# ai-study-lab 协作指南

这个仓库用于公开维护 AI、Agent、深度学习、算法和工程实践相关的中文学习资料。

后续 Codex 在本仓库内工作时，默认把任务理解为：维护公开学习资料、GitHub Pages 网站入口、README、导航、链接和发布质量。

## 维护边界

### 只处理公开学习资料相关内容

优先维护：

- `README.md`：公开仓库入口说明
- `index.html`、`catalog.json`：GitHub Pages 入口和目录数据
- 各专题目录下的 `README.md`、`课程索引.html`、`学习页.html`
- 面向公开阅读的学习笔记、课程拆解、资料索引、导航页
- GitHub Pages 可访问路径、站内链接、标题、目录和发布前检查

不处理：

- 个人日记、收件箱、主页、个人项目推进等内部内容
- 不属于本仓库的私有工作流文件
- 账号信息、隐私材料、未公开项目资料、敏感配置

如果任务需要跨出本仓库，先判断是否为了公开仓库发布质量所必需；不是必需就不要改。

## README 关系

本仓库的 `README.md` 是外部读者在 GitHub 上看到的主入口。更新公开学习资料时，同步检查 README 是否仍能说明仓库定位、主要入口和学习路径。

未经用户明确要求，不提交、不推送到远程。

## 内容组织

新增或整理公开资料时，优先沿用现有结构：

- 专题目录：如 `大语言模型/`、`Agent/`、`算法/`、`深度学习/`
- 学习页：优先放在对应专题的 `研习资料/`
- 原始或过程性笔记：优先放在对应专题的 `笔记/`
- 课程入口：优先维护 `README.md` 或 `课程索引.html`

大量新增学习页时，同步检查：

- `README.md`
- 对应专题的 `README.md` 或 `课程索引.html`
- 上一级导航入口是否能让读者找到新内容

## 公开目录生成

`catalog.json` 是 GitHub Pages 首页使用的公开目录，不直接代表仓库里所有文件。

维护首页目录时使用：

```bash
python scripts/build_catalog.py
```

脚本采用白名单发布：只有写入 `scripts/build_catalog.py` 的 `PUBLIC_COURSES` 的目录会进入 `catalog.json`。半成品、私有整理过程、生成工具和暂不适合公开的目录，即使本地存在，也不要放入白名单。

如果某个目录只是本地继续打磨，优先写入 `.gitignore` 并用 `git rm --cached` 从公开仓库下架，保留本地文件，不直接删除资料。

## 写作与链接规范

- 始终使用中文回答和维护文档，必要时保留英文术语
- 内容面向公开读者，避免只对 Vault 本人可理解的上下文
- 标题要清晰、可检索、可作为导航文本
- 外部链接使用标准 Markdown 链接
- Obsidian 内部可继续使用 `[[WikiLinks]]`，但公开入口页要优先考虑 GitHub 和 GitHub Pages 的可读性
- 不使用本机绝对路径作为公开链接
- 对来源、日期、课程名、论文名和版本信息尽量写清楚

## 发布质量检查

发布或准备发布前，至少检查：

- README 是否说明仓库定位、主要入口和学习路径
- 新增页面是否能从公开导航进入
- Markdown 链接、HTML 链接和图片路径是否可用
- 是否误包含私有 Vault 链接、日记、收件箱、个人账号或敏感信息
- GitHub Pages 入口页面是否能正常打开、标题是否合理、移动端是否基本可读

如果涉及页面视觉或交互改动，启动本地预览或用浏览器实际打开检查。

## 工具偏好

- 查找文件和文本优先用 `rg` / `rg --files`
- Python 包管理统一使用 `uv`
- 高噪声 CLI 输出优先用 `chop` 压缩
- 不为了小改动扫描整个仓库；先定位，再读取必要文件

## Git 约定

- 默认只做本地修改
- 不提交、不推送，除非用户明确要求
- 推送或发布前，先说明本地变更、远程差异和预期影响
- 遇到已有未提交改动，先保护用户改动，不擅自回滚
