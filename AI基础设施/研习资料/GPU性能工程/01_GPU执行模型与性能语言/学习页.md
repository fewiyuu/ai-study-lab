# 先抓主线 {#overview}
<!-- tag: overview -->
<!-- layout: grid2 -->

## box soft
### 先把问题改写成三句话
这个 kernel 的工作怎么分给硬件？数据怎么在寄存器、shared memory、L2、HBM 之间流动？SM 为什么会等？
这三句一旦问清楚，很多“先加 shared memory”“先调 BLOCK_SIZE”的冲动都会慢下来，变成可验证的假设。

## box info
### 本页的边界
本页不讲完整 CUDA 语法，不讲微架构深水区，也不直接做 profiler 截图分析。它只负责把判断词典搭起来，让你读后续单元时知道该看什么。

**1**先分清软件层级和硬件层级。
**2**再分清“有很多工作”与“工作真的被推进”的差别。
**3**最后把数据移动、资源占用和同步边界放进同一张表。

## box
### 一个固定锚点：向量加法 kernel
后面所有例子都围绕同一件事展开：`c[i] = a[i] + b[i]`。它足够小，能看出执行模型；又足够真实，能看出带宽、launch overhead、边界处理和资源占用。
向量加法案例复制
```python
__global__ void vec_add(const float* a, const float* b, float* c, int n) {
 int i = blockIdx.x * blockDim.x + threadIdx.x;
 if (i < n) {
 c[i] = a[i] + b[i];
 }
}

int threads = 256;
int blocks = (n + threads - 1) / threads;
vec_add<<<blocks, threads>>>(a, b, c, n);
```

# 状态账本：层级读写和症状 {#mechanism}
<!-- tag: trace -->

下面这张表把每一层的读写和症状摆出来。先按这个顺序看，再去看 warp、SM、shared memory 这些词，判断会稳很多。

层级
你写出来的对象
硬件看到的推进方式
出问题时的症状


grid
一个 kernel launch 的全部 block
分散到多个 SM 上执行，顺序不要假设固定
把 block 间顺序当成逻辑前提，会写出依赖错位的代码


block
同一组可以局部协作的线程
通常驻留在同一个 SM，可共享 block 级 shared memory
把 block 当成物理核心，或者错误假设 block 间同步


warp
32 个线程的调度和 SIMT 单位
调度器按 warp 选择可发射指令的候选
只看 thread 数，不看 warp 是否都在等同一种瓶颈


SM
资源容器和执行工作站
保存寄存器、shared memory、warp 上下文并发射指令
寄存器或 shared memory 太满，active warp 变少，延迟隐藏失败

# 例子：向量加法和 attention 两条线 {#example}
<!-- tag: example -->

向量加法用来暴露低算术强度，FlashAttention 用来暴露中间矩阵的 IO 路径。两个例子看的是同一件事：数据到底有没有必要经过慢速层级。

## box
### 例子 1：向量加法为什么是好起点
它很简单，但恰好能把工作分配、访存模式和算术强度同时看清楚。每个元素读两个值、写一个值，只做一次加法，算术强度很低，优先怀疑带宽。
向量加法账本复制
```text
每个元素:
- 读 a[i], b[i]
- 写 c[i]
- 只做 1 次加法
=> 计算很少, 数据移动占比很高
```

观察项
看到什么
该怎么想


FLOPs每个元素只有 1 次加法算术很少，先别把问题归到计算吞吐
bytes moved每元素约 12 bytes带宽通常比算力更关键
launch如果 kernel 很小，启动开销会显眼考虑融合、批量化或减少碎片 launch
boundary最后一个 block 可能越界必须有边界判断或 mask

## box info
### 例子 2：FlashAttention 如何减少中间矩阵记录下来
它更准确的变化是 IO 路径。标准 attention 如果显式物化 N x N 中间矩阵，就会把大量数据反复写回慢速内存；FlashAttention 的价值是把中间状态留在更快的层级，减少 HBM 往返。
Attention 公式复制
$$
 S = QK^T,\quad P = \operatorname{softmax}(S),\quad O = PV
 $$
 这里真正要问的是：S 和 P 必须完整落到慢速内存吗？如果答案是否定的，性能语言就会立刻转向 IO 和分块。
```text
S = QK^T
P = softmax(S)
O = P V
关键点在中间矩阵要不要完整写回慢速内存
```

# CUDA 读法：先问四件事 {#cuda}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box soft
### 读 CUDA 代码时先问四件事
1. 工作粒度：一个 thread、一个 warp、一个 block 各处理多少数据？
2. 访存模式：相邻 thread 是否访问相邻地址，能否形成合并访存？
3. 资源占用：每个 thread 用多少寄存器，每个 block 用多少 shared memory？
4. 同步边界：是否需要 block 内同步，是否误以为 block 间也能同步？

## box warn
### 不要把 block 大小当成固定答案
`256` threads per block 是常见起点，只是经验值。更大 block 可能提高并行度，也可能增加寄存器或 shared memory 压力；更小 block 可能提高调度灵活性，也可能让 active warp 不够。

对象
先看什么
常见误判


vec add kernel索引映射和边界判断以为 block 数越多就越快
tensor kernel访存和资源占用只看 thread 数，不看 warp 是否能继续推进
fused kernel中间量是否真的需要写回把“融合”理解成只是在一个函数里写更多代码

# Triton 读法：program、tile 和 mask {#triton}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box
### 块级思维怎么读
Triton 把线程细节上提成“一个 program 处理一个 tile”。你还是要关心并行度、mask、访存合并、寄存器和 cache 行为，但代码读法从“每个 thread 干什么”变成“每个 program 处理哪个块级向量”。
Triton tile 账本复制
```python
@triton.jit
def add_kernel(x_ptr, y_ptr, out_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
 pid = tl.program_id(axis=0)
 offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
 mask = offsets < n_elements
 x = tl.load(x_ptr + offsets, mask=mask)
 y = tl.load(y_ptr + offsets, mask=mask)
 tl.store(out_ptr + offsets, x + y, mask=mask)
```


## box info
### 三个关键字
`program_id` 是块级任务编号，`BLOCK_SIZE` 决定 tile 粒度，`mask` 负责边界安全。它们一起回答的是“这个 program 负责哪一段数据，怎么不越界，为什么这个 tile 大小值得这样选”。

<!-- tag: measure -->

先定基线，再看改动。很多“调好了”的参数，其实只是把瓶颈换到了另一层资源上，所以要先记录它改动了什么。

## box soft
### 为什么不能只靠感觉
“看起来像带宽瓶颈”只能算假设。要把它变成工作结论，至少要能说出你会看什么：kernel 时间、吞吐、带宽占用、warp stall 原因、launch 次数，或者更后面的 Roofline 位置。
这也是后续第 04 单元的入口：先测，再判断，再动手改。

### 失败后果
如果只拿 occupancy 当答案，就容易把问题修歪：block 变大、寄存器更多、shared memory 更满，最后 active warp 反而更少。看上去像更“满”，实际上更慢。

# 真实排错：把一条调优链走完整 {#why-it-slows}
<!-- tag: chain -->
<!-- layout: grid2 -->

## box soft
### 一条完整链
先看向量加法。它的每个元素只做一次加法，却要读两个输入、写一个输出。这个时候，性能瓶颈通常落在数据搬运。
如果你把 block 调大，只是为了让 occupancy 更高，但寄存器和 shared memory 跟着上涨，SM 能驻留的 warp 反而可能更少。表面上更满，实际更慢。

## box warn
### 失败的后果
常见错误是把“线程更多”直接等价成“更快”。结果是：边界判断没改好、launch overhead 没处理、缓存和访存形状也没变，最后只是把问题搬到另一个瓶颈上。
真正的下一步是先判断：这个 kernel 是带宽受限、资源受限，还是 launch 太碎？

阶段
看见什么
决定什么


读代码索引、边界、线程映射工作是否切对
算账FLOPs / bytes moved先看算力还是带宽
看资源寄存器、shared memory、block 大小驻留数量会不会下降
看结果stall reason、利用率、吞吐下一步改什么，要用结果来定

# 访存合并：地址形状先于参数 {#memory-path}
<!-- tag: bandwidth -->
<!-- layout: grid2 -->

## box soft
### 先看地址形状
GPU 的带宽先看地址形状，再看线程数。对于一个 warp 来说，如果线程访问的是连续、对齐、同类型的地址，硬件更容易把请求合并成少量内存事务；如果访问带着 stride、转置或者尾部补齐，事务数量就会上升，单次搬运的浪费也会变大。
这就是为什么同样是向量加法，连续布局通常很顺，而稍微换成跨步读取、打乱索引或切片视图后，性能会明显变差。算式没变，地址形状变了，结果就会变。

## box warn
### 别先加参数
很多调参动作看起来是在“增强并行”，实际只是让更多 warp 一起做同样低效的访问。线程数量增加以后，如果每个 warp 仍然在拆散地址、重复穿过慢速层级，吞吐不会像预期那样线性上升。
所以先问三个问题：地址是否连续，是否需要 stride，最后一段有没有尾部处理。把这三件事说清楚，shared memory tile、vectorized load、padding 和对齐才有意义。

访问形状
warp 看到的地址
事务效果
常见症状
先试什么


连续、对齐
相邻元素排在一起
更容易合并成少量事务
带宽利用率更稳定
保持连续布局和对齐


跨步、转置
地址间隔变大
事务增多，浪费上升
吞吐下降但算术并没变
改数据布局或做 tile 重排


尾部补齐
部分 lane 失活
最后一段更容易出现无效访问
边界处理变慢或越界
加 mask / padding / 边界分支

## box info
### 和后面单元怎么接
下一节会继续看到 memory hierarchy、shared memory 和 bandwidth。这里先记住一个判断句：当访问形状不好时，更多线程并不会自动换来更多有效带宽；先修地址形状，再谈更复杂的 tile 和缓存策略。

# 一个 warp 怎么读一个小 kernel {#warp-lane}
<!-- tag: lane -->
<!-- layout: grid2 -->

## box soft
### 先看最小版本
把例子缩到最小：一个 block 里正好放 32 个线程，刚好组成一个 warp。每个线程负责一个 float，grid 负责把输入切成若干个 32 元素区间。这个版本没有花哨算子，只有“谁读谁写、谁和谁一起推进”三件事，因此最适合看执行模型。
单 warp 拷贝复制
```python
__global__ void warp_copy(const float* in, float* out, int n) {
 int tid = blockIdx.x * blockDim.x + threadIdx.x;
 if (tid < n) {
 out[tid] = in[tid];
 }
}
```

读这段代码时，先画三层映射：`grid` 决定发多少个 block，`blockDim.x = 32` 让一个 block 正好等于一个 warp，`threadIdx.x` 决定 lane 号。这样一来，`tid` 既是元素下标，也是每个 lane 在同一轮里访问的地址位置。

## box warn
### 性能症状从哪里冒出来
当访问保持连续时，32 个 lane 往往能形成更少的内存事务，吞吐主要受带宽上限约束。只要把地址改成跨步读取，事务就会散开，warp 还是那个 warp，地址形状却更差了，结果常见是带宽利用率下降、memory stall 上升、表面线程数不变但速度变慢。
这类症状很容易被误读成“GPU 没有吃满”。更准确的说法是：warp 在跑，SM 也在调度，只是每轮数据搬运做得更碎，很多 lane 的时间花在等内存上。

步骤
grid / block / thread / warp
读写什么
看见的症状
下一步


1
grid 发出若干 block
把输入切成连续区间
block 数过少时并行度不够
先看 batch 和 launch 粒度


2
block 里 32 个 thread 组成一个 warp
lane 0..31 读写相邻元素
访问连续时事务少
保持对齐和连续布局


3
threadIdx.x 决定 lane 编号
每个 lane 负责一个位置
边界判断错会出现无效 lane
补 mask 或边界分支


4
warp scheduler 选择就绪 warp
等待内存返回或继续发射
memory dependency stall 上升
减少跨步访问和中间往返

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box info
### 练习钩子
把这个 kernel 改成 `out[tid] = in[tid * 2]`，再问自己三件事：lane 的地址还连续吗，warp 的事务数会怎么变，最先出现的症状会落在吞吐、stall 还是边界处理上。你不需要先背结论，只要把这三问答成可检查的句子，就已经能开始做性能判断了。

## box
### 边界提醒
这个例子故意把算术压到最低。先学会读执行模型和地址形状，再回头看 shared memory tile、矩阵乘和注意力时，判断会更稳。这里要先抓的是“谁在一起推进、谁在等内存、谁把事务拆散了”。

# launch 太碎时，先看交接成本 {#fusion-lanes}
<!-- tag: fusion -->
<!-- layout: grid2 -->

## box soft
### 把碎 kernel 合成一条链
有些程序kernel 太多、每个都太短。每次 launch 都会引入固定开销，CPU 还要不断和 GPU 交接。结果是，真正花在计算上的时间不多，花在启动、同步和排队上的时间反而显眼。
两个短 kernel复制
```python
normalize<<>>(x, y, n);
scale<<>>(y, out, n);
```

如果这两步之间没有必须落地的同步点，很多时候可以把它们合成一条链：先归一化再缩放，让中间结果留在寄存器或更近的层级，少一次写回，也少一次 launch。

## box warn
### 慢的后果
碎 launch 的典型症状GPU 利用率低、队列里有空洞、profile 里小 kernel 堆了一串。你会看到很多“看起来没做多少事”的时间，却很难把它直接归到一个算子上。
这时只调 block size 往往不够，因为问题根本不在单个 kernel 的内部，而在核与核之间的交接成本。

场景
看到什么
先改什么
如果不改


两个短 kernel
每个 kernel 都只有十几微秒
考虑融合成一条执行链
launch overhead 会吃掉很大一块时间


中间张量落地
一次写回、一次再读
让中间值留在更近的层级
HBM 往返增加，带宽压力更大


同步过多
CPU/GPU 交接频繁
合并 launch 或减少同步点
调度空洞更明显，吞吐波动更大

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box info
### 练习钩子
把这两个 kernel 画成一条时间线：哪里是计算，哪里是交接，哪里是写回。然后回答一句话：如果它们能融合，最先减少的是哪一类开销。这个问题会直接连到后面的矩阵算子、Triton tile 和 attention 融合。

## box
### 边界提醒
只有中间结果要被别的分支复用，或者同步边界必须保留时，融合才该收住。判断重点是中间值能不能留得更近；是否把步骤合成一条链，还要看复用和同步边界。

# FlashAttention 视角 {#flashattention}
<!-- tag: boundary -->
<!-- layout: grid2 -->

## box soft
### 核心直觉
FlashAttention 保持 exact attention，通过按块读取 Q、K、V，在片上内存里完成局部 attention，并用在线 softmax 维护归一化所需的最大值和分母。这样仍然得到精确结果，同时避免把完整 N x N 注意力矩阵写回慢速内存。

## box warn
### 它教会你看什么
同一个数学表达式可以有不同的数据移动路径。性能工程常常改的是中间量的停留位置、停留时长和写回方式。

问法
普通实现
FlashAttention 式问法


中间结果
可能显式保存 attention scores / probabilities
按 tile 计算并在线合并，尽量不物化完整 N x N 矩阵


瓶颈
只看矩阵乘 FLOPs
同时看 HBM 读写量、片上复用和 softmax 数值稳定


工程边界
认为算法快就一定端到端快
还要看序列长度、head dim、精度、因果 mask、硬件型号和框架集成

# 常见误区与排错 {#mistakes}
<!-- tag: debug -->

概念 A
概念 B
区别
一句判断


Occupancy
Utilization
occupancy 是可驻留 warp 的比例；utilization 是硬件实际忙碌程度。
occupancy 高只说明“有候选工作”，不说明“候选工作都有效推进”。


Latency
Bandwidth
latency 是一次操作等多久；bandwidth 是单位时间搬多少。
GPU 常用并发隐藏 latency，但带宽上限不能靠更多等待完全绕过。


CUDA thread
Triton program
CUDA thread 是线程级实例；Triton program 通常是块级 tile 实例。
读 Triton 时先找 tile 形状，不要逐行想象成单个 thread。


计算优化
IO 优化
计算优化减少或加速算术；IO 优化减少慢速内存流量或提高复用。
FlashAttention 的核心是 IO-aware，保持 exact attention，只改变中间结果的停留方式。

### 3 填空

### 4 解释

**Q:** Q4. 用三句话解释 grid、block、thread、warp、SM 的关系。

Explain: 要点：grid 包含 block；block 内线程会按 warp 组织；warp 在 SM 上被调度推进。答题时要提到 block 之间默认不能直接协作，也不要把 block 当成硬件核心。

### 5 choice

**Q:** Q5. occupancy 高一定代表 kernel 快吗？

- [ ] A. 一定，因为更多 warp 总会提升吞吐
- [ ] B. 不一定，它只是延迟隐藏的条件之一
- [ ] C. 一定无关，可以完全忽略

Answer: B. 不一定，它只是延迟隐藏的条件之一
Explain: occupancy 只是延迟隐藏的条件，不等于实际吞吐已经高。

### 6 choice

**Q:** Q6. 一个算子每个元素读写约 12 bytes、只做 1 次 FLOP，更可能先看计算吞吐还是内存带宽？

- [ ] A. 计算吞吐
- [ ] B. 内存带宽
- [ ] C. 只看 block 数

Answer: B. 内存带宽
Explain: FLOP/byte 很低时，先看带宽更合理。

### 7 choice

**Q:** Q7. arithmetic intensity 的定义更接近哪种说法？

- [ ] A. FLOPs + bytes moved
- [ ] B. FLOPs / bytes moved
- [ ] C. bytes moved / FLOPs

Answer: B. FLOPs / bytes moved
Explain: 它描述每搬一个字节做多少计算。

### 8 错误诊断

**Q:** Q8. 诊断：某 kernel GPU utilization 低，但每次运行只有十几微秒。你会先怀疑什么？

Explain: 优先怀疑 kernel 太小、launch 太碎、CPU/GPU 同步过密、SM 上没有足够可驻留工作。要能说出会看哪些时间线或 profiler 指标，而不是只写“线程不够”。

### 9 解释

**Q:** Q9. 解释 latency hiding：它隐藏了什么，没有消灭什么？

Explain: 它隐藏的是等待时间在整体执行中的可见程度，不是把内存访问或依赖消掉。优秀答案会区分“某次访问仍然慢”和“SM 没有一直等它”。

### 10 choice

**Q:** Q10. Triton 中 `tl.program_id(axis=0)` 更接近哪种概念？

- [ ] A. 当前 CUDA thread 的 lane id
- [ ] B. 当前块级 program/tile 的编号
- [ ] C. 当前 GPU 的 SM 编号

Answer: B. 当前块级 program/tile 的编号
Explain: 它对应块级 program 编号，不是单个 CUDA thread。

### 11 判断

**Q:** Q11. FlashAttention 的核心是用近似 attention 换速度。

- [ ] 对
- [ ] 错

Answer: false
Explain: FlashAttention 保持 exact attention，核心是改中间状态的停留方式和 HBM 往返。

### 12 代码阅读

**Q:** Q12. 读 Triton add kernel，解释 `offsets` 和 `mask` 分别解决什么问题。

Explain: offsets 把 tile 中每个位置映射到全局地址； mask 处理边界和越界问题。答案要能解释为什么没有 mask 会有越界风险。

### 13 迁移

**Q:** Q13. 用性能语言解释 FlashAttention 为什么关注 `N x N` 中间矩阵。

Explain: 要说明标准实现若物化 scores/probabilities，会让 N x N 中间结果反复经过 HBM；FlashAttention 通过分块和在线 softmax，把大量中间状态留在更快的片上层级。

### 14 综合解释

**Q:** Q14. 综合解释：看到一个慢 attention kernel，你的前三个诊断问题是什么？

Explain: 先问工作如何映射到 warp 和 SM；再问数据是否重复搬运、是否物化中间矩阵；最后问 profiler 或计时如何验证瓶颈，而不是凭感觉换参数。
