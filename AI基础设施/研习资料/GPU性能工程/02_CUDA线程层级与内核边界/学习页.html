先看全局：一段代码怎样被切成很多份

overview

最小直觉

CUDA kernel “同一段函数由很多线程同时跑”。这些线程会先被分成 block，再被安排进 grid。每个线程拿到自己的坐标，去处理自己那一份数据。

这意味着读 kernel 代码时，第一步先问“这段逻辑是给哪一层坐标用的”，再问它具体做什么。

这里先别追

先别钻 warp 调度、occupancy、寄存器压力和共享内存优化。它们会在后面的单元再回来。当前只要把坐标、边界和同步三件事看清就够了。

坐标追踪：从局部编号到全局编号

trace

一维向量加法

vec_add复制

__global__ void vec_add(const float* a, const float* b, float* c, int n) {
 int i = blockIdx.x * blockDim.x + threadIdx.x;
 if (i < n) {
 c[i] = a[i] + b[i];
 }
}

这一段代码的关键在于先把 i 算对，再决定要不要做加法。如果少了前半句，每个 block 都会从 0 重新开始算，结果就是重复处理同一段数据。

二维映射：矩阵里的 row 和 col 怎么来

matrix

从向量到矩阵

很多 kernel 处理矩阵或 tile。此时 threadIdx 也会从一维扩成二维：x 方向通常对应列，y 方向通常对应行。真正的下标不再只是“第几个元素”，而是“第几行、第几列”。

这一步很重要，因为很多越界和性能问题都出在 row/col 组合后是否和内存布局一致。

二维公式

2D index复制

int col = blockIdx.x * blockDim.x + threadIdx.x;
int row = blockIdx.y * blockDim.y + threadIdx.y;
int offset = row * width + col;

这三行里，前两行负责把线程坐标变成 row 和 col，最后一行负责把二维坐标压回线性地址。只要 width 写错，或者把 row 和 col 反过来，结果就会读写到完全不同的位置。

形状选择：什么时候用 1D，什么时候用 2D

choice

很多问题不必硬塞成一维

很多初学者会默认把所有数据都压成一维下标来算，因为这样公式看起来更短。但一旦工作天然有行和列，或者一个 tile 本身就是二维块，继续压成一维会让你更难看见边界和布局。

更稳的办法是先问：这个 kernel 的真实工作单位是什么？如果是向量，1D launch 很自然；如果是矩阵 tile，2D launch 通常更直观。这里追求的是读出来就能判断哪条边界会出错。

决定前先看三件事

  • 数据本身是不是有行列结构。
  • 线程是否需要按 row 和 col 分工。
  • 边界是否更容易在二维坐标里表达。

场景

优先形状为什么常见错法
向量加法1D每个线程处理一个元素，坐标关系最简单强行加上 y 维，反而让 guard 更绕矩阵逐元素处理2Drow 和 col 直接对应输入结构压成 1D 后忘了宽度，导致 offset 计算混乱矩阵 tile 协作2D块级协作和边界通常按二维切分更清楚只看 thread 数，不看 tile 的行列边界

2D launch复制

dim3 block(16, 16);
dim3 grid((width + block.x - 1) / block.x,
 (height + block.y - 1) / block.y);

matrix_kernel<<<grid, block>>>(input, output, width, height);

这里最重要的是你先决定行和列各自怎么分配，再让 launch 去表达这个决定。这样一来，最后一列和最后一行的 guard 也会自然分开，不会挤在一条长公式里。

你现在应该能做什么：给你一个新 kernel，你能先判断它应该用 1D 还是 2D launch，而不是默认把所有问题都压成一维。

块形状和最后一块：最容易漏的边界在这里

boundary

最后一个 tile 往往不满

不管你用 1D 还是 2D launch，最后一个 block 或 tile 都常常不是满的。你算出来的 grid 只是覆盖完整工作范围的外壳，真正有效的数据只占其中一部分。

这意味着最后一块既要能覆盖真实数据，又要让多出来的线程停在 guard 外面。很多新手会在这里犯两个相反的错误：要么把 guard 写得太严，漏掉最后几个元素；要么把 guard 写得太松，放进越界访问。

一个更完整的矩阵例子

partial tile复制

__global__ void row_major_copy(const float* in, float* out, int width, int height) {
 int col = blockIdx.x * blockDim.x + threadIdx.x;
 int row = blockIdx.y * blockDim.y + threadIdx.y;
 if (row < height && col < width) {
 int offset = row * width + col;
 out[offset] = in[offset];
 }
}

这段代码把二维 guard 拆开写。这样做的好处是：当最后一列不满时，col < width 能单独挡住列越界；当最后一行不满时，row < height 能单独挡住行越界。两个条件任何一个漏了，最后一块都可能写坏数据。

观察

正常情况出错症状要先查什么
最后一列多出来的线程被 col < width 挡住右边界数据被写歪或越界列 guard 是否存在最后一行多出来的线程被 row < height 挡住底边数据缺失或重复行 guard 是否存在二维 tile行列各自按自己的边界收尾边界一侧好，一侧坏guard 是不是只写了一半

你现在应该能做什么：看到一个不满的矩阵 tile 时，能先判断哪一维会越界、哪一维会缺数据，而不是只说“最后一块要特别处理”。

启动与边界：为什么要向上取整

launch

先选线程数，再补 block 数

实际写 kernel 时，经常先定每个 block 的线程数，再用元素总数去算需要多少个 block。因为总元素数不一定正好被 block 大小整除，所以 block 数通常要向上取整。

要点在于把边界外那部分线程交给 guard 处理。guard 不只是保险，它也是 launch 方式的一部分。

别把 launch 当成自动补洞

GPU 会帮你启动更多线程，但不会帮你决定这些线程该写到哪里。边界外的线程如果没有 guard，就会把写入带到非法地址；如果 guard 写错，就会把最后一段真实数据漏掉。

主机发起和设备执行：kernel 边界到底在哪

boundary

边界不是抽象词，是一条可见的交接线

CUDA 代码常常让人误会：既然 kernel 里有很多线程，它们会不会像普通函数调用一样，沿着同一个调用栈往下跑？实际过程不同。主机端先发起 launch，再把参数、grid 和 block 送到设备端；设备端只负责按这个配置执行。你在 host 里能写的，是“什么时候启动、启动几次、启动哪些阶段”；你在 device 里能写的，是“每个线程在自己的坐标下处理什么”。

这层边界如果没看清，后面很多错误会被误判。比如，程序卡住时，你可能以为是线程内部死循环；但真正原因可能是 host 没有等到前一个 kernel 完成，就去读还没写好的输出。反过来，设备端的 block 间也没有天然全局同步，所以 host/ device 两边各自的边界都要分清。

最小阶段链

two-kernel stage复制

stage1<<<blocks, threads>>>(input, mid, n);
cudaDeviceSynchronize();
stage2<<<blocks, threads>>>(mid, output, n);

这三行不只是“写法”。它们表达的是：先在一个阶段里把中间结果写出来，再明确等它完成，最后再启动下一阶段。这样做的好处是，把原来容易混淆的“kernel 内同步”换成了更清楚的“kernel 之间的阶段边界”。

同步边界：block 内能同步，block 间不要假设

sync

block 内

__syncthreads() 的边界只覆盖同一个 block。它的意思是这个 block 里的线程先在这里对齐，再往下执行。

block 间

不同 block 之间没有默认的全局同步关系。不要把“看上去都在同一个 kernel 里”误读成“它们会按你写的顺序一起到达某个点”。如果真需要阶段边界，通常要拆 kernel，或者使用更明确的协作机制。

数据所有权：谁写缓冲区，谁才能在下一步读

ownership

把缓冲区当成有主人的资源

CUDA 里很多错误不是算错，而是读早了。你可以把输出缓冲区、中间缓冲区、下一阶段输入都看成有“所有权”的资源：当前阶段负责写，下一阶段才负责读。只要读写顺序错了，结果就可能看起来像随机坏掉，但本质上只是拿了还没写完的数据。

这也是为什么 host 端的同步并不是可有可无。cudaDeviceSynchronize() 的意义让你知道前一个阶段真的写完了。没有这个边界，host 可能提前拿到旧值；有了这个边界，后续读写顺序才可控。

一个最小所有权链

ownership chain复制

fill_intermediate<<<blocks, threads>>>(input, mid, n);
cudaDeviceSynchronize();
consume_intermediate<<<blocks, threads>>>(mid, output, n);

这段链路里，mid 先是前一阶段的写入目标，随后变成下一阶段的读入来源。只有当中间阶段真的完成，mid 才算“从写转到读”。

失败实验室：三种常见错误怎么分

diagnosis

调试顺序

先查坐标，再查 guard，最后查同步。顺序反过来很容易把“编号错了”误看成“同步错了”，或者把“边界没挡住”误看成“GPU 执行乱序”。

排错顺序：先问三个问题，再看结果

checklist

第一问：编号对吗先确认是不是把局部编号当成了全局编号。这个错误最常见，也最容易把所有 block 的输出写重。

第三问：同步边界对吗最后看是不是把 block 内同步当成了全局屏障，或者把需要阶段边界的逻辑塞进了同一个 kernel。

编号每个 block 都在写同一段输出检查 blockIdx.x * blockDim.x先不要调 block 大小

边界最后一块偶发越界检查 if (i < n)先不要怀疑显存坏了

同步单 block 正常，多 block 不稳检查是否存在隐式顺序依赖先不要把 __syncthreads() 当全局同步

和下一节怎么接

bridge

下一步会看到什么

下一节开始看内存层次、访存合并和带宽。那时坐标不再只是“线程编号”，还会变成“这些线程写到的是连续地址还是分散地址”。如果这一页没把全局编号和边界 guard 看稳，后面的访存问题会很难读。

Q: Q18. 用 200 字以内解释 grid、block、thread、guard 和 __syncthreads() 的关系。

参考答案 / 评分要点

。一个合格答案要说明：grid 承载全部 block，block 里有多个 thread，thread 用全局编号处理自己的那份数据；guard 负责挡住多出来的线程； __syncthreads() 只让同一 block 内的线程对齐，不是全局屏障。写到这里就够了，不要再去展开 warp 或性能优化。
