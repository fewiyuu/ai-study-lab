先抓主线：数据是怎么变慢的

overview

最小直觉

GPU kernel 慢，常见原因不是“算不动”，而是“搬得不顺”。距离越远，访问代价越高；访问越碎，事务越多；事务越多，真正有用的字节占比就越低。这个判断要从地址和字节账本里推出来，不能只看线程数够不够。

这页沿用两个固定例子：一个是 c[i] = a[i] + b[i] 的向量加法，用来算有用字节；另一个是 row-major 矩阵 copy 和 transpose，用来判断地址是否连续、是否跨步。两个例子会贯穿后面的 trace、算账和失败实验室。

状态账本

先把“谁读、谁写、谁共享、谁过期”分清，再谈优化。很多内存问题其实不是缺一个更近的层级，而是把不该共享的东西共享了，或者把只用一次的数据搬近了。

层级 共享范围 读写方式 离太远时的症状 下一步小检查
register 单个线程 线程自己读写自己的工作值 寄存器压力高时会把变量挤出去，后面又得重新加载 先看局部中间值是否真的只给一个线程用
shared memory 一个 block block 内多线程读写同一份 tile 没复用就只是在多搬一次，还额外多一次同步 先问这块数据会不会在 block 内重复出现
L2 cache 多个 SM 硬件保留近期数据，程序不直接管理 局部性差时命中率低，还是得回到 HBM 先看访问是否近期会再次出现
HBM / global memory 整块 GPU 容量大，代价也最高 访问很碎时，吞吐看起来会被白白吃掉 先看线程映射和数据布局是否已经把地址弄碎
host memory CPU 侧 通常是上游来源，不是 kernel 热路径工作区 把它当成快速工作区，往往只会多一次搬运 先分清这一步是在喂数据，还是在做计算

一个固定起点

后面所有判断都围着两个场景转：FP32 向量加法和 row-major 矩阵 copy / transpose。前者用来算“每个元素到底搬了多少有用字节”，后者用来观察“线程坐标是顺着行走，还是顺着列跳”。

访存合并：warp 有没有整齐取货

trace
连续读 相邻线程访问相邻地址

warp 更容易合并成较少的 memory transaction，transaction 里真正有用的字节也更多。

连续但偏移 地址方向没变，起点偏了一格

访问仍然连续，但如果起点跨过了硬件喜欢的边界，就可能多出一次事务。

跨步读 相邻线程隔着很远

同样 32 个线程会扫过更散的地址，硬件常常要搬更多次，浪费的字节也更多。

块内复用 同一份数据会被多次用到

如果一个 block 会多次读同一块数据，把它留在 shared memory 或寄存器里，收益通常比单次访问更明显。

row-major 访问
# row-major matrix A[height][width]
float good = A[row * width + col];         # 相邻线程沿 col 读，地址更连续
float bad = A[(row + tid) * width + col];  # 相邻线程沿 row 跳，stride 更大

这里的“矩阵”只是载体，真正要看的，是线程在地址空间里横着走还是竖着跳。横着走更容易合并，竖着跳更容易把 transaction 撑大。

一个判断句

如果 threadIdx.x 每前进一步，地址也只前进一个元素大小，那通常更接近合并访问；如果 threadIdx.x 前进，但地址跨着一整行甚至几整行，带宽会先被布局吃掉。

接下来不要急着加 shared memory。先确认线程映射和数据布局是否已经把地址弄得太碎。

模式 warp 看到的地址 通常意味着什么 先查什么
连续读取 相邻线程读相邻地址 更容易合并，事务更少 线程映射是否和数据连续维度一致
连续但偏移 相邻线程仍然连续，只是整体偏了 4 B 或 8 B 方向没错，起点可能跨过边界 首地址和 tile 起点是否对齐
跨步读取 相邻线程隔着很多元素 transaction 数容易变多 数据布局能不能换，或者 block 维度能不能调
块内复用 同一块数据被多个线程重复使用 shared memory / register 可能有价值 这份数据是否会在 block 内反复读

和 profiler 怎么对上

这张记录单写完后，再去看 profiler 会更有方向。访问模式对应全局读写请求是否拉散；字节账本对应 achieved memory throughput 是否已经接近设备上限；复用次数对应 shared memory 是否真的减少了 HBM 访问；验证动作对应本次实验只改一个因素。

指标只负责告诉你现象，不会自动给出优化顺序。先修正地址形状，再确认复用，再比较有效带宽和峰值。遇到指标互相冲突时，先回到最小可复现实验，把输入规模、block 配置和计时方式固定住。

对齐边界：为什么差一个元素也会慢

alignment

连续还不够，还要看起点

很多人只检查“相邻线程是否读相邻元素”，就以为访存已经合并。这个判断还少一半：起始地址如果偏离硬件喜欢的边界，本来可以落在一段事务里的数据，可能会跨到两段事务里。地址方向是连续的，事务数量仍然会增加。

排查时可以先写出前三个线程的地址：A + offset + 4 * tid。当 offset 是 0、4、8 这类小偏移时，代码语义没有变，事务边界却可能变。对性能工程来说，这种“小偏移”不能只凭肉眼跳过。

一个手算办法

一个 warp 通常有 32 个线程。对 FP32 来说，连续读取的有用跨度大约是 32 × 4 B = 128 B。如果起点对齐，128 B 往往能落进更少的事务里；如果起点偏了 4 B，同样的 128 B 可能被拆成两段去搬。

$$32 \times 4\text{B} = 128\text{B}$$

这里的 32 是 warp 线程数，4B 是一个 FP32 元素大小。这个算式只是在帮你看跨度，不是在替代 profiler。它的作用是让你知道：连续访问也可能因为起点偏移而多出事务。

偏移检查
int addr0 = base + 4 * tid;
int addr1 = base + 4 + 4 * tid;
int addr2 = base + 64 * tid;
地址形式 相邻线程关系 额外风险 排查动作
A + 4 * tid 连续 主要看整体对齐 核对首地址和数据类型大小
A + 4 + 4 * tid 仍然连续 起点偏移可能多出一次事务 检查切片、padding、tile 起点
A + 64 * tid 跨步 地址天然拉散 优先改线程映射或数据布局

矩阵路径：copy 和 transpose 为什么看起来都像搬运，结果却不同

case

先看 copy

row-major 矩阵拷贝最适合拿来检查“地址是否连续”。如果 threadIdx.x 对着列方向走，warp 里的线程就会沿着同一行连续读写。这个时候，读和写两边都比较整齐，访存事务一般更少。

这类 case 的作用很明确：它告诉你，线程坐标本身不慢，慢的是坐标和数据布局错位后，地址变碎了。

再看 transpose

转置会把行和列交换。读的时候可能还是连续的，但写的时候会沿着列方向跨步；也可能反过来，写整齐、读跨步。结果看上去都是“搬一个矩阵”，本质却是某一边的访问变碎了，transaction 数先上去。

如果只盯着“结果正确”，很容易漏掉“哪一侧是连续、哪一侧是跨步”这件事。

matrix copy
__global__ void matrix_copy(const float* in, float* out, int width, int height) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    if (row < height && col < width) {
        int offset = row * width + col;
        out[offset] = in[offset];
    }
}

这段代码里，读和写都围着同一个 offset 走。只要 col 和连续维度对齐，copy 就能把访问顺着行铺开。

matrix transpose
__global__ void matrix_transpose(const float* in, float* out, int width, int height) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    if (row < height && col < width) {
        int in_offset = row * width + col;
        int out_offset = col * height + row;
        out[out_offset] = in[in_offset];
    }
}

这里的读和写不再走同一条路。哪一侧跨步大，哪一侧就更容易把 transaction 撑开。transpose 正好把这件事暴露得很明显。

case 读地址 写地址 更像什么瓶颈 先查什么
copy 连续 连续 更像纯带宽或 launch 开销 是否已经接近峰值，或者 kernel 是否太碎
transpose 一侧连续，另一侧跨步 另一侧跨步 更像访问形状问题 哪一边的 stride 被拉大了
tile 复用 先读一块，再多次复用 必要时再写出去 shared memory 可能有收益 这一块数据会不会在 block 内重复出现
失败后果

如果 copy 和 transpose 都只被看成“矩阵搬运”，你会很容易把优化方向弄反：该先调布局的时候去调 block，大多数情况下只是在换一个更贵的写法。先把哪一侧连续、哪一侧跨步说清，才知道该改数据布局、线程映射，还是后面的复用策略。

带宽账本：每秒到底搬了多少有用字节

bandwidth

有效带宽怎么记

这页使用“有效带宽”，只看算法真正需要读写的字节，再除以执行时间。这样算出来的数字，能直接拿去和理论峰值比量级，也能看出访问形状有没有把数据搬得太碎。

如果同样的计算，地址更连续、线程更整齐，实际事务数通常会下降；如果地址很碎，有效带宽就算得出来，也未必代表程序真的把数据搬得高效。

符号先说清

R 表示有用读字节，W 表示有用写字节，t 表示 kernel 执行时间。后面的公式只用这三个量，不把事务大小混进分子里。

$$BW_{eff} = \frac{R + W}{t}$$

这里的 R 和 W 是算法真正需要的字节数，和某一次硬件 transaction 的原始大小分开记录。先把“有用字节”记准，再去和时间相除，才知道这段代码的搬运效率在哪里。

例子 读字节 写字节 每元素总字节 备注
float32 向量加法 2 × 4 B 1 × 4 B 12 B 算术少，先怀疑带宽或访存形状
float32 逐元素 copy 1 × 4 B 1 × 4 B 8 B 几乎纯搬运，访问连续时更接近带宽上限
按列跨步读取 同样的有用字节，实际 transaction 往往更多 视输出而定 表面不变，搬运更碎 问题常常先出在布局，不一定是算力

一个可算的例子

设 N = 50,000,000，向量加法每个元素 12 字节有用流量，总有用字节就是 600,000,000 B。如果 kernel 跑了 0.5 ms，那么有效带宽就是 1.2 TB/s。这个数只是算术演示，真实判断还要对照硬件规格和输入形状。

怎么读这个数字

有效带宽高，不代表每一次 transaction 都漂亮；它只说明这段算法需要搬的字节，在给定时间里已经搬得很多了。有效带宽低，也不自动说明硬件很差，可能只是访问模式把事务切碎了。

数据复用：什么时候值得把数据搬近

reuse

先问复用次数

把数据搬进 shared memory，收益来自同一块数据会被 block 内多次使用。只用一次的数据，搬进去就只是在多走一趟路，收益未必覆盖 copy 和同步的代价。

反过来，如果同一份数据被很多线程重复读，那把它留在更近的层级，常常比继续从 HBM 反复取更合算。

这一页先记住的判断

shared memory 适合 block 内多次读的 tile；L2 cache 适合近期会再次出现的数据；register 适合线程自己的中间值；HBM 适合容量大但必须接受更高访问代价的数据。

tile 复用
# 一个 tile 先读进来，再被多个线程重复使用
tile[tid] = A[base + tid]
__syncthreads()
acc = tile[tid] + tile[tid + 1]
资源 共享范围 最适合的场景 误用时会怎样
register 单线程 局部中间值、少量循环变量 用太多会压低并发，反而让延迟更难藏住
shared memory 同一 block tile 重用、块内归约、重复读取的中间块 没复用就只是多搬一次，还可能多一次同步
L2 cache 多个 SM 近期会再次访问的数据 访问太跳时，命中率不会好看
HBM / global memory 整块 GPU 容量大、必须放在慢层级的数据 如果访问很碎，transaction 会先变多
先别追 bank conflict 这页先只看“值不值得搬近”。bank conflict、occupancy 下降和同步开销会在后面的单元继续回来；现在先把“复用是否足够”这个问题答准。

四线程地址账：先把 stride 手算出来

map

用四个线程看清一条规律

如果只看“32 个线程”，地址规律容易被数量淹没。把它缩成 4 个线程，反而更容易看出相邻线程是在连续走、偏移走，还是跨步走。这里先手算四个地址，再回到 warp 视角，判断 transaction 可能怎么变。

这个小表最适合拿来检查 row-major 矩阵的一行。因为一行里相邻元素本来就是连续的，只要 threadIdx.x 和列方向对齐，就能把“线程坐标怎么变”直接读成“地址怎么变”。

读写顺序

先写出 tid = 0, 1, 2, 3 的地址，再看每一步差多少字节；最后问一句：这四个地址如果放大到 32 个线程，会不会跨过更多事务边界。这样比直接盯着抽象公式更稳。

线程 A + 4 * tid A + 4 + 4 * tid A + 64 * tid 读到的结论
tid = 0 A A + 4 A 先看起点有没有偏移
tid = 1 A + 4 A + 8 A + 64 相邻线程差 4 B 还是 64 B
tid = 2 A + 8 A + 12 A + 128 连续、偏移、跨步三种形状开始分开
tid = 3 A + 12 A + 16 A + 192 放大到 warp 后，跨步最容易把事务变碎
四线程手算
for tid in range(4):
    print({
        "tid": tid,
        "contiguous": base + 4 * tid,
        "offset": base + 4 + 4 * tid,

失败后果

如果这四个地址里已经出现 64 B 级别的跳跃，那 32 个线程只会把地址拉得更散。先把四线程账算明白，再决定是改线程映射、改布局，还是回到 02 继续看坐标和边界。

读写账本：先看对象怎么流过 block

ledger

一个最小的 staging case

有些 kernel 会先把 A 的 tile 读进 shared memory，再由同一个 block 里的线程去算 c。这里要看的不是“用了没用 shared memory”，而是这块 tile 到底会不会被第二次、第三次读取。如果只读一次，这个 staging 只是在把 HBM 的读改成 shared memory 的读写加一次同步。

把对象流想清楚之后，症状会更容易分开：读写顺序清楚但仍然慢，通常是地址形状或带宽上限；读写顺序本身就不划算，shared memory 只是把慢的地方往前挪了一层。

先记三条线

谁读 A，谁写 tile，谁最后写 c。三条线没有理顺之前，复用判断很容易写错。最常见的误判是把“搬近了”当成优化完成，实际上只是换了一个更近的搬法。

tile staging
tile[tid] = A[base + tid]
__syncthreads()
x = tile[tid]
y = tile[tid + 1]
c[base + tid] = x + y
对象 进入前 进入后 读什么 写什么 错了的症状
A 的 tile 还在 HBM 或 global memory 被搬进 shared memory 每个线程先读一次原始数据 shared memory 里的暂存块 只读一次时，copy 和同步吃掉收益
tile 中间值 还没有被多个线程复用 block 内可以再次读取 同一份 tile 是否会被后续线程重复用到 少量寄存器或输出值 复用次数不够时，shared memory 不值
c 的输出 还没有写回 最终写回 global memory tile 的局部结果 输出数组 写回顺序错了，结果可能对但更慢
__syncthreads() tile 可能只写了一半 block 内所有线程都能看见完整 tile 前面的写入是否已经完成 无新数据，只改变可见顺序 缺了它，某些线程会读到旧值或半写入值，结果常常不稳定
失败后果

如果 tile 只读一次，shared memory 只会引入额外 copy 和同步，结果是表面上更近，实际更慢。如果 __syncthreads() 漏了，慢还不是最糟的，结果会变得不稳定：同一个输入，第一次对，第二次错。下一步先回到 02 的地址判断，再回到这页的复用判断。练习里的 Q12 和 Q14 正是在查这件事：复用是否真的存在，和布局问题到底谁先出现。

瓶颈判断：别把所有慢都直接叫带宽问题

diagnosis

先走四步

  1. 先数有用字节，不要先猜硬件事务。
  2. 再看地址形状，确认 warp 内是否连续、对齐、跨步太大。
  3. 再看复用，判断 shared memory 或 cache 有没有实际价值。
  4. 最后再把有效带宽和硬件峰值放在一起比。

什么时候别急着加 shared memory

如果数据只用一次，或者访问已经很连续，shared memory 可能只是多一次 copy。它真正有价值的时候，是同一块数据会被 block 内多次使用，并且复用足以覆盖同步成本。

观察到的症状 更像什么 先查什么 下一步动作
threadIdx.x 连续变，但地址跳得很大 访问形状有问题 线程映射和数据布局 先把连续维度和 threadIdx.x 对齐
有效带宽低很多，但算术很少 带宽或布局瓶颈 有用字节账本 估算能否靠布局或融合减少字节
shared memory 之后反而慢 复用不足或同步代价太高 是否只读了一次 先撤回 shared memory，再看是否真的需要
kernel 很短但不算快 launch 太碎 启动次数和同步点 考虑融合或批量化

排查记录单：把猜测写成可验证假设

triage

先写事实，再写动作

内存问题最容易陷入“感觉很像带宽瓶颈”。更稳的写法是把猜测压成四行：访问模式、字节账本、复用次数、下一步实验。四行都能填出来，后面才值得改代码；填不出来，先回到索引表达式和数据布局。

例如按列读 row-major 矩阵时，可以写成：warp 内地址跨一整行，有用字节按读写元素计算，同一元素没有块内复用，下一步先改线程映射或转成 tile 访问。这样记录之后，shared memory 是否值得加就会清楚很多。

这张单子怎么用

每次只改一个因素。先让地址连续，再测带宽；先确认复用次数，再引入 shared memory；先记录有效字节，再去看 profiler 指标。这样做的好处是失败结果也能留下信息：慢在哪个环节、下一步该删什么、该保留什么。

Q4. 解释为什么连续读取通常比跨步读取更容易合并。

参考答案 / 评分要点

连续读取让同一 warp 的线程访问相邻地址，硬件更容易把请求合并成较少的 memory transaction。跨步读取会把地址拉散，transaction 变多，真正有用的字节占比下降。答案要说出“相邻线程”“更少 transaction”“浪费字节更多”这三个点。
