# 先抓主线：数据是怎么变慢的 {#overview}
<!-- tag: overview -->
<!-- layout: grid2 -->

## box soft
### 最小直觉
GPU kernel 慢，常见原因“搬得不顺”。距离越远，访问代价越高；访问越碎，事务越多；事务越多，真正有用的字节占比就越低。这个判断要从地址和字节账本里推出来，不能只看线程数够不够。
这页沿用两个固定例子：一个是 `c[i] = a[i] + b[i]` 的向量加法，用来算有用字节；另一个是 row-major 矩阵 copy 和 transpose，用来判断地址是否连续、是否跨步。两个例子会贯穿后面的 trace、算账和失败实验室。

## box info
### 状态账本
先把“谁读、谁写、谁共享、谁过期”分清，再谈优化。很多内存问题其实把不该共享的东西共享了，或者把只用一次的数据搬近了。

# 访存合并：warp 有没有整齐取货 {#trace}
<!-- tag: trace -->

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box
row-major 访问复制
```python
# row-major matrix A[height][width]
float good = A[row * width + col]; # 相邻线程沿 col 读，地址更连续
float bad = A[(row + tid) * width + col]; # 相邻线程沿 row 跳，stride 更大
```

这里的“矩阵”只是载体，真正要看的，是线程在地址空间里横着走还是竖着跳。横着走更容易合并，竖着跳更容易把 transaction 撑大。

## box info
### 一个判断句
如果 threadIdx.x 每前进一步，地址也只前进一个元素大小，那通常更接近合并访问；如果 threadIdx.x 前进，但地址跨着一整行甚至几整行，带宽会先被布局吃掉。
接下来不要急着加 shared memory。先确认线程映射和数据布局是否已经把地址弄得太碎。

## box warn
### 和 profiler 怎么对上
这张记录单写完后，再去看 profiler 会更有方向。访问模式对应全局读写请求是否拉散；字节账本对应 achieved memory throughput 是否已经接近设备上限；复用次数对应 shared memory 是否真的减少了 HBM 访问；验证动作对应本次实验只改一个因素。

# 对齐边界：为什么差一个元素也会慢 {#alignment}
<!-- tag: alignment -->
<!-- layout: grid2 -->

## box soft
### 连续还不够，还要看起点
很多人只检查“相邻线程是否读相邻元素”，就以为访存已经合并。这个判断还少一半：起始地址如果偏离硬件喜欢的边界，本来可以落在一段事务里的数据，可能会跨到两段事务里。地址方向是连续的，事务数量仍然会增加。
排查时可以先写出前三个线程的地址：`A + offset + 4 * tid`。当 `offset` 是 0、4、8 这类小偏移时，代码语义没有变，事务边界却可能变。对性能工程来说，这种“小偏移”不能只凭肉眼跳过。

## box info
### 一个手算办法
一个 warp 通常有 32 个线程。对 FP32 来说，连续读取的有用跨度大约是 `32 × 4 B = 128 B`。如果起点对齐，128 B 往往能落进更少的事务里；如果起点偏了 4 B，同样的 128 B 可能被拆成两段去搬。

$$32 \times 4\text{B} = 128\text{B}$$
 这里的 32 是 warp 线程数，4B 是一个 FP32 元素大小。这个算式只是在帮你看跨度，不是在替代 profiler。它的作用是让你知道：连续访问也可能因为起点偏移而多出事务。

偏移检查复制

```python
int addr0 = base + 4 * tid;
int addr1 = base + 4 + 4 * tid;
int addr2 = base + 64 * tid;
```

额外风险

# 矩阵路径：copy 和 transpose 为什么看起来都像搬运，结果却不同 {#matrix-path}
<!-- tag: case -->
<!-- layout: grid2 -->

## box soft
### 先看 copy
row-major 矩阵拷贝最适合拿来检查“地址是否连续”。如果 `threadIdx.x` 对着列方向走，warp 里的线程就会沿着同一行连续读写。这个时候，读和写两边都比较整齐，访存事务一般更少。
这类 case 的作用很明确：它告诉你，线程坐标本身不慢，慢的是坐标和数据布局错位后，地址变碎了。

## box info
### 再看 transpose
转置会把行和列交换。读的时候可能还是连续的，但写的时候会沿着列方向跨步；也可能反过来，写整齐、读跨步。结果看上去都是“搬一个矩阵”，本质却是某一边的访问变碎了，transaction 数先上去。
如果只盯着“结果正确”，很容易漏掉“哪一侧是连续、哪一侧是跨步”这件事。

## box
matrix copy复制
```python
__global__ void matrix_copy(const float* in, float* out, int width, int height) {
 int col = blockIdx.x * blockDim.x + threadIdx.x;
 int row = blockIdx.y * blockDim.y + threadIdx.y;
 if (row < height && col < width) {
 int offset = row * width + col;
 out[offset] = in[offset];
 }
}
```

这段代码里，读和写都围着同一个 `offset` 走。只要 `col` 和连续维度对齐，copy 就能把访问顺着行铺开。

## box
matrix transpose复制
```python
__global__ void matrix_transpose(const float* in, float* out, int width, int height) {
 int col = blockIdx.x * blockDim.x + threadIdx.x;
 int row = blockIdx.y * blockDim.y + threadIdx.y;
 if (row < height && col < width) {
 int in_offset = row * width + col;
 int out_offset = col * height + row;
 out[out_offset] = in[in_offset];
 }
}
```

这里的读和写不再走同一条路。哪一侧跨步大，哪一侧就更容易把 transaction 撑开。transpose 正好把这件事暴露得很明显。

# 带宽账本：每秒到底搬了多少有用字节 {#bandwidth}
<!-- tag: bandwidth -->
<!-- layout: grid2 -->

## box soft
### 有效带宽怎么记
这页使用“有效带宽”，只看算法真正需要读写的字节，再除以执行时间。这样算出来的数字，能直接拿去和理论峰值比量级，也能看出访问形状有没有把数据搬得太碎。
如果同样的计算，地址更连续、线程更整齐，实际事务数通常会下降；如果地址很碎，有效带宽就算得出来，也未必代表程序真的把数据搬得高效。

## box info
### 符号先说清
`R` 表示有用读字节，`W` 表示有用写字节，`t` 表示 kernel 执行时间。后面的公式只用这三个量，不把事务大小混进分子里。

$$BW_{eff} = \frac{R + W}{t}$$
 这里的 `R` 和 `W` 是算法真正需要的字节数，和某一次硬件 transaction 的原始大小分开记录。先把“有用字节”记准，再去和时间相除，才知道这段代码的搬运效率在哪里。

## box
### 一个可算的例子
设 `N = 50,000,000`，向量加法每个元素 12 字节有用流量，总有用字节就是 `600,000,000 B`。如果 kernel 跑了 `0.5 ms`，那么有效带宽就是 `1.2 TB/s`。这个数只是算术演示，真实判断还要对照硬件规格和输入形状。

## box warn
### 怎么读这个数字
有效带宽高，不代表每一次 transaction 都漂亮；它只说明这段算法需要搬的字节，在给定时间里已经搬得很多了。有效带宽低，也不自动说明硬件很差，可能只是访问模式把事务切碎了。

# 数据复用：什么时候值得把数据搬近 {#reuse}
<!-- tag: reuse -->
<!-- layout: grid2 -->

## box soft
### 先问复用次数
把数据搬进 shared memory，收益来自同一块数据会被 block 内多次使用。只用一次的数据，搬进去就只是在多走一趟路，收益未必覆盖 copy 和同步的代价。
反过来，如果同一份数据被很多线程重复读，那把它留在更近的层级，常常比继续从 HBM 反复取更合算。

## box info
### 这一页先记住的判断
shared memory 适合 block 内多次读的 tile；L2 cache 适合近期会再次出现的数据；register 适合线程自己的中间值；HBM 适合容量大但必须接受更高访问代价的数据。

tile 复用复制

```python
# 一个 tile 先读进来，再被多个线程重复使用
tile[tid] = A[base + tid]
__syncthreads()
acc = tile[tid] + tile[tid + 1]
```

资源
共享范围
最适合的场景
误用时会怎样


register
单线程
局部中间值、少量循环变量
用太多会压低并发，反而让延迟更难藏住


shared memory
同一 block
tile 重用、块内归约、重复读取的中间块
没复用就只是多搬一次，还可能多一次同步


L2 cache
多个 SM
近期会再次访问的数据

## box soft

# 读写账本：先看对象怎么流过 block {#rw-ledger}
<!-- tag: ledger -->
<!-- layout: grid2 -->

## box soft
### 一个最小的 staging case
有些 kernel 会先把 A 的 tile 读进 shared memory，再由同一个 block 里的线程去算 c。这里要看的这块 tile 到底会不会被第二次、第三次读取。如果只读一次，这个 staging 只是在把 HBM 的读改成 shared memory 的读写加一次同步。
把对象流想清楚之后，症状会更容易分开：读写顺序清楚但仍然慢，通常是地址形状或带宽上限；读写顺序本身就不划算，shared memory 只是把慢的地方往前挪了一层。

## box info
### 先记三条线
谁读 A，谁写 tile，谁最后写 c。三条线没有理顺之前，复用判断很容易写错。最常见的误判是把“搬近了”当成优化完成，实际上只是换了一个更近的搬法。

> [!NOTE]
> **失败后果**
> 如果 tile 只读一次，shared memory 只会引入额外 copy 和同步，结果是表面上更近，实际更慢。如果 `__syncthreads()` 漏了，慢还不是最糟的，结果会变得不稳定：同一个输入，第一次对，第二次错。下一步先回到 02 的地址判断，再回到这页的复用判断。练习里的 Q12 和 Q14 正是在查这件事：复用是否真的存在，和布局问题到底谁先出现。

<!-- layout: grid2 -->

## box warn
### 先走四步
1. 先数有用字节，不要先猜硬件事务。

# 排查记录单：把猜测写成可验证假设 {#triage-card}
<!-- tag: triage -->
<!-- layout: grid2 -->

## box soft
### 先写事实，再写动作
内存问题最容易陷入“感觉很像带宽瓶颈”。更稳的写法是把猜测压成四行：访问模式、字节账本、复用次数、下一步实验。四行都能填出来，后面才值得改代码；填不出来，先回到索引表达式和数据布局。
例如按列读 row-major 矩阵时，可以写成：warp 内地址跨一整行，有用字节按读写元素计算，同一元素没有块内复用，下一步先改线程映射或转成 tile 访问。这样记录之后，shared memory 是否值得加就会清楚很多。

## box soft
### 怎么答才算会
这页的练习重点不在背术语，而在把“线程怎么读地址、字节怎么记、哪一步先查”说成能复用的判断。能算、能判、能解释下一步检查项，才算真的读懂。

### 检查重点
优先看三件事：线程映射有没有把连续维度对齐，访问地址有没有跨步或不对齐，复用是否足以让 shared memory 真正划算。答题时先说症状，再说原因，最后说下一步检查。

- [ ] B. shared memory
- [ ] C. L2 cache

Answer: A. register
Explain: register 是线程私有的工作区。

### 2 true/false

**Q:** Q2. 判断：shared memory 的共享范围通常是整个 GPU。

- [ ] 对
- [ ] 错


### 3 fill

**Q:** Q3. 填空：float32 向量加法每个元素的有用流量通常是 `____` bytes。


### 4 explain

**Q:** Q4. 解释为什么连续读取通常比跨步读取更容易合并。

Explain: 连续读取让同一 warp 的线程访问相邻地址，硬件更容易把请求合并成较少的 memory transaction。跨步读取会把地址拉散，transaction 变多，真正有用的字节占比下降。答案要说出“相邻线程”“更少 transaction”“浪费字节更多”这三个点。

<!-- end-practice-group -->

<!-- practice-group: 二、带宽账本 -->
<!-- quick-review: 答题提示 先写字节数，再写时间，最后写单位。 -->

### 5 choice

**Q:** Q5. 选择：有效带宽最接近哪种定义？

- [ ] A. 所有 transaction 字节 / kernel 时间
- [ ] B. 有用读写字节 / kernel 时间
- [ ] C. 线程数 / kernel 时间

Answer: B. 有用读写字节 / kernel 时间
Explain: 有效带宽只看算法真正需要搬的字节。

### 6 true/false

**Q:** Q6. 判断：有效带宽接近峰值，通常说明访问已经很整齐。

- [ ] 对
- [ ] 错


### 7 fill

**Q:** Q7. 填空：一个 float32 向量加法每个元素通常有 `____` bytes 的有用流量。


### 8 diagnosis

**Q:** Q8. 诊断：一个向量加法跑得慢，但它的访问是连续的。你先怀疑什么？

Explain: 如果访问已经连续，优先怀疑它本来就是带宽密集算子，或者 launch 太碎、并行度不够、同步太多。答案要能把“连续访问”与“带宽上限、launch 碎片、并行度不足”分开。

### 9 choice

**Q:** Q9. 选择：shared memory 最值得放什么样的数据？

- [ ] A. 只读一次的数据
- [ ] B. block 内会被多次使用的 tile
- [ ] C. 永远不会重复的临时值

Answer: B. block 内会被多次使用的 tile
Explain: shared memory 的价值来自块内复用。

### 10 true/false

**Q:** Q10. 判断：如果数据只用一次，搬进 shared memory 通常一定更快。

- [ ] 对
- [ ] 错


### 11 fill

**Q:** Q11. 填空：把数据搬近的前提之一，是同一份数据会被 block 内 `____` 次使用。


### 12 diagnosis

**Q:** Q12. 诊断：shared memory 之后反而慢了。你会先检查哪两件事？

Explain: 先查数据是否只用一次，或者复用不足以覆盖 copy 代价；再查 block 内同步、bank conflict、occupancy 是否把收益吃掉。答案不需要展开所有硬件细节，但要优先怀疑“复用不足”，别默认 shared memory 一定有收益。

<!-- end-practice-group -->

<!-- practice-group: 四、综合判断和下一步 -->
<!-- quick-review: 答题提示 先说症状，再说原因，最后说下一步。 -->

### 13 synthesis

**Q:** Q13. 用 150 字以内解释为什么“慢”不等于“算力不够”。

Explain: 优秀答案要说出：同样的算术，访问形状不同会让 transaction 数不同；有用字节更多时，有效带宽会更快被吃满；数据复用不足时，即使算术很少也可能慢；kernel 太碎时，启动和同步也会放大耗时。答题的重点是把“慢”拆成多个可能来源。

### 14 diagnosis

**Q:** Q14. 看到一个 kernel 有效带宽只有峰值的 20%，你前两步会做什么？

Explain: 先查地址是否连续、对齐、跨步太大，再查是否有足够的数据复用值得搬近。这个顺序很重要：先确认访问形状问题，再决定是否值得引入 shared memory 或做更大的结构改动。
