GPU 性能工程｜第 04 单元

测量、Roofline 与瓶颈定位

这页把一个 kernel 的时间、工作量和搬运字节放到同一张账上，先判断它更像带宽受限还是计算受限，再把“慢”压成一个能复测的假设。它接着上一页的访存形状判断，往下一页的 Triton 块级优化走；这里先不展开更细的计数器解释或微架构细节。

课程地图上一单元下一单元

看完先能做这三件事

  • 能写出一个最小但可信的 kernel 计时口径，知道为什么要同步、预热、重复运行和固定输入形状。
  • 能用 FLOP、字节数、时间和算术强度算出一个算子的上限，并判断它更接近带宽墙还是计算墙。
  • 能把 profiler 里看到的现象翻成下一步动作：先查数据是否搬得碎，再查复用是否足够，最后回到端到端复测。

这页要练的三件事

measurement
把时间测准

能区分 CPU 侧 launch 时间、GPU kernel 时间和端到端时间，知道何时需要 CUDA event、同步、warmup 和重复运行。

把上限算清

能从 FLOP、字节数和时间算出实际吞吐、有效带宽、算术强度和 Roofline 上限，并保留单位。

把指标变成动作

能把 Nsight Systems 的时间线和 Nsight Compute 的单 kernel 指标，写成下一轮可验证的优化假设。

先放一边的细节 先不展开具体 GPU 的所有硬件 counter、tensor core 指令编码、cache slice、bank conflict 和厂商库内部实现。这里先把测量链路学稳，后面的 Triton 和注意力优化会继续使用这套判断语言。

先把“慢”拆成几种可验证的情况

overview

先拿一个固定例子

这页沿用上一节的两个例子：FP32 向量加法和一个 FP16 矩阵乘。向量加法每个元素读两个输入、写一个输出，FLOP 很少，适合观察带宽限制。矩阵乘会反复使用 tile，FLOP 很多，适合观察计算上限和实际实现差距。

同一个计时结果，只有放进工作量和字节账本里才有意义。一个 kernel 跑 1 ms 可能很好，也可能很差；要看它应该搬多少字节、应该做多少计算，以及它离机器的可持续上限有多远。

一条顺手的排查顺序

推荐顺序是：先测端到端时间线，确认热点在哪；再给热点写 microbenchmark，测 kernel 自身；接着算 FLOP/byte 和 Roofline；最后回到 profiler 指标，判断下一轮实验应该改访存、并行度、计算路径还是 launch 粒度。

microbenchmark热点本身是否稳定、可复现CUDA event、固定 shape、warmup、重复运行可信 kernel 时间分布
工作量账本它理论上做了多少计算、搬了多少数据FLOP、读写字节、dtype、shape实际吞吐和算术强度

指标确认差距来自哪里Nsight Compute、stall、occupancy、memory throughput下一轮可验证假设

先把计时测准

timing

CPU 计时为什么容易误判

CUDA kernel launch 对 CPU 通常是异步的。CPU 侧用墙钟包住一次 launch，如果没有同步，测到的可能停在提交工作这一层，GPU 真正执行 kernel 的时间还没有纳入。端到端分析可以看墙钟，但 kernel microbenchmark 应该把异步边界写清楚。

一个可信计时至少要说明：输入 shape、dtype、warmup 次数、重复次数、同步位置、统计口径，以及是否包含 host-device copy。没有这些信息，优化前后的百分比很容易是噪声。

计时口径分开记

端到端时间用于判断用户是否真的变快；kernel 时间用于判断局部实现是否变快；profiler 时间用于解释为什么快或慢。不要把三个口径混成一句“快了 30%”。

CUDA event 计时骨架
# PyTorch sketch: measure GPU work, not only CPU launch overhead.
for _ in range(warmup):
    kernel(x, y, out)
torch.cuda.synchronize()

start = torch.cuda.Event(enable_timing=True)
end = torch.cuda.Event(enable_timing=True)
times_ms = []

for _ in range(iters):
    start.record()
    kernel(x, y, out)
    end.record()
    torch.cuda.synchronize()
    times_ms.append(start.elapsed_time(end))

print({

    "dtype": str(x.dtype),

Roofline 怎么算

roofline

先把三个量写清

Roofline 的横轴是算术强度，也就是每搬 1 byte 数据能做多少 FLOP。纵轴是性能，通常写成 FLOP/s。给定机器峰值计算性能和可持续内存带宽，某个算子的理论上限可以先用一个小公式估出来。

Roofline 模型用来给判断划边界，精度来自后续 profiler 复查。它的价值是把“我觉得慢”改写成“这个点在带宽墙下面、计算墙下面，还是远离两个屋顶”。远离屋顶时，下一步才去看 occupancy、stall、tensor core、访存布局、同步和 launch 粒度。

ridge point 只负责分界

ridge point = 峰值计算 / 可持续带宽。AI 低于它时，带宽更可能先成为上限；AI 高于它时，计算路径更值得检查。它只给方向，不替代 profiler 和真实实验。

$$ AI = \frac{FLOPs}{Bytes},\quad P_{roof} = \min(P_{peak},\; BW_{sustained} \times AI),\quad P_{actual} = \frac{FLOPs}{Time} $$

AI 是算术强度；P_peak 是目标硬件的峰值计算性能；BW_sustained 是可持续带宽；P_actual 是实测吞吐。算完以后要保留单位，例如 FLOP/byte、TFLOP/s、TB/s。

向量加法读 A、读 B、写 C

每个 FP32 元素约 2 FLOP、12 byte，有效优化通常先看访存连续性、带宽和 launch 粒度。

矩阵乘同一 tile 被多次复用

AI 可以很高，Roofline 往往指向计算墙；实际仍要确认 tensor core、tile 形状和数据布局。

远低于屋顶AI 判断不够

如果实测点离理论上限很远，下一步看 profiler：stall、occupancy、寄存器压力、分支发散、访存事务是否异常。

读 profiler

profiler

先看时间线，再看单 kernel

端到端慢时，先看时间线。你要找的是 kernel 是否真的占主要时间，还是被 host 逻辑、同步、CPU 调度、memcpy、launch 空洞或小 kernel 串行化吃掉了。一个 kernel 局部快了 30%，如果它只占端到端 3%，整体收益会很小。

每轮只改一个变量

workflow

把实验写成一条记录

性能排查最怕同时改 shape、block size、数据布局和测量口径。这样即使变快，也不知道是哪一步起作用。每一轮都应该写清：当前症状、证据、假设、最小改动、预期指标变化、回退条件。

例如“vector add 慢”太粗；“有效带宽只有目标 GPU 可持续带宽的 20%，且访存地址沿 threadIdx.x 跨步，先改线程映射让地址连续”更接近可执行假设。

端到端复测别省

局部 kernel 变快只是中间结果。真正交付前，要回到实际输入分布、batch、并发和完整模型路径里复测。性能工程最终服务真实工作负载，单点最高分只提供线索。

完整案例：一个 fused softmax 为什么局部变快，端到端不动

假设你在 attention 里替换了一个 fused softmax kernel。microbenchmark 里，单个 kernel 从 42 us 降到 28 us，看起来很漂亮；但真实推理请求的 p95 延迟只从 31.2 ms 到 31.0 ms。这时先暂停下结论，把读数放到同一张账本里：端到端里这个 kernel 占比是多少，前后是否多了 layout transform，是否因为 batch 和 sequence length 分布不同，导致 microbenchmark 的 shape 偏离主流 shape。

一个常见错误是只保留最快的 microbenchmark 结果。比如你用固定 B=1, H=32, S=2048 测了 1000 次，看到中位数下降，于是提交优化。但线上实际请求可能有大量 S=256 到 S=768 的短序列，softmax kernel 本身只占 4% 到 7%，真正的空洞在 launch、mask 构造和 host/device 同步之间。此时单 kernel 快 30%，端到端也只会动一点点。正确动作是回到 Nsight Systems：看时间线中 softmax 前后的 kernel、memcpy、同步点和 CPU 提交间隔，再看这个 kernel 在真实请求路径里到底占多少。

复测时可以按三层记录：第一层是单 kernel，用同一组输入、同一 stream、同一计时口径比较前后；第二层是局部子图，把 softmax 前后的 mask、scale、dropout 或 layout transform 一起纳入；第三层是端到端，把真实请求分桶后分别看 p50、p95 和 GPU 时间线。三层结果如果方向一致，优化判断就比较稳；如果第一层变快、第二层变慢，说明融合边界或布局转换吃掉收益；如果第二层变快、第三层没动，说明真实热点在别处，下一轮要重新排优先级。

记录里还要写清回退条件。比如“长序列收益超过 5%，短序列无退化，端到端 p95 至少下降 1%”可以作为继续推进的条件；如果短序列因为额外 mask 处理变慢，就要保留旧路径或按 shape 分派。这个条件让后续复测有明确裁判，也能避免只挑好看的图。下一次实验从这条记录继续，避免重新猜；同时留下命令、截图、驱动和库版本，方便别人复现。

检查项可信读法容易误判的读法下一步
shape 分布按真实 batch、head、sequence length 分桶统计只选一个最大 shape 当代表至少测主流 shape、长尾 shape 和最坏 shape
端到端占比先算优化对象占总延迟比例只看 kernel 自己的加速比用 Amdahl 思路估算可见收益上限
时间线空洞看 kernel 前后是否有 host 等待和同步把所有空洞都归因于 GPU 算得慢先减少 launch、同步或无用拷贝
数据布局确认优化没有引入额外 transpose、contiguous 或 copy只看新 kernel 内部更快把 layout transform 单独计入账本

如果这张表填完后发现 softmax 在真实路径只占 5%，那么单 kernel 加速 33% 的端到端上限大约也只有 1.7%。这个结果仍有价值，它说明下一轮要找更大的热点，或者把多个小操作融合成一次时间线更紧凑的路径。如果表里显示 layout transform 吃掉了收益，就要把数据布局设计和调用边界一起改；如果表里显示短序列 launch 空洞多，就优先考虑 batching、fusion 或减少 host 同步。你在练习区做 Q14 时，应该能用这条链条回答“为什么 microbenchmark 好看但端到端不动”。

证据放一起

ledger

为什么要放在一起

计时回答“多久”，Roofline 回答“上限在哪”，profiler 回答“为什么慢”。如果三种证据不放在一张表里，常见结果是：时间看的是一个 shape，字节算的是另一个 shape，结论却拿去解释同一个 kernel。

这页的做法很直接：先把证据写成行，再把每行对应到下一步。这样后面换成 Triton 或注意力内核时，你会知道自己究竟是在改访问形状、改复用，还是只是改了测量口径。

本页的锚点

同一个低 AI 向量加法，贯穿计时、Roofline、profiler、排查和复测。它会先给你一个带宽型答案；同一个 fused softmax，会先给你一个 launch 型答案；同一个高 AI matmul tile，会把你带向计算路径。不要把这三个 case 混成一个“慢”。

Q7. 小算账：FLOP=2e9，Bytes=12e9，Time=2ms。

参考答案 / 评分要点

算术强度是 2e9 / 12e9，约 0.167 FLOP/Byte。实际吞吐是 2e9 / 0.002s，等于 1e12 FLOP/s，也就是 1000 GFLOP/s 或 1 TFLOP/s。如果持续带宽按 2 TB/s 估算，带宽屋顶约 0.167 * 2 TB/s = 0.333 TFLOP/s；实际点若高于这个粗算屋顶，就说明字节估计、带宽口径或单位需要复查。合格答案要同时写公式、单位和复查动作。
