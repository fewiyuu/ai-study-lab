先把 Triton 读成块级算子语言

这节课不从抽象语法开始，而是从一维向量加法和二维 tile 开始。你要看到的是：一个 program 负责一块连续数据，program id 定位块，offsets 生成地址，mask 保护边界，块级计算完成读写。

如果你已经会 CUDA，最大的变化把入口从线程索引抬到 tile 级对象。这样更容易把矩阵乘、Softmax、LayerNorm 这些算子读成同一种数据流。

核心直觉：从线程思维到块级思维

入口

CUDA 的典型起手式$global_index = blockIdx.x \times blockDim.x + threadIdx.x$线程先得到一个标量 index，再读写一个或少量元素。这种写法贴近硬件线程层级，适合精细控制，但你要显式处理很多线程细节。

Triton 的典型起手式$offsets = pid \times BLOCK_SIZE + [0,1,\dots,BLOCK_SIZE-1]$program 先得到一整块 index，再用块级张量 load、计算和 store。编译器会处理很多线程映射细节，你则把注意力放在块的大小、布局和边界上。

mask：边界、默认值与数值语义

语义与边界

加法类 load

向量加法里，越界元素不应该参与结果。因为 store 也被 mask 掉，other=0.0 主要是为了让块级表达式完整。

$\begin{aligned} x &= load(mask, other=0) \\ y &= load(mask, other=0) \\ \text{store}(x+y, mask) \end{aligned}$单位元 0 不改变加法结果。

最大值归约

如果要做 max，越界位置不能填 0，因为真实数据可能全为负数。常见做法是填 -inf，让越界位置永远不会成为最大值。

$x = load(mask, other=-\infty), \quad m = max(x, axis=0)$填充值要是不会抢走最大值的哨兵。

操作越界位置常用 other理由

加法后 masked store0.0无效位置不会写回，填 0 让表达式可计算。

sum 归约0.00 是加法单位元，不改变总和。

max 归约-inf不会把越界位置错误地选为最大值。

min 归约+inf不会把越界位置错误地选为最小值。

常见坏例子：

对一行负数做 max 时，越界位置填

0.0

。如果真实值都是负数，最后结果会被越界位置的 0 污染。这语义错误。

你现在应该能做什么：为每个 tl.load 说清楚：mask 为假时是否会参与后续计算；如果会，other 应该是什么单位元或哨兵值。

绿色是当前 program 的有效 tile。黄色是这个 tile 中坐标落在矩阵外的元素，必须被 mask。

$ptrs[i,j] = base + offs_m[i] \times stride_m + offs_n[j] \times stride_n$offs_m[:, None] 把行 offsets 变成列形状，offs_n[None, :] 把列 offsets 放在行方向；两者相加会广播成 BLOCK_M x BLOCK_N 的地址矩阵。

为什么要显式写 stride？

真实深度学习张量可能经过转置、切片或布局变换，逻辑上是二维矩阵，物理内存却不一定按紧密连续行存放。stride 让 kernel 按真实内存布局寻址。

你现在应该能做什么：读懂二维 Triton kernel 里的 offs_m[:, None]、offs_n[None, :] 和二维 mask，并能判断 tile 是否碰到右边界或下边界。

性能参数：先提出假设，再自动调参

BLOCK_SIZE

Triton 降低了写 kernel 的门槛，但不会自动替你知道最佳块大小。块太小，program 太多、每个 program 的工作太少；块太大，寄存器压力、共享资源和并行度可能变差。性能工程的做法是：先根据瓶颈提出小范围候选，再测量。

num_warps / num_stages决定并行展开和流水加载强度。更多 warp 可能提高并行计算能力，也可能增加调度和资源压力。

现象第一轮怀疑可以尝试

带宽远低于预期，load/store 不连续offsets 或 stride 导致非合并访存检查数据布局，调整 tile 方向，让相邻元素尽量连续访问。

小 shape 很慢kernel 启动开销或 program 工作量过少考虑融合算子，增大每个 program 工作量，或让上层避免过碎调用。

块变大后反而慢寄存器压力上升，占用率下降缩小块，减少中间张量，查看编译报告或 profiler 指标。

最后一块浪费明显shape 与块大小不匹配选择更合适的块大小，或对常见 shape 专门调参。

常见错误与修正

调试优先级

错误 1：把 program id 当 thread id表现：用 pid 只处理一个元素，导致并行度和表达方式都不对。修正：用 tl.arange 生成块内多个 offsets。

错误 3：二维 tile 忘记 stride表现：逻辑矩阵和物理地址错位。修正：总是写出 stride_m 和 stride_n，不要默认紧密连续。

错误优先级

先排边界和语义，再排性能。比起“慢一点”，越界、错填充值、读写错位这些问题更会让 kernel 不可用。

你现在应该能做什么：拿到一个 Triton kernel 后，先检查 pid、offsets、mask、stride 和调参假设有没有对上，而不是立刻改 BLOCK_SIZE。

尾块链路：从 offsets 到语义

把 n=20、BLOCK_SIZE=8 的例子拆开看，前三个 program 都在同一件事上工作：先用 pid 选块，再用 offsets 生成全局位置，最后用 mask 决定哪些位置真的参与读写。最后一个 program 之所以容易出错，因为它唯一会把越界 offsets 暴露出来。

这也是为什么边界问题不能只靠“看起来结果差不多”来蒙过去。对加法、sum、max、min 而言，越界位置要么必须是单位元，要么必须是哨兵值；一旦 other 选错，块级表达式会把错误值当成合法数据，后面的 store 再正确也救不回来。

如果你把尾块理解成一个可推理对象，而不是一段“顺手处理一下的特殊代码”，你就会自然地先列出 offsets、有效元素、mask、other 和 store 条件，再去看性能。这样排查，bug 会落在边界、语义或布局中的某一类，而不是散成一团。

症状先查什么最小修法

最后一块数值飘load 和 store 是否共用同一张 mask把边界判断放在 offsets 之后，所有访问都走同一个 mask。

max 被 0 污染other 是否误填成 0.0max 改成 -inf，min 改成 +inf。

调大块后仍然错位是不是 layout 或 stride 写错先核对 stride，再调块；不要用更大的块掩盖地址错误。

尾块和前面块结果不一致是不是把边界判断分散到多个地方把同一套 mask 复用到 load、计算和 store。

块大小的取舍不是单调的

很多人第一次写 Triton 时，会下意识把 BLOCK_SIZE 往上加，想用更大的块把循环次数压下去。这个直觉只对了一半：块变大确实能让单个 program 做更多工作，但它也会把寄存器压力、边界浪费和调度成本一起抬高。对于形状比较碎的输入，越大的块不一定越好，反而可能让最后一块空洞更多。

一个更稳的想法是把候选块大小和输入 shape 对齐。比如 n=192 时，BLOCK_SIZE=64 会得到 3 个整块，BLOCK_SIZE=128 会得到 2 个 program 但尾块工作更重，BLOCK_SIZE=256 则会让一半以上的地址都变成 mask。你真正要验证的“谁让有效元素比例和资源占用最平衡”。

候选看到的现象先查什么

BLOCK_SIZE=64program 数量偏多，但尾块浪费低看调度开销是否能接受。

BLOCK_SIZE=128块数减少，单块工作量增加看寄存器和局部变量是否开始顶满。

BLOCK_SIZE=256很多 offsets 只是在 mask 下空跑看边界浪费是否已经盖过并行收益。

这就是块级思维的第二层：继续问“这组地址在硬件上值不值”。一旦你能把边界浪费、语义单位元和调参假设连成一条线，Triton 代码就不再只是短，而是可解释、可排查、可比较。

最后再补一条实践判断：如果你在 profile 里看到小 shape 明显吃亏，不要只盯着“把块改大”。先问是不是函数调用本身太碎、是不是最后一块被 mask 掉太多、是不是重复 launch 把开销摊平不了。换句话说，块大小只是一个旋钮，不是一个答案。

还有一个经常被忽略的细节：当你同时改 block size 和 num warps 时，不要把它们混成一个“更大更快”的判断。num warps 改的是并行展开和资源调度，block size 改的是每个 program 的工作集，两者会一起影响寄存器、occupancy 和边界浪费。真正靠谱的做法，是一次只改一个旋钮，把观察到的变化和假设对齐，再决定要不要继续扩候选空间。

如果只剩一个收尾句，那就是：Triton 把“先算地址、再定边界、最后看资源”的顺序讲得更清楚。顺序讲清楚了，性能判断才不会从一开始就乱掉。
