术语与误区

terms

术语直觉解释精确定义常见误解

容易混淆：FLOPs 和吞吐

FLOPs 是计算量，吞吐是单位时间完成多少工作。一个算子 FLOPs 多，不代表吞吐高；只有当硬件计算单元被持续喂饱，吞吐才高。

容易混淆：带宽和延迟

带宽像单位时间能搬多少货，延迟像第一件货到达前要等多久。大批量连续访问更看带宽，小而散的访问更怕延迟和合并失败。

你现在应该能做什么：把“算术强度、tile、归约、mask、融合”五个词分别放到最正确的上下文里。

三类性能画像

profiles

矩阵乘：复用换吞吐形状：C[M,N] = A[M,K] × B[K,N]压力：大量乘加，A/B tile 可复用，适合 Tensor Core 或向量化路径。优化抓手：BLOCK_M/N/K、数据布局、共享内存复用、warp 级调度、autotune。

Softmax：归约加稳定性形状：通常沿最后一维做 exp、sum、除法。压力：每个元素计算不多，但要读整行、求最大值、再求和并写回。优化抓手：行级并行、一次读入尽量完成多步、避免中间张量落 HBM。

LayerNorm：统计量和仿射形状：对每个 token 或样本的隐藏维 H 做均值和方差。压力：读 x，算 mean/var，归一化，再乘 gamma 加 beta。优化抓手：一行一个 program、两阶段归约、向量化 load/store、融合残差或 dropout。

矩阵乘：把复用做实

compute

$C[i,j] = \sum_{k=0}^{K-1} A[i,k] \times B[k,j]$FLOPs ≈ 2 × M × N × K。矩阵乘的核心A/B tile 会被多个输出复用。

在 Triton 或 CUDA 的块级实现里，一个 program 通常负责 C 的一个 tile，例如 16×16、32×64 或 128×128。它沿 K 维循环加载 A tile 和 B tile，累加到寄存器或局部 accumulator，最后写回 C。

Softmax：稳定归约和少搬中间结果

bandwidth

$m = \max(x), \quad y_j = \frac{\exp(x_j - m)}{\sum_t \exp(x_t - m)}$减去最大值防止指数溢出的数值稳定条件。

Softmax 不是逐元素独立函数，因为每个输出都依赖同一行的最大值和指数和。GPU 上的优化重点是尽量把一行读入后完成 max、exp、sum 和除法，避免把中间 exp 结果写回 HBM 再读回来。

LayerNorm：同一行上的两次统计量

reduction
$$ \mu = \operatorname{mean}(x), \quad \sigma^2 = \operatorname{mean}((x-\mu)^2), \quad y = \frac{x-\mu}{\sqrt{\sigma^2 + \epsilon}} \times \gamma + \beta$$LayerNorm 沿隐藏维 H 统计均值和方差，然后做归一化、缩放和偏移。 LayerNorm 的计算量相对不大，但一次输出依赖 mean 和 variance 两个统计量。朴素实现可能读 x 多次、写中间结果，再读回来做仿射；融合实现会尽量在一个 kernel 内完成统计和写回。 ## box info ### 判断句 LayerNorm 慢时，先问隐藏维是否适合一行一个 program、是否重复读 x、gamma/beta 是否连续、是否能与残差加法或 dropout 融合。 layernorm pseudocode复制 ```python row = program_id(0) offs = arange(0, BLOCK_SIZE) mask = offs < H x = load(X + row * stride_x + offs, mask=mask, other=0.0) mean = sum(x, axis=0) / H centered = where(mask, x - mean, 0.0) var = sum(centered * centered, axis=0) / H rstd = rsqrt(var + eps) gamma = load(Gamma + offs, mask=mask, other=0.0) beta = load(Beta + offs, mask=mask, other=0.0) y = centered * rstd * gamma + beta store(Y + row * stride_y + offs, y, mask=mask) ``` 算子依赖模式典型瓶颈第一优化方向 矩阵乘每个 C 元素依赖 K 维内积，A/B tile 可被多个输出复用。算力、tile 复用、布局。提高 tile 复用和 Tensor Core 利用率。 Softmax每个输出依赖整行最大值和总和。读写次数、数值稳定。避免中间矩阵落 HBM。 LayerNorm每个输出依赖同一行的均值和方差。统计量、重复读写。融合统计和仿射。 ## box warn 边界提醒 本页只覆盖前向性能直觉。反向传播会引入更多归约和中间量，但不会改变这三种算子的基本画像：矩阵乘偏复用，Softmax 偏带宽，LayerNorm 偏统计和融合。 **你现在应该能做什么：**在三种算子之间建立一个可复用的性能判断框架，而不是把它们看成三个互不相关的函数。 $$

对比与常见错误

compare
算子依赖模式典型瓶颈第一优化方向
矩阵乘每个 C 元素依赖 K 维内积，A/B tile 可被多个输出复用。tile 复用与计算吞吐。提高块复用与布局匹配。
Softmax每个元素依赖整行最大值和总和。归约、稳定性、中间结果写回。尽量一行内完成所有步骤。
LayerNorm每个元素依赖同一行的均值和方差。重复读写与统计量。融合统计和仿射。

错误 1：只看 FLOPs

FLOPs 高不等于快。矩阵乘通常更像计算密集，Softmax 和 LayerNorm 更容易卡在读写和归约。

错误 2：mask 只是防越界

在 Softmax 和 LayerNorm 里，mask 还会影响填充值语义。填错 other 会把归约结果改坏。

错误 3：把融合当万能答案

融合可能节省 HBM 往返，也可能因为寄存器压力过大而变慢。

错误 4：忽略布局

stride、转置和切片会改变真实地址。只按逻辑形状写 kernel 很容易错位。

你现在应该能做什么：把常见错误和正确的第一检查项一一对应起来，而不是只记“要融合”“要调块”。

算子账本

diagnostic

从输入形状开始看

矩阵乘、Softmax 和 LayerNorm 放在一起学，因为它们会把性能压力落在不同地方。矩阵乘的核心是 tile 复用，Softmax 的核心是整行依赖和稳定归约，LayerNorm 的核心是同一行的统计量和仿射变换。

当你只看“算子名字”时，很容易把三者都说成“GPU kernel”。但在性能工程里，名字不够，依赖模式才够。一个看起来简单的逐元素公式，可能因为行内归约变成带宽型；一个看起来计算很多的矩阵乘，可能因为 tile 复用极强而更接近计算密集。

第一优化方向怎么选

矩阵乘先看数据布局和块复用；Softmax 先看能不能把一行的 max、exp、sum、normalize 连着做完；LayerNorm 先看能不能把 mean、variance、scale、shift 关在一个 kernel 里，并减少重复读写。

这三种判断把“依赖在哪里”翻译成“数据应该在哪里停留”。只要数据停留位置不对，FLOPs 再漂亮也未必快。

算子先看什么第一步怎么做典型误区

矩阵乘A/B tile 复用和 K 维循环对齐布局、扩大复用、控制块大小只盯 GEMM 名字，不看数据摆放

Softmax整行最大值、指数和、归一化一行内完成稳定归约，减少中间写回把它误看成逐元素函数

LayerNorm均值、方差、gamma/beta 的读写按 token 行组织统计和仿射，尽量融合只看归一化，不看重复读写

一个可落地的判断顺序

假设你现在拿到一个小型推理模型，里面有一个 4096 维 hidden size 的 LayerNorm、一个中等长度的 Softmax、和一个矩阵乘。不要先问“哪个算子最酷”，先问“哪一个在搬运不必要的数据”。

最实用的顺序通常是：先看矩阵乘有没有稳定吃满复用，再看 Softmax 有没有把中间结果写回 HBM，最后看 LayerNorm 有没有重复读 x、gamma 和 beta。如果 LayerNorm 的瓶颈只是小于你的 profile 噪声，那就不要把它当主要优化目标；如果 Softmax 的中间张量被拆成多个 kernel，才值得优先收回来。

先看证据如果错了会怎样

矩阵乘tile 复用、布局和 Tensor Core 是否稳定算术强度被浪费，A/B tile 没有吃满复用。

Softmax是否有中间矩阵写回、是否先减 maxHBM 流量上升，数值稳定性也会出问题。

LayerNormx、gamma、beta 是否一次性处理重复读写把本来不重的算子拖慢。

diagnosis sketch复制

if profile == "slow":
 if op == "softmax":
 root = "too many HBM round-trips"
 elif op == "layernorm":
 root = "repeated reads of x/gamma/beta"
 else:
 root = "tile reuse or layout mismatch"

这段判断还有一个实际好处：它会逼你把“慢”拆成可归因的对象。很多时候真正需要修的把同一个算子拆成了过多 kernel，或者把本来可以在一行里完成的统计，拆成了多次读写。只要这种拆分还在，换更大的 GPU 也只是把问题往后推。

这个顺序的价值在于，它强迫你把“慢”拆成一个具体对象，而不是一个情绪。你能指出数据在哪里停留、什么时候落回 HBM、哪一步没有单位元语义，这才算真正开始做算子性能分析。

如果你还想继续往下挖，可以把它们和前面的表格再对一遍：矩阵乘的第一问题是布局和复用，Softmax 的第一问题是稳定归约和中间写回，LayerNorm 的第一问题是重复读写和统计量的组织。这个顺序一旦固定，很多“该不该融合”“该不该换块”的争论就能转成事实讨论，而不是口号讨论。

实践里最容易犯的错，是把“哪个算子更重要”问成抽象问题。真正要问的是：如果我现在只改一件事，它会不会直接减少 HBM 往返、减少重复读、或者让归约结果更稳定。这个问法一旦落地，你就会自然地把优化目标限定在一两个可测的方向里，而不是把所有算子都一起乱动。

把这套顺序再往前推一步，你还会发现一个更实在的判断：有些“优化”其实只是把一个慢问题改成另一个慢问题。比如把 Softmax 拆成多个 kernel，只是把中间写回切成了几段；把 LayerNorm 拆成多次统计，只是把本来能共享的数据读了好几遍。只要你能在表格里指出这一点，后面的改动就会更克制。

如果只给一个模型打补丁，你也可以先问最小改动是否已经足够：矩阵乘只要把布局对齐就能明显变快，Softmax 只要把中间值留在行内就能减少读写，LayerNorm 只要把统计和仿射放在同一个 kernel 里就能少一轮往返。这个判断比“改大块”更具体，也更容易在 profile 里验证。

真正落地时，最有效的动作通常把已经有的想法按顺序执行：先定位谁在重复搬运数据，再核对是不是在错误的地方做了拆分，最后才考虑融合、调块或换实现。这个顺序很朴素，但它会让你少走很多把复杂问题误判成“再试一次就好”的弯路。

你也可以把这条顺序写成一条简短的评审意见：先查数据路径，再查归约语义，最后查是否真的需要拆 kernel。只要这句话能写出来，页面里的三个算子就不只是名词，而是三种可验证的判断方式。

边界提醒

本页只覆盖前向性能直觉。反向传播会引入更多归约和中间量，但不会改变这三种算子的基本画像：矩阵乘偏复用，Softmax 偏带宽，LayerNorm 偏统计和融合。

你现在应该能做什么：拿到一个新算子时，先把它归到“复用、归约、统计、仿射”中的一类，再决定第一优化方向。

工程例子

场景：长序列 prefill 的注意力 kernel

假设一个 batch 中有若干长度为 4096 的序列，每个头维度为 64。朴素路径会生成 4096×4096 的分数矩阵和概率矩阵。以 FP16 粗略估算，一个矩阵约 32 MiB；每个头如果写 S 和 P，再读回来做后续步骤，HBM 流量会迅速放大。

分块路径不保存完整 S/P，而是让每个 program 负责若干 query 行，循环读取 K/V 块，维护 m、l 和输出累计值。最后只把 O 写回 HBM。

状态最重要

m 保护数值稳定，l 保存分母，acc 保存加权 V 的未归一化累计值。每个新块到来时，旧状态必须按新的最大值重标定。

你现在应该能做什么：从一段注意力伪代码里指出哪些状态必须保留，为什么不能直接拼接局部 softmax 的结果。
