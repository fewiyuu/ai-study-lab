核心讲解

8

1从注意力公式看优化对象

缩放点积注意力通常写作：

$O = \operatorname{softmax}(QK^T / \sqrt{d})V$Q、K、V 的形状可以简化成 [N, d]。朴素实现会先算出 S = QK^T，再对 S 的每一行做 softmax 得到 P，最后计算 O = PV。

真正需要留到最后的是 O。S 和 P 只是通往 O 的中间状态。如果每次都把 S、P 写入 HBM，再读回来继续计算，就会把大量时间花在搬运暂时数据上。

3分块不是近似，而是改变数据停留位置

分块做注意力时，可以把 Q 的一小块和 K/V 的一小块搬到片上资源，计算一个局部分数块。关键是不能只对局部块做 softmax，因为完整 softmax 的分母来自整行所有 key。

解决办法是在线 softmax。对每一行维护两个状态：当前最大值 m 和归一化因子 l。当新块分数 x 到来，先更新最大值 m_new = max(m_old, max(x))，再把旧累计值按新的最大值重标定，同时加入新块贡献。

$m_{new} = \max(m_{old}, \max(x)), \quad l_{new} = l_{old} \cdot \exp(m_{old} - m_{new}) + \sum \exp(x - m_{new})$输出累计值也需要同样重标定。这样每次只处理一块，但最终等价于对完整行做数值稳定 softmax。

5Triton 视角：块级张量让算法结构更直接

Triton Tutorials 的价值在于，它让你用 program id、块级 load、mask 和 tl.dot 表达 tile 算法，而不必从每个 CUDA thread 的索引开始写。注意力优化很适合这种模型：一个 program 负责一块 query 行，循环扫过 K/V 块，维护在线 softmax 状态。

但 Triton 不是自动性能魔法。BLOCK_M、BLOCK_N、num_warps、num_stages、数据布局和 mask 处理都要结合硬件测量。Triton 让实现更短，不代表可以跳过性能账本。

7训练和推理的边界不一样

训练时要考虑反向传播，需要保存或重算足够的信息；FlashAttention 的思路通常会用重算换取更低显存占用。推理时则分 prefill 和 decode：prefill 类似长序列注意力，decode 常常是单步 query 访问已有 KV cache。

因此，注意力优化不是一个单一答案。训练关注激活显存和反向成本；prefill 关注长序列吞吐；decode 关注 KV cache 布局、批量组织和访存模式。本页的分块 softmax 是理解这些分支的共同底座。

你现在应该能做什么：把注意力优化拆成“公式、IO、分块、数值、硬件、场景、边界”七个问题来问，而不是直接问“能不能写个更快 kernel”。

工程例子

场景：长序列 prefill 的注意力 kernel

假设一个 batch 中有若干长度为 4096 的序列，每个头维度为 64。朴素路径会生成 4096×4096 的分数矩阵和概率矩阵。以 FP16 粗略估算，一个矩阵约 32 MiB；每个头如果写 S 和 P，再读回来做后续步骤，HBM 流量会迅速放大。

分块路径不保存完整 S/P，而是让每个 program 负责若干 query 行，循环读取 K/V 块，维护 m、l 和输出累计值。最后只把 O 写回 HBM。

状态最重要

m 保护数值稳定，l 保存分母，acc 保存加权 V 的未归一化累计值。每个新块到来时，旧状态必须按新的最大值重标定。

你现在应该能做什么：从一段注意力伪代码里指出哪些状态必须保留，为什么不能直接拼接局部 softmax 的结果。

场景拆解

prefill

prefill 看大矩阵，decode 看单步请求

prefill 阶段像长序列矩阵计算：query 很多，K/V 块也很多，优化重点往往落在分块、复用和减少中间矩阵写回。decode 阶段更像单步请求：每次只生成少量 query，但要频繁访问 KV cache，瓶颈会更偏向访存组织、批处理和调度。

所以同一个 attention kernel，在 prefill 上可能要先看 tile、HBM 和在线 softmax；到了 decode 上，可能先看 KV cache 布局、batch 合并和 launch 开销。工程边界不是抽象概念，它会随着场景切换。

先看什么，后看什么

如果你拿到的是长序列 prefill，先问 S/P 是否会落 HBM，能不能把 softmax 状态保留在片上。

如果你拿到的是 decode，先问每步 query 的工作量是不是太小，KV cache 是否连续，batch 是否碎得过头。

场景主要压力第一判断先别急着做什么

prefill长序列、S/P 中间矩阵、HBM 往返能否用分块和在线 softmax 减少中间写回先别把问题归咎于 launch 次数

decode单步 query、KV cache 访问、batch 组织是否连续访问、是否过度碎片化先别只调大 tile

两者共同点都要保证数值稳定和边界正确mask、state、layout 是否对齐先别用局部最优结论推广全部场景

如果你把这件事讲给一个做推理部署的人听，可以直接落成一条排查顺序：先看是不是把 prefill 和 decode 混在同一个结论里，再看 tile 是否真的在省 HBM，最后再看是否已经碰到系统瓶颈。很多优化报告失败，因为它把两个场景说成了一个场景。

这个实验说明，工程边界场景变化之后你是不是还在拿旧假设做判断。只要你能把 prefill 和 decode 的失败模式分开，很多“同一个优化为什么前后不一致”的问题就会变得可解释。

换成更直接的说法：如果你已经把中间矩阵压到了最小、把在线 softmax 状态也保住了、但 decode 还是慢，那就该把注意力 kernel 之外的部分也纳入判断，比如 batch 是否太碎、KV cache 是否跨得太散、launch 是否在抢你的时间。工程边界不是退缩，而是知道问题已经不在同一层了。

最后的检查单可以很短：prefill 看矩阵和中间写回，decode 看 KV cache 和批处理，两个场景都要看数值稳定和边界 mask。只要你每次都能把当前场景、当前瓶颈和当前停止线写出来，注意力优化就不会变成一句“再调调 tile”就完事的空话。

再补一个最小判断：如果 decode 的慢主要来自 KV cache 的访问碎片，那把 tile 继续调大通常不会救它；如果 prefill 的慢主要来自中间矩阵写回，那把 batch 再拆细也不会救它。前者要看布局和 packing，后者要看分块和状态保留。把这两句分开记住，很多“同一个 kernel 为啥在两个阶段表现相反”的问题就不用猜了。

把判断落成一条检查线

如果你需要一个更机械的版本，可以直接按这四步走：第一步，把当前场景标出来，是 prefill 还是 decode；第二步，把中间矩阵和 KV cache 的去向写清楚，看它们是不是还在 HBM 上来回跑；第三步，确认在线 softmax 的状态有没有被保留住；第四步，明确停止线，看看问题是不是已经转到 batch 组织、调度或别的系统层。

这四步的好处是，它把“注意力优化”从一个很大的概念压成了一个很小的动作序列。你不需要每次都重新发明一套判断，只要按这条线检查，就能快速知道当前优化是在省什么、牺牲了什么、有没有越过工程边界。

如果你希望把它继续变成一个工作习惯，就在每次实验后补一句记录：当前是 prefill 还是 decode、观察到的瓶颈是什么、下一步准备改哪一层。这个记录很短，但它会逼你把“优化感受”变成“可回放的判断”，也更容易让别人复查你的结论。

常见错误

pitfalls

错误 1：只比较 FLOPs，不比较 HBM 流量

注意力优化的核心收益常常来自减少中间读写，而不是减少主要矩阵乘的计算量。只看 FLOPs 会误判瓶颈。

错误 2：局部 softmax 后直接拼接

每个块单独归一化再拼起来不是全局 softmax。必须维护全行的最大值和分母。

错误 3：忽略 mask 的边界

causal mask、padding mask、变长序列和越界 mask 会改变分数矩阵中哪些元素有效。mask 处理错，速度再快也不可用。

错误 4：用一个 shape 的结果推广所有场景

短序列、小 batch、不同 head_dim、不同精度和不同硬件可能给出完全不同的结论。优化报告必须说明形状边界。

错误 5：寄存器压力失控

过大的 tile 或过多融合会让每个线程使用太多寄存器，降低 occupancy，甚至发生寄存器溢出。

错误 6：只测 kernel，不测端到端

单 kernel 更快不代表请求吞吐更高。实际系统还会受到数据准备、调度、KV cache、同步和其他算子的影响。

你现在应该能做什么：把“快了但没用”的原因拆成形状、数值、资源和系统四类，而不是只盯 kernel 时间。

停止线

当你已经知道瓶颈是 KV cache、batch 组织或系统调度时，再继续缩 tile 往往不会解决根因。这个时候更该换的是路径，而不是继续堆 kernel 技巧。

decode 检查样例

假设你在一个小 batch 的 decode 服务里看到：Q 很小，KV cache 很散，launch 次数反而比矩阵乘时间更显眼。这个时候你真正要做的，把每个请求的 cache 访问整理得更连续。能不能做这件事，通常比能不能再省一点 FLOPs 更重要。

对应的检查动作也很简单：先看 cache 是否连在一起，再看 batch 是否拆得太碎，再看当前 tile 是否已经不再是瓶颈。你会发现，问题一旦落到 decode，很多优化目标都会从“更大块”变成“更好的 packing”。
