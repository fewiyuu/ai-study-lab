Unit 01｜边界与生命周期

Serving 边界与请求生命周期

本页只收一件事：先把 LLM Serving 的边界画清，再把一次请求拆成入口、排队、prefill、decode、返回和观测几个状态。后面的 OpenAI 兼容接口、Admission Control、路由、伸缩和 SLO 都会回到这条路径上。

课程地图下一单元

先确认的三件事

  • 画出一次请求从客户端到 token stream 的路径，并说出每层在读什么、写什么。
  • 区分 Serving 层和推理引擎层，知道哪些问题该在入口处理，哪些该交给 engine。
  • 看到 TTFT、TPOT、队列时间或 5xx 时，能先判断该查哪一层。

学习目标

Goals

先定义边界

能说清 Serving 的边界：它负责把模型能力封装成可运营的线上服务。

再追状态

能按接入、校验、路由、排队、prefill、decode、流式返回和完成记录描述一次请求。

最后先查哪层

看到 TTFT、TPOT、队列时间、429/503 或长尾卡顿时，先把症状放回生命周期，再决定是否改参数。

先抓主线

Overview

这门课从哪一层开始看

外部调用者看到的是 API 合同：请求怎么写、模型名怎么填、是否流式、超时时间多长、失败时返回什么。这个边界先于任何 GPU 调度出现。

进入服务层后，系统开始决定请求该不该接、该去哪个副本、是否要排队、是否要拒绝、是否要降级。只有到了引擎层，request 才会变成 sequence，开始处理 prompt、cache 和 token 生成。

边界句

Serving 层回答“请求怎么进来、去哪里、何时返回、何时记录”；推理引擎回答“已经进来的序列如何高效生成 token”。

后面的接口、路由、背压、伸缩、SLO 都是在这条边界上继续加决策。

请求合同

Contract

先看哪些字段

服务层接到请求后，最先读的是请求在当前系统里能不能安全接住的信号。model 决定路由候选，messages 或 prompt 决定 prefill 成本，stream 决定返回方式，max_tokens 决定 decode 上限，tenant 和 auth 决定是否进入可用范围。

这些字段放在一起看，才能解释为什么有些请求会在入口被拒绝，有些会进队列，有些会被送去更适合长上下文的副本。字段本身不复杂，关键是它们会一路改变后续状态。

这一层只解决什么

请求合同只负责把外部输入变成可执行的服务状态：合法、待排队、待路由、待降级或待拒绝。它还不负责生成 token，也不负责解释模型能力边界。

先把这一步看清，后面读 OpenAI 兼容接口、流式返回和错误码时，就知道哪些是协议细节，哪些是服务层决策。

字段 服务层先读什么 主要影响 错配后的症状
model 可用副本、版本、租户约束 路由候选和副本选择 请求被送去不匹配的副本，最后变成 4xx / 5xx
prompt / messages 上下文长度、历史规模、cache 命中 prefill 成本和 TTFT 短请求被长上下文拖慢，首 token 很晚才出现
stream 返回通道、连接寿命、客户端等待方式 响应节奏和超时边界 服务已经开始工作，客户端却先超时
max_tokens decode 上限、GPU 预算、队列压力 输出长度和 TPOT 请求本来能结束，结果因为上限过大而拖长
tenant / auth 隔离规则、配额、优先级 接纳、拒绝和公平性 该挡的请求没挡住，或者该保的租户被挤掉

request = parse_http_request()
validate(request.model, request.messages, request.stream, request.max_tokens)
target = route_to_model_replica(request.model, request.tenant, load, health)
enqueue(target, request)

核心机制

Mechanism
步骤 前态 后态 读什么 写什么 错了会怎样
入口校验 HTTP 请求和字段 合法或拒绝 鉴权、配额、模型名、上下文长度、当前负载 接纳结果、错误码、入口日志 把明显超限的请求推到后面，最后在引擎层 OOM
路由 待处理请求 绑定到某个副本 模型版本、租户、健康状态、队列长度、负载 副本选择、路由记录 把请求打到忙副本，TTFT 抬高
prefill sequence + prompt KV cache 可用 prompt 长度、prefix cache、上下文 token 上下文状态、cache block 首 token 变慢，TTFT 拉长
decode 已有上下文 逐步生成 token batch 混合、停止条件、输出长度、GPU 利用率 新 token、sequence 状态、统计指标 首 token 正常但后续 token 变慢，TPOT 变差
返回与观测 生成中的 token 客户端收到流式结果 网络状态、响应通道、错误和 usage 流式响应、usage、TTFT、TPOT、总时延 服务已经算完，但客户端看起来像卡住

这张表的作用很简单：先看对象状态在哪里变，再看哪一层最可能出问题。慢请求通常先拆成排队、prefill、decode 和返回四段，再去定位真正的瓶颈。

请求案例

Example

一个客服聊天请求怎么走

假设一个客服助手通过 OpenAI 兼容接口发起流式聊天请求：系统提示词较长，用户问题很短，服务有两个副本，后端使用高吞吐推理引擎。这个例子足够小，能看清路径，又不会把真实机制抹平。

请求先经过入口校验，再由服务层选择副本；副本内部把请求转成 sequence，先做 prefill 建立上下文，再做 decode 逐步生成 token，最后把结果流式返回，同时记录 usage 和时延。

看这个例子时要盯住什么

  • 入口还没放行前，GPU 不该背锅。
  • prefill 慢，多半先看 prompt 长度、prefix cache 和上下文压力。
  • decode 慢，常见问题是输出很长、batch 混合差、GPU 被别的请求挤占。
请求路径伪代码
request = parse_http_request()
validate(request.model, request.messages, request.stream, request.max_tokens)
target = route_to_model_replica(request.model, request.tenant, load, health)
enqueue(target, request)

sequence = target.start_sequence(request)
engine.prefill(sequence)  # prompt -> KV cache, TTFT 关键路径
while not sequence.finished():
    token = engine.decode_step(sequence)
    stream_to_client(token)

record_metrics(queue_time, ttft, tpot, total_latency, status)

完整走读

Walkthrough

把一条请求从头走到尾

1. 客户端发来请求，外层只知道这是一个 HTTP / OpenAI-compatible 入口。2. 服务层先做鉴权、配额、字段合法性和上下文长度检查。3. 如果请求合法，再按照模型名、租户、健康状态和负载去挑副本。

4. 副本拿到请求后才开始排队。5. 引擎把 request 转成 sequence，先做 prefill，把 prompt 编成可复用的上下文状态。6. 接下来进入 decode，逐步生成 token，期间会一直和 batch、GPU 利用率和停止条件打交道。7. token 经过流式连接回到客户端。8. 整个过程结束后，系统写入 usage、TTFT、TPOT、queue time 和 status。

这条走读的用处

它的作用是把症状归位。你以后看到“慢请求”，先问它卡在哪一步，再决定是该看排队、prefill、decode，还是客户端返回链路。

同样的顺序还能帮你分辨问题有没有被说混：队列变长、GPU 忙、stream 慢、客户端超时，这四个现象常常同时出现，但根因不一样。

两条请求怎么分叉

Trace

短请求和长请求，不一定卡在同一层

先看两个最常见的样子。A 是短 prompt、正常排队的客服问答；B 是长上下文、带系统提示词和历史对话的总结请求。两个请求都可能叫“慢”，但慢的层次不同。

A 如果 TTFT 高，常见原因是入口排队、路由等待或副本繁忙；B 如果 TTFT 高，常见原因是 prefill 更重、prefix cache 命中差或上下文长度把首次 token 推迟了。两者都慢，不代表同一个瓶颈。

先问三个问题

第一，先确认请求有没有真正进入引擎。第二，确认 prefill 有没有做完。第三，确认 decode 是否已经开始并变慢。把这三层顺序问清，很多“服务很慢”的描述就能分成可检查的症状。

如果 A 和 B 都慢，先怀疑队列、路由和副本数；如果 A 还好、B 明显慢，先怀疑 prompt 长度和 cache 命中；如果首 token 正常、后面 token 慢，就去看 decode 和 batch。把这三种分叉记住，后面看 SLO 就能先问对层。

请求样子 先观察什么 最可能卡在哪层 第一检查
短 prompt，TTFT 偏高 queue time、route wait、连接状态 入口或服务层 看排队、限流、健康副本是否不足
长 prompt，TTFT 偏高 上下文长度、prefix cache、prefill time prefill 看 prompt 是否太长、历史是否可裁剪
首 token 正常，后续变慢 输出长度、batch 混合、GPU 利用率 decode 看长输出、调度不均或资源是否被挤占

状态账本

Ledger

Serving 的每一层都在改状态

这页最重要的习惯是盯住“谁写了什么”。一个请求从入口到返回，会在多个地方留下状态：请求对象、队列位置、副本选择、sequence、KV cache、流式连接和指标记录。

只要你知道某个状态是在哪一步被写入的，就能反推它什么时候应该存在，什么时候应该失效，什么时候不该拿来排障。这个方法比单纯记流程图更稳定。

失效时最容易错的地方

如果请求在入口被拒绝，就不该再期待 sequence 或 KV cache；如果流式返回已经结束，连接和上下文都应该回收；如果副本切换了，旧副本上的缓存和队列状态不能继续当成当前事实。很多故障都发生在“状态还在被错误沿用”的时候。

对象 谁创建 谁读取 谁写入 / 失效 典型症状
请求对象 入口解析层 鉴权、路由、观测 在入口阶段被标记合法、拒绝或重试 字段不合法却进到后面，最后变成 5xx
队列位置 服务层 admission / router 调度器、健康检查、限流器 入队、出队、丢弃、降级 TTFT 长但引擎利用率不高，说明卡在排队
sequence 副本内部 prefill、decode、停止条件 状态随着 token 生成逐步变化 写成了 request 级别变量，调度后全乱
KV cache prefill 阶段 decode、后续 batch 被写入、复用、淘汰 prefill 很重但缓存没起作用，首 token 很慢
流式连接 返回链路 客户端、网关、超时控制 写入 token、关闭、重置 服务已经生成，但客户端看起来像没返回
指标记录 返回完成时 观测、容量规划、告警 写入 TTFT、TPOT、queue time、status 指标没分层，最后所有问题都被平均值吞掉

接纳与背压

Admission

为什么请求要分层接纳

Serving 最容易出问题的地方，常常出现在决定“先接谁、后接谁、要不要暂时不接”的那一步。高并发时，系统必须在接收、排队、拒绝和降级之间做选择。

这会直接影响 TTFT、错误率和资源利用率。队列是缓冲，也是风险；背压是保护系统的方法，也是让结果更可解释的控制点。

四个动作

接收：说明请求已经通过入口检查，并进入服务范围。排队：说明副本暂时没空，但还愿意等待。拒绝：说明当前负载下继续接收只会把问题放大。降级：说明系统选了一个更低成本或更低能力的返回方式，至少先保住可用性。

新请求HTTP 请求 / 外部合同

读：模型名、messages、stream、长度、当前负载。写：接纳检查结果。症状：如果这里没挡住超限请求，后面就会把问题推到引擎层。

等待执行服务层队列

读：副本健康、排队深度、租户配额。写：队列位置、等待时间。症状：TTFT 变长但 GPU 还没满，通常先看这里。

过载分支拒绝或降级

读：阈值、超时、重试压力。写：429 / 503 / 降级响应。症状：如果没有这个分支，系统会用无限等待把错误放大。

现象 优先动作 读什么 写什么
队列短、TTFT 稳 正常接收 副本健康、模型负载、上下文长度 接纳、路由、开始 prefill
队列增长但还能消化 适度排队 等待时间、峰值速率、重试情况 队列位置和等待时长
队列持续变长，TTFT 与错误率一起涨 拒绝或降级 排队深度、超时、重试放大 429 / 503、降级响应、告警

一个更具体的判断例子

假设现在有两条请求同时到达：一条是 120 字的短问答，一条是 8k token 的长总结。短问答排队 2 秒还能接受；长总结如果继续排队，很可能把后面所有短请求一起拖慢。这个时候，服务层不应该只盯着“还能不能进”，而要先问“进来之后，会不会把别的请求也挤坏”。

这就是 admission 的判断方式：先看当前负载是否已经逼近不可接受的等待区间，再看这个请求会不会放大等待。服务层在这里做的是判断系统还能不能保持可解释的延迟和错误率。

接纳策略伪代码
def admit(request, queue_depth, queue_time_p95, overload_threshold):
    if not request.auth_ok:
        return reject(401, "unauthorized")
    if request.prompt_tokens > request.max_context:
        return reject(400, "context too long")
    if queue_depth > overload_threshold:
        if request.is_long_running():
            return degrade("shorter_response")
        return reject(503, "busy")
    if queue_time_p95 > request.max_wait:
        return reject(429, "backpressure")
    return queue(request)

这段伪代码最重要的地方在顺序：先做硬门槛，再做负载判断，最后才把请求放入队列。顺序错了，拒绝会变得太晚，排队会变得太长，降级会被当成临时补丁。

你在练习里会用这段判断去回答三类题：哪一步该先看、什么情况应该拒绝、什么情况应该降级。等你能把这三类题答顺，背压就不再只是一个词，而是一个可执行的决策链。

这一步和后面的路由、prefill、decode 属于不同层。该拒绝的时候还继续排队，最后只会把短请求也拖慢。

路由与副本选择

Routing

路由是挑合适的副本

服务层要找的是对这个请求合适的副本。模型版本、租户、健康状态、上下文长度和当前排队情况，都会影响选择。

一个短问答和一个长总结，不该被当成同一类请求处理。前者更怕排队，后者更怕 prefill 压力；如果路由忽略了这些差别，负载均衡反而会把不平衡制造出来。

你要记住的判断

路由层读的是“谁还能接”，先看健康和负载，再看租户与模型版本，最后才决定要不要把请求送去某个副本。

所以，当你看到某个副本 TTFT 特别高，不要直接假设模型慢。它可能只是被塞进了太多长请求，或者它的 cache 命中更差。

请求特征 路由优先看什么 常见选择 如果路由错了
短问答 健康、当前排队、返回速度 接近空闲且健康的副本 TTFT 被不必要的排队抬高
长上下文总结 上下文容量、prefix cache、内存压力 更适合处理长 prompt 的副本 prefill 变重，首 token 迟迟不出
特定租户或版本 租户隔离、模型版本、健康约束 满足合同的指定副本 把请求送到不该去的副本，最后变成 4xx / 5xx
路由选择伪代码
def route_to_model_replica(model, tenant, queue_depth, health, prompt_tokens):
    candidates = [r for r in replicas if r.model == model and r.healthy]
    if tenant.is_pinned:
        candidates = [r for r in candidates if r.tenant == tenant]
    if prompt_tokens > LONG_CONTEXT_THRESHOLD:
        candidates = sorted(candidates, key=lambda r: (r.long_context_score, r.queue_depth))
    else:
        candidates = sorted(candidates, key=lambda r: (r.queue_depth, r.queue_time_p95))
    return candidates[0] if candidates else None

这段代码告诉你一件事：路由会同时读模型、租户、健康状态、上下文长度和等待情况，再决定把请求送到哪一个副本。

如果把路由简化成轮询，长请求很容易扎堆到同一个副本，TTFT 会持续变差；如果把路由简化成只看最空闲副本，健康副本和适配副本会被忽略，最后把错误变成更难排查的长尾。

你可以把这段判断接回练习题：当题目问“应该先看哪一层”时，如果问题表现为某个副本特别慢，就先从路由和副本合同开始，再决定是否继续看 decode。

故障实验室

Failure Lab

把症状切成三类

先看这个小实验。你不用真的跑服务，只要看输出，就能把“慢”拆成三种常见失败：入口排队过长、prefill 过重、decode 或连接返回变慢。每种失败都对应不同的下一步。

按钮切换的是同一条请求路径在不同状态下的日志。你要关注哪一层先露出问题，先看症状，再看日志。正常时 TTFT 不会让人等太久；长 prompt 时预填充会先变重；排队爆炸时引擎甚至还没开始做事。

观测顺序

先读输出，再读症状，再读下一步。不要反过来：先想修哪一层，再去找证据。这样很容易把本来只需要限流的情况，误修成引擎调参。

[case normal]
request accepted
route -> healthy replica
prefill -> 84 ms
decode -> steady
ttft -> 410 ms
next step: keep current routing, watch p95 and queue time

指标归因

Metrics

TTFT

看用户多久看到第一个 token。它会同时受入口排队、路由等待、prefill 和返回链路影响。

TPOT

看后续 token 的平均节奏。它更容易暴露 decode、batch 混合、GPU 利用率和长输出的压力。

Queue Time

看请求进入执行前等了多久。它先反映服务层决策，不等于引擎内核慢。

TTFT = 入口排队 + 路由等待 + prefill + 首次返回

这是一个拆分顺序：先把等待和计算分开，再决定该调队列、扩容、副本、cache 还是客户端超时。

症状 优先看哪层 下一步
TTFT 上升 入口、排队、prefill、返回 先拆 queue time，再看 prompt 长度和 prefix cache
TPOT 上升 decode、batch、GPU 看输出长度、batch 混合和副本负载
5xx 增多 入口或引擎错误 先分 4xx / 5xx，再看是请求问题还是系统问题

常见误区

Mistakes

误区 1：把所有 5xx 都归到模型质量

很多错误根本没走到模型输出，可能是鉴权、限流、队列、路由、超时或引擎 OOM。先看返回位置，再看模型内容。

误区 2：只看平均延迟

LLM 请求长度差异很大，平均值会把长尾藏起来。TTFT、TPOT、p95/p99 和队列时间比单个平均值更能解释体验。

误区 3：无限排队就能保护吞吐

队列能缓冲波峰，也会把延迟和资源压力往后推。没有 admission control，排队只是在放大拥塞。

误区 4：把 stream 当成性能优化

流式返回改善的是用户感知，不一定减少总计算量。它还会把连接管理、超时和返回链路的问题显出来。
