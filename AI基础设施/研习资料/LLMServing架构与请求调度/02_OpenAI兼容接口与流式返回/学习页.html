Unit 02｜接口合同与流式返回

OpenAI 兼容接口与流式返回

本页只讲一条边界：客户端如何通过 OpenAI 风格的请求和 SSE 流式返回拿到 token，服务端又如何把 stream、delta、finish_reason 和 [DONE] 串成一条可读的流。队列、路由、自动伸缩和容量规划先放到后面的单元。

课程地图上一单元下一单元

读完本页应该能做什么

  • 能区分 /v1/chat/completions 的请求合同、SSE 事件、choices[0].delta 和 [DONE] 各自负责什么。
  • 能手动拼接一段流式返回，知道哪些 chunk 没有正文、哪些字段只是结束信号。
  • 能用 TTFT、queue_wait、prefill、finish_reason 和取消状态解释“为什么看起来慢或断开”。

学习目标

Contract

看懂兼容接口

先把“兼容”理解成接口合同：客户端可以沿用熟悉的请求形状和响应形状，但这不自动保证模型能力、错误码、限流策略和 stream 节奏完全一致。

拆开流式返回

先区分传输层、chunk 和增量字段。SSE 负责单向推送，data: 行装载事件，delta 装载正文或状态变化，[DONE] 负责收尾。

把症状放回路径

看到 TTFT 变慢、半截回答、空 chunk、JSON 解析失败时，先把问题放回请求生命周期，再决定是看入口、流式解析、连接还是结束信号。

你现在应该能做什么：拿到一个 OpenAI 风格的流式调用示例时，先问“它返回的是一次完整 JSON，还是一串 SSE 事件”，再问“正文到底在什么字段里出现”。

概念骨架

Map

这张图只画公共边界：请求从客户端进来，先经过 API 合同，再进入 SSE 事件流，最后由客户端把增量内容拼回可展示文本。后面的单元会继续讨论队列、路由和伸缩，这里先不展开。

节点 它手里有什么 它依赖什么 错了会出现什么
HTTP 请求合同 model、messages、stream、max_tokens 等字段 客户端 SDK、服务端路由、鉴权和长度限制 请求还没进生成器，就先因为字段不合格被拒绝
SSE 事件流 一串 data: 行，外加空行和 [DONE] HTTP 长连接、代理超时、flush 时机 客户端只收到半截内容，或者把结束哨兵当成 JSON
增量拼接器 choices[0].delta.content 和结束状态 JSON 解析、chunk 顺序、buffer 状态 正文丢字、重复字、光标停住但其实服务端已经结束
展示与指标 屏幕上的半截文本、TTFT、TPOT、错误状态 前面三层的状态和日志 把服务慢、网络慢、用户取消混成同一件事

先抓主线，再看局部

读这张表时，先追“谁在写状态”，再追“谁在读状态”。只要把请求、事件、拼接器和展示层分开，很多流式问题就不会被一句“接口兼容”盖过去。

你现在应该能做什么：看到一个流式调用图时，先把每一层归到“合同、传输、解析、展示”四类，而不是只记住某个框架名。

术语与误区

Terms
术语 直觉 精确定义 常见误解
OpenAI-compatible 客户端能沿用熟悉的请求格式 服务端在端点、字段和返回形状上尽量贴近 OpenAI 风格 以为“兼容”就等于能力、错误和流速完全一致
SSE 服务端主动往客户端推事件 基于 HTTP 的单向事件流，适合逐段输出 token 以为它等于 WebSocket，或者只能浏览器前端用
data: 行 一条事件正文 事件帧里的负载行，外面还有空行和可能的前缀 把整条网络报文都当成 JSON
choices[0].delta 增量正文或增量状态 流式返回里承载角色、正文和局部变化的字段 把它和非流式的 message 混成同一个字段
[DONE] 结束哨兵 表示流结束，不是 JSON payload 对它直接 JSON.parse

这里先不追的东西

tool calling、多模态、函数调用细节、并发路由、batching 和自动伸缩都先放下。它们会在后面的单元回来；这页先把“怎么传、怎么收、怎么停”讲清。

你现在应该能做什么：看到 [DONE]、finish_reason、delta 时，先判断它们属于结束、增量还是状态字段，不再把它们混成一类。

请求合同

Contract

先看一个最小请求

最小请求样例 
{
  "model": "gpt-4o-mini",
  "messages": [
    {"role": "system", "content": "你是一个回答流式接口问题的助手。"},
    {"role": "user", "content": "解释 TTFT。"}
  ],
  "stream": true,
  "temperature": 0.2,
  "max_tokens": 128
}

这里的 stream 是整页的关键开关。它不决定模型会不会思考，只决定结果是一次性返回，还是拆成多段事件返回。

字段看什么

入口层先检查字段是否完整、是否超长、是否能路由到目标模型；生成层再决定它是否进入流式输出。两层都可能报错，但原因不同。

把字段放在这里看，不是为了背表，而是为了知道“问题从哪一层开始”。

字段 入口先检查什么 对后续有什么影响 错了会怎样
model 模型名是否存在、是否有权限 决定路由到哪个服务后端 直接 404 / 400，甚至在路由前就被拒
messages 角色顺序、内容长度、上下文总长 决定 prefill 的负担 太长时 TTFT 抬高，极端时被截断或拒绝
stream 是否允许流式返回 决定返回形状和客户端解析方式 客户端按一次性 JSON 读流式事件，马上乱套
temperature 是否在可接受范围内 影响采样，不影响流式协议 把采样参数误当成流式性能参数
max_tokens 是否超出服务限额 影响输出长度和 decode 时间 长输出拖慢后半段，不一定先体现在 TTFT
你现在应该能做什么：读一个请求时，先把 model、messages、stream、max_tokens 分成“路由、上下文、返回形状、输出长度”四类。

请求例子

Example

还是拿“解释 TTFT”这个问题做锚点。非流式和流式的差别，不在答案内容，而在你什么时候看到答案、怎么接住答案。

时点 非流式返回 流式返回 学习意义
开始 等完整答案算完 先收到角色 chunk 流式会更早暴露“系统已经在动了”
中间 看不到中间过程 多个 delta.content 逐段出现 客户端要自己拼接 buffer
结束 一次性带着完整 message.content 最后带 finish_reason，再发 [DONE] 结束信号和正文不是一回事

一段最小流式解析器

流式解析伪代码 
def read_stream(lines):
    parts = []
    for line in lines:
        if not line.startswith("data: "):
            continue
        payload = line[6:]
        if payload == "[DONE]":
            return "".join(parts)
        event = json.loads(payload)
        choice = event["choices"][0]
        delta = choice.get("delta", {})
        parts.append(delta.get("content", ""))
    return "".join(parts)

这段代码只做四件事：跳过无关行、识别结束哨兵、解析 JSON、把 delta.content 追加到 buffer。它很短，但已经足够暴露最常见的错误。

读这段代码时盯住什么

[DONE] 不是 JSON，所以它必须在 json.loads() 前被单独拎出来。delta 可能存在但没有正文，所以 content 要允许为空。

如果你把这两件事顺序颠倒，就会把正常流式当成异常，把异常当成正常。

你现在应该能做什么：看见一串流式响应时，先判断“这条线是在拼正文，还是在识别结束条件”，再决定自己该不该立即 JSON 解析。

流式拆包

Stream

流式返回最容易让人误判的地方，是把“更早看到字”误读成“模型更快”。实际情况只是：服务端更早 flush 了 chunk，客户端更早拿到了可显示的内容。

结束识别前态：收到一行数据

读：原始行内容。写：结束标志。错法：把 [DONE] 当 JSON，导致解析器在最后一行报错。

JSON 解析前态：一条 data payload

读：payload 字符串。写：事件对象。错法：连空行、前缀和结束哨兵一起解析，破坏正常流。

增量抽取前态：完整事件对象

读：choices / delta / finish_reason。写：当前增量。错法：把流式 chunk 当成一次性 message，正文读不到或读错。

正文拼接前态：已有一段 buffer

读：增量正文。写：累计文本。错法：每个 chunk 都覆盖旧 buffer，最后只剩最后几个字。

chunk 类型 有没有正文 客户端该做什么 最常见的误判
角色 chunk 通常没有 记录角色，不要展示空正文 以为没有 content 就是错误
正文 chunk 通常有 追加到 buffer 把每个 chunk 当成完整答案
结束 chunk 常常没有 读取 finish_reason，准备收尾 忽略结束状态，只看正文有没有变长
[DONE] 没有 关闭流，结束循环 继续 JSON 解析，或者把它当正文
你现在应该能做什么：看到一条流式日志时，先用“结束哨兵、JSON、增量、拼接”四步去解释，而不是一上来就说“接口坏了”。

完整走读

Walkthrough

还是用同一个锚点请求：stream: true，问题是“解释 TTFT”。这次不只看字段，而是把一次请求从首包、正文、结束、取消四个点顺着走到底。这样做的目的很直接：你以后看到半截回答时，能知道它卡在什么状态，而不是只知道“结果没出来”。

请求刚进入时，服务端先把角色 chunk 写出来。这个 chunk 往往没有正文，只是在告诉客户端“流已经开始，角色已经确定”。随后正文 chunk 一段一段出现，客户端 buffer 才开始变长。最后一个有效 chunk 会带上 finish_reason，例如 stop、length 或 cancel 一类状态。等到 [DONE] 到来，循环才真正收口。

时点 服务端在写什么 客户端在读什么 如果顺序错了会怎样
首个 chunk 角色、开始状态、首包可见性 确认流已经打开 把空正文当失败，误报一次正常开始
正文 chunk 增量 content 追加到 buffer 覆盖旧 buffer，最后只剩最后几个字
结束 chunk finish_reason 更新完成状态 把结束状态写进展示文本，页面上出现状态词
[DONE] 结束哨兵 关闭循环 继续 JSON 解析，最后一行直接报错
取消分支 断开或超时 保留 partial 文本，标记 interrupted 把主动取消写成 completed，后面就很难排障

最小状态机

流式状态机伪代码 
state = {
    "buffer": "",
    "done": False,
    "cancelled": False,
    "error": None,
}

for line in stream_lines:
    if line == "data: [DONE]":
        state["done"] = True
        break
    if not line.startswith("data: "):
        continue
    event = json.loads(line[6:])
    choice = event["choices"][0]
    delta = choice.get("delta", {})
    state["buffer"] += delta.get("content", "")
    if choice.get("finish_reason"):
        state["done"] = True

这段状态机只保留四个对象：buffer、done、cancelled 和 error。它比一个裸字符串更像真实客户端，因为流式接口本来就是在改状态，不是在一次性吐一个值。

为什么这段走读有用

它把“接口已经开始”“正文正在增长”“结束状态已经出现”“流已经关闭”拆成四个不同的观察点。以后你看日志、看 UI、看代理超时，都可以先把它们摆回这四个点，再决定该查哪一层。

如果你的实现把 finish_reason 直接写到展示文本里，或者把 [DONE] 当成普通 chunk，问题通常不是模型慢，而是客户端状态没收口。

你现在应该能做什么：拿到一条真实流式日志后，按首包、正文、结束、取消四个状态去复盘，而不是只检查“有没有最终答案”。

日志样本

Logs

真实排障时，最有用的不是“接口成功了还是失败了”，而是这一条流最后停在哪个状态。你可以先盯原始 SSE 行，再看客户端状态有没有跟上。

一段最小事件日志

SSE 事件日志 
data: {"choices":[{"delta":{"role":"assistant"},"finish_reason":null}]}
data: {"choices":[{"delta":{"content":"TTFT 是"},"finish_reason":null}]}
data: {"choices":[{"delta":{"content":"首 token 等待时间。"},"finish_reason":null}]}
data: {"choices":[{"delta":{},"finish_reason":"stop"}]}
data: [DONE]

这段日志的顺序很典型：先角色，再正文，再结束状态，最后是 [DONE]。如果你在中间把流关掉，buffer 里仍然会有部分可读文本；如果你把 [DONE] 当成 JSON，最后一行会先炸。

读日志时盯住什么

先看有没有 data:，再看最后一行是不是 [DONE]，然后检查 finish_reason 是否和用户看到的文本长度一致。若日志里只有正文，没有结束哨兵，通常是连接中断或代理提前断开；若 finish_reason 是 length，说明服务端按上限停了，不是解析器丢字。

这也是为什么“流式成功”不能只看 HTTP 200：HTTP 只说明连接开过，真正的状态还要看 chunk 顺序和最后一帧。

日志现象 说明什么 客户端状态 下一步检查
只看到 role chunk 流已打开，正文未到 连接正常，继续等 别把空正文当失败
看到 content，但没有 finish_reason 正文还在增长 保持 buffer，继续读取 不要提前收口
看到 finish_reason，但没有 [DONE] 结尾已写出，流还没完全关闭 继续读取最后哨兵 中间层可能还在 flush
没有 [DONE]，但连接断了 partial / interrupted 保留文本，标记中断 不要冒充 finished
你现在应该能做什么：拿到一段日志时，先看它停在哪个状态，再决定是重试、保留 partial 文本，还是只修解析器顺序。

客户端收流

Client

现在把视角切到客户端。真正决定体验的，不只是服务端发了什么，而是客户端怎么把这些 chunk 变成一条稳定的 UI 状态。这个过程里，最重要的不是字符串拼接本身，而是“什么时候显示、什么时候标完成、什么时候保留 partial、什么时候重试”这些状态切换。

一个稳妥的客户端会把流分成四个阶段：初始化、增长、收口、失败。初始化阶段先创建空 buffer 和 pending 状态；增长阶段不断追加 delta.content；收口阶段在 finish_reason 出现后把状态变成 finished，但继续等最后的 [DONE]；失败阶段如果连接先断，就保留 buffer，并把结果标成 interrupted 或 partial。这样做的好处是，用户看见的内容、日志里的状态和后续统计会一致。

事件 客户端 UI 状态 写什么 读什么 下一步检查
role chunk streaming 初始化消息壳 角色字段 不要展示空正文
content chunk streaming 追加 buffer delta.content 检查是否重复追加
finish_reason chunk finishing 写完成原因 finish_reason 不要立刻把连接判死，等最后哨兵
[DONE] finished 关闭 loading 结束标志 确认导出和日志已同步
socket close partial / interrupted 保留 buffer 连接错误、超时、取消信号 不要覆盖成 finished

客户端伪代码

客户端收流逻辑 
state = "streaming"
buffer = []

for event in stream_events:
    if event == "[DONE]":
        state = "finished"
        break

    chunk = json.loads(event)
    choice = chunk["choices"][0]
    delta = choice.get("delta", {})
    if delta.get("content"):
        buffer.append(delta["content"])

    reason = choice.get("finish_reason")
    if reason:
        state = "finishing"

if state != "finished":
    state = "partial"

这段代码强调三件事：先识别结束哨兵，再追加正文，最后才统一收口。它把“看见内容”和“状态完成”拆开了，所以即使流中途断掉，也不会把一半文本误标成完成。

为什么前端要保留 partial

因为用户已经看见了文本。只要已经展示过，后端状态就应该把它记成 partial，而不是从 UI 里擦掉重来。这样做不仅能减少误判，也方便排查到底是网络断了、代理关了，还是服务端主动结束。

如果你在这里把状态处理好，后面的日志、导出和重试策略都会轻很多。反过来，收流阶段一旦混乱，后面所有指标都会跟着乱。

你现在应该能做什么：自己写一个流式客户端时，先决定 UI 的四个状态：streaming、finishing、finished、partial，再决定每个 chunk 到来时改哪一个状态。

生命周期账本

Ledger

流式接口最麻烦的地方，不是某个字段，而是多个状态在不同时间被写入：请求对象先出现，连接随后打开，正文一段段积累，最后才有结束状态。你只要把这些状态记成账本，很多 bug 会立刻变得可查。

对象 谁创建 谁读取 谁写入 / 失效 错了会出现什么
请求对象 HTTP handler 鉴权、路由、日志、指标 被标记为接受、拒绝或重试 明明已经被拒，后面却还在尝试生成
流式连接 响应头写出后 客户端、代理、超时控制 由断开、超时或完成信号失效 服务已经发完，客户端却还等着“最后一口气”
文本 buffer 客户端或 UI 渲染、复制、导出 随着每个正文 chunk 追加 每个 chunk 覆盖旧值，最终只剩最后几个字
finish_reason 最后一个有效 chunk UI 状态、日志、错误分析 在完成、停止或截断时写入 把中断错看成正常完成，或者反过来
usage 常在结尾才出现 计费、观察、容量规划 在最终统计时写入 误以为每个 chunk 都会带 token 统计
取消信号 客户端或代理 服务端、监控、日志 在用户主动离开或网络断开时触发 把主动取消当成服务端故障

状态为什么重要

同一条回答里，finish_reason、usage、buffer 和连接状态都不是同一时刻写入的。只要你把它们硬塞进同一个时间点，日志和页面上的状态就很容易对不上。

你现在应该能做什么：排查半截回答时，先问“哪个对象还活着、哪个已经失效、哪个状态还没写到位”，而不是只看最终文本长度。

失败实验室

Failure Lab

这里不是为了做戏剧化报错，而是让你把常见失败和下一步检查绑在一起。三种情况里，只有一种是正常结束；另外两种都要求你先看结束信号，再看连接状态。

症状 更像什么 第一检查 下一步动作
最后一行是 [DONE]，但客户端还在解析 JSON 解析器顺序错了 确认结束哨兵是否在 JSON 解析前被拎出来 把 [DONE] 单独分支处理
正文只出来一半，连接就断了 中断或超时 看连接状态、代理超时和取消信号 把结果标成 interrupted / partial，而不是 finished
服务端已经结束，页面还显示卡住 客户端状态没收尾 看 finish_reason 是否被消费 把 buffer、结束状态和 UI 刷新一起收口 
data: {"choices":[{"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"choices":[{"delta":{"content":"你好"},"finish_reason":null}]}

data: {"choices":[{"delta":{},"finish_reason":"stop"}]}

指标分解

Metrics
TTFT = queue_wait + route_wait + prefill + first_chunk_write

queue_wait 是请求在入口等待的时间，route_wait 是挑副本和排队的时间，prefill 是把 prompt 转成可生成上下文的时间，first_chunk_write 是首个可见 chunk 写出到网络的时间。

这个公式不是为了背，而是为了把“慢”拆成几段。只看总耗时，很容易把网络、排队和计算揉成一个模糊感觉；拆开以后，每一段都能去找自己的证据。

指标 它主要在看什么 常见误判 第一检查
TTFT 用户等到第一个可见 chunk 的时间 以为它只等于模型推理时间 拆成 queue、route、prefill 和首包写出
TPOT 后续 token 的平均节奏 把首 token 慢和后续 token 慢混成一类 看 decode、batch 和输出长度
finish_reason 为什么停 把 stopped、length、cancel、error 混在一起 先看最后一个有效 chunk 和连接状态
usage 消耗了多少 token 把它当成每个 chunk 都有的实时字段 只在结尾或最终统计里找
取消计数 用户或代理是否主动断开 把主动取消当成服务端故障 看断开时机和日志里的 cancellation reason

stream 改了什么

stream: true 主要改的是“可见时点”。它让用户更早看到内容，但不自动缩短 prefill，也不自动减少总生成量。

你现在应该能做什么：当 TTFT 变慢时，先把等待和计算拆开；当 TPOT 变慢时，再去看 decode 和输出长度，而不是反过来。
