请求合同

Contract

先看一个最小请求

最小请求样例复制
{
 "model": "gpt-4o-mini",
 "messages": [
 {"role": "system", "content": "你是一个回答流式接口问题的助手。"},
 {"role": "user", "content": "解释 TTFT。"}
 ],
 "stream": true,
 "temperature": 0.2,
 "max_tokens": 128
}

这里的 stream 是整页的关键开关。它不决定模型会不会思考，只决定结果是一次性返回，还是拆成多段事件返回。

字段看什么

入口层先检查字段是否完整、是否超长、是否能路由到目标模型；生成层再决定它是否进入流式输出。两层都可能报错，但原因不同。

把字段放在这里看，为了知道“问题从哪一层开始”。

字段入口先检查什么对后续有什么影响错了会怎样
model模型名是否存在、是否有权限决定路由到哪个服务后端直接 404 / 400，甚至在路由前就被拒
messages角色顺序、内容长度、上下文总长决定 prefill 的负担太长时 TTFT 抬高，极端时被截断或拒绝
stream是否允许流式返回决定返回形状和客户端解析方式客户端按一次性 JSON 读流式事件，马上乱套
temperature是否在可接受范围内影响采样，不影响流式协议把采样参数误当成流式性能参数
max_tokens是否超出服务限额影响输出长度和 decode 时间长输出拖慢后半段，不一定先体现在 TTFT

> [!NOTE] 你现在应该能做什么：

> 读一个请求时，先把 model、messages、stream、max_tokens 分成“路由、上下文、返回形状、输出长度”四类。

> model

> messages

> stream

> max_tokens

请求例子

Example

还是拿“解释 TTFT”这个问题做锚点。非流式和流式的差别，不在答案内容，而在你什么时候看到答案、怎么接住答案。

一段最小流式解析器

流式解析伪代码复制
def read_stream(lines):
 parts = []
 for line in lines:
 if not line.startswith("data: "):
 continue
 payload = line[6:]
 if payload == "[DONE]":
 return "".join(parts)
 event = json.loads(payload)
 choice = event["choices"][0]
 delta = choice.get("delta", {})
 parts.append(delta.get("content", ""))
 return "".join(parts)

这段代码只做四件事：跳过无关行、识别结束哨兵、解析 JSON、把 delta.content 追加到 buffer。它很短，但已经足够暴露最常见的错误。

读这段代码时盯住什么

[DONE] 不是 JSON，所以它必须在 json.loads() 前被单独拎出来。delta 可能存在但没有正文，所以 content 要允许为空。

如果你把这两件事顺序颠倒，就会把正常流式当成异常，把异常当成正常。

> [!NOTE] 你现在应该能做什么：

> 看见一串流式响应时，先判断“这条线是在拼正文，还是在识别结束条件”，再决定自己该不该立即 JSON 解析。

流式拆包

Stream

流式返回最容易让人误判的地方，是把“更早看到字”误读成“模型更快”。实际情况只是：服务端更早 flush 了 chunk，客户端更早拿到了可显示的内容。

你现在应该能做什么：

看到一条流式日志时，先用“结束哨兵、JSON、增量、拼接”四步去解释，而不是一上来就说“接口坏了”。

完整走读

Walkthrough

还是用同一个锚点请求：stream: true，问题是“解释 TTFT”。这次不只看字段，而是把一次请求从首包、正文、结束、取消四个点顺着走到底。这样做的目的很直接：你以后看到半截回答时，能知道它卡在什么状态，而不是只知道“结果没出来”。

时点服务端在写什么客户端在读什么如果顺序错了会怎样
首个 chunk角色、开始状态、首包可见性确认流已经打开把空正文当失败，误报一次正常开始
正文 chunk增量 content追加到 buffer覆盖旧 buffer，最后只剩最后几个字
结束 chunkfinish_reason更新完成状态把结束状态写进展示文本，页面上出现状态词
[DONE]结束哨兵关闭循环继续 JSON 解析，最后一行直接报错
取消分支断开或超时保留 partial 文本，标记 interrupted把主动取消写成 completed，后面就很难排障

最小状态机

流式状态机伪代码复制
state = {
 "buffer": "",
 "done": False,
 "cancelled": False,
 "error": None,
}

for line in stream_lines:
 if line == "data: [DONE]":
 state["done"] = True
 break
 if not line.startswith("data: "):
 continue
 event = json.loads(line[6:])
 choice = event["choices"][0]
 delta = choice.get("delta", {})
 state["buffer"] += delta.get("content", "")
 if choice.get("finish_reason"):
 state["done"] = True

这段状态机只保留四个对象：buffer、done、cancelled 和 error。它比一个裸字符串更像真实客户端，因为流式接口本来就是在改状态，不是在一次性吐一个值。

为什么这段走读有用

它把“接口已经开始”“正文正在增长”“结束状态已经出现”“流已经关闭”拆成四个不同的观察点。以后你看日志、看 UI、看代理超时，都可以先把它们摆回这四个点，再决定该查哪一层。

如果你的实现把 finish_reason 直接写到展示文本里，或者把 [DONE] 当成普通 chunk，问题通常客户端状态没收尾。

> [!NOTE] 你现在应该能做什么：

> 拿到一条真实流式日志后，按首包、正文、结束、取消四个状态去复盘，而不是只检查“有没有最终答案”。

日志样本

Logs

真实排障时，最有用的这一条流最后停在哪个状态。你可以先盯原始 SSE 行，再看客户端状态有没有跟上。

读日志时盯住什么

先看有没有 data:，再看最后一行是不是 [DONE]，然后检查 finish_reason 是否和用户看到的文本长度一致。若日志里只有正文，没有结束哨兵，通常是连接中断或代理提前断开；若 finish_reason 是 length，说明服务端按上限停了，不是解析器丢字。

这也是为什么“流式成功”不能只看 HTTP 200：HTTP 只说明连接开过，真正的状态还要看 chunk 顺序和最后一帧。

日志现象说明什么客户端状态下一步检查
只看到 role chunk流已打开，正文未到连接正常，继续等别把空正文当失败
看到 content，但没有 finish_reason正文还在增长保持 buffer，继续读取不要提前收尾
看到 finish_reason，但没有 [DONE]结尾已写出，流还没完全关闭继续读取最后哨兵中间层可能还在 flush
没有 [DONE]，但连接断了partial / interrupted保留文本，标记中断不要冒充 finished

> [!NOTE] 你现在应该能做什么：

> 拿到一段日志时，先看它停在哪个状态，再决定是重试、保留 partial 文本，还是只修解析器顺序。

客户端收流

Client

现在把视角切到客户端。真正决定体验的，不只是服务端发了什么，而是客户端怎么把这些 chunk 变成一条稳定的 UI 状态。这个过程里，最重要的“什么时候显示、什么时候标完成、什么时候保留 partial、什么时候重试”这些状态切换。

事件客户端 UI 状态写什么读什么下一步检查
role chunkstreaming初始化消息壳角色字段不要展示空正文
content chunkstreaming追加 bufferdelta.content检查是否重复追加
finish_reason chunkfinishing写完成原因finish_reason不要立刻把连接判死，等最后哨兵
[DONE]finished关闭 loading结束标志确认导出和日志已同步
socket closepartial / interrupted保留 buffer连接错误、超时、取消信号不要覆盖成 finished

客户端伪代码

客户端收流逻辑复制
state = "streaming"
buffer = []

for event in stream_events:
 if event == "[DONE]":
 state = "finished"
 break

 chunk = json.loads(event)
 choice = chunk["choices"][0]
 delta = choice.get("delta", {})
 if delta.get("content"):
 buffer.append(delta["content"])

 reason = choice.get("finish_reason")
 if reason:
 state = "finishing"

if state != "finished":
 state = "partial"

这段代码强调三件事：先识别结束哨兵，再追加正文，最后才统一收尾。它把“看见内容”和“状态完成”拆开了，所以即使流中途断掉，也不会把一半文本误标成完成。

为什么前端要保留 partial

因为用户已经看见了文本。只要已经展示过，后端状态就应该把它记成 partial，而不是从 UI 里擦掉重来。这样做不仅能减少误判，也方便排查到底是网络断了、代理关了，还是服务端主动结束。

如果你在这里把状态处理好，后面的日志、导出和重试策略都会轻很多。反过来，收流阶段一旦混乱，后面所有指标都会跟着乱。

> [!NOTE] 你现在应该能做什么：

> 自己写一个流式客户端时，先决定 UI 的四个状态：streaming、finishing、finished、partial，再决定每个 chunk 到来时改哪一个状态。

生命周期账本

Ledger

流式接口最麻烦的地方，多个状态在不同时间被写入：请求对象先出现，连接随后打开，正文一段段积累，最后才有结束状态。你只要把这些状态记成账本，很多 bug 会立刻变得可查。

状态为什么重要

同一条回答里，finish_reason、usage、buffer 和连接状态都不是同一时刻写入的。只要你把它们硬塞进同一个时间点，日志和页面上的状态就很容易对不上。

> [!NOTE] 你现在应该能做什么：

> 排查半截回答时，先问“哪个对象还活着、哪个已经失效、哪个状态还没写到位”，而不是只看最终文本长度。

失败实验室

Failure

这里让你把常见失败和下一步检查绑在一起。三种情况里，只有一种是正常结束；另外两种都要求你先看结束信号，再看连接状态。

你现在应该能做什么：

当页面出现“半截内容”时，先判断它是正常结束、解析器错误还是连接中断，再决定要不要重试。

指标分解

Metrics

TTFT = queue_wait + route_wait + prefill + first_chunk_write

queue_wait 是请求在入口等待的时间，route_wait 是挑副本和排队的时间，prefill 是把 prompt 转成可生成上下文的时间，first_chunk_write 是首个可见 chunk 写出到网络的时间。

指标它主要在看什么常见误判第一检查
TTFT用户等到第一个可见 chunk 的时间以为它只等于模型推理时间拆成 queue、route、prefill 和首包写出
TPOT后续 token 的平均节奏把首 token 慢和后续 token 慢混成一类看 decode、batch 和输出长度
finish_reason为什么停把 stopped、length、cancel、error 混在一起先看最后一个有效 chunk 和连接状态
usage消耗了多少 token把它当成每个 chunk 都有的实时字段只在结尾或最终统计里找
取消计数用户或代理是否主动断开把主动取消当成服务端故障看断开时机和日志里的 cancellation reason

stream 改了什么

stream: true 主要改的是“可见时点”。它让用户更早看到内容，但不自动缩短 prefill，也不自动减少总生成量。

> [!NOTE] 你现在应该能做什么：

> 当 TTFT 变慢时，先把等待和计算拆开；当 TPOT 变慢时，再去看 decode 和输出长度，而不是反过来。

。好答案会说：兼容接口解决的是接入方式，不保证所有参数、上下文长度、错误码、工具调用和流式节奏都完全一致；能力清单能让客户端知道哪些功能可用、哪些要降级；这样上线和排障时就不会把“兼容”误读成“完全等价”。
