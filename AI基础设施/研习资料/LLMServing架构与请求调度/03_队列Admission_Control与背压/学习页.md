# 学习目标 {#goals}
<!-- tag: target -->

LLM Serving 的高并发问题，很多时候源于请求进入系统以后没有被清楚地分流。接口层继续接、队列继续堆、调度器继续尝试拼 batch，客户端还在等 stream，最后每一层都以为自己只是“慢一点”，实际已经把后面的资源拖进超时和重试风暴。

## box
### 能定位
看到一次请求，要能说出它现在卡在入口网关、服务队列、引擎调度器、prefill、decode、流式发送，还是客户端超时。定位不清，后面的控制策略会混在一起。

## box
### 能决策
请求来了以后，入口要先做判断。你需要根据队列深度、预计 token 工作量、SLO 剩余时间、KV cache 压力和租户优先级，决定接收、排队、拒绝、降级或让上游退避。

## box
### 能解释
当用户看到 429、503、stream 中断或首 token 很慢时，你要能把现象翻译回系统动作：是接纳策略在保护服务，还是排队策略过松，还是下游引擎已经没有有效容量。

读完这一段，你现在应该能做什么：先把“为什么慢”拆成“哪里排队、谁在拒绝、谁该退避”三个问题，再决定是否需要调大并发数。

# 概念骨架 {#map}
<!-- tag: map -->

这三个词经常一起出现，但它们不在同一个位置工作。Queue 是等待区，Admission Control 是门口的判断，backpressure 是向上游释放的压力信号。一个服务可以有队列但没有接纳控制，也可以有接纳控制但没有清楚的背压信号。生产问题往往出在后两者缺失：系统接了太多请求，又没有告诉调用方应该慢下来。

## box
### Queue
Queue 保存已经被服务接受、但还没有进入实际执行的请求。它能吸收短促流量尖峰，本身不创造容量。队列越长，排队等待越长，客户端的剩余 timeout 越少。对 LLM 来说，排队还会影响首 token 延迟，因为用户通常在第一个 token 前没有任何可见反馈。

## box
### Admission Control
Admission Control 在入口处判断请求能不能进入系统。判断依据可以是队列长度、预计 token 工作量、当前 KV cache、租户预算、用户等级、SLO 余量和异常重试信号。它的目的在于保护服务承诺，避免一个服务因为接入过多而让所有已接请求一起超时。

## box
### Backpressure
Backpressure 把服务端压力变成上游可理解的信号，例如 429、503、Retry-After、gRPC status、限流 headers、stream 早停原因。没有背压，上游通常会更快重试，形成正反馈：越忙越被打，越被打越忙。

概念它回答的问题常见指标出错时的症状


Queue已经接收的请求按什么顺序等待queue depth、queue wait p95、oldest request ageTTFT 变慢，连接占用变多，客户端超时增多
Admission Control新请求是否应该进入当前服务admitted、rejected、shed ratio、estimated work无保护时全局超时，有保护但过紧时误拒绝
Backpressure上游应该如何减速或重试429/503 rate、Retry-After、retry budget、client retry interval没有退避时重试风暴，错误码含混时调用方无法自适应

你现在应该能做什么：遇到 overload 现象时，先问“这是等待区过长、门口没关住，还是压力信号没传回去”。

# 术语与误区 {#terms}
<!-- tag: terms -->

接纳控制相关术语最容易被误用，因为它们都和“限制”有关。限制并发、限制队列、限制 token、限制连接，看起来像一件事，实际保护的是不同资源。LLM Serving 又多了 token 生成这个维度：一个请求的工作量由 prompt 长度、预估输出长度、KV cache 生命周期和 decode 步数共同决定，不能只按一条 HTTP 连接计算。

术语直觉解释精确含义常见误区


concurrency同时有多少请求在系统里正在运行、正在排队、正在 stream 的请求都可能占用不同资源只看 HTTP 并发，不看 token 工作量和 stream 持续时间
queue limit最多允许多少请求等着队列容量上限，通常和排队等待时间、SLO、内存上限一起设计把 limit 调大当扩容，导致旧请求在队列里等到没意义
max concurrent requests最多同时处理多少请求入口或 replica 层面的并发门限，可能不等于引擎内部 batch 大小以为它会自动保护 KV cache 和 prefill 峰值
estimated work请求大概要消耗多少算力基于 prompt tokens、max output tokens、cache 命中、模型和优先级估算只按请求数公平，忽略一个长 prompt 可能顶多个短问答
load shedding主动丢掉一部分请求过载时拒绝低优先级或不可能按时完成的请求，保护已接请求把所有拒绝都当事故，其实有些拒绝是系统自保
retry budget允许上游重试的空间约束重试频率、次数和抖动，避免失败请求立刻翻倍回流只返回错误码，不告诉上游等待多久或是否该换路由

一句实用判断：如果一个设置只改变“同时处理多少请求”，它不一定是 Admission Control；只有当它参与“新请求能不能进来”的决策，并且能产生清楚的接受或拒绝结果，才属于接纳控制的一部分。

# 接纳回路 {#control-loop}
<!-- tag: loop -->

Admission Control 可以看成一个短控制回路：读取当前系统状态，估算新请求成本，判断剩余预算，执行动作，再把结果写回指标和上游响应。这个回路要足够快，因为它在请求入口上；同时要读取多类信号，因为 LLM 请求的成本差异很大。

阶段读什么判断什么写什么


入口采样queue depth、running requests、oldest age、KV cache headroom当前服务是否已经接近拒绝阈值admission_snapshot 指标

策略决策阈值、租户配额、全局保护开关、降级规则admit、queue、reject、degrade 的哪一个风险最小decision label、reason code
反馈输出决策结果、推荐等待、可替代模型或输出上限上游是否能据此退避429/503、Retry-After、metrics、日志

# 状态账本 {#ledger}
<!-- tag: state -->

要设计接纳策略，先要有一份状态账本。这里的账本指入口层必须持续维护或快速读取的一组事实：已经有多少请求承诺给用户，队列里最老的请求等了多久，正在 decode 的请求还会占用多久，KV cache 是否接近碎片或容量上限，上游是否正在异常重试。

LLM Serving 的特殊之处在于，一个请求被接收以后，资源占用会跨阶段变化。prefill 可能短时间吃掉大量算力，decode 可能长时间占住 batch slot 和 stream 连接，KV cache 会随着生成增长。只记录“请求数”会把这些阶段差异抹平，策略很容易对长输出和短问答做出同样承诺。

账本项为什么要记过高时的动作过低或缺失的风险


queue_depth表示已接收但未执行的请求数量停止接收低优先级请求，或者只接短请求没有排队视图时，入口会继续把请求压进服务
oldest_queue_age_ms比平均等待更能暴露饥饿和队头阻塞触发 load shedding，或把大请求拆到低优先级路径只看平均值时，少数长等请求会被掩盖
running_prefill_tokensprefill 峰值决定短期算力压力限制长 prompt 或开启 chunked prefill 策略长 prompt 突增时，decode 流可能被拖慢
active_decode_sequencesdecode 阶段决定持续 batch slot 和 stream 占用限制 max output tokens，或按剩余输出预算降级只看 prefill 会低估长回答造成的尾部压力
kv_cache_headroomKV cache 不足会导致请求无法稳定推进拒绝长上下文、缩短输出、切换更小模型或等待释放内存压力会在调度器内部变成不可解释的慢或失败
client_deadline_left请求剩余等待时间决定排队是否还有意义剩余时间太短就直接拒绝，避免排队后超时系统处理了用户已经放弃的请求，浪费 GPU 时间
retry_rate_by_client识别上游是否正在用重试放大流量返回更强 backpressure，降低 retry budget把重试当新需求，会让错误峰值继续放大

一个好账本有两个特征：第一，能在入口决策时读到；第二，能在事故复盘时解释当时为什么这样决策。你现在应该能做什么：给一个 endpoint 列出最小状态账本，并说明最大队列长度只是其中一个字段。

# 引擎边界 {#engine}
<!-- tag: runtime -->

vLLM、TensorRT-LLM、SGLang、Ray Serve LLM 这类运行时都可能提供自己的并发、队列、batching 或调度参数。接纳控制要尊重这些机制，但不能把所有保护都丢给引擎内部。原因很简单：引擎看到请求时，服务层往往已经向客户端承诺了连接、超时、stream 语义和计费上下文。

入口层更适合做“是否承诺服务”的判断，引擎层更适合做“如何把已承诺请求高效执行”的调度。continuous batching、chunked prefill、prefix caching、KV cache 管理可以提高吞吐，但容量仍然有限。入口没有边界时，优化后的引擎也会被过量承诺压垮。

$$
 W = P + \alpha O + \beta K + \gamma S
 $$
 这里的 W 是粗略工作量，P 是 prompt tokens，O 是允许生成的输出 token 上限，K 是 KV cache 占用风险，S 是 stream 持续时间或连接成本。系数不需要一开始很准，但策略必须承认不同请求的成本不同。

接纳策略配置骨架复制

```json
{
 "endpoint": "chat-completions",
 "queue": {
 "max_depth": 128,
 "max_oldest_age_ms": 8000
 },
 "work_budget": {
 "max_prompt_tokens": 32000,
 "max_estimated_work": 48000,
 "prefer_short_when_overloaded": true
 },
 "backpressure": {
 "reject_status_when_queue_full": 503,
 "reject_status_when_client_budget_low": 429,
 "retry_after_ms": [500, 2000],
 "include_reason_code": true
 },
 "degrade": {
 "cap_max_new_tokens_under_pressure": 512,
 "disable_optional_tools_under_pressure": true
 }
}
```

问题入口层更适合引擎层更适合


是否承诺处理读客户端 timeout、租户预算、请求成本，决定接收或拒绝通常太晚，因为请求已经进入服务域
如何提升吞吐限制极端请求，维持可调度输入continuous batching、chunked prefill、prefix cache、KV 管理
如何解释错误返回 reason code、Retry-After、可观测 metrics暴露内部队列或调度失败信号
如何做租户保护按租户、用户、API key、优先级分配预算按已进入的请求执行公平调度或批处理

你现在应该能做什么：看到一个引擎参数时，判断它是在优化已接请求，还是在决定新请求能不能进来。两者都需要，边界要讲清楚。

# 接纳策略 {#policy}
<!-- tag: policy -->

最小可用的接纳策略不需要一开始就很复杂，但必须有明确的决策顺序。建议从三个判断开始：队列是否还有等待价值，请求工作量是否超过当前预算，上游是否能理解拒绝原因。只要这三件事写清楚，后续加租户权重、优先级、成本模型和动态阈值就不会乱。

最小接纳决策伪代码复制

```python
def decide_admission(req, state, policy):
 work = estimate_work(
 prompt_tokens=req.prompt_tokens,
 max_new_tokens=req.max_new_tokens,
 kv_headroom=state.kv_cache_headroom,
 stream=req.stream,
 )

 queue_budget_left = req.client_timeout_ms - state.oldest_queue_age_ms
 if queue_budget_left < policy.min_useful_queue_ms:
 return reject(status=429, reason="client_deadline_too_close", retry_after_ms=policy.short_retry_ms)

 if state.queue_depth >= policy.max_queue_depth:
 return reject(status=503, reason="queue_full", retry_after_ms=policy.long_retry_ms)

 if work > policy.max_work_when_overloaded and state.pressure == "high":
 if req.can_degrade:
 return degrade(max_new_tokens=policy.degraded_max_new_tokens, reason="work_too_large")
 return reject(status=429, reason="estimated_work_too_large", retry_after_ms=policy.long_retry_ms)

 return admit(queue="default", estimated_work=work)
```

这段伪代码有几个刻意的取舍。第一，客户端剩余时间太少时，继续排队会浪费服务端资源。第二，队列满时返回 503 更像服务当前不可用，工作量过大或预算不足时返回 429 更像调用方应该降速或调整请求。第三，降级要可解释，例如限制 `max_new_tokens`、关闭可选工具、切到更小模型，避免偷偷改变语义。

## box info
### 策略先保已接请求
已被接纳的请求已经占用客户端等待、服务端状态和用户预期。过载时继续接新请求，最常见后果是旧请求也超时，新请求也超时。拒绝一部分新请求，反而能让已承诺请求完成。

## box warn
### 拒绝必须带原因
只返回“busy”会让调用方猜。reason code 可以让网关、SDK 或业务层决定下一步：等待、换模型、缩短 prompt、降低输出上限，还是停止重试。

你现在应该能做什么：把一个口头规则，例如“忙的时候拒绝请求”，改写成可执行的判断顺序、错误码和 retry hint。

# 诊断实验室 {#diagnostics}
<!-- tag: debug -->

诊断 overload 时，只盯 GPU utilization 很容易误判。GPU 很忙可能是好事，说明 batch 被吃满；GPU 不忙也可能很糟，说明请求堵在网关、队列、KV 分配、网络或 stream 写出。更稳定的办法是沿请求生命周期逐段看：入口是否拒绝，队列是否增长，首 token 是否被拖慢，decode 是否稳定，客户端是否按提示退避。

症状优先看什么可能根因下一步动作


TTFT p95 上升，completion tokens/s 还可以queue wait、prefill tokens、oldest age入口排队过长，或长 prompt 把 prefill 拖慢限制队列等待，拆长上下文队列，观察 prefill 峰值
queue depth 不高，但 503 多KV cache headroom、engine reject、replica health引擎内部资源不足，入口队列没有反映真实瓶颈把引擎压力接回 admission state，避免入口误接
429 后流量继续上升client retry rate、Retry-After、SDK backoff上游没有退避，或错误码被业务层立即重试给 retry window，加 jitter，按 API key 限制重试预算
平均延迟正常，少数请求特别慢oldest queue age、priority starvation、long output队头阻塞或低优先级饥饿看 tail latency，给长请求单独队列或老化优先级
stream 开始快，后面断开多client disconnect、decode step latency、network write buffer输出很长、客户端超时、连接占用过久限制 max output，识别已断开请求并停止 decode

一个常用排查顺序是：先看入口是否拒绝，再看队列是否有意义，然后看引擎是否真实饱和，最后看客户端是否按信号退避。顺序错了会浪费时间，比如队列已经超过客户端 timeout，却还在分析 GPU 利用率为什么不满。

## box
### 最小排查提问
当你接手一条报警，可以先问四个问题：现在拒绝比例是多少，最老的排队请求等了多久，p95 prompt tokens 是否突增，上游在收到拒绝后有没有指数退避。四个答案放在一起，通常能把“扩容、限流、降级、修重试”区分开。

# 指标分解 {#metrics}
<!-- tag: metrics -->

LLM Serving 的延迟指标要拆开看。用户最敏感的是 TTFT，也就是从发起请求到收到第一个 token 的时间。TTFT 包含网络、入口、排队、调度、prefill 和第一次 decode。后续 tokens/s 又受 decode batch、KV cache、stream 写出影响。只看端到端 latency 会把不同瓶颈混在一起。

$$
 TTFT = T_{network} + T_{admission} + T_{queue} + T_{schedule} + T_{prefill} + T_{first\_decode}
 $$
 Admission Control 主要影响 T_admission 和 T_queue。好的接纳策略不保证每个请求都快，但它能限制 T_queue 无上限增长，并把无法及时完成的请求早点反馈给上游。

指标看 p50 还是 tail说明什么可执行动作


admission_rejected_total按 reason code 分组策略正在拒绝哪些请求区分 queue_full、work_too_large、deadline_too_close
queue_wait_msp95、p99、oldest已接请求是否还值得等待限制最大等待，触发拒绝或降级
estimated_work分 bucket 看 tail请求成本是否集中在长 prompt 或长输出按 work 分队列，给长请求单独预算
kv_cache_headroom最小值和持续低位是否接近内存或块分配瓶颈拒绝长上下文，缩短输出，释放已断开请求
client_retry_rate按 client、API key、status 分组上游是否放大失败返回 Retry-After，限制 retry budget，加 jitter
TTFTp95、p99 和分阶段 breakdown用户第一反馈被哪段拖慢把 queue、prefill、schedule 分开优化

你现在应该能做什么：看到 TTFT 变慢时，把它拆成可观测阶段，再判断是否真的是“模型响应慢”。

# 下一步 {#bridge}
<!-- tag: bridge -->

做完本单元，你已经有入口接纳和背压的基本框架。下一单元会把问题推进到副本路由和多租户：同一个请求如果应该被接收，它应该去哪个 replica，如何避免一个租户吃掉所有容量，如何在 routing、priority queue 和 tenant quota 之间保持一致。

从工程动作看，本单元之后最值得先做的是给现有 endpoint 补三类观测：接纳决策 reason code、排队等待分布、上游重试行为。只要这三类信号齐了，后面分析 routing、autoscaling 和 SLO 才有实证基础。

## box
### 带到下一单元的问题
当入口决定 admit 后，副本路由需要知道请求的 estimated work、tenant、deadline 和是否可降级。没有这些字段，路由器只能按表面负载分配，很难避免某个副本被长请求拖住。

## box
### 现在可以先做的检查
检查服务有没有记录每次 reject 的原因、有没有区分 429 和 503、有没有返回退避建议、有没有把客户端断开传回引擎停止生成。缺任何一项，后续调度都会少一块事实。

# 边界 {#boundary}
<!-- tag: scope -->

本页故意没有把所有调度问题都塞进来。continuous batching 的具体实现、prefill/decode 的内部队列、KV cache block allocator、跨副本负载均衡、autoscaling 触发器、SLO error budget 都和本主题相邻，但它们不替代 Admission Control。这里先把入口承诺和压力反馈讲清楚，是为了让后面的 runtime 和 routing 机制有稳定边界。

<!-- practice-group: 一、概念定位 -->

### 1 choice

**Q:** 哪个动作最像 Admission Control

- [ ] 已接请求在队列中等待 batch。
- [ ] 新请求到达时，根据队列和工作量决定是否接收。
- [ ] decode 阶段把多个 token 生成步骤合并调度。

Answer: 新请求到达时，根据队列和工作量决定是否接收。
Explain: Admission Control 的核心是新请求进门前的判断。

### 2 true/false

**Q:** 队列是否等于容量

- [ ] 对，排队更多就能处理更多。
- [ ] 不对，队列只是等待区，等待超过预算会让请求失去价值。

Answer: 不对，队列只是等待区，等待超过预算会让请求失去价值。
Explain: 队列只能吸收短峰值，不能创造 GPU、KV cache 或客户端等待预算。

### 3 debug-checklist

**Q:** 背压信号选择

- [ ] ，并且语义清楚。
- [ ] 或等价的退避窗口。
- [ ] ，例如 queue_full 或 estimated_work_too_large。
- [ ] 只返回一个没有说明的 busy。

Explain: 前三个都能帮助上游决策；最后一个通常会制造猜测，业务层可能立即重试。好的 backpressure 要让调用方知道“现在别打这么快、多久后再试、是否该缩短请求”。

### 5 scenario

**Q:** 长上下文压力

- [ ] 在 admission 日志中记录 estimated_work。

Explain: 限制长 prompt、降低输出上限、记录 estimated_work 都合理。只调大队列没有解决 KV cache 和 prefill 压力，反而会让长请求堆积更久。
