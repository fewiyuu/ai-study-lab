Unit 04｜副本路由与租户隔离

模型副本、路由与多租户

上一单元已经回答“新请求该不该进入服务”。这页从接纳后的下一步开始：同一个公开模型名背后可能有多个模型版本、多个租户池、多个 GPU 副本和不同缓存状态，路由层必须给出可解释的选择。

本页会用一条请求贯穿：先解析租户和模型别名，再过滤候选副本，最后在公平性、缓存亲和、延迟和灰度安全之间做取舍。自动伸缩、资源隔离和完整 SLO 规划会在后续单元展开。

课程地图上一单元下一单元

读完这页应该能做什么

  • 能追踪一次请求如何从 public model name 映射到 model family、version、tenant pool 和 replica。
  • 能设计一条最小 route log，留下 tenant、model version、candidate set、route reason、cache signal 和最终副本。
  • 能区分 hot tenant、cache-unfriendly routing、版本漂移和副本失效这几类常见问题。

学习目标

target

模型副本和路由看起来像负载均衡问题，但 LLM Serving 里的“负载”不只是一条 HTTP 连接。一个请求可能属于某个租户，命中某个 prompt 前缀缓存，使用某个模型版本，还会占用不同数量的 KV cache 和 decode slots。路由层的工作，是把这些信息变成可解释的分配结果。

能画路径

从 `model=gpt-x`、tenant、headers 和请求体出发，画出 model alias、version、tenant pool、candidate replicas 和 final replica 的路径。

能做取舍

知道什么时候优先 cache locality，什么时候优先 fairness，什么时候要把 canary、专属租户或低优先级流量隔开。

能查事故

看到 p99 上升、某租户被拖慢、cache hit 下降或某版本错误率异常时，能从 route log 和 replica metrics 找到入口。

这一页的重点不是写出最聪明的路由算法，而是把路由决策变成能读、能复盘、能逐步改进的系统动作。

课程位置

sequence

上一单元讲 Admission Control：容量不够时，新请求应该接收、排队、拒绝还是降级。接纳之后，系统才进入本页的问题：已接请求应该去哪个模型版本、哪个租户资源池、哪个 replica。

下一单元会讲自动伸缩、批处理和资源隔离。它们会改变副本数量、batching 行为和资源边界；本页先固定在“当前已有这些副本时，路由层怎么做选择”。这个顺序能避免一个常见误判：把 p99 上升直接归因到扩容不足，却没有看到路由把热租户、长请求或新版本流量压到了同一个副本。

阶段输入本页处理后续交给谁
Admission 之后已接请求、tenant、model name、deadline、estimated work解析模型、选择租户池、过滤候选副本replica 内部队列和引擎 scheduler
执行之前replica health、queue depth、cache signal、version policy打分、选路、写 route logprefill、decode、stream
事故复盘时p99、TTFT、cache hit、tenant error rate、route reason判断是路由倾斜、租户争用、版本异常还是缓存失效SLO、autoscaling、容量规划

读到这里，你应该先能把“该不该接”和“接了以后交给谁”分开。前者保护服务入口，后者决定请求落点。

概念骨架

map

一个生产 LLM endpoint 往往只有一个公开模型名，背后却有更多层：模型别名、版本、租户策略、资源池、副本和引擎状态。路由层越早把这些层写清楚，后面查问题越省力。

model alias

客户端看到的模型名，例如 `chat-prod`。它通常不是一个固定文件，而是指向某个模型家族和版本策略。

tenant policy

租户身份决定 quota、优先级、隔离池、灰度资格和日志字段。没有这层，公平性只能停在“每个请求都一样”。

replica set

同一模型版本的多个运行副本。副本可能负载不同、缓存不同、健康状态不同，不能只按请求数平均分。

路由层对象读取什么产出什么缺失时的症状
模型解析public model name、alias config、version policymodel family、target version、fallback version同名模型指到错误版本，灰度回滚困难
租户选择tenant id、quota、priority、isolation classtenant pool、rate class、admission budget热租户拖慢普通租户，VIP 和低优先级请求混跑
副本过滤health、warm state、capacity、model version、regioncandidate replicas请求打到冷副本、坏副本或版本不匹配副本
副本打分queue、inflight tokens、cache hint、recent errorsselected replica、route reasonp99 抖动、cache hit 下降、热点副本越来越热

一个好用的路由层不一定复杂，但它一定要把“为什么选它”写出来。否则每一次事故都会退回猜测。

术语与误区

terms

路由相关术语容易被普通负载均衡经验带偏。LLM Serving 的副本状态包含 GPU 队列、KV cache、模型版本、stream 生命周期和租户策略，这些状态变化比普通无状态 HTTP 服务更快。

术语直觉解释在 LLM Serving 中的含义常见误区
replica同一个服务的多个副本通常持有同一模型或同一版本的运行实例，背后占用 GPU、KV cache 和引擎队列以为副本是完全无状态，可以像普通 Web worker 一样随意轮询
router把请求送到某个后端读取模型、租户、健康、负载、缓存和版本策略后选择候选副本只看最少连接数，忽略长上下文和缓存亲和
tenant调用服务的一类客户或业务决定 quota、优先级、隔离池、审计字段和公平性规则只在鉴权层识别租户，路由和指标里不再带 tenant
affinity偏向把相似请求送到同一处可用于 prefix cache、模型热身状态、会话上下文或区域亲和为了命中缓存，把同一类请求压成单点热点
canary小流量试新版本按租户、比例、模型别名或请求特征把一部分流量导向新版本只记录总体错误率，不记录 version 和 route reason
fairness让各类请求不互相拖垮常通过配额、权重、隔离池、优先级队列和限流组合实现把公平性等同于“每个请求同权”，忽略 token 工作量差异

最实用的检查是：每个请求是否能回答这四件事：它代表哪个 tenant，目标 model version 是什么，候选 replicas 有哪些，最终选择的 reason code 是什么。

路由回路

trace

把路由写成一个短回路会更容易调试：解析身份，过滤候选副本，按策略打分，写出决策证据。每一步都应留下最少但够用的字段。

解析身份request identity

从 headers、token、model 字段和路由配置得到 tenant、model family、target version 和灰度资格。这里解析错，后面再均衡也会把流量送错。

过滤候选candidate set

先去掉 unhealthy、副本版本不匹配、租户无权限、区域不合适或冷启动未完成的副本。过滤阶段回答“谁能接”。

策略打分routing score

对候选副本按有效负载、缓存亲和、公平性欠账和近期错误打分。打分阶段回答“谁更适合接”。

提交结果route log

最终选择必须写入 route log：候选集合、选中副本、route reason、fallback 情况和关键分数。事故复盘依赖这些字段。

步骤读取写出错误后果
resolvemodel alias、tenant token、routing configtenant_id、model_version、policy class流量进错版本或错租户池
filterreplica health、version、pool membershipcandidate_replicas坏副本、冷副本或不兼容版本进入候选
scorequeue depth、inflight tokens、cache hint、fairness debtscore per replica、route reason热点副本持续被选中，p99 抖动
commitselected replica、fallback rule、request idroute log、metrics tags、trace span事故时无法还原请求为什么去那里

这一节的检查动作很简单：拿一条线上请求，能不能还原它的 candidate set。如果不能，route log 还不够用。

路由账本

state

路由层最怕“当时看起来没问题”。要让后来的人能复盘，日志不能只写 selected replica。至少要记录请求身份、候选范围、过滤原因、打分信号和最终动作。

字段示例复盘用途缺失风险
request_id`req-8f2a`串联网关、路由、副本和 stream 日志只能看聚合指标，无法追单个请求
tenant_id`tenant-pro`判断是否热租户、是否被隔离策略影响租户争用被平均值掩盖
model_version`chat-prod:v17`对齐 canary、回滚和版本错误率新旧版本问题混在一起
candidate_count`5 -> 3`知道过滤前后候选数量以为有很多副本，实际可用候选很少
filter_reason`version_mismatch, unhealthy`解释为什么某些副本不可选无法区分健康问题和策略问题
route_reason`cache_affinity_within_load_budget`解释最终选择只能看到结果，看不到决策依据
route log 最小字段

副本负载

score

普通服务常用 round-robin 或 least-connections。LLM 副本更难，因为请求的成本差异大：一个 64k prompt 的 prefill，和一个短问答的 decode，不应该按同一条连接计算。路由分数至少要看当前队列、运行中 token、KV cache headroom、错误率和缓存亲和。

$$effective\_load = w_q \cdot queue\_work + w_t \cdot inflight\_tokens + w_k \cdot kv\_pressure + w_e \cdot recent\_errors - w_c \cdot cache\_benefit$$

`queue_work` 表示等待中的估算工作量，`inflight_tokens` 表示正在占用 decode/prefill 的 token 压力，`kv_pressure` 表示 KV cache 余量不足带来的风险，`recent_errors` 惩罚近期异常，`cache_benefit` 是命中 prefix cache 或 warm state 的收益。

信号为什么要看错误用法更稳的读法
queue depth表示副本内部等待压力只数请求，不估工作量按 prompt tokens、max output 和等待时间估算 queue_work
inflight tokens比连接数更接近 GPU 工作把 stream 中连接都当成同等成本区分 prefill 压力和 decode 长尾
KV cache headroom决定是否还能承接长上下文只看 GPU utilization低 headroom 时惩罚长请求或切换池
cache hint命中缓存可降低 TTFT为了缓存把热点继续压到同一副本只在负载预算内奖励亲和
recent errors捕捉局部坏副本等健康检查彻底失败才摘除用短窗口错误率降低路由权重
副本打分伪代码
def score_replica(replica, request, policy):
    load = (
        policy.w_queue * replica.estimated_queue_work
        + policy.w_tokens * replica.inflight_tokens
        + policy.w_kv * replica.kv_pressure
        + policy.w_error * replica.recent_error_rate
    )
    cache_bonus = policy.w_cache * cache_affinity(replica, request.prefix_hash)
    fairness_penalty = tenant_debt(request.tenant_id, replica.pool_id)
    return load + fairness_penalty - cache_bonus

公式不要求一次定准权重。更重要的是先把信号拆开，这样 p99 变差时能知道是队列、token、KV cache、错误率还是缓存策略在起作用。

多租户隔离

tenant

多租户不是在鉴权里识别 `tenant_id` 就结束了。路由层要继续带着 tenant 信息做 quota、优先级、池选择和指标标签。否则一个热租户会通过长上下文、密集重试或高输出上限拖慢所有人。

策略适合场景代价诊断信号
共享池 + 权重租户多、负载较均匀、成本敏感隔离弱，热租户仍可能影响尾延迟tenant p99、per-tenant queue wait、fairness debt
专属池VIP、合规隔离、稳定大客户资源碎片和空闲成本pool utilization、reserved capacity、spillover rate
分级池免费/标准/企业不同承诺策略更复杂，需要清楚的降级路径class-level SLO、throttle rate、fallback reason
长请求隔离长上下文、批量总结、离线任务混入在线服务需要额外队列和容量预算prompt_tokens p95、prefill pressure、online p99
tenant policy 示例
tenant_policy = {
    "tenant-pro": {
        "pool": "enterprise",
        "priority": 90,
        "max_queue_work": 120_000,
        "canary_allowed": True,
        "spillover": "standard_when_enterprise_full"
    },
    "tenant-free": {
        "pool": "shared",
        "priority": 20,
        "max_queue_work": 8_000,
        "canary_allowed": False,
        "spillover": "reject_with_retry_after"
    }
}

如果指标只按 endpoint 聚合，你几乎看不到租户争用。至少把 `tenant_id`、`tenant_class`、`pool_id` 和 `route_reason` 作为指标或日志标签保留下来。

缓存亲和

cache

LLM 服务常希望相似 prompt、同一会话或相同 prefix 的请求落到已有缓存的副本上。这个策略能降低 TTFT，但它也会制造热点：如果所有相似请求都被路由到同一副本，缓存命中率会上升，p99 也可能上升。

路由选择收益风险保护条件
强 cache affinityTTFT 下降，prefill 重复工作减少热点副本更热，租户之间互相影响只在 effective_load 低于阈值时启用
弱 cache bonus兼顾缓存和负载均衡命中率不如强亲和把 cache 作为打分奖励，不作为硬过滤
忽略缓存实现简单，副本负载更平均重复 prefill 增多，TTFT 上升低峰、短 prompt 或缓存收益不明显时可接受
按租户分缓存池避免跨租户热点和合规风险缓存复用范围变小专属租户、敏感数据或稳定大客户
$$choose\_cached\_replica \iff cache\_hit = true \land effective\_load < load\_budget \land tenant\_budget > 0$$

这条规则故意把 cache hit 放在三个条件之一，而不是唯一条件。缓存命中只能说明“可能更快”，不能说明“这个副本现在还能稳稳接”。

排查 cache-aware routing 时，别只看全局 cache hit rate。要同时看命中后的 TTFT、选中副本的 effective_load、tenant p99 和 fallback reason。

版本与灰度

release

多版本路由让模型更新可控，但它也会让事故更难看清。一次错误率上升，可能来自新模型版本、某个 adapter、某个量化配置、某个租户触发的输入分布，也可能只是灰度流量被路由到了不稳定副本。

检查项为什么需要日志字段常见失败
alias 到版本映射确认公开模型名实际指向谁model_alias、model_version回滚后部分副本仍服务旧映射
canary eligibility限制哪些租户或请求能进新版本canary_allowed、canary_bucket高价值租户误入实验版本
版本级指标区分整体错误和单版本错误version、adapter、quantization聚合错误率看起来正常，单版本已异常
fallback path新版本失败时能否回到稳定版本fallback_version、fallback_reasonfallback 没写日志，事故后无法确认发生过
版本路由伪代码
def resolve_version(model_alias, tenant, request, rollout):
    stable = rollout[model_alias].stable_version
    canary = rollout[model_alias].canary_version
    if tenant.canary_allowed and in_bucket(request.id, rollout[model_alias].canary_percent):
        return canary, "canary_bucket"
    return stable, "stable_default"

灰度策略要和 route log 放在一起看。只知道“10% 流量进新版本”不够，还要知道是哪 10%、按什么规则进、失败后有没有 fallback。

指标分解

metrics

路由指标要同时能看全局、租户、版本和副本。只看全局 QPS、平均延迟和 GPU utilization，通常发现不了路由倾斜。

$$request\_latency = route\_ms + replica\_queue\_ms + prefill\_ms + decode\_ms + stream\_send\_ms$$

`route_ms` 是路由层决策本身的开销，`replica_queue_ms` 是被选副本内部等待。路由错误常表现为 route_ms 很小，但 replica_queue_ms 或 TTFT 在某些 tenant/version/replica 标签下异常。

指标维度标签说明异常解释
route_decision_totaltenant、model_version、route_reason每类路由决策数量某 reason 突然增加，说明策略或候选状态变化
candidate_countmodel_version、pool_id过滤后可用副本数量候选数下降会让少数副本承压
replica_queue_msreplica_id、tenant_class选中副本内部等待路由倾斜或副本局部过载
cache_affinity_hitprefix_hash bucket、tenant、replica缓存亲和是否生效hit 高但 p99 高，说明热点风险
version_error_ratemodel_version、canary_bucket按版本拆错误率新版本问题被 alias 聚合掩盖

指标维度不是越多越好。最少要保留 tenant、model_version、replica_id、pool_id 和 route_reason，先保证能回答“谁被送到哪里，为什么”。

下一单元会把“已有副本怎么选”推进到“副本数量、batching 和资源边界怎么变化”。这时路由指标会变成 autoscaling 和资源隔离的输入。

边界

scope

这里先不展开四件事：完整 autoscaling 策略、GPU/MIG 等资源隔离细节、服务网格层面的全局流量治理、计费和商业配额。它们都会影响路由，但如果现在一起学，容易把“当前请求为什么去这个副本”这条主线冲散。

暂不深入为什么先放下后续位置
autoscaling它改变副本数量；本页先学已有副本间如何选择Unit 05
batching它在副本内部影响吞吐；本页只看路由如何把请求交给副本Unit 05
SLO capacity planning它需要跨入口、路由、执行和观测综合分析Unit 06
框架部署细节Ray Serve LLM 等框架会把这些概念落成配置和部署形态Unit 07

边界清楚后，你现在应该能拿一条线上请求，解释它的模型版本、租户池、候选副本和最终 route reason。

synthesispolicy

写一条路由规则

写一条最小路由规则，要求同时处理 replica health、model_version、cache bonus 和 load budget。

答案至少包含：过滤条件、打分信号、最终 route_reason。不要把缓存命中写成唯一硬条件。

参考答案 / 评分要点

合格答案应先过滤 unhealthy、version mismatch 和 pool mismatch；然后用 queue_work、inflight_tokens、kv_pressure、recent_errors 计算负载；最后在负载预算内给 cache hit 加 bonus。route_reason 可以写成 `cache_affinity_within_load_budget` 或 `least_effective_load`。

synthesisledger

设计一个最小路由账本

为一个 OpenAI-compatible chat endpoint 设计 8-10 个 route log 字段，并说明其中 3 个字段能诊断什么问题。

至少覆盖 tenant、model_version、candidate set、selected replica、route_reason、cache 或 error signal。

参考答案 / 评分要点

可用字段包括 `request_id`、`tenant_id`、`tenant_class`、`model_alias`、`model_version`、`candidate_replicas_before/after`、`filter_reason`、`selected_replica`、`route_reason`、`cache_hint`、`recent_error_rate`、`score`。说明部分至少要能把字段映射到热租户、灰度版本异常、cache 热点或坏副本之一。
