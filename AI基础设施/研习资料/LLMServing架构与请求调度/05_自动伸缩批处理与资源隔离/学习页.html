术语与误区

Terms

最常混的几组词

术语 一句话 容易误读成什么
target_ongoing_requests 每个副本希望同时挂住多少个进行中的请求。 延迟上限。
continuous batching / in-flight batching 请求还在跑时，新的请求也能并进来，不必等整轮结束。 固定 batch。
prefill 把 prompt 喂进模型并建立上下文状态的阶段。 只是在做一次普通前处理。
decode 一个 token 一个 token 地往外生成。 和 prefill 一样的开销。
max_queued_requests 队列能堆多少请求。 GPU 显存上限。

先别混的三件事

  • 副本数变多，不等于单个副本内部的 batching 变好了。
  • GPU 利用率高，不等于短请求体验好。
  • 批处理吞吐上升，不等于系统整体更健康。

本页后面会把这三件事分别落到参数、队列和隔离策略上。

核心讲解

Mechanism

先分清伸缩对象

Ray Serve 看的是每个副本挂住了多少 ongoing requests。这个数高了，系统才会更想扩副本。扩出来的副本是服务入口，不是引擎内部的调度器。

所以，扩副本只能改变“有多少个执行入口”，不能自动解决“单个副本里该攒多大的 batch”这件事。后者要交给引擎层的 scheduler 和 token 预算。

层 读什么 写什么 看错会怎样
服务层 ongoing requests、queue depth、健康状态 副本数、路由落点、是否拒绝 把扩容和调度混成一个动作
引擎层 token 预算、KV cache、水位、prefill/decode 竞争 batch 形状、并发 token、逐 token 生成顺序 只盯副本数，忽略真正的瓶颈

批处理不是越大越好

batch 大一点，通常更容易把 GPU 吃满，吞吐会上去；但请求也更容易排队，首 token 时间会变长。对交互服务来说，这个代价往往比你想得更早出现。

$$\text{replicas}_{\text{start}} \approx \left\lceil \frac{\text{并发请求数}}{\text{target\_ongoing\_requests}} \right\rceil$$

这只是起点。长 prompt、长输出、冷启动时间和队列水位都会把真实值往上推。

动作 更直接保护什么 代价
降低 batch token 预算 TTFT、短请求体验 吞吐可能下降
提高 min_replicas 冷启动 闲置成本上升
按请求长度分桶 短请求不被长 prefill 拖住 资源利用率变复杂

资源隔离要隔到请求类型

真正难的是把不同类型的请求拆开。长上下文分析、短问答、批量摘要、灰度流量，最怕一起挤进同一个共享池。只要它们共享了同一条排队和同一组热副本，迟早会互相拖。

Ray Serve / Scheduler 伪代码 
def decide_pool(request, queue_depth, prompt_tokens):
    if prompt_tokens >= 6000:
        return "long-context-pool"
    if request.priority == "interactive" and queue_depth > 32:
        return "interactive-hot-pool"
    if request.priority == "batch":
        return "batch-pool"
    return "shared-pool"

def decide_scale(concurrency, target_ongoing_requests, queue_p95):
    base = math.ceil(concurrency / target_ongoing_requests)
    if queue_p95 > 1.5:
        base += 1
    return max(1, base)

这段代码不是生产实现，只是在把思路写成可检查的顺序：先分池，再决定副本数。

观测信号要分层看

GPU 利用率高，只说明卡被压着跑，不说明用户体验好。排队、TTFT、TPOT 和错误率要一起看，才能知道问题出在入口、批处理还是副本冷启动。

症状 先看什么 常见动作
TTFT 上升 队列时间、prefill、冷启动 保热副本、分桶、降 batch 预算
吞吐升了，但短请求更慢 长请求是否抢占了同池资源 拆池、独立队列、保留短请求预算
扩容后 p99 还是高 副本加载时间、健康检查、路由积压 确认热副本是否真的可用

完整走读

Worked Case

先看一轮真实得多的混合流量

假设你手里有一个交互聊天池和一个离线摘要池。聊天池每分钟 36 个请求，平均 prompt 900 token，平均输出 220 token；摘要池每分钟 12 个请求，平均 prompt 6400 token，平均输出 1024 token。两个池都从同一个模型家族出来，但它们应该承担的风险不同。

如果你把这两类请求硬放进同一个 shared pool，再用一个中等大小的 `target_ongoing_requests` 去兜住所有流量，表面上看副本数可能够了，实际上短请求会先被长 prefill 卡住。GPU 还在忙，TTFT 却先抬头，这就是“看起来健康，实际体验变差”的典型场景。

正确的做法不是先争论谁更重要，而是先问：哪类请求必须被保护，哪类请求可以等，哪类请求本来就应该为吞吐让步。这个问题的答案会直接决定分池、热副本和 token 预算。

先算一个可落地的起点

$$\text{start replicas} = \left\lceil \frac{\text{concurrency}}{\text{target\_ongoing\_requests}} \right\rceil$$

这个公式只负责给你一个起步值，不负责替你处理长 prompt、长输出和冷启动。

对聊天池来说，36 / 3 = 12，所以 12 个副本可以作为起点。对摘要池来说，12 / 2 = 6，所以 6 个副本也只是起点。它们不应该因为算出来都“差不多够”就强行合池，因为合池后被共享的是队列和 batch 预算，不只是副本数。

如果你把这两个池硬压成一个池，最容易漏掉的是长请求对短请求的隐性占用：prefill 会拉长，decode 也会变慢，最后你会在 p95 和用户反馈里才看到问题。

池子 输入画像 起始副本估算 如果直接合池会怎样 先看哪一个指标
interactive chat 36 r/m，prompt 900，output 220 12 短请求会被长 prefill 或更大的 batch 预算拖慢 TTFT、queue wait
batch summary 12 r/m，prompt 6400，output 1024 6 吞吐看起来更高，但短请求体验会先坏掉 TPOT、GPU 利用率、KV cache 水位
shared pool 两类请求都进同一条队列 12 到 15 之间看似都能跑 数字可能“看着对”，但 queue fairness 和 tail latency 会先失真 分位数、路由账本、冷启动时间

把估算写成一个小审计函数

容量审计伪代码 
def capacity_audit(concurrency, target_ongoing_requests, prompt_tokens, output_tokens, queue_p95):
    replicas = math.ceil(concurrency / target_ongoing_requests)
    if prompt_tokens >= 6000:
        replicas += 1
    if output_tokens >= 1024:
        replicas += 1
    if queue_p95 > 1.5:
        replicas += 1
    return max(1, replicas)

这段代码的价值不在于精确，而在于顺序：先有一个起点，再看长 prompt，再看长输出，最后才看排队。你如果反过来，只盯 queue_p95，往往会把真正的根因吞掉。

走读到这里，常见误判已经能看出来了

第一个误判是把副本数当成全部答案。副本够多，只能说明入口多，不说明 batch 和队列就稳了。第二个误判是把 batch 做大当成无条件增益。吞吐上去了，TTFT 也可能一起上去。第三个误判是把长请求当成“只是更慢一点”的请求。它们真正占住的是更长时间的上下文状态和更长的 decode 生命周期。

一旦你在这三件事里看见任何一件，下一步就不该再问“要不要再多加一台机器”，而应该先问“是哪类请求在争同一组资源”。

诊断回路

Triage

先拿一个真的能排查的场景

假设线上出现这样一组信号：交互聊天的 TTFT 从 0.9 秒涨到 2.4 秒，`queue_p95` 也跟着上升；与此同时，GPU 利用率看起来没有爆表，扩副本后 p99 只是在几分钟后慢慢回落。这个场景很典型，因为它把三种常见误判放在一起了：把问题直接归到 GPU、把扩副本当成唯一动作、把短请求和长请求混成一池。

这时第一步是先把症状拆成三个读数：排队是不是先坏了，prefill 是不是在抢时间，副本是不是还处在冷启动窗口。只要这三件事分开，你就知道该先动队列、分池，还是保热副本。

症状 先读什么 先写什么动作 如果读错会怎样
TTFT 抬头 queue wait、cold start、prefill 时长 保热副本、降 batch token、分桶 把问题当成纯算力不足
吞吐上升但短请求更慢 长请求占比、同池请求分布、TPOT 拆池、单独队列、保短请求预算 以为提升吞吐一定是正收益
扩副本后 p99 仍高 replica ready time、health check、routing backlog 先保可用副本，再谈继续扩容 只看副本数，不看可用性

把排查顺序写成一个小函数

故障分流伪代码 
def triage(ttft_p95, queue_p95, prompt_tokens, output_tokens, replica_ready_s):
    if replica_ready_s > 30:
        return "keep warm replicas"
    if prompt_tokens >= 6000:
        return "split long-context pool"
    if queue_p95 > 1.2 and ttft_p95 > 1.5:
        return "tighten queue and lower batch token budget"
    if output_tokens >= 1024:
        return "cap output and isolate batch jobs"
    return "hold steady and keep watching"

这个顺序很重要。先看副本是否真能接流量，再看请求是不是太长，再看队列和 batch。顺序一旦反过来，常见结果就是把“慢启动”误修成“更大的队列”，或者把“长请求争抢”误修成“继续扩容”。

实践时先试 练习区 里的第 5、8、10、11 题，再回来看这段函数能不能和你的答案对上。

Failure Case

一个最常见的冲突

交互聊天池里突然进来一批长摘要任务。GPU 看起来更忙了，吞吐也上去了，但短请求的 TTFT 开始抬头，客户会先感到“变慢”，而不是先感到“更省钱”。

现象 背后原因 先动哪层

练习前最后一把钥匙

              
如果你今天只带走一件事，就带走这句：伸缩看副本，批处理看 token，隔离看请求类型。只要这三件事仍然被一个公共池和一个公共队列捆在一起，问题就会以尾延迟、冷启动或长短请求互相伤害的形式回来。

            
做练习时，先把答案写成“对象 + 读什么 + 写什么 + 结果”四列，再谈怎么优化。这样更容易看出你到底是在调参数，还是只是在换说法。

scenarios = ["chat", "long_context", "batch_summary"]
for scenario in scenarios:
    fix(model_version="same", region="same", timeout="same")
    vary(scenario)
    record(["TTFT", "TPOT", "queue_wait", "p95", "replica_ready_s"])
    if scenario == "chat" and ttft_rises_first():
        lower_batch_budget()
    if scenario == "long_context" and queue_spike():
        split_pool()
    if scenario == "batch_summary" and short_chat_slows():
        isolate_batch_pool()

            
这个顺序的价值在于：你先拿到三条稳定曲线，再去解释副本、batch 和隔离分别出了什么问题。顺序错了，常见结果就是把“请求太长”误修成“batch 还不够大”，或者把“队列太深”误修成“再加几个副本”。

            
做练习 q12 时，先把“变化项”和“固定项”分开写，再去写指标。只写参数名，通常说明你还没把实验目的想清楚。

常见错误

          Mistakes
        

        

          

            
只盯 GPU 利用率

            
卡跑满了，不代表用户体验好。先看 TTFT 和队列时间，再看利用率。

          

          

            
把 `target_ongoing_requests` 当延迟上限

            
它只描述副本希望挂住多少进行中的请求，不表示 SLA，也不直接给出用户要等多久。

          

          

            
把长请求和短请求混在一起

            
一个共享池能省心，但一旦长 prefill 进来，短请求会先被拖慢。拆池通常比一味加机器更直接。

synthesisdiagnosis

              
某服务流量突增后，Ray Serve 已经扩了副本，但 p99 仍然高了几分钟。列出三个可能原因，并写出你会看什么指标或日志。

                参考答案 / 评分要点
                
至少要提到三类：1) 新副本冷启动慢或没真正进入健康状态，先看加载时间、就绪探针和副本日志；2) 队列已经积压，先看 queue depth、排队时间和 ongoing requests；3) 长请求或坏副本把流量拖住，先看 route reason、长短请求分布和 p95/p99 分层指标。
