# 核心讲解 {#lesson}
<!-- tag: Mechanism -->
<!-- layout: grid2 -->

## box
### 先分清伸缩对象
Ray Serve 看的是每个副本挂住了多少 ongoing requests。这个数高了，系统才会更想扩副本。扩出来的副本是服务入口，不是引擎内部的调度器。
所以，扩副本只能改变“有多少个执行入口”，不能自动解决“单个副本里该攒多大的 batch”这件事。后者要交给引擎层的 scheduler 和 token 预算。

## box info
### 批处理不是越大越好
batch 大一点，通常更容易把 GPU 吃满，吞吐会上去；但请求也更容易排队，首 token 时间会变长。对交互服务来说，这个代价往往比你想得更早出现。
$$\text{replicas}_{\text{start}} \approx \left\lceil \frac{\text{并发请求数}}{\text{target\_ongoing\_requests}} \right\rceil$$
 这只是起点。长 prompt、长输出、冷启动时间和队列水位都会把真实值往上推。

## box

真正难的是把不同类型的请求拆开。长上下文分析、短问答、批量摘要、灰度流量，最怕一起挤进同一个共享池。只要它们共享了同一条排队和同一组热副本，迟早会互相拖。

```python
def decide_pool(request, queue_depth, prompt_tokens):

 return "long-context-pool"
 if request.priority == "interactive" and queue_depth > 32:
 return "interactive-hot-pool"
 if request.priority == "batch":
 return "batch-pool"
```

# 完整走读 {#walkthrough}
<!-- tag: Worked Case -->
<!-- layout: grid2 -->

## box
### 先看一轮真实得多的混合流量
假设你手里有一个交互聊天池和一个离线摘要池。聊天池每分钟 36 个请求，平均 prompt 900 token，平均输出 220 token；摘要池每分钟 12 个请求，平均 prompt 6400 token，平均输出 1024 token。两个池都从同一个模型家族出来，但它们应该承担的风险不同。
如果你把这两类请求硬放进同一个 shared pool，再用一个中等大小的 `target_ongoing_requests` 去处理所有流量，表面上看副本数可能够了，实际上短请求会先被长 prefill 卡住。GPU 还在忙，TTFT 却先抬头，这就是“看起来健康，实际体验变差”的典型场景。
正确的做法先问：哪类请求必须被保护，哪类请求可以等，哪类请求本来就应该为吞吐让步。这个问题的答案会直接决定分池、热副本和 token 预算。

## box info
### 先算一个可落地的起点
$$\text{start replicas} = \left\lceil \frac{\text{concurrency}}{\text{target\_ongoing\_requests}} \right\rceil$$
 这个公式只负责给你一个起步值，不负责替你处理长 prompt、长输出和冷启动。
对聊天池来说，36 / 3 = 12，所以 12 个副本可以作为起点。对摘要池来说，12 / 2 = 6，所以 6 个副本也只是起点。它们不应该因为算出来都“差不多够”就强行合池，因为合池后被共享的是队列和 batch 预算，不只是副本数。
如果你把这两个池硬压成一个池，最容易漏掉的是长请求对短请求的隐性占用：prefill 会拉长，decode 也会变慢，最后你会在 p95 和用户反馈里才看到问题。

## box
### 把估算写成一个小审计函数
容量审计伪代码
复制
```python
def capacity_audit(concurrency, target_ongoing_requests, prompt_tokens, output_tokens, queue_p95):
 replicas = math.ceil(concurrency / target_ongoing_requests)
 if prompt_tokens >= 6000:
 replicas += 1
 if output_tokens >= 1024:
 replicas += 1
 if queue_p95 > 1.5:
 replicas += 1
 return max(1, replicas)
```

这段代码的价值不在于精确，而在于顺序：先有一个起点，再看长 prompt，再看长输出，最后才看排队。你如果反过来，只盯 queue_p95，往往会把真正的根因吞掉。

## box warn
### 走读到这里，常见误判已经能看出来了
第一个误判是把副本数当成全部答案。副本够多，只能说明入口多，不说明 batch 和队列就稳了。第二个误判是把 batch 做大当成无条件增益。吞吐上去了，TTFT 也可能一起上去。第三个误判是把长请求当成“只是更慢一点”的请求。它们真正占住的是更长时间的上下文状态和更长的 decode 生命周期。
一旦你在这三件事里看见任何一件，下一步就不该再问“要不要再多加一台机器”，而应该先问“是哪类请求在争同一组资源”。

<!-- tag: Triage -->
<!-- layout: grid2 -->

## box
### 先拿一个真的能排查的场景

## box info
### 把排查顺序写成一个小函数
故障分流伪代码
复制
```python
def triage(ttft_p95, queue_p95, prompt_tokens, output_tokens, replica_ready_s):
 if replica_ready_s > 30:
 return "keep warm replicas"
 if prompt_tokens >= 6000:
 return "split long-context pool"
 if queue_p95 > 1.2 and ttft_p95 > 1.5:
 return "tighten queue and lower batch token budget"
 if output_tokens >= 1024:
 return "cap output and isolate batch jobs"
 return "hold steady and keep watching"
```

这个顺序很重要。先看副本是否真能接流量，再看请求是不是太长，再看队列和 batch。顺序一旦反过来，常见结果就是把“慢启动”误修成“更大的队列”，或者把“长请求争抢”误修成“继续扩容”。

# 上线顺序 {#playbook}
<!-- tag: Playbook -->
<!-- layout: grid2 -->

## box
### 先排顺序，再调数字
真实上线时，顺序很重要。先切分请求类型，再决定热副本和队列上限；接着定 batch token 预算；然后才是 `target_ongoing_requests`。很多人会反过来，从一个数字开始乱调，最后调出来的只是“看着不堵”的表象。
一个简单的判断法是：如果你不先分池，就很难区分是扩容慢、batch 太大，还是长请求本来就不该和交互请求共池。先把问题缩成“哪一池出事”，再去看“哪一个 knob 该动”。

```python
 record(["TTFT", "TPOT", "queue_wait", "p95", "replica_ready_s"])
 if scenario == "chat" and ttft_rises_first():
 lower_batch_budget()
 if scenario == "long_context" and queue_spike():
 split_pool()
 if scenario == "batch_summary" and short_chat_slows():
 isolate_batch_pool()
```

这个顺序的价值在于：你先拿到三条稳定曲线，再去解释副本、batch 和隔离分别出了什么问题。顺序错了，常见结果就是把“请求太长”误修成“batch 还不够大”，或者把“队列太深”误修成“再加几个副本”。
做练习 q12 时，先把“变化项”和“固定项”分开写，再去写指标。只写参数名，通常说明你还没把实验目的想清楚。

# 常见错误 {#mistakes}
<!-- tag: Mistakes -->

只盯 GPU 利用率
卡跑满了，不代表用户体验好。先看 TTFT 和队列时间，再看利用率。


把 `target_ongoing_requests` 当延迟上限
它只描述副本希望挂住多少进行中的请求，不表示 SLA，也不直接给出用户要等多久。


把长请求和短请求混在一起
一个共享池能省心，但一旦长 prefill 进来，短请求会先被拖慢。拆池通常比一味加机器更直接。

### 1 choice

**Q:** Ray Serve 的 `target_ongoing_requests` 主要表示什么？

- [ ] . 每个副本希望同时挂住的进行中请求数
- [ ] . 单次生成允许输出的 token 上限
- [ ] . GPU 的显存上限

Answer: . 每个副本希望同时挂住的进行中请求数
Explain: 它是每个副本希望挂住的进行中请求数，不是延迟上限。

### 2 true/false

**Q:** 连续批处理等同于固定 batch，一批结束后才开始下一批。

- [ ] 对
- [ ] 错

Answer: false
Explain: 连续批处理允许新请求在运行过程中并入，不是等一批完全结束后再开始下一批。

### 3 fill

**Q:** 填空：LLM 请求在生成期间持续占用什么缓存？

Explain: 答案是 KV cache 。它会在请求生成期间一直保留上下文状态，所以长请求会更占资源，也更容易影响同池里的短请求。

### 4 choice

**Q:** vLLM 的 `max_num_batched_tokens` 直接限制什么？

- [ ] . 单轮批处理的 token 预算
- [ ] . 每秒 QPS 上限
- [ ] . 模型版本数量

Answer: . 单轮批处理的 token 预算
Explain: 它限制的是单轮批处理能吃下的 token 预算，不是每秒 QPS。
