## box soft
### 把体验写成 SLO
先区分首 token 等待、后续 token 节奏、完整请求完成和可用性，再把它们写成 p95 / p99、错误率和错误预算，而不是笼统地写“低延迟”。

# 概念导图 {#map}
<!-- tag: Signals → Capacity -->
<!-- layout: grid2 -->

## box
### 先把四层分开看
用户最先感受到的是首 token、后续 token 和总完成时间；服务层先看到的是队列、路由和拒绝；引擎层先看到的是 prefill、decode 和 KV cache；容量层最后看到的是副本数、利用率和错误预算。
很多排障讨论之所以绕，是因为把这四层混成了一层。先把层级分开，后面的指标才有位置。

## box info
### 一条顺序
体验承诺 → 指标定义 → 读数归位 → 容量估算 → 压测校准 → 告警与扩容。
如果顺序反了，最后通常只剩“看起来很忙”，却说不清忙在哪里。

| 层级 | 正在看什么 | 读什么指标 | 写什么状态 | 最常见的误判 |
| --- | --- | --- | --- | --- |
| 入口与队列 | 请求有没有被接住 | queue_wait、pending request、reject rate | 接纳结果、排队位置、等待时长 | 把排队问题看成模型慢 |
| 引擎前半段 | prompt 有没有被转成可生成上下文 | TTFT、prefill time、cache 命中 | KV cache、sequence 状态 | 把长 prompt 和短 prompt 当成同一类 |
| 引擎后半段 | 后续 token 生成节奏 | TPOT、decode time、output length | token 流、batch 状态 | 把首 token 慢和后续慢混在一起 |
| 容量与 fleet | 整套系统还能撑多少 | p95 / p99、GPU util、错误预算、pending replica | 副本数、告警阈值、限流动作 | 只看平均利用率，不看尾部 |

# 术语与误区 {#terms}
<!-- tag: Terms -->

| 术语 | 直觉 | 精确定义 | 常见误解 |
| --- | --- | --- | --- |
| TTFT | 用户等到首 token 的时间 | 从请求进入系统到第一个可见 token 出现之间的时间 | 误以为它只等于模型算力时间 |
| TPOT | 后续 token 的节奏 | 生成阶段每个 token 的平均时间，常用来描述 decode 速度 | 把它和 TTFT 混成一个指标 |
| p95 / p99 | 长尾体验 | 分位数，不是平均值 | 拿平均值代替尾部体验 |
| 错误预算 | 还可以接受多少失败 | 在一段时间里允许消耗的可靠性余量 | 只把它看成告警阈值 |
| queue_wait | 排队等待 | 请求在入口或调度器里等副本空出来的时间 | 把这段时间算到模型推理里 |
| GPU util | 资源忙不忙 | GPU 的使用率，只是容量线索之一 | 把它当成唯一扩容信号 |

## box warn
### 边界先放这里
这一页不讨论 tracing backend 的实现细节，也不把某个框架的指标名当成唯一标准。你要保住的是“指标背后对应哪一层、哪一个状态”。

# 指标拆解 {#metrics}
<!-- tag: Metrics -->

$$TTFT = queue\_wait + route\_wait + prefill + first\_chunk\_write$$
 `queue_wait` 是请求在入口等副本的时间，`route_wait` 是挑副本和确认健康状态的时间，`prefill` 是把 prompt 转成可生成上下文的时间，`first_chunk_write` 是首个可见 chunk 写出到网络的时间。

## box
### 这个分解的用处
总 TTFT 只有一个数，看不出慢在等待还是慢在计算。拆开以后，每一段都能去找自己的证据：排队看负载，prefill 看 prompt 长度和 cache，写出看连接和代理，路由看副本健康和热度。

## box info
### 先从哪个数开始看
如果 TTFT 上升而 TPOT 稳定，先看 queue_wait 和 prefill。
如果 TTFT 正常而 TPOT 变慢，先看 decode、batch 混合和输出长度。
如果两者一起变差，先怀疑容量和长尾。

| 指标 | 它主要在看什么 | 常见误判 | 第一检查 |
| --- | --- | --- | --- |
| TTFT | 首 token 的可见等待 | 把它当成纯模型算力 | 拆成 queue、route、prefill 和首包写出 |
| TPOT | 后续 token 的平均节奏 | 把首 token 慢和后续慢混在一起 | 看 decode、batch 和输出长度 |
| p95 / p99 | 尾部体验 | 把平均值当成全部 | 看长尾请求的分布和排队 |
| GPU util | 资源使用率 | 把它当成是否扩容的唯一依据 | 同时看 queue_wait、pending replica 和错误率 |
| 错误率 | 系统是否已经在失守 | 只盯成功率，不看失败类型 | 先分拒绝、超时、取消和真正故障 |

容量估算伪代码复制

```python
def estimate_replicas(qps, avg_input_tokens, avg_output_tokens, per_replica_tps, target_util=0.70, safety=1.20):
 demand_tps = qps * (avg_input_tokens + avg_output_tokens)
 usable_tps = per_replica_tps * target_util
 raw = demand_tps / usable_tps
 return math.ceil(raw * safety)
```

$$required\_replicas = \left\lceil \frac{QPS \times (input\_tokens + output\_tokens)}{per\_replica\_tps \times target\_utilization} \times safety\_factor \right\rceil$$
 这个数先给你一个起点。真正上线时，还要看长度分布、prefill / decode 的不对称、KV cache、冷启动和流量长尾。

# 状态追踪 {#trace}
<!-- tag: Trace -->

<!-- interactive: ['shape-tracer'] → keep as HTML -->

| 对象 | 谁创建 | 谁读取 | 谁写入 / 失效 | 典型症状 |
| --- | --- | --- | --- | --- |
| 请求对象 | 入口解析层 | 鉴权、路由、观测 | 在入口阶段被标记合法、拒绝或重试 | 字段不合法却进到后面，最后变成 5xx |
| 队列位置 | 服务层 admission / router | 调度器、健康检查、限流器 | 入队、出队、丢弃、降级 | TTFT 长但引擎利用率不高 |
| sequence | 副本内部 | prefill、decode、停止条件 | 状态随着 token 生成逐步变化 | 把 request 级别变量误当成 sequence 状态 |
| KV cache | prefill 阶段 | decode、后续 batch | 被写入、复用、淘汰 | prefill 很重，但缓存没起作用 |
| 流式连接 | 返回链路 | 客户端、网关、超时控制 | 写入 token、关闭、重置 | 服务已经生成，客户端看起来像没返回 |
| 指标记录 | 返回完成时 | 观测、容量规划、告警 | 写入 TTFT、TPOT、queue time、status | 指标没分层，所有问题都被平均值吞掉 |

# 容量案例 {#case}
<!-- tag: Worked Case -->
<!-- layout: grid2 -->

## box
### 一个最小但不失真的估算
假设现在有 25 QPS，平均输入 900 token，平均输出 500 token，单副本稳定吞吐 10,000 token/s，目标利用率 70%，安全系数 1.2。这个数字足够小，能手算；也足够真实，不会把容量问题抹平。
先把请求换成 token/s，再除以可用吞吐，最后乘上安全系数。结果是 6 个副本。

## box info
### 为什么不是算完就结束
这只是第一轮估算。真实流量里，输入长度和输出长度通常不是平均分布，prefill 和 decode 的代价也不一样；如果把这些差异忽略掉，估算会很快偏离。

| 输入 | 值 | 中间结果 | 说明 |
| --- | --- | --- | --- |
| QPS | 25 | 每秒 25 个请求 | 流量入口 |
| 平均输入 token | 900 | | prefill 负载 |
| 平均输出 token | 500 | | decode 负载 |
| 单副本稳定 token/s | 10,000 | 可用吞吐 = 7,000 token/s | 乘上目标利用率 70% |
| 安全系数 | 1.2 | 最终副本数 = 6 | 为尾部和抖动留余量 |

容量估算过程复制

```python
qps = 25
input_tokens = 900
output_tokens = 500
per_replica_tps = 10000
target_util = 0.70
safety = 1.20

demand_tps = qps * (input_tokens + output_tokens) # 25 * 1400 = 35000
usable_tps = per_replica_tps * target_util # 10000 * 0.70 = 7000
replicas = math.ceil((demand_tps / usable_tps) * safety) # ceil(5 * 1.2) = 6
```

# 告警门槛 {#alerts}
<!-- tag: Alerting -->
<!-- layout: grid2 -->

## box
### 看板不该只会“红了再说”
很多团队的第一版告警只有一句“延迟升高就报警”。这种写法的问题不在于太粗，而在于它没有把门槛和动作连起来。TTFT 升高时，可能该查队列；TPOT 变慢时，可能该查 decode；错误率上升时，可能该查拒绝、超时或下游断开。如果同一条告警同时覆盖这些状态，最后通常只会在最差的时候响，而且没人知道先做什么。
更好的方式，是把告警分成“预警”和“动作告警”。预警用来提醒趋势正在偏，动作告警用来触发限流、扩容或降级。这样做的好处很简单：告警不再只是噪声，而是能接到具体操作。

## box warn
### 先看一个小场景
上午看板显示 TTFT p95 连续 10 分钟高于 2.5 秒，queue_wait p95 也高于 1.5 秒，但错误率还只有 0.2%，GPU util 在 62%。这不是“服务已经坏了”，更像是排队在变长、尾部在抬高。这个时候如果只按错误率报警，告警会来得太晚；如果只按 GPU util 报警，告警又会来得太早。

| 信号 | 预警门槛 | 动作门槛 | 建议动作 |
| --- | --- | --- | --- |
| TTFT p95 | 连续 10 分钟 > 2.0 秒 | 连续 5 分钟 > 3.0 秒 | 先看 queue_wait、route_wait 和 prefill |
| TPOT p95 | 连续 10 分钟 > 50 ms | 连续 5 分钟 > 80 ms | 先看 decode、batch 混合和长输出 |
| queue_wait p95 | 连续 10 分钟 > 1.0 秒 | 连续 5 分钟 > 2.0 秒 | 先看接纳、路由和 pending replica |
| 错误率 | 连续 10 分钟 > 0.5% | 连续 5 分钟 > 1.0% | 先分拒绝、超时、取消和真实故障 |

告警规则伪代码复制

```python
def alert(ttft_p95, tpot_p95, queue_wait_p95, error_rate, gpu_util):
 warn = []
 action = []
 if ttft_p95 > 2.0:
 warn.append("TTFT warn")
 if ttft_p95 > 3.0 or queue_wait_p95 > 2.0:
 action.append("TTFT action")
 if tpot_p95 > 0.05:
 warn.append("TPOT warn")
 if tpot_p95 > 0.08:
 action.append("TPOT action")
 if error_rate > 0.01:
 action.append("error action")
 if gpu_util > 0.85 and queue_wait_p95 > 1.0:
 action.append("capacity action")
 return warn, action
```

## box
### 如果门槛设错了，会发生什么
门槛太低，告警会像噪声一样刷屏，最后没人理；门槛太高，真正要扩容的时候已经晚了。更糟的是，若门槛只盯 GPU util，排队和长尾会被藏起来，直到用户已经先看到慢请求或超时。告警的目标把你推到正确的下一步。

## box info
### 练习时怎么用它
你可以把自己服务的三条真实告警写成“预警 / 动作 / 建议动作”三列。只要列得出这三列，你就已经把“发现问题”往前推到“知道该做什么”了。

# 走读示例 {#walkthrough}
<!-- tag: Walkthrough -->
<!-- layout: grid2 -->

## box
### 把一张看板读成一条判断链
假设上午 10 点，线上看板突然出现一组变化：TTFT p95 从 1.1 秒涨到 3.6 秒，TPOT p95 还在 32 毫秒，queue_wait p95 变成 2.7 秒，GPU util 只有 58%，pending replica 却没有增加。这个时候最容易犯的错，是直接说“模型慢了”或者“副本少了”，然后马上去调大实例数。
更稳的读法，是先把这几个数字排成顺序：入口在等，路由和接纳没有把请求及时推到空闲副本，或者副本虽在，但上游已经把队列堆满了；引擎后半段并没有明显变慢，因为 TPOT 还平稳。这个组合更像前半段的问题，而不是 decode 的问题。
再往下看，如果长 prompt 的比例上升，prefill 就可能把首 token 往后推；如果长 prompt 没变，queue_wait 却一直上升，那就该查 admission、路由权重和副本健康。只盯 GPU util，会把排队和前半段的代价藏起来；只盯平均延迟，会把少数被卡住的请求埋掉。

## box warn
### 这条链最容易误导什么
- 把排队看成模型慢。
- 把首 token 慢看成 decode 问题。
- 把平均值当成容量结论。
- 把副本数当成唯一动作，忽略路由和接纳。

| 看板组合 | 先读什么 | 更像哪一层 | 下一步动作 | 如果看错会怎样 |
| --- | --- | --- | --- | --- |
| TTFT p95 高，TPOT 平稳，queue_wait 也高 | 排队和路由 | 入口 / 调度层 | 看 admission、route_wait、pending replica | 把问题误判成引擎算力，最后扩错地方 |
| TTFT p95 高，TPOT 平稳，prompt 长度明显上升 | prefill | 引擎前半段 | 看 prompt 分布、cache 命中、裁剪策略 | 把短时输入波动当成容量不足 |
| TTFT 正常，TPOT p95 变差，GPU util 也上升 | decode 和 batch | 引擎后半段 | 看输出长度、batch 混合、显存压力 | 只加入口副本，后半段还是堵 |
| 平均值还行，p99 已经变差 | 尾部 | 容量与 fleet | 看长尾请求、热点租户、错误预算 | 把少数慢请求压成“噪声” |

排障决策伪代码复制

```python
def diagnose_slo(ttft_p95, tpot_p95, queue_wait_p95, gpu_util, prompt_shift, pending_replica):
 if ttft_p95 > 2.0 and tpot_p95 <= 0.05:
 if queue_wait_p95 > 1.0 or pending_replica > 0:
 return "先看接纳、路由和排队"
 if prompt_shift:
 return "先看长 prompt、prefill 和 cache 命中"
 return "先看入口和副本健康"
 if ttft_p95 <= 2.0 and tpot_p95 > 0.05:
 return "先看 decode、batch 混合和输出长度"
 if gpu_util > 0.85 and queue_wait_p95 > 0.8:
 return "容量已经接近边界，先看尾部和扩容"
 return "先看分位数、长尾和租户分布"
```

## box
### 错误后果
如果你在这一类场景里只看 GPU util，通常会把排队、路由不均和前半段代价都漏掉。结果是：告警晚了，扩容晚了，用户已经先看到首 token 卡住或者请求超时。真正该修的，可能是接纳阈值、路由权重、prompt 长度控制，甚至是副本健康检查。

## box info
### 练习时怎么用这条链
你可以把自己服务里的一条慢请求，按这张表重写一遍：先写看板上的四个数，再写它更像哪一层，最后写下一步动作。只要这三步能写顺，容量讨论就不容易飘到平均值上去了。

### 读完后，先去看什么

如果你只看一个数，最容易看偏；如果你把一条慢请求和一张容量图一起看，判断会稳很多。

## box warn
### 这里先不追
先别钻具体看板的字段名、trace backend 的存储格式、某个框架的默认阈值，或者自动伸缩控制器的内部实现。先把“看见什么就该去看哪一层”这件事做顺。

观测查询示意复制

```text
TTFT p95:
histogram_quantile(0.95, rate(ttft_seconds_bucket[5m]))

TPOT p95:
histogram_quantile(0.95, rate(tpot_seconds_bucket[5m]))

Reject rate:
sum(rate(request_rejected_total[5m]))

Queue pressure:
sum(queue_depth) + sum(pending_replica)
```
