# 概念图 {#map}
<!-- tag: route -->

**入口层**
OpenAI 兼容 HTTP 入口、鉴权、流式返回、请求排队和拒绝策略。


**编排层**
Ray Serve deployment、replica、router、autoscaling、handle 和观测指标。


**执行层**
LLMServer 或等价服务单元，负责把模型配置、资源和引擎参数拼起来。


**后端层**
vLLM、SGLang、TensorRT-LLM、其他推理栈，负责真正的 prefill、decode 和 cache。

## box soft
**直觉：**入口层决定“请求怎么进来”，编排层决定“请求交给谁”，执行层决定“副本怎么组织资源”，后端层决定“token 怎么吐出来”。你先把这四层拆开，后面的参数才不会混成一团。

# 入口层：先看请求怎么进来 {#ingress}
<!-- tag: ingress -->
<!-- layout: grid2 -->

## box
### 入口层读什么、写什么
入口层读 HTTP 请求、头部、认证信息、上下文参数和流式开关；写的是响应流、队列状态、拒绝结果和可观测事件。它还会影响后续调度，因为它知道请求是短问答、长文档，还是结构化生成。

## box info
### 为什么入口会先成为瓶颈
高并发下，HTTP 连接、事件循环、序列化和流式转发都会占 CPU。很多人只盯 GPU 利用率，但真正先满的可能是入口副本。

## box soft
对象
读什么
写什么
坏了会怎样


OpenAiIngress
请求体、流式标记、认证、租户标签
排队决策、路由信号、响应流
TTFT 拉长，或者一开始就被误拒绝


Admission control
当前队列、ongoing requests、副本健康状态
接收、等待、拒绝、限流
排队无限长，最后一起超时
这里先记住一件事：入口层不是“转发一下就完了”。它会改变队列长度、等待时间和用户感受到的首 token 时间。

入口判断伪代码复制

```python
def admit(request, queue_depth, ongoing, healthy_replicas):
 if not healthy_replicas:
 return "reject: no healthy replica"
 if request.is_streaming and queue_depth > 200:
 return "reject: protect TTFT"
 if ongoing > healthy_replicas * 4:
 return "wait: backpressure"
 return "accept: route now"
```

## box warn
**你现在应该能做什么：**看到 p95 TTFT 变差时，先问“入口是否先满了”，而不是直接去改 GPU 参数。

# 引擎层：副本背后到底占了什么 {#engine}
<!-- tag: engine -->
<!-- layout: grid2 -->

## box soft
### LLMServer 是一个执行单元
它把模型来源、权重位置、engine kwargs、加速卡类型、并行方式和副本生命周期捆在一起。一个副本可能背后占多张 GPU，也可能跨节点。副本数、tensor parallel、pipeline parallel 和 GPU worker 数量，不是同一个旋钮。

## box
### 你该盯的是副本背后的资源
你要看每个副本到底分到多少显存、KV cache、prefill 能力和 decode 吞吐。副本数增多但每个副本都很小，未必比少量大副本更稳。

# 一次请求走读 {#walkthrough}
<!-- tag: trace -->

## box soft
想象一个团队已经在 Ray Serve LLM 上挂好了一个 OpenAI 兼容端点。入口只有 1 个副本，后面有 2 个 LLMServer 副本，每个副本背后各占 2 张 GPU。现在来了一个普通聊天请求：1200 个输入 token，180 个输出 token，来自两个租户之一。
这条请求不是一路直走。入口层先读租户、流式标记和当前队列；编排层再读健康副本、前缀相似度和 ongoing requests；执行层最后读模型配置、KV cache 和引擎参数。每一层都会写一个不同的状态：队列状态、副本选择、响应流和扩容信号。
如果你把它误当成“HTTP 一直往后转发”，你会在排障时漏掉前半段。TTFT 变差时，入口队列可能先满；TPOT 变差时，decode 或 batch 可能先出问题；冷启动明显时，部署层才是主因。

路径
读什么
写什么
坏了会怎样


客户端 → 入口
请求体、租户、流式标记
队列位置、接纳结果
TTFT 一开始就上去


入口 → 路由
队列深度、健康副本、前缀相似度
目标副本、等待/拒绝决策
缓存局部性和负载均衡打架


路由 → LLMServer
模型配置、资源绑定、引擎参数
推理任务、GPU 绑定
副本看着在线，实际上不能用


引擎 → 客户端
prefill 结果、decode 输出、stream 状态
token 流、结束信号
首 token 正常，后续 token 卡住

请求分层判定伪代码复制

```python
def choose_layer(ttft_p95, tpot_p95, queue_wait_p95, cold_start_rate):
 if ttft_p95 > 2.0 and tpot_p95 <= 0.05:
 if queue_wait_p95 > 1.0:
 return "ingress or routing"
 if cold_start_rate > 0.1:
 return "deployment rollout"
 return "prefill or backend handoff"
 if tpot_p95 > 0.05:
 return "decode or backend"
 return "healthy enough to keep watching"
```

## box info
**你现在应该能做什么：**拿到一条慢请求，先把它归到入口、路由、执行或部署层中的某一层，再决定下一步是限流、扩容、回滚还是查后端。

# 路由与账本：最空副本和最合适副本不是一回事 {#routing}
<!-- tag: routing -->
<!-- layout: grid2 -->

## box
### 为什么缓存亲和有用
如果一批请求共享系统提示词、few-shot 示例或前缀，送到同一副本可能提高 KV cache 命中率，少做一遍 prefill。但这只是局部最优，很多时候还要看整体队列。

## box warn
### 什么时候要放弃缓存亲和
如果某个副本已经排队很长，继续追求缓存命中会把 TTFT 伤得更厉害。此时应该先保护延迟，再谈局部性。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

账本项
读什么
写什么
坏了会怎样


前缀亲和
前缀相似度、缓存热度
路由到同一副本
局部命中高，但某个副本排队爆炸


负载均衡
队列深度、ongoing requests
路由到更空的副本
缓存局部性差，重复 prefill 变多


扩容信号
持续时间窗口内的负载和等待

## box soft
**你现在应该能做什么：**解释“最空副本”和“最高缓存命中副本”为什么可能不是同一个，并能说出什么时候要选前者、什么时候要选后者。

# 升级与回滚：部署形态一变，状态也要跟着变 {#release}
<!-- tag: release -->

## box soft
Ray Serve LLM 的升级不是只换一个镜像。你可能会同时改模型权重、engine kwargs、GPU 绑定、replica 上限、路由规则和观测阈值。任何一项变了，系统的状态账本都会变。
最容易出问题的是“看起来已经发布成功，实际上缓存还没热起来”。这时 TTFT、错误率和冷启动时间可能一起抖。做升级时，最好把旧副本保留到新副本完成一次真实流式请求，再开始放量。

状态
发布前
发布后
读 / 写
坏了会怎样


镜像标签
旧版本
新版本
读 CI 产物；写部署引用
版本和权重没对齐


模型权重
旧权重已热
新权重加载中
读存储位置；写显存和缓存
冷启动拖长 TTFT


引擎参数
旧 tensor parallel
新 tensor parallel
读配置；写 worker 绑定
吞吐变了，但没人知道为什么


路由规则
旧副本仍在接流量
新副本开始接流量
读健康状态；写流量切换
一个副本排队，另一个副本闲着

滚动升级伪代码复制

```python
def rollout(new_image, new_weights, new_kwargs):
 keep_old_replicas_warm()
 new_replica = launch_one_replica(new_image, new_weights, new_kwargs)
 wait_for_streaming_canary(new_replica)
 if canary.ttft_p95 > threshold or canary.errors:
 rollback_to_previous()
 return "rollback"
 shift_traffic_gradually()
 drain_old_replicas()
 return "done"
```

## box warn
**你现在应该能做什么：**看到一次发布后指标变坏，先判断是镜像、权重、参数、路由还是缓存热身出了问题，而不是把所有锅都丢给“模型不稳定”。

# 部署案例：同一个模型可以有四种落点 {#cases}
<!-- tag: cases -->
<!-- layout: grid2 -->

## box
### 案例 A：单机 vLLM 或 SGLang
适合试验、内部工具和低并发验证。它们都能提供 OpenAI 兼容入口，但基本的服务治理、灰度、队列上限和多模型编排还得自己补。

## box
### 案例 B：Ray Serve LLM + vLLM
适合多副本、多模型、需要自动伸缩和统一观测的团队。这里 Ray Serve 负责编排，vLLM 负责推理。

## box
### 案例 C：Kubernetes + KubeRay + RayService
适合已经有 K8s 平台的组织。RayService 把集群和 Serve application 放进同一个声明式部署里，便于升级和标准化运维。

## box
### 案例 D：TensorRT-LLM 性能优先
适合 NVIDIA GPU 栈、固定模型和性能优先目标。它更像“性能路径”，主线不在通用服务编排。

# 故障诊断：先看哪一层 {#diagnostics}
<!-- tag: diagnosis -->

<!-- interactive: ['terminal-lab'] → keep as HTML -->

## box
### 可见症状 → 先查什么
- TTFT 上升但 decode 正常：先看入口、队列、prefill。
- decode 变慢但首 token 正常：先看批处理、输出长度和显存。
- GPU 利用率不高却排队：先看路由偏斜、接纳和副本健康。

## box warn
### 不要直接跳到“加 GPU”
很多问题是入口层、排队、路由或冷启动造成的。你先分层定位，才能知道扩的是入口副本、模型副本，还是根本不该扩。

# 部署账本 {#ledger}
<!-- tag: ledger -->
<!-- layout: grid2 -->

## box soft
### 先看一个真实切换场景
某团队把单机 vLLM 迁到 Ray Serve LLM，第一版灰度只换了镜像，没改路由和热身策略。健康检查都过了，控制台也显示新副本已经上线，但一旦切过去，TTFT 就开始抖，队列跟着涨，旧副本还在吃一部分请求。问题“入口、路由、热身、切流”没有一起变。
这时最值钱的动作先把账本摊开：谁在接真实流量，谁还在预热，谁已经标成新版本，谁背着旧权重，谁还没从旧路由里退出。账本不清，放量和回滚都会乱。
你可以把它想成四笔钱：入口在记接纳，编排在记分发，执行层在记资源，发布层在记版本。只要其中一笔没对齐，线上就会出现“看起来已经成功，实际上还在抖”的假稳定。

## box info
### 排障时先问四个问题
- 新副本是不是已经开始吃真实请求，而不只是通过了健康探针？
- 旧副本是不是还在保留，方便你一键退回？
- 流量切换是不是太快，导致缓存还没热起来就被放大了？
- 指标变坏时，你是先回滚、限流，还是只是继续等？
这四个问题能把“上线成功”和“线上可用”分开。前者只说明配置能被接受，后者才说明真实请求能稳地流过去。

层
读
写
冷启动 / 故障症状


入口层
请求、租户、流式标记、队列深度
接纳、排队、拒绝、流式输出
TTFT 上升，或者请求直接被拦


编排层
健康副本、前缀相似度、ongoing requests
路由选择、扩缩容信号
一边排队，一边有副本闲着


执行层
模型配置、权重、engine kwargs
GPU 绑定、执行状态、token 流
加载慢、OOM、decode 不稳


发布层
旧镜像、旧权重、旧阈值
新镜像、新权重、新路由
新副本没热起来，旧副本又被切太快

## box
### 上线动作
最稳的做法先保留旧副本，再让新副本吃一小段真实流量。这里至少要覆盖一个长输入请求、一个流式请求和一个普通短问答。长输入能测 prefill，流式请求能测 TTFT 和持续输出，短问答能测基本路径有没有被误伤。
如果新副本第一次就明显变慢，先别急着怪模型。先看是不是权重还在拉取、引擎参数没对齐、GPU 绑定没成功，或者入口把大流量全压到了单个副本上。很多“模型很慢”的判断，其实是发布流程和路由策略先出错了。
这一步的目标很简单：让新副本经历一轮真实请求，确认它不只是活着，而且真的能接住流量、按正确参数工作、按预期速度返回结果。

## box warn
### 回滚条件
一旦 TTFT 持续高、错误率上升，或者旧副本已经开始退出但新副本还在抖，就不要继续赌。回滚不是认输，而是在账本没对齐时把损失关住。
如果问题只出在热身，就先把旧副本留着，把切流节奏放慢，再单独查权重、缓存和路由。别把“再等等看”误当成处理方案。
发布门禁的意义把“能启动”与“能交付”分开。启动成功只能证明资源和配置没有立刻报错，交付成功才意味着真实请求路径已经稳定。

```python
def gate_release(ttft_p95, error_rate, warmup_ok, old_replicas_alive):
 if not warmup_ok:
 return "hold: keep old traffic"
 if ttft_p95 > threshold_ttft or error_rate > threshold_error:
 return "rollback: keep old replicas warm"
 if not old_replicas_alive:
 return "hold: do not drain the last known-good path"
 return "shift traffic gradually"
```

## box soft
这段伪代码想表达的“发布门禁要和状态账本绑定”。如果热身还没完成，就不该切；如果旧路径已经没了，就不该继续试；如果 TTFT 和错误率都在抬头，就该先回滚，再查是哪一层没有对齐。
**你现在应该能做什么：**拿到一次灰度异常，先把它拆成“新副本是否可用、旧副本是否该退、流量是否该切慢”三件事，再决定继续放量还是退回。
