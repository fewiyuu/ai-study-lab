学习目标

goal

先把故障分层

学完后，你要能把一个 ticket 分成入口接纳、路由副本、执行引擎、发布回滚四层。TTFT 变差，不等于模型一定变慢；TPOT 变差，也不等于入口一定有问题。先分层，后下结论。

再把框架放回职责边界

你要能把 vLLM、SGLang、TensorRT-LLM、Ray Serve LLM 放进同一张图里，不再用“哪个最好”这种空问题判断，而是按服务编排、结构化生成、极致性能、平台交付来选。

最后把判断落到动作

你要能写出一个回滚条件、一个灰度门禁和一个上线检查表。能看见问题，还要能知道下一步该限流、回滚、预热，还是换一层去查。

你现在应该能做什么：拿到一条生产故障，不先问“模型是不是坏了”，而是先问“它卡在入口、路由、执行，还是发布”。

故障地图

第一判断 入口或 prefill

读队列等待、请求大小和首 token 之前的耗时，写下一个检查点：接纳阈值、热身状态和预填充压力。

第一判断 decode 或 cache

读后续 token 间隔、batch size、输出长度和显存压力，写下一个检查点：decode 路径和缓存命中。

第一判断 接纳与路由

读副本健康、路由偏斜和 ongoing requests，写下一个检查点：是该改路由，还是该限流。

第一判断 发布节奏

读热身结果、灰度比例和旧副本保留状态，写下一个检查点：先保留旧路径，再判断是否继续放量。

你现在应该能做什么：看到一个慢请求时，先把它放进“入口、路由、执行、发布”四层中的某一层，再决定下一步查什么。

术语与误区

terms
术语 先这样理解 最常见的误读
TTFT 从请求进来，到第一个 token 出来之前的时间。 把它和总生成时间混成一件事。
TPOT 首 token 之后，每个 token 平均花多久。 以为 TTFT 好了，整体就一定好了。
prefill 先把输入上下文读进来并构造状态。 把它当成“只是读一下 prompt”。
decode 在已有状态上继续吐出后续 token。 把 decode 的慢归到入口。
KV cache 保存中间状态，减少重复计算。 以为只要开了 cache，排队和冷启动就会自己消失。
replica 服务的一个实例，可能背后挂着多张 GPU。 把 replica 直接等同于 GPU 数。

先记住这一句：OpenAI-compatible 只是入口协议接得上，不代表所有后端行为、缓存策略、流式细节都完全一样。

你现在应该能做什么：听到“TTFT 变差”时，先追问是入口、prefill、队列，还是热身；听到“TPOT 变差”时，先追问是 decode、batch，还是 cache。

故障诊断

diagnosis 
现象：p95 TTFT 明显变高，但 TPOT 基本不变。
先看：入口排队、预热状态、prefill 压力。
再看：是不是请求太长，或者新副本还没热起来。

第一层判断：先看什么

  • TTFT 先升，通常先看入口和 prefill。
  • TPOT 先升，通常先看 decode 和 cache。
  • 排队先涨，通常先看接纳、路由和副本健康。
  • 冷启动先涨，通常先看发布、热身和切流节奏。

第二层判断：别先做什么

别一看到慢就先加 GPU。很多问题出在入口、路由、热身、版本切换或缓存局部性上。你先分层，再决定是扩容、限流、改路由，还是回滚。

故障分层伪代码
def triage(ttft_p95, tpot_p95, queue_wait_p95, cold_start_rate, oom_rate):
    if ttft_p95 > 2.0 and tpot_p95 <= 0.05:
        if queue_wait_p95 > 1.0:
            return "ingress / queue / routing"
        if cold_start_rate > 0.1:
            return "rollout / warmup"
        return "prefill or admission"
    if tpot_p95 > 0.05:
        return "decode / batch / cache"
    if oom_rate > 0:
        return "capacity or model size"
    return "keep watching"

你现在应该能做什么：把一条 ticket 先归到“入口、路由、执行、发布”中的一层，再继续看细指标，不再靠猜。

故障案例

cases

案例 1：TTFT 高，TPOT 稳

这类情况常见于入口队列变长、请求太长、或者新副本热身没完成。你先看的是接纳、排队和 prefill，而不是先去改 decode 参数。

案例 2：首 token 还行，后续 token 变慢

这类情况常见于 decode、batch 混合、KV cache 压力，或者极长输出把后半段拖慢。你先看 TPOT、输出长度和缓存压力，再决定要不要调 batch 或换引擎。

案例 3：GPU 利用率没满，却开始排队

这类情况常见于路由偏斜、接纳策略过严、租户配额不平衡，或者某些副本没有真正接到流量。你先看路由和副本账本，而不是先看 GPU 面板。

现象 最先查什么 先做什么动作
TTFT 高、TPOT 稳 入口队列、prefill、热身 看能不能先限流、缩短队列或延长热身
首 token 快、后续 token 慢 decode、cache、batch 看 TPOT、输出长度和 cache 命中
排队上涨、GPU 利用率不高 接纳、路由、副本健康 先查路由偏斜，再看是否需要放量或限流
新副本上线后抖动 灰度、热身、旧副本保留 先保留旧路径，别急着把流量全切过去

你现在应该能做什么：看到一条故障描述时，先把它翻译成“症状、最先查的层、第一动作”三句话。

灰度账本

rollout

一条真实的切流现场

某团队把单机 vLLM 迁到 Ray Serve LLM。健康检查先通过了，控制台也显示新副本已经上线，只有一件事没变好：真实请求一切过去，TTFT 就抬头，队列也开始涨。问题不在“模型不能跑”，而在“热身、路由、切流和回滚”没有一起变。

这类故障最容易骗过人，因为它会先给你一个看起来正常的表象。新副本活着，旧副本也还活着，看板上甚至还有绿色勾。但如果你没把流量账本摊开，就会把“已启动”误当成“已可用”。

这张账本要记什么

它记的不是抽象“健康”，而是四个可核对的状态：新副本是不是已经吃到真实请求，旧副本是不是还在提供回退路径，路由是不是把流量切得太快，热身是不是已经让首 token 的代价降下来。账本不清，放量和回滚都会乱。

层 发布前 发布后 读什么 写什么 坏了会怎样
镜像与权重 旧版本已热 新版本加载中 镜像标签、权重位置、加载进度 显存占用、模型状态 冷启动把 TTFT 拉高
路由与副本 旧副本接流量 新副本开始接流量 健康状态、前缀相似度、队列深度 副本选择、流量比例、等待时间 新旧副本同时忙或同时闲
热身与缓存 缓存热、请求稳 缓存半热、首批流量已切 首 token 统计、cache 命中、prefill 代价 热身完成标记、放量节奏 看起来上线成功，实际上用户先遇到抖动
回滚与保底 旧路径还在 旧路径可能被提前清理 错误率、TTFT、队列、旧副本存活 回滚、限流、保留旧副本 一旦新副本抖了，却没有安全退路
灰度闸门伪代码
def gate_rollout(ttft_p95, queue_wait_p95, tpot_p95, warmup_ok, old_path_alive):
    if not warmup_ok:
        return "hold: keep old path"
    if ttft_p95 > 2.0 and queue_wait_p95 > 1.0:
        return "slow ramp: do not push traffic too fast"
    if tpot_p95 > 0.05:
        return "inspect decode/cache before cutting over"
    if not old_path_alive:
        return "hold: do not drain the fallback path"
    return "shift a little more traffic"

如果你继续放量，会发生什么

如果 TTFT 已经在抬头，你还继续把流量往新副本推，常见结果不是“系统更快了”，而是更多请求叠进队列，首 token 继续拖长。等你再想回头时，旧路径可能已经被你自己清空了，回退会比继续放量更慢。

如果你回滚太晚，会发生什么

回滚慢一步，错误率和等待时间都会继续烧错误预算。更麻烦的是，旧副本如果已经被 drain 掉，你失去的不是一个版本，而是一条安全退路。上线门禁的作用，就是把这类损失关在门里，而不是等它扩到用户那边。

你现在应该能做什么：拿到一次灰度异常时，先问“新副本真吃流量了吗、旧副本还在吗、热身结束了吗、切流太快了吗”，再决定继续放量还是回滚。

动作桥

operation

把一条慢请求变成一个决策

假设你在值班，图上出现的是这样的组合：TTFT p95 从 1.3s 升到 4.8s，TPOT 基本平稳，queue_wait 在涨，GPU util 只有 55%，同时新副本刚刚开始吃流量。这个组合说明问题不在 decode，也不在“GPU 不够忙”，而在入口接纳、路由、热身和切流节奏。

这时如果你还继续放量，队列会继续长，首 token 还会被拖慢；如果你先加 GPU，却没把路由和热身问题看清，新的副本也只是换一种方式继续排队。先把症状翻译成层，再做动作，才不会把资源花在错误方向上。

动作顺序要怎么排

第一步看队列和路由，确认新副本是不是在真实接流量；第二步看热身和 cache，确认首 token 的代价有没有真正降下来；第三步看旧路径是否还在，确认回滚和限流还有退路。只有这三步都说得通，才继续放量。

把看板翻成值班记录

值班时最容易犯的错，是把看板只读成“红了还是绿了”。更有用的读法，是把它翻成一句句可以执行的话。TTFT 在涨，说明先看入口和预热；queue_wait 在涨，说明先看接纳和路由；TPOT 在涨，说明先看 decode 和 cache；冷启动在涨，说明先保留旧路径，不要立刻清掉回退窗口。

这不是给看板增加一句漂亮解释，而是把它变成 runbook 的入口。谁接到报警，谁就知道先开哪几条曲线、先看哪段时间窗、先决定是限流、回滚、扩容，还是继续观察。动作如果不在这里被写清，后面的讨论就会停留在“感觉像模型慢了”这一类空话上。

你可以把这页想成一个最小闭环：症状进入，层级分开，动作排队，最后回到结果。这个闭环里最关键的不是“判断快”，而是“不要把错误动作先做了”。一旦把 GPU util 误当成唯一信号，或者把热身未完成误当成正常上线，你就会把真正该保护的退路提前清掉。

看板读数 你先写进记录 你先执行什么 这一步在防什么错
TTFT p95 连续上升 入口、预热、prefill 先限流或减慢切流 防止把入口拥塞误判成 decode 问题
queue_wait 持续上升 接纳、路由、副本健康 先查路由偏斜和副本分布 防止只看 GPU util 就匆忙加副本
TPOT 变差 decode、batch、KV cache 先看后续 token 的节奏，再调引擎 防止把后半段拖慢误成首 token 问题
新副本上线后抖动 热身、旧路径、灰度比例 先保留旧副本，降低放量速度 防止把回滚窗口自己清掉
runbook 判定伪代码
def runbook_step(ttft_p95, queue_wait_p95, tpot_p95, warmup_ok, rollback_path):
    if ttft_p95 > 2.0 and queue_wait_p95 > 1.0:
        return "先写进记录：入口排队。先做：限流、减慢切流、确认热身。"
    if tpot_p95 > 0.05:
        return "先写进记录：后续 token 变慢。先做：查 decode、batch、cache。"
    if not warmup_ok:

    if not rollback_path:

故障到动作伪代码
def next_action(ttft_p95, tpot_p95, queue_wait_p95, gpu_util, warmup_ok, old_path_alive):
    if ttft_p95 > 2.0 and tpot_p95 <= 0.05:
        if queue_wait_p95 > 1.0:
            return "先限流，先查接纳和路由"
        if not warmup_ok:
            return "先保留旧副本，延长热身"
        return "先看 prefill 和首包写出"
    if tpot_p95 > 0.05:
        return "先看 decode、batch 和 cache"
    if gpu_util < 0.6 and queue_wait_p95 > 0.8:
        return "先查副本分布和接纳，不要先加 GPU"
    if not old_path_alive:
        return "先恢复回滚路径，再放量"
    return "继续监控，不要直接放量"

失败后果

如果你把“TTFT 高、TPOT 稳”误判成 decode 问题，通常会先调错旋钮，排队还在涨，用户也还在等。相反，如果你先把队列和热身看清，再决定是否限流或回滚，损失会小得多。这个判断顺序就是这页真正要你带走的动作。

同样地，如果 queue_wait 明明在涨，你却只盯 GPU util，最后往往会把真正该改的路由权重和接纳阈值留在原地。结果是副本越加越多，排队却没有真的消失。看板不该只是一个“有没有事”的标签，它应该直接指向下一步检查。

练习时怎么用这段

你可以直接拿自己服务里的一个慢请求，按“症状、第一层、第一动作、错误后果”四列重写一遍。能把这四列写顺，说明你已经不只是知道名词，而是能把故障变成操作。

如果你愿意再往前一步，就把这四列搬到团队的值班手册里：每个症状都写清先看哪条曲线、先做什么动作、做完之后再看什么。这样下次有人再看相同的看板，就不用从零讨论。

你现在应该能做什么：看到一条灰度异常，先把它写成“症状 → 第一层 → 第一动作 → 错误后果”四步，再决定继续放量还是回滚。

选型清单

choice

先问问题层级，再问框架名

如果问题主要在服务编排、多副本、灰度和自动伸缩，先看 Ray Serve LLM。如果问题主要在结构化生成和受约束输出，先看 SGLang。如果问题主要在固定模型和 NVIDIA 栈上的极致性能，先看 TensorRT-LLM。如果你只是想先把服务跑起来并保持通用吞吐，vLLM 仍然是最直接的起点。

vLLM

适合：快速上线、OpenAI 兼容服务、通用高吞吐部署。

  • 先看在线 serving、调度、prefix cache 和指标暴露。
  • 先验证真实 prompt / output 分布，不要只看默认 benchmark。
  • 不擅长的地方：服务治理、灰度、多模型编排仍要补一层。

SGLang

适合：结构化生成、复杂推理程序、受约束输出。

  • 先看 runtime、OpenAI 兼容接口、router 和结构化输出能力。
  • 先验证你的程序式生成是否真能带来收益。
  • 不擅长的地方：团队要自己补足生产运维经验。

TensorRT-LLM

适合：固定模型、NVIDIA GPU 栈、追求极致性能。

  • 先看 engine build、executor、batching、KV cache 和部署流程。
  • 先验证驱动、CUDA、TensorRT 版本和回滚成本。
  • 不擅长的地方：模型变化频繁时，维护复杂度会上升。

Ray Serve LLM

适合：多模型部署、多副本、自动伸缩、灰度和 Python / Ray 生态。

  • 先看 LLMConfig、deployment、replica、autoscaling 和路由。
  • 先确认底层引擎是谁，再谈服务编排。
  • 不擅长的地方：它解决服务层问题，不会自动消除底层引擎瓶颈。
你的主问题 优先看谁 必须验证什么
先把服务跑起来，保持通用吞吐 vLLM TTFT、TPOT、队列、显存和错误率
结构化 JSON、工具调用、复杂约束输出 SGLang 结构化输出成功率、TPOT 抖动、复杂请求是否拖慢普通请求
固定模型、固定硬件、追求极限性能 TensorRT-LLM 构建流程、版本兼容、量化精度、吞吐回归
多模型、多副本、自动伸缩、灰度发布 Ray Serve LLM 副本级队列、路由、冷启动和集群运维

你现在应该能做什么：不再问“哪个框架最好”，而是先问“我的主问题在服务层、生成控制层，还是性能层”。

常见错误

debug

错误 1：只看 QPS 和平均延迟

QPS 一样，不代表输入长度和输出长度一样。TTFT、TPOT、queue wait 和 token 分布不拆开，容量评审会偏掉。

错误 2：看到慢，就直接加 GPU

很多故障出在接纳、路由、热身或切流节奏上。你如果先加 GPU，可能只是在把错误放大。

错误 3：把 OpenAI-compatible 当成完全一致

接口形状接上了，不等于后端行为、缓存策略、流式细节和错误码都一样。上线前要按版本复查。

错误 4：把 Ray Serve LLM 当成推理内核

Ray Serve LLM 解决的是服务编排、路由和伸缩，不是 token 采样和 GPU 内核优化。把层搞反，选型就会跑偏。

错误 5：发布只换镜像，不看热身和回滚

新副本上线只是启动成功，不等于真实请求已经稳定。灰度要保留旧副本，等 canary 过了再放量。

错误 6：benchmark 当成承诺

公开 benchmark 的硬件、模型、输入分布和并发未必等于你的生产流量。选型必须有自己的回归集。

bridge

如果你已经在值班

先开 TTFT、TPOT、queue wait、错误率和冷启动这几条曲线，再看路由和副本健康。先判断是哪一层，再决定是限流、改路由、预热，还是回滚。

如果你正在选型

先按主问题分层：服务编排、结构化生成、极致性能、平台交付。框架名只是答案的一部分，真正重要的是这层要读什么、写什么、验证什么。

看到什么 先打开什么 下一动作
TTFT 先升 入口、排队、prefill、热身 先限流或延长热身，再看是不是接纳阈值太松
TPOT 先升 decode、batch、KV cache 先看后续 token 的耗时，再决定要不要调 batch 或换引擎
排队先涨 接纳、路由、副本健康 先查路由偏斜，再判断是改路由还是扩副本
冷启动先涨 镜像、权重、热身、灰度 先保留旧路径，别急着把流量全切过去

先知道，先别追：这里先不追 CUDA kernel、完整 K8s 平台、benchmark 设计和训练产物更新。它们会在别的笔记里回来；这页先负责把线上故障和选型说清楚。

你现在应该能做什么：拿到一个生产问题，先写出“症状 → 第一层 → 第一动作”，再继续深入。

练习

diagnostic

作答目标

这组题不考背名词，重点是看你能不能把一个故障或者选型拆成层、状态和动作。优先写出你先看什么、为什么先看它，再写下一步动作。

检查重点

先确认对象有没有分层，状态有没有说清，选择有没有给出理由。开放题用评分要点自查，不要靠硬匹配整段自然语言。

一、症状识别

先把 TTFT、TPOT、排队和冷启动分开。

先判断这条症状是卡在入口、路由、执行，还是发布。别把首 token 之前和之后混成一层。
choiceTTFT

TTFT p95 上升、TPOT 基本没变时，先看哪层？

先看第一个 token 之前发生了什么。

true/false协议

判断：OpenAI-compatible 等于后端行为完全一致。

入口形状相同，不代表缓存、流式和错误码也相同。

fillqueue

填空：如果慢请求先卡在队列，你先看的词是 ______。

写你最先查的那个状态，不要写一长串分析。

synthesisdistinction

把 TTFT 慢和 TPOT 慢的判断顺序写出来

回答时至少写出：先看什么、后看什么、为什么这两个顺序不一样。

参考答案 / 评分要点

好答案会把“首 token 之前”和“首 token 之后”分开。TTFT 慢先看入口接纳、队列、prefill 和热身；TPOT 慢先看 decode、batch、KV cache 和输出长度。只写“模型慢了”不够。

二、路径与状态

把接纳、路由、副本健康和回滚放回同一张账本里。

如果 GPU 没满却排队，先看路由和接纳，不要先跳到模型本体。
choicequeue

GPU 利用率不高，但请求开始排队。你先看哪组因素？

资源没满，不代表请求已经走对了路。

filltradeoff

填空：缓存亲和和排队时长冲突时，先保护的指标是 ____。

这里先保用户最先感受到的那个时间。

synthesisrecovery

写一个你会放进值班手册的回滚条件

回答时至少说清：触发条件、先做什么动作、为什么不先盲目加副本。

参考答案 / 评分要点

好答案会同时写出阈值、动作顺序和理由。像“TTFT p95 连续 10 分钟高于阈值且队列不断增长，先限流或回滚，再查入口和引擎”这样的答案就比较完整。

三、框架选型

把框架名换成问题层级，再做判断。

先问“我到底要解决哪一层”，再问“哪个框架更顺手”。
choice编排

Ray Serve LLM 更像哪类问题？

看它是在管服务层，还是在管引擎层。

choice生成控制

SGLang 更像哪类问题？

先看它更靠近生成控制，还是更靠近平台调度。

choice性能

TensorRT-LLM 更适合哪种场景？

这类问题先看硬件和性能边界。

synthesis组合

给一个多模型 API 组合，并说明入口、编排、引擎、后端各自做什么

题目里有普通聊天、结构化 JSON 输出、企业租户隔离和灰度发布。回答时至少写出每一层的职责，再说明为什么要这样分。

参考答案 / 评分要点

好答案会把层级分清：Ray Serve LLM 解决服务编排、路由和伸缩；vLLM 解决通用推理；SGLang 适合结构化生成更重的场景；KubeRay / RayService 适合集群交付。只说“我选某个框架，因为它更快”不够。

四、上线复盘

把现象、门禁和回滚条件连起来。

先写阈值，再写动作，最后写原因。这个顺序比“我觉得”更有用。
synthesisrollback

写一个上线回滚阈值，并说明第一动作

回答时至少说清：什么指标触发、先做什么动作、为什么不该先盲目放量。

参考答案 / 评分要点

好答案会同时写出触发条件、动作顺序和理由。比如“TTFT p95 连续 10 分钟高于阈值且错误率上升，先保留旧副本并回滚，再查入口、路由和热身”就比较完整。
