读的时候盯住什么

每次看到一个并行策略，都回到三个问题：它切了什么，谁负责同步，出问题时哪一笔状态最先变脏。这个问题比记术语更稳定。

概念导图

trace

这页的锚点例子只有一个：同一个训练 step。把它走一遍，就能看见后面所有并行策略是怎么改写这条路径的。

forward 之前 参数在设备上，激活还没有出现。

读的是模型参数和输入；写出的是中间激活。若激活太大，后面的 backward 会先被显存卡住。

backward 之后 激活被反向使用，梯度开始累积。

读的是 forward 留下的激活；写出的是每个参数的梯度。若某层没有参与反向，后面同步时就会出现缺口。

同步之后 各 rank 的梯度必须表达同一次更新语义。

读的是每个 rank 的局部梯度；写出的是对齐后的梯度或梯度分片。若同步顺序错了，参数会开始分叉。

step 之后 参数更新，优化器状态前移。

读的是梯度、动量和学习率状态；写出的是新的参数和新的优化器状态。若只有部分 rank 更新，下一轮 forward 就不再等价。

checkpoint 之后 训练状态可以恢复到同一条路径上。

读的是参数、优化器状态、随机数和数据进度；写出的是可恢复快照。若少存一项，恢复后会跳样本、重复样本或改写学习轨迹。

术语与误区

glossary

这几个词最容易被说得像口号。这里先把它们压回训练循环里的具体位置。

术语 在本页里的意思 常见误读 现在先怎么用
rank 参与同一训练组的进程身份。它先是通信里的名字，再是硬件上的位置。 把 rank 直接当成 GPU 编号，或当成机器编号。 先确认“谁在跟谁通信”，再看物理卡怎么映射。
world size 参与同一训练组的进程总数。 把 world size 当成机器数，或者当成单卡 batch 的别名。 先数进程，再谈设备、机器和拓扑。
process group 一个通信小组。数据并行、张量并行、流水线并行可以各有自己的组。 以为所有并行维度都必须共享同一个组。 把“谁需要一起做通信”写成组边界。
collective 一组进程一起完成的通信动作，比如 all-reduce、all-gather、reduce-scatter。 把 collective 等同于 all-reduce。 先问动作，再问语义；动作相同不代表数据方向相同。
global batch 一次参数更新真正看到的总样本数。 把单卡 micro batch 直接当成 global batch。 先乘 micro batch、梯度累积和 data parallel world size。
checkpoint 训练状态的持久化快照，至少包含权重、优化器状态、随机数和数据进度。 只保存模型权重就够了。 恢复训练时，先考虑优化器、随机数和数据进度。

核心机制

state ledger

如果只记住一条经验，就记住训练 step 的顺序：forward 读参数，backward 读激活写梯度，sync 对齐各 rank 的更新方向，step 写回参数和优化器状态。很多并行策略只是把这条顺序拆开、挪位或分片。

单卡训练循环伪代码
for batch in loader:
    optimizer.zero_grad()
    logits = model(batch["x"])
    loss = criterion(logits, batch["y"])
    loss.backward()  # DDP / FSDP 的 hook 可能在这里触发通信
    optimizer.step()
    if should_save:
        save_checkpoint(model, optimizer, batch_cursor, rng_state)
步骤 读什么 写什么 谁会变脏 错了以后会看到什么
forward 参数、输入 激活 显存里的中间结果 backward 找不到需要的中间量，或者激活顶满显存
backward 激活、loss 梯度 参数对应的梯度缓冲 某些参数的 grad 为 None，或者局部梯度和别的 rank 不一致
gradient sync 局部梯度 对齐后的梯度或梯度分片 通信队列和 bucket 各 rank 参数开始分叉，下一轮 forward 不再等价
optimizer step 梯度、动量、学习率状态 新的参数和新的优化器状态 优化器内部状态 loss 看起来在动，但恢复后轨迹变了，或只有部分 rank 更新
checkpoint 参数、优化器、随机数、数据位置 持久化快照 磁盘上的训练状态文件 恢复后重复样本、跳样本或学习率曲线对不上

读源码或读日志时，先抓四个落点：参数在哪一步被读，梯度在哪一步被聚合，优化器状态在哪一步被写，checkpoint 哪一步把数据进度一起存下。把这四个点串起来，很多故障就会变成具体的对象顺序问题。

判断句

如果你能把一个故障映射到“哪一步读错、写错、没同步、没保存”，你就已经在做训练系统诊断了。框架名字只是把这些动作包起来的外壳。

数据并行的同步链

dp

数据并行里，每个 rank 都能本地完成 forward 和 backward，但 optimizer.step() 之前必须把梯度对齐到同一份模型语义。

最小例子可以看成 4 个 rank。假设它们在同一层上得到的本地梯度分别是 0.5、1.5、2.0、4.0，那么 sum all-reduce 的结果是 8.0，平均之后是 2.0。这里最重要的是 all-reduce 之后每个 rank 拿到同样的更新方向。若这一步没对齐，下一轮 forward 读到的参数就会分叉，训练曲线也会慢慢拉开。

$$ g_{\text{sync}} = \frac{1}{N}\sum_{r=0}^{N-1} g_r $$

其中 N 是参与同一 process group 的 rank 数，g_r 是第 r 个 rank 的局部梯度，g_{\text{sync}} 是同步后的梯度。若训练脚本使用的是 sum 再除以 N，语义仍然等价；关键是别把 sum 直接当成最终更新方向。

环节 本地发生什么 同步后要保证什么 错了会怎样
forward 每个 rank 读取同一份参数，但看到不同样本。 参数副本在 step 之前还应该保持等价。 如果前一轮 step 没同步，后续 forward 就不是同一个模型。
backward 每个 rank 生成自己的局部梯度。 局部梯度要汇总成一次更新的语义。 如果只让 rank0 更新，其它 rank 会停在旧参数上。
all-reduce 梯度在通信组里求和或平均。 每个 rank 得到同样的更新结果或同样的梯度分片。 如果 collective 顺序不一致，作业可能卡死而不是报 Python 异常。
optimizer step 读取同步后的梯度和优化器状态。 所有 rank 的参数保持一致。 如果只有一部分 rank step，下一轮再同步也救不回来。
4-rank 同步伪码
local_grad = backward(batch)
shared_grad = all_reduce(local_grad, op="sum")
shared_grad = shared_grad / world_size
optimizer.step(shared_grad)

为什么要在 step 之前同步

因为 step 会写回参数和优化器状态。只要 step 用的梯度不一样，后面的参数轨迹就不一样，之后再做多少次 all-reduce 都只能同步“已经走偏的模型”。

一个常见坏例子

如果 rank0 看起来 loss 降得更快，而其它 rank 没跟上，通常是同步点或者过程组出了问题。先查同步顺序，再查数据是否真的分开了。

FSDP2 为什么先压常驻

shard

数据并行能解决“算力分散”，但不自动解决“状态太大”。如果参数、梯度和 Adam 状态都复制到每张卡上，模型大一点就会被长期常驻显存卡死。FSDP2 的思路是把这些状态切成分片，让每张卡只背自己那份，真正要算的时候再临时聚合，算完再重新切回去。

FSDP2 解决的是状态驻留：参数、梯度和 Adam 状态不再长期完整地躺在每张卡上，而是按分片保存。forward 前把当前模块需要的参数 all-gather 到位，backward 后把梯度 reduce-scatter 回去，step 时更新本地分片上的优化器状态。这样省下的是常驻显存，代价是额外的通信、重分片和更讲究的包装边界。

FSDP2 的包装顺序
model = build_model()
model = fully_shard(model)
optimizer = AdamW(model.parameters(), lr=lr)
对象 DDP 里是什么样 FSDP2 里是什么样 如果顺序错了会怎样
参数 每张卡都保留完整副本。 常驻时只保留分片，需要计算时再临时聚合。 如果只包根模块，很多叶子模块根本没被分片。
梯度 每张卡都会看到完整梯度或等价副本。 反向后通常会把结果切回本地分片。 如果包装边界不对，某些梯度会保留得太久，峰值显存下不来。
优化器状态 每张卡都保留完整 Adam 状态。 每张卡只维护自己那份状态。 如果在 fully_shard 之前创建优化器，优化器会抓到错误的参数对象。
通信 梯度同步集中在 backward 末尾。 通信被拆进 forward 前和 backward 后。 如果你把它当成“无通信版本的 DDP”，峰值和延迟都会判断错。

为什么常说先压常驻

真正先把模型顶爆的，往往是参数、梯度和优化器状态长期都躺在显存里。FSDP2 先处理这个账本，再去谈吞吐是否要继续调。

读到 reshard_after_forward 时看什么

这个开关本质上是在“显存峰值”和“重复聚合”的成本之间做交换。留得更久，峰值更高；切得更快，通信更多。后面 06 单元会把这笔账再展开。

从同步到分片

transition

很多人学到这里会问：既然数据并行已经把梯度同步好了，为什么还要再学 FSDP2？因为它解决的是不同问题。数据并行保留完整副本，FSDP2 把参数、梯度和优化器状态切成分片。

把两者连起来看，通信的形状就更清楚了。数据并行的 all-reduce 关心同一层梯度的求和或平均，FSDP2 的 all-gather 和 reduce-scatter 关心参数和梯度在“临时使用”和“长期保存”之间的往返。它们都通信，但服务的账本不同。一个回答“这一轮怎么同步”，另一个回答“这份状态怎么装进显存”。

问题 数据并行怎么答 FSDP2 怎么答 你该先盯什么
训练目标是否一致 用 all-reduce 把梯度对齐，保证同一次更新语义。 继续保持这一点，但把状态拆成分片保存。 先看同步点是否在 optimizer.step() 之前完成。
状态是否放得下 完整副本常驻显存，模型大时会被状态占满。 只保留自己那份分片，需要时再临时聚合。 先拆参数、梯度和 Adam 状态，再看激活是否仍是瓶颈。
通信是为了什么 为了把多卡看到的局部梯度变成同一份更新方向。 为了让参数和梯度在“可算”和“可存”之间切换。 别把两类通信混成一类，它们的触发时机不同。
失效时最先坏哪里 梯度不同步会导致模型副本开始分叉。 包装边界错会导致显存峰值降不下来，或优化器抓错参数对象。 先查是语义错、体积错，还是生命周期错。
两条路径的对照伪码
# 数据并行
loss.backward()
grad = all_reduce(grad, op="mean")
optimizer.step()

# FSDP2
params = all_gather(shard)
loss = forward(params)
loss.backward()
grad = reduce_scatter(grad)
optimizer.step(shard)

这段对照想让你看见什么

数据并行先保证“同样的梯度”，FSDP2 再保证“同样的参数语义只在需要时出现”。它们解决的是前后两个阶段的问题，所以不能互相替代成一句“都在省显存”的空话。

实践时先从哪儿查

如果你读源码或读日志，先查同步点是在哪里发生的，再查参数对象是何时被包装的。很多 FSDP2 的 bug 不是算法错，而是对象创建顺序错。

扩展案例

case

先看一个足够小、但不失真的例子：每张卡 micro batch = 2，梯度累积 = 4，数据并行 world size = 4。一次参数更新看到的总样本数是多少，直接决定了你怎么理解“扩卡以后训练语义变了没有”。

$$ G = m \times a \times d $$

其中 G 是 global batch，m 是单卡 micro batch，a 是梯度累积步数，d 是数据并行 world size。上面的例子得到 G = 2 \times 4 \times 4 = 32。

这个数为什么要算

如果你只把 world size 从 1 提到 4，而其他值不动，更新语义也会变大。很多“扩卡后训练不稳”的问题，先不是通信问题，而是 global batch 已经换了。

读数的时候别漏掉累积

梯度累积改变的是更新频率，不是“有没有同步”的要求。它把多个 micro batch 合并成一次更新，但每次更新前后仍要维持正确的同步语义。

策略 主要切什么 先解决什么 代价在哪里 什么时候先看它
数据并行 数据批次 把计算分散到多卡 梯度同步和 global batch 变化 模型能放下，只是想更快或更稳
FSDP2 参数、梯度、优化器状态 把状态驻留显存压下去 通信和重分片成本 状态本身已经顶满显存
张量并行 层内张量维度 把单层也切开 层内通信和实现复杂度 单个矩阵或 attention head 已经太大
流水线并行 模型深度 把前后层分到不同卡 micro-batch 调度和空泡 模型太深，单卡执行链太长

boundary

别急着讨论 bucket size、混精 loss scaling、pipeline bubble 或 optimizer offload。它们都要等你能说清 step 的读写顺序之后再看。

回来的位置

现在先完成的动作

只要能解释“为什么这一步会卡显存、卡通信或卡恢复”，就可以进入下一个单元了。

恢复链

recovery

checkpoint 需要保存的是一整套训练状态，不是单一权重。训练跑久了，真正会把你卡住的往往是中途断电、作业被抢占、节点挂掉之后还能不能从同一条轨迹接回去。

最小失败场景很常见：只存模型权重，没存优化器状态和数据位置。恢复后看起来还能继续跑，loss 也可能还会下降，但训练轨迹已经变了。Adam 的动量没接上，学习率调度器会回到错误的步数，数据加载器也可能从旧位置重新读样本，最后你会看到重复样本、跳样本或者曲线拐点不对。

保存项 为什么要存 恢复时接不上会怎样 下一步查什么
模型参数 这是训练的主体状态，没有它就没有同一份模型。 恢复出来的是空壳，或者完全不是原来的函数。 先确认参数是同一版本、同一形状、同一包装边界。
优化器状态 动量、二阶矩和其他内部历史会改变下一次更新。 loss 可能还能动，但曲线拐点和收敛速度会变。 先查 optimizer 的 state_dict 是否随权重一起保存。
学习率调度器 学习率在不同 step 可能已经不是初始值。 恢复后学习率跳回去，训练像被重置到早期阶段。 先查 scheduler 记录的步数是否一致。
随机数状态 dropout、采样和数据增强都依赖它。 恢复后同一轮训练的噪声图案变了，结果难以复现。 先查 RNG state 有没有单独保存。
数据进度 这决定你恢复后从哪一个 batch 接着读。 样本会重复、跳过，或者一个 epoch 被拆成两半重来。 先查 batch cursor、sampler 状态和 shard 位置。
恢复顺序伪码
state = load_checkpoint(path)
model.load_state_dict(state["model"])
optimizer.load_state_dict(state["optimizer"])
scheduler.load_state_dict(state["scheduler"])
set_rng_state(state["rng"])
resume_from(state["cursor"])

恢复顺序为什么重要

先恢复模型，再恢复优化器，再恢复调度器和随机数，最后恢复数据位置。顺序错了，表面上看还是“能跑”，实际上每个环节都可能在不同的训练时刻接上。

恢复失败时先看什么

先看缺的是哪一类状态，再看数值精度。训练系统里很多“恢复后效果变了”的问题，根子都在恢复轨迹不完整。

从实验到选型

selection

读实验表或者比较不同训练栈时，先看模型多大、显存多紧、通信链路有多贵、作业能不能容错，再决定用数据并行、FSDP2、张量并行还是流水线并行。

同一条“扩卡”请求，可能对应完全不同的系统改法。模型已经能放下，只想更快，就先看数据并行和通信优化；状态本身放不下，就看 FSDP2；单层矩阵或 attention head 放不下，就看张量并行；模型深度太长、单卡执行链太重，就看流水线并行。你要从症状回到账本，再决定该改哪一层。

你先看到的信号 优先看的策略 先别急着做什么 为什么这样排
模型能放下，但 step time 太慢 数据并行 + 通信优化 不要先上复杂分片 这里先检查梯度同步、输入管线和长尾 rank，很多问题只是通信和调度没调好。
参数和 Adam 状态本身已经挤爆显存 FSDP2 不要只加 GPU 数 副本太多时，加卡只会把同样的常驻状态复制更多份。
单层矩阵或 attention 维度过大 张量并行 不要只盯 batch size 问题已经发生在层内维度，不是 batch 还不够大。
模型太深，前后层串得太长 流水线并行 不要只做参数分片 这里的瓶颈是深度和调度，不只是状态体积。
训练经常被抢占或挂掉 checkpoint + 容错设计 不要只看单次收敛 长任务要先能恢复，再谈最终吞吐。

选型时最该问的三句话

第一句，瓶颈在显存还是在时间。第二句，问题在状态驻留还是在模型内部切分。第三句，失败后是需要更强的恢复，还是需要换一种训练结构。只要这三句能答出来，很多“应该用什么框架”的问题就会自己缩小。

常见错误

debug

下面这些是最容易把训练系统带歪的操作错误。先看症状，再看下一步。

step 18
all ranks finish backward
gradients reduced
optimizer.step() runs on every rank
checkpoint saved
症状 更可能的原因 先查什么
各卡 loss 开始分叉 梯度没同步，或者只有部分 rank 做了 step 先看 backward 和 optimizer step 的顺序，再看 process group 是否一致。
GPU 都在忙，吞吐却没涨 通信占比太高，或数据加载把前向卡住了 先看通信、输入管线和长尾 rank，再看模型名。
显存降了，step 反而更慢 分片带来的通信和重分片开销把收益吃掉了 先看 FSDP2 的通信路径，再看 batch 和激活是否需要重配。
恢复后样本重复或跳过 checkpoint 少存了数据进度、随机数或优化器状态 先核对“保存了什么”，再核对“恢复从哪里接上”。

排查顺序

先判断是同步错了、状态切错了，还是保存错了。只要把故障落回 step 账本，很多看起来像“框架问题”的事都会变成具体检查项。

检查重点

重点看你能不能把一个扩展策略还原到训练循环，而不是只说“用了某个框架所以会更快”。
