先抓主线

course

这门课讲的是训练系统的 runtime：当模型、激活、梯度和优化器状态不再适合单卡时，怎样切分、怎样通信、怎样恢复。第 01 单元先把单卡训练循环看清，再把后面的并行策略放回同一张状态账本里。

读的时候盯住什么

每次看到一个并行策略，都回到三个问题：它切了什么，谁负责同步，出问题时哪一笔状态最先变脏。这个问题比记术语更稳定。

术语与误区

glossary

这几个词最容易被说得像口号。这里先把它们压回训练循环里的具体位置。

核心机制

state

如果只记住一条经验，就记住训练 step 的顺序：forward 读参数，backward 读激活写梯度，sync 对齐各 rank 的更新方向，step 写回参数和优化器状态。很多并行策略只是把这条顺序拆开、挪位或分片。

代码
for batch in loader:
 optimizer.zero_grad()
 logits = model(batch["x"])
 loss = criterion(logits, batch["y"])
 loss.backward() # DDP / FSDP 的 hook 可能在这里触发通信
 optimizer.step()
 if should_save:
 save_checkpoint(model, optimizer, batch_cursor, rng_state)

判断句

如果你能把一个故障映射到“哪一步读错、写错、没同步、没保存”，你就已经在做训练系统诊断了。框架名字只是把这些动作包起来的外壳。

数据并行的同步链

dp

数据并行里，每个 rank 都能本地完成 forward 和 backward，但 optimizer.step() 之前必须把梯度对齐到同一份模型语义。

$$ g_{\text{sync}} = \frac{1}{N}\sum_{r=0}^{N-1} g_r $$

其中 N 是参与同一 process group 的 rank 数，g_r 是第 r 个 rank 的局部梯度，g_{\text{sync}} 是同步后的梯度。若训练脚本使用的是 sum 再除以 N，语义仍然等价；关键是别把 sum 直接当成最终更新方向。

代码
local_grad = backward(batch)
shared_grad = all_reduce(local_grad, op="sum")
shared_grad = shared_grad / world_size
optimizer.step(shared_grad)

一个常见坏例子

如果 rank0 看起来 loss 降得更快，而其它 rank 没跟上，通常是同步点或者过程组出了问题。先查同步顺序，再查数据是否真的分开了。

FSDP2 为什么先压常驻

shard

数据并行能解决“算力分散”，但不自动解决“状态太大”。如果参数、梯度和 Adam 状态都复制到每张卡上，模型大一点就会被长期常驻显存卡死。FSDP2 的思路是把这些状态切成分片，让每张卡只背自己那份，真正要算的时候再临时聚合，算完再重新切回去。

FSDP2 的包装顺序复制

对象DDP 里是什么样FSDP2 里是什么样如果顺序错了会怎样
参数每张卡都保留完整副本。常驻时只保留分片，需要计算时再临时聚合。如果只包根模块，很多叶子模块根本没被分片。
梯度每张卡都会看到完整梯度或等价副本。反向后通常会把结果切回本地分片。如果包装边界不对，某些梯度会保留得太久，峰值显存下不来。
优化器状态每张卡都保留完整 Adam 状态。每张卡只维护自己那份状态。如果在 fully_shard 之前创建优化器，优化器会抓到错误的参数对象。
通信梯度同步集中在 backward 末尾。通信被拆进 forward 前和 backward 后。如果你把它当成“无通信版本的 DDP”，峰值和延迟都会判断错。

为什么常说先压常驻

真正先把模型顶爆的，往往是参数、梯度和优化器状态长期都躺在显存里。FSDP2 先处理这个账本，再去谈吞吐是否要继续调。

读到 reshard_after_forward 时看什么

这个开关本质上是在“显存峰值”和“重复聚合”的成本之间做交换。留得更久，峰值更高；切得更快，通信更多。后面 06 单元会把这笔账再展开。

从同步到分片

transition

很多人学到这里会问：既然数据并行已经把梯度同步好了，为什么还要再学 FSDP2？因为它解决的是不同问题。数据并行保留完整副本，FSDP2 把参数、梯度和优化器状态切成分片。

问题数据并行怎么答FSDP2 怎么答你该先盯什么
训练目标是否一致用 all-reduce 把梯度对齐，保证同一次更新语义。继续保持这一点，但把状态拆成分片保存。先看同步点是否在 optimizer.step() 之前完成。
状态是否放得下完整副本常驻显存，模型大时会被状态占满。只保留自己那份分片，需要时再临时聚合。先拆参数、梯度和 Adam 状态，再看激活是否仍是瓶颈。
通信是为了什么为了把多卡看到的局部梯度变成同一份更新方向。为了让参数和梯度在“可算”和“可存”之间切换。别把两类通信混成一类，它们的触发时机不同。
失效时最先坏哪里梯度不同步会导致模型副本开始分叉。包装边界错会导致显存峰值降不下来，或优化器抓错参数对象。先查是语义错、体积错，还是生命周期错。
代码
# 数据并行
loss.backward()
grad = all_reduce(grad, op="mean")
optimizer.step()

# FSDP2
params = all_gather(shard)
loss = forward(params)
loss.backward()
grad = reduce_scatter(grad)
optimizer.step(shard)

实践时先从哪儿查

如果你读源码或读日志，先查同步点是在哪里发生的，再查参数对象是何时被包装的。很多 FSDP2 的 bug 对象创建顺序错。

扩展案例

case

先看一个足够小、但不失真的例子：每张卡 micro batch = 2，梯度累积 = 4，数据并行 world size = 4。一次参数更新看到的总样本数是多少，直接决定了你怎么理解“扩卡以后训练语义变了没有”。

其中 G 是 global batch，m 是单卡 micro batch，a 是梯度累积步数，d 是数据并行 world size。上面的例子得到 G = 2 \times 4 \times 4 = 32。

读数的时候别漏掉累积

梯度累积改变的是更新频率，不是“有没有同步”的要求。它把多个 micro batch 合并成一次更新，但每次更新前后仍要维持正确的同步语义。

策略主要切什么先解决什么代价在哪里什么时候先看它
数据并行数据批次把计算分散到多卡梯度同步和 global batch 变化模型能放下，只是想更快或更稳
FSDP2参数、梯度、优化器状态把状态驻留显存压下去通信和重分片成本状态本身已经顶满显存
张量并行层内张量维度把单层也切开层内通信和实现复杂度单个矩阵或 attention head 已经太大
流水线并行模型深度把前后层分到不同卡micro-batch 调度和空泡模型太深，单卡执行链太长

恢复链

recovery

checkpoint 需要保存的是一整套训练状态，不是单一权重。训练跑久了，真正会把你卡住的往往是中途断电、作业被抢占、节点挂掉之后还能不能从同一条轨迹接回去。

保存项为什么要存恢复时接不上会怎样下一步查什么
模型参数这是训练的主体状态，没有它就没有同一份模型。恢复出来的是空壳，或者完全不是原来的函数。先确认参数是同一版本、同一形状、同一包装边界。
优化器状态动量、二阶矩和其他内部历史会改变下一次更新。loss 可能还能动，但曲线拐点和收敛速度会变。先查 optimizer 的 state_dict 是否随权重一起保存。
学习率调度器学习率在不同 step 可能已经不是初始值。恢复后学习率跳回去，训练像被重置到早期阶段。先查 scheduler 记录的步数是否一致。
随机数状态dropout、采样和数据增强都依赖它。恢复后同一轮训练的噪声图案变了，结果难以复现。先查 RNG state 有没有单独保存。
数据进度这决定你恢复后从哪一个 batch 接着读。样本会重复、跳过，或者一个 epoch 被拆成两半重来。先查 batch cursor、sampler 状态和 shard 位置。
代码
state = load_checkpoint(path)
model.load_state_dict(state["model"])
optimizer.load_state_dict(state["optimizer"])
scheduler.load_state_dict(state["scheduler"])
set_rng_state(state["rng"])
resume_from(state["cursor"])

恢复失败时先看什么

先看缺的是哪一类状态，再看数值精度。训练系统里很多“恢复后效果变了”的问题，根子都在恢复轨迹不完整。

从实验到选型

selection

读实验表或者比较不同训练栈时，先看模型多大、显存多紧、通信链路有多贵、作业能不能容错，再决定用数据并行、FSDP2、张量并行还是流水线并行。

你先看到的信号优先看的策略先别急着做什么为什么这样排
模型能放下，但 step time 太慢数据并行 + 通信优化不要先上复杂分片这里先检查梯度同步、输入管线和长尾 rank，很多问题只是通信和调度没调好。
参数和 Adam 状态本身已经挤爆显存FSDP2不要只加 GPU 数副本太多时，加卡只会把同样的常驻状态复制更多份。
单层矩阵或 attention 维度过大张量并行不要只盯 batch size问题已经发生在层内维度，不是 batch 还不够大。
模型太深，前后层串得太长流水线并行不要只做参数分片这里的瓶颈是深度和调度，不只是状态体积。
训练经常被抢占或挂掉checkpoint + 容错设计不要只看单次收敛长任务要先能恢复，再谈最终吞吐。

选型时最该问的三句话

第一句，瓶颈在显存还是在时间。第二句，问题在状态驻留还是在模型内部切分。第三句，失败后是需要更强的恢复，还是需要换一种训练结构。只要这三句能答出来，很多“应该用什么框架”的问题就会自己缩小。
