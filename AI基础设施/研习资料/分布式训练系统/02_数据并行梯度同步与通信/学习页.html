数据并行、梯度同步与通信

本页接着 01 的训练步账本往下看：同一个 forward、backward、step，被复制到多个 rank 之后，哪些对象还留在本地，哪些对象必须同步，哪一步一旦顺序错了就会挂住。这里先把数据并行讲透；FSDP2 的分片和 Megatron Core 的多维并行只先点出边界，留给后续单元展开。

学习目标

goal

这单元只盯三件事：把代码里的通信对象认出来，把一次训练步里同步发生的时刻认出来，把通信账单和性能瓶颈认出来。你能把同一段 DDP 代码拆成这三层，就算抓住了主线。

再认时序

能顺着 forward → backward → 同步 → optimizer.step() 追一遍本地读写和 collective 读写的切换，知道同步发生在 backward 之后。

先看主线

route

我们用一个最小锚点：两张 GPU、同一模型、不同 batch。每个 rank 先本地 forward/backward，再把梯度合成同一个更新方向。你要抓的是：哪些值可以各自算，哪些值必须在 backward 之后对齐。

这页先把 DDP 想成什么

先把 DDP 看成“复制训练步”。每个 rank 都有自己的 batch 分片，但最终要把梯度合成同一份更新。

先别追什么

先别把 bucket、通信重叠、混合精度、ZeRO 风格状态分片、pipeline bubble 当成起点。它们会在 03、04、05、06 里回来；本页先把同步边界钉住。

样本分片

data

数据并行要成立，除了模型副本要一致，样本也要切开。两张 GPU 用同一份数据，梯度会重复；两张 GPU 用不同分片，梯度才是在互补信息上做合成。这个差别很小，结果却很大，所以它是 DDP 里最值得先查的地方之一。

情况rank 0 看到什么rank 1 看到什么对梯度有什么影响常见症状
正确分片样本 id 0, 1, 4, 5样本 id 2, 3, 6, 7两个 rank 提供互补梯度，合成后更新更稳。loss 变化正常，验证结果和单卡不一样。
忘了分片样本 id 0, 1, 2, 3样本 id 0, 1, 2, 3两个 rank 在重复提供同一批信息。训练看起来在跑，效果却像没扩大数据量。
分片有了，但没重设 epoch每轮顺序相同每轮顺序相同样本覆盖不充分，shuffle 失去意义。每轮曲线几乎复制上一轮，泛化会慢。
分片和 batch 不匹配最后一个 batch 很小最后一个 batch 很小或空局部 batch 波动，统计量可能不稳定。最后几个 step 波动大，或者某些实现会卡住。
代码
from torch.utils.data import DataLoader, DistributedSampler

sampler = DistributedSampler(
 dataset,
 num_replicas=world_size,
 rank=rank,
 shuffle=True,
 drop_last=True,
)
loader = DataLoader(dataset, batch_size=micro_batch, sampler=sampler)

for epoch in range(num_epochs):
 sampler.set_epoch(epoch)
 for batch in loader:
 step(batch)

这段代码里，sampler 读的是 dataset、world size 和 rank，写的是每个进程能看到的样本顺序。set_epoch(epoch) 则是在每轮训练开始时换掉洗牌种子，让不同 epoch 不再重复同一串顺序。省掉这一行，DDP 还能跑，但样本顺序会太整齐。

为什么这里先查它

很多“多卡训练没变好”的问题，最后都落在数据分片上。梯度同步本身可以是对的，但如果每个 rank 读的是同一批样本，你得到的只是重复计算，不是更好的优化方向。

和练习的关系

后面的诊断题里，先让你检查数据分片而不是先查算子，就是因为这个环节最容易把训练语义直接写歪。你把这一节看明白，很多“验证像单卡小 batch”的现象就会更好判断。

术语与角色

trace

在 DDP 里，rank 表示通信组里的身份号，world size 表示参与协作的进程总数，process group 表示能彼此做 collective 的那组进程。把这几个对象拆开，后面才不会把 sampler、collective 和 wrapper 混在一起。

训练步账本

state

把一次 step 拆成账本，比背 DDP 名词更有用。你只要看每步在读什么、写什么，就知道同步发生在哪个边界。

同步轨迹

trace

点每一步，看“本地读写”怎么切换到“同步读写”。这个小轨迹比单纯读代码更容易看出，梯度同步真正发生在 backward 之后。

代码
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group("nccl")
rank = dist.get_rank()
model = DDP(model.cuda(rank), device_ids=[rank])

for batch in loader:
 x, y = batch[0].cuda(rank), batch[1].cuda(rank)
 optimizer.zero_grad(set_to_none=True)
 loss = loss_fn(model(x), y)
 loss.backward() # 这里触发梯度同步
 optimizer.step() # 各 rank 用同一份梯度更新参数副本

这段骨架里，真正触发同步的是 loss.backward()。DDP 做的是把同步逻辑挂进反向过程里。

两卡推演

case

把这件事手算一遍，最容易看清同步到底在修正什么。假设两张 GPU 用同一个模型副本，rank 0 看到样本 A，rank 1 看到样本 B。它们在 forward 和 backward 阶段各自独立计算，直到梯度同步才开始合并信息。

代码
mean_grad = (g0 + g1) / 2
p_next = p - eta * mean_grad

这两行很短，却把 DDP 的责任边界说得很清楚：本地先算出局部梯度，再把局部梯度合成同一份更新方向，最后所有副本一起走向下一轮。如果某个 rank 提前跳到别的分支，或者把别人的样本读成自己的，第二轮开始参数就会分叉。

一个常见误会

如果你只看到“多个副本都在跑”，很容易误以为同步已经完成。实际上，forward 只是把同一份模型放到不同数据上做局部计算；同步发生在 backward 之后。

这个推演怎么用

读训练脚本时，把自己的模型、sampler 和 optimizer 套进这张表：哪一步写激活，哪一步写梯度，哪一步把局部梯度变成全局更新。你一旦能把脚本对上这张表，就不容易在 DDP 里看错职责。

all-reduce 到底在做什么

math

all-reduce 这个词只说明“把多个 rank 的值规约到一起”。在训练里最常见的是先求和，再除以 world size 得到平均梯度。把这两个动作拆开，是避免误读代码的关键。

其中 N 是 world size，G 是一个 rank 上所有梯度张量的总字节数。这个式子是在 ring all-reduce 的常见量级上做的粗估，目的是帮你判断通信会不会压住收益。要做精确估计，还得看具体实现和网络条件。

代码
grad = local_grad.clone()
dist.all_reduce(grad, op=dist.ReduceOp.SUM)
grad /= world_size # 平均是训练代码层面的约定，不是 all_reduce 自动保证的

如果你只看到 all_reduce(..., op=SUM)，却没有看到后面的除法，就不要默认它已经是平均梯度。很多误判就发生在这里。

梯度累积和通信频率

frequency

当显存不够装更大的 batch 时，常见做法是把一个大 batch 拆成几个 micro-batch。前几个 micro-batch 只做局部 backward，不立刻同步；最后一个 micro-batch 再统一同步并 step。这样做的目的很明确：不改更新方向的前提下，减少通信频率。

代码
for i, batch in enumerate(loader):
 if i % accum_steps != accum_steps - 1:
 with model.no_sync():
 loss = loss_fn(model(batch), y)
 loss.backward()
 else:
 loss = loss_fn(model(batch), y)
 loss.backward()
 optimizer.step()

如果忘了在前几个 micro-batch 上用 no_sync()，你会在每个 micro-batch 都做一次梯度同步，通信频率就没有降下来。反过来，如果最后一步忘了同步，参数副本会在下一轮开始前带着不同的累积结果往前走。

为什么它和全局 batch 连着

梯度累积会把一个 step 里看到的样本数放大，所以全局 batch 公式要把累积步数算进去。这个地方如果算错，学习率和 warmup 都会跟着跑偏。

和前面两节的关系

这节把“什么时候同步”再往前推了一层：前几个 micro-batch 先本地攒，最后一个 micro-batch 再合并。你看懂这个开关，前面的 all-reduce 和通信账单就更完整了。

通信账单

cost

GPU 数量上去以后，慢的未必是算子本身。常见瓶颈是通信字节、等待最慢 rank、以及全局 batch 改变后带来的优化器和学习率重配。

一个很实用的判断

如果你把 world size 从 4 提到 8，通信通常不会简单减半。梯度总量没变，协作人数变多，通信和等待的形状都会改。

先看每个 rank 的时间线

先看每个 rank 各自卡在哪一步，再看平均 step time。只看平均值，最容易把“某个 rank 走散了”误判成“算子整体慢”。

边界先钉住

boundary

这页先把 DDP 的同步边界讲清楚。bucket、通信重叠、loss scaling、激活检查点、optimizer state sharding、pipeline bubble 这些词都重要；本页先从同步边界开始，它们会在后面的单元回来。

现在可以忽略什么

先不要去改 NCCL 参数，也不要急着猜网络拓扑。只要你还没把“谁读、谁写、谁同步”三件事对齐，调参数通常只是把症状挪地方。

故障信号

debug

看错时序时，程序不一定立刻报错。多半是卡住、变慢、或者结果悄悄变歪。先把最常见的几种症状和第一个检查动作对起来。

这些输出是模拟出来的症状，用来先看见“谁先走偏了”，再去怀疑 NCCL、网络或者算子。

练习前提醒

check

错误 1：把 all-reduce 当成自动平均

all-reduce 只说明规约动作；平均梯度要看代码里有没有再除以 world size。

错误 2：把 DDP 当成显存分片

DDP 的核心是同步副本的梯度；把模型状态切小是 FSDP2 这类分片策略要做的事。

错误 3：只盯平均 step time

collective 会等最慢的 rank。只看平均值，很容易把“某个 rank 走散了”误看成“系统整体都慢”。

错误 4：把 group 混成一个

数据并行、张量并行、流水线并行的通信对象不同。到了 Megatron Core 里，这个区分会直接影响结果对不对。

下一步怎么接

bridge

如果你要继续读代码，先找 init_process_group、DistributedSampler、DDP(model)、loss.backward()、optimizer.step()。先把这几个点在训练脚本里对齐，再去看 03 的 fully_shard，你会更容易明白“复制副本”和“分片状态”为什么是两类不同问题。

一句话收束

DDP 先让多个副本在同一份梯度上更新；FSDP2 再解决副本本身放不下的问题；Megatron Core 再把通信对象拆到多个维度里。它们属于同一条训练主线上的不同层。
