数据并行、梯度同步与通信

课程地图上一单元下一单元

读完本页，你应该能做什么

  • 能从一段训练代码里指出 rank、world size、process group、DDP 和 all-reduce 分别在做什么。
  • 能顺着一次训练步说清楚：哪些值是每个 rank 本地算的，哪一步开始同步梯度，最后谁在写参数。

学习目标

goal

这单元只盯三件事：把代码里的通信对象认出来，把一次训练步里同步发生的时刻认出来，把通信账单和性能瓶颈认出来。你能把同一段 DDP 代码拆成这三层，就算抓住了主线。

先认对象

看到 rank、world size、process group、DDP、all-reduce，能说出它们各自负责什么，不再把身份号、协作范围和同步动作混成一团。

再认时序

能顺着 forward → backward → 同步 → optimizer.step() 追一遍本地读写和 collective 读写的切换，知道同步发生在 backward 之后。

最后看代价

能估算梯度同步的大致字节量，并判断问题是算子慢、通信慢，还是 rank 之间已经走散。

先看主线

route

我们用一个最小锚点：两张 GPU、同一模型、不同 batch。每个 rank 先本地 forward/backward，再把梯度合成同一个更新方向。你要抓的是：哪些值可以各自算，哪些值必须在 backward 之后对齐。

参数副本 → 本地 forward → 本地 backward → 梯度同步 → optimizer.step()

这页先把 DDP 想成什么

先把 DDP 看成“复制训练步”。每个 rank 都有自己的 batch 分片，但最终要把梯度合成同一份更新。

先别追什么

先别把 bucket、通信重叠、混合精度、ZeRO 风格状态分片、pipeline bubble 当成起点。它们会在 03、04、05、06 里回来；本页先把同步边界钉住。

样本分片

data

数据并行要成立，除了模型副本要一致，样本也要切开。两张 GPU 用同一份数据，梯度会重复；两张 GPU 用不同分片，梯度才是在互补信息上做合成。这个差别很小，结果却很大，所以它是 DDP 里最值得先查的地方之一。

最常见的接法是 DistributedSampler。它知道 rank 和 world size，会把数据集切成若干份，让每个进程看到不同的样本子集。训练时还要在每个 epoch 重新设种子，否则 shuffle 虽然开着，顺序却可能每轮都一样。

情况 rank 0 看到什么 rank 1 看到什么 对梯度有什么影响 常见症状
正确分片 样本 id 0, 1, 4, 5 样本 id 2, 3, 6, 7 两个 rank 提供互补梯度，合成后更新更稳。 loss 变化正常，验证结果和单卡不一样。
忘了分片 样本 id 0, 1, 2, 3 样本 id 0, 1, 2, 3 两个 rank 在重复提供同一批信息。 训练看起来在跑，效果却像没扩大数据量。
分片有了，但没重设 epoch 每轮顺序相同 每轮顺序相同 样本覆盖不充分，shuffle 失去意义。 每轮曲线几乎复制上一轮，泛化会慢。
分片和 batch 不匹配 最后一个 batch 很小 最后一个 batch 很小或空 局部 batch 波动，统计量可能不稳定。 最后几个 step 波动大，或者某些实现会卡住。
DistributedSampler 骨架 
from torch.utils.data import DataLoader, DistributedSampler

sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,
    rank=rank,
    shuffle=True,
    drop_last=True,
)
loader = DataLoader(dataset, batch_size=micro_batch, sampler=sampler)

for epoch in range(num_epochs):
    sampler.set_epoch(epoch)
    for batch in loader:
        step(batch)

这段代码里，sampler 读的是 dataset、world size 和 rank，写的是每个进程能看到的样本顺序。set_epoch(epoch) 则是在每轮训练开始时换掉洗牌种子，让不同 epoch 不再重复同一串顺序。省掉这一行，DDP 还能跑，但样本顺序会太整齐。

为什么这里先查它

很多“多卡训练没变好”的问题，最后都落在数据分片上。梯度同步本身可以是对的，但如果每个 rank 读的是同一批样本，你得到的只是重复计算，不是更好的优化方向。

和练习的关系

后面的诊断题里，先让你检查数据分片而不是先查算子，就是因为这个环节最容易把训练语义直接写歪。你把这一节看明白，很多“验证像单卡小 batch”的现象就会更好判断。

术语与角色

trace

在 DDP 里，rank 表示通信组里的身份号，world size 表示参与协作的进程总数，process group 表示能彼此做 collective 的那组进程。把这几个对象拆开，后面才不会把 sampler、collective 和 wrapper 混在一起。

对象 一句话 读什么 写什么 错了会怎样
rank 通信组里的身份号。 进程在组内的编号。 不写模型，只标识自己。 把它当机器号，排查时会看错组。
world size 参与协作的进程总数。 组里有多少个 rank。 影响 batch、通信量和平均语义。 把它当单机 GPU 数，会错估全局 batch。
process group 能彼此做 collective 的那组进程。 这次同步的成员集合。 决定谁要一起进入 all-reduce。 把 group 搞错，轻则结果错，重则 hang。
DistributedSampler 把数据切给不同 rank。 epoch、rank、world size。 样本顺序和分片。 所有 rank 读到同一批样本，等于重复训练。
DDP 给模型加上梯度同步逻辑。 参数、梯度和反向 hook。 同步后的梯度 bucket。 以为它会切模型参数，结果只是在同步副本。
all-reduce 把多个 rank 的值规约到一起。 本地梯度或其他张量。 同步后的结果张量。 把它当自动平均，漏掉代码里的显式除法。

训练步账本

state

把一次 step 拆成账本，比背 DDP 名词更有用。你只要看每步在读什么、写什么，就知道同步发生在哪个边界。

步骤 读什么 写什么 谁能独立做 谁必须一致 出错时的症状
读 batch 本地样本分片。 输入张量。 每个 rank 都能自己做。 样本分片方式。 所有 rank 看到同一批样本，验证像单卡小 batch。
forward 参数副本和 batch。 激活、loss。 每个 rank 独立做。 模型结构和参数初始值。 前向结果不一致，后面再同步也救不回来。
backward 激活、loss、局部梯度路径。 本地梯度。 每个 rank 独立做。 反向图必须一致。 有些参数没参与反向，某些 rank 会缺梯度。
all-reduce 本地梯度。 同步后的梯度。 不能单独完成。 进入顺序和 group 成员。 挂住、等待、或者某个 rank 先跑到别的分支。
optimizer.step() 同步后的梯度和优化器状态。 参数副本、动量、学习率相关状态。 每个 rank 都能做同一件事。 更新方向必须一致。 参数副本分叉，下一轮 loss 开始不一致。
sampler.set_epoch() epoch 和 shuffle 种子。 新的数据顺序。 每个 rank 都会看见同一 epoch 号。 数据分片和洗牌规则。 每轮样本顺序重复，训练效果看起来像没 shuffle。

同步轨迹

trace

点每一步，看“本地读写”怎么切换到“同步读写”。这个小轨迹比单纯读代码更容易看出，梯度同步真正发生在 backward 之后。

本地 forward 参数副本不变

读参数和 batch，写激活与 loss。此时每个 rank 还在各算各的。

本地 backward 本地梯度出现

读激活和 loss，写每个参数的局部梯度。梯度还没同步，副本之间仍可能不同。

梯度同步 同一份更新方向

all-reduce 把各 rank 的梯度合成同一份结果。这里顺序或 group 一旦错了，程序就容易挂住。

optimizer.step() 参数副本继续一致

读同步后的梯度和优化器状态，写回参数。只要前面同步正确，下一轮副本就还能保持一致。

DDP 训练步骨架 
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group("nccl")
rank = dist.get_rank()
model = DDP(model.cuda(rank), device_ids=[rank])

for batch in loader:
    x, y = batch[0].cuda(rank), batch[1].cuda(rank)
    optimizer.zero_grad(set_to_none=True)
    loss = loss_fn(model(x), y)
    loss.backward()  # 这里触发梯度同步
    optimizer.step()  # 各 rank 用同一份梯度更新参数副本

这段骨架里，真正触发同步的是 loss.backward()。DDP 做的是把同步逻辑挂进反向过程里。

两卡推演

case

把这件事手算一遍，最容易看清同步到底在修正什么。假设两张 GPU 用同一个模型副本，rank 0 看到样本 A，rank 1 看到样本 B。它们在 forward 和 backward 阶段各自独立计算，直到梯度同步才开始合并信息。

时刻 rank 0 rank 1 共同结果
forward 前 参数副本 p 参数副本 p 两边参数一样。
forward 后 局部 loss0 局部 loss1 每边看到的样本不同，loss 也不同。
backward 后 局部梯度 g0 局部梯度 g1 还没有同步，两个梯度可能不同。
all-reduce 后 (g0 + g1) / 2 (g0 + g1) / 2 两边拿到同一份更新方向。
optimizer.step() 后 p' = p - η · mean(g) p' = p - η · mean(g) 参数副本继续一致。
两卡手算公式 
mean_grad = (g0 + g1) / 2
p_next = p - eta * mean_grad

这两行很短，却把 DDP 的责任边界说得很清楚：本地先算出局部梯度，再把局部梯度合成同一份更新方向，最后所有副本一起走向下一轮。如果某个 rank 提前跳到别的分支，或者把别人的样本读成自己的，第二轮开始参数就会分叉。

一个常见误会

如果你只看到“多个副本都在跑”，很容易误以为同步已经完成。实际上，forward 只是把同一份模型放到不同数据上做局部计算；同步发生在 backward 之后。

这个推演怎么用

读训练脚本时，把自己的模型、sampler 和 optimizer 套进这张表：哪一步写激活，哪一步写梯度，哪一步把局部梯度变成全局更新。你一旦能把脚本对上这张表，就不容易在 DDP 里看错职责。

all-reduce 到底在做什么

math

all-reduce 这个词只说明“把多个 rank 的值规约到一起”。在训练里最常见的是先求和，再除以 world size 得到平均梯度。把这两个动作拆开，是避免误读代码的关键。

$$ B_{\mathrm{sync}} \approx 2 \cdot \frac{N-1}{N} \cdot G $$

其中 N 是 world size，G 是一个 rank 上所有梯度张量的总字节数。这个式子是在 ring all-reduce 的常见量级上做的粗估，目的是帮你判断通信会不会压住收益。要做精确估计，还得看具体实现和网络条件。

rank 本地梯度 sum all-reduce 之后 再除以 world size
0 0.5 8.0 2.0
1 1.5 8.0 2.0
2 2.0 8.0 2.0
3 4.0 8.0 2.0
sum 和 average 的分界线 
grad = local_grad.clone()
dist.all_reduce(grad, op=dist.ReduceOp.SUM)
grad /= world_size  # 平均是训练代码层面的约定，不是 all_reduce 自动保证的

如果你只看到 all_reduce(..., op=SUM)，却没有看到后面的除法，就不要默认它已经是平均梯度。很多误判就发生在这里。

梯度累积和通信频率

frequency

当显存不够装更大的 batch 时，常见做法是把一个大 batch 拆成几个 micro-batch。前几个 micro-batch 只做局部 backward，不立刻同步；最后一个 micro-batch 再统一同步并 step。这样做的目的很明确：不改更新方向的前提下，减少通信频率。

micro-batch 本地发生什么 是否同步梯度 对通信的影响
第 1 个 forward + backward 否 本地先攒梯度，不发通信。
第 2 个 forward + backward 否 继续累积本地梯度。
最后 1 个 forward + backward + all-reduce 是 一次性对齐再更新。
DDP 的累积骨架 
for i, batch in enumerate(loader):
    if i % accum_steps != accum_steps - 1:
        with model.no_sync():
            loss = loss_fn(model(batch), y)
            loss.backward()
    else:
        loss = loss_fn(model(batch), y)
        loss.backward()
        optimizer.step()

如果忘了在前几个 micro-batch 上用 no_sync()，你会在每个 micro-batch 都做一次梯度同步，通信频率就没有降下来。反过来，如果最后一步忘了同步，参数副本会在下一轮开始前带着不同的累积结果往前走。

为什么它和全局 batch 连着

梯度累积会把一个 step 里看到的样本数放大，所以全局 batch 公式要把累积步数算进去。这个地方如果算错，学习率和 warmup 都会跟着跑偏。

和前面两节的关系

这节把“什么时候同步”再往前推了一层：前几个 micro-batch 先本地攒，最后一个 micro-batch 再合并。你看懂这个开关，前面的 all-reduce 和通信账单就更完整了。

通信账单

cost

GPU 数量上去以后，慢的未必是算子本身。常见瓶颈是通信字节、等待最慢 rank、以及全局 batch 改变后带来的优化器和学习率重配。

成本项 你会看到什么 先查哪里 容易误判成什么
梯度字节 step time 里通信占比高。 总梯度大小和 world size。 以为只是 GPU 算力不够。
慢 rank 等待 平均吞吐不错，但尾部 rank 拉长。 数据加载、网络、降频、IO。 以为 all-reduce 自己很慢。
全局 batch 变化 loss 曲线和收敛节奏变了。 micro-batch × accum × world size。 只调 GPU 数，不调学习率和 warmup。
bucket / overlap 通信看起来更顺，但问题没消失。 同步是否被更早开始、是否真正重叠。 以为优化器逻辑已经变了。

一个很实用的判断

如果你把 world size 从 4 提到 8，通信通常不会简单减半。梯度总量没变，协作人数变多，通信和等待的形状都会改。

先看每个 rank 的时间线

先看每个 rank 各自卡在哪一步，再看平均 step time。只看平均值，最容易把“某个 rank 走散了”误判成“算子整体慢”。

故障信号

debug

看错时序时，程序不一定立刻报错。多半是卡住、变慢、或者结果悄悄变歪。先把最常见的几种症状和第一个检查动作对起来。

症状 更可能的原因 第一个检查动作
训练卡住不动 collective 顺序不一致，或者某个 rank 走进了别的分支。 看每个 rank 到达了哪一个同步点。
吞吐看起来上去了，但验证像单卡小 batch 每个 rank 读到了同一批样本。 打印少量样本 id，确认分片是否不同。
loss 震荡变大 world size 变了，但全局 batch、学习率或 warmup 没跟着改。 先把 batch 公式算清楚。
只有部分 rank 报错 模型图或条件分支在各 rank 不一致。 检查有没有按 rank 走不同逻辑。 
[rank 0] backward done
[rank 1] backward done
[rank 0] all-reduce ok
[rank 1] all-reduce ok
[rank 0] optimizer.step ok
[rank 1] optimizer.step ok

这些输出是模拟出来的症状，用来先看见“谁先走偏了”，再去怀疑 NCCL、网络或者算子。

练习前提醒

check

错误 1：把 all-reduce 当成自动平均

all-reduce 只说明规约动作；平均梯度要看代码里有没有再除以 world size。

错误 2：把 DDP 当成显存分片

DDP 的核心是同步副本的梯度；把模型状态切小是 FSDP2 这类分片策略要做的事。

错误 3：只盯平均 step time

collective 会等最慢的 rank。只看平均值，很容易把“某个 rank 走散了”误看成“系统整体都慢”。

错误 4：把 group 混成一个

数据并行、张量并行、流水线并行的通信对象不同。到了 Megatron Core 里，这个区分会直接影响结果对不对。
