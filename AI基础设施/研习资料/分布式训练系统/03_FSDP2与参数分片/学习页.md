# 学习目标 {#goals}
<!-- tag: goal -->

看懂状态
把参数、梯度、优化器状态和激活拆成两类：常驻状态和临时状态。FSDP2 主要改前者，后者还要靠别的办法控峰值。


看懂时序
知道 `fully_shard` 以后，参数什么时候是分片，什么时候短暂恢复完整，什么时候又被切回去。


看懂边界
能把 FSDP2 和 DDP、张量并行、Megatron Core 分开，不把“省显存”“切计算”“工程栈”混成一件事。

# 先抓主线 {#overview}
<!-- tag: overview -->

02 单元已经讲清楚，数据并行下多个 rank 会同步梯度，但每张卡仍然保留完整模型副本。03 要回答的是：如果模型和优化器状态已经大到“完整副本常驻”吃不下，训练系统要怎么继续跑。

FSDP2 的思路很直接：长期只保留分片，轮到某个模块计算时再把它需要的那部分参数临时聚齐，算完再切回去。这样省下的是常驻显存，不是把模型从计算里拿掉。

# 术语与账本 {#terms}
<!-- tag: terms -->

术语
先怎么理解
这页里的作用
常见误读


`fully_shard`
把模块参数改成分片表示的入口。
告诉 PyTorch 这个模块要按 FSDP2 的方式处理参数和状态。
把它当成新建了一个外层 wrapper，忽略了原地改模块能力这件事。


DTensor
一种带分布式布局信息的张量表示。
让“这个参数现在是分片的”变成可检查的状态，而不是口头说法。
以为只要名字还在，参数就是完整张量。


all-gather
把各 rank 手里的片临时聚成完整参数。
forward 或 backward 真要用到完整参数时再执行。
以为只会在训练开始发生一次。


reduce-scatter
规约梯度后，再把结果切成分片留下。
让每个 rank 只保留自己负责的梯度片。
把 FSDP2 误解成“不同步梯度”。


reshard_after_forward
forward 后要不要立刻切回分片。
控制显存峰值和通信次数的交换。
把它当成性能开关，而不是内存/通信取舍。

# 两卡推演 {#walkthrough}
<!-- tag: trace -->

把问题缩到最小：只有两张卡，世界规模是 2，模型里只看一个线性块。DDP 和 FSDP2 的差别，在这个最小场景里最容易看清楚。DDP 里，两张卡各自都持有完整参数和完整梯度；FSDP2 里，两张卡只长期保存一半参数，真正算这个块的时候才把两份分片临时拼起来。

## box soft
### 卡 0 看到什么
初始化后只保留本地分片。forward 前，卡 0 先拿到完整参数，然后执行这层计算。forward 结束后，如果配置要求立刻 reshard，它只留下自己的那一半。backward 到来时，它收到规约后的梯度分片，再把自己的分片更新掉。

## box info
### 卡 1 看到什么
流程一样，只是分片索引不同。卡 1 也不会一直背着整层参数走。它只在需要时参与聚齐，在计算结束后回到分片常驻状态。这个行为和 DDP 最大的差别，在于“完整参数”只活在短暂窗口里。

这张表的意义不只是“看流程”。它告诉你，FSDP2 的通信围绕模块边界不断做聚齐和切回。这样做的结果是，训练步骤里每个状态对象的寿命都变短了：参数不再常驻完整，梯度不再常驻完整，优化器状态也不再常驻完整。代价是你要更小心通信顺序、包装边界和 hook 是否真的被走到。

如果你只看“每卡显存变小”这一句，很容易误判成所有问题都解决了。真实情况是，FSDP2 把瓶颈从“长期常驻副本太大”换成“模块边界附近的聚齐成本”。所以你在调参时应该先判断哪一层在卡：是参数常驻太大，还是 forward 前后的临时聚齐太重，还是激活本身就已经把显存顶满。这个判断会直接决定你下一步是继续包模块，还是改 micro-batch，还是该去 04、05 单元里看别的并行维度。

> **这个推演和练习怎么对上** q1、q2、q3、q4 先确认你能读懂这张两卡表；q5、q6、q7 再看你能不能把“长期状态”和“峰值”分开算；q8 到 q13 则要求你把推演回接到真实脚本改动。

# 包装顺序 {#wrap-order}
<!-- tag: order -->

FSDP2 的一个核心工程习惯是自底向上包：先包子模块，再包父模块，最后才考虑根模块。目的让通信边界跟计算边界尽量贴近。

## box soft
### 为什么先包子模块
子模块先进入独立通信组后，forward 的聚齐和 backward 的规约就能围着更小的计算块转。这样更容易和当前层计算重叠，也不容易把整个模型拧成一个大块通信。

## box info
### 为什么不只包根模块
只包根模块，所有参数都落进一个大组，forward 前要聚齐太多内容，backward 后也要规约太多内容。通信会变得粗，而且更难隐藏在计算后面。

包装顺序示意

```python
from torch.distributed.fsdp import fully_shard

for layer in model.layers:
 fully_shard(layer)

fully_shard(model)

# 优化器要在 fully_shard 之后创建，
# 这样它拿到的是已经分片后的参数对象。
```

> **这段代码的检查点** 先确认子模块已经都包好，再检查优化器是不是在最后创建。顺序一错，后面的显存和状态都会看起来“像是没生效”。

# 迁移案例 {#migration-case}
<!-- tag: case -->

下面这个案例把前面的概念合成一条可执行的迁移路径。假设你手里是一份 8B 参数的训练脚本，原来用 DDP 跑 8 张卡，现在线上说法是“还是能跑，但太吃显存，后面想把 batch 再放大一点就顶住了”。这个场景里，FSDP2 要解决的“常驻状态能不能先瘦下来”。

迁移时最容易踩的坑，是把“包模块”和“建优化器”两件事分开了，却没有检查调用路径。真正的顺序应该是：先让子模块进入分片管理，再让根模块补上总入口，最后再创建优化器并开始训练。这样优化器、参数和梯度才会对着同一套对象工作。

步骤
DDP 时代怎么做
迁到 FSDP2 后怎么做
如果顺序错了会看到什么


1. 构造模型
实例化完整模块。
还是先实例化完整模块，但后面会逐层 `fully_shard`。
如果直接跳过这一步，后面的包装没对象可包。


2. 包装模块
通常不用额外处理。
先包子模块，再包父模块，最后再考虑根模块。
只包根模块，通信会变大，forward 前后都容易卡。


3. 建优化器
直接对完整参数建优化器。
等 `fully_shard` 做完，再对分片后的参数建优化器。
优化器抓到旧引用，显存和状态看起来都不对。


4. 跑 forward / backward
每卡都跑完整副本。
forward 前临时 all-gather，backward 后规约并分片。
如果绕过模块调用路径，hook 不会在正确时间触发。

迁移前后对照

```python
# 旧脚本：DDP 时代常见写法
model = build_model()
optimizer = AdamW(model.parameters(), lr=1e-4)
model = DDP(model)
loss = model(batch).loss

# 迁移后：FSDP2 时代的顺序
model = build_model()
for block in model.blocks:
 fully_shard(block)
fully_shard(model)
optimizer = AdamW(model.parameters(), lr=1e-4)
loss = model(batch).loss
```

这段代码背后的变化很清楚：DDP 让每个 rank 保留完整副本，FSDP2 把常驻状态换成分片。你不需要把每一层都重新设计成新的网络结构，只需要把“状态怎么驻留、什么时候聚齐、什么时候切回去”这三件事摆正。这样一来，模型表达没有变，训练账本变了。

如果你在迁移时发现显存没降、速度还慢，优先怀疑的是包装粒度和调用路径，而不是先怀疑 PyTorch。FSDP2 并不会替你消除激活，也不会自动把通信变成零成本；它只是在正确的模块边界上，把长期常驻状态压小。这个判断方式会直接影响你之后怎么改 batch、怎么调 checkpoint、怎么决定要不要继续往张量并行走。

> **这段迁移案例和练习题怎么接** q8、q9、q10 负责检查顺序和调用路径；q13 检查你能不能把“省显存但可能降吞吐”说成一条完整因果链。先把这个案例看懂，后面的诊断题就不会只剩术语。

# 最小改动清单 {#bridge}
<!-- tag: bridge -->

如果你现在手上已经有一份能跑的 DDP 训练脚本，切到 FSDP2 的第一步把“对象的寿命”按顺序重新摆一遍。下面这份清单能让你快速判断到底改了哪里。

对象
要不要改
怎么改
错了会怎样


模型构造
要。
保持原模型不变，但在包装前先保证子模块层级清晰。
如果模型本身结构混乱，包装边界会跟着乱。


参数包装
要。
按子模块 → 父模块 → 根模块的顺序 `fully_shard`。
边界太大，通信和显存峰值都不好看。


优化器
要。
放到所有 `fully_shard` 之后再创建。
优化器抓到旧参数，状态对不上。


训练调用
要检查。
用 `model(x)` 走模块调用路径，不要绕过 hook。
聚齐和切分不会按预期触发。


显存预期
要修正。
把“常驻状态下降”和“峰值立即减半”分开看。
如果还按 DDP 的口径看峰值，会误判 FSDP2 没起作用。

最小迁移骨架

```python
model = build_model()

# 1. 先把分片边界落到子模块，再补上总入口
for block in model.blocks:
 fully_shard(block)
fully_shard(model)

# 2. 再创建优化器，让它接住分片后的参数对象
optimizer = AdamW(model.parameters(), lr=1e-4)

# 3. 训练时保持模块调用路径
loss = model(batch).loss
loss.backward()
optimizer.step()
```

这段骨架的价值，是把“想法”变成最少的改动路径。你不需要先碰张量并行，也不需要先碰混合精度。先让常驻状态真的分片，再决定是否要继续往更宽的并行策略走。这样调试时，一旦显存没降、通信没藏住或者 hook 没触发，你就知道该回到哪一层看。

如果你想把这页内容直接转成排障动作，按这个顺序走最稳：先看包装边界，再看优化器创建时机，然后看训练调用路径，最后才去看通信是否能被计算掩住。这个顺序和 q8、q9、q10、q13 是对齐的，所以练习题不是孤立的，它们就是这份最小改动清单的检查器。

> **这里能直接做什么** 你现在可以拿一份自己的 DDP 训练脚本，对照这张表，先标出“模型构造、包装、优化器、调用路径、显存预期”五个点，再决定哪一步先改。这样就不会一上来把整套训练代码重写。

# 概念对照 {#compare}
<!-- tag: compare -->

DDP
每卡都保留完整模型副本，靠同步梯度保持更新一致。简单、稳定，但状态常驻显存大。


FSDP2
沿数据并行维度分片参数、梯度和优化器状态，按模块临时聚齐完整参数。它改的是状态常驻方式。


张量并行 / Megatron Core
切的是层内计算和更宽的训练栈组合，不是 FSDP2 的同义词。它们可以组合，但解决的问题层级不同。

对象
DDP 改什么
FSDP2 改什么
张量并行改什么


常驻状态
保留完整副本。
参数、梯度和优化器状态按 rank 分片常驻。
通常不是主要目标。


计算方式
每个 rank 做完整前后向。
仍然是完整前后向，只是参数按需聚齐。
把单层计算切开，让矩阵在多个 rank 间分担。


主要瓶颈
梯度同步和副本常驻。
常驻显存与临时通信。
层内计算和跨 rank 通信边界。

# 常见错误 {#mistakes}
<!-- tag: debug -->
<!-- layout: grid2 -->

## box
### 先建优化器，再 `fully_shard`
这样优化器可能抓住旧参数引用，后面看起来像参数没更新。修法是先完成所有分片包装，再创建优化器。

## box
### 直接调 `forward`
这会绕开模块调用路径，默认 hook 可能不执行。修法是用 `model(x)`；如果必须走别的方法，就要显式处理 hook 或 unshard。

## box
### 只包根模块
所有参数都会挤进一个大通信组，forward 前和 backward 后都更重。修法是按层或按清晰模块边界自底向上包。

## box
### 把 FSDP2 当成激活省内存方案
它主要分片参数、梯度和优化器状态。激活峰值还是要靠检查点、micro-batch、序列并行或其他策略。

forward 前后卡顿明显

## box soft
### 怎么答才算会
这组题看你能不能把“分片、聚齐、同步、常驻、峰值”分开。能说出对象、时间点和后果，才算真的会。

### 检查重点
优先检查三件事：你有没有把 DDP 和 FSDP2 分开，能不能说清生命周期的读写顺序，能不能解释为什么 FSDP2 不自动解决激活峰值。

<!-- practice-group: 一、状态与时序 -->
<!-- quick-review: FSDP2 的核心是分片常驻 + 按需聚齐。读题时先找对象，再找时间点。 -->

### 1 choice

**Q:** FSDP2 初始化后，参数通常以什么形式常驻？

- [ ] . 每张卡都保留完整参数副本
- [ ] . 只保留本 rank 负责的参数分片
- [ ] . 参数被删除，只剩梯度

Answer: . 只保留本 rank 负责的参数分片

### 2 true/false

**Q:** FSDP2 会完全避免梯度同步吗？

- [ ] . 会，所有 rank 彼此独立更新
- [ ] . 不会，仍然要同步，只是同步结果变成分片

Answer: . 不会，仍然要同步，只是同步结果变成分片

### 3 choice

**Q:** forward 前把需要的参数临时聚齐，通常靠什么 collective？

- [ ] . all-gather
- [ ] . reduce-scatter
- [ ] . broadcast

Answer: . all-gather

### 4 synthesis

**Q:** 用四句话描述 FSDP unit 的一次前向/反向生命周期。

Explain: 要点至少包括：初始化时参数是分片；forward 前会临时 all-gather；forward 后按配置切回分片或暂存完整参数；backward 后梯度会规约并切成分片。少写一步就说明时序没对齐。

<!-- end-practice-group -->

<!-- practice-group: 二、显存与峰值 -->
<!-- quick-review: 答题时先区分常驻状态和临时聚合，不要把两者混成一个数。 -->

### 5 diagnosis

**Q:** 为什么 FSDP2 的显存下降不等于峰值按 world size 直接缩小？

Explain: 应该提到两层：FSDP2 只把参数、梯度和优化器状态分片，所以长期常驻会下降；但 forward 前的 all-gather 仍会短暂形成完整参数，激活也还在，所以峰值不会简单变成 1/world_size。

### 6 scenario

**Q:** 如果当前瓶颈其实是激活，下一步优先看什么？

- [ ] . 继续把 FSDP 包装切得更碎
- [ ] . 先看激活检查点、micro-batch、序列长度或并行策略
- [ ] . 只要换成 DDP 就会省激活

Answer: . 先看激活检查点、micro-batch、序列长度或并行策略

### 7 calculation

**Q:** 在 8B 参数、BF16、Adam 两个 FP32 状态的简化模型里，DDP 与理想 FSDP 的关系是什么？

Explain: 不要求精确到小数，但必须说清：DDP 每卡持有完整副本，所以参数、梯度和 Adam 状态都按完整规模常驻；理想 FSDP 则把这些长期常驻状态按 world size 分掉。答题时如果只写“显存减小”，说明没有区分常驻和峰值。

<!-- end-practice-group -->

<!-- practice-group: 三、包装与调用路径 -->
<!-- quick-review: 这组题重点看顺序：先包模块，后建优化器；先走模块调用路径，再谈 hook。 -->

### 8 choice

**Q:** 为什么通常不建议只对 `root model` 调用 `fully_shard`？

- [ ] . 因为这样会把所有参数放进一个大通信组，难和计算重叠
- [ ] . 因为 root model 不能被 PyTorch 训练
- [ ] . 因为这样会自动变成张量并行

Answer: . 因为这样会把所有参数放进一个大通信组，难和计算重叠

### 9 true/false

**Q:** 优化器应该在 `fully_shard` 之前还是之后创建？

- [ ] . 之前
- [ ] . 之后

Answer: . 之后

### 10 error-diagnosis

**Q:** 你看到训练脚本写 `loss = model.forward(x).sum()`，可能有什么问题？

Explain: 必须指出：直接调 forward 可能绕开默认模块调用路径，导致 FSDP2 需要的 hook 不触发，参数不会在正确时间聚齐或切回分片。修法通常是改为 model(x) ；如果业务上必须走自定义方法，就要显式管理 hook 或 unshard。

<!-- end-practice-group -->

<!-- practice-group: 四、对照与综合 -->
<!-- quick-review: FSDP2 切状态，DDP 同步梯度，张量并行切层内计算。先说清这一句，再写长答案。 -->

### 11 synthesis

**Q:** 用一段话比较 DDP、FSDP2、张量并行。

Explain: 好答案要包含三层对照：DDP 复制完整模型、靠同步梯度保持一致；FSDP2 主要沿数据并行维度分片参数/梯度/优化器状态，减少常驻显存；张量并行切的是层内计算和权重矩阵，不是状态账本。答题时如果把 FSDP2 和张量并行混成一类，说明抽象层级还没分开。

### 12 boundary

**Q:** Megatron Core 在本页中的作用是什么？

Explain: 要点是：Megatron Core 提供的是更宽的训练栈能力，包含多种并行策略、分布式优化器和分布式检查点；本页只把它当作边界参照，用来提醒你 FSDP2 只是数据并行维度上的状态分片入口，不是完整训练栈本身。

### 13 interview

**Q:** 面试解释：为什么 FSDP2 省显存但可能降低吞吐？

Explain: 高分答案会把因果链写清：FSDP2 通过分片把长期常驻状态压小，所以能省显存；但 forward 前的 all-gather 和 backward 后的 reduce-scatter 会引入额外通信，如果这些通信不能很好重叠，就会吃掉一部分吞吐。
