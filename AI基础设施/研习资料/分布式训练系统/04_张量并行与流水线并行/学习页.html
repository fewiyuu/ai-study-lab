张量并行与流水线并行

这一页接着 03 的参数分片往下走，但重点换成模型内部怎么切：张量并行负责把同一层的矩阵和激活拆给多卡，流水线并行负责把模型深度和 micro-batch 的节拍排好。先把层内切分、阶段切分、数据副本同步分开，再看它们怎样同时出现在一条训练拓扑里。

课程地图 上一单元

读完本页应该能做什么

  • 能把 ColwiseParallel、RowwiseParallel、PrepareModuleInput 和 PrepareModuleOutput 连成一条可执行的层内切分链，而不是只会背名词。
  • 能顺着一个 H -> 4H -> H 的 MLP 和一个 3-stage、5 micro-batch 的流水线，画出 rank、stage、micro-batch 和通信动作的对应关系。
  • 能看懂什么时候该先查形状、什么时候该查调度、什么时候该查进程组边界，从而定位 TP / PP 的常见错误。

学习目标

goal

先分对象

张量并行切的是层内矩阵和激活，流水线并行切的是模型深度。两者都能扩卡，但它们要回答的问题不同。

最后看组合

把 TP、PP、DP 放进同一个 rank 拓扑里，知道每个维度各管什么，才不会把“并行”当成一个单一开关。

先抓主线

overview

03 单元解决的是“状态常驻太大怎么办”。04 进入模型内部并行：当单张卡已经能装下参数分片，但单层计算还是太慢、模型深度还是太长时，训练系统就会把层内矩阵和层序列拆开，让多个 rank 一起完成同一批前向和反向工作。

理解这页时先抓一个锚点：Transformer 的 MLP 里有一个 H -> 4H -> H 的两层结构。第一层适合按输出维切开，第二层适合按输入维切开；如果再把整个模型切成 3 个 stage，就会多出 micro-batch、stage 间传输和 bubble。这个锚点会贯穿后面的形状、时序和组合推演。

这页先用 PyTorch 的公共原语看清边界，Megatron Core 放到下一单元再把这些边界装进训练栈里。

术语与边界

terms
术语先怎么理解这页里的作用常见误读
TPTensor Parallelism，把同一层的矩阵切到多个 rank 上算。看层内形状、局部输出和什么时候要拼回完整结果。把它误读成按层切模型。
PPPipeline Parallelism，把模型按深度切成 stage。看 stage 划分、micro-batch 数量、调度顺序和气泡比例。以为 stage 越多吞吐一定越高。
DeviceMesh把 rank 排成明确的分布式网格。告诉 parallelize_module() 该沿哪条并行轴切。把它当成可有可无的包装参数。
process group一组需要一起做 collective 的 rank。限制通信只在该发生的 rank 之间发生。把 TP、PP、DP 的通信混成一个 group。
micro-batch送进流水线调度器的最小切片。决定管道能不能被填满，bubble 会不会太大。以为总 batch 足够大就不用管 micro-batch。

先别追的东西

Sequence Parallel、Context Parallel、Expert Parallel、zero-bubble schedule 和分布式优化器先不展开。它们会在后续单元或更后面的训练栈里回来。

张量并行追踪

tp

先看 MLP 的第一层：X=[B,H]，W1=[H,4H]。把输出维切成两份时，每个 rank 只拿到一半输出列；它算出来的是局部激活，不是完整结果。下一层如果还想保持分片，就让后续算子继续吃分片表示；如果要恢复完整输出，就在合适的位置做一次聚合。

代码
from torch.distributed.device_mesh import DeviceMesh
from torch.distributed.tensor.parallel import (
 ColwiseParallel,
 PrepareModuleInput,
 PrepareModuleOutput,
 RowwiseParallel,
 parallelize_module,
)

mesh = DeviceMesh("cuda", [0, 1])

tp_plan = {
 "mlp.fc1": ColwiseParallel(),
 "mlp.fc2": RowwiseParallel(),
 "mlp": [PrepareModuleInput(), PrepareModuleOutput()],
}

parallelize_module(mlp_block, mesh, tp_plan)

流水线时序

pp

流水线并行的难点不在“能不能切”，而在“切完以后怎么喂”。把 3-stage 模型拆成 5 个 micro-batch 时，前几步要先把 pipeline 填起来，后面才进入 steady state。stage 越多、micro-batch 越少，空转就越明显。

读这张表时先看什么

先看每个时间片有没有空白 stage，再看空白是不是由 micro-batch 不够造成的。如果 stage 2 比别的 stage 慢，后面的 stage 会等它，不是调度器本身“没写对”。

一个最小失败例子

4 个 stage 只给 2 个 micro-batch 时，后续 stage 很快空下来，前向看起来还能跑，吞吐却明显掉下去。这个现象是 bubble 变大，不是模型算错。

$$

\text{bubble} \approx \frac{S-1}{M+S-1}

$$

这里 S 表示 stage 数，M 表示 micro-batch 数。stage 越多、micro-batch 越少，空转比例越高。这个式子的作用让你在排查吞吐时先怀疑调度粒度。

组合推演

tp

TP 和 PP 不是互相替代的两个选项，它们通常和 DP 一起出现。一个更实用的看法是：DP 决定数据副本怎么同步，TP 决定单层怎么算，PP 决定层序怎么排。三者同时存在时，rank 往往会同时属于多个 group，但每次通信只应该落在一个明确的维度上。

怎么读这个推演

先从最外层的 DP 读起，再进到 PP，最后看 TP。这样你会发现三个维度的通信对象完全不同：一个管副本，一个管 stage，一个管层内矩阵。只要顺序倒了，你就容易把不同维度的 collective 认成同一种操作。

如果你要手算

先不管每个 rank 的精确编号，先写“谁和谁通信”。一旦你能把 TP 组、PP 相邻 stage、DP 同步组分别写出来，rank 编号只是把这个关系展开成表格而已。

代码桥接

bridge

如果你接下来要读 PyTorch 官方文档，最直接的入口就是 parallelize_module()。它把“这层用哪种并行方式”写成一个显式计划。看这类代码时，不要先问 API 细节，先问每个模块应该属于哪种切法。

代码
from torch.distributed.device_mesh import DeviceMesh
from torch.distributed.tensor.parallel import ColwiseParallel, RowwiseParallel, parallelize_module

mesh = DeviceMesh("cuda", [0, 1])

tp_plan = {
 "mlp.fc1": ColwiseParallel(),
 "mlp.fc2": RowwiseParallel(),
}

parallelize_module(model.mlp, mesh, tp_plan)

实现一致性

case

把 TP 和 PP 真正落到代码里时，最容易出问题的地方，模块树、并行计划和 stage 边界写成了三套互相独立的配置。表面上模型还能跑，实际上你已经在不同层级上回答不同问题：哪一层切开、哪一段放进同一 stage、哪一组 rank 负责同一副本、哪一处需要把局部结果拼回完整张量。

边界你在看什么对应动作检查重点一旦错位会怎样
模块树block、attention、MLP、residual 的嵌套关系决定哪一层进入 TP 计划名字是否稳定、层级是否清楚计划写进了错误的子模块，后面再调试会很乱
TP 边界层内矩阵是按列切还是按行切决定局部输出、局部输入、部分和的位置输入输出形状是否连续看起来能跑，结果却在某个边界上少了一次聚合
PP 边界哪些 block 放在同一个 stage决定激活从哪里传到哪里stage 的计算量和 micro-batch 是否配平一边空转，一边排队，吞吐掉得很快
DP 边界哪些 rank 共享同一份副本决定梯度在哪个 group 里同步同步 group 是否和 TP、PP 混用梯度重复聚合或聚合范围过大
代码
from torch.distributed.device_mesh import DeviceMesh
from torch.distributed.tensor.parallel import ColwiseParallel, RowwiseParallel, parallelize_module

mesh = DeviceMesh("cuda", [0, 1])

tp_plan = {
 "blocks.0.mlp.fc1": ColwiseParallel(),
 "blocks.0.mlp.fc2": RowwiseParallel(),
 "blocks.1.mlp.fc1": ColwiseParallel(),
 "blocks.1.mlp.fc2": RowwiseParallel(),
}

parallelize_module(model.blocks[:2], mesh, tp_plan)

# stage 0 只放前两个 block；后两个 block 的 stage 划分交给外层调度器。
# 如果 stage 边界穿过 fc2，后续 stage 看到的就会是半成品张量。

自查顺序

先画模块树，再写 TP 计划，最后才看 stage 调度。顺序一旦倒过来，你会把“半成品张量”和“stage 没填满”看成同一种错。

检查回路

loop

把一个 4-block Transformer 拆成 TP=2, PP=2, DP=2 时，先别看总吞吐，先把边界画出来：哪两个 block 属于同一 stage，哪一层走列并行，哪一层走行并行，梯度又在哪个副本组里收敛。这个顺序一旦乱掉，日志会同时出现形状、时序和同步三类信号，排查会很慢。

排查顺序

先看边界，再看形状，最后看调度。这个顺序很重要，因为很多问题表面像性能问题，实际上是布局问题；也有些看起来像布局问题，实际上是 stage 数和 micro-batch 数没配好。

故障实验室

lab

现在把前面的链条合成一次真实排错。假设你把一个 4-block Transformer 迁到 TP=2, PP=2，模型能跑，但第一种症状是吞吐掉得很快，第二种症状是某个 stage 总在等前一段，第三种症状是某个线性层的输出看起来像拼错了。只看一个症状很容易误判，必须按边界顺序排。

一个更小的复现办法，是先只保留一个 MLP block 和一条 stage 边界，把 ColwiseParallel / RowwiseParallel 跑通，再把 stage 切分加回去。这样你能先判断错在 TP 还是 PP：如果小模型都错，问题通常在层内布局；如果小模型正常、加上 stage 才出错，问题通常在流水线边界或 micro-batch。

常见错误

mistakes
常见错法直接症状先改哪里
把 TP 当成按层切模型形状和通信边界一直对不上先画层内矩阵，再写 TP 计划
只看 stage 数，不看 micro-batch管道没填满，bubble 很大先确认 micro-batch 是否足够
用一个全局 group 做所有通信collective 互相打架，排查很慢把 DP、TP、PP 的 group 分开
只记 API 名字，不看边界代码能跑，但结果错位先看模块树，再看 PrepareModuleInput / PrepareModuleOutput
