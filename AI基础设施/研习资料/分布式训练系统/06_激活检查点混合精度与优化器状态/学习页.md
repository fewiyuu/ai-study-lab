# 激活重算账本 {#checkpoint-ledger}
<!-- tag: ledger -->

这一步最重要的直觉是：checkpoint 不会“凭空省显存”，它只是把 forward 里的某些中间态不再常驻，等 backward 需要时再重跑一遍。只要你把读写顺序看清，后面再看 Megatron Core 的 selective recompute 就容易了。

对象
进入前
进入后
读取
写入 / 变更
错位症状


forward 激活
forward 结束后会常驻到 backward
部分激活被丢弃，只保留必要输入
模块输入、权重、必要的 RNG 状态
释放中间态，缩小峰值显存
如果 checkpoint 颗粒度太粗，显存没降多少；太细则重算过多


`preserve_rng_state`
随机层的前向 / 反向轨迹可能要保持一致
需要时保存并恢复 RNG 状态
dropout / 采样类算子的随机流
保证重算和原 forward 走同一条随机路径
如果关掉，训练可能还能跑，但数值轨迹和复现性会变差


recomputed forward
backward 还没请求中间态
backward 期间重新执行同一段 forward
当前输入、权重、激活边界
生成新的中间态给梯度计算用
如果重算块选错，额外计算大于省下的显存

PyTorch 激活检查点最小调用复制

```python
from torch.utils.checkpoint import checkpoint

def block_forward(x, mask):
 return transformer_block(x, mask)

hidden = checkpoint(block_forward, hidden, attention_mask, preserve_rng_state=True)
```

$$
 M_{\text{peak}} \approx M_{\text{param}} + M_{\text{grad}} + M_{\text{opt}} + r_{\text{act}} M_{\text{act}}
 $$
 `M_peak` 是峰值显存，`r_act` 是 checkpoint 后仍保留的激活比例，`M_act` 是原本的激活账。`r_act` 越小，省下的显存越多，但 backward 里要补回更多重算。

# Megatron Core 的选择式重算 {#recompute}
<!-- tag: recompute -->
<!-- layout: grid2 -->

## box
Megatron Core 的文档把 activation recompute 做成了可配置能力：你可以只挑最吃显存、又相对不太吃计算的子模块做重算。官方文档里最典型的锚点是 attention 里的 `core_attn`，因为它常常是大模型里最重的激活热点。
模式
适合谁
常见代价


不重算
激活本来就不大，或者先跑基线
显存压力最大，但最简单


selective / `core_attn`

# 混合精度路径 {#precision-path}
<!-- tag: precision -->
<!-- layout: grid2 -->

## box
PyTorch AMP 文档里最典型的 FP16 路径，是把 `torch.autocast` 和 `GradScaler` 一起看。前者决定哪些算子可以用低精度算，后者负责把 loss / grad 缩放到更安全的范围。两者都在训练里有位置，但职责不同。
组件 / dtype
负责什么
最容易误解成什么


`autocast`
让可支持的算子在合适的 dtype 下执行
自动接管整个训练


`GradScaler`
给 FP16 路径做缩放，减轻 underflow / overflow 风险
和 autocast 是一个开关


BF16
更看重稳定性和工程可用性
只是 FP16 的另一个名字


FP16
更省带宽和显存，但数值范围更窄
不需要再管缩放


FP8
更依赖框架 recipe、内核和通信支持
只要把 dtype 改掉就能跑

## box soft
### 怎么把它和 Megatron Core 连起来
Megatron Core 的训练栈会把 BF16、FP16、FP8 和 distributed optimizer 放在同一条训练路径里看。你调精度时，先看 compute dtype，再看通信 dtype，最后才看主权重和 optimizer state 的 dtype。

FP16 典型 AMP 训练步复制

```python
scaler = torch.amp.GradScaler()

with torch.autocast(device_type="cuda", dtype=torch.float16):
 loss = model(batch)

scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

这个骨架只说明分工：`autocast` 管算子 dtype，`GradScaler` 管缩放。BF16 训练是否需要同样的缩放策略，要看 recipe 和数值表现，不能直接照搬 FP16 的做法。

# 优化器状态分片 {#optimizer-state}
<!-- tag: optimizer -->

Megatron Core 的 distributed optimizer 文档给了一个很直观的流程：backward 结束后，梯度先在每个 DP rank 的 buffer 里聚合，再做 reduce-scatter；随后 optimizer.step() 更新主权重；最后 all-gather 把更新后的参数拼回 forward 可用的布局。你只要把这个顺序看清，就能解释为什么“能训练”不等于“能省显存”或者“能轻松恢复”。

步骤
进入前
进入后
读什么
写什么
错位症状


backward 完成
梯度还分散在计算图里
梯度 buffer 里有完整的 fp32 / bf16 视图
每个参数的局部梯度
聚合后的梯度缓冲
如果这一步没收尾，后面 step 的基础就不稳


reduce-scatter
全量梯度还在每个 rank 上
每个 DP rank 只保留自己那一段 fully reduced 梯度
DP 维上的 shard 范围
局部分片梯度
如果 shard 范围算错，通信和恢复都会错位


`optimizer.step()`
有分片梯度和主权重
主权重更新完成
分片梯度、主权重、优化器状态
更新后的主权重和 moment buffers
如果只保存了参数没保存状态，继续训练会像重开一样


all-gather
各 rank 只拿到自己的主权重片段
forward 可用的完整参数布局恢复
更新后的分片主权重
完整参数 buffer
如果布局没恢复，下一轮 forward 会找不到正确参数视图

$$
 M_{\text{opt, shard}} \approx \frac{M_{\text{opt}}}{DP}
 $$
 `DP` 是数据并行分片数，`M_opt` 是原本 optimizer state 的总账。这个式子只描述分片方向，不把通信、buffer 和 master weight 的开销一起算进去。

分布式优化器步骤骨架复制

```text
backward -> reduce-scatter -> optimizer.step() -> copy main weights -> all-gather -> next forward
```

# 一个能手算的排查例子 {#worked-case}
<!-- tag: case -->
<!-- layout: grid2 -->

## box
把三张账放进同一个场景里最有用。假设你在跑一个 24 层 decoder-only 模型，序列长度 4096，BF16 训练，DP=8，所有 transformer block 都开了 activation checkpoint。第一个症状是：打开 checkpoint 后，显存确实降了一截，但 backward 还是会在注意力层附近顶到峰值；第二个症状是：关掉 distributed optimizer 以后，显存又明显抬头；第三个症状是：如果你把 FP16 强行塞进去，loss 轨迹更容易抖。这三件事说明，激活账、状态账和 dtype 账分别在起作用，不能用同一个 knob 解释完。
这个例子里，正确的排查顺序是先问“哪一张账仍然是大头”，再决定改哪些配置。如果重算已经开了但 OOM 还在，先看 attention 热点和 checkpoint 颗粒度；如果激活省下来的不够，再看 optimizer state 有没有按 DP 分片；只有当数值不稳、overflow 或 NaN 出现时，才回头查 autocast 和 GradScaler。

## box soft
### 你该怎么读这个例子
- 先看症状落在哪一步：前向、反向、step，还是恢复。
- 再看这一步主要碰的是哪张账：激活、dtype、还是 optimizer state。
- 最后才决定改 checkpoint 颗粒度、精度策略，还是分片方式。

症状
最先怀疑的账
先看什么
下一步怎么改


checkpoint 已开，还是在 backward 顶峰值
激活账
重算颗粒度、是否命中 `core_attn`
先把重算放到真正热点上，而不是轻模块上


把 checkpoint 关掉后，显存只更早爆
激活账 + 状态账
optimizer state 是否仍然全量驻留
同时看激活重算和 distributed optimizer 的 shard


FP16 下 loss / grad 更容易抖
dtype 账
`autocast` 和 `GradScaler` 是否一起配
先修缩放和算子 dtype，再谈更激进的省显存策略


恢复 checkpoint 后状态不对
状态账
分片优化器状态和元数据是否一起保存
检查 shard 格式，不要只看权重文件能不能打开

这组症状先对应哪些开关复制

```text
if oom_at_backward:
 check(recompute_granularity, core_attn, optimizer_state_shard)
elif nan_or_overflow:
 check(autocast, GradScaler, fp16_recipe)
elif resume_fails:
 check(sharded_optimizer_state, checkpoint_metadata, dp_world_size)
```

## box warn
### 这个例子要防的坑
不要一看到 OOM 就先关 BF16，也不要一看到数值波动就先怀疑 checkpoint。先让症状落到正确的账本，再改对应开关，排查才会收敛。这个顺序会在 07 的恢复检查里继续用到，只是那一页会把保存和恢复说得更完整。

# 常见错误 {#mistakes}
<!-- tag: mistakes -->
<!-- layout: grid2 -->

## box
### 把激活检查点当成训练 checkpoint
activation checkpointing 是 backward 时重算 forward 片段；训练 checkpoint 才是保存模型、优化器和进度。前者省显存，后者保恢复。

## box
### 把 autocast 和 GradScaler 混成一个按钮
`autocast` 改的是算子 dtype，`GradScaler` 处理的是 FP16 的缩放。只看其中一个，FP16 训练很容易出 NaN 或 underflow。

## box
### 把 BF16 当成“自动稳”
BF16 往往更稳，但它仍然需要检查。kernel 支持、通信 dtype 和 optimizer state 都要看，不然只是把问题换个地方藏起来。

## box
### checkpoint 挂在太轻的块上
如果你把 checkpoint 放在显存并不重、计算却很贵的块上，省下的显存有限，吞吐却会掉得更明显。先挑 attention 这种真正热点。

## box
### 只看参数，不看 optimizer state
很多 OOM 来自 Adam 的 moment buffers、主权重和梯度账。只盯参数量，很容易误判瓶颈。

## box
### reduce-scatter 后不管布局
分片状态和 forward 可用布局需要分开看。更新后要恢复成模块能继续读的参数视图，不然下一轮 forward 会对不上。

**Q:** Q10：BF16 训练已经开了 checkpoint，但还是 OOM。你会先查哪一张账，为什么？

Explain: 好的答案会先把问题拆回三张账：如果 checkpoint 已经开了但 OOM 还在，先查激活账和 optimizer state 账，而不是立刻怀疑 BF16 本身。理由是 checkpoint 只改变激活账，BF16 主要改变 dtype 账；如果这两张账都还不够，才去看 distributed optimizer、DP 维和主权重是否仍然全量驻留。答案里最好写出下一步检查项，比如重算颗粒度、`core_attn`、DP 分片和 optimizer state 的布局。

### 12 synthesis

**Q:** Q12：把本页讲给刚学完 05 的同学听，要求先说顺序，再说一个典型失败。

Explain: 高分答案会把顺序说清楚：先用 activation checkpointing 处理激活账，再用 autocast / BF16 / FP16 处理 dtype 账，最后用 distributed optimizer 处理状态账。然后给一个典型失败，比如“checkpoint 开了但 OOM 还在”时先查激活热点和优化器状态，而不是立刻怀疑 BF16；或者“FP16 出 NaN”时先查 autocast 和 GradScaler，而不是去改 checkpoint。只复述名词还不够，关键是把顺序和失败映射讲出来。
