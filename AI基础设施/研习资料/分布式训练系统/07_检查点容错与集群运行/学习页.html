检查点、容错与集群运行

这一页接着 06 往下走，把“训练状态很大”变成“checkpoint 怎么保存、怎么恢复、怎么证明恢复正确”。重点不在单次 torch.save ，而在临时目录、分片状态、原子发布和恢复验证这一整条链。

学习目标

goal

把 checkpoint 当成训练状态

这一页先把“保存模型文件”改成“保存一轮训练继续所需的状态”。模型、优化器、scheduler、step、随机数、数据游标和并行元数据都要放进同一张账本里看。

把发布当成一笔事务

checkpoint 不该在一半写完时就被当成最新版本。先写临时目录，再提交完成标记，最后原子更新可见入口，这条顺序要比 API 名字更重要。

先抓主线

route

06 单元已经把激活检查点、混合精度和优化器状态讲清了。这一页继续往前走，不再停在“显存怎么省”，而是看省下来的训练状态怎么安全记录下来、怎么恢复、怎么迁移到新拓扑。

PyTorch Distributed Checkpoint 适合说明分布式 state_dict 的保存和加载；FSDP2 说明参数、梯度和优化器状态本来就是分片的；Megatron Core 进一步把分片 checkpoint、distributed optimizer 和对象存储策略放进同一套运行方式里。

这一页要回答的四个问题

  • 恢复训练时，到底哪些状态不能丢。
  • 分片 checkpoint 为什么能换 world size 继续读。
  • 异步保存为什么能降停顿，却不能替代原子发布。
  • 恢复成功后，应该怎么做最小冒烟验证。

状态账本

ledger

这张账本比“checkpoint 里要存什么”更有用。它直接告诉你每个对象在保存前是什么、保存后变成什么、读了谁、写了谁、错位时先出现什么症状。

保存链路

flow

真正可靠的 checkpoint 一个小事务：先把状态写进临时位置，再确认它完整，最后才把它变成对外可见的最新版本。这个顺序比文件名更重要。

代码
tmp_dir = save_dir + ".tmp"
save_dcp(state_dict=model_and_optim, storage=tmp_dir)
write_manifest(tmp_dir, step=current_step, rng=rng_state)
verify_checkpoint(tmp_dir)
publish_latest_atomically(tmp_dir, save_dir)

这段骨架写什么

写进去的不只是 tensor，还包括恢复需要的步数、随机状态和布局信息。最常见的错误是只写权重，把其他状态留在进程里。

重新分片

reshard

PyTorch DCP 和 FSDP2 最有用的一点，是它们允许恢复时使用和保存时不同的执行拓扑。前提是你保存的是分布式语义，而不是某个 rank 的私有布局。

代码
# 概念骨架：先拿到分布式 state_dict，再交给 DCP 处理。
state = {"model": model, "optim": optimizer, "step": step}
save(state, checkpoint_id)

# 恢复时不是直接替换对象，而是把状态写回已有对象。
loaded = load(checkpoint_id)
load_state_dict(model, loaded["model"])
load_state_dict(optimizer, loaded["optim"])

异步保存与发布

async

异步保存只解决一件事：让前台训练少等一会儿。它不自动解决“这个 checkpoint 什么时候算正式可用”。所以异步保存要和完成标记、版本发布、失败回收一起看。

代码
1. 分片文件都写完了吗
2. common state / 元数据都齐了吗
3. 保存的 step、RNG、调度器状态是否和训练步对齐
4. 恢复时是否还支持当前 world size
5. latest 是否只在验证后更新

恢复演练

case

把 checkpoint 的逻辑放到一个具体场景里，最容易看出哪里真的会错。下面这个例子故意选得不算极端：保存时 64 张卡，恢复时只能用 32 张卡；保存端用了分片状态，恢复端要按新的拓扑重新装配。这训练集群里很常见的迁移。

恢复动作保存时看到什么恢复时得到什么读什么写什么如果省掉这步会怎样
读 manifeststep、world size、分片策略、RNG 快照知道这份 checkpoint 属于哪一轮训练版本、入口、元数据路径恢复计划和校验策略目录能打开，但不知道该按什么语义恢复
加载分片 state_dict参数和优化器都以 shard 形式存在新 world size 下的可执行对象模型结构、参数名、分片切法新布局下的模型和 optimizer 状态文件能读，训练却和旧拓扑绑死
恢复 scheduler / step学习率曲线和全局步数一起保存恢复后从正确的学习率位置继续global step、warmup、decay 位置下一步的 LR 和日志起点看起来恢复了，其实像重新启动
最小冒烟测试checkpoint 已经存在且完整确认能继续训练而不是只会加载首个 batch 的 loss、step、LR、样本位置验证结果和告警状态文件没坏，但训练语义可能已经漂了
代码
1. 先读 manifest 和完成标记，确认这是完整版本
2. 再加载分片 state_dict，确认模型和 optimizer 都能写回
3. 接着恢复 scheduler、step 和 RNG
4. 最后跑一个 batch 的冒烟测试，看 loss 和 LR 是否对齐

为什么这个例子够用

它同时暴露了三类问题：保存格式问题、拓扑迁移问题和训练语义问题。只看其中一类，都会让你误以为 checkpoint 已经“差不多能用”。

发布后复查

verify

checkpoint 真正上线以后，还要过一遍复查。复查的对象“这份目录是不是我们想要的那一版”。最常见的错误，是目录已经完整了，但 step、world size、分片数或者 manifest 版本跟当前训练不一致。

复查动作读取对象写回对象如果对不上会看到什么下一步怎么做
核对 manifeststep、world size、分片策略、版本号复查结果和告警状态目录看着完整，但不知道是不是正确版本先把这份 checkpoint 标成待确认，不要继续覆盖它
核对首个 batchloss、LR、RNG、样本游标冒烟测试结果模型能跑，但 loss 或 LR 不在预期位置回到 scheduler 和数据游标查恢复链路
核对分片数当前 world size 与保存时 shard 布局重新分片后的布局加载不报错，但某些 rank 的 shard 分布不均回到 DCP / FSDP2 的 reshard 规则复核
核对回滚点上一个完整 checkpoint 的路径恢复入口坏目录被误当作最新版本把 latest 指回上一个确认完成的版本
代码
def verify_checkpoint_dir(path, expected_step, expected_world_size):
 manifest = read_manifest(path)
 assert manifest["step"] == expected_step
 assert manifest["world_size"] == expected_world_size
 assert manifest["status"] == "complete"
 step, lr, loss = run_one_batch_smoke_test(path)
 return {
 "step_ok": step == expected_step + 1,
 "lr_ok": lr_matches_schedule(lr),
 "loss_ok": loss_is_reasonable(loss),
 }

复查和下一次保存的关系

复查下一次保存的前置条件。只有这次 checkpoint 被确认完整，下一轮才有资格把它作为 rollback 点或 latest 基线。

存储契约

storage

checkpoint 的最后一层，存储契约。你要知道保存端把东西写到哪里，恢复端从哪里找，失败时哪些目录可以删，哪些目录必须保留。对象存储、本地盘、共享文件系统的行为不一样，回滚策略也不能混着写。

存储后端保存动作恢复动作失败时保留什么失败时清理什么
本地盘先写临时目录，再换 latest从最新完整目录读取上一个完整 checkpoint 和 manifest写到一半的 tmp 目录
共享文件系统各 rank 协作写分片按 shard 和策略重建布局manifest、common state、分片文件校验失败的半成品 shard
对象存储写桶、提交元数据、再发布指针从发布点或索引读取完整版本和索引文件未提交的临时对象前缀
代码
1. 只回滚到 manifest 标记为 complete 的目录
2. tmp 目录和写到一半的 shard 可以删
3. latest 只能在新版本验完后更新
4. 恢复失败先切回旧版本，再查新的保存链路

常见错误

fixes
错误为什么危险修法
只保存模型，不保存优化器 / scheduler / step恢复后动量、二阶矩和学习率轨迹断掉，训练看起来像重新开始。把恢复状态列成清单，保存和加载都按同一份 schema 检查。
让 rank 0 聚合全量状态再保存大模型下会把内存峰值、通信开销和单点失败一起放大。用 DCP 或 Megatron Core 的分布式 checkpoint，让各 rank 保存自己的 shard。
异步保存后立刻更新 latest后台还没写完，恢复端就会先撞上半成品。先写临时目录、再做完整性校验、最后原子发布。
只看文件能不能打开文件能读不代表训练语义正确，step、LR、RNG、数据游标都可能错。恢复后立刻做最小冒烟测试：step 对齐、LR 对齐、样本游标对齐。
把 load-time resharding 当成自动万灵药它只能在保存语义正确时工作，不能修复模型命名、布局或 optimizer 状态错误。先核对 FSDP2 / DCP 的分布式 state_dict，再谈新的 world size。
把 Megatron Core checkpoint 当成普通 torch.save会漏掉 common state、分片策略和 distributed optimizer 相关布局。按 dist_checkpointing 的语义检查 common / sharded / strategy 三部分。

Q: 解释为什么 rank 0 聚合全量状态不是大模型 checkpoint 的好方案。

参考答案 / 评分要点

。好的答案会说清楚：rank 0 如果要聚合全量状态，就会背上所有参数、优化器状态和元数据的峰值内存；通信上会把别的 rank 的 shard 都集中到一处，I/O 也会被单点拖慢；一旦 rank 0 出问题，整个 checkpoint 也可能失效。替代方案应该指向分布式 checkpoint、分片保存或由多个 rank 共同完成写出。
