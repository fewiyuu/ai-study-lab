# 学习目标 {#goals}
<!-- tag: goal -->

把 checkpoint 当成训练状态
这一页先把“保存模型文件”改成“保存一轮训练继续所需的状态”。模型、优化器、scheduler、step、随机数、数据游标和并行元数据都要放进同一张账本里看。


把恢复当成新拓扑输入
World size 变了、rank 编排变了、分片形状变了，都不自动等于失败。DCP 和 FSDP2 允许按新的执行环境重新分片，但前提是状态语义没有丢。


把发布当成一笔事务
checkpoint 不该在一半写完时就被当成最新版本。先写临时目录，再提交完成标记，最后原子更新可见入口，这条顺序要比 API 名字更重要。

# 保存链路 {#save-flow}
<!-- tag: flow -->

真正可靠的 checkpoint 一个小事务：先把状态写进临时位置，再确认它完整，最后才把它变成对外可见的最新版本。这个顺序比文件名更重要。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

checkpoint 事务骨架

```python
tmp_dir = save_dir + ".tmp"
save_dcp(state_dict=model_and_optim, storage=tmp_dir)
write_manifest(tmp_dir, step=current_step, rng=rng_state)
verify_checkpoint(tmp_dir)
publish_latest_atomically(tmp_dir, save_dir)
```

## box soft
### 这段骨架读什么
先读训练状态，再读临时目录，再读 manifest，最后才发布。只要“验证”和“发布”之间被打断，恢复端就可能把半成品看成正式版本。

## box info
### 这段骨架写什么
写进去的不只是 tensor，还包括恢复需要的步数、随机状态和布局信息。最常见的错误是只写权重，把其他状态留在进程里。

# 重新分片 {#reshard}
<!-- tag: reshard -->

PyTorch DCP 和 FSDP2 最有用的一点，是它们允许恢复时使用和保存时不同的执行拓扑。前提是你保存的是分布式语义，而不是某个 rank 的私有布局。

场景
保存时
恢复时
读什么
写什么
先验判断


同 world size 恢复
分片 state_dict
同一拓扑
模型名、参数名、分片切法
原样恢复参数和优化器状态
最容易验证，但不能证明迁移能力


不同 world size 恢复
分片 state_dict
新的 rank 数
逻辑参数、加载策略、reshard 规则
按新拓扑重切 shard
如果语义保存对了，就能继续训练


FSDP2 fully_shard
模型在 forward/backward 外 fully sharded
重新装配可执行模型
FQNs、state_dict 结构、模块层级
参数、梯度和优化器状态的恢复布局
要先确认不是把全量权重当成分片权重

DCP / FSDP2 读写顺序

```python
# 概念骨架：先拿到分布式 state_dict，再交给 DCP 处理。
state = {"model": model, "optim": optimizer, "step": step}
save(state, checkpoint_id)

# 恢复时不是直接替换对象，而是把状态写回已有对象。
loaded = load(checkpoint_id)
load_state_dict(model, loaded["model"])
load_state_dict(optimizer, loaded["optim"])
```

## box warn
### 这里的关键句
`state_dict()` 的名字没有变，但背后的布局已经不是单卡时代那套了。FSDP2 里，分片是默认语义的一部分；DCP 做的是把这套语义保存得能被新拓扑读回来。

# 异步保存与发布 {#async-publish}
<!-- tag: async -->

异步保存只解决一件事：让前台训练少等一会儿。它不自动解决“这个 checkpoint 什么时候算正式可用”。所以异步保存要和完成标记、版本发布、失败回收一起看。

指针比内容先更新时，恢复会读到坏目录

发布前检查清单

```text
1. 分片文件都写完了吗
2. common state / 元数据都齐了吗
3. 保存的 step、RNG、调度器状态是否和训练步对齐
4. 恢复时是否还支持当前 world size
5. latest 是否只在验证后更新
```

## box info
### Megatron Core 的对应点
Megatron Core 的 `filesystem_async` 策略把“怎样得到保存函数”和“怎样异步执行保存”拆开。你真正要检查的是：写桶、验证、发布三个阶段有没有被放进正确的顺序。

# 恢复演练 {#recovery-drill}
<!-- tag: case -->

把 checkpoint 的逻辑放到一个具体场景里，最容易看出哪里真的会错。下面这个例子故意选得不算极端：保存时 64 张卡，恢复时只能用 32 张卡；保存端用了分片状态，恢复端要按新的拓扑重新装配。这训练集群里很常见的迁移。

这个场景里，最先不能丢的是模型参数和优化器状态，最先不能乱的是 step 和学习率。要是你只盯着权重文件，恢复后可能照样能前向，但训练轨迹已经变了。真正该检查的，是保存端有没有把“谁持有什么 shard”以及“当前训练走到哪一步”同时写进去。

恢复动作
保存时看到什么
恢复时得到什么
读什么
写什么
如果省掉这步会怎样


读 manifest
step、world size、分片策略、RNG 快照
知道这份 checkpoint 属于哪一轮训练
版本、入口、元数据路径
恢复计划和校验策略
目录能打开，但不知道该按什么语义恢复


加载分片 state_dict
参数和优化器都以 shard 形式存在
新 world size 下的可执行对象
模型结构、参数名、分片切法
新布局下的模型和 optimizer 状态
文件能读，训练却和旧拓扑绑死


恢复 scheduler / step
学习率曲线和全局步数一起保存
恢复后从正确的学习率位置继续
global step、warmup、decay 位置
下一步的 LR 和日志起点
看起来恢复了，其实像重新启动


最小冒烟测试
checkpoint 已经存在且完整
确认能继续训练而不是只会加载
首个 batch 的 loss、step、LR、样本位置
验证结果和告警状态
文件没坏，但训练语义可能已经漂了

恢复时的检查顺序

```text
1. 先读 manifest 和完成标记，确认这是完整版本
2. 再加载分片 state_dict，确认模型和 optimizer 都能写回
3. 接着恢复 scheduler、step 和 RNG
4. 最后跑一个 batch 的冒烟测试，看 loss 和 LR 是否对齐
```

## box soft
### 这条链怎么和前面接上
前面的“保存链路”解决的是怎么把状态安全写出去；这里的“恢复演练”解决的是怎么把状态安全读回来。两边的关键字不一样，但顺序是对称的：先确认语义，再确认布局，最后确认训练行为。

## box info
### 为什么这个例子够用
它同时暴露了三类问题：保存格式问题、拓扑迁移问题和训练语义问题。只看其中一类，都会让你误以为 checkpoint 已经“差不多能用”。

# 发布后复查 {#post-check}
<!-- tag: verify -->

checkpoint 真正上线以后，还要过一遍复查。复查的对象“这份目录是不是我们想要的那一版”。最常见的错误，是目录已经完整了，但 step、world size、分片数或者 manifest 版本跟当前训练不一致。

这一步的目标很简单：把“能打开”升级成“能继续训练，而且知道自己在继续哪一轮”。如果你已经恢复出了模型，却还没确认样本游标和学习率曲线，那只能说这份 checkpoint 读进来了，还不能说它可运营。

复查动作
读取对象
写回对象
如果对不上会看到什么
下一步怎么做


核对 manifest
step、world size、分片策略、版本号
复查结果和告警状态
目录看着完整，但不知道是不是正确版本
先把这份 checkpoint 标成待确认，不要继续覆盖它


核对首个 batch
loss、LR、RNG、样本游标
冒烟测试结果
模型能跑，但 loss 或 LR 不在预期位置
回到 scheduler 和数据游标查恢复链路


核对分片数
当前 world size 与保存时 shard 布局
重新分片后的布局
加载不报错，但某些 rank 的 shard 分布不均
回到 DCP / FSDP2 的 reshard 规则复核


核对回滚点
上一个完整 checkpoint 的路径
恢复入口
坏目录被误当作最新版本
把 latest 指回上一个确认完成的版本

复查脚本骨架

```python
def verify_checkpoint_dir(path, expected_step, expected_world_size):
 manifest = read_manifest(path)
 assert manifest["step"] == expected_step
 assert manifest["world_size"] == expected_world_size
 assert manifest["status"] == "complete"
 step, lr, loss = run_one_batch_smoke_test(path)
 return {
 "step_ok": step == expected_step + 1,
 "lr_ok": lr_matches_schedule(lr),
 "loss_ok": loss_is_reasonable(loss),
 }
```

## box soft
### 这段复查链的作用
它把“看目录”变成“看语义”。如果只看文件存在与否，你会漏掉版本漂移、分片漂移和学习率漂移；如果连一个 batch 都不跑，你也发现不了恢复后轨迹已经变了。

## box info
### 复查和下一次保存的关系
复查下一次保存的前置条件。只有这次 checkpoint 被确认完整，下一轮才有资格把它作为 rollback 点或 latest 基线。

# 存储契约 {#storage-contract}
<!-- tag: storage -->

checkpoint 的最后一层，存储契约。你要知道保存端把东西写到哪里，恢复端从哪里找，失败时哪些目录可以删，哪些目录必须保留。对象存储、本地盘、共享文件系统的行为不一样，回滚策略也不能混着写。

这也是为什么异步保存不能只看“后台线程在跑”。如果存储契约没有定义清楚，后台跑得再快也只是把错误更快地写出去。对于大模型训练，最值钱的你能不能在任务被抢占、节点重启或者版本回退时，快速找到上一版完整状态。

存储后端
保存动作
恢复动作
失败时保留什么
失败时清理什么


本地盘
先写临时目录，再换 latest
从最新完整目录读取
上一个完整 checkpoint 和 manifest
写到一半的 tmp 目录


共享文件系统
各 rank 协作写分片
按 shard 和策略重建布局
manifest、common state、分片文件
校验失败的半成品 shard


对象存储
写桶、提交元数据、再发布指针
从发布点或索引读取
完整版本和索引文件
未提交的临时对象前缀

回滚规则

```text
1. 只回滚到 manifest 标记为 complete 的目录
2. tmp 目录和写到一半的 shard 可以删
3. latest 只能在新版本验完后更新
4. 恢复失败先切回旧版本，再查新的保存链路
```

## box warn
### 把契约写清楚，排错才快
一旦你把“哪些目录算正式版本”“哪些目录可以回收”“出错时指针回到哪里”写清楚，后面无论是 DCP、FSDP2 还是 Megatron Core，排错思路都会统一成同一套。

# 常见错误 {#mistakes}
<!-- tag: fixes -->

错误
为什么危险
修法


只保存模型，不保存优化器 / scheduler / step
恢复后动量、二阶矩和学习率轨迹断掉，训练看起来像重新开始。
把恢复状态列成清单，保存和加载都按同一份 schema 检查。


让 rank 0 聚合全量状态再保存
大模型下会把内存峰值、通信开销和单点失败一起放大。
用 DCP 或 Megatron Core 的分布式 checkpoint，让各 rank 保存自己的 shard。


异步保存后立刻更新 latest
后台还没写完，恢复端就会先撞上半成品。
先写临时目录、再做完整性校验、最后原子发布。


只看文件能不能打开
文件能读不代表训练语义正确，step、LR、RNG、数据游标都可能错。
恢复后立刻做最小冒烟测试：step 对齐、LR 对齐、样本游标对齐。


把 load-time resharding 当成自动万灵药
它只能在保存语义正确时工作，不能修复模型命名、布局或 optimizer 状态错误。
先核对 FSDP2 / DCP 的分布式 state_dict，再谈新的 world size。


把 Megatron Core checkpoint 当成普通 `torch.save`
会漏掉 common state、分片策略和 distributed optimizer 相关布局。
按 `dist_checkpointing` 的语义检查 common / sharded / strategy 三部分。

### 4 open

**Q:** 解释为什么 rank 0 聚合全量状态不是大模型 checkpoint 的好方案。

Explain: 好的答案会说清楚：rank 0 如果要聚合全量状态，就会背上所有参数、优化器状态和元数据的峰值内存；通信上会把别的 rank 的 shard 都集中到一处，I/O 也会被单点拖慢；一旦 rank 0 出问题，整个 checkpoint 也可能失效。替代方案应该指向分布式 checkpoint、分片保存或由多个 rank 共同完成写出。
