训练系统选型与实验阅读

这一页接着 07 往上收束，把 checkpoint、恢复、并行维度和训练报告放回同一张选型表里。重点不是记住“FSDP2 还是 Megatron Core 哪个更强”，而是先判断你的瓶颈属于状态分片、模型内部切分，还是实验证据不够。

课程地图上一单元

读完本页，你应该能做什么

  • 能把训练需求翻成三个维度：状态压力、并行切法和恢复约束。
  • 能读懂一份实验报告有没有把峰值显存、吞吐、通信和恢复成本说全。
  • 能写出一个最小验证表，说明为什么先从 FSDP2 或 Megatron Core 其中一边起步。

课程位置

position

这页接在 06 和 07 后面

06 讲激活检查点、混合精度和优化器状态，07 讲 checkpoint、恢复和容错。08 不再往低层追，而是把这些状态放进选型和实验阅读框架里，问清楚“该从哪种训练栈起步”。

这页要解决什么

很多训练报告只给一个 tokens/s 或一个最大规模结果。真正需要的是：这组数字支持了什么结论，没覆盖什么风险，是否足够支撑换框架。

边界

这里不替你下最终工程结论，也不替你改代码。它只给一个可以落地的判断顺序：先看约束，再看并行维度，最后看证据是否完整。

先抓主线

route

选型顺序最好不是“先问框架名”，而是“先问问题属于哪一类”。如果主要压力来自参数、梯度和优化器状态，FSDP2 往往更直接；如果单层、长序列或 MoE 已经需要切进模型内部，Megatron Core 的并行构件更贴题。

再往上看，实验阅读的重点是把结果拆成三层：稳态吞吐、峰值和尾部成本、恢复和迁移成本。只看平均 tokens/s，结论很容易跑偏。

读这页时先记住的四个词

  • 状态：模型、优化器、scheduler、RNG、数据游标。
  • 切法：DP、FSDP、TP、PP、CP、EP。
  • 证据：显存、吞吐、通信、恢复时间。
  • 约束：团队熟悉度、版本稳定性、可维护性。

术语账本

terms
术语 这页里的意思 容易误解成什么
FSDP2 / fully_shard 把参数、梯度和优化器状态放进分片语义，保留 PyTorch 模型结构。 以为它会自动替你解决所有通信和流水线问题。
Megatron Core 把 TP、PP、CP、EP、分布式优化器和 checkpoint 组织成训练构件库。 以为它只是一个更大的 DDP 包装器。
DCP / Distributed Checkpoint 按分布式 state_dict 保存和恢复状态，支持 load-time resharding。 以为它只是把文件拆成很多块。
实验阅读 看设置、看指标、看消融、看失败边界，再判断结论是否成立。 以为看到一个最大吞吐数字就能下结论。

判断对象 先读什么 写出什么结论 错了以后先看哪里

团队约束 调试速度、维护成本、版本升级 先起步还是先对照实验 团队是否有能力长期维护更复杂的并行栈

判断矩阵

matrix
判断维度 更偏向 FSDP2 更偏向 Megatron Core 必须实测
模型改动频率 模型结构还在频繁变化，想保留 PyTorch 原生迭代速度。 模型结构较稳定，愿意围绕并行构件重组训练栈。 包装或并行配置是否阻碍调试和迭代。
主要压力 参数、梯度、优化器状态占显存过大。 单层矩阵、长序列、pipeline 阶段或 MoE 专家需要切分。 峰值显存发生在哪个训练阶段。
并行维度 数据并行 + 参数分片就能覆盖主要需求。 需要 TP、PP、CP、EP 等多维组合。 并行维度增加后通信是否抵消收益。
checkpoint 希望和 PyTorch 生态、状态字典、现有保存逻辑更接近。 需要分片 checkpoint、跨并行维度恢复和训练栈级恢复策略。 保存、加载、迁移并行配置、故障恢复耗时。
团队能力 团队主要熟悉 PyTorch，系统人手有限。 团队能维护并行配置、启动脚本、模型适配和性能诊断。 新人上手、线上排障和版本升级成本。
读法 先看这一行的瓶颈属于哪一类，再看它会把你推向哪种栈。不要先问“这个框架行不行”，先问“这件事到底切什么”。

微案例

mini-case

同样是 8xGPU，为什么起点不同

如果你的 7B 级模型还在频繁改层数、attention heads 和激活检查点，先从 FSDP2 起步，能把状态压力和 PyTorch 代码迁移成本先压住；如果模型结构已经定型，但单层矩阵、长序列和通信都在顶满，先看 Megatron Core，才能把 TP / PP / CP / EP 的组合摆到位。

失败会长什么样

起步选错时，最常见的不是“完全跑不动”，而是你花了很多时间调 tokens/s，却仍然在 checkpoint、恢复或跨节点扩展上卡住。这个时候回去看症状和证据，比继续换参数更有效。

场景 先选什么 看什么证据 下一步去练习哪题
模型还在频繁改 FSDP2 状态压力、包装边界、恢复时间 练习 4、6、9 题
模型结构已定型 Megatron Core TP / PP / CP / EP、通信占比、扩展曲线 练习 5、7、8、11 题

实验阅读

reading

先读问题，不先读结论

把论文、文档或报告里的问题改写成一句话：它是在证明更省显存、更高吞吐、更好扩展、更易恢复，还是更容易迁移。目标不清楚，后面的数字就容易被误读。

记录配置，不只记录结果

至少摘出模型参数量、层数、hidden size、attention heads、序列长度、global batch、micro-batch、精度、GPU 型号、节点数、互联、并行维度、是否重算、优化器和 checkpoint 策略。

分开看峰值、稳态和尾部成本

稳态吞吐只是中间段；峰值可能出现在 forward、backward、optimizer step、all-gather 或 checkpoint 附近；尾部成本包括保存、恢复、排队和数据加载抖动。

看消融是否对应关键机制

如果结论是 FSDP2 包装策略更好，就应看到包装边界、reshard、混合精度或 checkpoint 的对比。如果结论是 Megatron Core 多维并行更好，就应看到 TP/PP/CP/EP 或 micro-batch 配置变化的对比。

看扩展曲线，不只看最大点

一个最大规模结果有价值，但选型更需要曲线：从 1 节点到多节点，从短序列到长序列，从小 batch 到大 batch，效率在哪里开始掉头。掉头位置通常暴露真正瓶颈。

把结论改写成自己的验收实验

读完不要只保留“某系统更快”。把结论改写成你的最小实验：在你的 GPU、模型、序列长度、batch 和团队约束下，跑 3 个对照，收集 5 个指标，决定下一步。

下面不是推荐固定参数，而是示范如何把口头需求变成可以执行的实验。真实项目要替换成自己的模型、硬件和版本。

selection-plan.md 实验计划骨架 
目标：在 8xGPU 单节点上训练 7B 级 Transformer，先验证可行性，再决定是否扩展到多节点。

候选 A：PyTorch FSDP2
- 并行：数据并行 + 参数/梯度/优化器状态分片
- 关注：模块包装边界、reshard 策略、激活检查点、state_dict 保存/恢复
- 指标：峰值显存、tokens/s、OOM 位置、checkpoint 保存/恢复时间

候选 B：Megatron Core
- 并行：数据并行 + 张量并行，必要时加入流水线并行
- 关注：TP 大小、micro-batch、pipeline bubble、分布式优化器、分片 checkpoint
- 指标：tokens/s、MFU、通信占比、pipeline bubble、恢复路径

验收：
1. 训练 200 step 无 OOM、无 loss 异常跳变。
2. 保存 checkpoint 后能在同配置恢复。
3. 改变 micro-batch 后曲线变化可解释。
4. 记录完整启动命令、版本、GPU 拓扑和关键日志。

同一组数字，为什么会得出不同结论

假设你手里有两份报告。A 方案是 FSDP2，稳态 tokens/s 稍低，但 checkpoint 保存和恢复更短，模型结构还在频繁改；B 方案是 Megatron Core，稳态 tokens/s 更高，但并行配置更早冻结，checkpoint 和恢复成本更高。两份报告都可能“对”，前提是你关心的目标不同。

如果目标是研究代码快速迭代、模型结构仍在大改，A 的分数会更高，因为它把调试和迁移成本压低了。如果目标是长时间稳定预训练、模型内部已经必须切成 TP / PP / CP / EP，B 的分数会更高，因为它更贴近大规模并行的结构。这里的关键不是把数字平均一下，而是把目标、代价和恢复要求一起放进判断。

对照项 方案 A：FSDP2 起步 方案 B：Megatron Core 起步 你要补问什么
稳态吞吐 可能不是最高，但常常足够接近可用。 通常更容易把大规模并行的吞吐推高。 你的目标是研究迭代速度，还是极限吞吐？
状态和恢复 更贴近 PyTorch 生态，保存 / 恢复路径通常更容易接住。 保存语义更依赖并行构件和分布式 checkpoint。 checkpoint 保存、恢复、迁移世界规模时要不要重分片？
工程维护 团队更容易在现有训练代码上继续改。 需要更早冻结模型结构和并行配置。 团队有没有能力长期维护更复杂的启动脚本和排障流程？
实验证据 应该看到包装边界、reshard、混合精度或 checkpoint 对比。 应该看到 TP / PP / CP / EP 或 micro-batch 的对比。 作者有没有把结论和关键机制对上？
decision-notes.txt 判断笔记骨架 
1. 当前主要压力是什么：状态、并行还是恢复。
2. 这份报告的目标是什么：稳态吞吐、扩展效率、恢复能力，还是迁移成本。
3. 结论和证据是否对上：包装边界、reshard、TP/PP/CP/EP、checkpoint、恢复时间。
4. 如果只有一个最大 tokens/s，就先把它当作不完整结果，不直接下最终结论。
5. 如果要起步，就选那个能最快把你的主约束暴露出来的方案。

把结果翻成结论时，先看三件事

第一，看报告是不是把“稳态吞吐”和“尾部成本”分开了。如果没有分开，就先把 checkpoint 和恢复成本补出来，再看谁更适合起步。第二，看结论是不是和关键机制对上了：FSDP2 的证据应该解释状态压力和包装边界，Megatron Core 的证据应该解释多维并行和布局成本。第三，看你现在的问题是不是已经到了需要改栈的级别，还是只需要调批次、重算或 checkpoint 语义。

这一层不难，但特别容易被跳过。很多人看到一个更高的 tokens/s 就直接抄结论，结果到恢复、跨节点、或者长序列阶段才发现并行方式并没有覆盖真正的瓶颈。选型报告读到最后，不是收一个名字，而是收一个可执行的下一步。

你看到的结论 它真正支持什么 还缺什么才能下决定
FSDP2 更容易起步 模型结构还在变、状态压力更大、PyTorch 迭代速度更重要。 还要确认 checkpoint、恢复和跨节点扩展是不是能接受。
Megatron Core 吞吐更高 模型内部并行已经必要，且团队能承受更复杂的并行构件。 还要确认保存、恢复、调试和版本升级成本。
tokens/s 最高 稳态训练速度更好。 还要补峰值显存、通信占比、checkpoint 和恢复时间。
恢复没问题 文件层面和部分语义层面都能接住。 还要确认 step、RNG、采样游标和并行布局都一致。

故障实验

lab

先看症状，再看结论

选型时最容易出错的，不是“没听过某个框架”，而是把一个表面症状直接翻译成框架结论。tokens/s 只能告诉你稳态吞吐，不能告诉你 checkpoint 太慢、恢复太慢、跨节点通信太重，也不能告诉你是不是因为并行维度选错了。

所以诊断顺序要反过来：先看症状属于状态压力、并行切法、恢复代价，还是实验设置本身有问题；再决定是先看 FSDP2，还是先看 Megatron Core，或者先回去补实验变量。

先别追的四件事

不要一看到慢就换栈，不要一看到 OOM 就强行 TP/PP，不要一看到 checkpoint 失败就只盯存储，也不要一看到单点结果就外推到多节点。你现在要学的是排查顺序，不是结论口号。

症状 先问的问题 先看什么证据 暂时别下什么结论
tokens/s 很高，但保存和恢复都慢 checkpoint 时间是不是被吞掉了 保存耗时、恢复耗时、latest 更新顺序 不要只说训练栈更快
optimizer step 处 OOM 是不是优化器状态和主参数太肥 优化器状态占用、显存峰值位置、state_dict 语义 不要立刻改成更复杂的模型并行
长序列下速度掉得很厉害 是不是 attention 激活、通信或 bubble 在作怪 TP / CP / PP 配置、micro-batch、重算开关 不要只拿短序列结果解释长序列
恢复后 loss 轨迹变了 是不是 step、RNG、采样游标没一起恢复 scheduler、global step、RNG、数据游标、完成标记 不要只说“文件能读”就算恢复成功
跨节点扩展很差 是通信、并行维度还是版本假设的问题 通信占比、扩展曲线、并行配置、版本快照 不要只凭单节点结论换系统 

    
    
    
  

  

    diagnosis-order.txt
    排查顺序
    
  

  
1. 先判断症状属于状态压力、并行切法、恢复代价，还是实验设置有问题。
2. 再找能直接解释症状的证据：峰值、通信、checkpoint、step、RNG、数据游标。
3. 如果证据只来自单点或单卡，就补一组对照实验，不要立刻换栈。
4. 如果结论需要跨节点或跨 world size，先回到公开入口核对版本和并行语义。
5. 只有当问题被同一类证据重复指向时，才把 FSDP2、Megatron Core 或 DCP 放到结论里。

  

    
这一段真正想训练你什么

    
训练系统选型不是“选一个最强的名字”，而是“在一堆不完整的证据里，先把最可能解释症状的那层挑出来”。这一步做对了，后面读实验、看 checkpoint、看恢复路径，才不会总是跳层。

  

  

    report-readout.txt
    报告读法
    
  

  
如果 tokens/s 高，但 save / resume 慢：
  先补 checkpoint 保存时间和恢复时间，再看 latest 发布顺序。

如果 optimizer step OOM：
  先补优化器状态、主参数 shard、mixed precision 路径，再看是否需要先从 FSDP2 起步。

如果长序列掉速：
  先补 TP / CP / PP、micro-batch 和 recompute，再看通信占比和 bubble。

如果跨节点扩展差：
  先补扩展曲线和版本快照，再判断是网络还是并行配置。

  

    
和练习怎么连起来

    
这段读法正好对应练习里的报告阅读题、场景判断题和复述题。你先用它把症状翻成“缺什么证据”，再回到练习去写完整答案。若你只能多补一项，优先补 checkpoint 和恢复时间；若你只能多改一项并行配置，优先改最能暴露压力来源的那项，而不是先换整个训练栈。

  






  

    
常见误判

    debug
  

  

    

      
误判 1：把框架名当结论

      
“我们要用 Megatron”或“我们要用 FSDP2”不是选型结论，只是候选方案。真正的结论必须包含约束、配置、指标和验证计划。

    

    

      
误判 2：只看平均吞吐

      
平均 tokens/s 可能掩盖 checkpoint、数据加载、通信抖动和恢复成本。预训练任务不能只用一个稳态数字做决策。

    

    

      
误判 3：把小规模实验外推到大规模

      
单节点实验看不出跨节点网络瓶颈，短序列实验看不出上下文并行压力，小 batch 实验看不出流水线填充效果。

    

    

      
误判 4：忽略 checkpoint 形态

      
训练跑通不代表系统可用。分片状态、并行维度、优化器状态和随机数状态保存不完整，会让恢复或迁移变成事故。

    

    

      
误判 5：同时改太多变量

      
如果同时改模型、batch、精度、并行维度和框架，就无法知道收益来自哪里。一次只改一个主变量，才知道证据指向什么。

    

    

      
误判 6：忽略人的调试成本

      
复杂训练栈的性能上限更高，但调试成本也更高。没人能定位并行配置和数值问题时，高上限反而会拖慢项目。

    

  





      

        

          
练习

          diagnostic
        

        

  

    作答目标
    
不要只报框架名。每题尽量写出“先看什么、为什么、下一步验证什么”。

  

  

    检查重点
    
你是否能把状态压力、并行维度、实验指标和恢复边界放在同一张判断表里。

  






  

    

      
一、概念识别

      
先确认你不会把选型问题和恢复问题混成一类。

    

    

      检查点
      先判压力，再判切法，再判证据。
    

  

  

    

      
choicestate pressure

      
FSDP2 更直接解决哪类问题？

      

        
        
        
      

      
      

      
想一想名字里“shard”主要切什么。

    


    

      
true/falseparallelism

      
Megatron Core 只是 DDP 包装器吗？

      
      
      
      

      
回忆它提供的 TP、PP、CP、EP 和 distributed optimizer。

    


    

      
mini-fieldsdecision order

二、场景判断

      
把实际需求翻译成候选方案和验证计划。

    

    

      答题方式
      每题至少写出起点、风险和要验证的指标。
    

  

  

    

      
choiceFSDP2

      
场景 A：研究代码频繁改模型，单节点 8 卡，主要 OOM 在 optimizer step

      

        
        
        
      

      
      

      
optimizer step OOM 指向优化器状态；研究代码频繁变化影响迁移成本。

    


    

      
choiceMegatron Core

      
场景 B：长序列预训练，attention 激活和通信成为主压力

      

        
        
        
      

      
      

      
不要只写 TP；长序列通常还要看序列/上下文相关切分和重算。

    


    

      
openexperiment design

      
把一句结论改写成验收实验

      
      
提示：固定变量、列指标、说明通过/不通过标准。

      
      
参考答案 / 评分要点
合格答案会把“快 20%”拆成可重复的实验：固定模型、序列长度、batch 和版本，只改并行或 checkpoint 策略；分别记录稳态 tokens/s、峰值显存、通信占比、checkpoint 时间和恢复结果；最后写清楚通过条件，例如 200 step 内无 OOM、恢复后第一步 loss 和 step 对齐。只写“再跑一次看看”不算验收实验。

      

    

  






  

    

      
三、报告阅读

      
练习把报告里的“结果”改写成“还缺什么证据”。

    

    

      最低要求
      看设置、看消融、看尾部成本。
    

  

  

    

      
choiceevidence gap

      
一个实验只给最大规模 tokens/s，你先追问什么？

      

        
        
        
      

      
      

      
只有一个吞吐数字时，缺的往往不是算术，而是上下文。

    


    

      
mini-fieldsdiagnosis

      
填空：扩展效率下降时先看哪三项

      

        
        
        
      

      
      

      
不要只写“更慢了”，要写出最可能先掉的环节。

    


    

      
opentail cost

      
解释：为什么恢复成本不能从 tokens/s 里看出来？

      
      
回答结构：先说 tokens/s 描述什么，再说它漏掉什么，最后给出补充指标。

      
      
参考答案 / 评分要点
好的答案会说明 tokens/s 主要描述稳态吞吐，不包含保存 checkpoint、恢复 checkpoint、排队等待、失败后重启和重新装配状态的成本。选型时如果只看 tokens/s，可能会把保存时间很长、恢复很慢的方案误判成更优。补充指标应该包括 checkpoint 保存/恢复时间、崩溃后的恢复成功率、恢复后第一步是否对齐，以及端到端运行成本。

参考答案 / 评分要点
高分答案会先说，这页不是再讲 checkpoint 本身，而是把 06 和 07 的状态、保存和恢复拿来做选型。接着讲判断顺序：先看问题属于状态压力还是模型内部切分，再看要不要 DP、FSDP2、TP、PP、CP、EP 组合，最后看报告有没有给出吞吐、峰值、恢复和尾部成本。再往下要指出最小入口是公开训练循环和 GPT 预训练脚本，而不是只看一个最大 tokens/s。
