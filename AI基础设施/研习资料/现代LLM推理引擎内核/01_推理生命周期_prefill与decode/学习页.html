先把整条主线看见

overview

LLM 推理先把 prompt 变成可复用的注意力状态，再一 token 一 token 地生成后续内容。前半段叫 prefill，后半段叫 decode。两段都在跑模型，但读写的对象和瓶颈不同。

prefill 做什么

prefill 读取完整 prompt，计算各层的 key / value，并把它们写进 KV cache。首 token 之前的等待，大头通常在这里。

decode 做什么

decode 读取已有 KV cache 和当前 token，每轮只推进一个新 token，并把新 token 的 key / value 追加进去。流式输出的节奏主要由这里决定。

这也是为什么同一个请求会同时出现两个体验：首 token 很慢，但后面一旦开始流式输出，看起来就快了；也可能首 token 不慢，但中间一直卡顿，因为 decode 的小步循环被队列和资源压力打断了。

prefill 和 decode 的读写顺序

trace

这一段只围绕一个锚定例子：同一条请求，prompt 里有 3000 个 token，最终要生成 400 个 token。这个例子足够长，能把 prefill、decode 和流式输出之间的关系拉开，又不会长到看不清主线。

prefill 读完整 prompt 和模型权重，写入每层的 K / V。

看点：prefill 的产物是后续 decode 能继续读的历史状态，答案要等后面的采样和停止条件一起决定。

decode 读历史 K / V 和当前 token，写入新 token 的 K / V。

看点：decode 在已有状态上继续向前走，每轮只追加新 token 的状态。

sampling 读 logits、temperature、top-p / top-k 和 stop 条件，选出一个 token。

看点：模型算出 logits 以后，还要经过选择，才会进入下一轮 decode。

stop 读停止符、长度上限和业务规则，释放 batch 占位和 KV cache。

看点：结束条件没判好，会把已经结束的请求当成还在运行。

步骤 状态从哪里来 状态写到哪里 如果这步放错了，会误判什么
prefill 完整 prompt 和已分配的 batch prompt 对应的 KV cache 会误以为首 token 慢只是算力问题，忘了它先在写状态
decode 历史 KV cache 和上一轮 token 新增 token 的 KV cache 会误以为 decode 只是读，不会继续吃显存
sampling 当前 logits 和约束条件 下一轮 token 会把“模型已经算完”误当成“token 已经确定”
stop 停止符、长度上限、业务规则 释放资源、结束流式输出 会把已完成请求继续算进并发和显存预算
最小生命周期伪代码
def infer(request):
    tokens = tokenize(request.text)
    kv = prefill(tokens)
    outputs = []
    while not stop_condition(outputs, request):
        logits = decode_step(kv, outputs[-1] if outputs else None)
        next_token = sample(logits, request.temperature, request.top_p)
        outputs.append(next_token)
        kv = append_kv(kv, next_token)
    release(kv)
    return detokenize(outputs)

这段伪代码最重要的地方不是函数名，而是顺序：先 prefill，再 decode，再 sample，再 append KV，最后释放。如果把 sample 放到 decode 前面，或者把 release 提前，状态就会错位。

指标读法

metrics

同一个请求可以“感觉很快”，但具体慢在哪一步要拆开看。TTFT、ITL 和 throughput 关注的对象不同，混在一起看会把排查方向带偏。

$$\text{TTFT} = \text{queue} + \text{prefill} + \text{first sample}$$ $$\text{ITL} \approx \text{decode loop} + \text{queue interference}$$ $$\text{throughput} = \frac{\text{generated tokens}}{\text{wall time}}$$

这里的 queue 指排队时间，prefill 指完整 prompt 的处理，first sample 指首 token 采样，decode loop 指后续每轮生成。式子只帮助你把指标和阶段对齐，真正的 profiling 还是要看日志和曲线。

指标 它在看什么 最常见的高值原因 先查什么
TTFT 从请求进入到首 token 出来 排队长、prompt 长、prefill 被切太碎 看 queue、prompt token 数、chunked prefill 和 token budget
ITL 相邻 token 之间的时间 decode 被挤压、batch 抖动、KV 读取压力大 看 decode 节奏、队列干扰和流式输出是否断续
throughput 单位时间产出的 token 总数 batch 太小、资源空转、抢占太多 看 batch 利用率和重复 preemption
一条实用判断： TTFT 高，不一定是模型慢；它可能是 queue 或 prefill。ITL 高，也不一定是采样慢；它可能是 decode 被别的请求打断了。

三个读法

cases

同一条请求慢，不代表慢的地方相同。下面三个场景把最容易混掉的判断拆开：首 token 慢、输出中间慢、总吞吐低。每个场景都只盯一个问题，先别急着把所有参数都拉进来。

场景一：TTFT 高，ITL 正常

更像是 queue 太长、prompt 太长，或者 prefill 被切得太细。你先看首 token 前的等待，不要先盯 decode。

先查什么为什么
waiting queue请求可能还没真正进 running。
prompt token 数prefill 要先把完整上下文写一遍。
chunked prefill切块过细会拉长首 token 前的等待。

场景二：首 token 快，后面断续

更像是 decode 被别的请求打断，或者 KV cache 读取压力上来了。这里要看持续输出阶段。

先查什么为什么
decode 节奏一轮一 token 的循环被打断，会让 ITL 抖。
队列干扰别的长请求可能在抢预算。
KV cache 压力历史状态越长，读取越重。

场景三：throughput 低

更像是 batch 太小、资源空转，或者抢占太多。这里看整台服务的总产出。

先查什么为什么
batch 利用率资源没被吃满时，吞吐会掉。
重复 preemption反复抢占会把工作打碎。
长 prompt 占比长 prefill 过多会压住 decode。
这三个读法的共同点： 先从症状回到阶段，再回到状态账本。这样后面的参数选择才不会乱。

请求推演

planning

把生命周期放到实际请求里看，最容易看清 queue、prefill 和 decode 是怎么相互挤压的。下面把一个长 prompt 和一个短 prompt 放到同一个调度窗口里，看看 TTFT、ITL 和窗口余量会怎么变。

轮次 长请求状态 短请求状态 窗口变化 会影响什么
t0 长 prompt 进入 waiting，等待 prefill 预算 短请求还没到 token budget 先被预留给 decode decode 节奏被保住，TTFT 还没开始计时
t1 长 prompt 进入 prefill，但只走一个 chunk 短请求到达 waiting 剩余预算被切出来给短请求的首 token 短请求的 TTFT 可能比长请求更好
t2 长请求继续分段 prefill 短请求已经进入 decode running 序列变多，KV headroom 变紧 ITL 主要看 decode 是否被挤压
t3 长请求如果继续占位，就可能触发 preemption 或回队 短请求继续流式输出 窗口开始偏向短请求的持续输出 吞吐是否稳定，取决于抢占和回队的频率

长请求带来的直接后果

长 prompt 会先把 prefill 压长，再把 KV cache 拉大。如果窗口不留足余量，它会一边拖 TTFT，一边把后面的请求挤出 batch。

短请求为什么有时更快

短请求需要的 prefill 预算更少，首 token 更容易抢到位置。但如果前面堆了太多长 partial prefills，短请求也会被卡在 waiting 里。

把一轮预算拆给长短请求
def split_budget(token_budget, running_decode, reserve, long_prefill_remaining, short_request_tokens):
    left = max(0, token_budget - running_decode - reserve)
    decode_share = min(running_decode, token_budget)
    long_share = min(long_prefill_remaining, max(0, left - short_request_tokens))
    short_share = min(short_request_tokens, max(0, left - long_share))
    return {
        "decode_share": decode_share,
        "long_prefill_share": long_share,
        "short_request_share": short_share,
    }
这段代码想让你看见什么： 一轮预算按阶段和优先级分配，先保 decode，再把余下部分切给 prefill。长请求和短请求的差别在于它们吃预算的方式不同。

参数取舍

tuning

参数的方向取决于你要保护哪一种等待。`max_num_batched_tokens` 决定这一轮能塞多少 token，`max_num_seqs` 决定这一轮能挂住多少条序列，`scheduler_reserve_full_isl` 决定你会不会先把完整输入长度算进去。

参数 它在控制什么 调大时常见结果 调小时常见结果
`max_num_batched_tokens` 单轮可用的 token 预算 吞吐可能更高，但长 prefill 更容易挤压 decode ITL 往往更稳，但 TTFT 可能更长
`max_num_seqs` 单轮可处理的序列数量 并发更高，但 KV headroom 更紧 队列更容易增长，但单轮压力更小
`scheduler_reserve_full_isl` 是否先按完整输入长度做准入 更保守，过度准入更少 更激进，但后面更容易出现抢占或重算

常见误读

很多人把 `max_num_seqs` 当成 token budget，或者把 `max_num_batched_tokens` 当成并发数。前者会把容量看偏，后者会把调参方向看错。

和 TTFT、ITL 的关系

如果你想稳 ITL，就不要让长 prefill 一口气占满预算；如果你想压 TTFT，就要给首 token 留出足够的可见窗口。两者很少能同时拉满。

实操顺序： 先看症状落在 TTFT 还是 ITL，再回到参数。顺序反过来，通常只会把问题调成另一个样子。

举个具体例子：如果一条请求有 3000 个 prompt token、400 个输出 token，`max_num_batched_tokens` 设得太小，TTFT 往往会更稳，但长请求要跑更多轮；设得太大，长 prefill 会更容易压住 decode，短请求看起来也会一起变慢。练习时就拿这组数字回看上面的状态账本和终端日志。下面这些公开文档按 2026-06-21 可见内容整理，后续版本如果改名或迁移，先回链接核对参数语义。
