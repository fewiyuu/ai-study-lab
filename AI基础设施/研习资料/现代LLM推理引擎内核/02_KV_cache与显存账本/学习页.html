谁在吃显存

overview

权重是固定账本，KV cache 是动态账本，临时 workspace 是按 kernel 走的短账。推理服务是否稳，取决于三者一起算，而不是只看模型参数量。

对象 会不会随 token 增长 谁写 如果判断错了，常见症状是什么
模型权重 不会。它基本是一次加载、长期驻留。 加载流程或启动脚本。 启动时就 OOM，或者模型根本装不进卡里。
KV cache 会。prompt 越长、输出越长，账就越大。 prefill 先写一段，decode 每步再追加。 单条请求能跑，但一上并发就爆显存、掉吞吐、TTFT 变长。
临时 workspace 不按 token 线性增长，但会随 kernel 和 batch 波动。 attention、归一化、采样和其他运行时算子。 看起来“还差一点点”，却在某些 batch 或 kernel 组合下突然 OOM。
prefix cache / 元数据 间接增长，主要看活跃前缀和块管理方式。 缓存层、路由层和块管理逻辑。 命中率低、碎片高、同样的前缀被重复算。

你现在应该能做什么：看到一个推理服务现象时，先问它是在写权重、写 KV，还是只是在临时占用 workspace。

prefill / decode / prefix cache 的读写顺序

trace

这张图只做一件事：把“谁在写、谁在读、谁可以复用”分开。你如果把 decode 当成重新写整段 prompt，就会把容量算大；你如果把 prefix cache 当成语义缓存，就会把命中率想错。

prefill 读模型权重和整段 prompt，写入 prompt 长度对应的 K/V。

症状：prompt 越长，首 token 越慢，KV 占用也越高。

decode 读历史 K/V 和新 token，给当前 token 追加新的 K/V。

症状：显存不是静止的，输出越长，KV 账本越厚。

prefix hit 读缓存里已经存在的前缀 K/V，跳过重复 prefill。

症状：TTFT 下降，重复模板越稳定，收益越明显。

prefix miss 前缀 token 不完全一致，只能重新写一遍 KV。

症状：看起来“内容差不多”，但 token 不同就不会复用。

步骤 主要读什么 主要写什么 如果把这一步看错了，会怎么误判容量
prefill 整段 prompt、模型权重 prompt 对应的整段 K/V 会以为首 token 慢只是算力问题，忘了它先把 KV 一口气写满。
decode 历史 K/V、当前 token 当前 token 新增的 K/V 会以为 decode 只是读历史，不会继续吃显存。
prefix hit 缓存中的前缀 K/V 几乎不写重复前缀 会把命中的前缀算成重复计算，错过复用带来的 TTFT 收益。
prefix miss 当前请求的真实 token 序列 新的前缀 KV 会以为语义相近就能复用，结果模板一抖动命中率就掉。

你现在应该能做什么：看到“慢”或“涨”的现象时，先分清是 prefill 写太多、decode 追加太久，还是前缀根本没命中。

把账换成数字

formula
$$\text{bytes/token} = 2 \times L \times H_{kv} \times D \times b$$ $$\text{bytes/request} = \lceil T / B \rceil \times B \times \text{bytes/token}$$

这里，\(L\) 是层数，\(H_{kv}\) 是 KV heads，\(D\) 是 head dim，\(b\) 是每个元素的字节数，\(T\) 是请求 token 数，\(B\) 是块分配单位。第一个公式算理论账，第二个公式把块分配的尾巴算进去。

先手算，再看服务配置
def estimate_kv_bytes(layers, kv_heads, head_dim, bytes_per_elem, tokens, block=16):
    bytes_per_token = 2 * layers * kv_heads * head_dim * bytes_per_elem
    allocated_tokens = ((tokens + block - 1) // block) * block
    bytes_per_request = bytes_per_token * allocated_tokens
    return bytes_per_token, allocated_tokens, bytes_per_request
先看哪一个量 layers、kv_heads、head_dim 和 dtype 直接决定单 token 账本；tokens 和 block 决定这一笔账会被放大多少。你如果先盯模型参数量，通常会算错。

为什么 KV heads 不能写错

config

很多人第一次算 KV cache，会把 `num_attention_heads` 直接代进公式。这个写法只有在普通多头注意力里才成立；一旦模型用了 GQA 或 MQA，真正决定 KV 账本大小的是 `num_key_value_heads`。这两个字段看起来只差一个名字，实际会把预算差出几倍。

最稳妥的做法，是先读模型配置里有没有 `num_key_value_heads`。有，就用它；没有，才退回到 `num_attention_heads`。这个顺序不是偏好，而是防止你把共享的 KV 头当成每个 query 头都要复制一份。

配置字段 应该怎么用 如果写错 会把什么判断带偏
`num_attention_heads = 32` 只描述 query 侧有多少头，不直接决定 KV 账本。 把它当成 KV heads，会把显存估大四倍。 会误以为 4K 请求一定放不下，或者误判需要更激进的压缩手段。
`num_key_value_heads = 8` 这才是 KV cache 公式里该放进去的数字。 把它漏掉，预算会明显偏大。 会把可服务并发算低，或者把峰值容量想得过于悲观。
`head_dim = 128` 决定每个头的向量长度。 把 head_dim 和 heads 混在一起，会让单位彻底乱掉。 会连“单 token 账”都算不清，更别说后面的并发账。
`dtype = BF16 / FP16 / FP8` 决定每个元素占几个字节。 把 dtype 当成模型名字的一部分，而不是容量变量。 会漏掉最直接的字节压缩来源。
从配置里挑出真正的 KV 头数
def kv_heads_from_config(cfg):
    if "num_key_value_heads" in cfg and cfg["num_key_value_heads"]:
        return cfg["num_key_value_heads"]
    return cfg["num_attention_heads"]

这段逻辑的重点不在 Python，而在判断顺序。对 GQA 模型来说，共享的是 KV，没共享的是 query；如果你反过来取字段，就会把复用关系看错，后面的预算、并发和选型都会跟着错。最常见的后果不是“算不出来”，而是“算得太保守”：你会误以为这张卡只能接很少请求，于是提前把优化方向带偏，甚至把本来够用的配置误判成不适合上线。

这和上面的算例怎么接起来 32 attention heads、8 KV heads、128 head dim、BF16、4096 tokens 这组数字之所以会从 2GiB 变成 512MiB，不是因为模型“变小”了，而是因为每个 query 头不再各自背一份独立 KV。这个差异如果没看见，容量判断会整整偏一档。

你现在应该能做什么：读到一个模型 config 时，先找 `num_key_value_heads`，再去算 KV cache；如果这个字段不存在，再退回到 `num_attention_heads`，不要反过来猜。

两个算例，先看量级

example

先记住这个直觉：模型能装进 24GB 显存，不等于它能稳稳接住多个长上下文请求。并发一高，吃掉你预算的往往是 KV cache，而不是权重本身。

配置 单 token 账本 请求 token 单请求 KV 读法
32 层 / 32 KV heads / 128 / BF16 512 KiB 4096 2 GiB 一条 4K 请求本身就是 GiB 级显存账。
同模型，但 KV heads = 8 128 KiB 4096 512 MiB GQA 把单 token 账本压到原来的 1/4。
同上，但 token = 2048 128 KiB 2048 256 MiB 长度减半，单请求 KV 也跟着减半。
块分配的尾巴 4097 个 token 如果按 16-token 块来放，会被分到 4112 个槽位。理论 KV 账没变，但摆放效率变差了，这就是后面 PagedAttention 要解决的部分。

你现在应该能做什么：拿到一组模型配置时，先把单 token 账、单请求账和并发账分开算，再谈是否要换模型、降并发或缩短上下文。

把预算推回可服务请求数

capacity

常见的误判是：看到 24GB 显存就默认“还能塞很多请求”。真正要做的是把总预算拆成三段，再倒推还剩多少活跃请求能放得下。这个顺序很重要，因为只看总显存会把 KV 账、临时 workspace 和碎片都漏掉。

步骤 公式或动作 例子 如果跳过这一步，会漏掉什么
1. 先扣权重 总显存 - 权重显存 24GB - 14GB = 10GB 会把固定成本和动态成本混在一起。
2. 再留运行时余量 再减 workspace、元数据和安全边界 实际可用常常小于 10GB 会在某些 kernel 组合或 batch 下突然 OOM。
3. 把单请求 KV 乘上并发 bytes/request × active requests 2GB × 8 = 16GB，显然放不下 会误以为单请求能跑就代表整台服务能稳住。
4. 再问有没有浪费和复用 块尾浪费、prefix hit、GQA、量化 8 KV heads 可以把单请求压到 512MiB 会错过真正能救容量的优化方向。

这个顺序也解释了为什么“模型能加载”只是起点。即使权重放得下，KV cache 也会把可服务并发吃掉；即使理论账能放下，块尾浪费和运行时开销也可能把你推过线。容量规划最怕的是把平均值当作上限，把上限当作承诺。

为什么这里不直接谈 block table 因为块表只能优化摆放效率，不能凭空变出更多字节。先把总预算和单请求账算准，后面再看如何减少浪费，顺序不能反。

把一台 24GB 卡算到能不能接 8 个请求

case

容量规划最容易犯的错，是把“平均请求”当成“峰值请求”，再把“能加载模型”当成“能稳定服务”。真正要做的是把一条服务链上的每一笔开销都扣掉，再看剩下的预算够不够你的目标并发。这个例子用一台 24GB GPU、14GB 权重、1GB 运行时余量和 8 个活跃请求来走完整个判断。

规划步骤 要算什么 32 KV heads 情况 8 KV heads 情况 这一步会改变什么判断
1. 扣固定成本 24GB - 14GB 权重 - 1GB 余量 剩下约 9GB 给 KV cache 和少量波动。 先知道动态账本只有一小段可用预算。
2. 算单请求 KV 4K tokens 的每请求账本 约 2GiB / 请求 约 512MiB / 请求 这一步决定“一个会话”本身有多贵。
3. 乘上并发 单请求 KV × 8 约 16GiB 约 4GiB 这一步直接告诉你能不能接住 8 个活跃会话。
4. 再加块尾浪费 4K+1 token 时的块分配尾巴 每请求还会多出若干槽位 同样会多出若干槽位 这一步解释了为什么“理论刚好够”在真实服务里常常不够。
5. 看复用是否真的命中 prefix hit 能不能减掉重复 prefill 只在前缀完全一致时才有用 只在前缀完全一致时才有用 这一步决定你能不能把首 token 压短，但不改独立请求的峰值 KV。
把规划逻辑写成一个检查器
def plan_capacity(total_gib, weight_gib, runtime_gib, kv_per_request_gib, active_requests, safety_gib=1.0):
    usable_gib = total_gib - weight_gib - runtime_gib - safety_gib
    required_gib = kv_per_request_gib * active_requests
    return {
        "usable_gib": usable_gib,
        "required_gib": required_gib,
        "fits": required_gib <= usable_gib,
    }

这个检查器的价值不在于“算得绝对精确”，而在于逼你把预算拆开。你只要把 `kv_per_request_gib` 填错，后面的结论就会整段漂移；你只要把平均上下文长度当成峰值，8 个请求看起来会很轻松，实际一遇到 4K 长尾就可能直接顶穿预算。反过来，如果你把 P95 长度、并发峰值和块尾浪费一起算进去，很多“看上去应该够”的系统会被你提前排除掉。

这个例子也说明了为什么有些优化顺序不能颠倒。prefix cache 可以把重复前缀的 prefill 省掉，但它不改变独立请求的上限账本；PagedAttention 可以让 KV 摆得更紧，但它也不会把 2GiB 变成 1GiB；KV quantization 真的会把每 token 的字节数压下去，但它带来的是精度、实现和硬件支持边界。规划时先问“当前瓶颈是哪一层”，再决定要不要换模型、改精度、调块大小或做分层缓存，顺序比口号重要。

失败后果 如果你把平均请求当成容量上限，服务上线后常见的第一类失败就是：测试集里看起来没问题，真实流量一进来就出现 OOM、排队变长或者尾部请求被拖慢。这个问题不是靠“再买一张卡”自然消失的，先算错，后面所有优化都只能修补误判。

你现在应该能做什么：拿到一台机器和一个并发目标时，先估出能接住的活跃请求数，再决定是降并发、改 KV heads、上 quantization，还是先解决 prefix 稳定性。

为什么 4097 个 token 会比 4096 更贵

waste

块管理解决的是“怎么放”，不是“理论上要放多少”。只要你按固定块大小分配，就会有尾部浪费。这个浪费不在公式的主项里，但它会在真实服务里吃掉你以为剩下的那一点点预算。

请求长度 块大小 分配槽位 浪费槽位 会造成什么判断偏差
4096 16 4096 0 看起来“刚好对齐”，最容易让人误以为块管理没有代价。
4097 16 4112 15 只多 1 个 token，却多吃了一个完整块里剩下的 15 个槽位。
4097 32 4128 31 块越大，尾部浪费越明显；摆放更省心，不代表更省容量。
8193 64 8256 63 长请求和大块一起出现时，尾巴会变成明显的成本。
把尾部浪费也算进去
def block_waste(tokens, block_size):
    allocated = ((tokens + block_size - 1) // block_size) * block_size
    waste = allocated - tokens
    return allocated, waste

这个小函数看起来朴素，但它解决的是容量规划里最常被漏掉的那一项。假设单 token 账已经算好，4097 个 token 在 16-token 块下会从 4096 变成 4112 槽位，这 15 个槽位会被一起记进显存账里。只要并发一高，尾巴不是“可以忽略的小数”，而是会和其他请求叠在一起，最终把你推过可用预算。做容量评估时，块大小越大，尾部浪费就越可能把平均值和峰值拉开。

这也是为什么 PagedAttention 不能只当成“节省显存”的口号。它让物理布局更灵活，减少了为每个请求预留连续大块的压力；但如果块设置得太大，或者请求长度分布很碎，尾部浪费仍然会积累。真正稳的做法，是同时看块大小、请求长度分布和并发峰值，再决定要不要缩块、换量化、压低上下文，或者先把 prefix 稳定性和调度策略理顺。

失败后果 最常见的失败是：你按理论账估出来“刚好够”，上线后却发现某些长度分布下仍然 OOM。问题不在算术错，而在你漏算了尾部浪费和实际请求分布。这个链条一旦漏掉，后面的优化会被你用错方向。

你现在应该能做什么：拿到一个长度分布时，别只算平均 token 数；先看块大小会吞掉多少尾巴，再决定容量预算到底要留多宽。

四种优化各管一层

bridge

这四种优化经常被混着说，但它们解决的是四个不同层面的问题。先分清“少写、少算、少浪费、少字节”，再决定哪一个是你当前该动的杠杆。

方法 主要改哪一层 它真正省掉什么 它不替你解决什么
PagedAttention 摆放和碎片 让 KV 按块放进物理显存，减少预留和碎片。 不改变理论上的 bytes/token。
Prefix cache 重复 prefill 完全相同的前缀可以直接复用已有 KV。 不做语义近似，也不替代 prompt 稳定性。
KV quantization 每个元素的字节数 把每 token 的账本压小，尤其对长上下文很敏感。 不减少 token 数，也不自动提高 prefix 命中率。
HiCache 驻留层级 把 KV 放到 GPU 以外的更慢层，换更大的可用容量。 不保证低复用短请求变快，搬运成本还可能上来。
下一步看什么 下一单元会把 PagedAttention 的块表和物理布局拆开讲。这里先记住：它管的是摆放效率，不是理论账本大小。

你看到的症状 更像哪一类问题 先看什么
模型能加载，但一上并发就 OOM KV budget 不够 先算活跃请求数乘每请求 KV，再看权重和运行时开销还剩多少余量。
TTFT 变长，但显存还没有打满 重复 prefill 没少掉 检查 prompt 模板是否稳定，prefix hit 是否真的上来了。
用了 PagedAttention，容量还是不够 理论 bytes/token 没变 回去看 kv_heads、dtype、head_dim 和上下文长度。
某些请求特别慢，但别的都正常 块浪费或读带宽压力 看 block size、长 prompt 和 decode 时对历史 K/V 的读取压力。
你把总 attention heads 直接塞进公式 公式用错了变量 确认是不是 GQA/MQA，真正要填的是 KV heads。
最后一句 “能加载模型”只说明权重放得下；“能高并发服务”还要看 KV cache、碎片、前缀复用和运行时开销一起能不能兜住。

你现在应该能做什么：把一个 OOM 或慢请求先拆成“权重够不够、KV 够不够、复用有没有命中、摆放有没有浪费”四个问题，而不是直接换大显存。

练习

diagnostic

怎么答才算会

先把对象说清楚，再给数量级，最后说瓶颈归类。开放题里只写“结论”，通常说明你还没有把账算完。

检查重点

答题时至少要能说出：谁在写、谁在读、谁能复用、谁在浪费，以及这些动作会让显存、TTFT 或吞吐发生什么变化。

第一组：概念和账本

先确认你能把 KV cache 的来源、增长方式和关键参数说清楚。

KV cache 随 token 数线性增长；GQA 影响 KV heads；精度影响每格字节数。
choice公式

KV cache 每 token 字节数最接近哪一个公式？

注意 Key 和 Value 是两份张量，公式里要用 KV heads，不是总 attention heads。

true/false增长

decode 阶段只读 KV cache，不会继续增加 KV cache。

参考答案 / 评分要点

错。decode 每步都会为新 token 追加新的 K/V，所以 KV cache 会继续增长。答对要同时说出“读历史 K/V”和“写新 K/V”这两个动作。

fill手算

32 层、8 个 KV heads、head dim 128、BF16，2048 tokens 约占多少 MiB？

参考答案 / 评分要点

单 token 约 128 KiB，2048 tokens 约 256 MiB。至少要写出“KV heads = 8，所以单 token 比 32 heads 情况少四倍”这一层推理。

choice复用

下面哪句话最准确地描述了 prefix cache？

这里考的是 token 级别一致性，不是语义相似度。

第二组：工程诊断和迁移

把账本用于真实引擎现象：OOM、吞吐下降、prefix miss、块管理和缓存层级。

PagedAttention 管摆放，prefix cache 管复用，KV 量化管每格大小。
error diagnosis容量

一个 24GB GPU 能加载 14GB 权重，但 8 个 4K 请求并发时 OOM。请写出你的排查顺序。

参考答案 / 评分要点

要先估算每请求 KV 账，再乘并发数，看是否已经接近剩余显存；然后再检查上下文分布、输出长度和运行时开销。最后才考虑缩短上下文、减少并发、换 GQA、压低精度或启用 prefix cache。只说“换大显存”不算排查。

choice选型

哪一种方法主要解决“摆放和碎片”问题？

先想它是在改“摆放方式”、还是在改“复用”、还是在改“每格大小”。

scenario transfer工作负载

什么工作负载适合层次化 KV cache？什么工作负载可能不适合？

参考答案 / 评分要点

适合的是长上下文、多轮会话、共享前缀高、KV 可复用的场景；不适合的是低复用、短请求、搬运成本高于收益的场景。答题时要出现“复用率”和“搬运成本”两个判断维度。

synthesis面试

用不超过 180 字，向后端工程师解释为什么“能加载模型”不等于“能高并发服务”。

参考答案 / 评分要点

要点是：权重只是固定成本，高并发主要被动态 KV cache 和运行时开销限制；上下文越长、并发越高，KV 账越大；prefix cache、GQA、PagedAttention 只能分别解决复用、每 token 大小和摆放效率的一部分问题。说清楚这三层关系就够了。

true/false范围

PagedAttention 会直接把理论上的 bytes/token 变小。

参考答案 / 评分要点

错。PagedAttention 改的是摆放和碎片，不改变理论 bytes/token；要真正把每 token 账本压小，得看 KV quantization 或改变 KV heads、head dim、dtype 这类变量。

true/false适用性

HiCache 一定更适合低复用、短请求的工作负载。

参考答案 / 评分要点

错。层次化缓存更适合能产生复用、值得搬运的长上下文或多轮场景；低复用短请求往往会被数据搬运和管理开销拖慢。
