为什么会有 PagedAttention

overview

如果每个请求都预留一整段连续 KV cache，服务很快会被两件事拖住：请求长度变化太大，很多请求又只用到预留区的一小部分。PagedAttention 关注的是 KV cache 的物理摆放方式；attention 的数学公式本身在这里不变。

逻辑层

模型看到的是一串连续 token。前缀、续写和分支都还是“序列”的概念。

物理层

显存里放的是固定大小的 block。一个请求可以占很多块，块之间不必挨在一起。

管理层

块表把逻辑位置映射到物理 block；prefix cache 和 copy-on-write 在这张表上做复用和分叉。

读这页时的直觉： 先把“序列长度”翻成“块数”，再看“块之间怎么共享”。先盯 kernel，容量问题很容易被看成纯算力问题。

块表在读写什么

state ledger

把 KV cache 看成块表之后，最重要的就不是“有没有这段数据”，而是“这段数据现在处于什么状态”。下面这张表把读写、共享和故障症状摆在一起。

对象 进入前是什么 进入后是什么 读什么 写什么 错了会怎样
逻辑序列 一串连续 token 被切成多个块的逻辑段 当前 token 位置、前缀长度 生成新的 token 位置 会把一个连续前缀误算成多个不相关片段
物理 block 空闲或被别的序列占用 被当前序列引用 引用计数、块大小、块哈希 写入新的 K/V 数据 会出现覆盖、重复分配，或尾块浪费被漏算

prefill 读 prompt token，写 4 个逻辑块中的前 3 个完整块和 1 个尾块。

两个请求共享前缀时会发生什么

example

设请求 A 和请求 B 的前 32 个 token 完全一致，block size 是 16。A 先进入 prefill，B 后来跟进。这里真正重要的不是“字符串看起来像不像”，而是 token 序列是否完全一致。

A 先进入

A 的前 32 个 token 会占 2 个完整 block。A 继续生成自己的后续 token 时，会在自己的尾块里追加。

A0 -> 0..15
A1 -> 16..31
A2 -> 32..47
A3 -> 48..63

B 复用前缀

B 可以直接复用 A0 和 A1，两块的内容不需要重算；如果 B 在共享块上继续写自己的分支，就要先复制出私有块。

B0 -> 0..15 (shared)
B1 -> 16..31 (shared)
B2 -> 32..47 (private after branch)
这例子告诉你的事： prefix cache 省的是重复 prefill，不会自动帮你缩短生成长度；PagedAttention 省的是摆放和碎片，不会把理论 bytes/token 变小。

前缀缓存和块管理是什么关系

prefix cache

前缀缓存做什么

它找的是“完全相同的前缀”，命中后直接复用已有 KV block。它不是语义相近就行，也不是看长度一样就行。

块管理做什么

它解决的是 block 怎么放、怎么指、怎么共享、怎么在写时复制成私有块。没有块管理，前缀缓存就没有安全的承载方式。

两者合起来

prefix cache 负责减少重复 prefill，PagedAttention 负责把复用的 KV 放进块表里，让它能被多个请求共享。

机制 主要解决什么 不解决什么 你在这页里先记住的判断
PagedAttention 块式摆放、碎片、块表映射 不会改变理论 bytes/token 它是物理布局层的优化
Prefix cache 重复前缀复用 不会做语义近似，也不会缩短输出长度 它省的是重复 prefill
HiCache 把 KV 放到 GPU 之外的更慢层 不会让低复用短请求天然更快 它是驻留层级的扩展，不是这节课的主角

最容易错的 4 个判断

mistakes

把“看起来一样”当成“可以复用”

prefix cache 不是语义缓存。token 序列不完全一样，就不能默认复用。

把 block size 当成越大越好

块越大，尾块浪费越容易堆起来。你会减少管理开销，但也会放大空洞。

把 PagedAttention 当成算子近似

它不是改变 attention 数学的算法，而是改 KV cache 的布局、引用和访问方式。

把共享前缀当成写共享块

共享只对读成立；一旦写，必须 copy-on-write，不然两个请求会互相污染。

block size 为什么不是越大越好

capacity

Block size 越大，块表里的指针数会变少，管理会更轻；但尾块浪费也会更大，尤其是请求长度分布很碎的时候。PagedAttention 解决的是“摆得更灵活”，不是“把每个 token 都变成更少字节”。

所以这里要分开看三件事：第一，块数决定要记多少条映射；第二，尾块浪费决定物理显存里会空出多少槽位；第三，block size 决定你是在减少碎片，还是在放大空洞。单看平均长度，很容易把这个取舍看反。

$$\text{blocks}(T, B) = \left\lceil \frac{T}{B} \right\rceil$$ $$\text{tail waste}(T, B) = (B - T \bmod B) \bmod B$$

这里 \(T\) 是 token 数，\(B\) 是 block size。第一个式子告诉你要分配多少块，第二个式子告诉你最后一块会空出多少槽位。两者合起来，才是这页真正关心的占用账。

block size 49 token 要几块 尾块浪费 块表与管理开销 更适合什么请求分布
8 7 块 7 槽位 块表更长，管理更细 短请求多、长度变化碎、想压小尾块
16 4 块 15 槽位 折中 常见的混合负载，既想要复用又不想把表做太长
32 2 块 15 槽位 表短，但空洞更大 长度更整齐、长请求占比高、管理开销更敏感
这张表真正想让你记住的事： 你不是在找“最省”的单个数，而是在找“尾块浪费、块表长度和复用稳定性”之间的平衡点。块越小，不一定整体更好；块越大，也不一定更稳定。

把这笔账写成一段小代码

code

这段代码的作用不是“跑分”，而是把公式写成可检查的步骤。你在读块管理时，最容易出错的地方就是把“理论 token 数”和“实际分配槽位”混成一个数；代码会把这两个量拆开。

下面的函数分别做两件事：`plan_blocks` 负责算块数、尾块和利用率；`shared_prefix_blocks` 负责判断前缀能安全共享到哪一个完整 block。后面做 copy-on-write 或 prefix cache 排障时，先跑这两个量，通常就知道问题落在哪一层。

块数、尾块和共享前缀检查
def plan_blocks(tokens, block_size=16):
    blocks = (tokens + block_size - 1) // block_size
    allocated = blocks * block_size
    waste = allocated - tokens
    utilization = tokens / allocated if allocated else 0
    return {
        "tokens": tokens,
        "block_size": block_size,
        "blocks": blocks,
        "allocated_slots": allocated,
        "tail_waste": waste,
        "utilization": round(utilization, 4),
    }


def shared_prefix_blocks(prefix_tokens, block_size=16):
    shared_blocks = prefix_tokens // block_size
    shared_tokens = shared_blocks * block_size
    return {
        "prefix_tokens": prefix_tokens,
        "block_size": block_size,
        "safe_shared_blocks": shared_blocks,
        "safe_shared_tokens": shared_tokens,
    }
prefix_tokens safe_shared_blocks safe_shared_tokens 读这行代码时要盯住什么
31 1 16 只够共享一个完整块；尾部 15 个 token 还不能算安全共享
32 2 32 刚好卡在完整块边界，这是最干净的复用边界
33 2 32 多出来的 1 个 token 不能让前两块之外的部分自动共享
如果这段代码算错了，通常说明什么： 不是公式写错，而是你把“完整块边界”看漏了。对于 prefix cache 来说，安全共享只发生在整块上，不发生在半块上。

三种常见症状在终端里长什么样

failure lab

很多块管理问题不是“算术不会”，而是你看到了症状，却不知道它到底是在说复用失败、尾块浪费，还是共享块被错误写入。这个小终端实验把三种常见场景拆开，让你直接对着输出判断问题类别。

prefix hit:
  shared blocks reused: 2
  recompute prefill: no
  tail waste unchanged: yes

看到 prefix hit 时

先确认是不是完整前缀命中，再看复用的是几块。这里的目标不是更快地“猜中”，而是知道重复 prefill 已经被省掉了。

看到 prefix miss 时

优先检查 tokenizer、模板、特殊 token 和前导空白。只要前缀 token 序列变了，块表里的缓存就不会自动接上。

看到 copy-on-write 时

这说明共享块已经被保护起来了。后续写入必须落到新的私有块上，不能继续往共享块里写。

这节课的排障顺序

先看前缀是否完全一致，再看块边界是否对齐，最后才看 block size 是否需要调小。把顺序反过来，往往会把问题定位错层。

把 32、33、48、49 token 放在一起看

boundary cases

这页最容易读歪的地方，不在 49 token，而在“刚好卡边界”和“只差 1 个 token”这两类情况。它们会让块数、共享边界和尾块浪费同时改变，所以你要把它们放在同一张表里看。

token 数 block size = 16 时的块数 是否刚好卡在完整块边界 尾块浪费 对 prefix cache 的意义
32 2 是 0 这是最干净的共享边界，前两块可以整块复用
33 3 否 15 只比 32 多 1 个 token，却已经逼出一个新块和 15 个空槽位
48 3 是 0 如果前缀稳定，这里还能做整块共享；一旦分支，就从第 4 块开始分开
49 4 否 15 只多 1 个 token，就把整条请求的物理占用推到下一档

这也是为什么这节课要把“完整块边界”讲得这么细。你如果把 33 当成 32 的“差不多版本”，就会误以为还能整块复用；你如果把 49 当成 48 的“差不多版本”，就会漏掉一整块的记账。真正会让线上容量拐弯的，往往就是这一个 token 的差异。

失败后果： 一旦把 33 当成 32、把 49 当成 48，排障时你会先去怀疑 kernel 或采样，而不是先看块边界和前缀稳定性。这个顺序一错，后面的优化就会落到错层。

把块边界和线上症状接起来

handoff

现在把问题换成一个更像线上排障的场景：团队说“同样的 prompt 第二次没有复用，TTFT 也没降”。如果你只看延迟，很容易先怀疑算子；但在这页的视角里，先问的是前缀是否完全一致、块表是否还指向同一批 block，以及最后一块是不是在每次请求里都被重新占掉了。

先做什么

先把输入拆成 token 序列，再检查块边界。只要 tokenizer、模板、特殊 token 或前导空白变了，prefix cache 就可能直接失效。

再看什么

再算 block 数和尾块浪费。一个本来卡在 32 的前缀，只要变成 33，就会多出一整块和 15 个空槽位，这就是容量线上最常见的“只差一点点”问题。

线上症状 先看哪一层 要检查的对象 如果这里没问题，下一步再看什么
TTFT 没降 prefix cache token 序列是否完全一致 再看块表是否真的复用了前缀 block
显存突然紧了 块数和尾块 block size 与请求长度分布 再看是不是该缩小 block size，或换更稳定的 prompt 模板
共享后结果互相污染 copy-on-write 共享块是否在写前复制成私有块 再看引用计数和块表更新是否同步

这就是这页真正想训练你的地方：不是记住“PagedAttention 很重要”，而是拿到一个看起来像性能问题的症状时，先把它拆成块边界、共享边界和写时隔离三个检查点。这样做的好处，是你能很快分清问题到底在布局层、复用层，还是在后面的调度层。

练习 hook： 去练习区先做 q7、q10 和 q15。它们分别对应共享边界、前缀失效和排障起手式，最适合拿来检验你是不是已经把这一页的链条串起来了。
