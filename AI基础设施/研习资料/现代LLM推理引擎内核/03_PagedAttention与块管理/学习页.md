# 为什么会有 PagedAttention {#overview}
<!-- tag: overview -->

如果每个请求都预留一整段连续 KV cache，服务很快会被两件事拖住：请求长度变化太大，很多请求又只用到预留区的一小部分。PagedAttention 关注的是 KV cache 的物理摆放方式；attention 的数学公式本身在这里不变。

逻辑层
模型看到的是一串连续 token。前缀、续写和分支都还是“序列”的概念。


物理层
显存里放的是固定大小的 block。一个请求可以占很多块，块之间不必挨在一起。


管理层
块表把逻辑位置映射到物理 block；prefix cache 和 copy-on-write 在这张表上做复用和分叉。

> [!NOTE] 读这页时的直觉：
> 先把“序列长度”翻成“块数”，再看“块之间怎么共享”。先盯 kernel，容量问题很容易被看成纯算力问题。

# 两个请求共享前缀时会发生什么 {#example}
<!-- tag: example -->

设请求 A 和请求 B 的前 32 个 token 完全一致，block size 是 16。A 先进入 prefill，B 后来跟进。这里真正重要的token 序列是否完全一致。

## box
### A 先进入
A 的前 32 个 token 会占 2 个完整 block。A 继续生成自己的后续 token 时，会在自己的尾块里追加。
```text
A0 -> 0..15
A1 -> 16..31
A2 -> 32..47
A3 -> 48..63
```


## box
### B 复用前缀
B 可以直接复用 A0 和 A1，两块的内容不需要重算；如果 B 在共享块上继续写自己的分支，就要先复制出私有块。
```text
B0 -> 0..15 (shared)
B1 -> 16..31 (shared)
B2 -> 32..47 (private after branch)
```


> [!NOTE] 这例子告诉你的事：
> prefix cache 省的是重复 prefill，不会自动帮你缩短生成长度；PagedAttention 省的是摆放和碎片，不会把理论 bytes/token 变小。

# 前缀缓存和块管理是什么关系 {#prefix}
<!-- tag: prefix cache -->

前缀缓存做什么
它找的是“完全相同的前缀”，命中后直接复用已有 KV block。它不是语义相近就行，也不是看长度一样就行。


块管理做什么
它解决的是 block 怎么放、怎么指、怎么共享、怎么在写时复制成私有块。没有块管理，前缀缓存就没有安全的承载方式。


两者合起来
prefix cache 负责减少重复 prefill，PagedAttention 负责把复用的 KV 放进块表里，让它能被多个请求共享。

机制
主要解决什么
不解决什么
你在这页里先记住的判断


PagedAttention
块式摆放、碎片、块表映射
不会改变理论 bytes/token
它是物理布局层的优化


Prefix cache
重复前缀复用
不会做语义近似，也不会缩短输出长度
它省的是重复 prefill


HiCache
把 KV 放到 GPU 之外的更慢层
不会让低复用短请求天然更快
它是驻留层级的扩展，不是这节课的主角

# 最容易错的 4 个判断 {#mistakes}
<!-- tag: mistakes -->
<!-- layout: grid2 -->

## box
### 把“看起来一样”当成“可以复用”
prefix cache 不是语义缓存。token 序列不完全一样，就不能默认复用。

## box
### 把 block size 当成越大越好
块越大，尾块浪费越容易堆起来。你会减少管理开销，但也会放大空洞。

## box
### 把 PagedAttention 当成算子近似
它改 KV cache 的布局、引用和访问方式。

## box
### 把共享前缀当成写共享块
共享只对读成立；一旦写，必须 copy-on-write，不然两个请求会互相污染。

# block size 为什么不是越大越好 {#capacity}
<!-- tag: capacity -->

Block size 越大，块表里的指针数会变少，管理会更轻；但尾块浪费也会更大，尤其是请求长度分布很碎的时候。PagedAttention 解决的是“摆得更灵活”，不是“把每个 token 都变成更少字节”。

所以这里要分开看三件事：第一，块数决定要记多少条映射；第二，尾块浪费决定物理显存里会空出多少槽位；第三，block size 决定你是在减少碎片，还是在放大空洞。单看平均长度，很容易把这个取舍看反。

$$\text{blocks}(T, B) = \left\lceil \frac{T}{B} \right\rceil$$
 $$\text{tail waste}(T, B) = (B - T \bmod B) \bmod B$$
 这里 \(T\) 是 token 数，\(B\) 是 block size。第一个式子告诉你要分配多少块，第二个式子告诉你最后一块会空出多少槽位。两者合起来，才是这页真正关心的占用账。

# 把这笔账写成一段小代码 {#code}
<!-- tag: code -->

这段代码的作用把公式写成可检查的步骤。你在读块管理时，最容易出错的地方就是把“理论 token 数”和“实际分配槽位”混成一个数；代码会把这两个量拆开。

下面的函数分别做两件事：`plan_blocks` 负责算块数、尾块和利用率；`shared_prefix_blocks` 负责判断前缀能安全共享到哪一个完整 block。后面做 copy-on-write 或 prefix cache 排障时，先跑这两个量，通常就知道问题落在哪一层。

块数、尾块和共享前缀检查复制

```python
def plan_blocks(tokens, block_size=16):
 blocks = (tokens + block_size - 1) // block_size
 allocated = blocks * block_size
 waste = allocated - tokens
 utilization = tokens / allocated if allocated else 0
 return {
 "tokens": tokens,
 "block_size": block_size,
 "blocks": blocks,
 "allocated_slots": allocated,
 "tail_waste": waste,
 "utilization": round(utilization, 4),
 }


def shared_prefix_blocks(prefix_tokens, block_size=16):
 shared_blocks = prefix_tokens // block_size
 shared_tokens = shared_blocks * block_size
 return {
 "prefix_tokens": prefix_tokens,
 "block_size": block_size,
 "safe_shared_blocks": shared_blocks,
 "safe_shared_tokens": shared_tokens,
 }
```

**Q:** 下面哪句话最准确地描述了 prefix cache？

- [ ] . 语义相近就能复用同一份 KV
- [ ] . 长度一样就能复用同一份 KV
- [ ] . 只有 token 前缀完全一致时才有机会复用
- [ ] . 同一个 batch 里的请求都能自动复用

Answer: . 只有 token 前缀完全一致时才有机会复用

### 6 debug-checklist

**Q:** 共享前缀后，为什么写入时还要复制一份私有块？

- [ ] 避免两个请求写坏同一个物理块
- [ ] 让共享块继续只读
- [ ] 把理论 bytes/token 变小

Explain: 前两项是对的，第三项不是。copy-on-write 解决的是写时隔离，不会改变理论 bytes/token。你如果把它当成压缩手段，就会误判容量。

### 7 synthesis

**Q:** A 和 B 的前 32 个 token 完全一致，block size = 16。B 复用 A 的前缀时，最多能安全共享到哪一层？

Explain: B 最多共享 A 的前两个完整 block，也就是 0..15 和 16..31 两块。因为共享只对读成立，一旦 B 要在共享块上继续写，就必须先复制成私有块。答题时要同时写出“完整块边界”和“写前复制”两个点。

### 8 choice

- [ ] . block size 从 16 变成 8
- [ ] . 请求里多了一个无关的说明文本，但 token 序列不变

Answer: . tokenizer 或模板改了，前缀 token 序列不再完全一致

### 9 scenario-transfer

**Q:** 如果 block size 从 16 改成 8，最直接的收益和代价分别是什么？

Explain: 收益是尾块浪费通常会变小，短请求更容易贴近真实占用；代价是块表可能变长、管理开销和元数据压力会上来。答题时要把“收益”和“代价”分开写，不要只写“更省显存”。

### 14 code-reading

**Q:** 看代码：`shared_prefix_blocks(31, 16)`、`shared_prefix_blocks(32, 16)`、`shared_prefix_blocks(33, 16)` 的区别是什么？

Explain: 31 只能安全共享 1 个完整块，也就是 16 个 token；32 刚好能共享 2 个完整块；33 仍然只能共享到前 32 个 token，因为多出来的 1 个 token 还没有让下一个块变成完整块。答题时必须写出“完整块边界”以及“不能共享半块”。

### 16 synthesis

**Q:** 如果你只剩 60 秒，怎么向同事解释“PagedAttention 解决的到底是什么”？

Explain: 可参考的答法：PagedAttention 把 KV cache 切成固定大小的 block，用块表把逻辑序列映射到物理显存；这样可以复用完全一致的前缀，减少重复 prefill，同时把尾块浪费和碎片显式地管起来。答题时要把“布局”和“复用”说出来，不要只说“更省显存”。
