Continuous batching 与调度循环

这一页只看 continuous batching 与调度循环：请求怎样在 waiting、running、preempted 和 finished 之间迁移，token budget 怎样决定本轮能推进多少 prefill / decode，chunked prefill 为什么可以和 decode 同轮共存。它承接 03 单元的块管理，也会把你送到 05 单元的算子层；更下游的路由、网关、多机扩展留到后面的单元。

continuous batching 在改什么

overview

静态 batch 的思路是先凑齐，再一起跑完。continuous batching 的思路是每一轮迭代都重排一次队伍：先让正在 decode 的请求继续前进，再把剩余 token budget 分给 chunked prefill，必要时再把长请求切块、把低优先级请求抢占出去。这样做的目的，是让 decode 不被长 prefill 长时间挡住，同时让新请求更快进入服务。

这页的锚点例子：

一组请求 A、B、C、D 会在不同轮次里交替出现。你后面看到的表格和代码，都会围着这组请求转。

调度循环怎么跑

mechanism

这一段把调度器拆成五步：新请求进入、预算检查、组 batch、执行 forward、回写状态。读的时候不要只盯最后一次 forward，要看每一步到底读了什么、写了什么、丢了什么。

代码
while engine_running:
 retire_finished_requests()
 free_kv_for_finished_requests()

 decode_batch = pick_decode_requests(running_queue, token_budget)
 prefill_batch = pick_prefill_requests(
 waiting_queue,
 token_budget,
 chunked_prefill=True,
 )

 batch = decode_batch + prefill_batch
 outputs = model_forward(batch)
 update_request_states(outputs)

 if kv_pressure_high():
 preempt_low_priority_requests(mode=preemption_mode)

调度窗口怎么看

window

调度窗口可以理解为“这一轮允许被看见和被推进的请求集合”。它受三类限制共同影响：token 预算、序列上限和 KV cache headroom。窗口太窄，短请求也会排队；窗口太宽，长 prefill 和过多 running 请求会把 KV 压紧，后面只能靠抢占收场。

限制它在保护什么窗口变窄时窗口变宽时
token budget单轮 forward 的计算量和 prefill / decode 比例prefill chunk 更小，TTFT 可能上升吞吐可能上升，但 decode 更容易被长 prefill 挤压
max_num_seqs同一轮能挂住多少条序列waiting queue 更容易增长并发更高，KV cache 压力也更高
KV headroom后续 token 写入和请求恢复的空间准入更保守，短期吞吐受限更容易 over-admission，抢占和重算成本会上来
读日志时的抓手：

如果 waiting 增长，看窗口是否太窄；如果 repeated preemption 增长，看窗口是否放得太宽；如果 TTFT 和 ITL 互相拉扯，回到 token budget 和 chunked prefill 的比例。

别把瞬时峰值当成稳定瓶颈，至少跨几个连续调度轮次一起看。

example

下面这个例子只跟踪 A、B、C、D 四个请求。A 和 B 先在 decode 阶段继续生成；C 是一个长 prompt；D 是一个短 prompt，但到得更晚。这样安排是为了让你一次看到 decode 优先、chunked prefill、等待队列和抢占四件事。

这张表最重要的结论：

一轮里既可以有 decode，也可以有 chunked prefill，还可以有抢占。continuous batching 关注每轮怎么重新组批，让不同状态的请求按预算继续前进。

参数怎么把系统推向不同方向

tuning

调参时不要只问“这个值大不大”，要问它在减少哪一种等待。max_num_batched_tokens 更多是在调每轮 token 预算，max_num_seqs 更多是在调并发请求数，max_num_partial_prefills 和 max_long_partial_prefills 影响长 prompt 能同时被切几段，scheduler_reserve_full_isl 则决定你在准入时会不会把整段输入长度一起算进去。

$$ \text{chunk} = \min(P_{\text{remaining}}, \max(0, B - D - R)) $$

这里 \(B\) 是这一轮的总 token 预算，\(P_{\text{remaining}}\) 是还没走完的 prompt tokens，\(D\) 是这轮已经给 decode 留出的 token，\(R\) 是安全余量或保留头寸。这个式子用来帮助你判断为什么长 prefill 会和 decode 争预算。

调参顺序建议：

先看 token budget 和 max_num_seqs，再看长 prefill 的并行数，最后才看抢占模式。顺序反了，容易把问题从预算层拖到抢占层。

Q: 为什么把 max_num_batched_tokens 调小，常常会改善 ITL，却可能让 TTFT 变差？

参考答案 / 评分要点

。把 max_num_batched_tokens 调小，单轮里能塞进来的 prefill chunk 变少，decode 请求被长 prefill 打断的次数也会下降，所以 ITL 往往更稳；但同样的长 prompt 需要更多轮才能跑完，首个 token 或完整请求完成前要等更久，所以 TTFT 可能变差。答题时要把“少打断 decode”和“prefill 分轮更细”都说出来。

Q: 如果开启 chunked prefill 之后 TTFT 变差，先检查哪三件事？

参考答案 / 评分要点

。可以先查：1）max_num_batched_tokens 是否过小，导致 prefill chunk 被切得太细；2）max_num_partial_prefills 和 max_num_long_partial_prefills 是否让长请求并行过多，挤压了短请求；3）scheduler_reserve_full_isl 是否没打开，或者请求长度分布变化后仍按老预算准入。只要能把“预算、长请求并行、准入检查”三件事说清楚，就算合格。
