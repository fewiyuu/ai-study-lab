第 04 单元

Continuous batching 与调度循环

这一页只看 continuous batching 与调度循环：请求怎样在 waiting、running、preempted 和 finished 之间迁移，token budget 怎样决定本轮能推进多少 prefill / decode，chunked prefill 为什么可以和 decode 同轮共存。它承接 03 单元的块管理，也会把你送到 05 单元的算子层；更下游的路由、网关、多机扩展留到后面的单元。

课程地图上一单元下一单元

读完本页应该能做什么

  • 你能看一条调度轨迹，分出 waiting、running、preempted 和 finished 各自什么时候变化。
  • 你能读懂 `max_num_batched_tokens`、`max_num_seqs`、`max_num_partial_prefills`、`scheduler_reserve_full_isl` 这些参数会把性能推向哪一边。
  • 你能从 TTFT、ITL、吞吐、等待队列长度和 preemption 次数里，判断问题是在预算、队列还是抢占。

先把问题定住

goals

这页先抓什么

先看请求怎样进入 waiting queue，怎样被 scheduler 选进 running queue，怎样在 token budget 不够时继续排队，怎样在显存压力下被抢占后回到等待队列。只要这条状态线没断，后面的参数和 benchmark 才有地方放。

先别展开什么

先不要把注意力放到路由、网关、分布式通信、speculative decoding、kernel fusion 或 MoE expert parallelism 上。那些内容会在后面的系统单元或算子单元里再回来。

读完这一节，你要能做什么： 拿到一条请求轨迹，先说出它现在在哪个队列，再判断这轮能不能进 batch，最后说明它如果被抢占，是 swap 回去还是 recompute。

continuous batching 在改什么

overview

静态 batch 的思路是先凑齐，再一起跑完。continuous batching 的思路是每一轮迭代都重排一次队伍：先让正在 decode 的请求继续前进，再把剩余 token budget 分给 chunked prefill，必要时再把长请求切块、把低优先级请求抢占出去。这样做的目的，是让 decode 不被长 prefill 长时间挡住，同时让新请求更快进入服务。

层次 静态 batch continuous batching 你要盯住的变化
请求进入 等一批请求凑齐后再开始 新请求可以在每轮迭代之间进入 观察 waiting queue 的长度和进入时机
prefill 常常整段一次性推进 长 prompt 可以被切成多个 chunk 观察 chunked prefill 是否抢占了过多预算
decode 容易和 prefill 一起被大 batch 牵着走 通常优先保住 decode 的节奏 观察 ITL 是否因为长 prefill 被拖慢
状态更新 一轮里状态变化少 每轮都可能有完成、继续、回队、抢占 观察 running / waiting / preempted 的迁移
这页的锚点例子： 一组请求 A、B、C、D 会在不同轮次里交替出现。你后面看到的表格和代码，都会围着这组请求转。

状态账本

ledger

调度器最重要的是知道每个请求现在处于什么状态，接下来要读什么、写什么、丢什么。下面这张表把状态迁移摆平，后面的预算和抢占都可以回到这里检查。

对象 进入前是什么 进入后是什么 读什么 写什么 / 变更什么 错了会怎样
waiting request 还没拿到本轮预算 继续等待，或被挑进 batch 队列顺序、优先级、前缀命中情况 排队位置、是否 chunked prefill 会把“没进 batch”误当成“服务挂了”
running request 已经在上一轮进入 batch 继续 decode，或推进一个 prefill chunk，或结束 当前剩余 token、KV cache 可用空间 新增 token 的状态、完成标记、流式返回 会把 decode 继续推进误看成“没有新状态”
partial prefill request 长 prompt 还没走完 保留已完成的 chunk，等待下一轮继续 剩余 prompt tokens、chunk 上限、token budget 已完成 chunk 的进度 会把长 prompt 当成一次性输入，预算算大
preempted request 本轮资源不够，先被让出位置 回 waiting，或走 swap / recompute 抢占模式、KV cache 压力、优先级 是否保留块、是否重算、是否重新排队 会把抢占当成错误，忽略它其实是在保住整体吞吐
finished request 还占着运行位和 KV 块 释放资源，退出调度循环 停止条件、采样结果、流式输出状态 释放 KV block、清掉 running 占位 会把已完成请求当成仍在占预算
先检查哪里： 如果你在日志里只看到“慢”，先不要碰 kernel。先回到这张表，看请求到底是 waiting、running、partial prefill 还是 preempted。

调度循环怎么跑

mechanism

这一段把调度器拆成五步：新请求进入、预算检查、组 batch、执行 forward、回写状态。读的时候不要只盯最后一次 forward，要看每一步到底读了什么、写了什么、丢了什么。

进入 新的请求到达后先进入 waiting queue，不等于已经拿到 batch 里的位置。

看点：进入队列只是建立候选状态，不代表已经消耗 token budget。

预算 调度器会同时看 `max_num_batched_tokens`、`max_num_seqs`、KV cache headroom 和抢占策略。

看点：预算由 token budget、seq 上限、KV headroom 和抢占策略一起决定。

组批 decode 请求通常先保住节奏，剩下的预算再给 chunked prefill。

看点：这一步决定 TTFT 和 ITL 谁更容易被照顾。

执行 模型只处理这一轮被选中的 token，下一轮要重新排队。

看点：continuous batching 的“continuous”来自每轮重新排队和重新组批。

回写 完成的请求释放资源，没完成的请求回到队列，必要时切换到 swap 或 recompute。

看点：状态回写决定下一轮能不能继续前进。

最小调度循环伪代码
while engine_running:
    retire_finished_requests()
    free_kv_for_finished_requests()

    decode_batch = pick_decode_requests(running_queue, token_budget)
    prefill_batch = pick_prefill_requests(
        waiting_queue,
        token_budget,
        chunked_prefill=True,
    )

    batch = decode_batch + prefill_batch
    outputs = model_forward(batch)
    update_request_states(outputs)

    if kv_pressure_high():
        preempt_low_priority_requests(mode=preemption_mode)
这段代码要你盯住的 4 个动作： 先退场已完成请求，再挑 decode，再把剩余预算留给 prefill chunk，最后回写状态。只要这四步中的任一步顺序错了，队列表现就会变形。

调度窗口怎么看

window

调度窗口可以理解为“这一轮允许被看见和被推进的请求集合”。它受三类限制共同影响：token 预算、序列上限和 KV cache headroom。窗口太窄，短请求也会排队；窗口太宽，长 prefill 和过多 running 请求会把 KV 压紧，后面只能靠抢占收场。

做线上判断时，不要只看平均输入长度。最好同时看 p95 prompt 长度、当前 decode 并发、KV 使用率峰值和 repeated preemption 数量；这四个量合在一起，才能说明到底是哪一层限制先撞墙。

限制 它在保护什么 窗口变窄时 窗口变宽时
token budget 单轮 forward 的计算量和 prefill / decode 比例 prefill chunk 更小，TTFT 可能上升 吞吐可能上升，但 decode 更容易被长 prefill 挤压
`max_num_seqs` 同一轮能挂住多少条序列 waiting queue 更容易增长 并发更高，KV cache 压力也更高
KV headroom 后续 token 写入和请求恢复的空间 准入更保守，短期吞吐受限 更容易 over-admission，抢占和重算成本会上来
读日志时的抓手： 如果 waiting 增长，看窗口是否太窄；如果 repeated preemption 增长，看窗口是否放得太宽；如果 TTFT 和 ITL 互相拉扯，回到 token budget 和 chunked prefill 的比例。 别把瞬时峰值当成稳定瓶颈，至少跨几个连续调度轮次一起看。

一轮一轮看 4 个请求怎么走

example

下面这个例子只跟踪 A、B、C、D 四个请求。A 和 B 先在 decode 阶段继续生成；C 是一个长 prompt；D 是一个短 prompt，但到得更晚。这样安排是为了让你一次看到 decode 优先、chunked prefill、等待队列和抢占四件事。

轮次 本轮输入 预算状态 调度决策 结果 要看出的原则
t0 A、B 正在 decode；C 刚到，prompt 很长；D 还没到 token budget 足够容纳少量 decode 和一段 prefill chunk A、B 继续 decode，C 只推进一个 chunk A 可能完成；B 继续；C 保留 prefill 进度 decode 不该被长 prefill 一口气挤掉
t1 C 还没走完；D 到达，且输入更短 剩余预算更紧 优先保住正在 decode 的请求，再把剩余 token 分给 D 和 C 的下一段 chunk D 可能比 C 更快进入 running 短请求不应永远排在长 prefill 后面
t2 B 还在生成；C 继续分段 prefill；D 转入 decode running 数接近上限 不再放新请求进来，直到下一轮有人完成或释放位 waiting 增长，但 batch 保持稳定 `max_num_seqs` 在这里开始起约束作用
t3 C 的 prefill 还没结束，但 KV cache headroom 变紧 需要给后来的请求留空间 低优先级或长尾请求 C 被抢占，之后按 recompute 或 swap 恢复 C 回到 waiting；别的请求继续前进 抢占的作用是保住整体吞吐
这张表最重要的结论： 一轮里既可以有 decode，也可以有 chunked prefill，还可以有抢占。continuous batching 关注每轮怎么重新组批，让不同状态的请求按预算继续前进。

参数怎么把系统推向不同方向

tuning

调参时不要只问“这个值大不大”，要问它在减少哪一种等待。`max_num_batched_tokens` 更多是在调每轮 token 预算，`max_num_seqs` 更多是在调并发请求数，`max_num_partial_prefills` 和 `max_long_partial_prefills` 影响长 prompt 能同时被切几段，`scheduler_reserve_full_isl` 则决定你在准入时会不会把整段输入长度一起算进去。

$$B = \text{max\_num\_batched\_tokens}$$ $$\text{chunk} = \min(P_{\text{remaining}}, \max(0, B - D - R))$$

这里 \(B\) 是这一轮的总 token 预算，\(P_{\text{remaining}}\) 是还没走完的 prompt tokens，\(D\) 是这轮已经给 decode 留出的 token，\(R\) 是安全余量或保留头寸。这个式子用来帮助你判断为什么长 prefill 会和 decode 争预算。

参数 主要控制什么 调大 / 调小会怎样 常见误读
`max_num_batched_tokens` 单轮总 token 预算 调小通常更利于 ITL，调大通常更利于 TTFT 和吞吐 把它当成“吞吐越大越好”的单旋钮
`max_num_seqs` 单轮最多处理多少个序列 调小会限制并发，调大能让更多请求进来，但也更吃 KV cache 把它当成 token budget 的替代品
`max_num_partial_prefills` 同时进行中的部分 prefill 数 调大能让更多长 prompt 同时切块推进，调小能让排队更保守 把长 prompt 和短 prompt 看成同一种调度成本
`max_long_partial_prefills` 超过阈值的长 prompt 同时推进的数量 调小后，短 prompt 更容易插队；调大后，长 prompt 更容易并行切块 忽略长请求和短请求的队列差异
`long_prefill_token_threshold` 多长算“长 prompt” 阈值降低会让更多请求走长 prompt 路径，阈值升高则相反 默认阈值一定适合自己的长度分布
`scheduler_reserve_full_isl` 准入时是否先检查整段输入是否装得下 打开后更保守，能减少 over-admission 和 KV thrashing 只看第一块就够了
`preemption_mode` 被抢占时是 swap 还是 recompute swap 保留块，recompute 丢块重来 抢占只是“暂停”，不会改变后续成本
`scheduling-policy` 先来先服务还是优先级调度 `priority` 能让高优先级请求先走，但更容易牺牲公平性 公平和低延迟永远可以同时拉满
调参顺序建议： 先看 token budget 和 `max_num_seqs`，再看长 prefill 的并行数，最后才看抢占模式。顺序反了，容易把问题从预算层拖到抢占层。

答案至少要包含：`max_num_batched_tokens`、长 prompt 并行数、以及请求长度分布或 `scheduler_reserve_full_isl` 之一。

参考答案 / 评分要点

可以先查：1）`max_num_batched_tokens` 是否过小，导致 prefill chunk 被切得太细；2）`max_num_partial_prefills` 和 `max_num_long_partial_prefills` 是否让长请求并行过多，挤压了短请求；3）`scheduler_reserve_full_isl` 是否没打开，或者请求长度分布变化后仍按老预算准入。只要能把“预算、长请求并行、准入检查”三件事说清楚，就算合格。
