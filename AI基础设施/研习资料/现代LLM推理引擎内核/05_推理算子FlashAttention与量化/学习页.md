# 算子路径 {#kernel}
<!-- tag: kernel -->

一次在线推理不是把 `model.forward` 跑一遍就结束。引擎会先对 prompt 做 prefill，把每层的 KV cache 写出来；之后每轮 decode 只推进一个或少量 token，再更新 KV cache、计算 logits、采样下一个 token。算子层关心的是 attention、矩阵乘、归一化、采样，以及框架为了拼 batch、重排 KV block、拷贝数据而产生的辅助 kernel。

伪代码：调度循环看到的 kernel 边界复制

```python
while engine_has_work:
 batch = scheduler.pick_prefill_and_decode_requests()
 token_metadata = kv_manager.map_tokens_to_cache_blocks(batch)

 hidden = embedding_and_position(batch.input_tokens)
 for layer in model.layers:
 q, k, v = layer.project_qkv(hidden)
 hidden = attention_backend.run(q, k, v, token_metadata)
 hidden = layer.mlp(layer.norm(hidden))

 next_tokens = sampler(logits(hidden), sampling_params)
 kv_manager.commit_new_blocks(batch, next_tokens)
```

Attention(Q, K, V) = softmax(QKT / sqrt(d)) V
直觉上，真正重的是把完整 score 矩阵写回 HBM 再读回来，显存往返会把时间吃掉。

# FlashAttention 的核心 {#flash}
<!-- tag: flash -->

FlashAttention 的要点是 IO-aware：把 Q/K/V 切成块，在 GPU 片上 SRAM 中做局部计算，同时维护 online softmax 所需的统计量，避免把完整 attention 矩阵落到 HBM。它保留精确 attention，省下来的是中间结果的搬运。

普通路径
算 score -> 写入 HBM -> 读回 score -> softmax -> 再乘 V。中间矩阵越大，显存压力越高。


FlashAttention 路径
分块读 Q/K/V -> 在片上完成局部 softmax 统计 -> 合并输出。减少中间结果往 HBM 的往返。


为什么适合长序列
长上下文让 attention score 矩阵迅速变大。少写中间矩阵，就能把瓶颈从 HBM 访问拉回更多有效计算。

## box warn
### 读法提醒
讲 FlashAttention 时，先说“减少 HBM 中间读写”，再说 tiling、online softmax 和重计算。顺序反过来，容易先陷进实现细节。

# 阶段差异 {#phases}
<!-- tag: phases -->

prefill 阶段要处理 prompt 中的多个 token。它更接近一个大矩阵问题，长 prompt 会带来明显的 attention 计算和中间状态压力。decode 阶段每轮通常只有新生成 token 参与查询，但它要访问越来越长的历史 KV cache，因此瓶颈常常变成“从 KV cache 读大量历史 K/V，再为少量 query 算结果”。

| 维度 | Prefill | Decode |
| --- | --- | --- |
| 输入形状 | 一次处理 prompt 的许多 token。 | 每步处理新 token，读取历史 KV。 |
| 主要压力 | 长序列 attention、矩阵乘、写 KV cache。 | KV cache 读取、batch 拼接、短 query attention。 |
| FlashAttention 直觉 | 减少长序列 attention 的中间矩阵读写。 | 需要结合 paged / radix / prefix cache 的布局和后端支持看收益。 |
| 调优入口 | max batch token、chunked prefill、attention backend。 | 并发度、KV cache dtype、block size、prefix cache、decode backend。 |

# 后端选择 {#backend}
<!-- tag: backend -->

vLLM 和 SGLang 都会根据硬件、dtype、head size、序列长度、量化配置和缓存布局选择或回退 attention backend。后端受兼容矩阵约束，性能只是其中一项；不满足时会回退或报错。你要先确认它在用哪条路径，再谈快不快。

| 检查项 | 为什么要看 | 常见误判 |
| --- | --- | --- |
| GPU 架构 | 不同 backend 的支持范围不同。 | 以为“能跑就一定是最快实现”。 |
| dtype | FP16、BF16、FP8、INT8 等会影响支持路径。 | 把量化收益和 backend 收益混成一件事。 |
| head size / shape | 某些实现对 head dim 有硬约束。 | 忽略 batch 中长短混排导致的回退。 |
| KV cache dtype | 会影响 decode 侧的读写模式和兼容性。 | 只盯模型参数，忘了缓存对象是另一笔账。 |

## box warn
### 判断口诀
先看兼容，再看性能；先看阶段，再看 kernel。顺序倒过来，排障会绕远。

# 具体案例 {#example}
<!-- tag: example -->

假设线上出现这样一组组合信号：32k 长 prompt 的首 token 时间明显变长，decode 侧 TPOT 也上升，显存峰值接近上限，日志里还偶尔出现 attention backend 回退，而业务团队最近刚把权重量化从 BF16 改到 INT4。这个案例最容易把人带偏，因为每个信号都像“主因”，但它们实际上可能来自不同层。

正确的切法是先把它拆开：prefill 侧先看长 prompt 和 FlashAttention 的 HBM 往返；decode 侧先看 KV cache 的读写和 block 回收；后端侧再看兼容矩阵和回退路径；量化侧最后再看是权重省下来了，还是 decode 侧的 cache 仍然在撑爆显存。每一层各看一遍，才不会把四个信号拎成一个结论。

| 信号 | 更像哪一层 | 优先看什么 | 常见误判 |
| --- | --- | --- | --- |
| 首 token 很慢 | prefill | 长 prompt、chunked prefill、backend、prefix cache | 先怪 sampler 或 HTTP 层 |
| TPOT 变慢 | decode | KV cache 读写、batch 形状、抢占和回收 | 只看单个 kernel benchmark |
| 显存接近上限 | KV cache / 权重 | cache 账本、dtype、并发 token | 把权重量化当成全部答案 |
| backend 回退 | 兼容矩阵 | GPU 架构、head size、dtype、框架版本 | 把“能跑”误解成“走了目标实现” |

最小判定流程复制

```text
1. 先分阶段：prefill 还是 decode。
2. 再分对象：backend、KV cache、权重还是激活。
3. 然后查日志：是否回退、是否抢占、是否重算。
4. 最后比指标：TTFT、TPOT、ITL、吞吐和显存峰值。
5. 如果一个动作只改了局部，别急着把它叫做全局优化。
```

### 失败后果
如果你先看错层，就可能把一个调度问题当成 kernel 问题，把一个 cache 问题当成量化问题，最后做了很多动作，却只是在不同层之间搬移瓶颈。

# 量化账本 {#quantization}
<!-- tag: quantization -->

量化改的是数值表示，调度路径不变。常见路径包括权重量化、KV cache 量化和激活量化。前者省的是模型常驻权重显存，后者省的是并发上下文保存 K/V 的显存。两者作用对象不同，所以收益曲线也不同。

| 对象 | 省什么 | 更常见的副作用 |
| --- | --- | --- |
| 权重 | 模型常驻显存。 | 反量化开销、kernel 支持差异、质量波动。 |
| KV cache | 长上下文和高并发下的 cache 显存。 | 需要 scale 或额外 metadata，decode 侧更敏感。 |
| 激活 | 中间状态峰值。 | 实现复杂，端到端收益不一定稳定。 |

KV cache 显存 ≈ layers × tokens × 2 × kv_heads × head_dim × bytes_per_element
这只是简化账本。实际还要加 scale、对齐和 block 管理等开销。

## box info
### 结论先行
权重量化不必然让 KV cache 也同比下降。若高并发长上下文仍然 OOM，问题通常还在 KV cache 账本上，而不在模型参数。

# 排查路线 {#diagnosis}
<!-- tag: diagnosis -->

当你遇到 TTFT 变慢、TPOT 变慢、显存不足、backend 回退或输出质量波动时，先别急着调参。先把现象归类，再沿着阶段、对象、兼容性和端到端指标逐层缩小范围。

## box
### 案例 A：长 prompt 首 token 很慢
先看 prefill，重点确认 chunked prefill、attention backend、prefix cache 和长 prompt 是否被分到过多轮次。

## box
### 案例 B：高并发长上下文 OOM
先看 KV cache。权重量化如果已经做过，下一步通常是 KV cache dtype、并发 token、block 管理和回收策略。

## box
### 案例 C：切换 backend 后回退
检查 GPU 架构、dtype、head size、KV cache dtype 和框架版本，不要把“跑起来了”误判成“走了目标实现”。

## box
### 案例 D：显存降了但速度没提上去
量化把存储省下来了，但反量化和带宽瓶颈可能把收益吃回去。先分别测 prefill 和 decode，再看 profiler。

# 资料路线 {#bridge}
<!-- tag: bridge -->

这页不要求你把 FlashAttention、量化和 backend 的源码全读完。它只要求你能沿着公开资料把问题定位到正确一层。先把文档分成三条线：优化文档回答“默认会怎么调”；引擎参数文档回答“你能不能把它调成你想要的样子”；profiling 文档回答“线上到底是不是你以为的瓶颈”。

例如，当你怀疑某个模型在 decode 侧变慢时，先看 vLLM 的 engine args 里是否启用了 chunked prefill、preemption、max-num-seqs 之类的约束，再看 scheduler API 里是否存在抢占或重算路径，最后回到 SGLang 的 benchmark / profiling 读 TTFT、TPOT、ITL 和吞吐。这样排查，顺序和资料顺序是一致的，不会把调度问题误读成 kernel 问题。

再往下一层看源码时，也不用一口气把仓库打开。先找入口，再找边界，再看对象流向。你真正需要跟踪的通常只有三件事：调度器把什么请求推进了 batch，backend 选了哪条实现，量化对象落到了哪里。三件事都确认完，源码阅读就从“漫游”变成“对照表”。

## box soft
### 阅读顺序
1. 先读优化和 tuning 文档，确定 backend、chunked prefill、cache 和量化策略的大方向。
2. 再读 engine args，核对你能设置的边界参数和它们的副作用。
3. 然后看 scheduler API，确认抢占、重算和 token budget 的行为。
4. 最后回到 profiling，确认线上数据和你的推理是不是一致。

## box info
### 源码触发条件
只有当文档已经把边界说明清楚、但现象仍然解释不通时，才去看源码。这样读代码的目的就很明确：“确认分支、对象和回退点”。
如果文档和日志已经能说明问题所在，就没有必要把学习预算花在不相关的实现细节上。

| 资料 | 主要问题 | 你拿它做什么 |
| --- | --- | --- |
| vLLM Optimization and Tuning | 默认优化策略是什么。 | 确认 chunked prefill、backend、cache 和量化的初始假设。 |
| vLLM Engine Arguments | 你能调哪些边界。 | 核对并发、抢占、分块 prefill 与长 prompt 的控制参数。 |
| vLLM Scheduler API | 调度器在什么条件下会抢占或重算。 | 把“服务突然变慢”拆成 token budget 或抢占路径问题。 |
| SGLang Benchmark and Profiling | 线上指标如何读。 | 对照 TTFT、TPOT、ITL、吞吐和显存峰值。 |

公开资料检查清单复制

```text
1. 先确认问题属于 prefill、decode 还是 KV cache 压力。
2. 再确认 backend 是否满足 dtype、head size、GPU 架构和框架版本。
3. 然后看量化对象是权重、KV cache 还是激活值。
4. 最后把 kernel benchmark 和端到端指标放在同一张表里对照。
5. 如果文档已经能解释现象，就先别跳到源码。
6. 如果文档解释不了，再用源码确认分支和回退点。
```

## box warn
### 你现在应该能做什么
拿到一份公开文档时，不再只是“读过”，而是能说出它回答的是哪一层问题，以及它和线上排障之间隔着哪一步。

## box soft
### 实现对照
把文档和实现对照时，最有用的盯住几个稳定的分支点：请求是走 prefill 还是 decode、backend 是否满足兼容矩阵、量化对象是否进入了正确的缓存层、以及抢占或重算有没有把预期收益抵消掉。
如果你看到日志里 backend 被回退，或者量化后延迟没变快，最稳妥的做法是先回到参数文档确认边界，再去看实现里的分支。这样你会更容易发现，是 dtype 不对、head size 不合、cache 布局不匹配，还是某个路径在 decode 侧偷偷触发了额外的反量化。

## box
### 实现检查表
- 调度器是否真的把高并发请求送进了同一个 batch。
- attention backend 是否按预期选中，而不是在日志里悄悄回退。
- 量化对象是否正好落在你想优化的层级上。
- profiling 是否同时记录了 TTFT、TPOT、ITL、显存峰值和输出质量。

## box info
### 失败后果
如果这四项里有一项没对齐，你可能会得到一种很糟糕的结果：单点 benchmark 看起来更快，端到端服务却更慢；显存看起来更省，真正的瓶颈却只是从权重挪到了 KV cache；文档看起来都支持，实际部署却一直在走回退路径。
这也是为什么这页把“资料路线”放在中间，而不是放在最后。

从文档到实现的最短闭环复制

```text
文档确认边界
 -> 参数核对兼容矩阵
 -> 观察日志是否回退
 -> 读取 profiler 指标
 -> 再决定是否看源码
 -> 最后回到练习区写下诊断结论
```

把这条链跑顺以后，你会更容易把“我看懂了”变成“我知道下一步该验证什么”。这也是这页最重要的迁移目标。

# 收尾判断 {#wrapup}
<!-- tag: wrapup -->

这一页只有一条判断顺序：先分阶段，再分对象，再看兼容，再看端到端指标。FlashAttention、backend 选择和量化都只是这条链上的不同位置，不要把它们混成一个动作。

如果某次改动让显存下降，也让单项 benchmark 变快，先别急着下结论。先看瓶颈是不是从权重挪到 KV cache，从 prefill 挪到 decode，或者从计算挪到回退路径。只要位置变了，结论就会变。

## box soft
### 这时该看什么
- TTFT 是否主要由 prefill 拉长。
- TPOT / ITL 是否主要由 decode 侧的 cache 读写拉长。
- backend 是否悄悄回退到更慢的实现。
- 量化是否引入了额外的反量化开销。

## box info
### 下一步动作
如果这四项都看过了，下一步就是把结论写进练习记录里：你要能用一两句话说清楚现象、阶段、对象和修正动作。这样做出来的记录，才算是能拿去复盘的记录。

收尾检查单复制

```text
1. 先说清楚问题属于哪一层：请求、缓存、backend 还是量化。
2. 再说清楚你看到的指标：TTFT、TPOT、ITL、吞吐或显存峰值。
```
