06｜结构化生成与约束解码

这一单元把“模型要输出 JSON、表格、枚举、工具参数或固定格式”这类需求，翻回推理引擎内部看。重点不是把输出包成漂亮格式，而是理解约束如何进入 token 选择过程，哪里会提前过滤，哪里会回退，哪里会让 decode 变慢。

先抓主线

learning

先看对象

区分“输出格式”与“约束对象”。前者是你希望得到什么形状的文本，后者是 decode 过程里允许哪些 token、哪些路径、哪些结束条件。

先看验证

能输出合法 JSON 不等于能稳定生产。要继续看 schema 完整性、backend 支持、超时、回退路径和质量漂移。

术语与边界

terms
术语直觉解释精确定义容易混淆的地方
structured outputs让模型输出看起来像 JSON、表格或固定标签。用约束把生成结果限制在某种可验证结构里。不是只在最后做字符串清洗。
guided decoding一边生成，一边限制可选 token。在采样前或采样时把不合法 token 排除掉。不是改模型参数本身。
constraint backend负责把 schema/grammar 变成可执行规则的组件。把结构约束转成 token-level 过滤或状态机。backend 不只影响正确性，也影响性能和兼容矩阵。
post-hoc validation生成后再检查格式对不对。在输出完成后执行解析、校验或修复。它和 guided decoding 不是一层，不能混着说。

约束怎么进 decode

pipeline

先看一个最小闭环。模型给出 logits，约束器把不允许的 token 变成不可选，采样器在剩下的 token 里挑一个，状态机再推进一步。这里的关键“谁在改可选集合，谁在记住当前位置”。

代码
state = init_constraint_state(schema_or_regex_or_grammar)
for step in range(max_new_tokens):
 logits = model.forward(context, state)
 allowed = backend.valid_tokens(state, logits)
 masked_logits = apply_mask(logits, allowed)
 token = sample(masked_logits, temperature, top_p)
 state = backend.advance(state, token)
 if backend.is_finished(state, token):
 break
对象输入前状态输出后状态读取什么写入什么错了会怎样
约束配置只有 prompt 和参数。带 schema / regex / grammar 的可执行约束。prompt、配置、开始位置。约束状态机、token 规则。模型会生成合法文本，但不一定合法结构。
token masklogits 还没过滤。非法 token 已被压掉。当前状态、词表、合法前缀。mask 后的 logits。会出现“看起来能采样，实际上已越界”。
推进状态还停在前一个 token 后。进入新的结构位置。刚采样的 token、语法规则。状态机指针、分支栈。会在括号、字段、结束符附近卡住。
结束判断还没满足收束条件。输出被接受并结束。state、EOS、结构闭合情况。停止或继续。会提前结束或无限追 token。

约束方法

methods

vLLM 和 SGLang 都把“结构化输出”做成公开能力，但支持的约束类型、后端名字和接入方式不完全一样。学习时不要先问“谁更高级”，先问“这条约束能不能落到当前 workload 上”。

vLLM 的公开路线

公开文档里能看到 choice、regex、json、grammar、structural tag 等选项，也能看到 outlines、lm-format-enforcer、xgrammar 这类 guided decoding backend。你要先看文档支持哪种约束，再决定是否继续往源码里下钻。

SGLang 的公开路线

公开文档里能看到 JSON schema、regex、EBNF 这些约束参数，也能看到不同 grammar backend。它的重点把约束放到 serving 和 reasoning 语境里一起看。

把约束落到一条请求上

case

假设你要从一段产品反馈里抽取联系人信息，最后必须输出一个对象，字段只有 name、company、email。这类任务最容易暴露约束解码的真实位置：它在每一步采样前收窄 token 集合。

阶段输入状态状态变化重点看什么错了会怎样
起始只有 prompt 和约束定义。backend 把 schema / regex / grammar 转成可执行规则。约束是否能被当前模型和 runtime 支持。请求看起来合法，实际一开始就会回退。
对象头还没进入字段。允许的 token 主要围绕对象起始、字段名和分隔符。前缀是否真的进入了结构空间。模型会继续吐自然语言而不是结构化内容。
字段值已经选定字段名。合法 token 变成字段值本身和闭合符号。字段类型、长度和格式限制是否过窄。会在引号、逗号或邮箱格式附近卡住。
收束对象接近完整。状态机确认闭合并允许结束。终止条件是否可达。会提前停，也可能一直追 token。

 "constraint": {
 "type": "json_schema",
 "schema": {
 "type": "object",
 "required": ["name", "company", "email"],
 "properties": {
 "name": { "type": "string" },
 "company": { "type": "string" },
 "email": { "type": "string", "format": "email" }
 }
 }

怎么把问题拆开

先分清是“输出无法解析”，还是“输出可解析但语义不对”。前者多半是约束和 backend 的问题，后者才更多回到 prompt、模型和数据分布。

练习区的第 5、6、8 题都是在追这条链：字段、前缀、终止条件、fallback 和状态推进。把它们串起来，你就能从一条错误日志直接回到该看哪一层。

backend、状态机和回退

backend

同样是“约束解码”，不同 backend 的工作方式不完全一样。你不用把每个实现都背下来，但要知道它们在解决什么问题：有的偏向把结构约束翻成可执行的 token 过滤，有的偏向把语法规则转成状态机，有的偏向和现成的约束库对接。工程上最重要的不是名字，而是你能不能从日志里分辨出“这次是约束对象不对、状态推进不对，还是 backend 不支持当前形状”。

读法输入约束内部状态输出 mask常见症状
schema对象、字段、类型、数组和必填项。当前位置在第几个字段、是否已经进入值区。只允许当前字段需要的 token。字段闭合、额外文本、类型错误。
regex模式、分支、重复和终止条件。当前前缀是否还在合法路径上。把不匹配前缀的 token 过滤掉。在某个前缀附近反复兜圈。
grammar语法规则、非终结符、终止符。解析栈、展开顺序、可归约状态。只暴露能让语法继续前进的 token。括号不闭合、语句卡住、结束不了。
choice有限枚举和固定标签。只要记住当前是否已经选定。只放行极少数合法 token。最稳，但表达空间很窄。

为什么会回退

回退通常backend 发现当前模型、dtype、shape 或约束形状不在支持范围里。系统为了能继续服务，会换一条较保守的路。这条路能跑，但不一定保留原来那条路径的速度和约束能力。

看日志时的顺序

先确认约束类型，再确认 backend 名字，再确认是否 fallback，最后看输出是否在采样前就已经被 mask 改掉。顺序如果倒过来，很容易把后验错误当成前置错误。

故障分流记法

1. 解析失败：先看 schema / grammar 规则是否写错。
2. 语法卡住：先看状态是否还能推进到下一个合法前缀。
3. backend 回退：先看模型、dtype、shape 和支持矩阵。
4. 结果合法但不对：再回到 prompt、数据分布和任务定义。

这就是本单元真正要你记住的动作：先判断问题究竟出在约束定义、状态推进、backend 兼容，还是输出内容本身。

三类失败怎么分流

triage

实际排障时，结构化输出最常见的三种失败看起来都像“格式坏了”。第一种是输出根本不能解析，第二种是输出卡在某个前缀上不动，第三种是系统为了兼容性偷偷回退了 backend。它们表面都表现为“结果不对”，但处理顺序完全不同。

失败类型最早看到的症状最可能的根因先做什么后做什么
解析失败输出像 JSON，但解析器报错。schema 写错、额外文本、字段类型不稳。检查 schema 和输出片段是否一一对应。再看 prompt 和数据分布。
前缀卡住模型在某个前缀上反复重复。regex / grammar 太窄，终止条件太晚。缩小规则，验证合法前缀能不能推进。再看状态机有没有被错误推进。
fallback日志里出现回退，速度明显变慢。模型、dtype、shape 或长度不在支持矩阵里。确认 backend 是否支持当前配置。再比较回退后性能损失有多大。
语义不对结构合法，但内容不符合任务。任务定义、prompt 或数据分布不稳。先保留结构，再改任务输入和数据。最后才考虑重设约束形状。
代码
1. 先判断 parse 失败、前缀卡住、还是 backend 回退。
2. 解析失败先查 schema、字段和额外文本。
3. 前缀卡住先查 regex / grammar / state。
4. 回退先查模型、dtype、shape 和支持矩阵。
5. 语义不对才回到 prompt、任务和数据分布。

这段和后面练习的关系

练习区第 5、6、8 题对应的就是这四类问题。你答题时如果能先分流，再说下一步，就说明已经把约束解码当成一条可追踪的工程路径，而不是一个格式标签。

一个小例子

example

假设你要抽取一条联系人记录，要求输出 `, , 三个字段，而且 ` 必须是合法邮箱。没有约束时，模型可能写成自然语言；加上 JSON schema 后，模型必须在字段、引号和类型上收束；如果再加 regex 或 grammar，连局部形式都要跟着变窄。

代码
prompt: "Extract a contact card from the paragraph."
constraint:
 type: json_schema
 schema:
 type: object
 required: ["name", "company", "email"]
 properties:
 name: { type: string }
 company: { type: string }
 email: { type: string, format: email }

第一步

先让 schema 说清结构，不要先急着优化 prompt 文案。

第二步

再确认 backend 能否把 schema 变成 token 级规则，而不是只做后验解析。

为什么这个例子有用

它把“约束对象”“结构对象”“输出对象”分开了。只要你能把这三层拆开，后面遇到 tool call、枚举、JSON、表格，判断就会快很多。

换句话说，结构化生成把允许的路径收窄。你在这里看见的是一个请求如何被 schema 牵住；你在后面的引擎单元里会继续看见，这根绳子会落到哪个 backend、哪段状态机、哪条日志。

terminal
