第 06 单元

结构化生成与约束解码

这一单元把“模型要输出 JSON、表格、枚举、工具参数或固定格式”这类需求，翻回推理引擎内部看。重点不是把输出包成漂亮格式，而是理解约束如何进入 token 选择过程，哪里会提前过滤，哪里会回退，哪里会让 decode 变慢。

本页基于 2026-06-21 可访问的公开文档：vLLM 的 Structured Outputs、SGLang 的 Structured Outputs，以及它们对应的官方仓库与示例。

读完后，你应该能做的事

  • 分清结构化生成、guided decoding、后验校验这三层分别在做什么。
  • 看懂 JSON schema、regex、grammar、choice 这些约束类型的适用边界。
  • 拿到一条输出错误日志时，先判断是约束设错、backend 回退，还是模型本身不稳定。

先抓主线

learning goals

先看对象

区分“输出格式”与“约束对象”。前者是你希望得到什么形状的文本，后者是 decode 过程里允许哪些 token、哪些路径、哪些结束条件。

先看位置

约束不是在样本结束后再修一遍，而是在生成过程中影响 token 选择。这个位置一错，后面所有性能与正确性判断都会偏。

先看验证

能输出合法 JSON 不等于能稳定生产。要继续看 schema 完整性、backend 支持、超时、回退路径和质量漂移。

概念骨架

map
prompt / 消息 → 约束来源
schema / regex / grammar / choice → guided decoding
token 过滤与状态推进 → 受约束的输出

这条链里，前半段决定“你想要什么”，中间段决定“模型此刻还能选什么”，后半段决定“结果是否真的可用”。如果你把约束只看成后处理，后面就很难解释为什么有些请求会变慢、会回退，甚至会卡住。

术语与边界

terms
术语 直觉解释 精确定义 容易混淆的地方
structured outputs 让模型输出看起来像 JSON、表格或固定标签。 用约束把生成结果限制在某种可验证结构里。 不是只在最后做字符串清洗。
guided decoding 一边生成，一边限制可选 token。 在采样前或采样时把不合法 token 排除掉。 不是改模型参数本身。
constraint backend 负责把 schema/grammar 变成可执行规则的组件。 把结构约束转成 token-level 过滤或状态机。 backend 不只影响正确性，也影响性能和兼容矩阵。
post-hoc validation 生成后再检查格式对不对。 在输出完成后执行解析、校验或修复。 它和 guided decoding 不是一层，不能混着说。

约束怎么进 decode

pipeline

先看一个最小闭环。模型给出 logits，约束器把不允许的 token 变成不可选，采样器在剩下的 token 里挑一个，状态机再推进一步。这里的关键不是“格式对不对”，而是“谁在改可选集合，谁在记住当前位置”。

伪代码：约束如何进入 token 选择
state = init_constraint_state(schema_or_regex_or_grammar)
for step in range(max_new_tokens):
    logits = model.forward(context, state)
    allowed = backend.valid_tokens(state, logits)
    masked_logits = apply_mask(logits, allowed)
    token = sample(masked_logits, temperature, top_p)
    state = backend.advance(state, token)
    if backend.is_finished(state, token):
        break

\(m_i = 0\) 表示 token \(i\) 允许，\(m_i = -\infty\) 表示 token \(i\) 不能选。

\(p'_i = \mathrm{softmax}((logits_i + m_i)/\tau)\) 。约束没有改模型本体，只是在采样前改了可选集合。

对象 输入前状态 输出后状态 读取什么 写入什么 错了会怎样
约束配置 只有 prompt 和参数。 带 schema / regex / grammar 的可执行约束。 prompt、配置、开始位置。 约束状态机、token 规则。 模型会生成合法文本，但不一定合法结构。
token mask logits 还没过滤。 非法 token 已被压掉。 当前状态、词表、合法前缀。 mask 后的 logits。 会出现“看起来能采样，实际上已越界”。
推进状态 还停在前一个 token 后。 进入新的结构位置。 刚采样的 token、语法规则。 状态机指针、分支栈。 会在括号、字段、结束符附近卡住。
结束判断 还没满足收束条件。 输出被接受并结束。 state、EOS、结构闭合情况。 停止或继续。 会提前结束或无限追 token。

约束方法

methods

vLLM 和 SGLang 都把“结构化输出”做成公开能力，但支持的约束类型、后端名字和接入方式不完全一样。学习时不要先问“谁更高级”，先问“这条约束能不能落到当前 workload 上”。

约束类型 适合什么问题 不适合什么问题 典型工程读法
choice 枚举答案、有限标签、固定选项。 自由文本列表、深层嵌套结构。 把输出空间压到最小，先验证流程。
regex 固定格式字符串、带模式的字段、简单 JSON 片段。 复杂层级、状态很多的递归结构。 看 token 过滤和前缀状态是否稳定。
json schema 对象、字段、类型、必填项、数组结构。 需要开放式推理链但只约束最终答案的场景。 适合把 API 输出变成可验证合同。
grammar / ebnf 结构复杂、需要精确定义语法的位置。 短枚举场景，过重。 更像一台小语法机，不只是“格式化”。

公开文档里能看到 choice、regex、json、grammar、structural tag 等选项，也能看到 outlines、lm-format-enforcer、xgrammar 这类 guided decoding backend。你要先看文档支持哪种约束，再决定是否继续往源码里下钻。

SGLang 的公开路线

公开文档里能看到 JSON schema、regex、EBNF 这些约束参数，也能看到不同 grammar backend。它的重点不是单一格式，而是把约束放到 serving 和 reasoning 语境里一起看。

现象 先看什么 说明了什么 下一步怎么做
字段顺序错乱 schema / grammar 的顺序约束 约束对象和 JSON 结构没有对齐 先把字段顺序收紧，再看语义质量
输出能 parse 但内容空洞 prompt、温度、样例 decode 成功，不代表任务成功 补输入信息，不要放松结构约束

把约束落到一条请求上

case study

假设你要从一段产品反馈里抽取联系人信息，最后必须输出一个对象，字段只有 name、company、email。这类任务最容易暴露约束解码的真实位置：它不是在模型结束后补一个格式，而是在每一步采样前收窄 token 集合。

你可以把它想成一条小小的状态机。状态机一开始只知道“要输出对象”，还不知道字段顺序；进入第一个字段后，它才知道这一步该允许哪些字符。到字段值阶段，可选 token 会变成字符串内容、闭合引号、逗号或结束符。你看到的变化不是模型本体换了，而是采样前的 mask 在每一步被重新计算。

阶段 输入状态 状态变化 重点看什么 错了会怎样
起始 只有 prompt 和约束定义。 backend 把 schema / regex / grammar 转成可执行规则。 约束是否能被当前模型和 runtime 支持。 请求看起来合法，实际一开始就会回退。
对象头 还没进入字段。 允许的 token 主要围绕对象起始、字段名和分隔符。 前缀是否真的进入了结构空间。 模型会继续吐自然语言而不是结构化内容。
字段值 已经选定字段名。 合法 token 变成字段值本身和闭合符号。 字段类型、长度和格式限制是否过窄。 会在引号、逗号或邮箱格式附近卡住。
收束 对象接近完整。 状态机确认闭合并允许结束。 终止条件是否可达。 会提前停，也可能一直追 token。
一个最小请求配置
{
  "prompt": "Extract a contact card from the paragraph.",
  "constraint": {
    "type": "json_schema",
    "schema": {
      "type": "object",
      "required": ["name", "company", "email"],
      "properties": {
        "name": { "type": "string" },
        "company": { "type": "string" },
        "email": { "type": "string", "format": "email" }
      }
    }
  }
}

失败后果

如果 schema 过窄，模型可能在字段名、引号或结束符附近被卡住；如果 backend 不支持当前模型或 dtype，系统可能回退到别的实现；如果你只做后验校验，token 已经花掉了，错误还是会回到用户手里。

怎么把问题拆开

先分清是“输出无法解析”，还是“输出可解析但语义不对”。前者多半是约束和 backend 的问题，后者才更多回到 prompt、模型和数据分布。

练习区的第 5、6、8 题都是在追这条链：字段、前缀、终止条件、fallback 和状态推进。把它们串起来，你就能从一条错误日志直接回到该看哪一层。

backend、状态机和回退

backend trace

同样是“约束解码”，不同 backend 的工作方式不完全一样。你不用把每个实现都背下来，但要知道它们在解决什么问题：有的偏向把结构约束翻成可执行的 token 过滤，有的偏向把语法规则转成状态机，有的偏向和现成的约束库对接。工程上最重要的不是名字，而是你能不能从日志里分辨出“这次是约束对象不对、状态推进不对，还是 backend 不支持当前形状”。

一个好用的读法，是把每个 backend 都当成三层：输入约束、内部状态、输出 mask。输入约束决定它接收的是 JSON schema、regex 还是 grammar；内部状态决定它记住的是字段位置、语法栈还是前缀状态；输出 mask 决定下一步能选什么 token。三个层一旦对不上，问题就会表现成不同症状。

读法 输入约束 内部状态 输出 mask 常见症状
schema 对象、字段、类型、数组和必填项。 当前位置在第几个字段、是否已经进入值区。 只允许当前字段需要的 token。 字段闭合、额外文本、类型错误。
regex 模式、分支、重复和终止条件。 当前前缀是否还在合法路径上。 把不匹配前缀的 token 过滤掉。 在某个前缀附近反复兜圈。
grammar 语法规则、非终结符、终止符。 解析栈、展开顺序、可归约状态。 只暴露能让语法继续前进的 token。 括号不闭合、语句卡住、结束不了。
choice 有限枚举和固定标签。 只要记住当前是否已经选定。 只放行极少数合法 token。 最稳，但表达空间很窄。

为什么会回退

回退通常不是“模型答错了”，而是 backend 发现当前模型、dtype、shape 或约束形状不在支持范围里。系统为了能继续服务，会换一条较保守的路。这条路能跑，但不一定保留原来那条路径的速度和约束能力。

看日志时的顺序

先确认约束类型，再确认 backend 名字，再确认是否 fallback，最后看输出是否在采样前就已经被 mask 改掉。顺序如果倒过来，很容易把后验错误当成前置错误。

故障分流记法
1. 解析失败：先看 schema / grammar 规则是否写错。
2. 语法卡住：先看状态是否还能推进到下一个合法前缀。
3. backend 回退：先看模型、dtype、shape 和支持矩阵。
4. 结果合法但不对：再回到 prompt、数据分布和任务定义。

这就是本单元真正要你记住的动作：不是看到“格式问题”就一股脑改 prompt，而是先判断问题究竟出在约束定义、状态推进、backend 兼容，还是输出内容本身。

三类失败怎么分流

triage

实际排障时，结构化输出最常见的三种失败看起来都像“格式坏了”。第一种是输出根本不能解析，第二种是输出卡在某个前缀上不动，第三种是系统为了兼容性偷偷回退了 backend。它们表面都表现为“结果不对”，但处理顺序完全不同。

先看最小分流法：如果 parse 失败，优先检查 schema、闭合符和额外文本；如果卡在前缀，优先检查 regex 或 grammar 是否把合法分支收得太窄；如果速度突然掉了，优先查 fallback 和支持矩阵。这个顺序能把大多数无效回滚挡在前面。

失败类型 最早看到的症状 最可能的根因 先做什么 后做什么
解析失败 输出像 JSON，但解析器报错。 schema 写错、额外文本、字段类型不稳。 检查 schema 和输出片段是否一一对应。 再看 prompt 和数据分布。
前缀卡住 模型在某个前缀上反复重复。 regex / grammar 太窄，终止条件太晚。 缩小规则，验证合法前缀能不能推进。 再看状态机有没有被错误推进。
fallback 日志里出现回退，速度明显变慢。 模型、dtype、shape 或长度不在支持矩阵里。 确认 backend 是否支持当前配置。 再比较回退后性能损失有多大。
语义不对 结构合法，但内容不符合任务。 任务定义、prompt 或数据分布不稳。 先保留结构，再改任务输入和数据。 最后才考虑重设约束形状。
先后顺序小抄
1. 先判断 parse 失败、前缀卡住、还是 backend 回退。
2. 解析失败先查 schema、字段和额外文本。
3. 前缀卡住先查 regex / grammar / state。
4. 回退先查模型、dtype、shape 和支持矩阵。
5. 语义不对才回到 prompt、任务和数据分布。

失败后果

如果你跳过分流，排障就会变成“哪里都改一点”。那样最常见的结果是，结构问题没解决，prompt 变复杂了，backend 也没确认，最后连日志都读不懂了。

这段和后面练习的关系

练习区第 5、6、8 题对应的就是这四类问题。你答题时如果能先分流，再说下一步，就说明已经把约束解码当成一条可追踪的工程路径，而不是一个格式标签。

一个小例子

example

假设你要抽取一条联系人记录，要求输出 ``, ``, `` 三个字段，而且 `` 必须是合法邮箱。没有约束时，模型可能写成自然语言；加上 JSON schema 后，模型必须在字段、引号和类型上收束；如果再加 regex 或 grammar，连局部形式都要跟着变窄。

对象、字段和数组位置都依赖这一步。

第一步

先让 schema 说清结构，不要先急着优化 prompt 文案。

第二步

再确认 backend 能否把 schema 变成 token 级规则，而不是只做后验解析。

第三步

最后再看输出质量。合法结构只是门槛，不代表内容已经准确。

为什么这个例子有用

它把“约束对象”“结构对象”“输出对象”分开了。只要你能把这三层拆开，后面遇到 tool call、枚举、JSON、表格，判断就会快很多。

换句话说，结构化生成不是把文本变漂亮，而是把允许的路径收窄。你在这里看见的是一个请求如何被 schema 牵住；你在后面的引擎单元里会继续看见，这根绳子会落到哪个 backend、哪段状态机、哪条日志。

快速复盘：先问“约束对象是什么”，再问“谁在改 token 集合”。
诊断JSON

5. 案例：输出看起来像 JSON，但解析总是失败。你会先查哪三件事？

提示：从 schema 完整性、额外文本、字段类型三个方向开刀。

参考答案 / 评分要点

先看 schema 是否把必填字段、类型和嵌套写清；再看输出里有没有额外自然语言；最后看字段值是不是被模型写成了不合法类型。

诊断regex

6. 案例：regex 约束下模型一直兜圈，不肯停。你怎么缩小范围？

提示：先怀疑规则太窄，再怀疑终止条件太晚，最后再看 backend 是否回退。

参考答案 / 评分要点

先把 regex 缩到一个更小的子集，验证前缀能否推进；再确认终止条件是否真的可达；如果还是卡住，再看 backend 是否把约束状态搞乱了。

场景reasoning

7. 案例：你要的是最终答案，结果内容跑到 reasoning_content。你会怎么判断这个问题归谁？

提示：这不是单纯的格式错误，要先看 reasoning 段和 answer 段是否被分开处理。

参考答案 / 评分要点

先确认模型是否把 reasoning 和 final answer 分开输出；如果是，就要看约束到底应该应用在答案段，还是在 reasoning 结束后再进入结构化输出。

诊断backend

8. 案例：结构化输出忽然变慢，日志里还出现 fallback。下一步你怎么排？

提示：模型、dtype、shape、上下文长度和 backend 支持范围都要看。

参考答案 / 评分要点

先确认 backend 是否支持当前模型和 dtype，再看 shape / head size / context length 是否触发回退，最后再比对慢的是 constraint 生成还是原始 decode。

三组：迁移与表达

把概念讲成工程判断。

快速复盘：能不能把“约束对象、执行位置、故障现象”讲成一句话。
读代码decode

9. 读这段伪代码，说出它在做什么。

伪代码片段
allowed = backend.valid_tokens(state, logits)
masked_logits = apply_mask(logits, allowed)
token = sample(masked_logits, temperature, top_p)
state = backend.advance(state, token)

提示：必须提到“合法 token 集合”和“状态推进”两个点。

参考答案 / 评分要点

这段代码先计算当前状态下哪些 token 允许，再把非法 token 的 logits 压掉，随后在剩下的 token 里采样，并把状态推进到下一步。

解释概念

10. 用一句话解释 guided decoding。

提示：一句话里要同时说到“生成过程中”与“限制 token”。

参考答案 / 评分要点

guided decoding 是在生成过程中用规则约束 token 选择，让输出沿着指定结构前进，而不是生成完后再补格式。

设计实验

11. 如果你要比较 schema、regex、grammar 三种约束，你会怎么做实验？

提示：至少要同时记录合法率、延迟、回退和输出质量。

参考答案 / 评分要点

至少设 baseline、schema、regex、grammar 四组，同时记录合法率、TTFT、TPOT、回退次数和内容质量。这样才能看出约束收益是不是被性能开销吃掉。
