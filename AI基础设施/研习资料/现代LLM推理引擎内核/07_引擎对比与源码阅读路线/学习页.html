概念骨架

map
请求入口 先看外部请求长什么样，再看内部请求对象怎么变化。

这一步决定你从哪里开始追踪。

术语与边界

terms
对象 直觉解释 精确定义 容易混淆的地方
vLLM 一个高性能 LLM serving / inference 引擎。 围绕请求调度、KV cache、后端和服务接口组织的推理系统。 不要把它等同于某一个 kernel 或某一个 feature。
SGLang 一个面向高性能 serving 和编排的框架。 把运行时、结构化输出、前缀复用和多模型场景组织在一起。 不要把它只看成“更会写 prompt 的工具”。
PagedAttention KV cache 的块式管理技术。 把分页思想引入 KV cache，减少碎片并提升共享和复用能力。 它不是完整 serving 框架。
benchmark 拿来比较速度和容量的数据。 必须和模型、硬件、输入输出长度、并发和配置一起解释。 单张图通常不能直接推出生产结论。

别拿什么直接比

不要拿单张 benchmark 图直接裁决生产选型。输入长度、输出长度、并发数、是否流式、是否启用 prefix cache、是否量化，这些上下文缺一项，结论都可能变。

比较的正确姿势

先看这个项目解决的层级，再看它默认打开了什么能力，最后才看它在你那类 workload 上的可复现表现。

源码阅读路线

route

阅读顺序尽量按“先主循环、再资源、再特性”走。你不需要一次读完整个项目，只要先把一条请求路径追通，就已经能继续往下拆。

最小搜索命令
rg -n "scheduler|engine|kv cache|cache manager|structured outputs|metrics" .
rg -n "request|prefill|decode|running|waiting|preempt" .
rg -n "backend|logits|sampling|grammar|json_schema|regex" .
步骤 先看什么 为什么这样看 常见信号
1 README / docs / quickstart 先确认这个项目把自己定位成什么。 服务入口、支持模型、feature 列表。
2 API handler / request object 把外部请求和内部状态对象对应起来。 request id、sampling params、prompt / output tokens。
3 scheduler / engine step 看本轮执行谁、给多少 token、是否抢占。 waiting / running、budget、preemption。
4 KV cache / prefix cache / block manager 把显存账本和请求状态对齐。 block、reuse、evict、fragmentation。
5 model runner / backend / logits 确认 token 如何从调度结果变成一次执行。 backend selection、fallback、sampling、structured outputs。
6 metrics / tracing / benchmark 把源码里的统计项映射到工程指标。 TTFT、TPOT、ITL、throughput、cache usage。

先别追的东西

现在先别钻 kernel 里的每个实现细节，也别把 gateway、路由器、分布式 RPC 和评测平台混进来。等你把这条路径跑通，再回头看这些边界会更稳。

把一条请求从头走到尾

walkthrough

如果你只想记一个阅读故事，就记这一条：请求从 API 进来，先变成内部 request object，再进入 scheduler；scheduler 决定它这一轮要不要执行、执行多少 token；cache manager 给它分配或复用 KV 块；model runner 把这一轮的 token 计算出来；最后 metrics 把 TTFT、TPOT、ITL 和 cache usage 记下来。

这条链的每一站都能变成源码里的关键词。你不是在“找某个文件”，而是在找角色：谁创建 request，谁修改状态，谁决定本轮预算，谁记录性能，谁负责 fallback。只要角色找对，文件名怎么变都不会把你带偏。

路径段 角色 主要读什么 主要写什么 出错时的症状
API / handler 把外部请求变成内部对象。 prompt、sampling params、stream 标志。 request id、内部上下文、初始状态。 请求看起来进来了，但内部字段缺失。
scheduler 决定本轮跑谁。 waiting / running 队列、token budget、抢占条件。 本轮执行集合、预算分配、预抢占状态。 吞吐、延迟或公平性开始明显异常。
cache manager 给请求分配和回收 KV 资源。 block table、prefix 命中、剩余显存。 块分配、复用、释放、碎片状态。 高并发时 OOM，或者复用收益看不见。
model runner / backend 把调度结果变成一次执行。 隐藏状态、约束、后端兼容矩阵。 logits、输出 token、fallback 记录。 能跑但慢，或者直接回退到备选路径。
metrics / tracing 把运行现象留成可复盘证据。 完成时间、队列时间、token 速率、cache 使用。 TTFT、TPOT、ITL、throughput、preemption 次数。 你只能感觉“慢了”，说不清慢在哪一层。

一个有用的判断

如果问题主要落在 queue time 和 token budget，上游多半是 scheduler；如果问题主要落在 cache usage 和高并发 OOM，先看 cache manager；如果问题主要落在 fallback 和支持范围，先看 backend。

为什么这段值得先读

它把“调度”“缓存”“执行”和“观测”串成了一条链。读源码时只要能把这条链映射到文件和函数，你就不会被目录结构绕住。

失败后果

如果你跳过这条主线，后面读到的会是“这个项目支持了什么”“那个项目又多了什么能力”，但你仍然不知道问题从哪一层开始出现。比较最后会退化成印象分，源码阅读也会变成四处翻目录。

按项目各读一条线

project read

真正开始读源码时，可以把 vLLM、SGLang 和 PagedAttention 分成三种读法。vLLM 更适合从 request / scheduler / cache manager / runner 的闭环去看；SGLang 更适合从 API、runtime、structured outputs 和 prefix cache 的组合去看；PagedAttention 更适合从论文的问题设定、块表和映射关系去看。

项目 先读哪条线 先找什么关键词 要得到什么图 常见误区
vLLM API → engine → scheduler → cache manager → runner。 scheduler、kv cache、structured outputs、metrics。 一条请求如何进、如何排、如何跑、如何退。 只盯特性列表，不看主循环。
SGLang API → runtime → prefix cache → structured outputs → executor。 structured outputs、prefix cache、benchmark、profiling。 前缀复用和结构化输出如何嵌到 serving 里。 只把它当成 prompt DSL。
PagedAttention 问题设定 → block table → 分配 → 映射 → 共享。 block、sharing、fragmentation、copy-on-write。 逻辑 token 怎样映射到物理块。 拿论文结论直接外推到所有场景。

为什么要分三条线

因为这三个对象的层级不一样。vLLM 和 SGLang 是完整系统，PagedAttention 是其中一类关键技术。读法不同，关注点也不同：系统看闭环，技术看映射，论文看问题定义。

看完你该拿到什么图

你至少应该能画出一条请求路径、一个缓存账本、一个 backend 回退路径，以及一个 benchmark 的条件列表。少了任何一个，选型和复盘都还差一层。

怎么读 benchmark

benchmark

一张 benchmark 图最容易骗人的地方，不在于它是不是漂亮，而在于它把上下文藏掉了。看似相同的 tokens/s，背后可能是不同模型、不同长度分布、不同并发、不同流式配置、不同 prefix cache 命中率，甚至不同 backend。你若不先问条件，就会把偶然结果当成稳定规律。

所以 benchmark 先读条件，再读结果。条件里至少要有模型、硬件、输入长度、输出长度、并发数、是否流式、是否量化、是否启用 prefix cache、是否启用 speculative decoding，以及统计的是平均值还是分位数。没有这组信息，图只能当线索，不能当结论。

必须补的条件 为什么要看 如果漏了会怎样
模型和版本 不同模型的结构和上下文行为不一样。 你不知道是在比项目还是在比模型。
硬件和 dtype backend 和 kernel 对架构、精度很敏感。 性能差异可能只是硬件差异。
输入 / 输出长度 prefill、decode、cache 压力会换位置。 TTFT 快不代表 TPOT 也快。
并发和流式 调度行为、队列和尾延迟都会变。 你看到的可能只是单请求的理想状态。
cache / 量化 / speculative 这些开关会同时改容量和速度。 很难知道收益来自哪一层。
最小对照实验
baseline:
  model: same
  hardware: same
  input_length: same
  output_length: same
  concurrency: same

compare:
  vllm:
    prefix_cache: on/off
    structured_outputs: on/off
  sglang:
    prefix_cache: on/off
    structured_outputs: on/off

失败后果

如果 benchmark 条件没补齐，最后很容易得出“这个项目更快”这种空结论。真正的工程问题会被遮住，后面换工作load 时又会重新撞墙。

选型复盘

review

假设你手里有三份材料：vLLM 的 benchmark 图、SGLang 的功能说明、PagedAttention 的论文。它们都在讲推理，但它们说的是三种不同层级的事。最容易出错的地方，不是信息太少，而是你把这三种层级混成了同一个问题，然后用同一个结论去回答。

复盘时可以直接把材料分成三句。第一句说系统层，哪个项目更适合你的请求路径和运维方式。第二句说能力层，哪个项目更适合结构化输出、前缀复用和路由需求。第三句说技术层，哪个机制只是在局部把容量、碎片或访问路径做得更好。这样写，讨论会自然收口。

层级 该问什么 该看什么材料 常见误判
系统层 它能不能接住你的线上请求和运维方式？ API、runtime、scheduler、metrics、README。 只看 feature 列表，不看闭环。
能力层 它对结构化输出、前缀复用和流式请求支持得怎样？ structured outputs、prefix cache、benchmark、示例。 把能力层直接当成最终性能。
技术层 哪一个机制真正解决了局部瓶颈？ PagedAttention、cache layout、block table、论文图。 把局部技术当成完整系统结论。
复盘模板
系统层：____ 更适合我的请求路径和运维方式，因为 ____。
能力层：____ 更适合我的结构化输出 / 前缀复用需求，因为 ____。
技术层：____ 说明了容量或碎片上的关键机制，但它不等于完整系统。

怎么用这段

你可以拿任何一组 benchmark 或 README 来填这个模板。填完以后，很多“谁更强”的争论就会自然变成“哪一层更适合我”。这比直接下结论更接近工程决策。

失败后果

如果不分层，评审里最常出现的结果就是：把技术点当成系统点，把局部结果当成全局结论，最后选型和源码阅读都偏航。

这一段和前面的 benchmark 读法是同一件事的两面：前者教你补条件，后者教你给条件分层。做完这一步，你就能把这门课收束成一个可复用的阅读和选型方法。

故障实验室

terminal lab

看引擎时最容易犯的错，是把“看起来像一个问题”的几类症状混成一类。下面几个场景先把分层感练出来。

一个小例子

example

假设你要做一个服务：重复的 system prompt 很多，还要求输出稳定的 JSON，而且有时会混进工具调用。你不能先问“哪个项目最强”，你要先拆成三件事：请求入口是否兼容、约束输出是否能在 decode 里执行、缓存和调度是否会因为额外规则变慢。

看入口

先确认你的请求是 OpenAI 兼容、native API 还是别的接口。

看约束

再确认结构化输出是 schema、regex 还是 grammar，是否需要 tool parser。

看运行时

最后再看 prefix cache、scheduler、backend 和 metrics，确认收益有没有被开销抵消。

最小判断顺序
1. 先确认问题属于哪一层：请求、调度、缓存、backend、还是观测。
2. 再确认工作负载：长 prompt、短 prompt、重复前缀、长输出、结构化输出。
3. 然后看约束和缓存会不会互相拖慢。
4. 最后才把 benchmark 结论拿来支持选型。
问题线索 先查的对象 想得到的答案 不该立刻做什么
重复前缀很多 prefix cache / radix cache 是否真的在复用前缀，是否命中稳定 先别把慢直接归给 model runner
JSON 输出不稳定 structured outputs / schema / backend 是否在 decode 时被约束 先别只改 prompt 文案
看起来同样快 benchmark 条件 是否真的是同一 workload 先别把一张图当结论
偶发 fallback 兼容矩阵 / dtype / shape 目标 backend 是否被限制 先别直接换项目

常见错误

mistakes

错误 1：从 kernel 开始读

kernel 只能解释单次计算怎么更快，不解释请求为什么被选中、cache 为什么够用、输出为什么停。修正：先读运行时主循环。

错误 2：把 benchmark 当结论

没有模型、硬件、长度分布和并发上下文的 benchmark，不足以支持选型。修正：先补条件，再看数值。

错误 3：把 PagedAttention 当完整引擎

它解决的是 KV cache 管理的重要问题，但不自动解决服务治理、路由、限流或可观测性。修正：把它放回引擎的一部分看。

错误 4：把“能跑”误认为“走了目标实现”

后端回退很常见，能跑不代表走的是你以为的那条路径。修正：先看兼容矩阵，再谈速度。

失败后果

如果你从 kernel 切入，通常会得到一堆局部优化名词，却不知道请求是怎么进来的、资源是怎么分的、为什么会回退。最后你会能讲单个实现，却不能解释一个线上现象。

如果你想把这页变成真的工程判断工具，最少要同时回答三件事：请求路径在哪一层、缓存账本有没有复用、benchmark 条件有没有对齐。只会其中一件，选型还是会飘。

综合路径

7. 把一次请求从入口到释放资源的路径说出来。

提示：写成“入口 -> 调度 -> 缓存 -> 执行 -> 输出 -> 释放”的链。

参考答案 / 评分要点

一条可接受的路径是：请求进入 API，创建内部 request object，进入 scheduler，分配 KV cache 或 prefix cache，进入 model runner / backend，生成并采样输出，最后判断停止并释放资源。

诊断日志

8. 如果只看日志，你会优先看哪四类信号？

提示：尽量把症状和层级对应起来。

参考答案 / 评分要点

优先看 TTFT、TPOT / ITL、backend 回退、KV cache 占用或 preemption 计数。这样能快速定位到是入口、调度、缓存还是执行层出了问题。

设计选型

9. 如果业务重前缀复用和结构化输出，你会先比较什么？

提示：从功能、后端、缓存、指标四个方向说。

参考答案 / 评分要点

先比较结构化输出支持、prefix cache 或 radix cache 机制、backend 兼容范围和实际 workload 下的 TTFT / TPOT / 命中率。只有这些都对齐了，选型才有意义。

10. 读这段路线代码，说出它提醒你先看什么。

路线代码片段
rg -n "scheduler|engine|kv cache" .
rg -n "structured outputs|json_schema|regex|grammar" .
rg -n "TTFT|TPOT|ITL|throughput|cache usage" .

提示：把检索顺序和阅读顺序联系起来。

参考答案 / 评分要点

这段命令提醒你先找调度、缓存和执行路径，再看结构化输出，最后把这些路径映射到 TTFT、TPOT 和 cache usage。它不是在搜目录，是在搜阅读入口。

解释benchmark

11. 用一句话解释为什么不能只拿一张 benchmark 图做选型。

提示：一定要点出上下文缺失和 workload 差异。

参考答案 / 评分要点

因为 benchmark 脱离模型、硬件、长度分布和并发上下文就没有可比性，单张图只能说明某个测试场景，不足以直接代表生产选型。
