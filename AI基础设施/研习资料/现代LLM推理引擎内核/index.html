课程地图

现代 LLM 推理引擎内核

7 个单元 AI 基础设施 vLLM PagedAttention SGLang TensorRT-LLM

推理引擎的难点不是“调用模型”，而是在不断变化的请求、上下文长度和显存压力下，让 GPU 以可控方式持续产出 token。

建议起点：先把 prefill/decode 和 KV cache 讲清楚，再读 PagedAttention、batching 和结构化生成。

返回公开学习首页

课程总论

这门课把 LLM 推理引擎看成一个实时操作系统：请求进入、显存分配、算子执行、token 返回，都要排队、复用和回收。

Overview
建议学习对象

想读 vLLM、SGLang、TensorRT-LLM 或自建推理服务的人，尤其是关心吞吐、延迟、显存和长上下文的人。

前置知识

需要理解 Transformer 解码、attention、GPU 基础性能语言、Python 服务基础；最好先学 GPU 性能工程。

学完以后

应能解释一个引擎为什么要做 KV cache 管理、continuous batching、prefix cache、约束解码和 kernel 优化。

基于哪些资料

资料以公开文档、论文和开源项目为主；课程地图只安排阅读路线，不代替源码逐行讲解。

推荐路径

先看请求如何变成一串 kernel，再看引擎如何在多请求之间分配时间和显存。

Route
01-02｜单请求账本

拆解 prefill、decode、logits、sampling、KV cache，知道一次生成请求为什么分成两个阶段。

先建立 token 生成的时间线。
03-04｜多请求引擎

用 PagedAttention 和 continuous batching 理解显存块、队列、公平性和吞吐之间的关系。

从单请求进入运行时。
05-06｜内核与约束

把 FlashAttention、量化、speculative decoding、结构化生成放到引擎循环里看。

区分 kernel 优化和语义约束。
07｜源码和系统对比

用 vLLM、SGLang、TensorRT-LLM 对照阅读，建立选择引擎的判断标准。

收束为工程选型能力。

KV cache 与显存账本

存在理由：KV cache 是长上下文和高并发的主要显存压力来源；不理解它，就无法理解后续任何调度和块管理。

PagedAttention 与块管理

存在理由：PagedAttention 把操作系统分页思想引入 KV cache；本单元解释为什么块式管理能减少碎片并提高吞吐。

04

Continuous batching 与调度循环

存在理由：在线推理不是静态 batch；本单元解释请求到达、token 级调度、抢占、队列和吞吐延迟取舍。

推理算子、FlashAttention 与量化

存在理由：引擎吞吐最终落到 kernel、backend、精度和数据布局；本单元把算子层优化放回 prefill/decode 的上下文。
