这一块先练判断：多智能体不是默认架构，只有当单智能体遇到明确约束时才值得引入。
Decision framework

先单智能体，再决定要不要拆成多智能体

这节课来自 Claude 官方文章《Building multi-agent systems: When and how to use them》。文章的态度很直白：大多数团队不需要多智能体。先把单智能体做好，再用证据判断是否需要拆分。

单智能体基线→ 发现真实瓶颈→ 匹配三类约束→ 估算协调成本→ 再决定拆分

学完本页，你应该能看着一个 Agent 项目说出：现在的问题是上下文、并行搜索、专业化，还是单纯的提示词和工具设计没做好。你也应该能拒绝“看起来更高级”的拆分方案。

本块学完要能做

  1. 解释为什么单智能体应作为第一版。
  2. 列出多智能体常见成本：上下文复制、协调消息、交接摘要、维护提示词。
  3. 判断一个任务是否落入三类适用场景：上下文保护、并行化、专业化。
  4. 用证据而不是感觉决定是否拆分。

1. 为什么先从单智能体开始

baseline first

单智能体不是低级版本。一个设计得当的单智能体，配上合适的工具、清楚的系统提示词和稳定的上下文管理，能完成的事情通常比直觉里多。先做单智能体，等于先建立一个可比较的基线。没有基线，就很难判断后面引入多智能体到底改善了什么。

多智能体会增加新的失败点。多一个智能体，就多一份提示词要维护，多一套工具权限要检查，多一次交接可能丢信息。很多团队把“规划、执行、测试、评审”拆给不同智能体，结果不是更稳，而是每次交接都要解释背景，最后协调花掉的 token 比真正做事更多。

单智能体基线要先回答

  • 它是否已经有完成任务所需的工具？
  • 系统提示词是否说明了目标、约束和停止条件？
  • 上下文是否被无关材料污染？
  • 失败是能力不足，还是任务说明不清？

判断句

如果单智能体还没有稳定基线，就先不要拆。拆分会把原来的问题分散到多个智能体里，排查会更难。

你现在应该能做什么：遇到一个“要不要多智能体”的需求时，先问单智能体版本跑得怎样，而不是先画多个角色框。

2. 多智能体的成本从哪里来

3-10x tokens

Claude 原文给出一个很实用的观察：在等价任务上，多智能体实现通常会用掉单智能体的 3 到 10 倍 token。这个数字不是数学定律，但足以提醒我们：多智能体不是免费并行。

成本主要来自三处。第一是上下文复制。每个子智能体都需要自己的任务说明、工具说明和必要背景。第二是协调消息。主智能体要分派任务，子智能体要汇报结果，必要时还要追问。第三是交接摘要。一个智能体做完后，另一个智能体无法直接继承完整思考过程，只能拿到被压缩过的结果。

成本来源具体表现坏结果缓解方式
上下文复制每个子智能体都加载同一批背景。token 增长快，模型注意力分散。只给子智能体完成任务所需的最小背景。
协调消息分派、汇报、追问、合并都要占上下文。协调比执行更贵。拆分前确认子任务真的独立。
交接摘要后续智能体只看到压缩后的结果。动机、边界条件、失败尝试丢失。能不交接就不交接，相关工作放同一智能体里。
提示词维护多个角色、多个工具权限、多个输出格式。改一个流程要同步多处提示词。保持角色少，接口清楚，测试覆盖关键路径。
正确示例：先跑单智能体基线pseudo
task = "研究某个市场趋势，并给出带来源的结论"

single_agent_result = run_agent(
    task=task,
    tools=["web_search", "read_document", "summarize"],
    budget_tokens=baseline_budget
)

measure(single_agent_result, metrics=[
    "答案覆盖度",
    "引用准确性",
    "总 token",
    "耗时",
    "失败原因"
])

这段伪代码的重点不是函数名，而是顺序。先让一个智能体在明确工具和预算下完成任务，再记录质量、成本和失败原因。只有当失败原因指向真实瓶颈，才进入拆分讨论。

3. 值得多智能体的三类约束

not style, constraint

多智能体有价值，不是因为它听起来像一个团队，而是因为它解决了单智能体越不过去的约束。Claude 原文反复强调：只有当收益足以抵消额外成本时，才应该使用多智能体。第一块先掌握三类判断，后面几块会分别展开。

上下文保护

当一个子任务会产生大量无关材料，继续塞进同一个上下文会污染后续推理时，可以用子智能体隔离它。

并行化

当任务天然能拆成多个独立搜索方向时，多个智能体可以同时探索更大的信息空间。

专业化

当不同子任务需要完全不同的工具、提示词或领域知识时，专用智能体可能比一个通用智能体更稳。

这三类约束要和“我想要更清楚的分工”区分开。人类团队按角色分工很自然，但智能体系统里的角色分工常常带来上下文丢失。测试、实现、评审如果都围绕同一个功能，反而可能应该留在同一个上下文里完成。

易混概念准确含义判断方式
单智能体 vs 多智能体不是低级和高级，而是单上下文执行和多上下文协作。看是否有单上下文无法处理的约束。
工具多 vs 需要专业化工具多可能先用工具搜索或按需加载，不一定拆智能体。看工具是否属于清晰分离的领域。
并行任务 vs 顺序阶段并行任务互不依赖，顺序阶段共享大量背景。看子任务是否需要频繁同步理解。
上下文压力 vs 上下文污染压力是太长，污染是混入当前推理不需要的信息。看压缩、清理、隔离哪种方法更合适。

4. 单智能体真的撑不住时，会出现什么信号

outgrowing

判断是否拆分，最好看可观察信号。第一类信号是上下文接近上限，或者还没到硬上限，质量已经随着上下文增长而下降。这里不要急着拆，先检查能不能压缩历史、清理无关工具结果，或者让工具按需返回更短信息。

第二类信号是工具太多。原文提到，当一个智能体有 15 到 20 个以上工具时，模型会花大量上下文和注意力理解选项。但这不自动等于要多智能体。更轻的办法是工具搜索：让模型按需发现工具，而不是一次加载全部定义。

第三类信号是子任务天然可并行，比如多来源研究、多个独立组件测试、多个候选方案探索。这里多智能体可能带来覆盖度收益。注意，收益主要是更全面，不一定是更快。并行智能体总计算量更大，最终合成也要时间。

上下文变差

输出开始忽略早期约束，或者被上一阶段无关材料带偏。

工具选择困难

工具列表太长，模型频繁选错工具或反复解释工具。

搜索空间太大

一个智能体无法覆盖足够多角度，结论单薄。

判断句

如果失败原因能通过更好的工具定义、上下文压缩、检索策略或评估标准解决，就先修这些。多智能体应该解决架构瓶颈，不该替代基本工程整理。

6. 常见误区、后果和修法

diagnosis
误区错因后果修法
把多智能体当默认架构把角色数量误认为能力提升。成本增加，问题更难定位。先跑单智能体基线，再看瓶颈。
按人类岗位拆分忽略智能体交接会丢上下文。规划、实现、测试之间反复解释背景。按上下文边界拆，不按职位名拆。
工具多就拆智能体没有先尝试按需工具发现。多出路由和提示词维护成本。先精简工具、分组工具或引入工具搜索。
只看并行，不看合成以为同时跑就一定更快。结果合并变慢，冲突结论难处理。提前定义合成格式和冲突处理规则。
没有成本指标只凭主观感觉评价效果。无法证明拆分是否值得。记录 token、耗时、错误率、覆盖度和维护成本。
错误示例：为了像团队而拆分anti-pattern
agents = [
    PlannerAgent(),
    CoderAgent(),
    TesterAgent(),
    ReviewerAgent()
]

# 所有智能体都在处理同一个功能，但每一步都要重新解释背景。
# 测试者不知道实现时做过哪些权衡，评审者看不到探索过程。
run_handoff_chain(agents, task="实现同一个功能")

这段错误示例的问题不是“有四个智能体”，而是拆分边界和上下文边界不一致。它把强相关的工作切开，每次交接都会压缩信息，最后容易形成传话游戏。

7. 例题讲解：什么时候该拒绝拆分

worked examples

例题 A：客服诊断

一个客服 Agent 要查订单历史，还要诊断技术故障。订单历史每次会返回几千 token，后续诊断并不需要完整订单细节。

判断：这可能适合上下文保护。查订单可以交给子智能体，只返回和诊断相关的摘要，避免把长订单历史塞进主上下文。

例题 B：实现同一个小功能

团队想让一个智能体规划、一个智能体编码、一个智能体写测试、一个智能体评审。功能很小，所有阶段共享同一批设计背景。

判断：先不要拆。这里更像顺序阶段共享上下文，拆分会增加交接损耗。让同一个智能体完成实现和测试，再用明确检查清单验收更合适。

正确示例：把拆分写成可验证假设pseudo
hypothesis = "订单查询污染了技术诊断上下文"

baseline = run_single_agent(task)
candidate = run_agent_with_order_lookup_subagent(task)

compare(baseline, candidate, metrics=[
    "诊断准确率",
    "无关订单 token 数",
    "总 token",
    "平均耗时",
    "失败案例"
])

好的拆分会对应一个可验证假设。你不是因为“多智能体更先进”才拆，而是因为你怀疑某个约束正在拖累质量，并准备拿指标检查这个假设。
