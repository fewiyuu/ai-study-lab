上下文保护：别让无关材料拖垮推理

多智能体最常见的有效理由之一，是保护上下文。模型能看见的上下文有限，质量也会随着无关信息累积而下降。子智能体的价值不是“多一个人”，而是给某个子任务一个干净的工作区。

任务产生材料→ 判断是否污染→ 隔离子任务→ 返回最小摘要→ 主上下文继续推理

学完本页，你应该能分清“上下文太长”和“上下文污染”，能判断一个拆分边界是否真的减少信息损耗，也能解释为什么按规划、编码、测试这种岗位式拆分经常适得其反。

本块学完要能做

  1. 解释上下文窗口、上下文压力和上下文污染。
  2. 用客服订单例子说明子智能体如何隔离噪声。
  3. 区分问题导向拆分和上下文导向拆分。
  4. 判断哪些边界适合拆，哪些边界应该留在同一个智能体里。

1. 上下文不是“聊天记录”这么简单

working memory

在 Agent 系统里，上下文包括用户任务、系统提示词、工具说明、历史消息、工具返回结果、中间计划和已经做过的尝试。模型下一步怎么判断，基本都依赖它当前看见的这些材料。

上下文窗口是模型一次能处理的材料上限。但实践中，不是只有“超出窗口”才出问题。即使还没到硬上限，如果上下文里混入大量当前推理不需要的材料，模型也会更容易忽略关键约束，或者把上一阶段的信息带到下一阶段。

上下文窗口

模型一次能看到的 token 范围。它决定能装多少材料。

上下文压力

材料越来越多，注意力分配变难，输出质量开始下降。

上下文污染

上下文里有大量当前任务不需要的信息，干扰模型判断。

你现在应该能做什么：看到一个 Agent 变笨时，不只问“是不是上下文满了”，还要问“上下文里是不是塞了太多和当前判断无关的东西”。

2. 客服订单例子：污染是怎么发生的

support case

原文里的例子很典型：客服 Agent 要处理用户说“我的订单不能用了”。它可能先查订单历史，再诊断技术问题。订单查询会返回购买记录、物流、地址、优惠、历史沟通等大量细节。可是真正诊断“为什么不能用”时，主上下文可能只需要订单状态和产品型号。

如果单智能体把完整订单历史都塞进同一条对话，后续技术诊断就要带着几千 token 的无关内容继续推理。模型可能开始引用不相关的物流信息，或者忽略技术故障里最关键的错误码。这就是上下文污染。

错误示例：单智能体把所有订单细节都带到诊断阶段pseudo
conversation = [
    user("我的订单 #12345 不能用了"),
    assistant("我先查一下订单"),
    tool_result("订单历史、购买记录、物流、地址、优惠、客服沟通……几千 token"),
    assistant("现在开始诊断技术问题")
]

# 技术诊断时，模型仍然背着大量无关订单细节。

判断句

如果某个工具返回的大部分内容只对一个短暂子任务有用，后续主推理不需要它，就要考虑隔离或摘要，不要让它长期留在主上下文里。

3. 上下文保护模式：子智能体只返回必要摘要

clean context

上下文保护的做法是：让子智能体在自己的干净上下文里完成高噪声子任务，然后只把主任务需要的结论返回给主智能体。子智能体可以看完整订单历史，主智能体只收到“订单有效、产品型号 A、最近一次激活失败、错误码 E42”这类压缩结果。

这里的关键不是“摘要越短越好”，而是“摘要要保留后续推理需要的字段”。如果摘要丢了错误码、产品型号或时间线，主智能体会缺关键证据；如果摘要把地址、优惠券、物流节点都带回来，污染并没有消失。

正确示例：隔离订单查询，只返回诊断字段pseudo
order_summary = order_lookup_subagent.run(
    task="读取订单历史，提取技术诊断需要的信息",
    return_fields=[
        "订单是否有效",
        "产品型号",
        "最近激活时间",
        "错误码",
        "需要人工确认的缺失信息"
    ]
)

main_agent.continue_diagnosis(user_problem, order_summary)
字段为什么保留为什么不保留更多
订单是否有效决定是否继续技术诊断。完整购买历史通常不影响当前故障。
产品型号不同型号可能对应不同故障路径。收货地址、优惠信息对技术判断无关。
错误码直接指向诊断分支。聊天历史里的寒暄会干扰模型。
缺失信息提示主智能体下一步追问。不要把“不确定”伪装成确定结论。

你现在应该能做什么：设计子智能体输出时，先写“主智能体后续推理需要哪些字段”，而不是让子智能体把自己看过的材料都总结一遍。

4. 按上下文拆分，不按问题类型拆分

context-centric

Claude 原文里有一个很重要的区分：问题导向拆分和上下文导向拆分。问题导向拆分会说“一个智能体规划，一个智能体实现，一个智能体测试，一个智能体评审”。听起来像专业团队，但这些阶段往往共享同一批背景。每次交接都会丢掉为什么这么做、试过哪些方案、哪些边界条件不能碰。

上下文导向拆分问的是：哪些工作可以在不共享大量背景的情况下独立完成？如果答案是“不能”，就不要拆。如果实现和测试都围绕同一个功能，同一个智能体已经知道设计原因、边界条件和失败尝试，它顺手写测试反而更自然。

问题导向拆分

按工作类型拆：规划、实现、测试、评审。风险是每个智能体都缺上一阶段的隐含上下文。

上下文导向拆分

按信息边界拆：订单查询、独立地区研究、黑盒验证、清晰 API 两侧。风险更可控。

判断句

如果两个子任务之间需要频繁解释“我为什么这么做”，它们大概率应该留在同一个上下文里。

5. 哪些边界适合拆，哪些不适合

boundary test

有效边界通常有一个共同点：子任务之间只需要少量清晰接口，不需要共享大量思考过程。比如分别研究亚洲和欧洲市场趋势，两个方向可以并行搜索，最后汇总。再比如有明确 API 合同的前后端组件，如果接口足够清楚，也可以分开做。

危险边界也有共同点：它们看起来是不同阶段，实际共享同一个理解。规划、实现和测试同一功能，测试者需要知道实现权衡，评审者需要知道探索过程。如果只拿到摘要，就容易误判。

边界类型是否适合拆原因检查问题
独立研究路径适合信息源和分析方向可以分开。各子任务是否能独立得出结果？
清晰接口的组件视情况适合接口合同能减少沟通。接口是否足够稳定？
黑盒验证适合验证者只需输入输出和测试标准。是否能不看构建过程就检查？
同一功能的顺序阶段通常不适合共享设计原因、失败尝试和边界条件。交接摘要会丢掉什么？
紧耦合组件不适合需要频繁同步状态。是否一直来回问对方？

你现在应该能做什么：拿到一个拆分方案时，用“接口是否清晰、是否共享大量背景、交接会丢什么”三问快速判断。

6. 拆分对比：传话游戏 vs 干净隔离

interactive

下面的切换图展示两种常见拆法。坏拆法让每个智能体接手同一件事的不同阶段，背景不断压缩。好拆法让子智能体处理可隔离的噪声任务，只把必要摘要交回主上下文。

岗位式拆分容易变成传话游戏

规划、实现、测试、评审都围绕同一个功能，实际需要共享大量背景。每次交接只能传摘要，信息会逐步变薄。

观察重点：测试者和评审者是否缺少“为什么这样实现”的上下文。

7. 常见误区、后果和修法

diagnosis
误区错因后果修法
把所有长上下文都拆给子智能体没有区分“长但相关”和“长且无关”。无谓增加协调成本。先判断材料是否会污染后续推理。
子智能体返回完整摘要害怕丢信息，于是把噪声又带回来。主上下文继续被污染。提前定义主任务需要的字段。
按岗位拆同一任务套用人类组织分工。交接摘要丢掉动机和边界。按上下文边界拆，强相关工作留在一起。
接口不清就并行拆低估同步成本。子智能体反复互相等信息。先稳定接口或让同一智能体处理。
只看摘要是否短把短当成好。关键证据被压掉。检查摘要是否足够支持下一步推理。

例题讲解：该不该拆订单查询

worked example

场景：用户反馈“我买的专业版无法激活”。客服 Agent 需要查订单，还要判断激活失败原因。订单系统会返回完整购买历史、发票、地址、促销、支付流水、客服沟通等材料。

适合拆的部分

订单查询适合放到子智能体里。它需要读取大量高噪声材料，但主诊断只需要少数字段：订单是否有效、产品版本、激活记录、错误码、缺失信息。

不适合拆的部分

技术诊断和追问用户最好留在主智能体上下文里。它们共享同一条推理主线，拆开会让后续智能体不清楚为什么要追问某个信息。

判断结果：可以拆订单查询，但不要把整个客服流程拆成“接待、查单、诊断、回复”四个岗位式智能体。拆分的目标是隔离订单噪声，不是模拟公司组织结构。
