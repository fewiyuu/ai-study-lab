并行化与专业化：什么时候值得多开几个上下文

第 1 块讲了先单智能体，第 2 块讲了上下文保护。现在进入另外两种常见理由：并行化和专业化。它们都能让多智能体变得有价值，但前提不同，风险也不同。

拆独立面向→ 并行搜索→ 蒸馏发现→ 主智能体合成→ 处理冲突

学完本页，你应该能判断一个任务是否真的可并行，能解释为什么多智能体研究更全面但通常更贵，也能设计一个有清晰路由的专业化智能体系统。

本块学完要能做

  1. 解释并行化的主要收益是覆盖度，不是必然省钱。
  2. 把复杂研究问题拆成互相独立的面向。
  3. 区分工具集专业化、系统提示词专业化和领域专业化。
  4. 识别专业化带来的路由错误和维护成本。

1. 并行化适合什么任务

search space

并行化适合大搜索空间任务。一个智能体在有限上下文里只能探索有限方向；多个子智能体可以同时查不同来源、不同地区、不同假设或不同候选方案。Claude 原文把研究任务作为典型例子：主智能体分析问题，拆成多个研究面向，子智能体分别搜索，再把发现蒸馏回来。

关键条件是子任务独立。亚洲市场趋势、欧洲市场趋势、北美市场趋势可以分别研究，因为信息源和分析路径相对独立。反过来，如果子任务必须共享大量中间状态，或者一个子任务的结论决定另一个怎么做，就不是真并行。

适合

多地区研究、多来源事实核验、多候选方案探索、多个独立组件测试。

不适合

同一功能的连续实现步骤、强依赖推理链、需要频繁同步的组件。

判断句

如果每个子任务不用等其它子任务就能完成初稿，它才可能适合并行。

你现在应该能做什么：看到一个“大任务”时，不急着拆，而是先问每个面向是否能独立产出可合成的结果。

2. 并行化买的是覆盖度，不是免费速度

coverage vs cost

并行运行多个智能体，墙钟时间可能下降，但总计算量通常上升。每个子智能体都有自己的上下文、工具调用和结果摘要，最后主智能体还要合成。原文明确提醒：多智能体通常比单智能体消耗更多 token，主要收益是更全面。

这和人类直觉有点不一样。我们常把“并行”理解成“更快”，但 Agent 系统还要付合成成本。五个子智能体同时跑，可能比一个智能体顺序查五个方向更全面；但如果每个子智能体都查了重复资料，最后还要合并冲突，成本就会膨胀。

适合小范围并行实验

当前子智能体数不多，信息重叠可控。可以试着拆成几个独立面向，并要求每个子智能体返回结构化发现。

观察重点：覆盖度是否提升，而不是只看启动了几个子智能体。

这个交互不是精确计费器，而是提醒你看三个变量：子智能体越多，成本越高；信息重叠越大，浪费越多；合成越难，主智能体越容易在冲突结论里迷路。

3. 研究型多智能体的最小流程

lead and subagents

研究型流程通常由一个主智能体和多个子智能体组成。主智能体先理解问题，拆出研究面向；每个子智能体在自己的上下文里搜索和阅读；子智能体返回蒸馏后的发现；主智能体再合成结论、处理冲突、补齐引用。

这个流程里的“蒸馏”很重要。子智能体不应该把所有搜索过程都扔回来，而应该返回结构化发现：结论、来源、证据强度、未解决问题、和其它面向可能冲突的地方。否则合成阶段会变成另一次上下文污染。

正确示例：研究型并行流程pseudo
query = "比较三个地区的企业 AI 采用趋势"

facets = lead_agent.decompose(query, by=[
    "亚洲市场",
    "欧洲市场",
    "北美市场"
])

findings = parallel_run([
    research_subagent(facet, return_format=[
        "核心结论",
        "来源",
        "证据强度",
        "不确定点"
    ])
    for facet in facets
])

report = lead_agent.synthesize(findings)
阶段主智能体做什么子智能体做什么常见错误
拆分找独立面向。还不执行。把强依赖步骤误拆成并行。
搜索等待结构化结果。独立查资料。多个子智能体查同一批来源。
蒸馏定义返回格式。压缩发现。返回大段原始材料。
合成对齐、去重、处理冲突。必要时补查。直接拼接子报告。

4. 专业化不是角色扮演，而是减少选择负担

specialization

专业化的目标，是让每个智能体面对更小、更清晰的工具和任务空间。一个通用智能体如果同时管理 CRM、营销、通知、数据分析、代码执行、文档写作，模型会花很多注意力理解“我现在该用哪个工具”。如果领域边界清楚，专业化智能体能更稳定。

原文把专业化分成几类：工具集专业化、系统提示词专业化和领域知识专业化。工具集专业化让某个智能体只掌握某类工具；系统提示词专业化让它按特定目标和风格工作；领域专业化让它带着某个业务领域的术语和规则。

工具集专业化

CRM 智能体只处理客户数据工具，消息智能体只处理通知和群聊。

系统提示词专业化

不同智能体有不同工作准则、输出格式和风险约束。

领域知识专业化

金融、医疗、法务等领域使用专门术语、规则和检查清单。

判断句

如果专业化不能明显减少工具选择、任务目标或领域规则的混乱，它就可能只是换了几个角色名。

5. 专业化的代价：路由会变难

routing

专业化系统需要一个路由者，决定用户请求交给哪个智能体。路由看似小事，实际是主要风险。请求如果被分错，后面再专业也没用。比如“给上周购买但未激活的客户发提醒”同时涉及 CRM 查询和消息发送，路由者要么会编排两个智能体，要么会把任务错交给其中一个。

专业化还会增加提示词维护成本。每个智能体都有自己的工具权限、系统提示词、输出格式和错误处理。业务一变，可能多处都要改。如果边界不清，维护成本会吞掉专业化收益。

错误示例：路由规则太模糊anti-pattern
if user_request contains "客户":
    send_to(CRM_AGENT)
elif user_request contains "消息":
    send_to(MESSAGE_AGENT)
else:
    send_to(GENERAL_AGENT)

# “给未激活客户发提醒”同时需要客户筛选和消息发送。
# 简单关键词路由会漏掉编排关系。
正确示例：路由先判断任务结构pseudo
plan = router.analyze(user_request)

if plan.requires(["customer_filter", "message_send"]):
    customers = crm_agent.find_customers(plan.filter)
    result = message_agent.send_reminder(customers, plan.message)
else:
    route_to_single_specialist(plan)

好的路由不是关键词匹配，而是识别任务结构：需要哪些能力、顺序是什么、输出要如何交接。你现在应该能做什么：设计专业化系统时，先写路由规则和跨智能体编排，再写各智能体的人设。

6. 工具多时，先考虑按需发现

tool search

工具太多是专业化的常见动机，但不是唯一解。第 1 块提过，工具数量达到 15 到 20 个以上时，模型会花大量上下文理解工具选项。此时可以先尝试工具搜索或按需加载，让模型只在需要时看到相关工具，而不是一次加载全部工具定义。

如果工具天然属于清晰业务域，比如 CRM、营销活动、消息通知，而且任务路由也清楚，专业化就更有意义。如果工具只是数量多但边界混乱，拆成多个智能体会把“工具选择问题”升级成“路由问题”。

情况优先方案理由
工具多但同属一个流程工具搜索 / 工具分组仍然需要同一上下文判断。
工具分属清晰业务域专业化智能体能减少每个智能体的工具选择负担。
请求常跨多个业务域路由 + 编排需要明确谁先做、结果如何传递。
工具权限风险不同专业化或权限隔离敏感工具不应暴露给所有任务。

7. 常见误区、后果和修法

diagnosis
误区错因后果修法
并行一定更快只看同时启动，不看总计算和合成。token 暴涨，合成更慢。比较覆盖度、耗时、token 和冲突率。
子智能体越多越全面忽略信息重叠。重复搜索，结论冗余。拆分前定义互斥面向和返回格式。
专业化等于角色名没有减少工具或任务空间。维护多套提示词却没有收益。检查专业化是否对应工具、提示词或领域差异。
关键词路由就够了忽略任务可能跨领域。请求被分错，或者只做了一半。路由先识别能力组合和执行顺序。
合成只是拼接没有处理冲突和证据强度。报告表面完整，结论互相打架。要求子智能体返回来源、置信度和不确定点。

例题讲解：研究任务怎么拆

worked example

任务：分析“企业在 2026 年采用 AI Agent 的主要障碍”。这个问题可以从技术成熟度、组织流程、合规风险、成本收益四个面向拆。每个面向可以独立查资料，最后合成。

适合并行的理由

四个面向的信息源和判断标准不同，互相依赖不强。每个子智能体可以独立返回结论、来源、证据强度和未解问题。

合成时要小心

技术面说“已经成熟”，合规面可能说“仍然高风险”。主智能体不能直接拼接，需要解释冲突来自不同约束。

判断结果：可以做并行研究，但要约束返回格式，避免每个子智能体都写一篇完整长报告。主智能体的工作是去重、对齐、处理冲突，而不是简单粘贴。
