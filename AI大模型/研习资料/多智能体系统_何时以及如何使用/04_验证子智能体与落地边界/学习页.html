Verification subagent

验证子智能体：让通过变成可证明的通过

Claude 原文认为，跨领域比较稳定的一种多智能体模式，是验证子智能体。它只负责测试或验证主智能体的工作。这个模式有效，不是因为它更会“评价”，而是因为验证通常可以用很少的上下文完成。

主智能体产出→ 验证标准→ 黑盒检查→ 报告失败→ 修正或放行

学完本页，你应该能设计一个不会偷懒的验证子智能体，能写出具体检查标准，也能识别“跑了一两个测试就说通过”的早期胜利问题。

本块学完要能做

  1. 解释验证子智能体为什么适合单独拆出来。
  2. 区分黑盒验证和继续参与实现。
  3. 写出具体、全面、包含负例的验证指令。
  4. 用最终采用清单判断多智能体是否值得落地。

1. 验证子智能体负责什么

sole responsibility

验证子智能体的职责很窄：测试或验证主智能体的产出。它不负责重新规划，不负责顺手改代码，不负责替主智能体解释为什么这样做。职责窄反而是优势，因为验证通常只需要输入、输出、规格和检查标准，不需要完整构建过程。

这正好避开多智能体常见的传话游戏。实现者需要知道很多历史原因，验证者却可以像用户或测试系统一样黑盒检查。它只问：这个结果是否满足规格？测试是否真的跑过？边界情况是否覆盖？不该通过的输入有没有失败？

看产出

读取最终代码、文档、报告或结构化结果。

看标准

按明确规格、测试套件、schema、规则或引用要求检查。

报失败

报告哪些检查失败、证据是什么、下一步该修哪里。

判断句

如果验证者必须知道实现过程里的大量权衡才能判断，它可能不是好拆分；如果它只需规格和结果就能检查，就是好边界。

2. 为什么黑盒验证有效

minimal context transfer

黑盒验证的意思是：验证者不依赖完整内部过程，只根据外部可观察行为判断结果。软件里是跑测试、lint、schema 校验；内容里是核对引用、检查政策规则、确认输出格式；业务流程里是检查是否满足交付条件。

这类工作天然需要的上下文少。验证者不必知道实现者为什么选某个库，只要知道测试应该怎么跑。它也不必知道报告写作过程，只要知道每个事实是否有来源、是否符合规则。上下文传递少，多智能体的交接成本就低。

正确示例：验证者只拿规格和产出pseudo
verifier.run(
    artifact="主智能体生成的代码或报告",
    criteria=[
        "运行完整测试套件",
        "检查 lint 和类型错误",
        "验证输出 schema",
        "尝试至少 3 个边界输入",
        "列出所有失败，不要只报告第一个"
    ]
)

这段伪代码的重点是 criteria。验证子智能体不能只收到“看看是否正常”。它需要可执行、可检查、可报告的标准。

3. 验证指令要具体到能执行

concrete criteria

验证子智能体的质量，很大程度取决于验证指令。模糊指令会让它走捷径。比如“确认能用”太松，它可能跑一个 happy path 就宣布通过。更好的指令是“运行完整测试套件，报告所有失败；再测试空输入、非法输入和权限不足三类负例”。

好的验证标准至少有四类：具体标准、全面检查、负例测试、明确禁止偷懒。负例测试很关键，因为很多系统只在正常输入上表现好，一到异常输入就暴露问题。

标准类型好写法差写法
具体标准运行完整测试套件，并报告所有失败。看看能不能跑。
全面检查覆盖正常、边界、异常和权限场景。试一两个例子。
负例测试输入无效参数，确认系统拒绝。只测成功路径。
明确要求未完成全部检查前不得标记通过。通过了就说一声。
错误示例：验证指令太松anti-pattern
请检查这个功能是否正常。
正确示例：验证指令可执行prompt
请运行完整测试套件，报告所有失败。
额外测试：空输入、超长输入、权限不足、网络失败。
如果任何检查未执行，请不要标记通过。
输出格式：通过项、失败项、未执行项、复现步骤。

你现在应该能做什么：写验证 prompt 时，避免“确认一下”这种松句子，改成可执行的检查清单。

4. 最大失败模式：早期胜利问题

early victory

原文把 early victory problem 说得很明确：验证子智能体最严重的失败，是跑了一两个测试，看到通过，就宣布成功。这很像人类调试时只测 happy path。问题是，验证者一旦过早放行，系统会给用户一种“已经可靠”的错觉。

解决办法不是让验证者“更认真一点”，而是把认真写成规则。要求它运行完整测试，覆盖多个场景和边界，主动尝试应该失败的输入，并且在未完成全部检查前不能标记通过。

还不能宣布通过

只跑成功路径会制造早期胜利。验证者需要检查边界、负例和完整测试结果。

观察重点：验证报告里有没有“未执行项”。没有跑的检查不能默认为通过。

5. 验证子智能体适合哪些场景

applications

验证子智能体适合那些标准明确、检查动作相对独立的场景。质量保障是最直观的：跑测试、lint、类型检查、schema 校验。合规检查也适合：文档是否符合政策、输出是否触犯规则、流程是否满足审批条件。

内容生成也能用验证子智能体，比如检查引用是否存在、事实是否和来源一致、输出是否符合格式。这里的重点仍然是标准明确。如果你只是让它“评价写得好不好”，就很容易变成主观反馈，而不是可靠验证。

质量保障

测试套件、lint、类型检查、schema 校验、输出格式检查。

合规检查

政策规则、敏感内容、权限要求、审批条件。

输出验证

内容是否满足规格，结构化字段是否完整。

事实核验

声明是否有来源，引用是否支持结论。

6. 多智能体落地前的最终清单

moving forward

整篇文章的落点很克制：多智能体强大，但不是默认选择。落地前至少确认三件事。第一，确实存在多智能体能解决的约束，例如上下文限制、并行机会或专业化需求。第二，拆分遵循上下文，而不是按问题类型或岗位名硬拆。第三，有清晰验证点，子智能体能在不需要完整上下文的情况下检查结果。

这三条可以变成采用清单。清单不是为了阻止你使用多智能体，而是避免架构先行。你要能说出：我们为什么拆、拆在哪里、成本如何衡量、验证怎么做、什么时候回退到单智能体。

问题通过标准如果答不上来
约束是否真实？有失败案例、指标或上下文证据。先做单智能体基线。
拆分边界是否按上下文？子任务共享背景少，接口清楚。重新合并或改边界。
成本是否可接受？token、耗时、维护成本都有记录。做小实验，不要直接重构。
验证是否明确？有具体、全面、含负例的检查。先补验证标准。
回退条件是什么？知道何时回到单智能体或工具搜索。先定义停止线。

7. 常见误区、后果和修法

diagnosis
误区错因后果修法
验证者参与实现职责边界不清。失去独立检查视角。验证者只检查产出和标准。
只测 happy path过早满足于第一个成功信号。隐藏边界问题。强制完整测试、边界和负例。
没有未执行项报告只写成功，不写没测什么。用户误以为全部通过。输出通过、失败、未执行三类。
验证标准太主观缺少可检查条件。不同验证者结论不一致。改成测试、schema、规则、来源核验。
拆分后不记录成本只看架构是否能跑。无法判断多智能体是否值得。持续记录 token、耗时、失败率和维护成本。

例题讲解：怎么写验证任务

worked example

场景：主智能体生成了一个用户导入功能。你要让验证子智能体检查它是否可靠。差的写法是“检查导入功能是否正常”。好的写法要给出输入范围、检查动作、失败报告格式和禁止提前通过的规则。

差写法

“帮我看一下导入功能有没有问题。”这会导致验证者只上传一个正常 CSV，看到成功就放行。

好写法

“用正常 CSV、空文件、缺列、重复用户、超大文件和非法编码测试导入。记录每项结果，未执行项单独列出，任何失败都不得标记通过。”

判断结果：验证任务能不能独立拆出来，取决于它是否有清晰标准。标准越清楚，验证子智能体越像可靠检查点；标准越模糊，它越像另一个会点头的聊天者。
