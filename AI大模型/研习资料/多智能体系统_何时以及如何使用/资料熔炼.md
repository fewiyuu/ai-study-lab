> [!abstract]
> 来源材料是 Claude 官方文章《Building multi-agent systems: When and how to use them》。核心观点：大多数团队不需要默认使用多智能体；只有当单智能体遇到明确约束，例如上下文保护、并行搜索空间、工具/提示词/领域专业化，且收益能覆盖协调成本时，才值得拆分。

## 领域术语表

| 术语 | 直觉解释 | 精确定义 | 常见误解 |
|---|---|---|---|
| 单智能体基线 | 先让一个 Agent 做到可比较水平 | 一个主上下文里完成任务、调用工具、输出结果的版本 | 以为单智能体一定低级 |
| 多智能体系统 | 多个 Agent 协作完成任务 | 多个独立上下文之间分派、执行、汇报、合成 | 以为 Agent 越多越强 |
| 上下文污染 | 无关材料拖累当前推理 | 上下文累积了后续任务不需要的信息，影响模型判断 | 只把它理解成上下文太长 |
| 并行化 | 多个独立方向同时探索 | 主智能体拆分独立面向，子智能体并行搜索或执行，再合成结果 | 以为并行一定更便宜 |
| 专业化 | 缩小工具和任务空间 | 按工具集、系统提示词或领域知识划分专用 Agent | 以为取几个角色名就是专业化 |
| 路由 | 决定请求交给谁 | 识别任务结构、能力组合和执行顺序 | 以为关键词匹配就够 |
| 验证子智能体 | 独立检查主产出 | 只负责测试、校验、核验或合规检查的 Agent | 让验证者继续参与实现 |
| 早期胜利问题 | 太快宣布通过 | 验证者只跑少量成功路径就标记通过 | 以为有一个测试通过就可靠 |

## 知识骨架

1. 先从单智能体开始：建立可比较基线，避免过早引入协调、提示词维护和交接摘要成本。
2. 决策框架：多智能体只在解决明确约束时有价值，包括上下文保护、并行化和专业化。
3. 上下文保护：把高噪声子任务隔离到子智能体，只返回主任务需要的最小字段。
4. 按上下文拆分：有效边界是信息能隔离，不是规划、实现、测试这类岗位名。
5. 并行化：适合大搜索空间和独立研究路径，主要提升覆盖度，不保证更便宜。
6. 专业化：适合工具、提示词或领域规则清晰分离的任务，但会引入路由复杂度。
7. 验证子智能体：黑盒检查主产出，标准要具体、全面、包含负例，避免 early victory。
8. 落地清单：确认真实约束、上下文边界、成本指标、验证点和回退条件。

## 可练习点

- 判断一个任务是否先该做单智能体基线。
- 识别上下文污染与上下文压力的区别。
- 设计子智能体返回字段，避免把噪声重新带回主上下文。
- 把研究任务拆成独立可并行面向。
- 判断专业化是否减少了工具选择和领域规则混乱。
- 写路由规则，处理跨领域任务。
- 把模糊验证指令改成具体检查清单。
- 为多智能体方案设计成本与质量对照实验。

## 资料边界

- 原文是工程经验文章，不是形式化论文；数字如 3-10x token 成本应理解为经验观察，不是固定定律。
- 原文没有展开具体实现框架，本课程页使用伪代码和场景题补足学习入口。
- 原文重点在“何时使用”，不是多智能体算法综述；后续如要学习具体框架，可另开 LangGraph、AutoGen、CrewAI 或 Claude Agent 架构专题。
