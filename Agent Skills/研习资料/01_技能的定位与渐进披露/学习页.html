Skill mental model

技能的定位与渐进披露

Agent Skills 是给智能体打包专业知识的一种方式。它把说明、脚本、参考资料和模板放在一个目录里，让智能体在需要时加载，而不是每次把全部资料塞进提示词。

这节先建立判断框架：什么时候需要 Skill，Skill 和 prompt、tool、MCP 有什么区别，为什么渐进披露能解决上下文浪费。

任务意图→技能描述→加载说明→读取资源→执行或生成

学完后，你应该能判断一个任务该不该做成 Skill，并能解释为什么 Skill 的价值不是“多写一段提示词”，而是把可复用流程放进智能体运行环境。

来源材料

Claude 博客《Equipping agents for the real world with Agent Skills》指出，Skills 是由 instructions、scripts、resources 组成的文件夹，供智能体动态发现和加载。

Agent Skills 开放标准说明了目录结构、SKILL.md、scripts、references、assets 与 progressive disclosure。

1. 为什么需要 Agent Skills

from general to specialized

通用智能体已经能读文件、写代码、调用工具，但它不自动知道你的组织习惯、项目术语、输出格式和脆弱步骤。每次把这些都写进聊天窗口，短期能用，长期会变成重复劳动。

Skill 的直觉是：把“新同事入职手册”放在智能体能读取的位置。用户不用每次解释模板在哪、审批怎么走、错误怎么查；智能体根据任务描述识别相关 Skill，然后读取完整说明。

正式说法里，Skill 是一个目录。目录至少有 `SKILL.md`，可选带 `scripts/`、`references/`、`assets/`。它不是模型参数微调，也不是让模型永久记住知识，而是在执行任务时动态加载。

常见错法是把 Skill 当成“更长的系统提示词”。如果只是把一篇长文塞进 `SKILL.md`，没有触发边界、步骤、资源使用规则和验证方法，智能体仍然会抓不到重点。

Prompt

适合一次性约束：这次用中文、这次按某种口吻、这次输出表格。它快，但不沉淀流程。

Skill

适合可复用流程：项目规范、领域方法、模板资产、评估步骤、固定排错顺序。

Tool

适合执行动作：读取数据库、发消息、运行命令、调用 API。Skill 常常负责教智能体怎样稳妥地使用工具。

判断问题Prompt 倾向Skill 倾向
这件事只做一次吗是否，之后还会反复做
有没有固定产物格式没有，随聊随改有模板、字段、验证规则
是否需要额外文件不需要需要脚本、参考文档或资产
失败是否有固定排查法没有有常见坑和修复顺序

2. 渐进披露解决的是上下文预算

progressive disclosure

渐进披露的直觉很简单：先只告诉智能体“有哪些技能、什么时候用”，等任务真的相关时，再加载完整说明；只有需要脚本、模板或参考资料时，才继续读取那些文件。

这和把所有资料一次性塞进上下文相反。一次性塞入会让模型在一堆不相关规则里分心，也会挤掉当前任务的重要信息。渐进披露把 Skill 分成三层：启动时读取 name 和 description；触发后读取 SKILL.md 正文；需要时再读 scripts、references、assets。

正式规范里，`description` 是触发入口。它不只是介绍语，而是智能体判断“这个 Skill 是否相关”的主要依据。正文则写操作流程、边界、示例和注意事项。资源文件保留细节，避免主说明过长。

判断句：如果一个 Skill 的主文件必须装下所有 API 文档、全部模板和所有例外情况，它就没有利用渐进披露，反而会拖慢智能体。

切换任务，观察加载层级

加载越多，不代表越好。关键是只在需要时拿需要的信息。

Prompt 就够了

一次性的措辞改写通常不需要 Skill。直接给要求，模型就能完成。

观察点：没有稳定流程、没有外部资源、没有脚本复用时，不要急着做 Skill。
概念示例：三层加载pseudo
startup: load name + description for every skill
if task matches description:
    read SKILL.md
if instructions mention needed resources:
    read references/api-errors.md
    run scripts/validate.py

3. Skill、工具和子智能体不要混用

concept boundaries

Skill 常被误认为“新的工具系统”。更稳的理解是：Skill 是能力手册，工具是动作接口。手册可以告诉智能体先查 schema、再调用 API、最后验证输出；但真正的 API 调用仍然由工具、命令或脚本完成。

它也不是子智能体。子智能体强调把任务委托给另一段独立执行上下文，适合并行研究、隔离复杂任务或调用专门角色。Skill 则更像一套可加载的操作规范，同一个智能体加载后自己按规范做事。

它也不等于微调。微调改变模型行为分布，适合大量样本下的风格或能力调整。Skill 不改模型权重，只在任务时提供结构化知识。好处是更新快、可审计、能和文件资产一起移动。

常见错法是遇到所有问题都做 Skill。真正要问的是：这个能力有没有可复用流程？有没有局部知识会被通用模型猜错？有没有模板、脚本或验证门槛？如果没有，聊天提示或普通工具就够了。

概念解决什么问题不解决什么问题
Prompt约束本次回答长期复用和资源打包
Skill复用流程、项目知识、模板和脚本直接替代工具执行权限
Tool执行外部动作或查询教智能体完整业务流程
MCP把外部系统作为工具暴露给智能体自动生成领域操作规范
Subagent隔离或委托复杂任务保存可复用项目手册

4. 好 Skill 来自真实任务，不来自空泛总结

grounded expertise

开放标准的最佳实践强调：不要只让模型凭通用知识生成 Skill。通用知识会给出“处理错误”“遵循最佳实践”这类正确但没用的话。真正有价值的 Skill 来自实际执行中的稳定模式。

提炼时要看四类材料：成功步骤、用户纠正、输入输出格式、项目上下文。比如用户反复说“不要用某个库”“先查这个配置”“输出必须是某种 frontmatter”，这些就是 Skill 的高价值内容。

从已有项目资料提炼也可以。内部文档、runbook、API 规格、代码审查意见、真实故障记录，都会暴露模型容易猜错的地方。Skill 不需要复述整个领域，只要补上智能体没有它就会做错的部分。

判断句：如果删掉某条指令后，智能体大概率仍然做对，那条指令可能不该占用主文件空间；如果那条指令来自真实踩坑，并且删掉后容易复发，就应该写进 Skill。

成功步骤

哪些动作按什么顺序做，最后产生了稳定结果。

用户纠正

用户指出过的偏好、禁用方案、边界和命名要求。

失败记录

哪些错误经常出现，错误信息长什么样，下一步怎么查。

5. 最小 Skill 应该长什么样

minimal example

一个最小 Skill 不需要复杂目录，但必须让智能体知道两件事：什么时候用，以及怎么做。`name` 和目录名要匹配，`description` 要写清触发场景，正文给出可执行步骤。

下面这个例子不是为了完整，而是为了看结构。它适合“整理会议纪要”的重复任务：先抓行动项，再按项目归属分组，最后输出到固定 Markdown 模板。

注意，正文里不需要解释 Markdown 是什么，也不需要讲会议纪要的历史。智能体已经知道这些。Skill 要补的是项目约定：哪些字段必须保留，行动项怎么归属，遇到负责人缺失时怎么处理。

判断句：最小 Skill 的价值不在长度，而在它能不能让智能体少问一次、少猜一次、少走一次错流程。

正确示例：最小 SKILL.mdSKILL.md
---
name: meeting-notes
description: Summarize meeting transcripts into project action notes. Use when the user provides a meeting transcript or asks to extract decisions, owners, and next actions.
---

## Workflow

1. Extract decisions, action items, blockers, and open questions.
2. Group action items by project name. If the project is unclear, put it under "Unassigned".
3. Preserve owner names and due dates when present.
4. Output Markdown using assets/action-note-template.md.

## Gotchas

- Do not invent owners.
- If a due date is implied but not explicit, mark it as "待确认".
错误示例：空泛但看似完整SKILL.md
---
name: meeting-notes
description: Helps with meetings.
---

Summarize the meeting professionally.
Follow best practices.
Be accurate and concise.
Handle edge cases appropriately.

6. 五个常见误区

gotchas

第一，Skill 过宽。一个 Skill 同时管数据分析、报告写作、数据库迁移和前端设计，触发时会带来互相干扰的规则。更好的做法是按一个稳定工作单元来拆。

第二，Skill 过窄。每个小动作一个 Skill，会让一个真实任务触发多个 Skill，增加上下文开销，也提高冲突概率。判断标准是：它能不能独立完成一类用户意图。

第三，description 写得像标题。`Process files` 这种描述对触发帮助很小。它要写清任务类型、用户意图和近似说法。

第四，把详细参考全塞主文件。主文件越长，智能体越难抓到当前任务真正相关的步骤。细节应该放 references 或 assets，并写明什么时候读取。

第五，没有验证环。Skill 让智能体做完某件事后，应该知道如何检查结果。没有验证的 Skill 容易产生漂亮但不可用的输出。

误区后果修法
把 Skill 当长提示词上下文变大，流程仍不稳定写触发、步骤、边界、验证
description 太短该触发时不触发写用户意图和典型任务
主文件装太多智能体读不出重点拆 references/assets
没有 gotchas真实踩坑反复发生把纠正写成具体规则
没有验证输出看着完成但不可靠加检查清单或脚本
