SKILL.md 结构与触发

这一节把 Agent Skill 拆成可检查的文件结构。你会看到一个 Skill 最少需要什么，哪些 frontmatter 字段真正影响触发，正文应该写什么，资源文件应该怎样被引用。

重点不是背规范，而是能判断一个 Skill 为什么没有触发、为什么触发错了，以及怎样把描述写到既不过宽也不过窄。

目录名→SKILL.md→frontmatter→description→正文步骤→资源引用

学完后，你应该能写出一个能被客户端发现的最小 Skill，并能审查 name、description、allowed-tools、compatibility、references 和 assets 的常见错误。

来源材料

Agent Skills Specification 规定：Skill 是至少包含 `SKILL.md` 的目录；`SKILL.md` 必须有 YAML frontmatter 和 Markdown 正文。

Optimizing skill descriptions 说明：description 是触发的主要入口，要围绕用户意图而不是内部实现来写。

1. Skill 首先是一个目录

directory contract

规范里最小的 Skill 是一个目录，目录里有 `SKILL.md`。目录不是装饰，它给客户端一个稳定边界：这个目录里的说明、脚本、参考资料和模板属于同一个能力。

目录名和 `name` 字段要匹配。这个要求看似机械，但它减少了客户端发现和索引时的歧义。如果目录叫 `pdf-tools`，frontmatter 里却写 `name: document-helper`，不同客户端可能很难判断到底该显示哪个名字。

可选目录有三个常见用途：`scripts/` 放可执行代码，`references/` 放按需读取的详细说明，`assets/` 放模板、图片、配置样例或数据文件。你也可以有其他目录，但要让智能体知道什么时候用。

判断句：如果一个 Skill 目录里有很多文件，却没有在 `SKILL.md` 中说明它们的用途，智能体通常不会自动知道应该读哪一个。

正确示例：清晰目录tree
invoice-audit/
├── SKILL.md
├── scripts/
│   └── validate_invoice.py
├── references/
│   └── tax-rules.md
└── assets/
    └── audit-report-template.md
目录用途典型内容
SKILL.md元数据和主流程触发说明、步骤、边界、验证
scripts/可重复执行的逻辑校验器、转换器、解析器
references/任务需要时再读的细节API 错误、业务规则、长文档
assets/静态资源模板、样例、配置、图片

2. Frontmatter 是客户端发现 Skill 的入口

metadata

YAML frontmatter 位于 `SKILL.md` 开头。必填字段是 `name` 和 `description`。可选字段包括 `license`、`compatibility`、`metadata` 和实验性的 `allowed-tools`。

`name` 有硬约束：最长 64 个字符，只能用小写字母、数字和连字符，不能以连字符开头或结尾，不能有连续连字符，并且要和父目录名匹配。这个字段适合作为机器可读标识，不适合写中文标题或长句。

`description` 最长 1024 字符，必须非空。它要说明 Skill 做什么、什么时候用。客户端启动时通常只加载所有 Skill 的 name 和 description，所以这个字段会直接影响触发。

`compatibility` 用来写环境要求，比如需要 Node.js、需要 uv、面向 Claude Code 或类似客户端。`allowed-tools` 用来声明预批准工具，但支持情况会随客户端变化，不能把它当成所有环境都执行的安全策略。

正确示例：字段够用且边界清楚YAML
---
name: invoice-audit
description: Audit invoice PDFs and spreadsheets for missing fields, tax mismatches, and duplicate invoice numbers. Use when the user asks to review invoices, reconcile billing files, or prepare an invoice audit report.
compatibility: Requires Python 3.12+ and uv for scripts/validate_invoice.py
allowed-tools: Read Bash(uv:*)
---
错误示例：name 不合规，description 不可触发YAML
---
name: Invoice_Audit!
description: Helps.
---

3. Description 要写用户意图，不写内部实现

trigger accuracy

触发描述的直觉是：用户会怎么说这件事，智能体看到这些话能不能联想到这个 Skill。写“调用 `scripts/parse.py` 处理输入”通常不够，因为用户不会这么描述需求。

好描述包含四个信息：任务对象、用户目标、产物或动作、触发时机。比如“Analyze CSV and tabular data files，compute summary statistics，generate charts，clean messy data”，比“Process CSV files”更容易触发，也更不容易误触发。

描述过窄会漏掉相关任务。比如只写“CSV analysis”，当用户说“帮我从 Excel 里找异常销售额”时可能不会触发。描述过宽会误触发。比如“helps with data”会让数据库迁移、数据清洗、报表排版都可能撞进来。

判断句：description 不应该像宣传语，也不应该像内部函数名。它应该像一句给智能体看的路标：用户说什么时，该走进这个目录。

切换描述问题

触发准确性来自描述，不来自正文里藏着的细节。

触发偏窄

描述只写了工具名或内部实现，用户换一种说法时智能体看不出相关。

修法：补用户意图、任务对象和典型说法。
描述问题改法
Helps with PDFs.太短，不知道做什么列出抽取、填表、合并、审查等具体任务
Use parse.py to run the pipeline.内部实现，用户不会这么说改成用户意图和产物
Does all document work.过宽，容易误触发限定文件类型、任务边界和产物

4. 正文要让智能体知道下一步怎么做

instructions

`SKILL.md` 正文没有严格格式，但它不是随笔。好的正文通常包括流程、默认选择、输入输出格式、常见坑和验证方法。越脆弱的任务，步骤越要具体；越开放的任务，说明目的和判断原则更重要。

正文最重要的工作是补上智能体不知道、容易猜错或需要遵守的项目约定。不要解释它已经知道的基础概念。比如 PDF Skill 不需要长篇解释 PDF 是什么，但应该说“扫描件先 OCR”“表格抽取失败时用哪个工具”“输出字段如何命名”。

当多个方案都可行时，正文要给默认方案，而不是把所有选项平铺成菜单。智能体看到一堆平级选择，容易试错。写“默认用 pdfplumber；扫描件再用 OCR”比“可以用 A/B/C/D”更可执行。

判断句：正文写完后，拿一个真实任务问自己：智能体触发 Skill 后，能不能按它一步步完成？如果还需要你在聊天里补大量说明，正文就没有写到位。

流程

先做什么，再做什么，什么时候停止。

默认

多种工具可用时选哪一个，何时切换。

边界

哪些任务不适用，哪些信息不能猜。

正文模板：短但可执行Markdown
## Workflow

1. Inspect the input file type and size.
2. Use pdfplumber for text PDFs. Use OCR only when text extraction returns empty content.
3. Extract required fields using references/field-schema.md.
4. Validate the result with scripts/validate_fields.py.
5. If validation fails, report missing fields instead of inventing values.

## Gotchas

- Do not infer invoice numbers from filenames.
- Keep amounts as decimals. Do not round tax values.

5. 资源引用要告诉智能体何时读取

on-demand files

渐进披露只有在资源按需读取时才有效。`references/` 里可以放详细文档，但主文件必须写清楚触发条件。比如“API 返回非 200 时读取 references/api-errors.md”，比“更多信息见 references/”更有用。

引用文件时使用相对路径，从 Skill 根目录出发。这样目录移动到不同客户端或不同机器时仍然可用。绝对路径会把 Skill 绑死在某台机器上，也不利于跨平台分享。

资源文件不要无限套娃。主文件指向参考文件，参考文件再指向更多参考文件，智能体会在查找上浪费上下文。开放标准建议保持引用浅，文件也尽量聚焦。

判断句：主文件负责告诉智能体“何时读哪一个文件”。如果它只说“参考资料在 references 里”，智能体可能会读太多，也可能一个都不读。

正确示例：带触发条件的引用Markdown
## References

- Read references/schema.md before writing SQL.
- Read references/error-codes.md only if the API returns a non-200 status.
- Use assets/report-template.md when producing the final report.
错误示例：没有读取条件Markdown
See references/ for details.
Use whatever template looks appropriate.

6. 写完后先做结构验证

validate

Skill 的第一层质量不是文采，而是能否被客户端识别。结构验证要检查目录名、`name`、frontmatter、必填字段、字段长度和文件引用。

开放标准提供 `skills-ref validate ./my-skill` 作为参考验证方式。即使当前客户端没有这个命令，你也可以按同样思路人工检查：父目录名是否匹配 name，description 是否非空且不过长，YAML 是否能解析，引用路径是否存在。

第二层验证是触发。你需要准备一些 should-trigger 和 should-not-trigger 的真实用户句子，看 description 是否能区分。触发测试不等于输出质量测试，但它能先解决“该用不用、不该用乱用”的问题。

判断句：一个 Skill 没有被加载时，不要先怀疑模型笨。先查它是否能被发现，再查 description 是否像用户会说的话。

检查项通过标准失败后果
目录名/name完全一致，合法字符客户端发现或显示混乱
description具体说明任务和时机漏触发或误触发
YAML可解析，无缩进错误Skill 无法注册
路径引用相对路径存在执行时找不到资源
正文有步骤和验证触发后仍不知道怎么做
