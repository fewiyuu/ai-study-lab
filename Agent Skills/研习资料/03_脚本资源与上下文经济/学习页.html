Scripts and context

脚本资源与上下文经济

这一节关注 Skill 的可执行部分：什么时候该把步骤写成说明，什么时候该把逻辑做成脚本，脚本怎样给智能体返回可用信息。

核心直觉是：模型擅长判断和整合，脚本擅长稳定、重复、可验证的处理。一个好 Skill 会把两者分工讲清楚。

一次性命令→脚本封装→结构化输出→错误消息→验证循环

学完后，你应该能设计一个适合智能体调用的脚本接口，并能解释为什么 stdout、stderr、退出码、--help 和 dry-run 会影响智能体下一步行动。

来源材料

Using scripts in skills 说明了 one-off commands、自包含脚本、PEP 723、uv run、非交互输入、--help、结构化输出、错误信息、幂等性、dry-run 和输出大小控制。

Best practices 指出，重复被智能体临时重写的逻辑适合沉淀到 scripts/。

1. 什么时候说明够用，什么时候要脚本

script boundary

Skill 正文适合写判断、步骤和边界；脚本适合做稳定、重复、可机械验证的处理。比如“先读 schema 再写查询”可以是说明，“检查 200 个字段是否符合 schema”更适合脚本。

一个简单判断：如果智能体每次都会临时写差不多的代码，那就该把它沉淀到 `scripts/`。临时代码会随着上下文、模型采样和用户措辞变化而漂移；脚本把核心逻辑固定下来，智能体只需要调用。

脚本还有一个价值：减少上下文消耗。让模型读一大段数据再手工判断，不如让脚本先输出结构化摘要。模型看到摘要后做决策，既省上下文，也更稳定。

判断句：脚本不是为了取代智能体，而是把“可重复、可验证、容易写错”的部分从语言推理里拿出来。

2. 一次性命令要可复现

one-off commands

并不是所有命令都要写进 `scripts/`。如果一个现成工具已经能完成任务，`SKILL.md` 可以直接写命令。比如 Python 生态可以用 `uvx` 运行固定版本工具，Node 生态可以用 `npx package@version`。

关键是可复现。命令里最好固定版本，并写清前置条件。`npx eslint .` 今天和半年后可能行为不同，`npx eslint@9.0.0 .` 更稳定。Python 脚本在你的仓库偏好里也应该优先用 uv 管理。

命令复杂到难以一次写对时，就该搬到脚本。比如带多层引号、多个输入输出、错误分支和临时文件的命令，写成脚本比让智能体每次拼 shell 更可靠。

判断句：一次性命令适合“调用一个成熟工具”；脚本适合“把一段脆弱流程变成稳定接口”。

正确示例：固定版本的一次性命令bash
uvx ruff@0.8.0 check .
npx eslint@9 --fix .
风险示例：版本漂移和环境不清bash
npx eslint .
python scripts/process.py  # 不知道依赖装在哪个环境

3. 自包含脚本让 Skill 更容易迁移

uv run

自包含脚本的意思是：脚本自己声明依赖，智能体只需要一个命令就能运行。Python 可以用 PEP 723 在文件顶部写依赖，然后用 `uv run scripts/extract.py` 执行。

这对 Skill 很重要。Skill 经常跨项目、跨机器、跨客户端使用。如果脚本依赖隐藏在某个全局环境里，别人拿到 Skill 后就很难复现。把依赖写在脚本里，客户端能更容易安装和缓存。

依赖也要控制版本。`beautifulsoup4>=4.12,<5` 比裸写 `beautifulsoup4` 更稳。需要特定 Python 版本时，在脚本元数据里写 `requires-python`，比在正文里模糊提醒更可靠。

判断句：一个脚本如果只能在作者电脑上跑，它还不是 Skill 资源；它只是本地小工具。

正确示例：PEP 723 + uvpython
# /// script
# requires-python = ">=3.12"
# dependencies = [
#   "beautifulsoup4>=4.12,<5",
# ]
# ///

from bs4 import BeautifulSoup
import argparse

parser = argparse.ArgumentParser()
parser.add_argument("html_file")
args = parser.parse_args()
运行方式bash
uv run scripts/extract.py input.html

4. 面向智能体的脚本接口要避免交互

non-interactive

智能体运行脚本时通常没有交互式终端。脚本如果弹出“请选择环境”“是否继续”的提示，智能体不能像人一样输入，任务会挂住。输入应该来自命令行参数、环境变量或 stdin。

`--help` 是智能体学习脚本接口的主要入口。帮助信息要短，说明用途、参数、默认值和例子。不要把帮助写成论文；它会进入上下文，过长反而影响使用。

错误信息要具体。`Error: invalid input` 没法指导下一步；`Error: --format must be one of json, csv, table. Received: xml` 能让智能体立刻修正参数。

判断句：脚本的输出不是给人看热闹，而是给智能体决定下一步。它要清楚、有限、可解析。

错误示例：会卡住的交互python
env = input("Target environment: ")
deploy(env)
正确示例：参数 + 清晰错误python
parser = argparse.ArgumentParser(description="Deploy a release.")
parser.add_argument("--env", choices=["dev", "staging", "prod"], required=True)
parser.add_argument("--tag", required=True)
args = parser.parse_args()
接口要素推荐做法原因
输入参数、stdin、环境变量不会阻塞
帮助简短 --help 和示例智能体能快速学会
错误说明收到什么、期望什么便于自我修正
退出码不同失败类型区分便于分支处理

5. stdout、stderr 和输出大小要设计

structured output

脚本输出会进入智能体上下文。结构化输出比对齐空格的表格更可靠。JSON、CSV、TSV 都能被程序继续处理，也能让智能体少猜字段边界。

数据和诊断要分开：stdout 输出干净数据，stderr 输出进度、警告和诊断。这样智能体可以捕获 stdout 做下一步，也不会丢失错误线索。

输出大小也要控制。很多智能体环境会截断高噪声输出。脚本如果可能产生大结果，默认应该输出摘要，提供 `--limit`、`--offset` 或 `--output`。需要完整内容时写文件，而不是把几万行直接吐到终端。

判断句：脚本不是说得越多越好。对智能体来说，能解析、能定位、能继续行动，比完整铺开日志更有用。

正确示例：stdout 给数据，stderr 给诊断python
print(json.dumps({"missing": 3, "duplicates": 1}))
print("Warning: 2 rows skipped because dates were empty", file=sys.stderr)
风险示例：不可解析的大段输出python
print("Processing file...")
print("Here are all rows:")
for row in rows:
    print(row)
print("Done, maybe some rows failed")

6. 智能体可能重试，脚本要有安全默认

idempotency

智能体看到错误后可能再次运行命令。脚本如果每次运行都会追加、删除或覆盖，就要考虑幂等性。能“存在则更新、不存在则创建”的脚本，比“重复创建就失败或重复写入”的脚本更适合智能体。

有状态或破坏性操作应该支持 `--dry-run`，让智能体先看到会发生什么。真正执行时可以要求 `--confirm` 或 `--force` 这类显式参数。这样 Skill 的正文可以规定：先 dry-run，检查摘要，再执行。

安全默认不是一句“请谨慎”。它应该体现在脚本接口里：默认不删除，默认限制范围，默认输出计划，默认拒绝模糊输入。

判断句：一个适合智能体的脚本，应该假设自己会被误用、重试、传错参数，然后仍然给出可恢复的结果。

风险脚本护栏Skill 正文写法
重复运行幂等 create-or-update允许重试，但检查输出
破坏性操作--dry-run + --confirm先预览计划，再执行
模糊输入枚举参数和清晰错误不要猜用户意图
输出过大--limit / --output默认摘要，必要时写文件
