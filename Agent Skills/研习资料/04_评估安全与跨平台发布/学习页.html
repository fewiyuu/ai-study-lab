Eval and portability

评估、安全与跨平台发布

最后一节把 Skill 从“写出来”推进到“可靠使用”。你会学习触发评估、输出质量评估、安全边界、版本维护和跨客户端发布。

一个 Skill 真正可用，不是因为它看起来像规范，而是因为它在真实任务、边界条件和不同客户端里都能给智能体稳定帮助。

触发评估→输出评估→安全边界→版本维护→跨端发布

学完后，你应该能为一个 Skill 设计最小 eval 集，区分 should-trigger 与 should-not-trigger，写出输出断言，并说明发布到其他客户端前要检查哪些兼容性问题。

来源材料

Evaluating skill output quality 说明了 evals/evals.json、with-skill/without-skill 对照、assertions、grading 和迭代。

Optimizing skill descriptions 说明了触发查询、近似负例、训练/验证拆分和多次运行触发率。

Agent Skills open standard 提供跨平台目录格式和客户端实现参考。

1. 先评估 Skill 会不会正确触发

trigger eval

Skill 只有被加载才有机会发挥作用。触发评估要准备两类用户查询：should-trigger 和 should-not-trigger。前者是应该触发 Skill 的真实任务，后者是关键词相近但不该触发的近似负例。

强负例比明显无关的句子更有价值。比如 CSV 分析 Skill 的负例不该是“今天天气怎样”，而应该是“写一个脚本把 CSV 上传到数据库”。它也包含 CSV，但任务是 ETL，不是分析。

由于模型行为有随机性，同一个查询最好运行多次，记录触发率。should-trigger 的触发率应该高于阈值，should-not-trigger 的触发率应该低于阈值。阈值可以从 0.5 起步，再根据实际要求调。

判断句：触发评估回答的是“该不该加载”。它不回答“加载后做得好不好”。这两个问题要分开测。

切换评估视角

可靠 Skill 至少要经过触发、输出和边界三类检查。

只测触发

触发测试告诉你 Skill 会不会被加载，但不证明输出好。

下一步还要测产物质量、错误处理和验证结果。
触发评估样例json
[
  {
    "query": "帮我把这个会议逐字稿整理成决策和待办",
    "should_trigger": true
  },
  {
    "query": "给我写一封邀请同事开会的邮件",
    "should_trigger": false
  }
]

2. 输出质量评估要有可观察断言

assertions

输出质量评估通常从 `evals/evals.json` 开始。每个测试包含 prompt、expected output、可选输入文件和 assertions。断言是可检查的成功条件。

好断言要具体、有证据。`输出文件是合法 JSON`、`报告包含三个行动项`、`图表有坐标轴标签` 都能检查。`输出很专业`、`质量很高` 太虚，不适合作为断言。

评估时可以跑 with-skill 和 without-skill 对照。这样你能知道 Skill 到底带来了什么提升，而不是只看一次漂亮输出。改进已有 Skill 时，也可以用旧版本做 baseline。

判断句：Skill 的价值要能被比较出来。它至少应该让结果更准确、更稳定、更符合格式，或者减少智能体走弯路的时间。

输出质量 eval 样例json
{
  "skill_name": "meeting-notes",
  "evals": [
    {
      "id": "decisions-actions",
      "prompt": "整理 transcript.txt，提取决策和待办。",
      "expected_output": "Markdown report with decisions, action items, owners, and open questions.",
      "files": ["evals/files/transcript.txt"],
      "assertions": [
        "The report includes a Decisions section.",
        "Each action item includes an owner or says 待确认.",
        "The output does not invent due dates."
      ]
    }
  ]
}
断言好坏理由
输出很好差不可观察
包含 5 个字段：title、owner、due、status、source好可检查字段
语气专业偏弱需要更具体的风格标准
没有编造负责人好能对照输入证据

3. 评分要记录证据，不只写 PASS

grading

评分不是给感觉打分，而是把每条断言和输出证据对上。一个断言通过时，要能指出文件、字段、段落或命令输出；失败时，也要写清缺了什么。

证据能帮助迭代。比如“图表有标题但没有 X 轴标签”比“图表不好”更能指导下一版 Skill。智能体或人类评审看到证据后，能判断是 Skill 指令不清、脚本缺失，还是断言写错。

有些质量很难做成机械断言，比如整体可读性、设计感、解释是否抓住重点。这时可以加入人工反馈或盲评，但仍然要尽量记录具体观察。

判断句：PASS 没有证据就不算真正验证。你要能回到输出里找到支撑它的东西。

评分记录样例json
{
  "assertion_results": [
    {
      "text": "Each action item includes an owner or says 待确认.",
      "passed": false,
      "evidence": "Action item 3 says 'Follow up with vendor' but has no owner field."
    }
  ],
  "summary": {"passed": 2, "failed": 1, "total": 3}
}

4. 迭代时看失败模式，不要只补关键词

iteration

优化 description 时，容易犯的错是把失败查询里的词硬塞进去。这样会过拟合：对训练查询有效，对新查询可能更差。更好的做法是找失败背后的类别。

如果 should-trigger 没触发，可能是描述太窄，没有覆盖用户意图。如果 should-not-trigger 误触发，可能是描述太宽，没有写边界。修描述时要用一般化表达，而不是复制某条测试句。

输出质量失败也一样。不要只给某个测试补丁，要看它反映的流程缺口。比如多个测试都漏 owner，说明 Skill 需要在流程或模板里明确 owner 字段；不是只在一个例子里加 owner。

判断句：评估不是为了把测试集刷满，而是为了发现 Skill 哪条规则不清、哪段流程缺失、哪个资源该脚本化。

失败现象可能原因改进方向
相关任务未触发description 太窄补用户意图和同义说法
近似任务误触发边界不清增加不适用范围或更具体对象
输出缺固定字段模板或流程缺字段更新 assets 或正文步骤
每次格式不同指令太自由提供模板和验证
脚本被错误调用--help 或错误信息不清改脚本接口

5. 安全要靠边界和验证，不靠一句提醒

safety

Skill 可能教智能体运行命令、发送消息、访问文件或调用付费 API。越接近真实副作用，越需要明确边界：哪些操作可做，哪些必须询问，哪些永远拒绝。

`allowed-tools` 可以声明预批准工具，但它是实验字段，不同客户端支持不同。不能把安全完全寄托在这个字段上。正文和脚本也要有安全默认：dry-run、确认参数、路径范围、最小权限和输出验证。

发布前要测试负面场景。比如用户要求删除目录、导出敏感数据、给全员发消息、绕过审批时，Skill 应该让智能体停下来，而不是“尽力完成”。

判断句：安全 Skill 的写法要具体到动作。不要写“谨慎处理敏感信息”，要写“不要输出 API Key；发现密钥只报告文件和行号，不复制值”。

权限

只允许需要的工具和最小范围。

确认

破坏性动作先 dry-run，再要求显式确认。

数据

不要复制密钥、PII 或内部令牌。

更具体的安全规则Markdown
## Safety

- Never print secret values. Report only file path and key name.
- For delete operations, run scripts/plan_delete.py --dry-run first.
- Do not send messages to external recipients without user confirmation.
- Refuse requests to bypass approval, logging, or access controls.

6. 开放标准解决格式，客户端能力仍要核查

portability

Agent Skills 开放标准的好处是给了统一目录格式：`SKILL.md`、`scripts/`、`references/`、`assets/`。这让 Skill 更容易在不同智能体客户端之间移动。

但跨平台不等于零检查。不同客户端可能使用不同工具名、不同权限模型、不同工作目录、不同 shell，也可能不支持某些 frontmatter 字段。`allowed-tools` 尤其要谨慎，因为它的支持情况会变化。

发布 Skill 前，要检查相对路径、脚本运行环境、依赖安装方式、文件名大小写、网络需求和许可证。Windows、macOS、Linux 对路径和可执行脚本处理也有差异。

判断句：开放标准让 Skill 可携带，兼容性验证让它真的能跑。二者缺一不可。

发布检查项要问的问题常见修法
路径是否全是相对路径从 Skill 根目录引用
依赖是否写清版本和安装方式用 uv/PEP 723 或锁定工具版本
权限目标客户端是否支持字段正文保留安全边界
脚本是否非交互、跨平台避免硬编码 shell 特性
许可证能否分发模板和资料写 license 或移除受限资源
