Agent Loop · ReAct

从思考、行动、观察理解 Agent

Agent 不等于“模型会聊天”，也不等于“模型能调用工具”。更稳的入口是一个循环：模型根据目标和状态提出下一步，程序执行被允许的动作，环境返回观察，模型再决定继续还是停下。这个循环如果没有状态、工具边界和停止条件，就很难调试，也很难放心交给它做事。

目标→ 思考→ 行动→ 观察→ 状态更新→ 停止或继续

学完本页，你应该能做三件事：读懂 ReAct 的基本思想，画出一个最小 Agent Loop，判断一个系统更像 Workflow 还是 Agent。

学习目标

  1. 解释 Thought、Action、Observation 分别解决什么问题。
  2. 分清普通 LLM 调用、工具调用、Workflow 和 Agent 的边界。
  3. 指出最小循环里的状态、工具权限、观察结果和停止条件。
  4. 诊断循环卡住、乱调工具、反复搜索或越权执行的常见原因。

一、ReAct 的直觉：一边推理，一边接触外部世界

REACT

ReAct 这个名字来自 reasoning and acting。直觉上，它反对把“推理”和“行动”拆成互不相干的两段。真实任务很少是先把完整计划写完，再机械执行到底。你查到一条资料后，下一步计划可能要改；工具报错后，原来的路线也要改；用户目标不清时，继续执行反而会把任务带偏。

ReAct 的做法是让模型交错生成思考痕迹和任务相关动作。思考用来解释当前状态、维护计划、处理异常；行动让模型接触外部知识库、搜索接口或环境；观察把外部反馈带回来，让下一轮判断有依据。这样一来，模型不只是“自己想”，也会被环境纠正。

ReAct 论文里最值得抓住的一点是：只靠 Chain-of-Thought 容易把错误沿着推理链传播下去，只靠行动计划又缺少中途修正。思考和行动交错后，模型可以先查证，再更新计划。你以后看到 Agent 轨迹时，不要只看它写了多少推理文本，要看它有没有用观察结果修正下一步。

最小轨迹：问答型 ReActTrace
Question: 2026 年某产品是否仍支持旧接口？
Thought: 我不能凭记忆回答，需要查官方文档。
Action: search_docs({ query: "legacy API support 2026" })
Observation: 文档显示旧接口将在 2026-09-30 停止支持。
Thought: 需要把日期和迁移建议一起告诉用户。
Action: final_answer("旧接口支持到 2026-09-30，之后应迁移到 v2。")

读完这一节，你应该能做的事：看到一段 Agent 轨迹时，指出哪一句是计划更新，哪一步是外部动作，哪条信息是观察结果。

二、最小循环：代码短，边界不能短

LOOP

循环体只有几步

一个最小 Agent Loop 通常这样运行：把用户目标和已有消息交给模型；模型返回最终回答或工具请求；程序执行工具；工具结果作为观察追加回消息；模型继续判断下一步。代码看起来不长，真正麻烦的是循环外面的东西。

循环外至少要有工具 schema、权限检查、最大步数、成本预算、日志、错误处理和人工确认。缺少这些，Agent 会从“自动化助手”变成“不可预测的执行器”。所以，Agent Loop 的重点不是写一个 `while true`，而是让每一轮都有可检查的输入、动作和输出。

一个健康循环要能回答两个问题：为什么这一步要调用这个工具？工具结果如何改变下一步？如果日志只能看到最终回答，看不到每轮观察和状态变化，后面就很难复盘。

交互演示：一步步看状态

目标：用户问“这个订单为什么还没发货？”
思考：不能猜原因，需要先查订单状态。
行动：调用 get_order_status(order_id)，程序检查用户是否有权查看。
观察：订单已付款，仓库缺货，预计明天补货。
停止：信息足够，生成回复；如果缺订单号，则停下并向用户澄清。

观察点：停止不是失败。缺少必要信息时停下来，比继续乱查更可靠。

伪代码：受控 Agent LoopTypeScript-like
const messages = [{ role: "user", content: userInput }];

for (let step = 0; step < MAX_STEPS; step++) {
  const response = await model({ messages, tools });
  messages.push({ role: "assistant", content: response.content });

  if (response.stop_reason !== "tool_use") {
    break;
  }

  const results = await runAllowedTools(response.tool_calls, {
    workspace,
    permissions,
    budget,
  });

  messages.push({ role: "user", content: results });
}

关键行在 `runAllowedTools`。模型只提出工具调用请求，真正执行的是程序。程序可以拒绝越权参数、限制工作目录、记录日志，也可以要求人工确认。删掉这层校验，Agent 的行动就只剩模型自觉。

读完这一节，你应该能做的事：给一个循环补上最大步数、工具校验和观察回传，避免它无限跑下去。

三、Workflow 和 Agent：控制权在谁手里

BOUNDARY

Workflow 是程序预先写好的路径。输入进来后，系统按固定步骤走：分类、检索、生成、校验。模型可以参与某个节点，但节点顺序由代码决定。Agent 则把一部分下一步选择权交给模型：它可以根据观察结果决定查资料、问用户、调用工具或停止。

两者没有高低之分。流程固定、输入边界清楚、验收标准明确时，Workflow 更稳定也更便宜。任务需要中途推理、环境反馈和灵活处理异常时，Agent 才有价值。很多产品标成 Agent，其实只是工作流加了几个 LLM 节点；这不丢人，误判才麻烦。

判断规则很直接：如果你能提前画出固定流程图，并且所有异常分支都能列完，用 Workflow。只要下一步动作必须由模型结合观察动态决定，就进入 Agent 设计空间，同时也要准备评测、追踪和护栏。

维度WorkflowAgent
控制权代码预定义步骤模型基于状态选择下一步
工具顺序固定或有限分支按观察动态选择
调试方式看节点日志看完整轨迹、工具结果和状态变化
适合场景流程稳定、验收清楚需要探索、异常处理或多轮决策
主要风险流程写死，灵活性不足成本、轮数、权限和可观测性更难控

读完这一节，你应该能做的事：看到一个 AI 系统时，先问“下一步由代码决定，还是由模型决定”。

四、状态：Agent 不是只靠聊天记录工作

STATE

状态是 Agent 当前知道什么、做过什么、还缺什么。短期状态可以放在 messages 里，长期状态通常要外化到文件、数据库、任务记录或记忆系统。只把所有东西塞进上下文窗口，看似简单，实际会带来噪声、重复、遗忘和不可恢复。

一个任务状态至少包含目标、已知事实、已执行动作、工具结果、待确认事项、剩余步数和失败记录。对长任务来说，还要有显式任务文件或 checkpoint。这样会话中断后，Agent 才能从断点恢复，而不是重新猜自己做过什么。

状态和消息不是同一个东西。消息是发给模型看的上下文，状态是系统维护的任务事实。很多状态不必原样注入模型，例如内部日志、权限字段、工具耗时。给模型看的内容应该经过筛选，足够做下一步判断，但不要把框架内部字段都暴露进去。

任务状态：帮助恢复和决策JSON
{
  "goal": "整理 ReAct 论文学习资料",
  "phase": "writing_exercises",
  "knownFacts": ["ReAct 交错生成 reasoning trace 和 action"],
  "done": ["读取摘要", "整理 Thought/Action/Observation"],
  "next": ["补错误诊断题", "检查导出按钮"],
  "stopIf": ["来源不足", "达到最大步数", "需要人工确认"]
}

读完这一节，你应该能做的事：为一个会跑多轮的 Agent 写出最小状态表，而不是只写 prompt。

五、工具：模型提出请求，程序负责执行

TOOLS

工具是 Agent 接触外部世界的接口。一个工具描述要包含用途、参数、返回值、限制和失败处理。只写“搜索资料”不够，最好写清什么时候使用、什么时候不要使用、需要哪些参数、结果可能为空时怎么办。

程序侧要做参数校验、权限检查和日志记录。模型可以说想删除文件，但程序不应该直接相信它。越接近外部副作用的工具，越需要白名单、工作区隔离、人工确认或 dry-run。读文件和搜索通常可以自动，发消息、转账、删除、提交代码就要更谨慎。

很多 Agent 失误不是模型太弱，而是工具定义让模型猜。工具名字含混、参数重复、描述没有排除条件，都会让模型选错工具。调试 Agent 的第一步经常不是换大模型，而是重写工具描述和返回结构。

正确示例：带限制的工具描述Tool Schema
{
  "name": "search_docs",
  "description": "Search official docs when the answer depends on current product behavior. Do not use when the user already provided the exact source text.",
  "input_schema": {
    "query": "string, concise search query",
    "product": "string, product area if known"
  },
  "returns": "Top passages with source URL, date, and confidence note. Empty result is valid."
}
错误示例：描述太空，模型只能猜Bad Tool
{
  "name": "do_search",
  "description": "Search things.",
  "input_schema": {
    "text": "string"
  }
}

读完这一节，你应该能做的事：为一个工具写清用途、禁用场景、参数和失败返回。

六、停止条件：能停下来才算可控

STOP

停止条件是 Agent 工程里最容易被低估的部分。一个循环可以因为信息足够而停止，也可以因为缺少必要参数、触发权限边界、达到最大步数、遇到连续失败、预算耗尽或需要人工确认而停止。停止不是失败，它是系统承认当前边界。

没有停止条件的 Agent 会反复搜索、重复问用户、在同一个错误上重试，甚至把一个小任务拖成大任务。停止条件要写在程序和 prompt 两边：程序用硬限制兜底，prompt 告诉模型哪些情况应该收束。

判断句：只要一个 Agent 可能执行超过一轮，就必须说明它何时继续、何时停、何时交给人。否则它只是看起来自动，实际上不可预测。

信息足够

观察结果已经能回答用户目标，不需要继续调用工具。

条件不足

缺少订单号、权限、文件路径或用户确认，继续执行会猜。

边界触发

达到最大步数、预算耗尽、连续失败或涉及高风险副作用。

读完这一节，你应该能做的事：为一个 Agent 写出至少 5 条停止条件，并把它们分成正常完成、需要澄清和风险兜底。

七、完整链路：从目标到可审计结果

CHAIN

把前面的内容串起来，一个可用 Agent 的主线是：用户给目标，系统注入边界和工具；模型根据状态选择下一步；程序执行被允许的工具；观察结果回到状态；模型继续或停止；日志保存轨迹；评测检查最终环境和执行过程。

这条链路里，LLM 负责解释状态和提出下一步，Harness 负责执行、状态、权限、追踪和验证。Harness 可以理解为模型外面的运行框架。它越清楚，Agent 越稳定。只盯着模型输出，很容易忽略真正决定成败的工程边界。

学完这一块，你应该能拿任何一个所谓 Agent 产品做初步拆解：它的目标入口在哪里，工具有哪些，观察如何回传，状态放在哪里，谁负责停止，哪些动作需要确认。

观察图里的回环：工具结果回到状态，而不是直接变成最终回答。状态更新后，模型才决定继续还是停下。

术语、易混概念和误区

REVIEW

术语表

术语直觉解释容易误解的地方
ReAct推理和行动交错进行的方法。不是让模型无限展示思考，而是让计划能随观察更新。
Thought模型对当前状态和下一步的解释。漂亮的 Thought 不是正确性的证据。
Action模型请求程序执行的工具或环境动作。Action 必须被 schema、权限和工具边界约束。
Observation工具或环境返回的结果。Observation 不是装饰，它决定下一轮是否改计划。
Harness模型外面的运行框架。很多失败来自 Harness 缺失，不是模型本身不会推理。
Stop condition循环结束或转人工的条件。没有停止条件，就没有可控 Agent。

易混概念

概念对准确区分判断提醒
Workflow / AgentWorkflow 路径固定；Agent 让模型动态选择下一步。先看控制权在代码还是模型。
Action / ObservationAction 是请求执行；Observation 是执行后的事实反馈。下一轮必须基于 Observation 更新。
Messages / StateMessages 是给模型看的上下文；State 是系统维护的任务事实。不要把所有状态都塞进聊天文本。
工具调用 / Agent工具调用是单次能力；Agent 是围绕目标多轮选择动作。有工具不等于有自主循环。

常见误区

误区为什么错修法
有工具就是 Agent工具调用只是能力入口，还要看是否有动态循环。检查是否基于观察选择下一步。
思考越长越可靠长推理可能把错误包装得更像真的。看观察和验证，不只看推理文本。
Agent 应该尽量自主自主度越高，评测、安全和恢复成本越高。从低风险、可验证动作开始放权。
停止是失败缺权限、缺信息或风险过高时，停止是正确行为。把停止写成正常分支。
换强模型就能解决循环问题工具描述、状态和日志差时，强模型也会乱走。先查工具、状态、停止条件和评测。
