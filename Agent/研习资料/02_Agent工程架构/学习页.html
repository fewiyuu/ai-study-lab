Agent Architecture · Harness

把循环放进可靠的运行框架

上一页把 Agent 看成“思考、行动、观察”的循环。工程上，循环只是最小骨架。真正决定稳定性的，是模型外面的运行框架：上下文怎么放，工具怎么描述，状态怎么保存，记忆怎么注入，长任务怎么恢复，失败怎么追踪。

Harness→ 上下文→ 工具→ 记忆→ 状态恢复→ 追踪验证

学完本页，你应该能从“我要做一个 Agent”拆出工程清单，而不是只写一个 prompt 或一个 while 循环。

学习目标

  1. 说清 Harness、上下文工程、工具设计和记忆层各自负责什么。
  2. 设计一个能跨 session 恢复的 Agent 任务状态。
  3. 识别工具描述、上下文污染和记忆注入失控带来的故障。
  4. 用 OpenClaw 这类架构线索理解渠道、消息、Agent 和工具的分层。

一、Harness：模型外面的运行框架

HARNESS

Harness 可以理解为包住模型的工程框架。它负责把目标、上下文、工具、权限、日志、评测和恢复机制组织起来。模型只是在这个框架里做判断：下一步该调用哪个工具，是否需要澄清，是否可以停下。

很多 Agent 调不稳时，第一反应是换更强模型。但 Tw93 的工程总结里有一个很有用的判断：工具定义、验证测试和 Harness 质量经常比模型大小更影响成功率。模型输出只是一部分，真正让系统可控的是执行层、状态层和验证层。

如果 Harness 不清楚，prompt 会变得越来越长，工具会散落在代码各处，权限会靠模型自觉，日志只剩最终回答。出错时，你很难判断是模型判断错、工具描述错、状态丢了，还是评测没覆盖。

结构草图：Harness 负责的不是一句 promptTypeScript-like
type AgentHarness = {
  buildContext(taskState): Message[];
  selectTools(taskState): ToolDefinition[];
  executeTool(call): Promise<ToolResult>;
  persistState(taskState): Promise<void>;
  trace(event): void;
  shouldStop(taskState): StopDecision;
};

读完这一节，你应该能做的事：把“模型负责什么”和“Harness 负责什么”分开说清楚。

二、上下文工程：不是窗口越长越好

CONTEXT

上下文窗口能放很多内容，但注意力不是免费的。无关材料越多，关键信号越容易被稀释。工程实践里常把这种现象叫 Context Rot：上下文越堆越长，模型看似知道更多，决策质量反而下降。

上下文要分层。常驻层放身份、硬约束和项目规则；按需层放 Skills、领域资料和工具说明；运行时层放当前任务事实、最近观察和待确认事项；持久层放历史记录、记忆和文件。每层只放自己该放的东西。

一个判断句很实用：如果某段信息不是当前决策必须使用，就不要默认塞进 messages。给模型的信息应该短、硬、可执行。长材料可以通过检索、文件接口或按需加载进入，不要把所有资料一次性压进 prompt。

常驻层

身份、硬规则、绝对禁止项。要短，不能变成百科全书。

按需层

Skills、领域知识、长文档。触发时加载，不用时不占上下文。

运行时层

当前目标、最近观察、待确认事项和剩余预算。

读完这一节，你应该能做的事：给一个 Agent 任务划分常驻、按需、运行时和持久信息。

三、工具设计：描述决定模型会不会选对

TOOLS

工具不是函数清单，而是 Agent 的能力边界。一个工具的名字、描述、参数、返回值和错误消息都会影响模型选择。工具描述只写能做什么还不够，还要写什么时候不该用、失败时返回什么、哪些参数必须来自用户确认。

工具数量也会影响稳定性。工具太多，模型会混淆相近能力；工具跨多个领域，模型容易把文件工具用于数据库问题，把搜索工具用于已知事实。工具设计可以逐步演进：先用少量粗工具验证任务，再把常见误用拆成更明确的专用工具。

工具返回要服务下一步决策。返回“成功”太粗，返回整页原文又太吵。更好的返回包含关键结果、错误类型、可选下一步和是否需要人工确认。工具消息如果携带框架内部字段，发给 LLM 前也要过滤，避免模型看到一堆不该解释的技术字段。

正确示例：工具返回服务下一步决策Tool Result
{
  "tool": "read_project_file",
  "status": "ok",
  "summary": "Found AGENTS.md with Chinese-only response rule.",
  "relevantLines": [1, 2, 3],
  "nextOptions": ["continue_editing", "ask_user_if_conflict"],
  "requiresConfirmation": false
}
错误示例：返回噪声太大Bad Result
{
  "status": "success",
  "html": "<html>...完整网页、导航、广告、脚本、无关评论...</html>"
}

读完这一节，你应该能做的事：为工具写清用途、禁用场景、返回结构和失败处理。

四、记忆系统：把稳定事实和操作流程分开放

MEMORY

Agent 没有天然的跨会话连续性。会话结束后，messages 通常就没了。要让 Agent 下次还能接着干，记忆必须作为基础设施单独设计，而不是事后往 prompt 里补几句。

可以按用途分四类：工作记忆放当前任务；程序性记忆放怎么做某事，例如 Skills；情景记忆放发生过什么，例如 JSONL 会话历史；语义记忆放稳定偏好和事实，例如 MEMORY.md。不同记忆的加载方式不同，不应该全部常驻。

记忆最大的风险是注入失控。稳定事实要短，操作流程要按需加载，历史记录要可检索，错误记忆要能回退。否则 Agent 会把旧事实当新规则，或者把一次偶然偏好永久写入系统提示。

记忆类型放什么加载方式
工作记忆当前任务目标、观察、待办运行时注入
程序性记忆Skills、操作步骤、领域规范按需加载
情景记忆会话历史、执行轨迹、失败记录检索或摘要后加载
语义记忆稳定偏好、长期事实、项目约定短而稳定地注入

读完这一节，你应该能做的事：判断一条信息应该放进当前上下文、Skill、历史记录还是长期记忆。

五、长任务恢复：状态显式写出来

RESUME

长任务不能只靠聊天历史。一个跨 session 任务至少要记录目标、当前阶段、已完成动作、未解决问题、下一步、关键文件和失败记录。这样 Agent 重启后可以先读状态，再继续任务。

显式状态还可以降低重复劳动。比如代码 Agent 已经跑过测试、发现两个失败、修了第一个，如果状态没有写下来，下一次可能重新跑全流程。状态文件不需要很复杂，但要稳定、结构化、可检查。

判断句：只要任务可能超过一次对话，就要外化状态。把状态写成文件或数据库记录，不是繁琐，而是给 Agent 留一个能回来的位置。

任务状态：可恢复记录JSON
{
  "taskId": "agent-course-2026-06-06",
  "goal": "逐个制作 Agent 学习资料",
  "phase": "writing_02_architecture",
  "done": ["01_ReAct与Agent循环"],
  "next": ["检查 02 页面", "继续 03 设计模式"],
  "openQuestions": ["Medium 原文是否需要再补充镜像来源"],
  "lastVerified": "01 页面有 20 道题和导出区"
}

读完这一节，你应该能做的事：为一个会跨天执行的 Agent 任务写出可恢复状态。

六、OpenClaw 线索：用分层架构落地 Agent

OPENCLAW

Tw93 用 OpenClaw 展示 Agent 如何落地。你不需要先记住某个框架的所有细节，先抓住分层思路：渠道接入、消息总线、Agent 核心、工具执行、持久化和可观测性应该分开。分开以后，换聊天渠道不会改 Agent 核心，换工具也不会改消息入口。

消息总线的价值是隔离渠道和 Agent。用户可能来自飞书、网页、命令行或后台任务，但 Agent 接收到的是统一消息。cron 和 heartbeat 这类主动触发也可以走同一套入口，让 Agent 不只被动响应用户。

这类架构的关键不是名字，而是把控制流、工具、副作用和状态拆清楚。你以后看任何 Agent 框架，都可以问：消息入口在哪里？工具执行在哪里？状态存在哪里？追踪事件在哪里？安全边界在哪里？

观察图里的分离：渠道变化、工具变化和记忆变化都不应该直接把 Agent 核心搅乱。

七、完整链路：稳定性来自分工清楚

CHAIN

把工程架构串起来：Harness 负责把模型放在受控环境里；上下文工程负责让模型只看到当前有用的信息；工具设计负责让行动明确可校验；记忆系统负责跨会话连续性；任务状态负责长任务恢复；追踪和评测负责确认结果。

这条链路里，prompt 只是其中一环。prompt 写得再好，如果工具返回噪声、状态丢失、记忆混乱、权限不清，Agent 仍然会表现不稳。工程上要优先把可观察、可恢复、可验证的结构搭起来。

学完这一块，你应该能从“我要做一个 Agent”的模糊想法，拆出最小工程清单：状态表、工具表、上下文层、停止条件、日志字段和验证方式。

术语、易混概念和误区

REVIEW

术语表

术语直觉解释容易误解的地方
Harness模型外面的运行框架，负责工具、状态、权限、追踪和验证。不要把 Harness 当成 prompt 本身。
Context Rot上下文噪声变多后，模型决策质量下降。长上下文不是免管理上下文。
Skills按需加载的程序性记忆，告诉 Agent 怎么做某类任务。描述符可以常驻，完整内容不应全部常驻。
Task state长任务的显式进度记录。它服务恢复和审计，不是流水账。
Semantic memory长期稳定事实或偏好。写入要谨慎，避免把临时情况固化。
Message bus把不同渠道消息转换成统一 Agent 输入的中间层。它用于隔离入口和核心逻辑。

易混概念

概念对准确区分判断提醒
常驻上下文 / 按需加载常驻放硬约束；按需加载放长知识和流程。默认少，触发时再加。
工作记忆 / 长期记忆工作记忆服务当前任务；长期记忆服务跨会话连续性。不要把临时状态写成长期偏好。
工具 handler / 工具描述handler 是真实执行代码；描述是给模型选择工具的说明。两者都要测试。
Trace / Final answerTrace 记录执行过程；Final answer 面向用户。只看最终回答无法复盘 Agent 为什么这么做。

常见误区

误区为什么错修法
上下文越长越好长上下文会带来噪声和注意力稀释。按频率、稳定性和当前任务相关性分层加载。
记忆就是向量数据库记忆按用途设计，可能是文件、摘要、历史、Skills 或数据库。先问记忆解决什么问题，再选存储。
工具返回越详细越好过长返回会污染上下文。返回关键字段、错误类型和下一步建议。
状态可以让模型自己记跨会话或长任务不能依赖模型临时上下文。显式写任务状态。
架构越复杂越专业过早引入多层系统会增加维护成本。先从最小可运行 Harness 做起。
