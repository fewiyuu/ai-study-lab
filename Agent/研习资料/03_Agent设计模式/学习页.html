按任务形状选择 Agent 设计模式

设计模式不是术语收藏。它的作用是帮你在动手前判断：这个任务能不能线性拆？是否需要先分类？能否并行探索？是否要一个编排器分派子任务？是否需要评估器反复挑错？如果任务形状没看清，系统很容易被做得过度复杂。

线性→ 路由→ 并行→ 编排→ 评估优化→ 自主循环

学完本页，你应该能根据任务风险、分解方式、验收标准和成本，选出足够简单的模式，而不是一上来就写复杂 Agent。

学习目标

  1. 说清 Prompt Chaining、Routing、Parallelization、Orchestrator-Workers、Evaluator-Optimizer 的差别。
  2. 判断一个任务该用 Workflow 模式还是更 Agentic 的模式。
  3. 识别过度设计：能线性解决却上多 Agent，或需要探索却写死流程。
  4. 用 ReAct、Plan-and-Execute、ReWOO、LATS 的区别理解自主度递增。

一、为什么要先学模式：少一点玄学，多一点选型

WHY

很多 Agent 讨论容易直接跳到“模型能不能自己做事”。工程上更实用的问题是：控制流应该由谁决定，任务能否拆成稳定步骤，错误能否自动检查，成本是否可接受。设计模式就是回答这些问题的工具。

模式不是框架，也不是固定代码。它更像任务形状的标签。比如线性任务适合 Prompt Chaining；输入类型差异大适合 Routing；搜索空间大适合 Parallelization；任务需要动态拆分适合 Orchestrator-Workers；输出需要反复检查适合 Evaluator-Optimizer。

判断句：优先选择能完成任务的最简单模式。能用两步工作流解决，就不要上自主 Agent；需要模型根据观察探索时，再把控制权逐步交给模型。

读完这一节，你应该能做的事：先描述任务形状，再谈选哪种 Agent 模式。

二、Prompt Chaining：把线性任务拆成几步

CHAINING

Prompt Chaining 是最朴素也最常用的模式：把一个复杂任务拆成顺序步骤，每一步 LLM 处理上一步输出。比如“先提取要点，再生成大纲，再写正文，再做格式检查”。它不是 Agent 循环，因为步骤顺序由代码固定。

它适合输入边界清楚、步骤稳定、每一步都能检查的任务。优点是可控、便宜、容易调试。缺点是遇到异常时灵活性不足，如果中间结果错了，后面步骤可能沿着错误继续。

设计时要给每一步明确输入和输出，不要让上一步生成一堆自由文本交给下一步猜。中间最好加检查点，例如 JSON schema 校验、长度限制、引用检查或人工确认。

流程示例：文章生成链Workflow
extract_notes(source) -> bullet_points
build_outline(bullet_points) -> outline
draft_article(outline) -> draft
check_claims(draft, source) -> verified_draft
polish_style(verified_draft) -> final_article

读完这一节，你应该能做的事：把一个线性任务拆成 3-5 个有明确输入输出的步骤。

三、Routing：先判断类型，再走专用路径

ROUTING

Routing 的直觉是先把输入分流。不同问题走不同处理路径：账单问题走账单工具，技术问题走技术知识库，简单问题走便宜模型，复杂问题走强模型。它解决的是“一套流程处理所有输入太粗”的问题。

路由适合输入类型差异明显的场景。关键不是分类标签多，而是分类后真的会走不同策略。如果分类完还是同一套处理，那路由只是装饰。路由也要有兜底：分类不确定时可以走人工、走通用流程，或向用户澄清。

路由器本身要小心误分。高风险任务不要只靠一次模型分类直接执行副作用动作。可以先分类，再让下游流程做权限和参数检查。

简单咨询

走轻量模型和 FAQ，低成本快速回答。

技术故障

走日志工具、知识库和排查模板。

退款申请

走订单工具和人工确认，不直接执行。

读完这一节，你应该能做的事：为一个客服或知识库系统设计 3 条有差异的路由路径。

四、Parallelization：并行探索更大的空间

PARALLEL

Parallelization 有两种常见用法。第一种是分段法：把任务拆成互不依赖的部分并发处理，例如研究一个主题时分别查论文、产品、竞品和风险。第二种是投票法：对同一问题跑多个候选答案，再汇总或取共识。

它适合搜索、研究、审查和高风险判断。优点是覆盖更广，能减少单一路径漏掉信息。代价是 token 和时间成本增加，汇总阶段也会变难。并行不是越多越好，子任务必须能独立完成，否则并行结果会互相缺上下文。

判断句：如果子任务之间需要频繁来回同步，就不要硬并行。能独立查、独立评、最后汇总，才适合并行。

流程示例：研究任务并行拆分Plan
lead_agent:
  facet_1 = "论文和原始方法"
  facet_2 = "工程实现和框架"
  facet_3 = "失败案例和风险"
  run_in_parallel([facet_1, facet_2, facet_3])
  synthesize(findings)

读完这一节，你应该能做的事：判断一个任务能不能拆成互不依赖的并行子任务。

五、Orchestrator-Workers：一个编排器分派多个工人

ORCHESTRATOR

Orchestrator-Workers 模式里，编排器负责理解目标、拆分任务、分派给 worker，并汇总结果。worker 只做局部工作。它比普通并行更动态，因为编排器可能根据任务内容决定分几块、派给谁、是否补一轮。

这个模式适合任务结构不固定，但子任务能相对独立完成的场景。例如代码仓库审查，编排器可以把前端、后端、测试、文档分给不同 worker；研究任务可以把不同资料源分给不同 worker。

风险在于交接损耗。worker 拿不到足够上下文会做偏，编排器收到结果太粗会汇总错。要给 worker 明确输入、输出格式和验收标准，不要只说“帮我看看”。

读完这一节，你应该能做的事：写出编排器给 worker 的输入、输出和验收标准。

六、Evaluator-Optimizer：让评估器推动改进

EVAL-OPT

Evaluator-Optimizer 模式包含两个角色：生成器产出结果，评估器检查结果并指出问题，然后生成器根据反馈修改。这适合输出质量可被规则或标准检查的任务，例如代码、文案、结构化报告、事实引用和安全策略。

评估器不能只说“还不错”。它要有明确标准：是否满足格式、是否引用来源、是否覆盖边界、是否通过测试、是否存在风险。标准越具体，优化越有效。否则循环会变成空泛互评，成本上升但质量不一定变好。

这个模式的边界是：评估器本身也可能偷懒或误判。高风险任务最好用自动测试、规则检查、人审或外部工具辅助，而不是只靠另一个 LLM 点头。

错误示例：评估器太空Bad Review
"请检查一下是否可以。"
"看起来不错，可以发布。"
正确示例：评估器有具体标准Review Criteria
检查：
1. 是否有来源链接。
2. 是否区分事实和推断。
3. 是否覆盖失败边界。
4. 是否有至少 3 个反例。
5. 是否能通过格式校验。

读完这一节，你应该能做的事：给一个生成任务写出可执行的评估标准。

七、Agentic 模式：自主度增加，成本和风险也增加

AGENTIC

ReAct、Plan-and-Execute、ReWOO、LATS 这些模式都更接近 Agentic 设计。ReAct 强调思考、行动、观察交错；Plan-and-Execute 先生成计划再逐步执行；ReWOO 倾向先规划工具调用和变量依赖，减少反复让模型思考；LATS 把搜索树、评估和回溯引入决策。

这些模式不是越高级越该用。它们增加了探索能力，也增加了 token 成本、实现复杂度和评测难度。任务简单、路径稳定时，用 Prompt Chaining 或 Routing 往往更好。只有当任务需要动态探索、工具结果会改变计划、候选路径需要比较时，Agentic 模式才值得上。

选型时可以问四个问题：任务是否需要中途观察？是否有多个可能路径？错误能否自动验证？高风险动作是否有护栏？如果这些问题答不清，先回到更简单的 Workflow 模式。

模式直觉适合场景
ReAct边想边做，观察修正计划需要工具反馈的问答、网页任务、环境交互
Plan-and-Execute先列计划，再逐步执行步骤能先粗分，但执行中仍需调整
ReWOO先规划工具和变量依赖，再执行工具调用依赖清楚、想减少反复推理成本
LATS像搜索树一样探索、评估、回溯候选路径多，需要比较和反思的复杂任务

读完这一节，你应该能做的事：解释为什么“更 Agentic”不一定更适合当前任务。

术语、易混概念和误区

REVIEW

术语表

术语直觉解释容易误解的地方
Prompt Chaining把任务拆成固定顺序的多个 LLM 步骤。它通常是 Workflow，不是自主 Agent。
Routing先分类输入，再走专用流程。分类后必须真的有不同处理路径。
Parallelization并行处理独立子任务或多个候选答案。子任务如果强依赖，就不适合硬并行。
Orchestrator-Workers编排器动态拆任务并分派给 worker。交接上下文不足会带来损耗。
Evaluator-Optimizer评估器指出问题，生成器根据反馈改进。评估标准必须具体，否则只是互相点头。
LATS用搜索、评估和回溯处理复杂决策。成本和实现复杂度更高，不适合简单任务。

易混概念

概念对准确区分判断提醒
Prompt Chaining / ReAct前者步骤固定；后者根据观察继续选择动作。看中途是否需要环境反馈改变路径。
Routing / OrchestratorRouting 选择预设路径；Orchestrator 动态拆分任务。看任务结构是否提前固定。
Parallelization / Multi-Agent并行可以只是多个调用；Multi-Agent 还涉及角色、上下文和协作协议。不要把“并发”自动叫多 Agent。
Evaluator / Human review评估器可自动检查；人审适合高风险和模糊标准。高风险场景不要只靠 LLM judge。

常见误区

误区为什么错修法
模式越复杂越强复杂模式会增加成本、延迟和调试难度。选能完成任务的最简单模式。
所有任务都适合 Agent流程固定时 Workflow 更稳定。先判断控制权是否需要交给模型。
并行越多越全面子任务不独立时，并行会制造重复和冲突。先确认能独立完成，再并行。
评估器说通过就可信评估器也会偷懒或误判。写具体标准，必要时加测试和人审。
路由只要分类准确就行分类后没有不同动作，路由没有价值。把分类标签和处理流程绑定。
