Evaluation · Trace · Safety

让 Agent 的行为可验证、可追踪、可限制

Agent 做得对不对，不能只看它最后说了什么。它可能说得很自信，但工具没跑、文件没改、测试没过，甚至在错误方向上高效执行。评测、追踪和安全边界就是为了回答三件事：它完成了吗？它为什么这么做？它有没有越界？

任务→ 运行轨迹→ 环境结果→ 评分器→ 追踪→ 安全边界

学完本页，你应该能为一个 Agent 任务写出最小评测方案、Trace 字段和安全护栏。

学习目标

  1. 解释 task、trial、grader、transcript、outcome 分别是什么。
  2. 区分只看最终回答的评测和检查环境结果的 Agent 评测。
  3. 设计 Trace 字段，能复盘每一步工具调用和状态变化。
  4. 写出权限、工作区隔离、人工确认和在线采样的基本规则。

一、为什么要先修评测：否则改了也不知道好坏

WHY

很多团队会先堆 prompt、工具和模型，评测放到最后。结果是：改了 prompt，不知道是否变好；换了模型，不知道是否退化；加了工具，不知道是否引入新误用。Agent 的波动比单轮问答更大，没有评测就只剩感觉。

Agent 评测比普通模型评测复杂，因为它不是“输入一句话，输出一个答案”就结束。它会多轮调用工具，改变环境状态，生成中间轨迹。最终评分不能只看回答文本，还要看它是否真的完成任务，环境有没有被正确修改，过程有没有越权。

判断句：目标清楚、结果可自动验证，是最适合 Agent 发挥的区域。目标模糊但自动化很强，系统会高效地往错误方向跑；目标清楚但验收靠人，吞吐量天花板会卡在人审速度上。

读完这一节，你应该能做的事：在改 Agent 前，先写出“怎么知道它变好了”。

二、评测结构：task、trial、grader

EVAL

Agent 评测的最小单位可以拆成三件事：task 是测什么，trial 是一次运行，grader 是怎么评分。一个 evaluation suite 就是一组 task。每个 task 可以跑多次 trial，因为 LLM 有随机性，工具环境也可能波动。

除了最终回答，还要保存 transcript 和 outcome。transcript 是完整执行记录，包括模型消息、工具调用、工具结果和错误；outcome 是环境最终状态，例如文件是否正确修改、测试是否通过、数据库是否写入。只看 transcript 可能知道它怎么想，只看 outcome 可能不知道为什么错，两者都要有。

Agent 评测要尽量贴近真实任务。比如代码 Agent 的评分，不是看它说“我修好了”，而是看测试是否通过、diff 是否合理、是否触碰了不该改的文件。

评测样例：任务、运行和评分器YAML-like
task:
  id: fix-login-timeout
  goal: 修复登录超时后没有重试的问题
  setup: checkout fixture repo
  success:
    - npm test passes
    - only auth module changed
trial:
  agent: coding-agent-v2
  model: claude-opus
grader:
  checks:
    - run tests
    - inspect changed files
    - compare behavior with expected scenario

读完这一节，你应该能做的事：给一个 Agent 任务写出 task、trial 和 grader。

三、评分器：规则、模型和人工各有位置

GRADER

评分器可以分三类。规则评分器最硬，例如测试通过、JSON schema 合格、文件存在、数值在范围内。LLM judge 能处理开放文本，但要有清晰 rubric。人工审查适合高风险、主观或规则还没稳定的场景。

好的评分器要避免“只看起来不错”。它要明确检查什么、如何失败、失败后记录什么。代码任务优先用自动测试；事实任务优先检查来源和引用；写作任务可以用 rubric，但最好也检查长度、结构、禁用词和事实覆盖。

评分器本身也会出错。LLM judge 可能被流畅文本骗过，人工也会疲劳。重要任务可以组合多个评分器：先规则筛，再 LLM 检查语义，最后抽样人审。

规则评分

测试、schema、权限、文件 diff。硬，但覆盖不到所有语义。

LLM judge

适合开放文本和综合判断。必须给 rubric。

人工审查

适合高风险和模糊标准。成本高，吞吐有限。

读完这一节，你应该能做的事：为一个任务选择至少两类评分器，并说明取舍。

四、Trace：记录过程，不只记录结果

TRACE

Trace 是 Agent 的执行轨迹。它应该记录每一步的输入、模型决定、工具调用、参数、结果、耗时、错误、状态变化和停止原因。Trace 的作用不是凑日志，而是让你能复盘：它为什么走到这里，哪一步开始偏了，工具返回是否误导了模型。

Trace 要结构化。只保存一大段聊天文本，后续很难查询。至少要有 step id、event type、tool name、tool input、tool output summary、state delta、cost、latency、error 和 safety decision。这样才能按工具、错误类型、成本和风险做统计。

追踪和评测要连起来。失败任务不只保存分数，还要保存导致失败的轨迹。否则你知道 pass rate 降了，却不知道为什么降。

Trace 事件：最小字段JSON
{
  "runId": "trial-042",
  "step": 3,
  "event": "tool_call",
  "tool": "shell",
  "inputSummary": "npm test",
  "outputSummary": "2 tests failed in auth retry",
  "stateDelta": {"phase": "diagnosing_failure"},
  "latencyMs": 4280,
  "costUsd": 0.014,
  "safety": "allowed"
}

读完这一节，你应该能做的事：列出一条 Agent trace 至少应该记录的字段。

五、线上采样：别只在离线集里自我感觉良好

ONLINE

离线评测能防回归，但真实用户输入会更乱。线上采样是在真实运行中抽取一部分任务做复盘：看成功率、失败类型、工具误用、成本、用户接管和安全拦截。它能发现离线数据集没有覆盖的边界。

线上采样不能侵犯隐私，也不能把敏感内容随便送给评审。需要脱敏、分级、抽样比例和权限控制。高风险动作要记录更多元数据，低风险问答可以只保留摘要和评分。

一个实用做法是把事件流作为底座：所有 tool_call、tool_result、state_update、stop_reason、安全拦截都作为事件发出。评测、告警和分析都消费同一条事件流，避免每个模块各记各的。

读完这一节，你应该能做的事：设计一个线上采样方案，说明抽什么、谁能看、怎么脱敏。

六、安全边界：先于功能，不是上线后补

SAFETY

Agent 一旦能调用 shell、发消息、写数据库、提交代码或操作订单，就有外部副作用。安全边界要先于功能。至少要回答：谁能用，能在哪用，能做什么，做了什么如何追踪，哪些动作需要确认。

常见护栏包括白名单授权、工作区隔离、路径检查、危险命令拦截、dry-run、人工确认、预算限制、速率限制和审计日志。模型不能自己决定越过这些边界。程序侧要硬拦，prompt 只能辅助。

工作区隔离尤其重要。shell 工具必须解析真实路径，防止 `../` 或符号链接越出工作目录。删除、提交、推送、发消息、退款这类动作要有更高确认级别。

安全护栏：路径检查示例TypeScript-like
const WORKSPACE = path.resolve("/workspace/project");

async function assertInsideWorkspace(cwd) {
  const real = await fs.realpath(path.resolve(cwd ?? WORKSPACE));
  const rel = path.relative(WORKSPACE, real);
  if (rel.startsWith("..") || path.isAbsolute(rel)) {
    throw new Error("outside workspace");
  }
  return real;
}

读完这一节，你应该能做的事：给一个有 shell 权限的 Agent 写出最小安全护栏。

七、常见反模式：看起来自动，实际不可控

ANTI-PATTERN

第一个反模式是没有评测就改 Agent。表现是每次改 prompt 都凭感觉，线上失败后也无法复现。修法是先建小评测集，哪怕只有 10 个真实任务，也比没有基线强。

第二个反模式是只看最终回答。Agent 可能没有调用该调用的工具，也可能做了危险动作后给出漂亮总结。修法是保存 transcript 和 outcome，并让 grader 同时检查过程与结果。

第三个反模式是把安全交给模型自觉。只在系统提示里写“不要做危险操作”不够。真正的边界要由程序执行：权限、路径、确认、日志、预算都必须是硬约束。

反模式后果修法
无评测改 prompt不知道变好还是变坏建立 task suite 和回归记录
只看最终回答过程越权或工具没跑也看不出检查 transcript 和 outcome
安全只写在 prompt模型失误时边界失效程序侧硬拦和审计
评分器太空LLM judge 只会说“不错”写具体 rubric 和负例

读完这一节，你应该能做的事：看到一个 Agent 方案时，指出它缺评测、缺追踪还是缺安全边界。

术语、易混概念和误区

REVIEW

术语表

术语直觉解释容易误解的地方
Task评测要完成的具体任务。任务描述要有可验证目标。
Trial同一个任务的一次运行。一次通过不代表稳定。
Grader给运行结果打分的规则、模型或人工流程。评分器本身也需要检查。
Transcript完整执行记录，包括消息、工具调用和结果。它不是最终答案，而是复盘材料。
Outcome环境最终状态，例如测试、文件、数据库变化。Agent 说完成不等于 outcome 正确。
Trace结构化事件流，用来追踪每一步。一大段聊天文本不等于可查询 Trace。

易混概念

概念对准确区分判断提醒
Transcript / Outcome前者记录过程；后者记录环境最终状态。Agent 评测通常两者都要看。
Offline eval / Online sampling离线防回归；线上发现真实输入边界。线上采样要脱敏和权限控制。
Rule grader / LLM judge规则硬但覆盖有限；LLM judge 灵活但会误判。重要任务可以组合评分器。
Prompt safety / Program safetyPrompt 提醒模型；程序硬性拦截越界动作。高风险边界必须由程序执行。

常见误区

误区为什么错修法
只看最终回答无法知道工具是否执行、过程是否越权。保存 transcript 和 outcome。
没有评测先改 prompt无法判断变化是提升还是退化。先建小任务集和回归记录。
LLM judge 说通过就可信模型评估器也会偷懒或被流畅文本骗过。加规则检查、负例和抽样人审。
安全靠系统提示模型可能忽略或误解提示。程序侧做权限、路径和确认。
线上数据随便拿来评审可能泄露隐私和敏感信息。脱敏、分级、限制访问。
