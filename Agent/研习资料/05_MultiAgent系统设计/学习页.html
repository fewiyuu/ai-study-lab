什么时候拆成多个 Agent

Multi-Agent 听起来很自然：规划一个、执行一个、审查一个，似乎更像团队。但工程经验并不总是这样。每多一个 Agent，就多一份 prompt、一次上下文交接、一个失败点和更多 token。很多任务用一个工具配置清楚的单 Agent 反而更稳。

单 Agent→ 上下文保护→ 并行探索→ 专业化→ 验证子 Agent

学完本页，你应该能判断一个任务是否真的需要 Multi-Agent，并写出拆分后的角色、上下文边界和验证方式。

学习目标

  1. 解释为什么 Claude 文章建议先从单 Agent 开始。
  2. 掌握 Multi-Agent 值得使用的三类场景：上下文保护、并行化、专业化。
  3. 设计验证子 Agent，避免“早胜利”问题。
  4. 识别上下文交接、电话游戏、工具过多和协作协议缺失带来的风险。

一、先从单 Agent 开始

SINGLE

一个设计得好的单 Agent，加上合适工具，能完成的事情通常比直觉更多。多 Agent 系统会引入协调成本：每个 Agent 都要维护 prompt、工具、上下文和错误处理。交接时信息会丢，多个 Agent 之间还可能互相放大幻觉。

Claude 文章里一个很清楚的提醒是：多 Agent 实现通常会比单 Agent 多用很多 token，原因包括重复上下文、协调消息和中间汇总。成本增加不是绝对坏事，但必须换来明确收益。

判断句：如果一个单 Agent 能通过更好的工具、上下文压缩和评测解决问题，就先不要拆。只有当单 Agent 的上下文、工具集或搜索空间真的到达边界，再考虑多 Agent。

读完这一节，你应该能做的事：解释为什么 Multi-Agent 不是默认升级路线。

二、决策框架：收益必须大于协调成本

DECISION

拆分前先问四个问题。第一，单 Agent 是否因为上下文污染而明显变差？第二，任务能否并行探索更大搜索空间？第三，不同子任务是否需要明显不同的工具、提示或专业知识？第四，是否需要一个独立验证者检查主 Agent 的工作？

如果答案都是否，多 Agent 多半只是增加复杂度。反过来，如果其中一项非常明确，例如检索任务需要多个方向并行搜索，或者代码生成需要独立测试者，那么多 Agent 才可能值回成本。

另一个判断是耦合度。需要频繁共享状态、来回同步理解的工作，最好留在同一个 Agent 里。能独立完成、只返回精简结果的工作，更适合拆给子 Agent。

信号适合拆不适合拆
上下文子任务资料很多且互不相关所有信息都要共享理解
搜索空间多个方向可独立探索下一步强依赖上一轮细节
工具集不同角色需要不同工具工具集高度重叠
验证可黑盒检查主 Agent 输出验证必须参与每一步决策

读完这一节，你应该能做的事：用四个问题判断是否拆成多个 Agent。

三、上下文保护：让子任务在干净上下文里工作

CONTEXT

上下文保护是 Multi-Agent 最常见的正当理由。主 Agent 的上下文里如果塞进大量订单、日志、网页或检索结果，后续推理会被污染。子 Agent 可以在独立上下文里处理这些材料，只把精简发现返回给主 Agent。

例如客服 Agent 需要查订单历史，同时还要诊断技术问题。订单查询可能产生大量无关 token。如果全塞进主上下文，会影响技术诊断。让一个子 Agent 专门查订单并返回“和当前问题有关的 3 条事实”，主 Agent 就能保持上下文干净。

上下文保护的前提是子任务边界清楚。子 Agent 不能返回整堆原文，否则只是把污染推迟。它应该返回压缩后的事实、来源、置信度和未解决问题。

子 Agent 返回：精简发现JSON
{
  "subtask": "order_lookup",
  "findings": [
    "订单 #123 已付款",
    "仓库缺货导致延迟",
    "预计 2026-06-08 补货"
  ],
  "irrelevantDataDropped": true,
  "sources": ["order_api"],
  "needsMainAgentDecision": "是否向用户提供替代方案"
}

读完这一节，你应该能做的事：设计一个子 Agent，让它过滤噪声而不是转发噪声。

四、并行化：用多个 Agent 覆盖更大搜索空间

PARALLEL

并行化适合研究、搜索、审查和信息探索。一个 lead Agent 可以把问题拆成多个面向，让子 Agent 并行查找不同资料源，最后汇总。这样能覆盖更多信息空间，减少单一路径漏掉关键资料的风险。

代价也很明确：token 更多，总耗时可能更长，汇总更难。并行子 Agent 的任务必须能独立完成。若每个子 Agent 都需要不断问其他 Agent 细节，就说明任务没有被拆对。

并行输出要标准化。每个子 Agent 返回发现、来源、反例、置信度和未解决问题。否则 lead Agent 会花大量上下文解析风格不同的报告。

论文方向

查原始方法、实验和局限。

工程方向

查框架、实践和工具链。

风险方向

查失败案例、成本和安全边界。

读完这一节，你应该能做的事：把一个研究任务拆成 3 个互不依赖的并行 Agent 子任务。

五、专业化：不同工具集和提示词适合不同角色

SPECIALIZE

专业化适合工具太多或领域差异太大时。一个 Agent 同时拿着数据库、浏览器、文件系统、支付、邮件和代码工具，容易选错工具。拆成专门角色后，每个 Agent 的工具集更窄，提示词更清楚，误用概率会下降。

专业化不只是给角色起名字。每个角色要有输入边界、工具列表、输出格式和不能做的事。比如 Research Agent 可以搜索和整理来源，但不能写入数据库；Code Agent 可以改文件和跑测试，但不能发送外部邮件；Verifier 可以跑测试和检查结果，但不能直接修代码。

如果两个角色工具高度重叠、上下文完全相同，只是名字不同，那就没有真正专业化。它们只会增加协调成本。

角色工具禁止项输出
Research Agent搜索、网页提取、引用整理不改项目文件带来源发现
Code Agent读写文件、运行测试不推送、不发消息diff 和测试结果
Verifier测试、静态检查、规则评分不修改主输出通过/失败和证据

读完这一节，你应该能做的事：为 3 个专业化 Agent 写出工具边界和输出格式。

六、验证子 Agent：最稳定的多 Agent 模式之一

VERIFY

验证子 Agent 的职责很窄：检查主 Agent 的工作。它不需要完整参与任务规划，也不需要理解所有背景，只要拿到输出、标准和工具，黑盒验证即可。这减少了上下文交接损耗，也避免“电话游戏”。

这个模式适合代码测试、引用检查、格式检查、政策合规和输出验收。Verifier 成功的关键是标准具体：运行完整测试，而不是“看看能不能用”；检查反例和边界，而不是跑一两个顺手用例。

常见失败是 early victory：验证子 Agent 跑了一个简单测试就宣布通过。修法是写明完整检查清单、负例、边界案例和必须报告的失败。

验证子 Agent 指令：避免早胜利Prompt
你是验证子 Agent。不要修改代码。
必须执行：
1. 运行完整测试套件。
2. 至少检查 2 个边界输入。
3. 检查 diff 是否触碰无关文件。
4. 报告所有失败，不要只报告第一个通过项。
只有全部检查通过，才能写“通过”。

读完这一节，你应该能做的事：为一个主 Agent 输出设计独立验证子 Agent。

七、失败模式：交接、幻觉和协作协议

FAILURE

多 Agent 最常见的问题是交接损耗。主 Agent 给子 Agent 的任务不清，子 Agent 返回太长或太粗，主 Agent 汇总时丢掉关键限定条件。每一次交接都是一次压缩，压缩就可能丢信息。

第二个问题是幻觉互相放大。一个子 Agent 提供了未经验证的结论，主 Agent 把它当事实，另一个 Agent 再基于这个“事实”继续扩写，错误会变得更像真的。解决方式是让返回结果带来源、置信度和未验证标记。

第三个问题是协作协议缺失。谁能派任务，谁能改文件，谁负责最终答案，失败如何升级，深度限制是多少，都要写清楚。没有协议，多 Agent 会像一群没有边界的聊天模型，而不是一个系统。

失败模式表现修法
交接损耗子 Agent 不知道真正目标明确输入、输出和验收标准
幻觉放大未验证结论被多次引用来源、置信度和验证状态必填
工具重叠多个 Agent 抢同一类动作缩窄工具集和角色边界
无限递归子 Agent 又生成更多子 Agent设置深度限制和派发权限

读完这一节，你应该能做的事：为 Multi-Agent 系统写出协作协议和失败兜底。

术语、易混概念和误区

REVIEW

术语表

术语直觉解释容易误解的地方
Multi-Agent多个 Agent 分工协作的系统。多个 LLM 调用并不自动等于 Multi-Agent。
Context protection把噪声子任务放到独立上下文里处理。子 Agent 不能把噪声原样返回。
Parallelization多个 Agent 并行探索不同方向。子任务必须相对独立。
Specialization用不同工具集和提示词处理不同职责。只改名字不算专业化。
Verification subagent专门验证主 Agent 输出的子 Agent。它要有具体检查清单，避免早胜利。
Early victory验证者只做少量检查就宣布通过。需要完整测试、负例和边界检查。

易混概念

概念对准确区分判断提醒
单 Agent / Multi-Agent单 Agent 上下文统一；多 Agent 分工但有交接成本。先看单 Agent 是否已到边界。
并行化 / 专业化并行化为覆盖更大空间；专业化为缩窄工具和提示。一个解决搜索空间，一个解决角色边界。
子 Agent / 工具工具执行确定函数；子 Agent 会用模型判断和压缩信息。简单确定动作优先做成工具。
验证子 Agent / 编排器验证者检查结果；编排器拆分和分派任务。不要让验证者参与主任务生成。

常见误区

误区为什么错修法
多 Agent 默认更强它会增加 token、延迟和交接失败点。先证明收益大于协调成本。
角色越多越像团队没有边界的角色只会互相干扰。写清工具、输入、输出和禁止项。
子 Agent 返回越多越好会把上下文污染带回主 Agent。返回精简发现、来源和置信度。
验证者说通过就结束容易出现 early victory。要求完整测试、边界和负例。
所有复杂任务都要拆高耦合任务拆开会频繁同步。耦合高时留在同一 Agent。
