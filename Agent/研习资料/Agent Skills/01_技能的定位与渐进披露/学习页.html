先定这页的目标

学习目标

这页只做一件事：帮你判断一个任务值不值得做成 Skill，并且知道一旦决定要做，说明、资源和执行逻辑应该怎么分层。它还会把 Skill 和 Prompt、Tool、MCP、subagent 分开，避免把不同层的东西混在一起。

为什么有些重复任务该做成 Skill

decision

先拿一个具体例子：团队每周都要把会议纪要整理成决策、行动项、负责人和待确认问题。这个任务值得做成 Skill，因为它重复出现、输出形状稳定、常常需要固定模板，还经常要补参考材料。每次重新在聊天里解释一遍，智能体都要重新猜一遍，这就是上下文浪费。

观察到的信号更像什么该不该做 Skill

这里的判断重点它有没有稳定的输入、稳定的输出和稳定的检查方式。

渐进披露先解决上下文预算

progressive

渐进披露的直觉很简单：先只让智能体看到能决定“要不要加载”的信息，后面再逐层展开。它把资料放在该出现的时候再出现。这样做的结果是，智能体先判断相关性，再读正文，再按需读资源，不会在不相关的细节里先耗掉上下文。

把相邻概念分开，才不会互相抢位置

role

这一页最容易混的，是 Skill、Prompt、Tool、MCP 和 subagent。它们都和“让智能体做事”有关，但负责的层不一样。你只要记住一件事：谁负责临时约束，谁负责稳定流程，谁负责执行动作，谁负责连接外部系统，谁负责独立分工，这五层就不会打架。

Prompt约束本次回答长期沉淀流程这次怎么说

Skill沉淀可复用做法替代执行工具以后都这么做

Tool执行动作或查询保存业务手册现在去做

MCP把外部系统接进来解释完整业务流程连上哪里

Subagent隔离或委托复杂任务替代可复用知识包交给谁独立做

把责任切开，主文件才不会发胖

层级分配

当用户说“把这类任务做成 Skill”时，真正要分的是责任，不是词汇表。Prompt 处理本次临时约束，Skill 记录稳定流程，Tool 执行动作或查询，MCP 连接外部系统，subagent 处理需要隔离上下文的复杂任务。把这五层混在一起，最常见的结果就是主文件塞满，或者工具被拿来写流程说明。

“这次先按这个口径来”Prompt不要把临时要求写成长期规则

“以后每次都按这个流程”Skill要流程散在聊天里，无法复用

“现在去取/写/跑一个动作”Tool不要把执行器当说明书写

“连到外部系统去读写”MCP只写连接边界把连接协议和业务规则混成一段

“这件事要隔离上下文单独处理”Subagent不要把委托任务的细节全塞进主说明

代码
if need only bound this answer:
 use Prompt
elif need stable repeatable workflow:
 use Skill
elif need run a command or query:
 use Tool
elif need connect to an external system:
 use MCP
else:
 delegate to subagent

这张图放在写作前面有个具体好处：你在拆一个新任务时，不会先问它叫不叫 Skill，而是先问它要留下哪种责任。只要这一步做对，后面的 SKILL.md 结构、模板拆分和资源下沉都会顺很多。要是这一步做错了，后面即使有漂亮的目录，也只是给错层级整理文件。

如果你拿不准，就先看“现在要不要立即执行、以后会不会重复、有没有外部系统、是否要隔离上下文”这四个问题。它们比术语更好用。

把一条请求拆成四个判断

状态账本

还是用“把会议稿整理成决策、行动项、负责人和待确认问题”这条请求做锚点。你先看它会不会再来、输出是不是固定、有没有共享模板或长规则、到底该由哪一层负责。前两项决定值不值得做成 Skill，后两项决定内容该放进主说明还是下沉到按需层。

  1. 识别重复性任务出现频率、相同场景、固定意图先判断是否值得沉淀把一次性任务误做成 Skill先问“下周还会不会来”
  2. 识别输出形状字段、顺序、必填项、禁猜项写进主说明触发后仍然输出散乱列出最少必须项
  3. 识别共享资源模板、词表、长规则、样例下沉到 references/ 或 assets/主文件变厚、上下文被吃掉把长细节拆出去
  4. 识别执行责任现在要说清、以后都照做，还是要动外部系统决定 Prompt / Skill / Tool / MCP / subagent职责混在一起按责任切层
代码
if task repeats:
 if output_shape_is_fixed:
 make the decision rule explicit
 if shared_resources_exist:
 split main instructions from long material
 if action belongs to a tool or external system:
 hand it to the right layer
 else:
 keep it as prompt or one-off workflow
else:
 do not promote it just because it looks neat

这个顺序的用处很实际：它逼你先看事实，再做归类。只要这四步里有一步跳过了，最常见的后果就明确了。要么把一次性的改写做成长期包，后面维护越来越重；要么把该沉淀的流程继续留在对话里，每次都要重新解释。前者浪费维护成本，后者浪费上下文。

错误症状说明更稳的做法

把一次请求落成文件

workflow

再看一个更完整的落地场景：用户只说“每周把会议稿整理成决策、行动项、负责人”。这时候你不是先写模板，也先判断它会不会再来、输出是否固定、有没有共享资源。确认这三点以后，才把流程写进 SKILL.md，把长规则下沉到 references/，把最终模板放进 assets/。

“每周都要整理一次”重复性决定是否做成 Skill把一次性任务做成长期包

“输出要固定成决策/待办/负责人”输出形状SKILL.md触发了也不知道怎么收尾

“同一套模板每次都要用”共享资源references/ + assets/主文件被长模板塞满

“这一步要不要交给工具去做”责任层级Tool / MCP / subagent执行器和说明书混在一起

代码
if task repeats and output_shape_is_fixed:
 choose Skill
 write trigger + workflow in SKILL.md
 move long rules to references/
 move the final template to assets/
else:
 keep it as prompt or one-off workflow

这个链路的价值在于，它把“该不该沉淀”和“该放哪一层”连在一起。要是你把模板先放进主文件，后面每次改流程都要翻一大段文字；要是你把该沉淀的任务继续留在聊天里，每次又都要重讲一遍规则。两种情况都会让维护成本上升。

学到这里，你已经可以直接把一个新的重复任务拆成四句：先判重复性，再判输出形状，再判资源，最后判责任层级。下一步去练习里把同样的拆分做一遍。

一个最小例子，够看清结构就行

minimal

还是拿“会议纪要整理”这个重复任务做例子。这个任务适合沉淀成 Skill，因为输入稳定、输出稳定、规则稳定，而且经常要读同一套参考模板。下面这个最小骨架为了让你看见：description 负责触发，SKILL.md 负责流程，references/ 负责长细节，assets/ 负责模板。

代码
---
name: meeting-notes
description: Turn meeting transcripts into decisions, action items, owners, and open questions. Use when the user asks to summarize meeting notes or extract next actions.
---

## Workflow

1. Read the transcript or notes.
2. Extract decisions, action items, owners, and open questions.
3. If the project template is fixed, read references/field-schema.md.
4. Use assets/meeting-note-template.md for the final format.

## Gotchas

- Do not invent owners or due dates.
- If a due date is missing, mark it as 待确认.
- Keep the output in one consistent template.

SKILL.md触发、流程、边界、默认选择长规则、长模板、全部样例

references/字段说明、错误处理、长规则最终排版模板

assets/模板、静态样例、复用资源会频繁变动的逻辑

把一次真实请求追一遍

decision

还是用会议纪要这个锚点。用户说：“把这段会议稿整理成决策和待办。” 这句话不够决定是否要做 Skill，必须再问三个问题：是不是反复出现、输出是否固定、有没有共享模板或参考资料。把这三个问题串起来，才能知道这一类稳定流程。

三个失败场景，直接看症状

failure

很多 Skill “看上去能触发，但用起来总差一口气”。下面这三个场景最常见：描述太窄、描述太宽、主文件太满。你可以把它们当成一次小型故障实验室，看症状、看原因、看修法。

故障定位伪代码复制

把判断规则收紧成一条线

decision

现在把前面的内容收成一句可执行的规则：一个任务值得做成 Skill，当它反复出现、输出形状稳定，并且会反复使用同一套资源或规则。只要其中一条不成立，就先别急着做 Skill。临时提示、单次命令、一次性说明，通常更便宜，也更不容易把课程的结构搞乱。

这个式子只是在提醒你：重复性、输出固定度和资源共享度要同时站得住，沉淀成 Skill 的回报才会高于维护成本。

代码
repeatable = task happens again and again
fixed_shape = output fields are stable
shared_resources = same template or long rule is reused

if repeatable and fixed_shape and shared_resources:
 choose Skill
else:
 keep it as prompt, tool, or one-off workflow

这个规则为了让判断落到能检查的东西上。你以后看到一个任务，先问自己这三条，再决定是不是继续往下做成 Skill。

常见误区，基本都卡在分层上

common

把 Skill 当成“更长的提示词”还是靠临场猜，流程不稳写触发、步骤、边界、资源和验证

把所有资料都塞进主文件上下文浪费，重点被挤掉把细节放进 references/ 和 assets/
