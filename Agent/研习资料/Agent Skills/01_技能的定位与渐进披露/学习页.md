# 先定这页的目标 {#goals}
<!-- tag: 学习目标 -->

这页只做一件事：帮你判断一个任务值不值得做成 Skill，并且知道一旦决定要做，说明、资源和执行逻辑应该怎么分层。它还会把 Skill 和 Prompt、Tool、MCP、subagent 分开，避免把不同层的东西混在一起。

读完后能做什么对应动作下一步


判断任务是否值得沉淀看重复频率、固定格式、资源依赖决定是继续写提示，还是做 Skill
分清能力层级区分 Prompt、Skill、Tool、MCP、subagent把动作、说明、连接、分工放回各自位置
读懂源材料边界把课程索引、课程说明、公共来源和邻近单元拼起来写出可复用的资料底稿

# 为什么有些重复任务该做成 Skill {#problem}
<!-- tag: decision frame -->

先拿一个具体例子：团队每周都要把会议纪要整理成决策、行动项、负责人和待确认问题。这个任务值得做成 Skill，因为它重复出现、输出形状稳定、常常需要固定模板，还经常要补参考材料。每次重新在聊天里解释一遍，智能体都要重新猜一遍，这就是上下文浪费。

把它做成 Skill 以后，用户不用每次重述全部规则，智能体也不用把整个流程临时记在对话里。任务一旦有稳定意图、稳定输出和稳定检查点，Skill 就开始比长提示更稳。

观察到的信号更像什么该不该做 Skill


每周都要做，规则几乎不变可复用流程是
输出格式固定，字段不能乱模板化产物是
要反复调用同一套参考资料资源依赖是
只是一句临时改写一次性提示否

这里的判断重点它有没有稳定的输入、稳定的输出和稳定的检查方式。

# 渐进披露先解决上下文预算 {#progressive}
<!-- tag: progressive disclosure -->

渐进披露的直觉很简单：先只让智能体看到能决定“要不要加载”的信息，后面再逐层展开。它把资料放在该出现的时候再出现。这样做的结果是，智能体先判断相关性，再读正文，再按需读资源，不会在不相关的细节里先耗掉上下文。

对一个 Skill 来说，最先被读到的是 `name` 和 `description`；任务匹配后再看 `SKILL.md`；只有真的需要模板、长规则、长文档或脚本时，才去读 `references/`、`assets/` 和 `scripts/`。这个顺序一乱，Skill 就会变成大段说明书，读得多，记不住，也不稳定。

层级读什么写什么读错后会怎样


发现层`name`、`description`“这是不是我要用的 Skill”漏触发或误触发
主说明层`SKILL.md`流程、边界、默认选择触发了但不知道怎么做
按需层`references/`、`assets/`、`scripts/`细节、模板、可执行逻辑主文件过长，或者资源根本没被读到

加载顺序示意复制

```text
1. 先看 name 和 description
2. 任务相关时再读 SKILL.md
3. 需要模板、长规则或脚本时，再读 references / assets / scripts
4. 任务结束后只保留结果，不把全部资料塞回上下文
```

# 把相邻概念分开，才不会互相抢位置 {#roles}
<!-- tag: role split -->

这一页最容易混的，是 Skill、Prompt、Tool、MCP 和 subagent。它们都和“让智能体做事”有关，但负责的层不一样。你只要记住一件事：谁负责临时约束，谁负责稳定流程，谁负责执行动作，谁负责连接外部系统，谁负责独立分工，这五层就不会打架。

概念主要作用不该拿来做什么本页的判断句


Prompt约束本次回答长期沉淀流程这次怎么说
Skill沉淀可复用做法替代执行工具以后都这么做
Tool执行动作或查询保存业务手册现在去做
MCP把外部系统接进来解释完整业务流程连上哪里
Subagent隔离或委托复杂任务替代可复用知识包交给谁独立做

如果一个问题只需要一次性约束，就用 Prompt；如果它需要稳定流程，就考虑 Skill；如果核心是动作，就看 Tool；如果核心是连接系统，就看 MCP；如果是分工和隔离，就看 subagent。

# 从资料到 Skill，要先把边界写清楚 {#source-flow}
<!-- tag: source boundary -->

这门课有明确来源。课程索引告诉你整门课的顺序，`README.md` 给出课程入口和配套文件，`资源熔炼.md` 记录了公共来源和拆块结果，邻近三节则把今天这页之后的细节接住。这一页先判断该不该做成 Skill，下一页再把判断写成 `SKILL.md` 的结构。

来源支持这页的哪部分可见产出


[课程索引](../index.html)课程顺序、单元角色、邻近衔接课程起点与下一步
[README.md](../README.md)学习顺序和配套文件课程入口与公共标签
[资源熔炼.md](../资源熔炼.md)公共来源、资料边界、拆块结果资料底稿的骨架
[02 SKILL.md 结构与触发](../02_SKILL.md结构与触发/学习页.html)下一步把思路落到文件结构从判断进入结构
[03 脚本资源与上下文经济](../03_脚本资源与上下文经济/学习页.html)后续把稳定流程放进脚本从说明进入执行

这四类来源各自回答不同问题：课程索引负责课程内部的顺序，`README.md` 负责读者从哪里进入，`资源熔炼.md` 负责公开来源和拆块边界，邻近单元负责前后衔接。以后如果上游内容更新，先按 2026-06-20 这个访问日期回查，再决定要不要刷新这页。

如果这一步不先写边界，后面的 Skill 很容易变成“看起来很完整，但没有告诉你该从哪里开始”的东西。

# 把责任切开，主文件才不会发胖 {#handoff-map}
<!-- tag: 层级分配 -->

当用户说“把这类任务做成 Skill”时，真正要分的是责任，不是词汇表。`Prompt` 处理本次临时约束，`Skill` 记录稳定流程，`Tool` 执行动作或查询，`MCP` 连接外部系统，`subagent` 处理需要隔离上下文的复杂任务。把这五层混在一起，最常见的结果就是主文件塞满，或者工具被拿来写流程说明。

信号该放哪层主文件里要不要写放错后的症状


“这次先按这个口径来”Prompt不要把临时要求写成长期规则
“以后每次都按这个流程”Skill要流程散在聊天里，无法复用
“现在去取/写/跑一个动作”Tool不要把执行器当说明书写
“连到外部系统去读写”MCP只写连接边界把连接协议和业务规则混成一段
“这件事要隔离上下文单独处理”Subagent不要把委托任务的细节全塞进主说明

分层检查伪代码复制

```text
if need only bound this answer:
 use Prompt
elif need stable repeatable workflow:
 use Skill
elif need run a command or query:
 use Tool
elif need connect to an external system:
 use MCP
else:
 delegate to subagent
```

这张图放在写作前面有个具体好处：你在拆一个新任务时，不会先问它叫不叫 Skill，而是先问它要留下哪种责任。只要这一步做对，后面的 `SKILL.md` 结构、模板拆分和资源下沉都会顺很多。要是这一步做错了，后面即使有漂亮的目录，也只是给错层级整理文件。

如果你拿不准，就先看“现在要不要立即执行、以后会不会重复、有没有外部系统、是否要隔离上下文”这四个问题。它们比术语更好用。

# 把一条请求拆成四个判断 {#decision-ladder}
<!-- tag: 状态账本 -->

还是用“把会议稿整理成决策、行动项、负责人和待确认问题”这条请求做锚点。你先看它会不会再来、输出是不是固定、有没有共享模板或长规则、到底该由哪一层负责。前两项决定值不值得做成 Skill，后两项决定内容该放进主说明还是下沉到按需层。

判断步读什么写什么错位症状下一步


1. 识别重复性任务出现频率、相同场景、固定意图先判断是否值得沉淀把一次性任务误做成 Skill先问“下周还会不会来”
2. 识别输出形状字段、顺序、必填项、禁猜项写进主说明触发后仍然输出散乱列出最少必须项
3. 识别共享资源模板、词表、长规则、样例下沉到 `references/` 或 `assets/`主文件变厚、上下文被吃掉把长细节拆出去
4. 识别执行责任现在要说清、以后都照做，还是要动外部系统决定 Prompt / Skill / Tool / MCP / subagent职责混在一起按责任切层

判定伪代码复制

```text
if task repeats:
 if output_shape_is_fixed:
 make the decision rule explicit
 if shared_resources_exist:
 split main instructions from long material
 if action belongs to a tool or external system:
 hand it to the right layer
 else:
 keep it as prompt or one-off workflow
else:
 do not promote it just because it looks neat
```

这个顺序的用处很实际：它逼你先看事实，再做归类。只要这四步里有一步跳过了，最常见的后果就明确了。要么把一次性的改写做成长期包，后面维护越来越重；要么把该沉淀的流程继续留在对话里，每次都要重新解释。前者浪费维护成本，后者浪费上下文。

错误症状说明更稳的做法


页里只有漂亮定义，没有判断动作学完还是不会选层回到重复性、输出形状、共享资源这三步
主文件又长又散所有细节都挤在一起把长规则和样例下沉到按需层
一遇到任务就先谈工具还没确认值不值得沉淀先判重复性，再谈实现

# 把一次请求落成文件 {#workflow-demo}
<!-- tag: workflow demo -->

再看一个更完整的落地场景：用户只说“每周把会议稿整理成决策、行动项、负责人”。这时候你不是先写模板，也先判断它会不会再来、输出是否固定、有没有共享资源。确认这三点以后，才把流程写进 `SKILL.md`，把长规则下沉到 `references/`，把最终模板放进 `assets/`。

用户说法你先判断什么落到哪一个文件错放后的后果


“每周都要整理一次”重复性决定是否做成 Skill把一次性任务做成长期包
“输出要固定成决策/待办/负责人”输出形状`SKILL.md`触发了也不知道怎么收尾
“同一套模板每次都要用”共享资源`references/` + `assets/`主文件被长模板塞满
“这一步要不要交给工具去做”责任层级Tool / MCP / subagent执行器和说明书混在一起

最小落地伪代码复制

```text
if task repeats and output_shape_is_fixed:
 choose Skill
 write trigger + workflow in SKILL.md
 move long rules to references/
 move the final template to assets/
else:
 keep it as prompt or one-off workflow
```

这个链路的价值在于，它把“该不该沉淀”和“该放哪一层”连在一起。要是你把模板先放进主文件，后面每次改流程都要翻一大段文字；要是你把该沉淀的任务继续留在聊天里，每次又都要重讲一遍规则。两种情况都会让维护成本上升。

学到这里，你已经可以直接把一个新的重复任务拆成四句：先判重复性，再判输出形状，再判资源，最后判责任层级。下一步去练习里把同样的拆分做一遍。

# 一个最小例子，够看清结构就行 {#example}
<!-- tag: minimal skill -->

还是拿“会议纪要整理”这个重复任务做例子。这个任务适合沉淀成 Skill，因为输入稳定、输出稳定、规则稳定，而且经常要读同一套参考模板。下面这个最小骨架为了让你看见：`description` 负责触发，`SKILL.md` 负责流程，`references/` 负责长细节，`assets/` 负责模板。

最小 SKILL.md 骨架复制

```text
---
name: meeting-notes
description: Turn meeting transcripts into decisions, action items, owners, and open questions. Use when the user asks to summarize meeting notes or extract next actions.
---

## Workflow

1. Read the transcript or notes.
2. Extract decisions, action items, owners, and open questions.
3. If the project template is fixed, read references/field-schema.md.
4. Use assets/meeting-note-template.md for the final format.

## Gotchas

- Do not invent owners or due dates.
- If a due date is missing, mark it as 待确认.
- Keep the output in one consistent template.
```

层放什么不要放什么


`SKILL.md`触发、流程、边界、默认选择长规则、长模板、全部样例
`references/`字段说明、错误处理、长规则最终排版模板
`assets/`模板、静态样例、复用资源会频繁变动的逻辑

这个骨架里最重要的不是字数，而是分层。主文件负责“怎么决定”，按需层负责“有什么细节”，模板负责“长什么样”。只要你在别的重复任务里也能说出这一句，分层就算真的学会了。

# 把一次真实请求追一遍 {#trace}
<!-- tag: decision tracer -->

还是用会议纪要这个锚点。用户说：“把这段会议稿整理成决策和待办。” 这句话不够决定是否要做 Skill，必须再问三个问题：是不是反复出现、输出是否固定、有没有共享模板或参考资料。把这三个问题串起来，才能知道这一类稳定流程。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

检查步读什么写什么错了会怎样


请求意图用户说了什么是不是重复任务把一次性需求当成 Skill
输出形状字段、顺序、固定格式主文件里应写的约束触发后仍然输出散乱

这张追踪表的作用，让你在看到一个重复任务时，先看它的输入、输出和资源，再决定要不要把它做成 Skill。只要这三步过了，后面的文件结构就好写很多。反过来，如果你在第一步就已经能看出它只是一次性改写，就不要继续往下推；那样只会多造一个维护对象。

很多 Skill “看上去能触发，但用起来总差一口气”。下面这三个场景最常见：描述太窄、描述太宽、主文件太满。你可以把它们当成一次小型故障实验室，看症状、看原因、看修法。

看这种输出时，重点“哪一层出问题”。描述层出问题，就去改触发；主文件过满，就去拆资源；规则写得太散，就去补边界和默认值。

```text

 move long rules to references/

 else:
 keep the current split and write the next unit
```

# 把判断规则收紧成一条线 {#decision-rules}
<!-- tag: decision rule -->

现在把前面的内容收成一句可执行的规则：一个任务值得做成 Skill，当它反复出现、输出形状稳定，并且会反复使用同一套资源或规则。只要其中一条不成立，就先别急着做 Skill。临时提示、单次命令、一次性说明，通常更便宜，也更不容易把课程的结构搞乱。

$$
 \text{Skill 值得做} \approx \text{重复性} \times \text{输出固定度} \times \text{资源共享度}
 $$
 这个式子只是在提醒你：重复性、输出固定度和资源共享度要同时站得住，沉淀成 Skill 的回报才会高于维护成本。

判断伪代码复制

```text
repeatable = task happens again and again
fixed_shape = output fields are stable
shared_resources = same template or long rule is reused

if repeatable and fixed_shape and shared_resources:
 choose Skill
else:
 keep it as prompt, tool, or one-off workflow
```

这个规则为了让判断落到能检查的东西上。你以后看到一个任务，先问自己这三条，再决定是不是继续往下做成 Skill。

# 常见误区，基本都卡在分层上 {#mistakes}
<!-- tag: common mistakes -->

先别追 `SKILL.md` 语法细节、脚本接口、评估器和跨平台打包；这四件事分别在后面的单元里再回头。现在只看一个问题：你是不是把不同层的内容混在一起了。

误区后果更稳的改法


把 Skill 当成“更长的提示词”还是靠临场猜，流程不稳写触发、步骤、边界、资源和验证
把所有资料都塞进主文件上下文浪费，重点被挤掉把细节放进 `references/` 和 `assets/`
看到重复就立刻做 Skill很多一次性任务也被抬高了先看是否有稳定输入、输出和检查点
把 Tool、MCP、subagent 和 Skill 混为一类职责重叠，后面更难维护先问“临时约束、稳定流程、动作、连接、分工”哪一个是主问题

判断一个 Skill 好不好，要看它有没有让智能体少猜、少问、少绕路。

# 下一步去哪里 {#bridge}
<!-- tag: bridge -->

这页结束后，最自然的下一步就是去看 `SKILL.md` 的结构与触发。那一页会把今天这里已经分好的边界，落到 `name`、`description`、正文、`references/` 和 `assets/` 的具体写法上。换句话说，这一页决定“要不要做 Skill”，下一页决定“写成什么样的 Skill”。

现在已经明白下一节要补什么现在别先追的东西

先做选择和判断题，把角色分开；再做短答，把边界说完整。导出的记录会保留每组标题和检查重点，方便你回头复盘。

### 1 choice

**Q:** 1. 下面哪种任务最像该做成 Skill？

- [ ] 只改一次的邮件措辞

- [ ] 对
- [ ] 错

Answer: 错

**Q:** 3. Skill 适合沉淀的是可复用的 ______ 和资源。


### 4 debug-checklist

**Q:** 4. 下面哪些信号说明值得做成 Skill？

- [ ] 任务反复出现
- [ ] 输出格式固定
- [ ] 每次都要重新解释一遍
- [ ] 需要脚本或参考资料支撑

Explain: 应选：任务反复出现、输出格式固定、需要脚本或参考资料支撑。不要把一次性任务、临时改写当成 Skill。真正的判断点是“是否有稳定流程和固定检查点”。

### 6 choice

**Q:** 6. 下面哪种说法最符合渐进披露？

- [ ] 先把所有资源塞入上下文
- [ ] 先看 name 和 description，相关时再读 SKILL.md，需要时再读 references / assets / scripts

### 7 true/false

**Q:** 7. 如果一个 Skill 的主文件必须装下全部细节，说明它没有利用渐进披露。

- [ ] 对
- [ ] 错


### 8 synthesis

**Q:** 8. 为什么渐进披露能省上下文？

Explain: 好的回答要说清：先用 `name` 和 `description` 判断相关性，相关后再读 `SKILL.md`，需要时才打开 `references/`、`assets/` 和 `scripts/`。这样可以把不相关的细节留在后面，避免一开始就占满上下文，还能让主文件只写流程和边界，不被长文档挤爆。

**Q:** 10. 哪个最像 MCP？

- [ ] 把外部系统接进来
- [ ] 沉淀固定流程和边界
- [ ] 临时约束一次回答

Answer: 把外部系统接进来

### 11 true/false

**Q:** 11. Subagent 和 Skill 是一类东西：都是为了保存可复用流程。

- [ ] 对
- [ ] 错


### 12 synthesis

**Q:** 12. 用 2-4 句话把 Prompt、Skill、Tool、MCP 和 subagent 分开。

Explain: 好的回答要把层级说清：Prompt 是这一次的临时约束，Skill 是可复用流程和做法，Tool 是执行动作或查询，MCP 是把外部系统接进来，subagent 是把复杂任务交给独立上下文处理。只要能把这五层说分明，就说明你没有把“怎么说、怎么做、接哪儿、交给谁”混成一团。

### 15 synthesis

**Q:** 15. 把“会议纪要整理 Skill”拆成目录、`SKILL.md`、`references/` 和 `assets/` 三层。

Explain: 目录负责把这项能力放进一个稳定位置；`SKILL.md` 负责触发说明、流程、边界和默认选择；`references/` 放长规则、字段说明、错误处理和细节；`assets/` 放模板、样例和静态资源。答题时如果把“怎么做”和“长细节”分开了，就说明你已经开始按 Skill 的方式组织资料了。

### 16 synthesis

**Q:** 16. 用 6-9 句话讲清楚本页的核心链路。

Explain: 好的回答应该从一个重复任务开始，先判断它是否有稳定输入、稳定输出和稳定检查点；然后说明为什么这种任务值得做成 Skill，而不是继续堆临时提示；再说渐进披露如何先用 `name` 和 `description` 判断相关性，再逐层打开 `SKILL.md`、`references/`、`assets/` 和 `scripts/`；最后收束到下一节的 `SKILL.md` 结构与触发。只要这条链路连起来，你就在按用途分层。
