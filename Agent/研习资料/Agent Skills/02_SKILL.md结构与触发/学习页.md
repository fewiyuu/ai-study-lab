# 先定这页的目标 {#goals}
<!-- tag: 学习目标 -->

这一页只解决一件事：让你拿起一个现成的 Skill 目录时，能快速看出它是被正确安装了，还是卡在目录、frontmatter、正文或支持文件这几层中的某一层。这样你不会把“没触发”简单归因为模型不行，而是能先定位到文件层面的原因。

你要检查的对象你要看什么常见症状


目录放置位置、目录名、是否属于当前作用域命令找不到，或技能只在一部分目录里可见
frontmatter`name`、`description` 和触发控制项技能能被找到，但不会自动加载，或总在不该用的时候冒出来
正文与支持文件说明是否够短、支持文件是否有明确读取条件主文件变厚、细节没有按需加载


---

# Skill 先是一个目录，再是一个文件 {#file-shape}
<!-- tag: 文件外形 -->

在 Claude Code 里，Skill 的第一层一个目录。普通布局下，目录名就是你输入的命令名；`SKILL.md` 提供主说明；其他文件只是补充材料。官方文档把它拆成三层：启动时先读元数据，触发后再读正文，需要时才打开支持文件。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

层级客户端先读什么这层负责什么错位后会怎样


元数据层`name`、`description`告诉 Claude 这是什么、什么时候用相关任务没触发，或过宽误触发
说明层`SKILL.md` 正文告诉 Claude 具体做法、边界和默认选择触发了，但下一步不知道怎么走
资源层支持文件把长规则、样例和模板留在需要时再读的位置主文件发胖，细节又不够用

最小目录骨架复制

```text
summarize-changes/
└── SKILL.md

report-generator/
├── SKILL.md
├── references/
│ └── field-map.md
└── templates/
 └── report.md
```

这页先把“能不能被发现、能不能被触发、细节放在哪”拆开。脚本怎么跑、评估怎么做、发布怎么管，放到后两节再看。

# 技能放在哪，Claude 才看得见 {#where-live}
<!-- tag: discovery -->

官方文档把技能发现分成几个位置：个人技能目录、项目 `.claude/skills/`、当前工作树下嵌套的 `.claude/skills/`，以及通过 `--add-dir` 或 `/add-dir` 加进来的额外目录。Claude 还会在会话中检测这些目录的变化，所以你改完文件后通常不需要重新搭一遍流程才能看到结果。

位置谁能看到有什么用写错会怎样


个人技能目录当前用户的所有项目放跨项目通用技能技能存在，但不在你当前找得到的位置
项目 `.claude/skills/`当前项目放这个仓库专用的流程你以为它属于项目，实际没进项目可见范围
嵌套 `.claude/skills/`子目录工作区给 monorepo 某个包提供自己的技能目录在，但不在当前工作子树里
`--add-dir` / `/add-dir`你显式加入的额外目录临时扩展可见范围文件放进去了，却没把目录加到加载范围里
会话中的 live change detection当前会话改文件后继续工作时能看到最新内容你改了文件，却还在读旧版本

这个层级比 frontmatter 更早。文件放对位置，后面的 `description` 才有机会被读到；文件放错位置，再好的描述也不会被看见。

# frontmatter 决定 Claude 怎么认识这个 Skill {#frontmatter}
<!-- tag: metadata -->

`SKILL.md` 顶部的 YAML frontmatter 不是装饰，它是启动阶段最先被读取的结构化信息。官方文档强调，`description` 是最重要的入口；`name` 主要影响列表展示；其他字段则控制触发方式、路径范围和调用方式。

字段它回答什么问题对触发的影响常见误用


`name`这项技能在列表里叫什么通常影响展示，不直接决定自动加载把它当成所有布局下的命令名来源
`description`什么时候该用这个 Skill直接影响自动加载判断只写内部实现，用户说法却对不上
`when_to_use`补充触发场景和示例说法和 `description` 一起形成触发说明把关键 use case 放到后面，列表里看不见
`disable-model-invocation`要不要让 Claude 自动触发设为 `true` 后只适合手动调用想要人工运行，却仍然被模型自动调用
`paths`哪些文件范围能激活把触发限制在匹配文件上忘了写范围，结果什么地方都可能触发

最小 frontmatter 骨架复制

```text
---
name: summarize-changes
description: Summarize uncommitted changes and flag anything risky. Use when the user asks what changed, wants a commit message, or asks to review a diff.
disable-model-invocation: true
paths:
 - "*.md"
---

## Instructions

Summarize the current diff in two or three bullet points.
List any risks that need follow-up.
```

布局你输入的命令名从哪来`name` 在这里干什么


普通 `~/.claude/skills/skill-name/` 或 `.claude/skills/skill-name/`目录名显示标签来自 `name`，但 `/skill-name` 仍由目录名决定
嵌套 `.claude/skills/`工作目录下的子目录路径再加目录名用于展示和区分同名技能
`.claude/commands/`文件名不承担技能命令名的决定作用
插件根 `SKILL.md`frontmatter 的 `name` 与插件目录名组合这是少数会直接参与命令名的例外

先把 `name` 当作展示标签，把 `description` 当作自动加载提示，再看 `paths` 和 `disable-model-invocation` 这种控制项。这样写，触发链路才不会乱。

# description 要写用户意图，不写内部实现 {#description}
<!-- tag: trigger accuracy -->

用户不会说“请运行 `scripts/parse.py`”，他们更可能说“帮我总结未提交改动”或者“检查这个 diff 里有没有风险”。所以 `description` 要写成用户会说的话，先交代任务对象、目标和常见触发场景，再谈实现细节。

写法问题更稳的改法


`Helps with docs.`太泛，什么文档任务都可能撞上写清楚是总结、重写、比对还是抽取
`Use parse.py to run the pipeline.`把实现写成触发条件，用户说法对不上改成用户目标和典型请求
`Summarize uncommitted changes and flag anything risky. Use when the user asks what changed or wants a commit message.`任务、结果和触发场景都清楚可以直接拿来做自动加载判断

官方文档还提醒了一点：技能列表里会显示 `description` 和 `when_to_use` 的组合，长度会被截断，所以关键 use case 要放前面。也就是说，你在给 Claude 一条能快速判断的路标。

把描述改成可触发的说法复制

```text
description: Summarize diffs, review uncommitted changes, and flag risks when the user asks what changed, wants a commit message, or asks for a quick review.
```

这一步的判断很简单：如果去掉内部函数名后，用户还是能看懂它何时该被用，描述就对了。

# 正文负责执行，写法要短、直、可操作 {#body}
<!-- tag: instructions -->

frontmatter 负责让 Claude 发现这个 Skill；正文负责让 Claude 真正做事。正文里最值钱的内容通常是步骤、默认选择、边界和输出形状，而不是一段长解释。官方文档明确建议：正文要说“做什么”，少说“为什么这样”。

应该放进正文不该挤进正文原因


步骤顺序、默认路径、失败时怎么处理冗长背景、重复定义、所有参考资料原文正文会一直留在上下文里，太长会持续付出 token 成本
输入输出格式、常见坑、何时停止把所有可选分支列成一长串菜单Skill 需要一个默认做法，而不是把选择权全丢给模型临场猜
需要读支持文件时的明确提示只写“更多信息见 references/”Claude 需要知道什么时候去打开哪一个文件

正文的最小工作块复制

```text
## Instructions

1. Read the current diff.
2. Summarize the change in two or three bullets.
3. List the risks that need follow-up.
4. If the task needs detailed field rules, open references/field-map.md.
5. If the output format is fixed, use templates/report.md.

## Gotchas

- Keep the main file concise.
- Do not repeat the same rules in multiple sections.
- Put long examples and reference material into supporting files.
```

这里先不追脚本怎么写、怎么测、怎么发布。那几层会在后面的单元里接住；本页只要求你把“执行说明”写得像说明，不像论文。

# 支持文件负责把细节拆出去 {#supporting-files}
<!-- tag: support files -->

支持文件的价值把主文件里不必每次都读的内容挪出去，让 Claude 按需打开。官方文档把它们分成几类：模板、示例输出、脚本和详细参考文档。`SKILL.md` 要写清它们是什么、什么时候读，而不是只扔一个目录名。

文件类型适合放什么什么时候打开如果不这样做会怎样


`references/`字段说明、业务规则、长解释、例外情况正文提到需要补细则时正文被长规则撑爆，或者细节永远没被读到
`templates/`固定输出样式、报告骨架、可填空模板格式已经决定，只差填内容时每次都重新排版，结果不稳定
`scripts/`可重复执行、容易写错、需要稳定验证的步骤正文说明该执行脚本时临时代码反复出现在上下文里
示例文件期望输出、对照样例、错误示范用户需要看成品长什么样时模型知道要做什么，却不知道结果长什么样

支持文件引用示例复制

```text
## Supporting files

- Read `references/field-map.md` when the request mentions fields or validation rules.
- Use `templates/report.md` when the output format is fixed.
- Open `scripts/validate.py` when you need a repeatable check before responding.
- Keep examples in `examples/` if the output shape needs a concrete sample.
```

记住一个简单原则：正文负责告诉 Claude 什么时候打开文件，支持文件负责承载真正会变厚的内容。

# 加载以后，内容会在会话里继续留着 {#lifecycle}
<!-- tag: 生命周期 -->

Skill 被调用以后，`SKILL.md` 不会只出现一秒就消失。它会进入当前会话，继续影响后面的判断和执行。所以正文必须写成“持续生效”的说明：先做什么、默认怎么做、什么情况要停、什么时候读支持文件。官方文档也建议把 `SKILL.md` 控制在 500 行以内，超出的部分就拆出去。

阶段读什么写什么如果这层写坏了会怎样


首次发现`name`、`description`、`paths`把候选技能放进上下文相关请求没被加载，或者不相关任务误进来
正文进入会话`SKILL.md` 的 Instructions步骤、边界、默认选择加载后看起来存在，但下一步还是不清楚
后续回合已经加载的技能内容继续约束当前会话正文写成一次性提醒，后面就失效了
按需打开文件`references/`、`templates/`、`scripts/`把长规则和样例拆开主文件过厚，或者细节根本没被读到

这也是为什么前面一再强调主文件短、支持文件按需读。Skill 不是一次性提示，它会在会话里跟着你走一段时间。

# 自动触发、手动调用和路径限制不是一回事 {#trigger-modes}
<!-- tag: invocation -->

同一个 Skill 可以被不同方式使用。最常见的是自动加载：Claude 读到 `description` 发现相关，就把它带进来。另一种是手动调用：你直接输入 `/skill-name`。还有一种是路径限制：只有在匹配 `paths` 的文件范围里才更容易被激活。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

模式你在 frontmatter 里控制什么谁来触发错误症状


自动加载`description`、`when_to_use`、`paths`Claude 根据请求和文件上下文判断该触发时没加载，或不该触发时也进来了
手动调用`disable-model-invocation: true`你通过 `/skill-name` 明确叫它你想手动跑，模型却仍然自动启动
范围限制`paths`只有匹配到文件范围才更容易被激活跨目录误触发，或者本该在某包里工作却没被看见

如果一个 Skill 需要切到更独立的执行位置，比如 fork 到单独 agent，那是后面更高一层的调度问题。这里先把“自动、手动、按路径”说清楚就够了。

# 一个最小例子，够看清整条链路 {#example}
<!-- tag: example -->

官方文档里的最小例子是一个 `summarize-changes` skill：目录名决定命令名，`description` 决定它何时会被自动加载，正文则告诉 Claude 该怎么总结 diff。这个例子最有用的地方，是它把目录、frontmatter、正文和支持文件的分工一次摆平了。

最小 Skill 示例复制

```text
summarize-changes/
└── SKILL.md

---
name: summarize-changes
description: Summarize uncommitted changes and flag anything risky. Use when the user asks what changed, wants a commit message, or asks to review a diff.
---

## Current changes

`git diff HEAD`

## Instructions

Summarize the changes in two or three bullet points, then list any risks you notice.
If the diff is empty, say there are no uncommitted changes.
```

片段它在做什么如果写错会发生什么


目录名 `summarize-changes`给普通布局下的命令名一个稳定来源你输入的命令和技能列表里的名字对不上
`description`告诉 Claude 什么时候自动加载自动触发要么太宽，要么太窄
正文 `Instructions`告诉 Claude 加载后怎么做技能被加载了，却不知道下一步
支持文件 / 动态输入把当前上下文带进来，减少猜测正文只剩抽象指令，和真实工作树脱节

# 把一次请求追一遍，能看出问题在哪一层 {#trace}
<!-- tag: trace -->

还是拿“帮我看看未提交改动里有什么风险”这个请求做锚点。Claude 先扫 `description`，如果匹配就加载正文；如果正文里说要看字段规则，就打开 `references/`；如果输出样式固定，就按指向读模板。整个过程里，哪一层写错，症状就会落在哪一层。

步骤读什么写什么错位后的症状


1. 启动扫描`name`、`description`是否值得加载相关请求没触发，或者不相关请求也进来了
2. 进入正文`SKILL.md` 的 Instructions步骤、边界、默认选择加载后却不知下一步，或正文太长读不动
3. 按需打开资源`references/`、`templates/`、`scripts/`把细节交给支持文件主文件越来越厚，支持文件却从没被读到
4. 选择触发方式`disable-model-invocation`、`paths`自动还是手动该自动的时候没人加载，该手动的时候又被模型抢先触发

$$
 \text{有效加载} = \text{metadata 匹配} + \text{正文可执行} + \text{支持文件按需读}
 $$
 这里的“加法”只是提醒你三层都要站住：如果任一层缺位，技能就会看起来存在，却不好用。

这页先不追提示词、评估、跨平台发布或 subagent 分工。它们都很重要，但它们不是“文件怎么被发现、怎么被触发、怎么按需读”的这一层问题。

# 判断规则和常见误区 {#rules}
<!-- tag: review -->

这页先不追脚本、评估、安全发布和 subagent 调度；这些会在后面的单元里接回来。现在只记一个判断线：能不能被发现、能不能被正确触发、支持文件能不能按需读，决定了一个 Skill 是不是写对了。

误区后果更稳的写法


把 `name` 当成所有布局下的命令名来源普通技能目录、插件根和命令文件会被混在一起先看布局，再看 `name` 的角色
`description` 只写内部实现用户请求和自动加载条件对不上写成用户会说的话，先放关键 use case
正文里塞满长解释触发后仍然需要大量上下文，主文件持续膨胀正文只写做法，长细节下沉到支持文件
只写“See references/”知道有资料，但不知道什么时候打开在正文里写触发条件和读取场景

先别追的东西现在应该专注什么它们会在哪一节回来

你现在应该能做什么下一步要看的页下一步要补的能力

# 练习 {#practice}
<!-- tag: diagnostic -->

先做结构判断题，再做触发与分层题，最后用一条完整链路把字段、正文和支持文件串起来。导出的记录会保留每组标题、检查重点和参考答案，方便你回头自查哪里看偏了。

<!-- practice-group: 一、先看文件外形 -->
<!-- quick-review: 先分目录、`SKILL.md`、frontmatter，再谈正文和支持文件。 -->

### 1 choice

**Q:** 1. 一个 `SKILL.md` 最少由哪两部分组成？

- [ ] YAML frontmatter + Markdown 正文
- [ ] README + scripts
- [ ] index.html + CSS

Answer: YAML frontmatter + Markdown 正文

### 2 true/false

**Q:** 2. `description` 会直接影响技能是否会被自动加载。

- [ ] 对

### 4 synthesis

**Q:** 4. 用 2-4 句话说清 `name` 和你输入的命令名有什么区别。

Explain: 好的回答应该说清：`name` 主要是展示标签，便于在技能列表里识别；普通技能目录里，用户输入的命令通常来自目录名，而不是 frontmatter 的 `name`；只有插件根 `SKILL.md` 这类特殊布局，`name` 才会更直接地参与命令名。只要把“展示”和“命令”分开，你就不会把两个层次混在一起。

### 7 debug-checklist

**Q:** 7. 下面哪些写法更像是在给技能加“只在相关文件里触发”的边界？

- [ ] paths 里写 `.md` 和 `docs/**`
- [ ] 只在 description 里写“文档相关”
- [ ] 把脚本目录写成要读取的说明
- [ ] 让正文明确说何时打开支持文件

Explain: 更像边界的是把 `paths` 写成明确的文件模式，并在正文里说明何时读支持文件。只靠“文档相关”这种宽描述，边界还是会散；把脚本目录写成说明也会让职责混掉。真正的边界是“触发范围”与“读取条件”都写清楚。

### 10 synthesis

**Q:** 10. 假设你在写一个“总结 diff 并标风险”的 Skill，说明正文、`references/` 和 `templates/` 分别放什么。

Explain: 正文应该写加载后要做的步骤，例如先读 diff、再总结、再列风险；`references/` 放字段规则、风险分类、特殊例外；`templates/` 放固定输出样式或报告骨架。好答案会明确说出“什么时候打开哪一个文件”，而不只是罗列目录名。

### 11 code-reading

**Q:** 11. 用 3-5 句话追踪这条请求：`帮我看看未提交改动里有什么风险`。说明它先会碰到哪层，再决定是否继续读正文和支持文件。

Explain: 合格答案会先说 Claude 在启动时读取 `description` 判断这个请求是否相关；如果匹配，就进入 `SKILL.md` 正文，按步骤总结改动和风险；如果正文提到字段规则或输出样式，再去读 `references/` 或 `templates/`。如果能进一步说出“支持文件按正文提示打开”，说明你已经把触发链路说通了。

### 12 debug-checklist

**Q:** 12. 下面哪些情况会让一个 Skill “看起来存在，却总是不对劲”？

- [ ] description 太泛
- [ ] 正文只写长背景，不写步骤
- [ ] 支持文件有，但正文从不提什么时候读
- [ ] frontmatter 写得越长越好

Explain: 正确的诊断点是前面三项：描述太泛会误触发或漏触发，正文没有步骤会让加载后的执行断掉，支持文件没被明确引用会让细节永远读不到。最后一项不是问题本身，问题在于字段是否对齐用途，不是越长越好。

### 13 synthesis

**Q:** 13. 用 6-9 句话把这页讲清楚：目录、frontmatter、正文、支持文件、自动触发和手动调用分别各管什么。

Explain: 好答案会从目录名和 `SKILL.md` 说起，再解释 frontmatter 负责让 Claude 发现并判断这个 Skill，正文负责执行，支持文件负责承载长规则和样例；随后再说明自动触发依赖 `description` 和 `paths`，手动调用则由 `disable-model-invocation: true` 之类的控制项收紧。只要这条链路能讲通，你就在理解一套加载机制。
