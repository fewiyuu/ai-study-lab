先抓住这节在解决什么

学习目标

这一节只问一个问题：Skill 里某一步到底该留在正文里，还是该下沉到 scripts/。判断对了，Skill 读取时的上下文会更短，逻辑会更稳，后面的评估也更容易做。

正文是不是只负责判断、步骤和边界内容越写越长，阅读成本吞掉上下文

一次性命令是不是只是在调用一个成熟工具版本和环境一变，命令就漂了

scripts/是不是承接重复、易错、可验证的处理每次都临时拼流程，错误会反复出现

评估能不能证明脚本真的省了上下文只看一次漂亮输出，看不出是否真的更稳

为什么要把一部分逻辑交给脚本

mechanism

把重复逻辑收进脚本的前提，是先看到它是重复、稳定、可验证的。scripts/ 不负责“想清楚问题”，它负责把已经想清楚的规则固定下来，避免模型每次临场重写。

位置读什么写什么失败症状

当一条流程每次都要临场拼出来时，脚本是在压缩重复工作，不是在给技能加花样。

什么时候只写说明，什么时候该脚本化

boundary

一条简单规则：如果任务主要靠判断、取舍、写法，就放正文；如果任务主要靠稳定输入输出、重复处理和机械验证，就考虑脚本。一次性命令适合调用成熟工具，脚本适合把脆弱流程变成稳定接口。

代码
uvx ruff@0.8.0 check .
uv run scripts/extract.py input.html

任务更适合原因

把一段结论写得更自然正文需要判断和表达，不是固定算法

校验 200 个字段是否符合 schemascripts/规则明确、可重复、可自动报错

解析 10MB 日志并统计错误scripts/数据太大，模型直接读会浪费上下文

决定报告结论如何措辞正文需要综合语境和取舍

先把重复步骤和判断步骤分开，再决定是留在正文里，还是收进脚本。

脚本接口要让 agent 一次看懂

interface

智能体运行脚本时通常没有交互式终端。脚本如果问“要不要继续”，它就会卡住。输入应该放在参数、stdin 或环境变量里；--help 要短，写用途、参数、默认值和例子；错误要说收到什么、该是什么；退出码要能分失败类型。

代码
# /// script
# requires-python = ">=3.12"
# dependencies = [
# "beautifulsoup4>=4.12,<5",
# ]
# ///

from bs4 import BeautifulSoup
import argparse

parser = argparse.ArgumentParser(description="Extract fields from HTML.")
parser.add_argument("html_file")
parser.add_argument("--limit", type=int, default=20)
parser.add_argument("--dry-run", action="store_true")
args = parser.parse_args()

接口要素推荐做法原因

输入参数、stdin、环境变量不会阻塞

--help简短帮助 + 一个例子agent 能快速学会怎么调用

错误说明收到什么、期望什么、下一步怎么改便于自我修正

退出码不同失败类型分开便于分支处理

stdout、stderr 和输出大小

output

脚本输出会进入上下文，所以它不是越多越好。更稳的写法是把可继续处理的数据放到 stdout，把进度、警告和诊断放到 stderr。大结果默认只给摘要，必要时写文件，不要把几万行直接吐到终端。

代码
print(json.dumps({"missing": 3, "duplicates": 1}))
print("Warning: 2 rows skipped because dates were empty", file=sys.stderr)

输出通道放什么什么时候用

stdout结构化数据、摘要、下一步要处理的结果要让后续步骤继续消费时

stderr警告、进度、诊断、失败原因要保留调试线索时

文件完整结果、大表、日志、可下载产物结果太大，不适合直接塞进上下文时

摘要最少但足够的判断信号想先看结论，再决定要不要展开时

把数据、诊断和大结果分开，不再让日志替你做决策。

重试、幂等和 dry-run

safety

智能体看到错误后可能再次运行命令，所以脚本要假设自己会被重试、传错参数、走到边界。幂等性意思是：同一命令重复运行，不会额外制造副作用。能 create-or-update 的脚本，通常比单纯 append 更适合自动化流程。

代码
plan_delete.py --dry-run
plan_delete.py --confirm

风险脚本护栏Skill 正文怎么写

重复运行幂等 create-or-update允许重试，但检查输出

破坏性操作--dry-run + --confirm先预览计划，再执行

模糊输入枚举参数和清晰错误不要猜用户意图

输出过大--limit / --output默认摘要，必要时写文件

看出哪些动作必须先预览，哪些动作可以直接更新，哪些动作应该默认拒绝。

怎么证明脚本真的有用

eval

脚本的价值要能被测出来。Agent Skills 的评估通常分两层：先看会不会触发，再看输出是否达标。触发评估用 should-trigger 和 should-not-trigger 查询；质量评估看 with-skill / without-skill 对照、可观察断言和证据评分。

代码
[
 {"query": "帮我把这份逐字稿整理成决策和待办", "should_trigger": true},
 {"query": "给我写一封邀请同事开会的邮件", "should_trigger": false}
]

评估层看什么证据是什么

触发评估相关任务会不会加载 Skillshould-trigger / should-not-trigger 的命中率

输出评估结果有没有固定字段、格式和边界断言、截图、文件内容、命令输出

基线对照有没有比没有 Skill 时更稳with-skill / without-skill 的对比结果

评分记录失败到底缺了什么PASS/FAIL 之外的文字证据

把“能不能触发、输出是否达标、改完是否真的更稳”这三件事分开看。

一个完整案例：把脆弱命令收进脚本

worked

看一个更像真实工作的例子。你有一个“整理会议逐字稿”的 Skill，正文里原本写了三段 shell 说明：先找文件、再抽标题、最后把结果写成 Markdown。第一次看，这样已经能做事；第二次、第三次、第四次做同一类任务时，它就开始浪费上下文，因为模型每次都在重组那几段说明。

步骤正文保留什么脚本接住什么如果写反了会怎样

从正文到脚本的切口复制

这个切口的判断很重要：正文里还是要告诉 agent 先看什么、后看什么；脚本里只负责把已经稳定的规则执行得一模一样。

你现在应该能把一个“看起来能做”的流程，拆成“正文讲清楚、脚本固定住、输出能继续接”的三段。

一个更细的阈值：什么时候已经该下沉了

decision

很多人卡住，因为不确定“重复到什么程度才该脚本化”。这节给一个更可执行的阈值：当一段流程已经满足下面三条里的两条，就该认真考虑下沉到 scripts/。

重复出现同一个流程是不是每次都要写如果每次都会临时重写，说明正文已经在承担实现层工作

错误代价高写错一次会不会把结果带偏比如字段漏一个、路径写错、输出截断，都会直接影响结果

输入输出稳定参数、文件、输出形状是不是已经固定固定了以后，脚本比正文更能压住差异

能被检查有没有办法用断言或文件比较确认结果能检查，就有理由让脚本接管

阈值判断草图复制

这个草图不是算法，只有一个用途：在写 Skill 时帮你做决策。它提醒你先看重复、再看代价、再看输入输出是否稳定。只要其中两个已经成立，正文就不该继续背实现。

你现在应该能把“该不该下沉”从一种模糊感觉，变成一组可以检查的条件。

迁移规则：怎么把正文里的步骤搬进脚本

migration

真正把事情做稳，靠的知道怎么搬。最稳的迁移顺序通常是：先把原正文里重复出现的步骤列出来，再找它们的输入、输出、错误和确认点，最后才决定哪一段进脚本，哪一段继续留在正文。

原步骤先问什么放哪一层最常见的坏结果

迁移前后对照复制

这条迁移规则有个关键点：正文不该消失。它还要保留顺序、边界和例外，因为脚本只负责稳定执行，不负责替代判断。

你现在应该能从一段正文里，先找重复步骤，再决定哪些搬进脚本，哪些留在正文，哪些交给评估去证明。

请求追踪：一条需求怎么落到脚本资源

trace

看一条更真实的请求：帮我把这段会议逐字稿整理成决策、待办和风险。这类请求先经过正文的判断层。正文先确认它是“整理逐字稿”而不是“写邮件”或“改文案”，再决定是否要读脚本资源、模板和输出约定。

识别任务请求是不是重复、稳定、可验证判断要不要脚本化把一次性判断当成脚本需求

进入正文目标、边界、例外决定读取顺序和默认做法正文太短，脚本没有上下文

调用脚本输入文件、参数、输出形状执行固定规则临场拼 shell，格式漂移

失败处理输入是否缺失、输出是否过大返回错误或摘要agent 以为成功，继续走错路

代码
1. 识别请求是不是重复流程。
2. 看正文能不能把边界说清楚。
3. 确认脚本接收什么输入、产出什么结果。
4. 检查失败时是不是会停在正确位置。

这个追踪告诉你：脚本不是第一层响应，正文才是。脚本只是把正文已经决定好的稳定处理接过去。

如果你把请求直接扔给脚本而跳过正文，最常见的后果就是参数不全、输出不稳，或者失败时没有人知道该回哪一层修。

失败诊断：卡住、刷屏、误删分别怎么看

failure

脚本资源最容易出问题的地方，在“遇到边界时会不会把 agent 卡住”。这节把三种最常见的故障放在一起看：交互式阻塞、输出过大、破坏性动作没有护栏。它们的修法不一样，但都能从同一条检查链开始。

脚本卡在等待输入把交互式提问写进了流程看是否用了 input()、隐藏确认或需要人工回车的步骤

stdout 爆满把诊断、进度和数据混在一起了把诊断移到 stderr，把大结果改成文件或摘要

重复运行后结果变乱脚本不是幂等的看是不是在 append、重复创建或覆盖已有结果

误删、误发、误写破坏性动作没有预览和确认先上 --dry-run，再要求显式确认

代码
1. 看输入是不是阻塞。
2. 看 stdout 是否被噪声占满。
3. 看是否有重复运行会放大的副作用。
4. 看破坏性动作是否先预览。

这个顺序之所以有用，是因为它从最容易让 agent 停住的地方开始：先看输入，再看输出，再看副作用。只要这三层过了，脚本大多就能继续走。

如果你把这条诊断链放到真实 Skill 里，先能快速定位的是“为什么卡”，而不是“到底该不该再补一段说明”。

怎么读导出的练习记录

review

导出记录不是为了把答案塞进一个 Markdown 文件就结束。它的作用，是让你在离开页面之后还能看出自己错在判断、接口、输出，还是评估。这一层很重要，因为脚本化的错误通常层次混了。

把重复任务选成正文还没分清判断层和执行层正文、命令与脚本

知道要脚本化，但输入还写成 input()接口还在阻塞脚本接口

答对了判断题，短答却说不清输出分流知道概念，但没把 stdout / stderr 分开输出与上下文

会说 dry-run，却没写出幂等和确认门安全只停在口头提醒安全与重试

读记录的顺序复制

如果一份记录里总是同一类题错，不要急着背答案。先找这一类题共同指向的层，再把那一层补稳。

下一步怎么接到真实 Skill

bridge

如果你要把这节落到真实 Skill 里，下一步先找一条每次都在重写的流程，问它是否稳定、重复、可验证。是的话，就给它脚本接口；不是的话，留在正文里就够了。

找出一条重复流程判断它是否该进 scripts/触发、评估、安全边界

给脚本补参数和 --help非交互输入、结构化输出、退出码跨平台可运行性

给大输出加摘要和文件stdout / stderr / --limit结果是否能稳定复现

先做最小 eval触发和输出对照安全和发布前检查

把一条临场拼出来的步骤，改写成一条可调用、可检查的脚本资源。

常见误区与边界

review

这节最容易犯的错，把脚本写成另一种更啰嗦的正文。脚本只接住稳定、重复、可验证的处理；判断、取舍和语义边界仍然应该留在正文里。

见到重复就脚本化过度工程，反而增加维护面先问是否稳定、重复、可验证

脚本里继续问人智能体会卡住，无法继续改成参数、环境变量或 stdin

stdout 打太多上下文被噪声占满默认摘要，必要时写文件

只看一次漂亮输出不知道改动是否真的更稳做 with-skill / without-skill 对照

安全只写一句提醒破坏性操作仍然会被误触发把 --dry-run、--confirm 和幂等写进接口
