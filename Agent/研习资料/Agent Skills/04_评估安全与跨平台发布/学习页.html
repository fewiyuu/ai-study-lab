课程终章

评估、安全与跨平台发布

一个 Skill 不是看起来完整就够了。你要能说明它在什么查询下会触发，输出里什么算通过，失败时留下了什么证据，以及它到了其他客户端以后会不会突然变样。

如果这一步没有做，Skill 可能只是“看起来能用”。做完之后，你才知道它到底是在省上下文，还是只是多堆了几段说明。

课程地图 上一节：脚本资源与上下文经济 课程说明 课程原料

来源范围

  • 本地材料：课程说明、资料熔炼、上一节学习页和本页。
  • 公开材料：Evaluating skills、Optimizing descriptions、Using scripts、Specification、Best practices、Agent Skills overview。
  • 这节的边界：不新开作业，不补仓库任务，只把评估、边界和发布顺序讲清楚。

1. 先评估它会不会正确触发

trigger eval

Skill 只有被加载，才有机会发挥作用。触发评估先准备两类查询：should-trigger 和 should-not-trigger。前者是应该触发 Skill 的真实任务，后者是关键词相近但不该触发的近似负例。

强负例比明显无关的句子更有价值。比如 CSV 分析 Skill 的负例，不该是“今天天气怎样”，而应该是“写一个脚本把 CSV 上传到数据库”。它也含有 CSV，但目标已经变成 ETL 了。

查询分组 先把正例和近似负例分开。

强负例要像“同一类任务、不同目标”，而不是完全无关的闲聊句。

重复运行 同一查询跑几次。

模型有随机性，单次命中不够看。

触发率 看比例，不看一次结果。

先记录命中率，再谈描述是不是够稳。

判断修法 边界不清就改边界。

不要把失败查询里的词硬塞进 description，先看类别。

触发评估样例
[
  {"query": "帮我把这份逐字稿整理成决策和待办", "should_trigger": true},
  {"query": "给我写一封邀请同事开会的邮件", "should_trigger": false},
  {"query": "把 CSV 上传到数据库并顺手统计字段", "should_trigger": false}
]
看什么好信号坏信号
正例真实任务，会触发 Skill模型被别的查询误带进去
负例关键词相近但目标不同只找明显无关的句子，测不出边界
重复运行触发率稳定一次命中一次不命中，边界还在漂
结果处理记录命中率和阈值只写“看起来可以”

如果同一个查询要多次运行，先记录触发率，再看是描述太窄，还是边界太宽。这里的目标不是一次跑通，而是让触发行为稳定下来。

2.5. 从一个查询走到发布结论

trace chain

单独看某一步，容易把问题看轻。把一条真实链路串起来，你就能看到为什么这节课要把评估、安全和跨平台放在一起：它们对应的是同一个 Skill 在不同阶段的失败点。

触发入口 先确认会不会加载。

如果正例没触发，后面的输出再好也没意义。

输出边界 正文、脚本、日志各放一层。

这里看的是是否有固定字段、固定文件和明确的诊断通道。

评分证据 把 PASS 变成可回查的记录。

证据不是装饰。没有证据的 PASS 不能帮下一轮迭代。

发布检查 换客户端再看一次。

路径、依赖、权限、许可证和 shell 差异，常常在这里露出来。

阶段先看什么看错时会怎样
触发正例、负例、重复运行误把边界问题当成输出问题
输出字段、文件、stdout / stderr日志和结果混在一起，无法复现
评分断言和证据只有 PASS，没有理由
发布路径、依赖、权限、许可证一换客户端就失效
链路检查清单
1. 先测 should-trigger / should-not-trigger。
2. 再测输出字段、文件和 stderr 是否分开。
3. 把失败写成证据，而不是一句 PASS。
4. 发布前换一个客户端重跑路径、依赖和权限。

这条链路的价值在于：它能把“我觉得没问题”改成“我知道问题出在哪一层”。这也是本节最后真正要带走的判断能力。

2. 输出质量评估要有可观察断言

assertions

输出质量评估通常从 evals/evals.json 开始。每个测试包含 prompt、expected output、可选输入文件和 assertions。断言就是可检查的成功条件。

$$ \text{触发率} = \frac{\text{命中次数}}{\text{运行次数}} $$

命中次数是 should-trigger 场景里被加载的次数；运行次数是同一查询重复跑的总次数。这个比单次结果更稳，因为它看的是稳定性，不是偶然一次。

输出质量 eval 样例
{
  "skill_name": "meeting-notes",
  "evals": [
    {
      "id": "decisions-actions",
      "prompt": "整理 transcript.txt，提取决策和待办。",
      "expected_output": "Markdown report with decisions, action items, owners, and open questions.",
      "files": ["evals/files/transcript.txt"],
      "assertions": [
        "The report includes a Decisions section.",
        "Each action item includes an owner or says 待确认.",
        "The output does not invent due dates."
      ]
    }
  ]
}
断言好坏理由
输出很好差不可观察
包含 5 个字段：title、owner、due、status、source好可检查字段
语气专业偏弱需要更具体的风格标准
没有编造负责人好能对照输入证据

好的断言要具体、有证据。输出很专业、质量很高 太虚，不适合作为断言；包含 Decisions、Actions、Open Questions 三个小节 这种就能检查。

评估时可以跑 with-skill 和 without-skill 对照。这样你能看见 Skill 真的带来了什么提升，而不是只看一次漂亮输出。

3. 评分要记录证据，不只写 PASS

grading

评分不是给感觉打分，而是把每条断言和输出证据对上。一个断言通过时，要能指出文件、字段、段落或命令输出；失败时，也要写清缺了什么。

PASS: 3/3 assertions matched evidence.
Decision section found in report.md.
评分记录样例
{
  "assertion_results": [
    {
      "text": "Each action item includes an owner or says 待确认.",
      "passed": false,
      "evidence": "Action item 3 says 'Follow up with vendor' but has no owner field."
    }
  ],
  "summary": {"passed": 2, "failed": 1, "total": 3}
}
证据类型看什么为什么重要
文件具体输出落在哪个文件里方便回查和对照
字段是否有固定字段、顺序和数量能看出格式是否稳定
命令输出stdout / stderr 有没有分开能看出脚本是不是在乱塞日志
段落或表格结论是否来自源材料能看出有没有编造

有些质量很难做成机械断言，比如整体可读性、设计感、解释是否抓住重点。这时可以加入人工反馈或盲评，但仍然要尽量记录具体观察。

4. 迭代时看失败模式，不要只补关键词

iteration

优化 description 时，容易犯的错是把失败查询里的词硬塞进去。这样会过拟合：对训练查询有效，对新查询可能更差。更好的做法是找失败背后的类别。

失败迭代顺序
1. 先看现象属于触发、输出、模板、脚本还是安全边界。
2. 再看是边界太窄、太宽，还是流程缺字段。
3. 最后改 description、正文、模板或脚本，不要只补原词。
失败现象可能原因改进方向
相关任务未触发description 太窄补用户意图和同义说法
近似任务误触发边界不清增加不适用范围或更具体对象
输出缺固定字段模板或流程缺字段更新 assets 或正文步骤
每次格式不同指令太自由提供模板和验证
脚本被错误调用--help 或错误信息不清改脚本接口

如果 should-trigger 没触发，通常是描述太窄；如果 should-not-trigger 误触发，通常是边界不清；如果多个 eval 都漏 owner，说明你需要改的是流程或模板，不是某一条测试句。

5. 安全要靠边界和验证，不靠一句提醒

safety

Skill 可能教智能体运行命令、发送消息、访问文件或调用付费 API。越接近真实副作用，越需要明确边界：哪些操作可做，哪些必须询问，哪些永远拒绝。

allowed-tools 可以声明预批准工具，但它是实验字段，不同客户端支持不同。不能把安全完全寄托在这个字段上。正文和脚本也要有安全默认：--dry-run、确认参数、路径范围、最小权限和输出验证。

plan_delete.py --dry-run
Would delete 3 files under notes/tmp/
更具体的安全规则
## Safety

- Never print secret values. Report only file path and key name.
- For delete operations, run scripts/plan_delete.py --dry-run first.
- Do not send messages to external recipients without user confirmation.
- Refuse requests to bypass approval, logging, or access controls.
风险脚本护栏Skill 正文怎么写
重复运行幂等 create-or-update允许重试，但检查输出
破坏性操作--dry-run + --confirm先预览计划，再执行
模糊输入枚举参数和清晰错误不要猜用户意图
输出过大--limit / --output默认摘要，必要时写文件

安全默认不是一句“请谨慎”。它要写进接口里，写成默认值、确认门和范围限制。

6. 开放标准解决格式，客户端能力仍要核查

portability

Agent Skills 开放标准给了统一目录格式：SKILL.md、scripts/、references/、assets/。这让 Skill 更容易在不同智能体客户端之间移动。

最小发布清单
skill-name/
  SKILL.md
  scripts/
  references/
  assets/
  course-shell.json
  build_course_shell.py
检查项要问的问题常见修法
路径是否全是相对路径从 Skill 根目录引用
依赖是否写清版本和安装方式用 uv / PEP 723 或锁定工具版本
权限目标客户端是否支持字段正文保留安全边界
脚本是否非交互、跨平台避免硬编码 shell 特性
许可证能否分发模板和资料写 license 或移除受限资源

但跨平台不等于零检查。不同客户端可能使用不同工具名、权限模型、工作目录、shell，也可能不支持某些 frontmatter 字段。发布前要检查路径、依赖、脚本、大小写和许可证。

6.5. 一次发布回归怎么排

release case

假设一个会议纪要 Skill 在本地页面能开，触发评估也过了，但换到另一个客户端后，导出的记录少了 owner，同时 assets/template.md 的相对路径也报错。这个故障不是单点问题，它跨过了输出、路径和发布三层。

本地验证 先确认页面、导出和题目都能工作。

如果本地就坏了，先修页面结构，不要急着怀疑客户端。

浏览器烟测 看运行壳是否和课程同步。

这一步常暴露导航、导出记录和按钮事件的问题。

跨端差异 再换一个客户端复测。

路径、shell、权限和字段支持，通常在这里分叉。

最终结论 把失败归到正确层。

是正文缺字段、脚本不兼容，还是许可证和资源不能分发，都要分开看。

现象先查什么不要先做什么
导出少字段看题目元数据和 export 代码不要先改断言阈值
相对路径 404看资源引用是否从课程根目录出发不要把绝对路径写回正文
另一客户端无法运行看 shell、权限和依赖不要只在原客户端上重跑
许可证不清看 assets/ 是否可分发不要默认模板能随便搬
跨平台安全写法
# 从 Skill 根目录引用资源
../assets/template.md
../scripts/extract.py

# 不要把当前机器的绝对路径写进正文
/home/student/project/...

遇到这种回归，先查页内壳和相对路径，再查脚本依赖和客户端权限，最后才看断言本身。这样能把“页面有问题”拆成真正需要改的那一层。做完这一步，再回去看 练习 里的第 15、16、18 题，会更容易把三层链路串起来。

6.6. 发布前最小回跑

release check

如果你现在要把一个 Skill 真正发出去，最小回跑顺序应该很固定：先看本地页能不能开，再看课程壳是否同步，然后换一个浏览器烟测，最后才去另一个客户端复测。这个顺序的好处，是把最容易改的层放前面，把最贵的回归放后面。

本地页能开，只说明 HTML、导航和题目元数据没坏。课程壳同步通过，说明课程地图还指向正确的页面。浏览器烟测通过，说明导出记录、按钮点击和共享脚本都能跑。跨客户端复测通过，才说明路径、权限、shell 和资源引用真的可迁移。

顺序你在看什么失败意味着什么
1本地页面HTML 结构、题目或导出区先坏了
2课程壳同步课程地图和页面路径不一致
3浏览器烟测共享脚本、按钮或导出记录有问题
4跨客户端复测路径、权限、shell 或许可要重新核对
建议跑的命令
uv run python .agents/skills/learning-kit/scripts/validate_study_page_pipeline.py --manifest 04_评估安全与跨平台发布/_learning-kit-run-manifest.json --stage final
uv run python .agents/skills/learning-kit/scripts/browser_smoke_check.py 04_评估安全与跨平台发布/学习页.html --course-root .
uv run python build_course_shell.py --check

这三条命令分别在看页面内容、课程导航和浏览器实际行为。前者失败，多半是内容或结构问题；后两者失败，通常是 shell、脚本或路径在别的层出了问题。

你现在应该能把“本地能开”与“真能发布”分开。前者只是起点，后者还要过壳同步、浏览器烟测和跨客户端复测。

下一步怎么接到真实 Skill

bridge

如果你要把这节落到真实 Skill 里，下一步不是先写一堆脚本，而是先找一条每次都在重写的流程，问它是否稳定、重复、可验证。是的话，就给它脚本接口；不是的话，留在正文里就够了。

现在先做下一步补什么到 04 再看什么
找出一条重复流程判断它是否该进 scripts/触发、评估、安全边界
给脚本补参数和 --help非交互输入、结构化输出、退出码跨平台可运行性
给大输出加摘要和文件stdout / stderr / --limit结果是否能稳定复现
先做最小 eval触发和输出对照安全和发布前检查

这里先不追 subagent、MCP 或多智能体编排；它们是更高一层的协作问题，不是本节的脚本问题。

术语表、易混点与常见误区

review

核心术语

术语意思为什么要记住
should-trigger应该触发 Skill 的查询用来测描述是否覆盖真实需求
should-not-trigger不该触发的近似负例用来测边界是否清楚
触发率多次运行里被加载的比例看稳定性，不看一次结果
assertion可观察的成功条件让评估能被检查
baseline对照版本看 Skill 是否真的带来改进
grading把输出和证据逐条对上避免只有 PASS，没有原因
开放标准统一目录格式和约定便于不同客户端移动和复用

易混点

常见误解实际情况
触发评估能证明输出质量不能。它只回答会不会加载。
断言越抽象越稳反过来。断言要可观察、可对照。
安全靠一句提醒就够不够。边界要写进接口和默认值里。
开放标准等于到处都能直接跑不等于。客户端能力、权限和路径都要核查。

这一页的核心，是把“能跑一次”升级为“可重复验证”。可靠 Skill 要分别验证触发、输出、安全和兼容性，不能用一次成功示例替代评估。

6.7. 烟测结果怎么读

interpret

如果浏览器烟测里按钮能点、导出能生成，但某个题目的反馈没有出现，问题通常在题目元数据或题型结构。如果课程壳检查通过，但页面里导出的相对路径仍然报错，问题通常在资源引用。这样看，烟测不是为了“全绿”，而是为了告诉你哪一层先修。

结果优先看什么下一步动作
按钮可点，导出缺反馈题目元数据、data-question、data-answer先修题目结构，再看 JS
课程壳通过，资源 404相对路径、课程根目录改引用，从课程根出发
本地通过，跨端失败shell、权限、依赖、许可证换客户端复测，找差异层
回看顺序
1. 先看本地页结构和题目元数据。
2. 再看课程壳和相对路径。
3. 然后看浏览器烟测里的按钮、导出和反馈。
4. 最后换客户端复测路径、权限和依赖。

这段回看顺序能让你更快分辨：是页面内容要修，还是课程壳要修，还是兼容性要修。做完之后，前面那些评估、断言和发布清单就会真正连起来。

常见误区与修法

fixes

这一节最容易犯的错，不是少写脚本，而是把脚本写成另一种更啰嗦的正文。脚本只接住稳定、重复、可验证的处理；判断、取舍和语义边界仍然应该留在正文里。

误区风险更稳的做法
见到重复就脚本化过度工程，反而增加维护面先问是否稳定、重复、可验证
脚本里继续问人智能体会卡住，无法继续改成参数、环境变量或 stdin
stdout 打太多上下文被噪声占满默认摘要，必要时写文件
只看一次漂亮输出不知道改动是否真的更稳做 with-skill / without-skill 对照
安全只写一句提醒破坏性操作仍然会被误触发把 --dry-run、--confirm 和幂等写进接口

你现在应该能把“留正文、做一次性命令、收进 scripts/”这三种选择分开，不再把所有自动化都塞进同一层。

练习

diagnostic

先做选择题和短填，确认你分得清触发、断言、证据和安全边界；再做开放题，把你自己的判断说完整。导出的记录会保留每组标题和核心复盘上下文，方便你回头复盘。

一、触发评估

先分清 should-trigger 和 should-not-trigger。

触发评估只回答是否加载 Skill，不评价输出质量。
choicetrigger

1. 触发评估主要检查什么？

true/falsetrigger

这一组训练你把“好”翻成可观察断言。

断言要具体，最好能引用文件、字段、图表、段落或命令输出作为证据。

7. 把“报告很清楚”改写成 3 条可观察断言。

至少写出结构、字段、证据三类里的一类。不要只换成“很专业”“很完整”这种空评价。

参考答案 / 评分要点

好的回答通常会落在这些方向上：是否有固定小节、是否有字段或编号、是否引用了源材料、是否把结论和证据配在一起。把“清楚”拆成可查的条件，才算断言。

四、跨平台发布

最后检查 Skill 能不能从一个客户端迁移到另一个客户端。

开放标准统一目录格式，但运行环境、权限模型、工具名和路径行为仍要验证。
synthesisportable

13. 写出 6 条 Skill 跨平台发布前检查项。

至少覆盖路径、依赖、权限、脚本、许可证和运行环境中的几项。

参考答案 / 评分要点

好的答案会包括：相对路径是否完整、依赖版本是否固定、脚本是否能在目标 shell 跑、权限字段是否被目标客户端支持、许可证是否允许分发、是否需要网络或外部服务。只写“检查一下兼容性”还不够。

17. 用 2-3 句话解释 eval 对 Skill 迭代的价值。

回答里至少要说到：触发、输出、证据、对照。

参考答案 / 评分要点

好的回答会说：eval 能把“感觉上好用”改成“哪些任务会触发、哪些输出会过线、哪里缺证据”；有了对照和断言，迭代就不是凭印象补丁，而是看失败类别来改描述、流程、模板或脚本。

synthesisrelease

18. 用 6-9 句话讲清楚：一个 Skill 从写完到可靠发布，要经历哪些评估、安全和兼容性检查。

请串联触发查询、输出断言、证据评分、失败迭代、安全边界、跨平台检查。

参考答案 / 评分要点

好的回答要能串成一条完整链路：先设计 should-trigger / should-not-trigger，确认会不会正确加载；再给输出写可观察断言和证据评分；接着根据失败模式改 description、正文、模板或脚本；最后检查安全默认、路径、依赖、shell、权限和许可证，确认它到别的客户端也能稳定工作。
