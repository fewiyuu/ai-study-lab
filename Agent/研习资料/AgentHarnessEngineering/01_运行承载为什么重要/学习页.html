运行承载工程 · 第 01 单元

运行承载为什么重要

本页先把运行承载层说清楚。它负责把模型输出接成受控行动，管住工具权限、状态保存、恢复点和追踪记录。读完这一页，你应该能把一个智能体故障先分到“模型判断”“工具执行”还是“承载边界”三类里，再决定下一步看哪里。

课程地图下一单元

读完本页你应该能做什么

  • 能画出模型、运行承载层、工具、状态和追踪之间的分工，不把所有责任都塞给提示词。
  • 能读懂一段智能体循环伪代码，指出终止条件、工具回填、恢复点和副作用边界。
  • 能为一个长任务列出最小状态表和追踪字段，判断哪里会重复写入、哪里会丢上下文。

学习目标

goal

这一单元不讲模型训练，也不讲提示词技巧。它只回答一件事：当模型要连续调用工具、跨步骤保留上下文，并在失败后继续工作时，外层系统要守住哪些边界。

对象 主要负责 不负责 没做好时的症状
模型 理解目标、选择下一步、生成工具意图、解释结果。 权限隔离、持久状态、幂等写入、故障恢复。 答案看起来连贯，动作却不稳定。
运行承载层 调度循环、执行工具、保存状态、控制恢复、记录 trace。 直接替模型思考。 系统能说话，却接不住长任务。
工具 完成受限动作，例如读文件、查库、发请求、写入外部系统。 自己决定权限边界和恢复策略。 参数错、输出长、重试重复副作用。

你现在应该能做什么：拿到一段智能体描述时，先把模型、承载层和工具拆开，再讨论它为什么稳或不稳。

先抓主线

overview

这一页用一个固定的锚点例子：文档审阅智能体。它要读几份资料，调用检索或解析工具，把中间结果压缩后喂回模型，再决定下一步。这个例子足够小，能把分工看清，也足够大，能把恢复、状态和追踪的问题露出来。

外层要把一次行动变成可控流程。承载层接住的是意图、状态、工具结果和外部副作用。

输入 用户目标、资料范围、允许访问的资源。

承载层先接住边界，再把消息送进模型。

模型输出 工具名、参数、继续推理还是结束。

这里还只是意图，不代表动作已经发生。

工具执行 参数校验、权限检查、超时、失败返回。

承载层决定这一步能不能做、做多久、做完回传什么。

记录 checkpoint、call_id、error、latency、approval。

这些字段让中断恢复和排查变成可能。

下一轮输入 摘要后的工具结果、状态变化、剩余目标。

外层把结果压缩成模型真正该看见的内容。

你现在应该能做什么：看到一条运行链时，先说清每一步是意图、执行、记录，还是回填，不要把它们混成一句“模型做完了”。

核心机制

trace

运行循环是这门课的骨架。模型只负责输出下一步，承载层负责解释输出、决定是否执行工具、把工具结果回填，再判断是否继续。只要这层逻辑没定好，系统就会在无限循环、提前结束或者重复动作里打转。

外层循环伪代码
while turn_count < max_turns:
    model_output = call_model(agent, input_items, state)

    if has_final_output(model_output) and not has_tool_calls(model_output):
        return final_result(model_output)

    if has_handoff(model_output):
        agent = next_agent(model_output)
        input_items = build_handoff_input(model_output)
        continue

    if has_tool_calls(model_output):
        tool_results = run_tools_with_limits(model_output.tool_calls)
        input_items = append_tool_results(input_items, tool_results)
        continue

raise MaxTurnsExceeded()
分支 承载层在看什么 常见误解 真正的排查点
最终输出 有没有可交付的答案，且没有再要工具。 模型只要说“完成”就能结束。 先看是否还有未执行的工具意图。
交接 下一步是不是要换到别的智能体或处理器。 交接只是换个提示词前缀。 输入要重建，状态要带过去。
工具调用 参数、权限、超时、错误格式、回填长度。 把工具当成普通文本补充。 先看工具执行器，再看模型。
轮数上限 是否已经反复修正到失去收敛。 多跑几轮总会好。 看 max_turns 和失败重试策略。

你现在应该能做什么：打开一段智能体循环代码，先找结束条件、交接条件、工具回填和上限控制，再看谁在决定下一步。

工具与权限

tools

工具面决定模型能做什么，也决定它不能做什么。一个稳的工具需要契约、限制、回传和追责四层约束；缺一层，长任务就很容易把错误放大成外部副作用。

层面 要写清楚的东西 对应的保护动作 错了会怎样
输入契约 字段、类型、必填项、默认值、错误格式。 结构化校验，失败早返回。 模型拼出来的半结构化文本被工具硬猜。
执行边界 超时、并发、重试、输出长度、资源上限。 限制每次调用的成本和时长。 一次工具调用拖垮整轮循环。
权限边界 工作空间、只读/可写范围、账户、网络。 把动作放进受限环境。 工具能碰到它不该碰的东西。
回填策略 摘要、来源、时间、版本、幂等键。 只把后续判断需要的部分送回模型。 全文回填把上下文塞满。
工具结果回填示例
{
  "source": "internal_doc_v3",
  "timestamp": "2026-06-21T10:15:00+08:00",
  "status": "ok",
  "idempotency_key": "report-20260621-001",
  "summary": "提取到 3 条结论，2 条待核对项"
}

这种回填更稳。模型下一轮看到的是来源、时间、状态和摘要，原文不会直接挤满上下文。它做判断时依然有上下文，但不会被噪声压住。

你现在应该能做什么：给一个工具写出输入契约、执行上限、权限范围和回填字段，而不是只给它一个函数名。

状态与会话

state

会话负责记住上下文，检查点负责记住执行位置。两者常常一起出现，但用途不同。前者帮模型理解“刚才说到哪了”，后者帮系统回答“任务已经走到哪一步了”。

状态类型 读取什么 写入什么 什么时候会过期 典型症状
对话状态 消息历史、摘要、用户偏好。 新消息、压缩后的摘要、目标说明。 上下文窗口被挤掉时。 模型忘了前文，但还在继续答。
任务状态 当前步骤、分支、等待点。 下一步、人工审批结果、子任务进度。 任务完成或被放弃时。 重启后不知道该接哪一步。
工具状态 调用 ID、参数、返回、错误、耗时。 调用记录、重试次数、结果摘要。 调用结束后仍要保留供排查。 只知道结果，不知道哪里出了错。
副作用状态 操作 ID、外部确认号、幂等键。 写库、发信、付款、文件改动的确认。 外部动作发生后立刻变成关键记录。 一重试就重复发信或重复写入。

当系统要跨进程、跨等待、跨重启继续运行时，状态不能只存在线程内存里。内存变量适合当前轮推理，不适合恢复；检查点适合恢复，不适合替代工具契约。

你现在应该能做什么：看到一个状态表时，能区分哪些字段只给模型看，哪些字段要给恢复逻辑看，哪些字段必须保留给审计。

运行记录

trace

最终答案通常不能解释失败。要排查智能体，得看它看到了什么、调用了什么、返回了什么、花了多久、在哪一步被拒绝、有没有重复执行副作用。trace 的作用，就是把这一串事件拆成能检查的层。

字段 建议内容 能帮你看什么
run_id 一次完整运行的稳定编号。 把一整次任务串起来。
agent_version / model 版本号、模型名、发布时间。 判断是不是升级后变了行为。
tool_name / call_id 工具名、调用编号。 判断问题出在模型还是工具。
args / result 摘要 入参摘要、出参摘要、来源标记。 判断回填是否太长、来源是否混了。
error / timeout / retry_count 错误信息、超时、重试次数。 判断循环是否卡住或重复执行。
checkpoint_id / approval 检查点编号、人工批准状态。 判断恢复点和等待点是不是完整。

好的运行记录只保留关键路径。出问题时能还原路径，才能分清该修模型、工具，还是承载边界。

你现在应该能做什么：为一次运行列出最小追踪字段，并说明其中两个字段分别帮你排查什么问题。

一条完整故障链

case

把运行承载层想清楚，最有效的方法不是再背一遍定义，而是看一条完整故障怎么一路走偏。还是用文档审阅智能体来演示：系统先进入人工批准等待，随后服务重启。恢复逻辑只知道当前停在 waiting_at，却不知道下一步动作和副作用状态。它重新回到 send_email，结果第二次发送被当成新动作，收件人收到重复邮件。

阶段 系统读什么 系统写什么 如果缺了会怎样
等待前 step、waiting_at checkpoint、next_action 重启后只能回到“在等什么”，回不到“下一步做什么”。
副作用前 approval、operation_id 外部确认号、幂等键 重试时识别不出这是同一动作。
副作用后 result、error、retry_count trace、状态摘要 只剩结果，不知道动作是否已经发生过。
断点与副作用的位置
if approval_ok:
    checkpoint(step="before_send", waiting_at="approval_wait", next_action="send_email")
    send_email(to=user_email, operation_id=operation_id)
    checkpoint(step="after_send", result="sent", operation_id=operation_id)
    trace("email_sent", operation_id=operation_id)

这段顺序里最关键的点，是把恢复点放在副作用前面，把副作用结果写进可追踪状态里。只要 checkpoint 只记到等待点，没记 next_action，重启后就会回到错误分支；只要 operation_id 没进确认表，重试就可能被当成新动作。

你现在应该能做什么：看到“重启后重复执行”的故障时，先检查恢复点有没有覆盖 next_action，再检查幂等键有没有进入确认表。

例子

example

还是看同一个文档审阅智能体。它可能在三类地方出错：重复写副作用、回填太多噪声、重启后丢步骤。下面这组案例是为了让你把“故障现象”直接对到“承载层缺口”。

[run 01] send_email() completed
[run 01] timeout while waiting for ack
[run 01] retrying same side effect
[warn] duplicate confirmation mail sent
现象 承载层缺口 先补什么
重试后同一客户收到两封确认邮件。 写操作没有幂等键，也没有外部确认状态。 先记录 operation_id 和确认号，再做恢复判断。
模型开始混淆来源，答复里塞进很多原文。 工具输出没有摘要、来源标记和长度上限。 先压缩工具结果，再把关键字段回填给模型。
人工审批一半时重启，系统不知道接哪一步。 等待点没有 checkpoint，任务状态只存在内存。 先保存当前步骤、等待点和下一步动作。

你现在应该能做什么：看到一个“系统不稳”的现象时，先问它是不是副作用、回填或恢复缺了一层，而不是先改提示词。

常见误区

debug

下面这些问题经常被说成“智能体不行”，实际上更像是承载层没定住。它们会直接变成线上故障的缺口。

错误做法 表面症状 更稳的修法
用提示词代替权限。 看起来有约束，实际还是能乱读乱写。 把工具和工作空间限制写进程序层。
只保存最终答案。 出错时不知道问题出在模型、工具还是网络。 保存 run、tool、error 和状态变化。
把会话当恢复机制。 聊天历史还在，执行位置却丢了。 把检查点和会话分开设计。
工具输出全文回填。 上下文很快被噪声塞满。 只回填摘要、来源、版本和必要结果。
没有最大轮数。 循环卡在自我修正或反复重试。 设置轮数上限、超时和升级点。
重试写操作。 重复发信、重复扣款、重复提交。 先做幂等键和副作用确认，再谈重试。

这些误区都把外部动作、状态和记录当成了模型记忆的一部分。模型可以参与判断，但承载层要把边界定住。

你现在应该能做什么：看到“智能体不稳”的反馈时，先按权限、状态、恢复、回填、轮数、幂等六个方向排查。

更新信号与边界

updates

公开更新里反复出现的词，基本都是这门课的信号：沙箱、工作空间、会话、检查点、恢复、追踪、版本迁移。它们说明智能体工程已经从“模型会说话”走向“系统能长期工作”。

沙箱与工作空间

关注文件、目录、挂载和权限边界。这个层面决定动作在哪个世界里发生。

会话与记忆

关注消息、摘要和上下文压缩。这个层面决定模型还能记住什么。

恢复与编排

关注 checkpoint、等待点、重试和人工介入。这个层面决定任务中断后怎么继续。

这一页先把循环、工具、状态和追踪放稳；外部协议、可靠性和版本迁移放到后面单元再展开。

你现在应该能做什么：读下一单元前，先把“这次讨论的是循环、工具、状态、追踪还是迁移”分清，再往下看。

操作桥接：把概念接到一个最小任务

bridge

把这一页真正落到工作里，可以从一个文档审阅智能体开始。它的目标很普通：读资料、提炼结论、保留来源、等待批准、再写入结果。这个任务之所以适合拿来练，是因为它同时包含可重放步骤和不可重复副作用，最容易看出承载层哪里该管、哪里不该管。

步骤 承载层做什么 要留下什么 如果漏掉会怎样
1. 接收任务 分配 run_id，记录资料范围和权限。 任务编号、目标、允许访问的工作空间。 后面出了问题，找不到是哪一次运行。
2. 读资料 调用检索或解析工具，限制输出长度。 来源、时间、版本、摘要、置信度。 上下文被原文塞满，来源开始混。
3. 形成草稿 把工具结果压缩成下一轮输入。 草稿版本、引用列表、待核对点。 模型看见的是噪声，不是可判断信息。
4. 等待批准 写 checkpoint，记住停在哪一步。 step、waiting_at、next_action、approval 状态。 服务重启后接不上步骤。
5. 写外部系统 执行有副作用的动作时加幂等键。 operation_id、确认号、重试结果。 重试一次就可能写两次。
6. 运行复盘 保留 trace，按 run、tool、state 三层回看。 run_id、call_id、error、latency、checkpoint_id。 只剩最终答案，无法判断错在哪层。

关键在每一步留下什么。如果你只记结果，不记来源和状态，后面就会把恢复、排查和审计混成一件事。只要把这六步写清楚，一个智能体方案能不能上生产，基本就有判断依据了。

最小上线清单
1. 工作空间和权限边界写清楚
2. tool 输入契约和超时写清楚
3. run_id / call_id / checkpoint_id 写清楚
4. 写操作加幂等键
5. 工具结果只回填摘要和来源
6. 版本变更前先做回归
7. 出现等待点前先落 checkpoint

你现在应该能做什么：把一个“能跑”的智能体需求改写成一份可检查的上线清单，里面至少要包含权限、状态、恢复、幂等和 trace。

排查路径

diagnosis

如果一个文档审阅智能体在重试后重复发信，先别从提示词改起。先复现同一条 run，再看是哪一个 tool call 触发了副作用，接着确认 checkpoint 是不是写在副作用之前，最后再看 operation_id 和 retry 策略有没有真正约束重复动作。

失败日志片段
[run 42] step=send_notification
[run 42] tool=send_email
[run 42] timeout waiting_ack
[run 42] retry=1
[warn] duplicate confirmation mail
排查顺序 看什么 说明什么 对应修法
1. 复现 run run_id、step、触发时机。 问题是不是稳定发生在同一步。 先固定一条最小复现路径。
2. 看 tool call call_id、args、result、error。 是不是工具返回慢，还是调用参数有问题。 先缩短工具输出，再补结构化错误。
3. 看 checkpoint checkpoint_id、waiting_at、next_action。 恢复点是不是在副作用之前。 把 checkpoint 写到外部动作前。
4. 看副作用 operation_id、确认号、幂等键。 重复动作有没有被显式识别。 让写操作先检查 operation_id。
5. 看 trace latency、retry_count、版本号。 问题是不是升级或重试引起的。 版本回归和重试上限一起收紧。

这条路径的目标，是把“看起来像模型失误”的现象拆回承载层问题。重复发信往往要从恢复点、幂等键和重试策略一起查。提示词只能影响模型判断，不能阻止重复副作用。

你现在应该能做什么：拿到一条故障日志时，先找 run、tool、checkpoint、operation_id、trace，再决定先补哪一层。

迁移边界

migration

这一页引用的公开资料都是可变的。只要 SDK 版本、release note、工具接口或者 checkpoint 结构一变，运行承载层的行为就可能跟着变。学习这部分时，不要只记住结论，要记住哪些字段必须重新核对。

变动项 先核对什么 为什么会影响承载层 上线前补什么
SDK 版本 Runner、会话、追踪和错误类型是否变了。 循环和记录入口可能换了。 更新回归任务和 shell 校验。
工具返回格式 来源、时间、状态、摘要字段是否还在。 回填字段变了，模型下一轮看到的东西就变了。 重新写工具契约和回填模板。
checkpoint 结构 step、waiting_at、next_action 是否还能恢复。 恢复分支依赖这些字段。 补恢复测试和断点恢复样例。
trace 语义 run_id、call_id、latency、error 是否仍可映射。 排查视图和审计字段会跟着受影响。 更新排查手册和日志脱敏规则。
迁移时的最小检查
1. 先对照 release note
2. 再核对工具返回字段
3. 再核对 checkpoint 是否可恢复
4. 再看 trace 字段是否还能映射
5. 最后跑一遍重复副作用回归

迁移时最容易犯的错，是只看“新 API 能不能编译”，却没看承载层有没有丢状态、丢幂等、丢 trace。编译通过不等于运行边界还稳，尤其是长任务和副作用动作，任何字段缺失都会变成真实故障。

你现在应该能做什么：遇到 SDK 或工具升级时，先拿这张表逐项核对，再决定要不要放行。
