# 术语与误区 {#terms}
<!-- tag: Vocabulary -->

| 术语 | 直觉解释 | 精确定义 | 常见误解 |
| --- | --- | --- | --- |
| Run | 一次从任务输入到结束的完整运行。 | 由 Runner 或等价承载层管理的一段执行过程，可能包含多个模型回合、工具调用和交接。 | 把 Run 当成一次模型 API 调用，忽略了中间可能发生多次工具往返。 |
| Turn | 循环中的一轮互动。 | 一次模型输出及其后续处理的最小推进单位，通常用于计算回合上限。 | 认为 turn 等同于用户聊天轮次；在智能体运行中，工具回填也会推动后续回合。 |
| Tool Call | 模型提出“我要用某个工具”。 | 包含工具名和结构化参数的请求，执行前仍要由宿主程序验证。 | 模型一说调用工具就立即执行，导致危险参数或越权动作绕过检查。 |
| Tool Result | 工具执行后的观察结果。 | 承载层把工具输出、错误或审批结果包装后放回模型上下文。 | 只把成功结果回填，失败时抛异常结束，模型无法修正计划。 |
| Final Output | 这次运行交付给调用方的结果。 | 当循环不再需要工具或交接，并满足输出约束时产生的最终响应。 | 把流式中间文本当成最终答案，提前展示未校验内容。 |
| Max Turns | 防止循环无限跑的护栏。 | 运行层设置的最大回合数；超过后应进入明确的失败或降级路径。 | 认为“模型会自己停”，没有给循环设置硬上限。 |
| Handoff | 把任务交给更合适的角色。 | 运行中从一个 agent 切换到另一个 agent 的控制转移，仍属于同一次运行的编排。 | 把交接当作工具调用，结果丢失当前任务的责任边界。 |
| Guardrail | 运行前后检查是否越界。 | 输入或输出侧的约束检查，触发后可阻止继续执行或阻止结果交付。 | 只在提示词里写“不要做”，却没有可执行的检查点。 |
| Human Approval | 敏感动作前让人确认。 | 工具执行前的暂停点，审批通过才继续，拒绝则把拒绝结果带回循环。 | 在工具执行后才让人确认，此时副作用可能已经发生。 |
| Checkpoint | 给长任务留恢复点。 | 持久执行系统保存的运行进度，用于中断后继续或重放。 | 把 checkpoint 当成普通日志；日志能看，checkpoint 要能恢复执行。 |

# 核心讲解 {#lesson}
<!-- tag: 8 Sections -->
<!-- layout: grid2 -->

1. 运行循环解决的是“连续行动”
普通聊天可以把模型输出直接交给用户。智能体任务不同：模型可能先判断需要查资料，再请求工具，工具返回结果后模型继续判断是否还缺信息，最后才给出结论。这种“推理、行动、观察、再推理”的往返就是运行循环。
OpenAI Agents SDK 的 Runner 文档把运行过程描述为一个会反复调用模型、执行工具、处理交接并最终产出结果的生命周期。学习这个单元时，你要把“模型输出”看成循环中的一步，而不是系统的全部。
`任务输入 → 模型回合 → 工具请求或最终输出 → 工具执行与结果回填 → 下一回合或结束`这条状态转移是一条实现路径：每个节点都要能记录、检查，并在需要时回放。

## box info
### 判断句
如果一个任务需要模型在看到工具结果后继续决定下一步，它就需要运行循环，而不只是一次模型调用。

### 2. 一次 turn 的最小结构：意图、动作、观察

turn 可以理解为运行循环的一次推进。模型看到当前上下文后，可能直接给出最终输出，也可能提出工具调用。工具调用执行后，结果会作为 observation 回填，下一次 turn 再基于这个 observation 继续。

这里最容易混淆的是：turn 不等于“用户说一句、助手回一句”。在智能体系统中，用户可能只说了一次任务，但系统在后台跑了多个 turn，每个 turn 都会消耗上下文、产生记录，并可能触发工具。

## box info
### 三个容易分不清的对象
**用户轮次**用户和系统的对话轮次，通常由产品界面定义。
**模型回合**一次模型读取上下文并生成输出，可能发生在后台。
**运行步骤**承载层记录的一次推进，可能包括模型、工具、审批或错误处理。
设计运行记录时，不要只按聊天消息计数。长任务排查更需要看到每次模型回合和工具执行。

### 3. 模型请求工具，宿主程序执行工具

工具调用的关键边界是：模型产生的是“请求”，不是“动作已经发生”。模型给出工具名和参数后，宿主程序要检查工具是否启用、参数是否符合 schema、当前用户是否有权限、是否需要人工审批、是否超过超时限制。

OpenAI Agents SDK 的工具文档区分了函数工具、托管工具和把 agent 暴露为工具等形式。无论形式如何，工程原则相同：工具面是运行层暴露给模型的能力清单，不是模型可以随意扩大权限的入口。

### 4. 工具结果要回填成“下一步可用的信息”

工具执行成功时，结果不能只是写入日志；它要以模型能理解的形式回填。工具失败时，也不一定要立刻让整个任务崩溃。很多失败可以作为观察返回，例如“文件不存在”“查询超时”“权限不足”，让模型改用另一条路径或向用户澄清。

回填结果的质量决定了下一回合的质量。含糊的返回值会让模型重复调用同一个工具；过长的返回值会挤压上下文；缺少错误类型会让模型把权限问题误判为信息缺失。

### 5. 终止条件必须是显式设计，不应只靠模型自觉

一次运行可以正常结束，也可以因为回合上限、工具超时、模型输出异常、护栏触发或人工拒绝而停止。SDK 文档中对 `max_turns`、错误处理、模型拒绝和护栏异常的说明，本质上都在提醒同一件事：循环需要硬边界。

硬边界的作用是让调用方知道发生了什么。超过回合上限时，系统应该返回“没有在预算内完成”的结构化结果，避免用户面对一个长时间无响应的任务。

### 6. 人工介入点应该放在副作用之前

人工审批适合放在会产生真实影响的动作前：发送邮件、提交表单、改写文件、发起付款、调用生产系统等。审批点的正确位置是工具执行前，而不是执行后。执行后再确认，只能变成事后通知。

一个可维护的审批设计至少要说明四件事：什么工具需要审批、审批人看到哪些参数、批准或拒绝如何回填给模型、暂停期间运行状态保存在哪里。

### 7. RunConfig 和运行参数属于控制面

模型、模型设置、追踪开关、工具行为、回合上限、上下文对象和错误处理，都属于运行控制面。把这些配置散落在业务代码里，会让同一个 agent 在不同入口表现不一致。

更稳的做法是把运行参数当成契约：哪些任务允许更高回合上限，哪些工具在某个环境中隐藏，哪些错误要转成用户可理解的输出，哪些运行必须打开追踪。这样后续升级 SDK 或调整模型时，才有明确的回归检查点。

### 8. 为持久执行留接口：先记录边界

LangGraph Durable Execution 强调 checkpoint、resume、replay、确定性和副作用隔离。放到本单元里，它提醒我们：即使当前只实现普通运行循环，也要避免把不可重复动作藏在模型回合里。

可恢复运行要求每个关键步骤都有记录：模型输入是什么、工具请求是什么、工具结果是什么、是否已经执行过会产生副作用的动作。否则进程重启后只能从头再来，或者更糟糕，重复执行已经发生过的动作。

# 运行循环案例 {#case}
<!-- tag: Walkthrough -->

场景：一个“销售异常分析助手”收到任务：读取本周订单数据，找出异常订单，生成给运营团队的摘要。它有三个工具：读取数据、统计异常、生成草稿。发送摘要需要人工确认。

**1 输入**运行层接收目标、工具清单、权限上下文。
**2 工具请求**模型提出意图，宿主程序还未执行。
**3 结果回填**工具输出整理为下一回合 observation。
**4 人工确认**副作用动作执行前暂停审批。
**5 结束**满足边界后产出 final output。

## box
### 步骤 1：输入进入运行层
用户给出目标，运行层同时带入模型、工具清单、回合上限、追踪设置和权限上下文。模型还没有执行任何工具。
控制点：运行层先决定这次 run 允许哪些工具可见。

## box
### 步骤 2：模型提出读取数据
模型请求 `read_orders(week="current")`。宿主程序检查工具是否启用、参数是否符合 schema、当前任务是否允许读取订单数据。
控制点：模型请求不等于工具已经执行。

## box
### 步骤 3：工具结果回填
读取工具返回 1,240 条订单的摘要，只把必要证据交回上下文。模型据此请求 `detect_outliers(metric="refund_rate")`，工具返回异常分组和证据。
控制点：回填结果既要足够解释下一步，又要避免无意义地占满上下文。

## box
### 步骤 4：敏感动作前暂停
模型生成给运营团队的摘要草稿，并请求发送。发送是外部副作用，因此运行层暂停，展示收件人、标题和正文摘要，等待人工确认。
控制点：人工确认放在发送前；拒绝时把拒绝原因回填给模型，让它修改草稿或结束。

## box
### 步骤 5：最终输出
人工批准后，发送工具执行成功，结果回填。模型生成最终输出：已发送、关键异常、使用的数据范围、后续建议。运行结束并保留记录。
控制点：最终输出要等任务和边界条件都满足后再交付，不能把最后一段流式文本直接当结果。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

贴近 SDK 语义的伪代码复制

```python
agent = Agent(
 name="Sales anomaly analyst",
 instructions="Analyze order anomalies and prepare an operations summary.",
 tools=[read_orders, detect_outliers, draft_summary, send_summary],
)

result = await Runner.run(
 agent,
 "Analyze this week's orders and prepare a summary for operations.",
 max_turns=8,
 run_config=run_config_with_tracing_and_tool_policy,
)

print(result.final_output)
```

| 循环点 | 模型负责 | 运行层负责 | 失败时怎么处理 |
| --- | --- | --- | --- |
| 选择读取数据 | 说明为什么需要订单数据。 | 验证工具是否可见、参数是否合法、用户是否有权限。 | 返回权限不足或请求澄清，不直接崩溃。 |
| 统计异常 | 选择指标并解释异常定义。 | 执行统计工具、限制运行时间、记录输入输出。 | 把超时或空数据作为观察回填。 |
| 发送摘要 | 生成草稿并提出发送意图。 | 暂停等待人工确认，确认后才执行发送。 | 拒绝后回填拒绝原因，让模型修改或结束。 |

# 从故障倒推边界 {#debug-chain}
<!-- tag: Debug Chain -->

运行循环出问题时，不要先改提示词。先问一个更具体的问题：这次故障发生在模型提出意图、宿主程序执行工具、结果回填，还是终止判断。能定位到循环点，修法才会落到代码和配置上。

继续沿用销售异常分析助手。假设系统在 8 个回合里反复调用 `detect_outliers`，最后没有给运营团队摘要。这个症状看起来像模型“不聪明”，但排查时要先看运行层有没有给模型可用的观察。

看到的症状
先查的边界
要打开的记录
常见修法


同一个工具被连续调用
结果回填是否清楚说明“没有更多数据”或“已经查过”。
tool result、observation 摘要、turn 序号。
把空结果、分页结束、查询范围写成结构化观察，并在超过上限时返回降级结果。


模型请求了用户无权访问的数据
工具可见性和权限校验是否基于当前用户上下文。
run config、tool list、user role、拒绝原因。
在运行开始时动态隐藏工具，或在执行前返回 `permission_denied` 观察。


发送动作发生在审批之前
审批点是否放在工具执行前。
approval state、tool call id、side effect id。
把 `send_summary` 拆成“生成草稿”和“确认后发送”，拒绝时只回填原因。


进程重启后重复发送摘要
checkpoint 是否记录了副作用已经发生，工具是否有幂等键。
checkpoint、idempotency key、发送结果。
把外部副作用封装为可去重活动，恢复时先查执行记录，再决定是否重放。

把可恢复错误转成观察复制

```python
async def run_tool_call(call, runtime):
 if not runtime.can_use(call.tool_name, call.args):
 return {
 "type": "tool_observation",
 "tool": call.tool_name,
 "ok": False,
 "error": "permission_denied",
 "recoverable": True,
 "message": "Current user cannot read order data. Ask for a permitted scope."
 }

 try:
 result = await runtime.execute(call)
 return {"type": "tool_observation", "tool": call.tool_name, "ok": True, "result": result.summary()}
 except TimeoutError:
 return {
 "type": "tool_observation",
 "tool": call.tool_name,
 "ok": False,
 "error": "timeout",
 "recoverable": True,
 "message": "The query timed out. Narrow the date range or stop with a partial result."
 }
```

这段伪代码重点看错误的形状。模型需要知道失败类型、是否可恢复、下一步应该缩小范围还是停止。运行层需要同时记录 `tool`、`ok`、`error` 和 `recoverable`，后续导出练习记录时也要保留这些线索。

做后面的练习时，可以按这个顺序自查：先说症状，再指出循环点，然后给出最小记录字段，最后给一个不会重复副作用的修法。这样写出来的答案更像工程契约，少停留在“让 agent 更谨慎”这种空泛说法。

# 常见错误 {#mistakes}
<!-- tag: Anti-patterns -->
<!-- layout: grid2 -->

## box warn
1. 没有设置回合上限
循环遇到反复失败的工具时可能一直重试。修法：为不同任务设置 `max_turns`，并设计超过上限后的降级输出。

## box warn
2. 把工具请求当成执行授权
模型参数可能不完整或越权。修法：每次工具执行前都做 schema、权限、环境和审批检查。

## box warn
3. 只记录最终答案
出错时无法判断是模型、工具、权限还是上下文问题。修法：记录每个模型回合、工具请求、工具结果和终止原因。

## box warn
4. 工具返回整块原始数据
上下文被无关内容挤满，下一回合质量下降。修法：返回摘要、证据、分页标识和必要样本。

## box warn
5. 副作用没有审批点
发送、提交、写入等动作可能在人工看到前已经发生。修法：将审批放在工具执行前，并让拒绝结果可回填。

## box warn
6. 失败只抛异常
模型无法根据失败调整计划。修法：区分可恢复错误和不可恢复错误，把可恢复错误转成结构化观察。

## box warn
7. 把流式输出当最终输出
中间 token 可能还没经过工具结果和输出约束。修法：区分 streaming event 与 final output。

## box warn
8. 把普通重试当持久执行
重试可能重复执行已经发生的副作用。修法：对关键步骤保存 checkpoint，并把不可重复动作封装成可去重的任务。

## box warn
9. SDK 升级后不做回归
运行、工具、错误处理或集成能力的更新可能改变边界行为。修法：根据更新记录检查关键路径，至少覆盖回合上限、工具审批、错误处理和追踪。

## box warn
10. 工具可见性不随上下文变化
低权限用户也看到高风险工具，模型更容易提出危险请求。修法：用运行上下文动态启用或隐藏工具。

### 17 设计

**Q:** Q17 为“文档整理助手”画运行循环。

Explain: 好的循环会写出：输入文档范围；模型请求读取文件；运行层检查路径和权限；工具返回摘要；模型生成草稿；写入前人工确认或展示 diff；批准后写入；最终输出包含改动摘要和未处理项。

### 18 边界说明

**Q:** Q18 为一个高风险工具写边界说明。

### 19 面试解释

**Q:** Q19 解释：为什么运行承载层不能只靠提示词？

Explain: 提示词是软约束，模型可能误解、遗忘或被上下文干扰。权限、终止、审批和记录需要宿主程序执行：工具不可见就不能被调用，超过上限就停止，高风险动作先审批，每一步都能追踪。
