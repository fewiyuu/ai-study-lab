工具面与工作空间

这一页接着前两单元往下走：02 讲运行循环怎么推进，03 讲工具面和工作空间怎么把“能做什么、在哪里做、怎么留痕、怎么验收”分开。读完后，你应该能为一个会读写文件或调用业务 API 的 Agent 画出边界图，而不是只会说它“能调用工具”。

学习目标

goal

这一单元只处理一件事：当智能体开始读文件、搜资料、调 API、跑代码或改写工作区时，哪些能力应该暴露给模型，哪些信息必须由运行时注入，哪些路径可以写，哪些产物算完成。

先抓主线

overview

这一页用同一个锚点例子贯穿到底：文件整理 Agent。它要读取输入目录中的文件，先生成 dry-run 计划，再在批准后执行移动或改写，最后输出 move_plan.md、move_report.json 和 trace id。这个例子足够具体，能把路径、权限、状态和验收放到一张图里看。

术语与易混点

vocabulary
术语直觉解释精确定义常见误解
工具 Tool模型可以请求外层执行的能力。由承载层实际调用的受限动作，例如搜索、写文件、发请求或更新图状态。模型一说调用，动作就自动发生。
工具面 Tool Surface给模型看的“行动菜单”。所有可见工具及其契约：名称、参数、返回、错误、批准条件。把工具面当成工具实现本身。
工作空间 Workspace工具真正读写的地方。文件、目录、运行时、依赖和副作用被限制的环境边界。只要有容器，就等于边界足够了。
InjectedState工具运行时能看到的内部状态。由运行时注入、模型 schema 不暴露的字段。让模型自己填写 user_id、workspace_id。
Command工具返回的不只是文本。同时表达状态更新和下一步路由的结构化结果。把 Command 当成复杂字符串。
Output Artifact用户或下游系统最终要拿到的文件。具有稳定路径、命名和校验方式的交付物。工具跑完就等于结果可交付。

边界提醒

这里先不展开外部协议、持久执行、追踪系统和版本迁移。它们会在后面几单元回来。现在先把工具面、工作空间和状态分开，不要把所有能力都叫成“函数调用”。

你现在应该能做什么：在讨论一个 Agent 时，能准确区分工具、工具面、工作空间、状态注入和输出物。

核心机制

mechanism

先看工具面。它要解决的是“给模型看什么”。如果工具列表过宽，模型会频繁选错；如果参数 schema 把内部状态也暴露出去，模型会开始编造 user_id、路径或权限字段。工具面契约越清楚越好。

代码
from agents import Agent, function_tool
from pydantic import Field

@function_tool
def search_project_docs(

 scope: str = Field(description="限制搜索范围，例如 input 或 docs"),
) -> str:
 """返回与 query 相关的文档摘要，找不到时返回 NOT_FOUND。"""
 ...

agent = Agent(
 name="Docs assistant",
 tools=[search_project_docs],
)

承载层看到什么

权限、超时、路径边界、日志和批准状态。

层读什么写什么错位症状
模型工具描述和参数约束。调用意图和业务参数。同一个工具被反复请求。
运行时内部状态、权限、目录策略。实际调用、错误、trace、批准记录。模型伪造内部字段，工具越权。

再看工作空间。它要解决的是“在哪里做”和“能留下什么”。这里的关键read_root、write_root、temp_root、破坏性动作审批和验收命令。路径边界如果没有写清，Agent 既可能读到不该读的内容，也可能把产物写丢。

工作空间 policy 需要落到路径和动作复制

  • read_root: ./input
  • write_root: ./output
  • temp_root: ./tmp
  • destructive_actions: require_approval
  • network: disabled_by_default
  • final_artifacts: report.md, result.json
  • verification: run uv run pytest and validate output schema

常见错误

pitfalls
  1. 工具描述过宽

模型会把相邻任务都塞给同一个工具。修法：拆成职责更窄的工具，并写清不适用场景。

  1. 把内部状态暴露给模型填写

模型可能编造 user_id、workspace_id 或权限字段。修法：用运行时注入状态。

  1. 没有错误语义

模型不知道失败是可重试、需换参还是要人工介入。修法：定义稳定错误码和下一步建议。

  1. 工作空间可写范围过大

误删、覆盖、污染仓库，或把临时文件留得到处都是。修法：限定写入根目录，并把破坏性动作设成审批项。

  1. 只返回自然语言

下游无法解析产物、状态和复验结果。修法：返回结构化摘要，同时保留文件产物和运行记录。

  1. 把工具数量当能力

工具越多，选择歧义、权限风险和测试成本越高。修法：先覆盖关键路径，再按失败案例增加工具。

看到的症状先查的边界要打开的记录常见修法
模型反复请求同一个工具结果回填是否清楚，是否告诉模型已经查过。tool result、observation 摘要、turn 序号。把空结果、分页结束和失败写成结构化观察。
模型请求了无权访问的数据工具可见性和权限是否基于当前用户上下文。run config、tool list、user role、拒绝原因。在运行开始时动态隐藏工具，或在执行前返回 permission_denied。
发送动作发生在审批之前审批点是否放在副作用动作前。approval state、tool call id、side effect id。先返回草稿，再在批准后执行发送。
进程重启后重复写入checkpoint 是否记录副作用已经发生，工具是否有幂等键。checkpoint、idempotency key、执行结果。把外部副作用封装成可去重活动，恢复时先查执行记录。

你现在应该能做什么：如果系统“看起来能跑但总出怪问题”，先从边界表和记录表里找线索，而不是先怪模型不够聪明。

审批链：把高风险动作拆成可见步骤

approval

同样是“模型想改文件”或“模型想发送摘要”，工程上要做的把动作拆成可见的几步：先给出草稿，再展示风险点，再让人批准或拒绝，最后才真正执行。这样做的目的很朴素，只有一句话：副作用一旦发生，后面再确认就晚了。

代码
def run_sensitive_tool(plan, approval):
 draft = build_draft(plan)
 if not approval["approved"]:
 return {
 "status": "rejected",
 "draft": draft,
 "reject_reason": approval.get("reason", "not approved"),
 }

 result = execute(plan)
 return {
 "status": "ok",
 "result": result,
 "audit": {"approved_by": approval["reviewer"], "operation_id": plan["operation_id"]},
 }

错误顺序

先执行再确认，会把审批变成事后通知，出问题时已经晚了。

这个链条和前面的工作空间 policy 是连在一起的。路径边界负责挡住不该碰的文件，审批链负责挡住该看但不能马上做的动作。两者缺一个，系统都会变成“看起来能跑，但一碰真实副作用就失控”。

你现在应该能做什么：为任何会改写文件、发送消息或发起外部请求的工具，先画审批链，再决定批准信息里要展示什么。

操作桥：从概念到一份最小上线清单

bridge

这节把本单元的抽象词翻回实际工作面。你要设计的一份能上线、能复验、能排错的清单。下面这张表就是最小模板。

工具列表名称、用途、参数、返回、错误、批准条件。避免模型随便调用或误选工具。

工作空间 policyread_root、write_root、temp_root、network、approval。限制路径和副作用。

状态字段run_id、workspace_root、approval、plan_id、artifact_id。让恢复和排错有抓手。

输出物清单哪些文件、摘要、报告、trace 算完成。把“做过动作”变成“可验收结果”。

验证方式测试命令、schema 检查、干跑结果、人工复核。让完成可复验，而不是靠口头确认。

故障链：路径越界怎么被挡住

diagnosis

这单元最常见的事故，工具在真正执行前没有把路径边界截住。比如文件整理 Agent 的任务只允许处理 input/，但工具意图里却出现了 ../private_notes.md。这类问题如果只靠提示词，模型下一轮还会继续试；如果把路径归一化、边界检查、拒绝原因和 trace 都放在执行链里，系统就会在副作用发生前把问题说清楚。

代码
from pathlib import Path

def guard_path(root: Path, candidate: str) -> Path:
 resolved_root = root.resolve()
 resolved_path = (root / candidate).resolve()
 if resolved_root not in resolved_path.parents and resolved_path != resolved_root:
 raise PermissionError("PATH_OUT_OF_SCOPE")
 return resolved_path

不要做什么

不要只在提示词里说“请小心”，那样挡不住真实副作用。

这段代码只做一件事：把越界路径挡在执行前。它不替模型做决策，也不替用户重新定义任务，只负责把边界变成机器可检查的结果。真正稳定的做法，是让拒绝结果本身进入 observation 和 trace，而不是只在日志里默默写一行失败。

你现在应该能做什么：看到一个会写文件的工具时，先想路径归一化和边界检查放在哪里，再想是否需要审批。

练习

diagnostic

怎么答才算会

先说对象，再说边界，再说症状。选择题用来分辨责任归属，短答题用来补状态和恢复路径，开放题用来检查你能不能把这些东西连成一套。

检查重点

重点看你有没有把工具面、工作空间、状态注入、输出物和验证方式分开；有没有把重试和幂等分开；有没有把 dry-run 和真正副作用分开。
