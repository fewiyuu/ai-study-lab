工具面与工作空间

这一页接着前两单元往下走：02 讲运行循环怎么推进，03 讲工具面和工作空间怎么把“能做什么、在哪里做、怎么留痕、怎么验收”分开。读完后，你应该能为一个会读写文件或调用业务 API 的 Agent 画出边界图，而不是只会说它“能调用工具”。

读完本页应该能做什么

  • 能把工具面、工作空间、状态注入和输出物边界分开说清楚。
  • 能为一个文件处理、资料研究或工单处理 Agent 写出最小可执行契约。
  • 能用状态表、trace 表和产物清单检查一个工具系统为什么会失控或难复验。

学习目标

goal

这一单元只处理一件事：当智能体开始读文件、搜资料、调 API、跑代码或改写工作区时，哪些能力应该暴露给模型，哪些信息必须由运行时注入，哪些路径可以写，哪些产物算完成。

对象 主要负责 输入从哪里来 没做好时的症状
工具面 定义能做什么、怎么调用、是否需要批准。 模型输出的工具意图。 模型到处找同一个工具，参数越界，错误语义混乱。
工作空间 定义在哪里做、能读什么、能写什么、如何清理。 任务上下文、目录策略、环境配置。 覆盖用户文件、读到不该看的内容、产物找不到。
状态注入 提供 user_id、workspace_id、session_id 等内部字段。 运行时上下文，而不是模型自己编造。 模型伪造身份、目录或权限字段。
输出物边界 定义哪些文件、摘要、记录和返回值算完成。 工具结果、文件产物、运行记录。 动作做了，结果却说不清。

你现在应该能做什么：拿到一个 Agent 需求时，先把“工具、工作区、状态、产物”四个词分开，再决定边界写在哪里。

先抓主线

overview

这一页用同一个锚点例子贯穿到底：文件整理 Agent。它要读取输入目录中的文件，先生成 dry-run 计划，再在批准后执行移动或改写，最后输出 move_plan.md、move_report.json 和 trace id。这个例子足够具体，能把路径、权限、状态和验收放到一张图里看。

输入用户目标、目录范围、风险提示。

承载层先接住任务，再决定暴露哪些工具和路径。

工具面list_files、propose_moves、apply_moves 等能力。

这里定义的是可调用能力，不是执行结果。

状态workspace_root、approved、plan_id、moved_files。

这些字段来自运行时，不让模型自己编造。

工作空间read_root、write_root、temp_root、destructive_actions。

路径边界决定工具能接触到什么文件和副作用。

输出move_plan.md、move_report.json、trace id。

结果要能让人复验，而不只是看见一段自然语言。

环节 读取什么 写入或改变什么 边界错位时的症状
任务输入 目录范围、任务目标、批准状态。 创建 run，设置初始策略。 任务一开始就看见过大的写权限。
模型回合 消息、工具说明、上轮观察。 工具意图、下一步计划。 模型不断提出同一组移动建议。
工具执行 工具名、参数、路径、批准状态。 文件读写、报告文件、错误码。 越权写入、重复副作用、错误不可恢复。
结果回填 工具输出、错误类型、证据摘要。 observation、trace、状态字段。 上下文被原始数据挤满，模型无法修正。
终止判断 最终输出、回合数、审批结果、产物清单。 final output、失败原因、降级状态。 任务看似结束，但没有可验收结果。

你现在应该能做什么：把一个工具系统拆成“输入、工具、状态、工作空间、输出”五段，再指出每段最容易错的地方。

术语与易混点

vocabulary
术语 直觉解释 精确定义 常见误解
工具 Tool 模型可以请求外层执行的能力。 由承载层实际调用的受限动作，例如搜索、写文件、发请求或更新图状态。 模型一说调用，动作就自动发生。
工具面 Tool Surface 给模型看的“行动菜单”。 所有可见工具及其契约：名称、参数、返回、错误、批准条件。 把工具面当成工具实现本身。
工作空间 Workspace 工具真正读写的地方。 文件、目录、运行时、依赖和副作用被限制的环境边界。 只要有容器，就等于边界足够了。
InjectedState 工具运行时能看到的内部状态。 由运行时注入、模型 schema 不暴露的字段。 让模型自己填写 user_id、workspace_id。
Command 工具返回的不只是文本。 同时表达状态更新和下一步路由的结构化结果。 把 Command 当成复杂字符串。
Output Artifact 用户或下游系统最终要拿到的文件。 具有稳定路径、命名和校验方式的交付物。 工具跑完就等于结果可交付。

边界提醒

这里先不展开外部协议、持久执行、追踪系统和版本迁移。它们会在后面几单元回来。现在先把工具面、工作空间和状态分开，不要把所有能力都叫成“函数调用”。

你现在应该能做什么：在讨论一个 Agent 时，能准确区分工具、工具面、工作空间、状态注入和输出物。

核心机制

mechanism

先看工具面。它要解决的是“给模型看什么”。如果工具列表过宽，模型会频繁选错；如果参数 schema 把内部状态也暴露出去，模型会开始编造 user_id、路径或权限字段。工具面不是功能越多越好，而是契约越清楚越好。

函数工具的关键是契约，不是函数体本身
from agents import Agent, function_tool
from pydantic import Field

@function_tool
def search_project_docs(
    query: str = Field(description="检索关键词，只填业务问题"),
    scope: str = Field(description="限制搜索范围，例如 input 或 docs"),
) -> str:
    """返回与 query 相关的文档摘要，找不到时返回 NOT_FOUND。"""
    ...

agent = Agent(
    name="Docs assistant",
    tools=[search_project_docs],
)
模型看到什么

工具名、用途、参数和错误语义。

承载层看到什么

权限、超时、路径边界、日志和批准状态。

层 读什么 写什么 错位症状
模型 工具描述和参数约束。 调用意图和业务参数。 同一个工具被反复请求。
运行时 内部状态、权限、目录策略。 实际调用、错误、trace、批准记录。 模型伪造内部字段，工具越权。

再看工作空间。它要解决的是“在哪里做”和“能留下什么”。这里的关键不是一个容器名字，而是 read_root、write_root、temp_root、破坏性动作审批和验收命令。路径边界如果没有写清，Agent 既可能读到不该读的内容，也可能把产物写丢。

工作空间 policy 需要落到路径和动作
## Workspace policy
- read_root: ./input
- write_root: ./output
- temp_root: ./tmp
- destructive_actions: require_approval
- network: disabled_by_default
- final_artifacts: report.md, result.json
- verification: run `uv run pytest` and validate output schema
边界问题 要写清的约束 没有约束的后果
能读哪里 允许读取的根目录、文件类型、隐藏文件、密钥文件。 泄露私有数据，或者把无关文件当成上下文。
能写哪里 目标目录、覆盖规则、dry-run、批准条件。 覆盖用户文件，产物找不到，工作区被污染。
能运行什么 允许命令、依赖安装、超时、网络、资源限制。 长时间卡死，环境漂移，副作用不可控。
如何验收 测试命令、输出文件清单、日志和 trace。 看似完成，却无法复验或定位失败。

状态注入解决的是“谁提供内部字段”。像 `user_id`、`workspace_id`、`session_id` 这类信息，应该由运行时注入，而不是让模型自己填写。工具执行后，结果也不总是字符串；如果它既要更新状态，又要决定下一步路由，`Command` 比普通返回值更合适。

状态注入和 Command 让内部字段不暴露给模型
from typing_extensions import Annotated
from langgraph.prebuilt import InjectedState
from langgraph.types import Command


def search_user_files(
    query: str,
    state: Annotated[dict, InjectedState],
) -> str:
    user_id = state["user_id"]
    workspace_id = state["workspace_id"]
    return scoped_search(user_id=user_id, workspace_id=workspace_id, query=query)


def create_ticket_tool(args) -> Command:
    ticket_id = create_ticket(args)
    return Command(
        update={"ticket_id": ticket_id, "ticket_status": "created"},
        goto="agent",
    )

你现在应该能做什么：判断一个字段该由模型填写，还是该由运行时注入；也能说出什么时候工具要返回状态更新，而不是只吐一句文本。

例子：文件整理 Agent

case

这个例子把工具面、工作空间和输出物串起来。Agent 读取输入目录中的文件，先生成 dry-run 计划，再在批准后执行移动或改写，最后输出 move_plan.md、move_report.json 和 trace id。它的目标不是把文件“改得很聪明”，而是把每一步都留成可复验的工程记录。

步骤 模型负责 运行层负责 失败时怎么处理
列目录 说清要按什么规则整理。 验证目录是否在 read_root 内。 返回 PATH_OUT_OF_SCOPE 或空结果观察。
提议移动 生成 move plan。 检查是否需要人工批准。 拒绝时回填审批原因，不直接执行。
执行移动 说明哪些文件可以迁移。 实际移动文件，记录 moved_files。 如果失败，保留已移动和未移动的差异。
交付结果 总结处理范围和剩余问题。 输出报告文件和 trace。 如果没有产物清单，任务不能算结束。

常见错误

    pitfalls
  

  

    
1. 工具描述过宽
模型会把相邻任务都塞给同一个工具。修法：拆成职责更窄的工具，并写清不适用场景。

    
2. 把内部状态暴露给模型填写
模型可能编造 user_id、workspace_id 或权限字段。修法：用运行时注入状态。

    
3. 没有错误语义
模型不知道失败是可重试、需换参还是要人工介入。修法：定义稳定错误码和下一步建议。

    
4. 工作空间可写范围过大
误删、覆盖、污染仓库，或把临时文件留得到处都是。修法：限定写入根目录，并把破坏性动作设成审批项。

    
5. 只返回自然语言
下游无法解析产物、状态和复验结果。修法：返回结构化摘要，同时保留文件产物和运行记录。

    
6. 把工具数量当能力
工具越多，选择歧义、权限风险和测试成本越高。修法：先覆盖关键路径，再按失败案例增加工具。

审批链：把高风险动作拆成可见步骤

    approval
  

  
同样是“模型想改文件”或“模型想发送摘要”，工程上要做的不是让模型多想一下，而是把动作拆成可见的几步：先给出草稿，再展示风险点，再让人批准或拒绝，最后才真正执行。这样做的目的很朴素，只有一句话：副作用一旦发生，后面再确认就晚了。

  

    

      
        

          步骤
          读取什么
          改变什么
          漏掉时的后果
        

      
      
        

          先出草稿
          工具参数、目标文件、收件人或移动计划。
          生成可审查的候选动作。
          人只能看到结果，来不及拦住副作用。
        

        

          展示风险点
          被改写的路径、覆盖范围、外部影响。
          把风险写进审批界面或 observation。
          审批人不知道到底会改什么。
        

        

          批准或拒绝
          人工判断和当前上下文。
          写入 approved、reject_reason。
          系统不知道这次能不能继续。
        

        

          真正执行
          批准结果、工具参数、路径边界。
          移动文件、发送摘要、写入外部系统。
          副作用被提前触发，后续很难回滚。
        

      
    

  

  
审批门要放在副作用前，而不是执行后

  
def run_sensitive_tool(plan, approval):
    draft = build_draft(plan)
    if not approval["approved"]:
        return {
            "status": "rejected",
            "draft": draft,
            "reject_reason": approval.get("reason", "not approved"),
        }

    result = execute(plan)
    return {
        "status": "ok",
        "result": result,
        "audit": {"approved_by": approval["reviewer"], "operation_id": plan["operation_id"]},
    }

  

    
正确顺序
先草稿，再审核，再执行，最后记录批准人和 operation_id。

    
错误顺序
先执行再确认，会把审批变成事后通知，出问题时已经晚了。

  

  
这个链条和前面的工作空间 policy 是连在一起的。路径边界负责挡住不该碰的文件，审批链负责挡住该看但不能马上做的动作。两者缺一个，系统都会变成“看起来能跑，但一碰真实副作用就失控”。

  
你现在应该能做什么：为任何会改写文件、发送消息或发起外部请求的工具，先画审批链，再决定批准信息里要展示什么。

故障链：路径越界怎么被挡住

    diagnosis
  

  
这单元最常见的事故，不是模型不会想，而是工具在真正执行前没有把路径边界截住。比如文件整理 Agent 的任务只允许处理 `input/`，但工具意图里却出现了 `../private_notes.md`。这类问题如果只靠提示词，模型下一轮还会继续试；如果把路径归一化、边界检查、拒绝原因和 trace 都放在执行链里，系统就会在副作用发生前把问题说清楚。

  

    

      
        

          步骤
          读取什么
          写入或改变什么
          错位时的症状
        

      
      
        

          路径归一化
          用户传入的相对路径或候选目标。
          把候选路径解析成绝对路径。
          父目录穿越没有被看见。
        

        

          边界检查
          resolved_root 和 resolved_path。
          判断是否仍在 read_root / write_root 内。
          工具碰到不该碰的文件。
        

        

          拒绝记录
          拒绝原因、调用者、原始参数。
          返回 PATH_OUT_OF_SCOPE 或 permission_denied。
          看见失败，但不知道为什么。
        

        

          复验
          trace、operation_id、输入输出摘要。
          留下可追踪的调用记录。
          下次还是只能靠猜。
        

      
    

  

  
路径守卫先挡住越界，再交给工具执行

  
from pathlib import Path

def guard_path(root: Path, candidate: str) -> Path:
    resolved_root = root.resolve()
    resolved_path = (root / candidate).resolve()
    if resolved_root not in resolved_path.parents and resolved_path != resolved_root:
        raise PermissionError("PATH_OUT_OF_SCOPE")
    return resolved_path

  

    
先做什么
先把候选路径规整成可比较的路径，再决定是否允许读写。

    
不要做什么
不要只在提示词里说“请小心”，那样挡不住真实副作用。

  

  
这段代码只做一件事：把越界路径挡在执行前。它不替模型做决策，也不替用户重新定义任务，只负责把边界变成机器可检查的结果。真正稳定的做法，是让拒绝结果本身进入 observation 和 trace，而不是只在日志里默默写一行失败。

  
你现在应该能做什么：看到一个会写文件的工具时，先想路径归一化和边界检查放在哪里，再想是否需要审批。
