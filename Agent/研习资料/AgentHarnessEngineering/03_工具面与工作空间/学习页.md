# 学习目标 {#goals}
<!-- tag: goal -->

这一单元只处理一件事：当智能体开始读文件、搜资料、调 API、跑代码或改写工作区时，哪些能力应该暴露给模型，哪些信息必须由运行时注入，哪些路径可以写，哪些产物算完成。

# 术语与易混点 {#terms}
<!-- tag: vocabulary -->

术语
直觉解释
精确定义
常见误解


工具 Tool
模型可以请求外层执行的能力。
由承载层实际调用的受限动作，例如搜索、写文件、发请求或更新图状态。
模型一说调用，动作就自动发生。


工具面 Tool Surface
给模型看的“行动菜单”。
所有可见工具及其契约：名称、参数、返回、错误、批准条件。
把工具面当成工具实现本身。


工作空间 Workspace
工具真正读写的地方。
文件、目录、运行时、依赖和副作用被限制的环境边界。
只要有容器，就等于边界足够了。


InjectedState
工具运行时能看到的内部状态。
由运行时注入、模型 schema 不暴露的字段。
让模型自己填写 user_id、workspace_id。


Command
工具返回的不只是文本。
同时表达状态更新和下一步路由的结构化结果。
把 Command 当成复杂字符串。


Output Artifact
用户或下游系统最终要拿到的文件。
具有稳定路径、命名和校验方式的交付物。
工具跑完就等于结果可交付。

# 核心机制 {#mechanism}
<!-- tag: mechanism -->

先看工具面。它要解决的是“给模型看什么”。如果工具列表过宽，模型会频繁选错；如果参数 schema 把内部状态也暴露出去，模型会开始编造 user_id、路径或权限字段。工具面契约越清楚越好。

函数工具的关键是契约，不是函数体本身复制

```python
from agents import Agent, function_tool
from pydantic import Field

@function_tool
def search_project_docs(
 query: str = Field(description="检索关键词，只填业务问题"),
 scope: str = Field(description="限制搜索范围，例如 input 或 docs"),
) -> str:
 """返回与 query 相关的文档摘要，找不到时返回 NOT_FOUND。"""
 ...

agent = Agent(
 name="Docs assistant",
 tools=[search_project_docs],
)
```

## box soft
模型看到什么
工具名、用途、参数和错误语义。

## box warn
承载层看到什么
权限、超时、路径边界、日志和批准状态。

层
读什么
写什么
错位症状


模型
工具描述和参数约束。
调用意图和业务参数。
同一个工具被反复请求。


运行时
内部状态、权限、目录策略。
实际调用、错误、trace、批准记录。
模型伪造内部字段，工具越权。

再看工作空间。它要解决的是“在哪里做”和“能留下什么”。这里的关键read_root、write_root、temp_root、破坏性动作审批和验收命令。路径边界如果没有写清，Agent 既可能读到不该读的内容，也可能把产物写丢。

工作空间 policy 需要落到路径和动作复制

```text
## Workspace policy
- read_root: ./input
- write_root: ./output
- temp_root: ./tmp
- destructive_actions: require_approval
- network: disabled_by_default
- final_artifacts: report.md, result.json
- verification: run `uv run pytest` and validate output schema
```

边界问题
要写清的约束
没有约束的后果


能读哪里
允许读取的根目录、文件类型、隐藏文件、密钥文件。
泄露私有数据，或者把无关文件当成上下文。


能写哪里
目标目录、覆盖规则、dry-run、批准条件。
覆盖用户文件，产物找不到，工作区被污染。


能运行什么
允许命令、依赖安装、超时、网络、资源限制。
长时间卡死，环境漂移，副作用不可控。


如何验收
测试命令、输出文件清单、日志和 trace。
看似完成，却无法复验或定位失败。

状态注入解决的是“谁提供内部字段”。像 `user_id`、`workspace_id`、`session_id` 这类信息，应该由运行时注入，而不是让模型自己填写。工具执行后，结果也不总是字符串；如果它既要更新状态，又要决定下一步路由，`Command` 比普通返回值更合适。

状态注入和 Command 让内部字段不暴露给模型复制

```python
from typing_extensions import Annotated
from langgraph.prebuilt import InjectedState
from langgraph.types import Command


def search_user_files(
 query: str,
 state: Annotated[dict, InjectedState],
) -> str:
 user_id = state["user_id"]
 workspace_id = state["workspace_id"]
 return scoped_search(user_id=user_id, workspace_id=workspace_id, query=query)


def create_ticket_tool(args) -> Command:
 ticket_id = create_ticket(args)
 return Command(
 update={"ticket_id": ticket_id, "ticket_status": "created"},
 goto="agent",
 )
```

你现在应该能做什么：判断一个字段该由模型填写，还是该由运行时注入；也能说出什么时候工具要返回状态更新，而不是只吐一句文本。

# 常见错误 {#mistakes}
<!-- tag: pitfalls -->
<!-- layout: grid2 -->

## box warn
1. 工具描述过宽
模型会把相邻任务都塞给同一个工具。修法：拆成职责更窄的工具，并写清不适用场景。

## box warn
2. 把内部状态暴露给模型填写
模型可能编造 user_id、workspace_id 或权限字段。修法：用运行时注入状态。

## box warn
3. 没有错误语义
模型不知道失败是可重试、需换参还是要人工介入。修法：定义稳定错误码和下一步建议。

## box warn
4. 工作空间可写范围过大
误删、覆盖、污染仓库，或把临时文件留得到处都是。修法：限定写入根目录，并把破坏性动作设成审批项。

## box warn
5. 只返回自然语言
下游无法解析产物、状态和复验结果。修法：返回结构化摘要，同时保留文件产物和运行记录。

## box warn
6. 把工具数量当能力
工具越多，选择歧义、权限风险和测试成本越高。修法：先覆盖关键路径，再按失败案例增加工具。

tool result、observation 摘要、turn 序号。
把空结果、分页结束和失败写成结构化观察。


模型请求了无权访问的数据
工具可见性和权限是否基于当前用户上下文。
run config、tool list、user role、拒绝原因。
在运行开始时动态隐藏工具，或在执行前返回 permission_denied。


发送动作发生在审批之前

# 审批链：把高风险动作拆成可见步骤 {#approval-chain}
<!-- tag: approval -->

同样是“模型想改文件”或“模型想发送摘要”，工程上要做的把动作拆成可见的几步：先给出草稿，再展示风险点，再让人批准或拒绝，最后才真正执行。这样做的目的很朴素，只有一句话：副作用一旦发生，后面再确认就晚了。

步骤
读取什么
改变什么
漏掉时的后果


先出草稿
工具参数、目标文件、收件人或移动计划。
生成可审查的候选动作。
人只能看到结果，来不及拦住副作用。


展示风险点
被改写的路径、覆盖范围、外部影响。
把风险写进审批界面或 observation。
审批人不知道到底会改什么。


批准或拒绝
人工判断和当前上下文。
写入 approved、reject_reason。
系统不知道这次能不能继续。


真正执行
批准结果、工具参数、路径边界。
移动文件、发送摘要、写入外部系统。
副作用被提前触发，后续很难回滚。

审批门要放在副作用前，而不是执行后复制

```python
def run_sensitive_tool(plan, approval):
 draft = build_draft(plan)
 if not approval["approved"]:
 return {
 "status": "rejected",
 "draft": draft,
 "reject_reason": approval.get("reason", "not approved"),
 }

 result = execute(plan)
 return {
 "status": "ok",
 "result": result,
 "audit": {"approved_by": approval["reviewer"], "operation_id": plan["operation_id"]},
 }
```

## box soft
正确顺序
先草稿，再审核，再执行，最后记录批准人和 operation_id。

## box warn
错误顺序
先执行再确认，会把审批变成事后通知，出问题时已经晚了。

这个链条和前面的工作空间 policy 是连在一起的。路径边界负责挡住不该碰的文件，审批链负责挡住该看但不能马上做的动作。两者缺一个，系统都会变成“看起来能跑，但一碰真实副作用就失控”。

你现在应该能做什么：为任何会改写文件、发送消息或发起外部请求的工具，先画审批链，再决定批准信息里要展示什么。

# 故障链：路径越界怎么被挡住 {#diagnosis-chain}
<!-- tag: diagnosis -->

这单元最常见的事故，工具在真正执行前没有把路径边界截住。比如文件整理 Agent 的任务只允许处理 `input/`，但工具意图里却出现了 `../private_notes.md`。这类问题如果只靠提示词，模型下一轮还会继续试；如果把路径归一化、边界检查、拒绝原因和 trace 都放在执行链里，系统就会在副作用发生前把问题说清楚。

步骤
读取什么
写入或改变什么
错位时的症状


路径归一化
用户传入的相对路径或候选目标。
把候选路径解析成绝对路径。
父目录穿越没有被看见。


边界检查
resolved_root 和 resolved_path。
判断是否仍在 read_root / write_root 内。
工具碰到不该碰的文件。


拒绝记录
拒绝原因、调用者、原始参数。
返回 PATH_OUT_OF_SCOPE 或 permission_denied。
看见失败，但不知道为什么。


复验
trace、operation_id、输入输出摘要。
留下可追踪的调用记录。
下次还是只能靠猜。

路径守卫先挡住越界，再交给工具执行复制

```python
from pathlib import Path

def guard_path(root: Path, candidate: str) -> Path:
 resolved_root = root.resolve()
 resolved_path = (root / candidate).resolve()
 if resolved_root not in resolved_path.parents and resolved_path != resolved_root:
 raise PermissionError("PATH_OUT_OF_SCOPE")
 return resolved_path
```

## box soft
先做什么
先把候选路径规整成可比较的路径，再决定是否允许读写。

## box warn
不要做什么
不要只在提示词里说“请小心”，那样挡不住真实副作用。

这段代码只做一件事：把越界路径挡在执行前。它不替模型做决策，也不替用户重新定义任务，只负责把边界变成机器可检查的结果。真正稳定的做法，是让拒绝结果本身进入 observation 和 trace，而不是只在日志里默默写一行失败。

你现在应该能做什么：看到一个会写文件的工具时，先想路径归一化和边界检查放在哪里，再想是否需要审批。

## box info
### 检查重点
重点看你有没有把工具面、工作空间、状态注入、输出物和验证方式分开；有没有把重试和幂等分开；有没有把 dry-run 和真正副作用分开。

Answer: NOT_FOUND,PERMISSION_DENIED,RETRYABLE_ERROR

### 6_command_use explain

**Q:** 6. 什么时候工具适合返回 Command，而不只是字符串？

Explain: 合格答案会说：当工具执行后不只是要给模型一句话，而是要同时更新状态字段并决定下一步去哪里时，用 Command 更合适。比如创建工单后，要把 ticket_id、ticket_status 写回状态，同时路由回 agent 继续后续流程。

### 7_policy checklist

**Q:** 7. 为文件整理 Agent 写一个最小 workspace policy。

Explain: 合格答案应限定 read_root、write_root 和 temp_root，并说明 destructive_actions 需要批准，network 默认关闭或受限，最后用固定命令或 schema 校验作为验证方式。重点是把路径和副作用都变成明确约束。

### 8_artifact debug-checklist

**Q:** 8. 下列哪些更像最终输出物边界的一部分？

- [ ] . move_report.json
- [ ] . trace id
- [ ] . files_written 列表
- [ ] . 模型用了几层 transformer

Answer: A,B,C

### 9_verify scenario-diagnosis

**Q:** 9. 一个文件整理 Agent 已经移动了文件，但用户说“我不知道它到底干了什么”。你会补哪三类记录？

Explain: 合格答案通常会补三类记录：一是 move_plan.md 或 move_report.json 这类文件产物；二是 workspace_root、approved、moved_files 这类状态字段；三是 run_id、operation_id、trace id 这类运行记录。这样才能把“做过动作”变成“可验收结果”。

### 10_diag_order choice

**Q:** 10. 工具误删文件后，哪种排查顺序更合理？

- [ ] 先看路径边界、批准状态、operation_id 和 trace。
- [ ] 先改按钮颜色，再加大字体。
- [ ] 先删除所有日志。

Answer: 先看路径边界、批准状态、operation_id 和 trace。

### 11_out_of_scope debug-checklist

**Q:** 11. 如果工具触碰到 `../private_notes.md`，你会先补哪两道拦截？

Explain: 合格答案会先做路径归一化和父目录穿越检查，然后把读写边界限制在 read_root 和 write_root 内。这样可以在真正执行前拒绝越界路径，而不是等副作用发生以后再补救。

### 12_bridge scenario-transfer

**Q:** 12. 设计一个“读取资料并生成摘要”的 Agent 工具面与工作空间方案，至少写出工具列表、读写路径、状态字段和验收方式。

Explain: 合格答案通常会列出 search_docs、extract_citation、write_summary 之类的工具；把 read_root 限在输入资料目录、write_root 限在 output/；保存 query、sources、gaps、draft_path 等状态字段；最后用 report.md、sources.json 和固定验证命令作为验收方式。
