状态、会话与可恢复执行

这一页接着 03 的工具面和工作空间往下走，专门拆长任务的恢复语义：会话历史给模型看，检查点给运行时继续跑，幂等键保护外部系统，追踪记录让人知道为什么会恢复成这样。

  • 能把同一个长任务拆成 session、checkpoint、业务副作用和 trace 四条线。

  • 能对重复跑一遍会不会写坏做出具体检查，也能指出要看哪一层。

学习目标

goal

这一页只解决长任务里的恢复问题：哪些内容给模型看，哪些内容给运行时续跑，哪些内容必须交给业务系统去重，哪些内容只是给人排查用。先把这四条线分开，后面的概念才不会缠在一起。

先知道，先别追

这一页先不展开外部协议、连接器、完整 durable workflow、版本迁移和多 agent 协作。它们会在后面的单元回来。现在先把 session、checkpoint、幂等键和 trace 的边界立住。

你现在应该能做什么：听到“保存上下文”时，先问对方是在说 session、checkpoint、幂等键，还是 trace。

状态账本

map

这里沿用同一个锚点例子：客服智能体处理工单 T-456，目标订单是 O-789，流程要先产出建议，再等待人工批准，最后写退款并留追踪记录。这个例子足够小，能把恢复链路看清楚，也足够真实，不会把问题简化没了。

会话

memory

OpenAI Agents SDK 里的 session 解决的是“下一次模型应该看到什么”。它会在 run 前取历史，在 run 后把新增项写回去。这样做的价值很直接：同一个工单继续问时，不用每轮手工拼历史，但也不会把执行恢复、业务去重和排查记录一起搅进来。

代码
from agents import Agent, Runner, SQLiteSession

agent = Agent(name="Support", instructions="回答要简洁，先给结论。")
session = SQLiteSession("ticket-T-456", "conversations.db")

first = await Runner.run(agent, "这张订单能退款吗？", session=session)
second = await Runner.run(agent, "继续上一步，补一段处理结论。", session=session)

# 第二次 run 会自动带上同一 session 的历史，不需要手工拼接输入项。

会话边界

session 只管模型输入历史，不管图状态从哪一步继续。它能帮用户接话，不能替 checkpoint 找回执行位置。

你现在应该能做什么：看到“继续刚才的话题”时，优先检查 session；看到“中断后继续执行”时，先别动 session，先找 checkpoint。

检查点

state

LangGraph 这一类持久执行把同一条执行线固定在 thread_id 上，再把图状态保存成 checkpoint。恢复时看同一条 thread，就能回到同一条状态线；这和聊天历史是两回事。

代码
from langgraph.types import Command, interrupt

def approval_node(state):
 decision = interrupt({
 "question": "是否批准发送退款通知？",
 "ticket_id": state["ticket_id"],
 "order_id": state["order_id"],
 })
 return {"approved": bool(decision)}

config = {"configurable": {"thread_id": "refund-T-456"}}

# 首次运行会停在 interrupt，checkpoint 会把当前状态记下来。
first = graph.stream_events({"ticket_id": "T-456", "order_id": "O-789"}, config=config, version="v3")

# 恢复时必须用同一个 thread_id，resume 值会成为 interrupt(...) 的返回值。
resumed = graph.stream_events(Command(resume=True), config=config, version="v3")

恢复实验

lab

下面这个实验把恢复中最容易出错的三种情况放在一起：同一条 thread 继续跑、thread 换了、interrupt 顺序换了。你不需要背概念，直接看恢复后到底落到哪里。

副作用

idempotency

checkpoint 只管图状态，不会替外部系统去重。退款、发信、发消息、写审批单这些外部副作用，要靠幂等键或 outbox 才能安全重跑。先问这一步能不能重复跑，再问重复跑会不会多写一次。

代码
refund_request_id = f"refund:{ticket_id}:{order_id}"

result = refund_api.upsert_refund(
 key=refund_request_id,
 order_id=order_id,
 reason="approved_by_agent",
)

# key 一样，业务系统应返回同一条结果，而不是再新建一条退款单。

恢复桥接

bridge

把 session、checkpoint、幂等键和 trace 放在同一张桥接表里，能直接看出它们什么时候接、什么时候断。这个表把谁保存什么、谁负责继续、谁负责去重、谁负责排查，放成一份操作清单。

代码
def handle_refund(agent, session, graph, ticket_id, order_id):
 thread_id = f"refund-{ticket_id}"
 refund_request_id = f"refund:{ticket_id}:{order_id}"

 with trace("refund_workflow", group_id=ticket_id, metadata={"ticket_id": ticket_id}):
 draft = await Runner.run(agent, "生成退款建议", session=session)
 decision = await graph.invoke(
 {"ticket_id": ticket_id, "order_id": order_id, "refund_request_id": refund_request_id},
 config={"configurable": {"thread_id": thread_id}},
 )
 if decision["approved"]:
 refund_api.upsert_refund(key=refund_request_id, order_id=order_id)

 flush_traces()

追踪

observability

trace 负责解释“为什么会变成这样”。它把一条长任务拆成多个 span，记录每次 run、工具调用、人工介入和恢复过程。到了排查阶段，你会想把同一个工单的多次 run 归到同一个 group_id 下面，这样恢复前后留下的证据才连得起来。

代码
from agents import Runner, trace, flush_traces

async def handle_ticket(agent, session, ticket_id):
 try:
 with trace(
 "refund_workflow",
 group_id=ticket_id,
 metadata={"ticket_id": ticket_id, "mode": "recoverable"},
 ):
 draft = await Runner.run(agent, "生成退款建议", session=session)
 final = await Runner.run(agent, f"复核这段建议：{draft.final_output}", session=session)
 return final.final_output
 finally:
 flush_traces()

trace 边界

trace 负责证据链，不能替代 session 的历史，也不能替代 checkpoint 的状态。

你现在应该能做什么：排查一次恢复事故时，先找 trace，再根据 trace 回到 thread_id、session 和幂等键。

落地检查

checklist

如果你要把一个会中断的 Agent 改成可恢复执行，先按这五步检查。顺序很重要：会话先对齐模型输入，thread_id 先对齐执行线，幂等键先对齐外部写入，trace 先对齐排查，最后再跑恢复测试。第一轮最容易错的地方，通常是把 session 继续拿去存执行状态，或者在 interrupt 前就做了不可逆写入。

下一步先看什么

官方文档先看 Sessions、Durable Execution、Persistence、Interrupts 和 Tracing。实际排查时，先找 session_id、thread_id、幂等键和 group_id 四个字段，再决定问题该往哪一层修。要是还看不出来，先故意让同一节点重跑一次，问题通常会在这一轮里暴露出来。

你现在应该能做什么：拿到一个新 Agent，按这五项找出最先会失败的地方，而不是先去加更多提示词。

退款案例

case

把四条线放回同一个流程里看，最容易看出它们各自负责什么。客服智能体先给退款建议，等人工确认，再写退款申请，最后把这一串动作记到 trace 里。

代码
state = {
 "ticket_id": "T-456",
 "order_id": "O-789",
 "refund_request_id": "refund:T-456:O-789",
 "approved": None,
 "refund_status": "not_submitted",
}

def review_refund(state):
 decision = interrupt({
 "ticket_id": state["ticket_id"],
 "order_id": state["order_id"],
 "refund_request_id": state["refund_request_id"],
 })
 return {"approved": decision}

def submit_refund(state):
 if not state["approved"]:
 return {"refund_status": "rejected"}

 result = refund_api.upsert_refund(
 key=state["refund_request_id"],
 order_id=state["order_id"],
 reason="approved_by_agent",
 )
 return {"refund_status": result.status}

会话保存什么用户和智能体围绕工单的对话、工具回包、后续补充说明。

检查点保存什么状态值、next、thread 配置、恢复点和待执行任务。

业务系统保存什么退款请求键、审批结果、提交状态和通知记录。

边界提醒

这一页只讲恢复和排查的基础面。外部协议、连接器、完整运行平台和版本迁移，留到后面的单元再看。

你现在应该能做什么：把一个“等待人工批准”的流程拆成 session、thread、幂等键和 trace 四个字段，并说明各自放在哪里。

常见错误

mistakes

先别追的东西

这里先不追外部协议、持久工作流框架、多 agent 协同和迁移策略。把恢复边界看明白以后，再去看那些更大的运行系统，理解会稳很多。

错误 1：把聊天历史当恢复状态

只把历史消息再喂给模型，恢复点就会丢掉，工具调用也更容易重复。

修法：聊天历史交给 session，执行位置交给 checkpoint。

错误 2：interrupt 前先做非幂等写入

先 create_refund 再 interrupt 等审批，恢复时节点从头再跑，create 可能再执行一次。

修法：把写入后移，或者改成 upsert 和幂等键。

错误 3：resume 时换了 thread_id

恢复时用了新 id，相当于开始一条新执行线。旧 checkpoint 找不到，流程只能重新开始。

修法：把 thread_id 存进业务任务记录，前后端都用同一个值。

错误 4：同一节点里 interrupt 顺序变动

多个 interrupt 的恢复值通常按位置对应。顺序一变，恢复输入就会落到错的字段上。

修法：保持顺序稳定，复杂表单拆节点，或者一次收集完整结构。

错误 5：把 trace 当状态库

trace 能看见发生过什么，但不能替你恢复执行位置。状态还是要放在 checkpoint 和业务系统里。

修法：trace 只做证据链，真正的状态放 checkpoint 和业务系统。

错误 6：任务结束了，trace 还没导出

后台 worker 先退场，trace 还留在缓冲里，排查时就只剩半截记录。

修法：在 trace 上下文结束后及时 flush_traces。

你现在应该能做什么：看到恢复事故时，先分成“历史没接上、位置没接上、写入没去重、记录没导出”四类，再找对应修法。
