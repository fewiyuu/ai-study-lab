目标一：看清外部边界
看到一个“让智能体查数据库、读文档、发工单”的需求时，先能问出：谁暴露能力，谁持有凭证，谁审查工具调用，谁记录副作用。

能把连接器失败拆成发现失败、认证失败、参数失败、权限失败、网络失败和副作用失败，而不是只写一个“重试”。

# 概念导图 {#map}
<!-- tag: Protocol Boundary -->

智能体运行循环
→
协议客户端
→
传输层
→
连接器 / MCP Server
→
工具、资源、提示模板
→
外部系统与副作用

直觉上，连接器像一个“受约束的翻译层”：它把外部系统的能力翻译成模型可发现、运行时可调用、平台可审计的能力。术语上，它承担协议适配、能力声明、参数约束、授权传递、结果归一化和错误表达。

边界读法

**能力边界**工具和资源说明智能体能看什么、能做什么。能力越宽，越需要过滤、审批和最小权限。
**身份边界**连接器必须明确用谁的凭证访问外部系统：用户、服务账号、租户代理，还是临时令牌。
**副作用边界**读操作、写操作、删除操作、发消息、付款、提交审批，风险级别不同，不能用同一套默认策略处理。

# 术语与误区 {#terms}
<!-- tag: Vocabulary -->

| 术语 | 直觉解释 | 精确定义 | 常见误解 |
| --- | --- | --- | --- |
| 协议 | 双方约好的说话方式。 | 一组消息格式、能力发现、请求响应、错误表达和生命周期规则。 | 以为协议只是 URL 或 SDK；实际还包括能力边界和交互语义。 |
| 连接器 | 把外部系统接到智能体运行时的适配器。 | 负责封装外部 API、数据源、认证、错误归一化、工具声明和结果返回的组件。 | 以为连接器只是函数包装；实际它是安全、授权和审计边界。 |
| MCP Server | 给模型应用暴露上下文和工具的服务端。 | 按 Model Context Protocol 提供 tools、resources、prompts 等能力，并通过标准传输与客户端通信。 | 以为 MCP server 必须部署在云端；实际也可以是本地 stdio 进程。 |
| 传输 | 消息怎么来回跑。 | MCP 规范里的 stdio 和 Streamable HTTP 等通信机制；实现还可能兼容旧 SSE。 | 把传输当业务协议本身；换传输不等于换掉工具语义。 |
| 资源 | 模型可读取的外部内容。 | MCP 中由 server 暴露、client 可读取并纳入上下文的数据对象，例如文件、记录、文档或 URI 指向的内容。 | 把资源当工具返回值；资源通常更像可发现、可选择、可读取的上下文材料。 |
| 授权 | 确认“谁可以用什么能力”。 | 围绕身份、权限范围、令牌、同意、租户隔离和过期策略建立访问控制。 | 以为只要环境变量里有 API key 就算授权；这只解决了凭证存放的一小部分。 |
| 托管连接器 | 平台替你维护的外部能力入口。 | 由模型平台或运行平台提供的连接器，例如托管 MCP 工具或文件搜索、网页搜索、代码解释器一类内置工具。 | 以为托管等于没有边界；仍要考虑工具过滤、审批、数据范围和可观测性。 |

# 核心讲解 {#lesson}
<!-- tag: 8 段讲解 -->
<!-- layout: grid2 -->

1. 为什么外部协议会成为运行承载问题
如果工具只是进程内函数，运行时只要知道函数名、参数和返回值。但外部连接器引入了另一个系统：它有自己的身份、权限、版本、延迟、错误类型和副作用。智能体运行承载层不能假装外部系统永远可用、返回永远可信、权限永远正确。
直觉上，协议把“模型想做事”翻译为“外部系统可接受的请求”。工程上，协议还要回答：能力如何被发现，参数如何被验证，失败如何被表达，危险动作如何被审批，结果如何进入上下文。
**关键判断：**只要工具调用跨出本地进程，连接器就同时承担能力适配、身份传递、错误边界和审计边界。

## box info
### 你现在应该能做什么
看到“接一个外部系统”的需求时，先不要急着写 API 调用。先列出：外部系统、协议客户端、连接器、能力声明、凭证来源、审批点和日志点。

### 2. MCP 的直觉：上下文协议

MCP，全称 Model Context Protocol，直觉上是“让模型应用用统一方式拿上下文和调用工具”的协议。它不是某个单一产品，也不是把所有 API 变成一样的魔法层。它定义的是客户端和服务端如何发现工具、读取资源、使用提示模板，以及如何通过传输发送请求。

从运行承载角度看，MCP 最有价值的地方是把外部能力从“写死在提示词里”移到“可声明、可发现、可过滤、可审批、可记录”的边界上。OpenAI Agents SDK 已经提供多种 MCP server 集成形态，包括本地 stdio、远程 Streamable HTTP、旧 SSE 兼容，以及托管 MCP 工具。不同形态的差异，不只是部署方式不同，也影响凭证、审批和网络失败处理。

> [!NOTE] 你现在应该能做什么：
> 能解释为什么 MCP 更像连接智能体和外部上下文的协议层，而不是“把任意 API 塞给模型”的快捷方式。

### 3. 工具、资源、提示模板要分开设计

很多连接器设计失败，是因为把所有东西都做成工具。工具适合表达动作：查一条记录、创建工单、发送消息、运行查询。资源适合表达上下文：文档、项目列表、配置片段、知识条目、可读取文件。提示模板适合表达可复用的交互框架：例如“根据这份事故记录生成复盘提纲”。

把资源误做成工具，会让模型每次都通过动作拿上下文，导致权限和日志变得混乱。把动作误做成资源，则会隐藏副作用。正确做法是先按“读上下文”和“做动作”拆边界，再考虑哪个能力暴露给模型，哪个能力只给人工审批或后端流程使用。

| 对象 | 适合承载 | 设计重点 | 风险 |
| --- | --- | --- | --- |
| Tool | 有输入、有执行、有结果的动作 | 参数 schema、权限、审批、幂等、错误 | 误触发外部副作用 |
| Resource | 可读取的上下文材料 | URI、可见范围、内容大小、缓存、敏感信息 | 泄露不该进入上下文的数据 |
| Prompt | 可复用任务模板 | 适用场景、变量、约束、输出格式 | 把业务规则藏进不可审计文本 |

> [!NOTE] 你现在应该能做什么：
> 能把一个连接器能力列表拆成 tools、resources、prompts，并说明每类对象的风险不同。

### 4. 传输选择：stdio、Streamable HTTP、托管连接器

传输不是最显眼的设计点，但它决定了很多运行特性。stdio 适合本地进程：客户端启动 server，通过标准输入输出交换协议消息。它适合本地工具、开发机工具、受控沙箱或命令行能力。Streamable HTTP 适合远程 server：跨网络连接、可部署到服务端、可结合 HTTP 认证和基础设施观测。旧 SSE 形态仍可能出现在已有实现中，但新设计应优先理解当前规范的 Streamable HTTP。

托管连接器则把一部分 server 生命周期、工具发现和平台集成交给平台。比如 OpenAI Agents SDK 中的 hosted MCP 工具和托管工具能力，能降低接入成本，但仍需要决定工具过滤、审批、连接器缓存、凭证范围和数据最小化。

**本地 stdio**适合可由客户端启动、需要本地文件或本地命令环境的 server。重点是进程隔离、路径限制、环境变量和超时。
**Streamable HTTP**适合远程、多客户端、可通过网关管控的 server。重点是认证、租户隔离、请求追踪、网络失败和版本兼容。
**托管连接器**适合希望快速接入平台已有能力的场景。重点是平台提供了什么控制点，哪些控制点仍要应用自己承担。

> [!NOTE] 你现在应该能做什么：
> 能根据部署位置、凭证来源、网络边界和审计要求，解释为什么某个连接器适合某种传输。

### 5. 授权不是“有 key 就行”

连接器最容易被低估的是授权。API key 只说明“请求能不能打出去”，不等于说明“模型能不能代表这个用户做这件事”。一个成熟连接器至少要区分：平台服务账号、用户 OAuth 授权、租户范围、工具级权限、资源级权限、临时凭证和人工审批。

如果一个智能体能读企业知识库又能发外部邮件，运行时必须知道两件事：读到的内容是否允许进入模型上下文，发出的内容是否需要人确认。否则，连接器会把“模型生成文本”直接升级成“外部系统副作用”。

$$
授权设计最小句式：主体是谁 → 能力是什么 → 资源范围在哪里 → 有效期多长 → 危险动作是否要审批 → 调用记录给谁看。
$$

> [!NOTE] 你现在应该能做什么：
> 能把“接入某 SaaS”改写成授权问题，而不是只问“有没有 API 文档”。

### 6. 工具过滤与审批是连接器的控制闸

OpenAI Agents SDK 的 MCP 集成支持工具过滤和人工审批一类控制点。它们背后的思想是：并不是 server 暴露的每个工具都应该直接交给模型使用，也不是每个工具调用都应该自动执行。运行承载层要在“能力发现”和“能力执行”之间加一层策略。

工具过滤可以按名称、风险、用户角色、任务阶段、租户或环境决定哪些工具可见。审批可以按副作用等级决定是否需要人工确认。比如读 Jira issue 可以自动执行，关闭 issue 或修改优先级则可能需要确认。过滤解决“看不看得见”，审批解决“能不能执行”。

> [!NOTE] 你现在应该能做什么：
> 能解释工具过滤和人工审批的差别，并能给连接器工具按风险分级。

### 7. 连接器错误要可诊断，而不是只返回失败文本

外部连接器错误通常不止一种：server 没启动、工具列表拉取失败、schema 不匹配、令牌过期、权限不足、参数验证失败、网络超时、外部 API 限流、工具产生重复副作用。对模型来说，一句“调用失败”不足以决定下一步；对工程师来说，也不足以定位根因。

可靠连接器应该把错误表达成结构化信息：错误类型、外部系统、工具名、请求标识、是否可重试、是否已产生副作用、建议恢复动作。这样运行循环才能决定重试、请求用户重新授权、降级到只读工具、要求人工介入，或停止任务。

结构化错误示例
JSON

```text
{
 "error_type": "authorization_expired",
 "system": "crm",
 "tool": "update_customer_note",
 "retryable": false,
 "side_effect": "not_started",
 "user_action": "重新授权 CRM 连接器",
 "trace_id": "run_2026_06_09_001"
}
```

> [!NOTE] 你现在应该能做什么：
> 能要求连接器返回可操作错误，而不是把所有异常压成一句自然语言。

### 8. 版本更新会改变连接器行为

智能体运行框架和协议实现更新很快。OpenAI Agents SDK 的更新记录中可以看到 MCP 支持、托管工具、代码解释器、沙箱路径、工具过滤和 MCP 相关修复持续变化。生产系统不能只看“升级成功”，还要看行为边界是否变化。

升级连接器相关依赖时，最少要回归四类测试：工具发现是否一致，工具参数 schema 是否一致，授权与审批策略是否仍生效，错误与 trace 是否仍可读。对于会写外部系统的工具，还要单独验证幂等键、重复提交保护和人工审批。

> [!NOTE] 你现在应该能做什么：
> 能把 SDK 更新记录转成回归测试清单，尤其关注 MCP、连接器、沙箱、工具过滤和追踪相关条目。

# 协议与连接器案例 {#cases}
<!-- tag: Cases -->
<!-- layout: grid2 -->

## box
### 案例一：本地文件分析 MCP server
**场景：**开发者想让智能体读取项目文件、运行轻量分析、返回候选修改点。
**连接方式：**stdio MCP server。客户端启动本地进程，server 暴露只读资源和少量分析工具。
**边界：**限制工作目录；只读资源默认可见；写文件或执行命令需要单独工具和审批；日志记录读取路径、工具名、参数摘要和结果大小。

## box warn
### 设计提醒
不要把整个用户主目录暴露成资源根目录。不要把 shell 执行工具和文件读取工具放在同一风险等级。不要在工具返回里塞入过大的文件内容。

## box
### 案例二：远程 CRM 连接器
**场景：**销售助手需要查询客户信息、追加跟进记录、生成下次联系建议。
**连接方式：**Streamable HTTP MCP server 或业务 API 包装工具。远程 server 通过 OAuth 或服务端令牌访问 CRM。
**边界：**查询客户是只读工具；追加跟进记录是写工具；修改客户状态是高风险工具。运行时按用户身份和客户归属过滤资源，并为写操作保留 trace_id 和幂等键。

## box info
### 设计提醒
如果模型读到了客户隐私信息，后续发邮件或生成外部消息时要避免越权传播。连接器需要区分“可进入上下文的数据”和“可对外输出的数据”。

## box
### 案例三：托管 MCP 工具接企业知识源
**场景：**内部问答智能体需要搜索企业知识库，并引用来源回答。
**连接方式：**平台托管 MCP 工具或托管文件搜索能力。
**边界：**应用仍要定义查询范围、用户权限映射、敏感内容过滤、工具调用记录和答案引用格式。托管降低接入成本，但不能替代业务权限判断。

## box
### 设计提醒
不要因为工具由平台托管，就默认可以跳过安全评审。托管只说明一部分运行和集成由平台负责，不说明所有内容都适合给当前用户看。

| 场景 | 推荐连接形态 | 优先控制点 | 回归测试 |
| --- | --- | --- | --- |
| 本地开发工具 | stdio MCP server | 目录限制、命令白名单、进程超时 | 工具发现、路径逃逸、失败恢复 |
| 企业 SaaS 数据 | Streamable HTTP 或托管连接器 | OAuth、租户隔离、资源过滤 | 令牌过期、权限变化、查询审计 |
| 写入外部系统 | 明确分级的工具接口 | 人工审批、幂等键、副作用记录 | 重复提交、撤销路径、审批绕过 |
| 知识检索 | 资源暴露或托管搜索工具 | 可见范围、引用、内容大小 | 越权内容、空结果、引用缺失 |

# 接入演练 {#bridge}
<!-- tag: Bridge -->

当你把一个外部系统接进智能体时，最容易出错的地方“接入后系统会不会把所有边界都混掉”。一个合格的接入演练，先要把能力发现、权限确认、调用路径和失败表达分开看。这样你才能知道模型是在看上下文，还是在真正执行动作；你也才能在审计时回答，为什么这次能看见、那次不能看见。

比较稳妥的做法是，先为每个连接器列出三个最小对象：资源、工具和审批点。资源只负责读，工具只负责做，审批点只负责放行高风险动作。然后再为它们补上最小记录：资源要有范围和大小，工具要有参数摘要和返回摘要，审批点要有谁确认、何时确认、为什么确认。只要其中一类对象没写清，后面就会出现“能用但不可审”的情况。

## box
### 接入检查
- 先列出模型能看见的能力，再列出真正能执行的能力。
- 把读、写和删除分开，不让同一把工具同时承担所有风险。
- 为高风险动作单独定义失败返回、审计字段和幂等策略。

## box info
### 失败提醒
如果一个连接器的错误只剩下一句“调用失败”，模型通常不会知道下一步该重试、换工具、请求授权还是结束任务。反过来，如果你在返回里塞了太多原始内容，排查虽然省事，却会把敏感信息带进后续上下文和日志。接入演练的目标，就是把“够用”和“不过量”卡在同一条线上。

## box info
### 连接器最小闭环
假设你要把一个资料整理助手接到云盘和任务系统里，第一版最容易犯的错就是一次性打开太多能力。看起来只是“先接上再说”，实际却会把读取、写入、删除和审批混进一条调用路径，后面一旦出现越权、空结果或重复写入，谁都说不清是资源暴露过宽，还是工具边界没定好。更稳的方式，是先只开一个只读目录和一个写入动作，把它们的授权、审计和失败返回都单独写出来。
接入时可以按四个问题往下拆。第一，这个能力对应的是资源、工具还是审批点。第二，模型能看见什么，真正能执行什么。第三，返回里有没有外部 id、时间戳和失败类型，方便后续追踪。第四，失败以后是可重试、要换范围，还是必须停下等人确认。只要这四个问题答不出来，连接器就还停留在演示阶段，不适合进入真实工作流。
检查点要看什么失败信号修法


资源范围目录、文档、对象是否最小化模型一次看到整库或整目录按任务收窄可见范围
工具边界读、写、删是否分开一把工具承担所有操作拆成单一职责动作
失败返回是否包含类型、重试性和副作用说明只剩“失败了”三个字返回结构化错误摘要
审计字段是否有主体、授权和外部 id只能靠时间猜发生了什么补全 trace 与业务记录
把这套闭环写进回归测试时，最好固定三种输入：一个只读查询，用来确认资源边界；一个需要审批的写入，用来确认授权和日志；一个明显越权的请求，用来确认拒绝是否足够清楚。只要这三种输入能稳定通过，你就不只是“连上了一个外部系统”，而是把它变成了可以审、可以查、可以回滚的工程能力。

## box info
### 接入后的回归剧本
再往前走一步，连接器的回归就不能只测“能不能连上”。你要用一个真实场景去压它：比如用户把一组公开资料放进云盘，助手先读目录，再生成摘要，最后把摘要写到任务系统。这个链路里最脆弱的地方，通常读取之后的范围是否继续扩大、写入动作有没有带上审批、失败时有没有把外部结果和内部状态分开记录。只要链路里有一个环节写不清，后续排查就会把连接器误当成模型问题。
所以回归脚本最好写成三步。第一步只读，确认模型只能看到约定范围内的资源。第二步带审批写入，确认工具调用和审计字段能一起落下。第三步故意制造一个失败，比如撤掉权限或返回空结果，确认系统会给出可操作的错误，而不是只冒出一条冷冰冰的失败消息。这样做的价值，是你能用同一套脚本同时检查可见性、执行性和恢复性。
回归步输入应该看到什么失败后怎么处理


只读公开目录摘要请求资源范围受限、返回可复述摘要先收窄资源，再查工具过滤
写入生成报告并保存审批记录、外部 id、结果摘要核对授权与幂等键
失败模拟拒绝或空返回结构化错误和下一步建议调整失败分支和提示语
练习时可以直接把这三步写进页内答案：你是否看懂了资源、工具和审批的分工，你是否知道失败信息里该保留什么，你是否能说出恢复时哪里需要幂等。答完这三个问题，连接器才算真正进入可维护阶段。

# 常见错误 {#mistakes}
<!-- tag: Anti-patterns -->

错误一：把连接器当“函数集合”
只写工具名和参数，不定义身份、资源范围、审批和日志。结果是原型能跑，生产环境无法审计。
**修法：**给每个工具标注读写属性、权限范围、风险等级、是否需要审批和 trace 字段。


错误二：把 API key 当授权模型
所有用户共用同一服务端 key，模型能读到或写入超出当前用户权限的数据。
**修法：**区分用户授权、服务账号、租户边界和工具级权限；敏感操作用临时凭证或人工确认。


错误三：资源暴露过宽
为了省事，把整个目录、整库文档或全部客户数据暴露给连接器。
**修法：**按任务和用户做资源选择；默认最小可见，必要时让模型请求更窄范围。


错误四：错误信息不可操作
工具失败只返回“失败了”，模型无法决定重试、重新授权、换工具还是停止。
**修法：**返回结构化错误，包含类型、是否可重试、是否产生副作用和建议动作。


错误五：忽略连接器版本变化
SDK 或 server 更新后，工具 schema、缓存、审批或沙箱路径变了，应用仍按旧假设运行。
**修法：**把版本更新记录转成连接器回归测试，尤其测工具发现、授权、审批、错误和 trace。


错误六：把过滤和审批混在一起
以为“模型看不到工具”就等于“危险动作安全”，或者以为“人工审批”就可以暴露所有工具。
**修法：**过滤控制可见能力，审批控制执行动作；两者配合，但不能互相替代。

### 1 选择

**Q:** 选择：MCP 最贴近哪种描述？

- [ ] . 一个把所有 API 自动变安全的插件市场
- [ ] . 一个让模型应用和上下文/工具服务端通信的协议
- [ ] . 一个只用于本地文件读取的命令行工具


### 2 判断

**Q:** 判断：连接器只要能返回工具结果，就不需要单独记录授权信息。

- [ ] 对
- [ ] 错


### 3 填空

**Q:** 填空：MCP 中更适合表示可读取上下文材料的对象是____。

Explain: resource（资源）。MCP 区分 resource 和 tool：resource 是可读取的上下文材料，不会直接产生副作用；tool 是动作，会改变外部状态。把文档、数据库记录等暴露为 resource，可以让模型在调用工具前先读取相关上下文，而不必把读取动作混在工具里。

### 4 选择

**Q:** 选择：stdio 传输最适合哪个场景？

- [ ] . 客户端启动的本地 MCP server
- [ ] . 需要公开给多个远程租户的 HTTP 服务
- [ ] . 不需要任何协议消息的静态网页


### 5 解释

**Q:** 解释：为什么不能把所有外部能力都做成工具？

Explain: 工具（tool）是执行动作，会改变外部状态并产生副作用；资源（resource）是读取上下文，只提供信息不产生副作用。如果所有外部能力都做成工具，模型会在读取文档时误触发写入，造成不必要的副作用；同时审计和权限控制会变得混乱，因为读和写共享同一个边界，无法区分哪些调用是安全的、哪些需要审批。

<!-- end-practice-group -->

<!-- practice-group: 二、边界设计 -->
<!-- quick-review: 速记 主体是谁，能做什么，看哪些资源，危险动作谁确认。 -->

### 6 场景

**Q:** 场景：知识库问答连接器应该暴露哪些资源和工具？

Explain: 资源：部门文档索引、可搜索的文档摘要、会议纪要根据权限过滤后的读取接口。工具：按部门过滤的搜索工具、基于权限的摘要生成工具、安全引用工具。权限规则：用户只能看到自己部门的数据；敏感文档需要额外审批；资源读取不产生副作用，不需要幂等；工具调用需要记录 trace 和审计日志。

### 7 场景

**Q:** 场景：CRM 写入工具需要哪些审批与幂等设计？

Explain: 审批：追加备注为低风险写，可以自动执行；修改客户状态为中风险，需要用户确认；删除标签为高风险，需要双人审批或延迟执行。幂等设计：每次写入附带 trace_id 作为幂等键；写入前查询是否已有相同 trace_id 的记录；CRM 端维护写入记录表，重复提交时直接返回已写入结果。

### 8 选择

**Q:** 选择：工具过滤主要解决什么问题？

- [ ] . 控制模型当前能看见哪些工具
- [ ] . 保证所有网络请求都不会失败
- [ ] . 自动修复外部 API 的业务错误


### 9 选择

**Q:** 选择：人工审批主要解决什么问题？

- [ ] . 对高风险动作执行前加确认
- [ ] . 让模型拥有更多上下文窗口
- [ ] . 替代所有权限校验


### 10 解释

**Q:** 解释：如何写出一个连接器授权最小句式？

Explain: 主体：用户 A（身份凭证）。能力：查询客户记录（只读）、修改客户状态（审批后写）。资源范围：自己团队的客户数据。有效期：会话级或 4 小时。审批：修改状态需要用户确认。审计：记录谁、何时、查询了哪些客户、修改了什么字段。

<!-- end-practice-group -->

<!-- practice-group: 三、错误诊断 -->
<!-- quick-review: 速记 不要只写失败；写错误类型、可重试性、副作用和用户动作。 -->

### 11 错误诊断

**Q:** 诊断：工具返回“CRM 调用失败”，缺了哪些信息？

Explain: 缺少 error_type（不知道是哪类错误）、retryable（不知道是否可重试）、side_effect（不知道是否已产生副作用）、user_action（用户应该做什么）、trace_id（和哪次调用关联）、status_code（HTTP 状态还是业务错误）、timestamp（何时发生）、affected_rows（影响范围）。没有这些信息，运行循环无法决定是重试、回滚、通知用户还是继续。

### 12 判断

**Q:** 判断：网络超时后，对所有写工具直接重试是安全的。

- [ ] 对
- [ ] 错


### 13 错误诊断

**Q:** 诊断：资源暴露过宽会造成哪些后续风险？

Explain: 上下文污染：模型会把不该读的文档内容带入后续推理。越权读取：用户 A 可能看到用户 B 的文档。输出泄露：敏感信息可能通过模型回复或日志泄露。日志敏感信息：如果资源读取记录没有脱敏，日志中会保留隐私数据。修法：按用户或部门过滤资源、对敏感字段脱敏、最小权限暴露、审计日志分级。

### 14 错误诊断

**Q:** 诊断：SDK 更新后连接器要回归哪些行为？

Explain: 工具发现：确认所有工具在升级后仍能被正确列出。schema：检查工具参数和返回值结构是否变化。授权：确认凭证和权限规则仍然生效。审批：验证高风险动作仍然需要审批。错误：检查错误类型和提示是否变化。trace：确认 trace 记录的内容和格式一致。写入操作：验证幂等和重复提交保护。

<!-- practice-group: 四、迁移与综合表达 -->

### 15 迁移

**Q:** 迁移：为会议系统设计一个连接器能力列表。

Explain: 资源：会议列表、会议纪要（按权限过滤）、会议室可用性。工具：查询会议、读取纪要、创建待办（审批后写）、发送会议总结（对外副作用，需要审批）。prompts：会议查询模板、纪要摘要模板。边界：读取纪要是资源，发送总结是工具；创建待办和发送总结需要用户确认。

### 16 迁移

**Q:** 迁移：什么时候不用 MCP，直接写业务 API 工具也合理？

Explain: 当能力数量很少、只有一两个简单 API 时，直接写业务 API 更轻量。当团队已经维护了一套成熟的 API 网关，不需要额外协议层。当部署边界很窄，只服务内部一个应用时，协议复用价值不高。当生态兼容性不是问题时，比如只支持一个模型或一个平台。MCP 的价值在于标准化、可复用和生态兼容，如果这些因素不重要，直接写 API 更合理。

### 17 综合解释

**Q:** 综合：用一段话解释外部协议与连接器为什么是运行承载层的一部分。

Explain: 智能体要执行动作，但外部系统有自己的边界：谁能访问、能做什么、出错怎么办。连接器就是运行承载层和外部系统之间的工程合同：它定义协议（MCP 或 HTTP）、暴露能力（工具和资源）、控制权限（授权和审批）、记录行为（trace 和审计）、处理失败（错误分类和恢复）。没有连接器，智能体只能调用本地函数，无法安全地扩展外部能力。

### 18 方案

**Q:** 方案题：设计一个最小可上线连接器验收清单。

Explain: 1. 协议兼容性：连接器使用 MCP 或 HTTP，schema 能被正确解析。2. 授权验证：谁、在什么范围、用什么凭证、有效期多久。3. 资源边界：只暴露最小必要资源，敏感数据脱敏。4. 工具分级：读操作自动执行，写操作需要审批。5. 错误处理：错误类型、可重试性、副作用、用户动作。6. 幂等设计：写操作附带幂等键，重复提交不重复执行。7. trace 记录：工具发现、调用参数、返回值、错误、耗时。8. 审计日志：谁、何时、调用了什么、结果如何。9. 版本回归：升级后验证工具发现、schema、授权、审批、错误、trace。10. 性能测试：正常负载和超时场景下的响应时间。11. 安全扫描：凭证不泄露、权限不越界。12. 回滚方案：灰度失败时的回滚条件和步骤。
