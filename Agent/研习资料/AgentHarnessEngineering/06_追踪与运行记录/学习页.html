Agent Harness Engineering

追踪不是日志，是运行结构。

第 06 单元 Trace Span Checkpoint Replay

一个智能体失败时，最终回答通常只告诉你“坏了”；运行记录要回答的是：哪一步坏、由谁触发、用了什么上下文、调用了哪个工具、失败前状态保存到哪里、恢复后会不会重复副作用。

本节的位置在“状态、会话与恢复”和“版本更新与迁移策略”之间：先能恢复，再要能解释每次恢复前后发生了什么。

目标与位置

学完本节，你要能从运行证据判断问题来源，而不是只读最后一段输出。

Goals
学习目标

能说清 trace、span、事件、日志、指标、checkpoint、thread_id、replay 的差异和连接方式。

工程目标

能设计一个最小运行记录格式，覆盖模型调用、工具调用、护栏、人类确认、错误、重试和恢复边界。

排查目标

能沿着父子 span、错误状态、耗时、输入输出摘要与 checkpoint id，定位失败发生在模型、工具、权限、上下文、网络还是恢复逻辑。

上一单元

外部协议与连接器：重点是工具、资源、授权和传输边界。

本单元

追踪与运行记录：重点是把运行边界记录成可读证据。

下一单元

版本更新与迁移策略：重点是用记录发现行为变化，并支持回归验证。

概念图

直觉：trace 像一次完整任务的目录，span 是目录下每个有起止时间的动作。

Map
Trace：一次端到端工作流workflow_name + trace_id + group_id + metadata
Agent Span：某个智能体开始工作智能体名、可用工具、交接对象、输出类型
Generation Span：模型生成输入摘要、输出摘要、用量、模型相关信息
Function Span：工具调用工具名、参数摘要、返回摘要、错误
Guardrail / Handoff Span护栏结果、是否交接、交给谁
Custom Span：业务边界checkpoint_id、thread_id、恢复点、外部写入 id
Trace 解决“整件事是什么”。
它把多次模型调用、工具调用和人工步骤归到同一个工作流里。
Span 解决“哪一步做了什么”。
它有开始时间、结束时间、父级关系和结构化数据，能形成调用树。
Checkpoint 解决“状态保存到哪里”。
它不是调试文本，而是恢复执行所需的图状态快照。
运行记录要把两者接起来。
同一个任务里，trace_id 用于看过程，thread_id / checkpoint_id 用于看恢复状态。
Trace 同一工作流下的完整运行树。

它回答“这次任务整体发生了什么”。

Span 带边界、父子关系和结构化数据的动作。

它回答“哪一步触发了后续错误”。

Checkpoint 保存恢复所需状态，不保存故事。

它回答“失败后要从哪里继续”。

Replay 从保存点继续执行后续步骤。

它回答“恢复后会不会重复副作用”。

trace_id + thread_id + checkpoint_id → 可恢复运行

少一个字段，过程、状态或恢复点就会断开，排查只能靠猜。

术语误区

这些词常被混用；混用会直接导致记录不可查、恢复不可控。

Terms
容易混淆 正确直觉 工程判断
日志 = 追踪 日志是事件文本或结构化消息；追踪是有父子关系和时间范围的运行树。 只有日志时，你能搜索关键词；有 trace 时，你能沿调用树定位哪个步骤触发了后续错误。
Trace = 一次 HTTP 请求 在智能体系统里，一次 trace 更适合代表一个逻辑工作流，可能包含多次模型调用和工具调用。 如果一次用户任务跨多个 Runner.run，要用更高层 trace 把它们包在一起。
Span = 普通函数日志 span 有开始、结束、父级和结构化 span_data；它应该表示一个有边界的动作。 不要把每一行调试信息都变成 span；用 span 表示模型调用、工具调用、护栏、交接和关键业务步骤。
Checkpoint = Trace checkpoint 保存状态，trace 展示过程。一个负责恢复，一个负责解释。 排查恢复失败时，需要同时记录 trace_id、thread_id、checkpoint_id。
Replay = 重放日志 replay 是从保存的检查点重新执行后续步骤；未保存的模型调用、API 请求或人工中断可能再次触发。 会产生副作用的步骤必须有任务边界和幂等策略，否则恢复或回放会重复写入。

讲解段

按“为什么记录、记录什么、怎么排查、怎么和恢复对齐”的顺序学习。

8 Parts
1

为什么智能体不能只看最终输出

智能体的输出是运行链条的最后一环。真正的错误可能发生在模型看错上下文、工具参数生成错误、工具返回被截断、权限被拒绝、护栏拦截、网络超时、恢复时重复执行，或者人工确认之后状态没有写回。只看最终输出，相当于只看事故后的结论，不看事故过程。

运行记录的目标不是“多留点日志”，而是让每个关键动作都有身份、时间、父级、输入摘要、输出摘要、错误状态和可关联的恢复点。

你现在应该能做什么：看到一个智能体失败案例时，先追问“失败发生在哪个运行步骤”，而不是马上改提示词。
2

Trace 与 Span 的基础读法

trace 是一次端到端工作流的外壳。OpenAI Agents SDK 的 trace 可带 workflow_name、trace_id、group_id 和 metadata。span 是 trace 里的一个动作，通常有 started_at、ended_at、trace_id、parent_id 和 span_data。

读 trace 时先看根节点：这次工作流是什么、属于哪个会话或任务组；再看子节点：哪个智能体运行、哪次模型生成、哪个函数工具调用、是否触发护栏、是否交接给其他智能体。父子关系比时间顺序更重要，因为它告诉你“是谁导致了谁”。

你现在应该能做什么：拿到一棵 span 树后，能分辨根工作流、智能体步骤、模型步骤、工具步骤和自定义业务步骤。
3

OpenAI Agents SDK 默认记录了什么

Agents SDK 默认启用 tracing。默认情况下，Runner.run、Runner.run_sync、Runner.run_streamed 会被包进 trace；每次智能体运行、模型生成、函数工具调用、护栏、交接，以及语音相关输入输出都会形成对应 span。也就是说，框架已经给了基础运行树。

你可以用更高层的 trace 把多次 Runner.run 合成一个逻辑工作流；也可以用 custom_span 标记自己的业务边界，例如“写入结果”“生成下载文件”“保存审批记录”。长运行 worker 中，默认批量导出可能不是即时可见；如果一个任务结束后需要立刻看到记录，可以在 trace 结束后调用 flush_traces。

你现在应该能做什么：知道哪些记录 SDK 已经自动生成，哪些业务边界需要自己补 custom_span。
4

记录字段要服务于排查，而不是堆满所有内容

一个有用的 span 至少要能回答：它做什么、何时开始结束、父级是谁、输入是什么类型、输出是否成功、失败是什么错误、是否重试、关联了哪个状态或外部写入。对模型生成与工具调用，输入输出很有价值，但也可能包含敏感数据。

OpenAI Agents SDK 的 generation_span 和 function_span 可能记录模型输入输出、工具输入输出；默认敏感数据采集是开启的，可以通过配置关闭。实践中不要把原始隐私内容、密钥、完整客户数据直接放入 metadata；更稳妥的是记录摘要、数量、哈希、外部记录 id 和脱敏后的错误类别。

你现在应该能做什么：设计 span_data 时能区分“排查必需字段”和“不该进入追踪系统的敏感原文”。
5

Durable Execution 让运行记录有了恢复坐标

LangGraph 的持久化层会把图状态保存为 checkpoints，并按 threads 组织。启用 checkpointer 后，每个 super-step 边界会形成状态快照；如果某一步失败，可以从最后成功的步骤恢复。它还支持查看状态历史、time travel、pending writes 等能力。

这里要分清：trace 记录“过程”，checkpoint 记录“状态”。一个排查页面最好能从 trace span 跳到对应 thread_id 和 checkpoint_id，也能从某个 checkpoint 找到当时执行的 trace。这样你既能看“当时为什么这么走”，也能看“恢复时会从哪里继续”。

你现在应该能做什么：解释为什么 trace_id 不能替代 checkpoint_id，也能说明两者需要用相关 id 连接起来。
6

回放不是从断掉的代码行继续

持久执行的关键点是：恢复时通常不是从原来崩溃的那一行继续，而是从合适的保存点重新执行后续步骤。LangGraph Durable Execution 要求工作流具备确定性；随机数、模型调用、文件写入、API 请求这类非确定性或带副作用的操作，应当放入任务或节点边界中。

如果一个节点里连着做了三件外部写入，恢复时很难知道哪件已经完成。更好的做法是把每个副作用拆成独立任务，并使用幂等键或“先查是否已写入”的策略。幂等的意思是：同一操作重复执行，也不会产生第二份不该有的结果。

你现在应该能做什么：看到会写文件、发请求、创建记录的步骤时，能判断它是否需要任务边界和幂等键。
7

排查 trace 的实用顺序

第一步看根 trace：workflow_name 是否正确，group_id 是否把同一会话串起来。第二步看错误 span：错误是模型、工具、护栏、交接还是自定义业务步骤。第三步看上游 span：错误前模型是否生成了错误工具参数，工具是否返回了异常或空结果，护栏是否修改了路径。

第四步看耗时：最长 span 可能暴露网络、外部 API、模型延迟或重试。第五步看恢复坐标：当前 checkpoint 是否已经包含成功节点的写入，replay 会从哪个边界开始，后续步骤是否会再次触发外部副作用。第六步把失败 trace 与成功 trace 对比，找结构差异，而不是只比文本。

你现在应该能做什么：按固定顺序阅读 trace，定位错误来源，并决定是修工具契约、上下文、权限、网络重试还是恢复逻辑。
8

运行记录最终要进入运营闭环

开发阶段的 trace 用来调试；生产阶段的运行记录还要支撑监控、告警、审计、成本分析和回归验证。常见指标包括成功率、工具错误率、护栏触发率、平均耗时、长尾耗时、重试次数、模型用量、恢复次数和重复副作用拦截次数。

记录策略要有保留期限和脱敏规则。高价值字段应结构化保存，方便聚合；高风险原文要少存或不存。版本升级时，用历史 trace 选择回归样本，比较 span 结构、工具调用参数、错误率和恢复路径，比只比较最终回答更可靠。

你现在应该能做什么：把单次追踪扩展成可持续运营的记录方案，并知道哪些字段适合做指标。

Tracing 案例

用一个“资料整理助手”示例，看 trace 与 checkpoint 如何协同定位问题。

Case

场景

用户要求助手读取一组公开资料，生成摘要，保存一份报告，并在失败后继续。一次失败发生在“保存报告”之后、“发送完成通知”之前。我们希望知道报告是否已经写入，恢复时会不会重复写入。

A

根 trace：workflow_name = research_report，group_id = conversation_42。

B

模型 span：生成计划，决定调用 read_sources 与 write_report。

C

工具 span：read_sources 成功，返回 6 条来源摘要。

D

自定义 span：write_report 记录 report_id、idempotency_key、checkpoint_id。

E

错误 span：notify_user 网络超时；检查 checkpoint，确认 write_report 的任务写入已保存。

OpenAI Agents SDK 示例

trace_workflow.py Python 
from agents import Agent, Runner, RunConfig, custom_span, flush_traces, trace

async def run_report_workflow(user_input: str, conversation_id: str):
    agent = Agent(
        name="report_agent",
        instructions="读取资料，生成摘要，并返回保存建议。"
    )

    run_config = RunConfig(
        trace_include_sensitive_data=False
    )

    try:
        with trace(
            "research_report",
            group_id=conversation_id,
            metadata={"workflow_version": "2026-06", "record_policy": "redacted"}
        ):
            result = await Runner.run(agent, user_input, run_config=run_config)

            with custom_span(
                "persist_report",
                data={
                    "report_id": "report_20260609_001",
                    "idempotency_key": "report_20260609_001",
                    "checkpoint_id": "ckpt_after_report_write"
                }
            ):
                save_report_once(result.final_output)

            return result.final_output
    finally:
        flush_traces()

要点：高层 trace 包住完整工作流；自动 span 记录智能体、模型和工具；custom_span 记录业务写入边界；关闭敏感原文采集；任务结束后刷新导出。

如何判断是否重复写入

1

查 trace 中 persist_report span 是否成功结束，并记录了 report_id。

2

查 state history 中对应 checkpoint 是否包含报告写入后的状态。

3

恢复前先用 idempotency_key 查询外部系统是否已有同一报告。

4

如果已有报告，恢复流程跳过写入，只继续通知或后续步骤。

5

如果没有报告，重新执行写入任务，并保留同一个幂等键。

追踪排查工作法

真正排查一个失败运行时，顺序很重要。先看根 trace，确认这次工作流是不是你要找的那一条；再看最晚失败的 span，判断问题是出在模型、工具、护栏还是人工确认；接着把失败 span 往上游回溯，核对工具参数和返回摘要；最后再回到 checkpoint，看当前状态是否已经包含某一步成功写入的结果。这样做的好处，是你不会把“恢复后又执行了一次”的问题误判成“模型又想了一次”。

如果你把这一步做扎实，后面的修法会明显更稳定：trace 负责告诉你哪一步坏了，checkpoint 负责告诉你恢复时会从哪里继续，幂等键负责告诉你外部系统是不是已经做过这件事。三者如果缺一个，排查就会变成猜测。对于写入型动作，最小闭环应该是“先记录 intent，再执行副作用，再写入 checkpoint，再导出 trace”。

证据重点看什么错位症状下一步
根 traceworkflow_name、group_id找错了运行线先校正任务边界
错误 span错误类型、父级、工具名只看见失败文本回溯上游输入
checkpointstate、thread_id、恢复点恢复后重复执行核对幂等和写入记录
trace diff工具顺序、耗时、用量升级后行为改变对比 golden run

如果你已经把排查顺序固定下来，下一次看到“工具返回空结果”就不需要先改提示词。你会先问：模型为什么会生成这个参数、工具为什么只返回空、空结果是不是被上游过滤掉、恢复时是否把成功的写入又触发了一次。这个顺序能把很多看似复杂的事故，拆成几步非常确定的检查动作。

再往前走一步，你就会发现追踪记录本身也应该服务于组织协作。值班同学需要的是“这一步失败了没有、失败前最后一个成功动作是什么、恢复后会不会重复执行”，产品或运营需要的是“用户有没有拿到结果、多久能恢复、是否需要人工介入”，研发则需要的是“哪一层 span 缺字段、哪一个 checkpoint 没有写入、哪一个副作用没有幂等”。如果记录只满足一种人，它就会在别的环节失效；如果记录能同时回答这三类问题，trace 才算真的进入可运营状态。

恢复前先问

  • 失败是否已经影响外部系统，还是只停在内部状态。
  • 当前 checkpoint 是否足够还原到安全的继续点。
  • 这次恢复会不会触发重复通知、重复写文件或重复扣费。

复盘后再写

把结论写成固定句式：哪条 trace 出了问题，哪一个 span 是最后成功点，哪一个 checkpoint 是恢复起点，哪一类副作用需要幂等保护。这个句式的价值不在于好看，而在于它能直接转成值班交接、事故复盘和回归测试的输入。以后每次遇到同类故障，你就不必重新发明排查方法，只需要替换其中的运行 id、工具名和错误类型。

如果你想把这套方法真正用起来，建议再加一层“记录模板”和“恢复模板”。记录模板负责把一次运行固定成同样的字段集合，比如 trace_id、group_id、thread_id、checkpoint_id、最后一个成功 span、失败类型和外部副作用 id。恢复模板则负责把这些字段变成明确动作：先判断副作用是否已经落地，再判断 checkpoint 是否可继续，再判断是否需要人工确认，最后才决定重新执行哪一步。这样一来，排查不再是看见错误就临时拼判断，而是从记录直接走向恢复。

这里最容易漏掉的一点，是 trace 和 checkpoint 的时间顺序。trace 记录的是“发生了什么”，checkpoint 记录的是“系统现在在哪儿”。如果恢复模板没有先看这两个时间点的关系，就会出现一种很典型的误判：日志上看任务已经“完成”，但其实写入动作只是部分完成；或者 checkpoint 已经写下来了，但外部系统的副作用还没来得及提交。真正靠谱的做法，是把失败前的最后一个成功 span 和 checkpoint 的状态一起读，确认系统和外部世界对同一件事是不是达成了一致。

模板字段用途恢复动作风险
trace_id / group_id找到同一运行线锁定故障上下文串错会话
thread_id / checkpoint_id定位恢复点判断从哪里继续重复执行已完成动作
最后成功 span确认边界找出最后可确认的输出把失败前动作当成失败动作
副作用 id确认外部系统状态决定跳过还是重做重复通知、重复写入

练习时你可以把一个真实任务拆成三句话写出来：这次运行是谁发起的，失败停在哪一步，恢复时为什么不会重复做事。只要这三句话能写顺，说明你已经把可观测性和可恢复性连成了一条线。你现在应该能做什么：拿到一条失败 trace，就按固定顺序判断它该重试、该跳过，还是该人工介入。

如果再往日常工作里落一步，最有用的做法是把这套模板直接塞进值班记录。每次故障只填四个空：根 trace、最后成功 span、checkpoint 位置、外部副作用 id。填完后马上写下一步动作，不要让记录停在“看过了”这种状态。这样做的好处很现实：下次同类故障出现时，你可以先比对字段，再比对现象，最后才看代码，排查时间会短很多，复盘结论也更容易被别人复用。

如果你想把它教给新人，最简单的说法就是：先找线，再找点，最后找状态。线是 trace，点是 span，状态是 checkpoint；三者连起来以后，故障就不再是模糊的一团，而是能被顺着读出来的一段过程。

把这三步固定下来，你就能把一次事故压缩成一张小卡片，卡片上只保留关键 id、最后成功点和恢复动作。这样写出来的记录，可以直接进入交接、复盘和回归，不会在下一轮排查时重新丢失上下文，下一班只看四个字段就能接手。

常见错误

多数可观测性失败，不是因为没有记录，而是记录没有工程边界。

Pitfalls
只记录自然语言日志

日志里写“开始执行工具”“执行完成”，但没有工具名、参数摘要、返回状态、父级 span 和错误类型。结果是能读故事，不能定位。

把敏感原文塞进 trace

为了排查方便保存完整提示词、客户数据、密钥或文件内容。短期好查，长期会制造合规和泄露风险。优先记录脱敏摘要和外部 id。

没有 group_id / thread_id

每次运行都是孤立记录，无法把同一会话、多次恢复和多个 trace 串起来。后续排查只能靠时间猜。

业务副作用没有 custom_span

框架自动记录模型和工具，但保存数据库、写文件、发通知这类业务边界没有记录。事故通常就发生在这些地方。

把 checkpoint 当作审计日志

checkpoint 是恢复状态，不一定适合直接阅读或长期审计。审计需要结构化事件；恢复需要状态快照；两者要关联但不能混成一个东西。

恢复时重复执行副作用

回放从保存点后重新执行，未被任务边界和幂等策略保护的外部写入可能重复发生。每个写操作都要能回答“重复执行会怎样”。
