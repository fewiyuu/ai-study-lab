07 版本更新与迁移策略｜智能体运行承载工程

智能体运行框架变化很快。真正危险的更新不只是 API 名称变化，而是默认模型、工具错误语义、会话持久化、沙箱边界和恢复逻辑发生变化。本页把更新读法、风险分级、迁移剧本和回归检查连成一套可执行方法。

核心讲解

8

01

先读版本号，不要先跑升级命令

Agents SDK 的发布说明把版本形式写成 0.Y.Z，并强调前导 0 代表 SDK 仍在快速演进。这里的关键建立一个动作：只要跨 minor 版本，就按“可能破坏公共接口或公共行为”处理。

直觉上，minor 版本像一道门。跨过去之前，要先问四个问题：公共接口有没有变，默认值有没有变，beta 功能有没有变，旧状态或旧工作区能不能被新代码继续使用。

迁移风险 = 接口变化 + 默认值变化 + 状态兼容风险 + 副作用重复风险

03

默认模型变化要当成产品行为变化

发布说明中的 0.16.0 提到 SDK 默认模型从 gpt-4.1 变为 gpt-5.4-mini，并带来 GPT-5 默认设置，例如 reasoning.effort="none" 和 verbosity="low"。如果你的 Agent 没显式写模型，升级后它并“系统换了模型和部分默认推理配置”。

迁移策略很直接：生产 Agent 显式写模型，回归测试覆盖典型任务、边界任务和拒绝场景，成本与延迟指标单独比较。若想保留旧行为，使用 Agent 或 RunConfig 的显式模型设置，或使用文档提到的默认模型环境变量。

05

沙箱边界变化要当成权限变更

0.14.0 引入 Sandbox Agents 作为 beta 功能区，文档描述了持久隔离工作区、文件、目录、Git repo、mount、snapshot 和 resume 等能力。0.17.0 又调整了本地 source materialization 的边界：来自 base_dir 外的本地主机路径需要通过 manifest 级别的额外路径授权。

这说明沙箱迁移的重点“哪些主机路径被允许进入工作区”。升级时应审计所有 LocalFile、LocalDir、mount、snapshot 和恢复流程。路径授权属于可信应用配置，不应该来自模型输出。

07

持久执行不等于副作用安全

LangGraph 的文档强调 checkpoint 能支持 human-in-the-loop、memory、time travel 和 fault tolerance，也提到 super-step、pending writes 与 durability mode。它可以避免某些成功节点在恢复时被重复执行，但外部副作用仍然需要你的应用自己设计。

如果一个节点已经发邮件、扣款、写外部数据库或触发部署，checkpoint 只能记录状态，不能自动撤销外部世界。迁移时要检查副作用节点是否幂等，是否有去重键，是否先记录 intent 再执行，是否能在恢复时判断“这件事已经做过”。

常见错误

Pitfalls

只跑单元测试，不跑行为回归

SDK 升级常常不破坏类型，却改变默认模型、错误语义和工具循环。修法是保留 golden run，并比较输出、工具序列、trace 和成本。

从模型输出生成权限授权

路径授权是可信应用配置。如果把模型输出直接变成 host path grant，迁移时会扩大文件边界。修法是白名单、只读优先、人工或配置审批。

把 checkpoint 当事务

checkpoint 保存图状态，不自动保证外部副作用原子性。修法是为副作用增加幂等键、去重表、intent 记录和恢复检查。

补充演练说明

把升级想成三层动作：第一层读 release notes，第二层找承载面，第三层做回归。很多失败都出在第二层被省略了。比如一个看起来只是模型默认值的更新，实际上会让同一条 trace 的工具调用序列、token 用量和拒绝语义一起变化；一个看起来只是沙箱路径的更新，实际上会让旧 checkpoint 需要重新跑路径授权演练。迁移剧本写得越清楚，灰度和回滚就越少依赖临场猜测。

检查清单怎么写才有用

一个有用的迁移检查清单，不会只写“升级依赖、跑测试、看结果”这种空句子。它应该能回答三类问题：第一，哪些默认值已经显式写出来，哪些还要继续锁定；第二，哪些状态需要迁移，哪些状态只要回放验证；第三，哪些失败一旦出现就必须停止灰度。写到这一步，迁移清单就不再只是文档，而是上线前的操作脚本。

实际执行时，建议把检查清单拆成四栏：变化项、影响承载面、验证方式、回滚条件。这样一来，任何一个升级点都能被放进同一个审查框架里比较。模型默认值、错误语义、沙箱路径、session 恢复、checkpoint 格式和追踪记录，都可以按这四栏去填，不会因为对象不同就写成完全不兼容的笔记。

迁移执行顺序

真正执行升级时，顺序比口号重要得多。先冻结版本锁和依赖范围，再把默认值显式写进配置，然后挑一组固定输入做 golden run，接着比对 trace、成本、错误和恢复路径，最后才进入小流量灰度。这个顺序的意义在于，你能先把最容易漂移的地方钉死，再看剩下的变化是不是可接受。

如果升级里涉及 checkpoint、session 或 sandbox，就要额外加一层状态和路径演练。换句话说，不只是模型输出会变，连失败以后怎么继续、继续时会不会重复动作、外部文件还能不能读到，都会跟着变。能过这层检查，升级才算真正安全；只要这层没做，版本号就只是看起来改过了，实际系统没有被理解。

迁移案例补充链

假设你有一个现成的资料整理智能体，升级前后最容易出问题的地方一般有三个。第一，默认模型变了，输出会更短或更快，但工具调用序列也可能少一步；第二，拒绝语义变了，原来会吞掉的失败现在可能变成显式错误；第三，沙箱路径变了，原来能读的本地资料现在可能因为授权不完整而读不到。只要这三类变化没被写进检查表，灰度阶段就会把“看起来能跑”误判成“真的安全”。

所以一个完整的迁移案例，最好先列出旧行为，再列出新行为，然后明确在哪些地方做补丁：模型显式配置、拒绝处理器、路径授权、状态迁移函数和恢复演练。最后把这些补丁放进一个 golden run 集合里，固定输入、固定工具夹具、固定回归断言，确保下次升级不会再从头猜一次。

更细一点看，迁移真正难的把升级拆成可验证的动作。你要先确认旧行为里哪些是必须保留的，比如一个任务必须能在中途恢复、一个写入动作不能重复执行、一个路径授权必须只读可见；然后确认新行为里哪些是可以接受的，比如回答更简洁、工具调用更少、拒绝更明确。只要这两类目标没有分开，团队就会在评审里争论“新版本是不是更聪明”，却没人回答“旧任务还能不能按原来的方式跑完”。

如果把这个案例放进真实上线流程，顺序应该是先做依赖锁定，再做行为快照，再做差异解释，最后才做小流量切换。行为快照要覆盖一条完整链路：输入是什么、模型选了哪些工具、每个工具的参数是什么、哪个 checkpoint 被写入、恢复后有没有重复副作用。差异解释则要把“预期变化”和“事故变化”区分开。比如输出更短可能是新模型的正常行为，但如果同样输入下工具调用少了一步、路径授权少了一层，那就迁移缺口。

快照项为什么要记升级后怎么看回滚信号

模型与提示词决定输出风格和工具倾向对比回答长度、工具序列和成本工具顺序或成本超阈值

权限与路径决定能读什么、能写什么确认只读范围和拒绝语义出现越权或空授权

状态与 checkpoint决定能否从中途恢复验证旧状态是否能继续恢复后跳错分支

副作用与幂等决定是否会重复做事检查 intent、commit 和去重表重复写入或重复通知

把这些项都写进迁移剧本以后，团队沟通也会明显更稳定。评审时不再只是说“这次升级风险不大”，而是能明确地说“模型默认值已锁定、拒绝分支已补、旧 checkpoint 已演练、路径授权已收窄、回滚条件已列出”。这种说法很平实，但它比“应该没问题”要可靠得多。真正可落地的迁移，不依赖勇气，依赖你能把每一个变化点都写成检查动作。

升级执行剧本

一个能直接拿去执行的剧本，应该从“准备、试跑、灰度、回滚、验收”五步写起。准备阶段先冻结依赖和配置，把模型版本、路径授权、状态 schema 和回归输入都写死；试跑阶段用同一条 golden run 观察工具序列、错误语义和 checkpoint 写入点；灰度阶段只放小流量，重点看失败是否集中在某一个动作，而不是整条链路都在漂；回滚阶段则要能把新版本产生的状态和旧 checkpoint 区分开，避免撤回时把旧状态也冲掉；验收阶段最后看的是业务结果、trace 差异和恢复演练是否同时通过。

这个剧本的关键，是每一步都要对应一个可以检查的证据。准备阶段看配置文件和 manifest，试跑阶段看 trace、成本和工具调用序列，灰度阶段看错误分布和恢复时间，回滚阶段看状态是否还在，验收阶段看同样输入是否仍能得到可接受的输出。只要你能把证据写清，迁移就不再是一次性赌注，而是一个可重复、可对比、可复盘的工程动作。你现在应该能做什么：拿到一个版本升级任务，先写出试跑和回滚证据，再决定能不能进灰度。

最后一步别忘了把剧本和练习绑在一起。练习题里如果只问“哪个版本更大”，学到的只是版本号；如果练习题问“升级后哪一个字段最先变、哪一个步骤最容易回滚、哪一类失败必须停灰”，学到的才是迁移判断。你可以把题目答案写成自己的上线检查单，再反过来用检查单验证一次升级。这样一来，学习页可以直接转成团队执行文档。

为了让迁移不只是概念，建议把每次升级都固定成同一个输出格式：变更说明、影响承载面、验证证据、回滚条件、复盘结论。这个格式写一次以后，下次遇到新版本就直接套用，不需要重新想结构。它的价值不在于文字整齐，而在于让团队每次都从同一组事实出发，避免把感觉当成判断。你现在应该能做什么：把一个升级任务拆成“验证前、验证中、验证后”三段，并能说出每段需要留下什么证据。

如果只记一个口诀，就记“先证据，后结论”。先拿到配置、trace、checkpoint 和回滚条件，再谈这次升级值不值得放行，这样你不会被表面的版本号带跑。

升级之后的后果，也要在剧本里提前写出来。最轻的情况是输出风格变化，团队只需要接受新的表述方式；更重的情况是工具调用顺序变化，导致某个写入动作提前或滞后；最麻烦的情况是状态恢复点变化，旧 checkpoint 不能直接续跑，必须先跑迁移函数。把这三种后果分开，才能决定是继续灰度、修补承载面，还是立刻回滚。

比如一个长任务在旧版本里跑到一半，新的版本发布后，最稳妥的做法先拿旧 thread_id 和新状态 schema 做一次恢复演练。演练里如果发现 checkpoint 读得出来，但图节点已经换名，那就先补迁移函数；如果发现恢复后第一步就会重发通知，那就先加幂等键；如果发现路径授权在新环境里变窄了，那就先收回灰度。这个案例告诉你，版本迁移把恢复、授权和副作用一起看。

回滚也要分层处理。版本回滚是把新代码撤下，状态回滚是把任务带回一个还能继续的位置，副作用回滚则通常做不到真正撤销，只能靠幂等、补偿或人工修正兜底处理。把这三层混在一起，团队就会误以为“回滚成功”意味着一切都回来了，实际上外部通知、文件写入或数据库提交可能已经发生。更稳的做法，是在剧本里明确写出：哪些变化可以直接回滚代码，哪些必须先迁移状态，哪些只能停住新会话再清理旧动作。你现在应该能做什么：看到一个旧 checkpoint，先判断它属于可续跑状态、需要迁移状态，还是已经进入只读收尾状态。

例如某次升级后，trace 里最先变短的是模型 span，紧跟着少了一次工具授权检查，结果恢复演练里第一次重跑就重复发出通知。碰到这种情况，最先修的不是文案，而是把回滚条件、幂等键和 golden run 一起补进剧本，再重新验证一遍。

练习时可以直接写一张小表，把“代码回滚、状态迁移、幂等补偿”三件事分开列出来。

。trace 记录了 LLM generation、tool call、handoff、guardrail 等每个 span 的输入输出和耗时。升级前后对比 trace，可以发现模型行为变化（如生成顺序不同、工具调用次数变化）、成本变化（延迟和 token 消耗）、错误模式变化（新的拒绝类型或错误频率）。普通日志只记录最终输出，trace 能看到中间每一步的差异。

11错误诊断

Q: 错误诊断：重复发邮件

参考答案 / 评分要点

。原因：发送邮件后进程崩溃，checkpoint 没有记录邮件已发送，恢复时重跑节点会再次发送。checkpoint 和副作用的边界：checkpoint 只保存图状态，不保存外部系统的副作用。幂等键或发送记录：发送前生成 trace_id 作为幂等键，发送后记录到外部系统或本地状态。恢复时检查：恢复后先查询是否已有相同 trace_id 的邮件记录，有则跳过。

12迁移

Q: 设计：旧 checkpoint 恢复演练

参考答案 / 评分要点

。1. 准备旧状态：记录旧 thread_id 和 checkpoint 内容。2. 升级代码：部署新版本，保留旧版本备份。3. 恢复执行：用旧 thread_id 在新代码下恢复。4. 比较 trace：对比升级前后同一任务的 trace，检查 span 差异和错误。5. 状态迁移：如果旧 checkpoint 字段不兼容，写迁移脚本或回滚。6. 验证输出：确认恢复后的任务输出和旧版本一致。

13迁移方案

Q: 迁移案例：Agents SDK 0.14 到 0.17

参考答案 / 评分要点

。升级前检查：阅读 changelog，标记 breaking 和 behavior 变更；列出使用的 API：Sandbox Agents、LocalDir、session、tracing；检查默认模型和 reasoning 设置。代码修改：更新依赖版本；显式设置模型参数；更新沙箱路径授权配置；修改错误处理逻辑。回归测试：用 golden run 集合验证普通任务、拒绝任务、工具失败、沙箱文件任务、恢复任务。灰度策略：先 5% 流量，监控拒绝率和 trace diff；异常率超过旧版本两倍时回滚。

14发布控制

Q: 写一条回滚条件

参考答案 / 评分要点

。如果拒绝场景异常率超过旧版本两倍，或恢复演练失败，或 golden run 中任一任务输出与旧版本不一致，就停止灰度并回滚。触发条件要可观测（能从 trace 和监控中读取）、和业务风险相关（不是无关的指标阈值）。

15测试设计

Q: 写一个 golden run 集合

参考答案 / 评分要点

。1. 普通任务：验证工具发现和 schema 正常。2. 拒绝任务：验证模型拒绝行为是否变化。3. 工具失败：验证错误处理和重试逻辑。4. 沙箱文件任务：验证 LocalDir 路径授权和文件读写。5. 恢复任务：验证 checkpoint 恢复和副作用幂等。每个任务对应一个迁移风险点，用固定输入确保可重复比较。

16综合解释

Q: 综合解释：把本单元讲成一个故事

参考答案 / 评分要点

。智能体 SDK 不只是普通依赖：它控制模型如何生成、如何调用工具、如何保存状态、如何处理错误。升级 SDK 可能改变默认模型、拒绝行为、沙箱边界、session 格式和 trace 内容。这些变化会影响运行循环的每个环节，所以迁移需要检查版本信号、定位承载面、验证可恢复性、确保副作用安全，并用 trace diff 验收。
