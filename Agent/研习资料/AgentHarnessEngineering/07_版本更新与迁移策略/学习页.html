Agent Harness Engineering｜Unit 07

版本更新与迁移策略

智能体运行框架变化很快。真正危险的更新不只是 API 名称变化，而是默认模型、工具错误语义、会话持久化、沙箱边界和恢复逻辑发生变化。本页把更新读法、风险分级、迁移剧本和回归检查连成一套可执行方法。

工程进阶 版本锁定 行为回归 可恢复执行

本页完成后

  • 能把 SDK 更新分成可直接吸收、需要配置锁定、必须迁移和需要灰度的几类。
  • 能解释 Agents SDK 的版本、运行、会话、追踪和沙箱更新为什么会改变智能体行为。
  • 能为一个已有智能体写出迁移清单、回归用例和恢复策略。

直觉上，升级不是“装一个新包”。智能体运行承载层会把版本变化放大：模型默认值改变会改变回答风格和成本，工具错误语义改变会改变循环是否继续，沙箱边界改变会影响文件可见性，持久化策略改变会影响中断后从哪里恢复。

术语与误区

Terms
术语 直觉解释 精确定义 常见误解
0.Y.Z 版本 还在快速演进的 SDK 版本号。 Agents SDK 发布说明把前导 0 解释为仍在快速变化；minor 版本可包含公共接口 breaking change，patch 版本用于非破坏性变更。 误以为 0.16 到 0.17 像成熟库的小版本升级一样低风险。
默认行为变更 你没写配置，但系统行为变了。 例如默认模型、默认 reasoning 设置、工具错误处理、handoff 历史包装、追踪开关等发生变化。 只看函数签名，没有检查“未显式配置”的路径。
行为回归 代码能跑，但输出、成本、权限或恢复路径变了。 升级后同一任务在模型选择、工具调用顺序、错误处理、状态保存、追踪记录或用户可见结果上偏离旧基线。 把没有异常等同于没有回归。
checkpoint 长流程的可恢复状态快照。 LangGraph 在配置 checkpointer 后按 thread 组织图状态快照，可用于 human-in-the-loop、memory、time travel 和 fault tolerance。 以为 checkpoint 会自动处理所有外部副作用，例如已经发出的邮件或已经写入的数据库。
durability 每一步状态落盘的时机。 LangGraph 文档中的持久化模式包括 sync、async、exit，分别在可靠性和性能之间取不同权衡。 只问“有没有持久化”，不问“什么时候持久化”。
迁移剧本 升级前后要做的固定动作清单。 包括版本锁定、差异阅读、配置显式化、状态兼容、回归用例、灰度范围、回滚条件和验收指标。 把迁移理解成一次性改 import 或改包版本。

核心讲解

8 Sections
01

先读版本号，不要先跑升级命令

Agents SDK 的发布说明把版本形式写成 0.Y.Z，并强调前导 0 代表 SDK 仍在快速演进。这里的关键不是记住某个版本，而是建立一个动作：只要跨 minor 版本，就按“可能破坏公共接口或公共行为”处理。

直觉上，minor 版本像一道门。跨过去之前，要先问四个问题：公共接口有没有变，默认值有没有变，beta 功能有没有变，旧状态或旧工作区能不能被新代码继续使用。

迁移风险 = 接口变化 + 默认值变化 + 状态兼容风险 + 副作用重复风险
02

把更新映射到运行承载面

同样是 SDK 更新，影响的位置不同，迁移动作也不同。模型默认值变化影响输出质量、成本和延迟；Runner 循环变化影响终止条件和错误处理；工具与 MCP 变化影响模型能否继续推进；session 与 checkpoint 变化影响恢复；sandbox 变化影响文件读写边界；tracing 变化影响排查证据。

所以阅读 release notes 时，不要只标“breaking”或“non-breaking”。更实用的标法是：它落在哪个承载面，是否影响已有任务，是否影响未显式配置的路径，是否影响中断后的恢复。

03

默认模型变化要当成产品行为变化

发布说明中的 0.16.0 提到 SDK 默认模型从 gpt-4.1 变为 gpt-5.4-mini，并带来 GPT-5 默认设置，例如 reasoning.effort="none" 和 verbosity="low"。如果你的 Agent 没显式写模型，升级后它并不是“同一个系统使用新版库”，而是“系统换了模型和部分默认推理配置”。

迁移策略很直接：生产 Agent 显式写模型，回归测试覆盖典型任务、边界任务和拒绝场景，成本与延迟指标单独比较。若想保留旧行为，使用 Agent 或 RunConfig 的显式模型设置，或使用文档提到的默认模型环境变量。

04

错误语义变化会改变运行循环

0.15.0 的发布说明把模型拒绝显式暴露为 ModelRefusalError，而不是空文本或结构化输出循环重试。这类变化非常重要，因为它改变的不是异常名称，而是 Runner 什么时候结束、什么时候继续、什么时候把错误交给业务处理。

正确的迁移不是吞掉异常，而是决定拒绝在你的系统里属于哪类结果：用户可见的安全回复、需要人工处理的事件、还是任务失败。然后为该错误配置明确的处理器，并把结果放进 trace 和回归用例。

05

沙箱边界变化要当成权限变更

0.14.0 引入 Sandbox Agents 作为 beta 功能区，文档描述了持久隔离工作区、文件、目录、Git repo、mount、snapshot 和 resume 等能力。0.17.0 又调整了本地 source materialization 的边界：来自 base_dir 外的本地主机路径需要通过 manifest 级别的额外路径授权。

这说明沙箱迁移的重点不是“工具还能不能读文件”，而是“哪些主机路径被允许进入工作区”。升级时应审计所有 LocalFile、LocalDir、mount、snapshot 和恢复流程。路径授权属于可信应用配置，不应该来自模型输出。

06

会话和 checkpoint 决定中断后恢复到哪里

Agents SDK 的 Sessions 文档提供多种会话实现，用于自动管理对话历史和跨回合恢复。LangGraph Durable Execution 则把图状态保存成 checkpoint，并按 thread 组织历史状态。两者都在回答同一个工程问题：任务中断后，系统拿什么作为继续执行的依据。

迁移时最容易漏掉的是旧状态兼容。新代码会用旧 session 或旧 checkpoint 恢复吗？状态字段改名了吗？旧工具结果还能被新工具解释吗？已有 thread_id 的历史运行会不会进入新分支？这些问题要在升级前演练，而不是等用户任务卡住时再排查。

07

持久执行不等于副作用安全

LangGraph 的文档强调 checkpoint 能支持 human-in-the-loop、memory、time travel 和 fault tolerance，也提到 super-step、pending writes 与 durability mode。它可以避免某些成功节点在恢复时被重复执行，但外部副作用仍然需要你的应用自己设计。

如果一个节点已经发邮件、扣款、写外部数据库或触发部署，checkpoint 只能记录状态，不能自动撤销外部世界。迁移时要检查副作用节点是否幂等，是否有去重键，是否先记录 intent 再执行，是否能在恢复时判断“这件事已经做过”。

08

追踪是迁移验收的证据，不只是日志

Agents SDK 文档说明 tracing 会记录 LLM generation、tool call、handoff、guardrail 和自定义事件，且默认启用。迁移时，trace 不只是事故后排查用，它也是升级前后的对照证据。

一个实用做法是保留一组 golden run：固定输入、固定工具夹具、固定业务期望。升级前后比较最终输出、工具调用序列、错误处理、token 使用、耗时、span 结构和敏感数据处理。只要这些变化无法解释，就不要把它归类为“安全升级”。

依赖锁定示例

Python 项目应把“允许自动吸收哪些版本”写清楚。下面示例表达的是：允许吸收 0.16 的 patch，暂不自动跨到 0.17。

  "openai-agents>=0.16,<0.17",
  "langgraph>=1.0,<1.1"
]

常见错误

Pitfalls

只跑单元测试，不跑行为回归

SDK 升级常常不破坏类型，却改变默认模型、错误语义和工具循环。修法是保留 golden run，并比较输出、工具序列、trace 和成本。

把 beta 功能当稳定契约

Sandbox Agents 文档明确标记为 beta。修法是把 beta 表面隔离在适配层，并对 Manifest、workspace、snapshot 和 resume 写专门测试。

从模型输出生成权限授权

路径授权是可信应用配置。如果把模型输出直接变成 host path grant，迁移时会扩大文件边界。修法是白名单、只读优先、人工或配置审批。

忽略旧 checkpoint

长任务升级后可能用新代码恢复旧状态。修法是保留状态 schema 版本，写迁移函数，并演练从旧 thread_id 继续执行。

把 checkpoint 当事务

checkpoint 保存图状态，不自动保证外部副作用原子性。修法是为副作用增加幂等键、去重表、intent 记录和恢复检查。

发布说明只读最新段

如果从 0.13 升到 0.17，中间每个 minor 的行为变化都要读。修法是逐段列出差异，不只看目标版本摘要。

补充演练说明

把升级想成三层动作：第一层读 release notes，第二层找承载面，第三层做回归。很多失败都出在第二层被省略了。比如一个看起来只是模型默认值的更新，实际上会让同一条 trace 的工具调用序列、token 用量和拒绝语义一起变化；一个看起来只是沙箱路径的更新，实际上会让旧 checkpoint 需要重新跑路径授权演练。迁移剧本写得越清楚，灰度和回滚就越少依赖临场猜测。

检查清单怎么写才有用

一个有用的迁移检查清单，不会只写“升级依赖、跑测试、看结果”这种空句子。它应该能回答三类问题：第一，哪些默认值已经显式写出来，哪些还要继续锁定；第二，哪些状态需要迁移，哪些状态只要回放验证；第三，哪些失败一旦出现就必须停止灰度。写到这一步，迁移清单就不再只是文档，而是上线前的操作脚本。

实际执行时，建议把检查清单拆成四栏：变化项、影响承载面、验证方式、回滚条件。这样一来，任何一个升级点都能被放进同一个审查框架里比较。模型默认值、错误语义、沙箱路径、session 恢复、checkpoint 格式和追踪记录，都可以按这四栏去填，不会因为对象不同就写成完全不兼容的笔记。

迁移执行顺序

真正执行升级时，顺序比口号重要得多。先冻结版本锁和依赖范围，再把默认值显式写进配置，然后挑一组固定输入做 golden run，接着比对 trace、成本、错误和恢复路径，最后才进入小流量灰度。这个顺序的意义在于，你能先把最容易漂移的地方钉死，再看剩下的变化是不是可接受。

如果升级里涉及 checkpoint、session 或 sandbox，就要额外加一层状态和路径演练。换句话说，不只是模型输出会变，连失败以后怎么继续、继续时会不会重复动作、外部文件还能不能读到，都会跟着变。能过这层检查，升级才算真正安全；只要这层没做，版本号就只是看起来改过了，实际系统没有被理解。

迁移案例补充链

假设你有一个现成的资料整理智能体，升级前后最容易出问题的地方一般有三个。第一，默认模型变了，输出会更短或更快，但工具调用序列也可能少一步；第二，拒绝语义变了，原来会吞掉的失败现在可能变成显式错误；第三，沙箱路径变了，原来能读的本地资料现在可能因为授权不完整而读不到。只要这三类变化没被写进检查表，灰度阶段就会把“看起来能跑”误判成“真的安全”。

所以一个完整的迁移案例，最好先列出旧行为，再列出新行为，然后明确在哪些地方做补丁：模型显式配置、拒绝处理器、路径授权、状态迁移函数和恢复演练。最后把这些补丁放进一个 golden run 集合里，固定输入、固定工具夹具、固定回归断言，确保下次升级不会再从头猜一次。

更细一点看，迁移真正难的不是“升级版本”，而是把升级拆成可验证的动作。你要先确认旧行为里哪些是必须保留的，比如一个任务必须能在中途恢复、一个写入动作不能重复执行、一个路径授权必须只读可见；然后确认新行为里哪些是可以接受的，比如回答更简洁、工具调用更少、拒绝更明确。只要这两类目标没有分开，团队就会在评审里争论“新版本是不是更聪明”，却没人回答“旧任务还能不能按原来的方式跑完”。

如果把这个案例放进真实上线流程，顺序应该是先做依赖锁定，再做行为快照，再做差异解释，最后才做小流量切换。行为快照要覆盖一条完整链路：输入是什么、模型选了哪些工具、每个工具的参数是什么、哪个 checkpoint 被写入、恢复后有没有重复副作用。差异解释则要把“预期变化”和“事故变化”区分开。比如输出更短可能是新模型的正常行为，但如果同样输入下工具调用少了一步、路径授权少了一层，那就不是正常变化，而是迁移缺口。

快照项为什么要记升级后怎么看回滚信号
模型与提示词决定输出风格和工具倾向对比回答长度、工具序列和成本工具顺序或成本超阈值
权限与路径决定能读什么、能写什么确认只读范围和拒绝语义出现越权或空授权
状态与 checkpoint决定能否从中途恢复验证旧状态是否能继续恢复后跳错分支
副作用与幂等决定是否会重复做事检查 intent、commit 和去重表重复写入或重复通知

把这些项都写进迁移剧本以后，团队沟通也会明显更稳定。评审时不再只是说“这次升级风险不大”，而是能明确地说“模型默认值已锁定、拒绝分支已补、旧 checkpoint 已演练、路径授权已收窄、回滚条件已列出”。这种说法很平实，但它比“应该没问题”要可靠得多。真正可落地的迁移，不依赖勇气，依赖你能把每一个变化点都写成检查动作。

升级执行剧本

一个能直接拿去执行的剧本，应该从“准备、试跑、灰度、回滚、验收”五步写起。准备阶段先冻结依赖和配置，把模型版本、路径授权、状态 schema 和回归输入都写死；试跑阶段用同一条 golden run 观察工具序列、错误语义和 checkpoint 写入点；灰度阶段只放小流量，重点看失败是否集中在某一个动作，而不是整条链路都在漂；回滚阶段则要能把新版本产生的状态和旧 checkpoint 区分开，避免撤回时把旧状态也冲掉；验收阶段最后看的是业务结果、trace 差异和恢复演练是否同时通过。

这个剧本的关键，是每一步都要对应一个可以检查的证据。准备阶段看配置文件和 manifest，试跑阶段看 trace、成本和工具调用序列，灰度阶段看错误分布和恢复时间，回滚阶段看状态是否还在，验收阶段看同样输入是否仍能得到可接受的输出。只要你能把证据写清，迁移就不再是一次性赌注，而是一个可重复、可对比、可复盘的工程动作。你现在应该能做什么：拿到一个版本升级任务，先写出试跑和回滚证据，再决定能不能进灰度。

最后一步别忘了把剧本和练习绑在一起。练习题里如果只问“哪个版本更大”，学到的只是版本号；如果练习题问“升级后哪一个字段最先变、哪一个步骤最容易回滚、哪一类失败必须停灰”，学到的才是迁移判断。你可以把题目答案写成自己的上线检查单，再反过来用检查单验证一次升级。这样一来，学习页不是看完就结束，而是可以直接转成团队执行文档。

为了让迁移不只是概念，建议把每次升级都固定成同一个输出格式：变更说明、影响承载面、验证证据、回滚条件、复盘结论。这个格式写一次以后，下次遇到新版本就直接套用，不需要重新想结构。它的价值不在于文字整齐，而在于让团队每次都从同一组事实出发，避免把感觉当成判断。你现在应该能做什么：把一个升级任务拆成“验证前、验证中、验证后”三段，并能说出每段需要留下什么证据。

如果只记一个口诀，就记“先证据，后结论”。先拿到配置、trace、checkpoint 和回滚条件，再谈这次升级值不值得放行，这样你不会被表面的版本号带跑。

升级之后的后果，也要在剧本里提前写出来。最轻的情况是输出风格变化，团队只需要接受新的表述方式；更重的情况是工具调用顺序变化，导致某个写入动作提前或滞后；最麻烦的情况是状态恢复点变化，旧 checkpoint 不能直接续跑，必须先跑迁移函数。把这三种后果分开，才能决定是继续灰度、修补承载面，还是立刻回滚。

比如一个长任务在旧版本里跑到一半，新的版本发布后，最稳妥的做法不是直接点继续，而是先拿旧 thread_id 和新状态 schema 做一次恢复演练。演练里如果发现 checkpoint 读得出来，但图节点已经换名，那就先补迁移函数；如果发现恢复后第一步就会重发通知，那就先加幂等键；如果发现路径授权在新环境里变窄了，那就先收回灰度。这个案例告诉你，版本迁移不是单点判断，而是把恢复、授权和副作用一起看。

回滚也要分层处理。版本回滚是把新代码撤下，状态回滚是把任务带回一个还能继续的位置，副作用回滚则通常做不到真正撤销，只能靠幂等、补偿或人工修正兜底。把这三层混在一起，团队就会误以为“回滚成功”意味着一切都回来了，实际上外部通知、文件写入或数据库提交可能已经发生。更稳的做法，是在剧本里明确写出：哪些变化可以直接回滚代码，哪些必须先迁移状态，哪些只能停住新会话再清理旧动作。你现在应该能做什么：看到一个旧 checkpoint，先判断它属于可续跑状态、需要迁移状态，还是已经进入只读收口状态。

例如某次升级后，trace 里最先变短的是模型 span，紧跟着少了一次工具授权检查，结果恢复演练里第一次重跑就重复发出通知。碰到这种情况，最先修的不是文案，而是把回滚条件、幂等键和 golden run 一起补进剧本，再重新验证一遍。

练习时可以直接写一张小表，把“代码回滚、状态迁移、幂等补偿”三件事分开列出来。

解释可观测性

解释：为什么 tracing 是迁移验收证据？

参考答案 / 评分要点

trace 记录了 LLM generation、tool call、handoff、guardrail 等每个 span 的输入输出和耗时。升级前后对比 trace，可以发现模型行为变化（如生成顺序不同、工具调用次数变化）、成本变化（延迟和 token 消耗）、错误模式变化（新的拒绝类型或错误频率）。普通日志只记录最终输出，trace 能看到中间每一步的差异。

回答结构：先说明 trace 记录什么，再说明升级前后比较什么，最后说明它能发现哪些普通日志看不到的问题。

checkpoint 的主要作用是什么？

错误诊断副作用

错误诊断：重复发邮件

参考答案 / 评分要点

原因：发送邮件后进程崩溃，checkpoint 没有记录邮件已发送，恢复时重跑节点会再次发送。checkpoint 和副作用的边界：checkpoint 只保存图状态，不保存外部系统的副作用。幂等键或发送记录：发送前生成 trace_id 作为幂等键，发送后记录到外部系统或本地状态。恢复时检查：恢复后先查询是否已有相同 trace_id 的邮件记录，有则跳过。

回答结构：原因｜checkpoint 与副作用的边界｜幂等键或发送记录｜恢复时检查。

迁移恢复演练

设计：旧 checkpoint 恢复演练

参考答案 / 评分要点

1. 准备旧状态：记录旧 thread_id 和 checkpoint 内容。2. 升级代码：部署新版本，保留旧版本备份。3. 恢复执行：用旧 thread_id 在新代码下恢复。4. 比较 trace：对比升级前后同一任务的 trace，检查 span 差异和错误。5. 状态迁移：如果旧 checkpoint 字段不兼容，写迁移脚本或回滚。6. 验证输出：确认恢复后的任务输出和旧版本一致。

至少包含：准备旧状态、升级代码、恢复执行、比较 trace、判断是否需要状态迁移。
