课程总论

重点不是让模型“更聪明”，而是让模型行为进入可管理的工程外壳。

Overview
核心问题

一次模型调用可以靠提示词推进，长任务则需要运行循环、工具分发、状态保存、权限边界和故障恢复。

学完能做

能画出一个真实智能体系统的运行承载层，并说明每个边界如何降低失控、丢状态和难排查的风险。

课程位置

位于基础工程之后，连接工具契约、可靠性工程、上下文工程和多角色协作。

基于哪些资料

课程结构来自公开文档的交叉整理，未把任何单一框架当作唯一答案。

Sources
OpenAI 智能体 SDK

官方文档用于理解运行循环、工具、会话、沙箱工作空间和结果状态。

SDK 更新记录

发布与变更说明用于建立迁移、版本锁定和行为变化的判断标准。

LangGraph Durable Execution

持久执行文档用于理解检查点、恢复、确定性和副作用隔离。

Temporal

平台文档用于补充长运行流程、崩溃恢复和工作流/活动拆分的通用工程语义。

MCP 规范

规范入口用于理解外部工具、资源、传输、授权和协议边界。

追踪与语义约定

OpenAI 追踪文档与 OpenTelemetry GenAI 语义约定用于设计运行记录、指标和排查入口。

建议学习对象

适合已经做过模型调用，却开始遇到长任务工程问题的人。

Audience
应用开发者

已经能接模型和工具，希望把零散调用整理成可维护的服务。

平台工程师

需要设计任务运行、权限、沙箱、记录、恢复和升级策略。

技术负责人

需要判断一个智能体方案是演示、原型还是可上线系统。

前置知识

不要求已经掌握某个框架，但需要知道模型应用的基本零件。

Before
模型调用

知道消息、系统指令、上下文窗口、响应和流式输出是什么。

工具调用

理解函数名、参数、返回值、错误和权限边界，而不只是把工具当成提示词补充。

工程基础

熟悉 API、文件系统、日志、重试、环境变量和最基本的服务部署概念。

推荐路线

先认清运行承载层的职责，再逐步进入工具、状态、外部连接和运维。

Roadmap
阶段一｜定义边界

先区分模型、运行循环、工具执行器、工作空间和人工控制点，避免把所有问题都塞回提示词。

对应 01-02
阶段二｜承载行动

再看工具面、文件工作区、外部协议和数据连接，理解行动能力从哪里来、如何被限制。

对应 03、05
阶段三｜承载时间

长任务的关键是状态、恢复和副作用控制；这一步把“跑完一次”升级为“中断后还能继续”。

对应 04
阶段四｜承载运营

最后建立追踪、指标、回放、版本迁移和验收标准，让系统可以被排查和迭代。

对应 06-07

单元设计

每个单元先说明为什么存在；具体单元内容后续再制作。

7 Units
01

运行承载为什么重要

存在理由：很多失败不是模型能力问题，而是外层没有定义谁负责状态、工具、权限、失败和记录。本单元先建立“运行承载层”的直觉。

打开学习页
02

运行循环与控制边界

存在理由：连续任务需要明确输入、模型调用、工具分发、结果回填、终止条件和人工介入点，否则循环会失控或过早结束。

打开学习页
03

工具面与工作空间

存在理由：智能体一旦能读写文件或执行代码，就必须把工具类型、工作目录、权限和输出物边界讲清楚。

打开学习页
04

状态、会话与可恢复执行

存在理由：长任务会跨回合、跨进程、跨时间运行。本单元讨论会话、检查点、恢复和副作用隔离，防止中断后从头重做或重复写入。

打开学习页
05

外部协议与连接器

存在理由：当工具不再只是本地函数，协议、传输、授权、资源暴露和安全提示会成为系统边界的一部分。

打开学习页
06

追踪与运行记录

存在理由：没有完整运行记录，就很难判断错误来自模型、工具、上下文、权限还是网络。本单元建立 trace、span、事件和指标的读法。

打开学习页
07

版本更新与迁移策略

存在理由：智能体运行框架变化很快。生产系统需要知道哪些更新会改变行为，哪些可以直接吸收，哪些必须回归测试。

打开学习页

资料边界

这页保留课程地图，并链接到已完成的单元讲义。

Scope

本课程不覆盖模型训练、提示词大全、前端产品设计、企业采购治理，也不比较所有智能体框架优劣。

单元顺序是基于公开资料的工程归纳：先运行边界，再工具和状态，再外部连接，最后可观测与迁移。后续制作单元时应继续回到这些资料核对细节。
