故障分流

可靠性问题地图

这一页先做可靠性分流：看到一个 Agent 失败时，不急着改提示词，先判断是状态丢了、工具失败了、副作用重复了、输出漂了，还是证据链不够。你会沿着同一个锚点例子把问题分类、找证据、决定下一步该去 02 还是 04、05、06。

读完本页后能做什么

  • 能把客服退款、搜索卡住、最终答案错误这三类症状分到不同的可靠性问题上。
  • 能说清一条失败证据链里该看什么：运行记录、工具参数、重试次数、状态变更、输出边界。
  • 能判断下一步该进入哪一单元：02 看恢复，03 看幂等，04 看重试，05 看结构化边界，06 看追踪。

学习目标

goal

本单元先练分流：同样是“Agent 失败”，有些问题要看状态恢复，有些要看工具调用，有些要看结构化边界和观测记录。先把问题分到对的桶里，后面的单元才有意义。

先判断症状

从“退款被重复执行”“搜索卡 40 分钟”“只有最终回复”这类表面现象，判断它更像哪一类工程问题。

再看证据

把 trace、工具返回、重试次数、状态写入点和输出格式放进同一条排查链里，先看过程，再看最终回答。

最后选下一步

知道哪类问题交给 02 / 03 / 04 / 05 / 06 解决，哪类问题只是暂时信息不足，需要先补运行记录。

先抓主线

triage

把可靠性当作一次分诊。用户看到的是“结果错了”或“流程停了”，你要先在恢复、幂等、重试、结构边界、观测语义五个方向里定位。它们常常一起暴露，但处理入口不同。

表面症状优先怀疑下一步看什么下一单元
重启后忘记执行过退款状态恢复 / 副作用边界流程历史、检查点、幂等键02 / 03
搜索工具卡住很久工具失败 / 超时 / 重试timeout、retry、error type、耗时04
模型输出格式总是漂结构化边界schema、字段、下游解析条件05
只能看到最后一句回复追踪不足trace、span、事件语义、敏感数据策略06

这张表的作用很实在：先决定你应该去补哪一层。不要把所有失败都硬塞成“模型不够聪明”；很多时候，问题在运行层、工具层或观测层。

核心机制

trace

这个单元的核心机制，是把一次运行拆成“症状、证据、归因、下一步”四层。你可以先拿一个最小案例走一遍：客服退款 Agent 在部署重启后重复读对话，于是又准备再退一次。这个症状本身不能告诉你根因；你要看它是否有流程历史、退款工具是否幂等、是否有可重放的检查点、是否有 trace 记录工具参数。

对象重启前重启后读什么写什么错了会怎样
流程状态停在“退款已发起”附近只剩部分对话记忆历史步骤、检查点继续执行位置把已完成步骤当成未执行
退款工具第一次调用成功再次收到相同意图幂等键、订单号、运行标识原结果或重复执行重复退款
观测记录有可能只留最终回答排查者看不到中间过程trace、span、error.type故障证据链只能猜，不能定位

这张矩阵的作用很直接：逼你问对问题。这个步骤到底读了什么、写了什么、写错后会出现什么症状。只要这三问没答清，很多“修复”其实都只是把症状盖住。

退款重启分流的最小判断
1. 先看是否有流程历史 / checkpoint
2. 再看退款工具是否使用幂等键
3. 再看 trace 是否记录了工具调用和错误类型
4. 如果三项都没有，先补运行层证据，再谈模型提示词

例子

case

案例 A：客服退款 Agent

用户已经点了退款，工具也成功返回过一次。后来系统重启，Agent 再次读到“用户想退款”，又准备调用退款接口。

这类问题优先看状态恢复和幂等。你要确认是否有可恢复的流程步骤，以及退款工具是否接受订单号、意图和运行标识组成的幂等键。

如果这两层都没有，重试只会把重复动作扩大。

案例 B：研究 Agent 卡 40 分钟

Agent 调用搜索工具后长时间没结果，前端只显示“生成失败”，日志里没有耗时和重试次数。

这类问题先看工具失败、超时和重试策略。模型是否会写答案，放到后面再判断。你需要知道是哪次调用超时、是否重试、失败是网络问题还是限流问题。

如果连这些都看不到，先补观测和错误分类。

先看到什么只有用户意图和一段对话文本

这一步只能说明“问题出现了”，不能说明问题在哪里。

再看什么流程是否记录了可恢复位置

有 checkpoint 才能把“重启后怎么办”变成可回答的问题。

继续看什么退款键是否能识别同一意图

没有幂等键，恢复和重试都可能重复副作用。

最后看什么trace 是否带工具参数、耗时和错误类型

看不见过程，就很难把一次失败和下一步动作连起来。

症状：退款工具成功过一次，但重启后又准备再退。
先看：02 的 checkpoint，03 的幂等键，06 的 trace。

常见误区

debug

误区 1：把所有失败都归因于模型

模型当然会错，但重复副作用、状态丢失、超时和证据链缺失通常是系统层问题。先看运行层，再看提示词。

误区 2：有重试就算修好了

重试只对一部分失败有用。遇到非幂等副作用，重试会把事故放大；遇到永久错误，重试只是拖时间。

误区 3：只保存最终回答就够了

最终回答只能算结果，不算证据链。没有 trace、span、工具参数和错误类型，你只能知道结果错了，不能知道哪一步错了。

误区 4：字段名各写各的也没关系

如果一个团队写 tool，另一个写 function_call，再有团队只写自然语言摘要，跨系统就没法汇总，也没法比较。

误区真正缺的东西先补什么
“模型不行”运行层分流看恢复、幂等、工具、trace
“多重试几次”错误分类区分可重试和不可重试
“看最后结果就够”运行证据补 span、参数、耗时、错误类型
“字段随便叫”统一语义对齐 GenAI 事件和指标术语

先别追的东西

boundary

这一页先不追完整的分布式工作流框架、红队评测、成本核算和所有编排平台实现。它们当然重要，但不该抢走你对“症状怎么分、证据怎么看、下一步该去哪里”的注意力。

02 的位置

当你已经确认问题是“中断后继续不了”时，再去看持久执行和检查点。

03 / 04 的位置

当你确认问题和重复副作用、超时、重试策略有关时，再进入幂等和工具失败单元。

05 / 06 的位置

当问题落在格式漂移、字段语义、观测缺失时，再看结构化输出和追踪语义。

怎么接到后续单元

next step

去 02 之前

先确认当前问题是否属于“流程中间断了、恢复不了”。如果是，带着具体的断点和状态读写位置去看检查点，先把恢复位置找出来。

去 03 / 04 之前

先找是否存在外部副作用、重复调用、超时、重试和错误分类。这里的关键词会决定你下一页该看幂等还是重试策略。

分流口诀
状态断了 -> 02
副作用会重复 -> 03
工具经常失败或卡住 -> 04
输出格式/字段会漂 -> 05
看不到过程 -> 06

先把术语放正

terms

这一页里，术语只负责帮你分清问题，不负责装饰表达。你只要先把四个词放对位置：checkpoint 说的是“从哪里继续”，幂等说的是“同一意图重复来一次会怎样”，trace 说的是“过程能不能被看见”，schema 说的是“下游按什么结构接收”。它们经常一起出现，但各自要解决的失败点不同。

术语直觉常见误解在本页里怎么用
checkpoint给长任务留一个继续点以为只要保存对话就够了判断 02 是否该先上场
幂等同一件事再来一遍，结果不应变坏以为“重试一次”总能修好判断 03 是否要先看
trace / span把过程拆成可见步骤以为最终回答就是完整日志判断 06 是否需要补证据链
schema告诉下游字段怎么排、怎么读以为格式只影响展示，不影响流程判断 05 是否要先收紧边界
error.type把失败先分成能处理的类别以为所有失败都只要再试一次判断 04 是否先做分类

你可以把这些词当成五个检查灯。灯亮了，说明对应一层已经出现问题；灯不亮，不代表没有问题，只代表你还没找到入口。先把词放正，后面再讲实现，会省很多绕路。

术语对照卡
checkpoint = 继续点
幂等 = 重复一次不应造成额外副作用
trace/span = 过程记录
schema = 结构边界
error.type = 失败分类

状态台账

ledger

当你分不清是恢复问题还是幂等问题时，先别争论概念，直接列台账：谁读了什么，谁写了什么，写完后谁会再读，什么状态会很快过期。客服退款最容易暴露这类问题，因为它同时有对话记忆、流程位置、外部副作用和排查证据。只要这四层里有一层没被记录清楚，你就很容易在“恢复问题”和“幂等问题”之间来回摇摆。

步骤读什么写什么何时会过时错了的症状下一步检查
用户意图解析对话、订单信息、当前目标“要退款”的运行标记当新的对话上下文覆盖它把旧意图当成新任务确认输入窗口和上下文来源
流程 checkpoint当前步骤、已完成步骤继续执行位置进程退出、部署重启、任务迁移后重启后从头来过去看持久执行和恢复语义
退款工具调用幂等键、订单号、请求摘要外部退款结果重复接到同一意图时再次扣款或再次退款确认幂等键是否稳定
trace / span工具名、耗时、错误类型运行证据采集缺失、脱敏过度、字段不统一时只能看最终回答去补追踪和事件语义
通知用户退款是否真的成功确认消息工具返回和通知之间发生中断时用户收到不一致通知检查状态写入顺序

读这张表时，先看每一行的“写”。如果你连状态是谁写的都说不出来，这一层就还只是口头概念。反过来，只要你能指出写入点，就能继续追问：下一次运行会从哪读、读到的值什么时候过期、写错时会出现什么症状。

退款台账的最小问法
1. 这一步写了什么状态？
2. 下一个步骤会从哪里读这个状态？
3. 状态一旦过期，症状会变成什么？
4. 这行状态有没有幂等键或 checkpoint？
5. 记录不到这一步时，先去补哪类运行记录？

这张台账的价值在于让猜测变成可验证。只要你能够把一项状态写清楚，它就能把修复方向带到正确的层：能恢复就去 02，能重放但不能重复副作用就去 03，连过程都看不见就去 06。

失败时间线

timeline

同一个退款故障，按时间看会更清楚。09:00 用户发起退款；09:01 Agent 调用退款工具并成功；09:05 部署重启，内存态丢失；09:06 Agent 再读对话，把“用户想退款”当成新的任务。这个过程最危险的地方，不在于每一步看起来都合理，而在于每一步都在各自的局部视角里合理。

时间看到的现象系统内部发生了什么如果缺这一层会怎样
09:00用户发出退款请求系统读到订单和意图如果没有意图标记，后面无法判断同一任务是否已开始
09:01退款工具返回成功外部副作用已经发生如果不记录幂等键，下一次调用看不出自己已经成功过
09:05服务重启内存态和临时上下文消失如果没有 checkpoint，流程会失去继续的位置
09:06Agent 再次读到退款意图把老意图当成新任务如果看不到 trace，团队会误把问题归给模型生成

把时间线拉出来以后，你会发现重复退款先表现为一个顺序错误：先有副作用，再有状态丢失，再有再次执行。这个顺序一旦说不清，很多修复就会错位：有人去改提示词，有人去加更大的上下文，有人去重试，结果都没有触到真正的断点。

时间线日志片段
09:00 user_intent = refund_order
09:01 tool_result = refund_success
09:05 deploy = restart
09:06 memory_state = partial
09:06 retry_action = refund_again

这段日志真正要你看的是读写顺序。第一次成功之后，系统还需要知道成功发生过；如果这个信息没有被写到流程历史、幂等键或 trace 里，重启后的第二次读取就会把同一件事看成新任务。

误判实验室

lab

下面这组误判非常常见：它们看起来都像“模型没做好”，但真正缺的层完全不同。把它们放在一起看，能帮你把“先改提示词”的冲动压住，改成“先找哪一层写错了、哪一层没记录、哪一层会重复”。

误判为什么容易看错真正缺什么继续错修会怎样先做的最小动作
把重启后重复退款当成提示词问题表面上看，Agent 又读了一遍用户意图，像是理解失败checkpoint、状态写入、幂等键你会把 prompt 写得更长，但重复副作用还是会发生先找流程历史和退款工具的幂等规则
把工具超时当成模型不会用工具用户看到的是“生成失败”，很容易把责任归给模型本身timeout、retry、错误分类、耗时记录你会反复换模型，却没碰到真正慢的那一层先看是哪一次调用慢、慢在哪里、是否被重试
把只看到最终回答当成“已经有日志了”最终回答确实是一段文本，所以很像证据trace、span、事件语义、脱敏策略团队会围着结果猜来猜去，排查时间越来越长先补过程记录，再谈结果解释
把字段漂移当成前端显示问题有些时候只是展示错了，看起来像 UI 毛病schema、字段定义、下游解析条件你会改界面，但真正的输入输出边界仍然松动先确认字段定义是否稳定，解析方是否按同一结构读
把重试当成万能兜底重试确实能救一部分临时失败，所以很容易被放大成默认策略错误类型、幂等语义、失败分层可重试错误和不可重试错误会被混在一起，事故可能被放大先区分可重试和不可重试，再决定是否重试

这张表要你做的事情很简单：看见一个故障时，先找最不该先动的东西。退款重复时，先别动提示词；工具超时先别换模型；字段漂移先别改页面。先把这一层稳住，后面的修复才不会把问题推到别的地方。

误判：把重复退款当成提示词问题。
先做：找 checkpoint、幂等键、状态写入点，再考虑提示词。
误判纠偏三问
1. 这类故障最容易被误判成哪一层？
2. 真正缺的是状态、幂等、追踪、结构边界，还是错误分类？
3. 如果只动表层修复，事故会不会转移到别的地方？

把这三问记住以后，你会更容易在同一批症状里分出不同层：恢复类问题先看 02，幂等类问题先看 03，工具失败类问题先看 04，结构漂移先看 05，过程看不见先看 06。这个顺序一旦稳定下来，排查会比“先试一个大模型”快得多。

处置演练

plan

如果你现在就是值班的人，面对“退款重启后又要再退一次”这种告警，先把动作顺序钉住，再考虑模型。一个能用的顺序通常是：先确认外部动作有没有发生，再确认流程是否还能续跑，再确认同一意图是否会重复触发副作用，最后再看 trace 里能不能把这条链条说清楚。顺序错了，修复就会错位；顺序对了，很多争论会自动结束。

你现在看到什么先做什么不要先做什么为什么
退款已经成功过一次，但又要执行第二次冻结新的外部动作，确认上一次是否已经生效不要先把 prompt 改长重复副作用比文本不优雅更危险
重启后流程找不到位置检查 checkpoint 和流程历史不要先强行重跑整条链丢掉的位置如果不先找回，重跑只会制造新状态
看不到工具调用与错误类型补 trace、span、事件语义不要先靠人工猜原因没有证据链，团队只会争论“像不像模型问题”
字段和解析规则已经漂了检查 schema 和下游解析条件不要先改展示文案界面上看起来对，不代表下游真的能读对

这个顺序背后有一条简单原则：先止损，再找位，再补证据，最后决定是否调模型。止损是为了不让副作用继续扩大；找位是为了知道任务卡在什么地方；补证据是为了把结论从猜测变成记录；调模型放在最后，因为模型通常不是唯一层，也不是最容易被误解的那层。

值班顺序单
1. 先止损：冻结可能重复的外部动作
2. 再找位：确认 checkpoint、已完成步骤、当前步骤
3. 再补证据：trace、span、error.type、耗时、幂等键
4. 最后调策略：决定是去 02、03、04、05 还是 06

失败后果：
- 少了第 1 步，副作用会继续扩大
- 少了第 2 步，流程会从错误位置重来
- 少了第 3 步，团队会继续猜
- 少了第 4 步，后续修复会送错门

把这份顺序单写成自己的版本很重要。你不需要照抄这几行，但你要保留同样的结构：止损、找位、补证据、再决定去向。只要结构没变，遇到“退款重复”“搜索卡住”“字段漂移”这三类故障，你就能很快把动作缩到最小，不会一上来就动到最容易误伤的层。

这也是本页最后的练习钩子：如果你要把这门课交给同事，先让他复述一次这个顺序，再让他拿自己的系统举一个故障例子。只要他能按“症状 -> 台账 -> 时间线 -> 顺序单 -> 下一页”讲通，说明他已经能开始分流了。

追踪回放

trace

最后再把同一个退款故障按 trace 回放一遍。这里的关键是看每一步是否能让你看见读了什么、写了什么、用了什么错误分类。trace 只剩下“任务失败”四个字时，对分流几乎没有帮助；trace 记录了调用、结果和错误类型时，你就能直接把问题送到正确的后续单元。

span / 事件记录了什么对分流有什么用缺失时会怎样
agent.start输入摘要、订单号、当前目标帮你确认问题到底是不是同一单任务你会把老任务和新任务混在一起
tool.call.refund工具名、参数摘要、幂等键帮你判断是否会重复副作用你不知道这次调用是不是第二次
tool.result.refund成功、失败、error.type、耗时帮你判断该去 03 还是 04你只能看到“失败了”，却不知道怎么失败
agent.finish最终回答、用户确认状态帮你对上“流程是否真的结束”结果看着完整，过程却可能断在中间

你会发现，真正有用的 trace 不是把所有文本都堆进去，而是把过程拆成几次有意义的写入。每个写入点都要回答三个问题：这一步有没有改变状态、有没有改变外部世界、有没有留下可回放的证据。只要这三个问题没答清，回放就还不能算回放。

trace 最小清单
1. 输入摘要和任务标识有没有记录
2. 工具调用有没有带幂等键
3. 工具结果有没有带 error.type 和耗时
4. 最终回答有没有和前面的步骤对得上
5. 如果没有这些字段，先去补哪一层

这就是本页最后一次把问题收束到下一步：有 checkpoint 的去 02，有幂等问题的去 03，有超时和失败分类的去 04，有格式和字段问题的去 05，没有过程证据的去 06。你现在已经能把一个故障拆到这一步了，接下来的课才会真正有用。

分流决策

route

当症状、台账和时间线都摆出来以后，最后一步就是把问题送去正确的下一页。这里不需要一套抽象框架，只需要一个稳定的判断表：什么信号说明该去 02，什么信号说明该去 03，什么信号说明该去 04、05 或 06。

看见什么先查什么下一页为什么先去这里
中断后恢复不了checkpoint、流程历史、继续执行位置02断点和恢复语义先决定任务能不能续跑
外部副作用会重复幂等键、请求摘要、写入顺序03先保证同一意图不会造成两次外部动作
工具慢或偶发失败timeout、retry、error type、耗时04先把失败类型分清，再决定是否重试
输出字段会漂schema、字段定义、解析条件05先把结构边界钉住，后面的步骤才有固定输入
过程看不见trace、span、事件语义、脱敏策略06先补证据链，排查才有落点

如果你愿意把每次排查都按这个表走，很多看起来像模型问题的故障会很快拆开。比如“只见最终回答”会直接指向 06；“重启后重复副作用”会同时指向 02 和 03；“工具慢且偶发失败”会把你推去 04。你不需要在第一页就解决一切，你只需要别把问题送错门。

路由器
看见什么 -> 先查什么 -> 去哪一页
断点恢复不了 -> checkpoint -> 02
外部副作用会重复 -> 幂等键 -> 03
工具超时或限流 -> timeout/retry/error type -> 04
输出字段漂移 -> schema/parse -> 05
过程看不见 -> trace/span -> 06

把这张表当成你的一次排查门牌。它不是结论，但它能让你少走一轮错路。

证据优先级

order

同一个故障常常会同时露出多个信号：对话、工具调用、checkpoint、trace 和错误分类。先后顺序很重要。先看能防止副作用继续扩大的信息，再看能恢复任务位置的信息，最后看能解释原因的记录。这样做不是为了把每条记录都看完，而是先把排查范围缩到最小。

先看什么它回答的问题什么时候优先看完之后去哪里
外部动作是否已经发生先止损还是先续跑退款、发信、删除、写库这类副作用存在时先去 03，再回 02
checkpoint / 事件历史任务从哪里继续流程中断、重启、迁移后去 02
trace / span / error.type哪一步失败、为什么失败过程不清、排查卡住、需要复盘时去 06 或 04
schema / 字段定义下游为什么读不动结果能产出但结构不稳定时去 05

这张表有一个实际用法：同样是“退款重启后又要再退一次”，如果先看最终回答，你会把注意力放错；如果先看外部动作和幂等键，你能立即判断是否需要止损；如果动作已经成功，再去找 checkpoint，就能知道是恢复问题还是副作用问题。

证据优先级清单
1. 外部动作有没有已经发生？
2. 流程能不能从 checkpoint 继续？
3. trace 里有没有工具调用、参数和 error.type？
4. schema 和字段定义会不会让下游读不动？
5. 这条线应该送去 02 / 03 / 04 / 05 / 06 的哪一页？

首轮排查卡

first pass

碰到新告警时，先做一轮短排查，不要立刻改策略。第一轮的目标很小：确认有没有副作用外溢，确认任务能不能续跑，确认你手上有没有足够证据把它送进正确单元。只要这三件事没完成，就先别把问题归给模型或提示词。

第 1 步你要拿到什么说明什么下一步
确认外部动作退款、发信、写库、删除是否已经发生先决定要不要止损必要时去 03
确认恢复位置checkpoint、事件历史、当前步骤看系统能不能续跑去 02
确认过程证据trace、span、error.type、耗时看你能不能定位失败点去 04 或 06
确认结构边界schema、字段定义、解析条件看下游是否读得动去 05

这张卡的意思很实际：先抓最容易扩大事故的部分，再抓最容易丢位置的部分，最后抓最容易解释不清的部分。你每次只要照着这四行问一遍，就不会在第一轮排查里把所有层混成一锅。

首轮排查三问
1. 这次有没有已经发生的外部动作？
2. 如果进程现在停掉，它还能从哪一步继续？
3. 我手里的证据够不够把它送到 02 / 03 / 04 / 05 / 06？

练习

diagnostic

怎么答才算会

目标是把一个真实症状分到对的可靠性问题上，并说出你下一步要看哪类证据。答题时先说判断，再说理由，最后说下一步。

检查重点

重点看三件事：你有没有把状态、工具、副作用、结构边界和追踪语义分开；你有没有把“看什么”说具体；你有没有把下一单元选对。

一、症状分流

先判断问题属于哪一类，不要直接跳到解决方案。

答题要点：先分流，再看证据，再选下一步。看不懂时先回到状态台账。
choicetriage

退款重启后重复执行

客服 Agent 已经成功调用过退款工具，但重启后又准备再退一次。最优先怀疑哪一类问题？

choicefailure

卡住 40 分钟

研究 Agent 调用搜索工具后卡了很久，最后只说“生成失败”。最先该看什么？

choicetrace

最终回答看不出过程

上线后只有最后一句回答，没人知道中间有没有调用工具。最先怀疑哪一层？

二、证据链

把症状变成可排查的记录，别停在模糊判断。

答题要点：列出你要看什么，别只写“去排查”。
debug checklistevidence

分流检查清单

下面哪些信息对判断“是恢复问题还是工具问题”最有用？选出你认为应该先看的一组，并说明为什么最终回答和语气不算核心证据。

展开反馈

正确方向是：流程历史 / checkpoint、工具参数 / 幂等键、trace / span / 错误类型 / 耗时。最终回复的语气不能直接帮助分流。

filltrace

写出最小证据链

用 3 到 5 个字段写出一次失败时你最想保留的证据。要求包含一个能定位过程、一个能定位工具、一个能定位错误的字段。写完后再补一句：如果这些字段都没有，你下一步先去哪里找。

答案目标：你要能说清“过程在哪里、工具做了什么、为什么失败”。

参考答案 / 评分要点

要点至少包括：一个运行标识（trace_id / run_id）、工具信息（工具名 / 参数摘要）、错误或耗时信息（error.type / duration / retry_count）。如果只写最终回答，不算完整。

三、下一步去哪里

判断完问题类型后，要知道该把它送到哪一单元。

答题要点：问题类型和后续单元要对上，别把问题送错门。
choiceroute

状态断了去哪里

如果你已经确认问题是“中断后恢复不了”，下一步最该看哪一单元？

choiceroute

重复副作用去哪里

如果问题是“同一个意图会不会再次扣款、再次退款、再次写文件”，下一步该去哪里？

synthesistransfer

给出分流理由

任选一个真实或假想的 Agent 失败，按“症状 -> 怀疑点 -> 证据 -> 下一单元”写出 4 句判断。不要写长篇分析，先把链条接完整。最好直接套用退款、卡住、字段漂移这三类例子之一。

答案目标：让别人看完就知道你为什么把它分到那一类，结论和理由都要在。

参考答案 / 评分要点

合格答案要有四段：症状、怀疑点、证据、下一步。判断是否合理，关键看问题类型是否和后续单元对得上，措辞是否漂亮放在后面。

四、回看误区

把容易误判的地方再压一遍，防止你回去还是先改提示词。

答题要点：能指出错在哪里才算会，能解释为什么错更重要。
true/falsemisconception

判断对错 1

“只要把提示词写长一点，重启后重复退款这类问题就会自己消失。”

true/falseobservability

判断对错 2

“最终回答保存下来了，就等于有了完整证据链。”

orderingdebug order

排序：先看什么

把下面四步排成合理顺序：A. 看 trace 和错误类型；B. 判断症状属于哪类问题；C. 决定去哪个后续单元；D. 找到流程历史或工具参数。

答案目标：从症状分流到证据，再到下一步。

参考答案 / 评分要点

合理顺序通常是：先 B 判断症状类别，再 D 找对应证据，接着 A 看 trace / 错误类型，最后 C 决定进入哪个单元。只要你的顺序能说明“先分流、再证据、后去向”，也可以接受。

orderingevidence order

证据优先级

把下面四类证据按更合理的排查顺序排一下：A. trace / span / error.type；B. 外部动作是否已经发生；C. checkpoint / 事件历史；D. schema / 字段定义。

答案目标：先止损，再找位，再补证据，最后看结构边界。

参考答案 / 评分要点

合理顺序通常是：B 先确认外部动作有没有发生，C 再确认能否从 checkpoint 继续，A 继续看 trace 和错误类型，D 最后检查 schema 是否让下游读不动。只要你的顺序能说明“先止损，再找位，再补证据，最后看边界”，就算合格。
