可靠性问题地图

这一页先做可靠性分流：看到一个 Agent 失败时，不急着改提示词，先判断是状态丢了、工具失败了、副作用重复了、输出漂了，还是证据链不够。你会沿着同一个锚点例子把问题分类、找证据、决定下一步该去 02 还是 04、05、06。

学习目标

goal

本单元先练分流：同样是“Agent 失败”，有些问题要看状态恢复，有些要看工具调用，有些要看结构化边界和观测记录。先把问题分到对的桶里，后面的单元才有意义。

再看证据

把 trace、工具返回、重试次数、状态写入点和输出格式放进同一条排查链里，先看过程，再看最终回答。

先抓主线

triage

把可靠性当作一次分诊。用户看到的是“结果错了”或“流程停了”，你要先在恢复、幂等、重试、结构边界、观测语义五个方向里定位。它们常常一起暴露，但处理入口不同。

搜索工具卡住很久工具失败 / 超时 / 重试timeout、retry、error type、耗时04

模型输出格式总是漂结构化边界schema、字段、下游解析条件05

只能看到最后一句回复追踪不足trace、span、事件语义、敏感数据策略06

核心机制

trace

这个单元的核心机制，是把一次运行拆成“症状、证据、归因、下一步”四层。你可以先拿一个最小案例走一遍：客服退款 Agent 在部署重启后重复读对话，于是又准备再退一次。这个症状本身不能告诉你根因；你要看它是否有流程历史、退款工具是否幂等、是否有可重放的检查点、是否有 trace 记录工具参数。

流程状态停在“退款已发起”附近只剩部分对话记忆历史步骤、检查点继续执行位置把已完成步骤当成未执行

退款工具第一次调用成功再次收到相同意图幂等键、订单号、运行标识原结果或重复执行重复退款

观测记录有可能只留最终回答排查者看不到中间过程trace、span、error.type故障证据链只能猜，不能定位

退款重启分流的最小判断复制

例子

case

案例 A：客服退款 Agent

用户已经点了退款，工具也成功返回过一次。后来系统重启，Agent 再次读到“用户想退款”，又准备调用退款接口。

这类问题优先看状态恢复和幂等。你要确认是否有可恢复的流程步骤，以及退款工具是否接受订单号、意图和运行标识组成的幂等键。

如果这两层都没有，重试只会把重复动作扩大。

常见误区

debug

误区 1：把所有失败都归因于模型

模型当然会错，但重复副作用、状态丢失、超时和证据链缺失通常是系统层问题。先看运行层，再看提示词。

误区 3：只保存最终回答就够了

最终回答只能算结果，不算证据链。没有 trace、span、工具参数和错误类型，你只能知道结果错了，不能知道哪一步错了。

误区真正缺的东西先补什么

先别追的东西

boundary

这一页先不追完整的分布式工作流框架、红队评测、成本核算和所有编排平台实现。它们当然重要，但不该抢走你对“症状怎么分、证据怎么看、下一步该去哪里”的注意力。

03 / 04 的位置

当你确认问题和重复副作用、超时、重试策略有关时，再进入幂等和工具失败单元。

怎么接到后续单元

next

去 02 之前

先确认当前问题是否属于“流程中间断了、恢复不了”。如果是，带着具体的断点和状态读写位置去看检查点，先把恢复位置找出来。

分流口诀复制

先把术语放正

terms

这一页里，术语只负责帮你分清问题，不负责装饰表达。你只要先把四个词放对位置：checkpoint 说的是“从哪里继续”，幂等说的是“同一意图重复来一次会怎样”，trace 说的是“过程能不能被看见”，schema 说的是“下游按什么结构接收”。它们经常一起出现，但各自要解决的失败点不同。

checkpoint给长任务留一个继续点以为只要保存对话就够了判断 02 是否该先上场

幂等同一件事再来一遍，结果不应变坏以为“重试一次”总能修好判断 03 是否要先看

trace / span把过程拆成可见步骤以为最终回答就是完整日志判断 06 是否需要补证据链

schema告诉下游字段怎么排、怎么读以为格式只影响展示，不影响流程判断 05 是否要先收紧边界

error.type把失败先分成能处理的类别以为所有失败都只要再试一次判断 04 是否先做分类

术语对照卡复制

状态台账

ledger

当你分不清是恢复问题还是幂等问题时，先别争论概念，直接列台账：谁读了什么，谁写了什么，写完后谁会再读，什么状态会很快过期。客服退款最容易暴露这类问题，因为它同时有对话记忆、流程位置、外部副作用和排查证据。只要这四层里有一层没被记录清楚，你就很容易在“恢复问题”和“幂等问题”之间来回摇摆。

用户意图解析对话、订单信息、当前目标“要退款”的运行标记当新的对话上下文覆盖它把旧意图当成新任务确认输入窗口和上下文来源

流程 checkpoint当前步骤、已完成步骤继续执行位置进程退出、部署重启、任务迁移后重启后从头来过去看持久执行和恢复语义

退款工具调用幂等键、订单号、请求摘要外部退款结果重复接到同一意图时再次扣款或再次退款确认幂等键是否稳定

trace / span工具名、耗时、错误类型运行证据采集缺失、脱敏过度、字段不统一时只能看最终回答去补追踪和事件语义

通知用户退款是否真的成功确认消息工具返回和通知之间发生中断时用户收到不一致通知检查状态写入顺序

退款台账的最小问法复制

这张台账的价值在于让猜测变成可验证。只要你能够把一项状态写清楚，它就能把修复方向带到正确的层：能恢复就去 02，能重放但不能重复副作用就去 03，连过程都看不见就去 06。

失败时间线

timeline

同一个退款故障，按时间看会更清楚。09:00 用户发起退款；09:01 Agent 调用退款工具并成功；09:05 部署重启，内存态丢失；09:06 Agent 再读对话，把“用户想退款”当成新的任务。这个过程最危险的地方，不在于每一步看起来都合理，而在于每一步都在各自的局部视角里合理。

09:00用户发出退款请求系统读到订单和意图如果没有意图标记，后面无法判断同一任务是否已开始

09:01退款工具返回成功外部副作用已经发生如果不记录幂等键，下一次调用看不出自己已经成功过

09:05服务重启内存态和临时上下文消失如果没有 checkpoint，流程会失去继续的位置

09:06Agent 再次读到退款意图把老意图当成新任务如果看不到 trace，团队会误把问题归给模型生成

时间线日志片段复制

这段日志真正要你看的是读写顺序。第一次成功之后，系统还需要知道成功发生过；如果这个信息没有被写到流程历史、幂等键或 trace 里，重启后的第二次读取就会把同一件事看成新任务。

误判实验室

lab

下面这组误判非常常见：它们看起来都像“模型没做好”，但真正缺的层完全不同。把它们放在一起看，能帮你把“先改提示词”的冲动压住，改成“先找哪一层写错了、哪一层没记录、哪一层会重复”。

把重启后重复退款当成提示词问题表面上看，Agent 又读了一遍用户意图，像是理解失败checkpoint、状态写入、幂等键你会把 prompt 写得更长，但重复副作用还是会发生先找流程历史和退款工具的幂等规则

把工具超时当成模型不会用工具用户看到的是“生成失败”，很容易把责任归给模型本身timeout、retry、错误分类、耗时记录你会反复换模型，却没碰到真正慢的那一层先看是哪一次调用慢、慢在哪里、是否被重试

把只看到最终回答当成“已经有日志了”最终回答确实是一段文本，所以很像证据trace、span、事件语义、脱敏策略团队会围着结果猜来猜去，排查时间越来越长先补过程记录，再谈结果解释

把字段漂移当成前端显示问题有些时候只是展示错了，看起来像 UI 毛病schema、字段定义、下游解析条件你会改界面，但真正的输入输出边界仍然松动先确认字段定义是否稳定，解析方是否按同一结构读

把重试当成万能兜底处理重试确实能救一部分临时失败，所以很容易被放大成默认策略错误类型、幂等语义、失败分层可重试错误和不可重试错误会被混在一起，事故可能被放大先区分可重试和不可重试，再决定是否重试

误判纠偏三问复制

把这三问记住以后，你会更容易在同一批症状里分出不同层：恢复类问题先看 02，幂等类问题先看 03，工具失败类问题先看 04，结构漂移先看 05，过程看不见先看 06。这个顺序一旦稳定下来，排查会比“先试一个大模型”快得多。

处置演练

plan

如果你现在就是值班的人，面对“退款重启后又要再退一次”这种告警，先把动作顺序钉住，再考虑模型。一个能用的顺序通常是：先确认外部动作有没有发生，再确认流程是否还能续跑，再确认同一意图是否会重复触发副作用，最后再看 trace 里能不能把这条链条说清楚。顺序错了，修复就会错位；顺序对了，很多争论会自动结束。

退款已经成功过一次，但又要执行第二次冻结新的外部动作，确认上一次是否已经生效不要先把 prompt 改长重复副作用比文本不优雅更危险

重启后流程找不到位置检查 checkpoint 和流程历史不要先强行重跑整条链丢掉的位置如果不先找回，重跑只会制造新状态

看不到工具调用与错误类型补 trace、span、事件语义不要先靠人工猜原因没有证据链，团队只会争论“像不像模型问题”

字段和解析规则已经漂了检查 schema 和下游解析条件不要先改展示文案界面上看起来对，不代表下游真的能读对

值班顺序单复制

把这份顺序单写成自己的版本很重要。你不需要照抄这几行，但你要保留同样的结构：止损、找位、补证据、再决定去向。只要结构没变，遇到“退款重复”“搜索卡住”“字段漂移”这三类故障，你就能很快把动作缩到最小，不会一上来就动到最容易误伤的层。

追踪回放

trace

最后再把同一个退款故障按 trace 回放一遍。这里的关键是看每一步是否能让你看见读了什么、写了什么、用了什么错误分类。trace 只剩下“任务失败”四个字时，对分流几乎没有帮助；trace 记录了调用、结果和错误类型时，你就能直接把问题送到正确的后续单元。

agent.start输入摘要、订单号、当前目标帮你确认问题到底是不是同一单任务你会把老任务和新任务混在一起

tool.call.refund工具名、参数摘要、幂等键帮你判断是否会重复副作用你不知道这次调用是不是第二次

tool.result.refund成功、失败、error.type、耗时帮你判断该去 03 还是 04你只能看到“失败了”，却不知道怎么失败

agent.finish最终回答、用户确认状态帮你对上“流程是否真的结束”结果看着完整，过程却可能断在中间

trace 最小清单复制

这就是本页最后一次把问题收束到下一步：有 checkpoint 的去 02，有幂等问题的去 03，有超时和失败分类的去 04，有格式和字段问题的去 05，没有过程证据的去 06。你现在已经能把一个故障拆到这一步了，接下来的课才会真正有用。

分流决策

route

当症状、台账和时间线都摆出来以后，最后一步就是把问题送去正确的下一页。这里不需要一套抽象框架，只需要一个稳定的判断表：什么信号说明该去 02，什么信号说明该去 03，什么信号说明该去 04、05 或 06。

中断后恢复不了checkpoint、流程历史、继续执行位置02断点和恢复语义先决定任务能不能续跑

外部副作用会重复幂等键、请求摘要、写入顺序03先保证同一意图不会造成两次外部动作

工具慢或偶发失败timeout、retry、error type、耗时04先把失败类型分清，再决定是否重试

输出字段会漂schema、字段定义、解析条件05先把结构边界钉住，后面的步骤才有固定输入

过程看不见trace、span、事件语义、脱敏策略06先补证据链，排查才有落点

路由器复制

把这张表当成你的一次排查门牌。它不是结论，但它能让你少走一轮错路。

证据优先级

order

同一个故障常常会同时露出多个信号：对话、工具调用、checkpoint、trace 和错误分类。先后顺序很重要。先看能防止副作用继续扩大的信息，再看能恢复任务位置的信息，最后看能解释原因的记录。这样做先把排查范围缩到最小。

外部动作是否已经发生先止损还是先续跑退款、发信、删除、写库这类副作用存在时先去 03，再回 02

checkpoint / 事件历史任务从哪里继续流程中断、重启、迁移后去 02

trace / span / error.type哪一步失败、为什么失败过程不清、排查卡住、需要复盘时去 06 或 04

schema / 字段定义下游为什么读不动结果能产出但结构不稳定时去 05

证据优先级清单复制

首轮排查卡

first

碰到新告警时，先做一轮短排查，不要立刻改策略。第一轮的目标很小：确认有没有副作用外溢，确认任务能不能续跑，确认你手上有没有足够证据把它送进正确单元。只要这三件事没完成，就先别把问题归给模型或提示词。

确认外部动作退款、发信、写库、删除是否已经发生先决定要不要止损必要时去 03

确认恢复位置checkpoint、事件历史、当前步骤看系统能不能续跑去 02

确认过程证据trace、span、error.type、耗时看你能不能定位失败点去 04 或 06

确认结构边界schema、字段定义、解析条件看下游是否读得动去 05
