持久执行与检查点

这一页先把“中断后继续”讲清楚：长任务为什么不能只靠内存、检查点到底保存什么、Workflow 和 Activity 为什么要分开、以及恢复时该用哪些字段把过程看回来。你会沿着“研究助理生成报告并等待确认后发送”这条锚点，把恢复点、事件历史、业务存储和 trace 证据串成一条能操作的链。

学习目标

foundation

本单元要把“持久执行”落到可判断的动作上：一个长任务断掉以后，系统到底该从哪里续、哪些动作可以复用、哪些状态必须被保存、哪些字段能把过程看回来。你先把这四件事分开，后面的幂等、重试和追踪才有位置。

再分编排和动作

把决定下一步做什么的 Workflow，和真正会碰外部世界的 Activity 分开。

先抓主线

roadmap

持久执行解决的是“进程没了、网络断了、用户晚点才确认，任务还能不能继续”。它不负责让外部系统永不失败，也不负责让模型输出完全稳定。它只做一件事：把已经确认过的执行事实留下来，让系统知道自己已经走到哪里，恢复时从正确位置接着走。

重启后不知道任务做到哪了checkpoint 不足事件历史、继续位置、已完成步骤02

核心机制

workflow

一个可恢复的长任务通常会被拆成两层：上层是 Workflow，负责记住流程骨架和下一步该做什么；下层是 Activity，负责真正碰外部世界。这个拆法的意义很直接：Workflow 要可重放，Activity 要可重试。把两者混在一起，恢复时就会把不该重复的动作重复掉。

Workflow 决策已有目标、计划、已完成步骤根据历史重新走一遍事件历史、signal、定时器下一步调度位置把已经走过的分支再走一遍

Activity 执行调用过 LLM、工具或数据库可以按历史结果复用幂等键、参数摘要、历史结果外部请求、状态变更重复扣款、重复发信、重复写入

检查点任务停在某个中间步骤恢复到可继续的位置步骤状态、摘要、引用继续点从头开始猜，或者跳错位置

最小边界示意复制

状态台账

当你分不清“恢复问题”还是“副作用问题”时，先不要争概念，直接列台账。看每一步谁在读、谁在写、写完以后谁会再读、状态会在什么时候过期。把这四个问题列清楚，很多看似模糊的故障会立刻露出位置。

最小检查点清单复制

历史收尾

continue-as-new

Event History 是恢复的基础，但历史越长，重放越慢。任务跑得久、活动很多、信号很多时，历史会越来越大。Temporal 里的 Continue-As-New 思路很简单：把当前最相关的状态传给一个新的 Workflow Execution，给它一份新的历史，让旧历史收尾，新历史继续往前跑。

任务很长，活动很多最新相关状态、继续位置、必要引用新的 Workflow Execution别把无关历史一并搬过去

历史开始膨胀能继续任务所需的最小状态同一个 Workflow Id，新的 Run Id新旧执行要能关联，但不要混成一条旧历史

代码版本切换太多当前版本能理解的状态摘要fresh Event History避免老代码和新代码在同一条历史里互相打架

最小恢复协议

read

知道 checkpoint 的定义还不够。真正能让长任务稳定跑下去的，是一套固定的读写顺序：先读历史，再决定下一步，接着跑一个 Activity，最后只写恢复必需的状态。顺序不能反过来。你如果先把“已完成”写进 checkpoint，再去跑外部动作，重启后就会以为这一步已经完成；你如果把完整 prompt、原文资料和所有工具返回都塞进历史，恢复虽然也许还能做，但历史会越来越大，重放会越来越慢。

确认窗口

success

最容易出事故的地方，往往是“动作已经成功，但系统还没来得及记住”。邮件发送、退款、提交 PR、删除文件，都有一个短窗口：外部世界已经变了，本地 checkpoint 还没写完，worker 就崩了。恢复后如果只看本地状态，就会把这个动作当成还没做，于是再做一遍。

这三行代码看的是顺序：外部动作先完成，结果再进入 checkpoint，最后才返回结束状态。只要这个顺序守住，恢复后就能判断“这一步已经成功”，不会把系统确认和外部成功混成一件事。

案例

worked

研究助理生成报告并等待确认后发送

一个 agent 接到任务：收集资料，生成中文报告，等待用户确认，再把报告发给项目组。任务可能持续十几分钟，也可能因为人工确认跨越一天。中间任何一步都可能因为 worker 重启、网络中断或部署发布而停住。

中断点推演

如果 T2 前 worker 崩溃：恢复后 replay 会重建 T0、T1，继续调度 T2。

如果 T3 后 worker 崩溃：恢复后不应重新生成草稿，而应读取已记录的 draft_id。

如果 T5 邮件已发但结果未确认：本页先把“成功但确认丢失”的窗口看见，完整修法放到 03 的幂等和副作用隔离。

操作桥

next

把这页放回真实系统时，先打开三处：Workflow history、业务存储、trace。先看历史里的 step 和 signal，确认恢复位置有没有被记住；再看业务存储里的 draft_id、approval_state、send_id，确认中间成果有没有稳定引用；最后看 trace 里的 tool.call.plan、tool.call.fetch、tool.call.send，确认失败发生在编排、工具还是发送窗口。

追踪字段

observability

持久执行解决的是“能不能继续”，追踪解决的是“为什么会这样继续”。两者都需要，但职责不同。历史负责恢复，trace 负责解释；如果把 trace 当成恢复状态，系统一旦要重放，就会把“观察记录”误当成“执行依据”。

workflow.start任务 id、输入摘要、目标帮你确认是否同一单任务你会把老任务和新任务混在一起

tool.call.plan工具名、参数摘要、计划版本帮你看计划是否重新生成了你不知道计划是不是第二次算出来的

tool.call.fetchsource_ids、查询条件、耗时帮你判断资料检索是否卡住你只能看到“失败了”，不知道慢在哪

tool.call.senddraft_id、幂等键、外部请求 id帮你判断有没有重复发送恢复后重发的风险看不见

workflow.waitapproval_state、等待位置帮你知道任务是在等人还是卡住了排查的人会误以为流程死了

常见错误

fixes

错误 1：把 LLM 调用写进可重放编排代码

LLM 输出会随模型版本、采样和服务状态变化。放在 Workflow 里会破坏 replay 的确定性。

修法：把 LLM 调用封装为 Activity，结果只留下引用和必要摘要。

错误 2：把 trace 当检查点

trace 很适合排查，但它不能驱动恢复状态机。trace 丢了不该影响业务继续，事件历史坏了才会影响恢复。

修法：恢复状态进事件历史或 durable store，trace 用相关 id 串起来。

错误 3：恢复后重复发送外部副作用

邮件、扣款、提交 PR、删除文件都要考虑“已经成功但确认丢失”的窗口。

修法：这一步先记录窗口，下一页去 03 处理幂等、唯一约束和外部状态查询。

错误 4：把完整 prompt 和工具结果都写进历史

这会放大成本、隐私和性能问题。历史只保留恢复必须的信息，大内容放受控存储。

修法：保存 task_id、plan_version、source_ids、draft_id、approval_state、idempotency_key。

错误 5：用系统时间、随机数、全局状态驱动 Workflow 分支

这些值在 replay 时可能不同，导致历史与代码生成的命令不一致。

修法：把不确定计算放进 Activity，或者使用 replay-safe 的历史事件结果。

错误 6：观测字段没有统一命名

一个服务叫 tool，另一个叫 function，另一个只写 message，后续就很难跨系统查询。

修法：统一用 gen_ai.operation.name、gen_ai.provider.name、gen_ai.request.model、gen_ai.tool.name、error.type。
