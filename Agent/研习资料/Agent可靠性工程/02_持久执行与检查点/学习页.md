# 学习目标 {#goals}
<!-- tag: foundation -->

本单元要把“持久执行”落到可判断的动作上：一个长任务断掉以后，系统到底该从哪里续、哪些动作可以复用、哪些状态必须被保存、哪些字段能把过程看回来。你先把这四件事分开，后面的幂等、重试和追踪才有位置。

先找恢复点
知道任务已经走到哪一步，哪些步骤完成过，哪些步骤还没有开始。


再分编排和动作
把决定下一步做什么的 Workflow，和真正会碰外部世界的 Activity 分开。


最后补证据
用 trace、span、错误类型、幂等键和运行标识，把恢复前后的过程串起来。

# 先抓主线 {#overview}
<!-- tag: roadmap -->

持久执行解决的是“进程没了、网络断了、用户晚点才确认，任务还能不能继续”。它不负责让外部系统永不失败，也不负责让模型输出完全稳定。它只做一件事：把已经确认过的执行事实留下来，让系统知道自己已经走到哪里，恢复时从正确位置接着走。

看到什么优先怀疑下一步看什么通常去哪里


重启后不知道任务做到哪了checkpoint 不足事件历史、继续位置、已完成步骤02
同一意图又准备执行一次外部动作副作用边界没隔开幂等键、业务唯一键、状态写入顺序03
任务卡住、超时或间歇失败工具失败 / 重试策略timeout、retry、error.type、耗时04
输出字段漂移，后续读不动结构化边界松了schema、字段定义、解析条件05
只能看到最后回答，看不见过程追踪缺失trace、span、事件语义、版本06

先把这张表记住。它是一张分流表：一旦看见症状，就先决定去补哪一层，不要直接把所有故障都归给模型。

# 核心机制 {#mechanism}
<!-- tag: workflow / activity -->

一个可恢复的长任务通常会被拆成两层：上层是 Workflow，负责记住流程骨架和下一步该做什么；下层是 Activity，负责真正碰外部世界。这个拆法的意义很直接：Workflow 要可重放，Activity 要可重试。把两者混在一起，恢复时就会把不该重复的动作重复掉。

对象重启前重启后读什么写什么错了会怎样


Workflow 决策已有目标、计划、已完成步骤根据历史重新走一遍事件历史、signal、定时器下一步调度位置把已经走过的分支再走一遍
Activity 执行调用过 LLM、工具或数据库可以按历史结果复用幂等键、参数摘要、历史结果外部请求、状态变更重复扣款、重复发信、重复写入
检查点任务停在某个中间步骤恢复到可继续的位置步骤状态、摘要、引用继续点从头开始猜，或者跳错位置

这张矩阵的重点是读写关系。Workflow 读历史，写下一步；Activity 读参数，写外部结果；checkpoint 读当前状态，写继续位置。只要这一层没说清，恢复时就容易把“已经做过但系统不记得”的动作再做一遍。

最小边界示意复制

```python
class ReportWorkflow:
 async def run(self, request):
 plan = await execute_activity(plan_with_llm, request)
 sources = await execute_activity(fetch_sources, plan)
 draft = await execute_activity(write_report, sources)

 approval = await wait_for_signal("approved")
 if approval.ok:
 await execute_activity(send_report_once, draft.id, idempotency_key=draft.id)
 return {"status": "done", "draft_id": draft.id}
```

这段伪代码只做一件事：把可重放的编排和真实副作用分开。LLM、检索、写报告、发邮件都不该直接放在可重放的编排里；等待人工确认应进入可恢复的等待状态，而不是挂住一个 HTTP 请求。

# 状态台账 {#ledger}
<!-- tag: ledger -->

当你分不清“恢复问题”还是“副作用问题”时，先不要争概念，直接列台账。看每一步谁在读、谁在写、写完以后谁会再读、状态会在什么时候过期。把这四个问题列清楚，很多看似模糊的故障会立刻露出位置。

步骤读什么写什么何时会过时错了的症状下一步检查


目标进入任务说明、订单号、用户意图任务标识、版本、初始状态当新一轮任务覆盖它把旧任务当成新任务确认输入窗口和任务 id
计划生成目标、约束、已有上下文plan_version、摘要、引用模型版本变化或重新规划时重启后重新生成一份完全不同的计划看事件历史是否保存了计划结果
资料检索计划、source_ids、检索条件source_ids、摘要、可引用结果数据源变化或重试后同一次任务看到不同资料确认 Activity 结果是否持久化
草稿写入检索结果、写入目标draft_id、摘要、存储引用正文被覆盖或文件移动后重启后找不到草稿检查 draft_id 是否进入历史
人工等待审批状态、用户信号approval_state、等待位置用户晚些时候才确认时流程卡住但不知道卡在哪里看 signal 和等待点是否被记录
确认发送draft_id、发送标识、幂等键发送结果、外部请求 id发送成功但确认丢失时恢复后又发一次先看 03，再看外部状态

读这张表时，先看“写什么”。如果你说不出写入点，这层还只是口头概念；如果你能说出写入点，下一步就能追问它会被谁再次读取，以及写错以后会出现什么症状。

最小检查点清单复制

```text
1. 这一步写了什么状态？
2. 下一个步骤会从哪里读这个状态？
3. 这个状态什么时候会过时？
4. 能不能靠 checkpoint 或事件历史把它恢复回来？
5. 读不到时，先去补哪类运行记录？
```

# 历史收尾 {#fresh-history}
<!-- tag: continue-as-new -->

Event History 是恢复的基础，但历史越长，重放越慢。任务跑得久、活动很多、信号很多时，历史会越来越大。Temporal 里的 Continue-As-New 思路很简单：把当前最相关的状态传给一个新的 Workflow Execution，给它一份新的历史，让旧历史收尾，新历史继续往前跑。

何时考虑保留什么新历史从哪里开始你要注意什么


任务很长，活动很多最新相关状态、继续位置、必要引用新的 Workflow Execution别把无关历史一并搬过去
历史开始膨胀能继续任务所需的最小状态同一个 Workflow Id，新的 Run Id新旧执行要能关联，但不要混成一条旧历史
代码版本切换太多当前版本能理解的状态摘要fresh Event History避免老代码和新代码在同一条历史里互相打架

这里最重要的一句是：checkpoint 只保住“下一次能继续”的最小状态。Continue-As-New 让你把这一点做成新的历史边界，避免把一条越来越长的历史硬拖到底。

# 最小恢复协议 {#recovery-protocol}
<!-- tag: read / run / write -->

知道 checkpoint 的定义还不够。真正能让长任务稳定跑下去的，是一套固定的读写顺序：先读历史，再决定下一步，接着跑一个 Activity，最后只写恢复必需的状态。顺序不能反过来。你如果先把“已完成”写进 checkpoint，再去跑外部动作，重启后就会以为这一步已经完成；你如果把完整 prompt、原文资料和所有工具返回都塞进历史，恢复虽然也许还能做，但历史会越来越大，重放会越来越慢。

对“研究助理生成报告并等待确认后发送”这个锚点来说，最小恢复状态很具体：任务 id、plan_version、source_ids、draft_id、approval_state、发送幂等键，以及能指向当前步骤的继续位置。真正的正文、长篇资料和外部系统返回不应该都待在历史里，它们应该放在业务存储或对象存储，只把引用留在历史里。这样做的好处是，重启后你只需要恢复“现在走到哪一步”和“接下来做什么”，不需要把整个世界重新喂给 worker 一遍。

步骤读什么写什么错位后症状先看哪里


读历史事件历史、signal、已有的继续位置当前要做的下一步把已经完成的步骤又执行一遍确认历史里是否有上一步结果
跑 Activity任务参数、稳定引用、必要摘要外部结果、返回 id、错误类型外部世界已经变了，但系统没记住看 Activity 是否有独立结果记录
写 checkpoint刚刚确认过的结果、继续位置最小恢复状态重启后从错误位置继续，或以为已经做完确认写入点是不是在动作之后
切下一轮当前状态、历史长度、是否继续等待新的历史边界或新的等待点历史越来越长，重放越来越慢考虑是否需要 Continue-As-New

最小恢复伪代码复制

```python
async def run_report_task(task_id):
 state = load_history(task_id)
 step = state.get("step", "plan")

 if step == "plan":
 plan = await execute_activity(plan_with_llm, task_id)
 save_checkpoint(task_id, step="fetch", plan_version=plan["version"])
 step = "fetch"

 if step == "fetch":
 sources = await execute_activity(fetch_sources, task_id)
 save_checkpoint(task_id, step="write", source_ids=sources["ids"])
 step = "write"

 if step == "write":
 draft = await execute_activity(write_report, task_id)
 save_checkpoint(task_id, step="wait", draft_id=draft["id"])
 step = "wait"

 if step == "wait":
 await wait_for_signal("approved")
 save_checkpoint(task_id, step="send", approval_state="approved")
 step = "send"

 if step == "send":
 await execute_activity(send_report_once, task_id, idempotency_key=task_id)
```

这段代码真正要你看的是写入时机。每次 Activity 结束之后，才写下一步 checkpoint；每个 checkpoint 只保留恢复所需的最小字段。这样做有两个直接后果：重启后知道应该从哪一步续，也不会把整份资料和所有工具返回都拖进历史里。

如果顺序写错，症状会很明确：写早了，会把还没真正发生的步骤标成已完成；写晚了，会让同一动作在恢复时再执行一次。你做 q7 时如果不知道该保留哪些字段，就回到这张表；你做 q10 时如果讲不清“为什么不能把完整 prompt 写进历史”，就把这段代码里的写入点按顺序复述一遍。

# 确认窗口 {#confirmation-window}
<!-- tag: success / ack -->

最容易出事故的地方，往往是“动作已经成功，但系统还没来得及记住”。邮件发送、退款、提交 PR、删除文件，都有一个短窗口：外部世界已经变了，本地 checkpoint 还没写完，worker 就崩了。恢复后如果只看本地状态，就会把这个动作当成还没做，于是再做一遍。

这个窗口要靠顺序来缩短：先让 Activity 完成并返回结果，再把结果写进 checkpoint 或历史，最后才继续下一步。不要把“成功标记”提前写进去，也不要把完整正文拖到最后才补。对于长任务来说，最稳的做法是把“动作成功”和“系统确认”之间的间隔压到最短，并且把这段间隔留下可查的记录。

场景外部世界本地记录典型症状先查什么


邮件已发出项目组已经收到邮件checkpoint 还没写到 send 完成恢复后又发一次发送结果和幂等键
草稿已保存正文已经写到对象存储draft_id 还没进入历史恢复后找不到草稿草稿写入点和历史写入点
审批已通过用户已经确认approval_state 还没写入恢复状态系统一直等人或重复问确认signal 到 checkpoint 的顺序

发送完成的最小顺序复制

```python
result = await execute_activity(send_report_once, draft.id, idempotency_key=draft.id)
save_checkpoint(task_id, step="done", send_id=result["request_id"])
return {"status": "done", "send_id": result["request_id"]}
```

这三行代码看的是顺序：外部动作先完成，结果再进入 checkpoint，最后才返回结束状态。只要这个顺序守住，恢复后就能判断“这一步已经成功”，不会把系统确认和外部成功混成一件事。

如果你在 q6 里看到“最终回复的语气”，就别把它当证据；如果你在 q10 里不知道该怎么讲“成功但确认丢失”，就回到这里，把这个窗口和上一段的最小恢复协议连起来看。你已经能把一次长任务的状态顺序说清楚了。

# 案例 {#case}
<!-- tag: worked case -->
<!-- layout: grid2 -->

研究助理生成报告并等待确认后发送
一个 agent 接到任务：收集资料，生成中文报告，等待用户确认，再把报告发给项目组。任务可能持续十几分钟，也可能因为人工确认跨越一天。中间任何一步都可能因为 worker 重启、网络中断或部署发布而停住。

**T0 目标进入**创建 workflow execution，记录 task_id、目标、发起人和版本。
**T1 计划生成**调用 LLM Activity 生成计划，历史保存结果，trace 记录 generation span。
**T2 资料检索**调用检索工具 Activity，保存 source_ids 和摘要，不把整份资料硬塞进历史。
**T3 草稿写入**正文写入业务存储，checkpoint 只保存 draft_id 和必要摘要。
**T4 人工等待**Workflow 等待 approved signal，任务暂停在可恢复状态里，而不是挂死一个请求。
**T5 确认发送**发送 Activity 使用 idempotency_key，避免恢复或重试导致重复发送。

## box warn
### 中断点推演
**如果 T2 前 worker 崩溃：**恢复后 replay 会重建 T0、T1，继续调度 T2。
**如果 T3 后 worker 崩溃：**恢复后不应重新生成草稿，而应读取已记录的 draft_id。
**如果 T5 邮件已发但结果未确认：**本页先把“成功但确认丢失”的窗口看见，完整修法放到 03 的幂等和副作用隔离。

# 操作桥 {#bridge}
<!-- tag: next check -->

把这页放回真实系统时，先打开三处：Workflow history、业务存储、trace。先看历史里的 step 和 signal，确认恢复位置有没有被记住；再看业务存储里的 `draft_id`、`approval_state`、`send_id`，确认中间成果有没有稳定引用；最后看 trace 里的 `tool.call.plan`、`tool.call.fetch`、`tool.call.send`，确认失败发生在编排、工具还是发送窗口。

# 追踪字段 {#trace}
<!-- tag: observability -->

持久执行解决的是“能不能继续”，追踪解决的是“为什么会这样继续”。两者都需要，但职责不同。历史负责恢复，trace 负责解释；如果把 trace 当成恢复状态，系统一旦要重放，就会把“观察记录”误当成“执行依据”。

span / 事件记录了什么对排查有什么用缺失时会怎样


workflow.start任务 id、输入摘要、目标帮你确认是否同一单任务你会把老任务和新任务混在一起
tool.call.plan工具名、参数摘要、计划版本帮你看计划是否重新生成了你不知道计划是不是第二次算出来的
tool.call.fetchsource_ids、查询条件、耗时帮你判断资料检索是否卡住你只能看到“失败了”，不知道慢在哪
tool.call.senddraft_id、幂等键、外部请求 id帮你判断有没有重复发送恢复后重发的风险看不见
workflow.waitapproval_state、等待位置帮你知道任务是在等人还是卡住了排查的人会误以为流程死了

真正有用的 trace 会把过程拆成几次有意义的写入，而不是把所有文本都堆进去。每个写入点都要回答一个问题：这一步有没有改变状态、有没有改变外部世界、有没有留下可回放的证据。只要这三个问题没答清，回放就还不能算回放。

trace 最小清单复制

```text
1. 输入摘要和任务标识有没有记录
2. 工具调用有没有带幂等键或稳定引用
3. 工具结果有没有带 error.type 和耗时
4. 人工等待点有没有记录 approval_state
5. 如果这些字段都没有，先去补哪一层
```

# 常见错误 {#mistakes}
<!-- tag: fixes -->
<!-- layout: grid2 -->

## box
### 错误 1：把 LLM 调用写进可重放编排代码
LLM 输出会随模型版本、采样和服务状态变化。放在 Workflow 里会破坏 replay 的确定性。
**修法：**把 LLM 调用封装为 Activity，结果只留下引用和必要摘要。

## box
### 错误 2：把 trace 当检查点
trace 很适合排查，但它不能驱动恢复状态机。trace 丢了不该影响业务继续，事件历史坏了才会影响恢复。
**修法：**恢复状态进事件历史或 durable store，trace 用相关 id 串起来。

## box
### 错误 3：恢复后重复发送外部副作用
邮件、扣款、提交 PR、删除文件都要考虑“已经成功但确认丢失”的窗口。
**修法：**这一步先记录窗口，下一页去 03 处理幂等、唯一约束和外部状态查询。

## box
### 错误 4：把完整 prompt 和工具结果都写进历史
这会放大成本、隐私和性能问题。历史只保留恢复必须的信息，大内容放受控存储。
**修法：**保存 task_id、plan_version、source_ids、draft_id、approval_state、idempotency_key。

## box
### 错误 5：用系统时间、随机数、全局状态驱动 Workflow 分支
这些值在 replay 时可能不同，导致历史与代码生成的命令不一致。
**修法：**把不确定计算放进 Activity，或者使用 replay-safe 的历史事件结果。

## box
### 错误 6：观测字段没有统一命名
一个服务叫 tool，另一个叫 function，另一个只写 message，后续就很难跨系统查询。
**修法：**统一用 `gen_ai.operation.name`、`gen_ai.provider.name`、`gen_ai.request.model`、`gen_ai.tool.name`、`error.type`。

# 边界与去向 {#boundary}
<!-- tag: boundary -->

这一页先不追完整的幂等实现、重试策略、结构化输出和追踪后端接入。它们当然重要，但不该抢走你对“恢复点在哪里、哪些动作会被重放、哪些状态必须被保存”的注意力。

去 03 之前
先确认问题是否集中在“外部动作会不会重复”，再进入幂等与副作用隔离。


去 04 之前
先确认问题是否集中在“工具慢、偶发失败或超时”，再进入重试策略。


去 05 / 06 之前
先确认问题是否集中在“字段漂移”或“过程看不见”，再看结构边界和追踪语义。

### 4 synthesis

**Q:** 用三句话解释恢复和重试的差别。

Explain: 合格答案要有三个点：1. 恢复发生在 worker / 进程 / 网络中断之后；2. 重试发生在某个 Activity 或工具动作失败之后；3. 恢复负责接着走，重试负责重新尝试单步动作。只要你把“运行层中断”和“动作层失败”分开，就算答到了要点。

### 5 classification

**Q:** 把以下动作分类：生成摘要、写数据库、调用支付、读取已保存的模型输出、发送邮件。

Explain: 可重放逻辑：选择下一步、组装计划、读取已有历史结果。可重试活动：写数据库、读取外部数据、生成摘要（如果允许重新调用）。不可重复副作用：调用支付、发送邮件。关键是说明“重复一次会不会改变外部世界”。
