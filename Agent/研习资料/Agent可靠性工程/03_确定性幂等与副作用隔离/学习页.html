确定性幂等与副作用隔离

这一页接着 02 的恢复语义往前走：当同一条退款、发信或写库意图因为重试、重放或人工重复提交再次进入系统时，Workflow 只负责决定下一步，Activity 才碰外部世界；幂等记录负责挡住重复到达，trace 负责留证据。先把边界放稳，重试策略、结构化输出和完整追踪后端放到后面的单元。

回到 02 持久执行与检查点 继续看 04 工具失败、超时与重试策略

读完本页应该能做什么

  • 能把一个外部动作拆成 Workflow 决策、Activity 执行、幂等记录和 trace 证据四层。
  • 能为退款、扣款、发信这类写操作设计最小幂等键，并判断同 key 冲突时该返回什么。
  • 能从一段重复调用记录里分清重放、重试和重复副作用，并知道下一步去 04、05 还是 06。

先把问题分清

目标

这页先处理重复请求到达时的四个判断：是不是同一条业务意图、是不是同 key 但参数变了、外部动作有没有已经真实执行过、过程能不能用 trace 说清楚。只要这四件事还混在一起，就先把边界拆开，再往下看。

先看退款这条线

主线

先用客服退款这条线把问题讲清楚：用户提交退款，系统要先决定这是不是同一条意图，再决定由谁去执行外部动作，最后把证据写回来。Workflow 负责判断和编排，Activity 负责和支付、邮件、数据库打交道，幂等记录负责识别重复到达，trace 负责留下排查线索。

确定性与重放

重放

Temporal 里的 Workflow Definition 负责定义流程。它会被重放，所以 Workflow 里能读到的时间、随机值或不稳定外部数据，都必须走 Temporal SDK 能记录进历史的路径；否则同一段历史在重放时会走出不同的命令序列。结论很直接：决定下一步做什么的逻辑放在 Workflow，真正会碰外部系统的动作放到 Activity。

代码
async def customer_refund_workflow(request):
 intent = build_refund_intent(request)
 draft = await execute_activity(build_refund_draft, intent)

 approval = await wait_for_signal("approved")
 if not approval.ok:
 return {"status": "waiting", "draft_id": draft["draft_id"]}

 refund = await execute_activity(
 create_refund,
 draft,
 idempotency_key=draft["idempotency_key"],
 )
 await execute_activity(send_confirmation, refund["confirmation_key"])
 return {"status": "done", "refund_id": refund["refund_id"]}

这段代码里，Workflow 只整理意图、等待确认和串起 Activity；创建退款和发送确认都在 Activity 里执行。这样做的价值重放时不会把支付和发信一起卷回去。

如果你只需要一个一次性的非确定性值，例如 UUID 或随机分支结果，可以用 Temporal 提供的 Side Effect 机制把结果写进历史。它只适合记录值，不适合把真正的外部写动作塞回 Workflow。

幂等记录

幂等

幂等要解决的是“重复到达”。同一条退款意图第二次进来时，系统不能把它当作一件新事去执行。在 Temporal 这类系统里，Activity 要能安全重复执行，尤其是带写操作的 Activity。最稳妥的做法，是把业务意图、请求摘要、执行状态和外部请求号一起记下来。

代码
def create_or_reuse_refund(intent):
 record = store.lookup(intent["idempotency_key"])

 if record and record["request_fingerprint"] != intent["request_fingerprint"]:
 return {"status": "conflict", "reason": "same key, different intent"}

 if record and record["status"] == "success":
 return {"status": "reused", "result_ref": record["result_ref"]}

 if not record:
 store.insert({
 "idempotency_key": intent["idempotency_key"],
 "request_fingerprint": intent["request_fingerprint"],
 "status": "pending",
 })

 result = payment_api.refund(
 order_id=intent["order_id"],
 amount=intent["amount"],
 key=intent["idempotency_key"],
 )

 store.mark_success(
 intent["idempotency_key"],
 provider_request_id=result["request_id"],
 result_ref=result["refund_id"],
 )
 return result

这里的关键在于把三种情况分清：第一次进来、同 key 同意图再次进来、同 key 但意图变了。前两种要能复用，第三种要冲突退出，不能悄悄沿用旧结果。

副作用隔离

边界

副作用隔离是把“决定做什么”和“真的去做”拆开。Temporal 的 Activity 负责碰外部世界，所以它可以失败、重试、心跳和恢复；Workflow 则负责保存流程骨架。少量一次性的非确定性值可以用 Side Effect 写入历史，但真正会改现实世界的动作，应该进入 Activity，再用幂等记录处理。

代码
def send_confirmation(refund_id, idempotency_key, provider_request_id):
 record = notify_store.lookup(idempotency_key)
 if record and record["status"] == "sent":
 return record["message_id"]

 message_id = mailer.send(
 subject="Refund confirmed",
 body=f"refund={refund_id}, provider_request_id={provider_request_id}",
 key=idempotency_key,
 )
 notify_store.mark_sent(idempotency_key, message_id)
 return message_id

这类代码的职责很窄：先查是否已经发过，再发，再写记录。编排留在 Workflow，外部效果留在 Activity，记录留在幂等存储和 trace。

案例：同一条退款重复进来

案例

下面这条线把前面的概念串起来：用户先提交退款，系统已经把请求送到支付网关；后来网关响应超时，Workflow 被重放或活动被重试；再过一会儿，用户又点了一次退款。你要能判断哪一步该复用旧结果，哪一步该重新执行，哪一步该直接冲突退出。

代码
def refund_activity(intent):
 record = store.lookup(intent["idempotency_key"])
 if record and record["request_fingerprint"] != intent["request_fingerprint"]:
 raise ConflictError("same key, different intent")
 if record and record["status"] == "success":
 return record["result_ref"]

 if not record:
 store.insert_pending(intent)

 result = payment_api.refund(
 order_id=intent["order_id"],
 amount=intent["amount"],
 request_key=intent["idempotency_key"],
 )

 store.mark_success(
 intent["idempotency_key"],
 provider_request_id=result["request_id"],
 result_ref=result["refund_id"],
 )
 return result["refund_id"]

落地顺序

落地

把这一页的思路落到真实系统时，顺序很重要：先定稳定的 idempotency_key，再补 request_fingerprint，然后把支付、发信、写库这类外部动作收进 Activity，最后把 attempt、provider_request_id 和 result_status 写进 trace。先做这四步，后面才知道自己是在复用旧结果、挡住重复副作用，还是在处理真正的新请求。

追踪字段

证据

trace 的作用是给排查留证据，幂等记录负责做决定。OpenAI Agents SDK 的 tracing 直接把一次运行里的 generation、tool、handoff、guardrail 和自定义事件收进 trace；OpenTelemetry 的 GenAI 事件与指标则把这些调用放进统一的观测语言。没有过程，就很难判断重复调用到底有没有被挡住。

代码
{
 "trace_id": "trace_2f6f1d...",
 "workflow_name": "customer_service_refund",
 "activity_type": "create_refund",
 "attempt": 2,
 "idempotency_key": "refund:order_123:v1",
 "request_fingerprint": "order_123|120.00|approved",
 "provider_request_id": "pay_8891",
 "result_status": "success"
}

这一段摘要已经足够回答三件事：同一条意图有没有重试，外部请求有没有对上，最终结果是复用还是冲突退出。它比一大串原始日志更适合做回放和回看。

常见错误

mistakes
错误为什么会出事应该怎么改
在 Workflow 里直接调支付、发信、写库重放时会把外部动作一起带回去，重复副作用很难挡把外部动作下沉到 Activity，让 Workflow 只做编排
有重试就不做幂等重试只会把不安全的动作放大先设计幂等键，再谈重试策略
同 key 复用不同金额、不同收款方的请求把新意图误判成旧请求，钱和状态都会错加入 request_fingerprint，冲突就返回新的业务意图版本
trace 里只有最终答案，没有 attempt / provider_request_id / key只能看结果，不能追过程把调用链的关键字段写进 span 和事件
把 Side Effect 当成外部请求入口Side Effect 只适合记录一次性非确定性值，不适合真实副作用外部请求仍然用 Activity，必要时再给它加幂等记录
把所有问题都归到 prompt 或模型质量很多重复问题根本在运行层和边界层先看 Workflow、Activity、幂等记录和 trace，再决定要不要碰 prompt

边界与去向

去向

到这里就够用了。你现在应该已经能把“重放”和“重复副作用”分开，也能在退款、扣款、发信这类动作上先写幂等键，再决定 Activity 怎么执行。剩下的几个相邻问题先别一起追：超时和退避去 04，结构化输出和字段漂移去 05，过程看不见和字段怎么统一去 06，历史太长要收尾再回 02 的 Continue-As-New。

。好答案要包含四层：Workflow 负责编排和重放；Activity 负责外部写操作和可重试执行；幂等记录负责识别同一意图并防止重复副作用；trace 负责留下能回看的证据链。只要有一层说成“也负责其他所有事情”，就说明边界还没分清。
