# 先看退款这条线 {#overview}
<!-- tag: 主线 -->

先用客服退款这条线把问题讲清楚：用户提交退款，系统要先决定这是不是同一条意图，再决定由谁去执行外部动作，最后把证据写回来。Workflow 负责判断和编排，Activity 负责和支付、邮件、数据库打交道，幂等记录负责识别重复到达，trace 负责留下排查线索。

对象
读什么
写什么
为什么要分开
错了会怎样


Workflow
历史、当前状态、业务意图
下一步调度、继续位置
它要能重放，不能把外部世界当作内存来用
重放时走出不同分支，或者把已经完成的步骤再做一次


Activity
参数、幂等键、必要摘要
外部请求、结果、错误类型
它可以失败、重试、恢复，但必须把副作用关在边界里

幂等记录
同一 key 是否来过、请求摘要是否变了
pending / success / conflict / result

Trace
调用链、尝试次数、外部请求号、错误类型
证据链
它回答“发生了什么”，不负责替你恢复状态
只能看到结果，不能解释原因

# 确定性与重放 {#mechanism}
<!-- tag: 重放 -->

Temporal 里的 Workflow Definition 负责定义流程。它会被重放，所以 Workflow 里能读到的时间、随机值或不稳定外部数据，都必须走 Temporal SDK 能记录进历史的路径；否则同一段历史在重放时会走出不同的命令序列。结论很直接：决定下一步做什么的逻辑放在 Workflow，真正会碰外部系统的动作放到 Activity。

动作
能不能放 Workflow
原因
例子


根据历史决定下一步
可以
这是 Workflow 的本职工作，重放时应得出同样决策
先查订单还是先查余额


读取当前时间或随机数
只在 SDK 记录过结果时才行
如果值每次变，重放就会分叉
为一次性分支生成 UUID


发起退款 HTTP 请求
不应直接放
外部动作会重复影响现实世界
调用支付网关创建退款


写入数据库或发送邮件
不应直接放
这些动作必须通过幂等边界或 Activity 保护
写状态表、发确认信

退款流程的边界示意复制

```python
async def customer_refund_workflow(request):
 intent = build_refund_intent(request)
 draft = await execute_activity(build_refund_draft, intent)

 approval = await wait_for_signal("approved")
 if not approval.ok:
 return {"status": "waiting", "draft_id": draft["draft_id"]}

 refund = await execute_activity(
 create_refund,
 draft,
 idempotency_key=draft["idempotency_key"],
 )
 await execute_activity(send_confirmation, refund["confirmation_key"])
 return {"status": "done", "refund_id": refund["refund_id"]}
```

这段代码里，Workflow 只整理意图、等待确认和串起 Activity；创建退款和发送确认都在 Activity 里执行。这样做的价值重放时不会把支付和发信一起卷回去。

如果你只需要一个一次性的非确定性值，例如 UUID 或随机分支结果，可以用 Temporal 提供的 Side Effect 机制把结果写进历史。它只适合记录值，不适合把真正的外部写动作塞回 Workflow。

# 幂等记录 {#idempotency}
<!-- tag: 幂等 -->

幂等要解决的是“重复到达”。同一条退款意图第二次进来时，系统不能把它当作一件新事去执行。在 Temporal 这类系统里，Activity 要能安全重复执行，尤其是带写操作的 Activity。最稳妥的做法，是把业务意图、请求摘要、执行状态和外部请求号一起记下来。

字段
用途
什么时候看它
看错会怎样


idempotency_key
识别同一条业务意图
每次 Activity 开始前
把同一件事当成不同请求，或者把不同请求当成同一件事


request_fingerprint
判断 key 相同但参数是否变了
key 已存在时
同 key 不同金额、不同收款人也被复用旧结果


status
区分 pending / success / conflict / failed
决定要复用、重试还是阻断
把还没完成的请求当成已经成功


provider_request_id
把本地记录对上外部系统
发生超时、重试、人工排查时
外部请求已经成功，但本地看不出来


result_ref
复用已完成结果
同 key 重复到达且 fingerprint 一致时
每次都重新走一遍外部副作用

幂等记录的最小处理复制

```python
def create_or_reuse_refund(intent):
 record = store.lookup(intent["idempotency_key"])

 if record and record["request_fingerprint"] != intent["request_fingerprint"]:
 return {"status": "conflict", "reason": "same key, different intent"}

 if record and record["status"] == "success":
 return {"status": "reused", "result_ref": record["result_ref"]}

 if not record:
 store.insert({
 "idempotency_key": intent["idempotency_key"],
 "request_fingerprint": intent["request_fingerprint"],
 "status": "pending",
 })

 result = payment_api.refund(
 order_id=intent["order_id"],
 amount=intent["amount"],
 key=intent["idempotency_key"],
 )

 store.mark_success(
 intent["idempotency_key"],
 provider_request_id=result["request_id"],
 result_ref=result["refund_id"],
 )
 return result
```

这里的关键在于把三种情况分清：第一次进来、同 key 同意图再次进来、同 key 但意图变了。前两种要能复用，第三种要冲突退出，不能悄悄沿用旧结果。

# 副作用隔离 {#isolation}
<!-- tag: 边界 -->

副作用隔离是把“决定做什么”和“真的去做”拆开。Temporal 的 Activity 负责碰外部世界，所以它可以失败、重试、心跳和恢复；Workflow 则负责保存流程骨架。少量一次性的非确定性值可以用 Side Effect 写入历史，但真正会改现实世界的动作，应该进入 Activity，再用幂等记录处理。

边界
读什么
写什么
允许重复吗
错放后的症状


Workflow
历史、signal、参数、前一步结果
下一步调度、继续位置
可以重放
同一条历史重放出不同命令序列


Activity
业务参数、幂等键、必要摘要
外部请求、结果、错误、请求号
可以重复执行，但结果要一致
重复扣款、重复写库、重复发信


Side Effect
一次性的随机值、UUID、短小的非确定性结果
历史中的已记录值
不应再执行
把会失败的逻辑塞进去，重放时难排查


Trace
运行事件、尝试次数、外部请求号、错误类型
证据链
可以持续追加
只剩最终答案，过程丢光

Activity 的最小职责复制

```python
def send_confirmation(refund_id, idempotency_key, provider_request_id):
 record = notify_store.lookup(idempotency_key)
 if record and record["status"] == "sent":
 return record["message_id"]

 message_id = mailer.send(
 subject="Refund confirmed",
 body=f"refund={refund_id}, provider_request_id={provider_request_id}",
 key=idempotency_key,
 )
 notify_store.mark_sent(idempotency_key, message_id)
 return message_id
```

这类代码的职责很窄：先查是否已经发过，再发，再写记录。编排留在 Workflow，外部效果留在 Activity，记录留在幂等存储和 trace。

# 落地顺序 {#bridge}
<!-- tag: 落地 -->

把这一页的思路落到真实系统时，顺序很重要：先定稳定的 `idempotency_key`，再补 `request_fingerprint`，然后把支付、发信、写库这类外部动作收进 Activity，最后把 `attempt`、`provider_request_id` 和 `result_status` 写进 trace。先做这四步，后面才知道自己是在复用旧结果、挡住重复副作用，还是在处理真正的新请求。

先做什么
为什么先做它
做完后能检查什么


定义 `idempotency_key`
先把同一件事识别出来，后面才谈复用
重复提交会不会命中同一条记录


补 `request_fingerprint`
同 key 但参数变了时，不能沿用旧结果
不同金额、收款方或版本能不能触发冲突


拆出 Activity
把支付、发信、写库关进可重试边界里
重放时会不会重复碰外部世界


补 trace 字段
让过程可回看，便于判断有没有被挡住
能不能从 trace 找到 attempt 和外部请求号

如果你是在维护已有系统，先改 schema 和记录，再改执行边界，最后补观测字段。先改提示词往往最省事，也最容易留下回头补不上的缺口。

# 追踪字段 {#trace}
<!-- tag: 证据 -->

trace 的作用是给排查留证据，幂等记录负责做决定。OpenAI Agents SDK 的 tracing 直接把一次运行里的 generation、tool、handoff、guardrail 和自定义事件收进 trace；OpenTelemetry 的 GenAI 事件与指标则把这些调用放进统一的观测语言。没有过程，就很难判断重复调用到底有没有被挡住。

字段
回答什么问题
典型位置
缺了会怎样


trace_id
这次端到端运行是哪一条
整条流程的根标识
你只能看见零散日志，串不成一条链


span_id / parent_id
哪一步是谁调用的
Workflow、Activity、tool、generation
看不清上下游关系


workflow_name / activity_type
这一步到底在做什么
运行名、活动名
只能看到一堆通用事件，没法分层


attempt
这是第几次重试
Activity 或外部调用记录
不知道是第一次超时还是第四次重试


idempotency_key
这次调用和上一次是不是同一件事
幂等记录和 Activity span
重复和复用分不清


request_fingerprint
同 key 但参数是否变了
请求摘要、校验日志
冲突看不见，只会误复用旧结果


provider_request_id / external_request_id
本地请求和外部系统是否对上
支付、邮件、数据库写入返回值
本地超时后不知道外部到底成功没有


error.type / result_status
这次是成功、冲突还是可重试失败
结束 span 或结果摘要
错误全混在一起，没法分流

一条可读的 trace 摘要复制

```json
{
 "trace_id": "trace_2f6f1d...",
 "workflow_name": "customer_service_refund",
 "activity_type": "create_refund",
 "attempt": 2,
 "idempotency_key": "refund:order_123:v1",
 "request_fingerprint": "order_123|120.00|approved",
 "provider_request_id": "pay_8891",
 "result_status": "success"
}
```

这一段摘要已经足够回答三件事：同一条意图有没有重试，外部请求有没有对上，最终结果是复用还是冲突退出。它比一大串原始日志更适合做回放和回看。

# 常见错误 {#mistakes}
<!-- tag: mistakes -->

错误
为什么会出事
应该怎么改


在 Workflow 里直接调支付、发信、写库
重放时会把外部动作一起带回去，重复副作用很难挡
把外部动作下沉到 Activity，让 Workflow 只做编排


有重试就不做幂等
重试只会把不安全的动作放大
先设计幂等键，再谈重试策略


同 key 复用不同金额、不同收款方的请求
把新意图误判成旧请求，钱和状态都会错
加入 request_fingerprint，冲突就返回新的业务意图版本


trace 里只有最终答案，没有 attempt / provider_request_id / key
只能看结果，不能追过程
把调用链的关键字段写进 span 和事件


把 Side Effect 当成外部请求入口
Side Effect 只适合记录一次性非确定性值，不适合真实副作用
外部请求仍然用 Activity，必要时再给它加幂等记录


把所有问题都归到 prompt 或模型质量
很多重复问题根本在运行层和边界层
先看 Workflow、Activity、幂等记录和 trace，再决定要不要碰 prompt

# 边界与去向 {#boundary}
<!-- tag: 去向 -->

到这里就够用了。你现在应该已经能把“重放”和“重复副作用”分开，也能在退款、扣款、发信这类动作上先写幂等键，再决定 Activity 怎么执行。剩下的几个相邻问题先别一起追：超时和退避去 04，结构化输出和字段漂移去 05，过程看不见和字段怎么统一去 06，历史太长要收尾再回 02 的 Continue-As-New。

**Q:** 判断：trace 可以代替幂等记录，因为它也保存了调用过程。

- [ ] 对
- [ ] 错

Answer: false
Explain: trace 只提供证据，不能替你决定是否复用旧结果。

### 10 synthesis

**Q:** 写出最小证据链：从用户重复提交到你确认是否重复副作用，按顺序列 4 个点。

Explain: 高分答案应按顺序写出：用户请求或任务进入系统、幂等键和请求摘要、外部请求号或结果引用、trace 里的 attempt / error.type / result_status。只写“看日志”不够，因为日志没有明确分层；只写“看 trace”也不够，因为 trace 还要对上幂等记录和外部请求号。

### 11 scenario

**Q:** 支付接口超时，但供应商后来确认已经扣款。下一步先做什么？

- [ ] 直接重试一次，反正有幂等就行
- [ ] 查幂等记录和供应商请求号
- [ ] 先改 prompt，再观察一轮

Answer: 查幂等记录和供应商请求号
Explain: 先查幂等记录和供应商请求号，确认是不是同一笔业务已经成功，不要盲目再打一次。

### 12 synthesis

**Q:** 把本页讲给只写普通 API 的工程师：Workflow、Activity、幂等记录、trace 四者分别负责什么。

Explain: 好答案要包含四层：Workflow 负责编排和重放；Activity 负责外部写操作和可重试执行；幂等记录负责识别同一意图并防止重复副作用；trace 负责留下能回看的证据链。只要有一层说成“也负责其他所有事情”，就说明边界还没分清。
