读完后可以直接做的事

  • 能把工具失败分成临时故障、终态故障、限流、权限失败、参数错误和业务冲突，不再看到“失败”就直接重试。
  • 能为一个工具同时写出单次超时、整体超时、最大尝试次数和退避规则，并看出预算是否已经超了。

先把问题分清

goal

这一页处理的是工具失败后的四个问题：它是临时故障还是终态故障，是单次执行慢了还是整体预算超了，是可以退避重试还是应该直接停下。只要这四件事还混在一起，重试就只会把错误放大。

看到什么 先问什么 这页给什么
接口超时 这是单次慢了，还是总预算已经见底 单次超时、整体超时、退避序列
429 限流 是临时拥堵，还是系统已经不该继续撞上去 限流窗口、退避、熔断
权限不足或参数非法 再试会不会还是同一个错 不可重试错误边界和人工接管条件

这一页读完后，你应该能做三件事：给工具选一个合理超时，给失败类型选下一步动作，给一次失败尝试补上可排查的证据字段。

先看一条工具调用线

main line

把工具调用想成一条窄管道：请求先进入执行器，再进入外部工具，最后把结果和错误带回来。问题通常不在“工具会不会失败”，而在“失败之后系统怎么分流”。如果把所有错误都当成可重试，系统会把参数错误、权限问题和业务冲突一遍遍打给同一个依赖。

对象 读什么 写什么 为什么要分开 错了会怎样
执行器 / Workflow 历史、当前目标、剩余预算 下一步调度、停止或接管 它要做决定，不直接碰现实世界 重放或重复调用时，决策和执行会缠在一起
工具 / Activity 参数、幂等键、超时、错误策略 外部请求、结果、错误类型、请求号 它可以失败，但要把副作用关在边界里 重试把真实副作用放大
追踪 / Trace 每次尝试、错误类型、耗时、外部请求号 证据链 它负责解释发生了什么，不负责决定要不要再试 最后只剩一个失败结果，看不见中间过程

这张表的意思很直接：工具失败会沿着一串读写关系展开。先把关系分清，再谈策略。

失败分类

taxonomy

先把失败拆成六类：临时网络故障、服务不可用、限流、权限失败、参数非法和业务冲突。它们看起来都像“没成功”，但后续动作完全不同。

临时故障 可重试，但要有限度

服务短暂抖动、连接超时、偶发 503 这类问题，通常先退避再试。

权限失败 不重试，先补权限或人工处理

再试也不会变好。继续重试只会浪费时间，还会污染日志。

参数非法 回到输入修正

这里要让模型重新构造参数，或先过结构化校验，避免继续撞同一个错误。

业务冲突 查状态，必要时请求用户决策

例如重复创建、状态已变、资源已不存在。这里要先确认现实世界的当前状态。

错误类型 是不是可重试 更合适的动作 必须记录什么
网络抖动可以短间隔退避后再试工具名、尝试次数、延迟
服务不可用有限重试退避、熔断、降级错误类型、持续时间、影响范围
限流按窗口重试等待可用窗口或降频限流键、等待时间、调用方
权限不足不重试重新授权或人工接管权限范围、用户、资源
参数非法不重试当前参数修正参数或过校验被拒字段、schema 版本
业务冲突通常不重试查询状态、解释冲突、请求决策冲突对象、当前状态、期望状态

这张表的作用是让你先看类别，再看动作。先分流，再处理，顺序不能反。

超时边界

timers

超时是一个预算集合。单次执行上限、整体完成上限、排队上限和心跳上限，约束的是不同阶段。Temporal 里的 Start-To-Close、Schedule-To-Close、Schedule-To-Start 和 Heartbeat Timeout，分别对应执行时长、总时长、排队时长和长任务进度信号。

时间边界 它限制什么 适合发现什么问题 智能体工具里的类比
单次执行上限工具真正运行多久内部卡住、慢查询、外部服务无响应一次网页抓取最多 20 秒
整体完成上限从调度到最终完成的总时间排队、重试和执行总耗时过长整个搜索工具最多 60 秒
排队上限任务等 worker 接手多久worker 不足、队列拥堵、路由错误浏览器工具 10 秒没人接手就失败
心跳上限长任务多久要报一次还活着长任务中途挂死或失联导出文件每 30 秒报告进度
检查项 读什么 写什么 症状
单次执行开始时间、结束时间、当前 attempttimeout_ms、实际耗时工具内部慢，但整体预算还没用完
整体预算用户请求总预算、剩余时间是否继续重试单次没爆，重试加起来先爆了
心跳长任务最新进度last_heartbeat_at任务卡住，外层还以为在跑

如果只设置单次超时，重试可能把总时间拖到用户已经不能接受的区间。这里要看所有尝试加起来还剩多少预算。

重试策略

policy

一个可控的重试策略至少要回答五件事：第一次失败后等多久、下一次等待如何增长、最长能等到多大、最多试几次、哪些错误绝不重试。Temporal Retry Policy 正是围绕这五件事组织的。

$$delay_n = \min(initial\_interval \times backoff\_coefficient^{n-1}, maximum\_interval)$$

这里的 initial_interval 是第一次重试前的等待，backoff_coefficient 是增长倍率，maximum_interval 是封顶值，n 是第几次重试。

参数 回答什么问题 写错会怎样
initial_interval第一次失败后等多久太短会把故障窗撞满，太长会让用户等太久
backoff_coefficient等待如何增长增长太慢会继续压依赖，太快会浪费可恢复机会
maximum_interval等待上限是多少没有封顶会把总时长拖成失控长尾
maximum_attempts最多试几次没有次数上限，策略就会一直拖
non_retryable_error_types哪些错误不该重试权限失败、参数错误、业务冲突会被重复放大
重试预算示意
attempts = 4
single_timeout = 5s
delays = [1s, 2s, 4s]
worst_case = attempts * single_timeout + sum(delays)
            = 4*5 + 1 + 2 + 4
            = 27s

这个公式的意义很朴素：重试会消耗预算。每加一次尝试，就要把单次超时和等待都算进去。做策略时，先看最坏情况能不能被用户预算接受，再看是否需要降级。

案例：客户资料同步工具

case

下面用“客户资料同步工具”演示怎么从错误类型、幂等性和时间预算推导策略。这个工具会写外部系统，所以不能把“失败”简单等同为“再试一次”。

工具失败处理契约示例
tool: sync_customer_status
idempotent: true, by sync_request_id
single_attempt_timeout: 8s
overall_timeout: 35s
retry:
  max_attempts: 3
  initial_interval: 2s
  backoff_coefficient: 2
  maximum_interval: 8s
  retryable_errors:
    - network_error
    - service_unavailable
    - rate_limit_after_wait
  non_retryable_errors:
    - invalid_customer_id
    - permission_denied
    - business_conflict
handoff:
  required_when:
    - non_retryable write error
    - unknown error after first write attempt
    - overall_timeout exceeded
trace:
  record:
    - trace_id
    - span_id
    - tool.name
    - tool.attempt
    - error.type
    - timeout.ms
    - idempotency_key
    - handoff.required
场景 先看什么 下一步
网络超时，但工具本身还没返回是不是临时失败，整体预算还剩多少有限重试，继续看 trace
同一 sync_request_id 再次出现幂等记录是否已成功，fingerprint 是否一致复用结果或冲突退出
权限不足或参数非法错误类型是不是终态修正参数、重新授权或人工接管
 
症状：客户资料同步失败，但没有明确错误类别。
先看：error.type、attempt、timeout.ms、overall_timeout。
顺序 要做什么 目的
1先写 `sync_request_id`把同一业务意图先识别出来
2补 `request_fingerprint`区分同 key 但参数变了的情况
3拆出 Activity把外部写操作关进可重试边界
4补 trace 字段让一次尝试的过程可回看

追踪字段

evidence

trace 的作用是留证据，不能代替策略。OpenAI Agents SDK 的 tracing 会把一次运行里的 generation、tool、handoff 和自定义事件串成 trace；OpenTelemetry 的 GenAI 语义则把这些调用放进统一命名里。没有这些字段，你就很难看出是第一次失败、第二次限流，还是第三次才撞到参数错误。

字段 回答什么问题 缺了会怎样
trace_id这是哪一条端到端流程只能看见零散日志，串不成链
span_id / parent_id哪一步是谁调用的看不清上下游关系
tool.name / activity_type这一步到底在做什么只有通用事件，没法分层
attempt这是第几次重试分不清第一次超时和第四次重试
idempotency_key是不是同一件事重复和复用分不清
request_fingerprint同 key 但参数是否变了冲突看不见，只会误复用旧结果
provider_request_id / external_request_id本地请求和外部系统是否对上本地超时后不知道外部到底成功没有
error.type / result_status这次是成功、冲突还是可重试失败错误全混在一起，没法分流
一条可读的 trace 摘要
{
  "trace_id": "trace_2f6f1d...",
  "activity_type": "sync_customer_status",
  "attempt": 2,
  "idempotency_key": "sync:customer_123:v1",
  "request_fingerprint": "customer_123|active|crm",
  "provider_request_id": "crm_8891",
  "result_status": "success"
}

这段摘要已经足够回答三件事：同一条意图有没有重试，外部请求有没有对上，最终结果是复用还是冲突退出。它比一大串原始日志更适合做回看。

落地顺序

bridge

把这一页的思路落到真实系统时，顺序很重要：先定稳定的 `idempotency_key`，再补 `request_fingerprint`，然后把支付、发信、写库这类外部动作收进 Activity，最后把 `attempt`、`provider_request_id` 和 `result_status` 写进 trace。先做这四步，后面才知道自己是在复用旧结果、挡住重复副作用，还是在处理真正的新请求。

先做什么 为什么先做它 做完后能检查什么
定义 `idempotency_key`先把同一件事识别出来，后面才谈复用重复提交会不会命中同一条记录
补 `request_fingerprint`同 key 但参数变了时，不能沿用旧结果不同金额、收款方或版本能不能触发冲突
拆出 Activity把支付、发信、写库关进可重试边界里重放时会不会重复碰外部世界
补 trace 字段让过程可回看，便于判断有没有被挡住能不能从 trace 找到 attempt 和外部请求号

如果你是在维护已有系统，先改 schema 和记录，再改执行边界，最后补观测字段。先改 prompt 往往最省事，也最容易留下回头补不上的缺口。

限流与熔断

limits

限流、服务不可用和超时很容易被混成一类“工具不稳定”。实际处理时要再分两层：限流说明你撞上了对方的容量边界，熔断说明你已经不该继续把请求往同一个依赖上压。两者都允许系统降速，但不等于都应该立刻重试。

状态 触发条件 读什么 写什么 症状 下一步
closed 错误率正常 最近失败率、延迟、attempt 正常调用和追踪 请求可以继续走 按既定重试策略执行
open 连续失败、限流窗口明显、依赖已经不健康 错误阈值、持续时间、剩余预算 暂停外部调用，记录熔断原因 继续撞只会放大失败 降级、排队或接管
half-open 等待一段时间后试探恢复 冷却时间、最新探测结果 少量探测请求 需要判断依赖是否恢复 成功则回到 closed，失败则回到 open
最小分流器伪代码
def decide_next_action(error, attempt, remaining_budget, breaker_state):
    if error.type in {"invalid_argument", "permission_denied", "business_conflict"}:
        return "stop"

    if breaker_state == "open":
        return "handoff"

    if error.type == "rate_limit":
        wait = error.retry_after or min(2 ** attempt, 8)
        if wait >= remaining_budget:
            return "handoff"
        return "retry_after_wait"

    if error.type in {"network_error", "service_unavailable"}:
        if remaining_budget <= 0:
            return "handoff"
        return "retry"

    return "handoff"

这段代码里，读取的是错误类型、重试轮次、剩余预算和熔断状态；写出去的是下一步动作。它的目的在于稳定：让已经终态失败的请求早点停下，让还可能恢复的请求在预算内再试一轮。

场景 先看什么 不要做什么 更合适的动作
429 限流retry_after、调用方、限流键立刻重复撞同一个窗口等窗口、降频或转备用通道
503 / service unavailable持续时间、错误率、剩余预算无上限地继续重试有限退避后熔断
单次超时是不是已经接近整体预算只盯着一次请求结果缩短单次等待或改异步接续

限流和熔断最容易犯的错，是把“还没成功”误看成“应该继续冲”。如果依赖已经明显失稳，先保护总预算和下游系统，再决定是否转接管，通常比继续压请求更有效。

例子 先看什么 结论
retry_after=12s, remaining_budget=8s窗口比预算长直接转接管，不要等到预算耗光
retry_after=4s, remaining_budget=25s窗口短于预算可以退避后再试，但要继续看尝试次数
连续三次 503依赖是否已明显失稳进入 half-open 探测，不要把完整请求再压上去

一条分流链

decision

把这些规则合在一起看，实际分流顺序只有一条：先看错误类别，再看依赖状态，再看剩余预算。这个顺序会先于“重试几次”决定结局。

如果你只记一条经验，就记住这三个判断：终态错误直接停，容量边界先看窗口，依赖失稳先看熔断。只要这三个判断先走完，后面的重试次数才有意义。

输入 先判什么 再判什么 最后动作 会留下什么证据
permission_denied 这是终态错误 不用看重试窗口 停止或接管 error.type、resource、scope
rate_limit 这是容量边界 看 retry_after 和预算 等窗口后再试或降频 retry_after、attempt、delay
service_unavailable 依赖已经不稳 看错误持续时间和熔断状态 有限退避后熔断 breaker_state、duration、trace_id
timeout 看是单次还是整体 看剩余预算和是否可异步接续 缩短单次等待、改异步或停止 timeout.ms、overall_timeout、attempt
从错误到动作的最短决策器
def next_action(error, remaining_budget, breaker_state):
    terminal = {"invalid_argument", "permission_denied", "business_conflict"}
    if error.type in terminal:
        return "stop"

    if breaker_state == "open":
        return "handoff"

    if error.type == "rate_limit":
        if error.retry_after and error.retry_after < remaining_budget:
            return "retry_after_wait"
        return "handoff"

    if error.type == "service_unavailable":
        return "retry" if remaining_budget > 0 else "handoff"

    if error.type == "timeout":
        return "retry" if error.single_attempt and remaining_budget > error.single_attempt else "handoff"

    return "handoff"

这段分流器有意写得很短。它给执行器划硬边界：能不能再试，先看错误类型；该不该停，先看熔断状态；能不能再等，先看剩余预算。

症状：权限失败但系统还在继续重试。\n先看：error.type、scope、resource、breaker_state。\n动作：停止并转接管。

常见错误

mistakes
错误 为什么会出事 应该怎么改
只设单次超时，不设整体预算重试会把总时间拖成失控长尾同时设置单次超时和整体完成上限
写操作没有幂等就自动重试重复副作用会直接放大先做幂等键和状态查询，再谈重试
把所有错误都当成暂时性故障权限失败、参数非法、业务冲突会被重复放大按错误类别设置 retryable 和 non-retryable
trace 里只留最终结果看不见尝试次数、耗时和外部请求号每次尝试都生成可关联 span
429 后立刻重试会把限流窗口撞得更满读取限流等待时间，或执行更保守的退避
把人工接管当成失败资金、权限、删除类动作会被硬挤回自动流把接管条件写进契约，保留上下文给人判断

接管台账

handoff

当错误已经不是“再等一轮就可能好”，系统就该从自动分流转到接管。接管不是失败收尾，而是把风险、证据和下一步动作整理成一个人能立刻接手的包。最常见的误区是只把错误扔给人工，没有留下足够的上下文，结果人工接手后还是得重新查一遍。

要接管的情形 先看什么 接管时写什么 谁来处理 接管后先做什么
终态错误 error.type、resource、scope 停止原因、可疑输入、上一次有效状态 业务负责人或值班人 补权限、修参数、或确认业务冲突
预算耗尽 remaining_budget、attempt、delay 已经用掉的时间、下一步建议 值班人或上游系统 降级、排队或转异步
外部系统不稳 breaker_state、error.type、duration 熔断时间、恢复探测条件 平台或基础设施值班人 等待恢复或切换备用通道
写操作不确定成功 idempotency_key、provider_request_id、result_status 是否可能重复副作用、现有证据链 业务负责人和必要的人审 查外部结果，再决定复用、冲突退出或补偿 
症状：系统已经连续重试多次，但错误没有变短。\n先看：error.type、remaining_budget、breaker_state。\n动作：接管并留下证据链。

这张台账的作用，是把“要停”具体化。你不是在决定系统该不该努力，而是在决定它继续努力的代价会不会比收益更高。

练习

diagnostic
作答目标

每题都尽量说出它属于哪一类失败、应该看什么字段、下一步为什么这样选。先把错误分成可重试、不可重试和需要接管三类，再回答具体参数。

检查重点

先看你能不能把可重试和不可重试分开，再看你能不能把单次超时和整体预算分开，最后才看 trace 字段和接管信息是否完整。

第一组：先分失败类型

这组只看错误分类，不急着谈具体参数。

复习句 临时故障可以有限重试，权限失败和参数非法要先停下来。
choicefailure class

哪一项最适合先做有限重试？

先看这是不是“短暂故障”。

true/falsebudget

判断：只要设置了最大重试次数，就不需要整体超时。

次数有限，不等于时间有限。

choiceterminal

权限不足、参数非法和业务冲突都属于哪一类错误？

先看是不是“再试也不会自然变好”的那类错误。

orderingdebug

排序：一次工具失败后，合理的处理顺序是什么？

把下列动作按顺序写出来：判断错误类别、检查幂等和副作用、计算剩余时间预算、决定重试 / 停止 / 接管、记录尝试。

参考答案 / 评分要点

推荐顺序是：先记录尝试，再判断错误类别，再检查幂等和副作用，接着计算剩余时间预算，最后决定重试、停止还是接管。只要顺序错了，后面的判断就容易建立在假设上。

第二组：看参数

这组看超时和重试参数是否成套。

复习句 单次超时、整体预算、退避序列、最大次数和错误边界要一起看。
choicepolicy

重试策略里，判断“第一次失败后等多久”的参数是哪个？

先看等待起点，不要先看次数。

true/falsebackoff

判断：backoff_coefficient 只影响单次等待，不影响总预算。

每次等待变长，总耗时也跟着变。

fillpolicy

填空：一个基础重试策略至少要包含哪五类信息？

把“等待、增长、封顶、次数、错误边界”都写上。

synthesisbudget

用一句话说明为什么整体预算要和最大重试次数一起看。

答题时至少说出“次数有限不等于时间有限”这一层，再补上等待和排队会叠加。

参考答案 / 评分要点

高分答案要说明：最大重试次数只限制尝试轮数，整体预算还会被每次单次超时、退避等待和排队时间一起消耗。如果不同时看整体预算，策略可能在次数没到之前就已经把用户能接受的时间耗光了。

第三组：看证据和案例

这组检查你是否能把 trace、幂等键和外部请求号串起来。

复习句 幂等记录决定结果能不能复用，trace 决定过程能不能看见。
choicetrace

要判断一次退款是否已经真实发过，最关键的两个字段组合是什么？

一个负责找同一意图，一个负责对外部系统。

true/falseevidence

判断：trace 可以代替幂等记录，因为它也保存了调用过程。

证据和决策不是一回事。

synthesisdiagnosis

写出最小证据链：从用户重复提交到你确认是否重复副作用，按顺序列 4 个点。

建议按“请求进入系统 - 幂等记录 - 外部请求号 - trace”来答。不要把结果和证据混在一起。

参考答案 / 评分要点

高分答案应按顺序写出：用户请求或任务进入系统、幂等键和请求摘要、外部请求号或结果引用、trace 里的 attempt / error.type / result_status。只写“看日志”不够，因为日志没有明确分层；只写“看 trace”也不够，因为 trace 还要对上幂等记录和外部请求号。

第四组：迁移到真实系统

把同一套边界规则迁移到写库、发信和同步场景里。

复习句 先查幂等，再查外部请求号，再决定重试还是冲突。
scenariotransfer

支付接口超时，但供应商后来确认已经扣款。下一步先做什么？

超时不等于失败，先确认现实世界有没有已经发生。

synthesisexplain

把本页讲给只写普通 API 的工程师：Workflow、Activity、重试策略、trace 四者分别负责什么。

答题时要同时说出“谁决定下一步、谁碰外部世界、谁限制重复、谁留证据”。

参考答案 / 评分要点

好答案要包含四层：Workflow 负责编排和重放；Activity 负责外部写操作和可重试执行；重试策略负责限制等待、次数和错误边界；trace 负责留下能回看的证据链。只要有一层被说成“也负责其他所有事情”，就说明边界还没分清。
