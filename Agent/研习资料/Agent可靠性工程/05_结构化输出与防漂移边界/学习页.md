# 学习目标 {#goals}
<!-- tag: goal -->

**钉住形状**
用结构化输出限制字段、类型、枚举和版本，先让后续程序能稳定解析。


**钉住语义**
用业务规则、权限、风险分流和 trace 字段，防止“看起来合法”的输出继续扩散错误。


**钉住恢复**
在 Temporal 这类持久执行系统里，把已验证的结构化决策保存成历史事实，而不是每次重放都重新生成。

# 先抓主线 {#map}
<!-- tag: map -->

这一单元的主线很短：先让输出有结构，再让结构过校验，再把校验结果写进追踪，最后把已验证的事实放进可恢复的执行系统。只要任意一层开始漂，后面的工具、回放和评测都会跟着歪。

# 一、漂移不是玄学，是接口契约被模型侵蚀 {#lesson-1}
<!-- tag: lesson 1 -->

先给直觉：只要后续程序要消费模型输出，模型就不再只是“写一段话”，而是在填一个接口。如果接口没有被系统明确定义，模型会按语言习惯补全、解释、压缩或改写字段。

漂移至少有四种。格式漂移是结构不对；字段漂移是名称和类型变了；语义漂移是字段值看似合法但业务含义错了；流程漂移是模型跳过确认、审批或权限边界，直接把动作推到执行层。

**格式**解析失败，后续程序拿不到稳定结构。
**字段**字段名或类型变化，映射层变脆。
**语义**字段合法，但业务含义变了。
**流程**模型越过确认或权限边界。


---

# 二、Structured Outputs：先把输出形状钉住 {#lesson-2}
<!-- tag: lesson 2 -->

结构化输出的价值，是把“请尽量按这个格式”升级成“这个接口只接受这种结构”。对象字段明确、必填项明确、类型明确、枚举封闭，才能让下游程序稳定解析和校验。

但是 schema 只能约束形状，不能证明事实正确。一个 JSON 可能完全合法，却仍然是错误订单、越权动作或不该执行的金额。形状是入口，业务校验才是门禁。

Schema 边界示例JSON Schema复制

```json
{
 "name": "agent_plan_gate",
 "schema": {
 "type": "object",
 "additionalProperties": false,
 "required": ["schema_version", "decision", "risk_level", "requires_human_approval"],
 "properties": {
 "schema_version": { "type": "string", "enum": ["plan_gate.v1"] },
 "decision": { "type": "string", "enum": ["allow", "revise", "block"] },
 "risk_level": { "type": "string", "enum": ["low", "medium", "high"] },
 "requires_human_approval": { "type": "boolean" }
 }
 },
 "strict": true
}
```

## box info
这层负责什么
限制字段、类型、枚举、版本和额外字段。它解决“程序能不能读”的问题。

## box warn
这层不负责什么
不负责判断退款是否允许、用户是否有权限、金额是否在窗口内。那些是业务校验。

# 三、边界要放在每一次系统交接处 {#lesson-3}
<!-- tag: lesson 3 -->

很多人只给最终回答做结构化输出，这不够。真正危险的地方常在中间交接：计划交给执行器、路由器交给子 Agent、模型参数交给工具、工具结果交给下一轮模型、失败摘要交给重试策略。每一次交接都可能漂移。

输入边界->
计划边界->
工具边界->
输出边界

边界位置
要约束什么
失败时怎么办


输入边界
用户意图、来源可信度、敏感数据、任务范围
拒绝、脱敏、澄清或转人工


计划边界
动作类型、顺序、风险等级、是否需要确认
要求重规划，或阻止高风险动作


工具边界
参数类型、资源 ID、权限、幂等键、超时和重试
不调用工具，返回结构化错误


输出边界
最终状态、证据、下一步、用户摘要
保留 trace，进入补救流程

# 四、追踪让漂移可定位：span 要记录边界结果 {#lesson-4}
<!-- tag: lesson 4 -->

trace 和 span 的作用，把一次步骤变成可排查的记录。它要能回答：哪一步坏了、哪个 schema 失败、哪个字段错了、哪次重试发生了、哪个版本引入了变化。

Trace 事件示例JSON复制

```json
{
 "trace_id": "run_2026_06_09_042",
 "span_name": "agent.plan_gate.validate",
 "attributes": {
 "gen_ai.operation.name": "chat",
 "agent.schema.name": "agent_plan_gate",
 "agent.schema.version": "plan_gate.v1",
 "agent.schema.validation": "failed",
 "agent.schema.error_field": "decision",
 "agent.model": "configured-model-id",
 "agent.retry.count": 1
 },
 "events": [
 { "name": "schema_validation_failed", "reason": "enum_value_not_allowed" }
 ]
}
```

## box soft
span 里放什么
模型名、工具名、步骤名、版本、尝试次数、耗时和父子关系。它们是长期有用的排查字段。

## box info
event 里放什么
一次工具调用的局部证据、脱敏后的参数、拒绝原因、澄清原因和短期采样结果。

# 五、Temporal 视角：重放时不要重新发明决策和副作用 {#lesson-5}
<!-- tag: lesson 5 -->

持久执行系统关心的是：任务中断后，能不能从历史里恢复，而不是从头再猜一次。Workflow 里保留确定性的编排状态，Activity 承担外部 I/O，模型输出进入系统前先结构化和校验。

**Workflow**保存确定性的任务状态和已验证决策引用。
**Activity**执行外部 I/O：调用模型、访问 API、写数据库、发送消息。
**幂等键**让重试或恢复不会重复制造副作用。

对象
读什么
写什么
错了会怎样


Workflow
确定性状态、分支、已验证决策
恢复后的流程位置
如果塞进随机模型决策，重放会漂


Activity
外部输入、参数、幂等键
真实副作用
如果重复执行，会产生双重写入


Replay
历史事件、已记录结果
复用结果，不重复动作
如果重新发明决策，结果不稳定

# 六、拒绝、修复、降级：结构化失败也要有流程 {#lesson-6}
<!-- tag: lesson 6 -->

结构化输出失败不是终点，而是可设计的分支。先拒绝非法结构，再把错误定位到字段，再决定是否让模型修复。如果涉及权限、金额、删除、发送、提交等高风险动作，不要让模型自动修复后直接执行。

生成结构化输出->
Schema 校验->
业务校验->
风险分流->
执行 / 草稿 / 人审 / 拒绝

失败处理骨架TypeScript-like复制

```ts
async function handlePlan(rawModelOutput) {
 const parsed = validateSchema("agent_plan_gate.v1", rawModelOutput);
 if (!parsed.ok) {
 recordSpanEvent("schema_validation_failed", parsed.errorSummary);
 return { status: "retry_or_repair", reason: parsed.errorSummary };
 }

 const checked = await validateBusinessRules(parsed.value);
 if (!checked.ok) {
 recordSpanEvent("business_validation_failed", checked.reason);
 return { status: "blocked", reason: checked.reason };
 }

 if (checked.value.risk_level === "high" || checked.value.requires_human_approval) {
 return { status: "draft_waiting_for_approval", plan: checked.value };
 }

 return { status: "ready_to_execute", plan: checked.value };
}
```

# 七、把边界落到真实项目：最小上线链路 {#bridge}
<!-- tag: bridge -->

如果你在做客服、退款、工单或审批 Agent，这一页的边界应该直接落到代码里。最小链路通常是：先让模型只产出结构化评估，再把结构化结果交给业务校验器，最后把通过与否写进 trace，并把可恢复的决策交给持久执行系统。这个顺序不能倒，因为每一步失败的含义不同。

先看一个真实感更强的场景。用户要求“把发票改到新邮箱”，系统收到的一个会影响外部写入的动作提议。这里最容易写坏的地方程序有没有把“合法结构”“权限合法”“状态合法”和“恢复合法”分开。只要把它们混在一起，排障时就会只剩下一句“结果不对”，却不知道错在 schema、权限还是幂等。

层
读什么
写什么
错了会怎样


模型输出
用户意图、上下文、候选动作
结构化评估结果
如果没有版本，后面无法对齐


schema 校验
字段、类型、枚举、额外字段
解析通过或可读错误
如果混入业务逻辑，错误会变得模糊


业务校验
权限、状态、金额、窗口、幂等键
允许、拒绝、草稿或接管
如果交回模型决定，越权会继续扩散


Trace / History
版本、字段、重试、恢复位置
可追踪、可复盘、可重放的事实
如果只留聊天记录，问题很难还原

这张表的读法很简单。第一行告诉你，模型输出只是候选，不是执行命令；第二行告诉你，解析错误是接口层问题，不是业务层问题；第三行告诉你，业务系统有权覆盖模型的“建议”；第四行告诉你，追踪和历史以后恢复和排障的依据。把四层混在一起，系统就会在最小错误上放大成生产事故。

最小落地链路骨架pseudo复制

```ts
const plan = validateSchema("refund_assessment.v1", modelOutput);
if (!plan.ok) {
 recordSpanEvent("schema_validation_failed", plan.errorSummary);
 return { status: "retry_or_repair" };
}

const checked = await validateBusinessRules(plan.value);
recordSpanEvent("business_validation_result", checked.ok ? "passed" : checked.reason);

if (!checked.ok) {
 return { status: "blocked", reason: checked.reason };
}

if (checked.value.requires_human_approval || checked.value.risk_level === "high") {
 return { status: "draft_waiting_for_approval", plan: checked.value };
}

return { status: "ready_to_execute", plan: checked.value };
```

## box soft
这条链路先读什么
先读结构化结果，再读业务上下文，再读当前权限和幂等状态。这样每一层的错误都能被单独定位。

## box warn
这条链路先写什么
先写验证结果，再写 trace 事件，再决定是否进入草稿、人审或执行。不要反过来先做副作用。

如果你把这条链路写对，后面的课程会自然连起来：06 里你会知道应该把哪几个字段写进 span，07 里你会知道哪些失败样本必须回放，08 里你会知道该在什么点暂停、回滚或接管。换句话说，这一条可复用的执行边界。

你现在应该能做什么：把一个自然语言执行请求拆成“结构化评估、schema 校验、业务校验、trace 记录、恢复保存”五步，并指出每步的输入和输出。

# 八、看一条失败链：先判断是结构、语义还是恢复问题 {#diagnosis}
<!-- tag: diagnosis -->

同样是“系统没按预期做事”，原因可能完全不同。结构失败说明程序连结果都读不进来；语义失败说明结果能读，但业务不成立；恢复失败说明前面的结果都对了，却在重放和副作用边界上出了问题。把这三类分开，是后面做排障、回放和门禁的前提。

拿“发票改邮箱”来举例。模型可能已经输出了合法 JSON，但如果 `search_vendor` 结果为空，而系统继续走写接口，这决策问题。相反，如果输出里混入了解释文字，程序连 JSON 都解析不出来，那就应该先回到 schema 边界，而不是去猜用户到底想干什么。

症状
先看哪里
下一步动作
最常见误判


输出无法解析
schema、额外字段、类型和枚举
拒绝、修复或重试一次
误当成“模型不聪明”


输出合法但业务不成立
权限、状态、金额、窗口、幂等键
阻断、草稿或人工确认


重放后结果变了
workflow 历史、Activity、replay 标记
把已验证决策保存为历史事实
误当成“偶发抖动”


外部副作用重复发生
幂等键、重试策略、边界位置
停止写路径，先修边界再重放
误当成“再跑一次就好”

最小诊断函数pseudo复制

```ts
function classifyFailure(trace) {
 if (!trace.schema_valid) return "shape";
 if (!trace.business_valid) return "semantic";
 if (trace.replay && trace.side_effect_executed) return "replay";
 if (trace.side_effect_repeated) return "idempotency";
 return "other";
}

const kind = classifyFailure(currentTrace);
if (kind === "shape") {
 return "先回 schema 和字段，不要修业务";
}
if (kind === "semantic") {
 return "先看权限、状态和幂等键，不要只改 prompt";
}
if (kind === "replay" || kind === "idempotency") {
 return "先停写路径，再检查历史和 Activity 边界";
}
```

这个函数的意义不在于你真的照搬它，而在于它逼你先回答“问题属于哪一层”。如果你把结构错误当业务错误，修复会绕远；如果把业务错误当结构错误，你会不停重试；如果把恢复错误当普通失败，你会重复制造副作用。

更实用的判断是：`shape` 先看 schema 和字段，`semantic` 先看权限和状态，`replay` 先看 history 和 Activity 边界。只要这三步顺序写对，工单、回放和上线门禁就能共用同一条诊断链，而不是每个团队各写一套猜测规则。

这也是为什么一个好的诊断页不会只给结论，而是给你下一次该打开的表、该看的字段和该停的路径。

这条判断链也能直接沿用到 06、07、08。

你现在应该能做什么：拿到一条 trace，先判断它是结构、语义还是恢复问题，再决定是拒绝、修复、阻断还是停写路径。

# 防漂移案例：退款智能体的结构化边界 {#case}
<!-- tag: case -->

场景：客服智能体根据用户对话判断是否退款。没有边界时，模型可能把“建议退款”直接变成“执行退款”，把订单号脑补成某个格式，或者把高风险动作绕过确认。

坏设计：自然语言计划直接执行

模型输出退款结论。
程序从文本里抽金额和订单号。
没有检查订单状态、权限、窗口和幂等键。
失败后重试，可能重复退款。


好设计：四层防漂移边界

模型只能输出 `refund_assessment.v1` schema。
枚举只允许 `deny`、`draft_refund`、`needs_human_review`。
业务系统校验订单、金额、权限、政策和幂等键。
trace 记录 schema 版本、业务校验、Activity id 和最终状态。

退款案例：模型不能直接表达执行副作用JSON复制

```json
{
 "schema_version": "refund_assessment.v1",
 "decision": "draft_refund",
 "confidence": "medium",
 "customer_visible_summary": "订单可能符合部分退款条件，建议人工确认。",
 "proposed_refund": {
 "order_id": "ord_7788",
 "amount": 98,
 "currency": "CNY",
 "reason_code": "service_unavailable"
 },
 "required_checks": [
 "order_exists",
 "operator_can_refund",
 "within_refund_window",
 "idempotency_key_reserved"
 ]
}
```

# 常见错误 {#mistakes}
<!-- tag: mistakes -->
<!-- layout: grid2 -->

**字段太自由**把 `action` 写成任意字符串，模型会创造程序不认识的动作。

**没有 schema 版本**上线后改字段，旧 trace 和新执行器混在一起，排查时不知道失败来自模型还是接口变更。

**把安全判断交给模型**模型可以建议风险等级，但不能自己决定越权执行。

**修复循环无限重试**结构化失败后不断要求模型重写，会消耗成本并掩盖系统缺陷。

**trace 字段无法统计**只记录一段文本日志，后续无法按 schema、模型、工具和错误字段聚合。

**恢复时重新调用模型**长任务恢复时再次生成计划，可能得到不同结果。

Explain: 应选 A、B、C。D 只是提示，不是边界。边界必须能被程序检查、记录和追踪。

### 3_notify_fields synthesis

**Q:** 给一个“发送客户通知”的 Agent 设计最小结构化输出字段

Explain: 合格答案至少要写出 schema_version、decision、risk_level、requires_human_approval 和 evidence_fields 之类字段，并说明字段值域。重点是“能执行、能校验、能追踪”。

### 4_refund_checks synthesis

**Q:** 合法 JSON 为什么还不能直接执行退款？

Explain: 可以写客户归属、订单状态、金额上限、退款窗口、幂等键、审批记录等。关键是要说明“合法 JSON ≠ 业务可执行”。

### 5_trace_fields debug-checklist

**Q:** 一次结构化输出失败后，你会记录哪些 trace/span 属性？

- [ ] schema_version
- [ ] schema_name
- [ ] error_field
- [ ] model_version
- [ ] retry_count
- [ ] tool_name

Explain: 这些字段里，至少要有 schema 名称、版本、失败字段、模型版本、重试次数和工具名。它们要能直接帮助你查“哪一步、哪一版、哪一类失败”。

### 6_temporal_plan synthesis

**Q:** 为什么长任务恢复时不应该重新让模型生成计划？

Explain: 重放时重新生成计划会引入不确定性，所以已接受的结构化决策应作为历史事实保存到 workflow 状态或事件历史里。外部副作用放到 Activity，并通过幂等键保护。

### 7_rewrite_flow synthesis

**Q:** 把这句话改造成可靠流程：“模型判断用户应该退款，于是调用 refund API。”

Explain: 合格答案会把“模型说要退款”改成“先评估，再校验，再确认，最后执行”。如果风险高或权限不足，必须停在草稿或人工确认阶段。
