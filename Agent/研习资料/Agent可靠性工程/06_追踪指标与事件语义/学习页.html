06 追踪、指标与事件语义｜智能体可靠性工程

从 05 的结构化输出往前走一步，问题就不再是“模型有没有给出一个像样的结果”，而是“这次失败能不能被定位、这一类问题能不能被统计、replay 和真实副作用能不能区分开”。这一页把 trace、span、event、metric 和 Temporal Event History 放到同一张图里看。

学习目标

goal

先看单次

trace 和 span 先回答“这一次到底在哪一步坏掉了”，适合看路径、耗时、尝试次数和父子关系。

最后看恢复

Temporal Event History 先回答“恢复过没有、重放发生过没有、真实副作用有没有被重复做”。

先抓主线

map

如果你只记一条线，就记这一条：请求先进入 span，span 里可以带 event，event 可以记录短期细节，metric 负责归类，Temporal history 负责长期恢复事实。它们不是同一回事，也不能互相替代。

字段矩阵

table

最容易出错的地方，“放在哪一层记录”。下面这张表先把职责分清，再看哪些字段只能进局部事件，哪些字段可以长期保留。

事件与 span

code

span 适合记录一次步骤本身，event 适合记录这一步里短期但很关键的细节。两者都能写字段，但写法不同：span 里的东西应该稳定、可聚合，event 里的东西可以更细，但保留期通常更短。

代码
{
 "trace_id": "trace_case_20260621_001",
 "span": "CallModel",
 "gen_ai.operation.name": "chat",
 "gen_ai.request.model": "gpt-4.1",
 "event": "tool.arguments.redacted",
 "tool.name": "send_invoice",
 "attempt": 2,
 "workflow_id": "claim-84",
 "run_id": "temporal-run-a7",
 "replay": false
}

event 里放什么

一次工具调用的局部证据、脱敏后的参数、澄清原因、拒答原因、短期采样结果。它们帮助你回看“当时为什么这么做”。

指标与基数

math

指标回答趋势，不回答个案。为了让趋势真的能查，标签必须低基数。直觉上可以把它看成“标签空间一乘，数据量就爆了”。

符号意思很简单：每个 label 的可取值越多，最终时序条目就越多。tool.name 和 agent.version 往往还能控制，trace_id、用户原文、完整路径通常会把序列数撑爆。

Temporal 对齐

workflow

Temporal 的 Event History 不是普通日志。普通日志只能告诉你“发生过什么”，Event History 还告诉你“发生的顺序、恢复的位置、哪一步是 replay 复用的结果”。这对智能体尤其重要，因为副作用必须被关在 Activity 边界里。

代码
{
 "trace_id": "trace_case_20260621_001",
 "workflow_id": "claim-84",
 "run_id": "temporal-run-a7",
 "activity_type": "CallModel",
 "attempt": 2,
 "replay": false,
 "search_attributes": ["claim-84", "invoice", "escalation"],
 "event_history": "ActivityTaskStarted -> ActivityTaskCompleted"
}

哪些字段只该做搜索

搜索属性适合放业务筛选条件，比如单号、流程类型、任务阶段。它们帮助你找回这次运行，但不承担诊断全部细节的责任。

读一条事故链

trace

把一条追踪链读顺，最实用的方法先问四个问题：这次是谁触发的、哪一步最慢、哪一步真的失败了、哪里出现了重复执行。只要这四个问题能答出来，trace 就能顺着走的证据链。

观察点该读什么该忽略什么错了会怎样
入口 span用户目标、模型版本、当前步骤和当前步骤无关的历史碎片会把这次事故误看成老问题
工具 span参数、attempt、error.type、耗时原文的无关修辞看不出是重试还是新决策
event脱敏后的局部证据、拒绝原因、澄清原因长期会爆量的原始全文既不安全，也不好检索
historyworkflow / run / replay / activity把 replay 当成再次执行的错觉会把恢复行为误判成新副作用
代码
{
 "complaint": "发票重复发送",
 "trace": [
 "entry: user asked for resend",
 "span: search_vendor returned empty_result",
 "event: redact vendor lookup payload",
 "span: update_invoice_email attempt=2 replay=false",
 "history: ActivityTaskCompleted"
 ],
 "next_check": [
 "Was the empty result handled with clarification?",
 "Did retry reuse the same idempotency key?",
 "Did replay run a real side effect?"
 ]
}

如果你在 trace 里只能看见“失败了”，还不够。你需要知道失败在哪一步被放大，哪一步本来可以拦住，哪一步因为 replay 或重试把一次错误变成了两次。这个判断会直接影响 07 的回归样本写法，也会影响 08 的接管和回滚策略。

默认采什么，默认不采什么

policy

最稳妥的做法是先给出一个最小默认集，再给出一个按需开放集。默认集只保留排障和趋势足够的字段，按需开放集只在明确需要时打开，而且要有时间限制和访问控制。这样做不是保守，而是把“能查”和“能看”分开。

默认集

trace_id、workflow_id、run_id、step name、attempt、error.type、duration、agent.version、tool.name。它们够你定位个案，也够你看趋势。

按需开放集

原文输入输出、工具参数全文、系统指令全文、RAG 片段全文。它们不能默认长期开、默认多人可见。

字段族默认策略需要时怎么开回头看要注意什么
输入输出摘要或脱敏摘要限时、限人、按工单查看后要回收权限
工具参数保留关键字段和版本必要时打开完整 payload注意是否含密钥或隐私
系统指令只保留引用或摘要对故障样本按需查看不要把它当普通日志
审计事实单独保存按审计角色访问别和调试细节混在一起

这一层的目标很简单：让系统默认足够可查，但不默认过度暴露。等你把这条线立住，07 的回放和 08 的验收才不会在同一个问题上互相打架。

排障顺序

runbook

当一条事故已经进入排障阶段，最重要的按顺序读证据。先看 trace 里发生了什么，再看 metric 是否只是趋势抖动，最后去 event 和 retention 里确认哪些内容该保留、哪些内容该收缩。顺序一旦乱掉，你就会把“这次为什么失败”错看成“系统所有地方都坏了”。

证据层优先看什么误读后果建议动作
tracestep、attempt、replay、error.type把个案误判成大范围退化先确认是哪一次任务出了问题
metric失败率、p95、重试率、人工接管率看不见长期趋势判断是否需要限流、降级或回滚
event局部证据、拒绝原因、澄清动作把原文全文当作默认保留对象改成摘要或脱敏摘要
historyworkflow / activity / replay 关系把恢复行为误当成新副作用重新核对边界和幂等策略
代码
{
 "signal": "duplicate invoice resend",
 "trace": ["entry", "tool attempt=2", "replay=false"],
 "decision": "check retry policy before changing business logic",
 "action": ["pause write path", "review boundary", "keep retention minimal"]
}

把排障顺序写成这样，后面的监控、合规和回归都会更顺：先定位一次事故，再看它是不是一类问题，最后再决定这个字段该不该长期可见。这个顺序正好会接到下一步的样本设计和上线验收上。

保留期与访问边界

policy

追踪越细，排障越快，但风险也越高。默认策略应该从最小必要信息开始，再决定哪些内容短期保留、哪些只留摘要、哪些进入专门的审计记录。

中期保留

脱敏后的 span 属性、错误类型、耗时、版本、attempt、replay。它们够用来排查，但不会把原始内容长期摊开。

数据类型默认保留访问范围判断标准
指标较长平台、SRE、开发负责人只含趋势，不含原文
脱敏 span 属性中等排障相关角色能定位问题，但不暴露原文
原文输入输出短期或按需显式授权、限时访问属于敏感内容，不能默认全开
审计事实按合规要求审计和安全角色和调试内容分开保存

案例

worked

场景：理赔智能体要发送补充材料清单。用户投诉说收到了两封几乎一样的邮件。要判断是模型再次决定发信，工具重试发信，还是 worker 重启后 replay 误触发了副作用。

看 metric

如果 tool_timeout_total、replay_count_total、manual_handoff_total 一起抬头，就说明这恢复、重试和人工接管同时在变坏。

先看什么接着看什么最后确认什么
外部写入有没有发生同一 tool call 是否发生了 retryreplay 期间是否只读历史
错误类型是什么attempt 和 idempotency key 是否一致Activity 边界是否隔离了副作用
哪个字段在暴涨是 trace、metric 还是 event是否需要把这个字段降级为摘要
代码
{
 "first_check": "Was there a real side effect?",
 "second_check": "Did the same call retry with the same idempotency key?",
 "third_check": "Was replay read-only?",
 "if_unknown": [
 "pause the write path",
 "inspect workflow/activity boundary",
 "compare trace, event, and metric separately"
 ]
}

真正能落地的监控，把每一类问题放进对应的观测层：趋势看 metric，个案看 trace，恢复顺序看 history，风险说明看 event。你在设计指标时越早把这四层分开，后面越容易做权限、保留期和排障手册。

如果把这三个层次记住，你就会更自然地把事件语义写成“发生了什么、为什么发生、该不该长期保留”。这也是为什么指标设计边界题：它决定了你以后能不能用同一套数据，同时支持趋势分析、事故定位和合规访问。

常见误区

traps

只存聊天记录

聊天记录缺少 step、attempt、parent-child、replay 和版本信息，排查时你只会知道“说过话”，不知道“做过什么”。

把原文输入输出默认长期保存

很多排障场景确实需要原文，但“需要排障”不等于“默认永久在线”。保留期和访问范围要先写清楚。

只看平均时延

平均值会把尾部问题藏起来。p95 / p99、失败率、重试率和人工接管率通常更能反映真实风险。

练习

diagnostic

先别急着写长答案。先把观测对象分对，再决定是查个案、看趋势，还是追 replay。

二、字段放对层

快速回顾：低基数给 metric，短期细节给 event，恢复事实给 history。 -->
5_label_checkdebug-checklist

Q: 勾出不应该作为 metric label 的字段

参考答案 / 评分要点

。应勾选 trace_id 和 user_input 。 agent.version 与 tool.name 往往是低基数，适合做指标标签。判断标准“值的种类会不会多到让时序爆炸”。

6_trace_treesynthesis

Q: 画一条最小 trace 树

参考答案 / 评分要点

。合格答案至少要包含：入口 trace、agent span、model span、tool span、一个 event 或 metric、以及一个和恢复有关的字段（例如 workflow_id / run_id / attempt ）。说明里要说清楚：span 负责路径，event 负责局部事实，metric 负责趋势，history 负责 replay 事实。

7_retentiondebug-checklist

Q: 把数据分到正确的保留期

参考答案 / 评分要点

。原文输入输出通常应短期保留或按需开放；脱敏 span 属性和计数型指标可以保留更久；审计事实按合规要求单独保存。关键“默认谁能看、看多久、出事时怎么追责”。

三、Temporal 对齐

快速回顾：replay 不是新副作用，history 记录的是恢复事实。 -->
8_temporal_idschoice

Q: 哪组字段最能把 trace 和 Temporal 对齐？

参考答案 / 评分要点

workflow_id + run_id + attempt + replay。workflow_id、run_id、attempt 和 replay 一起，才够把恢复事实对齐。

9_historysynthesis

Q: 为什么 Event History 不是普通日志？

参考答案 / 评分要点

。好答案会说明：Event History 记录的是 workflow 的事件顺序和恢复事实；replay 时会复用已记录的 Activity 结果，而不是重新执行外部副作用；普通日志只是顺序文本，不天然保证恢复语义。只写“能追踪”不够，要把“重放”和“复用”说出来。

10_replay_flagchoice

Q: 哪一条最像 replay，不像新执行？

参考答案 / 评分要点

attempt=2, replay=true, side effect not executed。replay=true 且 side_effect_not_executed，说明复用了记录结果。

11_policysynthesis

Q: 写一条最小可执行的观测保留策略

参考答案 / 评分要点

。合格答案至少要有三层：默认只保留低风险字段和趋势指标；短期保留原文输入输出、工具参数和局部 event；审计事实单独保存、权限单独管理。还要写清楚一个具体动作：谁能申请访问、访问多久、到期怎么回收。
