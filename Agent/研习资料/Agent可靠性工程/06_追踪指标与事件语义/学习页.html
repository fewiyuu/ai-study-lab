观测语义

追踪、指标与事件语义

从 05 的结构化输出往前走一步，问题就不再是“模型有没有给出一个像样的结果”，而是“这次失败能不能被定位、这一类问题能不能被统计、replay 和真实副作用能不能区分开”。这一页把 trace、span、event、metric 和 Temporal Event History 放到同一张图里看。

Trace Span / Event Metrics Temporal

读完这一页，你能直接做什么

  • 把一次事故拆成 trace、span、event、metric 和 history 五层证据。
  • 分清哪些字段能进 span，哪些应该只在事件里短期保留，哪些绝对不该进 metric label。
  • 看见 `workflow_id`、`run_id`、`attempt` 和 `replay` 时，知道它们是在解释恢复，不是在重复执行副作用。

学习目标

goal
先看单次

trace 和 span 先回答“这一次到底在哪一步坏掉了”，适合看路径、耗时、尝试次数和父子关系。

再看趋势

metric 先回答“一类问题是不是在变糟”，适合看分布、计数和比例，不适合承载原始输入文本。

最后看恢复

Temporal Event History 先回答“恢复过没有、重放发生过没有、真实副作用有没有被重复做”。

先抓主线

map

如果你只记一条线，就记这一条：请求先进入 span，span 里可以带 event，event 可以记录短期细节，metric 负责归类，Temporal history 负责长期恢复事实。它们不是同一回事，也不能互相替代。

01
输入 用户请求、工具参数、模型输出、外部返回
02
Span 记录一次步骤的耗时、父子关系、尝试和版本
03
Event 记录短期细节、局部证据、红线字段的脱敏结果
04
Metric 把次数、时长、比例压成低基数趋势
05
History 把 workflow / activity / replay 的事实固定下来

字段矩阵

table

最容易出错的地方，不是“有没有记录”，而是“放在哪一层记录”。下面这张表先把职责分清，再看哪些字段只能进局部事件，哪些字段可以长期保留。

Span 一次步骤

读上下文、写耗时、版本、父子关系和尝试次数。适合回答“哪一步慢了、哪一步坏了”。

Event 局部证据

写短期细节，比如一次工具调用的红线字段、脱敏后的输入摘要或澄清原因。

Metric 趋势聚合

把计数、时长、比例压成可聚合的低基数数据，不承载原始文本。

History 恢复事实

记录 workflow / activity / replay 的事实，回答“这次是不是重放，不是再次执行”。

对象 读什么 写什么 最怕放错到哪里
Trace / Span 路径、耗时、版本、尝试、父子关系 可排查的步骤链 把高基数字段塞进 metric
Event 局部细节、短期证据、脱敏结果 短期可读的事实片段 把原文输入输出长期公开保存
Metric 计数、分布、比例、窗口趋势 低基数趋势 把 trace_id、用户输入、完整路径放进 label
Temporal History workflow_id、run_id、activity、attempt、replay 恢复事实和事件顺序 把 replay 当成新副作用

事件与 span

code

span 适合记录一次步骤本身，event 适合记录这一步里短期但很关键的细节。两者都能写字段，但写法不同：span 里的东西应该稳定、可聚合，event 里的东西可以更细，但保留期通常更短。

06 事件与 span 样例
{
  "trace_id": "trace_case_20260621_001",
  "span": "CallModel",
  "gen_ai.operation.name": "chat",
  "gen_ai.request.model": "gpt-4.1",
  "event": "tool.arguments.redacted",
  "tool.name": "send_invoice",
  "attempt": 2,
  "workflow_id": "claim-84",
  "run_id": "temporal-run-a7",
  "replay": false
}
span 里放什么

模型名、工具名、步骤名、版本、尝试次数、耗时、父子关系。它们是长期有用的排查字段，不靠原文内容才能看懂。

event 里放什么

一次工具调用的局部证据、脱敏后的参数、澄清原因、拒答原因、短期采样结果。它们帮助你回看“当时为什么这么做”。

指标与基数

math

指标回答趋势，不回答个案。为了让趋势真的能查，标签必须低基数。直觉上可以把它看成“标签空间一乘，数据量就爆了”。

$$ \text{series\_count} \approx \prod_{i=1}^{n} \left| \text{label}_i \right| $$

符号意思很简单：每个 label 的可取值越多，最终时序条目就越多。`tool.name` 和 `agent.version` 往往还能控制，`trace_id`、用户原文、完整路径通常会把序列数撑爆。

指标 用途 建议标签 不要放进标签的东西
tool_timeout_total 看工具超时是不是在上升 tool.name、error.type、agent.version trace_id、用户输入、完整路径
gen_ai.client.operation.duration 看模型调用和外部操作是否变慢 operation、model、agent.version 完整 prompt、工具返回全文
replay_count_total 看恢复和重放是不是变多 workflow_id、run_state 原始消息文本、完整事件 payload
manual_handoff_total 看人工接管是否在扩大 agent_version、tool.name 用户隐私、订单原文、模型思维链

指标的作用不是“记得越多越好”，而是让你在趋势图里看见变化。要是 label 选错了，后面所有告警和仪表盘都会跟着歪。

Temporal 对齐

workflow

Temporal 的 Event History 不是普通日志。普通日志只能告诉你“发生过什么”，Event History 还告诉你“发生的顺序、恢复的位置、哪一步是 replay 复用的结果”。这对智能体尤其重要，因为副作用必须被关在 Activity 边界里。

06 追踪字段和 replay 对齐
{
  "trace_id": "trace_case_20260621_001",
  "workflow_id": "claim-84",
  "run_id": "temporal-run-a7",
  "activity_type": "CallModel",
  "attempt": 2,
  "replay": false,
  "search_attributes": ["claim-84", "invoice", "escalation"],
  "event_history": "ActivityTaskStarted -> ActivityTaskCompleted"
}
哪些字段用来对齐

workflow_id 说明是哪条长期流程，run_id 说明是哪一次运行，attempt 说明这是第几次尝试，replay 说明是不是重放。

哪些字段只该做搜索

搜索属性适合放业务筛选条件，比如单号、流程类型、任务阶段。它们帮助你找回这次运行，但不承担诊断全部细节的责任。

[trace] workflow_id=claim-84 run_id=temporal-run-a7
[span]  CallModel attempt=1 replay=false
[event] tool.arguments.redacted=true
[metric] tool_timeout_total{tool.name="send_invoice"} += 0

读一条事故链

trace table

把一条追踪链读顺，最实用的方法不是先背字段，而是先问四个问题：这次是谁触发的、哪一步最慢、哪一步真的失败了、哪里出现了重复执行。只要这四个问题能答出来，trace 就不是一堆彩色方块，而是能顺着走的证据链。

下面用一个补发发票的例子串起来。用户说“把发票发到新邮箱”，系统先查供应商，再补写邮箱。如果 search_vendor 返回空结果，正确的系统行为不是继续猜，而是澄清身份；如果 update_invoice_email 被调用了两次，重点就不是“邮件发了几封”，而是“哪一层失守了，模型重复决策、工具重试还是 replay 误触发”。

观察点 该读什么 该忽略什么 错了会怎样
入口 span 用户目标、模型版本、当前步骤 和当前步骤无关的历史碎片 会把这次事故误看成老问题
工具 span 参数、attempt、error.type、耗时 原文的无关修辞 看不出是重试还是新决策
event 脱敏后的局部证据、拒绝原因、澄清原因 长期会爆量的原始全文 既不安全，也不好检索
history workflow / run / replay / activity 把 replay 当成再次执行的错觉 会把恢复行为误判成新副作用
06 事故链小样本
{
  "complaint": "发票重复发送",
  "trace": [
    "entry: user asked for resend",
    "span: search_vendor returned empty_result",
    "event: redact vendor lookup payload",
    "span: update_invoice_email attempt=2 replay=false",
    "history: ActivityTaskCompleted"
  ],
  "next_check": [
    "Was the empty result handled with clarification?",
    "Did retry reuse the same idempotency key?",
    "Did replay run a real side effect?"
  ]
}

如果你在 trace 里只能看见“失败了”，还不够。你需要知道失败在哪一步被放大，哪一步本来可以拦住，哪一步因为 replay 或重试把一次错误变成了两次。这个判断会直接影响 07 的回归样本写法，也会影响 08 的接管和回滚策略。

默认采什么，默认不采什么

policy

最稳妥的做法是先给出一个最小默认集，再给出一个按需开放集。默认集只保留排障和趋势足够的字段，按需开放集只在明确需要时打开，而且要有时间限制和访问控制。这样做不是保守，而是把“能查”和“能看”分开。

对智能体来说，最容易误开的内容有四类：输入全文、输出全文、工具参数全文和系统指令全文。它们都可能包含用户隐私、业务机密或安全边界信息。真正能长期保留的，往往是脱敏后的摘要、错误类型、步骤名、版本和少量业务键。你越早把这条线画清楚，后面的审计、合规和排障就越不打架。

默认集

trace_id、workflow_id、run_id、step name、attempt、error.type、duration、agent.version、tool.name。它们够你定位个案，也够你看趋势。

按需开放集

原文输入输出、工具参数全文、系统指令全文、RAG 片段全文。它们不是不能看，而是不能默认长期开、默认多人可见。

字段族 默认策略 需要时怎么开 回头看要注意什么
输入输出 摘要或脱敏摘要 限时、限人、按工单 查看后要回收权限
工具参数 保留关键字段和版本 必要时打开完整 payload 注意是否含密钥或隐私
系统指令 只保留引用或摘要 对故障样本按需查看 不要把它当普通日志
审计事实 单独保存 按审计角色访问 别和调试细节混在一起

这一层的目标很简单：让系统默认足够可查，但不默认过度暴露。等你把这条线立住，07 的回放和 08 的验收才不会在同一个问题上互相打架。

排障顺序

runbook

当一条事故已经进入排障阶段，最重要的不是继续加字段，而是按顺序读证据。先看 trace 里发生了什么，再看 metric 是否只是趋势抖动，最后去 event 和 retention 里确认哪些内容该保留、哪些内容该收缩。顺序一旦乱掉，你就会把“这次为什么失败”错看成“系统所有地方都坏了”。

一个实用的原则是：个案靠 trace，趋势靠 metric，恢复靠 history，风险说明靠 event。只要你记得这四层分工，很多争论就会自动收束到一个问题上，就是“这条信号现在应该落在哪一层解决”。

证据层 优先看什么 误读后果 建议动作
trace step、attempt、replay、error.type 把个案误判成大范围退化 先确认是哪一次任务出了问题
metric 失败率、p95、重试率、人工接管率 看不见长期趋势 判断是否需要限流、降级或回滚
event 局部证据、拒绝原因、澄清动作 把原文全文当作默认保留对象 改成摘要或脱敏摘要
history workflow / activity / replay 关系 把恢复行为误当成新副作用 重新核对边界和幂等策略
06 最小排障记录
{
  "signal": "duplicate invoice resend",
  "trace": ["entry", "tool attempt=2", "replay=false"],
  "decision": "check retry policy before changing business logic",
  "action": ["pause write path", "review boundary", "keep retention minimal"]
}

把排障顺序写成这样，后面的监控、合规和回归都会更顺：先定位一次事故，再看它是不是一类问题，最后再决定这个字段该不该长期可见。这个顺序正好会接到下一步的样本设计和上线验收上。

保留期与访问边界

policy

追踪越细，排障越快，但风险也越高。默认策略应该从最小必要信息开始，再决定哪些内容短期保留、哪些只留摘要、哪些进入专门的审计记录。

短期保留

原文输入输出、工具参数全文、系统提示词全文、敏感事件 payload。只在需要排查时开放，过期后删除或只留脱敏摘要。

中期保留

脱敏后的 span 属性、错误类型、耗时、版本、attempt、replay。它们够用来排查，但不会把原始内容长期摊开。

数据类型 默认保留 访问范围 判断标准
指标 较长 平台、SRE、开发负责人 只含趋势，不含原文
脱敏 span 属性 中等 排障相关角色 能定位问题，但不暴露原文
原文输入输出 短期或按需 显式授权、限时访问 属于敏感内容，不能默认全开
审计事实 按合规要求 审计和安全角色 和调试内容分开保存

案例

worked case

场景：理赔智能体要发送补充材料清单。用户投诉说收到了两封几乎一样的邮件。要判断是模型再次决定发信，工具重试发信，还是 worker 重启后 replay 误触发了副作用。

看 trace

如果第二次出现新的 tool call id，先怀疑 planner 又做了一次决定；如果 call id 相同但 attempt 变了，先看幂等键；如果 replay=true 但还真的发了邮件，就要查副作用是否越过 Activity 边界。

看 metric

如果 tool_timeout_total、replay_count_total、manual_handoff_total 一起抬头，就说明这不是单点 bug，而是恢复、重试和人工接管同时在变坏。

信号 先怀疑什么 下一步看什么
新 tool call id 模型再次决策 prompt、planner、工具提示词
同 call id 不同 attempt 重试策略或幂等键 timeout、retry policy、request id
replay=true 还真的执行副作用 Activity 边界失守 workflow / activity 拆分、历史记录

这类问题的排查顺序最好固定成一条短链：先看有没有真实的写操作，再看是不是同一条调用在重试，最后看 replay 是否只是在恢复历史而不是再次触发副作用。只要这三个顺序搞反，你就会在“邮件发了几封”的表面现象里绕很久，却抓不到真正应该修的边界。

先看什么 接着看什么 最后确认什么
外部写入有没有发生 同一 tool call 是否发生了 retry replay 期间是否只读历史
错误类型是什么 attempt 和 idempotency key 是否一致 Activity 边界是否隔离了副作用
哪个字段在暴涨 是 trace、metric 还是 event 是否需要把这个字段降级为摘要
06 事故排查最小顺序
{
  "first_check": "Was there a real side effect?",
  "second_check": "Did the same call retry with the same idempotency key?",
  "third_check": "Was replay read-only?",
  "if_unknown": [
    "pause the write path",
    "inspect workflow/activity boundary",
    "compare trace, event, and metric separately"
  ]
}

真正能落地的监控，不是把所有字段都放出来，而是把每一类问题放进对应的观测层：趋势看 metric，个案看 trace，恢复顺序看 history，风险说明看 event。你在设计指标时越早把这四层分开，后面越容易做权限、保留期和排障手册。

定位层 一条事故只需要少量稳定 ID

这一层回答“是哪一次”，而不是“所有细节是什么”。

诊断层 错误类型、attempt、持续时间

这一层回答“为什么慢、为什么失败、为什么重试”。

恢复层 replay、history、Activity 边界

这一层回答“这是恢复还是新的副作用”。

如果把这三个层次记住，你就会更自然地把事件语义写成“发生了什么、为什么发生、该不该长期保留”。这也是为什么指标设计不是统计题，而是边界题：它决定了你以后能不能用同一套数据，同时支持趋势分析、事故定位和合规访问。

常见误区

traps
只存聊天记录

聊天记录缺少 step、attempt、parent-child、replay 和版本信息，排查时你只会知道“说过话”，不知道“做过什么”。

把 trace_id 当 metric 标签

这会把时序数据撑爆。trace_id 应该留在追踪里，不应该成为趋势图的标签。

把原文输入输出默认长期保存

很多排障场景确实需要原文，但“需要排障”不等于“默认永久在线”。保留期和访问范围要先写清楚。

把 Event History 当日志备份

Event History 的价值在恢复顺序和 replay 对齐。它不是为了替代所有日志，也不是用来堆原始大文本的。

只看平均时延

平均值会把尾部问题藏起来。p95 / p99、失败率、重试率和人工接管率通常更能反映真实风险。

敏感字段默认全开

输入、输出、工具参数和系统提示词可能含隐私或业务机密。默认最小采集，再按需开放。
