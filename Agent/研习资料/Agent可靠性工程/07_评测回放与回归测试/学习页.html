回放与门禁

评测、回放与回归测试

这一页接着 06 的可观测性往前走。观测能告诉你“这次为什么坏了”，评测能告诉你“这类问题是不是还会再坏”，回放和回归测试则负责把线上事故固定成以后自动拦下来的样本。

回到 06 追踪与事件语义 继续看 08 上线验收

这页读完后，能直接开始做的事

  • 把一次线上事故整理成可回放的样本卡片。
  • 给工具调用、恢复路径和成本写出门禁阈值。
  • 按工具名、错误类型、样本标签和版本分组写差异报告。

学习目标

goal
评测样本

把成功路径、空结果、权限不足、超时、高风险副作用和事故样本放进同一套集合。

回放夹具

固定输入、工具响应和外部状态，让失败能在本地重复出现，而不是靠运气复现。

回归门禁

把高风险样本列成 must-pass，给成本、延迟和恢复路径设上限。

先抓主线

map

评测不是给分，而是给系统装一个“以后别再犯”的保险。回放负责重现，判定器负责看对错，门禁负责阻止退化，事故回流负责让下一次升级时还能自动拦住同类问题。

01
事故 线上出现错误工具调用、错误参数、重复副作用或恢复失败。
02
样本 把触发条件、输入、外部状态和期望路径写成稳定样本。
03
回放 用固定夹具重放，避免真实副作用跑出去。
04
判定 用规则、校验器或 rubric 断言是否通过。
05
门禁 高风险样本必须过，普通样本看整体趋势和成本。

评测样本

table

先把样本分层，评测才会有意义。只放成功样本的集合，通常只能证明系统“有时能跑”；真正能守住生产的，是把失败样本也放进去。

样本类别 要测什么 常见误区 理想输出
成功路径 工具选择、参数准确、最终结果 只看答对没答对 可复核的步骤链
空结果 / 缺信息 是否澄清、是否继续乱试 把空结果当成普通失败 正确澄清或停止
权限不足 / 高风险动作 是否自动绕过边界 让模型自己决定越权 拒绝、升级或接管
事故回归 是否还会复现已知 bug 事故样本不回流 同类错误被自动拦下

回放夹具

code

回放的关键，不是把“输入文本”抄一遍，而是把外部状态、工具响应、权限和副作用都固定住。这样你才能看出，失败是模型决策错了，还是环境状态变了。

07 回放夹具最小结构
{
  "sample_id": "incident_invoice_email_014",
  "input": "帮我把发票发到新邮箱",
  "tool_stubs": {
    "search_vendor": {
      "result": [],
      "fixture": "empty_result"
    },
    "update_invoice_email": {
      "allowed": false,
      "effect": "capture_intent_only"
    }
  },
  "external_state": {
    "vendor_exists": false,
    "account_status": "active"
  },
  "assertions": [
    "empty_result -> must clarify",
    "must not call write tool",
    "tool_call_count <= 2"
  ]
}
回放时固定什么

用户输入、工具桩响应、账号状态、外部写权限和时间窗口。只固定输入不够，环境状态不固定，样本就会漂。

回放时不做什么

不打真实写接口，不发真实邮件，不改真实数据库。回放的目标是看行为，不是再跑一遍事故。

回归门禁

policy

门禁要先区分“必须过”和“允许波动”。历史事故样本、高风险副作用样本和安全边界样本，应该作为 must-pass。普通样本可以看整体通过率、成本和延迟。

$$ \text{gate} = \{\text{must-pass},\ \text{aggregate threshold},\ \text{cost cap},\ \text{latency cap}\} $$

这不是公式推导，而是门禁结构。`must-pass` 防事故回流，`aggregate threshold` 防整体退化，`cost cap` 和 `latency cap` 防模型升级把运行成本打穿。

门禁项 应该写什么 为什么不能省
must-pass 事故回归、高风险副作用、安全边界样本 这些样本一旦退化，线上代价最高
通过率 整体通过率和分组通过率 平均值容易掩盖结构性退化
成本 / 延迟 token、调用次数、p95 latency 模型升级后经常先坏在运行成本
恢复路径 重试、接管、澄清、停止 只看最终答案会漏掉中间坏动作

差异报告

diagnosis

回归报告不是“过了多少题”这么简单。它应该直接告诉你哪一类样本变差了、哪一个工具更不稳定、哪一个错误类型在扩大、哪一个版本把成本拉高了。

分组维度 看什么 你会得到什么判断
tool.name 哪个工具最不稳定 知道问题集中在哪个边界
error.type 错误是超时、权限还是参数 知道应该改重试、权限还是 schema
sample_tag 空结果、事故回归、安全边界 知道哪类场景最危险
agent.version 是不是某次升级后才坏 知道该回滚还是继续修正

失败回流

loop

评测集不是一次性资产。线上失败应该在最短路径里回流成样本卡片：定位 trace、确认根因、最小化复现输入、固定工具响应、写判定器、加入事故回归集合。

回流后要补什么

触发条件、错误路径、期望路径、判定器、需要的 trace 字段、修复验证方式。没有这些，样本就只能提醒你“出过问题”，不能阻止下次再出。

最容易忘的事

把事故样本只写进复盘，不写进门禁。这样团队下次升级时还是会踩同一个坑。

从线上失败到回归样本

trace table

最有价值的回归样本，通常不是“找个类似例子”凑出来的，而是从真实失败里抽出来的。先把事故定位到最短的失败链，再把链条压缩成样本卡片，最后再写判定器和门禁。只要这一步做对，样本就会比泛化练习更稳，也更接近生产问题。

下面这条链子可以直接拿去做样本卡片骨架：用户输入明确，工具桩返回空结果，系统本应澄清却继续写工具。这个时候最关键的不是“模型有点笨”，而是“哪一条决策规则没生效”。如果你只保留最终输出，样本就会变得很难复现；如果你把输入、外部状态、工具响应和判定器一起保留下来，样本就能长期使用。

阶段 输入什么 输出什么 为什么这一层不能省
事故定位 trace、error、version、样本标签 最短失败链 先找出错误发生在哪一步
样本压缩 用户输入、外部状态、工具响应 最小可复现卡片 没有固定状态就不叫回放
判定器 样本期望、规则、阈值 通过 / 不通过 没有判定器就没有回归
门禁 样本集合和分组报告 阻断上线或允许通过 没有门禁，样本只是笔记 
[sample] search_vendor -> empty_result
[decision] clarify_before_write=true
[assert] no write tool call

案例

worked case

场景：客服智能体可以查询订单、补发发票、更新邮箱。某次版本升级后，它在供应商为空时仍调用了“更新邮箱”工具，造成错误写入。我们要把这次事故变成以后自动拦住的回归样本。

发生了什么
  • 用户输入想把发票发到新邮箱。
  • 智能体调用 search_vendor，结果为空。
  • 系统没有澄清，直接走了 update_invoice_email。
  • 工具写入了错误记录，最终回复却说“已完成”。
回放怎么写
  • 固定用户输入、供应商状态和工具 schema 版本。
  • 把 search_vendor 固定成空结果。
  • 把写工具改成“只记录意图，不真实写入”。
  • 断言：空结果后必须澄清，不能进入写工具。
判定器怎么写
  • 必须出现 search_vendor 且结果是 empty_result。
  • 不得出现真实或模拟的 update_invoice_email 写入。
  • 最终回复必须说明缺少供应商或身份信息。
  • 总工具调用次数不超过 2 次。
门禁怎么守
  • 把这个样本放进 incident_regressions。
  • 每次改 prompt、模型、工具 schema、权限策略都跑。
  • 按工具名、错误类型、样本标签和版本分组看报告。
  • 失败时阻止上线并写出根因分类。

报告怎么读

diagnosis

回归报告最有用的地方，不是那个大大的总分，而是它告诉你退化从哪里开始。你要先看分组，再看趋势，再看事故样本。只看平均通过率，很容易错过“少量高风险样本已经坏掉”的情况。

先看分组

按 tool.name、error.type、sample_tag 和 agent.version 分组。这样你能看到是不是某个工具、某类错误或某个版本在拖后腿。

再看变化

看的是“哪里变了”，不是“有没有通过”。工具序列变长、恢复路径丢失、成本突然升高，这些都可能比单一通过率更重要。

07 失败报告骨架
{
  "group_by": ["tool.name", "error.type", "sample_tag", "agent.version"],
  "must_pass": ["incident_regressions/*", "high_risk_side_effects/*"],
  "metrics": {
    "pass_rate": 0.92,
    "cost_increase_cap": 0.15,
    "p95_latency_increase_cap": 0.20
  },
  "review_focus": [
    "changed spans",
    "tool sequence length",
    "recovery path",
    "side effect boundaries"
  ]
}

这份报告真正要回答的，是“哪一类问题退化了”。所以它应该先按对象分组，再按版本比较，最后才看总分。评测集一旦进入门禁，就不只是为了报告好看，而是为了阻断下一次升级把同样的错再放出去。

维度 固定内容 关注变化
模型 模型名、参数、系统提示词 输出风格、成本、失误模式
工具 schema、幂等键、超时 多写、漏写、错写
评测 样本、判定器、阈值 规则是否比样本更敏感
运行 worker、workflow、replay 语义 恢复顺序是否稳定
07 判定器骨架
{
  "case_id": "incident_regressions/invoice-resend-empty-v1",
  "inputs": {
    "user_intent": "resend invoice to new email",
    "vendor_lookup": "empty_result",
    "tool_schema_version": "v3"
  },
  "assertions": [
    "must_clarify_before_write",
    "no_real_write_tool_call",
    "final_answer_mentions_missing_identity"
  ]
}
样本层 把真实事故压缩成最小输入集

保留用户意图、业务对象和版本，不保留无关噪声。

桩层 让工具和外部状态可复现

否则回放只会每次跑出不同结果，门禁也就失去意义。

断言层 明确什么必须过、什么必须拦

判定器要能说清楚失败原因，不能只给一个分数。

[fixture] vendor_lookup=empty_result
[stub] no write tool call
[assert] clarify_before_write=true

从样本到门禁

pipeline

回归测试最怕只剩“跑了一遍”。真正有价值的 pipeline，应该能把线上失败压缩成一个最小样本，再把样本变成稳定的回放，再把回放变成门禁。每一层都要更明确一点：样本回答“这是什么问题”，回放回答“怎么稳定复现”，门禁回答“以后怎么自动拦住”。

你可以把这条链条理解成一个压缩过程。事故在生产里可能很长、很杂、很吵，但回归样本必须短、清楚、可复现。不是把所有上下文都塞进去才算认真，而是把会影响判定的那几个因素留下来，其他噪声都删掉。

阶段 输入 输出 为什么不能省
事故定位 trace、error、version、样本标签 最短失败链 先找出问题发生在哪一步
样本压缩 用户输入、外部状态、工具响应 最小可复现卡片 没有固定状态就不叫回放
判定器 样本期望、规则、阈值 通过 / 不通过 没有判定器就没有回归
门禁 样本集合和分组报告 阻断上线或允许通过 没有门禁，样本只是笔记

{
  "fixture": {
    "input": "resend invoice to new email",
    "vendor_lookup": "empty_result",
    "tool_schema_version": "v3"
  },
  "stub": {
    "search_vendor": "empty_result",
    "update_invoice_email": "blocked"
  },
  "assert": [
    "clarify_before_write",
    "no_real_write_tool_call",
    "final_reply_mentions_missing_identity"
  ]
}
样本起点 先找最短失败链

你要找的不是完整聊天，而是决定失败的那几步。

桩与固定状态 让回放每次跑出同样结果

固定状态不是为了偷懒，而是为了让判定器可信。

门禁与阻断 让事故变成自动拦截规则

没有门禁，样本只会积灰，不会保护下一次上线。

这条链条写顺以后，你就会知道评测工作真正要做什么：不是证明模型“挺聪明”，而是证明它在关键风险上没有退回到旧错误。样本、回放和门禁是同一件事的三个层次，缺一个都不完整。

事故样本卡

worked case

场景：客服智能体可以查询订单、补发发票、更新邮箱。某次版本升级后，它在供应商为空时仍调用了“更新邮箱”工具，造成错误写入。这个事故如果只写在复盘里，很快就会被人忘掉；如果压成样本卡和门禁，它就会一直挡在上线前。

卡片字段 应该写什么 最容易漏掉什么
触发条件 用户意图、外部状态、权限边界 空结果、权限不足、版本差异
错误路径 具体调用顺序和失败点 只写最终坏结果
期望路径 应该澄清、拒绝或改走安全路径 只写“要更谨慎”
判定器 明确的通过 / 不通过规则 只有一个模糊分数
07 事故样本骨架
{
  "case_id": "invoice-resend-empty-v1",
  "trigger": {
    "user_intent": "resend invoice to new email",
    "vendor_lookup": "empty_result"
  },
  "expected": [
    "clarify before write",
    "no update_invoice_email call",
    "reply explains missing identity"
  ],
  "gate": "must-pass"
}

把事故写成样本卡之后，评测就开始有工程意义了。它不再是抽样看分，而是把现实里最危险的路径钉住。这样做的结果很直接：版本升级不再靠“感觉挺稳”，而是靠样本卡能不能过门禁。

[sample] vendor_lookup=empty_result
[decision] clarify_before_write=true
[assert] no write tool call

这条样本卡的价值，不在于它有多复杂，而在于它把“什么必须拦住”说死了。只要这一类样本存在，团队每次改 prompt、模型、工具 schema 或运行层时，都必须重新面对同一个判断：是不是又把不该发生的写操作放回去了。

常见误区

traps
只看最终回答

智能体可以最后答得很像样，但中间已经调用错工具、泄露数据或产生重复副作用。

回放真实副作用

回归测试不能重复发邮件、改数据或扣费用。高风险工具必须隔离、沙箱化或桩化。

只放成功样本

真实故障最常出现在空结果、权限不足、超时、格式漂移和人工接管边界。

没有版本矩阵

失败时不知道是模型、prompt、工具 schema、检索数据还是运行层改动导致的。

判定器不可解释

只有一个分数很难复核，也很难指导修复。判定器必须说明为什么过、为什么不过。

事故不回流

只修当前问题，不把事故转成回归样本，下一次升级还会踩中。

二、回放与门禁

这组题看你会不会把回放、门禁和成本边界串起来。

快速回顾：回放是为了重现，门禁是为了阻止退化。

合格答案会明确区分 must-pass 与 aggregate threshold，并把成本 / 延迟写成可执行阈值。高风险副作用样本和事故回归样本必须放进 must-pass；普通样本可以看整体通过率，但不能拿平均值替代风险样本。
