08 上线前验收与运维边界｜智能体可靠性工程

前面几页已经把可观测性、回放和回归门禁搭好了，这一页要回答最后一个问题：系统可以上线了吗，出了事谁来停，停到什么程度，怎么回滚，谁来接管。上线不是一个开关，而是一组责任边界。

读完本页应该能做什么

  • 写出一份能给团队评审的上线验收清单。
  • 说明哪些情况必须暂停自动执行，哪些情况必须人工接管。
  • 把回滚、告警、版本和职责边界写成能落地的操作项。

学习目标

goal

验收

上线前要有证据，不只是演示。证据要覆盖恢复、追踪、成本、回滚和接管。

接管

当自动化不该继续时，系统要知道该停在哪里、谁来接、怎么通知。

先抓主线

map

如果 06 解决的是“怎么看”，07 解决的是“怎么回放、怎么拦下”，那 08 解决的就是“什么时候可以进生产、出了问题谁负责停和接”。这一页不再讲语义本身，而是把语义转成操作清单。

规定模型能做什么，不能做什么。

人工接管

什么情况不该继续自动化。

验收清单

table

一份合格的验收清单，应该把“运行、观测、变更、处置”放在同一张表里。只看模型效果，不看恢复和接管，评审会很快失真。

运维边界

policy

运维边界不是一句“加人工审核”就结束了。它要明确四件事：什么动作默认可以自动做，什么动作必须确认，什么情况必须暂停，什么情况必须通知。

必须确认或暂停

退款、发信、改数据、删数据、访问敏感内容、跨边界调用，或者任何超过成本阈值的动作。

动作类型默认策略谁来决定
可逆低风险动作可自动执行，但要可追踪系统
高风险写入动作必须确认或双重检查人和系统共同决定
越权或敏感动作默认拒绝权限和安全边界
成本超阈值动作暂停或降级运行门禁

回滚与暂停

workflow

回滚上线前就要写进边界的选项。要提前说明：哪种异常先暂停自动执行，哪种异常直接回滚上一版，哪种异常只降级不回滚。

代码
{
 "pause_if": [
 "replay_count_total rises sharply",
 "manual_handoff_total exceeds threshold",
 "tool_timeout_total spikes on a write tool",
 "sensitive_data_in_trace is detected"
 ],
 "rollback_if": [
 "must-pass incident regression fails",
 "high-risk side effect sample regresses",
 "cost or latency cap is exceeded after upgrade"
 ],
 "degrade_if": [
 "non-critical query path is slow",
 "an auxiliary retrieval path is unstable"
 ]
}

先回滚什么

先回滚把事故引入的版本层：模型、prompt、工具 schema、Workflow / Worker 版本，别只盯着最后的回答文本。

人工接管

handoff

人工接管的边界要比“出了事找人”具体得多。必须写清楚：什么信号会触发接管，谁接，接到什么状态，系统要不要暂停，接管后怎么恢复自动化。

接管后要补什么

上下文、最近步骤、工具尝试、已知限制、用户可见解释。接管把人需要的上下文补齐。

验收证据

evidence

验收记录要回答的是“这不是演示吗？”这个质疑。证据应该覆盖崩溃恢复、重复副作用、追踪定位、回归门禁和回滚演练。

所以验收证据最好写成一条最短闭环：先识别异常信号，再选处置动作，最后留下可复核的记录。这样你在上线前就已经把运行、回滚和接管的顺序说死了，真正出事时不会临场讨论太久。

代码
{
 "signal": "must-pass sample failed",
 "context": ["trace_id", "workflow_id", "version", "owner"],
 "action_order": ["pause", "investigate", "rollback_or_handoff"],
 "evidence": ["report", "trace", "notification"]
}

处置跑道

runbook

处置跑道上线前就要写好的执行顺序。它要把“看到什么信号”“先做什么动作”“由谁负责”“留下什么证据”说完整。这样一来，值班同学拿到告警时不会先问“要不要继续跑”，而是直接按跑道执行。

信号第一动作负责人留什么证据
门禁失败暂停发布值班 / 负责人评测报告、失败样本、版本号
重复副作用冻结写路径运维 / 开发trace、幂等键、重试记录
敏感数据进入追踪收缩访问范围安全 / 平台访问审计、脱敏记录、回收结果
成本或延迟越线降级或回滚业务 owner阈值、影响范围、处置时间
代码
{
 "pause": "stop high-risk automation first",
 "rollback": "revert the version that introduced the regression",
 "handoff": "attach trace_id, workflow_id, last step, and risk notes",
 "evidence": ["alert", "trace", "report", "rollback record"]
}

把这份跑道固定下来以后，验收就不再只是“有没有测试过”，而会变成“出了事时系统能不能真的停、真的退、真的交接”。这才是运维边界要解决的问题。

案例

worked

场景：一个理赔智能体已经接好了恢复、追踪和回放，但准备上线时，团队还是担心三个问题：会不会重复发信，会不会在高风险动作上越权，会不会出了事故没人知道该停哪一层。

写工具是否都被关进 Activity 边界。

每个高风险动作是否有幂等键和接管条件。

trace 里是否能区分 replay 和真实执行。

是暂停自动化，还是直接回滚版本。

是模型、prompt、工具 schema，还是 Worker 版本问题。

谁先接管，谁负责通知，谁记录复盘。

常见误区

traps

只看演示成功

演示成功不等于线上可承受。没有回滚、接管和门禁，演示再顺也只是一次性结果。

告警没有处置脚本

告警如果只说“失败率高”，值班人员还是不知道先停哪、查哪、回滚哪。

Worker、Workflow、工具版本混在一起

一旦恢复行为变了，不知道该回滚哪一层，故障就会拖长。

7_handoff_triggerchoice

Q: 哪一种情况更应该人工接管？

参考答案 / 评分要点

权限不足、敏感数据或高风险写入。越权、敏感和高风险写入通常应该接管，而不是继续自动化。

8_pause_rulesynthesis

Q: 写一条暂停自动化的规则

参考答案 / 评分要点

。合格答案要明确：什么信号触发暂停、通知谁、先停哪个自动化路径、谁判断是否回滚或接管、恢复条件是什么。只有“加告警”不够，因为告警不等于处置。

9_checklistdebug-checklist

Q: 从下面选出验收证据

。前四个应该勾选。供应商海报不是验收证据。验收要看恢复、拦截、门禁和回滚是否真的演练过，而不是看宣传材料好不好看。

10_cost_capfill

Q: 写出一个上线前必须检查的成本或延迟阈值

参考答案 / 评分要点

。可以写 p95 latency、token usage、工具调用次数、重试次数或外部写请求次数的阈值。关键是它要能直接阻断上线，不是挂在文档里看一眼就算。

11_boundary_notesynthesis

Q: 写三句话说明为什么边界必须先于模型优化

参考答案 / 评分要点

。边界先于模型优化，因为上线最先遇到的是权限、恢复、重复副作用和回滚问题，不是回答风格问题。即使模型更聪明，只要边界不稳，系统还是会在高风险动作上出事。比如发信、退款、删数据这类动作，系统层不先定规则，模型再好也不能直接放行。
