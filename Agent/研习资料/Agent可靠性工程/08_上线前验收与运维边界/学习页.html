验收与边界

上线前验收与运维边界

前面几页已经把可观测性、回放和回归门禁搭好了，这一页要回答最后一个问题：系统可以上线了吗，出了事谁来停，停到什么程度，怎么回滚，谁来接管。上线不是一个开关，而是一组责任边界。

回到 07 评测与回归 回到 06 追踪与事件语义

读完后应该能直接做的事

  • 写出一份能给团队评审的上线验收清单。
  • 说明哪些情况必须暂停自动执行，哪些情况必须人工接管。
  • 把回滚、告警、版本和职责边界写成能落地的操作项。

学习目标

goal
验收

上线前要有证据，不只是演示。证据要覆盖恢复、追踪、成本、回滚和接管。

边界

模型可以建议动作，但权限、风险和副作用边界必须由系统层决定。

接管

当自动化不该继续时，系统要知道该停在哪里、谁来接、怎么通知。

map

如果 06 解决的是“怎么看”，07 解决的是“怎么回放、怎么拦下”，那 08 解决的就是“什么时候可以进生产、出了问题谁负责停和接”。这一页不再讲语义本身，而是把语义转成操作清单。

验收清单

table

一份合格的验收清单，应该把“运行、观测、变更、处置”放在同一张表里。只看模型效果，不看恢复和接管，评审会很快失真。

检查项 要写什么 没有它会怎样
运行范围 任务类型、可调用工具、可产生的副作用 团队不知道系统到底能做什么
恢复与幂等 任务身份、恢复点、幂等键、崩溃演练 一旦恢复就可能重复副作用
追踪与指标 Trace 关联字段、敏感数据处理、告警阈值 出了问题看不见，或者看见太多敏感内容
变更与回滚 模型、提示词、工具、Workflow / Worker 版本 版本升级后不知道该回滚哪一层

运维边界

policy

运维边界不是一句“加人工审核”就结束了。它要明确四件事：什么动作默认可以自动做，什么动作必须确认，什么情况必须暂停，什么情况必须通知。

默认可以自动做

低风险查询、低风险建议、可逆且低成本的动作，前提是已经过恢复、追踪和回归门禁。

必须确认或暂停

退款、发信、改数据、删数据、访问敏感内容、跨边界调用，或者任何超过成本阈值的动作。

动作类型 默认策略 谁来决定
可逆低风险动作 可自动执行，但要可追踪 系统
高风险写入动作 必须确认或双重检查 人和系统共同决定
越权或敏感动作 默认拒绝 权限和安全边界
成本超阈值动作 暂停或降级 运行门禁

回滚与暂停

workflow

回滚不是失败之后临时想出来的动作，而是上线前就要写进边界的选项。要提前说明：哪种异常先暂停自动执行，哪种异常直接回滚上一版，哪种异常只降级不回滚。

08 最小回滚规则
{
  "pause_if": [
    "replay_count_total rises sharply",
    "manual_handoff_total exceeds threshold",
    "tool_timeout_total spikes on a write tool",
    "sensitive_data_in_trace is detected"
  ],
  "rollback_if": [
    "must-pass incident regression fails",
    "high-risk side effect sample regresses",
    "cost or latency cap is exceeded after upgrade"
  ],
  "degrade_if": [
    "non-critical query path is slow",
    "an auxiliary retrieval path is unstable"
  ]
}
先暂停什么

先停高风险自动动作，再停低风险辅助链路。这样能先止血，再决定是否回滚。

先回滚什么

先回滚把事故引入的版本层：模型、prompt、工具 schema、Workflow / Worker 版本，别只盯着最后的回答文本。

人工接管

handoff

人工接管的边界要比“出了事找人”具体得多。必须写清楚：什么信号会触发接管，谁接，接到什么状态，系统要不要暂停，接管后怎么恢复自动化。

[check] task_type=query
[check] tool_risk=low
[decision] auto_execute=true
[evidence] trace_id, workflow_id, version recorded
接管触发信号

权限不足、敏感数据、重复副作用、成本激增、门禁失败、连续超时，都是典型接管信号。

接管后要补什么

上下文、最近步骤、工具尝试、已知限制、用户可见解释。接管不是把问题丢给人，而是把人需要的上下文补齐。

验收证据

evidence

验收记录要回答的是“这不是演示吗？”这个质疑。证据应该覆盖崩溃恢复、重复副作用、追踪定位、回归门禁和回滚演练。

证据 最好是什么形式 说明什么
崩溃恢复演练 日志、截图、trace 流程能从中断点恢复
重复副作用拦截 回放记录、判定器 恢复不会重复写入
门禁结果 评测报告、分组差异 事故样本没有回退
回滚演练 操作步骤、责任人、通知链 线上出事知道怎么退

证据不是摆设，它的作用是让事故发生时的动作有依据。你如果只写“已经验收”，现场还是会回到经验主义：到底先停自动化还是先回滚版本，到底是模型问题还是边界问题，谁也说不清。

所以验收证据最好写成一条最短闭环：先识别异常信号，再选处置动作，最后留下可复核的记录。这样你在上线前就已经把运行、回滚和接管的顺序说死了，真正出事时不会临场讨论太久。

异常识别 replay 暴增、门禁失败、敏感内容泄露、成本越线

先把信号分清楚，再决定是不是要动生产。

处置动作 暂停、回滚、降级、接管

动作必须能直接执行，不能只写在总结里。

复核证据 trace、report、通知链、回滚记录

证据要支持复盘，也要支持以后追责和恢复。

08 处置前的信息包
{
  "signal": "must-pass sample failed",
  "context": ["trace_id", "workflow_id", "version", "owner"],
  "action_order": ["pause", "investigate", "rollback_or_handoff"],
  "evidence": ["report", "trace", "notification"]
}

处置跑道

runbook

处置跑道不是事故后补出来的文档，而是上线前就要写好的执行顺序。它要把“看到什么信号”“先做什么动作”“由谁负责”“留下什么证据”说完整。这样一来，值班同学拿到告警时不会先问“要不要继续跑”，而是直接按跑道执行。

最常见的误区有两个。一个是把所有异常都当成同一种问题，结果小抖动也把整条自动化停掉；另一个是把高风险异常当普通延迟，结果副作用已经发生了才开始补救。真正可用的跑道，必须先区分信号，再区分动作，最后区分交接。

信号 第一动作 负责人 留什么证据
门禁失败 暂停发布 值班 / 负责人 评测报告、失败样本、版本号
重复副作用 冻结写路径 运维 / 开发 trace、幂等键、重试记录
敏感数据进入追踪 收缩访问范围 安全 / 平台 访问审计、脱敏记录、回收结果
成本或延迟越线 降级或回滚 业务 owner 阈值、影响范围、处置时间
08 最小 runbook
{
  "pause": "stop high-risk automation first",
  "rollback": "revert the version that introduced the regression",
  "handoff": "attach trace_id, workflow_id, last step, and risk notes",
  "evidence": ["alert", "trace", "report", "rollback record"]
}
告警不是结论 告警只是“需要看一下”的起点

必须结合 trace、门禁和成本阈值，才能决定下一步动作。

结论要能执行 暂停、回滚、降级、接管

结论如果不能落地成按钮、脚本或通知，就是不合格处置。

交接要带上下文 最近步骤、工具尝试、已知限制

人接手时最缺的就是背景，不补背景就等于没接管。

把这份跑道固定下来以后，验收就不再只是“有没有测试过”，而会变成“出了事时系统能不能真的停、真的退、真的交接”。这才是运维边界要解决的问题。

案例

worked case

场景：一个理赔智能体已经接好了恢复、追踪和回放，但准备上线时，团队还是担心三个问题：会不会重复发信，会不会在高风险动作上越权，会不会出了事故没人知道该停哪一层。

上线前要问的事
  • 写工具是否都被关进 Activity 边界。
  • 每个高风险动作是否有幂等键和接管条件。
  • trace 里是否能区分 replay 和真实执行。
出事时要问的事
  • 是暂停自动化，还是直接回滚版本。
  • 是模型、prompt、工具 schema，还是 Worker 版本问题。
  • 谁先接管，谁负责通知，谁记录复盘。

常见误区

traps
只看演示成功

演示成功不等于线上可承受。没有回滚、接管和门禁，演示再顺也只是一次性结果。

权限边界交给模型

模型可以建议动作，但不能当权限系统。能不能做，必须由系统层决定。

告警没有处置脚本

告警如果只说“失败率高”，值班人员还是不知道先停哪、查哪、回滚哪。

没有明确下线条件

成本、失败率、重复副作用和敏感数据异常阈值要先写，不然事故一大就没人敢停。

Worker、Workflow、工具版本混在一起

一旦恢复行为变了，不知道该回滚哪一层，故障就会拖长。

把验收写成口号

“加日志、加告警、人工审核”不是验收。要写对象、触发条件、证据和责任人。

Workflow 负责长期流程和状态；Activity 负责外部动作和副作用；上线验收时分别看恢复语义和副作用安全。只说“一个管流程一个管工具”还不够，要补上验收视角。

synthesisboundary

写三句话说明为什么边界必须先于模型优化

先写直觉，再写对上线的影响，最后写一个具体场景。

答题目标：说清楚为什么“更聪明”不等于“更可上线”。

参考答案 / 评分要点

边界先于模型优化，因为上线最先遇到的是权限、恢复、重复副作用和回滚问题，不是回答风格问题。即使模型更聪明，只要边界不稳，系统还是会在高风险动作上出事。比如发信、退款、删数据这类动作，系统层不先定规则，模型再好也不能直接放行。
