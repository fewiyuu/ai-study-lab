# 学习目标 {#goals}
<!-- tag: goal -->

**验收**
上线前要有证据，不只是演示。证据要覆盖恢复、追踪、成本、回滚和接管。


**边界**
模型可以建议动作，但权限、风险和副作用边界必须由系统层决定。


**接管**
当自动化不该继续时，系统要知道该停在哪里、谁来接、怎么通知。

<!-- tag: table -->

一份合格的验收清单，应该把“运行、观测、变更、处置”放在同一张表里。只看模型效果，不看恢复和接管，评审会很快失真。

检查项

运行范围

追踪与指标
Trace 关联字段、敏感数据处理、告警阈值

变更与回滚

# 运维边界 {#boundary}
<!-- tag: policy -->

运维边界不是一句“加人工审核”就结束了。它要明确四件事：什么动作默认可以自动做，什么动作必须确认，什么情况必须暂停，什么情况必须通知。

## box soft
默认可以自动做
低风险查询、低风险建议、可逆且低成本的动作，前提是已经过恢复、追踪和回归门禁。

## box warn
必须确认或暂停
退款、发信、改数据、删数据、访问敏感内容、跨边界调用，或者任何超过成本阈值的动作。

动作类型
默认策略
谁来决定


可逆低风险动作
可自动执行，但要可追踪
系统


高风险写入动作
必须确认或双重检查
人和系统共同决定


越权或敏感动作
默认拒绝
权限和安全边界


成本超阈值动作
暂停或降级
运行门禁

# 回滚与暂停 {#rollback}
<!-- tag: workflow -->

回滚上线前就要写进边界的选项。要提前说明：哪种异常先暂停自动执行，哪种异常直接回滚上一版，哪种异常只降级不回滚。

08 最小回滚规则复制

```json
{
 "pause_if": [
 "replay_count_total rises sharply",
 "manual_handoff_total exceeds threshold",
 "tool_timeout_total spikes on a write tool",
 "sensitive_data_in_trace is detected"
 ],
 "rollback_if": [
 "must-pass incident regression fails",
 "high-risk side effect sample regresses",
 "cost or latency cap is exceeded after upgrade"
 ],
 "degrade_if": [
 "non-critical query path is slow",
 "an auxiliary retrieval path is unstable"
 ]
}
```

## box info
先暂停什么
先停高风险自动动作，再停低风险辅助链路。这样能先止血，再决定是否回滚。

## box soft
先回滚什么
先回滚把事故引入的版本层：模型、prompt、工具 schema、Workflow / Worker 版本，别只盯着最后的回答文本。

# 人工接管 {#handoff}
<!-- tag: handoff -->

人工接管的边界要比“出了事找人”具体得多。必须写清楚：什么信号会触发接管，谁接，接到什么状态，系统要不要暂停，接管后怎么恢复自动化。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

## box warn
接管触发信号
权限不足、敏感数据、重复副作用、成本激增、门禁失败、连续超时，都是典型接管信号。

## box
接管后要补什么
上下文、最近步骤、工具尝试、已知限制、用户可见解释。接管把人需要的上下文补齐。

# 验收证据 {#evidence}
<!-- tag: evidence -->

验收记录要回答的是“这不是演示吗？”这个质疑。证据应该覆盖崩溃恢复、重复副作用、追踪定位、回归门禁和回滚演练。

证据
最好是什么形式
说明什么


崩溃恢复演练
日志、截图、trace
流程能从中断点恢复


重复副作用拦截
回放记录、判定器
恢复不会重复写入


门禁结果
评测报告、分组差异
事故样本没有回退


回滚演练
操作步骤、责任人、通知链
线上出事知道怎么退

证据不是摆设，它的作用是让事故发生时的动作有依据。你如果只写“已经验收”，现场还是会回到经验主义：到底先停自动化还是先回滚版本，到底是模型问题还是边界问题，谁也说不清。

所以验收证据最好写成一条最短闭环：先识别异常信号，再选处置动作，最后留下可复核的记录。这样你在上线前就已经把运行、回滚和接管的顺序说死了，真正出事时不会临场讨论太久。

}

# 处置跑道 {#runbook}
<!-- tag: runbook -->

处置跑道上线前就要写好的执行顺序。它要把“看到什么信号”“先做什么动作”“由谁负责”“留下什么证据”说完整。这样一来，值班同学拿到告警时不会先问“要不要继续跑”，而是直接按跑道执行。

最常见的误区有两个。一个是把所有异常都当成同一种问题，结果小抖动也把整条自动化停掉；另一个是把高风险异常当普通延迟，结果副作用已经发生了才开始补救。真正可用的跑道，必须先区分信号，再区分动作，最后区分交接。

信号
第一动作
负责人
留什么证据


门禁失败
暂停发布
值班 / 负责人
评测报告、失败样本、版本号


重复副作用
冻结写路径
运维 / 开发
trace、幂等键、重试记录


敏感数据进入追踪
收缩访问范围
安全 / 平台
访问审计、脱敏记录、回收结果


成本或延迟越线
降级或回滚
业务 owner
阈值、影响范围、处置时间

08 最小 runbook复制

```json
{
 "pause": "stop high-risk automation first",
 "rollback": "revert the version that introduced the regression",
 "handoff": "attach trace_id, workflow_id, last step, and risk notes",
 "evidence": ["alert", "trace", "report", "rollback record"]
}
```

<!-- interactive: ['shape-tracer'] → keep as HTML -->

把这份跑道固定下来以后，验收就不再只是“有没有测试过”，而会变成“出了事时系统能不能真的停、真的退、真的交接”。这才是运维边界要解决的问题。

# 案例 {#case}
<!-- tag: worked case -->

场景：一个理赔智能体已经接好了恢复、追踪和回放，但准备上线时，团队还是担心三个问题：会不会重复发信，会不会在高风险动作上越权，会不会出了事故没人知道该停哪一层。

**上线前要问的事**

写工具是否都被关进 Activity 边界。
每个高风险动作是否有幂等键和接管条件。
trace 里是否能区分 replay 和真实执行。


**出事时要问的事**

是暂停自动化，还是直接回滚版本。
是模型、prompt、工具 schema，还是 Worker 版本问题。
谁先接管，谁负责通知，谁记录复盘。

# 常见误区 {#mistakes}
<!-- tag: traps -->
<!-- layout: grid2 -->

**只看演示成功**
演示成功不等于线上可承受。没有回滚、接管和门禁，演示再顺也只是一次性结果。

**权限边界交给模型**
模型可以建议动作，但不能当权限系统。能不能做，必须由系统层决定。

**告警没有处置脚本**
告警如果只说“失败率高”，值班人员还是不知道先停哪、查哪、回滚哪。

**没有明确下线条件**
成本、失败率、重复副作用和敏感数据异常阈值要先写，不然事故一大就没人敢停。

**Worker、Workflow、工具版本混在一起**
一旦恢复行为变了，不知道该回滚哪一层，故障就会拖长。

**把验收写成口号**
“加日志、加告警、人工审核”不是验收。要写对象、触发条件、证据和责任人。

# 练习 {#practice}
<!-- tag: diagnostic -->

练习要让你真正能写出上线前的操作单，不是只会说原则。

<!-- practice-group: 一、验收清单 -->
<!-- quick-review: 快速回顾：验收要有证据，不是有感觉。 -->

### 1_acceptance choice

**Q:** 哪一项最像验收证据？

- [ ] 一次演示视频
- [ ] 崩溃恢复、重复副作用、追踪定位和回滚演练记录
- [ ] 模型供应商宣传页

Answer: 崩溃恢复、重复副作用、追踪定位和回滚演练记录
Explain: 演示截图不够，验收要看恢复、追踪、回滚和门禁。

### 2_trace_vs_chat true/false

**Q:** Trace 等同于聊天记录吗？

- [ ] 等同
- [ ] 不等同

Answer: 不等同
Explain: Trace 应有步骤、耗时、错误和版本，不等于聊天记录。

### 3_boundary_rule fill

**Q:** 写一个必须在工具层完成的控制

Explain: 合格答案可以是权限检查、参数校验、幂等键、超时控制、写入限额等。关键是它必须在工具层发生，而不是把责任推回模型。

### 4_operational_note synthesis

**Q:** 写三句话解释 Workflow 和 Activity 的区别

Explain: Workflow 负责长期流程和状态；Activity 负责外部动作和副作用；上线验收时分别看恢复语义和副作用安全。只说“一个管流程一个管工具”还不够，要补上验收视角。

<!-- end-practice-group -->

<!-- practice-group: 二、运维边界 -->
<!-- quick-review: 快速回顾：边界要能停、能退、能接。 -->

### 5_rollout_plan synthesis

**Q:** 设计一个任务身份和追踪关联方案

Explain: 合格答案至少要把业务对象和运行对象一起写出来：用户 / 业务 ID 用来定位对象， trace_id 和 workflow_id 用来找运行记录，版本字段用来说明是哪一版行为。只写单一 ID 不够。

### 6_failures debug-checklist

**Q:** 写出三条上线前必须演练的失败路径

- [ ] 工具调用后崩溃
- [ ] 外部服务超时
- [ ] 结构化输出解析失败
- [ ] 敏感数据进入追踪
- [ ] 模型版本升级回归失败

Explain: 这题没有唯一答案，但要覆盖恢复、工具、观测、版本和安全五个方向。每条都要能写出触发方式、预期系统行为和验收证据，不然只是列了几个名词。

### 7_handoff_trigger choice

**Q:** 哪一种情况更应该人工接管？

- [ ] 普通查询变慢一点
- [ ] 提示词稍微变长
- [ ] 权限不足、敏感数据或高风险写入

Answer: 权限不足、敏感数据或高风险写入
Explain: 越权、敏感和高风险写入通常应该接管，而不是继续自动化。

### 8_pause_rule synthesis

**Q:** 写一条暂停自动化的规则

Explain: 合格答案要明确：什么信号触发暂停、通知谁、先停哪个自动化路径、谁判断是否回滚或接管、恢复条件是什么。只有“加告警”不够，因为告警不等于处置。

### 9_checklist debug-checklist

**Q:** 从下面选出验收证据

- [ ] 崩溃恢复演练
- [ ] 重复副作用拦截
- [ ] 门禁结果
- [ ] 回滚演练
- [ ] 供应商海报

Explain: 前四个应该勾选。供应商海报不是验收证据。验收要看恢复、拦截、门禁和回滚是否真的演练过，而不是看宣传材料好不好看。

### 10_cost_cap fill

**Q:** 写出一个上线前必须检查的成本或延迟阈值

Explain: 可以写 p95 latency、token usage、工具调用次数、重试次数或外部写请求次数的阈值。关键是它要能直接阻断上线，不是挂在文档里看一眼就算。

### 11_boundary_note synthesis

**Q:** 写三句话说明为什么边界必须先于模型优化

Explain: 边界先于模型优化，因为上线最先遇到的是权限、恢复、重复副作用和回滚问题，不是回答风格问题。即使模型更聪明，只要边界不稳，系统还是会在高风险动作上出事。比如发信、退款、删数据这类动作，系统层不先定规则，模型再好也不能直接放行。
