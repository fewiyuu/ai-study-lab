课程总论

可靠性不是最后补一个重试按钮，而是从运行设计、工具契约、状态保存和观测数据开始。

Overview
核心问题

智能体系统同时有模型不确定性、工具副作用、长运行状态和外部依赖，因此必须把失败当作常态来设计。

学完能做

能为一个长任务系统设计恢复点、重试策略、幂等保护、追踪字段、评测集和上线验收清单。

课程位置

位于运行承载工程和工具契约之后，是走向生产环境前的工程收束课程。

基于哪些资料

可靠性课程以持久执行、追踪语义、结构化输出和工具评测资料为主。

Sources
LangGraph Durable Execution

持久执行文档用于理解检查点、恢复、确定性、任务封装和中断续跑。

Temporal

平台文档用于理解崩溃恢复、工作流、活动、长时间等待和可靠执行模型。

OpenAI 智能体 SDK 追踪

追踪文档用于建立运行记录、span、敏感数据控制和排查路径。

OpenTelemetry GenAI

语义约定用于把模型调用、工具调用、事件和指标纳入统一观测语言。

OpenAI Structured Outputs

结构化输出指南用于约束最终结果和中间决策，减少格式漂移。

BFCL 与 SDK 更新

BFCL 用于工具调用评测维度，更新记录用于关注运行行为变化。

建议学习对象

适合准备把智能体系统从原型推进到长期运行的人。

Audience
工程实现者

需要处理超时、失败、重试、任务恢复、工具副作用和运行记录。

质量负责人

需要把评测、回放、回归和上线验收变成稳定流程。

平台负责人

需要判断哪些能力应由模型承担，哪些必须由外层系统保证。

前置知识

可靠性工程默认你已经理解运行承载层和工具契约。

Before
运行承载

知道模型调用、工具分发、状态保存、工作空间和追踪记录分别由谁负责。

工具契约

知道参数 schema、返回结构、错误语义和权限控制如何影响执行结果。

常规可靠性

理解超时、重试、幂等、日志、指标、告警和回归测试的基本含义。

推荐路径

按故障生命周期学习：先防丢，再防重，再能看见，最后能验证。

单元设计

8 Units
01

可靠性问题地图

存在理由：先把失败分类，才能知道哪些问题靠模型改进，哪些必须靠运行层、工具契约、状态管理或观测系统解决。

打开学习页
02

持久执行与检查点

存在理由：长任务会被网络、进程、部署和人工等待打断；检查点和恢复语义决定系统能否从正确位置继续。

打开学习页
03

确定性、幂等与副作用隔离

存在理由：恢复和重试会重新经过部分代码。必须区分可重放逻辑和真实副作用，避免重复扣款、重复写文件或重复调用外部服务。

打开学习页
04

工具失败、超时与重试策略

存在理由：工具调用是故障高发点。本单元建立超时、重试、熔断、人工确认和错误可见性的处理原则。

打开学习页
05

结构化输出与防漂移边界

存在理由：如果输出格式、字段含义或中间决策漂移，后续执行链路会被污染。结构化边界是可靠性的基础防线。

打开学习页
06

追踪、指标与事件语义

存在理由：可观测性要能回答“哪一步、哪个工具、哪个输入、哪次重试、哪个版本导致问题”，而不只是保存一段聊天记录。

打开学习页
07

评测、回放与回归测试

存在理由：智能体系统需要持续验证工具选择、参数准确、任务完成、异常恢复和成本边界；一次成功运行不足以证明可靠。

打开学习页
08

上线前验收与运维边界

存在理由：最后要把前面内容收束成可执行清单：权限、数据、日志、告警、回滚、版本升级和人工接管都要有明确标准。

打开学习页

后续推荐

可靠性课程之后，下一步取决于你的系统瓶颈。

Next

瓶颈是上下文质量：学习 Context Engineering，重点处理写入、选择、压缩、隔离和长期记忆。

瓶颈是外部集成：深入 MCP 规范和工具契约，重点处理授权、资源边界、结构化工具结果和安全提示。

瓶颈是组织协作：学习多智能体系统，重点判断何时拆分角色、何时保持单一运行循环。

资料边界

可靠性工程会借鉴多个框架，但不把某个框架当成唯一范式。

Scope

本课程不覆盖完整 SRE 体系、模型安全红队、企业合规审计、成本会计或所有编排平台的实现细节。

课程重点是智能体长任务的工程可靠性：持久执行、幂等副作用、工具失败、结构化边界、追踪、评测和上线验收。

所有课程单元入口已链接到真实存在的学习页。
