ReAct / Agent 循环

ReAct 与 Agent 循环

这一页先把 Agent 讲成一个受控循环：模型先判断下一步，程序执行被允许的动作，环境把观察结果送回来，再由模型决定继续、改计划还是停下。后面的工程架构、设计模式、评测安全和多 Agent 判断，都建立在这条主线上。

学习目标

goals

先抓住的三件事

第一，把 Agent 看成一轮一轮推进的控制回路。第二，知道每一轮里谁负责“想”、谁负责“做”、谁负责“记”。第三，能从运行轨迹里看出卡住、越权、重复搜索和提前停止分别出在哪一层。

这页的边界

这里先讲 ReAct、最小循环、状态台账、workflow 边界和停止条件。规划器、记忆检索、评测框架、长程恢复和多 Agent 协作放到后面的单元展开。

先抓主线

overview

一个任务，四次回头

最小的 Agent 会围绕“目标 → 判断下一步 → 执行动作 → 接收观察 → 更新状态”反复推进。只要中间有一步不清楚，下一轮就应该把不确定性暴露出来，别硬往前走。

这也是 ReAct 的核心：Reasoning 和 Acting 交错出现。思考要给下一步动作提供依据，动作要把外部事实带回来。

一个会贯穿全页的小请求

用户问：“这个订单为什么还没发货？”

这类问题很适合拿来讲 Agent，因为它不该直接猜答案。先查状态，再看库存，再判断是否需要追问或转人工，整个过程都能对应到一个具体回路。

ReAct 的直觉

react

推理和动作交错

ReAct 的价值不在于“多写一点思考文本”，而在于让模型每做一步都能被外部结果纠正。只靠内部推理，错误会沿着同一条链继续长大；只靠一次性计划，遇到事实变化时很难回头。

当模型在思考里承认“我现在还不知道”，然后发起查询或调用工具，它是在把不确定性暴露出来。观察结果回来之后，它才有资格更新结论。

常见误会

有人把 ReAct 读成“先想很多，再回答”。那会把真正有用的地方漏掉：观察结果如何改变下一轮动作。没有这一步，轨迹只是带着工具名的长推理。

目标 → 计划 / 判断 → 动作 → 观察 → 更新状态 → 决定继续或停止

这里真正要盯住的是观察如何改变下一步：能不能继续、该查什么、要不要转人工。

最小循环

loop
before 用户问题、上下文、已有事实

读入当前轮要解决什么。

after 下一步计划或工具请求

决定是继续查证、提问还是结束。

reads 工具 schema、权限、预算

程序校验后才执行。

writes 观察、报错、状态变更

结果必须回到下一轮，日志只能作为复盘材料。

symptom 不再重复、越权或空转

如果这一步没有做，Agent 就容易卡住。

一个健康的循环还要把边界控制和回流接上：谁能调用什么工具，失败后怎么记，状态缺失时要不要停。真正难的部分通常在外层约束，而不在 `while` 本身。

伪代码：受控 Agent Loop
messages = [("user", user_input)]
state = {"budget": 8, "need_followup": False}

for step in range(MAX_STEPS):
    proposal = model(messages, state, tools)
    if proposal.kind == "final":
        return proposal.text

    action = proposal.action
    if not allow(action, permissions, budget=state["budget"]):
        messages.append(("system", "action rejected"))
        state["need_followup"] = True
        continue

    observation = run_tool(action)
    messages.append(("tool", observation))
    state["budget"] -= 1

    if done(observation, state):
        return compose_answer(messages, state)

状态台账

state
对象 读什么 写什么 什么时候会脏 出错症状
messages 用户历史、工具返回、系统提示 每一轮新观察和新指令 工具结果没写回去，下一轮就看不到事实 模型重复问同一件事，或者凭空继续推
state 预算、已知事实、是否需要追问 更新任务事实和执行边界 状态和消息不同步时，模型会拿旧信息做决定 已经查到的结果被当成没查到
permissions 当前用户能做什么、工具允许什么 记录拒绝或授权结果 没有检查写动作时，越权风险会直接穿透 误删、误改、误执行有副作用的操作
budget 剩余轮数、时间或成本 每轮递减，必要时触发停止 无限循环、重复搜索、重复调用工具 任务没结束，额度先耗尽

读这个表时的动作先找“谁在读旧值”，再找“谁没把新值写回去”。很多 Agent 问题卡在状态衔接上。

一个例子走到底

example

订单为什么没发货

第 1 轮：模型先判断“不能凭记忆回答”，决定查订单状态。第 2 轮：程序执行只读查询，观察到“已付款、仓库缺货”。第 3 轮：模型把缺货和预计补货时间合并，判断今天不能发货。第 4 轮：如果客户要一句话回复，就把已知事实压缩成可发给用户的结论。

这个例子里，每一轮做的事都很小，但它暴露了完整机制：先查证，再更新，再决定下一步。只要观察回流被截断，后面就会退化成猜测。

从这个例子看出什么

Action 负责提出可执行动作，Observation 负责把外部事实带回下一轮。把这两层看清楚，Agent 才能从“能调用工具”变成“能把工具结果用起来”。

轮次 模型要做什么 程序要做什么 观察到什么 下一轮怎么变
1 决定先查订单 准备受控工具调用 还没有事实 继续进入查询
2 根据观察更新计划 执行只读查询 已付款，仓库缺货 需要结合补货时间判断
3 压缩成用户可读结论 检查是否还缺信息 今天大概率不能发货 可回复，也可转人工

Workflow 和 Agent 的分界

boundary

先看控制权在哪

当步骤顺序已经画死，分类、检索、总结、校验都由代码固定控制，系统更像 Workflow。模型可以参与其中某一步，但它不能改路径。下一步动作要由模型结合当前观察来决定时，才进入 Agent 设计空间。

这里看的是控制权。流程固定、验收清楚、异常分支可列完的时候，Workflow 更稳。需要探索、需要中途纠偏、需要处理不完整信息时，Agent 才值得上场。

判断句

能提前把所有步骤写死，就优先用 Workflow。不能提前写死、必须靠观察决定下一步，就用 Agent，但同时要准备权限、追踪、预算和停止条件。

停止条件

stop

停下来也是正常分支

一个好的 Agent 需要知道什么时候该停：信息已经够了、权限不够、用户没给关键参数、预算已经快耗尽、工具结果不可靠。把这些情况写成正常分支，比强行继续要安全得多。

什么时候继续

如果还缺的是能查到的事实，就继续查；如果缺的是用户才能补的输入，就先澄清；如果继续下去会越权或放大风险，就停下来。每一轮都要回答一个问题：它有没有减少不确定性。

[ok] 订单状态查到了
[ok] 仓库缺货信息回来了
[ok] 计划更新为“先回复客户，说明补货时间”

这个小实验的目的很直接：你要能从输出看出是“该继续查”、还是“该停下来”。如果结果没变，继续跑只是在耗预算；如果动作越权，继续跑只是在制造事故。

常见误区

mistakes

有工具就算 Agent

不算。单次工具调用只是能力入口，Agent 还要有多轮选择下一步的回路。

思考写得长就更可靠

不一定。长思考可能只是把错误包装得更像对的，真正该看的还是观察是否回流、状态有没有更新、下一轮有没有变。

停止说明系统失败

也不对。缺信息、缺权限、风险过高时，停下来才是正确动作。把停下来当成正常分支，系统才更可控。

换更强模型就能解决一切

不会。工具 schema、状态写回、日志和停止条件出了问题，再强的模型也会在错误边界里绕圈。

先别追的东西

boundary

这一节先不展开

先别追规划器、记忆检索、长链路恢复、多 Agent 协作和评测框架。它们都很重要，但要在你已经看懂循环、状态和停止条件之后再进来。

它们会在哪里回来

工程架构单元会把工具、记忆、权限和日志拼起来；设计模式单元会讲 routing、planning 和 reflection；评测追踪与安全会把这些机制变成验收条件；多 Agent 单元再讨论什么时候要拆角色、拆上下文。

分叉追踪

trace

同一条请求，三个结局

还是“这个订单为什么还没发货？”这条请求。第一种结局是查到了订单和库存，直接回复原因；第二种结局是订单号缺失，先向用户追问；第三种结局是查到信息却发现需要退款或改单，那就先停下来做权限确认。三种结局对应三种不同的停止条件。

这个分叉很重要，因为很多 Agent 页面只讲“继续查”，却不讲什么时候该换分支。没有分叉追踪，停止条件就会变成一句口号。

看分叉时先问三句

第一，当前轮还缺的是系统能查到的事实，还是只有用户能补的输入？第二，继续动作会不会引入副作用？第三，观察结果有没有已经足够让系统收口？

分支 触发条件 下一步 为什么这样走
继续查 还有可查事实没查完 发起下一次查询 目标还没被事实支撑住
先追问 缺少用户才能补的参数 停止并澄清 继续查不会新增信息
先停下 动作会越权或有副作用 请求确认或转人工 正确性不只看答案，还看后果

故障拆分

debug

四种最常见的坏轨迹

第一种是只会空想，不会发起工具调用。第二种是工具调用了，但观察没有写回。第三种是观察写回了，但状态没更新，下一轮还在拿旧信息做决定。第四种是该停的时候没停，导致越权、重复和空转一起出现。

这四种坏轨迹很值得记，因为它们分别对应四个不同的修法。只要把症状看混，修法就会变成瞎补。

对应的修法

空想问题优先看工具 schema 和触发条件；观察不回流先看 messages；状态没更新先看 state；该停不停先看 budget、权限和人工确认分支。修法的顺序应该从数据流往回找，不要从最终答案硬拧。

症状 最可能的根因 先看哪里 下一步检查
一直在想，没有行动 工具入口没被触发 模型输出和工具 schema 是不是任务本身需要查证
工具执行了，但又回到老问题 Observation 没进 messages 消息拼接点 下一轮能不能看到新事实
看起来知道了，其实还在拿旧信息决策 state 没更新 状态写入点 事实是否同时写到 state 和消息里
已经越权，还在继续跑 停止条件缺失 权限和预算检查 是否存在人工确认分支

这个表的用途很具体：你以后看到 Agent 报错，不要先问“模型为什么这么笨”，先问“是哪个状态没有回流”。大多数问题都能在这四个症状里对上号。

怎么把坏轨迹修回来

repair

修复顺序

先修入口，再修回流，最后修停止。入口问题通常是工具没有被触发，或者工具 schema 写得太散；回流问题通常是 observation 没有写回 messages 或 state；停止问题通常是预算、权限和人工确认分支没接上。

这个顺序很重要，因为如果你先修停止条件，却没把观察写回来，系统还是会重复空转。先修回流，再修收口，才有可能真正止住坏轨迹。

修法对应的检查点

入口看“模型有没有被允许请求工具”；回流看“工具返回有没有进入下一轮输入”；停止看“有没有清楚写出继续、追问、转人工的条件”。这三个检查点可以直接套到后面的工程单元。

坏轨迹修复清单
1. 先确认任务是否真的需要查证，而不是直接回答。
2. 检查工具 schema 是否清楚，模型是否真的能发起 action。
3. 检查 observation 是否写回 messages，state 是否同步更新。
4. 检查 budget、权限和人工确认分支是否存在。
5. 如果结果重复，先停下来，不要把重复查询当进展。

这张清单的意义是把调试动作固定下来。你下一次碰到 Agent 卡住时，不需要先想很大的理论，先按这五步检查，通常就能把问题缩到一个小层面。

一段日志怎么看

lab

把一段坏轨迹拆开

假设系统连续三轮都在查同一条订单，工具返回的内容完全一样，但模型还是继续发起查询。这个时候不要只说“它在重复”。要继续问：它是没收到观察，还是收到了但没写回；是状态没更新，还是停止条件没写；是工具入口没限制，还是模型根本没看到查询已经没有增量。

真正的调试顺序应该是先找数据流，再找控制流。先看 Observation 有没有进 messages，再看 state 有没有同步，再看 budget 和权限有没有拦住空转。

这段日志的结论

如果三轮查询都没有新增信息，继续查就是坏动作，不是进展。此时最合理的分支通常是停止并澄清，或者把任务转给人工。能把这一步说清楚，说明你已经会把日志读成工程问题，而不只是读成文本。

坏轨迹日志
[t1] Thought: 我先查订单状态
[t1] Action: get_order_status(order_id=123)
[t1] Observation: status=paid, warehouse=out_of_stock
[t2] Thought: 再查一次订单状态更稳
[t2] Action: get_order_status(order_id=123)
[t2] Observation: status=paid, warehouse=out_of_stock
[t3] Thought: 还是再查一次确认
[t3] Action: get_order_status(order_id=123)

读这段日志时，最关键的判断不是“它查了几次”，而是“每次查询之后有没有新增事实”。如果没有，就该停，不该再用同一个动作证明同一个事实。

权限与副作用

permissions

为什么这里要单独讲

一条看起来很普通的 Agent 轨迹，真正决定风险的往往是它“能不能改东西”。查询订单状态和执行退款，在表面上都像一次工具调用，但一个只是读，另一个会改变外部世界。

如果你不把读写分开，模型很容易把副作用动作当成普通步骤，结果就是系统会在没有确认的情况下把危险操作跑出去。

读 / 写 / 副作用

读操作只把事实带回来，比如查询库存、查询订单、读取规则。写操作会改变系统状态，比如取消订单、提交退款、修改地址。副作用动作通常需要更高权限或人工确认，因为一旦执行，回滚成本就会上去。

动作 类型 默认策略 如果放错会怎样
查询订单状态 读 可自动执行 风险较低，但仍要记录观察
查询库存 ETA 读 可自动执行 会影响下一轮判断，但不会改外部状态
提交退款 写 / 副作用 先确认权限 错误退款会直接产生业务损失
取消订单 写 / 副作用 先确认权限 会改变用户的实际结果

读完这个表，你应该能把“查事实”和“改事实”分开。后面的评测、安全和 Harness 单元，基本都会围着这条线展开。

把循环接到工程层

handoff

这一层开始回答“系统怎么跑”

ReAct 只回答了最小问题：模型如何在思考、行动和观察之间来回。真正做成产品时，还要再往外走一层，接上工具注册、权限校验、日志、预算、失败恢复和人工确认。循环本身只是执行关系，系统要靠外层工程把它包稳。

把这件事想清楚很重要。很多人第一次看 Agent，会把注意力放在“模型下一轮该想什么”上，但真正决定系统能不能上线的，往往是外层边界：这一步能不能调用，失败怎么记，返回值有没有写回，危险动作有没有挡住。

从循环到服务，最少要多出四件事

第一，工具要有名字、参数和返回值，不能只是口头描述。第二，权限要在程序侧拦住，不能只靠模型“自觉”。第三，日志要能让人复盘每一轮到底发生了什么。第四，预算和停止条件要能兜住空转，不然循环会一直跑到资源耗尽。

循环里的概念 工程层的对应物 如果这里断了会怎样 下一个单元会继续讲什么
Action 受控工具调用 模型会想做事，但程序不知该执行什么 工程架构里会讲工具编排、上下文和记忆
Observation 返回值、错误、状态变更 事实回不来，下一轮只能重复老判断 评测与追踪单元会把它变成可检查的轨迹
Stop condition 预算、权限、人工确认 系统要么空转，要么越权 安全单元会继续拆开风险边界
State 会话状态、任务状态、工作记忆 看起来知道了，其实下一轮还在读旧值 设计模式单元会讲状态如何驱动分支

你可以把这个映射当成过桥图。左边是概念，右边是落地。只要某个概念在右边找不到位置，它就还停留在“能说清”阶段，离“能上线”还差一步。

练习

diagnostic
答题目标

这组题要检查你能不能把 Agent 轨迹拆成真实的工程动作：谁在读什么、谁在写什么、哪里该停、哪里该查、哪里该拒绝。

检查重点

优先看控制权、状态回流和停止条件。开放题不用写长，但必须说明因果和边界。

1. 概念辨认

先把术语放回它们的工程位置。

复盘提示：先区分“能力入口”和“多轮回路”。很多错答都把工具调用直接当成 Agent。

2. 轨迹与状态

看得懂回路，才有资格调试回路。

复盘提示：Observation 要能改变下一轮。看不到写回，后面的推理就容易空转。
05fill

ReAct 的三个基本环节可以概括为：____、____、____。

06fill

程序包住模型、负责工具执行和边界控制的运行框架可称为 ____。

07ordering

把最小 Agent Loop 排序：A 工具返回观察；B 模型选择动作；C 用户给目标；D 程序执行工具；E 判断停止或继续。

参考答案 / 评分要点

顺序应是 C → B → D → A → E。答题时要说明：目标先进入系统，模型基于目标提出动作，程序执行动作，观察回流，最后判断是否继续或停止。少写一步或把观察和最终回答混在一起，都说明回路没理清。

08matching

匹配概念：Thought / Action / Observation / Stop condition 分别对应什么工程位置？

参考答案 / 评分要点

Thought 对应当前轮的判断与计划，Action 对应程序要执行的受控动作，Observation 对应工具或环境返回的事实，Stop condition 对应循环结束或转人工的分支。回答时要写工程位置，不要只翻译英文。

09code reading

读伪代码：为什么工具执行后要把结果追加回 messages？如果不追加会怎样？

参考答案 / 评分要点

因为下一轮模型要基于观察继续决定动作。若不追加，模型看不到新事实，只会重复旧判断、再次查询同一件事，或者在缺少证据的情况下直接下结论。

09berror diagnosis

如果观察没有写回状态，最常见的两个后果是什么？

参考答案 / 评分要点

常见后果是重复查询和重复推理。前者因为系统看起来“还没查过”；后者因为下一轮依然以旧信息为输入，继续沿着同一条错误路径走。答题时要把“没回流”和“重复空转”连起来。

09csynthesis

把“入口、回流、停止”三个修复点按你认为正确的顺序排出来，并解释为什么。

参考答案 / 评分要点

常见顺序是先入口、再回流、最后停止。理由是：如果动作入口都没接上，后面的状态和停止条件无从谈起；如果回流没接上，系统会反复空转；等前两层稳定后，再修停止条件才有意义。答题时要写出层次，不能只给一个名词列表。

3. 诊断与转移

把错误放回机制里，别只说“结果不对”。

复盘提示：先说错因，再说后果，再说修法。这样答案才像诊断，不像感想。
10error diagnosis

某 Agent 工具失败后只收到“失败”，然后反复重试。错因、后果、修法各写一句。

参考答案 / 评分要点

错因：观察没有给出可用信息，状态也没有更新。后果：模型只会在同一个错误上循环，可能耗尽预算。修法：让工具返回结构化错误、把失败原因写回状态，并在重试前判断是否需要改参数或转人工。

11error diagnosis

系统没有最大步数，也没有需要人工确认的分支。会出现什么风险？

参考答案 / 评分要点

至少应写出两类：预算或时间被耗尽，重复调用同一工具导致空转；以及遇到高风险动作时没有刹车，可能把越权或误操作直接执行出去。说清楚风险如何从“没有停下来的条件”里长出来，比只写“会出错”更重要。

12scenario transfer

客服机器人要查订单和退款。哪些动作可以自动，哪些动作需要确认？

参考答案 / 评分要点

只读查询通常可以自动，比如看订单状态、库存、物流时间；写数据或有副作用的动作通常需要确认，比如退款、改地址、取消订单。答案要按“读 / 写 / 副作用”分层，不能停在“有些自动，有些不自动”。

13scenario transfer

你要做一个读论文助手。它需要 Agent 还是 Workflow？给出你的判断条件。

参考答案 / 评分要点

如果任务路径稳定、步骤可预先写清，Workflow 就够；如果需要根据论文内容、用户追问和中途反馈动态选择下一步，就更像 Agent。好的答案要先写“判断条件”，再下结论，不要只报一个名词。

13bscenario transfer

订单需要退款，但当前用户没有权限。系统该怎么处理？

参考答案 / 评分要点

应该先停下来，不要硬执行退款；然后说明原因、请求人工确认或切到有权限的流程。好答案会明确提到“这一步有副作用”以及“不能把越权动作包装成普通查询”。

14short answer

用 2-3 句话解释 Workflow 和 Agent 的核心区别。

参考答案 / 评分要点

要点是控制权和路径。Workflow 的步骤由代码固定，Agent 让模型根据观察决定下一步。写得好的答案会提到“谁来决定下一步”和“遇到新信息时能不能改计划”。

15short answer

为什么说 Agent Loop 本身不复杂，复杂的是循环外的工程边界？

参考答案 / 评分要点

因为真正容易出问题的地方包括工具验权、状态写回、预算控制、错误记录和停止分支。答案最好至少提到工具、状态、权限、评测中的两个。

16short answer

什么时候 Agent 应该向用户澄清？

参考答案 / 评分要点

当缺少的是用户才能补的关键信息，或者继续查下去只会重复已有事实时，应该先澄清。答案要说出“缺什么”和“为什么不能继续盲查”。

16bsynthesis

把“继续查 / 先追问 / 先停下”三种分支各写一句判断规则。

参考答案 / 评分要点

继续查：还有可查事实能减少不确定性。先追问：缺的是只有用户能补的输入。先停下：继续动作会越权、带副作用或已经没法增加有效信息。答题时不要把三个分支混成一条。

17synthesis

用 5-7 句话向新人解释 Agent Loop。必须串联：目标、思考、行动、观察、状态、停止条件、工具权限。

参考答案 / 评分要点

好的答案会把七个要素串成一条过程线，避免挨个背术语。先说目标如何进入系统，再说思考如何形成下一步，接着说程序如何执行工具并检查权限，最后说观察如何回到状态里、停止条件如何让循环收口。

18code reading

模型给出很长 Thought，但没有调用任何能验证事实的工具。你如何判断这条轨迹有没有问题？

参考答案 / 评分要点

要先看任务性质：如果是开放式想法整理，长 Thought 未必有问题；如果是事实核验、查询或执行任务，没有观察回流通常就是问题。好答案会说明“任务需要查证时，长思考不能替代工具验证”。

19checklist

按本页的坏轨迹修复顺序，写出 5 个检查点。

参考答案 / 评分要点

参考检查点：任务是否需要查证；工具入口是否能触发；观察是否写回 messages；状态是否同步更新；预算、权限和人工确认是否存在。答案不必逐字相同，但要覆盖入口、回流、停止三层。

20synthesis

把“Action / Observation / Stop condition / State”分别翻译成工程上能检查的一件事。

参考答案 / 评分要点

参考对应：Action 对应受控工具调用和参数校验；Observation 对应返回值、错误和状态变更是否写回；Stop condition 对应预算、权限和人工确认是否真正生效；State 对应会话状态或任务状态是否同步更新。好答案会写出“怎么检查”，而不是只写名词翻译。
