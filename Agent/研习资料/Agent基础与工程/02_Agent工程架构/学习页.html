Agent Architecture / Harness

Agent 工程架构：把循环放进可靠运行框架

上一单元把 Agent 看成“思考、行动、观察”的循环。本页继续往工程层走：循环本身只说明模型怎样决定下一步，真正决定稳定性的，是模型外面的运行框架。你会把目标、上下文、工具、记忆、任务状态、追踪和权限放到同一张架构图里，学会判断一个 Agent 为什么能恢复、为什么能排错、为什么不会把一次临时观察当成永久规则。

课程地图上一单元：ReAct 与循环下一单元：设计模式

读完本页后能做什么

  • 把 Agent Loop 与 Harness 的职责分开：模型负责决策，框架负责边界、状态、执行和验证。
  • 为一个长任务写出最小可恢复状态，并说明哪些信息进上下文、工具结果、记忆或 trace。
  • 看到工具误选、上下文污染、记忆污染或状态丢失时，能沿工程链路定位问题。

学习目标

goals

从循环走到系统

你已经知道 Agent 会在推理、行动、观察之间循环。本页要补上的，是循环外面的工程外壳：谁准备上下文，谁执行工具，谁保存状态，谁记录证据，谁决定停止。

从“能跑”走到“能恢复”

一次演示成功不等于工程稳定。一个真实 Agent 要能跨会话继续任务，要能解释为什么调用某个工具，也要能在失败后定位是上下文、工具、记忆还是权限出了问题。

从概念走到清单

学完后，你应该能写出最小 Harness 清单：上下文层、工具契约、任务状态、记忆策略、权限边界、trace 字段和验证方式。这个清单比“写一个更长 prompt”更接近工程入口。

本页边界这里不讲某个框架的 API，也不展开多 Agent 协作。先把单个 Agent 的运行框架讲清楚，后面的设计模式、评测和 Multi-Agent 才有稳定落点。

架构主线：循环要被包进 Harness

overview

Agent Loop 解释的是模型内部的下一步选择：想一想，调用工具，读观察，再决定下一步。工程架构解释的是模型外面的责任分配：输入从哪里来，工具由谁执行，状态写到哪里，错误怎样回到模型，最终结果怎样验证。把这两层混在一起，系统会很快变成一团长 prompt。

一个可维护的 Agent 通常要有一条清楚的链路：用户请求进入 Harness；Harness 组装当前上下文和可用工具；模型输出行动意图；工具执行器检查权限并运行工具；观察结果被压缩成下一步可用的信息；任务状态和 trace 同步更新；验证器判断是否可以停止或需要修复。模型仍然重要，但模型只占其中一段。

入口用户目标、硬约束、风险级别

先把“要做什么”和“哪些事不能做”拆开，避免后面靠模型猜边界。

上下文常驻 + 按需 + 当前观察

只装入本轮决策需要的事实，长资料先检索或摘要。

模型计划、工具选择、停止判断

模型负责判断下一步，但不直接越过权限和执行层。

工具参数校验、执行、错误语义

工具结果要服务下一轮决策，避免把一大段原始输出塞回去。

证据状态、日志、评测、恢复点

后续排错和跨会话恢复，都依赖这里留下的结构化记录。

环节读什么写什么错位时的症状
请求入口用户目标、身份、权限、风险提示任务目标和硬约束Agent 开始做用户没有授权的事，或者把闲聊当任务执行。
上下文构建项目规则、按需资料、最近观察、记忆摘要本轮提示中的可见事实回答混入无关资料，关键约束被长文本淹没。
模型决策目标、上下文、工具描述、上轮观察计划、工具调用或停止理由工具误选、重复调用、过早停止、无法解释下一步。
工具执行模型意图、参数、权限、运行环境结果、错误类型、可选下一步工具返回“成功”但没有证据，或把敏感内部字段暴露给模型。
状态与 trace执行结果、关键观察、验证结论恢复点、事件日志、未解决问题换会话后任务从头再来，或者无法复盘失败发生在哪一步。

你现在应该能做什么看到一个 Agent 方案时，先画出请求、上下文、模型、工具、状态和 trace 的流向；缺哪一段，就先补哪一段。

Harness：模型外面的运行框架

harness

Harness 可以理解为包住模型的运行框架。它指一组工程责任，不等同于某一句 prompt 或某个模型 SDK：加载合适上下文、暴露合适工具、检查权限、执行行动、压缩观察、记录 trace、更新状态、判断是否需要验证或恢复。模型在 Harness 里面做决策，Harness 让这些决策有边界、有证据、有后路。

很多 Agent 不稳定时，直觉会去换更强模型。Tw93 的工程梳理给了一个很实用的提醒：工具定义、验证测试和 Harness 质量经常比单纯提升模型能力更影响成功率。原因很直接：模型只能根据它看到的上下文和工具说明行动；如果工具描述含糊、返回噪声很大、状态没有持久化，模型再强也会在错误输入上做判断。

最小 Harness 接口草图
type AgentHarness = {
  policy: PermissionPolicy
  context: ContextBuilder
  tools: ToolRegistry
  memory: MemoryStore
  state: TaskStateStore
  trace: TraceWriter
  evaluator: StopAndQualityCheck

  run(goal, previous_state): TraceEvent[]
}

模型负责的事

  • 理解当前目标和约束。
  • 在可用工具里选择下一步。
  • 根据观察修正计划。
  • 说明何时可以停止或需要澄清。

Harness 负责的事

  • 决定哪些资料进入上下文。
  • 把工具暴露成清楚的契约。
  • 在程序侧执行权限、超时和确认。
  • 把状态、trace、验证结果写成可恢复记录。

常见判断如果你无法回答“这个 Agent 失败后去哪里看证据”，那问题大概率不在模型，而在 Harness 还没有成形。你现在应该能把“模型判断”和“框架控制”分开列清楚。

上下文工程：窗口很长，也要分层

context

上下文窗口变长后，最容易出现的错觉是“能塞就塞”。但模型注意力有成本。无关材料越多，关键信号越容易被稀释；旧观察、长文档和重复规则堆在一起时，模型会更难判断哪条信息是当前任务的硬约束。这类现象常被称为 Context Rot：上下文越堆越多，决策质量反而下降。

工程上要把上下文按用途分层。常驻层放身份、硬规则和绝对边界；按需层放 Skills、领域资料、长操作手册；运行时层放当前目标、最近观察、未解决问题和剩余预算；持久层放历史 trace、长期偏好和可检索记忆。每一层都要有进入条件和淘汰条件。

$$ S = \frac{R + V + T}{N + H + M + 1} $$

这里的 S 是本轮决策的信号质量启发式，只用于辅助检查，不能当作数学定律。R 表示相关事实，V 表示验证证据，T 表示当前任务状态；N 表示无关噪声，H 表示过期历史，M 表示互相冲突的记忆。设计上下文时，目标是提高分子里的有效信号，压低分母里的干扰。

层级适合放什么进入条件移除或压缩时机
常驻层身份、硬规则、不可越过的安全边界每次运行都必须遵守发现规则重复、过细或已经被程序侧替代时压缩
按需层Skills、领域知识、长文档、工具使用指南任务触发对应领域或操作流程当前步骤不再需要，或只需保留摘要和引用
运行时层当前目标、最近观察、待确认问题、剩余预算本轮决策直接依赖观察被状态吸收，或已经被验证结论替代
持久层历史 trace、长期偏好、稳定事实、会话摘要通过检索、规则或人工确认进入过期、冲突、来源不明或只对旧任务有效时回退

应该进入本轮上下文

订单状态、退款权限、当前用户诉求、已确认事实、下一步需要调用的工具说明。

先不要进入本轮上下文

全部历史聊天、完整商品目录、无关客户案例、过期流程截图、框架内部调试字段。

你现在应该能做什么拿到一段材料时，先问它服务哪一轮决策；不能服务当前决策的内容，要么按需加载，要么摘要，要么留在持久存储里等待检索。

工具契约：描述决定模型会不会选对

tools

工具要按 Agent 的行动边界来设计，不能只看成函数清单。一个工具的名字、描述、参数、返回值和错误消息都会影响模型选择。工具描述只写“能做什么”还不够，还要写什么时候不该用、哪些参数必须来自用户确认、失败时返回什么、是否会产生外部副作用。

工具数量也会影响稳定性。工具太多，模型会混淆相近能力；工具跨多个领域，模型容易把文件工具用于数据库问题，把搜索工具用于已知事实。更稳的做法是先用少量粗工具验证任务，再把高频误用拆成更明确的专用工具。拆分依据应看模型能否稳定选对、工具结果能否帮助下一步判断。

工具结果：服务下一步决策
{
  "ok": false,
  "tool": "refund_status.lookup",
  "error_type": "permission_required",
  "summary": "订单已付款，但退款金额超过自动处理阈值。",
  "evidence": ["payment_status=paid", "amount=1280", "auto_refund_limit=500"],
  "next_options": ["ask_human_approval", "explain_policy_to_user"],
  "llm_visible": true
}
契约字段要写清什么没写清的后果
名称动作对象和动作范围，例如 lookup、create、delete、summarize模型在相似工具之间摇摆，或者调用了过强工具。
描述适用场景、禁用场景、是否有副作用模型把查询工具当执行工具，把危险工具用于低风险步骤。
参数必填字段、来源、格式、是否需要用户确认工具收到猜测参数，执行层很难判断该拒绝还是继续。
返回关键结果、证据、错误类型、可选下一步模型只看到“成功/失败”，无法修正计划。
错误可重试、需人工、权限不足、输入不合法等类型Agent 盲目重复调用，或者把程序错误解释成业务结论。

工具消息也要过滤应用层可能有内部 request id、权限标记、原始 HTML、数据库字段。发给模型前只保留决策需要的信息。否则模型会解释噪声，甚至把内部字段当业务事实。

记忆分层：稳定事实和操作流程分开放

memory

Agent 没有天然的跨会话连续性。会话结束后，当前 messages 往往就不可用了。要让 Agent 下次还能接着做，记忆必须作为基础设施设计，不能临时往 prompt 里补几句。记忆的关键问题是：这条信息下次为什么还应该影响决策；存储方案要服务这个答案。

可以按用途分四类：工作记忆服务当前任务；程序性记忆保存做事方法，例如 Skills 和操作流程；情景记忆保存发生过什么，例如执行轨迹、会话摘要、失败记录；语义记忆保存稳定偏好和长期事实。不同记忆的加载方式不同，写入门槛也不同。

记忆类型放什么加载方式风险控制办法
工作记忆当前目标、最近观察、临时待办运行时注入跨任务复用会污染新任务任务结束后归档或丢弃，只把必要结论写入状态
程序性记忆Skills、工具流程、领域操作步骤按触发条件加载全部常驻会挤占上下文保留短描述，完整流程按需装载
情景记忆会话历史、执行轨迹、失败样本检索、摘要或 trace 回放旧事件被误当当前事实保留时间、来源、任务 id 和有效期
语义记忆稳定偏好、长期事实、项目约定短而稳定地注入错误写入会长期误导需要写入理由、人工确认或回退机制

记忆最大的风险是注入失控。一次临时项目路径、一次失败猜测、一次用户口误，都不应该自动变成长期规则。稳定事实要短，操作流程要按需加载，历史记录要能检索和过期，错误记忆要能撤销。否则 Agent 会把旧事实当成新规则，或者把一次偶然偏好永久写入系统提示。

应该保守写入

“用户今天为了临时导出用了某个目录”只属于当前任务状态。它只对当天任务有效，不能写成长期偏好，也不应该影响明天的所有任务。

可以稳定保留

“用户希望中文回答”“仓库使用 uv 管理 Python 包”这类长期规则，可以短句保留，并在规则变化时更新。

你现在应该能做什么判断一条信息进入哪里：当前上下文、任务状态、程序性流程、历史 trace，还是长期语义记忆。判断依据是用途和有效期，不能只看它显得多重要。

任务状态：长任务要显式写出来

state

长任务不能只靠聊天历史。一个跨 session 的 Agent 任务至少要记录目标、当前阶段、已完成动作、未解决问题、下一步、关键文件或对象、失败记录和验证结果。这样 Agent 重启后可以先读状态，再继续任务，避免重新探索一遍。

显式状态还会降低重复劳动。比如代码 Agent 已经跑过测试、发现两个失败、修了第一个，如果状态没有写下来，下一次可能重新跑全流程，甚至覆盖别人中途做的改动。状态文件不需要复杂，但要稳定、结构化、可检查，并且能回答“我现在应该从哪里继续”。

可恢复任务状态示例
{
  "task_id": "refund-case-2026-06-21-042",
  "goal": "处理同一用户的退款工单，给出可执行答复",
  "phase": "waiting_for_human_approval",
  "done": ["确认订单已付款", "检查库存不足", "计算退款金额"],
  "open_questions": ["退款金额超过自动阈值，需要人工审批"],
  "next_step": "请求主管确认是否放行全额退款",
  "critical_objects": {
    "order_id": "A-1042",
    "payment_status": "paid",
    "refund_amount": 1280
  },
  "last_trace_event": "tool.refund_status.lookup:permission_required",
  "verification": ["金额已与支付网关返回一致", "仍未获得人工审批"]
}
字段用于恢复什么用于下一步决策什么缺失时症状
goal恢复任务目的判断后续动作是否偏题Agent 只记得局部工具结果，不知道整体要完成什么。
phase恢复当前阶段决定是继续执行、等待确认还是验证收尾换会话后重复做已经完成的阶段。
done恢复已完成动作避免重复调用工具工具重复执行，日志里出现相同副作用。
open_questions恢复阻塞点决定是否要问人、检索资料或改计划Agent 假装问题已解决，继续生成不可靠答案。
verification恢复已验证证据决定结果是否可以交付最终回答没有证据链，只剩模型自信表达。

判断句只要任务可能超过一次对话，就要外化状态。状态承担恢复点、决策依据和审计线索，不承担流水账角色。你现在应该能为一个跨天任务写出最小状态对象。

分层架构：从 OpenClaw 线索看落地方式

openclaw

Tw93 的文章用 OpenClaw 展示 Agent 如何落地。学习这一段时，不需要先记住框架细节，先抓住分层思路：渠道接入、消息总线、Agent 核心、工具执行、状态和记忆、追踪与评测应该分开。分开以后，换聊天渠道不会改 Agent 核心，换工具不会改消息入口，调整记忆策略也不会把工具执行层搅乱。

这类架构最值得学的是“变化隔离”。聊天入口可能来自网页、IM、IDE 或命令行；工具可能是搜索、文件、数据库、审批、浏览器；状态可能存文件、数据库或任务系统。只要边界清楚，系统可以逐步演进。边界不清时，每次加工具都要改 prompt、改入口、改日志，最后排错成本会超过开发成本。

channel

接收用户输入，保留身份、会话和权限上下文。

message bus

把消息变成统一事件，隔离不同入口格式。

agent core

组织目标、上下文、工具选择和停止判断。

tool executor

执行工具，处理权限、超时、错误和副作用。

memory/state

保存当前进度、长期偏好、历史 trace 和恢复点。

observability

记录事件、指标、评测结果和人工复盘证据。

你看到的问题优先检查的层原因
网页入口正常，IM 入口丢失用户身份channel / message bus问题发生在入口标准化之前，不应先改 Agent 核心。
模型知道要查订单，却调用了库存工具agent core / tool registry通常是工具命名、描述、示例或上下文选择问题。
工具执行成功，但下轮模型不知道结果tool executor / observation compression结果没有转成模型可用的观察，或被噪声淹没。
跨天恢复后重复退款检查memory/state已完成动作和阻塞点没有写进可恢复状态。
线上失败无法复盘observabilitytrace 没有记录输入、工具、错误类型和验证结论。

你现在应该能做什么以后看任何 Agent 框架，都可以问六个问题：消息入口在哪里，Agent 核心在哪里，工具在哪里执行，状态在哪里保存，权限在哪里拦截，trace 在哪里查看。

追踪与验证：让失败有证据

trace

没有 trace 的 Agent 很难工程化。用户只看到最终回答，开发者需要看到过程证据：模型为什么选这个工具，工具收到了什么参数，执行层是否拒绝了权限，返回结果被怎样压缩，状态是否更新，验证器是否通过。trace 的目的不是堆日志，它要回答“下一次怎样不再错”。

验证也要进链路。很多 Agent 在工具执行后直接生成自然语言答案，看起来顺畅，却没有检查证据是否足够、状态是否关闭、输出是否违反边界。一个最小验证器可以很简单：检查必需字段是否齐全，风险动作是否有人确认，工具结果是否有证据，状态是否留下恢复点，最终回答是否引用了可见事实。

事件字段记录内容排错价值验证钩子
step_id本轮决策序号和父步骤定位重复调用、循环和分叉超过预算时触发停止或人工确认
visible_context_hash本轮上下文摘要或哈希判断失败是否来自上下文污染或缺事实关键约束缺失时阻止交付
tool_call工具名、参数、权限检查结果判断是模型误选还是执行层拒绝高风险工具必须有确认来源
observation工具返回摘要、证据、错误类型判断返回是否足够支持下一步只有“成功”但无证据时要求补查
state_delta本轮写入或修改的任务状态判断恢复点是否完整未写入 next_step 时不允许结束长任务
quality_check最终答案是否满足目标和边界把“模型觉得完成”变成可检查条件缺证据、越权或未解决问题时返回修复环节
最小验证伪代码
def can_finish(task_state, trace_events, final_answer):
    if task_state["open_questions"]:
        return False, "仍有未解决问题"
    if not task_state.get("verification"):
        return False, "缺少验证证据"
    if any(event["risk"] == "high" and not event.get("confirmed_by") for event in trace_events):
        return False, "高风险动作缺少确认"
    if not final_answer.get("evidence"):
        return False, "最终答复没有引用可见证据"
    return True, "可以交付"

你现在应该能做什么给一个 Agent 任务补 trace 字段时，不用追求全量日志。先保证能复盘工具选择、权限检查、观察摘要、状态变化和最终验证。

故障诊断：沿链路找错，不靠猜

failure

Agent 故障常常表现为“模型答错了”，但根因可能在链路的任何位置：上下文太吵、工具描述重叠、工具返回没有错误语义、记忆把旧事实带进来、状态没有恢复、trace 不够定位。排查时不要从“换模型”开始，先沿工程链路看输入和证据。

case: context-rot
symptom: 模型忽略“退款需人工确认”，直接生成退款完成答复。
trace: visible_context 中包含 18 段历史聊天和 4 份无关流程。
diagnosis: 硬约束被长历史淹没。
repair: 把人工确认规则放入常驻层；历史聊天改为按需摘要；验证器检查高风险动作确认来源。
排查顺序要问的问题常见修法
1. 目标和边界用户目标、权限、风险约束是否明确进入 Harness？把硬约束提到常驻层或程序侧 policy。
2. 上下文本轮可见事实是否足够且不吵？分层、检索、摘要、移除过期历史。
3. 工具契约工具名字、描述、禁用场景、错误类型是否清楚？改名、拆分工具、补参数来源和错误语义。
4. 工具结果返回是否支持下一步，是否避免只给原始输出？返回 summary、evidence、next_options 和 error_type。
5. 记忆和状态旧事实是否误入当前任务？当前进度是否可恢复？加有效期、写入门槛、状态字段和回退机制。
6. trace 和验证失败能否定位到具体步骤？最终答复是否有证据？补 trace 字段、验证器和停止条件。

你现在应该能做什么遇到 Agent 错误时，按目标、上下文、工具、结果、记忆、状态、trace 的顺序查。这样能把“模型不行”的泛化判断拆成可修的工程问题。

边界：本页先不追哪些内容

boundary

Agent 工程架构会自然连到很多高级主题：多 Agent 协作、复杂规划器、长期记忆检索、自动评测、安全沙箱、模型路由、队列和分布式执行。它们都重要，但此刻先不要把它们混进来。单个 Agent 的 Harness 没有稳定之前，多 Agent 只会把边界问题放大。

暂时忽略

  • 多 Agent 之间的角色协商和消息协议。
  • 复杂 planner 的搜索算法和反思策略。
  • 向量数据库调参、召回排序和嵌入模型选择。
  • 企业级安全审计、灰度发布和队列调度。

现在要抓住

  • 模型决策与程序执行的边界。
  • 上下文、工具、记忆、状态各自的责任。
  • 长任务的恢复点和下一步字段。
  • trace 能否支撑一次失败复盘。

一个实用的最小版本可以很朴素：一个清楚的系统规则、一个 Context Builder、三五个工具、一份任务状态 JSON、一张 trace 表、一个最终检查函数。这个版本跑稳后，再加复杂规划、多 Agent 或记忆检索，才不会变成一堆看似高级但无法定位问题的组件。

你现在应该能做什么给自己的 Agent 项目画一张“最小可运行 Harness”图。只要图上能标出入口、上下文、模型、工具、状态、记忆、trace 和验证，就已经有了工程化起点。

练习

diagnostic

做题方式

先做本地可检查题，再做诊断题和迁移题。选择题的目标是确认术语和边界；开放题的目标是让你真的能拆一个 Agent 工程方案。

自查标准

好的答案会同时说清职责、证据和失败后果。只写“加 prompt”“换模型”“接数据库”通常不够，因为它没有回答边界、状态和验证在哪里。

一、架构边界

先确认你能把 Agent Loop、Harness、上下文和工具的责任分开。

复盘提示：如果你的答案把所有东西都写进 prompt，说明还没把模型决策和程序侧控制分开。
choicetools

当 Agent 反复在两个相似工具之间选错，优先检查什么？

choicecontext

下面哪类信息最适合按需加载，避免长期常驻在每一轮上下文里？

true/falsecontext rot

判断：长上下文模型可以替代上下文工程，所以不需要压缩、分层和按需加载。

short fillharness

Agent 外部运行框架通常称为哪个英文术语？

二、状态、记忆与恢复

这一组检查你能否把临时事实、长期记忆、程序流程和可恢复状态分开。

复盘提示：状态服务恢复和下一步决策，记忆服务跨任务连续性；把临时状态写成长期记忆是高频错误。
true/falsememory

判断：长期记忆应该有写入、更新和回退机制。

short fillcontext

上下文噪声变多导致决策变差，常被称为什么？

matchingmemory

把下面四类信息分别放到工作记忆、程序性记忆、情景记忆、语义记忆：当前工单下一步、如何查询支付错误码、上次工具超时记录、用户长期偏好中文回复。

参考答案 / 评分要点

当前工单下一步属于工作记忆或任务状态；如何查询支付错误码属于程序性记忆；上次工具超时记录属于情景记忆或 trace 摘要；用户长期偏好中文回复属于语义记忆。评分重点是说明用途和有效期，不靠背诵名称。

debug checkliststate

客服 Agent 跨天处理同一工单。你认为任务状态至少应该保存哪些字段？

参考答案 / 评分要点

前三项是核心字段。全部历史聊天原文不一定进入任务状态，可以作为附件或情景记忆保存，再按需摘要。状态只保存恢复和下一步决策需要的信息。

三、故障诊断

这一组要求你沿工程链路定位问题，避免把所有错误都归因给模型能力。

复盘提示：诊断答案至少包含症状、可能层级、证据和修复动作。只写“优化提示词”通常不够。
diagnosiscontext

Agent 每次启动都读取 20 份长文档，回答质量变差。错因、后果、修法各写一句。

参考答案 / 评分要点

错因：无差别加载造成 Context Rot。后果：关键约束被噪声稀释，模型引用过期或无关材料。修法：常驻层只留硬规则，长文档改为按需检索或摘要，把当前任务事实写入运行时状态。

diagnosistool result

工具返回完整 HTML 页面，模型开始引用导航栏广告内容。你会怎样改返回结构？

参考答案 / 评分要点

应把原始 HTML 留在调试侧，给模型返回摘要、关键字段、证据、错误类型和可选下一步。好答案会说明哪些字段给 LLM 看，哪些字段只留在应用层或日志里。

四、迁移设计

最后把本页方法迁移到你自己的 Agent 场景，检查是否能动手设计。

复盘提示：一个可用设计至少包含工具表、上下文层、状态字段、记忆策略、权限边界和验证方式。
scenariotransfer

设计一个论文阅读 Agent。哪些内容放 Skills，哪些放任务状态，哪些放长期记忆？

参考答案 / 评分要点

Skills 可放“如何读论文、如何抽取方法、如何做引用检查”；任务状态可放当前论文、已读章节、待查问题、下一步；长期记忆可放用户偏好的综述格式、长期研究方向和稳定术语表。不要把某篇论文的临时结论写成长期事实。

safetytool boundary

给 Agent 加一个 shell 工具。写出至少 4 条程序侧护栏。

参考答案 / 评分要点

可包括：限制工作目录；高风险命令需人工确认；禁止默认递归删除；设置超时；记录命令、退出码和关键输出；压缩高噪声日志；区分给模型看的摘要和只给应用层看的原始输出。好答案会说明这些护栏由程序执行，不靠模型自觉。

synthesisarchitecture

给一个你熟悉的工作流，写出最小 Harness 清单。

参考答案 / 评分要点

评分重点：是否有明确目标；是否区分常驻、按需和运行时上下文；工具是否有禁用场景和错误语义；状态是否能恢复；长期记忆是否有写入门槛；权限是否在程序侧；trace 是否能定位失败；验证是否能阻止无证据交付。

feynmanexplain

用 5-7 句话解释 Agent 工程架构，必须串联 Harness、上下文、工具、记忆、状态和 trace。

参考答案 / 评分要点

好解释会说：Agent 不只是模型循环；Harness 把模型放进受控环境；上下文工程决定模型本轮看什么；工具契约决定模型能做什么和怎样收到结果；记忆和状态保证跨会话连续性；trace 和验证让系统能复盘、恢复和交付。
