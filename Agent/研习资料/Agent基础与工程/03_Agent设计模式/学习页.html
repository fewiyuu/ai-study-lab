Agent Workflow / Design Patterns

Agent 设计模式：按任务形状选工作流

上一单元把 Agent 放进 Harness、上下文、工具、记忆和 trace 里。本页继续往前一步：先看任务形状，再决定用哪种工作流模式。线性拆分、路由、并行、编排器-工作者、评估器-优化器，都是把模型放进可控流程的方法；只有当任务真的需要观察驱动的探索时，才值得把控制权更多交给更 Agentic 的循环。

课程地图上一单元：Agent 工程架构

读完本页后你能做什么

  • 能先判断一个任务是线性拆分、类型分流、并行探索，还是需要动态编排。
  • 能为一个工作流写出输入、输出、检查点和失败后的下一步。
  • 能识别过度设计，并知道什么时候该升级到更自主的 Agent 循环。

学习目标

goals

先看任务形状

线性、分流、并行、动态拆分、反复优化，对应的控制方式不同。你要先看任务是什么形状，再决定流程怎么写。

再看控制边界

同样是“让模型做事”，有的任务适合固定工作流，有的任务要靠模型根据观察调整下一步。别把二者混成一类。

最后看升级条件

更复杂的模式不一定更好。只有当任务需要探索、分派、回溯或反复挑错时，才值得升到更 Agentic 的结构。

本页边界：这里讲模式选型，不讲某个框架的 API，也不讲多 Agent 协作。先把单个任务如何选工作流讲清楚。

先抓主线：模式是任务形状的答案

overview

Anthropic 的工程文章给了一个很实用的起点：先选最简单的方案，能用 workflow 完成的任务就先别写成自主 Agent。对这门课来说，设计模式要回答“这个任务该由谁决定下一步”。

你可以把选择顺序记成一条线：先线性拆分，再做输入分流；如果子任务彼此独立，再并行；如果任务结构不固定，就交给编排器分派；如果输出需要反复挑错，就加评估器；只有当任务必须根据观察不断调整，才往更 Agentic 的循环走。

任务信号先选模式常见误判
步骤固定、检查点清楚Prompt Chaining把固定工作流写成自主循环，反而更难查错。
输入类型差异大，后续动作不同Routing分类后还是同一条路径，路由就没有意义。
多个部分可以独立完成Parallelization子任务之间还要频繁同步，却硬拆成并行。
任务结构不稳定，需要动态拆分Orchestrator-Workers把动态分派误写成固定流程，灵活性不够。
输出可按明确标准反复检查Evaluator-Optimizer让评估器只说“不错”，没有可执行反馈。
必须根据观察继续探索更 Agentic 的循环任务其实很稳定，却过早引入自主性。

读完这一节，你应该能先描述任务形状，再谈选哪种模式。

选型矩阵：先问四个问题

selection

先不要急着背模式名。看任务时，先问四个问题：步骤是否固定？输入是否可分流？子任务是否独立？结果是否需要反复优化？这四个问题的答案，比“要不要上 Agent”更接近真正的工程入口。

Prompt Chaining先固定步骤，再逐步推进

适合提要点、排大纲、写初稿、查证、润色这一类顺序清楚的任务。

Routing先判断类型，再分到专用路径

适合客服、知识库、工单、意图分类等“入口很多、后续动作不同”的场景。

Parallelization拆成互不依赖的部分

适合研究、审查、搜索和投票式任务；子任务必须能独立完成。

Orchestrator-Workers先看目标，再动态分派

适合仓库审查、复杂调研、分工不固定的任务。

Evaluator-Optimizer让评估推动修改

适合代码、文案、报告、策略等能按明确标准检查的输出。

如果你看到...先用...还要补的一步
每一步输入输出都能提前写死Prompt Chaining加检查点，不要把自由文本直接传下去。
不同输入走不同工具Routing分类后要有专用路径和兜底路径。
多个子任务互不依赖Parallelization汇总时统一格式、来源和置信度。
任务结构会变，分工要动态决定Orchestrator-Workers把 worker 的输入和验收标准写清楚。
输出可以按规则反复挑错Evaluator-Optimizer评估标准必须具体、可执行。

读完这一节，你应该能把同一个任务从四个角度再看一遍，避免一上来就把它写成复杂 Agent。

同一工单换三种写法：什么时候会翻车

failure lab

把模式讲清楚，最好的办法是看同一个任务在不同模式下会发生什么。下面用一张客服工单来试：用户同时问账单、技术故障和退款。这个任务里，入口很多，动作不同，风险也不同。

如果你把它写成一条固定链，前面一旦分错，后面就会沿着错误分支一路跑下去；如果你只做分类，没有兜底，模糊问题就会被硬塞进某个专用路径；如果你把所有部分都并行拆开，却不管它们是否独立，最后汇总时会出现重复、冲突和错引用。这个例子最能说明：模式要和任务形状对上。

同一工单的三种组织方式Case Flow
chain:
  extract -> draft -> answer

routing:
  classify -> billing / tech / refund -> fallback

orchestrator:
  split -> assign -> merge -> verify
case: chain
symptom: 技术故障和退款被当成同一条线处理，前面一旦分错，后面整段都偏。
trace: extract 只生成统一摘要，没有把账单、技术、退款拆开。
diagnosis: 线性链缺少分流点。
repair: 先加 routing，再把高风险动作放到人工确认或单独步骤。
模式适合的任务形状翻车信号修法
Prompt Chaining步骤固定、检查点清楚前一步错了，后面一路跟错。在链中加分流点或重新设计输入输出。
Routing输入类型差异大分类后没有兜底，模糊输入被硬送进专用路径。补澄清分支、人工确认和权限检查。
Parallelization子任务彼此独立并行结果互相打架，汇总阶段重复很多。先确认依赖关系，再决定是否真的独立。
Orchestrator-Workers任务结构会变分工不清，worker 收到的信息太少。把 worker 的输入、输出和验收标准写死。
Evaluator-Optimizer输出能按规则检查评估器只会说“不错”，不能指出缺口。把来源、边界、重复和风险写成检查项。

这张表的用法很简单：先找任务形状，再找翻车信号，最后找修法。只要这三件事能对应上，你就知道自己是在选模式，不是在堆术语。

读完这一节，你应该能用一个具体工单解释为什么不同模式会翻车，以及翻车后该怎么修。

案例：同一张客服工单，四种模式怎么落地

case

假设用户发来一条工单：先问账单是否已扣款，再问页面报错怎么排查，最后又要求退费。这个输入里同时有查询、排障和高风险动作，最容易暴露“模式选错”的问题。把它直接写成一条固定链，前半段一旦分错，后面就会沿着错误方向写到底；把它直接交给自主循环，流程又会因为太自由而难以检查。

更稳的写法是先路由，再按需展开。账单类问题走账单工具，技术问题走知识库和日志，退款问题走人工确认或明确的审批步骤。只有当某个分支内部步骤很多、且结构并不固定时，才把那一支交给编排器去动态分派。这个例子能把四种模式的边界一次讲清：线性链适合顺序固定的段落，路由适合分流，编排器适合动态拆分，评估器适合最后检查答复是否能交付。

同一工单的推荐组织方式Recommended Flow
billing = route(ticket, kind="billing")
tech = route(ticket, kind="technical")
refund = route(ticket, kind="refund")

answers = [
    billing_tool.lookup(billing),
    knowledge_search.explain(tech),
    human_approval.review(refund),
]

final = evaluator.check(answers, evidence=True, risk="high")
组织方式优点最常见的坏结果修法
固定链容易实现，步骤直观。一旦前面分错，后面整段都偏。先增加分流点，再谈顺序。
纯路由入口清楚，动作可分开。模糊退款被硬塞进专用路径。每条路都要有兜底和澄清。
并行 + 汇总覆盖面广，能同时看多个角度。不同子任务互相打架，结论不一致。统一来源、格式和置信度。
编排器 + 工作者分工灵活，适合结构不固定的任务。worker 拿不到足够上下文。给 worker 固定输入、输出和验收标准。

看这个例子时，最好顺着三层来读：先看入口是不是该分流，再看分支里是不是该并行或动态拆分，最后看结果是不是还需要评估器收口。这样做的好处是，你会把“模式”当成一组可执行选择，而不是一组听起来高级的名词。

读完这一节，你应该能拿着一条真实工单，先决定先路由还是先拆分，再决定是否需要评估器收口。

Prompt Chaining：把线性任务拆成几步

chain

Prompt Chaining 的核心很朴素：任务顺序固定，每一步都看得见输入和输出。它适合“先提取、再组织、再写作、再检查”这一类工作。这里最容易做错的地方是每一步都写成自由文本，导致下一步只能猜。

工程上更稳的做法，是把每一步的产物收紧成可检查的中间结果。前一步负责提供结构，后一步只处理结构。这样一来，某一步错了，能直接回到那一步，不必怀疑整个 Agent 失灵。

线性链示例：文章生成Workflow
notes = extract_notes(source)
outline = build_outline(notes)
draft = write_draft(outline)
evidence = check_claims(draft, source)
final = revise(draft, evidence)
步骤输入输出检查点坏味道
提取原始资料要点列表是否遗漏关键事实直接把整段原文传给下一步。
组织要点列表大纲顺序是否合理大纲里混入未验证结论。
写作大纲初稿段落是否对应大纲写作阶段开始自行发挥，偏离结构。
查证初稿 + 原资料证据和修订意见每个结论是否能追到来源只让模型说“看起来对”，没有证据。
修订修订意见终稿有没有把错误和缺口补上把空话修漂亮，但事实还是没对齐。

读完这一节，你应该能把一个线性任务拆成 3 到 5 个清楚的步骤，并知道每一步该检查什么。

Routing：先判断类型，再走专用路径

routing

Routing 解决的是“同一个入口，但问题类型不同”。它先把输入分流，再把不同问题交给不同流程。这个模式的关键在于分类后真的会走不同动作。分类做完却还是同一条路，那只是换了一个名字，没有换掉流程。

Routing 也必须有兜底。输入不确定时，要么走通用流程，要么先澄清，不要把模糊输入直接送进高风险工具。尤其是退款、审批、删除这类动作，路由后仍然要有权限和参数检查。

路由规则示意Router
if input.kind == "billing":
  path: billing_tool
if input.kind == "technical":
  path: knowledge_search
if input.kind == "refund":
  path: human_approval
else:
  path: clarify_or_generalize
输入类型走哪条路为什么兜底
账单咨询账单工具需要结构化字段和可查证记录。字段缺失时先补问。
技术故障知识库 + 日志要先看证据，再给排查步骤。证据不足时先收集日志。
退款申请人工确认有副作用，不能只靠一次分类结果直接执行。金额或权限不清时暂停。
模糊问题澄清或通用流程分类信心不够时，不要强行下结论。先问一个最小澄清问题。

读完这一节，你应该能给一个客服或知识库系统设计一条清楚的分流路径和一条兜底路径。

Parallelization：并行探索更大的空间

parallel

并行有两种常见用法。第一种是分段法：把任务拆成互不依赖的部分，分别处理。第二种是投票法：对同一个问题跑多个候选答案，再汇总或取共识。两种做法都能扩大覆盖面，但前提是子任务真的能独立。

并行要看子任务是否独立。子任务如果需要频繁来回同步，并行只会制造重复、冲突和汇总噪声。真正适合并行的任务，应该能做到独立查、独立评、最后再合并。

并行研究拆分示例Parallel
lead_agent:
  facet_1 = "论文和原始方法"
  facet_2 = "工程实现和框架"
  facet_3 = "失败案例和风险"
  run_in_parallel([facet_1, facet_2, facet_3])
  synthesize(findings)
子任务是否独立输出形状汇总风险
查论文高方法、实验、结论只看论文会漏掉工程细节。
查产品高功能、接口、限制容易把宣传文案当事实。
查竞品高差异、取舍、边界结论太多时需要统一术语。
查风险高失败模式、护栏、警示如果和其他子任务强耦合，就不该硬并行。

读完这一节，你应该能判断一个任务能不能拆成几个互不依赖的并行子任务。

Orchestrator-Workers：一个编排器分派多个工作者

orchestrator

当任务结构不固定，但又能拆成若干局部工作时，就该看编排器-工作者模式。编排器负责理解目标、拆分任务、分派 worker、收回结果；worker 只做局部工作。它比普通并行更动态，因为编排器可以根据目标决定分几块、派给谁、要不要补一轮。

这个模式最怕交接太粗。worker 如果拿不到足够上下文，就会做偏；编排器如果只收到一堆散结果，也会汇总错。要让 worker 成功，必须把输入、输出和验收标准写成可检查的形状。

编排器给 worker 的最小任务单Task Split
{
  "goal": "比较三种 Agent 设计模式",
  "workers": [
    {"id": "A", "focus": "线性任务和 Prompt Chaining"},
    {"id": "B", "focus": "输入分流和 Routing"},
    {"id": "C", "focus": "并行与评估"}
  ],
  "expected_output": ["1 段结论", "1 个风险点", "1 个可复用判断"],
  "acceptance": ["术语一致", "每段有证据", "没有跨段混写"]
}
角色读什么写什么错位症状
编排器总目标、局部结果、剩余预算任务切片、顺序、回收规则拆得太碎或太粗，导致 worker 空转。
工作者局部目标、上下文、工具说明局部结果、证据、未解决问题结果太散，编排器无法汇总。
汇总层多个 worker 的局部结果统一结论、冲突说明、下一步把不同粒度的结果硬拼成一段话。

读完这一节，你应该能写出给 worker 的输入、输出和验收标准，而不是只说“帮我看看”。

Evaluator-Optimizer：让评估推动修改

evaluator

Evaluator-Optimizer 的核心是让评估标准真正可执行。生成器先产出结果，评估器再按具体标准指出缺口，生成器根据反馈修改。这个模式适合代码、报告、文案、策略和其他能被规则或样例检验的任务。

如果评估器只会说“可以”“不错”“再好一点”，这个循环就没有方向。好的评估器要能指出哪里缺来源、哪里不完整、哪里违反边界、哪里还没覆盖失败场景。没有这些具体点，优化只是在原地打转。

坏的评估反馈与好的评估反馈Review
坏：
"看起来不错，可以发布。"

好：
1. 结论没有来源。
2. 失败边界只写了成功路径。
3. 第三段和第二段重复。
4. 高风险动作缺少人工确认。
检查项看什么没写清的后果
来源每个结论是否能追到证据模型会把推断写成事实。
边界是否写出不能做什么只讲成功路径，失败时没有落点。
重复是否有无效重复和空泛拔高优化只会把空话修得更像样。
风险高风险动作是否有人确认看起来通过，实际上已经越权。

读完这一节，你应该能写出可执行的评估标准，而不是让评估器只做态度表态。

Agentic 边界：自主度增加，代价也增加

agentic

ReAct、Plan-and-Execute、ReWOO、LATS 都比普通 workflow 更接近 Agentic 设计，但它们并非越高级就越适合。它们的共同点是：让模型在更多步骤里根据观察或规划来决定下一步；它们的共同代价是：实现更复杂、成本更高、调试更难。

这就是为什么本页一直强调“先选最简单的模式”。如果任务步骤固定，workflow 往往更稳；如果任务需要环境反馈、工具调用、路径比较或回溯，再考虑更 Agentic 的方案。

模式控制方式适合什么不要误用成什么
ReAct思考、行动、观察交错需要工具反馈的问答、网页任务、环境交互固定流程的别名。
Plan-and-Execute先计划，再执行步骤大致清楚，但执行中可能调整把所有不确定性都提前写死。
ReWOO先规划工具调用和变量依赖工具依赖清楚、想减少反复推理成本和普通计划器混为一谈。
LATS搜索、评估、回溯候选路径多、需要比较和反思的任务简单任务上的默认选项。

读完这一节，你应该能说明为什么“更 Agentic”不一定更适合当前任务。

术语与误区

review
术语直觉解释容易混掉的点
Prompt Chaining把任务拆成固定顺序的多个 LLM 步骤。它通常属于 workflow。
Routing先分类输入，再走专用流程。分类后必须真的有不同处理路径。
Parallelization并行处理独立子任务或多个候选答案。子任务强依赖时就不适合硬并行。
Orchestrator-Workers编排器动态拆任务并分派给 worker。交接上下文不足会带来损耗。
Evaluator-Optimizer评估器指出问题，生成器根据反馈改进。评估标准必须具体，否则只是互相点头。
ReAct边想边做，观察会回流到下一步。它强调观察回流，不能拿来当固定步骤的代称。
误区为什么错修法
模式越复杂越强复杂模式会增加成本、延迟和调试难度。选能完成任务的最简单模式。
所有任务都适合 Agent流程固定时 workflow 更稳定。先判断控制权是否需要交给模型。
并行越多越全面子任务不独立时会制造重复和冲突。先确认能独立完成，再并行。
评估器说通过就可信评估器也会偷懒或误判。写具体标准，必要时加测试和人审。
路由只要分类准确就行分类后没有不同动作，路由没有价值。把分类标签和处理流程绑定。

读完这一节，你应该能直接说出一条模式判断和一条误判修法，而不是只背名称。

边界：本页先不追哪些内容

boundary

这一页只解决一个问题：任务形状不同，应该先选哪种模式。它不展开复杂规划树、长期记忆检索、多智能体协作，也不讲某个框架的具体 API。那些内容都重要，但要放在更合适的位置。

暂时不追的主题以后去哪里接现在先抓什么
更复杂的工具外壳和状态恢复02 Agent 工程架构先看清单个模式如何落地。
更自主的观察驱动循环01 ReAct 与 Agent 循环先明白何时需要、何时不需要。
评测、安全和更严的验收04 评测追踪与安全先学会给模式写检查点和错误语义。
多角色分工和协作协议05 MultiAgent 系统设计先别把并行调用误叫成多 Agent。

如果你要先把单个 Agent 做稳，最小闭环就是：一个清楚的任务目标、一种合适的工作流模式、一组可检查的中间结果、一个清楚的失败修法。这个闭环跑稳以后，再谈更复杂的规划、协作和自主演化，才不会把问题越做越散。

读完这一节，你应该能给自己的项目画一条“先用 workflow，后看是否升级”的路径。

练习

diagnostic

做题方式

先做能本地检查的题，再做需要判断的题。选择题和填空题用来确认模式边界；开放题用来确认你真的能把模式落到一个场景里。

自查标准

好的答案会同时说清任务形状、模式选择、检查点和失败后的下一步。只写“用 Agent”“上并行”“加评估”通常不够。

一、先选模式

先确认你能不能从任务形状直接看出 workflow 选择。

复盘提示：如果一个任务的步骤能提前写死，先从 Prompt Chaining 开始想。
choicelinear

“先提取要点，再生成大纲，再写正文，再查证”最接近哪种模式？

choicerouting

不同问题要走不同工具和不同路径时，优先考虑哪种模式？

true/falseboundary

判断：只要把流程写得更复杂，Agent 就一定更强。

true/falseparallel

判断：只要想让任务更快，就应该尽量拆成并行子任务。

二、工作流细节

这一组检查你能否把流程写成有输入、输出、检查点的链。

复盘提示：中间产物越结构化，越容易检查；如果下一步只能“猜”，前一步就没写好。
short fillworkflow

把任务拆成固定顺序多个 LLM 步骤的模式叫 ____。

orderingchain

给“生成带来源的摘要”排序：A 提取要点；B 写初稿；C 查证；D 生成大纲；E 修订。

参考答案 / 评分要点

合理顺序通常是 A 提取要点 → D 生成大纲 → B 写初稿 → C 查证 → E 修订。评分重点是说明每一步都有明确输入输出，查证和修订是链上的检查点。

choicefallback

输入不确定、分类信心也不够时，路由后的兜底应该优先是什么？

short fillcheck point

在“提取-组织-写作-____-修订”这条链里，最像检查点的是哪一步？

三、诊断与评估

这一组要求你沿链路看错。

复盘提示：如果评估器只能给态度，不能给证据，它就不能推动优化。
choicetool result

工具返回完整 HTML，模型开始引用导航栏广告。你会先改哪一类字段？

true/falseevaluation

判断：Evaluator 只说“看起来不错”就足够了。

diagnosismemory

为什么把“今天临时用的目录”写进长期记忆会出问题？写出错因、后果和修法。

参考答案 / 评分要点

错因：把一次任务状态误当成长期语义记忆。后果：下次任务继续沿用临时路径，污染新任务。修法：临时事实进任务状态或情景记忆，长期记忆只保留稳定规则，并设置来源、有效期和回退。

四、迁移与边界

最后把本页的方法迁移到一个真实场景，检查你是否真的会选模式。

复盘提示：先写模式，再写检查点，再写兜底。
choiceorchestrator

什么时候应该从 Routing 升级到 Orchestrator-Workers？

short fillreact

ReAct 属于更 agentic 的模式，因为它让模型在 ____ 后调整下一步。

scenariotransfer

为一个代码审查 Agent 选两种工作流模式，并说明为什么它们比直接上更 Agentic 的循环更合适。

参考答案 / 评分要点

好的答案通常会选 Prompt Chaining 处理固定检查流程，必要时加 Routing 把安全问题、性能问题和风格问题分流；如果仓库很大、模块很多，可以再加 Orchestrator-Workers 做动态分派。评分重点是：为什么这些模式足够，哪里需要检查点，哪里需要人工确认。
