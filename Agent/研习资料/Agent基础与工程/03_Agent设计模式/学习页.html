Agent 设计模式：按任务形状选工作流

上一单元把 Agent 放进 Harness、上下文、工具、记忆和 trace 里。本页继续往前一步：先看任务形状，再决定用哪种工作流模式。线性拆分、路由、并行、编排器-工作者、评估器-优化器，都是把模型放进可控流程的方法；只有当任务真的需要观察驱动的探索时，才值得把控制权更多交给更 Agentic 的循环。

读完本页应该能做什么

  • 能为一个工作流写出输入、输出、检查点和失败后的下一步。
  • 能识别过度设计，并知道什么时候该升级到更自主的 Agent 循环。

学习目标

goals

先看任务形状

线性、分流、并行、动态拆分、反复优化，对应的控制方式不同。你要先看任务是什么形状，再决定流程怎么写。

最后看升级条件

更复杂的模式不一定更好。只有当任务需要探索、分派、回溯或反复挑错时，才值得升到更 Agentic 的结构。

先抓主线：模式是任务形状的答案

overview

Anthropic 的工程文章给了一个很实用的起点：先选最简单的方案，能用 workflow 完成的任务就先别写成自主 Agent。对这门课来说，设计模式要回答“这个任务该由谁决定下一步”。

任务信号先选模式常见误判

读完这一节，你应该能先描述任务形状，再谈选哪种模式。

选型矩阵：先问四个问题

selection

先不要急着背模式名。看任务时，先问四个问题：步骤是否固定？输入是否可分流？子任务是否独立？结果是否需要反复优化？这四个问题的答案，比“要不要上 Agent”更接近真正的工程入口。

每一步输入输出都能提前写死Prompt Chaining加检查点，不要把自由文本直接传下去。

不同输入走不同工具Routing分类后要有专用路径和兜底处理路径。

多个子任务互不依赖Parallelization汇总时统一格式、来源和置信度。

任务结构会变，分工要动态决定Orchestrator-Workers把 worker 的输入和验收标准写清楚。

输出可以按规则反复挑错Evaluator-Optimizer评估标准必须具体、可执行。

同一工单换三种写法：什么时候会翻车

failure

把模式讲清楚，最好的办法是看同一个任务在不同模式下会发生什么。下面用一张客服工单来试：用户同时问账单、技术故障和退款。这个任务里，入口很多，动作不同，风险也不同。

案例：同一张客服工单，四种模式怎么落地

case

假设用户发来一条工单：先问账单是否已扣款，再问页面报错怎么排查，最后又要求退费。这个输入里同时有查询、排障和高风险动作，最容易暴露“模式选错”的问题。把它直接写成一条固定链，前半段一旦分错，后面就会沿着错误方向写到底；把它直接交给自主循环，流程又会因为太自由而难以检查。

Prompt Chaining：把线性任务拆成几步

chain

Prompt Chaining 的核心很朴素：任务顺序固定，每一步都看得见输入和输出。它适合“先提取、再组织、再写作、再检查”这一类工作。这里最容易做错的地方是每一步都写成自由文本，导致下一步只能猜。

读完这一节，你应该能把一个线性任务拆成 3 到 5 个清楚的步骤，并知道每一步该检查什么。

Routing：先判断类型，再走专用路径

routing

Routing 解决的是“同一个入口，但问题类型不同”。它先把输入分流，再把不同问题交给不同流程。这个模式的关键在于分类后真的会走不同动作。分类做完却还是同一条路，那只是换了一个名字，没有换掉流程。

路由规则示意Router

输入类型走哪条路为什么兜底处理

读完这一节，你应该能给一个客服或知识库系统设计一条清楚的分流路径和一条兜底处理路径。

Parallelization：并行探索更大的空间

parallel

并行有两种常见用法。第一种是分段法：把任务拆成互不依赖的部分，分别处理。第二种是投票法：对同一个问题跑多个候选答案，再汇总或取共识。两种做法都能扩大覆盖面，但前提是子任务真的能独立。

并行研究拆分示例Parallel

子任务是否独立输出形状汇总风险

读完这一节，你应该能判断一个任务能不能拆成几个互不依赖的并行子任务。

Orchestrator-Workers：一个编排器分派多个工作者

orchestrator

当任务结构不固定，但又能拆成若干局部工作时，就该看编排器-工作者模式。编排器负责理解目标、拆分任务、分派 worker、收回结果；worker 只做局部工作。它比普通并行更动态，因为编排器可以根据目标决定分几块、派给谁、要不要补一轮。

编排器给 worker 的最小任务单Task Split

角色读什么写什么错位症状

读完这一节，你应该能写出给 worker 的输入、输出和验收标准，而不是只说“帮我看看”。

Evaluator-Optimizer：让评估推动修改

evaluator

Evaluator-Optimizer 的核心是让评估标准真正可执行。生成器先产出结果，评估器再按具体标准指出缺口，生成器根据反馈修改。这个模式适合代码、报告、文案、策略和其他能被规则或样例检验的任务。

读完这一节，你应该能写出可执行的评估标准，而不是让评估器只做态度表态。

Agentic 边界：自主度增加，代价也增加

agentic

ReAct、Plan-and-Execute、ReWOO、LATS 都比普通 workflow 更接近 Agentic 设计，但它们并非越高级就越适合。它们的共同点是：让模型在更多步骤里根据观察或规划来决定下一步；它们的共同代价是：实现更复杂、成本更高、调试更难。

边界：本页先不追哪些内容

boundary

这一页只解决一个问题：任务形状不同，应该先选哪种模式。它不展开复杂规划树、长期记忆检索、多智能体协作，也不讲某个框架的具体 API。那些内容都重要，但要放在更合适的位置。

01choice

Q: “先提取要点，再生成大纲，再写正文，再查证”最接近哪种模式？

参考答案 / 评分要点

Prompt Chaining。步骤固定、检查点清楚时，先用 Prompt Chaining。

02choice

Q: 不同问题要走不同工具和不同路径时，优先考虑哪种模式？

参考答案 / 评分要点

Routing。输入类型差异大、后续动作不同时，先做 Routing。

03true/false

Q: 判断：只要把流程写得更复杂，Agent 就一定更强。

参考答案 / 评分要点

错。更复杂的模式不一定更强，成本和排错难度都会上升。

03btrue/false

Q: 判断：只要想让任务更快，就应该尽量拆成并行子任务。

参考答案 / 评分要点

错。子任务不独立时，不该硬拆成并行。

二、工作流细节

复盘提示：中间产物越结构化，越容易检查；如果下一步只能“猜”，前一步就没写好。 -->
04short-fill

Q: 把任务拆成固定顺序多个 LLM 步骤的模式叫 ____。

参考答案 / 评分要点

Prompt Chaining。固定顺序、多步串联的模式就是 Prompt Chaining。

05ordering

Q: 给“生成带来源的摘要”排序：A 提取要点；B 写初稿；C 查证；D 生成大纲；E 修订。

参考答案 / 评分要点

。合理顺序通常是 A 提取要点 → D 生成大纲 → B 写初稿 → C 查证 → E 修订。评分重点是说明每一步都有明确输入输出，查证和修订是链上的检查点。

06choice

Q: 输入不确定、分类信心也不够时，路由后的兜底处理应该优先是什么？

参考答案 / 评分要点

澄清或人工确认。Routing 的关键不是分类，而是分类后有清楚的专用路径和兜底处理路径。

06bshort-fill

Q: 在“提取-组织-写作-____-修订”这条链里，最像检查点的是哪一步？

参考答案 / 评分要点

查证。查证和修订是线性链里最典型的检查点。

三、诊断与评估

复盘提示：如果评估器只能给态度，不能给证据，它就不能推动优化。 -->
07choice

Q: 工具返回完整 HTML，模型开始引用导航栏广告。你会先改哪一类字段？

参考答案 / 评分要点

summary。给模型的结果应优先保留摘要、证据、错误类型和下一步，而不是原始长输出。

08true/false

Q: 判断：Evaluator 只说“看起来不错”就足够了。

参考答案 / 评分要点

错。评估器必须给出可执行标准，不能只说“不错”。

09error diagnosis

Q: 为什么把“今天临时用的目录”写进长期记忆会出问题？写出错因、后果和修法。

参考答案 / 评分要点

。错因：把一次任务状态误当成长期语义记忆。后果：下次任务继续沿用临时路径，污染新任务。修法：临时事实进任务状态或情景记忆，长期记忆只保留稳定规则，并设置来源、有效期和回退。

四、迁移与边界

复盘提示：先写模式，再写检查点，再写兜底。 -->
10choice

Q: 什么时候应该从 Routing 升级到 Orchestrator-Workers？

参考答案 / 评分要点

动态拆分。当任务结构不固定，需要根据目标动态拆分和分派时，才考虑 Orchestrator-Workers。

11short-fill

Q: ReAct 属于更 agentic 的模式，因为它让模型在 ____ 后调整下一步。

参考答案 / 评分要点

观察。ReAct 的关键是观察回流，模型根据环境反馈修正下一步。

12scenario transfer

Q: 为一个代码审查 Agent 选两种工作流模式，并说明为什么它们比直接上更 Agentic 的循环更合适。

参考答案 / 评分要点

。好的答案通常会选 Prompt Chaining 处理固定检查流程，必要时加 Routing 把安全问题、性能问题和风格问题分流；如果仓库很大、模块很多，可以再加 Orchestrator-Workers 做动态分派。评分重点是：为什么这些模式足够，哪里需要检查点，哪里需要人工确认。
