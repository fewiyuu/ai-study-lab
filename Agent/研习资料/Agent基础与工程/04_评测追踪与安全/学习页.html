评测、追踪与安全：把行为纳入验收

上一单元把工作流模式选清楚了，这一页继续往下收口：先把 task、trial、grader、transcript 和 outcome 拆开，再看 Trace 怎么保住过程证据，最后把安全边界落到 shell、文件、消息和 computer use 这些会产生外部动作的地方。你读到最后，应该能为一个 Agent 任务写出最小评测方案、最小 Trace 字段和最小护栏，而不是只盯着最终回答。

学习目标

goals

先把结果说清楚

你要能区分 task、trial、grader、transcript 和 outcome，知道谁负责定义目标，谁负责记录运行，谁负责评分，谁负责保存最终状态。

最后把边界卡住

你要能给 shell、文件写入、消息发送或 computer use 这类动作写出最小安全护栏，而不是只在提示词里写一句“请谨慎”。

先看主线：从结果到过程，再到边界

overview

这页按一个固定顺序往下走：先看结果是否可验收，再看过程是否可追踪，最后看动作是否越界。这个顺序比“先把工具接上，再补评测，最后再想安全”更稳，因为一旦系统开始自动运行，后补的内容往往只是在补记录，而不是补控制。

它完成了吗？outcome测试通过、文件改对、数据库状态正确只看文本，可能把“说完成”误当完成。

它为什么这么做？transcript / trace模型消息、工具调用、状态变化、错误只能看结果，无法复盘偏差从哪一步开始。

它有没有越界？permission / safety decision路径检查、确认弹窗、审计日志、拦截事件模型一旦失误，越界动作会直接落地。

评测task / trial / grader任务定义、运行配置、评分规则任务结果和回归趋势改了 prompt 以后，没人知道变好还是变坏。

追踪trace / transcript工具输入、输出摘要、状态变化、错误每一步发生了什么只留一句最终答复，排错时找不到分叉点。

安全permission / sandbox允许范围、确认规则、拦截事件当前动作是否能执行模型一句话说错，shell、文件或消息已经执行了。

评测结构：task、trial、grader、outcome

harness

最小评测单位可以拆成四层。task 是你要测什么，trial 是一次运行，grader 是怎么判定，outcome 是环境最后变成了什么。这个拆法的价值在于：你可以重复同一个 task，多次 trial，再看 grading 是否稳定，而不是被一次偶然的好结果骗过去。

Trace：记录过程，不只记录结果

trace

Trace 要解决的是复盘问题。你不仅想知道它最后对不对，还想知道它是哪一步开始错的、哪次工具返回改变了方向、哪一步把状态写坏了。只存一大段聊天文本不够，后续很难按工具、错误或成本做查询。

失败实验室：同一条 bug，在三层里怎么坏

failure-lab

最容易学会这页内容的方式，看同一个 bug 在评测、追踪和安全里分别怎么坏。下面用“登录超时后没有重试”这个任务做一个小实验：如果任务定义太宽，grader 可能根本不知道该看什么；如果 Trace 太粗，你只能看到“失败了”，看不到失败从哪一步开始；如果安全边界缺失，Agent 可能为了修复 bug 直接动到仓库外的文件。

评测任务看上去都像过了成功条件太宽，grader 没有硬边界收紧 task，固定 outcome 和改动范围。

追踪失败后说不出是哪里坏的Trace 只存聊天文本，没有结构化事件加 tool_call、tool_result、state_delta 和 error。

安全模型一失误就越权写入程序没有工作区和确认拦截把 permission 和 sandbox 变成硬约束。

事件形状追踪：从一条运行看到四个层级

shape

这个小追踪器把同一个任务的四个层级摆在一起：评测对象、追踪事件、安全状态和最终结果。你可以把它当成一个结构地图：当 task 写得太宽时，失败首先出现在评测层；当工具返回误导时，失败先出现在追踪层；当程序没拦住动作时，失败先出现在安全层。它的价值在于把“看起来都差不多”的问题分开。

评测层定义是否够清楚task、trial、grader目标模糊，分数就没意义。

追踪层执行路径是否完整tool_call、tool_result、state_delta只有最终回答，没有过程证据。

安全层动作是否被允许permission、sandbox、confirm越界动作直接发生，没有拦截记录。

结果层环境最后变成什么outcome、tests、diff答复听起来对，状态其实不对。

监控与抽查：离线评测之外的第二层眼睛

monitoring

离线评测能防回归，但真实输入会更乱。线上监控的作用，是从真实运行里抽样，看成功率、失败类型、工具误用、成本、用户接管和安全拦截。它能抓到离线集合没覆盖的边界，但前提是你把抽样、脱敏和权限一起设计好。

回流闭环：线上问题怎样变成下一轮评测

feedback-loop

线上监控抓到一条异常后，下一步把这条异常整理成一条可复现的评测样例。你要先保留最小上下文：原始输入、关键工具调用、失败输出、风险状态和用户最后看到的结果。这样做的目的很直接，后面的回归测试才能真正固定住同一类问题。

安全边界：先于功能，不是上线后补

safety

一旦 Agent 能调用 shell、写文件、发消息、提交代码或控制界面，它就会产生外部动作。安全边界的任务很简单：谁能用，能在哪用，能做什么，做了什么如何追踪，哪些动作必须再确认。真正起作用的是程序侧的硬约束，prompt 只负责提醒，不负责兜底处理。

案例：代码修复 Agent 的验收链

这里先不追：下一单元再接什么

boundary

这页只收最小闭环：评测、追踪、安全。更细的监控策略、完整的线上告警、跨任务的归因分析，以及多 Agent 级别的分工协作，都不在这一页展开。你现在先把“能验收、能复盘、能拦住”这三件事做实，后面的系统设计才有底。

Q: 用 4-6 句话解释：为什么 Agent 评测不能只看最终回答？

。好的答案要同时提到工具、环境结果和安全边界。核心意思是：最终回答只能说明模型说了什么，不能说明它有没有真的执行、执行得对不对、有没有越权、有没有把状态写坏。最好再补一句：评测要看 task / trial / grader，追踪要看 trace，安全要看 permission 和 confirm。
