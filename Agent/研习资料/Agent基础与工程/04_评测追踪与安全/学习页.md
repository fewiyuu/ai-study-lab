# 学习目标 {#goals}
<!-- tag: goals -->

先把结果说清楚
你要能区分 task、trial、grader、transcript 和 outcome，知道谁负责定义目标，谁负责记录运行，谁负责评分，谁负责保存最终状态。


再把过程留下来
你要能读一条结构化 Trace，找出工具失败、状态误写、成本上升和安全拦截分别落在哪些字段上。


最后把边界卡住
你要能给 shell、文件写入、消息发送或 computer use 这类动作写出最小安全护栏，而不是只在提示词里写一句“请谨慎”。

本页边界：这里只讲把行为纳入验收的最小工程闭环，不展开完整评测平台，也不讲多 Agent 协作。

# 先看主线：从结果到过程，再到边界 {#overview}
<!-- tag: overview -->

这页按一个固定顺序往下走：先看结果是否可验收，再看过程是否可追踪，最后看动作是否越界。这个顺序比“先把工具接上，再补评测，最后再想安全”更稳，因为一旦系统开始自动运行，后补的内容往往只是在补记录，而不是补控制。

要回答的问题看什么典型证据漏掉会怎样


它完成了吗？outcome测试通过、文件改对、数据库状态正确只看文本，可能把“说完成”误当完成。
它为什么这么做？transcript / trace模型消息、工具调用、状态变化、错误只能看结果，无法复盘偏差从哪一步开始。
它有没有越界？permission / safety decision路径检查、确认弹窗、审计日志、拦截事件模型一旦失误，越界动作会直接落地。

层级核心对象写入什么读什么常见症状


评测task / trial / grader任务定义、运行配置、评分规则任务结果和回归趋势改了 prompt 以后，没人知道变好还是变坏。
追踪trace / transcript工具输入、输出摘要、状态变化、错误每一步发生了什么只留一句最终答复，排错时找不到分叉点。
安全permission / sandbox允许范围、确认规则、拦截事件当前动作是否能执行模型一句话说错，shell、文件或消息已经执行了。

读完这一节，你应该能用一句话区分“结果能验收”“过程能追踪”“动作能拦住”三件事。

# 评测结构：task、trial、grader、outcome {#harness}
<!-- tag: harness -->

最小评测单位可以拆成四层。task 是你要测什么，trial 是一次运行，grader 是怎么判定，outcome 是环境最后变成了什么。这个拆法的价值在于：你可以重复同一个 task，多次 trial，再看 grading 是否稳定，而不是被一次偶然的好结果骗过去。

对代码 Agent 来说，outcome 往往比回答文本更重要。它测试是否通过、文件改得对不对、有没有碰到不该动的路径。对写作或研究 Agent 来说，outcome 可能是摘要质量、引文覆盖率、结构是否完整，但也一样需要明确的判定点。

最小评测样例YAML复制

```yaml
task:
 id: fix-login-timeout
 goal: 修复登录超时后没有重试的问题
 success:
 - npm test passes
 - only auth module changed
trial:
 agent: coding-agent-v2
 model: claude-opus
grader:
 checks:
 - run tests
 - inspect changed files
 - compare behavior with expected scenario
```

对象它负责什么读者该检查什么坏结果长什么样


task定义目标和成功条件目标是否可验证，边界是否写清任务描述很长，但没有可判定标准。
trial记录一次运行模型、工具版本、预算、时间只知道结果，不知道这次怎么跑的。
grader把结果变成分数或判定检查项是否能执行，是否覆盖真实风险只会给“看起来不错”这样的空判断。
outcome环境最终状态测试、diff、状态变更是否对齐目标回答听起来对，但环境其实没对齐。

读完这一节，你应该能写出一个 task，并说清楚它的 grader 到底检查什么。

# Trace：记录过程，不只记录结果 {#trace}
<!-- tag: trace -->

Trace 要解决的是复盘问题。你不仅想知道它最后对不对，还想知道它是哪一步开始错的、哪次工具返回改变了方向、哪一步把状态写坏了。只存一大段聊天文本不够，后续很难按工具、错误或成本做查询。

结构化 Trace 至少要记录 step id、event type、tool name、tool input、tool output summary、state delta、latency、cost、error 和 safety decision。对流程复杂的任务，这些字段比长对话更有用，因为它们能直接暴露“读了什么、写了什么、在哪里停住、谁拦住了它”。

Trace 事件示例JSON复制

```json
{
 "runId": "trial-042",
 "step": 3,
 "event": "tool_call",
 "tool": "shell",
 "inputSummary": "npm test",
 "outputSummary": "2 tests failed in auth retry",
 "stateDelta": {"phase": "diagnosing_failure"},
 "latencyMs": 4280,
 "costUsd": 0.014,
 "safety": "allowed"
}
```

字段它告诉你什么常见误读下一步该看什么


tool / inputSummary它调用了哪个工具、传了什么意图只看最终答案，忘了工具路径检查工具参数和调用顺序。
outputSummary / error工具返回了什么、是成功还是失败把失败当成模型胡说看失败是工具报错还是参数不对。
stateDelta状态被写成了什么状态变化被藏在长文本里检查状态是否在正确阶段更新。
latency / cost这一步花了多少时间和预算只在出错后才看成本比较不同步骤的耗时和预算。
safety动作有没有被允许把安全边界当成提示词看是谁拦住了动作、有没有审计记录。

读完这一节，你应该能拿一条 Trace，指出工具失败点、状态写入点和安全拦截点。

# 失败实验室：同一条 bug，在三层里怎么坏 {#failure-lab}
<!-- tag: failure-lab -->

最容易学会这页内容的方式，看同一个 bug 在评测、追踪和安全里分别怎么坏。下面用“登录超时后没有重试”这个任务做一个小实验：如果任务定义太宽，grader 可能根本不知道该看什么；如果 Trace 太粗，你只能看到“失败了”，看不到失败从哪一步开始；如果安全边界缺失，Agent 可能为了修复 bug 直接动到仓库外的文件。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

层症状故障原因修法


评测任务看上去都像过了成功条件太宽，grader 没有硬边界收紧 task，固定 outcome 和改动范围。
追踪失败后说不出是哪里坏的Trace 只存聊天文本，没有结构化事件加 tool_call、tool_result、state_delta 和 error。
安全模型一失误就越权写入程序没有工作区和确认拦截把 permission 和 sandbox 变成硬约束。

读完这个实验，你应该能把一个坏结果拆成三类故障，而不是把所有问题都怪给模型。

# 事件形状追踪：从一条运行看到四个层级 {#shape}
<!-- tag: shape -->

这个小追踪器把同一个任务的四个层级摆在一起：评测对象、追踪事件、安全状态和最终结果。你可以把它当成一个结构地图：当 task 写得太宽时，失败首先出现在评测层；当工具返回误导时，失败先出现在追踪层；当程序没拦住动作时，失败先出现在安全层。它的价值在于把“看起来都差不多”的问题分开。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

层级要看什么该读什么坏结果怎么暴露


评测层定义是否够清楚task、trial、grader目标模糊，分数就没意义。
追踪层执行路径是否完整tool_call、tool_result、state_delta只有最终回答，没有过程证据。
安全层动作是否被允许permission、sandbox、confirm越界动作直接发生，没有拦截记录。
结果层环境最后变成什么outcome、tests、diff答复听起来对，状态其实不对。

读完这一节，你应该能从“评测、追踪、安全、结果”四个层级里认出问题落点。

# 监控与抽查：离线评测之外的第二层眼睛 {#monitoring}
<!-- tag: monitoring -->

离线评测能防回归，但真实输入会更乱。线上监控的作用，是从真实运行里抽样，看成功率、失败类型、工具误用、成本、用户接管和安全拦截。它能抓到离线集合没覆盖的边界，但前提是你把抽样、脱敏和权限一起设计好。

这里最重要的习惯是：评测、监控和告警共用同一条事件流。不要让每个模块各记各的，否则你最后会得到三份对不上账的日志。事件流里至少要有 tool_call、tool_result、state_update、stop_reason 和 safety_block 这些事件。

事件谁写谁读为什么要单独记


tool_call执行层评测、监控、审计能看出调用了什么、何时调用、是否越权。
tool_result执行层评测、监控、调试能看出返回值是否误导了后续步骤。
state_update业务层复盘、恢复、审计能看出任务状态有没有被写坏。
stop_reason控制层监控、告警、人审能看出是完成、失败、超预算还是被拦截。
safety_block安全层安全、审计、复盘能看出谁拦住了高风险动作。

读完这一节，你应该能设计一个最小抽查方案，说清抽什么、谁能看、怎么脱敏。

# 回流闭环：线上问题怎样变成下一轮评测 {#feedback-loop}
<!-- tag: feedback-loop -->

线上监控抓到一条异常后，下一步把这条异常整理成一条可复现的评测样例。你要先保留最小上下文：原始输入、关键工具调用、失败输出、风险状态和用户最后看到的结果。这样做的目的很直接，后面的回归测试才能真正固定住同一类问题。

这个回流动作有一个常见误区：把事故摘要直接放进工单就结束了。工单适合协作，回归集才适合防复发。前者回答“发生了什么”，后者回答“下次还会不会再来”。

从线上抓到什么整理成什么回归时检查什么常见漏项


一次失败请求最小复现 task同类输入是否还会失败只记结论，不记上下文。
关键工具事件结构化 trace失败从哪一步开始只保留聊天文本。
越权或高风险动作safety_block / confirm 记录程序是否真的拦住了把拦截写成提示语。
最终用户结果outcome 摘要结果是否和成功条件一致只看模型自述，不看环境状态。

回流样例JSON复制

```json
{
 "task": "login timeout retry",
 "input": "timeout after 3s",
 "trace": ["tool_call: open file", "tool_result: timeout", "state_delta: retry_count=0"],
 "outcome": "test failed",
 "risk": "needs_confirm",
 "regression_check": "retry once and keep changes inside auth module"
}
```

这条样例进入回归集之后，下一轮评测就能直接问三个问题：它还会不会失败、Trace 能不能定位、护栏有没有拦住越界动作。只要这三问里有一问答不上来，这条线上问题就还没有真正关掉。

读完这一节，你应该能把线上事故整理成一条可回归的样例，而不是只留一条工单摘要。

# 安全边界：先于功能，不是上线后补 {#safety}
<!-- tag: safety -->

一旦 Agent 能调用 shell、写文件、发消息、提交代码或控制界面，它就会产生外部动作。安全边界的任务很简单：谁能用，能在哪用，能做什么，做了什么如何追踪，哪些动作必须再确认。真正起作用的是程序侧的硬约束，prompt 只负责提醒，不负责兜底处理。

最常见的护栏包括白名单、工作区隔离、路径检查、危险命令拦截、dry-run、人工确认、预算限制、速率限制和审计日志。对 shell 工具来说，`../`、符号链接、绝对路径和破坏性命令都要特别小心；对发消息或退款这类动作，确认级别往往更高。

路径检查示例TypeScript-like复制

```text
const WORKSPACE = path.resolve("/workspace/project");

async function assertInsideWorkspace(cwd) {
 const real = await fs.realpath(path.resolve(cwd ?? WORKSPACE));
 const rel = path.relative(WORKSPACE, real);
 if (rel.startsWith("..") || path.isAbsolute(rel)) {
 throw new Error("outside workspace");
 }
 return real;
}
```

护栏读什么写什么症状


工作区隔离真实路径、符号链接解析结果允许访问的根目录一旦路径越界，文件就可能被改到仓库外。
人工确认高风险动作的摘要确认记录、拒绝记录退款、提交、删除类动作直接落地。
预算限制token、时间、步骤数停止原因Agent 在错误方向上一直跑，不知道停。
审计日志谁发起、谁批准、谁执行可追溯记录出了问题找不到责任链。

读完这一节，你应该能给一个 shell 权限的 Agent 画出最小护栏，而不是只写一句“请谨慎操作”。

# 案例：代码修复 Agent 的验收链 {#case-study}
<!-- tag: case-study -->

把前面几层合起来看，最容易懂的是代码修复任务。一个合理的 task 会写清楚 bug、fixture 和成功条件；trial 负责记录这次运行用的模型、工具和预算；grader 先跑测试，再检查改动范围，最后确认没有引入新的高风险副作用。Trace 则告诉你它是先定位错误、先跑测试还是先改文件。

这里最关键的“怎么证明修好了”。如果测试通过，但它顺手删了不该删的文件，或者为了过测试绕开了权限边界，那 outcome 也不能算真正通过。所以在代码 Agent 里，结果检查、过程追踪和安全护栏必须一起看。

代码修复验收链Workflow复制

```python
task = {
 "goal": "fix login timeout retry",
 "success": ["npm test passes", "only auth module changed"]
}

trial = run_agent(task, model="claude-opus", budget="small")
trace = trial.trace
outcome = trial.outcome
grader = {
 "tests": outcome.tests_passed,
 "files": outcome.changed_files,
 "risk": outcome.safety_state,
}
```

层检查点如果错了会怎样下一步看什么


task目标和成功条件是否可验证任务写得很热闹，但没法判定对错补测试、补文件范围、补风险边界。
trace先做了什么、为什么转向失败之后不知道偏差是从哪一步开始的看工具调用顺序和状态写入。
outcome测试、diff、风险状态回答看起来对，环境其实没对齐检查自动测试和文件修改范围。
grader是否把硬结果和风险一起看评分器只会给印象分把规则、阈值和人工确认写进检查项。

读完这个例子，你应该能说清一个代码 Agent 的验收链需要哪些证据，缺哪一层会出什么问题。

# 误区：看起来自动，实际上不可控 {#pitfalls}
<!-- tag: pitfalls -->

误区为什么错修法


只看最终回答过程越权或工具没跑也看不出来。保存 transcript 和 outcome。
没有评测先改 prompt你不知道是变好还是变坏。先建小任务集和回归记录。
安全只写在 prompt模型一旦失误，边界就失效。程序侧做权限、路径和确认。
Trace 只存聊天文本后续无法按工具、错误或成本查询。改成结构化事件流。

症状可能原因先看哪里修法方向


分数忽高忽低任务定义不稳定或 trial 太少task / trial / budget固定任务边界，增加重复运行。
很难复盘为什么失败Trace 太粗tool_call / state_delta / error把过程变成结构化事件。
高风险动作误执行安全边界只写在 prompt 里permission / sandbox / confirm程序侧加硬拦截和审计。
监控看起来都挺好，但用户还是抱怨只看离线集，不看线上抽查事件流和抽样策略把在线抽查接进同一套记录。

读完这一节，你应该能见到一个坏结果时先判断是评测、追踪还是安全出了问题。

# 这里先不追：下一单元再接什么 {#boundary}
<!-- tag: boundary -->

这页只收最小闭环：评测、追踪、安全。更细的监控策略、完整的线上告警、跨任务的归因分析，以及多 Agent 级别的分工协作，都不在这一页展开。你现在先把“能验收、能复盘、能拦住”这三件事做实，后面的系统设计才有底。

现在先学先别追后面会回来


task / trial / grader完整 benchmark 平台更大规模的课程和评测工作流
trace / transcript / outcome复杂归因和统计建模后续监控与可靠性单元
permission / sandbox / confirm完整安全政策后续工具调用和多 Agent 设计

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
<!-- end-practice-intro -->

**概念**知道 task、trial、grader、outcome 的分工。
**追踪**知道 Trace 该记录什么。
**边界**知道安全护栏放在哪一层。
**迁移**能把这套方法搬到真实任务。

<!-- practice-group: 一、先把评测对象分清 -->
<!-- quick-review: 复盘提示 先问“测什么”，再问“怎么跑”，最后问“怎么判”。 -->

### 01 choice

**Q:** “测什么”在 Agent 评测里对应哪个对象？

- [ ] task
- [ ] trial
- [ ] trace

Answer: task

### 02 true/false

**Q:** 判断：只要最终回答正确，就不必看 outcome。

- [ ] 对
- [ ] 错

Answer: 错

### 03 short-fill

**Q:** 同一个 task 的 ____ 叫 trial。

Answer: 一次运行

### 04 ordering

**Q:** 按顺序排列：A 定义 task；B 跑 trial；C grader 检查；D 汇总 outcome。

Explain: 典型顺序是 A → B → C → D。评分时重点看你有没有说清楚 grader 依赖 outcome，而不是依赖一段好看的最终回答；也要提到 repeated trial 才能看稳定性。

<!-- end-practice-group -->

<!-- practice-group: 二、读 Trace 和监控事件 -->
<!-- quick-review: 复盘提示 先看工具，再看状态，再看成本和安全。 -->

### 05 choice

**Q:** 读一条失败 Trace 时，最先该看哪三类字段？

- [ ] tool / error / stateDelta
- [ ] title / author / date
- [ ] emoji / tone / layout

Answer: tool / error / stateDelta

### 06 true/false

**Q:** 判断：一大段聊天文本就足够当 Trace。

- [ ] 对
- [ ] 错

Answer: 错

### 07 code-reading

**Q:** 看这条事件：`tool=input_search`、`outputSummary=404`、`stateDelta.phase=diagnosing_failure`。你能判断出什么？还缺什么证据？

Explain: 应能判断它调用了搜索工具、返回失败、状态已经进入排错阶段。还缺的证据通常是：输入到底查了什么、后续有没有改路线、失败是不是来自权限或参数而不是目标本身。好的答案会把 tool、output、state 三者串起来。

### 08 true/false

**Q:** 判断：高风险动作的边界应该主要由程序执行，而不是只靠 prompt 提醒。

- [ ] 对
- [ ] 错

Answer: 对

### 09 short-fill

**Q:** `path.relative` 和 `realpath` 这一类检查，主要是为了 ____。

Answer: 防止越出工作区

### 10 scenario

**Q:** 给一个能发消息和改文件的 Agent，写出 3 条最小护栏。每条都要说清楚“拦什么、为什么拦、看什么证据”。

Explain: 合格答案通常会包含：工作区隔离或路径白名单，防止越界写文件；高风险动作二次确认，防止误发消息或误提交；审计日志或拦截事件，便于追责和复盘。要点是每条都要有对象、动作和证据，而不是只写“请小心”。

<!-- end-practice-group -->

<!-- practice-group: 四、迁移到真实任务 -->
<!-- quick-review: 复盘提示 先写证据链，再写动作链。 -->

### 11 scenario

**Q:** 为“自动修复 bug”的 Agent 设计一个最小 evaluation suite：写 2 个 task 和对应 grader。

Explain: 好的答案会给出可验证任务，比如“修复登录超时重试”和“修复文件导入失败”，并给出 grader：测试是否通过、改动范围是否正确、是否引入额外风险。不要只写“看修好了没有”，要写清楚检查项。

### 12 choice

**Q:** 做线上抽查时，最值得保留的事件组合是哪一组？

- [ ] tool_call / tool_result / safety_block
- [ ] title / author / summary
- [ ] tone / font / color

Answer: tool_call / tool_result / safety_block

### 13 synthesis

**Q:** 用 4-6 句话解释：为什么 Agent 评测不能只看最终回答？

Explain: 好的答案要同时提到工具、环境结果和安全边界。核心意思是：最终回答只能说明模型说了什么，不能说明它有没有真的执行、执行得对不对、有没有越权、有没有把状态写坏。最好再补一句：评测要看 task / trial / grader，追踪要看 trace，安全要看 permission 和 confirm。
