Multi-Agent 系统设计：什么时候拆、怎么交接、怎么验

本页把 Multi-Agent 当成一笔工程账来算。你会沿着一个“客户流失原因研究报告”的小例子，看单 Agent 在哪里卡住、哪些任务适合隔离或并行、交接格式怎么写、验证者怎么把关；长期记忆、A2A/MCP、完整分布式调度平台和多轮组织治理先放到后面。

课程地图上一单元：评测追踪与安全

读完本页后先能做什么

  • 能先找证据，再决定单 Agent 是继续做，还是拆成多个 Agent。
  • 能把 lead、研究子 Agent 和 verifier 的输入、工具、输出、禁止项和升级规则写清楚。
  • 能顺着交接损耗、幻觉放大、信任升级、成本失控和 early victory 定位常见故障。

学习目标

goals

先会判断是否该拆

多 Agent 的第一步是确认单 Agent 已经碰到真实边界：上下文被污染、搜索空间太宽、工具权限冲突，或结果需要独立验证。

再会写交接协议

你要能写清子 Agent 收到什么、能用什么工具、返回什么字段、不能做什么，以及遇到低置信度或来源冲突时如何升级。

最后会验收失败点

你要能识别交接损耗、幻觉放大、trust escalation、成本失控和 early victory，并知道应该回到哪张表或哪条 Trace 检查。

这一节先把判断框架钉住：不是先数角色名，而是先看证据够不够。

判断主线：Multi-Agent 是拿成本换能力

overview

Anthropic 的 Agent 文章有一个很实用的起点：先从最简单的系统做起，能用 workflow 或单 Agent 解决的，就不要急着把任务拆成一群角色。多 Agent 会带来重复上下文、协调消息、结果汇总、权限边界和调试成本。它值不值得，取决于这些成本能不能换来可见能力。

这页使用一个贯穿案例：你要让 Agent 为“某个产品的客户流失原因”写一份研究报告。单 Agent 可以搜索、阅读、总结；当资料很多、方向很多、工具权限很多时，单 Agent 可能开始丢重点、重复搜索、引用未验证来源。多 Agent 的价值来自明确分工：搜索、压缩、专门工具和验证分别有边界。

先问的问题适合继续单 Agent适合考虑 Multi-Agent证据
上下文是否撑得住资料少，所有事实都需要同一上下文判断大量资料互不相关，需要先过滤再汇总主上下文反复塞入无关网页、日志或订单。
搜索空间是否可并行下一步强依赖上一轮结果多个方向可以同时查，最后再汇总研究、竞品、风险、实现方案能独立探索。
工具面是否冲突工具少，权限差异小不同子任务需要不同工具和禁止项搜索者只读网页，代码者能改文件，验证者只能跑测试。
是否需要独立验证结果可由同一流程直接检查生成者容易自证通过，需要另一套检查视角来源覆盖、测试结果、安全边界需要单独验收。

这张表的用法很简单：四列里至少有一列出现清楚证据，再谈拆分。没有证据时，先改工具、上下文整理、任务边界和评测，往往比多加一个 Agent 更有效。

先看收益证据，再谈拆分；证据不清楚，就先别加角色。

拆分决策：先过四个闸门，再决定拆不拆

decision

把“是否拆分”变成一个闸门流程，会比凭感觉判断稳定。先让单 Agent 走一轮，记录它在哪里失败：是上下文太满，还是搜索路径太窄，是工具太多导致误用，还是结果缺少独立验证。只有失败点能被拆分修复，才进入多 Agent 设计。

单 Agent 先跑workflow / single agent

先确认更好的提示、工具、上下文压缩和评测不能解决问题。这个步骤防止把可控问题拆成协作问题。

收益证据context / search / tools / verification

拆分必须对应具体收益：隔离噪声、扩大搜索、缩窄工具权限，或让验证者独立检查主输出。

边界清楚input / tools / output / forbidden

每个 Agent 都要有输入、工具、输出和禁止项。边界不清时，多 Agent 只会把问题变成电话游戏。

验收可查trace / checklist / escalation

交接结果必须能追踪和验证。没有验收规则时，lead Agent 只能凭风格判断子 Agent 是否靠谱。

闸门必须留下的证据通过后才能做什么没通过时怎么处理
单 Agent 先跑失败 Trace、成本、上下文长度、错误类型讨论拆分收益先改工具、任务边界或上下文整理。
收益证据上下文污染、宽搜索、工具冲突、验证缺口之一定义子 Agent 角色继续单 Agent 或 workflow。
边界清楚输入、工具、输出、禁止项、升级条件写交接协议缩小任务，直到可委派。
验收可查Trace 字段、检查清单、失败升级路径上线或扩展协作先补验证者或人工 review。

先过这四个闸门，能挡住大多数为了“看起来高级”而拆分的冲动。

上下文保护：让噪声留在子上下文里

context

上下文保护是多 Agent 最常见的合理理由。主 Agent 负责最终判断，不该把所有网页、订单、日志、论文段落都吞进去。子 Agent 可以在独立上下文里读脏材料，只把压缩后的事实、来源、置信度和未解决问题交回来。

关键点是“压缩后返回”。如果子 Agent 把全文、截图、长日志原样转回来，主 Agent 仍然被污染，只是污染晚了一步。一个好的研究子 Agent 返回可检查的事实卡：短 claim、来源、置信度、证据和未解决问题。

子 Agent 返回的事实卡
{
  "subtask": "pricing_churn_research",
  "findings": [
    {
      "claim": "近三个月客户投诉集中在价格变更说明不清",
      "source": "support_tickets_2026_q2",
      "confidence": "medium",
      "evidence": "142 tickets tagged pricing_change"
    }
  ],
  "dropped": ["重复投诉原文", "与流失无关的配送问题"],
  "unknowns": ["价格变更是否已经影响续费率"],
  "needsLeadDecision": "是否继续查续费数据"
}
对象读什么写什么交给谁错了的症状
lead agent任务目标、子 Agent 摘要、冲突信号最终结论、补查任务用户或下一步 workflow把噪声都读进来，后面开始遗漏约束。
research subagent原始网页、票据、日志、文档事实卡、来源、置信度、unknownslead agent返回长篇原文，污染主上下文。
verifierlead 结论、事实卡、来源列表支持/不支持、缺口、反例lead agent 或人工 review只说“看起来可靠”，没有证据。

如果返回的是原文搬运，说明上下文保护还没起作用。

并行探索：lead 负责汇总，不替每条路走到底

parallel

宽研究任务适合并行。Anthropic 的多 Agent 研究系统就是一个典型形状：lead agent 拆出多个方向，subagents 独立搜索，最后把发现压缩回主上下文。这里的收益来自覆盖更多搜索空间；让几个 Agent 互相聊天本身不会带来稳定能力。

并行化有一个硬前提：子任务之间相对独立。如果每个子 Agent 都要频繁同步完整状态，那说明任务没有被拆对。研究报告可以拆成“客户反馈”“竞品变化”“价格策略”“技术缺陷”四路并行；但一个需要连续调试同一段代码状态的问题，通常不适合拆成四个互相猜测的 Agent。

子 Agent只读材料返回字段不要做lead 怎样用
Feedback Researcher客服工单、用户评论投诉主题、证据、频次、反例不写最终结论判断用户侧流失原因。
Market Researcher竞品公告、行业文章市场变化、替代方案、来源不改内部数据判断外部压力。
Product Analyst产品变更记录、指标摘要变更时间线、可能影响、缺失数据不访问敏感明细对齐时间线。
Verifier汇总稿、来源、事实卡支持度、冲突、缺口、必须重查项不重写报告阻止 unsupported claim 进入最终稿。

并行输出必须标准化。每个子 Agent 都用同一张返回表，lead 才能比较不同方向。如果一个返回随笔、一个返回 Markdown、一个返回代码块，lead 的上下文会被格式解析占满。

并行看的是能不能各自独立交活，不是能不能同时开口。

专业化边界：角色差异要落到工具、权限和禁止项

specialization

专业化要落到工具和权限上。只有当角色的工具、权限、提示、输入输出和禁止项真的不同，专业化才有意义。多个同质 Agent 分摊同一个混乱任务，只会提高协调成本。

一个常见工程收益是缩窄工具面。Research Agent 只能读公开网页和内部只读材料；Code Agent 能改文件但不能发邮件；Verifier 能跑测试和检查来源，但不能直接修结果。工具变窄之后，误用概率会下降，审计也更清楚。

角色输入允许工具禁止项输出格式
Lead Agent用户目标、评测标准、子结果任务分派、汇总、请求补查不伪造来源，不绕过 verifier最终结论、引用事实卡、未解决风险
Research Agent一个独立研究问题搜索、阅读、摘要不改文件，不写最终报告findings / sources / confidence / unknowns
Code Agent明确的代码任务和限制读写文件、运行测试不发消息，不改无关模块diff 摘要、测试结果、风险
Verifier Agent候选结果、来源、Trace测试、来源核对、规则检查不直接生成主答案pass / fail / evidence / required_fix

专业化越强，交接越要窄。不要让每个角色都拿完整任务背景和所有工具；那会把单 Agent 的问题复制多份。每个角色只拿完成当前子任务所需的最小上下文。

角色名不重要，工具和权限才决定它是不是另一个角色。

交接协议：把口头交接写成可验收字段

handoff

多 Agent 系统最容易坏在交接。人类团队里“我以为你知道”的问题，在 Agent 系统里会变成字段缺失、来源丢失、状态误写和权限越界。交接协议要把必要信息固定下来：短、完整、可验证。

多 Agent 交接协议
{
  "handoffId": "pricing-churn.market.01",
  "from": "LeadAgent",
  "to": "MarketResearcher",
  "task": "查明近三个月竞品是否有明显降价或迁移激励",
  "input": {
    "scope": "公开资料和已授权的竞品摘要",
    "deadline": "one search pass",
    "mustReturn": ["findings", "sources", "confidence", "counterExamples", "unknowns"]
  },
  "toolsAllowed": ["web_search", "page_extract"],
  "forbidden": ["write_files", "send_messages", "final_answer"],
  "escalateWhen": ["source_conflict", "low_confidence", "safety_risk"]
}
字段作用缺失后果验收方式
task限定子任务目标子 Agent 自行扩题，返回无关内容看返回是否回答同一个问题。
scope限定材料范围读到未授权或无关材料检查 sources 是否在范围内。
mustReturn固定返回字段lead 难以比较不同子结果字段缺失直接退回。
forbidden限制副作用子 Agent 越权写文件或发消息Trace 中不应出现禁止工具。
escalateWhen定义停机和上交条件低置信度结果被当作事实遇到冲突必须标记 unknowns 或升级。

OpenAI 的 Agents SDK 和 handoffs 相关材料也在强调这个方向：多 Agent 要把交接、guardrails、tracing 和 observability 放进接口。你可以不用某个具体 SDK，但不能省掉这些字段。

交接协议写清以后，lead 才不用靠猜来补字段。

验证子 Agent：职责越窄，越容易可靠

verification

验证子 Agent 是相对稳定的多 Agent 模式，因为它的目标窄：不负责生成主答案，只检查候选结果是否被证据支持、是否漏掉反例、是否越过边界。它不需要和主 Agent 共创答案，交接成本比“规划、执行、审查一起聊天”低。

验证者最常见的失败叫 early victory：只做了一个轻量检查，就宣布通过。防法是给它明确检查清单、负例和停止规则。验证者必须返回证据，不允许只写“看起来没问题”。

case: source
symptom: final claim 引用了“竞品全面降价”，但 sources 只有一篇社区帖。
diagnosis: 证据强度不足，不能支撑强结论。
verifier response: fail; required_fix=补官方价格页或把结论降级成“有用户讨论降价”。
检查项看什么通过标准失败时返回
来源支持每个关键 claim 的 sources来源直接支持，不靠推测补齐unsupported_claim / missing_source
范围边界scope、toolsAllowed、Trace没有越过材料和工具范围scope_violation / forbidden_tool
反例覆盖counterExamples、unknowns报告没有把冲突证据藏起来missing_counterexample
输出完整findings、confidence、unknowns字段齐全，低置信度被标出incomplete_handoff

验证器的任务很窄：把不该过的结果挡回去。

失败模式：多 Agent 会把小问题放大

failure

多 Agent 的失败往往来自系统放大小问题。一个子 Agent 漏了来源，lead 直接引用，verifier 又只看风格，就会形成幻觉放大。一个子 Agent 接触了不可信内容，lead 把它的摘要当成高信任事实，就会出现 trust escalation。

失败模式触发条件症状修法
交接损耗task、scope、mustReturn 不清子结果答非所问，lead 需要猜字段化交接；缺字段退回。
幻觉放大未验证 claim 被多次引用最终报告显得很一致，但来源不支撑关键 claim 必须挂 sources 和 verifier 状态。
trust escalation低信任输入经子 Agent 摘要后变高信任不可信网页内容进入最终事实层保留来源信任级别，lead 不自动升级信任。
工具重叠多个 Agent 都能改同一资源文件互相覆盖，审计难以归因单写者规则；verifier 只读。
成本失控并行方向无预算和深度限制重复搜索、无限补查、token 飙升限制轮次、预算、深度和停止原因。
early victory验证者检查太窄只跑一个正例就宣布通过加入反例、边界项和完整检查清单。
失败 Trace 片段
[
  {"agent": "Researcher", "event": "finding", "claim": "competitor lowered price", "sources": ["forum_post"], "confidence": "low"},
  {"agent": "Lead", "event": "merge", "claim": "competitor lowered price", "confidence": "high"},
  {"agent": "Verifier", "event": "pass", "checks": ["format_only"], "missingChecks": ["source_support", "counter_examples"]}
]

这段 Trace 的问题很清楚：Researcher 已经写了 low confidence，Lead 合并时把它升成 high，Verifier 只检查格式。根因在交接协议和验证清单：前者没有保护信任级别，后者没有覆盖来源支持。

只要看到 low 被无依据升成 high，就该回头改交接或验证。

运行台账：把设计落到可追踪事件

ops-ledger

多 Agent 设计写完以后，还要能在运行时查到证据。上一单元讲过评测、Trace 和安全边界；这一页把它们放回多 Agent 场景里。你至少要记录谁派了任务、谁接了任务、读了哪些来源、返回了什么置信度、有没有触发禁止项、验证者检查了哪些项目。

运行台账的粒度不需要很重，但字段要能回答三个问题：这次为什么拆、拆出去的角色做了什么、最终结果是怎么被验证的。最好也记上 budget 和 stop_reason，这样才看得出是一次正常的宽搜索，还是已经滑向无限补查。少了这些字段，系统出错时只能读长对话，无法定位是交接、上下文、权限还是验证失败。

多 Agent 运行事件
{
  "runId": "churn-report-2026-06-21",
  "event": "handoff_result",
  "from": "MarketResearcher",
  "to": "LeadAgent",
  "handoffId": "pricing-churn.market.01",
  "sources": ["competitor_pricing_page", "release_note"],
  "confidence": "medium",
  "unknowns": ["discount duration unclear"],
  "forbiddenToolUsed": false,
  "verifierStatus": "pending"
}
字段回答的问题谁写谁读缺失后的排错症状
handoffId这条结果来自哪个委派任务lead / dispatcherlead、verifier、审计多个子结果混在一起，无法追责。
sources关键 claim 的来源在哪里subagentverifier最终报告有结论，缺证据链。
confidence子 Agent 对结果有多确定subagentlead低置信度结果被无声升级。
unknowns还有哪些问题没解决subagent / verifierlead不确定性被吞掉，最终答案过度肯定。
budget / stop_reason这轮为什么停、停得是否及时lead / runtimelead、审计无法区分正常搜索和失控补查。
forbiddenToolUsed有没有越过工具边界runtime / guardrailverifier、审计越权动作被当成正常结果。
verifierStatus当前结果是否已验收verifierlead、用户未验证结果进入最终答案。

这张台账的目标只有一个：出问题时能定位到哪一层。

练习

diagnostic
答题目标

这组题检查你能不能把 Multi-Agent 变成一套判断，而不是一串角色名。好的答案会同时说明收益、边界、证据和失败时的升级方式。

作答提醒

先做客观题，再写开放题。开放题按“为什么拆、怎么交接、怎么验收”三步写，别只列角色名。

判断能证明单 Agent 已到边界。
交接能写出字段和禁止项。
验证能防止 early victory。
诊断能从 Trace 定位失败类型。

一、先判断该不该拆

这一组先确认你不会把 Multi-Agent 当成默认答案。

复盘提示先找上下文、搜索空间、工具面、验证缺口这四类证据。
choicedecision

设计 Agent 系统时，默认起点是什么？

true/falsecoupling

判断：如果两个子任务要反复同步完整状态，拆成两个 Agent 通常更稳。

choicecontext

订单、日志和网页材料很多，主 Agent 读完后开始忽略原始目标。这个拆分理由最接近哪一个？

orderingdecision

排序：A 写交接协议；B 单 Agent 先跑并留下失败证据；C 判断拆分收益；D 定义验收和升级规则。

参考答案 / 评分要点

推荐顺序是 B → C → A → D。常被跳过的是 B 和 D：没有失败证据就拆，会变成角色堆叠；没有验收规则就上线，会让 lead Agent 凭语气信任子结果。

二、写清角色、工具和交接

这一组检查你能不能把角色名落到字段、工具和禁止项上。

复盘提示角色必须有输入、工具、输出、禁止项和升级条件。
choicespecialization

专业化 Agent 的关键证据是什么？

short fillhandoff

交接协议里用来固定返回字段的键通常可以叫 ____。

true/falseboundary

判断：交接协议中的 forbidden 应该能限制工具和副作用。

protocolhandoff

为“研究客户流失原因”的 Research Agent 写一个交接草案：至少包含 task、scope、toolsAllowed、forbidden、mustReturn。

参考答案 / 评分要点

好的答案会把任务收窄到一个独立方向，例如“只查客服工单里的价格变更投诉”；toolsAllowed 只给搜索或只读数据；forbidden 明确不写最终报告、不改文件、不访问无授权明细；mustReturn 至少有 findings、sources、confidence、counterExamples、unknowns。

三、验证与失败诊断

这一组检查你能不能从 Trace 里找出多 Agent 的真实失败点。

复盘提示看 claim 的来源、confidence 是否被升级、verifier 检查了什么。
choiceverification

一个稳定的 verifier 最好承担哪种职责？

short fillverification

验证者只跑一个正例就宣布通过，这类失败通常叫 ____。

choicetrust

低信任网页内容经过子 Agent 摘要后，被 lead 当成高信任事实。这个风险最接近哪一个？

diagnosistrace

Trace 显示：Researcher 返回 confidence=low；Lead 合并为 confidence=high；Verifier 只检查 format_only。写出错因、后果、修法。

参考答案 / 评分要点

错因是信任级别在交接中被无依据升级，verifier 检查清单过窄。后果是低置信度 claim 进入最终事实层，形成幻觉放大。修法包括：保留 confidence，不允许 lead 自动升信任；verifier 必须检查 source_support、counterExamples 和 missingChecks；低置信度结果进入 unknowns 或升级人工判断。

四、迁移到真实系统设计

最后把本页方法搬到一个完整方案里。

复盘提示写方案时先给判断证据，再给角色和验收，不要先列一堆角色。
scenariosystem design

为“客户流失原因研究报告”设计一个 3-Agent 方案：谁是 lead，谁是 researcher，谁是 verifier？每个角色的输入、工具、输出和禁止项是什么？

参考答案 / 评分要点

合格答案先说明为什么值得拆，例如资料多、搜索方向并行、需要来源验证。Lead 负责拆任务和汇总；Researcher 只读授权材料并返回事实卡；Verifier 只检查来源支持、范围边界、反例和缺失字段。禁止项要明确：Researcher 不写最终报告，Verifier 不重写报告，所有角色都不能越过授权材料范围。

choicecost

控制多 Agent 成本时，哪一组措施最直接？

synthesisfeynman

用 5-7 句话解释：什么时候使用 Multi-Agent？必须串起单 Agent、协调成本、上下文保护、并行探索、专业化和验证者。

参考答案 / 评分要点

好的解释会先说：单 Agent 或 workflow 是默认起点。只有上下文被污染、搜索空间能并行、工具权限需要隔离，或结果需要独立验证时，才考虑 Multi-Agent。拆分后要有交接协议和标准返回字段，否则信息会丢。专业化必须体现在工具、权限和禁止项上。Verifier 要只验不改，检查来源、反例、范围和缺失字段，防止 early victory。最后要补一句：多 Agent 的收益必须超过协调成本。
