第一步

本地启动最小 Agent

这一节只处理最小闭环：把一个任务交给模型，让本地程序执行允许的动作，再回到终端或文件系统里确认结果。先别追消息队列、子 Agent、权限和记忆；这一页只要你能把最小 Agent 跑起来，并知道自己在看什么。

课程地图Agent 学习地图下一节：消息历史与 Agent 循环

读完本页应该能做什么

  • 能解释 Agent 和普通聊天的差别：前者会把模型给出的动作请求交给本地程序执行。
  • 能核对最小启动所需的三项配置：`ANTHROPIC_API_KEY`、`ANTHROPIC_BASE_URL`、`ANTHROPIC_MODEL`。
  • 能用 `hello.py` 这样的最小任务检查真实副作用，并判断文件到底生成在什么目录。

学习目标

goal

这一页不要求你理解整套 Claude Code 结构，只要求你拿住第一步：任务从哪里进来，模型返回了什么，本地程序又做了什么。学完之后，你应该能自己完成一次启动、一次配置核对、一次最小任务触发和一次结果复验。

你要抓住的对象

一个最小 Agent 由三部分组成：输入任务的命令行程序、会返回下一步建议的模型、以及实际执行动作的本地代码。以后看到输出里有“调用工具”或“执行命令”，先分清谁在提议，谁在执行。

这页的边界

这一页不讲长对话如何续写，也不讲多轮记忆、规划、子 Agent 或权限审批。那些都要等你先能把最小 Agent 跑稳，再往后页走。

先抓主线

flow

普通聊天模型只会给答案。Agent 多了一层本地程序：模型先判断下一步该做什么，返回文本或动作请求；本地程序再决定是否执行，把执行结果回写给模型。这个差别听起来小，实际决定了模型能不能把“写一个文件、跑一个命令、检查目录”变成真实副作用。

before一句自然语言任务

读入的是目标，不是答案。这里还没有任何本地副作用。

reads提示词 + 历史 + 工具能力

模型会根据已有上下文决定是直接回答，还是请求做事。

writes命令、文件、输出

真正改磁盘的是本地程序，不是模型文本本身。

after新的上下文状态

执行结果回到循环里，下一轮判断才有真实反馈。

先别追的东西 消息历史、长任务规划、子 Agent、权限审批都先放一边。它们都是这个闭环后面的放大器，不是第一步本身。

Quick Start

setup

作者在演示里用的是 `learn-claude-code` 这一套目录和入口。你不需要背所有命令，但要知道这四步分别在做什么：进入项目目录、确认 Python 环境、设置模型接口配置、启动 `s01_agent_loop.py`。这四步中，只要有一步错了，后面的“Agent 能做事”就会变成假象。

启动前的检查顺序
cd learn-claude-code
python --version
ANTHROPIC_API_KEY=你的密钥
ANTHROPIC_BASE_URL=兼容 Anthropic 协议的接口地址
ANTHROPIC_MODEL=你要调用的模型名
python s01_agent_loop.py

目录

决定相对路径和生成文件的落点。很多“文件不见了”的问题，其实是从错误目录启动造成的。

环境

决定 Python 能不能 import 需要的依赖。启动失败不一定是模型错，也可能只是当前解释器不对。

配置

决定请求发到哪里、调用哪个模型。Key、URL、Model 这三项一旦混了，后面会连错层。

配置核对

key / url / model

这三个配置项最容易被一起记住、一起写错。`ANTHROPIC_API_KEY` 是身份凭证，回答“谁在调用”；`ANTHROPIC_BASE_URL` 是请求入口，回答“发到哪里”；`ANTHROPIC_MODEL` 是模型名，回答“由谁来处理”。它们放在同一组，但职责完全不同。

配置项读什么写什么错了会怎样
ANTHROPIC_API_KEY身份凭证服务端识别调用者常见报错是 401 或 unauthorized
ANTHROPIC_BASE_URL接口地址请求发往哪个兼容服务请求会连错地方，甚至连到网页地址
ANTHROPIC_MODEL模型标识实际处理任务的模型名服务可能报 model not found 或 bad request

这里有个关键检查顺序：先看 Key 是否存在，再看 URL 是否真的是 API 地址，最后看模型名是不是服务商后台里支持的名字。这个顺序不是习惯问题，是为了少走回头路。

常见误区 把网页地址当 API 地址、把展示名当模型名、把 Key 粘贴进错的 shell，会让你以为是模型不稳定，其实是配置错层。

运行与观察

observe

启动 `s01_agent_loop.py` 之后，最值得看的不是“程序说成功了”，而是它有没有真的把任务变成了本地副作用。最小任务可以很朴素：让它创建一个 `hello.py`，内容打印 `Hello World`，然后再运行它。这个任务足够小，能把“模型提议”和“本地执行”分开看清。

最小任务提示
实现一个 hello.py 文件，内容是打印 Hello World，并运行它。

看终端

确认模型有没有返回动作请求，Agent 有没有真正执行命令，而不是只输出一段说明。

看目录

确认 `hello.py` 是否真的出现在当前工作目录里。文件存在，才说明动作落地了。

再手动跑一次

用普通命令执行生成文件，排除“只是说做了”的假成功。

手动复验
ls
python hello.py

find . -name "hello.py"

这一步里，真正重要的是工作目录。Agent 在哪个目录里启动，就会以哪个目录为基准创建相对路径文件。目录错了，后面所有结果都会看起来“跑通了”，但文件落在别处。

执行链路

checkpoints

最小 Agent 的流程可以拆成四个检查点：先看任务有没有具体到对象，再看配置有没有被当前 shell 真正读到，再看动作有没有落到本地程序，最后看结果有没有回到下一轮判断。只盯着终端上的一句“完成了”，你会错过中间最关键的断点。

关口看什么最小证据常见错位
输入任务是否指向具体对象文件名、目录、内容、动作至少说出三项只说“写个 hello”，模型会自己补默认值
配置Key、URL、Model 是否分层正确当前 shell 里能读到值，且请求发到 API 地址把展示名当模型名，把网页地址当接口地址
执行本地程序有没有真的动手目录里出现文件，终端里出现命令输出只有自然语言说明，没有副作用
回写结果有没有被下一轮看见下一次判断会引用刚才的执行结果每轮都像第一次，说明状态没有接上

这张表的作用，是把“模型说了什么”和“系统做了什么”分开。Agent 的能力来自闭环，不来自单次回答。你看到命令建议，只说明模型提出了动作；你看到文件生成，只说明本地程序执行了动作；你看到后续判断引用了前一步结果，才说明整条链路真的接上了。

最小复验顺序
pwd
ls
python hello.py

先看当前目录，再看文件是否真的在，再手动跑一次生成文件。这个顺序能把“看起来像做完了”的假象拆开，尤其适合排查相对路径和工作目录问题。只要目录层没对齐，后面的结果就会看上去正常，实际落点却是错的。

一轮拆解

worked case

`hello.py` 是这一页最好的锚点。它小到不会遮住机制，又足够完整，能让你看见一次 Agent 循环里真正发生的事：你给出任务，模型判断下一步，本地程序执行允许的动作，文件和终端输出变成结果，再回到上下文里继续判断。很多人第一次看 Agent，只盯着模型回了什么话，却没注意到最重要的变化发生在磁盘和终端上。

把任务说成“在当前目录创建 `hello.py`，内容打印 `hello agent`，然后运行它”，这条句子已经把对象、动作和验证方式都写清了。对象是文件，动作是创建和运行，验证方式是看文件是否真的在当前目录、程序是否真的执行、输出是否真的出现。你如果少写任一层，Agent 就很容易退回到“给你示例，不替你做事”的状态。

再细一点看，这句话其实分成三种质量不同的写法。第一种只说“写个 hello”，这种句子会把目录、内容、运行方式都留给模型猜。第二种说“写一个 hello.py”，已经有对象了，但还没有把内容和复验说清。第三种把目录、文件、内容和执行都写出来，才适合拿来判断 Agent 是否真的完成了动作。

环节输入输出你要看什么错了会怎样
任务输入明确的文件与动作Agent 开始形成计划任务是否具体到文件名、内容和执行命令指令太泛，模型只会给思路，不会落地
模型判断任务 + 历史 + 配置文本或动作请求模型有没有请求执行命令，而不只是解释怎么做你会看到一段像答案的文字，却没有副作用
本地执行动作请求`hello.py`、终端输出文件是否真的生成，命令是否真的跑过模型说做了，磁盘里却没有变化
结果回写执行输出下一轮上下文Agent 是否拿到真实反馈继续判断后续行动建立在空想上
三种任务写法
1. 写个 hello。
2. 写一个 hello.py。
3. 在当前目录创建 hello.py，内容打印 "hello agent"，然后运行 python hello.py。

第三种写法之所以更适合学习，不只是因为它更长，而是因为它把对象、动作和复验都暴露出来了。你看见对象，就能检查文件；你看见动作，就能检查命令；你看见复验，就能确认结果有没有落回系统。这样一来，你不是在猜 Agent 该不该成功，而是在检查它是否真的完成了每一步。

可直接复验的最小任务
请在当前目录创建 hello.py，内容打印 "hello agent"。
创建后运行 python hello.py，并告诉我运行输出。

复验时别只看“输出对不对”，还要看“文件在哪个目录、是不是当前目录、能不能单独运行”。这三件事缺一件，你就还没有把最小闭环跑实。

为什么这个例子值得保留 它同时触到目录、文件、副作用和结果回读。只要这四层能看懂，后面再换成写脚本、改配置或调用工具，思路还是同一条。

故障先后顺序

debug order

最小 Agent 的问题通常不会一上来就出在模型层。更常见的是：目录没切对、当前 shell 没拿到环境变量、配置地址和模型名写错、命令根本没执行。你如果先从最深层开始猜，只会把时间花在不相关的地方。

比较稳的顺序是：先看目录，再看环境，再看配置，最后才看模型和任务。这个顺序的好处，是每一步都能缩小范围，而不是把所有变量一起改掉。只要顺序固定，排查就会快很多。

把这件事讲得更实一点：目录层回答“文件会落到哪里”；环境层回答“解释器和依赖是不是同一套”；配置层回答“请求有没有发到能接收的服务”；执行层回答“动作有没有真的发生”。你每次看报错，其实都是在问哪一层先失真了。只要先把层级说对，后面的修改通常就只剩一个变量，不需要来回乱试。

现象先查什么属于哪一层下一步动作
找不到 `hello.py`当前工作目录目录层先 `pwd` / `ls`，再找文件
import 报错当前 Python 环境环境层确认解释器和依赖安装位置
401 / unauthorized`ANTHROPIC_API_KEY`配置层重新复制 Key，确认当前 shell 已生效
model not found`ANTHROPIC_MODEL`配置层对照服务商后台的模型标识
有输出但没文件工具能力和启动目录执行层确认 Agent 有没有真的获得本地执行能力
目录和环境最容易混在一起 目录错误会让文件落错地方，环境错误会让脚本根本跑不起来。两者都表现为“没有结果”，但解决办法完全不同，所以先拆开看。

这张表的用途是把症状和检查动作绑在一起。以后你再看到“结果不对”，不要先抽象地说“Agent 不稳定”，先把症状落到一个层级上，再按层级往下查。

这一页学完以后，你最应该保住的习惯 先问“问题发生在哪一层”，再决定“改哪个变量”。这比任何一次性修复都更有用。

把机制说完整

summary

如果你要把这一页压成一句更准的话，可以这样说：最小 Agent 是一个命令行程序，它把用户任务交给模型，把模型返回的动作交给本地程序，再把执行结果带回循环里。这里的关键不是“模型很强”，而是系统把提议、执行和复验拆成了不同层。

这也是为什么这一页要先讲目录，再讲配置，再讲运行，再讲故障。目录决定文件往哪落，配置决定请求往哪发，运行决定动作有没有执行，故障演练决定你能不能把症状归层。你如果只记住“先设置 Key”，还不够；你要知道 Key 只是整条链路的一环。

到这里为止，你已经可以开始看下一节了。但下一节不是去背更多术语，而是去看消息历史如何把单次交互变成循环，让 Agent 在前一次结果的基础上继续行动。真正有价值的，是你已经知道这条循环要靠什么启动、靠什么落地、靠什么复验。

再往后学，你会遇到工具调用、权限、子 Agent 和上下文压缩。现在先不用把这些概念提前混进来，它们只是在这个闭环上再叠一层规则。你现在最需要的是一套能落地的判断：任务有没有指向具体对象，动作有没有真正执行，结果有没有回到上下文。只要这三件事能分清，后面的章节就不会只剩名词。

这一页的最小结论
Agent 的三件事：提议、执行、复验。
目录决定落点，配置决定连向哪里，结果决定下一轮怎么判断。

状态账本

state ledger

最小 Agent 的关键不是某个神秘模型技巧，而是状态流转是否清楚：输入从哪里来，配置读到什么，执行写到了哪里，哪一步出了错。把这些状态列出来，排查速度会比“反复重跑一遍”快得多。

步骤读什么写什么错位症状
输入任务用户目标对话上下文任务太模糊，模型给出泛化文本
配置加载Key / URL / Model模型请求参数鉴权失败、连错地址、模型名无效
工具执行模型返回的动作文件、命令、终端输出没有本地副作用，只留下自然语言
结果回写执行输出下一轮上下文模型看不到真实反馈，后续判断漂移

这个账本有一个直接用途：如果你看到程序报错，先问自己错误发生在“输入、配置、执行、还是回写”。不要一上来就重装依赖。

好答案会先判断 shell 语法是否匹配，尤其是环境变量写法、临时赋值方式和当前会话是否真的接收到了值。然后再检查是不是 Key 根本没传进去，或者路径里有空格导致命令没按预期执行。

checklist复验

7. Agent 说已经生成 `hello.py`，你会用哪两步复验？

参考答案 / 评分要点

应当先确认当前目录里真的有 `hello.py`，再用普通命令运行它。只看终端里的自然语言总结不算复验。

rewrite任务描述

8. 把“写个 hello”改成更适合最小 Agent 执行的任务描述。

答案要具体到文件、内容、执行动作。模糊指令会让模型乱猜。

参考答案 / 评分要点

参考写法可以是：在当前目录创建 `hello.py`，内容打印 `hello agent`，然后运行 `python hello.py`，把输出告诉我。好的答案会包含文件名、目标内容和运行动作。

三、错误诊断

这一组要求你把现象和具体层级对上。不要只写“配置错了”，要说清是 Key、URL、Model 还是目录。

复盘提示：先归类层，再判断症状，最后给下一步检查。
diagnosis配置

9. 把 Base URL 和 Model 写反，会造成什么后果？怎么改？

参考答案 / 评分要点

后果通常是请求发错地址，或者模型名不被识别，出现连接失败、`model not found` 或 `bad request` 一类报错。修法是回到服务商后台，把接口地址和模型名分别复制对，不要凭直觉填。

diagnosis复验

10. 只看到终端里有“成功”，但没有检查文件，这个判断哪里不稳？

参考答案 / 评分要点

不稳的地方在于把自然语言总结当成了事实。最小 Agent 的判断标准应该是副作用：文件是否真的生成、命令是否真的跑过、输出是否可复验。没有看文件，就没有确认动作落地。

diagnosis路径

11. 你在上级目录启动 Agent，生成文件找不到。可能发生了什么？

参考答案 / 评分要点

最常见的是当前工作目录不对，导致相对路径文件生成在另一个目录里。也可能是你以为启动在项目目录，实际是在上级目录。先看 `pwd` 或等价命令，再去找文件，不要先怀疑模型。

synthesis边界

12. 为什么跑通 Hello 示例还不能说明你理解了工具调用原理？

要答到“模型提议”和“本地执行”这两层的分工，还要说出你还没看懂的部分是什么。

参考答案 / 评分要点

因为你可能只是碰巧把环境配对了，但还没有真正追踪“模型返回了什么、本地程序怎么决定执行、执行结果怎么回写”。跑通只说明这次链路成功，不说明你已经会解释每一步的职责和失败点。

四、迁移与表达

这一组把这节内容放回真实使用场景：换服务商、换目录、换一句解释，你还能不能稳住最小闭环。

复盘提示：能换平台，不代表能换掉检查顺序。
scenario迁移

13. 从阿里云百炼换到 Kimi 或 DeepSeek 时，哪三类信息需要重新确认？

参考答案 / 评分要点

至少要重新确认接口地址、模型名和当前 shell 里生效的 Key。更稳一点的答案还会提到当前环境是否真的支持兼容协议，以及启动目录有没有变。

short answer源码

14. 为什么 `s01_agent_loop.py` 是后续读源码的入口？

参考答案 / 评分要点

因为它最靠近第一轮真实交互，能看到输入如何进入、配置如何读入、模型请求如何发出、结果如何回到循环里。入口文件通常最适合用来建立源码地图。

interview表达

15. 用 2-3 句话解释：Agent 为什么能“帮你做事”？

参考答案 / 评分要点

可以答：Agent 不是只会生成文本，它会把模型提出的动作交给本地程序执行，所以能真正创建文件、运行命令、读取结果。模型负责判断下一步，本地程序负责把允许的动作落到磁盘或终端上。

checklist启动前检查

16. 写一份 5 项启动前检查清单。

清单至少覆盖目录、环境、Key、URL 和模型名。少一项就容易漏层。

参考答案 / 评分要点

典型清单可以是：确认在项目目录；确认当前 Python 环境对；确认 `ANTHROPIC_API_KEY` 已设置；确认 `ANTHROPIC_BASE_URL` 是 API 地址；确认 `ANTHROPIC_MODEL` 名称和服务商后台一致。写法不必一样，但这五层不能漏。

rubric复查

17. 设计一个最小通过条件：怎样才算本节 demo 真的跑通？

答案要把“终端输出”和“本地副作用”都说到，不能只看一边。

参考答案 / 评分要点

最小通过条件至少应包含：Agent 成功读取配置并启动；接受一个明确任务；在当前目录生成 `hello.py`；普通 `python hello.py` 能跑出预期输出；你能在目录里找到这个文件。这个条件同时覆盖启动、执行和复验。

synthesis费曼解释

18. 用 6-9 句话讲清楚：从输入任务到本地出现 `hello.py`，中间发生了什么。

请串联用户输入、模型请求、本地工具、运行目录、生成文件、手动复验。

参考答案 / 评分要点

高质量答案会顺着真实流程讲：用户输入任务；Agent 把任务和配置发给模型；模型返回文本或动作请求；本地程序在当前目录执行允许的命令；命令生成 `hello.py`；再用普通 `python hello.py` 复验输出；最后回头确认文件确实在目录里。少掉任一环，说明链路还没讲完整。

五、执行链路

这一组专门考你能不能把“提议、执行、回写”说得更具体，而不是只会重复关键词。

复盘提示：任务、动作、结果、再判断，这四步要连得上。
matching链路

19. 把对象和动作配对：`hello.py`、`ANTHROPIC_BASE_URL`、`pwd`、`python hello.py`。

请把每个词对应到“文件 / 配置 / 目录 / 复验命令”之一。

参考答案 / 评分要点

`hello.py` 是文件对象；`ANTHROPIC_BASE_URL` 是配置项；`pwd` 用来确认目录；`python hello.py` 是复验命令。答案只要把四个对象放回正确层级就算对。

true/false回写

20. 只要模型返回了命令建议，就说明 Agent 已经完成执行。

short answer层级

21. “目录错了”和“环境错了”的症状为什么容易混在一起？

参考答案 / 评分要点

因为两者都可能表现成“没有结果”或“文件找不到”。但目录错主要影响文件落点，环境错主要影响解释器和依赖是否能启动。答案只要把这两个影响面拆开，就已经抓住关键。

ordering复验

22. 把这三步排成更稳的复验顺序：运行文件、看当前目录、确认文件存在。

六、边界表达

这一组让你把本节的边界说清。会不会讲，不只是看你懂不懂，还看你会不会收住范围。

复盘提示：先讲本节解决什么，再讲本节不解决什么。
boundary范围

23. 用 3-5 句话说明：这一节讲到了哪里，在哪里停住了？

要明确说出不包括的内容，比如多轮记忆、子 Agent、权限审批、上下文压缩。

参考答案 / 评分要点

好答案会说清本节覆盖启动、配置、最小任务、结果复验和故障分层，但不展开长对话、消息历史、子 Agent、权限和记忆。这种边界表达能帮助你在后面章节不乱混概念。

summary表达

24. 用一句话解释：为什么这个 `hello.py` 例子比“会跑了”更重要？

参考答案 / 评分要点

因为它把对象、动作和复验同时暴露出来，让你能检查 Agent 是不是把任务真的变成了本地副作用，而不只是输出了一段像成功的说明。

scenario风险

25. 如果你只看终端消息不看目录，会漏掉什么？

参考答案 / 评分要点

会漏掉文件是否真的生成在当前目录、是否落到了别处、以及命令输出是不是只是描述而没有副作用。最小 Agent 的真假，很多时候就藏在目录里。

synthesis转述

26. 把“提议、执行、复验”这三个词各用一句话说清楚。

每句话都要能对应到本节里的一个实际检查点。

参考答案 / 评分要点

提议是模型根据任务和上下文给出的动作建议；执行是本地程序把允许的动作真正做出来；复验是你用目录检查和普通命令确认结果真的落地。三句都要落到具体检查点上，不能只重复定义词。
