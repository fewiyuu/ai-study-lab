1. 先分清两层循环

outer

源码里有两层 while True。外层循环在主函数里，负责不断等待用户输入；只要你不输入 q 或 exit，它就会继续问下一句。内层循环在 agent_loop 里，负责让模型完成当前这一个用户任务。当前任务结束后，内层循环退出，控制权回到外层循环。

外层循环做什么

  • 维护同一个 history 列表。
  • 等待用户输入 query。
  • 把用户输入包装成 role=user 的消息。
  • 调用 agent_loop(history)。
  • 从最后一条回复里打印文本。

内层循环做什么

  • 把当前 messages 发给模型。
  • 把模型响应追加进同一个列表。
  • 检查 stop_reason。
  • 如果是 end_turn，说明当前轮结束。
  • 如果是 tool_use，下一节会继续处理工具。

逐行拆解：两层循环骨架Python

history = []

while True:
 query = input("> ")
 if query in ("q", "exit"):
 break

 history.append({"role": "user", "content": query})
 agent_loop(history)

 response_content = history[-1]["content"]
 for block in response_content:
 if hasattr(block, "text"):
 print(block.text)

你现在应该能判断：history 不在 agent_loop 里面新建，因为它要跨多次用户输入持续累积；agent_loop 接收的是同一个列表引用，所以它追加的 assistant 消息会留在外层。

2. history 是消息列表，不是聊天文本

role

history 是一个列表，里面每个元素是一条消息。最小用户消息通常有两个字段：role 表示这条消息是谁说的，content 表示内容。输入 hello 后，外层循环会追加 {"role": "user", "content": "hello"}。

字段本节含义常见错法

正确示例：hello 后的消息列表Python-like

判断句：看到 history[-1]["content"] 时，要先问最后一条消息是谁追加的。如果是 assistant 响应，content 通常要按 blocks 遍历，而不是直接 print(content)。

3. agent_loop 的一轮做了什么

request

agent_loop 是当前任务的核心。它把 messages 发给模型，拿到 response，再把 response 作为 assistant 消息追加回 messages。追加之后再看 stop_reason。这一步要先追加再判断，因为无论后续是结束还是工具调用，模型刚才说过什么都应该进入历史。

点击观察当前步骤

用户输入 hello

外层循环拿到 query。此时还没有追加消息，history 仍然保持上一轮状态。

观察点：用户输入只是原始字符串，还不是 API 协议里的消息对象。

代码
response = client.messages.create(messages=messages)

if response.stop_reason == "end_turn":
 return

# 错误点：response 没有被 append 到 messages。
# 外层回去读 history[-1] 时，读到的仍然是 user 消息。

这类错误的后果很明显：外层想从最后一条消息里打印模型回复，却发现最后一条还是用户输入。修法是在判断结束条件前，把 assistant 响应追加到 messages。

4. content blocks 要逐个看

thinking

模型响应的 content block 列表。本节调试时能看到至少两类 block：thinking block 和 text block。代码遍历这些 block，找到带 text 属性的那一个，再打印 block.text。

thinking block

表示模型内部推理相关的块。不同 API 或模型对它的可见性、字段和处理方式不同。本节观察重点是：它不是要直接打印给用户的普通文本。

text block

包含可以展示给用户的自然语言文本。代码用 hasattr(block, "text") 找到它，然后打印出来。

正确示例：只打印有 text 的 blockPython

response_content = history[-1]["content"]

for block in response_content:
 if hasattr(block, "text"):
 print(block.text)

判断句：如果你把 response.content 当字符串处理，说明你还没进入 Agent 协议层；如果你能说出“遍历 block，识别 text block，再展示”，就抓住了本节的关键。

6. 公开讲义补强：消息协议最小心智模型

message

把这一页压缩成一句话：Agent 循环在维护一组有角色、有内容块、有停止原因的协议消息。协议消息的好处是程序能精确判断“谁说的、说了什么类型的内容、下一步该结束还是调用工具”。

最小消息历史例子JSON-like

常见误区

  • 把 history 理解成一整段字符串，导致无法解释多轮上下文。
  • 把 agent_loop 当成主输入循环，混淆“当前任务内部循环”和“下一次用户输入”。
  • 直接打印 response.content，没有按 block 类型过滤 text。
  • 先判断 stop_reason 再决定是否保存 assistant 响应，导致历史不完整。

你现在应该能做什么：看到一次模型响应时，先按 role -> content blocks -> stop_reason -> 是否继续循环 的顺序读，而不是把它当成普通字符串输出。
