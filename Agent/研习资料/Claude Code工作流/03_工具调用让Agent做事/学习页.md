# 1. tools 列表是在告诉模型“你能请求什么” {#tools}
<!-- tag: capability list -->

模型本身不知道你本地有哪些工具。Agent 每次请求模型时，会把 `tools` 列表一起传过去。这个列表说明工具名、用途和参数结构。模型看到这些说明后，才可能在响应里返回 `tool_use` block，请求本地程序执行某个工具。

这里要分清两件事：`tools` 列表只是能力说明，不是工具执行结果；模型返回 `tool_use` 也只是请求，不是已经执行。真正的执行发生在本地代码里，比如 `run_bash` 调用 `subprocess.run`。

正确示例：把工具能力传给模型Python

```text
response = client.messages.create(
 model=model,
 messages=messages,
 tools=tools,
)
```

工具名模型用它指定要调用哪个工具，例如 `bash`。
参数 schema告诉模型 input 里应该放什么字段，例如 `command`。
本地 handler真正执行工具的 Python 函数，例如 `run_bash`。

判断句：`tools` 让模型知道“可以请求 Bash”，但不能代替本地安全检查。模型给出的命令仍然要由本地程序决定是否执行。

# 2. tool_use block 是模型提出的动作请求 {#tool-use}
<!-- tag: request block -->

当模型判断只靠文字不够，需要本地收集信息或执行动作时，`response.stop_reason` 会变成 `tool_use`。这和上一节的 `end_turn` 不同：`end_turn` 表示本轮可以结束，`tool_use` 表示模型希望本地程序先做一件事，再把结果告诉它。

字幕里的调试示例中，`response.content` 里有 thinking block 和 tool_use block。代码会遍历 content，找到 `type == "tool_use"` 的块，再从 `block.input.command` 里取出命令。这个命令要传给工具 handler 的参数。

逐行拆解：识别 tool_use 并取出命令Python

```text
for block in response.content:
 if block.type == "tool_use":
 command = block.input["command"]
 output = run_bash(command)
 tool_results.append({
 "type": "tool_result",
 "tool_use_id": block.id,
 "content": output,
 })
```

字段作用读错后果

`block.type`判断这是 thinking、text 还是 tool_use。把 thinking 当命令执行会完全跑偏。
`block.input`工具参数，本节里关键字段是 `command`。取错字段会导致 handler 拿不到命令。
`block.id`后续 tool_result 要对应这次 tool_use。模型不知道哪个结果对应哪个请求。

# 3. 本地执行：run_bash 不是摆设 {#execute}
<!-- tag: local side effect -->

`run_bash` 是本节里让 Agent 从聊天变成行动的关键。它会拿到模型请求的 command，先做一些简单的危险命令拦截，再用本地 shell 执行。字幕里提到会屏蔽提权、关机、重启、危险删除等命令，目的给最小 Agent 加一道基础防线。

这里有个边界要讲清楚：本节的安全检查非常粗糙。它能让你理解“执行前应该有校验”，但不能当生产级安全策略。后面的权限系统会更系统地处理允许、拒绝、询问和规则匹配。

正确示例：最小 Bash handler 的意图Python

```text
def run_bash(command: str) -> str:
 blocked = ["sudo", "shutdown", "reboot", "rm -rf /"]
 if any(word in command for word in blocked):
 return "Blocked dangerous command."

 result = subprocess.run(
 command,
 shell=True,
 capture_output=True,
 text=True,
 )
 return result.stdout or result.stderr or "No output"
```

判断句：工具 handler 必须返回结果。无论命令成功、失败、无输出、被拦截，都要形成可写回 messages 的文本，否则模型无法知道刚才发生了什么。

# 4. messages 为什么会越来越长 {#messages}
<!-- tag: context grows -->

工具调用的核心不是“执行一次命令”而是“执行结果进入上下文”。第一次请求模型时，messages 里可能只有用户 prompt。模型返回 assistant 响应，请求工具，于是 messages 变成 2 条。工具执行后，tool_result 作为新消息追加进去，messages 变成 3 条。再次请求模型时，这 3 条会一起发过去。

如果模型还需要第二次工具调用，流程继续：assistant tool_use 追加进去，tool_result 再追加进去，messages 就可能变成 5 条。随着 messages 增长，模型看到的上下文更多，能知道哪些命令失败了、哪些路径不存在、下一步应该换什么办法。

观察点：没有 tool_result 时，模型无法利用本地执行反馈。

错误示例：执行了工具但不回写结果上下文断裂

```text
output = run_bash(command)

# 错误点：只在本地变量里拿到 output，没有 append 到 messages。
# 下一次请求模型时，模型不知道命令执行失败还是成功。
```

这类错误会让 Agent 退回“盲猜下一步”。它可能反复执行类似命令，因为它没看到上一次失败的输出。修法是把 tool_result 作为消息追加回去，并带上对应的 tool_use id。

# 5. 失败结果也有学习价值 {#errors}
<!-- tag: failure feedback -->

字幕示例里，模型请求执行类似 `find` 的命令，结果可能是找不到文件、拒绝访问或没有输出。初学者容易觉得“失败就没用”。Agent 视角正好相反：失败输出也是环境反馈。模型只有看到失败结果，才知道下一步要换命令、换路径、或者向用户解释限制。

工具输出应该怎样回写模型可以学到什么

成功输出把 stdout 写入 tool_result。已有足够信息，可以继续总结或下一步操作。
stderr 报错把错误文本写入 tool_result。命令失败原因、权限问题、路径问题。
No output明确写入无输出。命令执行了，但没有返回内容。
安全拦截写入 blocked 原因。这条路径不能走，要换安全方案。

判断句：工具结果给下一轮模型看的。只要会影响下一步判断，就应该进入 messages。

# 6. 公开讲义补强：一次工具调用的验收清单 {#public-guide}
<!-- tag: tool call checklist -->

把这一页压缩成一句话：工具调用是一个双向协议，模型只能提出“我想调用什么工具、带什么参数”，本地程序负责执行，并把真实结果回写给模型。只要结果没有回写，这次调用在下一轮模型眼里就等于没有发生。

`tools`能力目录。它告诉模型“可以请求哪些工具、参数长什么样”，不执行任何动作。
`tool_use`动作请求。它由模型生成，通常包含工具名、调用 id 和 input 参数。
`tool_result`执行反馈。它由本地程序生成，必须能对应到某次 `tool_use`。

最小工具调用三步Pseudo flow

```text
1. send(messages, tools=[bash_schema])
2. model -> tool_use(id="u1", name="bash", input={command: "pwd"})
3. local -> tool_result(tool_use_id="u1", content="...current directory...")
```

验收点为什么重要失败信号

工具名存在能路由到本地 handler。模型请求了未注册工具。
参数符合 schemahandler 能拿到必要字段。`command` 缺失或类型不对。
执行有边界防止危险命令直接运行。没有任何拦截或权限检查。
结果写回 messages下一轮模型才能基于事实继续。模型重复同一命令或继续盲猜。

你现在应该能做什么：审查任意工具调用实现时，按“声明能力、识别请求、本地执行、回写结果、继续循环”五步确认链路是否闭合。
