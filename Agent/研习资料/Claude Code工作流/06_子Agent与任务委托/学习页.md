# 1. 为什么需要任务委托 {#why}
<!-- tag: delegate work -->

一个主 Agent 可以直接读取文件、搜索内容、修改代码，但有些任务会消耗很多轮对话。例如“读取所有 Python 文件并总结每个文件的作用”，主 Agent 如果亲自做，就要把每一次读取、每一次中间判断都塞进自己的 messages。任务越大，主上下文越重，后面的推理也越容易被无关细节干扰。

子 Agent 的直觉很简单：把一个相对独立的子任务交出去，让它在自己的小上下文里折腾，最后只给父 Agent 一段整理好的结果。父 Agent 不关心子 Agent 读了多少文件、绕了几轮、试了几次，只关心最终产物能不能用于下一步决策。

## box warn
### 主 Agent 直接做的代价
- 所有中间消息都进入主上下文。
- 文件探索、失败尝试和细节会占用注意力。
- 主 Agent 之后还要继续对话，历史会越来越重。
- 权限如果过宽，所有步骤都在同一套能力里发生。

## box
### 委托给子 Agent 的价值
- 子任务有独立 `sub_messages`。
- 父 Agent 只接收最终文本结果。
- 子 Agent 可以使用更小的工具集。
- 主上下文保持短，下一轮推理更集中。

注意：子 Agent 不是神秘的新模型，也不是一定要多线程并行。它首先是一种“把任务包装成工具调用”的设计。父 Agent 看到的是 `task` 工具；本地实现负责把这个工具调用转换成一次新的 Agent 循环。

# 2. `task` 工具把委托变成普通工具调用 {#task}
<!-- tag: description / prompt -->

在视频里，模型响应里出现了一个 tool use block，工具名是 `task`。这说明父 Agent 并像调用 `read_file`、`bash` 一样调用 `task`。这会让 Agent 框架保持统一：模型只需要根据工具 schema 生成合法参数，本地代码再执行对应函数。

`task` 工具通常有两个关键参数：`description` 和 `prompt`。`description` 是短描述，用来让父 Agent 和日志知道这次委托大概在做什么；`prompt` 是交给子 Agent 的正式任务说明，应该写清目标、范围、输出格式和限制。子 Agent 不会天然知道父 Agent 的全部意图，委托 prompt 写得越含糊，子 Agent 的结果越可能偏。

正确示例：父 Agent 调用 taskJSON

```text
{
 "name": "task",
 "input": {
 "description": "Summarize Python files",
 "prompt": "读取当前项目中的所有 Python 文件，按文件名列出每个文件的核心职责。只总结职责，不要修改文件。"
 }
}
```

字段直觉作用写坏后的后果

`description`给委托任务一个短标签。日志和父 Agent 难以快速识别任务。
`prompt`真正交给子 Agent 的任务说明。范围不清、输出不稳、容易越权行动。
`task` schema告诉模型该工具需要哪些参数。模型可能返回自然语言，无法执行工具。

你现在应该能做什么：看到一段 tool use，就能判断它是不是在委托任务；看到一段委托 prompt，就能指出它是否说明了目标、范围和输出格式。

# 3. `run_sub_agent` 启动独立循环 {#subloop}
<!-- tag: new messages -->

父 Agent 收到 `task` 工具调用后，会进入 `run_sub_agent`。这个函数最重要的第一步，是新建一个 `sub_messages` 列表。它不是父 Agent 的 `messages`，也不是把父历史复制一整份进去。它是子 Agent 自己的工作台，用来保存子 Agent 与模型之间的多轮交互。

主 Agent 的循环通常是持续的：用户还会继续输入新 prompt，Agent 也会继续响应。子 Agent 则必须有边界，视频里示例是最多 30 轮。这个上限是安全边界：如果子 Agent 一直找不到结果、一直调用工具、一直循环，不能让它无限跑下去。

逐行拆解：子 Agent 循环的骨架Python

```text
def run_sub_agent(prompt: str) -> str:
 sub_messages = [{"role": "user", "content": prompt}]

 for _ in range(30):
 response = call_model(
 messages=sub_messages,
 tools=child_tools,
 )
 sub_messages.append(response)

 if response_has_tool_use(response):
 result = execute_child_tool(response)
 sub_messages.append(result)
 continue

 return final_text(response)

 return "子 Agent 达到轮数上限，未能完成任务。"
```

这段代码的关键不是语法，而是三个顺序：先创建独立消息列表，再用 `child_tools` 调模型，最后在结束时只返回最终文本。只要你抓住这三个点，就能读懂大多数“子 Agent 作为工具”的实现。

父子消息流观察器

父 Agent 调用 task
子 Agent 内部多轮
最终结果回父级


**1. parent messages**
用户请求：读取所有 Python 文件并总结。模型选择调用 `task`。
**2. sub_messages**
子 Agent 使用自己的消息列表，多轮调用 read / bash 等工具完成探索。
**3. tool_result**
只把最终总结文本写回父 Agent 的 messages，子上下文被丢弃。


父 Agent 把任务交出去
父 Agent 的响应里出现 `task` tool use。此时它还没有亲自读取所有文件，而是把任务说明交给本地 `run_sub_agent`。
观察点：子 Agent 在父 Agent 眼里就是一次工具调用，不是整段对话历史。

# 4. 上下文隔离：只返回最终 text {#context}
<!-- tag: discard child context -->

视频里最值得记住的一句注释是：只有最终 response content 结果会返回，child context 会被丢弃。这里的 context 可以粗略理解成子 Agent 的 `sub_messages`：它和模型交互的中间消息、工具结果、探索路径都留在子循环内部，不会整段并入父 Agent。

这带来一个明显好处：父 Agent 的 messages 仍然很短。视频示例里，子 Agent 可能已经和模型交互了七八次，但父 Agent 侧只看到一条 `task` 工具结果，例如“I found and read all 18 Python files...”。父 Agent 后续推理只需要理解这条整理后的结果，不需要消化每个文件读取过程。

## box soft
### 父上下文看到什么
1. 用户原始任务。
2. 父模型发起 `task` tool use。
3. `task` 的最终文本结果。

## box warn
### 父上下文看不到什么
1. 子 Agent 每次 read_file 的细节。
2. 子 Agent 中途失败或重试的路径。
3. 完整 `sub_messages` 历史。

错误示例：把子上下文全部塞回父级上下文污染

```text
# 错误思路：把子 Agent 的完整消息历史返回给父 Agent
return {
 "content": sub_messages
}

# 后果：父 Agent 的 messages 被中间过程淹没，后续推理更重。
```

上下文隔离为了控制信息密度。子 Agent 可以努力探索，父 Agent 只吸收压缩后的结论。这和上一节 todo 的思路也能连起来：复杂系统需要可见状态，但不需要把所有中间噪声都塞进主循环。

# 5. 工具权限收缩：子 Agent 不该无限外包 {#permission}
<!-- tag: child tools -->

父 Agent 拥有 `parent_tools`，其中包含 `task`。子 Agent 使用的是 `child_tools`，视频里提到它有 bash、read、write、edit 等工具，但没有 `task`。这意味着子 Agent 不能再把任务委托给另一个子 Agent。这个限制非常关键，否则委托可以无限嵌套，变成难以审计的递归外包。

权限收缩还有另一个价值：你可以根据任务给子 Agent 更小的能力集合。比如“只总结代码结构”的子 Agent 只需要 read 和搜索，不一定需要 write 或 edit；“尝试修复测试”的子 Agent 也许需要 edit，但不需要删除文件或访问外部系统。工具集越小，意外行为的空间越小。

能力父 Agent子 Agent设计含义

`task`有无父级可以委托，子级不能继续无限委托。
`read_file` / 搜索通常有可给适合资料收集、代码阅读、摘要任务。
`write_file` / `edit`视任务有可收缩写权限应按任务需要给，不默认放开。
外部系统工具视场景有谨慎给减少越权、误操作和审计压力。

正确示例：父子工具集区分Python

```text
parent_tools = [read_file, write_file, edit_file, bash, task]

child_tools = [
 read_file,
 bash,
 # 没有 task，避免无限委托
 # 写工具是否给，要看子任务是否真的需要
]
```

你现在应该能做什么：设计一个子 Agent 时，先问“它完成任务最少需要哪些工具”，而不是把父 Agent 的全部工具原样复制过去。

# 6. 公开讲义补强：什么时候该委托 {#public-guide}
<!-- tag: delegation checklist -->

把这一页压缩成一句话：子 Agent 适合处理边界清楚、需要独立探索、但最终只需给父 Agent 一个结论的子任务。它用隔离上下文和收缩权限来管理复杂度。

适合委托代码库摸底、资料检索、错误日志归因、候选方案比较、只读型分析任务。
不适合委托一步就能完成的动作、需要用户即时决策的操作、高风险写入或权限不清的任务。
输出契约委托 prompt 要写清输出格式，例如“列出结论、证据、风险、下一步”，方便父 Agent 接回主线。

合格的 task promptPrompt

```text
请只读分析当前项目的测试失败原因。
范围：只查看测试输出和相关源码，不修改文件。
输出：用三段返回：根因判断、证据文件/行号、建议修复步骤。
限制：不要继续委托其他 Agent，不要执行删除或写入操作。
```

检查项好委托坏委托

目标子任务有明确完成条件。“帮我看看”这种开放请求。
上下文只给完成子任务所需背景。把父 Agent 全部 history 直接塞过去。
工具按任务需要给最小工具集。默认复制父 Agent 的所有工具。
返回最终只回传压缩后的结论和证据。把完整 `sub_messages` 全量回传。

你现在应该能做什么：拿到一个复杂任务时，判断哪些部分适合交给子 Agent，并写出包含目标、范围、限制、输出格式的委托 prompt。
