07 Skill渐进式披露｜Claude Code 工作流

Skill 是一份可被 Agent 调用的能力说明书。它通常写在 `SKILL.md` 里，上半部分是 `name`、`description` 等元数据，下半部分是完整工作流。渐进式披露的做法是：启动时只把元数据放进 system prompt，等模型判断需要某个 Skill 并调用 `load_skill` 时，再把这个 Skill 的完整 body 加入 messages。

1. Skill 是可加载的能力说明书

SKILL.md

Skill 的直觉是“给 Agent 的操作手册”。当用户提出某类需求时，Agent 可以加载对应 Skill，获得更具体的步骤、约束、工具使用规则和输出规范。比如 PDF Skill 会告诉 Agent 如何创建 PDF，Code Review Skill 会告诉 Agent 如何审查代码。

最小示例：SKILL.md 文件结构Markdown

metadata 适合做什么

  • 让模型快速知道有哪些 Skill。
  • 帮助模型判断用户需求匹配哪个 Skill。
  • 构建短小的 Skill 列表。

body 适合做什么

  • 提供完整执行步骤。
  • 写清工具限制和验收标准。
  • 在真正需要时进入上下文。

你现在应该能做什么：看到一个 Skill 文件时，能指出哪部分用于“发现能力”，哪部分用于“执行能力”。

2. `SkillLoader` 启动时递归扫描

load_all

程序启动时会创建一个 SkillLoader 对象，并指向当前工作目录下的 skills 目录。因为 Skill 可能放在多层子目录里，所以加载器会递归查找所有 SKILL.md 文件。视频里示例项目有多个 Skill，例如 agent builder、code review、MCP builder、PDF。

逐行拆解：扫描并登记 SkillPython

这里的 rglob("SKILL.md") 表示递归匹配当前目录和所有子目录里的 SKILL.md。这比只扫描第一层目录稳，因为真实项目里的技能经常会按主题或来源分层存放。

创建 loader初始化 self.skills = {}。准备一个可按名称查询的 Skill 索引。

递归扫描查找所有 SKILL.md。覆盖多层目录，不漏技能。

解析文件拆出 metadata 和 body。为渐进式披露做结构分离。

登记字典按 name 存 metadata/body/path。后续 load_skill 可以快速读取。

3. 第一轮只给模型 Skill 描述

descriptions

加载器已经把所有 Skill 文件读到本地，但这不等于要把所有内容立刻发送给模型。get_descriptions() 会从字典中抽取每个 Skill 的 name 和 description，拼成简短列表，放到 system prompt 里。这样模型第一次请求时，只知道“有哪些 Skill、各自适合什么场景”。

正确示例：只组装 Skill 摘要Python

错误示例：启动时塞入所有 body上下文浪费

坏处上下文工程变差。模型要在大量无关流程里找相关信息，后续响应质量可能下降，成本和延迟也会增加。

4. 需要时再调用 `load_skill`

tool

当用户说“创建一个 PDF，内容是你好 Skill”，模型会先看 system prompt 里的 Skill 描述，判断 PDF Skill 相关，于是返回一个 tool use block：调用 load_skill，输入是 Skill 名称，例如 PDF 或 pdf。本地工具收到名字后，从 loader 字典里取出对应 body。

正确示例：load_skill 工具Python

上下文披露切换

上下文占用：低。模型只看到 Skill 名称和描述。

你现在应该能做什么：看到一次 load_skill 调用时，能说出它为什么发生在“匹配 Skill 之后、执行任务之前”。

5. 渐进式披露是一种上下文工程

context

上下文工程把信息放在合适的时间、以合适的粒度交给模型。Skill 系统的做法很典型：启动时发短目录，任务匹配后发完整说明，执行时再根据工具结果继续推进。

少量初始信息只发送 name 和 description，用于发现匹配能力。

按需详细说明模型调用 load_skill 后，再把完整 body 放进 messages。

执行后继续循环模型根据 Skill 工作流调用 write、bash 等工具，直到完成任务。

6. 公开讲义补强：渐进式披露设计清单

progressive

把这一页压缩成一句话：Skill 系统把“发现能力”和“执行能力”拆开。模型一开始只看到短目录，用它判断是否相关；确认相关后，再通过工具加载完整说明。这样既保留能力，又避免无关文档占满上下文。
