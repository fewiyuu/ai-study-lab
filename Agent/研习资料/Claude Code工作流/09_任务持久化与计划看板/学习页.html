09 任务持久化与计划看板｜Claude Code 工作流

本节把前面内存里的扁平 todo 升级成可持久化的任务系统。每个 task 是 `.tasks/task_N.json` 文件，里面保存 `id`、`subject`、`description`、`status`、`blocked_by`、`blocks`、`owner` 等字段。Agent 通过 `task_create`、`task_update`、`task_list`、`task_get` 四个工具创建、更新、查看和渲染任务。

1. 从内存 todo 到文件任务系统

memory

内存 todo manager 更适合短、线性、一次性的任务追踪。它把计划保存在当前进程里，程序一退出，计划也跟着消失。任务之间的关系也比较简单，通常只是 pending、in_progress、completed 这类状态切换。

内存 todo 的限制

  • 程序退出后状态丢失。
  • 更适合轻量线性流程。
  • 依赖关系不明显。
  • 只能通过程序状态或输出观察。

文件任务系统的收益

  • 每个任务是独立 JSON 文件。
  • 重启后可以重新加载。
  • 依赖关系可被显式记录。
  • 用户能直接查看 .tasks 目录。

你现在应该能做什么：看到一个任务追踪需求时，能判断它只是短期 todo，还是需要持久化、依赖和恢复能力。

2. 每个 task 是一个 JSON 文件

task_N.json

TaskManager 初始化时会创建 .tasks 目录，并扫描已有文件得到当前最大 id。空目录时 max id 是 0，下一条任务 id 就从 1 开始。创建任务时，next_id 用作文件名和任务 id，然后自增。

3. 四个工具组成任务系统接口

create/update/list/get

Agent 并不直接乱写 .tasks 目录，而是通过工具 schema 告诉模型可以调用哪些任务工具。视频里新增了四个工具：task_create、task_update、task_list 和 task_get。模型根据 schema 返回对应参数，本地函数再执行读写。

正确示例：task_create 输入JSON

逐行拆解：创建任务并保存Python

task_update

读取已有 task_N.json，按输入更新 status、add_blocked_by、add_blocks 等字段，再保存回文件。

task_list / task_get

task_list 读取所有任务并格式化成看板；task_get 读取某个任务的完整 JSON，适合查细节。

4. `blocked_by` 表达依赖，也允许并行

dependencies

任务依赖写进 JSON。示例中任务 2 transform 和任务 3 emit 都 blocked_by: [1]，意思是它们都必须等任务 1 parse 完成。任务 4 test blocked_by: [2, 3]，意思是 test 要等 transform 和 emit 都完成。

流程图式文字

  1. 先把目标拆成可独立验收的任务节点。
  2. 给每个节点写入 subject、description 和初始 pending 状态。
  3. 再补依赖：后置任务把前置任务 id 写进 blocked_by。
  4. 执行时只挑 blocked_by 为空、状态为 pending 的任务。
  5. 前置完成后解除依赖，再刷新看板。

验收清单

  • 重启后还能从 .tasks 恢复任务。
  • 每个任务有清晰标题、边界和状态。
  • 依赖方向能回答“我在等谁”。
  • 并行任务之间没有多余依赖。
  • task_list 能让人一眼看到阻塞点。

正确示例：用 task_update 添加依赖JSON

[
 {"name": "task_update", "input": {"task_id": 2, "add_blocked_by": [1]}},
 {"name": "task_update", "input": {"task_id": 3, "add_blocked_by": [1]}},
 {"name": "task_update", "input": {"task_id": 4, "add_blocked_by": [2, 3]}}
]

四个任务都是 pending

刚创建时，每个任务都有 JSON 文件，但还没有依赖关系。此时看板只能显示四个待办。

你现在应该能做什么：看到一个任务链，能用 blocked_by 判断哪些任务必须等待，哪些任务可以并行。

5. `task_list` 把 JSON 渲染成计划看板

visible

task_list 会读取 .tasks/task_*.json，排序后逐个格式化。它根据 status 决定 marker：pending 用 [ ]，in_progress 用 [>] 或类似符号，completed 用 [x]。再把 id、subject 和 blocked_by 信息拼成一行。

逐行拆解：任务列表渲染Python

文件持久化的代价

文件任务系统比内存 todo 更强，但也更复杂。并发写入可能覆盖，读取和写回要考虑原子性，多个 Agent 同时操作 .tasks 时还要考虑锁或数据库。工程上要根据任务规模选择复杂度。

Worked example：判断下一步能做什么

假设看板里有四个任务：parse 已 completed，transform 和 emit 都 blocked_by: [1]，test blocked_by: [2,3]。正确判断是：transform 和 emit 已经可以并行开始；test 仍然不能开始，因为它还在等待 2 和 3。这个例子检查的能从 blocked_by 推出执行顺序。
