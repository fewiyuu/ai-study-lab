# 1. 慢操作不能让 Agent 空等 {#why}
<!-- tag: non-blocking UX -->

传统工具调用通常是同步的：模型要求运行命令，本地执行命令，等命令返回后再把结果写入 messages。如果命令只是 `ls`，问题不大；如果是下载大文件、跑长测试、构建项目或等待外部服务，就会让主循环卡住。

后台任务的直觉是“先启动，后回收”。主 Agent 不需要一直盯着慢命令，它只要知道后台任务已经启动，并在后续轮次拿到结果即可。这样用户可以继续输入 prompt，Agent 也可以继续规划或处理其他事情。

## box warn
### 同步等待的问题
- 命令没结束前主循环不能继续。
- 用户看起来像被卡住。
- 多个慢任务无法自然并行。
- Agent 不能及时响应下一步需求。

## box
### 后台任务的收益
- 启动后立刻返回 task id。
- 多个任务可以同时运行。
- 完成结果进入通知队列。
- 主循环按自己的节奏拉取结果。

你现在应该能做什么：看到一个工具命令时，能判断它适合同步执行，还是应该放到后台。

# 2. `BackgroundManager` 管理任务状态和通知 {#manager}
<!-- tag: tasks / queue / lock -->

`BackgroundManager` 初始化时会创建三个关键对象。第一是 `tasks` 字典，key 是 task id，value 保存命令、状态和结果。第二是 `notification_q`，后台任务完成后把结果放进去。第三是 `lock`，用于保护队列和任务状态的读写，避免主线程和后台线程同时修改造成混乱。

最小示例：BackgroundManager 初始化Python

```text
class BackgroundManager:
 def __init__(self):
 self.tasks = {}
 self.notification_q = []
 self.lock = threading.Lock()
```

`tasks` 字典是“可查询状态”，`notification_q` 是“完成事件”。这两者不要混为一谈：一个任务 running 时已经在 `tasks` 里，但只有完成后才会进入通知队列。

对象保存什么谁读写

`tasks`task id、command、status、result。启动任务、执行线程、check 工具。
`notification_q`已经完成的后台任务通知。后台线程写入，主循环 drain。
`lock`同步保护。所有涉及共享状态的读写。

# 3. `background_run` 立刻返回，后台线程继续执行 {#run}
<!-- tag: thread start -->

工具 schema 里新增了 `background_run`，要求模型返回 `command`。本地收到命令后生成一个短 task id，把任务登记为 running，然后创建一个线程执行真正的命令。线程启动后，函数马上返回“background task started”。

视频里示例在 Windows 上运行类似 `timeout /T 2`、`timeout /T 4`、`timeout /T 6` 的命令。三个命令都启动为后台任务，所以它们可以并行等待。主 Agent 不需要等待 2、4、6 秒全部结束才继续。

## box soft
### 流程图式文字
1. 模型调用 `background_run(command)`。
2. 管理器生成 task id，并把任务登记为 running。
3. 后台线程执行命令，主循环立即拿到启动结果。
4. 命令结束后，线程更新 `tasks[task_id]`。
5. 线程把完成通知写入 `notification_q`。
6. 下一轮主循环 drain 队列，把结果注入 messages。

## box
### 适合放后台的任务
- 耗时超过一轮交互的构建、测试、下载。
- 结果可以稍后回收，不需要阻塞当前推理。
- 有明确 task id，可被查询和取消。
- 输出可以被截断、摘要或写入文件。

逐行拆解：启动后台任务Python

```text
def background_run(command):
 task_id = str(uuid.uuid4())[:8]
 self.tasks[task_id] = {
 "command": command,
 "status": "running",
 "result": None,
 }

 thread = threading.Thread(
 target=self.execute,
 args=(task_id, command),
 daemon=True,
 )
 thread.start()

 return f"Background task {task_id} started: {command}"
```

`daemon=True` 表示守护线程。主程序退出时，守护线程会自动终止，避免进程因为后台线程还没结束而挂住。代价是如果主程序退出太快，后台任务可能来不及收尾；生产系统通常还需要更完整的任务生命周期管理。

后台线程：执行命令并记录结果Python

```text
def execute(self, task_id, command):
 try:
 result = subprocess.run(
 command,
 shell=True,
 capture_output=True,
 text=True,
 timeout=300,
 )
 status = "completed"
 output = result.stdout or result.stderr
 except Exception as exc:
 status = "failed"
 output = str(exc)

 with self.lock:
 self.tasks[task_id]["status"] = status
 self.tasks[task_id]["result"] = output
 self.notification_q.append({
 "task_id": task_id,
 "status": status,
 "result": output,
 })
```

# 4. 主循环 drain 通知队列，再注入 messages {#notify}
<!-- tag: pull notifications -->

每轮 Agent loop 开始时，会先调用 `drain_notifications()`。drain 的意思“取出并清空”。它把当前队列里的通知全部拿出来，然后把队列清空，避免同一条完成通知重复注入。

如果拿到通知，主循环会把它们格式化成 `Background results`，追加到 messages。这样下一次发给模型时，模型不仅看到用户的新 prompt，也看到上一轮后台任务完成了什么。

正确示例：取出并清空通知队列Python

```text
def drain_notifications(self):
 with self.lock:
 notices = list(self.notification_q)
 self.notification_q.clear()
 return notices
```

后台任务时间线

启动任务
主循环继续
任务完成
下一轮注入


**1. background_run**
生成 task id，登记 running，启动 daemon thread。
**2. main loop**
立刻返回给模型和用户，不等待慢命令完成。
**3. execute**
后台线程执行 subprocess.run，完成后写入 notification_q。
**4. drain**
下一轮 Agent loop 取出通知，追加 Background results 到 messages。


任务启动后立刻返回
主线程只负责登记任务和启动线程，然后返回 task id。慢命令在后台线程里继续跑。

`check_background` 是主动查询状态的工具。如果不传 task id，可以列出所有后台任务；如果传入某个 task id，就返回这个任务的 command、status 和 result。

# 5. pull 模式、daemon 线程与通知类型 {#design}
<!-- tag: design choices -->

这套设计采用 pull，而不是 push。后台线程完成后不直接回调主循环，也不强行打断当前对话，只把结果放进 `notification_q`。主循环在每一轮开始时主动 drain 队列。这样后台线程不需要知道 Agent loop 的内部结构，主循环也能按自己的节奏处理结果。

这比回调式 push 更解耦。push 的问题是：后台线程什么时候把结果推给谁、主循环此时处于哪个阶段、是否会打断当前模型请求，都需要额外协调。队列式 pull 更朴素：完成事件先入队，下一轮自然被取走。

pull 队列后台线程只入队，主循环主动取，减少耦合。
daemon 线程主程序退出时自动结束，避免进程被后台线程挂住。
通知类型生产系统可给通知加 `type`，区分任务完成、失败、警告等事件。

## box soft
### Worked example：两个测试任务先后完成
如果同时启动 `task_a` 和 `task_b`，主循环不等待它们完成。`task_a` 先结束时只把结果放进 `notification_q`；当前模型请求不会被线程打断。下一轮 loop 开始时，`drain_notifications()` 取走 `task_a` 的通知并清空队列；如果 `task_b` 还在 running，`check_background(task_b)` 仍会显示 running。等 `task_b` 后续完成，它会在另一次 drain 中进入 messages。

## box warn
### 工程风险
- 短 task id 可能碰撞，生产系统要更稳。
- `subprocess.run(shell=True)` 需要命令安全检查。
- daemon 线程可能在主程序退出时被中断。
- 通知队列如果不 drain，会重复或堆积。
- 共享状态不加 lock，可能出现竞态条件。
