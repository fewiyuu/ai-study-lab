11 Agent团队与消息总线｜Claude Code 工作流

本节实现一个最小 Agent team：lead Agent 可以生成 teammate，例如 Alice；TeammateManager 把成员写进 `.teams/config.json`；MessageBus 把消息写进 `.teams/inbox/{name}.jsonl`；lead loop 和 teammate loop 每轮开始都会读取自己的 inbox，把消息注入 messages 后再请求模型。

1. Agent team 先要有团队状态和通信通道

team

多个 Agent 协作时，不能只靠“多开几个模型”。系统至少要知道团队里有哪些成员、成员负责什么角色、当前状态是什么，以及成员之间如何发送消息。本节用文件系统实现了一个简单团队底座。

团队系统需要保存

  • 成员名称，例如 Alice。
  • 成员角色，例如 coder。
  • 成员状态，例如 working / idle。
  • 成员线程引用。

团队系统需要通信

  • 向指定成员发送消息。
  • 读取自己的 inbox。
  • 读完后清空收件箱。
  • 把收件箱消息注入 messages。

你现在应该能做什么：看到一个多 Agent 系统时，先问“成员状态在哪里、消息怎么传、谁能生成谁”。

2. MessageBus：用 JSONL 文件做收件箱

inbox

MessageBus 初始化时创建 .teams/inbox 目录。发送消息时，它根据接收者名字找到对应文件，例如发给 lead 就写入 .teams/inbox/lead.jsonl。每条消息是一行 JSON，包含 type、from、content、timestamp 等字段。

消息字段的最小约定

  • type：区分普通消息、请求、响应或系统事件。
  • from：发送者，用来追溯来源。
  • to：接收者，便于调试和审计。
  • content：人类或模型可读的正文。
  • timestamp：排序和排查延迟。

验收清单

  • 发给谁就只写入谁的 inbox。
  • 每行 JSON 都能独立解析。
  • 读完 inbox 后不会重复注入。
  • 并发写入有锁、队列或追加策略。
  • 坏消息不会让整个 inbox 解析失败。

最小示例：写入 lead 收件箱JSONL

{"type":"message","from":"alice","content":"Hi, I am Alice, a coder.","timestamp":"2026-03-25T...Z"}

逐行拆解：发送消息Python

def send(to, sender, content, message_type="message"):
 message = {
 "type": message_type,
 "from": sender,
 "content": content,
 "timestamp": now_iso(),
 }
 inbox_path = inbox_dir / f"{to}.jsonl"
 with inbox_path.open("a", encoding="utf-8") as f:
 f.write(json.dumps(message, ensure_ascii=False) + "\n")
 return f"sent message to {to}"

正确示例：读取并清空 inboxPython

def read_inbox(name):
 inbox_path = inbox_dir / f"{name}.jsonl"
 if not inbox_path.exists():
 return []

 messages = [json.loads(line) for line in inbox_path.read_text().splitlines()]
 inbox_path.write_text("", encoding="utf-8")
 return messages

3. TeammateManager：生成队员并维护状态

config

spawn_teammate 工具要求模型返回 name、role 和 prompt。本地收到后，TeammateManager 会先检查成员是否已存在；不存在就把成员加入 config，状态设为 working，保存到 .teams/config.json，再创建一个线程运行 teammate_loop。

正确示例：config.jsonJSON

逐行拆解：生成 teammatePython

这里和子 Agent 有相似点：新队员不是拥有所有能力的完整主 Agent。它有自己的 system prompt 和工具集，并且工具集会被裁剪，避免它继续无限 spawn 新队员。

4. lead loop 与 teammate loop 的共同点和差异

two

lead Agent 的 loop 和 teammate loop 很像：每轮都先读自己的 inbox，把消息加入 messages，再请求模型；模型如果返回 tool use，就执行工具并把结果继续写回 messages。差异在于 system prompt、工具集和循环上限。

5. `/tm` 和 `/inbox` 是直接命令，不走 LLM

slash

main 函数对 /tm 和 /inbox 做了特殊处理。输入 /tm 时，程序直接读取 config 并列出团队成员；输入 /inbox 时，程序直接读取 lead 的 inbox。它们不会进入 agent loop，也不会请求模型。

/tm 输出

读取 .teams/config.json，显示 team name 和 members，例如 alice coder idle。

/inbox 输出

读取 .teams/inbox/lead.jsonl，显示消息，并在读取后清空。

Worked example：为什么要裁剪 teammate 工具

假设 Alice 是 coder，只负责实现一个小功能。如果 Alice 也能调用 spawn_teammate，她可能再生成 Bob，Bob 又生成 Carol，团队规模会在没有 lead 决策的情况下扩散。更稳的设计是：lead 拥有创建成员和查看全局状态的工具，teammate 只拥有完成自己任务、收发消息和报告结果所需的工具。

常见坑

  • 队员工具集没裁剪，导致 teammate 也能无限 spawn。
  • 读取 inbox 后不清空，导致消息重复进入上下文。
  • 成员状态只改内存，不保存 config。
  • JSONL 消息缺少 type/from/content，后续无法路由和解释。
  • 线程并发写 inbox 时没有考虑文件写入冲突。
