# 1. Agent team 先要有团队状态和通信通道 {#why}
<!-- tag: team basics -->

多个 Agent 协作时，不能只靠“多开几个模型”。系统至少要知道团队里有哪些成员、成员负责什么角色、当前状态是什么，以及成员之间如何发送消息。本节用文件系统实现了一个简单团队底座。

目录结构很直接：`.teams/config.json` 保存 team name 和 members；`.teams/inbox/` 存每个成员的收件箱文件。lead 是主 Agent，teammate 是被生成的队员，每个队员运行在独立线程里，有自己的 prompt、自己的工具集和自己的收件箱。

## box
### 团队系统需要保存
- 成员名称，例如 Alice。
- 成员角色，例如 coder。
- 成员状态，例如 working / idle。
- 成员线程引用。

## box warn
### 团队系统需要通信
- 向指定成员发送消息。
- 读取自己的 inbox。
- 读完后清空收件箱。
- 把收件箱消息注入 messages。

你现在应该能做什么：看到一个多 Agent 系统时，先问“成员状态在哪里、消息怎么传、谁能生成谁”。

# 2. MessageBus：用 JSONL 文件做收件箱 {#bus}
<!-- tag: inbox jsonl -->

MessageBus 初始化时创建 `.teams/inbox` 目录。发送消息时，它根据接收者名字找到对应文件，例如发给 lead 就写入 `.teams/inbox/lead.jsonl`。每条消息是一行 JSON，包含 type、from、content、timestamp 等字段。

读取 inbox 时，它会读出所有行并解析成消息列表，然后清空这个文件。这和上一节后台任务的 drain 通知队列很像：取出后清空，避免同一条消息反复注入。

## box soft
### 消息字段的最小约定
- `type`：区分普通消息、请求、响应或系统事件。
- `from`：发送者，用来追溯来源。
- `to`：接收者，便于调试和审计。
- `content`：人类或模型可读的正文。
- `timestamp`：排序和排查延迟。

## box
### 验收清单
- 发给谁就只写入谁的 inbox。
- 每行 JSON 都能独立解析。
- 读完 inbox 后不会重复注入。
- 并发写入有锁、队列或追加策略。
- 坏消息不会让整个 inbox 解析失败。

最小示例：写入 lead 收件箱JSONL

```text
{"type":"message","from":"alice","content":"Hi, I am Alice, a coder.","timestamp":"2026-03-25T...Z"}
```

逐行拆解：发送消息Python

```text
def send(to, sender, content, message_type="message"):
 message = {
 "type": message_type,
 "from": sender,
 "content": content,
 "timestamp": now_iso(),
 }
 inbox_path = inbox_dir / f"{to}.jsonl"
 with inbox_path.open("a", encoding="utf-8") as f:
 f.write(json.dumps(message, ensure_ascii=False) + "\n")
 return f"sent message to {to}"
```

正确示例：读取并清空 inboxPython

```text
def read_inbox(name):
 inbox_path = inbox_dir / f"{name}.jsonl"
 if not inbox_path.exists():
 return []

 messages = [json.loads(line) for line in inbox_path.read_text().splitlines()]
 inbox_path.write_text("", encoding="utf-8")
 return messages
```

# 3. TeammateManager：生成队员并维护状态 {#manager}
<!-- tag: config + thread -->

`spawn_teammate` 工具要求模型返回 `name`、`role` 和 `prompt`。本地收到后，TeammateManager 会先检查成员是否已存在；不存在就把成员加入 config，状态设为 working，保存到 `.teams/config.json`，再创建一个线程运行 `teammate_loop`。

队员完成自己的 loop 后，状态会从 working 改成 idle 并保存回 config。这样 `/tm` 查看团队时，就能看到 Alice 是 coder，当前 idle。状态落到 config 文件中。

正确示例：config.jsonJSON

```text
{
 "team_name": "default",
 "members": [
 {"name": "alice", "role": "coder", "status": "idle"}
 ]
}
```

逐行拆解：生成 teammatePython

```text
def spawn_teammate(name, role, prompt):
 if self.find_member(name):
 return f"{name} already exists"

 member = {"name": name, "role": role, "status": "working"}
 self.config["members"].append(member)
 self.save_config()

 thread = threading.Thread(target=teammate_loop, args=(name, role, prompt))
 self.threads[name] = thread
 thread.start()
 return f"spawned teammate {name}"
```

这里和子 Agent 有相似点：新队员不是拥有所有能力的完整主 Agent。它有自己的 system prompt 和工具集，并且工具集会被裁剪，避免它继续无限 spawn 新队员。

# 4. lead loop 与 teammate loop 的共同点和差异 {#loop}
<!-- tag: two loops -->

lead Agent 的 loop 和 teammate loop 很像：每轮都先读自己的 inbox，把消息加入 messages，再请求模型；模型如果返回 tool use，就执行工具并把结果继续写回 messages。差异在于 system prompt、工具集和循环上限。

lead 的 system prompt 是 team leader，工具集更强，可以 spawn teammate、list teammates、send message 等。teammate 的 prompt 是创建时传入的角色任务，工具集更小，通常只能收发消息或读取 inbox，不能再生成新 teammate。teammate loop 也有限轮，例如 50 轮。

### 流程图式文字
lead 调用 `spawn_teammate` → config 增加 Alice → Alice 线程启动 → Alice loop 读取 `alice.jsonl` → Alice 执行任务并调用 `send_message(to=lead)` → MessageBus 追加一行到 `lead.jsonl` → lead 下一轮读取自己的 inbox → 消息被注入 lead messages → lead 根据新信息继续决策。

# 5. `/tm` 和 `/inbox` 是直接命令，不走 LLM {#slash}
<!-- tag: slash commands -->

main 函数对 `/tm` 和 `/inbox` 做了特殊处理。输入 `/tm` 时，程序直接读取 config 并列出团队成员；输入 `/inbox` 时，程序直接读取 lead 的 inbox。它们不会进入 agent loop，也不会请求模型。

这种设计很实用：团队状态和收件箱是系统状态，用户有时只想查看，不需要模型解释。直接命令更快、更稳定，也避免把一个简单查询变成模型推理。

## box soft
### `/tm` 输出
读取 `.teams/config.json`，显示 team name 和 members，例如 `alice coder idle`。

## box soft
### `/inbox` 输出
读取 `.teams/inbox/lead.jsonl`，显示消息，并在读取后清空。

## box soft
### Worked example：为什么要裁剪 teammate 工具
假设 Alice 是 coder，只负责实现一个小功能。如果 Alice 也能调用 `spawn_teammate`，她可能再生成 Bob，Bob 又生成 Carol，团队规模会在没有 lead 决策的情况下扩散。更稳的设计是：lead 拥有创建成员和查看全局状态的工具，teammate 只拥有完成自己任务、收发消息和报告结果所需的工具。

## box warn
### 常见坑
- 队员工具集没裁剪，导致 teammate 也能无限 spawn。
- 读取 inbox 后不清空，导致消息重复进入上下文。
- 成员状态只改内存，不保存 config。
- JSONL 消息缺少 type/from/content，后续无法路由和解释。
- 线程并发写 inbox 时没有考虑文件写入冲突。
