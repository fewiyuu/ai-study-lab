# 1. 团队协议让协作可确认 {#protocol}
<!-- tag: request / response -->

第 11 块的 Agent team 已经有成员和消息总线，但那只是通信底座。团队协议是在通信之上定义“这类消息应该怎样被处理”。关机协议就是其中一种：lead 创建一个请求，等待 teammate 响应，再根据响应更新状态。

这种设计的好处是可追踪。请求有 `request_id`，状态从 pending 变成 approved 或 rejected；队员退出前可以发送确认；lead 后续可以查询请求状态。协议让团队动作不再是散乱消息，而是一条有状态的流程。

## box warn
### 没有协议的问题
- lead 不知道队员是否真的收到消息。
- 队员是否退出没有确认。
- 状态只靠自然语言猜。
- 多个请求时难以对应。

## box
### 关机协议提供
- 唯一 request id。
- pending / approved / rejected 状态。
- teammate 主动响应。
- lead inbox 中的确认消息。

FSM 是 finite state machine，也就是有限状态机。它强调系统只允许在有限状态之间按规则跳转。

# 2. `shutdown_request`：lead 发起关机请求 {#request}
<!-- tag: pending request -->

当用户要求“spawn Alice as coder and request her shutdown”，lead 先创建 Alice，然后再调用 `shutdown_request`。本地函数会生成 request id，把请求登记到 `shutdown_requests` 字典里，初始状态是 pending，target 是 Alice。

随后 MessageBus 把一条 shutdown 请求写入 Alice 的 inbox。消息里除了 type、from、content、timestamp，还会带上 request_id。这个 id 很重要：Alice 后续响应时，必须说明自己响应的是哪一个请求。

## box soft
### 流程图式文字
1. lead 创建 `request_id`，登记 pending 请求。
2. lead 把 `shutdown_request` 写入 Alice inbox。
3. Alice 下一轮读取 inbox，把请求交给模型判断。
4. Alice 调用 `shutdown_response`，返回 approve 和 reason。
5. 系统更新请求状态，并把确认写回 lead inbox。
6. 若批准，Alice 设置 `should_exit`，自然退出 loop。

## box
### 协议验收清单
- 每个请求都有唯一 `request_id`。
- 请求状态只在合法状态间跳转。
- 批准和拒绝都要给 lead 回信。
- 线程退出前有机会保存状态和结果。
- 成员最终状态能在 config 中查到。

逐行拆解：创建 shutdown requestPython

```text
def shutdown_request(teammate):
 request_id = str(uuid.uuid4())[:8]

 with lock:
 shutdown_requests[request_id] = {
 "target": teammate,
 "status": "pending",
 }

 bus.send(
 to=teammate,
 sender="lead",
 message_type="shutdown_request",
 content="Please shutdown gracefully.",
 extra={"request_id": request_id},
 )
 return f"shutdown request {request_id} sent to {teammate}"
```

Alice 收件箱中的消息JSONL

```text
{"type":"shutdown_request","from":"lead","content":"Please shutdown gracefully.","request_id":"a13f29c0"}
```

这里用 lock 是为了保护 `shutdown_requests`。团队系统里 lead 和 teammate 可能运行在不同线程，状态字典被同时读写时要避免竞态。

# 3. `shutdown_response`：Alice 读 inbox 后响应 {#response}
<!-- tag: approve / reject -->

Alice 的 teammate loop 每轮开始会读自己的 `alice.jsonl`。读到 shutdown request 后，这条消息被加入 Alice 的 messages。模型看到“请优雅关机”和 request_id，应该调用 `shutdown_response`，输入包括 `request_id`、`approve` 和 `reason`。

如果 `approve` 为 true，执行函数会把 `shutdown_requests[request_id].status` 从 pending 改为 approved，并设置 `should_exit = True`。这让 teammate loop 在当前工具结果处理完后自然 break。

正确示例：Alice 的响应工具调用JSON

```text
{
 "name": "shutdown_response",
 "input": {
 "request_id": "a13f29c0",
 "approve": true,
 "reason": "Acknowledged. I will shut down gracefully."
 }
}
```

逐行拆解：处理 shutdown_responsePython

```text
def handle_shutdown_response(sender, request_id, approve, reason):
 with lock:
 req = shutdown_requests.get(request_id)
 if not req:
 return False, "unknown request"

 req["status"] = "approved" if approve else "rejected"

 bus.send(
 to="lead",
 sender=sender,
 message_type="shutdown_response",
 content=reason,
 extra={"request_id": request_id, "approved": approve},
 )
 return approve, reason
```

如果 Alice 拒绝，状态会变成 rejected，teammate loop 不应该退出。拒绝也要回信给 lead，因为 lead 需要知道请求没有成功。

# 4. FSM：状态跳转比自然语言更可靠 {#fsm}
<!-- tag: finite state machine -->

关机协议可以看成一个小型有限状态机。请求刚创建时是 pending；Alice 同意后变成 approved；Alice 拒绝后变成 rejected。teammate 成员状态也会变化：Alice 创建时是 working，正常完成任务可变 idle，如果批准关机并退出 loop，则保存为 shutdown。

当前状态触发事件下一状态必须产生的副作用

无请求lead 调用 `shutdown_request`pending写入请求字典和 Alice inbox。
pendingAlice approve=trueapproved设置 `should_exit`，回信 lead。
pendingAlice approve=falserejected保存拒绝原因，回信 lead。
approvedteammate loop 收尾完成shutdown把成员状态保存到 config。

状态机切换

请求已发送
Alice 批准
线程退出
Alice 拒绝


pending→
approved→
shutdown

拒绝路径：pending → rejected，成员状态不进入 shutdown。


请求等待响应
lead 已经创建 request，并把 shutdown_request 写入 Alice inbox。此时请求状态是 pending，Alice config 仍是 working。

状态机的价值是减少含糊。系统不需要猜“Alice 是看字段：request status 是 approved 还是 pending，member status 是 working 还是 shutdown。

# 5. lead 通过 inbox 和查询工具确认结果 {#confirm}
<!-- tag: confirmation -->

Alice 批准关机后，除了设置 `should_exit`，还会通过 MessageBus 给 lead 发送一条 shutdown_response 消息。lead 下一轮 loop 开头读取 `lead.jsonl`，就能把“Alice 已确认关机”的消息加入 messages。这样模型后续决策时知道队员已经退出。

还有一个工具用于查询 shutdown status。lead 可以查看 `shutdown_requests` 字典，知道某个请求仍然 pending，还是已经 approved / rejected。视频里先从主线程角度 debug 时，因为 Alice 线程没有真正跑起来，所以 request 还停在 pending，Alice config 还是 working；放开线程后，Alice 才会读 inbox、响应、退出并变成 shutdown。

## box soft
### Worked example：pending 不等于失败
lead 查询到 request 仍是 pending 时，不能立刻断定 Alice 拒绝了。更合理的排查顺序是：Alice 线程是否启动；Alice 是否读到了自己的 inbox；消息里是否带了正确 `request_id`；Alice 是否调用了 `shutdown_response`；响应是否成功写回 lead inbox。只有这些环节都正常，才继续看业务层面的拒绝原因。

主线程视角创建请求、写入 Alice inbox、可查询 pending。
Alice 线程视角读取 inbox、调用 response、设置 should_exit。
确认视角lead inbox 收到响应，config 中 Alice 变 shutdown。

## box warn
### 常见坑
- 只发 shutdown 消息，不记录 request id。
- Alice 响应后没有回信给 lead。
- 批准后直接杀线程，没有让 loop 自然退出。
- `should_exit` 为 true 后没有把成员状态保存为 shutdown。
- 拒绝请求也没有记录 rejected 状态。
