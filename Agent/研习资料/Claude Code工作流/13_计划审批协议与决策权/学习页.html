13 计划审批协议与决策权｜Claude Code 工作流

计划审批协议用于高风险或需要把关的任务：teammate 不能直接开干，而是先提交 plan 给 lead；lead 查看计划后批准或拒绝，并把结果写回 teammate inbox。它和关机协议结构相似，但方向相反：关机是 lead 发起、teammate 决策；计划审批是 teammate 发起、lead 决策。

1. 计划审批和关机协议方向相反

decision

团队协议的重点谁发起请求、谁拥有决策权。关机协议中，lead 请求 teammate 关机，但 teammate 可以 approve 或 reject，因为它需要决定自己能否优雅退出。

协议发起方决策方批准后的动作拒绝后的动作

你现在应该能做什么：看到一个团队协议时，先画出“发起方 → 决策方 → 结果回传”的方向，而不是只看工具名。

2. teammate 用 `plan_approval` 提交计划

submit

示例里 lead 生成 Charlie，并让 Charlie 提交一个计划。Charlie 的 teammate loop 根据自己的 prompt 调用 plan_approval 工具。工具 schema 要求模型返回 plan 字段，这就是提交给 lead 审批的具体计划内容。

一个可审批 plan 应包含

  • 目标：要完成什么，不做什么。
  • 步骤：执行顺序和关键工具。
  • 风险：可能破坏什么或卡在哪里。
  • 回滚：失败后如何恢复。
  • 验收：怎样判断计划执行完成。

流程图式文字

Charlie 生成 plan → plan_approval 创建 request_id → 请求进入 plan_requests pending → 消息写入 lead inbox → lead 读取并审批 → 状态变 approved 或 rejected → 结果写回 Charlie inbox → Charlie 执行或修订。

正确示例：Charlie 提交计划JSON

{
 "name": "plan_approval",

 "plan": "I will inspect the codebase, identify risky files, propose edits, then wait for approval before writing changes."
 }
}

 request_id = str(uuid.uuid4())[:8]

 with lock:

 }

这里的 request tracker 和关机协议很像，只是保存的是 plan request，而不是 shutdown request。

3. lead 审批后把结果写回 teammate inbox

approve

lead 的 agent loop 读到 lead.jsonl 中的 plan request 后，请求模型判断是否 approve。视频示例把 prompt 改成 reject it，所以模型调用审批处理工具时，approve 是 false。于是 request status 从 pending 变成 rejected。

正确示例：lead 拒绝计划JSON

逐行拆解：更新审批状态并回信Python

注意：视频里 Charlie 这个 teammate loop 很快 end_turn 了，所以回信写入了 Charlie inbox，但 Charlie 未必立刻继续处理。真实系统里 teammate 往往会持续运行，下一轮读到拒绝后修改 plan 再提交。

4. 审批状态机：pending → approved / rejected

FSM

计划审批也是一个小型 FSM。request 创建时是 pending；lead 同意则变 approved；lead 拒绝则变 rejected。和关机协议不同的是，approved 不代表 teammate 退出，而是代表 teammate 可以继续按计划执行；rejected 则代表需要修订计划或等待新指令。

提交计划批准拒绝修订重提
pending→approved/rejected

状态字段让系统不必从自然语言里猜“lead 是不是同意了”。只要看 plan_requests[request_id].status 就能知道当前审批结果。

5. 为什么需要计划审批

risk

计划审批的核心价值是风险闸门。teammate 可以主动思考方案，但不能对高风险动作直接开干。它要先把 plan 发给 lead，让 lead 判断是否满足目标、是否足够具体、是否考虑了风险和回滚。

Worked example：拒绝不是终点

Charlie 提交“重写登录模块”的计划，但没有说明测试和回滚。lead 拒绝并写明原因：“缺少影响范围、测试步骤和回滚方案”。Charlie 下一轮读取自己的 inbox 后，不应该继续执行旧计划，而应补充新计划：限定只改认证入口、先加回归测试、保留旧路径开关、失败时回滚到原实现。这样 rejected 状态就变成提升计划质量的反馈。

保护高风险操作写文件、删文件、发消息、调用外部系统前先过审。

提高计划质量lead 可以要求补充步骤、风险、验收和回滚。

明确责任边界teammate 提案，lead 决策，结果回传。

常见坑

  • teammate 提交 plan 后直接执行，没有等待 lead 审批。
  • lead 更新状态后没有把结果发回 teammate inbox。
  • 拒绝原因太空，teammate 无法修订计划。
  • 多个 plan request 没有 request_id，审批结果对不上。
  • 把 shutdown 和 plan approval 的决策方搞反。
