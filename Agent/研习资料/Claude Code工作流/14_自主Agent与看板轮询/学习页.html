本节把 teammate 从“等 lead 发消息”推进到“自己扫描看板接任务”。lead 创建 `.tasks/task_N.json` 看板并 spawn Alice；Alice 的 autonomous loop 会自己查看 `.tasks`，读取任务 JSON，claim 可做任务，完成后进入 idle 状态，并继续按间隔轮询 inbox 和任务目录。如果一段时间没有新任务，才把状态改成 shutdown。

1. 自主性来自“自己发现任务”

pull

上一阶段的 teammate 主要靠 lead 发消息驱动：lead 给 Alice 发指令，Alice 读 inbox 后执行。本节的自主 Agent 不再等 lead 指派具体任务，而是自己扫描 .tasks 目录，从任务看板中找可以做的任务。

被动 teammate

  • 主要等待 lead 发消息。
  • 执行完一轮可能结束。
  • 任务选择由 lead 明确给出。

自主 Agent

  • 自己扫描 .tasks 看板。
  • 主动认领可执行任务。
  • 完成后进入 idle 并继续轮询。

你现在应该能做什么：判断一个 Agent 是“被动执行指令”，还是“能从任务源主动拉取工作”。

3. `claim_task` 防止多个 Agent 抢同一任务

atomic

当 Alice 选择一个任务后，会调用 claim_task(task_id)。这个函数不是简单写 owner，它会先在 lock 保护下读取任务文件，检查 owner 是否为空、status 是否 pending、blocked_by 是否为空。三项都通过，才把 owner 改成 Alice，status 改成 in_progress，并写回文件。

认领算法

  1. 读取候选任务 JSON。
  2. 确认 status 是 pending。
  3. 确认 owner 为空。
  4. 确认 blocked_by 中没有未完成前置。
  5. 在 lock 内写入 owner 和 in_progress。
  6. 失败时返回明确原因，而不是静默跳过。

可认领任务验收

  • 不会认领已 completed 的任务。
  • 不会抢占已有 owner 的任务。
  • 不会越过未完成依赖。
  • 并发认领只能成功一个 Agent。
  • 失败信息能帮助模型改选任务。

逐行拆解：claim_task 的核心检查Python

def claim_task(task_id, agent_name):
 with task_lock:
 task = load_task(task_id)

 if task["owner"]:
 return "already owned"
 if task["status"] != "pending":
 return "not pending"
 if task["blocked_by"]:
 return "blocked by other tasks"

 task["owner"] = agent_name
 task["status"] = "in_progress"
 save_task(task)
 return "claimed"

视频里 Alice 先 claim task1 和 task2，因为它们没有依赖。task3 是 A，blocked_by: [1]，第一次 claim 会失败。等 B 完成后，模型需要更新 task3 的 blocked_by，再重新 claim。

4. idle 不等于结束：还要继续轮询

idle

Alice 完成所有当前任务后，会调用 idle 工具。这里的 idle 进入空闲状态。系统把 Alice 在 config 中的 status 设为 idle，然后开始一段轮询窗口。

逐行拆解：idle 后继续轮询Python

这就是 autonomous loop 的关键：做完一批任务后，不马上退出，而是继续观察任务源。它从 lead 推任务，变成自己 pull 任务。

5. 依赖解除不能只靠“模型记得”

dependency

视频里有个很有价值的调试点：B 完成后，A 的 blocked_by: [1] 理论上应该解除。模型第一次直接 claim A 失败，因为 blocked_by 仍然非空。后面它修改 task3，把 blocked_by 清空，再 claim 成功。

Worked example：依赖解除的系统做法

task1 是 B，task3 是 A，A 的 blocked_by 是 [1]。当 Alice 把 task1 标记为 completed 时，系统可以扫描所有 pending 任务：如果某个任务的 blocked_by 包含 1，就检查 task1 是否 completed；确认完成后把 1 从 blocked_by 移除并保存。这样 Alice 下一轮扫描看板时，A 自然变成可认领任务，不需要模型记住去手动改 JSON。

脆弱做法

模型记得 B 已完成，然后手动改 A 的 blocked_by。如果模型忘了，A 会一直被阻塞。

更稳做法

系统在完成任务或 claim 前自动检查依赖，把 completed 前置转化为可认领条件。

错误示例：忽略依赖字段直接认领依赖违规

task["owner"] = "alice"
task["status"] = "in_progress"

# 问题：没有检查 blocked_by，A 可能在 B 未完成时被执行。
