15 Git Worktree任务隔离｜Claude Code 工作流

本节学习 Git Worktree：在同一个 Git 仓库下创建多个工作目录，每个目录可以 checkout 不同分支。普通分支切换只能让你在一个目录里交替做任务；worktree 则让 `repo/`、`dev-branch-3/`、`dev-branch-4/` 同时存在，每个目录绑定自己的分支，适合人类或 AI coding agent 并行推进多个任务。

1. 普通分支切换只能“轮流做”

checkout

传统开发里，你通常在一个仓库目录中使用 git switch -c dev-branch-1 开始任务一。新任务来了，你再切回 main，基于主分支创建 dev-branch-2。这当然能工作，但同一时刻这个目录只能处在一个分支上。

普通 checkout

  • 一个工作目录只能对应一个当前分支。
  • 多任务需要频繁切换分支。
  • 未提交改动容易阻塞或污染切换。

Git Worktree

  • 一个仓库可以挂出多个工作目录。
  • 每个目录绑定不同分支。
  • 不同任务可以在不同目录同时推进。

你现在应该能做什么：识别“分支切换”解决的是版本线问题，而 worktree 进一步解决“工作目录隔离”问题。

2. Worktree 是“共享 Git 数据的独立工作目录”

one

git worktree 允许同一个仓库创建多个工作目录。每个工作目录看起来像一个独立项目目录，可以打开、编辑、运行、测试，也可以处在自己的分支上。但它们背后仍然共享同一个 Git 对象数据，不需要完整克隆多份仓库。

流程图式文字

已有仓库 → 选择起点分支 main → 为任务创建新分支 → 用 git worktree add 挂出新目录 → 在新目录运行编辑和测试 → 任务合并后 git worktree remove 清理目录 → 如需继续查看，用已有分支重新挂载。

隔离是否成立

  • 每个任务有独立工作目录。
  • 每个工作目录绑定明确分支。
  • 命令在对应目录中运行。
  • 删除 worktree 前确认改动已提交或不再需要。
  • 清理目录不误删远程分支或本地分支。

3. 最小命令流程：add、list、remove、恢复

commands

视频里先初始化一个仓库，创建 README.md 并提交到主分支。日常真实开发中，你更多是从远程 clone 一个现成 repo。无论是 git init 还是 git clone，只要你在一个 Git 仓库里，就可以使用 worktree。

代码
mkdir repo
cd repo
git init
git branch -M main
echo "hi readme" > README.md
git add README.md
git commit -m "init readme"

有了基线提交后，你可以创建一个新 worktree。命令一般写成：git worktree add <目录路径> -b <新分支名> <起点分支>。这里的目录路径是新工作目录的位置，分支名是这个目录要绑定的新分支，起点分支通常是 main。

正确示例：为新任务创建独立目录和分支Shell

如果你再创建 dev-branch-4，旁边就会多出另一个目录。此时你可以在 repo/ 里看主分支，在 dev-branch-3/ 里做任务三，在 dev-branch-4/ 里做任务四。它们真的有不同文件树。

代码
git worktree list

# 可能看到：
# /path/to/repo abc1234 [main]
# /path/to/dev-branch-3 def5678 [dev-branch-3]
# /path/to/dev-branch-4 901abcd [dev-branch-4]

任务完成、PR 合并后，可以删除本地 worktree 目录。注意：git worktree remove 主要移除工作目录登记和目录本身，不等于自动删除这个分支。视频里 remove 后再看 branch，分支仍然保留，这就是“删除目录不影响分支，可随时恢复”的意思。

删除 worktree 目录，但分支仍在Shell

如果误删或想重新打开这个分支，不需要 -b 创建新分支。因为分支已经存在，只需要把现有分支重新挂成一个 worktree。

代码
git worktree add ../dev-branch-3 dev-branch-3

cd ../dev-branch-3
git branch --show-current
# dev-branch-3

你现在应该能做什么：为一个新任务创建 worktree，为已有分支恢复 worktree，并用 git worktree list 检查当前隔离目录。

4. 为什么 AI coding agent 特别需要 worktree

agent

人类开发者也会受益于 worktree，但 AI coding agent 的收益更直接。因为 agent 可以同时接多个任务、执行多个命令、修改多个文件。如果它们共用一个目录，任务之间很容易相互污染：一个 agent 切了分支，另一个 agent 的上下文就错了；一个 agent 改了文件，另一个 agent 的测试结果就不可靠。

场景不用 worktree 的风险使用 worktree 的好处

这里也能看出本节和前面任务看板的连接：看板负责“有哪些任务”，worktree 负责“每个任务在哪个隔离环境里执行”。一个成熟的 agent harness 往往会把任务分配、工作目录、分支、提交和审计记录串起来。

5. 常见误区：目录、分支和仓库不是一回事

pitfalls

学习 worktree 最容易混淆三个层次：工作目录、Git 分支、仓库数据。工作目录是你能打开和编辑文件的地方；分支是 Git 中的一条提交指针；仓库数据是 Git 保存对象、提交和引用的底层数据。

因此，git worktree remove ../dev-branch-3 删除的是工作目录，不是自动把 dev-branch-3 分支也删掉。反过来，如果你想创建全新的任务分支，用 -b；如果分支已经存在，就不要再 -b，直接指定已有分支。

代码
# dev-branch-4 已经存在时，这样会失败或造成误解
git worktree add ../dev-branch-4 -b dev-branch-4 main

# 正确：直接挂载已有分支
git worktree add ../dev-branch-4 dev-branch-4

另一个误区是把 worktree 当作“多 clone 几份仓库”。多 clone 当然也能隔离目录，但每份 clone 都有自己的远程、配置和对象数据，管理成本更高。worktree 更适合从同一个本地仓库派生多个任务工作台。

Worked example：恢复已删除目录

你完成 dev-branch-3 后执行了 git worktree remove ../dev-branch-3。第二天需要重新查看这条分支，不要再写 -b dev-branch-3，因为分支已经存在。正确做法是 git worktree add ../dev-branch-3 dev-branch-3。这一步重新创建工作目录，但复用原来的分支指针。
