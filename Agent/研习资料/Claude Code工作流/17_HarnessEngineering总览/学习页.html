Harness engineering

Harness Engineering总览

本节是 Claude Code 工作流 第一阶段的收束：Claude Code 不是单纯把 LLM 放进循环，而是一整套 Harness Engineering。模型像动力源，Harness 像外部约束和协作系统：工具、上下文、权限、任务、审批、通信、隔离、审计、Skill，都在让模型朝正确方向行动。

LLM+ Harness= Agent

学完本页，你应该能用统一框架串联前面章节，并迁移到 OpenCode、Codex、Claude Code 或其他 Agent 源码。

重点：Agent = Model + Harness，五个正交关注点，Skill 作为可复用能力包。

1. Agent 不是模型本身，而是模型加 Harness

model + harness

本节的核心句子是：Agent = LLM + Harness。LLM 是语言模型，负责理解、生成、推理、选择工具调用；Harness 是模型之外的一切工程结构，负责约束、补充、校验、授权、记录和调度。

如果只有模型，它可能给出不稳定的 tool use block：该调用工具时没调用，不该调用时乱调用，参数不合法，权限不安全，任务状态不可追踪。Harness 的作用不是替代模型，而是把模型的能力接到可控环境里，让它能真正完成任务。

可以把边界记成一句话：模型负责“下一步该说什么、该请求什么”，Harness 负责“这一步是否允许、如何执行、结果如何回到上下文、失败后如何恢复”。所以评估一个 Agent 不只看模型输出，还要看外部工程结构是否把输出变成可靠行动。

抽象公式：Agent 的最小组成Concept
Agent = LLM + Harness

LLM:
  - 理解用户意图
  - 生成自然语言
  - 选择工具调用
  - 根据工具结果继续推理

Harness:
  - 管理工具和上下文
  - 控制权限和审批
  - 持久化任务和状态
  - 调度子 Agent 与后台任务
  - 隔离工作目录并记录审计日志

你现在应该能做什么：看到一个 Agent 系统时，先分清“模型能力”和“Harness 工程能力”。

2. 五类正交关注点组成完整 Harness

orthogonal concerns

所谓正交关注点，就是彼此相对独立、但组合后形成完整系统的工程面向。工具管理、上下文管理、权限安全、任务协作、执行隔离可以分别设计，也可以组合成一个 Agent Harness。

工具管理

说明会显示在这里。

工具：让模型能做事
上下文：让模型知道该怎么做
权限：让危险动作先被约束
任务：让复杂工作可拆、可委托
隔离：让多个执行互不干扰且可追踪
这五类不是唯一分类，但足够作为阅读 Agent 源码的第一张地图。
关注点前面章节里的对应物解决的问题
工具tool use、专用工具、多工具分发模型如何从“说话”变成“行动”。
上下文消息历史、Skill、上下文压缩模型如何获得足够信息，又不被上下文撑爆。
权限计划审批、操作审批、安全路径高风险动作如何先被人或策略把关。
任务协作todo、`.tasks`、子 Agent、团队消息复杂工作如何拆分、委托、同步状态。
隔离审计Git Worktree、index、events.jsonl多个任务如何互不干扰，并能追踪结果。

术语边界

正交不是互不相关，而是可以分别推理。例如权限策略可以先独立设计，再套到 Bash、MCP、后台任务等不同工具来源上。

阅读源码流程

先找主循环，再找工具注册和消息结构；接着找权限、任务状态、持久化、隔离目录和日志。这样能快速判断项目的 Harness 成熟度。

3. 从 12 个模块回看 Claude Code Harness

series map

整个系列从最小 Agent 循环开始：本地启动、消息历史、工具调用、多工具、规划。这些是单 Agent 的基础。之后进入能力扩展：子 Agent、Skill、上下文压缩、任务持久化、后台任务。最后进入团队系统：消息总线、关机协议、计划审批、自主领任务、Git Worktree 隔离。

如果把它们放进 Harness Engineering 视角，就不是一堆零散功能，而是一条递进路径：先让模型能循环，再让它能行动；先让行动可控，再让任务可拆；先让协作可通信，再让执行可隔离和审计。

从零到可用

  1. 本地启动最小 Agent。
  2. 消息历史和循环。
  3. 工具调用和分发。
  4. 行动前规划。

从可用到可靠

  1. 子 Agent 和 Skill。
  2. 上下文压缩和任务持久化。
  3. 后台任务、团队通信和协议。
  4. 自主任务、Worktree 隔离和审计。
Worked example：把“自动修复测试”拆成 Harness 部件Design
用户目标：修复失败测试

LLM:
  - 读懂失败日志
  - 提出修改方案
  - 选择 read_file / edit_file / run_tests

Harness:
  - 只允许读取当前项目文件
  - 写入前展示 diff 或请求审批
  - 在隔离分支或工作目录运行测试
  - 记录工具调用、测试结果和最终变更
  - 失败时保留现场并让模型改计划

你现在应该能做什么：把一个 Agent 功能放回“循环、工具、上下文、任务、权限、隔离”这些大类中，而不是只记章节名。

4. Skill 是 Harness 的可复用知识与动作包

skills

视频后半段提到 skills.sh 和 frontend-slides skill。这里的重点不是 PPT 风格，而是 Skill 的工程意义：当你希望 Agent 做某类复杂工作时，不必每次把所有规则塞进 prompt，而是安装或编写一个 Skill，让 Agent 按技能说明加载流程、模板、工具和约束。

Skill 属于 Harness 的上下文和工具扩展层。它把“怎么做一类事”沉淀成可复用包：例如代码迁移、日志诊断、接口联调、性能分析或安全审查。模型仍然负责理解具体任务，但 Skill 给了它稳定方法论和资源。

一个可复用的 Skill 通常只描述通用流程、输入输出、依赖工具、检查清单和失败处理。它不应该依赖项目专属目录、个人化资料库或一次性任务，否则换一个项目就无法复用。

Skill 的角色：把一次性提示词变成可复用能力Concept
Without skill:
  用户每次都要解释风格、流程、文件结构、检查规则。

With skill:
  Agent 读取 SKILL.md
  -> 获取流程和约束
  -> 复用模板和脚本
  -> 产出更稳定
  -> 下次任务可重复使用

你现在应该能做什么：判断一个复杂任务是应该靠一次性 prompt，还是应该沉淀成 Skill。

5. 常见误区：把 Agent 神秘化或把 Harness 看轻

pitfalls

最常见误区是把 Agent 的能力完全归因于模型。模型当然重要，但如果没有工具协议、上下文压缩、权限边界、任务持久化和执行隔离，它很难稳定完成真实工程任务。

只看模型

关注模型聪不聪明，忽略工具结果如何回填、危险操作如何审批、任务状态如何保存、多个任务如何隔离。

看 Harness

关注模型之外的外部组件如何约束和放大模型能力，让输出变成可执行、可追踪、可恢复的工程过程。

错误示例：把所有问题都甩给模型设计缺失
如果模型误删文件：
  错误归因：模型不够聪明。

更工程化的追问：
  是否有权限系统？
  是否有路径白名单？
  是否有审批？
  是否有 dry-run？
  是否有审计日志？
  是否能恢复？

分析任意 Agent 产品时，可以用这个检查清单逐项对照：它怎样接入工具，怎样压缩上下文，怎样处理权限，怎样记录任务状态，怎样隔离执行环境，怎样把失败恢复成可继续的步骤。
