20 Hook系统｜Claude Code 工作流

Hook 是系统在固定主流程中预留的扩展口。Claude Code 工作流 这一节用 `.hooks.json` 配置 `session_start`、`pre_tool_use`、`post_tool_use` 三个时机，让你不修改 Agent loop 主体，也能在会话开始、工具执行前、工具执行后插入命令、记录日志、阻止操作或补充上下文。

1. Hook 是固定流程里的扩展口

extension

Hook 的直觉：框架把主流程定死，在关键节点留下口子，让使用者插入自定义逻辑，而不必修改框架源码。Vue 的生命周期、Android Activity 生命周期、测试框架的 setup/teardown、Web 框架 middleware、模板方法模式里的抽象步骤，都可以用这个直觉理解。

边界也要清楚：Hook 是旁路扩展，不是新的主流程。它可以观察、记录、阻断或补充上下文，但不应该悄悄替代 Agent loop 的工具执行、权限检查和消息回填。

你现在应该能做什么：用“固定主流程 + 预留扩展点”解释 Hook。

2. 本节三个 Hook 时机

events

代码里初始化了一个 hooks 字典，包含三个 key：pre_tool_use、post_tool_use、session_start。每个 key 对应一个列表，可以挂多个 Hook。Hook 配置来自 .hooks.json。

代码
{
 "session_start": [
 {"command": "echo session start ok"}
 ],
 "pre_tool_use": [
 {"matcher": "bash", "command": "echo before bash"},
 {"matcher": "read_file", "command": "echo before read"}
 ],
 "post_tool_use": [
 {"matcher": "*", "command": "echo post tool use ok"}
 ]
}

Hook触发时机典型用途

session_start会话开始时加载环境、记录启动、注入初始上下文。

pre_tool_use工具执行前检查、阻止、改写、记录即将执行的操作。

post_tool_use工具执行后记录结果、上报可观测性、补充消息。

3. `run_hooks` 如何运行 Hook

runtime

run_hooks(event, context) 是核心。它先创建 result：blocked=false、messages=[]。然后检查 workspace trust。如果当前工作区不可信，Hook 不运行。可信后，根据 event 取出对应 Hook 列表，按 matcher 判断是否匹配当前工具，再把 context 写入环境变量并运行 command。

session_start

说明会显示在这里。

  1. check workspace trust
  2. matcher 筛选
  3. 注入环境变量
  4. 执行 command
  5. 结果注入 messages/output

pre 可以阻止工具，post 可以把工具结果之后的额外信息补回上下文。

代码
result = {"blocked": False, "messages": []}

if not check_workspace_trust():
 return result

for hook in hooks[event]:
 if not matches(hook.get("matcher"), context):
 continue

 env["HOOK_EVENT"] = event
 env["HOOK_TOOL_NAME"] = context.get("tool_name", "")
 env["HOOK_TOOL_INPUT"] = json.dumps(context.get("tool_input", {}))
 env["HOOK_TOOL_OUTPUT"] = context.get("tool_output", "")[:10000]

 output = run_command(hook["command"], env=env)
 result = merge_hook_output(result, output)

Worked example：阻止访问密钥文件pre_tool_use

代码
context:
 event = "pre_tool_use"
 tool_name = "read_file"
 tool_input.path = ".env"

hook:
 matcher = "read_file"
 command = "check-sensitive-path"

hook output:
 {"blocked": true, "messages": ["Blocked reading .env because it may contain secrets."]}

Agent loop:
 不执行 read_file
 把 blocked message 作为工具结果或上下文反馈给模型
 让模型改用安全路径或请求用户确认

4. 信任标记、matcher、blocked 与 message

control

Hook 可以执行本地命令，所以必须先有工作区信任。视频中为了 debug 创建了类似 .claude/.claude_trusted 的标记文件，否则 check_workspace_trust 会拒绝运行 Hook。这是安全设计：不可信工作区里的 Hook 不应该随便执行命令。

Hook 命令输出如果是普通文本，可以仅打印或记录；如果是结构化 JSON，可以携带 blocked 和 messages。blocked=true 时，pre Hook 可以阻止工具执行；messages 可以补充到模型上下文，让模型知道 Hook 发现了什么。

结构化 Hook 输出：阻止危险操作JSON

你现在应该能做什么：判断一个 Hook 是记录型、增强上下文型，还是阻断型。

5. 常见误区：Hook 不是把所有逻辑塞进主循环

pitfalls

Hook 的价值是开闭原则：对主循环修改关闭，对扩展开放。你不需要改 Agent loop，就能在关键时机加日志、检查、通知、上下文注入、记忆写入或安全阻断。

Hook 做法

主循环只负责触发事件；扩展逻辑放进配置和 Hook command。

错误示例：不可信工作区直接运行 Hook安全风险

# 风险：任意项目里的 .hooks.json 都能执行本地命令
for hook in hooks[event]:
 run_command(hook["command"])

# 更稳：先检查 workspace trust，再运行 Hook。

设计 Hook 时还要避免隐藏副作用。记录型 Hook 应尽量只写日志；阻断型 Hook 要返回明确原因；增强上下文型 Hook 要控制输出长度。Hook 一旦能修改文件、发网络请求或执行命令，就必须纳入权限和审计视野。
