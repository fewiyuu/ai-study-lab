# 1. Memory 在 Agent loop 中动态读写 {#why}
<!-- tag: not static -->

`CLAUDE.md` 和 Skill 更像启动时加载的静态指导材料；Memory 不同，它是在 Agent loop 过程中读写的。开始时可能没有 memory，用户表达偏好后，模型会调用 `save_memory` 工具，把偏好保存下来。下一轮构建 system prompt 时，再把这些记忆注入给模型。

视频里的例子是：用户说自己喜欢 Python 文件里定义 `main` 函数。第一次模型可能先修改当前文件，再调用 `save_memory` 记录这个偏好。之后用户让生成 `greet.py`，即使没再次说明，模型也会生成带 `main()` 的 Python 文件。

术语边界：聊天历史保存当前会话的上下文，Memory 保存跨会话仍值得复用的稳定事实；项目配置描述当前代码库规则，Memory 描述用户偏好、长期约定或外部事实。三者都能影响模型，但生命周期和来源不同。

## box warn
### 无记忆
每轮像第一次合作，用户偏好需要反复说明。

## box
### 有记忆
跨会话保留偏好、反馈和项目参考，让 Agent 更像持续合作伙伴。

# 2. 保存记忆：写文件并重建索引 {#save}
<!-- tag: save_memory -->

`MemoryManager` 初始化时指定 `.memory` 目录，并维护内存中的 `memories` 字典。`save_memory(name, description, type, content)` 会先校验 memory type，再规范化 name，创建 `.memory` 目录，写入一个带 frontmatter 的 Markdown 文件。

记忆文件示例：Python main 偏好Markdown

```text
---
name: python_main_convention
description: User prefers main function in Python files
type: user
---

When creating Python files, always define a main function.
```

写入文件后，系统还会更新内存字典，并调用 `rebuild_index()`。索引文件 `.memory/memory.md` 像目录，列出当前有哪些 memory、类型和描述。这样模型可以先看到索引，再按需要引用具体记忆。

索引文件示例：memory.mdMarkdown

```text
# Memory Index

- python_main_convention [user]: User prefers main function in Python files
```

保存流程里有两个容易漏的细节。第一，name 要规范化成稳定文件名，避免同一偏好出现多个拼写版本。第二，保存后要重建索引，否则下一轮模型可能只看到旧目录，看不到刚写入的记忆。

# 3. 加载记忆：构建 system prompt 时注入 {#load}
<!-- tag: load prompt -->

每轮 Agent loop 开始时，`build_system_prompt` 会尝试加载 memory prompt。如果当前没有 memories，就只加入 memory guidance，告诉模型什么时候该调用 `save_memory`、什么时候不该保存。如果有 memory，就按类型分组，把 description 和 content 加入 system prompt。

无记忆
保存偏好
重建索引
注入提示词


无记忆启动
说明会显示在这里。
1. load_all
2. save_memory
3. rebuild_index
4. build_system_prompt
Memory 最终会占用上下文窗口，所以必须只存真正值得跨会话保留的信息。

Worked example：从用户反馈到下一轮行为Flow

```text
第 1 轮：
 用户：以后生成 Python 脚本时，请保留 main() 和 if __name__ == "__main__"。
 Agent：调用 save_memory，写入 python_entrypoint_preference。
 MemoryManager：写 .memory/python_entrypoint_preference.md 并重建 memory.md。

第 2 轮：
 用户：写一个统计 CSV 行数的脚本。
 build_system_prompt：注入这条用户偏好。
 Agent：生成带 main() 入口的 Python 脚本，不需要用户重复说明。
```

你现在应该能做什么：从 `.memory/*.md` 和 `.memory/memory.md` 反推出 system prompt 中会出现哪些记忆。

# 4. 什么值得进入 Memory {#criteria}
<!-- tag: criteria -->

视频强调：只有跨会话仍然有价值、且无法从当前工作区轻易推导出的知识，才值得进入 Memory。用户偏好就是典型例子：仓库里看不出用户是否喜欢 Python 文件必须有 `main()`，所以值得保存。

如果什么都存，Memory 会变成垃圾堆：越存越乱、越存越长，还会占用上下文窗口。更危险的是，过期记忆可能与当前真实状态冲突，让 Agent 产生幻觉式判断。

信息是否适合存原因

用户偏好 Python 文件使用 main 函数适合跨会话有价值，仓库不一定能推导。
当前目录有哪些文件不适合随时可读，且容易变化。
用户长期不喜欢直接改文件，先给计划适合稳定协作偏好。
刚刚工具调用返回的一次性错误通常不适合短期上下文即可，不应长期污染。

判断是否保存时可以用三问：下次还会用吗？当前仓库或工具能实时查到吗？如果它过期了会不会误导模型？只有前两个答案分别是“会、查不到”，且第三个风险可控，才适合进入 Memory。

# 5. 常见误区：Memory 不是聊天记录仓库 {#pitfall}
<!-- tag: pitfalls -->

Memory 的目标保存可复用的稳定知识。聊天历史用于当前会话，Memory 用于跨会话延续偏好和重要事实。上下文压缩解决“当前长会话太长”的问题，Memory 解决“下次不从零开始”的问题。

垃圾堆把所有临时信息都存入 memory。
过期记忆记忆不再真实，却继续影响模型。
上下文膨胀记忆太多，每轮 system prompt 都变长。

错误示例：把可实时读取的信息存成记忆不值得

```text
save_memory(
 name="current_file_list",
 type="project",
 content="The current folder has hello.py and greet.py"
)

# 问题：文件列表可以实时读取，而且会变化。
```

成熟的 Memory 系统还需要更新和删除。用户偏好改变时应覆盖旧记忆而不是叠加冲突版本；项目事实变化时应删除或标记过期；记忆太多时应按类型、更新时间和引用频率做整理，避免每轮 prompt 被历史负担拖慢。
