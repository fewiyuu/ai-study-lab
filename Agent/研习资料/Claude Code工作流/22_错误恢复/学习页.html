22 错误恢复｜Claude Code 工作流

真实 Agent 一定会遇到错误：模型输出被 max tokens 截断、上下文太长、网络抖动、API 超时、限流。如果没有恢复机制，主循环会在第一个错误处崩溃；有恢复机制后，系统会先判断错误类型，再选择继续、压缩或退避重试。

1. 错误不一定表示任务失败

robustness

一个常见误解是：Agent 失败就是模型不行。视频强调，很多失败这一轮需要换一种继续方式。输出被截断时可以让模型从停止处继续；上下文太长时可以先压缩；网络或限流时可以等一会儿再试。

恢复的边界是“尽量继续正确任务”，不是“任何错误都自动吞掉”。如果错误说明权限被拒绝、工具参数不合法或目标文件确实不存在，系统应把原因反馈给模型重新规划，而不是偷偷重试同一个动作。

有恢复

识别 max_tokens、overlong prompt、连接错误、限流，并尝试继续。

2. 三类恢复策略

strategy

本节代码在 Agent loop 中加入三类策略。第一类是 stop_reason == max_tokens，说明模型输出到一半被截断，系统追加 continuation message，让模型从停止处继续。第二类是 API error 中包含 overlong prompt，说明上下文超限，系统自动压缩历史。第三类是 connection error 或 rate limit，系统使用带随机抖动的 backoff sleep 后重试。

3. Loop 改造：try/catch + retry attempt

retry

早期 Agent loop 直接请求模型。错误恢复版在请求外包了一层 try/except，并用 for attempt in range(max_attempts) 限制重试次数。每次错误根据类型走不同分支。这样有针对性地恢复。

代码
for attempt in range(max_attempts):
 try:
 response = client.messages.create(messages=messages)
 break
 except APIError as e:
 if is_overlong_prompt(e):
 messages = auto_compact(messages)
 continue
 raise
 except (ConnectionError, RateLimitError):
 delay = backoff_delay(attempt) + random_jitter()
 time.sleep(delay)
 continue

请求成功后，还要检查 response.stop_reason。如果是 max_tokens，说明输出到达上限。这时追加一条用户消息，要求模型从停止处继续。

4. 自动压缩与一个设计疑点

auto

overlong prompt 表示请求太长。代码里的 auto compact 会把 messages 中一段 conversation text 发给模型，让模型生成 summary，再把 summary 包进 continuation message 继续执行。这个思路和前面上下文压缩一致：把完整历史压成可继续工作的摘要。

有疑问的做法

压缩最早 80000 字符，可能丢掉最近任务意图或最新错误现场。

更稳做法

保留最近消息和任务状态，把较早历史压缩成摘要。

设计风险：压缩错范围Context

conversation_text = serialize(messages)[:80000]

# 风险：截取的是最早部分。
# 恢复时可能更需要最近的用户指令、工具错误和当前任务状态。

更稳的 auto compact 输入通常至少保留四类信息：用户最新目标、当前计划或待办、最近工具结果、最近错误现场。较早闲聊和已经完成的中间过程可以压成摘要，但不能压掉“现在正在做什么”。

5. 常见误区：所有错误都重试

pitfalls

错误恢复不是无脑重试。可恢复错误可以恢复，不可恢复错误要明确停止并反馈原因。max_tokens 适合续写；overlong prompt 适合压缩；网络抖动适合退避重试；权限拒绝、工具参数错误、文件不存在等则可能需要反馈给模型改计划，而不是 sleep 后再试。

一个成熟系统还应该记录恢复次数、错误类型、延迟时间、压缩摘要和最终是否成功。这些信息对调试和信任感都很重要。
