# 1. 错误不一定表示任务失败 {#why}
<!-- tag: robustness -->

一个常见误解是：Agent 失败就是模型不行。视频强调，很多失败这一轮需要换一种继续方式。输出被截断时可以让模型从停止处继续；上下文太长时可以先压缩；网络或限流时可以等一会儿再试。

所以错误恢复的目标把“报错就崩溃”升级为“判断错误类型，再选择恢复路径”。这也是系统鲁棒性的一部分。鲁棒性指系统遇到异常输入、临时故障或边界情况时，仍能维持可接受行为。

恢复的边界是“尽量继续正确任务”，不是“任何错误都自动吞掉”。如果错误说明权限被拒绝、工具参数不合法或目标文件确实不存在，系统应把原因反馈给模型重新规划，而不是偷偷重试同一个动作。

## box warn
### 无恢复
第一次 API 错误、超时或截断就退出，用户只能重来。

## box
### 有恢复
识别 max_tokens、overlong prompt、连接错误、限流，并尝试继续。

# 2. 三类恢复策略 {#strategies}
<!-- tag: strategy -->

本节代码在 Agent loop 中加入三类策略。第一类是 `stop_reason == max_tokens`，说明模型输出到一半被截断，系统追加 continuation message，让模型从停止处继续。第二类是 API error 中包含 overlong prompt，说明上下文超限，系统自动压缩历史。第三类是 connection error 或 rate limit，系统使用带随机抖动的 backoff sleep 后重试。

恢复策略必须有上限，否则会变成无限重试或无限续写。

错误类型检测方式恢复动作

输出截断`stop_reason == "max_tokens"`追加 continuation message，最多恢复若干次。
上下文超限API error body 包含 overlong promptauto compact 历史，返回压缩后的 continuation。
网络/限流connection error 或 rate limitbackoff + jitter，等待后重试。

backoff 是逐步拉长等待时间，jitter 是给等待时间加少量随机抖动。两者常一起用：前者避免持续打爆服务，后者避免很多 Agent 在同一秒同时重试。

# 3. Loop 改造：try/catch + retry attempt {#loop}
<!-- tag: retry loop -->

早期 Agent loop 直接请求模型。错误恢复版在请求外包了一层 `try/except`，并用 `for attempt in range(max_attempts)` 限制重试次数。每次错误根据类型走不同分支。这样有针对性地恢复。

伪代码：分类恢复Python

```text
for attempt in range(max_attempts):
 try:
 response = client.messages.create(messages=messages)
 break
 except APIError as e:
 if is_overlong_prompt(e):
 messages = auto_compact(messages)
 continue
 raise
 except (ConnectionError, RateLimitError):
 delay = backoff_delay(attempt) + random_jitter()
 time.sleep(delay)
 continue
```

请求成功后，还要检查 `response.stop_reason`。如果是 `max_tokens`，说明输出到达上限。这时追加一条用户消息，要求模型从停止处继续。

max_tokens 续写消息Python

```text
if response.stop_reason == "max_tokens":
 max_output_recovery_count += 1
 if max_output_recovery_count <= 3:
 messages.append({
 "role": "user",
 "content": "Continue from where you stopped. Do not repeat previous content."
 })
 continue
 else:
 break
```

Worked example：一次失败如何选择恢复路径Decision

```text
现象 A：response.stop_reason = "max_tokens"
 判断：模型输出被截断，但请求成功。
 动作：追加 continuation message，最多续写 3 次。

现象 B：API error = "prompt is too long"
 判断：上下文超过服务端限制。
 动作：保留近期任务状态，压缩较早历史，再重发。

现象 C：tool_result = "Permission denied"
 判断：不是临时网络错误。
 动作：把拒绝原因给模型，让它改计划或请求用户授权。
```

# 4. 自动压缩与一个设计疑点 {#compact}
<!-- tag: auto compact -->

overlong prompt 表示请求太长。代码里的 auto compact 会把 messages 中一段 conversation text 发给模型，让模型生成 summary，再把 summary 包进 continuation message 继续执行。这个思路和前面上下文压缩一致：把完整历史压成可继续工作的摘要。

字幕里也指出一个值得注意的设计疑点：代码取的是前 80000 个字符，也就是较早的历史；但很多任务恢复更关心最近消息和用户最新要求。更稳的设计通常要保留最近上下文，压缩较早历史，或者按任务状态选择压缩范围。

## box warn
### 有疑问的做法
压缩最早 80000 字符，可能丢掉最近任务意图或最新错误现场。

## box
### 更稳做法
保留最近消息和任务状态，把较早历史压缩成摘要。

设计风险：压缩错范围Context

```text
conversation_text = serialize(messages)[:80000]

# 风险：截取的是最早部分。
# 恢复时可能更需要最近的用户指令、工具错误和当前任务状态。
```

更稳的 auto compact 输入通常至少保留四类信息：用户最新目标、当前计划或待办、最近工具结果、最近错误现场。较早闲聊和已经完成的中间过程可以压成摘要，但不能压掉“现在正在做什么”。

# 5. 常见误区：所有错误都重试 {#pitfall}
<!-- tag: pitfalls -->

错误恢复不是无脑重试。可恢复错误可以恢复，不可恢复错误要明确停止并反馈原因。max_tokens 适合续写；overlong prompt 适合压缩；网络抖动适合退避重试；权限拒绝、工具参数错误、文件不存在等则可能需要反馈给模型改计划，而不是 sleep 后再试。

无限重试没有 attempt 上限，可能卡死。
错误分类粗糙所有错误都 sleep，不解决根因。
恢复无日志用户不知道系统恢复过什么。

一个成熟系统还应该记录恢复次数、错误类型、延迟时间、压缩摘要和最终是否成功。这些信息对调试和信任感都很重要。

用户体验上，恢复过程应可见但不打扰：例如显示“连接错误，10 秒后第 2 次重试”或“上下文过长，正在压缩历史”。这样用户知道系统在处理问题，而不是无响应。
