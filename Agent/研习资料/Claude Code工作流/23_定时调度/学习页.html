Scheduler

定时调度

定时调度让 Agent 能处理未来工作：夜间跑测试、每周一写周报、每天固定时间检查状态。系统把“在时间 Y 执行 X”的意图保存成 schedule 记录，后台 check loop 对照时钟匹配 Cron 表达式，到点后把通知放进队列，Agent loop 再把通知注入 messages。

schedule_create→ schedule.json→ check_loop→ notification queue→ agent loop

学完本页，你应该能解释 Cron 五字段、调度记录结构，以及为什么通知回流后 Agent 不需要关心消息来自用户还是时钟。

1. Agent 需要“未来时间”的入口

future work

没有调度时，Agent 只能响应现在的输入。用户必须记得每天晚上输入“跑测试”，或每周一早上输入“写周报”。这类重复任务不应该依赖用户记忆，而应该记录成未来触发规则。

调度系统的解决方案由三部分组成：调度记录、时间检查器、通知队列。调度记录保存“什么时候做什么”；时间检查器持续看当前时间是否匹配；通知队列把触发结果送回主循环。

调度的边界是“未来时间触发输入”，不是“后台自动完成全部工作”。真正执行工具、检查权限、写入结果仍然交给同一个 Agent loop，这样定时任务不会绕过原有安全和上下文机制。

调度记录

持久化保存 Cron、prompt、是否重复等。

时间检查器

后台线程每分钟检查是否到点。

通知队列

把触发提示送回 Agent loop。

2. Cron 表达式：五个字段描述时间规则

cron

Cron 表达式常见五字段：分钟、小时、每月第几天、月份、每周第几天。星号表示任意值，`*/5` 表示每 5 个单位，具体数字表示固定时间。

字段含义范围
minute分钟0-59
hour小时0-23
day of month每月第几天1-31
month月份1-12
day of week每周第几天0-6
Cron 示例Text
*/5 * * * *     # 每 5 分钟
0 9 * * 1      # 每周一 9:00
30 14 * * *    # 每天 14:30

调度器的 `cron_matches` 会把表达式 split 成五段，再把当前时间拆成 minute、hour、day、month、weekday，逐项匹配。

公开实现时要明确时区、星期编号和边界日期。例如有的 Cron 约定周日是 0，有的也允许 7；如果系统跨时区运行，应在记录中保存 timezone，避免用户说“每天 9 点”却按服务器时区触发。

3. 调度记录：把未来意图写下来

schedule record

`schedule_create` 创建一条任务记录。模型必须提供 Cron 和 prompt；`recurring` 默认 true，表示重复任务；`durable` 默认 false，表示不一定持久化到磁盘。持久化任务会写到类似 `schedule.json` 的文件里，重启后可以加载。

调度记录结构JSON
{
  "id": "task_1",
  "cron": "0 9 * * 1",
  "prompt": "Write my weekly report",
  "recurring": true,
  "durable": true,
  "created_at": 1714200000,
  "last_fired": null,
  "jitter_offset": 0
}

`last_fired` 用于防止同一分钟重复触发。`jitter_offset` 用于给整点或半点任务加一点抖动，避免多个实例同时触发。重复任务如果超过一定时间，例如 7 天，可以自动过期清理，防止孤岛任务累积。

Worked example：创建每周报告提醒schedule_create
用户：每周一 9 点提醒我整理周报。

模型调用：
  schedule_create(
    cron="0 9 * * 1",
    prompt="整理上周完成事项、风险和本周计划。",
    recurring=true,
    durable=true,
    timezone="Asia/Shanghai"
  )

调度记录保存后：
  check_loop 每分钟检查一次
  命中后把 prompt 放入 notification queue
  Agent loop 像处理普通用户消息一样处理它

4. check loop 触发通知，Agent loop 统一处理

queue

`CronScheduler.start()` 会加载 durable schedules，并启动后台线程执行 `check_loop`。check loop 每秒醒来一次，但通过 `last_check_minute` 保证每分钟只检查一次任务。若当前时间匹配 Cron，就把通知 put 到队列，并更新 `last_fired`。

创建调度

说明会显示在这里。

1. 保存 schedule record
2. check_loop 对照时钟
3. put notification
4. drain 到 messages
Agent loop 只看到一条 user message，不需要知道它来自手动输入还是定时器。
伪代码：检查与通知回流Python
def check_loop():
    while not stop_event.is_set():
        now = datetime.now()
        if current_minute(now) != last_check_minute:
            last_check_minute = current_minute(now)
            for task in tasks:
                if cron_matches(task["cron"], now) and not fired_this_minute(task, now):
                    queue.put({"role": "user", "content": task["prompt"]})
                    task["last_fired"] = now.timestamp()
        stop_event.wait(1)

如果程序停机错过 durable 任务，重启时可以做 missed run 检查：读取 last_fired 和当前时间，判断中间是否有本该触发的时间点。处理策略要明确，是补发一次、标记 missed，还是按任务类型跳过，避免恢复后突然堆积大量过期任务。

5. 常见误区：调度器不是另一个 Agent

pitfalls

调度器不自己思考，也不自己调用模型。它只是时间触发器：到了时间，把 prompt 作为通知送回同一个 Agent loop。这样用户手动输入和定时触发走同一套消息处理、工具调用和权限系统。

误区

为定时任务再做一个独立 Agent loop，导致状态、权限、工具结果分裂。

正确

定时器只注入消息，所有执行仍由统一 Agent loop 处理。

另一个误区是忘记防重复触发。check loop 每秒醒来，如果没有 `last_check_minute` 和 `last_fired`，同一分钟内可能触发很多次同一个任务。

调度日志至少应记录 task id、cron、计划触发时间、实际触发时间、是否补发、入队结果和后续 Agent 执行状态。没有日志时，用户很难判断任务是没到点、没匹配、没入队，还是执行失败。
