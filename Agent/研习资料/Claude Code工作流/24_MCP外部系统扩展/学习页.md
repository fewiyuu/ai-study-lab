# 1. MCP 解决“工具写死”的扩展问题 {#why}
<!-- tag: extensibility -->

早期 Agent 的工具是写死在本地代码里的，比如 `bash`、`read_file`、`write_file`。如果想增加浏览器控制、数据库查询、业务系统操作，就要继续往源码里加函数。这样本地代码会越来越膨胀，也难复用其他团队已经做好的工具。

MCP 的本质是协议：外部系统按照协议提供工具列表和工具调用能力，Agent 作为 client 连接它们，把这些工具合并到自己的工具池中。模型看到的是统一的工具 schema，而不是每个外部系统的私有实现。

可以把 MCP 的边界记成：它标准化“发现工具、描述工具、调用工具”的接口，但不替 Agent 判断这个工具该不该用，也不替本地系统完成权限审批、审计和错误恢复。

## box warn
### 写死工具
新增能力要改源码、重部署，工具越来越臃肿。

## box
### MCP 接入
外部 server 提供工具，Agent 动态发现、注册和路由调用。

# 2. 三个核心组件：client、loader、router {#components}
<!-- tag: components -->

字幕里提到三个核心组件。MCP client 负责和 server 通信，例如 initialize、tools/list、tools/call。Plugin loader 负责扫描插件清单，找到有哪些 MCP server。MCP tool router 负责根据工具名把调用分发到正确 server。

组件职责直觉

MCP client连接 server，发送/接收 JSON-RPC 请求外部工具的客户端
Plugin loader扫描 `.cloud-plugin/plugin.json`MCP server 注册中心
MCP tool router注册 client，按工具名调用 server统一路由分发器

插件清单示例JSON

```text
{
 "name": "plugin1",
 "mcpServers": {
 "server1": {
 "command": "python",
 "args": ["mcp_simple_server.py"],
 "transport": "stdio"
 }
 }
}
```

这里的 JSON-RPC 是一种请求/响应协议格式，常见字段包括 `jsonrpc`、`id`、`method` 和 `params`。stdio transport 表示 client 通过子进程的标准输入输出通信；HTTP/SSE 则更像连接一个远程服务。传输方式不同，但工具描述和调用语义应保持一致。

# 3. 连接、列工具、合并工具池、路由调用 {#flow}
<!-- tag: flow -->

启动时，Plugin loader 扫描插件目录并读取 `plugin.json`，得到 MCP server 配置。系统为每个 server 创建 MCP client，通过 stdio 管道启动子进程并发送 initialize 请求。连接成功后，client 调用 `tools/list` 拿到工具 schema，再把这些工具加上命名前缀后合并到工具池。

工具池里同时有 native tools 和 MCP tools；模型只看到统一 schema。

JSON-RPC 交互示例JSON

```text
// initialize
{"jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {"protocolVersion": "..."}}

// tools/list
{"jsonrpc": "2.0", "id": 2, "method": "tools/list", "params": {}}

// tools/call
{"jsonrpc": "2.0", "id": 3, "method": "tools/call", "params": {"name": "list_files", "arguments": {"path": "."}}}
```

Worked example：外部数据库工具接入Flow

```text
插件清单：
 plugin = "analytics"
 server = "warehouse"
 tool = "query_sales"

启动：
 loader 读取 plugin.json
 client initialize warehouse server
 client tools/list 得到 query_sales schema
 tool pool 注册 mcp__analytics__warehouse__query_sales

调用：
 模型选择 mcp__analytics__warehouse__query_sales
 permission gate 检查是否允许查询数据库
 router 去掉前缀，向 warehouse server 发送 tools/call
 server 返回结果，Agent 作为 tool_result 回填给模型
```

# 4. 命名空间、native 优先与权限不可绕过 {#security}
<!-- tag: safety -->

MCP 工具名会加前缀，例如 `mcp__plugin1__server1__list_files`。前缀相当于命名空间，能避免外部工具和本地工具重名，也能让系统识别这个工具来自 MCP。构建工具池时先加入 native tools，再加入 MCP tools，并且外部工具不能覆盖核心 native 能力。

更重要的是权限控制不可绕过。MCP 工具虽然来自外部 server，但执行前仍要通过 permission gate。只读工具如 `list_files` 可以自动允许；高风险工具需要 ask 或 deny。否则外部 server 提供一个危险删除工具，就等于给系统开后门。

## box
### 安全接入
MCP 工具统一命名、统一 schema、统一权限检查。

## box warn
### 危险接入
外部工具绕过权限系统，可能执行删除、写入、网络或系统命令。

错误示例：MCP 工具绕过权限Unsafe

```text
if is_mcp_tool(tool_name):
 return mcp_router.call(tool_name, tool_input)

# 问题：外部工具没有经过 permission.check。
# 正确：native 和 MCP tool 都先走权限系统。
```

路由实现还要处理重名和降级。native 工具通常优先，因为它们是 Harness 的核心能力；MCP 工具加命名空间后再注册。如果某个 server 断开，工具池应标记这些工具不可用，而不是让模型继续看到一个会失败的工具。

# 5. 常见误区：MCP 不是插件魔法 {#pitfall}
<!-- tag: pitfalls -->

MCP 提供统一协议，但并不自动解决所有问题。你仍然要处理连接失败、server 进程退出、工具 schema 冲突、权限控制、日志审计和传输方式。MCP 支持 stdio，也支持 server-sent events 等传输机制；配置 server 时要清楚它如何通信。

只堆名词知道 MCP 名字，但不理解 initialize/list/call。
忽略权限MCP 工具直接执行，外部能力变成后门。
忽略命名外部工具覆盖 native 工具，核心能力被污染。

一个稳健的 MCP 接入应该做到：插件发现清晰、连接状态可见、工具命名不冲突、权限可控、失败可恢复、调用可审计。

MCP 调用日志至少应包含 server 名称、工具名、参数摘要、权限决策、请求 id、耗时、返回状态和错误信息。这样当外部系统慢、schema 变更或权限策略过宽时，才能定位问题发生在 Agent、本地 router 还是远端 server。
