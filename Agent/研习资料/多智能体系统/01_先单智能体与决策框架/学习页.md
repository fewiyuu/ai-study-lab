# 0. 学习目标 {#goal}
<!-- tag: goal -->

这页只训练一件事：先把单智能体基线跑清楚，再决定是否值得拆成多个上下文。你需要先能说出基线要记录什么，再能把多智能体需求分到三类约束里，最后能把“我们要多智能体”改写成可验证实验。

先说基线
能解释为什么单智能体要先跑，不要先拆。


再说约束
能把需求分到上下文保护、并行化、专业化三类。


最后说验证
能把拆分变成基线、指标、对照和回退条件。


---

# 1. 先抓主线 {#overview}
<!-- tag: overview -->

这门课的顺序很固定：先看单智能体基线，再看是否存在真实约束，然后才讨论拆分方式和协调成本。只要这个顺序反过来，判断就很容易变成“看起来更高级”。

单智能体基线→
真实约束→
协调成本→
小范围实验→
是否拆分

## box warn
### 一句话标准
只有当独立上下文真的解决了一个可观察瓶颈，并且指标证明收益大于协调成本时，多智能体才是工程方案。

你现在应该能做什么：听到“我们想要多智能体”时，先追问基线、约束和指标，而不是先开始命名角色。


---

# 1. 先把名词对齐 {#concepts}
<!-- tag: terms -->

多智能体系统不是“很多聊天机器人一起说话”。更准确地说，它是把任务分配到多个独立上下文里的编排方式。每个子智能体有自己的提示词、工具权限、输入材料和输出格式，主智能体负责分派、接收和合成。

这一定义一旦定下来，后面的判断就能落到两边：价值来自隔离、扩展、专业化和验证；代价来自复制、路由、交接和合成。任何拆分方案都要同时回答这两边。

概念直觉工程含义常见误解


单智能体基线先让一个 Agent 做到可比较的水平。一个主上下文里完成任务、调用工具、输出结果。把单智能体理解成低级版本。
多智能体系统多个 Agent 协作完成任务。多个独立上下文之间分派、执行、汇报、合成。以为 Agent 越多越强。
上下文污染无关材料拖累当前推理。上下文累积了后续任务不需要的信息。只把它理解成上下文太长。
并行化同时探索多个独立方向。用更多计算换更大的覆盖度，再由主智能体合成。以为并行一定更便宜。
专业化缩小工具和任务空间。按工具集、提示词或领域知识划分专用 Agent。以为取几个角色名就是专业化。

你现在应该能做什么：看到一张多智能体架构图时，先把每个框翻译成“它独立持有哪些上下文、工具和输出责任”。


---

# 2. 先跑单智能体基线 {#baseline}
<!-- tag: baseline -->

单智能体不是试试看。一个设计得当的单智能体，配上合适的工具、清楚的系统提示词和稳定的上下文管理，能完成的事情通常比直觉里多。先做单智能体，等于先建立可比较的基线；没有基线，就很难判断后面的拆分到底改善了什么。

多智能体会增加新的失败点。多一个智能体，就多一份提示词要维护，多一套工具权限要检查，多一次交接可能丢信息。很多团队把规划、执行、测试、评审拆给不同智能体，结果协调花掉的 token 比真正做事更多。

## box soft
### 单智能体基线要先回答
- 它是否已经有完成任务所需的工具？
- 系统提示词是否说明了目标、约束和停止条件？
- 上下文是否被无关材料污染？
- 失败是能力不足，还是任务说明不清？

## box warn
### 判断句
如果单智能体还没有稳定基线，就先不要拆。拆分只会把原来的问题分散到多个智能体里，排查会更难。

正确示例：先跑基线pseudo

```python
task = "研究某个市场趋势，并给出带来源的结论"

single_agent_result = run_agent(
 task=task,
 tools=["web_search", "read_document", "summarize"],
 budget_tokens=baseline_budget
)

measure(single_agent_result, metrics=[
 "答案覆盖度",
 "引用准确性",
 "总 token",
 "耗时",
 "失败原因"
])
```

这段伪代码的重点顺序。先让一个智能体在明确工具和预算下完成任务，再记录质量、成本和失败原因。只有当失败原因指向真实瓶颈，才进入拆分讨论。

# 3. 成本从哪里来 {#costs}
<!-- tag: cost -->

一个常见工程观察是：在等价任务上，多智能体实现可能用掉单智能体数倍 token。这个数字不是数学定律，但足以提醒我们，多智能体不是免费并行。

成本主要来自三处。第一是上下文复制。每个子智能体都需要自己的任务说明、工具说明和必要背景。第二是协调消息。主智能体要分派任务，子智能体要汇报结果，必要时还要追问。第三是交接摘要。一个智能体做完后，另一个智能体无法直接继承完整思考过程，只能拿到被压缩过的结果。

成本来源具体表现坏结果缓解方式


上下文复制每个子智能体都加载同一批背景。token 增长快，模型注意力分散。只给子智能体完成任务所需的最小背景。
协调消息分派、汇报、追问、合并都要占上下文。协调比执行更贵。拆分前确认子任务真的独立。
交接摘要后续智能体只看到压缩后的结果。动机、边界条件、失败尝试丢失。能不交接就不交接，相关工作放同一智能体里。
提示词维护多个角色、多个工具权限、多个输出格式。改一个流程要同步多处提示词。保持角色少，接口清楚，测试覆盖关键路径。

对照实验：基线与候选pseudo

```python
hypothesis = "订单查询污染了技术诊断上下文"

baseline = run_single_agent(task)
candidate = run_agent_with_order_lookup_subagent(task)

compare(baseline, candidate, metrics=[
 "诊断准确率",
 "无关订单 token 数",
 "总 token",
 "平均耗时",
 "失败案例"
])
```

好的拆分会对应一个可验证假设。你因为你怀疑某个约束正在拖累质量，并准备拿指标检查这个假设。

# 4. 识别三类约束 {#signals}
<!-- tag: signals -->

多智能体有价值，因为它解决了单智能体越不过去的约束。先把约束说清，后面才有拆分理由。这里的三类约束分别是上下文保护、并行化和专业化。

上下文保护
当一个子任务会产生大量无关材料，继续塞进同一个上下文会污染后续推理时，可以用子智能体隔离它。


并行化
当任务天然能拆成多个独立搜索方向时，多个智能体可以同时探索更大的信息空间。


专业化
当不同子任务需要完全不同的工具、提示词或领域知识时，专用智能体可能比一个通用智能体更稳。

你现在应该能做什么：先把需求归到三类约束之一，再讨论是否拆分，而不是先画出多个角色框。

# 5. 做决策 {#decision}
<!-- tag: decision -->

真正的判断顺序很固定：先有单智能体基线，再看真实约束，再估算协调成本，最后决定是否只做一个小范围实验。这里没有神秘门槛，只有证据是否足够。

信号优先动作为什么下一步


基线还不稳定继续打磨单智能体。没有可比较对象，拆分结论不成立。先修提示词、工具和评测。
高噪声材料拖累后续推理先做上下文隔离。子上下文能把污染关在局部。设计最小摘要接口。
多个独立方向可同时探索尝试并行化。覆盖度可能上升。定义合成格式和冲突处理规则。
工具、提示词、领域知识明显分离考虑专业化。专用上下文通常更稳。写清路由条件和返回格式。

正确示例：把拆分写成可验证假设pseudo

```python
plan = {
 "hypothesis": "订单查询污染了技术诊断上下文",
 "baseline": "单智能体直接查单并诊断",
 "candidate": "订单查询交给子智能体，主智能体只接收诊断字段",
 "metrics": ["诊断准确率", "无关 token 数", "总 token", "耗时", "失败案例"],
 "rollback": "如果摘要字段不稳定或成本上升，就回到单智能体"
}
```

这张表的核心把判断转成实验。实验能赢，才拆；实验赢不了，就先别拆。

# 6. 例子 {#examples}
<!-- tag: example -->
<!-- layout: grid2 -->

## box soft
### 例题 A：客服诊断
一个客服 Agent 要查订单历史，还要诊断技术故障。订单查询会返回几千 token，但后续诊断并不需要完整订单细节。
**判断：**这适合上下文保护。查订单可以交给子智能体，只返回和诊断相关的摘要，避免把长订单历史塞进主上下文。

## box warn
### 例题 B：实现同一个小功能
团队想让一个智能体规划、一个智能体编码、一个智能体写测试、一个智能体评审。功能很小，所有阶段共享同一批设计背景。
**判断：**先不要拆。这里更像顺序阶段共享上下文，拆分会增加交接损耗。让同一个智能体完成实现和测试，再用明确检查清单验收更合适。

案例假设：把拆分做成实验pseudo

```python
task = "研究某个市场趋势，并给出带来源的结论"

single_agent = run_agent(task)
multi_agent = run_agent_with_research_subagents(task)

compare(single_agent, multi_agent, metrics=[
 "结论覆盖度",
 "引用准确性",
 "总 token",
 "耗时",
 "失败原因"
])
```

这里的实验重点是比较，而不是“看起来像团队”。一旦你能把候选方案和基线放在同一张表里，多智能体就从口号变成工程选择了。

# 7. 常见误区 {#mistakes}
<!-- tag: debug -->

误区错因后果修法


把多智能体当默认架构把角色数量误认为能力提升。成本增加，问题更难定位。先跑单智能体基线，再看瓶颈。
按人类岗位拆分忽略智能体交接会丢上下文。规划、实现、测试之间反复解释背景。按上下文边界拆，不按职位名拆。
工具多就拆智能体没有先尝试按需工具发现。多出路由和提示词维护成本。先精简工具、分组工具或引入工具搜索。
只看并行，不看合成以为同时跑就一定更快。结果合并变慢，冲突结论难处理。提前定义合成格式和冲突处理规则。
没有成本指标只凭主观感觉评价效果。无法证明拆分是否值得。记录 token、耗时、错误率、覆盖度和维护成本。

错误示例：为了像团队而拆分anti-pattern

```python
agents = [
 PlannerAgent(),
 CoderAgent(),
 TesterAgent(),
 ReviewerAgent()
]

# 所有智能体都在处理同一个功能，但每一步都要重新解释背景。
# 测试者不知道实现时做过哪些权衡，评审者看不到探索过程。
run_handoff_chain(agents, task="实现同一个功能")
```

这段错误示例的问题拆分边界和上下文边界不一致。它把强相关的工作切开，每次交接都会压缩信息，最后容易形成传话游戏。

# 9. 把拆分落成一条交接链 {#bridge}
<!-- tag: bridge -->

最容易把多智能体做虚的地方，是只画角色，不写交接。真正有用的拆分，必须能从主智能体、子智能体到合成结果，沿着一条清楚的链路走完。每一步都要说明输入是什么、输出保留什么、哪些材料不该继续传下去。

还是用客服诊断这个例子。主智能体接到问题后，不必把完整订单历史和聊天记录都塞进一个上下文里。更稳的做法，是把“问题描述、时间范围、订单号、要看的字段”交给查单子智能体，把“异常证据、涉及哪些接口、建议回退动作”交给诊断子智能体，最后只把主任务需要的字段带回来。这样做的目的把信息边界说得更清楚。

阶段输入输出必须保留容易丢失


主智能体接单用户问题、任务目标、时间约束。拆分判断、路由条件、首轮假设。决策依据、成功标准。为什么先查这一块。
查单子智能体订单号、时间窗、必要字段。异常字段、状态变化、可复核证据。能支持诊断的最小事实集。整份原始日志和无关订单。
诊断子智能体异常字段、接口信息、失败样本。根因判断、风险等级、建议动作。结论和依据的对应关系。推理过程里那些不影响判断的旁枝。
主智能体合成来自各子智能体的摘要。可交付答案、下一步、回退条件。主任务的最终责任。把子任务细节再讲一遍。

正确示例：只传交接字段pseudo

```python
handoff = {
 "problem": "接口突然变慢",
 "scope": ["最近 24 小时", "同一订单类型", "同一入口"],
 "keep": ["异常时间点", "相关接口", "失败样本", "可复核证据"],
 "drop": ["完整聊天记录", "整份日志转储", "无关订单明细"],
}

lookup = order_subagent.run(handoff)
diagnosis = diagnosis_subagent.run({
 "evidence": lookup["evidence"],
 "root_cause": lookup["root_cause"],
 "rollback_hint": lookup["rollback_hint"],
})

final_answer = main_agent.combine({
 "summary": diagnosis["summary"],
 "risk": diagnosis["risk"],
 "next_step": diagnosis["next_step"],
})
```

这段流程的关键，是每个智能体都只拿自己需要的材料。只要交接字段够短，主智能体就能快速判断下一步；只要字段开始膨胀，说明拆分边界已经不干净了，继续拆反而更贵。

## box warn
### 回退条件
- 子智能体的输出仍然需要主智能体再读一遍全部原始材料。
- 两个子智能体返回了相同信息，只是换了说法。
- 主智能体无法用一眼看懂的字段判断该不该继续。
- 交接摘要开始超过主任务真正需要的长度。

你现在应该能做什么：拿一条真实任务写出一份交接协议，明确主智能体保留什么、子智能体只看什么、最后必须返回什么。只要这三件事说不清，先别拆分。

# 10. 小范围试点怎么判定 {#pilot}
<!-- tag: pilot -->

多智能体不该一上来就全量替换。更稳的做法，是拿一条真实任务做小范围试点：同样的输入、同样的预算、同样的验收标准，只把拆分结构换掉。这样才能看出结构变化到底带来了什么。

试点里最重要的记录要完整。单智能体和多智能体必须放到同一张记录表里，看质量、token、耗时、交接次数、人工追问次数和回退条件。只要其中任何一项明显变差，就不能只盯着“角色更多、过程更像协作”这一点下结论。

观察项单智能体记录多智能体记录怎么读危险信号


质量正确率、覆盖度、可用性。同一套指标。先看是否真的更好，再看好多少。答案看起来更长，但判断没有更稳。
成本总 token、工具调用、耗时。同一任务下的总消耗。成本要和质量一起看，不要单独报喜。结果没有明显变好，成本却先翻上去。
交接无交接或一次完成。分派、汇报、合成、追问。交接越多，越要看字段有没有收紧。每个子智能体都在重复背景。
回退能否直接修正。失败后能否快速退回单体。回退规则要在试点前写清楚。失败后还要重新设计整个路由。
维护一套提示词和工具逻辑。多套角色、接口和边界。改动是否会牵连更多位置。为了一个小修复，得同步改很多角色。

试点协议pseudo

```python
trial = {
 "task": "同一条真实任务",
 "baseline_budget": 4000,
 "candidate_budget": 4000,
 "shared_metrics": [
 "quality",
 "token",
 "time",
 "handoff_count",
 "manual_followup",
 ],
 "stop_rules": [
 "quality did not improve",
 "handoff_count stayed high",
 "summary_fields kept expanding",
 "rollback became hard",
 ],
}
```

这份协议的作用，是把“感觉更强”变成可以复核的判断。试点前先写指标和停手条件，试点后再看结果，顺序不能反过来。只要试点结束时还说不清哪项指标在改善、哪项成本在上升，这个候选方案就还不能算成熟。

## box info
### 停手信号
- 单智能体已经能稳定完成任务，多智能体只是在复述同一份背景。
- 子智能体输出不断膨胀，主智能体还要回去读原始材料。
- 交接次数增加后，耗时和人工追问一起变多。
- 失败后没有简单回退路径，只能继续补角色。

你现在应该能做什么：把任意一个真实任务写成小范围试点，明确基线、候选、指标和停手条件。只要这四件事都能落到纸面上，你就有资格谈要不要拆分。

如果连停手条件都写不出来，就先回到单智能体重跑一遍同一任务，把基线重新量清楚。

**基线**先知道单智能体能做到什么。
**约束**拆分要解决真实瓶颈。
**成本**上下文、协调、交接都会变贵。
**证据**用指标比较，不靠架构图。

### 1 choice

**Q:** 1. 大多数团队应该先从什么开始？

- [ ] . 单智能体基线
- [ ] . 四个角色同时拆分
- [ ] . 先做验证子智能体

Answer: . 单智能体基线

### 2 choice

**Q:** 2. 下列哪一项不是多智能体常见成本？

- [ ] . 上下文复制
- [ ] . 协调消息
- [ ] . 交接摘要
- [ ] . 单个智能体完全不需要工具

Answer: . 单个智能体完全不需要工具

### 3 fill

**Q:** 3. 本页提到的三类约束是：上下文保护、______、专业化。

Answer: 并行化

### 4 short-answer

**Q:** 4. 为什么没有单智能体基线时，不适合直接上多智能体？

Explain: 要点至少包含两件事：一是没有基线就无法比较多智能体是否真的提高了质量、覆盖度或稳定性；二是拆分后失败点变多，如果单智能体本来就没跑稳，很难判断问题是出在任务本身、工具、提示词还是拆分边界。

<!-- end-practice-group -->

<!-- practice-group: 二、边界区分 -->
<!-- quick-review: 复习一下 按上下文边界拆，通常比按人类岗位拆更稳。 需要频繁同步理解的工作，不适合拆开。 -->

### 5 true/false

**Q:** 5. 同一个小功能的规划、实现、测试天然适合交给三个智能体。

- [ ] 对
- [ ] 错

Answer: 错

### 6 short-answer

**Q:** 6. 当一个智能体有 20 个工具时，为什么不一定马上拆成多个智能体？

Explain: 重点工具是否真的构成不同上下文边界。如果工具可以分组、按需选择或通过工具搜索减少干扰，先拆智能体往往会引入更多路由和维护成本。

### 7 choice

**Q:** 7. 下列哪种任务更适合并行子智能体？

- [ ] . 三个彼此独立市场的趋势研究
- [ ] . 需要同一条状态不断来回同步的实现任务
- [ ] . 一个短功能的单次修复

Answer: . 三个彼此独立市场的趋势研究

### 8 explain

**Q:** 8. 用 1-3 句话解释“上下文污染”和“上下文太长”的区别。

Explain: 上下文太长强调材料总量接近或超过窗口；上下文污染强调即使没有超窗，也有太多当前推理不需要的材料，导致后续判断被干扰。两者经常一起出现，但处理方式不一样。

<!-- end-practice-group -->

<!-- practice-group: 三、错误诊断 -->
<!-- quick-review: 复习一下 常见错误是把噪声又带回来，或者把强相关工作切断。 修法通常是定义字段、合并上下文或稳定接口。 -->

### 9 diagnosis

**Q:** 9. 一个系统把 Planner、Coder、Tester、Reviewer 拆开处理同一小功能，结果 token 暴涨。错因、后果、修法分别是什么？

Explain: 错因通常是按人类岗位拆，而不是按上下文边界拆；后果是每次交接都要压缩背景，协调成本和 token 成本一起上升；修法是把强相关阶段留在同一智能体里，或者只拆出真正独立的子任务。

### 10 diagnosis

**Q:** 10. 团队说“多智能体感觉更稳”，但没有记录质量、token、耗时和失败案例。这个判断哪里不稳？

Explain: “感觉更稳”缺少基线和对照，没法知道改善来自结构还是偶然结果。没有指标，也就无法区分质量提升、成本恶化、偶发成功或验证覆盖不足。

### 11 true/false

**Q:** 11. 工具列表很长时，直接按工具类别拆智能体通常没有额外成本。

- [ ] 对
- [ ] 错

Answer: 错

### 12 short-answer

**Q:** 12. 为什么“并行运行多个智能体”不等于“总是更快更便宜”？

Explain: 并行只是在执行层面同时启动多个方向，后面还要合成、去重、处理冲突和验证结果。如果子任务并不独立，或者合成成本过高，整体反而可能更慢更贵。

<!-- practice-group: 四、场景迁移 -->
<!-- quick-review: 复习一下 好的答案会先找约束，再看成本。 只写“用多智能体更专业”不够。 -->

### 13 scenario

**Q:** 13. 客服 Agent 每次查订单会拉回几千 token，但技术诊断只需要订单状态摘要。你会怎么设计？

Explain: 合理答案通常会把订单查询放到子智能体，要求它只返回诊断所需字段，如订单是否有效、产品型号、错误码、最近激活时间和缺失信息，然后让主智能体继续技术诊断。

### 14 scenario

**Q:** 14. 研究“亚洲、欧洲、北美三个市场趋势”，是否适合多智能体？请说明。

Explain: 如果三个市场的资料和判断大体独立，就适合用并行子智能体分别研究，再由主智能体合成。回答里最好说明独立性、合成方式和成本控制点，而不是只说“可以并行”。

### 15 choice

**Q:** 15. 一个小型 bug 修复任务，涉及一个文件、一个测试。你会先单体还是多体？

- [ ] 单体
- [ ] 多体

Answer: 单体

### 16 short-answer

**Q:** 16. 什么情况下“专业化智能体”比一个通用智能体更合理？

Explain: 当不同子任务需要完全不同的工具集、提示词约束或领域知识，而且这些边界稳定清晰时，专业化更合理。重点是“边界清楚”，不是“名字更像岗位”。

### 17 interview

**Q:** 17. 面试官问：“为什么不一开始就做多智能体？”请用 2-4 句话回答。

Explain: 回答要点：先有单智能体基线，才能比较收益；多智能体会引入额外协调成本和失败点；只有当上下文保护、并行化或专业化真的带来可观察收益时，才值得拆分。

### 18 checklist

**Q:** 18. 如果你要验证多智能体是否值得，至少记录哪 5 个指标或证据？

Explain: 可写的证据包括：质量、覆盖度、总 token、耗时、失败原因、错误率、交接次数、回退条件、人工介入次数、维护成本。回答不必一模一样，但必须能比较基线和候选。

### 19 rewrite

**Q:** 19. 把“我们需要多个 Agent，因为这样更智能”改写成可验证的工程假设。

Explain: 一个合格版本应该包含：为什么要拆、拆哪一段、基线是什么、用什么指标比较、什么情况下回退。例子可以写成“把订单查询交给子智能体后，主上下文中的无关 token 明显下降，同时诊断准确率不下降；如果摘要字段不稳定或总成本上升，就回退到单智能体”。

### 20 synthesis

**Q:** 20. 用 6-9 句话讲清：什么时候应该从单智能体升级到多智能体。

Explain: 好答案会先说基线，再说约束，再说成本，再说实验。它应该能讲清：先跑单智能体，确认问题不是提示词和工具设计；如果是上下文污染、并行搜索空间或专业化边界，才做小范围拆分；拆分后要看指标是否真的改善，并保留回退条件。
