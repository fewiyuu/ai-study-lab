## Source-To-Unit Notes

- `Context protection` 支撑本单元的主线：把高噪声子任务隔离到子上下文里，只让主上下文接收后续推理需要的最小字段。
- `Context-centric decomposition` 支撑本单元的另一条主线：按信息边界拆分，不按“规划 / 实现 / 测试 / 评审”这类岗位名拆分。
- `A decision framework` 负责承接上一单元的判断框架，本单元只在已经存在明显上下文压力时继续往下拆。
- `Outgrowing single-agent architectures` 提供边界：一旦拆分会增加协调成本、丢失关键字段或把噪声带回主上下文，就不应继续拆。
- 《资料熔炼》补足课程术语、拆块顺序和例子族，尤其是订单查询、技术诊断和返回合同这条练习线。
- 现有 02 草稿可复用客服订单的锚定例子、分辨“长但相关”和“长且无关”的练习节奏，但最终页面要迁移到共享 shell，并把练习做成更强的诊断结构。

## Operation Facts

- 本单元讲的是工程判断，不是固定公式或硬阈值。
- 最适合反复使用的锚定例子，是“客服 / 订单查询 / 技术诊断”这一类高噪声任务：订单历史很长，但主智能体只需要少数字段。
- 页面需要显式说明子智能体的返回合同：哪些字段必须返回，哪些材料应留在子上下文里，哪些不确定项要直接暴露。
- 页面还需要说明为什么“规划、实现、测试、评审”这种阶段式拆分经常变成传话游戏。
- 本页不新增外部资料，不爬网，不补其他实现框架。
