上下文保护与按上下文拆分

本页只处理一个判断：当某个子任务会把大量无关材料带回主上下文时，怎样把它隔离到子智能体里，同时保住主智能体继续判断所需的字段。锚定例子一直是“客服查订单 + 技术诊断”，因为它同时有长材料、关键字段和明显的误判后果。

0. 学习目标

goal

这一页只处理一个工程判断：当高噪声材料会拖累后续推理时，怎样把它关进子上下文里，同时让主智能体只接收继续诊断所需的字段。阅读时先看状态变化，再看返回合同，最后用练习区确认你会不会把“长”和“污染”分开。

再写返回合同

能列出子智能体必须返回的字段，并把无关材料挡在外面。

1. 先抓主线

overview

上下文窗口说的是模型一次能看到多少材料；上下文压力说的是材料虽然还没超窗，但已经开始挤压注意力；上下文污染说的是材料里混进了当前判断不需要的东西。三者常常一起出现，但处理方式不同。

用户问题→

识别噪声→

子智能体查单→

最小摘要→

主智能体继续诊断

上下文窗口能装多少。总 token 和模型上限。超窗后直接丢信息。

上下文压力还没超窗，但开始挤。材料是否已经让注意力分散。关键约束被淹没。

上下文污染混进了不该留的东西。当前推理是否被无关材料干扰。主智能体开始追错线索。

2. 上下文状态快照

ledger

把上下文当成一种会被读取、写入和污染的状态，会比把它当聊天记录更容易判断。你要看的是：谁在读、谁在写、写进去的东西后面还会不会被继续用。

主智能体接单用户问题、时间范围、订单号。拆分判断和任务边界。问题范围混乱，后面很难收束。先写出必须保留的字段。

子智能体查单完整订单历史和工具返回。最小摘要、缺失项、证据来源。摘要仍然带着噪声。删掉与诊断无关的材料。

主智能体诊断摘要、错误码、必要上下文。结论和追问。开始引用物流、地址等无关信息。回看字段清单是否过肥。

回退决策质量、token、耗时、交接次数。继续拆或回退单体。没有可比证据，只剩感觉。先回到单智能体基线。

3. 客服订单：噪声和信号怎么分

example

客户说“专业版无法激活”时，订单查询通常会返回购买记录、地址、发票、物流、历史沟通和激活信息。真正要做技术诊断时，主智能体通常只需要订单是否有效、产品型号、最近激活时间、错误码和缺失信息。

代码
order_lookup_result = {
 "order_status": "active",
 "product": "Pro",
 "last_activation": "2026-06-20 14:22",
 "error_code": "E42",
 "address": "...",
 "invoice": "...",
 "shipping_history": "...",
 "chat_history": "..."
}

这段材料本身没有错，错的是后续上下文是否把不相关部分也继续背着走。上下文保护做的事，就是把它们分开。

失败后果

如果订单地址、发票和客服寒暄继续留在主上下文里，模型会更容易把物流问题当成激活问题；如果摘要只剩一句“订单正常”，主智能体又会失去继续追问的线索。

4. 手工追踪：一条订单如何变成摘要

trace

这一段把同一条订单从原始返回走到主智能体可用摘要。先看原始输入，再看允许保留的字段，再看丢弃项。只要其中一步多留一个字段，主上下文就会重新背上噪声；少留一个字段，后面就会卡诊断。

原始查单整份订单历史和工具返回。状态、型号、激活时间、错误码、缺失项。地址、发票、物流、寒暄。主智能体开始跟着无关细节走。

摘要构建白名单字段和证据来源。能支持下一步判断的最小字段。任何不在白名单里的材料。摘要变肥，主上下文又要回读原文。

主智能体诊断摘要、错误码、时间线。根因分支和下一步追问。重复出现的原始订单文本。诊断绕圈，不能直接进入判断。

回退判断质量、token、交接次数。是否真的减噪。没有比较价值的装饰性材料。试点说不清成败，只剩“看起来更完整”。

代码
ALLOWED_FIELDS = [
 "order_status",
 "product",
 "last_activation",
 "error_code",
 "missing",
 "evidence",
]

def summarize_order(result):
 summary = {key: result[key] for key in ALLOWED_FIELDS if key in result}
 summary["dropped"] = [k for k in result.keys() if k not in ALLOWED_FIELDS]
 return summary

这段代码的作用是做字段白名单。它只留下白名单字段，其他材料都被丢弃。白名单写得越清楚，主智能体越不容易把地址、发票和客服寒暄带进技术诊断。白名单写得越模糊，子智能体就越像一个小型垃圾回收站，最后还要主智能体自己清理一遍。

操作提示

看这段追踪时，先把每一列当成一次状态变换：读什么、保留什么、丢掉什么、错了以后会出现什么症状。你一旦能把症状说准，就能更快判断是字段太肥，还是字段太薄。

你现在应该能做什么：给任意一个高噪声工具结果写一份白名单，然后用“保留什么、丢弃什么、错了会怎样”三列把它检查一遍。

5. 返回合同：筛字段，保留最小摘要

contract

子智能体的返回接口应当只保留主智能体继续推理需要的结构化字段。如果摘要写成散文，主智能体通常还是得回去读原始材料，这就不算减噪。

子智能体返回合同pseudo

字段为什么保留为什么不保留更多

判断句

如果你不能清楚写出“主智能体后续要用什么字段”，就不要让子智能体先写长摘要。先写接口，再写内容。

6. 按上下文拆，不按岗位拆

decomposition

问题导向拆分会说“一个智能体规划，一个智能体实现，一个智能体测试，一个智能体评审”。这种拆法看起来像团队，实际上经常把同一批背景压成几段摘要，最后变成传话游戏。上下文导向拆分问的是：哪些工作可以在不共享大量背景的情况下独立完成？

边界类型是否适合拆原因检查问题

你现在应该能做什么：先把需求归到“独立”“可隔离”或“强相关”之一，再决定是否拆分；先画角色框只会把边界问题藏起来。

7. 先知道，先别钻

boundary

这一页先不追并行化、专业化、验证子智能体、路由策略和更复杂的协调机制。它们会在第 03 和第 04 单元回来，但现在先把“怎样把噪声留在子上下文里”这一件事做扎实。

并行化只把它当成一种可能的后续选择。第 03 单元。

专业化只记住它依赖清晰的工具和领域边界。第 03 单元。

验证子智能体只先记住验证者要看具体证据。第 04 单元。

路由只把它看成后续编排问题，不在这里展开。第 03 单元和第 04 单元。

回退条件先记住“字段开始膨胀就退回单体”。第 04 单元。

8. 操作桥：把这页变成你自己的判断清单

bridge

真正落地时，先做三步。第一步，把你正在处理的任务写成一句话，找出会制造噪声的那一段。第二步，列出子智能体必须返回的字段白名单，再把无关材料写进丢弃清单。第三步，把这两个清单和单智能体基线放到同一张表里比较。

你要写什么写成什么样合格信号不合格信号

你下一步该做什么

  • 拿一条真实任务，写出噪声源和字段白名单。
  • 把白名单和丢弃清单放进同一张检查表。
  • 和单智能体基线比较质量、token、耗时和交接次数。

如果你已经能把这三步写清楚，说明上下文保护已经从概念变成动作了。下一步再去看第 03 单元，你会更容易判断哪些任务值得并行，哪些任务应该继续保持单上下文。

9. 摘要实验室：少一个字段会怎样

lab

这一段专门演示一个常见失败：摘要看起来很整齐，但漏掉了一个关键字段。场景还是订单诊断，主智能体想判断是否继续排障，子智能体却把“地址、发票、物流”都保留下来，却漏掉了错误码。表面上摘要更长了，实际上可用信息更少了。

字段保留理由缺失后的后果是否进入主上下文

摘要守门代码pseudo

失败后果

如果这个守门器放行了错误码缺失的摘要，主智能体会卡在追问上；如果它放行了地址和发票，主智能体会继续背着噪声诊断。两种情况都说明摘要没有守住边界。

这段实验最值得记住的地方，检查顺序。先看主智能体需要什么字段，再看摘要里留下了什么，再看有没有把无关字段带回去。只要这三步能跑通，你就能直接用它检查练习 7 的答案。

你现在应该能做什么：拿一个你自己的高噪声任务，写一张“保留 / 丢弃 / 缺失后果”表，再用一段守门代码检查摘要是否合格。

10. 三步自检：白名单、黑名单、回退线

check

真正把上下文保护做稳，最后要落到一套自检顺序：先看白名单里有没有主智能体真正要用的字段，再看黑名单里有没有不该回来的噪声，最后看回退线是否清楚。这个顺序比“摘要写得短不短”更重要，因为短摘要也可能缺字段，长摘要也可能已经变成噪声。

检查点合格时长什么样不合格时长什么样你要怎么改

摘要守门器pseudo

失败后果

白名单守不住，主智能体会失去错误码和时间线；黑名单守不住，地址和发票会重新占位；回退线守不住，候选方案失败后会继续堆角色，最后很难再回到可比较的基线。

这一步是前面所有内容的收束：如果你能凭白名单、黑名单和回退线判断摘要是否可用，说明你已经能实际检查一个子智能体输出，而不只是会描述上下文保护的概念。

你现在应该能做什么：拿练习 7 的字段清单和练习 13 的试点计划，直接用这一张三步自检表复核一遍，看看你的摘要是不是已经能交给主智能体。

11. 小范围试点：把判断变成实验

pilot

多智能体不该一上来就全量替换。更稳的做法，是拿一条真实任务做小范围试点：同样的输入、同样的预算、同样的验收标准，只把拆分结构换掉。这样才看得出结构变化到底带来了什么。

代码
trial = {
 "task": "同一条真实任务",
 "baseline_budget": 4000,
 "candidate_budget": 4000,
 "shared_metrics": [
 "quality",
 "token",
 "time",
 "handoff_count",
 "manual_followup",
 ],
 "stop_rules": [
 "quality did not improve",
 "handoff_count stayed high",
 "summary_fields kept expanding",
 "rollback became hard",
 ],
}

观察项单智能体记录多智能体记录怎么读危险信号

质量正确率、覆盖度、可用性。同一套指标。先看是否真的更好，再看好多少。答案看起来更长，但判断没有更稳。

成本总 token、工具调用、耗时。同一任务下的总消耗。成本要和质量一起看，不要单独报喜。结果没有明显变好，成本却先翻上去。

交接无交接或一次完成。分派、汇报、合成、追问。交接越多，越要看字段有没有收紧。每个子智能体都在重复背景。

回退能否直接修正。失败后能否快速退回单体。回退规则要在试点前写清楚。失败后还要重新设计整个路由。
