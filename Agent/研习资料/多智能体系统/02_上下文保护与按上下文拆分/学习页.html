上下文保护

上下文保护与按上下文拆分

本页只处理一个判断：当某个子任务会把大量无关材料带回主上下文时，怎样把它隔离到子智能体里，同时保住主智能体继续判断所需的字段。锚定例子一直是“客服查订单 + 技术诊断”，因为它同时有长材料、关键字段和明显的误判后果。

读完本页应该能做什么

  • 能区分“上下文太长”和“上下文被污染”，知道先查哪一个。
  • 能写出子智能体的返回合同，明确哪些字段要保留、哪些材料要留在子上下文里。
  • 能判断一个任务应该按信息边界拆，还是先留在同一个上下文里继续做。

0. 学习目标

goal

这一页只处理一个工程判断：当高噪声材料会拖累后续推理时，怎样把它关进子上下文里，同时让主智能体只接收继续诊断所需的字段。阅读时先看状态变化，再看返回合同，最后用练习区确认你会不会把“长”和“污染”分开。

先识别污染

看到长工具结果时，先判断它是否属于后续推理的噪声源。

再写返回合同

能列出子智能体必须返回的字段，并把无关材料挡在外面。

最后判边界

能判断该按信息边界拆，还是先留在同一个上下文里继续做。

1. 先抓主线

overview

上下文窗口说的是模型一次能看到多少材料；上下文压力说的是材料虽然还没超窗，但已经开始挤压注意力；上下文污染说的是材料里混进了当前判断不需要的东西。三者常常一起出现，但处理方式不同。

这一页的顺序也很固定：用户提问先进入主智能体，主智能体判断有没有高噪声子任务；如果有，就把这部分交给子智能体；子智能体只回最小摘要；主智能体继续诊断；如果摘要开始膨胀，就退回单体基线。

用户问题→ 识别噪声→ 子智能体查单→ 最小摘要→ 主智能体继续诊断
概念直觉你要看什么写错后的症状
上下文窗口能装多少。总 token 和模型上限。超窗后直接丢信息。
上下文压力还没超窗，但开始挤。材料是否已经让注意力分散。关键约束被淹没。
上下文污染混进了不该留的东西。当前推理是否被无关材料干扰。主智能体开始追错线索。

你现在应该能做什么：看到一段很长的材料时，先判断它是否属于污染源，再决定是否隔离；超窗检查放在后面。

2. 上下文状态快照

ledger

把上下文当成一种会被读取、写入和污染的状态，会比把它当聊天记录更容易判断。你要看的是：谁在读、谁在写、写进去的东西后面还会不会被继续用。

阶段读什么写什么如果写错下一步检查
主智能体接单用户问题、时间范围、订单号。拆分判断和任务边界。问题范围混乱，后面很难收束。先写出必须保留的字段。
子智能体查单完整订单历史和工具返回。最小摘要、缺失项、证据来源。摘要仍然带着噪声。删掉与诊断无关的材料。
主智能体诊断摘要、错误码、必要上下文。结论和追问。开始引用物流、地址等无关信息。回看字段清单是否过肥。
回退决策质量、token、耗时、交接次数。继续拆或回退单体。没有可比证据，只剩感觉。先回到单智能体基线。

判断句

如果子智能体返回的内容，主智能体还要再读一遍原始材料，那说明子上下文没有真正减噪，拆分边界还不干净。

3. 客服订单：噪声和信号怎么分

example

客户说“专业版无法激活”时，订单查询通常会返回购买记录、地址、发票、物流、历史沟通和激活信息。真正要做技术诊断时，主智能体通常只需要订单是否有效、产品型号、最近激活时间、错误码和缺失信息。

坏流读：整份订单历史

写：一大段原始材料。症状：主智能体开始引用地址、物流和寒暄，技术诊断被拖偏。

好流读：高噪声订单材料

写：订单是否有效、产品型号、最近激活时间、错误码、缺失信息。症状：主智能体可以直接继续诊断。

薄摘要读：查单结论

写：一句“订单正常”。症状：没有错误码和时间线，主智能体不知道下一步该追问什么。

客服订单原始返回pseudo
order_lookup_result = {
  "order_status": "active",
  "product": "Pro",
  "last_activation": "2026-06-20 14:22",
  "error_code": "E42",
  "address": "...",
  "invoice": "...",
  "shipping_history": "...",
  "chat_history": "..."
}

这段材料本身没有错，错的是后续上下文是否把不相关部分也继续背着走。上下文保护做的事，就是把它们分开。

失败后果

如果订单地址、发票和客服寒暄继续留在主上下文里，模型会更容易把物流问题当成激活问题；如果摘要只剩一句“订单正常”，主智能体又会失去继续追问的线索。

4. 手工追踪：一条订单如何变成摘要

trace

这一段把同一条订单从原始返回走到主智能体可用摘要。先看原始输入，再看允许保留的字段，再看丢弃项。只要其中一步多留一个字段，主上下文就会重新背上噪声；少留一个字段，后面就会卡诊断。

阶段输入保留什么丢弃什么写错后的症状
原始查单整份订单历史和工具返回。状态、型号、激活时间、错误码、缺失项。地址、发票、物流、寒暄。主智能体开始跟着无关细节走。
摘要构建白名单字段和证据来源。能支持下一步判断的最小字段。任何不在白名单里的材料。摘要变肥，主上下文又要回读原文。
主智能体诊断摘要、错误码、时间线。根因分支和下一步追问。重复出现的原始订单文本。诊断绕圈，不能直接进入判断。
回退判断质量、token、交接次数。是否真的减噪。没有比较价值的装饰性材料。试点说不清成败，只剩“看起来更完整”。
摘要构建示例pseudo
ALLOWED_FIELDS = [
    "order_status",
    "product",
    "last_activation",
    "error_code",
    "missing",
    "evidence",
]

def summarize_order(result):
    summary = {key: result[key] for key in ALLOWED_FIELDS if key in result}
    summary["dropped"] = [k for k in result.keys() if k not in ALLOWED_FIELDS]
    return summary

这段代码的作用是做字段白名单。它只留下白名单字段，其他材料都被丢弃。白名单写得越清楚，主智能体越不容易把地址、发票和客服寒暄带进技术诊断。白名单写得越模糊，子智能体就越像一个小型垃圾回收站，最后还要主智能体自己清理一遍。

操作提示

看这段追踪时，先把每一列当成一次状态变换：读什么、保留什么、丢掉什么、错了以后会出现什么症状。你一旦能把症状说准，就能更快判断是字段太肥，还是字段太薄。

你现在应该能做什么：给任意一个高噪声工具结果写一份白名单，然后用“保留什么、丢弃什么、错了会怎样”三列把它检查一遍。

5. 返回合同：筛字段，保留最小摘要

contract

子智能体的返回接口应当只保留主智能体继续推理需要的结构化字段。如果摘要写成散文，主智能体通常还是得回去读原始材料，这就不算减噪。

最稳的写法，是先列出必须保留的字段，再明确哪些材料必须留在子上下文里，最后把缺失项直接暴露出来。这样主智能体不用猜，也不会把缺失当成事实。

子智能体返回合同pseudo
def build_order_summary(order_history, user_problem):
    return {
        "order_status": order_history["status"],
        "product_model": order_history["product_model"],
        "last_activation_time": order_history["last_activation_time"],
        "error_code": order_history["error_code"],
        "missing": ["需要人工确认的字段"],
        "evidence": ["状态记录", "激活记录", "错误日志"]
    }
字段为什么保留为什么不保留更多
订单是否有效决定是否继续技术诊断。完整购买历史通常不影响当前故障。
产品型号不同型号可能对应不同故障路径。收货地址、优惠信息对技术判断无关。
最近激活时间帮助确认故障发生时点。完整物流节点不必继续占位。
错误码直接指向诊断分支。聊天寒暄会干扰模型。
缺失信息提示主智能体下一步追问。不要把“不确定”伪装成确定结论。

判断句

如果你不能清楚写出“主智能体后续要用什么字段”，就不要让子智能体先写长摘要。先写接口，再写内容。

6. 按上下文拆，不按岗位拆

decomposition

问题导向拆分会说“一个智能体规划，一个智能体实现，一个智能体测试，一个智能体评审”。这种拆法看起来像团队，实际上经常把同一批背景压成几段摘要，最后变成传话游戏。上下文导向拆分问的是：哪些工作可以在不共享大量背景的情况下独立完成？

如果两个子任务需要频繁解释“我为什么这样做”，它们大概率应该留在同一个上下文里。只有当任务边界、工具边界或者数据边界足够清楚时，拆分才会真正减少污染。

边界类型是否适合拆原因检查问题
独立研究路径适合信息源和分析方向可以分开。各子任务是否能独立得出结果？
清晰接口的组件视情况适合接口合同能减少沟通。接口是否足够稳定？
黑盒验证适合验证者只需输入输出和标准。是否能不看构建过程就检查？
同一功能的顺序阶段通常不适合共享设计理由和失败尝试。交接摘要会丢掉什么？
紧耦合组件不适合需要频繁同步状态。是否一直来回问对方？
 
岗位式拆分：同一功能被切成连续交接，背景会越传越薄。

你现在应该能做什么：先把需求归到“独立”“可隔离”或“强相关”之一，再决定是否拆分；先画角色框只会把边界问题藏起来。

7. 先知道，先别钻

boundary

这一页先不追并行化、专业化、验证子智能体、路由策略和更复杂的协调机制。它们会在第 03 和第 04 单元回来，但现在先把“怎样把噪声留在子上下文里”这一件事做扎实。

相关概念这一页先怎么处理后面在哪里回来
并行化只把它当成一种可能的后续选择。第 03 单元。
专业化只记住它依赖清晰的工具和领域边界。第 03 单元。
验证子智能体只先记住验证者要看具体证据。第 04 单元。
路由只把它看成后续编排问题，不在这里展开。第 03 单元和第 04 单元。
回退条件先记住“字段开始膨胀就退回单体”。第 04 单元。

边界提醒

现在先别把路由、验证、并行、专业化一起端上来。只要还能把高噪声材料隔离干净，这一页的任务就算完成了。

8. 操作桥：把这页变成你自己的判断清单

bridge

真正落地时，先做三步。第一步，把你正在处理的任务写成一句话，找出会制造噪声的那一段。第二步，列出子智能体必须返回的字段白名单，再把无关材料写进丢弃清单。第三步，把这两个清单和单智能体基线放到同一张表里比较。

如果你面对的是客服、排障、资料检索、订单核对这一类任务，这个桥接方法会特别直接：先查单，再筛字段，再继续主诊断。只要某一步开始把字段越塞越多，或者主智能体还得重新回读原始材料，就说明拆分边界没有收紧。

你要写什么写成什么样合格信号不合格信号
噪声源一句话说清是哪段材料最容易拖累后续判断。你能指出具体字段或具体工具返回。只会说“信息很多”。
字段白名单列出主智能体真正要用的字段。字段数量有限，且能支持下一步判断。字段越列越长。
丢弃清单明确哪些材料不该留在主上下文里。地址、发票、寒暄之类被清楚挡住。把所有内容都“保险起见”留下。
回退条件写明什么时候放弃拆分。质量不升、摘要变肥、交接变多时能退回。只准进，不准退。

你下一步该做什么

  • 拿一条真实任务，写出噪声源和字段白名单。
  • 把白名单和丢弃清单放进同一张检查表。
  • 和单智能体基线比较质量、token、耗时和交接次数。

如果你已经能把这三步写清楚，说明上下文保护已经从概念变成动作了。下一步再去看第 03 单元，你会更容易判断哪些任务值得并行，哪些任务应该继续保持单上下文。

9. 摘要实验室：少一个字段会怎样

lab

这一段专门演示一个常见失败：摘要看起来很整齐，但漏掉了一个关键字段。场景还是订单诊断，主智能体想判断是否继续排障，子智能体却把“地址、发票、物流”都保留下来，却漏掉了错误码。表面上摘要更长了，实际上可用信息更少了。

你可以把这件事看成一次筛字段实验。主智能体需要的是能继续决策的字段，不是把原始记录再压一遍。只要错误码不在，后续诊断的分支就会变多；只要地址还在，主上下文就会继续背着无关信息。

字段保留理由缺失后的后果是否进入主上下文
order_status决定是否还值得继续诊断。主智能体不知道订单是否有效。是
product_model不同型号对应不同故障路径。主智能体无法路由到正确分支。是
last_activation_time帮助定位故障发生点。主智能体看不出故障前后关系。是
error_code直接指向诊断分支。主智能体只能继续追问，不能直接判断。是
address和技术诊断无关。主上下文被噪声占位。否
invoice和技术诊断无关。主智能体容易跟着财务信息走偏。否
摘要守门代码pseudo
REQUIRED = {"order_status", "product_model", "last_activation_time", "error_code"}
DROP = {"address", "invoice", "shipping_history", "chat_history"}

def gate_summary(summary):
    missing = sorted(REQUIRED - set(summary))
    kept_noise = sorted(set(summary) & DROP)
    if missing:
        return {
            "ok": False,
            "missing": missing,
            "next_action": "keep the query in the child context",
        }
    if kept_noise:
        return {
            "ok": False,
            "kept_noise": kept_noise,
            "next_action": "remove non-diagnostic fields before handing off",
        }
    return {"ok": True, "next_action": "send to the main agent"}

失败后果

如果这个守门器放行了错误码缺失的摘要，主智能体会卡在追问上；如果它放行了地址和发票，主智能体会继续背着噪声诊断。两种情况都说明摘要没有守住边界。

这段实验最值得记住的地方，不是代码写法，而是检查顺序。先看主智能体需要什么字段，再看摘要里留下了什么，再看有没有把无关字段带回去。只要这三步能跑通，你就能直接用它检查练习 7 的答案。

你现在应该能做什么：拿一个你自己的高噪声任务，写一张“保留 / 丢弃 / 缺失后果”表，再用一段守门代码检查摘要是否合格。

10. 三步自检：白名单、黑名单、回退线

check

真正把上下文保护做稳，最后要落到一套自检顺序：先看白名单里有没有主智能体真正要用的字段，再看黑名单里有没有不该回来的噪声，最后看回退线是否清楚。这个顺序比“摘要写得短不短”更重要，因为短摘要也可能缺字段，长摘要也可能已经变成噪声。

还是用订单激活失败这个例子。白名单应当保住订单状态、产品型号、最近激活时间、错误码和缺失信息；黑名单应当拦住地址、发票、物流和寒暄；回退线则告诉你什么时候必须退回单体。只要其中一项失守，主智能体就会重新背起不该背的材料，或者在诊断时找不到下一个分支。

检查点合格时长什么样不合格时长什么样你要怎么改
白名单字段少，但都能支持下一步判断。字段越列越多，甚至开始写成原始摘要。删掉和诊断无关的字段。
黑名单地址、发票、物流等都被明确挡住。无关字段仍然能回到主上下文。把丢弃清单写进子智能体返回合同。
回退线质量不升、摘要变肥、交接变多时能立即退回。只能继续补角色，不能回到基线。把回退条件写进试点协议。
摘要守门器pseudo
def accept_summary(summary):
    white = {"order_status", "product_model", "last_activation_time", "error_code", "missing"}
    black = {"address", "invoice", "shipping_history", "chat_history"}
    if not white.intersection(summary.keys()):
        return False, "missing diagnostic fields"
    if black.intersection(summary.keys()):
        return False, "noise leaked back into the main context"
    if len(summary.keys()) > len(white):
        return False, "summary is growing beyond the contract"
    return True, "safe to hand off"

失败后果

白名单守不住，主智能体会失去错误码和时间线；黑名单守不住，地址和发票会重新占位；回退线守不住，候选方案失败后会继续堆角色，最后很难再回到可比较的基线。

这一步是前面所有内容的收束：如果你能凭白名单、黑名单和回退线判断摘要是否可用，说明你已经能实际检查一个子智能体输出，而不只是会描述上下文保护的概念。

你现在应该能做什么：拿练习 7 的字段清单和练习 13 的试点计划，直接用这一张三步自检表复核一遍，看看你的摘要是不是已经能交给主智能体。

11. 小范围试点：把判断变成实验

pilot

多智能体不该一上来就全量替换。更稳的做法，是拿一条真实任务做小范围试点：同样的输入、同样的预算、同样的验收标准，只把拆分结构换掉。这样才看得出结构变化到底带来了什么。

试点协议pseudo
trial = {
    "task": "同一条真实任务",
    "baseline_budget": 4000,
    "candidate_budget": 4000,
    "shared_metrics": [
        "quality",
        "token",
        "time",
        "handoff_count",
        "manual_followup",
    ],
    "stop_rules": [
        "quality did not improve",
        "handoff_count stayed high",
        "summary_fields kept expanding",
        "rollback became hard",
    ],
}
观察项单智能体记录多智能体记录怎么读危险信号
质量正确率、覆盖度、可用性。同一套指标。先看是否真的更好，再看好多少。答案看起来更长，但判断没有更稳。
成本总 token、工具调用、耗时。同一任务下的总消耗。成本要和质量一起看，不要单独报喜。结果没有明显变好，成本却先翻上去。
交接无交接或一次完成。分派、汇报、合成、追问。交接越多，越要看字段有没有收紧。每个子智能体都在重复背景。
回退能否直接修正。失败后能否快速退回单体。回退规则要在试点前写清楚。失败后还要重新设计整个路由。

停手信号

  • 单智能体已经能稳定完成任务，多智能体只是在复述同一份背景。
  • 子智能体输出不断膨胀，主智能体还要回去读原始材料。
  • 交接次数增加后，耗时和人工追问一起变多。
  • 失败后没有简单回退路径，只能继续补角色。

你现在应该能做什么：把任意一条真实任务写成小范围试点，明确基线、候选、指标和停手条件。只要这四件事都能落到纸面上，才算开始做工程判断。

12. 常见误区、后果和修法

mistakes
误区错因后果修法
把长上下文都拆给子智能体没有区分“长但相关”和“长且无关”。协调成本白白增加。先判断材料是否会污染后续推理。
子智能体返回完整摘要害怕丢信息，于是把噪声又带回来。主上下文继续被污染。提前定义主任务需要的字段。
按岗位拆同一任务套用人类组织分工。交接摘要丢掉动机和边界。按上下文边界拆，强相关工作留在一起。
只看摘要是否短把短当成好。关键证据被压掉。检查摘要是否足够支持下一步推理。
没有回退条件只想着拆，不想着退。候选方案卡住，评估也卡住。把回退写进试点协议。
错误拆分链anti-pattern
agents = [
    PlannerAgent(),
    CoderAgent(),
    TesterAgent(),
    ReviewerAgent()
]

# 所有智能体都在处理同一个功能，但每一步都要重新解释背景。
# 测试者不知道实现时做过哪些权衡，评审者看不到探索过程。
run_handoff_chain(agents, task="实现同一个功能")

这个错误的根因是拆分边界和上下文边界不一致；四个智能体本身并不能解决这个问题。它把强相关工作切开，每次交接都会压缩信息，最后很容易变成传话游戏。

练习

diagnostic
先抓住一句话

上下文保护的目标，是把高噪声子任务隔离开，只让主智能体看到后续推理需要的字段。

答题时别漏的检查项
  • 这段材料会不会污染下一步？
  • 子任务需要返回哪些字段？
  • 哪些材料应该留在子上下文里？
  • 拆分会不会变成传话游戏？
窗口能装多少材料。
压力材料多到注意力开始挤。
污染无关材料干扰后续判断。
边界少量接口能否说清。

一、基础概念

先检查你能不能分清窗口、压力和污染，再看单体基线为什么要先跑。

复习一下上下文污染指的是混入当前推理不需要的材料。长但相关，和长且无关，处理方式不同。
choicebaseline

1. 当一个工具结果返回几千 token 时，主智能体首先要问什么？

true/falsecontext

2. 只要上下文没有超出窗口，就不可能发生上下文污染。

fillsummary

3. 子智能体返回的应该是主智能体继续推理需要的最小 ______。

short answerreasoning

4. 用 2-4 句话说明“上下文窗口”和“上下文污染”为什么不是一回事。

回答时要同时说“装得下多少”和“会不会混进不需要的材料”。

参考答案 / 评分要点

上下文窗口说的是模型一次能看到多少材料，偏向容量问题；上下文污染说的是材料里混入了当前推理不需要的东西，偏向干扰问题。两者可以同时出现，也可以单独出现，处理方式不同。

二、字段接口

这一组检查你能不能把子智能体的返回接口写得够短，但不缺关键证据。

复习一下摘要不是散文，接口要能支持主智能体继续做判断。地址、发票、寒暄通常不该回到主诊断。
choicecontract

5. 订单子智能体的返回更像什么？

true/falsenoise

6. 收货地址、发票和完整客服寒暄通常都应该返回给主诊断 Agent。

short answerevidence

7. 为“激活失败”设计订单子智能体输出，列出 5 个应该返回的字段。

至少包含订单状态、产品型号、最近激活时间、错误码和缺失信息中的几项。

参考答案 / 评分要点

合格答案应围绕诊断字段展开，例如：订单是否有效、产品型号、最近激活时间、错误码、缺失信息、证据来源。关键不是字数，而是这些字段能让主智能体继续判断。

三、拆分边界

这一组训练你区分“看起来能拆”和“真的应该拆”。

复习一下如果两个子任务总要解释“我为什么这样做”，它们大概率应该留在同一个上下文里。独立研究和黑盒验证更容易拆。
choiceboundary

8. 下列哪种边界最适合拆给不同子智能体？

true/falsehandoff

9. 如果两个子任务需要频繁解释“我为什么这样做”，它们通常应该拆开。

short answerdiagnosis

10. 用 2-4 句话解释为什么 Planner、Coder、Tester、Reviewer 容易把同一功能做成传话游戏。

参考答案 / 评分要点

要点是：这些阶段共享大量背景，但交接时只能传摘要；每次摘要都会丢掉动机、边界条件和失败尝试；最后测试者和评审者缺少“为什么这样做”的上下文，只能继续问人或回头读原材料。

四、试点与回退

最后一组把判断写成试点计划，看看你会不会停手，也会不会回退。

复习一下试点是为了看质量是否真的改善、成本是否真的值得。没有回退条件，拆分就不完整。
choicerollback

11. 如果候选方案的质量没明显提升，但 token 和交接次数都上升了，你该怎么做？

choicemetrics

12. 下列哪一条最像应该触发回退的信号？

synthesispilot

13. 把一条真实任务写成小范围试点计划：基线、候选、指标和停手条件分别是什么？

答案要能直接拿去比较，不能只写“试试看”。

参考答案 / 评分要点

合格答案至少要包含：同一条真实任务、单智能体基线、子智能体候选、共享指标（质量、token、耗时、交接次数、人工追问等）以及明确的停手条件。只要这四件事能落到纸面上，才算开始做工程判断。
