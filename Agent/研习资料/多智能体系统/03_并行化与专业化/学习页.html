并行化与专业化

这一页只处理两种拆分理由：并行化和专业化。你会先判断多个方向是否真的彼此独立，再判断工具、提示词和领域边界是否真的分开，最后把“要不要多开几个智能体”改写成可以拿去比较的试点。

0. 学习目标

goal

这一页只回答一个工程判断：什么时候值得把任务拆成多个上下文，什么时候先留在单智能体里更稳。你需要先看懂并行化为什么买的是覆盖度，再看懂专业化为什么买的是更窄的工具和任务边界，最后能把“想拆”改写成可以比较的实验。

先看并行

能判断多个方向是否真的彼此独立，是否值得同时探索。

再看专业化

能判断工具、提示词和领域知识是否真的分开。

最后看回退

能把拆分写成试点、指标、合成规则和回退条件。

1. 先抓主线

overview

这一页的顺序很固定：先看单智能体是否已经有基线，再看任务有没有真实约束，然后才讨论并行化、专业化和协调成本。顺序一倒，判断就容易只剩“看起来更高级”，却丢了适配性。

单智能体基线→ 真实约束→ 协调成本→ 小范围试点→ 是否拆分
约束先问什么看什么证据常见失败症状
并行化多个方向是否彼此独立？独立来源、独立判断、独立初稿。子智能体都在查同一批材料。
专业化工具、提示词或领域边界是否真的分开？更窄的权限、更稳定的返回格式。只是把角色名换得更像团队。
工具很多能否先按需发现，再考虑拆智能体？工具搜索、分组、懒加载是否能减负。路由和维护成本比工具选择更贵。

边界提醒

这一页先不追验证子智能体、评测平台、框架名字和落地流水线。先把并行和专业化判断清楚，后面再看验证和回退会轻很多。

2. 并行化适合什么任务

parallel

并行化适合大搜索空间任务。一个智能体在有限上下文里只能探索有限方向；多个子智能体可以同时查不同来源、不同地区、不同假设或不同候选方案。关键条件是每个子任务能否独立产出可合成的结果。

拆分读：原始任务

写：互斥面向和边界。症状：如果面向本身就互相依赖，后面合成会被拖垮。

搜索读：各自的面向

写：独立证据和初稿。症状：五个子智能体查到同一批材料，覆盖度并没有变高。

回传读：搜索结果

写：结论、来源、证据强度、未解决问题。症状：如果把整份原始材料都回传，主上下文会重新被污染。

合成读：多个局部结论

写：统一结论和下一步。症状：如果只拼接子报告，冲突和重复会一起留下来。

适合并行的信号不适合并行的信号你要看的证据
多个方向能各自得出初稿。每一步都依赖上一轮结果。各方向是否真的不等彼此也能推进。
信息源彼此差异大。所有方向都查同一批资料。搜索结果是否带来新信息。
合成规则可以提前写清。合成要靠主智能体临场拍脑袋。返回格式是否稳定、短而可比较。
研究型并行流程
query = "比较三个市场的企业 AI 采用障碍"

facets = lead_agent.split(query, by=[
    "亚洲市场",
    "欧洲市场",
    "北美市场"
])

findings = [
    research_subagent(facet, return_fields=[
        "结论",
        "来源",
        "证据强度",
        "未解决问题"
    ])
    for facet in facets
]

report = lead_agent.synthesize(findings, rules=[
    "去重",
    "标冲突",
    "保留不确定点"
])

3. 并行化提高覆盖度，也会抬高成本

cost

并行跑多个智能体，墙钟时间可能下降，但总计算量通常会上升。每个子智能体都要带自己的任务说明、必要背景和回传格式，最后主智能体还要合成、去重和解释冲突。多智能体通常比单智能体更全面，也通常更贵。

$$C_{total} \approx n(c_{ctx}+c_{coord}) + c_{synth}$$

n 是子智能体数，c_{ctx} 是每个子智能体需要的必要背景，c_{coord} 是分派和回传的协调成本，c_{synth} 是主智能体的合成成本。这个式子只是估算。

成本来源具体表现坏结果你可以怎么量
上下文复制每个子智能体都要加载同一批背景。token 增长快，注意力被分散。看每个方向的必要背景是否真的不同。
协调消息分派、汇报、追问、合并都要占上下文。协调比执行更贵。统计交接轮数和重复说明次数。
交接摘要后续智能体只能看到被压缩的结果。动机、边界条件和失败尝试丢失。检查摘要是否还能支持下一步判断。
维护成本多个角色、多个权限、多个输出格式。改一个流程要同步多处提示词。看修一个小问题是否要动很多位置。
基线对照伪代码
baseline = run_single_agent(task)
candidate = run_parallel_agents(task)

compare(baseline, candidate, metrics=[
    "结论覆盖度",
    "引用准确性",
    "总 token",
    "耗时",
    "失败原因"
])

判断句

如果并行后的覆盖度没有明显变高，却先把 token 和合成成本拉上去了，就先别把它当成好方案。

4. 专业化要减少选择负担

specialization

专业化的目标，是让每个智能体面对更小、更清晰的工具和任务空间。一个通用智能体如果同时管理 CRM、营销、通知、数据分析、代码执行和文档写作，模型会花很多注意力判断“我现在该用哪个工具”。当边界清楚时，专业化才会更稳。

工具集专业化

每个智能体只看自己领域的工具，权限更清楚，选择也更少。

系统提示词专业化

不同智能体有不同目标、风险约束和输出格式。

领域知识专业化

金融、医疗、法务等领域可以沉淀成稳定的术语和检查清单。

工具集读：工具类别

写：更窄的工具权限。症状：如果所有智能体仍然拿同一批工具，专业化只是换名字。

提示词读：不同工作准则

写：不同输出格式和停止条件。症状：如果只是把“专家”写进名字，边界并没有变窄。

领域读：领域规则

写：专门术语和检查清单。症状：每次任务都要重新解释背景，专业化就站不住。

专业化类型变窄前变窄后主要读取什么主要写出什么失败症状
工具集所有智能体都拿同一批工具。每个智能体只看自己领域的工具。工具类别和权限。更少的工具选择。工具仍然混在一起，选择压力没降。
提示词统一的工作准则和格式。不同智能体有不同约束。目标、风险、输出格式。更稳定的返回结构。只是改了名字，没有改行为。
领域通用背景到处都要重新解释。领域术语和规则固定下来。领域规则和检查清单。更窄的判断空间。每次任务都要重讲背景。

判断句

如果专业化没有明显减少工具选择、任务目标或领域规则的混乱，它仍然只是换了几个角色名，没有真正收窄边界。

5. 专业化的代价：路由会变难

routing

专业化系统需要一个路由者，决定请求交给谁。路由看起来只是分派，实际上是主要风险：分错了，后面再专业也没用。一个请求可能需要一个专家，也可能需要一个顺序编排，也可能需要先澄清再分派。

请求形态优先动作判断依据常见失败症状
一个领域就能完成交给单个专家。能力组合清楚，不需要别的上下文。路由把简单请求拆成多段。
要先查再发顺序编排两个专家。有明确的先后依赖。把依赖关系错当成并行。
跨多个业务域先分析任务结构，再决定怎么分派。需要多个能力同时参与。只按关键词匹配，漏掉编排关系。
路由置信度低先让通用智能体澄清。请求结构本身还不清楚。硬分派给某个专家，结果越做越偏。
路由判断伪代码
plan = router.analyze(user_request)

if plan.requires(["customer_lookup", "message_send"]):
    customers = crm_agent.find(plan.filter)
    result = message_agent.send(customers, plan.message)
elif plan.single_domain:
    result = route_to_specialist(plan)
else:
    result = general_agent.clarify(plan)
工具很多但同一流程：先做工具搜索或工具分组，不要急着拆成多个智能体。

路由四步

  1. 先识别请求需要哪些能力。
  2. 再判断能力之间是顺序依赖还是并行依赖。
  3. 然后定义每一步的输入输出格式。
  4. 最后在置信度低时先澄清，再分派。

观测指标

专业化上线后，至少要看路由错误率、跨专家交接失败、重复工具调用和最终任务完成率。没有这些指标，就很难判断专业化到底是在减负，还是在制造新的路由成本。

6. 工具很多时，先考虑按需发现

tool

工具太多是专业化的常见动机。工具数量一多，模型会花很多注意力理解选项，这时可以先试工具搜索、按需加载或工具分组。只有当工具天然属于清晰业务域，而且路由也清楚时，拆成多个专家才更有意义。

情况优先方案为什么不合适时的症状
工具多但同属一个流程工具搜索或工具分组。仍然需要同一个上下文判断。拆成多个智能体后，路由比工具选择更贵。
工具分属清晰业务域专业化智能体。每个智能体面对更窄的工具空间。各智能体仍然拿同一批工具，收益很小。
请求常跨多个业务域路由加编排。需要明确谁先做、结果如何传递。只按关键词匹配，漏掉交接关系。
工具权限风险不同专业化或权限隔离。敏感工具不该对所有任务开放。安全边界不清，权限扩得太大。
按需发现伪代码
plan = router.analyze(user_request)
tools = tool_registry.search(domain=plan.domain)

if len(tools) > 15:
    tools = tool_registry.load_only(plan.required_tools)

result = specialist.run(plan, tools=tools)

你现在应该能做什么

工具多的时候，先想工具搜索、分组和按需加载，再想拆智能体。只要工具边界还不清，专业化很容易把问题从“选工具”变成“选路由”。

7. 例子：两种很常见的拆法

example

两个最容易判断的例子，一种是并行研究，一种是专业化编排。前者看独立面向，后者看工具和业务边界。两种都能做，但判断依据不同。

例题 A：三个市场的趋势研究

亚洲、欧洲、北美三个市场的资料和判断路径彼此独立，适合拆成三个研究面向。每个子智能体只需要返回结论、来源、证据强度和未解决问题，主智能体负责去重和合成。

研究拆分示例
facets = ["亚洲市场", "欧洲市场", "北美市场"]
findings = [research_subagent.run(facet) for facet in facets]
final_report = lead_agent.combine(findings)

例题 B：CRM 加消息发送

“给未激活客户发提醒”同时涉及客户筛选和消息发送。这个请求更像顺序编排：先查客户，再发消息，不要硬拆成很多角色。

步骤输入输出容易丢失什么
客户筛选时间窗和业务条件。待提醒客户清单。筛选条件和来源。
消息发送客户清单和消息模板。发送结果和失败项。失败原因和回退动作。
任务更像哪种拆法为什么主智能体要保留什么不该继续带什么
三市场研究并行化。方向彼此独立，能各自出初稿。去重后的证据和冲突说明。每个市场的全部原始搜索过程。
未激活客户提醒专业化编排。先查后发，有明确顺序。客户清单、消息模板、发送结果。与发送无关的后台日志和临时列表。

8. 综合案例：一个混合请求先怎么拆

case bridge

现实里常见的是混合请求。比如用户先问“比较三个市场的采用障碍”，又要求“把结论转成给未激活客户的提醒话术”。这时不要先按岗位名拆，先把任务切开：研究部分适合并行，执行部分适合专业化，主智能体只保留两边交接需要的字段。

判断的关键是边界。研究分支需要独立面向、独立来源和独立初稿；执行分支需要客户筛选、消息发送和失败回报。只要一边的输出开始挤进另一边，说明路由没说清，拆分方式就该收紧。

阶段读什么写什么不该带什么失败症状
路由用户意图和约束。判断是并行、专业化还是单专家。完整调研笔记。把一个请求硬拆成很多层。
并行研究各市场面向。结论、来源、证据强度、未解决问题。其他市场的原始资料。重复搜索和冲突结论。
专业化执行客户清单和消息模板。发送结果、失败项、回退建议。研究材料和无关日志。路由错位，执行被研究污染。
主智能体合成局部结果。可交付答案和下一步。子任务的完整过程。报告过长，责任不清。
回退质量、token、交接次数。继续拆或回到单智能体。装饰性的流程描述。试点赢不了，还是硬拆。
混合请求路由伪代码
request = "比较三个市场的采用障碍，并给未激活客户发提醒"
plan = router.plan(request)

if plan.need_research:
    market_findings = parallel_research([
        "亚洲市场",
        "欧洲市场",
        "北美市场"
    ])
else:
    market_findings = general_agent.clarify(request)

if plan.need_action:
    customers = crm_agent.find_unactivated(plan.window)
    action_result = message_agent.send(customers, plan.template)
else:
    action_result = None

final_answer = main_agent.combine({
    "research": market_findings,
    "action": action_result,
    "rollback": "if either branch keeps expanding its summary, return to baseline"
})

如果这里拆错了，会坏在哪里

  • 把研究和执行放在同一个长上下文里，地址、物流和寒暄会污染技术判断。
  • 把三个市场都放进同一条顺序链里，覆盖度会下降。
  • 把 CRM 和消息发送都拆成独立角色，但没有先后顺序，路由会比任务更复杂。
  • 把所有工具一次性打开，模型会把时间花在选项上，判断就会被挤掉。

你现在应该能做什么：拿一个混合请求先画路由，再在每个分支里写返回合同。只要你还不能说清哪个分支负责并行，哪个分支负责专业化，就不要急着继续扩角色。

9. 小范围试点：把判断变成实验

pilot

不管是并行还是专业化，都不要一上来全量替换。更稳的方式，是拿一条真实任务做小范围试点，同样输入、同样预算、同样验收标准，只改拆分结构。这样才能看出结构变化到底带来了什么。

试点时最重要的是记录要完整。单智能体和多智能体必须放到同一张记录表里，看质量、token、耗时、交接次数、人工追问次数和回退条件。只要其中任何一项明显变差，就不能只盯着“角色更多、过程更像协作”这一点下结论。

观察项单智能体记录多智能体记录怎么读危险信号
质量正确率、覆盖度、可用性。同一套指标。先看是否真的更好，再看好多少。答案看起来更长，但判断没有更稳。
成本总 token、工具调用、耗时。同一任务下的总消耗。成本要和质量一起看，不要单独报喜。结果没有明显变好，成本却先翻上去。
交接无交接或一次完成。分派、汇报、合成、追问。交接越多，越要看字段有没有收紧。每个子智能体都在重复背景。
回退能否直接修正。失败后能否快速退回单体。回退规则要在试点前写清楚。失败后还要重新设计整个路由。
试点配置
trial = {
    "task": "同一条真实任务",
    "baseline_budget": 4000,
    "candidate_budget": 4000,
    "shared_metrics": [
        "quality",
        "token",
        "time",
        "handoff_count",
        "manual_followup",
    ],
    "stop_rules": [
        "quality did not improve",
        "handoff_count stayed high",
        "summary_fields kept expanding",
        "rollback became hard",
    ],
}
环节读什么写什么写错会怎样怎么回退
输入白名单任务目标、必要背景、约束。只给下一步真正要用的字段。背景越传越多，子任务开始复述原文。收紧字段，只保留下一步用得上的信息。
子任务回传局部结论和证据。结论、来源、证据强度、未解决问题。子报告变成长过程复述。删掉过程，只留可合成字段。
主智能体合成多份局部结果。统一答案和冲突说明。冲突被直接拼接，重复一起留下。先去重，再标冲突，再下结论。
失败回退质量、token、交接数。退回单体或更轻方案。继续补角色，路由越来越长。先退回基线，再改结构。
复盘记录基线与候选差异。可复查的决策理由。只记“感觉更顺”，下次还是照旧。把触发回退的条件写下来。

试点阶段最常见的误判，是把“分工更细”当成“结构更好”。真正要看的，是每次交接后字段有没有变少、责任有没有变清、回退有没有变容易。如果这些地方没有改善，再多的角色也只是把问题拆散，不是把问题解决。

阶段读什么写什么如果写错下一步检查
设定基线单智能体能做到什么。质量、token、耗时、失败案例。没有可比较对象。先把基线跑稳。
试点候选拆分后的输入输出。局部结果和交接字段。摘要开始膨胀。收紧字段白名单。
比较结果基线与候选指标。是否继续拆分。只能凭感觉判断。回到表格看数据。
决定回退成本与质量关系。回到单智能体或更轻方案。失败后还想继续堆角色。先退回基线，再重新设计。

停手信号

  • 单智能体已经能稳定完成任务，多智能体只是在复述同一份背景。
  • 子智能体输出不断膨胀，主智能体还要回去读原始材料。
  • 交接次数增加后，耗时和人工追问一起变多。
  • 失败后没有简单回退路径，只能继续补角色。

你现在应该能做什么：把任意一条真实任务写成小范围试点，明确基线、候选、指标和停手条件。只要这四件事都能落到纸面上，你就有资格谈要不要拆分。

10. 交接合同怎么写

handoff

试点真正容易失控的地方在交接。交接合同不是为了把流程写得漂亮，而是为了约束每一步只传下一步需要的字段。字段越短，主智能体越容易合成；字段越长，子智能体越像在复制原始上下文。

合同项要写什么为什么写错会怎样
目标这一轮要解决的具体问题。避免子智能体擅自扩题。范围越写越大，最后没人知道这轮到底交付什么。
输入白名单下一步必须用到的背景和约束。让后续智能体只看必要信息。把原始上下文整包传下去，污染后续判断。
回传白名单结论、来源、证据强度、未解决问题。保证主智能体可以直接合成。回传成过程记录，主智能体还得重读原材料。
异常处理什么时候停、什么时候退回单体。让失败有明确出口。出了问题只能继续补角色，路由越来越长。

    "goal": "只解决当前这一轮的问题",
    "input_whitelist": [

    "output_whitelist": [

    ],

你现在应该能做什么

你可以把任何一个子智能体的交接改写成“目标、输入白名单、回传白名单、异常处理”四件事。只要这四件事写不清，拆分就算做出来了，协作也还没有真正开始。

11. 先知道，先别钻

boundary

这一页先不追验证子智能体、早期胜利问题、框架选型和更复杂的评测流程。它们会在下一单元回来，但现在先把“要不要并行、要不要专业化”这两个判断做稳。

相关概念这一页先怎么处理后面在哪里回来
验证子智能体这里只把它当成下一步的检查者。第 04 单元。
早期胜利问题这里只提醒“不要只看一次通过”。第 04 单元。
回退条件这里只记住“成本和质量不对等就退回”。第 04 单元。
框架选型这里只不展开具体库名和工程细节。后续专题。
评测平台这里只看指标，不看平台名。后续专题。

边界提醒

现在先别把并行、专业化、验证和路由一次性混在一起。只要你已经能判断“哪些方向能并行、哪些边界值得专业化”，这一页就算完成了。

练习

diagnostic
先抓住一句话

并行化买的是覆盖度，专业化买的是更清楚的边界。两者都不是免费收益。

答题时别漏的检查项
  • 子任务是否真的独立？
  • 工具边界和领域边界是否清楚？
  • 路由和合成会不会吞掉收益？
  • 有没有回退到单智能体的条件？
并行看独立面向和覆盖度。
专业看工具、提示词和领域边界。
路由决定请求交给谁。
回退成本不值时回到基线。
