# 0. 学习目标 {#goal}
<!-- tag: goal -->

这一页只回答一个工程判断：主智能体已经能跑起来以后，什么情况下值得再拆一个验证子智能体，什么情况下先停在单体基线更稳。你需要先看懂验证为什么适合黑盒检查，再把标准写成可执行清单，接着识别早期胜利问题，最后判断多智能体到底是在放大质量，还是在增加表面通过的错觉。

先看职责
验证者只读产出和标准，不参与实现过程。


再看标准
标准要能跑、能测、能报告，不能只写一句“看看是否正常”。


最后看边界
能说清什么时候放行，什么时候回退到单智能体或自动测试。

# 1. 验证子智能体负责什么 {#verifier}
<!-- tag: verifier -->

验证子智能体的职责很窄：测试或验证主智能体的产出。它不负责重新规划，不负责顺手改代码，也不负责替主智能体解释为什么这样做。职责窄反而是优势，因为验证通常只需要输入、输出、规格和检查标准，不需要完整的实现历史。

这类拆分有一个好处：主智能体可以继续保留复杂上下文，验证子智能体只拿外部可观察的结果来判定。只要它能回答“结果是否满足规格、测试是否真的跑过、边界情况是否覆盖、该失败的输入有没有失败”，它就可以独立工作。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

验证子智能体伪代码复制

```python
verifier.run(
 artifact="主智能体生成的代码、文档或结构化结果",
 criteria=[
 "运行完整测试套件",
 "检查 lint 和类型错误",
 "验证输出 schema",
 "尝试边界输入和负例",
 "列出所有失败项和未执行项"
 ]
)
```

步骤主要读取什么主要写出什么如果做错会怎样


读产出最终代码、报告、schema 或结果文件。待验证对象。如果把实现历史也背进来，验证者会跟着参与实现。
对标准规格、测试、约束和规则。检查清单。如果标准只有一句“看看能不能用”，验证者会偷懒。
做检查对象和检查清单。通过、失败、未执行三类结果。如果只测成功路径，早期胜利就会发生。
报结果完整检查记录。可放行或需回退的判断。如果没写未执行项，读者会误以为全部都通过了。

# 2. 为什么黑盒验证有效 {#blackbox}
<!-- tag: blackbox -->

黑盒验证的意思很直接：验证者不依赖完整实现过程，只根据外部可观察行为判断结果。软件里，它看测试、lint、类型检查和 schema；内容里，它看引用、事实和格式；流程里，它看是否满足交付条件。只要结果可观察，验证就不必知道每一步是怎么写出来的。

这也是验证子智能体适合拆出来的根本原因。它的上下文可以比主智能体短得多，不需要带着所有历史权衡一起走。上下文少了，验证者就更像测试系统，而不是又一个会点头的聊天者。

场景验证者读什么不该读什么常见失败症状


代码检查测试输出、lint 结果、类型错误、schema。实现时的每次重构原因。验证者开始替实现者找理由。
文档核对引用、事实、格式和政策规则。长篇写作过程。最后只剩“读起来顺”这种主观判断。
流程放行是否满足交付条件、权限条件、审批条件。无关的中间草稿。没有正式条件，却口头宣布通过。
结果比对输入、输出和预期差异。内部思考链路。把调试过程误当成验证结果。

黑盒检查模板复制

```text
请只根据外部产出做检查，不要参与实现。
检查项：
1. 跑完整测试套件。
2. 检查边界输入和负例。
3. 报告所有失败项和未执行项。
4. 任何未执行的检查都不能默认通过。
输出格式：
- 通过项
- 失败项
- 未执行项
- 放行判断
```

# 3. 验证标准要具体到能执行 {#criteria}
<!-- tag: criteria -->

验证质量，最先看的是标准写得够不够硬。模糊标准会把验证者推向捷径。比如“检查这个功能是否正常”太松，它很可能只跑一个成功样例就宣布通过。更好的标准是把范围、动作和报告都写出来，连没有执行的检查也要列出来。

好的验证标准通常分成四类：具体标准、全面检查、负例测试、明确禁止偷懒。只要其中一类缺位，验证者就容易把“看起来可以”误读成“已经稳定可用”。

标准类型好写法差写法风险


具体标准运行完整测试套件，并报告所有失败。看看能不能跑。语句过松，验证者会偷懒。
全面检查覆盖正常、边界、异常和权限场景。试一两个例子。覆盖面太窄，隐藏问题会留下来。
负例测试输入无效参数，确认系统拒绝。只测成功路径。很多问题只在失败输入上露出来。
禁止偷懒未完成全部检查前不得标记通过。通过了就说一声。早期胜利会直接被放行。

可执行验证指令复制

```text
请运行完整测试套件，报告所有失败。
额外测试：空输入、超长输入、权限不足、网络失败。
如果任何检查未执行，请不要标记通过。
输出格式：通过项、失败项、未执行项、复现步骤。
```

你现在应该能做什么：把“检查一下”改写成一份具体到动作的验证清单，而不是一段礼貌但空洞的话。

# 4. 最大失败模式：早期胜利问题 {#early}
<!-- tag: early victory -->

验证子智能体最容易犯的错，是跑了一两个测试，看到通过，就提前宣布成功。问题不在于它“还不够认真”，而在于规则没写死。只要完整测试、边界输入、负例和报告格式没有被强制列出，早期胜利就会变成默认行为。

要压住这个问题，最有效的方法把“必须做什么”写成顺序明确的检查链。只跑成功路径不够，必须让验证者把没做的事情也写出来。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

观察到什么问题在哪里应该怎么改


只跑一个正常例子就通过。验证标准太松。强制完整测试和负例。
报告里没有未执行项。没测过的内容被默认通过。把未执行项单独列出来。
边界输入没测。成功路径掩盖了脆弱点。补空输入、超长输入、权限不足等边界。
结论只写“基本正常”。没有可复现的判断依据。写出通过项、失败项、复现步骤和放行判断。

# 5. 验证报告必须写出没做什么 {#report}
<!-- tag: report -->

可靠验证不是一句“通过”就够了。报告至少要让读者看出四件事：检查了什么、怎么检查、哪些通过、哪些没做。特别是未执行项，它能直接暴露验证盲区。没有这部分，读者很容易把“没测过”当成“没问题”。

所以验证报告最好写成结构化格式，而不是自由发挥。结构化为了让主智能体可以继续判断下一步应该修哪里、补哪里、回退到哪里。

验证报告模板复制

```text
验证结论：通过 / 不通过 / 无法判断

已执行检查：
- 检查项：
 方法：
 结果：
 证据：

失败项：
- 失败描述：
 复现步骤：
 可能影响：
 建议修复位置：

未执行项：
- 未执行原因：
 需要补充的环境、权限或数据：

放行判断：
- 是否允许放行：
- 如果不允许，阻塞原因：
```

字段作用不写会怎样


验证结论告诉主智能体能否继续。读者只看到细节，不知道是否阻塞。
已执行检查证明验证不是主观感觉。无法复现检查过程。
失败项指出具体修复目标。主智能体只能猜哪里坏了。
未执行项暴露验证盲区。没测的内容被误认为通过。
放行判断把失败和阻塞程度连起来。小问题和阻塞问题混在一起。

# 6. 验证回路：从请求到放行 {#flow}
<!-- tag: flow -->

验证子智能体真正有用的地方，把验证回路做短做硬。主智能体先给出产出，验证子智能体只看产出和标准，最后把结果压成可以决策的报告。这个回路一旦拉长，就会重新回到传话游戏：实现者解释自己、验证者追问自己、主智能体再补上下文，最后没人真的在做检查。

把它拆成四步看会更清楚。第一步是准备验证对象，只保留最终结果和必要的规格；第二步是执行检查，完整测试、边界和负例都要覆盖；第三步是写报告，把通过、失败、未执行分开；第四步是决定放行或回退。每一步都只做自己该做的那部分，才不会把验证者变成另一个实现者。

步骤读什么写什么常见失败症状修法


准备对象最终产出、规格、测试入口。验证范围。把实现过程也带进去，范围开始膨胀。只保留验证必须知道的最小字段。
执行检查检查清单、边界输入、负例。通过、失败、未执行。只测成功路径，验证者急着放行。把负例和边界写成强制步骤。
生成报告完整检查记录和证据。可复现的结论。只写“基本正常”，看不出哪里没测。把未执行项和复现步骤单独列出。
决定放行报告、失败项、成本和回退条件。放行或回退决策。有失败还硬放行，或不清楚何时回退。把阻塞项和停止线提前写好。

验证回路伪代码复制

```python
artifact = lead_agent.build(task)

verification_plan = {
 "checks": [
 "run full test suite",
 "cover edge inputs",
 "cover negative inputs",
 "check schema or policy rules"
 ],
 "report": [
 "passed items",
 "failed items",
 "not executed items",
 "evidence"
 ],
 "stop_rules": [
 "any blocking failure",
 "any skipped required check",
 "cost exceeds trial budget",
 "summary keeps expanding without new evidence"
 ]
}

report = verifier.run(artifact, verification_plan)

if report.blocking_failures:
 return lead_agent.repair(report)
if report.not_executed:
 return lead_agent.fill_gaps(report)
return lead_agent.release(report)
```

## box soft
### 回路里的三种信号
- 通过项说明当前检查已经满足最低门槛，但不代表可以放弃报告中的其他字段。
- 失败项说明系统需要修正，主智能体应该回到具体问题，而不是重新讲故事。
- 未执行项说明验证还没有闭环，没测过的内容不能被自动视为没问题。

## box warn
### 回路里最容易坏的地方
最常见的坏法有三个：一是验证者把对象带得太多，变成实现者的影子；二是检查写得太松，成功路径一通过就想结束；三是报告没写未执行项，结果读者误判放行条件。只要这三点任何一个没收住，验证回路就又会变长。

# 7. 失败诊断：验证放行为什么会错 {#diagnosis}
<!-- tag: diagnosis -->

最常见的失败，它只干了最容易干的那一小段。比如主智能体做了一个 CSV 导入功能，验证者先跑正常样例，看到成功就放行。问题往往“没测的地方被误判成过了”。空文件、缺列、重复用户、超大文件、非法编码，这些才是验证回路真正要抓住的边界。

诊断这种问题时，先别急着改实现。先看验证对象有没有带太多实现历史，检查清单是不是漏了边界输入，报告里有没有把未执行项写出来，放行判断有没有对应停止线。只要这四个位置里有一个断掉，验证者就会把“看起来正常”当成“已经可以放行”。

观察到的信号更可能的原因下一步先查什么如果还不行怎么办


只跑成功样例就通过。检查标准太松。验证任务里有没有强制完整测试和负例。把负例和边界写成硬性步骤。
报告里没有未执行项。验证者默认没测过也算过。报告模板是否要求未执行项单独列出。重新定义报告结构，再跑一次。
边界输入没被测到。验证任务没有覆盖空、缺、重、超、错这类输入。检查清单是否真的包含边界输入。补一轮边界测试，不要直接放行。
失败后主智能体只能重新讲一遍背景。回传字段太长，关键证据丢了。报告里有没有失败项、证据和复现步骤。把回传字段压短，只保留决策信息。

失败诊断函数复制

```python
def diagnose(report):
 if report.passed_only_sample:
 return "标准太松：补完整测试和负例"
 if report.missing_not_executed:
 return "报告不硬：把未执行项写进模板"
 if report.skipped_edges:
 return "边界没跑全：补空、缺、重、超、错"
 if report.summary_sprawls:
 return "回传太长：压缩成通过、失败、未执行"
 return "继续放行前，先看是否有阻塞项"

def next_action(report):
 diagnosis = diagnose(report)
 if "补" in diagnosis:
 return lead_agent.repair(report)
 if "压缩" in diagnosis:
 return lead_agent.trim_report(report)
 return lead_agent.release(report)
```

## box soft
### 诊断顺序
1. 先看检查有没有跑全，再看报告有没有写全。
2. 再看边界输入有没有覆盖。
3. 最后看回传是不是太长，导致主智能体读不到关键判断。

## box warn
### 失败后的正确动作
如果问题出在标准，回到验证指令；如果问题出在报告，回到模板；如果问题出在边界，补测试；如果问题出在回退条件，先写停止线。真正的修复把下一轮检查变得更硬。

你现在应该能做什么：拿到一份验证报告时，先判断它是在证明通过，还是只是在证明“跑过一遍”。只要你能从报告里分出标准、边界、未执行项和回退条件，就能很快判断问题应该回到哪一步。

# 8. 交接桥：验证结果怎么回到主智能体 {#bridge}
<!-- tag: bridge -->

验证子智能体不是终点，它的结果要交回主智能体做下一步。这里最容易犯的错，是把整份验证记录原样倒回去。那样主智能体又会被细节淹没，最后还得自己重新抽结论。更稳的做法，是把结果分成三层：可以直接放行的通过项、需要修正的失败项、以及暂时不能下结论的未执行项。

这个交接桥的核心是压缩，不是省略。压缩意味着你保留足够做判断的信息，例如复现步骤、失败原因、阻塞级别和建议修复位置；省略则会把关键证据丢掉。只要主智能体接收到的还是长段的解释、背景和过程，它就会重新被拉回实现细节里，验证子智能体的价值也会被削弱。

返回层级要保留什么要丢掉什么主智能体接收到后怎么用


通过项已确认的检查、证据和通过条件。无关的实现细节和重复说明。确认是否可以继续放行。
失败项失败原因、复现步骤、建议修复位置。冗长的探索过程。回到具体问题，安排修复。
未执行项未执行原因、缺失环境或权限。没有验证依据的主观推测。决定补测还是回退。
阻塞项会阻止放行的条件和停止线。只有“感觉不稳”这种模糊表述。明确下一轮是否还能继续拆分。

交接桥伪代码复制

```python
def handoff(report):
 payload = {
 "passed": report.passed_items,
 "failed": [
 {
 "item": item.name,
 "reason": item.reason,
 "evidence": item.evidence,
 "fix": item.fix_point,
 }
 for item in report.failed_items
 ],
 "not_executed": [
 {
 "item": item.name,
 "reason": item.reason,
 "missing": item.missing_input,
 }
 for item in report.not_executed_items
 ],
 "decision": report.blocking_reason,
 }
 return payload

def lead_agent_next_step(payload):
 if payload["failed"]:
 return lead_agent.repair(payload)
 if payload["not_executed"]:
 return lead_agent.schedule_retest(payload)
 return lead_agent.release(payload)
```

## box soft
### 压缩原则
- 把长报告变成短决策，但不要丢掉证据。
- 把失败项和未执行项分开，避免放行时混淆。
- 把修复位置写出来，主智能体才能直接回到问题点。

## box warn
### 交接错误的后果
如果交接桥做得太长，主智能体会重新读实现历史；如果交接桥做得太短，关键证据会消失；如果交接桥没有阻塞项，验证者就会把“没测完”伪装成“基本通过”。所以交接桥本身也要被验证。

# 9. 验证子智能体适合哪些场景 {#applications}
<!-- tag: applications -->

验证子智能体最适合标准明确、检查动作独立的场景。软件里，它检查测试、lint、类型和 schema；合规里，它检查政策、权限和审批条件；内容里，它检查引用和事实。关键检查条件能不能写清。

如果标准本身很主观、验证者必须理解完整实现权衡、或者任务小到同一智能体配一份清单就能做完，那就不一定要单独拆验证者。先把标准补清，再决定是否需要拆分。

场景适合验证什么不适合交给验证者什么典型收益


质量保障测试套件、lint、类型检查、schema。完整的实现思路和重构理由。把是否可用变成可复现的检查。
合规检查政策规则、敏感内容、审批条件。流程里无关的草稿和闲聊。把放行条件写成规则。
输出验证结构是否完整、字段是否齐全。主观的“写得顺不顺”。让结构问题先被发现。
事实核验引用是否存在、是否支撑结论。长篇写作过程和内部动机。把事实和判断拆开看。

## box warn
### 不适合单独拆验证者的情况
如果验证标准很主观、验证者必须理解完整实现权衡、检查环境无法给到子智能体，或者任务小到同一智能体配一份检查清单就能完成，就先别拆。先把规则写硬，再看是否真的需要一个独立检查者。

# 10. 多智能体落地前的采用清单 {#check}
<!-- tag: check -->

多智能体强大，但不是默认选择。落地前至少确认五件事：约束是否真实、拆分边界是否按上下文、验证是否明确、成本是否可接受、回退条件是否写清。只要有一项说不清，就先做小试点，不要直接把整个流程重构成多角色。

这一页的最后判断“拆了以后是否比单体更值得”。如果质量没有明显提升、成本和维护反而上升，就应该回退，而不是继续加角色名。

问题通过标准答不上来时怎么做


约束是否真实？有失败案例、指标或上下文证据。先跑单智能体基线。
边界是否按上下文？子任务共享背景少，接口清楚。重新合并或改边界。
验证是否明确？有具体、全面、含负例的检查。先补验证标准。
成本是否可接受？token、耗时和维护成本都记录了。做小试点，不要直接全量改造。
回退条件是什么？知道何时回到单智能体或工具搜索。先定义停止线。

## box info
### 上线后看什么
至少记录任务成功率、失败类型、总 token、平均耗时、未执行项、误放行次数和人工介入次数。没有这些记录，就很难判断多智能体是在帮忙，还是只是在制造更多协作痕迹。

# 11. 常见误区、后果和修法 {#mistakes}
<!-- tag: mistakes -->

验证子智能体最怕两件事：一是边界不清，二是报告不硬。前者会让它悄悄参与实现，后者会让它把没测过的东西当成通过。下面这张表把常见偏差写成可操作的修法。

误区错因后果修法


验证者参与实现职责边界不清。失去独立检查视角。只检查产出和标准，不碰实现过程。
只测成功路径默认第一个通过就够了。边界问题被隐藏。强制完整测试、边界和负例。
没有未执行项报告只写通过，不写没测什么。用户误以为全部通过。把未执行项单独列出来。
标准太主观缺少可检查条件。不同验证者结论不一致。改成测试、schema、规则和来源核验。
不记录成本只看架构是否能跑。无法判断是否值得继续拆。持续记录 token、耗时、失败率和回退次数。

# 12. 例子：怎么写验证任务 {#example}
<!-- tag: example -->

场景：主智能体生成了一个用户导入功能。你要让验证子智能体检查它是否可靠。差的写法是“检查导入功能是否正常”。好的写法会告诉它输入范围、检查动作、失败报告格式和禁止提前通过的规则。

## box redsoft
### 差写法
“帮我看一下导入功能有没有问题。”这会导致验证者只上传一个正常 CSV，看到成功就放行。

## box
### 好写法
“用正常 CSV、空文件、缺列、重复用户、超大文件和非法编码测试导入。记录每项结果，未执行项单独列出，任何失败都不得标记通过。”

CSV 导入验证伪代码复制

```python
verifier.run(
 artifact="CSV 导入功能",
 criteria=[
 "正常 CSV 能通过",
 "空文件被拒绝",
 "缺列被拒绝",
 "重复用户被标出",
 "超大文件有明确处理",
 "非法编码有明确错误"
 ],
 report=["通过项", "失败项", "未执行项", "复现步骤"]
)
```

检查项为什么要测没测会怎样


正常 CSV先确认主路径真的能跑。连基本功能都不通，却误以为通过。
空文件确认空输入不会静默成功。空数据被当成有效输入。
缺列确认 schema 校验存在。字段不全的文件混进系统。
重复用户确认重复冲突有明确处理。脏数据被直接写入。
超大文件确认容量边界有明确策略。性能和稳定性问题被忽略。
非法编码确认异常编码不会悄悄破坏导入。编码问题在生产里才暴露。

判断结果很简单：验证任务能不能独立拆出来，取决于它是否有清晰标准。标准越清楚，验证子智能体越像可靠检查点；标准越模糊，它越像另一个会点头的聊天者。

# 13. 落地边界：这一页先停在哪里 {#boundary}
<!-- tag: boundary -->

这一页先把四件事收住：黑盒验证、具体标准、负例测试、未执行项。再往外延伸，就会碰到评测平台、自动化编排、框架实现、多人协作和上线运维。这些都很重要，但不该和“验证子智能体是否值得拆出来”混在同一层判断里。

## box info
### 先别展开的相邻议题
- 评测平台和基准分数：后面可以接，但不是本页的判断核心。
- 框架与库名：实现层细节先放后面，别拿工具名代替边界判断。
- 自动化流水线：先把验证规则写硬，再考虑怎么编排执行。
- 多人协作流程：团队怎么分工是组织问题，不是本页的主线。
- 红队和对抗测试：属于更后面的质量强化，不影响本页的基本采用判断。

你现在应该能做什么：拿到一个验证任务时，先检查它是不是能黑盒、能写标准、能列负例、能报未执行项。只要这四件事有一件说不清，就先别急着拆成独立验证者。

### 10 short-answer

**Q:** 10. 一份文档需要满足政策规则，验证子智能体可以怎么检查？

### 11 short-answer
