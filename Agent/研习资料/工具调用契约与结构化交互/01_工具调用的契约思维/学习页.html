层 读什么 写什么 错了会怎样

术语和误区

terms
术语 直觉 容易误解
工具契约 模型、宿主程序和外部系统共用的一份说明。 只要有函数名，其他都可以靠模型猜。
输入结构 执行前必须给出的参数形状。 字符串越长越好，反正模型能读懂。
输出结构 结果要能继续被页面、程序或后续工具消费。 只要用户看得懂一句话就够。
副作用 工具会不会真的改外部状态。 “查询”和“创建”差不多，只是名字不同。
失败语义 失败后要告诉系统下一步怎么走。 所有错误都只回一句“失败了”。
评测样例 用来验证模型、宿主和工具面是否真的稳。 只测一次成功就算过关。

只读与写入

compare

工具调用最容易被忽略的差别，是“只是读一眼”和“真的改一笔”之间隔着一整套检查。前者更像问路，后者更像落笔。模型如果把两者混起来，就会出现很自然、但很危险的错误：把查询工具当成创建工具，把创建工具当成查询工具，或者把需要确认的动作当成普通读操作。

关键在后果。只读工具出错，通常先损失的是判断；写入工具出错，损失的是状态，往往还会波及别人。也正因为这样，宿主程序不能只看“这条调用像不像”，还要看“这条调用会不会真的改东西”。如果它会改东西，就必须先问三个问题：有没有权限、有没有确认、有没有足够具体的参数。

维度 只读工具 写入工具
目的 获取信息，帮助判断下一步。 改变外部状态，完成具体动作。
宿主程序关注点 参数是否足够明确，返回是否可继续使用。 是否有权限、是否需要确认、是否幂等。
用户感受 读完就知道接下来该问什么。 做完以后，外部世界真的变了。
失败时先看什么 字段是否缺、对象是否歧义、结果是否太模糊。 权限是否不足、确认是否遗漏、回填是否不完整。
最典型的坑 把查询写成“处理信息”，边界太宽。 把创建写成“帮我安排一下”，危险性被掩住。

一个很实用的判断

如果工具会写入外部状态，你就必须想清楚确认门槛、权限和回填。如果它只是只读查询，你就应该更关注参数是否具体、返回是否能直接用。两者的错误处理路径不一样，不能用同一套话糊过去。

再往前一步看，读和写还会影响日志、审计和回滚。只读工具的日志可以更多记录查询条件和命中结果；写入工具则要额外记住谁触发了动作、何时触发、写到了哪里、能不能回退。这个差别一旦写进合同，系统就会更容易排查，也更容易让后续工具接着做事。

你现在应该能做什么：看到工具说明时，先判断它到底是读、写，还是读完会触发下一步写入。

追问示范

ask

当用户只说“帮我把周二上午的同步会约上”，宿主程序要做的不是猜一个最像样的参数，而是把缺口逐个问清。这个动作很朴素，但它能把“看起来能跑”的调用，变成“真的能执行”的调用。下面这张表把常见缺口和追问方式放在一起，你可以直接照着读。

缺口 为什么不能跳过 追问示范
结束时间 没有结束时间，会议长度就只能猜。 “请告诉我结束时间，或者给我一个明确的时长。”
时区 同一个“上午”在不同地区对应不同钟点。 “请确认时区，是 Asia/Shanghai 还是别的时区？”
参与者 没有唯一 ID，重名的人很容易被拉错。 “请给我参与者的唯一 ID，或者告诉我该找哪位联系人。”
是否发送邀请 创建事件和发送通知不是同一件事。 “这次只创建事件，还是还要同步发邀请？”
追问分支示意
if (!intent.end_time) {
  return askUser("请补结束时间。");
}
if (!intent.timezone) {
  return askUser("请确认时区。");
}
if (!intent.attendee_ids || intent.attendee_ids.length === 0) {
  return askUser("请提供参与者的唯一 ID。");
}
if (intent.send_invites === undefined) {
  return askUser("这次只建事件，还是还要发送邀请？");
}

如果直接跳过会怎样

系统会把默认值偷偷补进去，看起来流程更短，实际上把错误留到了更后面。到了真正执行时，用户会看到错时间、错对象或错通知，这时候再改就已经晚了。

你现在应该能做什么：看到一个没填完的请求时，列出你要追问的字段，并把它们放到正确的顺序里。

案例

example

场景：用户说“帮我把周二上午的同步会约上”。这句话里真正可执行的信息只有会议意图和大致时间，参与者、时区、结束时间和是否发送邀请都还没明确。

字段 为什么要写清 含糊时会发生什么
title 让人和系统知道这是什么会议。 日历里多出一条“会议”但没人知道用途。
start_time / end_time 决定具体时间窗，避免“上午”这种模糊说法。 宿主程序可能补错默认值，或者把时长算错。
timezone 避免同一个时间被解释成不同城市的本地时间。 跨时区时最容易约错时间。
attendee_ids 邀请对象要落到唯一 ID，而不是昵称。 重名的人被一起拉进来，或者邀请发错人。
send_invites 决定这次是只创建事件，还是还要通知别人。 模型以为只是“建一下”，系统却把邀请发出去了。

先追什么

如果用户只给了“周二上午”，宿主程序还要追问结束时间、时区和参与者。如果是只读查询，就不要把这个请求硬塞进创建工具。

你现在应该能做什么：把一句模糊请求拆成几个必填字段，再判断哪些字段必须追问。

结果回流

loop

真正可靠的工具调用会继续回流结果，直到后续流程知道下一步该做什么。这个回流动作有两个目的：第一，让下一步知道这次发生了什么；第二，让失败时能继续分流，而不是把所有错误都塞成一句“没成功”。

把这件事讲得再具体一点：结果回流其实是在替后续步骤省猜测。页面、工作流、下一个工具和人工审阅都不该去猜“这次到底有没有写进去”“还能不能再试一次”“下一步是不是该追问用户”。如果结果里把状态、对象 ID、重试信息和短说明都留好，后续的路就清楚了；没有这些字段，整个系统就会一直围着一句模糊的“完成”打转。

回流字段 为什么要有 没有它会怎样
status 告诉后续步骤这次是创建成功、需要用户动作，还是失败。 程序只能猜这次算不算成功。
event_id / object_id 让后续步骤知道要接哪一条记录。 后面想继续修改、取消或查询时找不到对象。
retryable 告诉系统要不要再试一次。 失败后可能被盲目重试，或者本来能重试却被直接丢掉。
message 给人看的简短说明，帮助用户理解发生了什么。 用户只看到一个状态码，不知道下一步该做什么。
回流结果示例
{
  "status": "needs_user_action",
  "event_id": "evt_20260610_0830",
  "retryable": false,
  "message": "已创建草稿会议，但还缺少参与者确认"
}

一个常见误区

很多人会把“回填给模型看的内容”和“给用户看的文案”混成一句话。更稳的做法，是让结构里有可继续处理的字段，再给一条简短说明。这样宿主程序、页面和后续工具都能各取所需。

你现在应该能做什么：拿一个工具返回值，先检查有没有状态、对象 ID、是否可重试和一条能给人看的说明。

执行链工作台

trace

把“帮我约周二上午的同步会”走一遍，最容易看清工具契约到底在保护什么。这里不是背答案，而是按步骤追踪：先看请求里有什么，哪些字段缺口会让宿主程序停下，什么时候必须追问，什么时候才算真的写入。

步骤 读取 写入 典型症状
1. 读请求 “周二上午”“同步会”这类自然语言 一个待补全的意图 只有时间片段，没有结束时间和时区。
2. 补字段 标题、开始时间、结束时间、参与者、时区 一个足够窄的输入对象 字段没填齐，系统只能猜默认值。
3. 查门槛 是否需要确认、是否有权限、是否会通知别人 放行、拒绝或追问 写入动作被当成普通查询。
4. 真执行 已验证的参数 日历里的真实事件 时间窗错了，或者邀请发错了人。
5. 回填 event_id、状态、可重试性、用户说明 下一步还能继续消费的结果 只剩一句“已完成”，后续流程接不上。

function handleCreateEventRequest(userText) {

  if (!intent.title) {
    return askUser("请先给会议一个标题。");
  }
  if (!intent.start_time || !intent.end_time) {
    return askUser("请补全开始时间和结束时间。");
  }

    return askUser("请确认时区。");
  }
  if (intent.requires_confirmation) {

  }

    return reject(check.reason);
  }

  const result = createCalendarEvent(intent);
  return {

    event_id: result.id,
    retryable: false,
    message: "会议已创建，可以继续发送邀请或安排后续步骤"
  };

常见错误

练习

作答目标

Q12 讲一遍从请求到回填的流程

请按“请求 → 选择 → 校验 → 执行 → 回填”写，不用追求文采，但要把出错点说出来。

参考答案 / 评分要点

合格答案要把流程讲完整：用户发来请求，模型提出调用候选，宿主程序校验参数和权限，外部系统真正执行，最后把状态和 ID 回填给后续步骤。最好再补一句：最容易出错的地方通常是“选择”和“校验”，因为这两步最容易让相近工具、缺字段或越权动作混过去。

Q14 哪个检查顺序更稳？

把下面四步按更稳的顺序排出来：执行、校验、回填、选择工具。

参考答案 / 评分要点

较稳的顺序是：选择工具 → 校验 → 执行 → 回填。只要把“执行”放到“校验”前面，后面就很难再补救；如果把“回填”放到“执行”前面，那就更不合理。答案如果写成别的顺序，重点看有没有把选择、校验、执行和回填的职责区分开。

Q15 给这个错误写一个归类和修法

错误：工具说明里写的是“日历安排”，但系统里还有“查询日程”“创建日程”“取消日程”三个工具。模型经常选错。

参考答案 / 评分要点

这个错误主要属于“工具名和描述太泛”，也可能伴随“边界没有写清”。修法不是只调温度，而是把名称改成动作 + 对象，把描述里写清它负责什么、不负责什么，并在下一单元继续检查工具选择是否还会混淆。

Q16 Feynman：为什么回填不能只写一句“完成”

把答案写给“后续还要接着做别的工具”的同事看。

参考答案 / 评分要点

好答案要说明：后续步骤需要事件 ID、状态、是否可重试、是否要用户再确认，以及这次执行有没有真正改到外部状态。只写“完成”会让下游不知道是不是成功、能不能重试、还能不能继续追踪同一件事。

Q19 为什么回流不能只剩一句“完成”？

参考答案 / 评分要点

合格答案要说清楚三件事：后续步骤需要对象 ID 才能继续操作；排障需要知道这次是成功、需要用户动作还是失败；用户也需要一条简短说明来理解发生了什么。只写“完成”会让系统停在半空，既不能继续，也不好复盘。

Q20 给一个回流结果补齐四个字段

下面这个结果太短了：{"status":"created","message":"完成"}。请补出你认为必须补上的两个字段，并说明它们为什么重要。

参考答案 / 评分要点

常见应补字段包括 event_id 和 retryable，也可以补 needs_user_action 或更具体的状态说明。理由要落到后续动作：没有 ID，后续找不到对象；没有可重试性，失败后不知道该不该再试；没有更具体的状态，系统就只能猜。
