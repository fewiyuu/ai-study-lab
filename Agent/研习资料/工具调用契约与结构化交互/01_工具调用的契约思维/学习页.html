01｜工具调用的契约思维

这一页先把工具调用当成一份合同。你会把一个自然语言动作拆成工具名、用途边界、输入字段、输出字段、权限提示和失败语义，再看宿主程序怎么接住它。下一单元会继续讲工具描述、选择与分发。

课程位置

route

前一步你已经知道工具不是按钮，而是要被宿主程序接管的动作。

本页先把“合同”讲清：谁能调、要给什么、会返回什么、出错怎么回。

概念地图

map
层读什么写什么错了会怎样
模型用户意图、上下文、可用工具名一个调用候选会选错工具，或把查询当成写入。
宿主程序工具名、参数、权限、确认状态放行、拒绝或追问缺字段、越权或副作用会漏过去。
外部系统已验证的参数真实状态变更写错日程、写错对象、写错时间窗。
回填执行结果、事件 ID、错误类别下一步可继续用的结果只剩一句“已完成”，后续步骤接不住。

术语和误区

terms
术语直觉容易误解
工具契约模型、宿主程序和外部系统共用的一份说明。只要有函数名，其他都可以靠模型猜。
输入结构执行前必须给出的参数形状。字符串越长越好，反正模型能读懂。
输出结构结果要能继续被页面、程序或后续工具消费。只要用户看得懂一句话就够。
副作用工具会不会真的改外部状态。“查询”和“创建”差不多，只是名字不同。
失败语义失败后要告诉系统下一步怎么走。所有错误都只回一句“失败了”。
评测样例用来验证模型、宿主和工具面是否真的稳。只测一次成功就算过关。

输入 schema字段、类型、必填、枚举、额外字段策略校验目标空对象通过、字段漂移、错误值被塞进来。

你现在应该能做什么：拿到一个工具时，先把输入、风险和结果分开写，再看哪些字段不能让模型自己猜。

只读与写入

compare

工具调用最容易被忽略的差别，是“只是读一眼”和“真的改一笔”之间隔着一整套检查。前者更像问路，后者更像落笔。模型如果把两者混起来，就会出现很自然、但很危险的错误：把查询工具当成创建工具，把创建工具当成查询工具，或者把需要确认的动作当成普通读操作。

维度只读工具写入工具
目的获取信息，帮助判断下一步。改变外部状态，完成具体动作。
宿主程序关注点参数是否足够明确，返回是否可继续使用。是否有权限、是否需要确认、是否幂等。
用户感受读完就知道接下来该问什么。做完以后，外部世界真的变了。
失败时先看什么字段是否缺、对象是否歧义、结果是否太模糊。权限是否不足、确认是否遗漏、回填是否不完整。
最典型的坑把查询写成“处理信息”，边界太宽。把创建写成“帮我安排一下”，危险性被掩住。

一个很实用的判断

如果工具会写入外部状态，你就必须想清楚确认门槛、权限和回填。如果它只是只读查询，你就应该更关注参数是否具体、返回是否能直接用。两者的错误处理路径不一样，不能用同一套话糊过去。

再往前一步看，读和写还会影响日志、审计和回滚。只读工具的日志可以更多记录查询条件和命中结果；写入工具则要额外记住谁触发了动作、何时触发、写到了哪里、能不能回退。这个差别一旦写进合同，系统就会更容易排查，也更容易让后续工具接着做事。

你现在应该能做什么：看到工具说明时，先判断它到底是读、写，还是读完会触发下一步写入。

追问示范

ask

当用户只说“帮我把周二上午的同步会约上”，宿主程序要做的把缺口逐个问清。这个动作很朴素，但它能把“看起来能跑”的调用，变成“真的能执行”的调用。下面这张表把常见缺口和追问方式放在一起，你可以直接照着读。

代码
if (!intent.end_time) {
 return askUser("请补结束时间。");
}
if (!intent.timezone) {
 return askUser("请确认时区。");
}
if (!intent.attendee_ids || intent.attendee_ids.length === 0) {
 return askUser("请提供参与者的唯一 ID。");
}
if (intent.send_invites === undefined) {
 return askUser("这次只建事件，还是还要发送邀请？");
}

案例

example

场景：用户说“帮我把周二上午的同步会约上”。这句话里真正可执行的信息只有会议意图和大致时间，参与者、时区、结束时间和是否发送邀请都还没明确。

先追什么

如果用户只给了“周二上午”，宿主程序还要追问结束时间、时区和参与者。如果是只读查询，就不要把这个请求硬塞进创建工具。

你现在应该能做什么：把一句模糊请求拆成几个必填字段，再判断哪些字段必须追问。

诊断

debug
症状先查哪里典型修法
空对象也能通过输入 schema补 required，收紧字段类型。
相近工具老是选错工具名和描述把用途和不用途写清楚。
执行前没追问确认门槛和必填字段把缺口变成明确的追问点。
结果回填后接不上下一步输出结构加状态、ID、是否可重试。

结果回流

loop

真正可靠的工具调用会继续回流结果，直到后续流程知道下一步该做什么。这个回流动作有两个目的：第一，让下一步知道这次发生了什么；第二，让失败时能继续分流，而不是把所有错误都塞成一句“没成功”。

回流字段为什么要有没有它会怎样
status告诉后续步骤这次是创建成功、需要用户动作，还是失败。程序只能猜这次算不算成功。
event_id / object_id让后续步骤知道要接哪一条记录。后面想继续修改、取消或查询时找不到对象。
retryable告诉系统要不要再试一次。失败后可能被盲目重试，或者本来能重试却被直接丢掉。
message给人看的简短说明，帮助用户理解发生了什么。用户只看到一个状态码，不知道下一步该做什么。
代码
{
 "status": "needs_user_action",
 "event_id": "evt_20260610_0830",
 "retryable": false,
 "message": "已创建草稿会议，但还缺少参与者确认"
}

 }

常见错误

mistake
错误症状修法
工具名写得太泛模型把查询、创建和修改混在一起。名称里带上动作和对象，描述里写不用途。
输入字段太松空对象能通过，或者“上午”这种词直接进来。把必填、枚举和格式写清，必要时追问。
忽略权限和确认工具一被选中就直接写入外部状态。把 requires_confirmation 和权限检查放到宿主程序。
输出只有一句话用户看懂了，但下一步程序接不住。加状态、ID、是否可重试和下一步说明。
只测成功路径一到缺字段、歧义或越权就崩。把缺字段、歧义、权限和副作用样例一起测。

Q: Q19 为什么回流不能只剩一句“完成”？

参考答案 / 评分要点

。合格答案要说清楚三件事：后续步骤需要对象 ID 才能继续操作；排障需要知道这次是成功、需要用户动作还是失败；用户也需要一条简短说明来理解发生了什么。只写“完成”会让系统停在半空，既不能继续，也不好复盘。

Q: Q20 给一个回流结果补齐四个字段

参考答案 / 评分要点

。常见应补字段包括 event_id 和 retryable ，也可以补 needs_user_action 或更具体的状态说明。理由要落到后续动作：没有 ID，后续找不到对象；没有可重试性，失败后不知道该不该再试；没有更具体的状态，系统就只能猜。
