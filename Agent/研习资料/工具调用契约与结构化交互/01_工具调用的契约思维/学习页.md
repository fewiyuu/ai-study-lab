# 只读与写入 {#compare}
<!-- tag: compare -->

工具调用最容易被忽略的差别，是“只是读一眼”和“真的改一笔”之间隔着一整套检查。前者更像问路，后者更像落笔。模型如果把两者混起来，就会出现很自然、但很危险的错误：把查询工具当成创建工具，把创建工具当成查询工具，或者把需要确认的动作当成普通读操作。

关键在后果。只读工具出错，通常先损失的是判断；写入工具出错，损失的是状态，往往还会波及别人。也正因为这样，宿主程序不能只看“这条调用像不像”，还要看“这条调用会不会真的改东西”。如果它会改东西，就必须先问三个问题：有没有权限、有没有确认、有没有足够具体的参数。

维度
只读工具
写入工具


目的
获取信息，帮助判断下一步。
改变外部状态，完成具体动作。


宿主程序关注点
参数是否足够明确，返回是否可继续使用。
是否有权限、是否需要确认、是否幂等。


用户感受
读完就知道接下来该问什么。
做完以后，外部世界真的变了。


失败时先看什么
字段是否缺、对象是否歧义、结果是否太模糊。
权限是否不足、确认是否遗漏、回填是否不完整。


最典型的坑
把查询写成“处理信息”，边界太宽。
把创建写成“帮我安排一下”，危险性被掩住。

## box info
### 一个很实用的判断
如果工具会写入外部状态，你就必须想清楚确认门槛、权限和回填。如果它只是只读查询，你就应该更关注参数是否具体、返回是否能直接用。两者的错误处理路径不一样，不能用同一套话糊过去。

再往前一步看，读和写还会影响日志、审计和回滚。只读工具的日志可以更多记录查询条件和命中结果；写入工具则要额外记住谁触发了动作、何时触发、写到了哪里、能不能回退。这个差别一旦写进合同，系统就会更容易排查，也更容易让后续工具接着做事。

**你现在应该能做什么：**看到工具说明时，先判断它到底是读、写，还是读完会触发下一步写入。

# 追问示范 {#follow-up}
<!-- tag: ask -->

当用户只说“帮我把周二上午的同步会约上”，宿主程序要做的把缺口逐个问清。这个动作很朴素，但它能把“看起来能跑”的调用，变成“真的能执行”的调用。下面这张表把常见缺口和追问方式放在一起，你可以直接照着读。

缺口
为什么不能跳过
追问示范


结束时间
没有结束时间，会议长度就只能猜。
“请告诉我结束时间，或者给我一个明确的时长。”


时区
同一个“上午”在不同地区对应不同钟点。
“请确认时区，是 Asia/Shanghai 还是别的时区？”


参与者
没有唯一 ID，重名的人很容易被拉错。
“请给我参与者的唯一 ID，或者告诉我该找哪位联系人。”


是否发送邀请
创建事件和发送通知不是同一件事。
“这次只创建事件，还是还要同步发邀请？”

追问分支示意复制

```js
if (!intent.end_time) {
 return askUser("请补结束时间。");
}
if (!intent.timezone) {
 return askUser("请确认时区。");
}
if (!intent.attendee_ids || intent.attendee_ids.length === 0) {
 return askUser("请提供参与者的唯一 ID。");
}
if (intent.send_invites === undefined) {
 return askUser("这次只建事件，还是还要发送邀请？");
}
```

## box
### 如果直接跳过会怎样
系统会把默认值偷偷补进去，看起来流程更短，实际上把错误留到了更后面。到了真正执行时，用户会看到错时间、错对象或错通知，这时候再改就已经晚了。

**你现在应该能做什么：**看到一个没填完的请求时，列出你要追问的字段，并把它们放到正确的顺序里。

# 结果回流 {#result-loop}
<!-- tag: loop -->

真正可靠的工具调用会继续回流结果，直到后续流程知道下一步该做什么。这个回流动作有两个目的：第一，让下一步知道这次发生了什么；第二，让失败时能继续分流，而不是把所有错误都塞成一句“没成功”。

把这件事讲得再具体一点：结果回流其实是在替后续步骤省猜测。页面、工作流、下一个工具和人工审阅都不该去猜“这次到底有没有写进去”“还能不能再试一次”“下一步是不是该追问用户”。如果结果里把状态、对象 ID、重试信息和短说明都留好，后续的路就清楚了；没有这些字段，整个系统就会一直围着一句模糊的“完成”打转。

回流字段
为什么要有
没有它会怎样


status
告诉后续步骤这次是创建成功、需要用户动作，还是失败。
程序只能猜这次算不算成功。


event_id / object_id
让后续步骤知道要接哪一条记录。
后面想继续修改、取消或查询时找不到对象。


retryable
告诉系统要不要再试一次。
失败后可能被盲目重试，或者本来能重试却被直接丢掉。


message
给人看的简短说明，帮助用户理解发生了什么。
用户只看到一个状态码，不知道下一步该做什么。

回流结果示例复制

```json
{
 "status": "needs_user_action",
 "event_id": "evt_20260610_0830",
 "retryable": false,
 "message": "已创建草稿会议，但还缺少参与者确认"
}
```

## box warn
### 一个常见误区
很多人会把“回填给模型看的内容”和“给用户看的文案”混成一句话。更稳的做法，是让结构里有可继续处理的字段，再给一条简短说明。这样宿主程序、页面和后续工具都能各取所需。

**你现在应该能做什么：**拿一个工具返回值，先检查有没有状态、对象 ID、是否可重试和一条能给人看的说明。

### 1 choice

**Q:** Q1 工具契约的第一职责是什么？

- [ ] . 让模型看起来更像会用工具。
- [ ] . 让模型、宿主程序和外部系统对动作边界达成一致。
- [ ] . 让返回值尽量长一点，方便展示。
- [ ] . 让所有工具都能被同一个名字调用。

Answer: . 让模型、宿主程序和外部系统对动作边界达成一致。

### 2 true/false

**Q:** Q2 模型选中工具以后，调用已经执行了吗？

- [ ] 对
- [ ] 错

Answer: false
Explain: 选中工具只是调用候选；真正执行还要经过宿主程序的校验、权限和确认。

### 4 synthesis

**Q:** Q4 用 3-4 句解释模型、宿主程序和外部系统分别负责什么

Explain: 合格答案至少要说清三层：模型负责提出调用候选；宿主程序负责校验参数、权限和确认；外部系统负责真正执行并返回结果。好答案还会补一句：如果把任一层混掉，就会出现选错工具、越权执行或结果接不住。

### 7 rewrite

**Q:** Q7 改写一个模糊工具描述

Explain: 合格答案至少要写出动作、对象和边界，例如“search_policy_docs：搜索公开政策文档，只用于查询规则、条款和流程，不用于个人订单或执行退款”。如果把“不能做什么”也写出来，说明你已经在帮模型分边界。

### 9 choice

**Q:** Q9 缺少结束时间时，先做什么？

- [ ] . 直接执行，默认给一个 1 小时时长。
- [ ] . 先改工具名，让它看起来更专业。
- [ ] . 先追问结束时间，不要默认补值。
- [ ] . 只把错误消息写长一点。

Answer: . 先追问结束时间，不要默认补值。

### 10 synthesis

**Q:** Q10 为查单工具写一个失败返回

Explain: 答案不要求固定格式，但必须说清楚错误是“缺少订单号”“权限不足”还是“外部服务失败”，并且要告诉系统下一步是追问、拒绝还是重试。只写“失败了”不够，因为宿主程序不知道该把球传给谁。

### 11 choice

**Q:** Q11 工具请求进来后，第一步通常先查什么？

- [ ] . 工具名是否在白名单里。
- [ ] . 页面配色是否一致。
- [ ] . 最终回复的文风是否自然。
- [ ] . 是否已经有导出记录。

Answer: . 工具名是否在白名单里。

### 12 transfer

**Q:** Q12 讲一遍从请求到回填的流程

Explain: 合格答案要把流程讲完整：用户发来请求，模型提出调用候选，宿主程序校验参数和权限，外部系统真正执行，最后把状态和 ID 回填给后续步骤。最好再补一句：最容易出错的地方通常是“选择”和“校验”，因为这两步最容易让相近工具、缺字段或越权动作混过去。

### 13 choice

**Q:** Q13 哪个症状最像工具名太泛？

- [ ] . 模型在查询和创建之间来回跳。
- [ ] . 返回值里没有 event_id。
- [ ] . 用户没给结束时间。
- [ ] . 结果回填太慢。


### 14 ordering

**Q:** Q14 哪个检查顺序更稳？

Explain: 较稳的顺序是：选择工具 → 校验 → 执行 → 回填。只要把“执行”放到“校验”前面，后面就很难再补救；如果把“回填”放到“执行”前面，那就更不合理。答案如果写成别的顺序，重点看有没有把选择、校验、执行和回填的职责区分开。

### 15 debug

**Q:** Q15 给这个错误写一个归类和修法

Explain: 这个错误主要属于“工具名和描述太泛”，也可能伴随“边界没有写清”。修法把名称改成动作 + 对象，把描述里写清它负责什么、不负责什么，并在下一单元继续检查工具选择是否还会混淆。

### 16 synthesis

**Q:** Q16 Feynman：为什么回填不能只写一句“完成”

Explain: 好答案要说明：后续步骤需要事件 ID、状态、是否可重试、是否要用户再确认，以及这次执行有没有真正改到外部状态。只写“完成”会让下游不知道是不是成功、能不能重试、还能不能继续追踪同一件事。

### 18 fill

**Q:** Q18 填空：告诉系统失败后还值不值得再试，通常叫哪个字段？


### 19 synthesis

**Q:** Q19 为什么回流不能只剩一句“完成”？

Explain: 合格答案要说清楚三件事：后续步骤需要对象 ID 才能继续操作；排障需要知道这次是成功、需要用户动作还是失败；用户也需要一条简短说明来理解发生了什么。只写“完成”会让系统停在半空，既不能继续，也不好复盘。

### 20 code-reading

**Q:** Q20 给一个回流结果补齐四个字段

Explain: 常见应补字段包括 event_id 和 retryable ，也可以补 needs_user_action 或更具体的状态说明。理由要落到后续动作：没有 ID，后续找不到对象；没有可重试性，失败后不知道该不该再试；没有更具体的状态，系统就只能猜。
