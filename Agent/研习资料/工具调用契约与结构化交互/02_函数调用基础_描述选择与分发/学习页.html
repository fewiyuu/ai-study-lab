第 02 单元

02｜函数调用基础：描述、选择与分发

这一页把函数工具调用拆成一个可检查的闭环：工具怎样被命名和描述，模型怎样选择，参数怎样被收紧，宿主程序怎样校验并分发，结果怎样回填给后续推理。这里先不展开 JSON Schema 的全部细节、MCP 协议和错误回退；重点是你能沿着一次调用判断“该不该调、调哪个、能不能执行”。

课程地图上一单元下一单元

读完本页应该能做什么

  • 能把一个工具声明拆成名称、描述、参数结构和选择边界。
  • 能沿着一次用户请求追踪模型选择、参数生成、宿主校验、真实执行和结果回填。
  • 能诊断选错工具、漏填参数、越权执行和回填不可消费这几类常见失败。

学习目标

goals

看懂声明

看到一个工具声明时，能分出工具名、描述、参数 schema、输出语义和副作用等级。工具名主要服务路由；描述服务选择；schema 服务参数边界；输出服务后续推理。

追踪选择

能把一次用户请求拆成候选工具、选择理由、参数来源和缺失信息。遇到相近工具时，先看动作和对象，再看描述里的适用范围。

守住分发

能说明宿主程序为什么还要查白名单、schema、权限、确认和幂等。模型可以提出工具调用，但执行权要留在宿主程序手里。

你现在应该能做什么：拿到一个函数工具面，先判断它会不会被稳定选中，再判断它能不能被安全执行。

最小闭环

trace
请求 意图还没有落到动作

用户给出的通常是任务，不是接口调用。这里要先判断任务属于查询、写入还是需要追问。

选择 模型只给出调用候选

候选包含工具名和参数草稿。它说明“模型认为应该这样做”，还没有改动任何外部系统。

参数 schema 把文本收成对象

参数对象应该能被程序读取。缺少必填字段、类型错或值域过宽时，宿主程序要追问或拒绝。

分发 执行权在宿主程序

白名单、权限、确认、幂等和审计都在这里发生。模型不能绕过这层直接访问外部系统。

回填 结果进入下一轮推理

返回值要包含状态、对象 ID、失败类别和下一步建议。只写“完成”会让后续步骤接不住。

阶段 谁在写 谁在读 失败信号 处理动作
工具声明 开发者 模型和宿主程序 工具经常混选或从不被选中 改名、补边界、拆工具
调用候选 模型 宿主程序 未知工具名、参数缺失、越权意图 拒绝、追问、降级到普通回复
真实执行 宿主程序和外部系统 模型、用户、审计日志 外部错误、权限不足、重复写入 返回结构化错误并保留可复查信息
结果回填 宿主程序 模型后续推理 没有 ID、状态或下一步 补齐状态、句柄、可重试性和人类说明

你现在应该能做什么：把一次工具调用画成五步，并指出每一步的输入、输出和拦截点。

术语边界

vocab
术语 直觉 工程含义 常见混淆
tool / function 模型可以请求使用的能力 一段声明加一段宿主程序里的真实处理逻辑 把声明当成执行本身
name 路由键 宿主程序用它找到处理函数，模型也会从名字里读动作信号 为了好看起一个宽泛名字
description 选择说明 告诉模型适用场景、排除场景、风险和需要追问的信息 只写“处理相关问题”
parameters schema 可填写表单 把自然语言里的信息收成字段、类型、必填项和值域 把内部实现细节塞给模型
tool choice 选择策略 系统允许模型自动选择、禁止使用、强制使用或指定某个工具 以为工具列表存在就一定会被用对

读工具声明的顺序

先看名字能不能表达动作和对象，再看描述是否说明边界，然后看参数字段是否足以支持执行，最后看输出是否能被后续系统消费。这个顺序能快速发现“看起来能用，但实际会混选”的工具。

最容易混掉的边界

模型生成的是调用请求，不是执行结果。真实执行需要宿主程序把工具名映射到本地函数，校验参数，检查权限，必要时向用户确认，然后才调用外部系统。

描述与选择

selection

描述要解决什么

描述不是给用户看的产品文案，而是给模型的选择说明。它要把动作、对象、适用场景、排除场景和风险写清楚。越是相近工具越需要边界句，例如“只用于查询，不用于创建或退款”。

如果工具描述都写成“处理订单相关问题”，模型只能靠上下文猜。猜对时你会觉得系统聪明，猜错时问题很难复盘，因为声明里没有可检查的边界。

选择策略怎么参与

auto 让模型自己判断是否调用工具；none 禁止调用；required 要求至少调用一个；指定某个工具则把选择空间收窄到一个目标。策略不是质量保证，它只改变模型的选择空间。

工程上常见做法是：普通问答用自动选择，高风险写入先让模型给计划或追问，确认后再指定写入工具。这样可以减少“模型把查询句误当成执行句”的风险。

工具 弱描述 更稳的描述 为什么更稳
get_order_status 查询订单信息 查询单个订单的当前物流和支付状态。只读，不创建退款，不修改订单。 动作、对象和排除场景都清楚。
search_policy_docs 搜索相关内容 搜索公开政策文档，用于回答规则、条款和流程问题。不用于查询个人订单。 把知识库搜索和业务对象查询拆开。
create_refund_request 处理退款 为指定订单创建退款申请。会写入外部系统，必须在用户确认金额和原因后使用。 明确副作用和确认条件。

这个小实验只做切片提示：真实系统会结合上下文和工具描述判断。你要留意动词、对象、状态词和是否出现确认信号。

你现在应该能做什么：看到两个工具描述时，指出它们是否会抢同一个用户意图，并写出一条排除边界。

参数结构

schema

把 schema 当成模型要填写的表单

参数 schema 的任务是让模型把自然语言里的信息落到字段上。它不能替代业务权限，也不能保证事实正确；它先保证“对象形状能被程序读”。字段名要贴近业务对象，描述要解释取值含义，值域能收窄就收窄。

订单状态查询工具声明
{
  "name": "get_order_status",
  "description": "查询单个订单的当前物流和支付状态。只读，不创建退款，不修改订单。",
  "parameters": {
    "type": "object",
    "additionalProperties": false,
    "required": ["order_id"],
    "properties": {
      "order_id": {
        "type": "string",
        "pattern": "^ORD-[0-9]{8}$",
        "description": "订单号，例如 ORD-20260621"
      }
    }
  }
}

required

告诉宿主程序哪些字段缺了就不能继续。缺订单号时追问，比默认猜一个订单安全得多。

pattern / enum

把开放字符串收进可检查范围。订单号适合格式约束，退款原因、渠道、状态这类字段常适合枚举。

additionalProperties

让参数对象保持闭合，避免模型把解释、猜测和调试信息混进要执行的参数里。

检查层 检查内容 适合放在哪里 失败后动作
结构校验 字段、类型、必填、格式、枚举 参数 schema 和解析器 追问或要求模型重填参数
业务校验 订单是否属于当前用户、状态是否允许退款 宿主程序和业务服务 拒绝、解释原因或转人工
执行校验 是否重复提交、外部系统是否可用 分发器、幂等层、外部 API 返回可重试错误或保留任务状态

关键提醒：schema 让参数更可读、更可查，但高风险动作还要做权限、确认、归属和幂等检查。

分发账本

dispatcher

分发器的核心职责

分发器站在模型和真实系统之间。它不只是“按 name 调函数”，还要确认工具名在白名单里，参数能通过 schema，当前用户有权限，写入动作已经确认，重复请求不会造成重复副作用。

为什么不能直接执行

模型输出可以被提示词影响，也可能因为相近工具而选错。把任意 name 和 arguments 直接交给系统，相当于把路由、权限和审计全交给概率输出。安全的做法是把模型请求当作候选，而不是命令。

宿主程序分发伪代码
async function dispatchToolCall(call, user, registry) {
  const tool = registry[call.name];
  if (!tool) return reject("unknown_tool");

  const parsed = validateAgainstSchema(tool.parameters, call.arguments);
  if (!parsed.ok) return askForMissingFields(parsed.errors);

  const permission = await canUseTool(user, tool, parsed.value);
  if (!permission.ok) return reject("permission_denied");

  if (tool.sideEffect && !call.userConfirmed) {
    return askForConfirmation(tool.name, parsed.value);
  }

  const result = await tool.handler(parsed.value, { user });
  return normalizeToolResult(result);
}
顺序 检查 通过 失败
1 工具名是否在 registry 里 找到处理函数 拒绝未知工具，记录异常
2 参数是否符合 schema 得到规范化参数对象 追问缺失字段或要求重填
3 用户是否有权限 继续业务校验 拒绝并说明原因
4 写入动作是否已确认 执行外部调用 先请求用户确认
5 结果是否可回填 返回状态、ID、可重试性 补结构化错误，避免只返回异常字符串

你现在应该能做什么：写出一个最小分发器检查清单，并指出每个失败分支应该追问、拒绝还是执行降级。

订单工具组案例

case

get_order_status

只读工具。用于单个订单的物流、支付和售后状态查询。输入是订单号，输出要包含订单状态、物流状态和是否可发起退款。

search_policy_docs

知识库搜索工具。用于查政策、条款、流程和规则说明。它不应该接收个人订单号，也不应该改动订单状态。

create_refund_request

写入工具。用于创建退款申请。必须有订单号、退款原因、金额和用户确认；执行后要返回退款申请 ID。

用户请求 首选动作 应选工具 执行前检查 返回给模型的关键信息
“查一下 ORD-20260621 到哪了” 只读查询 get_order_status 订单号格式、订单归属 订单状态、物流节点、更新时间
“七天无理由退货规则是什么” 政策检索 search_policy_docs 查询词是否指向政策文档 命中文档、条款摘要、来源
“这个订单帮我直接退了” 高风险写入 先追问或确认，再用 create_refund_request 订单号、金额、原因、用户确认、权限、订单状态 退款申请 ID、状态、预计处理时间
“我东西没收到，能退吗” 先澄清 可能先查订单，再查政策 缺订单号；缺用户想查状态还是发起退款 追问订单号或意图，不要直接创建退款
退款申请工具声明草案
{
  "name": "create_refund_request",
  "description": "为指定订单创建退款申请。会写入外部系统；仅在用户确认退款金额和原因后使用。",
  "parameters": {
    "type": "object",
    "additionalProperties": false,
    "required": ["order_id", "reason", "amount_cents", "confirmed_by_user"],
    "properties": {
      "order_id": { "type": "string", "pattern": "^ORD-[0-9]{8}$" },
      "reason": {
        "type": "string",
        "enum": ["not_received", "damaged", "wrong_item", "changed_mind", "other"]
      },
      "amount_cents": { "type": "integer", "minimum": 1 },
      "confirmed_by_user": { "type": "boolean" }
    }
  }
}

案例的重点：同一个用户主题可以落到不同工具。选择稳定性来自动作边界、对象边界、参数边界和确认边界。

失败诊断

debug 
症状：
用户问“七天无理由规则”，模型选择 get_order_status。

定位：
订单查询工具和政策搜索工具都写成“处理订单问题”。

修法：
把一个描述收窄到“单个订单状态”，另一个收窄到“公开政策文档”。
症状 优先检查 常见根因 修法
经常选错相近工具 name 和 description 动作或对象边界太宽 拆工具、补排除场景、加入负例测试
调用参数缺字段 required 和字段描述 schema 太松或用户信息不足 补 required；缺信息时追问
写入动作没确认 side effect 标记和分发器 把选择当执行 写入前要求确认并记录确认文本
返回后无法继续 tool result 只给人类文本，没有机器字段 返回状态、对象 ID、错误类别和下一步

你现在应该能做什么：看到一个工具调用 bug，先把症状归到描述、schema、权限、确认或回填，而不是直接调 prompt。

选择测试矩阵

test

为什么要单独做这个矩阵

工具面重构完以后，不能只测“理想请求能不能调对”。你还要测边界请求、缺信息请求、高风险写入请求和无关请求。这样才知道问题落在模型选择、参数 schema，还是宿主程序分发。矩阵不追求题量很大，先覆盖四类分支：该调用、该追问、该确认、该拒绝。

测试输入 期望工具 期望分发动作 要抓的失败
查 ORD-20260621 到哪了 get_order_status 校验订单号和归属后执行只读查询 把订单号送进政策搜索；返回没有物流状态
七天无理由退货规则是什么 search_policy_docs 执行知识库搜索并返回来源 错误调用退款工具；把政策问题当个人订单查询
帮我退了这个订单 暂不直接执行 追问订单号、金额、原因，并要求明确确认 未确认就写入；默认猜金额或原因
给我讲个笑话 不使用工具 普通回答或按产品策略拒绝 为了“用工具”而强行调用无关工具

怎么读测试结果

如果理想请求能过、边界请求错了，先改描述和负例；如果模型选对了但参数缺字段还被执行，查 schema 和分发器；如果写入请求没有确认就通过，问题在宿主程序的执行门槛；如果无关请求也调用工具，检查工具选择策略和拒绝样例。

最小通过线

这四类输入都要有稳定预期：只读查询能执行，政策问题走文档，写入动作先确认，无关请求不调用工具。只要其中一类没有稳定预期，这个工具面就还不能算真正可维护。

你现在应该能做什么：为自己的工具组写一张小型测试矩阵，覆盖正例、边界例、写入确认例和无关拒绝例。

桥接到后续单元

bridge

接到结构化输出

本页只把参数 schema 当成“可填写表单”来用。下一单元会更细地看 JSON Schema：对象如何闭合，字段如何必填，枚举如何收窄，复杂数组和嵌套对象怎样被消费端稳定读取。

如果你在本页已经能指出“缺字段不能执行”“额外字段会污染下游”，下一页就会变成自然延伸：把这些直觉写成更严密的结构约束。

接到 MCP、错误回退和评测

MCP 会把工具发现、调用和结果语义做成协议边界；错误回退会处理调用失败、超时和重试；BFCL 会检查模型在工具选择、参数生成、多轮调用和无关工具拒绝上的表现。它们都需要今天这条链：描述、选择、分发、回填。

换句话说，今天学的是工具调用的“最小可复盘单元”。后面的协议、错误和评测，只是在更大范围里反复检查这个单元是否可靠。

后续主题 会继续追问什么 本页要带走的准备
结构化输出 对象能否被严格解析和消费 参数 schema 是执行前合约，不是装饰文本。

boundary

本页不承诺什么

  • 不承诺只靠描述就能让所有模型稳定选对工具。
  • 不承诺 schema 能解决权限、归属、事实正确性和外部系统状态。
  • 不承诺一次工具结果就足够回答复杂任务，多轮场景仍要记录状态。

本页已经足够做什么

  • 审查一组工具声明是否会相互抢意图。
  • 为只读、写入和政策搜索三类工具设计不同的描述边界。
  • 写出最小分发器检查顺序，并给每类失败安排下一步。

自测标准：你能不能拿一个真实业务动作，写出工具名、描述、必填参数、确认条件和回填字段。如果可以，这一页就已经能落地。

怎么答才算会

不要只背“工具调用就是函数调用”。每题都要落到一个可检查环节：描述是否会混选，参数是否能执行，分发器是否放行，结果是否能回填。

schema design写入

为退款申请工具列出最小参数字段

请列出 create_refund_request 至少需要哪些字段，并说明哪些字段适合枚举或布尔值。

参考答案 / 评分要点

至少应包含 order_id、reason、amount_cents、confirmed_by_user。reason 适合 enum，confirmed_by_user 适合 boolean，金额要有最小值。好答案还会提到订单归属、状态和可退款金额属于业务校验，不只靠 schema。

debug归因

诊断一次错误工具调用

症状：用户问“退货政策是什么”，模型调用了 create_refund_request，而且宿主程序真的创建了申请。请按“选择问题 / 分发问题 / 修法”写。

参考答案 / 评分要点

选择问题：政策问题被写入工具抢走，说明描述边界或工具选择样例有问题。分发问题：写入动作没有确认就执行，说明宿主程序漏了确认和业务校验。修法：收窄描述，政策问题走 search_policy_docs；退款申请必须检查订单、金额、原因和用户确认。

synthesis迁移

Feynman：讲一遍从请求到回填的闭环

把答案写给一个正在接入函数工具调用的同事看，尽量用 120-200 字讲清楚。

参考答案 / 评分要点

合格答案要串起六步：用户提出任务，模型根据工具声明给出调用候选，参数 schema 把信息收成对象，宿主程序校验白名单、权限和确认，外部系统执行并返回结果，宿主程序把状态、ID、错误类别和下一步回填给模型。好答案还会提醒：高风险写入不能把模型选择当成执行命令。
