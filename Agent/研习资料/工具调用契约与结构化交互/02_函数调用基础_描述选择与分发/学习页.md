# 学习目标 {#goals}
<!-- tag: goals -->

看懂声明
看到一个工具声明时，能分出工具名、描述、参数 schema、输出语义和副作用等级。工具名主要服务路由；描述服务选择；schema 服务参数边界；输出服务后续推理。


追踪选择
能把一次用户请求拆成候选工具、选择理由、参数来源和缺失信息。遇到相近工具时，先看动作和对象，再看描述里的适用范围。


守住分发
能说明宿主程序为什么还要查白名单、schema、权限、确认和幂等。模型可以提出工具调用，但执行权要留在宿主程序手里。

**你现在应该能做什么：**拿到一个函数工具面，先判断它会不会被稳定选中，再判断它能不能被安全执行。

## box soft
### 读工具声明的顺序
先看名字能不能表达动作和对象，再看描述是否说明边界，然后看参数字段是否足以支持执行，最后看输出是否能被后续系统消费。这个顺序能快速发现“看起来能用，但实际会混选”的工具。

## box warn
### 最容易混掉的边界
模型生成的是调用请求，不是执行结果。真实执行需要宿主程序把工具名映射到本地函数，校验参数，检查权限，必要时向用户确认，然后才调用外部系统。

# 描述与选择 {#description}
<!-- tag: selection -->
<!-- layout: grid2 -->

## box info
### 描述要解决什么
描述给模型的选择说明。它要把动作、对象、适用场景、排除场景和风险写清楚。越是相近工具越需要边界句，例如“只用于查询，不用于创建或退款”。
如果工具描述都写成“处理订单相关问题”，模型只能靠上下文猜。猜对时你会觉得系统聪明，猜错时问题很难复盘，因为声明里没有可检查的边界。

## box soft
### 选择策略怎么参与
`auto` 让模型自己判断是否调用工具；`none` 禁止调用；`required` 要求至少调用一个；指定某个工具则把选择空间收窄到一个目标。策略不是质量保证，它只改变模型的选择空间。
工程上常见做法是：普通问答用自动选择，高风险写入先让模型给计划或追问，确认后再指定写入工具。这样可以减少“模型把查询句误当成执行句”的风险。

# 参数结构 {#schema}
<!-- tag: schema -->

## box soft
### 把 schema 当成模型要填写的表单
参数 schema 的任务是让模型把自然语言里的信息落到字段上。它不能替代业务权限，也不能保证事实正确；它先保证“对象形状能被程序读”。字段名要贴近业务对象，描述要解释取值含义，值域能收窄就收窄。

订单状态查询工具声明复制

```json
{
 "name": "get_order_status",
 "description": "查询单个订单的当前物流和支付状态。只读，不创建退款，不修改订单。",
 "parameters": {
 "type": "object",
 "additionalProperties": false,
 "required": ["order_id"],
 "properties": {
 "order_id": {
 "type": "string",
 "pattern": "^ORD-[0-9]{8}$",
 "description": "订单号，例如 ORD-20260621"
 }
 }
 }
}
```

`required`
告诉宿主程序哪些字段缺了就不能继续。缺订单号时追问，比默认猜一个订单安全得多。


`pattern` / `enum`
把开放字符串收进可检查范围。订单号适合格式约束，退款原因、渠道、状态这类字段常适合枚举。


`additionalProperties`
让参数对象保持闭合，避免模型把解释、猜测和调试信息混进要执行的参数里。

检查层
检查内容
适合放在哪里
失败后动作


结构校验
字段、类型、必填、格式、枚举
参数 schema 和解析器
追问或要求模型重填参数


业务校验
订单是否属于当前用户、状态是否允许退款
宿主程序和业务服务
拒绝、解释原因或转人工


执行校验
是否重复提交、外部系统是否可用
分发器、幂等层、外部 API
返回可重试错误或保留任务状态

**关键提醒：**schema 让参数更可读、更可查，但高风险动作还要做权限、确认、归属和幂等检查。

# 分发账本 {#dispatch}
<!-- tag: dispatcher -->
<!-- layout: grid2 -->

## box info
### 分发器的核心职责
分发器站在模型和真实系统之间。它不只是“按 name 调函数”，还要确认工具名在白名单里，参数能通过 schema，当前用户有权限，写入动作已经确认，重复请求不会造成重复副作用。

## box warn
### 为什么不能直接执行
模型输出可以被提示词影响，也可能因为相近工具而选错。把任意 name 和 arguments 直接交给系统，相当于把路由、权限和审计全交给概率输出。安全的做法是把模型请求当作候选，而不是命令。

宿主程序分发伪代码复制

```js
async function dispatchToolCall(call, user, registry) {
 const tool = registry[call.name];
 if (!tool) return reject("unknown_tool");

 const parsed = validateAgainstSchema(tool.parameters, call.arguments);
 if (!parsed.ok) return askForMissingFields(parsed.errors);

 const permission = await canUseTool(user, tool, parsed.value);
 if (!permission.ok) return reject("permission_denied");

 if (tool.sideEffect && !call.userConfirmed) {
 return askForConfirmation(tool.name, parsed.value);
 }

 const result = await tool.handler(parsed.value, { user });
 return normalizeToolResult(result);
}
```

检查
通过
失败


1
工具名是否在 registry 里
找到处理函数
拒绝未知工具，记录异常


2
参数是否符合 schema
得到规范化参数对象
追问缺失字段或要求重填


3
用户是否有权限
继续业务校验
拒绝并说明原因


4

执行外部调用
先请求用户确认


5

返回状态、ID、可重试性
补结构化错误，避免只返回异常字符串

# 订单工具组案例 {#case}
<!-- tag: case -->

`get_order_status`
只读工具。用于单个订单的物流、支付和售后状态查询。输入是订单号，输出要包含订单状态、物流状态和是否可发起退款。


`search_policy_docs`
知识库搜索工具。用于查政策、条款、流程和规则说明。它不应该接收个人订单号，也不应该改动订单状态。


`create_refund_request`
写入工具。用于创建退款申请。必须有订单号、退款原因、金额和用户确认；执行后要返回退款申请 ID。


## box info
### 怎么读测试结果
如果理想请求能过、边界请求错了，先改描述和负例；如果模型选对了但参数缺字段还被执行，查 schema 和分发器；如果写入请求没有确认就通过，问题在宿主程序的执行门槛；如果无关请求也调用工具，检查工具选择策略和拒绝样例。

## box warn
### 最小通过线
这四类输入都要有稳定预期：只读查询能执行，政策问题走文档，写入动作先确认，无关请求不调用工具。只要其中一类没有稳定预期，这个工具面就还不能算真正可维护。

- 不承诺只靠描述就能让所有模型稳定选对工具。
- 不承诺 schema 能解决权限、归属、事实正确性和外部系统状态。
- 不承诺一次工具结果就足够回答复杂任务，多轮场景仍要记录状态。

## box soft
### 本页已经足够做什么
- 审查一组工具声明是否会相互抢意图。
- 为只读、写入和政策搜索三类工具设计不同的描述边界。
- 写出最小分发器检查顺序，并给每类失败安排下一步。

**自测标准：**你能不能拿一个真实业务动作，写出工具名、描述、必填参数、确认条件和回填字段。如果可以，这一页就已经能落地。

Explain: 合格答案要说清三层：模型负责根据上下文和工具声明提出调用候选；宿主程序负责查白名单、校验参数、检查权限和确认；外部系统负责真实查询或写入并返回结果。更好的答案会补一句：把任意一层混掉，都会造成混选、越权或回填不可用。

### 5_best_description choice

**Q:** 哪个描述更适合 `get_order_status`？

- [ ] . 处理订单相关事情。
- [ ] . 一个很强的订单工具。

Answer: . 查询单个订单的物流和支付状态。只读，不创建退款，不修改订单。

### 6_rewrite_policy rewrite

**Q:** 改写一个模糊的政策搜索工具描述

Explain: 合格答案类似： search_policy_docs ：搜索公开政策文档，用于回答规则、条款和流程问题；不用于查询个人订单，也不创建退款。评分看三点：动作明确、对象明确、排除场景明确。

### 7_choice_modes checklist

**Q:** 哪些场景适合收窄 `tool_choice`？

- [ ] ，可以指定退款申请工具。
- [ ] ，可以禁止工具调用。
- [ ] ，所以强制 required 就能保证正确。
- [ ] 。

Explain: 应选第一、第二和第四项。第三项是陷阱：强制调用只改变选择空间，不能修复工具描述和 schema 的边界问题。高风险动作通常要先确认，再指定具体工具。

### 8_refund_request choice

**Q:** 用户说“帮我直接退了”，第一步最稳是什么？

- [ ] . 立刻调用退款工具，金额默认填全额。
- [ ] . 确认订单、金额、原因和用户明确同意，再进入写入工具。
- [ ] . 用政策搜索工具代替退款工具。
- [ ] . 只回答“已完成”。

Answer: . 确认订单、金额、原因和用户明确同意，再进入写入工具。

### 9_dispatch_checks checklist

**Q:** 分发器执行前应该检查哪些项？

- [ ] 工具名是否在白名单 / registry 里。

### 11_refund_schema schema-design

**Q:** 为退款申请工具列出最小参数字段

Explain: 至少应包含 order_id 、 reason 、 amount_cents 、 confirmed_by_user 。 reason 适合 enum， confirmed_by_user 适合 boolean，金额要有最小值。好答案还会提到订单归属、状态和可退款金额属于业务校验，不只靠 schema。

### 15_failure_path debug

**Q:** 诊断一次错误工具调用

Explain: 选择问题：政策问题被写入工具抢走，说明描述边界或工具选择样例有问题。分发问题：写入动作没有确认就执行，说明宿主程序漏了确认和业务校验。修法：收窄描述，政策问题走 search_policy_docs ；退款申请必须检查订单、金额、原因和用户确认。

### 16_feynman_loop synthesis

**Q:** Feynman：讲一遍从请求到回填的闭环

Explain: 合格答案要串起六步：用户提出任务，模型根据工具声明给出调用候选，参数 schema 把信息收成对象，宿主程序校验白名单、权限和确认，外部系统执行并返回结果，宿主程序把状态、ID、错误类别和下一步回填给模型。好答案还会提醒：高风险写入不能把模型选择当成执行命令。
