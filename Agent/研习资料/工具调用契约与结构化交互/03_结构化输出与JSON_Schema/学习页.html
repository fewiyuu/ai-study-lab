第 03 单元

03｜结构化输出与 JSON Schema

这页只处理一件事：怎么把“模型看起来像在输出 JSON”推进到“系统真的能稳定消费这个结构”。你会先分清结构化回答、工具参数和协议级 schema，再用一个订单状态对象和一个会议风险对象把约束、枚举、必填和额外字段看清楚；错误处理、取消、评测和完整 MCP 协议留到后面单元。

课程地图上一单元下一单元

读完本页能做什么

  • 能说清 JSON、JSON Schema、Structured Outputs 和工具参数各自负责什么。
  • 能手工诊断一个 schema 里最常见的三个问题：必填缺失、值域过宽、额外字段失控。
  • 能把自然语言需求改写成可消费的结构，并判断什么时候该用结构化回答，什么时候该用工具参数。

学习目标

goal

这页先解决什么

先把“合法 JSON”与“可被系统稳定消费”分开。前者只管语法，后者还要看字段是否齐全、类型是否正确、值域是否收紧、额外字段是否会把下游打乱。

读完后要能做的动作

你应该能拿到一个模糊需求，先判断它属于结构化回答还是工具参数，再给出一个最小 schema 草案，并指出它最可能出错的地方。

术语和边界

terms

JSON

一种数据表示格式。它只说明写出来是否合法，不保证内容适不适合业务。

JSON Schema

一种用 JSON 描述 JSON 结构和约束的规范。它关心字段、类型、必填、枚举、数组长度和额外字段策略。

Structured Outputs

让模型输出服从结构约束的能力。它解决“输出可解析”，无法替代“事实绝对正确”。

工具参数 schema

它比普通结构化回答更敏感，因为一旦参数被下游执行，错误就会变成真实动作。

先记住一条线合法 JSON 是语法检查，schema 是结构检查，业务校验是动作检查。把这三层混在一起，页面、接口和评测都会变脆。

核心机制

trace

先用一个订单状态对象走一遍

假设有一个“查订单当前状态”的需求，输入只需要订单号，输出要给前端和后续规则继续用。这个对象适合看清“输入约束”和“输出约束”是两件事。

订单状态 schema
{
  "type": "object",
  "additionalProperties": false,
  "required": ["order_id"],
  "properties": {
    "order_id": {
      "type": "string",
      "pattern": "^ORD-[0-9]{8}$",
      "description": "订单号，例如 ORD-20260609"
    }
  }
}

为什么这样写

required 让输入对象至少有一个稳定锚点；pattern 把订单号约束成可识别格式；additionalProperties: false 防止模型往对象里塞一堆下游用不到的字段。

错在什么地方

如果只写 {"type":"object","properties":{"order_id":{"type":"string"}}}，模型可以给你一个空对象，也可以混进额外字段。结果是解析表面成功，业务层仍然不知道该不该继续。

层 读取 写入 / 约束 失败信号
输入 schema 订单号格式 只接受 ORD-YYYYMMDD 这类值 空对象或乱填字符串
结构化输出 查询意图、上下文 一个 JSON 对象 对象能解析，但字段含义不稳
业务消费 对象字段和类型 状态展示、追踪、后续分支 前端显示错、后续规则分支走偏

案例

example

案例 A：看起来像 JSON，但不够稳

weak schema
{
  "type": "object",
  "properties": {
    "result": { "type": "string" },
    "extra": { "type": "object" }
  }
}

这个 schema 的问题很直接：字段语义太泛、没有必填项、没有额外字段控制，也没有任何值域收紧。消费者看到它，只能猜 result 到底代表什么。

案例 B：会议风险对象

需求是“阅读会议纪要，输出是否有风险、风险类型、需要谁跟进、依据是什么”。这里最该做的是把自然语言拆成稳定字段，再把易漂移的地方收进枚举。

meeting-risk schema
{
  "type": "object",
  "additionalProperties": false,
  "required": ["risk_found", "risk_type", "owner", "evidence"],
  "properties": {
    "risk_found": { "type": "boolean" },
    "risk_type": {
      "type": "string",
      "enum": ["deadline", "scope", "dependency", "budget", "unknown"]
    },
    "owner": { "type": "string" },
    "evidence": { "type": "string" }
  }
}

这里的关键 risk_type 用枚举，是为了让后续分派有稳定入口；evidence 仍然保留字符串，是因为它需要承载解释，不适合强行枚举。

诊断

debug

一眼能看出来的三个问题

  • 没有必填项，空对象也可能通过。
  • 没有额外字段控制，模型可以塞进一堆多余信息。
  • 没有枚举或模式约束，消费者无法稳定判断值域。

阅读顺序

先看对象是否闭合，再看字段是否必要，再看类型是否够窄，最后看下游是否能直接消费。这个顺序比“先找语法对不对”更有效，因为语法对了也不代表系统能用。

消费桥接

bridge

为什么要单独讲这一段

结构化输出把模型产物收成一个对象，这只是起点。真正决定系统好不好用的，是对象怎么被解析、怎么被校验、怎么被归一、怎么进入工具调用或页面渲染。只要消费链里有一层没有固定语义，页面看起来再整齐，最后也会变成“读得出来，却用不起来”。

所以读 schema 时，要把注意力从“它是否只是一个 JSON”移到“这个 JSON 进来以后谁接、谁拦、谁改、谁记账”。这就是结构化输出和普通自然语言回答最大的分界线：前者要为后续动作负责，后者只要把意思说清。

判断顺序

先确认对象是否完整，再确认字段语义是否足够具体，再确认值域是否收紧，最后确认执行前还有没有业务门槛。这个顺序能把排查路径缩短很多，因为问题常常落在对象边界、字段含义和动作后果上，而不只是落在 JSON 语法上。

如果你在做项目，最好把每个字段都记成三列：它从哪来，下游怎么用，错了会影响谁。这个做法很朴素，但能把“字段设计”直接拉到“系统接口设计”的层面。

1. 解析

先把模型输出转成对象，确认结构确实能被读出来。

2. 结构校验

再检查 required、enum、pattern、items、minimum 和 additionalProperties 等约束。

3. 业务校验

接着判断对象是否满足场景规则，例如权限、归属、状态、时间窗和幂等。

4. 执行

最后才把对象送进工具、写入数据库、更新页面或触发通知。

层 要回答的问题 典型输入 典型失败
解析层 它能否被读出来 模型输出的字符串、函数返回值或结构化响应 括号错、类型错、字段层级错，导致对象读不出来
结构层 字段、类型和值域是否和 schema 对齐 required、enum、pattern、minimum、additionalProperties 空对象混进来、值域过宽、字段漂移或多余键污染消费端
业务层 这个对象能不能在当前场景继续走 权限、状态、归属、时间窗、幂等键、环境变量 对象通过结构检查，但动作本身不允许执行
执行层 动作发出去以后会改什么状态 工具调用、数据库写入、页面更新、通知发送 写错目标、重复执行、覆盖旧值、日志无法复盘
消费端伪代码
function handleMeetingRisk(rawText) {
  const parsed = parseJson(rawText);
  if (!parsed.ok) {
    return {
      ok: false,
      stage: "parse",
      message: "结构读不出来，先不要进入后续步骤"
    };
  }

  const data = parsed.value;
  const schemaCheck = validateSchema(data, meetingRiskSchema);
  if (!schemaCheck.ok) {
    return {
      ok: false,
      stage: "schema",
      message: "字段、类型或值域不合格",
      issues: schemaCheck.issues
    };
  }

  const businessCheck = validateBusinessRules(data);
  if (!businessCheck.ok) {
    return {
      ok: false,
      stage: "business",
      message: "对象合格，但当前场景不允许执行",
      issues: businessCheck.issues
    };
  }

  const action = buildTaskAction({
    owner: data.owner,
    riskType: data.risk_type,
    evidence: data.evidence
  });

  return execute(action);
}

常见故障路径

  • 只做解析，不做 schema 校验，空对象和宽字段会持续溜进下游。
  • 只盯 schema，不看业务状态，合法对象也可能触发错误动作。
  • 只看工具调用结果，不看前面的对象归一，排查时很难定位问题点。
  • 把失败都归到“模型不稳定”，会掩盖具体是哪一层没有拦住。

修法模板

先把失败归层：是 parse、schema、business 还是 execute。再把每一层的返回值写具体，最好把错在哪个字段、哪个规则、哪条状态分支都写出来。这样页面、日志和调试输出就能互相对应，后面排查时不用从头猜。

如果你要把这页的内容迁到真实项目里，就把 schema 放在最靠近输入的地方，把业务校验放在真正执行前，把执行日志放在最末端。这样出问题时，回放路径会清楚很多。

失败模式 现象 根因 修法
parse 失败 对象读不出来，页面和工具都拿不到结构 输出里混进了自然语言、引号不闭合或层级错位 先约束输出格式，再把解析失败当成独立错误处理
schema 失败 对象能读出来，但 required、enum 或 pattern 没过 字段太宽、值域太散、额外字段没人管 收紧字段定义，补足 required 和 additionalProperties
business 失败 对象看着对，执行时却被拒绝 权限、归属、状态、时间窗或幂等条件不满足 把业务规则放在执行前，并返回能定位问题的错误信息
execute 失败 前面都通过了，最后动作仍然落空 接口限流、服务超时、下游字段映射错误或重复提交 记录请求、响应和回退动作，给执行层单独做追踪

到这里应该形成的直觉 schema 只是接口的一部分，真正稳定的对象一定经过了结构、业务和执行三层。你现在应该能把一个模糊需求拆成“字段怎么长、谁来拦截、什么时候执行”三步。

迁移一致性

consistency

一个真实变化

假设会议风险对象从 v1 的 risk_found、risk_type、owner、evidence，升级到 v2。新版本增加 severity、action_due 和 source_span，同时把 owner 改成 assignee。这个变化一发生，上游生成、下游读取、适配层和测试数据都要跟着改。

这类改动最容易出问题的地方不在新字段本身，而在旧页面、旧脚本、旧提示词还继续认旧字段。表面上对象还能过，实际消费链已经分叉。你只看一次运行，很容易误以为系统正常；你看版本边界，问题才会显出来。

先看哪几层

先看上游是否停止产生旧字段，再看下游是否接受新字段，再看是否有兼容层保住旧版本。把这三件事拆开，版本迁移会清楚很多。只改 schema 不改消费方，通常会把问题推迟到线上。

如果没有兼容层，就要把旧版本显式拒绝，不能让它悄悄混进来。沉默失败最难排查，尤其是当对象还能解析、页面还能显示、日志却已经写错时。

1. 识别版本

先标出当前对象属于 v1 还是 v2，避免混用字段。

2. 生成兼容层

必要时把旧字段映射到新字段，把缺失字段补成默认值。

3. 切消费端

前端、API、脚本和提示词一起切换到新字段名和新值域。

4. 删除旧路径

确认没有遗留调用后，再撤掉旧 schema 和旧适配逻辑。

角色 旧版本看什么 新版本看什么 迁移动作
生产者 输出 risk_found / risk_type / owner / evidence 输出 severity / action_due / source_span / assignee 改生成模板，补字段来源，重新检查 required
消费者 按旧字段做展示、分派或写库 按新字段做展示、分派或写库 更新解析、映射和存储逻辑，避免字段名断裂
适配层 把旧对象直接交给业务代码 把新对象归一到统一内部模型 写一个版本转换器，专门处理映射和默认值
测试层 只测一次成功示例 覆盖旧版、新版、缺字段、错字段和多字段 把失败样例写进回归集，防止回退
版本适配器
function normalizeMeetingRisk(raw) {
  const parsed = parseJson(raw);
  if (!parsed.ok) {
    return fail("parse");
  }

  const v2 = parsed.value;
  if (v2.schema_version === "2") {
    return {
      risk_type: v2.risk_type,
      severity: v2.severity,
      owner: v2.assignee,
      evidence: v2.source_span,
      action_due: v2.action_due
    };
  }

  if (v2.schema_version === "1") {
    return {
      risk_type: v2.risk_type,
      severity: "medium",
      owner: v2.owner,
      evidence: v2.evidence,
      action_due: null
    };
  }

  return fail("unsupported_version");
}

如果迁移没做好

最常见的后果是字段名换了，旧 consumer 还在读旧名，结果所有对象都变成空值。另一种情况是 schema 变宽了，但适配层没有收拢，旧数据和新数据混在一起，最后日志、UI 和数据库出现三套解释。

还有一种更难查的情况：测试只覆盖新路径，旧路径静默失效。这个时候页面看起来能跑，回放却对不上，问题往往拖到业务事故后才被发现。

修法优先级

先加版本号，再加适配层，再切消费端，最后清旧路径。顺序不要反过来。反过来做会让旧对象和新对象互相污染，排查范围会变得很大。

这也是结构化输出和一般 JSON 配置的区别之一：前者一旦变更，影响的是一整条消费链；后者很多时候只要换个字段名就结束了。你在设计 schema 时，始终要把迁移成本算进去。

发布前检查

最少准备四个样本：旧版成功、新版成功、旧版缺字段、旧版多字段。逐条跑适配器，看输出模型是否统一，记录里是否保留版本号。这个步骤不花太久，但能提前暴露大多数字段迁移事故。

放行条件

任何一个样本没过，都先停在适配层，不要直接改消费端。消费端一旦跟着补丁修到一半，旧链路就可能继续漂移，最后连回滚点都找不到。

失败点 用户看到什么 系统内部发生什么 第一修法
旧 consumer 没改 页面字段空白，分派错误，导出内容不全 新字段进来了，旧映射还在读旧键 先补兼容映射，再改业务读取点
旧 prompt 没改 模型继续吐旧字段，结果和新 schema 对不上 生成端和校验端版本脱节 同步更新提示词和示例输出
只改测试样例 测试看起来通过，线上却持续报错 测试没覆盖真实流量里的旧路径 把旧版样例、异常样例和兼容样例一起纳入回归
没有回滚方案 一旦新版本有问题，无法快速退回 旧 schema 和旧适配器已经被删掉 保留一个可回切窗口，再做正式下线

你现在应该能做什么 拿到一个 schema 变更时，先列出生产者、消费者、适配层和测试层四个位置，再说清楚每一处要改什么、先改哪一处、哪一处要保留兼容。只要这四列能写出来，迁移风险就已经被拆开了，发版前也更容易做抽样检查。

抽样检查时，先看旧样本是否还能被归一，再看新样本是否能稳定落库。

边界

scope

这一页讲到哪

讲到 schema 层和应用消费层之间的接口：字段名、类型、枚举、必填和额外字段。你已经能判断一个结构够不够稳，也能把自然语言需求初步收敛成对象。

这一页不往哪去

不往错误恢复、取消流程、回退策略、完整 MCP 协议和评测基准扩展。那些概念会在后面单元重新出现，而且会和今天的结构约束接起来。

练习

diagnostic

怎么答才算会

不要只说“更严谨”或“更结构化”。答案要能落到对象、字段、值域和下游消费上；开放题尽量按“先看什么、错在哪里、怎么改”来答。

检查重点

先看 schema 是否闭合，再看字段是否有稳定语义，再看值域是否收紧，最后看是否能直接交给前端、解析器或工具执行。

synthesis操作桥

如果结构化输出通过了，但工具调用后的业务结果仍失败，你会按什么顺序检查？

参考答案 / 评分要点

好的顺序通常是：先确认对象是否能解析，再确认 schema 是否真通过，然后看业务约束是否满足，接着查权限、归属、状态和幂等，最后才看下游接口是否有自己的限制。答案如果只写“重试一下”或“让模型再试一次”，说明还没把结构层和执行层分开。

应该勾出前两项和最后一项。把所有信息塞回一个 text 字段会把结构重新打散，等于把 schema 的作用削掉。正确迁移的方向是让参数更具体、值域更窄、输出更贴近下游真正要用的字段。

参考答案 / 评分要点

合格答案要串起四件事：JSON 只是格式，JSON Schema 负责结构约束，Structured Outputs 让模型更稳定地服从结构，工具参数还会进一步进入执行链路。只要答案能说清“语法、结构、执行”这三层不同，就算比背定义更好。最好再补一句：结构化输出的目标是让下游能安全、稳定、可预测地继续工作。
