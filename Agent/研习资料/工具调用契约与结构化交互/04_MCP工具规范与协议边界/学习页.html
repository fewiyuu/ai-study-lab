04｜MCP 工具规范与协议边界

这页只讲 MCP 工具规范里的协议边界：你会看到 tools/list 和 tools/call 怎么把工具暴露给模型， inputSchema 与 outputSchema 怎么收紧输入输出， content 、 structuredContent 、 isError 怎么把结果分给人和机器。这里先不展开资源、prompts、授权和完整传输栈，它们会在后面的单元再回来。

学习目标

goals

先抓主线

这一页先把 MCP 看成一条协议化的调用链：模型给出调用候选，client 和 server 通过协议交换工具列表和工具调用，server 返回工具结果，再由宿主程序把结果送回模型继续推理。只要这条链里任意一层混掉，后面就会出现错选工具、结果不可消费、日志污染或权限误判。

先记边界，再记字段表：哪些信息属于协议消息，哪些属于工具执行结果，哪些属于宿主应用策略，哪些留给评测。资源、prompts、认证和传输细节会在后面再回来，这一页先不追它们的整套展开。

读完后要能做的动作

拿到一个 MCP 工具接入方案时，你能先判断它在说哪一层：协议层、工具层、结果层，还是评测层。再往下，你能检查 tool definition 有没有把名字、描述、输入约束和输出约束写清楚，工具结果有没有把人看的文本和机器看的结构分开。

如果一个失败看起来“像模型不聪明”，你应该能先追问：是协议请求本身不成立，还是工具执行失败，还是 stdout 把消息流弄脏了。

术语

vocab

host

宿主应用。它决定用户界面、权限提示、模型调用方式和最终要暴露哪些工具。很多人把 host 和 server 混为一谈，其实 host 管的是用户那一侧的体验和调度，server 才是能力提供方。

client

协议客户端。它负责和 server 说话，把工具列表和工具调用消息真正发出去。client 不负责业务本身，但负责把协议消息送到正确的地方。

server

MCP 服务端。它负责声明工具、验证输入、执行工具、返回结果。server 如果只会报“成功”，不会说明结果结构和失败语义，那工具面很快就会失控。

tool / result

tool 是可被调用的能力；result 是它返回给模型和宿主的东西。工具名是路由键，结果要能被后续步骤继续消费。

protocol error / tool error

协议错误表示这次请求在协议层就不成立，比如工具不存在、消息格式错、能力不支持；工具错误表示协议请求成立了，但工具执行失败，比如业务参数无效、权限不足、下游超时。

tool_choice

选择空间控制。它决定模型是自由选、必须选、只能选指定工具，还是先不走工具。这个控制会改变选择空间，但不会替你修掉工具描述不清或 schema 太松的问题。

层典型症状责任点下一步
协议错误工具名不存在、消息格式不对、能力没声明client / server / 协议会话先检查 tools/list、方法名和能力协商
工具错误参数无效、权限不足、业务拒绝、下游失败工具本身和业务服务看 inputSchema、业务规则和错误语义
传输错误stdio / stream 被日志、乱码或非消息内容污染传输层和运行时把日志和消息流分开，检查 stdout / stderr 归属

角色账本

ledger

这一张表的作用很简单：先看谁在读，谁在写，谁有权拦，谁只负责透传。只要角色关系一乱，工具面就会变成“所有人都能改，但没人负责”。

读表顺序

先看“读什么”，再看“写什么”，最后看“最容易犯的错”。如果一项角色既读又写同一类状态，就要追问这层负责校验、执行还是回填。

工具定义

mechanism

先看一个最小工具定义

这个例子用的是一个公开政策文档搜索工具。它的名字、描述、输入 schema 和输出 schema 都尽量收紧，这样模型更容易选对工具，工具执行结果也更容易继续消费。

最小 tool definition复制

{
 "name": "search_policy_docs",
 "title": "Search policy docs",
 "description": "Search public policy docs for rules, terms, and workflows. Do not use for personal orders or refunds.",
 "inputSchema": {
 "type": "object",
 "additionalProperties": false,
 "required": ["query"],
 "properties": {
 "query": {
 "type": "string",
 "description": "The topic or question to search for"
 },
 "category": {

 "enum": ["policy", "workflow", "faq"],
 "description": "Narrow the search scope"
 }
 }
 },
 "outputSchema": {
 "type": "object",
 "additionalProperties": false,
 "required": ["hits", "summary"],
 "properties": {
 "hits": {
 "type": "array",
 "items": {
 "type": "object",
 "additionalProperties": false,
 "required": ["title", "url"],
 "properties": {
 "title": { "type": "string" },
 "url": { "type": "string" }
 }
 }
 },
 "summary": { "type": "string" }
 }
 }
}
字段谁读谁写如果写错会怎样
nameclient、server、模型开发者路由失败，或者工具被混选
title人类、界面、调试日志开发者展示不清，但通常不影响路由
description模型、提示词、选择逻辑开发者动作边界不清，模型会猜错
inputSchema模型、解析器、校验器开发者参数太宽、空对象混进来、值域失控
outputSchema消费端、评测、回填链路开发者结果能返回，但后续无法稳定读取

这里最该盯的两件事

description 负责帮助模型做选择，inputSchema 负责把输入收紧成可验证对象。很多人只写“能查什么”，不写“不能查什么”，结果相近工具抢同一个意图，排查时只能靠猜。

另一件事是 outputSchema。它决定结果是否能被下游稳定消费。结果结构如果不稳，后面的 UI、规则引擎和评测集都会跟着散。

结果语义

result

tool result example复制

这三个字段分别干什么

content 是给人看的自然语言说明，通常用来讲结果摘要、提示下一步或解释失败原因。structuredContent 是给机器继续消费的结构化对象，最适合放 hits、state、id、status 这类后续会被程序读取的内容。isError 是这次工具执行有没有失败的标记，它会直接影响后续应该重试、追问还是停下来。

最容易掉进去的坑

只返回一段顺口的文本，结果会让人看得懂、程序接不住。只返回结构化对象，不给人类一句话说明，后续用户又会不知道为什么这个结果重要。把两种输出混在一个字段里，短期看似省事，长期会让回填链路越来越难维护。

字段主要服务谁适合放什么错放后的症状
content人类读者摘要、说明、下一步建议、错误解释结果能看懂，但程序没法稳定消费
structuredContent机器和下游系统结构化对象、列表、状态、id、计数、分支信息字段变来变去，消费端每次都要补丁
isError宿主、模型、调试者工具执行是否失败的布尔标记业务失败被当成功，或者成功被误判成失败

结果回填路由复制

function routeToolResult(result) {
 if (result.isError) {
 return {
 stage: "tool_error",
 nextAction: "retry_or_ask_for_clarification",
 humanMessage: result.content?.[0]?.text ?? "Tool failed."
 };
 }

 return {
 stage: "success",
 nextAction: "continue_reasoning",
 machinePayload: result.structuredContent,
 humanMessage: result.content?.[0]?.text ?? ""
 };
}

这段代码展示的是责任分工：失败就走错误分支，成功就把机器对象和人类说明分开。这样后面的选择、回填和重试才不会混成一锅。

边界案例

debug
案例属于哪一层为什么这样分你先看哪里
工具名不存在协议层错误工具根本没被合法发现，调用请求无法成立tools/list、工具命名、路由表
参数缺失或值域不对工具错误协议请求成立了，但输入对象没过校验inputSchema、业务规则、确认流程
日志写进 stdout传输层错误消息流被污染，协议字节都不干净了stdout / stderr、消息输出位置

把边界拆开以后，修法也会清楚很多

协议层错误先查工具是否存在、能力是否声明、消息格式是否正确；工具错误先查输入 schema、业务规则和权限；传输层错误先查日志有没有跑到 stdout。你不用把所有失败都归成“模型不稳定”，那样只会把排查越推越远。

迁移桥接

bridge

和上一单元怎么接

上一单元讲的是结构化输出和 JSON Schema，这一页讲的是协议化工具调用。两者都在用 schema，但位置不一样：结构化输出约束的是模型直接吐出来的内容，MCP 的 inputSchema 和 outputSchema 约束的是工具协议中的输入和输出。前者解决“回答能不能解析”，后者解决“动作能不能路由、结果能不能回填”。

所以当你在看一个系统时，先问它要的是哪一种：是只想让模型给一份结构化答案，还是要把动作真正交给外部系统执行。这个判断决定了你该用 Structured Outputs、function calling，还是再往下走到 MCP。

三个边界

Structured Outputs 适合模型自己输出可解析结构，重点在回答格式。Function calling 适合把调用候选交给宿主程序执行，重点在工具选择和参数收紧。MCP 适合跨进程、跨语言、跨服务地管理工具面，重点在协议化发现、调用、结果和边界。

这三者可以串起来，但不要混成一层。串起来时，分工才是清楚的；混成一层时，后面所有错误都会变成“看起来像 JSON 的问题”。

选择与执行分层复制

if 只需要结构化答案:
 用 Structured Outputs
elif 需要模型提出调用候选、宿主来执行:
 用 function calling
elif 需要跨服务的协议化工具面:
 用 MCP
能力主要约束什么谁执行真实动作常见误用
Structured Outputs模型输出的结构模型自己以为结构化输出已经能替代工具调用
Function calling模型选择工具、填写参数宿主程序把选择策略当成执行保证
MCP工具面、会话、消息、结果client / server / host 协作把它当成“再加一层封装的本地函数库”

这里先不追的概念：资源、prompts、授权框架、长会话状态和 transport 细节。它们会在后面的单元重新出现；现在先把“结构化回答”和“协议化工具面”分清。

评测

bfcl

为什么要单独看评测

工具面做得像样，不代表它真的稳定。BFCL 的价值不在“给分很高”，而在于它提醒你把工具调用拆成可测维度：工具选没选对、参数填没填对、能不能拒绝无关工具、能不能在多步和并行场景里保持一致。对你自己的业务系统来说，这些维度都应该单独写进评测集。

评测维度它在看什么为什么不能只看一次演示你本地怎么借用
工具选择模型能不能从候选里选对工具演示一次成功，不代表相近工具不会混选准备相近描述、相近意图和排除边界样本
参数抽取字段、类型、枚举、必填是否填对一次样例过了，空值、错值、宽值仍可能溜进去覆盖空对象、错字段、错枚举和多字段样本
多步 / 多工具一轮调用之后，下一轮还能不能接上单次成功不等于链路可维护准备两步以上的任务链和中间状态样本
拒绝无关工具模型能不能对不该用的工具说不能用不代表都该用加“应该直接回答，不应该调用工具”的样本
异常处理工具失败后还能不能继续解释或重试失败样本比成功样本更能看出边界是否稳加权限不足、参数缺失、下游超时和 stdout 污染样本

你该从 BFCL 学什么

要借用的是拆分方式，不是分数本身。工具调用能力要拆成很多个小问题去测，才能看出是选错工具、参数没填对，还是一到异常就崩。把这些维度写成业务自己的回归集，比只看一次成功截图有用得多。

上线前自查

如果你要把这一页的内容落到自己的系统里，先别急着把完整 MCP 接入想得太大。先检查工具定义、返回结构和日志通道，再准备少量高价值回归样本，这样才知道问题是出在协议、工具还是结果消费。

  • 工具名和描述只覆盖当前单元的边界，不把 prompts、resources 或 authorization 拉进主线。
  • inputSchema 把必填项、枚举和值域收紧到可验证范围。
  • content、structuredContent 和 isError 的分工保持稳定。
  • stdout 只输出协议消息，日志和调试信息走 stderr 或独立渠道。
  • 评测集包含相近工具、空参数、多步链路和失败样本。

实现一致性

如果你把 MCP 接进真实业务，最先要守住的是“同一件事在不同层说同一种话”。工具名、错误码、日志字段和评测样本要对应同一个边界，否则协议、宿主、工具和分析面会各写各的版本，排查时谁都以为问题在别处。

最实用的做法，是给每个工具配一条短说明、一组 schema 约束和一组失败样本，三者一起改，不单独改其中一项。这样当你后来改动工具名、参数或返回结构时，回归测试会直接告诉你是选择、参数、结果还是错误语义被破坏了。

常见误区

fix
误区为什么会坏修法
把 MCP 当成“让模型随便调用本地函数”的库会忽略会话、能力协商、传输、结果和边界，最后只剩一个壳。先画 host / client / server / tool / result 的责任账本，再写接入代码。
只写 description，不收紧 inputSchema模型可能把空值、宽值和多余字段一起塞进来。用 required、enum、format、additionalProperties 先把对象收窄。
把业务失败塞进协议 error模型看不到可修正信息，后续也难以判断是协议问题还是业务拒绝。工具内部失败用清楚的 tool result 返回，让模型看到可重试或可追问的线索。
把安全控制只写在工具描述里描述只能引导模型，挡不住错误执行或恶意请求。权限、确认、审计和最小暴露要在 host、server 和业务系统里做。
把 structuredContent 和 content 混成一个字段人类和机器都要各自猜一次，回填链路会越来越脆。让 content 负责说明，structuredContent 负责对象，isError 负责失败标记。

四个先查的问题

  • 工具是不是已经在 tools/list 里被合法暴露出来。
  • 输入对象是不是被 inputSchema 收紧到可验证范围。
  • 结果是不是把 content 和 structuredContent 分开了。
  • 失败是不是被正确标成 isError，而不是直接混在文本里。

怎么答才算会
