# 学习目标 {#goals}
<!-- tag: goals -->
<!-- layout: grid2 -->

## box soft
### 先抓主线
工具调用的失败路径可以拆成四条分支：模型拒绝、工具错误、用户取消和回退策略。它们对应的判断层不同，恢复动作也不同。先分开，后面才知道该停、该补问、该重试，还是该缩小动作范围。

## box info
### 读完后要能做的动作
拿到一条失败消息时，你能先判断它属于哪一层，再给出下一步。你也能给工具错误对象补出可执行字段，让调用方不用猜“这个失败能不能重试、要不要问用户、能不能直接换工具”。

**你现在应该能做什么：**先把失败分层，再看恢复动作，不再把所有异常都塞进同一个解释里。

# 术语 {#terms}
<!-- tag: terms -->
<!-- layout: grid2 -->

## box
### refusal
模型或策略明确拒绝继续执行。系统要把它保留成单独分支。

## box
### tool error
协议请求成立了，但工具执行失败了。参数无效、权限不足、下游超时、业务状态冲突都属于这一类。

## box
### cancel
请求还没完成，但调用方不需要它了。取消的核心是停止继续做无用功，并释放资源。

## box
### fallback
在当前动作不合适时，换成更窄、更慢或更需要确认的动作。回退要收窄范围。

**这里先记住：**拒绝和取消都可能要求停下，工具错误更常需要恢复；回退是一种要单独设计的策略。

# 模型拒绝 {#refusal}
<!-- tag: refusal -->

## box soft
### 先看一个高风险请求
用户要求生成可直接用于违规操作的步骤，或者要求把写入动作伪装成读取动作。这里的正确反应是拒绝并给出安全替代。拒绝要能被程序单独检测，这样宿主程序才知道该停在安全边界，而不会继续往下执行。

拒绝分支示例复制

```json
{
 "refusal": {
 "reason": "request_requires_disallowed_action",
 "message": "这类请求不能继续执行。可以改成只读解释、风险说明或合规替代方案。",
 "suggested_next_step": "ask_for_safe_alternative"
 }
}
```

检查点
应该看到什么
不该发生什么


分支识别
拒绝被单独标记出来
把拒绝混进普通成功对象


后续动作
停止执行或换成安全替代
继续调用写操作工具


用户沟通
告诉用户边界和可替代路径
只返回一个空泛的“失败”

# 错误对象 {#error-contract}
<!-- tag: contract -->

## box soft
### 别只写一个 failed
工具错误最有用的地方，是让调用方知道“为什么失败”和“下一步怎么走”。一个能被程序读的错误对象，至少应该告诉你是可重试还是需要用户输入，是权限问题还是参数问题，是局部失败还是整个动作都应该停下。

可恢复错误对象复制

```json
{
 "error": {
 "type": "validation_error",
 "code": "missing_field",
 "field": "date",
 "message": "日期字段缺失。",
 "retryable": false,
 "need_user_input": true,
 "safe_to_retry": false,
 "suggested_next_step": "ask_for_date"
 }
}
```

字段
读什么
写什么
写错后的症状


`type`
错误属于哪一类
validation / permission / timeout / conflict
不同层的问题被混成一类


`field`
哪一个输入出问题
具体字段路径
调用方不知道该补哪个值


`retryable`
这次是否值得重试
布尔值
把必然失败的动作反复撞墙


`need_user_input`
是否要补问用户
布尔值
自动重试把本该问的问题吞掉了


`safe_to_retry`
重试是否会放大副作用
布尔值
带写操作的请求被盲目重放

错误路由伪代码复制

```js
function handleToolError(error) {
 if (error.type === "permission") return "stop_and_request_authorization";
 if (error.need_user_input) return "ask_user_for_missing_field";
 if (error.retryable && error.safe_to_retry) return "retry_once_with_backoff";
 return "stop_with_clear_explanation";
}
```

# 失败链路演算 {#diagnosis}
<!-- tag: trace -->

## box soft
### 用一个私有报表导出请求串起四个分支
用户先说“把这个月的私有报表导出来，顺手发给外部同事”。这句话同时带着对象、权限和副作用三类信号。一个稳定的系统不会直接把它丢给导出工具，第一步先判断有没有权限、导出动作是不是写操作、外发是不是超出边界。这里如果第一步就错，后面所有恢复动作都会跟着跑偏。
继续往下看，系统可能先遇到模型拒绝，因为外发对象不在允许范围内；如果权限勉强通过，也可能在工具层撞到字段校验失败；如果用户随后点了取消，后面的结果要被忽略；如果写操作失败，就只能回退到只读预览或草稿，不能改调另一个更宽的发送工具。

失败链路分流伪代码复制

```js
function routeFailure(event) {
 if (event.refusal) return "stop_and_offer_safe_alternative";
 if (event.cancelled) return "ignore_late_result_and_release";
 if (event.error?.need_user_input) return "ask_for_missing_or_missing_authorization";
 if (event.error?.retryable && event.error.safe_to_retry) return "retry_once";
 if (event.scope === "write" && event.permission === "insufficient") return "fallback_to_readonly_preview";
 return "stop_with_explanation";
}
```

## box info
### 这里最容易错的地方
第一种错法是把拒绝和错误混成一类，于是系统看见边界问题就开始补参数。第二种错法是把取消当成一个普通失败，于是旧结果还会回来继续覆盖状态。第三种错法是把回退做宽，结果从“写失败”变成“换另一个更强的写工具”。这三种错法看起来都像恢复动作，实际上都在把系统往外推。
你可以用一个很短的自检问题来判断自己有没有写对：这一步是停、补问、重试，还是缩小动作？如果说不清，就说明错误对象或者回退策略还没有写实。

# 状态账本 {#ledger}
<!-- tag: ledger -->

## box soft
### 同一条请求在不同状态下，读写点完全不同
这类页面最怕把失败看成一次性事件。更好用的看法是：把请求当成一个有状态的对象。它从 pending 开始，可能走到 refused、errored、cancelled，也可能在回退后变成 previewed，最后才有机会进入 committed。每一步都要写清楚当前状态、谁在读、谁在写、哪些字段已经失效。
如果状态账本写不清，最常见的后果是旧结果晚到时还继续回填，或者错误已经发生，调用方却还在拿 success 路径处理。把状态记住，才能知道这一步应该停、该补问、该清理，还是该改成只读预览。

确认是否需要安全替代或人工说明

调用方不知道该补问、重试还是停下

cancelled

取消标记、清理动作
旧状态污染新状态
确认后续输出是否被忽略

状态转移伪代码复制

```js
function advance(state, event) {
 if (state === "pending" && event.refusal) return "refused";
 if (state === "pending" && event.error) return "errored";
 if (state === "pending" && event.cancelled) return "cancelled";
 if (state === "errored" && event.canPreview) return "previewed";
 if (state === "previewed" && event.confirmed) return "committed";
 return state;
}
```

## box info
### 怎么用这张账本排障
先找当前状态，再找谁在读、谁在写。只要发现状态已经进入 cancelled，就不要再把晚到结果当成当前结果；只要发现状态是 refused，就不要继续补参数；只要发现状态是 previewed，就别把它当成已经提交的写入。状态一旦写清，很多“看起来像模型问题”的症状就会立刻变成分支问题。
这也是这一页最想让你带走的东西：失败一组可追踪的状态迁移。能追状态，才能追恢复。

**你现在应该能做什么：**拿到一个请求状态，先说出它该读什么、该写什么、下一步该停、补问还是回退。

# 结果交接 {#handoff}
<!-- tag: handoff -->

## box soft
### 晚到结果先过状态门
当工具完成得晚，或者用户先取消再返回，宿主程序要先看状态，不要先看内容。内容可能是对的，时机已经错了。审批、检索、导出这几类任务都很容易出现这个问题：结果返回的时候，当前会话已经换了目标，或者当前请求已经不需要了。
结果交接最容易漏掉的三种情况是：取消之后的晚到结果、拒绝之后仍想回填的结果、错误后还没补齐输入就继续提交的结果。只要状态门没守住，系统就会把旧任务当新任务，把草稿当提交，把拒绝当成功。

结果交接伪代码复制

```js
function handoff(result, state) {
 if (state === "cancelled") return "drop_late_result";
 if (state === "refused") return "offer_safe_alternative";
 if (result.error?.need_user_input) return "ask_user";
 if (result.error?.retryable) return "retry_once";
 if (state === "previewed") return "wait_for_confirm";
 return "commit";
}
```

## box info
### 故障后果
最危险的后果失败后仍然继续写回。那会把取消变成成功，把拒绝变成误执行，把草稿变成提交。状态门一旦守不住，回写就会把本来已经结束的动作重新拉活。
因此你在做交接时，先判断状态，再判断内容，最后才决定要不要显示、重试或回退。这个顺序一旦反过来，系统就很容易把旧结果错当成新结果。

**你现在应该能做什么：**拿到一个晚到结果，先按状态决定丢弃、补问、重试还是提交。

# 取消 {#cancel}
<!-- tag: lifecycle -->

## box soft
### 取消：停止继续
用户关掉页面、主动中断长任务，或者宿主程序决定不再等待，都属于取消。取消最重要的是生命周期收束：把还在跑的工作停下来，把资源释放掉，把后续回写也一并截断。否则旧结果晚到以后，前端和状态机都会被旧消息污染。

# 回退 {#fallback}
<!-- tag: fallback -->
<!-- layout: grid2 -->

## box soft
### 回退先缩小动作
好的回退会把动作缩小。写操作回退到草稿或预览，自动执行回退到人工确认，完整结果回退到部分结果。回退要先问是否会改变外部状态，再问是否需要重新授权，最后才考虑能不能继续。

## box warn
### 坏回退的样子
权限失败以后，系统却去调用另一个更宽的工具；或者本来应该只读，结果换成了另一个可能写入的工具。这样的“回退”只是在扩大风险。

不能扩大外部状态变化


权限不足
自动重试同样的动作

确认权限边界是否变化


长任务被取消
继续等旧结果
忽略晚到结果，重新开始一个新请求
不要让旧状态覆盖新状态

# 案例 {#cases}
<!-- tag: cases -->
<!-- layout: grid2 -->

## box soft
### 案例 1：参数合法，但语义无效
用户想查“明天的订单状态”，输入字段都填对了，但日期格式和业务日历冲突。这里属于工具错误，应该返回字段路径和可修正提示。

## box info
### 案例 2：高风险请求被拒绝
用户想让系统生成一个会直接修改私有资源的动作链。这里的正确做法是拒绝并提供安全替代，再停止继续切换工具。

## box soft
### 案例 3：长任务被取消
用户在检索过程中点了取消。系统要把正在跑的任务停住，后面任何晚到结果都应忽略，不再回填到当前会话。

## box info
### 案例 4：回退到只读预览
写操作因为权限不足失败。系统不该再找一个更强的发送工具，而应回退到预览草稿、说明权限差异，必要时让用户重新授权。

<!-- layout: grid2 -->

## box soft
### 和上一单元怎么接
上一单元讲的是工具定义、结果分工和协议边界，这一页往下走一步：工具结果不仅要能返回，还要能说明失败后的动作。`content` 负责给人看，`structuredContent` 负责给机器接，错误对象负责告诉调用方下一步怎么走。

## box info
### 和下一单元怎么接
下一单元会把这些分支放进评测样本。那时你要检查的不只是“能不能成功调用”，还要检查“会不会拒绝不该做的事”“会不会在取消后停止”“会不会在失败时收窄动作”。

# 常见误区 {#mistakes}
<!-- tag: fix -->

误区
为什么会坏
修法


把所有失败都叫错误
拒绝、取消、错误和回退会被混成一层
先分支，再处理


把拒绝写进普通成功对象
调用方会误以为还能继续执行
让拒绝单独成分支


把取消当成普通失败重试
旧请求可能继续占资源，晚到结果还会污染状态
停止处理并忽略旧结果


把回退理解成“再换一个更强工具”
动作范围会越滚越大
把回退缩到更窄、更慢或更需要确认的动作


错误对象只给一个 failed 字段
调用方不知道该重试、补问还是停下
补齐 `type`、`field`、`retryable`、`need_user_input`

**四个先查的问题：**它是拒绝、错误、取消还是回退？这一步该停、补问、重试还是收窄？会不会改变外部状态？晚到结果会不会污染当前状态？

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 怎么答才算会
这页的练习看的是你能不能先分层，再选下一步。答案尽量按“这是什么分支、先看哪一层、下一步是什么”来写。

## box info
### 检查重点
先分清 refusal、error、cancel 和 fallback，再看错误对象里的字段是否足以指导调用方。开放题至少要说清分支、原因和动作，不要只写“重试”或“拒绝”。

<!-- end-practice-intro -->

<!-- practice-group: 第一组：分支识别 -->
<!-- quick-review: 快速复盘 拒绝看边界，错误看字段，取消看生命周期，回退看动作范围。 -->

### 1_branch_main choice

**Q:** 哪一类分支通常应该直接停止执行？

- [ ] . 工具参数缺少一个字段。
- [ ] . 用户关闭了当前请求。
- [ ] . 模型或策略明确拒绝继续执行。
- [ ] . 下游临时超时。

Answer: . 模型或策略明确拒绝继续执行。

### 2_cancel_truth true/false

**Q:** 取消是不是一种普通工具错误？

- [ ] 对
- [ ] 错

Answer: false
Explain: 取消属于生命周期控制，核心是停止继续做无用功。

### 3_cancel_word choice

**Q:** MCP 取消通知常见的消息名是什么？

- [ ] tools/cancelled
- [ ] notifications/cancelled
- [ ] response/cancelled
- [ ] cancel/notice

Answer: notifications/cancelled

### 4_branch_sort checklist

**Q:** 把四种症状归类

- [ ] 模型明确说“不能继续”。
- [ ] 参数合法但字段路径缺失。
- [ ] 用户点了取消，后续结果不该回填。
- [ ] 写操作失败后回到只读预览。

Explain: 四项都属于正确分类。合格答案要能说出每一类先看哪一层：拒绝看模型或策略，错误看工具和业务，取消看生命周期，回退看宿主策略。把四种症状都叫“错误”是这道题的反面答案。

<!-- end-practice-group -->

<!-- practice-group: 第二组：错误对象 -->
<!-- quick-review: 快速复盘 调用方真正需要的是下一步，不只是一个 failed 标记。 -->

### 5_error_field choice

**Q:** 哪一个字段最直接告诉调用方该补哪个值？

- [ ] retryable
- [ ] field
- [ ] safe_to_retry
- [ ] suggested_next_step

Answer: field

### 6_error_object checklist

**Q:** 错误对象应该至少包含哪些字段？

- [ ] 错误类型 type
- [ ] 字段路径 field
- [ ] 是否可重试 retryable
- [ ] 是否需要用户输入 need_user_input
- [ ] 一段宽泛 summary 就够了

Explain: 前四项是合格字段，最后一项不够。好的答案要说明：字段不是装饰，而是为了让调用方知道这次失败能不能重试、要不要追问、该停在哪里。只给一段 summary 会让后续动作继续猜。

### 7_error_fill synthesis

**Q:** 补全一个最小错误对象

Explain: 合格答案至少要包含错误类型、出错字段、是否可重试、是否需要用户输入和下一步建议。更好的答案会说明：如果是权限问题，就该停并请求授权；如果是缺字段，就该补问，而不是盲目重试。

<!-- end-practice-group -->

<!-- practice-group: 第三组：取消与回退 -->
<!-- quick-review: 快速复盘 取消是停下，回退是缩小，不要继续放大动作。 -->

### 8_fallback_truth true/false

**Q:** 回退是不是“再试一个更强工具”？

- [ ] 对
- [ ] 错

Answer: false
Explain: 回退应该收窄动作范围。

### 9_cancel_action choice

**Q:** 取消后最该做什么？

- [ ] . 继续等旧结果，看看能不能回来。
- [ ] . 释放资源，并忽略晚到结果。
- [ ] . 立刻换一个更强工具重跑。
- [ ] . 把取消当成普通成功继续回填。


### 10_fallback_case scenario

**Q:** 权限不足时如何回退？

Explain: 合格答案要先指出坏回退是“换一个更强的写工具”。好回退应该缩到只读预览、草稿、授权确认或公开资料搜索。更好的答案会说明：只要边界没变宽，就不能把回退变成另一次写入。

<!-- end-practice-group -->

<!-- practice-group: 第四组：迁移到评测 -->
<!-- quick-review: 快速复盘 评测要测拒绝、错误、取消和回退，不只测成功。 -->

### 11_bfcl_main choice

**Q:** BFCL 风格评测最应该包含什么？

- [ ] . 只测一次成功调用。
- [ ] . 工具选择、参数、拒绝、取消、多步和异常样本。
- [ ] . 只看 UI 文案。
- [ ] . 只看日志是否漂亮。

Answer: . 工具选择、参数、拒绝、取消、多步和异常样本。

### 12_eval_set checklist

**Q:** 哪些样本应该进你的失败路径回归集？

- [ ] 模型明确拒绝的样本。
- [ ] 参数缺字段、缺值或值域错误的样本。
- [ ] 用户取消后，旧结果晚到的样本。
- [ ] 写操作失败后回到只读预览的样本。
- [ ] 截图就足够的样本。

Explain: 前四项都应该进入回归集。第五项不够，因为成功截图只会掩盖边界问题。合格答案要说清楚：你测的是失败是否可分流、可恢复、可停下，而不是单点成功。

### 13_open_review synthesis

**Q:** 为什么失败路径必须进入契约和评测？

Explain: 合格答案要指出：失败路径决定系统是否安全、是否可恢复、是否会越权。没有契约，调用方只能猜；没有评测，问题只会在真实用户那里暴露。更好的答案会提到，拒绝、错误、取消和回退分别需要不同的字段和样本，不能合成一个统称。
