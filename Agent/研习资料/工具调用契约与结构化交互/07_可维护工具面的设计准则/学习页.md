# 术语账本 {#terms}
<!-- tag: terms -->

容易混用的说法
本页怎么理解
如果混在一起会怎样
先记住什么


工具面
模型可见、可选、可校验、可治理的动作集合。
把后端所有函数都摊给模型，选择面会过大。
先暴露动作，再暴露实现。


schema
结构、类型、枚举、必填项和局部约束。
把跨字段规则和权限规则误塞进结构定义里。
schema 管结构，不管全部业务。


返回值
给下一步继续消费的结果，不是执行日志。
调试信息和业务数据混在一起，后续步骤更难读。
结构要稳定，字段要少而清楚。


MCP 工具
被协议发现、描述、约束、调用的工具。
把普通远程接口简单包一层就当成协议工具。
工具面要先可发现，再可执行。


BFCL 分数
一组评测结果，不是上线结论。
只看总分，不看失败是选错、填错还是不该调。
拿它拆维度，不拿它代替回归集。


工具描述
用来区分适用边界的短文本。
把描述写成提示词补丁，越写越长，越长越乱。
描述要短、准、可判别。

# 核心机制 {#mechanism}
<!-- tag: mechanism -->

## box soft
### 先看一份最小工具定义
下面这个例子只保留搜索动作。它足够小，能看出 name、description、schema 和注解分别在约束什么，也足够真实，不会把问题缩成玩具。

最小工具定义复制

```json
{
 "name": "search_notes",
 "description": "只读。搜索知识库中的笔记，返回最相关的 10 条结果。",
 "inputSchema": {
 "type": "object",
 "properties": {
 "query": { "type": "string" },
 "scope": {
 "type": "string",
 "enum": ["project", "research", "knowledge"]
 },
 "limit": { "type": "integer", "minimum": 1, "maximum": 20 }
 },
 "required": ["query", "scope", "limit"],
 "additionalProperties": false
 },
 "annotations": {
 "readOnlyHint": true
 }
}
```

字段
它读什么
它写什么 / 约束什么
错了会怎样


`name`
动作和对象的最短标识。
写入一个可被模型稳定记住的选择点。
近义词太多，模型会互相误选。


`description`
适用边界和不适用边界。
写入判别信号，而不是宣传语。
描述过长，反而遮住差异。


`inputSchema`
参数结构、类型、枚举、范围。
把可填字段固定住，减少模糊输入。
执行前校验不了，只能到运行时才暴露错误。


`annotations`
读写属性、风险、行为线索。
写入权限门槛和确认策略。
写操作和读操作混在一起，安全边界会松。


`output`
下一步要消费的事实结果。
写入稳定返回，而不是一大段日志。
后续步骤难取字段，回放也难做。


`version`
旧调用还能不能继续用。
写入兼容期、迁移说明和回滚线索。
老样本和老提示词突然失效。

$$
 R = \frac{S \times D \times V}{C}
 $$
 `S` 是同义工具数，`D` 是暴露面，`V` 是版本漂移强度，`C` 是回归覆盖密度。这个式子不是正式定理，只是提醒你：工具越像、越多、越常改，而回归越少，维护风险就越高。

# 例子 {#example}
<!-- tag: example -->

下面用一组常见的知识库动作做例子。先别管实现，只看这组名字有没有给模型足够的判别信号。

工具
推荐动作
理由
如果不这样做


`search_notes`
保留
只读搜索，动作清楚，范围也清楚。
如果再起一个差不多的名字，模型会开始猜。


`find_note`
并入 `search_notes`
如果只是同义词，没有新的边界，就不该重复暴露。
同义按钮越多，误选越多。


`share_note`
单独保留
外发动作有副作用，不能和搜索动作混成一类。
模型会把“查找”和“发送”误当成相近动作。


`archive_note`
单独保留并加确认
归档会改变资源状态，最好显性标出风险。
用户以为在整理，实际上在改变可见性。

## box info
判断标准很简单：如果两个动作只是同义词，合并；如果差别在读写、外发或权限，就分开并把边界写进描述和注解。

# 失败链路 {#trace-lab}
<!-- tag: trace-lab -->

这一段把同一组工具放进四种真实情形里看：搜索、修改、外发、归档。你会看到，工具面一旦写得不稳，问题模型会在最该谨慎的地方做出错误动作。

情形
应该暴露什么
应该拦住什么
错误时的症状
怎么修


搜索笔记
只读工具、范围、limit 上限。
超大 limit、无关范围、空 query。
结果太多，后续步骤抓不到重点。
收紧 schema，把默认范围写清楚。


修改笔记
明确写操作、目标 id、确认门槛。
不带 id 的模糊修改、没确认就执行。
模型说的是“改一下”，系统做的是“改错对象”。
把写操作拆开，先查后改。


外发分享
收件人、权限级别、是否公开。
把外发动作藏在“整理”里。
用户以为只是在归档，实际上已经发出去了。
给外发动作单独命名，加确认。


归档资源
资源状态、是否可逆、回滚路径。
和删除混在一起。
后面找不到，回不来，也不知道谁改的。
把归档和删除分开，写回滚说明。

## box soft
### 一条最小操作链
先搜索，再决定是否修改，再看是否分享，最后才考虑归档。这个顺序不是装饰，它是在把高风险动作放到后面，让低风险动作先把上下文收齐。

操作顺序示意复制

```text
search_notes -> inspect_result -> decide_write -> confirm_if_needed -> share_note -> archive_note
```

$$
 \text{风险顺序} = \text{读} \rightarrow \text{轻写} \rightarrow \text{外发} \rightarrow \text{归档}
 $$
 这一条工作顺序：先读后写，先低风险后高风险。顺序倒过来，错误会更早发生，也更难回退。

# 状态台账 {#ledger}
<!-- tag: ledger -->

工具面真正难维护的地方，是很多状态不在一处。名字、参数、返回、权限、版本和评测样本都在读同一件事，但它们写回的是不同地方。

对象
读什么
写什么
什么时候算有效
失配症状
下一步检查


name / description
模型如何区分动作。
选择面和判别信号。
一眼能分开相近工具。
同义工具互相抢位置。
把两个相近工具写成一句互斥描述。


inputSchema
参数结构、类型、边界。
可校验的输入约束。
执行前就能发现缺参或越界。
到了运行时才报错。
把开放文本改成字段和枚举。


output / structuredContent
下一步要读取的结果。
稳定返回结构。
后续步骤能直接取字段。
返回里混着日志和业务结果。
把日志移出返回结构。


permissions / annotations
动作风险和读写属性。
确认门槛和拦截策略。
写操作不会悄悄发生。
用户以为在看，系统其实在改。
给高风险动作单独加确认。


version / changelog
旧调用和旧样本。
兼容期和迁移说明。
老提示词还能按预期工作。
老样本突然失效。
保留别名、默认值或适配层。


metrics / regression cases
失败归因和历史问题。
回归样本和修复记录。
改完以后能复查是否变好。
问题反复出现，但没人记得。
把这次失败补进回归集。

变更类型
兼容吗
应该补什么


新增可选字段
通常兼容
补一个用到新字段的正例。


新增必填字段
高风险
补旧提示词和缺参样本。


重命名字段
高风险
保留旧别名，补迁移说明。


改默认值
看语义
补边界样本，确认结果没反转。


拆分一个工具成两个
部分兼容
补选择样本，确认模型能分清边界。

# 设计准则 {#principles}
<!-- tag: principles -->

$$
 M = \text{清晰度} + \text{稳定性} + \text{可测性} + \text{可回滚性} - \text{冗余度}
 $$
 这个式子只是在提醒你：名字越清楚、结构越稳定、回归越完整、版本越能退回去，工具面就越好维护；同义动作越多、重复暴露越多，维护成本就越高。

P01最小暴露只暴露当前任务需要的动作。内部函数、临时调试能力和纯实现步骤不要直接进工具面。检查：删掉这个工具后，核心任务还能不能完成？
P02互斥命名名称表达动作和对象，描述表达适用边界。相近工具必须能被一句话区分。检查：两个工具能不能被同一句请求同时触发？
P03强 schema字段、类型、枚举、范围和必填项尽量明确，不要把结构化需求塞进一个 `text` 字段。检查：参数错了能不能在执行前暴露出来？
P04稳定返回返回结构服务后续推理和审计。业务数据、用户提示、错误对象和调试日志要分开。检查：下一步调用能不能稳定读到字段？
P05显性风险只读、写入、删除、外发、权限变更这些动作要在工具面上看得出来。检查：高风险工具有没有确认或授权门槛？
P06错误可恢复错误不只是字符串。要区分参数错、权限错、资源不存在、外部失败、超时和用户取消。检查：模型拿到错误后知道下一步该问什么吗？
P07版本可迁移schema 和语义变更要有兼容期、迁移说明、评测更新和回滚线索。检查：旧提示词和旧样本会不会被突然打断？
P08评测绑定每个重要工具都要有正例、负例、边界值和权限场景。新增工具必须新增评测。检查：有没有覆盖“应该不调用工具”的样本？
P09分层组合复杂流程不要让模型自己拼太多底层动作。能组合的步骤尽量放到执行器里。检查：模型是不是被迫记住一串业务顺序？
P10可观测治理要记录工具选择、参数、结果、错误、耗时和版本。没有这些记录，就很难改出下一版。检查：一次失败能不能回放并归因到具体一层？

# 常见误区 {#mistakes}
<!-- tag: mistakes -->

这些错误看起来都很省事，实际会把工具面慢慢推向不可维护。最麻烦的地方错了以后没人知道该改哪一层。

误区
表面症状
更好的做法


一个万能工具包住所有动作
看起来最省事，实际什么都难校验。
按真实用户意图拆成少量明确动作。


相近工具只改个名字就上线
模型开始在同义词之间猜。
合并同义工具，或者把边界写清楚。


返回里什么都塞
业务数据、日志、提醒混成一团。
把稳定结果和调试信息分开。


只测成功样本
看上去总分不错，实际一改就崩。
补无关请求、缺参、并行、权限和版本样本。


安全只写在描述里
工具还是会被误触发。
把风险写进注解、确认和执行策略。


改 schema 不留迁移说明
旧提示词和老样本突然失效。
保留别名、默认值、兼容期和回滚线索。

# 注册表与治理 {#registry}
<!-- tag: registry -->

工具面要能长期维护，最后一定会落到一张注册表。注册表的作用让维护者能回答：这个动作为什么存在，谁在用，改了会不会坏，坏了怎么退。

字段
为什么需要
维护时看什么


工具名
确认模型如何选择。
是否和其他工具重复或相似。


用途
确认为什么要暴露它。
是不是还服务当前场景。


输入 / 输出
确认结构和消费方式。
字段有没有改名、加默认值、加约束。


风险级别
确认是否需要门槛。
写操作、外发和权限变更有没有单独分层。


版本与负责人
确认变更归谁管。
旧版本有没有退路，谁来确认迁移。


回归编号
确认问题是否被记住。
这次失败有没有补进样本。

## box info
如果一张注册表里只剩工具名和一句宣传语，它就不是治理材料，只是目录页。能维护的工具面，最后都要能回到这张表上检查。

# 审查实操 {#audit-flow}
<!-- tag: audit-flow -->

这一段用一组最容易出问题的动作做实操：`search_notes`、`find_note`、`share_note`、`archive_note`。你要做的顺着这组动作跑一遍“合并、拆分、确认、回归”的判断流程。

步骤
看什么
做什么决定
常见坏法
落地动作


1. 读动作名
是不是同义词或近义词。
合并还是保留。
两个名字像，描述也像。
删掉重复暴露，保留一个主入口。


2. 看描述
边界有没有说清。
描述是否需要改写。
只写“智能”“快速”“一站式”。
补适用场景和不适用场景。


3. 看 schema
字段、枚举、范围、必填项。
是否能提前挡掉错参。
把所有内容塞进一个 text。
把开放文本拆成结构字段。


4. 看风险
读、写、外发、归档、删除。
是否需要确认。
写操作躲在普通按钮后面。
把高风险动作显性化。


5. 看版本
旧调用还能不能继续跑。
兼容还是破坏。
直接删字段、改默认值。
加别名、加适配、加迁移说明。


6. 看回归
失败样本有没有补进来。
要不要补正例和负例。
只测成功路径。
补一条失败、一条恢复、一条边界。

## box soft
### 一份可直接照着做的审查清单
先把 `find_note` 并回 `search_notes`，因为它只是同义重复；再把 `share_note` 单独挑出来，因为它有外发副作用；再把 `archive_note` 加上确认和回滚说明，因为它会改变资源可见性。最后，把这四个动作各补一条回归样本，尤其要补一条“本来不该调用”的负例。

审查结论模板复制

```text
动作名: ____ 
结论: 合并 / 保留 / 拆分 / 加确认
理由: 同义重复 / 读写不同 / 外发风险 / 归档可逆性 / 版本破坏
补充: 描述改写 / schema 收敛 / 权限提示 / 回归样本编号
```

$$
 \text{审查成本} \approx \text{同义动作数} + \text{风险动作数} + \text{版本变更数} - \text{回归覆盖}
 $$
 这不是精确公式，但很接近实际维护感受。动作越重复、风险越高、版本越常改，而回归越少，审查成本就越高。

# 变更单 {#change-case}
<!-- tag: change-case -->

把这些准则放进一个真实变更单里，会更容易看出它们不是口号。假设你现在要把知识库工具面从四个动作改成三个动作：`search_notes`、`share_note`、`archive_note`。这时，`find_note` 只是 `search_notes` 的同义词。

改动项
旧状态
新状态
为什么要改
会影响什么


工具名
`search_notes` 与 `find_note` 并存。
只保留 `search_notes`。
同义暴露会让模型在两个近义按钮之间摇摆。
选择分数、路由稳定性、提示词兼容。


描述
两项都写“查找资料”。
`search_notes` 明确只读搜索，`share_note` 明确外发。
描述越像，模型越容易把动作混成一类。
工具选择、误触发、审查成本。


schema
query 字段只有一个开放文本。
加上 scope、limit、是否公开等枚举。
结构更清楚，执行前能挡掉更多错参。
校验失败、缺参追问、默认值策略。


风险
外发和归档都没显性门槛。
share_note 和 archive_note 单独确认。
高风险动作不能藏在普通搜索后面。
确认流程、权限策略、审计记录。


版本
旧提示词直接调用旧动作名。
保留别名或适配层一段时间。
老样本和老提示词不能突然断路。
迁移说明、回滚路径、样本更新。

## box soft
### 这单改完以后要补什么
第一，删掉 `find_note` 的独立入口，但保留一段兼容期。第二，把 `share_note` 的确认策略写入工具说明，而不是放在补充备注里。第三，给 `archive_note` 补一条能回滚的失败样本，确保“归档后找不到”这类问题不会再次出现。第四，把这次变更写进注册表和回归集，留下编号，后面才能查。

变更单摘要复制

```text
删除同义工具 -> 收敛 schema -> 显性确认高风险动作 -> 写入迁移说明 -> 补回归样本
```

$$
 \text{收益} = \text{误选减少} + \text{校验提前} + \text{风险显性化} + \text{回归可追踪}
 $$
 这条链路真正改变的维护方式：你开始能把一个工具面改动拆成几个可检查步骤，而不是凭感觉改完就算。

# 判定链 {#decision-lab}
<!-- tag: decision-lab -->

如果你手里只有一条判断流程，这一页最想给你的就是下面这条。它不是协议，也不是框架，只是一条在真实审查里很管用的操作顺序：先看同义暴露，再看读写风险，再看版本回归，最后补样本。

工具面判定伪代码复制

```python
def audit_tool_surface(tools):
 decisions = []
 for tool in tools:
 if is_synonym(tool, tools):
 decisions.append((tool.name, "merge", "same action boundary"))
 continue

 if is_write_or_external(tool):
 decisions.append((tool.name, "separate", "needs confirmation"))
 else:
 decisions.append((tool.name, "keep", "read-only or low risk"))

 if schema_is_open_text(tool):
 decisions.append((tool.name, "tighten_schema", "move to fields and enums"))

 if version_breaks_old_calls(tool):
 decisions.append((tool.name, "add_migration", "keep alias and regression cases"))

 return decisions
```

判断点
先问什么
如果答案是“是”
如果答案是“否”
回归怎么补


同义暴露
是不是同一个动作换了两个名字？
合并入口，只保留一个主名称。
继续看风险和结构。
补“误选相近工具”的样本。


读写风险
是不是写、删、外发、归档？
单独命名并加确认。
保持只读或低风险状态。
补“未确认不执行”的样本。


schema 收敛
是不是把很多字段塞进一个 text？
拆成字段、枚举和范围。
保留现有结构。
补缺参和越界样本。


版本兼容
旧调用还能不能继续跑？
加别名、默认值或适配层。
记录为破坏性变更。
补旧提示词和迁移样本。

## box soft
### 这个判定链的价值
它把“我觉得这个工具面有问题”变成了“我知道应该合并哪一个名字、加在哪个确认、补哪类回归”。当你能把判断写成流程，维护就不再靠记性。
