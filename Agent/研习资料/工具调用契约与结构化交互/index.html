工具调用契约与结构化交互

7 个单元 结构化输出 函数调用 MCP BFCL

这门课关注模型和外部系统之间的交互契约：工具如何被描述，参数如何被约束，结果如何被结构化，错误如何被处理，能力如何被评测。

推荐起点：先理解“契约”而不是“函数名列表”，再进入具体结构和评测。

课程总论

工具调用的难点不是让模型输出一个 JSON，而是让交互双方对动作、参数、结果和风险有共同约束。

Overview
核心问题

模型生成的工具意图必须穿过接口描述、参数校验、权限边界、执行结果和错误反馈，才能成为可靠行动。

学完能做

能设计一个工具契约，说明 schema、返回结构、错误语义、授权提示和评测用例如何配合。

课程位置

承接运行承载工程，向可靠性工程和外部协议集成延伸。

基于哪些资料

资料选择集中在公开接口规范和评测基准。

Sources
OpenAI Function Calling

官方指南用于理解工具定义、参数生成、工具选择和执行回填。

OpenAI Structured Outputs

官方指南用于区分结构化响应和工具参数，并建立 schema 约束意识。

工具文档用于补充托管工具、本地运行工具、函数工具和工具错误处理。

MCP 规范

规范入口、Tools 和 Schema Reference 用于理解跨系统工具契约。

BFCL

SDK 更新记录

发布与变更说明用于关注结构化输出、工具执行和协议集成的行为变化。

建议学习对象

适合正在把工具调用接进业务、代码、数据或自动化系统的人。

Audience
接口设计者

需要把业务动作设计成模型可理解、系统可校验的工具。

模型应用开发者

已经会写工具调用示例，但经常遇到参数错、工具选错、返回难解析。

评测负责人

需要把工具调用质量拆成可测用例，而不是只看一次演示是否成功。

前置知识

先具备最基本的 API 和数据结构常识即可。

Before
API 基础

知道函数、参数、返回值、错误码、鉴权和幂等请求是什么意思。

数据结构

能读懂 JSON 对象、数组、枚举、必填字段和嵌套字段。

模型应用

知道模型会根据上下文生成工具调用意图，但真实执行发生在外部系统。

推荐路径

从“接口像合同”开始，再进入 schema、协议、错误和评测。

Roadmap
阶段一｜先立契约

把工具调用看成双方约定：名称、用途、参数、权限、返回值、错误和审计都要明确。

对应 01-02

对应 03
阶段三｜接入协议

把本地函数扩展到标准化工具服务器，理解资源、工具、传输、能力协商和安全提示。

对应 04-05
阶段四｜验证质量

7 Units
01

工具调用的契约思维

存在理由：如果只把工具当成“模型可以点的按钮”，就会忽略参数约束、执行权限、错误反馈和审计记录。本单元先建立接口合同意识。

打开学习页
02

函数调用基础：描述、选择与分发

存在理由：工具名称、描述、参数 schema 和运行分发共同决定模型能否选对工具、填对参数、等待正确结果。

打开学习页
03

结构化输出与 JSON Schema

存在理由：结构化响应和工具参数都依赖 schema，但用途不同。本单元帮助区分“回答结构化”和“行动结构化”。

打开学习页
04

MCP 工具规范与协议边界

存在理由：跨系统工具需要标准化协议来约定服务器能力、工具列表、资源、传输和授权边界，不能只靠本地函数包装。

打开学习页
05

错误、拒绝、取消与回退

存在理由：可靠工具契约必须说明执行失败、参数不合法、用户未授权、模型拒绝、请求取消时系统如何反馈和恢复。

打开学习页
06

从 BFCL 看工具调用评测

存在理由：工具调用能力需要拆成可观测指标：工具选择、参数准确、多轮一致、并行调用和真实交互，而不是看一次成功截图。

打开学习页
07

可维护工具面的设计准则

存在理由：工具越多，描述冲突、schema 漂移、权限混乱和上下文成本越明显。本单元收束命名、分组、版本和最小暴露策略。

打开学习页

资料边界

课程只讲工具交互契约，不替代具体业务接口设计。

Scope

本课程不覆盖所有 JSON Schema 细节、所有 MCP 实现库、所有排行榜模型，也不讨论模型训练或微调。

课程顺序基于公开规范和评测资料归纳：先契约，再结构，再协议，再错误和评测。后续单元制作时应继续核对官方文档的最新字段与行为。

所有课程单元入口已链接到真实存在的学习页。
