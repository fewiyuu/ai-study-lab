> [!question] 常见问题速查
> 1. S01 和 S02 的工具调用核心差别是什么？
> 2. `block.type` 和 `block.name` 分别判断什么？
> 3. handler map 的键和值是什么？
> 4. `read_file` 为什么比 Bash 读文件更可控？
> 5. `safe_path` 要防什么？
> 6. 新增一个工具要同步改哪几处？

> [!abstract] 核心主线
> S01 只有 Bash，所以工具调用可以硬编码到 `run_bash`。S02 有多个工具，必须用工具名分发：先确认 block 是 `tool_use`，再读取 `block.name`，用 `handlers.get(block.name)` 找到本地函数，把 `block.input` 作为参数传进去。`read_file` 这样的专用工具把能力收窄，配合 `safe_path` 可以限制访问范围，比所有事情都交给 Bash 更可控。

## 概念精讲

### 从硬编码到分发

S01：

```python
if block.type == "tool_use":
    output = run_bash(block.input["command"])
```

S02：

```python
handlers = {
    "bash": run_bash,
    "read_file": run_read_file,
    "write_file": run_write_file,
}

if block.type == "tool_use":
    handler = handlers.get(block.name)
    output = handler(**block.input)
```

判断句：`type=tool_use` 只说明“这是工具调用”，`name=read_file` 才说明“调用哪个工具”。

### handler map 是工具名到函数的契约

handler map 的键是工具名，值是本地函数。工具 schema 中暴露的名字必须和 handler map 中注册的名字一致。否则模型请求 `read_file`，本地却找不到对应函数。

找不到 handler 时，不应该直接调用 `None`。更稳妥的处理是返回明确错误，让模型知道这个工具不可用。

### 专用工具让能力收窄

`read_file` 只读取文件。它的 input 需要 `path`，本地函数先校验路径，再读取文本。

```python
def run_read_file(path: str) -> str:
    safe = safe_path(path)
    return safe.read_text(encoding="utf-8")
```

Bash 也能读文件，但 Bash 能做的事情太多，不容易限制。专用工具的输入字段更清楚，返回格式更稳定，安全边界更容易审计。

## 最小可操作样例

```python
from pathlib import Path

WORKSPACE = Path.cwd().resolve()

def safe_path(path: str) -> Path:
    candidate = (WORKSPACE / path).resolve()
    if not str(candidate).startswith(str(WORKSPACE)):
        raise ValueError("Path is outside workspace")
    return candidate

def run_read_file(path: str) -> str:
    return safe_path(path).read_text(encoding="utf-8")

handlers = {
    "read_file": run_read_file,
}

handler = handlers.get(block.name)
if handler is None:
    output = f"Tool {block.name} is unavailable."
else:
    output = handler(**block.input)
```

## 关键术语

| 术语 | 含义 | 容易混淆点 |
|---|---|---|
| `block.type` | block 类型 | 不能决定具体工具 |
| `block.name` | 工具名 | 要和 handler map 的键一致 |
| handler map | 工具名到函数的映射 | 不是工具 schema |
| `block.input` | 工具参数 | 需要本地校验 |
| `safe_path` | 路径安全检查 | 不是可选装饰 |

## 应用与迁移问答

> [!question] 为什么不用 Bash 包打天下？
> Bash 太通用，命令字符串难校验，危险操作多，跨平台差异也大。专用工具能力更窄，更容易限制输入和审计输出。

> [!question] `safe_path` 主要防什么？
> 防路径穿越。例如 `../../secret.txt` 可能访问工作区外的文件。`safe_path` 应把路径解析到绝对路径后，确认它仍然在工作区内。

> [!question] 专用工具是不是越多越好？
> 不是。专用工具应该对应稳定、常用、边界清楚的能力。过细的工具会增加 schema、handler、测试和维护成本。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 只看 `block.type`，忘记读取 `block.name`。
> - schema 里的工具名和 handler map 里的键不一致。
> - 找不到 handler 时直接调用 `None`。
> - 相信模型给出的 path，不做 `safe_path`。
> - 以为专用工具只是 Bash 的语法糖，没有看到安全边界变化。

## 费曼解释润色版

> [!quote]
> S01 只有一个 Bash 工具，所以代码可以看到工具调用就直接跑 `run_bash`。S02 有多个工具，就不能再硬编码。模型返回的 block 里，`type` 说明这是工具调用，`name` 说明它想调用哪个工具。程序用 `handlers.get(block.name)` 找到对应的本地函数，再把 `block.input` 传进去。以 `read_file` 为例，模型给出 path，本地函数先用 `safe_path` 限制路径，再读取文件内容。这样做比全部交给 Bash 更清楚，因为每个专用工具的能力、参数和安全边界都更容易检查。最后，工具输出仍然要写回 `tool_result`，让模型知道读取结果。

## 易错卡片

> [!warning] 误区：`type=tool_use` 已经说明工具名
> 正确说法：`type` 只说明是工具调用，具体工具看 `name`。

> [!warning] 误区：handler map 只是代码风格问题
> 正确说法：它是多工具系统的路由表。

> [!warning] 误区：schema 会保证 path 安全
> 正确说法：schema 只能描述字段，本地 handler 必须校验路径。

> [!warning] 误区：Bash 能做就不用专用工具
> 正确说法：专用工具让能力更窄，输入和输出更可控。
