> [!question] 常见问题速查
> 1. todo 工具和普通自然语言计划有什么不同？
> 2. todo item 为什么需要 `id`、`text`、`status`？
> 3. 为什么同一时间最多一个 `in_progress`？
> 4. 为什么计划项要有数量上限？
> 5. `runs_since_todo` 是做什么的？
> 6. todo 更新为什么要写回 messages？

> [!abstract] 核心主线
> todo 工具把计划变成显式、结构化、可追踪的任务状态。模型通过工具 schema 知道 todo 需要 `items`，每个 item 有 `id`、`text`、`status`。本地 update 函数校验这些字段，渲染出计划，并把计划作为工具结果写回 messages。后续模型每一轮都能看到当前计划，知道哪一步完成、哪一步正在做、哪一步还没开始。`runs_since_todo` 则用于提醒模型定期更新计划，避免做着做着跑偏。

## 概念精讲

### todo 是可见任务状态

todo 不是隐藏思维链。它不是让模型暴露全部推理，而是把需要用户和程序共同追踪的任务状态写出来。复杂任务中，todo 能让当前行动更可检查。

判断句：todo 的价值是“可见、可更新、可校验”，不是“让模型想得更长”。

### schema 让模型知道输入结构

todo 工具 schema 告诉模型：参数里需要 `items`，每个 item 需要 `id`、`text`、`status`。

```json
{
  "items": [
    {"id": "1", "text": "查看当前目录", "status": "in_progress"},
    {"id": "2", "text": "读取 requirements.txt", "status": "pending"}
  ]
}
```

如果没有 schema，模型很可能只返回一段自然语言计划，本地函数无法稳定解析。

### update 要校验

todo update 函数不应该照单全收。它至少要检查：

- `text` 不能为空。
- `status` 必须属于允许集合。
- 同一时间最多一个 `in_progress`。
- 计划项不能超过上限。
- id 应该稳定、可追踪。

## 最小可操作样例

```python
allowed = {"pending", "in_progress", "completed"}
validated = []

for item in items:
    task_id = item.get("id")
    text = item.get("text", "").strip()
    status = item.get("status")

    if not text:
        raise ValueError("Todo text cannot be empty")
    if status not in allowed:
        raise ValueError(f"Unknown status: {status}")

    validated.append({"id": task_id, "text": text, "status": status})

running = [item for item in validated if item["status"] == "in_progress"]
if len(running) > 1:
    raise ValueError("Only one task can be in progress")
```

渲染示例：

```text
> [ ] 1 查看当前目录
  [ ] 2 读取 requirements.txt
  [ ] 3 总结依赖用途
```

## 关键术语

| 术语 | 含义 | 容易混淆点 |
|---|---|---|
| todo tool | 更新计划的工具 | 不是执行任务的工具 |
| `items` | 计划项列表 | 不是随便一段文本 |
| `status` | 任务状态 | 不能随意发明 |
| `in_progress` | 当前正在做的任务 | 同一时间最多一个 |
| `runs_since_todo` | 距离上次更新 todo 的轮数 | 用于提醒模型同步计划 |

## 判断与行动清单

> [!todo] todo update 检查
> - [ ] items 是列表。
> - [ ] 每项有 id、text、status。
> - [ ] text 不为空。
> - [ ] status 在允许集合内。
> - [ ] 最多一个 `in_progress`。
> - [ ] 总数不超过上限。
> - [ ] 渲染结果能被用户读懂。
> - [ ] 工具结果写回 messages。

## 应用与迁移问答

> [!question] 为什么同一时间最多一个 `in_progress`？
> 这样能保持当前焦点清楚。多个进行中任务会让模型看起来并行推进，实际可能每条线都没收束。

> [!question] 为什么限制计划项数量？
> 太多任务会让计划本身变得脆弱。模型容易陷入过细步骤，用户也难以判断真实进度。

> [!question] `runs_since_todo` 为什么有用？
> 它记录模型多久没更新计划。连续几轮调用其他工具后，系统会提醒模型回到 todo，同步 completed 和 in_progress 状态。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把 todo 当成模型的隐藏思维链。
> - 只生成计划，不把计划写回 messages。
> - 允许多个 `in_progress`。
> - 计划拆得太细，超过上限。
> - 连续执行工具后忘记更新任务状态。

## 费曼解释润色版

> [!quote]
> todo 工具的作用是把 Agent 的计划变成可见状态。模型通过 schema 知道它要返回一个 items 列表，每项包含 id、text 和 status。本地 update 函数会校验这些字段，比如 text 不能为空，status 只能是允许值，而且同一时间最多一个任务处于 in_progress。校验通过后，计划会被渲染成用户能读懂的格式，并作为工具结果写回 messages。后续模型再次请求时，就能看到当前计划，而不是凭记忆继续行动。如果模型连续几轮没有更新 todo，`runs_since_todo` 会触发提醒，让模型同步任务状态。这样 Agent 不容易做着做着偏离原计划。

## 易错卡片

> [!warning] 误区：todo 是隐藏思维链
> 正确说法：todo 是显式任务状态，用于协作和控制。

> [!warning] 误区：计划越细越好
> 正确说法：过细计划会脆弱，应该限制条目数量。

> [!warning] 误区：多个任务可以同时 `in_progress`
> 正确说法：当前焦点应保持唯一。

> [!warning] 误区：todo 更新只是显示给用户看
> 正确说法：它还会写回 messages，约束模型下一步行动。
