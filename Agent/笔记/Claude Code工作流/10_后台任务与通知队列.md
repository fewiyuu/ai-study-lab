> [!question] 常见问题速查
> 1. 为什么慢操作要放到后台？
> 2. `BackgroundManager` 里 `tasks`、`notification_q`、`lock` 分别做什么？
> 3. `background_run` 如何启动任务并立刻返回？
> 4. 后台线程完成后如何把结果交回主 Agent？
> 5. 为什么使用 pull 队列，而不是直接回调？
> 6. `daemon=True` 的作用和代价是什么？

> [!abstract] 核心主线
> 同步工具调用会让 Agent 等慢命令结束后才能继续响应。后台任务机制用 `BackgroundManager` 管理任务状态和通知队列：`background_run(command)` 生成 task id，把任务记为 running，并启动 daemon 线程执行命令，然后立刻返回；后台线程用 `subprocess.run` 执行完后，更新 `tasks` 状态，并把完成通知放入 `notification_q`；下一轮 Agent loop 开始时，主线程 drain 通知队列，把后台结果作为 `Background results` 注入 messages。这样慢任务不会卡住主循环。

## 概念精讲

### 慢操作不该卡住主循环

同步工具调用的流程是：模型要求执行命令，本地执行，等待结束，再返回结果。如果命令要跑几分钟，主 Agent 就只能空等。

后台任务的做法是“启动后先返回，完成后再通知”。用户不必等慢命令结束才能继续交互。

判断句：后台任务不是让结果消失，而是把“等待结果”从主循环里移出去。

## BackgroundManager

初始化时有三个核心对象：

```python
class BackgroundManager:
    def __init__(self):
        self.tasks = {}
        self.notification_q = []
        self.lock = threading.Lock()
```

| 对象 | 作用 |
|---|---|
| `tasks` | 按 task id 保存 command、status、result |
| `notification_q` | 保存已经完成、等待注入 messages 的通知 |
| `lock` | 保护共享状态，避免线程同时读写造成竞态 |

`tasks` 和 `notification_q` 不一样。任务一启动就进入 `tasks`；只有后台线程完成后，结果才会进入通知队列。

## 启动后台任务

`background_run` 的流程：

1. 生成 task id。
2. 在 `tasks` 中登记 running 状态。
3. 创建后台线程。
4. 线程 target 指向 `execute(task_id, command)`。
5. `daemon=True`。
6. 启动线程并立刻返回 started 消息。

```python
def background_run(command):
    task_id = str(uuid.uuid4())[:8]
    self.tasks[task_id] = {
        "command": command,
        "status": "running",
        "result": None,
    }

    thread = threading.Thread(
        target=self.execute,
        args=(task_id, command),
        daemon=True,
    )
    thread.start()

    return f"Background task {task_id} started"
```

`daemon=True` 的作用是：主程序退出时，守护线程会自动终止，避免进程被后台线程挂住。代价是主程序退出太快时，后台任务可能没有机会收尾。

## 后台线程执行与通知

后台线程用 `subprocess.run` 执行命令。完成后更新状态，并把通知放进队列。

```python
def execute(self, task_id, command):
    try:
        result = subprocess.run(
            command,
            shell=True,
            capture_output=True,
            text=True,
            timeout=300,
        )
        status = "completed"
        output = result.stdout or result.stderr
    except Exception as exc:
        status = "failed"
        output = str(exc)

    with self.lock:
        self.tasks[task_id]["status"] = status
        self.tasks[task_id]["result"] = output
        self.notification_q.append({
            "task_id": task_id,
            "status": status,
            "result": output,
        })
```

这里的 lock 很关键。主线程可能正在 drain 队列，后台线程也可能同时 append 通知；不加锁会出现共享状态不一致。

## 主循环 drain 通知队列

每轮 Agent loop 开始时，会先 drain 通知队列。

```python
def drain_notifications(self):
    with self.lock:
        notices = list(self.notification_q)
        self.notification_q.clear()
    return notices
```

drain 的含义是“取出并清空”。如果只取不清，同一个后台结果会重复进入 messages。

主循环拿到通知后，会把它格式化成 `Background results`，追加到 messages。这样模型下一轮请求时就能知道后台任务完成情况。

## pull 而不是 push

这套设计选择 pull：

- 后台线程只负责把结果放入 `notification_q`。
- 主 Agent 每轮开始时主动拉取通知。
- 后台线程不需要理解 Agent loop 的内部结构。
- 主循环不会被后台线程任意打断。

push 或回调也能做，但耦合更强：后台线程要知道把结果推给谁、什么时候推、会不会打断当前对话。队列式 pull 更简单。

## `check_background`

`check_background` 是主动查询工具：

- 不传 task id：列出所有后台任务状态。
- 传 task id：查看某个任务的 command、status、result。

这和通知队列互补：通知队列用于“完成时提醒”，`check_background` 用于“用户或模型主动查状态”。

## 应用与迁移问答

> [!question] 什么任务适合同步执行？
> 很快、结果马上需要用于下一步推理的任务，比如列目录、读一个小文件。

> [!question] 什么任务适合后台执行？
> 下载、长测试、构建、等待服务启动、批量处理等耗时任务。

> [!question] 为什么通知可以加 type？
> 生产系统里通知不只代表任务完成，也可能是失败、超时、警告、进度更新。加 `type` 后主循环可以按类型处理。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把后台任务理解成“没有结果”。
> - 分不清 `tasks` 和 `notification_q`。
> - 忘记后台线程完成后才写入通知队列。
> - drain 队列后不清空，导致结果重复注入。
> - 共享状态不加 lock。
> - 以为 daemon 线程没有代价。
> - 忽略 `subprocess.run(shell=True)` 的命令安全风险。

## 费曼解释润色版

> [!quote]
> 后台任务机制是为了避免慢命令卡住 Agent 主循环。`BackgroundManager` 用 `tasks` 字典保存每个后台任务的 command、status 和 result，用 `notification_q` 保存已经完成但还没注入 messages 的通知。模型调用 `background_run` 时，本地生成 task id，把任务记为 running，并启动一个 daemon 线程执行命令，然后立刻返回 started。后台线程内部用 `subprocess.run` 执行命令，完成后把状态改成 completed 或 failed，并把结果放入通知队列。下一轮 Agent loop 开始时，主线程会 drain 这个队列，把后台结果格式化成 `Background results` 加入 messages。这样模型下一轮就知道后台任务完成了什么。这里选择 pull 队列，是为了让后台线程和主循环解耦；lock 则用于避免多个线程同时读写共享状态造成混乱。

## 易错卡片

> [!warning] 误区：后台任务启动后主 Agent 就不用管了
> 正确说法：主 Agent 后续要通过通知队列或 `check_background` 接收结果。

> [!warning] 误区：通知队列和任务状态字典是同一个东西
> 正确说法：`tasks` 保存当前状态，`notification_q` 保存待注入的完成通知。

> [!warning] 误区：push 回调一定更高级
> 正确说法：pull 队列更解耦，主循环按自己的节奏取结果。

> [!warning] 误区：daemon 线程只有好处
> 正确说法：它能避免进程挂住，但主程序退出时后台任务可能被中断。
