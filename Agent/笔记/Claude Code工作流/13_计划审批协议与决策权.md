> [!question] 常见问题速查
> 1. 计划审批协议和关机协议的方向有什么不同？
> 2. plan request 由谁发起，谁决策？
> 3. `plan_approval` 如何创建 request？
> 4. lead approve/reject 后如何通知 teammate？
> 5. approved 和 rejected 后 teammate 应该分别怎么做？
> 6. 为什么高风险任务需要计划审批？

> [!abstract] 核心主线
> 计划审批协议用于让 teammate 在执行高风险任务前先提交 plan，由 lead 审批。它和关机协议方向相反：关机协议是 lead 发起、teammate 决策；计划审批是 teammate 发起、lead 决策。teammate 调用 `plan_approval(plan)`，系统生成 request_id，把计划保存到 `plan_requests`，状态为 pending，并写入 lead inbox。lead 读取后调用审批处理工具，把状态改为 approved 或 rejected，并把结果和 reason 写回请求发起方的 inbox。approved 表示可以执行；rejected 表示需要修订计划。

## 概念精讲

### 两种团队协议的方向

| 协议 | 发起方 | 决策方 | 用途 |
|---|---|---|---|
| shutdown | lead | teammate | lead 请求队员优雅退出 |
| plan approval | teammate | lead | 队员提交计划，leader 审批 |

关键判断：谁承担后果，谁应该拥有决策权。

关机时，teammate 需要判断自己能否安全退出；计划执行时，lead 需要判断计划是否符合目标和风险边界。

### 计划审批是风险闸门

teammate 可以提出方案，但不能直接执行高风险动作。先提交 plan，等 lead 同意，再继续。

适合审批的场景：

- 修改文件。
- 删除文件。
- 调用外部系统。
- 执行成本高的任务。
- 涉及安全、权限、数据或业务后果的操作。

## 提交计划：`plan_approval`

Charlie 提交计划时：

```json
{
  "name": "plan_approval",
  "input": {
    "plan": "I will inspect the codebase, identify risky files, propose edits, then wait for approval before writing changes."
  }
}
```

本地处理：

1. 生成 request_id。
2. 在 `plan_requests` 里保存 from、plan、status=pending。
3. 向 lead inbox 写入 plan request。
4. 返回提交结果。

```python
def request_plan_approval(sender, plan):
    request_id = str(uuid.uuid4())[:8]

    with lock:
        plan_requests[request_id] = {
            "from": sender,
            "plan": plan,
            "status": "pending",
        }

    bus.send(
        to="lead",
        sender=sender,
        message_type="plan_approval_request",
        content=plan,
        extra={"request_id": request_id},
    )
```

## lead 审批

lead 读到 plan request 后，决定 approve 或 reject。

```json
{
  "request_id": "p19a8f2d",
  "approve": false,
  "reason": "The plan is too vague. Add concrete steps and risk checks."
}
```

处理逻辑：

- 根据 request_id 找到 request。
- approve=true：status 变 approved。
- approve=false：status 变 rejected。
- 根据 request["from"] 找到发起方。
- 把审批结果写回发起方 inbox。

```python
def handle_plan_approval(request_id, approve, reason):
    with lock:
        req = plan_requests.get(request_id)
        if not req:
            return "unknown plan request"

        req["status"] = "approved" if approve else "rejected"
        target = req["from"]

    bus.send(
        to=target,
        sender="lead",
        message_type="plan_approval_response",
        content=reason,
        extra={"request_id": request_id, "approved": approve},
    )
```

## 状态与后续动作

| 状态 | 含义 | teammate 后续 |
|---|---|---|
| pending | 等待 lead 审批 | 不应执行高风险动作 |
| approved | lead 同意计划 | 可以按计划执行 |
| rejected | lead 拒绝计划 | 读取原因，修订 plan，再提交 |

> [!warning] 与关机协议不同
> shutdown approved 后，teammate 退出；plan approved 后，teammate 不退出，而是继续按计划执行。

## 应用与迁移问答

> [!question] 为什么不能让 teammate 直接执行计划？
> 因为它可能没有完整目标和风险视角。计划审批让 lead 在执行前把关。

> [!question] lead 拒绝时为什么要给 reason？
> 没有 reason，teammate 不知道该如何修订计划，只能盲目重试。

> [!question] plan approval 和 todo planning 有什么关系？
> todo planning 是自己把计划显式化；plan approval 是把计划提交给有决策权的一方审批。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把计划审批的决策方误认为 teammate。
> - 忘记审批结果要写回 teammate inbox。
> - rejected 后不给修改原因。
> - approved 后错误地让 teammate shutdown。
> - teammate 没等 approved 就直接执行。
> - 多个 plan request 没有 request_id。
> - 把 plan request 状态和 teammate 生命周期状态混淆。

## 费曼解释润色版

> [!quote]
> 计划审批协议用于让 teammate 在高风险执行前先提案。它和关机协议方向相反：关机是 lead 请求 teammate，teammate 决定是否退出；计划审批是 teammate 把 plan 发给 lead，lead 决定是否允许执行。Charlie 调用 `plan_approval` 时，系统会生成 request_id，把 from、plan 和 pending 状态保存到 `plan_requests`，并把计划写入 lead inbox。lead 读取后根据计划质量决定 approve 或 reject。如果 approve，request 状态变 approved，Charlie 可以按计划执行。如果 reject，状态变 rejected，lead 应该给出 reason，并把结果写回 Charlie 的 inbox。Charlie 下一轮读到拒绝后，应修订计划再提交，而不是直接执行。这样团队系统就把高风险行动变成“先提案、再审批、再执行”。

## 易错卡片

> [!warning] 误区：plan approval 和 shutdown 的决策方一样
> 正确说法：shutdown 决策方是 teammate，plan approval 决策方是 lead。

> [!warning] 误区：plan approved 后 teammate 应该退出
> 正确说法：approved 后应继续执行计划。

> [!warning] 误区：rejected 只需要一个布尔值
> 正确说法：还需要 reason，否则 teammate 无法修订。

> [!warning] 误区：提交计划就是完成任务
> 正确说法：提交只是进入 pending，真正执行要等 approved。
