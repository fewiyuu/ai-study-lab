> [!question] 常见问题速查
> 1. 系统为什么先执行 `git rev-parse --show-toplevel`？
> 2. `.tasks`、`.worktrees/index.json`、`.worktrees/events.jsonl` 分别记录什么？
> 3. 模型第一轮直接调用 `worktree_create` 为什么会失败？
> 4. task JSON 为什么新增 `worktree`、`created_at`、`updated_at`？
> 5. `worktree_create_before` 和 `worktree_create_after` 有什么区别？
> 6. `index.json` 和 `events.jsonl` 的作用有什么不同？
> 7. 为什么只执行 `git worktree add` 还不算可审计？

> [!abstract] 核心主线
> 任务隔离下半节关注“可审计、可追踪”。系统启动时先用 `git rev-parse --show-toplevel` 找到 repo root，并初始化 `.tasks`、`.worktrees/events.jsonl` 和 `.worktrees/index.json`。模型如果在 task 不存在时直接调用 `worktree_create`，工具会返回 `Task not found`，促使模型先创建 task。真正创建 worktree 时，系统先写 `worktree_create_before` 事件，再执行 `git worktree add`，然后把 worktree entry 写入 `index.json`，把 task 的 `worktree` 绑定和 `status=in_progress` 写回 task JSON，最后写 `worktree_create_after` 事件。这样一次目录创建不只是 Git 命令，而是带有任务绑定、索引和事件日志的可审计过程。

## 概念精讲

### 启动时先确定 repo root

系统先执行：

```bash
git rev-parse --show-toplevel
```

它返回当前 Git 仓库的根目录。后续 `.tasks`、`.worktrees`、worktree 路径和 Git 命令都依赖这个根目录。

然后系统检查当前目录是不是 Git 仓库，因为如果不是，就不能使用 `git worktree`。

### 三类追踪文件

| 文件 / 目录 | 用途 | 回答的问题 |
|---|---|---|
| `.tasks/` | 保存 `task_N.json` | 任务是什么、状态如何、绑定哪个 worktree |
| `.worktrees/index.json` | 保存当前 worktree 索引 | 系统管理了哪些 worktree |
| `.worktrees/events.jsonl` | 保存事件日志 | 什么时候做过什么动作 |

`index.json` 更像当前状态快照；`events.jsonl` 更像时间线。

### 工具结果会纠正模型流程

视频里模型第一轮直接调用 `worktree_create`，传入 `task_id=1` 和 `task_id=2`。

但 `.tasks` 目录为空，所以工具返回：

```text
Task 1 not found
Task 2 not found
```

下一轮模型看到这个结果后，先调用 `task_list`，再调用 `task_create` 创建两个任务。

> [!warning] 错误结果也是上下文
> 工具失败不是纯失败，它会成为下一轮模型推理的输入。设计好错误消息，可以把模型从错误流程拉回正确流程。

### task JSON 新增字段

这一节的 task 比早期 task system 多了：

```json
{
  "worktree": "",
  "created_at": "...",
  "updated_at": "..."
}
```

字段含义：

- `worktree`：任务绑定的工作目录名称。
- `created_at`：任务创建时间。
- `updated_at`：任务最后更新时间。

创建 worktree 并绑定 task 后，`worktree` 会写成目录名，`status` 通常从 `pending` 变成 `in_progress`。

## 创建 worktree 的可审计流程

### 1. 写 before 事件

在执行 Git 命令前，先向 `.worktrees/events.jsonl` 追加一行：

```json
{"event":"worktree_create_before","ts":"...","task_id":1,"worktree":{"name":"auth-refactor","base_ref":"HEAD"}}
```

它记录的是“准备做什么”。

### 2. 执行 Git 命令

然后才运行：

```bash
git worktree add .worktrees/auth-refactor -b wt/auth-refactor HEAD
```

这里真正产生副作用：创建工作目录、创建/绑定分支。

### 3. 更新 index

Git 命令成功后，把 worktree entry 写入 `.worktrees/index.json`：

```json
{
  "name": "auth-refactor",
  "path": ".worktrees/auth-refactor",
  "branch": "wt/auth-refactor",
  "task_id": 1,
  "created_at": "...",
  "status": "active"
}
```

以后系统可以直接读 index，知道当前有哪些 worktree，以及每个目录对应哪个 task。

### 4. 绑定 task

系统再更新 task JSON：

```json
{
  "id": 1,
  "worktree": "auth-refactor",
  "status": "in_progress",
  "updated_at": "..."
}
```

这样从 task 可以找到 worktree，从 index 也能反查 task。

### 5. 写 after 事件

最后写：

```json
{"event":"worktree_create_after","ts":"...","task_id":1,"worktree":{"name":"auth-refactor","status":"active"}}
```

它记录的是“已经做完什么”。

## JSONL 的直觉

JSONL 是 JSON Lines：每一行都是一个独立 JSON。

适合事件日志，因为：

- 可以持续追加。
- 不需要每次重写整个文件。
- 方便按时间顺序读取。
- 某一行坏了，也不一定影响所有记录。

## 应用与迁移问答

> [!question] `index.json` 和 `events.jsonl` 是否重复？
> 不重复。`index.json` 是当前有哪些 worktree 的状态快照；`events.jsonl` 是创建前后发生过什么的时间线。

> [!question] 为什么创建 worktree 后要更新 task？
> 因为任务要知道自己在哪个隔离目录执行，以及状态是否已进入 `in_progress`。

> [!question] 为什么 before 事件要写在 Git 命令之前？
> 这样即使命令失败，也能知道系统原本打算执行什么，方便排查失败原因。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 只关注 `git worktree add`，忽略任务绑定和审计记录。
> - 把 `index.json` 和 `events.jsonl` 当成同一种东西。
> - 忘记 task 不存在时不能创建绑定 worktree。
> - 创建 worktree 后没有把 task status 改成 `in_progress`。
> - 只写 after，不写 before，失败时看不到意图。
> - 没有在 index 中记录 task_id，导致目录和任务无法互相追踪。

## 费曼解释润色版

> [!quote]
> 这一节讲的是把 Git Worktree 变成可审计的 Agent Harness 能力。系统启动时先用 `git rev-parse --show-toplevel` 找到仓库根目录，然后初始化 `.tasks`、`.worktrees/events.jsonl` 和 `.worktrees/index.json`。模型如果直接创建 worktree，但 task1 和 task2 还不存在，工具会返回 `Task not found`，引导模型先创建任务。创建任务后，每个 task JSON 会包含 `worktree`、`created_at` 和 `updated_at`。真正创建 worktree 时，系统先在 `events.jsonl` 里写 `worktree_create_before`，记录准备为哪个 task 创建哪个目录；再执行 `git worktree add`；成功后把 worktree 写入 `index.json`，并把 task 的 `worktree` 绑定和 `status=in_progress` 写回任务文件；最后再写 `worktree_create_after`。这样，系统不只是创建了一个目录，还能回答这个目录为哪个任务创建、何时创建、当前状态如何、前后发生过什么。

## 易错卡片

> [!warning] 误区：可审计就是能看到目录存在
> 正确说法：可审计需要事件日志、索引状态和任务绑定。

> [!warning] 误区：`events.jsonl` 可以替代 `index.json`
> 正确说法：事件日志记录历史，index 记录当前管理状态。

> [!warning] 误区：task 不存在也可以直接创建 worktree
> 正确说法：如果 worktree 要绑定 task，就必须先有 task。

> [!warning] 误区：只要 after 日志就够了
> 正确说法：before 日志能记录执行意图，失败排查时很关键。
