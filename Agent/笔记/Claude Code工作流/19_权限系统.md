> [!question] 常见问题速查
> 1. 为什么模型返回 tool use 只是行动建议？
> 2. 权限检查应该插在工具执行流程的哪里？
> 3. 为什么 Bash 需要特殊安全校验？
> 4. `PermissionManager.check` 的决策顺序是什么？
> 5. default、auto、plan 模式分别有什么含义？
> 6. 用户输入 yes、no、always 后系统分别怎么处理？
> 7. 连续拒绝计数有什么意义？

> [!abstract] 核心主线
> 权限系统把模型的 tool use 从“直接执行”改成“先过安全关”。模型返回工具调用只是行动建议，真正执行前由 `PermissionManager` 检查。流程通常是：先对 Bash 命令做危险模式校验，再匹配 deny rules；如果当前是 plan mode，则阻止写操作；如果是 auto mode，则可自动允许 read-only 工具；再匹配 allow rules；都不命中时 ask user。用户选择 yes 只允许本次，选择 no 会拒绝并增加连续拒绝计数，选择 always 会追加运行时 allow rule，让后续同类调用自动通过。权限系统是 Agent Harness 的系统加固层，让模型能做事但不越界。

## 概念精讲

### tool use 只是行动建议

早期 Agent loop：

```text
模型返回 tool_use -> 系统直接执行工具
```

加入权限系统后：

```text
模型返回 tool_use -> 权限检查 -> allow / ask / deny -> 决定是否执行
```

模型可以建议执行 `bash`、`write_file`、`read_file`，但它没有最终执行权。

### Bash 为什么特殊

Bash 能做太多事情：

- 列目录。
- 读文件。
- 写文件。
- 删除文件。
- 执行脚本。
- 提权。
- 拼接命令。

所以不能只看工具名是 `bash` 就放行，必须检查 `command` 内容。

## Bash 安全校验

`BashSecurityValidator` 会检查命令中是否包含危险模式：

| 模式 | 示例 | 风险 |
|---|---|---|
| shell metacharacters | `;`、`&&`、`|` | 命令注入或拼接执行 |
| sudo | `sudo ...` | 提权 |
| recursive delete | `rm -rf` | 删除大量文件 |
| command substitution | `$(...)`、反引号 | 隐藏执行命令 |
| IFS manipulation | `${IFS}` | 绕过空格过滤，命令注入 |

> [!warning] Bash 要看参数
> `bash: ls` 和 `bash: rm -rf /` 都叫 Bash，但安全等级完全不同。

## PermissionManager 的决策顺序

### 1. Bash 安全校验

如果 tool 是 `bash`，先检查 command 是否命中危险模式。命中就直接 deny。

### 2. deny rules

默认规则里有：

```json
{"tool": "bash", "content": "rm -rf /", "behavior": "deny"}
{"tool": "bash", "content": "sudo *", "behavior": "deny"}
```

deny rules 要优先，避免被后面的 allow rule 覆盖。

### 3. mode 决策

| mode | 含义 | 行为 |
|---|---|---|
| default | 默认模式 | 不确定时 ask user |
| auto | 自动模式 | read-only 工具可自动 allow |
| plan | 计划模式 | 阻止写操作，只允许先规划 |

plan mode 的意义是：当用户不希望 Agent 直接改东西时，让它先讲计划。

### 4. allow rules

默认可能允许：

```json
{"tool": "read_file", "path": "*", "behavior": "allow"}
```

用户选择 always 后，也会追加新的 allow rule。

### 5. ask user

如果前面都没有决策，就询问用户：

- `yes`：本次允许。
- `no`：本次拒绝。
- `always`：追加 allow rule，后续同类调用自动允许。

## always 与运行时规则

用户选择 always 后，系统会把规则加入当前 `PermissionManager.rules`。

例如用户对 `bash dir` 选择 always，后续同进程内相似 Bash 调用可能直接 allow，不再 ask。

这说明权限规则不仅可以来自默认配置，也可以在运行中被用户授权扩展。

## 连续拒绝计数

`consecutive_denials` 用于记录用户连续拒绝了多少次。

如果多次拒绝，说明模型一直想做用户不想让它做的动作。系统可以提示切换到 plan mode：

```text
你已经连续拒绝多次，建议切换到 plan mode，让 Agent 先给计划。
```

这也是 Harness 的反馈控制：用户拒绝是信号，系统应该调整 Agent 行动方式。

## 应用与迁移问答

> [!question] auto mode 是否意味着所有操作都自动允许？
> 不是。auto mode 适合自动允许 read-only 工具，写操作仍应谨慎处理。

> [!question] plan mode 为什么会拒绝写操作？
> 因为 plan mode 的目的就是先规划、先解释，不让 Agent 直接产生副作用。

> [!question] 权限系统和计划审批有什么关系？
> 都是在行动前插入治理层。计划审批偏任务级计划；权限系统偏工具级执行。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把 tool use 当成最终执行命令。
> - 只按工具名判断，不看参数。
> - 把 Bash 和普通 read_file 一样处理。
> - allow rules 放在 deny rules 前面。
> - 忘记用户选择 always 会改变运行时 rules。
> - 不把 deny reason 写回模型，导致模型不知道为什么失败。
> - 连续拒绝后仍让模型反复尝试同类危险动作。

## 费曼解释润色版

> [!quote]
> 权限系统的核心是：模型可以提出 tool use，但不能直接拥有执行权。系统在真正执行工具前调用 `PermissionManager.check`。如果工具是 Bash，要先用 `BashSecurityValidator` 检查命令里有没有 `sudo`、`rm -rf`、命令拼接、命令替换、IFS 注入等危险模式。然后先匹配 deny rules，因为危险规则必须优先；再看当前 mode，plan mode 会阻止写操作，auto mode 可以自动允许 read-only 工具；之后再匹配 allow rules。如果都不命中，就 ask user。用户选 yes 表示本次允许，选 no 表示拒绝并增加连续拒绝计数，选 always 表示把这类操作加入运行时 allow rules。这样，模型的行动从“直接执行”变成“经过安全关后执行”，既保留了 Agent 的行动能力，也降低了误删、越权和命令注入风险。

## 易错卡片

> [!warning] 误区：模型返回 tool_use 就应该立即执行
> 正确说法：tool_use 是建议，执行前要过权限系统。

> [!warning] 误区：Bash 工具可以整体 allow
> 正确说法：Bash 必须检查具体 command。

> [!warning] 误区：allow rule 可以覆盖危险 deny
> 正确说法：deny 应该优先，危险命令先拒绝。

> [!warning] 误区：连续拒绝只是用户情绪
> 正确说法：连续拒绝是 Harness 信号，说明模型行动方向需要调整。
