> [!question] 常见问题速查
> 1. 错误恢复解决什么问题？
> 2. 为什么 Agent 出错不一定是模型能力问题？
> 3. `max_tokens` 截断时如何恢复？
> 4. overlong prompt 时如何恢复？
> 5. 网络错误或限流为什么用 backoff + jitter？
> 6. retry attempt 的上限有什么意义？
> 7. 自动压缩只取最早 80000 字符有什么风险？

> [!abstract] 核心主线
> 错误恢复把“报错就崩溃”升级成“先判断错误类型，再选择恢复路径”。真实 Agent 会遇到模型输出被 `max_tokens` 截断、上下文过长、网络抖动、API 超时和限流。`max_tokens` 适合追加 continuation message，让模型从停止处继续；overlong prompt 适合 auto compact，把历史压缩成摘要后继续；connection error 和 rate limit 适合 backoff + jitter，等待后重试。Agent loop 因此需要 `try/except`、`for attempt in range(max_attempts)`、恢复次数上限和恢复日志。不是所有错误都适合无脑重试，权限拒绝、参数错误、文件不存在等应反馈给模型改计划。

## 概念精讲

### 错误不等于任务失败

很多失败不是任务真的失败，而是当前轮次需要换一种继续方式：

- 输出到一半被截断。
- 上下文太长，API 拒绝请求。
- 网络临时抖动。
- API 超时。
- 限流。

没有恢复机制时，主循环在第一个错误处停住。加入恢复机制后，系统能分类处理。

## 三类恢复策略

| 错误类型 | 检测方式 | 恢复方式 |
|---|---|---|
| 输出截断 | `response.stop_reason == "max_tokens"` | 追加 continuation message |
| 上下文超限 | API error body 包含 overlong prompt | auto compact 历史 |
| 网络 / 限流 | connection error / rate limit | backoff + jitter 后重试 |

### max_tokens 续写

当模型输出被截断：

```python
if response.stop_reason == "max_tokens":
    max_output_recovery_count += 1
    if max_output_recovery_count <= 3:
        messages.append({
            "role": "user",
            "content": "Continue from where you stopped. Do not repeat previous content."
        })
        continue
```

关键点：

- 不是重新开始。
- 要求从停止处继续。
- 必须有次数上限。

### overlong prompt 自动压缩

当 API 返回上下文超限：

1. 从 messages 中抽取 conversation text。
2. 让模型总结压缩。
3. 把 summary 包进 continuation message。
4. 用压缩后的 messages 继续请求。

> [!warning] 压缩范围要谨慎
> 字幕里指出，代码取最早 80000 字符可能不够稳。恢复时往往更需要最近用户指令、工具错误和当前任务状态。

更稳思路：

- 保留最近消息。
- 压缩较早历史。
- 保留任务状态、工具结果和用户最新意图。

### 网络 / 限流退避重试

网络错误和限流通常是临时问题。

```python
delay = min(base * (2 ** attempt), 30) + random.random()
time.sleep(delay)
```

这里的 jitter 是随机抖动，避免多个请求同时在固定时间重试，造成再次拥塞。

## Loop 改造

早期 loop：

```text
request model -> parse response -> run tools
```

错误恢复版：

```text
for attempt in max_attempts:
  try request model
  catch overlong -> compact
  catch network/rate -> backoff
  catch fatal -> raise/stop

check stop_reason
if max_tokens -> continuation
```

## token 估算

代码里用字符数 / 4 粗略估算 token。

优点：

- 简单。
- 可以提前判断是否接近阈值。

局限：

- 不等于真实 tokenizer。
- 中文、代码、JSON 的比例不同。
- 只能作为预警，不应作为精确计数。

## 应用与迁移问答

> [!question] 为什么不能所有错误都 sleep 后重试？
> 因为很多错误不是临时问题。例如工具参数错、权限拒绝、文件不存在，sleep 不会解决根因。

> [!question] max_tokens 为什么不是 API error？
> 请求成功了，只是模型输出达到上限。它需要续写，而不是压缩或重试请求。

> [!question] 错误恢复为什么要告诉用户？
> 用户需要知道系统是在恢复，不是卡住。恢复日志也能帮助调试。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把所有 Agent 错误都归因于模型。
> - max_tokens 时重新开始，而不是续写。
> - overlong prompt 时不压缩上下文。
> - retry 没有次数上限。
> - 限流后立即密集重试。
> - 自动压缩丢掉最近用户意图。
> - 不记录恢复过程，导致用户以为系统卡死。

## 费曼解释润色版

> [!quote]
> 错误恢复的核心是：Agent 出错后不要立刻崩溃，而要先判断错误类型，再选择恢复路径。如果模型响应的 `stop_reason` 是 `max_tokens`，说明输出被截断，可以追加一条 continuation message，让模型从停止处继续，但要限制最多续写次数。如果 API 返回 overlong prompt，说明上下文太长，需要 auto compact，把历史压缩成摘要后继续；不过压缩范围要小心，最好保留最近用户指令和任务状态。如果遇到 connection error 或 rate limit，通常是临时故障，可以用 backoff 加随机 jitter 等一会儿再重试。Agent loop 因此需要 try/catch、attempt 上限和恢复日志。不是所有错误都适合重试，权限拒绝、参数错误、文件不存在等要反馈给模型修改计划。这样系统就从“第一次报错就停”变成“可分类恢复、可继续工作”的更健壮 Harness。

## 易错卡片

> [!warning] 误区：错误恢复就是无限重试
> 正确说法：错误恢复是分类处理，并且必须有上限。

> [!warning] 误区：max_tokens 要重新请求同一个 prompt
> 正确说法：应该追加 continuation，让模型从停止处继续。

> [!warning] 误区：上下文超限只要删一些历史
> 正确说法：应保留关键最近上下文，压缩较早历史。

> [!warning] 误区：网络错误和参数错误可以同样处理
> 正确说法：网络错误可 backoff 重试；参数错误要改输入或计划。
