> [!question] 常见问题速查
> 1. 定时调度解决 Agent 的什么能力缺口？
> 2. 调度系统由哪三部分组成？
> 3. Cron 五个字段分别是什么？
> 4. 调度记录里为什么需要 `recurring`、`durable`、`last_fired`？
> 5. `check_loop` 如何触发任务？
> 6. 为什么调度器不是第二个 Agent loop？
> 7. jitter、自动过期和防重复触发分别解决什么问题？

> [!abstract] 核心主线
> 定时调度让 Agent 能处理未来时间的任务。用户不必每晚手动输入“运行测试”，也不必每周一记得“写周报”。系统通过 `schedule_create` 把“在时间 Y 执行 X”保存为调度记录；后台 `check_loop` 每分钟对照当前时间和 Cron 表达式；匹配后把通知放入 notification queue，并更新 `last_fired`；Agent loop 开头 drain notifications，把触发 prompt 作为 user message append 到 messages。调度器不是第二个 Agent loop，它只是时间触发器，真正的推理、工具调用和权限检查仍由统一 Agent loop 完成。

## 概念精讲

### 为什么需要调度

Agent 默认只能响应当前输入。

定时调度解决：

- 未来某个时间点开始工作。
- 每隔一段时间重复工作。
- 每周 / 每天固定时间触发任务。
- 用户不必记得手动发起。

典型例子：

- 每天夜间运行测试。
- 每周一早上写周报。
- 每 5 分钟检查一次状态。

## 三个组成部分

| 组成 | 作用 |
|---|---|
| 调度记录 | 保存 Cron、prompt、是否重复、是否持久化 |
| 时间检查器 | 后台线程定期检查当前时间是否匹配 |
| 通知队列 | 到点后把 prompt 送回 Agent loop |

## Cron 五字段

```text
minute hour day-of-month month day-of-week
```

| 字段 | 含义 | 范围 |
|---|---|---|
| minute | 分钟 | 0-59 |
| hour | 小时 | 0-23 |
| day of month | 每月第几天 | 1-31 |
| month | 月份 | 1-12 |
| day of week | 每周第几天 | 0-6 |

例子：

```text
*/5 * * * *     每 5 分钟
0 9 * * 1      每周一 9:00
30 14 * * *    每天 14:30
```

## 调度记录结构

```json
{
  "id": "task_1",
  "cron": "0 9 * * 1",
  "prompt": "Write my weekly report",
  "recurring": true,
  "durable": true,
  "created_at": 1714200000,
  "last_fired": null,
  "jitter_offset": 0
}
```

字段含义：

- `cron`：时间规则。
- `prompt`：到点后注入 Agent 的任务提示。
- `recurring`：是否重复。
- `durable`：是否持久化到磁盘。
- `created_at`：创建时间。
- `last_fired`：上次触发时间，用于防重复。
- `jitter_offset`：抖动偏移，避免整点集中触发。

## check loop

`CronScheduler.start()` 会：

1. 加载 durable schedules。
2. 创建后台线程。
3. 启动 `check_loop`。

`check_loop` 每秒醒一次，但通过 `last_check_minute` 保证每分钟只检查一次。

核心流程：

```python
for task in tasks:
    if expired(task):
        continue

    check_time = now + task["jitter_offset"]

    if cron_matches(task["cron"], check_time):
        queue.put({"role": "user", "content": task["prompt"]})
        task["last_fired"] = now
```

## 通知回流 Agent loop

Agent loop 开头调用：

```python
notifications = scheduler.drain_notifications()
for n in notifications:
    messages.append({
        "role": "user",
        "content": n["prompt"]
    })
```

从 Agent 角度看，它只收到一条 user message。它不需要知道这条消息来自用户手动输入，还是来自定时器。

> [!abstract] 关键点
> 调度器负责“何时触发”；Agent loop 负责“如何执行”。

## jitter、过期和防重复

### jitter

给整点或半点任务增加几分钟随机偏移。

目的：

- 避免多个实例同时触发。
- 避免流量集中在整点。

### 自动过期

重复任务超过一定时间未维护，例如 7 天，可以自动删除。

目的：

- 防止孤岛任务累积。
- 避免过期计划继续触发。

### 防重复触发

`last_check_minute`：避免同一分钟检查多次。

`last_fired`：避免同一个任务在同一分钟触发多次。

## 与后台任务对比

| 维度 | 后台任务 | 定时调度 |
|---|---|---|
| 解决问题 | 慢工作不阻塞主 loop | 未来工作何时开始 |
| 触发方式 | 用户现在发起 | 用户发起 + 时间触发 |
| 状态核心 | 运行中的任务 | 调度记录 |
| 回流方式 | 通知队列 | 通知队列 |

## 应用与迁移问答

> [!question] 调度器为什么不是第二个 Agent loop？
> 因为它不调用模型、不执行工具，只负责到时间后注入消息。

> [!question] durable=true 有什么意义？
> 重启后还能加载这条调度记录，不会因为进程结束丢失。

> [!question] 如果程序停机错过任务怎么办？
> 可以检查 last_fired 到当前时间之间是否有 Cron 匹配，最多回溯一定时间，把遗漏任务记录或补触发。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把调度器当成另一个 Agent。
> - 忘记通知要回流统一 messages。
> - 不校验 Cron 表达式。
> - 没有 last_fired，导致重复触发。
> - durable 任务不落盘。
> - 不清理过期重复任务。
> - 整点任务没有 jitter，多个实例同时触发。

## 费曼解释润色版

> [!quote]
> 定时调度让 Agent 能在未来时间自动开始工作。用户通过 `schedule_create` 创建一条调度记录，里面有 Cron 表达式、prompt、是否 recurring、是否 durable、created_at、last_fired 和 jitter_offset。Cron 用五个字段描述时间规则，比如 `0 9 * * 1` 表示每周一 9 点。调度器启动时加载 durable 任务，然后后台 `check_loop` 每分钟对照当前时间和 Cron 规则。匹配后，它不会自己调用模型，而是把 prompt 放入 notification queue，并更新 `last_fired` 防止重复触发。Agent loop 开头 drain 这个队列，把通知作为 user message append 到 messages，所以手动输入和定时触发走同一套 Agent 系统。jitter 用来避免整点同时触发，自动过期用来清理孤岛任务。调度器负责“什么时候触发”，Agent loop 负责“触发后怎么做”。

## 易错卡片

> [!warning] 误区：定时调度需要第二个 Agent loop
> 正确说法：调度器只注入消息，执行仍由统一 Agent loop 完成。

> [!warning] 误区：每秒检查就可以每秒触发
> 正确说法：通常用 last_check_minute 控制每分钟检查一次。

> [!warning] 误区：Cron 匹配后不用记录触发时间
> 正确说法：需要 last_fired 防重复触发。

> [!warning] 误区：所有调度都只存在内存里
> 正确说法：重要任务应 durable 持久化。
