上下文不只是 Prompt

定义

提示词工程主要关心指令怎么写。上下文工程关心的是模型这一次推理前能看到的全部信息：系统提示、用户消息、工具定义、工具返回、检索资料、记忆、输出格式、运行时配置，甚至上一轮错误。只改一句 prompt，通常只是在改上下文中的一个小部件。

Agent 和普通聊天最大的差别是它会循环行动。模型每次决定下一步时，都依赖当前窗口里留下的材料。如果错误工具结果、过时计划或相互冲突的规则留在窗口里，后面的推理会继续受影响。你现在应该能做的第一件事，是把一次模型调用拆成“它看到了什么”，而不是只看用户最后一句话。

对象进入上下文的样子典型风险
系统提示角色、边界、工具使用规则太抽象会不给行为抓手，太硬编码会脆
消息历史用户与 Agent 的多轮对话旧错误和过期目标继续污染
工具定义函数名、参数、描述、返回格式工具重叠导致误选
工具结果搜索结果、文件内容、日志、数据库结果原始结果太长，稀释重点
记忆用户偏好、项目规则、历史摘要误召回让窗口不再受控
输出格式JSON schema、报告结构、字段约束格式约束和任务目标冲突

Prompt Engineering 和 Context Engineering 的分界

概念区分

Prompt engineering 是必要的，但它偏向“静态写法”：怎样组织指令、怎样给 few-shot 示例、怎样说明输出格式。Context engineering 更像运行时系统设计：什么时候注入什么资料，哪些资料写到窗口外，哪些信息需要压缩，哪些子任务应该隔离。

判断规则很直接：如果问题能靠改一段指令解决，多半是 prompt engineering；如果问题涉及记忆、检索、工具、历史、状态、压缩、权限、成本或长任务协调，那已经进入 context engineering。你现在应该能看到，“上下文工程”不是换个名词，而是 Agent 变长、变会用工具之后出现的新工程面。

Prompt Engineering

优化一段指令如何表达。典型动作是重写系统提示、加示例、加格式要求、调整语气和边界。

Context Engineering

优化一次模型调用实际看见什么。典型动作是选择资料、保存状态、压缩历史、过滤工具、隔离子任务。

容易成功的场景

一次性分类、固定格式改写、短文本生成、规则很稳定的任务。

必须升级的场景

代码库修改、长研究任务、跨会话偏好、工具链复杂、结果需要持续验证的任务。

注意力预算：窗口很大也不能乱塞

机制直觉

长上下文的问题不只发生在“超出窗口”那一刻。Anthropic 把 context 当作有限资源来处理：token 越多，模型越难稳定抓住远距离依赖，关键信息也更容易被噪声稀释。工程上可以把这叫注意力预算。它不是模型内部的精确计量表，而是一个很实用的判断：每放入一段信息，都要问它是否值得占用模型的阅读能力。

这解释了为什么“把所有文档都贴进去”经常不如“给路径、索引和检索工具”。前者让模型一次读完全部材料，后者让模型先保留轻量指针，需要时再取。你现在应该能判断：上下文窗口是工作台，不是仓库；仓库可以很大，工作台要保持可操作。

四类上下文失败

诊断

LangChain 引用了几个很有用的失败词：poisoning、distraction、confusion、clash。它们的价值不在术语本身，而在诊断方式。失败发生时，不要只问“模型怎么又笨了”，要检查窗口里是否有错误信息、太多无关材料、含糊工具描述或互相冲突的规则。

上下文失败的麻烦在于它会积累。一次错误工具调用如果被完整保留，后续模型可能继续把它当证据；一条过期计划如果没有更新，Agent 可能努力执行已经被用户否定的目标。你现在应该能做的事，是读一段失败轨迹，标出哪几类上下文正在拖后腿。

失败类型表现修法
Context poisoning幻觉、错误日志或错误结论进入历史保留错误证据但明确标注失败原因和新结论
Context distraction材料太多，目标被稀释裁剪、摘要、按需检索，把计划放到近处
Context confusion工具和规则重叠，模型不知道该用哪个减少工具重叠，改清参数语义，做工具选择
Context clash系统提示、记忆、用户要求互相矛盾建立优先级，删除或覆盖过期上下文

最小高信号上下文

设计原则

Anthropic 的原则可以压成一句工程话：找出最小的高信号 token 集合，让目标行为更可能发生。最小不等于短。一个短但含糊的系统提示可能比一个结构清楚的长提示更差；一个长但格式稳定的任务规范，可能比十轮口头补丁更节省注意力。

高信号上下文通常有三个特点：它和当前决策直接相关；它的格式让模型容易定位；它和其它上下文不冲突。低信号上下文则相反：只是“也许有用”、结构混乱、重复、过时或缺少来源。你现在应该能用这三条筛掉一批看似有用的材料。

完整链路：一次 Agent 调用前发生什么

主线

把本页串起来：用户提出任务后，系统先决定模型本轮应该知道什么。静态部分包括系统提示和工具定义；动态部分包括当前消息、相关记忆、检索结果、运行时配置、输出格式和上一轮工具结果。模型做出下一步行动，工具执行后产生新材料。新材料不是自动全塞回去，而要被写入、选择、压缩或隔离。

这就是 context engineering 的主线。Prompt engineering 仍然重要，因为指令是上下文的一部分；但 Agent 的可靠性还取决于窗口之外的状态、工具、记忆和生命周期控制。你现在应该能解释：为什么 Agent 越长、越会用工具，越需要上下文工程。

输入用户目标、系统提示、历史消息进入候选上下文。
筛选从记忆、文件、工具和知识库中选择当前需要的材料。
行动模型生成回复或工具调用，工具返回新的观察。
维护把观察写入状态、压缩历史、隔离大对象，准备下一轮。
