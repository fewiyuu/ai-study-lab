Agent loop：控制点藏在每一步

LangChain

LangChain 文档把 Agent loop 拆成两个核心动作：模型调用和工具执行。模型调用会读入提示、消息、工具、模型配置和输出格式；工具执行会读取状态、长期存储或运行时配置，并把结果写回某个位置。上下文工程不是只发生在调用模型前，而是贯穿整个生命周期。

你现在应该能做一个基本判断：如果错误发生在模型选择工具之前，可能是 model context 没给对；如果错误发生在工具执行期间，可能是 tool context 没给对；如果错误发生在多轮之后，可能是 life-cycle context 没维护好。

Model call动态提示、消息、工具、模型和输出格式。
Tool execution工具读取 state、store、runtime context。
State update保存消息、认证状态、摘要或任务进展。
Next turn下一轮只选择需要的内容给模型。

Transient 与 Persistent：临时注入不等于保存

核心区分

Transient context 是当前这一次模型调用看到的内容。比如你根据消息数量临时让模型更简洁，或者根据当前任务临时隐藏某些工具，这些变化可以不写入状态。Persistent context 是跨步骤保存的内容，比如消息历史、摘要、用户偏好、认证状态、任务计划。

这个区分很重要。很多 Agent 失败不是因为没有上下文，而是把临时上下文误当长期事实保存了，或者把长期记忆每轮都完整塞进窗口。你现在应该能判断：一条信息是只影响当前调用，还是要成为后续任务的状态。

类型保存位置适合内容错法
Transient当前请求构造临时语气、当前文件摘要、动态工具列表误以为下一轮还会自动存在
Persistentstate / store / 文件 / 数据库任务计划、用户偏好、项目规则、摘要把过时内容永久保留

State、Store、Runtime Context

三类来源

State 是当前会话或当前任务内的短期状态。它可以保存消息、上传文件元数据、认证结果、当前计划和工具执行后的中间结果。Store 是跨会话长期存储，适合用户偏好、项目规则、历史经验和长期记忆。Runtime context 是调用时带入的配置，比如 user_id、权限、环境、地区、成本等级。

三个词看起来像框架名，其实是工程边界。State 更像“这次任务进行到哪”；Store 更像“这个用户或项目长期知道什么”；Runtime context 更像“这次运行的外部条件是什么”。你现在应该能把一条信息放到合适的位置，而不是全部塞进 message history。

State
短期、任务内、会变化。例：当前 TODO、已读文件、认证状态。
Store
长期、跨会话、可检索。例：用户偏好、项目规则、历史案例。
Runtime
调用配置、权限和环境。例：user_id、role、region、API key 引用。

Model context：提示、消息、工具、模型、格式

控制面

LangChain 把 model context 拆成五类：system prompt、messages、tools、model、response format。它们都可以动态变化。比如长对话时系统提示提醒模型简洁；用户未登录时隐藏支付工具；低成本环境选择更便宜模型；管理员和普通用户使用不同输出字段。

这不是为了复杂而复杂。真实应用里的上下文应该跟用户、任务阶段、权限和成本相关。你现在应该能读懂一个 middleware 的意图：它不是“写 prompt 的花活”，而是在模型调用前改造本轮可见信息。

示例：根据消息数量改变系统提示text
@dynamic_prompt
def state_aware_prompt(request):
    message_count = len(request.messages)
    base = "You are a helpful assistant."
    if message_count > 10:
        base += "\nThis is a long conversation. Be concise and use the latest goal."
    return base
控制项能动态调整什么常见用途
System prompt行为规则、语气、阶段说明长对话简洁、项目规范、角色差异
Messages历史消息、文件摘要、检索片段只注入当前任务相关材料
Tools可用工具集合按权限、状态或任务过滤工具
Model模型和参数按成本、难度、窗口大小切换
Response formatschema 或结构按用户角色输出不同字段

Tool context：工具也读写上下文

工具语义

工具不是只接收模型给的参数。真实工具经常需要读数据库连接、API key、用户 ID、权限、当前认证状态或历史偏好。LangChain 文档强调工具可以从 state、store、runtime context 读取，也可以写回 state 或 store。

这能减少模型窗口压力。比如支付工具不需要让模型看到完整 API key，只需要从 runtime context 里读取安全引用；偏好保存工具不需要把所有偏好写进消息，只要写入 store。你现在应该能说清：工具上下文既保护敏感信息，也让模型少看无关细节。

示例：工具读取 runtime context，而不是把密钥暴露给模型text
@tool
def fetch_order(order_id: str, runtime: ToolRuntime[Context]):
    user_id = runtime.context.user_id
    api_key = runtime.context.api_key
    return query_order_api(user_id=user_id, order_id=order_id, api_key=api_key)
错误示例：把敏感配置直接塞进模型消息text
System: 当前用户 API_KEY=sk_live_...
User: 查询订单 10086
Assistant: 调用 fetch_order

Life-cycle context：在步骤之间维护轨迹

Middleware

生命周期上下文控制的是模型调用和工具执行之间发生什么。最典型的例子是总结：当消息超过阈值时，系统用另一个模型把旧消息压缩成摘要，永久替换 state 里的旧消息，同时保留最近消息。这个动作不只是临时裁剪，而是改变了后续所有回合看到的历史。

生命周期控制还可以做日志、护栏、跳过工具执行、重复模型调用、清理工具结果。你现在应该能理解 middleware 的价值：它让上下文工程变成可插拔的生命周期逻辑，而不是散落在 prompt 里的口头提醒。

示例：超过 token 阈值后压缩旧消息text
agent = create_agent(
    model="gpt-5.4",
    tools=[...],
    middleware=[
        SummarizationMiddleware(
            model="gpt-5.4-mini",
            trigger={"tokens": 4000},
            keep={"messages": 20},
        )
    ],
)

一张上下文策略表

落地

设计 Agent 时，不要只写“需要记忆”和“需要 RAG”。更好的做法是列出每种信息的来源、进入窗口的条件、保存位置、过期规则和验证方式。这样才能避免两个常见错误：该保存的没保存，该临时注入的永久污染。

你现在应该能为一个业务 Agent 画出最小上下文策略表。表不需要复杂，但必须回答：谁读、谁写、何时进窗口、何时移除。

信息来源进入窗口条件保存/过期
用户写作偏好Store生成邮件或报告时用户更新后覆盖旧值
上传文件列表State用户问题引用文件时任务结束后清理
支付工具Runtime 权限用户已认证且有权限不写入模型消息
旧对话摘要Life-cycle middleware历史过长时保留摘要和最近消息
