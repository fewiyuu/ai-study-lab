四个动词：Write、Select、Compress、Isolate

策略总览

LangChain 把 Agent 上下文管理归纳成四个动作：写出去、选回来、压缩掉、隔离开。中文里可以对应 Offload、Retrieve、Reduce、Isolate。它们不是四个互斥模块，而是一套循环：工具结果太大先写到外部，只把引用放进窗口；需要时检索回来；历史太长就压缩；任务太杂就隔离给子 Agent 或环境。

你现在应该能用这四个动词描述任何长任务上下文策略。不要一上来就说“加 RAG”。先问：信息应该留在窗口里吗？不留的话写到哪里？下一轮怎么取？取回来太长怎么办？不同子任务会不会互相污染？

Write

把计划、观察、文件、记忆写到窗口外，留下可追踪引用。

Select

按当前目标从外部记忆、文件、工具和知识库取回相关内容。

Compress

把历史、工具结果或研究轨迹压成更短但高保真的形式。

Isolate

把子任务、环境、大对象或冲突上下文分开，减少互相干扰。

Write：窗口外保存，不是丢弃

Offload

Write context 的直觉是给 Agent 一个外部工作区。人做长任务会写笔记、保存草稿、建立文件夹；Agent 也需要类似能力。Scratchpad 适合当前任务，memory 适合跨会话，文件系统适合大对象和结构化产物，数据库适合可查询状态。

写出去的关键是可回收。只保存一大坨文本不够，还要保存路径、标题、时间、来源和用途。否则下一轮 Select 会变成碰运气。你现在应该能判断：工具返回 500 行日志时，不应该全进窗口，而应该写到文件或 state，再把摘要和路径给模型。

示例：工具结果写到外部，只返回引用text
result_path = "runs/search_2026_06_06.json"
save_json(result_path, raw_search_results)
return {
  "summary": "找到 12 篇候选资料，其中 3 篇与上下文压缩直接相关。",
  "artifact": result_path,
  "next_hint": "如需原文证据，请读取 artifact 中的 top_3 条目。"
}
错误示例：把原始结果全部返回给模型text
return raw_search_results  # 可能是几万 token，包含重复、广告、无关片段

Select：检索不是只有向量库

Retrieve

Select context 是把窗口外的信息带回来。RAG 是常见做法，但不是唯一做法。代码 Agent 常用文件名、目录结构、grep、glob、AST、知识图谱和 rerank 组合。Anthropic 提到 just-in-time retrieval：先保留轻量引用，需要时再让 Agent 自己探索。

Select 的难点是误召回。记忆系统如果把用户位置、旧偏好或过期项目规则错塞进当前窗口，模型会自然使用它。你现在应该能设计选择条件：为什么取这段？取多少？放在哪里？过期和冲突怎么处理？

检索方式适合场景风险
向量检索语义相似资料、知识库问答相似不等于有用，容易漏结构信息
grep / file search代码仓库、明确术语、日志需要 Agent 会分步探索
知识图谱 / AST代码依赖、实体关系构建成本高，更新复杂
工具选择检索工具数量很多工具描述重叠仍会误选

Compress：短不等于好，保真才重要

Reduce

Compress context 的目标不是把文本变短，而是保留下一步需要的决策信息。摘要可以压缩长对话、工具结果、子 Agent 发现；裁剪可以删除旧消息、重复输出、无用日志；rerank 可以让相关片段排前。

最危险的压缩是丢掉后来才会有用的细节。Anthropic 建议 compaction 先追求 recall，确保关键事实都保留，再逐步提高 precision，删掉冗余内容。你现在应该能判断一份摘要是否合格：它有没有保留目标、决策、未解问题、约束、失败尝试和下一步。

Isolate：把冲突和大对象分开

隔离

Isolate context 的核心是分而治之。子 Agent 可以独立探索一个子问题，只把结论交回主 Agent；沙盒可以保存图片、音频、数据表和代码执行状态，只把必要返回值给模型；state schema 可以把工具原始结果放在不可见字段，需要时再显式取出。

隔离不等于一定上多 Agent。多 Agent 会增加 token、协调和评估成本。适合隔离的任务通常有清楚边界、可并行探索、结果能被压缩汇报，或者不同上下文混在一起会冲突。你现在应该能判断：什么时候隔离能提升清晰度，什么时候只是把系统搞复杂。

隔离方式适合不适合
Sub-agent并行研究、代码审查、只读探索强依赖共享状态的细粒度协作
Sandbox代码执行、大文件、表格、图片处理需要模型直接逐 token 阅读的短材料
State schema工具原始结果、候选证据、草稿没有明确读取规则的杂物箱
文件系统长任务产物、项目笔记、可复查证据缺少命名规范时会难以检索

长任务组合拳：Compaction、Notes、Subagents

Anthropic

Anthropic 把长时任务的策略讲得很清楚：compaction 负责在窗口接近限制时保留连续性；structured note-taking 让 Agent 把目标、计划、进展写到窗口外；sub-agent architecture 让主 Agent 聚合结果，子 Agent 独立消耗大量上下文。

这三者的边界不同。对话式协作需要 compaction，因为用户和 Agent 的来回很重要；迭代开发需要 notes，因为任务有里程碑和未完成事项；复杂研究适合 subagents，因为并行探索能换来更广覆盖。你现在应该能按任务特征选策略，而不是盲目套一个“多智能体架构”。

任务特征优先策略原因
大量往返讨论Compaction保留目标、决策和最近对话
长期迭代开发Structured notesTODO、设计约束、失败尝试需要跨轮保留
多方向研究Subagents各自探索，主 Agent 汇总高密度结论
大文件处理Sandbox / file system原始对象不应直接进窗口

策略链路：从工具结果到下一轮决策

完整主线

一次工具返回后，工程系统要做四个判断。第一，结果是否需要完整保存；第二，下一轮是否需要立刻看到；第三，如果要看到，是否需要压缩；第四，是否应该和主上下文隔离。这个判断链比单个 RAG 模块更接近真实 Agent 工程。

你现在应该能把四个动词连起来讲：Write 解决窗口放不下，Select 解决需要时取回，Compress 解决取回后太长，Isolate 解决混在一起会污染。

Write保存原始观察、计划和证据，留下引用。
Select按当前目标取回最相关片段。
Compress把长片段压成可行动摘要。
Isolate让子任务或大对象留在独立上下文。
