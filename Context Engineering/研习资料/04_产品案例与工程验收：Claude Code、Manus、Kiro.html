三个案例：Claude Code、Manus、Kiro

产品观察

中文文章把三个产品放在一起看：Claude Code 代表代码 Agent 的长任务能力，Manus 展示围绕工具、缓存和文件系统的 Agent 优化，Kiro 代表从 Vibe Coding 走向 Spec-driven development 的流程化上下文。案例不是为了背产品细节，而是抽象可复用原则。

你现在应该能做一件事：看到一个产品能力，不只说“模型强”，而是问它给模型维护了哪些上下文、如何减少噪声、如何把目标留在近处、如何验证输出。

Claude Code

重视项目记忆、文件检索、压缩、实时 steering 和子任务隔离。

Manus

围绕 KV cache、工具遮蔽、文件系统记忆、TODO 复述和错误保留做优化。

Kiro

把 prompt 变成 requirements、design、tasks，再生成代码，让规范本身成为上下文。

Claude Code：按需探索和项目记忆

代码 Agent

Claude Code 的有效性不能只归因于模型。公开材料里反复出现几个上下文工程动作：用 CLAUDE.md 或项目文件保存长期规则；用 grep、glob、文件读取做 just-in-time retrieval；用 compaction 保留长任务连续性；用子任务隔离做并行探索或专项检查。

代码仓库尤其适合按需探索，因为文件名、目录、测试、提交历史和错误日志本身就是强信号。把整个仓库塞进窗口通常不现实，也不稳定。你现在应该能解释：代码 Agent 的上下文不是“所有代码”，而是目标、索引、相关文件、最近证据和可继续搜索的工具。

示例：代码 Agent 的上下文入口text
Goal: fix intermittent 403 after login.
Always read: CLAUDE.md, failing test output.
Likely files: src/auth/session.ts, src/api/login.ts, tests/auth/
Use grep for: "403", "session", "csrf", "refresh_token".
Keep a TODO list and update it after each tool result.

Manus：缓存、工具遮蔽、文件系统和错误证据

Agent 优化

Manus 案例强调一个很工程的点：Agent 输入往往远大于输出，稳定前缀可以利用 KV cache 降低成本和延迟。因此动态增删工具、插入时间戳、JSON 键顺序不稳定，都会破坏缓存收益。工具太多又会让模型选错，所以 Manus 倾向“遮蔽工具”而不是频繁移除工具定义。

另一个原则是把文件系统当作外部上下文。网页、文档、表格、图片和中间产物不必一直留在窗口，只要路径和摘要保留，Agent 就能需要时再读。错误也不应该全部抹掉，失败尝试和堆栈能降低重复犯错概率。你现在应该能判断：成本、延迟和上下文稳定性也是上下文工程的一部分。

做法解决的问题注意点
稳定前缀 / KV cache降低长输入成本和首 token 延迟避免动态时间戳和不确定序列化
工具遮蔽减少误选，同时保留工具定义稳定需要状态机或响应约束支持
文件系统记忆保存大对象和中间产物路径、摘要、命名必须可检索
TODO 复述把目标推回近处，减少中途偏航不能变成重复噪声
保留错误让模型看到失败证据要标注结论，避免 poisoning

Kiro：Spec-driven 把规范变成上下文

开发流程

Vibe Coding 的问题不是“凭感觉”一定错，而是 Prompt -> Code 的链路太短。用户一次说不清需求，模型直接生成代码，缺少需求、设计、任务拆解、测试和验收约束。Spec-driven development 把中间层显式化：Prompt -> Requirements -> Design -> Tasks -> Code。

从上下文工程角度看，spec 是高信号、可复查、可迭代的上下文。它比散落在聊天里的补丁更稳定，也能给后续生成、测试和审查提供共同依据。你现在应该能把“写规范”理解成上下文管理，而不是额外文档负担。

Prompt用户原始意图，通常含糊。
Requirements把目标、边界和验收写清。
Design / Tasks拆解架构、接口和执行步骤。
Code在明确上下文约束下生成和验证。

上下文工程要能验收

评估

上下文策略不能只靠感觉。LangChain 提醒先做 observability 和 evaluation：看每轮模型实际收到了什么、用了多少 token、调用了哪些工具、延迟和成本怎样、结果质量是否变好。没有这些证据，很难判断一次压缩、检索或工具过滤到底是在帮忙还是在伤害。

你现在应该能写一份验收清单：它覆盖可靠性、成本、延迟、工具选择、记忆污染、隐私和回归测试。上下文工程不是“加更多机制”，而是让 Agent 在更少噪声、更清楚目标、更可验证的轨迹里工作。

验收项看什么证据常见失败
可靠性任务成功率、回归集、人工评审只看单个 demo
成本/延迟token、缓存命中、工具调用次数上下文更聪明但成本爆炸
工具选择误选率、重试率、无效调用工具描述重叠
记忆质量召回准确率、过期规则、用户可控性误召回隐私或旧偏好
压缩质量复杂轨迹复现、关键事实召回摘要短但丢约束

常见反模式

避坑

第一个反模式是“越多越好”：把所有文档、所有工具、所有历史都塞进去。第二个反模式是“越短越好”：摘要把关键约束删掉。第三个反模式是“越自动越好”：长期记忆自动注入，却没有用户控制和过期规则。第四个反模式是“越多 Agent 越强”：协调成本和信息损耗没有被评估。

你现在应该能识别这些反模式，并给出修法：建立来源和优先级，给记忆加召回条件，给工具加清晰边界，用评估验证压缩和隔离，不让系统在复杂机制里自我感动。

反模式 1：把窗口当仓库。修法：窗口只放下一步决策需要的材料。
反模式 2：摘要只追求短。修法：保留目标、决策、约束、失败和下一步。
反模式 3：记忆自动注入不可控。修法：召回可解释、可删除、可过期。
反模式 4：多 Agent 不评估。修法：只在边界清楚、可压缩汇报时隔离。

小型工程 Playbook

行动清单

落地时可以按六步走。先画出当前 Agent 每轮看到的上下文；再标出失败来自缺信息、噪声、冲突还是工具误选；然后决定哪些信息写出去，哪些按需检索，哪些压缩，哪些隔离；接着加日志和评估；最后再优化缓存、延迟和用户控制。

你现在应该能从一个模糊需求开始，做出一份上下文策略草案，而不是直接写一个万能 prompt。

步骤动作产物
1记录每轮实际上下文trace / token 报告
2分类失败原因缺失、噪声、冲突、误选
3设计四类策略write / select / compress / isolate 表
4实现最小改动一个 middleware 或工具策略
5评估回归成功率、成本、延迟对比
6固化规范README、规则文件、spec、测试集
