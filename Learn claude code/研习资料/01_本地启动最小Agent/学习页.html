这一节解决一个很朴素的问题：为什么一个命令行程序接到“写一个 Hello World 并运行”的提示后，能真的在本地创建文件、执行程序，而不是只返回一段聊天文本。

1. Agent 和普通聊天有什么不同

从聊天到行动

普通聊天模型只负责生成文本。你问它“写一个 Hello World”，它通常会给一段代码，让你自己创建文件、粘贴代码、运行命令。Agent 多了一层本地程序：模型仍然负责判断下一步该做什么，但真正执行创建文件、运行命令、读取目录这些动作的是你本机上的代码。

这句话很重要：模型不会自己伸手改你的磁盘。它只会在响应里提出一个动作请求，比如“调用 Bash 执行这个命令”。本地 Agent 程序收到请求后，再决定是否执行、怎么执行、如何把结果写回上下文。以后看到工具调用时，先分清“模型提出动作”和“本地代码执行动作”。

最小链路

  1. 用户输入任务。
  2. Agent 把任务发给模型。
  3. 模型返回文本或工具调用请求。
  4. 本地程序执行允许的工具。
  5. 执行结果回到对话，模型继续判断。

你现在应该能判断：Agent 的能力来自模型和本地工具的配合，不是单靠模型“更聪明”。

不要把 demo 跑通误认为已经懂了

Hello 示例看起来很简单，但它已经包含后续所有章节的雏形：消息历史、工具列表、工具执行、结果回写、循环退出。第一节只要求你建立入口感，不要求一下子读懂每个源码分支。

2. Quick Start 真正在做什么

先让系统能跑

项目 README 里的 Quick Start 不是课程的附录，而是本节的第一条主线。它会让你拿到代码、安装依赖、配置模型接口，然后运行第一个 Python 文件。只要这几步有一步出错，后面的源码调试都会变成猜。

Quick Start 的关键不是把命令背下来，而是知道每条命令改变了什么。克隆或进入仓库，是确定工作目录。安装依赖，是让 Python 能 import 需要的包。配置环境变量，是告诉程序把请求发到哪个模型服务。运行 `s01_agent_loop.py`，才是启动最小 Agent。

正确示例：启动前检查思路PowerShell / Bash
# 1. 进入项目目录，后续生成文件会受当前目录影响
cd learn-claude-code

# 2. 检查 Python 和依赖环境
python --version

# 3. 配置模型服务需要的环境变量
# 名称以项目 README 为准，这里只展示三类信息
ANTHROPIC_API_KEY=你的密钥
ANTHROPIC_BASE_URL=兼容 Anthropic 协议的接口地址
ANTHROPIC_MODEL=你要调用的模型名

# 4. 运行第一节 Agent
python s01_agent_loop.py

如果你使用的是 Windows，PowerShell、Git Bash、cmd 对环境变量和命令写法的支持不完全一样。字幕里作者不喜欢 PowerShell，改用 Git Bash 演示，这不是知识点本身，只是提醒你：同一个项目在不同 shell 里可能需要不同写法。

动作它解决什么常见错法
进入项目目录让脚本、相对路径、生成文件都落在预期位置。在上级目录运行，结果找不到文件或文件生成在别处。
配置 API Key让模型服务知道是谁在调用。把 Key 写错、带空格、放错 shell。
配置 Base URL告诉 SDK 请求发到哪个兼容接口。以为兼容 Anthropic 接口就必须使用 Anthropic 官方服务。
配置 Model指定实际调用哪个模型。模型名和服务商后台不一致。

3. 三个配置项不要混

Key / URL / Model

本地 Agent 要调用大模型服务，至少需要三类信息。API Key 是身份凭证，Base URL 是服务入口，Model 是具体模型名。它们经常一起出现，但作用完全不同。把 Key 写对但 URL 错了，请求会发错地方；URL 对了但模型名错了，服务也可能拒绝。

“Anthropic 兼容接口”指的是请求格式兼容，不等于服务商只能是 Anthropic。字幕里提到可以用阿里云百炼、Kimi、DeepSeek 等平台，只要它们提供兼容接口，项目代码就可以用相似方式访问。这里要检查的是协议和字段，不是品牌名称。

配置检查器

输入三项配置的状态，观察 Agent 启动时最可能卡在哪里。

可以进入模型调用

配置通过后，程序才有机会把用户任务发送给模型。下一步要观察模型返回的是文本还是工具调用。

判断顺序：先看身份凭证，再看请求地址，最后看模型名和服务商后台是否一致。
错误示例：把 Base URL 当成模型名配置混淆
ANTHROPIC_API_KEY=sk-xxxx
ANTHROPIC_BASE_URL=glm-4.5
ANTHROPIC_MODEL=https://example.com/anthropic-compatible

这段配置的错因很直接：地址和模型名放反了。后果是 SDK 可能把 `glm-4.5` 当作请求地址，连接阶段就失败。修法是回到服务商后台，分别复制接口地址和模型名称，不要凭感觉填写。

4. Hello 示例要观察什么

文件、命令、目录

启动 S01 后，可以输入类似“实现一个 hello world 的 Python 程序并跑起来”。如果 Agent 正常工作，你应该观察到两类结果：终端里出现模型和工具执行过程，项目目录里出现新生成的 Python 文件。不要只看终端输出“成功”，还要确认文件到底生成在哪里。

运行目录是本节最容易被忽略的细节。你在哪个目录启动 Agent，它使用相对路径创建文件时，就会以那个目录为基准。字幕里作者发现文件似乎在 S01 目录里运行，又切到外面运行 `python hello.py` 检查。这一步的意义是确认“文件真实存在，并且可以被普通 Python 命令运行”。

正确示例：用最小任务触发本地动作Prompt
实现一个 hello.py 文件，内容是打印 Hello World，并运行它。

看终端

确认 Agent 有没有进入模型调用，有没有请求执行命令，有没有返回运行结果。

看目录

确认 `hello.py` 是否生成在当前工作目录，而不是你以为的目录。

再手动运行

用普通命令运行生成文件，排除只是 Agent 输出“我做了”的情况。

逐行拆解：手动复验生成结果Shell
ls
python hello.py

# 如果文件不在当前目录，先查找
find . -name "hello.py"

你现在应该能做一个基本复查：只要 Agent 声称生成并运行了文件，你就去目录里找文件、打开内容、手动运行。Agent 学习不能停在“它说成功了”，要训练自己检查真实副作用。

5. 启动失败时先查哪几处

不要盲目重跑

最小 Agent 启动失败通常不是大问题，真正消耗时间的是没有顺序地乱改。建议按“目录、Python、依赖、环境变量、模型接口、任务描述”排查。每一步只验证一个条件，避免同时改很多东西后不知道是哪一步修好了。

现象优先检查修法
找不到脚本文件当前目录是否是项目目录，文件名是否和 README 一致。先 `pwd` / `ls`，再运行脚本。
import 报错依赖是否安装到当前 Python 环境。确认虚拟环境和安装命令，不要混用多个 Python。
401 或鉴权失败API Key 是否存在、是否多了空格、是否过期。重新从平台复制 Key，按当前 shell 规则设置。
连接失败Base URL 是否能访问，是否是兼容接口。检查服务商文档和网络，不要把网页地址当 API 地址。
模型返回但没有创建文件工具能力是否开启，模型是否只生成了文本。输入更明确的任务，并检查后续工具调用章节。

第一节不要求你修完所有 Agent 问题。它的边界是：能启动，能调用模型，能用最小任务观察本地副作用。如果卡在工具调用细节，先记下来，下一节会进入消息循环和工具调用的源码。
