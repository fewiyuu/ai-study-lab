Tool dispatch

多工具分发与专用工具

S01 只有 Bash，代码可以硬编码 `run_bash`。S02 有多个工具，就需要一个分发映射：模型返回工具名，本地程序根据工具名找到对应函数，再把 input 参数交给它。

block.type→ block.name→ handlers.get→ run_read_file→ tool_result

学完本页，你应该能解释 `type=tool_use` 和 `name=read_file` 的分工，能读懂 handler map，也能说出为什么专用工具比一把 Bash 更安全、更可控。

本页以 `read_file(requirements.txt)` 为主例，讲工具分发和路径安全。

1. S01 到 S02 的核心变化

hardcode → dispatch

S01 只有一个 Bash 工具，所以代码看到 `tool_use` 后，可以直接调用 `run_bash`。这在最小 demo 里够用，但一旦有多个工具，硬编码就会失控。模型可能请求 `read_file`、`write_file`、`bash` 或其他工具，本地程序必须先看工具名，再决定调用哪个函数。

S02 的核心变化可以压缩成一句话：从“所有工具调用都走 Bash”改成“用 `block.name` 查 handler 映射”。这让新增工具变得清晰：你给模型暴露工具 schema，再在本地 handlers 字典里注册同名函数。

错误示例：只有 Bash 时的硬编码思路S01
if block.type == "tool_use":
    output = run_bash(block.input["command"])
正确示例：多工具分发思路S02
handlers = {
    "bash": run_bash,
    "read_file": run_read_file,
    "write_file": run_write_file,
}

if block.type == "tool_use":
    handler = handlers.get(block.name)
    output = handler(**block.input)

判断句：`block.type` 只告诉你“这是工具调用请求”，`block.name` 才告诉你“请求的是哪个工具”。

2. handler map 怎么选函数

name lookup

handler map 本质上是 Python 字典。键是模型会返回的工具名，值是本地函数。模型返回 `name="read_file"` 时，本地程序用 `handlers.get("read_file")` 找到 `run_read_file`。如果找不到，就应该返回明确错误，而不是默默忽略。

工具名必须统一。工具 schema 里叫 `read_file`，handler 字典也要有 `read_file`。如果 schema 暴露的是 `read_file`，本地却注册成 `readFile`，模型请求就会找不到 handler。命名一致是多工具系统最基础的契约。

选择模型返回的工具名

命中 run_bash

工具名是 bash，handler map 能找到本地的 run_bash 函数。下一步要检查 input 里是否有 command。

观察点：命中 handler 只是开始，参数校验和安全检查仍然要做。
错误示例：没有处理未知工具分发漏洞
handler = handlers.get(block.name)
output = handler(**block.input)

# 如果 block.name 不存在，handler 是 None。
# 这会在运行时抛错，而且错误信息不一定对模型有帮助。

更稳的写法是：找不到 handler 时，返回一个明确的 tool_result，告诉模型“这个工具不可用”。这样模型可以改用别的工具，或者向用户说明限制。

3. read_file 为什么是专用工具

specific tool

`read_file` 是一个专用工具。它只负责读取文件，不负责执行任意命令。模型返回 `name="read_file"`，同时在 input 里提供 `path`，本地程序把 path 传给 `run_read_file`。函数内部先做路径安全检查，再用 Python API 读取文本。

这比让模型通过 Bash 执行 `cat requirements.txt` 更可控。Bash 能做的事太多，既能读文件，也能删除文件、发网络请求、执行脚本。专用工具把能力收窄，让本地程序更容易检查参数、限制目录、返回稳定格式。

正确示例：read_file 的最小执行链Python
def run_read_file(path: str) -> str:
    safe = safe_path(path)
    return safe.read_text(encoding="utf-8")

block = ToolUseBlock(
    type="tool_use",
    name="read_file",
    input={"path": "requirements.txt"},
)

handler = handlers.get(block.name)
output = handler(**block.input)
对象负责什么不负责什么
模型选择 `read_file`，给出 path。不直接读取磁盘。
handler map把 `read_file` 映射到 `run_read_file`。不做业务逻辑。
`run_read_file`校验路径并读取文件。不执行任意 shell 命令。
tool_result把文件内容或错误反馈给模型。不替模型做最终总结。

4. safe_path 防的是什么

path traversal

路径参数不能完全相信模型。即使模型不是恶意的，它也可能生成 `../` 这样的路径，试图访问工作目录外的文件。`safe_path` 的目的就是把路径限制在允许的工作区里，防止路径穿越。

路径穿越的典型形式是 `../../secret.txt`。如果程序直接读取这个路径，就可能访问到项目外的敏感文件。稳妥做法是把用户提供的相对路径解析成绝对路径，再确认它仍然位于工作目录下。

正确示例：路径限制的思路Python
from pathlib import Path

WORKSPACE = Path.cwd().resolve()

def safe_path(path: str) -> Path:
    candidate = (WORKSPACE / path).resolve()
    if not str(candidate).startswith(str(WORKSPACE)):
        raise ValueError("Path is outside workspace")
    return candidate
错误示例：直接信任 path路径穿越
def run_read_file(path: str) -> str:
    return Path(path).read_text()

# 如果 path 是 "../../secret.txt"，程序可能读到工作区外的文件。

判断句：工具 schema 只能告诉模型 path 字段应该存在，不能保证 path 安全。安全判断必须在本地函数里做。

5. 为什么不用 Bash 包打天下

safe and controllable

Bash 的优点是通用，缺点也是通用。它几乎什么都能做，所以很难精确限制。专用工具把能力拆小：读文件就只读文件，写文件就只写文件，列目录就只列目录。能力越窄，参数越清楚，安全边界越容易检查。

全 Bash 的问题

  • 命令字符串难校验。
  • 不同系统命令差异大。
  • 容易混入危险操作。
  • 返回格式不稳定。

专用工具的好处

  • 输入字段明确。
  • 能限制工作目录。
  • 返回结构更稳定。
  • 新增能力更容易审计。

这不是说 Bash 不能用，而是不能把所有能力都压到 Bash 里。最小 Agent 可以先用 Bash 解释工具调用原理；真正扩展时，要逐步增加专用工具。
