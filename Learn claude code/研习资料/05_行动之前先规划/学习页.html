本页聚焦 todo 工具、状态校验、计划渲染和定期更新提醒。

1. 计划要显式，不要只藏在回复里

visible state

没有 todo 工具时，模型也可能在自然语言里说“我先做 A，再做 B”。问题是这种计划不稳定：下一轮模型可能忘掉、改写或跳过某一步。todo 工具把计划变成结构化工具调用，结果会进入 messages，后续每一轮都能看到当前任务状态。

这不是要求模型暴露隐藏思维链。todo 是可见的任务状态，和内部推理不同。它告诉用户和程序：现在有哪些任务，哪个正在做，哪些还没开始，哪些已经完成。你可以把它当作 Agent 的小看板。

只靠自然语言计划的问题

  • 状态不稳定，下一轮可能忘。
  • 用户难以检查当前进度。
  • 模型容易同时推进多条线。
  • 失败后不知道该回到哪一步。

todo 工具带来的控制

  • 计划以结构化 items 存在。
  • 每项都有 id、text、status。
  • 渲染后能直观看到当前任务。
  • 更新结果会写回 messages。

2. schema 告诉模型 todo 需要什么参数

id / text / status

模型之所以知道 todo 工具要返回 `items`，并且每个 item 里有 `id`、`text`、`status`，是因为工具 schema 已经把参数结构告诉了它。schema 的作用不是执行计划，而是约束模型生成符合本地函数预期的输入。

本节里的 todo item 至少有三部分：`id` 用来稳定识别任务；`text` 描述任务内容；`status` 表示状态。常见状态是 `pending`、`in_progress`、`completed`。如果 text 为空，或者 status 不是允许值，本地 update 函数应该拒绝或返回错误。

正确示例：todo 工具输入结构JSON
{
  "items": [
    {"id": "1", "text": "查看当前目录", "status": "in_progress"},
    {"id": "2", "text": "读取 requirements.txt", "status": "pending"},
    {"id": "3", "text": "总结依赖用途", "status": "pending"}
  ]
}
字段作用常见坏输入
`id`稳定追踪任务。重复、缺失、每次更新都换。
`text`描述任务内容。空字符串、过度笼统。
`status`表示进度状态。拼错、发明新状态、多个 in_progress。

3. update 不是照单全收

validate then render

todo update 函数会把传入的 items 逐个提取出来，检查 text 是否为空，status 是否属于允许集合，再把合法项放进 validated 列表。校验之后才渲染成用户能看的计划。这个顺序很重要：先校验，再展示，再写回 messages。

逐行拆解：校验 todo itemsPython
allowed = {"pending", "in_progress", "completed"}
validated = []

for item in items:
    task_id = item.get("id")
    text = item.get("text", "").strip()
    status = item.get("status")

    if not text:
        raise ValueError("Todo text cannot be empty")
    if status not in allowed:
        raise ValueError(f"Unknown status: {status}")

    validated.append({"id": task_id, "text": text, "status": status})

渲染时，当前任务可以用 `>` 标出来，完成任务可以用 `[x]`，待办任务用 `[ ]`。符号不是重点，重点是让用户和模型都能稳定看到同一个计划状态。

切换 todo 状态

校验通过

只有一个任务处于 in_progress，其他任务是 pending 或 completed，可以渲染并写回 messages。

观察点：todo 是给下一轮模型看的状态，不只是给用户看的进度条。

4. 约束让计划不变成噪声

guardrails

todo 工具有几个简单但有效的约束。第一，状态只能来自允许集合。第二，同一时间最多一个任务是 `in_progress`。第三，计划项不能无限多，示例里限制不超过 20 条。它们不是形式主义，而是在减少 Agent 的失控空间。

最多一个 `in_progress` 是为了避免一种隐蔽失败：模型看起来同时做很多事，实际上每条线都没收束。20 条上限是为了避免模型把任务拆成过多碎片。计划太细会脆弱，模型和用户都难以维护。

状态集合

`pending`、`in_progress`、`completed`。不允许随便发明状态。

单一当前任务

同一时间最多一个 `in_progress`，让当前焦点清楚。

条目上限

限制计划长度，防止拆得太细导致计划本身难维护。

错误示例：多个任务同时进行状态冲突
[
  {"id": "1", "text": "查看目录", "status": "in_progress"},
  {"id": "2", "text": "读取文件", "status": "in_progress"},
  {"id": "3", "text": "总结结果", "status": "pending"}
]

修法不是简单删掉一个任务，而是让计划反映真实顺序：当前只做一项，做完后再把下一项切到 `in_progress`。

5. runs_since_todo 是更新提醒

stay aligned

如果模型调用了 todo 工具，`runs_since_todo` 会重置为 0。之后每一轮如果调用的是 Bash、read_file、write_file 等其他工具，就加 1。达到阈值后，程序会向结果里加入类似“更新你的 todo”的提醒，迫使模型回头同步计划状态。

这不是为了烦模型，而是为了防止它越做越远。复杂任务中，模型可能连续执行几个工具后忘记更新计划。提醒机制把它拉回到可见计划，让 completed 和 in_progress 状态及时变化。

轮次工具`runs_since_todo`结果
1todo0生成初始计划。
2bash1执行目录命令。
3bash2换命令继续检查。
4bash3加入更新 todo 的提醒。
5todo0计划状态更新。

判断句：todo 更新不是任务执行本身，它是任务控制机制。它让 Agent 的行动可见、可追踪、可重新对齐。
