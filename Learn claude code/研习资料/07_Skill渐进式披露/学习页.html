Skill渐进式披露

Skill 是一份可被 Agent 调用的能力说明书。它通常写在 `SKILL.md` 里，上半部分是 `name`、`description` 等元数据，下半部分是完整工作流。渐进式披露的做法是：启动时只把元数据放进 system prompt，等模型判断需要某个 Skill 并调用 `load_skill` 时，再把这个 Skill 的完整 body 加入 messages。

scan SKILL.md→ parse metadata/body→ send descriptions→ load_skill→ send body

学完本页，你应该能解释 `SkillLoader` 如何扫描和解析 Skill，为什么不要一开始发送所有 body，以及 `load_skill` 工具如何让模型按需获得详细工作流。

本页聚焦 Skill 文件结构、递归加载、metadata/body 分离、`load_skill` 工具和上下文管理。

1. Skill 是可加载的能力说明书

SKILL.md

Skill 的直觉是“给 Agent 的操作手册”。当用户提出某类需求时，Agent 可以加载对应 Skill，获得更具体的步骤、约束、工具使用规则和输出规范。比如 PDF Skill 会告诉 Agent 如何创建 PDF，Code Review Skill 会告诉 Agent 如何审查代码。

一个典型 `SKILL.md` 可以分成两层。第一层是 frontmatter，也就是文件开头用 `---` 包住的元数据，包含 `name` 和 `description`。第二层是 body，也就是元数据下面的完整正文，写具体工作流。渐进式披露依赖的正是这次拆分：元数据短，适合一开始给模型；body 长，适合需要时再加载。

最小示例：SKILL.md 文件结构Markdown
---
name: pdf
description: Create PDF documents from text, Markdown, or generated content.
---

# PDF Skill

1. Clarify the target format and page size.
2. Generate a source document.
3. Render or compile it into a PDF.
4. Verify the output file exists and is readable.

metadata 适合做什么

  • 让模型快速知道有哪些 Skill。
  • 帮助模型判断用户需求匹配哪个 Skill。
  • 构建短小的 Skill 列表。

body 适合做什么

  • 提供完整执行步骤。
  • 写清工具限制和验收标准。
  • 在真正需要时进入上下文。

你现在应该能做什么：看到一个 Skill 文件时，能指出哪部分用于“发现能力”，哪部分用于“执行能力”。

2. `SkillLoader` 启动时递归扫描

load_all

程序启动时会创建一个 `SkillLoader` 对象，并指向当前工作目录下的 `skills` 目录。因为 Skill 可能放在多层子目录里，所以加载器会递归查找所有 `SKILL.md` 文件。视频里示例项目有多个 Skill，例如 agent builder、code review、MCP builder、PDF。

扫描到文件后，加载器读取文本，解析 frontmatter 和 body，再把结果放进一个字典。这个字典的 key 通常是 Skill 的 `name`，value 则包含 metadata、body 和 path。这样后续模型请求加载某个 Skill 时，可以通过名字快速找到完整内容。

逐行拆解：扫描并登记 SkillPython
class SkillLoader:
    def __init__(self, skills_dir):
        self.skills_dir = skills_dir
        self.skills = {}
        self.load_all()

    def load_all(self):
        for path in sorted(self.skills_dir.rglob("SKILL.md")):
            text = path.read_text(encoding="utf-8")
            metadata, body = parse_frontmatter(text)
            name = metadata["name"]
            self.skills[name] = {
                "metadata": metadata,
                "body": body,
                "path": str(path),
            }

这里的 `rglob("SKILL.md")` 表示递归匹配当前目录和所有子目录里的 `SKILL.md`。这比只扫描第一层目录稳，因为真实项目里的技能经常会按主题或来源分层存放。

步骤做什么为什么必要
创建 loader初始化 `self.skills = {}`。准备一个可按名称查询的 Skill 索引。
递归扫描查找所有 `SKILL.md`。覆盖多层目录，不漏技能。
解析文件拆出 metadata 和 body。为渐进式披露做结构分离。
登记字典按 `name` 存 metadata/body/path。后续 `load_skill` 可以快速读取。

3. 第一轮只给模型 Skill 描述

descriptions only

加载器已经把所有 Skill 文件读到本地，但这不等于要把所有内容立刻发送给模型。`get_descriptions()` 会从字典中抽取每个 Skill 的 `name` 和 `description`，拼成简短列表，放到 system prompt 里。这样模型第一次请求时，只知道“有哪些 Skill、各自适合什么场景”。

这一步是渐进式披露的第一层：披露能力目录，而不是披露全部说明书。假设只有 4 个 Skill，把 body 都发给模型也许还能忍；但如果有 100 个、1000 个 Skill，一开始就塞完整 body 会让上下文窗口迅速被占满，模型也更难判断当前任务真正相关的信息。

正确示例：只组装 Skill 摘要Python
def get_descriptions(self):
    lines = []
    for name, skill in self.skills.items():
        desc = skill["metadata"].get("description", "")
        lines.append(f"- {name}: {desc}")
    return "\n".join(lines)
错误示例：启动时塞入所有 body上下文浪费
system_prompt = "Available skills:\n"
for skill in loader.skills.values():
    system_prompt += skill["body"]  # 错误：每次都发送完整工作流

坏处不是“代码会立刻崩”，而是上下文工程变差。模型要在大量无关流程里找相关信息，后续响应质量可能下降，成本和延迟也会增加。

4. 需要时再调用 `load_skill`

tool gated body

当用户说“创建一个 PDF，内容是你好 Skill”，模型会先看 system prompt 里的 Skill 描述，判断 PDF Skill 相关，于是返回一个 tool use block：调用 `load_skill`，输入是 Skill 名称，例如 `PDF` 或 `pdf`。本地工具收到名字后，从 loader 字典里取出对应 body。

然后 body 会作为工具结果加入 messages。下一轮模型再请求时，messages 里已经有三类关键信息：用户需求、模型调用 `load_skill` 的动作、`load_skill` 返回的完整 Skill 工作流。此时模型才真正拥有执行 PDF 任务所需的详细步骤。

正确示例：load_skill 工具Python
def load_skill(name: str) -> str:
    skill = loader.skills.get(name)
    if not skill:
        return f"Skill not found: {name}"
    return skill["body"]

上下文披露切换

上下文占用：低。模型只看到 Skill 名称和描述。

先给能力目录

模型根据简短描述判断是否需要某个 Skill。它还没有看到完整 body，所以上下文保持轻。

观察点：渐进式披露不是不让模型知道能力，而是延迟发送详细说明。

你现在应该能做什么：看到一次 `load_skill` 调用时，能说出它为什么发生在“匹配 Skill 之后、执行任务之前”。

5. 渐进式披露是一种上下文工程

context engineering

上下文工程不是只追求少发字，而是把信息放在合适的时间、以合适的粒度交给模型。Skill 系统的做法很典型：启动时发短目录，任务匹配后发完整说明，执行时再根据工具结果继续推进。

这和前面几节可以连成一条线：工具 schema 让模型知道能调用什么；todo 把计划变成可见状态；子 Agent 把复杂子任务隔离出去；Skill 则把大量工作流说明按需加载。它们都在解决同一类问题：Agent 不只是“会聊天”，还要管理上下文、状态、工具和权限。

少量初始信息

只发送 `name` 和 `description`，用于发现匹配能力。

按需详细说明

模型调用 `load_skill` 后，再把完整 body 放进 messages。

执行后继续循环

模型根据 Skill 工作流调用 write、bash 等工具，直到完成任务。

常见坑集中看

  • 把 Skill body 当成 description，导致目录过长。
  • 解析 frontmatter 不稳，`name` 或 `description` 丢失。
  • 只扫描 skills 第一层目录，漏掉子目录里的 `SKILL.md`。
  • `load_skill` 对大小写、别名和不存在的 Skill 没有处理。
  • 加载 Skill 后不把结果写回 messages，模型下一轮仍然看不到完整工作流。
