上下文压缩与无限会话

模型上下文窗口有上限，Agent 的 messages 却会随着工具调用不断增长。本节的压缩机制分三层：micro compact 替换旧的长 tool_result；客户端按 token 阈值自动触发 auto compact；模型也可以调用 `compact` 工具手动触发压缩。压缩后，长历史会被摘要替代，会话得以继续。

messages grow→ micro compact→ estimate tokens→ auto compact→ summary resume

学完本页，你应该能解释旧工具结果如何被替换、为什么要估算 token、auto compact 如何把 messages 变成摘要，以及压缩可能丢失哪些关键约束。

本页聚焦 micro compact、token 阈值、manual compact、transcript 保存、摘要替换和压缩风险。

1. 为什么 Agent 需要上下文压缩

bounded window

Agent 循环会不断把用户消息、模型响应、tool use、tool result 写入 messages。读取文件、列目录、运行命令这类工具结果可能很长。如果用户让 Agent “逐个读取目录下所有 Python 文件”，messages 很快就会堆满大量历史输出。

模型的上下文窗口不是无限的。超过窗口后，模型无法完整接收历史；即使没超过，大量旧工具结果也会稀释当前任务重点。上下文压缩的目标不是美化历史，而是减少无关细节，让 Agent 继续保留足够的任务连续性。

不压缩的后果

  • 旧 tool_result 占用大量上下文。
  • 后续请求成本和延迟变高。
  • 模型被无关输出干扰，任务焦点变弱。
  • 长会话最终触达窗口上限。

压缩后的收益

  • 保留最近关键结果。
  • 旧细节替换为短标记或摘要。
  • 会话能继续推进。
  • 父循环更容易保持任务连续性。

直觉上，micro compact 是“把旧工具输出缩短”，auto compact 是“把整段对话改写成续航摘要”。前者轻，后者重；前者保留 messages 结构，后者会重建 messages。

2. micro compact：旧工具结果替换成短标记

tool_result cleanup

micro compact 每轮 Agent loop 开始时先跑。它会遍历 messages，找到 content 是 list 的 user 消息，再从里面收集 `tool_result`。收集时不仅保存结果本身，还保存 message index 和 part index，因为后面要回到原 messages 里原地替换。

它不会处理所有 tool_result。示例里 `keep_recent = 3`，也就是最近三条工具结果保留原文，旧结果才进入待清理列表。这样模型仍能看到最近工具调用的真实输出，不至于刚读完的结果马上被压掉。

逐行拆解：收集旧 tool_resultPython
tool_results = []

for message_index, message in enumerate(messages):
    if message["role"] != "user":
        continue
    if not isinstance(message.get("content"), list):
        continue

    for part_index, part in enumerate(message["content"]):
        if part.get("type") == "tool_result":
            tool_results.append((message_index, part_index, part))

然后它会构造 tool use id 到工具名的映射。原因是不同轮次即使都调用 Bash，tool use id 也不同；替换时要知道某个旧结果来自哪个工具，才能写成 `previously used bash` 这类可读标记。

正确示例：只清理旧且过长的结果Python
to_clear = tool_results[:-keep_recent]

for message_index, part_index, part in to_clear:
    content = part.get("content", "")
    tool_use_id = part.get("tool_use_id")
    tool_name = tool_name_map.get(tool_use_id, "tool")

    if isinstance(content, str) and len(content) > 100:
        messages[message_index]["content"][part_index]["content"] = (
            f"previously used {tool_name}"
        )

这个操作很粗，但很有效：几千字符的命令输出会变成十几个字符。模型仍然知道“这里曾经用过 bash”，但不再反复读取旧输出的细节。

3. token 估算决定是否进入重压缩

estimate threshold

micro compact 后，程序会估算 messages 消耗多少 token。示例里用的是粗略算法：把 messages 转成字符串，长度除以 4。真实 tokenizer 会更复杂，英文、中文、符号和 JSON 结构都会影响 token 数；但工程上先用估算值可以快速判断是否接近阈值。

视频里阈值示例是 5 万 token。低于阈值，就继续正常 Agent loop；高于阈值，就触发 auto compact，把长历史压成摘要。这个阈值不是绝对真理，而是保护上下文窗口的工程参数。

最小示例：token 粗估Python
def estimate_tokens(messages):
    return len(str(messages)) // 4

messages = micro_compact(messages)

if estimate_tokens(messages) > 50_000:
    messages = auto_compact(messages)
阶段触发条件处理强度保留信息
micro compact每轮先跑轻保留 messages 结构和最近工具结果。
auto compact估算 token 超过阈值重把历史浓缩成 continuity summary。
manual compact模型调用 `compact` 工具重走同一套 auto compact 流程。

你现在应该能做什么：看到一个长会话时，能判断“先用微压缩清旧工具结果，不够再用摘要压缩整段历史”。

4. auto compact：保存 transcript，再用摘要替换历史

summary resume

auto compact 是重压缩。它会先创建 `.transcripts` 目录，把当前完整 messages 序列化保存到磁盘。这样做有两个价值：第一，压缩后如果发现摘要漏了信息，还能回看原始 transcript；第二，审计时可以知道压缩前到底发生过什么。

保存后，程序构造一个给模型的摘要请求，大意是“summarize this conversation for continuity”，再附上 conversation text。示例代码只取 messages 文本的前 8 万字符用于摘要。模型返回 summary 后，原 messages 会被替换成两条左右的短消息：一条说明这是压缩后的历史，一条包含摘要。

逐行拆解：auto compact 的核心流程Python
def auto_compact(messages):
    transcript_dir.mkdir(exist_ok=True)
    transcript_path = transcript_dir / "conversation.json"
    transcript_path.write_text(json.dumps(messages, ensure_ascii=False))

    conversation_text = str(messages)[:80_000]
    response = call_model([
        {
            "role": "user",
            "content": "Summarize this conversation for continuity:\n"
                       + conversation_text,
        }
    ])
    summary = extract_text(response)

    return [
        {"role": "user", "content": "Previous conversation was compacted."},
        {"role": "assistant", "content": summary},
    ]

压缩前后 messages

上下文占用：高。旧工具输出仍在历史中。

长历史不断增长

多轮工具调用会让 messages 里塞入大量 tool_result。模型每次请求都要带上这些历史。

观察点：压缩目标是保留继续任务所需的信息，而不是保留所有原文。

`compact` 工具的手动触发也是这个思路。模型响应里出现 `compact` tool use 后，本地没有像 Bash 那样调用具体处理函数，而是把 `manual_compact` 标记设为 true，随后进入 auto compact 流程。

5. 压缩会带来信息丢失风险

lossy summary

压缩不是免费的。micro compact 会丢掉旧工具结果的具体内容；auto compact 更明显，它把一大段历史变成摘要。摘要能保留主线，但可能漏掉用户后面强调的限制、某个失败原因、某个关键文件路径或“不允许做什么”的约束。

视频里特别提醒：auto compact 只取前 8 万字符做摘要。如果关键要求出现在 8 万字符之后，它可能根本没进入摘要输入。即使进入了，模型摘要时也可能没保留下来。因此压缩系统需要配套风险控制：保存 transcript、让摘要明确保留约束、压缩后让模型复述当前任务状态。

保存原文

压缩前写入 `.transcripts`，方便事后审计和恢复。

摘要保留约束

prompt 要要求保留用户目标、禁止事项、当前进度和未解决问题。

压缩后校准

让模型确认下一步和关键限制，避免摘要漏信息后继续跑偏。

错误示例：摘要没有保留禁止事项压缩风险
# 原历史后段有一句：
用户：不要修改原始字幕文件，只能在 course-forge 目录新增资料。

# 压缩摘要只写：
用户正在制作 Learn Claude Code 学习资料。

# 后果：
模型可能误以为可以改原始字幕。

你现在应该能做什么：使用或设计压缩机制时，不只问“压缩了多少”，还要问“哪些信息不能丢”。
