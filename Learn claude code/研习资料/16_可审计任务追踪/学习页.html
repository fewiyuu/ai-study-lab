可审计任务追踪

上一节学的是 `git worktree` 命令。本节进入源码调试：系统如何发现 repo root，初始化 `.tasks` 和 `.worktrees`，在模型误调用工具时引导它先创建 task，再创建 worktree，并把每一次 worktree 创建前后写入 `events.jsonl`，同时把活跃 worktree 记录到 `index.json`。

detect repo root→ create task→ worktree create before→ git worktree add→ index + task + after log

学完本页，你应该能解释：为什么仅执行 Git 命令还不够，以及 Agent Harness 如何用 task JSON、worktree index、JSONL 事件日志把一次任务执行变成可审计记录。

重点：`git rev-parse --show-toplevel`、`.worktrees/events.jsonl`、`.worktrees/index.json`、task 的 `worktree/status/updated_at` 字段。

1. 先确定仓库根目录和可用性

bootstrap

系统启动时先执行 `git rev-parse --show-toplevel`，检测当前 Git 仓库根目录。这个 root 很重要，因为后面 `.tasks`、`.worktrees`、worktree 目录路径、Git 命令执行位置，都要围绕这个根目录展开。

随后初始化 task manager、event bus 和 worktree manager。task manager 创建 `.tasks`；event bus 创建 `.worktrees/events.jsonl`；worktree manager 创建 `.worktrees/index.json`。这些初始化不是装饰，它们定义了后续所有任务追踪数据要写到哪里。

启动阶段：检测 repo root 和 Git 可用性Shell / Python
git rev-parse --show-toplevel

def is_git_repo(repo_root):
    result = run_git(["rev-parse", "--is-inside-work-tree"], cwd=repo_root)
    return result.stdout.strip() == "true"

`.tasks/`

保存 task_1.json、task_2.json 等任务状态。

`.worktrees/events.jsonl`

追加事件，一行一个 JSON，用于审计。

`.worktrees/index.json`

保存当前 worktree 索引和状态。

你现在应该能做什么：读到初始化代码时，判断它是在准备任务状态、事件日志还是 worktree 索引。

2. 模型会先犯错，系统用工具结果纠偏

tool loop

视频里的 prompt 要求创建两个 worktree：一个做 auth refactor，一个做 UI login。模型第一轮直接调用 `worktree_create`，并传入 `task_id=1`、`task_id=2`。但此时 `.tasks` 目录是空的，task1 和 task2 还不存在。

工具没有默默创建 worktree，而是返回 `Task 1 not found`、`Task 2 not found`。下一轮模型看到结果后，先调用 `task_list` 确认没有任务，再调用 `task_create` 创建两个任务。这个过程说明：工具返回值是模型下一轮推理的重要上下文，错误结果也能把模型拉回正确流程。

任务创建后新增字段：worktree 和时间戳JSON
{
  "id": 1,
  "subject": "auth refactor",
  "description": "Refactor authentication system",
  "status": "pending",
  "owner": "",
  "worktree": "",
  "blocked_by": [],
  "created_at": "2026-04-06T...",
  "updated_at": "2026-04-06T..."
}

和更早的 task system 相比，这一节多了 `worktree`、`created_at`、`updated_at`。`worktree` 用来把任务和工作目录绑定；时间戳用来判断任务何时创建、何时更新，是审计的基本材料。

3. `events.jsonl` 记录“做命令前后发生了什么”

events.jsonl

当 `worktree_create` 真正开始执行时，系统不是立刻跑 `git worktree add`。它先向 `events.jsonl` 写一条 `worktree_create_before` 事件，记录 timestamp、task_id、worktree name、base_ref 等信息。然后执行 Git 命令。成功后，再写一条 `worktree_create_after`。

JSONL 是“JSON Lines”的缩写：每一行都是一个独立 JSON。它很适合事件日志，因为可以不断 append，不需要每次读写整个大 JSON 文件。审计时只要按行读取，就能看到事件发生顺序。

worktree_create_before

说明会显示在这里。

1. 先写 before 事件
2. 再执行 `git worktree add`
3. 更新 index 和 task
4. 最后写 after 事件
审计的关键是顺序：先留痕，再执行，再登记结果。即使命令失败，也应该能看到失败前的意图。
事件日志示例：创建前和创建后JSONL
{"event":"worktree_create_before","ts":"...","task_id":1,"worktree":{"name":"auth-refactor","base_ref":"HEAD"}}
{"event":"worktree_create_after","ts":"...","task_id":1,"worktree":{"name":"auth-refactor","branch":"wt/auth-refactor","status":"active"}}

4. `index.json` 和 task JSON 形成双向追踪

index

Git 命令成功后，系统把新 worktree 写入 `.worktrees/index.json`。这个文件通常包含一个 `worktrees` 数组，每个 entry 记录 name、path、branch、task_id、created_at、status。之后系统不用每次都解析 Git 输出，也能快速知道当前管理了哪些 worktree。

worktree index 示例JSON
{
  "worktrees": [
    {
      "name": "auth-refactor",
      "path": ".worktrees/auth-refactor",
      "branch": "wt/auth-refactor",
      "task_id": 1,
      "created_at": "2026-04-06T...",
      "status": "active"
    }
  ]
}

同时，系统还会更新 task JSON：把 `worktree` 字段写成对应 name，把 `status` 改成 `in_progress`，并刷新 `updated_at`。这样从 task 可以找到 worktree，从 worktree index 也能反查 task_id。

文件记录什么回答的问题
`.tasks/task_1.json`任务内容、状态、owner、worktree、时间戳这个任务现在由哪个工作目录执行？
`.worktrees/index.json`所有 managed worktree 的 name、path、branch、task_id、status系统当前管理了哪些隔离目录？
`.worktrees/events.jsonl`worktree 创建前后、时间戳、payload什么时候做过什么动作？动作前后发生了什么？

你现在应该能做什么：定位一个任务对应的 worktree，并说明哪些文件能支持审计和追踪。

5. 常见误区：只会创建目录还不够

pitfalls

如果系统只是执行 `git worktree add`，它确实创建了目录，但很难回答后续问题：这个目录对应哪个 task？是谁在什么时候创建的？创建前后有没有成功？任务状态有没有从 pending 变成 in_progress？如果这些问题回答不了，任务隔离就只能算“文件系统隔离”，还不是可审计的 Agent Harness。

脆弱做法

只运行 Git 命令，不更新 task，不写事件日志，也不登记 index。目录虽然存在，但系统视角里没有可追踪记录。

稳健做法

执行前写 before 事件，执行后写 index、绑定 task、写 after 事件。之后可以审计、恢复、清理和追踪。

错误示例：只有命令，没有追踪状态不可审计
run_git(["worktree", "add", ".worktrees/auth-refactor", "-b", "wt/auth-refactor", "HEAD"])

# 问题：
# 1. task_1.json 没有记录 worktree。
# 2. index.json 不知道这个目录存在。
# 3. events.jsonl 没有 before/after 日志。
# 4. 后续无法回答谁在什么时候为哪个任务创建了目录。
