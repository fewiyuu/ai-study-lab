1. 为什么复杂系统需要消息规范化

preflight

早期 Agent loop 很简单：用户发一条消息，模型返回文本或工具调用，系统执行工具，再把工具结果追加到 messages。此时直接把 messages 发给 API 往往能跑。

但系统变复杂后，消息里会混入更多内部结构：工具超时、用户取消、压缩替换、调试元数据、不同 SDK 返回的 block 对象、工具结果引用。内部结构对 Harness 有用，但 API 未必接受。API 是后端协议，它只接受规定字段和合法顺序；不合法就会返回 400。

直接发送内部消息

可能包含 `_iteration`、SDK 对象、空 content、未配对 tool_use 等 API 不接受的内容。

发送前规范化

把内部消息转换成 API 协议格式，提前消除可预期的 Bad Request。

你现在应该能做什么：判断消息规范化是“请求前预处理”，不是模型推理本身。

2. API 协议的三类约束

protocol

视频提到三个典型要求。第一，user 和 assistant 消息需要严格交替；如果连续两个 user 或连续两个 assistant，可能要合并。第二，assistant 发出的 tool_use 必须有对应的 user tool_result；否则模型看到了工具调用，却没有看到工具结果。第三，只能发送协议定义字段，内部元数据会导致 400。

约束为什么需要规范化动作
role 交替API 期望对话结构稳定。合并相邻同 role 消息。
tool_use 配 tool_result工具调用必须有结果闭环。收集已有 result id，检查缺失结果。
字段白名单API 不接受内部调试字段。过滤 `_` 开头或协议外字段。
协议直觉：只发送 API 能理解的消息JSON
[
  {"role": "user", "content": "创建 greet.py"},
  {"role": "assistant", "content": [
    {"type": "tool_use", "id": "toolu_1", "name": "write_file", "input": {"path": "greet.py"}}
  ]},
  {"role": "user", "content": [
    {"type": "tool_result", "tool_use_id": "toolu_1", "content": "created"}
  ]}
]

3. 规范化流程：清理、校验、合并

pipeline

规范化函数通常分三步。第一步遍历消息，构造 clean list：保留 role 和 content，把 content block 中 API 不接受的字段剥掉。第二步收集已有 tool_result 的 `tool_use_id`，用于判断 assistant 消息中的 tool_use 是否都有结果。第三步合并相邻同 role 消息，让 user/assistant 更接近 API 期望的交替结构。

内部消息

说明会显示在这里。

1. 过滤内部字段
2. 收集 tool_result id
3. 合并相邻同 role
规范化不是越激进越好。它应该移除协议不接受的部分，同时保留模型继续推理所需的有效内容。
伪代码：normalize_messages 的基本意图Python
def normalize_messages(messages):
    clean = []
    for message in messages:
        clean.append(strip_internal_metadata(message))

    existing_results = collect_tool_result_ids(clean)
    clean = drop_or_fix_unmatched_tool_use(clean, existing_results)

    return merge_adjacent_same_role(clean)

4. 调试暴露的 bug：把有效 content 清空

bug

视频调试里，第一次发送用户 prompt 没变化。第二轮 messages 有三条：用户 prompt、assistant 返回的 thinking/tool_use block、user 工具结果。规范化时，代码试图把 content block 当 dict 处理，并过滤 `_` 开头字段。但实际 SDK 返回的 thinking block、tool_use block 可能不是普通 dict，而是对象或特殊 block 类型。

结果是：assistant 的 content 被处理成空 list。随后系统把空 assistant content 发送给 API，触发 400 Bad Request。这个 bug 说明：规范化层如果写得不够鲁棒，反而会破坏原本有效的 tool_use 链路。

错误示例：只处理 dict，导致 block 被清空Bug
for block in message["content"]:
    if isinstance(block, dict):
        block_item = {k: v for k, v in block.items() if not k.startswith("_")}
        content.append(block_item)

# 如果 block 是 SDK block 对象而不是 dict，这里就什么也不保留。
# assistant content 可能变成 []，下一轮请求直接 400。
更稳思路：先把 SDK block 转成协议 dict，再过滤字段Python
def normalize_block(block):
    if hasattr(block, "model_dump"):
        block = block.model_dump()
    elif hasattr(block, "to_dict"):
        block = block.to_dict()
    elif not isinstance(block, dict):
        block = {"type": "text", "text": str(block)}

    return {k: v for k, v in block.items() if not k.startswith("_")}

你现在应该能做什么：看到 400 错误时，不只怪模型或 API，而会检查 messages 在规范化前后是否丢失了关键内容。

5. 常见误区：规范化不是简单删除

pitfalls

消息规范化的目标是“转换成合法协议”，不是“看不懂就删掉”。尤其是 tool_use、thinking、text、tool_result 等 block，可能来自不同 SDK，有对象形式、dict 形式或 Pydantic model 形式。正确做法是先统一表示，再过滤字段。

误删内容

把非 dict block 全部丢弃，导致 assistant content 为空。

漏删元数据

把 `_iteration` 等内部字段发给 API，可能触发 400。

忽略配对

tool_use 没有 tool_result，下一轮模型上下文断裂。

更稳的规范化还要保留足够日志：规范化前消息数量、规范化后消息数量、丢弃了哪些字段、是否出现空 content、是否存在未配对 tool_use。这样一旦出错，能快速定位是 API 协议问题还是规范化逻辑问题。
