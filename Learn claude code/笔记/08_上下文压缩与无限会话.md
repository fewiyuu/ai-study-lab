> [!question] 常见问题速查
> 1. 为什么 Agent 需要上下文压缩？
> 2. micro compact 压缩的是什么？
> 3. 为什么要保留最近几条 tool_result？
> 4. token 估算如何触发 auto compact？
> 5. auto compact 为什么要先保存 transcript？
> 6. 压缩可能丢失哪些关键信息？

> [!abstract] 核心主线
> Agent 的 messages 会随着用户消息、模型响应、tool use 和 tool_result 不断增长，而模型上下文窗口有上限。本节压缩机制分三层：第一层 micro compact 每轮先跑，把旧的长 tool_result 替换成 `previously used tool_name`，只保留最近几条真实结果；第二层是客户端估算 token，如果超过阈值就触发 auto compact；第三层是模型调用 `compact` 工具手动触发同样的重压缩。auto compact 会先把完整 messages 保存到 `.transcripts`，再让模型生成 continuity summary，最后用摘要替换原来的长历史。

## 概念精讲

### 上下文窗口是有边界的

Agent loop 会持续把信息写进 messages：

- 用户请求。
- assistant 的 tool use。
- user 侧的 tool_result。
- assistant 的下一轮响应。

读取文件、列目录、运行命令的结果可能很长。旧工具结果如果一直保留原文，会占用大量上下文，让后续模型请求越来越重。

判断句：压缩不是为了让模型忘掉一切，而是为了保留继续任务所需的信息。

## 三层压缩机制

### 1. micro compact：替换旧 tool_result

micro compact 每轮 Agent loop 开始时先执行。它会遍历 messages，找到 content 是 list 的 user 消息，从里面收集 `tool_result`。

收集时要保存三类信息：

| 信息 | 作用 |
|---|---|
| message index | 知道结果在哪条 message 里 |
| part index | 知道结果在 content list 的哪个位置 |
| part 本身 | 检查 content、tool_use_id 等字段 |

示例逻辑：

```python
to_clear = tool_results[:-keep_recent]

for message_index, part_index, part in to_clear:
    content = part.get("content", "")
    tool_use_id = part.get("tool_use_id")
    tool_name = tool_name_map.get(tool_use_id, "tool")

    if isinstance(content, str) and len(content) > 100:
        messages[message_index]["content"][part_index]["content"] = (
            f"previously used {tool_name}"
        )
```

`keep_recent = 3` 表示最近三条工具结果不处理。这样模型仍能看到刚刚发生的真实输出。

### 2. token 阈值：判断是否重压缩

micro compact 后，程序会估算 messages 的 token 数。示例里使用粗估：

```python
def estimate_tokens(messages):
    return len(str(messages)) // 4
```

这不是精确 tokenizer，只是工程估算。若估算值超过阈值，例如 50,000，就触发 auto compact。

### 3. auto compact：摘要替换整段历史

auto compact 是重压缩。它会：

1. 创建 `.transcripts` 目录。
2. 把当前完整 messages 序列化保存到磁盘。
3. 构造摘要 prompt，请模型 summarize this conversation for continuity。
4. 从模型响应中取 summary。
5. 用压缩提示和 summary 替换原 messages。

示意：

```python
return [
    {"role": "user", "content": "Previous conversation was compacted."},
    {"role": "assistant", "content": summary},
]
```

manual compact 也是类似流程。模型调用 `compact` 工具后，本地把 `manual_compact` 标记设为 true，随后进入 auto compact。

## 关键区分

| 机制 | 处理对象 | 强度 | 风险 |
|---|---|---|---|
| micro compact | 旧 tool_result 的 content | 轻 | 丢失旧工具输出细节 |
| auto compact | 整个 messages 历史 | 重 | 摘要可能漏掉关键约束 |
| manual compact | 由模型主动触发的 auto compact | 重 | 同 auto compact |

## 风险与修法

> [!warning] 压缩是有损的
> micro compact 会丢掉旧工具输出原文。auto compact 会把长历史压成摘要，可能漏掉用户限制、失败原因、文件路径或当前待办。

视频里还有一个明显风险：auto compact 只取 messages 文本的前 8 万字符做摘要。如果关键要求出现在后面，就可能根本没有进入摘要输入。

典型事故：

```text
原历史后段：
用户：不要修改原始字幕文件，只能在 course-forge 目录新增资料。

压缩摘要：
用户正在制作 Learn Claude Code 学习资料。

后果：
模型可能误以为可以修改原始字幕。
```

降低风险的做法：

- 压缩前保存 transcript。
- 摘要 prompt 明确要求保留用户目标、禁止事项、当前进度、未完成任务、关键文件路径。
- 压缩后让模型复述当前任务状态和限制。
- 对高风险任务，压缩摘要后人工或程序检查关键约束是否仍在。

## 应用与迁移问答

> [!question] 为什么不直接删除所有旧 tool_result？
> 直接删除会让历史结构断裂。替换成 `previously used tool_name` 至少保留了“曾经用过哪个工具”的线索。

> [!question] 为什么 auto compact 要保存 transcript？
> 因为摘要可能漏信息。保存原始 transcript 能让系统或人回看压缩前完整历史。

> [!question] micro compact 和 auto compact 最大区别是什么？
> micro compact 保留 messages 结构，只改旧工具结果内容；auto compact 会用摘要重建 messages。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 以为 `len(str(messages)) // 4` 是精确 token 计算。
> - 忘记 micro compact 只处理旧结果，最近结果要保留。
> - 不理解为什么需要 tool_use_id 到 tool name 的映射。
> - 把 auto compact 当成无损压缩。
> - 压缩摘要没有保留用户限制。
> - 没有保存 transcript，压缩后无法追溯。
> - 手动 `compact` 和自动阈值触发混为两套完全不同流程。

## 费曼解释润色版

> [!quote]
> Agent 的 messages 会随着工具调用不断变长，而模型上下文窗口有上限，所以长会话需要压缩。第一层是 micro compact，它每轮先找出旧的 `tool_result`，只保留最近几条真实结果，把更旧且很长的内容替换成 `previously used tool_name`。这样 messages 的结构还在，但旧工具输出变短了。第二层是 token 阈值，程序用 `len(str(messages)) // 4` 粗略估算上下文大小，超过阈值就触发 auto compact。auto compact 会先把完整 messages 保存到 `.transcripts`，再请模型为连续性生成 summary。最后，原来的长历史会被压缩提示和摘要替代。模型也可以通过调用 `compact` 工具手动触发这套流程。压缩的代价是可能丢失关键限制，所以摘要必须尽量保留目标、禁止事项、当前进度和未完成任务。

## 易错卡片

> [!warning] 误区：压缩是无损的
> 正确说法：压缩会丢细节，必须管理摘要质量和 transcript 追溯。

> [!warning] 误区：micro compact 和 auto compact 都只是删文本
> 正确说法：micro compact 替换旧 tool_result；auto compact 重建 messages。

> [!warning] 误区：只要保存 summary 就够了
> 正确说法：压缩前还应保存完整 transcript，防止摘要漏信息。

> [!warning] 误区：token 估算越简单越可靠
> 正确说法：简单估算只能当工程近似，不能替代真实 tokenizer 判断。
