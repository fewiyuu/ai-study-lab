> [!abstract] 核心主线
> S03 的 todo manager 更适合短、线性、一次性的内存计划；S07 的任务系统把计划持久化到 `.tasks/task_N.json`。每个 task 文件保存 `id`、`subject`、`description`、`status`、`blocked_by`、`blocks`、`owner` 等字段。Agent 通过 `task_create` 创建任务，通过 `task_update` 更新状态和依赖，通过 `task_list` 渲染看板，通过 `task_get` 查看单个任务详情。依赖关系写进 `blocked_by` 后，系统既能表达顺序，也能看出可并行分支。

## 概念精讲

### 从内存 todo 到持久化任务

内存 todo 的特点是轻：适合短任务、线性任务、一次性跟踪。但程序退出后，状态就没了。

持久化任务系统把每个任务写成 JSON 文件。这样即使程序崩溃或重启，也可以从 `.tasks` 目录重新读取任务状态。

判断句：todo 是短期行动清单；持久化任务系统是可恢复、可依赖、可查看的计划状态。

### 每个任务是一个 JSON 文件

示例：

```json
{
  "id": 2,
  "subject": "transform phase",
  "description": "Transform parsed structures into intermediate representation.",
  "status": "pending",
  "blocked_by": [1],
  "blocks": [],
  "owner": null
}
```

关键字段：

| 字段 | 含义 |
|---|---|
| `id` | 任务编号，也对应文件名 |
| `subject` | 看板短标题 |
| `description` | 任务详细说明 |
| `status` | 当前状态 |
| `blocked_by` | 当前任务依赖哪些前置任务 |
| `blocks` | 当前任务阻塞哪些后续任务 |
| `owner` | 任务负责人或归属 |

TaskManager 初始化时会扫描 `.tasks` 目录，计算已有最大 id，让新任务从 `max_id + 1` 开始，避免重启后 id 冲突。

## 四个任务工具

### `task_create`

创建任务，需要 `subject` 和 `description`。本地函数会分配 id、设置默认状态、保存成 `task_N.json`。

```python
def create_task(subject, description):
    task = {
        "id": self.next_id,
        "subject": subject,
        "description": description,
        "status": "pending",
        "blocked_by": [],
        "blocks": [],
        "owner": None,
    }
    self.save(task)
    self.next_id += 1
    return task
```

### `task_update`

读取某个任务文件，更新状态或依赖，再保存回去。更新 `blocked_by` 时通常要做去重，避免同一个依赖重复出现。

```python
task["blocked_by"] = list(
    set(task["blocked_by"] + add_blocked_by)
)
```

### `task_list`

读取所有 `task_*.json`，按 id 排序并渲染成看板。

### `task_get`

读取单个任务完整 JSON。适合查看 description、owner、依赖等细节。

## 依赖与并行

示例任务链：

```text
1 parse
2 transform  blocked_by [1]
3 emit       blocked_by [1]
4 test       blocked_by [2, 3]
```

解释：

- task2 必须等 task1 完成。
- task3 也必须等 task1 完成。
- task2 和 task3 彼此不阻塞，所以 parse 完成后可以并行。
- task4 必须等 task2 和 task3 都完成。

> [!warning] 易错点
> `blocked_by: [1]` 的方向是“我被 1 阻塞”，不是“我阻塞 1”。方向写反会导致执行顺序错误。

## 看板渲染

`task_list` 会把 JSON 任务渲染成可读文本：

```text
[ ] #1 parse phase
[ ] #2 transform phase blocked by 1
[ ] #3 emit phase blocked by 1
[ ] #4 test phase blocked by 2,3
```

常见 marker：

| marker | 含义 |
|---|---|
| `[ ]` | pending |
| `[>]` | in_progress |
| `[x]` | completed |

看板只显示 id、subject 和阻塞关系，方便快速扫视；详情交给 `task_get`。

## 工程边界

文件持久化更强，但比内存 todo 更复杂：

- 可能有并发写入覆盖。
- 需要考虑原子写回。
- 多个 Agent 同时操作 `.tasks` 时可能出现状态冲突。
- 依赖字段如果只维护一侧，可能不一致。
- 大规模任务可能更适合数据库，而不是一堆 JSON 文件。

## 应用与迁移问答

> [!question] 什么时候仍然用内存 todo？
> 短、线性、一次性的任务，比如“先读文件，再改一处，再跑测试”。这类任务没必要引入文件持久化。

> [!question] 什么时候用持久化任务系统？
> 任务较长、可能中断、需要恢复、需要表达依赖或并行时，持久化任务系统更合适。

> [!question] 为什么不只在一个 JSON 文件里保存所有任务？
> 一个文件更简单，但单个任务独立文件更方便查看、更新和拆分。代价是要考虑多文件一致性。

## 费曼解释润色版

> [!quote]
> 这一节的任务系统是把内存 todo 升级成可持久化的计划看板。TaskManager 启动时会创建 `.tasks` 目录，并扫描已有 `task_N.json` 计算下一个 id。模型通过 `task_create` 创建任务，本地把每个任务写成一个 JSON 文件，保存 subject、description、status、blocked_by 等字段。之后模型可以用 `task_update` 更新任务状态或依赖关系。比如 transform 和 emit 都 `blocked_by [1]`，表示它们都要等 parse 完成，但它们彼此不依赖，所以可以并行。test `blocked_by [2,3]`，表示它要等 transform 和 emit 都完成。`task_list` 会读取所有 JSON 文件，把任务渲染成带 marker 和 blocked-by 的看板。这样任务不仅能被模型使用，也能被人直接从文件和看板中查看。代价是文件持久化要考虑并发写入、原子保存和依赖一致性。

## 易错卡片

> [!warning] 误区：任务持久化只是把 todo 换个地方存
> 正确说法：它还引入了依赖、恢复、查看和更复杂的写入一致性问题。

> [!warning] 误区：`blocked_by [1]` 表示当前任务阻塞 1
> 正确说法：表示当前任务被 1 阻塞。

> [!warning] 误区：task2 和 task3 都依赖 task1，所以 task2 与 task3 互相依赖
> 正确说法：它们共享前置任务，但彼此不阻塞，可以并行。

> [!warning] 误区：文件持久化一定比内存 todo 更好
> 正确说法：它更适合长任务和依赖任务，但复杂度也更高。
