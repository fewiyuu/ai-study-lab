> [!question] 常见问题速查
> 1. 为什么发送给模型前要做消息规范化？
> 2. API 协议对 messages 有哪些常见要求？
> 3. user / assistant 严格交替是什么意思？
> 4. 为什么 `tool_use` 必须有对应 `tool_result`？
> 5. 内部元数据为什么可能导致 400？
> 6. 调试中 assistant content 为什么被清空？
> 7. 如何让 block 规范化兼容不同 SDK？

> [!abstract] 核心主线
> 消息规范化是在把内部 `messages` 发给模型 API 前做的协议预处理。随着系统加入工具超时、用户取消、上下文压缩、内部元数据和不同 SDK block，对话历史可能不再满足 API 要求。规范化的目标是：过滤内部字段、保证 user/assistant 更接近交替结构、检查 `tool_use` 和 `tool_result` 是否配对、合并相邻同 role 消息，并避免发送 API 不接受的字段。视频调试里也暴露了一个 bug：代码只按 dict 处理 content block，遇到 SDK 返回的特殊 block 对象时把 assistant content 清成空 list，最终触发 400 Bad Request。这说明规范化层必须先把 block 转成协议 dict，再过滤字段，不能简单删除看不懂的内容。

## 概念精讲

### 消息规范化是什么

消息规范化发生在发送给模型 API 之前：

```python
normalized_messages = normalize_messages(messages)
client.messages.create(messages=normalized_messages)
```

它不是模型推理，也不是工具执行，而是 Harness 层的请求预处理。

目标是把内部消息列表转换成 API 能接受的格式。

### 为什么需要规范化

早期 Agent loop 简单时，可以直接把 `messages` 发给模型。

系统复杂后，messages 可能混入：

- 工具超时信息。
- 用户取消信息。
- 上下文压缩替换信息。
- `_iteration` 等内部调试元数据。
- SDK 返回的特殊 content block 对象。
- 未闭环的 `tool_use`。

这些内容对内部系统有用，但 API 可能不接受。不处理就容易得到 400 Bad Request。

## API 协议的三类约束

| 约束 | 含义 | 规范化动作 |
|---|---|---|
| role 交替 | user / assistant 消息应保持稳定对话顺序 | 合并相邻同 role 消息 |
| tool 闭环 | assistant 的 `tool_use` 要有 user 的 `tool_result` | 收集 result id，检查缺失结果 |
| 字段白名单 | API 只接受协议定义字段 | 删除 `_` 开头或协议外字段 |

> [!warning] API 协议由后端控制
> 如果后端要求某种格式，Harness 最好在发送前适配，而不是等后端报错后再浪费一次交互。

## 规范化的基本流程

### 1. 清理消息

遍历 messages，构造 clean list：

- 保留 `role`。
- 保留合法 `content`。
- 过滤 `_` 开头的内部字段。
- 保留 text、tool_use、tool_result 等有效 block。

### 2. 收集已有工具结果

遍历 clean messages，收集：

```python
existing_results = {"toolu_1", "toolu_2"}
```

用 set 的好处是去重和快速判断某个 tool_use 是否已有结果。

### 3. 合并相邻同 role

如果出现连续 user 或连续 assistant，可以把 content 合并，减少违反 role 交替协议的风险。

```text
user + user -> user
assistant + assistant -> assistant
```

## 调试中发现的 bug

第二轮 messages 有三条：

1. user prompt。
2. assistant 返回 thinking block + tool_use block。
3. user 返回 tool_result。

规范化时，代码假设 content block 是 dict：

```python
if isinstance(block, dict):
    block_item = {k: v for k, v in block.items() if not k.startswith("_")}
    content.append(block_item)
```

但某些 SDK 返回的是 `ThinkingBlock`、`ToolUseBlock` 这类对象，不是普通 dict。

结果：

- block 没被保留。
- assistant content 变成空 list。
- 发送给 API 后触发 400 Bad Request。

> [!failure] 这里的关键教训
> 规范化不是“看不懂就删”。应该先把 SDK block 转成协议 dict，再过滤字段。

## 更稳的修复思路

```python
def normalize_block(block):
    if hasattr(block, "model_dump"):
        block = block.model_dump()
    elif hasattr(block, "to_dict"):
        block = block.to_dict()
    elif isinstance(block, dict):
        block = block
    else:
        block = {"type": "text", "text": str(block)}

    return {
        k: v
        for k, v in block.items()
        if not k.startswith("_")
    }
```

核心原则：

- 先统一表示。
- 再过滤字段。
- 不要丢掉有效 tool_use。
- 不要产生空 assistant content。

## 400 Bad Request 排查清单

> [!todo] 先比较规范化前后 messages
> - [ ] 消息数量是否异常变化。
> - [ ] role 顺序是否合理。
> - [ ] assistant content 是否被清空。
> - [ ] `tool_use` 是否还在。
> - [ ] `tool_result` 是否有对应 `tool_use_id`。
> - [ ] 是否存在 `_` 开头内部字段。
> - [ ] 是否存在 API 不接受的 SDK 对象。
> - [ ] 是否存在空 content。
> - [ ] 是否存在连续同 role 且未合并。

## 应用与迁移问答

> [!question] 规范化是否一定能避免所有 400？
> 不能。它只能提前处理已知协议问题。未知协议变化、SDK 差异和 bug 仍可能导致 400。

> [!question] 为什么不要把所有非 dict block 删除？
> 因为不同 SDK 可能用对象表示 text、thinking、tool_use。删除它们会丢掉模型下一轮推理所需内容。

> [!question] 换模型 SDK 时最容易坏在哪里？
> content block 的结构最容易不兼容：同样是 tool_use，有的 SDK 给 dict，有的给对象，有的字段名不同。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把消息规范化理解成压缩上下文。
> - 忘记规范化发生在发 API 前。
> - 只删除内部字段，不检查 tool_use/tool_result 配对。
> - 把 SDK block 对象当 dict 处理。
> - assistant content 被清空后还继续发送。
> - 遇到 400 只怪模型，不比较规范化前后的 messages。

## 费曼解释润色版

> [!quote]
> 消息规范化是 Agent Harness 在发送请求给模型 API 前做的一次协议预处理。内部 messages 里可能有工具超时、用户取消、压缩替换、调试元数据和 SDK 特殊对象，但 API 只接受它规定的字段和结构。规范化要做几件事：过滤 `_` 开头的内部字段，尽量保持 user 和 assistant 交替，检查 assistant 的 `tool_use` 是否有对应 user 的 `tool_result`，并合并相邻同 role 消息。这样可以减少 400 Bad Request。视频调试里出现了一个反例：代码只按 dict 处理 content block，遇到 SDK 的 ThinkingBlock 和 ToolUseBlock 对象时没有保留内容，导致 assistant content 变成空 list，最终请求 400。这个 bug 说明规范化不是简单删除，而是先把不同 SDK 的 block 转成统一协议 dict，再过滤不该发的字段。遇到 400 时，第一步应该比较规范化前后的 messages，看关键 tool_use、tool_result 和 content 是否被破坏。

## 易错卡片

> [!warning] 误区：消息规范化就是上下文压缩
> 正确说法：规范化是协议适配；压缩是上下文长度治理。

> [!warning] 误区：内部字段无所谓，API 会忽略
> 正确说法：API 可能直接返回 400。

> [!warning] 误区：非 dict block 都可以删
> 正确说法：很多有效 block 可能是 SDK 对象，必须先转换。

> [!warning] 误区：规范化后 content 空了也可以发
> 正确说法：空 assistant content 很可能违反协议或丢失工具调用链路。
