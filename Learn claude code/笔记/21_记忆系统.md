> [!question] 常见问题速查
> 1. Memory 和 CLAUDE.md / Skill 有什么区别？
> 2. `MemoryManager` 管理哪些文件？
> 3. `save_memory` 会写入什么内容？
> 4. `.memory/memory.md` 索引有什么用？
> 5. Memory 如何进入 system prompt？
> 6. 什么信息值得保存进 Memory？
> 7. 为什么不能把所有聊天内容都保存成 Memory？

> [!abstract] 核心主线
> 记忆系统让 Agent 跨会话保留用户偏好和重要事实。它不是启动时固定加载的 CLAUDE.md 或 Skill，而是在 Agent loop 中动态读写。`MemoryManager` 管理 `.memory` 目录和内存中的 `memories` 字典。模型在用户表达稳定偏好时调用 `save_memory(name, description, type, content)`，系统把记忆写成带 frontmatter 的 Markdown 文件，并重建 `.memory/memory.md` 索引。下一轮 `build_system_prompt` 会加载这些 memory，按类型分组注入 system prompt。值得进入 Memory 的信息应满足两个条件：跨会话仍有价值，并且无法从当前工作区轻易推导。

## 概念精讲

### Memory 的位置

`CLAUDE.md` 和 Skill 通常在启动时加载，像固定说明和技能包。

Memory 不同：

- 可能一开始没有。
- Agent loop 过程中可以创建。
- 可以跨会话保留。
- 下一轮构建 system prompt 时注入。

> [!abstract] 直觉
> Memory 是“长期协作偏好和稳定事实”，不是当前聊天记录。

## 保存记忆

当用户说：

```text
I like to define a main method in Python files.
```

模型可以调用 `save_memory`：

```json
{
  "name": "python_main_convention",
  "description": "User prefers main function in Python files",
  "type": "user",
  "content": "When creating Python files, always define a main function."
}
```

系统会：

1. 校验 type。
2. 规范化 name。
3. 创建 `.memory` 目录。
4. 写入 `.memory/python_main_convention.md`。
5. 更新内存中的 `memories` 字典。
6. 重建 `.memory/memory.md` 索引。

## 单个 memory 文件

```markdown
---
name: python_main_convention
description: User prefers main function in Python files
type: user
---

When creating Python files, always define a main function.
```

frontmatter 记录元数据，正文记录具体记忆内容。

## memory index

`.memory/memory.md` 像目录：

```markdown
# Memory Index

- python_main_convention [user]: User prefers main function in Python files
```

作用：

- 快速知道当前有哪些记忆。
- 按名称、类型、描述检索。
- 避免每次无脑读所有长内容。

## 加载到 system prompt

每轮 Agent loop 开始时，`build_system_prompt` 会调用 memory prompt 构建逻辑。

如果没有 memory，只加入 memory guidance，告诉模型什么时候该保存记忆。

如果有 memory，会按 type 分组：

```text
## Memory

### user
- python_main_convention: User prefers main function in Python files
When creating Python files, always define a main function.
```

然后这段进入 system prompt，影响下一轮模型选择工具和生成内容。

## 什么值得进入 Memory

两个标准：

1. 跨会话仍然有价值。
2. 无法从当前工作区轻易推导。

| 信息 | 是否适合 | 原因 |
|---|---|---|
| 用户喜欢 Python 文件定义 main 函数 | 适合 | 稳定偏好，仓库推导不出 |
| 当前目录有 hello.py | 不适合 | 可实时读取，容易变化 |
| 用户希望改文件前先给计划 | 适合 | 稳定协作偏好 |
| 一次性工具错误 | 通常不适合 | 短期上下文即可 |

> [!warning] Memory 会占上下文
> Memory 最终会进入 system prompt，所以越多越贵，也越容易干扰模型。

## Memory 的风险

如果什么都存：

- Memory 变成垃圾堆。
- 上下文窗口被占用。
- 过期信息继续影响模型。
- 模型依赖错误记忆，产生幻觉式响应。

所以 Memory 需要整理、压缩、删除和更新。

## 应用与迁移问答

> [!question] Memory 和上下文压缩有什么区别？
> 上下文压缩处理当前长会话；Memory 处理跨会话的稳定偏好和事实。

> [!question] Memory 和 Skill 有什么区别？
> Skill 是一类任务的可复用流程和模板；Memory 是用户或项目的长期偏好与事实。

> [!question] Memory 是否应该自动保存所有内容？
> 不应该。应该只保存稳定、有价值、当前上下文无法推导的信息。

## 练习 / 盲区复盘

> [!failure] 容易卡住的地方
> - 把 Memory 当聊天记录仓库。
> - 把可实时读取的仓库状态存入 Memory。
> - 忘记 Memory 会进入 system prompt，占用上下文。
> - 不更新过期 Memory。
> - 重复保存同一偏好。
> - 不区分用户偏好、反馈、项目事实和参考资料。

## 费曼解释润色版

> [!quote]
> 记忆系统让 Agent 不必每次都像第一次合作。它和 CLAUDE.md、Skill 不一样：CLAUDE.md 和 Skill 更像启动时加载的固定说明，Memory 是 Agent loop 中动态读写的长期信息。用户表达稳定偏好时，模型可以调用 `save_memory`，传入 name、description、type 和 content。系统会把它写到 `.memory/<name>.md`，文件里有 frontmatter 和正文，同时更新内存中的 memories 字典，并重建 `.memory/memory.md` 索引。下一轮 `build_system_prompt` 会把这些记忆按类型分组注入 system prompt，所以模型生成新 Python 文件时会记得用户偏好，比如总是定义 main 函数。值得保存的记忆必须跨会话有价值，而且不能从当前仓库轻易推导。不能把所有聊天都存进去，否则 Memory 会变成垃圾堆，占用上下文，还可能因为过期信息误导模型。

## 易错卡片

> [!warning] 误区：Memory 就是保存全部聊天历史
> 正确说法：Memory 保存跨会话稳定偏好和重要事实。

> [!warning] 误区：能从仓库读出来的信息也要存
> 正确说法：可实时读取的信息不应长期记忆化。

> [!warning] 误区：Memory 越多越好
> 正确说法：Memory 会占上下文，过多会污染判断。

> [!warning] 误区：Memory 写入后永远不需要维护
> 正确说法：Memory 需要更新、删除、压缩和去重。
