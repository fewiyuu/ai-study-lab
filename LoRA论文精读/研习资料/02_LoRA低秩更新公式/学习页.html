LoRA 低秩更新公式

把 Delta W 写成 BA，读懂形状、参数量、初始化和 merge

学完你应该能做什么

  1. 写出 LoRA 的核心前向公式，并解释每个矩阵的形状。
  2. 根据 d、k、r 计算一个 LoRA 模块的可训练参数量。
  3. 说明为什么 A 随机初始化、B 初始化为 0。
  4. 解释 alpha/r 的缩放作用，以及 merge 为什么能消除额外推理延迟。

1. 直觉：不直接改 W，而是学 W 的改变量

low-rank

LoRA 的核心不是“加一个小网络”，而是把原来要学习的权重改变量拆成两个小矩阵。预训练权重 W0 保持不动，下游任务只学习 Delta W。但 Delta W 仍然可能和 W0 一样大，于是 LoRA 进一步约束：让 Delta W = BA，其中中间维度 r 很小。

用形状看会更清楚。假设原线性层权重 W0 的形状是 d x k，输入 x 是 k 维，输出是 d 维。LoRA 让 A 先把输入从 k 维压到 r 维，再让 B 从 r 维还原到 d 维。这样 BA 的形状仍然是 d x k，可以和 W0 相加。

术语上，这叫低秩参数化更新。它不是说原模型权重 W0 必须低秩，而是说下游任务需要的改变量可能只占少数方向。LoRA 把可训练自由度限制在这些低维方向里。限制得太小可能表达不够，限制得合适就能省大量参数。

判断句：看到一个 LoRA 公式时，先检查 BA 是否和 W0 同形状。如果形状对不上，后面讲得再漂亮也不能加到原线性层上。你现在应该能用形状判断一个 LoRA 实现是否写反。

h = W0x + Delta W x = W0x + BAx
W0 in Rd x k,   B in Rd x r,   A in Rr x k,   r << min(d,k)

2. 参数量：为什么两个小矩阵会省这么多

parameters

一个普通 d x k 权重矩阵有 d*k 个参数。如果全量微调这个矩阵，任务更新也可以覆盖这 d*k 个自由度。LoRA 只训练 A 和 B，参数量变成 r*k + d*r = r(d+k)。如果是 Transformer 里常见的方阵 d_model x d_model，每个目标矩阵就是 2*d_model*r。

这不是魔法，而是明确牺牲了一部分表达自由度。LoRA 只能表示 rank 不超过 r 的更新矩阵。论文的经验发现是：语言模型下游适配常常不需要满秩更新，小 r 就能捕捉主要任务方向。你要记住的是“低秩假设”，不是“免费变强”。

当 LoRA 应用于多层 Transformer 时，还要乘以层数和每层选中的目标矩阵数。论文里常用 W_q 和 W_v，也就是每层两个目标矩阵。GPT-3 175B 有 96 层，d_model=12288，如果 r=8 且每层两个矩阵，参数量大约 3770 万个，FP16 约 72MB。论文也展示了更小预算，比如 4.7M 参数配置。

|Theta| = r*k + d*r = r(d+k)
若 W 是 dmodel x dmodel：每个目标矩阵约 2 x dmodel x r
LoRA 参数量计算器2 x modules x layers x d x r
估算结果
判断提示：对一个方阵投影，LoRA 每个目标矩阵训练 A 和 B 两个矩阵，所以参数量约为 2 x d_model x r。再乘以层数和目标矩阵数。

判断句：当你看到“LoRA 参数量很小”，应该马上问：rank 是多少，作用了多少层，作用了几个目标矩阵，隐藏维度是多少。你现在应该能手算一个 LoRA 配置的大致参数量。

3. 初始化：为什么 B 从 0 开始

initialization

论文使用的初始化很朴素：A 随机高斯初始化，B 初始化为 0。这样训练刚开始时 BA=0，LoRA 分支不会改变预训练模型的输出。模型一开始等同于原模型，然后训练逐步学出任务更新。

如果 A 和 B 都随机，刚开始就会给原模型加上一段随机扰动。预训练模型本来已经有可用行为，随机扰动可能让训练初期不稳定。反过来，如果两者都初始化为 0，梯度会出现对称性问题：其中一个矩阵可能拿不到有效方向，训练启动会受影响。随机 A、零 B 是一种常用折中。

这里的判断不要停在“论文这么做”。你要能解释删改后果：B 不为 0 会扰动底座输出；A 和 B 都为 0 会让低秩分支起步困难；忘记冻结 W0 则会变成“LoRA + 部分全量微调”，参数和显存收益都会跑偏。

正确示例 正确示例：LoRA 初始化思路
# A: r x in_features，随机初始化
A = normal(mean=0, std=small_value, shape=(r, in_features))

# B: out_features x r，初始化为 0
B = zeros(shape=(out_features, r))

# 初始时 Delta W = B @ A = 0
delta = B @ A
错误示例 错误示例：两个矩阵都随机扰动底座
A = normal(shape=(r, in_features))
B = normal(shape=(out_features, r))

# 问题：刚开始 Delta W != 0
# 预训练模型输出被随机分支改写，训练初期可能更不稳。

判断句：初始化是否合理，要看训练第 0 步 LoRA 分支有没有改变原模型输出。你现在应该能诊断 LoRA 初始化错误。

4. alpha/r：控制 LoRA 分支的尺度

scaling

LoRA 前向通常不是直接加 BAx，而是加上 (alpha/r) * BAx。这里 alpha 是缩放常数，r 是 rank。缩放的目的，是让改变 rank 时，LoRA 分支输出的量级不至于失控，也减少每次改 rank 后大幅重调学习率的负担。

直觉上，rank 变大代表分支里有更多方向。如果不做缩放，不同 rank 的输出尺度可能差很多。论文提到，在 Adam 下调 alpha 和调学习率有相近味道，但它不是学习率本身。学习率控制优化步长，alpha/r 控制 LoRA 分支进入主干输出的幅度。

实际项目里常见错误是只记公式但忘记配置语义。alpha 太小，LoRA 分支影响弱；太大，训练可能不稳或覆盖预训练行为。不同库还会把 lora_alpha、scaling、rank 写成不同字段，验收时要确认最终乘到输出上的系数。

h = W0x + alphar BAx
缩放直觉alpha / r
r=4, alpha=8
scale=2
r=8, alpha=8
scale=1
r=16, alpha=8
scale=0.5
判断提示：rank 改了，LoRA 分支的表达方向数变了；缩放项用于控制它进入主分支的量级。

你现在应该能区分：rank 控制低秩空间大小，alpha 控制分支强度，学习率控制参数更新步长。

5. merge：为什么部署时可以零额外延迟

merge

LoRA 在训练时有两条路径：原始线性层 W0x 和低秩分支 BAx。但 BA 的结果本身就是一个和 W0 同形状的矩阵。部署时可以先算出 W = W0 + BA，以后推理仍然只跑一个普通线性层。这就是论文说“不引入额外推理延迟”的关键。

merge 不是把 LoRA 扔掉，而是把任务更新吸收到原权重里。切换任务时，可以从 W0 出发，加载另一个任务的 B'A'。工程上要小心重复 merge：如果同一个 LoRA 权重被加了两次，输出就错了。也要记住，merge 后如果还想继续训练 LoRA，需要知道当前权重是否已经包含 LoRA 更新。

这里还有一个边界：如果一个 batch 里混合不同任务，并且每个样本要用不同 LoRA，提前 merge 成单一权重就不方便。可以选择不 merge，动态按样本选择 LoRA 分支，但这样会牺牲一部分延迟优势。论文也提到这个限制。

逐行拆解 逐行拆解：merge 和 unmerge 的状态
# 训练时
y = x @ W0.T + scaling * (x @ A.T @ B.T)

# 部署前 merge
W_merged = W0 + scaling * (B @ A)
y = x @ W_merged.T

# 切换任务时，从干净 W0 加另一个 LoRA
W_task_b = W0 + scaling_b * (B_b @ A_b)

merge 前

便于训练 LoRA、切换不同低秩分支；推理要额外算低秩路径。

merge 后

推理就是普通线性层；适合单任务部署或服务当前任务。

重复 merge

同一份 Delta W 被加两次，模型输出偏移。要记录 merge 状态。

判断句：“LoRA 零额外推理延迟”默认指已经 merge 到权重后的推理路径。你现在应该能说明这句话的前提。

6. 常见错法：公式看似会了，代码却写坏了

diagnosis

A/B 形状写反

如果 B@A 不能得到 d x k，就无法和 W0 相加。先用形状验收。

忘记冻结 W0

底座参数继续训练，显存、优化器状态和任务 checkpoint 都会变重。

B 不为 0

初始输出被随机 LoRA 分支扰动，训练起点不再是预训练模型。

alpha 当学习率

alpha 控制 LoRA 输出尺度，不等同于优化器步长。两者都会影响训练，但位置不同。

重复 merge

多次把同一 LoRA 加到 W0，会让更新翻倍。需要 merge 状态标记。

rank 盲目越大越好

rank 越大参数越多，不保证指标单调提升。论文实验里小 r 已经很强。
