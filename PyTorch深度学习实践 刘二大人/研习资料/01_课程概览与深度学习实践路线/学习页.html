训练闭环

先抓主线

深度学习里最稳定的骨架只有一条：数据进入模型，模型给出预测，预测和标签算出 loss，loss 通过反向传播写入梯度，优化器读取梯度并更新参数，评估阶段再检查模型有没有真的学到东西。你后面看任何一段训练代码，都可以先按这条链路找对象，不要先沉进一堆 API 名字里。

$$ \theta \leftarrow \theta - \eta \nabla_{\theta}\mathcal{L} $$

\theta 是参数，\eta 是学习率，\nabla_{\theta}\mathcal{L} 是 loss 对参数的梯度。这个式子只做一件事：把“往哪改、改多大”讲清楚。

步骤 读取什么 写入或改变什么 错一步会看到什么 下一步先查什么
数据 batch 的输入 x 和标签 y 把样本送进训练循环 维度不对、标签类型不对、batch 为空 DataLoader、batch size、dtype
前向 输入 x 和当前参数 生成预测 pred / y_hat 输出形状错、模型没接上数据 forward 里读了什么、写了什么
loss 预测和标签 压成一个标量目标 loss 保持向量形状，后面无法 backward criterion 的输入约定
backward loss 把梯度写到参数的 grad 上 grad 为空、图断了、报错说不能求导 参数有没有参与计算图
step 参数上的 grad 更新参数值 参数不动、loss 不降、方向反了 学习率、zero_grad、step 顺序
eval 验证集、模型状态 关闭训练态行为，停止写入梯度 验证集结果乱跳、显存涨、dropout 干扰 model.eval() 和 no_grad()

从最小例子开始

线性入口

线性模型是这门课第一块可计算的骨架。它把训练链路里的每个动作都露出来了：输入乘权重、再加偏置、再和标签比较。路径越短，越容易看见哪里出错。只要你能把这个例子看懂，后面换成逻辑斯蒂回归、多维特征、卷积网络，骨架都还是同一条。

$$ \hat{y} = wx + b,\qquad \mathcal{L} = \frac{1}{n}\sum_{i=1}^{n}(\hat{y}_i-y_i)^2 $$

\hat{y} 是预测，w 和 b 是参数，\mathcal{L} 是均方误差。把这三个符号放在一起看，就能知道模型到底在优化什么。

x 真实 y 参数 预测 \hat{y} 残差 看出来的事
1 2 w=1, b=0 1 -1 模型还没对上斜率
3 6 w=2, b=0 6 0 这组参数更接近目标关系

这里还有一个常见分界：如果没有偏置 b，模型会被固定在过原点的直线上。对 y=2x+1 这样的关系来说，单靠 w 就不够用。这个例子说明，参数少一项，模型能力就少一层。

把对象和状态追一遍

读写追踪

学习这页时，最值钱的动作是追对象。x 和 y 从哪里来，pred 和 loss 是谁算出来的，grad 写到哪里，谁会真的改参数，这些问题只要有一个答不清，后面调参就很容易靠运气。下面这条小追踪把同一批对象穿起来，读的时候可以只盯一列。

数据批次 x, y

读自 DataLoader。这里只负责提供当前 batch，不负责计算，也不负责更新。

预测 pred / \hat{y}

由模型前向算出来。它读输入和参数，写出预测，不应该直接改参数。

标量 loss 一个数

只有把这一步压成标量，后面的 backward 才能顺利继续。

梯度 grad / \nabla_\theta \mathcal{L}

梯度被写到参数上。这里写错，后面 step 就只是在拿坏方向走。

更新后参数 w, b

优化器读取 grad，真正改参数值。没清梯度、顺序错、学习率不合适，常常都在这里暴露。

对象 从哪来 谁读它 谁会写它 错了会怎样
x batch 输入 model 通常不改 前向读不到正确形状
y batch 标签 criterion 通常不改 loss 目标错，训练方向全偏
pred model(x) criterion 每个 batch 重新算 输出形状不对，loss 算不下去
grad loss.backward() optimizer 反向写入 参数更新方向失真
参数 初始化或上一步更新后 model / optimizer optimizer.step() 模型看起来在跑，实际不变

把一个 batch 走完整

具体案例

现在把抽象链路收进一个最小 batch。假设输入是 x=[1,2,3]，标签是 y=[2,4,6]，先从一个最朴素的线性模型开始，取 w=1、b=0。第一次前向时，预测会是 [1,2,3]，这组值离标签还有明显距离，loss 也就不会小。这里最重要的是看清楚：同一批数据在 forward、loss、backward、step 之间是怎么穿过一遍的。

如果你把参数换成 w=2、b=0，预测就会直接贴到标签上。这个变化说明两件事。第一，模型参数在决定输出。第二，loss 的高低和参数位置是连着的，所以训练在找一组更好的参数。课程后面讲梯度下降时，这个例子会继续用，只是把“怎么找更好的参数”换成了“沿着损失下降”。

这个例子留下的判断法

以后看到一段训练代码，先问四个问题：输入对不对、预测算没算、loss 有没有压住、参数有没有真的变。能答上这四个问题，后面的排错会快很多。

手算一轮 batchpython
x = [1, 2, 3]
y = [2, 4, 6]
w, b = 1, 0
pred = [w * xi + b for xi in x]   # [1, 2, 3]
loss = sum((pi - yi) ** 2 for pi, yi in zip(pred, y)) / len(x)

这段伪代码适合拿来检查两个常见误区。一个是把 batch 当成单个样本，结果看不出平均误差；另一个是只盯某一行公式，忘了 loss 其实是从预测和标签的整批比较来的。你读训练脚本时，只要能指出数据在这一步读了什么、写了什么，就离理解代码近了一大步。

阶段 输入 输出 读写关系 如果错了
forward x 和参数 pred 读输入，写预测 模型和数据接不上
loss pred 和 y 标量 loss 读预测和标签，写目标值 后面不能正常 backward
backward loss grad 把梯度写到参数上 优化器拿不到方向
step grad 更新后的参数 读梯度，改参数 看起来训练了，参数没真变

再补一条常见失配：batch 形状和 dtype 不对

很多训练脚本看起来已经跑通，真正出问题的地方却在输入批次。比如分类任务里，模型输出的是 [32, 10] 的 logits，标签却还停在 [32, 1] 的浮点数。代码不一定立刻报错，但 loss 会走偏，梯度也会跟着歪。你看到的“训练能跑”，其实只是数据格式勉强接上了。

先看 shape 再进 losspython
for x, y in loader:
    print("x:", x.shape, x.dtype)
    print("y:", y.shape, y.dtype)

    logits = model(x)
    target = y.squeeze(-1).long()
    loss = criterion(logits, target)
检查点 看到什么 要做什么
x.shape batch 维度有了，但特征维度被压扁 先确认 DataLoader 是否把样本展开成了预期形状
y.dtype 标签还是 float 分类任务通常要转成 long 或正确的 one-hot 形式
criterion 输入和标签维度对不上 对照 loss 的约定，先改标签再改模型
loss 数值忽大忽小，像在乱跳 先看是否 broadcast 了，或把类别和回归混用了

这个问题的后果不止是报错。更麻烦的是，它会让训练曲线看起来像在下降，但下降方向并不对应真实标签。等你把模型换大、学习率调小、优化器换一圈，问题还在，因为真正错的是输入约定。先把 shape 和 dtype 打印出来，很多假象会马上露出原形。

练习钩子

拿一段现成训练代码，先跑一个 batch，记录 x.shape、y.shape、y.dtype 和 loss 的输入约定。然后把标签改成正确类型，再对比一次结果。

把一轮训练跑通

检查流程

把这页再往前推一步，就能得到一轮真正可检查的训练流程。先把一个 batch 送进模型，再把预测送进 loss，接着清梯度、反向、更新，最后切到评估态看一眼验证结果。只要这条链路能自己讲清楚，后面的章节就不会只是名词表。

拿一个最小例子来说：x=[1,2,3]，y=[2,4,6]，模型先取 w=1、b=0。第一次 forward 得到的预测是 [1,2,3]，这时 loss 还偏大。把参数换成 w=2、b=0，预测会直接贴上标签。这个变化不需要大模型才能看见，线性模型就足够把“参数会改输出”这件事露出来。

一轮训练和一次评估python
for epoch in range(2):
    model.train()
    for x, y in loader:
        pred = model(x)
        loss = criterion(pred, y)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

    model.eval()
    with torch.no_grad():
        val_pred = model(val_x)
        val_loss = criterion(val_pred, val_y)

这段代码最值得盯的是状态切换。train() 让模块回到训练态，eval() 让模块改成评估态，torch.no_grad() 则把评估阶段的梯度建图关掉。只要这三步有一个漏掉，训练脚本就可能“看起来能跑”，但状态已经不对了。

阶段 当前对象 读什么 写什么 常见故障
训练前向 model x 和参数 pred 输入形状错、forward 没接通
训练 loss criterion pred 和 y 标量 loss loss 保持向量形状，后面无法 backward
训练反向 autograd loss 参数 grad 图断了、grad 为空、更新方向丢失
训练更新 optimizer grad w / b 忘了 zero_grad、学习率太大或太小
评估 model + no_grad 验证集 x, y val_pred / val_loss 验证结果抖动、显存浪费、模式没切对

这条流程能直接接到练习区。题目 5 到 8 会让你检查更新顺序和模式切换；题目 9 到 12 会让你根据症状回看是哪一步出了问题。你只要把这轮流程记住，后面学梯度下降时，重心就会变成“为什么朝这个方向走”，而不是“到底谁在改参数”。

另一个高频坑：梯度累积把更新方向带偏

假设你在做二分类，batch size 只有 8，前一轮训练结束后没有清掉梯度，下一轮又直接 backward。此时 optimizer.step() 读到的方向混了前两轮样本，单个 batch 看起来都对，累计起来的参数却开始往错误区域滑动。现象通常很像“训练还能跑，验证集先稳后抖”。

错误顺序和修正顺序python
# 错误写法：上一轮 grad 继续留着
for x, y in loader:
    pred = model(x)
    loss = criterion(pred, y)
    loss.backward()
    optimizer.step()

# 修正写法：每个 batch 先清梯度
for x, y in loader:
    pred = model(x)
    loss = criterion(pred, y)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()
检查点 看到什么 要做什么
zero_grad 第二个 batch 的梯度明显比第一个 batch 大 把清梯度放到每个 batch 的最前面
backward 参数参与了计算图，但 grad 的数值一路叠加 确认每轮只保留当前 batch 的误差信号
step 训练曲线前期下降，后期开始发散 检查学习率和梯度是否同时放大了步长
eval 训练集还能降，验证集却开始抖 对照验证集，判断是否由累计梯度影响泛化

后果很直接：训练 loss 可能还会下降，因为旧梯度里也有方向信息；但这条方向并不对应当前 batch。到了验证集，你会看到曲线忽上忽下，或者模型只在前几个 epoch 有改善，后面开始发散。这个问题如果拖到更大的模型里，排查成本会翻倍。

练习钩子

你现在可以做三件事：给任意训练循环标出 zero_grad()、backward()、step() 的位置；把缺失 zero_grad() 的代码改回去；用一个小 batch 观察参数是否真的变化。

第一次读脚本先找哪五个点

阅读顺序

拿到一份陌生的 PyTorch 项目时，先把五个位置找出来：数据怎么来、模型怎么建、loss 用什么、优化器怎么配、循环怎么跑。这个顺序和这页的课程顺序一样，对应的就是训练链路的真实顺序。先把入口找对，后面遇到报错才知道要往哪一层回溯。

  1. 先找数据入口，确认 batch 的 x 和 y 是什么形状、什么类型。
  2. 再找模型定义，确认 forward 里读了什么、写了什么。
  3. 然后找 loss 和 optimizer，确认谁负责评估，谁负责更新。
  4. 最后看训练循环，确认 zero_grad、backward、step 的顺序。

这一步很像在读地图。先确定自己站在什么位置、下一站在哪里，再去看每条小路的细节。等你把这五个位置找出来，脑子里就有了对象关系。没有这个顺序，很多代码会显得都认识，真正跑起来却说不清错在哪。

读代码时先记什么

先记输入、输出、状态变化。对训练脚本来说，状态变化通常就是参数、梯度、运行模式和数据批次。只要这四类状态盯住了，很多 bug 会自己显形。

把症状往回对

故障桥

这页还有一个作用，是给后面排错留桥。你以后在训练里看到 loss 不降、验证集乱跳、参数看似在跑却不收敛，先别急着换模型。先回到这页，把症状对回链路上的某一步：输入是否正确、前向是否连对、loss 是否标量、梯度是否写进去、参数是否真的更新、评估模式是否切对。这个映射比“多试几个超参数”更有用。

例如，loss 如果一直很大，先看标签、尺度和模型输出是否还停在随机初始化附近；如果训练集 loss 降了，验证集却完全不跟，先看是否过拟合，再看评估时是否切到 model.eval()；如果参数看上去完全不动，先看 zero_grad()、backward()、step() 的顺序，再看学习率是否被设成了几乎看不见的数。

症状 先回看哪一步 最常见的检查点
loss 不降 数据 / forward / loss 标签、形状、目标函数
梯度一直没有变化 backward 图断没断、参数有没有参与计算
参数像没动 step 学习率、zero_grad、step 顺序
验证集波动大 eval train / eval 模式、no_grad

你后面学到更复杂的网络，也可以继续沿用这个方法。CNN、RNN、Embedding、GRU 只是把模型结构换了，训练链路还是那一套。先把这条映射练熟，后面遇到新结构时就不会慌。

PyTorch 里对应哪些对象

框架映射

到了 PyTorch，概念不会消失，只是换了对象名。nn.Module 管模型，loss 负责把预测和标签压成目标，optimizer 负责更新参数，DataLoader 负责给出 batch，model.train() 和 model.eval() 负责切换运行状态。读代码时只要先把这五类对象找出来，基本就能知道一段训练脚本在做什么。

PyTorch 训练一步python
for x, y in loader:
    pred = model(x)
    loss = criterion(pred, y)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

这段代码里最容易被漏掉的是 zero_grad()、backward() 和 step() 的顺序。先清梯度，再反向写梯度，最后让优化器读取梯度更新参数。顺序一旦乱掉，训练看起来仍然会“跑”，但参数状态已经不再可信。

怎么把它用到别的脚本里

读任何训练脚本，都先找这四行：前向、loss、清梯度、更新。它们一旦顺序错了，其他代码再漂亮也只是装饰。

对象 主要职责 读什么 写什么 常见误区
DataLoader 分 batch 提供数据 数据集 不改模型 以为它会顺手做训练
Module 定义前向与参数 x 和当前参数 pred 把 forward 和训练混成一件事
Criterion 算损失 pred 和 y 标量 loss 让 loss 保持向量形状
Optimizer 更新参数 grad w / b 忘了清梯度或顺序错

再看一个更隐蔽的场景：评估态没切

有些脚本在训练集上表现正常，一到验证集就波动，原因是代码还停在训练态。dropout 继续随机丢神经元，batchnorm 继续更新统计量，验证结果自然会摇摆。你看到的是运行模式没换对。

训练态与评估态的区别python
model.train()
for x, y in loader:
    pred = model(x)
    loss = criterion(pred, y)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

model.eval()
with torch.no_grad():
    val_pred = model(val_x)
    val_loss = criterion(val_pred, val_y)
状态 会发生什么 最容易漏掉的点
train() dropout 生效、batchnorm 更新统计量 把训练和验证一起跑在这个状态里
eval() 关闭训练态特有行为 只切 model，却忘了 no_grad
no_grad() 不再为评估建图 显存和时间被白白浪费
val_loss 更接近真实泛化情况 把验证曲线当成训练曲线看

这个错误的后果很清楚：你会把训练态的随机性误判成模型能力不足，然后开始乱改学习率、层数或优化器。真正该修的是运行状态。把状态切对，很多“玄学波动”会立刻变平。

练习钩子

拿一个现成训练脚本，给验证部分补上 model.eval() 和 torch.no_grad()，然后对比两次验证曲线。你会直接看到状态切换对结果的影响。

旧写法和现在的写法

版本边界

这门课的内容本身是稳定的，但旧视频里的写法有些地方已经不适合照搬。你不需要把每个历史 API 名字都背下来，真正该留住的是对象关系、训练顺序和状态切换方式。旧课里看见的东西，最好先判断它是“概念”还是“当年的写法”。

旧课常见写法 现在更稳的写法 为什么这样改
Variable 单独拿出来讲 直接用 Tensor 和 autograd 现在参数和梯度都已经统一到 Tensor 体系里
.data 直接取值 loss.item() 或 tensor.detach() .data 容易绕开 autograd，排错时更危险
评估时只切 model.eval() model.eval() 加 with torch.no_grad() 前者切层行为，后者停止建图，两个作用不一样
反向后再去清梯度 先 optimizer.zero_grad()，再 loss.backward() 避免上一轮的梯度残留到下一轮

边界提醒

如果你现在只是想看懂课程顺序，可以先忽略这些历史细节；等你开始自己写训练脚本，再回来对照当前写法。真正要记的是：梯度写到哪儿、参数谁来改、评估时哪些状态要关掉。

误区 现场症状 立即修法
把 loss.backward() 当成更新参数 以为只要 backward 了就训练完了 记住 backward 只写 grad，真正改参数的是 optimizer.step()
忘了 zero_grad() 梯度一轮一轮叠加，loss 乱跳 把清梯度放到每个 batch 的前面

只盯训练集 loss 训练很好看，实际效果却差 把验证集和测试集一起看，先判断是否过拟合
