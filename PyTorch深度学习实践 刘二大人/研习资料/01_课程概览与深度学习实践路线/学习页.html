先看它在课程里的位置

课程入口

这一页先把整门课的学习顺序立住：从这里出发，怎么一路走到线性模型、梯度下降、反向传播，再走到更复杂的网络结构。路线看懂了，后面每一讲才知道自己是在补哪一块。

如果把这门课看成一条训练链路，这一页就是链路图。它先告诉你该先盯什么，再告诉你哪些内容暂时不用追到实现细节。学习时要知道下一步该翻哪一页，为什么翻那一页。

训练闭环

先抓主线

深度学习里最稳定的骨架只有一条：数据进入模型，模型给出预测，预测和标签算出 loss，loss 通过反向传播写入梯度，优化器读取梯度并更新参数，评估阶段再检查模型有没有真的学到东西。你后面看任何一段训练代码，都可以先按这条链路找对象，不要先沉进一堆 API 名字里。

\theta 是参数，\eta 是学习率，\nabla_{\theta}\mathcal{L} 是 loss 对参数的梯度。这个式子只做一件事：把“往哪改、改多大”讲清楚。

从最小例子开始

线性入口

线性模型是这门课第一块可计算的骨架。它把训练链路里的每个动作都露出来了：输入乘权重、再加偏置、再和标签比较。路径越短，越容易看见哪里出错。只要你能把这个例子看懂，后面换成逻辑斯蒂回归、多维特征、卷积网络，骨架都还是同一条。

\hat{y} 是预测，w 和 b 是参数，\mathcal{L} 是均方误差。把这三个符号放在一起看，就能知道模型到底在优化什么。

把对象和状态追一遍

读写追踪

学习这页时，最值钱的动作是追对象。x 和 y 从哪里来，pred 和 loss 是谁算出来的，grad 写到哪里，谁会真的改参数，这些问题只要有一个答不清，后面调参就很容易靠运气。下面这条小追踪把同一批对象穿起来，读的时候可以只盯一列。

把一个 batch 走完整

具体案例

现在把抽象链路收进一个最小 batch。假设输入是 x=[1,2,3]，标签是 y=[2,4,6]，先从一个最朴素的线性模型开始，取 w=1、b=0。第一次前向时，预测会是 [1,2,3]，这组值离标签还有明显距离，loss 也就不会小。这里最重要的是看清楚：同一批数据在 forward、loss、backward、step 之间是怎么穿过一遍的。

这个例子留下的判断法

以后看到一段训练代码，先问四个问题：输入对不对、预测算没算、loss 有没有压住、参数有没有真的变。能答上这四个问题，后面的排错会快很多。

手算一轮 batchpython

x = [1, 2, 3]
y = [2, 4, 6]
w, b = 1, 0
pred = [w * xi + b for xi in x] # [1, 2, 3]
loss = sum((pi - yi) ** 2 for pi, yi in zip(pred, y)) / len(x)

这段伪代码适合拿来检查两个常见误区。一个是把 batch 当成单个样本，结果看不出平均误差；另一个是只盯某一行公式，忘了 loss 其实是从预测和标签的整批比较来的。你读训练脚本时，只要能指出数据在这一步读了什么、写了什么，就离理解代码近了一大步。

阶段输入输出读写关系如果错了
forwardx 和参数pred读输入，写预测模型和数据接不上
losspred 和 y标量 loss读预测和标签，写目标值后面不能正常 backward
backwardlossgrad把梯度写到参数上优化器拿不到方向
stepgrad更新后的参数读梯度，改参数看起来训练了，参数没真变

再补一条常见失配：batch 形状和 dtype 不对

很多训练脚本看起来已经跑通，真正出问题的地方却在输入批次。比如分类任务里，模型输出的是 [32, 10] 的 logits，标签却还停在 [32, 1] 的浮点数。代码不一定立刻报错，但 loss 会走偏，梯度也会跟着歪。你看到的“训练能跑”，其实只是数据格式勉强接上了。

先看 shape 再进 losspython

for x, y in loader:
 print("x:", x.shape, x.dtype)
 print("y:", y.shape, y.dtype)

 logits = model(x)
 target = y.squeeze(-1).long()
 loss = criterion(logits, target)
检查点看到什么要做什么
x.shapebatch 维度有了，但特征维度被压扁先确认 DataLoader 是否把样本展开成了预期形状
y.dtype标签还是 float分类任务通常要转成 long 或正确的 one-hot 形式
criterion输入和标签维度对不上对照 loss 的约定，先改标签再改模型
loss数值忽大忽小，像在乱跳先看是否 broadcast 了，或把类别和回归混用了

这个问题的后果不止是报错。更麻烦的是，它会让训练曲线看起来像在下降，但下降方向并不对应真实标签。等你把模型换大、学习率调小、优化器换一圈，问题还在，因为真正错的是输入约定。先把 shape 和 dtype 打印出来，很多假象会马上露出原形。

练习钩子

拿一段现成训练代码，先跑一个 batch，记录 x.shape、y.shape、y.dtype 和 loss 的输入约定。然后把标签改成正确类型，再对比一次结果。

把一轮训练跑通

检查流程

把这页再往前推一步，就能得到一轮真正可检查的训练流程。先把一个 batch 送进模型，再把预测送进 loss，接着清梯度、反向、更新，最后切到评估态看一眼验证结果。只要这条链路能自己讲清楚，后面的章节就不会只是名词表。

一轮训练和一次评估python

这段代码最值得盯的是状态切换。train() 让模块回到训练态，eval() 让模块改成评估态，torch.no_grad() 则把评估阶段的梯度建图关掉。只要这三步有一个漏掉，训练脚本就可能“看起来能跑”，但状态已经不对了。

另一个高频坑：梯度累积把更新方向带偏

假设你在做二分类，batch size 只有 8，前一轮训练结束后没有清掉梯度，下一轮又直接 backward。此时 optimizer.step() 读到的方向混了前两轮样本，单个 batch 看起来都对，累计起来的参数却开始往错误区域滑动。现象通常很像“训练还能跑，验证集先稳后抖”。

错误顺序和修正顺序python

# 错误写法：上一轮 grad 继续留着
for x, y in loader:
 pred = model(x)
 loss = criterion(pred, y)
 loss.backward()
 optimizer.step()

# 修正写法：每个 batch 先清梯度
for x, y in loader:
 pred = model(x)
 loss = criterion(pred, y)
 optimizer.zero_grad()
 loss.backward()
 optimizer.step()
检查点看到什么要做什么
zero_grad第二个 batch 的梯度明显比第一个 batch 大把清梯度放到每个 batch 的最前面
backward参数参与了计算图，但 grad 的数值一路叠加确认每轮只保留当前 batch 的误差信号
step训练曲线前期下降，后期开始发散检查学习率和梯度是否同时放大了步长
eval训练集还能降，验证集却开始抖对照验证集，判断是否由累计梯度影响泛化

后果很直接：训练 loss 可能还会下降，因为旧梯度里也有方向信息；但这条方向并不对应当前 batch。到了验证集，你会看到曲线忽上忽下，或者模型只在前几个 epoch 有改善，后面开始发散。这个问题如果拖到更大的模型里，排查成本会翻倍。

练习钩子

你现在可以做三件事：给任意训练循环标出 zero_grad()、backward()、step() 的位置；把缺失 zero_grad() 的代码改回去；用一个小 batch 观察参数是否真的变化。

第一次读脚本先找哪五个点

阅读顺序

拿到一份陌生的 PyTorch 项目时，先把五个位置找出来：数据怎么来、模型怎么建、loss 用什么、优化器怎么配、循环怎么跑。这个顺序和这页的课程顺序一样，对应的就是训练链路的真实顺序。先把入口找对，后面遇到报错才知道要往哪一层回溯。

这一步很像在读地图。先确定自己站在什么位置、下一站在哪里，再去看每条小路的细节。等你把这五个位置找出来，脑子里就有了对象关系。没有这个顺序，很多代码会显得都认识，真正跑起来却说不清错在哪。

把症状往回对

故障桥

这页还有一个作用，是给后面排错留桥。你以后在训练里看到 loss 不降、验证集乱跳、参数看似在跑却不收敛，先别急着换模型。先回到这页，把症状对回链路上的某一步：输入是否正确、前向是否连对、loss 是否标量、梯度是否写进去、参数是否真的更新、评估模式是否切对。这个映射比“多试几个超参数”更有用。

症状先回看哪一步最常见的检查点
loss 不降数据 / forward / loss标签、形状、目标函数
梯度一直没有变化backward图断没断、参数有没有参与计算
参数像没动step学习率、zero_grad、step 顺序
验证集波动大evaltrain / eval 模式、no_grad

PyTorch 里对应哪些对象

框架映射

到了 PyTorch，概念不会消失，只是换了对象名。nn.Module 管模型，loss 负责把预测和标签压成目标，optimizer 负责更新参数，DataLoader 负责给出 batch，model.train() 和 model.eval() 负责切换运行状态。读代码时只要先把这五类对象找出来，基本就能知道一段训练脚本在做什么。

代码
for x, y in loader:
 pred = model(x)
 loss = criterion(pred, y)
 optimizer.zero_grad()
 loss.backward()
 optimizer.step()

这段代码里最容易被漏掉的是 zero_grad()、backward() 和 step() 的顺序。先清梯度，再反向写梯度，最后让优化器读取梯度更新参数。顺序一旦乱掉，训练看起来仍然会“跑”，但参数状态已经不再可信。

怎么把它用到别的脚本里

读任何训练脚本，都先找这四行：前向、loss、清梯度、更新。它们一旦顺序错了，其他代码再漂亮也只是装饰。

对象主要职责读什么写什么常见误区
DataLoader分 batch 提供数据数据集不改模型以为它会顺手做训练
Module定义前向与参数x 和当前参数pred把 forward 和训练混成一件事
Criterion算损失pred 和 y标量 loss让 loss 保持向量形状
Optimizer更新参数gradw / b忘了清梯度或顺序错

再看一个更隐蔽的场景：评估态没切

有些脚本在训练集上表现正常，一到验证集就波动，原因是代码还停在训练态。dropout 继续随机丢神经元，batchnorm 继续更新统计量，验证结果自然会摇摆。你看到的是运行模式没换对。

训练态与评估态的区别python

model.train()
for x, y in loader:
 pred = model(x)
 loss = criterion(pred, y)
 optimizer.zero_grad()
 loss.backward()
 optimizer.step()

model.eval()
with torch.no_grad():
 val_pred = model(val_x)
 val_loss = criterion(val_pred, val_y)
状态会发生什么最容易漏掉的点
train()dropout 生效、batchnorm 更新统计量把训练和验证一起跑在这个状态里
eval()关闭训练态特有行为只切 model，却忘了 no_grad
no_grad()不再为评估建图显存和时间被白白浪费
val_loss更接近真实泛化情况把验证曲线当成训练曲线看

这个错误的后果很清楚：你会把训练态的随机性误判成模型能力不足，然后开始乱改学习率、层数或优化器。真正该修的是运行状态。把状态切对，很多“玄学波动”会立刻变平。

练习钩子

拿一个现成训练脚本，给验证部分补上 model.eval() 和 torch.no_grad()，然后对比两次验证曲线。你会直接看到状态切换对结果的影响。

旧写法和现在的写法

版本边界

这门课的内容本身是稳定的，但旧视频里的写法有些地方已经不适合照搬。你不需要把每个历史 API 名字都背下来，真正该留住的是对象关系、训练顺序和状态切换方式。旧课里看见的东西，最好先判断它是“概念”还是“当年的写法”。

边界提醒

如果你现在只是想看懂课程顺序，可以先忽略这些历史细节；等你开始自己写训练脚本，再回来对照当前写法。真正要记的是：梯度写到哪儿、参数谁来改、评估时哪些状态要关掉。
