用PyTorch实现线性回归：Module、Loss与Optimizer

这一讲把线性回归从“能手算”推进到“能写代码”。你会看到 `nn.Module` 怎样登记参数，`MSELoss` 怎样把预测和标签压成一个标量，`SGD` 怎样读取梯度并改参数。这里不重复推导梯度，重点放在对象分工和顺序。

概念骨架

先看对象

这一页只有一条主线：输入 batch 先进模型，模型给出预测，损失函数把预测和标签压成一个数，反向传播把梯度写到参数上，优化器读梯度并更新参数。你只要把这五个对象认准，后面的代码就不再是黑箱。

把同一批样本从输入走到更新

完整样本链

这一讲最适合用一批固定样本来读：x=[[1.0],[2.0],[3.0]]，y=[[2.0],[4.0],[6.0]]。它和 nn.Linear(1, 1) 对齐，最后一维就是特征数，前一维是 batch。只要把这组样本记住，后面读 model(x)、criterion(pred, y)、optimizer.step()，你就不会总在对象之间跳来跳去。

$$ \hat y = wx + b,\qquad \mathcal{L} = \frac{1}{n}\sum_{i=1}^{n}(\hat y_i - y_i)^2 $$

这里的 \hat y 是预测，w 和 b 是参数，\mathcal{L} 是均方误差。把这三个符号放在同一行里看，Module、Loss、Optimizer 就都能对回去了。

x

[3, 1]

送进 model(x)

最后一维特征数

通常不改

如果还是 [3]，线性层会找不到清楚的特征轴

criterion(pred, y)

两批同形状张量

标量 loss

预测和标签

损失数值

如果标签形状或 dtype 错了，loss 的语义会偏掉

optimizer.step()

参数值还没变

下一轮参数

.grad

更新参数值

如果这行放错位置，训练看起来在跑，参数却不动

代码
import torch
import torch.nn as nn

model = nn.Linear(1, 1)
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)

x = torch.tensor([[1.0], [2.0], [3.0]])
y = torch.tensor([[2.0], [4.0], [6.0]])

pred = model(x)
loss = criterion(pred, y)

optimizer.zero_grad()
loss.backward()
optimizer.step()

print("x.shape =", x.shape)
print("pred.shape =", pred.shape)
print("loss =", loss.item())

先在自己的脚本里做这四个打印

先打印 x.shape、pred.shape、loss.item() 和某个参数的 .grad。如果这里都对不上，后面的调参基本是在绕远路。先把 shape 和状态看清，Module、Loss、Optimizer 才是同一条链上的三个位置。

形状契约：`[N]` 和 `[N, 1]` 不是一回事

先把输入摆正

线性回归这一页里，最常见的第一处失误不是公式，而是 shape。nn.Linear(1, 1) 把最后一维当特征维，所以一批输入通常写成 `[N, 1]，而不是简单的 [N]`。对模型来说，这“有没有特征轴”的差异。

先把一维样本整理成 batch复制

对象beforeafterreadswrites / mutates错了会怎样
x_raw[N]还没进模型原始样本列表不改线性层看不出特征轴，后面的判断会绕
x = x_raw.reshape(-1, 1)一维张量[N, 1]当前样本数补出显式特征维如果只改数值不改 shape，模型仍然不会知道每个样本有几个特征
model(x)输入形状清楚pred.shape = [N, 1]最后一维特征数输出预测如果输入和 in_features 不一致，会直接报 shape 错
criterion(pred, y)预测和标签同形状标量 loss两个 batch 张量压成一项损失如果标签还是一维，loss 可能算得出来，但语义已经不够清楚

为什么先查 shape，再查别的

因为 shape 错时，后面的所有读写都可能是错位的。你会看见 loss 还能跑，但它实际上是在错误约定上跑。把输入摆成 [N, 1]，比盯着 SGD 改学习率更先解决问题。

症状可能原因第一步修法
报矩阵乘法维度错误输入最后一维和 nn.Linear 的 in_features 对不上先把输入 reshape 成 [N, 1] 或把 in_features 改对
loss 能算，但效果怪标签和预测的 shape 语义不一致先打印 x.shape、y.shape、pred.shape 再进 loss
训练像在跑，参数却没真正变形状和参数登记都没问题，但梯度交接顺序错了回到 zero_grad -> backward -> step 的顺序检查

把这个检查变成习惯

每次写线性回归，先打印三项：x.shape、y.shape、pred.shape。如果这三项都对，再看 loss.item() 和 grad。这一步很短，但能替你挡掉一大半“代码能跑，语义却偏了”的问题。

如果你想把这一页的核心动作缩成一句话，就写：先把输入摆成 [N, 1]，再让 Module、Loss、Optimizer 按顺序接上。这个顺序对了，后面的训练才有意义。

Module：把层、参数和 forward 绑在一起

对象登记

nn.Module 的作用很具体：把参数、子层和前向路径收进一个模型对象里。__init__ 负责登记层，forward 负责定义数据怎么走。只要层是挂在 self 上的，model.parameters() 就能把它们交给优化器。

代码
import torch
import torch.nn as nn


class LinearRegression(nn.Module):
 def __init__(self):
 super().__init__()
 self.linear = nn.Linear(1, 1)

 def forward(self, x):
 return self.linear(x)

这里最容易漏的一步

层如果没有挂在 self 上，代码也许还能写，但优化器未必看得见它。你会以为自己在训练，实际上只是做了一次前向。

Loss 与 Optimizer：一个算偏差，一个读梯度

职责分开

线性回归在这一步才真正接上 PyTorch 的训练对象。MSELoss 把预测和标签压成一个标量，SGD 只负责根据参数上的梯度更新参数。criterion 不会改参数，optimizer 也不会重新算 loss。

\hat y_i 是第 i 个样本的预测，y_i 是标签，\theta 指参数，\eta 是学习率。前一个式子把偏差压成单个数，后一个式子告诉优化器怎么改参数。

代码
model = LinearRegression()
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)

这里的分工不能乱

如果把 criterion 当成优化器，或者把 step() 当成重新算梯度，训练脚本看起来会像跑着，结果却会偏。只记两句话就够：criterion 负责算偏差，optimizer 负责读梯度并更新参数。

训练循环：四行代码各管一段状态

顺序不能乱

训练循环最容易被写成模板，但每一行都在改状态。zero_grad() 清掉上一轮的梯度，backward() 把当前梯度写进去，step() 才读取这些梯度并改参数。顺序一倒，梯度就会累积或者被清空。

代码
x = torch.tensor([[1.0], [2.0], [3.0]])
y = torch.tensor([[2.0], [4.0], [6.0]])

for epoch in range(2):
 pred = model(x)
 loss = criterion(pred, y)

 optimizer.zero_grad()
 loss.backward()
 optimizer.step()

 print(epoch, loss.item())

顺序的意义

zero_grad() 放在 backward() 前面，是为了让每个 batch 只看当前梯度。PyTorch 默认会累积梯度，顺序写错，上一轮的值就会混进来。

手动更新、优化器更新和 `torch.no_grad()`

同一动作的三种写法

理解 SGD 的最好方式，是先手动写一遍更新。你会看到它本质上就是把参数减去学习率乘以梯度。手动写的时候要包上 torch.no_grad()，否则更新本身也会被记进计算图。

代码
lr = 0.01

with torch.no_grad():
 for p in model.parameters():
 p -= lr * p.grad

预测阶段最小片段复制

代码
model.eval()

with torch.no_grad():
 pred = model(torch.tensor([[4.0]]))
 print(pred)

这一步和 SGD 的关系

optimizer.step() 做的事和上面的手动更新是一回事，只是把更新规则封装起来了。后面你换成 Adam、momentum 或别的优化器，改的是更新细节，不是这条读写链路。

训练态和预测态要分开

边界清楚

线性回归本身没有 Dropout 和 BatchNorm，但训练态和预测态分开的习惯要从这里开始。model.eval() 切的是模块行为，torch.no_grad() 切的是是否继续建图。它们不做同一件事。

为什么两个动作常一起写

eval() 负责把模块切到评估行为，no_grad() 负责不再记梯度。前者管模块，后者管图；只写一个，常常不够稳。

下一讲会把输出换成二分类概率，损失约定也会跟着变。这里先把训练闭环和预测边界看稳，不提前跳到下一页的损失形式。

16_zero_before_backwarderror diagnosis

Q: 16. 为什么 optimizer.zero_grad() 要放在 backward() 前面？

参考答案 / 评分要点

。PyTorch 的梯度默认会累积，不会自动覆盖上一轮的值。把 zero_grad() 放在 backward() 前面，才能保证这一轮 batch 只用当前梯度，不被上一轮残留影响。如果你是故意做梯度累积，那是另一种设计；默认训练循环不应该把这个顺序反过来。
