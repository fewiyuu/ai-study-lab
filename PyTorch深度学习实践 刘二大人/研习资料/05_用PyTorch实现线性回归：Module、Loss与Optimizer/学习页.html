第 05 讲 · 训练闭环落地

用 PyTorch 实现线性回归：Module、Loss 与 Optimizer

这一讲把线性回归从“能手算”推进到“能写代码”。你会看到 `nn.Module` 怎样登记参数，`MSELoss` 怎样把预测和标签压成一个标量，`SGD` 怎样读取梯度并改参数。这里不重复推导梯度，重点放在对象分工和顺序。

课程地图 上一讲：反向传播 下一讲：逻辑斯蒂回归

读完这一页，先能做什么

  • 把 `model(x) -> loss -> zero_grad -> backward -> step` 排顺。
  • 说清 `Module`、`Loss`、`Optimizer` 各自读什么、写什么。
  • 分开看训练态、预测态和手动更新，避免把职责混掉。

概念骨架

先看对象

这一页只有一条主线：输入 batch 先进模型，模型给出预测，损失函数把预测和标签压成一个数，反向传播把梯度写到参数上，优化器读梯度并更新参数。你只要把这五个对象认准，后面的代码就不再是黑箱。

数据批次 x, y

读自 DataLoader。这里先只看一批样本，不把样本换来换去。

预测 pred / y_hat

由模型前向算出来。它读输入和参数，先不改参数。

标量 loss 一个数

损失函数把一批样本压成一个标量，后面的反向才有入口。

梯度 grad / .grad

`loss.backward()` 不改参数，只把梯度写到叶子参数上。

更新后参数 w, b

优化器读取 `.grad`，真正改参数值。顺序错，参数就不会变。

步骤 读什么 写什么 / 改什么 错一步会看到什么 下一步先查什么
`nn.Module` 输入维度和层配置 把层和参数登记到模型里 优化器找不到参数 层是不是挂在 `self` 上
`criterion` 预测和标签 标量 loss loss 形状不对，后面不好 `backward()` 预测和标签的 shape / dtype
`backward()` 计算图和标量 loss 把梯度写到参数的 `.grad` `.grad` 为空或还是旧值 是否真的建图，参数是否参与计算
`optimizer.step()` 参数和 `.grad` 下一轮参数值 参数看起来在训练，数值却没动 参数是否传给优化器，`step()` 是否在 `backward()` 后面
`model.eval()` / `no_grad` 当前模块状态 关闭训练态行为或关闭建图 评估时还在攒图，或者层行为没切 推理和训练有没有分开

把同一批样本从输入走到更新

完整样本链

这一讲最适合用一批固定样本来读：x=[[1.0],[2.0],[3.0]]，y=[[2.0],[4.0],[6.0]]。它和 nn.Linear(1, 1) 对齐，最后一维就是特征数，前一维是 batch。只要把这组样本记住，后面读 model(x)、criterion(pred, y)、optimizer.step()，你就不会总在对象之间跳来跳去。

这组样本还有一个好处：它足够小，能把线性回归的最小闭环全部摊开。`w=1, b=0` 时，预测是 `[1, 2, 3]`；`w=2, b=0` 时，预测正好贴住标签。这样一来，你能直接看见参数变化怎样改输出，损失怎样随之变化，优化器又怎样把参数往更合适的位置推。

$$ \hat y = wx + b,\qquad \mathcal{L} = \frac{1}{n}\sum_{i=1}^{n}(\hat y_i - y_i)^2 $$

这里的 \hat y 是预测，w 和 b 是参数，\mathcal{L} 是均方误差。把这三个符号放在同一行里看，`Module`、`Loss`、`Optimizer` 就都能对回去了。

对象 / 代码位 before after reads writes / mutates 如果写错，现场会怎样
x [3, 1] 送进 model(x) 最后一维特征数 通常不改 如果还是 [3]，线性层会找不到清楚的特征轴
model(x) 只有参数，没有本轮预测 pred 输入和参数 输出预测张量 如果模型层没登记好，预测能算，参数却可能没被优化器看见
criterion(pred, y) 两批同形状张量 标量 loss 预测和标签 损失数值 如果标签形状或 dtype 错了，loss 的语义会偏掉
loss.backward() 只有 loss，没有梯度 weight.grad 和 bias.grad 计算图和标量 loss 把梯度写到叶子参数上 如果参数没参与图，.grad 还是空的
optimizer.step() 参数值还没变 下一轮参数 .grad 更新参数值 如果这行放错位置，训练看起来在跑，参数却不动
把同一批样本跑完一轮
import torch
import torch.nn as nn

model = nn.Linear(1, 1)
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)

x = torch.tensor([[1.0], [2.0], [3.0]])
y = torch.tensor([[2.0], [4.0], [6.0]])

pred = model(x)
loss = criterion(pred, y)

optimizer.zero_grad()
loss.backward()
optimizer.step()

print("x.shape =", x.shape)
print("pred.shape =", pred.shape)
print("loss =", loss.item())
症状 你先查什么 为什么先查它
x 还是 [3] 输入 shape 线性层把最后一维当特征，[3] 会把这层语义弄模糊
loss 能算，但参数没变化 zero_grad()、backward()、step() 的顺序 参数更新依赖前面已经写好的梯度
预测阶段还在攒图 model.eval() 和 torch.no_grad() 一个切模块行为，一个停建图，两个作用不一样
损失对象报“不能直接调用” 是不是少了 () nn.MSELoss 是类，nn.MSELoss() 才是实例

先在自己的脚本里做这四个打印

先打印 x.shape、pred.shape、loss.item() 和某个参数的 .grad。如果这里都对不上，后面的调参基本是在绕远路。先把 shape 和状态看清，Module、Loss、Optimizer 才是同一条链上的三个位置。

形状契约：`[N]` 和 `[N, 1]` 不是一回事

先把输入摆正

线性回归这一页里，最常见的第一处失误不是公式，而是 shape。`nn.Linear(1, 1)` 把最后一维当特征维，所以一批输入通常写成 `[N, 1]`，而不是简单的 `[N]`。对模型来说，这不是排版差异，而是“有没有特征轴”的差异。

如果你把 `[1, 2, 3]` 直接喂进去，视觉上它还是三个数，但框架会把这组数据当成没有显式特征维的一维张量。线性层、损失函数和后面的打印结果就可能开始各说各话。先把 batch 维和特征维摆正，是这一讲最便宜的检查。

先把一维样本整理成 batch
x_raw = torch.tensor([1.0, 2.0, 3.0])
y_raw = torch.tensor([2.0, 4.0, 6.0])

x = x_raw.reshape(-1, 1)
y = y_raw.reshape(-1, 1)

print("x_raw.shape =", x_raw.shape)
print("x.shape =", x.shape)
print("y.shape =", y.shape)
对象 before after reads writes / mutates 错了会怎样
x_raw [N] 还没进模型 原始样本列表 不改 线性层看不出特征轴，后面的判断会绕
x = x_raw.reshape(-1, 1) 一维张量 [N, 1] 当前样本数 补出显式特征维 如果只改数值不改 shape，模型仍然不会知道每个样本有几个特征
model(x) 输入形状清楚 pred.shape = [N, 1] 最后一维特征数 输出预测 如果输入和 `in_features` 不一致，会直接报 shape 错
criterion(pred, y) 预测和标签同形状 标量 loss 两个 batch 张量 压成一项损失 如果标签还是一维，loss 可能算得出来，但语义已经不够清楚

为什么先查 shape，再查别的

因为 shape 错时，后面的所有读写都可能是错位的。你会看见 `loss` 还能跑，但它实际上是在错误约定上跑。把输入摆成 `[N, 1]`，比盯着 `SGD` 改学习率更先解决问题。

症状 可能原因 第一步修法
报矩阵乘法维度错误 输入最后一维和 `nn.Linear` 的 `in_features` 对不上 先把输入 reshape 成 `[N, 1]` 或把 `in_features` 改对
loss 能算，但效果怪 标签和预测的 shape 语义不一致 先打印 `x.shape`、`y.shape`、`pred.shape` 再进 loss
训练像在跑，参数却没真正变 形状和参数登记都没问题，但梯度交接顺序错了 回到 `zero_grad -> backward -> step` 的顺序检查

把这个检查变成习惯

每次写线性回归，先打印三项：`x.shape`、`y.shape`、`pred.shape`。如果这三项都对，再看 `loss.item()` 和 `grad`。这一步很短，但能替你挡掉一大半“代码能跑，语义却偏了”的问题。

如果你想把这一页的核心动作缩成一句话，就写：先把输入摆成 `[N, 1]`，再让 `Module`、`Loss`、`Optimizer` 按顺序接上。这个顺序对了，后面的训练才有意义。

Module：把层、参数和 forward 绑在一起

对象登记

`nn.Module` 的作用很具体：把参数、子层和前向路径收进一个模型对象里。`__init__` 负责登记层，`forward` 负责定义数据怎么走。只要层是挂在 `self` 上的，`model.parameters()` 就能把它们交给优化器。

最小模型定义
import torch
import torch.nn as nn


class LinearRegression(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = nn.Linear(1, 1)

    def forward(self, x):
        return self.linear(x)
对象 / 代码位 before after reads writes / mutates 常见症状
`self.linear = nn.Linear(1, 1)` 只是一个局部想法 注册成子模块 输入输出维度 weight / bias 进入模型注册表 如果不挂在 `self` 上，优化器可能拿不到参数
`forward(self, x)` 没有固定入口 定义 `model(x)` 的路径 输入张量和已注册层 输出预测张量 只有类，没有 `forward`，模型就没法跑前向
`model.parameters()` 没有参数集合 得到可训练参数迭代器 模块注册表 把参数句柄交给优化器 如果模型里没登记层，优化器拿到的可能是空的
`nn.Linear(1, 1)` 只知道是线性层 一个输入特征到一个输出特征的仿射变换 最后一维特征数 输出仍是 `[N, 1]` 如果输入还是 `[N]`，特征轴就不清楚了

这里最容易漏的一步

层如果没有挂在 `self` 上，代码也许还能写，但优化器未必看得见它。你会以为自己在训练，实际上只是做了一次前向。

Loss 与 Optimizer：一个算偏差，一个读梯度

职责分开

线性回归在这一步才真正接上 PyTorch 的训练对象。`MSELoss` 把预测和标签压成一个标量，`SGD` 只负责根据参数上的梯度更新参数。`criterion` 不会改参数，`optimizer` 也不会重新算 loss。

$$ \mathcal{L} = \frac{1}{n}\sum_{i=1}^{n}(\hat y_i - y_i)^2,\qquad \theta \leftarrow \theta - \eta \, \nabla_{\theta}\mathcal{L} $$

\hat y_i 是第 i 个样本的预测，y_i 是标签，\theta 指参数，\eta 是学习率。前一个式子把偏差压成单个数，后一个式子告诉优化器怎么改参数。

`criterion` 和 `optimizer` 的初始化
model = LinearRegression()
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)
对象 读什么 写什么 先别混成什么
`criterion = nn.MSELoss()` 预测和标签 标量 loss 不是参数更新器
`optimizer = SGD(model.parameters(), lr=...)` 参数和它们的 `.grad` 下一轮参数值 不是 loss 计算器
`criterion(pred, y)` 当前 batch 的预测和标签 一项损失 不是训练循环本身
`optimizer.step()` 当前参数和 `.grad` 更新后的参数 不是自动反向传播

这里的分工不能乱

如果把 `criterion` 当成优化器，或者把 `step()` 当成重新算梯度，训练脚本看起来会像跑着，结果却会偏。只记两句话就够：`criterion` 负责算偏差，`optimizer` 负责读梯度并更新参数。

训练循环：四行代码各管一段状态

顺序不能乱

训练循环最容易被写成模板，但每一行都在改状态。`zero_grad()` 清掉上一轮的梯度，`backward()` 把当前梯度写进去，`step()` 才读取这些梯度并改参数。顺序一倒，梯度就会累积或者被清空。

输入批次 x, y

线性层输入写成 `[N, 1]`，这样最后一维才是特征轴。

前向输出 pred

预测张量的形状通常和标签一致，便于 loss 直接比较。

标量 loss 一个数

训练通常从标量 loss 出发，反向传播才知道从哪里开始回传。

梯度 weight.grad / bias.grad

`backward()` 把梯度写到叶子参数上，不直接改参数值。

更新后参数 w, b

`optimizer.step()` 读 `.grad`，写出下一轮参数。

最小训练循环
x = torch.tensor([[1.0], [2.0], [3.0]])
y = torch.tensor([[2.0], [4.0], [6.0]])

for epoch in range(2):
    pred = model(x)
    loss = criterion(pred, y)

    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

    print(epoch, loss.item())
阶段 输入 输出 读写关系 如果错了
forward `x` 和当前参数 `pred` 读输入，写预测 模型和数据接不上
loss `pred` 和 `y` 标量 `loss` 读预测和标签，写目标值 后面不能顺利 `backward()`
backward `loss` `grad` 把梯度写到参数上 优化器拿不到方向
step `grad` 更新后的参数 读梯度，改参数 看起来训练了，参数没真变

顺序的意义

`zero_grad()` 放在 `backward()` 前面，是为了让每个 batch 只看当前梯度。PyTorch 默认会累积梯度，顺序写错，上一轮的值就会混进来。

手动更新、优化器更新和 `torch.no_grad()`

同一动作的三种写法

理解 `SGD` 的最好方式，是先手动写一遍更新。你会看到它本质上就是把参数减去学习率乘以梯度。手动写的时候要包上 `torch.no_grad()`，否则更新本身也会被记进计算图。

手动更新参数
lr = 0.01

with torch.no_grad():
    for p in model.parameters():
        p -= lr * p.grad
预测阶段最小片段
model.eval()

with torch.no_grad():
    pred = model(torch.tensor([[4.0]]))
    print(pred)
动作 读什么 写什么 什么时候用 常见提醒
手动更新 参数和 `.grad` 直接改参数值 理解更新规则，或做极简实验 一定要放进 `torch.no_grad()`
优化器更新 `model.parameters()` 和 `.grad` 下一轮参数值 正式训练 优化器不会替你重新算梯度
预测 参数和输入 输出结果 推理、验证、部署前检查 不要把预测态和训练态混成一件事

这一步和 `SGD` 的关系

`optimizer.step()` 做的事和上面的手动更新是一回事，只是把更新规则封装起来了。后面你换成 `Adam`、`momentum` 或别的优化器，改的是更新细节，不是这条读写链路。

训练态和预测态要分开

边界清楚

线性回归本身没有 `Dropout` 和 `BatchNorm`，但训练态和预测态分开的习惯要从这里开始。`model.eval()` 切的是模块行为，`torch.no_grad()` 切的是是否继续建图。它们不做同一件事。

动作 做什么 不做什么 什么时候用
`model.train()` 切回训练态 不更新参数，也不自动求梯度 每个训练 epoch 开头
`model.eval()` 切到评估态 不关闭梯度建图 验证和推理前
`with torch.no_grad()` 停止建图 不改变模块行为 验证、推理、手动改参数

为什么两个动作常一起写

`eval()` 负责把模块切到评估行为，`no_grad()` 负责不再记梯度。前者管模块，后者管图；只写一个，常常不够稳。

下一讲会把输出换成二分类概率，损失约定也会跟着变。这里先把训练闭环和预测边界看稳，不提前跳到下一页的损失形式。

故障实验

先看症状

训练脚本出了问题，先看症状，再找对应的状态交接点。下面的几种情况，基本都能回到这一页里的几个动作。

现象：第二个 batch 的梯度像还带着第一个 batch 的值。
先查 `optimizer.zero_grad()` 是否放在 `loss.backward()` 前面。
症状 可能原因 第一步检查
梯度像在累积 `zero_grad()` 太晚，或者根本没调 把 `zero_grad()` 放回 `backward()` 前面
参数不动 `step()` 没执行，或者优化器没拿到参数 检查 `optimizer.step()` 和 `model.parameters()`
评估时还在建图 少了 `eval()` 或 `no_grad()` 把训练态和预测态分开写
损失对象不能直接用 把类当成实例了 补上 `()`，写成 `nn.MSELoss()`

15. 参数几乎不动时，你会先查哪三步？

至少说出 `zero_grad()`、`backward()`、`step()` 的顺序，以及参数有没有真的传给优化器。

16. 为什么 optimizer.zero_grad() 要放在 backward() 前面？

答案要提到：梯度默认会累积；清零放在前面，当前 batch 才能单独算。

参考答案 / 评分要点

PyTorch 的梯度默认会累积，不会自动覆盖上一轮的值。把 `zero_grad()` 放在 `backward()` 前面，才能保证这一轮 batch 只用当前梯度，不被上一轮残留影响。如果你是故意做梯度累积，那是另一种设计；默认训练循环不应该把这个顺序反过来。
