# 把同一批样本从输入走到更新 {#anchor-case}
<!-- tag: 完整样本链 -->

这一讲最适合用一批固定样本来读：`x=[[1.0],[2.0],[3.0]]`，`y=[[2.0],[4.0],[6.0]]`。它和 `nn.Linear(1, 1)` 对齐，最后一维就是特征数，前一维是 batch。只要把这组样本记住，后面读 `model(x)`、`criterion(pred, y)`、`optimizer.step()`，你就不会总在对象之间跳来跳去。

这组样本还有一个好处：它足够小，能把线性回归的最小闭环全部摊开。`w=1, b=0` 时，预测是 `[1, 2, 3]`；`w=2, b=0` 时，预测正好贴住标签。这样一来，你能直接看见参数变化怎样改输出，损失怎样随之变化，优化器又怎样把参数往更合适的位置推。

$$
 \hat y = wx + b,\qquad
 \mathcal{L} = \frac{1}{n}\sum_{i=1}^{n}(\hat y_i - y_i)^2
 $$
 这里的 `\hat y` 是预测，`w` 和 `b` 是参数，`\mathcal{L}` 是均方误差。把这三个符号放在同一行里看，`Module`、`Loss`、`Optimizer` 就都能对回去了。

# 形状契约：`[N]` 和 `[N, 1]` 不是一回事 {#shape-contract}
<!-- tag: 先把输入摆正 -->

线性回归这一页里，最常见的第一处失误不是公式，而是 shape。`nn.Linear(1, 1)` 把最后一维当特征维，所以一批输入通常写成 ``[N, 1]``，而不是简单的 ``[N]``。对模型来说，这“有没有特征轴”的差异。

如果你把 ``[1, 2, 3]`` 直接喂进去，视觉上它还是三个数，但框架会把这组数据当成没有显式特征维的一维张量。线性层、损失函数和后面的打印结果就可能开始各说各话。先把 batch 维和特征维摆正，是这一讲最便宜的检查。

先把一维样本整理成 batch复制

```python
x_raw = torch.tensor([1.0, 2.0, 3.0])
y_raw = torch.tensor([2.0, 4.0, 6.0])

x = x_raw.reshape(-1, 1)
y = y_raw.reshape(-1, 1)

print("x_raw.shape =", x_raw.shape)
print("x.shape =", x.shape)
print("y.shape =", y.shape)
```

### 把这个检查变成习惯

# Module：把层、参数和 forward 绑在一起 {#module}
<!-- tag: 对象登记 -->

`nn.Module` 的作用很具体：把参数、子层和前向路径收进一个模型对象里。`__init__` 负责登记层，`forward` 负责定义数据怎么走。只要层是挂在 `self` 上的，`model.parameters()` 就能把它们交给优化器。

最小模型定义复制

```python
import torch
import torch.nn as nn


class LinearRegression(nn.Module):
 def __init__(self):
 super().__init__()
 self.linear = nn.Linear(1, 1)

 def forward(self, x):
 return self.linear(x)
```

`self.linear = nn.Linear(1, 1)`
只是一个局部想法
注册成子模块
输入输出维度
weight / bias 进入模型注册表
如果不挂在 `self` 上，优化器可能拿不到参数


`forward(self, x)`
没有固定入口

### 这里最容易漏的一步
层如果没有挂在 `self` 上，代码也许还能写，但优化器未必看得见它。你会以为自己在训练，实际上只是做了一次前向。

# Loss 与 Optimizer：一个算偏差，一个读梯度 {#loss-optimizer}
<!-- tag: 职责分开 -->

线性回归在这一步才真正接上 PyTorch 的训练对象。`MSELoss` 把预测和标签压成一个标量，`SGD` 只负责根据参数上的梯度更新参数。`criterion` 不会改参数，`optimizer` 也不会重新算 loss。

$$
 \mathcal{L} = \frac{1}{n}\sum_{i=1}^{n}(\hat y_i - y_i)^2,\qquad
 \theta \leftarrow \theta - \eta \, \nabla_{\theta}\mathcal{L}
 $$
 `\hat y_i` 是第 `i` 个样本的预测，`y_i` 是标签，`\theta` 指参数，`\eta` 是学习率。前一个式子把偏差压成单个数，后一个式子告诉优化器怎么改参数。

`criterion` 和 `optimizer` 的初始化复制

```python
model = LinearRegression()
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)
```

对象
读什么
写什么
先别混成什么


`criterion = nn.MSELoss()`
预测和标签
标量 loss
不是参数更新器


`optimizer = SGD(model.parameters(), lr=...)`
参数和它们的 `.grad`
下一轮参数值
不是 loss 计算器


`criterion(pred, y)`
当前 batch 的预测和标签
一项损失
不是训练循环本身


`optimizer.step()`
当前参数和 `.grad`
更新后的参数
不是自动反向传播

## box warn
### 这里的分工不能乱
如果把 `criterion` 当成优化器，或者把 `step()` 当成重新算梯度，训练脚本看起来会像跑着，结果却会偏。只记两句话就够：`criterion` 负责算偏差，`optimizer` 负责读梯度并更新参数。

# 训练循环：四行代码各管一段状态 {#loop}
<!-- tag: 顺序不能乱 -->

训练循环最容易被写成模板，但每一行都在改状态。`zero_grad()` 清掉上一轮的梯度，`backward()` 把当前梯度写进去，`step()` 才读取这些梯度并改参数。顺序一倒，梯度就会累积或者被清空。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

最小训练循环复制

```python
x = torch.tensor([[1.0], [2.0], [3.0]])
y = torch.tensor([[2.0], [4.0], [6.0]])

for epoch in range(2):
 pred = model(x)
 loss = criterion(pred, y)

 optimizer.zero_grad()
 loss.backward()
 optimizer.step()

 print(epoch, loss.item())
```

输入
输出
读写关系
如果错了


forward
`x` 和当前参数
`pred`
读输入，写预测
模型和数据接不上


loss
`pred` 和 `y`
标量 `loss`
读预测和标签，写目标值
后面不能顺利 `backward()`


backward
`loss`
`grad`
把梯度写到参数上
优化器拿不到方向


step
`grad`
更新后的参数
读梯度，改参数
看起来训练了，参数没真变

## box info
### 顺序的意义
`zero_grad()` 放在 `backward()` 前面，是为了让每个 batch 只看当前梯度。PyTorch 默认会累积梯度，顺序写错，上一轮的值就会混进来。

# 手动更新、优化器更新和 `torch.no_grad()` {#update-path}
<!-- tag: 同一动作的三种写法 -->

理解 `SGD` 的最好方式，是先手动写一遍更新。你会看到它本质上就是把参数减去学习率乘以梯度。手动写的时候要包上 `torch.no_grad()`，否则更新本身也会被记进计算图。

手动更新参数复制

```python
lr = 0.01

with torch.no_grad():
 for p in model.parameters():
 p -= lr * p.grad
```

预测阶段最小片段复制

```python
model.eval()

with torch.no_grad():
 pred = model(torch.tensor([[4.0]]))
 print(pred)
```

读什么
写什么
什么时候用
常见提醒


手动更新

直接改参数值
理解更新规则，或做极简实验
一定要放进 `torch.no_grad()`


优化器更新
`model.parameters()` 和 `.grad`
下一轮参数值
正式训练
优化器不会替你重新算梯度


预测
参数和输入
输出结果
推理、验证、部署前检查
不要把预测态和训练态混成一件事

## box warn
### 这一步和 `SGD` 的关系
`optimizer.step()` 做的事和上面的手动更新是一回事，只是把更新规则封装起来了。后面你换成 `Adam`、`momentum` 或别的优化器，改的是更新细节，不是这条读写链路。

# 训练态和预测态要分开 {#prediction-boundary}
<!-- tag: 边界清楚 -->

线性回归本身没有 `Dropout` 和 `BatchNorm`，但训练态和预测态分开的习惯要从这里开始。`model.eval()` 切的是模块行为，`torch.no_grad()` 切的是是否继续建图。它们不做同一件事。

动作

不做什么

`model.train()`
切回训练态
不更新参数，也不自动求梯度
每个训练 epoch 开头


`model.eval()`
切到评估态
不关闭梯度建图
验证和推理前


`with torch.no_grad()`
停止建图
不改变模块行为
验证、推理、手动改参数

## box info
### 为什么两个动作常一起写
`eval()` 负责把模块切到评估行为，`no_grad()` 负责不再记梯度。前者管模块，后者管图；只写一个，常常不够稳。

下一讲会把输出换成二分类概率，损失约定也会跟着变。这里先把训练闭环和预测边界看稳，不提前跳到下一页的损失形式。

### 12_eval_reason synthesis

**Q:** 12. 为什么评估时常把 `model.eval()` 和 `torch.no_grad()` 一起用？

Explain: 好答案会说：`model.eval()` 负责把模型切到评估态，后面如果有 `Dropout`、`BatchNorm` 之类的层，它们的行为会跟训练时不同；`torch.no_grad()` 负责停止建图，验证和推理时不用再保留梯度，省内存也省计算。两者管的不是同一件事，所以通常一起写。

### 15_params_not_move diagnosis

**Q:** 15. 参数几乎不动时，你会先查哪三步？

Explain: 稳妥的顺序是：先确认 `optimizer.step()` 有没有真的执行；再确认 `optimizer.zero_grad()`、`loss.backward()`、`optimizer.step()` 的顺序是否正确；最后确认 `model.parameters()` 是否真的传进了优化器，参数是否参与了这张图。若能再补一句“梯度值是不是接近 0”，会更完整。

### 16_zero_before_backward diagnosis

**Q:** 16. 为什么 `optimizer.zero_grad()` 要放在 `backward()` 前面？

Explain: PyTorch 的梯度默认会累积，不会自动覆盖上一轮的值。把 `zero_grad()` 放在 `backward()` 前面，才能保证这一轮 batch 只用当前梯度，不被上一轮残留影响。如果你是故意做梯度累积，那是另一种设计；默认训练循环不应该把这个顺序反过来。
