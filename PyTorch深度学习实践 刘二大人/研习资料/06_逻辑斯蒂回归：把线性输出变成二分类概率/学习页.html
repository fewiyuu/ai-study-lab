逻辑斯蒂回归：把线性输出变成二分类概率

这一页只盯一件事：线性分数 logit 怎么变成 0 到 1 之间的概率，概率怎么再变成类别，PyTorch 里该把 BCELoss 和 BCEWithLogitsLoss 放在什么位置。你不用先背整套推导，先把输入、输出、阈值和损失函数的约定看准。

读完以后，先能判断一段二分类代码到底是在处理 logits 还是概率；再往后看第 07 讲时，注意力会转到特征维和矩阵形状，而不是继续卡在输出层。

上一讲：线性回归 课程地图 下一讲：多维特征输入

先抓住这四个对象

  • logit：模型的原始分数，还不是概率。
  • sigmoid：把分数压到 0 到 1。
  • threshold：把概率切成类别。
  • BCEWithLogitsLoss：直接吃 logits 的损失。

学习目标

先定边界

这页先解决什么

把二分类模型的输出链路看清楚：线性分数、概率、阈值、标签、损失函数分别负责什么。只要这个链路顺了，后面读二分类训练脚本时就不会把概率、loss 和评估指标混在一起。

先别追的东西

多分类的 softmax、交叉熵推导、概率校准和 F1 / ROC 的细节先放一边。这里先把“二分类怎么接损失”钉住，别把问题提前扩成整个分类家族。

目标 读到什么算懂了 能立刻检查什么
输出含义 知道二分类输出先是分数，再是概率，最后才是类别。 能分清 logit 和 probability。
损失约定 知道 BCELoss 和 BCEWithLogitsLoss 吃的不是同一个输入。 看见 loss 就能猜到前向末层是否已经做了 sigmoid。
阈值判断 知道阈值是决策规则，不是模型参数。 能解释为什么 0.5 不是绝对固定。

对象链路

先看数据怎么走

二分类里最容易搞乱的，不是公式本身，而是对象的顺序。模型先给出原始分数，sigmoid 再把它压成概率，阈值把概率切成类别，损失函数则拿训练标签去约束模型。把这四步分开，你就知道每一步该读什么、写什么、哪里不能乱插一层。

logit [B, 1]

这一步只负责把输入变成分数，不负责解释成概率。

probability [B, 1]

sigmoid 把分数压到 0 到 1，才有“像概率”的读法。

prediction [B, 1]

阈值只做决策，不改模型参数。

loss scalar

训练时只拿标量 loss 继续往后传。

label [B, 1] float

BCE 的标签通常要和输出对齐，且保持 0/1 的浮点形式。

步骤 读什么 写什么 错了会看到什么
linear 输入 x 和权重 w、b 原始分数 z 把 z 当概率，后面所有判断都偏了
sigmoid 原始分数 z 0 到 1 的 p 误以为输出已经是最终分类结果
threshold 概率 p 类别 pred 阈值设得太高或太低，正负类偏向会变
criterion logits 和标签 标量 loss loss 还在向量形状，或者输入先后顺序错了

sigmoid

把分数压成概率

sigmoid 做的事很单纯：把任意实数压到 0 到 1 之间。它不负责判断正负类，只负责把模型的原始分数改写成可以解释的概率值。真正的分类决策还在后面，要靠阈值来切。

$$ \sigma(z)=\frac{1}{1+e^{-z}},\qquad p=\sigma(z) $$

z 是模型输出的原始分数，p 是正类概率。z=0 时，p=0.5，这也是常见阈值的中线位置。

z sigmoid(z) 阈值 0.5 的判断
-2 约 0.119 更像负类
0 0.5 卡在边界上
2 约 0.881 更像正类

这里最容易混的一点

概率是给人读的，类别是给任务用的。sigmoid 只负责把分数变成概率，真正把样本判成正类还是负类的，是阈值这一步。

损失约定

先对齐输入

二分类里最常见的两种写法，差别只在损失函数吃什么。BCELoss 直接吃概率，所以模型 forward 里要先做 sigmoid；BCEWithLogitsLoss 直接吃 logits，内部会帮你做 sigmoid，数值上更稳，也更少出错。

$$ \mathrm{BCE}(p,y)=-\big[y\log(p)+(1-y)\log(1-p)\big] $$

y 是 0/1 标签，p 是模型给出的正类概率。真实标签是 1 时，p 越接近 1，loss 越小；真实标签是 0 时，p 越接近 0，loss 越小。

损失 输入 forward 末层该长什么样 常见失误
BCELoss 概率 forward 里已经做过 sigmoid 把 logits 直接喂进去
BCEWithLogitsLoss logits forward 末层只返回线性分数 先 sigmoid 再喂进去，等于重复一遍
推荐写法：forward 只返回 logitspython
import torch
from torch import nn

class BinaryModel(nn.Module):
    def __init__(self, in_features):
        super().__init__()
        self.linear = nn.Linear(in_features, 1)

    def forward(self, x):
        return self.linear(x)  # 只返回 logits

model = BinaryModel(8)
criterion = nn.BCEWithLogitsLoss()
错误写法：把概率再喂给 BCEWithLogitsLosspython
logits = model(x)
prob = torch.sigmoid(logits)
loss = nn.BCEWithLogitsLoss()(prob, y.float())
# BCEWithLogitsLoss 内部已经做过 sigmoid 了

标签也要对齐

二分类标签常写成 0/1，但进 BCE 时通常要是浮点张量，形状也最好和输出对齐。输出是 [B, 1]，标签也尽量保持 [B, 1]；这样不会把广播当成正确答案。

PyTorch 对应物

把约定落到代码

PyTorch 里，真正要记住的不是某个类名，而是它们各自读写什么。Module 组织参数，forward 产生 logits，criterion 把 logits 和标签压成 loss，optimizer 读取梯度并更新参数。评估时再把 logits 变成概率，最后才做阈值判断。

对象 before after read / write
model.forward 输入 x logits 读输入和参数；写出原始分数
criterion logits、float 标签 标量 loss 读预测和标签；写训练目标
optimizer 参数的 .grad 新的参数值 读梯度；写参数
threshold 概率 类别 读概率；写决策结果
训练与评估的最小分界python
logits = model(x_batch)
loss = criterion(logits, y_batch.float())

optimizer.zero_grad()
loss.backward()
optimizer.step()

model.eval()
with torch.no_grad():
    prob = torch.sigmoid(model(x_batch))
    pred = (prob >= 0.5).long()

这三行最容易错

  • y_batch.float() 忘了转，BCE 可能直接卡住。
  • 先 sigmoid 再喂 BCEWithLogitsLoss，等于重复一遍。
  • 把阈值判断塞进训练损失里，后面就很难调试。

单样本手算

把公式算一遍

先盯一个样本。假设线性层给出的分数是 z=2.0，真实标签是 y=1。这时 sigmoid(2.0) 约等于 0.881，模型对正类已经很自信。如果真实标签换成 0，同样的分数就会让损失明显变大。这种对比比空讲公式更能看出 BCE 在奖励什么、惩罚什么。

$$ p=\sigma(z),\qquad \mathcal{L}_{y=1}=-\log(p),\qquad \mathcal{L}_{y=0}=-\log(1-p) $$

这里的 p 是正类概率。y=1 时，p 越接近 1，loss 越小；y=0 时，p 越接近 0，loss 越小。BCE 看的不是“最后判对没”，而是“真类概率留了多少”。

情形 z p = sigmoid(z) y loss 读法
预测对了且很自信 2.0 0.881 1 0.127 模型把概率留给了真类，所以惩罚很轻。
预测错了且很自信 2.0 0.881 0 2.127 同样的分数，标签翻过来，惩罚立刻变重。
边界附近 0.0 0.500 1 0.693 边界位置的惩罚中等，说明模型还没给出足够偏向。
把单样本追踪写进脚本python
import torch

z = torch.tensor([[2.0]])
y1 = torch.tensor([[1.0]])
y0 = torch.tensor([[0.0]])

p = torch.sigmoid(z)
loss_y1 = torch.nn.functional.binary_cross_entropy(p, y1)
loss_y0 = torch.nn.functional.binary_cross_entropy(p, y0)

print(round(p.item(), 3))
print(round(loss_y1.item(), 3))
print(round(loss_y0.item(), 3))

这个例子要看什么

你不用死记小数，只要记住方向：真类概率越接近 1，y=1 的 loss 越低；真类概率越接近 0，y=0 的 loss 越低。把这个样本看懂，batch 只是把同一规则复制多次。

小 batch 追踪

把一批样本走完

先用一个小 batch 看完整链路，比空讲公式更容易发现接口问题。设三个样本的标签分别是 y=[1,0,1]，模型给出的 logits 是 z=[2.0,-1.0,0.0]。你可以先不管参数怎么来的，只看这三个数在 sigmoid、阈值和 BCEWithLogitsLoss 里分别会变成什么。

样本 logit z sigmoid(z) threshold=0.5 直觉解释
样本 1 2.0 约 0.881 正类 分数很高，模型明显偏正类。
样本 2 -1.0 约 0.269 负类 分数偏低，模型更像在拒绝正类。
样本 3 0.0 0.5 边界 这里刚好卡在阈值上，最适合检查代码里到底用的是 > 还是 >=。

把这张表继续往后推

如果训练时用的是 BCEWithLogitsLoss，你不会先手工写出概率，而是直接把 logits 和 y.float() 交给损失。损失会在内部把 logits 转成概率，再计算每个样本的惩罚。也就是说，sigmoid 不是必须显式出现在训练图里，但它的作用并没有消失，只是被损失函数接手了。

为什么这个追踪有用

很多人把训练和评估混着写：训练时算概率、评估时又拿 logits 做阈值，或者写了两次 sigmoid。只要把这三个样本手算一遍，你就能马上看出哪一步写重了、哪一步少了一层。

把这个小 batch 写进训练脚本python
logits = torch.tensor([[2.0], [-1.0], [0.0]])
y = torch.tensor([[1.0], [0.0], [1.0]])

prob = torch.sigmoid(logits)
pred = (prob >= 0.5).long()

print(prob.squeeze().tolist())
print(pred.squeeze().tolist())

故障信号

如果样本 3 这种边界值在训练和评估里表现不一致，先查阈值比较符号；如果正负类都开始偏向同一边，先查是否重复 sigmoid 或者标签 shape 没对齐。这个小 batch 的作用，就是把“看起来差不多”变成“哪一行错了”。

标签对齐

shape / dtype / 位置

二分类最常见的另一类错误，不是数学，而是接口。输出是 [B, 1]，标签却是 [B]；输出是 logits，标签却已经是 one-hot；训练里忘了转成 float，最后把类别索引当成概率用。这些问题有时不会立即报错，但会把损失和梯度悄悄改坏。

项目 推荐写法 为什么这样更稳
输出形状 [B, 1] 和单个正类概率对齐，后面阈值也更直观。
标签形状 [B, 1] 避免广播把维度偷偷扩掉。
标签 dtype float BCE 系列损失期待的是 0/1 浮点值，不是类别索引。
评估时的比较 pred = (prob >= threshold) 把业务规则和训练目标分开，调阈值才不会动到 loss。
把 shape 检查写进训练前python
logits = model(x_batch)
y_batch = y_batch.float().view(-1, 1)

assert logits.shape == y_batch.shape, (logits.shape, y_batch.shape)
loss = criterion(logits, y_batch)

这里的检查顺序别反过来

先看输出和标签形状，再看 dtype，最后才看阈值和指标。如果你把顺序反过来，往往会在“精确率怎么这么差”这种表面症状上打转很久，真正的问题却只是 view 或 float() 少了一步。

训练与评估状态账本

把状态分开记

二分类里，训练和评估只差几行代码，但状态不一样。训练时要留下计算图，让 loss 能把梯度写回参数；评估时通常不需要梯度，只想看概率和阈值后的结果。model.eval() 和 torch.no_grad() 解决的是不同层的问题，别混成一个动作。

位置 训练时 评估时 写错会怎样
model.train() / model.eval() 保持训练态 切到评估态 Dropout、BatchNorm 之类的层行为不对
torch.no_grad() 通常不用 关闭梯度记录 显存白白增加，后面还可能把评估图接回训练
sigmoid + threshold 通常不放进 loss 用来产出可读的概率和类别 把决策规则塞进训练里，调试会很乱

这点和第 05 讲的关系

第 05 讲里你已经见过 optimizer.step() 和 zero_grad() 的位置。这里多出来的只是输出层的读法：训练时读 logits 和标签，评估时读概率和阈值。链路还是同一条，只是读写对象换了一点。

先别把评估写成训练

如果你在验证阶段还在记录梯度，或者把 sigmoid 写进损失之前又写进评估之后两次，就说明状态没有分开。先把训练态、评估态和决策规则各自放回自己的位置，再看指标会不会更稳。

错例实验室

把症状接回原因

下面这三类错最典型：一类是输出层放错位置，一类是标签 shape 没对齐，一类是阈值和评估状态分开没做好。别把它们都归成“模型没学会”，先看对象链路断在哪一段。

症状：loss 降得很慢，pred 常常挤在 0.5 附近。
先怀疑：forward 和 loss 两边都做了 sigmoid。
先查：模型末层和 criterion 的配合方式。

怎么把错例和前文接起来

重复 sigmoid 对应的是损失约定那一节；shape 不对对应的是标签对齐；阈值太高对应的是评估一节。只要你能把症状直接指回上面的某一节，就说明你不是在背模板，而是在按对象链路排查。

调试时先看哪一层

先看 forward 是否输出 logits，再看 loss 是否吃对输入，再看标签 dtype 和 shape，最后才看阈值和指标。顺序一乱，就会把本来很小的问题扩大成“训练完全没法用”。

把三个错例写进检查脚本python
cases = {
    "repeat_sigmoid": "forward 和 loss 两边都做了 sigmoid",
    "shape_mismatch": "标签是 [B]，输出是 [B,1]",
    "threshold_high": "threshold 比 0.5 更高",
}

for name, hint in cases.items():
    print(name, "->", hint)

这段实验的目的

不是让你多记三条口号，而是让你看到：二分类的错误大多都能回到同一条链路上。只要你知道自己在查 logits、概率、标签还是阈值，排错就会快很多。

阈值与评估

决策和指标分开看

概率不是最终动作，阈值才是。默认 0.5 只是常见起点，不是铁律。阈值调低，模型更愿意报正类，召回常会上去，误报也可能增多；阈值调高，模型更保守，精确率可能更高，漏报也可能更多。指标要和业务代价一起看，不要只盯 loss。

prob=0.80, threshold=0.50
pred = positive
这是最常见的正常分界。
调法 更常见的变化 适合谁先看
调低阈值 召回上升，误报也可能上升 先看漏报成本高的场景
调高阈值 精确率上升，漏报也可能上升 先看误报成本高的场景
只看 accuracy 类别不平衡时容易骗自己 谁都不该只看这一项

阈值扫一遍

同一组概率，换不同决策

同一批概率，阈值换一下，结果就会变。下面这组概率取 [0.12, 0.41, 0.55, 0.78]，真实标签取 [0, 0, 1, 1]。你可以先按 0.5 试一遍，再把阈值改成 0.3 和 0.7，看看正类数量、误报和漏报怎样变化。这个追踪比空讲“阈值可调”更能说明问题。

prob y 阈值 0.3 阈值 0.5 阈值 0.7
0.12 0 负类 负类 负类
0.41 0 正类 负类 负类
0.55 1 正类 正类 负类
0.78 1 正类 正类 正类

从这张表能看出来什么

阈值 0.3 会把 0.41 也判成正类，所以正类数量增加得更快；阈值 0.7 会把 0.55 压回负类，所以更保守。换句话说，阈值是在“漏报”和“误报”之间挪位置，不是在改模型本身。

为什么值得单独看一遍

很多人只知道阈值能调，却没试过同一组概率在不同阈值下怎么改判。这一页把概率和决策拆开后，最该练的就是这类扫表。等你回到真实任务里，才知道自己到底是在调模型，还是在调决策规则。

阈值扫一遍的最小脚本python
probs = torch.tensor([0.12, 0.41, 0.55, 0.78])
thresholds = [0.3, 0.5, 0.7]

for threshold in thresholds:
    pred = (probs >= threshold).long()
    print(threshold, pred.tolist())

这个练习的目标

不是把三个数字背下来，而是让你形成一个习惯：看到结果不对时，先问“阈值改了吗”，再问“模型输出变了吗”。只要这个顺序对了，很多二分类错误都能更快定位。

常见误区

先排错再调参
错法 先看到什么症状 改哪一层
把 logit 当概率 数值会小于 0 或大于 1，却还拿来做阈值判断 先 sigmoid，再做概率解释
重复 sigmoid 梯度变弱，loss 下降慢 forward 和 loss 只保留一处 sigmoid
标签 dtype / shape 不对 BCE 报错、广播失真、训练结果怪 把标签转成 float，并和输出对齐
只看 accuracy 类别不平衡时，看起来很高，实际没用 把 precision / recall 一起看

这里先不追

概率校准、ROC / PR 曲线、focal loss、多标签分类都属于后面的工作。当前只要把“logit → 概率 → 阈值 → 类别 → BCE 约定”看稳，这一讲就够用了。

练习
