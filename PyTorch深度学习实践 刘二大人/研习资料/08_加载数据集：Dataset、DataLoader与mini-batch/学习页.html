加载数据集：Dataset、DataLoader 与 mini-batch

这一页只处理一条数据链：原始表格或图片目录怎样被 Dataset 切成单样本，DataLoader 怎样把样本装成 batch，训练循环又怎样把 batch 送进模型、loss 和 optimizer。

先别追多进程读取、pin_memory、缓存和复杂的数据增强。先把样本、batch、最后一批和标签 dtype 看稳，后面换成图片、文本或更大的数据时，入口还是这条线。

课程地图 上一讲：07 多维特征输入 下一讲：09 多分类问题

读到这里先确认

  • 能说出 Dataset、DataLoader、batch 各自做什么。
  • 能算出一个 epoch 里大约有多少个 batch，并知道最后一批可能更小。
  • 能在第一批数据上检查 shape、dtype 和标签约定。

学习目标

先定边界

这页先解决什么

把一个 CSV 或图片目录送进训练循环之前，先把职责拆开：谁读单样本，谁拼 batch，谁负责更新参数。先把这个分工看清，后面读代码会轻很多。

先别追什么

num_workers、pin_memory、分布式采样、复杂增强先放一边。它们只是在这条入口线外面加层数，不改变这里的分工。

术语 先抓住什么 常见误解
Dataset 按索引返回一个样本 以为它负责打乱顺序或组 batch
DataLoader 把样本拼成 batch 并按设置迭代 以为它会自动修正特征和标签
epoch 完整看一遍训练集 和一次参数更新混在一起
batch size 每次迭代最多拿多少样本 以为最后一批也一定等于这个数

概念骨架

先看整条线

同一组锚点数据会在三层上换形状：一整张表、一个样本、一批样本。看 shape 的变化，比背 API 名字更快。

原始数据 [1000, 5]

一张表，4 列特征 + 1 列标签，还是未分 batch 的状态。

Dataset.__getitem__(i) x_i [4], y_i [1]

只负责按索引取出一个样本，不负责打乱顺序，也不负责拼 batch。

DataLoader(batch_size=32) x_batch [32, 4], y_batch [32, 1]

把单样本收成一批，交给训练循环。训练循环看到的就是这一层。

尾批 [8, 4] / [8, 1]

数据量不能整除 batch size 时，最后一批通常更小。写代码时要允许这个情况。

对象 before after 读什么 写什么 错了会怎样
Dataset.__len__ 原始数据还在磁盘或内存里 给出样本总数 样本数 迭代预算 len(loader) 和 epoch 计数会偏
Dataset.__getitem__(i) 只有一个索引 i 一个样本 (x_i, y_i) 原始存储、索引、切片 单样本特征和标签 特征和标签混在一起，dtype 也可能不对
DataLoader 一堆单样本 一个 batch batch_size、shuffle、drop_last x_batch、y_batch 顺序偏差、尾批 shape 变化或读码困难
训练 step 当前 batch 和旧参数 新参数 x_batch、y_batch、grad pred、loss、参数值 optimizer.step() 看起来在跑，参数其实没变

状态追踪

把入口看成流水线

这页里最值得记住的是“状态怎么变”。原始表格在 Dataset 里变成单样本，在 DataLoader 里变成 batch，最后在训练循环里变成预测、损失和参数更新。

先读一遍流程

如果你能说清每一步读什么、写什么、为什么要分开，这一讲就已经过半了。后面再看图片目录、文本列表或 CSV，只是换了输入来源，思路相同。

只看一件事

训练代码最怕“一个函数什么都干”。这里先把职责切开：Dataset 只管单样本，DataLoader 只管 batch，训练循环只管前向、反向和更新。

位置 输入 输出 读到的状态 写入的状态
__len__ 原始数据 样本总数 数据里有多少行 迭代范围
__getitem__ 一个索引 一个样本 原始数组、切片、转换规则 单样本特征和标签
DataLoader Dataset + 迭代设置 batch 迭代器 样本总数、batch size、shuffle、drop_last 当前 batch 的 shape
训练循环 batch 更新后的参数 预测、loss、grad 模型参数值

批次数与尾批

先算清楚再写代码

一个 epoch 里有多少个 batch，先看总样本数 N 和 batch size B。默认最后一批保留，所以 batch 数按向上取整；如果你显式写了 drop_last=True，才会把尾巴截掉。

$$ N_{\text{batches}} = \left\lceil \frac{N}{B} \right\rceil \qquad b_{\text{last}} = N - (N_{\text{batches}} - 1)B $$

N 是样本总数，B 是 batch size，b_last 是最后一批的实际大小。它不一定等于 B。

按样本数快速算 batch 数
N = 1000
B = 128
print((N + B - 1) // B)  # 8

drop_last=False

保留尾批，数据不会白白丢掉。代价是最后一批可能比前面小，代码里要接受 b != B 的情况。

drop_last=True

每批形状更整齐，适合那些写死 batch 维度的旧代码。代价是尾批会被丢掉，少量样本不会参与训练。

CSV 具体例子

concrete case

把这条链放进一个具体场景会更清楚。假设你手里有一个 diabetes.csv，一共 1000 行，前 4 列是特征，最后 1 列是标签。Dataset 负责把这一整张表切成一条一条样本；DataLoader 负责把样本按 batch_size 装起来，给训练循环一个稳定入口。

这就是为什么训练前要先看第一批数据。你只要把 Dataset.__getitem__(17) 和 next(iter(loader)) 打印出来，就能知道切片、标签列、batch 维度和尾批是否正常。这个动作很便宜，但它能提前挡住一半以上的入口错误。

位置 你会看到什么 这一步做什么 错了会出现什么
dataset[17] x_17 [4], y_17 [1] 取出第 17 条样本，保持特征和标签分离 如果标签没切开，x 维度会多一列，后面模型输入也会跟着偏
next(iter(loader)) x_batch [32, 4], y_batch [32, 1] 把 32 条样本拼成一批 如果 __getitem__ 返回了 Python 对象，拼批会先在这里报错
第一个 epoch 的最后一批 可能是 [8, 4], [8, 1] 把剩下的样本也送进训练 如果写死 view(32, -1)，最后一批会炸
criterion(pred, y_batch) 标量 loss 把当前 batch 的预测误差压成一个数 如果 y_batch 的 dtype 不对，loss 会在这里报类型或形状错误
先把单样本和 batch 打出来
sample_x, sample_y = dataset[17]
print(sample_x.shape, sample_y.shape)

x_batch, y_batch = next(iter(loader))
print(x_batch.shape, y_batch.shape)
print(x_batch.size(0), "real batch size")
这个检查很值

它能直接告诉你：样本切片有没有切对，batch 拼接有没有拼对，最后一批有没有按真实大小保留。很多人把错误留到 loss.backward() 之后才看，其实真正该看的地方在入口。

图片目录例子

same job, different source

如果数据源换成图片文件夹加一份标签表，职责分工还是不变。Dataset 负责读单个样本，这次单样本变成一张图片和它的标签；DataLoader 负责拼 batch，这时 batch 里的单样本形状从表格行变成 [C, H, W]。

文件类型会变，入口职责不会变。你真正要盯的是：单样本 shape 是什么，标签怎么对齐，batch 维度在哪一层加上去，训练循环是不是只接收 (x_batch, y_batch)。只要这几件事稳定，后面的 CNN、RNN 或分类头都还能沿着同一条入口往下接。

数据来源 Dataset 负责什么 单样本 shape DataLoader 负责什么
CSV 表格 切出特征列和标签列，转成张量 [4] 或 [1] 拼成 [B, 4]、[B, 1]
图片文件夹 读单张图片、做必要的 resize / normalize、对齐标签 [C, H, W] 拼成 [B, C, H, W]
文本列表 读单条文本、截断或清洗、对齐标签 token 序列 拼成 batch 序列
图片数据集只换读取动作，不换职责
class ImageDataset(Dataset):
    def __getitem__(self, index):
        image = self.load_one_image(index)   # 读单张图片
        label = self.labels[index]           # 对齐标签
        return image, label

loader = DataLoader(dataset, batch_size=16, shuffle=True)
这段骨架强调的是职责分工

图片、文本、表格换来换去，Dataset 这一层都在做“单样本准备”；DataLoader 这一层都在做“batch 组织”。如果你把两层混在一起，后面再加增强、缓存或多进程，只会更难改。

batch 到 loss 的接口

从入口接到训练目标

这一讲的入口包含两件事：把 batch 拼出来，把 batch 送到正确的 loss 上。DataLoader 不关心你最后用 BCEWithLogitsLoss 还是 CrossEntropyLoss；它只保证 batch 按设置被吐出来。真正决定标签 shape 和 dtype 的，是你后面那一层目标函数。

所以训练前的检查顺序应该是：先看 x_batch，再看 y_batch，然后看 pred 和 criterion。如果顺序反了，很多错误会被推迟到 loss.backward() 或者更后面才冒出来，看起来像优化器的问题，实际上是入口接口没对齐。

任务类型 批次里标签长什么样 loss 期待什么 错了会怎样
二分类 / 多标签 y_batch [B, 1] 或 [B, C] float 0/1 值 标签 dtype 不对，loss 先报错
多分类 y_batch [B] long 类别索引 one-hot 被当成类别索引，shape 或类型会错
回归 y_batch [B, 1] 或 [B] float 目标值 把类别标签硬塞进回归，会让目标语义失真
两种常见 loss 的入口差异
# 二分类 / 多标签：标签通常是 float
pred = model(x_batch)                # [B, 1]
loss = torch.nn.BCEWithLogitsLoss()(pred, y_batch.float())

# 多分类：标签通常是 long 类别索引
pred = model(x_batch)                # [B, C]
loss = torch.nn.CrossEntropyLoss()(pred, y_batch.long())
看懂这段，就知道入口错在哪一层

如果 x_batch 的 shape 先错，问题在 Dataset；如果 batch 维正常但 y_batch 的 dtype 不对，问题多半还在 Dataset；如果两者都对却报 loss 错，才去看模型输出和损失函数的约定。

症状 先怀疑什么 下一步检查
mat1 and mat2 shapes cannot be multiplied 特征列切多了或切少了 打印一个样本的 x.shape
expected scalar type Long but found Float CrossEntropyLoss 的标签 dtype 不对 看 y_batch.dtype，必要时在 Dataset 里改
最后一批报 shape 错 代码写死 batch size 改成 x_batch.size(0) 或显式 drop_last=True

这里真正的习惯是：先检查入口，再进入 loss。这样你看到的每个报错，都能回到一个具体位置，避免在训练循环里盲目换参数。

训练、验证、测试三套 loader

同一份 Dataset，三种配置

同一个 Dataset 往往会被三个 DataLoader 共用。训练集、验证集、测试集的差别主要在 loader 配置上：训练时要打乱顺序，验证和测试通常按原顺序读；训练允许尾批参与更新，验证和测试则常按任务需要选择是否保留尾批。

数据拆分 常见配置 为什么这样设 错了会看到什么
训练集 shuffle=True, drop_last=False 打散样本顺序，尾批也参与更新 顺序偏差、过拟合到数据排列或少掉最后几条样本
验证集 shuffle=False, drop_last=False 只看真实评估分布，不改变样本顺序 如果乱序，会让你更难对照样本和预测
测试集 shuffle=False, drop_last=False 保存原始顺序，便于复查样本级结果 如果把测试集打乱，复盘单样本会更麻烦
同一份 Dataset 的三种 loader
train_loader = DataLoader(train_dataset, batch_size=32, shuffle=True, drop_last=False)
val_loader = DataLoader(val_dataset, batch_size=64, shuffle=False, drop_last=False)
test_loader = DataLoader(test_dataset, batch_size=64, shuffle=False, drop_last=False)
这段代码能帮你分清“数据拆分”和“读取方式”

训练 / 验证 / 测试先在数据层面拆开，再分别给不同的 loader 配置。别把“要不要 shuffle”当成全局开关，它是每个 loader 自己的读取策略。

保留 batch 维，再做展平

shape safety

这一点经常在图片任务里冒出来。假设一批图片是 [8, 3, 28, 28]，你想把它喂给全连接层，就得先展平成 [8, 2352]。这里的 8 只是这一次 loader 真正吐出来的 batch 大小。

所以代码里更稳的写法是先读 x_batch.size(0)，再把其余维度压平。只要最后一批变成 [5, 3, 28, 28]，这种写法还是能继续跑；如果你写死 view(32, -1)，尾批就会直接炸。这里的关键是训练入口要接受当前 batch 的真实大小。你把这一层处理好，后面接 nn.Linear、CrossEntropyLoss 或别的分类头，形状才不会在中途突然变窄。

展平时保留当前 batch 大小
for x_batch, y_batch in loader:
    flat = x_batch.reshape(x_batch.size(0), -1)
    pred = classifier(flat)
写法 适合场景 尾批会怎样 下一步
view(32, -1) 你真的确认 batch 永远是 32 最后一批会报 shape 错 只在 drop_last=True 且业务允许时考虑
reshape(x.size(0), -1) batch 大小会变 自动跟着当前 batch 走 更适合训练入口
x.size(0) 任何需要读取当前 batch 数的地方 不会把尾批当成固定长度 先在训练前打印一次，确认入口一致
先保住这一步，后面就轻很多

很多看起来像“模型坏了”的报错，其实都是这里写死 batch 长度造成的。你只要在入口处接受动态 batch，后面的 loss、更新和评估都能更稳定地接上。

排错顺序

先查哪里

如果训练脚本第一眼就不对，先看数据入口，再看优化器。数据入口错了，后面所有 loss 和梯度都只是替错误数据服务。这个顺序感很重要，因为 Dataset / DataLoader 的问题通常在很前面就能看出来。

  1. 先看 __len__：样本总数对不对，len(loader) 会不会跟着偏。
  2. 再看 __getitem__：单样本是不是被切对，标签有没有混进特征。
  3. 再看第一批 shape：x_batch 和 y_batch 的维度是不是跟 loss 约定一致。
  4. 再看 dtype：BCE 和 CrossEntropyLoss 要求的标签形式不一样。
  5. 最后才看 shuffle、drop_last 和手写代码是否支持尾批。
先查什么 你要看什么 典型症状 下一步
__len__ 是不是样本数 epoch 计数不对 把长度打印出来
__getitem__ 特征和标签是否分开 输入维度多一列或少一列 打印一个具体索引的返回值
第一批 shape batch 维是否加对 loss 接口报 shape mismatch 看 x_batch.size(0)
dtype 标签类型是否和 loss 一致 BCE / CE 报类型或 target 错误 改数据切片时就统一类型

代码与例子

逐行看职责

下面三段代码都围绕同一组锚点数据。第一段把原始数组切成样本，第二段把样本装进 loader，第三段先看第一批数据，确认 shape 和 dtype。

自定义 Dataset 骨架
import torch
from torch.utils.data import Dataset, DataLoader

class DiabetesDataset(Dataset):
    def __init__(self, data):
        self.x = torch.tensor(data[:, :-1], dtype=torch.float32)
        self.y = torch.tensor(data[:, [-1]], dtype=torch.float32)

    def __len__(self):
        return len(self.x)

    def __getitem__(self, index):
        return self.x[index], self.y[index]
这段代码只做三件事

存原始数据、按索引取一条、把特征和标签分开。标签先保持为单列张量，后面用什么 loss 再决定要不要改 shape。

DataLoader 接入训练循环
dataset = DiabetesDataset(data)
loader = DataLoader(dataset, batch_size=32, shuffle=True, drop_last=False)

for epoch in range(2):
    for x_batch, y_batch in loader:
        pred = model(x_batch)
        loss = criterion(pred, y_batch)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()
训练循环里，DataLoader 只提供 batch

shuffle=True 只影响顺序，drop_last=False 只影响尾批是否保留。真正改参数的是 optimizer.step()。

第一批数据检查
x_batch, y_batch = next(iter(loader))
print(x_batch.shape, y_batch.shape)
print(x_batch.dtype, y_batch.dtype)

这个检查最好在训练前就做。第一批能过，至少说明 Dataset 的切片、DataLoader 的拼批和标签 dtype 没把最基础的边界弄坏。

误区诊断

错因 / 后果 / 修法

DataLoader 本身不修正语义，很多报错在进入模型之前就埋好了。下面几种症状最值得先看。

TypeError: object of type 'DiabetesDataset' has no len()
检查：`__len__` 是否返回样本数。
症状 先查哪里 修法
