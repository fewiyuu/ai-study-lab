08｜加载数据集：Dataset、DataLoader 与 mini-batch

这一页只处理一条数据链：原始表格或图片目录怎样被 Dataset 切成单样本，DataLoader 怎样把样本装成 batch，训练循环又怎样把 batch 送进模型、loss 和 optimizer。

学习目标

先定边界

这页先解决什么

把一个 CSV 或图片目录送进训练循环之前，先把职责拆开：谁读单样本，谁拼 batch，谁负责更新参数。先把这个分工看清，后面读代码会轻很多。

先别追什么

num_workers、pin_memory、分布式采样、复杂增强先放一边。它们只是在这条入口线外面加层数，不改变这里的分工。

术语先抓住什么常见误解
Dataset按索引返回一个样本以为它负责打乱顺序或组 batch
DataLoader把样本拼成 batch 并按设置迭代以为它会自动修正特征和标签
epoch完整看一遍训练集和一次参数更新混在一起
batch size每次迭代最多拿多少样本以为最后一批也一定等于这个数

概念骨架

先看整条线

同一组锚点数据会在三层上换形状：一整张表、一个样本、一批样本。看 shape 的变化，比背 API 名字更快。

状态追踪

把入口看成流水线

这页里最值得记住的是“状态怎么变”。原始表格在 Dataset 里变成单样本，在 DataLoader 里变成 batch，最后在训练循环里变成预测、损失和参数更新。

只看一件事

训练代码最怕“一个函数什么都干”。这里先把职责切开：Dataset 只管单样本，DataLoader 只管 batch，训练循环只管前向、反向和更新。

位置输入输出读到的状态写入的状态
__len__原始数据样本总数数据里有多少行迭代范围
__getitem__一个索引一个样本原始数组、切片、转换规则单样本特征和标签
DataLoaderDataset + 迭代设置batch 迭代器样本总数、batch size、shuffle、drop_last当前 batch 的 shape
训练循环batch更新后的参数预测、loss、grad模型参数值

批次数与尾批

先算清楚再写代码

一个 epoch 里有多少个 batch，先看总样本数 N 和 batch size B。默认最后一批保留，所以 batch 数按向上取整；如果你显式写了 drop_last=True，才会把尾巴截掉。

N 是样本总数，B 是 batch size，b_last 是最后一批的实际大小。它不一定等于 B。

代码
N = 1000
B = 128
print((N + B - 1) // B) # 8

drop_last=True

每批形状更整齐，适合那些写死 batch 维度的旧代码。代价是尾批会被丢掉，少量样本不会参与训练。

CSV 具体例子

concrete

把这条链放进一个具体场景会更清楚。假设你手里有一个 diabetes.csv，一共 1000 行，前 4 列是特征，最后 1 列是标签。Dataset 负责把这一整张表切成一条一条样本；DataLoader 负责把样本按 batch_size 装起来，给训练循环一个稳定入口。

位置你会看到什么这一步做什么错了会出现什么
dataset[17]x_17 [4], y_17 [1]取出第 17 条样本，保持特征和标签分离如果标签没切开，x 维度会多一列，后面模型输入也会跟着偏
next(iter(loader))x_batch [32, 4], y_batch [32, 1]把 32 条样本拼成一批如果 __getitem__ 返回了 Python 对象，拼批会先在这里报错
第一个 epoch 的最后一批可能是 [8, 4], [8, 1]把剩下的样本也送进训练如果写死 view(32, -1)，最后一批会炸
criterion(pred, y_batch)标量 loss把当前 batch 的预测误差压成一个数如果 y_batch 的 dtype 不对，loss 会在这里报类型或形状错误
代码
sample_x, sample_y = dataset[17]
print(sample_x.shape, sample_y.shape)

x_batch, y_batch = next(iter(loader))
print(x_batch.shape, y_batch.shape)
print(x_batch.size(0), "real batch size")

> 这个检查很值 它能直接告诉你：样本切片有没有切对，batch 拼接有没有拼对，最后一批有没有按真实大小保留。很多人把错误留到 loss.backward() 之后才看，其实真正该看的地方在入口。

图片目录例子

same

如果数据源换成图片文件夹加一份标签表，职责分工还是不变。Dataset 负责读单个样本，这次单样本变成一张图片和它的标签；DataLoader 负责拼 batch，这时 batch 里的单样本形状从表格行变成 [C, H, W]。

数据来源Dataset 负责什么单样本 shapeDataLoader 负责什么
CSV 表格切出特征列和标签列，转成张量[4] 或 [1]拼成 [B, 4]、[B, 1]
图片文件夹读单张图片、做必要的 resize / normalize、对齐标签[C, H, W]拼成 [B, C, H, W]
文本列表读单条文本、截断或清洗、对齐标签token 序列拼成 batch 序列
代码
class ImageDataset(Dataset):
 def __getitem__(self, index):
 image = self.load_one_image(index) # 读单张图片
 label = self.labels[index] # 对齐标签
 return image, label

loader = DataLoader(dataset, batch_size=16, shuffle=True)

> 这段骨架强调的是职责分工 图片、文本、表格换来换去，Dataset 这一层都在做“单样本准备”；DataLoader 这一层都在做“batch 组织”。如果你把两层混在一起，后面再加增强、缓存或多进程，只会更难改。

batch 到 loss 的接口

从入口接到训练目标

这一讲的入口包含两件事：把 batch 拼出来，把 batch 送到正确的 loss 上。DataLoader 不关心你最后用 BCEWithLogitsLoss 还是 CrossEntropyLoss；它只保证 batch 按设置被吐出来。真正决定标签 shape 和 dtype 的，是你后面那一层目标函数。

任务类型批次里标签长什么样loss 期待什么错了会怎样
二分类 / 多标签y_batch [B, 1] 或 [B, C]float 0/1 值标签 dtype 不对，loss 先报错
多分类y_batch [B]long 类别索引one-hot 被当成类别索引，shape 或类型会错
回归y_batch [B, 1] 或 [B]float 目标值把类别标签硬塞进回归，会让目标语义失真
代码
# 二分类 / 多标签：标签通常是 float
pred = model(x_batch) # [B, 1]
loss = torch.nn.BCEWithLogitsLoss()(pred, y_batch.float())

# 多分类：标签通常是 long 类别索引
pred = model(x_batch) # [B, C]
loss = torch.nn.CrossEntropyLoss()(pred, y_batch.long())

> 看懂这段，就知道入口错在哪一层 如果 x_batch 的 shape 先错，问题在 Dataset；如果 batch 维正常但 y_batch 的 dtype 不对，问题多半还在 Dataset；如果两者都对却报 loss 错，才去看模型输出和损失函数的约定。

症状先怀疑什么下一步检查
mat1 and mat2 shapes cannot be multiplied特征列切多了或切少了打印一个样本的 x.shape
expected scalar type Long but found FloatCrossEntropyLoss 的标签 dtype 不对看 y_batch.dtype，必要时在 Dataset 里改
最后一批报 shape 错代码写死 batch size改成 x_batch.size(0) 或显式 drop_last=True

训练、验证、测试三套 loader

同一份

同一个 Dataset 往往会被三个 DataLoader 共用。训练集、验证集、测试集的差别主要在 loader 配置上：训练时要打乱顺序，验证和测试通常按原顺序读；训练允许尾批参与更新，验证和测试则常按任务需要选择是否保留尾批。

代码
train_loader = DataLoader(train_dataset, batch_size=32, shuffle=True, drop_last=False)
val_loader = DataLoader(val_dataset, batch_size=64, shuffle=False, drop_last=False)
test_loader = DataLoader(test_dataset, batch_size=64, shuffle=False, drop_last=False)

> 这段代码能帮你分清“数据拆分”和“读取方式” 训练 / 验证 / 测试先在数据层面拆开，再分别给不同的 loader 配置。别把“要不要 shuffle”当成全局开关，它是每个 loader 自己的读取策略。

保留 batch 维，再做展平

shape

这一点经常在图片任务里冒出来。假设一批图片是 [8, 3, 28, 28]，你想把它喂给全连接层，就得先展平成 [8, 2352]。这里的 8 只是这一次 loader 真正吐出来的 batch 大小。

展平时保留当前 batch 大小复制

写法适合场景尾批会怎样下一步
view(32, -1)你真的确认 batch 永远是 32最后一批会报 shape 错只在 drop_last=True 且业务允许时考虑
reshape(x.size(0), -1)batch 大小会变自动跟着当前 batch 走更适合训练入口
x.size(0)任何需要读取当前 batch 数的地方不会把尾批当成固定长度先在训练前打印一次，确认入口一致

> 先保住这一步，后面就轻很多 很多看起来像“模型坏了”的报错，其实都是这里写死 batch 长度造成的。你只要在入口处接受动态 batch，后面的 loss、更新和评估都能更稳定地接上。

排错顺序

先查哪里

如果训练脚本第一眼就不对，先看数据入口，再看优化器。数据入口错了，后面所有 loss 和梯度都只是替错误数据服务。这个顺序感很重要，因为 Dataset / DataLoader 的问题通常在很前面就能看出来。

先查什么你要看什么典型症状下一步
__len__是不是样本数epoch 计数不对把长度打印出来
__getitem__特征和标签是否分开输入维度多一列或少一列打印一个具体索引的返回值
第一批 shapebatch 维是否加对loss 接口报 shape mismatch看 x_batch.size(0)
dtype标签类型是否和 loss 一致BCE / CE 报类型或 target 错误改数据切片时就统一类型

代码与例子

逐行看职责

下面三段代码都围绕同一组锚点数据。第一段把原始数组切成样本，第二段把样本装进 loader，第三段先看第一批数据，确认 shape 和 dtype。

代码
import torch
from torch.utils.data import Dataset, DataLoader

class DiabetesDataset(Dataset):
 def __init__(self, data):
 self.x = torch.tensor(data[:, :-1], dtype=torch.float32)
 self.y = torch.tensor(data[:, [-1]], dtype=torch.float32)

 def __len__(self):
 return len(self.x)

 def __getitem__(self, index):
 return self.x[index], self.y[index]

> 这段代码只做三件事 存原始数据、按索引取一条、把特征和标签分开。标签先保持为单列张量，后面用什么 loss 再决定要不要改 shape。

DataLoader 接入训练循环复制

> 训练循环里，DataLoader 只提供 batch shuffle=True 只影响顺序，drop_last=False 只影响尾批是否保留。真正改参数的是 optimizer.step()。

代码
x_batch, y_batch = next(iter(loader))
print(x_batch.shape, y_batch.shape)
print(x_batch.dtype, y_batch.dtype)

这个检查最好在训练前就做。第一批能过，至少说明 Dataset 的切片、DataLoader 的拼批和标签 dtype 没把最基础的边界弄坏。
