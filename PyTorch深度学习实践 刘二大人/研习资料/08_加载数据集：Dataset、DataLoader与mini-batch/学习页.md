# 批次数与尾批 {#mechanism}
<!-- tag: 先算清楚再写代码 -->

一个 epoch 里有多少个 batch，先看总样本数 `N` 和 batch size `B`。默认最后一批保留，所以 batch 数按向上取整；如果你显式写了 `drop_last=True`，才会把尾巴截掉。

$$
 N_{\text{batches}} = \left\lceil \frac{N}{B} \right\rceil
 \qquad
 b_{\text{last}} = N - (N_{\text{batches}} - 1)B
 $$
 `N` 是样本总数，`B` 是 batch size，`b_last` 是最后一批的实际大小。它不一定等于 `B`。

按样本数快速算 batch 数复制

```python
N = 1000
B = 128
print((N + B - 1) // B) # 8
```

## box soft
### `drop_last=False`
保留尾批，数据不会白白丢掉。代价是最后一批可能比前面小，代码里要接受 `b != B` 的情况。

## box warn
### `drop_last=True`
每批形状更整齐，适合那些写死 batch 维度的旧代码。代价是尾批会被丢掉，少量样本不会参与训练。

# CSV 具体例子 {#case-study}
<!-- tag: concrete case -->

把这条链放进一个具体场景会更清楚。假设你手里有一个 `diabetes.csv`，一共 1000 行，前 4 列是特征，最后 1 列是标签。`Dataset` 负责把这一整张表切成一条一条样本；`DataLoader` 负责把样本按 `batch_size` 装起来，给训练循环一个稳定入口。

这就是为什么训练前要先看第一批数据。你只要把 `Dataset.__getitem__(17)` 和 `next(iter(loader))` 打印出来，就能知道切片、标签列、batch 维度和尾批是否正常。这个动作很便宜，但它能提前挡住一半以上的入口错误。

# 图片目录例子 {#image-case}
<!-- tag: same job, different source -->

如果数据源换成图片文件夹加一份标签表，职责分工还是不变。`Dataset` 负责读单个样本，这次单样本变成一张图片和它的标签；`DataLoader` 负责拼 batch，这时 batch 里的单样本形状从表格行变成 `[C, H, W]`。

文件类型会变，入口职责不会变。你真正要盯的是：单样本 shape 是什么，标签怎么对齐，batch 维度在哪一层加上去，训练循环是不是只接收 `(x_batch, y_batch)`。只要这几件事稳定，后面的 CNN、RNN 或分类头都还能沿着同一条入口往下接。

数据来源
Dataset 负责什么
单样本 shape
DataLoader 负责什么


CSV 表格
切出特征列和标签列，转成张量
`[4]` 或 `[1]`
拼成 `[B, 4]`、`[B, 1]`


图片文件夹
读单张图片、做必要的 resize / normalize、对齐标签
`[C, H, W]`
拼成 `[B, C, H, W]`


文本列表
读单条文本、截断或清洗、对齐标签
token 序列
拼成 batch 序列

图片数据集只换读取动作，不换职责复制

```python
class ImageDataset(Dataset):
 def __getitem__(self, index):
 image = self.load_one_image(index) # 读单张图片
 label = self.labels[index] # 对齐标签
 return image, label

loader = DataLoader(dataset, batch_size=16, shuffle=True)
```

> **这段骨架强调的是职责分工** 图片、文本、表格换来换去，Dataset 这一层都在做“单样本准备”；DataLoader 这一层都在做“batch 组织”。如果你把两层混在一起，后面再加增强、缓存或多进程，只会更难改。

# batch 到 loss 的接口 {#loss-bridge}
<!-- tag: 从入口接到训练目标 -->

这一讲的入口包含两件事：把 batch 拼出来，把 batch 送到正确的 loss 上。`DataLoader` 不关心你最后用 `BCEWithLogitsLoss` 还是 `CrossEntropyLoss`；它只保证 batch 按设置被吐出来。真正决定标签 shape 和 dtype 的，是你后面那一层目标函数。

所以训练前的检查顺序应该是：先看 `x_batch`，再看 `y_batch`，然后看 `pred` 和 `criterion`。如果顺序反了，很多错误会被推迟到 `loss.backward()` 或者更后面才冒出来，看起来像优化器的问题，实际上是入口接口没对齐。

任务类型
批次里标签长什么样
loss 期待什么
错了会怎样


二分类 / 多标签
`y_batch [B, 1]` 或 `[B, C]`
float 0/1 值
标签 dtype 不对，loss 先报错


多分类
`y_batch [B]`
long 类别索引
one-hot 被当成类别索引，shape 或类型会错


回归
`y_batch [B, 1]` 或 `[B]`
float 目标值
把类别标签硬塞进回归，会让目标语义失真

两种常见 loss 的入口差异复制

```python
# 二分类 / 多标签：标签通常是 float
pred = model(x_batch) # [B, 1]
loss = torch.nn.BCEWithLogitsLoss()(pred, y_batch.float())

# 多分类：标签通常是 long 类别索引
pred = model(x_batch) # [B, C]
loss = torch.nn.CrossEntropyLoss()(pred, y_batch.long())
```

> **看懂这段，就知道入口错在哪一层** 如果 x_batch 的 shape 先错，问题在 Dataset；如果 batch 维正常但 y_batch 的 dtype 不对，问题多半还在 Dataset；如果两者都对却报 loss 错，才去看模型输出和损失函数的约定。

症状
先怀疑什么
下一步检查


`mat1 and mat2 shapes cannot be multiplied`
特征列切多了或切少了
打印一个样本的 `x.shape`


`expected scalar type Long but found Float`
`CrossEntropyLoss` 的标签 dtype 不对
看 `y_batch.dtype`，必要时在 `Dataset` 里改


最后一批报 shape 错
代码写死 batch size
改成 `x_batch.size(0)` 或显式 `drop_last=True`

这里真正的习惯是：先检查入口，再进入 loss。这样你看到的每个报错，都能回到一个具体位置，避免在训练循环里盲目换参数。

# 训练、验证、测试三套 loader {#split-config}
<!-- tag: 同一份 Dataset，三种配置 -->

同一个 `Dataset` 往往会被三个 `DataLoader` 共用。训练集、验证集、测试集的差别主要在 loader 配置上：训练时要打乱顺序，验证和测试通常按原顺序读；训练允许尾批参与更新，验证和测试则常按任务需要选择是否保留尾批。

数据拆分
常见配置
为什么这样设
错了会看到什么


训练集
`shuffle=True`, `drop_last=False`
打散样本顺序，尾批也参与更新
顺序偏差、过拟合到数据排列或少掉最后几条样本


验证集
`shuffle=False`, `drop_last=False`
只看真实评估分布，不改变样本顺序
如果乱序，会让你更难对照样本和预测


测试集
`shuffle=False`, `drop_last=False`
保存原始顺序，便于复查样本级结果
如果把测试集打乱，复盘单样本会更麻烦

同一份 Dataset 的三种 loader复制

```python
train_loader = DataLoader(train_dataset, batch_size=32, shuffle=True, drop_last=False)
val_loader = DataLoader(val_dataset, batch_size=64, shuffle=False, drop_last=False)
test_loader = DataLoader(test_dataset, batch_size=64, shuffle=False, drop_last=False)
```

> **这段代码能帮你分清“数据拆分”和“读取方式”** 训练 / 验证 / 测试先在数据层面拆开，再分别给不同的 loader 配置。别把“要不要 shuffle”当成全局开关，它是每个 loader 自己的读取策略。

# 保留 batch 维，再做展平 {#shape-case}
<!-- tag: shape safety -->

这一点经常在图片任务里冒出来。假设一批图片是 `[8, 3, 28, 28]`，你想把它喂给全连接层，就得先展平成 `[8, 2352]`。这里的 `8` 只是这一次 loader 真正吐出来的 batch 大小。

所以代码里更稳的写法是先读 `x_batch.size(0)`，再把其余维度压平。只要最后一批变成 `[5, 3, 28, 28]`，这种写法还是能继续跑；如果你写死 `view(32, -1)`，尾批就会直接炸。这里的关键是训练入口要接受当前 batch 的真实大小。你把这一层处理好，后面接 `nn.Linear`、`CrossEntropyLoss` 或别的分类头，形状才不会在中途突然变窄。

展平时保留当前 batch 大小复制

```python
for x_batch, y_batch in loader:
 flat = x_batch.reshape(x_batch.size(0), -1)
 pred = classifier(flat)
```

写法
适合场景
尾批会怎样
下一步


`view(32, -1)`
你真的确认 batch 永远是 32
最后一批会报 shape 错
只在 `drop_last=True` 且业务允许时考虑


`reshape(x.size(0), -1)`
batch 大小会变
自动跟着当前 batch 走
更适合训练入口


`x.size(0)`
任何需要读取当前 batch 数的地方
不会把尾批当成固定长度
先在训练前打印一次，确认入口一致

> **先保住这一步，后面就轻很多** 很多看起来像“模型坏了”的报错，其实都是这里写死 batch 长度造成的。你只要在入口处接受动态 batch，后面的 loss、更新和评估都能更稳定地接上。

# 排错顺序 {#debug-order}
<!-- tag: 先查哪里 -->

如果训练脚本第一眼就不对，先看数据入口，再看优化器。数据入口错了，后面所有 loss 和梯度都只是替错误数据服务。这个顺序感很重要，因为 Dataset / DataLoader 的问题通常在很前面就能看出来。

1. 先看 `__len__`：样本总数对不对，`len(loader)` 会不会跟着偏。
2. 再看 `__getitem__`：单样本是不是被切对，标签有没有混进特征。
3. 再看第一批 `shape`：`x_batch` 和 `y_batch` 的维度是不是跟 loss 约定一致。
4. 再看 `dtype`：`BCE` 和 `CrossEntropyLoss` 要求的标签形式不一样。
5. 最后才看 `shuffle`、`drop_last` 和手写代码是否支持尾批。

先查什么
你要看什么
典型症状
下一步


`__len__`
是不是样本数
epoch 计数不对
把长度打印出来


`__getitem__`
特征和标签是否分开
输入维度多一列或少一列
打印一个具体索引的返回值


第一批 `shape`
batch 维是否加对
loss 接口报 shape mismatch
看 `x_batch.size(0)`


`dtype`
标签类型是否和 loss 一致
BCE / CE 报类型或 target 错误
改数据切片时就统一类型

# 代码与例子 {#examples}
<!-- tag: 逐行看职责 -->

下面三段代码都围绕同一组锚点数据。第一段把原始数组切成样本，第二段把样本装进 loader，第三段先看第一批数据，确认 shape 和 dtype。

自定义 Dataset 骨架复制

```python
import torch
from torch.utils.data import Dataset, DataLoader

class DiabetesDataset(Dataset):
 def __init__(self, data):
 self.x = torch.tensor(data[:, :-1], dtype=torch.float32)
 self.y = torch.tensor(data[:, [-1]], dtype=torch.float32)

 def __len__(self):
 return len(self.x)

 def __getitem__(self, index):
 return self.x[index], self.y[index]
```

> **这段代码只做三件事** 存原始数据、按索引取一条、把特征和标签分开。标签先保持为单列张量，后面用什么 loss 再决定要不要改 shape。

DataLoader 接入训练循环复制

```python
dataset = DiabetesDataset(data)
loader = DataLoader(dataset, batch_size=32, shuffle=True, drop_last=False)

for epoch in range(2):
 for x_batch, y_batch in loader:
 pred = model(x_batch)
 loss = criterion(pred, y_batch)
 optimizer.zero_grad()
 loss.backward()
 optimizer.step()
```

> **训练循环里，DataLoader 只提供 batch** shuffle=True 只影响顺序，drop_last=False 只影响尾批是否保留。真正改参数的是 optimizer.step()。

第一批数据检查复制

```python
x_batch, y_batch = next(iter(loader))
print(x_batch.shape, y_batch.shape)
print(x_batch.dtype, y_batch.dtype)
```

这个检查最好在训练前就做。第一批能过，至少说明 `Dataset` 的切片、`DataLoader` 的拼批和标签 dtype 没把最基础的边界弄坏。

### 1_dataset_return choice

**Q:** 1. `Dataset.__getitem__` 通常返回什么？

- [ ] 一个样本及其标签
- [ ] 一个完整 epoch 的统计
- [ ] 一次参数更新结果

Answer: 一个样本及其标签
Explain: Dataset 负责按索引返回一个样本；DataLoader 才负责把样本拼成 batch。

### 3_batch_size fill

**Q:** 3. `DataLoader(dataset, batch_size=32)` 表示每次迭代最多返回多少个样本？

Answer: 32
Explain: batch_size 是每次迭代最多取的样本数，最后一批可能更少。

### 5_len_dataset fill

**Q:** 5. `len(dataset)` 在这里通常返回什么？

### 6_shuffle_train choice

**Q:** 6. 为什么训练集常设 `shuffle=True`？

- [ ] 让标签自动变成 long
- [ ] 减少样本顺序带来的偏差
- [ ] 让最后一批一定等于 batch_size

Answer: 减少样本顺序带来的偏差
Explain: 打乱顺序可以减少样本顺序带来的偏差；测试集通常不需要。

**Q:** 7. 1000 个样本，`batch size` 设为 128，大约有多少个 batch？

Answer: 8
Explain: 1000 / 128 ≈ 7.8，默认要向上取整。

### 9_dataset_tasks diagnosis

**Q:** 9. 下面哪些动作更应该放在 `Dataset.__getitem__` 里？

- [ ] 读取原始 CSV 或图片文件
- [ ] 切出特征列和标签列
- [ ] 转成模型需要的 tensor 和 dtype
- [ ] 把样本顺序打乱
- [ ] 组 batch

Explain: 更应该放进 Dataset.__getitem__ 的是“读单样本”和“整理单样本”的工作：读取原始文件、切特征和标签、转换 dtype。 shuffle 和组 batch 属于 DataLoader 。

### 10_tail_batch synthesis

**Q:** 10. 最后一批只有 8 个样本，代码写死 `view(32, -1)` 时会怎样改？

Explain: 要点至少要有三句：第一，最后一批可能小于 32，所以写死 view(32, -1) 会把尾批当成固定形状处理。第二，改成 x.size(0) 读取当前 batch 的真实大小，或者在允许丢尾巴时用 drop_last=True 。第三，如果一定要固定 batch，就要说明这是业务取舍，不是 DataLoader 自动保证的。

### 11_ce_dtype fill

**Q:** 11. `CrossEntropyLoss` 的标签通常是什么 `dtype` / 形式？

Answer: long
Explain: CrossEntropyLoss 通常吃的是 long 类别索引，不是 one-hot。

**Q:** 12. `BCEWithLogitsLoss` 的标签通常是 float 的 0/1 值。

- [ ] 对
- [ ] 错

Answer: true

### 13_same_loop choice

**Q:** 13. 为什么 MNIST 和自定义 CSV 最后都能进入同一种训练循环？

- [ ] 因为模型会自动识别文件格式
- [ ] 因为它们最后都变成同一种 batch 入口
- [ ] 因为 DataLoader 会把标签都改成 one-hot

Answer: 因为它们最后都变成同一种 batch 入口
Explain: 它们最后都会变成 DataLoader 产出的 `(x_batch, y_batch)`。训练循环只关心这一层。

### 14_feynman synthesis

**Q:** 14. 用 5-7 句话说明：这页里哪些东西会变，哪些东西不会变？

Explain: 好的回答至少要说到：不管是 MNIST 还是 CSV，最后都要变成 (x_batch, y_batch) ；训练循环本身只关心 batch、shape、dtype 和更新顺序；真正会变的是 Dataset.__getitem__ 里的读法、切片和转换； DataLoader 只负责拼批和迭代；如果标签约定不同，差异会先在 loss 入口暴露出来。
