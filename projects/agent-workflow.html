智能工作流引擎

可运行、可恢复、可观测的 Agent 工作流系统实战项目。

需要 Python · 状态机 · 异步编程 · LLM API

项目目标

GOALS

核心目标：实现一个 Agent 工作流引擎，能接收任务描述、规划步骤、调用工具、处理异常并恢复。

为什么做这个：Agent 面试最常问的是"怎么保证可靠执行"。一个能跑通的工作流引擎是最好的回答。

预期产出：可运行的 CLI 工具 + 可视化 Trace + 项目文档。

前置知识

PREREQUISITES
  • Agent 基础与工程（ReAct 循环、架构模式）
  • Context Engineering（状态、压缩、隔离）
  • Harness Engineering（运行外壳、环境设计）

实施步骤

STEPS

MVP 阶段（1-2 周）

  • 实现任务解析器：把自然语言任务拆成步骤列表
  • 实现工具注册表：Python 函数作为工具，JSON Schema 描述
  • 实现 ReAct 循环：观察 → 思考 → 行动 → 观察
  • 实现 CLI 界面：输入任务、查看执行过程和结果

增强阶段（2-3 周）

  • 添加状态机：把执行过程变成有状态机，支持暂停和恢复
  • 添加错误处理：工具调用失败时的重试、回退和报告策略
  • 添加记忆系统：跨任务的学习和上下文复用
  • 添加工具结果验证：用 schema 校验工具返回值

进阶阶段（1-2 周）

  • 添加 OpenTelemetry Trace：每个步骤生成 span，可观测
  • 添加持久化：任务状态保存到 SQLite/Redis，崩溃后可恢复
  • 添加子 Agent：复杂步骤委托给专门的子 Agent
  • 添加人机协作：遇到不确定时暂停并请求用户确认

技术栈建议

STACK
Runtime: Python 3.11+ + asyncio LLM: GPT-4o / Claude 3.5 / Qwen2.5 State: SQLite (本地) / Redis (分布式) Trace: OpenTelemetry + Jaeger (可选) Schema: Pydantic + JSON Schema CLI: Click / Rich

验收标准

CHECKLIST
  • 能接收自然语言任务，自动拆分为可执行步骤
  • 工具调用失败时，有重试和回退策略，不直接崩溃
  • 执行过程可观测：每个步骤有输入、输出、耗时
  • 支持暂停和恢复：kill 进程后重启能继续执行
  • 有完整的测试：单元测试 + 集成测试（至少 5 个场景）
  • README 包含架构图、运行方式和扩展指南
