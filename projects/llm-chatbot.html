LLM 应用线 · 实战项目

智能问答助手

基于 RAG 的文档问答系统，支持上传文档、向量检索、多轮对话和引用溯源。

需要Python · FastAPI · 向量数据库 · LLM API 加分React · Docker · CI/CD

项目目标

GOALS

核心目标：搭建一个能理解用户上传文档并基于文档内容回答问题的系统。

为什么做这个：RAG 是 LLM 应用落地的核心模式。掌握它，你能在面试中讲清楚检索增强的完整链路。

预期产出：一个可运行的 Web 服务 + 一份项目文档 + 一次 3 分钟演示。

前置知识

PREREQUISITES

完成以下课程后再开始项目：

  • LLM 基础与应用工程（tokenizer、embedding、context）
  • 提示工程（结构化输入输出、 few-shot）
  • RAG 论文精读（检索增强的原理和边界）

实施步骤

STEPS

MVP 阶段（1-2 周）

  • 搭建 FastAPI 后端骨架，实现健康检查接口
  • 集成向量数据库（Chroma/Milvus），实现文档插入和查询
  • 接入 OpenAI / 本地 LLM，实现基础问答
  • 实现一个简单的 HTML 前端，支持上传文件和提问

增强阶段（2-3 周）

  • 添加文档切分策略（按段落 / 语义 / 滑动窗口）
  • 实现重排序（reranking）提升检索精度
  • 添加多轮对话上下文管理
  • 实现引用溯源：回答中标注来源段落

进阶阶段（1-2 周）

  • 添加评测：准备 20 个标准问题，计算 recall 和 relevance
  • 添加混合检索：关键词 + 向量 + 重排序
  • 添加用户反馈：点赞/点踩，用于迭代优化
  • Docker 化部署，编写 docker-compose.yml

技术栈建议

STACK
Backend: Python + FastAPI + httpx Vector DB: ChromaDB (本地) / Milvus (生产) Embedding: sentence-transformers / OpenAI text-embedding-3 LLM: GPT-4o-mini / Claude 3.5 / 本地 Qwen2.5 Frontend: Vanilla JS + Fetch (MVP) / React (进阶) Deploy: Docker + docker-compose

验收标准

CHECKLIST
  • 能上传 PDF/TXT 文档，并基于文档内容回答问题
  • 回答能标注来源段落，方便验证
  • 支持多轮对话，上下文不丢失
  • 评测集上 Top-3 召回率 > 70%
  • 有完整的 README：安装、配置、运行、测试
  • 能在本地用 docker-compose up 一键启动

面试常见问题

INTERVIEW

Q1: 为什么用向量检索而不是全文搜索？什么时候应该混合使用？

Q2: 文档切分粒度怎么选？太细会怎样，太粗会怎样？

Q3: 如果检索到的段落不相关，你的系统怎么发现和缓解？

Q4: 多轮对话的上下文怎么管理？会不会超出 LLM 的上下文窗口？

Q5: 如果用户上传了 1000 份文档，检索速度变慢怎么办？
