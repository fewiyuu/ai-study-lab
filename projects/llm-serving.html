LLM Serving 服务

基于 vLLM 的高吞吐 LLM 推理服务实战项目。

需要 Python · Docker · GPU · CUDA

项目目标

GOALS

核心目标：搭建一个能同时处理多个请求、高吞吐低延迟的 LLM 推理服务。

为什么做这个：Infra 面试必问推理优化。从 PagedAttention 到 batching 调度，亲手实现一遍才能真正理解。

预期产出：可部署的推理服务 + 性能 benchmark + 监控面板。

前置知识

PREREQUISITES
  • GPU 性能工程（CUDA 基础、显存管理）
  • 现代 LLM 推理引擎内核（PagedAttention、KV Cache）
  • LLM Serving 架构（调度、批处理、限流）

实施步骤

STEPS

MVP 阶段（1-2 周）

  • 用 vLLM 启动一个本地推理服务，跑通文本生成
  • 实现 OpenAI 兼容的 API 格式（/chat/completions）
  • 编写压测脚本，测量吞吐（tokens/s）和延迟（TTFT / TPOT）
  • 对比单请求 vs 多请求的吞吐差异

增强阶段（2-3 周）

  • 实现请求调度：支持 prefill 和 decode 分离调度
  • 添加动态批处理：根据显存和请求长度动态调整 batch size
  • 添加请求优先级：区分实时请求和后台任务
  • 添加负载均衡：多实例分发策略

进阶阶段（1-2 周）

  • 添加 Prometheus 监控：QPS、延迟、显存、队列长度
  • 添加日志和 tracing：每个请求的生命周期可追踪
  • 添加限流和降级：保护服务不被突发流量压垮
  • Docker 化 + docker-compose，支持一键启动

技术栈建议

STACK
Engine: vLLM / SGLang / TGI Model: Qwen2.5-7B / Llama-3.1-8B (单卡可跑) API: FastAPI / OpenAI-compatible Benchmark: locust / aihub-benchmark Monitor: Prometheus + Grafana (可选) Deploy: Docker + docker-compose

验收标准

CHECKLIST
  • 单卡 7B 模型，并发 10 请求时吞吐 > 500 tokens/s
  • 支持 OpenAI 格式的 API，兼容现有客户端
  • 有完整的 benchmark 报告：吞吐、延迟、显存占用
  • 有基本的监控指标：QPS、队列长度、错误率
  • 支持限流：超过容量时返回 429 而不是崩溃
  • README 包含部署指南、性能调参建议和故障排查

面试常见问题

INTERVIEW

Q1: PagedAttention 为什么能减少显存碎片？

Q2: prefill 和 decode 阶段为什么不能简单 batch 在一起？

Q3: 如果请求长度差异很大，怎么避免短请求被长请求阻塞？

Q4: 显存不足时，你选 chunked prefill 还是 request swapping？

Q5: 怎么测量和优化 TTFT（Time To First Token）？
