# 先抓主线 {#overview}
<!-- tag: from call to environment -->

工具使用要分层看：先能输出合法调用，再能选对工具，再能把参数填准，再能根据观察改下一步，最后才是在有状态的环境里稳定完成任务。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

这个分法对应课程前后关系：前面看接口和日志，后面看轨迹和环境。你现在先把层级坐标记住，后面每篇论文都能对号入座。

# 术语与误区 {#terms}
<!-- tag: terms -->

| 术语 | 直觉 | 更准确的说法 | 常见误判 |
| --- | --- | --- | --- |
| 工具 | 模型能请求外部系统做一件事。 | 带名字、描述、参数约束、返回值和运行限制的可调用接口。 | 只盯着函数名，忽略权限、错误、延迟和副作用。 |
| Action / Observation | 模型做一步，系统回一步。 | 行动产生工具调用或环境动作；观察把结果、错误和状态变化送回模型。 | 把观察当成附加说明，没意识到它会改变下一步决策。 |
| 环境 | 会记状态的任务世界。 | 跨轮维护状态、终止条件和奖励的系统，动作会影响后续观察。 | 把无状态工具当环境，结果只会得到一次性的回答。 |
| 奖励 | 告诉模型这条轨迹好不好。 | 由任务结果或环境状态计算出来的反馈信号。 | 以为奖励越密越好，结果模型学会钻局部规则。 |

# 能力层级 {#ladder}
<!-- tag: ladder -->

层级
能做什么
典型失败
先看哪里


L0 格式遵循
输出合法 JSON、字段名和类型正确。
结构破损、字段缺失、类型不对。
schema、解析器、结构化输出约束。


L1 工具选择
从多个工具里挑出相关工具。
选错工具、没选到必须工具、工具太多时混淆。
工具描述、分组和检索顺序。


L2 参数构造
把自然语言目标转成正确参数。
值幻觉、单位错、漏参数。
参数标注、校验错误回传、调用示例。


L3 多步规划
根据观察调整下一步。
不读观察、重复调用、提前结束。
轨迹、action/observation 对、阶段评分。


L4 环境适应
在有状态环境里完成任务并处理失败。
恢复不了异常、越权调用、对终止迟钝。
状态、奖励、沙盒、审计和回退。

## box warn
### 核心判断
一次合法调用只能证明模型至少碰到了 L0 或 L1/L2 的一部分。只有当观察真的改变后续行动，而且任务在状态变化中还能收束，才开始接近 Agentic 工具使用。

# 核心机制 {#mechanism}
<!-- tag: diagnose before train -->
<!-- layout: grid2 -->

## box
### 先分层，再决定训练手段
如果模型连工具名和参数都老出错，直接上 RL 只会把问题揉成一团。分层的意义，是把“失败了”拆成能定位的层：接口问题先修接口，轨迹问题先修观察和规划，环境问题再看状态、奖励和权限。
这也是为什么课程把 Agents SDK、ReAct、ToolBench 和 OpenEnv 放在同一条线上：它们对应同一条链上的不同层。

## box info
$$D = C_{interface} + S_{state} + L_{latency} + R_{risk}$$
这里的 `D` 只是教学上的难度拆解。课堂里用它来判断先修哪一层。`C` 表示接口复杂度，`S` 表示状态连续性，`L` 表示反馈延迟，`R` 表示副作用风险。

### 层级和工程对象怎么对应

| 层级 | 看什么对象 | 读什么 | 写什么 / 改什么 | 出错时的症状 |
| --- | --- | --- | --- | --- |
| L0-L2 | 工具 schema、参数、返回值 | 工具描述、字段约束、错误消息 | 调用文本、参数值、结构化输出 | 解析失败、字段缺失、参数错位 |
| L3 | 轨迹 | 前一步观察、任务目标、历史动作 | 下一步行动、停止信号、改计划 | 重复调用、忽略观察、提前结束 |
| L4 | 环境状态 | 状态、奖励、终止条件、权限边界 | episode 里的动作、回退策略、最终决策 | 越权、奖励投机、任务卡死 |

这张表的重点在读写关系。你要先问“模型读了什么、写了什么”，再问“该用什么方法修”。

# 训练怎么选 {#training-map}
<!-- tag: method map -->

同一条失败日志，在不同层级下应该指向不同的方法。这个判断顺序很重要，因为它会决定你是在补数据、改提示、收紧 schema，还是换成状态化环境。

| 层级 | 最值得先看什么 | 优先方法 | 别急着做什么 |
| --- | --- | --- | --- |
| L0 | 调用文本是否符合 schema。 | 结构化输出、字段约束、解析器回传、少量正反例。 | 别先上奖励，解析都过不去时奖励只会把错误藏起来。 |
| L1 | 工具名、描述、分组和检索顺序是否混淆。 | 工具描述改写、工具分组、检索器、对比样本。 | 别把所有工具一股脑暴露给模型。 |
| L2 | 参数值是否被正确抽取、补齐和校验。 | 带参数标注的 SFT、错误回传样本、参数对齐示例。 | 别只看最终答案对不对，参数错而答案对的样本会误导训练。 |
| L3 | 第一步观察有没有改变第二步行动。 | ReAct 轨迹、阶段评分、失败轨迹重写、多步偏好比较。 | 别在没有轨迹的情况下讨论“规划能力”。 |
| L4 | 状态、奖励、终止和权限是否形成闭环。 | OpenEnv 类环境、episode 训练、奖励检查、沙盒审计。 | 别把生产工具直接当实验环境。 |

你可以把这张表当成一个极简决策树：先看是不是格式问题，再看是不是工具选择问题，再看是不是参数问题，最后才看是否真的进入环境训练。这样做的好处，是你不会把最便宜能修的问题留到最后。

# 状态账本 {#state-ledger}
<!-- tag: state ledger -->

下面这张账本用“同一条订单任务”串起读、写、失效和下一步检查。它的作用是把隐藏状态摊开，让你在调试时知道到底是哪个环节把任务带偏了。

| 步骤 | 读什么 | 写什么 | 什么时候还是有效的 | 错了会怎样 | 下一步查什么 |
| --- | --- | --- | --- | --- | --- |
| 查订单状态 | order_id、当前状态、失败原因。 | 把 observation 追加进轨迹。 | 当下一个动作还没发生时。 | 如果没读到 reason，下一步很容易走错方向。 | 先看 observation 有没有被完整写回上下文。 |
| 决定下一步 | 当前 observation、任务目标、历史动作。 | 下一步 action 或停止信号。 | 当任务目标还没变化时。 | 重复调用、误判状态、提前结束。 | 看 action 前有没有真正引用前一步结果。 |
| 执行有副作用的工具 | 权限、审计、回退条件。 | 真实系统状态。 | 只在权限明确、边界明确时有效。 | 越权、误操作、回退困难。 | 先确认是否能在沙盒或只读模式下复现。 |
| 读取奖励 | 最终状态、任务成功条件。 | 训练信号。 | episode 结束、或者环境给出终止信息时。 | 奖励投机、局部最优、学会钻规则。 | 检查 reward 是否真的对应任务成功，别只看表面合规。 |

读这张表时，最有用的问题是：这一步读了什么、写了什么、有没有把下一步需要的信息保住。一旦某一步把关键状态丢了，后面所有训练都会沿着错误的上下文继续。

# 常见层级陷阱 {#layer-traps}
<!-- tag: mistake map -->
<!-- layout: grid2 -->

## box warn
### 把所有错误都归成“模型不行”
这会让你直接上 RL，结果把最便宜的接口问题、描述问题和参数问题藏进更贵的训练里。更稳的做法，是先用一条失败样本判断它属于哪层。
如果连 JSON 都破，先修 L0；如果工具选错，先修 L1；如果参数抽错，先修 L2；只有当观察没有改变下一步，才进入 L3；只有当状态化闭环真的存在，才轮到 L4。

## box info
**排错顺序**
1. 先看能不能解析。
2. 再看是不是工具选错。
3. 再看是不是参数错。
4. 再看 observation 有没有进下一步。
5. 最后再看环境状态、奖励和权限。
这个顺序的好处是，你不会在 L0 问题上浪费 L4 的训练成本。

如果你以后要自己做评测，最先要加的是一个能把失败分层的记录格式。没有这个分层，训练和诊断会互相打架。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

字段破损
这个输出说明问题先卡在格式层。先修 schema、字段名和类型，别先讨论轨迹和奖励。


观察被忽略
这说明模型看到了结果，却没有把结果带进下一步。先看轨迹和观察回填，再决定要不要换训练算法。


权限越界
这类错误在环境侧更危险，因为它会真的改动系统状态。先补权限模型，再谈闭环训练。

# 操作桥 {#operation-bridge}
<!-- tag: from symptom to next reading -->

这一节把前面的分层判断落到实际动作上。你拿到一条失败样本后，不要先急着翻整篇论文，也不要把它直接扔给更重的训练方法。先把样本拆成四件事：它看见了什么、做了什么、哪里断了、下一步该去看哪类资料。只要这四件事说清楚，后面的阅读顺序就会稳定很多。

这一步很像给失败样本做路由。路由的意思先把样本送到正确的检查台。字段破损应该去 schema 台，工具混淆应该去工具描述台，参数错误应该去参数回填台，观察没进下一步应该去轨迹台，状态和奖励没闭环应该去环境台。你先把台子找对，再决定要不要换方法。

如果你把所有样本都交给更重的训练，表面上看是在做统一处理，实际上是在把低层故障和高层故障混在一起。这样会带来两个后果：一是诊断成本越来越高，因为每个错误都要穿过更多层；二是训练信号越来越脏，因为本来可以靠规则、示例或约束修掉的问题，被你塞进了轨迹或环境里。

桥接章节的目标，是让你把“看见一个坏样本”变成“三步动作”：先定层级，再找证据，再选资料。这个顺序比“先想训练法”更稳，因为训练法永远建立在问题已经被说清楚的前提上。问题没说清楚时，最容易出现的就是方法选得很重，修得却很轻。

## box
### 样本处理四步
第一步，摘出原始症状，例如字段缺失、工具选错、参数写错、观察没进下一步，或者状态和奖励没有闭环。第二步，把症状落到层级上，先判 L0-L2，只有在轨迹和环境信号明确时才往后走。
第三步，写下证据。证据最好是能直接复述出来的日志片段、轨迹片段、表格字段，或者一个能让人复查的状态表。第四步，给出下一篇要看的资料类型，而不是直接给出“应该怎么训练”。
可以把这个流程想成一个轻量排障单：症状、层级、证据、下一步，四栏就够了。比如一条日志如果写着“工具调用合法，但第二步没有读取 observation”，那就先记 L3；如果写着“字段名缺失，结构直接报错”，就先记 L0。你不需要在这一层就把全部修法写满，只要把第一站选对，后面就会轻很多。
这个流程的好处是，你不会把所有问题都往 RL 里塞。很多样本根本不需要更复杂的奖励设计，只需要更好的 schema、工具分组、轨迹标注，或者一个更清楚的环境边界。
```text
症状
 -> 层级
 -> 证据
 -> 下一篇资料
 -> 训练或工程修法
```

## box warn
### 常见误读
最常见的误读，是把所有失败都往最显眼的地方放。比如看到一条轨迹里有工具调用，就以为问题一定在规划；看到一个环境任务失败，就以为一定要换 RL。实际上，很多更早的错误会伪装成更晚的错误。字段破损会像规划失败，工具描述太糊会像参数错，参数错会像轨迹不稳，环境边界没写清会像奖励投机。
第二种误读，是只看结果，不看读写关系。你应该关心模型读了什么、写了什么、下一步有没有用到上一步结果，而不是只盯着最后答对没有。只看最后结果，很容易把一个低层错误拖成高层问题，最后把诊断成本抬上去。
第三种误读，是把“方法名”当成“问题答案”。ReAct、Toolformer、ToolBench、OpenEnv 都有名字，但名字本身不会告诉你该先修什么。你还是要回到样本，先判层级，再看这篇资料到底补的是接口、轨迹，还是环境。
第四种误读，是把“先读资料”误当成“先背结论”。这类课的价值不在于背名字，而在于看到一条日志就知道应该去哪一层找下一步。你能把层级说对，后面的阅读才会省力。
如果你愿意再多做一步，就把样本写成“小卡片”：一行症状、一行层级、一行证据、一行下一步。这样做的好处是，后面的 ReAct、ToolBench 和 OpenEnv 资料都能直接挂到这张卡片上，不会各讲各的。

## box
### 你现在应该能做什么
拿到一条新的失败日志时，先写出三件事：它属于哪层、最直接的证据是什么、下一步该去看哪类资料。能把这三件事写出来，就说明这一页的坐标系已经能用了。
如果一条样本同时出现了接口错和轨迹错，先修更低的那层。低层没通的时候，高层信号常常只是表象。这个判断会直接影响你后面是补 schema、改工具描述，还是读 ReAct 和环境训练资料。
这一节把概念转成动作。你读完这一节，应该能把失败样本从“看起来很乱”变成“知道先看哪里”。

## box
### 补一个小例子
假设你看到一条日志：模型查到“订单待审核”，却还是继续发起“催发货”工单。把它放进桥接表后，你会先记成 L3，再去看轨迹，而不是先追环境奖励。轨迹上要找的第一步观察有没有进入第二步动作。
轨迹摘要
action/observation
```text
action 1: get_order_status(order_id=A1024)
observation 1: status=pending_review
action 2: create_ticket(type=shipping_delay)
```

看到什么
先判什么
下一步


第二步仍然沿着发货走
L3
查观察回填和轨迹样本


第一步字段就破了
L0

这个小例子有一个直接后果：如果你把它误判成环境问题，就会先去改 reward，结果最该修的轨迹和回填被放在后面。下一次遇到同类样本时，先按这张表走一遍，基本就能少绕一圈。
你现在应该能把一条样本写成“症状、层级、证据、下一步”四栏，并且知道它先该去看哪篇资料。这个动作本身就足够支撑后面的单元阅读了。

# 例子 {#example}
<!-- tag: worked cases -->

## box
### 例 1：看上去会调用工具，实际上没读观察
用户说：“查一下订单是否已经发货，如果没有，帮我创建催办工单。”模型先查到结果，再直接创建工单：
轨迹片段
JSON-like
```text
call get_order_status({"order_id": "A1024"})
observation: {"status": "pending", "reason": "payment_review"}

call create_ticket({"order_id": "A1024", "type": "shipping_delay"})
```

这里第一步观察没有改变第二步行动。真正的问题是 L3：模型没把 `payment_review` 读进计划。你可以把它想成一个小型排错实验：输入、观察、下一步动作都在，但状态没有被真正消费，所以轨迹断在中间。
这个例子之所以重要，是因为它能把“结果看起来对”和“过程真的对”分开。后者才是后训练和工具使用 RL 真正要优化的对象。

## box warn
### 修正后的轨迹
轨迹片段
ReAct-style
```text
thought: 订单未发货，但原因是 payment_review，不该创建物流催办。
action: ask_payment_review_status({"order_id": "A1024"})
observation: {"review_eta": "24h", "can_escalate": false}
final: 告知用户订单在支付审核中，预计 24 小时内更新。
```

这条轨迹的关键不在“多调用了一次”，而在于第二步是由第一步观察推出的，而不是硬接着往下点。

## box
### 常见错误
把所有失败都归因于模型“不够聪明”，然后直接上 RL。这个做法通常会把 L0/L1/L2 的错误藏进训练里，最后只剩下更难读的轨迹。
**修法：**先标注失败层级。如果是 L1/L2，先修工具描述、schema、示例和错误回传；如果是 L3/L4，再考虑轨迹数据、环境和奖励。

- [ ] 错

Answer: false
Explain: 无状态工具一次只处理一次调用；环境会保留状态，动作会影响后续观察。

### 4 synthesis

**Q:** 用 120 字以内解释：为什么 function calling 不等于 agentic 工具使用？

Explain: 合格答案会先说 function calling 只证明模型能输出符合 schema 的调用，再说它并没有证明模型会读观察、会修正下一步、会在状态变化中完成任务。提到“单步调用”和“多步观察”通常就能抓住主线；如果还能补上环境状态或错误恢复，会更完整。

### 5 error-diagnosis

**Q:** 模型第一次查询返回“账户余额不足”，但第二步仍然调用“立即转账”。这主要是哪一层失败？

- [ ] L1
- [ ] L3
- [ ] L4

Answer: L3
Explain: 这是多步规划失败，不是单次格式错误。先修轨迹和观察回填，再看是否需要更重的训练。

### 6 scenario-transfer

**Q:** 为“浏览网页并完成多页表单”设计训练评测时，为什么普通无状态工具不够？

Explain: 因为网页会保持状态，前一步动作会改变后一步能看到什么；多页表单还有终止条件和局部错误恢复，单次无状态工具调用只能描述一次点击或一次提交，不能表达整个过程。合格答案最好能说出“状态连续性”和“观察会被动作改变”这两个点。

### 7 checklist

**Q:** 一个客服 agent 可以退款、改地址、发送邮件。你会直接把这些生产工具接进 RL 环境吗？为什么？

- [ ] 沙盒
- [ ] 副作用
- [ ] 权限
- [ ] 审计
- [ ] 回退
- [ ] 奖励投机

Explain: 不应直接接进生产工具。更稳妥的做法是先做只读版本、模拟器或沙盒版本，保留人工确认和回退机制，再逐步放开权限。只说“不安全”不够，最好说明哪些能力先模拟、哪些先只读、哪些必须人工确认。

### 8 synthesis

**Q:** 把本单元讲给一个刚会 function calling 的工程师：能力层级为什么能节省训练成本？

Explain: 好的答案会说明：分层先把问题定位到具体层，避免把简单的格式错误、工具混淆或参数错当成 RL 问题；只有当轨迹和环境层真的需要学习闭环时，再引入更重的训练。能明确举出一个低层问题，例如 JSON 破损或参数错位，不该直接上 RL，就说明理解到位了。
