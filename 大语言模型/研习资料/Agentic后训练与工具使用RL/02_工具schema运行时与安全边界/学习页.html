课程单元

工具 schema、运行时与安全边界

本页只讲一件事：怎样把工具调用拆成 schema、运行时和安全边界三层。先把这三层分清，后面读 ReAct、Toolformer、ToolBench、API-Bank 和 OpenEnv 时，才知道每篇材料是在补接口、补轨迹，还是补环境与权限。

课程地图

读完本页后你应该能做什么

  • 能区分 schema、运行时、guardrail、handoff、tracing 的职责边界。
  • 能读一个工具定义，判断它缺了哪些参数约束和权限约束。
  • 能把自然语言安全要求改写成可执行的运行时策略。

学习目标

Foundation / Interface

拆开三层

把“模型能生成工具调用”拆成接口描述、运行时处理和安全策略三件事，不再用一个 function calling 标签覆盖全部问题。

读 schema

理解名称、描述、参数类型、必填项、枚举、范围和额外字段限制为什么会影响模型选择工具和填写参数。

设计边界

知道何时用只读工具、沙盒、人工确认、输出脱敏、trace 审计和动态禁用来限制副作用。

先抓主线

Schema -> Runtime -> Boundary

工具调用不是“模型输出一个 JSON”这么简单。可靠系统至少有三层：schema 告诉模型能填什么，运行时决定怎样解析、校验、执行和回传观察，安全边界决定哪些调用即使格式正确也不能执行。

工具意图 → schema 约束 → 运行时校验 → 权限与 guardrail → 执行观察与 trace

一句话判断

如果一个约束可以通过“参数格式”表达，它属于 schema；如果它依赖用户、环境、状态、权限、风险或历史轨迹，它属于运行时和安全边界。

本单元不做什么

这里不训练模型，也不展开 ReAct 轨迹。目标是先把工具执行面的工程结构讲清楚，为后面的行动-观察数据和环境 RL 做准备。

术语与误区

Terms
术语 直觉解释 精确定义 常见误解
工具 schema 告诉模型“这个工具能填哪些格子”。 工具名、描述和参数结构的机器可读说明，常用 JSON Schema 风格表达类型、必填项、枚举、范围和额外字段规则。 以为 schema 等于安全。schema 只能约束形状，不能判断用户是否有权限删除文件。
运行时 真正接住工具调用并执行的系统。 负责解析模型输出、验证参数、注入上下文、选择工具、调用函数或外部服务、处理异常、返回观察和记录日志的执行层。 以为模型直接调用 API。实际是模型提出调用，运行时决定是否执行。
Guardrail 调用前后的一道检查门。 对输入、工具参数、工具输出或最终回答执行的策略检查；可以放行、拒绝、改写、触发人工确认或终止流程。 以为只在最终回答做安全过滤就够。工具输入和工具输出也可能泄密或产生副作用。
Handoff 把任务转交给更合适的 agent 或流程。 当当前 agent 不应继续处理时，把上下文转给另一个专业 agent、人工队列或审批流程。 把 handoff 当普通工具调用。handoff 通常改变责任边界和后续策略。
Tracing 把一次运行拆成可审计的时间线。 记录模型生成、工具调用、函数输入输出、guardrail 结果、handoff 和错误等 span，支持调试、评测和事故复盘。 只把 trace 当调试日志。训练数据清洗、奖励归因和安全审计都依赖它。

Schema 设计

Tool Interface

1. Schema 解决的是“可表达性”问题

模型面对工具列表时，并不知道你的后端函数签名。它只能根据工具名称、描述和参数说明判断：该不该用、什么时候用、每个参数应该填什么。schema 越含糊，模型越容易用错工具；schema 越过宽，模型越容易塞入运行时没准备处理的字段。

一个实用 schema 需要同时服务两类读者：模型读描述来决策，运行时读结构来校验。只给模型自然语言提示而没有结构约束，会让错误延迟到执行时才爆；只给结构而不给清晰描述，会让模型选错工具。

好 schema = 明确工具意图 + 收窄参数空间 + 保留必要上下文 + 明确不能表达的约束

最小检查清单

  • 工具名是否动词化、单一职责？
  • 描述是否说明何时用、何时不用？
  • 必填参数是否真的必填？
  • 枚举、范围、格式是否足够收窄？
  • 是否禁止无意义的额外字段？
  • 危险动作是否只在 schema 中描述，而没有运行时权限检查？
一个较稳的工具 schema JSON Schema 风格 
{
  "name": "search_customer_orders",
  "description": "按客户 ID 查询订单摘要。只用于读取订单状态，不用于退款、改地址或取消订单。",
  "parameters": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "内部客户 ID，不要填姓名、邮箱或自然语言描述。"
      },
      "status": {
        "type": "string",
        "enum": ["open", "paid", "shipped", "cancelled"],
        "description": "要过滤的订单状态。"
      },
      "limit": {
        "type": "integer",
        "minimum": 1,
        "maximum": 20,
        "description": "最多返回多少条记录。"
      }
    },
    "required": ["customer_id"],
    "additionalProperties": false
  }
}

这个 schema 做了几件事：工具名说明动作和对象；描述强调只读边界；customer_id 避免模型填入邮箱或自然语言；status 用枚举降低拼写和语义漂移；limit 限制成本；additionalProperties: false 拒绝模型临时编造字段。

名称

优先用 verb_object，如 create_ticket。不要用 handle_request 这种泛化名字。

描述

描述不仅说功能，也要说不用场景。模型经常通过描述边界决定是否调用。

参数

参数越接近业务原子字段越好。把“query”当万能参数会把风险推给后端。

运行时

Runtime

在 agent 系统里，模型通常只生成一个“想调用某工具并传入这些参数”的意图。运行时才是执行者。它需要把模型输出转成结构化调用，检查参数是否合法，补入用户身份、租户、会话、权限等上下文，再决定是否真的执行。

1

暴露工具

运行时把当前可用工具及 schema 放进模型上下文。动态禁用的工具不应出现在模型可见列表里。

2

模型提议

模型选择工具并生成参数。这个阶段只产生意图，不应该直接触碰外部系统。

3

解析校验

运行时解析 JSON、校验必填项、类型、枚举、范围和额外字段，必要时要求模型重试。

4

执行或拒绝

运行时检查权限、速率、沙盒、审批和 guardrail；通过后调用函数或外部服务。

5

回传观察

工具输出被脱敏、截断、格式化后回到模型，trace 记录每一步。

运行时必须处理错误

工具调用失败不是异常边角料，而是 agent 训练和评测的核心信号。错误可能来自 schema 不匹配、权限不足、资源不存在、网络失败、工具返回过长、状态已变化或动作不可逆。运行时应把错误转成模型可理解的观察，同时保留机器可读错误码。

观察不是裸输出

工具返回的原始数据可能包含密钥、内部 ID、超长日志或与任务无关的信息。观察应只包含下一步推理需要的最小信息，并标注成功、失败、部分成功或需要确认。

运行时包装工具调用的伪代码 Python-like 
async def run_tool_call(call, user_context):
    tool = registry.get(call.name)
    if tool is None:
        return observation(error="unknown_tool")

    args = parse_json(call.arguments)
    schema_result = validate(args, tool.schema)
    if not schema_result.ok:
        return observation(error="schema_error", details=schema_result.summary)

    policy_result = check_policy(
        user=user_context.user,
        tenant=user_context.tenant,
        tool=tool.name,
        args=args,
        risk=tool.risk_level
    )
    if policy_result.requires_approval:
        return request_human_approval(tool.name, args, policy_result.reason)
    if not policy_result.allowed:
        return observation(error="permission_denied", details=policy_result.reason)

    with trace_span("tool_call", tool=tool.name, redacted_args=redact(args)):
        raw_output = await tool.invoke(args, timeout=tool.timeout)
        safe_output = redact_and_limit(raw_output, max_tokens=1200)
        return observation(status="ok", content=safe_output)

这段伪代码的关键不是某个框架 API，而是责任分离：模型不负责权限判断；schema 不负责业务策略；工具函数不应该自己承担所有安全控制；trace 不应该泄露敏感参数。

决策轨迹

Shape Trace

同一条工具请求，在模型、运行时和观察阶段长得不一样。模型看到的是自然语言目标和工具列表，运行时看到的是结构化参数和策略结果，回传给模型的则是经过脱敏和压缩的观察。把这三种形态放在一条轨迹里看，最容易发现问题究竟出在参数、权限，还是回传信息。

模型先看到的对象before: 自然语言目标

模型先把“帮我查发票”翻成工具名和参数草稿。这个阶段最常见的错误是参数看上去像样，但身份、租户或风险信息还没进来。

运行时真正处理的对象after: 权限和策略结果

运行时读的不是“像不像”，而是“能不能做”。同样的参数，在只读查询里能放行，在写入动作里可能直接要求人工确认。

回到模型里的对象after: observation

工具执行后，模型拿到的是压缩过的观察，不是原始输出。观察如果没有保留关键错误，下一步就会继续沿着错误上下文推理。

步骤 看见什么 写回什么 失败症状
提议层 工具名、参数草稿、请求目标。 结构化调用。 字段对了，但身份或上下文不在。
裁决层 用户身份、租户、风险、审批状态。 allow / deny / approve。 该拦没拦住，或者该放没放。
回传层 工具输出、错误码、脱敏后的摘要。 下一轮可继续用的 observation。 模型看不到关键错误，重复走错路。

为什么要单独看这层

如果 trace 里只有“模型说了什么”，你看不出运行时为什么拒绝；如果只有“工具返回了什么”，你看不出它是在放行后执行、审批中等待，还是直接被拦截。把三层都记录下来，错误才有机会被准确归类。

状态账本

Read / Write / Fail

工具调用里最容易漏掉的不是函数名，而是状态。模型先看到了什么，运行时又补进了什么上下文，工具执行之后系统状态有没有变化，观察回传时有没有把下一步需要的信息保住，这些都决定了后面还能不能继续做对。

下面这张账本把 schema、运行时、权限检查、工具执行和观察回传放到同一张表里。你读表的时候，别先急着背术语，先看每一步“读了什么、写了什么、什么时候失效、出错后症状是什么”。

步骤 读什么 写什么 什么时候还有效 错了会怎样 下一步查什么
schema 解析 工具名、参数类型、必填项、枚举和范围。 把模型输出变成结构化参数。 当模型输出能被稳定解析时。 字段缺失、类型错、额外字段混入。 先看工具定义和参数约束。
policy 检查 用户、租户、环境、风险级别和审批状态。 允许、拒绝或要求人工确认。 当身份和环境信息还是最新时。 越权、误放行、该拦没拦住。 先看运行时有没有注入正确上下文。
工具执行 校验通过的参数、超时、可回滚条件。 真实系统状态或外部服务请求。 只有在权限和上下文都一致时。 副作用、网络失败、状态已经变了。 先看是否要沙盒、只读或可回滚。
观察回传 原始输出、错误码、脱敏规则、任务上下文。 给模型的 observation 和 trace 记录。 当下一步还要继续推理时。 泄密、截断过度、模型看不到关键错误。 检查观察是否保留了下一步需要的信息。

为什么这张表有用

工具调用里的很多 bug 不是“模型太笨”，而是状态在某一步丢了：参数没校验、身份没注入、执行没有回滚、观察没脱敏。把这些步骤拆开之后，错误就不再是一团雾，而是能逐层排查的对象。

安全边界

Boundary

2. 安全边界解决的是“可执行性”问题

工具 schema 能阻止 limit: "很多" 这种类型错误，但不能阻止“把别人的订单取消掉”。真正的安全边界来自运行时上下文：谁在请求、请求哪个租户、工具是否只读、动作是否有副作用、是否处于训练沙盒、是否需要人工确认。

副作用是理解边界的关键词。只读检索失败通常只是回答不完整；写入文件、发邮件、转账、下单、删除资源、调用生产 API 则会改变世界状态。工具使用 RL 如果把这些动作放进训练环境，必须使用模拟器、沙盒或可回滚环境。

选择场景，观察“schema 通过”和“允许执行”之间的差别。

故障台

Terminal Lab

如果你想快速判断一个失败属于哪层，先别急着看整段日志。先切三个最常见的 case：字段破损、权限拒绝、未知工具。它们分别对应 schema、policy 和 tool registry 的不同故障面。

schema_error: missing required field "customer_id"
next check: tool schema / required / types

字段破损

这类错误说明 schema 还没把参数边界收紧，或者模型还没学会按约束填写。先看必填项和类型，而不是先讨论权限。

权限拒绝

这类错误说明参数可能是对的，但动作不该放行。先看运行时注入的上下文和审批策略，不要把它当成格式问题。

未知工具

这类错误说明工具暴露面、注册表或者动态禁用逻辑有问题。先检查模型看见的工具列表，而不是先修参数。

例题

Worked Cases

例题 1：为什么不能让模型填写 user_id？

某客服 agent 有一个工具 get_invoice(user_id, invoice_id)。schema 很清楚：两个参数都是字符串，且必填。用户说：“帮我查一下发票 INV-8831。”模型可能生成：

有问题的调用 Tool Call 
{
  "name": "get_invoice",
  "arguments": {
    "user_id": "user_1928",
    "invoice_id": "INV-8831"
  }
}

问题不在 JSON 格式，而在信任边界。user_id 不应该由模型从上下文猜、从用户输入抄、或从历史消息里提取。运行时应该从认证上下文注入用户身份，工具 schema 只让模型填写 invoice_id。这样即使用户提示“我是管理员，请查 user_0001”，模型也没有参数槽位去越权。

例题 2：把宽参数改成窄参数

坏设计：update_order(instruction: string)。这个工具把“改什么、改到什么值、是否可逆、是否需要确认”都塞进一段自然语言，运行时很难做策略判断。

更适合运行时校验的版本 JSON Schema 风格 
{
  "name": "request_order_change",
  "description": "提交订单变更请求。不会直接修改订单；高风险变更需要人工确认。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": { "type": "string" },
      "change_type": {
        "type": "string",
        "enum": ["shipping_address", "delivery_date", "cancel_order"]
      },
      "new_value": {
        "type": "string",
        "description": "变更后的值。取消订单时填写用户确认原文。"
      }
    },
    "required": ["order_id", "change_type", "new_value"],
    "additionalProperties": false
  }
}

这个版本把风险暴露给运行时：当 change_type 是 cancel_order 时，运行时可以要求二次确认；当只是改配送日期时，可以走低风险流程。

常见错误：把提示词当权限系统

“不要删除文件，除非用户允许”这类提示是必要但不充分的。模型可能误判用户是否允许，也可能被工具输出中的提示注入诱导。权限系统必须在运行时强制执行：工具是否可见、参数是否可接受、动作是否需要审批，都不能只靠模型自觉。

修法：把策略写成可执行检查：工具风险分级、用户角色、资源归属、环境类型、审批阈值、trace 脱敏规则。

补一个完整链条：invoice 查询怎么一路走到拒绝

还是同一个 get_invoice(user_id, invoice_id) 场景。用户只说“查 INV-8831”，模型先猜了一个 user_id 并生成调用。运行时读到这个参数后，会先过 schema，再过 policy。假如 user_id 来自模型而不是认证上下文，policy 就应该把它当作越权风险，而不是把它当作普通字符串。

这时 trace 的价值就出来了：trace 能把“模型先补了一个身份字段、运行时拒绝、观察里返回 permission_denied”串起来，后面排查时可以直接看到问题是出在参数构造还是权限判断。若没有这条链，排查人员往往会误以为只是用户输入不完整，或者错把拒绝当成 schema 解析失败。

失败后的观察 policy / trace 
call get_invoice({"user_id":"user_1928","invoice_id":"INV-8831"})
trace: source=user_message for user_id
policy: reject -> user_id must come from auth context
observation: permission_denied / retry without user_id

这条链路说明一个实用判断：当参数看上去“格式对了”，但身份来源不对时，问题就不在 schema，而在运行时和安全边界。下一步该修的是注入策略、权限规则和 trace，而不是再给模型更多提示词。

这页先不展开的内容有五个：ReAct 的完整提示模板、Toolformer 的数据生成细节、ToolBench 的任务集合、OpenEnv 的代码接口、沙盒和上线审计的具体实现。现在先把层级坐标记住，它们会在后面的单元重新出现。

如果你在读后面的论文时开始纠结“到底先用 SFT 还是先用 RL”，先回到这页，看看问题卡在哪一层。很多时候答案不是换训练法，而是先把失败症状说清楚。

再往后读时，记住一个简单规则：接口问题先看可解析性，轨迹问题先看观察是否改变计划，环境问题先看状态、奖励和权限有没有闭环。这个规则比任何一篇单独论文都更早派上用场。

设计与诊断

作答结构 先指出风险，再归类到层级，最后给出可执行修法。
error diagnosisschema

run_admin_action(action: string) 这类工具 schema 有什么问题？请指出至少 3 个。

答题脚手架：问题 1 属于哪层；会导致什么失败；怎么改。

参考答案 / 评分要点

至少应指出三类问题：工具名过泛，无法表达具体动作；action 太宽，运行时无法稳定做策略判断；描述没有写清只读、写入、危险操作的差别；没有把权限、审批和 trace 规则拆开。合格答案要把这些问题分别归到 schema 或运行时层，而不是只说“太模糊”。

scenario transferguardrail

为发送邮件工具设计安全边界：调用前、调用后、trace 三个位置各要检查什么？

至少写出 2 个调用前检查、1 个调用后检查、1 个 trace 记录规则。

参考答案 / 评分要点

调用前应检查收件人是否合法、发信人是否有权限、是否需要人工确认、是否超过频率限制。调用后应检查是否成功、是否需要重试、以及邮件内容是否做了脱敏。trace 至少要记录工具名、风险等级、审批结果和脱敏后的参数。只写“要安全”不够，必须是系统可执行的规则。

解释handoff

解释 handoff 和普通工具调用的区别。

用“谁继续负责这个任务”作为核心判断点。

参考答案 / 评分要点

普通工具调用是“模型提议，运行时执行，返回观察”；handoff 则把任务和上下文交给另一个 agent、人工队列或审批流程，责任主体会变化。答案里如果能提到“后续策略改变”或“trace 里会看到责任转移”，就更完整。

策略化runtime

把“不要让模型乱删文件”改写成至少 4 条运行时可执行策略。

不要只写“提示模型不要删”，要写系统能检查的条件。

参考答案 / 评分要点

可执行策略至少应包含：限定可操作目录；禁止通配符删除；高风险删除必须审批；默认在沙盒或可回滚环境执行；记录删除前后 trace；对模型输出做参数白名单检查。合格答案会把“不要乱删”拆成系统可以拦截、审计和回退的规则，而不是停在提示词层。

综合迁移

把本单元连接到后续 ReAct 和环境 RL：工具调用会变成行动-观察轨迹。

核心连接 训练数据里的每一步行动都应该能追溯到 schema、运行时状态和安全结果。
综合解释RL 衔接

为什么 trace 对工具使用 RL 很重要？

答题顺序：先说明 trace 记录什么，再说明没有 trace 会导致什么训练问题。

参考答案 / 评分要点

trace 能把工具调用、参数、审批、错误和观察串成可回放的时间线，因此它既能帮助复盘错误，也能帮助构造训练数据和奖励。没有 trace，工具失败很难定位到 schema、运行时或安全策略哪一层，RL 训练也容易学到错误归因。好的回答至少要提到“错误归因”和“数据过滤”中的一项。

费曼解释synthesis

用一段话讲完整闭环

要求包含这 5 个词：schema、运行时、guardrail、观察、trace。

参考答案 / 评分要点

合格答案会把流程说完整：模型先根据 schema 决定是否调用、调用什么参数；运行时负责解析、校验、执行或拒绝；guardrail 在关键节点拦截高风险输入和输出；工具结果被整理成观察回传给模型；trace 则把整个闭环记录下来，便于复盘、评测和审计。只要把这五个环节连起来，答案就已经抓住主线。
