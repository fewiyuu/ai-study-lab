ReAct：推理与行动交替

本页把 ReAct 当成一种轨迹组织方式来读：模型先给出可检查的想法，再采取动作，环境返回观察，下一轮再把观察纳入判断。这样读，后面的 Toolformer、ToolBench、API-Bank 和 OpenEnv 就不会混在一起。

学习目标

trajectory

分清轨迹

能从一段 ReAct 记录里分出 Thought、Action、Observation 和 Final Answer，知道哪一步来自模型，哪一步来自环境。

找出断点

能看出轨迹哪一步坏了，并判断该回头修参数、工具选择、观察回传，还是停止条件。

术语与误区

Terms
术语直觉解释精确定义常见误解
Thought模型在当前一步要怎么想。为了决定下一步动作而写出的中间判断，通常会吸收前文和环境反馈。把它当成最终答案草稿，结果下一步动作根本没变。
Action模型对外部世界做了什么。向工具、检索系统或环境发出的可执行请求。以为有了 Action 就结束了；没有 Observation，循环还没闭合。
Observation外部世界回来的证据。工具、检索器或环境返回给下一轮推理使用的信息。把 Observation 当成装饰日志，下一轮 Thought 完全不看它。
Final Answer这轮任务的终局回答。当证据足够时给出的最终输出，不再继续行动。把它当成每一步都该有的字段；多数步骤只需要 Thought / Action / Observation。
Trajectory一条可回放的解决路径。Thought、Action、Observation 交替出现的多步记录。只看最后一句，看不见中间为什么错。
Stop condition什么时候该停下来回答。当当前证据足够支撑回答时，结束行动并输出 Final Answer。总想着再查一次，最后只是在重复调用同一工具。

轨迹机制

shape

ReAct 的关键，不在于 Thought 写得长，而在于动作要能接上观察，观察要能推动下一轮判断。只要某一轮里 Thought、Action、Observation 断开，后面留下的就只是日志，不再是可学习的轨迹。

断点

如果 Action 和 Thought 没对上，先修动作；如果 Observation 没被用进下一轮，先修读回来的方式；如果没有停下来的时机，先修停止条件。

轨迹语法

formula
$$ \text{ReAct trajectory} = (Thought, Action, Observation)_{1..T} + Final\ Answer $$

T 表示循环轮数；Thought 是下一步判断，Action 是对外执行，Observation 是环境回传，Final Answer 是证据足够后的终局回答。

代码
{
 "task": "确认 ReAct 论文的年份",
 "steps": [
 {
 "thought": "我还缺年份，先查论文页面。",
 "action": {
 "tool": "search_web",
 "arguments": {
 "query": "ReAct Synergizing Reasoning and Acting in Language Models"
 }
 },
 "observation": {
 "status": "ok",
 "year": 2022
 }
 },
 {
 "thought": "年份已经明确，可以直接回答。",
 "action": {
 "tool": "finish",
 "arguments": {}
 },
 "observation": {
 "status": "n/a"
 }
 }
 ],

}

轨迹读写表

read

把 Thought、Action、Observation 分开看，最容易知道问题出在哪一层。下面这张表把每一步读什么、写什么、什么时候会失效、错了会出现什么症状放在一起。

读表小窍门

如果你只能描述“最后答对了没有”，还不算会读 ReAct。能指出哪一步读错、写错、或者没有把观察接回下一轮，才算真正抓到了轨迹结构。

训练桥接

SFT

ReAct 先是一种轨迹范式，再是训练素材。常见用法有三种：一是把轨迹当示范数据，训练模型学会下一步 Thought / Action；二是把轨迹当评测记录，拆出是想错、选错工具，还是没读 Observation；三是把轨迹当环境 rollout 的雏形，后面再接奖励和优化框架。

代码
def to_sft_pairs(trajectory):
 pairs = []
 history = []
 for step in trajectory["steps"]:
 prompt = {
 "task": trajectory["task"],
 "history": history[:]
 }
 target = {
 "thought": step["thought"],
 "action": step["action"]
 }
 pairs.append((prompt, target))
 history.extend([
 {"thought": step["thought"]},
 {"action": step["action"]},
 {"observation": step["observation"]}
 ])
 return pairs

样本拆分

concrete

这一节把同一条 ReAct 轨迹拆成三种常见样本：监督微调样本、评测记录、环境 rollout 片段。这样做的意义是让你一眼看出，哪一层要保留什么，哪一层又该删掉什么。

最容易犯的错

很多人只保留最终答案，或者把整段轨迹原封不动塞进训练集。前者会丢掉关键监督，后者会把无关噪声也带进去。

样本类型输入里保留什么输出或标签保留什么常见丢失点
SFT 样本任务、历史上下文、上一步 Observation。下一步 Thought / Action。只保留最后答案，模型学不会中间判断。
评测记录每一步 Thought、Action、Observation 和错误码。正确与否、错误归因、回放路径。只看终局，看不见是哪一步坏了。
rollout 片段状态、动作、反馈、终止条件。奖励入口、环境返回、下一状态。把文本轨迹直接当状态序列，忽略环境语义。

同一轨迹的拆分伪代码

复制

def split_react_sample(trace):
 sft = []
 eval_rows = []
 rollout = []
 for i, step in enumerate(trace["steps"]):
 sft.append({
 "prompt": {
 "task": trace["task"],
 "history": trace["steps"][:i]
 },
 "target": {
 "thought": step["thought"],
 "action": step["action"]
 }
 })
 eval_rows.append({
 "step": i + 1,
 "thought": step["thought"],
 "action": step["action"],
 "observation": step["observation"]
 })
 rollout.append({
 "state": step["observation"],
 "action": step["action"],
 "next_state": trace["steps"][i + 1]["observation"] if i + 1 < len(trace["steps"]) else None
 })
 return {"sft": sft, "eval": eval_rows, "rollout": rollout}

怎么判断拆分对不对

如果你把任务描述、历史上下文和上一步观察都放进了 SFT 样本，模型就有机会学到“下一步怎么想”；如果你保留了每一步动作和观察，评测就能定位错在什么地方；如果你把状态变化也保留下来，rollout 才有继续往后推的意义。

常见失败

最常见的失败有三个：一是只保留最终答案，二是把 Observation 当成解释性文字，三是把文本轨迹误当成环境状态。修法也很直接：先确认每一步的读写对象，再决定数据该怎么切。

操作桥接

source

如果你要把 ReAct 用到真实任务里，第一步通常不是训练，而是把一条轨迹做成“可以回放、可以切分、可以定位失败”的数据。这样你才知道下一步是在做监督样本、评测样本，还是环境 rollout 片段。

先修哪里

这时别急着改最终答案。先看 Thought 是否在表达缺口，Action 是否真的去取证，Observation 是否保留了下一轮能用的信息。轨迹结构先稳住，后面再谈策略。

任务侧对象要保留什么要删掉什么为什么
SFT 输入任务目标、历史上下文、上一轮 Observation。与当前动作无关的长解释。训练要学的是下一步怎么接，不是复述整段聊天。
评测记录每一步动作、观察、错误码、最终结果。对结果没有帮助的背景闲话。评测要能定位“错在什么层”，不是只看最后对没对。
rollout 片段状态、动作、反馈、终止条件、奖励入口。不能映射到环境状态的文本装饰。环境训练要能继续推进，不是把自然语言原样搬进去。

轨迹归一化伪代码

复制

def normalize_react_trace(trace):
 normalized = []
 for step in trace["steps"]:
 item = {
 "task": trace["task"],
 "thought": step["thought"].strip(),
 "action_name": step["action"]["tool"],
 "action_args": step["action"]["arguments"],
 "observation": step["observation"],
 }
 normalized.append(item)
 return {
 "samples": normalized,
 "has_final_answer": bool(trace.get("final_answer")),
 "can_be_sft": all("thought" in s and "action_name" in s for s in normalized)
 }

看这段伪代码时要盯住什么

它把一条自然语言轨迹拆成了三种可检查对象：任务、动作和观察。这样一来，监督微调、评测和 rollout 读到的是同一份事实，而不是三份互相打架的文本。

如果这一步做坏了

漏掉 Observation，模型会学会盲动；把所有日志都塞进去，评测会被噪声淹没；把用户自然语言直接当状态，rollout 就会失去环境语义。

这个桥接一旦做好，后面再读 ToolBench、API-Bank 和 OpenEnv，就能直接问：它是在补哪一段轨迹，是在改哪一种监督，还是在补哪一种状态反馈。

例题

worked

例 1：问答里怎么用检索

ReAct 论文里最直观的场景之一，是问答任务先查证据，再回答。模型先判断缺什么事实，再调用检索或维基 API，拿到观察后再收束答案。这样做的好处，是证据链能被人看到，也能被后续训练复用。

问答轨迹复制

 "task": "ReAct 论文什么时候发表？",
 "steps": [
 {
 "thought": "我需要确认论文年份，先查论文标题。",
 "action": {
 "tool": "search_web",
 "arguments": { "query": "ReAct Synergizing Reasoning and Acting in Language Models" }
 },
 "observation": { "status": "ok", "year": 2022 }
 }
 ],
 "final_answer": "ReAct 论文发表于 2022 年。"
}

例 2：交互任务里的下一步

交互式任务里，Observation 不只是“查到了什么”，还包括环境状态有没有变化。比如购物任务里，模型先查价格和库存，再决定是否继续筛选；环境任务里，模型看到门没开，就会改动作，而不是照着上一轮的计划原封不动再试一次。

交互轨迹复制
Thought: 我还缺尺寸信息，先筛选可选尺码。
Action: filter(size="M", color="black")
Observation: M 码库存不足，L 码可用。
Thought: 计划要改成 L 码，再继续看配送。

例 3：看起来像 ReAct，实际不是

坏轨迹复制
Thought: 我应该查订单状态。
Action: get_weather(city="北京")
Observation: 北京晴。
Final Answer: 订单已经发货。

这条轨迹的病根很清楚：Thought 说的是订单，Action 调的是天气；Observation 也没有帮助回答订单问题。最终答案看上去顺，但中间没有一条线把它撑住。

ReAct 的坑通常不在术语本身，而在轨迹有没有真的闭环。下面这张表按“问题样子 -> 为什么错 -> 下一步查什么”来读。

故障台

terminal

要快速判断一个失败是轨迹问题还是工具问题，先切成三个最常见的 case：缺观察、错动作、旧证据。它们分别对应 ReAct 里三种典型断点。

错动作

这类错误说明计划和执行没对齐。先看 Thought 是否真的在引导动作，而不是只写了一个像样的句子。

旧证据

这类错误说明观察已经过期，但计划还没更新。先看停止条件和重试策略，再看是否需要换工具。
