课程单元

ReAct：推理与行动交替

本页把 ReAct 当成一种轨迹组织方式来读：模型先给出可检查的想法，再采取动作，环境返回观察，下一轮再把观察纳入判断。这样读，后面的 Toolformer、ToolBench、API-Bank 和 OpenEnv 就不会混在一起。

课程地图上一单元下一单元

读完本页应该能做什么

  • 能从一段轨迹里分出 Thought、Action、Observation 和 Final Answer。
  • 能看出轨迹哪一步坏了，并判断该回头修参数、动作还是观察。
  • 能把 ReAct 轨迹转成示范数据、评测记录或环境 rollout 的雏形。

学习目标

trajectory / interface

分清轨迹

能从一段 ReAct 记录里分出 Thought、Action、Observation 和 Final Answer，知道哪一步来自模型，哪一步来自环境。

找出断点

能看出轨迹哪一步坏了，并判断该回头修参数、工具选择、观察回传，还是停止条件。

接到后训练

能把 ReAct 轨迹转成示范数据、评测记录或环境 rollout 的雏形，知道它和 RL 算法不同。

学习边界

这页先把轨迹讲顺，不展开完整 prompt 模板、奖励函数和优化器。那些内容属于后面的训练单元。

先抓主线

source boundary

课程地图里，02 已经把接口、运行时和安全边界拆开；03 在这个基础上看轨迹怎样生成、怎样被读、怎样接到训练和评测。ReAct 论文把 reasoning traces 和 task-specific actions 交替起来，观察会推动下一步计划更新。这个单元先把这种循环读顺，再去看 Toolformer、ToolBench、API-Bank 和 OpenEnv。

读轨迹的顺序

先看“缺什么证据”，再看“做了什么动作”，最后看“环境回了什么”。ReAct 的价值在于把中间步骤显式化，错误不再只藏在最终答案里。

先别追的细节

完整 prompt 模板、奖励函数和优化器都不在这页展开。03 只负责把轨迹结构、读法和失败症状讲清楚。

术语与误区

Terms
术语 直觉解释 精确定义 常见误解
Thought 模型在当前一步要怎么想。 为了决定下一步动作而写出的中间判断，通常会吸收前文和环境反馈。 把它当成最终答案草稿，结果下一步动作根本没变。
Action 模型对外部世界做了什么。 向工具、检索系统或环境发出的可执行请求。 以为有了 Action 就结束了；没有 Observation，循环还没闭合。
Observation 外部世界回来的证据。 工具、检索器或环境返回给下一轮推理使用的信息。 把 Observation 当成装饰日志，下一轮 Thought 完全不看它。
Final Answer 这轮任务的终局回答。 当证据足够时给出的最终输出，不再继续行动。 把它当成每一步都该有的字段；多数步骤只需要 Thought / Action / Observation。
Trajectory 一条可回放的解决路径。 Thought、Action、Observation 交替出现的多步记录。 只看最后一句，看不见中间为什么错。
Stop condition 什么时候该停下来回答。 当当前证据足够支撑回答时，结束行动并输出 Final Answer。 总想着再查一次，最后只是在重复调用同一工具。

轨迹机制

shape tracer

ReAct 的关键，不在于 Thought 写得长，而在于动作要能接上观察，观察要能推动下一轮判断。只要某一轮里 Thought、Action、Observation 断开，后面留下的就只是日志，不再是可学习的轨迹。

模型先看的对象 before: 目标 + 现有上下文

这一步还不知道缺什么证据。最常见的失误，是 Thought 看起来合理，Action 却没有依赖它。

运行时接住的对象 after: 结构化动作

运行时把动作发到工具或环境里。这里最怕参数对了，动作却选错了对象。

回到模型里的对象 after: observation

观察必须进入下一轮 Thought。若 Observation 没改变计划，轨迹就只是在堆日志。

读法

先问“这一轮缺的是什么”，再问“动作有没有补到这个缺口”，最后问“环境回来的证据有没有被下一轮吸收”。这三个问题能把大多数 ReAct 轨迹读通。

断点

如果 Action 和 Thought 没对上，先修动作；如果 Observation 没被用进下一轮，先修读回来的方式；如果没有停下来的时机，先修停止条件。

轨迹语法

formula / code
$$ \text{ReAct trajectory} = (Thought, Action, Observation)_{1..T} + Final\ Answer $$

T 表示循环轮数；Thought 是下一步判断，Action 是对外执行，Observation 是环境回传，Final Answer 是证据足够后的终局回答。

两步轨迹样例 
{
  "task": "确认 ReAct 论文的年份",
  "steps": [
    {
      "thought": "我还缺年份，先查论文页面。",
      "action": {
        "tool": "search_web",
        "arguments": {
          "query": "ReAct Synergizing Reasoning and Acting in Language Models"
        }
      },
      "observation": {
        "status": "ok",
        "year": 2022
      }
    },
    {
      "thought": "年份已经明确，可以直接回答。",
      "action": {
        "tool": "finish",
        "arguments": {}
      },
      "observation": {
        "status": "n/a"
      }
    }
  ],
  "final_answer": "ReAct 论文发表于 2022 年。"
}

这段样例要看什么

第一轮先补证据，第二轮再收束回答。Action 不是把思考换个格式重说一遍；Observation 也不是备注，而是下一轮决策的输入。

轨迹读写表

read / write / fail

把 Thought、Action、Observation 分开看，最容易知道问题出在哪一层。下面这张表把每一步读什么、写什么、什么时候会失效、错了会出现什么症状放在一起。

步骤 读什么 写什么 什么时候还有效 错了会怎样
Thought 任务目标、历史上下文、上一轮 Observation。 下一步判断、检索意图、是否继续行动。 当前证据还没变，或下一步计划仍然成立。 后面动作和证据脱节，轨迹像在自说自话。
Action 当前 Thought 和可用工具。 向工具、检索器或环境发出的请求。 动作确实对应当前缺口。 工具选错、参数错、重复调用同一无效动作。
Observation 工具输出、错误码、环境反馈。 下一轮能继续用的证据。 证据还没过期，也没被截断得太狠。 下一轮 Thought 没变化，循环断在这里。
Final Answer 前面累积的证据。 终局回答。 证据足够支撑当前结论。 答案流畅，但没有对应的证据链。

读表小窍门

如果你只能描述“最后答对了没有”，还不算会读 ReAct。能指出哪一步读错、写错、或者没有把观察接回下一轮，才算真正抓到了轨迹结构。

训练桥接

SFT / eval / rollout

ReAct 先是一种轨迹范式，再是训练素材。常见用法有三种：一是把轨迹当示范数据，训练模型学会下一步 Thought / Action；二是把轨迹当评测记录，拆出是想错、选错工具，还是没读 Observation；三是把轨迹当环境 rollout 的雏形，后面再接奖励和优化框架。

轨迹材料 能拿来做什么 需要保留什么 最常见的风险
SFT 示例 监督模型学下一步行动与中间判断。 任务、历史上下文、下一步 Thought / Action。 只学会表面格式，没学会观察驱动决策。
评测日志 拆分错误归因。 每一步动作、观察、工具结果、最终答案。 只看终局，漏掉中间一步的错因。
环境 rollout 接到更严格的训练闭环。 状态、动作、反馈、终止条件和奖励入口。 把 ReAct 直接当 RL 算法，忽略后续框架。
轨迹转示范的伪代码 
def to_sft_pairs(trajectory):
    pairs = []
    history = []
    for step in trajectory["steps"]:
        prompt = {
            "task": trajectory["task"],
            "history": history[:]
        }
        target = {
            "thought": step["thought"],
            "action": step["action"]
        }
        pairs.append((prompt, target))
        history.extend([
            {"thought": step["thought"]},
            {"action": step["action"]},
            {"observation": step["observation"]}
        ])
    return pairs

桥接时要记住的边界

ReAct 本身不是优化算法。它更像一条可读的轨迹模板。后面要不要用 SFT、偏好优化、RL，或者环境训练，是下一层的问题。

样本拆分

concrete chain

这一节把同一条 ReAct 轨迹拆成三种常见样本：监督微调样本、评测记录、环境 rollout 片段。这样做的意义是让你一眼看出，哪一层要保留什么，哪一层又该删掉什么。

同一条轨迹，三种读法

对 SFT 来说，重点是下一步 Thought / Action 该怎么写；对评测来说，重点是错误发生在哪一步；对 rollout 来说，重点是状态、动作、反馈和终止条件是否闭合。

最容易犯的错

很多人只保留最终答案，或者把整段轨迹原封不动塞进训练集。前者会丢掉关键监督，后者会把无关噪声也带进去。

样本类型 输入里保留什么 输出或标签保留什么 常见丢失点
SFT 样本 任务、历史上下文、上一步 Observation。 下一步 Thought / Action。 只保留最后答案，模型学不会中间判断。
评测记录 每一步 Thought、Action、Observation 和错误码。 正确与否、错误归因、回放路径。 只看终局，看不见是哪一步坏了。
rollout 片段 状态、动作、反馈、终止条件。 奖励入口、环境返回、下一状态。 把文本轨迹直接当状态序列，忽略环境语义。
同一轨迹的拆分伪代码 
def split_react_sample(trace):
    sft = []
    eval_rows = []
    rollout = []
    for i, step in enumerate(trace["steps"]):
        sft.append({
            "prompt": {
                "task": trace["task"],
                "history": trace["steps"][:i]
            },
            "target": {
                "thought": step["thought"],
                "action": step["action"]
            }
        })
        eval_rows.append({
            "step": i + 1,
            "thought": step["thought"],
            "action": step["action"],
            "observation": step["observation"]
        })
        rollout.append({
            "state": step["observation"],
            "action": step["action"],
            "next_state": trace["steps"][i + 1]["observation"] if i + 1 < len(trace["steps"]) else None
        })
    return {"sft": sft, "eval": eval_rows, "rollout": rollout}

怎么判断拆分对不对

如果你把任务描述、历史上下文和上一步观察都放进了 SFT 样本，模型就有机会学到“下一步怎么想”；如果你保留了每一步动作和观察，评测就能定位错在什么地方；如果你把状态变化也保留下来，rollout 才有继续往后推的意义。

常见失败

最常见的失败有三个：一是只保留最终答案，二是把 Observation 当成解释性文字，三是把文本轨迹误当成环境状态。修法也很直接：先确认每一步的读写对象，再决定数据该怎么切。

操作桥接

source to task

如果你要把 ReAct 用到真实任务里，第一步通常不是训练，而是把一条轨迹做成“可以回放、可以切分、可以定位失败”的数据。这样你才知道下一步是在做监督样本、评测样本，还是环境 rollout 片段。

一个真实点的客服场景

用户说“帮我查 INV-8831”。模型如果先补一个 user_id，再去查单，那条轨迹已经暴露了一个重要信号：它在往身份推断走，而不是在往证据检索走。这个信号对训练和评测都很值钱。

先修哪里

这时别急着改最终答案。先看 Thought 是否在表达缺口，Action 是否真的去取证，Observation 是否保留了下一轮能用的信息。轨迹结构先稳住，后面再谈策略。

任务侧对象 要保留什么 要删掉什么 为什么
SFT 输入 任务目标、历史上下文、上一轮 Observation。 与当前动作无关的长解释。 训练要学的是下一步怎么接，不是复述整段聊天。
评测记录 每一步动作、观察、错误码、最终结果。 对结果没有帮助的背景闲话。 评测要能定位“错在什么层”，不是只看最后对没对。
rollout 片段 状态、动作、反馈、终止条件、奖励入口。 不能映射到环境状态的文本装饰。 环境训练要能继续推进，不是把自然语言原样搬进去。
轨迹归一化伪代码 
def normalize_react_trace(trace):
    normalized = []
    for step in trace["steps"]:
        item = {
            "task": trace["task"],
            "thought": step["thought"].strip(),
            "action_name": step["action"]["tool"],
            "action_args": step["action"]["arguments"],
            "observation": step["observation"],
        }
        normalized.append(item)
    return {
        "samples": normalized,
        "has_final_answer": bool(trace.get("final_answer")),
        "can_be_sft": all("thought" in s and "action_name" in s for s in normalized)
    }

看这段伪代码时要盯住什么

它把一条自然语言轨迹拆成了三种可检查对象：任务、动作和观察。这样一来，监督微调、评测和 rollout 读到的是同一份事实，而不是三份互相打架的文本。

如果这一步做坏了

漏掉 Observation，模型会学会盲动；把所有日志都塞进去，评测会被噪声淹没；把用户自然语言直接当状态，rollout 就会失去环境语义。

这个桥接一旦做好，后面再读 ToolBench、API-Bank 和 OpenEnv，就能直接问：它是在补哪一段轨迹，是在改哪一种监督，还是在补哪一种状态反馈。

例题

worked cases

例 1：问答里怎么用检索

ReAct 论文里最直观的场景之一，是问答任务先查证据，再回答。模型先判断缺什么事实，再调用检索或维基 API，拿到观察后再收束答案。这样做的好处，是证据链能被人看到，也能被后续训练复用。

问答轨迹 
{
  "task": "ReAct 论文什么时候发表？",
  "steps": [
    {
      "thought": "我需要确认论文年份，先查论文标题。",
      "action": {
        "tool": "search_web",
        "arguments": { "query": "ReAct Synergizing Reasoning and Acting in Language Models" }
      },
      "observation": { "status": "ok", "year": 2022 }
    }
  ],
  "final_answer": "ReAct 论文发表于 2022 年。"
}

例 2：交互任务里的下一步

交互式任务里，Observation 不只是“查到了什么”，还包括环境状态有没有变化。比如购物任务里，模型先查价格和库存，再决定是否继续筛选；环境任务里，模型看到门没开，就会改动作，而不是照着上一轮的计划原封不动再试一次。

交互轨迹 
Thought: 我还缺尺寸信息，先筛选可选尺码。
Action: filter(size="M", color="black")
Observation: M 码库存不足，L 码可用。
Thought: 计划要改成 L 码，再继续看配送。

例 3：看起来像 ReAct，实际不是

坏轨迹 
Thought: 我应该查订单状态。
Action: get_weather(city="北京")
Observation: 北京晴。
Final Answer: 订单已经发货。

这条轨迹的病根很清楚：Thought 说的是订单，Action 调的是天气；Observation 也没有帮助回答订单问题。最终答案看上去顺，但中间没有一条线把它撑住。

常见错误

debug table

ReAct 的坑通常不在术语本身，而在轨迹有没有真的闭环。下面这张表按“问题样子 -> 为什么错 -> 下一步查什么”来读。

问题样子 为什么错 下一步查什么
只写 Thought，不写 Action / Observation。 这更像 chain-of-thought，不像 ReAct。 先补外部动作，再看观察是否进入下一轮。
Action 和 Thought 说的是两件事。 动作没有跟着当前缺口走。 先修工具选择和参数，再看是否需要重写 Thought。
Observation 由模型自己编。 外部校正丢了，轨迹失去意义。 把 Observation 重新绑定到工具或环境输出。
只评 Final Answer。 中间的想错、调错、读错都被藏起来了。 把每一步都拆开看，给错误找归属层。
观察到了新证据，下一轮计划没变。 轨迹停在日志层，没有进入更新层。 检查下一轮 Thought 是否真的吸收了 Observation。

故障台

terminal lab

要快速判断一个失败是轨迹问题还是工具问题，先切成三个最常见的 case：缺观察、错动作、旧证据。它们分别对应 ReAct 里三种典型断点。

loop_detected: observation missing, model repeated the last action
next check: action / observation boundary

缺观察

这类错误说明轨迹没闭环。先看工具有没有返回可用信息，再看运行时有没有把它接进下一轮。

错动作

这类错误说明计划和执行没对齐。先看 Thought 是否真的在引导动作，而不是只写了一个像样的句子。

旧证据

这类错误说明观察已经过期，但计划还没更新。先看停止条件和重试策略，再看是否需要换工具。

先知道，先别追

boundary

这页先不展开的内容有五个：完整 ReAct prompt、Toolformer 的数据生成流程、ToolBench 和 API-Bank 的 evaluator 细节、OpenEnv 的实现、以及更完整的环境 RL 优化框架。现在先把轨迹坐标记住，它们会在后面的单元重新出现。

如果你在读后面的论文时开始纠结“先用 SFT 还是先上 RL”，先回到这页，看看失败症状卡在哪一层。很多时候要先把轨迹读顺，训练方法才有意义。

再往后读时，记住一条最实用的线索：接口问题先看可解析性，轨迹问题先看观察有没有改变计划，环境问题先看状态、奖励和权限有没有闭环。

synthesisrelation

用三句话解释 Thought、Action、Observation 的关系。

脚手架：第一句讲 Thought，第二句讲 Action，第三句讲 Observation 怎样影响下一轮。

参考答案 / 评分要点

合格答案要把三者连成闭环：Thought 负责判断缺口，Action 负责向外部取证，Observation 负责把外部结果送回下一轮。若只是分别定义三个词，但没有说它们如何互相影响，答案还没抓住 ReAct 的主线。

轨迹诊断

练习从中间步骤定位问题，而不是只看最终答案。

答题顺序 先找断点，再说为什么断，最后给出最小修法。
error diagnosistrace

诊断这段轨迹的问题。

坏轨迹 
Thought: 我需要查订单状态。
Action: get_weather(city="北京")
Observation: 北京晴。
Final Answer: 订单已经发货。

至少指出两个问题，并说清楚该先修 Thought、Action 还是 Observation。

参考答案 / 评分要点

这段轨迹至少有两个问题：Action 选错工具，根本没有查订单；Observation 也没能支撑“订单已发货”。合格答案应先把错点归到行动层，再说如果继续往下做，需要补对的工具和新的 Observation，而不是直接改最后一句。

synthesisconstruct

为一个客服工具任务写 2 轮 ReAct 轨迹。

任务可以设为查订单、查物流或改地址。要写出两轮 Thought / Action / Observation 和最终回答。

参考答案 / 评分要点

合格答案要有两轮以上的闭环：第一轮先把缺口补出来，第二轮必须基于第一轮观察继续行动或直接收束。若第二轮和第一轮没有任何依赖关系，说明你写的是并列步骤，不是 ReAct。

它不是优化器，先把范式和算法分开。

comparisonrollout

比较 ReAct 和环境 RL：各写出 2 个相同点和 2 个不同点。

可以从状态、动作、反馈、奖励、采样和优化框架这几项里选。

参考答案 / 评分要点

相同点通常是都有状态、动作和反馈，也都关心多步决策。不同点在于：ReAct 主要是一种可读的轨迹组织方式，而环境 RL 还需要奖励函数、采样策略和优化框架；ReAct 里的观察通常是文本或工具结果，环境 RL 的反馈更像可定义的状态和奖励。

synthesisexplain

综合解释：把 ReAct 讲给刚学工具调用的人。

要求包含这 4 个词：Thought、Action、Observation、trace。

参考答案 / 评分要点

合格答案要把主线说顺：模型先用 Thought 判断缺什么，再用 Action 去外部取证，工具或环境回来的 Observation 再推着下一轮更新；trace 让这条链可回放、可诊断、可复用。若只讲“会调用工具”而没有说观察如何改变下一步，就还没讲到 ReAct 的重点。
