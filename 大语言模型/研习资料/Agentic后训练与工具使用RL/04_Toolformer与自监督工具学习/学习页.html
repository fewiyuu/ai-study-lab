Toolformer 与自监督工具学习

这一页把 Toolformer 当成一种“自监督工具调用数据生成法”来读：先用少量示例让模型提议候选 API 调用，再把真实执行后的结果接回语言模型，看这条调用是否真的让后文更容易预测。你先把这条筛选链路读顺，再去看后面的工具评测和环境训练，就不会把它们混成一类。

读完本页应该能做什么

  • 能把一条 Toolformer 候选样本拆成“示例、提议调用、执行结果、过滤判定”四步。
  • 能解释为什么它保留的是“能降低未来 token 损失”的调用，而不是“看起来像工具调用”的调用。
  • 能把 Toolformer 放回课程主线，说明它和 03 的轨迹读法、05 的任务评测、06 的环境训练分别解决什么问题。

学习目标

self-supervised tooling

拆出流程

能把一条 Toolformer 候选样本拆成少量示例、候选调用、执行结果和过滤判定，知道每一步在训练里做什么。

读懂筛选

能用“未来 token 损失是否下降”解释为什么某条调用被保留，为什么另一条看起来合法的调用会被丢掉。

接回课程主线

能把 Toolformer 和 03 的 ReAct 轨迹、05 的多步评测、06 的环境训练分开看，知道它在课程里解决的是哪一层问题。

本页边界

这页先不展开链式调用、交互式搜索、沙盒权限和环境奖励。先把“数据怎么来、怎么筛、怎么进入微调”读顺，再往后走。

先抓主线

source boundary

Toolformer 的核心想法很直接：让模型先在普通文本里提议候选 API 调用，再执行这些调用，把返回结果接回上下文，最后只保留那些真的让后文更容易预测的样本。这里的关键不是“会不会调用工具”，而是“这次调用有没有给语言模型带来可见的信息增益”。

为什么它重要

如果只看工具格式，模型很容易学会“像在用工具”；如果看未来 token 损失，过滤器会把真正有帮助的调用留下来。前者只是外形，后者才是训练信号。

别急着混淆的概念

先别把它和 ReAct、ToolBench、API-Bank、OpenEnv 混在一起。ReAct 负责轨迹读法，Toolformer 负责训练数据生成，后两者再把问题推向评测和环境闭环。

术语与边界

terms
术语 直觉解释 精确定义 常见误解
Demonstration 给模型看的少量工具示例。 放在提示里帮助模型学会 API 文本格式的少量上下文示例。 把它当成全量监督数据；Toolformer 只需要少量示例。
Candidate call 模型提议的一次工具调用。 在普通文本中采样出来、带有工具名和参数的候选 API 调用。 以为只要格式像调用，就自动算样本。
Filtering 筛掉没用的候选。 根据执行后的返回值是否降低未来 token 损失来决定保留与否。 把合法格式和有用样本混为一谈。
Future-token loss 后文是否更容易被预测。 比较是否看到返回值时的语言模型损失差异，用作工具有用性的信号。 以为这只是工具调用本身的打分，不看后文。
Self-supervised finetuning 用筛出来的调用继续训练模型。 把保留的样本重新写回语言模型训练格式，让模型学会何时调用、如何调用、如何利用结果。 把它误读成环境 RL 或在线交互。

数据流程

pipeline
1

给少量示例

先为每个 API 提供少数 demonstrations，让模型知道该工具的文本格式、参数形状和返回样子。

2

在文本里采样位置

模型在普通语料中寻找可能插入工具调用的位置，这一步不是人工手工标注出来的。

3

生成候选调用

模型写出工具名和参数，形成一条候选调用。格式不完整或明显出界的样本会先丢掉。

4

执行并接回结果

真实执行调用，把返回值插回上下文，再观察它是否帮助后文更容易被预测。

5

筛选并微调

只保留能降低未来 token 损失的调用，把它们交给标准语言模型微调。

这条流程和 ReAct 的区别很明显：ReAct 先让人读懂轨迹，Toolformer 先把可用的轨迹“生产”出来。前者更像轨迹读法，后者更像轨迹制造线。

筛选公式

loss filter
$$ \Delta L = \min(L_{\text{call only}}, L_{\text{call + result}}) - L_{\text{call + result}} $$

这里的 L 表示未来 token 的语言模型损失。若 \Delta L 足够大，说明工具返回值确实帮了后文预测；否则这条候选调用就不值得保留。

怎么读这个公式

先比较没有结果时的基线，再看加入结果后的损失。工具调用本身的格式不是重点，能不能减少后文不确定性才是重点。

为什么要看未来 token

因为“像工具调用”不等于“对训练有帮助”。把视线放到后文，过滤器才能筛掉空转的调用，留下真正提高信息量的样本。

对象 before after 读什么 写什么 / 改什么 出错症状
候选调用 只有少量文本上下文 带工具名和参数的调用片段 前文、示例、采样位置 生成 API 文本 格式不闭合、参数越界、调用位置不对
工具返回值 尚未执行 插回上下文的结果文本 候选调用本身 为后文补信息 结果和后文无关，损失没有下降
过滤器 所有候选样本 保留的训练样本 有无返回值时的损失差 筛掉无效调用 误保留噪声样本，模型学会空转调用

工具矩阵

mechanisms
工具 解决的短板 典型候选位置 训练时看什么 这一页要记住什么
Calculator 精确算术 数字、比例、换算、比较 结果是否让数值后文更容易预测 最容易看懂“结果真的有用”这件事
Question answering 事实性缺口 实体、定义、背景说明 答案是否降低后文不确定性 检索到的事实要能进入后文，而不是只停在工具输出里
Search 开放式证据查找 需要背景信息或额外证据的段落 返回结果能不能支撑后文 搜索不是万能，关键在于能否改善后文预测
Translation 语言理解辅助 外语短语、低资源表达 翻译结果是否帮助接下来续写 翻译不是最终目的，而是给语言模型补可读证据
Calendar 时间参照 日期、星期、时间相关句子 当前日期信息是否真的影响后文 模型不是在学“会查时间”，而是在学“什么时候需要时间”

例子

worked case

算术样本

文本里出现“84 杯拿铁，平均每天卖出……”时，模型可以提议一个计算器调用：Calculator(84 / 7) → 12。如果后文真的要写“平均每天 12 杯”，这个返回值就会显著减少后续 token 的不确定性，样本更可能被保留。

候选调用 
原文本：这家店一周卖出 84 杯拿铁，平均每天卖出……
候选：<API> Calculator(84 / 7) → 12 </API>
筛选点：如果后文要写出“12 杯”，返回值 12 会降低未来 token 损失。

搜索样本

如果文本缺的是背景事实，模型会提议检索类调用。这里的重点不是搜索结果看起来多完整，而是搜索结果是否真的帮助后文更自然地续写。没帮助，就该被过滤掉。

候选调用 
原文本：这篇论文发表于……
候选：<API> Search("Toolformer paper publication date") </API>
筛选点：如果检索结果让后文年份、作者或提交时间更容易预测，样本才值得保留。

这个例子要看什么

你真正要盯的是“后文有没有被结果帮助到”。如果后文并不使用这个返回值，或者返回值只是把一句本来就能写出的句子绕远了一点，这个样本就不该留下。

样本回放

shape tracer

如果你想真正看懂 Toolformer，不要只看“模型会不会调用工具”，要看一条候选样本从普通文本变成训练样本时，哪些东西被读走、哪些东西被写回、哪些东西被过滤掉。这个过程很像把一张没标注的卡片，按顺序补成“缺口、调用、结果、保留/丢弃”四个状态。只要这四个状态有一个没接上，后面就会出现两种典型错误：一种是样本合法但没收益，另一种是样本有收益却被误删。

先找缺口 before: 普通文本 + 未完成句子

这一轮先确认缺的是什么。如果句子里的缺口根本不是数值，而是事实、时间或定义，就不该把计算器硬塞进去。

生成候选 after: 结构化调用

模型把缺口转成候选调用。这里要检查的是工具名、参数和位置三件事，任何一个偏掉，后面的结果都很可能无效。

写回结果 after: result enters context

工具返回值不是备注，而是新的上下文证据。它必须真的进入后文，才能影响未来 token 的预测损失。

做过滤 after: retained sample

如果结果没有降低未来 token 损失，样本就丢掉；如果结果帮助了后文预测，样本才保留。这里保留下来的不是“工具调用格式”，而是“调用 + 结果 + 可用后文”。

读这条回放时要盯住什么

第一步看缺口是不是适合某个工具；第二步看调用参数是否真的对准缺口；第三步看返回值有没有进入后文；第四步再看过滤器有没有把样本留下。顺序不能倒。很多人一上来就看“格式对不对”，结果会把一堆无效调用误当训练样本。

最常见的两种失配

一种是“工具能跑，但后文根本没用上结果”，这类样本会在损失比较里被丢掉；另一种是“后文其实需要结果，但候选位置选错了”，这类样本看起来合法，实际上对上下文没有帮助。前者是过滤问题，后者是采样问题。

步骤 读什么 写什么 什么时候还有效 错了会怎样
缺口定位 普通文本、未完成句子、上下文语义 需要哪类工具的判断 缺口确实能被某个工具补上 工具和任务不匹配，候选调用从源头就偏了
候选生成 少量 demonstrations、工具格式、当前位置 工具名、参数、API 标记 工具名和参数都符合接口 格式合法但信息无关，或者参数越界
结果回写 工具返回值和后文句子 带结果的新上下文 返回值真被后文使用 返回值漂在上下文外，像没发生过一样
过滤决策 有无返回值时的未来 token 损失 保留或丢弃样本 结果确实让后文更容易预测 把空转样本留了下来，模型学会胡乱调用
候选过滤伪代码 
def should_keep(call, result, future_text, threshold):
    loss_without_result = lm_loss(call, future_text, visible_result=False)
    loss_with_result = lm_loss(call, future_text, visible_result=True)
    improvement = loss_without_result - loss_with_result
    if improvement >= threshold:
        return True, improvement
    return False, improvement

这段伪代码说明了过滤器真正看的对象：不是调用格式本身，而是“看到结果前后，后文损失差了多少”。如果你把改进值和工具调用次数混在一起看，就会误以为“多调用几次就更像 Toolformer”，实际上这是把训练目标看偏了。

故障台

terminal lab

一条 Toolformer 样本常见的失败，不在于它能不能执行，而在于执行之后能不能改变后文。下面把三个最常见的故障拆开：工具合法但没收益、参数看似正确但位置错、结果出现了但没有回到上下文。你可以把它们当成三个排查层。

filter: candidate call executed, but future-token loss did not improve
next check: does the result actually answer the local text gap?

合法但没收益

这类错误说明工具调用本身可以执行，但执行后并没有改善后文预测。先看返回值是不是补到了真正的缺口，再看是不是该换工具，而不是继续加调用次数。

参数正确但位置错

这类错误说明模型知道工具怎么写，却不知道应该插在哪个局部文本旁边。这里的问题更像采样位置，而不是接口格式；修法通常是回到候选提议那一步。

结果有了但没回写

这类错误最容易被忽略。工具实际上已经给出有用结果，但结果没有进入后文，于是下一轮还是在缺证据的地方打转。修法不是换工具，而是修结果回写和上下文拼接。

故障排查顺序 
1. 看候选位置是否和文本缺口对齐
2. 看工具参数是否匹配当前缺口
3. 看工具结果是否回写到后文
4. 看过滤器是否确实按损失差保留样本
5. 只在前四步都成立时，再判断是不是该换工具

把这条顺序记住，你就能把“样本看起来很像调用”与“样本真的能学到调用”分开。前者常常只是表面像，后者才是训练上值钱的东西。

课程对照

compare
03 ReAct

先教你读轨迹。重点是 Thought、Action、Observation 怎样闭环，以及哪一步断了。

04 Toolformer

先教你造数据。重点是候选调用怎么被采样、执行、过滤，再进入微调。

05 任务评测

再往后看多步任务怎么测。重点会转到参数正确性、步骤成功率和评测器设计。

一句话区分

ReAct 更像“可读轨迹模板”，Toolformer 更像“自监督数据生产器”，ToolBench 和 API-Bank 更像“多步任务评测器”，环境 RL 则把问题推进到状态和奖励闭环里。

为什么要这样分

因为每一层的错误都不一样：轨迹层错在读写闭环，数据层错在筛选标准，评测层错在指标和样本覆盖，环境层错在状态、奖励和副作用。

局限

limits
不擅长链式调用

候选调用是单次生成和单次筛选，训练里没有天然的多轮工具链，因此复杂组合任务覆盖有限。

不是交互式浏览

搜索或检索只是一次性结果，模型不能像人那样反复改写查询、翻页、对比多份来源。

不看调用成本

过滤器关注的是预测收益，不会自动把延迟、费用、失败率和副作用纳入决策。

对工具接口很敏感

工具名、参数格式和返回文本一变，样本分布就会变；这也是为什么 text-sequence 接口这么重要。

不解决权限与沙盒

论文关注的是学习调用能力，不是生产系统里的权限、审计和回滚。

样本保留率可能不高

很多候选调用最后会被过滤掉，说明“会提议”不等于“真有训练价值”。

这些局限会把你自然带到下一节：如果任务要求多步评测，就要看 ToolBench 和 API-Bank；如果任务要求环境里真的完成动作，就要看 OpenEnv 和环境 RL。

衔接下一步

bridge
下一步要看什么 这页已经准备了什么 下一步真正新增的东西 常见失败会长什么样
ToolBench / API-Bank 知道调用格式、过滤思路和可复用样本 多步任务评测、参数正确性和成功率指标 只会看单次调用，不会评估多步完成率
环境 RL 知道调用结果要回到上下文里 状态、奖励、终止条件和副作用控制 把文本调用直接当成环境交互，忽略真实状态
生产系统 知道样本要可执行、可筛选、可回放 权限、审计、回滚和成本控制 只会生成“看起来像调用”的文本，没法上线

最实用的判断句

当你看到一个工具学习方案，先问它是“造数据”、还是“做评测”、还是“做环境训练”。这一页讲的是前者，不是后两者。

把方法改到自己的工具上

operation bridge

如果你要把 Toolformer 的思路迁到自己的工具，第一步不是训练，而是先把工具压成稳定的文本接口。模型能生成候选调用，前提是工具名、参数和返回值都能被写成一句可回放的文本。只要接口太散，候选调用就会一会儿像参数表，一会儿像自然语言，最后过滤器再强也救不回来。

先做什么

先选一个返回值短、格式稳、能直接接回上下文的工具。比如汇率、计算器、短搜索结果或日期查询。然后写出几条 demonstrations，让模型知道工具名、参数顺序和返回值长什么样。最后才轮到候选采样和损失筛选。

先别做什么

先别把重型工具、长篇 API 返回和多轮浏览一起塞进去。Toolformer 这条路最怕接口过长、返回过碎、上下文回写过难。只要结果不能直接改变后文预测，样本就很难留下来。

改造步骤 要检查什么 通过时的样子 不过时的症状
接口文本化 工具名、参数、返回值能否稳定表示 调用格式像一句可回放的文本 接口字段太散，样本无法统一
缺口对齐 调用位置是否对准文本缺口 候选调用紧贴真正需要补证据的句子 调用离缺口很远，结果进不了后文
结果回写 返回值能否直接进入上下文 后文真的用上了工具返回值 结果停在工具输出里，像旁白
损失比较 返回值是否减少未来 token 的不确定性 有结果时的后文更容易预测 样本合法却没有训练收益
接口设计小样 
Calculator(expr) -> numeric result
Search(query) -> short evidence string
FX(amount, from, to, date) -> converted amount + rate

如果你自己的工具做不到“短、稳、可回写”，就先别急着想过滤器。

这段桥接的意思很实在：Toolformer 方法的门槛不是“有没有工具”，而是“能不能把工具压成一条稳定的文本轨迹”。一旦接口不稳，后面再强的筛选都只能筛掉大部分样本，剩下来的也未必好用。你现在应该能做的，不是直接训练，而是先检查工具接口、返回值长度和回写方式。

一个边界案例

case / practice hook

再看一个更接近真实工具的例子：文本里说“机票预算是 1200 元，想查今天能不能换成美元”，模型如果把这句话交给汇率工具，就必须先确认返回值够不够短，够不够稳定，能不能直接回写到句子里。这里的判断标准和计算器一样，都是看返回值有没有真的降低后文的不确定性；不同的是，汇率和日期这类工具更容易带进来噪声、格式波动和时间过期问题。

这个例子怎么走

先把“预算”和“换汇”拆开，再写候选调用；如果工具返回的是一长段说明而不是短结果，后文往往接不住。换句话说，候选调用本身可能没错，错的是它不适合被回写进这条上下文。

失败会长什么样

最常见的是三种：结果太长、时间过期、返回值和文本缺口不对应。只要其中一个出现，过滤器就会把样本打回去；如果你硬保留，模型学到的就不是“何时调用工具”，而是“如何把无关结果拼进句子里”。

场景 候选调用该长什么样 返回值通过时的样子 失败信号
换汇 FX(1200, CNY, USD, 2026-06-21) 短结果，能直接写回预算句 返回一大段说明，或者日期太旧
日期 Calendar(today) 一句稳定日期，足够进入后文 返回格式不一，模型不知道怎么接
短搜索 Search("Toolformer publication date") 一条简短事实，补到缺口里 结果太长、主题漂移、后文不用
边界判断小样 
if result_is_short and result_fits_gap and result_changes_future_tokens:
    keep_sample()
else:
    drop_sample()

这段边界案例真正想提醒你的是：Toolformer 不只是“把工具装进模型”，它还要求工具结果能在同一句话的局部缺口里落地。你如果把这一点忘了，后面的练习很容易只写出一个能跑的 API，而不是一个能训练的样本。现在应该能做的，是拿你手头任意一个工具，先判断它适不适合被写成一条短、稳、可回写的文本轨迹。

作答目标

不要只背“自监督”这个词。每题都尽量说出样本、执行、筛选和后续训练之间的关系。

检查重点

重点看你能否分清：哪一步是在造候选，哪一步是在过滤，哪一步是在进入微调，哪一步已经超出 Toolformer 的范围。

脚手架：第一句讲候选怎么来，第二句讲怎么执行，第三句讲怎么筛掉无效样本。

参考答案 / 评分要点

合格答案要把流程讲成闭环：模型先基于少量示例在普通文本中提议候选调用，工具真实执行后返回结果，再比较有无结果时的未来 token 损失，只有真正降低损失的样本才保留。若只说“模型学会了调用工具”，但没说筛选标准，答案还不完整。

损失与过滤

练习把“看起来对”改成“真的有训练价值”。

答题顺序 先判是否会降低未来 token 损失，再谈格式和工具名。
debug diagnosisfilter

诊断这条样本为什么可能被丢掉。

候选样本 
原文本：今天的天气不错，适合出门。
候选：<API> Calculator(2 + 2) → 4 </API>
后文：今天的天气不错，适合出门。

至少指出两个理由，并说清楚是哪个环节把它筛掉的。

参考答案 / 评分要点

这条样本大概率会被丢掉，因为返回值 4 并没有帮助后文预测“今天的天气不错，适合出门”。调用虽合法，但和上下文缺口无关，所以有无返回值时的未来 token 损失差异很小，过滤器会把它视为无效候选。

true/falseloss

判断：Toolformer 主要是在比较工具结果本身的打分。

fillmetric

填空：Toolformer 过滤时重点看的是未来 token 的 ________。

scenariofilter

给这个候选调用写一个保留或丢弃的判断。

候选：<API> Translation("bon voyage") → "一路顺风" </API>。后文只是在继续写天气，不再提旅行。

参考答案 / 评分要点

更合理的判断是丢弃：翻译结果没有帮助后文预测天气句子，因此有无工具结果对未来 token 损失的影响很小。若后文马上要写旅行或告别语，这条样本才可能保留。
