ToolBench / API-Bank 与任务评测

这一页把 ToolBench、API-Bank 和 StableToolBench 放在同一条线上看：先分清它们各自要测什么，再看多步工具任务的评测器到底读什么、写什么、漏掉什么。读完这一页，你应该能把“最终答案对了”与“工具路径真的对了”分开判断。

学习目标

multi-step

分清三类基准

ToolBench、API-Bank、StableToolBench 分别对应不同层面的工作：一个偏大规模工具学习框架，一个偏可运行任务评测，一个偏稳定性修复。

写出最小规约

能给一个多步工具任务写出最小成功条件、允许工具、稳定环境和复核方式，不把一个分数当成全部。

先抓主线

source

想判断一个模型会不会用工具，不能只看最终答案。比如“把明天 15:00 的会议改到 16:00”这类任务，模型可能给出一个看起来正确的结果，但中间没有查空闲、没有更新参会者、甚至改错了会议对象。对这种任务，评测要看的是路径：计划有没有读对，调用有没有对准，参数有没有填对，结果有没有真的落到最后一步。

这页和前后单元的关系

04 讲的是 Toolformer 怎么造工具调用数据；这一页讲的是多步任务怎么评测；06 才进入状态、奖励和环境训练。

术语与边界

terms
术语直觉解释这里要怎么用常见误读
ToolBench大规模工具学习和评测框架。用来把真实 API、数据构造、训练和评测放到同一条线上看。把它只当成一个 leaderboard 名称，忽略了数据和训练部分。
API-Bank可运行的工具增强 LLM 基准。重点看 planning、retrieval、calling 三类步骤怎样一起被评测。只盯最终答案，漏掉中间工具步骤。
StableToolBench让评测更稳定的后续基准。用虚拟 API server、缓存和 API 模拟器降低线上漂移。把线上 API 的临时变化误当成模型能力。
Runnable evaluation评测器能真实跑任务，同时检查路径和结果。适合看调用路径、参数和结果是否真的完成任务。以为只要文本像答案，就等于任务成功。
Task completion任务是否真的按要求完成。作为最终层判断，但不能替代路径和参数检查。把完成度和可读性混成一回事。

评测地图

benchmarks

这三条公开线索的区别，不在于名字多长，而在于它们分别把注意力放在不同层面。ToolBench 更像把工具学习的“数据、训练、评测”放在一张图里；API-Bank 更像把一个任务拆成可运行步骤；StableToolBench 则告诉你，评测如果不稳定，分数本身会漂。

一条实用判断

当你看到一个工具基准，先问它是在测“会不会写答案”，还是在测“会不会完成多步任务”，还是在测“同一任务能不能稳定比较”。

别把分数看成同一种分数

同样叫 pass rate，有的在测最终结果，有的在测路径，有的在测稳定性。口径不同，分数就不能直接互换。

评测流程

state

拿“会议改期”当例子：用户要把明天 15:00 的会议改到 16:00，模型需要先看日历，再找空闲，再改会议，再发通知。评测器如果只看最后一句“已改到 16:00”，就会漏掉改错会议、漏通知、参数填错这些问题。真正有用的评测，是把任务规格、计划、调用、观察和评分连起来看。

代码
def score_trace(task_ok, plan_ok, call_ok, args_ok, stable_ok):
 if not task_ok:
 return "fail: wrong task"
 if not plan_ok:
 return "fail: wrong plan"
 if not call_ok or not args_ok:
 return "fail: tool path mismatch"
 if not stable_ok:
 return "fail: unstable result"
 return "pass"

轨迹工作坊

trace

把一条多步工具任务真正看清，最稳的方法是把轨迹拆成表：每一步读什么、写什么、有没有推进任务、有没有把后文带偏。下面仍然用“会议改期”做锚点，但这次同时看最终结果和中间步骤，把每一步都落到可复核的状态上。你会发现，多步评测真正贵的地方，在于判断中间每一步是否都在往目标状态走。

这一张表真正教什么

它把“调用路径”从一个抽象词变成了可见对象。你现在可以逐行问：这一步读了什么、写了什么、有没有推进最终状态。如果某一步既没写入，也没推进任务，它就是多余动作，评测器不该给它加分。

最容易漏的点

很多轨迹看上去已经成功，但其实只做了状态的一半。改期任务里最常见的是：会议改了，通知没发；或者通知发了，会议对象改错了。路径层的评测就是专门抓这个。

轨迹复核清单

复制

1. 目标对象是否唯一
2. 每一步是否读到了当前状态
3. 调用参数是否指向同一个对象
4. 结果是否进入下一步
5. 最终状态是否满足全部约束

这条清单可以直接拿去做人工复核。只看最后一句话的判分，通常会把第 2 到第 4 步的错误都藏起来。真正有价值的评测器，应该能把错误定位到具体一步，而不是只给一个总分。

稳定性隔离

stability

StableToolBench 最重要的提醒是：工具任务的分数不只来自模型，也可能来自外部 API。一个真实线上接口如果今天返回空闲、明天返回满位，模型分数就会随着环境一起跳。评测要把会变化的部分隔离掉，让比较尽量只反映模型差异。

不稳定时会发生什么

同一条输入今天 pass、明天 fail，模型好像忽然变笨。其实往往是接口状态变了，或者缓存命中了不同结果。没有稳定层，这种噪声会直接污染基准分数。

层冻结前是什么冻结后是什么读什么写什么 / 改什么失败信号
API 状态真实线上服务，随时变化虚拟 API server 或固定快照同一请求的历史响应把不确定性关进可控环境同题多跑结果抖动
缓存可能命中不同路径固定缓存策略请求是否重复、返回是否一致避免前一次调用影响后一次前后两次跑出不同答案
返回格式线上服务可能临时改字段模拟器返回固定 schema字段名、顺序、类型统一解析入口评测器能跑，但解析结果变形
请求顺序并发或先后顺序不受控固定顺序重放上一条请求是否影响下一条减少偶然命中同一任务的轨迹不稳定

稳定评测配置样例

复制

benchmark:
 use_virtual_api_server: true
 freeze_api_state: true
 cache_responses: true
 deterministic_order: true
 compare_runs: 3

这段配置的意思很直接：先让环境稳定，再谈模型好坏。否则你看到的可能只是线上接口的波动，不是模型的实际能力。对工具任务来说，稳定性本身就是评测的一部分。

工具选择工作坊

case

多步工具评测里，最常见的误判是“模型会用工具，但工具选偏了”。比如用户说“看看我明天有没有空，然后把两点的会改到三点”，这里至少包含两种动作：先读状态，再写状态。若评测器只看最终有没有出现“已改到三点”，模型就可能先乱改，再回头补一个看起来合理的结论。下面这张小表就是用来抓这种偏差的。

这张表怎么用

先把任务短语圈出来，再判断它更像读、写，还是先读后写。这样做的好处是，你能在写 benchmark 规约时先分任务，再决定哪些步骤必须进总分。

最容易遮住的错误

如果模型先读错，再把最终结果写对，纯结果分数会放过它；如果模型先写对，再把中间观察丢掉，路径分数才能把它抓出来。多步评测最怕把“结论像对的”当成“过程真的对”。

选择规则草图

复制

def choose_tool(goal_text):
 if any(word in goal_text for word in ["看看", "查询", "检查", "有没有空"]):
 return "read_only_tool"
 if any(word in goal_text for word in ["改", "更新", "取消", "预订"]):
 return "mutating_tool"
 if any(word in goal_text for word in ["证据", "条件", "理由"]):
 return "retrieve_then_decide"
 return "ask_for_disambiguation"

这段草图提醒你写评测时先定义动作类型。只要读写不分，模型就可能把查询当修改，把修改当确认，最后分数看起来还行，实际任务却没完成。

下次你读任何一个多步工具 benchmark，先把题面里的动作标出来，再问自己：这题是在测读、测写，还是测读后写。这个问题一旦想清楚，成功条件和失败标签就好写很多。

指标怎么读

metrics

一个分数不够。多步工具任务至少要分开看四件事：任务有没有完成，工具选没选对，参数填没填对，结果在重复运行里稳不稳。只看 pass rate，最容易把“路径错但结果碰巧对了”的样本放过去，也最容易把“工具正确但环境抖动”误算成模型变差。

怎么读这些指标

先看完成度，再看路径，再看稳定性。顺序反过来，模型很容易被一个漂亮分数骗过去。

为什么要分层

如果工具选择、参数正确和稳定性都混在一个总分里，你很难知道问题是模型不会用工具，还是环境本身在波动。

任务桥接

operation

如果你要复现一个公开 benchmark，或者给自己的工具任务做评测，先别写分数公式。先把任务分布、允许工具、成功条件、稳定环境和复核方式固定下来。只要这五样不固定，后面的分数很容易只是当时环境的截屏。

代码

allowed_tools:
 - calendar.lookup
 - calendar.update
 - notification.send
success:
 - target slot is free
 - meeting moved to the target slot
 - attendees are notified
stability:
 - use a frozen fixture
 - rerun the same prompt three times
review:
 - automatic check for task completion
 - manual check for path mismatch and leakage
