# 学习目标 {#goals}
<!-- tag: multi-step evaluation -->

分清三类基准
ToolBench、API-Bank、StableToolBench 分别对应不同层面的工作：一个偏大规模工具学习框架，一个偏可运行任务评测，一个偏稳定性修复。


读懂评测链路
能把任务规格、计划、调用、观察、复核这几层按顺序读出来，知道评测器在哪一层开始发力。


写出最小规约
能给一个多步工具任务写出最小成功条件、允许工具、稳定环境和复核方式，不把一个分数当成全部。

这页先讲评测，不讲环境强化学习本身。环境、奖励和沙盒边界放到后面的单元再追。

# 先抓主线 {#overview}
<!-- tag: source boundary -->

想判断一个模型会不会用工具，不能只看最终答案。比如“把明天 15:00 的会议改到 16:00”这类任务，模型可能给出一个看起来正确的结果，但中间没有查空闲、没有更新参会者、甚至改错了会议对象。对这种任务，评测要看的是路径：计划有没有读对，调用有没有对准，参数有没有填对，结果有没有真的落到最后一步。

## box info
### 为什么要把路径算进去
如果只看最终文本，模型很容易学会绕开工具；如果把调用路径、参数和执行结果一起看，模型才会被迫学会真正完成任务。

## box warn
### 这页和前后单元的关系
04 讲的是 Toolformer 怎么造工具调用数据；这一页讲的是多步任务怎么评测；06 才进入状态、奖励和环境训练。

# 术语与边界 {#terms}
<!-- tag: terms -->

术语
直觉解释
这里要怎么用
常见误读


ToolBench
大规模工具学习和评测框架。
用来把真实 API、数据构造、训练和评测放到同一条线上看。
把它只当成一个 leaderboard 名称，忽略了数据和训练部分。


API-Bank
可运行的工具增强 LLM 基准。
重点看 planning、retrieval、calling 三类步骤怎样一起被评测。
只盯最终答案，漏掉中间工具步骤。


StableToolBench
让评测更稳定的后续基准。
用虚拟 API server、缓存和 API 模拟器降低线上漂移。
把线上 API 的临时变化误当成模型能力。


Runnable evaluation
评测器能真实跑任务，同时检查路径和结果。
适合看调用路径、参数和结果是否真的完成任务。
以为只要文本像答案，就等于任务成功。


Task completion
任务是否真的按要求完成。
作为最终层判断，但不能替代路径和参数检查。
把完成度和可读性混成一回事。

# 评测地图 {#benchmark-map}
<!-- tag: benchmarks -->

这三条公开线索的区别，不在于名字多长，而在于它们分别把注意力放在不同层面。ToolBench 更像把工具学习的“数据、训练、评测”放在一张图里；API-Bank 更像把一个任务拆成可运行步骤；StableToolBench 则告诉你，评测如果不稳定，分数本身会漂。

基准
主要解决什么
评测形态
最该盯住什么
容易漂的地方


ToolBench / ToolLLM
把大规模真实 API 的工具学习和评测放进同一框架。
看任务、调用和训练数据怎么连起来。
工具库够不够大，调用链是否能覆盖真实场景。
如果只看最终分数，会忽略数据构造和调用路径。


API-Bank
把工具增强 LLM 的能力拆成可运行的任务。
看 planning、retrieval、calling 三层是否都对。
每一步是否真的执行，还是只是写得像执行。
如果任务环境不固定，结果会掺进外部波动。


StableToolBench
把线上 API 不稳定的问题从评测里隔离出去。
用虚拟 API server、缓存和模拟器做更稳定的比较。
同一条输入在多次运行里能不能得到一致、可比的结果。
如果不用稳定环境，模型分数可能只是当天 API 的状态。

## box info
### 一条实用判断
当你看到一个工具基准，先问它是在测“会不会写答案”，还是在测“会不会完成多步任务”，还是在测“同一任务能不能稳定比较”。

## box warn
### 别把分数看成同一种分数
同样叫 pass rate，有的在测最终结果，有的在测路径，有的在测稳定性。口径不同，分数就不能直接互换。

# 评测流程 {#eval-pipeline}
<!-- tag: state ledger -->

拿“会议改期”当例子：用户要把明天 15:00 的会议改到 16:00，模型需要先看日历，再找空闲，再改会议，再发通知。评测器如果只看最后一句“已改到 16:00”，就会漏掉改错会议、漏通知、参数填错这些问题。真正有用的评测，是把任务规格、计划、调用、观察和评分连起来看。

环节
读什么
写什么 / 改什么
出错症状
下一步检查


任务规格
目标、约束、允许工具
评测边界
模型学到的目标和任务真实目标不一致
先核对 success condition


计划
任务规格和工具清单
步骤序列
最终答案碰巧对了，但路径不对
看 planning 是否拆对


调用
API schema 和当前上下文
工具名、参数、对象
格式能跑，语义不对
检查参数是否指向正确对象


观察
工具返回值
新的上下文证据
结果出现了，但后文没有用
看回写与后续动作是否一致


评分
完整轨迹和成功条件
分数和失败标签
一个分数盖住了路径错误
拆成完成度、路径和稳定性三层

评测器伪代码
复制

```python
def score_trace(task_ok, plan_ok, call_ok, args_ok, stable_ok):
 if not task_ok:
 return "fail: wrong task"
 if not plan_ok:
 return "fail: wrong plan"
 if not call_ok or not args_ok:
 return "fail: tool path mismatch"
 if not stable_ok:
 return "fail: unstable result"
 return "pass"
```

# 轨迹工作坊 {#trace-workshop}
<!-- tag: trace table -->

把一条多步工具任务真正看清，最稳的方法是把轨迹拆成表：每一步读什么、写什么、有没有推进任务、有没有把后文带偏。下面仍然用“会议改期”做锚点，但这次同时看最终结果和中间步骤，把每一步都落到可复核的状态上。你会发现，多步评测真正贵的地方，在于判断中间每一步是否都在往目标状态走。

步骤
输入 / 读取
输出 / 写入
评测器看什么
失败会落在哪里


确认任务
“明天 15:00 改到 16:00”
任务边界、目标时间、对象范围
是否把目标状态锁对
任务定义漂移，后面所有调用都在错目标上跑


查空闲
日历里的候选时间段
可用时间列表
有没有真的查，还是直接猜一个时间
模型跳过检索，直接给出自以为正确的时间


选时间
可用时间列表 + 会议约束
16:00 这一条候选
是否选到了符合约束的 slot
参数对不上对象，或者把冲突时间也选进来


更新会议
会议 id、目标时间、参会者
新的会议状态
状态有没有真的被写入
只改了文本，没有改到真实对象


通知参会者
更新后的会议状态
通知记录
任务要求有没有包含这一步
前面做对了，最后漏通知

## box info
### 这一张表真正教什么
它把“调用路径”从一个抽象词变成了可见对象。你现在可以逐行问：这一步读了什么、写了什么、有没有推进最终状态。如果某一步既没写入，也没推进任务，它就是多余动作，评测器不该给它加分。

## box warn
### 最容易漏的点
很多轨迹看上去已经成功，但其实只做了状态的一半。改期任务里最常见的是：会议改了，通知没发；或者通知发了，会议对象改错了。路径层的评测就是专门抓这个。

轨迹复核清单
复制

```text
1. 目标对象是否唯一
2. 每一步是否读到了当前状态
3. 调用参数是否指向同一个对象
4. 结果是否进入下一步
5. 最终状态是否满足全部约束
```

这条清单可以直接拿去做人工复核。只看最后一句话的判分，通常会把第 2 到第 4 步的错误都藏起来。真正有价值的评测器，应该能把错误定位到具体一步，而不是只给一个总分。

# 稳定性隔离 {#stability-lab}
<!-- tag: stability lab -->

StableToolBench 最重要的提醒是：工具任务的分数不只来自模型，也可能来自外部 API。一个真实线上接口如果今天返回空闲、明天返回满位，模型分数就会随着环境一起跳。评测要把会变化的部分隔离掉，让比较尽量只反映模型差异。

## box info
### 稳定环境要隔离什么
最先隔离的是 API 状态和缓存，其次是请求顺序和返回格式。把这几层冻住以后，评测器才知道自己在比模型，也知道哪些波动来自当天线上服务。

## box warn
### 不稳定时会发生什么
同一条输入今天 pass、明天 fail，模型好像忽然变笨。其实往往是接口状态变了，或者缓存命中了不同结果。没有稳定层，这种噪声会直接污染基准分数。

层
冻结前是什么
冻结后是什么
读什么
写什么 / 改什么
失败信号


API 状态
真实线上服务，随时变化
虚拟 API server 或固定快照
同一请求的历史响应
把不确定性关进可控环境
同题多跑结果抖动


缓存
可能命中不同路径
固定缓存策略
请求是否重复、返回是否一致
避免前一次调用影响后一次
前后两次跑出不同答案


返回格式
线上服务可能临时改字段
模拟器返回固定 schema
字段名、顺序、类型
统一解析入口
评测器能跑，但解析结果变形


请求顺序
并发或先后顺序不受控
固定顺序重放
上一条请求是否影响下一条
减少偶然命中
同一任务的轨迹不稳定

稳定评测配置样例
复制

```text
benchmark:
 use_virtual_api_server: true
 freeze_api_state: true
 cache_responses: true
 deterministic_order: true
 compare_runs: 3
```

这段配置的意思很直接：先让环境稳定，再谈模型好坏。否则你看到的可能只是线上接口的波动，不是模型的实际能力。对工具任务来说，稳定性本身就是评测的一部分。

# 工具选择工作坊 {#selection-workshop}
<!-- tag: case -> trace -> failure -->

多步工具评测里，最常见的误判是“模型会用工具，但工具选偏了”。比如用户说“看看我明天有没有空，然后把两点的会改到三点”，这里至少包含两种动作：先读状态，再写状态。若评测器只看最终有没有出现“已改到三点”，模型就可能先乱改，再回头补一个看起来合理的结论。下面这张小表就是用来抓这种偏差的。

任务片段
更像哪类工具
先读什么
会写什么
错在哪一步最容易被遮住


“看看我明天有没有空”
read / retrieve
日历状态、空闲段
只输出观察结果
如果把查询当更新，后文会把空闲当成已修改


“把两点的会改到三点”
write / update
原会议对象、目标时间、冲突约束
新的会议状态
如果对象没对齐，模型最后一句再像样也没用


“先查证据，再决定是否退款”
retrieve + decide
订单、条件、规则
判定动作
如果跳过检索，后续决策会变成猜测


“把说明翻译成中文再继续写”
transform / read-then-write
原文与术语
可回写的短结果
如果结果太长，后文接不住，路径就断了

## box info
### 这张表怎么用
先把任务短语圈出来，再判断它更像读、写，还是先读后写。这样做的好处是，你能在写 benchmark 规约时先分任务，再决定哪些步骤必须进总分。

## box warn
### 最容易遮住的错误
如果模型先读错，再把最终结果写对，纯结果分数会放过它；如果模型先写对，再把中间观察丢掉，路径分数才能把它抓出来。多步评测最怕把“结论像对的”当成“过程真的对”。

选择规则草图
复制

```python
def choose_tool(goal_text):
 if any(word in goal_text for word in ["看看", "查询", "检查", "有没有空"]):
 return "read_only_tool"
 if any(word in goal_text for word in ["改", "更新", "取消", "预订"]):
 return "mutating_tool"
 if any(word in goal_text for word in ["证据", "条件", "理由"]):
 return "retrieve_then_decide"
 return "ask_for_disambiguation"
```

这段草图提醒你写评测时先定义动作类型。只要读写不分，模型就可能把查询当修改，把修改当确认，最后分数看起来还行，实际任务却没完成。

下次你读任何一个多步工具 benchmark，先把题面里的动作标出来，再问自己：这题是在测读、测写，还是测读后写。这个问题一旦想清楚，成功条件和失败标签就好写很多。

# 指标怎么读 {#metrics}
<!-- tag: metrics -->

一个分数不够。多步工具任务至少要分开看四件事：任务有没有完成，工具选没选对，参数填没填对，结果在重复运行里稳不稳。只看 pass rate，最容易把“路径错但结果碰巧对了”的样本放过去，也最容易把“工具正确但环境抖动”误算成模型变差。

指标
读什么
最能抓到的失败
最容易漏掉什么


任务完成率
最终状态是否符合要求
任务根本没做完
路径错误但结果碰巧正确


工具选择准确率
有没有选对 API
调用错工具
选对工具但参数还错


参数正确率
参数、对象、时间点是否对齐
把会议改错对象，或者时间填错
返回值没有被后文使用


步骤覆盖率
计划里的关键步骤是否真的执行
漏掉查询、确认或通知步骤
步骤看起来很多，但都没真正推进任务


稳定性
同一任务多次运行是否一致
API 漂移、缓存差异、环境噪声
模型本身的路径问题


成本 / 延迟
调用是否过多、过慢、过贵
空转、反复调用、慢路径
任务本身有没有完成

## box info
### 怎么读这些指标
先看完成度，再看路径，再看稳定性。顺序反过来，模型很容易被一个漂亮分数骗过去。

## box warn
### 为什么要分层
如果工具选择、参数正确和稳定性都混在一个总分里，你很难知道问题是模型不会用工具，还是环境本身在波动。

# 任务桥接 {#operation-bridge}
<!-- tag: operation bridge -->

如果你要复现一个公开 benchmark，或者给自己的工具任务做评测，先别写分数公式。先把任务分布、允许工具、成功条件、稳定环境和复核方式固定下来。只要这五样不固定，后面的分数很容易只是当时环境的截屏。

步骤
要固定什么
通过时长什么样
不过时长什么样


任务分布
任务类型、难度、工具覆盖面
不同任务之间可比
样本偏到某一类任务


工具白名单
允许调用哪些 API
模型知道边界，不会乱试
评测集和训练集工具混在一起


成功条件
最终状态、必要步骤、不能漏掉的动作
路径错误能被抓出来
只看最终一句话


稳定环境
fixture、缓存、API 状态
同题多次运行可比
今天的分数和明天不是一回事


复核方式
自动判分还是人工复核
知道哪里需要人看
把所有判断都压成一个数字

最小 benchmark 规约
复制

### 6 synthesis

**Q:** 用三句话解释为什么只看 pass rate 不够。

Explain: 合格答案至少要说出：pass rate 只能看到任务有没有做完，看不到中间有没有选对工具、填对参数、走对步骤。它也看不出同一任务在多次运行里是否稳定，所以会把 API 漂移、缓存差异和路径错误藏起来。若答案只说“会误导”，但没有举出路径或稳定性问题，说明还没读到评测层。

### 10 open

**Q:** 为会议改期任务写最小成功条件。

Explain: 合格答案应该同时覆盖最终状态和关键动作。例如：会议必须改到指定时间；改动必须作用在正确的会议对象上；如果任务要求通知参会者，还必须把通知发出去。若只写最终时间，没写对象和通知，说明成功条件还不完整。

### 11 choice

**Q:** 设计自己的 benchmark 时，最先冻结哪一层更有用？

- [ ] . 任务分布和稳定环境
- [ ] . 页面配色
- [ ] . 最终答案的长度

Answer: . 任务分布和稳定环境

### 12 synthesis

**Q:** 给一个最小 benchmark 规约写四项。

Explain: 一个合格的最小规约，至少要有：任务分布、允许工具、成功条件和稳定环境。更好的答案还会补上复核方式和泄漏边界。只要这些要素固定了，评测才不会把任务完成、路径错误和环境漂移混成一团。
