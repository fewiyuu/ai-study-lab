从 SFT 到环境强化学习

这一页把 05 的评测视角往前推一步：当任务的关键不再只是“答案对不对”，而是“上一步动作会不会改变下一步观察”，就该把工具任务从静态示范换成环境交互。读完后，你应该能分清什么时候继续做 SFT，什么时候改成环境训练，以及为什么环境类和 reward_func 必须按 episode 来写。

学习目标

route

先识别切换信号

当任务结果取决于上一步留下的状态、权限或中间产物时，就不能只拿完成答案来训练。

先写出最小骨架

你要能把一个工具任务改成环境类、公开工具方法和 reward_func 三部分，而不是把逻辑全塞进一段 rollout 代码里。

先抓主线

big

SFT 看到的是已经写好的样本。它擅长学格式、学常见路径、学专家当时怎么回答，但它看不到模型自己的动作会不会改掉下一步的观察结果。只要任务还是“输出一段像样的回答”，SFT 往往够用；一旦任务变成“先动一下，再看环境怎么变，再决定下一步”，SFT 就少了最关键的反馈。

TRL 的文档把这件事分得很清楚：SFTTrainer 适合 language modeling 或 prompt-completion，GRPOTrainer 的基础输入是 prompt-only；当任务需要多轮交互时，OpenEnv 把环境变成一个带状态的对象，GRPOTrainer 通过 environment_factory 把它接进训练环。这样，模型学的就连续动作后的结果。

本页只讲切换条件和最小骨架。你在这里要学会判断：是继续补示范样本，还是把任务改成环境；是把奖励放在离线评测里，还是放进 episode 末尾读回来的环境状态里。

训练阶段怎么选

信号先做什么先不要做什么
格式稳定，结果不依赖后续状态继续用 SFT，先把常见路径和格式学稳。不要为了“更高级”直接上环境训练。
下一步观察依赖上一步动作改成环境训练，让 observation 进入闭环。不要只拿最终答案做监督。
只有偏好分数，没有交互环境先做离线偏好或评测，再考虑是否能补环境。不要把静态分数假装成 episode 奖励。
动作可能触达真实对象先留在评测和沙盒，不要直接让模型探索生产系统。不要把训练和上线混在同一套工具里。

术语与边界

terms

SFT

从完成好的样本里学下一步该写什么。它看得见答案，不看见动作改变了什么。

Observation

环境返回给模型的输入，可能是文本、状态摘要或工具结果。它来自环境，不来自静态样本。

Reward

episode 结束后读取的反馈。它可以存在环境实例里，再由 reward_func(environments, **kwargs) 统一读出。

术语在代码里看哪里误读后会出什么问题
SFTprompt / completion 或 text。把静态示范误当成状态训练。
Episodereset 到终止之间的一轮交互。状态没清干净，上一轮结果泄漏到下一轮。
Observation环境方法返回的内容。把模型自己的输出当成环境反馈。
Reward环境实例上的 reward 或相似字段。只读文本 heuristics，reward hacking 很快出现。
Prompt-onlyGRPOTrainer 的基础输入形态。把 preference 数据直接塞给环境训练。

SFT 为什么会卡住

cutover

拿“给客户写跟进邮件草稿”做例子。SFT 可以让模型学会语气、结构和常见措辞，但它只看到“最后一版草稿长什么样”。如果客户有没有回复、纪要里有没有缺失信息、草稿是否经过审批，都会改变下一步应该怎么做，那么单次示范就不够了。

环境训练的价值就在这里：模型先做一个动作，环境返回新的观察，训练信号再从这条轨迹里读出来。你关心的不只是“写得像不像”，还包括“是否清空了旧状态”“是否把错误恢复做对”“是否把风险动作拦在 episode 里”。

这也是为什么这页和 05 的分界线很重要。05 关注评测，回答“做完以后好不好”；06 关注训练，回答“做的时候怎么从结果里学到东西”。

同一任务的两种样本形态

形态样本长什么样看不见什么
SFTprompt + completion，或者一段整理好的对话。看不见动作之后状态怎么变。
环境 RLprompt → action → observation → reward。不再只依赖已写好的答案文本。
离线偏好两个候选结果加比较信号。看不见自己在环境里的交互轨迹。

一轮 episode 的状态台账

state

这张表是本页最核心的检查点。每一步都要回答四件事：读什么、写什么、什么时候会变脏、错了以后会出现什么症状。只要这四列对齐，环境训练就不容易写成“看起来像 RL”的样子。

两轮 episode 的差别

trace

同一个“给客户写跟进邮件草稿”的任务，第一轮和第二轮看起来只差一步，实际差的是环境有没有被清干净。下面这张表把两轮 episode 摊开：第一轮可以正常计分，第二轮如果沿用共享实例，reward 会带着上一轮状态跑。

轮次初始观察动作读回的状态如果写错会怎样
episode 1请基于会议纪要写跟进邮件草稿，不要直接发送。draft_reply(tone="polite", length="short")reward = 1.0，done = False如果 reward 直接读文本，可能被“polite”这个词骗过。
episode 2同样的任务，但环境应该重新初始化。reset() 后再发动作只读这轮的状态，不继承上一轮分数。如果 reset 没清零，第二轮的奖励会从旧状态开始。

这张表该怎么用

先看起点：reset 返回的观察是不是当前任务。再看过程：工具方法有没有改写环境状态。最后看终点：reward_func 读回来的值是不是只属于这一轮。只要其中一项对不上，训练信号就会偏。

  • 起点错了，后面的轨迹都跟着错。
  • 过程错了，模型学不到动作和状态的对应关系。
  • 终点错了，reward 曲线会比真实任务更乐观。

把一轮训练摊开看

trace

下面这条轨迹把一轮训练压成五步。左边是你在代码里会看到的入口，右边是这一步的状态形态。你切换时，要特别留意 SFT 样本和 environment observation 的区别：前者是静态示范，后者是动作之后才出现的反馈。

奖励从哪里来

bridge

OpenEnv 文档里的最小例子是 Echo：环境执行一次动作，环境对象里记录 reward，reward_func 再从 environments 列表里读出来。这个结构放到别的任务里也一样：你先把环境状态写进去，再把 episode 的结果统一读回来。

下面这段是教学版的邮件草稿环境。它不是生产代码，但它保留了 docs 里的骨架：reset 清状态，公开方法当工具，reward_func 只读环境实例。这样的结构比把所有逻辑塞进 rollout 循环里更容易查错。

环境训练最小骨架python
class MailDraftEnv:
 def __init__(self):
 self.reward = 0.0
 self.done = False
 self.last_observation = ""

 def reset(self, task: str | None = None, audience: str | None = None, **kwargs):
 """Start a new episode."""
 self.reward = 0.0
 self.done = False
 self.task = task or "写跟进邮件草稿"
 self.audience = audience or "客户"
 self.last_observation = "请基于会议纪要写跟进邮件草稿，不要直接发送。"
 return self.last_observation

 def draft_reply(self, tone: str, length: str) -> str:
 """
 Draft a reply for the user.
 Args:
 tone: 邮件语气
 length: 邮件长度
 Returns:
 草稿文本
 """
 self.last_observation = f"草稿已更新：{tone}/{length}"
 self.reward = 1.0 if tone in {"polite", "neutral"} else 0.4
 return self.last_observation

 def submit_for_review(self) -> str:
 """Finish the episode."""
 self.done = True
 return "submitted for review"


def reward_func(environments, **kwargs):
 return [env.reward for env in environments]

职责分工

对象负责什么最常见 bug
environment_factory给每次生成创建一个新的环境实例。传了共享实例，episode 互相污染。
reset清状态，装载当前样本，返回初始观察。忘记清零 reward 或 done。
公开工具方法让模型发出可理解的动作。方法名过泛、参数没类型、docstring 不完整。
reward_funcepisode 结束后统一读出奖励。直接看文本，导致奖励与真实状态脱钩。

在 OpenEnv 文档里，工具必须是带类型提示和说明的公开方法。step(action) 这种泛型入口对模型来说太空，不利于它学会选工具。

同一个任务，两种训练视角

example

还是用“给客户写跟进邮件草稿”这件事。SFT 版本里，你会把一条完整示范写成 prompt-completion 或对话样本。环境版本里，模型先看初始任务，再发动作去生成草稿，环境根据草稿质量、语气、是否尊重限制来给反馈。两者看上去都在“写邮件”，实际学到的是不同东西。

环节SFT 版本环境版本
起点一条整理好的示范。reset 返回初始观察。
中间变化看不到。公开工具方法会改写环境状态。
失败信号答案不像样或格式错。状态没清零、动作不合规、reward 读错源。
更新依据示范文本和标签。episode 末尾读回来的奖励和观察。

先走哪条路

decision

这一步不要凭感觉。先问三个问题：任务有没有持续状态，下一步观察会不会被上一步动作改变，风险能不能在训练环境里隔离。只要其中任意一项答不清，就先不要把它当成纯 SFT 问题。

TRL 的文档把这个分界说得很直接：GRPOTrainer 可以做 agent 训练，但环境模式和工具模式解决的问题不同。前者要状态连续，后者更多是单次函数调用。不要把这两个入口混成一件事。

最常见的失误

failure
错误症状先查哪里怎么改
environment_factory=shared_env不同 episode 之间互相污染。看是不是每次生成都创建了新环境。传类或工厂，不要传共享实例。
reset 不清零 reward第二轮分数一上来就不对。看 reward、done、缓存有没有被重置。每轮开始都把 episode 状态清干净。
reward 只看文本分数会被格式或关键词骗过。看 reward 是否读到了真实环境状态。让 reward 来源和任务状态对齐。
工具方法叫 step(action)模型不知道该选哪一个动作。看工具名字和参数是否具体。改成带语义的公开方法名，并补 typed args 和 docstring。

这里先不追

boundary

先把这几类细节放一边：OpenEnv 的容器部署、多 GPU rollout、异步 reward、reward shaping、上线审批。它们都重要，但不影响你先把一轮环境 episode 读写接对。

这些内容会在后面的 07 和 08 里回来。现在你只需要记住：能不能训练，先看状态有没有隔离；训练会不会学歪，再看 reward 有没有从真实 episode 里读回来。

故障台：一条最常见的错误链

failure

错误链

坏写法python
shared_env = MailDraftEnv()

trainer = GRPOTrainer(
 model=model,
 train_dataset=dataset,
 reward_funcs=reward_func,
 environment_factory=shared_env,
)

def reward_func(environments, **kwargs):
 return [1.0 if "polite" in env.last_observation else 0.0 for env in environments]

这段代码同时犯了两个错：environment_factory 传的是共享实例，不是工厂；reward_func 读的是文本片段，不是 episode 里真正的状态。

怎么修

先修哪里修复动作为什么先修它
environment_factory改成类或工厂，确保每次 rollout 拿到新实例。先把 episode 隔离起来，避免状态交叉污染。
reset清空 reward、done 和临时缓存，再返回初始观察。否则 reward 会沿用上一轮残留。
reward_func从环境实例里读出 episode 结果，而不是从文本 heuristics 读分。这样训练信号才真正和任务目标对齐。

如果这三处任意一处没修好，模型都会看起来“会做题”，但实际上是在学错误的奖励来源。最常见的表面现象，就是 reward 曲线升了，真实任务成功率却没有跟着上去。

这条故障链和本页前面的状态台账一一对应。你只要能把错误写法、症状和修复动作串起来，环境训练的基本排错就算过关了。

Q: 12. 把“浏览网页找信息”改写成环境接口。

参考答案 / 评分要点

。合格答案应该把接口拆成五件事： reset 返回初始观察， 工具方法 负责打开页面、点击、搜索等动作， 状态 记录当前页、已看内容和未完成任务， reward 读任务完成度和合规性， done 说明什么时候 episode 结束。只列函数名不够，必须把状态和奖励的来源说清楚。

14synthesis

Q: 14. 综合解释：用 120-180 字讲清 SFT 到环境 RL 的切换。

参考答案 / 评分要点

。合格答案要把切换条件说清楚：SFT 适合学静态样本和格式，GRPO / 环境训练适合学状态连续任务； reset 负责清空旧 episode 并返回初始观察，模型做出 action 后得到新的 observation， reward_func 再从环境实例里读出奖励。最后要点出：当任务的成败取决于连续交互，而不是单次答案时，才需要把训练改成 environment RL。
