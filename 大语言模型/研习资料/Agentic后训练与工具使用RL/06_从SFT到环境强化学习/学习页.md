# 先抓主线 {#overview}
<!-- tag: big picture -->
<!-- layout: grid2 -->

## box
SFT 看到的是已经写好的样本。它擅长学格式、学常见路径、学专家当时怎么回答，但它看不到模型自己的动作会不会改掉下一步的观察结果。只要任务还是“输出一段像样的回答”，SFT 往往够用；一旦任务变成“先动一下，再看环境怎么变，再决定下一步”，SFT 就少了最关键的反馈。
TRL 的文档把这件事分得很清楚：`SFTTrainer` 适合 language modeling 或 prompt-completion，`GRPOTrainer` 的基础输入是 prompt-only；当任务需要多轮交互时，OpenEnv 把环境变成一个带状态的对象，`GRPOTrainer` 通过 `environment_factory` 把它接进训练环。这样，模型学的就连续动作后的结果。
本页只讲切换条件和最小骨架。你在这里要学会判断：是继续补示范样本，还是把任务改成环境；是把奖励放在离线评测里，还是放进 episode 末尾读回来的环境状态里。

## box warn
### 训练阶段怎么选
信号
先做什么
先不要做什么


格式稳定，结果不依赖后续状态
继续用 SFT，先把常见路径和格式学稳。
不要为了“更高级”直接上环境训练。


下一步观察依赖上一步动作
改成环境训练，让 observation 进入闭环。
不要只拿最终答案做监督。


只有偏好分数，没有交互环境
先做离线偏好或评测，再考虑是否能补环境。
不要把静态分数假装成 episode 奖励。


动作可能触达真实对象
先留在评测和沙盒，不要直接让模型探索生产系统。
不要把训练和上线混在同一套工具里。

# 术语与边界 {#terms}
<!-- tag: terms -->

SFT
从完成好的样本里学下一步该写什么。它看得见答案，不看见动作改变了什么。


Episode
一次完整交互，从 `reset` 开始，到终止条件满足或步数用尽结束。


Observation
环境返回给模型的输入，可能是文本、状态摘要或工具结果。它来自环境，不来自静态样本。


Action
模型在环境里发出的动作，通常对应一个公开工具方法，比如 `draft_reply()`、`move()` 或 `open_page()`。


Reward
episode 结束后读取的反馈。它可以存在环境实例里，再由 `reward_func(environments, **kwargs)` 统一读出。


`environment_factory`
传给 `GRPOTrainer` 的环境构造入口。文档的意思很直接：传工厂或类，不要传一个共享实例。

术语
在代码里看哪里
误读后会出什么问题


SFT
`prompt` / `completion` 或 `text`。
把静态示范误当成状态训练。


Episode
`reset` 到终止之间的一轮交互。
状态没清干净，上一轮结果泄漏到下一轮。


Observation
环境方法返回的内容。

Reward

Prompt-only

# SFT 为什么会卡住 {#sft-cutover}
<!-- tag: cutover -->
<!-- layout: grid2 -->

## box
拿“给客户写跟进邮件草稿”做例子。SFT 可以让模型学会语气、结构和常见措辞，但它只看到“最后一版草稿长什么样”。如果客户有没有回复、纪要里有没有缺失信息、草稿是否经过审批，都会改变下一步应该怎么做，那么单次示范就不够了。
环境训练的价值就在这里：模型先做一个动作，环境返回新的观察，训练信号再从这条轨迹里读出来。你关心的不只是“写得像不像”，还包括“是否清空了旧状态”“是否把错误恢复做对”“是否把风险动作拦在 episode 里”。
这也是为什么这页和 05 的分界线很重要。05 关注评测，回答“做完以后好不好”；06 关注训练，回答“做的时候怎么从结果里学到东西”。

## box
### 同一任务的两种样本形态
形态

看不见什么

# 奖励从哪里来 {#reward-bridge}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box
OpenEnv 文档里的最小例子是 Echo：环境执行一次动作，环境对象里记录 reward，`reward_func` 再从 `environments` 列表里读出来。这个结构放到别的任务里也一样：你先把环境状态写进去，再把 episode 的结果统一读回来。
下面这段是教学版的邮件草稿环境。它不是生产代码，但它保留了 docs 里的骨架：`reset` 清状态，公开方法当工具，`reward_func` 只读环境实例。这样的结构比把所有逻辑塞进 rollout 循环里更容易查错。
环境训练最小骨架
python
```python
class MailDraftEnv:
 def __init__(self):
 self.reward = 0.0
 self.done = False
 self.last_observation = ""

 def reset(self, task: str | None = None, audience: str | None = None, **kwargs):
 """Start a new episode."""
 self.reward = 0.0
 self.done = False
 self.task = task or "写跟进邮件草稿"
 self.audience = audience or "客户"
 self.last_observation = "请基于会议纪要写跟进邮件草稿，不要直接发送。"
 return self.last_observation

 def draft_reply(self, tone: str, length: str) -> str:
 """
 Draft a reply for the user.
 Args:
 tone: 邮件语气
 length: 邮件长度
 Returns:
 草稿文本
 """
 self.last_observation = f"草稿已更新：{tone}/{length}"
 self.reward = 1.0 if tone in {"polite", "neutral"} else 0.4
 return self.last_observation

 def submit_for_review(self) -> str:
 """Finish the episode."""
 self.done = True
 return "submitted for review"


def reward_func(environments, **kwargs):
 return [env.reward for env in environments]
```


## box warn
### 职责分工
对象
负责什么
最常见 bug


`environment_factory`
给每次生成创建一个新的环境实例。
传了共享实例，episode 互相污染。


`reset`
清状态，装载当前样本，返回初始观察。
忘记清零 reward 或 done。


公开工具方法
让模型发出可理解的动作。
方法名过泛、参数没类型、docstring 不完整。


`reward_func`
episode 结束后统一读出奖励。
直接看文本，导致奖励与真实状态脱钩。
在 OpenEnv 文档里，工具必须是带类型提示和说明的公开方法。`step(action)` 这种泛型入口对模型来说太空，不利于它学会选工具。

# 故障台：一条最常见的错误链 {#failure-lab}
<!-- tag: failure lab -->
<!-- layout: grid2 -->

## box
### 错误链
坏写法
python
```python
shared_env = MailDraftEnv()

trainer = GRPOTrainer(
 model=model,
 train_dataset=dataset,
 reward_funcs=reward_func,
 environment_factory=shared_env,
)

def reward_func(environments, **kwargs):
 return [1.0 if "polite" in env.last_observation else 0.0 for env in environments]
```

这段代码同时犯了两个错：`environment_factory` 传的是共享实例，不是工厂；`reward_func` 读的是文本片段，不是 episode 里真正的状态。

## box warn
### 怎么修
先修哪里
修复动作
为什么先修它


`environment_factory`
改成类或工厂，确保每次 rollout 拿到新实例。
先把 episode 隔离起来，避免状态交叉污染。


`reset`
清空 reward、done 和临时缓存，再返回初始观察。
否则 reward 会沿用上一轮残留。


`reward_func`
从环境实例里读出 episode 结果，而不是从文本 heuristics 读分。
这样训练信号才真正和任务目标对齐。
如果这三处任意一处没修好，模型都会看起来“会做题”，但实际上是在学错误的奖励来源。最常见的表面现象，就是 reward 曲线升了，真实任务成功率却没有跟着上去。

## box soft
这条故障链和本页前面的状态台账一一对应。你只要能把错误写法、症状和修复动作串起来，环境训练的基本排错就算过关了。

### 8 choice

**Q:** 8. 如果 reward 曲线升高，但真实任务成功率没升高，你会先查什么？

- [ ] 奖励定义和环境最终状态是否真的对齐
- [ ] 学习率一定太低
- [ ] 数据集 prompt 数量一定太少

Answer: 奖励定义和环境最终状态是否真的对齐
Explain: 这通常是 reward hacking 或读错奖励来源。先查 reward 定义、环境最终状态和是否只用了文本 heuristics。

### 9 debug-checklist

**Q:** 9. 哪一组检查最适合先做？

- [ ] environment_factory 是否每次创建新实例
- [ ] reset 是否把 reward / done / 临时缓存清零
- [ ] 而不是环境状态

Explain: 前两项是最先检查的状态边界；第三项是奖励来源检查。三个点都值得查，但排序上应该先确认 episode 有没有被污染，再确认 reward 有没有读错源。

### 10 synthesis

**Q:** 10. 设计一个手动 baseline 检查。

Explain: 合格答案至少要写清四件事： 任务 是什么、 基线 是谁、 指标 怎么量、 异常 说明了什么。比如可以把随机策略、强模型和人工策略放在同一环境里跑，再比较成功率、错误恢复率和是否越权。重点你能不能通过 baseline 把“环境能不能学”这件事看出来。

### 11 choice

**Q:** 11. 哪个场景最适合先上环境训练？

- [ ] 下一步观察会被上一步动作改变，但副作用可控
- [ ] 只是把一句固定 JSON 格式写对
- [ ] 只是把输出字数变长

Answer: 下一步观察会被上一步动作改变，但副作用可控
Explain: 如果下一步观察依赖上一步动作，就要先把任务写成环境而不是继续堆静态示范。

### 12 迁移

**Q:** 12. 把“浏览网页找信息”改写成环境接口。

Explain: 合格答案应该把接口拆成五件事： reset 返回初始观察， 工具方法 负责打开页面、点击、搜索等动作， 状态 记录当前页、已看内容和未完成任务， reward 读任务完成度和合规性， done 说明什么时候 episode 结束。只列函数名不够，必须把状态和奖励的来源说清楚。

### 13 choice

**Q:** 13. 把 `rollout_func` 迁移到 `environment_factory` 时，先保留在哪个对象里？

- [ ] 环境类本身：状态、reset、工具方法、docstring
- [ ] 只保留一个共享环境实例
- [ ] 只保留 reward 曲线图

Answer: 环境类本身：状态、reset、工具方法、docstring
Explain: 真正该保留的是环境类本身：状态、reset、工具方法和可读的 docstring。rollout 循环的控制权交给 trainer。

### 14 synthesis

**Q:** 14. 综合解释：用 120-180 字讲清 SFT 到环境 RL 的切换。

Explain: 合格答案要把切换条件说清楚：SFT 适合学静态样本和格式，GRPO / 环境训练适合学状态连续任务； reset 负责清空旧 episode 并返回初始观察，模型做出 action 后得到新的 observation， reward_func 再从环境实例里读出奖励。最后要点出：当任务的成败取决于连续交互，而不是单次答案时，才需要把训练改成 environment RL。
