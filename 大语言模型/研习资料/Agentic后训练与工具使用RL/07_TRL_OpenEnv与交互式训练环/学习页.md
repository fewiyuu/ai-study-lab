# 先抓主线 {#overview}
<!-- tag: 主线 -->
<!-- layout: grid2 -->

## box
SFT 看到的是已经写好的样本。它擅长学格式、学常见路径、学专家当时怎么回答，但它看不到模型自己的动作会不会改掉下一步的观察结果。只要任务还是“输出一段像样的回答”，SFT 往往够用；一旦任务变成“先动一下，再看环境怎么变，再决定下一步”，SFT 就少了最关键的反馈。
TRL 的文档把这件事分得很清楚：`SFTTrainer` 适合语言建模或 prompt-completion，`GRPOTrainer` 的基础输入是 prompt-only；当任务需要多轮交互时，OpenEnv 把环境变成一个带状态的对象，`GRPOTrainer` 通过 `environment_factory` 把它接进训练环。这样，模型学的是连续动作后的结果，不只是单次输出。
本页只讲切换条件和最小骨架。你在这里要学会判断：是继续补示范样本，还是把任务改成环境；是把奖励放在离线评测里，还是放进 episode 末尾读回来的环境状态里。

# 术语与边界 {#terms}
<!-- tag: 术语 -->

SFT
从完成好的样本里学下一步该写什么。它看得见答案，不看见动作改变了什么。


Episode
一次完整交互，从 `reset` 开始，到终止条件满足或步数用尽结束。


Observation
环境返回给模型的输入，可能是文本、状态摘要或工具结果。它来自环境，不来自静态样本。


Action
模型在环境里发出的动作，通常对应一个公开工具方法，比如 `draft_reply()`、`move()` 或 `open_page()`。


Reward
episode 结束后读取的反馈。它可以先存在环境实例里，再由 `reward_func(environments, **kwargs)` 统一读出。


`environment_factory`
传给 `GRPOTrainer` 的环境构造入口。文档的意思很直接：传工厂或类，不要传一个共享实例。

## box
术语
在代码里看哪里
误读后会出什么问题


SFT
`prompt` / `completion` 或 `text`。
把静态示范误当成状态训练。


Episode
`reset` 到终止之间的一轮交互。
状态没清干净，上一轮结果泄漏到下一轮。


Observation
环境方法返回的内容。
把模型自己的输出当成环境反馈。


Reward
环境实例上的 `reward` 或相似字段。
只读文本 heuristics，奖励欺骗 很快出现。


Prompt-only
GRPOTrainer 的基础输入形态。
把 preference 数据直接塞给环境训练。

# 一轮 episode 的状态台账 {#state-ledger}
<!-- tag: 状态 -->

这张表是本页最核心的检查点。每一步都要回答四件事：读什么、写什么、什么时候会变脏、错了以后会出现什么症状。只要这四列对齐，环境训练就不容易写成“看起来像 RL”的样子。

步骤
读取
写入 / 变更
什么时候会脏
错了会怎样


`reset()`
数据集列、任务描述、可选上下文。
清空上一轮的 reward、done、last_observation，返回初始观察。
如果不清零，下一轮会继承旧分数。
reward 泄漏，当前 episode 也会被污染。


公开工具方法
动作参数、当前环境状态。
修改环境状态，写入新观察或错误信息。
如果方法名太泛、参数太空，模型很难学会选工具。
工具 schema 弱，模型会卡在不稳定动作上。


`reward_func(environments, **kwargs)`
环境实例列表和 episode 里存下来的状态。
输出 `list[float]`，供 trainer 计算更新信号。
如果从文本 heuristics 读分数，容易被钻空子。
奖励欺骗，分数上升但真实任务成功率不变。


trainer update
模型完成、环境回报、参考策略差异。
更新模型参数。
如果前面三步没接对，更新会朝错方向走。
模型学会模板或长度偏好，偏离任务本身。

# 交互式训练环 {#loop}
<!-- tag: 训练环 -->
<!-- layout: grid2 -->

## box
### 为什么 SFT 不够
SFT 可以教模型“像训练数据那样调用工具”。但环境任务的难点在于：同一个动作在不同状态下结果不同，后续观察还取决于前面的动作。只模仿轨迹会把模型锁在已有策略里，遇到新状态时容易乱掉。
RL 的价值是让模型试出不同轨迹，再用环境反馈比较哪条轨迹更好。GRPO 这类方法会在一组候选 completions 内做相对比较，因此奖励不一定要非常细，但必须能稳定地区分成功和失败。
一次 episode ≈ 初始提示 + reset 后观察 + 多轮 action / observation + 最终 reward

## box info
### 先看这个判断
如果任务的下一次观察依赖上一次动作，就优先考虑 environment；如果每次工具调用都独立、没有持久状态，普通 tools 模式通常更简单。

### 训练器在幕后做什么

当你传入 `environment_factory` 后，`GRPOTrainer` 会自动处理多轮循环：创建环境、发现工具、生成回复、解析工具调用、执行环境工具、把结果放回对话，再继续生成，直到模型停止调用工具或达到长度限制。你不需要自己写完整 rollout loop。

这也是 `environment_factory` 与手写 `rollout_func` 的核心差异：前者把交互协议标准化，适合大多数“模型直接调用环境工具”的训练；后者适合外部 agent server 自己掌控生成和交互协议的场景。

三条责任线

**模型**根据当前上下文决定是否调用工具、调用哪个工具、传什么参数。
**环境**保存状态，执行动作，返回观察，记录 reward、done、错误等信号。
**trainer**组织采样和训练，把 episode 文本和奖励变成 GRPO 的优化信号。

观察重点：模型不应该直接知道 hidden state；它只能通过工具结果看到环境愿意暴露的 observation。

# 环境类怎么写 {#env-class}
<!-- tag: 接口 -->

最小环境类有三块：构造时准备客户端或状态；`reset(**kwargs)` 在每个 episode 开始时清理并返回初始观察；公开工具方法把模型动作转成环境 `step`，并保存奖励、结束标记或错误信息。

最小结构示意
Python

```python
from datasets import Dataset
from trl import GRPOConfig, GRPOTrainer

ENV_URL = "https://example-env.hf.space"

class TaskEnv:
 def __init__(self):
 self.client = make_client(base_url=ENV_URL)
 self.reward = 0.0
 self.done = False

 def reset(self, **kwargs) -> str | None:
 self.reward = 0.0
 self.done = False
 initial = self.client.reset(task_id=kwargs.get("task_id"))
 return initial.message

 def act(self, command: str) -> str:
 """
 Execute one command in the task environment.

 Args:
 command: The command to execute.

 Returns:
 The new observation after executing the command.
 """
 if self.done:
 raise ValueError("Episode is already finished.")
 result = self.client.step(command)
 self.reward = float(result.reward)
 self.done = bool(result.done)
 return result.observation

def reward_func(environments, **kwargs) -> list[float]:
 return [env.reward for env in environments]

dataset = Dataset.from_dict({
 "prompt": [[{"role": "user", "content": "Solve the task by using the tool."}]] * 64,
 "task_id": list(range(64)),
})

trainer = GRPOTrainer(
 model="Qwen/Qwen3-0.6B",
 args=GRPOConfig(log_completions=True, max_completion_length=1024),
 train_dataset=dataset,
 reward_funcs=reward_func,
 environment_factory=TaskEnv,
)
trainer.train()
```

`__init__`适合放全局配置、轻量状态和客户端准备。多环境训练时，重连接成本高的客户端常延迟到 `reset` 中创建。
`reset(**kwargs)`是 episode 边界。这里清空 reward、done、历史缓存，并用数据集列选择任务、难度或具体环境。
公开工具方法方法名、参数类型、docstring 会影响工具 schema。名字越贴近任务语义，模型越容易学会正确调用。

## box warn
### 为什么不推荐只暴露 `step(action)`
`step(action)` 对程序员清楚，但对语言模型不清楚。模型看到的是工具名字和参数说明；`guess(word)`、`click(selector)`、`run_code(code)` 比 `step(action)` 给出更多行动先验。训练要让环境能跑通，也要让模型知道“此刻有哪些合法动作”。

# 奖励、停止与长度预算 {#reward-bridge}
<!-- tag: 故障 -->
<!-- layout: grid2 -->

奖励：优先让环境判断结果
环境任务的奖励最好从最终状态或环境内置判定来读。比如 Wordle 任务可以用是否猜中作为奖励，不必奖励每一步像不像人类策略。
简单奖励常常更稳：成功 1.0，失败 0.0。GRPO 比较同一提示下多条候选轨迹的相对好坏，过度 shaped reward 反而可能让模型钻局部规则。

## box
### 危险信号
- 随机策略和强模型的分数接近。
- reward 只检查格式，不检查环境状态。
- 失败动作没有错误反馈，模型无法知道哪里错。
- episode 被长度截断，却误判为模型不想继续。

| 问题 | 表现 | 检查 | 修法 |
| --- | --- | --- | --- |
| 长度预算太小 | 模型做到一半停止，长游戏尤其明显。 | 查看 max_completion_length 是否覆盖整局对话，而非单轮输出。 | 提高总长度预算，压缩工具结果，避免重复回传完整历史。 |
| 并发连接不足 | 训练启动后环境服务报连接或 session 错误。 | 确认每个 generation 都可能打开一个环境实例和 WebSocket 连接。 | 复制专用 Space、本地容器化运行，或配置足够的 max_concurrent_envs 。 |
| 状态泄漏 | 不同样本互相影响，reward 异常波动。 | 检查是否传入共享环境实例，或 reset 没有清理状态。 | 传类 / 工厂；每个 episode 初始化独立状态。 |
| 错误不可学习 | 模型反复调用非法动作。 | 检查工具方法遇到非法动作时是否返回清楚的错误消息。 | 用异常或明确 observation 告诉模型动作无效、游戏结束或参数错误。 |

$$
 \text{reward\_func}(\text{environments}) \rightarrow [r_1, r_2, \dots, r_n]
 $$
 最低要求是：返回长度和环境列表一致，且每个分数都来自对应 episode 的最终可验证状态，不要从一段文本里的表面特征硬猜。

# 多环境训练 {#multi-env}
<!-- tag: 并发 -->
<!-- layout: grid2 -->

## box
### 为什么每个 episode 都要独立
OpenEnv 的训练要求每条采样路线使用独立环境实例，不共享同一份状态。这样做很朴素：如果样本之间互相看见彼此的历史，模型学到的就是环境残留。
`environment_factory` 的职责是给 trainer 一个每次都能重新造出来的环境。这个细节一旦写错，很多故障看起来像训练不稳定，实际是 episode 边界失效。

### 并发先看什么
对象
要独立什么
错了会看到什么


环境实例

前一轮结果串到后一轮


连接 / session
每个实例自己的会话生命周期
session limit、连接被挤爆、响应延迟


奖励读取
只读当前 episode 的最终状态
reward 记到别的样本头上

# 例题 {#examples}
<!-- tag: 例子 -->

## box
### 例 1：Echo 环境为什么仍然有教学价值
Echo 环境看起来太简单：模型调用 `echo(message)`，环境返回回声并给奖励。但它能验证交互式训练的所有关键接口：工具发现、工具调用解析、环境执行、reward 保存、reward_func 读取、trainer 训练。
Echo 包装类的关键点
Python
```python
class EchoToolEnv:
 def __init__(self):
 self.env = EchoEnv(base_url=ENV_URL)
 self.reward = 0.0

 def reset(self, **kwargs) -> str | None:
 self.reward = 0.0
 return None

 def echo(self, message: str) -> str:
 """
 Echo the message back from the environment.

 Args:
 message: The message to echo.

 Returns:
 The echoed message from the environment.
 """
 observation = self.env.step(EchoAction(message=message))
 self.reward = observation.observation.reward
 return observation.observation.echoed_message

def reward_func(environments, **kwargs):
 return [env.reward for env in environments]
```

读这段代码不要停在“它会 echo”。要看的是：reward 从环境返回的 observation 写回 `self.reward`，最后由 `reward_func` 交给 trainer。

## box
### 例 2：Wordle 任务里的状态与增量观察
Wordle 之所以更像真正环境，是因为每次猜词后，下一次反馈依赖前面的猜测。很多环境客户端会返回完整历史；如果每轮都把完整历史塞回工具结果，token 会膨胀，模型也会被重复内容干扰。
只回传新增反馈
Python
```python
def guess(self, guess: str) -> str:
 """
 Make a guess in Wordle.

 Args:
 guess: The guessed word, formatted as a five-letter word.

 Returns:
 The new feedback after this guess.
 """
 if self.done:
 raise ValueError("Game over.")

 result = self.client.step(TextArenaAction(message=guess))
 full_feedback = result.observation.messages[0].content
 new_feedback = full_feedback[len(self._last_full_feedback):]
 self._last_full_feedback = full_feedback
 self.reward = float(result.reward)
 self.done = bool(result.done)
 return new_feedback
```

这里的关键是：环境内部可以保存完整状态，模型只看到必要的新 observation。这样会降低上下文成本，也更接近真实工具使用中的审计边界。

## box warn
### 常见错误：把 done 当成自动停止
`done=True` 通常只是环境状态。模型并不会因为 Python 变量变成 True 就自动停止调用工具。你需要通过工具返回、异常消息或系统提示，把“任务已结束，不要再调用工具”的信息放回模型可见上下文。
**修法：**在工具方法开头检查 `self.done`；如果已经结束，抛出清楚的错误，例如 `ValueError("Game over.")`。trainer 会把错误作为工具响应反馈给模型，模型才有机会学到停止行为。

### 1 单选

**Q:** 1. 什么时候该用 environment 而不是 stateless tools？

- [ ] 下一步观察取决于上一步动作留下的状态
- [ ] 模型参数还不够大
- [ ] 输出格式还没有固定

Answer: 下一步观察取决于上一步动作留下的状态
Explain: 状态连续性是切换信号。只要下一步观察取决于上一步动作，静态示范就少了关键反馈。

### 2 判断

**Q:** 2. GRPOTrainer 的环境循环只要给 `reward_func` 就够了吗？

- [ ] 对
- [ ] 错

Answer: false
Explain: 光有 reward_func 还不够。GRPOTrainer 还需要 environment_factory 造出独立环境，并在训练时跑多轮交互。

### 3 短填

**Q:** 3. GRPOTrainer 的基础输入形态是什么？

Answer: prompt-only
Explain: TRL 的 dataset formats 里，GRPOTrainer 对应的是 prompt-only。环境反馈由 environment_factory 和 reward_func 提供，不靠 preference 样本本身。

### 4 单选

**Q:** 4. `reward_func` 最稳妥的读取对象是什么？

- [ ] 环境实例里 episode 期间存下来的状态
- [ ] 模型刚生成的文本长度
- [ ] tokenizer 的配置

Answer: 环境实例里 episode 期间存下来的状态
Explain: reward_func 应该读取环境实例里 episode 期间存下来的状态，例如 env.reward，而不是直接读模型刚生成的文本。

### 8 单选

**Q:** 8. 如果 reward 曲线升高，但真实任务成功率没升高，你会先查什么？

- [ ] 奖励定义和环境最终状态是否真的对齐
- [ ] 学习率一定太低
- [ ] 数据集 prompt 数量一定太少

Answer: 奖励定义和环境最终状态是否真的对齐
Explain: 这通常是 奖励欺骗 或读错奖励来源。先查 reward 定义、环境最终状态和是否只用了文本 heuristics。

### 9 排查清单

**Q:** 9. 第二轮分数一上来就不对时，先查哪三层？

- [ ] 第 1 层：
- [ ] 第 2 层：
- [ ] 第 3 层：

Explain: 顺序应是：先确认 environment_factory 有没有为每个 episode 造新实例，再看 reset 有没有清掉旧状态，最后看 reward_func 有没有从当前 episode 的环境状态里读分。顺序比单独的检查项更重要。

### 10 综合

**Q:** 10. 设计一个手动对照检查。

Explain: 合格答案至少要写清四件事： 任务 是什么、 对照组 是谁、 指标 怎么量、 异常 说明了什么。比如可以把随机策略、强模型和人工策略放在同一环境里跑，再比较成功率、错误恢复率和是否越权。重点在于，你能不能通过对照把“环境能不能学”这件事看出来。

### 11 单选

**Q:** 11. 哪个场景最适合先上环境训练？

- [ ] 下一步观察会被上一步动作改变，但副作用可控
- [ ] 只是把一句固定 JSON 格式写对
- [ ] 只是把输出字数变长

Answer: 下一步观察会被上一步动作改变，但副作用可控
Explain: 如果下一步观察依赖上一步动作，就要先把任务写成环境而不是继续堆静态示范。

### 12 迁移

**Q:** 12. 把“浏览网页找信息”改写成环境接口。

Explain: 合格答案应该把接口拆成五件事： reset 返回初始观察， 工具方法 负责打开页面、点击、搜索等动作， 状态 记录当前页、已看内容和未完成任务， reward 读任务完成度和合规性， done 说明什么时候 episode 结束。只列函数名不够，必须把状态和奖励的来源说清楚。

### 13 单选

**Q:** 13. 把 `rollout_func` 迁移到 `environment_factory` 时，先保留在哪个对象里？

- [ ] 环境类本身：状态、reset、工具方法、docstring
- [ ] 只保留一个共享环境实例
- [ ] 只保留 reward 曲线图

Answer: 环境类本身：状态、reset、工具方法、docstring
Explain: 真正该保留的是环境类本身：状态、reset、工具方法和可读的 docstring。rollout 循环的控制权交给 trainer。

### 14 综合

**Q:** 14. 综合解释：用 120-180 字讲清 SFT 到环境 RL 的切换。

Explain: 合格答案要把切换条件说清楚：SFT 适合学静态样本和格式，GRPO / 环境训练适合学状态连续任务； reset 负责清空旧 episode 并返回初始观察，模型做出 action 后得到新的 observation， reward_func 再从环境实例里读出奖励。最后要点出：当任务成败取决于连续交互，而不是单次答案时，就需要把训练改成 environment RL。
