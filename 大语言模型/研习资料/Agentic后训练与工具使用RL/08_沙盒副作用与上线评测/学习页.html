08｜沙盒、副作用与上线评测

这一页把 07 页的环境训练往生产边上推一步。你会看到同样是 action / observation / reward，到了上线前就要多看三件事：动作会不会改到真实对象，观测里有没有把副作用读出来，以及回退和审计能不能把风险兜住。

课程地图 上一单元

读完本页应该能做什么

  • 判断一个动作只能进沙盒，还是已经可以进入灰度。
  • 从状态台账里看出 episode 有没有脏数据、漏审计或共享实例。
  • 把上线闸门写成能执行、能记录、能停机的评测条件。

学习目标

route

先分动作等级

查询、可撤销写入、对外外发、真实删除，不要混成一层。动作等级不同，闸门也不同。

先把上线条件写明

不要只写“任务成功”，还要写越权、回退、审计和外部影响的门槛。

先抓主线

flow

最容易看混的是：模型输出一个动作，不等于系统已经安全。上线前要把路径拆成五步，分别看谁在写、谁在读、谁能停。

核心讲解

chain

把场景想成一个邮件助理。用户说“把昨天那个退款单通知发给客户”，模型会先查订单，再生成草稿，最后选择发出。这里最容易被忽略的点它到底碰的是测试租户还是生产租户，写的是草稿还是正式外发，记录的是一条可回滚的审计，还是一条只剩最终结果的日志。

同一个动作名，在沙盒里可能只是一次练习，在生产边上就会变成真实副作用。评测要做的事，是把这两种情形拆开：看清动作目标、看清状态变化、看清是否留下回退证据。只要这三件事里有一件缺位，分数再漂亮也不能说明系统可上线。

判定时的四个锚点

  • 目标对象是不是当前回合的对象。
  • 凭证和租户是不是测试范围。
  • 状态差异有没有被明确记下来。
  • 回退路径是不是在动作前就能找到。

如果这四个锚点里有一个空着，就要先停在沙盒，不要把它误当成“可以先试试”的灰度。

步骤你看到什么它说明什么失败会怎样
dry-run测试租户、假收件人、无提交先验证语义和参数如果仍有外部写入，环境本身就不干净
shadow mode真实输入、无提交先看分布和策略命中只能说明看见了流量，不能说明副作用可控
small gray rollout少量真实流量看回退、审计、异常处理如果停止条件缺失，问题会跟着流量一起扩大
full live全量真实流量只在闸门全部通过后才考虑没有审计和回退，就等于把风险放到最大

一个能看出副作用的壳

复制

`def run_with_gates(env, action):

before = env.snapshot()

result = env.step(action)

after = env.snapshot()

diff = after.diff(before)

audit = {

"tenant": env.tenant_id,

"target": action.target,

"diff": diff,

"approved": env.approval_state,

"rollback": result.rollback_available,

}

env.audit_log.append(audit)

if diff.has_external_side_effect and not audit["approved"]:

raise RuntimeError("unapproved side effect")

return result`

失败会怎么暴露

如果 snapshot() 读的是共享对象，diff 会带着上一轮的痕迹；如果 approval_state 没进日志，事故后就查不出是谁放行；如果 rollback_available 永远写真，闸门就会高估稳定性。

这个链条的目标很朴素：每一步都能看见对象变化，出错时能停，回头还能说明白。

练习钩子

拿 07 页的环境类改一改：先把 snapshot() 和 audit_log 加上，再想哪些动作只能跑 dry-run，哪些动作可以进入最小灰度。你写出来以后，应该能一眼看出哪一步最容易把真实对象误写进去。

案例链

example

这一段把讲解落到一个可排查的例子上。假设邮件助理在测试租户里运行正常，管理员看到的结果也都对，但一旦接到真实客户名单，系统就开始出现“看上去完成了、实际上已经越界”的情况。评测如果只看最终文本，会把危险路径漏掉。

if run.target_tenant != "sandbox":

return "stop: target tenant is not sandbox"

if run.sent_to_real_recipient:

return "stop: external side effect detected"

把案例改成练习

你可以直接拿这条链做一张小表：对象是不是测试租户，动作是不是外发，审计是不是完整，回退是不是可用。任何一项不稳，就先停在沙盒；如果四项都稳，再考虑最小灰度。这样写出来的判定，才真的能拿去挡上线。

术语与分层

terms

只读查询

不改外部对象，但仍可能越权读取、泄露数据或消耗配额。

不可逆动作

外发邮件、真实删除、资金变更，出了问题不能只靠“再来一轮”修复。

灰度

把风险缩到小流量里，先看副作用，再决定要不要放大。

动作层级典型动作默认处理
只读查公开文档、查状态、查日志先看越权和隐私，再看结果是否正确。
可撤销写入建测试工单、写草稿、调试记录先进沙盒，再看是否有回退路径。
不可逆外发发给真实客户、删真实数据、触发付款先加人工闸门，再做影子或最小灰度。

状态台账

state

这里最重要的环境类在每个 episode 里到底读什么、写什么、什么时候会脏。只要状态台账一乱，后面的 reward 和评测都会跟着偏。

`class TaskEnv:

def __init__(self):

self.reward = 0.0

self.done = False

self.audit_log = []

self.last_diff = {}

def act(self, tool_name, **params):

if self.done:

raise ValueError("episode finished")

result = self.client.call(tool_name, **params)

self.last_diff = result.state_diff

self.audit_log.append({

"tool": tool_name,

"params": params,

"diff": result.state_diff,

})

self.reward = float(result.reward)

self.done = bool(result.done)

return result.observation

先盯这四个字段

reward、done、audit_log、last_diff 是最容易出错的地方。reset() 不清空，下一轮会继承上一轮的痕迹；act() 不写差异，评测就只剩表面结果；reward_func() 只读文本，就很容易被钻空子。

真正该检查的调用前后状态有没有变脏，评测有没有读到该读的那部分。

步骤读取写入 / 清空脏了会怎样下一步检查
reset()任务参数、测试对象、可选上下文清空 reward、done、audit_log、last_diff下一轮继承上一轮状态，reward 泄漏先确认每个 episode 都是新状态
公开工具方法动作参数、当前状态写入 state diff、观察、错误信息工具语义不清，模型学不会选动作看方法名、参数和 docstring 是否具体
reward_func()环境实例列表和 episode 状态输出可比较的 reward只读文本 heuristics，reward hacking 很快出现确认 reward 真的是从环境状态读回来的
trainer update完成轨迹、奖励、参考策略差异更新模型参数前面三步接错，更新方向也会错先把日志、回退和审计补齐再训练

评测闸门

gate
$$ \text{GoLive} = \text{Success} \land \neg \text{Violation} \land \text{Rollback} \land \text{Audit} $$

Success 看任务是否完成，Violation 看有没有越权，Rollback 看出错后能不能停住，Audit 看证据链是否完整。这里在写一个能停机的条件。

实现一致性

contract

沙盒、离线评测、灰度上线这三层能不能互相说明问题，关键在于它们是不是读同一份状态契约。这里的契约，不只是接口名一样，还包括对象 id、审批状态、审计记录、回退结果和副作用边界都能按同一种语义被读回来。只要其中一层把字段省掉、改名、或者改成别的含义，数字看起来还会动，但它们已经不能再证明同一件事。

层级必须共享可以不同不同了会怎样
训练环境对象 id、state diff、reward、done数据规模、样本难度、随机种子如果契约字段缺失，后面的评测就接不上训练语义。
离线评测审批状态、审计记录、回退结果流量规模、批次大小、观察窗口如果只看分数不看证据，危险路径会被当成能力提升。
灰度上线真实对象 id、外部影响标记、停止条件放量比例、观察时长、人工复核频率如果回退链不在场，异常只会随着流量一起扩散。

`def assert_contract(env):

required = [

"object_id",

"approval_state",

"audit_log",

"rollback_result",

"side_effect_boundary",

]

missing = [name for name in required if not hasattr(env, name)]

if missing:

raise AssertionError(f"missing contract fields: {missing}")

if env.side_effect_boundary not in {"sandbox", "shadow", "gray", "live"}:

raise AssertionError("unexpected side effect boundary")

return True`

练习钩子

把 07 页的 environment、这一页的 gate、还有你的评测脚本并排看一遍。只要你发现某个字段在其中一层被删掉，就说明这一层已经不能再拿来解释另两层的结果了。能写出这张对照表，你才算真正把训练和上线连在一起。

故障台

logs

先看输出，不先看结论。你要找的是哪一层开始脏了，是共享状态、审计缺失，还是动作已经越过了沙盒边界。

复制

如果你已经有 07 页的环境类，下一步补什么

先补 audit_log、state_diff、rollback_result 和 approval_state。然后把会碰真实对象的工具拆成 dry-run 和 execute 两条路径，先跑影子模式，再做最小灰度。

第一次最常见的失败是共享环境实例和漏记审计字段。只要这两项还在，reward 再高也不能说明可上线。

你现在要改什么为什么先验失败
环境类的 reset清空上一轮的脏状态下一轮继承旧 reward、旧审计或旧对象 id
工具方法把动作和副作用写清楚模型只会看到一个泛化的 step，学不会选工具
reward_func从环境状态读回结果只看文本表面，reward hacking 很快出现
评测脚本把闸门和回退条件固定下来任务成功率看起来高，但危险路径没人拦

常见错误

mistakes

最常见的错评测把危险藏起来了。下面这几条，优先按顺序排查。

先知道，先别追

boundary

这一页先不追异步 rollout、分布式扩容、复杂 IAM、奖励模型重写和自动化事故复盘。它们都重要，但要等你已经能读懂状态台账、闸门表和故障日志再展开。

如果你现在只想继续往前走，先回头检查三件事：动作有没有碰到真实对象，日志有没有记下 state diff 和审批，出错以后有没有回退。

16面试

Q: 面试中被问“工具使用 RL 怎么安全上线”，用 5-7 句话回答。

参考答案 / 评分要点

。合格回答通常会先说：工具动作会改真实对象，所以不能直接在生产里探索。然后依次说沙盒、负例评测、影子模式、小流量灰度、审计和回退。最后要补一句：上线看的是副作用是否可控，不只是任务是否成功。

17迁移

Q: 向产品经理解释：为什么不能直接让模型在生产里继续探索？

参考答案 / 评分要点

。答案里至少要说清：真实成本会被消耗、用户会被直接触达、数据会被污染、出错后回滚成本会升高。最后最好补一句，为什么离线学习、影子模式和灰度是更稳的替代方案。
