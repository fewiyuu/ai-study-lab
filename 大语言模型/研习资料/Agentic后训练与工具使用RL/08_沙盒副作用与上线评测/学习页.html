课程单元

沙盒、副作用与上线评测

这一页把 07 页的环境训练往生产边上推一步。你会看到同样是 action / observation / reward，到了上线前就要多看三件事：动作会不会改到真实对象，观测里有没有把副作用读出来，以及回退和审计能不能把风险兜住。

课程地图 上一单元

读完这页先会什么

  • 判断一个动作只能进沙盒，还是已经可以进入灰度。
  • 从状态台账里看出 episode 有没有脏数据、漏审计或共享实例。
  • 把上线闸门写成能执行、能记录、能停机的评测条件。

学习目标

route

先分动作等级

查询、可撤销写入、对外外发、真实删除，不要混成一层。动作等级不同，闸门也不同。

先看状态边界

episode 里的 reward、done、审计和 state diff 要按回合清空，不然下一轮会继承上一轮的痕迹。

先把上线条件写明

不要只写“任务成功”，还要写越权、回退、审计和外部影响的门槛。

这一页接着 07 页继续讲 environment，但视角换成上线前评测。你现在不只是在问“模型会不会动手”，而是在问“动手以后会留下些什么，出了事怎么停”。

先抓主线

flow

最容易看混的是：模型输出一个动作，不等于系统已经安全。上线前要把路径拆成五步，分别看谁在写、谁在读、谁能停。

任务意图 用户想完成什么

先确认对象、范围和约束。对象没定清，后面的副作用评估也会跟着飘。

工具动作 模型发出的调用

动作本身可以很小，但它可能触发写入、外发、删除或审批流。

沙盒状态 测试租户、测试凭证、测试对象

动作只允许改这里。只要状态离开这个范围，就该停下来看闸门。

评测读回 state diff、审计、回退结果

评测看的是动作留下了什么，不只是模型说得像不像。

上线闸门 继续、灰度、停机

闸门不是装饰。没有审计、没有回退、没有停止条件，默认就不该继续放量。

先按这条链回溯，再去看日志。只看最终回答，往往会把危险路径藏起来。

核心讲解

chain

把场景想成一个邮件助理。用户说“把昨天那个退款单通知发给客户”，模型会先查订单，再生成草稿，最后选择发出。这里最容易被忽略的点不是语句本身，而是它到底碰的是测试租户还是生产租户，写的是草稿还是正式外发，记录的是一条可回滚的审计，还是一条只剩最终结果的日志。

同一个动作名，在沙盒里可能只是一次练习，在生产边上就会变成真实副作用。评测要做的事，是把这两种情形拆开：看清动作目标、看清状态变化、看清是否留下回退证据。只要这三件事里有一件缺位，分数再漂亮也不能说明系统可上线。

这个链条要依次看什么

先看对象有没有指错，接着看动作有没有真的提交，再看审计和回退有没有被写进记录。顺序一乱，问题就会被藏进“看起来成功”的结果里。

判定时的四个锚点

  • 目标对象是不是当前回合的对象。
  • 凭证和租户是不是测试范围。
  • 状态差异有没有被明确记下来。
  • 回退路径是不是在动作前就能找到。

如果这四个锚点里有一个空着，就要先停在沙盒，不要把它误当成“可以先试试”的灰度。

步骤 你看到什么 它说明什么 失败会怎样
dry-run 测试租户、假收件人、无提交 先验证语义和参数 如果仍有外部写入，环境本身就不干净
shadow mode 真实输入、无提交 先看分布和策略命中 只能说明看见了流量，不能说明副作用可控
small gray rollout 少量真实流量 看回退、审计、异常处理 如果停止条件缺失，问题会跟着流量一起扩大
full live 全量真实流量 只在闸门全部通过后才考虑 没有审计和回退，就等于把风险放到最大
一个能看出副作用的壳 
def run_with_gates(env, action):
    before = env.snapshot()
    result = env.step(action)
    after = env.snapshot()
    diff = after.diff(before)
    audit = {
        "tenant": env.tenant_id,
        "target": action.target,
        "diff": diff,
        "approved": env.approval_state,
        "rollback": result.rollback_available,
    }
    env.audit_log.append(audit)
    if diff.has_external_side_effect and not audit["approved"]:
        raise RuntimeError("unapproved side effect")
    return result

失败会怎么暴露

如果 snapshot() 读的是共享对象，diff 会带着上一轮的痕迹；如果 approval_state 没进日志，事故后就查不出是谁放行；如果 rollback_available 永远写真，闸门就会高估稳定性。

这个链条的目标很朴素：每一步都能看见对象变化，出错时能停，回头还能说明白。

练习钩子

拿 07 页的环境类改一改：先把 snapshot() 和 audit_log 加上，再想哪些动作只能跑 dry-run，哪些动作可以进入最小灰度。你写出来以后，应该能一眼看出哪一步最容易把真实对象误写进去。

案例链

example

这一段把讲解落到一个可排查的例子上。假设邮件助理在测试租户里运行正常，管理员看到的结果也都对，但一旦接到真实客户名单，系统就开始出现“看上去完成了、实际上已经越界”的情况。评测如果只看最终文本，会把危险路径漏掉。

轨迹步 现场发生了什么 应该读到什么 失败说明什么
选对象 模型拿到了真实客户 id 只应拿到测试对象 id 目标边界已经错位
生成草稿 文案、语气、模板都正常 草稿只留在沙盒，不触达外部 文本正确不代表动作安全
正式提交 调用了发送接口 先过审批或灰度开关 副作用已经落地
记录证据 只有结果，没有 state diff 要有对象、参数、差异、批准和回退 事故后无法复盘
检查轨迹的最小函数 
def validate_mail_flow(run):
    if run.target_tenant != "sandbox":
        return "stop: target tenant is not sandbox"
    if run.sent_to_real_recipient:
        return "stop: external side effect detected"
    if not run.audit_log:
        return "stop: missing audit log"
    if not run.rollback_plan:
        return "stop: no rollback plan"
    return "pass"

失败后果

一旦真实收件人被触达，草稿再正确也没法撤回外部影响。评测如果不看目标对象和审计，只盯任务成功率，就会把最危险的一步藏起来。上线时真正要拦的是“真实对象 + 外部提交 + 证据缺失”这个组合。

把案例改成练习

你可以直接拿这条链做一张小表：对象是不是测试租户，动作是不是外发，审计是不是完整，回退是不是可用。任何一项不稳，就先停在沙盒；如果四项都稳，再考虑最小灰度。这样写出来的判定，才真的能拿去挡上线。

术语与分层

terms

只读查询

不改外部对象，但仍可能越权读取、泄露数据或消耗配额。

可撤销写入

会改状态，但能回滚。适合先灰度，不适合直接放到生产对象上。

不可逆动作

外发邮件、真实删除、资金变更，出了问题不能只靠“再来一轮”修复。

影子模式

看见真实输入，但不提交动作。它适合找分布差异，不适合证明万无一失。

灰度

把风险缩到小流量里，先看副作用，再决定要不要放大。

审计

记录动作、参数、状态差异、批准和回退结果。没有审计，事后基本说不清。

动作层级 典型动作 默认处理
只读 查公开文档、查状态、查日志 先看越权和隐私，再看结果是否正确。
可撤销写入 建测试工单、写草稿、调试记录 先进沙盒，再看是否有回退路径。
不可逆外发 发给真实客户、删真实数据、触发付款 先加人工闸门，再做影子或最小灰度。

状态台账

state

这里最重要的不是“有没有一个环境类”，而是环境类在每个 episode 里到底读什么、写什么、什么时候会脏。只要状态台账一乱，后面的 reward 和评测都会跟着偏。

最小守门骨架 
class TaskEnv:
    def __init__(self):
        self.reward = 0.0
        self.done = False
        self.audit_log = []
        self.last_diff = {}

    def reset(self, **kwargs):
        self.reward = 0.0
        self.done = False
        self.audit_log.clear()
        self.last_diff = {}
        return "initial observation"

    def act(self, tool_name, **params):
        if self.done:
            raise ValueError("episode finished")
        result = self.client.call(tool_name, **params)
        self.last_diff = result.state_diff
        self.audit_log.append({
            "tool": tool_name,
            "params": params,
            "diff": result.state_diff,
        })
        self.reward = float(result.reward)
        self.done = bool(result.done)
        return result.observation

def reward_func(environments, **kwargs) -> list[float]:
    return [env.reward for env in environments]

先盯这四个字段

reward、done、audit_log、last_diff 是最容易出错的地方。reset() 不清空，下一轮会继承上一轮的痕迹；act() 不写差异，评测就只剩表面结果；reward_func() 只读文本，就很容易被钻空子。

真正该检查的不是“有没有调用成功”，而是调用前后状态有没有变脏，评测有没有读到该读的那部分。

步骤 读取 写入 / 清空 脏了会怎样 下一步检查
reset() 任务参数、测试对象、可选上下文 清空 reward、done、audit_log、last_diff 下一轮继承上一轮状态，reward 泄漏 先确认每个 episode 都是新状态
公开工具方法 动作参数、当前状态 写入 state diff、观察、错误信息 工具语义不清，模型学不会选动作 看方法名、参数和 docstring 是否具体
reward_func() 环境实例列表和 episode 状态 输出可比较的 reward 只读文本 heuristics，reward hacking 很快出现 确认 reward 真的是从环境状态读回来的
trainer update 完成轨迹、奖励、参考策略差异 更新模型参数 前面三步接错，更新方向也会错 先把日志、回退和审计补齐再训练

评测闸门

gate
$$ \text{GoLive} = \text{Success} \land \neg \text{Violation} \land \text{Rollback} \land \text{Audit} $$

Success 看任务是否完成，Violation 看有没有越权，Rollback 看出错后能不能停住，Audit 看证据链是否完整。这里不是在做一个漂亮分数，而是在写一个能停机的条件。

一票否决的信号

  • 生产凭证跑评测。
  • 动作已经触达真实对象，却没有回退。
  • 审计字段空着，日志只能看见最终答案。
  • 异常率上升时系统还继续放量。
指标 看什么 不合格说明什么 下一步
task_success 任务有没有做成 任务本身就没跑通 先修功能，不要先谈放量
policy_violation_rate 有没有越权、绕过审批、触达禁区 模型会做对事，但走了危险路径 回看动作级日志和策略拦截点
rollback_success 出错后能不能停住并恢复状态 一旦出错，影响会继续扩散 先补回退演练，再考虑灰度
audit_complete 动作、参数、差异、批准是否齐 事后不好复盘，也不好追责 把审计字段写进环境和评测脚本
unexpected_diff_rate 有没有不该出现的状态变化 模型可能在隐藏路径上乱写 回到沙盒和状态台账排查

实现一致性

contract

沙盒、离线评测、灰度上线这三层能不能互相说明问题，关键在于它们是不是读同一份状态契约。这里的契约，不只是接口名一样，还包括对象 id、审批状态、审计记录、回退结果和副作用边界都能按同一种语义被读回来。只要其中一层把字段省掉、改名、或者改成别的含义，数字看起来还会动，但它们已经不能再证明同一件事。

这件事在 agent 场景里很常见。训练环境为了简单，可能先不接审批流；评测脚本为了快，可能只读最终结果；上线前的灰度窗口则必须把审批、审计和回退全都拉进来。三层之间如果没有一份共同契约，模型就会在“看起来成功”的评测里拿分，却在真实副作用上失控。

层级 必须共享 可以不同 不同了会怎样
训练环境 对象 id、state diff、reward、done 数据规模、样本难度、随机种子 如果契约字段缺失，后面的评测就接不上训练语义。
离线评测 审批状态、审计记录、回退结果 流量规模、批次大小、观察窗口 如果只看分数不看证据，危险路径会被当成能力提升。
灰度上线 真实对象 id、外部影响标记、停止条件 放量比例、观察时长、人工复核频率 如果回退链不在场，异常只会随着流量一起扩散。
检查契约是否一致 
def assert_contract(env):
    required = [
        "object_id",
        "approval_state",
        "audit_log",
        "rollback_result",
        "side_effect_boundary",
    ]
    missing = [name for name in required if not hasattr(env, name)]
    if missing:
        raise AssertionError(f"missing contract fields: {missing}")
    if env.side_effect_boundary not in {"sandbox", "shadow", "gray", "live"}:
        raise AssertionError("unexpected side effect boundary")
    return True

为什么这里要单独检查契约

如果训练环境没有这些字段，模型学到的是“简化版世界”。如果评测脚本没有这些字段，分数就只是在读一个缩水后的世界。如果上线灰度没有这些字段，风险就会在真实对象上直接放大。三层都写全，才能让同一件事在不同阶段都能被看见。

练习钩子

把 07 页的 environment、这一页的 gate、还有你的评测脚本并排看一遍。只要你发现某个字段在其中一层被删掉，就说明这一层已经不能再拿来解释另两层的结果了。能写出这张对照表，你才算真正把训练和上线连在一起。

落地桥接

bridge
上线闸门伪代码 
def should_go_live(metrics):
    return (
        metrics["task_success"] >= 0.9
        and metrics["policy_violation_rate"] == 0
        and metrics["rollback_success"] >= 0.99
        and metrics["audit_complete"]
    )

如果你已经有 07 页的环境类，下一步补什么

先补 audit_log、state_diff、rollback_result 和 approval_state。然后把会碰真实对象的工具拆成 dry-run 和 execute 两条路径，先跑影子模式，再做最小灰度。

第一次最常见的失败是共享环境实例和漏记审计字段。只要这两项还在，reward 再高也不能说明可上线。

你现在要改什么 为什么 先验失败
环境类的 reset 清空上一轮的脏状态 下一轮继承旧 reward、旧审计或旧对象 id
工具方法 把动作和副作用写清楚 模型只会看到一个泛化的 step，学不会选工具
reward_func 从环境状态读回结果 只看文本表面，reward hacking 很快出现
评测脚本 把闸门和回退条件固定下来 任务成功率看起来高，但危险路径没人拦

常见错误

mistakes

最常见的错不是模型太笨，而是评测把危险藏起来了。下面这几条，优先按顺序排查。

错误 为什么危险 更稳的做法
用生产凭证跑评测 一次错误轨迹就可能触发真实数据修改或真实通知。 用测试身份、测试租户、测试副本和外发拦截。
只评最终答案 模型可能走危险路径拿到正确结果，副作用被遮住。 评完整轨迹、工具参数、状态差异和权限合规。
把人工确认放在动作后面 高风险动作已经发生，再确认只是确认既成事实。 在不可逆调用前展示对象、影响和回退方案。
把影子模式当成真实上线 看到了分布，不等于看到了副作用。 影子后面接最小灰度，再决定是否放大。
没有停止条件 异常率上升时系统继续扩大流量，风险会被放大。 为越权、工具错误、审计缺失、回滚失败设阈值。

先知道，先别追

boundary

这一页先不追异步 rollout、分布式扩容、复杂 IAM、奖励模型重写和自动化事故复盘。它们都重要，但要等你已经能读懂状态台账、闸门表和故障日志再展开。

如果你现在只想继续往前走，先回头检查三件事：动作有没有碰到真实对象，日志有没有记下 state diff 和审批，出错以后有没有回退。

练习

diagnostic
作答目标

把答案落到动作、状态、证据和回退四个对象上。别只写“要安全”，要写出安全由什么组成。

synthesis矩阵

为“创建退款草稿”设计 4 格评测矩阵。

至少包含：正常退款、金额超限、缺少审批、工具超时。

先写场景，再写允许动作、禁止动作和观察指标。不要只写“能不能成功”。

参考答案 / 评分要点

合格答案应该把每一格都写成“场景 -> 动作 -> 指标”的关系。例如正常退款看草稿是否正确；金额超限看模型是否拒绝或降级；缺少审批看它是否先闸门而不是直接写入；工具超时看它是否停住并回退。只写“正常/异常”不够，要能对应到动作和回退。

synthesis切换

综合解释：用 120-180 字讲清 SFT 到环境 RL 的切换。

必须说到 prompt-only、reset、observation、reward_func 和 episode。

先给直觉，再给术语，最后说你现在能做什么。

合格答案要说明：prompt-only 适合静态样本，环境 RL 适合状态连续任务；reset 清空旧 episode 并返回初始观察；模型执行动作后得到新的 observation；reward_func 再从环境实例里读出奖励。最后要点出：只要成败取决于连续交互，就该把训练切到环境里。

清单评审

写一个 8 项以内的工具 agent 上线评审清单。

每项尽量写成“是否已经……”的问题。

参考答案 / 评分要点

清单里至少要有：工具分层、凭证隔离、状态差异、负例评测、回退演练、审计完整、影子/灰度策略、停止条件。写成判断句会比写成描述句更容易落地。

面试解释

面试中被问“工具使用 RL 怎么安全上线”，用 5-7 句话回答。

先讲风险，再讲沙盒、评测、灰度、审计和回退。

参考答案 / 评分要点

合格回答通常会先说：工具动作会改真实对象，所以不能直接在生产里探索。然后依次说沙盒、负例评测、影子模式、小流量灰度、审计和回退。最后要补一句：上线看的是副作用是否可控，不只是任务是否成功。

迁移沟通

向产品经理解释：为什么不能直接让模型在生产里继续探索？

用业务风险语言解释，不只说技术安全。

参考答案 / 评分要点

答案里至少要说清：真实成本会被消耗、用户会被直接触达、数据会被污染、出错后回滚成本会升高。最后最好补一句，为什么离线学习、影子模式和灰度是更稳的替代方案。

综合解释Feynman

用“任务意图 → 工具行动 → 环境副作用 → 评测与回退”串起本页。

像教别人一样解释，至少包含一个具体工具例子。

参考答案 / 评分要点

合格答案要把四段链条讲清楚，并且至少举一个具体工具例子，比如发邮件、建工单或改文档。要点不是把名词串起来，而是说出每一步会留下些什么、哪里会出副作用、什么时候需要回退。
