# 核心讲解 {#lesson}
<!-- tag: chain -->
<!-- layout: grid2 -->

把场景想成一个邮件助理。用户说“把昨天那个退款单通知发给客户”，模型会先查订单，再生成草稿，最后选择发出。这里最容易被忽略的点它到底碰的是测试租户还是生产租户，写的是草稿还是正式外发，记录的是一条可回滚的审计，还是一条只剩最终结果的日志。
同一个动作名，在沙盒里可能只是一次练习，在生产边上就会变成真实副作用。评测要做的事，是把这两种情形拆开：看清动作目标、看清状态变化、看清是否留下回退证据。只要这三件事里有一件缺位，分数再漂亮也不能说明系统可上线。

这个链条要依次看什么
先看对象有没有指错，接着看动作有没有真的提交，再看审计和回退有没有被写进记录。顺序一乱，问题就会被藏进“看起来成功”的结果里。

## box warn
### 判定时的四个锚点
- 目标对象是不是当前回合的对象。
- 凭证和租户是不是测试范围。
- 状态差异有没有被明确记下来。
- 回退路径是不是在动作前就能找到。
如果这四个锚点里有一个空着，就要先停在沙盒，不要把它误当成“可以先试试”的灰度。

步骤
你看到什么
它说明什么
失败会怎样


dry-run
测试租户、假收件人、无提交
先验证语义和参数
如果仍有外部写入，环境本身就不干净


shadow mode
真实输入、无提交
先看分布和策略命中
只能说明看见了流量，不能说明副作用可控


small gray rollout
少量真实流量
看回退、审计、异常处理
如果停止条件缺失，问题会跟着流量一起扩大


full live
全量真实流量
只在闸门全部通过后才考虑
没有审计和回退，就等于把风险放到最大

一个能看出副作用的壳
复制

`def run_with_gates(env, action):
 before = env.snapshot()
 result = env.step(action)
 after = env.snapshot()
 diff = after.diff(before)
 audit = {
 "tenant": env.tenant_id,
 "target": action.target,
 "diff": diff,
 "approved": env.approval_state,
 "rollback": result.rollback_available,
 }
 env.audit_log.append(audit)
 if diff.has_external_side_effect and not audit["approved"]:
 raise RuntimeError("unapproved side effect")
 return result`

## box info
### 失败会怎么暴露
如果 `snapshot()` 读的是共享对象，diff 会带着上一轮的痕迹；如果 `approval_state` 没进日志，事故后就查不出是谁放行；如果 `rollback_available` 永远写真，闸门就会高估稳定性。
这个链条的目标很朴素：每一步都能看见对象变化，出错时能停，回头还能说明白。

## box soft
### 练习钩子
拿 07 页的环境类改一改：先把 `snapshot()` 和 `audit_log` 加上，再想哪些动作只能跑 dry-run，哪些动作可以进入最小灰度。你写出来以后，应该能一眼看出哪一步最容易把真实对象误写进去。

### 把案例改成练习

# 术语与分层 {#terms}
<!-- tag: terms -->

只读查询
不改外部对象，但仍可能越权读取、泄露数据或消耗配额。


可撤销写入
会改状态，但能回滚。适合先灰度，不适合直接放到生产对象上。


不可逆动作
外发邮件、真实删除、资金变更，出了问题不能只靠“再来一轮”修复。


影子模式
看见真实输入，但不提交动作。它适合找分布差异，不适合证明万无一失。


灰度
把风险缩到小流量里，先看副作用，再决定要不要放大。


审计
记录动作、参数、状态差异、批准和回退结果。没有审计，事后基本说不清。

动作层级
典型动作
默认处理


只读
查公开文档、查状态、查日志
先看越权和隐私，再看结果是否正确。


可撤销写入
建测试工单、写草稿、调试记录
先进沙盒，再看是否有回退路径。


不可逆外发
发给真实客户、删真实数据、触发付款
先加人工闸门，再做影子或最小灰度。

# 状态台账 {#ledger}
<!-- tag: state -->

这里最重要的环境类在每个 episode 里到底读什么、写什么、什么时候会脏。只要状态台账一乱，后面的 reward 和评测都会跟着偏。

最小守门骨架
复制

`class TaskEnv:
 def __init__(self):
 self.reward = 0.0
 self.done = False
 self.audit_log = []
 self.last_diff = {}

 def reset(self, **kwargs):
 self.reward = 0.0
 self.done = False
 self.audit_log.clear()
 self.last_diff = {}
 return "initial observation"

 def act(self, tool_name, **params):
 if self.done:
 raise ValueError("episode finished")
 result = self.client.call(tool_name, **params)
 self.last_diff = result.state_diff
 self.audit_log.append({
 "tool": tool_name,
 "params": params,
 "diff": result.state_diff,
 })
 self.reward = float(result.reward)
 self.done = bool(result.done)
 return result.observation

def reward_func(environments, **kwargs) -> list[float]:
 return [env.reward for env in environments]`

## box warn
### 先盯这四个字段
`reward`、`done`、`audit_log`、`last_diff` 是最容易出错的地方。`reset()` 不清空，下一轮会继承上一轮的痕迹；`act()` 不写差异，评测就只剩表面结果；`reward_func()` 只读文本，就很容易被钻空子。
真正该检查的调用前后状态有没有变脏，评测有没有读到该读的那部分。

步骤
读取
写入 / 清空
脏了会怎样
下一步检查


`reset()`
任务参数、测试对象、可选上下文
清空 reward、done、audit_log、last_diff
下一轮继承上一轮状态，reward 泄漏
先确认每个 episode 都是新状态


公开工具方法
动作参数、当前状态
写入 state diff、观察、错误信息
工具语义不清，模型学不会选动作
看方法名、参数和 docstring 是否具体


`reward_func()`
环境实例列表和 episode 状态
输出可比较的 reward
只读文本 heuristics，reward hacking 很快出现
确认 reward 真的是从环境状态读回来的


trainer update
完成轨迹、奖励、参考策略差异
更新模型参数
前面三步接错，更新方向也会错
先把日志、回退和审计补齐再训练

# 评测闸门 {#gate}
<!-- tag: gate -->
<!-- layout: grid2 -->

$$
 \text{GoLive} = \text{Success} \land \neg \text{Violation} \land \text{Rollback} \land \text{Audit}
 $$
 `Success` 看任务是否完成，`Violation` 看有没有越权，`Rollback` 看出错后能不能停住，`Audit` 看证据链是否完整。这里在写一个能停机的条件。

## box
### 一票否决的信号
- 生产凭证跑评测。
- 动作已经触达真实对象，却没有回退。
- 审计字段空着，日志只能看见最终答案。
- 异常率上升时系统还继续放量。

指标
看什么
不合格说明什么
下一步


task_success
任务有没有做成
任务本身就没跑通
先修功能，不要先谈放量


policy_violation_rate
有没有越权、绕过审批、触达禁区
模型会做对事，但走了危险路径
回看动作级日志和策略拦截点


rollback_success
出错后能不能停住并恢复状态
一旦出错，影响会继续扩散
先补回退演练，再考虑灰度


audit_complete
动作、参数、差异、批准是否齐
事后不好复盘，也不好追责
把审计字段写进环境和评测脚本


unexpected_diff_rate
有没有不该出现的状态变化
模型可能在隐藏路径上乱写
回到沙盒和状态台账排查

# 实现一致性 {#consistency}
<!-- tag: contract -->

沙盒、离线评测、灰度上线这三层能不能互相说明问题，关键在于它们是不是读同一份状态契约。这里的契约，不只是接口名一样，还包括对象 id、审批状态、审计记录、回退结果和副作用边界都能按同一种语义被读回来。只要其中一层把字段省掉、改名、或者改成别的含义，数字看起来还会动，但它们已经不能再证明同一件事。

这件事在 agent 场景里很常见。训练环境为了简单，可能先不接审批流；评测脚本为了快，可能只读最终结果；上线前的灰度窗口则必须把审批、审计和回退全都拉进来。三层之间如果没有一份共同契约，模型就会在“看起来成功”的评测里拿分，却在真实副作用上失控。

层级
必须共享
可以不同
不同了会怎样


训练环境
对象 id、state diff、reward、done
数据规模、样本难度、随机种子
如果契约字段缺失，后面的评测就接不上训练语义。


离线评测
审批状态、审计记录、回退结果
流量规模、批次大小、观察窗口
如果只看分数不看证据，危险路径会被当成能力提升。


灰度上线
真实对象 id、外部影响标记、停止条件
放量比例、观察时长、人工复核频率
如果回退链不在场，异常只会随着流量一起扩散。

检查契约是否一致
复制

`def assert_contract(env):
 required = [
 "object_id",
 "approval_state",
 "audit_log",
 "rollback_result",
 "side_effect_boundary",
 ]
 missing = [name for name in required if not hasattr(env, name)]
 if missing:
 raise AssertionError(f"missing contract fields: {missing}")
 if env.side_effect_boundary not in {"sandbox", "shadow", "gray", "live"}:
 raise AssertionError("unexpected side effect boundary")
 return True`

## box warn
### 为什么这里要单独检查契约
如果训练环境没有这些字段，模型学到的是“简化版世界”。如果评测脚本没有这些字段，分数就只是在读一个缩水后的世界。如果上线灰度没有这些字段，风险就会在真实对象上直接放大。三层都写全，才能让同一件事在不同阶段都能被看见。

## box soft
### 练习钩子
把 07 页的 environment、这一页的 gate、还有你的评测脚本并排看一遍。只要你发现某个字段在其中一层被删掉，就说明这一层已经不能再拿来解释另两层的结果了。能写出这张对照表，你才算真正把训练和上线连在一起。

# 常见错误 {#mistakes}
<!-- tag: mistakes -->

最常见的错评测把危险藏起来了。下面这几条，优先按顺序排查。

错误
为什么危险
更稳的做法


用生产凭证跑评测
一次错误轨迹就可能触发真实数据修改或真实通知。
用测试身份、测试租户、测试副本和外发拦截。


只评最终答案
模型可能走危险路径拿到正确结果，副作用被遮住。
评完整轨迹、工具参数、状态差异和权限合规。


把人工确认放在动作后面
高风险动作已经发生，再确认只是确认既成事实。
在不可逆调用前展示对象、影响和回退方案。


把影子模式当成真实上线
看到了分布，不等于看到了副作用。
影子后面接最小灰度，再决定是否放大。


没有停止条件
异常率上升时系统继续扩大流量，风险会被放大。
为越权、工具错误、审计缺失、回滚失败设阈值。

- [ ] . 回答字数

Answer: . policy violation rate
Explain: 越权要看策略违反率，不要看回答长不长。

### 6 debug-checklist

**Q:** 为“邮件助理 agent”勾出必须进负例集的 3 项。

- [ ] 绕过审批群发邮件
- [ ] 读取无关联系人
- [ ] 伪造发件人
- [ ] 统计字数

Explain: 必须进负例集的是： 绕过审批群发邮件 、 读取无关联系人 、 伪造发件人 。这三项分别对应越权、隐私和身份伪造。 统计字数 不该作为核心负例，因为它不构成副作用。

### 7 choice

**Q:** 哪种日志最能说明模型绕过了沙盒？

- [ ] . state diff + approval record + target object
- [ ] . answer length + token count
- [ ] . GPU temperature

Answer: . state diff + approval record + target object
Explain: 能还原状态差异和批准链的日志，最容易看出模型是不是绕过了沙盒。

### 8 true/false

**Q:** 常规任务成功率够高就可以上线。

- [ ] 对
- [ ] 错

Answer: false
Explain: 成功率不能覆盖越权、恢复和审计。

### 9 short-fill

**Q:** 填写一个能衡量恢复路径的指标。

Explain: 可以写 回滚成功率 、 安全停机率 ，或者英文 rollback success 、 safe stop rate 。关键是这个指标要看得出系统在出错后有没有停住或恢复，而不是只看有没有把任务做完。

### 10 synthesis

**Q:** 为“创建退款草稿”设计 4 格评测矩阵。

Explain: 合格答案应该把每一格都写成“场景 -> 动作 -> 指标”的关系。例如正常退款看草稿是否正确；金额超限看模型是否拒绝或降级；缺少审批看它是否先闸门而不是直接写入；工具超时看它是否停住并回退。只写“正常/异常”不够，要能对应到动作和回退。

### 11 choice

**Q:** 这段 trainer 配置最大的问题是什么？

- [ ] . 共享同一个环境实例会让 episode 互相污染
- [ ] . reward_func 不能返回浮点数
- [ ] . train_dataset 不能叫 dataset

Answer: . 共享同一个环境实例会让 episode 互相污染
Explain: 共享同一个环境实例会让 episode 互相污染。

### 12 迁移

**Q:** 一个文档 agent 可以读取、改写和分享公司文档。请给出上线前的权限分层。

Explain: 合格答案要分三层： 只读 先过权限和隐私检查； 可撤销写入 放在测试租户或可回滚草稿里； 对外分享 必须加人工闸门、审批和审计。答案里如果只写“读写分开”还不够，要说明每层的限制是什么。

### 13 choice

**Q:** 为什么影子模式适合上线前发现真实分布问题？

- [ ] . 能看见真实输入，但不提交动作
- [ ] . 可以直接改真实数据
- [ ] . 只要回答更长就行

Answer: . 能看见真实输入，但不提交动作
Explain: 影子模式能看见真实输入，但不提交动作，所以能对比分布而不制造副作用。

### 14 synthesis

**Q:** 综合解释：用 120-180 字讲清 SFT 到环境 RL 的切换。

Explain: 合格答案要说明： prompt-only 适合静态样本，环境 RL 适合状态连续任务； reset 清空旧 episode 并返回初始观察；模型执行动作后得到新的 observation ； reward_func 再从环境实例里读出奖励。最后要点出：只要成败取决于连续交互，就该把训练切到环境里。

<!-- practice-group: 四、上线表达 -->

### 15 清单

**Q:** 写一个 8 项以内的工具 agent 上线评审清单。

Explain: 清单里至少要有：工具分层、凭证隔离、状态差异、负例评测、回退演练、审计完整、影子/灰度策略、停止条件。写成判断句会比写成描述句更容易落地。

### 16 面试

**Q:** 面试中被问“工具使用 RL 怎么安全上线”，用 5-7 句话回答。

Explain: 合格回答通常会先说：工具动作会改真实对象，所以不能直接在生产里探索。然后依次说沙盒、负例评测、影子模式、小流量灰度、审计和回退。最后要补一句：上线看的是副作用是否可控，不只是任务是否成功。

### 17 迁移

**Q:** 向产品经理解释：为什么不能直接让模型在生产里继续探索？

Explain: 答案里至少要说清：真实成本会被消耗、用户会被直接触达、数据会被污染、出错后回滚成本会升高。最后最好补一句，为什么离线学习、影子模式和灰度是更稳的替代方案。

### 18 综合解释

**Q:** 用“任务意图 → 工具行动 → 环境副作用 → 评测与回退”串起本页。

Explain: 合格答案要把四段链条讲清楚，并且至少举一个具体工具例子，比如发邮件、建工单或改文档。要点说出每一步会留下些什么、哪里会出副作用、什么时候需要回退。
