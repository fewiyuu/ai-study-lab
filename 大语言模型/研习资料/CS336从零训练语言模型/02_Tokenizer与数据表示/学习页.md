# 先抓对象：这页里有哪四层表示 {#goals}
<!-- tag: object map -->

先不要把 tokenizer 理解成“切词器”这么粗的东西。这里至少有四层对象：人看到的文本、底层的 bytes、tokenizer 切出来的 token、模型和测试实际使用的 token id。很多 bug 就是因为这四层被混成一层，打印出来的东西看似对，实际上已经换了对象。

文本比如 `low lower low`。这是给人看的字符串，不是模型输入。

bytes / Unicode字符背后还有编码层。BPE 和特殊 token 的实现经常要先落到 bytes 才稳。

tokentoken 是词表里的片段，可能是词、词片段、空格加片段，甚至 byte 片段。

token id训练时真正进模型的是整数 id。id 本身不带语义，语义来自词表和 embedding。

# 从文本到 id：先看代价，再看收益 {#problem}
<!-- tag: representation -->

字符级切分最稳，但序列会很长；词级切分更像人读法，但遇到新词、拼写变化、多语言和符号时会掉得很快。BPE 这类子词方法折中在中间：常见片段会被合并短，少见片段还能退回更小单位拼出来。

切分层优点代价对训练的影响

字符简单，覆盖广序列长上下文窗口更容易被吃满
词直观，片段少新词和变体麻烦词表外问题会变多
字节 / 子词可逆，能兜底实现更细，序列可能变长词表、长度和计算成本要一起看

$$\text{总成本} \approx \text{序列长度} \times \text{每步计算} + \text{词表大小带来的参数成本}$$
    这不是精确公式，只是一个检查框架：token 越碎，序列越长；词表越大，embedding 和输出层参数越多。比较 tokenizer 时不能只看“词表大不大”。

# bytes 与 Unicode：先保证可逆，再谈合并 {#unicode}
<!-- tag: encoding -->

Assignment 1 的 tokenizer 需要同时处理 ASCII、Unicode、特殊符号和长文本。这里最稳的做法，是先把文本放到一个可逆的底层表示上，再做 BPE 合并和 special token 匹配。这样不会因为字符集不同，整条链路就断掉。

层读什么写什么错了会怎样

bytes原始文本编码后的字节可逆基础符号解码回不去原文
Unicode 字符人类可读文本字符边界多语言和表情符号容易出问题
tokenbytes / 字符 / special token词表片段切分边界和测试对不上

# 完整走一遍：一条文本怎样穿过 tokenizer {#trace}
<!-- tag: trace -->

拿一个测试里很常见的输入来走：`Héllò hôw <|endoftext|> are ü? 🙃`。这类文本有重音、空格、special token 和表情符号，刚好能把 tokenizer 的关键分支都撞一遍。你要看的不是“结果像不像”，而是每一步到底在保护什么。

步骤输入对象输出对象看什么一旦错了会怎样

1. 原文`Héllò hôw <|endoftext|> are ü? 🙃`字符串文本里同时有普通词和 special token后续如果把 special token 当普通文本，边界就会乱
2. 编码字符串bytes / Unicode 片段重音字符和表情符号必须还能回去若不保留可逆性，decode 会丢字符或变形
3. special 匹配字节流完整的 `<|endoftext|>`这个标记要被保留成一个整体被切成几段后，训练和 decode 都会失真
4. BPE 合并基础片段更短的常见 token常见片段会更像词片段，不常见片段继续拆pair 统计或 merge 顺序错了，词表就不对
5. id 映射tokentoken id每个 token 要对应一个稳定整数同一 id 如果语义漂移，模型会查错 embedding
6. decodetoken id 序列原文回到原字符串，special token 也要回来如果回不去，测试会在 roundtrip 里直接爆掉

对应的检查顺序复制

```python
text = "Héllò hôw <|endoftext|> are ü? 🙃"
ids = tokenizer.encode(text)
roundtrip = tokenizer.decode(ids)

# 1) 先看 special token 有没有被保留
# 2) 再看 ids 是否稳定
# 3) 最后看 roundtrip 是否回到原文
```

这就是这页最实用的诊断顺序：先确认 special token 没被拆，接着看 id 序列有没有稳定，再看 decode 能不能回到原文。如果第 1 步就错了，后面 BPE 再对也没用；如果第 1 步没错但第 3 步错了，通常是 bytes / Unicode 的可逆性出了问题。

## box warn
### 这条链路的失败含义
如果能 encode 但不能 roundtrip，问题多半不在模型，而在 tokenizer 本身；如果 special token 被拆碎，问题常出在最长匹配或分块边界；如果 `ids` 稳定但测试仍失败，通常要回头看词表和 merges 的顺序。

# 前导空格为什么重要 {#spacing}
<!-- tag: spacing -->

很多 GPT 风格 tokenizer 会把“单词前的空格”当成 token 的一部分。这个设计看起来别扭，但很实用：它让 `Hello` 和 `Hello` 不是同一个片段，模型就能区分“句首的 Hello”和“句中的 Hello”。这也是为什么你在测试里会看到 `["Hello", ",", " how", " are", " you", "?"]` 这类结果，而不是把空格丢掉后再拼回去。

输入token 读法你应该注意什么错了会怎样

`Hello`句首词片段没有前导空格和句中同词可能被当成不同对象
` Hello`带前导空格的词片段空格本身承载了边界信息如果把空格删掉，token 序列会变短但语义边界丢了
`Héllò hôw`重音字符 + 带空格片段Unicode 和空格会同时参与切分encode / decode 和测试都可能错位

空格和 token 的最小对照复制

```python
text_a = "Hello, how are you?"
text_b = " Hello, how are you?"

ids_a = tokenizer.encode(text_a)
ids_b = tokenizer.encode(text_b)

# 常见现象：
# text_b 的 token 往往会把前导空格保留下来，
# 所以它和 text_a 的 token 序列不一样。
```

这不是“多余的空格被保留了”，而是 tokenizer 在保留边界。对训练来说，空格是文本结构的一部分；对模型来说，它告诉模型哪些 token 是句首、哪些 token 是句中延续。你如果在预处理时手滑去掉空格，测试看起来可能还能过一部分，但 token 序列、pair 统计和 BPE 结果都会悄悄偏掉。

## box warn
### 一眼能查的错误
如果 `encode` 后的 token 明显少了一个前导空格片段，先查文本清洗和规范化；如果只有某些语言或标点出问题，再回头看 bytes / Unicode 的分支。这个错误和 BPE 本身不一定有关。

# 手算 BPE：先统计 pair，再看合并顺序 {#bpe}
<!-- tag: BPE -->

拿 `low lower low` 做玩具语料，把空格记成 `_`。起点先按字符切：`l o w _ l o w e r _ l o w`。BPE 每一轮只做一件事：统计相邻 pair，挑最高频的一对合并。它不看语义，只看局部频率。

轮次当前序列高频 pair合并后

0`l o w _ l o w e r _ l o w``(l,o)` 与 `(o,w)` 都出现 3 次需要 tie-break；这里假设先合并 `l+o -> lo`
1`lo w _ lo w e r _ lo w``(lo,w)` 出现 3 次`low _ low e r _ low`
2`low _ low e r _ low``(low,_)` 出现 2 次常见边界也会继续短化

pair 计数的最小代码块复制

```python
from collections import Counter

def count_pairs(tokens):
    pairs = Counter()
    for a, b in zip(tokens, tokens[1:]):
        pairs[(a, b)] += 1
    return pairs
```

读这段代码要盯住两件事：`zip(tokens, tokens[1:])` 只生成相邻窗口；`pairs[(a, b)] += 1` 在累计 pair 频率。pair 统计一旦错，后面的 merge 顺序就全错。

# special token：最长匹配和边界保留 {#specials}
<!-- tag: specials -->

special token 不能像普通文本那样被拆开再合并。它们是 tokenizer 里单独保留的标记，用来表示结束、分隔、系统消息、padding 或其他约定。遇到 special token 时，最重要的是保持它的完整性和可逆性。

## box info
### 常见例子
`<|endoftext|>`、padding token、文档边界 token、聊天模板里的角色标记。

## box warn
### 容易错的地方
如果 special token 被切碎，后面的 encode / decode 可能还能跑，但语义已经变了，测试也会失配。

环节读什么写什么错误症状

encode原文和 special token 列表完整保留 special token 的 id字符串会被拆碎成普通片段
decodetoken id 序列还原成原文边界符号丢失或重复
train_bpe训练语料与 special token词表和 mergesspecial token 被合并走

# 分块预处理：大语料为什么要先找边界 {#chunking}
<!-- tag: pre-tokenization -->

大语料不能每次都整文件读进去再处理。Assignment 1 里的预处理示例会先估算若干 chunk，再向后扫描到 special token 边界。这样每个进程都能独立处理自己的 chunk，不会把一个特殊标记拦腰切开。

# 流式编码：encode 和 encode_iterable 不是一回事 {#iterable}
<!-- tag: streaming -->

在小文本上，`encode` 和 `encode_iterable` 看起来像两种写法；在大文本上，它们的意义完全不同。`encode` 通常先把整段文本读进内存，再返回一串 id；`encode_iterable` 则要一边读一边产出 token id，避免为了一个测试样本就把整份语料塞进内存。

Assignment 1 里这不是“可选优化”，而是测试边界的一部分。`test_encode_iterable_memory_usage` 会把输入文件限制到很小的内存预算里；如果你的实现把流式接口写成了“先拼成完整字符串，再统一 encode”，这个测试会先爆在内存，不会等到 BPE 算完。

接口读什么写什么适合什么场景典型失败

`encode`完整字符串整段 token id 列表小文本、一次性处理、调试 roundtrip内存够用，但大文件会吃满
`encode_iterable`文件对象或可迭代文本流逐个 token id大语料、分块计数、低内存测试如果先拼整串文本，内存限制会先失败

流式接口的最小读法复制

```python
with open(FIXTURES_PATH / "tinystories_sample.txt") as f:
    ids = []
    for token_id in tokenizer.encode_iterable(f):
        ids.append(token_id)

# 和 encode 的差别在于：
# 这里直接消费文件流，不先把整份文本拼成一个大字符串。
```

## box warn
### 如果把它写成 encode + for-loop，会坏在哪里
短文本上看不出来，长文本上就会把“流式处理”退化成“先全读进内存再处理”。测试用的 5M 语料正是为这个错误准备的；它不会告诉你 BPE 错了，只会告诉你内存用法不对。

这节和上面的分块预处理连在一起看最有用：分块负责把大文件切成可处理的边界，`encode_iterable` 负责在这个边界上逐步吐出 token id。一个管“切哪里”，一个管“怎么读”，两者合起来才是大语料路径。

# id 到 batch：next-token 训练真正吃什么 {#ids}
<!-- tag: batch -->

Tokenizer 输出的是一长串整数 id，但训练还没开始。next-token 任务要把这串 id 切成输入和目标：`x` 是前面的部分，`y` 是右移一位后的目标。模型在每个位置预测“下一个 token 是谁”。

右移一位的最小样本复制

```python
tokens = [101, 203, 44, 19, 8]
x = tokens[:-1]  # [101, 203, 44, 19]
y = tokens[1:]   # [203, 44, 19, 8]
```

位置x 看到什么y 期待什么读法

0`101``203`看到第一个 token，预测下一个。
1`203``44`上下文再多一位，目标继续右移。
2`44``19`每个位置都形成一个分类任务。
3`19``8`右移一位后，最后一个目标仍然存在。

这一步和 03 页的训练循环直接接上：模型输出的是每个位置对词表的分数，loss 用 `y` 来监督。这里如果错位，后面所有训练都在认真学错误目标。

# 排错入口：tokenizer 出问题通常坏在哪里 {#mistakes}
<!-- tag: mistakes -->
<!-- layout: grid -->

可逆性坏了encode 和 decode 互相对不上，优先查 bytes / Unicode 处理，不要先怪 BPE。

special token 被切碎先检查最长匹配和边界扫描。special token 不是普通字符片段。

merge 顺序不对pair 统计和 tie-break 有问题时，词表可能还能长出来，但顺序已经偏了。

速度太慢常见原因是反复扫描、重复分配和 chunk 切分不合理，而不是“Python 就是慢”。

batch 错位把 `x` 和 `y` 对错位，等于训练了另一个任务。

词表成本没算清词表过大和序列过长都要付钱，比较 tokenizer 时要同时看参数和长度。

### 1 解释

**Q:** Q1. 区分文本、bytes、token、token id。

Explain: 文本是人看到的字符串；bytes 是编码后的底层字节；token 是 tokenizer 词表里的片段；token id 是 token 在词表里的整数编号。训练循环真正吃的是 token id，tokenizer 定义 token 边界和映射。

### 2 判断

**Q:** Q2. 判断：tokenizer 只负责切分，不影响训练成本。

- [ ] 对
- [ ] 错

Answer: false
Explain: tokenizer 会改变序列长度、词表大小、embedding/output 参数量和 attention 的计算长度。

### 3 解释

**Q:** Q3. 为什么要先经过 bytes / Unicode，而不是直接做 BPE？

Explain: 先有 bytes / Unicode，是为了把不同语言和符号放到一个可逆、稳定的底层表示上。这样 BPE 处理的对象更统一，special token 也更容易保留边界。直接对高层文本做 BPE，很多字符集和边界情况会变得不稳。

<!-- end-practice-group -->

### 4 手算

**Q:** Q4. token 序列 `a b a b a` 中最高频 pair 是什么？

Explain: 相邻 pair 是 (a,b) 、 (b,a) 、 (a,b) 、 (b,a) 。两种 pair 都出现 2 次，存在并列最高频。若选择合并 a b 为新 token ab ，序列可变成 ab ab a 。

### 5 代码阅读

**Q:** Q5. `ids=[4,5,6,7]`，`context=3`，`x` 和 `y` 分别是什么？

Answer: x=[4,5,6]; y=[5,6,7]
Explain: y 是 x 右移一位后的目标 token。

### 6 代码阅读

**Q:** Q6. 读 `find_chunk_boundaries` 这类代码，它读什么、写什么、返回什么？

Explain: 函数读取文件和 special token；根据文件大小先估边界，再向后扫描到 token 边界；写出每个 chunk 的 start / end；返回一组可用于并行处理的边界。如果 special token 被切开，后面的计数、合并和测试都会错。

### 7 对比

**Q:** Q7. 词表变大可能带来哪两个好处和哪两个代价？

Explain: 好处：常见片段能用更少 token 表示，序列可能变短；高频词片段的表示更稳定。代价：embedding 和 output head 随词表变大；低频 token 更难学好；词表过细或过粗都会影响泛化和成本。

### 8 场景迁移

**Q:** Q8. special token 在训练和 decode 里分别起什么作用？

Explain: 训练中，special token 可以标记文档边界、结束符或模板位置，让模型学习何时该停。decode 里它们会决定输出如何被还原成文本。若 special token 缺失或被拆碎，样本边界会乱，解码结果也会失真。

### 9 错误诊断

**Q:** Q9. 训练好的 tokenizer 可以直接换一套词表吗？

Explain: 不能直接换。模型学到的是旧词表下 id 到 embedding、hidden state 和输出词表的关系；新词表会改变 id 的语义，special token 也可能变。通常需要重新训练或至少做专门适配。

### 10 场景诊断

**Q:** Q10. 训练 loss 看起来下降，但生成质量不稳定，先查什么？

Explain: 先查 x/y 是否右移正确、special token 是否被保留、分块边界是否切断 token、训练和生成时的 tokenizer 是否一致。loss 下降只说明当前任务上的 next-token 预测变好了，不代表所有下游生成都稳。

### 11 综合解释

**Q:** Q11. 用 6-8 句话讲清楚文本如何变成训练 batch。

Explain: 答案应从原始文本开始，说明文本会被编码成 bytes / Unicode 片段，再由 tokenizer 和 BPE 合并成 token，token 查词表得到 token id。长文本会按边界切成 chunk，special token 要保留完整；训练时按 context window 切出固定长度片段，再构造 x = ids[:-1] 和 y = ids[1:] ，让模型在每个位置预测下一个 token id。

### 12 诊断

**Q:** Q12. 看到 special token 被切碎时，你会先查哪三件事？

Explain: 先查 tokenizer 是否按最长匹配保留 special token；再查大文件分块时是否把边界切到了 special token 中间；最后查 encode / decode 是否对同一个特殊标记保持对称。只改 merge 顺序通常不够。

### 13 多选

**Q:** Q13. 真实文本追踪里，先查哪一步最能快速定位问题？

- [ ] special token 是否被完整保留
- [ ] id 序列是否稳定
- [ ] roundtrip 是否回到原文
- [ ] 先把词表扩大一倍

Explain: 前三项都应先查。这个追踪表的目的就是先确认 special token、id 序列和 roundtrip 是否稳定；词表扩大不是第一动作。

### 14 多选

**Q:** Q14. 哪些检查应该放在 special token 被切碎之前？

- [ ] 先查 longest-match 规则
- [ ] 先查 chunk 边界是否切断了 special token
- [ ] 先查 encode / decode 对称性
- [ ] 先把词表扩大到足够大再说

Explain: 前三项都应该先查。词表变大不是 special token 被切碎时的第一处理动作；先确认匹配规则、边界和对称性，往往更快定位问题。
