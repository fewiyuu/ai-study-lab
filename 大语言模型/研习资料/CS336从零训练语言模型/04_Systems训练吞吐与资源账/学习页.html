先看一笔资源账

goal

上一单元里，训练循环看起来像一串 Python 调用：forward、loss、backward、optimizer step。到了 Systems，问题变成：这些调用在 GPU 上到底消耗了什么资源，哪些资源会让吞吐停住。吞吐可以先理解为单位时间完成多少训练工作，例如 tokens/s、samples/s 或 step/s。资源账则是在问：一轮 step 的时间花在算矩阵乘、读写显存、跨设备传张量，还是等待同步。

先拿一个小 batch 做锚点。假设模型有两张 GPU，每张卡各跑 10 条样本；一次 step 里会出现参数、激活值、梯度、optimizer state、attention 中间量、跨 rank 通信缓冲区。模型数学上没有变，但这些对象的存放位置和移动时机变了。Assignment 2 的测试正是在检查这些对象是否被正确计算、保存、同步和还原。

$$ T_{\mathrm{step}} \approx T_{\mathrm{compute}} + T_{\mathrm{memory}} + T_{\mathrm{comm}} + T_{\mathrm{sync}} $$

T_compute 是矩阵乘、softmax、归一化等算术时间；T_memory 是读写 HBM、保存激活、读 optimizer state 的时间；T_comm 是 all-reduce、all-gather、reduce-scatter 这类跨设备传输；T_sync 是某个 rank 或 kernel 等其它工作完成的空等时间。真实 profile 会更复杂，但这四项足够帮你读 Assignment 2 的第一批瓶颈。

对象 什么时候出现 主要占哪笔账 常见错误信号
parameters forward 前就存在，optimizer step 后更新 显存；FSDP 下还涉及 all-gather rank 之间参数不等价，full params 还原失败
activations forward 产生，backward 读取 显存和读写带宽 backward 需要的张量没保存，梯度对不上
attention scores / LSE attention forward 与 backward 之间 softmax 计算、显存流量 输出接近但梯度不对，或 saved tensor shape 不对
gradients backward 逐层产生 显存；DDP/FSDP 下涉及通信 optimizer step 前没有同步，parallel baseline 不等价
optimizer state optimizer 初始化后逐步积累 显存；sharded optimizer 下涉及所有权 分片更新后参数和非分片 AdamW 不一致

这一页先不追 CUDA 线程块、pipeline parallel、ZeRO 细节和 kernel 级别调优。先把“资源账”立住，你才知道后面哪类优化是语义必须，哪类优化只是性能加分。

你现在应该能做什么：看到一个训练变慢或测试失败的现象，先把它归到“算、存、传、等”中的一类或几类，再去查对应的公式、通信或同步点。

瓶颈怎么读

profile

profile 和 benchmark 是两种不同问题。benchmark 问“这段代码每秒能处理多少工作”；profile 问“时间具体花在哪些 kernel、哪些内存拷贝、哪些同步点上”。当 tokens/s 低时，不要只看 GPU utilization 一个数字。一个训练 step 可能算力很忙，也可能看起来 GPU 很忙但主要在搬数据，或者每张卡都能算却互相等待通信。

读 profile 时先按可观察信号分组。矩阵乘和卷积一类 kernel 时间高，通常先看形状、batch、融合和 Tensor Core 友好程度；大量小 kernel 串行出现，先看 Python 调度、算子切分和能否融合；memcpy 或显存读写占比高，先看中间张量是否太大、是否重复 materialize；collective 操作占比高，先看 DDP/FSDP 的通信时机和数据量。

forward 账compute + memory

Transformer block 会读参数、写激活。attention 如果显式保存完整 score matrix，序列长度一上来，显存和带宽都会先吃紧。

backward 账saved tensors + gradients

backward 既要重放依赖关系，也要读取 forward 保存的关键张量。FlashAttention 测试要求保存每个 query 的 log-sum-exp，因为 backward 要用它恢复 softmax 归一化。

通信账rank 之间移动张量

DDP 通常同步梯度；FSDP 还要在 forward/backward 周围 gather 参数、在 backward 后 reduce-scatter 梯度。通信放错位置，数值可能对不上，也会让设备空等。

更新账parameter + state

AdamW 这类优化器会维护额外 state。sharded optimizer 的目标是让每个 rank 只维护自己负责的 state，同时更新效果仍等价于非分片优化器。

症状 先怀疑 下一步检查
tokens/s 低，矩阵乘 kernel 占大头 算力账 看 shape 是否让 GEMM 太小或太碎，batch/seq 是否合理
显存接近上限，OOM 或 batch 被迫很小 显存账 看 activation、attention score、optimizer state 是否可重算、分片或融合
多卡 step 比单卡快得很少 通信账 看 all-reduce、all-gather、reduce-scatter 的数据量和是否阻塞计算
GPU 利用率忽高忽低 同步账 看 rank 是否数据不均、某个 collective 是否等待慢 rank

你现在应该能做什么：拿到一个 profile 截图或 benchmark 数字时，先写出“慢在哪一笔账”，再决定该看 kernel、显存、通信还是分布式等价性。

对象怎么流

ledger

这一节把一次 step 里最容易混淆的对象摆成账本。你不用先背 DDP 或 FSDP 的名词，先看对象在 forward、backward、同步和更新时分别由谁读、由谁写。只要这张表没对上，后面的代码和测试就会一直错在同一类地方。

锚点还是那组最小情形：2 张 GPU，每张卡各看 10 条样本。rank 之间看见的数据不一样，但最后都要得到同一套等价参数。差别只在参数、梯度、optimizer state 和通信缓冲区的归属。

一次 step 的时序
for batch in loader:
    logits = model(inputs)      # 读参数，写激活
    loss = criterion(logits, targets)
    loss.backward()             # 读 saved tensors，写梯度
    sync_or_reduce(model)       # DDP / FSDP 在这里处理通信
    optimizer.step()            # 读梯度和 state，写参数和 state
阶段 读什么 写什么 出错时最像什么
forward parameters、输入 batch、必要的 mask activations、attention 中间量、LSE 输出值错、shape 对不上、saved tensor 不够
backward saved tensors、loss 依赖、局部梯度 gradients forward 对了，梯度不对，或某个参数没有 grad
after backward 本地梯度、通信缓冲区、分片信息 同步后的梯度或 shard 梯度 多卡参数不等价、reduce-scatter 形状不对、通信没等完
optimizer.step() 梯度、optimizer state、当前参数 更新后的参数和 state 和非并行 baseline 的权重越走越偏
next iteration 更新后的参数 新的 activations / 新的梯度 上一步的同步问题会在下一轮放大

这张账本还有一个实用点：它能帮你分清“模型语义错了”和“对象生命周期错了”。前者常见于 attention 公式、mask、scale；后者常见于 DDP/FSDP 的同步时机、参数归属、dtype 以及 optimizer state 的位置。

你现在应该能做什么：拿到一个训练问题时，先写出哪个对象在什么阶段被读写，再决定要查 forward、backward 还是同步。

先知道，先别钻

boundary

这一页要先立住的是训练 step 的资源账、attention 的 LSE、DDP 的同步时机、FSDP 的 shard / full 归属。下面这些概念会在后面的系统优化和实现细节里再回来，现在先放一边。

先放一边的概念 现在怎么处理 后面再回来时看什么
CUDA 线程块 / warp 先只把它看作 kernel 级别的性能细节 kernel 为什么慢、为什么要分块读写
pipeline parallel 先不纳入本页的资源账 跨 stage 的等待和负载不均
ZeRO stages 先和 FSDP 的 shard / gather / scatter 区分开 参数、梯度、optimizer state 的更细粒度分工
activation checkpoint 先把它当作“用计算换显存”的下一层手段 哪些中间量可以重算，哪些必须保存

你现在应该能做什么：看到这些词时，先知道它们属于后续优化层，不要把本页的语义问题和更深的性能问题混在一起。

attention 变慢在哪里

attention

标准 scaled dot-product attention 的公式很短，但朴素实现会产生一个形状为 batch × n_queries × n_keys 的 score matrix。序列越长，这个矩阵越大；如果还把 softmax 概率 P 也保存下来，forward 看似直接，backward 会把显存和内存带宽都推高。FlashAttention2 的动机之一，就是把 attention 当作“如何分块读写、如何少存中间量”的系统问题。

$$ S = \frac{QK^\top}{\sqrt d},\qquad P = \operatorname{softmax}(S),\qquad O = PV,\qquad L_i = \log \sum_j e^{S_{ij}} $$

Q、K、V 的最后一维是 head dimension d。S 是每个 query 对每个 key 的分数；L_i 是第 i 个 query 行的 log-sum-exp。Assignment 2 的 attention 测试要求 autograd forward 保存一个形状为 (batch_size, n_queries) 的 LSE 张量，用它在 backward 中稳定地恢复 softmax 相关量。

测试里 attention 的语义表面
batch_size = 4
n_queries = 128
n_keys = 128
D = 64
q = torch.randn(batch_size, n_queries, D, requires_grad=True)
k = torch.randn(batch_size, n_keys, D, requires_grad=True)
v = torch.randn(batch_size, n_keys, D, requires_grad=True)

o = impl(q, k, v, is_causal)
maybe_lse = [t for t in o.grad_fn.saved_tensors if t.shape == (q.shape[0], q.shape[1])]

assert len(maybe_lse) == 1
torch.testing.assert_close(o, o_ref, rtol=1e-2, atol=1e-2)

这段测试给了三个实现约束。第一，函数返回的是 O，形状应沿用 query 位置：(batch_size, n_queries, D)。第二，causal mask 需要保留；开启时第 i 个 query 不能看未来 key。第三，forward 不能只返回正确输出，还要把 backward 需要的稳定中间量保存到 autograd context。输出对了但 LSE 没保存，测试也会失败，因为 backward 无法用同一套数值路径复现梯度。

失败现象 先看哪里 原因
forward 输出和 reference 不接近 scale = 1 / sqrt(d)、mask、softmax 维度 score matrix 的数值语义错了，后面再省显存也没意义
提示没有找到 LSE saved tensor ctx.save_for_backward(...) 保存对象 测试需要一个 (batch_size, n_queries) 的 log-sum-exp 张量
forward 过了，backward 梯度不匹配 dQ、dK、dV 推导和保存张量 backward 需要 forward 的归一化信息，不会单独再做一次 softmax
Triton 测试被跳过 是否有 CUDA GPU 这是环境限制，不等于 PyTorch 语义实现已经失败

你现在应该能做什么：在写 FlashAttention2 的 PyTorch 版本之前，先用朴素公式确认输出、LSE 形状和 backward 梯度三件事；再把“少存大矩阵、分块读写”的优化放进 Triton 版本。

DDP 通信检查什么

distributed

Distributed Data Parallel 的基本想法是：每个 rank 保留一份完整模型，吃不同的数据切片，backward 后把梯度同步到等价于“单个模型看完整 batch”的结果。这里最容易犯的错，是把“每张卡都能 backward”误认为“多卡训练已经等价”。如果 optimizer step 前梯度没有同步，两个 rank 会沿不同方向更新，几步之后参数就不再相同。

Assignment 2 的 DDP adapter 暴露两个入口：get_ddp(module) 返回包装后的模块，负责参数广播和梯度同步机制；ddp_on_after_backward(ddp_model, optimizer) 在 backward 结束、optimizer step 之前运行，用来完成或等待梯度同步。测试用两类 toy model，包括 tied weights，目的是让你处理共享参数、非训练参数和 rank 初始化差异。

测试阶段 它在验证什么 实现含义
初始化后 rank 0 参数广播到其它 rank；不需要梯度的固定参数保持不变 get_ddp 不能只保存 module，还要建立一致初始状态
非并行 baseline 更新后 baseline 已经 step，DDP 还没 step，因此 rank 0 上参数应暂时不同 测试还会看训练时序是否合理
DDP backward 后 每个 rank 只看本地 10 条样本，但同步后梯度应等价于完整 20 条样本 同步通常要做平均或等价缩放，不能只把梯度相加后直接 step
DDP step 后 DDP 参数和非并行 baseline 再次相等 ddp_on_after_backward 的时机必须在 optimizer step 之前

通信重叠是 DDP 的性能点：梯度在 backward 中逐层产生，越早准备好的梯度越早可以异步通信。source adapter 的说明要求“asynchronously communicating gradients as they are ready”，并且每个 parameter tensor 单独通信。学习阶段可以先把语义做对，再读 profile 看通信是否挡住了后面的 backward。

你现在应该能做什么：看到 DDP 参数不等价时，先检查三件事：初始化广播有没有发生，backward 后 step 前有没有等待所有梯度通信完成，梯度缩放是否和非并行完整 batch 的 loss 语义一致。

FSDP 分片改变了哪笔账

sharding

DDP 每个 rank 都存完整参数、完整梯度和完整 optimizer state。模型变大后，显存先成为瓶颈。Fully Sharded Data Parallel 的核心变化是：把参数、梯度和 optimizer state 的一部分分给不同 rank；计算某一层时再临时 all-gather 出完整参数，backward 后用 reduce-scatter 把梯度归还到各自 shard。它省显存，但增加通信和时序复杂度。

对象 DDP FSDP 排错问题
parameters 每个 rank 一整份 平时只保留 shard；计算时 all-gather full params 是否能按名字还原
gradients 每个 rank 同步后有完整梯度 reduce-scatter 后每个 rank 只保留本地 shard 梯度 grad shape 是否等于本地 param.data shape
optimizer state 每个 rank 一整份 跟参数 shard 对齐分片 分片更新后是否等价于非分片 optimizer
mixed precision 可选，主要影响算和带宽 通信和计算可用 compute_dtype，master weight 保持 fp32 梯度 dtype 是否回到 master weight dtype

Assignment 2 的 FSDP 测试很具体。get_fsdp(module, compute_dtype=None) 要返回包装器；fsdp_on_after_backward 在 backward 后、step 前处理同步；fsdp_gather_full_params 要把分片参数 all-gather 回完整 state dict。测试会比较 FSDP 模型和非并行 baseline 的参数，还会检查 compute_dtype=torch.float16 时，计算通信可以降精度，但 master weights 和 optimizer 更新仍按 fp32 语义走。

你现在应该能做什么：把 FSDP 的“省显存”理解成对象归属变化，重点看 full、shard、同步和 dtype。每次出错时，先问参数、梯度、optimizer state 现在应该是 full 还是 shard，dtype 应该是 compute dtype 还是 master dtype。

排错速查

debug

Systems 页最常见的误区，是把所有失败都理解成“数学公式错了”。有些失败确实是公式错，例如 attention 的 scale、mask 或 backward 推导；但 DDP/FSDP 的很多失败来自对象生命周期：参数什么时候 full，什么时候 shard，梯度什么时候同步，optimizer 读到的是本地状态还是全局等价状态。

现象 优先检查 不要先做什么
attention forward 通过，backward 不通过 LSE、保存张量、dQ/dK/dV 依赖 不要先改 Triton block size；PyTorch 语义还没闭合
