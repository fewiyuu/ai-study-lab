课程第 1 节

角色转变与环境设计

本页只讲一件事：当智能体已经能连续做任务时，人类工程师的重心要从“自己把代码写完”转到“把它工作的环境设计好”。这里先不追更大范围的仓库治理和吞吐管理，只先把角色转变、能力块、验收证据和反馈回路这条主线讲清。

课程地图原文下一节

读完本页后，先能做三件事

  • 把模糊需求改成可执行的能力块，写出输入、输出、边界和证据。
  • 判断一个任务卡住时，先分辨是目标不清、环境缺口，还是证据不够。
  • 为一个前端或后端改动补出可复核证据：测试、日志、截图、diff 或命令输出。

学习目标

start here

这节课讲的是：智能体已经能连续做任务以后，人类工程师不能只盯着单次输出，而要把任务边界、工作环境和验收证据一起摆好。真正影响结果的，不只是提示词长短，而是任务说明、目录入口、测试和反馈能不能接住它。

顺序要固定：先定边界，再补环境，最后用证据和反馈收口。顺序一乱，后面的步骤都会变成补洞。

这一节先看什么

  • 角色怎么变：从逐行实现转到设计工作台。
  • 环境怎么摆：文档、工具、测试、权限和反馈各放哪。
  • 证据怎么留：让“做完了”变成能复核的事实。

先抓主线

big picture

原文给出的工程判断很直接：工程师要为智能体搭工作台。这个工作台要让任务可读、动作可执行、结果可验证、错误可回收。

这门课先不讲抽象模型，而是把工程条件拆成可操作的东西：任务卡、目录说明、测试、日志、权限和规则。后面几节会继续把这些对象拆开。

先别追的东西

先不钻模型参数、推理细节、多智能体编排和大规模权限治理。那些主题会在后面的单元里回来；这一节先把一个任务写成能执行、能验收、能复盘的形式。

核心机制：能力块与反馈闭环

mechanism
目标层 先把“想做什么”变成可拆分的能力块。

读什么：需求、场景、约束。
写什么：输入、输出、不做事项、验收条件。

环境层 把能力块放到智能体能读懂的工作台里。

读什么：仓库说明、脚手架、已有测试、权限边界。
写什么：目录、任务模板、命令入口。

证据层 结果必须回到可复核的证据。

读什么：测试输出、日志、截图、diff。
写什么：通过/失败状态和原因。

反馈层 重复出现的问题要进入流程，不要只留在聊天里。

读什么：反复出错的模式。
写什么：文档、lint、模板、审查清单。

收敛层 这样下一次任务就更少返工。

读什么：历史失败。
写什么：更稳的 harness。

步骤 之前状态 之后状态 读什么 写什么 错了会怎样
需求卡 一句口语需求 能力块 需求、场景、约束 输入、输出、边界、证据 任务范围会越写越散
仓库入口 散落的说明和脚手架 可读工作台 README、目录、权限 入口、模板、命令 Agent 不知道从哪开始，也不知道哪里算完成
执行输出 只有完成感 可复核结果 测试、日志、截图、diff 通过/失败和原因 “做完了”只能停留在口头上
反馈规则 聊天里的重复提醒 流程里的规则 重复失败模式 文档、lint、清单 同类错误会不断重来

状态账本：任务从说明到证据怎么流

ledger

这类工作最容易漏掉的是状态。任务说明、仓库入口、执行输出和反馈规则分散在不同位置，谁读、谁写、谁会过期，要先分开看。把它们写成一张账本，才能知道问题卡在哪一层，而不是只盯着最后那份结果。

对象 当前状态 有效状态 读什么 写到哪里 过期的症状 下一步检查
任务说明 一句口语需求 能力块 场景、约束、输入 输出、边界、证据 任务会被理解成很多版本 先看有没有缺输入、输出或边界
仓库入口 散落的说明和脚手架 可读工作台 README、目录、权限 入口、模板、命令 Agent 不知道从哪开始 先看入口是否能定位任务
执行输出 只有完成感 可复核结果 测试、日志、截图、diff 通过/失败和原因 结果没法复核 先看证据是否能独立验证
反馈规则 聊天里的重复提醒 流程里的规则 重复失败模式 文档、lint、清单 同类错误反复出现 先看规则有没有落回仓库 
1. 先看任务卡有没有写清输入、输出和边界
2. 再看仓库里有没有可读入口和命令
3. 再看证据能不能复核
4. 最后看重复失败有没有写回规则

例子：把“做一个报表页”改写成可执行任务

example

原始说法很常见：做一个报表页。问题不在于它短，而在于它什么都没说清。智能体可以把它理解成很多不同的页面：有筛选器、有图表、有导出按钮，还是只有列表。只要边界不清，后面生成的结果就会看起来很努力，但很难验收。

把它拆成能力块之后，任务会变成：输入是筛选条件和表格数据，输出是一个可以查看、筛选、导出的页面；不做事项是暂时不碰权限体系和复杂图表；证据是截图、交互录屏、测试和 diff。

最小任务说明

输入：现有订单列表和一个日期筛选条件。

输出：一个能筛选、能分页、能导出的报表页。

边界：先不做权限分层和可视化图表。

证据：至少给出本地测试结果、页面截图和关键 diff。

版本 说了什么 缺什么 会怎样
原始说法 做一个报表页 输入、输出、边界、证据 每个人都能理解成不同东西
能力块版 输入是订单和日期筛选，输出是可筛选、可分页、可导出的页面 先不做权限和复杂图表 任务开始变得能执行
验收版 补上测试、截图和关键 diff 还要补本地检查入口 结果能复核，不只是看起来完成 
需求：做一个报表页。

常见误区

mistakes

这些误区的共同点，是把环境当成聊天记录的延长。只要任务说明、测试入口和反馈规则没落到仓库里，下一轮还会回到原点。

误区 真实问题 会造成什么 怎么改
只加长提示词 把问题当成“模型没听懂” 下次还是得重新解释 把重复要求写进文档、模板或测试
把大需求直接丢给 Agent 没有拆成能力块 范围扩张、审查变慢 先写输入、输出、边界和证据
只看最终结果 忽略过程证据 失败点不清楚，返工困难 要求测试、日志、截图或 diff
每次都在聊天里纠错 反馈没有进入流程 同一个坑反复出现 把第三次出现的问题写回模板、README 或 lint

这里先不追

boundary

这一节只解决角色转变和环境设计的起点。多智能体调度、权限分层、仓库级知识注入、评测体系和吞吐管理都属于后面的单元，不要现在就把它们混进来。先把“任务怎么切、环境怎么摆、证据怎么留”做稳，后面才接得住更复杂的治理。

下一步怎么接

下一节会把“环境”拆得更细：应用要怎样变得可读，哪些日志、截图和状态能进入反馈回路。你现在先带着这节的能力块模板去看下一页。

操作桥：把原文主线落到手边任务

bridge

如果你现在就要开始改一个真实任务，先从三步做起：把需求写成能力块，给每个能力块补一个证据要求，再把重复出现的错误写回仓库说明或模板。这个动作比“再加一点提示词”更能改变后面的执行质量。

操作顺序可以很短：先写任务说明，再跑一次最小实现，最后把失败模式改写成可检查的规则。这样下次就能靠环境本身挡住同类问题。

三步桥

2. 给能力块补上证据和边界。

3. 把重复错误写回模板、README 或 lint。

练习

diagnostic
练习目标

这组题要检查你能不能把一段模糊需求改写成能执行、能验收、能沉淀的任务。先判边界，再判证据，最后看反馈能不能写回流程。

检查重点

优先看你有没有把“做什么”说清，再看“怎么证明做完了”，最后看“重复错误该写到哪里”。开放题都带了参考答案要点，别只凭感觉作答。

一、概念辨认

先确认你没有把 harness、prompt 和能力块混在一起。

4. 为什么“做一个报表页”还不够？

参考答案 / 评分要点

要点至少包括：报表页的输入是什么、输出长什么样、是否需要筛选/分页/导出、先不做什么、怎么证明它真的可用。只说“需求太模糊”还不够，要指出它缺了哪些输入、输出、边界和验收信息。

参考答案 / 评分要点

应该选中“导出这个动作”作为已知信息；其余多数都还缺。更重要的是，题目没有说清导出格式、页面数据来源和如何证明做完。答题时别只停在“需求模糊”，要把缺口列出来。

参考答案 / 评分要点

好的答案会说明：智能体能持续执行后，人类的工作重点会转向任务设计和环境设计。要把任务拆成可执行的能力块，把环境做成可读、可测、可验证的工作台，再把失败模式写回流程。这样“做完”会落到证据上，而不是只停在感觉里。最后要点出：环境设计会决定智能体能走多远，重复问题能不能真正收住。
