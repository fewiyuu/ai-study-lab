Role Shift · Harness

从写代码转向设计环境

OpenAI 这篇文章讲的不是“怎样把提示词写得更长”。它讲的是一种角色变化：当 Codex 这样的智能体已经能连续写代码、跑测试、修复失败、开 PR，人类工程师的主要工作就从逐行编码，转向设计它工作的环境。

这个环境可以叫 harness。直觉上，它像一套可执行的工作台：仓库结构、文档、脚手架、测试、日志、验收标准、审查流程和反馈机制都在里面。模型越强，这套工作台越重要，因为强模型会放大环境里的好约束，也会放大环境里的含糊和漏洞。

目标→环境→能力块→Agent执行→证据→反馈→再沉淀

学完后要能做什么

  1. 解释为什么“让模型再努力一点”不是稳定的工程方法。
  2. 说清 harness 的组成：文档、工具、测试、反馈和权限。
  3. 把一个大目标拆成可执行、可验收、可复用的能力块。
  4. 诊断 Agent 卡住时，判断是模型问题、环境问题还是验收问题。

学完本块后，你应该能把一个含糊需求改写成 Agent 可以开始工作的工程任务。

概念骨架

MODEL

Harness 的直觉是“让 Agent 做对事的外部条件”。如果只有一句 prompt，模型需要自己猜项目结构、质量标准、测试方式、边界和失败处理。猜得准时看起来很神，猜错时很难复盘。Harness 把这些东西变成仓库里的显式对象。

正式一点说，harness 是围绕智能体执行过程建立的一组可读、可执行、可验证的约束和反馈回路。它不替代模型能力，但决定模型能力能不能稳定落到工程结果上。一个好 harness 会减少来回解释，让 Agent 看到目标、工具、证据和停止条件。

这里的角色变化不是“人类不重要了”。相反，人类负责的东西更靠近系统设计：定义什么算完成、哪些错误不能接受、哪些信息必须沉淀、哪些判断不能交给模型。你现在应该能判断：当 Agent 输出不稳时，先检查它的工作环境，而不是只责怪模型。

本块主线

  • 大目标不能直接丢给 Agent，先要拆成能力块。
  • 能力块需要明确输入、输出、验收证据和失败处理。
  • Agent 执行后，人类反馈应沉淀到文档、脚本或测试里。
  • 沉淀后的环境会降低下一次任务的解释成本。

讲义主线

LESSON

下面的小节按“问题、做法、判断”推进。每节最后都有一句检查点，用来确认你不是只记住了标题。

1. 变化不是少写几行代码，而是换了控制点

ROLE

传统开发里，人类把很多时间放在实现细节上：写函数、改类型、连接口、修测试。Agent-first 的开发方式不是把这些动作全部取消，而是让模型承担更多机械执行。人类的控制点移到更上游：任务怎么切、规则怎么写、证据怎么收、错误怎么复盘。

这件事最容易被误解成“以后只要下命令”。真正的问题是，如果环境没有结构，命令越大越危险。模型可以写出大量代码，但它不知道你的仓库边界、产品口味、风险偏好和历史决策。人类如果不把这些信息外化，Agent 只能靠概率猜。

所以角色变化的关键不是放手，而是把隐性判断变成显性系统。你不必手写每个组件，但要告诉系统什么叫好组件、什么依赖不能跨、什么测试必须跑、什么结果不能合并。

读完这一节，你应该能把“人类掌舵”解释成控制环境和验收，而不是只发一句需求。

2. Agent 卡住时，先问环境缺了什么

ENV

很多失败看起来像模型不够聪明：它改错文件、忘记跑测试、重复实现已有工具、给出无法验证的总结。往下看，常见原因是环境里没有告诉它该读哪里、该怎么运行、结果怎样算对、哪些工具优先用。

一个成熟的 harness 会把这些缺口提前补上。比如项目根目录有简短入口说明，任务目录有执行计划模板，测试脚本能一键运行，常见错误有修复说明，审查流程会要求 Agent 附上证据。模型不需要凭空记住这些，它可以在仓库里读到。

这也是为什么“再提示一次”不是稳定修法。聊天提醒只对当前会话有效；写进仓库的规则、脚本和测试才会影响下一次、下一个 Agent、下一条分支。

缺目标

Agent 不知道完成标准，就会倾向于产出看起来完整的东西。

缺工具

Agent 没法快速验证，只能依赖文字推断。

缺反馈

同类错误不会变成系统规则，下次还会出现。

读完这一节，你应该能把一次失败改写成“环境缺口清单”。

3. 把大目标拆成可验收的能力块

CAPABILITY

大目标通常太宽，例如“做一个后台系统”或“重构搜索体验”。Agent 可以开始写，但很难知道优先级。能力块更小：建立用户表 schema、生成路由骨架、写查询封装、补端到端测试、接入日志字段。每块结束时都能拿出证据。

能力块不是任务清单的换皮。它要能被复用。比如“生成带权限检查的 API 路由”如果沉淀成模板、测试和说明，后续所有路由都能用；如果只是让 Agent 写一次代码，它就只是一次性产物。

拆块时最有用的顺序是：先定义当前块的输入输出，再定义验收证据，最后定义不做什么。范围边界写清楚，Agent 才不会把局部任务扩大成全局重构。

读完这一节，你应该能把一个模糊产品目标拆成 3 到 5 个有验收证据的能力块。

4. 让 Agent 用证据说话

EVIDENCE

Agent 的自然语言总结很有用，但不能单独当作完成证明。它可能漏掉失败测试，也可能把没跑过的验证说成已经完成。Harness 要求 Agent 附上证据：执行了哪些命令，结果是什么，哪些截图或日志能证明问题被修复。

证据不是为了怀疑模型，而是为了让人类审查更快。高吞吐开发里，人类没有时间重跑每个细节。清晰证据可以让审查者直接看风险点：测试是否覆盖关键路径、日志是否显示正确状态、diff 是否只碰了相关文件。

好的证据也帮助 Agent 自我纠错。它看到失败输出，才能下一步修复；它看到覆盖率或性能指标，才能判断是否达标。没有证据，循环就变成“生成、相信、合并”。

读完这一节，你应该能为一个 Agent 任务写出至少三类验收证据。

5. 每次纠错都要问：能不能沉淀进系统

FEEDBACK

人类反馈如果只停留在聊天里，会不断重复。比如“不要在服务层直接访问 UI 状态”“日志要带 request_id”“这个目录不要放业务逻辑”。这些话说一遍可以，第三遍还在说，就说明 harness 没跟上。

沉淀方式不一定都是代码。简单规则可以放进 AGENTS.md 或目录说明；流程性要求可以变成任务模板；可机械判断的边界应该变成 lint、测试或脚本；需要审美判断的内容可以先写成示例，再逐步提炼。

反馈沉淀的目标是降低下一次解释成本。Agent 下次进入仓库时，不需要读懂你所有历史聊天，只要沿着仓库里的入口和工具走。

读完这一节，你应该能判断一条反馈应该进入文档、模板、测试还是 lint。

6. 完整链路：目标进入环境，环境约束执行

CHAIN

把这一块串起来：人类先把目标拆成能力块；每个能力块写清输入、输出、边界和证据；Agent 在已有仓库知识和工具里执行；执行结果通过测试、日志、截图或 PR diff 被检查；人类反馈再沉淀到文档或机制里。

这条链路的核心判断是：不要把模型当成唯一变量。模型只是系统的一部分。环境越清楚，模型越容易发挥；环境越含糊，模型越容易输出表面完整、内部不稳的结果。

以后看到 Agent 生成大量代码时，先不要急着兴奋或否定。问四个问题：目标是否被拆小，约束是否可读，证据是否可查，反馈是否能沉淀。能回答这四个问题，才算进入 harness engineering 的思路。

读完这一节，你应该能复述 harness 的完整闭环，并用它检查一个真实任务。

最小例子：把概念落到工程动作

EXAMPLES

例子 1：把一句大需求改成 Agent 可执行任务

原始需求“做一个客户后台”太大。下面的写法把它压成一个能力块，Agent 能开始，也能知道什么时候停。

正确示例：能力块任务说明Markdown
# 能力块：客户列表页最小可用版本

目标：实现客户列表页，支持分页、按名称搜索、查看客户状态。

输入：已有 customers 表、现有后台布局、权限中间件。
输出：路由、页面组件、查询封装、单元测试和一个端到端冒烟测试。

不做：不做客户编辑，不改数据库结构，不重写后台布局。

验收：
- pnpm test:unit customers 通过
- pnpm test:e2e admin-customers 通过
- 截图显示搜索为空、有结果、无权限三种状态

关键不是字多，而是范围、输入、输出和证据都可检查。Agent 看到这个任务，不需要猜“后台系统”到底包括什么。

例子 2：错误示例，只有愿望没有环境

下面这种提示很常见。它看起来给了目标，但没有告诉 Agent 从哪里开始，也没有验收标准。

错误示例：含糊需求Prompt
帮我把后台做好，代码写漂亮一点，能用就行。

错因是缺范围、缺证据、缺边界。“漂亮一点”没有工程含义，Agent 只能靠自己的默认风格发挥。

例子 3：把反馈沉淀成规则

如果你第三次提醒 Agent 不要跨层调用，就应该把规则写成仓库机制。

正确示例：反馈进入仓库YAML
rule: no_ui_import_in_service
applies_to: src/services/**/*.ts
forbidden_imports:
  - src/ui/**
message: 服务层不能导入 UI 模块。请把展示逻辑放到组件层，服务层只返回业务数据。

这条规则会影响后续所有分支。它比在聊天里反复说“注意分层”稳定得多。

易混概念对照

COMPARE
概念组容易混的地方区分方式工程判断
Prompt / Harness都能影响 Agent 行为。Prompt 是当前任务说明；Harness 是可复用的工作环境。重复出现的要求应进入 harness。
能力块 / 待办项都可以写在任务列表里。能力块有可复用产物和验收证据；待办项可能只是动作。能复用的能力优先沉淀。
人类审查 / 人类手写都需要工程判断。审查关注目标、证据和风险；手写关注实现细节。高吞吐时把人类注意力用在边界和例外上。
完成 / 可证明完成看起来都像任务结束。可证明完成需要测试、日志、截图等证据。没有证据的完成不适合合并。
