# 学习目标 {#goals}
<!-- tag: start here -->
<!-- layout: grid2 -->

## box
这节课讲的是：智能体已经能连续做任务以后，人类工程师不能只盯着单次输出，而要把任务边界、工作环境和验收证据一起摆好。真正影响结果的，不只是提示词长短，而是任务说明、目录入口、测试和反馈能不能接住它。
顺序要固定：先定边界，再补环境，最后用证据和反馈收尾。顺序一乱，后面的步骤都会变成补洞。

## box soft
### 这一节先看什么
- 角色怎么变：从逐行实现转到设计工作台。
- 环境怎么摆：文档、工具、测试、权限和反馈各放哪。
- 证据怎么留：让“做完了”变成能复核的事实。

# 先抓主线 {#overview}
<!-- tag: big picture -->
<!-- layout: grid2 -->

## box
原文给出的工程判断很直接：工程师要为智能体搭工作台。这个工作台要让任务可读、动作可执行、结果可验证、错误可回收。
这门课先不讲抽象模型，而是把工程条件拆成可操作的东西：任务卡、目录说明、测试、日志、权限和规则。后面几节会继续把这些对象拆开。

## box warn
### 先别追的东西
先不钻模型参数、推理细节、多智能体编排和大规模权限治理。那些主题会在后面的单元里回来；这一节先把一个任务写成能执行、能验收、能复盘的形式。

# 例子：把“做一个报表页”改写成可执行任务 {#example}
<!-- tag: example -->
<!-- layout: grid2 -->

## box
原始说法很常见：*做一个报表页*。问题不在于它短，而在于它什么都没说清。智能体可以把它理解成很多不同的页面：有筛选器、有图表、有导出按钮，还是只有列表。只要边界不清，后面生成的结果就会看起来很努力，但很难验收。
把它拆成能力块之后，任务会变成：输入是筛选条件和表格数据，输出是一个可以查看、筛选、导出的页面；不做事项是暂时不碰权限体系和复杂图表；证据是截图、交互录屏、测试和 diff。

## box soft
### 最小任务说明
输入：现有订单列表和一个日期筛选条件。
输出：一个能筛选、能分页、能导出的报表页。
边界：先不做权限分层和可视化图表。
证据：至少给出本地测试结果、页面截图和关键 diff。

版本
说了什么
缺什么
会怎样


原始说法
做一个报表页
输入、输出、边界、证据
每个人都能理解成不同东西


能力块版
输入是订单和日期筛选，输出是可筛选、可分页、可导出的页面
先不做权限和复杂图表
任务开始变得能执行


验收版
补上测试、截图和关键 diff
还要补本地检查入口
结果能复核，不只是看起来完成

# 常见误区 {#mistakes}
<!-- tag: mistakes -->

## box
这些误区的共同点，是把环境当成聊天记录的延长。只要任务说明、测试入口和反馈规则没落到仓库里，下一轮还会回到原点。

| 误区 | 真实问题 | 会造成什么 | 怎么改 |
| --- | --- | --- | --- |
| 只加长提示词 | 把问题当成“模型没听懂” | 下次还是得重新解释 | 把重复要求写进文档、模板或测试 |
| 把大需求直接丢给 Agent | 没有拆成能力块 | 范围扩张、审查变慢 | 先写输入、输出、边界和证据 |
| 只看最终结果 | 忽略过程证据 | 失败点不清楚，返工困难 | 要求测试、日志、截图或 diff |
| 每次都在聊天里纠错 | 反馈没有进入流程 | 同一个坑反复出现 | 把第三次出现的问题写回模板、README 或 lint |

# 这里先不追 {#boundary}
<!-- tag: boundary -->
<!-- layout: grid2 -->

## box
这一节只解决角色转变和环境设计的起点。多智能体调度、权限分层、仓库级知识注入、评测体系和吞吐管理都属于后面的单元，不要现在就把它们混进来。先把“任务怎么切、环境怎么摆、证据怎么留”做稳，后面才接得住更复杂的治理。

## box info
### 下一步怎么接
下一节会把“环境”拆得更细：应用要怎样变得可读，哪些日志、截图和状态能进入反馈回路。你现在先带着这节的能力块模板去看下一页。

# 操作桥：把原文主线落到手边任务 {#bridge}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box
如果你现在就要开始改一个真实任务，先从三步做起：把需求写成能力块，给每个能力块补一个证据要求，再把重复出现的错误写回仓库说明或模板。这个动作比“再加一点提示词”更能改变后面的执行质量。
操作顺序可以很短：先写任务说明，再跑一次最小实现，最后把失败模式改写成可检查的规则。这样下次就能靠环境本身挡住同类问题。

## box soft
### 三步桥
1. 先把需求拆成能力块。
2. 给能力块补上证据和边界。
3. 把重复错误写回模板、README 或 lint。

```
任务：
输入：
输出：
边界：
证据：
```

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box
练习目标
这组题要检查你能不能把一段模糊需求改写成能执行、能验收、能沉淀的任务。先判边界，再判证据，最后看反馈能不能写回流程。

## box soft
检查重点
优先看你有没有把“做什么”说清，再看“怎么证明做完了”，最后看“重复错误该写到哪里”。开放题都带了参考答案要点，别只凭感觉作答。

<!-- end-practice-intro -->

<!-- practice-group: 一、概念辨认 -->
<!-- quick-review: 复盘提示 先找对象，再找边界。 -->

### 1 choice

**Q:** 1. harness 最接近哪种说法？

- [ ] 让智能体工作的工程环境
- [ ] 一段更长的提示词
- [ ] 某个特定模型名称

Answer: 让智能体工作的工程环境
Explain: harness 让任务能执行、能验证、能沉淀的一组环境条件。

### 2 true/false

**Q:** 2. 判断：只要智能体写得足够多，人类就不需要再定义验收证据了。

- [ ] 对
- [ ] 错

Answer: 错
Explain: 写得多不等于可证明完成。证据是让结果能被复核，不是多写几句总结。

### 3 fill

**Q:** 3. 能力块至少要写清哪四项？

Answer: 输入;输出;边界;证据
Explain: 能力块的最小闭环是输入、输出、边界和证据。

### 4 synthesis

**Q:** 4. 为什么“做一个报表页”还不够？

Explain: 要点至少包括：报表页的输入是什么、输出长什么样、是否需要筛选/分页/导出、先不做什么、怎么证明它真的可用。只说“需求太模糊”还不够，要指出它缺了哪些输入、输出、边界和验收信息。

<!-- end-practice-group -->

<!-- practice-group: 二、拆写与比较 -->
<!-- quick-review: 复盘提示 先写输入输出，再补边界和证据。 -->

### 5 choice

**Q:** 5. 下面哪种首页任务写法最完整？

- [ ] 做一个首页，好看一点就行。
- [ ] 输入是品牌名和三条重点信息，输出是一个能展示主视觉、简介和入口的首页，先不做登录和后台管理，验收看截图和交互检查。
- [ ] 先把首页做出来，细节以后再补。

Answer: 输入是品牌名和三条重点信息，输出是一个能展示主视觉、简介和入口的首页，先不做登录和后台管理，验收看截图和交互检查。
Explain: 完整任务要同时写出输入、输出、边界和验收。

### 6 choice

**Q:** 6. 下列哪项更像 harness 的职责？

- [ ] 把重复要求写进模板和测试
- [ ] 每次都重新口头解释一遍
- [ ] 只在最后总结一句完成了

Answer: 把重复要求写进模板和测试
Explain: 重复出现的要求应该进入环境，而不是只留在对话里。

### 7 fill

**Q:** 7. 分别写出 prompt 和 harness 更像什么。

Answer: 一次性任务说明;可复用工程环境
Explain: Prompt 是一次任务说明，harness 是可复用的工程环境。

### 8 debug-checklist

**Q:** 8. 下面这段需求里，哪些信息已经够用，哪些还缺？

- [ ] 已经够用：页面主题
- [ ] 已经够用：导出这个动作
- [ ] 还缺：要导出什么格式
- [ ] 还缺：页面输入是什么
- [ ] 还缺：验收证据是什么

Explain: 应该选中“导出这个动作”作为已知信息；其余多数都还缺。更重要的是，题目没有说清导出格式、页面数据来源和如何证明做完。答题时别只停在“需求模糊”，要把缺口列出来。

<!-- end-practice-group -->

<!-- practice-group: 三、诊断与修正 -->
<!-- quick-review: 复盘提示 诊断题要写“原因 + 结果 + 修法”。 -->

### 9 fill

**Q:** 9. Agent 产出了很多文件，但没人知道怎么验证。至少补哪三项？

Answer: 测试命令;检查标准;截图或日志要求
Explain: 证据要能复核，所以至少要有命令、标准和日志或截图。

### 10 choice

**Q:** 10. 同一个错误第三次出现时，应该把它写到哪里？

- [ ] 写回模板、README 或 lint
- [ ] 只在聊天里再提醒一次
- [ ] 等下次再说

Answer: 写回模板、README 或 lint
Explain: 重复错误要进入流程，而不是只留在聊天记录里。

### 11 choice

**Q:** 11. 下面哪一版更像可执行的后台任务？

- [ ] 帮我把后台做好，先把常见问题都处理掉。
- [ ] 做一个管理后台，输入是用户和订单数据，输出是能查看和筛选数据的页面，先不做权限系统，验收看测试、截图和关键操作。
- [ ] 后台先能跑起来，其他以后再说。

Answer: 做一个管理后台，输入是用户和订单数据，输出是能查看和筛选数据的页面，先不做权限系统，验收看测试、截图和关键操作。
Explain: 可执行任务要写出输入、输出、边界和验收。

### 12 choice

**Q:** 12. 哪一句最接近这页对 harness engineering 的解释？

- [ ] 把任务写长一点，模型就会自己补齐剩下的。
- [ ] 把目标边界、工具入口、证据和反馈写进工作台，比单独调提示词更重要。
- [ ] 把所有事情都交给自动化，人工只看结果。

### 13 fill

**Q:** 13. 列出你会检查的 4 个 harness 要素。

Answer: 目标;环境;证据;反馈
Explain: 这一页的主线就是这四个要素。

### 14 synthesis

**Q:** 14. 讲一段 6-8 句话的话，说明为什么 Agent-first 会把人类的重心推向环境设计。

Explain: 好的答案会说明：智能体能持续执行后，人类的工作重点会转向任务设计和环境设计。要把任务拆成可执行的能力块，把环境做成可读、可测、可验证的工作台，再把失败模式写回流程。这样“做完”会落到证据上，而不是只停在感觉里。最后要点出：环境设计会决定智能体能走多远，重复问题能不能真正收住。
