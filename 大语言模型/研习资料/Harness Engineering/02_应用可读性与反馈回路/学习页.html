应用可读性与反馈回路

这一页只讲一件事：故障出现时，怎样把界面、日志、指标、trace 和工作树连成一条可回看的证据链。先不追 APM 平台、告警编排或浏览器自动化框架的细节，先练会判断该看哪一层、补哪一类证据。

学习目标

start

故障一出现，Agent 先要知道自己看到了什么，再知道下一步该读哪一层。应用可读性指的是界面、日志、指标和 trace 能不能把状态、失败和修复路径摆到台面上。

把这件事做对以后，修 UI 不会只剩一句“看起来好了”，修后端也不会只剩一句“测试过了”。你会开始要求证据，并知道证据该落在哪一层。

读完先能做的事

  • 从一条模糊报错里拆出入口、状态、证据和下一步检查点。
  • 判断一个问题该先看截图、DOM、日志、指标还是 trace。
  • 把“修好了”改成可复查的证据包，保留下次还看得懂的材料。

先抓主线

big

原文强调的是把工程环境搭成 Agent 能直接工作的台面：任务要读得懂，动作要做得出，结果要验得了，错了还能回收。

这门课的主线就是把工程条件设计成模型能用的工作台。以后你写任务说明、整理仓库说明、设测试门槛或定审查规则，都会回到这条线。

这页的边界

这一页先只讲可读性和反馈回路，不展开 APM 平台、告警编排、分布式采样、全链路压测或浏览器自动化框架。那些内容留到后面的系统课；这里先把“读什么、比什么、记录什么”说清。

一句工作定义

应用可读性 = 目标可见 + 状态可见 + 证据可回看 + 路径可复现。

这里的“目标”是任务要达成的状态，“状态”是当前发生了什么，“证据”是可以带回审查的材料，“路径”是下一次还能不能沿着同样的路得到同样结果。

表面回答什么问题最先看什么少了会怎样
截图界面是不是长对了视觉结果看不出布局、遮挡和状态
DOM / 页面状态按钮、开关、输入框的真实状态是什么当前结构和属性截图看起来对，实际状态还错
日志请求走到了哪一步request_id、route、error_code只能猜发生在哪个环节
指标问题是偶发还是持续退化p95、错误率、吞吐单次快照误导判断
trace跨调用的路径卡在哪span、下游、等待点跨服务问题看不见断点
工作树修复是否真的落地分支、diff、复跑结果口头说修了，实际上没跑通

证据表面怎么串起来

trace

同一个问题要在不同表面上对照。浏览器里看到的是结果，日志里看到的是事件，指标里看到的是窗口内的趋势，trace 里看到的是路径，工作树里看到的是修复是否真的落地。

故障走查：支付按钮点了没反应

case

把场景定得具体一点：用户在结算页点了提交，按钮立刻变灰，页面没有跳转，后台最后记下的是 PAYMENT_GATEWAY_TIMEOUT。这类问题最容易在两个地方看偏：只看截图，会把前端状态当成全部；只看日志，会把支付网关当成唯一嫌疑。更稳的做法是把同一请求在 UI、日志、指标和 trace 里对齐。

Agent 先看 DOM，确认按钮是否被禁用、有没有重复提交标记、有没有出现错误提示；再用 request_id 找到同一条日志，确认请求有没有到达网关；接着看指标窗口，判断这是短时间抖动还是持续升高；最后看 trace，确认等待点落在谁那里。每一步都在回答同一个问题，但每一步都把判断范围收得更小。

为什么这条顺序重要

如果先看指标，你只知道今天变慢了；如果先看日志，没有 request_id，就只能猜是哪一笔请求；如果先看截图，可能把“界面禁用”误当成“提交失败”。正确顺序的价值不在于更长，而在于少走回头路。

层看到什么能判断什么下一步错了会怎样
截图 / DOM按钮变灰，错误提示是否出现前端状态是否真的切换查交互事件和状态属性把视觉问题误判成后端问题
日志request_id、route、error_code请求有没有到达网关，失败类型是什么找同一条请求的下游日志没有字段就串不起来
指标p95、错误率、吞吐这是单次失败还是持续退化看同窗口前后的变化把一次抖动当成长期退化
tracespan、等待点、下游调用卡点在哪一跳看耗时最长的 span知道慢了，却不知道慢在谁那里
工作树diff、测试、复跑结果修复是否真的落地回到当前分支复跑口头说修了，实际没改到位
1. 先看浏览器截图和 DOM，确认前端状态。
2. 再找 request_id，对齐同一条日志。
3. 接着看指标窗口，分清单次故障和持续退化。
4. 如果问题跨服务，再看 trace。
5. 最后回到工作树和测试输出，确认修复落地。

证据包怎么写

bundle

证据包的目标很简单：让后来看的人不用重新问一遍“入口在哪、复现什么、哪里能看到失败、修复有没有真正跑过”。如果一个问题已经进到 PR 或 issue 讨论里，证据包最好固定成同一套字段。这样做的作用很直接：减少反复追问，也减少重复复现。

一个可复用的证据包至少应当同时保留界面、日志、指标和路径信息。界面告诉你现象，日志告诉你事件，指标告诉你趋势，路径告诉你卡点。少掉任何一层，审查者都得自己补一段推理。

证据包的最小字段

  • 入口：在哪个页面、哪个按钮、哪条接口上复现。
  • 现象：截图或 DOM 里能直接看到什么。
  • 事件：对应的 request_id、route 和 error_code。
  • 趋势：相关指标是单次抖动还是持续变化。
  • 路径：trace 里卡在哪一跳，工作树里改了什么。

写证据包时，顺序比措辞更重要。先写入口和现象，确保复现点一眼能找得到；再写 request_id、route、error_code，把同一条请求串起来；最后写指标窗口、trace 和工作树，说明这次是局部抖动、持续退化，还是修复没落地。少掉任意一层，后面的人都要先替你补一段推理。

如果你只写“问题已修复”，信息其实还没闭合。审查者需要知道原来是哪一步坏了、修复落在哪一层、复跑有没有重新经过同一路径。证据包的作用就在这里：它把问题说得更容易被重走一遍。

故障链路：从样本到证据包

sequence

把故障样本写成一条链，Agent 就不会每次都从零开始猜。最小样本最好同时包含入口、现象、请求、趋势和修复结果。这样做的好处很简单：每一步都能接到下一步，后面的人不用重新拼时间线。

下面这条链路把一个“提交失败”的问题拆成五个时间点。你可以直接照着改成你自己的样本：先写用户在什么页面点了什么，再写页面发生了什么，再写日志和指标里出现了什么，最后写工作树里改了什么。样本越短越好，但每个时间点都不能空。

怎么用这条链

  • 先把问题缩成一个按钮、一个接口或一个页面。
  • 再把界面、日志、指标和 trace 按时间顺序摆出来。
  • 最后写清楚修复后哪一步应该变成什么样。

提交失败时间线复制

t0 用户进入 /checkout，按钮可点击。
t1 用户点击提交，按钮变灰，页面没有跳转。
t2 日志出现 request_id=r-2481，route=/checkout，error_code=PAYMENT_GATEWAY_TIMEOUT。
t3 指标窗口里 error_rate 上升，p95 也从 210ms 升到 842ms。
t4 trace 显示最长 span 卡在 payment gateway，下游没有返回。
t5 工作树补了重试和超时回退，复跑后按钮恢复可用。
时间点看到什么说明了什么如果少看这一层会怎样
t0按钮可点击入口和初始状态正常会把“按钮坏了”误当成起点问题
t1按钮变灰，页面未跳转交互状态已经发生变化只看文字说明，忽略真实 UI 状态
t2request_id 和错误码请求到了哪一步、失败类型是什么找不到同一请求，日志无法聚合
t3error_rate 和 p95 上升这是趋势回退，要看窗口变化把持续问题当成单次抖动
t4trace 卡在下游支付网关真正断点在跨服务路径只盯页面，不知道卡在哪一跳
t5工作树复跑通过修复已写进当前分支只听口头确认，没法复查

这条链路给出的判断很明确：如果只看 t1，你会把问题当成交互异常；如果只看 t2，你会以为是网关故障；如果只看 t3，你会以为是整体变慢；如果只看 t4，你会以为下游服务一定是根因。真正有用的做法，是把这些层放在同一条时间线上，让每一步都把下一步的范围收窄。

把链路写回 issue 的最小句式

入口是哪个页面，现象是什么，request_id 是多少，指标窗口怎样变化，trace 卡在什么 span，工作树里改了什么。这样写出来的记录既能给人看，也能让 Agent 直接接着查。

如果这几项没有写全，后面的人就只能补问入口、补问状态、补问路径。一次问题的修复成本，很多时候不在代码本身，而在补这几层证据。

检查项合格长什么样不合格长什么样后果
入口具体页面、按钮、接口只写“有问题”无法复现
现象截图、DOM、状态变化只有口头描述看不见真实状态
事件request_id、route、error_code没有字段，只有长段文本日志无法聚合
趋势窗口内 p95、错误率、吞吐只给一个瞬间值误判单次抖动
路径trace 和工作树 diff 都能对上只说“应该修好了”没有落地证据

写回流程

feedback

反馈回路要把第三次出现的同类问题直接落到 issue 模板、PR 模板、日志规范或脚手架里。按钮状态、日志字段、trace 缺口、工作树污染，只要重复出现，就该被写成规则。这样下一次碰到同类故障，Agent 读到的就是可重复的工作方式，而不是临时说明。

如果你已经有一条错误样本，先别急着写长分析，先判断它应该被写进哪里：字段名缺失写进日志规范，界面状态缺失写进 UI 检查项，跨服务断点写进 trace 采集约定，工作树混乱写进并行规则。把错因放回流程，反馈才会真的留下来。

写回的位置

常见位置只有四类：issue 模板、PR 模板、日志规范、脚手架或测试入口。问题不一定要进同一个地方，但每类问题都要有固定入口，让下次的人不用重新摸索。

重复症状现在的临时做法应该写回哪里下一次会少什么
按钮状态经常不清楚人眼口头确认UI 检查项 / DOM 断言反复猜状态
日志无法聚合请求手工翻日志日志规范 / request_id 约定重新拼请求
性能回退总是后知后觉等人反馈“变慢了”指标窗口 / 告警规则凭感觉判断趋势
跨服务卡点总找不到只看接口返回值trace 采集约定重新猜下游
修复结果总回退只看聊天里的确认PR 模板 / 复跑步骤重复确认是否落地

状态台账

ledger

台账的作用是把“看起来像好了”拆成可追踪的资源变化。只要状态会变、会失效、会被别的 Agent 或别的请求覆盖，就应该把读写关系写出来。

证据怎么选

choose

证据要和问题类型对上。视觉类问题先看截图和 DOM，交互类问题先看状态和日志，性能类问题先看指标和 trace，修复落地类问题先看工作树和测试输出。顺序错了，审查者只会在错误的层上来回绕。

先别混

不要把“页面看起来正常”当成“流程一定正常”，也不要把“测试通过”当成“交互就一定可用”。同一问题常常需要两层证据：一层证明逻辑，一层证明运行时状态。

问题类型第一证据第二证据别先拿来替代的东西
按钮点了没反应DOM + 截图日志单元测试结果
接口偶发 500日志 + request_idtrace最终页面截图
页面越来越慢指标窗口trace单次响应时间
修完又回退工作树 diff测试输出一句“已修复”
跨服务卡住trace下游日志单个接口返回码

证据包演示

lab

点击不同证据包，观察同一个问题在审查成本上的变化。这个演示关注的是哪个包能让下一步检查少猜一点，哪个包能把后续动作接上。

对照三个包时，不要只看信息量，要看下一步能不能少猜。文字总结适合汇报，测试输出适合确认逻辑，测试加截图加日志才适合让审查者直接复核运行时状态。遇到跨服务问题，再把 trace 叠进去，才能把卡点和下一步动作连起来。

例子：一条请求日志该长什么样

example

如果日志里没有结构化字段，Agent 只能读到一堆句子，没法把一条请求串到另一条请求上。一个最小可用的日志至少要能回答三件事：这是谁的请求、走到哪一步了、最后为什么失败或变慢。

下面这条样例里，request_id 把同一次请求串起来，route 告诉你走到哪条路径，error_code 和 latency_ms 让你知道是失败还是变慢，ui_state 说明前端状态和后端事件有没有对上。

读这条样例时先看什么

  • 先找 request_id，确认你看的是同一条请求。
  • 再看 route 和 error_code，确认问题落在哪个环节。
  • 最后看 ui_state 和 latency_ms，确认界面状态与时延是否对上。

结构化日志示例复制

{
 "request_id": "r-2481",
 "route": "/checkout",
 "user_id": "u-17",
 "status": "error",
 "error_code": "PAYMENT_GATEWAY_TIMEOUT",
 "latency_ms": 842,
 "ui_state": "submit_disabled"
}
字段回答什么少了会怎样
request_id同一条请求怎么串起来日志散成一片，没法对齐
route请求走到了哪条路径不知道是哪个页面或接口
error_code失败类型是什么只能看到“失败了”，看不到失败类别
latency_ms是否慢到异常很难判断是失败还是性能回退
ui_state前端真实状态是什么看不出界面和后端是否对上

这里先不追

boundary

这一节只处理“能看见、能对照、能复查”。OpenTelemetry 的完整配置、APM 平台选型、采样率设计、告警编排和浏览器自动化框架都不在这里展开。先把证据面接起来，再往更复杂的系统里走。

后面会怎么回来

第 03 节会把这种读法搬到仓库知识、目录、README 和模板上。到了更后面的单元，你会再次看到 trace、采样和自动化，不过那时它们已经是在更大的系统里做分工了。

下一步怎么做

bridge

如果你现在就在一个真实项目里，先拿一个页面或一个请求做样本。先看截图和 DOM，确认问题是否真的出在视觉或状态；再找同一条 request_id 的日志，把事件串起来；然后看指标窗口和 trace，确认这次是偶发、持续还是跨服务卡住。最后回到工作树，确认修复和测试都真的落地。

这条顺序的作用是把“猜”换成“查”。只要你能把同一问题在不同表面上对齐，Agent 的修复就会少很多反复解释。

症状先看哪层

  • 页面样式乱了，先看截图和 DOM。
  • 接口偶发失败，先看日志和 request_id。
  • 页面越来越慢，先看指标窗口，再看 trace。
  • 修完又回退，先看工作树 diff 和测试输出。
症状先看哪层补什么常见误判
页面样式乱了截图 + DOM布局、状态、遮挡只看文字描述
接口偶发 500日志 + request_id失败类型、路径、上下文只看最终页面
页面越来越慢指标窗口 + trace趋势、断点、等待点只看一次响应时间
修完又回退工作树 diff + 测试输出复跑证据、提交状态只听口头确认
1. 先看浏览器截图和 DOM，确认问题是否出在界面或状态。
2. 再用 request_id 找到同一条日志，确认失败发生在哪一步。
3. 然后看指标窗口，判断这是单次抖动还是持续退化。
4. 如果问题跨服务，再看 trace，找到卡住的 span。
5. 最后回到工作树和测试输出，确认修复真的落地。

证据联读样本

遇到一个灰掉的按钮时，先别急着下结论。截图告诉你结果长什么样，DOM 告诉你这个结果是被哪一层状态写出来的，日志告诉你这条请求有没有走到对应的接口，指标窗口告诉你错误是不是集中爆发，trace 告诉你具体卡在哪个服务或哪段等待上。

这五层放在一起看，修复目标就会从“页面看起来不对”变成“禁用状态来自前端还是后端、超时是局部还是全局、回写要落在哪一层”。判断一旦清楚，Agent 的回写就会更稳，审查者也不用重新拼故事。

一条样本怎么读

  • 先确认截图里的按钮状态，看看是样式问题还是业务锁定。
  • 再看 DOM 里的 disabled、aria-busy、data-state，确认状态来源。
  • 再看同一条 request_id 的日志，确认请求有没有走到支付或提交接口。
  • 最后看最近一段时间的指标和 trace，确认是局部抖动还是链路阻塞。
信号你在找什么读到什么程度就够下一步
截图按钮、文案、布局、遮挡知道界面有没有直接暴露异常转去 DOM 看状态从哪来
DOMdisabled、hidden、aria-、data-知道前端状态是怎么写出来的转去日志看请求是否到位
日志request_id、route、error_code知道同一条请求走到了哪一步转去指标看是不是持续退化
指标窗口错误率、p95、吞吐、重试知道问题是单点还是成片出现转去 trace 看卡点位置
Trace最慢的 span、等待段、重试段知道是哪一跳拖住了整条链路回到修复或回写动作

证据联读速查

截图：按钮灰掉，旁边没有错误提示。
DOM：<button disabled data-state="pending">Submit</button>
日志：request_id = r-2481，route = /checkout，error_code = PAYMENT_GATEWAY_TIMEOUT
指标：过去 15 分钟 error_rate 上升，p95 变长
Trace：checkout span 正常进入，但 payment-gateway 等待最久
结论：先修下游超时，再回看按钮是否应该继续锁定

把信号串成断点

signals

真实项目里，最容易误判的地方在于，报错之前哪一步已经变形。一个按钮卡住的例子就很典型：前端先把按钮固定住，随后接口请求超时，最后 trace 显示支付服务等待最久。你如果只盯着最终页面，看到的是灰色按钮；你如果只盯着日志，看到的是超时；你如果只盯着 trace，看到的是下游等待。单独看每一层都像答案，合起来才知道该先修哪一跳。

这个断点链的作用，是把“修页面”拆成“确认状态来源、确认请求去向、确认慢点位置、确认回写位置”四个动作。修复回到这个顺序以后，后续复发也更容易被同一套证据抓住。

四个断点

  • 状态断点：按钮为什么被固定住，锁定标记来自哪一层。
  • 请求断点：这次请求有没有到达目标接口，是否卡在网关前后。
  • 趋势断点：错误率和时延是在一条线上抖动，还是整段窗口都在变差。
  • 回写断点：修复应该落在前端状态、后端接口，还是两边都要改。
断点先读什么会写什么断错了的后果
状态断点DOM、组件状态、按钮属性锁定或解锁的可见状态只修样式，业务状态还在卡住
请求断点request_id、route、error_code请求是否成功到达目标接口只修前端，后端超时仍然继续
趋势断点错误率、p95、重试次数问题是偶发还是持续把系统性退化误判成单次噪声
回写断点修复说明、测试输出、工作树谁要改、改完怎么证明口头说修好了，实际没有复现闭环

最小断点模板

1. 截图：按钮状态、提示文案、遮挡情况。
2. DOM：按钮属性、状态类名、关键 data-*。
3. 日志：request_id、route、error_code、latency_ms。
4. 指标：近 15 分钟错误率和 p95 的变化。
5. Trace：最长 span 和等待点。
6. 结论：先修哪一层，修后怎么复查。

练习时不要只会说“我看完了”。你要能说出哪一层是起点、哪一层是证据、哪一层是回写入口。只要这三件事说清楚，下一次碰到同类问题，你就能先收集证据再动手。
