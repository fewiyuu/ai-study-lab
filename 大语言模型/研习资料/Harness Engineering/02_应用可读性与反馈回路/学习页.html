应用可读性不是让界面更漂亮，而是让 Agent 能观察状态、复现问题、判断修复是否真的发生。
Application Legibility

让应用状态变成 Agent 能读的证据

当 Agent 能大量写代码后，瓶颈会从“谁来实现”转向“谁来验证”。如果一个应用只能靠人眼点击、靠口头描述问题，Agent 就只能猜。应用可读性要把界面、日志、指标、追踪和本地环境做成 Agent 可以读取的证据。

这一页关注 OpenAI 原文里很实用的一点：为了让 Codex 在多个工作树里修复和验证问题，团队让应用环境可启动、可观察、可查询。Agent 不只看代码，还能看浏览器、看日志、看性能数据，然后把这些证据带回 PR。

独立环境→UI状态→日志→指标→追踪→验证→PR证据

学完后要能做什么

  1. 解释应用可读性和普通可观测性的区别。
  2. 设计一个让 Agent 复现 UI 问题的最小反馈回路。
  3. 把日志、指标和 trace 转成可用于验收的证据。
  4. 诊断为什么“人眼看一下”会成为 Agent 高吞吐的瓶颈。

学完本块后，你应该能为一个前端或全栈任务设计可观察、可复现、可验证的运行环境。

概念骨架

MODEL

应用可读性的直觉是：系统运行时发生了什么，Agent 能不能自己看见。如果只能由人类描述“按钮点了没反应”，Agent 得先猜复现步骤，再猜原因。可读性高的应用会给出 DOM、截图、控制台错误、接口日志、请求耗时和业务状态。

正式说法上，它是面向智能体的可观测性。普通可观测性面向人类运维和开发者；Agent 可读性还要求信息能被工具抓取、能被提示压缩、能和代码改动关联。日志字段、命令输出、截图文件、trace 链接都应该能进入任务上下文。

你现在应该抓住一个判断：如果修复过程必须依赖人类手动点击和口头描述，Agent 的执行能力再强也会被验证环节卡住。

本块主线

  • 每个任务最好在隔离工作树里运行，避免状态互相污染。
  • UI 状态要能被 Agent 获取：截图、DOM、路由、控制台错误。
  • 后端状态要能被 Agent 查询：日志、指标、trace、错误码。
  • 验收不是“看起来能用”，而是证据能证明关键路径通过。

讲义主线

LESSON

下面的小节按“问题、做法、判断”推进。每节最后都有一句检查点，用来确认你不是只记住了标题。

1. 代码生成变快后，验证会先卡住

QA

当 Agent 能快速实现功能，人工验证就会暴露为瓶颈。一个人同时看十几个 PR，不可能每个都手动启动、点击、读日志。如果应用没有把状态变成证据，审查者只能在“相信 Agent”和“全部重跑”之间摇摆。

验证瓶颈不是小问题。它会影响任务拆分方式、PR 大小和合并速度。Agent 写得越快，越需要把验证前置到环境里：启动脚本要稳定，测试要能指向关键路径，日志要能按请求查，截图要能说明 UI 状态。

所以应用可读性首先服务验证。它让 Agent 在提交结果前先做一轮自查，也让人类审查时能看证据，而不是从零复现。

读完这一节，你应该能解释为什么高吞吐 Agent 团队会更需要自动化证据。

2. UI 不是只能给人看，也要给 Agent 看

UI

前端问题经常卡在“状态看不到”。按钮是不是 disabled，弹窗有没有打开，表单错误显示在哪里，路由是否跳转成功。这些信息如果只能靠人眼，Agent 很难自主验证。通过浏览器工具、DOM 快照、截图和控制台日志，Agent 才能把界面状态带回代码循环。

这不意味着所有 UI 都要写成机器专用。更实际的做法是保留稳定的测试标识、清晰的错误文案、可查询的路由状态和截图输出。Agent 不需要美学判断时，也能知道当前页面处于什么状态。

UI 可读性的判断句是：如果一个人能指出问题，但 Agent 无法从任何工具读到同样事实，环境还不够可读。

读完这一节，你应该能为一个页面列出 Agent 可观察的 4 类 UI 信号。

3. 日志、指标和 trace 要能回答具体问题

OBS

日志不是越多越好。Agent 需要的是可定位的问题：这个请求是谁发的，经过哪些步骤，失败在哪里，耗时多少，关联的资源 ID 是什么。没有结构化字段，日志会变成一堆文本，模型只能模糊总结。

指标和 trace 也一样。比如目标是“页面首屏接口 800ms 内返回”，验收证据就应该能查到接口耗时分布，而不是写一句“感觉快了”。如果目标是修复支付失败，就要能看到错误率、错误码和相关调用链。

运行时证据要和任务目标绑定。每个功能任务都可以问：它的成功状态在日志里是什么、在指标里是什么、在 UI 里是什么。

日志

定位发生了什么，最好有 request_id、user_id、resource_id。

指标

判断数量和趋势，例如耗时、错误率、吞吐。

Trace

追踪一次流程经过哪些服务和步骤。

读完这一节，你应该能把一个模糊性能目标改成可查询的指标验收。

4. 隔离环境让多个 Agent 可以并行验证

WORKTREE

如果所有 Agent 共用同一个本地服务，一个任务改了数据库，另一个任务的测试可能突然失败。隔离工作树和独立运行实例能减少这种干扰。每个任务有自己的端口、环境变量、日志目录和测试数据，证据才对应当前 diff。

隔离环境也让失败更容易复现。审查者看到 PR 时，能根据任务里的命令重新启动同样环境。Agent 自己也可以在失败后清理、重启、再验证，而不是依赖人类猜当前机器状态。

这件事的成本不低，但它会换来并行能力。高吞吐开发不是让所有 Agent 挤在同一个环境里抢状态，而是给每条任务一块可控的工作台。

读完这一节，你应该能说明为什么独立工作树是 Agent 并行开发的基础设施。

5. 反馈闭环要短，但不能省掉证据

LOOP

短反馈闭环的意思是：Agent 改完后马上运行相关验证，看到失败就继续修。它不应该等人类晚上统一检查。单元测试、端到端测试、浏览器截图、日志查询都可以成为闭环的一部分。

但闭环短不等于随便。Agent 如果只跑了一个宽泛命令，却没有覆盖目标路径，就会产生虚假的安全感。更好的任务说明会写明“运行哪条命令，观察哪个状态，保存哪类截图或日志”。

人类审查的重点也会变化：不只看代码，还看闭环是否成立。一个 PR 的证据如果无法证明目标路径通过，就应该要求补验证，而不是争论代码看起来是否合理。

读完这一节，你应该能为一个修复任务设计执行、观察、再执行的闭环。

6. 完整链路：从复现到证明修复

CHAIN

把这一页串起来：任务进入独立工作树，Agent 启动应用；它通过浏览器和日志工具复现问题；根据 DOM、截图、错误日志和 trace 找原因；修改代码后跑相关测试；最后把截图、命令输出和查询结果写进 PR 证据。

这条链路的价值是让验证从“人类手感”变成“环境事实”。它不会取消人工审查，但会让人工审查更集中。人类看证据是否覆盖关键路径，看异常情况是否被处理，看 Agent 有没有绕过失败。

以后遇到“Agent 修不好 UI bug”时，先问它能看到什么。看不到状态，就先补状态；看不到日志，就先补日志；无法独立运行，就先补运行环境。

读完这一节，你应该能用 UI、日志、指标和 trace 设计一个可复盘的修复流程。

EXAMPLES

例子 1：UI 问题的复现说明

把“页面坏了”改成 Agent 能复现的说明。

正确示例：复现与证据Markdown
# 任务：修复客户搜索为空状态

复现：
1. 启动当前工作树：pnpm dev --port 4312
2. 打开 /admin/customers
3. 搜索 zzz-no-result

预期：显示空状态文案和新建客户入口。
证据：保存搜索前、搜索后两张截图；记录控制台错误；运行 e2e:customers-search。

这段说明给了路径、状态、预期和证据。Agent 不需要猜 bug 在哪里。

例子 2：性能目标要落到指标

“快一点”没有验收价值。下面把它改成可查询目标。

正确示例：指标验收Text
目标：/api/customers/search 在本地测试数据下 p95 < 800ms。

验证：
- 运行 pnpm bench customers-search
- 查询 trace 中 searchCustomers span 的 p95
- PR 中贴出优化前后 p50/p95 对比

指标不需要复杂，但要能回答是否达标。

常见坑：错因、后果和修法

PITFALLS
误区错因后果修法
只靠人眼验证应用状态没有工具化。Agent 无法自主复现和证明修复。补浏览器、截图、DOM、日志和测试入口。
日志没有结构化字段把日志当散文写。无法按请求或资源定位问题。增加 request_id、user_id、route、error_code 等字段。
所有任务共用环境没有隔离工作树和端口。状态互相污染，证据不可靠。每个任务独立启动服务和测试数据。
性能目标含糊只说“快一点”。修复后无法判断是否达标。写 p50/p95、错误率、吞吐等指标目标。
证据和目标不匹配修 UI 只贴单元测试，修逻辑只贴截图。审查者仍要补验证。按问题类型选择截图、日志、指标或测试。

易混概念对照

COMPARE
概念组容易混的地方区分方式工程判断
可观测性 / Agent 可读性都看系统状态。可观测性面向人和运维；Agent 可读性要求状态能进工具和上下文。Agent 任务要能自动收集关键证据。
截图 / DOM都描述界面。截图看视觉结果，DOM 看结构和状态。视觉问题看截图，交互状态常看 DOM。
日志 / Trace都用于排查运行时。日志记录事件，trace 记录流程路径和耗时。跨服务问题优先看 trace。
共享环境 / 独立工作树都能运行应用。共享环境快但容易污染；独立环境更适合并行 Agent。高并行任务优先隔离。
