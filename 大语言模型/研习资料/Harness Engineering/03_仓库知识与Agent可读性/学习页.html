仓库知识与 Agent 可读性

这一页讲的是仓库知识怎么变成 Agent 真能读、真能用的入口。重点在于把 README、AGENTS、目录边界、任务模板、脚手架和验收命令放到正确位置，让 Agent 进仓库后先知道“从哪读、能改哪、怎么收尾”。

你可以把这页当成一张仓库地图的制作说明：入口要短，边界要硬，模板要可复制，验证要能复跑。只要其中一项空着，Agent 就会自己补解释，最后把时间花在猜测上。

读完本页应该会做什么

  • 判断一条仓库规则应该放在 AGENTS、README 还是目录 README。
  • 把口号式说明改成带入口、边界和验证方式的可执行说明。
  • 把一次常见失误写回模板、脚手架或目录说明里。

主线：把仓库知识做成可执行入口

map

仓库知识是一组分层入口。全局约定写在 AGENTS，项目介绍和常用启动写在 README，目录职责和例外写在目录 README，重复结构交给模板和脚手架，验收方法写成命令和证据。

这样做的好处很直接：Agent 进仓库后，先读到能改变它动作的材料，再读到能证明完成的材料。它不用把整个仓库当成搜索题来做。

这一页的工作定义

仓库可读 = 入口短、边界清、模板稳、脚手架固定、验证命令可复跑。

仓库不可读 = 信息分散、边界含混、重复靠手写、验证只能口头描述。

对象 该写什么 谁会读 缺了会怎样
AGENTS.md 仓库级约定、目录边界、验证要求 Agent 和维护者 每次都要重新解释基础规则
README.md 项目做什么、怎么启动、常见入口 人和 Agent 从搜索开始，入口不清
目录 README 该目录的职责、禁区、例外和命令 当前目录的执行者 改错目录或把任务扩大
任务模板 目标、输入、边界、验收、证据 发任务的人和接任务的 Agent 交接后仍然缺字段
脚手架 固定文件、固定字段、生成方式 需要重复创建的人 重复结构靠复制，越来越乱
测试命令 怎么复跑、怎么看失败、怎么看证据 审查者和 Agent 做完了也说不清怎么证明

入口地图：从猜测走向可执行

trace
无入口 Agent 只能靠文件名和搜索猜结构，常见症状是来回翻目录、改错层级、重复问起点。

看到目录很多，不等于看到了入口。

单一入口 一个清楚的 README 告诉 Agent 先读什么、项目做什么、常用命令在哪里。

入口短，Agent 才会先进入正确的范围。

目录入口 每个重要子目录都有自己的 README 或 AGENTS 片段，职责和禁区落在最近的位置。

边界越近，改错目录的概率越低。

可执行入口 模板、脚手架、测试命令和证据要求都能被复跑，Agent 不用猜验收方式。

入口不只告诉它怎么读，还告诉它怎么结束。

成熟度 读到什么 还会猜什么 典型症状
低 只有目录名和零散代码 入口、边界、验证 反复搜索、反复提问
中 有 README，但边界不够近 哪些目录能改、哪些不能碰 常把相关目录一起动掉
高 入口、边界、模板和命令都在 只剩具体实现细节 审查更快，复跑更稳

目录边界：把职责放到最近的位置

boundary

目录边界的判断标准很简单：这条信息会不会改变当前目录里的正确动作。会，就放在这里；不会，就不要把它塞进来占注意力。

对子目录来说，目录 README 比顶层 README 更重要，因为它更接近真实动作。对仓库来说，AGENTS 比临时聊天记录更重要，因为它能稳定地被下一次读到。

三个检查问题

  • 没有这份说明，Agent 会不会改错目录？
  • 这份说明会不会把本次任务扩成别的范围？
  • 这份说明能不能直接告诉它怎么验证？
位置 应该写什么 不建议写什么 缺失时的信号
AGENTS.md 仓库级规则、公共边界、验证要求 单个目录的局部细节 Agent 先问最基础的问题
README.md 项目介绍、启动路径、常用入口 每个子目录的例外 入口分散，读者要自己拼
目录 README 这个目录的职责、禁区、验证命令 整个仓库的通用原则 改动越界，检查成本高
issue / PR 模板 目标、复现、证据、写回 长篇背景说明 交接后仍缺字段
测试与脚手架 怎么生成、怎么复跑、怎么验收 业务政策和架构宣言 重复结构只能手写

任务模板：把口头要求改成可执行字段

template

任务模板把“帮我处理一下”变成一组能执行、能验收、能写回的字段。它让 Agent 不用猜目标、边界和证据。

如果同一类任务总是缺同一个字段，就把这个字段固化到模板里，后面直接复用。

模板先补什么

  • 目标：用户或系统最终看到什么状态。
  • 范围：允许改动什么、禁止碰什么。
  • 验证：用什么命令和证据证明完成。

脚手架演示：同一类说明，三种写法

lab

脚手架和模板的差别在于：模板告诉你字段长什么样，脚手架直接把结构生成出来。这里看三种仓库入口写法，判断 Agent 会不会少猜。

欢迎来到本项目，请认真工作。
版本 能确定什么 还要猜什么 适合交给谁
口号版 只有态度，没有动作 入口、边界、验证 谁都不适合
半成品 知道目录，但边界不全 能改哪里、怎么验收 会继续补问的人
可执行版 目标、范围、验证、证据都清楚 只剩具体实现 可以直接交给 Agent

具体场景：给 checkout 目录补入口

case

假设你接手一个 `src/checkout/` 目录。里面已经有页面、测试和提交逻辑，但没有目录级 README，也没有说明这个目录能碰什么、不能碰什么、怎么验证。新来的 Agent 第一次进来，通常会从最显眼的文件名开始翻，接着把支付协议、错误提示和页面状态一起当成一个问题处理。

这时候补入口的目标很明确：不是写一篇更长的介绍，而是把最常用的动作放在最近的位置。先让它知道这里负责什么，再让它知道哪里是边界，最后让它知道怎么证明自己做完了。只要这三件事稳定，后面的改动就会收敛很多。

目录说明越近，越像一张现场图。它不是给人看的背景材料，而是告诉执行者“现在往哪走、哪条线不能碰、改完要拿什么回来”。如果这三件事写在不同位置，Agent 就得先拼图，再开始工作，成本会从阅读阶段就抬高。

这个场景里，Agent 最常问的三句

  • 这个目录到底负责页面、状态还是接口。
  • 能不能碰支付协议和账户体系。
  • 改完以后要跑哪一个命令、贴什么证据。
阅读顺序 先看到什么 脑中先回答什么 下一步动作
1 目录名和 README 标题 这是不是当前任务的最近位置 先确认职责，不急着翻实现
2 目录边界和禁区 哪些东西不能顺手一起改 把范围先收住
3 验证命令和证据要求 怎么证明这次改动是对的 先跑命令，再看结果
4 模板或脚手架里的重复结构 下次怎么少写一遍 把缺口写回固定入口
现场对象 原始缺口 补什么 预期变化
目录 README 没有人告诉 Agent 这个目录做什么 职责、入口、常用命令 Agent 先读最近的说明，而不是全仓搜索
目录边界 支付协议、账户体系、优惠券逻辑都混在一起 禁区和例外 Agent 先停在局部修复，不会顺手扩大
任务模板 只写“修一下提交问题” 目标、允许改动、验收、证据 交接后不需要继续追问
脚手架 固定结构只能手写或复制 生成命令和默认文件 新目录的新任务可以直接起步
目录 README 与 AGENTS 片段
## checkout 目录负责什么
这里放结算页、提交状态和相关测试。

## 目录边界
不要改支付协议，不要改账户体系，不要顺手扩结算流程。

## 先看什么
先看根 README 的启动说明，再看这里的验证命令。

## 怎么验证
先跑组件测试，再跑浏览器复现。

## 证据要求
提交时附上截图、日志字段和复跑命令。

为什么这段说明不能再写薄

如果只写“这个目录处理 checkout 相关功能”，Agent 仍然会继续问支付、账户、优惠券、状态机和日志到底算不算这里的责任。它不会因为你说得更温和，就自动知道边界在哪里。

一旦目录边界模糊，后面就会出现两种连锁反应：一是任务范围被扩到别的目录，二是验证时只看当前页面，不会回头看根入口。最后即使改对了，也很难说清为什么对。

故障链路：入口不清时，任务怎样一路跑偏

chain

如果入口不清，Agent 往往会先去找最像入口的文件名。它可能先读主页面、再读测试、再看接口，最后还是不确定这个目录到底负责什么。接下来它会扩大搜索范围，直到把本来只该局部修复的任务，扩成一个需要更多上下文的判断题。

这条链路的危险不在于它“不会做”，而在于它会做得很认真。只要开始范围扩张，后面补一句“不要改太多”通常不够用。应该补的是入口、边界、验收和写回位置，让同类任务下次从一开始就落在正确范围里。

故障链路的观察点

先看它从哪一层开始读，再看它在哪一步把任务扩大，最后看有没有把缺口写回稳定入口。

观察点 表面现象 真实含义 最小修补
入口只有一句口号 Agent 还在翻目录 它不知道从哪里开始读 补 README 的启动说明和第一步
边界只写“注意范围” 它还是去碰无关目录 它不知道哪些对象不能动 补目录边界和禁区
验证只写“检查一下” 改完后没法复查 它不知道该交什么证据 补命令、截图、日志字段
写回只留在聊天里 下一次还是重复问同一件事 信息没有进入稳定入口 写回 README、AGENTS 或模板
最小观测记录
T0 进入仓库，只看到目录名，没有最近入口。
T1 打开目录后，只看到一句“这里是 checkout”。
T2 继续搜索页面、测试和接口，开始把任务当成整条结算链处理。
T3 发现没有验证命令，只能回头补问“改完怎么证明”。
T4 问题被扩大：从一个目录补强，变成整个仓库的范围讨论。

修补顺序：
1. 在目录 README 写职责和禁区。
2. 在根 README 写启动路径。
3. 在模板里写验证命令和证据。
4. 在脚手架里写重复结构的默认生成方式。
步骤 读到什么 开始发生什么偏差 修补点
T0 只有目录名和一堆文件 从搜索开始，找不到最短路径 在目录 README 写清职责和入口
T1 知道目录负责 checkout，但不知道边界 把支付协议也当成可改对象 把禁区写进目录 README 或 AGENTS
T2 看到了报错和截图 只修当前症状，不知道怎么复查 补验收命令和证据要求
T3 任务完成，但模板没变 下次还是会问同样的问题 把缺口写回模板或脚手架说明

- 只改了 checkout 目录的提交状态和相关测试

## 证据
- 组件测试输出
- 浏览器截图
- 日志字段
- 复跑命令

## 写回
- 如果目录缺入口，补目录 README
- 如果模板缺字段，补 issue / PR 模板
- 如果命令总跑错，补脚手架说明

这条链路怎么变成练习题

你在练习里要做的，不是背一套漂亮话，而是能看到“缺入口会先导致搜索扩张，缺边界会导致范围扩张，缺验证会导致结果不可复查”。只要你能把这个链路写出来，下次见到类似任务，就能先补对位置。

进一步说，真正要防的是连锁反应：入口不清会让阅读顺序变长，边界不清会让修改对象变多，验证不清会让审查回到聊天。你只要把这三步拆开，很多看起来很大的问题其实都只是一个缺口没补到位。

写回链路：把一次失败沉淀成下一次能读到的规则

writeback

仓库知识最值钱的地方，不是“这次任务做完了”，而是“下次不用再问同样的问题”。所以每次任务结束后，要沿着同一条链路检查：这次缺的是入口、边界、模板、脚手架，还是验证命令。找到缺口以后，只写回最靠近使用场景的地方，不顺手开一篇新说明。

比如一个 checkout 目录的任务，如果 Agent 反复问从哪里开始，那就说明入口不够近；如果它把支付协议也改了，那就说明边界不够硬；如果审查总追着要截图，那就说明模板没把证据字段固化；如果固定结构每次都手写，那就说明脚手架没接上。每一种失败，都对应一个明确的写回位置。

写回判断顺序

  1. 先判断是入口问题、边界问题、模板问题还是脚手架问题。
  2. 再找最接近使用场景的位置。
  3. 最后只补最小字段，不做顺手改写。
问题类型 优先写回到哪里 写回时只补什么 不该顺手做什么
从哪里开始读 README 第一步和常用入口 不要改成项目全史
哪些目录能动 目录 README 或 AGENTS 职责、禁区、例外 不要把别的目录说明带进来
改完怎么证明 PR 模板 截图、日志、命令 不要只写“已验证”
重复结构怎么生成 脚手架说明 生成命令和默认文件 不要只留下复制粘贴做法
失败现象 真实缺口 写回位置 下一次会怎样
总问从哪读起 入口不近 README 先读入口，不再从搜索开始
总改到无关目录 边界不硬 目录 README 或 AGENTS 先停在正确层级，再开始改
PR 总被追问截图 证据字段没固化 PR 模板 默认提交可复查证据
重复结构总要手写 脚手架没接上 脚手架说明或生成脚本 固定结构能直接生成
写回最小补丁清单
1. 入口不清：在 README 加一段“先看什么”。
2. 边界不硬：在目录 README 加职责和禁区。
3. 证据不全：在 PR 模板加截图、日志、命令字段。
4. 结构重复：在脚手架说明里写生成命令。
5. 常见误区反复出现：补一条反例或检查清单。

这一链路的结束方式

不是把这次任务写成长复盘，而是把最小补丁落到固定入口。下一次 Agent 读到这些补丁时，就不需要再从聊天里找上下文，也不会把一个局部任务扩大成整仓库问题。

如果你愿意再往前走一步，就把“问题类型 -> 写回位置 -> 最小字段”做成一张固定速查表。这样每次处理完任务，不是额外发明一个新结论，而是把相同的缺口填回同一条入口。

一致性链路：同一条规则要落在三处

consistency

仓库知识真正难的地方，不是写出一句正确的话，而是让这句话在三个地方说的是同一件事：根目录告诉你从哪开始，目录说明告诉你这里能做什么，模板告诉你交付时要带什么。只要三处口径不一致，Agent 就会在不同文件里读到不同答案。

一个常见场景是：根 README 说先读启动说明，目录 README 却没有写边界，PR 模板也没写证据字段。结果就是入口有了，执行时仍然会问边界，交付时仍然会追证据。页面看起来“有文档”，实际还是靠人盯着补空洞。

三处一致，才算真的写进去

  • 根 README 负责起点。
  • 目录 README 负责当前目录的职责和禁区。
  • 模板负责验证和写回字段。
位置 应该回答的问题 如果漏掉会怎样 修补动作
根 README 先从哪里开始读 Agent 还会从搜索开始 加第一步和常用入口
目录 README 当前目录能做什么、不能做什么 任务会顺手扩到别处 加职责、禁区和例外
PR / issue 模板 改完怎么证明 交付时反复追问证据 加截图、日志和命令字段
脚手架 重复结构怎么生成 固定结构靠复制粘贴 加生成命令和默认文件
一致性检查清单
1. 根 README 是否写了第一步。
2. 目录 README 是否写了职责和禁区。
3. 模板是否写了验证命令和证据字段。
4. 脚手架是否写了生成方式。
5. 这四处说的是不是同一件事。

如果一致性没补齐，会出现什么

最常见的结果是：新人看完觉得“文档都有”，真正开始做的时候还是不知道先读哪份、先改哪里、最后交什么。也就是说，文档存在，但入口没有合并，任务只能继续靠问答拼出来。

练习里的第 9 题，就是把“请认真工作”改成可执行的 AGENTS 片段。那道题其实就在检查这条链路：你能不能把一句正确但空的说法，写成能落到根 README、目录 README 和模板里的同一条规则。

最小自检

检查一致性时，不要先问哪一份文档写得最长，先问同一条规则有没有在三个位置说同一句话。最常见的漏点不是没有写，而是只写在一个位置，其他位置还沿用旧说法。

如果根 README 有起点、目录 README 有边界、模板里却没有证据字段，这不是“文档挺全”，而是“规则没落稳”。修补时先补最近的位置，再回头校准上一级，最后再跑一次页面验证。

检查对象 常见漏点 你会看到什么 先补哪里
根 README 第一步没写 Agent 仍从搜索开始 补启动说明和入口
目录 README 边界没写硬 改动会顺手扩出去 补职责、禁区和例外
PR / issue 模板 证据字段缺失 交付时还要临时追问 补截图、日志、命令
脚手架 生成方式没写 固定结构只能复制粘贴 补默认文件和生成命令
一致性自检顺序
1. 先看根 README 的第一步。
2. 再看目录 README 的职责和禁区。
3. 然后看模板里的证据字段。
4. 最后确认四处说的是不是同一件事。
三处口径示例
根 README：先看启动说明。
目录 README：这个目录负责 checkout 页面和提交状态，不碰支付协议。
PR 模板：提交时附截图、日志和复跑命令。

如果其中一处没写，Agent 还是会补问同一个问题。
所以真正的修补不是多写一句，而是让三处都指向同一个动作。

练习里的对应点

第 3 题、第 6 题和第 12 题都在测这件事：你能不能把职责、边界和证据拆开，再写回不同位置。只要这三题答顺了，说明你已经能用同一条规则去校准根入口、目录入口和交付模板。

再往前半步，就是把同一条规则放进一次真实改动里复查：根 README 给起点，目录 README 给禁区，模板给证据。只要其中一处和另外两处不一致，下一次 Agent 就会重新问同一个问题，这时就该先补最近的那一层，而不是先加更多解释。

这个检查并不复杂，难的是别把“写过了”误当成“写稳了”。把三处都跑一遍，你才能判断问题到底是入口没写、边界没写硬，还是模板没有把证据字段固定住。

做完这个检查以后，再回头看导航和页面锚点是否一致。它们本来就是同一件事的两个版本：一个写给读者走路，一个写给页面自己跳转。

这一步会直接减少下一次追问和返工，也让页面更容易复查和写回。

状态台账：读、写、失效、回写

ledger

仓库知识会失效。目录变了、命令变了、模板变了、边界变了，原来的说明就不一定还对。台账把“读了什么、写了什么、什么时候失效、失效后写回哪里”讲清楚。

只要台账在，下一次碰到同类问题，先查哪里失效就比重新猜快得多。

台账看的不是内容量

看的是它能不能帮助你定位：问题是入口错了、边界错了、命令错了，还是证据错了。

对象 读什么 写什么 何时失效 失效症状 回写位置
AGENTS.md 全局规则、验证方式 仓库级边界、公共约定 仓库根规则变化 Agent 反复问基础问题 仓库根 AGENTS
README.md 项目介绍、启动路径 入口、命令、常见路径 启动方式变化 跑错命令或找不到入口 README 本身
目录 README 目录职责、禁区、例外 本目录规则、验证命令 目录职责变化 改错目录或跨层修改 最近的目录 README
任务模板 目标、范围、证据 补齐缺失字段 常见任务换了一类 交接后仍然缺字段 issue / PR 模板
脚手架 固定结构、生成方式 固定文件和默认说明 重复结构改版 重复手写、结构漂移 脚手架说明或生成脚本

写回位置：把失误留到下次读得到的地方

bridge

写回的目标是让下一个任务少踩同一个坑。模板缺字段，就写回模板；目录边界缺说明，就写回最近的目录 README；命令总跑错，就写回启动文档或脚手架说明。

写回位置越近，下一次越容易被读到。写到聊天记录里，通常等于没写回。

最小写回顺序

  1. 先记清缺了什么字段。
  2. 再找最接近使用场景的位置。
  3. 只补最小字段，不顺手重写整页。
目录 README 最小补丁
## 这个目录负责什么
这里放 checkout 页面和提交状态相关的内容。

## 不能碰什么
不要改支付协议，不要改账户体系。

## 先看什么
先看 README 的启动说明，再看本目录的验证命令。

## 怎么验证
先跑组件测试，再跑浏览器复现。

## 证据要求
提交时附上截图、日志字段和复跑命令。
失败信号 说明缺什么 先写回哪里 下次怎么受益
Agent 总问从哪里开始 入口不清 README 下一次能直接找到起点
Agent 改到无关目录 边界不清 目录 README 或 AGENTS 知道该停在哪一层
PR 反复追问证据 验收不清 PR 模板 默认带上截图和命令

练习目标

这组题考的是你能不能把仓库知识拆成入口、边界、模板、脚手架和验证命令。要说清补什么、放哪、为什么能改变 Agent 的动作。

检查重点

先看入口是否清楚，再看边界是否贴近目录，最后看证据和写回位置是否足够稳定。开放题按“对象、位置、作用、证据、写回”来答。

参考答案 / 评分要点

合格答案会把说明写到最近的目录里，明确这个目录负责什么、不能碰什么、怎么验证。评分重点在职责、禁区和命令三个部分。例如：这个目录负责 checkout 页面和提交状态；不要改支付协议和账户体系；先跑组件测试，再跑浏览器复现。

三、模板与脚手架

把重复结构变成固定字段，再让命令生成。

复盘提示模板管字段，脚手架管生成。

synthesis

至少写出一个职责、一个禁区和一个验证动作。

参考答案 / 评分要点

好答案会把口号改成具体说明，例如：这个目录负责 checkout 页面和提交状态；不要改支付协议和账户体系；先看 README 的启动说明；先跑组件测试，再跑浏览器复现；提交时附上截图、日志字段和复跑命令。只写“请认真工作”没有信息密度。

四、失败诊断与写回

看懂症状，再把缺口补回下次能读到的位置。

复盘提示写回要离使用场景近。

true/false

11. 判断：如果 PR 总被追问证据，最该补的是 PR 模板里的证据字段。

diagnosis

12. 说出一次失败后应该写回哪里，以及为什么不能只留在聊天记录里。

至少说出一个写回位置和一个后续受益点。

参考答案 / 评分要点

合格答案会把失误写回最接近的稳定位置，例如目录 README、AGENTS、PR 模板或脚手架说明。原因是聊天记录下一次通常读不到，只有仓库里的稳定入口才会被后续任务看到。更好的答案还会说明下一次怎么受益，比如减少重复追问、减少改错目录、默认带上验证命令和证据字段。
