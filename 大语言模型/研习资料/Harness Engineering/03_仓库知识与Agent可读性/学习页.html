Repository Knowledge

把仓库知识写成 Agent 能读的地图

OpenAI 原文强调，Codex 能不能高效工作，很大一部分取决于仓库本身是不是容易被它读懂。人类工程师熟悉的口头约定、历史原因、目录边界和风格偏好，如果没有写进仓库，Agent 每次都要重新猜。

这一页把仓库知识当成 harness 的核心部分。它不是写一份长到没人读的文档，而是把 Agent 最需要的入口、规则、模板、脚手架和示例放在它会查到的位置。

入口文档→目录边界→模板→脚手架→示例→测试命令→审查规则

学完后要能做什么

  1. 判断哪些仓库知识应该写进 AGENTS 或 README。
  2. 为一个目录设计 Agent 友好的入口说明。
  3. 区分文档规则、代码模板、脚手架和测试约束的适用场景。
  4. 诊断 Agent 重复踩坑时，仓库知识缺在哪里。

学完本块后，你应该能给一个项目补一页让 Agent 少问问题的仓库地图。

概念骨架

MODEL

仓库知识的直觉是：新同事第一天会问的问题，Agent 也会问。项目怎么启动，代码放哪，测试怎么跑，哪些目录不能动，常见任务有没有模板。人类可以口头解释，Agent 更依赖可读取文件。

正式说，仓库知识是存放在项目内、能被 Agent 检索和遵守的上下文。它包括 AGENTS.md、README、目录说明、任务模板、代码生成脚手架、测试命令、设计记录和反例。它的目标不是完整记录一切，而是让 Agent 在关键分叉点做对选择。

判断句是：如果同一类问题反复需要人类提醒，说明这条知识还没有进入仓库，或者进入的位置不够靠近任务现场。

本块主线

  • 入口文档回答“我是谁、在哪里、先读什么”。
  • 目录说明回答“这里能放什么、不能放什么”。
  • 模板和脚手架回答“照什么格式做”。
  • 测试命令和审查规则回答“怎样证明做完”。

术语表：先把词放稳

TERMS
术语直觉解释准确边界常见误解
AGENTS.md给 Agent 的项目级协作说明。写任务约定、命令、边界和风格偏好。把所有知识都塞进去。
目录边界某个目录负责什么、不负责什么。能阻止跨层引用和乱放文件。只靠目录名字让 Agent 猜。
脚手架可运行的生成工具或模板。把重复结构变成命令或示例。把脚手架当成文档的替代品。
示例可复制的本地范式。展示项目真实风格，比抽象规则更好用。用过期示例误导 Agent。
上下文预算Agent 一次能有效处理的信息有限。入口文档要短而准，复杂细节分层链接。写一份巨大总文档。

讲义主线

LESSON

下面的小节按“问题、做法、判断”推进。每节最后都有一句检查点，用来确认你不是只记住了标题。

1. 入口文档要短，先解决定位问题

ENTRY

Agent 进入仓库时，最需要的不是公司历史，而是定位：这个仓库做什么，当前任务常改哪些目录，常用命令是什么，哪些文件不能随便动。如果入口文档太长，关键信息会被淹没。

好的入口文档通常分层。顶层 AGENTS.md 写全局原则和常用命令；子目录 README 写本目录边界；复杂流程再链接到更细文档。Agent 可以先读短入口，再按任务深入。

入口文档的检查方法很简单：假设一个新 Agent 只有 2 分钟读文档，它能不能知道从哪里开始、怎么验证、哪些坑别踩。

读完这一节，你应该能把一份冗长说明压成 Agent 第一屏能读完的入口。

2. 目录边界比抽象架构口号更有用

BOUNDARY

很多架构规则写得很抽象，例如“保持模块清晰”。Agent 需要更具体的边界：src/services 不能导入 UI，components 不能直接访问数据库，routes 只做参数解析和权限检查。这些规则越靠近目录，越容易被遵守。

目录边界应该写“可以做什么”和“不要做什么”。只写职责，Agent 仍可能把相邻层的逻辑放进来；写出反例和修法，会更稳。比如“如果需要格式化展示文本，请在 view model 层处理，不要在 service 层拼 UI 文案”。

边界不是为了束缚开发，而是减少 Agent 的搜索空间。它知道不能走哪条路，就能更快找到正确位置。

读完这一节，你应该能为一个目录写出职责、禁区和替代做法。

3. 模板比口头风格偏好更可靠

TEMPLATE

人类常说“照现有风格写”。如果仓库里示例分散、版本不一，Agent 很可能复制到过期写法。模板把风格变成可复制结构：文件布局、命名、错误处理、测试形态、日志字段都在里面。

模板不一定复杂。一个最小路由样例、一个标准测试文件、一个任务说明模板，就能显著降低误差。更重复的任务可以做成脚手架命令，让 Agent 运行命令生成骨架，再填业务逻辑。

模板也要维护。过期模板会比没有模板更糟，因为 Agent 会把它当成权威。每次架构迁移后，要同步更新模板和示例。

规则

适合说明边界和原则。

示例

适合展示项目真实写法。

脚手架

适合重复生成结构。

读完这一节，你应该能判断一个重复任务该写文档、给示例还是做脚手架。

4. 仓库知识要照顾上下文预算

CONTEXT

Agent 不是读得越多越好。上下文太杂会让关键规则被稀释。仓库知识要有层级：顶层只写全局约定，目录说明只写本目录规则，任务模板只写当前任务需要填的字段。

信息要放在 Agent 会找的地方。比如修改 API 路由时，路由目录里的 README 比根目录深处的长文档更有效；生成组件时，组件目录里的示例比产品规范更直接。

好的仓库知识有一个特点：它让 Agent 少读无关内容。少读不是偷懒，而是减少错误关联。

读完这一节，你应该能按“全局、目录、任务”三层组织上下文。

5. 仓库知识必须跟着真实反馈更新

MAINTAIN

文档一旦和真实代码不一致，Agent 会更困惑。它可能读到旧命令、旧目录、旧权限规则，然后生成看似合理但已经过期的改动。仓库知识需要像代码一样维护。

更新的触发点很明确：人类审查指出重复问题；任务失败暴露缺少命令；架构迁移改变目录职责；新脚手架替代旧手写流程。这些都应该反映到入口文档、模板或规则里。

维护仓库知识不是额外文书工作，它是提高下一次 Agent 成功率的投资。

读完这一节，你应该能从一次失败中提取需要更新的仓库知识。

6. 完整链路：入口、边界、模板、验证

CHAIN

把这一页串起来：Agent 从入口文档定位项目；进入相关目录后读边界说明；用模板或脚手架生成符合风格的结构；按文档里的命令验证；人类审查后把新反馈写回文档或工具。

这个链路解决的是“Agent 第一次来项目时像不像新同事”。新同事需要 onboarding，Agent 也需要，只是它的 onboarding 更适合写进仓库、靠工具执行。

以后你看到 Agent 反复问“这个项目怎么跑”“文件放哪”“测试命令是什么”，就该补仓库知识，而不是每次重新解释。

读完这一节，你应该能为一个目录设计 Agent onboarding 的最小版本。

最小例子：把概念落到工程动作

EXAMPLES

例子 1：目录级 README

目录说明要短，直接告诉 Agent 怎么改。

正确示例：routes 目录说明Markdown
# src/routes

职责：定义 HTTP 路由、参数解析、权限检查和响应格式。

不要做：
- 不直接访问数据库；调用 src/services
- 不拼 UI 文案；返回结构化错误码
- 不在路由里写复杂业务分支

新增路由：复制 _template.route.ts，并补 route.test.ts。
验证：pnpm test routes/<name>

这比“保持路由层干净”更能指导 Agent。

例子 2：任务模板

任务模板让每次委托都带上同样的关键字段。

正确示例：Agent 任务模板Markdown

例子 3：错误示例，入口文档太空

只欢迎 Agent，没有工作信息。

错误示例：无效 AGENTSMarkdown
欢迎来到本项目。请认真工作，写出高质量代码，注意保持风格一致。

这些话没有可执行信息。Agent 仍不知道目录、命令、边界和验收。

常见坑：错因、后果和修法

PITFALLS
误区错因后果修法
AGENTS 写成口号只有态度要求，没有工程信息。Agent 仍要猜命令、目录和边界。写常用命令、目录规则、验证方式和禁区。
文档过长把所有背景塞进一个文件。关键规则被稀释，读取成本高。按全局、目录、任务分层。
示例过期代码迁移后没有更新模板。Agent 复制旧写法。把模板维护纳入架构变更清单。
只写允许，不写禁止边界说明不完整。Agent 把相邻层逻辑混进来。写职责、禁区和替代位置。
没有验收命令文档只讲怎么写，不讲怎么查。Agent 无法证明完成。每类任务附最小验证命令。

易混概念对照

COMPARE
概念组容易混的地方区分方式工程判断
AGENTS / README都能写说明。AGENTS 偏 Agent 协作规则；README 偏人类和项目介绍。Agent 专用约定放 AGENTS，更通用入口放 README。
规则 / 示例都能引导风格。规则说明边界，示例展示真实写法。新模式优先给示例。
模板 / 脚手架都能复用结构。模板靠复制，脚手架靠命令生成。重复多、字段固定时做脚手架。
全局文档 / 目录文档都能提供上下文。全局文档写通用原则，目录文档写局部边界。信息放到离任务最近的位置。
