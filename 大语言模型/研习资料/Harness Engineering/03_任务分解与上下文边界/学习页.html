课程第 03 节

任务分解与上下文边界

这一页把“让 Agent 做事”压成两个动作：先把模糊需求拆成一个能执行、能验收的任务卡，再把它需要读取的上下文收窄到刚好够用。这里不追更复杂的调度系统或多 Agent 编排，先练会写清楚目标、输入、边界、证据和停手条件。

课程地图上一节下一节原文

读完本页应该能做什么

  • 把一句模糊需求拆成目标、输入、产出、验收证据和禁止范围。
  • 为 Agent 选择足够的上下文包，避免读少了乱猜、读多了跑偏。
  • 从一次任务失败里判断是目标不清、边界过宽、证据不足还是上下文过期。

学习目标

start here

这一节承接前两页：第 01 节说人类要设计 Agent 的工作环境，第 02 节说结果要能被证据回看。现在要处理中间最容易松掉的一环：任务到底怎么交给 Agent。

一个好任务不能只写“帮我改一下登录页”。它至少要说明目标状态、当前输入、允许改动的位置、不能碰的边界、验收证据和停手条件。少了其中一项，Agent 可能仍然很努力，但它会把努力花在错误范围里。

本页的练习对象

我们用一个持续出现的小场景：结算页提交按钮会卡住。你会把它从一句报错拆成任务卡，再为它选择上下文包，最后判断失败应该写回哪里。

主线：先拆任务，再给上下文

map

Harness engineering 的重点是环境。任务分解就是环境的一部分：它决定 Agent 看到什么目标、能动什么文件、用什么证据证明完成。上下文边界也是环境的一部分：它决定 Agent 读哪些规则、样例和反例，哪些材料暂时不该进入窗口。

直觉上，你可以把任务卡看成给 Agent 的工单，把上下文包看成它开工前桌面上放的材料。工单太粗，Agent 会自己补意图；材料太多，Agent 会把无关规则也当成线索。两者都要刚好够用。

一句工作定义

可执行任务 = 目标状态 + 输入材料 + 改动边界 + 验收证据 + 停手条件。

上下文边界 = 当前任务必须读取的规则、样例、命令和证据，不包含会诱导跑偏的历史材料。

对象 它回答什么 典型材料 失控症状
任务卡 这次到底要完成什么 目标、输入、输出、验收、禁止范围 Agent 改了很多地方，但没人知道算不算完成
上下文包 开工前必须读什么 AGENTS、README、相关文件、测试命令、相似实现 Agent 读少了乱猜，读多了引入无关规则
边界条件 哪些东西不能碰 目录范围、接口契约、权限、数据模型、不变量 局部修复破坏架构或产品约束
反馈写回 失败后该沉淀什么 模板字段、文档补丁、测试、反例、脚手架 同类任务下一次继续踩坑

把模糊需求拆成能力块

task split

能力块关注 Agent 动作的可判定粒度，和模块拆分、TODO 清单不同。比如“修结算页卡住”太粗；“复现按钮卡住、定位状态来源、补失败提示、跑通提交路径、留下证据包”就更容易执行。

目标 用户点击提交后，要么成功进入下一步，要么看到明确失败提示。

目标要描述状态，不要只写“修好”。

输入 结算页路径、复现步骤、截图、request_id、最近一次失败日志。

输入要让 Agent 从同一个现场开始。

动作 允许改结算页状态处理、错误提示和相关测试。

动作范围要比仓库范围小。

证据 测试输出、浏览器截图、日志字段和复跑步骤。

证据决定审查者能不能复查。

停手 不重构支付网关，不改账户体系，不扩展结算流程。

停手条件保护任务不被顺手扩大。

能力块 具体动作 验收方式 边界
复现 按给定路径重放按钮卡住 截图或日志能证明同一现象 不先改代码
定位 判断卡在 UI 状态、请求、后端还是下游 有 request_id、错误码或 DOM 状态 不靠猜测归因
修复 只改本次缺陷需要的状态处理或提示 相关测试和浏览器复跑通过 不顺手改支付架构
写回 把新增证据字段、复现步骤或模板缺口补回仓库 后续任务能复用 不写无关大文档

上下文边界怎么画

boundary

上下文边界的核心判断是：这份材料会改变当前任务的正确动作吗？会，就放进上下文包；不会，就不要让它占用注意力。Agent 可以读很多东西，但读得多不等于理解得准。无关材料会制造错误联想，过期材料会制造旧规则。

边界要按任务收，不按“看起来相关”收。修结算按钮时，相关的是结算页状态、提交请求、错误提示、测试命令和近期失败日志；支付网关长期重构计划、旧版埋点讨论、另一个页面的视觉规范，除非任务明确需要，否则先不放进来。

三个判定问题

  • 没有这份材料，Agent 会不会改错对象？
  • 这份材料会不会让 Agent 扩大任务范围？
  • 这份材料是否比当前代码、测试或模板更新？
材料 放入条件 不放入时的风险 放错时的风险
AGENTS / 项目约定 影响测试、风格、权限或目录规则 违反项目约定 读到上级泛规则，忽略本目录规则
README / 启动说明 需要运行项目或复现问题 无法启动或跑错命令 把历史命令当成当前命令
相关源码 包含当前行为或相似实现 只能凭描述改代码 读入太多相邻模块，改动越界
测试与失败日志 能证明问题或验收修复 无法复查 用旧失败覆盖当前现场
设计讨论 包含当前不可破坏的不变量 破坏隐藏约束 把长期设想误当成本次目标

上下文包：给 Agent 的开工桌面

context pack

上下文包应该像一张小桌面：左边是任务卡，中间是必须读的规则和文件，右边是验收证据。不要把整间办公室都搬进来。一个好的上下文包让 Agent 能立刻回答三件事：我要达成什么、我能改哪里、我怎么证明自己做完了。

如果任务需要跨前端和后端，包里可以有两条路径，但仍然要写清主线。比如结算页卡住：主线是提交状态和失败提示；后端日志只是用来确认原因；支付网关重构不是本次目标。

最小上下文包

  • 任务卡：目标、输入、边界、验收。
  • 入口文件：相关页面、接口、测试或脚手架。
  • 项目规则：AGENTS、目录 README、命令说明。
  • 证据：复现步骤、截图、日志、失败输出。
上下文包模板

输入材料：复现路径、失败截图、request_id、最近一次日志
允许改动：结算页状态处理、错误提示、相关测试

禁止范围：不重构支付网关，不改账户体系，不扩大结算流程
验收证据：测试输出、浏览器截图、日志字段、复跑步骤
停手条件：验收通过后只写回必要规则或模板缺口
组成 作用 检查问题 常见缺口
任务卡 固定本次目标 有没有明确完成状态 只写“优化”“修复”“完善”
文件入口 减少搜索成本 Agent 从哪里开始读 让 Agent 全仓库乱找
规则入口 避免违反约定 哪些规则会影响这次改动 只给代码不给项目约定
证据入口 让验收可复查 用什么证明完成 只有口头总结，没有输出

从人类意图到 Agent 交接

handoff

交接要把人类意图翻译成工程对象：目标状态、改动范围、可读上下文和证据。这个翻译过程越清楚，Agent 越不需要自己猜产品意图、架构约束或验收标准。

交接时最危险的是隐性限制。比如“别影响老用户优惠券”“不要改支付网关”“只处理移动端布局”，这些限制如果没有写进任务卡，Agent 很可能从局部最容易修的位置下手，最后造成范围外影响。

交接检查顺序

  1. 先写目标状态，不先写实现方案。
  2. 再写允许改动的范围。
  3. 补上必须读取的上下文。
  4. 列出验收证据和停手条件。
字段 好写法 坏写法 坏写法会导致什么
目标 提交成功进入下一步，失败显示错误提示 修一下提交问题 完成状态不明确
输入 复现步骤、截图、日志、相关测试 你自己看看 Agent 从错误入口开始
范围 只改结算页状态处理和错误提示 相关地方都可以改 改动扩散到无关模块
验收 测试输出、截图、日志字段齐全 看起来可以就行 审查者无法复核
停手 不重构支付网关，不改账户体系 顺便优化一下 任务变成开放重构

状态台账：读、写、失效、回写

ledger

任务交给 Agent 后，你要能追踪它读了什么、写了什么、哪些信息已经失效、失败后该写回哪里。台账的价值在于把一次失败沉淀成下一次可读的任务边界。

失败链路：任务怎样一步步跑偏

debug

任务跑偏通常源于错误搜索空间。模糊目标会让 Agent 先猜问题类型；缺少输入会让它扩大检索；缺少边界会让它把局部修复升级成重构；缺少证据会让它在看起来能跑通的地方停下。

所以自审时不要只看最终改了哪些文件，要回放它每一步为什么会做那个动作。只要能指出“哪一个字段没给，导致哪一次猜测变大”，下一版任务卡就有明确修补点。

回放顺序

  • 先看目标：它能否判断完成状态。
  • 再看输入：它能否复现同一个现场。
  • 再看边界：它能否知道哪些方向要停住。
  • 最后看证据：它能否留下可复查结果。
时间点 任务卡缺口 Agent 的合理猜测 跑偏动作 修补字段
T0 只写“修提交问题” 问题可能在 UI、接口、支付、账户或优惠券 从全仓搜索 submit 和 payment 目标状态 + 复现入口
T1 没有 request_id 和日志 只能靠代码结构猜失败点 开始读支付网关适配层 输入证据 + 最小日志片段
T2 没有禁止范围 如果支付网关可疑，就可以改它 重构 gateway timeout 分支 允许改动 + 禁止范围
T3 验收只写“测试通过” 单测绿灯即可交付 没有重新打开结算页截图 验收证据 + 浏览器复跑步骤
T4 没有写回要求 任务结束后不更新模板 下次仍然缺 request_id 字段 写回位置 + 模板补丁

范围矩阵：读、改、禁止、回写

scope

边界要靠可执行权限建立。更稳的写法是把对象拆开：哪些文件只允许读取，哪些文件允许修改，哪些模块必须禁止，哪些发现要写回模板。这样 Agent 不需要靠风格猜测你的意图，审查者也能直接判断某个改动是否越界。

范围矩阵适合放在复杂 issue、PR 描述或任务交接文档里。它把“上下文边界”从抽象原则变成可检查条目：一列对象，一列动作权限，一列验收证据，一列写回位置。

矩阵的使用时机

  • 需求涉及多个目录，但本次只想改其中一层。
  • 存在高风险模块，例如支付、权限、迁移、计费。
  • 任务需要读旧决策，但不能顺手推翻旧决策。
  • 验收需要 UI、日志、测试、文档一起对齐。
对象 可读 可改 禁止动作 验收证据 写回位置
结算页组件 状态流、错误文案、提交按钮逻辑 允许 不能顺手重做整页布局 截图、组件测试、e2e 复跑 PR 描述
支付网关适配层 错误码、超时语义、日志字段 默认不允许 不改协议、不改重试策略 日志字段引用 任务卡边界
账户与权限模块 只读相关调用链 禁止 不改鉴权、不改角色判断 无改动说明 PR 审查说明
测试目录 既有用例、fixture、e2e 脚本 允许 不删除覆盖异常分支的旧用例 测试输出 测试说明
issue / PR 模板 现有字段 只补最小缺口 不重写整套流程 模板 diff 仓库模板
边界自审清单
交付前自审：
1. 我能说清本次目标对应的用户可见状态或系统可见状态。
2. 我能列出 Agent 必须读取的上下文，并删掉不会改变当前动作的材料。
3. 我能指出至少一个禁止范围，尤其是高风险模块和长期架构决策。
4. 我能解释每个改动文件为什么属于“允许改动”。
5. 我留下的证据能让另一个人复跑，不需要回到聊天记录猜过程。
6. 如果任务暴露出模板或文档缺口，我知道写回到哪个稳定位置。

自审闭环：交付前最后看一次边界

review

任务完成前，要把“我做完了”翻译成“别人能复查”。这一步经常暴露上下文边界的真实质量：如果你说不清为什么读了这些文件、为什么改了这些文件、为什么没有改高风险模块，那么任务卡里的边界还不够硬。

自审闭环的顺序很简单：先拿原始任务卡对照结果，再拿范围矩阵对照改动文件，接着拿验收证据对照目标状态，最后把暴露出来的缺口写回稳定位置。这个动作不需要很重，但它能阻止同类任务每次都重新踩坑。

闭环判断

  • 目标：结果能否用一句用户可见状态说明。
  • 范围：每个改动文件都能落到允许范围。
  • 证据：截图、日志、测试输出可以互相印证。
  • 写回：模板、README、AGENTS 或测试说明得到最小补丁。
自审问题 通过信号 失败信号 修补动作
目标是否完成 成功路径和失败路径都有可见状态 只能说“代码改了”或“测试过了” 补用户状态、系统状态和复跑步骤
上下文是否足够 引用的日志、截图、文件都解释了当前动作 读了大量无关背景，交付说明讲不清原因 删掉无用材料，保留会改变动作的证据
范围是否守住 改动集中在允许目录，高风险模块只有只读证据 顺手改了协议、权限、计费或全局样式 拆出新任务，当前任务回到原边界
证据是否可复查 另一个人能按命令、截图和日志复跑 结论依赖聊天记录或个人记忆 补命令输出摘要、截图路径、request_id
经验是否写回 缺失字段回到模板，常见误区回到清单 下次任务仍要重新解释同一个规则 补最小模板字段或目录级规则
交付说明模板
交付说明：
目标结果：说明用户或系统现在能看到什么变化。
关键输入：列出本次使用的截图、日志、request_id、复现入口。
改动范围：列出改动文件，并说明它们分别落在哪个允许范围。
未改范围：列出读过但未改的高风险模块，说明为什么停在只读。
验收证据：贴测试命令、浏览器截图、日志字段、手动复跑步骤。
写回结果：说明是否补了模板、README、AGENTS、测试说明或反例清单。
后续任务：如果发现更大的架构问题，单独开新任务，不混进当前交付。

案例：把结算页任务拆错以后会发生什么

原始任务如果只写“修提交问题”，Agent 通常会从最显眼的按钮或日志开始搜。它可能先修一个看起来相关的支路，然后把时间花在更大的支付链路上，却忘了最初要的是“提交成功后进入下一步，失败时显示明确错误提示”。这种偏差很常见，因为任务卡没有把目标状态、边界和证据绑在一起。

一旦出现这种偏差，最有效的动作不是再加几句解释，而是把任务卡重写成可执行版本，再把这次暴露出来的缺口写回模板。这样下一次拿到同类任务的人，能直接读到复现入口、允许改动和停手条件。

步骤 看到的信号 错误推进 正确修正
读任务 只有“修提交问题” 把范围理解成整个支付链路 补目标状态、输入和停手条件
找上下文 看见按钮 disabled、日志超时 直接修改适配层和重试逻辑 先确认允许改动的文件和禁止模块
做改动 测试通过，但页面状态没恢复 只盯单测结果，不回看浏览器 补 e2e、截图和手动复跑步骤
交付后 模板仍然缺 request_id 字段 同类任务下次继续靠猜 把缺口写回 issue / PR 模板
坏卡与修正版
坏卡：
修一下结算页提交问题，测试跑通就行。

修正版：
目标：提交成功后进入下一步；提交失败时停留在当前页并显示明确错误提示。
输入：/checkout 的复现路径、request_id、截图、最后一次失败日志。
允许改动：结算页状态处理、错误提示、相关测试。
禁止范围：支付网关协议、账户体系、优惠券逻辑。
验收：组件测试、e2e 截图、日志字段、手动复跑步骤。
写回：如果模板缺少复现入口或日志字段，把字段补回 issue / PR 模板。

练习

拿你手边的一个任务，把它按上面的结构重写一遍。先写目标状态，再补输入和边界，最后列出验收证据和写回位置。写完以后，你应该能把它直接放进 issue 或交给 Agent。

现场诊断

如果你发现任务已经开始改高风险模块，就先停下来，把结果退回到任务卡修正，而不是继续追着代码打补丁。这个动作能把“局部修复”重新拉回“可验收交付”。

诊断时只看四件事：目标是否完整、输入是否可复现、边界是否锁住高风险模块、证据是否能复查。四件事里缺任何一件，任务卡就还不算完成。

一旦范围越过边界，后面再加说明也只能扩大误差，正确动作是拆新任务并把缺口写回模板内。

落到仓库：哪里写任务，哪里写规则

repo

任务分解要落到仓库。稳定的做法是把可复用部分写回仓库：任务模板写在 issue 或 PR 模板里；目录规则写在 AGENTS 或 README 里；测试命令写在开发文档或脚手架里；反复出现的错误写成反例或检查清单。

写回时要克制。一次任务失败不一定需要一篇长文档，它可能只需要在模板里加一个字段：复现入口、禁止范围、验收证据、日志字段。字段越靠近使用场景，下次越容易被读到。

写回位置速查

  • 任务字段缺失：补 issue / PR 模板。
  • 目录规则缺失：补 AGENTS 或目录 README。
  • 命令经常跑错：补开发文档或脚手架提示。
  • 同类错误反复出现：补测试、反例或审查清单。
失败信号 说明缺什么 写回哪里 下一次怎么受益
Agent 总问从哪里开始 入口不清 任务模板 / README 直接找到入口文件和复现路径
Agent 改到无关目录 边界不清 AGENTS / 目录说明 知道哪些目录不能碰
Agent 跑错测试 验收命令不清 开发文档 / 脚手架 按任务类型选正确命令
PR 反复要求补证据 证据字段不清 PR 模板 默认提交截图、日志、测试输出

14. 用 6-8 句话解释：为什么任务分解和上下文边界是 harness 的一部分。

参考答案 / 评分要点

好的答案会说：harness 不是一条 prompt，而是 Agent 工作的工程环境。任务分解决定 Agent 要达成什么、能改哪里、怎么证明完成；上下文边界决定它开工前应该读哪些规则、文件和证据。目标不清会让 Agent 自己补意图，边界不清会让改动越界，上下文过多或过少都会造成错误判断。失败后把模板、规则、测试或反例写回仓库，下一次 Agent 才能直接读到。答案要有因果链，不要只说“更高效”。
