第 4 讲

架构约束与品味不变量

这一页讲的是：当你觉得“代码写得好不好”时，怎样把这种判断落到可检查的规则上。重点不是把所有审美都变成硬规矩，而是把会长期影响系统的部分写成不变量、反例、测试和审查标准，让 Agent 也能稳定执行。

收尾注：规则为什么必须写回最近的位置

closing-note

如果一条规则只在聊天里出现过一次，它就很容易在下次被忘掉。工程品味要真正变成约束，必须把它写回最近的位置：目录 README、lint、测试、PR 模板和 review 清单。这样做以后，规则不再依赖某个人记得，而是依赖结构本身。

这也是为什么上面的例子都要先拆成边界，再配反例，再写验证，最后写回。这让同一条规则在不同环节都能被同一批人和工具读到。你要保住的“下次能拦住”。

最小回写句式

  1. 这条规则在阻止什么。
  2. 它应该被谁先看到。
  3. 它会放到哪个最近入口。
句子写回位下次谁会读到
这个目录只读权限结果，不生成权限判断README进入目录的人和 Agent
service 不依赖 UIlint / 测试合并前的检查流程
必须附验证命令和截图PR 模板审查人和复查人

深潜链：一个支付目录的完整回路

deep-dive

把上面的判断再往前推一步，就会得到一个完整回路。支付目录先收到 review 争议，再把争议拆成边界、不变量、验证和回写四件事；然后把每一件事放到最近的位置，保证下一次出现时能直接读到。这个回路之所以重要，是因为它让“口头提醒”变成了“固定结构”。

如果目录边界没写清，Agent 就会自己补权限判断；如果 lint 没拦住，跨层依赖会继续发散；如果 PR 模板没补证据，复查会一直来回问；如果 review checklist 里没写规则，老问题还会反复出现。问题这几条没有一起闭环。

这也是为什么你不能只盯着一处。目录 README 讲职责，lint 讲机械边界，模板讲证据，checklist 讲复审。四个地方一起工作，才能把工程品味从“感觉不对”收紧到“下次必拦”。

完整闭环的最小动作

  1. 从 review 争议里抽出边界句。
  2. 把边界句收成不变量。
  3. 把不变量送进 lint、测试或模板。
  4. 把最小反例写回最近入口。
环节输入输出失败后果

收尾故事：规则真正落地时长什么样

closing

真正把工程品味写成约束，把一条争议拆成几层。先看这条争议是不是会反复出现，再看它能不能被机械检查，再看它应该靠近哪个入口，最后看它会不会被下一位维护者直接读到。这个过程为了让规则在真实工作里能活下来。

如果一个目录总被问权限来源，说明它的责任边界没有读清。此时你不需要把所有架构理论拉进来，只要把“这个目录只读权限结果，不生成判断”写到最近的 README，把跨层引用写进 lint，把验证命令和截图写进 PR 模板，再把重复问到的问题写进 checklist。这样四个位置一起工作，下一次争议就不必重新从头开始。

这条链路的结尾“新人和 Agent 进来时会被同一条规则拦住”。真正稳定的约束，就是能被读、能被查、能被复用、还能在下次改动前先挡一下。

如果你要自己复现这一页的做法

  1. 抓一条会反复出现的争议。
  2. 写成一句可以机械检查的不变量。
  3. 把它放回最近的入口，而不是堆进大文档。
  4. 补一个最小反例和一个最小验证。
争议类型先写什么再写什么最后写什么
权限来源目录边界lint / 测试PR 模板和 checklist
依赖方向不变量架构测试例子和反例
交付证据模板字段验证命令截图和日志要求

终局案例：从一条评论到一套规则

final-case

再把视角收紧一点。假设你在支付目录里第一次收到评论：“这里像是自己在算权限。”这个句子没有告诉你具体哪一行错了，但它已经足够说明一件事：目录的责任边界没有被读者稳定识别出来。下一步不是辩解，而是去找一个最小的可检查对象。

你可以把它收成两层。第一层是目录边界：这个目录只能读权限结果，不能自己发起权限判断。第二层是验证方式：如果出现新来源，lint 或架构测试必须先报错。第三层是回写位置：README 里要写明该目录的职责、禁区和验证命令。这样做以后，评论就不再只是感觉，而是变成能执行、能复查的规则。

这条链路里最重要的地方，是你把“觉得不稳”拆成了几个具体动作。每一个动作都比前一个更强一点：评论只是提醒，不变量可以被读，lint 可以拦错，README 可以回写。规则真正稳定下来，靠的这几层一起工作。

把评论收紧时的顺序

  1. 先找这条评论真正想阻止什么。
  2. 再写成一句边界明确的不变量。
  3. 然后决定它是否能被工具拦住。
  4. 最后把它写回离问题最近的位置。
评论层对应动作对应位置失效症状
“这里像是自己在算权限”识别目录边界README读者不知道这目录在管什么
“跨层依赖看起来不稳”检查依赖方向lint / 测试任务会继续越层碰撞
“交付证据不够”补验证和截图PR 模板复查只能靠口头补充
“下次还会问同一句”写成 checklistreview 清单同样的问题重复出现

主线：把品味拆成可执行对象

map

这页的起点很简单：人看到一段代码时，会觉得“这里不对劲”。Agent 不会自动理解这种不对劲，所以要把判断拆成它能读、能检查、能拒绝的对象。

这一页只关心四种对象：不变量、反例、测试和审查标准。先有判断，再选载体，最后写回到最近的位置。这样一条偏好才会变成可执行规则。

学习路线

先看什么是判断，再看什么能变成规则，接着看规则怎样落到 lint、测试和 review，最后看怎样把一次失败写回仓库。

顺序你先看什么要回答什么下一步动作
1品味判断哪里会让维护变难把判断缩成一句话
2不变量和反例什么不能破，破了会怎样写成可检查规则
3测试和审查标准怎样机械拦住错误把规则放进工具和模板
4写回位置同类错误下次在哪里读到补最近的目录或模板

示例：把“不要乱写权限”改成规则

example

假设一个模块需要处理账户操作。你本来会说：“权限要谨慎一点。”这句话有方向，但对 Agent 没有约束力。它不知道什么叫谨慎，也不知道哪些地方算越界。

现在换成一条不变量：这个目录只能读取权限结果，不能自己生成权限判断；所有 allow / deny 必须来自统一 policy 层。这样写以后，Agent 至少知道两件事：哪里不能自己发明判断，哪里需要去找真正的决策源。

再往前一步，把这条规则写进 lint 或架构测试，错误就会在合并前被拦住。最后再在 review checklist 里补一句“是否有目录越权、是否有新的权限来源”，人和工具就会检查同一件事。

这个例子里有哪些可检查对象

  • 目录边界：模块能不能自己做权限判断。
  • 规则来源：allow / deny 是否都来自 policy 层。
  • 验证方式：lint、架构测试、review checklist。

写回桥：把一条规则放回最近的位置

bridge

写回的原则只有一个：让下一个遇到同类问题的人，能在离问题最近的地方读到它。目录级边界写到目录 README，机械能检查的规则写到 lint 或测试，交付时需要看的东西写到 PR 模板。

不要把所有内容都堆进一份大文档。那样做看上去很完整，实际上下一次还是要从头搜索。

最小写回顺序

  1. 先说清规则是什么。
  2. 再说它应该落到哪里。
  3. 最后说它会阻止什么错。

写回最小清单复制

1. 目录边界：写进最近的目录 README。
2. 机械规则：写进 lint 或测试。
3. 交付证据：写进 PR 模板。
4. 复审关注点：写进 review checklist。

边界：这一页先追到哪里

boundary

这一页不负责讨论所有工程美学，也不负责把每一种代码风格都升格成硬规则。它只处理那些会长期伤害维护和协作的部分：依赖方向、权限来源、目录职责、证据字段和审查标准。

如果你已经在想“哪些东西可以只留作建议”，那就是下一层的问题。先把必须硬化的部分守住，再谈保留余地。

  • 代码风格统一。
  • CI 平台选型。
  • 形式化证明。
  • 框架内部实现细节。

易混概念对照

compare
概念组容易混的地方区分方式工程判断
品味 / 规则都影响代码形状。品味是判断来源，规则是可传递表达。能表达清楚的品味尽量规则化。
建议 / 不变量都可能写在文档里。建议可权衡，不变量违反会破坏系统。不变量要少而硬。
反例 / 测试都能暴露错误。反例用于理解边界，测试用于自动阻止。先用反例讲清，再看能否自动化。
审查 / CI都守质量。CI 守机械底线，审查看设计权衡。不要让人类重复做 CI 的事。

落地流程：把判断变成约束

workflow

先看一个很常见的场景：支付模块 review 时，你不断听到“这里不稳”“这里像是越权”“这里会不会散”。这三句话都在指向同一类问题，但它们还不是约束。你真正要做的，是把这些判断拆成几个可检查的对象：依赖方向、权限来源、目录职责、验证命令和回写位置。

拆完以后，再决定每个对象写到哪里。能机械检查的，放进 lint、测试或架构校验；会被人反复问到的，放进目录 README 或 PR 模板；只用于理解边界的，放进反例说明。这样做的目的很直接：同类错误下次不用重新解释，工具和人会同时看到同一条线。

如果你只把它写成一段长文，下一次还是得搜索；如果你把它写进最近的入口，Agent 就会在执行时撞到它。约束的价值，不在于说得更重，而在于读得到、查得到、拦得住。

最小落地顺序

  1. 先抓信号，确认它不是一次性的吐槽。
  2. 再把信号缩成一句规则，别让它继续漂着。
  3. 然后选落点，优先选离问题最近的地方。
  4. 最后补验证和反例，让规则真的能工作。

约束写法模板复制

判断：
1. 先把模糊感受说成可检查对象。
2. 只保留会长期伤害维护的部分。
3. 给出一个最小反例。
4. 指明最近的写回位置。
5. 补上一个能在合并前运行的检查。

$$

\text{约束强度} \approx \text{可检查性} + \text{就近性} + \text{可复用性}

$$

一条规则越容易被读到、越容易被验证、越容易复用，下一次越不需要再解释同一件事。

失败回写：一次 review 争议怎样变成规则

repair

再看一个更接近真实工作的例子。某个支付目录每次改动都会被追问同一句话：“权限从哪来？”这说明问题不在单次实现，而在规则没有写清。你如果每次都靠口头解释，Agent 和人都只是在临时补洞；下次换一个人、换一个任务，洞还是会出现。

正确的做法是把争议转成稳定对象：目录边界、规则来源、验证方式、回写位置。这样一来，review 看到的“这里的权限只能来自 policy 层；如果出现新来源，lint 和架构测试会先拦住”。

这一步的关键是少而硬。规则不是越多越好，规则多了以后，真正影响维护的部分反而会被噪声盖掉。你要保留的是能长期阻止返工的那部分，不是把所有审美都变成硬限制。

把争议变成规则时先问三件事

  • 它会反复让谁多花时间？
  • 它能不能在合并前被检查？
  • 下次遇到同类问题，最先应该在哪读到它？

别写得太散service 不依赖 UI，跨层引用视为违规lint / 架构测试依赖方向继续发散

回写最小清单复制

1. 找到重复争议。
2. 写成一句不变量。
3. 选择最近的写回位。
4. 加上一个最小验证。
5. 保留一个能说明边界的反例。

规则集：四种写回位分别负责什么

ruleset
写回位它负责什么适合放什么句子不适合放什么
目录 README写清职责、禁区和最近的入口目录边界、依赖方向、常用验证命令抽象的大道理和全仓库总则
lint / 测试拦住能机械判断的错误跨层引用、缺少字段、违反结构的实现审美偏好和需要讨论的权衡
review checklist帮人重复检查最容易漏的点权限来源、证据字段、回归验证、反例每次任务都不同的临时思路
PR 模板把交付证据固定下来截图、命令、结果、风险说明实现细节和冗长复述

这四个位置不是互相替代的。目录 README 负责最近的职责边界，lint 和测试负责机械刹车，review checklist 负责重复关注点，PR 模板负责证据。把它们混在一起，结果就是每次都在错误的位置读规则。

真正稳定的写法是少而硬。你只保留足够拦住返工的句子，别把每个审美都写成红线。这样 Agent 在执行时才知道什么不能碰，人类在审查时才知道什么必须拦。

判断顺序

  1. 先问这条规则是不是机械可查。
  2. 再问它应该靠近哪个入口。
  3. 最后问它会阻止哪一种重复错误。

案例链：支付目录怎样把争议变成规则

casebook

假设一个支付目录连续三次被 review 卡住。第一次，审查人说“这里像是自己在算权限”；第二次，审查人说“跨层依赖看起来不稳”；第三次，审查人说“交付证据不够”。这三个反馈看上去不是一件事，其实都指向同一件事：目录边界没有被写成稳定规则。

这时候不要继续写解释信。你应该先把争议拆开：权限来源写到哪个层，跨层依赖要不要机械拦，交付证据写到哪里，验收命令放哪里。拆的时候不要追求面面俱到，只保留会长期让人返工的部分。真正该写进规则的，是以后还能复用、能被读到、能被检查的那几条。

然后再把它们放回最近的位置。目录 README 负责讲边界，lint 负责拦跨层引用，PR 模板负责收证据，review checklist 负责把“再看一眼”变成固定动作。这个顺序的意义很简单：下次同类问题出现时，读的人不用重新问一遍，Agent 也不会再把它当成自由发挥的空间。

这类争议最容易漏掉的东西

  • 争议到底是边界问题，还是实现选择问题。
  • 它是否已经重复出现，足够升格成规则。
  • 它应该被谁先看到，目录还是工具还是模板。
  • 它会不会在别的目录再次发生。

目录边界：
1. 这个目录只读权限结果，不生成权限判断。
2. service 层不能直接依赖 UI 层。
3. 交付时必须附验证命令和截图。
4. 新争议出现时，先写回 README，再补 lint 或 checklist。
