把工程品味变成可执行约束

OpenAI 原文里有一个很关键的判断：在 Agent-first 的世界，人类工程师仍然需要品味。区别在于，品味不能只留在脑子里。你要把它转成 Agent 可以遵守的架构约束、示例、审查标准和不变量。

这一页讲怎样把“代码要好”拆成可执行对象：依赖方向、状态归属、错误处理、命名、测试边界、性能预算、人工确认。Agent 可以机械执行很多任务，但它需要清楚哪些形状永远不能破坏。

品味判断→不变量→边界→反例→测试→审查→规则更新

学完后要能做什么

  1. 解释品味和可执行约束之间的关系。
  2. 把一条抽象架构偏好改写成 Agent 可遵守的不变量。
  3. 设计审查标准，区分必须阻止的问题和可以接受的风格差异。
  4. 诊断为什么“让代码优雅点”不能稳定指导 Agent。

学完本块后，你应该能把一条主观工程偏好转成规则、示例或测试。

概念骨架

MODEL

工程品味的直觉是：知道什么代码以后会让人痛苦。比如临时状态塞到全局、跨层依赖、重复业务规则、错误被吞掉、测试只覆盖快乐路径。人类资深工程师能凭经验识别这些问题，但 Agent 需要更明确的信号。

可执行约束是把品味落地后的形态。它可以是一条依赖规则、一个测试、一个模板、一个反例、一个 PR checklist。不是所有品味都能立刻自动化，但很多都能先写成不变量：永远不要让 UI 层直接访问数据库，所有外部动作必须可审计。

判断句是：如果你无法说明一条品味被违反时会造成什么后果，就很难让 Agent 稳定遵守它。

本块主线

  • 先把品味翻译成会影响长期维护的风险。
  • 再把风险写成不变量：必须保持什么，禁止破坏什么。
  • 能机械判断的变成测试或 lint，不能机械判断的变成示例和审查清单。
  • 审查发现新风险后，把它补回约束系统。

讲义主线

LESSON

下面的小节按“问题、做法、判断”推进。每节最后都有一句检查点，用来确认你不是只记住了标题。

1. 品味要从感觉变成可讨论的风险

TASTE

“这段代码不好”对人类同事可能有交流价值，对 Agent 帮助很小。它不知道不好在哪里。是命名含糊、状态放错层、错误处理不完整，还是未来扩展会重复？品味要先翻译成风险。

风险说清后，下一步才是约束。比如“这个写法会让权限逻辑分散”，可以转成“所有权限判断只能出现在 policy 层”；“这个组件太懂业务流程”，可以转成“组件只接收 view model，不直接调用 service”。

品味外化不是降低标准，而是把标准变成团队和 Agent 都能复用的语言。

读完这一节，你应该能把一句“代码不优雅”改写成具体风险。

2. 不变量保护系统长期形状

INVARIANT

不变量是系统变化时必须保持的条件。它比普通建议更硬。比如“所有写操作必须经过审计日志”“后台页面不能绕过权限中间件”“服务层不能依赖 UI”。这些条件一旦破坏，后续功能会越来越难维护。

Agent 很适合执行局部任务，但它未必知道全局形状。把不变量写清楚，可以防止它为了完成当前任务破坏长期结构。尤其是当任务很多、PR 很多时，不变量就是系统不散架的骨架。

不变量要少而硬。太多会变成噪声；太软会变成建议。真正的不变量应该能回答“违反后会造成什么具体后果”。

读完这一节，你应该能为一个项目写出 3 条少而硬的不变量。

3. 反例让 Agent 知道边界在哪里

NEGATIVE

只给正确示例，有时不够。Agent 可能照着正确示例写出相似但越界的版本。反例能说明边界：这个写法为什么错，后果是什么，应该改到哪里。

反例尤其适合表达架构品味。比如跨层导入、吞错误、在 UI 里拼 SQL、在测试里 mock 掉真正要验证的对象。这些错误不一定语法失败，但长期会伤系统。

一个好反例包含三部分：错法、后果、修法。只骂错法没有用，Agent 需要知道替代路径。

读完这一节，你应该能为一条架构规则写出一个反例。

4. 能机械判断的约束应该自动化

AUTO

有些约束可以交给工具：禁止跨层导入、测试必须带某个标签、API 返回必须有错误码、迁移文件不能改历史记录。这些不应该每次靠人类审查。

自动化不是只写测试。Lint、类型检查、脚本、CI gate、schema 校验都可以成为约束。原则是：如果一个规则能被程序稳定判断，就不要让人类在十几个 PR 里重复看。

但不要假装所有品味都能自动化。复杂设计权衡仍要人类判断。自动化负责守底线，人类负责看方向和例外。

硬规则

用 lint、CI 或测试阻止。

半结构规则

用模板和 checklist 提醒。

复杂判断

保留人类审查，但写清判断维度。

读完这一节，你应该能判断一条规则该自动化还是放进审查清单。

5. 审查标准要看路径，不只看结果

REVIEW

Agent 生成的代码可能通过测试，但路径仍然有问题：绕过权限、复制业务逻辑、扩大 diff、引入隐性状态。审查标准要覆盖这些路径风险。否则系统会在绿色测试下慢慢变乱。

一个适合 Agent PR 的审查清单会问：是否遵守目录边界，是否复用已有能力，是否附上相关证据，是否扩大任务范围，是否破坏不变量，是否留下新的文档或测试缺口。

审查不是挑刺。它是把人类品味重新注入 harness 的时刻。每次审查如果发现重复问题，就该更新规则。

读完这一节，你应该能为 Agent PR 写一份不只看测试的审查清单。

6. 完整链路：品味到规则，再回到品味

CHAIN

把这一页串起来：人类先把品味说成风险；风险变成不变量、反例、模板和测试；Agent 在这些约束下执行；PR 审查检查是否破坏长期形状；新发现的问题再沉淀成规则。

这条链路让品味不再只依赖某个资深工程师在线。它变成仓库的一部分，能被多个 Agent 和多个人类审查者共享。

以后当你想对 Agent 说“写得优雅一点”时，先停一下。问自己：优雅具体避免什么风险？有没有示例？能不能测试？如果能回答，就把它写进 harness。

读完这一节，你应该能把抽象品味转成 Agent 可执行的约束闭环。

最小例子：把概念落到工程动作

EXAMPLES

例子 1：把品味改成不变量

“不要写乱”太抽象。下面把它改成可检查规则。

正确示例：架构不变量Markdown
不变量：权限判断只存在于 policy 层。

禁止：
- route 里直接判断 user.role
- component 里根据权限隐藏真实数据访问

原因：权限分散会导致审计困难和绕过风险。

验证：
- policy 单元测试覆盖 allow/deny
- route 测试只断言调用 policy
- lint 禁止 src/routes 导入 src/auth/roles

这条规则能被 Agent 理解，也能部分自动化。

例子 2：反例写法

反例要说清错因和修法。

错误示例：跨层权限判断TypeScript
export async function updateInvoice(req) {
  if (req.user.role !== 'admin') {
    throw new Error('forbidden');
  }
  return invoiceService.update(req.body);
}

问题不在语法，而在权限规则散落到 route。修法是调用 policy 层，让权限逻辑集中可测。

例子 3：审查清单

Agent PR 的审查要看证据和不变量。

正确示例：PR 审查清单Markdown
- [ ] diff 只覆盖任务范围
- [ ] 没有跨越目录边界
- [ ] 复用了已有 service/template
- [ ] 新行为有测试或截图证据
- [ ] 没有破坏权限、日志、错误处理不变量
- [ ] 新反馈已补到文档或规则

清单把人类审查的注意力放到长期风险上。

常见坑：错因、后果和修法

PITFALLS
误区错因后果修法
用形容词指导 Agent说优雅、干净、合理，但不解释风险。Agent 按默认风格发挥，结果不稳。改成不变量、示例和反例。
不变量太多把所有偏好都升格成硬规则。Agent 被噪声淹没，人类也难维护。只保留会造成长期风险的硬约束。
只给正例没有展示边界外的错法。Agent 学到形状但不知道禁区。补反例：错因、后果、修法。
能自动化却靠审查没有把机械规则写进工具。人类重复检查，漏检概率高。用 lint、测试、schema 或 CI gate。
只看测试绿忽略实现路径风险。系统结构在绿色测试下劣化。审查目录边界、复用、权限和 diff 范围。

易混概念对照

COMPARE
概念组容易混的地方区分方式工程判断
品味 / 规则都影响代码形状。品味是判断来源，规则是可传递表达。能表达清楚的品味尽量规则化。
建议 / 不变量都可能写在文档里。建议可权衡，不变量违反会破坏系统。不变量要少而硬。
反例 / 测试都能暴露错误。反例用于理解边界，测试用于自动阻止。先用反例讲清，再看能否自动化。
审查 / CI都守质量。CI 守机械底线，审查看设计权衡。不要让人类重复做 CI 的事。
