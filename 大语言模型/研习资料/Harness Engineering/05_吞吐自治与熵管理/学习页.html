高吞吐之后，要管理熵

OpenAI 原文最后讨论的是更长期的变化：当智能体能夜以继日地产出分支、修 bug、加功能，工程组织会得到更高吞吐。但吞吐不是免费午餐。代码、任务、文档、测试和决策记录都会产生熵。

这一页讲怎样在 Agent-first 开发里管理熵：限制并行范围、保持小 PR、定期清理、合并重复能力、更新文档、让审查回路收敛。Agent 可以制造很多产物，人类要负责让系统仍然像一个整体。

并行任务→小PR→证据队列→审查→合并→清理→规则更新

学完后要能做什么

  1. 解释高吞吐为什么会带来系统熵。
  2. 设计多个 Agent 并行工作时的队列、范围和审查规则。
  3. 识别重复能力、过期文档、测试噪声和未收敛分支。
  4. 制定清理循环，让产出回到稳定系统。

学完本块后，你应该能为一个 Agent 团队设计“产出、审查、清理、沉淀”的节奏。

概念骨架

MODEL

熵的直觉是系统越来越散。多个 Agent 同时改代码，可能产生重复实现、相似但不一致的工具、过期文档、互相冲突的迁移、测试噪声和难以审查的大 diff。每个改动单看都合理，合起来系统变乱。

高吞吐工程需要两个方向同时存在：让 Agent 快速尝试，也让系统定期收敛。收敛包括合并重复能力、删除临时脚手架、更新文档、压缩任务队列、关闭过期分支、把经验转成规则。

判断句是：如果团队只统计 Agent 产出了多少 PR，却不统计多少被合并、多少被清理、多少规则被沉淀，那只是增加库存，不是提高工程能力。

本块主线

  • 吞吐提升会增加并行分支和审查压力。
  • 并行必须配范围控制、小 PR 和证据要求。
  • 系统会产生熵：重复、冲突、过期、噪声。
  • 清理循环把产出收敛成稳定能力和规则。

术语表：先把词放稳

TERMS
术语直觉解释准确边界常见误解
吞吐单位时间内完成并验证的工程任务量。不是生成代码行数，而是可合并成果。把 PR 数量当成唯一指标。
自治Agent 在边界内自主执行多步任务。需要停止条件、权限和证据。理解成无需人类审查。
熵系统结构和信息逐渐变乱。表现为重复、冲突、过期和难追踪。只把熵理解成代码格式问题。
收敛把分散产物合并成稳定系统。包括合并、删除、重构、文档更新和规则沉淀。只要 PR 合并就算收敛。
清理循环定期处理熵的工作节奏。查看未合并分支、重复能力、过期文档和失败测试。等系统乱了再大重构。

讲义主线

LESSON

下面的小节按“问题、做法、判断”推进。每节最后都有一句检查点，用来确认你不是只记住了标题。

1. 吞吐不是生成速度，而是可合并成果

THROUGHPUT

Agent 可以快速生成代码，但工程吞吐不能只看产出量。真正有价值的是经过验证、审查、合并并能长期维护的成果。如果十个 PR 里八个无法合并，生成速度反而变成审查负担。

所以吞吐指标要包括合并率、返工率、平均审查时间、失败原因、证据完整度和后续缺陷。只看代码行数会鼓励 Agent 写得更多，而不是写得更可用。

高吞吐的目标是让更多正确改动进入系统，不是让仓库里堆满未收敛分支。

读完这一节，你应该能区分生成速度和工程吞吐。

2. 并行任务要控制碰撞面

PARALLEL

多个 Agent 并行时，最怕改同一片区域。一个改 schema，一个改接口，一个改前端状态，最后冲突不只在 Git 层，还在设计层。任务分配要看碰撞面：文件、数据模型、接口契约、测试环境和产品决策。

控制碰撞面的方式包括：任务按能力块划分，标注相关目录，避免多个 Agent 同时改核心抽象，给每个任务独立工作树，要求 PR 小而有证据。越核心的模块，并行越要谨慎。

并行不是越多越好。没有审查和清理能力支撑的并行，会把瓶颈从实现转移到合并。

读完这一节，你应该能为多个 Agent 任务判断是否会互相碰撞。

3. 熵通常来自看起来合理的小局部

ENTROPY

系统变乱很少是因为某个 PR 明显灾难。更常见的是局部合理：为当前功能复制一个小工具，为当前测试加一个特殊 mock，为当前 bug 写一个临时分支。Agent 很擅长完成局部目标，也容易积累局部例外。

熵的信号包括：同类工具出现多个版本、文档和代码说法不一致、测试命令越来越多但没人知道该跑哪个、PR 描述里的临时说明没有沉淀、目录边界被小例外打穿。

识别熵要看系统整体，而不是只看单个任务是否完成。

重复

多个工具做相似事，但命名和行为不同。

过期

文档、模板或测试仍指向旧流程。

冲突

多个分支对同一契约做不同假设。

读完这一节，你应该能列出 Agent 高吞吐后最常见的熵信号。

4. 清理循环要固定发生，不等到大崩盘

CLEAN

清理循环是高吞吐开发的卫生机制。它可以每天或每周运行：查看未合并分支，关闭过期任务，合并重复能力，删除临时文件，更新入口文档，补测试缺口，把审查反馈变成规则。

清理不是低价值杂活。没有清理，Agent 的速度会不断制造库存。库存越多，审查越慢，冲突越多，最后团队会对 Agent 失去信任。

清理循环要有产物。比如一个合并后的工具、一个删除的旧模板、一条新增 lint、一份更新后的目录说明。只开会讨论不算清理。

读完这一节，你应该能设计一份高吞吐 Agent 项目的每周清理清单。

5. 自治越高，边界越要硬

AUTONOMY

让 Agent 自主跑更久，并不意味着给它更多自由。自治越高，越需要硬边界：最大步数、预算、权限、可修改目录、必须验证的命令、高风险动作人工确认。否则一个错误方向会被放大很多步。

自治任务适合明确、可验证、可回滚的工作。比如补测试、迁移一类 API 调用、修复明确错误、生成文档索引。不适合把产品方向、权限策略、数据删除这类高风险判断完全交给 Agent。

边界清楚时，自治可以提高吞吐；边界含糊时，自治只会提高事故速度。

读完这一节，你应该能判断一个任务适不适合让 Agent 长时间自治执行。

6. 完整链路：产出之后必须收敛

CHAIN

把这一页串起来：任务队列按能力块分配；多个 Agent 在隔离环境里执行；每个 PR 小而有证据；人类按不变量和证据审查；合并后进入清理循环；重复能力、过期文档和新规则被统一处理。

这条链路让高吞吐不至于变成高混乱。Agent 的优势是快速执行，人类和 harness 的职责是让执行结果不断收敛成稳定系统。

以后评估 Agent 工作流时，不要只问“它能不能写”。还要问“写完后如何收敛”。没有收敛机制，吞吐会变成熵。

读完这一节，你应该能完整解释 Agent 高吞吐工作流为什么必须有清理和收敛。

例子 2：清理循环清单

清理要产出实际变更。

正确示例：每周熵清理Markdown
- [ ] 关闭 7 天未更新且无证据的 Agent 分支

清理循环的目标是收敛，不是整理心情。

常见坑：错因、后果和修法

PITFALLS
误区错因后果修法
把 PR 数当吞吐只看产出，不看合并和质量。库存上升，审查压力变大。看合并率、返工率、证据完整度和缺陷。
并行不看碰撞面多个 Agent 改同一契约。冲突和设计分歧增多。按目录、数据模型和接口契约标注风险。
不清理临时产物任务完成后没有收敛。临时工具和文档长期污染仓库。固定清理循环，删除或合并临时内容。
自治边界太软给 Agent 长时间运行但无预算和权限限制。错误方向被放大。设置步数、预算、目录、工具和人工确认。
反馈不进入系统审查意见只留在 PR 评论。同类问题在并行任务里重复出现。把重复反馈写入规则、模板或测试。

易混概念对照

COMPARE
概念组容易混的地方区分方式工程判断
生成速度 / 工程吞吐都和速度有关。生成速度看产出，工程吞吐看可合并成果。不要用代码行数衡量成功。
并行 / 碰撞并行提高速度，碰撞增加返工。好的任务划分减少碰撞面。核心契约变更要降低并行度。
自治 / 放任都减少人工介入。自治有边界和证据，放任没有控制。越自治越需要硬护栏。
清理 / 重构都能改善系统。清理是固定卫生循环，重构通常是较大结构调整。不要等到必须大重构才处理熵。
