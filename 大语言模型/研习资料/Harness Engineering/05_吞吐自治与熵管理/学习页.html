吞吐自治与熵管理

这一页讲的是：当智能体已经能连续做任务时，工程系统怎样避免越做越散。重点不是追求更大的产出数字，而是让并行、自治和清理形成一条能收敛的链路，让产物继续变成稳定能力。

收尾链：把一周的结果落到下一周

summary

如果一周的结果没有被写回下一周，那它就只是一段已经结束的忙碌。高吞吐周最怕的是“忙完就散”，所以你要把结果拆成四类：进主线的、需要降并行的、必须清理的、必须写回的。只有拆出来以后，下一周的队列才会变得更干净。

进主线的东西告诉你哪条路径是真正有效的；需要降并行的东西告诉你哪些碰撞面太大；必须清理的东西告诉你哪些熵已经不能再拖；必须写回的东西告诉你哪些争议已经够格变成规则。四类信息一起看，才能让下周站在上周的结果上继续收紧。

这条收尾链最后想做的事情很朴素：让人和 Agent 下次进来时，先读到更好的入口，而不是读到更长的乱账。

下一周先看什么

  1. 先看队列有没有高碰撞任务。
  2. 再看台账有没有补足证据。
  3. 然后看清理循环有没有按时发生。
  4. 最后看规则有没有写回最近入口。
周复盘结论下一周应该怎么接放到哪里不接会怎样
有些任务真进了主线保持这条路径任务队列下一周重新试错
有些任务碰撞太大降低并行度队列规则冲突继续扩散
有些东西必须清掉固定清理节奏清理循环熵会继续升高
有些争议已经反复出现写回最近入口README / checklist / 模板同类问题再次复现

最后一公里：把复盘写回下一周

last-mile

复盘如果只停在“这周很忙”，它就没有价值。你要把周复盘写成下一周的输入：哪些任务要降并行，哪些任务要分离契约，哪些重复 helper 要合并，哪些过期分支要关掉，哪些 review 反馈要写回模板。这样下一周从一个已经收紧过的队列开始。

把复盘写回去的顺序也要固定。先写队列规则，再写台账字段，再写清理节奏，最后把重复出现的争议写进 README、checklist 或模板。写回之后，下一次 Agent 和人类就会读到同一套约束，不会再每轮都重新发明一次流程。

高吞吐真正怕的不是忙，而是忙完之后没人把东西收回来。只要复盘链不断，系统就会一直朝着更稳的方向收敛。

下一周的输入清单

  1. 把高碰撞任务单独排队。
  2. 给台账补上证据字段。
  3. 把重复 helper 和过期分支写进清理清单。
  4. 把重复争议写回最近入口。
上周发现下一周输入放到哪里结果
高碰撞任务挤在一起收紧并行度任务队列冲突变少
证据经常漏补验证字段PR 模板复查更快

同一句 review 反复出现写回 checklistreview 清单下次不会再白问一遍

台账复盘：一周高吞吐后，怎么查哪里撞车

ledger-review

如果一周结束后你只看见很多 PR，却看不出哪里出了问题，那说明台账还不够细。台账的作用让你能回头找出：哪个任务碰了同一块文件，哪个任务的证据还没补齐，哪个任务其实该降并行，哪个任务留下了新的重复 helper。只要这几项能写清，后面的复盘就不需要凭记忆。

拿到台账后，先查高碰撞面任务。看它们是不是同时改了同一个契约、同一个目录或者同一条运行命令。然后查证据字段，看看是否每个任务都有验证命令、截图或日志。最后查回退和清理项，确认重复的东西有没有合并，过期的东西有没有关掉。如果这些地方都是空的，那这个周报就只是“忙”，不是“收住了”。

台账复盘真正想要的，把下一周最先该动的地方找出来。找到之后，就把它写回队列规则、清理循环或 PR 模板。这样复盘才不是回忆，而是下一轮执行的起点。

台账复盘的固定顺序

  1. 先查碰撞面最大的任务。
  2. 再查证据字段有没有补齐。
  3. 然后查合并和回退是不是正常。
  4. 最后把问题写回最近的入口。
台账项怎么查发现问题意味着什么写回哪里
碰撞面看改动文件和共享契约队列排法不对任务队列规则
证据字段看验证命令、截图和日志交付缺少可复查材料PR 模板
重复 helper看是否出现多份相似实现熵正在增长清理循环
过期分支看是否无人再改系统里有旧库存清理循环

学习目标

goal

这页要练的是：先看吞吐是不是健康，再看并行任务是不是撞车，最后看熵是不是被固定的清理循环拉回来了。

你不需要把每一次并行都做大，先学会把任务排队、把边界写硬、把清理动作固定下来。

读完后你应该能做什么

  • 区分真正的吞吐和只是库存变多。
  • 按碰撞面给并行任务收紧队列。
  • 给重复、过期和冲突安排清理回路。
能力对应动作观察信号
看吞吐统计合并成果和返工率不是只看 PR 数
排队列按碰撞面和风险排序高风险任务不会被乱并行
做清理固定清理循环重复、过期和冲突不会越积越多

主线：吞吐越高，越要管熵

map

Agent 产出越多，系统越容易出现重复实现、未合并分支、过期说明、冲突和审查噪声。表面上看是“更快了”，实际可能只是库存更多。

这一页要做的是把速度和收敛分开看：先让任务跑得动，再让结果收得回。没有收敛机制，吞吐只会把熵放大。

学习路线

先看吞吐怎么定义，再看并行怎么控碰撞面，接着看熵怎么积累，最后看清理、自治边界和收敛链路怎么接起来。

顺序你先看什么要回答什么下一步动作
1吞吐定义什么才算真的完成别再拿 PR 数当速度
2并行和队列哪些任务会撞在一起把碰撞面写出来
3熵来源哪些小动作会让系统变散找重复、过期和冲突
4清理和自治边界怎样把产出拉回稳定系统定清理循环和硬护栏

吞吐模型：看可合并成果，不看库存

throughput

工程吞吐不是代码写了多少，也不是 Agent 发了多少条消息。它更像“进入系统并站得住的成果”有多少：能合并、能复用、能长期维护的改动才算数。

只看生成速度会把人带偏。因为生成越快，审查、返工和冲突也可能越快，最后把团队拖进库存堆里。

先看哪几个数

  • 合并率：最后真正进入主线的比例。
  • 返工率：同类错误是否一直重来。
  • 审查时间：队列是不是已经堵住。
  • 证据完整度：PR 有没有可复核材料。
指标健康信号风险信号你该先查什么
合并率多数小 PR 在合理时间内合并PR 很多，但长期停在 review是不是任务边界太大
返工率重复失败原因逐周减少同类审查意见反复出现规则有没有写回模板或 lint
证据完整度PR 附测试、截图、日志或指标只有自然语言总结证据字段是不是缺了
库存年龄过期分支定期关闭或重排大量分支无人认领清理循环有没有跑

吞吐判断清单复制

1. 看进入主线的成果，不看生成的数量。
2. 看合并率，不看 PR 库存。
3. 看返工率，不看单次热闹。
4. 看证据完整度，不看口头完成感。

队列台账：先算碰撞面，再排并行

queue

多个 Agent 并行时，最怕改到同一片区域。真正要先看的碰撞面在哪里：文件、目录、接口契约、数据模型、测试环境和产品决策。

碰撞面越大，并行度越要收紧。核心契约变更不能像普通小任务那样乱排，否则 Git 冲突还没出现，设计冲突就先出现了。

任务碰撞面建议并行度验收证据
改文档说明低高diff + 目录说明
补单元测试中中测试命令 + 失败前后对比
改接口契约高低接口说明 + 回归测试
改 schema很高很低迁移脚本 + 回滚方案

熵来源：看起来合理的小局部

entropy

系统变乱通常很多局部都“挺合理”。为当前功能复制一个 helper、为当前测试加一个临时 mock、为当前 bug 开一个过渡分支，这些动作单看没错，合起来就把系统拖散了。

所以识别熵时，要看重复、过期、冲突和噪声，而不是只看单个任务有没有完成。

常见熵信号

  • 同类工具出现多个版本。
  • 文档和代码说法不一致。
  • 测试命令越来越多，但没人知道该跑哪个。
  • 临时说明没有沉淀回仓库。
熵信号通常怎么出现会造成什么先怎么修
重复多个工具做相似事，但命名和行为不同没人知道该用哪一个合并重复能力
过期文档、模板或测试还指向旧流程Agent 按旧路径走更新入口说明
冲突多个分支对同一契约做不同假设后续改动越来越难合收紧并行度
噪声审查意见堆在评论区，没有变成规则同类错误反复出现写回模板、lint 或清单

清理循环：固定发生，不等大崩盘

clean

清理循环高吞吐系统的卫生机制。它可以每周固定跑：关掉过期分支，合并重复 helper，更新入口文档，补测试缺口，把重复反馈变成规则。

如果不清理，Agent 的速度会不断制造库存。库存越多，审查越慢，冲突越多，最后团队会开始怀疑整个工作流。

清理循环要产出什么

  • 一个合并后的工具。
  • 一个删掉的旧模板。
  • 一条新增 lint 或测试。
  • 一份更新后的目录说明。
清理动作输入输出为什么要固定跑
关掉过期分支未更新且无证据的任务更短的队列避免库存继续膨胀
合并重复 helper三个相似实现一个稳定实现减少重复维护
更新入口文档旧命令、旧目录、旧说明新的可读入口下一次任务能直接接上
把反馈写回规则重复 review 意见模板、lint 或清单让同类错误少重来

自治边界：越自主，越要硬护栏

autonomy

让 Agent 自主跑更久，不等于给它更多自由。自治越高，越需要硬边界：最大步数、预算、权限、可修改目录、必须验证的命令、高风险动作的人类确认。

边界一软，错误方向会被连续放大。边界一硬，自治才会变成效率，而不是事故速度。

适合自治的工作

  • 补测试。
  • 迁移一类 API 调用。
  • 修复明确错误。
  • 生成文档索引。
维度允许自治需要人工确认
代码范围任务指定目录、小型测试、局部文档跨核心模块、公共 API、数据库 schema
验证运行既有测试、截图、日志查询、格式化跳过关键验证或改写验证标准
权限读取代码、创建分支、生成本地证据删除数据、改密钥、发布生产变更
时间和预算在限定轮次内修复明确失败连续多轮失败、开始大范围重构

完整链路：产出之后必须收敛

chain

把这一页串起来：任务队列按能力块分配；多个 Agent 在隔离环境里执行；每个 PR 小而有证据；人类按不变量和证据审查；合并后进入清理循环；重复能力、过期文档和新规则被统一处理。

这条链路让高吞吐不至于变成高混乱。Agent 的优势是快速执行，人类和 harness 的职责是让执行结果不断收敛成稳定系统。

环节输入输出如果缺了会怎样
队列分配任务和碰撞面可并行的任务组并行撞车
执行证据本地任务和验证命令可复核的结果“做完了”只能停在感觉上
审查收尾不变量和证据合并或回退队列越来越长
清理循环重复问题和过期产物规则、文档、删除和合并熵继续长大

lab

例子：一个高吞吐周应该长什么样

example

假设一个团队这周开了很多 Agent 任务。好的结果：文档改动有新的入口，测试改动能复跑，契约改动被单独排队，重复 helper 被合并，过期分支被关闭，重复 review 意见被写回模板。

如果看不到这些收敛动作，就说明吞吐只是把库存堆高了。真正健康的周报应该能讲清楚：哪些东西进了主线，哪些东西被清理掉，哪些规则被补上。

这周的清理产物

  • 一个合并后的 helper。
  • 一条新增 lint。
  • 一份更新过的目录 README。
  • 一组被关闭的过期分支。
版本说了什么还缺什么判断
只看 PR 数库存很多合并率、返工率、清理动作不能说明吞吐健康
看合并率开始进入主线清理产物和规则沉淀比纯计数更接近真实吞吐
看清理循环产物、规则和文档都在收敛周边还可以继续优化这才像可持续系统

写回桥：把重复变成规则

bridge

清理把重复出现的东西写回流程。最容易写回的是目录 README、任务模板、lint 和 review 清单，因为这些地方下次最容易被读到。

如果同一条 review 意见已经出现三次，就别再靠聊天提醒了。把它写回能让下一次任务直接读到的位置，才算真的收住。

最小写回顺序

  1. 先数重复出现了几次。
  2. 再找最接近任务入口的位置。
  3. 最后把它写成规则、模板或 lint。

每周清理清单复制

1. 关闭 7 天未更新且无证据的分支。
2. 合并重复 helper，删除旧版本。
3. 更新 README 中过期命令。
4. 把本周重复 review 反馈转成规则。
5. 标记高碰撞任务，下周单独排期。

边界：这一页先追到哪里

boundary

这一页不追多智能体编排的所有细节，也不追权限治理、模型评测或组织管理的大框架。它只回答一件事：当吞吐上来以后，系统怎么靠队列、边界、证据和清理把自己收住。

如果你已经开始想“怎样统一全部工作流”，那是下一层的问题。先把这一页的硬护栏定住，再谈更大范围的治理。

先别追的附近概念

  • 多智能体调度策略。
  • 组织级绩效指标。
  • 大规模权限框架。
  • 模型评测细节。
问题优先写到哪里为什么
这个任务很高风险，不能多人并行任务队列先控制碰撞面，再谈吞吐

周节奏：把清理排进固定循环

steady

如果一个团队这周产出很多，最容易犯的错就是“先忙完再说”。但吞吐系统里，清理要跟着节奏走。你可以把一周拆成三个固定动作：周初看队列，周中看冲突，周末做清理。这样一来，重复、过期和争议都不会拖到下周再堆一层。

周初看队列，主要是调整碰撞面。哪些任务可以并行，哪些必须单独排，哪些要放到同一条链路里，先在入口写清。周中看冲突，主要是确认台账：谁改了同一块地方，谁的证据还没补齐，谁的回退风险更高。周末做清理，主要是合并重复能力、关掉过期分支、更新旧命令、回写重复 review 意见。

这个节奏的价值在于，它把“看起来有空再做”的事情，变成了每周都要完成的固定工作。只要节奏在，熵就很难无限长大。只要节奏乱了，清理就会不断被推迟，下一周的队列就会更长，冲突也会更难看。

一周三拍

  1. 周初先收队列，标高碰撞面任务。
  2. 周中查台账，补证据，找冲突。
  3. 周末做清理，把重复和过期收回去。
时间点要看什么要做什么结果信号
周初任务队列和碰撞面降并行、分层排期高风险任务不会挤在一起
周中台账和证据补验证、查回退、看谁改了同一块问题能定位到具体任务
周末重复、过期、冲突合并、关闭、更新、回写系统重新收紧

周节奏清单复制

周初：
1. 收队列，标高碰撞任务。
2. 降低共享契约改动的并行度。

周中：
3. 查台账，确认每个任务的证据。
4. 找出同文件、同契约、同目录的冲突。

周末：
5. 合并重复 helper。
6. 关闭过期分支。
7. 更新入口文档和命令。
8. 回写重复 review 意见。

流程链：从排队到清理

pipeline

高吞吐不是一味让更多 Agent 同时开跑。先要做的是排队：把高风险任务和低风险任务分开，把碰撞面大的任务降并行，把会改同一契约的任务放在同一条队列里。这样做的目的是减少互相顶掉的概率，而不是单纯把总数做大。

排完队以后，还要看台账。台账不是装饰，它要写清每个任务读了什么、改了什么、留下了什么证据。只要台账能把“谁碰了同一块地方”写清，后面就更容易看见冲突是怎么出现的。没有台账，冲突只会变成一句“这周好像有点乱”。

最后才是清理。清理吞吐系统的一部分。它负责把重复 helper 合并、把过期分支关掉、把旧命令更新掉、把重复 review 意见写回模板。没有这一步，吞吐只会把更多垃圾送进系统。

排队三原则

  1. 高碰撞面任务单独排。
  2. 改同一契约的任务一起排。
  3. 可验证、可回滚的任务可以更并行。
环节读什么写什么出错会怎样
排队风险、碰撞面、共享契约优先级和并行度多个任务互相覆盖
台账任务范围、改动文件、证据谁碰了什么冲突只能事后猜
清理重复、过期、冲突、噪声删除、合并、更新、回写系统越做越散

清理链：让熵有固定出口

cleanup

清理循环要做的事情很具体。它先找重复：相似 helper、重复规则、重复说明、重复 review 意见。接着找过期：已经不用的分支、旧命令、旧文档、旧入口。最后找冲突：两个任务改同一文件、同一条契约、同一个流程。只有这三类东西被稳定处理，系统才不会越跑越乱。

很多人以为清理是“有空再做”，其实不对。高吞吐之后，清理是固定的节奏，不做它，熵会把后面的任务越拖越慢。你越想省掉清理，后面就越要花更多时间回头补。

每周清理优先级

  • 先关高风险、无证据的分支。
  • 再合并重复能力和规则。
  • 然后更新入口文档和命令。
  • 最后把重复 review 意见写回流程。
熵信号具体样子动作不做会怎样
重复 helper相似代码分布在多个文件合并成单一实现后面每次改动都要改多处
过期分支没人再改但还留着关闭或归档任务树越长越乱
重复 review 反馈同一句提醒总出现写回模板或 checklist每次都要重新解释

收敛：吞吐、自治和清理要一起看

convergence

真正能持续的系统，不是只有吞吐，也不是只有自治，更不是只有清理。吞吐负责把结果送出来，自治负责让 Agent 自己跑，清理负责把过程中的重复、过期和冲突收回去。三者必须同时存在，少一个都会变形。

如果只有吞吐，没有清理，系统会越来越散；如果只有自治，没有边界，错误会被连续放大；如果只有清理，没有吞吐，系统又会变成慢动作。收敛让每一次扩张都能回到稳定状态。

所以这个单元最后想保住的，“更能活”。Agent 可以更快，但快完以后要能看见队列、看见台账、看见清理、看见回写。这样吞吐才不会变成熵的助推器。

收敛判断

  1. 是否能看出哪些任务真进了主线。
  2. 是否能看出哪些地方在互相碰撞。
  3. 是否有固定节奏处理重复和过期。
  4. 是否把争议写回了最近的入口。

$$

\text{可持续吞吐} \approx \text{交付} - \text{碰撞} - \text{重复} - \text{过期}

$$

这里判断方式：只要碰撞、重复或过期一直增加，所谓的吞吐就会被自己吃掉。

复盘链：把一周的高吞吐看完整

postmortem

真正有用的周复盘，不是简单统计“这周做了多少”。你要能把队列、台账、冲突、清理和回写连在一起看。比如某周看起来有很多 PR，但如果队列里高碰撞任务没有降并行，台账里没有写清改动范围，review 里总是追问同一句话，那这周其实只是把不稳定的东西快速扩散了。

复盘时先看入口：任务是不是被分到了合适的队列，是否有明确的碰撞面记录，是否有必须的验证证据。然后看出口：哪些东西进入了主线，哪些东西被清理掉，哪些争议被写回了 README、lint 或 checklist。最后再看节奏：周初有没有调队列，周中有没有补证据，周末有没有做清理。如果这个节奏断了，熵通常会在下一周更快地长出来。

这类复盘的重点给下一周一个具体起点。把问题写成规则、把规则放进最近的入口、把入口变成下次能直接读到的位置，复盘才算真的闭环。

复盘时先问的四个问题

  1. 这周真正进入主线的是什么。
  2. 哪些任务因为碰撞面太大而该降并行。
  3. 哪些重复和过期产物必须被清掉。
  4. 哪些争议应该写回最近的入口。
