学习目标

goal

先认链路

把字符串、token、token id、embedding 向量、带长度限制的输入序列放进同一条链路里看，不再把它们混成一个概念。

先会排错

遇到答非所问、截断异常、padding 污染或窗口不够时，先查输入链路，先别急着怀疑模型“没学会”。

先抓主线

overview

一条最小输入链

模型不会直接吃“句子”。它先把字符串切成 token，再把 token 查成 token id，接着用 embedding 表把 id 变成连续向量，最后把这些向量连同位置信息送进 Transformer。context window 则决定这次输入和生成总共能占多少 token。

这条链路里，token id 只是查表用的编号；embedding 向量才是真正进入计算的表示，语义关系也体现在向量空间里。另一个常见误会是把窗口当成“模型知识上限”，它实际限制的是这一轮能看见多少上下文。

建议你先记住的四个判断

  • token 数不能直接用字数代替。
  • id 大小不能代表语义远近。
  • embedding 先查表，后续上下文混合在 Transformer 里发生。
  • 窗口预算要把系统提示、用户问题、历史、检索文本和预留输出一起算进去。

核心机制

trace

01 原始文本

02 token

03 token id

04 embedding 向量

字符串先进入 tokenizer

这一步的读入对象是人类写下来的文本。输出是 token 片段和对应编号。这里先把句子切开，再决定每个片段在词表里的位置。

切分结果是 token

token 是 tokenizer 按词表和切分规则切出来的模型输入片段。它有时接近字，有时接近词，有时比词更细。切法会影响后面的长度预算和特殊标记位置。

编号只是查表钥匙

token id 是离散编号。它的作用是让程序在 embedding 表里定位到对应的一行，模型真正处理的是查表得到的向量。

向量才进入计算

embedding 把 [B, N] 变成 [B, N, D]。这里的 B 是 batch，N 是序列长度，D 是隐藏维度。后续层处理的是这组连续向量，而不是整数编号。

步骤前状态 / 形状后状态 / 形状读什么写什么错了会怎样

原始文本字符串待切分文本人类输入无如果文本没真正进入 tokenizer，后面所有检查都失效。

tokenizer字符串token 列表与 token id词表与切分规则切分结果、编号切法不对会导致长度预算、特殊标记和检索拼接都出错。

embedding[B, N][B, N, D]token id、embedding 表连续向量如果把 id 直接送进后续层，模型根本没有可计算的表示。

context window待拼接输入已截断 / 已预算输入系统提示、历史、检索、输出预算保留或丢弃的 token关键证据被挤掉时，模型只能在窗口内瞎猜。

为什么这个顺序重要

顺序错了，问题会变得很像“模型不会”。比如没算 token 数，就不知道为什么 prompt 一拼长就答偏；没做 embedding 形状检查，就不知道为什么张量维度不对；没看窗口预算，就不知道是证据没进来，还是模型真的没判断出来。

这里先不追的东西

先别把注意力放到 attention head 分工、位置编码细节、长上下文压缩、分块检索策略和微调格式上。它们都重要，但都建立在这条输入链路已经看懂的前提下。下一节会直接接到 Transformer 与 Attention；等你学到 RAG 和 Agent，再回来看这一节里的长度预算，会更容易判断证据该放哪里、该删哪里。

文本请解释 RAG人能读懂，模型不能直接计算

token请｜解释｜R｜AG切分结果，实际切法由 tokenizer 决定

一个最小窗口预算

$$\text{used} = s + h + r + o$$

其中 s 是系统提示，h 是历史对话，r 是检索文本，o 是预留输出。先算总占用，再和窗口上限比较，才知道该减哪一块。

检查 token 与长度复制

text = "请解释 RAG 和长上下文的区别。"
tokens = tokenizer.tokenize(text)
ids = tokenizer.encode(text, add_special_tokens=True)

print(tokens)
print(ids)
print(len(ids))

这段代码的检查重点只有一个：不要拿字数代替 token 数。长度一旦超预算，先查 tokenizer 结果，再查后面拼进来的历史和检索片段。

切分实测：同一段输入会怎么变

trace

中文、英文、代码放在一起看

真实工作里，你很少只处理一种文本。更多时候，是中文问题、英文术语和代码片段混在一起。tokenizer 会把它们拆成不同的片段形态：中文专业词可能被切成几个子词，英文长词可能被拆开，代码里的符号和空格也会变成切分信号。

这就是为什么“看起来字不多”的 prompt 仍然可能很长。只要混进了路径、JSON、函数名、报错和术语，token 数就会比肉眼预期高很多。对这种输入，最稳妥的习惯是直接打印 tokenizer 的结果，不要靠眼睛估 token 数。

一个最小切分脚本

看三种输入的 token 数复制

samples = {
 "中文": "请解释长上下文为什么会超窗。",
 "英文": "Explain why long context can overflow the window.",
 "代码": "tokenizer(prompt, truncation=True, return_tensors='pt')",
}
for label, text in samples.items():
 pieces = tokenizer.tokenize(text)
 ids = tokenizer.encode(text, add_special_tokens=True)
 print(label, len(pieces), len(ids))

这段脚本的目的，是让你看到三类输入的切分差异。通常你会发现：英文长词和代码片段的 token 数增长比直觉更快，中文专业词则可能被拆成更细的片段。

输入类型常见切分现象下一步检查
中文说明专业词、数字和标点可能各自占位看是否接近窗口边界
英文长句词根、词缀、空格模式会影响 token 数看是否和中文预算一致
代码与路径括号、引号、斜杠、下划线都会增加片段数看是否需要先摘出关键片段

这个实测的意义，是让你把“输入长短”改写成一个可以验证的动作。以后你看到混合型 prompt，就先跑一次 tokenizer，再决定是否要删历史、压缩检索，或者把代码和说明拆开。这样做的好处是，很多原来靠猜的判断都能落在一个明确的检查步骤上。

做完这个小实测后，再回到练习 1、2、5 和 17 复答一次。你会更容易说清楚：到底是哪里把 token 数推高了，为什么同样一段文本在不同模型里会有不同的预算压力，以及为什么“字数不多”并不等于“窗口很轻松”。

最小流程

pipeline

从输入到可检查张量

最小输入流程是：准备文本，调用 tokenizer，得到 input_ids 和 attention_mask，再送入模型。padding 负责补齐 batch 里短样本，attention mask 告诉模型哪些位置是真 token，哪些只是补齐。

一个稳妥的做法是把这四件事都做成检查动作：先打印 token 数，再看头尾片段，再确认 mask，最后检查实际送入模型的形状。这样你更容易判断问题出在 tokenizer、长度预算还是后续层。

如果你把这一步写进代码习惯，很多“模型奇怪”的问题会提前变成输入检查问题。比如一个 prompt 明明只有几百字，却在拼接检索和历史之后接近窗口上限；或者某个模型的特殊标记位置错了，结果看起来像“角色混乱”。这些问题在模型层里不容易看出来，在输入层里一眼就能看见。

错误和修法

静默截断示例复制

encoded = tokenizer(
 long_prompt,
 max_length=4096,
 truncation=True,
 return_tensors="pt"
)
answer = model.generate(**encoded)

这段代码能跑，但不够稳。它没有说明到底截掉了哪一段，也没有确认关键证据是否还在。对长文档任务，更可靠的做法是先统计长度、明确截断方向、再决定要不要分块和重排。

编码后验收复制

encoded = tokenizer(
 prompt,
 max_length=8192,
 padding="longest",
 truncation=True,
 return_tensors="pt"
)

print(encoded["input_ids"].shape)
print(encoded["attention_mask"].sum(dim=1))
print(tokenizer.decode(encoded["input_ids"][0][-80:]))

第一行看 batch 和序列长度，第二行看真实 token 数，第三行看尾部是否真的保留了你要的内容。删掉这三步，程序还在，判断就没了。

窗口预算演算台

lab

把预算拆成能看见的块

一段 prompt 变长以后，最难的是看不出是哪一块把窗口吃掉了。把系统提示、用户问题、历史对话、检索片段和预留输出拆开，是为了让你先看总账，再看每一笔明细。

下面这个小实验用三种场景来模拟：短句问答、长历史对话、RAG 拼接。每次切换场景时，先看总占用，再看剩余，再看最容易被挤掉的对象。这样你能把“答非所问”翻译成一个具体检查动作。

预算公式复制

总占用 = 系统提示 + 用户问题 + 历史对话 + 检索片段 + 预留输出
先看总占用，再看剩余额度，再决定削减哪一块。
短句长历史RAG 拼接
短句：系统提示 700 + 用户问题 120 + 预留输出 300 = 1120判断：窗口很宽松，重点是确认 tokenizer 的切分和特殊标记。短句：系统提示 700 + 用户问题 120 + 预留输出 300 = 1120
判断：窗口很宽松，重点是确认 tokenizer 的切分和特殊标记。长历史：系统提示 700 + 历史对话 8000 + 用户问题 180 + 预留输出 1200 = 10080判断：窗口已经明显吃紧。下一步先删旧轮次，再确认关键事实没有被挤掉。

RAG 拼接：系统提示 700 + 用户问题 140 + 8 段检索文本 8400 + 预留输出 1200 = 10440

判断：要先重排检索结果，保留最相关的少数段，再检查末尾证据有没有进入窗口。

场景最容易出问题的部分先做什么
短句问答特殊标记和基础切分看 tokenizer 输出和长度
长历史对话旧轮次把新问题挤掉删旧轮次并预留输出
RAG 拼接检索片段太多，证据尾部消失重排、裁剪、再拼接

这个演算台的目的，是让你养成一个顺序：先算 token 预算，再看输入结构，最后才谈模型效果。只要输入已经不稳，后面的 attention、检索、对话模板都只是放大问题。它也会直接影响后面的评测结果，因为很多“模型不稳定”的现象，其实是预算和截断没有统一。

真实例子：一轮长 prompt 怎么被挤掉

case

把一轮客服问答拆成账本

假设你在做客服问答。系统提示占 620 token，用户问题占 140，历史对话占 1320，检索片段占 2840，预留输出占 900。总占用是 5820。若窗口上限只有 4096，这一轮会超出 1724 token；若窗口是 8192，它表面上能放下，但只要再补 8 个 few-shot 例子，每个 140 token，就会逼近 9300，把尾部证据推到窗口外。

这个例子先让你看哪一块该减：系统提示通常保留，用户问题必须完整保留，历史对话可以压缩，检索片段适合重排后裁短，预留输出不能随便砍。长度预算不是最后才补的步骤，它直接决定输入结构。

如果你只看总 token，很容易把问题误判成“模型不懂”。实际上，很多失败只是因为最该保留的证据被历史和检索挤出窗口了。这个判断在 RAG、Agent、长对话和代码生成里都会反复出现。

块token先不动还是先减如果减错

system620先不动角色和约束会丢

user140先不动问题本身不完整

history1320优先压缩前文重复占位

retrieval2840先重排再裁短关键证据被挤出尾部

output reserve900不能随便砍模型没地方完成回答

这张表的读法很直接：先固定住不能丢的块，再处理可以压缩的块。删历史不等于删结论，裁检索不等于删证据，减少预留输出会直接伤到回答完整度。你真正要找的是“最少删哪一块，才不会把答案结构打坏”。

一行预算检查复制

case_blocks = {"system": 620, "user": 140, "history": 1320, "retrieval": 2840, "reserve": 900}
used = sum(case_blocks.values())
limit = 4096
overflow = used - limit
print(used, overflow)
if overflow > 0:
 print("先压缩 history，再重排 retrieval，最后确认 reserve")

这段代码的作用，是让你一眼看到超出的 token 来自哪里。遇到这种超额时，先看最后一个仍然影响答案的块，通常是检索尾部或最旧的历史轮次。

做完这个检查后，回到练习 11、12 和 17 再答一遍。目标是把“要删哪一块、为什么先删它”说清。只要你能把这轮预算讲明白，后面的 prompt 结构、RAG 拼接和 Agent 观察记录都会更好改。

一个能直接迁移的判断顺序

先看是否超窗，再看是否丢证据，再看是否挤掉输出预算，最后才决定删历史还是裁检索。这个顺序能帮你避免两个常见误判：第一，把“放得下”误当成“放得对”；第二，把“模型答偏”误当成“模型不懂”。只要你手里有一份输入账本，这两个误判都会少很多。

如果要把这节课变成日常动作，最有用的做法是：每次做长 prompt 前，先写一个很小的 token 账本；每次改 RAG 片段前，先看证据是否被挤到尾部；每次追问回答偏差时，先检查输入是否真的含有那条关键条件。这个动作不花太多时间，但能把很多排错路径缩短一大截。

输入验收

check

长 prompt 出问题时先看这四步

先看 tokenizer 输出，再看 attention_mask 和 padding，再看尾部 decode，最后回头算 system、user、history、retrieval 和 reserve 的账。这个顺序能把“像理解错了”拆成四个检查点：切分有没有变长，输入有没有被截断，补齐有没有混进计算，关键条件有没有留到最后。

只要其中一项没过，后面的 attention、检索和对话模板都只能在错误输入上继续工作。很多时候，模型本身没有先坏，先坏的是输入账本。

顺序检查什么为什么先看它

1tokenizer 输出和真实 token 数先确认输入是否比眼睛看到的更长。

2attention_mask 和 padding先确认补齐位置有没有被误算成内容。

3decode 的尾部片段先确认关键条件有没有留在窗口里。

4system / user / history / retrieval / reserve先看是哪一块把窗口吃掉了。

最小排错动作复制

encoded = tokenizer(prompt, truncation=True, return_tensors="pt")
print(encoded["input_ids"].shape)
print(encoded["attention_mask"].sum(dim=1))
print(tokenizer.decode(encoded["input_ids"][0][-60:]))

这三行够用来找大多数输入问题。先看长度，再看有效 token，再看尾部内容。它们不替代完整评测，但能让你很快知道问题是切分、截断还是证据位置。

常见误区

debug

最常见的五个误区

误区错因后果修法

把 token 当字数用肉眼长度代替 tokenizer 结果窗口预算失真用目标模型 tokenizer 统计

把 id 当语义向量看到整数就想比较大小误读模型输入区分 id 查表和 embedding 计算

以为 embedding 会读上下文把静态查表和上下文混在一起解释不清“苹果”的不同语境说明上下文混合发生在后续层

以为窗口越长越好忽略成本和噪声延迟高、费用高、重点被淹没先检索筛选，再放入窗口

只开 truncation 不看结果把“能运行”当“输入正确”关键证据丢失打印长度和解码片段

如果你遇到“模型突然忘了前文”或者“回答和文档末尾对不上”，先不要改模型参数，先回到这张表看输入端有没有把真正要看的内容送进去。很多问题在这里就能找到头绪。

三组经常混在一起的概念

概念对准确区别判断方法

token / wordword 是语言里的词；token 是 tokenizer 产生的模型输入片段。看 tokenizer 输出，不用肉眼猜。

token id / embeddingid 是离散编号；embedding 是查表得到的连续向量。看形状：[B, N] 常变成 [B, N, D]。

context window / 模型知识窗口是单次输入长度；知识来自训练参数和当前输入。问自己：证据是否真的进了这次输入。

padding / truncationpadding 是补齐短样本；truncation 是裁掉长样本。padding 配 mask，truncation 查保留内容。

概念区分的目标，是让你在代码里看到 input_ids、attention_mask、max_length 这些字段时，马上想到它们各自对应哪一步：谁在切分，谁在补齐，谁在截断，谁在屏蔽。这个判断习惯会在后面的 RAG、Prompt、Eval 和 Agent 里反复用到。

先知道，先别钻

boundary

这节课已经够你开始算输入预算了，但还没有进入 attention head 的分工、位置编码的数学细节、长上下文压缩、RAG 分块策略、微调格式和训练损失。先把“文本如何变成可计算输入”这件事看懂，再去看后面的单元会顺很多。下一节会直接接到 Transformer 与 Attention；等你学到 RAG 和 Agent，再回来看这一节里的长度预算，会更容易判断证据该放哪里、该删哪里。

如果你现在还在分不清 tokenizer 和 embedding，可以先只记一句：前者负责切分和编号，后者负责把编号变成可计算向量。最实用的检查顺序是：打印 tokenizer 输出，数 token，确认 mask，再看尾部 decode。只要这四步稳，很多“模型好像没理解”的感觉都会变成具体的输入问题。

往后怎么接

bridge

接 Transformer

下一节会看向量怎样在序列里互相影响。这里先记住：Transformer 读的是 embedding 后的序列，不是原始字符串。

接 Agent

Agent 调工具、写观察、再把结果送回模型时，也会重复面对 token 预算和截断问题。输入链路看懂了，后面才知道哪里该压缩，哪里该保留。

diagnostic

怎么答才算会

先把对象说清，再说依赖和后果。结构化题尽量用“错因、表现、修法”来答；开放题至少要提到 tokenizer、token id、embedding、context window 这四个对象里的相关关系。

检查重点

这页最重要的不是背词，而是能不能看懂输入链路、形状变化和长度预算。答题时如果提到 token 数、mask、截断或预算，记得说明你是在检查哪一步。

3ordering

4synthesis

Q: 4. 为什么不能根据 token id 的大小判断两个 token 语义是否接近？

参考答案 / 评分要点

。因为 token id 只是词表里的编号，编号大小不表示语义距离。真正的语义关系来自 embedding 和后续上下文表示，不来自编号本身。一个合格答案要把“编号”和“表示”分开。

二、形状和代码阅读

记住最常见的形状转变： [B, N] 进入 embedding 后通常变成 [B, N, D] 。 -->
5choice

Q: 5. input_ids 形状是 [2, 128]，hidden size 是 768，embedding 输出形状通常是哪一个？

参考答案 / 评分要点

[2, 128, 768]

6code reading

Q: 6. padding 后为什么还需要 attention mask？

参考答案 / 评分要点

。padding 会把短样本补到和长样本同样长，但补出来的位置不是内容。attention mask 用来告诉模型哪些位置要参与计算、哪些位置要忽略，否则补齐部分会被误当成真实 token，影响结果。

7choice

Q: 7. attention_mask.sum(dim=1) 常用来检查什么？

参考答案 / 评分要点

真实 token 数

8scenario-transfer

Q: 8. 为什么不同模型的特殊标记不能随便混用？

参考答案 / 评分要点

。不同模型对开始、结束、分隔、填充等特殊标记的约定不同，词表也不同。混用会让输入格式不对，轻则长度和对话角色错位，重则模型读到的是错误标记。

三、窗口预算与故障判断

窗口只管这一次输入和生成，不管模型训练时见过什么。 -->
9synthesis

Q: 9. 用一句话解释 context window。

参考答案 / 评分要点

。context window 是模型一次输入和生成过程中能同时处理的 token 长度上限。好的答案要强调“单次”和“长度上限”，不要写成“模型懂多少知识”。

11checklist

Q: 11. 你要把系统提示、历史对话、5 段检索文本放入模型。应该先检查哪三件事？

参考答案 / 评分要点

。至少要检查三件事：各部分 token 数是否超预算；关键证据是否真的在保留片段里；是否给输出预留了足够长度。更完整的答案可以再加上截断方向、重复片段和检索重排。

12error diagnosis

Q: 12. 模型忽略了文档末尾的关键条件，可能和 context window 有什么关系？

参考答案 / 评分要点

。最常见的原因是文档末尾被截掉、被压缩掉，或者在拼接检索文本时被别的内容挤出窗口。好的答案要把“没看见”与“没判断出来”区分开。

四、错误诊断和迁移

先找输入链路，再决定问题出在窗口、切分还是模型。 -->
13error diagnosis

Q: 13. 有人说“这段只有 2000 个汉字，所以肯定不到 2000 token”。错在哪、后果是什么、怎么修？

参考答案 / 评分要点

。错因是把字数当成 token 数。后果是窗口预算不准，可能把 prompt 拼太长。修法是用目标模型 tokenizer 统计 token，再按 token 预算安排系统提示、历史、检索和输出。

14error diagnosis

Q: 14. 有人用 token id 的差值衡量两个词相似度。错因和修法是什么？

参考答案 / 评分要点

。错因是把离散编号当成连续语义坐标。修法是区分“id 负责查表”和“embedding 负责表示”，语义相似度应该看向量空间或任务表现，而不是看编号差值。

18synthesis

Q: 18. 多轮聊天里，为什么要管理历史消息，而不是无限追加？

参考答案 / 评分要点

。因为窗口有限，历史越长越容易挤掉新问题、系统提示或关键证据；同时更长输入会提高延迟和成本。好的答案要把“限长”和“预算”都说出来。

五、表达输出

把 tokenizer、token id、embedding、context window、截断和 attention mask 串起来。 -->
19synthesis

Q: 19. 用 6-9 句话解释：一段文本是怎样变成大模型可处理输入的。

参考答案 / 评分要点

。参考答案应先说明文本被 tokenizer 切分，再编号成 token id，然后通过 embedding 查表变成向量，接着由 padding、mask 和 window 预算控制输入长度，最后再进入模型。评分时看有没有把对象、顺序和边界说清。

20checklist

Q: 20. 如果让别人写一个 prompt 编码函数，你会用哪 6 条清单验收它？

参考答案 / 评分要点

。至少要验收：是否按目标 tokenizer 统计 token；是否加入正确特殊标记；是否返回 attention mask；是否明确 truncation 策略；是否检查尾部解码片段；是否预留输出并做总 token 预算。答案不要求逐字一致，但要覆盖这六项里的大部分。
