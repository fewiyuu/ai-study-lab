# 先抓主线：一个模型从训练到调用会经历什么 {#overview}
<!-- tag: overview -->
<!-- layout: grid2 -->

## box
### 把“模型变好”拆成几种动作
预训练发生在基础模型形成阶段。训练目标通常是根据前文预测下一个 token，模型通过海量样本更新参数，学到语言结构、常识片段、代码模式和大量统计关联。它解决的是通用能力起点，成本高，数据和训练工程都重。
微调发生在已有模型之后。它仍然会更新参数，只是数据规模、目标范围和成本小得多。它适合把模型拉向稳定任务口径、输出格式、领域表达或偏好行为。推理发生在模型参数基本固定之后，工程师主要改本次输入、上下文、工具结果、解码参数和输出约束。
应用工程大多在推理侧和检索链路里做文章。embedding 负责把文本变成向量来召回，reranking 负责对候选材料做精排，generation 负责根据 prompt 和上下文组织答案。它们属于调用侧，不在训练阶段内。

# 阶段边界：同一句“让模型更好”，可能指四种改变 {#boundary}
<!-- tag: boundary -->

**状态账本**先看读写方向，再看名词。训练会把状态写进参数；推理会把状态写进输出、日志和会话痕迹；评估会把状态写进失败分类和下一轮动作。

动作
读入
写出
副作用
什么时候会过期
常见误判


预训练
海量通用文本、代码、混合语料
基础模型参数 `theta` 和 checkpoint
训练成本高，周期长，常常先把算力和数据一起固定住
当目标是私有文档、频繁更新的规则或短周期业务口径时
把“给模型看一份新文件”叫预训练


微调 / 对齐
任务样本、指令样本、偏好样本、领域样本
原模型参数或少量适配参数
可能把口径学窄，也可能把通用能力挤掉一部分
当任务口径改得快，样本和评估集跟不上时
把微调当作频繁更新的知识库


推理
prompt、上下文、工具结果、检索片段
答案、JSON、代码、分数、日志和会话记录
不会改参数，但会留下 trace、缓存和外部工具结果
每次调用结束后，本次输入就失效
以为调 temperature 会补缺失知识


评估反馈
真实样例、期望答案、错误类型、指标
失败地图和下一轮工程动作
只改决策，不改模型
当样本太少、覆盖不全，或者只看了少数成功样例时
只看几个成功样例就决定微调

# 预训练：让基础模型先学会预测下一个 token {#pretrain}
<!-- tag: pretrain -->
<!-- layout: grid2 -->

## box
### 它改变的是基础参数
预训练的直觉很朴素：给模型一段 token 序列，让它根据前面的 token 预测下一个 token。真实下一个 token 已知，模型预测错了，就通过损失函数和反向传播更新参数。样本足够多时，这个目标会把语言结构、事实片段、代码模式、问答格式和一些推理痕迹压进参数里。
这里要避免一个常见误会：参数更像统计压缩，不是逐篇原文存放的文件柜。模型不会把每篇文章原样放进某个格子里。它形成的是统计模式，所以会有过时、混淆、编造和不确定。新政策、私有文档、每天变化的资料，通常更适合通过检索和上下文交给模型，也更适合放在外部存储里更新。

## box
### 训练目标长什么样
$$
 \mathcal{L}(\theta) = - \sum_{t=1}^{T} \log p_{\theta}(x_t \mid x_{`x_t` 是当前位置的真实 token，`x_{<t}` 是它前面的 token，`theta` 是模型参数。训练时最小化这个损失；推理时用学到的概率分布继续生成。
最小训练循环的读法复制
```python
for batch in token_batches:
 x = batch[:, :-1] # 前缀 token
 y = batch[:, 1:] # 真实下一个 token
 logits = model(x) # 参数 theta 参与前向计算
 loss = cross_entropy(logits.reshape(-1, vocab), y.reshape(-1))
 loss.backward()
 optimizer.step() # 这里会更新参数
 optimizer.zero_grad()
```


**检查点**如果一个方案只是把一份制度文档放进 prompt、向量库或上下文窗口，它没有做预训练。预训练意味着参数被训练过程更新，通常需要很大的数据和算力。

# 微调：在已有模型上稳定改变任务行为 {#finetune}
<!-- tag: finetune -->
<!-- layout: grid2 -->

## box
### 它适合改变“怎么答”
微调是在基础模型或指令模型上继续训练。它可以让模型更稳定地遵守某类格式，学习某个任务的判断口径，适应领域表达，或通过偏好数据更贴近人类反馈。常见做法包括全量微调、LoRA/QLoRA 这类参数高效微调、指令微调和偏好对齐。
微调适合稳定、重复、高价值的问题。比如每天有大量同类客服质检文本，需要输出固定 JSON 字段；或模型总是在同一类标签边界上摇摆，而你已经有高质量标注样本和评估集。它不适合当成“把最新资料同步进模型”的唯一手段。

## box
### 微调前先问四个问题
问题为什么要问如果答案是否定

失败是否稳定重复？微调适合改长期模式先保存失败样例，别急着训练
是否有高质量样本？坏样本会把错误口径写进参数先做数据清洗和标注规范
是否有评估集？没有评估就无法判断整体是否退化先建 20-50 条真实样例做最小 Eval
prompt / RAG / rerank 是否已排查？很多问题在推理链路能低成本解决先修输入、上下文、召回和排序

一条指令微调样本的边界复制

```json
{
 "instruction": "判断客服质检文本是否存在承诺超范围。",
 "input": "客服表示：这个赔付一定能通过，明天到账。",
 "output": {
 "label": "违规",
 "reason": "承诺了无法保证的赔付结果和到账时间。"
 }
}
```

这条样本教的是任务口径和输出格式。它可以帮助模型在同类文本上更稳定，但不适合承载每天变化的政策手册。资料频繁变化时，把资料放进检索系统，再用 prompt、schema 或微调约束回答方式，通常更容易维护。

# 推理：参数固定后，工程师主要改输入和解码 {#inference}
<!-- tag: inference -->
<!-- layout: grid2 -->

## box
### 推理阶段能控制什么
普通推理调用时，模型参数通常固定。你能控制的是 system prompt、用户输入、检索上下文、工具返回、函数调用约束、输出 schema、temperature、top_p、max_tokens、stop sequence 等。本次调用结束后，这些设置不会永久改变模型参数。
很多应用质量问题发生在这里。模型没有引用证据，先检查证据是否真的放进上下文；输出 JSON 不稳定，先检查 schema、示例和格式约束；答案忽长忽短，先看 `max_tokens` 和任务拆分；答案发散，先看采样参数和问题边界。

## box
### 解码参数读法
控制项影响什么不能解决什么

`temperature`采样发散程度，越低越保守不能补缺失证据或私有知识
`top_p`候选 token 的概率截断范围不能修复错误上下文
`max_tokens`输出长度上限不能保证答案完整，只会截断
`response_format` / schema输出结构约束不能替代任务定义和字段语义
上下文材料模型本次可读的证据不能修复召回不到材料的问题

推理调用关注输入、上下文和解码参数复制

```python
messages = [
 {"role": "system", "content": "只根据给定证据回答；没有证据就说无法判断。"},
 {"role": "user", "content": user_question},
 {"role": "tool", "content": retrieved_policy_snippets}
]

response = client.responses.create(
 model="deployed-instruction-model",
 input=messages,
 temperature=0.2,
 max_output_tokens=400,
)

# 这段代码没有训练模型。它改变的是本次输入和生成控制。
```

# embedding、reranking、generation：应用链路里的三种调用角色 {#roles}
<!-- tag: roles -->
<!-- layout: grid2 -->

## box
### 先看输入输出
**embedding** 把文本变成向量，常用于相似度检索、聚类和去重。它的输出通常是一串浮点数，适合拿去做相似度计算，而不是直接拿来回答问题。
**reranking** 通常接收“查询 + 候选文档”，输出相关性分数或排序。它不负责从全库召回，而是对已经召回的候选做精排。它更细，也更慢，所以常放在召回之后。
**generation** 负责生成文本、代码或结构化结果。它可以基于 prompt 和检索材料组织答案，但如果没有给它证据，它对最新或私有知识不一定可靠。

## box
### 三者对照
角色输入输出典型位置

embedding一段文本或 query向量召回相似文档、聚类、去重
rerankingquery + 候选文本相关性分数或排序召回之后，进入上下文之前
generationprompt + 上下文 + 工具结果自然语言、代码、JSON最终回答或执行指令

RAG 中三种角色的最小链路复制

```python
query_vec = embedding_model.encode(user_question)
candidates = vector_store.search(query_vec, top_k=30)

ranked = reranker.score_pairs(user_question, candidates)
context = take_top(ranked, k=5)

answer = generation_model.generate(
 prompt=build_prompt(user_question, context),
 temperature=0.2,
)
```

**判断句**要“找资料”，先看 embedding 和 reranking；要“组织答案”，再看 generation；要“长期稳定改变任务行为”，才看微调和评估集。

# 代码读写：先看一条调用链到底改了谁 {#readwrite}
<!-- tag: read/write -->
<!-- layout: grid2 -->

## box
### 把一条调用拆成四个动作
同一段代码里，最容易混在一起的是“读到什么”和“写出去什么”。如果不先拆这两件事，后面就很容易把推理当训练，把外部工具结果当参数记忆。
读法很简单：先找输入，再找模型是否产生新的状态，最后看这个状态是只留在本次调用里，还是会落到文件、日志、缓存或者 checkpoint。只要落点不同，工程决策就不同。

## box
### 三类动作的读写账本
动作读什么写什么最常见的错读

预训练大语料、token 序列、损失信号参数、checkpoint、优化器状态把“加载语料”误当成“完成训练”
微调任务样本、标签、偏好数据参数或 adapter把“改提示词”误当成“已经改参数”
推理prompt、上下文、工具结果、schema答案、分数、日志、缓存痕迹把 temperature 当成知识补丁

读写顺序的最小示意复制

```python
# 读：当前问题、上下文、检索片段
messages = build_messages(question, context, retrieved_docs)

# 写：这次调用的输出，不改参数
result = client.responses.create(
 model=model_name,
 input=messages,
 temperature=0.2,
 max_output_tokens=400,
)

# 如果这一层还要留下副作用，通常写到日志、数据库或缓存
log_interaction(question, result.output_text)
```

**检查点**如果代码只看到 `result = ...`，那还不够。还要问一遍：这次调用改了模型参数吗，还是只写出了结果和日志。这个判断会直接影响你后面是改 prompt、补检索，还是去做微调比较。

**最小回放**如果系统只允许你保留一条记录，优先保留问题、上下文、输出和当时的 trace。后面排查经常要回看中间状态；只留最终答案，通常只能知道错了，不能知道错在召回、排序还是约束。

**状态分界**训练状态通常放在参数、adapter、checkpoint 和优化器状态里；调用状态通常放在消息、上下文、缓存和日志里。看到文件落点不同，就能知道下一步是回训练流程，还是回推理流程。界面上把这两类按钮放在一起，最容易让人误判。

# 决策链路：效果差时先定位，再选择动作 {#decision}
<!-- tag: decision -->
<!-- layout: grid2 -->

## box
### 同一个“答错”，可能来自不同层
假设内部助手把“报销超过 5000 元需要两级审批”答成“一级审批”。如果检索上下文里没有最新制度，先修 RAG：更新资料、重建索引、检查召回。若上下文里有正确条款，但模型没有引用，先修 prompt 和输出约束：要求只根据证据回答，列来源，资料冲突时拒答。
如果问题长期把同类质检文本分错类，并且你有稳定样本、标注规范和 Eval，再考虑微调。三种动作都可能让少数样例变好，但维护成本和风险完全不同。工程决策先从失败分类入手，再看技术选型。

## box
### 失败到动作的映射
症状先查哪里优先动作先别急着做

缺最新政策或私有文档上下文里有没有证据RAG / 上下文注入完整预训练
召回结果相关但顺序乱候选文档和 query 是否匹配reranking盲目塞更多片段
输出格式不稳定schema、示例、temperatureprompt、结构化输出约束、低温度马上微调
同类任务长期大量重复样本、评估集、标注口径评估集 + 高质量样本 + 微调只靠越来越长的 prompt
答案忽长忽短或截断任务拆分和 token 上限调整 max tokens，拆分步骤把截断归因于模型没学会

<!-- interactive: ['terminal-lab'] → keep as HTML -->

**真实排错顺序**先看证据有没有进来，再看证据有没有被排到前面，再看输出约束有没有把答案固定住，最后才问要不要训练。这个顺序不是形式主义，目的是把低成本动作放在前面，避免把可修的链路问题直接推成训练问题。

# 排查桥：把一个失败样例拆到可行动 {#operation}
<!-- tag: operation -->
<!-- layout: grid2 -->

## box
### 先保留现场，再谈方案
当一个 LLM 应用答错时，最有用的第一步是保存一条完整失败现场。至少记录六样东西：原始用户问题、系统提示、检索到的上下文、候选文档排序、模型实际输出、期望行为。缺哪一项，后面的判断就容易靠感觉。
例如“报销审批答错”这类问题，先把检索上下文打印出来。如果正确条款没有出现，问题更接近资料更新、切分、索引、召回或权限过滤。如果正确条款出现了，但答案仍然没引用，问题更接近 prompt 约束、输出格式、证据引用规则或 generation 的任务描述。如果条款和 prompt 都没问题，却在同类分类任务上长期摇摆，再检查是否有足够样本支持微调。
这条路径能保护你少走弯路：RAG 问题用 RAG 证据修，格式问题用结构约束修，排序问题用 reranking 修，稳定行为问题才进入微调比较。每一步都要能留下对比结果，否则只是把一个不确定动作换成另一个不确定动作。

## box
### 失败现场的读写账本
检查点读取什么写下什么结论下一步动作

问题和期望原问题、业务规则、人工期望答案错误是事实错、格式错、排序错还是判断口径错给失败样例贴一个主标签
检索上下文top-k 文档、chunk、来源、更新时间模型是否看到了必要证据没看到就修索引、召回、权限或资料同步
候选排序召回候选和 rerank 分数正确片段是否被排到可用位置候选有但靠后，就看 reranking 或上下文截断
生成约束system prompt、schema、示例、temperature模型是否被要求只按证据回答并按格式输出约束弱就先改 prompt / schema / 解码配置
稳定性证据同类失败数量、样本质量、Eval 覆盖是否值得进入微调实验有稳定证据才做训练方案对比
把这张账本真正用起来时，先填一条实际失败样例，再看哪一列是空的。空在“检索上下文”，就修索引和召回；空在“生成约束”，就改 prompt、schema 或温度；三列都不空还反复错，才值得去比较微调。

最小排查记录结构复制

```json
{
 "case_id": "policy-reimburse-001",
 "user_question": "报销超过 5000 元需要几级审批？",
 "expected": "需要两级审批，并引用最新制度条款。",
 "retrieved_context": {
 "has_required_policy": false,
 "top_chunks": ["旧版报销制度：超过 5000 元由直属主管审批"]
 },
 "model_output": "需要一级审批。",
 "primary_failure": "missing_evidence",
 "next_action": "更新资料、重建索引，并用同一问题回归测试"
}
```

## box
### 修完以后要回归同一批样例
排查写完结论以后还没结束。你至少要用同一批失败样例再跑一次，比较改动前后的证据命中、候选排序、输出格式和最终答案。比如这个案例的第一轮结论是 `missing_evidence`，修复动作就应该让最新制度条款进入 top-k，并且最终答案能引用它。如果只是答案偶然变对，但上下文里仍然没有证据，问题没有真正解决。验收时要同时看“证据找到了”和“答案按证据说了”。这能避免把偶然正确当成系统修复。

## box
### 把诊断写成可复查的句子
好的诊断句通常有四段：我观察到什么、它说明哪一层出问题、我先做哪个低成本动作、我用什么样例验证。写成“因为 top-k 没有最新制度，所以先更新索引并回归 20 条制度问答”比写成“模型不行，微调一下”更可执行，也更容易被别人复查。

**练习钩子**后面的诊断题不要只写“应该微调”或“应该 RAG”。先写你读到了什么现场证据，再写为什么这个动作比另一个动作更接近根因；最后补一句你会怎么回归验证。

# 误区诊断：把边界说清，方案才不会乱 {#diagnosis}
<!-- tag: diagnosis -->
<!-- layout: grid2 -->

## box redsoft
### 五个高频误区
1. 把 RAG 和微调混在一起。RAG 把材料放进当前上下文；微调改变模型参数或适配参数。
2. 用微调同步频繁变化的事实。事实每周更新时，检索系统比参数记忆更容易维护。
3. 拿 embedding 模型直接问答。embedding 输出向量，生成模型才负责组织自然语言答案。
4. 用 temperature 修事实错误。temperature 改采样倾向，不会凭空增加证据。
5. 没有 Eval 就训练。少数样例变好不代表整体变好，微调可能让别的能力退化。

## box
### 易混概念快查
概念组区别最快判断

pretraining / finetuning预训练学通用基础参数；微调用较小数据适配任务或行为看数据规模、目标和是否从基础模型继续训练
finetuning / prompting微调改参数；prompting 改本次输入调用结束后模型是否永久改变
RAG / long contextRAG 包含检索、筛选、注入；长上下文只是窗口更大看有没有召回、排序和证据管理
embedding / rerankingembedding 做向量召回；reranking 对候选精排看输入是单文本，还是 query 加候选对

**回到下一节**下一页看 HuggingFace 最小示例时，先认出自己是在做推理调用：加载已有模型，组织输入，读取输出。代码里出现训练循环、loss、backward、optimizer step，才是在训练。

- [ ] 模型全部参数
- [ ] 本次输入、上下文和解码配置
- [ ] 训练语料本身

Answer: 本次输入、上下文和解码配置
Explain: 普通推理不会永久更新模型参数，主要改变本次输入、上下文和解码参数。

### 3 ordering

**Q:** 3. 哪个顺序最符合常见模型生命周期？

- [ ] 预训练 -> 微调/对齐 -> 部署推理
- [ ] 推理 -> 预训练 -> 微调/对齐
- [ ] reranking -> 预训练 -> embedding

Answer: 预训练 -> 微调/对齐 -> 部署推理
Explain: 典型顺序是先预训练形成基础模型，再微调或对齐，最后部署后推理调用。

### 4 true/false

**Q:** 4. prompt 调整和微调都会永久改变模型参数。

- [ ] 对
- [ ] 错

Explain: prompt 改本次输入；微调改参数或适配参数。

### 5 scenario

**Q:** 5. 企业制度每周更新，问答经常答错。第一优先级更接近哪项？

- [ ] 从头预训练一个模型
- [ ] 检查资料更新、索引、召回和上下文注入
- [ ] 只把 temperature 降到 0

Answer: 检查资料更新、索引、召回和上下文注入
Explain: 资料每周更新，优先把最新证据放进检索和上下文，微调不是同步资料的首选。

### 6 choice

**Q:** 6. 哪种情况更适合认真比较微调方案？

- [ ] 只看见一次输出格式错了
- [ ] 资料今天刚更新，模型不知道
- [ ] 大量稳定任务反复出错，已有高质量样本和 Eval

Answer: 大量稳定任务反复出错，已有高质量样本和 Eval
Explain: 稳定样本、标注规范和评估集是进入微调讨论的基本条件。

### 7 debug-checklist

**Q:** 7. 你准备微调前，会先检查哪些项？

- [ ] 失败是否稳定重复
- [ ] 样本和标注是否可靠
- [ ] 是否有固定评估集
- [ ] prompt、RAG、rerank 是否已经排查
- [ ] 是否只是想尝试新技术

Explain: 应优先勾选前四项。微调前要确认失败稳定、样本可靠、有评估集，并且已经排查低成本动作。“只是想尝试新技术”不是工程理由。好的答案还会说明：没有 Eval 时，微调可能只是在少数样例上显得更好。

### 12 scenario

**Q:** 12. 终端案例显示：检索上下文里没有最新制度条款。下一步优先做什么？

- [ ] 更新资料、重建索引、检查召回
- [ ] 直接做完整预训练
- [ ] 只把输出长度调大

Answer: 更新资料、重建索引、检查召回
Explain: 上下文没有召回最新制度时，先修资料、索引和召回链路。

### 13 synthesis

**Q:** 13. 有人说“RAG 就是微调的一种”。请写出错因、后果和修法。

Explain: 好的答案会说：RAG 是在推理时检索外部材料并注入上下文，通常不改变模型参数；微调会训练参数或适配参数。把两者混在一起，会导致频繁变化资料被错误地拿去训练，维护成本变高，还难以及时更新事实。修法是先判断问题需要外部证据、排序优化、格式约束，还是稳定行为适配。

### 14 synthesis

**Q:** 14. 用 6-8 句话解释：预训练、微调、推理、embedding、reranking、generation 各自负责什么，并给出一个效果差时的决策顺序。

Explain: 高质量答案会把六个词放在同一张工程地图里：预训练形成基础参数，微调用任务样本稳定行为，推理用固定参数处理本次输入；embedding 输出向量用于召回，reranking 给候选排序，generation 组织最终答案。面对效果差，先检查模型有没有证据，再检查 prompt、schema、temperature 和输出约束，然后检查召回与排序；只有在任务稳定、大量重复、有样本和评估集时，才进入微调比较。
