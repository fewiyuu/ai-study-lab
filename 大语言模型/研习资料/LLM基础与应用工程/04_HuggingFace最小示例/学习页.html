这一页把 Hugging Face 最小示例讲成一条能验收的调用链：先判断该用 pipeline、AutoTokenizer + AutoModelForSequenceClassification，还是 AutoTokenizer + AutoModelForCausalLM + generate；再看输入怎么变成张量，输出怎么回到可读结果；最后用常见错误检查这段 demo 是否真的站在正确的模型头和解码路径上。

学完这页你能做什么

  • 能先判断一个 Hugging Face demo 是快速跑通任务，还是需要拆开看 tokenizer、模型头和输出字段。
  • 能读懂 input_ids、attention_mask、logits 和 generate() 各自负责哪一步。

学习目标

GOAL

这一页要解决什么

Hugging Face 的示例常常看起来很短，实际却藏着三条不同的路径：最快的标准任务入口是 pipeline，想看中间字段就拆成 AutoTokenizer + AutoModelForSequenceClassification，想续写文本就换成 AutoModelForCausalLM + generate()。这页的任务是把这三条路分清楚，并能看出一段 demo 读了什么、写了什么、错在哪一层。

这门课前一节已经把“训练、适配、调用”分开了。现在要把那条边界落到代码上：看到一个 Hugging Face 例子，先判断它是快速验收、手动检查，还是生成调用，再决定要不要继续往下拆。

先记住三句话

  • pipeline 适合快速跑通标准任务。
  • 手动拆开 tokenizer 和 model，才看得到 input_ids、attention_mask、logits。
  • generate() 读的是 prompt 和解码参数，写的是续写出来的 token 序列。

先抓主线

ROUTE

一条 demo 的最小链路

你可以把 Hugging Face 的最小示例看成一条固定顺序：先选任务，再选模型头，再让 tokenizer 把文本变成模型能吃的张量，最后读输出并做后处理。pipeline 会把这条链打包起来；手动拆开时，你能自己看到中间对象和失败点。

这页不讲训练、微调或部署框架。它只负责让你在运行一段示例时，能判断示例到底在读什么、写什么、有没有把任务和模型头配对好。

路径读写

TRACE
pipeline字符串 -> 结果字典

读取任务名、模型名和原始文本，内部依次完成预处理、模型前向和后处理。适合先验收一条标准任务链。

tokenizer文本 -> input_ids / attention_mask

读取文本和模型词表，写出批量张量。这里最容易错的是 padding、截断和模型名不一致。

SequenceClassificationbatch -> logits

读取已经对齐的张量，写出每个类别的分数。它不负责续写文本，也不负责生成完整答案。

CausalLMprompt -> token 序列

读取 prompt、采样参数和设备设置，写出续写 token。max_new_tokens 控制新增长度，skip_special_tokens 决定最后怎么读回文本。

路径选择表

你要的结果先用什么你会读什么你会写什么
快速确认标准任务可跑通pipeline结果标签或生成文本最终答案
想看类别分数AutoTokenizer + AutoModelForSequenceClassificationlogits分数和标签 id
想续写一段文本AutoTokenizer + AutoModelForCausalLM + generate()prompt 和生成参数token 序列和解码后的文本

一眼能看出的边界

如果代码里只有 pipeline，你通常看不到中间 logits；如果代码拆开 tokenizer 和 model，你可以验收每个张量；如果代码用的是分类头，却去调用 generate()，大概率就是路径错了。

pipeline：先跑通标准任务

PIPELINE

它适合什么

pipeline 把预处理、模型前向和后处理打包成一个入口。它最适合做标准任务的快速验收，比如情感分类、问答、摘要和文本生成。你先用它确认模型名、任务名和设备设置基本可用，再决定要不要拆开看中间字段。

它不适合你想盯着 logits、attention 或自定义后处理的场景。那时继续把任务打包在 pipeline 里，只会让你少看见关键中间量。

最小示例

标准任务链
from transformers import pipeline

clf = pipeline(
    task="sentiment-analysis",
    model="distilbert-base-uncased-finetuned-sst-2-english"
)
result = clf("This demo is small, but it is enough to verify the path.")
print(result)
对象读什么写什么如果错了会怎样
pipeline任务名、模型名、输入文本最终标签、分数或生成文本任务和模型头不匹配时，错误会在这里被掩盖
内部 tokenizer原始字符串模型输入张量看不见 input_ids 时，很难判断长度和 padding
内部模型张量 batch分数或 token 序列当你需要中间值时，pipeline 太粗

tokenizer：文本先变成模型能吃的输入

TOKEN

它到底写了什么

tokenizer 负责把字符串变成模型输入。对一条文本来说，它通常会写出 input_ids、attention_mask，必要时还会写出 token_type_ids。批量输入时，它还会把不同长度的文本补成同一个矩形 batch。

这一步最常见的三个错误是：tokenizer 和 model 不是一对、没有处理 padding 或 truncation、没有把张量放到同一个设备上。输入看起来只是几行字，到了模型前面就变成了要对齐的张量结构。

最小示例

tokenizer 输出
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("distilbert-base-uncased")
batch = tok(
    ["good demo", "a slightly longer demo"],
    padding=True,
    truncation=True,
    return_tensors="pt"
)
print(batch["input_ids"].shape)
print(batch["attention_mask"].shape)
检查对象进入前出来后它读什么它写什么错了的症状
AutoTokenizer一段文本或文本列表批量张量模型词表、特殊符号、最大长度input_ids、attention_mask模型能跑，但输入被截断或补齐错位
padding=True不同长度的句子同宽 batch当前 batch 的最长句子补齐 token没有 padding 时，batch 无法堆成张量
truncation=True超长输入限制在模型最大长度内模型的长度上限被截掉的尾部长文本报错或关键信息消失

手动分类：想看 logits 时就别停在 pipeline

CLASSIFY

为什么要拆开

当你只需要一个标签，pipeline 已经够用；当你需要每个类别的原始分数、要做自定义阈值、或者要核对模型到底把哪一类看高了，就要把 tokenizer 和分类头拆开。这样你能直接读到 outputs.logits，再自己决定怎么转成概率和标签。

$$p_i = \frac{e^{z_i}}{\sum_j e^{z_j}}$$

这里 z_i 是第 i 个类别的 logit，p_i 是 softmax 后的概率。argmax 选最大概率对应的类别 id。

最小示例

手动分类链路
import torch
from transformers import AutoTokenizer, AutoModelForSequenceClassification

name = "distilbert-base-uncased-finetuned-sst-2-english"
tok = AutoTokenizer.from_pretrained(name)
model = AutoModelForSequenceClassification.from_pretrained(name)

batch = tok("A tiny example is enough to see logits.", return_tensors="pt", truncation=True)
with torch.inference_mode():
    outputs = model(**batch)
logits = outputs.logits
probs = logits.softmax(dim=-1)
label_id = probs.argmax(dim=-1)
print(logits)
print(label_id)
对象读什么写什么你该看什么常见错法
AutoModelForSequenceClassificationtoken batch类别分数logits把它当成能续写文本的模型
outputs.logits模型前向结果每一类的原始分数分数差值、阈值、排序直接拿原始分数当概率
softmaxlogits归一化概率概率是否稳定把 softmax 当成修正模型知识的办法

生成：要续写就用 Causal LM

GENERATE

它读什么，写什么

生成式模型的最小路径是：tokenizer 先把 prompt 变成输入张量，AutoModelForCausalLM 再根据 prompt 和解码参数往后续写 token，最后把 token 序列解码回文本。这里最容易混的是长度参数：max_new_tokens 只管新增 token 数，max_length 常常管总长度。

如果你想让结果可复现，先关掉采样，设置 do_sample=False，再用较小的 max_new_tokens 验收链路。等路径确认后，再谈更复杂的解码控制。

最小示例

续写链路
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM

name = "gpt2"
tok = AutoTokenizer.from_pretrained(name)
model = AutoModelForCausalLM.from_pretrained(name)

inputs = tok("A short explanation:", return_tensors="pt")
with torch.inference_mode():
    out = model.generate(**inputs, max_new_tokens=40, do_sample=False)
text = tok.decode(out[0], skip_special_tokens=True)
print(text)
参数控制什么不负责什么最常见误解
max_new_tokens新增 token 数不改模型知识以为它等于总长度
max_length通常是总长度不保证回答完整把它当成“想写多少就写多少”的开关
skip_special_tokens解码时去掉特殊符号不改变生成内容本身以为 decode 会重写模型输出

选哪条路

DECIDE

先按目标选入口

如果你只想确认一个标准任务能跑通，先用 pipeline。如果你需要看分数、阈值或中间张量，就拆成 tokenizer 和任务专用模型。如果你要的是续写，就直接上 AutoModelForCausalLM + generate()。这三条路径都能跑，但它们解决的问题不同。

换路线的信号也很明确：一旦你开始问“这一层具体读了什么”，就不该继续停在 pipeline；一旦你开始问“为什么这条答案长度不稳定”，就该看生成参数；一旦你开始问“这个输出分数为什么这样分布”，就该看 logits。

场景推荐入口你先检查什么什么时候换路线
标准分类 / 问答 demopipeline模型名、任务名、输入输出是否能跑通需要看中间分数时
要检查标签分数手动分类链路input_ids、attention_mask、logits需要生成文本时
要续写文本生成链路prompt、设备、max_new_tokens只想看标准标签时

同一句输入，三条路径怎么变

CASE

同一个 prompt，不同的中间对象

假设输入都是 "This demo is small, but it is enough to verify the path."。如果你走 pipeline，你看到的通常是一个标签或者一段最终文本；如果你拆成分类链路，你会先得到 input_ids 和 attention_mask，再得到 logits；如果你走生成链路，你会读到 prompt，模型往后续写 token，再把整段 token 解码成文本。

这个案例的价值在于，它让“路径”不再是抽象概念。你能直接问：这一步到底在读字符串、读张量，还是读模型分数；它到底在写标签、写 logits，还是写 token 序列。只要你开始这样问，很多 demo 的写法就会突然变得很清楚。

同一句输入的三种读写

路径读什么写什么你会不会看见中间字段
pipeline输入字符串标签或最终文本通常看不见
分类链路token 张量logits、概率、标签 id能看见，而且适合验收
生成链路prompt 和解码参数续写 token 和解码文本能看见 token 序列，但通常不看 logits

出错时怎么读这个案例

如果你把分类 demo 写成了 generate，第一眼就会在这里露馅：你本来应该看到 logits，结果却在找 sequences；如果你把生成 demo 写成分类头，结果会变成“能跑但不续写”。这个对照能帮助你把错误直接定位到模型头，而不是把锅扔给 Hugging Face 本身。

同一句输入，三条最小代码

pipeline
result = pipeline("sentiment-analysis", model=name)(text)
手动分类
batch = tok(text, return_tensors="pt")
logits = model(**batch).logits
生成续写
inputs = tok(text, return_tensors="pt")
out = model.generate(**inputs, max_new_tokens=40)

这三行各自暴露了什么

pipeline 暴露的是任务入口；手动分类暴露的是 logits；生成链路暴露的是 prompt 和新增 token。你一旦把这三行放在一起看，很多 demo 的差别就不再只是“API 名字不同”，而是“它到底在做哪种工作”不同。

误区诊断

DIAG
AttributeError: 'SequenceClassifierOutput' object has no attribute 'sequences'

说明：你把分类头当成了生成模型来用。先换对模型头，再看是否真的需要 generate。
症状最可能的层先查什么修法
模型能跑但结果像猜的任务 / 模型头不配模型名和任务名是否对应换成任务专用类或改对 pipeline 任务
CPU 和 CUDA 混在一起设备model 和 batch 的 device统一搬到同一设备，再做前向
长文本报错或丢尾部tokenizerpadding、truncation、max_length先截断或分块，再验收输入长度
想看分数却只有最终标签入口太粗是否仍停在 pipeline拆开看 logits

这里先不追

BOUNDARY

这页先不展开的东西

这页先不追量化、LoRA、微调策略、批量服务、缓存、推理加速、KV cache 和多 GPU 分布式。它们都重要，但它们解决的是下一层的问题，不是这页要教的最小调用边界。

现在先把这三件事做扎实：知道什么时候该用 pipeline，什么时候该拆出 tokenizer 和任务头，什么时候该进 generate()。只要这三件事稳了，后面的检索、提示、评估和 Agent 才能顺着接下去。

8. 为什么多句批量输入时，attention_mask 不能随便省掉？

好的答案会说：批量里短句会被 padding 到同一长度，attention_mask 告诉模型哪些位置是真 token、哪些位置只是补齐。没有它，模型可能把补齐位也当成有效输入，影响分类分数或生成结果。完整答案最好再补一句：这也是为什么要同时检查 tokenizer、padding 和 truncation。

9. 如果报错里同时出现 CPU tensor 和 CUDA tensor，最先该查什么？

回答时按“错因、后果、修法”写，别只说“搬到 GPU”。

参考答案 / 评分要点

应先查模型和 batch 是否在同一 device 上。错因通常是模型被放到了 GPU，但输入还留在 CPU，或者反过来。后果是前向直接报错，或者某些张量先被隐式搬运导致性能差。修法是统一把 model 和 batch 搬到同一设备，然后再做前向和生成。

参考答案 / 评分要点

高质量答案会把步骤串成一条线：先判断任务，再让 tokenizer 把文本变成张量，把张量送进对应模型头，读出 logits 或生成 token，最后做 softmax、decode 或标签映射。排查顺序应是：先看任务和模型头是否对应，再看 tokenizer 是否配套和输入长度是否合理，再看设备是否一致，最后才看后处理和生成参数。结尾最好补一个“能跑但不对”的例子，比如分类头误当生成模型、长文本被截断、或者把 max_length 和 max_new_tokens 搞混。
