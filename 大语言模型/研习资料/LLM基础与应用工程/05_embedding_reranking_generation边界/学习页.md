# 先抓主线 {#overview}
<!-- tag: ROUTE -->
<!-- layout: grid2 -->

## box
### 一条最小检索链
用户提问后，系统先把问题和文档变成向量，用向量相似度捞出一批可能相关的候选。接着 reranker 重新看 query 和候选的成对关系，把最能回答当前问题的片段排前面。最后 generation 只根据这些片段组织回答。
这条链路看的是谁最适合当前一步。embedding 负责覆盖面，reranking 负责精细判断，generation 负责表达和组织。把三件事分开后，系统更容易控制速度、成本和答案稳定性。

## box soft
### 先记住三句话
- embedding 先找一批可能相关的材料，不负责证明答案一定正确。
- reranking 只在少量候选里重新排顺序，不适合扫全库。
- generation 负责基于材料写回答，不负责替你找证据。

# Embedding：快速召回候选 {#embedding}
<!-- tag: ROLE 1 -->
<!-- layout: grid2 -->

## box
### 它解决什么
embedding 把 query 和文档片段压成向量，再用相似度找最近的候选。它解决的是规模问题：你不能把全库都交给 reranker，也不能直接把全库塞进 generation，所以先用向量检索把范围缩小。
它的边界也很清楚。向量近，只说明“像”，不说明“真的包含答案”。如果用户问“报销超过 5000 元需要几级审批”，embedding 可能先把“报销流程总览”找出来，但真正写了“超过 5000 元需要两级审批”的段落未必排在最前面。

## box soft
### 最小判断
看什么说明什么不说明什么


相似度高语义接近不等于事实命中
TopK 大一点覆盖更稳不等于答案更准
候选很多召回更宽不等于后面能直接回答

向量召回的最小骨架复制

```python
query_vec = encoder.encode(query)
doc_vecs = encoder.encode(documents)
scores = similarity(query_vec, doc_vecs)
top_ids = scores.argsort()[::-1][:10]
candidates = [documents[i] for i in top_ids]
```

# Reranking：对候选做精细判断 {#rerank}
<!-- tag: ROLE 2 -->
<!-- layout: grid2 -->

## box
### 它和 embedding 的差别
reranking 接在召回后面。它看到的是 query 和候选文档的成对输入，因此能更细地判断“这个片段是不是直接回答当前问题”。和 embedding 相比，它更准，也更贵，因为每个候选都要单独跑一次相关性评分。
工程上常见的顺序是：embedding 从大库召回几十条，reranker 把这些候选重新排一遍，再把前几条送进 generation。这个顺序本身就是成本控制：先粗后细，避免拿昂贵模型去扫全库。

## box warn
### 一个容易犯的错
把 reranker 当成全库搜索器。这样做会让 query-document pair 数量爆炸，延迟和成本一起上去。reranker 最适合做“候选已经不多，但顺序还不可靠”的那一步，不适合做第一阶段召回。
候选重排骨架复制
```python
pairs = [(query, doc) for doc in candidates]
scores = reranker.predict(pairs)
ranked = sorted(zip(candidates, scores), key=lambda item: item[1], reverse=True)
final_docs = [doc for doc, score in ranked[:5]]
```


## box
步骤读什么写什么症状


成对打分query + candidate相关性分数如果候选太多，处理时间明显变长
重排输出分数和候选顺序重新排序后的 TopN如果把分数当证据，会误以为高分一定事实正确
送入 generation少量高分片段prompt 上下文如果上下文太长，最后证据会被挤掉

# 故障台：三种坏结果怎么切 {#lab}
<!-- tag: LAB -->
<!-- layout: grid2 -->

## box
### 先看输出，再看层
这页的排错顺序固定：先看候选有没有进来，再看排序有没有把关键片段推上去，最后看生成是不是把证据说歪了。你如果一上来就改 generation，很容易把问题藏起来；一上来就改 TopK，也可能只是把噪声放大。
下面的终端台把三个最常见的坏结果做成可切换案例。每个案例都写了“先查什么”和“别先动什么”，这样你在真实系统里看到类似现象时，能先停在正确层面，而不是把三层一起改。
- 案例一：候选里根本没有答案，重点查切分、query 表达和 embedding。
- 案例二：正确片段在列表里但排不前，重点查 reranker 的 pair 输入和分数。
- 案例三：证据已经在上下文里，但回答还是飘，重点查 prompt 和生成参数。

[召回缺证据]

# 桥接：把一条请求走完 {#bridge}
<!-- tag: FLOW -->
<!-- layout: grid2 -->

## box
### 从 query 到 answer 的四个检查点
拿“报销超过 5000 元需要几级审批？”这个句子做例子，先别急着改模型。先看 query 有没有被切成可检索的表达，再看 embedding 召回里有没有包含审批规则的片段，再看 reranker 有没有把这条规则往前推，最后看 generation 有没有把那条证据真的写进回答。
这四个检查点的顺序很重要，因为它直接决定你在系统里先改哪一层。很多人会先盯最终答案，但最终答案只能告诉你“错了”，不能直接告诉你“哪里错”。如果你把中间结果记下来，就能把一次失败拆成可以修的局部问题。
一条请求的最小追踪日志复制
```text
query: 报销超过 5000 元需要几级审批？
embedding recall: 规则总览 / 审批表 / 示例
rerank top: 直接写审批级数的片段
prompt context: 只保留最相关证据
answer: 根据资料，超过 5000 元需要...
```


## box soft
### 对应的追查顺序
步骤你在看什么如果这里出错下一步怎么动


召回候选里有没有那条规则答案证据没进来改切分、query 和 TopK
重排正确片段排在什么位置证据进来了但排得靠后缩候选，检查 pair 输入和分数
生成prompt 是否只保留关键证据证据到了上下文，回答还是偏收紧回答约束和解码参数
读这张表时，别把“候选很多”当成“答案更稳”。检索问题里更重要的是证据有没有被稳定传到下一层，以及下一层有没有把它保留下来。

## box
### 为什么这一段要单独拿出来
因为真正的工程排障，往往先看到“系统看起来都在工作，但结果不对”，而问题未必出在某个 API 返回错了。这个时候，你需要的一条能落到日志、候选、分数、上下文和答案上的排查链。只要你能把这条链顺起来，embedding、reranking 和 generation 的边界就不会混。
你可以把它想成四个固定检查点：第一步看 query 有没有被表达成容易检索的样子，第二步看候选里有没有直接证据，第三步看正确片段有没有排到前面，第四步看生成是不是只依赖这些证据。只要其中任何一步断了，后面的回答都会变得不稳。
在真实场景里，最好把这条链写成一份固定记录：query、召回候选、重排分数、最终上下文、回答和下一步动作。这样你比较的就候选有没有更准、排序有没有更稳、回答有没有更贴近证据。
如果你以后碰到别的检索系统，也可以沿用同样的读法：先看有没有召回到证据，再看证据有没有被排上来，再看证据有没有真的喂给生成模型。这个顺序稳定，换的是模型，不换的是检查逻辑。
排错时，最好把每一步的结果写进同一份记录。下次复看时，只要对照这份快照，就能先定位到层，再决定是改 query、重排还是 prompt，而不是同时乱动三层。
- 召回空了，先改切分、query 和 TopK。
- 重排不准，先查 pair 输入和分数。
- 生成跑偏，先收紧上下文和回答约束。
把这三条先放进固定检查单，后面看到同类问题就能直接定位，而不必从头猜一遍。

# Generation：基于材料组织回答 {#generation}
<!-- tag: ROLE 3 -->
<!-- layout: grid2 -->

## box
### 它该做什么
generation 的输入应该是用户问题、经过筛选的上下文和回答要求。它的输出是自然语言答案，也可以是表格、步骤或结构化文本。它擅长组织语言，不擅长在没有材料时保证事实正确。
如果检索材料里没有答案，好的生成行为应该直接承认缺少依据，别凭语气把猜测写成事实。很多系统喜欢把错误归咎于 generation 太弱，但更常见的情况是前面的检索和排序没有把证据送对。

## box soft
### 生成参数最小表
参数控制什么常见误解


`max_new_tokens`新增 token 数把它当总长度
`max_length`通常约束总长度以为它能自动补足证据
`do_sample`是否采样以为关掉采样就能修复事实问题

只根据上下文回答复制

```python
prompt = f"""请只根据给定资料回答。
如果资料不足，请直接说缺少依据。

资料：
1. {final_docs[0]}
2. {final_docs[1]}

问题：{query}
回答："""
answer = generator.generate(prompt, max_new_tokens=120, do_sample=False)
```

# 完整链路：从用户问题到回答 {#chain}
<!-- tag: CHAIN -->
<!-- layout: grid2 -->

## box
### 最小工程动作
1. 把文档切片并预先算好 embedding，存入向量库或本地索引。
2. 用户提问时计算 query embedding，先召回一批候选。
3. 把 query 和候选成对送给 reranker，重排后保留少量高质量片段。
4. 把片段拼成 prompt，再交给 generation 写成回答。
5. 记录 query、候选、重排分数、最终上下文和回答，方便排查。

## box info
### 错误归因顺序
如果回答没引用到关键事实，先看召回候选里有没有答案。没有，问题多半在切分、embedding 或 query 改写；有但排很低，问题更像 reranking；如果候选已经进了上下文但回答仍然错，才看 generation、prompt 和解码参数。

## box
### 一次错误回答的中间结果怎么读
现象优先怀疑什么先看什么


候选里根本没有答案召回或切分query、chunk、embedding、TopK
正确片段在列表里但排很后rerankingquery-document pair、重排分数
正确片段已经进上下文但回答还错generationprompt、上下文顺序、解码参数
最小诊断记录复制
```text
query
candidates with scores
reranked top-N
final prompt context
final answer
```

# 同句案例 {#case}
<!-- tag: CASE -->
<!-- layout: grid2 -->

## box
### 同一句问题，三条路径怎么分工
把同一句“报销超过 5000 元需要几级审批？”分别交给三条路径看，最容易看出边界。embedding 先找“像这类问题”的候选，reranking 再找“真的写了这条规则”的片段，generation 最后只负责把片段写成可读答案。你要看的重点放在它在每一步读了什么、写了什么。
路径读什么写什么中间字段可见吗


embeddingquery 和文档文本向量和候选看不到答案文本，只看近邻
rerankingquery + candidate 对重排分数能看见排序变化，但不直接写答案
generationprompt、上下文和解码参数自然语言回答看得到答案，但看不到候选排序细节

## box
同一句案例：embedding复制
```python
query_vec = encoder.encode("报销超过 5000 元需要几级审批？")
ranked_docs = retriever.search(query_vec, top_k=20)
```

同一句案例：reranking复制
```python
pairs = [(query, doc) for doc in ranked_docs]
scores = reranker.predict(pairs)
final_docs = [doc for doc, score in sorted(zip(ranked_docs, scores), key=lambda item: item[1], reverse=True)[:5]]
```

同一句案例：generation复制
```python
prompt = build_prompt(query, final_docs)
answer = generator.generate(prompt, max_new_tokens=80, do_sample=False)
```


## box warn
### 出错时怎么读这个案例
如果结果像对了但事实不对，先看 reranker 是否把正确片段排进前几条；如果正确片段已经在上下文里，才看 generation 有没有把证据读进去。这个案例的作用，是把“看起来都在工作”拆成具体的一步步检查。

# 误区诊断 {#diagnosis}
<!-- tag: DEBUG -->
<!-- layout: grid2 -->

## box redsoft
### 最常见的六个误区
误区错因后果修法


embedding 相似就是答案正确把语义接近当事实命中找回话题对、证据错的片段保留较大 TopK，再用 reranker 检查直接证据
reranker 可以直接扫全库忽略每个 query-document pair 都要算成本和延迟暴涨先召回，再精排少量候选
生成模型可以替代检索把表达能力当成证据能力模型会把猜测写成事实把资料放进上下文，缺证据时直接说缺少依据
TopK 越大越好以为多给上下文一定更稳噪声挤掉关键证据按问题长度和窗口预算调 K
只看最终回答，不记中间结果把系统当黑盒不知道该改哪一层保存 query、候选、分数、上下文和答案
中英文、代码、表格都用同一种检索策略忽略数据形态结构化信息检不准关键词检索、字段过滤和混合检索一起看

## box
### 先查哪一层
症状最可能的层先查什么修法


候选里没有正确片段embedding / 切分query、chunk、TopK改切分、改召回或改 query
正确片段在列表里但排很后rerankingquery-document pair 和分数缩小候选，再重新排
片段已经进上下文但回答还错generationprompt、上下文顺序、解码参数让回答只依据证据，不让它补猜测

**Q:** 8. 为什么 generation 不能替代检索？

Explain: 好的答案会说：generation 负责组织表达，不负责找证据。如果上下文里没有答案，它仍可能写得很顺，但这只是语言能力，不是事实证明。完整答案最好再补一句：所以要先检索、再重排、再让生成模型基于上下文回答。

### 9 diagnosis

**Q:** 9. 如果召回出来的候选里根本没有正确片段，最先该查什么？

Explain: 应先查切分、query 表达和 embedding 召回设置。错因通常是 chunk 太粗、query 没写准，或者 TopK 太小。后果是正确证据根本没进后面的排序和生成。修法是改切分、改召回或改 query，而不是先改生成模型。

- [ ] optimizer state 、 weight_decay 、 momentum

Answer: truncation 、 max_length 、是否要分块
Explain: 应优先勾选 truncation 、 max_length 和是否要分块。 optimizer state 不相关。好答案还会补一句：先确认证据是不是真的进了窗口，再决定是截断、分块还是重排。

### 12 scenario

**Q:** 12. 你只想确认一个问答 demo 能不能跑通，第一步更应该看什么？

- [ ] 先看候选里有没有答案，再决定要不要加 reranking
- [ ] 先把 generation 调到最强，再看结果
- [ ] 先把 prompt 写长一点，避免漏信息

Answer: 先看候选里有没有答案，再决定要不要加 reranking
Explain: 正确思路是先用 embedding 召回或一个最小的标准链路把路径跑通，再决定要不要拆成更细的 reranking 和 generation。若只是验证路径可用，先把系统搭到能看见候选和回答的程度，比一开始就优化所有细节更重要。

### 13 scenario

**Q:** 13. 你要让模型续写一句说明文字，而不是输出标签。你会换成哪条路径，为什么？

- [ ] generation，因为它按 prompt 续写 token
- [ ] embedding，因为它离输入更近
- [ ] reranking，因为它会重新排序答案

Answer: generation，因为它按 prompt 续写 token
Explain: 应把结果路径换成 generation，因为它会根据 prompt 往后续写 token。分类头只会输出分数，不会帮你继续写文本。答案里最好再说明：这就是为什么“任务目标”先于“模型名字”。

### 14 synthesis

**Q:** 14. 用 6-8 句话解释：一个检索问答系统从输入到输出经历了哪些步骤，以及你会按什么顺序排查一个“能跑但不对”的结果。

Explain: 高质量答案会把步骤串成一条线：先把 query 变成向量，再召回候选，再用 reranker 重新排，最后把少量片段送进 generation。排查顺序应是：先看召回里有没有证据，再看证据有没有排前，最后看证据有没有真的进入 prompt 和回答。结尾最好补一个“能跑但不对”的例子，比如召回太窄、重排太慢或生成时把猜测写成事实。
