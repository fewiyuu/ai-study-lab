# 一、把 prompt 当成接口契约 {#contract}
<!-- tag: contract -->
<!-- layout: grid2 -->

## box
### prompt 先是约定，再是文本
在工程里，prompt 要把话说成可执行的约定。它要告诉程序：模型读哪几块材料、哪些条件不能猜、返回后谁来做最后决定。只要结果会进入下游，就要把任务、输入、约束、输出和失败出口写清。
这页用的词很少：任务、输入、约束、输出、验证。每个词都对应一层读写关系，少一个就会把错藏到下一层。

## box soft
### 一条调用链
`用户需求`一段自然语言
`prompt 模板`任务 + 输入 + 约束
`模型候选`文本或 JSON 草稿
`程序验证`parse / validate / retry
`业务动作`接受、拒绝或转人工


**输入形态**混合说明读的是用户原话和系统规则的混合文本，写不出稳定边界。常见症状是后面每一层都觉得自己该补一点。


**模板形态**任务 + 输入 + 约束 + 输出读的是稳定规则，写的是本次变量。改动集中在数据层，接口本身保持稳定。


**候选形态**文本或结构化数据模型写出的是候选结果，程序还要验证后才能通过。它可以顺口，但不代表可执行。


**验证形态**通过 / 重试 / 拒绝程序读的是原始输出和 schema，写的是下一步状态。格式错、枚举错、缺字段都要在这里截住。


**业务形态**人工复核或外部动作只有验证过的结构才能进入外部动作。这里错了，后面会真的做错事，而不只是“答案不优雅”。

对象读什么写什么副作用或症状


任务业务目标和当前动作单一、可检查的要求输出会把多个动作搅在一起
输入事实材料、偏好、上下文字段化的材料分层模型把偏好当成事实
约束高风险条件、缺失信息、禁做项返回规则和拒绝规则缺证据时还会猜
输出schema、枚举、空值、错误字段结构化结果程序不能稳定解析
验证原始输出和 schema通过、重试、转人工坏结果直接进入下游

# 二、结构化输入：把材料放进正确的格子 {#input}
<!-- tag: input -->
<!-- layout: grid2 -->

## box
### 任务字段：一次只做一件事
任务字段回答“模型现在要做什么”。一条 prompt 最好只放一个主要动作：抽取、判断、改写、排序或总结。多个动作可以串成流程，但不要塞在同一段含混指令里。比如“请读这封邮件，判断情绪，再总结原因，还要写回复话术”，看起来省事，实际上会让错因很难定位。
更稳的写法是把动作拆开。先做分类，再抽取原因，最后生成回复。这样坏输出一出现，你能马上说清是哪一步偏了。

## box
### 事实和偏好分开
上下文是模型工作的材料。事实材料要和风格偏好分开。事实材料包括用户原文、产品说明、历史记录、检索片段；偏好包括语气、长度、受众和禁用词。混在一起时，模型很容易把偏好当事实，也容易把事实当建议。
最常见的做法是给每一块加清楚标题，并用分隔符包住原文。分隔符的作用是让边界更清楚。遇到长文、日志和用户输入时，优先分块，不要把所有东西接成一句。

字段要写什么不要写什么看到什么症状


任务单一动作和成功标准多段并列要求输出难以比较
输入原文、记录、参数把规则混进原文模型读不出事实边界
偏好语气、长度、受众高风险业务决策模型替程序做决定
约束缺失时返回什么只写“不要胡编”缺证据时仍会猜

投诉邮件 triage prompt复制

```text
任务：
从投诉邮件中抽取问题，并判断是否需要人工跟进。

输入：
<email>
{email_text}
</email>

规则：
1. 只使用邮件中出现的信息，不补充未出现的原因。
2. 如果提到退款、账号安全、监管投诉，escalation_level 至少是 human_review。
3. 如果无法判断模块，module 设为 null。

输出：
只返回结构化结果，不要附加解释。
```

# 三、结构化输出：让结果能被程序接住 {#output}
<!-- tag: output -->
<!-- layout: grid2 -->

## box
### 先定 schema，再谈模型回答
结构化输出最常见的是 JSON。它的价值在于字段名、类型和嵌套关系固定，程序可以解析、校验和入库。好的 schema 还会写字段含义、枚举范围、缺失值规则和错误字段。
严格模式要求 schema 足够清楚。这里最实用的几个点是：对象里把 `additionalProperties` 设成 `false`，把字段列进 `required`，再把输入与输出的边界写死。这样做的目的很直接：程序知道什么时候该继续，什么时候该重试。
格式验证只解决“能不能解析”，不解决“内容对不对”。模型可以返回合法 JSON，但分类错了、证据摘错了、理由不充分。格式过了以后，还要继续做业务验证。

## box soft
### 一个更稳的 JSON 约定
字段类型说明校验


`summary`字符串一句话概括问题不能为空
`module`字符串或 null原文明确提到的模块没有就返回 null
`escalation_level`枚举`none`、`human_review`、`urgent`不能自由造词
`evidence`字符串数组原文支持判断的短句不能编造新事实
`error`字符串或 null信息不足或解析失败时说明原因失败要有出口

JSON Schema 片段复制

```json
{
 "type": "json_schema",
 "json_schema": {
 "name": "ticket_triage",
 "strict": true,
 "schema": {
 "type": "object",
 "additionalProperties": false,
 "required": ["summary", "module", "escalation_level", "evidence", "error"],
 "properties": {
 "summary": { "type": "string" },
 "module": { "type": ["string", "null"] },
 "escalation_level": {
 "type": "string",
 "enum": ["none", "human_review", "urgent"]
 },
 "evidence": {
 "type": "array",
 "items": { "type": "string" }
 },
 "error": { "type": ["string", "null"] }
 }
 }
 }
}
```

## box info
### 几个边界
`strict: true` 约束 schema 是否真的生效；`additionalProperties: false` 防止模型临时加字段；`required` 防止关键字段被漏掉。它们管的是“结果长什么样”，不管“结果有没有判断错”。

# 四、状态账本：这一步写了什么副作用 {#operation}
<!-- tag: ledger -->
<!-- layout: grid2 -->

## box
### 把隐形状态摆到台面上
一轮 prompt 调用里，最容易乱的是状态怎么走。输入、候选输出、解析结果、验证结果、人工复核、外部动作都可能变。账本要记录每一步读了什么、写了什么、什么时候会过期、错了会出现什么症状。
这张表还要把副作用单独拎出来。解析和校验只是在本地判断；转人工、发工单、写数据库，才是会影响外部世界的动作。两者混在一起，排错会很慢。

## box soft
### 状态、读写和症状
状态读什么写什么副作用错了的症状


`pending`原始邮件和模板待调用的 prompt无任务、输入混在一起
`called`prompt 和模型版本原始输出无只剩一段文本，没法回看
`parsed`原始输出结构化对象无JSON 解析失败或字段不齐
`validated`对象和 schema可执行结果可能进入外部动作枚举越界、证据为空
`rejected`失败原因重试记录或人工单可能转人工坏结果被继续往下传
`escalated`高风险标记人工处理任务发工单或通知该转人工却还自动通过

状态流转最小示例复制

```python
state = "pending"
raw = call_model(prompt)
state = "called"

parsed = parse_result(raw)
if not parsed["ok"]:
 state = "rejected"
else:
 state = "parsed"
 if validate_business(parsed["data"]):
 state = "validated"
 else:
 state = "rejected"
```

# 五、同一个邮件，怎样走完一轮 {#example}
<!-- tag: example -->
<!-- layout: grid2 -->

## box
### 锚定例子：充值后余额没到账
同一个输入，结构化之后会变得很清楚。假设邮件内容是“充值后余额没到账，客服再不回我就投诉。”这条信息同时包含问题、紧急程度和证据线索。这里要做的是把它拆成可解析的字段，而不是把它写得更像回复。
样例输入复制
```text
充值后余额没到账，客服再不回我就投诉。
```

这个例子里，`summary` 写问题摘要，`module` 写到账或支付相关模块，`escalation_level` 直接升到 `human_review`，`evidence` 保留原文短句，`error` 留给缺失信息或解析失败。

## box soft
### 一条合格输出长什么样
结构化结果复制
```json
{
 "summary": "充值后余额未到账，用户要求尽快处理",
 "module": "billing",
 "escalation_level": "human_review",
 "evidence": [
 "充值后余额没到账",
 "客服再不回我就投诉"
 ],
 "error": null
}
```

原文片段字段作用


充值后余额没到账`summary` / `evidence`说明问题本体
客服再不回我就投诉`escalation_level` / `evidence`说明要转人工
没到账`module`把问题落到支付或到账模块
没有明确时效`error` 或人工复核告诉程序哪里缺材料
同样的写法也能处理低风险请求，只是字段会落到 `none`，不会自动升级。关键是让每一类输入都走稳定的出口。

# 六、读代码、看状态、处理失败 {#workflow}
<!-- tag: workflow -->
<!-- layout: grid2 -->

## box
### 先看输入，再看失败层
这页的排错顺序固定：先看输出有没有进入可解析状态，再看字段和值对不对，最后看业务判断有没有越界。你如果一上来就改模型语气，很容易把问题藏起来；一上来就改业务规则，也可能只是把格式错当成语义错。
这段代码的职责很窄：`json.loads` 只管把字符串变成数据对象，后面的枚举检查、证据检查和业务校验，才是决定结果能不能继续往下走的地方。

## box soft
### 最小解析流程
parse_result复制
```python
import json

ALLOWED_LEVELS = {"none", "human_review", "urgent"}

def parse_result(raw_text):
 try:
 data = json.loads(raw_text)
 except json.JSONDecodeError:
 return {"ok": False, "reason": "json_parse_failed"}

 required = ["summary", "module", "escalation_level", "evidence", "error"]
 missing = [key for key in required if key not in data]
 if missing:
 return {"ok": False, "reason": "missing_fields", "fields": missing}

 if data["escalation_level"] not in ALLOWED_LEVELS:
 return {"ok": False, "reason": "bad_enum"}

 if data["error"] is None and not data["evidence"]:
 return {"ok": False, "reason": "missing_evidence"}

 return {"ok": True, "data": data}
```

输出先发生什么真正问题


前面多了“以下是结果：”JSON 解析失败格式不稳
少了 `error` 字段字段检查失败接口约定不完整
`escalation_level="high"`枚举校验失败schema 太松
有 JSON，但证据是空数组业务验证失败判据和证据脱节

## box warn
### 三种坏结果怎么切
格式漂移
字段缺失
证据为空

[格式漂移]
先查：前言、代码围栏、尾注。
观察：json.loads 直接失败。
别先动：业务规则。
原因：格式没过，后面都不会执行。
[格式漂移]
先查：前言、代码围栏、尾注。
观察：json.loads 直接失败。
别先动：业务规则。
原因：格式没过，后面都不会执行。
[字段缺失]
先查：schema、required、输出模板。
观察：JSON 能解析，但关键字段没回来。
别先动：把结果直接交给下游。
原因：接口没有写清楚，解析通过不等于可用。
[证据为空]
先查：抽取规则、输入边界、回复条件。
观察：结果看起来完整，但 evidence 为空，error 仍是 null。
别先动：扩大模型自由度。
原因：判据和证据脱节时，模型很容易补猜。

重试提示复制

```text
上一次输出没有通过 JSON 解析。
请只返回 JSON，不要加解释、前言或代码围栏。
必须保留字段：summary, module, escalation_level, evidence, error。
如果信息不足，error 要写出原因，其余字段保持可解析。
```

# 七、分流决策：哪种失败走哪条路 {#fallback}
<!-- tag: fallback -->
<!-- layout: grid2 -->

## box
### 先把失败分成四类
一轮调用里，最值钱的“输出失败时走哪条路”。结构化 prompt 至少要准备四个出口：可以继续执行、需要重试、应该转人工、必须拒绝。四条路分别对应不同的失败层。
把出口写清楚以后，程序才知道该怎么处理 JSON 解析失败、字段缺失、证据不足和高风险升级。否则所有坏结果都会被同一种方式对待，后面就很难修。

## box soft
### 四路分流表
失败类型先看什么走哪条路副作用


格式失败JSON 能不能解析重试输入要收紧，重试提示要具体
字段失败required 和枚举重试或拒绝schema 要更明确
证据不足evidence / error转人工或拒绝不能把猜测写成结果
高风险命中升级条件转人工会产生外部动作
低风险可解释字段完整且证据足够继续执行进入下游业务流程

## box
### 同一个输入，三种不同出口
“充值后余额没到账”这类输入，若 JSON 解析失败，先重试；若字段完整但证据不够，转人工；若明确提到投诉和账号安全，直接升级。这个分流顺序很关键：先看格式，再看字段，再看证据，最后看风险。把这条顺序写稳，后面的日志、工单和回归样本才知道自己该记什么，也让 07 页的标签更好写些。
最小分流伪代码复制
```python
parsed = parse_result(raw)
if not parsed["ok"]:
 action = "retry" if parsed["reason"] == "json_parse_failed" else "reject"
elif parsed["data"]["escalation_level"] == "human_review":
 action = "escalate"
elif parsed["data"]["error"] and parsed["data"]["evidence"]:
 action = "continue"
else:
 action = "reject"
```

读这段伪代码时，先问自己一件事：每个分支有没有明确的出口。没有出口的失败，最后就会被默认当成成功，问题会顺着下游扩散。

# 七、错误诊断：先看层，再动手改 {#diagnosis}
<!-- tag: diagnosis -->
<!-- layout: grid2 -->

## box
### 先把症状对回一层
这页最常见的错，某一层的约定没有写稳。JSON 前面多了前言，先查格式层；字段缺了，先查 schema 和 required；证据为空，先查抽取规则和输入边界；高风险请求没有转人工，先查 gate 条件。
排错时先锁一层，再决定改 prompt、改 schema，还是改程序校验。一次改太多，后面很难知道哪一步真的起作用。

## box soft
### 症状到动作
症状先查什么下一步动作不要先动什么


JSON 前有前言输出格式收紧 schema 和重试提示业务规则
关键字段缺失required、模板补字段并重跑下游展示
`evidence` 为空抽取规则、输入边界检查原文是否真的提供证据放宽校验
高风险没转人工gate 条件把升级条件写成明确规则继续调语气
解析通过但内容错业务验证把格式层和语义层分开只看 JSON 是否合法

# 八、把这页接到 Eval {#bridge}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box
### 为什么 07 页会接得上
当你把 prompt 写成接口以后，下一步就自然变成：这些输出在固定样本上是不是稳定。07 页会把本页的字段、枚举、证据和错误字段变成 Eval 样本、判分规则和回归样本。换句话说，06 页解决“怎么让输出可控”，07 页解决“怎么证明它真的更好”。
如果现在已经有一个功能，先把它拆成样本、期望输出和失败标签，再往后看评测。这样你不会把“看起来没问题”误认为“可以上线”。

## box soft
本页能力到 07 页变成什么典型样本看什么


任务字段输入分类和期望行为普通咨询、投诉、退款、信息不足是否把动作说清
输出 schema参考答案或期望结构合法 JSON、必须字段、枚举值格式和字段是否稳定
evidence 字段判分时的支持证据引用是否真的来自输入有没有把猜测写成事实
error 字段失败标签和回归标签解析失败、字段缺失、规则冲突失败能不能复现
retry / fallback坏样本和降级样本高风险请求、资料不足请求该重试还是该转人工

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 作答目标
这组题不考你会不会把 prompt 写得漂亮，而是看你能不能把一次调用拆成任务、输入、约束、输出和验证五层。选择题和填空题先检查边界，开放题再看你能不能把错因、后果和修法说清。

## box info
### 检查重点
先看你有没有把事实和偏好分开，再看输出 schema 是否稳定，最后看你能不能把坏结果反推到具体一层。开放题回答时，优先写“错在哪里、会造成什么、怎么改”。

<!-- end-practice-intro -->

<!-- practice-group: 一、先分清边界 -->
<!-- quick-review: 复习 prompt 负责接口说明，权限系统和业务校验放在程序侧。 -->

### 1 选择

**Q:** 1. Prompt 在 LLM 应用工程里更接近什么？

- [ ] 让模型服从的咒语
- [ ] 调用方和模型之间的接口契约
- [ ] 替代数据库校验的规则

Answer: 调用方和模型之间的接口契约
Explain: 工程里的 prompt 需要固定输入边界、输出格式和失败规则。

### 2 判断

**Q:** 2. 只要模型返回合法 JSON，就说明内容一定正确。

- [ ] 对
- [ ] 错

Answer: 错
Explain: JSON 只保证形状，不能保证判断、证据和业务规则都正确。

### 3 填空

**Q:** 3. 一个稳的 prompt 至少要拆出任务、输入、______ 和输出。

Answer: 约束
Explain: 一个稳的 prompt 至少要拆出任务、输入、约束和输出。

### 4 synthesis

**Q:** 4. 为什么要写清“信息不足时返回什么”？

Explain: 要点有三层：一是信息不足时模型不能靠猜测补完整；二是下游程序要知道这是一个失败分支；三是可以把失败分到重试、人工处理或保守默认值。答到其中两点即可。回答里最好顺手说出一个具体出口，比如 error 、 insufficient_evidence 或 null 。

<!-- end-practice-group -->

<!-- practice-group: 二、读输入和输出 -->
<!-- quick-review: 复习 事实材料、偏好和规则要分开。输出字段要有类型、枚举和缺失值规则。 -->

### 5 填空

**Q:** 5. 如果 `priority` 只能是低、中、高，schema 里应该把它写成 ______，不要让模型自由造词。

Answer: enum,枚举
Explain: 枚举可以限制模型自由造词，让程序更容易校验。

### 6 选择

**Q:** 6. 做风险分类时，最值得放进 few-shot 的通常是哪类样本？

- [ ] 最简单的正例
- [ ] 容易混淆的边界样本
- [ ] 与任务无关但写得漂亮的样本

Answer: 容易混淆的边界样本
Explain: few-shot 最有用的是边界样本，最简单正例信息量不够。

### 7 解释

**Q:** 7. 为什么很多抽取或分类任务要让模型返回 `evidence` 字段？

Answer: 检查判断是否被原文支持,回头核对判断是不是有材料支撑
Explain: evidence 字段让你回头核对判断是不是有材料支撑。

### 8 scenario

**Q:** 8. 用户说“帮我总结这封投诉邮件并判断是否要升级”，你会把 prompt 拆成哪些输入字段？

Explain: 常见拆法是： email_text 放原文， task 说明是总结和升级判断， escalation_rules 说明退款、账号安全、投诉等条件， output_format 说明返回 JSON。答题时要区分事实字段和偏好字段，别把“语气友好一点”混进事实区。

<!-- end-practice-group -->

<!-- practice-group: 三、代码和流程阅读 -->
<!-- quick-review: 复习 先解析，再校验字段和值，最后进入业务动作。失败要走明确分支。 -->

### 9 代码阅读

**Q:** 9. 在上面的 Python 例子里，`json.loads` 负责检查什么？它不负责检查什么？

Answer: 只负责格式解析,不负责内容正确;只检查格式,不检查语义
Explain: json.loads 不能替你判断语义和业务规则。

### 10 代码阅读

**Q:** 10. 如果删除 `escalation_level in ALLOWED_LEVELS` 的检查，可能出现什么问题？

Answer: 模型会临时造词，程序接不住;模型会写出新枚举，程序接不住
Explain: 枚举检查防止模型自由扩写字段取值。

### 11 排序

**Q:** 11. 给一次模型调用排序：A 业务执行，B 填充模板，C 解析 JSON，D 校验字段，E 调用模型。

Answer: B-E-C-D-A
Explain: 先填模板，再调用模型，再解析，之后校验，最后业务执行。

### 12 应用

**Q:** 12. 第一次输出不是合法 JSON，第二次重试提示应该补充什么信息？

Explain: 要点有两层：先明确失败原因，比如“上次输出不是合法 JSON”；再明确下一次必须遵守的格式，比如“只返回 JSON，保留指定字段，不要加前言或代码围栏”。如果你还能补上缺失字段或枚举限制，说明你已经把失败当作接口问题处理，而不是只催模型再来一次。

<!-- end-practice-group -->

<!-- practice-group: 四、错误诊断和迁移 -->
<!-- quick-review: 复习 坏输出要反推到任务、上下文、约束、schema 或程序校验中的某一层。 -->

### 13 diagnosis

**Q:** 13. 模型经常在 JSON 前写“以下是结果：”。这属于哪类问题？你会怎么处理，让程序还能稳定接住？

Explain: 这是输出格式不稳的问题。处理方法是把格式要求写得更具体，必要时切到严格 JSON Schema，让程序只接受可解析结果；同时在重试提示里明确“不要加前言或解释”。如果你只会说“让模型更听话”，还差一步。

### 14 transfer

**Q:** 14. 你已经有一版结构化 prompt，下一页 Eval 会先拿它的哪几类输出做样本？

Explain: 最先会拿普通样本、边界样本、高风险样本和历史坏例做样本。原因很直接：结构化 prompt 已经把接口写出来了，Eval 需要检查看起来稳定的输出在不同风险层级上是不是还稳定。答题时最好顺手点出 summary 、 escalation_level 、 evidence 和 error 这些字段会怎样变成评测材料。
