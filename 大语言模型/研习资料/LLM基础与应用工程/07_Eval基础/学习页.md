# 先抓主线 {#overview}
<!-- tag: 回路 -->

## box soft
Eval 先做四件事：选样本、定规则、跑基线、看失败分布。它的作用把“感觉变好了”变成“能指出哪里变了”。
如果一个改动让平均分高了，但高风险样本掉了，Eval 会先把这件事暴露出来；如果一个改动把输出格式弄乱了，Eval 会先在解析或字段层面报出来，而不是等到后面的业务流程再出错。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

阶段读什么写什么坏了的症状


样本集真实任务、坏例、高风险点固定输入和期望行为只剩顺手挑的样例
被测系统prompt、检索、工具、规则一次输出输出混乱但原因说不清
判分器输出和判分规则分数、标签、解释开放题靠感觉打分
错误分析失败样本和标签错因分布只有平均分，没有修法
基线比较上一轮结果变化结论改动前后没有参照

# 样本和基线 {#mechanism}
<!-- tag: 机制 -->
<!-- layout: grid2 -->

## box
### 样本先按风险分层
最小 Eval 一开始不需要几千条样本。先挑出 20 到 50 条代表性输入就够：普通请求、边界请求、高风险请求和历史坏例。普通样本看主流程，边界样本看会不会乱猜，高风险样本单独看，历史坏例拿来做回归。
样本集的价值不在“多”，而在“能把问题分开”。一条样本最好同时保留输入、期望行为、风险标签和来源线索。这样当结果变差时，你才能看出来是常识误导、资料冲突、格式不稳，还是拒答策略太松。
只保存输入，不写期望行为，后面就会退化成人工浏览。

## box soft
### 样本卡要写哪些字段
字段写什么为什么要留乱写会怎样


`id`稳定编号方便回头查坏例追不回来
`input`原始请求或原始上下文让样本可复现人看不出原始任务
`expected_behavior`期望行为或拒答条件让判分有参照结果只能靠感觉比
`risk`low / medium / high决定看分还是看门槛高风险问题被平均分稀释
`tags`退款、冲突、缺失、引用等方便按错因切片失败只剩一串散点

样本 JSONL 记录复制

```json
{"id":"kb_01","input":"退款多久到账？","expected_behavior":"如果材料里没有明确时效，先说明缺证据，再转人工或要求补充来源","risk":"high","tags":["knowledge_base","missing_info"]}
{"id":"kb_02","input":"怎么改头像？","expected_behavior":"给出明确步骤，避免升级","risk":"low","tags":["how_to","stable"]}
{"id":"kb_03","input":"余额少了，但我也不确定是不是自己花了。","expected_behavior":"先复述不确定点，再要求补证据","risk":"high","tags":["ambiguous","high_risk"]}
{"id":"kb_04","input":"资料冲突时以哪一版为准？","expected_behavior":"说明版本优先级，必要时拒答","risk":"medium","tags":["version","conflict"]}
```

# 指标与判分 {#metric}
<!-- tag: 判分 -->
<!-- layout: grid2 -->

## box
### 不同任务用不同尺子
确定性任务先用硬指标。分类任务可以看 accuracy、precision、recall；抽取任务可以看字段是否缺失、字段值是否越界；JSON 任务可以先看解析率，再看字段完整率和字段正确率。硬指标便宜、稳定、可重复，适合先筛掉格式和明确错误。
开放文本任务先写 rubric，再让 LLM-as-judge 或人工复核去执行。rubric 要写清什么算满足、什么算扣分、什么算失败。LLM-as-judge 可以帮你初筛，但它也会偏爱长答案、礼貌答案或某种写法，所以高风险任务还要抽样人工复核。
平均分只能做第一层筛选，不能单独决定上线。上线前至少要看分布、失败列表和高风险样本。一个系统可以有很高的平均分，同时在少量高损害样本上翻车。

## box soft
### 把分数写成可比较的量
$$
 \text{accuracy} = \frac{\text{passed}}{\text{total}}, \qquad
 \text{high-risk pass rate} = \frac{\text{high-risk passed}}{\text{high-risk total}}
 $$
 `passed` 是通过的样本数，`total` 是全部样本数。第二个式子单独看高风险样本，不让普通样本把关键失败稀释掉。
任务类型先看什么再看什么别只看什么


分类accuracy / recall混淆在哪一类总分
抽取解析率字段完整性、字段值正确性字面格式
RAG 问答答案是否被材料支持引用是否对、缺证据时是否拒答只看回答像不像
改写事实有没有保住语气、长度、约束是否遵守只看是否顺口
RAG 判分说明复制
```text
2 分：直接回应问题，关键事实都能在 context 里找到，引用短句准确。
1 分：大体正确，但漏了一个限制，或者引用不完整。
0 分：context 不支持的内容，或者资料不足时仍然猜测。
```

# 最小流程 {#workflow}
<!-- tag: 流程 -->

先写边界，再写样本
先明确这个功能解决什么，不解决什么。边界决定样本范围，也决定哪些失败必须拦截。比如“根据内部知识库回答产品使用问题”，不包括法律意见、医疗建议和账号敏感操作。
边界写清后，再收普通样本、边界样本、高风险样本和历史坏例。样本少的时候，更要优先覆盖风险，而不是把简单题堆满。


先定判分，再跑基线
能硬判就硬判，不能硬判就写 rubric。第一次跑出来的结果就是基线，后面所有改动都拿它比较。不要先看新版本是否好看，再回头挑样本；那样容易把比较对象改掉。
最小流程只做一件事：把当前版本和基线放在同一批样本上比较，再把失败样本归类。


把失败变成下一轮输入
失败样本要分门别类：解析失败、字段缺失、判断错、证据不支持、拒答过度、拒答不足。错因比单个分数更能指导下一次改动。
新坏例一旦出现，就回流进样本集。Eval 只有被当成回归检查时才真的有用。

最小 Eval 伪代码复制

```python
def run_eval(samples, predict, judge=None):
 rows = []
 for sample in samples:
 raw = predict(sample["input"])
 parsed = parse_output(raw)
 score = judge(sample, parsed) if judge else hard_check(sample, parsed)
 rows.append({
 "id": sample["id"],
 "risk": sample.get("risk", "normal"),
 "score": score,
 "passed": score["passed"],
 "reason": score["reason"],
 })

 baseline = summarize(rows)
 failures = [row for row in rows if not row["passed"]]
 return baseline, failures
```

步骤读什么写什么坏了的症状


收样本真实任务、坏例、风险点固定输入和期望行为只有顺手挑的几个 demo
跑系统当前 prompt / 模型 / 规则一批原始输出每次都在看单条效果
判分输出和规则分数、标签、解释开放题只靠感觉
看失败失败样本和错因分布结论只有总分，没有修法
定回归旧坏例下次必跑的样本集老问题反复出现

# 样本台账 {#ledger}
<!-- tag: 台账 -->
<!-- layout: grid2 -->

## box
### 先看谁在读，谁在写
Eval 里最容易乱的地方，样本、规则、模型版本和回归集的状态同时在变。台账要记录每一步读了什么、写了什么、什么时候会过期、错了会出现什么症状。
当你看见分数波动时，先查这张表，确认是样本换了、规则换了、模型换了，还是回归集没跑。否则很容易把状态变化误认成模型能力变化。

## box soft
### 状态、读写和症状
对象读什么写什么过期信号先查什么


样本卡原始输入、期望行为、风险标签稳定编号、来源线索任务规则变了但样本没改样本版本是否冻结
rubric判分条件、边界、失败项满分、扣分、失败出口不同人打分差很大边界条件是否写实
基线第一次跑出来的结果当前比较参照被改成“更好看”的结果比较对象有没有变
回归集旧坏例和高风险样本每次改动都要重跑的列表老问题隔几轮又出现新坏例是否已回流
这张表的作用很朴素：分数变了，先看状态有没有变。只要台账没对齐，后面的结论就会很滑。

# 落地检查 {#operation}
<!-- tag: 检查 -->
<!-- layout: grid2 -->

## box
### 当你真的要跑一轮 Eval
先锁定样本版本，再锁定 rubric，再冻结基线，最后跑一轮并把失败分层。顺序不能乱：如果先改样本再看分数，比较对象已经变了；如果先看总分再翻失败列表，高风险问题会被平均分盖住。
这一步对应前面的 [基线与回归](#baseline)，也对应练习里的 17a 和 19。答题时要能说清每一步先查什么、先不动什么。

## box soft
### 故障快查表
症状先查什么先别动什么


平均分变好，高风险掉了高风险样本和失败标签先别改判分口径
同一条结果前后不稳定样本版本和 rubric 是否冻结先别补更多样本
旧坏例又出现回归集有没有真跑先别换新的 demo
能解析但业务还错格式层和业务层是否分开先别只看 JSON 是否合法

一轮回归检查单复制

```text
1. 样本版本是否冻结
2. rubric 是否和上次一致
3. 高风险样本是否单独看
4. 失败列表是否按错因切片
5. 新坏例是否回流
```

这一段把 Eval 从“看结果”变成“按顺序查对象”。后面无论做 RAG 还是 Agent，这个顺序都还会再用。

# 案例 {#case}
<!-- tag: 案例 -->
<!-- layout: grid2 -->

## box
### 知识库问答的 8 条起步 Eval
如果你刚开始做知识库问答，不要先追大样本。先写 8 条能把风险分开的样本：2 条明确数字题，2 条条件题，1 条无答案题，1 条版本敏感题，1 条要求引用来源的题，1 条容易被常识误导的题。数量不大，但足够让你看到检索、上下文和生成哪里会坏。
这 8 条样本的目标是先把最容易漏掉的风险放在眼前。后面一旦出现新坏例，再把它补进回归集。

## box soft
### 样本长什么样
样本类型例子它检查什么


明确数字题“退费要多久到账？”检索是否找得到精确值
条件题“什么情况下可以升级人工？”条件是否完整
无答案题资料里根本没有时效说明会不会硬猜
版本敏感题“旧版和新版规则哪个优先？”版本优先级是否清楚
引用题“请给出依据短句。”证据链是否真实
常识诱导题看起来像常识，但资料未写会不会把常识当事实
冲突题不同资料说法不一致能不能说明冲突
高风险拒答题账号安全、退款争议、合规投诉是否及时升级或拒答

样本字段模板复制

```json
{
 "id": "kb_05",
 "input": "用户问：...",
 "expected_behavior": "回答 / 拒答 / 转人工的规则",
 "source_ids": ["doc_a", "doc_b"],
 "risk": "high",
 "tags": ["missing_info", "version", "quote"]
}
```

# 基线与回归 {#baseline}
<!-- tag: 基线 -->
<!-- layout: grid2 -->

## box
### 基线是第一次跑出来的结果
Eval 的基线就是当前版本第一次跑出来的结果，不是理想答案。后续所有改动都要和它比。这样做的重点是把比较条件固定住，不让每次改动都顺手换一套样本或口径。
回归的意思是：旧坏例以后还要继续跑。一次修好不等于以后都好，样本集如果不回流，老问题会以很像新问题的形式重新出现。

## box soft
### 看到不同结果时先看哪里
平均分涨了
均值差不多
旧错复发

平均分从 0.82 升到 0.86，但高风险样本的通过率掉了。
下一步：先看高风险样本和失败列表，再看有没有把简单题做得更顺手。
平均分从 0.82 升到 0.86，但高风险样本的通过率掉了。
下一步：先看高风险样本和失败列表，再看有没有把简单题做得更顺手。
平均分差不多，但边界样本的失败类型变了。
下一步：比较失败标签，而不是只看总分。
旧坏例再次出现，说明回归样本没有真正跑起来。
下一步：把这条样本放回必跑集，并检查是不是改动把旧问题重新打开了。
第一次跑出的结果要保留。不要为了“更好看”回头改基线。

对象第一次跑后续比较错了的症状


基线当前版本第一次结果固定参照比较对象被偷偷换掉
回归样本先收集旧坏例每次都跑老问题偶尔出现、没人发现
高风险门槛单独看通过率和总分并行少量高损害错误被稀释

# 接到下一页 {#bridge}
<!-- tag: 迁移 -->
<!-- layout: grid2 -->

## box
### 从 06 页的结构化输出到 07 页的样本集
06 页已经把输出字段写稳了，07 页就把这些字段转成评测材料。比如 `summary` 可以对应“是否准确概括”，`module` 可以对应“分类是否对”，`escalation_level` 可以对应“高风险是否及时升级”，`evidence` 可以对应“判断有没有材料支持”，`error` 可以对应“失败时是否有明确出口”。
这样做的好处是，Eval 不必重新发明一套输入结构。它直接站在 06 页的输出上，检查结果稳不稳、为什么不稳。

## box soft
### 字段怎么变成评测项
06 页字段07 页怎么评后面 08 页会怎么继续用


`summary`摘要是否准确、是否漏掉关键点RAG 的回答摘要和证据是否一致
`module`分类是否命中正确模块按模块拆分检索路径和答案模板
`escalation_level`高风险是否及时升级决定是否进入人工或拒答分支
`evidence`证据是否真能支撑判断检查检索结果和引用片段
`error`失败时是否说明原因把失败拆到检索、生成或规则层

从结构化输出到 Eval 样本复制

```json
{
 "id": "kb_11",
 "input": "用户问：退款多久到账？",
 "expected_behavior": "如果资料里没有明确时效，先说明缺证据，再转人工",
 "source_ids": ["policy_refund_2026"],
 "risk": "high",
 "tags": ["missing_info", "quote", "escalation"]
}
```

# 错误诊断 {#diagnosis}
<!-- tag: 排查 -->

只看好看的 demo错因：样本是挑顺手的，不是从真实风险里来的。后果：边界样本一多就露馅。修法：保留普通、边界、高风险和历史坏例四层。
只看平均分错因：均值把少量严重错误冲淡了。后果：高风险问题会被藏住。修法：单独看高风险通过率和失败列表。
rubric 写得太虚错因：只写“答案好就高分”。后果：两个人给同一条输出会分得很散。修法：把满分、扣分和失败条件写出来。
样本集只收常见问题错因：常见不等于重要。后果：低频高损害错误没有门槛。修法：把投诉、退款、合规、冲突、缺失信息放进来。
一次通过就不再跑错因：把 Eval 当一次性检查。后果：模型、prompt、资料库一变，旧问题又回来。修法：把坏例放进回归集，每次改动都跑。
离线通过就等于线上稳定错因：忘了离线样本只覆盖已知风险。后果：真实流量里出现新分布时没人预警。修法：离线 Eval 之后接线上监控和坏例回流。

症状可能原因下一步检查


平均分涨了，但高风险样本掉了改动偏向简单题先看高风险分层和失败标签
同一条输出两个人分差很大rubric 太虚把扣分点和边界条件写实
旧坏例再次出现回归集没有跑起来把样本放进必跑集
解析通过，但业务还是错只看格式，不看语义补业务层校验和错因标签
看起来都不错，后面却翻车只收了常见样本补边界、高风险和冲突样本

# 三组易混概念 {#comparison}
<!-- tag: 对照 -->

## box
Eval、监控、指标、rubric、golden set 和人工复核各自负责的事不一样。后面的 RAG 和 Agent 会重新用到这些词，但位置会变。

概念组容易混的地方准确区分工程判断


离线 Eval / 线上监控都在看系统表现离线 Eval 用固定样本比较改动；线上监控看真实流量里的异常上线前跑 Eval，上线后看监控，坏例回流样本集
指标 / Rubric都给结果打分指标是计算规则，rubric 是给人或判分模型看的说明确定性任务先用指标，开放文本任务需要 rubric
Golden set / 全量测试集都包含样本和期望行为Golden set 小而精，先覆盖关键风险；全量测试集更大，适合稳定统计早期先做 golden set，后面再扩充
LLM-as-judge / 人工复核都能评开放文本判分模型便宜，人工复核更稳自动判分用于筛查，高风险样本仍要人工抽样看

## box soft
### 这里先记住的边界
这页先把离线评测讲稳，线上监控和反馈回流只点到为止。RAG 里的“证据支持”和 Agent 里的“工具动作是否成功”会在后面重新出现，到那时再把指标和判分规则接回来就行。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 作答目标
这组题看你能不能把一个 LLM 功能变成可复查的样本集和判分规则。先做选择、判断和填空，再用诊断题和迁移题把错因、后果和修法说清楚。

## box info
### 检查重点
先看你有没有把普通样本、边界样本、高风险样本和历史坏例分开，再看你会不会把平均分和高风险通过率一起看，最后看你能不能把失败样本回流成下一轮回归集。开放题按“错因、后果、修法”答。

<!-- end-practice-intro -->

<!-- practice-group: 一、先认 Eval 在做什么 -->
<!-- quick-review: 复习 Eval 用固定样本和规则比较改动，不等于线上监控，也不等于一次性验收。 -->

### 1 choice

**Q:** 1. Eval 在 LLM 应用里的核心作用是什么？

- [ ] 让系统永远不出错
- [ ] 用固定样本比较系统改动
- [ ] 替代线上监控

Answer: 用固定样本比较系统改动
Explain: Eval 先解决的是“前后怎么比”，不是“让系统永远不出错”。

### 2 true/false

**Q:** 2. 离线 Eval 通过后，线上监控就不重要了。

- [ ] 对
- [ ] 错

Answer: 错
Explain: 离线 Eval 看固定样本，线上监控看真实流量中的表现和异常。

### 3 fill

**Q:** 3. Rubric 指的是用于开放任务判分的 ______。

Answer: 判分量表;判分规则
Explain: Rubric 写的是什么算满分、扣分和失败。

### 4 synthesis

**Q:** 4. 为什么失败样本要进入回归样本集？

Explain: 失败样本回流可以防止旧问题反复出现，也能让问题可复现、可定位。好的回答会补一句：回归样本约束下一次改动。只说“以后还能测”还不够，要点出它和修法之间的关系。

<!-- end-practice-group -->

<!-- practice-group: 二、样本和指标怎么选 -->
<!-- quick-review: 复习 先分层样本，再选指标。确定性任务优先硬指标，开放任务用 rubric 或 LLM-as-judge。 -->

### 5 fill

**Q:** 5. 为客服分类器设计一条 Eval 样本，至少应包含哪些字段？

Answer: input;expected_behavior;risk;tags
Explain: 样本至少要能复现任务、说明期望行为、标出风险和标签。

### 6 choice

**Q:** 6. RAG 问答除了答案是否正确，还要看哪两件事？

- [ ] 答案是否被材料支持，引用是否准确
- [ ] 回答是否足够长，语气是否礼貌
- [ ] 格式是否花哨，排版是否整齐

Answer: 答案是否被材料支持;引用是否准确
Explain: RAG 不能只看答案像不像，还要看证据链。

### 7 choice

**Q:** 7. JSON 抽取任务最先应该检查什么？

- [ ] 回答是否有礼貌
- [ ] 解析率、字段完整性、字段值正确性
- [ ] 回答是否足够长

Answer: 解析率;字段完整性;字段值正确性
Explain: 结构化任务先看格式和字段，再看业务正确性。

### 8 scenario

**Q:** 8. 给一个你认为必须加入客服 Eval 的高风险样本，并说明检查点。

Explain: 一个合格答案通常会包括：高风险样本是什么、为什么高风险、应该看什么门槛。比如退款争议、账号安全、合规投诉、数据丢失，检查点会是是否及时升级、是否拒答、是否保留证据链。只写类别名不够，必须把检查点说清。

<!-- end-practice-group -->

<!-- practice-group: 三、读代码和读结果 -->
<!-- quick-review: 复习 失败列表和错因标签比总分更能指导修法。 -->

### 9 code-reading

**Q:** 9. 上面的伪代码里为什么要返回 `failures`，而不是只返回 accuracy？

Explain: failures 让你知道错的是哪条样本、哪类风险、哪种错因。只返回 accuracy，后面只能看到一个数字，看不见修法。合格答案至少要说出“定位样本”和“分类错因”两件事。

### 10 code-reading

**Q:** 10. 样本里的 `risk` 字段有什么用？

Answer: 单独看高风险样本;把高风险问题从总分里拆出来
Explain: risk 字段用来单独看高损害样本，避免被平均分稀释。

### 11 diagnosis

**Q:** 11. 新版本平均分从 0.82 涨到 0.86，但投诉类样本从 90% 降到 60%。你会直接上线吗？为什么？

- [ ] 会
- [ ] 不会

Answer: 不会;不会直接上线
Explain: 总分涨了不代表高风险样本也稳了。

### 12 synthesis

**Q:** 12. 为什么每次改动都要和固定基线比较？

Explain: 基线是固定参照，不是理想答案。它的作用是保证比较对象不变，让你看出改动到底是提升、退化，还是只是换了一批更容易的样本。合格答案至少要说出“比较对象一致”和“能发现退化”两点。

<!-- end-practice-group -->

<!-- practice-group: 四、错误诊断和迁移 -->
<!-- quick-review: 复习 离线 Eval 先看样本和指标，线上监控和坏例回流是下一步。 -->

### 13 diagnosis

**Q:** 13. 团队只用 5 条最好看的样例展示新 prompt。错在哪里？

Explain: 错在样本是挑顺手的，不是从真实风险里来的。后果是边界和高风险样本完全没被检验。修法是把普通、边界、高风险和历史坏例四层都保留。能把这四层说清，就算答对。

### 14 diagnosis

**Q:** 14. Rubric 只写“答案好就给高分”。会导致什么问题？怎么改？

Explain: 这样会让不同人、甚至同一人前后打分都不稳定。改法是写清满分、扣分和失败条件，尤其是缺证据、错引用、越权猜测这些边界。答题时别只说“写具体一点”，要点出判分维度。

### 15 diagnosis

**Q:** 15. 如果所有开放题都只让判分模型打分，不抽样人工看，风险是什么？

Explain: 风险是判分模型会偏爱某种写法，或者把长答案、礼貌答案误当成高质量答案。补救方法是抽样人工复核，把自动判分和人工判断对齐。至少要写出“偏差”和“抽样复核”两点。

### 16 diagnosis

**Q:** 16. 产品规则变了，但 Eval 样本和期望答案没改，会产生什么后果？

Explain: 后果是旧标准继续当作真标准，结果会把正确改动误判成失败，或者把应该失败的改动误判成通过。修法是更新样本和期望行为，并重新跑基线。答题时要说清“标准变了，样本也要变”。

### 17a transfer

**Q:** 17a. 你改了客服总结 prompt，先比较哪三类样本？

- [ ] 普通样本
- [ ] 边界样本
- [ ] 高风险样本
- [ ] 随机广告文案

Explain: 先看普通样本、边界样本和高风险样本，最好再加历史坏例。随机广告文案和任务无关，不该进这题。

### 17 application

**Q:** 17. 你改了客服总结 prompt。上线前至少要比较哪三类结果？

Explain: 至少要比较普通样本、边界样本和高风险样本，最好再补历史坏例。比较时别只写“看整体分数”，要顺手说明每类样本看什么：普通样本看主流程、边界样本看是否乱猜、高风险样本看是否及时升级或拒答。

### 18 interview

**Q:** 18. 用 2-3 句话区分离线 Eval 和线上监控。

Explain: 好的回答会说：离线 Eval 用固定样本比较改动，适合上线前做回归检查；线上监控看真实流量中的异常，适合上线后发现新分布或新坏例。两者前后接力。答题时要把“固定样本”和“真实流量”说出来。

### 19 scenario

**Q:** 19. 为什么高风险样本可以单独设门槛，而不只服从总平均分？

- [ ] 因为高风险问题不能被少量普通样本稀释
- [ ] 因为高风险样本一定比较简单
- [ ] 因为总平均分一定没有意义

Answer: 因为高风险问题不能被少量普通样本稀释
Explain: 高风险样本要单独设门槛，避免被平均分稀释。

### 20 checklist

**Q:** 20. 如果你要真的跑一轮 Eval，应该先锁定哪五件事？

Explain: 合格答案通常会包含：样本版本、rubric、基线、高风险单独看、失败按错因切片、新坏例回流。更好的回答还会说出顺序：先冻结样本和 rubric，再跑基线，再看失败分层，最后回流回归集。
