# 学习目标 {#goals}
<!-- tag: GOALS -->
<!-- layout: grid2 -->

## box soft
### 先把主线抓稳
RAG 的核心把资料变成可检索证据，再把证据交给生成模型。离线段负责整理文档，在线段负责回答问题。两段都要能打印中间结果，后面才好排错。
这页只管最小闭环，不管大而全的系统工程。你先学会怎么查证据、怎么保留来源、怎么识别证据没进上下文，再谈复杂编排。

## box
### 本页的边界
- 先不追完整的多轮对话记忆。
- 先不追复杂的 query rewrite 和多路路由。
- 先不追专门的评测平台，把关键日志和样例保住就够。

# 先抓主线 {#overview}
<!-- tag: MAP -->

原始文档→切块→向量化→索引→检索→重排→上下文→回答

这条链路里，前半段把资料变成可搜的对象，后半段把候选证据变成可读的回答。只要你记住“证据先被选出来，再被交给模型”，就不会把 RAG 误成单纯的向量库。

## box info
### 一眼看懂的判断
如果结果错了，先问“证据有没有进候选”，再问“候选有没有进最终上下文”，最后才问“模型有没有照着证据说”。

# 核心机制 {#mechanism}
<!-- tag: MECHANISM -->
<!-- layout: grid2 -->

## box
### 离线段做什么
离线段接收的是文档。它要抽取正文、按语义边界切块、给每个 chunk 加 metadata、算向量、写索引。metadata 通常至少包括来源文件、章节标题、页码或更新时间。没有这些信息，后面的引用和排错都会变钝。
chunk 粒度要平衡两件事：太小会丢上下文，太大又会让检索变模糊。真正要看的“切完之后还能不能回答原问题”。

## box
### 在线段做什么
在线段接收的是用户问题。问题先向量化，再去索引里找候选，必要时重排或过滤，然后把最相关的证据按顺序拼进 prompt。生成模型并不负责找证据，它只负责基于已经给到的证据生成回答。
所以在线段的关键“前面的步骤有没有把对的证据送进来”。

| 阶段 | 读什么 | 写什么 | 错了会怎样 |
| --- | --- | --- | --- |
| 文档抽取 | 原始 PDF、Markdown、FAQ | 清洗后的正文 | 目录、页脚、噪声混进 chunk |
| 切块 | 正文和标题层级 | chunk + overlap | 条件和答案被切断 |
| 向量化 | chunk 文本 | embedding 向量 | 语义近似找不到候选 |
| 检索 | query 向量 + 索引 | TopK 候选 | 正确证据没进候选 |
| 重排 | 候选 + 过滤条件 | 最终上下文顺序 | 有证据但被排到后面 |
| 生成 | 问题 + 上下文 | 回答 + 来源 | 明明有证据却答错 |

# 易混概念 {#compare}
<!-- tag: COMPARE -->

| 概念 | 前者 | 后者 | 判断方法 |
| --- | --- | --- | --- |
| RAG / 微调 | 先找外部资料，再基于资料回答 | 更新模型参数，让模型学到稳定行为 | 知识常变、要引用来源，优先 RAG |
| chunk / context | 索引里的可检索片段 | 一次请求里真正交给模型阅读的材料 | chunk 是候选，context 是当前可读证据 |
| 召回失败 / 生成失败 | 正确证据没进候选或上下文 | 上下文里有证据，但回答没按证据来 | 先看候选，再看上下文，最后看回答 |
| 语义检索 / 关键词检索 | 擅长近义表达和概念相似 | 擅长函数名、错误码、数字和符号 | API 名、编号、错误码常要混合检索 |

# 检索策略怎么选 {#strategy}
<!-- tag: CHOICE -->
<!-- layout: grid2 -->

## box
### 语义检索、关键词检索、混合检索
语义检索适合问法和文档措辞不一样的情况，像“谁负责审批”“这条制度怎么走”这类问题通常更适合它。关键词检索适合函数名、错误码、编号、产品型号、专有术语这些需要精确命中的材料。混合检索则把两者合起来，用语义覆盖意思相近的问法，用关键词处理不能错的字段。
你不需要把三种方法背成名词表，真正要记的是判断轴：知识是不是常变、符号是不是必须精确、问题是不是会出现同义改写。制度类问题常常既需要语义，也需要精确词；API 文档更是这样。只靠一种检索方式，很多时候不是慢，而是漏。

## box
场景优先策略原因

公司制度语义 + 关键词既有条款意思，也有编号和更新时间
API 文档关键词 + 语义函数名、错误码和参数值不能错
课程笔记语义为主问法通常比原文更口语化
故障手册混合检索既要找症状，也要抓精确报错

# 常见误区 {#mistakes}
<!-- tag: MISTAKES -->
<!-- layout: grid2 -->

## box warn
### 误区 1：RAG 等于向量库
错因是把索引组件当成完整系统。后果是只解决候选搜索，没有解决上下文选择、引用和评估。修法是把整条链拆成文档处理、切块、检索、重排、上下文和生成。

## box warn
### 误区 2：文档越多越直接塞
错因是看到长上下文就想跳过检索。后果是噪声多、成本高、引用难查。修法还是先选证据，再把少量高质量片段交给模型。

## box warn
### 误区 3：chunk 固定长度就够
错因是只按字符数切，不看语义边界。后果是答案和条件被拆开。修法是按标题、条款和段落切，再用少量 overlap 处理边界。

## box warn
### 误区 4：没有无答案处理
错因是所有问题都强行生成答案。后果是模型在资料缺失时编造。修法是让 prompt 明确资料不足时直接承认不足，并把无答案问题纳入评估。

# 状态台账 {#ledger}
<!-- tag: LEDGER -->
<!-- layout: grid2 -->

## box
### 证据流转台账
步骤读写风险


chunk正文 + 标题片段文本 + metadata没有来源，就不能回指原文
indexchunk 向量检索结构更新没同步时会命中旧答案
retrievequery + 索引候选列表召回不足会把答案挡在门外
pack候选 + 预算最终上下文塞太多会引入噪声
generate问题 + 上下文答案 + 引用不守证据会出现幻觉

## box soft
### 查错时先看哪一层
1. 候选里有没有正确 chunk。
2. 正确 chunk 有没有进最终上下文。
3. 上下文里有没有足够的来源标识。
4. 生成时有没有被额外提示或顺序带偏。
这张台账的作用，是把“感觉不对”变成“哪一步写错了”。只要每一步能记录下来，后面就能复查。

# 例子 {#example}
<!-- tag: EXAMPLE -->
<!-- layout: grid2 -->

## box
### 教学例子：报销制度
原文里写着“超过 5000 元需要直属主管和部门负责人两级审批”。离线段切块时，这句话要和制度标题、页码或更新时间绑在一起。在线段里，用户问“5000 以上报销谁批”，检索应该把这条 chunk 找出来，再放进最终上下文。
理想回答带着来源标识说明“我依据哪一条制度回答”。如果最后没有来源，问题通常不在措辞，而在检索、重排或上下文拼接。
步骤输入输出失败时看什么

切块报销制度正文包含“两级审批”的 chunk是不是把条件切掉了
检索“5000 元以上报销谁批”候选 chunk 列表有没有把这条规则召回
拼上下文TopK 候选按顺序排列的证据区正确 chunk 是否被噪声盖住
生成问题 + 证据区带来源的答案模型是否守住了材料

## box
### 最小 prompt 约束
```text
只根据资料回答。
资料不足时直接说资料不足。
回答后列出使用的来源编号。
不要把常识补成制度。
```

这个约束不是装饰。它的作用是把“凭记忆补全”压回“受证据约束的回答”。
检索后如何写入 prompt复制
```text
问题：
5000 元以上报销谁批？

资料：
[1] 超过 5000 元需要直属主管和部门负责人两级审批。
[2] 普通报销由直属主管审批。

要求：
只根据资料回答，给出来源编号。
```

# 排错顺序 {#rag-debug}
<!-- tag: DEBUG -->
<!-- layout: grid2 -->

## box
### 四类失败最常见
- 正确证据没被召回。
- 证据召回了，但被切坏了。
- 证据进了上下文，但排在错误位置或被噪声淹没。
- 上下文已经有证据，模型还是没有按证据回答。

## box
### 最小排查日志
日志项为什么要记没记会怎样

query知道问题到底问了什么不知道是检索错还是提问错
TopK 候选看证据有没有被召回只能猜检索层出了什么问题
最终上下文看证据有没有真正送进模型不知道是召回错还是拼接错
回答与来源看答案是否忠于材料无法做回归比较

# 同一条问题的回放链路 {#walkthrough}
<!-- tag: TRACE -->
<!-- layout: grid2 -->

## box
### 从 query 到答案，哪里最容易跑偏
拿“5000 元以上报销谁批？”这句来回放，最先看到的不是答案，而是问题本身有没有被正确理解。问题太短时，系统会依赖检索去补上下文；问题太长时，系统会先丢掉一部分问法，只留下更像关键词的那一段。两种情况都说明，RAG 一串会受到长度、顺序和预算影响的操作。
第一步，query 会被转成 embedding，并在索引里找候选。这里真正有价值的相似度对应的片段有没有包含完整条件。第二步，候选会进入重排。重排不是把“更像”的排前面那么简单，它还要处理标题、来源、新旧版本和过滤条件。第三步，候选被拼进上下文后，生成模型只看得到这部分材料，所以最后回答是否守住证据，很大程度上取决于上下文有没有把正确片段放在模型注意得到的位置。
如果你把这条链看成一份回放记录，就会发现一个很实用的排错原则：先看证据有没有被找出来，再看有没有被放进最终上下文，最后才看模型有没有按材料回答。很多“模型胡说”的问题，其实在上游就已经决定了。

## box
### 一条最小回放日志
RAG 回放日志复制
```text
[query] 5000 元以上报销谁批？
[retrieve] top1: 普通报销由直属主管审批。
[retrieve] top2: 超过 5000 元需要直属主管和部门负责人两级审批。
[rerank] top1: 超过 5000 元需要直属主管和部门负责人两级审批。
[pack] context = [规则条款, 来源文件, 更新时间]
[generate] 5000 元以上报销需要直属主管和部门负责人两级审批。来源：[1]
```

这条日志看起来很普通，但它把每一步的输入和输出都暴露出来了。真正做排错的时候，你要做的不是猜，而是把错误停在哪一步写出来。

| 步骤 | 观察点 | 正常时 | 异常时 |
| --- | --- | --- | --- |
| query | 问题是否明确 | 知道问的是哪个条款 | 问题太模糊，召回只能靠运气 |
| retrieve | 正确 chunk 是否出现 | 条款类 chunk 进入候选 | 关键条款没有被召回 |
| rerank | 正确 chunk 是否排前 | 更相关的片段排到前面 | 旧版本或噪声排在前面 |
| pack | 上下文是否过满 | 只保留最关键证据 | 噪声把条款挤出去 |
| generate | 回答是否守证据 | 答案带来源，语义和材料一致 | 回答顺了，但依据错了 |

# 版本回放案例 {#replay-case}
<!-- tag: CASE -->
<!-- layout: grid2 -->

## box
### 旧版和新版都在库里时，为什么会答错
想象一个课程笔记库有两个版本：旧版说“RAG 用向量相似度找相关片段”，新版补上了“还要保留来源、更新时间和回答引用”。用户问的是“为什么 RAG 要保留 metadata”。如果检索只看语义相似度，旧版和新版都会进候选，因为两份资料都在讲 RAG 的结构。问题就在这里：语义相似不等于版本正确，也不等于最适合当前问题。
离线段里，chunk 的 metadata 不只是装饰。它告诉系统这个片段来自哪份文档、哪一版、哪一页、什么时候更新。在线段里，retrieval 先把候选找出来，rerank 再决定谁排前面，pack 再把少量片段塞进上下文。只要其中任何一步忽略了版本，答案就可能引用旧材料。你看到的表面症状是“回答不对”，根因常常是“旧版本进了上下文”。
这个案例特别适合训练排错顺序：先看候选里是不是混进了旧版，再看重排有没有把旧版压下去，再看最终上下文有没有把新版本留下。只要你能把这三层说清楚，就知道该改的是索引、metadata、rerank，还是上下文预算。

## box
版本过滤骨架复制
```text
def pick_chunks(candidates, query, cutoff):
 fresh = []
 for chunk in candidates:
 if chunk.meta["updated_at"] >= cutoff:
 fresh.append(chunk)
 ranked = rerank(fresh, query)
 return ranked[:3]

chunks = pick_chunks(candidates, question, "2026-01-01")
```

症状更像哪一层先查什么

总是答旧规则索引 / metadatachunk 是否保留 updated_at 和版本号
旧版排在新版前面rerank重排有没有把版本和时效纳入判断
回答没引用pack / generate最终上下文有没有来源编号
问法一变就答偏retrievequery 改写和候选覆盖够不够
你可以把这段代码当成一个很小的防线：先把旧版本挡在外面，再把候选压到最少，最后才让模型开口。它的价值让你知道“版本感知”要落在哪一步。

## box info
### 练习时怎么用这段案例
如果你答题时只会说“要加 metadata”，还不够。你还要能补一句：metadata 放在哪里、谁来读它、谁来用它决定排序，以及没有它时会出现什么症状。能把这四个问题答出来，才算真的会做最小 RAG。

# 资料不足时怎么答 {#missing-answer}
<!-- tag: FALLBACK -->
<!-- layout: grid2 -->

## box
### 真正该学会的“先停下来”
很多 RAG 系统最难的没有答案时怎么答。比如用户问“下个月报销额度会不会变”，但知识库里只有上个月的制度和一份草案。这个时候，系统如果硬回答，就会把旧规则说成新规则；如果直接拒答，用户又会觉得系统没用。最小流程的正确做法是先把缺口说清楚：材料里没有新版本，当前检索到的只是旧制度或草案，系统不能把它当作最终答案。
这个动作看起来像“不会答”，其实是在保护系统的可信度。你可以继续给出下一步建议，比如“请补充最新制度”“请确认文档版本”，但前提是先把资料缺口和来源不确定性说明白。RAG 的底线资料不够时不乱编。

## box
资料不足时的模板复制
```text
当前检索到的资料不包含这个问题的最新答案。
我找到了旧版制度 / 草案，但没有找到可以确认的正式版本。
如果你需要准确结果，请补充最新文档或确认版本号。
```

情况该怎么说不要怎么做

只有旧版本说明找到的是旧材料把旧版本说成当前规则
找到草案说明草案不能当最终答案把草案当正式制度
完全没找到说明资料不足凭常识补一个看似合理的答案

# 最小实现桥 {#implementation}
<!-- tag: OPS -->
<!-- layout: grid2 -->

## box
### 最小 RAG 的操作桥
从文档到答案的最小链复制
```text
docs = load_docs()
chunks = chunk_docs(docs)
index = build_index(chunks)

question = "5000 元以上报销谁批？"
candidate_chunks = retrieve(index, question, top_k=5)
selected_chunks = rerank(candidate_chunks, question)[:3]
context = pack_context(selected_chunks, token_budget=1200)
answer = generate(question, context)
print(answer)
```

这段代码里，真正值得盯的是每一步写出了什么：chunk、index、候选、上下文和答案。只要中间状态能看见，问题就能定位。
代码行读写漏掉后的症状

`chunk_docs`文档正文chunk + metadata无法回指来源
`build_index`chunk 向量索引结构检索不稳定或查不到旧版本
`retrieve`query + index候选列表正确证据进不了候选
`pack_context`候选 + 预算prompt 证据区噪声挤掉关键片段
`generate`问题 + context答案 + 引用上下文有证据但回答没守住

## box
### 实现时最容易漏的两件事
1. 来源 metadata 没进 chunk 结构，后面只能拿到一段文本。
2. 上下文预算没显式控制，结果是候选太多、噪声太大、回答失焦。
$$\text{context budget} = \text{question} + \sum_{i=1}^{k}\text{chunk}_i + \text{source tags} + \text{answer reserve}$$
 这条式子不要求你算出精确 token 数，只要求你知道预算是分配出来的，不是“检索到了就全塞”。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-group: 一、概念和边界 -->
<!-- quick-review: REMIND RAG 的第一职责是把证据接进回答。 -->

### 1 单选

**Q:** 1. RAG 最准确的描述是哪一个？

- [ ] 先检索外部资料，再基于资料生成回答
- [ ] 把所有资料微调进模型
- [ ] 只要接入向量库就完成

Answer: 先检索外部资料，再基于资料生成回答

### 2 判断

**Q:** 2. chunk 只需要保存文本，不需要保存来源信息。

- [ ] 对
- [ ] 错

Answer: 错

### 3 简答

**Q:** 3. 为什么 chunk 太小和太大都可能出问题？

Explain: 太小会切断条件和结果，检索到也不完整；太大又会让语义变钝、候选噪声变多、上下文预算变紧。答到“上下文被切断”和“检索变模糊”就算抓住了要点。

### 4 填空

**Q:** 4. 一次请求中真正放进 prompt 给模型阅读的材料，通常叫 `________`。

Answer: context

<!-- end-practice-group -->

<!-- practice-group: 二、流程排序 -->
<!-- quick-review: REMIND 先建索引，再回答问题。 -->

### 5 排序

**Q:** 5. 给离线动作排序：文档抽取、切分、计算 embedding、保存索引、添加 metadata。

Explain: 合理顺序一般是：文档抽取 → 切分 → 添加 metadata → 计算 embedding → 保存索引。关键说明 metadata 要和 chunk 一起进入索引。

### 6 排序

**Q:** 6. 给在线动作排序：用户问题、query embedding、召回、重排、拼 prompt、生成回答。

Explain: 合理顺序一般是：用户问题 → query embedding → 召回 → 重排 → 拼 prompt → 生成回答。若强调排错，也可以补一句“先看召回，再看上下文，再看生成”。

### 7 解释

**Q:** 7. 为什么最小 RAG 也应该保留来源编号或文件名？

Explain: 因为引用能让回答回指原文，排错能追到具体 chunk，资料更新时也知道该重建哪一段索引。没有来源，回答只能算“像是对的”。

### 8 边界

**Q:** 8. 为什么评估集里要包含“资料中没有答案”的问题？

Explain: 因为真实系统会遇到资料缺口。只有把无答案问题放进评估，才能检查系统是否会承认缺口，而不是硬编一个看起来顺的答案。

<!-- end-practice-group -->

<!-- practice-group: 三、排错诊断 -->
<!-- quick-review: REMIND 先看证据有没有进入上下文。 -->

### 9 诊断

**Q:** 9. 最终回答错了，召回 TopK 里没有正确片段。问题可能在哪几层？怎么查？

Explain: 先查文档抽取和切分，再查 query 表达和 embedding，再查索引更新是否同步。因为正确证据没有进候选，生成层通常不是第一责任人。

### 10 诊断

**Q:** 10. 正确制度条款被切成两块，单块都无法回答完整问题。怎么修？

Explain: 优先按标题、条款、段落切，再给边界加少量 overlap。修法的目标让答案依赖的条件和结论能一起进入候选。

### 11 诊断

**Q:** 11. 最终上下文里有正确证据，但模型回答成另一个数字。你会改哪里？

Explain: 先看上下文顺序、是否有噪声干扰、prompt 是否要求只根据资料回答，再看生成参数和模型本身的稳定性。此时问题通常在生成层或上下文组织，而不是检索层。

### 12 诊断

**Q:** 12. 系统能回答，但无法给出处，也难以查错。设计上漏了什么？怎么补？

Explain: 漏的是 chunk 级 metadata 和回答级来源记录。补法是让索引里保留来源文件、章节、页码、更新时间，并在回答里输出所用来源编号或文件名。

<!-- end-practice-group -->

<!-- practice-group: 四、场景迁移 -->
<!-- quick-review: REMIND 要先说资料形态，再说方案。 -->

### 13 场景

**Q:** 13. 公司制度每月更新，RAG 和微调哪个更适合作为第一方案？为什么？

Explain: 通常先用 RAG，因为制度会变，检索最新材料比反复改模型参数更直接。只要答案需要依据和来源，RAG 比微调更容易维护。

### 14 场景

**Q:** 14. API 文档里有大量函数名和错误码，为什么可能要混合关键词检索？

Explain: 因为函数名、错误码和参数值往往要求精确匹配，纯语义检索可能不稳。混合关键词检索可以补住专有名词、数字和符号型信息。

### 15 应用

**Q:** 15. 一本 200 页手册做 RAG，你会怎样设计 chunk 的 metadata？

Explain: 至少要有文档名、章节名、页码范围、版本号或更新时间。若文档会分版发布，还可以加产品型号、地区或适用范围，方便过滤和回溯。

### 16 清单

**Q:** 16. 一次 RAG 请求至少记录哪 6 项，才能方便排错？

Explain: 至少记录 query、TopK 候选、最终上下文、回答文本、来源标识、索引版本或更新时间。这样才能判断问题出在召回、拼接还是生成。

<!-- end-practice-group -->

<!-- practice-group: 五、面试追问 -->
<!-- quick-review: REMIND 回答时不要把 RAG 简化成“向量数据库”。 -->

### 17 面试

**Q:** 17. 面试官问：什么时候用 RAG，什么时候考虑微调？

Explain: 知识频繁变化、需要来源和引用时，先用 RAG。要改的是稳定行为、格式或长期风格时，再考虑微调。答案里最好说出“知识更新频率”这条判断轴。

### 18 面试

**Q:** 18. 面试官问：chunk size 怎么影响 RAG 效果？

Explain: chunk 太小会丢上下文，太大又会让语义变钝、命中率下降。更好的说法是：chunk size 影响召回、上下文预算和答案完整性，需要根据文档结构调整。

### 19 面试

**Q:** 19. 面试官问：RAG 能完全消除幻觉吗？为什么？

Explain: 不能完全消除。RAG 只是把回答约束到外部证据上，证据选错、上下文组织差、模型不守约束时仍会出错。好的回答应同时讲到检索错误和生成错误。

### 20 面试

**Q:** 20. 面试官问：你会怎样评估一个最小 RAG 系统？

Explain: 准备一组有标准来源的问题，分开看召回是否命中、最终上下文是否含证据、回答是否忠于证据、无答案问题是否能拒答。要能说明“哪一层坏了”比只报一个总分更重要。

<!-- end-practice-group -->

<!-- practice-group: 六、完整表达 -->
<!-- quick-review: REMIND 必须串联：文档切分、embedding、索引、召回、重排、上下文、prompt、generation、引用、评估。 -->

### 21 费曼解释

**Q:** 21. 用 7-10 句话解释：一个最小 RAG 系统如何从文档变成有依据的回答。

Explain: 答案里至少要交代离线段、在线段、来源、排错和一种失败方式。能把“文档切块、embedding、索引、召回、重排、上下文、生成、引用、评估”串起来，就说明主线已经通了。

### 22 设计

**Q:** 22. 设计一个“课程笔记问答”的最小 RAG：文档如何切、metadata 存什么、如何检索、如何生成、如何评估。

Explain: 好的答案会先说文档结构，再说切块和 metadata，再说检索和上下文拼接，最后说怎么用样本、来源和无答案问题做评估。重点是把方案写成一条链，而不是一堆组件名。
