LLM 基础 09｜Agent 最小流程：让工具循环能停下来

这一页把 Agent 讲成一个受控循环，避开“让模型自己想办法”的黑盒想象。你会沿着同一个订单查询例子看清：模型只提出下一步动作，程序负责校验工具、执行工具、记录观察和判断是否停止。学完后，你应该能设计一个小工具集，也能看日志判断 Agent 是卡在计划、工具、状态、权限，还是停止条件。

学习目标

goals

先把 Agent 缩到能检查的一步

这节课不从“自主智能体”这种大词开始。工程上更稳的入口是一个小循环：给定目标和状态，模型选择一个结构化动作；应用代码校验动作和权限；工具执行后返回观察；系统更新状态，再决定继续、澄清、人工确认或停止。

08 页的 RAG 链路是固定路径：检索、拼上下文、生成回答。Agent 在这里多出“路径选择”这一层。它可以根据观察决定下一步查订单、查政策、问用户，或者直接回答。路径一旦可变，就必须把工具边界、状态和停止条件写清楚。

本页不追的内容

  • 不讲多 Agent 协作和复杂 handoff，只保留单个 Agent 的闭环。
  • 不讲长期记忆系统，只讲当前任务需要的短期状态。
  • 不绑定某个框架，把 SDK、planner、浏览器自动化放到后续专题。
  • 不让模型直接执行高风险动作，程序侧始终保留权限和确认。

先抓主线

map

目标→状态→模型选择动作→程序校验→工具执行→观察→继续或停止

Workflow

路径由程序固定安排。模型只在某个步骤里填答案或提取字段。优点是可控，风险主要来自每一步质量。

Agent

模型根据目标、状态和观察选择下一步。风险多了路径选择、重复调用、越权动作和无法停止。

核心机制

mechanism

模型提出请求，程序执行动作

Agent 里的模型不直接联网、查库、发邮件。模型输出的是结构化动作，例如 ask_user、call_tool、final。应用代码负责解析动作、检查 schema、校验权限、调用工具、记录日志，再把观察结果写回状态。

这个分工很重要。模型擅长在语义上下文里选择下一步；权限、事务和审计属于程序侧。只要工具能读写外部资源，执行权就应该留在程序侧。

循环能继续，也必须能停

每一步结束后都要问两个问题：信息是否足够，风险是否可接受。足够就返回 final；缺关键输入就 ask_user；达到步数、预算、失败次数或风险阈值就停止或转人工。

停止条件是硬边界。缺少停止条件的 Agent 会反复搜索、重复提问、持续消耗 token，或者把一次工具失败误当成继续行动的理由。

ledger

动作协议

schema

把“下一步”收进三种动作

第一版先控制动作面。常见最小集合是：final 返回答案，ask_user 请求澄清，call_tool 请求工具。动作必须有固定字段，程序只接受 schema 合法、权限合法、工具名在白名单里的请求。

动作里可以保留 reason 字段用于日志，但理由不能当作权限证明。模型说“我需要查这个订单”不代表它有权查；权限仍由程序根据用户身份、资源归属和工具范围决定。

动作类型模型输出程序侧处理常见错误

final最终回复检查是否有足够证据、是否越界承诺证据不足却编答案

ask_user一个澄清问题结束本轮，等待用户输入反复问同一个问题

call_tool工具名和参数schema 校验、权限校验、执行工具未知工具、参数错、越权资源

handoff转人工或转专门流程暂停自动执行，写明原因高风险动作仍自动执行

工具规格

tools

工具描述要写“什么时候不用”

很多工具描述只写能力，比如“查询订单状态”。这不够。Agent 还需要知道什么时候不该用：缺订单号不能查，订单不属于当前用户不能查，查询政策不等于承诺补偿，发送邮件需要人工确认。

工具越强，边界越要硬。只读工具可以自动化得多一些；写入外部系统、发消息、改订单、退款、删除文件这类动作，要默认进入确认或人工审批。

工具面先小后大

工具多会扩大动作空间，也会增加误选工具和参数错配。最小 Agent 先给 2-3 个工具就够：查订单、查政策、请求人工确认。等 Eval 覆盖了常见路径，再逐步增加工具。

如果一个任务用固定 workflow 就能稳定完成，优先用 workflow。Agent 适合路径需要根据观察动态变化的场景。

字段get_order_statuslookup_shipping_policyrequest_human_review

用途查询当前用户订单状态查发货和缺货政策暂停并请求人工处理

参数order_idpolicy_topicreason、risk_level

返回状态、预计时间、错误码政策摘要、来源、更新时间工单号、等待状态

限制只能查当前用户订单不能生成赔付承诺高风险动作必须走这里

失败处理无权限、没找到、服务超时资料缺失、版本过旧创建失败时停止并说明

订单例子

worked

先看固定 workflow

固定 workflow 可以很好用。比如客服问答固定执行“查 FAQ → 生成答案 → 记录日志”。模型没有选择下一步路径，只在固定位置生成文本。这种结构更容易控，也更容易测。

fixed-workflow.py复制

def answer_with_rag(question):
 chunks = retrieve(question)
 answer = generate_answer(question, chunks)
 log({"question": question, "chunks": chunks, "answer": answer})
 return answer

再看受控 Agent 循环

下面的循环让模型选择动作，但执行权仍在程序侧。注意三个保护点：max_steps 限制循环，validate_action 检查 schema 和权限，未知工具被写成观察结果，不能直接执行。

minimal-agent-loop.py复制

def run_agent(goal, state, model, tools, max_steps=3):
 for step in range(max_steps):
 action = model.decide(goal=goal, state=state, tools=tool_specs(tools))

 if not validate_action(action, state):
 state["observations"].append({
 "ok": False,
 "error_code": "invalid_or_unauthorized_action"
 })
 continue

 if action["type"] == "final":
 return action["message"]

 if action["type"] == "ask_user":
 return action["question"]

 if action["type"] == "call_tool":
 tool = tools.get(action["tool"])
 if tool is None:
 state["observations"].append({
 "ok": False,
 "error_code": "unknown_tool"
 })
 continue

 observation = tool.call(action["args"])
 state["observations"].append(observation)
 update_state_from_observation(state, observation)

 return "我先停止处理：信息不足或已达到自动处理上限。"

这段代码该怎么读

模型每轮只产生动作，程序负责决定能不能执行。工具返回后，观察要写回状态。下一轮模型看到的是新的状态，随后基于新事实继续。循环末尾的停止文本也不是失败，它说明系统没有把不确定性继续放大。

一轮日志怎么读

trace

Agent 调试最容易漏掉的是“这一轮到底把什么交给了下一轮”。下面用同一个订单例子走一轮日志。最终回复只是一部分，还要同时看模型动作、程序校验、工具观察和状态更新。

输入状态stategoal=解释未发货，order_id=A-1024，steps_left=2订单号是否来自当前用户，本轮是否还有步数

模型动作actioncall_tool get_order_status(order_id=A-1024)动作类型、工具名、参数是否符合 schema

程序校验guardrailowner(user_id)==current_user，工具是只读权限是否由程序判断，而不是由模型自述判断

工具观察observationok=true，status=paid_waiting_stock，eta=tomorrow观察是否足够支持回答，是否带错误码或来源

状态更新known_facts追加“订单已付款，仓库缺货，预计明天补货”下一轮能否看到这个事实，避免重复查询

停止判断stop_checkenough_info=true，risk=read_only是否应该 final，而不是继续调用工具

如果停止没执行

即使观察已经足够回答，Agent 仍可能继续查政策、查物流、查仓库，最后把一个简单问题拖成多轮工具链。修法是把 enough_info、steps_left、failure_count 和 risk_flags 放进每轮末尾的硬检查，prompt 提醒只能作为辅助手段。

练习区会怎么用这条链

后面的诊断题会要求你按这条日志链定位问题。循环不停先看停止判断和观察写回；越权查询先看 guardrail；重复提问先看 known_facts；高风险动作先看风险标记和人工确认。

实际排查时可以把每一轮日志压成五行：输入状态、模型动作、程序校验、工具观察、停止判断。五行里哪一行缺字段，就先修哪一层。比如输入状态没有 steps_left，就先补状态和递减逻辑；工具观察只有一句“失败”，就先补错误码；停止判断没有读取 risk_flags，就先把高风险动作接到转人工。

修完以后，下一次日志应该能说明“为什么这轮没有继续”：例如 enough_info=true 导致 final，或 risk=write 导致 request_human_review。这就是最小验收，也方便后续稳定回归测试。

故障实验

failure

Agent 出问题时，先看它在哪个边界坏掉：状态没写回、工具太宽、权限没拦住，还是停止条件没执行。提示词调整放在定位之后。

循环不停停止判断和工具观察没有 max_steps、观察没写回、模型不知道信息已够加停止条件，压缩观察，要求足够信息时 final

重复提问known_facts用户已提供信息没有进入状态把输入写入结构化状态，并在下一轮传入

误调用工具工具描述和 schema工具用途太泛，没有写不适用条件缩小工具面，写限制和失败类型

越权查询权限校验把资源范围交给模型自觉判断程序侧校验用户、资源和动作权限

高风险动作未确认风险标记和 handoff写工具直接暴露给模型默认阻断，转人工或确认后执行

评测路径

eval

Agent Eval 不只看最终答案

07 页已经讲过样本、指标、基线和回归。到了 Agent，评测对象要扩展到路径：是否该调用工具、工具选得对不对、参数是否合规、是否越权、步数是否合理、需要停时有没有停。

如果只看最终答案，一个 Agent 可能碰巧答对，但中间查了不该查的资源，或者多调用三次工具才得到结果。这种路径在生产里会变成成本、延迟和合规风险。

最小验收样本

  • 信息足够：应直接 final，不额外查工具。
  • 缺订单号：应 ask_user，不猜订单。
  • 有订单号且属于当前用户：应调用只读查询工具。
  • 订单属于别人：应拒绝或澄清，不执行查询。
  • 需要补偿承诺：应转人工，不自动发送。
  • 工具超时：应记录失败并按重试/停止规则处理。

指标看什么通过标准失败后怎么查

任务完成最终是否解决用户问题回答与观察一致看证据和 final 文本

工具选择是否调用了正确工具该查才查，该问才问看 action log 和工具描述

参数合规工具参数是否满足 schema类型、必填项、资源范围合法看 validate_action 输出

权限安全是否访问不该访问的资源无越权读写看 user_id、resource_owner、guardrail

停止质量是否该停时停下不空转，不过度重试看 steps_left、failure_count、enough_info

边界

boundary

什么时候不要上 Agent

路径固定、风险高、可用规则明确时，优先 workflow 或普通工具调用。比如“把一段文本改成 JSON”“从固定知识库检索再回答”“按表单字段生成摘要”，路径本身无需动态选择，引入 Agent 只会增加不确定性。

Agent 的价值在于路径需要随观察变化：缺字段就问用户，查到缺货就查政策，发现高风险就转人工。没有这种分叉，先别引入循环。

下一页会追什么

这页把循环跑通，下一页要检查循环能不能上线：每多一步工具调用都会带来延迟，每次模型决策都会消耗成本，工具失败和重试会影响可靠性。09 页的停止条件和日志，是 10 页讨论成本、延迟、可靠性的前置条件。

现在先把最小 Agent 控住：小工具、小状态、强校验、清日志、能停止。之后再扩展框架、长期记忆、多 Agent 和更复杂的任务编排。

。用户输入没有写回 known_facts ，或者下一轮没有把结构化状态传给模型。修法是把订单号写入状态，并让模型看到已知事实。

15诊断

Q: 15. Agent 用用户提供的订单号查到了别人的订单。错在哪里？怎么修？

参考答案 / 评分要点

。错在把资源范围交给模型或工具默认放行。程序侧要校验当前用户、订单归属和动作权限；无权限时返回 observation 或拒绝，而不是执行查询。

16诊断

Q: 16. Agent 自动给用户发了补偿承诺邮件。为什么危险？应该加什么护栏？

参考答案 / 评分要点

。这是外部写入和业务承诺，风险高。应把发送动作默认转人工或要求显式确认，工具规格里标记风险，并在程序侧阻断未确认执行。

17设计

Q: 17. 为“会议纪要整理助手”设计 3 个以内工具，并说明哪些动作需要人工确认。

参考答案 / 评分要点

。可用工具：读取会议文本、查参会人/议题、生成待办草稿。发送纪要、创建外部任务、通知参会人需要人工确认。状态至少保存会议 id、已读文本摘要、待办草稿和确认状态。

18面试

Q: 18. 用 2-3 句话区分 workflow、tool use 和 Agent。

参考答案 / 评分要点

。Workflow 的路径由程序固定，模型只执行其中某个步骤。Tool use 是模型请求一个工具，应用代码执行并返回结果。Agent 把 tool use 放进循环，模型能根据观察选择下一步或停止。

19应用

Q: 19. Agent 的 Eval 除了最终答案，还应该评估什么？

参考答案 / 评分要点

。要评估工具选择、参数合法性、权限安全、步数和成本、是否该问用户、是否该转人工、是否该停不停，以及日志能否解释路径。
