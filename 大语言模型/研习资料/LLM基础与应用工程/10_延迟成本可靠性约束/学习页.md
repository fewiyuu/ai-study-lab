# 先抓主线 {#overview}
<!-- tag: flow -->

这条链路是一条带预算的读写链。用户发起请求后，系统先看路由和预算，再拼上下文、调用模型、查工具、做后验检查，最后决定返回、回退还是转人工。每一段都可能是瓶颈，也都可能写坏状态。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

阶段主要读取主要写入错了的症状

路由用户意图、风险标记、预算选择快路径/普通路径/高风险路径该转人工的动作被自动执行
上下文拼接证据块、历史消息、缓存prompt 长度、证据顺序慢、贵、旧材料混进来
模型生成上下文、输出长度限制答案草稿、工具请求输出太长、参数不稳
回退与重试error_code、timeout、retry_countfallback、handoff、backoff无限重试、用户一直等
收尾阶段耗时、成本估算日志、告警、结果后面只剩“失败/成功”两种模糊结论

# 状态账本 {#ledger}
<!-- tag: ledger -->

这页里的状态只看每一轮请求必须读写什么。账本写错了，慢、贵、不稳就会一起出现。先看对象，再看读写，再看症状。

对象进入本轮前本轮读取本轮写入错了会怎样

`request_meta`用户、场景、风险级别要走快路径还是高风险路径路由选择该转人工的动作被直接放行
`context_budget`剩余 token、目标长度当前上下文和证据量裁剪、压缩、早返回上下文越堆越长，延迟和成本一起涨
`cache_entry`缓存键、版本、过期时间是否命中、是否仍然新鲜命中或重算结果旧政策、旧答案、错用户复用
`risk_flags`写入动作、外部消息、退款承诺是否需要确认或人工复核handoff、stop、confirm自动发出不该发的消息
`observability`空或上一轮日志摘要阶段耗时、错误码、重试数日志、告警、跟踪 id出了问题只剩“失败”两个字

## box info
### 先看账本，再看答案
同样是“答对了”，一种可能是缓存命中，一种可能是花了三次重试才答对。账本能告诉你这次请求到底读了什么、写了什么、有没有把旧状态带进去。

# 延迟拆解 {#latency}
<!-- tag: latency -->
<!-- layout: grid2 -->

## box
### 先找最长的一段
客服摘要为什么会卡六秒，通常某一段在拖尾：排队、上下文拼接、模型生成、工具调用或重试。先把总延迟拆开，才能知道该删哪段。
$$T_{total} \approx T_{queue} + T_{context} + T_{model} + T_{tool} + T_{retry}$$
 `T_{queue}` 是排队时间，`T_{context}` 是拼接和压缩上下文的时间，`T_{model}` 是模型生成时间，`T_{tool}` 是工具调用和等待返回的时间，`T_{retry}` 是重试带来的额外时间。

## box soft
### 先量时，再动手
record-stage-times.py复制
```python
stage_ms = {}
stage_ms["queue"] = wait_queue(req)
stage_ms["context"] = build_context(req)
stage_ms["model"] = run_model(req)
stage_ms["tool"] = call_tools(req)
stage_ms["retry"] = retry_if_needed(req)
total_ms = sum(stage_ms.values())
```

先把每一段记下来，再决定是压上下文、减少串行工具、还是把结果先流式返回。没有这张表，只靠感觉很容易把力气花错地方。

分段常见触发最先看什么先做什么

排队并发高、限流、资源忙429、队列长度、请求峰值限流、退避、分流
上下文历史太长、证据太多prompt 长度、chunk 数压缩、过滤、少塞无关历史
模型生成输出太长、模型太大输出 token、首 token 时间限制字数、先出第一屏
工具调用串行查库、外部系统慢工具耗时、返回码并行、缓存、缩小工具面
重试盲重试、失败没分类retry_count、error_code先分错，再决定是否重试

# 成本估算 {#cost}
<!-- tag: cost -->
<!-- layout: grid2 -->

## box
### 成本要同时看 token、输出、工具调用、重试和人工复核
一条请求的成本通常来自四处：输入 token、输出 token、工具调用和重试。要是还需要人工复核，人工时间也要算进去。单次看起来不贵，放大到真实流量时就会变成预算问题。
$$C_{total} \approx n_{in}p_{in} + n_{out}p_{out} + n_{tool}p_{tool} + n_{retry}p_{retry} + n_{human}p_{human}$$
 `n_{in}`、`n_{out}`、`n_{tool}`、`n_{retry}` 和 `n_{human}` 分别是输入 token、输出 token、工具调用、重试次数和人工复核次数。`p_*` 是各自的单次成本。

## box soft
### 先压哪一项
estimate_cost.py复制
```python
estimated_cost = (
 in_tokens * p_in
 + out_tokens * p_out
 + tool_calls * p_tool
 + retries * p_retry
 + human_reviews * p_human
)
```

限制输出长度压的是输出 token；缓存压的是重复请求；减少重试压的是重复调用；路由到更小的模型压的是生成成本。不要拿一种手段去解决所有成本问题。

动作主要压哪一项常见副作用适合什么场景

压缩上下文输入 token、上下文处理时间可能丢掉证据历史长、噪声多
限制输出长度输出 token、生成时间回答可能不完整摘要、提要、固定字段
缓存重复计算和重复工具调用过期、错用户、版本不一致稳定问题、重复问题
路由到小模型生成成本复杂任务质量可能掉简单分类、短答
减少重试重复 token 和重复工具调用短暂错误可能不再自动恢复错误可分类、失败可控

# 三个场景 {#cases}
<!-- tag: cases -->

同一套技术，放到不同业务里，优先级会变。客服摘要、知识问答和退款工单，不该用同一条线去判断“好不好”。先看场景，再决定延迟、成本和可靠性各让多少。

客服自动摘要
目标是让坐席先看到关键信息。可以接受短暂流式返回，但不能把旧会话、错用户或不完整状态带进去。


知识库问答
目标是快、稳、能回指来源。适合缓存、检索和流式输出，但资料过旧时要能拒答或更新。


退款工单
目标不是快，而是别出错。可以慢一点，但高风险动作必须先确认，最好默认转人工。

场景先保什么可以省什么最先加的护栏

客服自动摘要第一屏、事实准确长篇解释摘要字段约束、超时回退
知识库问答来源可追、答案不编重复检索和重复生成缓存、版本号、引用
退款工单别自动做错事一点点延迟人工确认、权限校验、审计日志

# 客服摘要回放 {#replay}
<!-- tag: trace -->
<!-- layout: grid2 -->

## box
### 把一条请求拆开看
假设坐席要把最近 12 分钟的对话压成三条要点，并附上一个下一步动作。这个请求看起来只是“写一段摘要”，实际会经过路由、拼历史、检索、模型生成、格式校验、写日志、必要时回退或转人工。只要其中一段出问题，用户感受到的就“卡住了”“贵了”“错动作发出去了”。
这类请求适合拿来做回放，因为它把三种问题放在同一条链上：上下文太长会拖慢第一屏，输出太长会抬高成本，结果格式或权限不稳会把失败推到后面。你只要能把这一条链看清，后面再遇到知识问答、工单分流、自动回复，判断顺序就不会乱。
- 先确认这是不是高风险请求，是否允许直接执行写动作。
- 再看缓存能不能直接命中，是否需要重新拼上下文。
- 然后量每一段耗时，找出最慢的那一段，而不是先换模型。
- 最后看错误码和回退出口，保证失败能被接住。

## box soft
### 最小预算表
阶段先看什么预算怎么花常见失手

路由风险级别、任务类型高风险动作先留后手把写动作当成普通答复
拼历史会话长度、检索块、权限保留必要证据，删掉噪声把无关历史一起塞进 prompt
生成输出长度、首 token 时间先拿到第一屏，再补后续一口气追求完整答案，等太久
校验schema、字段、写入动作先拦住错格式和高风险写入看到像答案就直接放行
回退error_code、retry_count有限重试、退避、转人工把所有失败都当成临时波动

## box
### 回放代码
trace_request.py复制
```python
def trace_request(req):
 log("start", req.id, req.kind, req.risk)
 if cache_hit(req) and cache_fresh(req):
 return cache_answer(req)

 ctx = load_context(req)
 stage_ms = {}
 stage_ms["context"] = time_block("context", lambda: ctx)
 draft = run_model(req, ctx)
 stage_ms["model"] = draft.latency_ms

 if draft.error_code == "429":
 return fallback(req, "rate-limit")
 if draft.schema_error or draft.risk == "write":
 return request_human_review(req, draft)

 save_log(req, stage_ms, draft.cost_estimate)
 return draft
```

这段代码最重要的点在于，它把阶段耗时、错误码、缓存命中和回退出口放进了同一个流程。这样做以后，排障时能直接问：是缓存错了、上下文太长、模型太慢，还是失败后没有出口。

## box soft
### 三种动作的后果
动作主要改变副作用适合什么时候用

压缩上下文减少输入 token 和拼接时间可能丢掉一部分证据历史长、噪声多、上下文明显超标
流式输出让第一屏更早出现总计算量未必下降用户先要看到结果，再补后续
加缓存减少重复请求和重复工具调用要管过期、权限和版本重复问题多、答案稳定、可复用
转人工把高风险动作挡住人工成本会上升退款、补偿、写入外部系统

## box soft
### 失败路径怎么读
如果请求在六秒左右才出字，先别急着换模型。先看它是卡在拼历史、外部工具，还是等重试。拼历史慢，通常意味着上下文和检索块太多；工具慢，通常意味着串行调用或外部系统响应差；重试慢，通常意味着错误分类不清，失败被重复放大。三类慢的修法不同，所以第一步是把它们分开记。
排障时最值得留下的几个能复算的字段：`request_id`、`stage`、`latency_ms`、`error_code`、`cache_hit`、`retry_count`、`risk_flag`。字段一旦齐，后面才知道要压上下文、换路径、改退避，还是补人工确认。
字段为什么要留看不到会怎样

`request_id`把同一条请求串起来日志散成几段，没法回放
`stage`知道慢在路由、上下文、生成还是回退只能说“整体慢”
`latency_ms`知道是哪一段拖尾无法比较两次优化前后
`error_code`区分 429、timeout、schema、权限把不同故障当成一类
`cache_hit`判断是不是重复请求不知道慢是不是因为重复算
`retry_count`看失败有没有被放大看不出重试在抬高成本
如果你把这一组字段补进日志，后面的监控才会有用。没有它们，监控页只能告诉你“今天好像慢了”，却没法指出该改路由、上下文、重试，还是人工确认。

## box info
### 这个回放要你记住什么
1. 先找最慢的一段，不要先把锅甩给模型。
2. 先看错误码和状态账本，不要只看最终答案。
3. 先确认失败有没有被接住，再看它有没有跑出来。
4. 先判断高风险动作要不要留人工确认，再谈自动化程度。
把这条链看明白以后，后面的优化顺序就会很具体：该压的压上下文，该省的省缓存，该拦的拦高风险写入，该补的补日志和监控。它就是你每次排障时都能直接拿出来用的判断顺序。

# 429 退避演练 {#ratelimit}
<!-- tag: backoff -->
<!-- layout: grid2 -->

## box
### 把限流当成一条可复算的链
高峰期里很常见的一种情况是：第一轮请求撞到 429，第二轮重试还在同一波峰里，第三轮又把用户等候时间拉长。这个时候，继续加重试次数只会让成本更高。更合理的做法，是先看限流信号，再决定要不要退避、降并发、分流到小路径，或者直接切到稍后再试。
这类问题适合单独练，因为它把三件事放在一起了：延迟会因为等待拉长，成本会因为重复调用抬高，可靠性会因为错误被反复放大而变差。你只要会读 429、retry_count、queue_depth 和 backoff_ms，就能判断自己是在救火，还是在重复踩同一个坑。
- 先封顶重试次数，避免限流被放大成多次计费。
- 再把退避节奏和并发闸门绑在一起，别只改一个参数。
- 如果错误持续出现，就切到人工或稍后再试，不要继续刷请求。

## box soft
### 退避节奏表
轮次等待读什么写什么风险

第 1 次200msstatus=429retry_count=1继续撞到同一波峰
第 2 次600msqueue_depthbackoff_ms=600用户仍然等
第 3 次1800ms错误是否持续handoff=true重复花钱，结果还不稳
backoff_plan.py复制
```python
def next_delay(retry_count, queue_depth):
 base = 200
 delay = base * (retry_count + 1) ** 2
 if queue_depth > 50:
 delay += 300
 return min(delay, 2000)
```


## box info
### 失败后果和练习入口
盲重试会把一次限流变成多次排队、多次计费和更晚的失败。对用户来说，这更慢、更贵、结果还不稳定。把 429 处理清楚，后面才谈得上整体可靠性。
你可以直接拿上一节的时序表和这里的退避表对照：如果 queue_depth 已经很高，就不该继续把请求往同一条路径上送；如果 retry_count 已经到顶，就该转人工或稍后再试。练习时，先复算一遍等待时间，再判断该不该继续重试。

# 上线动作 {#workflow}
<!-- tag: workflow -->
<!-- layout: grid2 -->

## box
### 先写顺序，再写 prompt
上线最容易漏掉的是顺序：先查缓存还是先拼上下文，先记录错误还是先重试，先流式还是先等完整结果。顺序一乱，副作用就会跟着乱。
serve_request.py复制
```python
def serve_request(req):
 budget = plan_budget(req)
 if cache_hit(req) and cache_fresh(req):
 return cache_answer(req)

 chunks = load_context(req)
 if budget.too_tight:
 chunks = compress(chunks)

 result = call_model(req, chunks)
 if result.timeout:
 return fallback(req, "timeout")
 if result.schema_error or result.risk == "write":
 return request_human_review(req, result)

 persist_log(req, chunks, result)
 return result
```


## box soft
### 每一步都在读写什么
缓存命中会减少成本，但前提是 key 对、版本对、权限也对。上下文压缩能省时间和钱，但会牺牲一部分证据。转人工看起来慢，却能把高风险动作挡住。

步骤读写副作用错了的症状

查缓存cache_key、fresh_until、permission命中结果减少重复计算旧答案、错用户复用
拼上下文历史、检索结果、预算prompt 片段改变后续模型输入上下文过长，延迟升高
调用模型prompt、长度限制答案草稿、工具请求生成 token、可能触发工具输出太长、格式不稳
处理失败error_code、retry_countbackoff、fallback、handoff增加或抑制重试无限重试、一直卡住
记录日志阶段耗时、结果log_id、metrics给后续排障留下证据只剩“成功/失败”两种模糊结论

# 误区诊断 {#diagnosis}
<!-- tag: diagnosis -->

这张表的顺序很重要：先看阶段，再看错误码，再看缓存和重试，最后才看 prompt。只改 prompt，通常只会把症状挪开。

症状优先看哪里常见误判修法

平均延迟不高，但用户说慢P95/P99 和阶段耗时只看平均值找尾部最慢的一段，先动那段
成本突然上去输出 token、重试、工具调用只看输入 token限制输出、压重试、加缓存
旧政策被发给用户cache_key、版本、fresh_until以为缓存本身坏了给缓存加过期和权限边界
日志只有“失败”error_code、stage、latency_ms改 prompt 就够了先补日志字段，再谈优化
退款/补偿自动发出risk_flags、handoff、confirm回答写得更稳就能解决默认转人工或显式确认

## box info
### 排障顺序
1. 先看是哪一段慢，或者是哪一类失败。
2. 再看 error_code、retry_count、cache_key 和 risk_flags。
3. 最后才看 prompt、文案和模型选择。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 练习目标
这组题只看一件事：你能不能把一条请求拆成可量时、可算账、可回退的几段。客观题可以直接检查，开放题按参考答案自查，不要只写空泛结论。

## box
### 作答提醒
答题时按“先看哪一段、为什么慢、怎么压、失败时怎么办”来写。不要先讲抽象结论，再回头找证据。

<!-- end-practice-intro -->

<!-- practice-group: 一、概念边界 -->
<!-- quick-review: 复习 三类约束常常一起变坏，不能只盯一个指标。 -->

### 1 choice

**Q:** 1. LLM 功能上线时，最先要一起看的三类约束是什么？

- [ ] 延迟、成本、可靠性
- [ ] 美观、标题、字号
- [ ] 上下文、分词、Embedding

Answer: 延迟、成本、可靠性
Explain: 这三项常常一起变坏，不能分开看。

### 2 true/false

**Q:** 2. 流式输出一定会减少总成本。

- [ ] 对
- [ ] 错

Answer: 错
Explain: 流式输出能让用户更早看到结果，但总 token 和工具调用没变时，成本不一定下降。

### 3 fill

**Q:** 3. P95 关注的是最慢约 ______ 的请求。

Answer: 5%

<!-- end-practice-group -->

<!-- practice-group: 二、延迟拆解 -->
<!-- quick-review: 复习 慢请求通常不是单一原因，先量时，再动手。 -->

### 4 choice

**Q:** 4. 如果一条请求很慢，最先该做什么？

- [ ] 先把延迟拆成排队、上下文、模型、工具、重试几段
- [ ] 立刻换成更长的 prompt
- [ ] 把所有日志关掉

Answer: 先把延迟拆成排队、上下文、模型、工具、重试几段
Explain: 先分段，才知道该删哪一段、并哪一段或回退哪一段。

### 5 fill

**Q:** 5. 延迟公式里，`T_context` 对应上下文拼接和压缩。

Answer: T_context,上下文

### 6 synthesis

**Q:** 6. 流式输出能改善什么，不能改善什么？

Explain: 好答案会说：流式输出主要改善感知延迟，让用户更早看到第一屏；它不自动减少输入 token、模型计算量或外部工具等待。若慢点在排队、检索、权限检查或串行工具，流式只能让结果更早露出一部分，不会把总时长变短。写得更好的答案还能补一句：高风险动作和 schema 错误也不能靠流式解决。

<!-- end-practice-group -->

<!-- practice-group: 三、成本与回退 -->
<!-- quick-review: 复习 重试、缓存和限流都能改账，但各自有副作用。 -->

### 7 choice

**Q:** 7. 下列哪一项通常会直接抬高单次请求的成本？

- [ ] 重试一次
- [ ] 缩短日志字段
- [ ] 提前显示第一屏

Answer: 重试一次
Explain: 重试会重复消耗输入、输出和工具调用。

### 8 choice

**Q:** 8. 看到 429 时，最合理的第一步通常是什么？

- [ ] 立刻无限重试
- [ ] 有限重试并做退避，同时降低并发
- [ ] 把错误码吞掉

Answer: 有限重试并做退避，同时降低并发
Explain: 429 表示限流，不是“多跑几次就会好”。

### 9 synthesis

**Q:** 9. 为什么缓存要同时看过期时间和权限边界？

Explain: 好答案会说：缓存能省重复计算，但必须有过期时间，不然旧政策、旧价格或旧状态会被继续返回；还必须有权限边界，不然一个用户的结果可能被另一个用户复用。写得更完整的答案会补上版本号或 fresh_until，说明数据新鲜度和权限都是缓存的一部分。

<!-- end-practice-group -->

<!-- practice-group: 四、日志与迁移 -->
<!-- quick-review: 复习 先看阶段和错误码，再看 prompt。 -->

### 10 code-reading

**Q:** 10. 代码里 `persist_log(req, chunks, result)` 的副作用是什么？

Answer: 写日志,记录排障材料

### 11 choice

**Q:** 11. 如果日志里只有“失败”，你最先该补什么字段？

- [ ] error_code
- [ ] hero
- [ ] 标题

Answer: error_code
Explain: 没有错误码，就分不清超时、限流、schema 还是权限。

### 12 synthesis

**Q:** 12. 为“客服自动摘要”写 3 条 SLO 或监控指标。

Explain: 参考答案可以包含：P95 总延迟、JSON 解析成功率、摘要字段缺失率、缓存命中率、转人工率、超时率。好的答案会把指标和业务后果连起来，例如“P95 超过 4 秒坐席就开始等”“解析失败率上升说明 schema 或输出长度有问题”“转人工率飙升说明模型或权限边界在退化”。
