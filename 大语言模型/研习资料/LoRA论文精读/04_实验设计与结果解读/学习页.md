# 1. 先认清基线：每个方法到底训练什么 {#baselines}
<!-- tag: baselines -->

论文的实验表里名字很多，先别急着看谁加粗。第一步是认清每个 baseline 的训练对象。Fine-tuning 更新全部参数；BitFit 只训练 bias；PreEmbed 训练输入端特殊 token 的 embedding；PreLayer 训练每层里的特殊 token 激活；Adapter 在模型内部插入小层；LoRA 训练并联到已有权重上的低秩矩阵。

这一步很重要，因为参数量接近不代表机制接近。Adapter 和 LoRA 都可能只有几十万或几百万参数，但 adapter 是顺序插入模块，LoRA 是并联低秩更新。PreEmbed 参数少，但它用的是上下文位置。BitFit 便宜，但表达能力受限。读表时要把“训练对象”和“推理路径”放在一起看。

论文有些数字来自作者复现实验，有些带星号的是引用前人结果，有些带 dagger 的是为了公平比较而采用受限设置。表里这些符号不只是脚注，它们影响可比性。你现在应该能先看实验设置，再看指标高低。

**FT**更新全部预训练参数，表达强，但每任务参数和原模型同量级。

**BitFit**只训练 bias，参数很少，通常作为极轻量基线。

**PreEmbed**训练特殊 prefix/infix token 的 embedding，占用序列长度。

**PreLayer**训练每层特殊 token 激活，参数比 PreEmbed 多。

**Adapter**插入瓶颈模块，参数少但推理可能增加顺序计算。

**LoRA**训练低秩矩阵 A/B，merge 后推理走原线性层。

判断句：表格里的方法名如果不能翻译成“训练什么”，就还没开始读实验。你现在应该能把每个 baseline 放回训练路径里。

# 2. RoBERTa / DeBERTa：自然语言理解任务 {#nlu}
<!-- tag: glue -->

第一组主要看 NLU，也就是自然语言理解任务。RoBERTa base/large 和 DeBERTa XXL 在 GLUE benchmark 上测试，指标包括 MNLI、SST-2、MRPC、CoLA、QNLI、QQP、RTE、STS-B 等。这里的核心问题是：LoRA 用远少于全量微调的参数，能否接近或超过全量微调和 adapter 类方法。

论文表格显示，RoBERTa base 的 LoRA 只训练约 0.3M 参数，平均分 87.2，高于全量微调表中 86.4；RoBERTa large 的 LoRA 约 0.8M 参数，平均分 89.0，和全量微调 88.9 接近并略高；DeBERTa XXL 的 LoRA 约 4.7M 参数，平均分 91.3，也达到全量微调水平。这里说在这些任务上，它没有因为参数少而明显牺牲质量。

读这组实验要注意公平性。论文为了和 adapter 更公平比较，还做了受限设置，例如相同 batch size、序列长度 128，并避免某些任务从已经适配 MNLI 的模型起步。带 dagger 的行就是这种设置下的结果。你现在应该能解释为什么实验设置会影响基线比较。

数值对比scale**RoBERTa base FT 参数**125.0M
**RoBERTa base LoRA 参数**0.3M
**RoBERTa large FT 参数**355.0M
**RoBERTa large LoRA 参数**0.8M

# 3. GPT-2：生成任务不是只看准确率 {#nlg}
<!-- tag: generation -->

GPT-2 实验看的是 E2E NLG Challenge，属于生成任务。生成任务常用 BLEU、NIST、METEOR、ROUGE-L、CIDEr 这类指标，不像分类任务只有 accuracy。论文把 GPT-2 medium 和 large 都拿来比较，重点看 LoRA 在 NLG 模型上是否仍然有优势。

表中 GPT-2 medium 的 LoRA 只训练 0.35M 参数，但 BLEU、NIST、METEOR、ROUGE-L、CIDEr 都达到或超过列出的基线。GPT-2 large 的 LoRA 训练 0.77M 参数，也在多数指标上表现很强。这里说明 LoRA 不只在理解任务有效，在生成任务上也能作为强基线。

不过生成指标本身有噪声和局限。BLEU 高不一定代表所有生成质量都好，ROUGE 更偏重重叠，CIDEr 来自图像描述评估传统。读这组实验时，重点看到作者覆盖了不同任务类型，并用多个指标避免单一评价偏差。

## box
### BLEU / NIST
偏 n-gram 重叠和信息量，常用于生成质量比较。

## box
### METEOR / ROUGE-L
关注匹配、召回或最长公共子序列，能补充 BLEU。

## box
### CIDEr
衡量候选文本和参考文本的相似性，E2E 表中也作为一个生成指标。

判断句：生成任务不能只说“准确率提升”。你现在应该能识别论文在不同任务类型上用了不同指标。

# 4. GPT-3：大模型压力测试 {#gpt3}
<!-- tag: gpt-3 -->

GPT-3 175B 是论文最有代表性的压力测试。这里比较 WikiSQL、MultiNLI-matched 和 SAMSum 三类任务：自然语言到 SQL、自然语言推理、对话摘要。全量微调要训练约 175,255.8M 参数；LoRA 有 4.7M 和 37.7M 两个参数预算。这个对比直接服务论文动机：大模型多任务适配不能每次都复制完整模型。

表格显示，LoRA 在 WikiSQL、MNLI-m、SAMSum 上达到或超过全量微调与其他 PEFT 基线。比如 4.7M 参数的 LoRA 在 MNLI-m 上是 91.7，在 SAMSum 的 ROUGE-1/2/L 是 53.8/29.8/45.9；37.7M 参数的 LoRA 在 WikiSQL 上达到 74.0。论文还标注了任务上的典型波动：WikiSQL 约 ±0.5%，MNLI-m 约 ±0.1%，SAMSum 约 ±0.2/±0.2/±0.1。

这里有一个读表细节：更多可训练参数不总是更好。prefix 方法在参数变多时可能反而下降，论文怀疑特殊 token 太多会让输入分布远离预训练分布。LoRA 的结果相对稳定，但也不能推出“参数越多越强”。

# 5. 读实验的四个检查问题 {#fairness}
<!-- tag: fairness -->

第一，比较对象是否合理？如果一个方法训练 0.3M 参数，另一个训练 355M 参数，你要同时看质量和成本。只说“谁分高”是不完整的。

第二，设置是否公平？序列长度、batch size、初始化、是否引用前人结果、是否调参充分，都会影响表格。论文用星号和 dagger 标注部分设置，这些脚注要读。

第三，指标是否支持主张？LoRA 的主张有几部分：任务质量、可训练参数量、训练吞吐、显存、推理延迟、checkpoint 大小。单个 GLUE 平均分不能支持所有主张，必须结合方法设计和工程数字。

第四，结论能否外推？论文覆盖了 RoBERTa、DeBERTa、GPT-2、GPT-3 和多个 NLP 任务，但没有覆盖所有模型和所有领域。你可以把 LoRA 当强基线，不能把表格数字当成新任务的保证。

## box
### 只看加粗
会忽略参数量、方差和设置差异。读表不能只追最高分。

## box warn
### 忽略脚注
星号和 dagger 影响可比性。脚注常常比表头更关键。

## box warn
### 过度外推
论文实验支持论文范围内结论，不自动覆盖所有现代模型和任务。

# 补充案例：把一张实验表审成可信结论 {#audit-chain}
<!-- tag: audit -->

读 LoRA 的实验结果时，最容易犯的错误是把表格当排行榜：哪一行分数最高，就说哪种方法最好。论文真正想证明的一个组合命题：在冻结大模型主体参数的前提下，低秩更新能用很少的可训练参数接近或超过全量微调，并且保存、切换和部署成本明显更低。这个命题比“分数高一点”复杂，所以读表时必须把实验对象、参数预算、任务类型、指标和工程代价连起来。

先看一个具体场景：你要给一个客户支持分类模型选微调方案。全量微调可能在验证集上比 LoRA 高 0.2 分，但它需要保存完整模型副本，训练时优化器状态也跟着膨胀；LoRA 只保存 adapter，checkpoint 只有几十 MB，多个客户或多个任务可以共享同一个底座。此时 0.2 分不是唯一答案，你要问这 0.2 分是否稳定、是否超过方差、是否值得多出的存储和部署复杂度。实验表里的平均分只能回答“质量是否接近”，不能替你回答“生产系统是否划算”。

论文里的 RoBERTa、DeBERTa、GPT-2、GPT-3 表格分别承担不同证据功能。RoBERTa 和 DeBERTa 主要说明在经典 NLU 任务上，LoRA 与全量微调接近；GPT-2 的生成实验说明 LoRA 不只适用于分类；GPT-3 的实验说明当底座变大时，少量 adapter 参数仍可能有效。把这些表合在一起，才能形成“跨模型、跨任务的参数高效证据”。如果只摘一张最高分表，结论会变窄；如果把所有表都说成“LoRA 普遍最优”，结论又会过宽。

读表第一步：确认训练对象FT 训练全部参数，Adapter 增加旁路模块，Prefix 训练输入侧连续提示，BitFit 只训练 bias，LoRA 训练低秩矩阵。方法名相似，但预算和部署路径不同。你现在应该能先问“这行到底训练了什么”。

读表第二步：确认成本口径参数量、显存、checkpoint、吞吐和延迟不是同一个指标。LoRA 的核心优势常常体现在保存和切换任务，而不是每张表都拿最高分。你现在应该能把质量指标和成本指标分开读。

读表第三步：确认任务边界GLUE、E2E、WebNLG、WikiSQL 等任务的评价指标不同，数据分布也不同。某个任务上的提升不能自动外推到所有生成任务、所有领域任务或所有现代模型。

下面这张“审表清单”可以直接拿来检查论文表格。它为了避免把实验读成口号。

| 检查问题 | 你要在表里找什么 | 如果漏看会怎样 |
| --- | --- | --- |
| 质量是否接近 | 平均分、单任务分、方差或多次运行标注 | 把噪声当成真实提升，或者低估一个稳定但略低的方法 |
| 预算是否公平 | 可训练参数、底座大小、是否使用同一预训练模型 | 拿不同容量的模型直接比较，结论偏向大预算方法 |
| 设置是否一致 | batch、学习率、序列长度、调参次数、脚注说明 | 忽略实现差异，把工程设置误读为方法本身差异 |
| 部署是否成立 | 是否能 merge、是否增加推理延迟、是否需要多套模型副本 | 只在训练表里赢，部署时成本反而不可接受 |

在复述论文时，可以把一行实验结果翻译成三句话。第一句说事实：在某个模型和任务集合上，LoRA 用多少可训练参数取得了什么指标。第二句说解释：这说明任务相关更新可能能由低秩矩阵表达，因此不必更新全部权重。第三句说边界：这个结论依赖论文覆盖的模型、任务、超参和评估方式，不能直接保证新领域。三句话都说完整，才是合格的论文复述。

```python
def read_lora_result(row, baseline):
 same_base = row.base_model == baseline.base_model
 quality_close = row.score >= baseline.score - row.expected_noise
 much_smaller = row.trainable_params < baseline.trainable_params / 100
 deploy_gain = row.checkpoint_mb << baseline.checkpoint_mb
 if same_base and quality_close and much_smaller and deploy_gain:
 return "supports parameter-efficient adaptation, within this task setup"
 return "needs more checks before claiming LoRA wins"
```

失败后果也要讲清楚。假设你只看 GLUE 平均分，忽略了参数量和 checkpoint，就会把全量微调、Adapter、Prefix、LoRA 混成同一类方法；假设你只看 LoRA 参数少，忽略质量表，就会把“省钱”误读成“有效”；假设你只看最高分，忽略脚注，就可能把不完全同设定的结果拿来并排比较。论文精读训练的把一张表拆成可验证的主张。

再做一次手工演练。看到 “LoRA 在 RoBERTa large 上用 0.8M 参数接近全量微调 355M 参数” 这类句子时，不要立刻写进笔记。先把它拆成四栏：对象是 RoBERTa large，不是所有 Transformer；任务是 GLUE 类理解任务，不是任意生成任务；指标是论文表里的任务分数和平均值，不是线上业务收益；成本是可训练参数和保存 adapter 的成本，不等于总推理成本为零。拆完以后，这句话才会变成一个可用结论：在这些 NLU 设置中，LoRA 用极小的任务参数达到了接近全量微调的质量，说明低秩适配是强基线。

如果你要把这张表用于项目决策，还要补一个“缺失证据”列表。论文表格通常不会直接告诉你自己的数据分布、标签噪声、线上延迟预算、模型切换频率和回滚流程。比如内部有 30 个客户、每个客户一个 adapter，LoRA 的 checkpoint 优势会非常明显；如果只有一个长期模型且对质量极敏感，全量微调也可能值得比较。实验表支持的是方法能力，项目决策还要接上业务约束。能分清这两层，读论文就不会被表格牵着跑。

## box
### 可以直接说
在论文覆盖的若干 NLP 任务中，LoRA 用远少于全量微调的可训练参数取得了相近质量。这个说法有模型、任务和指标边界。

## box warn
### 需要补证据再说
LoRA 在我们的业务上一定更省。还要看线上延迟、adapter 管理、评估集代表性、服务端是否允许动态加载或提前 merge。

## box
### 不要这样说
LoRA 全面超过全量微调。这个说法把部分表格结果扩大成全局结论，也忽略了不同任务和设置的差异。

还有一个经常被忽略的点：论文里的表格是结果，不是实验设计本身。你复现或迁移时要反推设计：基线要有全量微调或强 PEFT 方法，数据切分要固定，调参预算要可说明，训练成本要记录，结果要有至少一次 sanity check。若只拿 LoRA 跑一遍，然后说“分数还行”，它只能证明你跑通了代码，不能证明 LoRA 是合理选择。好的实验解读会逼你补齐对照组，而不是只复述作者的最终表格。

把它落成笔记时，推荐用“结论卡片”而不是流水账。卡片第一行写主张：LoRA 在该表中支持参数高效适配。第二行写证据：模型、任务、指标、参数量差异。第三行写限制：脚注、设置差异、任务范围、是否只是一组 seed。第四行写行动：如果迁移到新任务，先跑小 rank LoRA、全量或强基线、保存加载检查，再讨论是否扩大 rank 或 target modules。这样的笔记以后能直接指导实验，而不是只留下“表 2 很强”。

| 卡片行 | 写什么 | 示例 |
| --- | --- | --- |
| 主张 | 这张表最多支持什么 | LoRA 是强参数高效基线 |
| 证据 | 模型、任务、指标、参数量 | RoBERTa large / GLUE / 接近 FT / 0.8M vs 355M |
| 限制 | 哪里不能外推 | 不自动覆盖长文本生成、领域迁移或线上延迟 |
| 行动 | 下一步怎么验证 | 同设定基线、小 rank sweep、保存加载与 merge 检查 |

最后看一个容易误判的例子：某同学读到 GPT-3 表里 LoRA 用 4.7M 参数在部分指标上超过更大参数的方案，于是写下“参数越少反而越好”。这句话的问题把一个局部观察改写成了单调规律。更稳妥的改法是：在论文的 GPT-3 任务设置里，LoRA 的小参数 adapter 取得了有竞争力的结果，说明更大的任务参数量并不自动带来更好指标；但具体方法仍要看 target、rank、数据和评估指标。这个改写保留了表格给出的惊喜，也没有把它讲成普适定律。

如果你在自己的笔记里发现“证明”“总是”“全面优于”“只要”等词，要停下来重读表头和脚注。实验论文的语言应该先给范围，再给强度。范围包括模型、任务、指标、基线和设置；强度包括接近、超过、显著降低成本、没有增加延迟等。先写范围，再写强度，结论就会自然收住。你现在应该能把一句夸大的实验解读改成可复核的论文语言。

最后给自己一个三分钟检查动作：任选一行 LoRA 结果，先遮住方法名，只根据参数量、指标和脚注猜它能支持什么主张；再把方法名露出来，检查自己的主张是否因为“LoRA”这个名字而变得更夸张；最后写下如果要在自己任务上复验，还缺哪一个对照组、哪一个成本指标、哪一个部署检查。这个动作能把读表从“看作者怎么说”变成“我能不能独立判断”。

如果时间只够做一个动作，就做“反向读表”：先写你希望这张表证明什么，再逐项确认表里是否真的给了证据。没有给出的部分要写成待验证，而不是藏在结论里。这样做会让你的论文复述天然带边界，也能帮你在组会里回答别人追问：这个结论来自哪一列，缺的证据准备怎么补。

把这个动作做熟以后，你读其他 PEFT 论文也能复用：先拆主张，再拆证据，最后拆缺口。表格不再只是结果展示，而是实验设计是否可信的检查入口。

这一步尤其适合在复述前做，因为它能把“我记得作者说了什么”改成“我知道证据支持到哪里”。

结论越短，越要把范围写清楚。

这就是读实验的底线。

也是复述的底线。

练习时可以用一个固定句式逼自己读完整：“在某模型、某任务和某设置下，LoRA 用某个量级的可训练参数，达到了接近或超过某基线的某指标；因此它支持某个有限主张，但还需要检查某个边界。”这个句式听起来笨，但很管用，因为它会自动阻止两种极端：一是只说“LoRA 更好”，二是只说“论文表明低秩有效”却不给证据。你现在应该能把任意一行实验表改写成带范围、带依据、带风险的结论。

> [!NOTE]
> **练习钩子：**
> 等会做题时，不要先写“LoRA 好”。先写比较对象、指标、参数量和边界，再判断这张表支持哪一个主张。

### 04_03 短答

**Q:** PreEmbed 和 PreLayer 的训练对象有什么不同？

Explain: PreEmbed 训练词嵌入层，PreLayer 训练词嵌入和额外前馈层。PreEmbed 只调整 token 表示，PreLayer 还能调整 token 之间的交互方式。所以 PreLayer 表达能力更强，但参数量和延迟也更高。LoRA 不训练词嵌入，而是训练注意力层的低秩分支，与这两者都不同。

### 04_04 短答

**Q:** Adapter 和 LoRA 都参数少，为什么推理路径不同？

Explain: Adapter 在 Transformer 层内插入额外的前馈子层，推理时必须经过这些新增层，增加了延迟。LoRA 通过低秩矩阵直接修改权重，推理时可以把 B A 合并回 W，不增加额外层，因此延迟和原模型几乎相同。

### 04_05 短答

**Q:** 实验表里有些结果来自前人报告。读结论时为什么要注意这一点？

Explain: 前人报告的结果可能来自不同的训练设置、数据版本或评估协议。直接比较可能不公平。应该检查：是否使用相同底座、相同数据集划分、相同训练步数和相同评估指标。如果不一致，结论的可比性就会下降。

### 04_06 短答

**Q:** 为什么论文要用受限设置和 adapter 做公平比较？

Explain: 受限设置（如只训练少量参数）排除了"参数多所以效果好"的干扰。和 adapter 比较是因为 adapter 也是参数高效方法，可以控制参数量相近的情况下比较哪种结构更有效。公平比较能说明 LoRA 的结构优势，而不只是参数效率。

### 04_07 短答

**Q:** RoBERTa large 中 LoRA 0.8M 参数达到接近 FT 的平均分，这能说明什么？

Explain: 说明在 RoBERTa 这类中等规模模型上，极少量参数（0.8M）就能接近全量微调（330M）的效果。这意味着预训练模型已经编码了丰富的任务知识，只需要低秩更新就能调动这些知识，而不是重新训练大量参数。

### 04_08 短答

**Q:** DeBERTa XXL 上 LoRA 4.7M 与 FT 1500M 参数对比，重点应该看什么？

Explain: 重点看参数量效率：LoRA 只训练 4.7M 参数就达到 FT 1500M 的效果，参数效率提升约 300 倍。同时要看推理延迟：LoRA merge 后没有额外层，而 FT 和 Adapter 都有额外计算。这说明 LoRA 在超大规模模型上依然保持高效。

### 04_13 诊断

**Q:** 诊断：有人只看表格加粗项就说 LoRA 全面碾压所有方法。按错因、后果、修法回答。

Explain: 错因：忽略了表格中未加粗的数据和统计显著性。后果：高估 LoRA 的适用范围，低估其他方法在特定任务上的优势。修法：看所有任务的表现分布，检查统计显著性，注意每个方法的最佳任务类型和失败模式。

### 04_14 诊断

**Q:** 诊断：有人只比较分数，不比较可训练参数量和推理延迟。按错因、后果、修法回答。

Explain: 错因：把分数当作唯一指标，忽略了工程约束。后果：选择了分数略高但参数量大、延迟高、部署困难的方法。修法：在固定参数预算和延迟约束下比较分数，或者画出帕累托前沿（分数 vs 参数量 vs 延迟）。

### 04_15 诊断

**Q:** 诊断：有人把 2021 年论文表格直接当作所有 2026 年模型的默认结论。按错因、后果、修法回答。

Explain: 错因：忽略了模型规模、架构、训练数据和任务类型的变化。后果：把 2021 年 GPT-2 的结论直接套用到 2026 年的大模型，导致方法选择错误。修法：在目标模型和任务上重新验证，关注论文的适用范围和边界条件。

### 04_16 面试追问

**Q:** 用 1-3 句话解释 LoRA 论文的实验为什么有说服力。

Explain: 实验覆盖了不同模型规模（RoBERTa 到 DeBERTa XXL）、不同任务类型（NLI、SQL、摘要）和不同基线方法（FT、Adapter、Prefix），在控制参数预算的情况下展示了 LoRA 的分数、延迟和参数量优势，结论有跨模型和跨任务的证据支撑。

### 04_17 场景

**Q:** 如果你要在自己任务上验证 LoRA，至少要记录哪 6 个实验条件？

Explain: 1. 底座模型名称和版本。2. 数据集名称、划分方式和样本量。3. 训练步数和批次大小。4. 学习率和调度策略。5. LoRA 的 rank、alpha、target modules。6. 评估指标和随机种子。至少这 6 项才能复现和比较。

### 04_18 费曼解释

**Q:** 用 6-9 句话解释 LoRA 论文的实验设计和主要结论。

Explain: LoRA 论文的实验设计覆盖了多个模型规模（从 RoBERTa 到 DeBERTa XXL）和任务类型（NLI、SQL、摘要），在控制参数预算的前提下比较了 LoRA 与全量微调、Adapter、Prefix 等方法。主要结论是：LoRA 用极少量参数就能接近全量微调的效果，且推理延迟不增加，因为低秩更新可以合并回原始权重。论文还通过 SVD 分析解释了为什么小 rank 有效：预训练权重 W 的奇异方向已经编码了丰富的任务知识，Delta W 只需要在 W 的顶部奇异方向上做小幅调整即可调动这些知识。
