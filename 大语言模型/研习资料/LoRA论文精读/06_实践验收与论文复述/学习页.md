# 1. 从论文到项目：先写验收清单 {#checklist}
<!-- tag: checklist -->

真正做 LoRA 微调时，不要只把配置复制进训练脚本。先写一个最小验收清单：底座模型是什么，任务数据是什么，target modules 是哪些，rank 和 alpha 是多少，哪些参数可训练，保存的 adapter 文件多大，merge 前后输出是否一致，和哪些 baseline 比较。

这份清单的作用是防止“以为在训练 LoRA，其实什么都没训”或“以为只训 LoRA，其实底座也被更新”。最常见的两个检查是：打印 trainable parameters，并列出可训练参数名。只看 loss 下降不够，因为底座误更新也会让 loss 下降，但你已经失去 PEFT 的工程收益。

实验设计也要有 baseline。至少要有原始底座零样本/少样本表现、一个合理 LoRA 配置，以及如果资源允许的全量微调或小样本全量微调对照。没有 baseline，就无法判断 LoRA 的质量和成本是否值得。

## box
### 参数检查
确认只有 LoRA A/B 或允许的 bias 可训练。

## box
### 配置检查
记录 rank、alpha、target modules、dropout、学习率。

## box warn
### 部署检查
验证 merge 前后输出接近，避免重复 merge。

# 2. rank、alpha、target modules 怎么选 {#config}
<!-- tag: configuration -->

初始配置可以保守：先从常见的 `q_proj/v_proj` 和中等 rank 开始，例如 `r=8` 或 `r=16`，再根据验证集、显存和过拟合情况调整。论文告诉你小 rank 可能很强，但不是让你跳过验证。rank 是容量，alpha 是分支尺度，target modules 是更新位置，它们会一起影响训练。

如果 loss 降不动，先别马上把 rank 拉满。先检查 target module 是否命中、学习率是否合理、数据格式是否正确、标签是否参与 loss、LoRA 参数是否真的 requires_grad。很多“rank 不够”的问题其实是配置没生效。确认训练正常后，再做 rank sweep。

如果过拟合明显，可以降低 rank、加 LoRA dropout、减少训练 epoch、调低学习率，或者扩大数据。LoRA 参数少不代表不会过拟合，尤其是小数据任务。你现在应该能把“容量不够”和“训练配置错误”分开排查。

# 3. 保存、加载和 merge 的状态管理 {#merge}
<!-- tag: state -->

LoRA 产物通常只保存 adapter 权重和配置，不保存完整底座。加载时要用同一个或兼容的底座模型，再把 LoRA 权重挂上去。部署可以选择保留动态 LoRA 分支，也可以 merge 到原权重后保存合并模型。两种方式各有用途。

状态管理的核心是避免混乱：当前模型是干净底座、挂载未 merge LoRA、已经 merge 的模型，还是 merge 后又卸载过？很多线上错误来自重复合并、加载错底座、把 adapter 当完整模型发布、或者量化模型 merge 后精度路径没测。

验收时要做一个小测试：同一批输入，未 merge 的 LoRA 模型和 merge 后模型输出应当非常接近。允许有极小浮点误差，但不能语义大变。如果差异大，先查 scaling、dtype、是否重复 merge、target module 命名和加载顺序。

逐行拆解 逐行拆解：merge 前后输出验收复制

```text
# 伪代码：实际 API 随框架不同会变化
model_lora = load_base_with_lora(base_path, lora_path)
logits_before = model_lora(input_ids).logits

model_merged = merge_lora(model_lora)
logits_after = model_merged(input_ids).logits

max_diff = (logits_before - logits_after).abs().max()
print("max diff:", max_diff)

# 若差异很大，先查 scaling、dtype、是否重复 merge。
```

### adapter 权重
小、便于分发，但加载时必须指定兼容底座。

## box
### 合并模型
部署简单，推理路径普通，但文件变大，不便于频繁切换任务。

## box
### 状态错误
重复 merge 或加载错底座，会让输出异常且难排查。

# 4. LoRA 的适用边界 {#limits}
<!-- tag: boundaries -->

LoRA 适合的典型场景是：底座模型已经具备相关能力，下游任务主要需要风格、格式、领域偏好或特定输出映射；你希望低成本保存多个任务版本；部署希望少增加延迟。分类、抽取、指令风格、结构化输出、领域问答、摘要和轻量对齐都常见。

LoRA 不一定适合所有场景。如果任务需要大量新知识、预训练模型完全没有相关能力、下游数据分布差异极大，或你需要从底层表示到高层行为都大幅改变，小 rank LoRA 可能不够。此时可以提高 rank、扩展 target modules、结合继续预训练，或者考虑全量/部分微调。

还有一个现代语境：LoRA 之后出现了很多变体和组合，例如量化底座上训练 LoRA 的 QLoRA，调整方向和幅度的 DoRA，自适应 rank 的方法等。这些不改变论文的基本理解。先学会 LoRA 的 `Delta W=BA`，再看后续方法才不会迷路。

## box
### 适合
底座已有能力，任务更新像重排、强调或格式适配。

## box warn
### 谨慎
任务需要大量新能力、新语言、新知识或强分布迁移。

## box
### 替代方案
更高 rank、更多 target modules、继续预训练、部分或全量微调。

# 5. 如何完整复述 LoRA 论文 {#paper}
<!-- tag: feynman -->

复述论文可以按四步走。第一步讲问题：大模型全量微调在多任务部署中太贵，每个任务保存完整模型不现实。第二步讲方法：冻结预训练权重，对部分线性层学习低秩更新 `Delta W=BA`，训练 A/B，部署时可 merge。第三步讲证据：在 RoBERTa、DeBERTa、GPT-2、GPT-3 上，LoRA 用远少的可训练参数达到接近或超过全量微调和其他 PEFT 基线的指标，并且没有 adapter 那类额外推理延迟。第四步讲解释和边界：rank 扫描和子空间分析显示下游更新可能有低 intrinsic rank，但这不是所有任务的定理。

表达时不要把论文讲成“LoRA 是最好的微调方法”。更准确的是：LoRA 提供了一种简单、可 merge、参数高效的低秩适配方法，在论文覆盖的模型和任务上表现强，并给出了低秩更新的实证解释。它的优势来自任务参数少、显存和 checkpoint 低、推理路径可合并；限制包括混合任务 batch、target/rank 选择依赖经验，以及小 rank 对分布差异大的任务未必足够。

你现在应该能把 LoRA 从公式、工程和实验三个角度讲清楚。真正会用论文，能在新任务上设计验证，发现配置错误，并知道什么时候该扩大 LoRA 或换别的方法。

一段稳妥复述：
问题 → 方法 → 实验证据 → 机制解释 → 边界

全量微调太贵Delta W=BAWq/Wv 等落点实验接近 FT小 rank 有边界

# 6. 常见工程错误、后果和修法 {#pitfalls}
<!-- tag: diagnosis -->
<!-- layout: grid -->

## box
### 可训练参数为 0
target module 没命中或冻结逻辑错误。修法：打印参数名和 requires_grad。

## box
### 底座也在训练
PEFT 变成部分全量微调。修法：冻结底座并检查 optimizer 参数组。

## box
### adapter 当完整模型
部署缺少底座无法推理。修法：明确保存 adapter 和 base 版本。

## box warn
### merge 前后差异大
可能 scaling、dtype、重复 merge 或加载顺序错误。修法：做 logits diff 测试。

## box warn
### rank 盲调
未排除数据和配置问题就加容量。修法：先验收链路，再 sweep。

## box warn
### 论文表达夸大
把局部实验说成普适定理。修法：补上任务范围和边界。

# 补充案例：从论文复述走到项目验收 {#delivery-case}
<!-- tag: delivery -->

最后一页要把前面的概念落到一件事上：你能不能在真实项目里把 LoRA 训对、验收清楚、讲得稳妥。论文复述能把方法、证据、工程路径和边界串成可执行检查。一个项目里最危险的状态你以为自己在训练 LoRA，实际上 target module 没命中、底座也被 optimizer 更新了，或者保存出来的 adapter 依赖一个没人记录版本的 base model。

设想你接手一个中文客服意图分类任务。你决定用 LoRA 微调一个 7B 底座。第一天不应该直接跑三天训练，而是先做最小验收：打印所有可训练参数名，确认只有 LoRA 的 `A/B` 矩阵在训练；统计 trainable params，确认数量与配置量级一致；跑一个极小 batch，确认 loss 能下降；保存 adapter 后重新加载，确认 logits 与保存前一致；最后再 merge 一份模型，比较 merge 前后输出差异。这个顺序朴素，但能挡住大部分工程事故。

| 阶段 | 必须检查 | 不检查的后果 |
| --- | --- | --- |
| 训练前 | target modules 命中、base 冻结、trainable params 量级正确 | 训练对象错了，后续指标没有解释价值 |
| 训练中 | loss 曲线、验证集、显存、吞吐、学习率日志 | 只看到最后分数，不知道失败来自优化还是数据 |
| 保存加载 | adapter config、base model 版本、tokenizer、dtype | 换机器后加载不起来，或悄悄加载错底座 |
| merge 部署 | merge 前后 logits diff、重复 merge 防护、推理延迟 | 离线验证正常，线上输出漂移或延迟不符合预期 |
| 论文复述 | 问题、方法、证据、机制解释、适用边界 | 只会背“低秩有效”，无法指导新任务决策 |

实践验收还要区分“训练成功”和“方法有效”。训练成功是链路层面的：参数确实在更新，保存加载正常，merge 不出错。方法有效是任务层面的：在同一数据、同一评估、合理基线下，LoRA 达到目标指标，并且成本优势成立。两者缺一不可。只做训练成功，你可能得到一个能跑但没有业务价值的 adapter；只看任务分数，你可能不知道它是 LoRA 的效果，还是底座被意外全量更新了。

```python
def accept_lora_run(report):
 assert report.trainable_names_match_lora
 assert report.base_params_frozen
 assert report.adapter_reload_logits_diff < 1e-4
 assert report.merged_logits_diff < 1e-3
 assert report.metric_beats_baseline_or_meets_budget
 return "run accepted with reproducible adapter and bounded claim"
```

再把论文复述压成一个能开口讲的版本：LoRA 针对全量微调成本高的问题，冻结预训练权重，只在部分线性层上训练低秩矩阵 `A` 和 `B`，让权重更新写成 `Delta W = B A`。训练时只更新少量 adapter 参数，部署时可以把增量合回原权重，所以不一定增加推理路径。论文在 RoBERTa、DeBERTa、GPT-2、GPT-3 等设置上比较了全量微调、Adapter、Prefix、BitFit 等方法，展示了接近或超过基线的任务质量和更低的参数/存储成本，并用 rank 与子空间分析解释为什么小 rank 可能有效。边界是：rank、target modules、任务分布和底座能力都会影响结果，论文证据不能自动覆盖所有任务。

这段复述有五个骨架。第一，问题：为什么需要参数高效微调。第二，方法：低秩增量怎么接到线性层。第三，工程：冻结、保存 adapter、merge、推理延迟。第四，实验：哪些模型、哪些任务、对比哪些基线。第五，边界：小 rank 不是永远够，LoRA 依赖强底座，混合任务部署和配置选择也有代价。你讲任何一篇方法论文，都可以用类似骨架；只是 LoRA 这里尤其需要把公式和工程状态讲在一起。

错误复述“LoRA 证明了只要很小 rank 就能替代全量微调。”这句话过头，因为它没有限定模型、任务、指标和实验范围，也把“接近或超过部分基线”说成了“替代”。

合格复述“LoRA 在论文覆盖的多个 NLP 设置中，用低秩 adapter 取得了接近全量微调的质量，同时显著降低任务参数和 checkpoint 成本。”这句话保留了证据范围。

工程复述“我会先验 target 命中、冻结、保存加载和 merge 差异，再比较任务指标。”这说明你能把论文方法变成项目流程。

失败后果同样要能说出来。target module 写错，训练日志可能看似正常，但 LoRA 矩阵根本没接到关键层；base 没冻结，指标可能好看，却不再是参数高效微调；保存 adapter 时没记录 base 版本，别人无法复现；merge 后没做 diff，上线后输出漂移才发现 dtype 或重复 merge 有问题；复述时不讲边界，决策者会把 LoRA 当成无需验证的银弹。验收清单的意义就是在低成本阶段暴露这些问题。

交付时还需要一份很短的运行报告。报告不必写成长论文，但必须让下一个人能复现：底座模型名称和版本、数据版本、LoRA 配置、target modules、rank、alpha、dropout、训练步数、评估脚本、adapter 路径、merge 产物、关键指标和已知限制。少了这些字段，今天能跑通不代表下周还能解释结果。LoRA 的优势是 adapter 小、任务切换方便；如果元数据缺失，这个优势会被运维混乱抵消。

| 字段 | 必须写清楚的内容 | 为什么重要 |
| --- | --- | --- |
| 模型 | base model、revision、tokenizer、dtype | adapter 不是完整模型，底座错了就无法复现 |
| 配置 | r、alpha、dropout、target modules、bias 策略 | 决定训练对象和增量规模 |
| 训练 | 数据版本、步数、batch、学习率、seed | 用于解释波动和复跑差异 |
| 产物 | adapter 路径、merge 路径、评估脚本 hash | 避免保存了文件却不知道该加载哪一个 |
| 结论 | 指标、成本、适用范围、失败样例 | 防止把单次实验写成无限承诺 |

最后，把验收和复述连起来看。验收回答“这个 adapter 是否可信”，复述回答“这个方法为什么值得这样做”。如果你只能验收，说明你会跑项目但未必懂论文；如果你只能复述，说明你会讲概念但未必能落地。LoRA 学到这里，目标是两者合一：用论文解释为什么低秩适配可能有效，用工程检查证明你手上的 adapter 确实按这个方法工作。

一个完整收尾动作可以这样做：先打开训练配置，确认 target modules 与模型层名一致；再看参数打印，确认只有 LoRA 参数参与优化；然后跑小样本过拟合，确认梯度链路通；训练结束后保存 adapter，重新加载同一底座做 logits 对比；需要部署时再 merge，并保存 merge 前后的差异记录；最后写五句复述，把结论限定在你的数据和论文证据范围内。做完这些，你才真正拥有一个可解释、可复现、可交付的 LoRA 结果。

如果验收中出现异常，也要按层定位，不要直接推翻方法。参数名不对，先查 target modules；loss 不动，先查数据、标签和学习率；reload 后输出变了，先查底座版本、dtype 和 adapter 路径；merge 后漂移，先查 scaling、重复 merge 和精度转换；评估分不稳，先查样本量、seed 和错误样例。这样排查的好处是每一步都有明确责任对象。你现在应该能把“LoRA 效果不好”拆成若干可验证假设，而不是停在一句模糊抱怨。

最终复述也可以带上这套排查意识：LoRA 一种训练状态管理方法。它的贡献同时落在数学约束、工程产物和实验证据上；漏掉任意一侧，理解都会偏。会收尾的人，会把 adapter 文件、实验记录和论文语言放在一起交付。

收尾时再补一句人工检查：随机抽一个失败样例，确认它是模型能力问题、数据标注问题，还是输出格式问题。这个小动作能防止你只看平均分。

验收合格的标志，是别人能按你的记录重新加载、重新评估、重新解释。

这也是学习页最后要训练的真实能力。

练习时，请用“证据能支持到哪里”来约束表达。不要为了显得懂论文，把 LoRA 讲成万能方法；也不要因为它有边界，就忽略它在存储、切换任务和部署路径上的真实优势。你现在应该能完成三件事：写一份 LoRA 项目的最小验收清单，定位常见工程错误，并用一段有证据范围的话复述论文贡献。

### 06_01 短答

**Q:** 训练前为什么要打印可训练参数名？

Explain: 打印可训练参数名可以确认：1. target modules 是否命中正确层；2. 底座权重是否被冻结；3. 可训练参数量是否符合预期。如果 target modules 写错，可能 LoRA 完全没有生效，但训练仍会正常进行，只是效果很差。

### 06_02 短答

**Q:** target module 没命中会出现什么现象？

Explain: LoRA 层不会被初始化，模型行为等同于原始底座，训练时只有底座参数在更新（如果底座没冻结）。结果是：训练 loss 可能下降，但这不是 LoRA 的效果；验证时如果底座冻结，效果会回到原始水平。

### 06_03 短答

**Q:** 为什么不应该一开始就把 rank 拉到很大？

Explain: 大 rank 增加参数量和过拟合风险。如果数据集小或任务简单，大 rank 会记住训练数据而不是泛化。应该从较小 rank（如 4-8）开始，根据验证集表现和 SVD 方向分析逐步调整。

### 06_04 短答

**Q:** alpha 调整的是 LoRA 分支的什么？

Explain: alpha 调整 LoRA 分支的输出缩放：scale = alpha / rank。alpha 越大，LoRA 分支对最终输出的影响越大。如果 alpha 设置不当，可能导致训练不稳定（太大）或更新太弱（太小）。通常设 alpha = 2 * rank 作为起点。

### 06_05 多选

**Q:** LoRA 实验至少应该记录哪些 baseline 或对照？

- [ ] 原始底座表现
- [ ] LoRA 配置表现
- [ ] 可训练参数量
- [ ] 今天的天气

Explain: 按本组复盘提示自查：答案需要能回到训练对象、参数量、指标、状态或边界，而不是只给一个词。

### 06_06 场景

**Q:** LoRA 小数据训练过拟合，你会尝试哪些调整？

Explain: 1. 减小 rank 或增加正则化。2. 增加 dropout。3. 提前停止（early stopping）。4. 增加数据量或数据增强。5. 检查 target modules 是否过多，减少不必要的层。

### 06_07 判断

- [ ] 正确
- [ ] 错误

Answer: 错误
Explain: 对照本页正文里的训练对象、任务边界或部署状态。

### 06_08 短答

**Q:** 如何验证 merge 前后模型输出是否一致？

Explain: 1. 取一批测试输入，分别通过未 merge 模型和 merge 后的模型。2. 比较输出 logits 或生成文本的差异。3. 检查差异是否在数值精度范围内（如 fp16 误差）。4. 如果差异大，检查 merge 代码是否正确处理了 alpha 和 rank。

### 06_09 诊断

**Q:** 诊断：线上模型重复 merge 同一份 LoRA，输出明显变差。按错因、后果、修法回答。

Explain: 错因：merge 状态没有记录，每次切换任务时又从原始底座重新 merge，而不是先 unmerge。后果：多次 merge 把多份 LoRA 叠加到底座上，输出偏移越来越严重。修法：记录 merged/unmerged 状态；切换任务时先 unmerge 回到干净底座，或保持未 merge 状态动态加载。

### 06_10 诊断

**Q:** 诊断：LoRA adapter 加载到不兼容底座上。按错因、后果、修法回答。

Explain: 错因：LoRA 的 rank 和 target modules 与底座不匹配，或底座架构不同。后果：加载失败或运行时报 shape 不匹配；即使加载成功，输出也是错的。修法：检查底座名称、版本和架构；确认 LoRA 配置与底座兼容；加载前验证 shape。

### 06_11 场景

**Q:** 一个服务要频繁切换多个 LoRA，保存合并模型和动态加载各有什么取舍？

Explain: 保存合并模型：加载快，切换任务只需换模型文件；但每个任务需要单独存储完整模型，磁盘占用大。动态加载：只保存 LoRA 权重文件，切换时加载 LoRA 到共享底座；磁盘占用小，但切换有加载延迟，且需要管理底座 + LoRA 的组合。

### 06_12 短答

**Q:** 为什么量化或半精度场景下更要测试 merge 前后差异？

Explain: 量化或半精度会损失数值精度。merge 操作涉及矩阵乘法和加法，精度损失可能累积。如果 merge 前后差异大，说明当前精度设置下 merge 不可靠，需要提高精度或检查 merge 算法。

### 06_13 短答

**Q:** 列出 3 类适合优先尝试 LoRA 的任务。

Explain: 1. 领域适配：在通用模型上微调特定领域（如法律、医学）。2. 风格控制：调整模型输出风格（如正式、简洁、创意）。3. 任务增强：为特定 NLP 任务（如摘要、分类）提升性能。这些任务不需要改变模型底层知识，只需调动和放大已有能力。

### 06_14 短答

**Q:** 列出 2 类小 rank LoRA 可能不够的任务。

Explain: 1. 需要大幅改变底层知识的任务：如从通用模型变成代码专家，需要更新深层语义和语法表示。2. 需要多任务同时高表现的场景：小 rank 可能只能优化一个任务方向，多任务竞争时表现下降。

### 06_15 诊断

**Q:** 诊断：“LoRA 已经证明全量微调没有必要。”按错因、后果、修法回答。

Explain: 错因：把 LoRA 在特定任务上的高效性推广为所有场景都不需要全量微调。后果：在需要大幅改变模型底层知识的任务上失败，因为 LoRA 只能调动已有知识，不能重写基础能力。修法：理解 LoRA 的边界——适合调动已有知识，不适合从零建立新知识；在复杂任务上保留全量微调的选项。

### 06_16 短答

**Q:** 学习 QLoRA、DoRA 等后续方法前，为什么要先懂 Delta W=BA？

Explain: Delta W=BA 是 LoRA 的核心机制：低秩更新可以合并回原权重。QLoRA 在此基础上加入量化，DoRA 在此基础上分解为幅度和方向。如果不懂 Delta W=BA，就无法理解这些后续方法为什么有效、它们的约束是什么、以及它们与 LoRA 的关系。

### 06_17 面试追问

**Q:** 用 1-3 句话说明 LoRA 的核心贡献。

Explain: LoRA 的核心贡献是：用低秩矩阵 BA 近似任务更新 Delta W，只训练少量参数就能接近全量微调效果，且推理时可以把更新合并回原权重，不增加延迟。

### 06_18 清单

**Q:** 写一份 8 项 LoRA 训练验收清单。

Explain: 1. 可训练参数名正确，底座冻结。2. 训练 loss 下降，验证 loss 不过拟合。3. 验证集分数达到预期。4. merge 前后输出一致。5. 不同随机种子结果稳定。6. 目标模块和 rank 设置合理。7. 保存的 LoRA 权重可以加载到正确底座。8. 推理延迟符合预期。

### 06_19 费曼解释

**Q:** 用 8-10 句话完整复述 LoRA 论文：问题、方法、实验、机制解释和边界都要出现。

Explain: LoRA 论文解决的问题是：大模型全量微调参数量大、存储和部署成本高。提出的方法是：冻结预训练权重 W，只训练低秩矩阵 B 和 A，使得 Delta W = B A 近似任务更新。实验覆盖了不同模型规模和任务类型，证明 LoRA 用极少参数就能接近全量微调效果。机制解释通过 SVD 发现 Delta W 的方向与 W 的顶部奇异方向重合，说明小 rank 有效是因为任务更新只在已有知识上做放大。边界是：LoRA 适合调动已有知识，不适合需要大幅改变底层知识的任务；rank 选择和 target modules 需要根据任务调整。
