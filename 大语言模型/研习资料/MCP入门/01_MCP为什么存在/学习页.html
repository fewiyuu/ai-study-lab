Context Integration · Why MCP Exists

先看痛点：AI 为什么需要一套连接协议

聊天模型只会“说话”还不够。真正有用的 AI 应用经常要读文件、查数据库、调用搜索、打开工单、读取日历、执行代码或访问业务系统。问题在于，每个 AI 应用如果都单独写一套连接方式，每个外部系统也单独适配不同平台，集成很快会变成重复、脆弱、难审计的工程。

MCP 解决的是这层连接问题。它让 AI 应用和外部系统之间有一套共同语言：外部系统用 MCP server 暴露可用能力，AI 应用通过 MCP client 发现并调用这些能力，Host 负责把结果放回对话或工作流里。先理解这件事，后面看 tools、resources、prompts、transports 才不会散。

AI 应用→统一协议→数据/工具/工作流→可控行动

这一页学完，你要能做到

  2. 用一句话定义 MCP，并说明它标准化的是哪一层。

1. 上下文集成挑战：模型需要现实世界，但现实世界很分散

PROBLEM

上下文可以先理解成“模型这一次工作时能看到、能使用的信息”。它不只包括用户刚输入的问题，也包括文件内容、数据库记录、网页结果、工具执行返回值、历史对话、业务规则和操作权限。模型训练时学到的是一般知识，但你现在要它处理的往往是具体系统里的新数据。

旧式聊天界面的问题是：用户必须把信息手动贴进来，模型才能看到。这个方式在小问题上可行，一旦进入工程场景就很别扭。比如你让 AI 帮你排查线上报错，它需要 Sentry 错误、代码片段、部署版本、日志、配置差异和最近提交；靠人工复制粘贴，不仅慢，还容易漏掉关键证据。

更麻烦的是“行动”。查资料只是读，AI 应用还可能需要创建工单、写入表格、发起审批、改文件、运行测试。每个动作都涉及权限、参数、确认和审计。没有协议时，开发者只能给每个产品写自定义桥接代码。桥接越多，权限越散，出问题时越难追踪。

学到这里，你应该能说清：MCP 出现之前，AI 应用最痛的不是“不会聊天”，而是很难稳定、可控地连接外部系统。

2. MCP 的直觉定义：给 AI 应用接外部系统的共同语言

DEFINITION

MCP，全称 Model Context Protocol，直译是模型上下文协议。直觉上，它是一套开放标准，用来规定 AI 应用怎样和外部资源通信。外部资源可以是文件、数据库、API、工具、提示词模板，也可以是更复杂的业务工作流。

正式一点说，MCP 关注的是“上下文交换”和“能力调用”的协议层。它规定客户端和服务器如何建立连接、声明能力、列出工具、读取资源、调用工具、返回结果。它不规定你用什么大模型，也不规定 AI 应用要怎样推理、怎样管理长期记忆、怎样设计产品界面。

一个好判断句是：如果问题是“模型怎么连上外部系统并安全地使用它们”，就可能落在 MCP 的范围；如果问题是“模型权重怎么训练”“提示词怎么写得更好”“Agent 要不要多步规划”，那就不是 MCP 本身解决的事，虽然它会和这些问题配合。

问题MCP 是否直接负责原因
让 AI 应用读取本地文件列表负责连接和能力描述文件系统可以由 MCP server 暴露资源或工具。
让模型变得更聪明不直接负责MCP 不训练模型，也不改变模型参数。
让不同 AI 客户端复用同一个数据库连接能力适合标准接口让同一个 server 被不同 Host 接入。
决定回答语气是否礼貌不直接负责这更像 prompt、产品策略或模型行为问题。

学到这里，你应该能用一句话介绍 MCP：它是 AI 应用连接外部数据、工具和工作流的开放协议。

3. 旧做法：每个应用都写自己的连接器

BEFORE

没有统一协议时，一个 AI 应用想接数据库，就写一套数据库连接器；想接 GitHub，再写一套 GitHub 连接器；想接企业内部 API，又写一套鉴权、参数、错误处理和权限判断。另一个 AI 应用想用同样能力，还得再适配一遍。外部系统越多，AI 应用越多，重复工作会按组合数量增长。

这种模式的第一个后果是维护成本高。每个连接器都有自己的参数名、错误格式、权限检查、日志方式和版本兼容问题。第二个后果是复用差：一个团队写好的连接能力，很难自然被另一个客户端使用。第三个后果是安全边界散：API key 到处放，权限在不同代码里判断，审计时不知道某个动作究竟从哪条路径发生。

错误示例：把外部系统连接逻辑塞进 Host 代码Python
class AssistantApp:
    def answer(self, question):
        if "日历" in question:
            calendar_data = call_calendar_api(self.calendar_token, question)
        elif "数据库" in question:
            rows = query_database(self.db_password, question)
        elif "仓库" in question:
            issues = call_github_api(self.github_token, question)
        # 更多系统继续往这里堆，权限和错误处理也一起膨胀。

这段伪代码的问题不在语法，而在结构：Host 被迫知道太多外部系统细节。系统越多，主应用越臃肿，也越难判断哪些能力该暴露给模型。学到这里，你应该能识别“连接器散落在 Host 内部”这种结构风险。

4. MCP 的方案：把能力放到 server，用协议暴露给 client

SOLUTION

MCP 把连接方式拆开：AI 应用作为 Host，内部创建 MCP client；外部能力由 MCP server 暴露。Host 不需要把每个系统的细节都写死在自己代码里，而是通过 client 和 server 通信。server 告诉 client：我有哪些 tools、resources、prompts，需要什么参数，返回什么内容。

这带来一个很实际的变化：同一个 server 可以被多个 Host 复用。比如一个文件系统 server 可以被 Claude Desktop、IDE 插件或自定义 Agent 接入；一个企业内部知识库 server 可以被多个 AI 应用读取。开发者不再为每个组合写定制集成，而是围绕标准接口维护能力。

正确方向：Host 只管理 MCP 会话和能力注册Python
available_tools = []

for session in app.mcp_server_sessions():
    tools_response = await session.list_tools()
    available_tools.extend(tools_response.tools)

conversation.register_available_tools(available_tools)

# 模型决定调用某个工具时，Host 再把调用路由到对应 MCP server。

注意这里的关键不是“代码变短”，而是职责更清楚。Host 负责协调对话和权限体验；client 负责协议连接；server 负责把某个外部系统的能力描述清楚并执行。学到这里，你应该能画出“Host → Client → Server → 外部系统”的基本方向。

5. 安全边界：统一协议不等于放开所有权限

SECURITY

MCP 文档强调安全优先，这不是装饰语。AI 应用接外部系统时，风险来自两边：模型可能误用工具，外部工具可能能力过大，用户也可能不清楚某次调用会造成什么后果。协议能把能力描述、参数结构、调用结果和通知机制标准化，但具体权限仍然要由 Host、server 和部署环境认真设计。

一个健康的 MCP 集成应该能回答这些问题：这个 server 暴露了哪些工具；每个工具能读还是能写；参数是否验证；用户是否需要确认；凭证存在哪里；日志是否记录调用；不同用户看到的能力是否不同；server 出错时是否返回清楚的错误，而不是让模型猜。

好的边界

工具能力窄，参数明确，危险动作需要确认，凭证集中管理，调用可审计。模型看到的是受控能力，不是无限制 shell。

危险边界

server 暴露过大的写权限，工具描述含糊，参数不校验，用户确认缺失，API key 到处复制。协议存在，但风险仍然很大。

学到这里，你应该能判断：MCP 是安全设计的接口基础，不是自动安全按钮。真正的安全要看权限、确认、日志和部署方式。

6. MCP 不是什么：边界说清，学习才不跑偏

BOUNDARY

MCP 不是一个大模型。它不会让模型权重更强，也不会替代 tokenizer、推理服务或训练框架。MCP 也不是完整 Agent 框架。Agent 里的规划、记忆、任务拆解、反思、评估和人机协同，都可能使用 MCP 接入工具，但 MCP 不决定 Agent 该怎么思考。

MCP 也不等于普通 REST API。REST API 是外部服务提供的业务接口；MCP server 可以在里面调用 REST API，但它额外承担“面向 AI 应用描述能力”的工作，例如列出工具、提供输入 schema、暴露资源、返回结构化内容。你可以把 MCP server 看成 AI 应用和原始系统之间的适配层，但这个适配层有统一协议。

MCP 也不是插件目录本身。插件或注册表可以帮助发现和安装 server，但协议关注的是 client 与 server 如何通信。把这几个概念分开，你就不会把“装了一个 MCP server”“启用了一个插件”“模型会自动做事”混为一谈。

容易混淆准确理解判断句
MCP vs 模型MCP 是连接协议，模型是生成和推理组件。换模型不等于换 MCP；接工具也不等于训练模型。
MCP vs Agent 框架Agent 框架组织行动流程，MCP 提供外部能力接口。Agent 可以用 MCP，但 MCP 不负责完整规划。
MCP vs APIAPI 是外部服务接口，MCP server 可把 API 包装成 AI 可发现的能力。API 面向程序员，MCP server 还要面向 AI 应用描述能力。
MCP vs 插件插件负责安装或扩展入口，MCP 负责通信协议。插件可以包含 MCP server，但插件不是协议本身。

学到这里，你应该能在听到 MCP 时先问：这里说的是协议、server、工具能力，还是某个客户端里的插件配置？

7. 最小知识链路：一句需求如何落到 MCP

CHAIN

假设用户说：“帮我查一下今天日历里和客户有关的会议，然后把准备事项写到项目笔记里。”这个需求涉及读日历、判断会议、写文件。没有 MCP 时，Host 可能要直接集成日历 API 和文件系统 API；使用 MCP 时，可以由日历 server 暴露日程资源或工具，由文件系统 server 暴露写入能力。

链路可以这样拆：Host 接收用户目标；Host 中的 MCP client 连接日历 server 和文件系统 server；client 发现可用工具；模型在 Host 的控制下选择读取日历；server 返回日程内容；模型整理准备事项；用户确认后，Host 调用文件写入工具；文件系统 server 执行写入并返回结果。

这条链路的重点是“每一步都有边界”。模型不直接拥有你的电脑；它通过 Host 暴露的工具描述选择行动。server 不应该偷偷越权；它只执行自己声明并被允许的能力。用户不是完全旁观；危险或写入动作可以要求确认。学到这里，你应该能把一个 AI 外部行动需求拆成读什么、写什么、谁执行、谁确认。

8. 常见误区、后果和修法

PITFALLS

误区 1：MCP 会让模型自动变强

错因是把连接能力和模型能力混在一起。后果是接了很多 server，回答仍然可能不会推理。修法是分开看：MCP 解决接入，模型和应用逻辑解决推理与表达。

误区 2：有 MCP 就可以放心给写权限

错因是把协议当安全保证。后果是模型或用户误操作可能造成真实写入。修法是最小权限、参数校验、用户确认和审计日志。

误区 3：所有 API 都要包装成 MCP

错因是追新。后果是多一层维护成本，却没有复用收益。修法是先看是否需要被 AI 应用发现、调用、复用和审计。

误区 4：server 描述随便写也能用

错因是低估工具描述的重要性。后果是模型选错工具或传错参数。修法是写清工具用途、边界、输入 schema 和失败情况。

误区 5：把 MCP 等同某个客户端配置

错因是只见过 Claude Desktop 或 IDE 里的配置。后果是理解停留在安装层。修法是回到 Host、Client、Server 和协议消息。

误区 6：只关心工具，不关心资源和提示词

错因是把 MCP 当函数调用集合。后果是忽略可读上下文和可复用工作流模板。修法是同时理解 tools、resources、prompts 的不同角色。
