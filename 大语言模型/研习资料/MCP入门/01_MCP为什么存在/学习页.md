# 1. 先看问题：AI 应用为什么会卡在连接层 {#problem}
<!-- tag: 问题 -->

把上下文先理解成“这次工作里模型能看到、能使用、能引用的所有信息”。聊天记录只是其中一部分，真正做事时还会碰到文件、数据库、工单、日历、日志、配置、权限和返回结果。问题的关键在于，现实世界的信息和动作分散在很多系统里，接入方式又不统一。

如果每个 AI 应用都自己写一套连接器，团队会得到很多重复代码：今天写给桌面助手，明天再写给 IDE，后天再写给自动化机器人；每个连接器都要维护认证、参数、错误、日志、权限和版本。课程里说的碎片化，关键瓶颈在连接方式太散。

## box warn
痛点读什么写什么坏掉时的症状


连接重复每个应用自己的适配代码多份同义连接逻辑每接一个客户端就重写一遍
权限分散不同系统的账号与 token各处都管凭证泄露点变多，审计也散
能力不透明外部系统到底能做什么手写一堆 glue code模型选错接口或乱传参
结果难复用某次调用返回了什么把返回值塞进应用上下文工具已经成功，模型却像没看见

所以这一单元先把“为什么要统一连接”说清楚，再进入协议名词。你现在应该能做什么：看到一个 AI 应用需求时，先判断它是在解决推理问题，还是在解决接外部系统的问题。

# 2. 一句话定义：MCP 标准化的是连接层 {#definition}
<!-- tag: 定义 -->

MCP 可以先用一句话概括：它是一套让 AI 应用以统一方式发现、读取和调用外部能力的协议。这里的重点是统一方式；模型负责推理和表达，MCP 负责让外部系统以可发现、可调用、可审计的方式出现在 AI 应用面前。

如果再压紧一点，MCP 解决的是三件事：第一，外部系统不必为每个 Host 单独写适配；第二，Host 不必为每个系统单独学习私有接口；第三，结果、权限和边界可以用同一种协议表达出来。它像是把接线规则标准化，接口层统一，设备本身保持各自的工作方式。

## box info
层负责什么不负责什么


模型理解、推理、生成答案替你设计协议和权限
MCP标准化外部能力接入提升模型智力本身
应用编排流程、展示结果、确认风险代替外部系统的真实执行

这页后面所有例子都围绕这个边界展开：只要你看到“AI 应用要和外部系统稳定协作”，就把 MCP 放进候选列表；如果问题只是模型本身不会想，那先别把锅甩给协议。你现在应该能做什么：用一句话向别人解释 MCP 的职责边界。

# 3. 三个角色：Host、Client、Server 的职责不能混 {#roles}
<!-- tag: 角色 -->

课程里最容易混的地方，就是把 MCP 的三个词按普通 Web 语境去猜。这里要按职责理解，不按部署位置理解。Host 是用户实际使用的 AI 应用；Client 是 Host 内部的协议处理器；Server 是外部能力提供者。名字像网络，不代表一定是远程网络。

角色主要读什么主要写什么坏掉的症状


Host用户输入、工具结果、权限提示对话、任务编排、确认流程结果进了系统但没回到对话
ClientServer 描述、请求参数、会话状态协议消息、能力发现、调用请求连接成功但工具列表为空
Server请求、凭证、资源状态、参数工具结果、资源内容、错误说明能力说得清，执行却不稳


Host
它决定“谁能用、什么时候用、结果怎么回到用户视野里”。在 IDE、桌面助手、内部自动化里，Host 是那个真正面对人的应用。


Client / Server
Client 是协议会话；Server 是能力端。一个 Host 可以连多个 Server，每个 Server 只提供自己能稳定解释的能力边界。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

这一层要记住一个判断：如果你描述的是“谁在对用户负责”，那是 Host；如果你描述的是“谁在维护协议会话”，那是 Client；如果你描述的是“谁在把外部能力暴露出来”，那是 Server。你现在应该能做什么：看到一段架构说明时，先把三个角色各自的读写边界分开。

# 4. 能力类型：tools、resources、prompts 各有分工 {#capabilities}
<!-- tag: 能力 -->

课程里的第二个常见误区，是把所有能力都当成“调用函数”。MCP 里，tools、resources、prompts 的角色不同。tools 是做动作；resources 是给模型看的内容；prompts 是固定工作流或提示模板。把三者混在一起，会让模型看到的边界失真。

tools
适合“查一批数据”“创建一条记录”“执行一项受控动作”。它需要输入 schema、边界说明和失败返回。


resources
适合“读取文档”“查看目录”“拿到 schema 说明”。它更像上下文供应，不一定要触发动作。


prompts
适合重复格式稳定的工作流，比如纪要、审查、摘要、检查清单。它让 Host 复用一套提示组织方式。

能力类型速记复制

```text
tools = 让模型做事
resources = 让模型看见
prompts = 让模型按固定方式思考或表达
```

## box
能力输入关注点输出关注点常见失败


tool参数是否完整、是否越权结果、错误、审计线索字段缺失、动作过大
resource路径、过滤条件、可见性文本、结构化内容、片段读到了不该看的内容
prompt任务目标、格式、约束稳定可复用的提示模板格式漂移、指令太散

这一节要把名词和作用对应起来，判断“这项能力到底是在动手、在读内容，还是在组织工作流”。你现在应该能做什么：拿到一个外部能力需求时，先判断它属于 tool、resource 还是 prompt。

# 5. 一次会话怎么走：初始化、发现、调用、返回 {#lifecycle}
<!-- tag: 流程 -->

把一次 MCP 交互拆成四步最容易看清楚：先连起来，再发现能力，再发起调用，最后把结果带回 Host 的上下文。真正排错时，这四步任何一步坏了，表面症状都可能是“模型没有用上工具”，但根因并不一样。

## box
步骤谁在读谁在写检查点失败症状


初始化Host 读配置，Client 读 serverClient 建立会话server 命令、地址、环境变量是否存在找不到进程或握手失败
能力发现Client 读 tool/resource/prompt 描述Client 缓存可用能力列表是否为空、schema 是否可读连上了但看不到工具
调用Host 读用户意图，Client 读参数Server 执行动作输入是否完整、是否越权参数错、权限拒绝、结果不稳
返回Host 读结果Host 把结果放回上下文结果是否进入对话工具成功了，但答案里没出现

<!-- interactive: ['terminal-lab'] → keep as HTML -->

这个流程里最重要的是把“连接成功”与“可用能力”分开，也把“server 返回结果”与“Host 把结果纳入回答”分开。你现在应该能做什么：看到一段 MCP 故障描述时，先说出它卡在初始化、发现、调用还是返回哪一步。

# 6. 状态跟踪：把一次工具调用拆成读写表 {#trace}
<!-- tag: 跟踪 -->

MCP 有会话状态、权限状态和结果状态。只要有状态，就有“读了什么、改了什么、哪些内容已经过期”的问题。下面这张表适合手工排查：每一步都明确写出读、写、失效点和症状。

## box
步骤读什么写什么什么时候失效错了会怎样


Host 决策用户目标、当前对话是否允许调用、是否要确认用户改口或切换任务后误触发高风险动作
Client 连接server 配置、初始化状态会话、能力缓存server 重启、版本变更后看见旧工具列表或握手失败
Server 执行参数、权限、资源状态结果、日志、资源变更写入后立即或某个事务结束后返回成功但实际未完成
Host 回写tool result对话上下文、下一轮推理依据如果没回写，就一直是空的模型像没看到结果

## box info
```text
先看连接是否活着
再看能力是否发现
再看参数是否正确
最后看结果有没有回到 Host
```


这张账本的意义在于：它逼你把“看不见的东西”变成可检查的读写关系。你现在应该能做什么：遇到工具调用故障时，按状态账本逐步定位，不要笼统地说“协议有问题”。

# 7. 安全边界：统一协议不等于放开权限 {#security}
<!-- tag: 安全 -->

课程反复强调的一件事是：协议可以统一接口，不能替你自动解决安全。MCP server 暴露什么能力、允许写到哪里、需要不要确认、日志怎么记，这些仍然要自己设计。协议只把边界说得更清楚，不会把边界自己变安全。

## box
边界应该允许不应该允许风险信号


读取范围只读当前用户可见的数据跨租户、跨账户、跨项目读取“顺手全拿来给模型看看”
写入范围明确路径、明确对象、明确确认任意路径、静默覆盖、默认删除“先给大权限，后面再收”
参数校验字段、类型、必填项、范围都清楚任意字符串都能进执行层工具说明只写“do task”
审计记录谁在什么时候做了什么只有成功/失败，没有上下文出问题后无法追踪

安全检查清单复制

```text
1. 只暴露必要能力
2. 输入先校验再执行
3. 高风险操作要确认
4. 结果和错误都要能审计
5. Server 不替代 Host 的用户体验
```

这一节最要防的错觉，是把“协议标准化”误读成“安全自动完成”。它不会。你现在应该能做什么：看见一个 MCP server 方案时，先问它的读取范围、写入范围、确认方式和审计方式。

# 8. 常见误区：先把边界说清楚，后面才好用 {#mistakes}
<!-- tag: 误区 -->
<!-- layout: grid2 -->

## box
### 误区 1：MCP 会让模型自动变强
错因是把接入能力和推理能力混在一起。后果是接了很多 server，答案还是不稳。修法是分开看模型、应用和协议。

## box
### 误区 2：有协议就可以放开所有权限
错因是把规范当安全。后果是高风险写入被误触发。修法是最小权限、参数校验、确认和审计一起做。

## box
### 误区 3：所有 API 都值得包装
错因是追求统一而忽略收益。后果是维护层变厚，但复用收益很小。修法是先看是否会被多个 AI 应用长期共享。

## box
### 误区 4：只关心工具，不关心资源和提示词
错因是把 MCP 当函数集合。后果是可读上下文和工作流模板被忽略。修法是同时考虑 tools、resources、prompts。

## box
### 误区 5：把 client 当成用户端，把 server 当成远程机器
错因是按普通 Web 直觉读名词。后果是架构图反了。修法是回到职责，别先猜部署位置。

## box
### 误区 6：结果返回了，模型一定会用上
错因是把执行层和对话层混为一体。后果是工具已经成功，回答却像没发生。修法是检查 Host 是否把结果回写到上下文。

这些误区用于排错。只要你能把错因、后果、修法说出来，就已经比只会复述定义更接近能动手的状态。你现在应该能做什么：看到一个新方案时，先指出它最容易踩的两三个边界错误。

# 9. 采用判断：什么时候值得用 MCP，什么时候先别用 {#decision}
<!-- tag: 判断 -->

课程 03 的主线，在这里先压成一句操作判断：当一个外部能力需要被多个 AI 应用稳定复用，而且涉及权限、审计、可发现性时，MCP 的价值会明显上升；如果只是一次性脚本、单个点对点接口、低风险临时任务，先用更轻的方式通常更合理。

再往前推一步，如果你还说不清这项能力到底读什么、写什么、谁来确认、谁负责留痕，那就先不要急着包装成 MCP。很多项目卡住的地方，能力边界本身还没有理顺。先把边界说清楚，再决定要不要加协议层，后面接入和排错都会省很多力气。

## box soft
现在就要会先放一放后面再回来


为什么需要统一连接层传输层怎么编02 里看调用链和实现边界
Host / Client / Server 的职责进程托管方式02 里看部署形态和状态细节
tools / resources / prompts 的分工协议扩展字段03 里看接入与取舍
是否值得多加一层协议具体鉴权实现真正接入时再定

## box
场景特征更像 MCP更像普通 API / 脚本


要给多个客户端复用是否
需要明确权限和审计是可选
只做一次临时任务否是
能力边界窄、定义稳定是也可直接暴露 API
高风险写入很多要先收紧再决定先不要直接放开

## box soft
```text
先问会不会复用
再问有没有权限和审计
最后问是不是值得多一层协议
```


这里的结论是“值得用时再用”。MCP 的价值来自复用和治理，来自把外部能力放进统一入口。你现在应该能做什么：拿到一个外部能力需求时，先给出“采用、暂不采用、先原型”中的一个判断，并说明理由。

# 10. 端到端案例：共享日历查询怎么落成 MCP server {#case-study}
<!-- tag: 案例 -->

现在把前面的抽象压到一个具体场景：公司里有三个 AI 客户端都要查团队日历。桌面助手负责给用户整理会议准备事项，IDE 助手负责在提交前提醒是否撞会，日报机器人负责汇总当天安排。三者都需要读取同一类数据，但它们面对的用户不同，调用时机不同，展示方式也不同。如果每个客户端各自写一套日历连接器，认证、过滤字段、返回格式、审计位置都会散开，后面改权限时还得改三次。

把这个能力收进一个 MCP server 后，设计就会变得更稳。Host 只需要决定用户是否允许这次读取、允许读取多长时间范围、是否要显示敏感字段；Client 负责把请求稳定地发给 server；Server 负责从日历系统里拿出结构化事件，再按约定格式返回。这样的设计真正有价值的地方在于，多端复用一次，边界统一一次，审计记录也统一一次。

它不是把现成 API 外面套个名字就结束了。真正要收住的是契约：工具名要稳定，参数要能解释，返回值要能被后续步骤继续使用，权限也要能按场景收紧。只要这些东西还在变，临时脚本就会在不同客户端里反复长出来；当这些东西稳定下来，MCP 才开始省维护。

## box info
步骤读什么写什么结构变化检查点


Host 收到请求用户目标、当前身份、时间范围是否允许调用、是否提示确认从自然语言变成受控调用意图用户到底要查哪一段日历
Client 组装请求server 描述、工具 schema、上下文协议消息从“查日历”变成带字段的调用字段是否完整、是否超出范围
Server 执行查询日历存储、权限、过滤条件结构化事件列表从原始记录变成对模型友好的事件对象是否只返回用户可见内容
Host 回写结果tool result对话上下文、摘要、下一步动作从执行结果变成可继续推理的材料模型能不能看到返回内容

日历工具草图复制

```json
{
 "name": "calendar.search",
 "description": "读取当前用户可见的团队日历事件",
 "inputSchema": {
 "type": "object",
 "properties": {
 "startDate": { "type": "string" },
 "endDate": { "type": "string" },
 "keywords": { "type": "array", "items": { "type": "string" } },
 "includePrivate": { "type": "boolean", "default": false }
 },
 "required": ["startDate", "endDate"]
 }
}
```

这个草图的重点在输入和输出的边界。startDate 和 endDate 让调用范围可控，keywords 让筛选条件可解释，includePrivate 则提醒你：敏感字段必须默认关闭。返回值也要尽量结构化，例如 title、start、end、location、attendees、sensitivity，避免让 Host 先从一大段自然语言里重新找字段。结构化结果更容易被摘要、排序和过滤。

## box
失败模式症状根因修法


输出太自由模型分不清标题、时间、地点server 返回整段自然语言改成结构化事件数组
范围太大查到不相关甚至敏感的会议默认读取范围过宽把日期范围和可见性写死在 schema 里
权限太松一个客户端能看到另一个场景的数据Host 没有按用户和场景收紧入口在 Host 侧加确认和权限过滤
重复适配三份代码都要同步改每个客户端各写一套连接器把能力收回一个 server

这个案例还有一个实用判断：如果你的调用只会在一个脚本里用一次、也不需要多个客户端复用，那直接调普通 API 往往更轻；如果你已经能预见多个 Host 要共用同一套日历访问、还要统一权限和审计，那就更接近 MCP 的价值区间。你现在应该能做什么：把一个共享日历需求拆成 Host、Client、Server 三层，并说出哪一步最容易出错。

# 11. 写入型变体：当能力不只是读取时，边界要再收紧 {#write-path}
<!-- tag: 写入 -->

读取型能力的风险主要在“看见了什么”；写入型能力的风险多了一层，因为它会改变真实系统。比如同样是日历场景，如果你把能力从“查会议”扩展到“创建会议”“改时间”“撤销邀请”，那么输入校验、确认流程、幂等性和审计记录就都要重新设计。这里不能只看功能能不能跑，还要看是否能安全地重复执行。

一个实用的设计办法，是把写入动作拆成两层：第一层只接受足够小的结构化参数，例如会议标题、时间段、参与者、地点和时区；第二层在 Host 里做确认，把用户看得见的摘要和真实执行参数放在一起，让用户确认后再发给 server。这样做的目的是防止模型在看见多个相近选项时直接走到危险写入，而没有任何人工确认。

## box
阶段读什么写什么需要额外检查什么失败后会出现什么


提案用户意图、现有日程、冲突信息待确认的执行草案是否真的要写入模型先替用户做了决定
确认草案摘要、参数、影响范围确认结果用户是否看懂将要发生什么确认过快，实际参数却错了
执行确认后的参数、当前权限创建或修改结果幂等性、冲突处理、审计重复创建、时间冲突、无法回滚
回写执行结果、事件 ID、错误说明对话上下文、后续操作建议结果是否清楚到可以继续追问用户不知道到底改没改成功

写入前确认模板复制

```text

标题：团队复盘
时间：6 月 24 日 15:00-16:00
参与者：A、B、C
地点：线上会议室

```

如果这一步没有做稳，典型症状是“会议被改错”“邀请多发一次”“用户不清楚改了什么”。这类问题很少靠模型自己解决，必须靠 Host 的确认面、server 的校验和日志一起处理。你现在应该能做什么：把一个读取型能力和一个写入型能力的风险点分开说清楚。

# 12. 版本变更：工具改了，缓存和调用也要跟着改 {#versioning}
<!-- tag: 版本 -->

前面看的是“这次调用怎么走”，这一节看“工具变了以后会发生什么”。MCP 场景里，server 升级、工具重命名、输入字段变化、返回结构变化，都会让旧客户端出现隐性故障。最常见的症状连得上、列表也看得到，真正调用时却总是报错或拿到旧字段。

一个典型场景是：日历 server 从 `calendar.search` 升级成 `calendar.search_v2`，把 `startDate` 和 `endDate` 改成了 `range.start` 与 `range.end`。如果 Host 侧还缓存着旧 schema，模型就会按旧字段发起调用，结果看起来像“工具坏了”，实际上是版本不一致。排查时先看工具描述版本，再看客户端缓存刷新，再看模型得到的 schema 是否更新。

这里最容易踩的坑，是把“工具还在”误当成“调用契约还在”。只要字段名、必填项、返回结构或权限策略有一个没同步，问题就会出现在最不显眼的地方：有时是参数错位，有时是摘要层读不到新字段，有时是旧缓存一直把模型往错误方向带。版本管理的核心让 Host、Client 和 Server 在同一版契约上说同一种话。

## box

工具名calendar.searchcalendar.search_v2旧调用直接找不到入口先看工具列表是否刷新

返回结构字符串摘要事件数组Host 摘要逻辑要更新再看结果是否回写正确
权限策略默认可读按场景收紧旧客户端会碰到拒绝最后看确认面与日志

旧新字段对照复制

```json
{
 "calendar.search_v2": {
 "range": {
 "start": "2026-06-21T00:00:00+08:00",
 "end": "2026-06-21T23:59:59+08:00"
 },
 "keywords": ["周会", "复盘"]
 }
}
```

这个例子最想让你记住的是：调用失败不一定是能力不存在，也可能是版本没同步、缓存没刷新、模型拿到旧 schema。把问题拆开以后，排错顺序就很明确了。先确认 server 当前暴露什么，再确认 Host 缓存了什么，再确认模型实际看到什么，最后才看业务逻辑有没有偏差。你现在应该能做什么：遇到“工具明明在，但模型老是调错”的问题时，先检查版本和缓存。

**Q:** 1. MCP 主要解决哪类问题？

### 2 判断

**Q:** 2. MCP 本身是一种大语言模型。

- [ ] 对
- [ ] 错

Answer: 错
Explain: MCP 是协议。它让模型更容易接入外部能力，但不替代模型本身。

### 3 简答

**Q:** 3. 用一句话定义 MCP。

Explain: 要点至少包含“标准协议”“外部能力”“AI 应用”三部分，例如：MCP 是让 AI 应用以统一方式发现和调用外部能力的协议。评分时看是否说清连接层边界，以及是否把协议和模型区分开。

### 4 解释

**Q:** 4. 本节里的“上下文”不只指聊天记录，还包括哪些东西？列 4 个。

Explain: 可以包括文件内容、数据库记录、日志、配置、工单、日历、工具返回值、权限提示等。重点是看是否把“信息来源”说成了“聊天记录之外的现实系统”。

### 23 synthesis

**Q:** 23. 如果同一日历能力要给桌面助手和 IDE 共用，你会把哪些事情放在 Host，哪些事情放在 Server？

Explain: Host 侧负责用户确认、场景切换、结果展示和上下文回写；Server 侧负责连接日历系统、过滤可见数据、返回结构化事件和记录审计。好答案会明确区分“用户体验”与“外部执行”。

- [ ] 需要统一审计和权限

Answer: 只做一次临时任务
Explain: 单次低风险、不会复用的任务，先用普通 API 或脚本更轻。

### 7 解释

**Q:** 7. 为什么说 MCP 不是完整 Agent 框架？

Explain: MCP 只解决外部能力接入与发现，不负责规划、记忆、反思、长任务管理或完整执行编排。回答时要把“协议层”和“智能体层”分开。

### 8 解释

**Q:** 8. 每个 AI 应用都写自己的连接器，会带来哪 3 个问题？

Explain: 常见答案包括：重复开发、维护成本高、权限和日志分散、跨客户端难复用、协议不统一导致工具描述不稳定。不要只写“麻烦”。

### 9 排序

**Q:** 9. 排序：初始化 / 能力发现 / 调用 / 返回。

- [ ] 初始化->能力发现->调用->返回
- [ ] 调用->初始化->返回->能力发现
- [ ] 返回->调用->能力发现->初始化

Answer: 初始化->能力发现->调用->返回
Explain: 正确顺序是先初始化，再发现能力，再发起调用，最后返回结果。

### 10 代码阅读

**Q:** 10. 一个工具 schema 为什么要写 required 字段？

Explain: required 字段让调用方知道哪些参数必须提供，减少模型乱猜和 server 执行时的缺参失败。好答案会把“参数完整性”和“模型选参稳定性”都写出来。

### 11 诊断

**Q:** 11. 工具描述只写“do task”，可能造成什么后果？

Explain: 后果通常是模型选错工具、传错参数、无法判断边界。修法是把用途、输入、输出、限制和失败情况写清楚。

### 12 诊断

**Q:** 12. Server 成功返回内容，但模型回答像没看到结果。问题可能在哪？

Explain: 常见问题在 Host：结果没有被写回对话上下文，或者被错误过滤、截断、丢进了别的状态里。回答要指出“执行成功”和“结果进入上下文”是两件事。

**Q:** 13. 哪个场景更适合 MCP？

- [ ] 多个 AI 客户端需要复用同一个知识库查询能力
- [ ] 一次性导出一个公开 CSV
- [ ] 本地临时算一道数学题

Answer: 多个 AI 客户端需要复用同一个知识库查询能力
Explain: 多客户端复用、边界稳定、权限明确时，MCP 的价值更大。

### 14 判断

**Q:** 14. Quickstart 跑通后，就可以直接把 server 放到生产环境。

- [ ] 对
- [ ] 错

Answer: 错
Explain: 跑通 quickstart 不等于已经适合生产；还要补权限、审计、监控和版本治理。

### 15 清单

**Q:** 15. 判断是否采用 MCP 时，你会看哪 5 个条件？

Explain: 可以从复用、权限、审计、边界稳定性、跨客户端需求、维护成本、是否需要可发现性来答。关键是别只写“看需求”。

### 16 反例

**Q:** 16. 举一个不急着用 MCP 的场景，并说明原因。

Explain: 比如一次性本地脚本、单用户临时任务、没有复用价值的简单 API。判断时要解释为何轻量方案更省成本，并点出复用、权限或审计为什么不构成硬需求。
