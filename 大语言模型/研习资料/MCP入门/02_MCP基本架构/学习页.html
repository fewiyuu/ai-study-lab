Architecture · Roles And Message Flow

把 MCP 看成一条受控的外部能力通道

上一节说清了 MCP 为什么存在。这一节开始看结构。你不需要一上来背规格细节，先把四层关系看清：用户使用 Host，Host 里有 MCP Client，Client 和 MCP Server 通信，Server 再连接文件、数据库、API 或系统工具。

这个结构的价值在于职责分开。Host 不必把所有外部系统细节写死；Server 不需要知道具体大模型是谁；Client 负责按协议发请求、收响应；用户和 Host 可以围绕工具权限、确认和结果展示做控制。后面学任何 MCP server，都可以用这张图定位它在哪一层。

Host→Client→Server→外部系统

这一页结束时，你要能做到

  1. 解释 Host、Client、Server 的职责差异。
  2. 区分 tools、resources、prompts、sampling 和 transports。
  3. 按顺序说出一次 MCP 会话从初始化到工具调用的过程。
  4. 看到报错时，初步判断是配置、传输、能力发现、参数还是权限问题。

1. 架构图：四层关系先放稳

MAP

这张图不是装饰。以后遇到 MCP 问题，先问：Host 是否找到了 server 配置；Client 是否成功建立连接；Server 是否声明了能力；调用参数是否符合 schema；外部系统是否执行成功。很多报错都能沿着这条线定位。

你现在应该能做什么：看到一个 MCP 配置文件或日志，先判断它涉及 Host、Client、Server 还是外部系统。

2. 三类角色：不要把 Host、Client、Server 混在一起

ROLES

Host

Host 是用户实际使用的 AI 应用，例如桌面聊天应用、IDE、内部助手或自动化 Agent。它负责管理用户界面、模型调用、对话状态、权限体验和工具结果展示。Host 可以同时连接多个 MCP server。

MCP Client

Client 是 Host 内部的协议处理组件。它负责和某个 MCP server 建立会话、发送请求、接收响应、处理通知。你平时不一定直接看到它，但所有能力发现和工具调用都要经过它。

MCP Server

Server 是外部能力的提供者。它可以包装文件系统、数据库、浏览器、代码仓库或业务 API。它负责声明自己能做什么，校验输入，执行动作，返回结果。

一个容易误会的地方是：server 不等于远程云服务器。MCP server 可以是本地进程，也可以连接远程服务；关键不是部署位置，而是它按 MCP 协议向 client 暴露能力。同样，client 也不等于“前端用户”。client 是 Host 内部的协议端。

你现在应该能说清：Host 是用户正在用的应用，Client 是 Host 里的协议连接器，Server 是能力提供方。

3. 能力类型：tools、resources、prompts 各管一类事

CAPABILITIES
能力直觉典型用途判断句
Tools可调用动作查询数据库、创建工单、写文件、运行脚本如果它会“做一件事”并可能改变状态，多半是 tool。
Resources可读取内容文件、文档、表结构、日志片段、配置内容如果它主要是“给模型看资料”，多半是 resource。
Prompts可复用提示模板代码审查模板、会议总结模板、排查流程模板如果它是“怎么提问或怎么组织任务”的模板，多半是 prompt。
Samplingserver 请求 Host 让模型生成server 需要模型帮忙做中间推理或补全文本如果 server 反过来需要模型输出，就涉及 sampling。
Transports通信通道stdio、HTTP 类连接、事件流如果问题是“消息怎么传”，就在 transport 层。

最常见的混淆是把所有能力都叫工具。这样会让设计变粗糙：本来只是给模型读一个文件，却写成一个“读取动作”；本来是固定的工作流模板，却每次让模型临时猜提示词。MCP 把这些能力分开，是为了让 Host 和模型知道每种能力应该怎样展示、选择和调用。

你现在应该能做什么：看到一个能力描述，判断它更像 tool、resource 还是 prompt。

4. 一次会话怎么开始：初始化、能力发现、调用、返回

LIFECYCLE

一次 MCP 会话可以先按四步理解。第一步，Host 根据配置启动或连接 server，并由 client 发起初始化。第二步，双方交换协议版本和能力信息。第三步，Host 通过 client 请求 server 列出 tools、resources 或 prompts。第四步，模型或用户触发某个能力时，Host 把调用请求发给 server，server 执行后返回结果。

这个流程看似普通，但对排错很有用。如果 server 根本没启动，能力列表当然为空；如果初始化版本不兼容，后面调用不会稳定；如果工具列表能看到但调用失败，问题可能在参数 schema、权限或外部系统；如果 server 返回了结果但模型没用上，问题可能在 Host 的结果注入或提示词。

逐行拆解：一次工具调用的抽象流程pseudo
Host starts MCP server process
Client sends initialize request
Server replies with protocol version and capabilities
Client asks: list available tools
Server replies: search_docs, create_ticket, read_file
Model chooses: search_docs({"query": "refund policy"})
Client sends tool call to Server
Server executes and returns structured result
Host passes result back into the conversation

你现在应该能按顺序复述：先连接和初始化，再能力发现，再调用，再把结果交回 Host 和模型。

5. Tools：动作要有名字、说明、输入结构和边界

TOOLS

Tool 是 MCP 中最容易让人兴奋的一类能力，因为它能让 AI 应用“做事”。但越能做事，越需要描述清楚。一个好工具至少要说明：名字是什么，适合什么时候调用，需要哪些参数，参数类型是什么，会不会写入或删除东西，失败时怎么返回。

工具描述不是给人看的注释而已。Host 可能会把工具信息交给模型，让模型判断是否调用。如果工具名含糊、描述空泛、输入 schema 不准，模型就容易误选工具或传错参数。比如 do_action 这种名字几乎没有信息；search_customer_tickets 至少让人知道它是搜索客户工单，不是创建或删除。

正确示例：工具说明要收窄能力JSON
{
  "name": "search_customer_tickets",
  "description": "Search read-only customer support tickets by keyword and date range.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "keyword": {"type": "string"},
      "date_from": {"type": "string"},
      "date_to": {"type": "string"}
    },
    "required": ["keyword"]
  }
}
错误示例：名字和边界都含糊JSON
{
  "name": "do_task",
  "description": "Do whatever the user asks.",
  "inputSchema": {
    "type": "object"
  }
}

你现在应该能判断一个 tool 描述是否足够清楚：名字要具体，说明要写边界，输入结构要能约束模型。

6. Resources：给模型看的内容，不一定要变成动作

RESOURCES

Resource 更像“可读取的上下文”。它可以是某个文件、一段文档、数据库 schema、日志、图片元数据或其他内容。它的重点不是执行动作，而是让 Host 或模型知道这里有资料可读。很多场景下，把内容作为 resource 暴露比写一个“读取工具”更自然。

举个例子，代码仓库里的 README、配置文件、数据库表结构，都可以作为资源。模型读资源后再决定下一步。如果把所有资源读取都做成工具，调用会变得像一串动作；如果把稳定内容暴露成 resource，Host 可以更清楚地展示、缓存或引用。

最小例子：资源描述关注 URI 和内容类型JSON
{
  "uri": "file:///workspace/README.md",
  "name": "Project README",
  "description": "High-level project setup and usage notes.",
  "mimeType": "text/markdown"
}

Resources 的边界也要清楚。读资源可能看起来安全，但资源里可能包含 secret、个人信息或内部数据。server 仍然要按用户、路径、权限和数据类型做限制。你现在应该能做什么：把“给模型看资料”和“让模型执行动作”分开设计。

7. Prompts 与 Sampling：工作流模板和反向模型请求

PROMPTS

Prompts 是可复用的提示模板。它适合把常见工作流做成稳定入口，例如“总结会议纪要”“生成代码审查报告”“按公司格式整理客户反馈”。它不是普通文档，也不是工具动作，而是把任务说明、参数和输出格式组织起来，减少每次临时写提示词的不稳定。

Sampling 则更特殊：server 可以请求 Host 让模型生成内容。直觉上，这是 server 对 Host 说：“我现在需要模型帮我完成一段生成或判断。”这类能力要谨慎，因为它让 server 反过来使用模型资源，也可能涉及上下文和权限。你刚入门时不必深挖细节，但要知道它和 tool call 的方向不同。

Prompt 的好用场景

任务格式稳定，组织方式重复，用户经常需要同一类输出。比如固定格式的周报、代码审查、学习复盘。

Sampling 要多问一句

server 为什么需要模型生成？会带入哪些上下文？谁支付成本？用户是否知道？这些问题不清楚时，不要随手打开。

你现在应该能判断：prompts 是工作流入口，sampling 是 server 请求 Host 调模型，二者都不是普通工具调用。

8. Transports：消息怎么传，不等于能力是什么

TRANSPORTS

Transport 是通信通道。MCP 支持不同传输方式，常见理解可以先分成本地进程通信和网络通信。比如本地 server 可能通过标准输入输出和 Host 通信；远程或长连接场景可能用 HTTP 类方式或事件流。传输方式影响部署、认证、延迟和调试，但不改变 tool、resource、prompt 的基本含义。

新手常把“server 怎么启动”和“server 暴露什么能力”混在一起。配置里看到 command、args、url、headers 这类字段，多半是在处理启动和传输；看到 list_tools、inputSchema、resource URI，才是在看能力。排错时也要分层：连接不上先看 transport 和配置；连接上但工具缺失，再看 server 能力声明。

最小例子：本地 stdio server 的配置长相JSON
{
  "mcpServers": {
    "filesystem": {
      "command": "node",
      "args": ["./filesystem-server.js", "/workspace"]
    }
  }
}

这段配置只说明 Host 如何启动并连接 server，不说明 server 内部有哪些工具。能力要在连接成功后由 server 声明。你现在应该能把“连接配置问题”和“工具能力问题”分开排查。

交互观察：一次调用卡在哪一层

INTERACTIVE

点击不同阶段，观察常见症状和排查方向。

1. 配置启动
Host 找到 server 配置，启动本地进程或连接远程地址。
2. 初始化
Client 和 Server 交换协议版本与能力。
3. 能力发现
Host 请求 tools/resources/prompts 列表。
4. 工具调用
模型选择工具，Host 发送参数，Server 执行。
5. 结果注入
Host 把返回结果交回对话，模型继续回答。

配置启动

常见症状是 server 找不到、命令路径错误、工作目录不对、环境变量缺失或网络地址不可达。

观察提示：先定位阶段，再改配置。不要看到工具不能用，就直接重写 server。

9. 常见坑：架构名词背会了，调用还是会错

PITFALLS

把 client 当用户端

错因是按普通 Web 习惯理解 client。后果是架构图读反。修法：记住 MCP client 通常在 Host 内部，是协议连接器。

把 server 当远程机器

错因是被 server 这个词带偏。后果是忽略本地进程 server。修法：按“能力提供方”理解 server，而不是按部署位置理解。

用 tool 表达所有内容

错因是只熟悉函数调用。后果是资源读取和工作流模板都变成动作，体验混乱。修法：读内容优先想 resource，固定流程想 prompt。

工具 schema 太宽

错因是为了省事允许任意对象。后果是模型传参不可控。修法：明确字段、类型、必填项和边界。

连接失败就怀疑模型

错因是没分层。后果是排错方向错。修法：先看配置、进程、传输和初始化，再看工具选择。

返回结果不给模型

错因是工具执行和对话注入断开。后果是 server 成功了，模型却像没看到结果。修法：检查 Host 如何把 tool result 放回上下文。
