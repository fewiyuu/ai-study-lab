02 · 角色、能力与消息流

MCP 基本架构

这一页只讲一件事：把 MCP 看成一条可排查的消息链。你先把 Host、Client、Server、tools、resources、prompts、transports 和会话状态分开，后面看真实 server 时，就知道该看配置、看 schema、看回写，还是看权限。

读完本页应该能做什么

  • 能把一段 MCP 描述拆成 Host、Client、Server 三层，并说出各自读写什么。
  • 能判断一项能力是 tool、resource 还是 prompt，并解释为什么。

1. 先把对象放稳：这一页在讲什么

概览

上一页说的是为什么 MCP 会出现，这一页开始看它怎么工作。最容易学乱的地方有两个：一是把角色名按普通 Web 直觉理解，二是把 tools、resources、prompts 都当成“可以调用的接口”。先把这两层分开，后面看任何 server 时才不会只记住表面名词。

你可以先拿一个最小场景练手：Host 里有一个 AI 应用，Client 负责和 MCP server 说话，Server 提供一个“查项目文档”的能力。先看谁负责用户体验，谁负责协议消息，谁负责外部能力。

对象最小任务最先看什么错了会怎样
Host决定要不要调用外部能力用户意图和确认体验高风险动作被直接触发
Client建立会话、发现能力、发请求server 描述和工具列表连上了却看不到可用能力
Server执行查询或动作并返回结果输入参数和权限边界能力看起来存在，执行却不稳定

这里先不追协议细节、传输实现、SDK 版本兼容和具体厂商封装。先把结构和消息流看懂，再去读真实 server 的配置和实现。

2. 架构图：先看消息怎么流

架构图

学习 MCP 时，最有用的是消息流。你可以把它理解成一条有边界的管道：用户目标进入 Host，Host 把意图交给内部的 Client，Client 去找 Server，Server 执行后返回结果，Host 再把结果放回对话或工作流里。这里真正要检查的是“在哪一步变成了结构化调用，在哪一步又回到了自然语言回答”。

Host 前态：只有用户意图和当前对话。

后态：决定是否要调用，是否要确认，结果是否回到用户视野。

Client 前态：只有 server 配置。

后态：建立会话、得到能力列表、把调用请求发出去。

Server 前态：等待参数和权限。

后态：返回工具结果、资源内容或拒绝信息。

这张图的用途很朴素：把“连接成功”“能力可见”“参数正确”“结果回写”拆成四件事。很多故障出在你把这四件事混在一起。

3. 三类角色：Host、Client、Server 各管哪一段

角色

这里按职责理解，不按部署位置理解。Host 是面向用户的 AI 应用，Client 是 Host 内部负责协议连接的那一层，Server 是把外部能力暴露出来的那一层。你读到任何一个 server，先问它属于哪一层，再看它部署在哪儿。

角色读什么写什么典型症状
Host用户意图、工具结果、确认信息对话、流程编排、回写上下文结果进了系统，但没回到回答里
Clientserver 描述、能力列表、会话状态协议消息、调用请求已经连上，却读不到工具
Server参数、权限、资源状态工具结果、资源内容、拒绝原因能力说得清，执行却不稳

先别把 Agent、编排器、插件框架和部署拓扑混进来。它们以后会再见，但本页先只管这三层的职责边界。

4. 能力类型：tools、resources、prompts 不做同一件事

能力

如果你把所有能力都当成函数调用，MCP 会立刻变糊。tools 是“做动作”，resources 是“给内容”，prompts 是“给固定工作流入口”。这三个词的区别在于它们让模型看到的世界不同：动作要边界，内容要可读，提示模板要可复用。

tools

适合查一批数据、创建一条记录、执行一项受控动作。重点是输入 schema、边界和失败返回。

resources

适合读取文档、目录、schema 说明。重点是让模型看见内容，不一定触发动作。

prompts

适合纪要、审查、摘要、检查清单这类固定格式工作流。重点是复用提示组织方式。

能力类型速记
tools = 让模型做事
resources = 让模型看见
prompts = 让模型按固定方式思考或表达
误用会发生什么更合适的理解
把 resource 当 tool读内容被写成动作，边界变粗先想“给模型看什么”
把 prompt 当 tool工作流模板被误当执行接口先想“固定怎么问”
把 tool 当 resource动作被静态化，参数和权限丢失先想“要执行什么”

5. 传输层：会话怎么连起来

传输

传输层讲的是消息怎么到达，不讲能力本身是什么。你在旧页里会看到 command、args、url、cwd、env 这些字眼，它们描述的是“Host 怎样把 server 起起来或连上去”。这两层如果混了，排错时就会盯错地方。

[host] start server
[client] spawn mcp-server-file
[client] error: command not found
配置项它在管什么错了的症状
command本地 server 命令找不到进程或命令无法执行
args启动参数server 起了但行为不对
url远程连接地址握手失败或网络不可达
cwd / env工作目录和环境变量路径找不到、凭证缺失、配置不生效

这节只要记住一件事：传输层出错时，问题通常在“怎么连”，不在“能做什么”。先别急着改工具描述。

6. 会话流程：初始化、发现、调用、返回

流程

一次 MCP 会话最好拆成四步看：先初始化，再发现能力，再发起调用，最后把结果回写到 Host 的上下文。这样做的好处是，很多看起来相似的故障会被拆成不同层级。连上了但看不到工具，和工具能看到但参数不对，本来就是两件事。

步骤谁在读谁在写检查点失败症状
初始化Host 读配置，Client 读 serverClient 建立会话command、url、env、cwd 是否正确找不到进程或握手失败
能力发现Client 读 tools / resources / prompts 描述Client 缓存能力列表列表是否为空、schema 是否可读连上了却看不到能力
调用Host 读用户意图，Client 读参数Server 执行动作输入是否完整、是否越权参数错、权限拒绝、结果不稳
返回Host 读结果Host 把结果放回上下文结果是否真的进入对话工具成功了，但答案里没出现

先看连接是否活着，再看能力是否发现，再看参数是否正确，最后看结果有没有回到 Host。

这页的排错顺序也是这四步：先初始化，再发现，再调用，最后回写。你现在应该能把一次故障放到正确的步骤上，别把所有问题都叫“协议问题”。

7. 状态账本：把隐性状态摆到桌面上

状态

MCP 的会话、权限和缓存都有状态。只要有状态，就会出现“读了什么、写了什么、什么时候过期、错了会是什么症状”的问题。下面这张账本是本页最实用的排错表。

步骤读什么写什么什么时候失效错了会怎样
Host 决策用户目标、当前对话是否允许调用、是否要确认用户改口或切换任务后误触发高风险动作
Client 连接server 配置、初始化状态会话、能力缓存server 重启、版本变更后看见旧工具列表或握手失败
Server 执行参数、权限、资源状态结果、日志、资源变更写入后立即或事务结束后返回成功但实际未完成
Host 回写tool result对话上下文、下一轮推理依据如果没回写，就一直是空的模型像没看到结果
状态排查口诀
先看连接是否活着
再看能力是否发现
再看参数是否正确
最后看结果有没有回到 Host

这张表的作用是逼你把“看不见的状态”变成“能读、能写、能失效”的对象。以后看任何黑盒 server，都先问这四步里哪一步的状态坏了。

8. 安全边界：统一入口不等于放开权限

安全

课程里关于安全的提醒很直接：协议可以统一接口，但不会替你完成安全设计。能力暴露到什么范围、写入是否要确认、日志怎么记、参数怎么校验，这些都要自己收紧。真正稳的做法是先把边界和确认机制写清楚。

边界应该允许不应该允许风险信号
读取范围只读当前用户可见的数据跨租户、跨账户、跨项目读取顺手全拿来给模型看看
写入范围明确路径、明确对象、明确确认任意路径、静默覆盖、默认删除先给大权限，后面再收
参数校验字段、类型、必填项、范围都清楚任意字符串都能进执行层工具说明只写一句空话
审计记录谁在什么时候做了什么只有成功/失败，没有上下文出问题后无法追踪
安全检查清单
1. 只暴露必要能力
2. 输入先校验再执行
3. 高风险操作要确认
4. 结果和错误都要能审计
5. Server 不替代 Host 的用户体验

这一节只要记住一个判断：协议标准化的是接口，风险仍然要靠权限、确认、校验和审计来兜住。

9. 排错路径：按症状反推是哪一层坏了

排错

如果你只记一个调试办法，就记这个：先定位层，再找症状对应的读写点。传输层坏了，通常先看到的是起不来、连不上、环境变量缺失；能力发现坏了，常见症状是列表为空；调用层坏了，常见症状是缺参、越权、返回错误；回写层坏了，最迷惑，因为 server 明明成功了，模型却像没看见。

症状优先怀疑哪层先看什么别先急着改什么
command not found / network unreachable传输层command、url、cwd、env工具 schema
连上了但 tool list 为空能力发现server 是否注册能力模型提示词
调用时报缺参或类型错调用层input schema、必填字段返回摘要格式
server 成功但答案没变化回写层Host 是否把结果放回上下文重新写 server 逻辑

别把“工具能看到”当成“工具能用”。也别把“server 返回成功”当成“模型已经用上结果”。这两个判断常常把人带偏。

这部分的目标是让你把故障先放进正确的盒子里。能定位层，后面的修复才有意义。

10. 配置桥接：从 command 到 schema 怎么接起来

配置

实际读 server 时，先碰到的通常是启动配置。Host 看到的 command、args、url、cwd、env，决定了 server 能不能被拉起来；server 拉起来以后，才轮到它把 tools、resources、prompts 说清楚。排错时如果直接冲向工具定义，很容易把问题看歪。

一个最小配置样子
{
  "command": "mcp-server-files",
  "args": ["--root", "./docs"],
  "env": {
    "MCP_TOKEN": "***"
  }
}
字段Host 读什么Server 依赖什么坏掉的症状
command要启动哪个进程本地可执行文件直接找不到命令
args要带什么参数命令行解析结果server 起了但根目录错了
cwd在哪个目录启动相对路径解析看起来有配置，实际上没读到文件
env要传哪些环境变量凭证、开关、路径认证失败或功能被静默关闭
 
[host] spawn mcp-server-files
[client] error: command not found

这条链的关键点很简单：先让进程起来，再让 schema 可见，再让调用成功。只要其中一段断了，后面的 tool 描述再漂亮也没有用。

11. 共享案例：同一个能力给多个 Host 用时会发生什么

案例

把一个“查项目文档”的能力给桌面助手、IDE 助手和日报机器人一起用，最容易看出 MCP 的价值。三个 Host 的用户不同，场景不同，展示方式不同，但它们都需要同一份结构化结果。如果每个 Host 都自己接同一个文档系统，改权限、改字段、改错误提示时就要改三遍。

Host关注点Server 负责最容易出错的地方
桌面助手用户确认和摘要展示返回文档标题、路径、摘要结果太长，用户看不完
IDE 助手上下文插入和代码定位返回可引用片段和来源片段没有位置，没法继续操作
日报机器人批量汇总和格式稳定返回结构化条目自然语言太多，无法汇总
Host 输入 用户目标 + 当前上下文

输出：是否调用、调用哪个工具、是否需要确认。

Client 输入 工具 schema + 会话状态

输出：带参数的结构化调用请求。

Server 输出 标题、位置、摘要、可见性

输出：能被摘要、排序、引用的结果。

这个案例的坏味道也很明显：如果返回的是一长段自由文本，Host 就要再做一轮解析；如果返回的是结构化结果，三个 Host 都能直接用。你现在应该能说清，为什么统一 server 比三份连接器更稳。

12. 版本变化：schema 改了以后为什么旧客户端会坏

版本

会话能连上，不代表版本一定一致。server 一旦改了工具名、字段名或返回结构，旧客户端最先撞到的往往不是“完全不能用”，而是“能看到，但调用错”。这类问题最迷惑，因为表面上协议还通，真正坏的是 schema 和缓存。

旧 schema 和新 schema
{
  "before": {
    "name": "calendar.search",
    "input": { "startDate": "2026-06-21", "endDate": "2026-06-21" }
  },
  "after": {
    "name": "calendar.search_v2",
    "input": { "range": { "start": "2026-06-21", "end": "2026-06-21" } }
  }
}
变化点表面症状真正问题先查哪里
工具名变了调用找不到入口旧客户端还在喊旧名字工具列表和缓存
字段结构变了参数校验失败模型还按旧字段发参input schema
返回结构变了Host 摘要出错回写逻辑还按旧格式读结果结果解析和回写
权限策略变了以前能用，现在被拒绝新策略收紧了访问范围确认和审计日志

排查顺序很固定：先看 server 当前暴露什么，再看 Host 缓存了什么，再看模型看到什么，最后才看业务逻辑。这样做能把“协议坏了”的错觉拆开，避免你直接去重写 server。

13. 最小实战：从一段 server 说明读出调用路径

实战

下面这段只是一个最小说明。你读它时，顺序要先看配置，再看能力，再看返回值。这样做的原因很实际：如果进程都起不来，后面 tools 写得再漂亮也没有用；如果 schema 没写清楚，模型就会在参数上乱猜；如果返回值太自由，Host 就很难把结果继续用起来。

最小 server 说明
{
  "command": "mcp-server-notes",
  "args": ["--root", "./knowledge"],
  "tools": [
    {
      "name": "notes.search",
      "inputSchema": {
        "type": "object",
        "properties": {
          "query": { "type": "string" },
          "limit": { "type": "number", "default": 5 }
        },
        "required": ["query"]
      }
    }
  ]
}
对象读什么写什么如果错了会怎样
command / argsserver 怎么起启动进程和根目录进程起不来，或者读错目录
tool name模型该找哪个动作调用目标工具选错或压根找不到
inputSchema参数长什么样必填字段和默认值模型缺参，server 直接拒绝
result shapeHost 怎么继续用结果摘要、排序、引用的材料结果像一段散文，后续步骤接不住

如果这个 server 真正跑起来，调用链会是这样的：Host 先读到配置，Client 发起连接，Server 暴露 notes.search，模型根据 schema 传入 query 和 limit，Server 返回结构化结果，Host 再把结果写回上下文。你可以把这条链手画出来，看看哪一步负责“起进程”，哪一步负责“发现能力”，哪一步负责“让结果还能继续被用”。

失败模式最先露出的症状更可能的根因
目录没配对能启动但查不到文件cwd 或 args 指错根目录
schema 太松模型乱传参数inputSchema 没写 required
结果太散Host 摘要时丢字段返回值没有稳定结构

这条链是你以后读任何 server 的起点。先确认配置能把进程拉起来，再看工具 schema，最后才看结果格式。练习区里有一题会专门让你判断这个链条断在哪一步。

14. 结果回写：server 成功以后，Host 还要做什么

回写

很多人第一次排查 MCP，都会卡在同一个地方：server 明明已经返回了，为什么用户看到的答案里还是没有结果。答案通常不在 server，而在 Host。server 负责把能力执行完，Host 负责把执行结果放回对话或工作流，再交给模型继续推理或继续展示。

拿 notes.search 这个例子来说，server 最好返回一组能继续处理的字段；如果只给一段自然语言，Host 还要再拆字段。Host 看到这些字段后，可以先摘要，再排序，再引用，再决定要不要发起下一次调用。这样一来，工具结果不是句号，而是下一步动作的材料。

层读什么写什么如果漏掉
Server参数、权限、资源状态工具结果、错误、审计线索执行没完成，或结果不可信
Hosttool result、当前任务、用户目标对话上下文、摘要、下一步动作工具成功了，但模型没看见
Model回写后的上下文继续回答、继续调用、继续解释答案还是停在旧状态
返回形态Host 读起来的成本后续动作坏味道
结构化条目低，字段一眼能看清摘要、排序、引用、继续调用几乎没有，适合多次使用
自由文本摘要中，需要再拆字段可读，但容易丢掉位置和顺序适合展示，不适合继续加工
一整段长文本高，后续要先解析很难稳定回写一旦变长，就容易把上下文挤乱
回写模板
1. 先读 tool result，确认返回值不是空的
2. 再把结果摘成当前任务能继续用的结构
3. 最后把摘要或原始结果写回上下文
4. 如果结果缺字段，回到 server 端看 schema 或返回格式

最常见的失误是把“展示给人看得懂”当成“给模型继续用得上”。前者只要顺眼，后者要字段稳定、顺序稳定、来源稳定。

这一步的质量很容易被低估。结果回写做得好，模型下一轮回答会接着前一步继续；做不好，用户会感觉“工具已经跑了，但系统像没看到”。

15. 下一步怎么读真实 server

边界

读完这一页，你就能用结构化方式看 MCP server 了：先看它暴露的是 tool、resource 还是 prompt，再看它的启动方式和传输配置，再看它的调用流程和回写方式，最后看权限、确认和审计有没有收紧。真正开始动手时，顺序通常是先读配置，再读 schema，再读错误返回。

先看配置怎么连，再看能力怎么说，再看状态怎么变，再看结果有没有回到 Host。

这里先不追更细的协议版本、SDK 差异、底层 transport 细节和完整实现惯例。那些内容会在更具体的 server 实例里再出现。现在你已经有了一个稳定的读法：先问结构，再问状态，再问安全。
