# 1. 先把对象放稳：这一页在讲什么 {#overview}
<!-- tag: 概览 -->

上一页说的是为什么 MCP 会出现，这一页开始看它怎么工作。最容易学乱的地方有两个：一是把角色名按普通 Web 直觉理解，二是把 tools、resources、prompts 都当成“可以调用的接口”。先把这两层分开，后面看任何 server 时才不会只记住表面名词。

你可以先拿一个最小场景练手：Host 里有一个 AI 应用，Client 负责和 MCP server 说话，Server 提供一个“查项目文档”的能力。先看谁负责用户体验，谁负责协议消息，谁负责外部能力。

## box info
对象最小任务最先看什么错了会怎样


Host决定要不要调用外部能力用户意图和确认体验高风险动作被直接触发
Client建立会话、发现能力、发请求server 描述和工具列表连上了却看不到可用能力
Server执行查询或动作并返回结果输入参数和权限边界能力看起来存在，执行却不稳定

## box soft
这里先不追协议细节、传输实现、SDK 版本兼容和具体厂商封装。先把结构和消息流看懂，再去读真实 server 的配置和实现。

# 2. 架构图：先看消息怎么流 {#architecture}
<!-- tag: 架构图 -->

学习 MCP 时，最有用的是消息流。你可以把它理解成一条有边界的管道：用户目标进入 Host，Host 把意图交给内部的 Client，Client 去找 Server，Server 执行后返回结果，Host 再把结果放回对话或工作流里。这里真正要检查的是“在哪一步变成了结构化调用，在哪一步又回到了自然语言回答”。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

这张图的用途很朴素：把“连接成功”“能力可见”“参数正确”“结果回写”拆成四件事。很多故障出在你把这四件事混在一起。


---

# 3. 三类角色：Host、Client、Server 各管哪一段 {#roles}
<!-- tag: 角色 -->

这里按职责理解，不按部署位置理解。Host 是面向用户的 AI 应用，Client 是 Host 内部负责协议连接的那一层，Server 是把外部能力暴露出来的那一层。你读到任何一个 server，先问它属于哪一层，再看它部署在哪儿。

| 角色 | 读什么 | 写什么 | 典型症状 |
| --- | --- | --- | --- |
| Host | 用户意图、工具结果、确认信息 | 对话、流程编排、回写上下文 | 结果进了系统，但没回到回答里 |
| Client | server 描述、能力列表、会话状态 | 协议消息、调用请求 | 已经连上，却读不到工具 |
| Server | 参数、权限、资源状态 | 工具结果、资源内容、拒绝原因 | 能力说得清，执行却不稳 |

## box warn
先别把 Agent、编排器、插件框架和部署拓扑混进来。它们以后会再见，但本页先只管这三层的职责边界。

# 4. 能力类型：tools、resources、prompts 不做同一件事 {#capabilities}
<!-- tag: 能力 -->

如果你把所有能力都当成函数调用，MCP 会立刻变糊。tools 是“做动作”，resources 是“给内容”，prompts 是“给固定工作流入口”。这三个词的区别在于它们让模型看到的世界不同：动作要边界，内容要可读，提示模板要可复用。

tools
适合查一批数据、创建一条记录、执行一项受控动作。重点是输入 schema、边界和失败返回。


resources
适合读取文档、目录、schema 说明。重点是让模型看见内容，不一定触发动作。


prompts
适合纪要、审查、摘要、检查清单这类固定格式工作流。重点是复用提示组织方式。

能力类型速记复制

```text
tools = 让模型做事
resources = 让模型看见
prompts = 让模型按固定方式思考或表达
```

| 误用 | 会发生什么 | 更合适的理解 |
| --- | --- | --- |
| 把 resource 当 tool | 读内容被写成动作，边界变粗 | 先想“给模型看什么” |
| 把 prompt 当 tool | 工作流模板被误当执行接口 | 先想“固定怎么问” |
| 把 tool 当 resource | 动作被静态化，参数和权限丢失 | 先想“要执行什么” |

# 5. 传输层：会话怎么连起来 {#transports}
<!-- tag: 传输 -->

传输层讲的是消息怎么到达，不讲能力本身是什么。你在旧页里会看到 command、args、url、cwd、env 这些字眼，它们描述的是“Host 怎样把 server 起起来或连上去”。这两层如果混了，排错时就会盯错地方。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

| 配置项 | 它在管什么 | 错了的症状 |
| --- | --- | --- |
| command | 本地 server 命令 | 找不到进程或命令无法执行 |
| args | 启动参数 | server 起了但行为不对 |
| url | 远程连接地址 | 握手失败或网络不可达 |
| cwd / env | 工作目录和环境变量 | 路径找不到、凭证缺失、配置不生效 |

这节只要记住一件事：传输层出错时，问题通常在“怎么连”，不在“能做什么”。先别急着改工具描述。


---

# 6. 会话流程：初始化、发现、调用、返回 {#lifecycle}
<!-- tag: 流程 -->

一次 MCP 会话最好拆成四步看：先初始化，再发现能力，再发起调用，最后把结果回写到 Host 的上下文。这样做的好处是，很多看起来相似的故障会被拆成不同层级。连上了但看不到工具，和工具能看到但参数不对，本来就是两件事。

| 步骤 | 谁在读 | 谁在写 | 检查点 | 失败症状 |
| --- | --- | --- | --- | --- |
| 初始化 | Host 读配置，Client 读 server | Client 建立会话 | command、url、env、cwd 是否正确 | 找不到进程或握手失败 |
| 能力发现 | Client 读 tools / resources / prompts 描述 | Client 缓存能力列表 | 列表是否为空、schema 是否可读 | 连上了却看不到能力 |
| 调用 | Host 读用户意图，Client 读参数 | Server 执行动作 | 输入是否完整、是否越权 | 参数错、权限拒绝、结果不稳 |
| 返回 | Host 读结果 | Host 把结果放回上下文 | 结果是否真的进入对话 | 工具成功了，但答案里没出现 |

## box info
先看连接是否活着，再看能力是否发现，再看参数是否正确，最后看结果有没有回到 Host。

这页的排错顺序也是这四步：先初始化，再发现，再调用，最后回写。你现在应该能把一次故障放到正确的步骤上，别把所有问题都叫“协议问题”。

# 7. 状态账本：把隐性状态摆到桌面上 {#state-ledger}
<!-- tag: 状态 -->

MCP 的会话、权限和缓存都有状态。只要有状态，就会出现“读了什么、写了什么、什么时候过期、错了会是什么症状”的问题。下面这张账本是本页最实用的排错表。

| 步骤 | 读什么 | 写什么 | 什么时候失效 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| Host 决策 | 用户目标、当前对话 | 是否允许调用、是否要确认 | 用户改口或切换任务后 | 误触发高风险动作 |
| Client 连接 | server 配置、初始化状态 | 会话、能力缓存 | server 重启、版本变更后 | 看见旧工具列表或握手失败 |
| Server 执行 | 参数、权限、资源状态 | 结果、日志、资源变更 | 写入后立即或事务结束后 | 返回成功但实际未完成 |
| Host 回写 | tool result | 对话上下文、下一轮推理依据 | 如果没回写，就一直是空的 | 模型像没看到结果 |

状态排查口诀复制

```text
先看连接是否活着
再看能力是否发现
再看参数是否正确
最后看结果有没有回到 Host
```

这张表的作用是逼你把“看不见的状态”变成“能读、能写、能失效”的对象。以后看任何黑盒 server，都先问这四步里哪一步的状态坏了。

# 8. 安全边界：统一入口不等于放开权限 {#security}
<!-- tag: 安全 -->

课程里关于安全的提醒很直接：协议可以统一接口，但不会替你完成安全设计。能力暴露到什么范围、写入是否要确认、日志怎么记、参数怎么校验，这些都要自己收紧。真正稳的做法是先把边界和确认机制写清楚。

| 边界 | 应该允许 | 不应该允许 | 风险信号 |
| --- | --- | --- | --- |
| 读取范围 | 只读当前用户可见的数据 | 跨租户、跨账户、跨项目读取 | 顺手全拿来给模型看看 |
| 写入范围 | 明确路径、明确对象、明确确认 | 任意路径、静默覆盖、默认删除 | 先给大权限，后面再收 |
| 参数校验 | 字段、类型、必填项、范围都清楚 | 任意字符串都能进执行层 | 工具说明只写一句空话 |
| 审计记录 | 谁在什么时候做了什么 | 只有成功/失败，没有上下文 | 出问题后无法追踪 |

安全检查清单复制

```text
1. 只暴露必要能力
2. 输入先校验再执行
3. 高风险操作要确认
4. 结果和错误都要能审计
5. Server 不替代 Host 的用户体验
```

这一节只要记住一个判断：协议标准化的是接口，风险仍然要靠权限、确认、校验和审计来处理。

# 9. 排错路径：按症状反推是哪一层坏了 {#debug-path}
<!-- tag: 排错 -->

如果你只记一个调试办法，就记这个：先定位层，再找症状对应的读写点。传输层坏了，通常先看到的是起不来、连不上、环境变量缺失；能力发现坏了，常见症状是列表为空；调用层坏了，常见症状是缺参、越权、返回错误；回写层坏了，最迷惑，因为 server 明明成功了，模型却像没看见。

| 症状 | 优先怀疑哪层 | 先看什么 | 别先急着改什么 |
| --- | --- | --- | --- |
| command not found / network unreachable | 传输层 | command、url、cwd、env | 工具 schema |
| 连上了但 tool list 为空 | 能力发现 | server 是否注册能力 | 模型提示词 |
| 调用时报缺参或类型错 | 调用层 | input schema、必填字段 | 返回摘要格式 |
| server 成功但答案没变化 | 回写层 | Host 是否把结果放回上下文 | 重新写 server 逻辑 |

## box
别把“工具能看到”当成“工具能用”。也别把“server 返回成功”当成“模型已经用上结果”。这两个判断常常把人带偏。

这部分的目标是让你把故障先放进正确的盒子里。能定位层，后面的修复才有意义。

# 10. 配置桥接：从 command 到 schema 怎么接起来 {#config-bridge}
<!-- tag: 配置 -->

实际读 server 时，先碰到的通常是启动配置。Host 看到的 command、args、url、cwd、env，决定了 server 能不能被拉起来；server 拉起来以后，才轮到它把 tools、resources、prompts 说清楚。排错时如果直接冲向工具定义，很容易把问题看歪。

一个最小配置样子复制

```json
{
 "command": "mcp-server-files",
 "args": ["--root", "./docs"],
 "env": {
 "MCP_TOKEN": "***"
 }
}
```

| 字段 | Host 读什么 | Server 依赖什么 | 坏掉的症状 |
| --- | --- | --- | --- |
| command | 要启动哪个进程 | 本地可执行文件 | 直接找不到命令 |
| args | 要带什么参数 | 命令行解析结果 | server 起了但根目录错了 |
| cwd | 在哪个目录启动 | 相对路径解析 | 看起来有配置，实际上没读到文件 |
| env | 要传哪些环境变量 | 凭证、开关、路径 | 认证失败或功能被静默关闭 |

<!-- interactive: ['terminal-lab'] → keep as HTML -->

这条链的关键点很简单：先让进程起来，再让 schema 可见，再让调用成功。只要其中一段断了，后面的 tool 描述再漂亮也没有用。

# 11. 共享案例：同一个能力给多个 Host 用时会发生什么 {#case-study}
<!-- tag: 案例 -->

把一个“查项目文档”的能力给桌面助手、IDE 助手和日报机器人一起用，最容易看出 MCP 的价值。三个 Host 的用户不同，场景不同，展示方式不同，但它们都需要同一份结构化结果。如果每个 Host 都自己接同一个文档系统，改权限、改字段、改错误提示时就要改三遍。

| Host | 关注点 | Server 负责 | 最容易出错的地方 |
| --- | --- | --- | --- |
| 桌面助手 | 用户确认和摘要展示 | 返回文档标题、路径、摘要 | 结果太长，用户看不完 |
| IDE 助手 | 上下文插入和代码定位 | 返回可引用片段和来源 | 片段没有位置，没法继续操作 |
| 日报机器人 | 批量汇总和格式稳定 | 返回结构化条目 | 自然语言太多，无法汇总 |

<!-- interactive: ['shape-tracer'] → keep as HTML -->

这个案例的坏味道也很明显：如果返回的是一长段自由文本，Host 就要再做一轮解析；如果返回的是结构化结果，三个 Host 都能直接用。你现在应该能说清，为什么统一 server 比三份连接器更稳。

# 12. 版本变化：schema 改了以后为什么旧客户端会坏 {#versioning}
<!-- tag: 版本 -->

会话能连上，不代表版本一定一致。server 一旦改了工具名、字段名或返回结构，旧客户端最先撞到的往往“能看到，但调用错”。这类问题最迷惑，因为表面上协议还通，真正坏的是 schema 和缓存。

旧 schema 和新 schema复制

```json
{
 "before": {
 "name": "calendar.search",
 "input": { "startDate": "2026-06-21", "endDate": "2026-06-21" }
 },
 "after": {
 "name": "calendar.search_v2",
 "input": { "range": { "start": "2026-06-21", "end": "2026-06-21" } }
 }
}
```

| 变化点 | 表面症状 | 真正问题 | 先查哪里 |
| --- | --- | --- | --- |
| 工具名变了 | 调用找不到入口 | 旧客户端还在喊旧名字 | 工具列表和缓存 |
| 字段结构变了 | 参数校验失败 | 模型还按旧字段发参 | input schema |
| 返回结构变了 | Host 摘要出错 | 回写逻辑还按旧格式读结果 | 结果解析和回写 |
| 权限策略变了 | 以前能用，现在被拒绝 | 新策略收紧了访问范围 | 确认和审计日志 |

排查顺序很固定：先看 server 当前暴露什么，再看 Host 缓存了什么，再看模型看到什么，最后才看业务逻辑。这样做能把“协议坏了”的错觉拆开，避免你直接去重写 server。

# 13. 最小实战：从一段 server 说明读出调用路径 {#walkthrough}
<!-- tag: 实战 -->

下面这段只是一个最小说明。你读它时，顺序要先看配置，再看能力，再看返回值。这样做的原因很实际：如果进程都起不来，后面 tools 写得再漂亮也没有用；如果 schema 没写清楚，模型就会在参数上乱猜；如果返回值太自由，Host 就很难把结果继续用起来。

最小 server 说明复制

```json
{
 "command": "mcp-server-notes",
 "args": ["--root", "./knowledge"],
 "tools": [
 {
 "name": "notes.search",
 "inputSchema": {
 "type": "object",
 "properties": {
 "query": { "type": "string" },
 "limit": { "type": "number", "default": 5 }
 },
 "required": ["query"]
 }
 }
 ]
}
```

| 对象 | 读什么 | 写什么 | 如果错了会怎样 |
| --- | --- | --- | --- |
| command / args | server 怎么起 | 启动进程和根目录 | 进程起不来，或者读错目录 |
| tool name | 模型该找哪个动作 | 调用目标 | 工具选错或压根找不到 |
| inputSchema | 参数长什么样 | 必填字段和默认值 | 模型缺参，server 直接拒绝 |
| result shape | Host 怎么继续用结果 | 摘要、排序、引用的材料 | 结果像一段散文，后续步骤接不住 |

如果这个 server 真正跑起来，调用链会是这样的：Host 先读到配置，Client 发起连接，Server 暴露 `notes.search`，模型根据 schema 传入 `query` 和 `limit`，Server 返回结构化结果，Host 再把结果写回上下文。你可以把这条链手画出来，看看哪一步负责“起进程”，哪一步负责“发现能力”，哪一步负责“让结果还能继续被用”。

## box
失败模式最先露出的症状更可能的根因


目录没配对能启动但查不到文件cwd 或 args 指错根目录
schema 太松模型乱传参数inputSchema 没写 required
结果太散Host 摘要时丢字段返回值没有稳定结构

这条链是你以后读任何 server 的起点。先确认配置能把进程拉起来，再看工具 schema，最后才看结果格式。练习区里有一题会专门让你判断这个链条断在哪一步。

# 14. 结果回写：server 成功以后，Host 还要做什么 {#result-writeback}
<!-- tag: 回写 -->

很多人第一次排查 MCP，都会卡在同一个地方：server 明明已经返回了，为什么用户看到的答案里还是没有结果。答案通常不在 server，而在 Host。server 负责把能力执行完，Host 负责把执行结果放回对话或工作流，再交给模型继续推理或继续展示。

拿 `notes.search` 这个例子来说，server 最好返回一组能继续处理的字段；如果只给一段自然语言，Host 还要再拆字段。Host 看到这些字段后，可以先摘要，再排序，再引用，再决定要不要发起下一次调用。这样一来，工具结果不是句号，而是下一步动作的材料。

层读什么写什么如果漏掉


Server参数、权限、资源状态工具结果、错误、审计线索执行没完成，或结果不可信
Hosttool result、当前任务、用户目标对话上下文、摘要、下一步动作工具成功了，但模型没看见
Model回写后的上下文继续回答、继续调用、继续解释答案还是停在旧状态

| 返回形态 | Host 读起来的成本 | 后续动作 | 坏味道 |
| --- | --- | --- | --- |
| 结构化条目 | 低，字段一眼能看清 | 摘要、排序、引用、继续调用 | 几乎没有，适合多次使用 |
| 自由文本摘要 | 中，需要再拆字段 | 可读，但容易丢掉位置和顺序 | 适合展示，不适合继续加工 |
| 一整段长文本 | 高，后续要先解析 | 很难稳定回写 | 一旦变长，就容易把上下文挤乱 |

回写模板复制

```text
1. 先读 tool result，确认返回值不是空的
2. 再把结果摘成当前任务能继续用的结构
3. 最后把摘要或原始结果写回上下文
4. 如果结果缺字段，回到 server 端看 schema 或返回格式
```

## box warn
最常见的失误是把“展示给人看得懂”当成“给模型继续用得上”。前者只要顺眼，后者要字段稳定、顺序稳定、来源稳定。

这一步的质量很容易被低估。结果回写做得好，模型下一轮回答会接着前一步继续；做不好，用户会感觉“工具已经跑了，但系统像没看到”。

# 15. 下一步怎么读真实 server {#bridge}
<!-- tag: 边界 -->

读完这一页，你就能用结构化方式看 MCP server 了：先看它暴露的是 tool、resource 还是 prompt，再看它的启动方式和传输配置，再看它的调用流程和回写方式，最后看权限、确认和审计有没有收紧。真正开始动手时，顺序通常是先读配置，再读 schema，再读错误返回。

## box soft
先看配置怎么连，再看能力怎么说，再看状态怎么变，再看结果有没有回到 Host。

这里先不追更细的协议版本、SDK 差异、底层 transport 细节和完整实现惯例。那些内容会在更具体的 server 实例里再出现。现在你已经有了一个稳定的读法：先问结构，再问状态，再问安全。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
<!-- end-practice-intro -->

**角色**Host、Client、Server 各管一层。
**能力**tools、resources、prompts 不做同一件事。
**排错**先定位层，再看读写和症状。
**安全**统一接口不等于放开权限。

<!-- practice-group: 一、概念定位 -->
<!-- quick-review: 检查点 先区分连接层、能力层和应用层。 -->

### 1 choice

**Q:** 1. MCP 主要解决哪类问题？

- [ ] AI 应用连接外部数据和工具
- [ ] 训练更大的模型
- [ ] 压缩图片体积

Answer: AI 应用连接外部数据和工具
Explain: MCP 解决的是 AI 应用与外部系统之间的标准连接。

### 2 判断

**Q:** 2. MCP 本身是一种大语言模型。

- [ ] 对
- [ ] 错

Answer: 错
Explain: MCP 是协议。它让模型更容易接入外部能力，但不替代模型本身。

### 3 简答

**Q:** 3. 用一句话定义 MCP。

Explain: 要点至少包含“标准协议”“外部能力”“AI 应用”三部分，例如：MCP 是让 AI 应用以统一方式发现和调用外部能力的协议。评分时看是否说清连接层边界，以及是否把协议和模型区分开。

<!-- end-practice-group -->

<!-- practice-group: 二、角色与能力 -->
<!-- quick-review: 检查点 动作、内容、模板，不要混成一类。 -->

### 4 填空

**Q:** 4. 填空：AI 应用通常是 ____，它内部创建 ____，外部能力由 ____ 暴露。

Explain: 答案应对应 Host / Client / Server。评分重点是角色边界是否正确，以及三者的读写职责是否清楚。

### 5 比较

**Q:** 5. MCP server 和普通 API 的关系是什么？

- [ ] MCP server 可以包装普通 API，但更强调可发现、可复用和可审计
- [ ] MCP 就是普通 API 的另一个名字
- [ ] 普通 API 不能被任何 AI 应用使用

Answer: MCP server 可以包装普通 API，但更强调可发现、可复用和可审计
Explain: MCP server 可以把 API 变成标准化能力入口，但协议关注的是接入方式和治理。

### 6 解释

**Q:** 6. MCP 为何不等于完整 Agent 框架？

Explain: MCP 只解决外部能力接入与发现，不负责规划、记忆、反思、长任务管理或完整执行编排。回答时要把“协议层”和“智能体层”分开。

<!-- end-practice-group -->

<!-- practice-group: 三、流程与诊断 -->
<!-- quick-review: 检查点 先看链路，再看模型。 -->

### 7 排序

**Q:** 7. 排序：初始化 / 能力发现 / 调用 / 返回。

- [ ] 初始化->能力发现->调用->返回
- [ ] 调用->初始化->返回->能力发现
- [ ] 返回->调用->能力发现->初始化

Answer: 初始化->能力发现->调用->返回
Explain: 正确顺序是先初始化，再发现能力，再发起调用，最后返回结果。

### 8 代码阅读

**Q:** 8. 一个工具 schema 为什么要写 required 字段？

Explain: required 字段让调用方知道哪些参数必须提供，减少模型乱猜和 server 执行时的缺参失败。好答案会把“参数完整性”和“模型选参稳定性”都写出来。

### 9 诊断

**Q:** 9. Server 成功返回内容，但模型回答像没看到结果。问题可能在哪？

Explain: 常见问题在 Host：结果没有被写回对话上下文，或者被错误过滤、截断、丢进了别的状态里。回答要指出“执行成功”和“结果进入上下文”是两件事。

<!-- end-practice-group -->

<!-- practice-group: 四、状态与安全 -->
<!-- quick-review: 检查点 先问状态，再问权限。 -->

### 10 诊断

**Q:** 10. 工具描述只写“do task”，可能造成什么后果？

Explain: 后果通常是模型选错工具、传错参数、无法判断边界。修法是把用途、输入、输出、限制和失败情况写清楚。

### 11 选择

**Q:** 11. 下面哪种读取范围更安全？

- [ ] 只读当前用户可见的数据
- [ ] 跨租户、跨账户、跨项目读取
- [ ] 顺手把所有内容都返回给模型

Answer: 只读当前用户可见的数据
Explain: 读取范围要先收紧到当前用户可见内容，再考虑是否需要更细的权限分层。

### 12 checklist

**Q:** 12. 如果日志出现 command not found 或 network unreachable，你先查哪三项？

Explain: 优先查 command / url / env / cwd 这些传输层配置，再看 server 是否真的启动。不要先改 tool schema，也不要先怀疑模型提示词。

<!-- end-practice-group -->

<!-- practice-group: 五、最小实战 -->
<!-- quick-review: 检查点 先看命令，再看必填字段，再看结果结构。 -->

### 13 选择

**Q:** 13. 最小 server 说明里，如果能启动但查不到文件，最先怀疑什么？

- [ ] cwd 或 args 指错根目录
- [ ] 模型记性不好
- [ ] 结果格式太漂亮

Answer: cwd 或 args 指错根目录
Explain: 如果能起进程但查不到内容，先怀疑根目录和工作目录。

### 14 选择

**Q:** 14. 如果模型总是漏掉关键参数，最先怀疑什么？

- [ ] inputSchema 没写 required
- [ ] Host 的标题太短
- [ ] Server 结果太稳定

Answer: inputSchema 没写 required
Explain: 如果模型乱传参，先看 inputSchema 是否把必填字段写清楚。

### 15 解释

**Q:** 15. 结构化结果为什么更适合多 Host 复用？

Explain: 结构化结果更容易被 Host 摘要、排序、引用和回写，也更容易被多个 Host 共用。好答案会提到“后续还能继续用”这件事。

### 16 诊断

**Q:** 16. 如果 server 已经返回了，但用户还是看不到结果，你先查哪一层？

Explain: 先查 Host 的回写层，看 tool result 有没有真正进入上下文。好答案会提到：server 执行完成和结果回到对话不是一回事。

### 17 比较

**Q:** 17. 下面三种返回形态里，哪一种最适合给多个 Host 继续使用？为什么？

Explain: 最好选结构化条目。因为它最容易被摘要、排序、引用和再次调用。好答案会比较“结构化条目 / 自由文本摘要 / 一整段长文本”三者的后续可用性。
