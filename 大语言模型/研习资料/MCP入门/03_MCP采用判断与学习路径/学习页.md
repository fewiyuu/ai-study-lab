# 1. 先判断值不值得 MCP {#decision}
<!-- tag: decision -->

MCP 适合处理长期复用、边界清楚、需要被 AI 应用发现和调用的外部能力。它会增加一层 server、schema、配置、日志和版本维护，所以不能只问“能不能做”，要问“做成 MCP 后，复用和治理收益是否超过维护成本”。

贯穿本页的案例是共享工单查询。客服助手、研发 IDE 和值班机器人都想查同一套工单；数据有内部权限，查询结果要能审计，后面还可能加“创建工单”或“更新状态”。这个场景已经超过一次性脚本，更像一个值得评估的 MCP server。

判断条件支持采用 MCP 的信号先不用 MCP 的信号下一步检查


复用多个 Host 或多个团队需要同一能力只做一次本地脚本或临时导出列出会调用它的客户端和任务
权限会读内部资料、写业务系统、触发财务或通知动作公开数据、低风险、无写入写清最小权限和用户确认点
审计需要知道谁调用、何时调用、传了什么参数失败后不需要追踪责任链决定日志字段和留存范围
发现模型需要从工具描述和 schema 里稳定选择能力固定代码路径直接调用 API检查 tool 描述、输入 schema 和错误返回
维护外部系统变化时希望 server 一处更新新增 server 会比直接调用更难维护估算部署、监控、版本兼容成本

# 2. 学习路径按角色切 {#roles}
<!-- tag: roles -->

同一门 MCP 课，对不同角色的“够用”标准不一样。使用者要先会配置和授权；Server 开发者要会把外部系统包装成能力；Client/Host 开发者要会把工具选择、用户确认和结果回写接进应用；架构决策者要能决定哪些系统值得包装，哪些风险必须先管住。

角色先学什么能做什么就够进入下一步常见误区


使用者配置现成 server、授权、看工具列表能说明某个 server 会读写什么，凭证在哪里，失败时看哪类日志只会安装，不看权限和调用风险
Server 开发者tools、resources、prompts、输入 schema、错误返回能做一个只读 tool，并把读取范围、错误和日志写清楚一开始就暴露写入、删除、付款这类高风险动作
Client/Host 开发者工具发现、上下文注入、用户确认、结果回写能解释工具结果怎样回到对话，以及危险动作如何确认把工具列表塞给模型后就不管展示和失败处理
架构决策者采用标准、权限治理、审计、版本和部署模型能列出候选系统、采用优先级、风险清单和生产化门槛把所有 API 都包装成 MCP，增加不必要的层

如果你现在只是想“会用”，不要被 SDK 和协议细节拖走。先跑通一个可靠的现成 server，能看懂它暴露了哪些能力、需要哪些权限、失败时在哪一层出错。等你能解释这条链路，再决定要不要写自己的 server。

# 3. Quickstart 不是终点，是第一条可观察链路 {#quickstart}
<!-- tag: trace -->

初学 MCP 不建议从完整规格读起。先跑一个 quickstart，让 Host 能启动或连接 server，并看到一个最小工具；然后对照日志解释每一步发生了什么。你要观察的连接、能力发现、调用和返回是否各自成立。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

quickstart 复盘模板复制

```text
1. server 怎样启动：command / args / cwd / env
2. 初始化是否成功：版本、传输、握手响应
3. 暴露了哪些能力：tools / resources / prompts
4. 第一次调用传了什么参数：是否符合 schema
5. 返回结果怎样进入 Host：展示、上下文回写、错误提示
```

# 4. Server 开发从只读能力开始 {#server-path}
<!-- tag: server -->

写 MCP server 时，最稳的起点是只读能力。共享工单查询可以先只允许读取当前用户有权查看的工单，返回标题、状态、负责人、更新时间和链接。等只读范围、日志和错误返回稳定后，再考虑创建工单、更新状态这类写入动作。

步骤读取什么写入或改变什么错了会出现什么症状下一步检查


定义 tool业务动作、输入字段、返回字段工具名称、描述、schema模型选错工具或传错参数让描述包含用途、限制和必要参数
收紧范围当前用户、项目、租户、权限查询过滤条件跨项目或跨租户返回数据先只读当前用户可见范围
调用外部系统API、数据库或服务凭证请求日志和错误映射失败信息含糊，Host 只能看到通用错误区分鉴权失败、参数错误、系统不可达
返回结果外部系统响应结构化 tool result模型拿到一长段文本，无法继续引用字段保留 ID、状态、时间、链接等可继续使用的字段
加写入动作用户确认、变更字段、幂等标识业务系统状态误写、重复写、无法回滚先加确认，再加审计和失败补偿

共享工单查询的最小 tool 草图复制

```json
{
 "name": "search_tickets",
 "description": "Search tickets visible to the current user by keyword and status.",
 "inputSchema": {
 "type": "object",
 "required": ["keyword"],
 "properties": {
 "keyword": { "type": "string", "description": "Ticket keyword or ID" },
 "status": { "type": "string", "enum": ["open", "closed", "any"] }
 }
 },
 "returns": ["id", "title", "status", "owner", "updated_at", "url"]
}
```

这段草图不是某个 SDK 的完整代码，它只展示 server 开发者要先写清的契约：工具做什么、哪些参数必填、返回哪些结构化字段、权限由当前用户范围决定。

# 5. Client/Host 负责把工具变成可靠体验 {#client-host}
<!-- tag: host -->

如果你做 Host 或 MCP client，拿到工具列表只是开始。你还要决定工具信息怎样进入模型上下文，模型选择工具后怎样展示给用户，危险动作是否需要确认，返回结果怎样放回对话，失败时怎样让用户知道该重试、授权还是换查询条件。

Host/Client 环节读取写入或改变常见失败修法


工具注册Server 暴露的工具名、描述、schema模型可见的工具上下文工具太多或描述含糊，模型乱选筛选工具，保留清楚描述和限制
用户确认工具风险、参数、当前会话意图确认弹窗或拦截结果写入动作没有确认对写入、删除、外发和花钱动作设置确认
调用调度模型选择和参数发给 server 的调用请求参数缺失或越权调用前校验 schema 和权限
结果回写tool result 和错误对话上下文、UI 卡片、引用server 成功但模型没看到结果检查回写位置、截断策略和错误提示

# 6. Demo 到生产，缺口通常在治理层 {#production}
<!-- tag: production -->

Quickstart 跑通只证明链路能走一遍。生产环境要补的是权限、审计、监控、版本、错误处理和变更流程。尤其是写入类工具，不能只靠模型“应该不会乱点”。确认、幂等、回滚和日志要先设计好。

生产化项目要回答的问题共享工单案例里的落点


最小权限Server 能读写的范围是否小于用户真实权限？只返回当前用户可见工单，不跨租户、不跨项目
用户确认哪些调用必须确认？确认内容展示什么？更新状态、创建工单、外发消息前展示参数和影响
审计日志谁在什么时候用什么参数调用了什么工具？记录用户、tool、参数摘要、结果状态和工单 ID
错误映射失败能否区分权限、参数、网络、业务规则？把 403、缺字段、工单不存在、服务不可达分开返回
版本兼容schema 改动会不会打断旧 Host？新增字段先做可选，必填字段变更要有版本说明
监控告警怎样知道 server 卡住、失败率升高或调用异常？看启动失败、调用错误率、平均延迟、异常写入次数

## box
### 写入能力的红线
创建、删除、付款、发消息、更新业务状态这类能力，至少要有用户确认、最小权限、审计日志和错误回滚思路。缺其中任一项，就先停在原型或只读能力。

# 7. 成本收益：把结论写成三栏 {#tradeoff}
<!-- tag: tradeoff -->

采用判断最终要落成一段工程说明。推荐用三栏写：收益是什么，成本是什么，风险怎么管。这样比“建议采用 MCP”更有用，因为别人能看到你为什么这么判断，也能看到下一步要验证什么。

结论收益成本风险控制下一步


采用多客户端复用，同一套权限和审计要维护 server、schema、部署和监控只读先行，写入加确认和日志做一个只读原型，接入一个 Host 验证
暂不采用收益有限，普通 API 更直接MCP 层会增加配置和排错成本保留直接脚本，记录将来复用信号等出现多客户端复用再评估
先原型可能复用，但范围还不稳定先承担少量 server 开发成本只暴露只读 tool，不接写入动作用 1-2 周观察调用频率和故障类型

采用判断写法复制

```text
建议：先做只读 MCP server 原型。
收益：客服助手、研发 IDE、值班机器人都能复用同一套工单查询。
成本：需要维护 server、schema、权限配置、日志和部署。
风险控制：只读先行；写入动作暂缓；查询范围限制在当前用户可见工单。
验证点：两周内是否有多客户端复用，失败主要发生在传输、schema 还是权限层。
```

# 8. 场景判断：从信号到建议 {#scenario-lab}
<!-- tag: lab -->

下面三个场景故意很接近。读的时候先别急着背答案，按复用、权限、审计、发现和维护成本逐项扫一遍。一个好的结论可以是采用，也可以是暂不采用，或者先做只读原型。

场景复用权限/审计维护成本建议


企业知识库要给多个 AI 助手查询高高，需要按用户权限过滤可接受评估 MCP，只读原型先行
个人临时调用一次天气 API低低MCP 层偏重直接 API 或脚本
AI 要创建和更新工单中到高高，涉及写入和责任追踪高先只读 MCP，再设计确认、审计和回滚

# 9. 从判断走到一个可验收原型 {#prototype-bridge}
<!-- tag: bridge -->

采用判断写完后，不要直接把完整业务系统交给 MCP。更稳的做法是把结论拆成一个只读原型：能力范围小、输入字段少、返回结构稳定、失败能分类。共享工单查询可以先只做 `search_tickets`，不做创建、更新、删除。这样能验证 MCP 是否真的带来复用和治理收益，同时把写入风险留到下一轮。

这个原型的验收方式也要提前写清。你看三个 Host 是否能用同一套工具描述，权限过滤是否按当前用户生效，失败时能否区分缺参数、无权限、工单不存在和外部系统不可达。只有这些信号稳定，才值得继续评估写入能力。

原型任务先做什么验收信号暂时不做什么


能力范围只暴露 `search_tickets`，查询当前用户可见工单客服助手、研发 IDE、值班机器人能复用同一能力不做创建、删除、批量更新
输入契约只保留 `keyword`、`status`、`limit` 三类参数模型传参稳定，缺参时能返回明确错误不暴露任意 SQL 或复杂过滤表达式
输出契约返回 `id`、`title`、`status`、`owner`、`updated_at`、`url`Host 能展示卡片、引用链接，并把结构写回上下文不返回整段内部备注或无关字段
权限与审计按当前用户过滤，记录 tool、参数摘要、结果状态能回答“谁查了什么，是否越权失败”不把凭证放进模型上下文
失败分类区分缺参数、无权限、无结果、外部系统不可达用户知道该补关键词、申请权限、改条件还是稍后重试不只返回 `failed` 或空数组

原型验收备忘录复制

```text
Prototype: read-only search_tickets MCP server
Scope:
- read current-user-visible tickets only
- expose keyword/status/limit as input
- return id/title/status/owner/updated_at/url

Expected first failures:
- missing keyword -> schema validation message
- unauthorized project -> permission error, not empty success
- ticket system down -> external system unavailable
- Host does not show result -> inspect result writeback, not server query

Acceptance:
- at least two Host surfaces can reuse the same tool description
- audit log can reconstruct user, tool, parameter summary, result status
- no write action is exposed before confirmation and rollback are designed
```

这条桥接的重点是把学习从“我懂 MCP 概念”推进到“我知道第一个可验收产物是什么”。如果原型只被一个地方调用，失败也不需要追踪，那就回到普通 API 或脚本；如果多个 Host 都复用了它，并且权限、日志、错误分类开始发挥作用，MCP 的收益就有了证据。后面的练习会让你把同一套判断写成自己的三步学习计划和采用说明。

# 10. 原型失败时，先回退判断 {#fallback}
<!-- tag: fallback -->

原型失败不一定说明 MCP 学错了，可能只是这个场景不适合。判断时要把失败分成两类：一类是实现问题，比如 schema 不清楚、结果没回写、权限过滤写错；另一类是采用假设不成立，比如只有一个客户端使用、调用频率很低、审计没有真实需求。前者继续修原型，后者应该回退到普通 API 或脚本。

观察到的现象更可能的问题先做的检查决策变化


Host 看不到工具工具注册或筛选问题查工具发现、nav/context 注入和工具描述继续修原型，不急着否定 MCP
工具能调用，但结果没进回答结果回写问题查 Host 的 result mapper、截断和上下文写入继续修 Host/Client 路径
只有一个脚本会调用，之后也不会复用采用假设不成立查真实调用方和未来复用计划回退到普通 API 或脚本
写入工具需要大量确认和回滚设计风险高于当前收益先拆成只读查询，写入动作后置先只读原型，不上完整写入
每次外部系统变更都要改多个 Host复用和维护收益开始出现比较集中 server 维护和分散连接器维护成本继续评估 MCP 化

这个回退表能防止两个极端：一种是原型遇到问题就说 MCP 不行，另一种是场景明显低复用还继续把它包装成 server。工程取舍的好答案能说明哪条证据改变了你的判断，并把下一次验证写成可观察的日志、调用方、权限检查、复用证据和下周复测记录。

## box info
### 看到日志时先问三件事
如果日志是 `tool not found`，先查 Host 有没有加载和筛选工具；如果日志是 `permission denied`，先查当前用户范围和 server 权限过滤；如果一周只被一个脚本调用一次，问题就不在日志，而在采用假设本身。前两种继续修原型，第三种回退到普通 API 更合理。练习区的场景题也按这个顺序判断。

# 11. 常见误区 {#mistakes}
<!-- tag: mistakes -->

误区错在哪里修正方式


所有 API 都要 MCP 化把标准当潮流，忽略维护成本先看复用、权限、审计和跨客户端需求
quickstart 跑通就能上线只验证了链路，没有验证治理补最小权限、确认、日志、监控和版本兼容
Server 开发从写操作开始风险高，错误后果大先做只读能力，再逐步加确认和写入
只学配置，不学消息流出错时不知道查哪一层能解释启动、初始化、发现、调用、返回和回写
架构判断只看“先进”没有说明收益、成本和风险用三栏模板写出采用理由和验证点

学到这里，你应该能给一个需求写出“采用 / 暂不采用 / 先做原型”的判断。下一步如果要继续学，可以挑一个只读外部系统，从配置、schema、权限和结果回写四个点开始做最小 server。

### 1 choice

**Q:** 1. 哪个场景更适合评估 MCP？

- [ ] 多个 AI 客户端需要复用同一个知识库查询能力
- [ ] 一次性导出一个公开 CSV
- [ ] 本地临时算一道数学题

Answer: 多个 AI 客户端需要复用同一个知识库查询能力
Explain: 多客户端复用、权限过滤和审计需求明显时，MCP 更值得评估。

### 2 true/false

**Q:** 2. Quickstart 跑通后，就可以直接把 server 放到生产环境。

- [ ] 对
- [ ] 错

Answer: 错
Explain: Quickstart 只验证链路，不代表权限、审计、监控和版本兼容已经准备好。

### 3 checklist

**Q:** 3. 判断是否采用 MCP 时，你会看哪 5 个条件？

- [ ] 复用
- [ ] 权限
- [ ] 审计
- [ ] 发现
- [ ] 维护成本
- [ ] 流行程度

Explain: 前 5 项是本页的采用判断条件。“流行程度”不是工程理由；如果要采用，也要落回复用、权限、审计、发现或维护成本。

### 4 scenario

**Q:** 4. 举一个不急着用 MCP 的场景，并说明原因。

Explain: 好答案要包含一个具体场景，以及至少两个理由：低复用、低权限风险、无审计需求、调用点固定、维护 MCP server 成本偏高。只写“不需要”不够。

<!-- end-practice-group -->

<!-- practice-group: 二、学习路径 -->
<!-- quick-review: 检查点 使用者先会配置和看风险，开发者再写 server/client。 -->

### 5 choice

**Q:** 5. 如果你只是想会用 MCP server，前三步更接近哪一项？

- [ ] 配置现成 server、授权、看工具列表
- [ ] 一开始写完整远程部署平台
- [ ] 先背完整协议规格

Answer: 配置现成 server、授权、看工具列表
Explain: 使用者的第一目标是会配置、会授权、会判断 server 会读写什么。

### 6 choice

**Q:** 6. Server 开发最稳的起点是什么？

- [ ] 只读 tool
- [ ] 删除文件 tool
- [ ] 付款 tool

Answer: 只读 tool
Explain: 只读能力风险低，适合先练命名、schema、错误返回、权限和日志。

### 7 ordering

**Q:** 7. 排序：工具发现 / 用户确认 / 调用调度 / 结果回写。

Explain: 常见顺序是：工具发现 -> 用户确认或风险拦截 -> 调用调度 -> 结果回写。答案应指出结果回写经常被忽略：server 成功返回，不等于模型已经看到结果。

### 8 synthesis

**Q:** 8. 架构决策者评估 MCP 时，应该先列出哪些东西？

Explain: 至少应包含：候选外部系统、会复用的 Host/团队、读写权限边界、审计要求、部署和维护成本、是否先做只读原型。不要只写“列需求”。

<!-- end-practice-group -->

<!-- practice-group: 三、生产化检查 -->
<!-- quick-review: 检查点 权限、确认、审计、错误、版本、监控。 -->

### 9 choice

**Q:** 9. 工单更新 tool 上线前，哪组设计最不能省？

- [ ] 用户确认、最小权限、审计日志
- [ ] 更长的工具名字、更多颜色
- [ ] 把所有字段都返回给模型

Answer: 用户确认、最小权限、审计日志
Explain: 写入、删除、付款、外发消息这类动作至少要有确认、权限和日志。

### 10 choice

**Q:** 10. 共享工单查询的读取范围，哪个更适合作为第一版？

- [ ] 只读当前用户可见工单
- [ ] 跨租户读取所有工单
- [ ] 默认返回公司全部内部数据

Answer: 只读当前用户可见工单
Explain: 最小权限要先收紧到当前用户可见范围，不能因为模型要回答就跨权限读数据。

### 11 diagnosis

**Q:** 11. 为什么错误返回不能只写“调用失败”？

Explain: 好答案会说明：Host、用户和开发者需要知道失败来自权限、参数、外部系统不可达、业务规则还是版本失配。只写“调用失败”会让模型和用户都不知道下一步该授权、改参数还是重试。

### 12 checklist

**Q:** 12. 一个 MCP server 从 demo 到生产，至少补哪 6 件事？

Explain: 参考项：最小权限、用户确认、审计日志、错误映射、监控告警、版本兼容、部署方式、凭证管理、幂等和回滚。写入类工具应特别提到确认和审计。

<!-- end-practice-group -->

<!-- practice-group: 四、场景表达 -->
<!-- quick-review: 检查点 把收益、成本和风险放在同一段里。 -->

### 13 choice

**Q:** 13. 企业内部知识库要接入多个 AI 助手，较稳的建议是什么？

- [ ] 评估 MCP，只读原型先行
- [ ] 直接让模型读全库
- [ ] 永远不用 MCP

Answer: 评估 MCP，只读原型先行
Explain: 多个 AI 助手复用内部知识库，通常有权限过滤和审计需求。

### 14 choice

**Q:** 14. 个人只想临时调用一次天气 API，较稳的建议是什么？

- [ ] 暂不采用 MCP
- [ ] 立刻做完整 MCP server
- [ ] 先设计企业级审计平台

Answer: 暂不采用 MCP
Explain: 单次、低风险、低复用任务，普通 API 或脚本更清楚。

### 15 diagnosis

**Q:** 15. AI 要能创建和更新工单。采用 MCP 前必须补哪些安全设计？

Explain: 至少应包含：用户确认、最小权限、审计日志、写入参数校验、幂等或重复提交防护、失败回滚或补偿、错误分类。好答案会先建议只读原型，再逐步开放写入。

### 16 planning

**Q:** 16. 请给自己设计一个 3 步 MCP 学习路径。

Explain: 路径应按角色切。比如使用者：跑通现成 server -> 看工具列表和权限 -> 复盘失败日志。Server 开发者：只读 tool -> schema 和错误返回 -> 权限日志。架构角色：候选系统 -> 采用矩阵 -> 只读原型验证。

### 17 interview

**Q:** 17. 面试官问“什么时候不该用 MCP？”请用 3 句话回答。

Explain: 好答案会说：低复用、低风险、调用路径固定的一次性任务不急着 MCP；普通 API 或脚本更省维护成本；等出现多客户端复用、权限治理或审计需求，再评估 MCP。

### 18 synthesis

**Q:** 18. 用 6-9 句话解释：如何判断一个外部能力是否值得包装成 MCP server，以及下一步该怎么学。

Explain: 评分看 4 点：先给采用判断条件；能区分采用、暂不采用、先原型；能按角色说明学习路径；能指出从 demo 到生产还要补权限、确认、审计、错误和监控。不要写成“MCP 很重要”这种空结论。
