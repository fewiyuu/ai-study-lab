论文导读：参数记忆与非参数记忆

这一页只读 RAG 论文的开场问题：语言模型为什么需要把“记住的知识”拆成参数记忆和非参数记忆，又为什么检索要先进入生成链路。先把这条主线看顺，后面的概率公式、组件训练和实验解读才有落点。

学习目标

goal

先识别问题你要能说出：为什么单靠模型参数记住知识不够，问题不只在“会不会背”，还在“能不能更新、引用和检查”。

再看链路你要能按顺序复述 Figure 1：输入先变成 query，再去检索，再把文档作为条件送进生成器，最后对文档路径做边际化。

最后做区分你要能区分 closed-book、retrieve-and-extract 和 RAG，知道它们分别在哪一步把“外部知识”接进来。

先抓主线

overview

这页在回答什么

论文一开始先问一个很实际的问题：很多任务需要外部事实，但语言模型把知识压在参数里以后，更新慢、溯源难、出错时也不好查。RAG 的第一步，就是把“记住知识”这件事拆成两种记忆：一部分在参数里，一部分放到可替换、可检索的外部语料里。

所以这一页的作用让你看见论文为什么从这里起步。只要这层没看懂，后面的 RAG-Sequence、RAG-Token 和训练目标都会被误读成普通工程拼接。

一眼看懂的主线

输入问题、声明或待生成文本。

Query把输入变成检索用的查询表示。

检索从外部语料找出 top-k 文档。

生成把文档当条件，生成答案或文本。

边际化对“用哪篇文档生成”这个隐藏选择求和。

术语和边界

terms
术语直觉精确定义这一页先怎么看
参数记忆知识被写进模型权重里。模型参数里保存的分布式知识表示，更新要走训练或微调。把它当“难改但会泛化”的记忆。
非参数记忆知识放在外部文本库里。检索系统读到的文档集合，通常可替换、可重建、可检查。把它当“易换但依赖索引”的记忆。
知识密集型任务答题前先得找到事实。输出质量强依赖外部知识，单靠流畅语言不够。这页的例子都按这个标准看。
边际化先允许多篇文档都可能对，再把它们合起来。对潜在文档选择变量求和或近似求和。先记住“隐藏选择”，下一节再看公式。

> 先知道，先别追 这一页先不展开 MIPS 的索引结构、DPR 的训练细节、BART 的解码策略，也不追实验表里的所有数值。它们会在后面的单元里回来。现在只要抓住：外部知识为什么要进来，以及它在概率上怎么参与生成。

核心机制

trace
步骤读什么写什么错了会怎样
query encoder输入 x查询向量 q(x)检索入口偏掉，后续文档不对。
检索器query 和外部索引top-k 文档 z拿到的知识不相关，生成再流畅也没用。
生成器输入 x 与文档 z条件分布 p(yx,z)文档没真正参与，RAG 只剩拼接壳。
边际化多个文档路径的概率总体输出 p(yx)只看单篇文档会低估或误解模型输出。

把公式走一遍

worked

下面这个算例是我自己构造的，不是论文原始数值。目的很简单：把“检索概率乘上生成概率，再对多个文档求和”这件事真的算一遍。很多人第一次读 RAG 的时候，会以为只要检索分数高，答案就稳了；这个例子会直接把这个误解拆开。

问题设成“Washington 的首都是哪里？”，候选文档只有两篇：一篇讲 Washington state，一篇讲 Washington, D.C.。如果只看检索，state 文档可能排得很前；如果只看生成，模型也可能根据常见语料猜对首都。真正的 RAG 看的是两条路径一起贡献多少，而不是单看哪条路径更像“正确”。

文档 z检索概率 p(zx)生成概率 p(yx,z)路径贡献阅读时怎么解释
z 1 ：Washington (state)0.550.180.099检索像是相关的，但对当前答案帮助不大。
z 2 ：Washington, D.C.0.450.910.4095检索不一定排第一，但生成器在这条路径上更容易写出正确答案。
合计0.55 × 0.18 + 0.45 × 0.910.5085这就是边际化：把多条候选文档路径的贡献加起来。

$$p(y \mid x) = p_\eta(z_1 \mid x)p_\theta(y \mid x, z_1) + p_\eta(z_2 \mid x)p_\theta(y \mid x, z_2)$$

如果你把这条式子手算一遍，就会明白为什么论文不能只谈“检索到了没有”，还得谈“生成器在这条路径上贡献了多少”。检索和生成同一条概率链路的两部分。

> 这个算例真正想让你看到什么 检索分数高，不等于答案一定对；生成分数高，也不等于模型真的看懂了证据。RAG 的价值在于，它把“文档是否被找到”和“文档是否真的帮助生成”分开了。以后你读实验结果时，也要用这两个问题去看，而不是只看最终准确率。

公式里的符号怎么读

symbol

先把角色认清

p_η(z|x) 是检索侧的分布，回答“给定输入 x，哪篇文档 z 更值得被拿出来”。p_θ(y|x,z) 是生成侧的条件概率，回答“在看见输入和文档之后，应该怎么写输出 y”。这两个符号常被一起写，但它们负责的事情不一样。

字母 η 和 θ 只是把参数块区分开：前者偏检索器，后者偏生成器。你不需要在第一页就背住字母本身，但要知道它们在公式里代表不同的学习对象。

这一页先别追的东西

先别把注意力放到索引实现、向量维度、近似最近邻库或后面训练时的具体 loss 细节。当前页只需要回答两件事：这些符号各自代表谁，以及它们怎么拼成一个可以求和的生成概率。等到第 02 节和第 03 节，这些实现问题会回来。

符号读法代表什么常见误读
x输入问题、声明或待生成的文本条件。把 x 当成已经检索完的内容。
q(x)query输入经过编码后的检索信号。以为 query 只是原句复制。
z文档路径被检索到的候选文档或证据片段。把 z 误当成固定答案。
𝒵 K候选集top-k 检索到的文档集合。以为边际化只会看 top-1。
p_η(zx)检索分布在输入下选中某篇文档的概率。把它看成纯排序分数，不看概率意义。
p_θ(yx,z)生成分布给定输入和文档后的输出概率。以为生成器只是把文档原样抄出来。

训练时到底更新什么

training

这一页还有一层很容易被跳过的意思：RAG 不是把检索器和生成器随便串起来就完了，它还要回答“谁在学习、学什么、为什么这样分工”。论文里没有直接给检索监督标签，而是让模型从输入输出对里间接学出文档选择。这意味着训练时最重要的信号，来自最后能不能把答案写对，而不是来自人工给哪篇文档打了标签。

这也是为什么 query encoder、document encoder 和 generator 的角色不能混。query encoder 需要朝着更容易命中文档的方向更新；generator 需要朝着更容易在证据条件下写出答案的方向更新；document encoder 则常被固定住，避免每次训练都把索引重建一遍。你如果把这三件事混成一个“统一编码器”，就会同时碰到训练不稳定和检索失真两个问题。

组件读取什么更新什么为什么这样分如果做反了会怎样
query encoder输入 x把输入变成更能命中文档的表示让检索分布朝着更相关的文档收敛query 不稳，检索一路偏。
document encoder文档语料通常固定，少改或不改避免索引频繁失效和重建索引一变就要重算，系统很难维护。
generator输入 x 和文档 z学会在证据条件下生成 y把证据和表达接在一起模型会生成得很顺，但不一定真的看文档。
边际化目标多条候选路径的概率把“哪篇文档更对”传回前面的分布让训练不依赖硬标签检索监督只训练生成器时，检索器学不到方向。

> 为什么这一段对后面很重要 第 03 节会讲 DPR、BART 和 MIPS 时，你会再次看到这种分工。先把训练视角读懂，后面就不会把“检索器为什么这样训练”看成一个孤立实现问题，而会知道它和整条概率链路是绑在一起的。

30 秒怎么把这一页讲出来

retell

一个可直接复述的场景

有人问你：“RAG 不就是接个向量库吗？”你如果只回答“对啊，先检索再生成”，这个答法太短，也会把论文的关键点说丢。更稳的说法是：这篇论文先指出，知识密集型任务不能只靠参数记忆，因为知识会更新、需要可检查来源，而且不同任务对表达的要求也不一样；于是它把外部文档当成非参数记忆，让输入先形成 query，再检索文档，再让生成器在文档条件下写答案，最后对多篇文档路径做边际化。

这段话里最关键的顺序不能乱。只要你把“为什么需要外部知识”“外部知识在什么位置进入”“最后为什么还要对多篇文档求和”这三步说清，基本就不会把 RAG 读偏。

这类回答最常丢的东西

第一，漏掉“参数记忆和非参数记忆”的区分，就会把外部文档说成普通补充资料。第二，漏掉“边际化”，就会把 RAG 说成单篇检索后的拼接。第三，漏掉“更新和可检查”，就会把这篇论文讲成一个只有召回率的工程技巧。

所以这页要把“问题、位置、概率、边界”四件事串起来。后面读实验时，你也要按这四件事去检查结论有没有被说过头。

30 秒复述模板text

问题：为什么只靠参数记忆不够？
答：因为知识密集型任务需要可更新、可检查的外部证据。RAG 把检索接到生成前面，让 query 先找文档，再让生成器在文档条件下写答案，并对多篇文档路径求和。

问题：这和普通向量库有什么区别？
答：普通向量库只负责找资料，RAG 还把“文档如何影响生成”写进概率模型里，所以它不仅有检索，还有条件生成和边际化。
你说了什么够不够为什么
“先检索，再生成。”不够只有流程，没有解释为什么要这样做，也没有说边际化。
“RAG 让外部知识进入生成概率。”接近了已经有机制味道，但还没讲清参数记忆和非参数记忆的分工。
“它把检索、条件生成和边际化连成一条链。”够了这已经能把这页的主线、术语和概率关系一起说清。

例子

example

一个最小场景

问题是“Washington 的首都是哪里？”如果只靠参数记忆，模型可能答对，但你看不见它用了什么来源；如果知识库更新，模型本身也不容易马上改。RAG 的做法是先去检索关于华盛顿州和华盛顿特区的文档，再让生成器根据文档输出答案。这样你既能看结果，也能回看来源。

这里真正要注意的是：检索到文档，不等于答案自动正确。生成器还可能忽略、混淆或组合错误，所以论文讲的是“把外部知识接进概率模型”，不是“接上库就万无一失”。

为什么要多加一步

如果一个任务只需要把文档原句抄出来，retrieve-and-extract 就够了；但如果任务要改写、概括、组合多条证据，生成式路径更合适。RAG 选的是后者，所以文档进入后还要经过生成器，而不是直接当最终答案。

Figure 1 的最小伪代码pseudo

def rag_forward(x):
 q = query_encoder(x)
 docs = retriever.search(index, q, top_k=K)
 scores = []
 for z in docs:
 p_z = retrieval_prob(x, z)
 p_y = generator_prob(x, z)
 scores.append(p_z * p_y)
 return sum(scores)

$$p(y \mid x) = \sum_{z \in \mathcal{Z}} p(z \mid x)\, p(y \mid x, z)$$

这里的 z 是“选到哪篇文档”的隐藏变量。你可以先把它理解成多条可能来源的总和，而不是只盯着一篇文档。

任务对照

compare
路径怎么用知识适合什么任务这一页要记住的分界
Closed-book只靠参数记忆。不需要显式外部来源，或者只是一般语言生成。简单，但更新和追溯都弱。
Retrieve-and-extract先检索，再从文档里抽原文。答案在文本里原样出现，且不需要改写。看得到来源，但表达能力弱。
RAG先检索，再条件生成，并对文档路径边际化。既需要外部事实，又需要生成新文本。这是这一页真正要你读懂的路径。

> 对照时别混掉的点 这三个路径差别不在“有没有知识”，而在知识先进入哪里、最后有没有生成、以及文档路径是不是被显式纳入概率模型。你先把这三处分开，后面读公式就不会把 RAG 误读成普通检索拼接。

常见误区

debug

误区 1：把 RAG 读成工程拼接

论文里的 RAG 是端到端概率模型，不只是“先搜一段，再贴进 prompt”。后者是应用形态，前者是方法定义。

误区 2：以为检索到了就等于答对了

检索只是把候选知识拿上来，真正输出还取决于生成器怎么用这些文档。文档选得对，答案仍可能被模型写歪。

误区 3：把非参数记忆理解成不训练

外部索引本身不是权重，但 query encoder 和生成器是会学的；论文里固定文档编码器，是为了避免频繁重建索引。

误区 4：只看最终文本，不看来源路径

RAG 的价值之一是让你回看来源。如果只看最后一句话，你会漏掉它到底是从哪篇文档来的、是不是被模型误读了。

同一机制在不同任务里怎么出现

task

这篇论文没有把 RAG 只限定成“问答系统”。它想说明的是：凡是输入和输出都能写成文本、而且输出离不开外部事实的任务，都可以试着把检索接进来。这个角度很重要，因为它把“检索增强”从一个单点技巧，变成了一套通用的建模方式。

你读任务例子时，先别急着看任务名，先问自己三个问题：输入是什么，外部知识在哪里被用到，生成器到底是在抄文档还是在组织文档。只要这三问能答出来，任务名换成别的，你也不会乱。

任务输入 x输出 y外部知识在做什么如果只靠参数记忆，会丢什么
Open-domain QA自然语言问题。答案文本。提供事实、实体关系、时间和定义。你可能答得流畅，但很难保证事实更新。
FEVER事实声明。支持、反驳或信息不足。提供证据来判断声明是否成立。模型容易只凭语言模式猜标签。
Jeopardy 问题生成答案实体或线索要求。问题形式的文本。提供可转写成问题的背景信息。模型可能写得像题，但不够精确。
知识更新型问答问题 + 新近资料。可解释的答案。让最新资料替换掉旧记忆。旧参数会拖住答案，导致“看起来对，实际上旧”。

> 这个表背后的共同点 这四类任务的共同点“都不能只靠流畅生成来交差”。RAG 的价值，是让模型在需要事实时能先接触证据，再决定怎么说。这个差别在问答里很明显，在分类里也一样存在。

错了以后怎么查

failure

这页最有用的地方，你能不能在答案出错时把问题定位到某一层。RAG 的故障通常某一层把错误往下传：query 不准、检索偏了、生成器没用好文档，或者边际化把少数路径盖过去了。你只看最后一句，往往找不到真正的原因。

下面这段“伪日志”让你知道排查时应该看什么：先看输入有没有把问题说对，再看检索命中了什么文档，最后看生成器到底有没有受这些文档影响。

排查时可先读的最小日志log

query: "Washington capital"
top_docs:
 - doc_1: Washington (state)
 - doc_2: Washington, D.C.
generator:
 answer: "Washington, D.C."
symptom: answer fluent, evidence ambiguous
症状最可能的层下一步看什么修法方向
答案流畅，但事实不对。检索或边际化。top-k 文档是不是偏题，文档权重是不是被错误分配。先修检索，再谈生成。
答案像对了，但来源说不清。生成器与证据对齐。生成时是否真的读取了 z，还是只借用了输入里的表面模式。增加证据约束或改写解码策略。
换了索引后，答案还是旧的。非参数记忆更新。索引是否真的重建，文档编码是否同步更新。把外部知识替换流程接到真实更新上。
检索结果很多，但模型总盯着错误文档。query 表示或加权。query 是否把问题压歪，权重是否过度偏向某条路径。重看 query encoder 和边际化的权重设计。

notes

第一，Figure 1 的链路，最好画成一条从输入到边际化的箭头。第二，参数记忆和非参数记忆的区分，最好写成一张两列表。第三，至少留一个任务例子和一个失败例子，这样你下次回看时能直接判断自己有没有把“检索”和“生成”混成一件事。

这一页之后的接法

下一节会把这条链路写成概率模型，重点是 RAG-Sequence 和 RAG-Token 怎么把不同文档路径纳入生成。读到那里时，别再问“要不要检索”，而是问“检索的文档如何进入概率公式”。

你要写什么建议格式目的
主线“输入 -> query -> 检索 -> 生成 -> 边际化”保证你能复述流程。
区分参数记忆 / 非参数记忆 / retrieve-and-extract / RAG防止把不同路径混成一个口号。

知识更新的完整链路

update

这一页最开始说“参数记忆难更新”，这里把这句话落到一个具体的更新场景里。假设一条制度、地址或发布日期发生变化，参数记忆往往还会给出旧答案，因为旧知识已经被压进权重里，不会随着一份新文档自动改掉。RAG 的非参数记忆把这类事实放到可替换的外部语料中，所以更新时可以先换文档，再重建索引，再检查检索和生成有没有一起变。

你可以把这个过程理解成四个检查点：资料是否替换、索引是否重建、检索是否命中、答案是否跟着变。只要其中任何一步没接上，系统就会出现“看起来像模型没学会，其实是新知识没真正进来”的假象。这个链路之所以重要，是因为它把“知识更新”从一个抽象说法，变成了能逐层排查的操作问题。

检查点你在看什么读到什么算正常出问题时最像哪里坏了
资料替换新的事实有没有进语料。旧文档被新版本替换掉。源数据本身还停留在旧版本。
索引重建新文档有没有重新编码并入库。检索入口能访问到新文档向量。文档换了，但索引没换。
检索命中query 是否真的找到了新文档。top-k 里能看到最新证据。query encoder 或检索策略还在偏向旧文档。
答案变化生成器是否跟着新证据改写。输出文本和新资料一致。生成器没用上 z，或者边际化权重没把新路径抬起来。

更新问题的最小排查顺序workflow

1. 先确认新事实是否进入语料
2. 再确认索引是否重建
3. 再看检索是否命中新文档
4. 最后看生成结果是否真的变化

> 这条链路为什么能帮助你判断论文 如果你以后看到某个 RAG 系统“明明改过资料，答案却还是旧的”，不要先怀疑生成器。先走这四步，往往能定位到是资料、索引、检索还是生成哪一层没有更新好。论文把外部知识当非参数记忆，正是为了让这类问题能被单独检查出来。

> 练习钩子 拿一条你熟悉的事实，试着做一次更新实验：改掉事实后，看检索和答案是否同步变化。只要它们不同步，你就知道更新链路里还有一步没接上。

> 最小自测 把一个旧事实和一个新事实放进同一份文档里，先看检索结果会不会切到新版本，再看答案有没有真的改写。这个小动作能直接检出“资料没换”“索引没重建”“生成没看见证据”三类问题。

一眼判断错在哪一层

spot

如果你只想用这一页做一次快速排错，最省事的办法拿一个小问题做定位：先问“检索命中了什么”，再问“生成器有没有用证据”，最后问“边际化有没有把错误路径抬高”。这三问能把大多数 RAG 错误拆成可查的层，而不会把所有问题都说成“模型不行”。

先问什么看什么常见结论
检索命中了吗top-k 文档是否和问题相关。如果不相关，先修 query 或索引。
生成用了证据吗输出是否真的受 z 影响。如果没用上，修生成器或解码策略。
路径权重对吗边际化时哪些文档占了主要权重。如果错误路径太高，修分布和训练信号。

> 练习钩子 你下次看到一个答案似乎“说得通”，先不要下结论，先把这三问过一遍。只要有一问答不上来，你就知道问题不该往“语言流畅”上找，而应该回到检索、生成或边际化本身。
