为什么只靠模型参数不够

MOTIVATION

论文开头承认一个事实：大规模预训练语言模型确实会把大量知识压进参数。它们可以在下游任务上微调，也能回答不少事实问题。问题是，这种知识不容易被精确访问。模型知道某些事实，不等于它能在每次需要时稳定说出来，也不等于它能告诉你依据来自哪里。

参数记忆还有一个更实际的麻烦：更新成本高。世界变化了，比如国家领导人、公司 CEO、产品文档、法律条款发生变化，纯参数模型通常要继续训练或重新对齐，才能稳定改变行为。外部索引则可以直接替换或增量更新。RAG 的直觉就在这里：让模型保留语言生成能力，同时把可改、可查的知识放在外部。

这篇论文并不是今天工程里常见的“把文档塞进 prompt”的简单 RAG 流程。它提出的是一个可微调的概率模型：检索器给出一组可能相关的文档，生成器在每个文档条件下生成答案，最后把文档当作潜变量边际化。你读这篇论文时，要同时抓住两层：工程直觉是“先查资料再回答”，模型形式是“对潜在文档求和”。

读到这里，你应该能做一件事：解释为什么 RAG 不是为了让模型“显得更聪明”，而是为了让知识可以被查、被换、被检查。

参数记忆和非参数记忆怎么分工

MEMORY

参数记忆指的是模型权重里隐含的知识。BART 这类 seq2seq 模型在预训练中学到语言模式、实体关系、常识和表达能力。它强在生成自然语言，也能把片段组织成答案。但它的弱点也明显：你很难直接打开参数看见某条知识，也很难只改某一条事实。

非参数记忆在论文里是一个 Wikipedia 文档索引。它由 2100 万个 100 词片段组成，每个片段被 DPR 的文档编码器变成向量，再放进 FAISS 索引用 MIPS 检索。它强在可替换、可扩展、可检查。你可以看见被取回的片段，也可以用新的 Wikipedia dump 替换旧索引。

两者结合时，检索器负责从外部记忆里找候选文档，生成器负责读输入和文档后生成输出。注意，论文不是把检索结果当作固定正确答案。它把文档当成 latent variable，也就是训练数据里没有直接标注的隐藏选择。模型通过最终答案的概率来学习哪些文档有用。

记忆类型优势短板RAG 中的角色
参数记忆表达、概括、补全上下文难更新、难溯源、可能幻觉BART 生成器 pθ
非参数记忆可检查、可替换、可扩展检索可能漏证据，也需要索引维护Wikipedia dense index
判断句：如果知识需要频繁更新或需要引用依据，外部记忆比只改模型参数更自然。

RAG 在三类系统之间的位置

COMPARISON

论文把 RAG 放在 closed-book QA、open-book extractive QA 和通用 seq2seq 之间。Closed-book QA 像 T5，不用检索，直接从参数里生成答案。Extractive QA 先检索文档，再从文档中抽取答案片段。RAG 也检索文档，但它不是抽取一个 span，而是生成答案。

这个位置很关键。抽取式系统在证据文档里有原文答案时很强，但如果答案需要多个文档合成，或者相关文档提供的是线索而非原句，抽取就受限。生成式模型能合成答案，但没有外部资料时容易编。RAG 试图吃下两边的好处：保留生成能力，同时把生成过程接到检索文档上。

论文的实验也围绕这个定位展开：开放域 QA 上和 DPR、REALM、T5 比；MS-MARCO 和 Jeopardy 上看生成质量；FEVER 上看分类和证据检索。这不是随便选几个数据集，而是在验证同一个问题：一个统一的 retrieval-augmented seq2seq 形式，能不能跨问答、生成、分类任务工作。

Closed-book

只用参数记忆。优点是结构简单，缺点是更新和溯源困难。

Retrieve-and-extract

检索后抽取文本 span。适合答案在文档中原样出现的任务。

RAG

检索后生成，并对检索文档边际化。适合需要资料又需要生成表达的任务。

Figure 1 的链路：从 query 到边际化

FIGURE 1

Figure 1 展示了 RAG 的核心链路。输入 x 先进入 query encoder，得到 q(x)。检索器在文档索引里用 MIPS 找 top-K 文档 z1 到 zK。每个文档和输入一起送入生成器，生成器给出 pθ(y|x,z)。最后模型不是只选一个文档，而是把不同文档条件下的生成概率按检索概率加权求和。

这里最容易误解的是“检索器返回文档”这一步。论文里的文档不是硬标签，而是隐藏变量。训练数据只给输入和目标输出，没有告诉模型哪篇 Wikipedia 片段该被检索。RAG 通过目标答案的边际似然，把检索器和生成器连起来训练。

p(y|x) ≈ Σz ∈ top-K pη(z|x) · pθ(y|x,z)
你现在应该能做什么：拿任意一个 RAG 回答过程，指出哪一步是检索概率，哪一步是生成概率，哪一步在把多个文档的可能性合起来。

三个任务例子暴露同一个需求

TASKS

论文用问答、事实核查、Jeopardy 问题生成说明 RAG 的通用性。问答任务里，输入是问题，输出是答案。事实核查里，输入是 claim，输出是支持、反驳或信息不足。Jeopardy 生成更有意思：输入是答案实体，输出是一条包含事实线索的问题。

这些任务表面不同，但共同点是都需要具体外部知识。模型不能只靠语言流畅度过关。比如问“middle ear 的定义”，答案需要解剖事实；判断 The Divine Comedy 的结构，需要文献知识；给 Washington 生成 Jeopardy 问题，需要实体相关事实。RAG 的检索文档就是这些事实的来源。

从学习角度看，你不要把 RAG 只记成“问答系统”。这篇论文的野心是把 retrieval-augmented generation 做成通用 seq2seq 配方。任何能写成输入文本到输出文本的任务，只要需要外部知识，都可以尝试接入同一套形式。

任务输入 x输出 y外部知识的作用
Open-domain QA自然语言问题答案文本提供实体、日期、定义等事实。
FEVER事实声明分类标签提供支持或反驳声明的证据。
Jeopardy QGen答案实体问题线索提供可用于生成精确线索的事实。

读论文第一遍最容易踩的坑

PITFALLS

误区 1：把 RAG 当工程 prompt 拼接

论文的 RAG 是概率模型，有训练目标和潜变量边际化。工程拼接是后来的常见应用形态，但不能替代论文里的方法。

误区 2：以为检索文档都有监督标签

论文强调没有直接检索监督。训练数据是输入输出对，文档选择通过边际似然间接学习。

误区 3：把非参数记忆理解成不可训练

索引本身不是 trainable parameters，但 query encoder 会训练，检索分布会变。论文固定 document encoder，是为了避免频繁重建索引。

误区 4：认为 RAG 自动有依据

RAG 能让访问过的文档可检查，但生成仍可能忽略、误读或组合错误。事实性要靠检索、训练、解码和评估一起保证。

def rag_forward(x):
    query = query_encoder(x)
    docs = mips_search(index, query, top_k=K)
    weighted_probs = []
    for z in docs:
        retrieval_prob = p_eta(z, x)
        generation_prob = p_theta(y, x, z)
        weighted_probs.append(retrieval_prob * generation_prob)
    return sum(weighted_probs)
