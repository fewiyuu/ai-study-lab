概率模型：RAG-Sequence 与 RAG-Token

这一页只盯一件事：检索到的文档 z 在概率上怎么进入生成链路。先看文档为什么是 latent variable，再看两种模型把求和放在什么位置，最后把它们和解码、分类任务连起来。

这一页结束时，你先能

  • 说清 p_η(z|x)、p_θ(y|x,z)、top-K 截断和边际化分别负责什么。
  • 区分 RAG-Sequence 和 RAG-Token 的求和位置。
  • 解释为什么分类任务长度为 1 时，两种模型会等价。

学习目标

goal

先分清对象

你要先知道文档 z 是怎么从候选集里被抽出来的，再看它怎么进入生成分布。别把检索分数和最终答案混成一个量。

再看位置

你要能指出公式里求和是包住整段，还是包住每个 token。这个位置一变，模型假设就变了。

最后看用途

你要能把这两种形式接到解码和分类任务上，知道论文为什么说它们不是纯粹的数学变体。

先看主线

overview

这一页在回答什么

论文先把输入 x 送去检索，再拿回 top-K 文档 z。真正关键的是：z 不是提前知道的答案，而是一个要被边际化的潜变量。模型先假设“哪篇文档负责生成”这件事有多个可能，再把这些可能的贡献加起来，得到总体的 p(y|x)。

这里最容易读错的地方不是公式符号，而是求和位置。RAG-Sequence 让同一篇文档负责整段输出；RAG-Token 允许每个 token 都重新在候选文档里边际化一次。一个是整段路径上的求和，一个是逐 token 的求和。

一条可追的链路

before输入只是文本。

读取 x 以后，query encoder 先给出检索表示。

after候选文档进来。

检索器返回 top-K 文档，不是最终答案。

reads文档 z 和输入 x。

生成器在证据条件下估计 token 概率。

writes总体输出概率。

把多篇文档路径的贡献合起来。

failure检索偏了，后面都偏。

常见症状不是语法坏，而是来源不对。

术语和符号

terms
符号或术语 直觉 精确意思 这一页先怎么读

潜变量 训练时看不见，但模型必须对它积分或求和。 文档选择在观测上不直接给出，只能隐式估计。 先记住“隐藏选择”。

RAG-Sequence

model

它在说什么

RAG-Sequence 假设同一篇文档负责整个输出序列。也就是说，模型先选一篇文档，再在这篇文档的条件下把整段 y 写完。文档是一个潜变量，但这个潜变量在整段输出期间保持不变。

这带来的直觉很简单：如果答案本来就是围绕一份稳定证据展开的，整段共用一篇文档会更自然。它也让公式里“先对文档求和，再对 token 连乘”的结构特别清楚。

公式长什么样

$$p_{\text{RAG-Sequence}}(y \mid x) \approx \sum_{z \in \text{top-}k(p(\cdot \mid x))} p_{\eta}(z \mid x)\, p_{\theta}(y \mid x, z)$$

如果把 p_θ(y|x,z) 再拆成 token 乘积，z 的求和就包住了整段序列。读的时候别漏掉：这里的文档选择只做一次。

步骤 读什么 写什么 错了会怎样
检索 输入 x top-K 文档候选 候选偏了，后面再流畅也没用。
条件生成 输入 x 和某篇文档 z 整段输出的条件概率 如果 z 没真正参与，模型只是表面接了检索。
边际化 多篇文档路径的贡献 总体 p(y|x) 如果只看 top-1，会把模型学成硬检索器。

RAG-Token

model

它在说什么

RAG-Token 允许每个 token 都重新对文档做边际化。直觉上，这比整段共用一篇文档更灵活：不同 token 可以依赖不同证据片段，尤其适合一个答案里要拼接多条事实的场景。

它和 Sequence 的关键区别，不在“有没有检索”，而在“检索选择是整段一次完成，还是每个 token 都参与一次”。这个差别会直接影响生成自由度和解码方式。

公式长什么样

$$p_{\text{RAG-Token}}(y \mid x) \approx \prod_i \sum_{z \in \text{top-}k(p(\cdot \mid x))} p_{\eta}(z \mid x)\, p_{\theta}(y_i \mid x, z, y_{1:i-1})$$

这里求和包住的是每个 token 的条件概率。读的时候要注意：外层是连乘，内层是逐 token 的文档边际化。

两种模型最短对照text
Sequence: 先选一篇文档，再用同一篇文档生成整段。
Token: 每个 token 都可以在候选文档里重新边际化一次。

解码差异

decode

为什么解码不能一把梭

论文在测试时要近似 arg max_y p(y|x)。RAG-Token 可以像标准自回归 seq2seq 那样，把每一步的混合分布塞进 beam decoder；但 RAG-Sequence 的似然不是常规的 per-token 分解，不能直接照搬一次 beam search 解决。

所以解码差异不是“实现风格”，而是模型分解方式带出来的后果。模型怎么边际化，解码就得怎么近似。

对应到论文里的意思

RAG-Token 可以被看成普通的 autoregressive seq2seq 变体；RAG-Sequence 则要针对每篇文档单独跑 beam，再把候选再比较一次。前者更像逐步混合，后者更像多条整段路径的比较。

模型 解码直觉 论文里的结果 你要记的点
RAG-Token 像标准自回归生成器那样逐 token 解码。 可以把混合后的每步概率送进 beam decoder。 token 级边际化和自回归很接近。
RAG-Sequence 按文档分别估计整段候选，再比较整段路径。 不能只跑一次常规 beam 就结束。 整段路径上的似然是关键。

分类为什么会等价

classification

论文的那句话怎么理解

论文说，sequence classification 任务里如果把类别看成长度为 1 的目标序列，那么 RAG-Sequence 和 RAG-Token 会等价。原因不复杂：目标只有一个 token 时，就没有“整段共用文档”和“每个 token 换文档”的展开空间。

这也是为什么 FEVER 这类三分类任务能直接被写成 seq2seq 的输出形式。你不是在把分类硬塞进生成，而是在把标签写成一个长度为 1 的序列。

这页想提醒你什么

分类任务里，RAG 的核心不在“输出长文本”，而在“输出可以被写成序列”。一旦目标长度是 1，两种 RAG 形式在概率结构上的差别就消失了。

长度为 1 时的最小判断pseudo
target y = "supports"  # length = 1

两种模型手算

worked example

先算 Sequence

还是用“Washington 的首都是哪里？”这个例子。假设检索器返回两篇文档：`z1` 讲 Washington state，`z2` 讲 Washington, D.C.。RAG-Sequence 先对整篇文档做一次选择，再在这一篇文档条件下生成整段答案。

文档 `p_η(z|x)` `p_θ(y|x,z)` 路径贡献
`z1`：Washington state 0.55 0.18 0.099
`z2`：Washington, D.C. 0.45 0.91 0.4095
合计 0.55 × 0.18 + 0.45 × 0.91 0.5085

这一步要看的不是谁排第一，而是“高检索分 × 高生成分”能不能把最终概率抬起来。Sequence 的整段概率来自整段路径的加总。

再算 Token

现在把答案拆成两个 token，假设第一个 token 更依赖 `z2`，第二个 token 也更依赖 `z2`。RAG-Token 允许每一步都重新对文档做边际化，所以每个位置都能有自己的文档混合。

token 文档 1 贡献 文档 2 贡献 该步总概率
第 1 个 token 0.55 × 0.30 = 0.165 0.45 × 0.90 = 0.405 0.570
第 2 个 token 0.55 × 0.40 = 0.220 0.45 × 0.80 = 0.360 0.580
整段 0.570 × 0.580 0.3306

这个数值未必比 Sequence 大或小，关键是它表达了另一种假设：不同 token 可以从不同文档里取证据。你在论文里看到“Token 更灵活”，本质上就是这张表。

$$\text{Sequence: } \sum_z p_\eta(z\mid x)\prod_i p_\theta(y_i\mid x,z,y_{1:i-1})\qquad \text{Token: } \prod_i \sum_z p_\eta(z\mid x)p_\theta(y_i\mid x,z,y_{1:i-1})$$

把两个式子并排看，差别只在求和的位置，但模型假设完全不同。一个是整段共用证据，一个是逐 token 混合证据。

论文里有一句很短，但很关键：sequence classification 任务里，如果目标类被看成长度为 1 的序列，那么两种模型等价。很多人把这句当成附带说明，实际上它是在告诉你，RAG 的公式不是只服务长文本生成，它也能把标签当短序列来处理。

所以你读这一页时，最好把“框架图、概率公式、解码、分类退化”当成一条链，而不是四个孤立片段。这样后面回头看实验表，就不会只剩“哪个分数更高”这一层。

把原文拆成四句来读

第一句先看 Figure 1：retriever、generator、latent document、marginalization 都在图里。你读图时只需要确认谁负责选文档，谁负责把证据变成输出，不要在这一步就跳到训练细节。

第二句看 2.1 Models：Sequence 和 Token 的差别不在词面，而在求和放在整段外面还是每个 token 里面。读公式时，先把“谁被积掉、谁被保留”圈出来，再看生成器条件里到底带不带前缀。

落到排错日志

operation bridge

如果你把这页知识拿去做系统，最有用的不是背公式，而是知道哪里该留日志。你至少要能看见：检索器选了什么，生成器读了什么，最后是哪条路径把答案抬起来的。Sequence 和 Token 的差别，会直接体现在日志的长度和粒度上。

日志最小字段

如果你把这一页的思路塞进真实系统，日志至少要保留 query、top-K 文档、每篇文档的分数、生成前缀、最终输出和人工复核标签。少于这个集合，很多错误只能猜，不能查。

Sequence 场景里更要留“整段共用哪篇文档”的记录；Token 场景里更要留“每一步选了哪几篇文档”的记录。两者的差别不在名词，而在你能不能把错误定位到某一个位置。

三类常见失真

第一类是检索压偏：问题编码把关键词压丢了，top-K 看起来相关，实际上答不上。第二类是生成没读证据：文档找对了，但解码器没把文档信息接到输出。第三类是边际化错位：路径权重算错，最后把本来应该合并的证据拆散了。

你排查时不要先问“模型为什么不聪明”，先问“错误发生在哪一层”。先定位层，再讨论参数和数据，效率会高很多。

对象 读什么 写什么 哪里失真 常见症状

完整追踪案例

trace case

具体场景：同一个问题，两个文档，两种模型

假设用户问的是“Washington 的首都是哪里？”。检索器返回 `z1 = Washington state` 和 `z2 = Washington, D.C.`。如果你只看 top-1，很容易把整段答案写向 `z1`，最后输出 Olympia；如果你在 Token 级别逐步混合证据，又没有把路径记清楚，答案可能一半像州、一半像首都。

这个场景的关键不是“有没有搜到相关文档”，而是“哪一层负责把证据和输出对齐”。Sequence 需要整段共用一条文档路径，Token 允许每个位置重新混合文档，但两者都要求你把边际化放在正确的位置。

你应该盯住的信号

先记 query，再记 top-K 和分数，再记第一个 token 的候选文档贡献，最后记第二个 token 的贡献。只要这四类信息都在，你就能把错误切成三块：候选没进来、生成没用上、或者求和位置错了。

这比直接看最终字符串更有用，因为最终字符串只告诉你“坏了”，不会告诉你“坏在谁手里”。

步长 Sequence 观察点 Token 观察点 如果这一步错了
第 1 步 整段只认一篇文档，先确认是哪一篇。 给当前 token 算出各文档的混合权重。 如果文档选错，后面 token 再顺也救不回来。
第 2 步 前缀继续沿着同一条文档路径走。 允许新 token 再偏向另一篇文档。 如果前后文档混乱，答案会前后打架。
最终汇总 整段路径的概率被一次性合并。 每个位置的混合结果再连乘。 如果求和位置错了，看起来像近似，实际上是模型假设变了。
最小日志片段text
query: Washington capital
top_k: [z1=Washington state, z2=Washington, D.C.]
step_1: sequence_path=z2, token_path=z2, p_theta(y1|x,z2)=high
step_2: sequence_path=z2, token_path=z2, p_theta(y2|x,z2)=high
remark: if sequence_path and token_path diverge, check marginalization order first

失败后果很直接：如果你把 Sequence 的“一次选文档”误写成 Token 的“每步选文档”，日志看上去会一直正常，但整段答案其实已经把证据切碎了。排错时不能只看最终输出，要把路径和 token 一起看。

练习钩子

回到第 8 题时，先用这张表把回答说完整：你会保留哪些字段，哪一行说明 Sequence，哪一行说明 Token，哪一行说明边际化顺序。能说清这三点，就说明你已经能把公式落到真实调试里了。

什么时候选 Sequence，什么时候选 Token

如果任务更像单段回答，而且证据在整段里基本一致，Sequence 往往更省心，因为它把整段放进同一条文档路径里，日志和解释都更集中。你会更容易回答“这段话到底靠哪篇文档支撑”。

如果任务更像逐步生成，前后 token 可能分别依赖不同证据，Token 会更顺手，因为它允许每个位置重新混合文档。它不是“更高级”，只是把边际化放到了更细的粒度上。

先问：证据是否需要整段共用？
再问：输出是否会在 token 级别切换证据？
如果前者是，优先 Sequence。
如果后者是，优先 Token。
两种都不是默认正确，关键是任务形状要和边际化假设对齐。

常见误区

mistakes

误区 1：把 top-K 当最终答案

top-K 是近似边际化的候选范围，不是答案集合。模型后面还有生成概率要算。

误区 2：以为 Token 每步都重新检索

论文公式里 top-K 仍然来自同一个 p_η(z|x)。变化的是边际化位置，不是每一步真的重新查索引。

误区 3：只看检索分

某篇文档检索分高，不代表它对最终答案贡献一定大。贡献是检索概率和生成概率一起乘出来的。

误区 4：把边际化理解成平均

边际化是按概率加权求和，不是简单平均，也不是多数投票。
