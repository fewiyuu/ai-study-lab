latent variable看不见但影响结果的隐藏选择。训练数据没有给出哪个文档负责生成答案，z 因此是潜变量。不是随机噪声，而是模型要边际化的文档选择。
marginalize把多个可能文档的贡献加起来。对 top-K 文档求加权和或逐 token 求和。不是简单投票，也不是只拿最高分文档。
top-K approximation只用检索分最高的 K 个文档近似全库求和。全 Wikipedia 求和不可行，所以截断到 top-K。K 不是越大越好，计算和噪声都会增加。
RAG-Sequence一篇文档负责整段输出。先对完整序列概率 pθ(y|x,z) 加权求和。不是只能生成单句，而是假设同一 z 支撑整个 y。
RAG-Token每个 token 都可以从不同文档吸收信息。每一步生成 token 前对文档求和，再连乘。不是每个 token 重新检索，论文中 top-K 仍来自输入 x。
autoregressive前面的 token 影响后面的 token。pθ(yi|x,z,y1:i-1) 逐 token 生成。不代表文档只能逐 token 检索。

为什么文档是潜变量

LATENT

训练数据给的是输入 x 和目标输出 y，比如问题和答案。它没有告诉模型“应该检索哪篇 Wikipedia 片段”。但生成答案显然依赖某些文档。这个没有标注、却影响答案生成的文档 z，就是潜变量。

把 z 当潜变量后，模型不需要人工证据标签也能训练。它会尝试让那些能帮助生成正确 y 的文档获得更高贡献。贡献由两部分相乘：检索器认为 z 对 x 重要的概率 pη(z|x)，以及生成器在 x 和 z 条件下生成 y 的概率 pθ(y|x,z)。

这一步是论文的关键。RAG 不是“检索器先选一个文档，然后生成器无条件相信它”。它把多个候选文档都纳入概率计算。这样，如果几个文档都提供有用线索，它们都可以影响最终似然。

p(y|x) = Σz pη(z|x) · pθ(y|x,z)
判断句：看到“没有证据监督却训练检索器”，就要想到潜变量和边际似然。

RAG-Sequence：同一文档负责整段输出

SEQUENCE

RAG-Sequence 假设同一个检索文档 z 支撑完整输出序列 y。生成器在给定 x 和 z 后，自回归地产生 y 的每个 token。然后模型对 top-K 文档的完整序列概率做加权求和。

这个假设适合答案主要来自一个证据片段的任务。比如一个定义题、一个短事实问答，通常一篇文档足以支撑完整答案。RAG-Sequence 的直觉也比较容易解释：先问“如果读这篇文档，生成整个答案的概率是多少”，再把每篇文档的可能性加起来。

pRAG-Seq(y|x) ≈ Σz ∈ top-K pη(z|x) · Πi pθ(yi|x,z,y1:i-1)

注意公式里的求和在外层，连乘在内层。也就是说，每个文档先给出整段答案概率，再由检索概率加权。这个结构让它在解码时不能直接套一个普通 beam search，因为普通 beam search 需要每一步都有单一的 token 转移概率。

你现在应该能做什么：解释为什么 RAG-Sequence 的“同一个文档”是假设，不是检索器只能返回一篇文档。

RAG-Token：每个 token 都可以混合文档

TOKEN

RAG-Token 把文档边际化放到每个 token 的生成步骤里。每生成一个 token，模型都会计算这个 token 在不同文档条件下的概率，再按 pη(z|x) 加权求和。然后把每一步的结果连乘成完整序列概率。

这让 RAG-Token 更适合需要多个文档组合的输出。论文在 Jeopardy 问题生成里观察到，一个问题可能包含两个事实线索，不同 token 对不同文档的 posterior 更高。比如生成 Hemingway 相关问题时，某些 token 更依赖提到 The Sun Also Rises 的文档，另一些 token 更依赖提到 A Farewell to Arms 的文档。

pRAG-Tok(y|x) ≈ Πi Σz ∈ top-K pη(z|x) · pθ(yi|x,z,y1:i-1)

公式差别很小，但含义很大。RAG-Sequence 是“先每篇文档生成整段，再混合整段概率”；RAG-Token 是“每一步先混合文档，再生成下一个 token 的分布”。你可以把它理解成文档负责粒度不同。

判断句：如果输出需要多个证据片段协作，RAG-Token 的建模假设通常更灵活。

两个公式只差求和位置，学习时要慢慢看

FORMULA

RAG-Sequence 和 RAG-Token 最容易被背混。不要先死记公式，先看“文档选择发生几次”。RAG-Sequence 对整段输出只选一次文档变量；RAG-Token 对每个 token 都混合一次文档变量。于是求和的位置不同。

如果你看到求和在连乘外面，它就是 Sequence：每个 z 负责完整 y。如果求和在每个 token 的连乘里面，它就是 Token：每一步都对 z 求和。这个判断比背公式更稳。

分类任务为什么会退化成同一种形式

CLASSIFICATION

论文提到，当目标类别被看成长度为 1 的目标序列时，RAG-Sequence 和 RAG-Token 等价。原因很简单：输出只有一个 token，就不存在“每个 token 可以换文档”和“整段共用文档”的差别。

这点在 FEVER 里很有用。论文把 supports、refutes、not enough info 映射成输出 token，然后按 seq2seq 方式训练分类。因为目标长度是 1，两种 RAG 形式没有建模差异。

这也提醒你，RAG 论文里的“生成”不是只能输出长文本。只要任务可以写成输入序列到输出序列，标签也可以是一个短输出。seq2seq 框架的统一性，正是论文想强调的一部分。

公式题常见误区

PITFALLS

误区 1：把 top-K 当最终答案集合

top-K 是近似边际化范围，不是最终答案。模型还要计算生成概率。

误区 2：以为 RAG-Token 每步重新检索

论文公式里 top-K 仍由 pη(·|x) 得到。Token 级别变的是边际化位置，不是每个 token 重新查询索引。

误区 3：只看检索概率

一个文档检索分高，但生成目标答案概率低，最终贡献也可能不大。贡献来自 pη 和 pθ 的乘积。

误区 4：把边际化理解成平均

边际化是按概率加权求和，不是简单平均，更不是多数投票。
