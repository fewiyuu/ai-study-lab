# 学习目标 {#goals}
<!-- tag: goal -->

把目标函数说准
你要能看懂论文 2.4 Training 的核心句子：给定训练对 `(x_j, y_j)`，最小化目标答案的 negative marginal log-likelihood。这里的 marginal 指文档变量被合并掉了，不需要手工标注“答案来自哪篇文档”。


把训练信号追清
你要能说出为什么一个答案级损失可以改变检索器。直觉上，能让正确答案概率更高的文档路径会被奖励；经常把模型带偏的路径会被压低。这个信号通过 `p_η(z|x)` 回到 query encoder。


把工程边界立住
你要知道论文没有重训整套 DPR 文档编码器。它固定 `BERT_d` 和预建索引，只 fine-tune `BERT_q` 与 BART generator。这样牺牲了一部分自由度，换来可运行的训练流程。

# 训练主线 {#overview}
<!-- tag: overview -->
<!-- layout: grid2 -->

## box
### 从“会生成”到“会用证据生成”
上一节讲到 RAG-Sequence 和 RAG-Token 都把文档 z 当成潜变量。到了训练环节，论文要解决的问题很实际：数据集通常只给问题和答案，不给证据文档标签。如果直接要求 retriever 学“正确文档 id”，训练数据不够；如果只训练 generator，检索器又不能适应任务。
RAG 的做法是把文档选择放在概率路径里。对每个训练样本，retriever 先给出 top-K 文档和概率，generator 再在每篇文档条件下给目标答案概率，最后把这些路径贡献加起来。损失只看目标答案的边际概率：答案概率越高，损失越低；能解释答案的文档路径自然得到更强信号。
这个设计让 RAG 的训练目标和普通 sequence-to-sequence 模型保持相似：仍然优化答案的 log-likelihood。但它比普通生成模型多了一层可学习检索分布，所以训练时不仅让生成器更会写答案，也让 query encoder 更会把输入拉向有用证据。

## box soft
### 一条训练样本怎么走
读入训练对(xj,yj)
检索 top-KBERTq
逐文档打分BART
路径相加Σz
反向更新η,θ
**input**只有问题和目标答案。例如 question = “谁创立了 X？”，target = “某人”。没有给证据文档 id。
**retrieve**query encoder 生成检索向量。固定的文档索引返回 top-K 文档及相似度，形成 `p_η(z|x)` 的近似范围。
**generate**每篇文档都试着解释答案。BART 计算 `p_θ(y|x,z)`，文档好不好要通过答案概率体现。
**loss**把文档路径贡献相加。边际概率高，`-log` 损失低；边际概率低，训练会推高更有用的路径。
**update**只更新 query encoder 和 generator。`BERT_d` 与文档 index 固定，避免每步训练都重建巨大的向量库。

| 阶段 | 读什么 | 写什么 | 如果读错会怎样 |
| --- | --- | --- | --- |
| 训练对 | x_j 、 y_j | 本轮监督目标 | 把答案标签当文档标签，会误解整套训练目标。 |
| 检索 | 输入、query encoder、固定索引 | top-K 文档和 p_η(z|x) | 把 top-K 当答案集合，会漏掉生成器条件概率。 |
| 生成 | 输入、候选文档、目标答案前缀 | p_θ(y|x,z) | 只看检索分数，无法知道文档是否真的支持目标答案。 |
| 边际化 | 每条文档路径的检索概率和生成概率 | p(y|x) | 只取 top-1 会把潜变量近似读得过窄。 |
| 反向传播 | negative marginal log-likelihood | 更新 BERT_q 和 BART | 以为 retriever 完全冻结，会看不懂 learned retrieval 的消融提升。 |

# 目标函数 {#objective}
<!-- tag: formula -->
<!-- layout: grid2 -->

## box
### 一句话版
RAG 训练时最小化的是目标输出的 negative marginal log-likelihood。写成训练样本 `j` 的形式，就是：
\[
 \mathcal{L}_j = -\log p(y_j \mid x_j)
 \]
 这里的 `p(y_j|x_j)` 已经把候选文档 z 的贡献合并进去。损失不要求“第几篇文档是标准证据”。
如果用 RAG-Sequence 的直觉写出内部结构，可以先看成：
\[
 p(y \mid x) \approx \sum_{z \in \mathrm{top}\text{-}K(x)}
 p_{\eta}(z \mid x)\, p_{\theta}(y \mid x,z)
 \]
 top-K 是近似范围，`p_η` 是检索器，`p_θ` 是生成器。训练信号来自整条路径乘积，不能只看检索分数。

## box soft
### 一个两文档手算例子
假设同一个问题检索到两篇文档。第一篇看起来更相关，但生成器在它条件下不太能写出目标答案；第二篇检索分数低一些，却更能支持答案。边际化会把两条路径都算进来。
文档`p_η(z|x)``p_θ(y|x,z)`路径贡献

d10.600.100.060
d20.400.700.280
合计--`p(y|x)=0.340`
这一轮损失约为 `-log(0.34) ≈ 1.08`。更重要的是路径贡献暴露了一个训练方向：d2 虽然初始检索概率低，但它对目标答案更有帮助，训练会倾向于让类似 d2 的路径更容易被检索到。

# 信号怎么回传 {#signal}
<!-- tag: gradient -->
<!-- layout: grid2 -->

## box
### 没有文档标签，也能训练 retriever
这个点非常容易卡住：如果没有标准文档标签，retriever 怎么知道该调向哪里？答案是，retriever 不直接学习“这篇文档是 gold”。它学习的是：哪些文档路径让目标答案的边际概率变高。只要 `p_η(z|x)` 参与了 `p(y|x)` 的计算，答案级损失就能给它方向。
直觉上，某篇文档被检索出来后，如果 generator 在它条件下能稳定写出正确答案，那这条路径的乘积贡献会大。训练会提高这种路径在相似输入上的权重。相反，检索概率高但生成目标答案概率低的路径，会在损失里显得“不划算”。
这里并非把 generator 当裁判强行给文档打标签，更像是联合模型在同一个目标下分配责任：retriever 负责把有用证据带进来，generator 负责把证据转成目标输出。两个部分通过边际似然绑在一起。

## box warn
### 三种读法的差别
读法会怎么训练问题

把文档当已知标签对文档 id 做分类多数知识密集任务没有这样的监督。
只训练 generatorretriever 固定，答案损失只改写作能力检索不能适应任务，消融中通常吃亏。
RAG 的边际似然答案损失同时影响检索分布和生成分布需要 top-K 近似与稳定的索引设计。

# 固定索引 {#fixed-index}
<!-- tag: implementation -->
<!-- layout: grid2 -->

## box
### 论文到底更新了什么
论文说 jointly train retriever and generator，但要读完整句。RAG 训练时更新的是 query encoder `BERT_q` 和 BART generator；document encoder `BERT_d` 与预先构建的文档索引保持固定。这样 retriever 仍然被训练，因为 query tower 会变；只是文档 tower 不跟着变。
为什么这样做？如果 `BERT_d` 也更新，Wikipedia 每个 chunk 的向量都可能变化，索引就需要周期性重建。论文的检索库规模是约 2100 万个 100-word disjoint passages。这种规模下，重建文档向量和 MIPS 索引成本很高。
所以这里有一个很工程的折中：固定文档空间，让 query encoder 学会在这个空间里移动；同时让 generator 学会利用检索到的文本。模型没有获得“全量重排文档空间”的自由，但训练流程变得可负担。

## box soft
### 训练伪代码
fixed document index, trainable query encoder + generator复制
```python
# document_encoder = BERT_d, fixed after building the index
# query_encoder = BERT_q, trainable
# generator = BART, trainable

doc_vectors = load_precomputed_wikipedia_vectors()
index = load_mips_index(doc_vectors)

for x, y in training_pairs:
 q = query_encoder(x) # trainable BERT_q
 docs = index.search(q, top_k=K) # fixed document vectors
 path_probs = []

 for z in docs:
 retrieve_prob = retriever_score(q, z)
 target_prob = generator.prob(y, x, z)
 path_probs.append(retrieve_prob * target_prob)

 marginal_prob = sum(path_probs)
 loss = -log(marginal_prob)
 loss.backward()
 optimizer.step() # updates BERT_q and BART only
```


| 对象 | 训练中状态 | 这样设计的原因 | 如果误读会怎样 |
| --- | --- | --- | --- |
| BERT_q | 更新 | 让输入 query 更贴合任务需要的证据。 | 会误以为 retriever 没有 learned retrieval。 |
| BART generator | 更新 | 让生成器学会在检索文本条件下输出目标。 | 会把训练效果全部归给检索器。 |
| BERT_d | 固定 | 避免训练中反复更新全库文档向量。 | 会低估重建 index 的工程成本。 |
| Wikipedia vector index | 固定 | MIPS 检索需要稳定向量空间。 | 会想象每个 batch 后都能免费刷新向量库。 |

# 消融证据 {#ablation}
<!-- tag: ablation -->
<!-- layout: grid2 -->

## box
### 为什么消融能验证训练目标
如果训练目标真的能给检索器信号，那么 learned retrieval 应该比 frozen retriever 更好。论文的 Retrieval Ablations 正是这样做的：冻结 retriever 再训练模型，对比正常可学习检索；再用 BM25 替换神经检索，看不同任务的表现。
结果的解读要细一点。论文报告 learned retrieval improves results on all tasks，说明任务损失确实帮助检索器适应下游任务。BM25 在 FEVER 上表现强，论文解释与 FEVER claim 的实体中心特征有关；在 open-domain QA 上，differentiable retrieval 的优势更明显。
这给本页一个很好的闭环：训练目标已经落到可检验行为上。只要把 retriever 固定，性能就会掉；把检索换成 BM25，任务差异也会出现。训练目标、参数更新范围和消融结果三者能互相印证。

## box info
### 三组设置怎么读
设置检索器是否被任务训练读表时看什么本页结论

learned retriever是，至少 `BERT_q` 更新整体任务表现边际似然能给检索分布任务信号。
frozen retriever否相比 learned retrieval 的下降只训练 generator 不够。
BM25否，非神经稀疏检索按任务看强弱实体匹配型任务可能强，但开放问答更依赖可学习检索。

> **自审问题** 如果你能用一句话解释“FEVER 上 BM25 强”为什么不推翻 learned retrieval 的价值，就说明你已经开始按任务边界读消融了。更准确的说法是：BM25 在特定实体匹配场景里有优势，但 RAG 的可微检索在更需要语义匹配和答案生成的开放问答里更重要。

# 工程设置 {#setup}
<!-- tag: setup -->
<!-- layout: grid2 -->

## box
### Appendix C 给了哪些背景
论文附录 C 的训练设置不需要背，但能帮助你理解为什么固定索引是合理选择。作者使用 Fairseq 训练，采用 mixed precision，主要在 8 张 32GB V100 GPU 上训练。论文也说明一个 GPU 可以训练和推理，只是速度和规模会受限。
检索部分使用 FAISS 做 MIPS。论文说 CPU 上的 MIPS 足够快，不一定要把索引放在 GPU 上；但文档 index vectors 放在 CPU 上大约需要 100GB 内存，后续可以压缩到 36GB。这个数字足以说明：如果每次更新文档编码器都要重算全库向量，训练成本会被索引维护拖住。

## box warn
### 把数字换成设计判断
论文信息设计含义学习时重点

8 张 32GB V100 + mixed precision端到端训练本身已经不轻。训练目标要考虑可运行性。
FAISS MIPS 可在 CPU 上跑检索不一定抢 GPU 资源。瓶颈不只在生成器前向。
index vectors 约 100GB，可压缩到 36GB文档向量库是重量级状态。固定索引是规模约束下的选择。
代码后续迁移到 HuggingFace Transformers实现生态变化，但论文目标不变。不要把库 API 和论文机制混成一件事。

# 数据边界 {#boundary}
<!-- tag: data -->
<!-- layout: grid2 -->

## box
### 多答案和候选过滤只是边界处理
附录 D 里有一些数据处理细节，容易被误读成训练目标的一部分。NQ 和 WebQuestions 可能有多个 answer annotations，论文把它们作为多个 `(q, a)` 训练对处理。TriviaQA 会过滤掉不出现在 top 1000 检索文档中的候选答案。CuratedTrec 会用 top 1000 文档里的 regex 匹配数量选择 target。
这些处理说明训练样本的 `y_j` 有来源约束，数据集需要适配成输入输出对。但它们没有改变本页的主结论：RAG 的训练监督仍然落在答案文本上，文档选择仍然通过边际似然隐式学习。

## box soft
### 边界表
数据集/场景论文处理和训练目标的关系

NQ / WebQuestions多个答案标注作为多个 `(q,a)` pair。增加目标答案样本，不提供 gold document。
TriviaQA过滤不在 top 1000 文档中的候选答案。提高候选目标与检索语料的一致性。
CuratedTrec用 regex 在 top 1000 文档中匹配 target。让答案目标更可验证，但仍按答案似然训练。

# 训练排查案例 {#debug-case}
<!-- tag: case -->
<!-- layout: grid2 -->

## box
### 把论文机制换成一条真实训练日志
假设你在训练一个企业制度问答 RAG。输入 `x` 是“试用期员工能申请年假吗？”，目标答案 `y` 是“满足连续工作时间要求后可以按规定申请”。训练数据没有标注哪份制度文件支持答案，只给了问题和目标回复。初始检索时，top-1 是“试用期考核办法”，top-2 是“员工休假管理规定”。前者和问题词面更像，后者才真正支持目标答案。
如果只看检索分数，你会以为 top-1 应该被强化；如果只看最终答案，你又不知道是哪篇文档发挥了作用。RAG 的边际似然把两件事放在同一张表里：每篇文档既有检索概率，也有在该文档条件下生成目标答案的概率。路径贡献最高的文档，才是这轮样本里最能解释目标答案的证据路径。
候选文档检索概率目标答案概率路径贡献排查结论

试用期考核办法0.550.080.044词面相关，但证据不足。
员工休假管理规定0.300.620.186更能支持目标答案，应该获得更强任务信号。
薪酬福利总则0.150.120.018主题邻近，但不构成核心证据。

## box soft
### 建议保留的最小日志
训练 RAG 时，最好不要只记录最终 loss。你至少要能回放每个样本的 top-K 文档、检索概率、生成目标答案概率和路径贡献。这样 loss 下降时，你才知道是 generator 单独变强，还是 query encoder 真的开始把问题推向更好的证据。
one training example audit log复制
```json
{
 "question": "试用期员工能申请年假吗？",
 "target": "满足连续工作时间要求后可以按规定申请",
 "top_k": [
 {"doc": "试用期考核办法", "retrieve_prob": 0.55, "target_prob": 0.08, "path": 0.044},
 {"doc": "员工休假管理规定", "retrieve_prob": 0.30, "target_prob": 0.62, "path": 0.186},
 {"doc": "薪酬福利总则", "retrieve_prob": 0.15, "target_prob": 0.12, "path": 0.018}
 ],
 "marginal_prob": 0.248,
 "loss": 1.394
}
```

这条日志还能直接服务练习题 11：如果检索结果长期不变，就查 optimizer 是否包含 query encoder，检索分数是否进入边际概率，top-K 是否被缓存成不可更新的固定结果。排查的目标在于确认训练目标真的能改变检索分布，而不只是在最终答案上看到 loss 下降。

# 常见误区 {#mistakes}
<!-- tag: debug -->

误区 1：把训练目标看成检索分类
如果你说“RAG 训练时让模型预测正确文档”，就偏了。论文这里没有文档监督。更稳的表达是：训练目标最大化目标答案边际概率，文档变量在 top-K 候选上被边际化。


误区 2：以为 retriever 完全固定
论文固定的是 document encoder 和 index，并非整个 retriever。query encoder 会跟着任务损失更新，所以 learned retrieval 的效果仍然成立。


误区 3：把 BM25 强解释成神经检索无用
BM25 在 FEVER 这类实体中心 claim 上强，不代表开放问答里也同样合适。消融要按任务读，不能拿一个任务的优势覆盖全部结论。


误区 4：忽略固定索引的代价来源
固定索引源自工程成本：更新文档 encoder 会导致全库文档向量和索引维护变重。读 Appendix C 的内存数字时，要把它和训练设计连起来。


误区 5：只记住 `-log`
普通生成模型也会用负对数似然。RAG 的特别之处在 marginal：文档路径被加总，检索概率和生成概率共同决定目标答案概率。


误区 6：把数据预处理当模型机制
多答案展开、top 1000 过滤、regex 选择 target 都是数据边界处理。它们帮助构造训练样本，但不改变 RAG 通过答案似然训练检索和生成的主结构。

# 下一节怎么接 {#bridge}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box
### 从训练目标接到解码
训练时我们用目标答案 `y_j` 计算边际概率。推理时没有目标答案，模型要自己生成。于是问题会变成：给定同样的检索候选，RAG-Sequence 和 RAG-Token 怎么做 decoding？它们的文档路径是整段共享，还是每个 token 都重新混合？这会影响 beam search 和任务适配。

## box soft
### 复习时的最短链路
1. 第 02 节：文档 z 是潜变量，Sequence 和 Token 的求和位置不同。
2. 第 03 节：训练目标是 `-log p(y_j|x_j)`，文档标签缺失时仍能训练 query encoder。
3. 第 04 节：推理时没有 `y_j`，需要看解码和任务输出格式。

### 1 choice

**Q:** 1. RAG 在 2.4 Training 中最核心的训练目标是哪一个？

- [ ] . 最小化目标答案的 negative marginal log-likelihood。
- [ ] . 直接预测 gold document 的 id。
- [ ] . 只训练 BM25 的词项权重。

Answer: . 最小化目标答案的 negative marginal log-likelihood。

### 2 true/false

**Q:** 2. RAG 训练必须拿到每个答案对应的标准证据文档标签。

- [ ] 对
- [ ] 错

Answer: false

### 3 choice

**Q:** 3. 论文中 RAG 训练主要更新哪两个对象？

- [ ] . BERT_d 和 Wikipedia index。
- [ ] . BERT_q 和 BART generator。
- [ ] . BM25 和 regex target selector。

Answer: . BERT_q 和 BART generator。

### 4 open

**Q:** 4. 用一句话解释“没有文档标签，也能训练 retriever”。

Explain: 合格答案要包含：训练损失来自目标答案的边际概率；文档 z 是潜变量；有用文档路径通过 p_η(z|x)p_θ(y|x,z) 获得更高贡献；信号会回到 query encoder。

### 5 order

**Q:** 5. 哪个顺序最接近一轮 RAG 训练？

- [ ] . 重建文档索引 → 标注 gold document → 训练 BM25。
- [ ] . 生成答案 → 再检索文档 → 删除低分答案。
- [ ] . 读入 (x,y) → 检索 top-K → 计算各文档下目标答案概率 → 求和 → 反向更新。

Answer: . 读入 (x,y) → 检索 top-K → 计算各文档下目标答案概率 → 求和 → 反向更新。

### 6 calculation

**Q:** 6. d1 的路径贡献是 0.60×0.10，d2 是 0.40×0.70。哪个文档对本轮目标答案贡献更大？

- [ ] . d1，因为检索概率更高。
- [ ] . d2，因为路径贡献 0.28 大于 0.06。
- [ ] . 两者一样，因为都在 top-K 里。

Answer: . d2，因为路径贡献 0.28 大于 0.06。

### 7 true/false

**Q:** 7. 固定 `BERT_d` 和 index 的意思是 retriever 完全不会被训练信号影响。

- [ ] 对
- [ ] 错

Answer: false

### 8 open

**Q:** 8. 用自己的话复述 `p(y|x)=Σ_z p_η(z|x)p_θ(y|x,z)` 在训练里的含义。

Explain: 要点：对 top-K 文档路径求和；每条路径由检索概率和生成目标答案概率共同决定；损失看的是合并后的目标答案概率；这让文档选择成为可学习但不需要直接标注的潜变量。

### 9 diagnosis

**Q:** 9. learned retriever 比 frozen retriever 好，最直接支持哪个判断？

- [ ] . 答案级边际似然能给 query encoder 有用训练信号。
- [ ] . 文档索引每一步都被重新构建。
- [ ] . BM25 在所有任务上都失败。

Answer: . 答案级边际似然能给 query encoder 有用训练信号。

### 10 choice

**Q:** 10. BM25 在 FEVER 上强，比较稳的解释是哪一个？

- [ ] . 神经检索没有训练价值。
- [ ] . FEVER 不需要任何文档。
- [ ] . FEVER claim 更偏实体中心，词项匹配可能更占优势；这不覆盖开放问答任务。

Answer: . FEVER claim 更偏实体中心，词项匹配可能更占优势；这不覆盖开放问答任务。

### 11 scenario-transfer

**Q:** 11. 你训练一个 RAG 系统时发现答案损失下降，但检索结果几乎不变。你会先查哪三件事？

Explain: 建议检查：query encoder 是否真的进入 optimizer；retrieval score 是否参与可微的边际概率计算；top-K/索引调用是否把梯度路径切断成纯缓存结果。还可以查看 retriever 参数梯度范数和训练前后相似问题的 top-K 变化。

### 12 synthesis

**Q:** 12. 用 2-3 句话概括本页在 RAG 论文里的作用。

Explain: 合格答案要说出：这一页解释 RAG 如何用答案级负边际对数似然训练；文档没有直接标签，作为潜变量在 top-K 中被求和；训练更新 query encoder 和 generator，固定 document encoder/index，并通过消融证明 learned retrieval 有价值。
