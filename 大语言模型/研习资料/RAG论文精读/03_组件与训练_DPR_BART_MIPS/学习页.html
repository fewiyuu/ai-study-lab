学习目标

goals

先抓住三件事

这一页的边界

这里先不展开 DPR 的预训练细节，也不讲完整解码过程。你只需要先把组件、索引、训练这三层接上。解码差异和任务适配放回后续单元。

先看主线

overview

1输入 x 进入 query encoder，得到 q(x)。

2文档 z 先经 document encoder 变成 d(z)，离线写进索引。

3MIPS 在索引里找 top-K 文档，给出 p_\eta(z|x)。

4BART 读取 x + z，生成目标 y。

5训练只把梯度回传给 query encoder 和 BART，索引和 document encoder 固定。

DPR 检索器

retriever

它读什么，写什么

DPR 是一个 bi-encoder。query encoder 读输入 x，document encoder 读文档 z。两边都变成向量后，只靠内积 d(z)^\top q(x) 做打分。这样一来，文档向量可以离线算好，查询向量在线算，检索就能跑到大规模索引上。

论文里把这个问题落成 MIPS：给定一个查询向量，找内积最大的那些文档向量。FAISS 只是实现这个检索的工具，不是问题本身。

为什么要分成两路

如果把问题和每篇文档放在一起做联合编码，检索会更细，但代价很高。RAG 需要的是先把文档编码成可复用的记忆，再让查询去选它。这样大索引才能常驻在外部记忆里，而不是每个问题都重算一遍全库。

BART 生成器

generator

它怎么接上检索结果

论文没有发明新的生成器，而是直接用 BART-large。做法很直：把输入 x 和检索到的内容 z 拼在一起，让 seq2seq 模型在这个上下文里生成输出 y。这里的关键“BART 能不能把检索到的证据用成答案”。

论文把 BART generator 参数称为 parametric memory。意思很具体：模型权重里已经存着很多语言和事实模式，检索到的文本只是把它们拉出来、校正方向，不是完全替代权重记忆。

别把它理解成复读机

生成器不是把文档原句抄出来就算完成任务。它要做的是整合、改写、补全和选择。如果检索到的证据不完整，BART 仍然会依赖自己的参数记忆；这也是为什么论文后面会观察到 parametric 和 non-parametric memory 互相配合。

训练目标

training
$$ \min_{\eta, \theta}\; -\log p(y \mid x) \approx -\log \sum_{z \in \text{top-}K(p(\cdot \mid x))} p_\eta(z \mid x)\,p_\theta(y \mid x, z) $$

这里的 \eta 对应 retriever，θ 对应 generator。z 是 latent document，意思是训练时没有直接给“该检索哪篇文档”的标签，只给输入/输出对 (x, y)。

论文为什么不更新 document encoder

更新 BERT_d 不是不行，而是会让索引不断过期，训练成本很高。论文明确说：固定 document encoder 和索引，只 fine-tune query encoder 与 BART。这个选择把“训练信号怎么回传”与“索引怎么维护”分开了。

训练时的最小读写关系复制

# offline: build the non-parametric memory once
doc_vectors = BERT_d(documents)
index = FAISS.build(doc_vectors)

# training step: only retriever query side + generator receive gradient
q = BERT_q(x)
docs = index.search(q, top_k=K)
loss = -log(sum(p_eta(z|x) * p_theta(y|x, z) for z in docs))
loss.backward()
update(BERT_q, BART)

> 检查这段伪代码时看三件事 BERT_d 和 index 是离线写入；BERT_q 在线读取输入并参与更新；BART 读的是 x + z，最后和 retriever 一起接收训练信号。只要这三件事顺了，为什么“固定索引”就不难理解。

工程设置

setup

论文里真实做了什么

  • 非参数知识源用 December 2018 Wikipedia dump，切成约 21M 个 100-word chunks。
  • 文档索引用 FAISS 加 HNSW 近似，在 CPU 上维护；论文给出的索引内存大约 100GB，压缩后约 36GB。
  • 训练用 Fairseq、mixed precision 和 8 张 32GB V100，优化器是 Adam。

这些数字为什么要记

它们提醒你：非参数记忆不是“免费外挂”。索引有版本，有内存成本，也有更新成本。论文把这个成本摊开，正是为了说明为什么固定索引是一个有意识的系统选择。

训练排查案例

debug

一个真实会遇到的症状

假设你把旧索引换成了新索引，但某些问题的答案反而更差了。最常见的现象：检索出来的文档看起来相关，但答案总是对不上；或者模型在标题类问题上开始顺手续写旧知识。

这时不要先怪 BART。先看索引版本、文档编码器是否和文档源一起更新，再看 query encoder 有没有把问题推到旧分布里。如果索引本身已经过期，生成器再强也只能在错误证据上工作。

怎么一步步查

  1. 先打印当前索引版本和文档源时间，确认检索库是不是你以为的那一版。
  2. 再看 top-K 文档的标题和正文是否真的覆盖问题所需事实，而不是只在表面词面上相似。
  3. 如果检索结果对了但答案仍偏，再看 BART 是否在过度依赖参数记忆。
症状最先看哪里常见根因下一步检查
回答像旧知识索引版本热替换后没有重建文档向量比较索引时间戳和文档源时间戳
检索结果词像但事实不对query encoder问题被送到了表面相似但任务不相关的文档看 top-K 文档是否真的覆盖答案证据
检索对了但答案还是绕BART generator生成器过度依赖 parametric memory对比带检索和不带检索时的 partial decoding

> 这段案例想让你形成的习惯 先分清坏在“索引版本”“检索分布”还是“生成器使用证据”哪一层。RAG 的故障定位，和它的组件划分是一致的。

索引热替换

hot-swap

同一个问题，换一版索引会发生什么

论文拿 December 2016 和 December 2018 两版 Wikipedia 做对照，问题模板保持不变，只替换文档库。这个实验的意义很直接：如果索引版本和问题事实版本一致，模型就能答出对应年份的世界知识；如果版本错位，答案会明显变差。这里最值得你记住的，“非参数记忆真的可以通过换索引更新”。

更细一点看，热替换需要四步：先用新文档重算 d(z)，再把文档向量写进新索引，然后保持同一问题模板不变，最后比较输出是否切到新事实。这个流程一旦跑通，就说明模型不是只会在参数里背旧知识。

热替换时该记录什么

至少记录四项：dump 日期、chunk 切分规则、索引版本、问题模板。这样你才知道某次回答到底是在读哪一版记忆。若答案没变，先查文档向量是否重算，再查索引是否真替换，最后才查生成器是否把旧参数记忆顶在前面。

实验条件你做了什么你预期看到什么结果不对时优先查哪里
2016 对齐 2016旧问题模板 + 旧索引答案跟 2016 版事实一致先看文档向量是否确实来自 2016 dump
2018 对齐 2018同一模板 + 新索引答案跟 2018 版事实一致再看 query encoder 是否把问题推回老分布
版本错位2018 索引配 2016 问题，或反过来正确率明显下滑先怀疑知识源版本错了，再怀疑生成器顶住旧答案

索引热替换的最小检查顺序复制

# 1) rebuild the non-parametric memory from a new dump
new_doc_vectors = BERT_d(new_documents)
new_index = FAISS.build(new_doc_vectors)

# 2) ask the same question against different memories
old_answer = model.answer("Who is the President of Peru?", index=old_index)
new_answer = model.answer("Who is the President of Peru?", index=new_index)

# 3) compare outputs
print(old_answer)
print(new_answer)
print("changed:", old_answer != new_answer)

> 这段检查给你的判断 如果换索引后答案切到新事实，说明非参数记忆生效；如果答案不动，先查文档编码是否重算，再查索引是否真替换，最后再查 query encoder 和 generator 有没有继续偏向旧分布。做完这一步，你就知道“知识更新”到底是事实，还是口头描述。

梯度怎么回到检索器

gradient

把一条训练路径拆开看

先有查询向量 q(x)，再有 top-K 文档候选 z。generator 在每个候选上都算一次 p_\theta(y|x, z)，然后把这些概率按 p_\eta(z|x) 加权求和。最后得到的 p(y|x) 检索器和生成器共同决定的边际概率。

这意味着梯度通过总概率把信号分给被选中的候选。更像答案的文档会拿到更大的正向贡献，不相关的文档会被压下去。你读这条链时，重点看“谁在给候选加权”，而不是只盯着最后的 loss 数值。

如果正确文档没进 top-K

那它就进不了边际化，retriever 也收不到这篇文档带来的正向信号。RAG 的训练质量因此很依赖召回质量。候选集里没有证据，generator 再强也只能在错误集合里做最优组合，这也是论文为什么反复强调 top-K 近似和检索质量。

$$

p(y \mid x) \approx 0.7 \cdot 0.9 + 0.3 \cdot 0.2 = 0.69

$$

这里可以把 0.7 看成文档 A 的检索概率，0.9 看成它在该文档上的生成概率；0.3 和 0.2 则是文档 B 的对应值。A 的贡献更大，loss 下降时，query encoder 会更倾向把 A 类文档找上来。

消融怎么读

ablation

learned retrieval / frozen retriever / BM25

论文在 ablation 里冻结 retriever，再和 learned retrieval、BM25 互相对照。结论：在大多数任务上，训练出来的 retrieval 信号确实给了模型更好的任务对齐；但在像 FEVER 这种强实体词重叠任务上，BM25 也可能很强。

这说明了什么

如果检索器只会找“词看起来像”的文档，它不一定最适合答案生成；如果能学到任务相关的文档分布，它就能把生成器带到更有用的证据上。消融的价值，就是把这个差别拆出来。

对照控制了什么主要看什么读法
learned retrieval vs frozen retriever检索器是否继续学习任务信号能不能改进检索分布如果 learned retrieval 更好，说明 retriever 不只是“入口”，还是可学习部件
dense retriever vs BM25向量检索与词重叠检索任务更依赖语义相似还是词面重叠FEVER 这类强实体任务更容易让 BM25 占优
fixed index vs hot-swapped index知识源版本模型更新知识的方式换索引就能改世界知识，但前提是你真的把文档重新编码并重建索引

Q: 6. 解释 -log p(y|x) 为什么会把检索器也带着学。

参考答案 / 评分要点

。因为 p(y|x) 把 top-K 文档的生成概率按 p_\eta(z|x) 加权求和。某篇文档如果更能帮助生成正确答案，它在总概率里的权重就会上升；反过来，不相关文档会被压下去。于是，优化生成 loss 也会改变检索分布。

三、诊断与修法

关键词：固定索引、learned retrieval、BM25、索引版本、重建成本。 -->
7error diagnosis

Q: 7. 有人说“既然训练 retriever，就应该每一步都重建 document encoder 和索引”。错因、后果、修法分别是什么？

参考答案 / 评分要点

。错因：把可学习的 query side 和高成本的文档侧混成一件事。后果：索引频繁过期、训练变慢、系统维护成本高。修法：固定 document encoder 和索引，只更新 query encoder 与 generator；如果要换知识源，用热替换索引，而不是在训练循环里反复重建。

8error diagnosis

Q: 8. 为什么论文里 BM25 在某些任务上也能很强？这会推翻 learned retrieval 吗？

参考答案 / 评分要点

。不会推翻。BM25 在实体词重叠强、任务更依赖表面词匹配时可能表现很好，比如 FEVER。论文的意义是说明 learned retrieval 在大多数任务上能继续改进检索分布，而不是说 BM25 彻底没用。正确的读法是看任务偏好，而不是看谁“更新”。

9error diagnosis

Q: 9. 系统答对了一部分，但你发现它总在某些标题上“自己接话”。这说明你应该先查哪里？

参考答案 / 评分要点

。先查生成器是否过度依赖 parametric memory，再看检索分布是否把相关文档送上来了。这个题的重点先分辨“检索没找对”还是“找对了但生成器没有用好”。

10scenario-transfer

Q: 10. 如果把这套思路用到企业知识库问答，你会优先保留哪些中间记录？

参考答案 / 评分要点

。至少要保留：输入问题、查询向量或查询日志、检索到的 top-K 文档及其版本、生成答案、以及当次索引版本。这样出现错答时，才知道问题是 query 偏了、文档过期了，还是 generator 没用好证据。

12synthesis

Q: 12. 把“DPR、BART、MIPS、固定索引、negative marginal log-likelihood”串成一段完整解释。

参考答案 / 评分要点

。好的复述会说：DPR 把问题和文档分别编码，MIPS/FAISS 在固定索引里找 top-K 文档；BART 把 x + z 变成答案；训练时用 -log p(y|x) 作为目标，通过边际化把检索分布也带着学。固定 document encoder 和索引，是为了避免每次更新都重建大规模非参数记忆。
