DPR 检索器：两个编码器配一个索引

DPR

RAG 的检索器 pη(z|x) 基于 DPR。DPR 是 bi-encoder 架构：问题输入 x 经过 query encoder 得到 q(x)，文档 z 经过 document encoder 得到 d(z)。两个向量做内积，内积越大，文档越可能被检索出来。

bi-encoder 的好处是文档向量可以提前算好并放进索引。在线请求只需要算 query 向量，然后在索引中做近邻搜索。如果每次都把 query 和 2100 万文档一起编码，根本跑不动。代价是它不像 cross-encoder 那样能逐 token 精细比对 query 和文档。

pη(z|x) ∝ exp(d(z)T q(x)), d(z)=BERTd(z), q(x)=BERTq(x)

论文使用预训练 DPR 初始化检索器。这个 DPR 之前在 TriviaQA 和 Natural Questions 上学过检索包含答案的文档。这一点要记住：RAG 说“无直接检索监督”指的是下游 RAG 微调阶段没有证据标签，但检索器初始化本身用过 QA 检索监督。

MIPS 与索引：为什么要近似搜索

MIPS

MIPS 是 Maximum Inner Product Search，也就是在大量文档向量里找和 query 向量内积最大的 K 个。论文的 Wikipedia 被切成 2100 万个 100 词片段。如果每次请求都暴力计算所有内积，延迟和成本都会很高。

所以论文用 FAISS 构建索引，并使用 HNSW 近似加速检索。近似搜索的目标不是数学上百分百精确地找全局 top-K，而是在可接受的速度和内存下找到足够好的候选。RAG 的训练和推理都依赖这个检索步骤。

附录提到，MIPS 用 CPU 已经足够快。原始版本索引向量放在 CPU 上大约需要 100GB 内存；后续用 FAISS 压缩工具可以降到 36GB。这个细节说明 RAG 不是只有模型参数成本，外部记忆的存储和检索成本也必须算。

最小例子：内积检索结构Python
import numpy as np

query = np.array([0.8, 0.1, 0.3])
docs = np.array([
    [0.7, 0.2, 0.2],
    [0.1, 0.9, 0.1],
    [0.8, 0.0, 0.4],
])

scores = docs @ query
top_ids = np.argsort(scores)[::-1][:2]
print(top_ids, scores[top_ids])
判断句：只要文档规模大到不能全量读进生成器，就需要索引和近似检索。

BART 生成器：参数记忆负责表达和整合

BART

论文用 BART-large 作为生成器 pθ(yi|x,z,y1:i-1)。BART 是 encoder-decoder 模型，预训练目标是去噪。它适合 seq2seq 任务，能读输入和检索文档，然后生成答案、问题或分类标签。

RAG 把输入 x 和检索文档 z 简单拼接后送入 BART。这个做法很直接，但含义重要：生成器不是只看检索文档，也不是只看原始输入，而是同时看 query 和外部证据。它的参数 θ 就是论文所谓 parametric memory。

BART 的优势在于语言生成和知识整合。比如文档提供线索但没有原文答案时，生成器可以综合线索生成答案；这也是 RAG 在开放域 QA 中能超过纯抽取式方法的原因之一。抽取模型必须在文档里找 span，而 RAG 可以生成文档中没有原样出现的答案。

训练目标：没有证据标签也能更新检索器

TRAINING

训练时，论文给定下游任务的输入输出对 (xj, yj)，最小化目标答案的 negative marginal log-likelihood。直觉是：如果某些文档让生成器更容易产生正确 y，这些文档路径的概率就会得到更好的训练信号。

这里的“端到端”要精确理解。论文联合训练 query encoder 和 BART generator，但不更新 document encoder，也不实时重建文档索引。原因很现实：如果 document encoder 变化，所有 Wikipedia 文档向量都要周期性重算并更新索引，代价很高。

固定 document encoder 带来一个稳定索引，query encoder 学会把输入映射到这个固定文档空间中更有用的位置。生成器则学会利用检索文档生成目标输出。这个设计是性能、成本和工程可行性的折中。

Loss = Σj -log p(yj|xj)

规模细节：这不是玩具检索

SCALE

论文使用 2018 年 12 月的 Wikipedia dump，切成互不重叠的 100 词块，总共 2100 万个文档。每个文档编码成 728 维向量，索引不是 trainable parameters，但它是系统的一大块外部记忆。

附录 G 说，RAG 有 BERT-base query encoder、BERT-base document encoder 和 BART-large。其中 document encoder 不训练，BART-large 约 406M 参数。论文给出的 trainable parameters 总量约 626M，明显少于 T5-11B，但在开放域 QA 上表现很强。

这个对比的意义不是“参数越少越好”，而是说明外部记忆可以替代一部分把事实硬压进模型参数的需求。模型不必把所有事实都背进权重；它可以学会访问、读取和利用外部文本。

组件和训练常见误区

PITFALLS

误区 1：DPR 等于普通 embedding API

DPR 是为检索训练的 bi-encoder。一般 embedding 模型也能做向量检索，但训练目标和数据可能不同。

误区 2：端到端就是全部参数都更新

论文固定 document encoder 和索引，只更新 query encoder 与 generator。端到端指最终任务损失能影响检索和生成路径。

误区 3：索引不是参数就没有成本

索引会占内存、磁盘和检索延迟，还要维护版本。非参数不等于免费。

误区 4：BART 只负责复述检索文本

BART 负责生成，可以整合线索、改写答案，也可能在证据不足时犯错。
