RAG-Token 为什么像普通自回归模型

TOKEN DECODING

RAG-Token 在每个 token 步骤先对文档求和，得到一个边际后的 next-token distribution。这个分布长得像普通 seq2seq 模型的下一词概率，所以可以直接放进标准 beam decoder。你可以把文档混合看成发生在模型内部的一步，解码器看到的是一个已经混好的 token 分布。

论文把这个转移概率写成 p′θ(yi|x,y1:i-1)=Σz pη(z|x)pθ(yi|x,z,y1:i-1)。重点是，它仍然按 token 自回归展开。每一步都有明确的下一 token 概率，因此 beam search 可以正常扩展候选。

p′θ(yi|x,y1:i-1) = Σz ∈ top-K pη(z|x) · pθ(yi|x,z,y1:i-1)
判断句：只要模型能在每一步给出单一的 next-token distribution，就能接普通自回归解码。

RAG-Sequence 为什么解码更麻烦

SEQUENCE DECODING

RAG-Sequence 的概率不是一个普通的逐 token 转移概率。它先让每个文档 z 生成完整序列，再把完整序列概率加权求和。于是你不能只跑一次 beam search，因为不同文档下可能产生不同候选序列，而最终分数需要把同一个候选在所有文档下的概率加起来。

论文提出 Thorough Decoding：对每个文档各跑 beam search，收集候选集合 Y。对于某个候选 y，如果它没有出现在某篇文档的 beam 里，还要额外对这篇文档跑 forward pass，估计 pθ(y|x,z)，再乘 pη(z|x) 并求和。这个做法更完整，但当输出长、候选多时很慢。

Fast Decoding 则进一步近似：如果候选 y 没有在某篇文档的 beam search 中出现，就把 pθ(y|x,zi) 近似为 0。这样省掉额外 forward pass。论文在 Open-MS MARCO 和 Jeopardy 生成任务上用 Fast Decoding，因为 Thorough Decoding 没有带来收益。

伪代码：RAG-Sequence Thorough Decodingpseudo
Y = set()
for z in top_k_docs:
    Y.update(beam_search(generator, x, z))

for y in Y:
    score = 0
    for z in top_k_docs:
        score += p_eta(z, x) * forward_prob(generator, y, x, z)
choose_best(Y, score)

开放域 QA：把问题答案当文本对

QA

开放域 QA 是论文最直接的任务。输入 x 是问题，输出 y 是答案文本。RAG 用答案的负边际 log-likelihood 训练，不需要抽取答案 span，也不需要手工标注哪篇文档是证据。

论文比较了四个数据集：Natural Questions、TriviaQA、WebQuestions、CuratedTrec。指标是 Exact Match。这个指标很严格，要求生成答案和标准答案匹配。对于开放域 QA，RAG-Sequence 和 RAG-Token 都比很多 closed-book 和 open-book baseline 强。

附录还有一些细节：QA 测试时，RAG-Token 报告使用 15 个检索文档，RAG-Sequence 报告使用 50 个检索文档；QA 中使用 greedy decoding，因为 beam search 没有改善结果。这些细节说明解码和 K 值不是固定教条，要看任务和开发集表现。

输入输出

x = question，y = answer。

评估

Exact Match，看答案字符串是否命中。

生成任务：MS-MARCO 与 Jeopardy

GENERATION

MS-MARCO NLG 用问题生成完整句子答案。论文刻意不使用数据集中给的 gold passages，而是把它当开放域生成任务，只用问题和答案训练。这样更能测试 RAG 能不能自己从 Wikipedia 里找到有用知识。

Jeopardy question generation 更能体现 RAG-Token 的灵活性。输入是答案实体，输出是一条 Jeopardy 风格问题。这样的输出常常需要两个事实线索。论文用 Q-BLEU-1 和人评检查事实性、具体性，结果显示 RAG 生成更具体、更事实。

自动指标只能看一部分。BLEU、Rouge、Q-BLEU 都无法完全判断生成内容是否真实，所以论文加入人类比较评估。评估者看同一个输入下 BART 和 RAG 的生成结果，判断哪一个更 factual 或更 specific。这是生成任务里很重要的一层证据。

任务输入输出指标
MS-MARCO NLG问题完整句子答案BLEU-1、Rouge-L
Jeopardy QGen答案实体问题线索BLEU-1、Q-BLEU-1、人评

FEVER：分类也能写成生成

FEVER

FEVER 要判断一个 claim 是 supported、refuted，还是 not enough information。论文把这些类别映射成单个输出 token，用 seq2seq 的方式训练。因为输出长度为 1，RAG-Sequence 和 RAG-Token 在这个任务上等价。

FEVER 的难点不只是分类，还包括从 Wikipedia 检索相关证据并做蕴含判断。论文没有使用检索证据监督，这和很多复杂 pipeline 不同。结果虽然没有超过最强 FEVER 专用系统，但已经接近使用强检索监督的 pipeline。

附录 E 说明，FEVER 还有证据句抽取子任务，但论文主要做 claim classification。原因之一是 Wikipedia dump 不同，直接做证据抽取并不简单。这是读论文时要注意的边界：不要把它的 FEVER 结果误读成完整 FEVER pipeline 全部子任务都解决了。

判断句：如果类别可以写成短文本标签，seq2seq 模型就能把分类当生成处理。

解码与任务适配常见误区

PITFALLS

误区 1：训练公式等于解码算法

训练最大化目标似然，测试要近似找最高概率输出。两者相关，但不是同一步。

误区 2：RAG-Sequence 能直接单次 beam search

它的文档边际化在完整序列层面，需要对候选序列跨文档重新计分。

误区 3：所有任务都用同一 K 值

论文在不同任务和不同 RAG 形式下使用不同 K，开发集和成本都影响选择。

误区 4：自动指标能证明事实性

BLEU/Rouge 看文本重合，事实性仍需要证据检查或人评。
