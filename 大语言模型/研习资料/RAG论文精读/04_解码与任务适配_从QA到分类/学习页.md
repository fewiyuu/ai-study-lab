# 先看主线 {#overview}
<!-- tag: overview -->

**1**RAG-Token 先把每个 token 当成标准自回归步来解码。
**2**RAG-Sequence 先按文档分别 beam search，再把候选跨文档重计分。
**3**QA 用短答案和 EM，Open MS-MARCO 和 Jeopardy 用生成指标与人评。
**4**FEVER 把分类标签写成单 token，因此两种模型在这个任务上等价。
**5**论文的解码选择和任务指标一起决定了结果怎么读；结果表本身不会替你判断层次。

> **先记一个直觉** 当输出本身是一个短标签或短答案时，解码近似会尽量贴近标准 beam search；当输出要跨多个文档拼接内容时，候选集合和重计分就比单次搜索更重要。你先把这条线记住，后面的任务差异就不会散。

# 术语和边界 {#terms}
<!-- tag: terms -->

Thorough Decoding
RAG-Sequence 的严格近似。先对每篇文档分别做 beam search，再把没进所有 beam 的候选补回概率里，重新算文档边际。


Fast Decoding
RAG-Sequence 的更快近似。它把“没从某篇文档 beam 出来的候选”直接当作近似 0，少做额外 forward pass。


任务标签当序列
FEVER 的 supports、refutes、not enough information 都可以映射成一个输出 token，所以分类也能走生成模型这条路。

术语
在论文里负责什么
读它时先看什么

候选序列搜索

marginal probability
跨文档重计分
候选是不是来自多个文档的合并结果


label accuracy
FEVER 的分类指标
输出是否已被压成单 token 标签

# RAG-Token 怎么解码 {#token}
<!-- tag: token -->
<!-- layout: grid2 -->

## box
### 它为什么像标准自回归模型
RAG-Token 在测试时的形式和普通 seq2seq 很接近：给定前文 token，下一步就是选下一个 token。论文说这时可以把 `p_θ(y_i|x, z, y_它更适合长一点、需要在不同文档间切换证据的生成任务。Jeopardy 就是论文里最典型的例子：问题里可能要拼两个事实，token 级切换能让模型在不同文档之间来回取信息。

## box soft
### 第二步看什么
别先盯着 beam size。先看每个 token 的候选是怎么被文档后验加权的，再看某个 token 更像哪篇文档的证据。论文图 2 的意思就在这里：生成 “Sun” 时，一篇文档更强；生成 “A Farewell to Arms” 时，另一篇文档更强。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

RAG-Token 的最小解码读写关系复制

```python
# standard autoregressive decoding is enough here
for i in range(1, len(y) + 1):
 beam_state = beam.step(previous_tokens=y[:i-1], context=x)
 doc_posterior = retriever.posterior(x, y[:i-1])
 y_i = decoder.pick_next_token(beam_state, doc_posterior)
```

> **看这段时抓三件事** 先看它是不是标准按 token 逐步展开；再看文档后验是怎么参与当前步的；最后看 beam state 是否继续沿着前面的候选推进。RAG-Token 的重点让每一步的文档选择更细。

# RAG-Sequence 为什么更麻烦 {#sequence}
<!-- tag: sequence -->
<!-- layout: grid2 -->

## box
### 不能只跑一次 beam search 的原因
RAG-Sequence 的边际概率并不是每个 token 的普通独立形式，所以不能直接把标准 beam decoder 跑一遍就结束。论文的做法是：先对每个文档单独 beam search，得到一组候选，再对这些候选做跨文档的概率重计分。
这个过程就是论文说的 Thorough Decoding。它更完整，但也更慢，因为候选集一旦变大，就要做更多 forward pass。Fast Decoding 则把没进 beam 的候选近似成 0，少算很多次。

## box warn
### 什么时候该担心速度
只要输出序列变长，或者每篇文档的 beam 候选一多，Thorough Decoding 的代价就会上来。论文自己说，对于生成任务，Fast Decoding 更实用；而 QA 的答案短，Thorough Decoding 的额外代价不一定值得。

 这里的 `y` 是完整序列，`z` 是候选文档。RAG-Sequence 的麻烦点就在于：你要对整段 `y` 在多个文档之间重新算概率，而不是只看单个 token 的局部选择。

# 开放域 QA {#qa}
<!-- tag: qa -->
<!-- layout: grid2 -->

## box
### 为什么 QA 适合先看 EM
论文在 NQ、WebQuestions、CuratedTrec 和 TriviaQA 这类开放域 QA 上报告 Exact Match。原因很直接：这些任务的答案通常短，先看字符串是否命中，比先看长句的 n-gram 质量更贴近任务本身。
Appendix A 里还给了实现细节：RAG-Sequence 用 50 个检索文档，QA 里用 Thorough Decoding，但 greedy decoding 已经足够，beam search 没带来更好的结果。这个信息很重要，因为它说明 QA 的输出不像长生成那样依赖复杂解码近似。

## box info
### 读 QA 结果时先看什么
先看答案是否短、是否唯一，再看 EM 能不能稳定反映对错。QA 页里如果答案看起来很短，却去强调 Rouge，就会把问题读歪。这里最关键的是“检索有没有把正确证据送上来”，同时看答案是否命中。

# 生成任务 {#generation}
<!-- tag: generation -->
<!-- layout: grid2 -->

## box
### MS-MARCO 和 Jeopardy 为什么不能一把尺子量到底
Open MS-MARCO 关注的是抽象回答和信息重写，论文报告 BLEU-1 和 Rouge-L。Jeopardy question generation 更看事实性和具体性，除了 BLEU-1 之外，论文还引入 Q-BLEU-1 和两种人评。关键差别在任务里“好答案”的定义不同。
论文还观察到：RAG-Sequence 在 MS-MARCO 上比 BART 好，而 RAG-Token 在 Jeopardy 上更占优。一个更像完整答案重写，一个更像从多篇文档里拼出精确事实，所以下一步不能只看总分。

## box soft
### 人评在这里补什么
人评主要看 factuality 和 specificity。自动指标能看词面重合，但很难判断“事实是否站得住”。Jeopardy 的句子往往很短，却可能因为一个事实错位而完全错答案，所以人评是必要的补充。

# FEVER 分类 {#classification}
<!-- tag: classification -->
<!-- layout: grid2 -->

## box
### 分类为什么也能写成生成
FEVER 要判断一个 claim 是 supported、refuted，还是 not enough information。论文把这三个类别映射成单个输出 token，然后像生成任务一样直接训练。因为输出长度只有 1，RAG-Sequence 和 RAG-Token 在这里实际上等价。
Appendix E 还补了一句：论文用的是 claim classification，主文没有展开完整的证据句抽取 pipeline。这里的意义在于说明 RAG 可以不用检索证据监督，也能做分类式任务。

## box warn
### 读 FEVER 结果时最容易错的地方
容易把“分类任务”误看成“证据抽取任务已经一并解决”。实际上论文在这部分只做了标签预测，证据句抽取只是附带背景。读表 2 时要先确认：报告的是 label accuracy，而不是完整证据管线的最终质量。

$$
 \text{supports} \rightarrow t_1,\quad
 \text{refutes} \rightarrow t_2,\quad
 \text{not enough info} \rightarrow t_3
 $$
 这里的 `t_1, t_2, t_3` 是单 token 标签。因为输出长度被压成 1，RAG-Sequence 和 RAG-Token 在这一页的实际差别就被抹平了。

# 常见误区 {#pitfalls}
<!-- tag: pitfalls -->
<!-- layout: grid2 -->

## box redsoft
### 误区 1：把所有任务都当生成
短答案、长生成和分类标签需要的指标不一样。硬把它们塞进同一套读法，会把结论读偏。

## box redsoft
### 误区 2：把指标上涨当任务变好
BLEU-1 变高，不代表事实性真的提升；label accuracy 变好，也不等于证据句抽取已经解决。

## box redsoft
### 误区 3：把解码差异当训练差异
训练里的边际化和测试时的近似搜索属于不同层。看论文结果时要先分层，再下结论。

## box redsoft
### 误区 4：把 FEVER 只读成抽取任务
本页只讲标签预测层，证据抽取是另一层。混在一起，会把论文实际贡献看小或看大。

# 一条完整例子 {#worked-case}
<!-- tag: case -->
<!-- layout: grid2 -->

## box
### 用 Hemingway 的 Jeopardy 例子走一遍
论文图 2 用 “Hemingway” 作为输入，展示了 RAG-Token 的文档后验如何在生成过程中切换。生成 “Sun” 的时候，一篇文档更像正确证据；生成 “A Farewell to Arms” 的时候，另一篇文档更像正确证据。这个例子说明的token 级后验会跟着当前生成位置改变。
如果把它换成 RAG-Sequence，候选会先按每篇文档分别搜，再对整段生成结果重新计分。于是，哪怕两个候选看起来都能说对部分事实，最后被保留下来的序列也可能不同。这个差别就是解码和任务适配真正落地的地方。

## box info
### 手算时看哪三步
1. 先看当前 token 对哪篇文档的后验更高。
2. 再看这个 token 是否改变了后续候选的 beam 状态。
3. 最后把整个序列放回任务指标里，判断它是在做事实重写，还是在拼接事实。

步骤
你看到什么
接下来查什么


生成当前 token
某篇文档后验更高
这一篇文档是否真的解释了当前 token


更新 beam
候选序列继续展开
后续 token 是否开始切到别的文档


回到任务指标
完整输出已形成
它更像事实性问题，还是更像长句答案

把这个例子塞进解码判断复制

```python
tokens = ["Hemingway", "...", "Sun", "...", "A Farewell to Arms"]
for token in tokens:
 doc_post = posterior_docs(x, prefix=tokens_before(token))
 active_doc = max(doc_post, key=doc_post.get)
 beam.update(token, active_doc)
```

> **如果这里做错，最先坏在哪** 如果你把“当前 token 看到哪篇文档”读错，后面的 beam 会继续在错误文档上展开；如果你把任务指标也看错，就会把一个事实拼接问题误判成普通句子生成。这个例子就是为了让你在纸上先抓到这两个分叉。

# 任务透镜 {#task-lens}
<!-- tag: lens -->
<!-- layout: grid2 -->

## box
### 同一条输出，在三个任务里怎么看
同样是一段带检索的输出，放到 QA 里，你先关心答案能不能对上；放到 Jeopardy 里，你先关心问题是不是具体、准确；放到 FEVER 里，你先关心标签对不对。这个透镜很重要，因为它说明“解码方法相同”并不代表“任务读法相同”。
也可以反过来想：如果你只盯着 BLEU-1，Jeopardy 的任务会被简化成词面重合；如果你只盯着 label accuracy，FEVER 的证据约束会被你丢掉；如果你只盯着 EM，生成任务里的事实丰富度也会被压扁。任务格式决定指标，而指标又决定你怎么看解码。
具体到操作时，你可以拿同一条输出分别用 EM、Q-BLEU-1 和 label accuracy 读一遍，再看结论是否冲突；如果冲突，先查任务格式，再查指标顺序，最后才查模型本身。

## box warn
### 失败时最先查哪里
先查任务透镜，再查模型。很多时候模型没坏，坏的是你给它选的读法。这个单元最实用的动作，就是把输出先归类成短答案、长生成、还是单 token 标签，再决定下一步看什么。

### 1 choice

**Q:** 1. 哪个说法最准确？

- [ ] RAG-Token 可以像标准 beam decoder 那样按 token 逐步解码。
- [ ] RAG-Sequence 可以直接用一次标准 beam search 完成全部解码。
- [ ] 两种模型在测试时完全不需要候选搜索。

Answer: RAG-Token 可以像标准 beam decoder 那样按 token 逐步解码。
Explain: RAG-Token 的测试时形式最接近标准自回归 beam decoder；RAG-Sequence 需要按文档分别搜候选再重计分。

### 2 true/false

**Q:** 2. Thorough Decoding 的意思是“把候选按文档分开搜完以后，再把没出现在 beam 里的候选补回概率”。

- [ ] 对
- [ ] 错


### 3 fill

**Q:** 3. 用自己的话解释：Fast Decoding 为什么会更快？

Explain: 好答案会说：Fast Decoding 直接把没从某篇文档 beam 出来的候选近似为 0，所以不用额外对这些候选再跑一遍 forward pass；它更快，但代价是少算了部分跨文档候选的概率。它适合输出较长、候选很多、而且我们更关心速度的场景。

### 4 ordering

**Q:** 4. 按任务把输出和指标配起来：QA、MS-MARCO、Jeopardy、FEVER。

Explain: QA 通常看短答案，重点是 EM；Open MS-MARCO 是完整回答，常看 BLEU-1 和 Rouge-L；Jeopardy 是事实性问题生成，除了 BLEU-1 还看 Q-BLEU-1 和人评；FEVER 是标签分类，最后看 label accuracy。顺序可以变，但任务和指标不能互换。

### 5 matching

**Q:** 5. 把任务和评估方式连起来：哪个更适合 EM，哪个更适合人评？

- [ ] QA 更适合 EM
- [ ] Jeopardy 更需要人评
- [ ] FEVER 更适合 label accuracy
- [ ] 生成任务不需要 BLEU 或 Rouge

Explain: 前 3 项是对的，最后一项是错的。生成任务当然会看 BLEU / Rouge，只是不能只看这两个指标。这个题的重点是区分“短答案命中”“长句生成质量”和“分类标签准确率”。

### 6 synthesis

**Q:** 6. 为什么 Jeopardy 只看 BLEU-1 不够，还要看 Q-BLEU-1 和人评？

Explain: Jeopardy 的生成重点是能不能写出准确、具体的事实性问题。BLEU-1 只看表层词重合，Q-BLEU-1 会更强调实体匹配，而人评能进一步判断 factuality 和 specificity。三者合起来才比较接近任务真正关心的质量。

### 7 diagnostic

**Q:** 7. 有人说“既然 RAG-Sequence 训练时是边际化，那测试时直接一次 beam search 就够了”。错因、后果、修法分别是什么？

Explain: 错因：把训练目标和测试时的近似搜索混成一回事。后果：会漏掉 RAG-Sequence 需要按文档重计分这一步，得到过于乐观或过于粗糙的解读。修法：先分清训练是边际化，测试是近似 arg max，再看 Thorough / Fast 分别补了什么。

### 8 diagnostic

**Q:** 8. 为什么 FEVER 的结果不能被直接读成“证据句抽取管线已经做完”？

Explain: 论文在主文里做的是 claim classification，把标签写成单 token，主要评的是 label accuracy；Appendix E 还说明证据句抽取不是本页主任务。这个结果说明 RAG 可以把分类也写成生成，但没有覆盖完整 FEVER 抽取管线。

### 9 diagnostic

**Q:** 9. 系统在 Jeopardy 上的 BLEU-1 变高，但人评没怎么变。你先怀疑哪里？

Explain: 先怀疑生成器更会迎合词面重合。BLEU-1 只能说明 n-gram 更像参考答案，不足以证明问题更具体或更准确。这个题的核心是区分“指标上涨”和“任务真正变好”。

### 10 scenario-transfer

**Q:** 10. 如果你要给一个新的 RAG 任务选解码和指标，先按什么顺序做判断？

Explain: 先看任务输出是短标签、短答案还是长生成；再看答案是否经常需要跨文档拼接事实；最后按任务选择指标：短答案常看 EM，长生成看 BLEU / Rouge，事实性问题再加人评，分类任务则看 label accuracy。这样先把任务本体和评估本体分开，解码选择才不会乱。

### 11 fill

**Q:** 11. 用 2-3 句话概括本页在 RAG 论文里的作用。

Explain: 好答案会说：这一页把 RAG 的测试时解码差异和任务适配写清楚，说明了 RAG-Token 和 RAG-Sequence 为什么要用不同的近似；它还把 QA、生成和分类分别对应到不同的输出格式和指标，帮你读懂论文结果而不混淆层次。只要能把这两层讲清，句子长短可以自由调整。

### 12 synthesis

**Q:** 12. 把“RAG-Token、RAG-Sequence、Thorough Decoding、Fast Decoding、EM、BLEU-1、Rouge-L、Q-BLEU-1、label accuracy、FEVER”串成一段完整解释。

Explain: 好的复述会说：RAG-Token 在测试时能像标准自回归模型那样按 token 解码，所以可以直接用 beam decoder；RAG-Sequence 因为要在整段输出上对多个文档重计分，所以需要 Thorough Decoding 或 Fast Decoding 这样的近似。不同任务的输出格式不同，QA 更常看 EM，MS-MARCO 看 BLEU-1 和 Rouge-L，Jeopardy 还要看 Q-BLEU-1 和人评，而 FEVER 把分类标签写成单 token 后就看 label accuracy。这样一来，解码策略和指标都和任务本体对上了。
