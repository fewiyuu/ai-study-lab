术语和指标

terms

ablation

把一个组件拿掉或替换掉，看结果怎么变。这里主要看冻结检索器、换 BM25、改 K 值。

human evaluation

人评主要补 factuality 和 specificity。它不是在重复 BLEU / Rouge，而是在看自动指标看不到的部分。

label accuracy

FEVER 这里把分类标签写成单 token，所以最后看的是标签是否判对，而不是长句生成质量。

表 1：开放域 QA 的核心结论

table 1

这张表先告诉你什么

Table 1 里，RAG-Sequence 在 Natural Questions、WebQuestions、CuratedTrec 和 TriviaQA 这些开放域 QA 任务上都站在很靠前的位置。NQ 上它的 EM 高于 DPR 和 closed-book T5-11B，这支持了论文的主张：检索增强生成可以在知识密集型 QA 上超过只靠参数记忆的模型。

但读表时要注意比较对象。RAG 的 retriever 初始化来自 DPR，而 DPR 本身用过检索监督，所以这不是“完全从零学检索”的证据。更稳的说法是：RAG 在已有检索初始化上，能通过最终答案训练把检索和生成接起来。

读 QA 结果时先抓什么

先抓答案长度，再抓指标。开放域 QA 的答案通常短，EM 比 Rouge 更贴任务本体。这里的关键不是“分数越多越强”，而是“短答案命中率是否提高，同时证据是否真的被送进候选”。

表 2-4：生成质量不只看分数

table 2-4

MS-MARCO 和 Jeopardy 不能用同一把尺子

Open MS-MARCO 关注完整回答，论文看 BLEU-1 和 Rouge-L。Jeopardy question generation 更看事实性和具体性，所以除了 BLEU-1，论文还看 Q-BLEU-1 和人评。这里的关键不是指标越多越好，而是任务本身对“好答案”的定义不同。

结果也对应这个差异：RAG-Sequence 在 MS-MARCO 上比 BART 更强，RAG-Token 在 Jeopardy 上更占优。一个更像完整回答重写，一个更像从多篇文档里拼出精确事实，所以不能拿同一条结论硬套。

人评补什么

人评主要看 factuality 和 specificity。自动指标能抓词面重合，但抓不住“事实是否站得住”这件事。Jeopardy 的答案往往很短，一个事实错位就会把整句带偏，所以人评不是装饰，而是必要补充。

表 6：Frozen 和 BM25 消融告诉我们什么

ablation

Frozen retriever 在测什么

Frozen retriever 是把检索器冻住，不让它跟下游任务一起学。结果里，训练检索器的 RAG 通常更好，说明 learned retrieval 真的有用。检索不是一层固定前处理，而是任务模型的一部分。

BM25 消融则在问另一件事：dense retriever 是不是总会赢。答案不是。开放域 QA 里 dense 更强，但 FEVER 上 BM25 很有竞争力，因为它更吃实体词重合。这个结果最该提醒你的，是检索方式要跟任务文本形态对齐。

先别把消融读成胜负表

消融不是为了宣布“谁永远最强”，而是为了判断组件是否有用、在什么任务上有用、为什么有用。读的时候要把变化、任务和解释连起来，不要只盯着最高分。

消融改变了什么说明什么
Frozen检索器不随任务更新任务适配后的检索通常更好。
BM25用词重合检索替代 dense retrieverdense 不总是赢，实体任务词法检索很强。
K 值变化测试时取更多文档更多证据可能提升，也可能带来噪声和成本。

Worked example：读消融不要只背“谁赢了”

观察能支持的结论不能外推的结论
Learned retriever 通常优于 Frozen下游任务损失能让 query encoder 更适配任务。所有组件都必须一起训练。
BM25 在 FEVER 上很强实体词重合明显时，词法检索可能优于 dense 检索。BM25 总是优于 DPR。
K 变大有时提升、有时加噪声K 是任务相关超参数，需要开发集和错误分析。top-K 越大一定越好。

图 3：检索更多文档不总是简单变好

figure 3

Figure 3 在看什么

Figure 3 看测试时检索文档数量 K 的影响。NQ 上，RAG-Sequence 随 K 增大单调提升；RAG-Token 在 10 个文档附近达到峰值。MS-MARCO 上，RAG-Token 检索更多文档会提高 Rouge-L，但可能牺牲 BLEU-1；RAG-Sequence 影响小一些。

这说明 K 是任务和模型相关的超参数。更多文档会提高召回，也会带来更多噪声、更长上下文和更高计算成本。实践里，K 不是越大越好，而是要跟任务和错误模式一起调。

该怎么调

如果候选里经常没有证据，先考虑增大 K 或改检索；如果候选里已经有证据但生成开始发散，继续增大 K 可能只会加噪声。你要调的是“证据够不够”和“噪声会不会淹没答案”，不是一个抽象数字。

增大 K 的好处

提高正确证据进入候选的机会。

增大 K 的代价

更多计算、更多噪声、边际化更复杂。

选择方式

看 dev set、任务指标和错误分析。

一个完整案例

worked case

把一张表读成四句话

假设你现在只看 Table 6，看到 learned retriever 比 Frozen 好、BM25 在 FEVER 上又很强、K 变大有时能涨分有时会掉分。很多人会立刻写“RAG 的检索器很强，BM25 也不错，K 越大越好”。这三句里，只有第一句接近原意，后两句都太满。

更稳的写法是：Table 6 先说明 learned retrieval 真的有用；再说明词法检索在实体重合明显的任务上仍然能打；最后说明 K 不是单调增益的旋钮，而是要和任务、噪声和成本一起看。这样写出来的不是胜负口号，而是可执行的判断。

把这三句再往前挪一步，你就能知道下一步该查什么。若是 learned retrieval 收益明显，后面该看 query encoder 是否过拟合；若是 BM25 在 FEVER 上强，后面该看任务是不是实体中心；若是 K 变大开始掉分，后面该看候选噪声和上下文长度。结果页真正帮你的，不是背结论，而是把下一步检查点点出来。

先写观察，再写解释，最后写下一步。不要把三件事混成一句“所以 RAG 很强”。如果你想把这一页真正学会，最有效的做法就是从 Table 1 或 Table 6 里挑一行，按下面这张小表重写一遍。

观察更稳的解释下一步该查什么

把结果读成动作python
def read_result(row):
    task = row.task
    metric = row.metric
    baseline = row.baseline
    claim = summarize(task, metric, baseline)
    next_check = decide_next_check(task, baseline)
    return claim, next_check
为什么这一步重要 你在结果页最常犯的错，不是记不住数字，而是把数字当成结论。真正稳的读法，是把每一行数值翻译成一个可检查动作：去看任务、去看任务形态、去看噪声来源、去看下一轮实验该怎么设计。这样读表，才不会把论文结论说得过满。
如果你写不出“下一步该查什么”，就说明你还没把结果读成判断。把一句“RAG 在 FEVER 上很好”改成“FEVER 上 BM25 很强，说明实体词重合任务不能只迷信 dense retriever；下一步要查任务是否本来就偏实体中心”，这才算真正读到了消融。
读错这一步的后果很具体：你会把 Table 6 当成胜负榜，进而在真实系统里误删词法检索、误把 K 当常数、误把分类结果当证据抽取完成。把这段改正的方法也很具体：任选一张结果表，按“观察 / 解释 / 下一步”重写三行，再拿练习 10、12 去对照你写的边界。

先别追

boundary

这页暂时不展开的东西

先别追这四个方向：检索器的预训练细节、后续 RAG 变种、证据句抽取的完整 pipeline、以及更晚出现的评估框架。它们都重要，但不属于这一页的主线。你先把“结果怎么读”“消融怎么读”“边界怎么收”讲顺，再去追这些分支，阅读效率会高很多。

如果你现在就往这些方向钻，最容易发生的事不是学得更快，而是把本页的结论和后续方法混成一团。结果页里最值钱的不是某个单独分数，而是它帮你判断了哪些说法值得信，哪些说法还要再查。

读到这里要做的最后一件事

把“表 1 说明什么、表 2 / 4 说明什么、表 6 说明什么、Figure 3 说明什么”各写一句，然后在每句后面补一个不该过度外推的提醒。这样你就不是在背结论，而是在练结论的边界。

$$ \Delta score = score_{model} - score_{baseline} $$

这里的 Δscore 只表示相对差距，不表示因果已经完全成立。要把差距解释成“方法真的更好”，还要回到任务、指标、比较对象和边界。

QA 通常看 EM；MS-MARCO 看 BLEU-1 和 Rouge-L；Jeopardy 还要看 Q-BLEU-1 和人评；FEVER 是标签分类，最后看 label accuracy。顺序可以变，但任务和指标不能乱配。

4. 有人看到 BM25 在 FEVER 上很强，就说“dense retriever 没必要”。错因、后果、修法分别是什么？

参考答案 / 评分要点

错因：把任务形态忽略了。FEVER 是实体词重合很强的分类任务，BM25 本来就占优势。后果：把任务特征误读成方法优劣，进而对所有任务做错误外推。修法：先看任务文本形态，再看比较对象，最后才下方法结论。

diagnosticK 值

5. 如果模型在某个任务上 K 取大后更差，你会先怀疑什么？

参考答案 / 评分要点

先怀疑噪声和上下文长度，而不是默认“模型坏了”。K 变大会提高召回，也会增加噪声和计算成本；如果候选已经够用，再加 K 可能反而冲掉生成质量。正确做法是结合 dev set 和错误分析调，而不是信奉 top-K 越大越好。

6. 用自己的话解释：Frozen retriever 在这里到底检验什么？

答题时至少说到“冻结了什么”“和 learned retrieval 的差别”“为什么它能帮你判断检索是否真的被任务学到了”。

参考答案 / 评分要点

Frozen retriever 把检索器固定住，不让它随下游任务更新。这样就能对照 learned retrieval，看“让 query encoder 跟任务一起学”到底有没有收益。它检验的是检索是不是任务模型的一部分，而不是固定前处理。
