Index hot-swapping：更新知识不一定要重训

HOT-SWAP

RAG 的一个重要优势是外部记忆可以替换。论文用 2016 年和 2018 年的 Wikipedia 索引做演示：准备 82 个在两个年份间发生变化的世界领导人职位问题，例如“Who is the President of Peru?”，然后用不同索引回答。

结果很清楚：2016 索引对 2016 领导人回答正确率 70%，2018 索引对 2018 领导人正确率 68%；如果索引和年份错配，准确率很低。这个实验说明模型的回答确实受外部索引版本影响。换索引可以改变世界知识，而不需要重新训练 BART 参数。

但这个结论也有边界。热替换能更新外部事实来源，不等于所有任务行为都会自动变好。如果 query encoder 不会把新问题检索到正确文档，或者生成器忽略文档，换索引也救不了。热替换证明的是非参数记忆的可编辑性，不是完整系统自动正确。

证明了什么

外部索引版本会影响 RAG 的事实回答，知识可以通过替换索引更新。

没证明什么

没证明所有事实都能自动更新，也没证明检索和生成永远忠于新索引。

Null document：资料缺失时该怎么处理

NULL DOC

附录 F 讨论了 null document。直觉很合理：有些输入可能检索不到有用文档，那就额外加入一个空文档，让模型可以选择“没有资料可用”。形式上，如果检索 k 个文档，就再加一个 empty document，对 k+1 个结果边际化。

论文尝试了三种 null document logit：学习一个空文档 embedding、学习一个静态 bias、用神经网络预测 logit。但这些方法没有提升表现，所以主模型省略。对 Open MS-MARCO 这类经常检索不到有用 Wikipedia 文档的问题，模型似乎会学会检索一组固定文档，作为不依赖检索的信号。

这段很值得读，因为它提醒你：论文不是没有考虑“资料不足”，而是当时的机制没有带来收益。今天工程 RAG 仍然需要无答案处理，只是可能用阈值、拒答策略、证据充分性评估，而不是论文里的 null document。

错误示例：把没有证据的问题硬答掉Text
检索结果为空。
模型回答：根据资料，答案是 2018 年的 X。

问题：资料里根本没有这个答案，回答却伪装成有依据。
判断句：没有有用文档时，系统应该显式处理证据不足，而不是让生成器自由补。

Retrieval collapse：检索器可能学坏

COLLAPSE

附录 H 提到，早期实验中某些任务会出现 retrieval collapse：检索组件学会对不同输入取回同一批文档。随后生成器学会忽略这些文档，整个 RAG 表现退化得像 BART。

可能原因有两个。第一，有些任务对事实知识的需求不明确，检索文档对目标输出帮助不稳定。第二，长输出序列可能给检索器更模糊的梯度。生成器很强时，它可能绕开检索，靠参数记忆和语言模式完成任务。

这说明 RAG 不适合盲目套到所有任务上。任务必须让检索有用，训练目标也要能给检索器足够清楚的信号。否则你只是多接了一个索引，模型却没真正用它。

Broader Impact：更事实也不等于无风险

IMPACT

论文的 broader impact 很克制。积极面是：RAG 更接近真实知识，生成时幻觉更少，访问过的知识可以被检查，也可以更容易控制和更新。它可以用于医学、工作辅助等知识密集场景。

风险也直接来自这份能力。外部知识源可能有错误和偏见，Wikipedia 也不是完全事实、完全中立。RAG 作为语言模型也可能被用来生成误导内容、冒充他人、自动生产垃圾信息或钓鱼内容。检索增强让内容看起来更有依据，滥用时反而更危险。

所以 RAG 的安全边界不只在模型，还在资料源、检索策略、引用呈现、拒答机制、审计日志和使用场景。对于高风险领域，只说“我们接了知识库”远远不够。

收益对应风险需要的控制
更事实错误资料也会被放大资料治理、版本记录、来源权重
可溯源伪引用或误读引用引用验证、证据片段展示
易更新新索引引入偏见或污染索引审核、回滚、变更记录

整篇论文的最终抓手

SYNTHESIS

读完这篇论文，你要带走的不是“RAG 等于检索加生成”这句空话，而是一条完整链路：外部文档作为非参数记忆，DPR 给出文档分布，BART 在文档条件下生成，训练时对潜在文档边际化，测试时用近似解码产生输出，最后通过多任务实验、消融和热替换验证收益与边界。

RAG 的强处是把知识从模型权重里拿出一部分，放到可查、可换的外部文本中。它的难处是检索和生成之间的信用分配：哪个文档该被取回，生成器该多信文档，资料不足时怎么停下来，任务不需要事实时检索会不会坍塌。

这也是今天做 RAG 工程时仍然绕不开的问题。不同的是，今天很多系统不再训练论文式的 RAG 概率模型，而是用 embedding、reranker、prompt、long context 和工具调用搭流程。但论文里的核心判断仍然有用：外部记忆是否真的被访问、是否可更新、是否减少幻觉、是否能被评估。

你现在应该能做什么：看到一个 RAG 系统，能分别检查知识源、检索、生成、更新、拒答和评估，而不是只看最终回答。

最后一块常见误区

PITFALLS

误区 1：换索引等于万事更新

换索引只更新外部知识源，query encoder、生成器和 prompt 仍可能出错。

误区 2：没有 null document 就不用拒答

论文没采用 null document，不代表工程系统可以强行回答无证据问题。

误区 3：RAG 适合所有生成任务

检索需求不明确的任务可能出现 retrieval collapse。

误区 4：外部知识天然可信

知识源可能错误、过时、有偏见。RAG 会继承并放大这些问题。
