这节课先看清什么

goal

先看对象

这一页只处理语言建模目标。你要先知道模型在学什么，再去看 tokenizer、Transformer、优化器和系统细节。

先看方向

训练时，前文是输入，后一位是真实标签。模型不是在复述整句，而是在给正确的下一位 token 提高概率。

先看判断线

看到训练代码时，你应该能先说出 x、y、logits、softmax 和 loss 分别承担什么角色。

把这一节学完，不代表你已经会写完整训练系统；它只表示你已经能把训练目标和表示方式分开。后面的单元会继续补 tokenizer 的切分方式、Transformer 的内部结构、训练循环和性能瓶颈。

先看一遍闭环

overview

最小闭环只有五步：文本进来，切成 token，把前文做成 x、把后一位做成 y，模型给每个位置输出 logits，最后用 y 去算交叉熵，loss 再回到参数更新。

输入前 一段自然语言

模型还没看到离散编号，只看到原始文本。

切分后 token 列表

文本被拆成可训练的符号序列，后面的步骤只处理这些符号。

对齐后 x 负责输入，y 负责监督

前文喂给模型，后一位留作答案。这里一旦错位，后面再算 loss 也会偏。

前向后 [T, V] 或 [B, T, V]

每个位置都有一组词表分数，V 是词表大小。

回到参数 一个标量 loss

损失越大，说明正确 token 的概率越低；梯度就会把它往上推。

你可以把这条线记成一张读写图：前面的步骤负责把“该预测什么”说清楚，后面的步骤负责把“模型猜得对不对”变成可计算的损失。只要 x / y 错位没做好，后面再好的模型也会朝错误目标学习。

环节 读什么 写什么 错了会怎样
切分文本 原始句子 token 序列 长度暴涨，或者词表粒度失控
构造监督 token 序列 x / y 错位对 模型学成复制当前 token
模型前向 x 和参数 logits 最后一维对不齐，loss 没法按位置比较
计算损失 logits 和 y 一个标量 loss 如果 target 读错，loss 再小也没有意义
更新参数 loss 和优化器状态 新的参数 看起来在训练，实际没有学到东西

先把术语钉住

terms

这几件事最容易混在一起：token、输入、标签、分数、概率、损失。看见它们时，先问“它是表示、目标、分数，还是损失”。

术语 它是什么 不要和什么混 为什么重要
token 模型实际看到的离散符号 不要直接等同于“词” 它决定序列长度、词表大小和训练成本
x 输入序列，负责给上下文 不要和 y 混成同一列 它告诉模型“看到什么以后再预测”
y 真实下一个 token 不要当成模型输出 它是监督信号，决定 loss 怎么算
logits 每个位置对词表中每个 token 的分数 不要直接当概率 它要先经过 softmax 才能解释成分布
softmax 把分数变成概率分布 不要和 loss 混在一起 它让“正确答案的概率”能被比较
交叉熵 衡量正确 token 被分到多大概率的损失 不要把它理解成聊天质量分数 它只在 base 目标里看下一位 token 的预测质量

token 不等于词

一个 token 可能是一个字、一个词、一个子词片段，甚至是标点。你如果把 token 当成“单词”，训练成本和上下文长度的估算都会偏掉。

logits 还没变成概率

logits 只是分数。只有经过 softmax，才能把“谁更像正确答案”变成概率分布，再去和真实标签比较。

闭环里每一步读什么、写什么

trace

这里看的是状态，不是口号。训练时每一步都在改同一个对象：先改样本表示，再改模型输出，再改参数。你只要把读写顺序理顺，很多看起来复杂的问题会突然变简单。

步骤 读什么 写什么 什么时候有效 错了的症状
输入文本 原始句子 token 序列 切词器和词表已固定 长度异常，或者同一句话切得不稳定
构造 x / y token 序列 输入 x 与目标 y 已经有稳定的 token 编号 模型像在抄答案，loss 看着低但没学会预测
前向传播 x 和参数 logits 张量维度对齐，最后一维对应词表大小 V shape 报错，或者无法逐位置比较
损失函数 logits 和 y loss 标量 每个位置的真实 token 都能对上 loss 数字有了，但和任务目标没关系
反向和更新 loss 和优化器状态 新参数 梯度没有被截断或清零错位 训练“在跑”，曲线却纹丝不动
$$ L = - \frac{1}{T} \sum_{t=1}^{T} \log p_{\theta}(y_t \mid x_{\le t}) $$

这里的 T 是错位后序列的长度，x 是输入，y 是对应位置的真实下一个 token，p_{\theta} 是模型对正确答案的概率。只要正确 token 的概率上升，-log 就会下降，loss 也会跟着变小。

公式里最关键的不是符号本身，而是方向：模型先产出分数，再把分数变成概率，再把真实标签拿来比较。训练就是沿着这个方向，把正确 token 的概率慢慢推高。

从单句到 batch，形状怎么变

shape

单句很好懂，但训练不会只看一句。实际训练时，会把多条样本打包成 batch，这时你要同时盯住两个轴：batch 维 B 和时间步维 T。B 表示一次喂给模型多少条样本，T 表示每条样本里有多少个位置。

如果样本长度不一样，通常会补 padding，再配一个 mask。mask 的作用很朴素：把填充位置排除在损失之外。这样做以后，模型不会把空位当成真 token 学进去。

对象 典型形状 读什么 写什么 错了会怎样
单条 x [T] 一条 token 序列 当前位置之前的上下文 如果直接把整句塞进去，y 就没法错位
单条 y [T] 同一条 token 序列 后一位真实 token 如果没错位，模型学到的是复制，不是预测
batch x [B, T] 多条样本打包后的输入 整个 batch 的前文 如果 T 没统一，后面的矩阵运算会乱掉
batch y [B, T] 多条样本打包后的标签 每个位置的下一 token 如果 y 对不齐，loss 会看见错误监督
logits [B, T, V] x 和参数 每个位置对词表 V 的分数 最后一维不是 V，交叉熵就没法按词表算
mask [B, T] 哪些位置是真数据 把 padding 位排除掉 padding 被算进 loss，会把训练带偏
$$ L = \frac{1}{\sum_{b,t} m_{b,t}} \sum_{b=1}^{B}\sum_{t=1}^{T} m_{b,t}\, \bigl(-\log p_{\theta}(y_{b,t}\mid x_{b,\le t})\bigr) $$

m_{b,t} 是 mask。它等于 1 时，这个位置参与平均；它等于 0 时，这个位置被忽略。这样写的好处是，长度不一样的样本也能放在一起训练，而且 padding 不会污染 loss。

batch 里的平均不是装饰

如果你不把有效位置平均掉，长样本会比短样本更重。训练看起来还在算 loss，但不同长度的样本已经不在同一尺度上比较了。

为什么这里要看形状

shape 不是形式主义。它直接告诉你哪一维是样本、哪一维是时间、哪一维是词表。三个维度错一个，训练就会在不该错的地方报错，或者更糟，悄悄学偏。

padding 和 mask 为什么要一起看

mask

当 batch 里的句子长短不一样时，通常要把短句补到同样长度。补出来的 PAD 只是占位符，不是真 token。如果你让它参与 loss，模型会被迫去“预测空位”，训练就会朝错误方向偏一点。

最稳妥的做法，是把每个位置都标一个 mask：真实位置记 1，填充位置记 0。这样，loss 只在真实 token 上累积，padding 位不会影响平均值。这个逻辑看起来简单，但它正是很多训练脚本的隐形坑。

样本 x y mask 说明
样本 A 我 / 喜 / 欢 / 预测 喜 / 欢 / 预测 / token 1 / 1 / 1 / 1 完整样本，所有位置都参与 loss
样本 B 语 / 言 / PAD / PAD 言 / 模 / PAD / PAD 1 / 1 / 0 / 0 后两位是补位，只用于对齐 batch 形状
带 mask 的 loss 片段 
loss_per_pos = cross_entropy(logits.reshape(-1, V), y.reshape(-1), reduction="none")
mask_flat = mask.reshape(-1).float()
masked_loss = (loss_per_pos * mask_flat).sum() / mask_flat.sum()

这段代码只做了一件事：先算出每个位置的损失，再用 mask 把补位去掉，最后按有效位置数平均。只要你记住这一点，看到 “padding” 和 “loss” 同时出现时，就知道先检查 mask，而不是先怀疑模型结构。

典型故障

如果短句和长句混在一起训练，而你没有 mask，loss 往往会被 PAD 拉低或拉高，曲线看起来不稳定，但根因不是学习率，而是有效位置数算错了。

先查哪一项

先查 y 里有没有把 PAD 当真答案，再查 mask 有没有和 y 同步，再查 loss 平均时是不是只除以有效位置数。这个顺序通常比改模型参数更快。

用一个短句跑一遍

example

假设句子是“我 喜 欢 预测 下一个 token”。训练时，输入拿前五个 token，目标拿后五个 token。模型看到的是前文，监督信号是后一位。

x 输入
我 喜 欢 预测 下一个
y 目标
喜 欢 预测 下一个 token
最小训练伪代码 
tokens = ["我", "喜", "欢", "预测", "下一个", "token"]
x = tokens[:-1]
y = tokens[1:]

V = vocab_size
logits = model(x)                  # shape: [T, V]
loss = cross_entropy(logits.reshape(-1, V), y.reshape(-1))
loss.backward()
optimizer.step()

这里先把 reshape(-1, V) 当成一个记号：它表示把 batch 里的所有位置摊平，逐位置和目标对齐。真正重要的是哪一位是输入，哪一位是标签。

真实 token 概率 单位置 loss = -log(p) 直觉
0.90 0.105 模型几乎把正确答案排在最前面
0.50 0.693 模型有把握，但还不稳
0.10 2.303 模型对正确答案给得太少
0.01 4.605 模型几乎没把正确 token 当回事

当你把这张表和上面的伪代码放在一起看，训练曲线就不再只是一个数字了。它是在告诉你，正确 token 的概率有没有被推高，还是模型一直在把分数给别的位置。

从 logits 到概率，再到 loss

softmax

这一步是很多初学者第一次真正卡住的地方。logits 只是分数，真正可比较的是 softmax 之后的概率分布。只要你把这个转换看懂，loss 的数字就不再神秘。

假设某个位置上，模型对三个候选 token 的 logits 分别是 2.0、1.0、0.0。softmax 会把它们变成一组加起来等于 1 的概率。因为指数会放大差异，logit 高一点的 token 会拿到更多概率，但不会直接等于 1。

候选 token logit softmax 后概率 如果它是正确答案，loss 大概多大
A 2.0 0.67 0.40
B 1.0 0.24 1.43
C 0.0 0.09 2.41
手算 softmax 的思路 
import math

logits = [2.0, 1.0, 0.0]
exps = [math.exp(x) for x in logits]
total = sum(exps)
probs = [x / total for x in exps]

print([round(p, 3) for p in probs])
$$ p_i = \frac{e^{z_i}}{\sum_j e^{z_j}}, \qquad L = -\log p_{\text{correct}} $$

这里的 z_i 是 logits，p_i 是概率。正确答案的概率越高，-log 越小，loss 也越小。训练就是不断调参数，让正确 token 的 z_i 相对更高一点。

常见误读

如果你把 logits 当成概率，看到 2.0、1.0、0.0 这种数时会误以为“最大值就是答案”。其实真正比较的是 softmax 之后的分布，而不是原始分数本身。

训练里真正被推动的是什么

被推动的不是“某个 token 的绝对分数”，而是正确 token 相对其他 token 的位置。这个相对关系一旦看懂，很多 loss 曲线的变化就能解释得通。

读训练代码时先找哪几行

code

训练脚本看起来长，真正决定这节课成败的往往就几行。先找输入怎么来、目标怎么错位、分数怎么变成 loss、loss 怎么回到优化器。只要这四件事看懂了，其它花哨的参数先放后面。

读代码的最短路径 
tokens = tokenize(text)             # 原始文本 -> token 序列
x = tokens[:-1]                     # 看到前文
y = tokens[1:]                      # 预测后一位
logits = model(x)                   # [B, T, V] 或 [T, V]
loss = cross_entropy(logits, y)     # 逐位置对齐监督
loss.backward()
optimizer.step()
行号 它读什么 它写什么 最常见的误读 一眼先查什么
tokenize(text) 原始文本 token 序列 把 token 当成词 切分后的长度有没有变化得离谱
x = tokens[:-1] token 序列 输入前文 把 x 和 y 写成同一条切片 这里有没有少掉最后一位
y = tokens[1:] token 序列 监督标签 忘了错开一位 这里有没有从第二位开始取
logits = model(x) x 和参数 词表分数 把 logits 直接当概率 最后一维是不是 V
cross_entropy(...) logits 和 y loss 以为只要 loss 下降就一定学对 目标是不是下一位 token

如果你只看模型结构，很容易漏掉真正出错的地方。很多训练问题不在层数、头数或激活函数，而在这几行里某个切片、维度或 target 对齐写错了。

故障实验室：三种最像真的报错

lab

下面这三个调试场景，基本就是这节课最常见的三类症状。点不同案例，看输出会怎么变。你要练的是先认症状，再回到闭环里找原因。

step 1  x shape: [2, 5]
step 2  y shape: [2, 5]
step 3  logits shape: [2, 5, 32000]
step 4  loss: 2.31
note    batch、错位和词表都对齐，训练可以继续

错位失败的假象

这种情况最麻烦，因为 loss 可能还会下降。你看到的不是“学会了”，而是“把当前 token 复制得更熟了”。

词表不匹配的含义

如果 target id 超过 logits 的词表维度，问题往往不在优化器，而在 tokenizer、词表大小或数据编号没有统一。

loss 曲线怎么读

diagnosis

训练时最常见的误解，是把“loss 在变小”直接等同于“模型在变强”。更稳的读法是：先看训练 loss 的方向，再看验证 loss 是否跟上，最后看两者之间的距离有没有越来越大。

如果训练 loss 降得很快，但验证 loss 一直不动，通常不是“模型太弱”，而是数据切分、目标错位或过拟合在说话。如果训练和验证一起下降，而且下降幅度接近，说明模型大概率真的在学通用模式。

曲线现象 更像什么 下一步先查哪一类问题
训练 loss 持续下降，验证 loss 也下降 大体正常 继续看样本质量和收敛速度
训练 loss 下降很快，验证 loss 不动 过拟合、数据泄漏或监督写错 先查训练 / 验证分流，再查 input / target
两条曲线都很抖 batch 太小、mask 不稳或学习率不合适 先查 batch、mask 和学习率，而不是先加层数
训练 loss 几乎不变 模型没在学，或者梯度链路断了 先查监督对齐，再查 backward / step 有没有真的执行

为什么要同时看两条曲线

训练 loss 只告诉你“模型对已见数据的拟合程度”，验证 loss 才告诉你“它有没有在陌生数据上保持预测能力”。只看一条，容易把记忆当能力。

这页的收尾动作

以后你读训练日志时，先问三件事：目标对了吗，mask 对了吗，训练和验证分开了吗。只要这三个问题没答清，先别急着改模型规模。

从一条日志回到问题源头

log

假设你看到下面这组日志：训练 loss 从 2.9 掉到 1.4，验证 loss 只从 3.0 掉到 2.8，样本数量正常，显存也没有爆。这个时候最容易犯的错，是立刻去调更大的模型或更小的学习率。更稳的做法是先回到这一页的闭环，检查监督是否真的对齐。

一眼能读懂的日志 
step 1200 | train loss 1.42 | val loss 2.79 | mask ok
step 1600 | train loss 1.31 | val loss 2.78 | target shift ???
step 2000 | train loss 1.24 | val loss 2.77 | repeat rate up
日志信号 更像什么 先查哪一处 为什么
train loss 快速下降，val loss 几乎不动 训练在记忆，泛化没跟上 训练 / 验证分流、重复样本、目标错位 数据或监督先出问题，比模型结构更常见
mask ok，但 repeat rate 升高 模型开始偏向重复模式 目标切片、温度、采样设置 训练目标和生成设置不是同一件事，但会互相影响
loss 抖得厉害 更新不稳定 batch、mask、学习率 输入和平均方式不稳时，曲线会先乱掉

这条链的重点是顺序：先判断日志在说哪类问题，再回去查对齐、mask 和数据分流，最后才考虑调参。只要顺序对了，很多看似复杂的训练失败会直接缩成几个可以检查的点。

常见误判

训练 loss 降得快，不代表下一位 token 的学习方式就对了。它只说明模型在当前目标上更擅长了；如果目标写错，下降得再漂亮也没用。

这页最实用的回看动作

下次看到训练日志，先把“目标对齐、mask、训练验证分离”三个检查点默念一遍，再决定要不要改学习率或模型规模。

最常见的几种错法

debug

这页最值得记住的不是定义，而是排错顺序。先查对齐，再查维度，再查训练和验证是否分开，最后才怀疑模型不够大。

症状 最可能的原因 先查什么

loss 与能力

用 4 句以内解释：为什么交叉熵下降，能说明模型更接近真实 token，却不等于聊天能力已经变强。

答案里要说清三层关系：下一位 token 的概率、base objective、后训练能力。

参考答案 / 评分要点

好的答案至少要提到三点：第一，交叉熵下降意味着模型给正确下一位 token 分到更高概率；第二，这只是在优化 next-token objective，还没有覆盖指令跟随、偏好对齐和对话格式；第三，聊天能力通常还要依赖后续的微调、偏好优化和解码设置。只要把这三层分开，答案就够了。
