Tokenizer 与 token 流

这一页只讲 tokenizer 在训练管线里的职责：把原始文本变成稳定的 token id，再把 token id 组织成可训练的 token 流窗口。先把切分、特殊 token、padding 和 EOS 这些边界看清，后面学 Transformer 和训练循环时才不会把锅甩错地方。

上一节：语言建模全景 课程地图 下一节：Transformer 最小实现

读完本页应该能做什么

  • 能看懂一份 tokenizer 配置，知道 encode/decode、normalization、pre-tokenization、model、post-processing、decoding 分别在做什么。
  • 能把一段 token 流切成训练窗口，分清 x 输入、y 目标、padding 和 EOS 的位置。
  • 能检查常见错误：词表和 embedding 不一致、边界 token 漏掉、padding 进 loss、tokenizer 与 checkpoint 不匹配。

先抓主线

map

Tokenizer 的作用很朴素：把人能读的字符串，变成模型能稳定处理的 token id 序列。它负责把同一段文本在同一套规则下切成同一种序列，这样训练、评估和复现才说得上话。

公共管线

pipeline

Hugging Face 的 Tokenizers 文档把公共管线写得很清楚：先归一化，再预切分，再走具体模型，再做后处理，最后负责 decode。你不用把每个库的内部实现都背下来，但要知道这些名字指的是哪一层。

子词切分

subword

Hugging Face 的 tokenization 概述把常见的子词方法放在一起讲：BPE、WordPiece 和 Unigram。它们共同要解决的都是同一个问题，让常见片段尽量短、让词表别膨胀得太快，同时让罕见词还能拆得出来。

为什么常见片段会被留住

因为频繁出现的片段，在训练中更值得分配一个稳定的 id。这样一来，常见词、词根、标点组合和代码片段都能复用同一套表示。

为什么稀有片段会被拆开

如果所有稀有词都占一个独立 id，词表会爆炸。拆成子词后，模型能共享片段，也更容易处理没见过的新组合。

子词切分是大多数通用 LLM 的折中方案

真正该选哪种切分，取决于语料、语言混合程度、是否要对齐现成 checkpoint，以及你愿不愿意一起改 embedding 和训练配置。

同一句话的真实切法

example

拿一句混合文本来做完整检查：在 README 里写 tokenizer.json，然后检查 vocab_size。这类句子同时包含中文、英文、下划线、点号和逗号，最适合看 tokenizer 到底有没有把边界处理好。

代码
sample = "在 README 里写 tokenizer.json，然后检查 vocab_size"
ids = tokenizer.encode(sample)
text = tokenizer.decode(ids)

assert max(ids) < tokenizer.vocab_size
print(ids[:12])
print(text)

# 如果 text 丢了空白、标点或文件名片段，先回头查 normalization 和 special token。
# 如果 ids 超过 vocab_size，先查 tokenizer.json 和模型 embedding 是否同一份配置。

如果这一步坏了，会怎样

切分一旦不稳定，训练窗口、vocab_size、embedding 和评估都会跟着偏。模型表面还能跑，实际上输入单位已经变了。

练习时先做的动作

先随机抽一条语料做 encode / decode，再看 id 范围和 special token。你能把这个动作做熟，后面查训练数据时会省很多时间。

token 流

stream

把文本切成 token id 之后，训练最常见的操作是滑动一个窗口。你给模型前文 x，再把后一位 y 留给 loss。这个错位是 next-token 训练的核心。

这里的 t 是已经切好的 token 序列。x 负责输入，y 负责监督；只要两者没错开一位，模型就会学歪。

特殊 token

boundary

special token 是训练契约的一部分。它们告诉模型哪里开始、哪里结束、哪里只是补位，必要时还会告诉 decode 该不该把它们留回去。

一个容易忽略的点

有些字节级方案可以靠更细粒度的回退覆盖未知输入。真正要盯的是：这些 token 的 id 是否和 tokenizer 配置、数据文件和模型 embedding 一起保存。

文件检查

artifact

在实际训练里，tokenizer 不是一段抽象概念，它是一组可以检查的文件和 id 契约。llm.c 的公开 README 就给了一个很实用的例子：.bin 文件前面有短 header，后面跟着 token 流，token id 使用 GPT-2 tokenizer 对应的编号。你不需要照搬这个格式，但很适合拿来做检查思路。

代码
import json

cfg = json.load(open("tokenizer.json", encoding="utf-8"))
vocab_size = cfg["model"]["vocab_size"]
special_tokens = [item["content"] for item in cfg.get("added_tokens", [])]

print("model type:", cfg["model"]["type"])
print("vocab_size:", vocab_size)
print("special tokens:", special_tokens)

# 对 token 文件再做一次抽样 decode：
# 1. 读出前几个 id
# 2. 确认最大 id 小于 vocab_size
# 3. decode 回原文，检查空白、标点和边界 token 是否正常

一条样本怎么坏掉

failure

把错误从头走一遍，比只看报错更快。还是用同一句混合文本来想：在 README 里写 tokenizer.json，然后检查 vocab_size。它先经过 normalization，再被切成片段，随后变成 token id，最后才进入 x / y 窗口和 batch。任何一步写错，后面看起来像是模型在学，实际上是在放大输入契约的偏差。

代码
sample = "在 README 里写 tokenizer.json，然后检查 vocab_size"
ids = tokenizer.encode(sample)

assert max(ids) < tokenizer.vocab_size
x = ids[:-1]
y = ids[1:]
assert len(x) == len(y)

# 若 batch 里有 PAD，loss 平均前先把 PAD 位置 mask 掉。
# 先确认输入契约，再确认监督契约，最后再看模型本身。

把这页带进后面的训练页

后面看到 embedding、loss、checkpoint 或验证集时，先回想这一条链：原始文本、切分、id、x / y、mask。只要其中一步写错，后面的数字就会把错误放大。

怎么选

decision

Tokenizer 选择要同时看兼容性、语料分布和维护成本。要对齐公开 checkpoint 时，tokenizer 往往必须跟着走；要从零训练一个 base model 时，就要把语料、边界 token 和评估口径一起想。

这里先不把“选型”写成结论题

真正的决定通常要看你是不是要兼容现成模型、是否控制全部训练数据、以及你能不能接受 embedding 一起改。这里先学会看条件，下一页看模型结构时再接着往下走。
