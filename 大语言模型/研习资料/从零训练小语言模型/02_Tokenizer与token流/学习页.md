# 先抓主线 {#overview}
<!-- tag: map -->

Tokenizer 的作用很朴素：把人能读的字符串，变成模型能稳定处理的 token id 序列。它负责把同一段文本在同一套规则下切成同一种序列，这样训练、评估和复现才说得上话。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box info
### 你现在只要先记住一件事
tokenizer 不是“把文本压缩一下”这么简单，它是在给训练过程定单位。单位一旦错，长度、词表、mask、评估口径都会跟着偏。

# 公共管线 {#pipeline}
<!-- tag: pipeline -->

Hugging Face 的 Tokenizers 文档把公共管线写得很清楚：先归一化，再预切分，再走具体模型，再做后处理，最后负责 decode。你不用把每个库的内部实现都背下来，但要知道这些名字指的是哪一层。

tokenization pipeline复制

```text
normalization
pre-tokenization
model
post-processing
decoding
```

阶段
读什么
写什么
漏掉时最先坏什么


normalization
原始字符串
统一后的文本
同一句话在不同输入法、空白或 Unicode 形式下切法不一致


pre-tokenization
规范化后的文本
更小的候选片段
标点、空格、代码块的边界被切歪


model
片段序列
token id
词表和 id 对不上，embedding 查表会出错


post-processing
token ids
补上 BOS / EOS 等控制位

decoding
token ids
可读文本
你无法抽样确认 token 流是不是还原正确

# 子词切分 {#subword}
<!-- tag: subword -->

Hugging Face 的 tokenization 概述把常见的子词方法放在一起讲：BPE、WordPiece 和 Unigram。它们共同要解决的都是同一个问题，让常见片段尽量短、让词表别膨胀得太快，同时让罕见词还能拆得出来。

切法
直觉
优点
代价


字 / 字符切分
每个字都单独看
简单，边界清楚
序列长，词表利用率不高


整词切分
词是最小单位
常见词很紧凑
新词、拼写变化和代码符号容易炸


子词切分
常见片段合并，罕见词拆开
长度、覆盖率和词表规模之间更平衡
要接受“一个词被拆成几个片段”

## box soft
### 为什么常见片段会被留住
因为频繁出现的片段，在训练中更值得分配一个稳定的 id。这样一来，常见词、词根、标点组合和代码片段都能复用同一套表示。

## box info
### 为什么稀有片段会被拆开
如果所有稀有词都占一个独立 id，词表会爆炸。拆成子词后，模型能共享片段，也更容易处理没见过的新组合。

## box warn
### 子词切分是大多数通用 LLM 的折中方案
真正该选哪种切分，取决于语料、语言混合程度、是否要对齐现成 checkpoint，以及你愿不愿意一起改 embedding 和训练配置。

## box soft
### 这里能学到什么
你可以直接拿一个混合句子去查：切分有没有把文件名和标点保住，decode 能不能回到原文，token id 会不会越过词表上限。这比只背“BPE 是什么”更接近真实排错。

## box warn
### 如果这一步坏了，会怎样
切分一旦不稳定，训练窗口、vocab_size、embedding 和评估都会跟着偏。模型表面还能跑，实际上输入单位已经变了。

## box info
### 练习时先做的动作
先随机抽一条语料做 encode / decode，再看 id 范围和 special token。你能把这个动作做熟，后面查训练数据时会省很多时间。

# token 流 {#stream}
<!-- tag: stream -->

把文本切成 token id 之后，训练最常见的操作是滑动一个窗口。你给模型前文 x，再把后一位 y 留给 loss。这个错位是 next-token 训练的核心。

<!-- interactive: ['shift-demo'] → keep as HTML -->

$$
 x = [t_0, t_1, \dots, t_{n-2}], \qquad y = [t_1, t_2, \dots, t_{n-1}]
 $$
 这里的 `t` 是已经切好的 token 序列。`x` 负责输入，`y` 负责监督；只要两者没错开一位，模型就会学歪。

对象
内容
读什么
写什么
错了的症状


token 流
一串 id
文档边界后的连续序列
训练窗口的原材料
断句、拼接和边界处理会直接影响后面的监督


x
前文 token
上下文
模型输入
如果把 y 也塞进来，模型会像在抄答案


y
后一位 token
正确答案
loss 的 target
如果 y 没右移，训练目标就错了


mask
有效位标记
哪些位置是真样本
排除 padding
padding 被算进 loss，会污染曲线

# 特殊 token {#specials}
<!-- tag: boundary -->

special token 是训练契约的一部分。它们告诉模型哪里开始、哪里结束、哪里只是补位，必要时还会告诉 decode 该不该把它们留回去。

token
用途
通常写入哪里
漏掉后的症状


BOS / start
标记序列开始
post-processing 或数据拼接
模型不知道“开头”是什么，某些任务的起始上下文变乱


EOS
标记文档结束
文档拼接、样本切分
多篇文本会被当成一篇续写，边界污染最常见


PAD
批处理对齐
batch padding
如果没 mask，模型会被训练去预测空位


UNK / byte fallback
兜底处理未知输入
词表或字节级回退
遇到新符号时直接崩掉，或者 decode 不回原文

## box info
### 一个容易忽略的点
有些字节级方案可以靠更细粒度的回退覆盖未知输入。真正要盯的是：这些 token 的 id 是否和 tokenizer 配置、数据文件和模型 embedding 一起保存。

# 文件检查 {#artifact}
<!-- tag: artifact -->

在实际训练里，tokenizer 不是一段抽象概念，它是一组可以检查的文件和 id 契约。llm.c 的公开 README 就给了一个很实用的例子：`.bin` 文件前面有短 header，后面跟着 token 流，token id 使用 GPT-2 tokenizer 对应的编号。你不需要照搬这个格式，但很适合拿来做检查思路。

tokenizer 检查伪码复制

```python
import json

cfg = json.load(open("tokenizer.json", encoding="utf-8"))
vocab_size = cfg["model"]["vocab_size"]
special_tokens = [item["content"] for item in cfg.get("added_tokens", [])]

print("model type:", cfg["model"]["type"])
print("vocab_size:", vocab_size)
print("special tokens:", special_tokens)

# 对 token 文件再做一次抽样 decode：
# 1. 读出前几个 id
# 2. 确认最大 id 小于 vocab_size
# 3. decode 回原文，检查空白、标点和边界 token 是否正常
```

检查对象

错了会怎样
修法

训练脚本和 tokenizer 配置对不上
把配置和 checkpoint 一起保存


.bin token 文件

先做抽样 decode，再看最大 id

# 一条样本怎么坏掉 {#failure}
<!-- tag: failure chain -->

把错误从头走一遍，比只看报错更快。还是用同一句混合文本来想：`在 README 里写 tokenizer.json，然后检查 vocab_size`。它先经过 normalization，再被切成片段，随后变成 token id，最后才进入 x / y 窗口和 batch。任何一步写错，后面看起来像是模型在学，实际上是在放大输入契约的偏差。

步骤
你在看什么
第一眼症状
先查哪里


原始文本
空格、标点、文件名、大小写
同一句话在不同输入里切法不一致
normalization 是否改坏了该保留的字符


token id
最大 id 是否小于 vocab_size
embedding 行数不够，或者 decode 回不去
tokenizer.json 和模型配置是否同一份


x / y 窗口
后一位是否真的右移一格
模型像在复述输入
监督切片有没有和输入切片错开


batch + PAD
短句和长句是否一起对齐
loss 数字看着变了，但不是因为模型学对了
mask 是否把 PAD 位置排除掉

首查顺序复制

```python
sample = "在 README 里写 tokenizer.json，然后检查 vocab_size"
ids = tokenizer.encode(sample)

assert max(ids) < tokenizer.vocab_size
x = ids[:-1]
y = ids[1:]
assert len(x) == len(y)

# 若 batch 里有 PAD，loss 平均前先把 PAD 位置 mask 掉。
# 先确认输入契约，再确认监督契约，最后再看模型本身。
```

## box warn
### 故障的代价
这类错误通常不会第一眼爆炸。模型还能前向、还能降 loss，但学的对象已经变了，所以排错时要先看输入单位和监督切片，而不是先改学习率。

## box info
### 把这页带进后面的训练页
后面看到 embedding、loss、checkpoint 或验证集时，先回想这一条链：原始文本、切分、id、x / y、mask。只要其中一步写错，后面的数字就会把错误放大。

# 怎么选 {#choice}
<!-- tag: decision -->

Tokenizer 选择要同时看兼容性、语料分布和维护成本。要对齐公开 checkpoint 时，tokenizer 往往必须跟着走；要从零训练一个 base model 时，就要把语料、边界 token 和评估口径一起想。

场景
更合适的做法
原因
别忽略的代价


要继续训练某个公开 checkpoint
沿用它的 tokenizer
embedding、特殊 token 和评测口径已经绑定
换 tokenizer 等于改输入契约，复现会很难


从零训练 mixed-language / code corpus
先用候选 tokenizer 跑 baseline，再根据语料分布决定是否自训或扩展
需要看长度、覆盖率和边界 token 是否真的合适
自训会增加前期工作量和文件管理复杂度


语料里有很多新符号或结构化标记
考虑新增 special token 或重新训练 tokenizer
这些符号如果老被拆碎，训练窗口会很不稳定
special token 的 id 和 mask 规则必须一起更新

## box warn
### 这里先不把“选型”写成结论题
真正的决定通常要看你是不是要兼容现成模型、是否控制全部训练数据、以及你能不能接受 embedding 一起改。这里先学会看条件，下一页看模型结构时再接着往下走。

### 9 synthesis

**Q:** Q9 你拿到一个 `.bin` token 文件，准备送进训练程序。列 5 项你会先检查的内容。

### 10 debug-checklist

**Q:** Q10 如果你要继续训练一个公开 checkpoint，下面哪些条件会让你优先复用它现成的 tokenizer？

- [ ] 需要和 checkpoint 对齐
- [ ] 要沿用公开评测口径
- [ ] tokenizer 已经和 embedding 绑定
- [ ] 你完全想换掉语料和 id 契约

Explain: 前面三项都支持复用现成 tokenizer；最后一项反而说明你要重新考虑输入契约。这个题的关键是“换了 tokenizer 会不会把既有模型和评测一起改坏”。

### 12 transfer

**Q:** Q12 把这一页压成一条下一步动作：下次看到 tokenizer 配置和训练脚本时，你先检查什么，为什么？
