学习目标

goal

先看顺序

这页先盯住 batch → logits → loss → grad → update → log → checkpoint。只要顺序一乱，后面讨论再多超参也只是绕圈。

先看证据

一条可信的训练记录至少要回答三件事：模型有没有在学、更新稳不稳、断掉以后能不能接回同一个位置。

先抓主线

map

把一次训练 step 看成一条状态转移链更有用。数据进来以后，模型先前向得到 logits，再把目标 token 的误差压成 loss，随后 backward 写梯度，优化器和学习率计划决定参数怎么动，最后日志和 checkpoint 负责把这次变化留下来。

batch数据迭代器、batch size、序列长度当前输入、目标 token、计数器样本越界、长度不对、micro-step 和 global step 混掉
forward模型参数、输入 tokenlogits、loss输出形状不对、目标错位、loss 异常高或异常低
backwardloss 和计算图每个参数的 .grad梯度残留、未清零、梯度消失或爆炸
update梯度、优化器状态、学习率新参数、新优化器状态恢复后曲线跳、权重没变、学习率位置错
logstep、loss、lr、tokens/s、grad norm训练记录看不出训练在学、在慢、还是在坏
checkpoint权重、优化器、调度器、随机数、配置可恢复状态只能重训，不能续训

你现在应该能做什么

拿到一段训练日志时，先判断它是在描述 batch、forward、update 还是 recovery，再找对应检查点。这个顺序会让你更快定位问题。

一次训练 step

loop

下面这段伪代码只保留一次训练 step 的关键动作。它的价值不在于“能不能运行”，而在于让你看清每一行是在读什么、写什么、依赖什么，以及哪些顺序不能换。

代码
model.train()
optimizer.zero_grad(set_to_none=True)

x, y = next(train_iter) # x/y 是当前 batch
logits = model(x) # 读参数，写 logits
loss = cross_entropy(logits[:, :-1], y[:, 1:])
loss.backward() # 只写梯度，不改参数

if grad_clip is not None:
 clip_grad_norm_(model.parameters(), grad_clip)

optimizer.step() # 真正改参数
scheduler.step() # 推进学习率计划

if step % log_every == 0:
 log({
 "step": step,
 "train_loss": float(loss),
 "lr": scheduler.get_last_lr()[0],
 "tokens_seen": tokens_seen,
 })

if step % save_every == 0:
 save_checkpoint(model, optimizer, scheduler, step, tokens_seen)

读什么

如果这行漏掉或换序，会怎样

zero_grad

上一轮残留的梯度缓存

把 .grad 清空

旧梯度会跨 step 混进来，更新幅度变脏

clip_grad_norm_当前梯度范数把过大的梯度压回可控区间偶发坏 batch 可能把参数一下推飞
optimizer.step()梯度和优化器状态新参数参数不动，训练表面上还在跑，实际上没学
scheduler.step()当前 step 和调度器状态新学习率恢复后 lr 会错位，曲线很难比较
save_checkpoint当前权重和运行状态可恢复文件中断后只能从头来，或者恢复后曲线跳变

顺序不能乱的地方

zero_grad 一般放在前面，loss.backward() 只管写梯度，optimizer.step() 才改参数，scheduler.step() 需要和更新次数对齐。把这几步分清，后面的诊断才有意义。

优化器与学习率

AdamW

优化器决定参数沿着什么方向动，学习率决定每一步跨多大。AdamW 会记住梯度的近期均值和幅度，再把权重衰减单独拆出来；warmup 负责把早期步幅抬得慢一点，cosine decay 负责把后期步幅收回来。

g_t 是当前梯度，m_t 和 v_t 是一阶、二阶矩，\theta_t 是参数，\eta 是学习率，\lambda 是权重衰减。这里的重点是：AdamW 不只存参数，还存自己的状态。

t 是当前 step，Twarmup 是 warmup 步数，Ttotal 是总步数，\eta_{\max} 是峰值学习率。你要先知道曲线在哪一段，才知道恢复时该不该对得上。

先记一个判断

有效 batch 变了以后，学习率曲线也要重新看；两次实验的更新频率不同，曲线就不该直接对比。

检查点与恢复

state

checkpoint 保存的是训练现场。想从第 20000 step 继续同一次训练，就要把权重、优化器状态、调度器状态、step、随机数状态和训练配置放在一起保存。这样恢复时才能接回同一条训练轨迹。

代码
checkpoint = {
 "step": step,
 "tokens_seen": tokens_seen,
 "model": model.state_dict(),
 "optimizer": optimizer.state_dict(),
 "scheduler": scheduler.state_dict(),
 "scaler": scaler.state_dict(), # 混合精度时需要
 "rng": {
 "torch": torch.get_rng_state(),
 "cuda": torch.cuda.get_rng_state_all(),
 },
 "config": train_config,
 "last_metrics": {
 "train_loss": train_loss,
 "val_loss": val_loss,
 "lr": current_lr,
 },
}

save_atomically(checkpoint, path)

原子保存的意思

先写临时文件，确认写完整后再替换正式文件。这样可以减少“写到一半断掉，留下损坏 checkpoint”的风险。训练里最怕的是中断后你以为自己保存成功了。

恢复时的状态台账

ledger

恢复训练最容易出问题的地方是状态分叉。权重看起来对了，不代表训练现场已经对了。下面这张表把常见状态拆开，方便你按“读什么、写什么、什么时候会旧、第一眼查什么”来排查。

恢复时的第一轮检查

先看 step 和 lr，再看 loss 和 tokens_seen，最后看 grad norm。这三层信息分别告诉你进度、曲线和更新有没有接上。

日志怎么读

trace

日志的作用是把训练过程变成可比较的轨迹。你不需要每个字段都盯住，但要知道哪些字段放在一起，能判断学习、速度、稳定性和恢复是否正常。

如果一次更新处理 196608 个 token，而 step time 是 1.12 秒，吞吐大约是 175k tokens/s。这类数字不是装饰，它们能告诉你数据加载、算子或通信有没有掉速。

代码
step=12000 train_loss=3.58 val_loss=3.71 lr=1.8e-4 grad_norm=0.91 tokens/s=174k ckpt=run_012000.pt
step=12016 train_loss=3.49 val_loss=3.69 lr=1.7e-4 grad_norm=0.88 tokens/s=176k ckpt=run_012016.pt
resume step=12017 train_loss=3.50 val_loss=3.70 lr=1.7e-4 grad_norm=0.89 tokens/s=175k ckpt=run_012016.pt

常见读法

step 连续但 lr 回跳，先查 scheduler；loss 单点尖峰而下一步恢复，先查坏 batch 或数值问题；tokens/s 突然掉，先查数据加载和系统瓶颈。先把字段和症状一一对上，再下结论。

恢复实战

lab

把恢复训练当成一次有顺序的操作。下面这条小流程把权重、优化器、调度器和日志检查拆开，你可以照着排查一遍，看看哪里会先断。

为什么要先做短检查

短检查的作用是把明显错位拦在第一分钟内。你不需要等到长跑后才知道 lr 回到了 warmup，或者 optimizer 还停在第 0 step。

例子与诊断

cases

这部分把前面的概念压成几个可检查的例子。先算一个数，再看一条日志，最后看几种恢复失败的典型样子。你不需要把每个问题背下来，但要知道它们会从哪里露出破绽。

G 是 GPU 数，M 是每张卡的 micro batch，T 是序列长度，A 是梯度累积步数。例子里如果 G=4、M=8、T=1024、A=6，一次参数更新对应 196608 个 token。横轴如果只写 step，就很容易看错训练速度。

常见误区

debug

这页最容易出的错，常常是把状态链路看短了。训练能跑不代表训练对，日志看着正常也不代表恢复正确。

这里先别追的东西

先别把混合精度 scaler、ZeRO、FSDP、通信重排和 kernel 级优化一股脑塞进来。那些都是真的问题，但它们属于后面那条系统线，不属于这页的主线。
