系统性能

GPU 系统与性能瓶颈

这一页接在训练循环之后，专门处理“训练能跑，但跑得慢、显存爆、扩卡不涨、数值发散”这些系统问题。你要把一次 step 拆成可测量的显存、吞吐、kernel、dtype、attention、通信和数据输入链路，而不是把所有现象都归因于 GPU 不够强。

上一节：训练循环、优化器与检查点 课程地图 下一节：Scaling、验证集与评估

读完本页，你应该能做什么

  • 能用参数、梯度、优化器状态、激活值、临时 buffer 和 allocator reserved 列出显存账本。
  • 能用同一口径计算 tokens/s，区分 micro-batch、sequence length、grad accumulation 和 GPU 数的影响。
  • 能读 profiler 的时间线和 self CUDA time，把瓶颈初步归到数据输入、算子、数值精度、attention 或通信。
  • 能给出一套先测量、再改动、再复测的排查顺序，并说明每个优化动作可能牺牲什么。

学习目标

goal

先会记账

看到 OOM 时，先把显存拆成参数、梯度、优化器状态、激活值、临时 buffer、通信 buffer 和 allocator reserved。只看“模型参数多大”会漏掉训练峰值。

先会计时

看到训练慢时，先统一 tokens/s 的口径，再拆 step 时间。micro-batch、序列长度、累积步数和 GPU 数一变，吞吐比较就可能失真。

先会定位

优化前先拿证据：dataloader 等待、kernel 自身耗时、dtype 数值问题、attention 中间压力、NCCL 通信和 GPU 拓扑都要分开看。

本页边界

这里不教完整 CUDA 编程，也不展开分布式训练工程。你只需要学会把一次小语言模型训练 step 拆成可观察的系统账本；下一页再用这些指标进入 scaling、验证集和评估。

性能主线

map

性能排查要从一条训练 step 的时间线开始。CPU 准备 batch，数据被拷到 GPU，模型前向和反向启动一串 kernel，优化器更新参数，分布式场景还要同步梯度或参数分片。慢训练通常不是一个点坏了，而是某段等待、读写或同步占了太多时间。

input dataloader wait

如果 GPU utilization 周期性掉到低位，先查 CPU 解码、token 拼接、pin memory 和 H2D 拷贝。

forward kernel chain

前向会启动 matmul、attention、LayerNorm、mask 和 elementwise kernel。小 kernel 太多时，GPU 可能在调度和读写上花时间。

backward activation peak

反向需要前向中间值，OOM 常在这里出现。记录 peak memory 比只看 step 结束后的 allocated 更有用。

optimizer state traffic

AdamW 会读写 moments 和参数。参数量大时，optimizer state 和它的内存读写会成为一块实成本。

comm synchronization

多卡训练会加入同步。每卡 batch 太小或网络慢时，GPU 可能在等通信，而不是继续计算。

record evidence

日志要能复现判断：step time、tokens/s、allocated/reserved、dtype、GPU 数、通信时间和 profiler 截图。

阶段 主要读写 常见症状 第一条证据
数据输入 CPU 样本、token batch、host-to-device copy GPU utilization 有空洞，step time 抖动大 dataloader time、H2D copy time、GPU timeline gap
前向 / 反向 参数、激活值、attention 中间张量、梯度 OOM、self CUDA time 集中、长序列变慢 peak memory、profiler 按 kernel 排序
优化器 梯度、参数、moments、master weights 大模型 step 更新慢，显存超过参数直觉 optimizer state size、update kernel time
通信 梯度 bucket、参数 shard、NCCL buffer 多卡扩展效率低，等待 all-reduce NCCL 时间、通信/计算重叠、拓扑

读法先把慢训练归到一段时间线，再讨论技巧。没有基线、没有同口径 tokens/s、没有 profiler 的优化建议，通常只是猜。

术语与误区

terms
术语 直觉解释 精确定义 常见误解
peak memory 训练过程中最满的时候。 一次 step 内出现过的最高显存占用，常在 backward。 只看 step 结束时的 allocated，会低估峰值。
tokens/s 每秒处理多少 token。 通常按全局 step 的有效 token 数除以 step_seconds。 把 micro-step 当完整 step，会把数字算歪。
kernel GPU 上执行的一小段工作。 由 CPU 发起、在 GPU 上并行执行的函数，可能只是模型层的一部分。 把 kernel 当神经网络层，容易误读 profiler。
BF16 / FP16 低精度训练的两种常见格式。 BF16 保指数范围，FP16 更省位数；二者都要看数值稳定。 只看速度，不看 loss scale 和 NaN，会翻车。
NCCL all-reduce 多卡同步梯度。 分布式训练里的 collective communication，用来把各卡梯度合并。 通信慢，不一定是网络坏，也可能是每卡 batch 太小。
roofline 看算力和带宽谁先顶住。 用 FLOPs/bytes moved 的直觉判断更像 compute-bound 还是 memory-bound。 把它当精确诊断器，会过度简化。

显存账本

memory

训练显存不等于模型文件大小。参数会占显存，梯度也会占显存；Adam 这类优化器还有 moments；反向传播要保留激活值；框架、通信库和 kernel workspace 也会临时要空间。OOM 排查的第一步是把账本列出来。

$$M_{train} \approx P(b_{param}+b_{grad}+b_{opt}) + M_{act} + M_{tmp} + M_{comm} + M_{reserved}$$

P 是参数个数；b_param、b_grad、b_opt 是每个参数对应的参数、梯度和优化器状态字节数。激活值和临时 buffer 会随 batch、序列长度、实现和 checkpoint 策略变化。

小算例

假设有 300M 参数，参数 BF16 约 2 bytes，梯度 BF16 约 2 bytes，Adam 两个 moments 用 FP32 共 8 bytes。不算激活值，每个参数约 12 bytes，参数态和优化器态已经接近 3.35 GiB。

如果还保留 FP32 master weights，就再加 4 bytes/parameter。这个额外项常被初学者漏掉。

峰值在哪里

峰值常出现在 backward，而不是 step 结束。你要记录 max_memory_allocated、max_memory_reserved，再按 forward、backward、optimizer step 分段观察。

如果反向才 OOM，优先怀疑激活值、attention 中间张量、临时 workspace 或通信 buffer。

显存项 什么时候出现 受什么影响 常用处理
参数 模型加载后长期存在 参数量、dtype、是否有 master weights 更低精度、模型分片、缩小模型
梯度 反向传播后存在，更新前后清理 参数量、梯度累积、分片方式 梯度分片、累积策略、及时 zero_grad
优化器状态 第一次更新后长期存在 AdamW、Adafactor、SGD 等优化器选择 更省状态的优化器、ZeRO/FSDP 分片
激活值 前向保存，反向读取 micro-batch、seq_len、层数、checkpointing 减小 micro-batch/seq_len、activation checkpointing
临时与通信 buffer kernel、attention、NCCL、allocator 运行时出现 实现、拓扑、bucket、动态 shape 固定形状、预留空间、调 bucket、换实现

检查动作报 OOM 时，不要只说“显存不够”。写出 batch、seq_len、dtype、参数量、优化器、是否 activation checkpointing，以及 allocated/reserved/peak 的数字。

吞吐口径

tokens/s

训练速度通常用 tokens/s 表示，但这个数字只有在口径一致时才有意义。你要说清楚统计的是单卡还是全局，是训练 tokens 还是有效更新 tokens，是否包含 dataloader、checkpoint 和评估时间。

$$tokens/s = \frac{n_{gpu} \times micro\_batch \times seq\_len \times grad\_accum}{step\_seconds}$$

这里的 step_seconds 应该是一次 optimizer update 的端到端时间。如果你用 micro-step 时间当分母，或者忘了 grad accumulation，比较就会偏。

配置 计算 tokens/s 读法
1 GPU，micro=8，seq=1024，accum=4，step=2.0s 1 × 8 × 1024 × 4 / 2.0 16,384 这是全局训练吞吐，也是单卡吞吐。
4 GPU，micro=8，seq=1024，accum=4，step=2.4s 4 × 8 × 1024 × 4 / 2.4 54,613 比 1 卡快 3.33 倍，扩展效率约 83%。
4 GPU，micro=4，seq=1024，accum=4，step=1.5s 4 × 4 × 1024 × 4 / 1.5 43,691 step 变短但 tokens/s 变低，不能只看秒数。
timing_skeleton.py
import time
import torch

def measure_step(train_step, warmup=5, repeat=20):
    for _ in range(warmup):
        train_step()
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(repeat):
        train_step()
    torch.cuda.synchronize()
    return (time.perf_counter() - start) / repeat

# CPU 计时前后都要同步，否则可能只测到 kernel 提交时间。
# 真实报告还要写清 batch、seq_len、grad_accum、GPU 数和是否包含 dataloader。

检查动作每次改 batch、accum、GPU 数、dtype 或 dataloader 后，都重新记录同一套口径。吞吐变快但验证 loss 变差时，要回到训练语义，不把它当纯系统收益。

kernel 与 roofline

kernel

Python 里的一层 Transformer，到 GPU 上会变成很多 kernel：矩阵乘、softmax、mask、dropout、LayerNorm、elementwise、optimizer update。性能问题常来自两类：kernel 太碎，启动和内存读写占比高；或者大 kernel 没有喂满 Tensor Core，算力利用不足。

profiler 现象 更可能的原因 先查什么 可能动作
大量很短的 elementwise kernel 算子没融合、batch 太小、Python 控制流太碎 self CUDA time、kernel 数量、shape 是否很小 torch.compile、fused op、增大 micro-batch
matmul 时间长但 GPU 利用率不高 shape 不利于 Tensor Core、dtype 或 layout 不合适 矩阵维度、dtype、是否走 FP16/BF16/TF32 路径 调整 hidden size 对齐、autocast、检查 contiguous/layout
softmax / attention 占比随序列长度暴涨 attention 中间矩阵和 HBM 读写压力大 seq_len、head 数、attention 实现、是否物化分数矩阵 FlashAttention、减小 seq_len、activation checkpointing
$$arithmetic\ intensity = \frac{FLOPs}{bytes\ moved}$$

roofline 的直觉是：如果每搬一字节数据只做很少计算，瓶颈更像内存带宽；如果每搬一字节能做大量矩阵乘，瓶颈更可能是算力或 Tensor Core 利用。它不是精确诊断结论，但能提醒你看 HBM 往返和 kernel 形状。

fused_attention_boundary.txt
优化 kernel 前先问四个问题：
1. profiler 里时间是否真的花在这些 op 上？
2. 这些 op 是 compute-bound 还是 memory-bound？
3. 框架版本是否已经内置融合或高效实现？
4. 改实现以后，数值、mask、dropout、训练/评估模式是否仍然一致？

精度与 attention

dtype

混合精度的目标是让适合低精度的矩阵乘走快路径，同时守住容易不稳定的归约、softmax、loss 和归一化统计。dtype 改动会同时影响显存、吞吐和数值稳定，不能只看速度。

dtype / 机制 主要收益 主要风险 先看指标
FP32 动态范围和精度稳，便于调试 显存和吞吐成本高 基线 loss、grad norm、tokens/s
FP16 + GradScaler 显存低，Tensor Core 快 梯度下溢、scale 波动、NaN loss scale、NaN 首发位置、grad norm
BF16 指数范围接近 FP32，训练更稳 尾数精度较低，硬件需支持 验证 loss、吞吐、关键层输出范围
TF32 在支持的 NVIDIA GPU 上加速 FP32 matmul 矩阵乘精度近似，结果可能有细小差异 是否启用、loss 曲线是否可接受
$$scores\ shape \approx batch \times heads \times seq\_len \times seq\_len$$

标准 self-attention 会比较每个 token 和每个 token。序列长度翻倍，分数矩阵元素数会变成四倍。如果实现显式物化中间矩阵，显存和 HBM 读写都会变重。

边界

FlashAttention 主要解决 attention 的 IO 和中间显存问题。它不会修复 dataloader 等待、optimizer state 太大、NCCL 通信慢或训练超参错误。

多卡通信

parallel

多卡训练增加的是全局计算能力，也增加了同步成本。DDP 常见路径是每张卡算自己的 micro-batch，然后对梯度做 all-reduce；FSDP/ZeRO 会把参数、梯度或优化器状态分片，省显存但加入更多通信和调度约束。

现象 可能原因 证据 动作与代价
8 卡只比 1 卡快 3 倍 通信占比高、每卡 batch 太小、dataloader 等待 NCCL 时间、timeline 空洞、每卡 tokens 数 增大每次同步前计算量；代价是有效 batch 或显存变化
DDP 显存仍然太高 每卡都保留完整参数、梯度和 optimizer state 每卡 memory summary 与参数态估算对不上 考虑 FSDP/ZeRO；代价是通信和实现复杂度增加
跨节点后吞吐掉很多 互联带宽和延迟变差，拓扑不匹配 NCCL 日志、节点内/节点间对比、拓扑图 优先节点内通信、调整并行策略；代价是资源调度限制
$$scaling\ efficiency = \frac{throughput_N}{N \times throughput_1}$$

这个指标只在训练口径一致时有意义。GPU 数变了以后，如果 micro-batch、累积步数、数据加载路径或 checkpoint 频率也变了，就要把这些变化写进报告。

小模型的多卡陷阱

模型太小或每卡 batch 太小时，通信税会显得很重。小规模实验上看到扩展效率差，不必直接推断“大模型也会差”；你要先确认计算量是否足够覆盖同步成本。

profiler 诊断

profile

profiler 的作用不是给出一句“优化建议”，而是把等待、计算、通信和内存峰值摆到同一张证据表里。一个稳的排查流程是：建立基线，确认口径；分段计时；看时间线空洞；按 self CUDA time 排 kernel；记录显存峰值；最后只改一个主要变量。

    record_shapes=True,
    profile_memory=True,
    with_stack=False,
) as prof:
    for step, batch in enumerate(loader):
        if step >= 20:
            break
        loss = train_one_step(batch)

    sort_by="self_cuda_time_total",

# CUDA 计时是否同步，batch/seq_len/dtype/GPU 数是否写进报告。
读表步骤 要看什么 常见判断
1. 时间线 GPU 是否有长空洞，CPU 和 GPU 是否错位 空洞常指向 dataloader、CPU 同步或 H2D 拷贝。
2. self CUDA time 哪个 kernel 真正占 GPU 时间 matmul、attention、norm、optimizer update 要分开看。
3. shape 慢 kernel 对应的输入形状 小 shape 可能启动开销重，大 shape 可能算力或带宽重。
4. memory allocated、reserved、peak 和分段峰值 reserved 高不等于泄漏，可能是缓存、碎片或临时 workspace。
5. 复测 只改一个变量后的同口径指标 优化动作要同时报告 tokens/s、显存和验证 loss。

检查动作把 profiler 结果写成“现象 → 证据 → 判断 → 改动 → 复测”。如果只有“用了某个优化技巧”，这份报告还不能说明问题真的被定位了。

常见错误

pitfalls

未同步就计时

CUDA 默认异步执行。CPU 计时如果前后不加 torch.cuda.synchronize()，很可能只测到提交任务的时间。

只看 allocated

reserved、峰值、非 PyTorch 分配、NCCL buffer 和临时 workspace 都可能影响 OOM。报告里只写 allocated 不够。

频繁 .item()

每步把 GPU tensor 拉回 CPU 可能触发同步，让时间线出现空洞。日志可以降频，必要指标批量记录。

把 batch 改动当纯系统优化

micro-batch、累积步数和 GPU 数改变后，有效 batch 也可能改变。训练曲线变化不能只按吞吐解释。

忽略数据输入

GPU 慢不一定是 GPU 算得慢。CPU 解码、采样、拼接、拷贝和日志同步都可能让 GPU 等数据。

把单个技巧当万能药

FlashAttention、torch.compile、activation checkpointing、FSDP 都有边界和代价。先定位瓶颈，再选工具。
