05 GPU 系统与性能瓶颈｜从零训练小语言模型

这一页接在训练循环之后，专门处理“训练能跑，但跑得慢、显存爆、扩卡不涨、数值发散”这些系统问题。你要把一次 step 拆成可测量的显存、吞吐、kernel、dtype、attention、通信和数据输入链路，而不是把所有现象都归因于 GPU 不够强。

学习目标

goal

先会记账

看到 OOM 时，先把显存拆成参数、梯度、优化器状态、激活值、临时 buffer、通信 buffer 和 allocator reserved。只看“模型参数多大”会漏掉训练峰值。

先会计时

看到训练慢时，先统一 tokens/s 的口径，再拆 step 时间。micro-batch、序列长度、累积步数和 GPU 数一变，吞吐比较就可能失真。

先会定位

优化前先拿证据：dataloader 等待、kernel 自身耗时、dtype 数值问题、attention 中间压力、NCCL 通信和 GPU 拓扑都要分开看。

本页边界

这里不教完整 CUDA 编程，也不展开分布式训练工程。你只需要学会把一次小语言模型训练 step 拆成可观察的系统账本；下一页再用这些指标进入 scaling、验证集和评估。

性能主线

map

性能排查要从一条训练 step 的时间线开始。CPU 准备 batch，数据被拷到 GPU，模型前向和反向启动一串 kernel，优化器更新参数，分布式场景还要同步梯度或参数分片。慢训练通常某段等待、读写或同步占了太多时间。

术语与误区

terms
术语直觉解释精确定义常见误解
peak memory训练过程中最满的时候。一次 step 内出现过的最高显存占用，常在 backward。只看 step 结束时的 allocated，会低估峰值。
tokens/s每秒处理多少 token。通常按全局 step 的有效 token 数除以 step_seconds。把 micro-step 当完整 step，会把数字算歪。
kernelGPU 上执行的一小段工作。由 CPU 发起、在 GPU 上并行执行的函数，可能只是模型层的一部分。把 kernel 当神经网络层，容易误读 profiler。
BF16 / FP16低精度训练的两种常见格式。BF16 保指数范围，FP16 更省位数；二者都要看数值稳定。只看速度，不看 loss scale 和 NaN，会翻车。
NCCL all-reduce多卡同步梯度。分布式训练里的 collective communication，用来把各卡梯度合并。通信慢，不一定是网络坏，也可能是每卡 batch 太小。
roofline看算力和带宽谁先顶住。用 FLOPs/bytes moved 的直觉判断更像 compute-bound 还是 memory-bound。把它当精确诊断器，会过度简化。

显存账本

memory

训练显存不等于模型文件大小。参数会占显存，梯度也会占显存；Adam 这类优化器还有 moments；反向传播要保留激活值；框架、通信库和 kernel workspace 也会临时要空间。OOM 排查的第一步是把账本列出来。

P 是参数个数；b_param、b_grad、b_opt 是每个参数对应的参数、梯度和优化器状态字节数。激活值和临时 buffer 会随 batch、序列长度、实现和 checkpoint 策略变化。

峰值在哪里

峰值常出现在 backward，而不是 step 结束。你要记录 max_memory_allocated、max_memory_reserved，再按 forward、backward、optimizer step 分段观察。

如果反向才 OOM，优先怀疑激活值、attention 中间张量、临时 workspace 或通信 buffer。

显存项什么时候出现受什么影响常用处理
参数模型加载后长期存在参数量、dtype、是否有 master weights更低精度、模型分片、缩小模型
梯度反向传播后存在，更新前后清理参数量、梯度累积、分片方式梯度分片、累积策略、及时 zero_grad
优化器状态第一次更新后长期存在AdamW、Adafactor、SGD 等优化器选择更省状态的优化器、ZeRO/FSDP 分片
激活值前向保存，反向读取micro-batch、seq_len、层数、checkpointing减小 micro-batch/seq_len、activation checkpointing
临时与通信 bufferkernel、attention、NCCL、allocator 运行时出现实现、拓扑、bucket、动态 shape固定形状、预留空间、调 bucket、换实现

检查动作报 OOM 时，不要只说“显存不够”。写出 batch、seq_len、dtype、参数量、优化器、是否 activation checkpointing，以及 allocated/reserved/peak 的数字。

吞吐口径

tokens/s

训练速度通常用 tokens/s 表示，但这个数字只有在口径一致时才有意义。你要说清楚统计的是单卡还是全局，是训练 tokens 还是有效更新 tokens，是否包含 dataloader、checkpoint 和评估时间。

这里的 step_seconds 应该是一次 optimizer update 的端到端时间。如果你用 micro-step 时间当分母，或者忘了 grad accumulation，比较就会偏。

代码
import time
import torch

def measure_step(train_step, warmup=5, repeat=20):
 for _ in range(warmup):
 train_step()
 torch.cuda.synchronize()
 start = time.perf_counter()
 for _ in range(repeat):
 train_step()
 torch.cuda.synchronize()
 return (time.perf_counter() - start) / repeat

# CPU 计时前后都要同步，否则可能只测到 kernel 提交时间。
# 真实报告还要写清 batch、seq_len、grad_accum、GPU 数和是否包含 dataloader。

检查动作每次改 batch、accum、GPU 数、dtype 或 dataloader 后，都重新记录同一套口径。吞吐变快但验证 loss 变差时，要回到训练语义，不把它当纯系统收益。

kernel 与 roofline

kernel

Python 里的一层 Transformer，到 GPU 上会变成很多 kernel：矩阵乘、softmax、mask、dropout、LayerNorm、elementwise、optimizer update。性能问题常来自两类：kernel 太碎，启动和内存读写占比高；或者大 kernel 没有喂满 Tensor Core，算力利用不足。

$$ arithmetic\ intensity = \frac{FLOPs}{bytes\ moved} $$

roofline 的直觉是：如果每搬一字节数据只做很少计算，瓶颈更像内存带宽；如果每搬一字节能做大量矩阵乘，瓶颈更可能是算力或 Tensor Core 利用。它不是精确诊断结论，但能提醒你看 HBM 往返和 kernel 形状。

代码
优化 kernel 前先问四个问题：
1. profiler 里时间是否真的花在这些 op 上？
2. 这些 op 是 compute-bound 还是 memory-bound？
3. 框架版本是否已经内置融合或高效实现？
4. 改实现以后，数值、mask、dropout、训练/评估模式是否仍然一致？

精度与 attention

dtype

混合精度的目标是让适合低精度的矩阵乘走快路径，同时守住容易不稳定的归约、softmax、loss 和归一化统计。dtype 改动会同时影响显存、吞吐和数值稳定，不能只看速度。

$$ scores\ shape \approx batch \times heads \times seq\_len \times seq\_len $$

标准 self-attention 会比较每个 token 和每个 token。序列长度翻倍，分数矩阵元素数会变成四倍。如果实现显式物化中间矩阵，显存和 HBM 读写都会变重。

多卡通信

parallel

多卡训练增加的是全局计算能力，也增加了同步成本。DDP 常见路径是每张卡算自己的 micro-batch，然后对梯度做 all-reduce；FSDP/ZeRO 会把参数、梯度或优化器状态分片，省显存但加入更多通信和调度约束。

$$ scaling\ efficiency = \frac{throughput_N}{N \times throughput_1} $$

这个指标只在训练口径一致时有意义。GPU 数变了以后，如果 micro-batch、累积步数、数据加载路径或 checkpoint 频率也变了，就要把这些变化写进报告。

profiler 诊断

profile

profiler 的作用把等待、计算、通信和内存峰值摆到同一张证据表里。一个稳的排查流程是：建立基线，确认口径；分段计时；看时间线空洞；按 self CUDA time 排 kernel；记录显存峰值；最后只改一个主要变量。

代码

from torch.profiler import profile, ProfilerActivity

with profile(
 activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
 record_shapes=True,
 profile_memory=True,
 with_stack=False,
) as prof:
 for step, batch in enumerate(loader):
 if step >= 20:
 break
 loss = train_one_step(batch)
 prof.step()

print(prof.key_averages().table(
 sort_by="self_cuda_time_total",
 row_limit=20,
))

常见错误

pitfalls

未同步就计时

CUDA 默认异步执行。CPU 计时如果前后不加 torch.cuda.synchronize()，很可能只测到提交任务的时间。

只看 allocated

reserved、峰值、非 PyTorch 分配、NCCL buffer 和临时 workspace 都可能影响 OOM。报告里只写 allocated 不够。

频繁 .item()

每步把 GPU tensor 拉回 CPU 可能触发同步，让时间线出现空洞。日志可以降频，必要指标批量记录。

把 batch 改动当纯系统优化

micro-batch、累积步数和 GPU 数改变后，有效 batch 也可能改变。训练曲线变化不能只按吞吐解释。

忽略数据输入

GPU 慢不一定是 GPU 算得慢。CPU 解码、采样、拼接、拷贝和日志同步都可能让 GPU 等数据。

把单个技巧当万能药

FlashAttention、torch.compile、activation checkpointing、FSDP 都有边界和代价。先定位瓶颈，再选工具。

自测答案方向

check

计算题方向

显存账本要写清每个项对应的字节数；tokens/s 要统一为一次 optimizer step 的口径；扩展效率用 throughput_N / (N × throughput_1)，不要把 micro-step 当成完整 step。

诊断题方向

优先写证据和首查点：时间线空洞先看 dataloader 和同步，NaN 先看首发 op 和 dtype，扩卡差先看 NCCL 和每卡 batch，反向 OOM 先看 activation peak。
