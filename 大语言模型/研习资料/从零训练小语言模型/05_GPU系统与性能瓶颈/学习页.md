# 术语与误区 {#terms}
<!-- tag: terms -->

术语
直觉解释
精确定义
常见误解


**peak memory**
训练过程中最满的时候。
一次 step 内出现过的最高显存占用，常在 backward。
只看 step 结束时的 allocated，会低估峰值。


**tokens/s**
每秒处理多少 token。
通常按全局 step 的有效 token 数除以 step_seconds。
把 micro-step 当完整 step，会把数字算歪。


**kernel**
GPU 上执行的一小段工作。
由 CPU 发起、在 GPU 上并行执行的函数，可能只是模型层的一部分。
把 kernel 当神经网络层，容易误读 profiler。


**BF16 / FP16**
低精度训练的两种常见格式。
BF16 保指数范围，FP16 更省位数；二者都要看数值稳定。
只看速度，不看 loss scale 和 NaN，会翻车。


**NCCL all-reduce**
多卡同步梯度。
分布式训练里的 collective communication，用来把各卡梯度合并。
通信慢，不一定是网络坏，也可能是每卡 batch 太小。


**roofline**
看算力和带宽谁先顶住。
用 FLOPs/bytes moved 的直觉判断更像 compute-bound 还是 memory-bound。
把它当精确诊断器，会过度简化。

# 显存账本 {#memory}
<!-- tag: memory -->

训练显存不等于模型文件大小。参数会占显存，梯度也会占显存；Adam 这类优化器还有 moments；反向传播要保留激活值；框架、通信库和 kernel workspace 也会临时要空间。OOM 排查的第一步是把账本列出来。

$$M_{train} \approx P(b_{param}+b_{grad}+b_{opt}) + M_{act} + M_{tmp} + M_{comm} + M_{reserved}$$
 `P` 是参数个数；`b_param`、`b_grad`、`b_opt` 是每个参数对应的参数、梯度和优化器状态字节数。激活值和临时 buffer 会随 batch、序列长度、实现和 checkpoint 策略变化。

## box soft
### 小算例
假设有 300M 参数，参数 BF16 约 2 bytes，梯度 BF16 约 2 bytes，Adam 两个 moments 用 FP32 共 8 bytes。不算激活值，每个参数约 12 bytes，参数态和优化器态已经接近 3.35 GiB。
如果还保留 FP32 master weights，就再加 4 bytes/parameter。这个额外项常被初学者漏掉。

## box info
### 峰值在哪里
峰值常出现在 backward，而不是 step 结束。你要记录 `max_memory_allocated`、`max_memory_reserved`，再按 forward、backward、optimizer step 分段观察。
如果反向才 OOM，优先怀疑激活值、attention 中间张量、临时 workspace 或通信 buffer。

# 吞吐口径 {#throughput}
<!-- tag: tokens/s -->

训练速度通常用 tokens/s 表示，但这个数字只有在口径一致时才有意义。你要说清楚统计的是单卡还是全局，是训练 tokens 还是有效更新 tokens，是否包含 dataloader、checkpoint 和评估时间。

$$tokens/s = \frac{n_{gpu} \times micro\_batch \times seq\_len \times grad\_accum}{step\_seconds}$$
 这里的 `step_seconds` 应该是一次 optimizer update 的端到端时间。如果你用 micro-step 时间当分母，或者忘了 grad accumulation，比较就会偏。

配置
计算
tokens/s
读法


1 GPU，micro=8，seq=1024，accum=4，step=2.0s
1 × 8 × 1024 × 4 / 2.0
16,384
这是全局训练吞吐，也是单卡吞吐。


4 GPU，micro=8，seq=1024，accum=4，step=2.4s
4 × 8 × 1024 × 4 / 2.4
54,613
比 1 卡快 3.33 倍，扩展效率约 83%。


4 GPU，micro=4，seq=1024，accum=4，step=1.5s
4 × 4 × 1024 × 4 / 1.5
43,691
step 变短但 tokens/s 变低，不能只看秒数。

timing_skeleton.py复制

```python
import time
import torch

def measure_step(train_step, warmup=5, repeat=20):
 for _ in range(warmup):
 train_step()
 torch.cuda.synchronize()
 start = time.perf_counter()
 for _ in range(repeat):
 train_step()
 torch.cuda.synchronize()
 return (time.perf_counter() - start) / repeat

# CPU 计时前后都要同步，否则可能只测到 kernel 提交时间。
# 真实报告还要写清 batch、seq_len、grad_accum、GPU 数和是否包含 dataloader。
```

**检查动作**每次改 batch、accum、GPU 数、dtype 或 dataloader 后，都重新记录同一套口径。吞吐变快但验证 loss 变差时，要回到训练语义，不把它当纯系统收益。

# kernel 与 roofline {#kernel}
<!-- tag: kernel -->

Python 里的一层 Transformer，到 GPU 上会变成很多 kernel：矩阵乘、softmax、mask、dropout、LayerNorm、elementwise、optimizer update。性能问题常来自两类：kernel 太碎，启动和内存读写占比高；或者大 kernel 没有喂满 Tensor Core，算力利用不足。

matmul 时间长但 GPU 利用率不高
shape 不利于 Tensor Core、dtype 或 layout 不合适
矩阵维度、dtype、是否走 FP16/BF16/TF32 路径
调整 hidden size 对齐、autocast、检查 contiguous/layout


softmax / attention 占比随序列长度暴涨
attention 中间矩阵和 HBM 读写压力大
seq_len、head 数、attention 实现、是否物化分数矩阵
FlashAttention、减小 seq_len、activation checkpointing

$$arithmetic\ intensity = \frac{FLOPs}{bytes\ moved}$$
 roofline 的直觉是：如果每搬一字节数据只做很少计算，瓶颈更像内存带宽；如果每搬一字节能做大量矩阵乘，瓶颈更可能是算力或 Tensor Core 利用。它不是精确诊断结论，但能提醒你看 HBM 往返和 kernel 形状。

# 精度与 attention {#precision}
<!-- tag: dtype -->

混合精度的目标是让适合低精度的矩阵乘走快路径，同时守住容易不稳定的归约、softmax、loss 和归一化统计。dtype 改动会同时影响显存、吞吐和数值稳定，不能只看速度。

$$scores\ shape \approx batch \times heads \times seq\_len \times seq\_len$$
 标准 self-attention 会比较每个 token 和每个 token。序列长度翻倍，分数矩阵元素数会变成四倍。如果实现显式物化中间矩阵，显存和 HBM 读写都会变重。

## box warn
### 边界
FlashAttention 主要解决 attention 的 IO 和中间显存问题。它不会修复 dataloader 等待、optimizer state 太大、NCCL 通信慢或训练超参错误。

# 多卡通信 {#communication}
<!-- tag: parallel -->

多卡训练增加的是全局计算能力，也增加了同步成本。DDP 常见路径是每张卡算自己的 micro-batch，然后对梯度做 all-reduce；FSDP/ZeRO 会把参数、梯度或优化器状态分片，省显存但加入更多通信和调度约束。

可能原因
证据
动作与代价

通信占比高、每卡 batch 太小、dataloader 等待
NCCL 时间、timeline 空洞、每卡 tokens 数
增大每次同步前计算量；代价是有效 batch 或显存变化


DDP 显存仍然太高
每卡都保留完整参数、梯度和 optimizer state
每卡 memory summary 与参数态估算对不上
考虑 FSDP/ZeRO；代价是通信和实现复杂度增加


跨节点后吞吐掉很多
互联带宽和延迟变差，拓扑不匹配
NCCL 日志、节点内/节点间对比、拓扑图
优先节点内通信、调整并行策略；代价是资源调度限制

$$scaling\ efficiency = \frac{throughput_N}{N \times throughput_1}$$
 这个指标只在训练口径一致时有意义。GPU 数变了以后，如果 micro-batch、累积步数、数据加载路径或 checkpoint 频率也变了，就要把这些变化写进报告。

## box warn
### 小模型的多卡陷阱
模型太小或每卡 batch 太小时，通信税会显得很重。小规模实验上看到扩展效率差，不必直接推断“大模型也会差”；你要先确认计算量是否足够覆盖同步成本。

# profiler 诊断 {#profiler}
<!-- tag: profile -->

profiler 的作用把等待、计算、通信和内存峰值摆到同一张证据表里。一个稳的排查流程是：建立基线，确认口径；分段计时；看时间线空洞；按 self CUDA time 排 kernel；记录显存峰值；最后只改一个主要变量。

profiler_minimal.py复制

```python
import torch
from torch.profiler import profile, ProfilerActivity

with profile(
 activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
 record_shapes=True,
 profile_memory=True,
 with_stack=False,
) as prof:
 for step, batch in enumerate(loader):
 if step >= 20:
 break
 loss = train_one_step(batch)
 prof.step()

print(prof.key_averages().table(
 sort_by="self_cuda_time_total",
 row_limit=20,
))

# 看表之前先确认：warmup 是否完成，是否包含 dataloader，
# CUDA 计时是否同步，batch/seq_len/dtype/GPU 数是否写进报告。
```

读表步骤
要看什么
常见判断


1. 时间线
GPU 是否有长空洞，CPU 和 GPU 是否错位
空洞常指向 dataloader、CPU 同步或 H2D 拷贝。


2. self CUDA time
哪个 kernel 真正占 GPU 时间
matmul、attention、norm、optimizer update 要分开看。


3. shape
慢 kernel 对应的输入形状
小 shape 可能启动开销重，大 shape 可能算力或带宽重。


4. memory
allocated、reserved、peak 和分段峰值
reserved 高不等于泄漏，可能是缓存、碎片或临时 workspace。


5. 复测
只改一个变量后的同口径指标
优化动作要同时报告 tokens/s、显存和验证 loss。

**检查动作**把 profiler 结果写成“现象 → 证据 → 判断 → 改动 → 复测”。如果只有“用了某个优化技巧”，这份报告还不能说明问题真的被定位了。

# 常见错误 {#mistakes}
<!-- tag: pitfalls -->
<!-- layout: grid2 -->

## box warn
### 未同步就计时
CUDA 默认异步执行。CPU 计时如果前后不加 `torch.cuda.synchronize()`，很可能只测到提交任务的时间。

## box warn
### 只看 allocated
`reserved`、峰值、非 PyTorch 分配、NCCL buffer 和临时 workspace 都可能影响 OOM。报告里只写 allocated 不够。

## box warn
### 频繁 `.item()`
每步把 GPU tensor 拉回 CPU 可能触发同步，让时间线出现空洞。日志可以降频，必要指标批量记录。

## box warn
### 把 batch 改动当纯系统优化
micro-batch、累积步数和 GPU 数改变后，有效 batch 也可能改变。训练曲线变化不能只按吞吐解释。

## box warn
### 忽略数据输入
GPU 慢不一定是 GPU 算得慢。CPU 解码、采样、拼接、拷贝和日志同步都可能让 GPU 等数据。

## box warn
### 把单个技巧当万能药
FlashAttention、torch.compile、activation checkpointing、FSDP 都有边界和代价。先定位瓶颈，再选工具。

**Q:** Q7. 解释为什么 seq_len 从 1024 改到 4096 后，attention 可能突然成为显存和时间瓶颈。

Explain: 好答案会指出 attention 分数矩阵随 seq_len² 增长：1024 到 4096 是长度 4 倍，分数元素约 16 倍。若实现物化中间矩阵，显存和 HBM 读写都会上升。可查 attention 实现、head 数、activation checkpointing、FlashAttention 和 micro-batch；同时说明 FlashAttention 不处理 dataloader 或通信瓶颈。

### 12 diagnosis

**Q:** Q12. 你接手一个小模型训练：loss 正常下降，但 tokens/s 只有预期的一半。写一份排查顺序。

Explain: 好答案会先固定口径并同步计时，确认是否包含 dataloader 和 checkpoint；随后看时间线空洞，排查 CPU 解码、H2D copy 和 .item() ；再按 self CUDA time 找 matmul、attention、norm、optimizer 等热点；同时记录 allocated/reserved/peak 和 dtype；多卡时补查 NCCL 与每卡 batch。每次只改一个变量，复测 tokens/s、显存和验证 loss。

### 13 synthesis

**Q:** Q13. 设计一次“打开 BF16 + FlashAttention”的对比实验，说明你会记录哪些字段，怎么判断这次改动有效。

Explain: 好答案会保留一个 FP32 或原 dtype baseline，固定数据、batch、seq_len、accum、GPU 数和训练步数；记录 tokens/s、step time、peak allocated/reserved、验证 loss、NaN/grad norm、dtype、CUDA/PyTorch/FlashAttention 版本和硬件型号。判断有效要同时满足吞吐或显存改善、loss 曲线无异常、数值稳定，并说明改动只覆盖 attention 和矩阵乘路径，不代表 dataloader 或通信已优化。

### 14 synthesis

**Q:** Q14. 用一段话讲完整个单元。

Explain: 好的解释会把训练 step 说成一条链：输入先到 GPU，前后向启动 kernel，优化器改参数，通信负责跨卡同步，dtype 决定数值与吞吐，显存和 attention 中间张量决定能不能放下。慢训练先看时间线和显存，再看 kernel 形态和 dtype，再看通信和数据输入；不要一上来就换 GPU 或盲开技巧。
