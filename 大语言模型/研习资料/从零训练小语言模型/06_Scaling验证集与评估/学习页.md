# 术语与误区 {#terms}
<!-- tag: Vocabulary -->

术语
直觉解释
精确定义
常见误解


**训练 loss**
模型在刚见过或正在学习的数据上犯错多少。
通常是每个 token 的平均交叉熵，记录在训练 batch 上。
训练 loss 下降不等于泛化变好；它可能只是记住训练集或数据更简单。


**验证 loss**
模型在没参与参数更新的数据上犯错多少。
固定 held-out 数据上的平均交叉熵，用于调参、选 checkpoint 和判断泛化。
反复用验证集选模型，会让验证集也被间接训练。


**困惑度 PPL**
平均每步像是在多少个等可能 token 中做选择。
若 loss 使用自然对数，PPL = exp(loss)。
不同 tokenizer、不同文本分布上的 PPL 不能直接比较。


**测试集**
最后一次验收用的封存数据。
不用于日常调参，不用于选择 checkpoint，只在报告最终结论时使用。
把测试集看很多次再改模型，会把测试集变成验证集。


**Scaling law**
规模变大时，loss 往往按平滑规律下降。
经验性幂律关系，常考察参数量、训练 tokens、算力和 loss 的关系。
它不是自然定律；数据质量、架构、优化器和训练配方都会改趋势。


**污染 contamination**
考试题提前出现在训练或调参材料里。
评估样本、答案、近重复、解析或任务模板泄漏到训练、验证、提示或调参流程中。
污染不只等于完全重复；近重复、答案泄漏和多次调榜也会抬高分数。

# Loss 曲线：先看过程是否可信 {#loss}
<!-- tag: Curve Reading -->
<!-- layout: grid2 -->

## box
**直觉：**loss 曲线是训练过程的体温计。它不告诉你模型会不会写代码、会不会推理，但能告诉你训练有没有明显发烧：学习率太大、数据断流、评估口径变了、过拟合或还没学够。
**术语：**语言模型训练常用交叉熵 loss。对一个序列，模型在每个位置给下一个 token 一个概率；如果正确 token 的概率越高，负对数概率越低，loss 越低。
$$CrossEntropy = - \mathrm{mean}_{tokens}\log p(x_t\mid x_{这里的意思很简单：训练时先看平均交叉熵，再把它换成 perplexity 时，要知道两者本质上是同一个信号。
训练时至少记录四条信息：训练 loss、验证 loss、学习率、有效 token 数。只记录 step 编号，会让不同 batch、不同 accumulation、不同 GPU 数的实验难以比较。

## box warn
### 读曲线的顺序
1. 先确认 x 轴是 step、token 还是 wall-clock。
2. 看训练 loss 是否平滑下降，有没有尖峰、平台或突然断崖。
3. 看验证 loss 是否同步下降，是否开始反弹。
4. 把曲线变化和学习率、数据 shard、checkpoint 恢复、batch 改动对齐。
5. 最后才比较不同模型的最终 loss。

观察
更可能的含义
先检查什么
不要急着做什么


训练 loss 降，验证 loss 升
过拟合、验证集太小或训练 / 验证分布不一致。
验证集大小、去重、分域 loss、checkpoint 选择频率。
不要只因为训练 loss 好看就继续加 epoch。


训练和验证 loss 都不降
欠拟合、学习率不合适、模型太小、数据管线错误。
梯度是否非零、学习率曲线、标签 shift、batch 内容。
不要直接归因于“模型能力不足”。


loss 周期性尖峰
坏数据 shard、学习率调度、数值溢出、恢复 checkpoint 问题。
尖峰 step 对应的数据、dtype、梯度范数、NaN/Inf。
不要只做滑动平均把问题抹平。


验证 loss 突然大幅变好
可能是真进步，也可能是评估脚本、tokenizer 或验证集被换了。
eval commit、数据 hash、tokenizer 版本、prompt 模板。
不要立刻宣布突破。

# 验证集：不是随便留一小撮数据 {#validation}
<!-- tag: Held-out Data -->
<!-- layout: grid2 -->

## box soft
### 验证集解决什么问题
训练集像练习册，验证集像阶段小测。练习册分数提高说明你熟悉练习题，小测分数提高才更像学到了可迁移规律。
验证集是从参数更新中隔离出来的数据，用于选择 checkpoint、比较超参和早停。它不该参与训练梯度，也不该被数据清洗规则反复“优化”。

## box info
### 预训练验证集要分域
语言模型预训练数据通常混合网页、代码、百科、论文、对话和书籍。一个总验证 loss 可能掩盖分布差异：总 loss 下降时，代码域可能变好，新闻域可能变差，低资源语言可能几乎没学到。
最小做法是保留一个总验证集，再按来源或任务类型建立若干小验证集。每次评估同时报告总 loss 和分域 loss，这能帮你判断数据配方改变到底影响了哪里。

步骤
为什么要做
如果跳过会怎样


先文档级去重，再切分训练 / 验证 / 测试
减少近重复样本泄漏。
同一篇文章可能同时出现在训练和验证里。


总验证集 + 分域验证集
看清楚哪些域变好，哪些域变差。
总 loss 下降时，局部退化会被藏起来。


开发 / 封存 / 测试三层
让调参和最终比较分开。
验证集会被人反复看坏，最后不再可信。


固定 hash、tokenizer、eval batch 和随机种子
让每次评估可复核。
你看到的改进可能只是协议漂移。

## box warn
最低配置
至少保留：固定数据 hash、固定 tokenizer、固定 eval batch、固定随机种子、文档级隔离、总 loss、分域 loss、每次评估的代码版本。否则你看到的变化可能来自评估过程，而不是模型。

# Scaling law 直觉：规模趋势是预算工具 {#scaling}
<!-- tag: Scaling -->
<!-- layout: grid2 -->

## box
**直觉：**在很多语言模型实验里，参数更多、训练 tokens 更多、算力更多时，验证 loss 会按相对平滑的方式下降。把横轴和纵轴都取对数，许多曲线会接近直线，这就是幂律的直观图像。
**术语：**Scaling law 是经验规律，不是保证。它依赖数据分布、模型族、优化配方、tokenizer、训练长度和评估集。你可以用它规划实验，但不能把小样本拟合结果当成大模型承诺。
$$L(N,D)\approx L_{\infty}+A\cdot N^{-\alpha}+B\cdot D^{-\beta}$$
 $$C_{\mathrm{dense}}\approx 6\cdot N\cdot D$$
 这里的 N 是非 embedding 近似参数量，D 是训练 token 数，C 是粗略训练 FLOPs。公式只表达方向：参数和数据都能降低 loss，但边际收益递减。

## box
### 不要这样理解 scaling law
- 不要把单次训练曲线外推到 100 倍规模。
- 不要只增大模型、不增加数据，然后期待无限变好。
- 不要忽视数据质量；更多低质重复数据可能破坏趋势。
- 不要把 benchmark 分数和预训练 loss 的 scaling 混成一件事。

实验类型
目的
固定什么
改变什么
看什么结论


模型 scaling
看同一数据预算下，模型变大是否带来更低验证 loss。
数据配方、tokenizer、训练 tokens、优化器家族、评估集。
参数量、层数、宽度或 head 数。
loss 是否按 log-log 趋势下降，是否出现欠训练。


数据 scaling
看同一模型下，更多 token 是否继续带来收益。
模型、学习率策略、数据质量规则、评估集。
训练 token 数或 epoch 数。
验证 loss 是否继续下降，重复数据是否导致过拟合。


算力最优
在固定 FLOPs 预算内分配模型大小和训练 tokens。
硬件、训练代码、评估协议、候选模型族。
N 与 D 的组合。
哪组配置在同等算力下验证 loss 更低。

# 评估协议：loss 之外还要看任务行为 {#eval}
<!-- tag: Evaluation -->
<!-- layout: grid -->

## box soft
### 内在评估
验证 loss、perplexity、分域 loss。优点是便宜、稳定、适合训练中频繁运行；缺点是不直接等于用户任务能力。

## box info
### 下游 benchmark
问答、常识、数学、代码、阅读理解、多语言等。优点是更接近能力展示；缺点是容易受 prompt、解码和污染影响。

## box warn
### 人工与部署指标
偏好、事实性、安全性、延迟、成本、失败案例。优点是贴近使用；缺点是昂贵且主观，需要严格协议。

## box
### 评估不是一个数字
一个模型可能验证 loss 更低，但数学题更差；也可能 benchmark 准确率更高，但输出校准、鲁棒性或效率更差。多指标评估的意义是暴露取舍，而不是寻找一个万能分数。
给一个小语言模型评估表时，至少列出 accuracy、loss、校准、鲁棒性、效率和污染检查。

## box
### Prompt 和解码参数也是评估的一部分
同一个 benchmark，zero-shot、few-shot、chain-of-thought、温度、top-p、最大输出长度、停止词和评分脚本都会改变结果。没有固定协议的分数不能严肃比较。
报告时要写明 prompt 模板、示例选择、随机种子、解码参数、后处理规则和评估脚本版本。对于生成式任务，还要说明多次采样、pass@k 或人工评分规则。

# 污染与泄漏：分数太好时先怀疑边界 {#contamination}
<!-- tag: Contamination -->
<!-- layout: grid2 -->

## box
**直觉：**污染就是考试题或答案提前进了复习材料。语言模型训练语料来自大规模网页，公开 benchmark、题解、答案解析、排行榜讨论都可能混进去。
**更准确地说：**污染不只包括完全重复的测试样本，还包括近重复、答案字段、解析文本、题目模板、few-shot 示例、调参时反复看测试集，以及模型开发者根据 benchmark 错题改数据。
污染最危险的地方是它会把“记忆或调榜”伪装成“能力”。更麻烦的是，污染不一定总让分数明显升高；有些污染影响很小，有些只影响特定任务或大模型。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

风险类型
例子
可能后果
缓解方法


训练污染
训练网页中出现 benchmark 原题和答案解析。
benchmark 分数高估，模型未必能泛化到新题。
hash / 近重复过滤；保留数据快照；报告污染分析。


验证污染
根据验证集表现反复修改数据过滤规则。
验证 loss 对未来数据不再可靠。
开发验证集和封存验证集分层；限制查看频率。


提示泄漏
few-shot 示例与测试样本同源，或包含答案模式。
模型学会格式而非任务能力。
固定示例池；按任务来源隔离；报告 prompt。


时间泄漏
用训练截止日期之后发布的题目做“未污染”判断，却没有记录语料时间。
无法证明测试晚于训练。
记录数据时间戳；使用动态或时间敏感评估。


调榜泄漏
多次提交公开 leaderboard 后按榜单反馈改模型。
测试集逐渐变成验证集。
保留私有最终测试集；限制提交次数；报告选择过程。

## box warn
### 污染与背景知识的区别
训练集中出现某篇百科文章，不一定等于某个阅读理解题被污染；如果问题、答案和具体标注没有泄漏，模型可能只是拥有背景知识。污染判断要看样本、答案、任务模板和评估协议是否足以让模型“背题”。

# 下一步 {#bridge}
<!-- tag: Bridge -->

这一页先把“结果能不能信”讲清楚。下一页会把这些判断接到公开训练配方和开放报告上：你要继续看数据、超参、评估和阶段记录怎么一起披露，而不是只盯着一个验证分数。

这里补一个最常见的协议错误：同一个模型在两次 benchmark 上分数不同，原因可能prompt 模板、few-shot 示例、温度和停止词换了。只要这几个条件变了，分数就不再是同一把尺子。

本页留下的证据
下一页怎么用
如果缺失会怎样


训练 loss、验证 loss、PPL
判断公开训练报告里哪些数值能比较，哪些只是过程信号。
只能看最终分数，容易把训练过程和泛化结论混在一起。


验证集分域、切分顺序、评估协议
核对下一页公开配方里的数据说明和评估设置。
很难判断一个更好分数来自模型、数据还是口径变化。


污染检查与报告边界
继续看训练记录、版本和开放材料时，先问结论能不能复核。
benchmark 分数再高，也可能只是记住了题目或调榜过程。

失败后果
如果你把协议差异当成模型改进，后面会继续错下去：数据配方会越改越乱，排行榜会越看越高，但最终模型在真实任务里的行为还是不稳。先把协议写清，后面才有资格谈 scaling 或配方改进。

# 自测答案方向 {#check}
<!-- tag: Check -->

练习 1-4 检查点
练习 1 选 B；练习 2 为错；练习 3 约为 exp(2.30)≈10；练习 4 应覆盖数据 shard、学习率调度、梯度范数、数值异常、checkpoint 恢复、评估脚本和日志同步，并说明如何复测。

练习 5-7 检查点
练习 5 选 B；练习 6 为错；验证集方案应包含文档级隔离、固定 hash、分域 loss、开发 / 封存 / 测试分层；每天根据封存验证集改规则会造成调参泄漏。

练习 8-10 检查点
练习 8 中 A 与 B 的 N×D 都是 1000M·B，粗略 FLOPs 相同；比较参数量时应固定数据配方、训练 tokens、tokenizer、评估集和优化器；过度外推风险来自点数少、规模跨度大、数据 / 优化 / 架构变化和置信度不足。

练习 11-14 检查点
练习 11 选 D；练习 12 为错；污染排查应包含原题 / 答案 / 解析重叠、近重复、时间边界、prompt 示例、调榜记录、干净子集或改写测试；评估结论要写清协议和不能外推的范围。
