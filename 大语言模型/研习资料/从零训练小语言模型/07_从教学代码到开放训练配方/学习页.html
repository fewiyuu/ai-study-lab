训练配方

从教学代码到开放训练配方

这页不教你再跑一次训练，而是教你看懂一份公开训练报告：哪一段是在证明代码链路跑通，哪一段是在证明数据和评估可审计，哪一段只是下载说明。读完后，你应该能把一次 run 拆成数据、超参、评估、阶段记录和产物链路五块。

读完本页应该能做什么

  • 能区分 toy run、open weights 和 fully open training。
  • 能从报告里抽出数据配方、超参账本、评估协议和 checkpoint 谱系。
  • 能写出一份小模型训练报告骨架，并补上不能外推的边界。

学习目标

goal

这页的目标不是帮你写一篇发布文案，而是把一份公开训练材料拆成可检查的证据。每一类证据都回答不同的问题：这条 run 是否跑通、这份数据从哪来、训练设置是否够完整、评估协议是否站得住、最终模型来自哪些阶段、哪些结论只能保守地说。

读懂层级

能把 toy run、open weights、fully open training 分开说清，不把“能下载”误读成“可审计”。

抽出证据

能从数据、超参、评估、阶段记录和产物链路里抽出一份能回查的清单。

写出边界

能在报告结尾明确写出可靠结论和不能外推的部分，避免把一次 run 说成全部结论。

看到什么 先问什么 为什么要问
只有最终权重和一张截图 数据、超参、评估协议在哪里 这只说明结果存在，不说明结果可查。
有配置但没有数据版本 数据来源、混合比例、过滤和去重是否记录 数据变了，训练结论也会跟着变。
有分数但没有评估说明 验证集、prompt、shot、解码参数和污染检查在哪里 没有协议的分数，很难比较。
有 stage 名称但没有 checkpoint 谱系 起止 checkpoint、token budget 和选择规则在哪里 最终模型通常来自多阶段决策。
先知道，先别钻 这页先不追 benchmark 排名、数据清洗流水线细节和完整 scaling fit；它只负责把报告里的证据链理顺。

证据链

trace

把一份公开训练材料看成一条证据链，先看每一步读了什么，再看每一步写出了什么。前一步写的内容，会决定后一步有没有资格下结论。把鼠标移到左侧步骤上，你会看到证据从“原始 run”变成“可审计报告”的过程。

原始 run 一串训练日志和最终截图

读到的是 step、loss、吞吐、NaN、OOM；写出去的是“这次执行发生了什么”。

数据卡 来源、token 数、混合比例、过滤、去重

读到的是样本构成；写出去的是“模型到底从什么文本里学到什么”。

配置 layers、hidden、heads、LR、batch、dtype

读到的是训练设置；写出去的是“为什么这个 run 的动态长这样”。

评估 验证集、benchmark、污染检查、样例

读到的是协议；写出去的是“这个分数该怎么解释”。

产物包 权重、代码、日志、checkpoint、模型卡

读到的是前面的证据；写出去的是“别人能否按同一条链路复查”。

读法 先看每一步“读了什么”，再看“写出了什么”。如果一环只给结论，不给来源，后面的判断就会变轻。

术语误区

vocabulary

先把几个最容易混在一起的词拆开，后面写计划和看报告才不会乱。这里的重点不是记名词，而是知道每个词对应哪一层证据。

toy run

教学用的小规模训练，主要验证代码路径、loss 下降、恢复和采样是否正常。

recipe

训练配方。它不是单个 YAML，而是数据、模型、优化、系统、评估和阶段决策的组合说明。

open weights

只发布最终权重或少量模型文件。它能支持推理或微调，但未必能审计数据和训练决策。

fully open

更高层级的开放：权重、训练数据、代码、配置、日志、评估和中间检查点尽量可访问。

data mix

数据混合比例。它回答“每类数据在训练中占多少”，不是简单列数据集名字。

checkpoint lineage

检查点谱系。它说明最终模型来自哪些中间点，经历了哪些继续训练、筛选或回退。

contamination

评估污染。训练数据中如果包含测试题或近似答案，benchmark 分数会被抬高。

reproducibility boundary

可复现边界。它告诉你哪些结论能重现，哪些只能近似，哪些不能外推。

先别追的旁支 先不要把注意力放在 leaderboard 排名、数据集命名争议，或者完整 benchmark 作者学术史上；这页只处理“报告里该写什么”。

记录账本

ledger

训练配方不是一份配置文件，而是一组能互相对上的账本。你只要能从记录里复原有效 batch、token budget、评估协议和阶段边界，这份报告就已经比“只给一个 config”更接近可审计。

$$ B_{\text{eff}} = B_{\text{micro}} \times N_{\text{GPU}} \times A_{\text{acc}} \qquad C_{\text{rough}} \approx \frac{T_{\text{tokens}}}{r_{\text{tok/s}} \times 3600} \times p_{\text{gpu}} $$

B_micro 是每卡 micro-batch，N_GPU 是 GPU 数，A_acc 是梯度累积步数。T_tokens 是总 token 数，r_tok/s 是吞吐，p_gpu 是每小时 GPU 单价。这个估算只给数量级，不替代真实 profiler。

账本 要写什么 漏掉后会怎样
数据账 来源、版本、token 数、混合比例、过滤、去重、许可证、污染检查。 别人知道你用了什么名字，却不知道你到底喂给了模型什么文本。
模型账 layers、hidden、heads、context、vocab、位置编码、norm、初始化。 参数量有了，结构没有，结果就很难复查。
优化账 optimizer、学习率计划、warmup、weight decay、clip、effective batch。 训练动态会变，但你看不出为什么变。
系统账 GPU、dtype、并行策略、checkpointing、吞吐、显存和依赖版本。 你只能说“跑过了”，不能说“为什么只跑到这个速度”。
评估账 验证集大小、评估间隔、prompt、shot、解码参数、污染检查、失败样例。 分数会看起来很好，但比较口径不清。
结论账 哪些发现稳、哪些退步、哪些结论不能外推。 报告会把一次 run 说得比它真正支持的更多。
手感 你先把“有效 batch”和“总训练量”写对，再谈结果是不是公平。只写 micro-batch 不够，单写 GPU 型号也不够。

阶段记录

state

真实训练经常分阶段：先做 sanity run，确认链路稳定；再做 base pretraining，建立主曲线；最后进入 annealing 或继续训练，收束到更高质量的数据和更谨慎的学习率。阶段记录要把起止 checkpoint、token budget、数据切换和选择规则写成能回头查的形式。

阶段 读什么 写什么 写错后会怎样
Stage 0
sanity run		 代码路径、loss、恢复、采样、显存。 确认训练链路没断，先排查 NaN、OOM 和错误 shape。 你会把实现问题误当成数据问题。
Stage 1
base pretraining		 通用数据、学习率计划、验证 loss、吞吐。 记录起点 checkpoint、token budget、有效 batch 和评估间隔。 后面看不到主曲线从哪里开始变稳。
Stage 2
annealing / continued training		 高质量数据、数据切换、退火策略、污染检查。 记录数据混合、学习率变化、结束 checkpoint 和选择依据。 最终模型像是“一步到位”，其实是阶段结果。
阶段记录模板
stage:
  name: base_pretraining
  start_checkpoint: step_0
  end_checkpoint: step_180000
  token_budget: 300B
  data_mix:
    web: 60%
    code: 20%
    math: 20%
  lr_schedule: warmup + cosine
  eval_every: 2000 steps
  selection_rule: lowest validation loss with stable contamination check
边界提示 这里先不展开完整 scaling-law 拟合和阶段预算推导；这页只需要把 stage 记录写成能回头找的形式。

产物链路

artifact

开放训练不是只交一个权重文件。更稳的做法是把产物排成一条可追踪的链：模型卡指向权重，权重指向配置，配置指向训练代码，日志指向 checkpoint，评估表指向脚本和数据版本。只要一环断了，别人就很难复查。

链路 应该对上什么 常见断点
模型卡 → 权重 版本号、下载地址、适用范围。 卡片更新了，文件还是旧版。
权重 → 配置 layers、hidden、context、tokenizer、dtype、seed。 只给权重，不给结构。
配置 → 训练代码 启动脚本、参数解析、依赖版本、训练入口。 人能看见参数，不能复跑。
日志 → checkpoint step、stage、token budget、恢复点。 只保留最终曲线，断掉阶段谱系。
评估表 → 脚本和数据版本 prompt、shot、解码参数、验证集与污染检查。 分数看得见，协议找不到。 
toy run 只证明训练链路跑通。你还不知道数据版本、评估协议和产物链路是否完整。
最小开放包 小团队不一定能公开所有原始数据，但至少可以公开：训练脚本、配置、数据来源清单、处理脚本摘要、token 统计、loss 曲线、评估脚本、最终权重、若干中间 checkpoint 和训练报告。

报告断点实验室

failure lab

这一小节把“看报告”变成“排查报告”。假设你拿到三种常见材料：只给权重、只给分数、或者给了一部分配置和日志。你要先判断哪一环断了，再决定下一步追问什么。这里不追华丽排版，只追证据链是否闭合。

只给权重时，你能下载模型，但还不知道数据、配置、评估和 stage 是否完整。
看到的症状 最可能缺的证据 下一步先问什么 如果不补会怎样
只有下载按钮和权重文件 数据账、超参账、stage 记录、评估协议。 数据从哪来，训练怎么配，分数怎么算。 你只能说“有模型”，不能说“模型怎么来的”。
有 MMLU 分数，但没说明 prompt 和 shot 评估协议。 版本、prompt、shot、解码参数、污染检查。 分数看起来高，但没法横向比较。
有 stage 名字，却不知道前后 checkpoint checkpoint 谱系。 起止 checkpoint、token budget、数据切换和选择依据。 最终模型像是一个黑盒，不像一个阶段化结果。
有配置，但没有依赖版本和 seed 环境账和随机性账。 框架版本、dtype、并行策略、seed 和恢复点。 同一份配置，别人跑出来的曲线可能完全不同。
证据链审计口诀
先看数据从哪来，再看训练怎么配；
先看 stage 从哪起，再看 checkpoint 怎么收；
先看分数按什么协议算，再看结果能不能比；
先看产物互相能不能对上，再决定要不要相信这份报告。
失败后要做什么 如果材料只剩权重或只剩分数，不要先补修辞；先补协议和版本。修辞只能让页面看起来像报告，不能让报告真的可审计。

toy run 对照

compare

同样是“训练模型”，toy run 和公开训练报告要回答的问题完全不同。前者先问链路有没有 bug，后者先问结论是否可审计、可查、可复现到合理程度。

维度 toy run 常见写法 公开训练报告应补齐
目标 “跑通 GPT 训练，loss 会下降。” 模型目标、预算、成功标准、不能外推的边界。
数据 “用了 TinyShakespeare / 一个小数据集。” 来源、版本、token 数、混合比例、过滤、去重、污染检查、许可证。
超参 只写 batch、learning rate、step 数。 模型结构、optimizer、LR schedule、warmup、dtype、并行、有效 batch、seed、硬件。
评估 看训练 loss、验证 loss、生成几个样例。 验证集定义、benchmark 协议、污染检查、失败任务、样例选择规则。
阶段 一个命令从头跑到尾。 stage 划分、数据切换、退火、checkpoint 谱系、异常恢复、最终选择依据。
产物 代码片段、最终 loss、少量截图。 权重、配置、训练脚本、日志、评估脚本、模型卡、中间 checkpoint。
关键判断 toy run 只回答“这条训练链路有没有 bug”；公开训练报告还要回答“这个模型结果是否可信、可查、可复查”。两者不能互相代替。

一页报告怎么读

example

下面用一个教学示例演示读法。这个片段是虚构的，只用来练习“先找证据，再下结论”的顺序，不代表某个真实项目。你要看的不是数字本身，而是数字后面有没有协议、版本和链路。

教学示例：报告片段
# 教学示例：小语言模型训练片段

## 数据配方
- sources: cleaned web 55%, code 25%, math 20%
- snapshot: 2026-06-21
- token_count: 320B
- filtering: length > 64, dedup, contamination scan

## 训练设置
- model: 120M / 12 layers / 768 hidden / 12 heads / 1024 context
- optimizer: AdamW, lr 3e-4, warmup 2k, cosine decay
- batch: micro 8 x 4 GPU x accum 4 = 128 effective batch

## 评估
- valid loss: 2.31
- contamination: pass on held-out set
- benchmark: report only, protocol separate
片段里的线 它支持什么结论 还缺什么 下一步追问
数据配方、snapshot、token_count、filtering 这次 run 有明确的数据来源和处理痕迹。 具体数据版本、许可证、样本抽查和切分种子。 能否把同一版数据再找回来，是否会碰到污染。
model、optimizer、batch 训练设置足够判断有效 batch 和模型规模。 依赖版本、dtype、并行策略、恢复点和 seed。 如果训练动态变了，问题更像模型、优化还是系统。
valid loss、contamination、benchmark protocol separate 作者知道验证损失和基准协议不能混着说。 完整 benchmark 协议、任务级结果、失败样例和选择依据。 分数是否可信，是否能和别的报告比较。
这一页能支持什么 它支持你判断“这次 run 的骨架写出来了”；它还不能单独证明可复现、可比较和 fully open。要把这些结论补齐，还需要日志、脚本、数据版本和 checkpoint 谱系。

复现边界

boundary

好报告不是假装一切都能复现，而是诚实说明哪些部分可复现、哪些部分只能近似、哪些结论不能外推。边界写清楚以后，别人更容易相信你的结果。

这一页能帮助你写什么 这一页先不追什么 下一步应该补什么
训练报告的证据链、数据账、超参账、阶段记录和产物链路。 完整 benchmark 实现、数据清洗流水线和 scaling-law 拟合细节。 把自己手头的小模型材料填进模板，再检查哪一环缺证据。

下一步

bridge

把 07 的字段带到 08：问题、预算、数据、模型、指标和停机线。你不用先追“完整方案”，先把这几个字段写成自己的版本，下一页就能直接落成实验计划。

问题

先写一句话实验问题：你要验证什么关系，控制了什么变量。

预算

把 GPU 小时、token 数、重跑次数和人工排查时间写成上限。

数据

列出来源、版本、过滤、去重、切分和污染检查，不要只写数据集名称。

边界

把这次实验不能外推的地方提前写出来，下一页就能更快收束。

只做一件事 打开 08 页之前，先把你手头那次小实验的四个字段写下来：问题、预算、数据、模型/指标。写完再继续，速度会更快。

练习

diagnostic

怎么答才算会

先分层级，再补证据，最后写边界。你不需要把报告写得很长，但每个答案都要能指到具体字段、协议或产物。

检查重点
