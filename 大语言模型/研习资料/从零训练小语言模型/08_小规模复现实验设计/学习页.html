复现计划

小规模复现实验设计

这一页把前面的训练、评估和开放配方判断方法收束成一份能执行、能失败、能复盘的实验计划。你不需要复现大模型规模，但要把问题、预算、数据、模型、指标和停机线写在同一份记录里。

上一节：从教学代码到开放训练配方 课程地图

读完本页，你可以先写出

  • 一条可比较的问题句，说明固定什么、改变什么、观察什么。
  • 一份预算上限，知道什么时候该停、什么时候还能继续。
  • 一份能回查的实验记录，能分清主指标、诊断指标和边界。

学习目标

goal
先定问题

把“训练一个模型”改写成一条能比较的实验问题，写清固定项、变化项和主指标。

再定预算

把 GPU 小时、token 上限、重跑次数和人工排查时间写成硬边界。

最后定边界

先写停机线，再写能说什么、不能说什么，避免把一次小实验说成普遍结论。

看到什么 先补什么 为什么要补
“训练一个模型” 固定项、变化项、主指标 问题太大，结果就不可比较。
“大概跑两天” GPU 小时、token 上限、重跑次数 预算不清，失败就会越跑越贵。
只有最终分数 数据版本、评估协议、停机线 分数没法解释，也没法复查。
只写参数量 上下文长度、batch、优化器、数据计划 规模本身不等于实验设计。
本页边界

这里先不追完整 benchmark 实现、全量 scaling fit、后训练配方和大规模系统优化。先把小实验写成可执行计划，再谈扩展。

复现框架

trace

把小规模复现实验看成一条检查链：先定问题，再定预算，然后锁定数据、模型、指标，最后写停机线。顺序反过来，后面的判断就会变成替换口径。

question 问题定义

先写固定什么、改变什么、看什么。问题没写清，后面所有数字都只是在跑。

budget 资源上限

预算决定你能比较多少点、允许多少重跑，以及什么时候该停。

data 数据账

数据要有来源、版本、去重、切分和污染检查；没有这些，结果不可追溯。

model 模型账

参数量之外，还要写上下文长度、batch、优化器、学习率计划和 checkpoint 频率。

metric 评估账

主指标回答原问题，诊断指标告诉你训练是否健康。两类指标不能混成一类。

stop 停机线

先写失败判断，再开跑。这样你才知道该继续、回滚还是放弃。

判断顺序

先定问题，再定预算，然后锁数据、模型和指标，最后写停机线。这个顺序不能省，不然你会把资源花在不可比的结果上。

术语与误区

vocabulary

这几个词在复现实验里最容易混。先分清它们，后面的计划才写得稳。

术语 直觉 它回答什么 常见误解
复现 让别人按你的记录得到同类现象或同类判断。 这次实验该怎么被别人看懂、看回去。 重点是把趋势和判断写清，不是复制所有规模。
预算 你愿意花掉的资源上限。 最多能烧多少 GPU 小时、token 和重跑次数。 不是报销表，也不是“先跑再说”。
主指标 回答原问题的指标。 你到底想验证什么。 不能频繁换，不然实验问题也跟着变。
诊断指标 判断实验健康的指标。 训练是否稳定、数据是否异常、系统是否发热。 不能拿来替代主指标。
停机条件 提前写好的失败判断。 何时暂停、回滚或放弃。 不要等曲线已经坏掉，再临时找理由。
不可外推结论 这次实验不能证明的范围。 哪些结果只能说到这里。 写清边界，反而更可信。
主指标和诊断指标

主指标负责回答“有没有变好”；诊断指标负责回答“为什么会变好、变坏，还是根本没学到”。把两者混在一起，后面很容易选错停止点。

预算估算

budget

直觉：预算不是“我大概能跑两天”。预算要在训练前写清楚：最多能烧掉多少资源，以及失败时会损失什么。

$$B_{\mathrm{eff}} = B_{\mathrm{micro}} \times N_{\mathrm{GPU}} \times A_{\mathrm{acc}}$$ $$C_{\mathrm{rough}} \approx \frac{T_{\mathrm{tokens}}}{r_{\mathrm{tok/s}} \cdot 3600} \times p_{\mathrm{gpu}} \times N_{\mathrm{GPU}}$$

B_micro 是每卡 micro-batch，N_GPU 是 GPU 数，A_acc 是梯度累积步数。T_tokens 是总 token 数，r_tok/s 是吞吐，p_gpu 是单卡小时单价。公式只给数量级，不替代真实 profiler。

tiny: 先验证链路、loss 和恢复，不急着比较结论。
场景 粗略规模 适合回答什么 不适合什么
tiny run 120M 参数 / 300M tokens / 1 GPU 验证训练链路、恢复、日志和最初的 loss 下降。 验证大规模趋势。
comparison run 120M 参数 / 3B tokens / 1 GPU 比较 raw vs dedup、两个 tokenizer 或两个学习率计划。 直接证明模型越大越好。
stress run 350M 参数 / 24B tokens / 4 GPU 看更强的趋势、看数据配方和模型规模的交互。 在问题还不清楚时先花这么多钱。

先写五行

最小案例

walkthrough

先看一个最小但完整的例子。这个例子不追规模，只追“能不能把一条实验计划写完整”。

具体情况

你要比较一个 120M 模型在 raw 数据和 dedup 数据上的验证 loss。两组实验共用 tokenizer、评估集、优化器和学习率计划，只改数据清洗规则。

如果训练 loss 下降、验证 loss 不动，说明这个结论还没出来；这时先查验证集切分、去重和污染，而不是先把模型加大。

Trace 表
对象 先固定什么 变化是什么 如果错了会怎样
数据 来源、版本、切分 raw / dedup 结果不可比，或者验证集被污染。
模型 120M、context length、optimizer 不变 如果这部分也变，结果会混进别的因素。
指标 validation loss 再看 tokens/s、grad norm、NaN / Inf 只看一个数，容易把系统问题当成模型问题。
停机 NaN / Inf、OOM、验证不再改善 一旦触发就停 坏实验会继续烧预算。
失败后的结论

这个例子如果失败，最常见的原因不是“模型太小”，而是变量没有固定住。先把数据、模型和评估协议重新对齐，再判断结果有没有意义。这样做的好处，是你能知道问题是在实验设计里，还是在模型里。

预算算例

runbook

这一段把“要不要开跑”说具体一点。假设你要比较 raw vs dedup，先跑一个 120M 模型，目标是 300M tokens，单卡稳定吞吐大约 900 tok/s，单卡成本按 1.8 元 / 小时算。这个 run 的作用是把计划跑成一个能复查的最小闭环，不是先拿它去证明结论。

算账
$$hours \approx \frac{T_{\mathrm{tokens}}}{r_{\mathrm{tok/s}} \cdot 3600}$$ $$cost \approx hours \times p_{\mathrm{gpu}}$$

按单卡估算时，r_tok/s 是单卡吞吐，p_gpu 是单卡每小时价格。代入 300M tokens、900 tok/s 和 1.8 元 / 小时，可以得到约 92.6 小时、166.7 元。

这个数字很适合拿来做 tiny run 或 comparison run。它的价值不在于“省钱”，而在于你现在知道预算窗口大概在哪里，后面的停止条件也可以据此写出来。

Trace 表
对象 输入 输出 错了会怎样
预算 300M tokens, 900 tok/s, 1.8 元 / 小时 约 92.6 小时，约 166.7 元 如果想象得太乐观，后面就会超预算。
比较 raw vs dedup 只改一类变量 如果同时改模型和数据，结果就拆不开。
停机 预测成本、验证 loss、NaN / Inf 继续 / 暂停 / 回滚 如果停机线写晚了，坏实验会一直烧资源。
失败后的动作

如果预测成本已经比预算高出 20% 以上，先停下来缩 token target，或者重新收窄问题。不要在预算失控时继续把 GPU 加上去，那样只会把同一个问题放大。

budget-note.txt
question: compare raw vs dedup
tokens_target: 300M
throughput_per_gpu: 900 tok/s
gpu_price_per_hour: 1.8
projected_hours: ~92.6
projected_cost: ~166.7
stop_if:
  - projected_hours exceeds 110
  - validation loss does not improve
  - NaN / Inf appears repeatedly

这段算账和练习 4-6 对应：练习 4 让你写预算上限，练习 5 让你算有效 batch，练习 6 让你先去重再切分。三件事其实是同一条链上的不同位置。

案例对照

cases

下面四个案例都在同一层级上比较：每次只动一个主要变量。这样你才知道结果究竟来自数据、模型还是系统。

案例 改什么 主指标 失败线
数据去重复现 raw vs dedup 固定验证集 loss 验证集被去重规则污染，或两组 token 分布差太远。
模型规模对照 60M vs 120M 验证 loss / PPL 较大模型因为 token 不够而训练不稳。
上下文长度实验 512 vs 1024 长文本验证 loss 训练 token 不足，或切片方式改变了对照条件。
系统优化复现 FlashAttention / checkpointing tokens/s 和显存峰值 速度变快，但 loss 曲线被 batch 或数值设置带偏。
怎么读这些案例

先看变量是不是只动了一类，再看主指标有没有回答原问题，最后看失败线是不是提前写好。只要三件事里有一件模糊，结果就不适合继续放大。

怎么继续写

先写一条问题句，再写预算，再写数据，再写模型和停机线。你不需要先把所有案例都想完，先把一个计划写到能执行。

停机与错误

failure

小实验最容易犯的错，是把坏实验继续跑大。停机线的作用，是让你在资源消耗掉之前先停下来，把问题拆清楚。

信号 先查什么 更可能是什么 不要急着做什么
连续 NaN / Inf 学习率、dtype、loss scale、梯度范数 数值不稳。 不要继续拉长训练。
训练 loss 下降，验证 loss 不动 验证集大小、切分、tokenizer、数据重复 泛化还没起来，或验证集有问题。 不要只看训练曲线继续加步数。
tokens/s 太低 batch、加载、精度、通信、checkpointing 系统瓶颈。 不要先加算力，先找瓶颈。
验证分数突然变好很多 代码版本、数据 hash、tokenizer、prompt 协议漂了，或者数据被碰过。 不要立刻宣布突破。
先写停机线

如果你还没写停机线，先停三分钟，把四条停机条件补上：数值失败、系统失败、数据失败、结论失败。写完再开跑，能省很多回头路。

开跑前清单

如果你今天真的要开一个小复现实验，先按这个顺序做：第一，写问题句；第二，写预算上限；第三，写数据和评估协议；第四，写停机线；第五，准备一次 sanity run。这个顺序不是形式，而是为了让每一步都能单独回看。

动作 完成标志 失败时看哪里

开跑清单

runbook

真正开跑之前，先把一轮 sanity run 当成检查表，而不是比较实验。这个阶段只回答三件事：链路通不通、loss 会不会降、恢复是否可靠。只要这三件事还没确认，后面的数据对照就没有资格往下走。

先跑什么

sanity-run.sh
#!/usr/bin/env bash
# 1. 先跑最小数据
# 2. 确认 loss 下降
# 3. 保存 checkpoint
# 4. 重新载入 checkpoint
# 5. 再跑一小段验证恢复
python train.py --config configs/mini.yaml --max_tokens 50000000
python eval.py --checkpoint outputs/last.ckpt --split validation

这不是正式实验。它只是在告诉你：训练、保存、恢复、评估这四条链路现在都能用。

观察什么

  1. 前 50-100 steps 里，loss 应该有下降趋势。
  2. 重启后应该从最近 checkpoint 接着走，不要从头开始。
  3. 如果出现 NaN 或 OOM，先修数值与显存，再谈比较。
  4. 如果这些都正常，再去做 raw vs dedup、tokenizer 或上下文长度的对照。

sanity run 的价值，在于它把“能不能跑”从“跑得好不好”里拆出来。拆开之后，你就不会把链路问题误当成配方问题。

信号 说明 下一步
loss 先降再稳 链路基本通。 可以进入比较实验。
checkpoint 不能恢复 保存/加载路径有问题。 先修恢复，再继续。
一上来就 NaN / Inf 数值设置不稳。 先查学习率、dtype 和 loss scale。
吞吐明显低于预期 系统还没准备好。 回到预算推演，先查数据加载和显存。
为什么要先做这一轮

如果连最小训练都跑不稳，你根本分不清后面的问题是数据、模型还是系统。把 sanity run 先做掉，后面的比较实验才有资格解释结果。

实验模板

template

下面这份模板适合直接抄到笔记里。先填空，再扩写，不要先追漂亮表达。

    - evaluation set

填空顺序

先补问题，再补预算，再补数据，接着补模型和指标，最后补停机线。这样写出来的计划更容易复查，也更容易停下来。

故障复盘

diagnosis

实验计划真正有用的时候，是它能帮助你在出问题时缩小排查范围。下面这个复盘把“验证 loss 没有改善”拆成一条可追的链路：先看症状，再看该查哪层，最后决定要不要停。

一条常见故障链

症状 优先怀疑 下一步检查
训练 loss 下降，验证 loss 不动 数据或评估口径有问题 核对验证集版本、切分方式和污染检查。
训练曲线抖动大 优化器、学习率或 batch 不稳 对照模型账，先看 lr 和有效 batch。
结果波动大到难以复现 随机种子、数据版本或解码设置不固定 把评估账补完整，再跑一次同协议复查。
训练跑得动，但成本超预期 预算和停机线太宽 回看预算账，缩 token 上限或重跑次数。

先做哪一步

  1. 先停掉明显失控的 run，不要继续烧资源。
  2. 把症状写成一句话：是训练不稳，还是验证不稳。
  3. 回到对应账本，确认数据、模型、评估、停机哪一层最可疑。
  4. 只改一类变量，再做下一轮比较。

这条链路最重要的地方，是它把“猜原因”变成了“按账本找原因”。如果你今天能追到是哪一层出了问题，下一次实验计划就会更短，也更便宜。

复盘时常见的错法

最常见的错法是看到验证 loss 不动，就直接换模型、换数据、换学习率，最后也不知道哪一项起了作用。更稳的做法是先确认症状属于哪一层，再只动一个变量。这样哪怕结果还是失败，你也知道失败发生在哪里。

一个完整故障示范

假设你做的是 raw vs dedup 对照。第一轮看到的现象是：训练 loss 正常下降，但验证 loss 到第 3 次评估后几乎不动。这个时候不要先换模型，也不要马上把去重结论写出来。先查数据账，看验证集是不是和训练集近重复了；再查评估账，看是不是验证集版本、切分种子或解码参数变了；如果这两层都没问题，再回到模型账，检查 batch 和学习率是不是太保守，导致模型只是在训练集上记得更快。

如果你把这三层都查完，结果还是不动，那就把停机账收紧：先停止继续加 token，把本轮结论写成“当前预算和当前配置下，去重没有带来明显验证收益”，而不是直接写成“去重没用”。前者是可复查的判断，后者只是过度外推。

步骤 看什么 写进记录什么 下一步动作
现象 训练 loss 正常降，验证 loss 停住。 现象本身，不加判断。 先不要换变量，先分层排查。
数据层 验证集版本、切分种子、近重复。 数据是否同版、同切分、同污染检查。 如果这一层有问题，先修数据再重跑。
评估层 prompt、解码、随机种子、主指标。 协议是否完全相同。 如果协议变了，结果不能直接比较。
模型层 batch、学习率、context length。 训练动态是否匹配预算。 如果模型太保守，再考虑调参。
停机层 预算、重跑次数、结论边界。 本轮能说什么、不能说什么。 把局部结论写小，别把边界写丢。
