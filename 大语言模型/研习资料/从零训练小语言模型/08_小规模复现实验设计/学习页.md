# 术语与误区 {#terms}
<!-- tag: vocabulary -->

这几个词在复现实验里最容易混。先分清它们，后面的计划才写得稳。

术语
直觉
它回答什么
常见误解


**复现**
让别人按你的记录得到同类现象或同类判断。
这次实验该怎么被别人看懂、看回去。
重点是把趋势和判断写清，不是复制所有规模。


**预算**
你愿意花掉的资源上限。
最多能烧多少 GPU 小时、token 和重跑次数。
不是报销表，也不是“先跑再说”。


**主指标**
回答原问题的指标。
你到底想验证什么。
不能频繁换，不然实验问题也跟着变。


**诊断指标**
判断实验健康的指标。
训练是否稳定、数据是否异常、系统是否发热。
不能拿来替代主指标。


**停机条件**
提前写好的失败判断。
何时暂停、回滚或放弃。
不要等曲线已经坏掉，再临时找理由。


**不可外推结论**
这次实验不能证明的范围。
哪些结果只能说到这里。
写清边界，反而更可信。

## box warn
主指标和诊断指标
主指标负责回答“有没有变好”；诊断指标负责回答“为什么会变好、变坏，还是根本没学到”。把两者混在一起，后面很容易选错停止点。

# 预算估算 {#budget}
<!-- tag: budget -->
<!-- layout: grid2 -->

## box
**直觉：**预算不是“我大概能跑两天”。预算要在训练前写清楚：最多能烧掉多少资源，以及失败时会损失什么。
$$B_{\mathrm{eff}} = B_{\mathrm{micro}} \times N_{\mathrm{GPU}} \times A_{\mathrm{acc}}$$
 $$C_{\mathrm{rough}} \approx \frac{T_{\mathrm{tokens}}}{r_{\mathrm{tok/s}} \cdot 3600} \times p_{\mathrm{gpu}} \times N_{\mathrm{GPU}}$$
 `B_micro` 是每卡 micro-batch，`N_GPU` 是 GPU 数，`A_acc` 是梯度累积步数。`T_tokens` 是总 token 数，`r_tok/s` 是吞吐，`p_gpu` 是单卡小时单价。公式只给数量级，不替代真实 profiler。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

场景
粗略规模
适合回答什么
不适合什么


tiny run
120M 参数 / 300M tokens / 1 GPU
验证训练链路、恢复、日志和最初的 loss 下降。
验证大规模趋势。


comparison run
120M 参数 / 3B tokens / 1 GPU
比较 raw vs dedup、两个 tokenizer 或两个学习率计划。
直接证明模型越大越好。


stress run
350M 参数 / 24B tokens / 4 GPU
看更强的趋势、看数据配方和模型规模的交互。
在问题还不清楚时先花这么多钱。

# 预算算例 {#sanity}
<!-- tag: runbook -->

这一段把“要不要开跑”说具体一点。假设你要比较 raw vs dedup，先跑一个 120M 模型，目标是 300M tokens，单卡稳定吞吐大约 900 tok/s，单卡成本按 1.8 元 / 小时算。这个 run 的作用是把计划跑成一个能复查的最小闭环，不是先拿它去证明结论。

## box soft
算账
$$hours \approx \frac{T_{\mathrm{tokens}}}{r_{\mathrm{tok/s}} \cdot 3600}$$
 $$cost \approx hours \times p_{\mathrm{gpu}}$$
 按单卡估算时，`r_tok/s` 是单卡吞吐，`p_gpu` 是单卡每小时价格。代入 300M tokens、900 tok/s 和 1.8 元 / 小时，可以得到约 92.6 小时、166.7 元。
这个数字很适合拿来做 tiny run 或 comparison run。它的价值不在于“省钱”，而在于你现在知道预算窗口大概在哪里，后面的停止条件也可以据此写出来。

300M tokens, 900 tok/s, 1.8 元 / 小时
约 92.6 小时，约 166.7 元
如果想象得太乐观，后面就会超预算。


比较
raw vs dedup
只改一类变量
如果同时改模型和数据，结果就拆不开。


停机

继续 / 暂停 / 回滚

## box warn
失败后的动作
如果预测成本已经比预算高出 20% 以上，先停下来缩 token target，或者重新收窄问题。不要在预算失控时继续把 GPU 加上去，那样只会把同一个问题放大。

budget-note.txt复制

```text
question: compare raw vs dedup
tokens_target: 300M
throughput_per_gpu: 900 tok/s
gpu_price_per_hour: 1.8
projected_hours: ~92.6
projected_cost: ~166.7
stop_if:
 - projected_hours exceeds 110
 - validation loss does not improve
 - NaN / Inf appears repeatedly
```

这段算账和练习 4-6 对应：练习 4 让你写预算上限，练习 5 让你算有效 batch，练习 6 让你先去重再切分。三件事其实是同一条链上的不同位置。

# 开跑清单 {#runbook}
<!-- tag: runbook -->

真正开跑之前，先把一轮 sanity run 当成检查表，而不是比较实验。这个阶段只回答三件事：链路通不通、loss 会不会降、恢复是否可靠。只要这三件事还没确认，后面的数据对照就没有资格往下走。

先跑什么
sanity-run.sh复制
`#!/usr/bin/env bash
# 1. 先跑最小数据
# 2. 确认 loss 下降
# 3. 保存 checkpoint
# 4. 重新载入 checkpoint
# 5. 再跑一小段验证恢复
python train.py --config configs/mini.yaml --max_tokens 50000000
python eval.py --checkpoint outputs/last.ckpt --split validation`
这不是正式实验。它只是在告诉你：训练、保存、恢复、评估这四条链路现在都能用。


观察什么

前 50-100 steps 里，loss 应该有下降趋势。
重启后应该从最近 checkpoint 接着走，不要从头开始。
如果出现 NaN 或 OOM，先修数值与显存，再谈比较。
如果这些都正常，再去做 raw vs dedup、tokenizer 或上下文长度的对照。

sanity run 的价值，在于它把“能不能跑”从“跑得好不好”里拆出来。拆开之后，你就不会把链路问题误当成配方问题。

说明
下一步


loss 先降再稳
链路基本通。
可以进入比较实验。


checkpoint 不能恢复
保存/加载路径有问题。
先修恢复，再继续。


一上来就 NaN / Inf
数值设置不稳。
先查学习率、dtype 和 loss scale。


吞吐明显低于预期
系统还没准备好。
回到预算推演，先查数据加载和显存。

## box warn
为什么要先做这一轮
如果连最小训练都跑不稳，你根本分不清后面的问题是数据、模型还是系统。把 sanity run 先做掉，后面的比较实验才有资格解释结果。

# 自测答案方向 {#check}
<!-- tag: Check -->

练习 1-3 检查点
练习 1 要把问题写成“固定什么、改变什么、看什么”；练习 2 选“训练 loss 下降就能直接证明结论”为错；练习 3 要能补出问题句里的固定项、变化项和观察项。

练习 4-6 检查点
练习 4 属于预算项，答案应围绕 GPU 小时、token 上限、重跑次数；练习 5 用有效 batch 公式算出 64；练习 6 选先去重再切分，验证集要先隔离再使用。

练习 7-9 检查点
练习 7 要固定数据配方、tokenizer、评估集和优化器；练习 8 要把样例计划里的模型字段补完整；练习 9 需要写出一条完整计划，并能说清主指标和诊断指标分别做什么。

练习 10-12 检查点
练习 10 应该在 NaN / Inf、OOM、验证不再改善等停机线上作判断；练习 11 选“每天根据验证集改规则还能继续当封存集”为错；练习 12 要写清能说什么、不能说什么，以及下一步该怎么补。

# 故障复盘 {#debug}
<!-- tag: diagnosis -->

实验计划真正有用的时候，是它能帮助你在出问题时缩小排查范围。下面这个复盘把“验证 loss 没有改善”拆成一条可追的链路：先看症状，再看该查哪层，最后决定要不要停。

一条常见故障链


症状
优先怀疑
下一步检查


训练 loss 下降，验证 loss 不动
数据或评估口径有问题
核对验证集版本、切分方式和污染检查。


训练曲线抖动大
优化器、学习率或 batch 不稳
对照模型账，先看 lr 和有效 batch。


结果波动大到难以复现
随机种子、数据版本或解码设置不固定
把评估账补完整，再跑一次同协议复查。


训练跑得动，但成本超预期
预算和停机线太宽
回看预算账，缩 token 上限或重跑次数。


先做哪一步

先停掉明显失控的 run，不要继续烧资源。
把症状写成一句话：是训练不稳，还是验证不稳。
回到对应账本，确认数据、模型、评估、停机哪一层最可疑。
只改一类变量，再做下一轮比较。

这条链路最重要的地方，是它把“猜原因”变成了“按账本找原因”。如果你今天能追到是哪一层出了问题，下一次实验计划就会更短，也更便宜。

## box warn
复盘时常见的错法
最常见的错法是看到验证 loss 不动，就直接换模型、换数据、换学习率，最后也不知道哪一项起了作用。更稳的做法是先确认症状属于哪一层，再只动一个变量。这样哪怕结果还是失败，你也知道失败发生在哪里。

## box soft
一个完整故障示范
假设你做的是 raw vs dedup 对照。第一轮看到的现象是：训练 loss 正常下降，但验证 loss 到第 3 次评估后几乎不动。这个时候不要先换模型，也不要马上把去重结论写出来。先查数据账，看验证集是不是和训练集近重复了；再查评估账，看是不是验证集版本、切分种子或解码参数变了；如果这两层都没问题，再回到模型账，检查 batch 和学习率是不是太保守，导致模型只是在训练集上记得更快。
如果你把这三层都查完，结果还是不动，那就把停机账收紧：先停止继续加 token，把本轮结论写成“当前预算和当前配置下，去重没有带来明显验证收益”，而不是直接写成“去重没用”。前者是可复查的判断，后者只是过度外推。

步骤
看什么
写进记录什么
下一步动作


现象
训练 loss 正常降，验证 loss 停住。
现象本身，不加判断。
先不要换变量，先分层排查。


数据层
验证集版本、切分种子、近重复。
数据是否同版、同切分、同污染检查。
如果这一层有问题，先修数据再重跑。


评估层
prompt、解码、随机种子、主指标。
协议是否完全相同。
如果协议变了，结果不能直接比较。


模型层
batch、学习率、context length。
训练动态是否匹配预算。
如果模型太保守，再考虑调参。


停机层
预算、重跑次数、结论边界。
本轮能说什么、不能说什么。
把局部结论写小，别把边界写丢。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 作答目标
这些题训练你拿到实验结果后能做判断：问题是否清楚、预算是否够用、数据是否可回查、停机是否及时。

## box info
### 检查重点
答案要包含原因、证据和下一步动作。只写“过拟合”“数据污染”“模型太小”通常不够。

<!-- end-practice-intro -->

<!-- practice-group: 一、先定问题 -->
<!-- quick-review: 关键 问题句要固定项、变化项、观察项三件事都写出来。 -->

### 1 choice

**Q:** 练习 1：选择实验问题

- [ ] 想把模型训练得更好
- [ ] 固定模型大小和 tokenizer，比较 raw vs dedup 对验证 loss 的影响
- [ ] 先把所有数据都试一遍，再看结果
- [ ] 训练一个更大的模型

Answer: 固定模型大小和 tokenizer，比较 raw vs dedup 对验证 loss 的影响
Explain: 实验问题要写固定项、变化项和主指标；只写“训练一个模型”太泛。

### 2 true/false

**Q:** 练习 2：训练 loss 的边界

- [ ] 对
- [ ] 错

Answer: 错
Explain: 训练 loss 下降只能说明优化还在继续，不足以证明实验问题已经被回答。

### 3 mini-fields

**Q:** 练习 3：补全问题句

- [ ] 固定什么
- [ ] 改变什么
- [ ] 看什么

Explain: 答案里要同时出现固定项、变化项和观察项。只写“做一次比较”还不够，必须让别人知道这次比较究竟在比什么。

<!-- end-practice-group -->

<!-- practice-group: 二、预算与数据 -->
<!-- quick-review: 关键 预算写上限，数据写版本、切分和污染检查。 -->

### 4 choice

**Q:** 练习 4：预算里该写什么

- [ ] GPU 小时、token 上限、重跑次数
- [ ] 只写 GPU 型号
- [ ] 只写模型名字
- [ ] 只写最终分数

Answer: GPU 小时、token 上限、重跑次数
Explain: 预算需要资源上限和重跑成本；GPU 型号本身不够。

### 5 fill

**Q:** 练习 5：有效 batch

Answer: 64
Explain: 有效 batch = micro-batch × GPU 数 × 梯度累积步数，这里是 4×2×8=64。

### 6 choice

**Q:** 练习 6：验证集切分顺序

- [ ] 先随机切成训练和验证，再去重
- [ ] 先去重和近重复检测，再做文档级切分
- [ ] 每个 epoch 重新抽验证集
- [ ] 把验证集里表现好的样本移回训练集

Answer: 先去重和近重复检测，再做文档级切分
Explain: 先去重和近重复检测，再做文档级切分，能减少训练 / 验证泄漏。

<!-- end-practice-group -->

<!-- practice-group: 三、模型与指标 -->
<!-- quick-review: 关键 比较模型大小时，数据、tokenizer、评估集和优化器家族都要固定。 -->

### 7 choice

**Q:** 练习 7：控制变量

- [ ] 数据配方、训练 token、tokenizer、评估集和优化器
- [ ] 只固定 GPU 型号
- [ ] 只固定训练天数
- [ ] 每个模型都换一个 tokenizer

Answer: 数据配方、训练 token、tokenizer、评估集和优化器
Explain: 比较模型大小时，数据配方、tokenizer、评估集和优化器家族都要固定。

### 8 mini-fields

**Q:** 练习 8：补全模型账

- [ ] 参数量
- [ ] 上下文长度
- [ ] 优化器

Explain: 这三项只是样例字段，重点是知道模型账里不能只写参数量。上下文长度和优化器也会改变显存、训练动态和复现难度。

### 9 synthesis

**Q:** 练习 9：写一条计划

Explain: 合格答案要能看出：固定了哪些变量，改变了哪一项数据处理，主指标是验证 loss，诊断指标可以是 tokens/s、grad norm、NaN / Inf 或 OOM。写得再短，也要能让别人知道怎么复做。

<!-- end-practice-group -->

<!-- practice-group: 四、停机与边界 -->
<!-- quick-review: 关键 停机线优先于继续投入，边界优先于漂亮结论。 -->

### 10 choice

**Q:** 练习 10：第一停机线

- [ ] 连续出现 NaN / Inf
- [ ] tokens/s 有一点波动
- [ ] 训练曲线很平滑
- [ ] 验证集按计划按时跑完

Answer: 连续出现 NaN / Inf
Explain: 连续 NaN、梯度爆炸、loss 持续异常上升，都是应该先停下来的信号。

### 11 true/false

**Q:** 练习 11：验证集边界

- [ ] 对
- [ ] 错

Answer: 错
Explain: 封存验证集应该保持稳定；如果每天根据它改规则，它就不再可信。

### 12 synthesis

**Q:** 练习 12：写边界

Explain: 好答案会明确说出：这次结果只对当前数据切分、预算和模型规模成立；不能直接外推到更大模型、更长训练或别的数据分布。若能顺手写出下一步该补什么，会更好。

<!-- end-practice-group -->
