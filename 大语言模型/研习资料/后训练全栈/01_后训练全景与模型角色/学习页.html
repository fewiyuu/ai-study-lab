课程入口

后训练全景与模型角色

这一页先把后训练里最容易混的几个对象分开：base model、SFT checkpoint、policy model、reference model、reward model 和 judge。先看它们各自读什么、写什么、谁更新参数，再看它们怎样串成 DPO、RLHF 和发布前评测的流水线。把这一层理顺，后面的 SFT 模板、偏好优化和评测单元才不会只剩名词。

读完本页应该能做什么

  • 能说清 base、policy、reference、reward、judge 在后训练里的位置和职责。
  • 能用一张表追踪同一个 checkpoint 在不同阶段为什么会扮演不同角色。
  • 能把一条 DPO 或 RLHF 配方画成对象链，并指出常见错误会坏在哪里。

学习目标

goal

这节先把对象关系讲清楚，再进入算法公式。你读完后，应该能把一条后训练流水线拆成“谁生成、谁约束、谁打分、谁验收”四类动作，并且看见某个对象一旦该冻结却被更新，整个闭环会怎么坏掉。

更具体一点，你应该能拿到一份 DPO、RLHF 或 GRPO 的简化配方时，先把 policy、reference、reward、judge、SFT checkpoint 放到正确位置，再决定下一步要看损失、看评测，还是看数据模板。只要这一步稳，后面的单元就不会变成一串名词。

先抓主线

overview

后训练要解决的问题很直接：预训练模型会续写，但不会天然按指令办事，也不会天然知道什么回答更稳、更像人要的结果。后训练的工作，就是把“会说话”改造成“会按任务说话”，再把“按任务说话”改造成“能接受偏好约束、能通过评测和发布检查”。

这条主线里，base model 是起点，SFT checkpoint 是第一次被指令数据塑形后的结果，policy model 是真正被优化的对象，reference model 负责给 policy 提供稳定参照，reward model 给训练内环提供分数，judge / evaluator 则站在训练外环看结果是否值得放行。关键不是名字，而是读写权限。

如果你先把这层角色关系看透，后面的 DPO、ORPO、PPO、GRPO 只是换了优化方式；这层没看透，公式越看越像在背标签。第 01 单元就是来把这层地基打平的。

关键术语

terms
base model

预训练出来的底座，只学到了通用语言规律。它能续写，但不等于能按指令稳定回答。

SFT checkpoint

监督微调后的中间点。它已经看过示范答案，适合作为 policy 或 reference 的共同起点。

policy model

训练时真正被更新的模型。它负责提出候选回答，也负责承受损失函数或奖励信号。

reference model

保持稳定的参照模型。它不应该跟着 policy 一起漂，否则“相对变化”就失去意义。

reward model

把回答映射成可优化分数的模型或函数，主要服务训练内环。

judge / evaluator

训练外环或发布前使用的评审者。它看的是结果能不能放行，不负责给 policy 提供梯度。

这组词最容易混的地方在职责，不在字面。两个模型名字一样长，不代表在同一环里；一个函数看起来像打分，也不代表它一定是训练信号。先看环，再看名。

角色账本

trace

先用这张表把对象的读写关系钉住。你之后只要忘了谁更新、谁冻结、谁打分，就回来看这一页。它把权限和状态直接列出来，方便后面排错。

对象 前置状态 后续状态 读什么 写什么 / 是否更新 出错症状
base model 通用预训练权重 可进入 SFT 起点 通用文本分布 通常不在后训练这一步直接改 把它当 chat model，用错期待值
SFT checkpoint 已看过示范答案 可分叉成 policy 和 reference 指令-回答样本 在 SFT 阶段被更新，之后常被复用 分不清“训练结果”和“训练起点”
policy model 来自 SFT checkpoint 或其变体 随着优化继续变化 prompt、偏好对、奖励或损失 更新参数 把冻结对象写成 trainable，约束会失效
reference model 与 policy 同源但冻结 保持稳定 当前 policy 的偏离程度 不更新参数 reference 漂了，KL / 相对概率就失去锚点
reward model 偏好数据或规则函数 给训练内环提供分数 候选回答、偏好标签、规则 可能更新，也可能只是函数 分数能涨，不代表真实质量能涨
judge / evaluator 训练外的评审器 给发布前检查结论 最终输出、基准集、人工规则 通常不参与梯度更新 把一个高分 judge 误当成上线许可

如果你以后再看别的后训练页，可以先把对象往这张表里塞。只要表格里有一行你填不满，说明那个对象你还没真正搞清楚它是在读什么、写什么。

为什么会分成两份

state trace

第一次接触后训练时，最容易误会的一点是：既然 policy 和 reference 一开始来自同一个 checkpoint，为什么不干脆让它们一直共用一份权重？答案在于“比较”这件事本身需要一个稳定的参照。如果参照也跟着变，系统就会失去刻度，训练信号只剩自己追自己。

可以把它看成一次很普通的分工：policy 负责今天继续往前走，reference 负责保留昨天的位置，reward 负责告诉你这一步偏好有没有更好，judge 负责在外面看今天的结果值不值得放行。关键是谁在动、谁不动。

起点 policy 和 reference 都从同一个 SFT checkpoint 出发。

这样后面的差异就能明确归因到训练过程，而不是初始化。

更新 只有 policy 接受损失或奖励更新。

如果把 reference 也改了，后面的相对约束就没有意义。

比较 比较的是 policy 相对 reference 的偏离。

DPO、PPO、GRPO 都需要这把尺子，不然只剩自我追赶。

外环 judge 不给梯度，只给结论。

它检查的是训练后模型是否真的更稳、更对题、更适合发布。

这个分工一旦建立起来，很多错误就会变得很好识别。你看到某个实现把 reference 也训练了，先不要看损失公式，先问一句：比较还剩不剩锚点？如果锚点没了，后面的分数再漂亮也只是漂在空中。

角色怎么读写

matrix

下面这张矩阵把“谁读什么、谁写什么”说得更细一点。它不是算法公式，却常常比公式更能帮你排错。

对象 主要读取 主要写入 适合的位置 不适合的位置
policy prompt、reward、reference 候选回答分布 训练内环 作为稳定参照
reference 当前 policy 的偏离程度 不写入 对照基线 被梯度更新
reward model / rule 候选回答、偏好标签 分数或偏好方向 训练内环信号源 单独充当最终发布裁决
judge 最终输出、基准集、人工标准 评测结论 训练外环、发布前检查 直接替代训练目标
$$ \mathcal{L}_{total} = \mathcal{L}_{pref} + \beta \cdot KL(\pi_{\text{policy}} \parallel \pi_{\text{ref}}) $$

\mathcal{L}_{pref} 是偏好或奖励带来的优化项，\pi_{\text{policy}} 是正在更新的策略，\pi_{\text{ref}} 是冻结的参照策略，\beta 是控制 policy 偏离 reference 多远的权重。你不用先背完整推导，但要知道每个符号都对应一类职责。

如果你以后在别的单元里看到“mask、loss、KL、reward、judge”这些词，先不要急着背。先问自己：这个对象是读信号的，还是写信号的；是内环的，还是外环的。很多表面难题，拆成这两问就简单了。

最小案例

case

把角色放进一个真实一点的客服场景，最容易看出谁在训练内环，谁在训练外环。假设用户问：“订单为什么还没发货？” policy 先生成一条回答，reward 判断这条回答是否足够清楚、是否承认了不确定性、是否给出了下一步动作；reference 则负责给 policy 提供参照，让它知道自己有没有偏离原来的回答风格。

角色 这一步看什么 这一步写什么 如果放错位置，最先坏什么
policy 用户问题、示范偏好、reward、reference 当前回答分布 回答会越来越会讨好提示，稳定性反而下降
reference policy 的偏离程度 不写入 一旦漂移，无法判断 policy 到底改了多少
reward / rule 回答是否清楚、是否安全、是否有下一步动作 分数或偏好方向 如果直接替代 judge，模型会把一次打分当成最终真理
judge 最终回复、回归集、人工标准 是否放行的结论 如果把它塞回训练里，外环会污染内环

这个案例里，先盯谁读这个回答，谁改这个回答，谁只看结果。如果你把 reward 和 judge 互换，短期看起来也许还能跑，但长期会出现一种很典型的现象：模型开始学会顺着评测器说话，任务本身反而做得更差。

prompt → policy 生成候选
        → reward / rule 给分
        → reference 约束偏离
        → policy 更新
        → judge 在外环验收

这就是你以后看后训练配方时最先要问的事：我现在看到的是训练内环，还是训练外环？如果答案不清楚，就先别看公式，先把对象表补出来。

DPO 流程

code reading

后训练页里经常会看到这种最小代码结构。它的作用是让你在看到配方时，能把“谁冻结、谁更新、谁比较”一眼认出来。

policy / reference 分叉
# SFT checkpoint 只复制一次，然后分成两个职责
policy = load_checkpoint(sft_checkpoint)
reference = load_checkpoint(sft_checkpoint)
freeze(reference)

for prompt, chosen, rejected in batch:
    score_chosen = policy_score(policy, prompt, chosen)
    score_rejected = policy_score(policy, prompt, rejected)
    ref_gap = policy_gap(policy, reference, prompt, chosen, rejected)
    loss = dpo_loss(score_chosen, score_rejected, ref_gap)
    update(policy, loss)

# reference 不更新；它只提供稳定锚点

读这段代码时，你只需要盯住四个动作：两次加载、一次冻结、一次更新。只要 reference 也被送进 update，或者 policy 的分数比较里没有 reference 的影子，整条链就已经和 DPO 的基本意图脱节了。

这类代码最常见的误读是把“同一个 checkpoint 被加载两次”当成重复劳动。实际含义是分工：一个对象继续往前走，一个对象保留原位，比较才有意义。

偏好信号怎么进 loss

worked case

这一步最值得看的是信号方向。假设同一个 prompt 下有两个回答：chosen 更稳，rejected 更虚。训练先看 policy 有没有把 chosen 的相对分数抬高，把 rejected 压下去。reference 还在旁边提供稳定参照，让这个方向不至于乱飘。

$$ \Delta = \big(\log \pi_{\text{policy}}(chosen) - \log \pi_{\text{policy}}(rejected)\big) - \big(\log \pi_{\text{ref}}(chosen) - \log \pi_{\text{ref}}(rejected)\big) $$

如果 \Delta 变大，说明 policy 比 reference 更偏向 chosen；如果 \Delta 没朝这个方向走，损失就没有在推动正确的比较关系。重点在相对差。

观测 说明 该检查什么 下一步动作
chosen 得分高，rejected 也一起被抬高 policy 可能在整体变“更会讨好” reference 是否还在做参照 检查冻结与比较项是否正确
chosen / rejected 都没变 训练信号没有真正进入更新 loss 是否连接到了 policy 检查 optimizer、梯度和数据流
分数很好看，但人工一读就觉得更空 信号可能被 prompt 形式牵着走 judge / reward 是否和测试集同分布 加独立回归集和人工抽检

这就是为什么只看最终分数不够。偏好优化里，真正重要的是一条相对关系有没有被稳定地拉出来。你以后如果看到某个训练说“chosen 变好了”，但没法告诉你 rejected 和 reference 怎么动，那其实还没把链条讲完整。

这个小例子还有一个常见误会：很多人会把 reward、reference、judge 全都当成“给分的东西”。实际差别在于，谁参与更新，谁只给参照，谁只做验收。把这三类动作分开，后面的 DPO 公式会好读很多。

如果你在日志里看到 policy 的 chosen 分数在涨，但 judge 仍然指出回答空泛，下一步就别继续堆训练轮数。先回到这张表，看 reward 关注的是不是偏好方向，judge 看的是不是最终可用性，再决定是修 reference、换评测集，还是回到 SFT 样本本身。

RLHF / GRPO

inner loop

在 RLHF 或 GRPO 里，policy 先基于 prompt 生成候选，再由 reward model 或规则函数给分，随后 policy 按奖励和约束更新。reference 常常用来限制 policy 的漂移，judge 则站在训练外部看这一轮训练之后的结果是否真有改善。

训练内环
promptpolicyreward
训练外环
referencejudge
环节 输入 输出 是否更新 policy 常见失误
生成候选 prompt 多个回答 否 候选太少，比较没有信息量
打分 候选回答 reward / preference 否 把奖励函数当成最终真值
约束更新 reward、reference、policy 新 policy 是 reference 漂移，KL 失去锚点
外环评测 最终模型输出 judge 结论 / 报告 否 把评测分当成训练信号继续套回去 
现象：policy 看起来在改，reference 也在跟着改。
下一步：检查是否把 reference 加进了 optimizer 参数列表。
后果：KL 或相对概率没有稳定锚点，训练解释变差。
修正：冻结 reference，并确认它只读不写。

这里先记住一个判断：reward 负责训练内环，judge 负责训练外环。它们可以同源，但职责不能混成一件事。

日志里看到什么 优先怀疑什么 先查哪一层 对应回到本页哪一块
reward 变高，回答更像模板 reward 过度追格式 训练内环的打分规则 偏好信号怎么进 loss
KL 很小，输出却开始跑偏 reference 没有真正冻结 optimizer 和复制时机 为什么会分成两份
judge 打高分，但人工一读就不稳 judge 和训练分布太近 训练外环和评测集 常见错误

读这张表时，关键不是记住每个名字，而是把“症状 -> 原因 -> 该查哪一层”连起来。只要你能把这条链讲顺，日志就不只是数值，而是线索。

cases

同一个 SFT checkpoint 先被复制成 policy 和 reference。chosen / rejected 进入比较后，只有 policy 继续走。

案例 2：GRPO 训练环

这些案例的重点，不在于你记住某个缩写，而在于你能看出同一个对象在不同阶段的身份变化。它在某一页里是训练成果，在下一页里又是起点或基线，这并不矛盾；真正会出问题的是职责没有切开。

base model → SFT checkpoint → {policy, reference}
                                   ↓
                          reward model / rule function
                                   ↓
                            judge / evaluator

常见错误

debug

错误 1：把 base model 当 chat model

base 只说明它是预训练底座，不说明它已经会按指令回答。直接把它当产品模型，会高估它的服从性。

错误 2：让 reference 跟着 policy 一起更新

reference 一旦漂移，就不再是稳定参照。很多相对约束和 KL 控制都会失去意义。

错误 3：把 judge 高分当成真实能力

judge 只是评审器，不是训练目标本身。它可能偏向某种措辞、某类题型或某个提示词。

错误 4：只看最终分数，不看角色链

后训练里最常见的定位错误，就是把“分数很好”误读成“整个链条都对”。先看谁打分，再看谁被更新。

症状 通常原因 先检查什么 下一步动作
模型分数变高，但回答风格更讨好 reward 或 judge 过拟合提示词 评测集和训练信号是否同分布 分开训练内外环，增加独立回归集
reference 与 policy 越来越像 reference 被错误更新或周期性重置 optimizer 参数列表和复制时机 冻结 reference，确认只复制不回写
术语全都认识，但说不出谁负责什么 只记名词，没有看动作 读写关系表是否能填满 先按对象、再按环、最后按算法重读

调参前先核一遍

diagnosis

当训练看起来能跑，但结果还是不稳时，先不要换模型、加轮数或改学习率。先按对象顺序核查：policy 是否唯一可训练，reference 是否冻结，reward 和 judge 是否分在不同环，chosen / rejected 是否真的来自同一个 prompt。只要这四项里有一项错位，后面的分数通常都只是表面正常。

这一步的价值在于把问题定位到对象边界，而不是被最终分数带跑。很多训练失败不是算法公式错了，而是权责表已经错位：该更新的没更新，该冻结的被更新，内环和外环混成了一件事。

症状 常见原因 第一检查 下一步动作
chosen / rejected 一起涨，回答还是空 reward 只看形式，不看任务完成度 reward 的判分标准和样本分布 增加独立评测集和人工抽检
policy 和 reference 一起漂 reference 被错误加入更新 optimizer 参数列表和复制时机 冻结 reference，确认只读不写
judge 打高分，真实样本却不稳 judge 过拟合提示词或题型 judge 看的是训练样本还是留出样本 把 judge 留在外环，用独立回归集校验
四步核查顺序
1. 先看 policy 是否在更新
2. 再看 reference 是否冻结
3. 再看 reward / judge 是否分在不同环
4. 最后看 chosen / rejected 是否来自同一 prompt

如果这四步里有一步说不清，就先回到前面的角色账本和 trace。对象边界对了，再谈算法、学习率和 batch 才有意义。你也可以直接回到练习区，先答 Q7、Q11 和 Q12，看自己是不是已经把对象和信号分开了。

边界与下一步

boundary

这一页只负责把角色关系立住，不展开 SFT 的模板设计、loss_mask、packing、偏好损失公式或 RL 训练细节。那些内容分别在后面的单元里展开；你现在只要知道它们会沿着这条角色链继续往下走。

接下来先不要追更复杂的推导。先不碰 tokenizer、数据清洗、packing、PPO 的完整目标函数、batch 内采样细节和多卡训练配置。等你在下一页把样本和 loss_mask 看明白，再回来学这些东西，顺序会顺很多。

如果你已经能把“谁更新、谁冻结、谁打分、谁验收”这四句话说顺，下一步就去看 SFT 数据模板与 loss_mask。那一页会把“对象是谁”继续压成“样本长什么样、哪些 token 真被学到”。

Q11. 错误诊断：训练时把 reference 也设成 trainable，会出现什么问题？

答题脚手架：先说 reference 的作用，再说它更新后作用如何被破坏，最后给修法。

参考答案 / 评分要点

参考答案至少要提到三点：reference 失去稳定参照；policy 的相对偏移不再可解释；KL 或类似约束会失去锚点。修法是把 reference 冻结，并在训练开始前明确复制来源。

Q17. 解释 prompt、policy、reward、reference、judge 在一次 RLHF 实验中的关系。

答题脚手架：每个角色至少出现一次，并说明是否在训练内环。

参考答案 / 评分要点

一份合格答案要把顺序说出来：prompt 进入 policy，policy 生成候选，reward 对候选打分，reference 提供稳定参照，policy 根据 reward 和 reference 的约束更新，judge 在训练外环检查是否真的更好。

synthesis复述

Q18. 用 150 字以内给新人讲清楚本节。

答题脚手架：先说后训练目的，再说谁生成、谁约束、谁打分、谁验收。

参考答案 / 评分要点

答题时不要堆术语。先说后训练的目的，再用一条短链条解释：base 是底座，SFT 之后得到 checkpoint，policy 负责更新，reference 负责参照，reward 给训练信号，judge 负责外环验收。
