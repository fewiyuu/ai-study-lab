# 学习目标 {#goals}
<!-- tag: goal -->

这节先把对象关系讲清楚，再进入算法公式。你读完后，应该能把一条后训练流水线拆成“谁生成、谁约束、谁打分、谁验收”四类动作，并且看见某个对象一旦该冻结却被更新，整个闭环会怎么坏掉。

更具体一点，你应该能拿到一份 DPO、RLHF 或 GRPO 的简化配方时，先把 policy、reference、reward、judge、SFT checkpoint 放到正确位置，再决定下一步要看损失、看评测，还是看数据模板。只要这一步稳，后面的单元就不会变成一串名词。


---

# 先抓主线 {#overview}
<!-- tag: overview -->

后训练要解决的问题很直接：预训练模型会续写，但不会天然按指令办事，也不会天然知道什么回答更稳、更像人要的结果。后训练的工作，就是把“会说话”改造成“会按任务说话”，再把“按任务说话”改造成“能接受偏好约束、能通过评测和发布检查”。

这条主线里，base model 是起点，SFT checkpoint 是第一次被指令数据塑形后的结果，policy model 是真正被优化的对象，reference model 负责给 policy 提供稳定参照，reward model 给训练内环提供分数，judge / evaluator 则站在训练外环看结果是否值得放行。关键不是名字，而是读写权限。

如果你先把这层角色关系看透，后面的 DPO、ORPO、PPO、GRPO 只是换了优化方式；这层没看透，公式越看越像在背标签。第 01 单元就是来把这层地基打平的。

# 关键术语 {#terms}
<!-- tag: terms -->

**base model**
预训练出来的底座，只学到了通用语言规律。它能续写，但不等于能按指令稳定回答。


**SFT checkpoint**
监督微调后的中间点。它已经看过示范答案，适合作为 policy 或 reference 的共同起点。


**policy model**
训练时真正被更新的模型。它负责提出候选回答，也负责承受损失函数或奖励信号。


**reference model**
保持稳定的参照模型。它不应该跟着 policy 一起漂，否则“相对变化”就失去意义。


**reward model**
把回答映射成可优化分数的模型或函数，主要服务训练内环。


**judge / evaluator**
训练外环或发布前使用的评审者。它看的是结果能不能放行，不负责给 policy 提供梯度。

这组词最容易混的地方在职责，不在字面。两个模型名字一样长，不代表在同一环里；一个函数看起来像打分，也不代表它一定是训练信号。先看环，再看名。

# 为什么会分成两份 {#trace}
<!-- tag: state trace -->

第一次接触后训练时，最容易误会的一点是：既然 policy 和 reference 一开始来自同一个 checkpoint，为什么不干脆让它们一直共用一份权重？答案在于“比较”这件事本身需要一个稳定的参照。如果参照也跟着变，系统就会失去刻度，训练信号只剩自己追自己。

可以把它看成一次很普通的分工：policy 负责今天继续往前走，reference 负责保留昨天的位置，reward 负责告诉你这一步偏好有没有更好，judge 负责在外面看今天的结果值不值得放行。关键是谁在动、谁不动。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

这个分工一旦建立起来，很多错误就会变得很好识别。你看到某个实现把 reference 也训练了，先不要看损失公式，先问一句：比较还剩不剩锚点？如果锚点没了，后面的分数再漂亮也只是漂在空中。

# 最小案例 {#mini-case}
<!-- tag: case -->

把角色放进一个真实一点的客服场景，最容易看出谁在训练内环，谁在训练外环。假设用户问：“订单为什么还没发货？” policy 先生成一条回答，reward 判断这条回答是否足够清楚、是否承认了不确定性、是否给出了下一步动作；reference 则负责给 policy 提供参照，让它知道自己有没有偏离原来的回答风格。

```text
prompt → policy 生成候选

```

这就是你以后看后训练配方时最先要问的事：我现在看到的是训练内环，还是训练外环？如果答案不清楚，就先别看公式，先把对象表补出来。

# DPO 流程 {#dpo}
<!-- tag: code reading -->

后训练页里经常会看到这种最小代码结构。它的作用是让你在看到配方时，能把“谁冻结、谁更新、谁比较”一眼认出来。

policy / reference 分叉复制

```python
# SFT checkpoint 只复制一次，然后分成两个职责
policy = load_checkpoint(sft_checkpoint)
reference = load_checkpoint(sft_checkpoint)
freeze(reference)

for prompt, chosen, rejected in batch:
 score_chosen = policy_score(policy, prompt, chosen)
 score_rejected = policy_score(policy, prompt, rejected)
 ref_gap = policy_gap(policy, reference, prompt, chosen, rejected)
 loss = dpo_loss(score_chosen, score_rejected, ref_gap)
 update(policy, loss)

# reference 不更新；它只提供稳定锚点
```

读这段代码时，你只需要盯住四个动作：两次加载、一次冻结、一次更新。只要 reference 也被送进 update，或者 policy 的分数比较里没有 reference 的影子，整条链就已经和 DPO 的基本意图脱节了。

这类代码最常见的误读是把“同一个 checkpoint 被加载两次”当成重复劳动。实际含义是分工：一个对象继续往前走，一个对象保留原位，比较才有意义。

# 偏好信号怎么进 loss {#preference-signal}
<!-- tag: worked case -->

这一步最值得看的是信号方向。假设同一个 prompt 下有两个回答：chosen 更稳，rejected 更虚。训练先看 policy 有没有把 chosen 的相对分数抬高，把 rejected 压下去。reference 还在旁边提供稳定参照，让这个方向不至于乱飘。

$$
 \Delta = \big(\log \pi_{\text{policy}}(chosen) - \log \pi_{\text{policy}}(rejected)\big)
 - \big(\log \pi_{\text{ref}}(chosen) - \log \pi_{\text{ref}}(rejected)\big)
 $$
 如果 `\Delta` 变大，说明 policy 比 reference 更偏向 chosen；如果 `\Delta` 没朝这个方向走，损失就没有在推动正确的比较关系。重点在相对差。

观测
说明
该检查什么
下一步动作


chosen 得分高，rejected 也一起被抬高
policy 可能在整体变“更会讨好”
reference 是否还在做参照
检查冻结与比较项是否正确


chosen / rejected 都没变
训练信号没有真正进入更新
loss 是否连接到了 policy
检查 optimizer、梯度和数据流


分数很好看，但人工一读就觉得更空
信号可能被 prompt 形式牵着走
judge / reward 是否和测试集同分布
加独立回归集和人工抽检

这就是为什么只看最终分数不够。偏好优化里，真正重要的是一条相对关系有没有被稳定地拉出来。你以后如果看到某个训练说“chosen 变好了”，但没法告诉你 rejected 和 reference 怎么动，那其实还没把链条讲完整。

这个小例子还有一个常见误会：很多人会把 reward、reference、judge 全都当成“给分的东西”。实际差别在于，谁参与更新，谁只给参照，谁只做验收。把这三类动作分开，后面的 DPO 公式会好读很多。

如果你在日志里看到 policy 的 chosen 分数在涨，但 judge 仍然指出回答空泛，下一步就别继续堆训练轮数。先回到这张表，看 reward 关注的是不是偏好方向，judge 看的是不是最终可用性，再决定是修 reference、换评测集，还是回到 SFT 样本本身。

# 例子 {#cases}
<!-- tag: cases -->

**案例 1：DPO 配方**
同一个 SFT checkpoint 先被复制成 policy 和 reference。chosen / rejected 进入比较后，只有 policy 继续走。


**案例 2：GRPO 训练环**
policy 先生成多个候选，reward model 给分，reference 约束偏移，最后 policy 按反馈改写分布。


**案例 3：同模型换角色**
同一个 checkpoint 在一阶段是训练成果，在下一阶段可以变成 policy 起点或 reference 基线。名字没变，职责变了。

这些案例的重点，不在于你记住某个缩写，而在于你能看出同一个对象在不同阶段的身份变化。它在某一页里是训练成果，在下一页里又是起点或基线，这并不矛盾；真正会出问题的是职责没有切开。

```text
base model → SFT checkpoint → {policy, reference}
 ↓
 reward model / rule function
 ↓
 judge / evaluator
```

# 常见错误 {#mistakes}
<!-- tag: debug -->
<!-- layout: grid2 -->

## box
### 错误 1：把 base model 当 chat model
base 只说明它是预训练底座，不说明它已经会按指令回答。直接把它当产品模型，会高估它的服从性。

## box
### 错误 2：让 reference 跟着 policy 一起更新
reference 一旦漂移，就不再是稳定参照。很多相对约束和 KL 控制都会失去意义。

## box
### 错误 3：把 judge 高分当成真实能力
judge 只是评审器，不是训练目标本身。它可能偏向某种措辞、某类题型或某个提示词。

## box
### 错误 4：只看最终分数，不看角色链
后训练里最常见的定位错误，就是把“分数很好”误读成“整个链条都对”。先看谁打分，再看谁被更新。

# 调参前先核一遍 {#debug-loop}
<!-- tag: diagnosis -->

当训练看起来能跑，但结果还是不稳时，先不要换模型、加轮数或改学习率。先按对象顺序核查：policy 是否唯一可训练，reference 是否冻结，reward 和 judge 是否分在不同环，chosen / rejected 是否真的来自同一个 prompt。只要这四项里有一项错位，后面的分数通常都只是表面正常。

这一步的价值在于把问题定位到对象边界，而不是被最终分数带跑。很多训练失败权责表已经错位：该更新的没更新，该冻结的被更新，内环和外环混成了一件事。

症状
常见原因
第一检查
下一步动作


chosen / rejected 一起涨，回答还是空
reward 只看形式，不看任务完成度
reward 的判分标准和样本分布
增加独立评测集和人工抽检


policy 和 reference 一起漂
reference 被错误加入更新
optimizer 参数列表和复制时机
冻结 reference，确认只读不写


judge 打高分，真实样本却不稳
judge 过拟合提示词或题型
judge 看的是训练样本还是留出样本
把 judge 留在外环，用独立回归集校验

四步核查顺序复制

```text
1. 先看 policy 是否在更新
2. 再看 reference 是否冻结
3. 再看 reward / judge 是否分在不同环
4. 最后看 chosen / rejected 是否来自同一 prompt
```

如果这四步里有一步说不清，就先回到前面的角色账本和 trace。对象边界对了，再谈算法、学习率和 batch 才有意义。你也可以直接回到练习区，先答 Q7、Q11 和 Q12，看自己是不是已经把对象和信号分开了。

<!-- quick-review: 速记 policy 更新，reference 冻结，reward 给训练信号，judge 做外环判断。 -->

### 1 choice

**Q:** Q1. 哪个对象通常直接被训练目标更新？

- [ ] A. reference model
- [ ] B. policy model
- [ ] C. judge / evaluator

Answer: B. policy model
Explain: policy model 才是被优化的一方。

### 2 true/false

**Q:** Q2. reference model 在多数 DPO / RLHF 配方里应该冻结。

- [ ] 对
- [ ] 错

Answer: 对
Explain: reference 的价值来自稳定，不来自继续学习。

### 3 fill

**Q:** Q3. 填空：SFT 主要学习的是什么？

Explain: 答案可以写成“指令到回答的映射”“示范答案分布”“按任务输出的行为模式”。关键要说清楚 SFT 学的是可模仿的响应方式。

### 4 choice

**Q:** Q4. 哪个对象更适合用于发布前回归检查？

- [ ] A. judge / evaluator
- [ ] B. reference model
- [ ] C. optimizer state

Answer: A. judge / evaluator
Explain: 发布前需要的是外环判断，不是训练中的梯度信号。

### 5 synthesis

**Q:** Q5. 用一句话区分 reward model 和 judge。

Explain: 一个合格答案要同时说出“reward 负责训练内环的分数或梯度方向”“judge 负责训练外环或发布前判断”“两者可以使用相似模型，但不能混同角色”。

### 6 choice

**Q:** Q6. 为什么同一个 SFT checkpoint 可以同时成为 policy 起点和 reference 起点？

- [ ] A. 让两个对象从同一起点分叉
- [ ] B. 让 reference 继续学习，policy 冻结
- [ ] C. 让 reward 直接替代 policy

Answer: A. 让两个对象从同一起点分叉
Explain: policy 要继续更新，reference 要冻结；同起点便于比较偏离。

### 7 choice

**Q:** Q7. 哪一条最贴近 DPO 的对象顺序？

- [ ] A. policy 和 reference 轮流更新，chosen / rejected 只做展示
- [ ] B. SFT checkpoint 复制成 policy 和 reference，reference 冻结，policy 用 chosen / rejected 和 reference 的差异更新
- [ ] C. judge 先训练，再把分数倒回去改 tokenizer

Answer: B. SFT checkpoint 复制成 policy 和 reference，reference 冻结，policy 用 chosen / rejected 和 reference 的差异更新
Explain: DPO 先分叉出 policy 和 reference，再用偏好对推动 policy 更新。

### 8 true/false

**Q:** Q8. base model 天然等于 chat model。

- [ ] 对
- [ ] 错

Answer: 错
Explain: base 只是底座，不等于已经按指令对齐。

### 9 choice

**Q:** Q9. 在 GRPO 风格训练里，reward function 的主要作用是什么？

- [ ] A. 给候选答案提供可优化分数
- [ ] B. 保存 tokenizer 词表
- [ ] C. 替代所有人工评测

Answer: A. 给候选答案提供可优化分数
Explain: reward 的作用是给训练内环提供方向，不是替代全部评测。

### 10 fill

**Q:** Q10. 填空：reference 常用于计算相对概率或什么约束？

Explain: 通常是 KL 约束。答案只要表达“限制 policy 不要偏离 reference 太远”就算抓住了意思。

### 11 error-diagnosis

**Q:** Q11. 错误诊断：训练时把 reference 也设成 trainable，会出现什么问题？

Explain: 参考答案至少要提到三点：reference 失去稳定参照；policy 的相对偏移不再可解释；KL 或类似约束会失去锚点。修法是把 reference 冻结，并在训练开始前明确复制来源。

### 12 choice

**Q:** Q12. 用一个 LLM judge 打高分就宣布模型上线，最先应该补什么？

- [ ] A. 独立评测集和人工抽检
- [ ] B. 更大的学习率
- [ ] C. 让 reference 继续训练
- [ ] D. 把同一个 judge 继续当 reward

Answer: A. 独立评测集和人工抽检
Explain: 单个 judge 分数不够，先补独立回归和人工核查。

### 13 scenario-transfer

**Q:** Q13. 你要训练一个客服模型，请列出最小后训练对象表。

Explain: 参考答案不需要写很长，但必须包含：起点模型、被更新的 policy、冻结的 reference、提供训练信号的 reward 或规则函数、以及训练外的评测集或 judge。只列算法名不算完成。

### 14 choice

**Q:** Q14. 偏好数据 chosen / rejected 最直接服务于哪个阶段？

- [ ] A. tokenizer 训练
- [ ] B. DPO 或 reward modeling
- [ ] C. 模型量化

Answer: B. DPO 或 reward modeling
Explain: chosen / rejected 是偏好数据，直接供偏好优化或奖励建模使用。

### 15 true/false

**Q:** Q15. 一个规则判分函数既可以是 reward，也可以是 judge。

- [ ] 对
- [ ] 错

Answer: 对
Explain: 同一个函数放进训练内环或评测外环，职责会变。

### 16 fill

**Q:** Q16. 填空：如果模型为了拿高 reward 学会钻漏洞，这通常叫做什么？

Explain: 标准说法是 reward hacking。只要你能说出“模型学会利用奖励函数漏洞”就算抓住了核心。

### 17 synthesis

**Q:** Q17. 解释 prompt、policy、reward、reference、judge 在一次 RLHF 实验中的关系。

Explain: 一份合格答案要把顺序说出来：prompt 进入 policy，policy 生成候选，reward 对候选打分，reference 提供稳定参照，policy 根据 reward 和 reference 的约束更新，judge 在训练外环检查是否真的更好。

### 18 synthesis

**Q:** Q18. 用 150 字以内给新人讲清楚本节。

Explain: 答题时不要堆术语。先说后训练的目的，再用一条短链条解释：base 是底座，SFT 之后得到 checkpoint，policy 负责更新，reference 负责参照，reward 给训练信号，judge 负责外环验收。
