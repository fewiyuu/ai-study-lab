`apply_chat_template`
把结构化对话渲染成模型真正看到的文本。

# 先抓主线 {#overview}
<!-- tag: big picture -->

这页的主线很直接：`messages` 或 `prompt/completion` 先经过模板和 tokenizer，变成一条连续 token 序列；接着用 mask 标出哪些位置要算 loss；最后如果开了 packing，再决定多条样本怎样拼进同一段输入里。
三层责任是分开的。`apply_chat_template` 负责格式；`assistant_only_loss` 和 `completion_only_loss` 负责训练哪些位置；`packing=True` 只负责提高 token 利用率，不负责修数据质量。你在这一页里要盯住的每一步的读写关系。

**这一页先不展开**
DFT、chunked_nll、LoRA、学习率调度和分布式细节会放到后面的训练配置和实验单元里。这里先把样本、mask 和边界看稳。

# 状态账本 {#terms}
<!-- tag: ledger -->

这张表就是这页的状态账本：每一步读什么、写什么、谁在负责、错了先坏哪一层。只要这三层不混，后面的配置和排错会清楚很多。


对象
读什么
写什么
副作用
loss 通常看哪里


`messages`
结构化对话，`system`、`user`、`assistant` 先分开存。
`apply_chat_template()` 把角色和控制 token 变成模型输入。
如果把角色 token 先写进 `content`，模板会再写一遍。
通常只训练 assistant 回复，或至少只让 assistant 区进入 loss。


`prompt/completion`
prompt 是上下文，completion 是答案，边界直接写在字段里。
字段已经给出边界，训练器按这个边界生成 labels。
如果 prompt 也被算进 loss，模型容易复读题目。
默认只看 completion；如果改成全序列训练，要重新检查 prompt 标签。


`add_generation_prompt`
在末尾补一个 assistant 起始位置，让模型知道接下来轮到谁说话。
模板或 tokenizer 负责补，不需要把这段手写进内容里。
如果手写进内容里，再套模板，角色边界会重复。
它影响的是生成起点，训练目标仍按样本标签来定。


`assistant_only_loss`
只让 assistant 回答参与 loss。
模板要能指出 assistant 的生成区。
模板若找不到 assistant 区，mask 也无从落点。
看 assistant 对应的位置是否被标成有效标签。


`completion_only_loss`
只让 completion 部分参与 loss。
prompt/completion 的字段边界要清楚。
边界不清时，prompt 会悄悄进目标区。
看 completion 的标签是否完整，prompt 是否被忽略。


`packing=True`
把多个短样本装进同一个上下文窗口。
packing 只改装箱方式，不改样本内容。
它会把边界压得更紧，边界错了会更难查。
看 EOS、样本边界和 packed 后的 labels 是否串了。


`ignore_index=-100`
在 loss 里忽略某些位置。
标签里用这个值标出不训练的位置。
输入还在，变化的是 loss 标记，文本内容保持原样。
它本身不改文本，只改训练信号。


**先看边界，再看效率**
先认输入格式，再认标签边界，最后才看 packing。顺序一旦倒过来，排错会绕远路。

# 入口选择 {#pathchoice}
<!-- tag: decision tree -->

同一批后训练数据，常见只有三种入口：对话、问答对、纯文本。入口选错，后面的 mask 只能跟着错。你要先确认样本属于哪一类，再决定谁来补模板、谁来划目标区。
这一步的判断很直接：如果原始数据已经有角色，先走 `messages`；如果只有任务和答案，先走 `prompt/completion`；如果是零散文本，就先整理出清楚边界，再送进训练器。边界不清时，先查 labels，模型本身通常排在后面。


样本起点
最合适的入口
第一步先做什么
最容易犯的错


聊天记录、客服记录、角色对话
`messages`
保住角色边界，让模板补齐格式，再设 assistant-only 的目标区。
把 role token 先写进内容，再让模板重复写一遍。


题目 + 标准答案、指令 + 回答
`prompt/completion`
先把 prompt 和 completion 切开，再确认 completion-only 的标签范围。
把 prompt 也当目标区，模型开始复读题目。


散文、文档段落、代码块集合
先预处理，再转成前两类之一
先找出切点和边界，再决定是做问答对还是对话样本。
把一长串文本直接塞进训练器，边界只能靠运气。


简单决策骨架复制
`def choose_sft_path(sample):
 if "messages" in sample:
 return {
 "path": "chat",
 "format": "apply_chat_template",
 "loss": "assistant_only_loss=True",
 "check": "labels only on assistant tokens",
 }

 if "prompt" in sample and "completion" in sample:
 return {
 "path": "completion",
 "format": "prompt/completion",
 "loss": "completion_only_loss=True",
 "check": "prompt labels are ignored",
 }

 return {
 "path": "preprocess_first",
 "format": "manual segmentation",
 "loss": "create clear boundary before training",
 "check": "split text before SFTTrainer sees it",
 }`

**这一步的检查顺序**
先看样本属于哪类，再定模板和标签，最后才谈 packing。只要入口选对，后面很多问题会直接消失；入口选错，后面的参数只是在替错误兜底处理。


**第一失败信号**
如果模型开始学会复述提问，常见原因是入口选错后，prompt 区被当成了目标区。这个时候应该回到这张表，重新看样本属于哪类。

# 手工标一遍 labels {#labelmap}
<!-- tag: trace -->

把标签手工标一遍，比只记两个开关更有用。你要看的只有三件事：模型看见了什么，答案从哪里开始，哪些位置应该被忽略。
下面用同一条最小样本，分别看 `messages` 和 `prompt/completion`。先别管具体 tokenizer id，先看边界。


场景
上下文
目标区
labels 怎么标
为什么这样放


`messages`
`system`、`user` 和模板控制 token。
`assistant` 回复。
上下文位置设成 `-100`，assistant 区保留有效标签。
上下文只提供条件，答案区才应该推动参数更新。


`prompt/completion`
`prompt`。
`completion`。
`prompt` 设成 `-100`，`completion` 保留有效标签。
字段本身已经给了边界，训练器只要按边界算 loss。


packed samples
上一条样本的末尾到 `EOS` 为止。
下一条样本从自己的起点开始。
每条样本内部独立标注，不能把前一条的答案延到下一条。
packing 只改变装箱，不改变样本责任边界。


手工展开示意复制
`# 伪代码：只看边界关系，不依赖具体 tokenizer id
tokens = [
 "<bos>", "<sys>", "请用三句话解释 packing。",
 "<usr>", "loss mask 是什么？",
 "<asst>", "loss mask 会把不该训练的位置标成忽略。",
 "<eos>"
]
labels = [
 -100, -100, -100,
 -100, -100,
 -100, "loss mask 会把不该训练的位置标成忽略。",
 id("<eos>")
]

# prompt/completion 的同类样本更直接：
prompt_tokens = ["<bos>", "loss mask 是什么？"]
completion_tokens = ["loss mask 会把不该训练的位置标成忽略。", "<eos>"]
prompt_labels = [-100, -100]
completion_labels = [id("loss mask 会把不该训练的位置标成忽略。"), id("<eos>")]
`
如果这里的 labels 少遮了一段，上下文就会变成目标；如果目标区少放了一段，模型会学到“看见答案，但不用为答案负责”。这两种错，在训练日志里常常都只表现成一个看似正常的 loss 曲线。

**这一步最有用的检查**
先把一条样本打印成可读文本，再看 labels 是否只覆盖答案区。只要这一步能看对，`assistant_only_loss` 和 `completion_only_loss` 的差别就不再抽象。

# 代码读写与副作用 {#inspect}
<!-- tag: audit -->

真正出问题的时候，最省时间的做法是把一条样本完整打印出来：原始字段、模板化文本、token、labels、EOS、packed 后的边界。这里的目标是找出“哪一步把边界弄坏了”。
你可以把这件事想成四个检查点。先看输入是否已经脏了，再看模板是否重复写了控制 token，再看 labels 是否覆盖了错误位置，最后才看 packing 有没有把多个样本缝错在一起。每一步都在读前一步，也在写下一步的副作用。
样本审计脚本骨架复制
`def inspect_sample(example, tokenizer):
 print("=== raw example ===")
 print(example)

 print("=== formatted text ===")
 if "messages" in example:
 text = tokenizer.apply_chat_template(example["messages"], tokenize=False)
 else:
 text = example["prompt"] + example["completion"]
 print(text)

 print("=== token pieces ===")
 ids = tokenizer(text, add_special_tokens=True)["input_ids"]
 print(ids)
 print(tokenizer.decode(ids))

 print("=== labels view ===")
 labels = build_labels(example, tokenizer)
 for i, (tok, lab) in enumerate(zip(ids, labels)):
 print(i, tok, lab)

 print("=== boundary check ===")
 print("eos seen:", ids[-1] == tokenizer.eos_token_id)
 print("ignore count:", sum(1 for x in labels if x == -100))
`


检查点
读什么
写什么
副作用
下一步


raw example
原始字段是否已经混入 role token 或重复模板。
清理 source data，修正字段边界。
输入在进训练器前就脏了。
先清理 source data，再谈 labels。


formatted text
模板化之后，角色和控制 token 是否出现一次就够。
模板函数生成最终可读文本。
control token 被手写一次、模板又补一次。
把模板职责交给 tokenizer 或模板函数。


token pieces
解码后能不能看到答案区和边界。
生成 input_ids 并保留 EOS。
EOS 缺失，或者答案区和上下文缠在一起。
先关 packing，再单条样本解码。


labels view
哪些位置被标成 `-100`，哪些位置保留有效标签。
mask / labels 对齐。
上下文被训练，或者答案区被整段忽略。
回到边界，重新标一次。


boundary check
EOS 和 packed 边界有没有清晰结束。
packed 后的连续序列。
上一条样本的尾巴接到了下一条的开头。
修边界，再开 packing。


**典型坏样本长什么样**
最常见的坏样本看起来很正常：loss 在降，输出也像在学，但模型把提问和回答一起背了。这个时候先看 labels 是否只覆盖答案区，再看 packed 序列里有没有把前后样本缝成一条。


**检查顺序**
原始字段、模板化文本、labels、packing，四步里只要有一步错了，后面都会被带偏。你要找的是“哪一步改坏了边界”。

# packing 的副作用 {#packing}
<!-- tag: efficiency -->

packing 的目标只有一个：让短样本少浪费 padding，把更多真实 token 塞进同一次前向。它只管装箱，不管样本质量或 mask 对不对。
看 packed 例子时，最好拿两条短样本一起想：A 结束处要有自己的 `EOS`，B 开始处要重新起一段。A 的 labels 只负责 A，B 的 labels 只负责 B。把中间那条边界去掉，训练器就会把两条样本当成一条长答案。
这就是为什么 packing 总是最后开。先单样本确认 labels 对齐，再看两条样本拼起来有没有边界泄漏。只要其中一个位置错了，packing 就会把错误压得更紧，排错更难。
最小配置开关复制
`from trl import SFTConfig

chat_cfg = SFTConfig(assistant_only_loss=True)
prompt_completion_cfg = SFTConfig(completion_only_loss=True)
pack_cfg = SFTConfig(packing=True)
`


检查项
不开 packing 时
开 packing 后
你要额外盯什么


EOS / 样本边界
一条样本一条边界
多条样本共享一个上下文
每段是否都清楚结束，边界是否会串过去


padding
短样本会留空位
空位少很多
别把节省下来的 padding 当成训练质量提升


labels
只和单条样本对齐
必须对 packed 后的连续序列重新检查
有没有跨样本误标、漏标或把上一条答案延到下一条


**先把这三件事检查完，再考虑开 packing**
第一，模板已经正确；第二，EOS 和样本边界能在解码结果里看见；第三，labels 不会把 user/system 也算成目标。少做这一步，packing 只会把问题压得更密。

# 边界说明 {#mistakes}
<!-- tag: pitfalls -->

把 `messages` 当最终训练文本，结果自己先写了一遍 role token，再让模板写一遍。
把 `assistant_only_loss` 理解成“只看 assistant，用户内容完全不进输入”，其实它只是把非目标位置从 loss 里拿掉。
把 `packing=True` 理解成提分开关，忽略了它只解决 token 利用率，不解决数据质量。
把 `-100` 当成“这段不存在”，其实它只是不参与损失，输入和上下文通常还在。


**一个够用的判断顺序**
先看样本、模板、mask，再决定 packing。顺序一旦倒过来，排错会绕远路。

### 5 code-reading

**Q:** Q5. 代码阅读：这条样本里哪些部分应参与 loss？

Explain: 要点有三条：1）system 和 user 仍然进入输入上下文；2）真正作为目标的通常是 assistant 回答；3）如果是 assistant-only loss，非 assistant 位置应被忽略，常见做法是把这些位置的标签设成 -100 。答到“上下文可见”和“参与 loss”是两件事，就算抓住了核心。

### 6 choice

**Q:** Q6. padding 位置通常怎样处理？

- [ ] . 设为 ignore，不参与 loss
- [ ] . 变成新的 assistant 目标
- [ ] . 自动变成更高学习率

Answer: . 设为 ignore，不参与 loss
Explain: padding 会出现在输入里，但不该参与损失。

### 7 true/false

- [ ] 对
- [ ] 错

Answer: 对
Explain: TRL 文档说明它依赖 chat template 中的 generation 标记；有些模型会自动补，但模板未必天然具备。

### 8 choice

- [ ] . 把 completion_only_loss=True
- [ ] . 把 completion_only_loss=False
- [ ] . 把 packing=True

Answer: . 把 completion_only_loss=False
Explain: prompt-completion 数据默认就只算 completion；如果想训练全序列，可以设 completion_only_loss=False。

### 10 true/false

**Q:** Q10. `packing` 能修复错误模板吗？

- [ ] 对
- [ ] 错

Answer: 错
Explain: packing 不负责修数据，只负责把数据更紧凑地装进上下文。

### 11 synthesis

**Q:** Q11. 诊断：启用 packing 后，两个样本的答案像串在一起了。先查什么？

Explain: 至少说到三点：1）先关掉 packing，确认单样本是否正常；2）检查 EOS 和样本边界是否保留；3）打印 packed 后的 decoded 文本和 labels，看有没有跨样本误标。只要能把“边界”和“标签”两件事说出来，方向就对了。

### 12 choice

**Q:** Q12. 打开 `packing` 前，先检查什么最稳妥？

- [ ] . 先把学习率调大
- [ ] . 先确认边界、EOS 和 labels
- [ ] . 先把评测集删掉

Answer: . 先确认边界、EOS 和 labels
Explain: 先看边界、EOS 和 labels，再看效率开关。

### 13 synthesis

**Q:** Q13. 用 80-120 字解释 `assistant_only_loss` 和 `completion_only_loss` 的区别。

Explain: assistant_only_loss 针对 conversational 数据，只在 assistant 回复上算 loss； completion_only_loss 针对 prompt-completion 数据，只在 completion 部分算 loss，默认还会根据数据格式自动决定是否只训 completion。好答案还会指出：不要把 prompt 里本来只是上下文的部分也一起算进目标。
