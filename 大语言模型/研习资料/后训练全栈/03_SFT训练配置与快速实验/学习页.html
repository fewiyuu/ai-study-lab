SFT 训练配置与快速实验

学习目标

start here

本页的目标不是把所有训练参数背下来，而是让你能把一份 SFT 配置拆开，看出每个旋钮在控制什么。

如果你能把一份小实验配置读成“样本怎么进来、token 怎么装、哪些参数被训练、什么时候留日志”，你就已经能开始跑 smoke test 了。

层级 你要看什么 错了通常会怎样 先查哪一步
输入层 数据是 messages、prompt/completion，还是已经整理好的纯文本。 模板和 loss mask 对不上，模型开始学错位置。 回看上一单元的样本边界。
训练层 batch、accumulation、max_length、packing、LoRA、learning_rate。 OOM、训练太慢、更新太猛或容量不够。 看 batch 与显存预算。

先抓主线

big picture

这页的主线很简单：把上一单元已经验证过的样本边界，翻译成一份能跑的小实验。训练器不需要你先想“模型会不会变强”，它先要你回答“这份配置能不能稳定地把数据喂进去，把参数更新出去，并把结果留下来”。

拿一个常见的小实验当锚点更好记：1 张 24GB GPU、一个 0.5B 到 1.5B 的 chat model、3000 条短指令数据。这个规模不追求极限效果，先追求三个东西：能跑、能解释、能回滚。只要这三件事成立，后面才有资格放大。

数据检查 先看样本形状和字段

这里要确认上一单元的 labels、EOS、模板和边界都已经对齐。

smoke test 先用最小配置跑起来

目标是把配置的每个旋钮都放在你能解释的位置上，不是一次拿到最好分数。

读信号 loss、样本和 checkpoint 一起看

只看 loss 不够，输出是否复读、保存是否成功、日志是否完整，都要一起看。

放大实验 确认稳定后再扩大

packing、batch、数据量、步数都可以放大，但要一个一个改，不能一起动。

这一步先不追求所有可能的训练技巧。DFT、分布式并行、FlashAttention 和更复杂的保存策略都可以留到以后；这页先把“怎么起步、怎么判断、怎么回滚”讲清楚。

预算与取舍

state ledger

配置最容易出错的地方，不在单个参数，而在参数之间的预算关系。你改了一个地方，往往是另一层先坏掉。下面这张表就是这页的总账本。

预算 主要看什么 什么时候会变旧 最先出现的症状 下一步先查什么
数据预算 样本字段、数据量、长度分布、是否已经和上一单元的标签边界对齐。 你换了数据源、字段名或模板。 loss 看起来正常，输出风格却很怪。 回看输入格式和模板。
token 预算 每步真正喂给优化器多少 token。 你改了 max_length、packing 或 accumulation。 要么 OOM，要么吞吐低得不合理。 重算有效 batch 和 tokens per step。
内存预算 batch、序列长度、bf16、gradient checkpointing、LoRA 容量。 你改了模型大小或 target modules。 训练前几个 step 就炸显存。 先降长度或 micro batch。
参数预算 全参还是 LoRA，LoRA 的 r、alpha 和 target_modules。 你换了模型结构或任务难度。 输出学不动，或者学得太快、太窄。 看可训练参数是否够。
观测预算 日志、验证、保存、随机种子、输出目录。 你缩短了训练步数或换了实验路径。 跑完了却不知道哪里坏，或者没有可回滚的 checkpoint。 先修 logging 和 save 策略。

这页的判断顺序很稳定：先看输入，再看 token 和内存，再看参数容量，最后看观测。顺序反过来，最容易把问题误判成“模型不行”。

最小配方

smoke test

最小配方的作用只有一个：让你在最少变量下看见配置是否自洽。先把第一轮跑通，确认 loss、样本输出、checkpoint 都能对上，再决定是否开 packing、加数据或改更大的学习率。

smoke 配方
from trl import SFTConfig

smoke = SFTConfig(
    output_dir="outputs/sft-smoke",
    per_device_train_batch_size=1,
    gradient_accumulation_steps=8,
    max_length=2048,
    packing=False,  # 第一轮先关掉，等标签和边界稳了再开
    bf16=True,
    gradient_checkpointing=True,
    learning_rate=2e-4,
    logging_steps=5,
    save_steps=50,
    save_total_limit=2,
    seed=42,
)
步骤 要做什么 你要看的信号 如果失败，先退回哪里
第一轮 小 batch、短步数、packing 先关掉。 loss 是否下降、输出是否开始像任务答案、checkpoint 是否真的落盘。 先退回样本边界和标签。
第二轮 同一份数据下，先只开 packing，不改别的旋钮。 吞吐有没有变好、输出边界有没有串味。 如果串味，先关 packing 再看 EOS。
第三轮 确认稳定后，再只改一个变量，比如 batch 或学习率。 改动是否真的带来更好的速度或更稳的收敛。 如果变化说不清，回到第一轮。

这页里最值得记住的一句话是：先把第一轮做成“能解释”，再把第二轮做成“更高效”。如果顺序反了，packing 和大 batch 只会把问题压得更密。

一轮走查

concrete case

拿一组很小的训练样本做走查，比只看配置名更容易发现问题。这里用四条短指令数据作为锚点：它们都很短，适合看 packing、长度和日志。

锚点数据
1. “请用三句话解释 packing。”
2. “把这个客服回复改得更稳一点。”
3. “列出三个常见 SFT 失败信号。”
4. “说明为什么先关 packing 再排错。”
你看到的对象 在配置里对应什么 应该怎么判断 如果走错，最先坏什么
短样本很多 packing、max_length、gradient_accumulation_steps 先算 token 利用率，再决定要不要拼窗。 样本很短却占满上下文，padding 浪费明显。
单卡 24GB per_device_train_batch_size、bf16、gradient_checkpointing 先把第一轮跑起来，再考虑把 batch 或长度往上推。 一上来就 OOM，或者只能靠猜哪一步把显存撑爆。
想要任务样式变化 use_peft、lora_r、target_modules、learning_rate 先确认数据和模板，再看 LoRA 有没有接到正确层。 容量不够时看起来像没学，容量过大时又会把风格推歪。
需要可回看证据 logging_steps、save_steps、save_total_limit、output_dir 确认训练中段和最终点都能找到。 实验结束后只能凭记忆说“好像变了”。
$$ B_{\text{eff}} = D \times B \times A,\quad T_{\text{step}} \approx B_{\text{eff}} \times L $$

先看 B_eff，再看 L。B_eff 决定每次更新看多少样本，L 决定这些样本平均有多长。packing 改的是 L，gradient accumulation 改的是 B_eff，它们不在同一层。

失败信号 说明 先退回哪一步
loss 下降很快，但输出里还在复读 prompt 数据层和训练层没有接上，标签或模板大概率还没对齐。 先回上一单元重看样本边界。
开 packing 后速度变好，但输出边界变乱 装箱方式改变了，样本边界没有重新确认。 先关 packing，再单样本解码。
checkpoint 有了，但中间过程不好解释 观测层设置太弱，训练过程没有留下足够信号。 先调 logging_steps 和 save_steps。

这段走查的重点，是把“看配置”变成“看路径”。样本先经过模板，随后进入 token 预算，再进入参数更新，最后留下日志和 checkpoint；哪一步都不能跳。你只要能把这条路径说清，后面的调参就不会乱。

batch 与显存

math + trace

batch 的问题，表面上是在问“每次喂多少样本”，实际上是在问“每次优化器更新到底看了多少 token”。这个问题如果只看 per_device_train_batch_size，会漏掉梯度累积和序列长度。

$$ B_{\text{eff}} = D \times B \times A $$

D 是设备数，B 是每卡 micro batch，A 是 gradient accumulation steps。这个式子只数样本，不数 token。

$$ T_{\text{step}} \approx D \times B \times A \times L $$

L 是处理后的平均序列长度。开 packing 以后，L 往往更接近窗口上限；关掉 packing 时，L 更接近原始样本长度的平均值。

旋钮 它在读什么 它在写什么 调大之后最容易发生什么 第一检查点
per_device_train_batch_size 每张卡一次装几条样本。 micro batch。 显存直接上升，step 变慢。 先看是不是 OOM。
gradient_accumulation_steps 要攒几次梯度再更新一次。 有效 batch 和更新频率。 更新间隔变长，日志的 step 含义更容易看错。 区分 micro batch 和 optimizer step。
max_length 单条样本允许保留多长。 序列长度上限。 显存和计算成本明显上升。 先确认你真的需要这么长。
packing 短样本是否被拼进同一窗口。 平均序列长度和 padding 浪费。 吞吐变好，但边界更难看，标签一旦错会被压得更紧。 先确认 EOS 和 labels 已经对齐。
快速估算
def estimate_tokens_per_step(devices, micro_batch, accum, avg_len):
    effective_batch = devices * micro_batch * accum
    tokens_per_step = effective_batch * avg_len
    return effective_batch, tokens_per_step

# 例子：1 张卡、micro batch=1、accum=8、平均长度约 1024
# effective_batch = 8
# tokens_per_step ≈ 8192

如果你算出来的 token 量和显存表现对不上，优先检查是不是把 padding、packing 或平均长度估错了，而不是急着改学习率。

packing

efficiency

packing 的目标只有一个：把短样本之间的空白压下去，让 GPU 少算 padding。它解决的是效率，不是质量。坏样本、错模板、漏标签，packing 一个都修不了。

检查项 不开 packing 时 开 packing 后 你要额外盯什么
EOS / 样本边界 一条样本一条边界。 多条样本共享一个上下文窗口。 每段是否都清楚结束，边界是否会串过去。
padding 短样本会留空位。 空位少很多。 别把节省下来的 padding 当成训练质量提升。
labels 只和单条样本对齐。 必须对 packed 后的连续序列重新检查。 有没有跨样本误标、漏标或把上一条答案延到下一条。
先后顺序
1. 先关 packing 跑通
2. 单样本解码，确认 EOS 和 labels
3. 再开 packing，看边界是否串味
4. 若串味，先关 packing，再回去看上一单元

如果你还在读样本边界，packing 可以先不碰。把它留到第一轮 smoke test 之后，会更稳。

这里先别追 DFT、chunked NLL、FlashAttention、分布式 sharding 和更复杂的吞吐优化；它们会在后面的系统化训练内容里重新出现。

LoRA 与学习率

capacity

LoRA 解决的是“改多少参数”和“用多大步子”这两个问题。r 决定可训练容量，target_modules 决定哪几层真的被接上可训练分支，learning_rate 决定每次更新迈多大步。

LoRA 配方
from trl import SFTConfig

lora_smoke = SFTConfig(
    use_peft=True,
    lora_r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    target_modules=[
        "q_proj",
        "k_proj",
        "v_proj",
        "o_proj",
    ],
    learning_rate=2e-4,
)
旋钮 它读什么 它改什么 调大后可能出现什么 先看哪类信号
lora_r 任务需要多少可训练容量。 低秩分支的宽度。 容量更大，但更容易过拟合或显存更紧。 先看输出风格是否还很僵硬。
target_modules 模型里真正存在什么线性层名字。 哪些层接 LoRA。 名字写错了，可能根本没训练到你以为的层。 先核对模型结构里的模块名。
lora_alpha LoRA 更新的缩放关系。 低秩更新的幅度。 缩放太大，输出容易跑偏。 看训练早期样本是否迅速失真。
learning_rate 当前数据规模和 LoRA 容量。 每次参数更新的步幅。 太小像没学，太大容易把风格推歪。 先看 loss 曲线和少量样本输出。

一个常见误区是：把 LoRA 学不动直接理解成“模型容量不够”，然后马上改成全参训练。通常更稳的做法是先检查数据、target_modules 和学习率，再决定要不要增加容量。

日志与 checkpoint

observability

训练不只是更新参数，还要留下证据。没有日志和 checkpoint，失败后你只能猜。这个部分要做的事很简单：让训练过程可读、可回滚、可对比。

项目 读什么 写什么 变旧的信号 第一动作
logging_steps 你多久能看到一次训练信号。 loss、lr、step 等日志。 日志太稀，坏掉了也看不出来。 把日志频率调到能看见趋势。
save_steps 训练在哪些点落盘。 checkpoint 目录。 保存太少，出问题时没有中间点。 至少留一个中间点和最终点。
save_total_limit 磁盘还能留多少份历史。 自动清理旧 checkpoint。 实验很多时，旧目录把盘吃满。 限制历史版本数量。
output_dir 你想把这次实验放到哪里。 配置、状态、日志、checkpoint。 多个实验混到一个目录里。 每次实验用独立目录。
seed 你要不要尽量让实验可复现。 初始化随机状态。 同一份配置两次跑法差很大。 固定种子后再比较配置。
推荐目录形状
outputs/
  sft-smoke/
    config.json
    trainer_state.json
    checkpoint-50/
    checkpoint-100/
    samples_before_after.md
    notes.md

日志不只是给“以后查看”准备的，它还是帮你判断这次实验有没有资格继续放大的证据。看不见信号的训练，通常不值得继续堆步数。

排障顺序

terminal lab

当训练出问题时，最省时间的顺序通常不是“先改学习率”，而是先看症状，再把问题收回到最小检查点。下面这个小型失败实验就是为了训练这个顺序。
你看到的症状 最可能的原因 第一检查点 下一步动作
一上来就 OOM batch、max_length、packing 或显存设置太激进。 先降 micro batch 或长度。 再开 gradient checkpointing，看是否过关。
loss 长时间不动 学习率太小、数据太少、或者标签没有真正参与训练。 先看 labels 和可训练参数。 再检查 learning_rate。
输出开始复读题目或角色标记 数据模板、loss mask 或数据形状和当前配置不匹配。 先回到上一单元，重看样本边界。 再检查 assistant_only_loss / completion_only_loss。
checkpoint 落盘慢 保存频率过高、目录太慢、或者保存了不必要的完整权重。 先调 save_steps 和 save_total_limit。 再看是否只保存 adapter。

这页里最重要的不是记住每个报错，而是建立顺序：先缩小问题，再找原因，最后才改参数。这样你不会把本来简单的边界问题，误判成训练算法问题。

常见误区

boundary note

这页有几个很容易被顺手追下去的概念，但现在不值得分心。先把它们放到边上，等后面的单元再回来。

概念 现在先别追什么 它会在哪一页回来
DFT / chunked NLL 先别把这些当成 smoke test 的默认选项。 更深入的训练实现或大规模配置页。
FlashAttention / 低层加速 先别用底层优化替代 batch 和长度的检查。 系统和性能单元。
分布式并行 / sharding 先别把多卡复杂度和最小实验混在一起。 更大规模训练的工程页。
偏好优化 / reward 先别提前展开算法细节。 下一单元和再下一单元。

最常见的错误，是一上来就把所有技巧都叠进去：packing、LoRA、长上下文、大学习率、复杂保存策略一起上。更稳的顺序是先做一个能解释的 smoke test，再逐个放大变量。

Q6. 用一句话说明为什么 packing 不能修复错误模板。

参考答案 / 评分要点

合格答案至少要点出两层：packing 只是把短样本拼进同一窗口，它不会修正原始样本里已经写错的模板，也不会自动把错误的 labels 变对。它省的是 padding，修正边界要靠前面的样本检查。

Q9. 解释为什么你通常先看数据，再看是否该增大 LoRA。

答案要至少提到“容量不够”和“数据没对齐”是两类问题，不能一股脑都往容量上推。

参考答案 / 评分要点

好答案会先说数据和模板要先对齐，因为如果标签边界都不对，增大 LoRA 只是在更大容量上学错东西。只有在数据已经稳定、输出仍然太僵硬或太窄时，才值得考虑增大 r、换 target_modules 或调整学习率。
