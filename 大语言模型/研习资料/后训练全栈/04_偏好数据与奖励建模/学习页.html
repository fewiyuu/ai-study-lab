学习目标

what

读样本

能判断一条偏好样本是不是同一任务下的可比样本，还是只是字段齐全。

读风险

能把噪声、RM 过拟合和 reward hacking 分开，知道下一步该先查哪一层。

先抓主线

overview

偏好数据与奖励建模这一页，核心看清一条短链路：同一个 prompt 生成多个回答，标注或规则先选出更好的那一个，奖励模型再把这个偏好学成分数，后面的策略优化再去追这个分数。

2. 偏好判断

标注者或 judge 根据 rubric 选出 chosen / rejected。

4. 策略优化

policy 继续朝高 reward 方向走，同时保留 reference 作为锚点。

偏好样本长什么样

chosen

直觉

偏好样本同一个上下文里两个回答的相对判断。chosen 只是这一对里更合适的那个，不代表它已经完美。

先看这几个字段

prompt 提供上下文，chosen 和 rejected 提供比较对象，rubric 说明胜负依据，metadata 记录样本怎么来的。

字段它在读什么它在写什么一眼看不出的风险
prompt任务约束、上下文、用户意图给比较提供同一问题prompt 太短，任务目标不清
chosen胜出的回答供 reward model 读为正例chosen 只是相对更好，不等于最终最优
rejected输掉的回答供 reward model 读为负例rejected 如果太差，模型只会学到边角信号
rubric评价准则把“更好”具体化rubric 模糊时，偏好会混入个人口味

偏好样本 JSONL 示例

复制

{
 "prompt": "请解释为什么后训练需要偏好数据。",
 "chosen": "偏好数据让模型知道两个回答里谁更好，而不只是模仿一个标准答案。",
 "rejected": "偏好数据就是更高级的数据，模型看完就会自动变聪明。",
 "metadata": {
 "rubric": ["正确性", "具体性", "是否回答了任务"],
 "annotator_count": 3,
 "agreement": "2/3"
 }
}

判断一条样本值不值得训练，别只看字段齐不齐。先问三件事：同一个 prompt 吗，胜负理由能复核吗，rubric 真的在管任务而不是管口味吗。

样本到分数的角色链

state

这一页最容易弄混的，不是公式，而是对象职责。样本负责提供比较材料，reward model 负责把比较学成分数，policy 负责继续被更新，reference 负责留住锚点，judge 负责在训练外面做检查。

标注噪声怎么进来

noise

偏好数据脏，通常不是因为“有人手滑”这么简单。更常见的是任务目标没说清、rubric 写得太散、两个回答差别太小，或者展示顺序、长度、语气把判断带偏了。

$$ loss=-\log \sigma\!\left(r_\phi(x, chosen)-r_\phi(x, rejected)\right) $$

这个式子只要求 chosen 的分数比 rejected 高。它不需要绝对分数有统一尺度，只需要排序方向对。

先把这几件事做完

  • 把 rubric 拆成可判断的维度，不要把“好答案”写成一句空话。
  • 给 tie / equally good / cannot judge 留出口，别强迫二选一。
  • 随机化展示顺序，并保留长度、位置、模型来源这些元数据。
  • 用 held-out preference set 看稳定性，不要只看训练 loss。

Reward Model 怎么学

reward

reward model 的工作给回答打分。输入是 prompt 和候选回答，输出是一个标量。训练时，它看成对偏好；使用时，它可以给多个候选排序，也可以给 RLHF 里的 policy 更新提供奖励。

它不该被怎么用

reward 值通常不能跨模型、跨数据集直接比大小。真正该看的，是排序是否稳定、差值是否合理、换一批 prompt 以后还准不准。

reward model 最小训练逻辑

复制

for batch in preference_pairs:
 prompt = batch["prompt"]
 chosen = batch["chosen"]
 rejected = batch["rejected"]

 r_chosen = reward_model(prompt, chosen) # 标量分数
 r_rejected = reward_model(prompt, rejected) # 标量分数

 margin = r_chosen - r_rejected
 loss = -log_sigmoid(margin).mean()
 loss.backward()
 optimizer.step()
阶段输入输出要盯的检查点
偏好标注prompt + 多个候选回答chosen / rejectedrubric、可比性、展示偏差
RM 训练成对偏好样本标量 reward排序准确率、过拟合、分数校准
Policy 优化prompt + policy 采样回答更新后的策略reward hacking、KL 失控、能力退化
离线评测held-out prompts + 人工样本胜率、错误类型、风险结论评测集污染、judge 偏差、只看单指标

具体算一遍：一对样本怎么变成梯度

mechanism

偏好建模最容易学空，是因为很多讲法只停在“chosen 要高，rejected 要低”这一句。真正有用的理解，要能把一条样本拆成数值，再看这个数值会怎样推动更新。

如果顺序写反了

若标签噪声把 chosen / rejected 反过来，差值会朝错误方向推动。你看到的“梯度在把模型往错的方向拉”。

步骤输入 / 数值输出 / 含义如果错了会发生什么
取分数r_chosen = 1.40，r_rejected = 0.20先得到一对可比较的标量如果输入维度错了，RM 只能输出乱分
算差值margin = 1.40 - 0.20 = 1.20chosen 比 rejected 高如果差值接近 0，模型还不够确定
过 lossloss = -log_sigmoid(1.20)把“更偏好 chosen”变成可优化目标如果把 margin 写反，梯度方向会错
看结果训练后拿 held-out pairs 再测检查排序是否真的稳定只看训练 loss 可能误判已经学会

数值走查小例子

复制

chosen = 1.40
rejected = 0.20
margin = chosen - rejected = 1.20
loss = -log_sigmoid(margin)

如果把标签翻转：
chosen = 0.20
rejected = 1.40
margin = -1.20
loss 会推动模型朝错误方向更新

这一步看懂以后，很多噪声问题就能立刻落地。你不再只说“数据脏”，而是能具体说出脏在哪里：差值本来应该往一个方向走，结果标签把它推反了。这样查问题，比只喊“要清洗”有用得多。

这类样本最常见的失真点

  • 回答差异很小，但标注者硬要分胜负。
  • chosen 更长，却不是更正确。
  • rejected 太差，模型只学到“垃圾回答不好”，没学到“好答案长什么样”。
  • 样本来自不同任务，比较本身就不成立。

把它接到下一步真正要训练 policy 的时候，你已经不该再纠结“样本格式齐不齐”了。你要问的是：这条样本学出来的偏好，会不会被 policy 进一步放大成投机行为。

rubric：把“更好”写清楚

rubric

偏好数据常常因为“更好”没有写清。拿同一个客服问题来说，chosen 可能是先确认退款条件再给步骤，rejected 可能是只回一句“请耐心等待”。如果 rubric 只写“helpful”，不同标注者会把更长、更积极、更多格式，甚至更像客服的话术都算进去。

一个坏 rubric 常见什么样

“更专业”“更自然”“更完整”这类词看起来顺眼，实际上几乎没有可判定边界。它们会让标注者把个人审美写进数据，也让后面的 RM 学歪。

维度好写法坏写法
正确性是否真的回答了任务，是否有事实错误“更对”“更像答案”
可执行性用户拿到后能不能直接照着做“更有帮助”
安全边界该拒绝时有没有拒绝，该提醒时有没有提醒“更稳妥”
格式要求标题、长度、列表、引用是否按任务要求来“更专业”

rubric 最小模板

复制

1. 正确性：有没有答对任务
2. 可执行性：用户能不能直接用
3. 安全边界：该拒绝时有没有拒绝
4. 格式要求：是否满足任务指定的形式

如果你自己改不出这四项，就别急着训练。先把 rubric 写成能判断的句子，再去采样和比较。训练前多花一点时间，把“更好”说清楚，通常比后面修一堆 reward hacking 便宜得多。

实践钩子打开你手头的一条偏好样本，把“更好”改成四个能判定的维度；如果改不出来，这条样本还不适合直接训练。

失败实验室：错一层会怎样

failure

这一页真正想让你学会的，把错误分门别类。很多训练问题看起来像一回事，实际上错在完全不同的层上。下面这张表就是最短的排查路线。

这个实验室想提醒你的，是排查顺序。别一上来就改 loss 或换 trainer。先看样本，再看 RM，再看 policy。越是后训练，越要先弄清楚问题到底落在哪一层。

Reward Hacking 怎么出现

proxy

reward hacking 的意思很直接：模型把奖励模型当成目标本身去优化，结果学会了骗分，而不是把任务真的做对。它看起来像训练正常推进，实际上只是代理目标被钻了空子。

训练时的防护

  • 保留 reference model，让 policy 不是无锚点地往前冲。
  • 用 KL 约束控制偏离，不让奖励成为唯一方向。
  • 把 reward 和内容质量拆开看，别让一个分数背太多责任。

评测时的防护

  • 抽看最高 reward 样本，专门找高分低质的反例。
  • 保留人类偏好测试集，不让 policy 训练直接见到。
  • 做对抗提示、边界任务和回归评测，先看退化再谈发布。

操作桥：如果你明天就要开做

next

假设你要给一个客服模型做偏好建模。你手里已经有 SFT 后的模型，也已经能从同一个 prompt 采样出两个候选回答。现在不要急着进复杂算法，先把这四步做稳：

代码
1. 同一 prompt 的候选回答能被比较。
2. rubric 里已经写清楚“为什么 chosen 胜出”。
3. reward model 只负责排序，不负责最终放行。
4. reference model 冻结，policy 单独更新。
5. held-out preference 和人工抽检都已经准备好。

如果这五条里有一条说不清，先回到数据和评测，不要先改 trainer。

下一步怎么接

bridge

这里先不往下追

这一页到这里就停：DPO / ORPO 的完整目标式、PPO / GRPO 的 rollout 和 policy 更新、judge prompt 的设计，都放到后续单元。先把偏好样本、reward model 和 hacking 看清，再往下读会顺很多。

下一步你该去看什么为什么放在下一页
DPO / ORPOreference model、pairwise objective、隐式 reward这一页只铺偏好数据和 reward 的地基
PPO / GRPOrollout、KL、采样、policy update训练环比这页更长，先把角色链看懂再进环
评测与回归held-out preference、人工抽检、能力折损这一页已经告诉你为什么只看 reward 不够

复习时的优先级如果你还分不清 chosen 和 reward 的关系，就回到结构和 trace；如果你已经能说清这条链，再往下一页会更稳。
