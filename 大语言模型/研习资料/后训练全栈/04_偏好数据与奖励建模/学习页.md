# 先抓主线 {#overview}
<!-- tag: overview -->

偏好数据与奖励建模这一页，核心看清一条短链路：同一个 prompt 生成多个回答，标注或规则先选出更好的那一个，奖励模型再把这个偏好学成分数，后面的策略优化再去追这个分数。

**1. 候选回答**
同一个 prompt 下采样多个回答，先让它们真的可比。


**2. 偏好判断**
标注者或 judge 根据 rubric 选出 chosen / rejected。


**3. 奖励学习**
reward model 学的是“谁更好”，不是“答案长什么样”。


**4. 策略优化**
policy 继续朝高 reward 方向走，同时保留 reference 作为锚点。

这一页先不追 DPO、ORPO、PPO、GRPO 的完整公式。先把样本、分数和风险看清，后面的单元才不会变成背缩写。

# 样本到分数的角色链 {#trace}
<!-- tag: state trace -->

这一页最容易弄混的，不是公式，而是对象职责。样本负责提供比较材料，reward model 负责把比较学成分数，policy 负责继续被更新，reference 负责留住锚点，judge 负责在训练外面做检查。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

主要读取
主要写入

reward model
prompt、chosen、rejected、rubric

回答分布
分数涨了，人看却变差


reference
policy 的偏离程度
不写入

**排错顺序**先看谁在读，谁在写，再看谁不该动。只要 reference 还会被优化，或者 judge 被塞回训练内环，这条链就已经歪了。

# 标注噪声怎么进来 {#noise}
<!-- tag: noise -->

偏好数据脏，通常不是因为“有人手滑”这么简单。更常见的是任务目标没说清、rubric 写得太散、两个回答差别太小，或者展示顺序、长度、语气把判断带偏了。

$$P(chosen \succ rejected \mid x)=\sigma\!\left(r_\phi(x, chosen)-r_\phi(x, rejected)\right)$$
 $$loss=-\log \sigma\!\left(r_\phi(x, chosen)-r_\phi(x, rejected)\right)$$
 这个式子只要求 chosen 的分数比 rejected 高。它不需要绝对分数有统一尺度，只需要排序方向对。

噪声类型
表面现象
先查什么


任务歧义
一个回答更详细，另一个更简洁，谁都没错。
rubric 是否把目标拆开了。


标注者分歧
同一对样本在不同人手里反过来。
一致性统计和复核比例。


展示偏差
长回答、靠前位置、固定格式更容易赢。
答案顺序是否随机化，是否记录长度和位置。


过弱对比
rejected 太差，模型只能学到低价值边角。
候选是否来自同一任务、难度是否接近。

## box soft
### 先把这几件事做完
- 把 rubric 拆成可判断的维度，不要把“好答案”写成一句空话。
- 给 tie / equally good / cannot judge 留出口，别强迫二选一。
- 随机化展示顺序，并保留长度、位置、模型来源这些元数据。
- 用 held-out preference set 看稳定性，不要只看训练 loss。

# Reward Model 怎么学 {#reward-model}
<!-- tag: reward model -->

reward model 的工作给回答打分。输入是 prompt 和候选回答，输出是一个标量。训练时，它看成对偏好；使用时，它可以给多个候选排序，也可以给 RLHF 里的 policy 更新提供奖励。

## box info
### 它在读什么
读的是 prompt、候选回答、偏好标签和 rubric。它关心的是“谁更好”，不是“这段文字是否像标准答案”。

## box warn
### 它不该被怎么用
reward 值通常不能跨模型、跨数据集直接比大小。真正该看的，是排序是否稳定、差值是否合理、换一批 prompt 以后还准不准。

reward model 最小训练逻辑
复制

```python
for batch in preference_pairs:
 prompt = batch["prompt"]
 chosen = batch["chosen"]
 rejected = batch["rejected"]

 r_chosen = reward_model(prompt, chosen) # 标量分数
 r_rejected = reward_model(prompt, rejected) # 标量分数

 margin = r_chosen - r_rejected
 loss = -log_sigmoid(margin).mean()
 loss.backward()
 optimizer.step()
```

阶段
输入
输出
要盯的检查点


偏好标注
prompt + 多个候选回答
chosen / rejected
rubric、可比性、展示偏差


RM 训练
成对偏好样本
标量 reward
排序准确率、过拟合、分数校准


Policy 优化
prompt + policy 采样回答
更新后的策略
reward hacking、KL 失控、能力退化


离线评测
held-out prompts + 人工样本
胜率、错误类型、风险结论
评测集污染、judge 偏差、只看单指标

# 具体算一遍：一对样本怎么变成梯度 {#mechanism}
<!-- tag: mechanism -->

偏好建模最容易学空，是因为很多讲法只停在“chosen 要高，rejected 要低”这一句。真正有用的理解，要能把一条样本拆成数值，再看这个数值会怎样推动更新。

## box soft
### 假设这条样本长这样
同一个 prompt 下，`chosen` 的 reward 是 `1.40`，`rejected` 的 reward 是 `0.20`。差值是 `1.20`，说明模型已经偏向 chosen，但还不一定足够稳。

## box info
### 如果顺序写反了
若标签噪声把 chosen / rejected 反过来，差值会朝错误方向推动。你看到的“梯度在把模型往错的方向拉”。

步骤
输入 / 数值
输出 / 含义
如果错了会发生什么


取分数
`r_chosen = 1.40`，`r_rejected = 0.20`
先得到一对可比较的标量
如果输入维度错了，RM 只能输出乱分


算差值
`margin = 1.40 - 0.20 = 1.20`
chosen 比 rejected 高
如果差值接近 0，模型还不够确定


过 loss
`loss = -log_sigmoid(1.20)`
把“更偏好 chosen”变成可优化目标
如果把 margin 写反，梯度方向会错


看结果
训练后拿 held-out pairs 再测
检查排序是否真的稳定
只看训练 loss 可能误判已经学会

数值走查小例子
复制

```text
chosen = 1.40
rejected = 0.20
margin = chosen - rejected = 1.20
loss = -log_sigmoid(margin)

如果把标签翻转：
chosen = 0.20
rejected = 1.40
margin = -1.20
loss 会推动模型朝错误方向更新
```

这一步看懂以后，很多噪声问题就能立刻落地。你不再只说“数据脏”，而是能具体说出脏在哪里：差值本来应该往一个方向走，结果标签把它推反了。这样查问题，比只喊“要清洗”有用得多。

## box warn
### 这类样本最常见的失真点
- 回答差异很小，但标注者硬要分胜负。
- chosen 更长，却不是更正确。
- rejected 太差，模型只学到“垃圾回答不好”，没学到“好答案长什么样”。
- 样本来自不同任务，比较本身就不成立。

**把它接到下一步**真正要训练 policy 的时候，你已经不该再纠结“样本格式齐不齐”了。你要问的是：这条样本学出来的偏好，会不会被 policy 进一步放大成投机行为。

# rubric：把“更好”写清楚 {#rubric}
<!-- tag: rubric design -->

偏好数据常常因为“更好”没有写清。拿同一个客服问题来说，chosen 可能是先确认退款条件再给步骤，rejected 可能是只回一句“请耐心等待”。如果 rubric 只写“helpful”，不同标注者会把更长、更积极、更多格式，甚至更像客服的话术都算进去。

## box soft
### 一个好 rubric 要做到什么
它要把“正确性、可执行性、安全边界、格式要求”拆开，让标注者知道自己到底在比什么，而不是把所有偏好塞进一个“更好”的词里。

## box warn
### 一个坏 rubric 常见什么样
“更专业”“更自然”“更完整”这类词看起来顺眼，实际上几乎没有可判定边界。它们会让标注者把个人审美写进数据，也让后面的 RM 学歪。

维度
好写法
坏写法


正确性
是否真的回答了任务，是否有事实错误
“更对”“更像答案”


可执行性
用户拿到后能不能直接照着做
“更有帮助”


安全边界
该拒绝时有没有拒绝，该提醒时有没有提醒
“更稳妥”


格式要求
标题、长度、列表、引用是否按任务要求来
“更专业”

rubric 最小模板
复制

```text
1. 正确性：有没有答对任务
2. 可执行性：用户能不能直接用
3. 安全边界：该拒绝时有没有拒绝
4. 格式要求：是否满足任务指定的形式
```

如果你自己改不出这四项，就别急着训练。先把 rubric 写成能判断的句子，再去采样和比较。训练前多花一点时间，把“更好”说清楚，通常比后面修一堆 reward hacking 便宜得多。

**实践钩子**打开你手头的一条偏好样本，把“更好”改成四个能判定的维度；如果改不出来，这条样本还不适合直接训练。

# Reward Hacking 怎么出现 {#hacking}
<!-- tag: proxy failure -->

reward hacking 的意思很直接：模型把奖励模型当成目标本身去优化，结果学会了骗分，而不是把任务真的做对。它看起来像训练正常推进，实际上只是代理目标被钻了空子。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

问题
典型表现
修复方向


标注噪声
样本本身把胜负标错，或者 rubric 不一致。
清洗数据、改 rubric、多标注者复核。


RM 过拟合
训练和验证上都像样，换分布就失真。
扩充验证集、正则化、早停、覆盖难例。


Reward hacking
policy 找到骗过 RM 的表面模式。
限制优化强度、更新 RM、加人工红队和多目标评测。

## box soft
### 训练时的防护
- 保留 reference model，让 policy 不是无锚点地往前冲。
- 用 KL 约束控制偏离，不让奖励成为唯一方向。
- 把 reward 和内容质量拆开看，别让一个分数背太多责任。

## box soft
### 评测时的防护
- 抽看最高 reward 样本，专门找高分低质的反例。
- 保留人类偏好测试集，不让 policy 训练直接见到。
- 做对抗提示、边界任务和回归评测，先看退化再谈发布。

# 三个例子 {#examples}
<!-- tag: worked cases -->

例 1：一条能直接训练的偏好样本
prompt 说的是“解释为什么后训练需要偏好数据”，chosen 讲清楚相对判断和 reward 代理，rejected 只给空泛赞美。这个样本能训练，因为胜负理由和任务本身对得上。


例 2：看起来完整，其实不可比
chosen 在解释任务，rejected 在跑题；字段齐了，信息却不在同一条线上。这样的样本会让模型学到很差的边角信号。


例 3：reward 高了，人却不满意
训练后回答越来越长，格式越来越漂亮，但事实错得更多。这个时候先看高 reward 样本，再看 rubric，最后才是调训练强度。


# 操作桥：如果你明天就要开做 {#operation-bridge}
<!-- tag: next action -->

假设你要给一个客服模型做偏好建模。你手里已经有 SFT 后的模型，也已经能从同一个 prompt 采样出两个候选回答。现在不要急着进复杂算法，先把这四步做稳：

步骤
你在做什么
你在检查什么
做错时会怎样


准备样本
同一 prompt 下采样两个回答，写清 rubric。
两个候选是不是同一任务，胜负理由能不能复核。
样本齐了，但比较没成立。


训练 RM
让 chosen 的分数高于 rejected。
排序准确率、验证集表现、分数差值。
训练集看着好，换 prompt 就掉。


约束 policy
在 policy 更新时保留 reference 锚点。
reference 是否冻结，KL 是否在控制漂移。
policy 会把分数涨幅变成投机模式。


外环检查
用人工抽检和 held-out preference 看结果。
高 reward 样本是否真的更好。
你会把“分数上涨”误读成“任务完成”。

开工前的最小配置清单
复制

```text
1. 同一 prompt 的候选回答能被比较。
2. rubric 里已经写清楚“为什么 chosen 胜出”。
3. reward model 只负责排序，不负责最终放行。
4. reference model 冻结，policy 单独更新。
5. held-out preference 和人工抽检都已经准备好。

如果这五条里有一条说不清，先回到数据和评测，不要先改 trainer。
```

## box soft
### 一个最短的判断顺序
先问样本能不能比，再问 reward model 学没学稳，最后问 policy 会不会把 reward 当漏洞来钻。这个顺序很土，但很管用，因为它把最容易出错的层放在最前面。

比如客服退款场景里，chosen 给出“先确认订单是否在退货期，再告诉用户提交工单”的步骤，rejected 只回一句“请耐心等待”。如果 rubric 只写 `helpful`，标注者很容易把礼貌话术也算成好答案。你在练习里可以把这条样本改写成四个可判定维度，再看它还剩不剩可训练的边界。

这也是下一页会继续用到的起点。DPO / ORPO 会把偏好直接写进目标式；PPO / GRPO 会把 reward 放进更长的训练环。你只要先把“样本—分数—更新—检查”这条线记住，后面的差别就不会散掉。
