# 先抓主线 {#overview}
<!-- tag: overview -->

这一页把偏好样本压成可直接优化的目标式。DPO 把 chosen / rejected 的差异和 reference 的差异放到同一个 margin 里；ORPO 则把偏好压力并进 SFT，不再显式跑一份 reference model。

可以先把它看成三步：先看一对回答里谁更好，再看 policy 是否比 reference 更会区分这对样本，最后看这个差异会不会被长度、格式或安全模板这种代理信号带偏。

# 关键术语 {#terms}
<!-- tag: symbols -->
<!-- layout: grid2 -->

## box soft
### policy gap
`log πθ(chosen | x) - log πθ(rejected | x)`。读的是 policy 自己更偏向哪一边，不是绝对好坏。

## box info
### reference gap
`log πref(chosen | x) - log πref(rejected | x)`。读的是锚点原本怎么偏，负责保住比较尺度。

## box warn
### margin
`policy gap - reference gap`。它告诉你 policy 比 reference 多会区分多少；不是“回答是否绝对正确”。

## box
### β / λ
这类系数管的是偏好压力的强弱。太弱，模型学不动；太强，模型容易绕到代理信号上。

**先别追的变体**这页先不展开其他直接偏好优化变体。你只要先把 DPO 的锚点、ORPO 的单阶段偏好压力和 margin 的读法分开，后面的算法名就不会混成一团。

# DPO 的读法 {#dpo}
<!-- tag: reference anchor -->

DPO 的核心把偏好关系写成一个可直接优化的 margin。它并不需要显式训练 reward model，但它仍然需要 reference model 作为锚点。没有这个锚点，policy 只是在和自己比。

$$m=\Big[\log \pi_{\theta}(y^+|x)-\log \pi_{\theta}(y^-|x)\Big]-\Big[\log \pi_{\text{ref}}(y^+|x)-\log \pi_{\text{ref}}(y^-|x)\Big]$$
 $$L_{\text{DPO}}=-\log \sigma(\beta m)$$
 **读法：**前一层是 policy 自己对 chosen 和 rejected 的偏向，后一层是 reference 原本的偏向。把两者相减，才知道 policy 相对锚点多会区分这对样本。

DPO 参考实现骨架
复制

```python
def dpo_loss(policy_pos, policy_neg, ref_pos, ref_neg, beta):
 policy_gap = policy_pos - policy_neg
 ref_gap = ref_pos - ref_neg
 margin = policy_gap - ref_gap
 return -torch.log(torch.sigmoid(beta * margin))

# 这里只展示结构：先算 policy gap，再减 reference gap，
# 最后把 margin 送进 sigmoid。
```

步骤
示例数值
你该读到什么
错了会怎样


policy gap
`-6.2 - (-7.0) = 0.8`
policy 已经更偏向 chosen。
如果这一步算反，后面的 margin 全错。


reference gap
`-6.5 - (-6.8) = 0.3`
reference 也有一点偏向 chosen，但没有 policy 强。
如果 reference 也跟着更新，这一步就不再是锚点。


margin
`0.8 - 0.3 = 0.5`
policy 比 reference 多会区分 0.5。
只看 chosen 比 rejected 高，不能替代 margin。


β · margin
`0.1 × 0.5 = 0.05`
这条样本的更新压力很轻。
β 太小，模型可能学得太慢；太大，容易推过头。


σ(βm)
`≈ 0.512`
目标仍然在正确方向上，但还没拉开很多。
如果你把这个数直接当质量分，会误读训练状态。

一个实用判断是：如果 chosen 的绝对概率升了，但 reference 原本就更偏 chosen，margin 可能并没有多大变化。DPO 看的正是“policy 相对 reference 有没有更会区分”。

# ORPO 的读法 {#orpo}
<!-- tag: SFT + preference -->

ORPO 的位置更像“把两件事绑在一起做”：一边继续做 SFT，一边给 chosen / rejected 加偏好压力。它的好处是少了一份显式 reference model 的前向成本，适合想把 instruction tuning 和 preference alignment 放到同一阶段的人。

但它把约束换了位置。少了显式 reference，不等于少了风险。λ 取值、数据配比、chosen / rejected 的可比性，都会直接影响它是不是会把模型推去学代理信号。

方法
是否有显式 reference model
偏好压力放在哪里
适合什么场景
主要风险


DPO
有
policy 相对 reference 的 margin
想保住锚点，同时绕开 reward model + PPO 的复杂环路。
reference 选择、β、数据噪声和过训练都会影响稳定性。


ORPO
没有显式 reference model
SFT 项 + odds-ratio 偏好项
想把 instruction tuning 和 preference alignment 放在单阶段里。
λ、数据比例和偏好对质量更敏感，容易把代理信号放大。


SFT
通常没有
只学 chosen / answer
先把格式、指令遵循和领域表达打底。
只会模仿，不知道为什么另一个回答更差。

ORPO 的结构示意
复制

```python
sft_loss = cross_entropy(chosen_tokens)
pref_score = logp(chosen_tokens) - logp(rejected_tokens)
pref_penalty = preference_penalty(pref_score)
loss = sft_loss + lambda_orpo * pref_penalty

# 这里先看结构，不把实现细节写死：
# 1) 继续保留 SFT 的正向学习
# 2) 再把 chosen / rejected 的相对压力加进去
```

**一句话区分**DPO 是“带锚点地读 margin”，ORPO 是“在 SFT 里加偏好压力”。两者都在做 preference optimization，只是刻度和成本不同。

# 怎么选 {#decision}
<!-- tag: practical choice -->

实际做实验时，最常见的问题顺序选错。你如果还没有稳定的 SFT 基线，先不要急着上 DPO 或 ORPO。你如果已经有偏好对，但预算紧，想少掉一份显式 reference forward，可以先看 ORPO。你如果更在意锚点、想把“相对 reference 的偏移”看得更稳，DPO 通常更合适。

场景
优先选谁
为什么
第一条检查


只有示范数据，想先把格式和指令遵循打底
SFT
先把基础能力稳定住，再谈偏好压力。
看 loss mask、样本格式和是否已经跑通小 batch。


已经有偏好对，预算还算紧，但想把偏好项并进训练
ORPO
单阶段更直接，少一个显式 reference forward。
看 λ、chosen / rejected 是否可比、SFT 和偏好项会不会打架。


你最在意比较尺度，想保留锚点
DPO
reference model 保持刻度，margin 更好解释。
看 reference 是否冻结、β 是否合适、margin 是否真的在拉开。


出现长度上升、模板化、评测下降
先停
这通常proxy 失真。
抽查高分样本，确认是不是被长度或格式代理骗了。

**实验顺序**先把 SFT 起点和偏好数据看稳，再选 DPO 或 ORPO。不要为了省一步就把锚点、偏好对比和训练监控一起省掉。

# 同一条样本，三种设置 {#lab}
<!-- tag: parameter lab -->

下面这组设置用同一条偏好样本来说明一个更具体的问题：不是所有“margin 变大”都值得高兴，也不是所有“省掉 reference”都能直接上。你要看的是，训练设置改了以后，损失曲线、样本间隔和输出样式分别会往哪边走。

设置
你在改什么
短期现象
常见副作用
该怎么判断


DPO + 低 β
把偏好压力收得很轻。
loss 下降慢，margin 只会一点点涨。
模型看起来稳，但其实学得太慢。
先看验证 margin 是否也在涨；如果连验证都不动，说明更新太软。


DPO + 高 β
把偏好压力放得很重。
margin 拉得快，训练曲线看着漂亮。
模型容易把长度、格式或安全模板当成捷径。
抽查最高分样本，看看有没有统一口吻、过长回答或避重就轻。


ORPO + 高 λ
把偏好项压进 SFT，而且权重更重。
chosen 的拟合更强，短期会看到更明显的对齐感。
SFT 和偏好项打架，模型可能学得更像模板。
同时看 instruction following、长度分布和 held-out preference，别只看训练 loss。

## box soft
### 一个最小判断
如果你把一条样本从 DPO 低 β 换成 DPO 高 β，看到的是 margin 拉得更快、回答更长、语气更一致，那不一定是“模型更会对齐了”，也可能只是它更会讨好偏好代理。这个时候该查的高分样本长什么样。

## box warn
### 别误读的地方
ORPO 省掉显式 reference，不等于省掉比较。它只是把比较写进了另一项 loss。若 chosen / rejected 本身不稳，或者 λ 太大，模型会把“更像 chosen”误学成“更像模板”。

训练时该盯的最小日志
复制

```text
step 142 | policy_gap=0.8 | ref_gap=0.3 | margin=0.5 | beta=0.1
step 143 | policy_gap=0.9 | ref_gap=0.3 | margin=0.6 | beta=0.1
step 144 | policy_gap=1.1 | ref_gap=0.3 | margin=0.8 | beta=0.1

读法：
1. margin 在涨，不代表真实质量一定在涨。
2. 如果参考样本的长度和模板感也一起涨，先查代理信号。
3. 如果 reference 没冻结，先修锚点，再谈调参。
```

这就是这页最想留下的一条手感：DPO 和 ORPO 都不是单看一个标量就能收工的方法。你需要同时看样本长什么样、锚点有没有动、训练曲线是慢慢爬还是突然拉开，以及这些变化是不是只是把回答推向了更像“高分模板”的方向。读懂这层，下一页进入 PPO / GRPO 时，才知道该把采样、reward、KL 和 policy update 放在什么顺序里看。

# 常见误区 {#mistakes}
<!-- tag: check before training -->

误区
真正含义
先查什么


DPO 只是更简单的 PPO
DPO 不走在线 rollout，那条训练环和 PPO / GRPO 不是一回事。
先看目标式，再看是否需要采样、reward 和 KL。


reference model 可有可无
在 DPO 里 reference model 是比较刻度，不是装饰。
先看是否冻结，再看是否和 policy 从同一 checkpoint 出发。


ORPO 去掉 reference 就更自由
ORPO 把约束换成单阶段偏好压力。
先看 λ，再看 chosen / rejected 是否真可比。


margin 高了就代表任务做好了
margin 只是代理量，真实质量还要看 held-out 样本和人看结果。
先看最高分样本，再看独立评测。

**别把问题混在一起**如果你看到指标变好但样子变啰嗦，先别急着换 trainer。很多时候只是模型学会了更会讨好分数，而不是更会做任务。

# 边界与失真 {#boundary}
<!-- tag: when to stop -->

偏好优化最容易出问题的地方，proxy 信号继续变好，但真实质量开始变差。这个页面只负责把这件事看见，不负责替你把下一轮训练写完。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

信号
先查什么
常见误读


train margin 上升，val margin 不动
held-out preference set、样本覆盖、是否过拟合训练偏好。
把训练分数上升误读成能力上升。


回答越来越长
长度分布、最高分样本、rubric 是否把“详细”当成“更好”。
把长当成会做题。


reference 和 policy 一起漂
冻结设置、加载路径、训练脚本里谁在更新。
把锚点漂移误看成算法不稳定。


reward / margin 继续涨，但人看更差
抽查高分样本，确认有没有格式模板、拒答模板或长度代理。
把 proxy 目标当成真实目标。

## box
### 停止规则
不要把“preference loss 继续下降”当成继续训练的充分理由。只要验证偏好、独立评测、长度分布、KL / 漂移和人工抽检里有一项开始松动，就该先停下来查原因，而不是先加 epoch。

如果这一层看不稳，下一页的 PPO 和 GRPO 只会把同样的问题放大。
