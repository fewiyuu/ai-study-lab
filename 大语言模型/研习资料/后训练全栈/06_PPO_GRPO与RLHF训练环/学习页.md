# 学习目标 {#goals}
<!-- tag: Foundation for RL -->

看懂训练对象
分清 policy model、reference model、reward model 或 reward function 的职责。直觉上，policy 是正在学习的学生，reference 是不希望偏离太远的旧版本，reward 是评卷标准。


看懂优化信号
知道为什么奖励不能直接当成唯一目标。RLHF 的核心“在不毁掉原有能力的前提下，让更符合偏好的生成更常出现”。


看懂 PPO/GRPO 差别
PPO 通常要估计 advantage 并依赖 value 估计稳定更新；GRPO 倾向用同一 prompt 下多条回答的组内相对分数构造 advantage，减少或移除单独 value model 的需要。

# 训练环总览 {#map}
<!-- tag: Rollout → Reward → KL → Update -->

Prompt Batch
→
Rollout
→
Reward
→
KL Penalty
→
Policy Update

先给模型一批 prompt，让当前 policy 采样生成回答，这叫 rollout。然后把回答送进 reward model、规则奖励或 judge，得到每条回答的分数。接着计算 policy 与 reference 在这些回答上的差异，形成 KL 惩罚。最后用 PPO 或 GRPO 把“净收益更高”的 token 轨迹推高。

**01 Prompt**输入问题、任务或环境状态。prompt 分布决定模型会在哪些场景里学习。
**02 Rollout**当前 policy 生成一个或多个 response。采样温度、长度上限、stop 条件都会影响训练数据。
**03 Reward**给完整回答或中间步骤打分。奖励可以来自 reward model、可验证规则、工具环境或模型评审。
**04 KL**惩罚 policy 相对 reference 的过度偏移。它稳定器。
**05 Update**根据 advantage 更新 policy。更新幅度太小学不到，太大会崩。

# 术语与误区 {#terms}
<!-- tag: Key Terms -->

| 术语 | 直觉解释 | 精确定义 | 常见误解 |
| --- | --- | --- | --- |
| Policy | 正在被训练的生成模型。 | 给定上下文后，对下一个 token 或完整 response 轨迹给出概率分布的模型。 | 把 policy 当成固定模型。RL 阶段中，policy 每轮都会变，所以 rollout 数据也会变。 |
| Rollout | 让当前模型真的生成一批回答。 | 从当前 policy 按采样策略得到 response，并记录 token、logprob、长度、奖励等训练所需信息。 | 以为 rollout 只是普通推理。训练用 rollout 必须保存概率、mask 和奖励对齐信息。 |
| Reward | 这条回答“值得鼓励”的程度。 | 由 reward model、规则、环境反馈或 judge 对 response 给出的标量或分步信号。 | 以为 reward 越高模型越好。奖励一旦有漏洞，模型会优先学会利用漏洞。 |
| KL Penalty | 别为了高分跑得离旧模型太远。 | 衡量 policy 分布与 reference 分布差异的惩罚项，常按 token logprob 差近似加入目标。 | 把 KL 当成“安全评测”。KL 只限制分布漂移，不保证事实、安全或无偏。 |
| Advantage | 这次回答比预期好多少。 | 用于 policy gradient 的相对收益信号，常见形式是 reward 减 baseline 或经过标准化的相对分数。 | 把 reward 和 advantage 混用。reward 是原始分数，advantage 是“该不该增加这条轨迹概率”的更新信号。 |
| Reference Model | 训练前的锚点。 | 通常是 SFT 后冻结的模型，用来计算 KL 或 logprob ratio，防止 policy 过度漂移。 | 以为 reference 会跟着训练更新。多数 RLHF 设定里 reference 是冻结的。 |

易混 1：Reward vs Advantagereward 是评分，advantage 是更新方向。两个回答都可能得高分，但如果其中一个只是“符合预期”，它的 advantage 不一定大。
易混 2：KL vs ClipKL 约束 policy 与 reference 的距离；PPO 的 clip 约束新旧 policy ratio 的更新幅度。一个管“别偏离旧模型太远”，一个管“这一轮别改太猛”。
易混 3：DPO vs RLHFDPO 用静态偏好对直接优化；RLHF/PPO/GRPO 通常在线生成新样本再学习。在线性带来更强探索，也带来更高不稳定性。

# Rollout：训练数据来自当前模型 {#rollout}
<!-- tag: Online Samples -->
<!-- layout: grid2 -->

RLHF 与 SFT 的第一个关键区别是数据来源。SFT 读取人工或合成数据里的标准答案；RLHF 则让当前 policy 面对 prompt 自己生成回答。因为 policy 每次更新后都会变化，下一批 rollout 的分布也会变化，这叫在线训练。
rollout 要记录的不只是 response 文本，还包括每个 token 在当前 policy 下的 log probability、attention mask、是否越界、生成长度、stop 原因，以及稍后要对齐的奖励。没有这些记录，后面的 KL 和 policy update 就无法可靠计算。

 response y ~ πθ(· | x)
 x 是 prompt，y 是采样出的回答，πθ 是当前 policy。采样不是装饰，它决定模型会从哪些自己制造的行为中学习。

## box info
### 你现在应该能做什么
看到一份 RLHF 训练日志时，先问：prompt 从哪里来？每个 prompt 采样几条回答？采样温度多大？是否保存了 logprob？回答被截断了吗？这些问题比先问“用了哪个算法”更基础。

| Rollout 设置 | 太保守时 | 太激进时 | 检查方式 |
| --- | --- | --- | --- |
| temperature / top-p | 样本太像，reward 差异小，学不到新行为。 | 低质量或格式错误样本增多，奖励模型压力变大。 | 抽样看回答多样性、错误率和长度分布。 |
| 每个 prompt 的 response 数 | 难以比较同题下多个候选。 | 算力成本高，可能被少数 prompt 主导。 | 看每个 prompt 的奖励方差和组内排序。 |
| 最大长度 | 长推理被截断，奖励不可信。 | 模型学会啰嗦，KL 和训练成本上升。 | 看截断率、平均长度、单位 token 收益。 |

# Reward：把偏好变成可优化信号 {#reward}
<!-- tag: Score Is Not Truth -->
<!-- layout: grid2 -->

reward 是 RLHF 的方向盘。它可以来自人类偏好训练出的 reward model，也可以来自可验证规则，比如数学题是否答对、代码测试是否通过、工具调用是否完成目标。推理模型训练中，规则奖励和可验证奖励尤其重要，因为“答案是否正确”比“看起来像好答案”更可靠。
但是 reward 只是代理目标。代理目标的意思是：它不是最终真正想要的全部品质，只是一个可以计算、可以训练的近似信号。模型会优化你给它的信号，而不是你心里想象的完整目标。

 Rtotal(x, y) = Rtask(x, y) - β · KL(πθ(·|x) || πref(·|x))
 任务奖励负责“做得好”，KL 惩罚负责“别偏太远”。β 越大，训练越保守。

## box
### 奖励黑客现象
reward hacking 指模型找到奖励函数漏洞，而不是学到真实能力。例如奖励长度时，模型可能变得冗长；奖励格式时，模型可能堆格式；只用 judge 打分时，模型可能学会迎合 judge 的表面偏好。

Reward Model从 chosen/rejected 偏好数据训练，适合复杂主观偏好，但会继承标注噪声。
Rule Reward由规则直接计算，适合数学、代码、格式、工具结果等可验证任务。
Environment Reward来自环境状态变化，常见于工具使用、网页操作、游戏或代理任务。
Judge Reward由另一个模型评审，覆盖面广，但要防止偏见、串谋和评分漂移。

# KL：防止模型为了奖励跑偏 {#kl}
<!-- tag: Stability Dial -->

KL divergence 衡量两个概率分布的差异。在 RLHF 里，我们常关心：当前 policy 对生成 token 的概率，和 reference model 对同样 token 的概率差多少。若 policy 为了拿高 reward 把某些奇怪 token 概率抬得很高，KL 会变大，净奖励会被扣掉。

KL token 近似项：log πθ(yt|x,y<t) - log πref(yt|x,y<t)
 实现中常对生成 token 求和或求均值，再乘上 KL 系数。不同库会有具体近似与归一化差异，应以当前文档为准。

KL 系数不是越大越好。太小，模型可能奖励冲高但语言质量下降；太大，模型几乎不敢改变，reward 上不去。实际训练会同时看 reward、KL、response length、entropy、clip fraction、胜率和人工样本。

# Policy Update：把好轨迹概率推高 {#update}
<!-- tag: Gradient Step -->
<!-- layout: grid2 -->

policy update 的目标是改变模型参数，让带来正 advantage 的生成更可能出现，让负 advantage 的生成更不可能出现。对语言模型来说，这个更新落在生成 token 的 log probability 上：某条 response 的 advantage 为正，就提升这串 token 在相同上下文下的概率；advantage 为负，就压低。
PPO 的名字里有 proximal，意思是“更新要靠近原策略”。它会用新旧 policy 概率比值控制单次更新幅度，避免模型因为一批 rollout 的奖励信号就大幅跳动。常见做法还会估计 value，用 GAE 或类似方法得到 advantage，再用 clipped objective 更新 policy。

 PPO clipped objective：min(rtAt, clip(rt, 1-ε, 1+ε)At)
 rt 是新旧 policy 对同一 token 的概率比。clip 让“这一轮学太猛”的收益不再继续变大。

## box
### 实现时要盯住什么
- **ratio：**新 policy 相对 rollout 时 old policy 的变化幅度。
- **clip fraction：**有多少 token 或样本触发了裁剪。
- **entropy：**分布是否过早变窄，导致探索下降。
- **value loss：**PPO 中 value 估计是否拖累训练。
- **KL：**policy 是否整体偏离 reference。

训练环伪代码
conceptual loop

```text
for step in training_steps:
 prompts = sample_prompt_batch(dataset)

 # 1. rollout: 用当前 policy 生成回答，并保存 logprob
 responses, old_logprobs = policy.generate_with_logprobs(prompts)

 # 2. reward: 对回答打分
 task_reward = reward_fn(prompts, responses)

 # 3. KL: 与冻结 reference 比较，得到漂移惩罚
 ref_logprobs = reference.logprobs(prompts, responses)
 kl_penalty = beta * (old_logprobs - ref_logprobs)
 rewards = task_reward - kl_penalty

 # 4. advantage: PPO 用 value/baseline；GRPO 可用组内标准化
 advantages = estimate_advantage(rewards, prompts, responses)

 # 5. policy update: 提升正 advantage 轨迹，压低负 advantage 轨迹
 loss = policy_gradient_loss(policy, prompts, responses, old_logprobs, advantages)
 loss.backward()
 optimizer.step()
```

# PPO vs GRPO：两种构造更新信号的路线 {#compare}
<!-- tag: Algorithm Contrast -->

PPO 的基本形态
PPO 常用于传统 RLHF 配方。它通常需要 policy、reference、reward model，以及用于估计预期收益的 value model 或 value head。它的优势是方法成熟、稳定性工具多；代价是训练组件多、显存和调参复杂度高。

适合：主观偏好、对话质量、已有 reward model 的场景。
关注：value loss、ratio clipping、KL、reward hacking。
风险：组件越多，故障来源越多。


GRPO 的基本形态
GRPO 常见于推理强化学习语境。它对同一个 prompt 采样一组回答，用组内 reward 的均值和标准差做归一化，得到每条回答的相对 advantage。这样可以减少对单独 value model 的依赖，尤其适合有可验证奖励的任务。

适合：数学、代码、格式可验证、同题多样采样的场景。
关注：每组样本数、组内 reward 方差、长度偏置、KL。
风险：组内分数都差不多时，相对优势信号会变弱。

| 维度 | PPO | GRPO | 判断建议 |
| --- | --- | --- | --- |
| 优势估计 | 常用 value/baseline 估计 advantage。 | 同一 prompt 的多条回答做组内相对比较。 | 若 value 估计难、但可为同题生成多答案，GRPO 更自然。 |
| 奖励来源 | 常搭配 reward model，也可用规则奖励。 | 常搭配可验证奖励或规则奖励。 | 奖励越可验证，越适合让模型在线探索。 |
| 训练成本 | 组件多，value 相关开销和调参更多。 | 少一个 value 组件，但需要同题多样本 rollout。 | 不要只看算法名，算 prompt 数、response 数和总 token。 |
| 稳定器 | clip、KL、value loss、entropy 等。 | 组内标准化、KL、ratio 裁剪或类似约束。 | 稳定器是训练成功的主角之一，不是附属参数。 |

$$
GRPO 组内优势直觉：Ai = (ri - mean(rgroup)) / std(rgroup)
 同一道题生成多条回答，奖励高于组内平均的回答被鼓励，低于平均的回答被压低。真实实现还会处理标准差为零、长度归一化、KL 和裁剪细节。
$$

# 案例与工作示例 {#examples}
<!-- tag: Worked Cases -->
<!-- layout: grid2 -->

## box
### 案例 1：数学题的 GRPO 训练
同一个数学 prompt 采样 8 条回答。规则奖励只看最终答案是否正确，同时额外检查是否按要求输出。若 2 条答对、6 条答错，答对样本在组内有正 advantage，答错样本为负 advantage。policy 更新后，模型更倾向生成答对轨迹中的 token 模式。
关键点：如果所有 8 条都答错，组内 reward 方差可能很小，更新信号弱。此时需要更好的 prompt 分布、采样策略、课程难度或奖励 shaping，而不是盲目加大学习率。

## box
### 案例 2：客服对话的 PPO 训练
reward model 根据“是否有帮助、是否礼貌、是否遵守政策”给分。PPO 用 value head 估计预期收益，再根据 advantage 更新 policy。KL 约束防止模型为了迎合 reward model 变得模板化、冗长或过度拒答。
关键点：主观 reward model 的漏洞很难完全消除，所以必须把训练指标、抽样人工检查和回归评测放在一起看。

GRPO 组内优势计算
minimal example

```text
prompt = "求 17 * 23，并给出简短推理"
rewards = [1, 0, 0, 1, 0, 0, 0, 0] # 1 表示最终答案正确，0 表示错误

mean_r = sum(rewards) / len(rewards) # 0.25
std_r = standard_deviation(rewards) # 约 0.43
advantages = [(r - mean_r) / (std_r + 1e-6) for r in rewards]

# 正确样本 advantage 约为 +1.73，错误样本约为 -0.58
# 更新方向：提高正确回答轨迹概率，降低错误回答轨迹概率，同时受 KL 约束。
```

## box warn
### 常见错误：只看 reward 曲线
如果 reward 上升但 KL、长度、重复率、人工样本质量同时变坏，训练可能是在奖励函数上过拟合。一个成熟的 RLHF 实验记录至少要同时看 reward、KL、生成长度、通过率、拒答率、样本质检和离线评测。
**修法：**为每次训练保留固定 prompt 集做回归检查，并定期比较 policy、reference 和上一个 checkpoint 的输出。

# 五个高频误区 {#mistakes}
<!-- tag: Failure Modes -->

误区 1：RLHF 一定比 DPO 高级不一定。若没有可靠奖励、在线采样成本太高，DPO 类方法可能更稳、更划算。
误区 2：KL 越低越好KL 太低可能说明模型没学动。正确问题是：在能力和安全不退化的前提下，KL 是否支撑 reward 改善。
误区 3：规则奖励天然安全规则奖励只覆盖规则写到的部分。模型可能满足格式和答案，却学会不良中间过程或投机策略。
误区 4：多采样一定提升 GRPO多采样提升比较机会，也增加成本。若 prompt 太简单或太难，组内差异不足，优势信号仍然弱。
误区 5：训练稳定只靠学习率学习率重要，但 KL 系数、clip、batch 组成、奖励尺度、长度归一化和采样策略同样会决定稳定性。
误区 6：reward model 是客观裁判reward model 学的是偏好数据里的模式。标注偏差、数据覆盖不足和 judge 习惯都会进入训练目标。

# 机制演示：reward / KL / 更新信号 {#mechanism}
<!-- tag: trace -->

**净奖励：**Rnet = Rraw - β × KL(policy, reference)
原始奖励描述“这条回答好不好”，KL 约束描述“偏离参考模型有多远”，beta 控制收紧还是放松。

reward / KL 计算

```python
raw_reward = 5
kl = 0.4
beta = 2.0
net_reward = raw_reward - beta * kl
```

<!-- interactive: ['terminal-lab'] → keep as HTML -->

净奖励不是“分数减分数”这么简单。训练时要同时看原始奖励、KL 约束和 beta 的权重方向。

| 变量 | 作用 | 看什么 |
| --- | --- | --- |
| raw reward | 原始评分 | 样本本身是否更好 |
| KL | 偏离惩罚 | policy 有没有跑偏 |
| beta | 约束强度 | 更新是收紧还是放松 |

只看 reward 会忽略漂移；只看 KL 会忽略任务收益。后训练要同时把两件事读出来。

先会读 net reward，再去看 rollout、advantage 和 policy update，最后再判断有没有能力折损。

### 1 选择

**Q:** Q1. 下面哪个顺序最接近 RLHF 的基本训练环？

- [ ] . reward → rollout → prompt → policy update → KL
- [ ] . prompt → rollout → reward/KL → advantage → policy update

Answer: . prompt → rollout → reward/KL → advantage → policy update
Explain: 训练环先从 prompt 出发，让 policy rollout，随后得到 reward 和 KL 约束，再算 advantage 并更新 policy。
