后训练全栈 / Evaluation Gate

评测、判别器与回归检查

后训练不是训练完再看一眼分数。它需要一套能长期运行的评测系统：用确定性指标抓硬错误，用 judge 处理开放答案，用回归测试保护已有能力，用人工抽检校准自动结论。

本页完成后

  • 能区分 benchmark、自动评测、judge、判别器、回归测试各自解决的问题。
  • 能写出一个可复用的 judge 规格：任务、rubric、输出格式、校准样本和失败检查。
  • 能为一次 SFT/DPO/RL 实验设计发布前能力折损检查。

学习目标

60-90 分钟

把评测从“分数”改成“合同”

评测不是排行榜数字，而是发布前必须满足的行为约束：哪些能力必须涨，哪些能力不能跌，哪些安全边界不能破。

理解 judge 的可靠性边界

judge 可以是 LLM、分类器、reward model 或 verifier。它能降低人工成本，但需要 rubric、校准集、一致性检查和人工抽检。

建立回归测试思维

每次后训练都要和基线模型比较，特别关注能力折损：格式变好了，数学是否跌了？拒答更安全了，正常求助是否被误拒？

概念骨架

从样本到发布门禁
任务与风险 → 评测集 → 打分器 / Judge → 聚合与阈值 → 回归门禁

直觉上，评测系统像发布前的仪表盘和刹车。仪表盘告诉你不同能力的变化，刹车在关键指标跌穿时阻止上线。术语上，它是一组固定输入、固定采样规则、固定评分函数和固定发布阈值。

对每个能力 m，记录差值：Δm = score(model_new, eval_m) - score(model_base, eval_m)。
发布门禁不是只看平均分，而是检查：关键能力 Δm ≥ 最小允许变化，风险能力 violation_rate ≤ 上限。

你现在应该能做什么

看到一个后训练实验时，先问：它声称提升了什么？可能牺牲什么？用什么固定样本和评分器证明？和哪个基线比？

术语与误区

容易混淆的词
术语 直觉解释 精确定义 常见误解
Benchmark 一套公开或内部题库。 固定任务集合、样本、评分规则和报告口径，用于比较模型能力。 把 benchmark 分数当成产品质量。公开榜单常覆盖不了你的真实用户分布。
自动评测 让机器批量跑题、打分、汇总。 从输入集、模型输出、评分器、统计聚合到报告生成的可复现流程。 以为自动评测不需要人工。评分器本身也需要校准和抽检。
Judge 替你判断“这个回答好不好”的裁判。 基于 rubric 对输出进行 pointwise、pairwise 或 ranking 评价的模型或规则系统。 把 judge 当真理。它会受位置、长度、措辞、模型同源性和提示泄漏影响。
判别器 回答“属于哪类 / 是否合格”的检查器。 把输入和输出映射到类别、风险标签、合格/不合格或连续分数的模型。 把所有 judge 都叫 reward model。判别器可以只做安全分类、格式检查或事实核验。
能力折损 某项训练让模型在别的地方变差。 新模型相对基线在特定能力、任务、人群或交互约束上的显著下降。 只看目标任务上涨。SFT、DPO、RL 都可能引入遗忘、过拒、风格固化或推理下降。
回归测试 每次改模型都跑的老题。 用于检测已知能力、边界行为和历史 bug 是否被新训练破坏的固定测试套件。 只在代码里做回归。模型也需要回归，因为权重更新会改变行为分布。

核心讲解

评测是发布合同

1. 后训练评测的真正问题

后训练会把模型从“会续写”推向“会遵循指令、会拒绝、会按偏好回答”。问题是：模型能力不是单轴变化。一次 SFT 可能提高格式遵循，却让模型更啰嗦；一次 DPO 可能提高偏好胜率，却让模型过度迎合；一次 RL 可能提高推理任务分数，却引入奖励投机。

所以评测不能只问“新模型是否更好”。它要拆成三个问题：目标能力是否提升，关键旧能力是否保持，风险行为是否仍在边界内。

一次后训练发布至少要回答：
Gain：目标指标是否上升？
Regression：非目标指标是否下跌？
Risk：安全、合规、隐私、拒答和工具调用边界是否稳定？

判断句

如果一个实验报告只展示目标 benchmark 的涨幅，不展示基线对比、非目标能力和失败样本，它还不能支撑发布决策。

2. 评测集先按“能力面”组织

能力面是你想保护或提升的一类行为，例如指令遵循、数学推理、代码生成、多轮记忆、中文写作、拒答边界、事实一致性、工具调用、结构化输出。一个评测集不要只按题源命名，更要说明它覆盖哪个能力面。

每个能力面至少需要四类样本：正常正例、边界样本、历史失败样本、对抗样本。正常正例看日常质量，边界样本看规则是否稳定，历史失败样本看修复是否回退，对抗样本看模型是否会被诱导偏离。

评测系统的四层结构

输入层

任务、样本、采样参数、上下文模板。没有固定输入，就没有可比较结果。

评分层

精确匹配、规则、测试执行、分类器、LLM judge、人工标注。评分器要和任务类型匹配。

决策层

聚合、置信、阈值、失败样本查看和发布门禁。决策层负责说“能不能上线”。

你现在应该能做什么：为任何一个能力面列出“输入样本、评分方式、通过阈值、失败后动作”。

Judge 与判别器

不是把问题丢给另一个模型

Judge 适合什么

开放式回答、写作质量、多轮任务完成度、偏好胜负、帮助性、解释清晰度、格式是否满足复杂约束。这些任务很难用单个字符串匹配。

Judge 不适合什么

有明确答案的数学计算、代码是否能运行、JSON 是否可解析、链接是否存在、隐私字段是否泄露。这些应优先用确定性检查。

1. Pointwise、Pairwise、Ranking

方式 输入 输出 适用场景 风险
Pointwise 一个问题 + 一个回答 分数或标签 质量分、是否合格、是否违规 不同题目间分数不一定可比，rubric 写弱会漂移。
Pairwise 同一问题 + 两个回答 A 胜 / B 胜 / 平 DPO 前后的偏好胜率、模型版本比较 位置偏置明显，需要交换 A/B 顺序复判。
Ranking 同一问题 + 多个回答 排序 多候选生成、rerank、采样策略比较 长列表会降低一致性，成本也更高。

2. 一个可用 judge 规格包含什么

任务定义

明确输入是什么、要判断什么、哪些信息不能使用。例如只能根据给定证据判断事实一致性。

Rubric

把“好”拆成可观察标准：正确性、完整性、约束遵循、拒答边界、语气、引用证据。

输出格式

强制 JSON 或固定字段，包含 verdict、score、reason、failure_tags，便于后续聚合。

校准样本

准备人工已判的样本，看 judge 与人类是否一致，并定期抽检漂移。

常见 judge 偏差

  • 长度偏差：更长的回答看起来更认真，但未必更正确。
  • 位置偏差：pairwise judge 可能偏爱 A 或 B，所以要交换顺序复判。
  • 风格偏差：judge 偏好礼貌、套话或结构化，但目标任务可能更需要简洁准确。
  • 同源偏差：用同一家族模型做 judge 时，可能偏好相似表达模式。
  • 提示泄漏：待评回答如果包含“评分指令”，可能影响 judge。
  • 证据越界：事实核验 judge 如果使用自身世界知识，可能绕过给定证据。

你现在应该能做什么

设计 judge 时，不只写“请打分”。你应该能同时写出 rubric、输出字段、校准样本、偏差控制和人工抽检规则。

自动评测流水线

Eval Harness

自动评测的价值不是“无人参与”，而是“同一套流程可以重复运行”。在后训练中，每个候选 checkpoint 都应该用同样的输入、采样参数、评分器和报告口径评估，否则版本之间不可比。

1. 固定输入
保存 prompt、system message、few-shot 示例、工具说明、上下文长度、采样参数。采样温度变化会改变输出分布，不能随手改。
2. 候选回答生成
对基线模型和新模型跑同一批样本。开放生成任务最好保存原始输出，方便之后重新评分和人工复盘。
3. 多类评分
能用确定性评分就先用确定性评分；不能确定性评分时，再用 judge 或人工。评分器要记录版本。
4. 聚合报告
按能力面、难度、语言、风险类别拆分分数，展示均值之外的失败样本和长尾下降。
5. 发布门禁
把“目标涨幅”“最大可接受回退”“风险违规率”“人工抽检结论”写成规则，而不是临时拍脑袋。

评分器选择原则

任务类型 优先评分方式 为什么
JSON、SQL、工具参数 解析器 + schema + 单元测试 格式和执行结果可验证，judge 不应替代解析。
数学、代码、可执行题 答案检查、测试用例、执行沙箱、pass@k 结果能被程序验证，模型评价容易受解释文字干扰。
摘要、写作、客服回复 Rubric judge + 人工抽检 质量依赖语境和偏好，单一字符串指标太弱。
安全拒答、隐私、政策 规则过滤 + 分类器 + 人工复核 误放和误拒都重要，需要按风险等级拆分。

你现在应该能做什么

面对一个新任务，你应该先判断能否用确定性评分；如果不能，再决定是否用 judge，并写清楚人工如何校准 judge。

能力折损与回归检查

Regression Gate

1. 为什么后训练会伤害旧能力

SFT 会把模型推向训练样本的回答风格，如果样本过窄，模型会变得模板化。DPO 会放大偏好数据里的选择标准，如果偏好数据奖励“更自信、更长、更讨好”，模型就可能过度迎合。RL 会让模型追逐奖励函数，如果奖励不覆盖真实任务，模型可能学会钻评分器的空子。

能力折损不是异常，而是权重更新后的常见副作用。后训练实验越接近发布，越要把回归检查当作默认步骤。

典型折损信号

  • 正常问题被错误拒答，安全性看似提高但有用性下降。
  • 模型更会写结构化回答，但事实错误更多。
  • 中文、长上下文或小语种能力下降。
  • 代码和数学推理下降，但聊天偏好分上涨。
  • 回答变长、延迟和成本上升，产品体验变差。

2. 回归测试集怎么维护

固定核心集

每次发布必跑，覆盖产品主路径、关键能力和高风险边界。不要频繁替换，否则趋势不可比。

历史失败集

把线上事故、人工复盘和红队样本加入回归，防止修过的问题再次出现。

刷新探索集

定期加入新场景，发现固定核心集没覆盖的问题。刷新集可探索，核心集负责趋势。

例题与工作样板

Worked Examples

例 1：一个 pairwise judge 的规格

场景：你训练了一个客服助手，希望比较新旧模型在“准确、可执行、不过度承诺”上的偏好胜率。不要只写“哪个回答更好”，要把标准拆开。

judge-spec.json 结构化规格 
{
  "task": "比较两个客服回复哪个更适合发送给用户",
  "judge_type": "pairwise",
  "input_fields": ["user_question", "policy_context", "answer_a", "answer_b"],
  "rubric": [
    "答案必须解决用户的问题，不能只表达歉意",
    "答案只能使用 policy_context 中给出的政策",
    "如果信息不足，应说明需要补充什么，而不是编造",
    "优先选择准确、简洁、下一步明确的回答",
    "不要因为回答更长就判胜"
  ],
  "bias_controls": [
    "对每个样本交换 answer_a 和 answer_b 再评一次",
    "输出 tie 时不强行判胜",
    "每 100 个样本抽检 20 个与人工标签对比"
  ],
  "output_schema": {
    "winner": "A | B | tie",
    "confidence": "low | medium | high",
    "failure_tags": ["fabrication", "missing_next_step", "policy_misuse"],
    "reason": "不超过 80 字"
  }
}

这个规格能被自动化系统调用，也能被人工复盘。关键点是：judge 的输入、判断标准、偏差控制和输出结构都被固定了。

例 2：回归测试的能力矩阵

场景：DPO 后模型在偏好胜率上提高，但你担心它变得更讨好、更爱拒答。能力矩阵要把目标项和保护项分开。

能力面 样本来源 评分器 发布阈值
客服偏好胜率 近期真实问题脱敏 + 人工 pairwise 标签 pairwise judge + 抽检 相对基线胜率 ≥ 55%，人工抽检一致率 ≥ 80%
事实一致性 带证据的政策问答 证据约束 judge + 引用检查 不低于基线 1 分；严重编造率 ≤ 1%
正常求助误拒 无风险的日常任务请求 拒答分类器 + 人工复核 误拒率不能高于基线 0.5 个百分点
结构化输出 JSON / 表格 / 工具参数任务 schema 解析 + 单元测试 可解析率 ≥ 98%，关键字段缺失率 ≤ 1%

例 3：发布门禁伪代码

这段伪代码强调“保护指标不能被平均分盖住”。即使目标能力上涨，如果关键保护项下跌，也应该阻止发布。

regression-gate.py 伪代码 
def release_gate(report):
    checks = []

    checks.append(report.delta("target_preference") >= 2.0)
    checks.append(report.delta("math_reasoning") >= -1.0)
    checks.append(report.delta("normal_helpfulness") >= -0.5)
    checks.append(report.rate("unsafe_allow") <= 0.01)
    checks.append(report.rate("wrong_refusal") <= report.baseline_rate("wrong_refusal") + 0.005)
    checks.append(report.human_audit("critical_failures") == 0)

    if all(checks):
        return "pass"
    return "block_and_review_failures"

真实系统还要记录样本版本、模型版本、judge 版本、采样参数和失败样本链接。否则分数无法复现。

常见错误：让 judge 直接读“模型 A 是新模型”

如果 judge 知道哪个是新模型，它可能被实验预期影响。pairwise 评测应隐藏版本身份，并交换顺序再评一次。

修法：只给 judge 看任务输入和两个匿名回答；输出 A/B/tie；对冲突样本进入人工复核。

拿一个实际后训练实验，写出 4 行门禁表：目标能力、一个必须保护的能力、一个安全边界、一个回归样本来源。每行再补一个阈值和一个失败后的动作。写完后先问自己：如果胜率上升，但投诉也上升，哪一行会先把发布挡住？

        

        

          
把流程写成复盘口径

          
复盘时最容易发生的事，是大家都在讲不同层面的“好”。有人讲实验分数，有人讲样本感觉，有人讲用户投诉，最后谁都没法回答“到底该不该发”。把流程写成统一口径后，争论就会少很多：先固定样本，再看 judge，再看回归，再看人工抽检，最后才讨论发布。

          

            
              

                复盘口径
                看什么
                输出什么
              

            
            
              

                实验分数
                目标任务是否提升
                是否值得继续试
              

              

                回归分数
                旧能力是否保持
                哪类保护项被打破
              

              

                风险样本
                错误是否可接受
                是否必须阻止发布
              

              

                人工抽检
                自动结论有没有偏
                是否需要改 judge 或改数据
              

            
          

          
只要这四个口径都清楚，你就不会把“评分器变了”误当成“模型变好了”。这也是回归测试真正值钱的地方。

        

        

          
单样本复盘：从一条回答到一条门禁

          
再把视角缩小到一条样本。用户问“这个方案能不能直接退款”，模型回了一个看起来很合理的回答，但实际上这超出权限。这个样本一开始会在自动评分里拿到不错的分数，因为语言流畅、语气稳定；可只要把它放进回归和权限边界里看，问题就暴露了。

          

            
              

                你看到的
                背后要补的检查
                发布时会怎么处理
              

            
            
              

                回答很顺
                有没有编造承诺
                仍然要过事实与权限检查
              

              

                judge 选了胜者
                judge 有没有被提示诱导
                必要时交换顺序再评
              

              

                回归集没掉分
                这个样本是不是本该进安全集
                把它加入失败集再看
              

              

                人工觉得可接受
                是否只是“语气好”而不是“动作对”
                权限项没过就不能发布
              

            
          

          
这一层是很多人最容易漏掉的：模型回答本身看起来没问题，不代表系统层面的动作没问题。后训练评测的责任，就是把这种“看起来对”的答案拆开，让错误无法继续伪装。

        

        

          
案例串联：从分数到动作

          
你可以把一次发布看成四层：第一层是“这次实验有没有真的更好”，第二层是“有没有把旧能力打坏”，第三层是“哪类错误最危险”，第四层才是“能不能上线”。很多团队把这四层混成一张平均分表，于是看起来有结论，实际上没有动作。

          

            
              

                层级
                代表问题
                典型产物
                如果缺失会怎样
              

            
            
              

                实验层
                新模型是不是更好
                judge 胜率、自动评分
                只知道分数，不知道来源
              

              

                回归层
                旧能力有没有掉
                固定回归集、趋势图
                新模型把老问题又带回来
              

              

                风险层
                哪些错误最伤
                失败样本、危险标签
                低频高损害错误被平均分盖住
              

              

                动作层
                现在到底能不能放量
                门禁、回滚、人工抽检
                明明该停，却还在继续推
              

            
          

          
这张表的用处不是记住概念，而是让你看到证据链。你先知道测到的是什么，再知道它能不能变成发布决定。

        

        

          
可复用门禁模板

          
如果你要给一个新实验写门禁，最省事的方式不是从头发明规则，而是复用一个固定骨架：能力项、保护项、安全项、运行项。然后给每项补“样本从哪来”“评分器是什么”“失败后怎么处理”。

          

            
              

                门禁项
                样本来源
                评分器
                失败动作
              

            
            
              

                目标能力
                近期真实任务
                自动评分 + judge
                回看是否真有提升
              

              

                保护能力
                固定回归集
                单元规则 / 人工抽检
                阻止发布并回退
              

              

                安全项
                高风险样本
                拒答分类器 + 规则
                转人工或关闭入口
              

              

                运行项
                线上监控
                阈值告警
                限流、熔断、回滚
              

            
          

          
这个模板的好处是，它把“要不要发”变成“哪一项先失败”。这样写出来的门禁才会真的能执行，而不是停在口号。

        

        

          
判分工作流：从一个样本到一次决策

          
场景可以再具体一点。假设客服助手在一个样本上被 judge 判成 A 更好，但投诉里说 A 其实承诺了不存在的补偿。此时不要争论“到底谁对”，而要把样本沿着固定工作流走完：先锁定输入，再匿名化回答，再交换顺序复判，最后把结果和回归集、投诉样本一起比对。

          

            
              

                步骤
                读取什么
                写回什么
                错了会怎样
              

            
            
              

                固定样本
                原始用户输入与任务说明
                样本 hash、版本号
                题库变化会伪装成模型进步
              

              

                匿名回答
                两个候选答案
                A/B 标识隐藏
                judge 受版本预期影响
              

              

                交换顺序
                同一对回答再评一次
                位置偏差结果
                先后顺序决定胜负
              

              

                对照回归集
                固定保护样本
                回归差值
                旧能力悄悄掉了还不知道
              

              

                人工抽检
                冲突样本和高风险样本
                复核结论
                高损害错误被平均分盖过去
              

            
          

          
review-flow

          
1. 固定样本与版本
2. 记录 judge 输入
3. 交换 A/B 顺序复判
4. 对照回归集和失败集
5. 对高风险样本做人工复核
6. 再决定是否能进门禁

          
这六步的核心不是形式，而是顺序。只要某一步没有证据，你就不能把它硬拼成“可以发布”。评测系统的价值就在这里：它把模糊争论变成可以回看的证据链。

        

        

          
你现在应该能做什么

          
从自己的实验里挑一个失败样本，按这六步写出判断链。写的时候别只写结论，要把“样本有没有变、judge 有没有偏、回归集有没有固定、人工复核有没有介入”都写出来。写完后你就能更快发现，是模型真的变了，还是评测流程自己变了。

        

      


      

        

          
练习

          20 题混合练习
        

        

          

            作答目标
            
练习不是背术语，而是能为一次后训练实验写出评测计划：该测什么、怎么测、怎么防 judge 偏差、怎么判断能否发布。

          

          

            检查重点
            
每个开放题尽量按“对象 → 风险 → 检查方法 → 失败后动作”回答。

          

        


        

          

            

              
一、概念辨析

              
确认你能分清评测系统里的对象。

            

            

              答题提醒
              先判断任务是否有确定答案，再决定是否需要 judge。
            

          

          

            

              
选择评分器选择

              
哪个任务最适合优先使用确定性评分？

              

                
                
                
              

              
有明确机器可验规则时，不要先请 judge。

              
              

            


            

              
判断边界

              
Benchmark 分数可以直接代表产品质量。

              
              
              
想想真实用户分布、产品边界和风险样本。

              
              

            


            

              
填空回归

              
填空：回归测试主要保护的是模型的什么？

              
              
不是只证明新能力上涨。

              
              

            


            

              
选择判别器

              
下列哪个更像判别器而不是生成模型？

              

                
                
                
              

              
判别器输出类别、标签或分数。

              
              

            


            

              
判断能力折损

              
能力折损只会发生在强化学习阶段。

              
              
              
SFT 和 DPO 也会改变输出分布。

              
              

            


            

              
解释术语

              
用 3 句话区分自动评测、judge 和人工抽检。

              
              
脚手架：自动评测是流程；judge 是评分器之一；人工抽检用于校准和复核。

对照本节内容检查答案。

              

            

          

        


        

          

            

              
二、Judge 设计

              
把“让模型打分”改写成可复用的 judge 规格。

            

            

              答题提醒
              好的 judge 设计要包含 rubric、输出格式、偏差控制和校准样本。
            

          

          

            

              
选择偏差控制

              
Pairwise judge 最需要控制哪类偏差？

              

                
                
                
              

              
如果 A/B 交换后结论变了，就说明有问题。

              
              

            


            

              
判断输出格式

              
Judge 的 reason 字段越长越好。

              
              
              
评测输出要服务聚合和复盘，不能让解释本身变成噪声。

              
              

            


            

              
填空偏差控制

              
填空：为避免 pairwise 位置偏差，应对 A/B 做什么？

              
              
不是只随机一次。

              
              

            


            

              
判断证据约束

              
事实核验 judge 应尽量只使用给定证据。

              
              
              
否则 judge 可能用自身世界知识替模型补洞。

              
              

            


            

              
设计Rubric

              
为摘要任务写一个三条 rubric。

              
              
脚手架：每条 rubric 都应该能在输出里被观察，而不是抽象地说“质量高”。

              
              

                参考
                
对照本节内容检查答案。

              

            


            

              
错误诊断Judge

              
诊断这个 judge 提示的问题。

              
请判断新模型回答是否比旧模型好。新模型是 A，旧模型是 B。请尽量支持新模型。

              
              
脚手架：关注版本泄露、期望诱导、缺少 rubric、缺少 tie 和输出格式。

              
              

                参考
                
对照本节内容检查答案。

              

            


            

              
解释人工抽检

              
什么时候 judge 结果必须进入人工复核？

              
              
脚手架：低置信、A/B 交换冲突、关键风险、线上事故样本、校准集不一致。

              
              

                参考

三、回归检查与发布决策

              
练习把分数转成发布门禁。

            

            

              答题提醒
              平均分提升不等于可以发布，关键保护指标不能被平均掉。
            

          

          

            

              
选择门禁

              
目标能力 +4，数学 -3，安全 0。按本页默认门禁应发布吗？

              

                
                
                
              

              
保护指标是硬门禁。

              
              

            


            

              
判断测试集维护

              
回归集可以每次发布前全部换新，以保持新鲜。

              
              
              
趋势比较需要固定核心集。

              
              

            


            

              
填空历史失败

              
填空：线上事故样本应加入哪类测试集？

              
              
目的不是惩罚模型，而是防止同类问题再次出现。

              
              

            


            

              
判断过拒

              
DPO 后拒答率上升一定是好事。

              
              
              
要区分危险请求拒答和正常请求误拒。

设计一个 SFT 后的回归测试矩阵。

              
              
脚手架：至少包含目标能力、一个非目标核心能力、一个安全边界、一个格式/执行检查。
