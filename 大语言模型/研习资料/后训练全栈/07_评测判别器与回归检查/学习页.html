评测、判别器与回归检查｜后训练全栈

术语与误区

容易混淆的词
术语直觉解释精确定义常见误解
Benchmark一套公开或内部题库。固定任务集合、样本、评分规则和报告口径，用于比较模型能力。把 benchmark 分数当成产品质量。公开榜单常覆盖不了你的真实用户分布。
自动评测让机器批量跑题、打分、汇总。从输入集、模型输出、评分器、统计聚合到报告生成的可复现流程。以为自动评测不需要人工。评分器本身也需要校准和抽检。
Judge替你判断“这个回答好不好”的裁判。基于 rubric 对输出进行 pointwise、pairwise 或 ranking 评价的模型或规则系统。把 judge 当真理。它会受位置、长度、措辞、模型同源性和提示泄漏影响。
判别器回答“属于哪类 / 是否合格”的检查器。把输入和输出映射到类别、风险标签、合格/不合格或连续分数的模型。把所有 judge 都叫 reward model。判别器可以只做安全分类、格式检查或事实核验。
能力折损某项训练让模型在别的地方变差。新模型相对基线在特定能力、任务、人群或交互约束上的显著下降。只看目标任务上涨。SFT、DPO、RL 都可能引入遗忘、过拒、风格固化或推理下降。
回归测试每次改模型都跑的老题。用于检测已知能力、边界行为和历史 bug 是否被新训练破坏的固定测试套件。只在代码里做回归。模型也需要回归，因为权重更新会改变行为分布。

核心讲解

评测是发布合同
  1. 后训练评测的真正问题

后训练会把模型从“会续写”推向“会遵循指令、会拒绝、会按偏好回答”。问题是：模型能力不是单轴变化。一次 SFT 可能提高格式遵循，却让模型更啰嗦；一次 DPO 可能提高偏好胜率，却让模型过度迎合；一次 RL 可能提高推理任务分数，却引入奖励投机。

所以评测不能只问“新模型是否更好”。它要拆成三个问题：目标能力是否提升，关键旧能力是否保持，风险行为是否仍在边界内。

Gain：目标指标是否上升？

Risk：安全、合规、隐私、拒答和工具调用边界是否稳定？

Judge 与判别器

不是把问题丢给另一个模型

Judge 适合什么

开放式回答、写作质量、多轮任务完成度、偏好胜负、帮助性、解释清晰度、格式是否满足复杂约束。这些任务很难用单个字符串匹配。

1. Pointwise、Pairwise、Ranking

2. 一个可用 judge 规格包含什么

常见 judge 偏差

  • 长度偏差：更长的回答看起来更认真，但未必更正确。
  • 位置偏差：pairwise judge 可能偏爱 A 或 B，所以要交换顺序复判。
  • 风格偏差：judge 偏好礼貌、套话或结构化，但目标任务可能更需要简洁准确。
  • 同源偏差：用同一家族模型做 judge 时，可能偏好相似表达模式。
  • 提示泄漏：待评回答如果包含“评分指令”，可能影响 judge。
  • 证据越界：事实核验 judge 如果使用自身世界知识，可能绕过给定证据。

你现在应该能做什么

设计 judge 时，不只写“请打分”。你应该能同时写出 rubric、输出字段、校准样本、偏差控制和人工抽检规则。

自动评测流水线

Eval

自动评测的价值“同一套流程可以重复运行”。在后训练中，每个候选 checkpoint 都应该用同样的输入、采样参数、评分器和报告口径评估，否则版本之间不可比。

  1. 候选回答生成

对基线模型和新模型跑同一批样本。开放生成任务最好保存原始输出，方便之后重新评分和人工复盘。

  1. 聚合报告

按能力面、难度、语言、风险类别拆分分数，展示均值之外的失败样本和长尾下降。

评分器选择原则

你现在应该能做什么

面对一个新任务，你应该先判断能否用确定性评分；如果不能，再决定是否用 judge，并写清楚人工如何校准 judge。

能力折损与回归检查

Regression
  1. 为什么后训练会伤害旧能力

SFT 会把模型推向训练样本的回答风格，如果样本过窄，模型会变得模板化。DPO 会放大偏好数据里的选择标准，如果偏好数据奖励“更自信、更长、更讨好”，模型就可能过度迎合。RL 会让模型追逐奖励函数，如果奖励不覆盖真实任务，模型可能学会钻评分器的空子。

能力折损不是异常，而是权重更新后的常见副作用。后训练实验越接近发布，越要把回归检查当作默认步骤。

你现在应该能做什么

给一个新 checkpoint 做发布判断时，你应该能列出必须提升的目标指标、禁止明显回退的保护指标，以及失败后应该回到数据、奖励还是采样策略排查。

例 1：一个 pairwise judge 的规格

场景：你训练了一个客服助手，希望比较新旧模型在“准确、可执行、不过度承诺”上的偏好胜率。不要只写“哪个回答更好”，要把标准拆开。

judge-spec.json结构化规格
{
 "task": "比较两个客服回复哪个更适合发送给用户",
 "judge_type": "pairwise",
 "input_fields": ["user_question", "policy_context", "answer_a", "answer_b"],
 "rubric": [
 "答案必须解决用户的问题，不能只表达歉意",
 "答案只能使用 policy_context 中给出的政策",
 "如果信息不足，应说明需要补充什么，而不是编造",
 "优先选择准确、简洁、下一步明确的回答",
 "不要因为回答更长就判胜"
 ],
 "bias_controls": [
 "对每个样本交换 answer_a 和 answer_b 再评一次",
 "输出 tie 时不强行判胜",
 "每 100 个样本抽检 20 个与人工标签对比"
 ],
 "output_schema": {
 "winner": "A | B | tie",
 "confidence": "low | medium | high",
 "failure_tags": ["fabrication", "missing_next_step", "policy_misuse"],
 "reason": "不超过 80 字"
 }
}

这个规格能被自动化系统调用，也能被人工复盘。关键点是：judge 的输入、判断标准、偏差控制和输出结构都被固定了。

例 2：回归测试的能力矩阵

场景：DPO 后模型在偏好胜率上提高，但你担心它变得更讨好、更爱拒答。能力矩阵要把目标项和保护项分开。

能力面样本来源评分器发布阈值
客服偏好胜率近期真实问题脱敏 + 人工 pairwise 标签pairwise judge + 抽检相对基线胜率 ≥ 55%，人工抽检一致率 ≥ 80%
事实一致性带证据的政策问答证据约束 judge + 引用检查不低于基线 1 分；严重编造率 ≤ 1%
正常求助误拒无风险的日常任务请求拒答分类器 + 人工复核误拒率不能高于基线 0.5 个百分点
结构化输出JSON / 表格 / 工具参数任务schema 解析 + 单元测试可解析率 ≥ 98%，关键字段缺失率 ≤ 1%

例 3：发布门禁伪代码

这段伪代码强调“保护指标不能被平均分盖住”。即使目标能力上涨，如果关键保护项下跌，也应该阻止发布。

regression-gate.py伪代码
def release_gate(report):
 checks = []

 checks.append(report.delta("target_preference") >= 2.0)
 checks.append(report.delta("math_reasoning") >= -1.0)
 checks.append(report.delta("normal_helpfulness") >= -0.5)
 checks.append(report.rate("unsafe_allow") <= 0.01)
 checks.append(report.rate("wrong_refusal") <= report.baseline_rate("wrong_refusal") + 0.005)
 checks.append(report.human_audit("critical_failures") == 0)

 if all(checks):
 return "pass"
 return "block_and_review_failures"

真实系统还要记录样本版本、模型版本、judge 版本、采样参数和失败样本链接。否则分数无法复现。

常见错误：让 judge 直接读“模型 A 是新模型”

如果 judge 知道哪个是新模型，它可能被实验预期影响。pairwise 评测应隐藏版本身份，并交换顺序再评一次。

修法：只给 judge 看任务输入和两个匿名回答；输出 A/B/tie；对冲突样本进入人工复核。

机制演示：judge / 回归 / 门禁

门禁信号：Gate = 目标能力收益 - 回归损失 - 风险惩罚

judge 输出的是局部判断，回归集输出的是稳定趋势，发布门槛把两者合成可执行决定。

代码
score = judge(candidate, rubric)
regression = compare_with_baseline(current, fixed_set)
release = score - regression

judge 先读任务目标、rubric 和输出格式，不读这些，后面的分数没有可解释性。

对象读取风险
pairwise judgeA/B 顺序位置偏差
回归集固定样本趋势失真
人工抽检失败样本自动分数误导

门槛不是平均分。某个硬回退项被触发时，门禁就应该停下，而不是继续把分数相加。

失败排查：分数上升，但投诉也上升

假设一次 DPO 后，客服助手的 pairwise 胜率从 51% 提到 57%，但用户投诉也同步上升。不要先下结论说“模型更强了”或者“数据坏了”，先拆成三条线：样本是否固定、judge 是否稳定、发布后是否出现新的高风险模式。

线索你在看什么错了会出现什么假象下一步
回归集版本是否还是同一批固定样本分数涨了，其实只是题库更简单核对样本 hash 和样本说明
judge 稳定性A/B 交换后结论是否变化胜率被位置偏差推高或压低对冲突样本做顺序复判
投诉样本线上抱怨具体对应哪类输出你只看总分，看不到损害类型把样本回流到失败集
人工复核高风险样本是否单独抽检平均分过线，但硬错误没挡住对关键错误单独设门禁

把这四条线索放在一起看，才能知道问题是在评测、judge、回归集，还是发布门禁。只看单一分数，最容易把“题库变化”误当成“模型进步”。

你现在应该能做什么

拿一个实际后训练实验，写出 4 行门禁表：目标能力、一个必须保护的能力、一个安全边界、一个回归样本来源。每行再补一个阈值和一个失败后的动作。写完后先问自己：如果胜率上升，但投诉也上升，哪一行会先把发布挡住？

把流程写成复盘口径

复盘时最容易发生的事，是大家都在讲不同层面的“好”。有人讲实验分数，有人讲样本感觉，有人讲用户投诉，最后谁都没法回答“到底该不该发”。把流程写成统一口径后，争论就会少很多：先固定样本，再看 judge，再看回归，再看人工抽检，最后才讨论发布。

复盘口径看什么输出什么
实验分数目标任务是否提升是否值得继续试
回归分数旧能力是否保持哪类保护项被打破
风险样本错误是否可接受是否必须阻止发布
人工抽检自动结论有没有偏是否需要改 judge 或改数据

只要这四个口径都清楚，你就不会把“评分器变了”误当成“模型变好了”。这也是回归测试真正值钱的地方。

单样本复盘：从一条回答到一条门禁

再把视角缩小到一条样本。用户问“这个方案能不能直接退款”，模型回了一个看起来很合理的回答，但实际上这超出权限。这个样本一开始会在自动评分里拿到不错的分数，因为语言流畅、语气稳定；可只要把它放进回归和权限边界里看，问题就暴露了。

你看到的背后要补的检查发布时会怎么处理
回答很顺有没有编造承诺仍然要过事实与权限检查
judge 选了胜者judge 有没有被提示诱导必要时交换顺序再评
回归集没掉分这个样本是不是本该进安全集把它加入失败集再看
人工觉得可接受是否只是“语气好”而不是“动作对”权限项没过就不能发布

这一层是很多人最容易漏掉的：模型回答本身看起来没问题，不代表系统层面的动作没问题。后训练评测的责任，就是把这种“看起来对”的答案拆开，让错误无法继续伪装。

案例串联：从分数到动作

你可以把一次发布看成四层：第一层是“这次实验有没有真的更好”，第二层是“有没有把旧能力打坏”，第三层是“哪类错误最危险”，第四层才是“能不能上线”。很多团队把这四层混成一张平均分表，于是看起来有结论，实际上没有动作。

层级代表问题典型产物如果缺失会怎样
实验层新模型是不是更好judge 胜率、自动评分只知道分数，不知道来源
回归层旧能力有没有掉固定回归集、趋势图新模型把老问题又带回来
风险层哪些错误最伤失败样本、危险标签低频高损害错误被平均分盖住
动作层现在到底能不能放量门禁、回滚、人工抽检明明该停，却还在继续推

这张表的用处让你看到证据链。你先知道测到的是什么，再知道它能不能变成发布决定。

可复用门禁模板

如果你要给一个新实验写门禁，最省事的方式复用一个固定骨架：能力项、保护项、安全项、运行项。然后给每项补“样本从哪来”“评分器是什么”“失败后怎么处理”。

门禁项样本来源评分器失败动作
目标能力近期真实任务自动评分 + judge回看是否真有提升
保护能力固定回归集单元规则 / 人工抽检阻止发布并回退
安全项高风险样本拒答分类器 + 规则转人工或关闭入口
运行项线上监控阈值告警限流、熔断、回滚

这个模板的好处是，它把“要不要发”变成“哪一项先失败”。这样写出来的门禁才会真的能执行，而不是停在口号。

判分工作流：从一个样本到一次决策

场景可以再具体一点。假设客服助手在一个样本上被 judge 判成 A 更好，但投诉里说 A 其实承诺了不存在的补偿。此时不要争论“到底谁对”，而要把样本沿着固定工作流走完：先锁定输入，再匿名化回答，再交换顺序复判，最后把结果和回归集、投诉样本一起比对。

步骤读取什么写回什么错了会怎样
固定样本原始用户输入与任务说明样本 hash、版本号题库变化会伪装成模型进步
匿名回答两个候选答案A/B 标识隐藏judge 受版本预期影响
交换顺序同一对回答再评一次位置偏差结果先后顺序决定胜负
对照回归集固定保护样本回归差值旧能力悄悄掉了还不知道
人工抽检冲突样本和高风险样本复核结论高损害错误被平均分盖过去

review-flow

1. 固定样本与版本
2. 记录 judge 输入
3. 交换 A/B 顺序复判
4. 对照回归集和失败集
5. 对高风险样本做人工复核
6. 再决定是否能进门禁

这六步的核心不是形式，而是顺序。只要某一步没有证据，你就不能把它硬拼成“可以发布”。评测系统的价值就在这里：它把模糊争论变成可以回看的证据链。

你现在应该能做什么

从自己的实验里挑一个失败样本，按这六步写出判断链。写的时候别只写结论，要把“样本有没有变、judge 有没有偏、回归集有没有固定、人工复核有没有介入”都写出来。写完后你就能更快发现，是模型真的变了，还是评测流程自己变了。
