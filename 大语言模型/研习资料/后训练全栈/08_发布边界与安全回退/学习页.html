后训练全栈 / Unit 08

发布边界与安全回退

这节课把后训练的最后问题讲清楚：模型上线前不能只看排行榜、平均胜率或一组 demo。你需要定义可发布边界，检查幻觉、拒答、隐私和偏差，并准备模型失效时的回退路径。

本页完成后

  • 能把模型失败分成幻觉、拒答、隐私、偏差和策略失效。
  • 能为一个 LLM 功能写出发布门槛、拒答规则和回退策略。
  • 能解释为什么“更强模型”仍然需要监控、人审和降级方案。

学习目标

发布前决策

从分数转向风险

理解模型评测分数只是平均表现，发布决策还要看低频高损害错误、权限边界和用户补救能力。

拆开安全失败

区分事实错误、过度拒答、隐私泄露、群体偏差和越权执行，不把所有问题都塞进“模型不稳定”。

设计回退策略

为不确定、高风险或系统异常的请求安排回退：检索、澄清、拒答、人审、旧流程、只读模式或关闭入口。

你现在应该能做什么：看到一个准备上线的 LLM 功能时，先问“失败会伤害谁、伤害多大、能不能补救”，而不是只问“模型答得像不像”。

概念骨架

Boundary → Fallback
使用场景 → 风险清单 → 发布门槛 → 回退路径 → 监控复盘

发布边界的直觉很简单：先说清楚模型被允许做什么，再说清楚模型不该做什么，最后说清楚模型不确定或失控时系统应该怎么退。术语上，这叫把能力边界、风险控制、运行监控和事故响应放进同一个上线流程。

你现在应该能做什么：把发布看成一个闭环，而不是训练完成后的单点审批。

术语与误区

关键概念
术语 直觉解释 精确定义 常见误解
发布边界 给模型画上线使用范围。 对任务、用户、输入、输出、权限、风险等级和可接受失败率的明确定义。 以为“模型能力强”就等于“所有场景可上线”。
幻觉 模型把不确定的内容说得像真的。 输出与可验证事实、给定上下文或系统状态不一致，但表达上显得自信或完整。 以为只要接入检索就不会幻觉；检索只能减少部分事实缺口。
拒答 模型说“我不能帮你”。 模型基于安全、隐私、权限、能力不足或策略约束，不提供用户请求的目标内容。 把所有拒答都当成安全；过度拒答会伤害可用性，漏拒答会伤害安全。
隐私泄露 模型吐出了不该暴露的信息。 训练数据、上下文、日志、工具响应或其他用户信息被不当记忆、推断、输出或外传。 只盯训练数据泄露，忽视 prompt、RAG 文档、插件返回、日志和导出记录。
偏差 模型对某些人或场景系统性不公平。 在群体、语言、地区、职业、年龄、性别、文化或领域上出现稳定的质量差异或刻板输出。 以为偏差只是“说了冒犯的话”；许多偏差表现为漏召回、低质量建议或不合理拒答。
回退策略 模型不可靠时系统怎么退。 在不确定、高风险、异常或越权条件下，切换到更保守、更可控流程的机制。 把回退理解成“报错页面”；好的回退会尽量保留用户目标并降低风险。

核心讲解

上线前四问

1. 发布边界先于发布指标

一个模型在通用问答榜上表现很好，不代表它适合处理医疗建议、财务决策、客户投诉、内部权限操作或儿童用户。发布边界要回答：目标用户是谁，模型能接触什么数据，能产生什么动作，哪些输出必须被阻止，失败后谁承担成本。

后训练通常会提高指令遵循和偏好对齐，但也可能带来新问题：模型更会迎合用户，可能更自信地编造；安全数据过强，可能导致正常问题也拒答；偏好数据覆盖不均，可能让小语种、少数群体或冷门领域质量更差。

可发布性 = 能力收益 - 风险暴露 + 可控回退。
如果风险暴露不可观测，或回退不可执行，就不应该只凭离线均分上线。

判断句

发布边界不是“模型最多能做什么”，而是“在可监控、可解释、可补救条件下，系统允许模型做什么”。

2. 幻觉：从“答错”升级为“错误影响”

幻觉的核心不是模型犯错，而是模型把错误包装成可执行事实。发布评测要按影响分层：闲聊事实错误、资料摘要误引、代码建议错误、法律医疗建议错误、权限状态错误，严重程度完全不同。

常用控制方法包括：让模型承认不确定，要求引用可见来源，把事实性任务转到检索或数据库查询，输出前做一致性检查，对高风险答案加人审或二次判别器。注意，检索增强不是万能解法：检索结果错、引用错位、上下文过长或模型忽略证据，仍会产生幻觉。

幻觉评测要看三件事

  • 事实正确率：答案是否被来源支持。
  • 校准程度：不确定时是否降低语气、请求澄清或拒绝编造。
  • 损害半径：错误是否会驱动用户做不可逆行动。

3. 拒答：安全与可用性的双边界

拒答有两类错误：应该拒而没有拒，是安全漏判；应该答而拒绝，是过度保守。发布前要准备“拒答矩阵”，把请求按内容风险、用户权限、上下文充分性和任务可逆性分组。

好的拒答不是机械说“不”。它应该说明无法满足的边界，提供安全替代，例如给出一般原则、建议咨询专业人士、转人工客服、引导用户提供非敏感信息，或把危险操作改成只读说明。

拒答矩阵示例

请求类型模型行为替代路径
明显违法或伤害性操作拒答提供安全、合法、高层次信息
缺少权限的账户操作拒答或要求认证跳转权限流程或人工审核
事实不确定的专业建议澄清或保守回答引用来源、提醒限制、人审
正常但敏感的帮助请求谨慎回答最小必要信息、避免收集隐私

4. 隐私：不是只防训练数据泄露

LLM 应用的隐私风险来自多处：训练集中可能含个人信息，RAG 文档可能越权召回，prompt 可能包含用户敏感数据，工具调用可能返回内部字段，日志和导出记录也可能二次传播。发布边界要覆盖数据进入、模型处理、工具访问、输出展示和日志留存。

最小化原则很实用：不需要的隐私数据不要收集，能在权限系统里裁剪的不要交给模型，能用结构化结果表达的不要让模型自由复述。对敏感字段要做脱敏、权限检查和输出过滤，但不要把过滤器当成唯一防线。

隐私检查清单

  • 输入是否包含身份证、电话、地址、健康、财务、账户或儿童信息。
  • 检索结果是否只来自当前用户有权访问的文档。
  • 工具响应是否返回了模型无需知道的内部字段。
  • 日志、评测样本和导出记录是否会保留原始敏感文本。

5. 偏差：看均值看不出的质量差异

偏差不是只发生在敏感话题。客服模型可能对方言用户理解更差；招聘助手可能给某些背景候选人更低质量建议；医疗问答可能对女性、老人或少数族裔症状解释不足；代码助手可能偏向英语生态。上线前要做分组评测，而不是只看总体准确率。

偏差还会和拒答交织：某些群体的问题更容易被误判为违规，某些语言的安全规则更粗糙，某些地区的事实知识更新不足。后训练数据、偏好标注、评测集和线上反馈都可能放大这种差异。

偏差不是一个单指标

至少同时看：不同群体的任务成功率、拒答率、幻觉率、语气差异、建议可执行性和用户申诉率。只看“是否出现冒犯词”会漏掉更隐蔽的质量不公平。

6. 回退策略：按风险选择退路

触发条件：模型置信不足

先请求澄清，或给出有限范围的回答，并明确哪些信息缺失。

回退路径：澄清 / 保守回答

低风险问题的重点是避免编造，同时不打断用户任务。

监控指标：澄清后成功率

如果澄清后仍频繁失败，说明路由、提示词或知识源需要修。

观察重点：回退不是统一按钮，而是按风险、权限和可逆性选择不同退路。

7. 发布门槛应写成可执行规则

“模型安全性良好”不是门槛，“在 500 条高风险拒答集中，漏拒答率不超过 1%，并且所有漏判样本经人工复核后有补丁计划”才是门槛。门槛要能被测试、能被复现、能触发上线或暂停决策。

发布门槛通常包含四类：能力门槛、安全门槛、隐私门槛、运行门槛。运行门槛容易被忽略，例如延迟、成本、超时、工具失败、日志可观测性、人工接管 SLA 和一键关闭开关。

最小发布包

  • 任务定义和禁止范围。
  • 红队样本、回归样本、分组评测样本。
  • 拒答和回退规则。
  • 线上监控、抽检、人审和事故流程。

你现在应该能做什么：把“模型上线”拆成可审查对象：边界、风险、门槛、回退、监控，而不是只讨论模型参数和训练算法。

案例与清单

Worked Examples

案例 1：企业知识库助手

场景：员工可以询问内部制度、项目文档和客户资料。模型接入 RAG，并能总结搜索结果。

风险错误表现发布控制
幻觉引用不存在的政策条款要求答案带文档片段引用；没有证据时说不知道
隐私跨部门召回客户信息检索前做权限过滤；工具响应只返回最小字段
拒答正常制度问题被安全策略误拒把安全拒答和权限拒答分开评测
回退检索服务超时降级到“无法查询最新文档”，提供人工入口

常见错误

团队只测“回答是否流畅”，没有测“引用是否存在”“用户是否有权看引用”“检索失败时模型是否编造”。上线后用户会把模型语气当成系统事实，错误成本会被放大。

修法：把事实性回答绑定到来源，把权限检查放在检索前，把检索失败纳入回退测试。

案例 2：客服自动回复

场景：模型根据用户投诉生成回复，可查询订单状态，但不能直接退款。

发布边界样例 policy.md 
允许：
- 解释公开售后政策
- 查询当前用户本人的订单状态
- 生成安抚性回复草稿

禁止：
- 承诺超出政策的赔偿
- 查看其他用户订单
- 直接执行退款、改价、封号等不可逆操作

必须回退：
- 用户身份未确认 → 要求登录或转人工
- 政策条款冲突 → 引用条款并转人工复核
- 情绪激烈或威胁自伤 → 触发人工优先队列
- 工具超时 → 告知暂时无法查询，不编造订单状态

这段边界把“模型能说什么”和“系统能做什么”分开。模型可以写回复草稿，但不可逆动作仍要走权限系统或人工审核。

案例 3：医疗健康问答

高风险领域的回退更严格。模型可以解释常识、提醒风险信号、建议就医路径，但不应替代医生诊断，不应根据模糊描述给确定治疗方案。

发布门槛：严重症状识别、危险建议拒答、隐私最小化、紧急情况转人工或当地急救建议，都应该有专门测试集。

发布前检查清单

能力

核心任务成功率、长尾任务、上下文长度、工具失败、不同语言和不同用户表达。

安全

危险请求漏拒、正常请求误拒、越狱、间接 prompt 注入、过度代理行为。

隐私

敏感输入、越权检索、日志留存、输出脱敏、第三方工具传输。

运行

延迟、成本、限流、熔断、灰度、人工接管、事故复盘和回滚开关。

五个最容易混淆的点

容易混淆区别实务判断
拒答 vs 回退拒答是回答策略；回退是系统路径。拒答后仍可给安全替代或人工入口。
过滤器 vs 安全架构过滤器只处理一层输入/输出；安全架构覆盖权限、工具、日志和监控。不要把敏感数据先交给模型再指望输出过滤。
幻觉率 vs 损害风险幻觉率是频率；损害风险还看错误后果。低频但高损害错误也可能阻止发布。
公平语气 vs 公平质量不冒犯不代表质量公平。按群体看成功率、拒答率和建议可执行性。
离线评测 vs 线上监控离线评测覆盖已知风险；线上监控发现分布漂移和未知风险。上线后仍要抽检、告警、回流样本。

你现在应该能做什么：拿一个真实功能，从任务、数据、权限、输出和异常五个角度写出第一版发布检查表。

你现在应该能做什么

          
现在你可以拿一个真实功能，写出“低风险 → 中风险 → 高风险 → 事故”的四档动作表。每一档都要能说清楚，发生什么就该停、该退、该转人工，以及事故样本最后回流到哪里。

        

        

          
灰度失败复盘：别把平均分当终点

          
很多发布事故不是“模型突然坏了”，而是系统把不同风险混在一起看。低风险问答、带证据的事实查询、高损害动作，三类请求对错误的容忍度完全不同。你如果只看平均成功率，就会把最危险的请求当成普通样本处理。

                错误损害小，平均值能部分反映问题
              

              

                中风险事实
                引用是否存在、证据是否足够
                无证据时回退到检索或提示不知道
                事实错误会以“看起来很自然”的方式扩散
              

              

                高损害动作
                权限、身份、不可逆操作
                直接拒答或转人工
                一次错判可能就是一次事故
              

              

                事故请求
                告警、投诉、越权样本
                熔断、回滚、关闭入口
                此时的目标不是提分，是减损
              

            
          

          
一旦你把请求分层，发布边界就从抽象口号变成了实操顺序：先能看见风险，再决定回退方式，最后才是灰度放量。

        

        

          
事故日志怎么回流

          
一次事故最有价值的部分，不是当下多紧张，而是它能不能变成下一轮评测样本。你应该记录触发条件、系统动作、失败后果和回流结果，然后把它们加入回归集或失败集。这样后训练和上线监控才会互相咬合，而不是各做各的。

        

        

          
事故后的回滚动作链

          
当告警真的响了，系统不该继续讨论“平均分是不是还行”。正确顺序通常是：先关高风险入口，再切旧版本，接着保留样本和日志，最后把问题分流给人工和数据回流。这样做的目的不是惩罚模型，而是尽快把损害半径压小。

          

            
              

                动作
                目的
                如果漏掉会怎样
              

            
            
              

                关闭高风险入口
                先停止最危险的路径
                事故继续扩散
              

              

                切旧版本
                回到已知可控状态
                新问题没有缓冲区
              

              

                保留样本
                为后续分析留证据
                只能凭印象复盘
              

              

                人工分流
                让重要请求先得到人接管
                系统还在自动放大损害
              

              

                回流到评测集
                把事故变成新门禁
                同类事故会再次出现
              

            
          

          
这条链路的本质是“先减损，再复盘，再修规则”。只要顺序对了，事故就会变成下一次上线更稳的原因；顺序错了，事故只会被换个名字重复一遍。

        

        

          
风险分层检查表

          
当你拿到一个新功能时，先不要急着写 prompt。先把请求分层：哪些是纯信息问答，哪些需要证据，哪些涉及权限，哪些一旦错就不可逆。分层以后，再去写回退动作和监控阈值，顺序不能反。

          

            
              

                分层问题
                判断标准
                对应动作
              

            
            
              

                这是不是纯信息问答？
                错了也不会造成实质损害
                允许回答，但保留最小监控
              

              

                这是不是证据驱动任务？
                答案必须依赖可见来源
                检索、引用、无证据不答
              

              

                这是不是权限或身份任务？
                用户身份决定可见内容
                先验证身份，再允许动作
              

              

                这是不是不可逆任务？
                错一次就很难补救
                默认拒答或人工接管
              

            
          

          
只要这张表还没写清楚，发布边界就不能算完成。它的作用是把“风险感觉”变成“系统动作”，让上线后每次出事都能知道该退到哪一步。

        

        

          
发布链路：从请求进入系统到最终动作

          
拿“用户未登录却想查订单”这个场景来说，发布链路不是一句“拒答”就结束了。真正的链路应该是：先检查身份，再判断请求是否属于高损害动作，再决定是直接拒答、转人工、还是返回最小信息。这个顺序一旦错了，系统就可能先泄露再补救，或者先误拒再丢掉用户体验。

          

            
              

                链路环节
                读取什么
                动作
                失败表现
              

            
            
              

                身份检查
                用户是否已登录
                决定是否能继续查订单
                未登录也被当成本人处理
              

              

                权限检查
                这个用户能看哪些字段
                只开放必要信息
                越权看到别人的订单或地址
              

              

                风险判断
                请求是不是不可逆或高损害
                必要时直接拒答
                模型替代系统做了不可逆决定
              

              

                回退路径
                当前动作是否可撤回
                转人工、只读、关闭入口
                事故发生后没有退路
              

            
          

          
把这条链路写清楚之后，你会发现发布边界不是一个单点规则，而是一串从输入到动作的控制点。每个控制点都应该有监控和回退，不然就是把责任推给最后一步。

        

        

          
你现在应该能做什么

          
现在你可以拿一个业务功能，写一条从“请求进入系统”到“最终动作结束”的控制链。链条里至少要有身份、权限、风险、回退四步。写完以后，你就能更快看出系统到底是“不能做”，还是“还没到能做的那一步”。

未登录查订单：一条样本怎么走完

          
把“用户未登录却想查订单”拆开看，最容易犯的错误是只写一句“拒答”。这样写的规则太短，短到你很难知道它到底挡住了什么。更稳妥的方式，是把请求沿着身份、权限、风险、动作、日志一路走过去。这样你就能知道，系统究竟是应该直接拒，还是应该转人工，还是只给最小信息。

          

            
              

                环节
                读取什么
                允许动作
                失败后果
                修法
              

            
            
              

                身份检查
                用户是否已登录
                继续查订单，或先引导登录
                未登录也被当成本人处理
                把身份校验放在最前面
              

              

                权限检查
                这个用户能看哪些字段
                只返回订单号、状态、时间这类最小字段
                越权看到地址、电话、备注
                把返回字段做白名单
              

              

                风险判断
                请求是不是涉及退款、改价、封号
                高风险时直接拒答或转人工
                模型代替系统做了不可逆动作
                高损害动作默认不放行
              

              

                输出生成
                模型拿到的上下文和工具结果
                回答必须和证据一致
                看起来很自然，但其实在编造
                输出前做最小化和引用检查
              

              

                事故日志
                本次请求、动作、后果、回流结果
                把样本送回失败集
                同类问题以后还会再来一次
                把事故变成新评测项
              

            
          

          
order-policy.py

          
def handle_order_query(user, request):
    if not user.logged_in:
        return {"action": "prompt_login", "reason": "no_identity"}
    if not user.can_view_order:
        return {"action": "deny", "reason": "permission_block"}
    if request.kind in {"refund", "price_change", "account_lock"}:
        return {"action": "human_handoff", "reason": "high_risk"}
    return {"action": "answer_with_min_fields", "reason": "low_risk"}

          
这段伪代码的重点不是写法，而是顺序。先身份，再权限，再风险，最后才是回答。只要顺序正确，系统就能在必要的时候停下来，不会把“模型会说话”误当成“系统可以放行”。

        

        

          
失败后的信号

          
如果系统把未登录请求直接答了，事故日志里通常会看到越权字段外泄；如果系统一味拒答，事故日志里会出现转人工爆量和正常请求误拒。前者说明边界太宽，后者说明边界太窄。它们都不是“模型不够强”，而是控制链的某一环写错了。

        

        

          
你现在应该能做什么

          
从你自己的产品里挑一个请求，按这条链路写一版动作表：先写身份，再写权限，再写风险，再写回退和日志。写完以后，你应该能说清楚“系统在这一步到底该做什么”，而不是只说“模型应该更安全一点”。

        

        

          
发布前五件事

          
在一个功能真正进入灰度之前，最少要过五件事：先知道它处理什么请求，再知道它碰不碰证据，再知道它是不是涉及身份和权限，再知道它一旦错会不会不可逆，最后还要知道错了以后怎么退。这个顺序听起来像流程，但其实是在帮你决定风险预算。

          

            
              

                检查项
                核心问题
                最小动作
              

            
            
              
请求类型这是什么场景先分层

              
证据需求答案是否需要外部来源检索或无证据不答

              
身份权限用户是否有资格看见它先验证再放行

              
不可逆性错了能不能补救高风险默认拒答

            
          

          
如果这五件事没想清楚，灰度就会变成碰运气。把它们先写在纸上，再写到系统里，事故率通常会明显下降，因为你终于知道应该把刹车装在哪里。

          
更实用的做法，是把这五件事直接变成一次发布前的检查顺序：先看请求是否属于高风险场景，再看是否需要证据，接着看用户身份和权限，随后确认有没有不可逆动作，最后确认回退开关是否真的可用。这样写出来的门禁，不会只在文档里正确，而是能在灰度时真的起作用。

          
最后再记住一句话：边界不是为了限制模型，而是为了限制损害半径。这个思路一旦稳住，发布、回退、监控和复盘就会自然接成一条线。
