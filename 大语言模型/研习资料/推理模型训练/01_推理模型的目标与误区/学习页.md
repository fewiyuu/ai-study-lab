# 学习目标 {#goals}
<!-- tag: foundation -->

先说清目标
读到“推理训练”时，你要先能说出：这里关心的是可验证任务上的求解可靠性，而不是回答更长、标签更像、格式更整齐。


再拆信号
你要能把一条训练链路拆成任务分布、采样、奖励、更新和评测，并指出每一层可能把信号带偏的位置。


最后做判断
看到奖励函数、benchmark 提升、蒸馏轨迹或分布式框架时，能判断它是在增强能力，还是只是在优化表面结果。

## box info
### 读这页时，先问自己三件事
第一，目标有没有落在可验证成功上。第二，模型学到的是探索和修正，还是长度和格式。第三，证据是不是来自未污染评测，而不是单个榜单分数。

# 先抓主线 {#overview}
<!-- tag: training loop -->
<!-- layout: grid2 -->

## box
### 一个最小闭环
推理训练先看题目分布，再让模型对同一题采样多个候选，再把答案交给规则、测试或判题器验证，最后把奖励送回优化器。这个闭环里，最容易出错的奖励到底指向什么。
如果奖励只盯着长度、标签或格式，模型会很快学会投机。真正要看的，是它有没有更愿意探索、检查、纠错，并把这些行为转成更高的可验证成功率。
把这件事想成一个解题实验就容易了：题目是输入，候选解法是中间过程，最终答案和测试结果是反馈。实验设计本身先决定了你最后看到的是“真会做”还是“会表演”。

## box info
### 观察顺序
题目怎么来 → 候选怎么采样 → 奖励怎么判 → 参数怎么更新 → 评测怎么验。只要其中一环偷换目标，后面的分数就不可信。
这条顺序很重要，因为后面的改进都只能放大前面的定义。你如果在奖励里奖励长度，模型会先学会把长度变长；你如果在评测里只看一个榜单，模型会先朝那个榜单的漏洞靠拢。

# 术语与边界 {#terms}
<!-- tag: vocabulary -->

推理模型
在多步求解任务上更愿意探索、验证和修正的模型。它不是“输出带 think 标签”的模型。


可验证奖励
能通过答案匹配、测试用例、规则环境或判题器把输出映射成反馈的奖励。它的关键是可判定，而不是“看起来专业”。


长链思考
一种可观察的输出行为。它可能有用，也可能只是重复、绕圈和格式投机。


GRPO
把同一题的多个候选放在一组里比较相对优势的优化方法。它不负责替你定义什么叫对。


泛化能力
模型在未污染题、变体题和不同难度层级上还能保持稳定解题的能力。单个榜单涨分不等于泛化完成。

## box
### 这几个词为什么要分开
“长链思考”描述的是输出表面，“可验证奖励”描述的是反馈来源，“推理模型”描述的是训练后形成的行为倾向，“泛化能力”描述的是离开训练题以后还能不能站得住。把它们混在一起，最常见的后果就是把格式稳定当成能力稳定。
本页后面的所有例子都在做同一件事：把这些词重新拆开。你会看到同一个模型，换一种奖励、换一套采样、换一个评测集，结论就可能完全不同。

## box warn
### 这里先别追的东西
KL 细节、rollout 调度、GRPO 公式、verl 分布式实现和 Open-R1 数据配方，后面单元会展开。本页只负责把目标和误区钉住。
如果你现在就去抠优化公式，反而容易把注意力从“奖励在鼓励什么”挪开。先把目标和证据认清，再回头看算法，会更省力。

# 例子 {#example}
<!-- tag: worked cases -->

## box
### 案例 1：长度奖励
如果奖励写成“超过 1200 token 加分，出现 boxed 答案再加分”，模型最容易学到的是写得更长、加标签、套格式。它可能更像在“表现推理”，但没有验证最终答案是否真的对。
错误奖励函数复制
```python
def reward(output, problem):
 score = 0.0
 if "<think>" in output and "</think>" in output:
 score += 0.3
 if len(tokenize(output)) > 1000:
 score += 0.3
 if has_boxed_final_answer(output):
 score += 0.4
 return score
```


## box soft
### 案例 2：更接近可验证目标
更稳的做法是先抽取最终答案，再用规则或测试器判对错。格式可以给很小的辅助分，但不能盖过正确性。
可验证奖励函数复制
```python
def reward(output, problem):
 final = extract_final_answer(output)
 if final is None:
 return -0.2

 if verify_answer(final, problem.reference_answer):
 return 1.0

 if has_clear_reasoning_structure(output):
 return 0.05
 return 0.0
```

# 同题三奖励 {#case-lab}
<!-- tag: input / output -->

同一题不会因为题目换了就突然变难懂，真正变的是你怎么奖励它。下面把同一个小样本放进三种奖励里，看模型最可能学到什么。

同一题的设定
写在奖励里
模型更可能学到什么
最先坏掉的地方


数学题，答案可以直接判定
只奖励最终正确性
探索、检查、纠错
奖励覆盖不全时，可能不会主动写清步骤


同样的数学题
奖励 think 标签和 1200 token 以上长度
长输出、模板化、中间过程堆砌
正确性被挤到边缘，样本越来越像作文


同样的数学题
只盯某个公开 benchmark 的分数
适配榜单格式、采样设置和题型漏洞
榜单涨分，外部任务不一定跟着涨

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box
### 失败后果
如果你把“正确性”放得太靠后，模型会先学会把输出写得像解题，而不是先学会解题本身。等你再回头看日志，reward 可能已经很好看了，但样本会越来越整齐、越来越长、越来越像一套固定话术。
这就是为什么本页一直强调：同一题、同一模型、同一框架，不同的奖励定义会导向不同的训练行为。你把什么信号放进了框架。

## box info
### 看日志时先看哪一行
先看奖励项有没有直接对应最终答案，再看格式项是不是被放大，再看评测是不是和训练题过于相似。这个顺序很朴素，但它能帮你很快判断一个实验是在学能力，还是在学投机。
如果你愿意把这个顺序记成一句话，就是：先看任务，再看奖励，最后才看分数。分数永远是结果，不是定义。

# 一轮日志怎么读 {#log-reading}
<!-- tag: trace / failure -->

这一段把同一个训练步拆成日志、判断和下一步动作。你以后看到 reward 曲线、长度曲线和评测曲线时，最怕的曲线看起来很好却在错误方向上变好。

日志项
你看什么
如果正常
如果偏了


reward
有没有直接对应最终答案成功
正确样本更高，错误样本更低
长度变长就涨分，格式更像就涨分


length
输出是不是突然持续变长
只在需要解释时稍微变长
重复、绕圈、固定模板上升


eval
未污染题和错误样本有没有同步改善
外部任务和训练内任务一起改善
单榜单涨分，别的任务不动

训练日志片段复制

```text
step=142
reward=0.84
mean_length=1287
eval/math=+0.2
eval/ood=+0.0
note=样本开始大量使用 boxed 结尾和固定 think 模板
```

## box
### 这条日志最先说明什么
如果 reward 很高，但 mean_length 持续上升、eval/ood 没有跟上、样本还开始固定用同一种模板，第一怀疑对象奖励定义。这个步子已经在告诉你：模型学会的是刷分路径，不一定是解题路径。
换句话说，日志里最危险的组合“训练看起来成功，外部判断没有一起成功”。这时候你要回去看 reward 是否直接验证答案、采样是否太单一、评测是否太接近训练题。

## box info
### 把这一步接到练习区
Q6、Q7 和 Q9 专门练的是这种读法：先看奖励有没有偏，再看采样和评测有没有跟上。Q17 则把它推进一步，要求你设计一组反投机评测，把“看起来会做”和“真的会做”区分开。
如果你已经能根据这一条日志说出“最先查哪一层”，说明你在读训练信号。后面进到 Open-R1 和 verl 时，这个判断方式会继续用得上。

# 奖励项怎么排优先级 {#reward-grid}
<!-- tag: operation bridge -->

下面这张表是一个更实用的检查顺序。你不需要一次设计完美奖励，但至少要知道哪些项该放前面，哪些项只能做辅助，哪些项一旦放大就会把目标带歪。

给一份最常见的配置看：`primary=correctness`，`auxiliary=[format, length]`，`eval=[in_domain, ood]`。只要主奖励不是 correctness，先别急着调学习率或换框架，因为后面的改动很可能只是在放大错误主目标。

奖励项
优先级
怎么用
常见误用


最终答案正确性
最高
先作为主奖励，再谈其他补充项
被长度分和格式分挤到后面


验证器 / 测试用例
高
把答案成功映射成可判定反馈
验证器只覆盖表面格式，不覆盖真实任务


格式辅助分
低
只在不影响正确性的前提下轻量使用
把它放到主奖励位置


长度或 think 标签
最低
最多做观察项，不做主目标
直接奖励长度和模板化写法

奖励项检查片段复制

```python
def check_reward_terms(config):
 primary = config["reward"]["primary"]
 auxiliary = config["reward"].get("auxiliary", [])

 if primary != "correctness":
 return "主奖励应该先对齐最终答案正确性"
 if "length" in auxiliary:
 return "长度只能做观察项，不要放大"
 if "think_tag" in auxiliary and "correctness" not in primary:
 return "think 标签不能代替验证器"
 return "ok"
```

## box
### 如果顺序写反了
模型会先学会被奖励的最容易那部分：长度、标签、模板、固定结尾。等这些东西学得足够熟，你再想把正确性拉回来，成本就会明显变高，因为模型已经形成了稳定的刷分路径。
所以最稳的做法“把主奖励排对顺序，再谨慎加辅助项”。这也是为什么同一个训练框架，换一份奖励配置，结果就可能完全不同。

## box info
### 这一节和练习怎么接
Q6 让你抓住长度奖励的问题，Q9 让你从代码里识别主奖励偏差，Q10 则要你自己写一份奖励、采样和评测一起成立的目标。学完这一节后，你应该能在看到一份 reward config 时，先指出主次顺序，而不是先夸它写得完整。
这就是本页最后一个完整链条：先看目标，再看奖励，再看检查项，最后才看分数。后面的单元会继续把这个顺序延长到复现和系统层。
如果你以后在日志里看到 ``reward``、``mean_length`` 和 ``eval/ood`` 三个数字一起变了，就先对照这一节看主奖励有没有被写歪。很多时候，真正该修的不是模型，而是配置顺序。
Q17 继续做的，就是把这一套检查顺序变成反投机评测：去掉格式提示、换到未见题、再看错误样本。这样你不仅能发现偏，还能把偏差定位到具体一层。

# 常见误区 {#mistakes}
<!-- tag: failure modes -->

| 误区 | 为什么会错 | 更稳的检查法 |
| --- | --- | --- |
| 把长 CoT 当目标 | 长度最容易被投机，重复和废话也会变长。 | 看正确率、验证通过率和错误修正质量。 |
| 把 SFT 蒸馏当 RL 证明 | 模仿强模型轨迹不等于学会了通过探索拿奖励。 | 区分“会模仿”和“会优化”。 |
| 把 GRPO 当魔法 | 算法只管怎么用奖励，不会自动修正错误目标。 | 先审题目和奖励，再谈算法。 |
| 只看训练 reward | reward 上升可能只是适配了脚本。 | 同时看离线评测、错误样本和未污染任务。 |
| 忽略可读性 | 纯追求推理奖励会把表达和稳定性拖坏。 | 把语言稳定、重复率和通用任务一起看。 |

## box info
### 误区不是用来背的，是用来查的
如果 reward 涨了，先检查奖励是不是写偏了；如果单榜单涨了，先检查数据污染和评测方差；如果输出越来越长，先检查长度是不是被你写进奖励里了。每个误区都对应一个下一步检查，而不是一个抽象感叹。
这一点也解释了为什么本页一直把“证据”和“能力”分开说。只要你能明确下一步该看什么，这个误区就已经开始被你处理了。

增加候选、换题型、看不同难度层级的表现

这张表的作用很直接：当你以后在后续单元看到训练结果不对劲，先别急着换框架，先把症状归到某一层。这个动作能帮你少走很多弯路，因为大多数问题“你奖励了不该奖励的东西”。

本页到这里为止，已经把你需要的检查顺序定下来了。后面你看到任何一个训练配方，都可以先拿这张表做第一轮筛查，再决定要不要往下钻。

### 6_length 错误诊断

**Q:** Q6 某训练奖励为：超过 1200 token 加 0.5，答案有 boxed 加 0.5。哪些问题必须指出？

- [ ] 没有验证最终答案是否正确
- [ ] 奖励会鼓励长度投机
- [ ] 这种奖励一定能提升泛化

Explain: 至少要勾出前两项。更完整的回答还会补一句：reward 上升不等于能力提升，长度和格式可以被投机。

### 7_bench 错误诊断

**Q:** Q7 一个模型在单个数学榜单涨了 8 分。哪些排查项必须继续做？

- [ ] 检查训练集和评测集是否污染
- [ ] 检查采样次数和温度是否变化
- [ ] 检查题目分布是否太窄
- [ ] 只看训练 reward 曲线就够了

Explain: 前三项都要继续查；最后一项误区。单榜单涨分只能说明证据变强了，不等于泛化已经完成。

### 8_distill 错误诊断

**Q:** Q8 为什么“蒸馏了 R1 的推理轨迹”不等于“复现了 R1 的 RL 训练机制”？

Explain: 蒸馏更接近模仿已有轨迹，在线 RL 关心的是在新任务上通过探索和奖励更新策略。前者能复制结果风格，后者才能说明模型学会了用奖励驱动求解。

### 9_read code-reading

**Q:** Q9 读下面的奖励函数，最主要的问题是什么？

- [ ] . 它没有验证正确答案，容易奖励格式和长度投机
- [ ] . 它用了 Python
- [ ] . 它没有使用 KL

Answer: . 它没有验证正确答案，容易奖励格式和长度投机
Explain: 这个函数主要奖励长度和标签，正确性只占很少或没有。

### 10_design synthesis

**Q:** Q10 设计一个小型数学推理 RL 目标。

Explain: 答案至少要包含四块：选择可验证的任务分布；奖励由答案正确性主导；采样说明候选数或温度；评测说明要用未污染题和错误样本分析。只写“用 GRPO 训练”不够。

### 11_code scenario-transfer

**Q:** Q11 代码竞赛任务里，哪些说法应该保留？

- [ ] 可以用测试用例直接判定任务成功
- [ ] 也可能针对测试集和模板投机
- [ ] 隐藏测试和扩展测试能缓解一部分风险
- [ ] 只要能过公开样例，就能证明泛化完成

Explain: 前三项应保留，第四项不应保留。代码任务适合可验证奖励，但也最容易被测试集和模板投机，所以必须保留隐藏测试、扩展测试和错误样本分析。

### 12_verl debug-checklist

**Q:** Q12 在用 verl 跑 GRPO 之前，你会先检查哪五件事？

- [ ] 奖励是否真的验证最终答案
- [ ] 的候选数和温度是否足够
- [ ] 训练题和评测题是否有污染风险
- [ ] KL 或其他约束是否会防止模型跑偏
- [ ] 评测是否覆盖未污染题和错误样本

Explain: 至少勾出 reward、rollout、data、KL、eval 五项中的四项，并能说明为什么它们和“目标是否靠谱”有关。若只勾运行环境或显存，不算答对。

### 13_bridge synthesis

**Q:** Q13 你如何用 Open-R1 的 SFT、GRPO、评测三个部分分别回答不同问题？

Explain: SFT 说明可以模仿什么样的起点；GRPO 说明如何通过奖励驱动探索；评测说明这些变化是否真的转化成未污染任务上的稳定提升。三者回答的是不同层次的问题。

### 14_refute argument

**Q:** Q14 反驳时，哪些要点必须提到？

- [ ] 目标不是长度和 think 标签
- [ ] 奖励必须验证答案成功
- [ ] 还需要 RL、采样和评测这几环
- [ ] 只改提示词就足够

Explain: 前三项都应出现，第四项是错误说法。好的反驳要把“目标、奖励、RL、评测”四层都说清楚，而不是只说“不是”。

### 15_summary summary

**Q:** Q15 用 150 字解释 DeepSeek-R1 给推理训练带来的核心启发和边界。

Explain: 核心启发要说清：可验证奖励配合 RL 能激励出更强的探索、检查和修正行为。边界要说清：它不等于所有任务都能靠长度或格式解决，也不自动保证可读性、通用能力或无污染评测。

### 16_feynman explain

**Q:** Q16 判断奖励是否合适时，第一步先看什么？

- [ ] . 奖励是否真的验证最终答案
- [ ] . 框架名字是不是最新
- [ ] . GPU 数量是不是更大

Answer: . 奖励是否真的验证最终答案
Explain: 先看奖励定义，再看框架；框架只是执行工具。

### 17_eval debug-checklist

**Q:** Q17 如果你怀疑模型只学会了格式投机，哪些评测设计要保留？

- [ ] 去掉格式提示再测一次
- [ ] 换到未见题分布
- [ ] 分析错误样本和失败类型
- [ ] 只看训练 reward 曲线

Explain: 前三项都应该保留，最后一项不是反投机评测。更好的答案还会补上隐藏测试或不同采样设置下的稳定性检查。

### 18_compare compare

**Q:** Q18 比较“用 R1 轨迹做 SFT 蒸馏”和“用可验证奖励做在线 RL”的优缺点。

Explain: 蒸馏的优点是便宜、稳定、容易起步，缺点是只能模仿已有轨迹；在线 RL 的优点是能通过探索学新策略，缺点是更吃奖励设计、采样和系统稳定性。两者不冲突，但证据强度不同。

### 19_report diagnosis

**Q:** Q19 训练后模型 reward 上升，但人工看样本发现反复自我否定、最后答案常错。写一个简短失败模式报告。

Explain: 要把现象、原因假设和验证实验分开写。常见原因包括奖励覆盖不全、过度惩罚简洁输出、采样太窄或验证器和真实任务不一致。下一步实验要能区分是哪一环出了问题。

### 20_role summary

**Q:** Q20 哪个说法最准确地概括三者角色？

- [ ] . 三者都是同一件事，只是名字不同
- [ ] . R1 讲路线，Open-R1 讲复现，verl 讲系统怎么放大目标
- [ ] . 只要装上 verl，目标就会自动正确

Answer: . R1 讲路线，Open-R1 讲复现，verl 讲系统怎么放大目标
Explain: R1 讲路线，Open-R1 讲复现，verl 讲系统会把目标放大。
