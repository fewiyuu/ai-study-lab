单元位置：奖励入口先钉住

position

前两单元已经把问题铺开了：推理模型训练要解决的不是“把回答写长”，而是“让模型在可验证任务上学会更稳地解题”。这一单元往前走一步，专门处理训练闭环里最容易出错的那一层：题目怎么构造、答案怎么抽取、验证器怎么判分、奖励怎么落到日志里。

这一步很关键。后面的 rollout、优势估计、KL 约束和分布式训练都只是放大器；如果奖励定义本身有漏洞，系统会把漏洞放大得更快。先把奖励入口看清，后面几单元读起来就不会只剩术语。

概念骨架：可验证奖励是一条链

concept map

把可验证奖励想成一个裁判函数会太短。真实训练里，裁判前后还有几层：任务先决定你能不能判分，抽取器决定你有没有对齐到最终答案，验证器决定什么叫正确，奖励函数决定怎么把判分变成学习信号，日志决定你后面能不能排错。

输入状态 题目池 + 目标定义

读的是任务是否足够可判分，是否会产出有差异的样本。

中间状态 final answer / code block / JSON

读的是 completion，写出的是验证器真正要看的对象。

判分状态 对错、部分得分、异常

读的是抽取后的对象和约束，写出的是可训练的反馈。

输出状态 标量 reward

写进优化器的是“哪条轨迹更好”的信号，不是整段思维链本身。

回放状态 prompt / completion / reason

如果训练变差，先靠日志定位是任务、抽取器还是验证器出问题。

任务类型 验证方式 最常见的坑
数学 答案抽取 + 等价比较 格式不同但语义相同的答案被误杀
代码 沙箱执行 + 单元测试 + 隐藏测试 写死公开样例、超时、访问危险模块
规则输出 JSON / 字段 / 约束检查 模型先学会漂亮壳子，再学会绕过规则
主观任务 人工判断或偏好模型 难以稳定自动化，不适合这一单元的第一版实验

术语边界：把词拆开

terms

可验证奖励

能通过规则、测试、环境反馈或判题器自动判断的奖励。重点是可判定，不是“看起来专业”。

验证器

把模型回答映射成分数的程序或规则。数学等价比较、代码测试、JSON 检查都可以是验证器。

答案抽取器

从长回答里取出最终答案的逻辑。它常常比想象中更重要，因为训练会很快利用格式细节。

任务构造

选择、生成、过滤和包装题目的过程。它决定模型练什么，也决定 reward 分布有没有学习价值。

奖励塑形

除了最终对错，还给格式、步骤、约束满足度等中间信号。塑形能降稀疏性，也可能把目标带偏。

奖励漏洞

模型用不符合真实目标的方式拿到高分，例如输出特定字符串绕过解析器，或写死测试样例。

这几组词要分开看：可验证奖励描述反馈来源，验证器描述怎么判，任务构造描述题目怎么来，奖励塑形描述怎么把判分写成可训练信号。混在一起，最常见的后果就是把格式稳定当成能力稳定。

来源分工：DeepSeek-R1 说明为什么可验证任务能撑起推理 RL；Open-R1 把路线拆成更细的公开流程；verl 负责系统侧的 rollout、reward 和更新；TRL 提供 GRPO 和 reward 函数的公开实现入口。

讲解 1：为什么 R1 类路线偏爱可验证任务

lecture 1

DeepSeek-R1 的公开路线把数学和代码放在前面，不是因为这类题目“更高贵”，而是因为它们能把反馈做得足够稳定。强化学习需要大量反馈，如果每条回答都要人来打分，成本和一致性都会成为瓶颈。可验证任务把反馈从人工偏好转成自动裁判，模型就可以在大量采样中看见哪些轨迹更容易走到正确答案。

这一点很重要：训练信号一旦能稳定给出，模型就会自己去找更有用的推理路径。它可能开始多试几步，开始回头检查，开始对比候选。这里的价值不是“思维链更长”，而是“有更高的可验证成功率”。

[balanced] 同一题下有对有错，reward 分布能拉开。
判断点 适合继续做 RL 吗 理由
能直接判对错 通常适合 反馈便宜，判分稳定
有隐藏测试或环境反馈 更适合 更难靠表面格式投机
主要靠主观评分 要谨慎 分数波动大，验证器难稳定
题目太容易或太难 先重构任务 reward 没有差异，学习信号会变薄
边界提醒

主观写作、风格评价、长文总结这些题型不是这单元的第一选择。它们可以做研究，但不适合拿来做推理 RL 的起步实验。

讲解 2：奖励函数要拆成几层

lecture 2

一个训练可用的奖励函数通常不止一句 answer == gold。模型会输出长文本，可能没有最终答案，可能答案格式不同但语义等价，可能代码超时，可能触发解析异常。奖励函数要把这些情况分开，否则训练日志里会混进大量不可解释的噪声。

最稳妥的写法，是把正确性、格式、异常路径拆成三层：正确性是主奖励，格式只帮抽取，不抢主位，异常必须单独记录。DeepSeek-R1、Open-R1 和 TRL 的公开材料都能找到这种分层思路，只是名字不同。

$$ R = w_c R_{\text{correct}} + w_f R_{\text{format}} - w_e R_{\text{error}} $$

R_correct 只看目标是否真正达成，R_format 只负责辅助解析，R_error 把解析失败、超时和执行异常单独记出来。权重里，正确性应当最大，格式项通常只做低权重辅助。

路径 应记录什么 常见错误
答案缺失 missing_final_answer 静默给 0，后面看不出是没答还是答错
格式不合规 format_failed 把格式失败和正确性失败混成一类
验证器报错 verifier_error 验证器出问题却被算成模型错误
最终判错 wrong_answer 只剩一个总分，后面无法回放原因
一个很实在的判断

如果格式奖励比正确性更显眼，模型很快会学会“看起来像在解题”，但不一定真的更会解题。这里的权重设计，不是为了把曲线做漂亮，而是为了让训练信号更接近目标。

讲解 3：任务构造决定 reward 分布

lecture 3

同一个 reward 函数，放在不同任务分布上，训练效果会完全不同。太容易的题，大多数 rollout 都得 1；太难的题，大多数 rollout 都得 0。真正有学习价值的，往往是当前模型“偶尔能答对、但还不稳定”的任务，因为同一 prompt 下会出现对错差异，优化器才有东西可学。

所以任务构造不是杂务，它本身就是训练设计的一部分。你要看难度有没有铺开、题型有没有混合、题目会不会泄露答案、验证成本会不会过高。Open-R1 这种复现路线之所以要把数据和训练拆开，就是为了让这些边界能被单独检查。

输入状态 题型、难度、是否泄露

如果题目太单一，训练时只会看到一种行为。

中间状态 同一题的多个回答

要看的是差异，而不是单个最优答案。

判分状态 全对 / 全错 / 有对有错

能拉开 reward 才有训练价值。

过滤状态 保留 balanced 样本

全对全错都容易把信号变薄，泄露题会让 reward 失真。

输出状态 可训练的样本集合

进入训练前，先确认样本还保留了差异和可回放信息。

[balance] reward 有分层，当前模型能从“偶尔对”里学到东西。
保留 丢掉 原因
有对有错的题 全对题 前者能提供相对优势，后者几乎没有训练差异
验证边界清楚的题 需要主观判定的题 这单元先练自动判分，不先碰偏好判分
来源和评测可隔离的题 直接来自 benchmark 的题 防止训练和评测混在一起
难度有层次的题 模板重复的题 覆盖面太窄，模型很快学会套路

讲解 4：数学型可验证奖励

math reward

数学题的验证通常从最终答案开始。问题在于答案经常有等价形式：1/2、0.5、50% 在很多题里语义一致；x=2 和 2 也可能等价。只做字符串比较会误伤正确答案，也会逼着模型迎合格式，而不是解决问题。

更稳的做法，是先抽取最终答案，再做数学等价比较，最后再看格式是否符合约定。格式项可以帮助解析，但不应该把正确答案压下去。

任务例子

题目：若 3x + 5 = 20，求 x。标准答案：5。可接受形式：5、x=5、5.0。不可接受形式：只写推理过程但没有最终答案。

completion final answer correct format reward
... 最终答案：5 5 yes yes 1.1
... x=5.0 5.0 yes yes 1.1
... 我会先移项 missing no no 0.0
... 答案是 6 6 no yes 0.1
数学奖励伪代码
def reward_math(prompt, completion, gold):
    final = extract_final_answer(completion)
    if final is None:
        return {"reward": 0.0, "correct": False, "reason": "missing_final_answer"}

    correct = equivalent_math(final, gold)
    format_ok = has_boxed_or_final_marker(completion)

    reward = 1.0 if correct else 0.0
    reward += 0.1 if format_ok else 0.0
    return {
        "reward": reward,
        "correct": correct,
        "format_ok": format_ok,
        "final_answer": final,
    }
设计注意

先抽取最终答案，再做等价比较；格式奖励只能辅助，不能让错误答案靠外壳拿高分；多答案题要把顺序、集合和误差容忍度写清楚。

讲解 5：代码型可验证奖励

code reward

代码任务的奖励通常来自测试。模型输出函数实现，验证器把代码放进受限环境运行，公开测试和隐藏测试都通过才给高分。相比数学题，代码奖励更接近真实行为验证，但也更容易碰到安全、超时、依赖和非确定性问题。

任务例子

实现 dedupe_keep_order(xs)：删除列表重复元素，并保留首次出现顺序。这里最有价值的不是“写出来”，而是“公开样例之外还能稳稳通过隐藏测试”。

奖励漏洞

如果只给公开样例，模型可能写死样例输出；如果执行环境不隔离，模型可能读文件、联网或调用危险 API；如果没有超时限制，坏代码会拖垮训练。

检查项 为什么重要 日志字段
沙箱执行 防止模型代码访问系统资源或外部网络 sandbox_status
超时限制 防止无限循环拖慢 rollout timeout
隐藏测试 降低写死公开样例的收益 hidden_pass_rate
部分得分 让奖励不至于只有 0/1 passed_tests
代码奖励伪代码
def reward_code(completion, tests):
    code = extract_python_code(completion)
    if not code:
        return {"reward": 0.0, "reason": "no_code_block"}

    result = run_in_sandbox(
        code=code,
        tests=tests,
        timeout_seconds=2,
        blocked_modules=["os", "subprocess", "socket"],
    )

    if result.error:
        return {"reward": 0.0, "reason": result.error_type}

    passed = result.passed_tests
    total = result.total_tests
    reward = passed / total
    if passed == total:
        reward += 0.2
    return {"reward": reward, "passed": passed, "total": total}
读这段代码时，重点看什么

extract_python_code 先把可执行对象抽出来；run_in_sandbox 决定安全边界；passed / total 提供部分得分；隐藏测试决定样例之外的真实价值。

讲解 6：规则型可验证奖励

rule reward

规则型任务不一定考数学或代码，而是考结构化约束。例如“输出 JSON，字段必须完整，数值总和必须等于 100，理由不超过 80 字”。这种任务适合训练模型遵守格式、约束和工具接口，但必须小心：格式奖励很容易让模型学会漂亮壳子，而忽略真实任务质量。

任务例子

给定三项预算需求，输出 JSON：{"plan":[...],"total":100}。要求每项有 name 和 percent，百分比为整数，总和等于 100，不允许额外字段。

奖励拆分

JSON 可解析、字段完整、百分比加总、名称来自输入需求，这几项应该拆开记。只看总分，后面很难知道模型到底是在遵守规则，还是在写漂亮外壳。

状态 读什么 写什么 症状
JSON 可解析 文本结构 进入下一步检查 如果失败，后面都不用算
字段完整 键集合 是否有额外字段 模型可能补一堆不该有的说明
百分比正确 各项 percent 总和是否等于 100 数值对不齐，计划就不能执行
名称对齐 allowed_names 是否使用输入需求 模型可能答对格式，却没有对上需求
规则奖励伪代码
def reward_budget_json(completion, allowed_names):
    data = parse_json_only(completion)
    if data is None:
        return {"reward": 0.0, "reason": "invalid_json"}

    reward = 0.0
    reward += 0.2

    plan = data.get("plan")
    if isinstance(plan, list) and set(data.keys()) == {"plan", "total"}:
        reward += 0.2

    percents = [item.get("percent") for item in plan or []]
    names = [item.get("name") for item in plan or []]
    if all(isinstance(x, int) for x in percents) and sum(percents) == 100:
        reward += 0.4

    if set(names).issubset(set(allowed_names)) and len(names) == len(allowed_names):
        reward += 0.2

    return {"reward": reward, "parsed": data}
为什么要这样拆

如果只给 JSON 是否成功一个总分，模型会更愿意优化“像 JSON”而不是真的满足约束。把解析、字段、总和、名称分别记下来，后面才知道是哪一层在掉链子。

讲解 7：奖励尺度服务于训练，不服务于好看

lecture 7

奖励尺度会影响优势估计和梯度大小。0/1 奖励简单、稳定、解释性强，但在难题上会稀疏；连续奖励更密集，但如果塑形项不可靠，模型会优化错误目标。一个实用原则是：主目标权重大，辅助目标权重小；硬性错误直接归零或强惩罚；每个奖励项都要能单独统计。

别为了让训练曲线看起来平滑，就随手加奖励项。训练曲线上升，只说明模型更会拿你的分，不说明它真的更会推理。最后真正有用的是那些可以回放、可以拆分、可以定位的分数。

推荐

主奖励清晰，辅助奖励低权重，异常路径单独记日志。这样你能看见每一层在帮忙，还是在添乱。

谨慎

步骤奖励、风格奖励、长度奖励容易把模型推向形式主义。它们适合局部实验，不适合遮住主目标。

避免

只看总分，不记录原因；奖励项互相抵消；正确性被格式分淹没。到那一步，你其实已经不知道模型学到了什么。

尺度 优点 风险
0/1 简单、稳定、好解释 在难题上过于稀疏
部分得分 能保住一点训练差异 容易被塑形项带偏
连续塑形 更密集 如果设计不好，模型会朝错误目标最优化

任务构造清单：从题目到训练样本

checklist

下面这份清单可以直接拿去检查一个小型可验证奖励实验。它把任务构造拆成八个问题，任何一项没想清楚，训练里都可能变成隐藏故障。

检查点 先问什么 常见失误
目标 模型到底要学会什么能力？ 把“格式正确”当成最终目标
输入 Prompt 是否包含足够信息？ 泄露答案或测试细节
输出 回答格式是否可解析？ 最终答案位置不固定
验证 验证器能处理等价答案吗？ 字符串比较过窄
奖励 主奖励和辅助奖励怎么分？ 格式分太高
难度 当前模型会出现有对有错吗？ 题太易或太难
污染 训练题和评测题隔离了吗？ 直接拿 benchmark 当训练题
回放 能否回放 prompt、completion、原因？ 只剩总分，没法排错
这份清单怎么用

先拿它筛任务，再看 reward。顺序别反了。很多训练事故不是算法错，而是任务设计从一开始就没留下差异。

常见错误：奖励一错，训练就会很认真地错下去

mistakes
问题 症状 修正思路
只做字符串判等 等价答案被判错，模型过度迎合格式 加答案抽取和等价比较
格式分太高 模型学会漂亮格式，但正确率不升 格式只做低权重辅助或硬门槛
没有隐藏测试 代码模型写死公开样例 公开样例只做说明，隐藏测试负责奖励
忽略全对全错题 rollout 没有相对差异 按当前正确率过滤样本
不记录失败原因 reward 低时无法判断是解析、执行还是答案错 保留结构化日志
把评测集拿来训练 benchmark 提升但泛化下降 来源隔离、去重、做模板变体检查
这里的优先级

先改验证器，再看任务，再看奖励，最后才看优化算法。算法会放大问题，但很少是问题的起点。

失败路径：reward 上升时先别急着庆祝

failure path

训练里最麻烦的一种情况是：reward 曲线上去了，准确率却没怎么动。这个时候先别急着改算法，也先别急着换模型。先看验证器是不是被刷了，格式分是不是过高，日志里是不是有大量异常被静默吃掉，训练集和评测集是不是混了。

先查什么 为什么 看到什么算危险
验证器是否被刷 模型可能学会了投机格式 高分答案和真实正确性明显脱节
格式分是否过高 外壳可能盖过主目标 正确率没升，格式越来越规整
日志是否缺失 没有回放线索就很难排错 只有总分，没有失败原因
训练 / 评测是否混了 分数上升可能只是记住了题 benchmark 涨了，换题就掉

最稳的排查顺序：先看验证器，再看任务，再看日志，最后再考虑算法。算法是放大器，不是救火队。

3. 简答：为什么全对题和全错题都不适合作为主要 RL 样本？

答题目标：从 reward 差异、优势估计和训练信号三个角度回答。要点是“没差异就学不到比较关系”。

参考答案 / 评分要点

高分答案至少要提到三点：第一，全对题的 reward 几乎全一样，模型看不到谁更好；第二，全错题也几乎全一样，训练信号会变薄；第三，优势估计需要差异，才能知道该往哪条轨迹上调概率。加分点是补一句：更合适的是“当前模型偶尔能对、但还不稳定”的题。

6. 简答：reward 上升但准确率不升，你会先查什么？

答题目标：写出排查顺序，不要只给结论。优先从“验证器、格式分、日志、评测隔离”这几层查。

参考答案 / 评分要点

参考顺序可以是：先查验证器有没有被刷；再查格式分是不是过高；再查日志里有没有大量解析失败或异常被算成 0；再查训练集和评测集是否混了；最后再看 reward 的分布是不是被单一模式挤平。只写“模型过拟合”不够，必须能落到具体层。

synthesisverifier compare

7. 迁移：数学 verifier 和代码 verifier 最大差异是什么？

答题目标：先说出一个共同点，再说出一个决定性差异。可以从“等价比较”与“运行环境”两个角度切。

参考答案 / 评分要点

共同点是两者都要把模型输出转成可判分对象。决定性差异是：数学 verifier 主要做等价比较和格式抽取，代码 verifier 还要跑受限执行环境、处理超时和隐藏测试。高分答案最好再提一句：代码 verifier 对安全和非确定性更敏感。

9. 设计：为一个 JSON 输出任务写奖励拆分

参考答案 / 评分要点

要点是把 JSON 可解析、字段完整、数值总和、名称约束分开。总分可以有，但每一项最好单独留下日志。这样后面如果 reward 看着高却没学会规则，你能马上知道是哪一层在伪装。
