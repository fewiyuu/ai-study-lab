# 术语边界：把词拆开 {#terms}
<!-- tag: terms -->
<!-- layout: grid -->

## box
### 可验证奖励
能通过规则、测试、环境反馈或判题器自动判断的奖励。重点是可判定，不是“看起来专业”。

## box
### 验证器
把模型回答映射成分数的程序或规则。数学等价比较、代码测试、JSON 检查都可以是验证器。

## box
### 答案抽取器
从长回答里取出最终答案的逻辑。它常常比想象中更重要，因为训练会很快利用格式细节。

## box
### 任务构造
选择、生成、过滤和包装题目的过程。它决定模型练什么，也决定 reward 分布有没有学习价值。

## box
### 奖励塑形
除了最终对错，还给格式、步骤、约束满足度等中间信号。塑形能降稀疏性，也可能把目标带偏。

## box
### 奖励漏洞
模型用不符合真实目标的方式拿到高分，例如输出特定字符串绕过解析器，或写死测试样例。

## box warn
这几组词要分开看：`可验证奖励`描述反馈来源，`验证器`描述怎么判，`任务构造`描述题目怎么来，`奖励塑形`描述怎么把判分写成可训练信号。混在一起，最常见的后果就是把格式稳定当成能力稳定。

## box info
**来源分工**：DeepSeek-R1 说明为什么可验证任务能撑起推理 RL；Open-R1 把路线拆成更细的公开流程；verl 负责系统侧的 rollout、reward 和更新；TRL 提供 GRPO 和 reward 函数的公开实现入口。

# 讲解 1：为什么 R1 类路线偏爱可验证任务 {#lecture1}
<!-- tag: lecture 1 -->

DeepSeek-R1 的公开路线把数学和代码放在前面，因为它们能把反馈做得足够稳定。强化学习需要大量反馈，如果每条回答都要人来打分，成本和一致性都会成为瓶颈。可验证任务把反馈从人工偏好转成自动裁判，模型就可以在大量采样中看见哪些轨迹更容易走到正确答案。

这一点很重要：训练信号一旦能稳定给出，模型就会自己去找更有用的推理路径。它可能开始多试几步，开始回头检查，开始对比候选。这里的价值“有更高的可验证成功率”。

# 讲解 2：奖励函数要拆成几层 {#lecture2}
<!-- tag: lecture 2 -->

一个训练可用的奖励函数通常不止一句 `answer == gold`。模型会输出长文本，可能没有最终答案，可能答案格式不同但语义等价，可能代码超时，可能触发解析异常。奖励函数要把这些情况分开，否则训练日志里会混进大量不可解释的噪声。

最稳妥的写法，是把正确性、格式、异常路径拆成三层：正确性是主奖励，格式只帮抽取，不抢主位，异常必须单独记录。DeepSeek-R1、Open-R1 和 TRL 的公开材料都能找到这种分层思路，只是名字不同。

$$
 R = w_c R_{\text{correct}} + w_f R_{\text{format}} - w_e R_{\text{error}}
 $$
 `R_correct` 只看目标是否真正达成，`R_format` 只负责辅助解析，`R_error` 把解析失败、超时和执行异常单独记出来。权重里，正确性应当最大，格式项通常只做低权重辅助。

路径
应记录什么
常见错误


答案缺失
missing_final_answer
静默给 0，后面看不出是没答还是答错


格式不合规
format_failed
把格式失败和正确性失败混成一类


验证器报错
verifier_error
验证器出问题却被算成模型错误


最终判错
wrong_answer
只剩一个总分，后面无法回放原因

## box warn
一个很实在的判断
如果格式奖励比正确性更显眼，模型很快会学会“看起来像在解题”，但不一定真的更会解题。这里的权重设计，为了让训练信号更接近目标。

# 讲解 3：任务构造决定 reward 分布 {#lecture3}
<!-- tag: lecture 3 -->

同一个 reward 函数，放在不同任务分布上，训练效果会完全不同。太容易的题，大多数 rollout 都得 1；太难的题，大多数 rollout 都得 0。真正有学习价值的，往往是当前模型“偶尔能答对、但还不稳定”的任务，因为同一 prompt 下会出现对错差异，优化器才有东西可学。

所以任务构造不是杂务，它本身就是训练设计的一部分。你要看难度有没有铺开、题型有没有混合、题目会不会泄露答案、验证成本会不会过高。Open-R1 这种复现路线之所以要把数据和训练拆开，就是为了让这些边界能被单独检查。

# 讲解 4：数学型可验证奖励 {#lecture4}
<!-- tag: math reward -->

数学题的验证通常从最终答案开始。问题在于答案经常有等价形式：`1/2`、`0.5`、`50%` 在很多题里语义一致；`x=2` 和 `2` 也可能等价。只做字符串比较会误伤正确答案，也会逼着模型迎合格式，而不是解决问题。

更稳的做法，是先抽取最终答案，再做数学等价比较，最后再看格式是否符合约定。格式项可以帮助解析，但不应该把正确答案压下去。

## box info
任务例子
题目：若 `3x + 5 = 20`，求 `x`。标准答案：`5`。可接受形式：`5`、`x=5`、`5.0`。不可接受形式：只写推理过程但没有最终答案。

```python
def reward_math(prompt, completion, gold):
 final = extract_final_answer(completion)
 if final is None:
 return {"reward": 0.0, "correct": False, "reason": "missing_final_answer"}

 correct = equivalent_math(final, gold)
 format_ok = has_boxed_or_final_marker(completion)

 reward = 1.0 if correct else 0.0
 reward += 0.1 if format_ok else 0.0
 return {
 "reward": reward,
 "correct": correct,
 "format_ok": format_ok,
 "final_answer": final,
 }
```

## box warn
设计注意
先抽取最终答案，再做等价比较；格式奖励只能辅助，不能让错误答案靠外壳拿高分；多答案题要把顺序、集合和误差容忍度写清楚。

# 讲解 5：代码型可验证奖励 {#lecture5}
<!-- tag: code reward -->

代码任务的奖励通常来自测试。模型输出函数实现，验证器把代码放进受限环境运行，公开测试和隐藏测试都通过才给高分。相比数学题，代码奖励更接近真实行为验证，但也更容易碰到安全、超时、依赖和非确定性问题。

## box info
### 任务例子
实现 `dedupe_keep_order(xs)`：删除列表重复元素，并保留首次出现顺序。这里最有价值的“公开样例之外还能稳稳通过隐藏测试”。

## box warn
### 奖励漏洞
如果只给公开样例，模型可能写死样例输出；如果执行环境不隔离，模型可能读文件、联网或调用危险 API；如果没有超时限制，坏代码会拖垮训练。

检查项
为什么重要
日志字段


沙箱执行
防止模型代码访问系统资源或外部网络
`sandbox_status`


超时限制
防止无限循环拖慢 rollout
`timeout`


隐藏测试
降低写死公开样例的收益
`hidden_pass_rate`


部分得分
让奖励不至于只有 0/1
`passed_tests`

代码奖励伪代码复制

```python
def reward_code(completion, tests):
 code = extract_python_code(completion)
 if not code:
 return {"reward": 0.0, "reason": "no_code_block"}

 result = run_in_sandbox(
 code=code,
 tests=tests,
 timeout_seconds=2,
 blocked_modules=["os", "subprocess", "socket"],
 )

 if result.error:
 return {"reward": 0.0, "reason": result.error_type}

 passed = result.passed_tests
 total = result.total_tests
 reward = passed / total
 if passed == total:
 reward += 0.2
 return {"reward": reward, "passed": passed, "total": total}
```

## box soft
读这段代码时，重点看什么
`extract_python_code` 先把可执行对象抽出来；`run_in_sandbox` 决定安全边界；`passed / total` 提供部分得分；隐藏测试决定样例之外的真实价值。

# 讲解 6：规则型可验证奖励 {#lecture6}
<!-- tag: rule reward -->

规则型任务不一定考数学或代码，而是考结构化约束。例如“输出 JSON，字段必须完整，数值总和必须等于 100，理由不超过 80 字”。这种任务适合训练模型遵守格式、约束和工具接口，但必须小心：格式奖励很容易让模型学会漂亮壳子，而忽略真实任务质量。

## box info
### 任务例子
给定三项预算需求，输出 JSON：`{"plan":[...],"total":100}`。要求每项有 `name` 和 `percent`，百分比为整数，总和等于 100，不允许额外字段。

## box warn
### 奖励拆分
JSON 可解析、字段完整、百分比加总、名称来自输入需求，这几项应该拆开记。只看总分，后面很难知道模型到底是在遵守规则，还是在写漂亮外壳。

规则奖励伪代码复制

```python
def reward_budget_json(completion, allowed_names):
 data = parse_json_only(completion)
 if data is None:
 return {"reward": 0.0, "reason": "invalid_json"}

 reward = 0.0
 reward += 0.2

 plan = data.get("plan")
 if isinstance(plan, list) and set(data.keys()) == {"plan", "total"}:
 reward += 0.2

 percents = [item.get("percent") for item in plan or []]
 names = [item.get("name") for item in plan or []]
 if all(isinstance(x, int) for x in percents) and sum(percents) == 100:
 reward += 0.4

 if set(names).issubset(set(allowed_names)) and len(names) == len(allowed_names):
 reward += 0.2

 return {"reward": reward, "parsed": data}
```

## box soft
为什么要这样拆
如果只给 JSON 是否成功一个总分，模型会更愿意优化“像 JSON”而不是真的满足约束。把解析、字段、总和、名称分别记下来，后面才知道是哪一层在掉链子。

# 讲解 7：奖励尺度服务于训练，不服务于好看 {#lecture7}
<!-- tag: lecture 7 -->

奖励尺度会影响优势估计和梯度大小。0/1 奖励简单、稳定、解释性强，但在难题上会稀疏；连续奖励更密集，但如果塑形项不可靠，模型会优化错误目标。一个实用原则是：主目标权重大，辅助目标权重小；硬性错误直接归零或强惩罚；每个奖励项都要能单独统计。

别为了让训练曲线看起来平滑，就随手加奖励项。训练曲线上升，只说明模型更会拿你的分，不说明它真的更会推理。最后真正有用的是那些可以回放、可以拆分、可以定位的分数。

## box soft
### 推荐
主奖励清晰，辅助奖励低权重，异常路径单独记日志。这样你能看见每一层在帮忙，还是在添乱。

## box warn
### 谨慎
步骤奖励、风格奖励、长度奖励容易把模型推向形式主义。它们适合局部实验，不适合遮住主目标。

## box info
### 避免
只看总分，不记录原因；奖励项互相抵消；正确性被格式分淹没。到那一步，你其实已经不知道模型学到了什么。

# 失败路径：reward 上升时先别急着庆祝 {#failure-path}
<!-- tag: failure path -->

训练里最麻烦的一种情况是：reward 曲线上去了，准确率却没怎么动。这个时候先别急着改算法，也先别急着换模型。先看验证器是不是被刷了，格式分是不是过高，日志里是不是有大量异常被静默吃掉，训练集和评测集是不是混了。

先查什么
为什么
看到什么算危险


验证器是否被刷
模型可能学会了投机格式
高分答案和真实正确性明显脱节


格式分是否过高
外壳可能盖过主目标
正确率没升，格式越来越规整


日志是否缺失
没有回放线索就很难排错
只有总分，没有失败原因


训练 / 评测是否混了
分数上升可能只是记住了题
benchmark 涨了，换题就掉

## box info
最稳的排查顺序：先看验证器，再看任务，再看日志，最后再考虑算法。算法是放大器，不是救火队。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 先看什么
先判断题目能不能被稳定验证，再看奖励函数有没有把正确性、格式和异常拆开，最后检查日志里有没有留下回放线索。下面的题会故意把几类常见错误混在一起。

### 检查重点

<!-- practice-group: 组 1｜任务判断与边界 -->
<!-- quick-review: 回头看 #position 、 #terms 和 #lecture1 ，先认任务，再认奖励。 -->

### 1 choice

**Q:** 1. 单选：哪类任务最适合拿来做第一版可验证奖励实验？

- [ ] . 要求模型写一篇“有感染力”的散文
- [ ] . 给定函数签名，实现代码并通过单元测试
- [ ] . 评价一个商业战略是否“高级”
- [ ] . 让模型自由聊天并看起来更礼貌

Answer: . 给定函数签名，实现代码并通过单元测试
Explain: 代码任务有明确输入输出和测试，最容易把奖励做成稳定判分。

### 2 debug-checklist

**Q:** 2. 多选：一个适合留下来继续做 RL 的题，应该有哪些特征？

- [ ] ，reward 能拉开差异
- [ ] ，能自动判分
- [ ] ，训练更快
- [ ] ，和评测集分开

Explain: 应选“有对有错”“验证边界清楚”“来源可隔离”。“直接泄露答案”是典型坏样本，因为它让 reward 看起来很高，但学到的是泄露，不是推理。

### 3 synthesis

**Q:** 3. 简答：为什么全对题和全错题都不适合作为主要 RL 样本？

Explain: 高分答案至少要提到三点：第一，全对题的 reward 几乎全一样，模型看不到谁更好；第二，全错题也几乎全一样，训练信号会变薄；第三，优势估计需要差异，才能知道该往哪条轨迹上调概率。加分点是补一句：更合适的是“当前模型偶尔能对、但还不稳定”的题。

<!-- end-practice-group -->

<!-- practice-group: 组 2｜奖励函数与日志 -->
<!-- quick-review: 回头看 #lecture2 、 #lecture4 、 #lecture5 和 #lecture6 。 -->

### 4 choice

**Q:** 4. 单选：设计数学奖励时，最先应该确认什么？

- [ ] . 格式奖励权重是不是够大
- [ ] . 最终答案能不能稳定抽取
- [ ] . 训练曲线会不会更平滑
- [ ] . 模型是不是写得更长

Answer: . 最终答案能不能稳定抽取
Explain: 先确认最终答案能不能稳定抽取，后面的等价判断和格式项才有意义。

### 5 debug-checklist

**Q:** 5. 多选：一个可用的代码奖励，至少应该有哪几项保护？

- [ ] 沙箱执行
- [ ] 超时限制
- [ ] 隐藏测试
- [ ] 只看公开样例

Explain: 应选“沙箱执行”“超时限制”“隐藏测试”。“只看公开样例”会让模型学会写死样例，是常见漏洞。

### 6 diagnosis

**Q:** 6. 简答：reward 上升但准确率不升，你会先查什么？

Explain: 参考顺序可以是：先查验证器有没有被刷；再查格式分是不是过高；再查日志里有没有大量解析失败或异常被算成 0；再查训练集和评测集是否混了；最后再看 reward 的分布是不是被单一模式挤平。只写“模型过拟合”不够，必须能落到具体层。

### 7 synthesis

**Q:** 7. 迁移：数学 verifier 和代码 verifier 最大差异是什么？

Explain: 共同点是两者都要把模型输出转成可判分对象。决定性差异是：数学 verifier 主要做等价比较和格式抽取，代码 verifier 还要跑受限执行环境、处理超时和隐藏测试。高分答案最好再提一句：代码 verifier 对安全和非确定性更敏感。

<!-- end-practice-group -->

<!-- practice-group: 组 3｜设计与诊断 -->
<!-- quick-review: 回头看 #lecture3 、 #lecture7 和 #checklist 。 -->

### 8 synthesis

**Q:** 8. 设计：写一个最小可验证奖励实验

Explain: 高分答案应该把实验写成闭环：先选一个有对有错的可判分任务，再写清楚抽取和验证规则，接着说明 reward 由哪些子项组成，最后补上过滤条件和日志字段。若能再写出“为什么不用主观题做第一版”，就更完整。

### 9 rule-reward

**Q:** 9. 设计：为一个 JSON 输出任务写奖励拆分

Explain: 要点是把 JSON 可解析、字段完整、数值总和、名称约束分开。总分可以有，但每一项最好单独留下日志。这样后面如果 reward 看着高却没学会规则，你能马上知道是哪一层在伪装。

### 10 diagnosis

**Q:** 10. 诊断：为什么任务构造时要先看 reward 分布，而不是直接进训练？

Explain: reward 分布能告诉你样本里有没有可学的差异。全对和全错都让信号变薄，balanced 样本才更容易拉开优势。训练前看分布，能省掉很多“训练很久但学不到”的时间。

### 11 debug-checklist

**Q:** 11. 多选：下面哪些样本更应该先丢掉？

- [ ] 题目直接泄露答案
- [ ] 题目直接来自目标 benchmark
- [ ] 当前模型有对有错
- [ ] 格式可稳定解析

Explain: 应丢掉“答案泄露”和“benchmark 泄露”。“有对有错”和“格式可解析”恰恰是有价值的样本特征。

### 12 choice

**Q:** 12. 单选：在设计 reward 尺度时，下面哪个优先级更合理？

- [ ] . 格式分最大，正确性次之，异常可以忽略
- [ ] . 正确性最大，格式辅助，异常单独记
- [ ] . 只要曲线平滑，分怎么分都行
- [ ] . 先把长度奖励做高，再看别的

Answer: . 正确性最大，格式辅助，异常单独记
Explain: 正确性是主目标，格式只做辅助，异常要单独记，不能反过来。
