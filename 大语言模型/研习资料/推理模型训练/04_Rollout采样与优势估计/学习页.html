推理训练闭环

Rollout、采样与优势估计

这一页只讲一件事：同一道题生成多条回答以后，怎么把“哪条更好”变成可训练的信号。你会看到 rollout、采样温度、组内 baseline、优势归一化和方差来源如何连成一条链；KL、PPO 细节和分布式实现先不展开，下一页再接。

读完本页应该能做什么

  • 能解释为什么同题多采样会比单条回答更适合做组内比较。
  • 能手算一组 reward 的均值、标准差和每条优势，并说出正负号代表什么。
  • 能根据方差、成功率和有效 rollout 率判断是采样、奖励还是题目本身出了问题。

学习目标：先把比较单位找对

goals

这一页只处理推理 RL 里最靠前的一层：同一道题采样多条回答以后，怎样把“哪条更好”变成训练信号。你会反复看到 rollout、采样温度、组内 baseline、优势和方差这五个词，因为它们共同决定更新方向是否稳定。

这里的 anchor 例子固定用同一题的多个 completion。题目不变，模型不变，变化的是采样参数和 reward 分布。这样最容易看出训练信号到底来自答案本身，还是来自采样和比较方式。

Prompt→ 多条 completion→ reward 向量→ 组内 baseline→ advantage→ policy update

这一页要拿住什么

能说明 rollout 为什么不是“只跑一次就够”，能手算一组 reward 的均值和优势，也能判断一批训练信号抖不抖、浪不浪费预算。

先别追什么

KL 约束、PPO 剪切、分布式 rollout 调度先放一边。那几层会在 05 和 06 继续展开，这一页先把比较单位和更新方向钉住。

你看到的对象 它在这一页里代表什么 最容易看错的地方
单条 completion 一次从 prompt 到完整回答的采样结果 把它当成整个训练闭环
一组 completion 同一题的多个候选，用来做组内比较 组内差异太小就没有优势信号
reward 验证器、规则或 reward model 给出的分数 把 reward 当成真正学到的能力
advantage 某条回答相对参照水平好多少 忘了 baseline，就只剩绝对分数

训练闭环：先采样，再比较，再回写

loop

推理训练里的闭环不是“答对就行”。更准确的顺序是：先让模型对同一题生成多条候选，再把这些候选送进 reward 计算，再把 reward 和同组参照值比较，最后才进入参数更新。采样给你候选集合，baseline 给你参照点，优势给你更新方向。

这一页最关键的阅读顺序也一样：先看采样如何决定候选集合，再看 reward 如何压缩成标量，再看标量如何变成 advantage。只看最终分数，很容易把“模型偶尔采到了正确路径”误读成“训练已经学会了正确路径”。

输入状态 同一道题

读的是任务和当前策略，写入的是同一 prompt 的采样入口。

采样状态 G 条 completion

写出来的是候选集合。采样温度太低时，这里几乎没有差异。

奖励状态 reward 向量

每条回答都被压成一个分数，后面才能比较谁更好。

参照状态 组内均值或 value 预测

baseline 读的是 reward，写出的是“这组回答的普通水平”。

更新状态 advantage 加权的梯度

优势为正的轨迹被提高概率，优势为负的轨迹被压低概率。

步骤 读什么 写什么 坏掉时的症状
采样 prompt、temperature、top-p、长度上限 一组 completion 每条回答都太像，组内没有差异
评分 completion、答案抽取结果、验证器 reward reward 记到了格式，不记正确性
比较 reward 向量 baseline、advantage 全局分数看着涨，组内信号却很弱
更新 advantage、logprob、KL 约束 新的 policy 模型被推到只会刷分的路径上
你现在应该能做什么看到一段训练日志时，先定位它属于采样、评分、比较还是更新，再判断问题从哪一步开始偏。

术语表：同一批回答里，谁是谁

terms

这几个词不能混。rollout 说的是一次完整生成过程，completion 说的是生成出的文本本身，baseline 说的是参照水平，advantage 说的是相对参照的好坏，variance 说的是训练信号抖动的程度。它们不是同义词，只是常常一起出现。

Rollout

从 prompt 采样到完整回答的一次轨迹。LLM 场景里常把一条 completion 当成一个 rollout。

Completion

模型实际生成出来的文本。它可能是答案、步骤、代码，也可能是无效内容。

Baseline

参照水平。最常见的是同题组内均值，也可以是 value 预测或标准化后的参照。

Advantage

reward 减 baseline 之后的相对好坏。正值表示值得提高概率，负值表示要压低。

Variance

同样设置下更新信号有多抖。它高的时候，训练就像在噪声里找方向。

Group size

同一题采样多少条候选。组越大，比较越稳；预算越紧，覆盖面越小。

现象 最该用的词 为什么
同一题生成 8 条回答 group size / rollout batch 它描述的是采样了多少个候选
某条回答得 1 分，组均值是 0.5 advantage = +0.5 它描述的是相对组内参照的好坏
同样参数训练时稳时抖 variance 它描述的是更新信号的波动程度
每题只生成一条回答 single rollout 这时没有组内比较，优势信号很难稳定出现
先记住这个边界这页先把组内比较和优势估计讲透。PPO 的裁剪、KL 的调节、分布式 rollout 的调度都属于下一层，不要提前把它们塞进当前单元。

采样：多采几条，是为了看见差异

sampling

采样参数的作用很直接：它们决定模型会不会探索到不同回答。温度太低，几条 completion 会挤在同一条路径上，组内 reward 很接近；温度太高，候选会更散，但无效样本、截断和离题也会增加。top-p、长度上限和 stop 条件会继续改变这个候选集合。

这一页里最实用的判断不是“更随机更好”或“更保守更稳”，而是：这组采样能不能给我足够的组内差异，同时又不把预算浪费在无效 completion 上。TRl 的 GRPO Trainer 里，`num_generations`、`reward_funcs` 和 reward 归一化本来就是绑在一起看的。

低温：同一题的 4 条 completion 几乎一样，group std 很小，advantage 很难拉开。
每题采样伪代码
for prompt in prompts:
    completions = []
    for _ in range(num_generations):
        completion = model.generate(
            prompt,
            temperature=0.8,
            top_p=0.95,
            max_new_tokens=2048,
        )
        completions.append(completion)
    rewards = [reward_fn(prompt, item) for item in completions]
    # 后面再进入 advantage 计算
参数 它主要改变什么 过头时的风险
temperature 候选的随机性和探索范围 太低看不到差异，太高无效样本变多
top-p 保留的 token 候选集合大小 太小容易收成同一路径，太大容易放大噪声
num_generations 同一题采多少条 completion 组内比较更稳，但同样预算下题目覆盖会下降
max_new_tokens 回答能走多长 太短截断思路，太长拖高成本并引入废话
你现在应该能做什么看到一组 completion 时，能先判断差异是够不够，再判断随机性是不是已经把无效样本放大了。

优势估计：奖励要先减参照

advantage

advantage 解决的是一个很实际的问题：reward 只是绝对分数，绝对分数里混着题目难度、采样噪声和验证器尺度。训练要用的不是“这条回答有几分”，而是“这条回答相对这一组的普通水平好多少”。

最常见的做法是把同一题的一组 reward 减去组内均值，必要时再除以组内标准差。这样可以把不同题目的尺度压得更接近，也能让正负号更稳定地对应“该不该提高概率”。

$$ A_i = R_i - \mu_G \qquad \tilde A_i = \frac{R_i - \mu_G}{\sigma_G + \varepsilon} $$

其中 $R_i$ 是第 $i$ 条回答的 reward，$\mu_G$ 是同组 reward 均值，$\sigma_G$ 是同组 reward 标准差。第一式给你方向，第二式把不同题目的尺度拉近。

输入状态 reward 向量

读的是同一题的四条回答。只看绝对分数还看不出该往哪推。

比较状态 组内均值

baseline 只负责给比较参照，不负责定义最终能力。

中心化状态 advantage 向量

正值表示高于参照，负值表示低于参照。

更新状态 正推 / 负压

训练不会奖励“分数高”本身，而是奖励相对更好的轨迹。

优势计算伪代码
def group_advantage(rewards):
    mean_r = sum(rewards) / len(rewards)
    var_r = sum((r - mean_r) ** 2 for r in rewards) / len(rewards)
    std_r = (var_r ** 0.5)
    adv = [r - mean_r for r in rewards]
    norm = [a / (std_r + 1e-8) for a in adv]
    return mean_r, std_r, adv, norm
样本 reward advantage 解读
A 1 +0.5 高于同组平均，应该上推
B 0 -0.5 低于同组平均，应该下压
C 1 +0.5 同样的正确样本，在别组里优势可能不同
D 0 -0.5 如果整组都错，单个 0 分并不一定有信息
你现在应该能做什么给出一组 reward 时，能手算均值、标准差和每条 advantage，还能说出为什么组内参照比原始分数更适合更新策略。

方差：信号太抖，更新就不稳

variance

方差说的是：同样的训练设置，为什么有时很稳、有时很抖。推理 RL 特别容易高方差，因为 completion 很长，reward 常常稀疏，题目难度又不均匀，采样还带随机性。不是模型突然变笨，而是信号本来就吵。

这一页里最实用的判断是：如果组内差异太小，advantage 会接近没有信息；如果组内差异太大，但大多来自异常样本，更新方向也会被噪声带歪。方差高不高，直接决定这一批 rollout 值不值得继续用。

稀疏奖励

大多数回答都是 0，少数回答才有分。模型很少看到细粒度反馈，只知道最后成败。

题目难度混杂

简单题全对、难题全错，中间题反而更有训练信号。混得太厉害，组内比较就不稳。

采样噪声

temperature、top-p 和长度上限都会改变候选集合。候选集合变了，advantage 也会跟着变。

手段 直觉 代价 适用场景
baseline 只奖励“比参照更好”的部分 baseline 估计错会引入偏差或弱信号 几乎所有策略梯度训练
同题多采样 同一 prompt 内直接比较好坏 生成成本按采样数增加 GRPO / self-consistency 类训练
reward 标准化 让不同题目的尺度更接近 极端小组可能标准差不稳定 题目难度差异大时
过滤无信息题 全对或全错的小组优势接近 0 可能改变数据分布 训练预算紧张时
过程奖励 给中间步骤反馈，不只看最终答案 标注或 verifier 更复杂 能稳定评估步骤的任务
你现在应该能做什么遇到训练曲线抖动时，先看 reward 分布、组内标准差、无效 completion 比例和题目难度，而不是直接调学习率。

状态台账：哪一步读、哪一步写

state

推理训练的麻烦之处在于，很多错误不是“模型坏了”，而是状态在某一步被写错了：采样写出了几乎一样的 completion，验证器把正确答案抽错了，baseline 读到了错误的 reward，或者更新时把低信息样本当成了高信息样本。

这张台账把最常见的读写关系摆平。你以后看日志时，优先顺着这张表找错位，而不是直接去调学习率。

对象 更新前 更新后 读取了什么 写入了什么 错了时的症状
采样器 单个 prompt 一组 completion temperature、top-p、长度限制 候选集合 组内差异太小，优势几乎全靠噪声
reward 函数 completion 原文 reward 向量 答案抽取、测试结果、规则约束 标量分数和失败原因 格式被奖励，正确性没被看见
baseline reward 向量 中心化后的参照 同组 reward 组均值或标准化参照 分数看着好，组内却没有比较价值
update advantage、logprob 新 policy 正负优势、KL 约束 参数变化 模型更会刷分，不一定更会解题 
组内很平：同一题的 reward 几乎一样，baseline 只能吃到噪声，先查采样和题目难度。
你现在应该能做什么看到 reward 和评测不一致时，能先判断是采样、抽取、baseline 还是更新环节出了偏差。

奖励和 baseline 怎么选

reward choices

同样都是可验证任务，不同的奖励设计会把模型引向不同的地方。最稳妥的优先级通常是：先保证最终答案可判，再考虑格式、长度或中间步骤。baseline 也一样，先用同题组内均值做参照，只有在你确实需要更平滑的估计时，才考虑更复杂的 value 预测。

这一节不是让你背“最佳配置”，而是让你在看到训练配置时，先看主奖励是不是正对最终目标，再看辅助项有没有压过主奖励，再看 baseline 是否和这组样本真的属于同一层次。

配置项 适合什么时候用 常见误用
最终答案正确性 只要任务能稳定判分，就应该放在第一位 被长度和格式项挤到后面
格式辅助分 只在不影响正确性的前提下轻量使用 把它写成主目标
长度信号 更多时候只做观察，不做主奖励 奖励更长的废话输出
组内均值 baseline 同一题有多个 completion 时最直接 把不相关样本混在一起比较
value baseline 你已经能稳定估计价值，而且需要更平滑的梯度时 在问题还没判稳时先上复杂 baseline

一个直接判断

如果你看到的 reward 配置里，长度和格式项比正确性更靠前，先别急着调 optimizer。先把奖励顺序改回任务目标，再看训练曲线。很多看起来像“优化器不稳”的问题，其实是奖励排错了位。

样本效率：预算花在哪一层

efficiency

同样一批 token 预算，花法不同，学到的东西会很不一样。多题少采能提高覆盖面，少题多采能提高组内比较质量。前者适合 reward 比较密、题目差异很大的情况；后者适合可验证任务、需要从同题中挖出少量正确轨迹的情况。

这一页的判断标准不是“采样数越大越好”，而是“这组采样产生的有效 rollout 是否足够多”。有效 rollout 指的是非空、未截断、可解析、能被 verifier 真正评分的样本。

$$ \text{cost per positive advantage} = \frac{\text{total tokens}}{\text{positive advantage count}} $$

这不是最终指标，但它能提醒你：如果一轮训练花了很多 token，却几乎没有正优势样本，预算多半烧在无效采样上了。

指标 怎么读 它说明什么
有效 rollout 率 可解析且能被评分的 completion 占比 预算有没有被无效样本吞掉
组内 reward 方差 同一题多条回答的分数差异 这组有没有比较价值
pass@k 多次采样里是否至少有一次成功 模型是不是“偶尔会”，适不适合继续挖
每正优势 token 成本 生成多少 token 才得到一个正优势样本 这一轮训练值不值

关键权衡：探索、稳定和成本不能同时拉满

tradeoff

推理 RL 里的采样设计没有一组永远正确的参数。你把 temperature 调高，探索更多，但无效样本和方差也可能上来；你把组大小调大，组内比较更稳，但同样预算下题目覆盖会下降；你把格式过滤收紧，训练更干净，却可能把少数正确但不规整的路径过滤掉。

所以这一页的判断不是“哪种参数最好”，而是“哪一层的损失最能接受”。如果当前最缺的是比较信号，就先让同题多采样；如果当前最缺的是覆盖面，就别把预算都压在一个组上；如果当前最缺的是干净的 reward，就先修验证器，再想其它参数。

探索 vs 稳定

探索太少，学不到新路径；探索太多，训练信号很吵。先看 reward 分布，再动采样温度。

组大小 vs 覆盖面

组越大，同题比较越可靠；题目越多，任务覆盖越广。两者都吃生成预算。

稀疏奖励 vs 细粒度奖励

最终答案奖励简单可靠，但信号稀疏；过程奖励更细，但更容易引入 verifier 偏差。

误区诊断：这些偏差会把比较信号带歪

pitfalls

这一页最容易犯的错，不是公式写错，而是把错误信号当成了正确改进。reward 高不一定代表能力高，组内平均也不一定代表公平，更多采样也不一定更省。你要先看信号质量，再看信号数量。

奖励看起来高

先查验证器是不是只看了模板、长度或容易绕过的格式，再查训练题有没有泄露。

组内全对或全错

这两种组都很少给出可比较的优势。前者几乎没有学习空间，后者还没到能挖出正确路径的阶段。

温度一高就乱

先把温度拉回到能产生少量差异的区间，再看 top-p 和长度上限，不要先改优化器。

优势极端大

通常说明 reward 标准差太小、异常样本太多，或者 baseline 把本来不该比较的东西也压进去了。

现象 先查什么 通常怎么修
reward 上升，held-out 不涨 验证器漏洞、格式过拟合、训练题污染 加未见题、抽检高 reward 样本、收紧奖励规则
大部分组全零 模型起点太弱、题目过难、答案抽取失败 回到冷启动、降难度、修抽取器
completion 经常截断 max_new_tokens、stop 条件、推理长度分布 提高长度上限或改奖励里对长度的处理
训练曲线抖动很大 组内方差、采样噪声、无效 completion 比例 先查采样和 reward，再看学习率
你现在应该能做什么看到训练异常时，能把问题归因到采样、奖励、baseline、题目分布或评测，而不是笼统说“RL 不稳定”。

11. 诊断：样本效率差怎么回事

参考答案 / 评分要点

通常顺序是：先查样本是否被截断或格式错误，再查 temperature / top-p / max_new_tokens，随后再看题目是否过难、reward 是否太稀疏。有效 rollout 率低时，先降低无效样本通常比先调学习率更有用。

12. 设计：一个最小实验

设计一个小型数学推理 RL 实验的采样与优势估计方案。必须包含：每题采样数、temperature / top-p、reward 形式、baseline、至少 3 个诊断指标。

答案目标：让采样、奖励和日志连在一起，而不是只写一串参数。

参考答案 / 评分要点

好答案应当把采样数和题目难度绑在一起，把 reward 形式和可验证条件绑在一起，把 baseline 和组内均值或标准化绑在一起。诊断指标至少要有有效 rollout 率、组内 reward 方差和 held-out 评测；更好的答案还会补上 all-wrong group rate、positive advantage token cost 和高 reward 样本抽检。
