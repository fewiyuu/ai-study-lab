训练闭环：先采样，再比较，再回写

loop

推理训练里的闭环不是“答对就行”。更准确的顺序是：先让模型对同一题生成多条候选，再把这些候选送进 reward 计算，再把 reward 和同组参照值比较，最后才进入参数更新。采样给你候选集合，baseline 给你参照点，优势给你更新方向。

步骤读什么写什么坏掉时的症状
采样prompt、temperature、top-p、长度上限一组 completion每条回答都太像，组内没有差异
评分completion、答案抽取结果、验证器rewardreward 记到了格式，不记正确性
比较reward 向量baseline、advantage全局分数看着涨，组内信号却很弱
更新advantage、logprob、KL 约束新的 policy模型被推到只会刷分的路径上

术语表：同一批回答里，谁是谁

terms

这几个词不能混。rollout 说的是一次完整生成过程，completion 说的是生成出的文本本身，baseline 说的是参照水平，advantage 说的是相对参照的好坏，variance 说的是训练信号抖动的程度。它们不是同义词，只是常常一起出现。

现象最该用的词为什么
同一题生成 8 条回答group size / rollout batch它描述的是采样了多少个候选
某条回答得 1 分，组均值是 0.5advantage = +0.5它描述的是相对组内参照的好坏
同样参数训练时稳时抖variance它描述的是更新信号的波动程度
每题只生成一条回答single rollout这时没有组内比较，优势信号很难稳定出现

先记住这个边界

这页先把组内比较和优势估计讲透。PPO 的裁剪、KL 的调节、分布式 rollout 的调度都属于下一层，不要提前把它们塞进当前单元。

采样：多采几条，是为了看见差异

sampling

采样参数的作用很直接：它们决定模型会不会探索到不同回答。温度太低，几条 completion 会挤在同一条路径上，组内 reward 很接近；温度太高，候选会更散，但无效样本、截断和离题也会增加。top-p、长度上限和 stop 条件会继续改变这个候选集合。

每题采样伪代码复制

参数它主要改变什么过头时的风险
temperature候选的随机性和探索范围太低看不到差异，太高无效样本变多
top-p保留的 token 候选集合大小太小容易收成同一路径，太大容易放大噪声
num_generations同一题采多少条 completion组内比较更稳，但同样预算下题目覆盖会下降
max_new_tokens回答能走多长太短截断思路，太长拖高成本并引入废话

看到一组 completion 时，能先判断差异是够不够，再判断随机性是不是已经把无效样本放大了。

优势估计：奖励要先减参照

advantage

advantage 解决的是一个很实际的问题：reward 只是绝对分数，绝对分数里混着题目难度、采样噪声和验证器尺度。训练要用的“这条回答相对这一组的普通水平好多少”。

$$ A_i = R_i - \mu_G \qquad \tilde A_i = \frac{R_i - \mu_G}{\sigma_G + \varepsilon} $$

其中 $R_i$ 是第 $i$ 条回答的 reward，$\mu_G$ 是同组 reward 均值，$\sigma_G$ 是同组 reward 标准差。第一式给你方向，第二式把不同题目的尺度拉近。

代码
def group_advantage(rewards):
 mean_r = sum(rewards) / len(rewards)
 var_r = sum((r - mean_r) ** 2 for r in rewards) / len(rewards)
 std_r = (var_r ** 0.5)
 adv = [r - mean_r for r in rewards]
 norm = [a / (std_r + 1e-8) for a in adv]
 return mean_r, std_r, adv, norm

你现在应该能做什么

给出一组 reward 时，能手算均值、标准差和每条 advantage，还能说出为什么组内参照比原始分数更适合更新策略。

方差：信号太抖，更新就不稳

variance

方差说的是：同样的训练设置，为什么有时很稳、有时很抖。推理 RL 特别容易高方差，因为 completion 很长，reward 常常稀疏，题目难度又不均匀，采样还带随机性。信号本来就吵。

稀疏奖励

大多数回答都是 0，少数回答才有分。模型很少看到细粒度反馈，只知道最后成败。

采样噪声

temperature、top-p 和长度上限都会改变候选集合。候选集合变了，advantage 也会跟着变。

你现在应该能做什么

遇到训练曲线抖动时，先看 reward 分布、组内标准差、无效 completion 比例和题目难度，而不是直接调学习率。

状态台账：哪一步读、哪一步写

state

推理训练的麻烦之处在于，很多错误状态在某一步被写错了：采样写出了几乎一样的 completion，验证器把正确答案抽错了，baseline 读到了错误的 reward，或者更新时把低信息样本当成了高信息样本。

对象更新前更新后读取了什么写入了什么错了时的症状
采样器单个 prompt一组 completiontemperature、top-p、长度限制候选集合组内差异太小，优势几乎全靠噪声
reward 函数completion 原文reward 向量答案抽取、测试结果、规则约束标量分数和失败原因格式被奖励，正确性没被看见
baselinereward 向量中心化后的参照同组 reward组均值或标准化参照分数看着好，组内却没有比较价值
updateadvantage、logprob新 policy正负优势、KL 约束参数变化模型更会刷分，不一定更会解题

你现在应该能做什么

看到 reward 和评测不一致时，能先判断是采样、抽取、baseline 还是更新环节出了偏差。

奖励和 baseline 怎么选

reward

同样都是可验证任务，不同的奖励设计会把模型引向不同的地方。最稳妥的优先级通常是：先保证最终答案可判，再考虑格式、长度或中间步骤。baseline 也一样，先用同题组内均值做参照，只有在你确实需要更平滑的估计时，才考虑更复杂的 value 预测。

配置项适合什么时候用常见误用
最终答案正确性只要任务能稳定判分，就应该放在第一位被长度和格式项挤到后面
格式辅助分只在不影响正确性的前提下轻量使用把它写成主目标
长度信号更多时候只做观察，不做主奖励奖励更长的废话输出
组内均值 baseline同一题有多个 completion 时最直接把不相关样本混在一起比较
value baseline你已经能稳定估计价值，而且需要更平滑的梯度时在问题还没判稳时先上复杂 baseline

一个直接判断

如果你看到的 reward 配置里，长度和格式项比正确性更靠前，先别急着调 optimizer。先把奖励顺序改回任务目标，再看训练曲线。很多看起来像“优化器不稳”的问题，其实是奖励排错了位。

样本效率：预算花在哪一层

efficiency

同样一批 token 预算，花法不同，学到的东西会很不一样。多题少采能提高覆盖面，少题多采能提高组内比较质量。前者适合 reward 比较密、题目差异很大的情况；后者适合可验证任务、需要从同题中挖出少量正确轨迹的情况。

$$ \text{cost per positive advantage} = \frac{\text{total tokens}}{\text{positive advantage count}} $$

这不是最终指标，但它能提醒你：如果一轮训练花了很多 token，却几乎没有正优势样本，预算多半烧在无效采样上了。

关键权衡：探索、稳定和成本不能同时拉满

tradeoff

推理 RL 里的采样设计没有一组永远正确的参数。你把 temperature 调高，探索更多，但无效样本和方差也可能上来；你把组大小调大，组内比较更稳，但同样预算下题目覆盖会下降；你把格式过滤收紧，训练更干净，却可能把少数正确但不规整的路径过滤掉。

探索 vs 稳定

探索太少，学不到新路径；探索太多，训练信号很吵。先看 reward 分布，再动采样温度。

稀疏奖励 vs 细粒度奖励

最终答案奖励简单可靠，但信号稀疏；过程奖励更细，但更容易引入 verifier 偏差。

误区诊断：这些偏差会把比较信号带歪

pitfalls

这一页最容易犯的错，把错误信号当成了正确改进。reward 高不一定代表能力高，组内平均也不一定代表公平，更多采样也不一定更省。你要先看信号质量，再看信号数量。

组内全对或全错

这两种组都很少给出可比较的优势。前者几乎没有学习空间，后者还没到能挖出正确路径的阶段。

温度一高就乱

先把温度拉回到能产生少量差异的区间，再看 top-p 和长度上限，不要先改优化器。

优势极端大

通常说明 reward 标准差太小、异常样本太多，或者 baseline 把本来不该比较的东西也压进去了。

现象先查什么通常怎么修
reward 上升，held-out 不涨验证器漏洞、格式过拟合、训练题污染加未见题、抽检高 reward 样本、收紧奖励规则
大部分组全零模型起点太弱、题目过难、答案抽取失败回到冷启动、降难度、修抽取器
completion 经常截断max_new_tokens、stop 条件、推理长度分布提高长度上限或改奖励里对长度的处理
训练曲线抖动很大组内方差、采样噪声、无效 completion 比例先查采样和 reward，再看学习率

diagnostic

怎么答才算会

先分清 rollout、reward、baseline 和 advantage 的边界，再手算一组小样本，最后把异常归因到采样、奖励、题目难度或组大小。能说出下一步该查什么，比只写结论更重要。

检查重点

题型要和能力对齐：判断题看概念边界，手算题看 group mean 和正负号，诊断题看能不能先定位层级，设计题看你有没有把采样、奖励和日志绑在一起。

Q: 2. 多选：温度太低时会发生什么

参考答案 / 评分要点

。更合理的选择是“组内差异变小”和“advantage 更难拉开”。温度低不会必然带来更多无效 completion，反而常常更稳定；只是稳定得过头以后，组内比较会变弱。

3synthesis

Q: 3. 简答：为什么每题只采一条很难做组内比较

参考答案 / 评分要点

。要点：一条回答没有组内对照，就很难知道它到底是“真的好”还是“只是这题刚好容易”；baseline 也会失去组内比较的意义。好的回答要写出“差异不足导致 advantage 不稳”和“采样预算只换来单点分数，没有比较信息”这两层。

temperature。如果组内差异太小，先调采样温度最直接；学习率不是第一步。

参考答案 / 评分要点

11diagnostic

Q: 11. 诊断：样本效率差怎么回事

参考答案 / 评分要点

。通常顺序是：先查样本是否被截断或格式错误，再查 temperature / top-p / max_new_tokens，随后再看题目是否过难、reward 是否太稀疏。有效 rollout 率低时，先降低无效样本通常比先调学习率更有用。

12synthesis

Q: 12. 设计：一个最小实验

参考答案 / 评分要点

。好答案应当把采样数和题目难度绑在一起，把 reward 形式和可验证条件绑在一起，把 baseline 和组内均值或标准化绑在一起。诊断指标至少要有有效 rollout 率、组内 reward 方差和 held-out 评测；更好的答案还会补上 all-wrong group rate、positive advantage token cost 和高 reward 样本抽检。
