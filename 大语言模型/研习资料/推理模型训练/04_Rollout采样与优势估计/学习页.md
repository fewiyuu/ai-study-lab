# 训练闭环：先采样，再比较，再回写 {#loop}
<!-- tag: loop -->

推理训练里的闭环不是“答对就行”。更准确的顺序是：先让模型对同一题生成多条候选，再把这些候选送进 reward 计算，再把 reward 和同组参照值比较，最后才进入参数更新。采样给你候选集合，baseline 给你参照点，优势给你更新方向。

这一页最关键的阅读顺序也一样：先看采样如何决定候选集合，再看 reward 如何压缩成标量，再看标量如何变成 advantage。只看最终分数，很容易把“模型偶尔采到了正确路径”误读成“训练已经学会了正确路径”。

# 术语表：同一批回答里，谁是谁 {#terms}
<!-- tag: terms -->

这几个词不能混。rollout 说的是一次完整生成过程，completion 说的是生成出的文本本身，baseline 说的是参照水平，advantage 说的是相对参照的好坏，variance 说的是训练信号抖动的程度。它们不是同义词，只是常常一起出现。

**Rollout**从 prompt 采样到完整回答的一次轨迹。LLM 场景里常把一条 completion 当成一个 rollout。
**Completion**模型实际生成出来的文本。它可能是答案、步骤、代码，也可能是无效内容。
**Baseline**参照水平。最常见的是同题组内均值，也可以是 value 预测或标准化后的参照。
**Advantage**reward 减 baseline 之后的相对好坏。正值表示值得提高概率，负值表示要压低。
**Variance**同样设置下更新信号有多抖。它高的时候，训练就像在噪声里找方向。
**Group size**同一题采样多少条候选。组越大，比较越稳；预算越紧，覆盖面越小。

# 采样：多采几条，是为了看见差异 {#sampling}
<!-- tag: sampling -->

采样参数的作用很直接：它们决定模型会不会探索到不同回答。温度太低，几条 completion 会挤在同一条路径上，组内 reward 很接近；温度太高，候选会更散，但无效样本、截断和离题也会增加。top-p、长度上限和 stop 条件会继续改变这个候选集合。

这一页里最实用的判断：这组采样能不能给我足够的组内差异，同时又不把预算浪费在无效 completion 上。TRl 的 GRPO Trainer 里，`num_generations`、`reward_funcs` 和 reward 归一化本来就是绑在一起看的。

# 优势估计：奖励要先减参照 {#advantage}
<!-- tag: advantage -->

advantage 解决的是一个很实际的问题：reward 只是绝对分数，绝对分数里混着题目难度、采样噪声和验证器尺度。训练要用的“这条回答相对这一组的普通水平好多少”。

最常见的做法是把同一题的一组 reward 减去组内均值，必要时再除以组内标准差。这样可以把不同题目的尺度压得更接近，也能让正负号更稳定地对应“该不该提高概率”。

$$
 A_i = R_i - \mu_G
 \qquad
 \tilde A_i = \frac{R_i - \mu_G}{\sigma_G + \varepsilon}
 $$
 其中 $R_i$ 是第 $i$ 条回答的 reward，$\mu_G$ 是同组 reward 均值，$\sigma_G$ 是同组 reward 标准差。第一式给你方向，第二式把不同题目的尺度拉近。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

优势计算伪代码复制

```python
def group_advantage(rewards):
 mean_r = sum(rewards) / len(rewards)
 var_r = sum((r - mean_r) ** 2 for r in rewards) / len(rewards)
 std_r = (var_r ** 0.5)
 adv = [r - mean_r for r in rewards]
 norm = [a / (std_r + 1e-8) for a in adv]
 return mean_r, std_r, adv, norm
```

# 方差：信号太抖，更新就不稳 {#variance}
<!-- tag: variance -->

方差说的是：同样的训练设置，为什么有时很稳、有时很抖。推理 RL 特别容易高方差，因为 completion 很长，reward 常常稀疏，题目难度又不均匀，采样还带随机性。信号本来就吵。

这一页里最实用的判断是：如果组内差异太小，advantage 会接近没有信息；如果组内差异太大，但大多来自异常样本，更新方向也会被噪声带歪。方差高不高，直接决定这一批 rollout 值不值得继续用。

稀疏奖励
大多数回答都是 0，少数回答才有分。模型很少看到细粒度反馈，只知道最后成败。


题目难度混杂
简单题全对、难题全错，中间题反而更有训练信号。混得太厉害，组内比较就不稳。


采样噪声
temperature、top-p 和长度上限都会改变候选集合。候选集合变了，advantage 也会跟着变。

# 奖励和 baseline 怎么选 {#reward-choices}
<!-- tag: reward choices -->

同样都是可验证任务，不同的奖励设计会把模型引向不同的地方。最稳妥的优先级通常是：先保证最终答案可判，再考虑格式、长度或中间步骤。baseline 也一样，先用同题组内均值做参照，只有在你确实需要更平滑的估计时，才考虑更复杂的 value 预测。

这一节让你在看到训练配置时，先看主奖励是不是正对最终目标，再看辅助项有没有压过主奖励，再看 baseline 是否和这组样本真的属于同一层次。

只在不影响正确性的前提下轻量使用

# 样本效率：预算花在哪一层 {#efficiency}
<!-- tag: efficiency -->

同样一批 token 预算，花法不同，学到的东西会很不一样。多题少采能提高覆盖面，少题多采能提高组内比较质量。前者适合 reward 比较密、题目差异很大的情况；后者适合可验证任务、需要从同题中挖出少量正确轨迹的情况。

这一页的判断标准“这组采样产生的有效 rollout 是否足够多”。有效 rollout 指的是非空、未截断、可解析、能被 verifier 真正评分的样本。

$$
 \text{cost per positive advantage}
 =
 \frac{\text{total tokens}}{\text{positive advantage count}}
 $$
 这不是最终指标，但它能提醒你：如果一轮训练花了很多 token，却几乎没有正优势样本，预算多半烧在无效采样上了。

指标
怎么读
它说明什么


有效 rollout 率
可解析且能被评分的 completion 占比
预算有没有被无效样本吞掉


组内 reward 方差
同一题多条回答的分数差异
这组有没有比较价值


pass@k
多次采样里是否至少有一次成功
模型是不是“偶尔会”，适不适合继续挖


每正优势 token 成本
生成多少 token 才得到一个正优势样本
这一轮训练值不值

# 关键权衡：探索、稳定和成本不能同时拉满 {#tradeoff}
<!-- tag: tradeoff -->

推理 RL 里的采样设计没有一组永远正确的参数。你把 temperature 调高，探索更多，但无效样本和方差也可能上来；你把组大小调大，组内比较更稳，但同样预算下题目覆盖会下降；你把格式过滤收紧，训练更干净，却可能把少数正确但不规整的路径过滤掉。

所以这一页的判断“哪一层的损失最能接受”。如果当前最缺的是比较信号，就先让同题多采样；如果当前最缺的是覆盖面，就别把预算都压在一个组上；如果当前最缺的是干净的 reward，就先修验证器，再想其它参数。

探索 vs 稳定
探索太少，学不到新路径；探索太多，训练信号很吵。先看 reward 分布，再动采样温度。


组大小 vs 覆盖面
组越大，同题比较越可靠；题目越多，任务覆盖越广。两者都吃生成预算。


稀疏奖励 vs 细粒度奖励
最终答案奖励简单可靠，但信号稀疏；过程奖励更细，但更容易引入 verifier 偏差。

# 误区诊断：这些偏差会把比较信号带歪 {#pitfalls}
<!-- tag: pitfalls -->

这一页最容易犯的错，把错误信号当成了正确改进。reward 高不一定代表能力高，组内平均也不一定代表公平，更多采样也不一定更省。你要先看信号质量，再看信号数量。

## box
### 奖励看起来高
先查验证器是不是只看了模板、长度或容易绕过的格式，再查训练题有没有泄露。

## box warn
### 组内全对或全错
这两种组都很少给出可比较的优势。前者几乎没有学习空间，后者还没到能挖出正确路径的阶段。

## box warn
### 温度一高就乱
先把温度拉回到能产生少量差异的区间，再看 top-p 和长度上限，不要先改优化器。

## box
### 优势极端大
通常说明 reward 标准差太小、异常样本太多，或者 baseline 把本来不该比较的东西也压进去了。

现象
先查什么
通常怎么修


reward 上升，held-out 不涨

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 怎么答才算会
先分清 rollout、reward、baseline 和 advantage 的边界，再手算一组小样本，最后把异常归因到采样、奖励、题目难度或组大小。能说出下一步该查什么，比只写结论更重要。

## box info
### 检查重点
题型要和能力对齐：判断题看概念边界，手算题看 group mean 和正负号，诊断题看能不能先定位层级，设计题看你有没有把采样、奖励和日志绑在一起。

<!-- end-practice-intro -->

<!-- practice-group: 一、术语与采样 -->
<!-- quick-review: review 先看“生成了什么”，再看“怎么比较”。 -->

### 1 choice

**Q:** 1. 单选：rollout 最接近什么

- [ ] 只指一个 token
- [ ] 一次从 prompt 到完整回答的生成轨迹
- [ ] 验证器给出的最终分数

Answer: 一次从 prompt 到完整回答的生成轨迹
Explain: rollout 指的是从 prompt 到完整回答的一次生成轨迹，completion 是它的具体文本结果。

### 2 debug-checklist

**Q:** 2. 多选：温度太低时会发生什么

- [ ] 组内差异变小
- [ ] advantage 更难拉开
- [ ] 无效 completion 一定更多
- [ ] 同题比较更容易看出好坏

Explain: 更合理的选择是“组内差异变小”和“advantage 更难拉开”。温度低不会必然带来更多无效 completion，反而常常更稳定；只是稳定得过头以后，组内比较会变弱。

### 3 synthesis

**Q:** 3. 简答：为什么每题只采一条很难做组内比较

Explain: 要点：一条回答没有组内对照，就很难知道它到底是“真的好”还是“只是这题刚好容易”；baseline 也会失去组内比较的意义。好的回答要写出“差异不足导致 advantage 不稳”和“采样预算只换来单点分数，没有比较信息”这两层。

### 4 choice

**Q:** 4. 单选：最适合先调的采样项

- [ ] temperature
- [ ] 学习率
- [ ] optimizer 类型

Answer: temperature
Explain: 如果组内差异太小，先调采样温度最直接；学习率不是第一步。

<!-- end-practice-group -->

<!-- practice-group: 二、手算与比较 -->
<!-- quick-review: review 均值、标准差和正负优势是三个最基本的检查点。 -->

### 5 calculation

**Q:** 5. 单选：reward=[1,0,1,0] 的均值

- [ ] 0.25
- [ ] 0.5
- [ ] 0.75

Answer: 0.5
Explain: 均值是 0.5；对应的 advantage 序列是 [+0.5, -0.5, +0.5, -0.5]。

### 6 diagnostic

**Q:** 6. 诊断：全错组意味着什么

Explain: 要点：这类组几乎没有组内比较信号，advantage 接近没有信息。优先查题目是否过难、模型起点是否太弱、答案抽取是否把正确答案误判掉。能把“全错组”说成“没有比较价值的小组”就已经抓到重点了。

### 7 diagnostic

**Q:** 7. 单选：128 条 completion 的分配

- [ ] 64 题每题 2 条
- [ ] 16 题每题 8 条

Answer: 16 题每题 8 条
Explain: 如果你更需要从同题里挖出少量正确路径，16 题每题 8 条通常更适合组内比较。

### 8 diagnostic

**Q:** 8. 诊断：reward 上升但评测不涨

Explain: 常见原因包括：验证器有漏洞、模型开始只刷格式、训练题和评测题过近、采样太保守导致看起来 reward 高但没有真正探索。第一步通常先看 reward 分布和高 reward 样本，再去检查 held-out 题是否被污染。

<!-- end-practice-group -->

<!-- practice-group: 三、状态与失败路径 -->
<!-- quick-review: review 先定位层级，再谈修正动作。 -->

### 9 debug-checklist

**Q:** 9. 多选：advantage 极端大可能意味着什么

- [ ] reward 标准差太小
- [ ] 异常样本太多
- [ ] baseline 把不该比较的东西也压进去了
- [ ] KL 一定坏了

Explain: 更合理的是前三项。advantage 极端大常常reward 尺度太窄、异常值太多，或者 baseline 选错了比较对象。KL 可能后面也会受影响，但不是第一层原因。

### 10 choice

**Q:** 10. 单选：先查哪一层

- [ ] optimizer 动量
- [ ] reward 分布、组内方差和无效 completion 比例
- [ ] 学习率余弦退火

Answer: reward 分布、组内方差和无效 completion 比例
Explain: 先看采样和组内差异，很多高方差问题在这里就已经能解释了。

### 11 diagnostic

**Q:** 11. 诊断：样本效率差怎么回事

Explain: 通常顺序是：先查样本是否被截断或格式错误，再查 temperature / top-p / max_new_tokens，随后再看题目是否过难、reward 是否太稀疏。有效 rollout 率低时，先降低无效样本通常比先调学习率更有用。

### 12 synthesis

**Q:** 12. 设计：一个最小实验

Explain: 好答案应当把采样数和题目难度绑在一起，把 reward 形式和可验证条件绑在一起，把 baseline 和组内均值或标准化绑在一起。诊断指标至少要有有效 rollout 率、组内 reward 方差和 held-out 评测；更好的答案还会补上 all-wrong group rate、positive advantage token cost 和高 reward 样本抽检。
