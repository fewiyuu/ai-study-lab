推理训练算法

GRPO / PPO 与 KL 稳定性

这一页把推理 RL 里最容易混在一起的三层关系拆开：PPO 负责怎么迈步，GRPO 负责拿什么当比较单位，KL 负责把更新拉回参考策略。先跟着一个固定的四条回答例子走一遍，再看哪些日志信号说明训练已经开始跑偏。

学习目标：先把三层关系拆开

goals

这一页只处理推理 RL 的三个动作。PPO 决定怎么更新，GRPO 决定拿什么当比较单位，KL 决定更新能离参考模型多远。只要这三层分开，很多“算法问题”其实就会变成约束层、奖励层或比较层失配。

下面贯穿全页的固定例子是一道数学题的四条回答。我们把它写成 [1, 1, 0, 0] 的 reward 向量，这样你可以直接手算组内均值、优势方向和 KL 风险。

层 它盯什么 它改什么 错了会怎样
PPO 当前策略对旧策略的概率比 单次更新的幅度 一步迈太大，训练抖动或崩
GRPO 同一 prompt 下的多条回答 组内比较后的优势方向 比较单位选错，信号会变噪
KL 当前策略和参考策略的距离 语言分布漂移 模型为了刷分把行为推偏

概念骨架：同一批回答怎么进更新

map

推理训练的顺序很固定：先对同一题采样多条回答，再把每条回答打分，再和组内参照值比较，最后才进入参数更新。采样给你候选集合，baseline 给你参照点，优势给你方向。

这一页的阅读顺序也一样。先看一组回答怎么被评分，再看这些分数怎么变成优化方向。只看最终 reward，很容易把“偶然采到正确路径”误读成“模型已经学会了正确路径”。

输入状态 同一道题

读的是任务和当前策略，写入的是同一 prompt 的采样入口。

采样状态 四条 completion

写出来的是候选集合。采样太保守时，这里几乎没有差异。

奖励状态 [1, 1, 0, 0]

每条回答都被压成一个分数，后面才能比较谁更好。

参照状态 组内均值 0.5

baseline 读的是 reward，写出的是这组回答的普通水平。

更新状态 优势加权梯度

优势为正的轨迹被提高概率，优势为负的轨迹被压低概率。

术语边界：谁和谁比较

terms

这些词不能混。policy 是当前要训练的模型，old policy 是采样当时记录下来的版本，reference policy 是负责把行为拉回来的锚点，advantage 是相对参照值的好坏，KL 是当前策略和参考策略的距离。

术语 直觉 你在日志里看什么 常见误解
policy 当前模型 新一轮更新后的 logprob 把它和参考模型当成一回事
old policy 采样时的旧版本 ratio 里的分母 以为它会一直更新
reference policy 参考锚点 KL 的比较对象 以为只要有 clip 就不需要它
advantage 这条回答比组内普通水平好多少 正负号和幅度 把绝对 reward 当成 advantage

这里先别追 value model、reward model 训练、分布式 rollout 和更复杂的 credit assignment。它们会在 06 和 07 再出现；这一页先把更新规则和稳定性信号钉住。

PPO：把单次更新卡住

ppo

PPO 先看当前策略对旧策略的概率比值，再决定这一批样本能把参数推多远。直觉上，它是一个“不要一下迈太大”的约束。它解决的是更新幅度，不解决奖励是否可靠，也不解决训练跑远以后会不会慢慢偏掉。

$$r_t(\theta)=\frac{\pi_\theta(y_t \mid x)}{\pi_{\theta_{\text{old}}}(y_t \mid x)}$$ $$L_{\text{PPO}}(\theta)=\mathbb{E}\left[\min \left(r_t(\theta)A_t,\ \operatorname{clip}(r_t(\theta),1-\epsilon,1+\epsilon)A_t\right)\right]$$

这里 r_t 是新旧策略的概率比，A_t 是 advantage，\epsilon 是裁剪范围。clip 的意思不是“完全不动”，而是“这一批样本别把更新幅度推过头”。

对象 更新前 更新后 读什么 写什么
old policy 采样时的版本 保持不变，直到下一轮采样 样本生成时的 logprob ratio 的分母
current policy 要更新的参数 按 clipped objective 走一小步 reward、advantage、KL 新一轮 token 概率
clip range 固定阈值 不变 ratio 和 advantage 这条样本还能不能继续放大梯度
单步更新伪代码
for x in prompts:
    y = sample(policy, x)
    r = reward_fn(x, y)
    baseline = group_mean(x)
    advantage = (r - baseline) / (group_std(x) + delta)
    ratio = exp(logprob(policy, y) - logprob(old_policy, y))
    ppo_term = min(ratio * advantage, clip(ratio, 1 - eps, 1 + eps) * advantage)
    loss = -ppo_term + beta * kl(policy, ref_policy)
    step(loss)

GRPO：把 baseline 换成组内比较

grpo

GRPO 常见于可验证推理任务。对同一道题先采样多条回答，再把每条回答和同组的平均水平比较。这样做的好处是少依赖单独训练 value model，坏处是更依赖采样质量和 reward 的区分度。

$$\bar r=\frac{1}{G}\sum_{i=1}^{G} r_i,\qquad s_r=\sqrt{\frac{1}{G}\sum_{i=1}^{G}(r_i-\bar r)^2}$$ $$A_i=\frac{r_i-\bar r}{s_r+\delta}$$

这里 G 是同一题采样的回答数，\bar r 是组内均值，s_r 是组内标准差，\delta 是防止除零的小常数。它们共同决定优势是不是有方向、方向稳不稳。

回答 reward 组内均值 normalized advantage 读法
A 1 0.5 +1 该提高概率
B 1 0.5 +1 该提高概率
C 0 0.5 -1 该压低概率
D 0 0.5 -1 该压低概率

这里最容易错的地方是把“同组全对”或“同组全错”当成稳定训练信号。前者会让优势区分变弱，后者会让组内比较几乎失去方向。GRPO 需要的不是更多口号式鼓励，而是有差异的候选回答。

奖励链：多分数怎么合成一条信号

reward

同一道题常常不只有一个分数。答案对不对、格式合不合规、长度是不是过头，都会进入总 reward。这里最容易出问题的是把某一项权重抬得太高，结果模型开始优化那一项，其他目标反而变差。

还是用四条回答的例子。A 和 B 正确，C 和 D 错误。假设我们再加一个格式分和长度分，那么总 reward 不是单个数字，而是几个分量按权重相加后的结果。组内比较仍然看总 reward，但你要知道每个分量在背后推了什么。

$$R_{\text{total}} = w_c R_{\text{correct}} + w_f R_{\text{format}} + w_l R_{\text{length}} - w_k R_{\text{length penalty}}$$

这里 w_c、w_f、w_l、w_k 是权重。R_{\text{correct}} 负责正确性，R_{\text{format}} 负责格式，R_{\text{length}} 负责长度激励，R_{\text{length penalty}} 负责长度惩罚。权重的任务，是让正确性继续是主信号。

分量 读什么 写什么 权重太高时的症状
正确性 答案和 verifier 是否答对 如果太低，模型会先学格式，不先学做题
格式 输出结构、标签、模板 是否可解析 如果太高，模型会把模板写得越来越工整，答案未必更对
长度 token 数、截断情况 长短偏好 如果太高，输出容易膨胀，KL 也容易被带上去
长度惩罚 是否过长 压缩倾向 如果太高，模型会变得过短，推理步骤被压没
奖励合成伪代码
def total_reward(sample):
    correct = score_correctness(sample.answer)
    format_score = score_format(sample.text)
    length_bonus = score_length(sample.text)
    length_penalty = score_length_penalty(sample.text)

    return (
        1.0 * correct
        + 0.2 * format_score
        + 0.1 * length_bonus
        - 0.3 * length_penalty
    )
input correctness=1 or 0

先确认答案本身。这个分量不对，后面的分量只能修饰错误。

input format=0.2 或 0

格式分只负责可解析，不负责正确答案本身。

input length bonus / penalty

长度分量很容易把模型推向过短或过长，必须看权重。

output total reward

总 reward 决定组内谁比谁更高，但来源要能拆开看。

handoff 优势向量

总 reward 再变成优势，才会进入更新。权重错了，优势会稳定地把模型带偏。

这条链的失败后果很直接。格式权重过高时，reward 会看起来更好，评测却不一定涨；正确性权重过低时，模型会先学会规整的外壳，再学会做题；长度惩罚过强时，推理步骤会被压得太短，KL 读数也未必能告诉你为什么。到练习区先做第 10 题，再回来看第 11 题，通常就能判断是哪一项在主导。

KL：把更新拉回参考策略

kl

KL 不是用来宣布模型“好不好”的终极指标，它更像一条参考线。KL 太小，通常说明模型没怎么学动；KL 太大，通常说明模型为了短期 reward 把行为推得太远。真正要找的是“学得动但不跑偏”的区间。

$$J = \mathbb{E}[reward] - \beta \cdot \overline{\mathrm{KL}}(\pi_\theta || \pi_{\text{ref}})$$

这里 \beta 是 KL 强度，\overline{\mathrm{KL}} 是平均偏离量。\beta 越大，参考策略拉得越紧；\beta 太小，模型更容易为了刷分跑偏。

KL 读数 reward 走势 输出形态 先看什么
很低 也低 几乎没变 是不是学不动，或 beta 太强
中等 稳步上升 长度和格式还正常 继续看未见题和样本质量
很高 表面上升 长度膨胀、语气怪、模板化 先查奖励漏洞和学习率

状态台账：每一步读什么、写什么

state

训练里最难追的，不是公式本身，而是状态怎么传下去。PPO 读旧策略，GRPO 读组内 reward，KL 读参考策略。只要这三条 handoff 有一条错了，表面上看起来像“训练不稳定”，实际上往往是状态读写顺序已经乱了。

对象 更新前 更新后 读什么 写什么 错了的症状
current policy 当前参数 下一轮参数 reward、advantage、KL 新一轮 token 概率 更新后更会刷分，但更不会做题
old policy 采样时冻结 下一轮重采样 当前批次的 logprob ratio 的分母 ratio 失真，clip 也失去意义
reference policy 参考锚点 通常不动 当前策略的偏离量 KL 惩罚项 模型慢慢漂走，语言分布变怪
reward batch 未评分 评分后向量 completion 和 verifier 组内均值、方差和 advantage 学到的是格式，不是正确性
KL 强度 beta 固定超参 继续沿用或重新设定 reward / KL / eval 的共同走势 loss 里的拉回力度 太小会跑偏，太大会学不动
单步 handoff 伪代码
for x in prompts:
    completions = sample(current_policy, x, n=4)
    rewards = [score(x, y) for y in completions]
    mean_r = sum(rewards) / len(rewards)
    std_r = std(rewards) + delta
    advantages = [(r - mean_r) / std_r for r in rewards]

    for y, a in zip(completions, advantages):
        ratio = exp(logprob(current_policy, y) - logprob(old_policy, y))
        clipped = clip(ratio, 1 - eps, 1 + eps)
        policy_loss += -min(ratio * a, clipped * a)

    policy_loss += beta * kl(current_policy, ref_policy)
    step(policy_loss)
before current policy, old policy, ref policy

读的是当前参数、采样时冻结的旧参数和参考锚点。

after reward=[1,1,0,0]

写出来的是一组分数。这里已经看不到原始回答的长文本，只剩可比较的数值。

hand-off mean=0.5, std=0.5

baseline 只负责参照，不负责决定正确答案。它把 reward 变成相对方向。

mutate policy loss

优势为正的样本上推，优势为负的样本下压，KL 同时把策略拉回参考边界。

symptom 刷分、漂移或学不动

日志读错时，症状会先出现在 reward、KL、长度和评测里，而不是出现在公式里。

步骤 读什么 写什么 hand-off 错在哪 最先出现的症状
采样 prompt、temperature、top-p、max_new_tokens 四条 completion 采样太保守，组内没有差异 reward 方差很小
评分 completion、verifier、答案抽取 [1, 1, 0, 0] 把格式当正确性 reward 上升但评测不涨
比较 reward 向量 mean=0.5, std=0.5, A=[+1,+1,-1,-1] 全组同分或几乎同分 advantage 接近 0 或噪声大
更新 advantage、ratio、KL 新的 policy ratio 失真或 KL 太弱 模型开始刷分、变长或跑偏

调参链：beta、clip 和采样一起看

tuning

这一页最常见的误判，是把 beta、clip、采样温度拆开调，然后把结果当成不同问题。更稳妥的做法，是拿同一组四条回答来观察：采样温度决定候选是否有差异，clip 决定这一批样本能推多远，beta 决定模型离参考策略多远时就该收手。三者一起看，才能知道变化是来自搜索、更新还是约束。

还是回到 [1,1,0,0] 这组 reward。若采样太保守，同组回答会很像，GRPO 的优势会靠近 0；若 clip 太松，单批样本可以把 ratio 推得很大；若 beta 太小，reward 看上去能涨，KL 和长度却会先飙。调参时要看的不是“哪个超参最强”，而是“这组回答有没有拉开正确差异，同时没有把策略推离可读区间”。

设置 reward KL 长度 第一症状 先动哪一项
beta 偏小，clip 偏松 上升快 很快升高 明显变长 训练集分数好看，评测下滑 先收紧 beta，再查 reward
beta 偏大，clip 偏紧 起不来 长期很低 几乎不变 模型学不动 先放松 beta，再看采样差异
采样太保守 小幅波动 中低 稳定 组内优势接近 0 先提温度或扩大候选差异
三档配置伪代码

    {"name": "conservative", "temperature": 0.4, "clip_eps": 0.1, "beta": 1.0},
    {"name": "balanced",      "temperature": 0.8, "clip_eps": 0.2, "beta": 0.3},
    {"name": "loose",         "temperature": 1.0, "clip_eps": 0.4, "beta": 0.05},
]

    rewards = [1, 1, 0, 0]
    mean_r = 0.5
    std_r = 0.5
    advantages = [(r - mean_r) / std_r for r in rewards]

case 1 reward 起不来

优点是 KL 很稳，缺点是模型几乎不动。这个设置适合验证是否被约束锁死。

case 2 reward 和 KL 同步可读

四条回答里能拉开正负优势，输出长度也还正常，是最适合继续观察的区间。

case 3 reward 很快升，KL 跟着飙

短期看起来进步很快，长周期常常会变成刷分、变长和评测掉头。

这条链路的检查顺序很具体：先看四条回答是不是还能分出高低，再看 ratio 有没有被单批样本拉得太大，最后看 KL 和长度有没有开始偏离可读区间。只要这三步都稳，后面的 reward 提升才值得继续追。

算例：同一组 reward 怎么变成方向

examples

还是用同一道题的四条回答。A 和 B 得 1 分，C 和 D 得 0 分。组内均值是 0.5，标准差是 0.5，所以这组优势会变成 [+1, +1, -1, -1]。这不是在背公式，而是在看“谁该被提高概率，谁该被压低概率”。

回答 reward 组内均值 标准差 advantage 读法
A 1 0.5 0.5 +1 上推
B 1 0.5 0.5 +1 上推
C 0 0.5 0.5 -1 下压
D 0 0.5 0.5 -1 下压
机制 比较对象 主要防什么 防不住什么
PPO clip 当前策略 vs 旧策略 这一批样本的单次更新过大 长期稳定地偏离参考行为
KL penalty 当前策略 vs 参考策略 长期跑偏和语言分布漂移 奖励定义本身有漏洞

这个算例的价值不在数字本身，而在读法。PPO 关心这一批样本会不会把步子迈太大，GRPO 关心同题内谁比谁更好，KL 关心整体是不是已经离参考模型太远。

失稳诊断：看日志，不看感觉

failure

训练不稳时，先别急着加大模型、换优化器或改一堆参数。先看 reward、KL、长度和评测是不是朝同一个方向走。很多时候，表面上的“学会了”只是 reward 在刷，而真正的评测已经掉头。

路径 reward KL 长度 评测 下一步
学不动 低 很低 稳定 低 查 beta、采样和题目难度
刷分 高 偏高 变长 不涨或下滑 查 verifier、格式奖励和样本污染
被拉死 起不来 贴近 0 几乎不变 不动 放松 beta，看看模型是不是被 KL 锁住 
step 1200 | reward=0.62 | KL=0.38 | length=412 | eval=0.58
训练在学，输出还正常，评测也跟着走。
日志样子 更像什么问题 先查哪一层
reward 低，KL 很低，输出几乎没变 学不动或 KL 太强 beta、采样质量、题目难度
reward 高，KL 很高，输出变长 过优化或奖励漏洞 verifier、长度奖励、学习率
reward 高，KL 正常，评测掉头 训练集适配过度 held-out 题、样本污染、格式偏差
把这三种路径和上一节连起来看：采样坏，reward 方差先塌；评分坏，reward 会和评测分叉；更新坏，KL 和长度最先出问题。这样读日志，才能把问题定位到具体层，而不是停在“训练不稳”这个总标签上。

7. 简答：为什么 PPO clip 不能替代 KL？

脚手架：先说 clip 比较旧策略和当前策略，再说 KL 比较当前策略和参考策略，最后说奖励定义错了时两者都救不了。

参考答案 / 评分要点

好答案至少写出三点：clip 约束的是这一批样本的单次更新，不会自动阻止长期漂移；KL 约束的是当前策略离参考策略的距离，能把模型往回拉；如果 reward 或 verifier 本身有漏洞，clip 和 KL 都可能只是在更平稳地学错。
