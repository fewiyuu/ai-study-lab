<!-- tag: goals -->

# 概念骨架：同一批回答怎么进更新 {#map}
<!-- tag: map -->

推理训练的顺序很固定：先对同一题采样多条回答，再把每条回答打分，再和组内参照值比较，最后才进入参数更新。采样给你候选集合，baseline 给你参照点，优势给你方向。

这一页的阅读顺序也一样。先看一组回答怎么被评分，再看这些分数怎么变成优化方向。只看最终 reward，很容易把“偶然采到正确路径”误读成“模型已经学会了正确路径”。

# PPO：把单次更新卡住 {#ppo}
<!-- tag: ppo -->

PPO 先看当前策略对旧策略的概率比值，再决定这一批样本能把参数推多远。直觉上，它是一个“不要一下迈太大”的约束。它解决的是更新幅度，不解决奖励是否可靠，也不解决训练跑远以后会不会慢慢偏掉。

$$r_t(\theta)=\frac{\pi_\theta(y_t \mid x)}{\pi_{\theta_{\text{old}}}(y_t \mid x)}$$
 $$L_{\text{PPO}}(\theta)=\mathbb{E}\left[\min \left(r_t(\theta)A_t,\ \operatorname{clip}(r_t(\theta),1-\epsilon,1+\epsilon)A_t\right)\right]$$
 这里 `r_t` 是新旧策略的概率比，`A_t` 是 advantage，`\epsilon` 是裁剪范围。clip 的意思“这一批样本别把更新幅度推过头”。

单步更新伪代码复制

```python
for x in prompts:
 y = sample(policy, x)
 r = reward_fn(x, y)
 baseline = group_mean(x)
 advantage = (r - baseline) / (group_std(x) + delta)
 ratio = exp(logprob(policy, y) - logprob(old_policy, y))
 ppo_term = min(ratio * advantage, clip(ratio, 1 - eps, 1 + eps) * advantage)
 loss = -ppo_term + beta * kl(policy, ref_policy)
 step(loss)
```

# GRPO：把 baseline 换成组内比较 {#grpo}
<!-- tag: grpo -->

GRPO 常见于可验证推理任务。对同一道题先采样多条回答，再把每条回答和同组的平均水平比较。这样做的好处是少依赖单独训练 value model，坏处是更依赖采样质量和 reward 的区分度。

$$\bar r=\frac{1}{G}\sum_{i=1}^{G} r_i,\qquad s_r=\sqrt{\frac{1}{G}\sum_{i=1}^{G}(r_i-\bar r)^2}$$
 $$A_i=\frac{r_i-\bar r}{s_r+\delta}$$
 这里 `G` 是同一题采样的回答数，`\bar r` 是组内均值，`s_r` 是组内标准差，`\delta` 是防止除零的小常数。它们共同决定优势是不是有方向、方向稳不稳。

**这里最容易错的地方**是把“同组全对”或“同组全错”当成稳定训练信号。前者会让优势区分变弱，后者会让组内比较几乎失去方向。GRPO 需要的有差异的候选回答。

# 奖励链：多分数怎么合成一条信号 {#reward}
<!-- tag: reward -->

同一道题常常不只有一个分数。答案对不对、格式合不合规、长度是不是过头，都会进入总 reward。这里最容易出问题的是把某一项权重抬得太高，结果模型开始优化那一项，其他目标反而变差。

还是用四条回答的例子。A 和 B 正确，C 和 D 错误。假设我们再加一个格式分和长度分，那么总 reward 几个分量按权重相加后的结果。组内比较仍然看总 reward，但你要知道每个分量在背后推了什么。

$$R_{\text{total}} = w_c R_{\text{correct}} + w_f R_{\text{format}} + w_l R_{\text{length}} - w_k R_{\text{length penalty}}$$
 这里 `w_c`、`w_f`、`w_l`、`w_k` 是权重。`R_{\text{correct}}` 负责正确性，`R_{\text{format}}` 负责格式，`R_{\text{length}}` 负责长度激励，`R_{\text{length penalty}}` 负责长度惩罚。权重的任务，是让正确性继续是主信号。

奖励合成伪代码复制

```python
def total_reward(sample):
 correct = score_correctness(sample.answer)
 format_score = score_format(sample.text)
 length_bonus = score_length(sample.text)
 length_penalty = score_length_penalty(sample.text)

 return (
 1.0 * correct
 + 0.2 * format_score
 + 0.1 * length_bonus
 - 0.3 * length_penalty
 )
```

<!-- interactive: ['shape-tracer'] → keep as HTML -->

**这条链的失败后果**很直接。格式权重过高时，reward 会看起来更好，评测却不一定涨；正确性权重过低时，模型会先学会规整的外壳，再学会做题；长度惩罚过强时，推理步骤会被压得太短，KL 读数也未必能告诉你为什么。到练习区先做第 10 题，再回来看第 11 题，通常就能判断是哪一项在主导。

# KL：把更新拉回参考策略 {#kl}
<!-- tag: kl -->

KL 不是用来宣布模型“好不好”的终极指标，它更像一条参考线。KL 太小，通常说明模型没怎么学动；KL 太大，通常说明模型为了短期 reward 把行为推得太远。真正要找的是“学得动但不跑偏”的区间。

$$J = \mathbb{E}[reward] - \beta \cdot \overline{\mathrm{KL}}(\pi_\theta || \pi_{\text{ref}})$$
 这里 `\beta` 是 KL 强度，`\overline{\mathrm{KL}}` 是平均偏离量。`\beta` 越大，参考策略拉得越紧；`\beta` 太小，模型更容易为了刷分跑偏。

单步 handoff 伪代码复制

```python
for x in prompts:
 completions = sample(current_policy, x, n=4)
 rewards = [score(x, y) for y in completions]
 mean_r = sum(rewards) / len(rewards)
 std_r = std(rewards) + delta
 advantages = [(r - mean_r) / std_r for r in rewards]

 for y, a in zip(completions, advantages):
 ratio = exp(logprob(current_policy, y) - logprob(old_policy, y))
 clipped = clip(ratio, 1 - eps, 1 + eps)
 policy_loss += -min(ratio * a, clipped * a)

 policy_loss += beta * kl(current_policy, ref_policy)
 step(policy_loss)
```

# 调参链：beta、clip 和采样一起看 {#tuning}
<!-- tag: tuning -->

这一页最常见的误判，是把 beta、clip、采样温度拆开调，然后把结果当成不同问题。更稳妥的做法，是拿同一组四条回答来观察：采样温度决定候选是否有差异，clip 决定这一批样本能推多远，beta 决定模型离参考策略多远时就该收手。三者一起看，才能知道变化是来自搜索、更新还是约束。

还是回到 `[1,1,0,0]` 这组 reward。若采样太保守，同组回答会很像，GRPO 的优势会靠近 0；若 clip 太松，单批样本可以把 ratio 推得很大；若 beta 太小，reward 看上去能涨，KL 和长度却会先飙。调参时要看的“这组回答有没有拉开正确差异，同时没有把策略推离可读区间”。

```python
 {"name": "balanced", "temperature": 0.8, "clip_eps": 0.2, "beta": 0.3},

 std_r = 0.5

```

### 1 choice

**Q:** 1. 单选：PPO clip 最准确的作用是什么？

- [ ] 完全防止 reward hacking
- [ ] 限制单次更新的幅度
- [ ] 取代 KL 作为长期锚点

Answer: 限制单次更新的幅度
Explain: clip 只限制单次更新幅度，不能替代奖励定义，也不能替代 KL 锚点。

### 2 判断

**Q:** 2. 判断：KL 越小越好。

- [ ] 对
- [ ] 错

Answer: 错
Explain: KL 太小常常说明模型几乎没学动；稳定训练要找的是学得动但不跑偏。

### 7 解释

**Q:** 7. 简答：为什么 PPO clip 不能替代 KL？

Explain: 好答案至少写出三点：clip 约束的是这一批样本的单次更新，不会自动阻止长期漂移；KL 约束的是当前策略离参考策略的距离，能把模型往回拉；如果 reward 或 verifier 本身有漏洞，clip 和 KL 都可能只是在更平稳地学错。

### 8 diagnosis

**Q:** 8. 单选：reward 上升、KL 上升、长度翻倍意味着什么？

- [ ] 模型已经稳定学会了
- [ ] 更像过优化或奖励漏洞
- [ ] 说明 beta 一定太大

Answer: 更像过优化或奖励漏洞
Explain: 这更像过优化或奖励漏洞，而不是稳定进步。

### 9 choice

**Q:** 9. 单选：下面哪组日志更像 KL 过强？

- [ ] reward 升、KL 升、长度变长
- [ ] reward 起不来、KL 长期贴近 0、输出几乎不变
- [ ] reward 和 KL 都稳定上升

Answer: reward 起不来、KL 长期贴近 0、输出几乎不变
Explain: reward 起不来、KL 长期贴近 0、输出几乎不变，通常更像模型被拉得太紧。

### 10 设计

**Q:** 10. 设计：一个小型监控面板。

Explain: 好答案会把 reward、KL、评测、长度或 entropy 放在同一个面板里，并说明每个指标对应的故障层级。例如 reward 和 KL 同涨但评测下滑，说明更可能是奖励漏洞或过优化；如果 KL 很低而 reward 也低，说明模型可能学不动；如果长度异常膨胀，要查格式奖励和样本偏置。

### 11 diagnosis

**Q:** 11. 单选：reward 上升但评测不涨，先查什么？

- [ ] 先把学习率翻倍
- [ ] 先查 verifier、奖励定义和评测是否同源
- [ ] 先把 batch size 减半

Answer: 先查 verifier、奖励定义和评测是否同源
Explain: 先查 verifier、奖励定义和评测是否同源，而不是先动优化器。

# 参考资料 {#sources}
<!-- tag: source -->

- [DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning](https://arxiv.org/abs/2501.12948)：这一页的主线，说明推理训练为什么会围绕采样、奖励、更新和稳定性来组织。
- [DeepSeekMath: Pushing the Limits of Mathematical Reasoning in Open Language Models](https://arxiv.org/abs/2402.03300)：提供 GRPO 和数学推理场景的背景，帮助理解组内比较为什么有用。
- [Proximal Policy Optimization Algorithms](https://arxiv.org/abs/1707.06347)：提供 PPO、ratio、clip 和局部更新约束的原始定义。
- [TRL GRPO Trainer](https://huggingface.co/docs/trl/en/grpo_trainer)：给 reward_funcs、组内比较和训练器参数一个公开的实现入口。
- [verl / HybridFlow](https://github.com/verl-project/verl)：帮助把 PPO、GRPO、KL 和 rollout/update 的关系放到实际训练框架里看。
