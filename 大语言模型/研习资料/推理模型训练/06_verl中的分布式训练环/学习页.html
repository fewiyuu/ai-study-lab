学习目标：先把这一轮闭环画出来

goals

这一页只处理 verl 的系统闭环，不先追更大的并行拓扑。你要先看懂的是：谁负责发起一轮训练，谁负责采样，谁负责 reward，谁负责把旧策略快照固定下来，谁最后把梯度送回去。

下面用一个最小例子做主线。一个 prompt 进入控制器，产生一组 completions，reward manager 给每条回答打分，trainer 再根据 group baseline 形成优势并更新策略。读懂这一轮，后面的分布式配置、异步训练和日志排障才有落点。

目标 你要盯什么 看错会怎样
控制流 prompt 如何被 controller、actor 和 trainer 接住 会把“谁发起、谁返回”说混
状态流 old_log_probs、ref_log_probs、reward_scores、advantages 何时冻结 会把快照当成实时值
时序 同步、异步、reward 慢、snapshot 旧的差别 会把吞吐问题误判成算法问题

这里先别钻：把相邻问题先放一边

boundary

这一页先不处理多机吞吐调参、kernel 级 profiling、checkpoint 恢复、reward model 训练和更长链的 credit assignment。它们都重要，但现在先追，只会把这一轮闭环冲散。

先看什么：控制面、reward 面、更新面和 snapshot 时机。暂时略过：模型并行、通信优化、显存分配和更复杂的恢复策略。它们会在后面的复现、性能或部署内容里重新出现。

概念骨架：同一批样本怎么进更新

map

把这一页当成一条数据流就行。prompt 先被 controller 接住，actor 或 rollout worker 生成 completions，reward manager 给每条回答打分，controller 把 old_log_probs 和 ref_log_probs 固定下来，trainer 再把 advantages 送进更新。

如果你把 reward 当成终点，就容易忽略快照和时序。训练走偏时，先检查分数、快照和更新是否对齐。

输入状态 prompt batch

读的是任务和 batch 配置，写入的是本轮训练的入口。这里出错，后面每一步都会错位。

采样状态 completions

actor 把同一 prompt 变成多条回答。差异太小，组内比较就会变钝。

奖励状态 reward_scores

reward manager 读回答文本和任务规则，写出每条回答的分数。慢在这里，整条环就会等。

快照状态 old / ref logprobs

这里写的是快照，读取时要按冻结值理解。把它当实时值，ratio 和 KL 都会失真。

更新状态 advantages

trainer 读 reward、baseline 和快照，写回参数更新。优势方向错了，模型会往错误的方向稳稳地走。

下一轮状态 new policy

新策略成为下一轮 rollout 的起点。异步越深，这里和采样点的距离越大。

四个角色：谁负责什么，谁不能越界

roles

这一页里最容易混的，是把系统角色和算法角色混成一团。verl 里真正要分开的，是控制器、采样 worker、reward manager、trainer 和 reference policy。它们可以在同一台机器上，也可以拆开部署，但职责不能混。

角色 主要读什么 主要写什么 写错时的症状
single controller prompt、worker 状态、当前 step 调度顺序、batch 归属、下一轮入口 任务会乱序，后面的快照对不上
actor / rollout worker prompt、当前 policy、采样参数 completions、old_log_probs、轨迹文本 采样太保守或快照过旧
reward manager completion、任务规则、verifier reward_scores、分组统计 reward 慢、reward 漏洞、评分和评测分叉
trainer reward_scores、advantages、old_log_probs、ref_log_probs 参数更新、日志、下一轮 policy 更新方向错、KL 飙高或几乎不动
reference policy 当前 completion 和 token 序列 ref_log_probs、KL 约束的锚点 如果不固定，KL 就失去参照

从文档到代码：先认入口，再认职责

code path

如果你准备从 verl 文档跳到仓库代码，先看公开入口，不要先钻到实现细节。`trainer/main_ppo.py`、`trainer/ppo/ray_trainer.py`、`workers/engine_workers.py` 和 `workers/reward_manager` 分别对应训练主入口、PPO/Ray 训练壳、worker 承载层和 reward 计算层。

接着顺着一次更新的顺序往下看：`generate_sequences` 先产出 rollout，`compute_log_prob` 记录采样时的概率，`compute_ref_log_prob` 固定参照，`update_actor` 才把更新落到参数上。这个顺序本身就是排障顺序，能帮你先判断问题出在样本生成、快照冻结，还是 reward 回传。

入口 先看什么 它在链路中的作用 错了会冒出什么症状
trainer/main_ppo.py 训练主循环从哪里启动 把配置、资源和训练步骤串起来 batch 归属和同步点容易对不上
trainer/ppo/ray_trainer.py 调度、采样和更新的衔接 把 PPO 的更新流接到 worker 体系 reward 回来晚，训练仍按旧节拍跑
workers/engine_workers.py worker 怎么承载 rollout 和更新 负责实际执行和资源绑定 算力看起来够，实际 worker 已经抢占
workers/reward_manager reward 从哪进来、在哪写回 把 verifier 或规则分数写回 batch 队列堆积，GPU 空转，reward 延迟扩大
generate_sequences rollout 什么时候真正开始 生成 completions 并冻结采样时状态 old_log_probs 失去参照时点
一轮公开入口顺序
main_ppo -> ray_trainer -> engine_workers -> reward_manager
       -> generate_sequences -> compute_log_prob -> compute_ref_log_prob -> update_actor

一轮训练步：从 prompt 到参数更新

flow

先把一次更新写成伪代码，比只看名词更稳。verl 的训练环由快照、评分和更新按顺序串成，单独看任何一层都容易把时序看歪。

单轮训练伪代码
for prompt in prompts:
    completions = actor.rollout(prompt, group_size=group_size)
    reward_scores = reward_manager.score(prompt, completions)
    old_log_probs = actor.snapshot_log_probs(prompt, completions)
    ref_log_probs = reference_policy.log_probs(prompt, completions)
    advantages = normalize_by_group(reward_scores, prompt.group_id)
    trainer.update(completions, old_log_probs, ref_log_probs, advantages)

controller.commit_step(prompt.step_id)
controller.enqueue_next_batch()
步骤 读什么 写什么 如果顺序错了
rollout prompt 和当前 policy completions 样本太少，组内比较没信号
reward completion、规则、verifier reward_scores reward 和评测很快分叉
snapshot 采样时的 policy old_log_probs / ref_log_probs ratio 和 KL 对应的是两个时间点
update reward、baseline、快照 新的 policy 参数 训练看似在走，实际在漂

状态台账：哪些东西是快照，哪些东西是实时值

state

这一页最值得养成的习惯，是把每个对象分成三类：实时读、写成快照、只在下一步消费。分不清这三类，就会把 old_log_probs、ref_log_probs 和 reward_scores 当成同一种数据看。

对象 更新前是什么 更新后变成什么 读它的人 写它的人 错了会出现什么
prompt batch 待处理任务 一轮 rollout 的入口 controller、actor 上游数据管线 题目和日志对不上
completions 空 同一 prompt 的多条回答 reward manager、trainer actor / rollout worker 组内差异太小，比较变钝
old_log_probs 采样时的概率快照 ratio 分母 trainer actor 在采样时冻结 ratio 失真，更新幅度不可信
ref_log_probs 参考策略的概率快照 KL 锚点 trainer reference policy KL 约束失去参照
reward_scores 空 每条回答的分数 trainer reward manager reward 漏洞或评测分叉
advantages reward_scores + group baseline 正负更新方向 trainer trainer 内部计算 所有样本都朝同一方向，或信号全没了
step snapshot 当前 step 编号 下一轮是否继续的依据 controller controller 异步深了以后，staleness 会悄悄变大

同步与异步：快一点，还是准一点

timing

verl 里最实际的权衡，是把 rollout、reward 和 update 绑得多紧。绑得紧，系统更容易读懂，但会等；绑得松，吞吐会上去，但 snapshot 会开始滞后。这里的关键量，就是快照滞后。

$$\text{staleness} = s_{\text{train}} - s_{\text{snapshot}}$$

这里 strain 是当前训练步，ssnapshot 是采样时冻结的策略步。差值越大，更新看到的 policy 就越旧。

模式 好处 代价 常见症状
同步 时序清楚，排障容易 reward 慢时整轮都要等 GPU 看起来在空转
colocate 组件贴得近，通信短 资源抢占更明显 rollout 和 reward 抢算力
standalone 边界清楚，方便扩展 队列和网络开销更显眼 日志会先在队列层堆起来
one-step-off async 吞吐更高 staleness 变大 reward 还在涨，评测已经开始掉头

两个例子：一组回答和一条慢链路

example

先看同一道题的四条回答。你要盯的是：分数怎么变成组内比较，比较怎么变成优势，优势怎么影响更新。然后再看 reward 慢的时候，系统会先卡在哪。

回答 reward group baseline advantage 这条轨迹的意义
A 1.0 0.5 +0.5 应该被提高概率
B 1.0 +0.5 同组里表现好，更新方向相同
C 0.0 -0.5 应该被压低概率
D 0.0 -0.5 这条轨迹帮你拉开比较边界
$$\bar r_{\text{group}} = \frac{1}{n}\sum_{i=1}^{n} r_i$$ $$A_i = r_i - \bar r_{\text{group}}$$

这里 r_i 是第 i 条回答的 reward，n 是这一组的回答数。均值定了参照点，优势只负责告诉 trainer 该往上推还是往下压。

第二个例子是链路慢。rollout 只要 1 秒，reward 需要 6 秒，update 还要 2 秒，controller 先把等待时间写出来；日志上先出现的是耗时和队列，模型质量结论还没到这一层。很多看起来像算法坏掉的症状，实际是 reward 先堵了。

具体案例：一条 batch 在异步里怎么变形

假设同一个 prompt 要求模型比较三条训练日志。第 1 条回答把“reward 慢”说对了，第 2 条回答把“快照旧”说对了，第 3 条回答把“异步太深”说对了。rollout worker 在 step 1198 采样，reward manager 在 step 1200 才把分数送回，trainer 看到的 old_log_probs 仍然来自 step 1198，reference policy 已经走到 step 1200。表面上看，这些样本还在同一轮里，实际上它们已经跨了两个时点。

这类问题最容易出现在 reward 慢、队列堆积和长链异步同时存在的时候。你会看到 reward 平均值还在上升，KL 却开始抖，输出长度也变长。只看 reward，容易把系统拖慢误读成模型进步；只看 KL，又容易错过 reward 端的堆积。

环节 读什么 写什么 滞后后会发生什么
rollout worker prompt、当前 policy completions、old_log_probs 样本看起来新，参照已经旧了
reward manager completion、规则、verifier reward_scores 分数回来太慢，队列开始堆积
reference policy 当前 completion ref_log_probs KL 仍然在算，但锚点已经换步
trainer reward、baseline、快照 advantages、参数更新 更新看上去照常进行，实际上在追赶旧批次
日志片段
step 1198 | rollout=0.9s | reward_pending=6.2s | queue=8 | snapshot_step=1198
step 1199 | rollout=1.0s | reward_pending=6.0s | queue=9 | snapshot_step=1198
step 1200 | rollout=1.1s | reward=1.3s | update=2.0s | KL=0.06 | eval=0.24

从这段日志里，第一眼不要先猜算法坏了没有，而是先问三件事：reward 是否回得太慢，snapshot 是否滞后，异步是否比你以为的更深。这个顺序比盯总 reward 更有用，因为它把系统问题先分层，再谈调参。

练习时可以把这条链写成四步：输入是什么，哪一步变慢，哪一个快照变旧，最终症状在哪里露出来。你能把这四步说顺，才算真的读懂这一页。

故障诊断：先找是哪一层先坏

failure

训练环出问题时，先看日志里谁先慢、谁先旧、谁先乱，再动超参。层级找对以后，很多问题都能缩到 reward、snapshot、async 或 role mismatch 这四类。

step 1200 | rollout=1.1s | reward=1.4s | update=2.0s | KL=0.38 | eval=0.57
链路还顺，训练和评测在同一方向。
日志样子 更像什么问题 先查哪一层
reward 低，KL 很低，输出几乎不变 学不动或 KL 太强 beta、采样质量、题目难度
reward 高，KL 很高，长度翻倍 过优化或奖励漏洞 verifier、长度奖励、学习率
reward 慢，队列深，GPU 闲 reward manager 是瓶颈 reward worker、队列和并发数
snapshot_step 落后很多 快照旧或异步太深 old_log_probs、ref_log_probs、step 对齐

操作桥接：把日志直接接到动作

operation

这一段只做一件事：把日志信号翻成排障动作。这里按运行系统来读，每个信号后面都要跟一个先查对象。

最小判断顺序是：先看 reward 是否慢，再看队列是否堆积，再看 snapshot_step 是否落后，最后才去改超参。这个顺序之所以稳，是因为它先找系统瓶颈，再看算法表现。对一页教程来说，这比先争论 KL 权重更有用。

信号 先查对象 要确认什么
reward 用时突然变长 reward manager 评分函数、verifier、并发数、队列长度
rollout 很快但训练跟不上 controller 和 trainer 检查旧批次是否仍在占资源，同步点是否过密
KL 很低但评测也低 reward 规则和采样质量 检查信号是否太弱，题目分布是否太窄
KL 很高、长度继续涨 reward 设计 检查奖励漏洞、长度奖励或学习率是否过大
日志到动作的小脚本
if reward_time > rollout_time * 3:
    investigate("reward manager", "verifier", "queue")
elif snapshot_step < train_step - 1:
    investigate("controller", "old_log_probs", "ref_log_probs")
elif kl > kl_threshold and eval_drop:
    investigate("reward rule", "length bonus", "policy update")
else:
    investigate("sampling quality", "group baseline", "next batch alignment")

这段脚本的价值在于，它把“看到什么”转成“先查什么”。如果你把顺序反着来，很容易先陷在学习率、batch size 这些二级旋钮里，却忽略真正卡住系统的那一层。

读到这里，你应该能做两件事：拿一条日志先判断是 reward 慢、快照旧还是异步深；然后立刻说出先查哪个角色、哪个状态和哪个边界。这个能力比背名词更接近真实排障。

演练输入 第一步 第二步 第三步
reward=7.0s, queue=14, GPU idle 查 reward worker 查 verifier 和 batch 并发 看是否需要拆分 reward 任务
snapshot_step 比 train_step 落后 5 查 controller 的同步点 查 old_log_probs 是否晚冻结 确认 KL 约束是否在旧参照上计算
KL 持续上升、length 拉长、eval 下降 查 reward 规则 查长度奖励和门限 回看最近是否放松了更新强度

把这张表当成最后一层校验：先识别信号，再锁定层级，最后决定动作。这个顺序能让你把 06 页读成一张可执行的排障地图，而不是只停在概念对照。

练习

diagnostic

怎么答才算会

先把控制流、状态流和时序分开，再回到日志。能说出谁写入 old_log_probs、reward 慢会卡住哪一层、异步会引入什么代价，比只会背组件名更重要。

检查重点

这一页的题目会混着出：角色边界、状态快照、同步与异步、故障诊断。多数题都能从本页的表和追踪图里直接定位答案。

解释snapshot

8. 简答：stale snapshot 会把哪两个量一起带偏？

脚手架：说清楚旧快照为什么会让 ratio 和 KL 的参照都不准，再说这会让更新看起来稳定，实际在漂。

参考答案 / 评分要点

好答案会指出两类量：old_log_probs 影响 ratio，ref_log_probs 影响 KL。快照越旧，更新越像在拿“过去的策略”比较“现在的策略”，于是更新幅度和约束都不再对应当前状态。结果常见于异步太深或 snapshot 更新滞后。
