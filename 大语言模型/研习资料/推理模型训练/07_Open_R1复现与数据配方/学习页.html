概念骨架：同一批样本怎么进更新

map

把这一页当成一条数据流就行。题目先从题源进入，教师模型生成候选 trace，清洗步骤去掉污染和低质量样本，学生模型或 policy 在过滤后的数据上训练，最后再用独立评测复测。

如果你把 reward 当成终点，就容易忽略快照和时序。训练走偏时，先检查分数、样本来源和去污染是否对齐。

输入状态 题源

读的是题目集合和任务边界，写入的是一个可生成的 prompt 批次。这里错了，后面每一步都会偏。

采样状态 teacher trace

教师模型把题目变成多条候选回答。差异太小，后面就学不到多样路径。

清洗状态 filtered set

验证器、去污染和过滤规则在这里起作用。没过这一步的样本不该进训练集。

训练状态 student / policy

学生模型读过滤后的样本，写回参数更新。这里一旦把训练信号和任务目标混掉，结果会很快漂。

评测状态 score report

独立评测读的是模型和协议，不读训练集。这个区别决定分数能不能拿来做结论。

步骤 读什么 写什么 常见失败症状
题源抽取 问题集合与任务标签 prompt 批次 训练和评测题混在一起
教师生成 题目与采样设置 trace、最终答案 只得到格式好看的短答
清洗去污 候选 trace、benchmark 列表 可训练样本集 近重复题泄漏到评测
学生训练 过滤后的样本 学生参数或 policy 更新 学到格式，没学到推理
独立复测 模型、协议、采样设置 分数和方差 分数能看，结论站不住

术语与误区：先把词放回工作面

terms

这页里最容易混的词有五个：复现、数据配方、蒸馏、去污染和 pass@1。它们分属不同层级，混着说就会把“训练对象”说成“评测对象”。

术语 直觉解释 精确定义 常见误解
复现 按相近条件重做一段实验链路 对齐 base model、数据、训练和评测后，验证一个声明能否重复 把接近 benchmark 分数当成完整复现
数据配方 训练数据的菜谱 题源、模板、生成设置、验证器、过滤、去污染和混合比例的组合 只报样本数，不报筛选逻辑
蒸馏 让学生模仿教师轨迹 用教师生成的 reasoning trace 或答案做监督学习 把蒸馏说成纯 RL
去污染 先把训练集和评测集隔开 检查训练样本是否和 benchmark、近重复题或答案模板重叠 以为清洗只是删坏样本
pass@1 一次作答能答对的概率估计 在指定采样设置下，对每题抽样并估计正确率 忽略采样次数和方差

复现层级：先问复现哪一层

reproduce

“Open-R1 复现 DeepSeek-R1”这句话本身不够精确。R1 路线包含纯 RL 的 R1-Zero、带 cold-start 的 R1、多阶段训练，以及把大模型推理能力蒸馏到小模型。开源复现通常不会一次覆盖所有层。

复现可信度 = 对象清楚度 × 条件对齐度 × 证据完整度

这里的对象，是你到底在复现评测、蒸馏、RL 还是整条路线；条件，是 base model、prompt 模板、采样设置和评测协议；证据，是数据来源、去污染记录和实验日志。

层级 它真正证明什么 容易被误读成什么 还要补看什么
评测复现 同一协议下分数是否稳定 模型本体能力已经完全复刻 采样设置、答案抽取、方差
蒸馏复现 教师轨迹能否迁移到学生 学生已经拥有教师的全部 RL 过程 trace 来源、过滤规则、模板
RL 复现 可验证奖励能否继续优化 policy 只要 reward 高就一定更强 reward 定义、KL 锚点、group baseline
路线复现 公开路径是否能串成闭环 单个脚本跑通就等于路线复刻 数据、训练、复测三者是否一致

数据配方：先看筛选逻辑，再看样本数

data

Open-R1 这类复现最容易出错的地方在于样本说明不够细，样本少只是表层问题。对推理模型来说，能学的不是最终答案本身，而是答案背后的轨迹、过滤理由和去污染结果。

如果一条数据只剩 final answer，没有 trace 和 verifier，后面就很难判断它是可学、可验，还是会把 benchmark 一起污染掉。

字段 为什么要记 缺了会怎样
题源 决定任务边界和难度分布 训练和评测题混在一起
teacher_model 决定教师轨迹从哪里来 学到的能力来源说不清
trace 学生真正要模仿的对象 只能学到最终答案格式
verifier 决定样本是否可验 好看但不可靠的样本混进来
decontamination 决定是否会偷看 benchmark 分数可能只是泄漏
mix_ratio 决定不同任务的权重 简单题把训练目标带歪
最小审计字段

  "id": "math_000123",
  "domain": "math",
  "problem": "题目文本",
  "teacher_model": "deepseek-ai/DeepSeek-R1",
  "generation": {
    "temperature": 0.6,
    "max_new_tokens": 8192,
    "num_generations": 4
  },
  "trace": "教师模型生成的推理过程",

    "contaminated": false
  }

蒸馏路径：把强模型轨迹变成学生能力

distill

蒸馏和 RL 在这里承担不同的训练信号。蒸馏先把教师已经找到的解题轨迹拿来做监督学习，RL 再用可验证奖励继续优化 policy。前者学“像什么”，后者学“哪一个更好”。

蒸馏 / RL 分工伪码 
if has_teacher_trace:
    train_student_with_sft(problem, trace, final_answer)
else:
    completions = policy.sample(problem, k=group_size)
    rewards = verifier.score(problem, completions)
    policy.update_with_group_relative_reward(completions, rewards, kl_anchor)
SFT 目标：最大化 P(trace, final_answer | problem) RL 目标：在 KL 锚点约束下最大化 reward
路线 输入 训练信号 最容易出错的地方
SFT 蒸馏 高质量 trace 数据 交叉熵监督 模板和 EOS 不对齐，推理被截断
GRPO 继续训练 可验证 prompt 集 组内相对优势 + reward + KL reward 过强或被钻空子
混合路线 先蒸馏、后 RL 先学格式，再学探索 把两个阶段的目标混成一个

一个必要边界

如果模型是用 R1 生成数据训练出来的，这个结果说明“R1 能力可以被蒸馏到这个 base model 和规模上”；它还不能说明“我们从零复现了 R1 的 RL 过程”。两者的证据链不同。

评测闭环：分数必须带条件

eval

Open-R1 和 DeepSeek-R1 这类材料最该一起读的是分数背后的采样数、温度、最大长度、答案抽取和去污染。条件一变，分数的解释就会变。

检查项 为什么重要 出错症状
温度和 top-p 决定采样分布 同一模型分数起伏很大
每题采样数 决定 pass@1 的方差 一次采样看起来“忽高忽低”
答案抽取 决定最终判分是否准确 答案对了但解析失败
上下文长度 决定完整推理能否装下 长链思考被截断
去污染 决定分数是否来自泄漏 训练集和 benchmark 互相偷看
重复运行 决定结论是否稳定 一次好看，第二次掉下来
评测对象 代表任务 主要风险 复现时记录什么
数学推理 AIME 2024、MATH-500 题量小、答案抽取、近重复污染 采样数、答案格式、去污染结果
科学问答 GPQA Diamond 多选解析和记忆混在一起 prompt 格式、选项顺序、输出模板
代码生成 LiveCodeBench、Codeforces 类任务 测试覆盖不足、沙箱差异、时间限制 测试用例、执行环境、超时和通过率

复测算例：同一模型为什么会出不同结论

recheck

把同一个 7B 模型放到两种复测设置里，结论可能完全不同。单次采样时，某一轮刚好撞对答案，分数看起来高；把每题采样次数提高后，正确率的估计会更稳，但计算成本也会明显上升。

设置 采样方式 你会看到什么 如果只看分数会误判什么
Run A 每题 1 次采样，温度 0.7 分数波动大，单次结果看起来很戏剧化 把一次碰巧答对当成稳定能力
Run B 每题 32 次采样，温度 0.7 分数更稳，但计算更贵 把更稳定的估计误读成模型突然变强
两种复测配置 
# Run A
temperature: 0.7
num_generations: 1
max_new_tokens: 8192

# Run B
temperature: 0.7
num_generations: 32
max_new_tokens: 8192

这一步的失败 consequence

如果你只报一个总分，不写采样数、答案抽取和去污染，别人没法判断这个分数是模型更强，还是评测设置更稳。复现报告里最该固定的是协议，不是单个数值。

对照这两种设置，你会看到同一个模型在不同统计口径下得到不同答案。Run A 更像瞬时观察，Run B 更像稳定估计。复现报告里应该先定口径，再谈结果；如果两份报告只差在 `num_generations`，就不能把差值直接写成模型提升。

你现在应该能做什么：看一条复测结果时，先问它是单采样还是多采样，再问答案怎么抽取、benchmark 有没有去污染。这个顺序比先看分数更可靠。

例题：读一个复现实验声明

example

假设声明

“我们用 220k 数学推理 traces 训练 7B 模型，在 MATH-500 上接近某公开蒸馏模型。”

要问什么 为什么要问 不问会怎样
复现层级是什么 先分清是蒸馏、RL 还是评测复现 把不同层级混成一件事
trace 从哪来 决定学生学到的到底是什么 不知道能力来源
过滤和去污染怎么做 决定数据是否可验、可学 训练集和 benchmark 互相渗透
评测协议是什么 决定分数能不能比较 一个分数变成另一套分数
结论边界到哪 决定这句话能否写进报告 把局部结果说成全局复现
审计顺序 
1. 先贴层级：评测复现 / 蒸馏复现 / RL 复现 / 路线复现
2. 再查数据：题源、teacher_model、trace、verifier、去污染
3. 再查训练：base model、模板、长度、采样和 reward
4. 再查评测：协议、采样数、答案抽取、方差
5. 最后写结论：只说这一条协议下能说明什么

结论边界

这个声明如果没有把层级、数据来源和评测协议都说清楚，只能说明“在这个条件下接近某公开模型”，不能直接写成“完整复现了 R1”。

动作 公开入口 读什么 写什么 第一处常见失败

GRPO 训练 src/open_r1/grpo.py prompt 集、reward、KL 锚点 policy 更新 reward 被钻空子，KL 约束失去锚点

最短审读顺序 
1. 找 data generation，确认题源和模板
2. 找 decontaminate，确认训练集和 benchmark 是否隔开
3. 找 sft.py，确认学生学到的是 trace 还是只学到答案格式
4. 找 grpo.py，确认 reward 和 KL 是否分开
5. 找 eval 配置，确认采样数、抽取和方差是否固定

第一处常见失败

如果你在 data generation 前没有固定题源和模板，后面会得到看起来整齐、实际不可比的样本。若去污染没把 benchmark 排除，分数会把数据泄漏当成进步。SFT 如果没对齐 chat template 和 EOS，长推理会在最不该断的地方被截断。GRPO 如果没有把 reward 和 KL 锚点分开看，更新很快就会变成 reward hacking。

你现在应该能做什么：看到 Open-R1 的一段实验说明时，先把它标成生成、清洗、蒸馏、RL 还是复测，再去找具体文件或配置。这样你不会把仓库入口当成统一入口。

1. 先问这句话属于 data / distill / rl / eval 哪一层
2. 再找对应来源负责回答什么
3. 再确认来源有没有 snapshot、版本或访问日期
4. 再把仓库动作和评测协议分开
5. 缺任一层，就不要把它写成完整复现结论

跳过这一步会发生什么

你会把论文里的路线、仓库里的实现、评测工具里的协议和项目页里的公开边界拼成一条不存在的证据链。结果看起来很完整，实际每一层的责任都没分清。

你现在应该能做什么：拿到一条 Open-R1 相关 statement 时，先画出“路线、动作、协议、边界”四格，再去读细节。先分层，后细读，这样才能知道自己读到的是证据，还是解释。

最小诊断例：一条样本怎么从生成走到复测

mini-lab

把下面这个小样本当作本页的锚点：题源是一个数学题，教师模型生成了 4 条候选 trace，其中 1 条通过 verifier，另外 3 条被过滤掉。学生训练时只吃到通过的那条；之后复测要用独立评测协议重新跑。

步骤 输入 输出 如果这步做错
生成 题目 + prompt 4 条候选 trace 只得到格式整齐的短答，推理链不完整
验证 候选 trace + 规则 1 条通过、3 条失败 错误答案混进训练集
去污染 训练样本 + benchmark 列表 可训练样本集 题目和评测题重叠，后面分数虚高
蒸馏 过滤后的 trace 学生 checkpoint 学生学到答案格式，没学到推理步骤
复测 学生模型 + 评测协议 分数和方差 协议换了，分数不能比较
从样本到结论 
题源
  ↓
教师生成 4 条 trace
  ↓
verifier 只保留 1 条
  ↓
去污染后进入训练集
  ↓
学生蒸馏
  ↓
独立协议复测
  ↓
只在这个协议下写结论

这条链最容易错在哪里

最容易错的是把“通过 verifier 的样本”直接等成“训练集就是对的”，或者把“复测分数提高”直接等成“路线已经完整复现”。前者会漏掉去污染，后者会漏掉训练和评测协议的差异。

你现在应该能做什么：拿一条 Open-R1 样本，自己从生成、验证、去污染、蒸馏到复测画一遍箭头。只要这条链能说清，你就知道为什么一个数字不能单独代表复现成功。

失败模式：把问题定位到哪一层

failure

Open-R1 这类页最有价值的地方，不是知道它“做了什么”，而是知道它“哪里最先坏”。如果你把故障归到错误层，后面会花很多时间修错对象。

下面这张表把本页的核心失败模式收束成可检查的信号。它和前面的数据表、评测表连在一起读，能帮你从现象回到原因。

症状 先怀疑什么 下一步查什么 常见误判
分数提升很快，但复测掉得明显 评测协议或去污染不稳 采样数、答案抽取、benchmark 重复运行 把波动当成稳定提升
学生模型答得像，但推理短而空 蒸馏只学到格式 trace 是否保留推理链，模板是否过强 把格式像当成能力像
训练集很干净，但 benchmark 也跟着涨 去污染没做细 近重复检测、题源重叠、答案模板 把泄漏来的分数当成模型进步
RL 训练 reward 高，实际任务没变好 reward 被钻空子 verifier 定义、KL 锚点、group baseline 把 reward 当成最终目标
长推理经常被截断 上下文或模板设置太短 max_new_tokens、chat template、EOS 以为是模型不会思考
最小排障顺序 
1. 先看这是评测问题、数据问题、蒸馏问题还是 RL 问题
2. 再看样本是否有 trace、verifier 和去污染
3. 再看训练设置是否把模板、长度和 reward 搞乱
4. 再看分数是否因为采样数或抽取方式而波动
5. 最后决定要回到 06、07 还是 08 页

排障规则

不要先猜“模型坏了”。先看协议、样本和去污染。只要这三层任何一层没对齐，很多看起来像模型能力的问题，其实是复现条件的问题。

diagnostic

怎么答才算会

先把复现层级、数据配方和评测条件分开，再回到声明和表格。能说出哪一层在证明什么，比背仓库名更重要。

检查重点

题目会混着出：层级辨析、字段识别、去污染、采样方差和下一步行动。多数题都能从本页的表格和追踪图里直接定位答案。

4. 简答：为什么不能只看 benchmark 分数就下结论？

脚手架：先说采样设置和去污染会改变解释，再说层级不清时分数不能单独证明路线复现。

参考答案 / 评分要点

好答案会写出两层原因：第一，分数依赖采样数、温度、答案抽取和去污染，换条件就会变；第二，接近一个 benchmark 只说明某个协议下接近，不代表你已经复刻了完整训练链路。只要没有说清数据来源、训练路径和评测协议，就不能把局部结果写成完整复现。

8. 简答：如果一条数据只剩 final_answer，没有 trace 和 verifier，会缺什么判断能力？

脚手架：先说它能不能被程序检查，再说学生能不能学到过程，最后说这会如何影响数据筛选。

参考答案 / 评分要点

好答案会指出三件事：第一，只有最终答案时，很难判断样本是否真的可验；第二，学生只能学结果，学不到轨迹；第三，训练集里可能混进看起来正确、实际上无法确认来源的样本。这样一来，数据配方就失去了最关键的筛选能力。
