# 复现层级：先问复现哪一层 {#reproduce}
<!-- tag: reproduce -->

“Open-R1 复现 DeepSeek-R1”这句话本身不够精确。R1 路线包含纯 RL 的 R1-Zero、带 cold-start 的 R1、多阶段训练，以及把大模型推理能力蒸馏到小模型。开源复现通常不会一次覆盖所有层。

复现可信度 = 对象清楚度 × 条件对齐度 × 证据完整度

这里的对象，是你到底在复现评测、蒸馏、RL 还是整条路线；条件，是 base model、prompt 模板、采样设置和评测协议；证据，是数据来源、去污染记录和实验日志。

层级
它真正证明什么
容易被误读成什么
还要补看什么


评测复现
同一协议下分数是否稳定
模型本体能力已经完全复刻
采样设置、答案抽取、方差


蒸馏复现
教师轨迹能否迁移到学生
学生已经拥有教师的全部 RL 过程
trace 来源、过滤规则、模板


RL 复现
可验证奖励能否继续优化 policy
只要 reward 高就一定更强

# 数据配方：先看筛选逻辑，再看样本数 {#data}
<!-- tag: data -->

Open-R1 这类复现最容易出错的地方在于样本说明不够细，样本少只是表层问题。对推理模型来说，能学的答案背后的轨迹、过滤理由和去污染结果。

# 蒸馏路径：把强模型轨迹变成学生能力 {#distill}
<!-- tag: distill -->

蒸馏和 RL 在这里承担不同的训练信号。蒸馏先把教师已经找到的解题轨迹拿来做监督学习，RL 再用可验证奖励继续优化 policy。前者学“像什么”，后者学“哪一个更好”。

蒸馏 / RL 分工伪码
复制

```python
if has_teacher_trace:
 train_student_with_sft(problem, trace, final_answer)
else:
 completions = policy.sample(problem, k=group_size)
 rewards = verifier.score(problem, completions)
 policy.update_with_group_relative_reward(completions, rewards, kl_anchor)
```

SFT 目标：最大化 P(trace, final_answer | problem)
 RL 目标：在 KL 锚点约束下最大化 reward

路线
输入
训练信号
最容易出错的地方


SFT 蒸馏
高质量 trace 数据
交叉熵监督
模板和 EOS 不对齐，推理被截断


GRPO 继续训练
可验证 prompt 集
组内相对优势 + reward + KL
reward 过强或被钻空子


混合路线
先蒸馏、后 RL
先学格式，再学探索
把两个阶段的目标混成一个

## box info
### 一个必要边界
如果模型是用 R1 生成数据训练出来的，这个结果说明“R1 能力可以被蒸馏到这个 base model 和规模上”；它还不能说明“我们从零复现了 R1 的 RL 过程”。两者的证据链不同。

# 复测算例：同一模型为什么会出不同结论 {#recheck}
<!-- tag: recheck -->

把同一个 7B 模型放到两种复测设置里，结论可能完全不同。单次采样时，某一轮刚好撞对答案，分数看起来高；把每题采样次数提高后，正确率的估计会更稳，但计算成本也会明显上升。

设置
采样方式
你会看到什么
如果只看分数会误判什么


Run A
每题 1 次采样，温度 0.7
分数波动大，单次结果看起来很戏剧化
把一次碰巧答对当成稳定能力


Run B
每题 32 次采样，温度 0.7
分数更稳，但计算更贵
把更稳定的估计误读成模型突然变强

两种复测配置
复制

```text
# Run A
temperature: 0.7
num_generations: 1
max_new_tokens: 8192

# Run B
temperature: 0.7
num_generations: 32
max_new_tokens: 8192
```

## box warn
### 这一步的失败 consequence
如果你只报一个总分，不写采样数、答案抽取和去污染，别人没法判断这个分数是模型更强，还是评测设置更稳。复现报告里最该固定的是协议，不是单个数值。

对照这两种设置，你会看到同一个模型在不同统计口径下得到不同答案。Run A 更像瞬时观察，Run B 更像稳定估计。复现报告里应该先定口径，再谈结果；如果两份报告只差在 `num_generations`，就不能把差值直接写成模型提升。

你现在应该能做什么：看一条复测结果时，先问它是单采样还是多采样，再问答案怎么抽取、benchmark 有没有去污染。这个顺序比先看分数更可靠。

### 第一处常见失败
如果你在 data generation 前没有固定题源和模板，后面会得到看起来整齐、实际不可比的样本。若去污染没把 benchmark 排除，分数会把数据泄漏当成进步。SFT 如果没对齐 chat template 和 EOS，长推理会在最不该断的地方被截断。GRPO 如果没有把 reward 和 KL 锚点分开看，更新很快就会变成 reward hacking。

你现在应该能做什么：看到 Open-R1 的一段实验说明时，先把它标成生成、清洗、蒸馏、RL 还是复测，再去找具体文件或配置。这样你不会把仓库入口当成统一入口。

# 最小诊断例：一条样本怎么从生成走到复测 {#mini-lab}
<!-- tag: mini-lab -->

把下面这个小样本当作本页的锚点：题源是一个数学题，教师模型生成了 4 条候选 trace，其中 1 条通过 verifier，另外 3 条被过滤掉。学生训练时只吃到通过的那条；之后复测要用独立评测协议重新跑。

步骤
输入
输出
如果这步做错


生成
题目 + prompt
4 条候选 trace
只得到格式整齐的短答，推理链不完整


验证
候选 trace + 规则
1 条通过、3 条失败
错误答案混进训练集


去污染
训练样本 + benchmark 列表
可训练样本集
题目和评测题重叠，后面分数虚高


蒸馏
过滤后的 trace
学生 checkpoint
学生学到答案格式，没学到推理步骤


复测
学生模型 + 评测协议
分数和方差
协议换了，分数不能比较

从样本到结论
复制

```text
题源
 ↓
教师生成 4 条 trace
 ↓
verifier 只保留 1 条
 ↓
去污染后进入训练集
 ↓
学生蒸馏
 ↓
独立协议复测
 ↓
只在这个协议下写结论
```

## box warn
### 这条链最容易错在哪里
最容易错的是把“通过 verifier 的样本”直接等成“训练集就是对的”，或者把“复测分数提高”直接等成“路线已经完整复现”。前者会漏掉去污染，后者会漏掉训练和评测协议的差异。

你现在应该能做什么：拿一条 Open-R1 样本，自己从生成、验证、去污染、蒸馏到复测画一遍箭头。只要这条链能说清，你就知道为什么一个数字不能单独代表复现成功。

# 失败模式：把问题定位到哪一层 {#failure}
<!-- tag: failure -->

Open-R1 这类页最有价值的地方，知道它“哪里最先坏”。如果你把故障归到错误层，后面会花很多时间修错对象。

下面这张表把本页的核心失败模式收束成可检查的信号。它和前面的数据表、评测表连在一起读，能帮你从现象回到原因。

症状
先怀疑什么
下一步查什么
常见误判


分数提升很快，但复测掉得明显
评测协议或去污染不稳
采样数、答案抽取、benchmark 重复运行
把波动当成稳定提升


学生模型答得像，但推理短而空
蒸馏只学到格式
trace 是否保留推理链，模板是否过强
把格式像当成能力像


训练集很干净，但 benchmark 也跟着涨
去污染没做细
近重复检测、题源重叠、答案模板
把泄漏来的分数当成模型进步


RL 训练 reward 高，实际任务没变好
reward 被钻空子
verifier 定义、KL 锚点、group baseline
把 reward 当成最终目标


长推理经常被截断
上下文或模板设置太短
max_new_tokens、chat template、EOS
以为是模型不会思考

最小排障顺序
复制

```text
1. 先看这是评测问题、数据问题、蒸馏问题还是 RL 问题
2. 再看样本是否有 trace、verifier 和去污染
3. 再看训练设置是否把模板、长度和 reward 搞乱
4. 再看分数是否因为采样数或抽取方式而波动
5. 最后决定要回到 06、07 还是 08 页
```

## box info
### 排障规则
不要先猜“模型坏了”。先看协议、样本和去污染。只要这三层任何一层没对齐，很多看起来像模型能力的问题，其实是复现条件的问题。

### 4 explain

**Q:** 4. 简答：为什么不能只看 benchmark 分数就下结论？

Explain: 好答案会写出两层原因：第一，分数依赖采样数、温度、答案抽取和去污染，换条件就会变；第二，接近一个 benchmark 只说明某个协议下接近，不代表你已经复刻了完整训练链路。只要没有说清数据来源、训练路径和评测协议，就不能把局部结果写成完整复现。
