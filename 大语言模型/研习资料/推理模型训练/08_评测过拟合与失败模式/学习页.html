链路地图：样本、协议、抽取、报告

map

把这一页当作一条评测链路来读。题目先进入协议，协议决定怎么采样，采样结果再经过答案抽取和判分，最后写成报告。只要其中任一步换了，最后的分数都可能换意义。

关键术语与误区

terms

这页里最容易搅在一起的词不多，但每个词都对应一层失真。先把词放回工作面，再谈原因。

$$ S_{report} = f(M, P, C, F) $$

M 表示底层任务能力，P 表示采样和抽取协议，C 表示训练和测试边界，F 表示输出表面。这个式子是读报告的顺序，不是精确的统计定律。

评测台账：记录什么才可比

evaluation

高分报告最怕只留下一个数字。真正能复查的记录，需要把模型、协议、去污染和抽取方式都写清楚。否则下一次复测时，你只能猜这次变化到底来自哪里。

奖励投机：分数怎么被钻空子

reward-hacking

只要 verifier 和格式规则有空子，模型就可能学会朝空子走。它得到的 reward 会变高，但任务本身并没有更好。这里最重要的是看清楚，模型是在优化题目，还是在优化判分器。

Rcorrect 来自真实答案是否正确，Rformat 来自输出是否符合模板，Rshortcut 是模型借判分规则拿到的额外分数。这个拆分能帮你看出 reward 是在奖励任务，还是在奖励捷径。

格式依赖：同一能力为什么会掉分

format

有些模型答题并没有变弱，只是输出表面一变，答案抽取器就找不到它了。这个问题很容易被误读成能力退化，实际上更多时候是格式、模板和判分规则在起作用。

代码
def extract_answer(text):
 if "boxed" in text:
 return text.split("boxed")[-1]
 if "####" in text:
 return text.split("####")[-1].strip()
 return None # 格式一变，分数就可能掉

如果评测协议里没写清抽取器，别人拿到同一段输出时，可能得到两种完全不同的分数。那时候你以为在比较模型，其实是在比较规则。这个差异会直接影响复测、去污染后的复验，以及你对“格式依赖”的判断。

把这个小案例和前面的 reward-hacking、evaluation 两节连起来看，就能明白为什么评测页必须把抽取器写进记录里。没有抽取器，分数只是一个结果；有了抽取器，分数才有解释空间。

练习时你要能说出两件事：第一，为什么自然语言结论和 boxed answer 不能随便混；第二，为什么“抽取器更严格”不等于“模型更差”。如果这两点说不清，后面的 benchmark 过拟合也很难看准。

过拟合诊断：开发集、测试集和报告集

overfit

过拟合不只发生在训练曲线上。评测页里最常见的是开发集过拟合、报告集过拟合和题源污染。它们的共同点，是看起来都像“越来越会做题”，实际上只是越来越会认题。

$$ O = N_{repeat} + D_{drift} + S_{single} $$

Nrepeat 表示近重复，Ddrift 表示口径漂移，Ssingle 表示只盯单一集合。只要其中一项没处理，分数上升都要先打问号。

工作流：一条异常分数怎么拆回可查对象

workflow

把这页的所有概念放到同一个记录里看，会更清楚。假设你手里有一条数学题的复测记录：上周分数是 0.42，这周是 0.71。先别急着写“模型进步了”，先把记录拆成协议、数据、verifier 和结论四层。

同一模型的两次复测

Run A：temperature=0.7、num_generations=1、answer_extractor=boxed_answer，结果很像“模型会不会碰巧答对”。

Run B：temperature=0.7、num_generations=32、answer_extractor=boxed_answer，结果更像“这个模型在同一协议下的稳定命中率”。

如果这两次分数差很多，先别去猜模型本体变强或变弱，先看采样次数是不是把方差压下来了。单次采样和多次采样不是同一种证据。

手里有什么先看什么它说明什么看错会怎样
一条总分采样数、温度、抽取器这个分数能不能和上一轮比把协议变化当成能力变化
一份日志verifier 命中和错误样式reward 或判分器有没有被钻空子把 reward 投机写成模型变强
一组样本题源和近重复是不是开发集或测试集被认出来了把记题当成推理
一份报告边界说明和复测条件结论是不是写过头了把局部结果说成路线结论
差异点属于哪一层如果不记下来会怎样正确解释
num_generations 从 1 变成 32协议把方差变化写成能力变化这只是把同一题的采样估计做得更稳
answer_extractor 从宽松模板变成严格模板协议把抽取器变化误读成模型退化这是判分口径变了，不是模型参数变了
题源里加入了近重复样本数据把污染来的分数当成泛化这是记忆效应，不是新的推理能力
verifier 从答案匹配改成格式匹配判分器把模板优势写成任务进步这是 reward 规则改变后的假提升

最小审计脚本

复制

def audit_run(run):
 checks = []
 checks.append(("protocol", run["temperature"], run["num_generations"], run["answer_extractor"]))
 checks.append(("data", run["train_overlap"], run["dev_overlap"], run["test_overlap"]))
 checks.append(("verifier", run["verifier_rule"], run["reward_terms"]))
 checks.append(("report", run["variance"], run["boundary_note"]))
 return checks

def compare_runs(run_a, run_b):
 diff = {}
 for key in ["temperature", "num_generations", "answer_extractor", "verifier_rule"]:
 if run_a[key] != run_b[key]:
 diff[key] = (run_a[key], run_b[key])
 return diff

# 只要 diff 里出现协议项，就先别写能力结论。

失败 consequence

如果你只记总分，不记协议、近重复和 verifier 规则，后面就没法复测，也没法解释为什么这次高分和上次不是同一种高分。这个时候“看起来进步了”只是幻觉，真正能复查的只有一条残缺的分数线。

这个工作流就是后面练习题的底稿：第 1、3 题练协议，第 4、5 题练失真类型，第 7、8 题练污染和分层，第 10、12 题练动作顺序。你只要能按这条链走一遍，就已经能把大多数异常分数拆开了。实际读报告时，顺序建议固定成四步：先看协议差异，再看数据边界，再看 verifier / extractor，最后才写结论。

练到这里，最重要的能说清楚它们为什么不是同一种证据。一个数字可以好看，两个数字才能开始比较，但只有协议、数据和判分器都被写下来，比较才真的有意义。

常见错误：最容易把哪一步看错

mistakes

这页最常见的错，先把分数当成结论，再回头找解释。只要顺序反了，后面每一步都会看错。

这一步的关键提醒

如果你只记住一个动作，就记住“先问协议，再问数据，最后才问模型”。这能把很多看起来像能力问题的事，先拦回到评测问题上。

评测报告：把结论写回可复测格式

report

评测页最容易写坏的地方，是报告句子太快走到“结论”，却没留下别人能复测的条件。一个可复测的报告，至少要把对象、协议、分数、边界和下一次怎么复验写在同一段里。

同一模型，两种抽取器

同一个模型，在 boxed_answer 下的分数是 0.71，在严格要求 #### 的抽取器下变成 0.54。这里不要先写“模型退化”，先写“抽取器更严格，口径变了”。

如果这两个数写进报告，读者至少能知道你比较的同一模型在两种判分规则下的表现。这个差别决定了结论能不能外推。

报告句子读者能复查什么如果缺了会怎样
模型、任务、协议都写清楚是否在同一口径下比较分数不可比
写出方差或重复运行结果这个分数稳不稳一次碰巧碰对也会被写成进步
写出去污染和近重复检查分数是不是来自记忆污染来的结果会被误写成泛化
写出结论边界这句话能不能直接外推局部结果会被写成全局规律
检查项要对比什么如果差异很大

结论该怎么写

抽取器

宽松 vs 严格

格式和评分口径在放大差异

先写格式敏感，再写能力

题集

开发 vs 测试

近重复或污染可能在起作用

先写边界，再写泛化

采样

1 次 vs 32 次

方差被压下来了

先写稳定性，再写提升

报告草稿框架

复制

1. 对象：哪一个模型、哪一类题、哪一套协议
2. 数据：是否做了去污染、近重复检查和样本分层
3. 结果：分数、方差、重复运行、答案抽取方式
4. 解释：是能力变化、格式依赖，还是 verifier / extractor 变化
5. 边界：这句话能不能外推到别的数据集或别的任务

交叉校验代码

复制

def report_delta(strict_score, loose_score, dev_score, test_score):
 notes = []
 if abs(strict_score - loose_score) > 0.1:
 notes.append("format_sensitive")
 if dev_score - test_score > 0.1:
 notes.append("possible_overfit_or_contamination")
 return notes

# 先记录 notes，再写结论句。

失败 consequence

没有这个模板，报告会很快滑向“分数高所以更强”的句式。那种写法看起来干脆，实际上把可复测性删掉了。别人拿到你的报告，只能知道你看到了一个数，不能知道这个数是怎么来的。

把这条链和前面的 workflow、trace-lab 连起来，就能形成最后一个判断闭环：先看协议是否变化，再看抽取器是否变化，再看开发集和测试集是否出现偏差，最后才决定这次提升能不能写成能力变化。这个顺序不是形式主义，它直接决定你会不会把评测设置当成模型本体。

练习的落点也很明确：第 1 题和第 3 题帮你固定协议，第 5 题和第 8 题帮你区分格式和污染，第 12 题则要求你把这些信息写成一个能交出去的三步或五步模板。能写出模板，才算真的会做评测诊断。

这就是本页最后的动作链：先拆协议，再查数据，再分失真类型，最后把结论收回到可复测格式。你学会的“怎么给分数加边界”。

练习时可以把第 12 题的答案直接写成这五步模板。只要你能把它写顺，后面不管遇到的是 reward 投机、格式依赖还是开发集过拟合，都能先把报告写回可查格式，再谈结论。
