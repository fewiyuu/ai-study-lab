提示工程基础与提示词结构

从任务说明、上下文、输入材料和输出格式开始，学会写可检查的 prompt。

这一块先把提示工程的基础打稳：模型并不会自动知道你的真实意图，它只能根据当前 prompt 工作。你要学会把任务说清楚，把资料框出来，把结果格式定下来，再用输出反过来检查 prompt 是否需要改。

学完你应该能做什么

  1. 用自己的话解释提示工程解决的核心问题。
  2. 把一段 prompt 拆成 instruction、context、input data、output indicator。
  3. 把模糊请求改写成具体、可执行、可检查的提示词。
  4. 为信息抽取、客服、学习助手等任务设计稳定输出格式。
  5. 判断什么时候用简单结构，什么时候用 CRISPE 或结构化提示词。

提示工程解决什么问题

definition

提示工程关注的是提示词的开发、优化和测试。它不直接改模型参数，也不是寻找一句永远有效的口令。它真正处理的是任务表达问题：怎样把目标、背景、材料和结果要求说清楚，让模型更稳定地完成工作。

大模型只能根据当前输入生成后续内容，不能直接读取你脑子里的真实意图。你写“帮我分析一下”，模型会猜分析角度；你写“从目标用户、核心风险、可执行下一步三个角度分析，每部分不超过 80 字”，模型的自由度就少很多，输出也更容易检查。

原文强调，简单翻译、普通总结这类任务可能不太依赖复杂提示技巧。但当任务涉及推理、幻觉控制、上下文限制、工具调用或固定格式输出时，提示词结构会明显影响结果质量。提示工程在这里的作用，是把模型能力接到具体应用上。

目标要做什么
背景为什么做
材料处理什么
规则按什么边界
格式结果长什么样
测试是否稳定
判断句如果一个结果“不像你想要的”，先检查 prompt 是否清楚表达了任务、材料、边界和格式，再判断是不是模型能力不足。

一个 prompt 的四个基本组成

anatomy

原文把提示词拆成四个组成：指令、上下文、输入数据、输出指示。这个拆法很实用，因为它让你知道一段提示词到底缺了什么。指令告诉模型做什么；上下文告诉模型任务处在什么场景；输入数据是这次要处理的材料；输出指示规定答案形状。

并不是每个任务都必须把四个部分写满。翻译一句话可能只需要指令和输入数据；基于文档问答需要上下文和输入材料；信息抽取如果要给程序解析，就必须写清输出格式。结构的价值不是把 prompt 写长，而是让任务可以被检查、修改和复用。

Instruction

任务动作，例如总结、分类、翻译、提取、检查错误。

Context

背景和限制，例如读者是谁、使用场景是什么、只能依据哪些资料。

Input Data

本次要处理的具体文本、代码、表格、用户问题或资料片段。

Output Indicator

输出格式、字段、长度、语言、排序方式和缺失值处理。

Example

不是必需部分，但复杂格式常靠示例固定输出形状。

Evaluation

提示词不是写完就结束，还要用样例观察输出是否稳定。

你现在应该能做什么看到一段混乱提示词时，把里面的任务、背景、材料和格式要求分别圈出来。

指令：先把任务动作说清楚

instruction

指令是 prompt 的骨架。它回答“模型现在要做什么”。好的指令通常包含动作、对象和判断标准。比如“总结这篇文章”只是动作；“用 5 个要点总结下面文章，保留关键数字，每点不超过 30 字”才是可检查的任务说明。

原文提到，OpenAI 的提示工程最佳实践建议把指令放在开头，并用分隔符隔开指令和材料。这么做不是形式主义。长材料里可能也有命令式句子，如果没有边界，模型可能把材料里的内容误当成新指令。

错误示例 模糊指令
错误示例：不可检查的任务
帮我分析这个产品，写详细点，不要太长。

问题在于“分析”“详细”“不要太长”都没有标准。模型不知道分析维度，也不知道输出长度。

改进示例 明确动作、维度和长度
正确示例：产品分析指令
请从 3 个角度分析下面产品描述：
1. 目标用户
2. 核心卖点
3. 主要风险

每个角度用 2 句话回答。不要添加描述中没有的信息。

产品描述：
"""
{product_description}
"""
判断句如果一条指令里的形容词无法被第三个人验收，例如“高级”“详细”“自然”，就要把它改成数量、结构、读者或标准。

上下文：让模型知道任务环境

context

上下文不是“多给一点背景”这么简单。它的作用是减少歧义。整理同一段会议记录，给 CEO 看、给研发负责人看、给实习生看，重点会完全不同。上下文要说明读者、目标、限制、资料边界和决策场景。

但上下文不是越多越好。无关背景会占用 token，也会把模型带偏。好的上下文应该能改变模型的取舍：哪些信息更重要，哪些信息不能使用，资料不足时怎么处理。

任务弱上下文更稳的上下文
会议纪要整理一下会议读者是研发负责人，突出风险、依赖、负责人和截止日期。
知识库问答回答用户问题只基于给定资料回答；资料未提及时明确说明。
文本改写写自然一点面向技术博客读者，保留事实和术语，删掉宣传性表达。
你现在应该能做什么给一个任务补充必要背景，并删掉不会改变输出判断的背景。

输入材料：把资料边界标清楚

input data

输入材料是模型要加工的对象，可以是一段文章、一条用户反馈、一个表格、一段代码或一组知识库片段。材料一长，就要用清晰边界包起来。常见写法是三引号、XML 标签、Markdown 标题或代码块。

边界尤其适合处理“资料里也有指令”的情况。例如你让模型总结一段恶意文本，文本里写着“忽略上面的规则”，没有边界时模型更容易被干扰。有边界之后，你可以明确告诉模型：边界里的内容只是材料，不是任务指令。

正确示例：资料边界和不足处理
任务：
请只基于  中的资料回答用户问题。

规则：
- 如果资料没有提到答案，请回答“资料中未提及”。
- 不要使用资料外的常识补全。


{retrieved_passages}


用户问题：
{question}

这段提示把任务、规则、资料和问题分开了。以后出现错误时也更容易调试：是检索资料不够，还是规则不清，还是输出格式没有限制。

判断句只要材料可能很长、带噪声、包含命令式句子，就应该用边界把它和任务说明分开。

输出指示：规定结果长什么样

output

输出指示决定结果是否能被阅读、比较和复用。给人读的结果可以是要点、表格、结论加理由；给程序处理的结果要更严格，通常需要 JSON 字段、类型、缺失值规则和“只输出 JSON”的约束。

原文用公司名称和成立时间抽取作为例子：如果只说“提取公司信息”，模型可能输出自然语言；如果给出 JSON 示例，后续程序就可以稳定解析。格式示例不仅告诉模型字段名，也告诉它层级、数据形状和缺失值处理方式。

不稳写法

“提取下面文本里的公司信息，用 JSON 输出。”字段不清，缺失值不清，可能多写解释。

稳定写法

定义字段：name、founded_year、location；缺失填 null；只输出 JSON。

正确示例：可解析的信息抽取
从下面文本中提取公司信息，并以 JSON 数组输出。

字段：
- name：公司名称
- founded_year：成立年份，未知填 null
- location：总部地点，未知填 null

只输出 JSON，不要解释。

文本：
"""
{text}
"""
你现在应该能做什么为信息抽取任务写出字段定义、缺失规则和输出限制。

角色提示：身份要服务任务

role prompting

角色提示是在 prompt 中指定模型扮演的身份，比如面试官、教师、客服、编辑、代码审查员。它会影响模型的语气、知识选择和判断标准。原文举了 Java 面试官的例子：角色不只是“像谁说话”，还规定了互动方式：一次问一个问题，等待候选人回答。

角色提示最常见的问题是空泛。“你是专家”不会自动带来好结果，因为专家仍然需要知道任务、标准和输出形式。更稳的角色写法会包含身份、服务对象、工作边界、流程和禁止事项。

角色示例 技术编辑
正确示例：角色 + 标准 + 输出
你是一名中文技术文章编辑。

任务：
把给定 AI 技术文稿改成自然、准确、适合博客发布的中文。

工作标准：
1. 保留原意，不新增未经原文支持的信息。
2. 删除空泛宣传语。
3. 术语第一次出现时补一句简短解释。
4. 只输出修改后的正文。

原文：
"""
{draft}
"""

提示词框架：防遗漏，不是套模板

framework

原文介绍了两个框架：基础四段式和 CRISPE。四段式关注 instruction、context、input data、output indicator。CRISPE 则把角色能力、背景洞察、任务陈述、表达风格、多方案实验拆开。结构化提示词常见模块还包括 Role、Profile、Skills、Rules、Workflow、Initialization。

框架的价值是防止遗漏关键要素，不是让每个任务都变成超长提示。翻译一句英文不需要写完整角色设定；长期使用的学习助手、客服助手、代码审查助手，才更适合结构化提示词，因为它们需要固定身份、规则、流程和输出模板。

框架适合场景容易错法
四段式大多数一次性任务把材料和指令混在一起
CRISPE需要角色、背景、风格、多方案的复杂任务机械填满每个模块
结构化提示词长期复用助手或工作流规则冗长但任务不清
最小模板：四段式 prompt
任务：
请你{执行动作}。

背景：
{读者 / 使用场景 / 目标 / 限制}

输入材料：
"""
{input}
"""

输出要求：
1. {格式}
2. {长度或字段}
3. {边界规则}

常见误区：坏 prompt 通常坏在哪里

mistakes

第一类误区是模糊。比如“优化一下”“写得高级”“详细分析”。这些词听起来像要求，其实没有验收标准。修法是换成读者、长度、结构、字段、例子和边界。

第二类误区是否定式过多。原文建议少说“不要做什么”，多说“应该怎么做”。例如客服场景里，不只是说“不要问密码”，而是写清楚遇到账号验证时引导用户去官方帮助页面。

第三类误区是格式不可解析。只说“用 JSON”不够，还要字段、类型、缺失值和是否允许解释。第四类是上下文污染：材料、规则、问题混在一起，模型很难知道哪部分是命令。第五类是一次要求太多，研究、写作、校对、排版全塞进一个请求，错误时无法定位。

模糊词

高级、自然、详细、不要太长。换成可检查标准。

负向堆叠

连续写“不要”。改成遇到某情况时执行某动作。

无边界材料

任务、资料、问题混在一起。用分隔符拆开。

完整链路好的 prompt 先说明任务，再给必要背景，用边界放入材料，最后规定输出形状。复杂任务可以加角色、示例和流程；简单任务保持简单。
