01 基础

提示工程基础与提示词结构

这一节只做一件事：把一条模糊请求改成能检查、能复用的 prompt。你会看到，同一个任务只要补上不同层次的信息，模型输出的稳定性就会明显不一样。

学习目标

目标

这一节教你把 prompt 写成一份能交付、能检查、能迭代的说明。先抓住四个要素，再看它们怎么配合，最后把模糊请求改成可执行的版本。

拆出指令、上下文、输入数据和输出指示给每一段内容标出角色能说清楚哪一句在做任务，哪一句在定边界，哪一句在定格式

把模糊要求改成具体要求把“详细一点”翻成可检查的字段输出能被人工或程序验收，而不是只看起来像回答了

判断先用哪种 prompt 起手在 zero-shot、few-shot 和结构化提示之间选起点能解释为什么先试这个，而不是一上来就堆技巧

先抓主线

主线

提示工程先解决一个最朴素的问题：模型不会自己替你补边界。你写得越清楚，模型越容易把任务、材料、限制和输出形式对齐。这个单元先把“怎么写清楚”做稳，后面的单元再谈推理、样例和知识增强。

1. 任务动作模型要做什么

2. 上下文为什么这样做

3. 输入数据拿什么来处理

4. 输出指示结果要长什么样

四要素

结构

这四个词一张读写关系表。你先看模型读了什么，再看你写了什么，最后问一句：这里如果写错，最明显的症状是什么。

instruction用户的任务目标任务动作模型只知道“要说话”，不知道要做事

context场景、约束、身份、规则回答边界输出方向漂，或忽略重要限制

input data待处理材料分析对象模型会把材料和指令混在一起

output indicator你想要的结果形状格式、长度、字段结果能读，但不能直接用

结构怎么稳

机制

提示词写稳，靠的不是长，而是顺序和边界。先把任务放在前面，再把材料围起来，最后把结果格式提前说清楚，模型就更容易把“看什么、做什么、交什么”分开。

这在提醒你检查四个缺口：任务有没有说清、材料有没有围好、输出有没有定住、信息不全时要怎么处理。

更稳一点的结构

这里多了一层 output indicator。它的作用把输出收住，让人工和程序都能检查。

带输出格式的 prompt复制

提取下面文本中的公司名称和成立时间。

以 JSON 格式输出：
[
 { "name": "XXX", "establish_time": "XXX" }
]

文本内容：
"""
{text input here}
"""

把坏 prompt 改到能执行

步骤先看什么改写时写什么如果跳过这一层

  1. 任务动作模型到底该做总结、抽取、分类还是改写把动词写出来，别只写“分析一下”“帮我看看”模型只能猜你想要它说点什么
  2. 材料边界哪些内容算输入，哪些内容不算用分隔符、标签或字段把输入框起来模型会把说明、样例、正文混成一锅
  3. 输出形状结果是列表、表格、JSON 还是段落把字段、长度、顺序和缺失处理写清楚结果可能对题，却不能直接用
  4. 额外约束是否需要角色、语气、风格或拒答规则只保留真正影响结果的约束，删掉空泛修饰模型会接住语气，却抓不住交付物

这条流程怎么用在真实工作里如果你做的是知识库问答，就先把资料范围写死；如果你做的是内容抽取，就先定 JSON 字段；如果你做的是写作助手，就先定读者、语气和长度。不要一开始就加很多技巧，先把最常出错的那一层固定住。

诊断路径

路径

下面这条链把同一个任务从“看起来像 prompt”拉到“可以直接交付”的状态。你可以把它当成一次现场排查：先看原始请求，再看每一层补了什么，最后看哪一个缺口会让结果跑偏。

补齐后的版本

把任务、边界和缺失处理写清楚之后，模型才知道自己要抽什么、保留什么、缺什么时怎么标。

补齐后的 prompt复制

请从会议记录中抽取待办事项，输出为表格。
字段只保留 owner、task、deadline、source。
如果会议里没有 owner 或 deadline，请直接写“缺失”，不要自行补造。
如果同一事项被重复提到，只保留一次，并在 source 中注明原句。

检查点原始版本的状态补齐后的状态最常见的症状

任务动作“整理一下”抽取待办事项只得到一段概述，拿不到可执行动作

输入边界没说只看会议记录明确只从会议记录中抽取模型会把背景说明、聊天内容也混进来

输出形状“输出给我”表格 + 四个字段结果能看，但不能直接贴进文档或系统

缺失处理“不要乱说”缺少时写“缺失”模型会回避问题，或者自己编一个值

重复处理没有规则重复事项只保留一次同一件事会被拆成多个重复条目

为什么这一步重要

如果你没有把“缺失时怎么写”说清楚，模型就会在真实资料不完整时自己补。这个补法在演示里看不出来，在实际工作里最容易出错。很多人以为自己在做提示工程，实际上只是在给模型一个模糊方向。

如果你没有把输出形状说死，后续系统就要再做一次人工清洗。这样做不是不能用，只是把本来应该在 prompt 里解决的问题，转移到了后处理里。

你现在可以怎么试把自己常用的一条 prompt 拿出来，先只改一项：要么改输出形状，要么改缺失处理，要么改输入边界。不要三样一起改，这样你才能看出是哪一层真正起作用。

角色与分隔符

边界

很多坏 prompt 信息挤在一起。你把角色、材料和要求全写成一段，模型就很容易把它们混读。这里最有效的做法，把不同层次的内容隔开。

把它拆开

角色只负责口吻，规则只负责边界，材料只负责输入。拆开以后，模型知道哪里是任务，哪里是规则，哪里是客户原话。

拆开的写法复制

你是一位客服助手。
任务：根据用户描述回复登录问题。
规则：不要索取用户名和密码；不要重复客户原话。
输入：
"""
客户：我登录不了账号。
"""
输出：先判断问题类型，再给出下一步处理建议。

位置该放什么它负责什么放错时会发生什么

角色老师、客服、面试官、编辑说话口吻与视角模型会学到语气，但不知道要做什么

规则禁止项、长度、格式、处理顺序回答边界模型会照着最近一条规则跑，前面的规则容易被冲掉

输入正文、问题、记录、表格被处理的材料模型会把说明文字当成材料的一部分

分隔符"""、###、标题、块状结构切开层次输入和指令混在一起，模型更容易误读

为什么分隔符有用

分隔符的作用很朴素：告诉模型“这一段到那一段是输入，不是说明”。如果没有这个边界，模型会把说明句、角色句和正文一起读，最后输出就容易像“看懂了全部内容”，实际上是把好几层混在一起复述。

对程序来说，分隔符还有另一个好处：你以后要替换输入时，只需要换一块，不需要重写整条 prompt。这个习惯对长期复用很重要，因为它把“内容变化”和“结构变化”分开了。

你可以马上做的检查把你现在的一条 prompt 划成三块：角色、规则、输入。只要有一块说不清，就先补那一块，不要急着加更多词。很多时候，真正该删的是混进去的解释句，不是再加一层修饰。

把单次 prompt 变成模板

复用

一条 prompt 今天能用，不等于明天也好用。想让它能反复改，你要先把会变的地方圈出来，把不会变的地方固定住。最常见的做法，是把角色、任务、输入和输出改成占位符。

替换一次，看哪里会变

实例化后的版本复制

你是一位教学助理。
任务：请根据下面材料完成摘要。
输入：{一篇 800 字以内的课程笔记}
规则：控制在 5 条要点以内；不要编造事实；保留术语。
输出：先给结论，再给 5 条要点。
如果信息不足，请先写明缺少什么，再继续。

这一步的意义是把临时写出来的句子变成一个小工具。以后你只要换材料，不必重新想整个结构。

槽位该放什么最容易写坏的地方写坏后的表现

角色教学助理、客服、编辑、分析师把角色写成任务模型会换口吻，但还是不知道交什么

任务摘要、抽取、分类、改写、对照只写“帮我看看”模型只能自由发挥

输入一段笔记、一条记录、一份表格材料和说明混在一起模型把说明也当成要处理的内容

规则长度、语气、禁止项、缺失处理只写“不要乱说”模型知道回避，却不知道怎么交付

输出字段、顺序、格式把格式藏到最后才提结果能看，不能直接复用

如果只保留一层，通常会坏在哪里

只保留角色，模型会像在“扮演”，但不会稳定产出。只保留任务，模型会知道要做什么，却不知道做到什么程度。只保留输出，模型可能把字段写得很像样，却没有真实内容。模板化为了把最容易变的地方和最不该变的地方分开。

练一下把你自己常用的一条 prompt 改成模板，至少留出 4 个占位符：角色、任务、输入、输出。改完后再问一次：如果明天换一个任务，这条模板还剩多少能继续用。

改写后

结构化 prompt复制

`请从目标用户、核心风险、可执行下一步三个角度分析这个产品。

每个角度控制在 60 到 80 字。

如果资料不足，先写明缺少什么，再给出你能确认的部分。

输出请按“用户 / 风险 / 下一步”三个小标题排列。`

改动为什么这样改会少掉的错误

这一页里最值得抄走的一句如果一条 prompt 以后要重复用，先把输出样子定住，再写输入和任务。这样你以后改的是内容，不是整套骨架。

失败实验室

诊断

同一件事，少补一层，输出就会换一种坏法。这个实验室不追求像真的终端，只追求你一眼看出症状和缺口。

常见症状最可能缺哪层先补什么

方法对照

选法

起手时不要先问“哪种技巧最强”，先问任务要不要样例、要不要结构、要不要分步推理。大多数时候，先用最省力的写法试一轮，再决定要不要升级。

先后顺序

  • 先试 zero-shot，看看任务本身能不能说清。
  • 如果格式开始漂，再补少量示例。
  • 如果还要长期复用，就把 prompt 结构固定下来。

这一页先不追少样本选例、instruction tuning、CoT、RAG、自动提示优化。它们都建立在“你已经会把任务说清楚”这个前提上。

常见误区

误区

只写“详细一点”

这类话听起来像要求，实际上没有可检查边界。更好的写法是把角度、长度、字段和禁止项写出来。

以为角色提示能替代任务

“你是面试官”“你是老师”“你是客服”只是开场视角，不是交付物。角色、任务、材料和格式仍然要分开说。

一上来就堆很多示例

示例是为了稳定格式，不是为了让 prompt 看起来更专业。简单任务先用 zero-shot；只有格式漂、答案不稳时，再考虑 few-shot。

下一步怎么用

行动

把这一页落到真实任务里，最省事的办法是先改一条你最常用的 prompt，不要从零发明一个新模板。先补任务动作，再补边界，再补输出形状，最后才考虑要不要加示例。

输出变成长段废话先补 output indicator02，看少样本怎么稳格式

格式开始稳定，但内容还会跑偏先补任务角度和缺失处理02，继续把样例和指令写实

任务需要多步推理才能做完先把中间步骤拆开03，看看推理型提示

必须依赖外部知识才能答先把资料范围收紧05，往知识增强那边接

这里先不追

边界

这一节先不追 CoT、Self-Consistency、RAG、生成知识、主动优化和多模态提示。它们会在后面的单元继续出现，但前提都是你已经能把一个 prompt 的任务、上下文、材料和输出形式分清楚。现在最重要的动作，是把模糊请求改成可检查请求。

什么时候再回来
