零样本、少样本与指令优化

从直接提问开始，学会什么时候补指令、什么时候补示例，以及怎样避免示例把模型带偏。

这一块先把提示工程的基础打稳：模型并不会自动知道你的真实意图，它只能根据当前 prompt 工作。你要学会把任务说清楚，把资料框出来，把结果格式定下来，再用输出反过来检查 prompt 是否需要改。

学完你应该能做什么

  1. 区分 zero-shot、few-shot、instruction prompting 和 role prompting。
  2. 判断什么时候只需要明确指令，什么时候需要示例。
  3. 选择覆盖、相关、平衡、格式一致的 few-shot 示例。
  4. 识别多数标签偏差、近因效应偏差和常见令牌偏差。
  5. 用格式约束让模型输出更容易阅读、复用和解析。

Zero-shot：先直接把任务交给模型

zero-shot

zero-shot prompting 是不给示例，直接用任务描述和输入材料让模型完成任务。原文里的情感分类例子很典型：只给“文本：今天的天气真不错！情感分类：”，模型可能直接输出“积极”，也可能输出一段解释。

这说明 zero-shot 的优点和风险都很明显。它成本低、上下文短、适合快速试任务；但如果标签集合、输出格式、解释长度没有限制，模型会按自己的习惯补内容。对于常见任务，例如翻译、摘要、简单分类、文本提取，zero-shot 往往值得先试。

最小示例：zero-shot 情感分类
请对下面文本进行情感分类。

标签只能选择一个：积极、消极、中性。
只输出标签，不要解释。

文本：
今天的天气真不错！
判断句任务常见、标签清楚、输出短时，先用 zero-shot；如果结果开始飘，再补示例或更严格格式。

Few-shot：用样例固定模型的输出模式

few-shot

few-shot prompting 是在 prompt 里给少量“输入 → 输出”示例，让模型模仿当前任务的标签、格式和判断方式。原文把情感分类改成三组样例：精彩小说是积极，电影一般是消极，关于友谊是中性。新文本跟在后面，模型就更容易按同样格式输出。

few-shot 不会更新模型参数，它只是利用当前上下文里的示例来临时学习任务模式。它通常比 zero-shot 更稳定，但会消耗更多 token。当示例和输入都很长时，还可能挤占上下文窗口。

正确示例：few-shot 情感分类
文本：这是我读过最精彩的一本小说！
情感分类：积极

文本：这部电影内容一般般。
情感分类：消极

文本：这是一部关于友谊的电影。
情感分类：中性

文本：今天的天气真不错！
情感分类：

适合补示例

标签或风格很特殊，zero-shot 输出格式不稳，需要模型模仿判断尺度。

不急着补示例

任务很简单，输出已经稳定，或上下文预算很紧。

上下文学习：模型不是被训练了，而是在模仿当前模式

in-context

原文提到，少样本提示利用的是大模型的上下文学习能力。上下文学习指的是：模型可以从 prompt 里的少量示例中捕捉当前任务的输入分布、标签空间和输出格式，而不需要更新参数。

这个点容易被误解。few-shot 示例不是“教会模型一个永久能力”，而是在当前对话或当前请求里给它一个临时模式。换一批示例、换示例顺序、换标签词，模型的输出都可能变化。

对象few-shot 中的作用容易错在哪里
标签空间告诉模型可选答案有哪些标签混用：积极/正面/好评同时出现
输入分布告诉模型真实输入长什么样示例太简单，真实任务更复杂
格式告诉模型字段顺序和输出形状示例格式前后不一致
你现在应该能做什么解释 few-shot 的价值：它主要固定当前任务的模式，而不是永久改变模型。

示例选择：质量比数量重要

examples

few-shot 示例不是越多越好。原文引用多篇研究说明，示例选择、示例顺序、标签空间和输入分布都会影响表现。一个好的示例集应该和真实输入相关，覆盖常见情况和边界情况，标签尽量平衡，格式保持一致。

如果示例都太干净，模型遇到真实噪声会不稳；如果示例标签不平衡，模型会偏向多数标签；如果示例离真实任务太远，模型学到的是错误模式。few-shot 的核心不是“凑几道样题”，而是选能表达任务边界的样题。

相关

示例输入要接近真实输入，不要拿短句训练长文任务。

多样

覆盖常见情况、边界情况和容易混淆的情况。

平衡

分类任务中，各标签不要严重失衡。

一致

字段名、标点、标签写法和答案格式保持一致。

有序但不偏

避免模型总是模仿最后一个示例标签。

可测试

示例集要在独立输入上验证，不只看一个输出。

示例偏差：few-shot 为什么会被带偏

bias

原文列了三类偏差：多数标签偏差、近因效应偏差、常见令牌偏差。多数标签偏差发生在示例中的某个标签出现太多时；近因效应偏差是模型倾向重复靠近末尾的标签；常见令牌偏差是模型更容易生成常见词，而不是罕见标签。

这些偏差提醒我们：few-shot 示例本身也是 prompt 的一部分，会改变模型行为。你不能只关注示例内容对不对，还要看标签分布、顺序和标签词是否自然。原文还提到，有研究用无内容输入估计模型对各答案的偏差，再进行校准；实际使用时，至少要做小样本测试。

多数标签偏差

5 个示例里 4 个是积极，新输入容易也被判积极。

近因效应偏差

最后一个示例标签影响过强，模型跟着复读。

常见令牌偏差

模型偏向更常见的标签词，罕见标签更难出现。

判断句few-shot 输出不稳时，不只改指令，也要检查示例的标签分布、顺序和格式。

Instruction prompting：直接告诉模型你的意图

instruction

原文在 few-shot 之后提出一个自然问题：既然示例是为了解释意图，为什么不直接把意图告诉模型？这就是 instruction prompting。现代大语言模型大多经过指令微调，能更好理解自然语言任务说明，并根据指令完成任务。

指令提示适合把任务、标签、标准和输出格式直接写出来。比如情感分类里，不只是给文本，还明确说“分类结果可以是积极、消极或中性”。这会比完全裸问更稳，也比 few-shot 更省 token。

正确示例：instruction 情感分类
对下面文本进行情感分类。

分类结果只能是：积极、消极、中性。
只输出分类结果，不要解释。

文本：今天的天气真不错！
情感分类：

指令模型的发展和指令微调有关。原文提到 FLAN、T0、InstructGPT、ChatGPT、TK-Instruct 等模型。你不需要在这一节背论文细节，但要理解一个实用结论：面对指令模型时，清楚、具体、正向的任务说明非常重要。

角色和格式：让指令更像工作说明

role + format

在真实任务中，instruction 往往会和角色、上下文、格式约束一起使用。角色固定视角，指令说明任务，格式约束让结果可解析。比如客服质检任务，需要“质检员”视角、判断标准、JSON 输出字段和不合格时的改写建议。

组合示例：客服质检
你是一名客服质检员。

任务：
判断下面客服回复是否合格。

判断标准：
- 是否回应用户问题
- 是否避免索要敏感信息
- 是否给出下一步操作
- 语气是否礼貌

输出 JSON：
{
  "result": "合格/不合格",
  "reasons": ["原因1", "原因2"],
  "revision": "如果不合格，给出改写版本；如果合格，填 null"
}

输入：
{conversation}
你现在应该能做什么为中等复杂任务写出角色、任务、标准、输出格式四件套。

优化路径：不要一上来写超长 prompt

iteration

prompt 设计是迭代过程。比较稳的做法是先写最小 zero-shot，观察输出问题，再决定补什么。如果任务不清，补 instruction；如果格式不稳，补 output indicator；如果判断标准漂移，补 few-shot；如果视角很重要，补 role；如果任务太大，拆成多个子任务。

一开始写很长的 prompt，看起来全面，实际上难调试。输出错了，你不知道是角色设定、示例、格式、上下文还是任务本身出了问题。每次只改一个关键点，保留版本，才能知道哪条规则有效。

Zero-shot快速测试
补指令任务更清楚
补格式输出可解析
补示例标准更稳定
拆任务降低复杂度
测试版本保留证据
判断句prompt 优化不是把规则越堆越多，而是根据具体输出问题补最小有效约束。

组合使用：根据失败原因选技术

selection

zero-shot、few-shot、instruction、role、format 不是互斥关系。你可以先用指令说明任务，再用少量示例固定判断尺度，最后用 JSON 约束输出。关键是知道每个要素解决什么问题。

输出问题优先补什么例子
模型不知道做什么instruction明确分类、提取、总结或改写动作
格式每次不同output format字段定义、JSON 示例、缺失值规则
判断尺度漂移few-shot给边界样例和标签示例
视角或语气不对role编辑、客服、面试官、代码审查员
任务太复杂拆任务先提取信息，再分析，再生成结论
完整链路先用 zero-shot 试模型已有能力；再用 instruction 降低歧义；用 few-shot 固定特殊标准；用格式约束提高复用性；最后通过样例测试检查是否真的稳定。
