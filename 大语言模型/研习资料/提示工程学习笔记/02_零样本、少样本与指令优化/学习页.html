第 2 节 · 示例与指令

零样本、少样本与指令优化

这一节先把 zero-shot、few-shot、instruction prompting 和 role prompting 分开看。重点不是把 prompt 写长，而是先判断最小有效写法：任务已经很清楚时先试 zero-shot，输出格式或标签边界不稳时再补 few-shot，任务边界和结果形状不够清楚时再把指令、角色和格式拆开写。CoT、RAG、工具调用这些内容放到后面的单元再展开。

读完后可以做什么

  • 能先判断一条任务该从 zero-shot、few-shot、instruction prompting 还是 role prompting 起手。

学习目标

goal

这一页只解决一件事：当任务还没有复杂到要上推理链、检索或工具时，怎么先选对提示方式。先把任务动作、样本、角色、输出格式这四个东西分开，再谈怎样把它们组合起来。

你要会的事怎么检查常见失手点
判断先用 zero-shot 还是 few-shot看任务是否已经有清楚的标签和格式一上来就堆示例
把模糊请求改成可执行 prompt能写出任务、边界和输出形状只写“详细一点”“专业一点”
看出 few-shot 的偏差来源能指出标签分布、顺序和格式不一致把示例当成越多越好

先抓主线

overview

提示工程不是先学一堆花哨技巧，而是先把最小有效提示写对。模型会按你给出的任务说明、材料边界和输出格式去补全内容，所以真正决定结果的，往往不是句子有多长，而是你先补了哪一层。

zero-shot先跑一个最小版本

任务已经足够清楚、标签集合也不大时，先用一句话把问题交给模型。

few-shot再用示例固定模式

当标签边界、输出格式或风格容易漂时，用少量输入输出示例把样式钉住。

instruction把该做什么说透

适合补充任务动作、长度、字段、禁止项和缺失值处理方式。

role只改变说话姿态

角色能影响视角和语气，但不能替代任务、材料或格式说明。

方法主要补什么解决什么问题什么时候别急着加
zero-shot只有任务快速验证任务是否能做结果已经稳定时，不必先加示例
few-shot任务 + 示例固定标签、格式和风格任务本身还没说清时，先补指令
instruction任务 + 约束把动作、边界和输出写明白只是想换语气时，不要过度写长
role视角 + 语气让回答更像某个身份或场景不要拿角色代替交付物

状态账本

ledger

把 prompt 当成一个小流程看，会更容易排错。你写进去的内容、模型读到的内容、最后必须保住的内容，不是同一件事。下面这张账本先把它们分开。

对象读什么写什么写错后的症状
instruction任务目标要做什么模型知道要回应，却不知道要完成哪种动作
context场景、角色、边界在什么条件下做输出方向漂，或者跑去别的场景
input data待处理材料把要分析的内容圈起来模型把说明和正文混在一起
output indicator交付要求格式、字段、长度、缺失处理结果看着像答完了，但后续不能直接用
先抓一个判断如果你发现结果能读，但不能直接接到程序或文档里，先查 output indicator；如果模型连该做什么都没看准，先查 instruction。

代码读写

structure

把 prompt 写稳，靠的不是堆词，而是把几层内容分开。先写任务，再写材料，再写输出形状，最后再加角色或风格。这样模型更容易知道哪些是输入，哪些是要求，哪些是最后必须交出来的结果。

最小可检查 prompt

抽取任务骨架
任务：请从下面文本中提取公司名称和成立时间。

输入材料：
"""
{text}
"""

输出格式：
1. 只输出 JSON
2. 字段固定为 name 和 establish_time
3. 如果找不到时间，写 null

这类骨架先把任务、材料和输出形状分开。后面再补角色、示例或更严格的语言风格，通常才不会改一层坏一层。

读写顺序

构件作用何时加入常见失手
instruction告诉模型要做什么几乎总是先写只写“帮我看看”
context告诉模型在什么场景下做任务依赖背景时加入把背景写成废话
input data告诉模型处理什么材料有具体文本、表格或问题时加入把材料和指令混在一起
output indicator告诉模型结果长什么样需要列表、表格、JSON 时加入只写“简洁一点”
role告诉模型用什么身份说话需要视角或语气时加入以为角色能替代任务

先记一个规则简单任务可以只用 instruction；要是你已经在交付格式上吃过亏，就先把 output indicator 写死，再考虑要不要加角色或示例。

方法选择

decision

真正有用的不是记住某个术语，而是知道什么时候该停在最小写法，什么时候该加一层。下面这张表就是一个实用的选择顺序：先看任务是否已经清楚，再看输出是否稳定，最后才考虑要不要补示例或角色。

现场信号先试什么为什么什么时候升级
标签很清楚，任务很短zero-shot先看模型能不能直接完成如果多输出解释或标签漂移，再补 instruction
格式容易跑偏instruction + output indicator先把结构钉住如果仍然学不稳，再加 few-shot
示例能明显教会标签边界few-shot让模型模仿输入输出模式要长期复用时，把角色和格式一起固定
需要稳定语气、身份或场景role + instruction角色先定说话姿态，任务再定交付物如果角色写了但结果仍散，再补示例
顺手检查如果你发现自己已经写了三层要求，但输出还乱，先停一下。常见原因不是模型太笨，而是任务、边界和格式还没有被拆开。

示例偏差

bias

few-shot 不是“示例越多越好”。示例一旦开始带偏标签分布、顺序或词形，模型就会学到错误的模式。这个问题不是少样本本身的问题，而是示例设计的问题。

偏差触发条件现场症状怎么缓和
多数标签偏差示例里某个标签占比太高模型倾向重复那个标签让标签分布更平衡
近因效应偏差最后一个示例的标签特别显眼模型容易追着末尾标签跑打散顺序，别让同一标签总排最后
常见令牌偏差标签词本身太常见或太强势模型偏向生成常见词检查标签是否过于含糊或过于惯用
顺序敏感示例排列不稳定同样的题，换个顺序就变了固定一版顺序，再做对比测试

一个常见的校准思路

用 N/A 估计偏差
文本：N/A
情感分类：积极、消极、中性

这个输入本身没有内容，目的是看模型对标签空间有没有天然偏向。它不是正式答案，而是一个校准点。

别把示例当真相

few-shot 的样例是为了教模式，不是为了给事实背书。要是示例都不统一，模型学到的就是混乱的映射：有时学格式，有时学标签，有时只学到末尾那一个词。

真实排错

debug

把这页学到的东西落到实战时，顺序很重要：先判断任务是不是已经清楚，再决定要不要补示例，最后才补角色和更严格的输出格式。这个顺序能让你少走很多弯路，也能让后面的 prompt 更容易维护。

步骤先看什么改写时写什么如果做反了会怎样
1. 任务动作模型到底是要分类、抽取还是改写把动词写出来模型不知道自己要做什么
2. 材料边界哪些内容算输入，哪些不算用分隔符或标签圈起来输入和说明混在一起
3. 输出形状结果是列表、表格还是 JSON把字段、长度和缺失规则写清楚答案看得懂，但不能直接用
4. 最后收束是否需要角色、语气或示例只保留真正影响结果的约束prompt 被空话和姿态词撑大

结构化 prompt 模板

输入：{{材料}}
要求：{{长度、字段、边界、缺失值}}
输出：{{结果格式}}

失败实验室

这里不是在追求更长的 prompt，而是在看症状怎么对应缺口。真正有用的修法，是先补最先坏掉的那一层。

先别追

boundary

这一页先不展开 CoT、Self-Consistency、RAG、APE、Active Prompting 和 Directional Stimulus Prompting。它们会在后面的单元继续出现，但前提都是你已经知道怎么把一条 prompt 的任务、材料和输出形状写稳。

如果你现在还在犹豫“要不要加示例”，先回到前面的选择表：任务已经清楚时先试 zero-shot；格式和标签开始漂时再补 few-shot；需要身份、语气或场景时，再把 role 和 output indicator 一起补上。

练习

diagnostic

怎么答才算会

先看任务是否已经清楚，再判断要不要补示例、补角色或补输出格式。选择题和判断题要能当场核对；改写题要把错因、后果和修法写出来，不要只交一个更长的 prompt。

8. 模糊请求改写

诊断并改写：帮我分析一下这个产品，写详细点，不要太长，也不要乱说。

12. 费曼解释

用自己的话解释：什么时候先用 zero-shot，什么时候再补 few-shot，什么时候要把 role 和 output indicator 一起写进去；再说一句这页为什么先不追 CoT、RAG 或自动化 prompt 工具。

请串联：任务清楚度、格式稳定性、示例作用、角色边界、后续单元。

参考答案 / 评分要点

合格答案至少要说清：任务简单、标签清楚时先用 zero-shot；当格式或标签边界开始漂时，再补 few-shot；需要身份、语气或场景时，把 role 和 output indicator 一起写进 prompt。还要说明这一页先不追 CoT、RAG、工具调用和自动化 prompt，因为那些方法都建立在“先把任务写清楚”这个前提上。
