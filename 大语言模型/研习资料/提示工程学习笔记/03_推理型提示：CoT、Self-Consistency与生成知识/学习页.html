推理型提示：CoT、Self-Consistency 与生成知识

把复杂问题拆成可检查的步骤，理解什么时候先推理、什么时候多路径投票、什么时候先补背景知识。

这一块先把提示工程的基础打稳：模型并不会自动知道你的真实意图，它只能根据当前 prompt 工作。你要学会把任务说清楚，把资料框出来，把结果格式定下来，再用输出反过来检查 prompt 是否需要改。

学完你应该能做什么

  1. 区分普通 CoT、Zero-shot CoT、Few-shot CoT 和 Self-Consistency。
  2. 判断一个任务是否真的需要推理型提示。
  3. 为多步问题设计“关键步骤 + 最终答案”的稳定输出格式。
  4. 说明 Self-Consistency 为什么要多条相对独立的推理路径。
  5. 用生成知识提示辅助常识推理，同时知道它不能替代可靠检索。

为什么需要推理型提示

reasoning

普通 few-shot 对分类、翻译、生成这类任务很有效，但原文指出，它在算术、常识、符号推理等任务上并不总是够用。原因很直接：这些任务不是只要选一个标签，而是要经过多个中间判断。

推理型提示的核心，是把“直接给答案”改成“先给关键步骤，再给最终答案”。它不改变模型参数，只是改变生成路径。对多步问题来说，中间步骤能减少跳步，也方便人检查错在哪里。

问题多条件
条件识别有什么
步骤拆解先后顺序
中间结果可检查
最终答案单独标出
风险可能错因
判断句如果错误常来自漏条件、算错中间量、跳过隐含前提，就考虑推理型提示；如果只是格式转换，不要硬加长推理。

Chain-of-Thought：示范中间推理步骤

cot

Chain-of-Thought，简称 CoT，直觉上就是“先写解题过程，再写答案”。原文介绍的经典差异是：传统提示只给问题和答案，模型可能直接套答案模式；CoT 示例会展示推理过程，模型更容易模仿这种步骤。

CoT 的价值不是让模型说得更多，而是让模型遵循一个解题节奏：识别条件、推导中间量、合成答案。原文也提醒，CoT 不是在所有模型上都同样有效，较小模型可能效果不如标准提示。

正确示例：CoT 输出结构
请解决下面问题。先给出简洁推理步骤，再给出最终答案。

问题：
小明有 12 个苹果，送给小红 3 个，又买了原来一半数量的苹果，现在有几个？

输出格式：
步骤：
1. ...
2. ...
3. ...
最终答案：...
你现在应该能做什么把“直接回答”改成“关键步骤 + 最终答案”的 CoT prompt。

Zero-shot CoT：没有示例时先触发分步分析

zero-shot cot

Zero-shot CoT 不提供人工示例，只在问题后加入类似“Let's think step by step”或“请逐步分析”的触发语。原文提到，这个简单提示在一些复杂推理任务上能明显改善表现，但通常不如 Few-shot CoT 稳。

中文场景里，不要只写“想清楚点”。更稳的写法是把输出结构也写上，特别是要求最终答案单独一行。这样既能引导模型分步，也方便后续检查或解析。

错误示例 触发语太松
错误示例：答案位置不固定
帮我算一下这道题，想清楚点。
改进示例 步骤与答案分离
正确示例：Zero-shot CoT
请逐步分析问题，最后单独给出最终答案。

输出格式：
关键步骤：
1. ...
2. ...
最终答案：...

Few-shot CoT：用示例教模型怎样推理

few-shot cot

Few-shot CoT 是把 few-shot 和 CoT 结合起来。示例中不仅有问题和答案，还包含推理链。模型会模仿示例的推理粒度、步骤格式和答案位置。它更适合重复出现的一类任务，比如同类应用题、同类逻辑判断、同类业务决策。

关键是示例和目标任务要同构。如果前面示例只给答案，最后却要求新题写推理，模型会收到冲突信号。示例里的推理链也要正确，否则模型会学到错误路径。

正确示例：Few-shot CoT 样例格式
示例：
问题：一本书 100 页，每天读 20 页，几天读完？
推理步骤：
1. 总页数是 100 页。
2. 每天读 20 页。
3. 100 / 20 = 5。
最终答案：5 天。

现在回答：
问题：{new_question}
推理步骤：
判断句Few-shot CoT 的示例必须和目标输出同构：都有问题、关键步骤和最终答案。

Self-Consistency：多条推理路径投票

self-consistency

CoT 仍然可能生成错误中间步骤。Self-Consistency 的想法是：不要只信一次推理，而是多次采样得到不同推理路径，再对最终答案做多数投票。原文把它概括为三步：构造 CoT 示例，生成多个 reasoning path，用 majority vote 选择最一致答案。

这里的重点是“路径要有差异”。温度太低，几次结果几乎一样，投票没有意义；温度太高，答案过于分散，也难聚合。原文提到论文里采样很多次，实际应用中常见做法是先采样 5 次以上，再看答案是否集中。

流程示例：Self-Consistency
1. 对同一个问题独立调用模型 5 次。
2. 每次都要求输出：关键步骤 + 最终答案。
3. 抽取每次最终答案。
4. 多数投票。
5. 如果答案分散，标记为“不稳定”，改提示或人工复核。

适合

算术、常识推理、逻辑判断这类单次结果波动的任务。

边界

多数答案不等于真实答案；没有证据时，多路径也可能共享同一个错前提。

生成知识提示：先列背景知识，再回答

knowledge

Generated Knowledge Prompting 适合常识推理。原文说它包含两步：知识生成和知识集成。第一步让模型生成与问题相关的一组事实；第二步把这些知识作为上下文，再回答原问题。

它和 RAG 的区别很重要：RAG 使用外部资料，生成知识提示使用模型自己生成的候选知识。后者可以帮助模型唤起常识，但不能当作可靠来源。涉及医学、法律、金融、最新事实或需要引用来源的任务，应优先检索外部资料。

正确示例：生成知识两阶段提示
第一步：
请列出回答该问题需要用到的背景知识。
要求：只写与问题直接相关的可检验事实；不确定的内容标注“需核验”。

问题：为什么沙漠昼夜温差大？

第二步：
请只基于上面的背景知识回答问题。
如果知识不足，请说明不足。

答案：
判断句生成知识可以作为思考草稿，不能当作引用来源；需要事实可信度时，用 RAG 或人工资料。

方法选择：步骤不清、单次不稳、背景不足

selection

CoT、Self-Consistency 和 Generated Knowledge 解决的问题不同。CoT 解决“步骤不清”；Self-Consistency 解决“一次推理不稳”；Generated Knowledge 解决“需要先唤起背景知识”。把它们混在一起会让 prompt 变长，也会增加成本。

方法主要解决成本风险
Zero-shot CoT临时多步推理低不如示例稳定
Few-shot CoT重复型推理任务中示例质量影响大
Self-Consistency单次推理波动高多数不等于正确
Generated Knowledge常识背景不足中生成知识可能不准
你现在应该能做什么看到一个复杂任务时，先判断失败原因，再选提示策略。

推理输出格式：可读、可查、可解析

format

推理型提示要避免把步骤、答案、不确定性混在一段里。对学习场景，可以让模型写关键步骤；对产品接口，更适合输出简洁推理摘要、最终答案和风险字段。最终答案最好单独标出。

推荐格式：推理摘要
请按以下结构回答：

已知条件：
- ...

关键步骤：
1. ...
2. ...

最终答案：
...

可能的错误来源：
- ...

如果要输出 JSON，不要让模型在 JSON 外写解释。可以把详细推理压缩为 reasoning_summary，保留可检查依据，而不是要求输出很长的内心过程。

判断句推理输出要服务检查和使用，不是越长越好。

常见误区：推理文本不等于正确证明

mistakes

第一类误区是把 CoT 当成事实校验。模型能写出看似合理的步骤，但中间条件、计算或背景事实仍然可能错。第二类误区是在简单任务上强行加 CoT，导致输出变慢、变长、难解析。

第三类误区是 Self-Consistency 路径不独立。如果让模型在一次回答里写 5 条路径，这些路径会互相影响，不如多次独立调用。第四类误区是用 Generated Knowledge 替代检索，把模型自己生成的知识当真实来源。

合理化错误

步骤写得顺，不代表答案对。

投票误用

多数答案只是更一致，不保证真实。

知识幻觉

生成知识是候选背景，不是证据来源。

完整链路复杂题先判断是否需要步骤；单次结果不稳再采样投票；缺背景时先生成候选知识或检索资料；最终答案始终单独标出并保留风险。
