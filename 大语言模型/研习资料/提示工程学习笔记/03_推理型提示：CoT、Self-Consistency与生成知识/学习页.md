# 学习目标 {#goals}
<!-- tag: goal -->

这一页先处理三个常见卡点：多步问题里漏条件，答案看起来对但路径不稳，或者模型把自己补出来的背景当成事实。你要学会把“推理步骤”“最终答案”和“候选知识”分开看，这样后面的提示才知道该控制哪一层。

| 检查点 | 现场看什么 | 失手信号 |
| --- | --- | --- |
| CoT | 题目是否需要中间计算、条件串联或常识拆解 | 只追求一句话答案，看不到漏条件 |
| Self-Consistency | 单次推理是否容易走偏，路径之间是否能独立采样 | 把同一条想法复制很多次 |
| Generated Knowledge | 题目是否先缺背景、但可以先整理候选规律 | 把生成内容当外部证据 |
| 边界 | 事实是否必须可追溯，是否该转向检索或下一节 | 继续往上堆 prompt 名字 |

> **先记住一件事** 推理型提示改变的是回答路径，不是给模型装上新知识。它的作用是把原本容易省略、混在一起或互相污染的步骤拆开。

# 先抓主线 {#overview}
<!-- tag: overview -->

这一单元排在基础 prompt 之后、搜索和分解之前。你已经会把任务、材料和格式写清楚，现在要继续往前一步：让模型把中间判断显出来，让多条路径彼此独立，让候选知识先过一遍再进入最终答案。

**1. 观察失败**漏条件 / 走偏 / 背景不够
**2. 选方法**CoT / Self-Consistency / Generated Knowledge
**3. 控制路径**步骤显形 / 多路径采样 / 先生成知识
**4. 看症状**答案是否更稳、可检查、可追踪

| 方法 | 核心动作 | 输出里要留下什么 | 常见错法 |
| --- | --- | --- | --- |
| CoT | 把中间推理写出来 | 步骤、条件、最终答案 | 只把答案写长 |
| Self-Consistency | 采样多条独立路径再投票 | 最终答案的共识 | 把同一条推理重复很多次 |
| Generated Knowledge | 先生成候选知识，再回答 | 候选规律、解释、缺口 | 把候选知识当事实库 |

> [!NOTE]
> **锚点例子**
> 一个班 40 人，女生占 45%。如果直接拍答案，很容易把 45% 和 40 人之间的换算说糊；把步骤写出来，先算女生人数，再得男生人数，就更容易检查。

# 一步一步看：同一道题怎么走成三种提示 {#mechanism}
<!-- tag: trace -->

先把一题拆成最小的可见对象，很多判断就会清楚。下面这张追踪图用同一个例子：一个班 40 人，女生占 45%。你可以切换看三种写法，各自会把什么东西留下来、什么东西丢掉。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

| 方法 | 输入状态 | 输出状态 | 读写重点 | 错了的症状 |
| --- | --- | --- | --- | --- |
| CoT | 题目里有多个条件 | 步骤可检查 | 读条件，写中间计算 | 直接跳答案，漏条件 |
| Self-Consistency | 单次 CoT 容易走偏 | 多条路径后有共识 | 读各路径最终答案，写多数票 | 路径太像，投票没信息 |
| Generated Knowledge | 背景不够但可先整理规律 | 候选知识先成形 | 读问题，写候选规律，再写回答 | 把候选知识当事实库 |

# CoT：让中间步骤显形 {#cot}
<!-- tag: trace -->

CoT 的意思很直接：不要只看最终答案，把中间判断也写出来。对算术、常识、多条件问题来说，这一步能暴露漏条件、错单位和跳步。少样本 CoT 会给出一个完整示范；Zero-shot CoT 则是在提示里直接要求模型先一步步思考。

## box soft
### 直接问法
没有步骤要求的提示复制
```text
一个班有 40 人，女生占 45%，男生有多少人？
```

这类问法可以答对，但你很难看出模型是怎么到答案的，也很难判断它在哪一步算错。

## box info
### CoT 写法
要求中间步骤的提示复制
```text
请先写出计算步骤，再给最终答案。

题目：
一个班有 40 人，女生占 45%，男生有多少人？

输出格式：
1. 计算步骤
2. 最终答案
```

CoT 让每一步都能被人检查。题目越容易漏条件，这个拆分越有用。

| 位置 | 读什么 | 写什么 | 出错时症状 |
| --- | --- | --- | --- |
| instruction | 任务和题目 | 要求先写步骤 | 模型直接跳到结论 |
| reasoning steps | 题目里的条件 | 中间计算、判断或归纳 | 漏条件、单位混乱 |
| final answer | 步骤得出的结果 | 单独标出的最终答案 | 步骤和答案搅在一起 |

> **一个实用判断** 如果题目只是单步抽取、简单翻译或固定格式输出，就别硬把它拉成 CoT；中间步骤本身不提供额外信息，反而会把输出撑散。

| 场景 | 为什么 CoT 不一定更好 | 更合适的下一步 |
| --- | --- | --- |
| 固定抽取 | 答案可以直接映射，不需要多步推理 | 把格式写清楚，别加额外步骤 |
| 事实敏感 | 关键来源是否可信 | 转向检索或证据化流程 |
| 过长解释 | 步骤越多，越容易把真正关键的判断淹没 | 压缩成 2-4 个可检查动作 |

# Self-Consistency：让多条路径彼此独立 {#consistency}
<!-- tag: vote -->

Self-Consistency 处理的是另一类问题：CoT 已经把步骤写出来了，但某一条路径仍然可能在中途走偏。它的做法是让模型采样多条独立推理路径，再对最终答案做多数投票。重点不在“多写几遍”，而在“先分开采样，再汇总结果”。

## box
### 流程草图
Self-Consistency 的最小流程复制
```text
1. 构造一个能触发推理的提示
2. 让模型采样多条 reasoning path
3. 只比较每条路径的最终答案
4. 用多数投票得到最后结果
```

如果所有路径都来自同一次连贯回答，它们会互相影响，票数看起来很多，实际独立性却不够。

## box warn
### 采样时要盯住什么
检查点要看什么出错会怎样

路径独立性每条路径是不是独立采样答案看似很多，其实互相带偏
温度是否保留适度随机性太低会全都一样，太高会散开
投票对象投的是最终答案，不是整段推理长解释会把差异冲淡

| 位置 | 读什么 | 写什么 | 症状 |
| --- | --- | --- | --- |
| sample path | 同一个题目 | 独立 reasoning path | 路径太像，投票没有信息量 |
| vote | 各路径最终答案 | 共识结果 | 把整段解释当票，噪声变大 |
| final answer | 多数答案 | 最后输出 | 只看到一条路径的影子 |

# 完整案例：客服对话怎么走成三种提示 {#worked-case}
<!-- tag: case -->

假设用户说：“物流很慢，包装也破了，我是不是该退货？” 这句话表面上像一个简单的售后咨询，实际上包含了三层信息：先是投诉事实，再是潜在意图，最后才是行动建议。很多 prompt 的问题，你没有告诉它先看哪一层、后看哪一层。

下面把同一个问题拆成三种写法。你先不用急着记术语，先看每种写法把什么留下来了。CoT 负责把判断过程显出来，Self-Consistency 负责让“退货”还是“咨询”这类标签别被单次偶然结果带歪，Generated Knowledge 负责先整理与售后判断相关的候选规律，再组织答案。三者看起来都在“想一想”，但实际控制的对象不一样。

## box soft
### 版本 A：CoT
先拆事实，再下判断复制
```text
请先把用户原话拆成事实，再判断退货意图是否明确。

用户原话：
物流很慢，包装也破了，我是不是该退货？

输出格式：
1. 事实
2. 判断依据
3. 最终标签
```

这一版最适合做的是“看清楚模型有没有把事实和结论混在一起”。如果它直接把“包装破了”当成“已经决定退货”，你就知道步骤链里少了一个中间判断。

## box info
### 版本 B：Self-Consistency
先独立采样，再比较最终标签复制
```text
请对同一段用户对话独立思考 5 次。
每次都先写出判断依据，再给最终标签。
最后只比较最终标签，不要比较整段解释。

用户原话：
物流很慢，包装也破了，我是不是该退货？
```

这一版最适合看“标签稳不稳”。有时某条路径会更偏向“退货”，另一条会更偏向“售后咨询”。如果独立采样后的多数票稳定，那说明这个任务更适合用 Self-Consistency 收束；如果每条路径都长得一样，那就说明你其实没有采到独立性。

## box warn
### 版本 C：Generated Knowledge
先列候选规律，再做判断复制
```text
先列出 3 条与“是否要退货”相关的候选规律。
例如：什么样的说法更像明确退货请求，什么样的说法更像单纯抱怨。
然后再根据这些候选规律判断用户意图。
如果规律不够，请明确写出缺口，不要补造流程。
```

这一版最适合做的是“先把判断标准摆出来”。它在帮你整理思路。你后面仍然要决定：这条判断是不是需要更可信的资料、售后政策或可追溯规则。

## box
### 同一个问题的三种失败方式
方法最常见的失手你在输出里会看到什么下一步怎么修


CoT把抱怨直接当成退货请求事实和判断粘在一起先列事实，再下结论
Self-Consistency所有路径都在重复同一句话票数高，但没有新信息分开采样，减少路径相关性
Generated Knowledge把候选规律写成流程或政策看起来很完整，实际上来源不明显式标注“候选”，必要时转 RAG

把三种写法并排放在一起看，最容易看见的“每个方法在改哪一层”。CoT 改的是推理步骤的透明度，Self-Consistency 改的是路径偶然性，Generated Knowledge 改的是背景的组织方式。你一旦把这一层想清楚，就不会把它们都当成“让模型想多一点”的同义词。

这个案例还有一个很实用的提醒：如果你在售后、医疗、财务或政策场景里工作，最终答案不是看起来顺就行。你得知道答案是从哪里来的。到了这个层面，Generated Knowledge 只能帮你整理候选解释，不能替你证明事实。这个分界线要记牢，不然你会把“会说”误当成“会证实”。

> [!NOTE]
> **这个案例可以直接拿去改 prompt**
> 先把用户原话拆成事实，再让模型判断意图；如果单次判断不稳，就做多路径采样；如果判断标准本身还不清楚，就先列候选规律。三步分别对应三种方法，不要混用。

# Generated Knowledge：先生成候选知识，再回答 {#knowledge}
<!-- tag: knowledge -->

Generated Knowledge Prompting 的思路是先让模型写出与问题相关的规律、常识或背景，再用这些候选知识去回答问题。它适合背景不够但可以先整理思路的场景。它不等于检索，也不等于证据校验，所以在事实敏感任务里只能当起草步骤，不该直接当结论。

## box soft
### 第一阶段：生成候选知识
先列知识，再回答复制
```text
问题：为什么热水比冷水更容易溶解糖？

请先列出 3 条相关知识或规律。
不要直接回答问题。
```


## box info
### 第二阶段：基于知识回答
用候选知识组织答案复制
```text
请只使用上一步列出的知识回答问题。
如果知识不够，请明确写出缺口，而不是补造事实。
```


| 方法 | 知识从哪里来 | 输出里保留什么 | 更适合什么场景 |
| --- | --- | --- | --- |
| Generated Knowledge | 模型自己先整理出候选解释 | 候选规律、常识、缺口 | 先起草解释，再看是否需要外部资料 |
| RAG | 外部文档、数据库、网页或笔记 | 可追溯证据 | 事实敏感、来源明确、需要引用的任务 |

## box soft
### 候选知识不是结论
Generated Knowledge 的第一阶段只是在帮你找可能有用的规律。它能把“该往哪想”先写出来，但不能证明这些规律一定在当前任务里成立。第二阶段的职责，是把这些候选规律变成一段可读答案，同时保留缺口。

## box info
### 一个小型故障例子
如果模型在第一阶段写出“热水能加快分子运动”“糖在高温下更易溶解”，但第二阶段没有标出“这只是常识级解释，不是实验数据”，它就很容易把解释写成定论。事实敏感任务里，这就是该转向 RAG 的信号。

> **边界放这里** 如果你已经知道有可信资料、文档或数据库可查，就不要在这里停太久。Generated Knowledge 适合整理候选解释；RAG 负责把外部资料带进来。

# 方法选择 {#compare}
<!-- tag: diagnosis -->

选方法时先看失败类型，再看术语。题目会漏条件，就先考虑 CoT；同一条链容易飘，就看 Self-Consistency；背景不够但可以先列规律，就考虑 Generated Knowledge；如果事实必须可追溯，就转向 RAG。

| 场景 | 先试什么 | 为什么 | 失败后看哪里 |
| --- | --- | --- | --- |
| 多步数学、逻辑或常识推导 | CoT | 先把步骤显形，检查漏条件 | 看中间步骤是否跳步 |
| 单次推理结果不稳 | Self-Consistency | 多条独立路径投票更稳 | 看路径是否真的独立 |
| 背景不足，但能先整理规律 | Generated Knowledge | 先起草候选知识，再组织回答 | 看候选知识有没有被当成证据 |
| 事实必须可追溯 | RAG | 外部资料能提供来源 | 看检索是否命中、证据是否够 |

## box
### 同一个问题的三种写法
- CoT：先写计算步骤，再给最终答案。
- Self-Consistency：采样多条步骤不同的路径，只比较最终答案。
- Generated Knowledge：先列出相关规律，再用这些规律回答。

## box warn
### 先试哪一步
当你还不确定该用哪种方法时，先从最小版本开始。能直接答对就不要先加路径投票；步骤已经显形但答案还散，再加 Self-Consistency；背景不够但能先整理知识，再用 Generated Knowledge。

# 决策实验：先看错因，再选方法 {#decision-lab}
<!-- tag: decision -->

真正做 prompt 时，最难的不是知道 CoT、Self-Consistency 和 Generated Knowledge 的名字，而是看到一个失败样本后，能不能立刻判断要补哪一层。先看任务有没有多步条件；如果有，就让步骤显形。再看单次路径是否不稳；如果不稳，就做独立采样和投票。最后再看任务是不是缺背景但不缺事实来源；只有这类题，才适合先让模型整理候选知识。

这一步的直觉很简单：你在对症下药。CoT 解决的是过程不可见，Self-Consistency 解决的是偶然走偏，Generated Knowledge 解决的是背景空白。只要你把这三种失败方式分开，后面的 prompt 改写就不会乱。

## box soft
### 四种常见任务信号
任务信号先做什么不要先做什么为什么


多步计算、条件串联、逻辑拆解先用 CoT 把步骤写出来不要只说“详细一点”问题不在答案长短，而在中间步骤是否可检查
同一题单次结果常走偏先做独立采样，再做多数投票不要把同一条想法重复很多次需要的是路径独立性，不是字数重复
背景不够，但能先整理候选规律先生成候选知识，再组织回答不要把候选知识当成外部证据这一层只负责起草，不负责证明
事实必须可追溯，或者要引用来源直接转向检索或可验证资料不要继续堆生成式提示再会说也不能替代来源

## box info
### 最小切换顺序
先判断，再选方法复制
```text
1. 先判断任务是不是多步问题
2. 如果是，先让步骤显形
3. 如果步骤已经显形但答案还不稳，再做独立采样和投票
4. 如果任务只是缺背景，就先列候选知识
5. 如果任务要求证据或来源，立刻转向检索
```

先用最小版本试一轮，不要一开始就把三种方法叠在一起。先看有没有漏条件，再看答案稳不稳，最后才问是不是要先补知识。这样改 prompt，才知道是哪里真的起作用了。

> [!NOTE]
> **一个判断模板**
> 如果你能把错因说成“漏步骤”“路径不稳”“背景不足”三类之一，下一步通常也就清楚了。漏步骤优先 CoT，路径不稳优先 Self-Consistency，背景不足优先 Generated Knowledge；要证据时则停在这里，转向检索。

把这一套练熟之后，你看 prompt 就不会只看措辞，而是会先问它控制了哪一层。这个习惯比记术语更重要，因为后面的 ToT、LtM、RAG、主动优化，都是在同一套判断上继续展开。

## box warn
### 再看一个小案例
用户说“我想换货，但客服只让我等”。如果模型直接输出“建议立刻退货”，问题通常把“换货诉求”和“退货决策”混成了一层。先拆事实，再判断意图，最后才给动作，CoT 这时最先上场。

## box
### 对应修法
症状先修哪里提示里要加什么


事实和结论粘在一起CoT要求先列事实，再写判断依据
同一题每次都飘Self-Consistency要求独立采样多次，只比较最终标签
背景缺失但还想先起草Generated Knowledge先列候选规律，再标出缺口

> **练习钩子** 把你现在手里的一条 prompt 按这三列重写一次：先写症状，再写先修哪里，最后写提示里要加什么。只要这一步能做出来，后面的案例就不容易混。

# 常见误区 {#mistakes}
<!-- tag: debug -->

| 误区 | 现场症状 | 更可能的原因 | 怎么修 |
| --- | --- | --- | --- |
| 把 CoT 写成“越长越好” | 步骤很多，关键判断反而看不见 | 只加字数，没有加结构 | 把步骤压成可检查的 2-4 步 |
| 把 Self-Consistency 写成“同一条答案说很多次” | 票数很高，信息却没有增加 | 路径没有独立采样 | 分开采样，再只比最终答案 |
| 把 Generated Knowledge 当事实库 | 答案看起来流畅，但来源不清 | 候选知识没有被区分为“自己生成的” | 第二阶段只许用上一步内容，并标出缺口 |
| 在需要证据时还停留在生成知识阶段 | 回答能讲通，却不能追溯 | 任务本身要求外部来源 | 转向 RAG 或其他可检索路径 |

## box soft
### 快速自查
如果你已经能说出每一步读什么、写什么、依赖什么，就不会把这三种方法混着用了。先看任务缺的是步骤、稳定性还是背景，再决定提示怎么改。

## box info
### 写给自己的一句提醒
能解释不等于能证明。CoT 让过程更清楚，Self-Consistency 让路径更稳，Generated Knowledge 让候选解释先成形；需要外部证据时，还是要回到检索和来源。

### 13 diagnostic

**Q:** 13. CoT 输出混在一起

Explain: 好答案会指出：中间结果和最终答案冲突，且没有把“女生人数”和“男生人数”分开标记。修法是先写计算步骤，再单独写最终答案字段，避免把 18 和 22 混成一个结论。

- [ ] 独立采样多条路径，再只比最终答案
- [ ] 用多数投票选最终答案

### 15 scenario-transfer

**Q:** 15. Generated Knowledge 的边界

Explain: 不能只用 Generated Knowledge。因为这是事实敏感任务，必须能追溯来源；生成知识只能帮助你先想可能的答案或背景，不足以证明政策年份和文件名。更合适的是转向 RAG 或可检索资料。
