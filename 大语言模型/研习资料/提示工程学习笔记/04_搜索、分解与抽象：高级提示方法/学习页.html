搜索、分解与抽象：高级提示方法

从一条推理链升级到可分解、可搜索、可回退、可评估的复杂任务处理流程。

这一块先把提示工程的基础打稳：模型并不会自动知道你的真实意图，它只能根据当前 prompt 工作。你要学会把任务说清楚，把资料框出来，把结果格式定下来，再用输出反过来检查 prompt 是否需要改。

学完你应该能做什么

  1. 解释 Least-to-Most 为什么不是普通列步骤。
  2. 区分 Self-Ask、ReAct、ToT、Step-Back 和 APE 的主要用途。
  3. 为复杂任务设计子问题、候选路径、评分标准和停止条件。
  4. 识别伪 ToT、伪 ReAct、无评测集 APE 等常见错误。
  5. 根据任务结构选择分解、搜索、抽象、工具反馈或自动优化。

本块主线：从一条链到可控流程

overview

第 03 块讲的是先推理、多路径投票和先补背景知识。第 04 块再往前走一步：复杂任务有时不是一条推理链能解决的。问题可能需要拆成子问题，也可能需要比较多个候选方案，甚至需要调用工具观察结果。

原文从 Least-to-Most、Tree of Thoughts、Step-Back、APE 等方法切入。它们的共同点是：不满足于一次性生成答案，而是把问题处理过程显式组织出来。你要学会先判断任务结构，再选提示方法。

任务困难优先方法核心动作
大问题太复杂Least-to-Most拆成有依赖顺序的小问题
需要补中间事实Self-Ask提出追问并回答中间问题
需要工具反馈ReAct推理、行动、观察交替
多条方案要比较ToT生成候选、评分、剪枝、回溯
细节太多容易跑偏Step-Back先抽象原则，再回到问题
提示本身要优化APE生成候选提示，用评分函数筛选

Least-to-Most：先拆小，再逐步变大

decompose

Least-to-Most Prompting 的直觉是分治。原文说它先进行问题拆解，再有序回答子问题，并把前一个子问题的答案作为后一个子问题的上下文。它解决的是 CoT 泛化不足：旧示例的推理链不一定能迁移到新问题。

真正的 LtM 不是“列出步骤”。它要求子问题有依赖关系，而且从容易到困难逐步累积。每个子答案都应该帮助后续问题，而不是一堆并列要点。

正确示例：Least-to-Most 结构
请按 Least-to-Most 方式处理：
1. 先把原问题拆成有依赖顺序的子问题。
2. 从最基础的子问题开始逐一回答。
3. 每个子答案都作为后续问题的上下文。
4. 最后汇总成原问题的答案。

原问题：
{complex_question}
判断句如果子问题之间没有先后依赖，只是并列清单，就还不是 Least-to-Most。

Self-Ask：让模型提出必要追问

follow-up

Self-Ask 适合多跳问答。它要求模型先判断是否需要追问，再提出 follow-up questions，回答中间问题，最后合成答案。原文没有把 Self-Ask 展开成单独小节，但它和 LtM、ReAct 处在同一类问题里：把隐藏的中间问题显式化。

Self-Ask 的关键不是问得多，而是每个追问都服务原问题。它也很适合接搜索：模型负责提出中间事实问题，搜索或数据库负责回答，模型再做组合。

流程示例：Self-Ask
Question：某论文方法是否适合实时客服系统？
Are follow up questions needed here：Yes.
Follow up：该方法是否需要多轮模型调用？
Intermediate answer：需要。
Follow up：实时客服系统的主要限制是什么？
Intermediate answer：延迟、成本、稳定性。
So the final answer is：适合复杂问题升级流程，不适合作为所有请求默认路径。
你现在应该能做什么把复杂问答改写成“需要哪些中间事实？每个事实该问什么？最后怎样组合？”

ReAct：推理、行动、观察交替

tools

ReAct 把 Reasoning 和 Acting 结合起来。它不是只让模型在文本里思考，而是让模型根据当前推理选择行动，例如搜索、查询数据库、运行计算，再读取观察结果并更新下一步。

它适合知识密集型问答、网页检索、数据查询、代码调试、交互式任务。前提是工具边界清楚：能用哪些工具、参数怎么写、工具失败怎么办、什么时候停止、哪些动作需要确认。

正确示例：ReAct 循环
Thought：我需要确认 Tree of Thoughts 的定义。
Action：search("Tree of Thoughts deliberate problem solving language models")
Observation：资料显示 ToT 强调候选想法、状态评估和搜索。
Thought：现在需要和 CoT 做对比。
Action：search("Chain of Thought prompting intermediate reasoning")
Observation：CoT 主要是一条中间推理链。
Final：ToT 与 CoT 的关键区别是：CoT 生成一条链，ToT 搜索多条候选路径。
判断句没有工具列表、没有观察结果、没有停止条件的“边想边做”，不是可靠 ReAct。

Tree of Thoughts：把推理当成搜索树

search

原文说 ToT 解决的是 CoT“一条路走到底”的问题。它允许模型在中间过程探索多个推理路径，评估不同状态，并具备前瞻和回溯能力。树上的每个节点代表当前问题状态和到目前为止的思考过程。

一个完整 ToT 至少包含四件事：思考分解、候选想法生成、状态评估、搜索算法。没有候选数量、评分标准、保留规则和停止条件，就很容易变成“几个专家聊天”的伪 ToT。

Thought 分解

把中间过程切成合适大小的状态。

候选生成

为当前状态生成多个下一步想法。

状态评估

判断哪些候选更有希望。

搜索策略

BFS、DFS、剪枝、回溯或其他策略。

停止条件

达到答案、超过深度、评分过低或成本到顶。

成本控制

ToT 往往需要多次模型调用，不能乱用。

修正伪 ToT：加入评分和保留规则
请生成 3 个候选方案；
按“正确性、成本、可执行性”分别打分；
保留前 2 个继续展开；
如果后续发现方案不可行，回到上一层选择备选方案；
最多展开 3 层，最后输出评分最高方案。

Step-Back：先抽象原则，再回到细节

abstraction

Step-Back Prompting 的核心是“退一步”。原文说，它先抽象出问题背后的基础原理和高级概念，再利用这些原则解决具体问题。这个方法适合细节很多、直接推理容易被局部条件带偏的任务。

例如物理题可以先问涉及哪些定律，再代入具体数值；历史或多跳问答可以先问更高层级的概念或背景，再回到具体时间范围。Step-Back 不是逃避细节，而是先建立解题框架。

正确示例：Step-Back 两阶段
第一步：请先后退一步，说明解决这个问题涉及哪些基础概念、原则或定律。
不要直接计算。

第二步：请基于这些原则，回到原问题并逐步求解。

原问题：
{question}
判断句如果模型在细节里绕晕，先让它抽象原则，再回到具体条件。

APE：自动生成和筛选提示

optimization

APE，Automatic Prompt Engineer，解决的是提示词人工设计成本高、凭感觉优化不稳定的问题。原文把它概括为：给定输入输出样例，让模型生成候选指令；用评分函数评估候选；再生成语义相近的变体继续搜索。

APE 的关键不是“让模型帮我优化 prompt”这句话，而是有样例、有指标、有测试。分类、抽取、格式化这类可评分任务更适合 APE。开放写作也可以做，但要先定义评分规则。

APE 基本流程
1. 准备代表性输入输出样例。
2. 生成多个候选提示。
3. 用准确率、格式符合率或人工评分打分。
4. 保留高分候选。
5. 对高分候选生成变体。
6. 在独立测试集上验证。
判断句没有评测集和评分函数的“自动优化 prompt”，通常只是文案改写。

方法对照：什么时候用哪一个

comparison
方法最适合不适合检查点
Least-to-Most可线性拆解的复杂问题多路线搜索子问题是否有依赖顺序
Self-Ask多跳事实问答开放创作追问是否服务原问题
ReAct需要工具和观察反馈无工具任务Action 是否明确可执行
ToT规划、搜索、比较方案简单任务是否有候选、评分、回溯
Step-Back需要先抽象原则的任务纯格式转换抽象原则是否能指导细节
APE可量化评估的提示优化无评测集任务是否有样例和评分函数
完整链路复杂任务先判断结构：能线性拆就 LtM；需要外部观察就 ReAct；要比较方案就 ToT；细节太多先 Step-Back；要优化提示本身就 APE。

常见误区：方法名字不能替代流程

mistakes

第一类误区是把“列步骤”当成 Least-to-Most，但子问题没有依赖顺序。第二类误区是把“三个专家讨论”当成 ToT，却没有候选、评分、剪枝和回溯。第三类误区是把 ReAct 写成口号，没有工具定义和观察结果。

第四类误区是 APE 没有评测集，只让模型写几个“更好看的 prompt”。第五类误区是 Step-Back 抽象得太空，抽象原则无法指导具体解题。方法名本身没有价值，真正有价值的是流程约束。

伪 ToT

多角色聊天没有评分和回溯，不等于树搜索。

伪 ReAct

没有工具、观察和停止条件，只是在写过程。

伪 APE

没有样例和指标，无法判断哪个提示更好。
