04 搜索、分解与抽象

搜索、分解与抽象：高级提示方法

这一页处理的是复杂任务的几种常见卡点：题目太大、路径太多、细节把规则盖住、事实不能靠猜、提示本身不稳，或者输出需要朝一个方向收束。LtM 负责分解，ToT 负责搜索，Step-Back 负责抽象；RAG、Generated Knowledge、APE、Active Prompting 和 DSP 则把同一套思路延伸到证据、样例和方向控制上。

学习目标

goal

先抓四层

这页的判断顺序很固定：先看任务是不是必须分解，再看要不要搜索多条路径，再看是否需要先抽象原则，最后才看证据、样例和提示优化。

你要盯住的对象

对象分不清，方法就会混：LtM 管子问题顺序，ToT 管候选路径，Step-Back 管抽象层，RAG 管外部证据，Generated Knowledge 管候选知识，APE 管提示候选，Active Prompting 管样例选择，DSP 管方向刺激。

症状先动哪一层先看什么失手信号
题目必须先后依赖分解LtM子问题顺序乱，前一步答了也帮不上后一步
同一题有多条路搜索ToT只有聊天，没有评分、剪枝和回溯
细节把规则盖住抽象Step-Back还在细节里打转，没有先提一般原则
事实不能靠猜证据RAG回答流畅，但找不到可追溯依据
提示本身不稳优化APE只在开发样例上更好，换数据就掉

先抓主线

overview

这一页排在 CoT 和自我一致性之后。上一页已经把“把步骤写出来”和“让多条推理路径投票”讲过了，这一页继续往前走：有些任务不只是步骤多，而是必须先拆问题、再搜路径、再抽象原则，最后把证据、样例和方向也接进来。

LtM：先把题拆成能依次回答的子问题

decomposition

Least-to-Most 把依赖顺序排出来。前一个子问题的答案要能喂给后一个子问题；如果拆完之后每个问题都还是独立的，那只是把题目写碎了，不是分解。

看哪里会卡

如果子问题顺序反了，后面的题就接不上前面的答案；如果子问题太大，模型还是会跳步；如果只写“分步骤思考”，却没把子问题列出来，那还是普通 CoT，不是 LtM。

位置读什么写什么失手信号
问题拆解原始任务和依赖关系有顺序的子问题子问题彼此独立，顺序不重要
逐题回答前一题答案 + 当前问题当前子答案前一步的结果没被真正用上
总答案所有子答案最终结论中间答案和总答案混在一行

> [!NOTE]

> 小例子

> “出差住宿超标能不能报销？”可以先拆成“标准是什么”“有没有例外”“缺什么材料”。这比直接写一段长解释更容易检查。

ToT：先把可行路径找出来，再筛掉不好的路

search

Tree of Thoughts 不是“请三位专家聊天”。它关心的是状态、评分和回溯：先生成多个候选状态，再评估这些状态值不值得继续走，必要时剪枝，走不通就退回上一层。

ToT 的关键不是“多”

真正的区别在评分和剪枝。只要没有评分标准，多个“专家”就只是多份文字；只有把候选状态和评分放在一起，才叫搜索。

阶段读什么写什么错法
候选生成当前状态多个下一步候选候选都差不多，没有搜索意义
状态评估候选状态可行性分数只凭感觉，不给标准
回溯失败的分支回到父状态一路写到底，不回头看

> 该停的时候 如果题目本来就只有一条明显路径，比如固定抽取、简单翻译、标准格式填表，就别硬把它做成 ToT。搜索的成本只值得花在多路可比的任务上。

Step-Back：先抽象，再回到具体题

abstraction

Step-Back 处理的是另一种失败：题目里的细节太多，把真正该用的原则盖住了。它要求模型先后退一步，先说出这个问题更一般地属于哪类规则，再拿规则回到具体问题。

看哪里会错

如果模型一直在题目表面绕圈，说明还没抽象出规则；如果抽象成空话，回不来具体条件，那也不行。好用的 Step-Back 要能从一般原则回到原题，不是只会说大道理。

位置读什么写什么失手信号
抽象具体问题里的条件一般规则或原则还在照着细节念，没有抽出规则
推理规则 + 原题条件具体结论规则说了半天，回不到原题

RAG 和 Generated Knowledge：一个接证据，一个接候选知识

grounding

这两种方法都在补“知识”，但来源不同。RAG 先去外部资料里找证据，再回答；Generated Knowledge 先让模型自己列出候选事实，再拿这些候选事实组织回答。前者是证据化，后者是候选化，不能混着用。

Generated Knowledge

生成知识提示骨架复制

先列出 3-5 条与问题相关的候选事实或规律。
再只基于这些候选事实回答。
如果候选事实本身不可靠，就不要把它当作可追溯来源。
方法知识从哪里来适合什么不能替代什么
RAG外部文档、数据库、网页、笔记事实敏感、需要引用来源的任务不能替代证据追溯
Generated Knowledge模型自己生成的候选知识常识解释、写作构思、先起草不能替代外部证据

> [!NOTE]

> 判断句

> 如果题目要求“能追溯”，先接 RAG；如果只是先起草候选解释，再看 Generated Knowledge。候选知识可以帮你想，但不能自动变成事实。

APE、Active Prompting 与 DSP：优化提示、挑样例、推方向

optimization

APE

先生成候选提示，再用评分函数筛选，最后对高分候选做语义相近的重采样。它优化的是整段提示，不是单个词。

DSP

先由策略模型生成方向刺激，比如关键词或限制条件，再把刺激和原始输入合并。它优化的是生成方向，不是把事实盖掉。

这里 u(q) 是问题 q 的不确定性，p_k(q) 是第 k 个答案被采到的概率。分布越分散，不确定性越高，越值得拿去标注。

Active Prompting 的最小流程

Active Prompting 工作流复制

1. 对同一批问题重复采样
2. 计算每题的不确定性
3. 选择最不稳定的问题
4. 人工标注后做 few-shot 示例

DSP 的判断点

DSP 里最容易出错的是刺激词盖过原始输入。它适合内容总结、内容创作和方向性很强的任务，但前提是刺激要收住，不然输出只剩“方向词”，没有原文里的真实限制。

完整案例：同一个报销问题，怎么走完一条可检查的流程

case

把这几种方法放在一起看，最容易发现它们改的不是同一个位置。下面还是用“出差住宿超标能不能报销？”这道题：如果你直接问答案，模型可能会把“标准、例外、材料、结论”混在一行；如果你先抽象，再分解，再接证据，就能看见每一步在读什么、写什么、哪里会卡住。

第二步：把规则拆成子问题

这一步用 LtM。抽象出来的规则再往下拆，就变成一串能依次回答的问题：标准是什么、例外是什么、证据缺什么、材料是否齐全。前一个答案必须能喂给后一个问题，否则就是散列出来的碎问句，不是分解。

LtM 子问题链复制

1. 住宿报销的默认标准是什么？
2. 什么情况下可以超标报销？
3. 如果是会议主办方指定酒店，需要补哪些材料？
4. 现有信息是否足够支持“可以报销”的结论？

第三步：把证据接进来

如果这题要求可追溯，就不能停在候选知识。RAG 会把制度条款放进上下文，再要求模型只根据条款回答。这样最后得到的有出处的判断。

RAG 证据块复制

<context>
[制度-12] 出差住宿按城市等级执行，超标部分原则上由个人承担。
[制度-18] 若因会议主办方指定酒店导致超标，需提交说明并由负责人审批。
</context>

请只依据以上条款回答：能否报销、依据是什么、还缺什么材料？

第四步：如果提示本身不稳，再做优化

如果你发现同一类报销题总是漏掉“例外”或“材料”，这时才考虑 APE 或 Active Prompting。前者把整段提示拿去评分，后者把最不稳定的问题挑出来补样例。它们修的是提示和样例，不是报销规则本身。

阶段读什么写什么如果这一步错了
Step-Back题目细节一般规则与边界只会背细节，不知道先看哪条原则
LtM规则与依赖有顺序的子问题子问题之间断开，答案接不上
RAG制度条款可追溯依据答案流畅，但没人知道依据来自哪里
APE / Active提示和样例分布更稳的提示或样例开发集看着不错，换场景就掉

> 这个案例的用法 以后遇到别的复杂题，也可以照着这条顺序走：先抽象出规则层，再把规则拆成子问题，接着看需不需要证据，最后才考虑提示优化和样例选择。

失败实验室：同一条任务链，三种常见坏法

failure

很多 prompt 看起来“差不多”，其实坏在不同位置。这个小实验还是用报销题来对照：你会看到同一条任务链里，哪一步坏了，后面会出现什么症状，应该回到哪一层修。

坏法二：把 ToT 写成群聊

让模型“想三种方案”却不写评分标准。这样做会让候选数量变多，但没有剪枝、没有回溯，最后只是堆出更多文本。

坏法表面症状真正缺的东西应该回哪一层
LtM 只列问题题目被切碎，但顺序松散子问题之间的依赖回到分解
ToT 只会群聊候选很多，判断很少评分、剪枝、回溯回到搜索
Step-Back 只会拔高抽象词很多，具体结论少从原则回到原题的桥回到抽象后的推理

修法模板

从坏法改成可用 prompt复制

1. 先判断题目卡在哪一层：分解、搜索、抽象、证据、样例或方向。
2. 只改这一层，不要把所有方法一起堆上去。
3. 写清输入、输出和下一步依赖。
4. 如果这一步仍然不稳，再往下检查评分、证据或样例。

你在这里应该先看什么

先看对象，再看流程，再看失败症状。只要你能把一个坏 prompt 归类到某一层，就已经比“再多加一点提示”更接近问题了。接下来可以先修顺序，再修评分，再修证据，最后才修样例和方向。

> 练习钩子 把你手里任意一条复杂 prompt 拿来，先判断它卡在分解、搜索、抽象、证据、样例还是方向，再只改一层。这样改出来的版本，通常比“整体重写一遍”更稳。

改写示范：同一题，三种层级的 prompt

rewrite

很多人写复杂 prompt 的时候，第一反应是“把要求再说细一点”。这页更建议换一种方式：先判断问题在哪一层，再只把那一层写清楚。下面用同一个任务来演示：你要让模型帮你评估一篇文章值不值得继续读，或者一项方案值不值得继续做。

版本二：LtM 改写

LtM 改写复制

请把“这篇文章好不好”拆成 4 个子问题：
1. 文章要解决什么问题？
2. 它给出了哪些证据或例子？
3. 这些证据是否支持它的主张？
4. 读完之后，我应该继续读、放弃读，还是只保留其中一部分？
先逐题回答，再给总判断。

版本三：Step-Back + RAG

抽象后接证据复制

先把这个问题抽象成“如何判断一篇文章是否值得继续投入时间”。
请先说出判断这类内容时一般看哪几条原则，再根据给出的资料逐条核对。
如果资料不足，请说明缺口，不要直接下结论。

这三版分别在改什么

版本改的是哪一层适用时机坏了会怎样

普通问法几乎没改只适合非常简单的判断输出会飘，原因不清楚

LtM把判断拆成子问题文章评估、方案评估、审稿子问题顺序乱，结论接不上

Step-Back + RAG先抽象原则，再接证据规则很多、事实敏感、需要引用依据抽象成空话，或证据没接上

这条链怎么检查

如果你能从普通问法一路改到 LtM，再改到 Step-Back + RAG，说明你已经在做层级控制，而不是只是在加字。检查时只问三个问题：先看什么、后看什么、缺什么证据。

遇到失败时回哪一步

如果模型一直跳答案，先回 LtM；如果规则说得很散，先回 Step-Back；如果结论对，但找不到依据，先回 RAG。这样修，比把整段 prompt 全改一遍更省力。

> 实践钩子 把你最近要处理的一条真实任务写成三版 prompt：普通问法、LtM 版、Step-Back 版。写完以后只比较一个地方：每版到底让模型先看了什么。

选法演练：同一批任务，怎么落到不同的提示结构

decision

很多人学到这里，还是会在“到底该用哪种方法”上打转。最实用的判断方式看任务的第一道门槛在哪里：先后依赖、路径分叉、规则抽象、事实可追溯、提示不稳定，还是样例总挑错。下面这组例子把同一套判断法写得更细一点。

任务 B：方案比较

“三个选题哪个更值得做？”这类题有多条路，所以优先 ToT。要把候选、评分和回溯写清楚，不能只让模型自由发散。

错法示范：把所有方法堆进一条 prompt

混搭 prompt复制

请先抽象原则，再拆子问题，再找多个候选路径，再检索证据，再生成提示并评分。

这类写法的问题对象没分开。模型会知道你想要很多动作，但不知道第一步应该抓什么。

修正示范：只改一层

只改一层的 prompt复制

你现在只做一件事：先判断这个任务卡在哪一层。
如果是事实问题，先接证据；
如果是多路比较，先列候选状态并评分；
如果是规则被细节盖住，先抽象原则；
如果是后续子问题依赖前一步答案，先拆成有顺序的子问题。
不要同时展开所有方法。
任务信号第一步不该先做什么修完后能检查什么
事实敏感接 RAG不该先靠生成知识补猜证据是否真的进入上下文
多路比较接 ToT不该先写一堆没有评分的想法候选是否有分数，是否需要回溯
规则提炼接 Step-Back不该先堆细节原则是否回到了原题
依赖顺序接 LtM不该把子问题写成平铺的清单前一步答案是否真的进入下一步

> 实践钩子 挑一条你最近会用到的任务，把它先归到四种信号之一，再只改一层。改完以后，问自己一句：下一步能不能直接检查这层有没有做好。

方法对照

compare
你看到的信号先动哪一层优先方法第一动作
子问题有先后顺序分解LtM列出有依赖顺序的子问题
同一题有多条可行路搜索ToT生成候选、打分、剪枝、回溯
细节把规则盖住了抽象Step-Back先写一般原则，再回到原题
事实必须可追溯证据RAG先检索，再回答
能先起草候选知识候选化Generated Knowledge先生成候选事实，再组织回答
提示本身不稳优化APE生成候选提示并评分
样例总是选错样例选择Active Prompting找不稳定的问题再标注
需要朝一个方向写方向控制DSP先生成刺激，再合并输入

> 一句话分法 分解看依赖，搜索看候选，抽象看规则，RAG 看证据，Generated Knowledge 看候选知识，APE 看提示候选，Active Prompting 看样例分布，DSP 看方向刺激。

路由检查：先判断任务信号，再选方法

route

把这些方法真正用起来时，最容易出错的地方是第一步。你拿到一条任务时，不要先想名字，先看它的信号：是要先后依赖、还是要多路比较、还是要先抽象、还是要接证据。下面用一条“制度问答”来走一遍路由。

路由表

信号方法下一步错法症状

事实敏感RAG先放证据，再回答结论流畅，但找不到来源

顺序依赖LtM先拆子问题，再逐题回答子问题平铺，没有传递关系

多路比较ToT生成候选、评分、剪枝看起来热闹，实际没有搜索

规则被盖住Step-Back先抽象原则，再回到原题一直停在细节层

可直接复制的路由 prompt

路由骨架复制

请先判断这个任务卡在哪一层：事实、顺序、路径还是抽象。
如果是事实问题，先给出需要的证据，再回答。
如果是顺序问题，先拆成有依赖的子问题，再逐题回答。
如果是路径问题，列出候选方案并说明评分标准。
如果是抽象问题，先提炼原则，再回到具体题。
最后说明你为什么选这个方法。

错误路由会发生什么

如果把事实题当成 LtM，模型可能会把子问题写得很顺，但证据仍然空着；如果把规则题当成 ToT，候选会很多，原则却没提出来；如果把顺序题当成 RAG，拿再多资料也解决不了前后依赖。问题的关键通常不在“方法不够多”，而在第一步选错了。

> 实践钩子 下次你拿到一条复杂任务，先用这一页的四个信号做一次路由。只要能先说清楚“这题卡在哪一层”，后面的提示就会简单很多。

Q: 13. LtM 改写

参考答案 / 评分要点

。合格答案要把问题拆成依赖顺序明确的子问题，比如先问标准、再问例外、再问材料，最后再汇总结论。只写“请分步骤思考”还不够。

14scenario-transfer

Q: 14. RAG 改写

参考答案 / 评分要点

。合格答案要先放入制度条款或检索到的片段，再明确要求“只根据资料回答”。如果资料不足，要让模型直接说出缺口，而不是自己补猜。

15scenario-transfer

Q: 15. Step-Back 或 ToT 的选择

参考答案 / 评分要点

。如果卡点是规则被细节盖住，先用 Step-Back；如果卡点是可行路径很多、需要比较和剪枝，先用 ToT。答案里要说清“题目卡在哪里”，不能只报方法名。

16synthesis

Q: 16. 费曼解释

参考答案 / 评分要点

。高分答案要说清：分解、搜索、抽象、证据、样例和方向本来就在改不同的层。题目有依赖顺序时用 LtM，路径很多时用 ToT，细节盖住规则时用 Step-Back，事实敏感时接 RAG，想先铺候选知识时用 Generated Knowledge，提示本身不稳时看 APE，样例选择错了看 Active Prompting，需要朝一个方向写时用 DSP。把 prompt 写长，只会把这些层混在一起。
