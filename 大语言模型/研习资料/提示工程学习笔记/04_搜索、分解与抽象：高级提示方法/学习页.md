# 学习目标 {#goals}
<!-- tag: goal -->
<!-- layout: grid2 -->

## box
### 先抓四层
这页的判断顺序很固定：先看任务是不是必须分解，再看要不要搜索多条路径，再看是否需要先抽象原则，最后才看证据、样例和提示优化。

## box info
### 你要盯住的对象
对象分不清，方法就会混：LtM 管子问题顺序，ToT 管候选路径，Step-Back 管抽象层，RAG 管外部证据，Generated Knowledge 管候选知识，APE 管提示候选，Active Prompting 管样例选择，DSP 管方向刺激。

| 症状 | 先动哪一层 | 先看什么 | 失手信号 |
| --- | --- | --- | --- |
| 题目必须先后依赖 | 分解 | LtM | 子问题顺序乱，前一步答了也帮不上后一步 |
| 同一题有多条路 | 搜索 | ToT | 只有聊天，没有评分、剪枝和回溯 |
| 细节把规则盖住 | 抽象 | Step-Back | 还在细节里打转，没有先提一般原则 |
| 事实不能靠猜 | 证据 | RAG | 回答流畅，但找不到可追溯依据 |
| 提示本身不稳 | 优化 | APE | 只在开发样例上更好，换数据就掉 |

# 先抓主线 {#overview}
<!-- tag: overview -->

这一页排在 CoT 和自我一致性之后。上一页已经把“把步骤写出来”和“让多条推理路径投票”讲过了，这一页继续往前走：有些任务不只是步骤多，而是必须先拆问题、再搜路径、再抽象原则，最后把证据、样例和方向也接进来。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

| 方法 | 先读什么 | 先写什么 | 错了会怎样 |
| --- | --- | --- | --- |
| LtM | 完整任务和依赖关系 | 按顺序排列的子问题 | 拆成碎句，但前后没有依赖 |
| ToT | 当前状态和候选分支 | 多个候选路径和评分 | 像聊天，不像搜索 |
| Step-Back | 具体题目里的细节 | 抽象原则或一般规则 | 一直停在细节上，不回到规则层 |

# LtM：先把题拆成能依次回答的子问题 {#decompose}
<!-- tag: decomposition -->

Least-to-Most 把依赖顺序排出来。前一个子问题的答案要能喂给后一个子问题；如果拆完之后每个问题都还是独立的，那只是把题目写碎了，不是分解。

## box soft
### 提示骨架
LtM 提示骨架复制
```text
请把问题拆成 3-5 个有依赖顺序的子问题。
每个子问题先单独回答，再把前一题的答案带到下一题。
最后只在最后一行写总答案，不要把中间步骤和总答案混在一起。
```


## box info
### 看哪里会卡
如果子问题顺序反了，后面的题就接不上前面的答案；如果子问题太大，模型还是会跳步；如果只写“分步骤思考”，却没把子问题列出来，那还是普通 CoT，不是 LtM。

| 位置 | 读什么 | 写什么 | 失手信号 |
| --- | --- | --- | --- |
| 问题拆解 | 原始任务和依赖关系 | 有顺序的子问题 | 子问题彼此独立，顺序不重要 |
| 逐题回答 | 前一题答案 + 当前问题 | 当前子答案 | 前一步的结果没被真正用上 |
| 总答案 | 所有子答案 | 最终结论 | 中间答案和总答案混在一行 |

> [!NOTE]
> **小例子**
> “出差住宿超标能不能报销？”可以先拆成“标准是什么”“有没有例外”“缺什么材料”。这比直接写一段长解释更容易检查。

# ToT：先把可行路径找出来，再筛掉不好的路 {#search}
<!-- tag: search -->

Tree of Thoughts 不是“请三位专家聊天”。它关心的是状态、评分和回溯：先生成多个候选状态，再评估这些状态值不值得继续走，必要时剪枝，走不通就退回上一层。

## box soft
### 最小流程
ToT 搜索骨架复制
```text
1. 为当前问题生成多个候选状态
2. 给每个状态打分
3. 保留分数高的状态，剪掉低分状态
4. 如果走不通，回到上一步重新扩展
```


## box warn
### ToT 的关键不是“多”
真正的区别在评分和剪枝。只要没有评分标准，多个“专家”就只是多份文字；只有把候选状态和评分放在一起，才叫搜索。

| 阶段 | 读什么 | 写什么 | 错法 |
| --- | --- | --- | --- |
| 候选生成 | 当前状态 | 多个下一步候选 | 候选都差不多，没有搜索意义 |
| 状态评估 | 候选状态 | 可行性分数 | 只凭感觉，不给标准 |
| 回溯 | 失败的分支 | 回到父状态 | 一路写到底，不回头看 |

> **该停的时候** 如果题目本来就只有一条明显路径，比如固定抽取、简单翻译、标准格式填表，就别硬把它做成 ToT。搜索的成本只值得花在多路可比的任务上。

# Step-Back：先抽象，再回到具体题 {#abstract}
<!-- tag: abstraction -->

Step-Back 处理的是另一种失败：题目里的细节太多，把真正该用的原则盖住了。它要求模型先后退一步，先说出这个问题更一般地属于哪类规则，再拿规则回到具体问题。

## box soft
### 提示骨架
Step-Back 提示骨架复制
```text
先把这个问题抽象成一个更一般的规则问题。
写出适用的原则、它的边界条件，以及原题对应到哪一条规则。
再回到具体题目给出结论，不要一上来就堆细节。
```


## box info
### 看哪里会错
如果模型一直在题目表面绕圈，说明还没抽象出规则；如果抽象成空话，回不来具体条件，那也不行。好用的 Step-Back 要能从一般原则回到原题，不是只会说大道理。

| 位置 | 读什么 | 写什么 | 失手信号 |
| --- | --- | --- | --- |
| 抽象 | 具体问题里的条件 | 一般规则或原则 | 还在照着细节念，没有抽出规则 |
| 推理 | 规则 + 原题条件 | 具体结论 | 规则说了半天，回不到原题 |

# RAG 和 Generated Knowledge：一个接证据，一个接候选知识 {#grounding}
<!-- tag: grounding -->

这两种方法都在补“知识”，但来源不同。RAG 先去外部资料里找证据，再回答；Generated Knowledge 先让模型自己列出候选事实，再拿这些候选事实组织回答。前者是证据化，后者是候选化，不能混着用。

## box soft
### RAG
RAG 提示骨架复制
```text
只根据 <context> 里的条款回答。
如果条款不足，就直接说缺口，不要自己补猜。

<context>
[制度-12] 出差住宿按城市等级执行，超标部分原则上由个人承担。
[制度-18] 若因会议主办方指定酒店导致超标，需提交说明并由负责人审批。
</context>
```


## box info
### Generated Knowledge
生成知识提示骨架复制
```text
先列出 3-5 条与问题相关的候选事实或规律。
再只基于这些候选事实回答。
如果候选事实本身不可靠，就不要把它当作可追溯来源。
```


| 方法 | 知识从哪里来 | 适合什么 | 不能替代什么 |
| --- | --- | --- | --- |
| RAG | 外部文档、数据库、网页、笔记 | 事实敏感、需要引用来源的任务 | 不能替代证据追溯 |
| Generated Knowledge | 模型自己生成的候选知识 | 常识解释、写作构思、先起草 | 不能替代外部证据 |

> [!NOTE]
> **判断句**
> 如果题目要求“能追溯”，先接 RAG；如果只是先起草候选解释，再看 Generated Knowledge。候选知识可以帮你想，但不能自动变成事实。

# APE、Active Prompting 与 DSP：优化提示、挑样例、推方向 {#optimization}
<!-- tag: optimization -->

APE
先生成候选提示，再用评分函数筛选，最后对高分候选做语义相近的重采样。它优化的是整段提示，不是单个词。


Active Prompting
先测出哪些问题最不稳定，再把这些边界样例拿去人工标注。它优化的是示例分布，不是最稳的题。


DSP
先由策略模型生成方向刺激，比如关键词或限制条件，再把刺激和原始输入合并。它优化的是生成方向，不是把事实盖掉。

$$
 u(q)=1-\max_k p_k(q)
 $$
 这里 `u(q)` 是问题 `q` 的不确定性，`p_k(q)` 是第 `k` 个答案被采到的概率。分布越分散，不确定性越高，越值得拿去标注。

## box soft
### APE 的最小闭环
APE 工作流复制
```text
1. 生成候选提示
2. 用开发样例评分
3. 保留高分候选
4. 对高分候选做语义相近变体
5. 在独立测试集上复查
```


## box warn
### Active Prompting 的最小流程
Active Prompting 工作流复制
```text
1. 对同一批问题重复采样
2. 计算每题的不确定性
3. 选择最不稳定的问题
4. 人工标注后做 few-shot 示例
```


## box
### DSP 的判断点
DSP 里最容易出错的是刺激词盖过原始输入。它适合内容总结、内容创作和方向性很强的任务，但前提是刺激要收住，不然输出只剩“方向词”，没有原文里的真实限制。

# 完整案例：同一个报销问题，怎么走完一条可检查的流程 {#case-chain}
<!-- tag: case chain -->

把这几种方法放在一起看，最容易发现它们改的不是同一个位置。下面还是用“出差住宿超标能不能报销？”这道题：如果你直接问答案，模型可能会把“标准、例外、材料、结论”混在一行；如果你先抽象，再分解，再接证据，就能看见每一步在读什么、写什么、哪里会卡住。

## box soft
### 第一步：先抽象出规则层
这一步用 Step-Back。你先不问“能不能报”，而是问“报销判断一般看什么”。得到的不是答案，而是一个更高的观察框架：先看标准，再看例外，再看证据缺口。
Step-Back 先问法复制
```text
先把“出差住宿超标能不能报销？”抽象成规则问题。
请先说出报销判断通常要看哪几类原则，再说明原题分别对应哪一类原则。
最后再回到原题，不要一开始就下结论。
```


## box info
### 第二步：把规则拆成子问题
这一步用 LtM。抽象出来的规则再往下拆，就变成一串能依次回答的问题：标准是什么、例外是什么、证据缺什么、材料是否齐全。前一个答案必须能喂给后一个问题，否则就是散列出来的碎问句，不是分解。
LtM 子问题链复制
```text
1. 住宿报销的默认标准是什么？
2. 什么情况下可以超标报销？
3. 如果是会议主办方指定酒店，需要补哪些材料？
4. 现有信息是否足够支持“可以报销”的结论？
```


## box warn
### 第三步：把证据接进来
如果这题要求可追溯，就不能停在候选知识。RAG 会把制度条款放进上下文，再要求模型只根据条款回答。这样最后得到的有出处的判断。
RAG 证据块复制
```text
<context>
[制度-12] 出差住宿按城市等级执行，超标部分原则上由个人承担。
[制度-18] 若因会议主办方指定酒店导致超标，需提交说明并由负责人审批。
</context>

请只依据以上条款回答：能否报销、依据是什么、还缺什么材料？
```


## box
### 第四步：如果提示本身不稳，再做优化
如果你发现同一类报销题总是漏掉“例外”或“材料”，这时才考虑 APE 或 Active Prompting。前者把整段提示拿去评分，后者把最不稳定的问题挑出来补样例。它们修的是提示和样例，不是报销规则本身。

| 阶段 | 读什么 | 写什么 | 如果这一步错了 |
| --- | --- | --- | --- |
| Step-Back | 题目细节 | 一般规则与边界 | 只会背细节，不知道先看哪条原则 |
| LtM | 规则与依赖 | 有顺序的子问题 | 子问题之间断开，答案接不上 |
| RAG | 制度条款 | 可追溯依据 | 答案流畅，但没人知道依据来自哪里 |
| APE / Active | 提示和样例分布 | 更稳的提示或样例 | 开发集看着不错，换场景就掉 |

> **这个案例的用法** 以后遇到别的复杂题，也可以照着这条顺序走：先抽象出规则层，再把规则拆成子问题，接着看需不需要证据，最后才考虑提示优化和样例选择。

# 失败实验室：同一条任务链，三种常见坏法 {#failure-lab}
<!-- tag: failure lab -->

很多 prompt 看起来“差不多”，其实坏在不同位置。这个小实验还是用报销题来对照：你会看到同一条任务链里，哪一步坏了，后面会出现什么症状，应该回到哪一层修。

坏法一：只会列一串问题
把 LtM 写成“先问几个问题，再回答”。看起来像分解，实际上没有依赖顺序。结果是前一个答案没进下一个问题，拆题只是把句子切碎。


坏法二：把 ToT 写成群聊
让模型“想三种方案”却不写评分标准。这样做会让候选数量变多，但没有剪枝、没有回溯，最后只是堆出更多文本。


坏法三：把 Step-Back 写成空话
先加一个抽象词，再继续往下堆细节，问题还是会卡在原题和原则之间没接上。这里抽象层没有真正把结论带回原题。

| 坏法 | 表面症状 | 真正缺的东西 | 应该回哪一层 |
| --- | --- | --- | --- |
| LtM 只列问题 | 题目被切碎，但顺序松散 | 子问题之间的依赖 | 回到分解 |
| ToT 只会群聊 | 候选很多，判断很少 | 评分、剪枝、回溯 | 回到搜索 |
| Step-Back 只会拔高 | 抽象词很多，具体结论少 | 从原则回到原题的桥 | 回到抽象后的推理 |

## box soft
### 修法模板
从坏法改成可用 prompt复制
```text
1. 先判断题目卡在哪一层：分解、搜索、抽象、证据、样例或方向。
2. 只改这一层，不要把所有方法一起堆上去。
3. 写清输入、输出和下一步依赖。
4. 如果这一步仍然不稳，再往下检查评分、证据或样例。
```


## box
### 你在这里应该先看什么
先看对象，再看流程，再看失败症状。只要你能把一个坏 prompt 归类到某一层，就已经比“再多加一点提示”更接近问题了。接下来可以先修顺序，再修评分，再修证据，最后才修样例和方向。

> **练习钩子** 把你手里任意一条复杂 prompt 拿来，先判断它卡在分解、搜索、抽象、证据、样例还是方向，再只改一层。这样改出来的版本，通常比“整体重写一遍”更稳。

# 改写示范：同一题，三种层级的 prompt {#rewrite-demo}
<!-- tag: rewrite demo -->

很多人写复杂 prompt 的时候，第一反应是“把要求再说细一点”。这页更建议换一种方式：先判断问题在哪一层，再只把那一层写清楚。下面用同一个任务来演示：你要让模型帮你评估一篇文章值不值得继续读，或者一项方案值不值得继续做。

## box soft
### 版本一：普通问法
普通问法复制
```text
帮我分析这篇文章好不好。
```

这个问法最大的问题不是短，而是没有告诉模型先看什么。模型可能在摘要、观点、证据、结构之间乱跳，最后给你一段像样但难检查的评价。

## box info
### 版本二：LtM 改写
LtM 改写复制
```text
请把“这篇文章好不好”拆成 4 个子问题：
1. 文章要解决什么问题？
2. 它给出了哪些证据或例子？
3. 这些证据是否支持它的主张？
4. 读完之后，我应该继续读、放弃读，还是只保留其中一部分？
先逐题回答，再给总判断。
```


## box warn
### 版本三：Step-Back + RAG
抽象后接证据复制
```text
先把这个问题抽象成“如何判断一篇文章是否值得继续投入时间”。
请先说出判断这类内容时一般看哪几条原则，再根据给出的资料逐条核对。
如果资料不足，请说明缺口，不要直接下结论。
```


## box
### 这三版分别在改什么
版本改的是哪一层适用时机坏了会怎样


普通问法几乎没改只适合非常简单的判断输出会飘，原因不清楚
LtM把判断拆成子问题文章评估、方案评估、审稿子问题顺序乱，结论接不上
Step-Back + RAG先抽象原则，再接证据规则很多、事实敏感、需要引用依据抽象成空话，或证据没接上

## box
### 这条链怎么检查
如果你能从普通问法一路改到 LtM，再改到 Step-Back + RAG，说明你已经在做层级控制，而不是只是在加字。检查时只问三个问题：先看什么、后看什么、缺什么证据。

## box soft
### 遇到失败时回哪一步
如果模型一直跳答案，先回 LtM；如果规则说得很散，先回 Step-Back；如果结论对，但找不到依据，先回 RAG。这样修，比把整段 prompt 全改一遍更省力。

> **实践钩子** 把你最近要处理的一条真实任务写成三版 prompt：普通问法、LtM 版、Step-Back 版。写完以后只比较一个地方：每版到底让模型先看了什么。

# 选法演练：同一批任务，怎么落到不同的提示结构 {#decision-lab}
<!-- tag: decision lab -->

很多人学到这里，还是会在“到底该用哪种方法”上打转。最实用的判断方式看任务的第一道门槛在哪里：先后依赖、路径分叉、规则抽象、事实可追溯、提示不稳定，还是样例总挑错。下面这组例子把同一套判断法写得更细一点。

任务 A：政策问答
“住宿超标能不能报销？”这类题先看证据，所以优先 RAG。你要的把制度条款先放进上下文。


任务 B：方案比较
“三个选题哪个更值得做？”这类题有多条路，所以优先 ToT。要把候选、评分和回溯写清楚，不能只让模型自由发散。


任务 C：规则提炼
“这篇文章值不值得继续读？”这类题先抽象原则，再回到原文，所以 Step-Back 往往比直接下判断更稳。

## box
### 错法示范：把所有方法堆进一条 prompt
混搭 prompt复制
```text
请先抽象原则，再拆子问题，再找多个候选路径，再检索证据，再生成提示并评分。
```

这类写法的问题对象没分开。模型会知道你想要很多动作，但不知道第一步应该抓什么。

## box soft
### 修正示范：只改一层
只改一层的 prompt复制
```text
你现在只做一件事：先判断这个任务卡在哪一层。
如果是事实问题，先接证据；
如果是多路比较，先列候选状态并评分；
如果是规则被细节盖住，先抽象原则；
如果是后续子问题依赖前一步答案，先拆成有顺序的子问题。
不要同时展开所有方法。
```


| 任务信号 | 第一步 | 不该先做什么 | 修完后能检查什么 |
| --- | --- | --- | --- |
| 事实敏感 | 接 RAG | 不该先靠生成知识补猜 | 证据是否真的进入上下文 |
| 多路比较 | 接 ToT | 不该先写一堆没有评分的想法 | 候选是否有分数，是否需要回溯 |
| 规则提炼 | 接 Step-Back | 不该先堆细节 | 原则是否回到了原题 |
| 依赖顺序 | 接 LtM | 不该把子问题写成平铺的清单 | 前一步答案是否真的进入下一步 |

> **实践钩子** 挑一条你最近会用到的任务，把它先归到四种信号之一，再只改一层。改完以后，问自己一句：下一步能不能直接检查这层有没有做好。

# 方法对照 {#compare}
<!-- tag: compare -->

| 你看到的信号 | 先动哪一层 | 优先方法 | 第一动作 |
| --- | --- | --- | --- |
| 子问题有先后顺序 | 分解 | LtM | 列出有依赖顺序的子问题 |
| 同一题有多条可行路 | 搜索 | ToT | 生成候选、打分、剪枝、回溯 |
| 细节把规则盖住了 | 抽象 | Step-Back | 先写一般原则，再回到原题 |
| 事实必须可追溯 | 证据 | RAG | 先检索，再回答 |
| 能先起草候选知识 | 候选化 | Generated Knowledge | 先生成候选事实，再组织回答 |
| 提示本身不稳 | 优化 | APE | 生成候选提示并评分 |
| 样例总是选错 | 样例选择 | Active Prompting | 找不稳定的问题再标注 |
| 需要朝一个方向写 | 方向控制 | DSP | 先生成刺激，再合并输入 |

> **一句话分法** 分解看依赖，搜索看候选，抽象看规则，RAG 看证据，Generated Knowledge 看候选知识，APE 看提示候选，Active Prompting 看样例分布，DSP 看方向刺激。

# 路由检查：先判断任务信号，再选方法 {#route-check}
<!-- tag: route check -->

把这些方法真正用起来时，最容易出错的地方是第一步。你拿到一条任务时，不要先想名字，先看它的信号：是要先后依赖、还是要多路比较、还是要先抽象、还是要接证据。下面用一条“制度问答”来走一遍路由。

## box soft
### 原始任务
“出差住宿超标能不能报销？” 这类题表面上像一个简单问答，实际上先要看制度条款，再看例外，再看材料。任务信号是事实敏感，所以第一步应该接 RAG，而不是直接让模型自由发挥。

## box info
### 路由表
信号方法下一步错法症状


事实敏感RAG先放证据，再回答结论流畅，但找不到来源
顺序依赖LtM先拆子问题，再逐题回答子问题平铺，没有传递关系
多路比较ToT生成候选、评分、剪枝看起来热闹，实际没有搜索
规则被盖住Step-Back先抽象原则，再回到原题一直停在细节层

## box
### 可直接复制的路由 prompt
路由骨架复制
```text
请先判断这个任务卡在哪一层：事实、顺序、路径还是抽象。
如果是事实问题，先给出需要的证据，再回答。
如果是顺序问题，先拆成有依赖的子问题，再逐题回答。
如果是路径问题，列出候选方案并说明评分标准。
如果是抽象问题，先提炼原则，再回到具体题。
最后说明你为什么选这个方法。
```


## box warn
### 错误路由会发生什么
如果把事实题当成 LtM，模型可能会把子问题写得很顺，但证据仍然空着；如果把规则题当成 ToT，候选会很多，原则却没提出来；如果把顺序题当成 RAG，拿再多资料也解决不了前后依赖。问题的关键通常不在“方法不够多”，而在第一步选错了。

> **实践钩子** 下次你拿到一条复杂任务，先用这一页的四个信号做一次路由。只要能先说清楚“这题卡在哪一层”，后面的提示就会简单很多。

### 13 scenario-transfer

**Q:** 13. LtM 改写

Explain: 合格答案要把问题拆成依赖顺序明确的子问题，比如先问标准、再问例外、再问材料，最后再汇总结论。只写“请分步骤思考”还不够。

### 14 scenario-transfer

**Q:** 14. RAG 改写

Explain: 合格答案要先放入制度条款或检索到的片段，再明确要求“只根据资料回答”。如果资料不足，要让模型直接说出缺口，而不是自己补猜。

### 15 scenario-transfer

**Q:** 15. Step-Back 或 ToT 的选择

Explain: 如果卡点是规则被细节盖住，先用 Step-Back；如果卡点是可行路径很多、需要比较和剪枝，先用 ToT。答案里要说清“题目卡在哪里”，不能只报方法名。

### 16 synthesis

**Q:** 16. 费曼解释

Explain: 高分答案要说清：分解、搜索、抽象、证据、样例和方向本来就在改不同的层。题目有依赖顺序时用 LtM，路径很多时用 ToT，细节盖住规则时用 Step-Back，事实敏感时接 RAG，想先铺候选知识时用 Generated Knowledge，提示本身不稳时看 APE，样例选择错了看 Active Prompting，需要朝一个方向写时用 DSP。把 prompt 写长，只会把这些层混在一起。
