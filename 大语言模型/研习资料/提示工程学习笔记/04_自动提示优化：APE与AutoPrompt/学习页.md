# 学习目标 {#goals}
<!-- tag: goals -->

先抓住这个单元的边界：这一页不再讨论“怎么把问题写清楚”，而是讨论“怎么把提示本身和示例选择做成可评估的流程”。你需要先判断任务缺的是触发词、提示模板，还是更值得标注的样例，再决定用 AutoPrompt、APE 还是 Active Prompting。

先看对象AutoPrompt 关注触发 token，APE 关注整段提示，Active Prompting 关注最不稳定的样例。
再看流程候选生成、评分、重采样、人工标注、独立测试，这些步骤各自负责不同的检查点。
最后看边界没有评测集的提示优化，只是在换一种说法；没有不确定性度量的样例选择，也只是随手挑例子。

# 先抓主线 {#overview}
<!-- tag: overview -->

提示工程发展到这一层，很多问题已经不是“写一句更好的 prompt”能解决的。输入分布会变，少量样例会过拟合，任务也可能要求模型先找一个适合的触发模式。于是，提示优化开始变成三类动作：生成候选、评估候选、挑选更值得继续的候选。

这页的主线很简单：先看 AutoPrompt 怎么从数据里学触发词，再看 APE 怎么把整段提示放进搜索与评分循环，最后看 Active Prompting 怎么把标注预算花在最不稳定的问题上。三者都它们把人手里最容易漂的部分固定成了流程。

方法优化对象输入输出最怕什么


AutoPrompt触发 token / pattern标注样例、masked LM能激活目标答案的触发序列只在训练样例上触发
APE整段提示输入输出对、评分函数高分候选 prompt没有独立测试集
Active Prompting样例选择题库、多次采样、不确定性分数更值得人工标注的样例把稳定样例当边界样例

# 状态台账 {#ledger}
<!-- tag: ledger -->

自动提示优化表面上是在找更好的文字，实际上是在跟一组状态打交道：样例集、候选提示、评分函数、重采样结果和独立测试。只要其中一个状态错位，最后得到的“最优提示”就可能只是局部最优。

状态当前读什么会写什么错位症状


开发样例输入输出对候选评分的基准没有代表性，候选会偏到局部词面
候选提示任务描述、样例、触发词下一轮评估对象候选彼此太像，搜索没有分辨率
评分函数准确率、格式、人工偏好候选排名指标不一致，最后只剩看起来顺眼
独立测试集未参与生成的样例泛化验证开发集高分，测试集明显下滑
不确定性分布多次采样结果需要人工标注的样例把稳定问题当成难题，标注预算浪费

# 核心机制 {#mechanism}
<!-- tag: mechanism -->

这三个方法看起来不同，其实都在做同一件事：把“凭感觉写 prompt”拆成可观察的中间状态。AutoPrompt 观察触发词是否真的能把模型拉向目标标签，APE 观察候选提示在样例集上的分数，Active Prompting 观察模型在哪些问题上最不稳定。

## box
### AutoPrompt 的状态变化
输入是一个带掩码的位置，目标是找到一组触发 token，让模型更容易在 masked position 上给出目标答案。核心“找出哪些 token 对结果最敏感”。

## box
### APE 的状态变化
输入是示例对和评分规则，输出是候选 prompt 的排名。它关心的是整段提示在开发样例上的表现，所以必须把生成和评估分开，不然就会只优化出一段更像说明文的文字。

APE 评分骨架

```text
score(prompt) = 任务指标(prompt, dev_set) - 过拟合风险(prompt, dev_set)

其中：
- 任务指标可以是准确率、格式合规率或人工评分
- dev_set 是开发样例
- 过拟合风险在独立测试集上通常会暴露出来

选择规则：
1. 先在开发样例上排序
2. 再在独立测试集上复核
3. 如果 test 分数掉得明显，回到候选池重新采样
```

步骤读什么写什么坏掉时的症状


触发词搜索样例、梯度信号、候选 token更能激活目标输出的 token只学到训练集里的局部词
候选提示生成输入输出对、任务描述多条 prompt 候选候选彼此几乎一样，没有搜索意义
评分开发样例、准确率、格式合规率、人工判断候选分数没有独立指标，只剩“看起来更顺”
重采样高分候选、相近语义变体更稳的下一轮候选搜索停在第一轮，过早收敛
样例筛选多次采样结果、不确定性分布最不稳定的样例把简单题当难题，示例价值很低

# AutoPrompt 触发词追踪 {#trace}
<!-- tag: trace -->

AutoPrompt 的关键看触发 token 会不会把模型输出推向目标标签。为了检查这件事，可以把一个小样例的输入、触发词和输出变化排成追踪表。这里的重点是“写入了什么触发位置”和“模型行为变了没有”。

步骤输入结构检查点失败信号


1原句 + [MASK]模型原始预测基线就已经偏了
2原句 + 触发 token + [MASK]预测是否向目标标签移动触发词对预测几乎没影响
3换一批样例复测触发是否跨样例稳定只在一条样例上有效

> [!NOTE]
> **判断句**
> 如果一个触发词只能把单条训练样例推对，那它更像局部碰巧有效，不像可迁移的触发模式。

# AutoPrompt 小实验 {#autoprompt-demo}
<!-- tag: demo -->

AutoPrompt 的核心，可以用一个很短的分类任务看清楚：你先把原始句子放进模板，再在触发位置插入几个候选 token，然后看 [MASK] 的预测是不是朝目标标签移动。这个过程不像 APE 那样在整段提示上搜索，而是在几个很窄的插槽里找最敏感的词。

模板位置插入前插入后要看什么


触发位 1[MASK]great [MASK]预测是否偏向目标类
触发位 2great [MASK]great useful [MASK]多个 token 组合后是否更稳
换样例复测样例 A 有效样例 B / C 也有效有没有跨样例稳定性

AutoPrompt 检查顺序

```text
1. 准备一个带 [MASK] 的模板。
2. 在触发位尝试候选 token。
3. 观察目标标签的概率有没有上升。
4. 把同一组 token 换到另一批样例上复测。
5. 如果只对单条样例有效，回到候选池继续搜索。
```

## box info
### 它和 APE 的差别
APE 看的是整段提示有没有更稳；AutoPrompt 看的是少量 token 能不能把预测方向拉过去。

## box warn
### 它最怕什么
它最怕在一条样例上看起来很灵，但换一批输入就失灵，所以复测比单次命中更重要。

> [!NOTE]
> **失败后果**
> 如果你把触发词当成普通自然语言修辞去写，最后常常只是在局部样例上碰巧有效；一旦换一批任务，输出方向就会掉回去。

# 主动提示小实验 {#active-lab}
<!-- tag: lab -->

Active Prompting 的直觉可以再落一层：同一批问题先采样多次，再算它们的分歧程度。分歧越大，说明模型越不稳，也越值得人工标注。分歧很小的问题通常不该占预算，因为它们不会帮你学到新的边界。

$$
 u(q) = 1 - \max_k p_k(q)
 $$
 这里 `u(q)` 是问题 `q` 的不确定性，`p_k(q)` 是第 `k` 个答案被采到的概率。概率越分散，不确定性越高。

问题采样结果最大概率不确定性是否优先标注


Q1A, A, A, A, A1.000.00否
Q2A, B, A, C, B0.400.60是
Q3A, A, B, A, A0.800.20通常不优先

选择最不稳定样例

```text
1. 对每个问题重复采样 k 次。
2. 统计答案分布。
3. 按不确定性从高到低排序。
4. 选择前 N 个问题人工标注。
5. 把这些高不确定性样例作为 few-shot CoT 示例。
```

> [!NOTE]
> **失败后果**
> 如果你只挑模型本来就很稳的问题，标注预算会花在无效样例上，示例也不会补到真正的边界。

<!-- tag: bridge -->

# 具体案例 {#case-study}
<!-- tag: case -->

假设你在做客服工单分类，任务是把用户输入分成“退款、物流、账号、其他”四类。最开始你只有一段普通提示：“请判断这条消息属于哪一类”。这时模型容易被表面词汇带跑，尤其是遇到“退款”和“取消订单”这种边界很近的表达。

把这个任务交给自动提示优化后，流程就变成了另一种样子：先准备开发样例，生成多个候选提示，测开发集分数，再把高分候选放到独立测试集上复核。如果是 AutoPrompt，还会在模板里的触发位置上尝试不同 token，看看哪一组能更稳定地拉向目标标签。这个案例的关键“对对象、分数和验证路径都更清楚”。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

候选开发集分数测试集分数结论


Prompt A0.860.67开发集好看，但可能过拟合关键词。
Prompt B0.810.79开发和测试都稳，优先考虑。
Prompt C0.880.54像是学到了局部样例，不适合直接上线。

候选筛选伪代码

```python
candidates = generate_prompts(samples, n=8)
scored = []
for prompt in candidates:
 dev_score = evaluate(prompt, dev_set)
 scored.append((prompt, dev_score))

scored.sort(key=lambda item: item[1], reverse=True)
top_prompt = scored[0][0]
test_score = evaluate(top_prompt, test_set)

if test_score < scored[0][1] - 0.1:
 # 回到候选池，重新搜索，而不是直接把当前高分版本定稿
 candidates = resample_variants(top_prompt)
else:
 top_prompt
```

> [!NOTE]
> **失败后果**
> 如果你把开发集分数当终点，最后常常会得到一条“在例子里很好看、在新样本上就失灵”的提示。这个案例要记住的复核顺序。

再往前多看一步，开发集和测试集分开，不只是为了形式完整，而是为了避免你把“候选生成时见过的数据”当成真正的泛化能力。只要测试集还没看，候选就只是暂时好用；只有跨过复核，它才值得被当成可复用流程。

# 最小公式 {#formula}
<!-- tag: formula -->

APE 里最常见的决策可以写得很简单：在开发样例上先排分，再拿测试集复核。真正重要的你有没有把“排名依据”和“泛化依据”分开。

$$
 \text{choose}(p) = \arg\max_p \big(\text{score}_{dev}(p)\big)
 $$
 这里的 `score_dev` 是开发样例上的分数；真正收尾前，还要看 `score_test` 是否仍然稳定。若测试集明显下降，说明候选只是记住了开发集。

## box info
### 读这个式子要注意
它只负责“先选谁”，不负责“最终能不能上”。最终能不能上，要看独立测试集。

## box warn
### 最容易写错的地方
把开发集分数当成最终结论，或者把测试集拿来参与生成候选，都会把验证环节污染掉。

# 例子 {#example}
<!-- tag: example -->

看一个最小的 APE 闭环。先给模型几条输入输出对，让它生成候选指令；然后用统一的评分函数去比，最后把高分候选再做语义相近的变体搜索。这里最容易犯的错，是把“更会说话的提示”误当作“更有效的提示”。

APE 的最小闭环

```text
1. 准备开发样例：输入 x 和参考输出 y。
2. 让 LLM 根据样例生成多条候选指令。
3. 用准确率、格式合规率或人工评分给候选打分。
4. 选出高分候选，再生成语义相近的变体。
5. 在独立测试集上复核，确认不是只记住了开发样例。
```

AutoPrompt 的例子则更像“找钥匙”。你不给模型完整提示，而是让它在可插入的位置上搜索触发 token；这些 token 不一定是人类读起来最自然的词，但它们可能更能激活模型内部的目标模式。这里要特别小心：能激活，不代表可解释；能解释，也不代表稳定。

## box info
### AutoPrompt 适合看什么
它适合看 token 如何影响分类或触发答案，不适合把它当成自动写作文工具。

## box warn
### APE 适合看什么
它适合看提示整体是否更稳，不适合只在一小批样例上凭分数定终身。

# 方法对照 {#compare}
<!-- tag: compare -->

把三种方法放在一起看，最重要的是对象不同。AutoPrompt 优化的是触发 token；APE 优化的是完整提示；Active Prompting 优化的是样例选择。对象一变，评估方式、失败形态和后续动作都会变。

真正的决策顺序可以再说具体一点：先确认你要改的是模板、触发位还是样例池，再决定是做评分搜索、触发词搜索，还是不确定性排序。这个顺序比“自动优化”这个名字重要得多，因为名字不会告诉你该看哪一个失败信号。

方法优化对象典型流程常见误判


AutoPrompt触发 token搜索触发词 - 看预测变化 - 保留有效触发词把触发词当成人类可读的自然语言模板
APE提示模板生成候选 - 评分 - 变体重采样 - 独立测试只看开发集分数，不看泛化
Active Prompting示例样例多次采样 - 算不确定性 - 选择最摇摆的问题 - 人工标注优先选稳定题，导致示例信息量太低

## box info
### 决策树
如果任务的输出格式已经固定，但提示本身不稳，优先看 APE；如果只想找更敏感的 token，优先看 AutoPrompt；如果你更担心样例选错，优先看 Active Prompting。

## box warn
### 回退规则
开发集高分但测试集掉分，说明要回到候选池；触发词只在局部样例有效，说明要换一批样例；样例总是很稳，说明不确定性排序没有抓到边界。

> [!NOTE]
> **判断句**
> 如果你的任务已经有稳定模板，只想让系统自己挑样例，Active Prompting 更合适；如果模板本身需要改，先看 APE；如果你在找触发模式而不是模板，先看 AutoPrompt。

# 组合策略 {#sequence}
<!-- tag: sequence -->

真实工作里，很少有任务只属于一种方法。更常见的情况是：你先用 APE 把模板写稳，再用 AutoPrompt 试探关键触发位，最后用 Active Prompting 补上最不稳定的样例。这个顺序不是固定教条，但它比“随机换方法”更接近可复用流程。

举一个客服总结场景。假设你已经有固定输出格式，但模型经常把“退款”与“取消订单”混掉，同时少数边界样例又特别摇摆。先看模板是否该由 APE 重写，再看是否有少量触发 token 能拉住输出方向，最后用 Active Prompting 把最摇摆的工单补进示例池。三步合在一起，比单独调一个方法更稳。

顺序当前动作下一步信号失败后回到哪里


1APE 重写模板格式更稳但边界仍乱回到模板候选池
2AutoPrompt 搜索触发位局部样例变好回到触发词候选
3Active Prompting 补样例边界样例明显增加回到题库重算不确定性

组合策略伪代码

```python
if prompt_shape_is_clear:
 prompt = ape_search(dev_set, test_set=None)
if a_small_number_of_tokens_change_predictions:
 prompt = auto_prompt(prompt, masked_template)
if boundary_examples_are_missing:
 examples = active_prompting(pool, k=5)

run_final_check(prompt, examples, held_out_test_set)
```

> [!NOTE]
> **失败后果**
> 如果顺序搞反，你很容易先挑到“看起来会说话”的提示，再去找理由解释它为什么有效；那样最后得到的往往是局部有效，而不是可复用的工作流。

换句话说，方法名只是入口，真正决定质量的是检查顺序。先确认对象，再看候选，再看复核，最后才决定是不是可以把它变成稳定流程。只要这条线清楚，AutoPrompt、APE 和 Active Prompting 就不会被你混成一锅。

# 常见误区 {#mistakes}
<!-- tag: mistakes -->

第一类误区，是把自动优化理解成“让模型自己夸自己写得好”。如果没有独立评估，这不是优化，是循环自证。第二类误区，是把 AutoPrompt 和 APE 混成一类，结果把“触发词搜索”误写成“长提示生成”。第三类误区，是把 Active Prompting 当成随机挑错题，最后选出来的样例并不代表模型真正不稳的地方。

伪优化只在生成候选的同一批样例上打分，得到的高分很可能只是过拟合。
伪样例选择按题目表面难度挑样例，不按不确定性挑样例，示例价值会很低。
伪触发词把触发词当作自然语言修辞优化，最后常常偏离真正要激活的模式。

症状最可能原因先检查什么下一步怎么改


开发集分数高，测试集掉得快APE 过拟合是否有独立测试集扩大样例，重新评分
触发词看起来奇怪但有效AutoPrompt 找到了敏感 token是不是只在少量样例上成立复核跨样例稳定性
示例都很常规，模型还是不稳Active Prompting 没挑到边界样例不确定性分布是否真的拉开重新采样和排序

### 8 debug-checklist

### 9 diagnostic

**Q:** 9. 伪 APE 诊断

Explain: 错因：只在少量样例上自我证明；后果：很容易过拟合成更会说话的文字；修法：引入评分函数、保留开发集和独立测试集，并做语义相近变体搜索。

### 10 diagnostic

**Q:** 10. 样例选择诊断

Explain: 应优先选分歧大的问题。Active Prompting 的目标是找到模型最不稳定、最值得人工标注的边界样例，而不是重复标注已经稳定的问题。

### 11 scenario

**Q:** 11. 触发词误用

Explain: 先看它是不是只记住了训练样例，再看是否在独立样本上稳定，最后检查触发词是不是依赖了很窄的上下文模式。要把“局部有效”与“可迁移有效”分开。

### 12 synthesis

**Q:** 12. 费曼解释

Explain: 高分答案要说清三件事：优化对象不同，流程不同，验证方式也不同。AutoPrompt 盯触发词，APE 盯整段 prompt，Active Prompting 盯样例选择；这些都需要独立评估，不然只是在把文字改得更顺而已。

<!-- end-practice-group -->

<!-- practice-group: 五、迁移与解释 -->
<!-- quick-review: review 迁移题要说对象、流程和验证。 -->

### 16 scenario

**Q:** 16. 选择哪种方法

Explain: 如果目标是整段提示，先看 APE；如果想找更敏感的触发词，再看 AutoPrompt；如果开发样例本身不稳定，才考虑 Active Prompting。答题时要写出对象和为什么不是另外两种。

### 17 synthesis

**Q:** 17. 费曼解释

Explain: 高分答案应当把“自动提示优化”说成流程问题：对象触发词、提示模板和样例选择；过程必须包含候选、评分、复核；验证必须看独立测试或不确定性分布。只说“更自然”不够。
