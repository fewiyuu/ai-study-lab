知识增强、主动优化与多模态提示

把提示工程从“写好一句话”升级为外部知识、示例选择、方向刺激、集成验证和非文本输入的处理流程。

这一块先把提示工程的基础打稳：模型并不会自动知道你的真实意图，它只能根据当前 prompt 工作。你要学会把任务说清楚，把资料框出来，把结果格式定下来，再用输出反过来检查 prompt 是否需要改。

学完你应该能做什么

  1. 解释 RAG 为什么能缓解知识更新、私有资料和幻觉问题。
  2. 区分 RAG、生成知识提示、Active Prompting、DSP 和 Prompt Ensembling。
  3. 写出“只依据证据回答，证据不足则说明”的 RAG prompt。
  4. 为摘要、图表、关系链和数学题设计更可靠的提示结构。
  5. 根据任务失败原因选择补知识、补示例、补方向、补验证或补模态信息。

学习主线：从提示词到信息流程

workflow

文章后半部分的主线，是把提示工程扩展成增强语言模型的工作流。模型有参数记忆，但知识会过期，也可能编造来源；模型能写文字，但不擅长稳定计算、图表观察和关系遍历。此时 prompt 需要接入外部知识、示例筛选、方向刺激、集成验证或多模态输入。

这不是把所有技术堆在一起。你要先问任务缺什么：缺事实，优先 RAG；缺高价值示例，用 Active Prompting；缺输出方向，用 DSP；输出波动大，用集成；图片、图表或关系结构是核心，就把观察和结构写进 prompt。

RAG：把外部知识接到模型旁边

retrieval

原文说，大模型的参数记忆再强，也很难记住所有知识，尤其会遇到知识更新不及时、生成虚假信息、引用不存在来源等问题。RAG，Retrieval-Augmented Generation，检索增强生成，就是把外部资料检索出来，再放进 prompt 让模型基于证据回答。

典型 RAG 有两段：索引构建和检索生成。索引构建阶段准备资料、切块、向量化并存储；检索生成阶段根据用户问题找相关片段，把片段和问题组合成 prompt，再生成答案。

资料加载

文档、网页、数据库、笔记、日志或 API 结果。

分块

把长资料切成适合检索的小块。

向量化

把文本块转成 embedding，方便语义检索。

检索

根据问题找相关片段，可加重排。

生成

把证据和问题放进 prompt。

引用

输出依据和证据不足说明。

正确示例：RAG 问答 prompt
任务：只根据  中的资料回答问题。
若资料不足以支持结论，请回答“无法根据现有资料判断”。


[制度-12] 员工出差住宿标准按城市等级执行，超出标准部分原则上由个人承担。
[制度-18] 如因会议主办方指定酒店导致超标，需提交说明并由负责人审批。


问题：出差住宿超标能报销吗？

输出格式：
1. 结论：可以 / 不可以 / 视情况而定 / 无法判断
2. 依据：引用制度编号
3. 需要补充的信息：如有
判断句RAG 能提供证据入口，但检索错、资料错、prompt 约束弱时，答案仍会错。

RAG 与生成知识：外部证据和候选知识不是一回事

knowledge

原文在 RAG 后介绍 Generated Knowledge Prompting：不接外部知识，而是让模型先生成相关知识，再把这些知识作为上下文回答问题。它适合常识联想和初步分析，但不能当作可靠来源。

区别很简单：RAG 的知识来自外部资料，应该能追溯；生成知识来自模型自身，可能有幻觉。凡是涉及公司政策、医学建议、法律条文、最新事实或可引用来源，优先 RAG。常识解释、写作构思、低风险分析，可以用生成知识辅助。

方法知识来源适合风险
RAG外部文档、数据库、网页、笔记私有知识、最新事实、可引用问答检索不准、上下文污染
生成知识模型自己生成的候选知识常识推理、构思、低风险解释把幻觉当事实
你现在应该能做什么判断一个问题需要外部证据，还是只需要模型先唤起背景知识。

Active Prompting：把人工标注用在最不稳定的问题上

active

Active Prompting 借鉴主动学习。原文说它分四步：不确定性评估、选择、标注、推理。做法是对题库里的问题多次采样，找出模型最不稳定的问题，再让人工标注这些问题，作为 CoT 示例。

它的直觉很实用：如果人工标注预算有限，不要优先标注最简单的问题，而要标注模型最摇摆、最容易错的边界样例。这样 few-shot 示例更有教学价值。

流程示例：Active Prompting
候选题池：100 道应用题
1. 每道题让模型采样 5 次答案。
2. 统计答案分布。
3. 选择答案分歧最大的 10 道题。
4. 人工标注答案和关键推理步骤。
5. 把这 10 道题作为 few-shot CoT 示例。
6. 在独立测试集上评估。
判断句Active Prompting 不是随机选 few-shot 示例，而是优先选模型最不确定的问题。

Directional Stimulus Prompting：给输出一个方向刺激

dsp

DSP，Directional Stimulus Prompting，原文说它通过策略语言模型生成关键词或其他刺激信息，再和用户输入组合，指导下游大模型向某个方向生成。普通使用中，你也可以手工写“方向刺激”：关键词、重点实体、写作角度、必须覆盖的事实。

DSP 适合摘要、改写、内容创作和风格控制。风险是刺激信息会改变模型关注点，所以客观摘要、新闻事实、法律判断中要谨慎，刺激词不能覆盖或扭曲原文事实。

正确示例：DSP 摘要
任务：总结下面新闻，保持客观，不添加原文没有的信息。

方向刺激：
- 交易金额
- 涉及公司
- 监管风险
- 对用户的直接影响
- 后续时间节点

原文：
{news_text}

输出格式：
- 一句话结论
- 关键事实
- 风险与不确定性
- 后续观察点
判断句刺激信息只负责引导关注点，不能当作事实来源。

Prompt Ensembling：多个提示共同降低偶然性

ensemble

Prompt Ensembling 是提示层面的集成。你可以设计不同 prompt、不同示例、不同推理路径或不同验证方式，让它们对同一问题生成候选答案，再用投票、规则校验或验证器聚合。

它适合答案波动较大的推理、分类、抽取任务。但要注意：如果所有提示都缺证据，集成只会让错误更一致。有效集成需要候选路径有差异，聚合前要把答案标准化。

流程示例：差异化路径集成
对同一道题使用 4 条路径：
A. 直接列方程求解。
B. 先抽象数学原理，再求解。
C. 拆成子问题逐个解。
D. 写出 Python 表达式计算。

聚合规则：
1. 若 3 条以上一致，输出该答案。
2. 若不一致，检查单位、方程和代入验证。
3. 输出结论、分歧原因和验证结果。
判断句把同一个 prompt 重复问 5 次，不等于高质量集成；候选路径要有真实差异。

多模态提示：先观察，再推理

multimodal

很多任务的关键信息不在文字里，而在图片、截图、图表、扫描件、视频帧中。多模态提示要把“看见什么”和“推断什么”分开。模型应先描述可见信息，再基于可见信息回答问题。

图表题尤其要指定标题、坐标轴、单位、图例、趋势、异常点。截图题要说明看不清的内容，不要猜。专业影像、精确计数、小字号 OCR 等场景需要专用工具或人工复核。

正确示例：图表解读提示
请先观察图表，再回答问题。

观察步骤：
1. 读取标题、坐标轴、单位和图例。
2. 描述主要趋势和异常点。
3. 区分图中可见事实和你的解释。
4. 看不清的地方标注“不确定”。

问题：
{question}
你现在应该能做什么写出让模型先观察、再推理、最后说明不确定性的多模态 prompt。

图提示与数学提示：结构和验证要显式

graph math

基于图的提示适合关系复杂的问题，例如公司股权、任务依赖、知识图谱、人物关系、因果链。把实体、边、方向和属性写清楚，比把关系塞进长段落更容易检查。

数学推理提示则要强调变量、方程、单位、分步求解和验算。模型不擅长稳定计算，长推导不代表正确。能用计算工具验证时，不要只靠自然语言推导。

图提示模板

节点：A 公司、B 公司、C 项目。边：A 控股 B；B 投资 C。要求模型只沿给定边推理。

数学提示模板

定义变量、列已知条件、建立方程、分步求解、单位检查、代入验算、最终答案。

判断句关系复杂时先写节点和边；需要计算时先写变量、方程和验算。
