# 学习目标 {#goals}
<!-- tag: goals -->

这一页的边界很明确：它讲的是提示工程怎么接入知识、样例、方向、验证和多模态输入，而不是继续讨论“怎样把一句话写得更完整”。你需要先看任务缺哪一种支撑，再决定是接 RAG、做 Active Prompting、试 APE、加 DSP，还是把图表和数学结构先转成可观察的形式。

先看证据RAG 和生成知识提示都在处理知识，但一个来自外部资料，一个来自模型自己生成的候选知识。
再看样例Active Prompting 和 APE 都碰样例，但前者在挑最不稳定的问题，后者在搜索更好的整段提示。
最后看边界多模态、图和数学提示都先改观察方式，再改输出约束。

# 先抓主线 {#overview}
<!-- tag: overview -->

提示工程走到这一步，问题通常不再是“句子写得够不够顺”，而是“模型缺的是哪一类支撑”。有的任务缺事实，需要外部证据；有的任务缺好的示例，需要先找边界样例；有的任务缺方向，需要把关键词或刺激信息加进去；还有的任务输入本身就不是纯文本，提示结构就要跟着变。

因此，这一页的主线一条判断线：先看知识来源，再看样例怎么选，再看输出方向怎么控，最后看输入是不是图、关系或数学对象。只要对象分清，后面的流程就不会乱。

场景要补什么优先方法先看什么失败


事实更新快外部证据RAG检索不到、证据混进来、回答没引用依据
边界样例多高价值示例Active Prompting挑到的都是稳定题，标注预算浪费
提示本身不稳更好的整段 promptAPE只在开发集上更好，测试集掉分
输出需要被引导方向刺激DSP刺激词盖过原文事实
输入有图表/关系/数学结构观察方式多模态提示 / 图提示 / 数学提示把文本模板直接套进去

# 状态台账 {#ledger}
<!-- tag: ledger -->

这一页里的方法都带着状态在走：知识片段、候选知识、样例池、候选 prompt、方向刺激、验证结果、图像或数学输入。只要你把状态放错位置，最后的输出就会看起来像“优化过”，其实只是换了一种漂移。

状态读什么写什么错位症状


检索片段问题、索引、重排器上下文证据证据相关但答非所问，或完全找不到支撑
候选知识问题和模型先验可用于回答的常识句把模型自己编的内容当事实
不确定性分布多次采样结果高价值样例只挑最稳的问题，边界没被补到
候选 prompt样例和评分函数下一轮变体开发集高分，测试集掉分
方向刺激关键词、角度、限制下游生成约束刺激把事实盖住，内容开始跑偏
模态结构图、表、关系、公式可观察的提问方式把纯文本问法直接套到非文本输入上

# RAG：先把证据接上来 {#rag}
<!-- tag: retrieval -->

RAG 的作用很朴素：模型常常没有足够可靠的证据。把外部文档、网页、数据库或笔记检索出来，再放进 prompt，模型回答时就有了可追溯的依据。它最有价值的地方，“回答有证据来源可查”。

## box soft
### 最小流程
资料加载 - 切块 - 向量化 - 检索 - 组合上下文 - 生成答案。每一步都在读写不同状态，顺序错了，最后回答就会飘。

## box info
### 读这个流程时看什么
看检索是否真的拿到相关片段，看上下文里有没有噪声，看 prompt 有没有要求只依据证据回答。RAG 不是自动正确，它只是把“证据”这条路铺出来。

RAG 问答骨架复制

```text
任务：只根据 中的资料回答问题。
如果资料不足，直接说“无法根据现有资料判断”。


[制度-12] 员工出差住宿标准按城市等级执行，超标部分原则上由个人承担。
[制度-18] 若因会议主办方指定酒店导致超标，需提交说明并由负责人审批。


问题：出差住宿超标能报销吗？

输出：结论 / 依据 / 需要补充的信息
```

<!-- interactive: ['shape-tracer'] → keep as HTML -->

步骤读什么写什么错了会怎样


切块长文档适合检索的小片段块太大，检索命中但上下文塞不下
检索问题与索引相关片段找来一堆相关但不关键的片段
组合上下文证据和问题可回答的 prompt证据被问题淹没，回答仍会空转
生成证据和约束带依据的回答没加“只依据证据”的约束，就会补话

# 生成知识：先想，再回答 {#generated-knowledge}
<!-- tag: knowledge -->

Generated Knowledge Prompting 和 RAG 很像，但知识来源不同。RAG 把外部资料拉进来，生成知识提示则让模型先自己生成一组相关事实，再用这些候选知识去回答问题。它更像“先把脑子里的候选想法铺开”，而不是“先去查证”。

方法知识来源适合风险


RAG外部文档、数据库、网页、笔记私有知识、最新事实、可引用问答检索不准、上下文污染
生成知识模型自己生成的候选知识常识解释、写作构思、低风险分析把幻觉当成事实

生成知识两步法复制

```text
1. 先让模型列出 3 到 5 条与问题相关的候选事实。
2. 再要求模型只基于这些候选事实回答问题。
3. 如果候选事实本身不可靠，就不要把它当作可引用来源。
```

> [!NOTE]
> **判断句**
> RAG 的关键是证据可追溯，生成知识的关键是先把模型的候选认知显式化。两者都能帮回答，但只有前者适合做事实依据。

# Active Prompting：把标注预算花在最不稳的题上 {#active}
<!-- tag: active -->

Active Prompting 借鉴主动学习。它先对同一批问题多次采样，算出每个问题的分歧程度，再把最不稳定的问题交给人工标注。这样做为了把有限的标注预算花到模型真正摇摆的边界上。

$$
 u(q)=1-\max_k p_k(q)
 $$
 这里 `u(q)` 是问题 `q` 的不确定性，`p_k(q)` 是第 `k` 个答案被采到的概率。分布越分散，不确定性越高。

问题采样结果最大概率不确定性是否优先标注


Q1A, A, A, A, A1.000.00否
Q2A, B, A, C, B0.400.60是
Q3A, A, B, A, A0.800.20通常不优先

选择最不稳定样例复制

```text
1. 对每个问题重复采样 k 次。
2. 统计答案分布。
3. 按不确定性从高到低排序。
4. 选择前 N 个问题人工标注。
5. 把这些高不确定性样例作为 few-shot CoT 示例。
```

步骤读什么写什么失败信号


采样同一问题的多次输出答案分布只得到一个答案，说明没看出分歧
排序答案分布不确定性排名挑到最稳的问题，样例价值低
标注高不确定问题可用的 CoT 示例标注后仍没有覆盖边界
推理标注后的示例新的 few-shot 提示示例格式和任务不一致

# APE：把整段提示放进搜索循环 {#ape}
<!-- tag: ape -->

APE 关注的是整段提示。它先让模型生成多个候选 prompt，再用评分函数排出高低，最后对高分候选做语义相近的变体重采样。这里最重要的“候选和复核分开”。如果候选生成和评分混在一起，最后就只是在开发样例上把话说得更像。

APE 评分骨架复制

```text
score(prompt) = 任务指标(prompt, dev_set) - 过拟合风险(prompt, dev_set)
```

步骤读什么写什么坏掉的症状


候选生成样例和任务描述多条 prompt 候选候选彼此太像，没有搜索意义
评分开发样例候选分数没有独立指标，只剩“看起来顺”
重采样高分候选语义相近变体搜索停在第一轮，过早收敛
复核独立测试集泛化结论开发集高分，测试集掉得明显

候选筛选伪代码复制

```python
candidates = generate_prompts(samples, n=8)
scored = []
for prompt in candidates:
 dev_score = evaluate(prompt, dev_set)
 scored.append((prompt, dev_score))

scored.sort(key=lambda item: item[1], reverse=True)
top_prompt = scored[0][0]
test_score = evaluate(top_prompt, test_set)

if test_score < scored[0][1] - 0.1:
 candidates = resample_variants(top_prompt)
else:
 top_prompt
```

# 完整案例：客服政策问答的三段修复 {#case-study}
<!-- tag: case -->

假设你在做一套客服政策问答。用户问的是“出差住宿超标能不能报销”，而且公司制度会定期更新。最开始的提示只有一句：`请回答用户问题`。这个版本的问题很明显：它既没有把外部证据接进来，也没有约束回答格式，结果就是模型容易凭印象回答。

这类任务可以按三段修复。第一段先接 RAG，把制度片段检索进上下文；第二段如果政策解释需要方向，再用 DSP 写出“先给结论、再给依据、最后说缺口”的刺激信息；第三段如果你发现样例一直挑得不对，再用 Active Prompting 补边界样例，或者用 APE 把提示整段改稳。这个案例的重点每一步都只修一个问题。

阶段输入动作输出失败信号


RAG用户问题 + 制度库检索“住宿标准”“超标审批”片段带制度编号的证据上下文只检到相邻但不关键的段落
DSP证据上下文 + 回答任务写出方向刺激：结论 / 依据 / 缺口回答结构更稳刺激太强，把制度事实盖住
APE / Active Prompting开发样例 / 未稳样例搜索提示或补边界题更稳的 prompt 或示例池开发集很好看，测试集还是掉分

<!-- interactive: ['shape-tracer'] → keep as HTML -->

三段修复的最小 prompt复制

```text
任务：只依据 中的制度回答问题。
输出格式：
1. 结论
2. 依据（必须引用制度编号）
3. 需要补充的信息

如果证据不足，请直接说明“证据不足，不能定稿”。

问题：出差住宿超标能报销吗？
```

> [!NOTE]
> **失败后果**
> 如果只加长 prompt、不接证据、不固定输出结构，最后得到的仍然是“看起来认真”的回答。真正稳住结果的，是把每一步只负责一件事。

把这个案例继续往下做时，回到练习里的顺序也一样：先看对象是否是外部证据，再看输出结构是否固定，再检查样例和刺激有没有越界，最后才判断要不要回到 APE 或 Active Prompting。这个顺序一旦写成习惯，你在别的任务里也会知道该先动哪一层。比如财务制度问答、医疗常识摘要、产品更新说明，先补的都“更对的状态”。

检查点在这道题里看什么对应到哪一组练习出错后回哪里


证据制度编号 12 / 18 是否真的进上下文对象辨认 1、2回 RAG
结构结论 / 依据 / 缺口是否都出现状态追踪 5、8回 APE 或 DSP
边界答案有没有把“例外”漏掉失败诊断 9、10回检索或刺激设计
复核开发集和测试集是否一致迁移与解释 13、14、16回样例选择或提示搜索

你自己做这道题时，可以先把制度编号写到草稿最前面，再看结论有没有被这些编号支撑住。如果草稿里只有“可以报销”却没有编号，说明你还停在推测层；如果编号有了但结论还不收尾，就回到 RAG 的检索和上下文拼接。这个自查动作正好对应练习 1、5、9 和 13，能把判断、验证和回退连成一条线。

这就是这一页想让你形成的操作感：先找证据，再定结构，再看样例和方向，最后回到测试。只要这个顺序清楚，后面的练习题就给你下一次真实任务的检查表。

# DSP：给输出加一个方向刺激 {#dsp}
<!-- tag: dsp -->

Directional Stimulus Prompting 的思路是先生成关键词或方向信息，再把这些刺激和原始输入拼在一起，告诉下游模型“往哪儿说”。它很适合摘要、改写和内容创作，因为这些任务常常不缺事实本身，缺的是“该突出什么”。

## box soft
### 它适合的情况
摘要要抓重点，改写要守住角度，创作要保持风格。方向刺激把这件事显式写出来，模型就不必自己猜。

## box warn
### 它要小心的情况
如果刺激词覆盖了原文事实，摘要就会开始偏；如果任务本来要求客观，刺激就不能比事实更强。

DSP 摘要骨架复制

```text
任务：总结下面新闻，保持客观，不添加原文没有的信息。

方向刺激：
- 交易金额
- 涉及公司
- 监管风险
- 对用户的直接影响
- 后续时间节点

原文：
{news_text}
```

步骤读什么写什么风险


训练策略模型输入和刺激标注对可生成刺激的 policy LM刺激太泛，等于没引导
生成刺激用户输入关键词或方向刺激和任务不相关
组合输入原始输入与刺激新的 prompt刺激盖住事实
再训练反馈和偏好更稳的策略模型只记住局部偏好

# 多模态提示：先改观察方式，再改指令 {#multimodal}
<!-- tag: multimodal -->

当输入不再是纯文本，提示工程就不只是“怎么问”，而是“先看什么、怎么对齐”。图像、表格、截图、版式、图表和流程图都要求你先把观察对象说清楚，再决定回答格式。这里最容易犯的错，是把图像任务写成普通文本抽取题，结果模型只会复述看见的字，不会真正比较结构。

输入类型先看什么提示里要写什么常见失败


图片主体、关系、文本、动作请先描述可见事实，再回答问题只抄 OCR，不看图中关系
表格列名、行、比较维度按列对比，先给结论再给依据只做逐行复述，没有比较
图表趋势、极值、拐点先说变化方向，再说数值证据只读标题，不读走势
关系图节点、边、路径先列关键节点，再解释路径含义把图当成普通段落阅读

> [!NOTE]
> **判断句**
> 多模态提示的重点是把“看”和“答”分开。先把可见结构说清楚，再让模型做推理，通常比直接问结论稳得多。

# Prompt Ensembling：让多个提示互相校验 {#ensemble}
<!-- tag: ensemble -->

提示集成的思路很直白：单个提示会偏，多个不同角度的提示一起看，就能把偶然性压下去。它不追求某一个提示最花哨，而是追求多个提示对同一问题给出一致方向。这个方法常见于复杂推理、验证和稳健性检查。

做法看什么优点风险


多提示投票答案一致性压低单个提示的偶然性提示太像，投票没信息量
步骤验证中间推理是否站得住能发现某一步出错只看最终答案，错因藏起来
自然语言推断式校正前后结论是否互相支持适合做一致性修正把修正当成新的事实来源

集成判断骨架复制

```text
1. 用三个不同角度的提示分别回答。
2. 比较中间步骤有没有冲突。
3. 如果最终答案一致但理由冲突，先修理由。
4. 如果答案本身不一致，回到提示设计重新拆问题。
```

# 图与数学：把结构写进提示 {#graph-math}
<!-- tag: graph / math -->

图提示和数学提示都在提醒一件事：有些任务的核心结构。图里重要的是节点、边和路径；数学题里重要的是条件、变量、约束和中间变换。提示如果不先写结构，模型就容易只在表面措辞上打转。

## box soft
### 图提示先看什么
节点是什么、边代表什么、路径如何比较。把这些写出来，再问“最短路径”“最关键节点”之类的问题，模型才知道要比较什么。

## box info
### 数学提示先看什么
条件、目标、符号和推理顺序。题目越容易混淆条件，越要把已知量和要求量先分开列出来。

$$
 \text{已知条件} + \text{中间变换} \rightarrow \text{结论}
 $$
 这里的关键每个条件都要能落到一个可检查的变换步骤里。条件没写清，后面就会把答案算在错误对象上。

对象先写什么再问什么常见错误


图节点、边、路径谁连接谁、哪条路径更短只说“图里有什么”，不说“关系是什么”
数学题已知量、未知量、约束先推哪一步、用哪个恒等式直接套公式，不看条件

# 策略选择 {#strategy5}
<!-- tag: decision -->

把这一页收起来时，最实用的判断顺序。先问任务缺什么，再看应该把哪一种状态接进来。这个顺序能避免你把所有问题都写成“请更详细回答”，也能避免你把本来需要证据的任务写成风格控制题。

如果任务缺...先用什么再检查什么不要先做什么


外部证据RAG检索是否命中、证据是否足够先让模型自由发挥
边界样例Active Prompting不确定性排名是否真的拉开先挑最稳的题
整段 prompt 质量APE测试集是否保持稳定只看开发集分数

### 5 ordering

**Q:** 5. RAG 的最小顺序

Explain: 提出问题 - 检索片段 - 组合上下文 - 生成答案。顺序要对，因为证据必须先进入上下文，回答才能跟着证据走。

### 6 choice

**Q:** 6. APE 先看什么

- [ ] 开发样例
- [ ] 独立测试集
- [ ] 人工偏好标签

Answer: 开发样例

### 7 debug-checklist

**Q:** 7. 多次采样看什么

- [ ] 答案分布是否分散
- [ ] 哪一题最容易被模型背出
- [ ] 不确定性是否能把边界题排出来
- [ ] 是否只采到一个答案

Explain: 应优先看答案分布是否分散、不确定性是否能把边界题排出来。只采到一个答案说明没看出分歧，不足以支撑样例选择。

### 8 short-fill

**Q:** 8. 方向刺激的读写关系

Explain: DSP 读取原始输入，再写入方向刺激，最后把两者组合成新的 prompt。只要刺激信息盖过原始输入，输出就容易跑偏。

### 9 diagnostic

**Q:** 9. RAG 没有依据

Explain: 先怀疑检索和上下文拼接。错因可能是没检到相关片段，或者 prompt 没有限制“只依据证据回答”；后果是回答会自己补话；修法是检查检索结果和证据约束。

### 10 choice

**Q:** 10. DSP 过度刺激

- [ ] 刺激把事实盖住了
- [ ] 检索不到任何片段
- [ ] 采样只有一个答案

Answer: 刺激把事实盖住了

### 11 scenario

**Q:** 11. 多模态问法

Explain: 因为图表的核心趋势、极值、拐点和比较关系。提示如果没有先说明要看哪些结构，模型就会退回成普通文本复述，忽略图的关键信息。

### 12 debug-checklist

**Q:** 12. 集成提示的用途

- [ ] 用多个角度的提示互相校验答案
- [ ] 只保留最花哨的那个提示
- [ ] 答案一致但理由冲突时先修理由
- [ ] 只看最终答案，不看过程

Explain: 应选“用多个角度的提示互相校验答案”“答案一致但理由冲突时先修理由”。集成的意义是压低偶然性，不是把提示堆得更多。

### 13 scenario

**Q:** 13. 方法选择

Explain: 如果格式已经固定、但提示本身不稳，先看 APE；如果漏的是边界样例，才看 Active Prompting；如果需要把输出方向写得更清楚，再考虑 DSP。答题时要写清对象和回退顺序。

### 15 choice

**Q:** 15. 图提示先看什么

- [ ] 节点、边、路径
- [ ] 情绪、语气、修辞
- [ ] 只要一个标题就够了

Answer: 节点、边、路径

### 14 synthesis

**Q:** 14. 费曼解释

Explain: 高分答案要说清三件事：对象不同，流程不同，验证方式也不同。RAG 在补证据，生成知识在补候选认知，Active Prompting 在挑边界样例，APE 在筛整段 prompt，DSP 在控制方向；这些都不是“更长的 prompt”能替代的。关键是先识别缺口，再决定补哪种状态。

### 16 synthesis

**Q:** 16. 费曼解释

Explain: 高分答案要说明：证据解决的是事实可追溯，样例解决的是边界覆盖，方向解决的是输出侧重点，验证解决的是稳定性，模态解决的是观察对象。混着用会导致流程断裂，所以要把它们拆成不同状态，出错时回到对应环节，而不是继续加长 prompt。
