第1讲：一堂课搞懂 LLM 大模型的原理

网页聊天界面看起来像“输入问题，模型回答”。P1 要拆掉这层界面感：LLM 每一步看到的是一串 token，它根据已经出现的内容，为 vocabulary 里的候选 token 打分，再选出下一个 token。完整回答只是这个动作反复执行后的结果。

课程地图 下一节：剖析大型语言模型

读完本页应该能做什么

  • 用“prompt -> token -> 概率分布 -> 采样 -> 追加 token”解释一段回答怎样长出来。
  • 区分 token、token id、vocabulary 和 Hugging Face token，避免把同名词混用。
  • 判断 chat template、system prompt、历史消息和 RAG 证据分别改变了模型看到的哪一部分输入。
  • 读懂最小 Hugging Face 路线：tokenizer 管文字和编号，model 管概率，generate 管循环。

先把本讲的检查点立起来

goals

P1 的主线

这一讲的核心是看懂大语言模型怎样把一段文字回答拆成可执行的步骤，背模型名字放在后面。用户看到的是聊天框，模型内部处理的是 token 序列；用户等待的是完整回答，模型每一步真正产出的是下一个 token 的概率分布。

所以学习顺序可以很朴素：文字先被 tokenizer 切成 token，再映射成 token id；模型根据已经出现的 token 计算下一个位置的分数；生成策略从这些分数里选一个 token；新 token 追加回上下文，循环继续，直到遇到停止条件。

本页只要求你掌握这些

  • 用一个流程图解释 prompt 如何变成回答。
  • 说明 token、vocabulary、token id 和 Hugging Face token 的差异。
  • 判断一个输出异常更可能来自 prompt、chat template、上下文、采样还是证据不足。
  • 能读懂 tokenizer、model、generate 和 pipeline 在最小代码里的分工。

现在你应该能做什么：看到一个 LLM 回答时，先按“输入表示、模型打分、生成策略、上下文更新”四层来问问题。

全流程地图

trace

先把一轮生成放在同一张表里。这里的“模型”可以先当成一个已经训练好的函数，它接收 token id 序列，输出下一个位置对 vocabulary 中每个候选 token 的分数。真实系统会有更多工程包装，但这张地图足够定位大部分入门误解。

1用户文字整理成 prompt，聊天模型还会套入角色模板。system、user、assistant 的边界是否清楚。

2tokentokenizer 把文字切成模型认识的单位。中文、空格、特殊符号是否被切成意料之外的片段。

3token id每个 token 被映射成 vocabulary 里的编号。编号只是索引，不携带人类可读含义。

4logits / probability模型给下一个 token 的所有候选项打分，再转成概率。最高分 token 与采样策略可能不一致。

5next token贪心、top-k、top-p、temperature 等策略选出一个 token。随机性、重复、跑题常从这里放大。

6更新后的上下文新 token 被追加，下一轮继续。超出上下文窗口时，旧信息可能被截断。

token 边界

tokenizer

同一个词，两个 token

本讲最容易混的地方是 token。LLM 里的 token 是模型处理文本的基本单位，可以是一个中文字、一个英文词片段、一个标点、一个空格模式，也可以是特殊控制符。Hugging Face 网站或 API 里说的 access token 则是权限凭证，和模型切词没有关系。

vocabulary 是模型认识的 token 清单。token id 是 token 在这张清单里的编号。编号方便模型计算，但编号本身没有语义顺序，不能说“id 越大含义越复杂”。

术语这一讲怎么用不要误会成

token模型处理文字的片段单位。总是一个完整词或一个中文字。

vocabulary模型可选 token 的全集。自然语言里的全部词语。

token idtoken 在 vocabulary 里的整数索引。带有大小顺序的语义标签。

Hugging Face token访问模型仓库或 API 的权限凭证。LLM 生成时的文本 token。

最小观察代码

实际学习时，不需要先训练模型。先拿 tokenizer 做 encode / decode，就能看到“文字”和“模型输入”之间的边界。

tokenizer encode/decode复制

from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")
text = "LLM 会逐步预测下一个 token。"

ids = tokenizer.encode(text)
tokens = tokenizer.convert_ids_to_tokens(ids)

print(ids)
print(tokens)
print(tokenizer.decode(ids))

这段代码检查三件事：切出来多少个 token，token id 是什么，decode 回来后是否仍然是原文。之后遇到“模型为什么不懂我的格式”，先看 tokenizer 输出往往比猜模型性格更有效。

生成循环

next-token

大语言模型看起来一次写出一整段，内部更接近一个循环：根据已有序列预测下一个 token，选出它，追加，再预测下一个。这个动作也叫自回归生成，重点是“新输出会成为下一步输入的一部分”。

t_i 是下一个 token，前面的 token 构成条件；z_i 可以理解为模型给 vocabulary 中每个候选 token 的原始分数。

停止条件很关键

模型本身只会继续给下一个 token 打分。何时停下来，通常要靠最大 token 数、结束符、停止词、工具调用协议或应用层规则。很多“回答被截断”和“模型停不下来”的问题，根源常在停止条件配置，而不在模型知识本身。

现在你应该能做什么：解释“一个回答”为什么其实是一连串 token 决策，并能把重复、截断、随机性归到生成参数里检查。

聊天模板

prompt

聊天要先套模板

聊天模型通常在训练时见过固定格式：system 指令、user 消息、assistant 回复之间有特殊标记。推论时如果没有使用同样或兼容的 chat template，模型看到的输入格式就会陌生。结果可能表现为角色错乱、把用户问题复述成答案、或者漏掉系统约束。

apply_chat_template复制

from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")
messages = [
 {"role": "system", "content": "你是一个严谨的课程助教。"},
 {"role": "user", "content": "用三句话解释 LLM 的生成流程。"},
]

prompt = tokenizer.apply_chat_template(
 messages,
 tokenize=False,
 add_generation_prompt=True,
)
print(prompt)

模板改变的是模型实际看到的输入

如果只在应用层保存了“system: 你要严谨”，但没有把它按模型需要的格式写进 prompt，模型就不会真正看到这条约束。反过来，如果历史消息堆得太长，新的关键要求可能被挤到上下文窗口外。

角色用途排错时看什么

system设定回答规则、身份和边界。是否进入最终 prompt；是否被后续用户指令冲淡。

user表达当前任务和输入材料。任务是否具体，证据是否完整。

assistant历史回答或生成起点。历史中是否有错误结论被继续引用。

上下文与幻觉

context

LLM 的回答只根据当前上下文和模型参数产生。它不会自动知道你心里省略的资料，也不会天然区分“我记得的模式”和“这次输入里有证据”。这就是幻觉问题的基本入口：当模型没有足够证据时，它仍然会按语言模式补出看起来顺畅的内容。

幻觉先按证据问题处理

不要一上来把幻觉归结为“模型坏了”。更可操作的检查顺序是：用户问题是否需要外部事实；上下文里有没有足够证据；证据是否被明确引用；生成参数是否太松；应用层是否要求模型在不确定时承认不知道。

这也是后续 RAG、工具调用、评估和安全章节会继续展开的原因。P1 只要先记住：LLM 可以先看成基于上下文生成下一个 token 的模型，事实校验要靠证据、工具和评估链路补上。

Hugging Face 最小观察路线

implementation

本讲不要求你训练模型，但可以用 Hugging Face 的工具把“tokenizer 管表示，model 管打分，generate 管循环”看清楚。下面代码的目的在于把各层职责分开，速度优化先放到后面的部署主题里。

代码
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

name = "Qwen/Qwen2.5-0.5B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(name)
model = AutoModelForCausalLM.from_pretrained(name, torch_dtype="auto")

messages = [
 {"role": "system", "content": "你是一个严谨的课程助教。"},
 {"role": "user", "content": "用三句话解释 LLM 的生成流程。"},
]

prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
inputs = tokenizer(prompt, return_tensors="pt")

output_ids = model.generate(
 **inputs,
 max_new_tokens=96,
 temperature=0.7,
 top_p=0.9,
)

print(tokenizer.decode(output_ids[0], skip_special_tokens=True))

代码对象负责什么出问题时的症状

AutoTokenizer文字、token、token id 之间的转换；聊天模板。特殊标记错、角色格式错、切词意外。

AutoModelForCausalLM给下一个 token 的候选项打分。模型类型不匹配、权重加载失败、设备或精度问题。

generate循环采样、追加 token、处理停止条件。重复、截断、太随机、太保守。

pipeline把 tokenizer、model、生成参数包装成易用接口。方便但隐藏细节，不适合第一次排错。

排错演练

failure

下面三个案例都来自 P1 的基本模型观：模型只根据当前上下文预测下一个 token。点击案例时，先不要急着改模型，先找“模型实际看到的东西”和“你以为模型看到的东西”之间的差距。

角色混乱chat template、system/user/assistant 边界。模型实际看到的格式可能和训练格式不一致。

事实编造上下文证据、RAG 返回内容、不确定性规则。模型会补齐语言模式，不会自动查证。

回答太短或突然停下max_new_tokens、停止词、结束符。停止条件可能先于完整回答触发。

重复或绕圈temperature、top-p、重复惩罚、prompt 约束。生成循环会把前一步输出继续喂给下一步。

微型排错记录

debug

把 P1 的概念用于真实代码时，最有用的习惯是留下排错记录。下面这条记录模拟一个常见问题：你明明写了“请用三句话解释”，模型却输出一长段，甚至开始补充无关背景。先把一次请求拆开看。

最小检查代码

打印 prompt 与长度复制

prompt = tokenizer.apply_chat_template(
 messages,
 tokenize=False,
 add_generation_prompt=True,
)
inputs = tokenizer(prompt, return_tensors="pt")

print(prompt)
print(inputs["input_ids"].shape)
print(tokenizer.convert_ids_to_tokens(inputs["input_ids"][0][-20:]))

这段代码不负责改答案，它只负责让你看见输入。最后 20 个 token 可以帮助你确认生成起点附近有没有正确的 assistant 标记，以及当前问题是否真的靠近序列末尾。若这里已经错了，继续调 temperature 只会让问题更难定位。

检查动作你要看到什么如果不符合，下一步怎么改

打印最终 promptsystem、user、assistant 边界清楚，末尾有生成起点。改用模型 tokenizer 自带的 apply_chat_template。

看 input_ids.shape长度没有逼近上下文窗口，当前问题没有被挤掉。压缩历史，移除无关轮次，保留当前任务和证据。

看末尾 token最后一段 token 对应当前用户问题或 assistant 起始标记。检查消息顺序，避免把回答模板放进 user 内容。

收紧生成参数长度、随机性和停止条件与任务匹配。降低 temperature，设置 max_new_tokens 和明确停止规则。

这条记录的价值在于把“模型不听话”改写成可验证的问题。你能看到 prompt，就能判断模型是否真的读到了你的约束；你能看到 token 长度，就能判断旧消息是否占掉窗口；你能看到生成参数，就能解释为什么回答会变长、重复或发散。P1 讲的 token 和 next-token 预测，最终要服务于这种排错能力。

现在你应该能做什么：遇到输出异常时，先打印最终 prompt 和 token 序列，再判断模型到底读到了什么。

带走一条判断线

LLM 的“会说话”可以先拆成一个机械过程：把上下文变成 token id，模型对 vocabulary 里的下一个 token 打分，生成策略选一个 token，再把它接回去。这个解释不覆盖所有工程细节，但足够支撑你继续学 tokenizer、Transformer、RAG、评估和部署。

现在你应该能做什么：不用玄学词汇，给别人讲清楚一段 LLM 回答是怎样一步一步长出来的。
