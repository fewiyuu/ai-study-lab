# 先把任务说清 {#goals}
<!-- tag: 目标 -->
<!-- layout: grid2 -->

## box soft
### P4 先解决什么问题
生成式 AI 的评估，先别急着给一个总分。你要先知道自己是在新闻摘要、开放式问答、语音合成、代码修复，还是安全对抗。任务一变，输入材料、输出格式、标准答案和风险代价都会变。
这一讲真正重要的地方，是把一个分数拆成可以检查的部件。benchmark 负责组织测试，指标 负责打分，aggregation 负责合成结果，评委 负责处理没有唯一答案的场景，安全测试负责看攻击面。只要其中一环错位，分数就会变成漂亮但误导人的数字。

## box
### 学完后要能做四件事
- 判断一个排行榜量的是任务能力，还是格式遵守、提示词设计或资料污染。
- 把自动指标和人工评审分开说清楚，不把“分数高”直接等同于“真实可用”。
- 把泄题、提示词敏感、jailbreak、提示词注入和 bias 放进同一张评估表里。
- 给真实系统补上速度、成本、下限和失败样本，同时看平均值。

角色真正关心的问题常见误区该补的证据


模型使用者这个模型适不适合我的任务直接拿总榜判断自己的样本、小规模对照、失败样本
模型开发者哪个版本更值得继续优化只看平均分下限、拒答率、耗时和成本
标注 / 评审者怎么把主观判断变成稳定规则把评分规则说得太抽象明确定义、盲测和一致性记录
安全检查者模型会不会被绕过或污染只测单轮聊天攻击预算、间接注入和泄题检查

# 评估链路 {#map}
<!-- tag: 链路 -->

先把一轮评估写成一条链。task 决定你到底想量什么，benchmark 决定你拿什么材料去量，提示词决定模型看到什么，指标 决定怎么打分，aggregation 决定怎么合成结果，最后 decision 才能说服别人选模型或改模型。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

步骤读什么写什么错位症状


task业务目标和风险边界能力定义任务没定清，后面所有分数都可能测偏。
benchmark测试资料、输入格式、参考答案可比的评估流程资料不同，模型之间不可比。
指标模型输出和标准 / 偏好局部分数只奖励格式或词面重合，忘了真实目标。
aggregation一批样本分数总分、分位数、拒答率平均值盖住最差样本。
decision总分、失败样本和成本上线、回炉、再测只看榜单，不看部署代价。

# 术语边界 {#terms}
<!-- tag: 术语 -->
<!-- layout: grid2 -->

## box
### 先把几件事拆开
术语这一讲里的意思常见误解


benchmark一整套评估资料和流程，也可口语上指测试集。只把它当成题库。
ground truth参考答案、参考偏好或规则标签。以为所有生成任务都有唯一标准答案。
指标把输出变成分数的规则。把分数当成任务本身。
aggregation把样本分数合成总分的方法。默认平均一定合理。
人类评委让人类按 评分规则 评分。以为主观判断不需要校准。
LLM 评委让语言模型当评委。以为模型评委天然比人类客观。
verifier专门检查答案、推理或偏好的模型。以为它只是换了提示词的聊天模型。
资料泄露测试题或相似内容进入训练资料。模型背不出原题就一定没见过。
提示词注入把恶意指令塞进用户输入、网页或文档。只把聊天框里的攻击当作全部风险。

## box soft
### 先分开看，再回到任务
这一讲会碰到 BLEU、ROUGE、BERTScore、MOS、人类偏好、verifier、jailbreak 和 提示词注入。它们各管一类问题：内容、主观偏好、安全和泄题。
先把“评分规则”“评委”“攻击面”拆开，再回头看后面的模型训练、对齐和红队测试，会稳很多。
看到一个指标名，先说它在量什么，再说它没覆盖什么。

# 自动指标：先看它奖励什么 {#metric}
<!-- tag: 指标 -->
<!-- layout: grid2 -->

## box
### exact match、BLEU / ROUGE、BERTScore 的边界
`exact match` 要求输出和标准答案完全一致，适合固定格式、分类题和选择题。但它会把“格式不合格”和“内容不对”混在一起，模型只多说一句解释就可能吃亏。
`BLEU` 和 `ROUGE` 主要看词汇或片段重合，常用于翻译和摘要。它们能奖励和参考答案接近的表达，也会误伤同义改写，甚至让“看起来很像”的答案拿到不错的分数。`BERTScore` 更偏语义相似，适合看改写是否接近参考含义，但它还是会受任务类型、所用模型和对齐方式影响。

## box soft
### 最常见的坑
- 把格式遵守混进知识评估，最后测到的是“会不会按要求只输出一个字母”。
- 把词面重合混成事实正确，最后鼓励抄句式而不是解决问题。
- 把语义相似混成可部署质量，最后忽略了错误事实、漏点和拒答能力。
指标的关键，不在于高级不高级，而在于它是不是对着任务目标。

`score(model) = Aggregate(metric(output_i, reference_i))`
这里 `output_i` 是第 `i` 个样本的模型输出，`reference_i` 是对应的参考答案或偏好，`Aggregate` 负责把一批样本分数合成总结果。这个式子只适用于有清楚参考答案的场景。没有唯一参考时，指标 可能换成人类偏好、LLM 评委分数、拒答率或安全通过率。

指标最适合什么容易漏掉什么部署时要补什么


exact match固定格式、选择题、分类任务解释、改写、格式差异先做答案规范化，再单独看格式遵守。
BLEU / ROUGE翻译、摘要、文本相似度事实错误、同义改写、覆盖不足加事实核验和人工抽检。
BERTScore语义相近的生成任务任务需求、事实正确性、长度控制结合任务目标和失败样本看。
拒答式评分高风险问答、医学、法律、开放域 QA“会猜”与“会承认不知道”的差异把答错和拒答分开记分。

评估聚合伪代码复制

```python
def aggregate(scores, mode="mean"):
 scores = list(scores)
 if not scores:
 return 0.0
 if mode == "mean":
 return sum(scores) / len(scores)
 if mode == "min":
 return min(scores)
 if mode == "p10":
 index = max(0, int(0.1 * (len(scores) - 1)))
 return sorted(scores)[index]
 return sum(scores) / len(scores)
```

平均分适合看总体趋势，但它会把少数灾难性失败压得很轻。下限和分位数能把最差情况拉出来，适合报站、医疗提醒和其他不能接受小概率暴走的场景。高风险任务里，拒答也要单独记，不要把“没答”直接算成“没能力”。

# 案例与分桶 {#example}
<!-- tag: 案例 -->
<!-- layout: grid2 -->

## box
### 拿摘要模型做一轮完整评估
假设你有一个新闻摘要模型。你想知道它是真的会摘要，还是只是会复述开头和标题。更稳的做法，是先把样本分桶：短新闻、长新闻、标题党新闻、数字密集新闻、跨领域新闻。每个桶都用同一套 提示词、输出长度和评分规则，再看差异。
这样做很快就能看出，摘要任务里常见的失败不止一种：有的模型短文很稳，长文开始漏点；有的词面重合不错，事实却错；还有些模型在固定长度下只是把前两句抄得更像，真正的信息压缩没有做好。只看一个总分，很难知道问题到底出在哪个桶。

## box soft
### 跟着一个样本走一遍
步骤我们看什么会暴露什么问题


输入文章新闻正文和标题文章太长时，模型是否丢失关键段落
生成摘要固定字数或固定句数模型是否只会复述开头
自动指标ROUGE、BERTScore、事实核验词面重合和真实信息是否一致
人工抽检最差 10 例、数字错误例、漏点例平均分下面藏了什么失败
决策上线、回炉、换任务分数高但成本太高，或错误类型太危险

# 评估记录 {#review}
<!-- tag: 复盘 -->
<!-- layout: grid2 -->

## box
### 把一次评估写成四栏记录
好用的评估记录，通常会把样本桶、评分方式、失败类型和下一步动作放在同一张表里。这样你不会只看到“平均分 88”这类结论，还能同时看到长新闻掉分、数字密集样本出错、LLM 评委偏向格式、人工抽检抓到事实错误这些局部现象。
这一步很值钱：当结果要交给团队时，别人关心的不只是分数，还关心这个分数覆盖了什么、漏掉了什么、最差样本在哪里，以及下一轮要改提示词、改数据、改评委还是改部署约束。
如果你手里只有一串总分，最容易发生的事就是把不同桶的差异抹平。更好的写法，是把一轮评估拆成三个句子：先说样本怎么分桶，再说每个桶用什么规则打分，最后说哪个桶的失败最影响部署。这样写出来的记录，后面的人能直接复测，也能直接讨论要不要补长文样本、改提示词模板，或者把某一类高风险输入单独拎出去。
再往前一步，可以把每个桶的输入特征、评分规则、失败现象和修正动作放进同一张小表。这样，回头看结果时，大家看的是一条完整链路，而不是一串孤零零的数字。

## box soft
### 最小复盘清单
栏目要写什么为什么要写


样本桶短文、长文、数字密集、跨领域、攻击样本看出模型在哪类输入上掉链子
评分规则exact match、ROUGE、BERTScore、人工、LLM 评委看出分数到底奖励了什么
失败类型漏点、幻觉、格式错、偏见、被注入把“错”翻译成能修的对象
决策结果上线、回炉、补数据、换任务让评估直接服务下一步动作
这张清单的作用，是把“分数低”翻成可操作的修正方向。样本桶不清楚，就很难知道该补数据还是改提示词；失败类型不清楚，就很难知道是事实核验、拒答规则还是评委偏差出了问题。
现在你应该能做什么：拿到一次评估结果时，先按样本桶重写记录，再决定要改数据、改规则，还是改部署边界。

复盘汇总脚本复制

```python
def build_review(report):
 rows = []
 for item in report:
 rows.append({
 "bucket": item["bucket"],
 "auto_score": item["auto_score"],
 "human_score": item.get("human_score"),
 "failure": item.get("failure", "unknown"),
 "decision": item.get("decision", "revise"),
 })
 return rows
```

这个脚本很小，但它逼你把“评分结果”改写成“可复盘记录”。如果你发现某个桶一直低分，下一步就该看提示词、上下文长度、评委偏好或数据污染，而不是只把总分再报一遍。

一旦最差桶持续掉分，平均分再高也没有意义。对很多真实系统来说，一次明显失控就足以让部署方案改口。

# 把这页用到下一步 {#bridge}
<!-- tag: 桥接 -->
<!-- layout: grid2 -->

## box
### 如果你在做自己的评估表
最容易犯的错，是拿一串平均分直接交差。真正有用的做法，是先把样本按任务难度和风险分桶，再给每个桶定统一的 prompt、统一的评分规则和统一的输出限制。这样一来，你才知道模型到底是在某一类输入上掉链子，还是评分方式本身偏了。
可以先拿摘要、问答或客服回复做一轮小测试。每个桶只挑 3 到 5 条样本，人工看一遍失败类型，再写回一张四栏记录：样本桶、评分规则、失败类型、决策结果。这个动作很小，但它能把“看分数”变成“看问题”。

## box soft
### 一条最小操作线
步骤你要做什么产物


1先定义 task 和风险边界一句话任务定义
2按难度或风险给样本分桶一个小样本表
3给每个桶固定评分规则统一的 metric 说明
4挑失败样本回看一张失败回放表
5再决定要不要继续训练、改提示词或换任务下一步动作
现在你应该能做什么：拿到一个新模型时，先写评估表，再去看总分。

# 失败回放 {#replay}
<!-- tag: 回放 -->
<!-- layout: grid2 -->

## box
### 先看一张小表，再看结论
同一套摘要评估里，最常见的误判，是把不同桶的表现差异忽略掉。短新闻往往给人一种错觉：分数高，任务就稳。可一旦样本换成长新闻、数字密集新闻，问题就会露出来。
所以更稳的看法，是把失败样本回放到具体桶里。你不只是记“错了”，还要记“错在什么材料上、错在什么规则下、错到什么程度”。只有这样，下一轮改的才会是对的地方。回放时最好同时记录样本来源、输入长度和人工判定理由，这样下一轮抽样时就能直接沿着同一条线比对。

## box soft
### 一张可复用的回放表
样本桶现象为什么会误判下一步


短新闻ROUGE 高，人工也过得去样本简单，容易掩盖模型边界当作基线，不当作全部证据
长新闻平均分掉得不多，但关键信息漏掉平均值把长文失败盖住了补长桶样本，单独看下限
数字密集新闻数字和时间偶尔错词面相似不能保证事实准确加事实核验和人工抽检
跨领域新闻词汇重合低，但事实对了不少词面指标把改写误伤了把语义指标和人工判断一起看

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box soft
### 把回放接到下一轮实验
举个具体例子：如果长新闻的失败主要集中在段尾漏点，不要先急着换模型。先把输入截断、摘要长度和评分规则固定住，再补三到五条长样本做回放。这样你能分清，掉分到底来自上下文太长，还是模型根本没学会抓收尾信息。
如果数字密集样本总是出错，就单独拉一条数字核验链：把日期、金额、人名拆出来，先比对摘要，再让人工只查这三类字段。这样做的目的让下一轮改动直接对准错误源。
失败类型先做什么你会看清什么


长文漏点固定截断和摘要长度问题是不是出在上下文太长
数字错误单独抽字段核验问题是不是出在事实提取不稳
评委偏好做人类对照问题是不是被格式和风格带偏
现在你应该能做什么：看到一个坏样本，先说它落在哪个桶，再说下一轮该改哪一环。

# 人类与 LLM 评委 {#judge}
<!-- tag: 评委 -->
<!-- layout: grid2 -->

## box
### 没有标准答案时怎么办
写小说、写诗、语音合成、开放式对话，往往没有唯一 ground truth。这时可以用人类评分，也可以用 LLM 作为评委。课程里提到的 MOS，就是让很多人给 1 到 5 分，再把主观印象汇成一个平均值。
人类评委更接近真实使用者，但成本高、速度慢，也会受长度、格式、顺序和提示词影响。LLM 评委省力，却容易偏向自己的输出、偏向更像“改过的答案”、偏向带链接或格式更漂亮的回答。先解释再打分，通常比只给一个数字更接近人类判断，因为解释能把依据说清楚。

## box soft
### 评委也要校准
- 先抽小样本人类对照，确认评委和人类的相关性。
- 评审时随机打乱顺序，降低格式和位置偏好。
- 把评分规则写清楚，单独统计一致性、分歧和异常样本。
如果 LLM 评委只是因为“答案看起来更像人写的”就加分，系统会慢慢学会讨好评委，而不是解决任务。

评委优势风险适合的使用方式


人类评委最接近使用者体验贵、慢、主观波动大做小样本校准和高风险抽检
LLM 评委便宜、快、易扩展自我偏袒、格式偏好、来源偏见先做人类对照，再放大使用
verifier可以专门学评分规则若评分规则不稳，也会被带偏做更稳定的评分或反馈组件

# 泄题与安全 {#safety}
<!-- tag: safety -->
<!-- layout: grid2 -->

## box
### 评估还要看攻击面
资料泄露会让模型看起来分数很高，但它可能只是见过题。jailbreak 是绕过安全拒答，让模型做不该做的事。提示词注入则是把恶意指令藏进网页、文档或上下文里，诱导模型偏离原任务。
这三类问题很容易被“单轮聊天、单一指标、单一数据集”遮住。真正要看的，是模型在未知输入、未见资料和恶意环境里的表现。

## box soft
### 部署时别漏掉的指标
- 攻击预算：对手试几次、换几种说法、藏几层指令。
- 攻击成功率：到底有多高。
- 失败分类：是泄题、注入、拒答失效，还是偏见放大。
- 环境敏感性：把恶意内容放到网页、PDF、表格里会不会更危险。

风险典型症状为什么会发生更稳的修法


资料泄露榜单高分，换题就掉测试题或相似内容进了训练语料改写题、时间后移题、前缀续写检查

# 常见误区 {#diagnosis}
<!-- tag: mistake -->

## box
看评估结果时，先做三步：先问量什么，再问怎么算，再问有没有被泄题、注入或偏见扭曲。下面这些错法最常见。
错误说法 / 场景错因后果修法


“排行榜第一，所以一定适合我的任务。”榜单任务、提示词和聚合方式可能与你的应用不同。选到不适合领域、延迟或成本要求的模型。先用自己的样本做小规模验证。
“exact match 低，模型一定不知道答案。”生成式输出可能多解释、换格式或换单位。把格式错误误判成知识错误。先做规范化解析，再单独看格式。
“BLEU / ROUGE 高，摘要一定好。”词汇重合无法保证事实正确或覆盖完整。奖励照抄、漏点或语义错误。结合事实核验和人工抽样。
“LLM 评委便宜，所以全量直接用。”模型评委可能自我偏袒、偏向格式和来源。自动评分系统性偏移。先做人类小样本对照，再放大使用。
“通过一次安全测试就安全。”攻击者会多轮尝试，也会把指令藏进环境。部署后被 jailbreak 或 injection 击穿。报告攻击预算、失败类别和环境注入测试。

# 练习 {#practice}
<!-- tag: 诊断 -->

<!-- practice-intro -->
## box soft
### 怎么答才算会
这组练习的重点，是把任务、指标、评委和安全风险分开。先做选择和填空，再做诊断和综合表达。

## box info
### 检查重点
答题时先说对象，再说分数，再说偏差来源。不要把 benchmark、指标、评委、verifier 和安全攻击混成一个东西。

<!-- end-practice-intro -->

<!-- practice-group: 一、基础概念 -->
<!-- quick-review: REMIND benchmark 既可能指测试资料，也可能指整套评估流程。 -->

### p4-q1 单选

**Q:** benchmark 在这一讲里最准确的理解是？

- [ ] . 只有测试题集
- [ ] . 一整套测试资料、输入流程、评分和聚合方式
- [ ] . 模型最后输出的分数

Answer: . 一整套测试资料、输入流程、评分和聚合方式

### p4-q2 填空

**Q:** 评估里的指标通常被理解成什么？

Answer: 评分函数;打分函数

### p4-q3 综合

**Q:** 为什么 benchmark 和 指标 不能直接合并成一个词？

Explain: 好的回答会把 benchmark 说成“测试流程或测试资料”，把 指标 说成“评分函数”，把 aggregation 说成“合成总结果的方法”。最后要指出：混在一起会让人误以为一个高分就能代表所有任务、所有材料和所有评分规则。

<!-- end-practice-group -->

<!-- practice-group: 二、自动指标 -->
<!-- quick-review: REMIND format、overlap 和 factuality 是三回事。 -->

### p4-q4 单选

**Q:** 模型答对了内容，但多写了一句解释。exact match 这时最容易暴露什么问题？

- [ ] . 格式遵守被混进评估，任务边界变了
- [ ] . 模型一定没有知识
- [ ] . exact match 可以自动识别同义改写

Answer: . 格式遵守被混进评估，任务边界变了

### p4-q5 单选

**Q:** BLEU / ROUGE 高，但摘要仍然差，最典型的原因是什么？

- [ ] . 词汇重合和任务目标完全相同
- [ ] . 指标只看 token id，不看文本
- [ ] . 词汇重合不等于事实正确、覆盖完整或表达自然

Answer: . 词汇重合不等于事实正确、覆盖完整或表达自然

### p4-q6 检查清单

**Q:** 给摘要模型做上线前评估时，哪些数字值得一起报？

- [ ] 平均分
- [ ] 或下限
- [ ] 拒答率
- [ ] 攻击成功率
- [ ] 词表大小

Explain: 至少要同时看平均分、最差样本或下限、拒答率和攻击成功率。词表大小不是这个层面的部署指标。高风险任务里，平均分好看不够，尾部失败和攻击失败更要单独写出来。

### p4-q7 单选

**Q:** MOS 是什么？

- [ ] . 多头注意力分数
- [ ] . 多人主观评分的平均值
- [ ] . 词表上的原始 logits

Answer: . 多人主观评分的平均值

### p4-q8 填空

**Q:** verifier 常可以先理解成什么角色？

Answer: 验证器;校验器

### p4-q9 综合

**Q:** 如果 LLM 评委 总偏爱更漂亮的 Markdown 或“看起来像改过”的答案，你会怎么校准？

Explain: 优秀答案会把“先做人类对照，再放大 LLM 评委”的顺序写清楚，并指出随机顺序和明确评分规则可以减少格式偏好、来源偏好和自我偏袒。最后要补上异常样本回看，不要把评委输出当绝对真相。

- [ ] . 测试题或相似内容进了训练资料
- [ ] . 模型回答太长
- [ ] . 评委给分太低

### p4-q11 单选

**Q:** 下面哪种情况更像 提示词注入？

- [ ] . 让模型把一句话写得更长
- [ ] . 让模型先解释再打分
- [ ] . 在网页或文档里塞入指令，让模型偏离原任务

Answer: . 在网页或文档里塞入指令，让模型偏离原任务

### p4-q12 综合

**Q:** 给一个摘要模型设计一套上线前评估方案

Explain: 好的方案会同时报告平均分、最差样本、事实错误率、拒答率、首 token 延迟、token 速度、成本和攻击成功率。若 8% 的样本会捏造数字或在攻击下失控，这个模型就不适合高风险部署。
