# 概念骨架：从 Foundation Model 到持续学习 {#skeleton}
<!-- tag: SKELETON -->
<!-- layout: grid2 -->

本讲把参数更新前的通用模型称为 foundation model。它可能是 LLaMA、Gemma、DeepSeek、GPT 的某个可微调版本。为了特定任务调整参数后，得到 fine-tuned model。这个过程也叫 post training，因为它发生在通用模型已经被训练出来、交到你手上之后。
后训练和 P7 的 alignment 有相似处，都可能发生在预训练之后。区别在于，本讲默认你并不知道 foundation model 过去怎么被训练，不知道它见过什么资料，也不知道哪些能力依赖哪些参数。在这种“学习历程来路不明”的前提下动参数，风险会明显变高。

## box warn
### 为什么要持续学习？
知识会过期，例如总统换人；模型可能缺技能，例如某种语言、领域或模态；模型概念可能需要更新，例如关系建议不该一律分手；模型也可能需要遗忘隐私、版权或不安全内容。

Foundation model
已释出，训练历程不透明


Post training
Fine-tuning
Editing / Merging
Test-time training


Fine-tuned model
行为变了，旧能力也可能变

判断主线：目标改到了吗？相关说法会跟着改吗？无关能力有没有被破坏？

| 路线 | 它在骨架里的位置 | 最先问的问题 |
| --- | --- | --- |
| Fine-tuning | 直接用训练目标和 loss 更新参数 | 训练目标有没有同时写出要保护的旧能力？ |
| Model editing | 定位知识位置，再局部编辑参数 | 这是少量事实编辑，还是一整套技能学习？ |
| Model merging | 把多个后训练结果用 task vector 合起来 | 这些模型是否来自同一底座，merge 后是否会互相干扰？ |
| Test-time training | 测试样本来了以后临时或连续更新参数 | 是否允许推理时多花计算并改变模型状态？ |

# 术语表：先把容易混的词分开 {#terms}
<!-- tag: TERMS -->

| 术语 | 直觉解释 | 准确边界 | 常见误解 |
| --- | --- | --- | --- |
| Foundation model | 你拿到手、准备继续使用或改造的通用模型。 | 本节特指参数可被更新的模型，训练历程通常不透明。 | 把 chat UI 里会记住上下文误当成参数被改了。 |
| Post training | 模型释出后，为你的需求继续训练。 | 必须有参数更新；prompt、RAG、in-context learning 不算本节的学习。 | 以为“行为变了”就一定是参数变了。 |
| Reliability | 你想改的目标真的改到了。 | 目标题或目标任务表现应达到预期。 | 只看这一项就宣布微调成功。 |
| Generality | 相关说法、近义问法、反向问法也跟着正确。 | 相关输入的行为应一致，不只记住一个训练句子。 | 把背下训练样例当成学会知识。 |
| Locality | 无关能力不被改坏。 | 无关事实、安全拒答、格式输出、旧任务表现都要检查。 | 以为新任务正确就不必测旧能力。 |
| Catastrophic forgetting | 模型旧能力成片崩坏。 | 常出现在新训练目标没有显式保护旧能力时。 | 以为用了 LoRA 就不会遗忘。 |
| LoRA / adapter | 只开放少量额外参数让模型学。 | 通过缩小可调范围减少遗忘风险。 | 忘记它也可能 learn less，目标能力学不够。 |
| Task vector | 微调前后参数的差，像某项能力的方向。 | θ A - θ 0 可用于 merge、subtraction 或类比。 | 以为任何两个模型都能无条件相加。 |
| Test-time training | 测试样本来了，模型先现场训练一下再回答。 | 推理时会更新参数，和 reasoning 只多算不改参数不同。 | 把长思考、chain-of-thought 当成 TTT。 |

# 讲义：四条后训练路线和三把验收尺 {#lecture}
<!-- tag: LECTURE -->

Reliability
你要改的东西真的改到了。问“鲁夫吃了什么果实？”模型应回答尼卡果实。


Generality
相关表达也会跟着改。问“谁吃了尼卡果实？”或换一种说法，模型也应能推到鲁夫。


Locality
无关知识不被破坏。问“乔巴吃什么果实？”不应也变成尼卡果实；安全、格式、数学、语言能力也不应乱掉。

## box
### 1. 先别动参数：Prompt、RAG、上下文工程优先
后训练成本高，风险也高。很多时候你以为必须微调，其实只是 prompt 写得不够明确，或没有给模型外部资料。字幕里提到总统换人的例子：如果模型坚持旧答案，可以先在上下文里说明新事实，或让模型联网、检索资料。经常变化的事实尤其适合 RAG，因为事实变了只要更新资料库，不必反复改模型参数。
判断句：如果需求可以通过上下文、检索或工具稳定解决，就先不要改参数。只有当部署环境不能依赖这些外部信息，或者你需要模型长期学会某种行为时，才进入后训练。

## box
### 2. Fine-tuning：直观，但容易把旧能力带坏
直接微调的流程最像普通机器学习：把目标转成训练资料，定义 loss，决定哪些参数可调，再从 foundation model 的参数开始做 gradient descent。训练资料可以是一段继续预训练文本，可以是 SFT 问答，也可以是 DPO 这样的偏好对。它的优点是通用、工具成熟、能教模型学技能；问题是 loss 只关心你写进去的目标。
字幕用一个极端例子：只训练模型回答“谁是全世界最帅的人？”到“李宏毅”。目标题成功后，看似 reliability 达成；但一问“谁是美国总统”“谁是肥仔”，模型也答“李宏毅”。这说明模型可能学到错误规则：“谁是”后面都接同一个答案。loss 没有惩罚这个错误，因为训练资料没有告诉它哪些无关问题不该变。
你可以加入 locality 资料，比如“谁是美国总统？”要答“川普”。这会好一点，但仍可能把模型教成短答风格，导致原来丰富的回答变成一个词。后训练资料不是越多越保险，资料的分布、风格和覆盖面都会改变模型。

$$
Lfine-tune = Σ cross_entropy(model(prompt), target)
 这个 loss 只知道训练资料里的目标，不知道你心里想保护哪些旧能力。
$$

## box
### 3. 灾难性遗忘：不是忘一点，是能力块崩掉
Catastrophic forgetting 指模型在后训练中大面积遗忘旧能力。它一个能力区块整体崩坏。字幕给了两个典型例子：LLaMA 2 Chat 用中文资料继续训练后，会说中文了，却丢失了安全拒答；让模型学语音情绪任务后，能判断情绪了，却忘了如何输出 JSON。
根因很朴素：你没有在 loss 里要求模型保持旧能力，它就没有义务保持。训练只会最小化你给它的目标，不会自动理解“请只改这一点，别动其他地方”。以后看到“目标任务准确率上升”，你要立刻追问：安全、格式、常识、原语言、原任务有没有一起测？

## box
### 4. 减少遗忘：缩小范围、加偏好、做 Replay
第一类办法是缩小可调参数范围。LoRA 和 adapter 不更新整个模型，而是在少量参数上学习。范围小，模型比较不容易偏离原来的逻辑；但范围太小也可能学不动，论文标题里常见的说法是 learn less and forget less。它把学习能力和遗忘风险一起压低。
第二类办法是改 loss。你可以惩罚微调后参数离原模型太远，也可以给不同参数不同权重：重要参数少动，不重要参数多动。问题是“哪个参数重要”本身就是研究问题。你也可以把想保护的旧行为写进训练资料，让 locality 进入 loss。
第三类办法是 experience replay：新资料和旧资料一起训练，让模型一边学新技能，一边复习旧能力。难点是开源模型通常只开权重，不开训练资料。语言模型会说话，所以可以让 foundation model 自己生成一批问答，近似原来的训练分布，但这仍然只是近似。

| 方法 | 改哪一步 | 直觉 | 代价 |
| --- | --- | --- | --- |
| LoRA / adapter | 可调参数范围 | 只调少量参数，少忘一点 | 可能学不够 |
| 参数距离 regularization | loss | 微调后不要离原模型太远 | 重要参数怎么识别仍难 |
| Locality 训练资料 | loss 和资料分布 | 把想保护的旧行为也写进目标 | 资料风格可能带来新偏差 |
| Experience replay | 资料混合 | 新知识和旧能力一起复习 | 通常拿不到原训练资料 |

## box
### 5. Model editing：先找知识位置，再局部编辑
Model editing 的目标是比 gradient descent 更精准：先找出某个知识对应的神经元、表示或参数，再直接编辑这些位置，希望只改相关知识，保持 locality。字幕介绍了经典方法 ROME，也就是 Rank-One Model Editing。
ROME 的例子是把“Space Needle is in Seattle”改成“Space Needle is in Taipei”。它先遮住关键 token，再逐层替换 representation，观察哪个位置最能恢复 Seattle 的概率，由此定位“Space Needle → Seattle”这条知识可能在哪里被取出。定位之后，ROME 修改某层 feed-forward network 的参数，让同一个 key 输出新的 value。
编辑不只是把目标改到。ROME 还会保护其他关系，例如 Eiffel Tower → Paris、Khufu Pyramid → Egypt。换句话说，model editing 也要同时考虑 reliability 和 locality。它适合少量精确知识编辑；如果你要教一整套领域技能，局部编辑通常不够。

$$
目标：k* 经过 W* 输出 v*
 保护：k1, k2 ... 经过 W* 仍输出原来的 v1, v2 ...
 编辑不是只改目标，还要显式保护不该变的关系。
$$

## box
### 6. Model merging：把参数差当成能力方向
Model merging 的想法是：如果两个模型来自同一个 foundation model，一个微调出了能力 A，另一个微调出了能力 B，能不能不用训练资料，直接把参数差加起来，得到同时有 A 和 B 的模型？很多实验显示，这件事在一些场景下真的可行。
微调前后参数的差叫 task vector。θA - θ0 可以看成能力 A 的方向。把中文能力和 alignment 能力的 task vector 加到同一个底座上，可能得到既能中文回答、又能保持安全拒答的模型。反过来，减掉某个 task vector 可用于 machine unlearning，让模型忘掉某些不该知道或不该说的东西。
Model merging 最大吸引力是几乎不用训练资料和算力。但它不是总会成功。merge 后两个能力可能互相干扰，模型也可能出现奇怪行为。现在有 mergekit 这类工具实现多种合并方法，也有研究试图在训练时就让模型更容易被 merge。

$$
task vector: τA = θA - θ0
 merge: θmerged = θ0 + τA + τB
θ0 是原 foundation model，τA 可以理解为某项能力的方向。
$$

## box
### 7. Test-time training：测试时现场改参数
传统流程是先 training，再 testing。Test-time training 把这两个词放在一起：测试样本来了，模型不马上回答，而是先用这笔样本或一个 batch 的样本现场更新参数，再用更新后的模型回答。它属于 test-time computing 的一种，但和 reasoning 不同：reasoning 不改参数，TTT 真的改参数。
一种直观做法是近邻微调：在训练集中找和当前测试输入最接近的有标签样本，用这些近邻样本临时微调模型。全局模型可能无法 fit 每个局部，但为当前输入附近重新调一下，可能更准。另一类方法没有 label，会把测试 batch 当 unlabeled data，用 semi-supervised learning 的思路训练，例如 TENT 的 entropy minimization。
TTT 的前提是你愿意为每个输入多花计算，并允许模型在推理时改参数。它不是普通 API 调用里的默认行为，也不是把上下文塞长一点。

| TTT 方法 | 是否需要 label | 训练资料来自哪里 | 适合直觉 |
| --- | --- | --- | --- |
| 近邻 TTT | 需要相似样本的 label | 训练集中与测试输入接近的样本 | 局部特化 |
| TENT | 不需要 label | 测试 batch | 让输出更确定 |
| Pretext task TTT | 通常不需要人工 label | 测试样本构造的辅助任务 | 先适应输入，再做主任务 |
| SUTA | 单句也可尝试 | 当前语音样本 | 为每句话定制语音识别模型 |

## box
### 8. 连续 TTT：越学越久，也越需要刹车
普通 TTT 往往对每个输入临时更新，用完就退回原 foundation model。连续 TTT 则把更新保留下来：第一笔输入学到一点，第二笔在这个基础上继续学，理论上模型会越来越适应环境。这很符合大众对 AI 的想象，但实践中很容易崩。
每次只用一笔或少量无标签资料更新，很容易引入 catastrophic forgetting。模型在当前样本上变好，却把通用能力弄坏；下一笔再从坏掉的模型继续更新，错误会累积，最后 model collapse。Dynamic SUTA 的思路是把更新分成 fast update 和 slow update：fast 针对当前输入，slow 负责长期稳定，再配合 reset 发现坏掉时回滚。

## box
### 9. 路线选择：先写验收集，再选方法
如果需求是频繁变化的新事实，RAG 往往比微调合适；如果是新领域技能，可以考虑 LoRA、adapter 或 SFT；如果是少量精确事实，model editing 更贴近问题；如果你已有多个同底座模型，model merging 值得试；如果每个测试输入的分布差异大，TTT 才有意义。
最重要的顺序先写验收集。验收集至少包含：目标样例、相关改写、无关事实、旧任务、安全拒答、格式输出。后训练是否值得做，要看这些指标一起变化。

# 例子：把抽象风险落到具体问题 {#example}
<!-- tag: EXAMPLE -->
<!-- layout: grid2 -->

## box
### 例 1：为什么“谁是”微调会跑偏
训练资料只有一笔：“谁是全世界最帅的人？”到“李宏毅”。对 loss 来说，只要这个输入后面接出目标 token 就好。模型可以学到正确规则，也可以学到粗糙规则：“看到谁是，就接李宏毅”。两个模型在训练样例上的 loss 都低，但第二个模型的 locality 很差。
检查方法：除了目标题，还要问“谁是美国总统”“谁是英国总统”“台湾最高的山是什么”。如果无关问题也被拖到新答案附近，说明微调并没有成功。

## box
### 例 2：为什么加 locality 资料仍可能带偏风格
加入“谁是美国总统？”到“川普”可以阻止部分错误，但如果所有训练答案都是短词，模型可能学到“回答要短”。它不再像原来那样解释背景，只吐出一个词。这个问题回答风格被训练资料改了。
检查方法：locality 资料要覆盖内容，也要覆盖风格、长度、格式和安全策略。旧能力不是只有知识问答。

决策顺序：不要一上来微调post-training gate

```text
if prompt_or_context_solves_it:
 do_not_update_parameters()
elif retrieval_or_tool_can_supply_facts:
 use_rag_or_tooling()
else:
 define_reliability_generality_locality_tests()
 choose_post_training_method()
```

Model editing 例子
把 Space Needle 的地点从 Seattle 改成 Taipei 时，不能只看目标输入是否输出 Taipei，还要保护 Eiffel Tower → Paris 这类无关事实。


Model merging 例子
一个模型学中文，一个模型保留 alignment。若两者来自同一底座，可以尝试把两个 task vector 加到一起，再验收中文能力和安全拒答。


TTT 例子
语音识别里，每句话的口音、噪声和场景不同。SUTA 这类方法会先用当前语音做一次适应，再输出辨识结果。

核心直觉
后训练的关键控制改动范围。你希望目标问题变好，同时相关问题能泛化、无关问题不被污染、旧能力不突然遗忘。


术语边界
**RAG** 不改参数，只把新资料放进上下文；**fine-tuning** 改一批参数；**model editing** 尝试局部改知识；**model merging** 合并参数差；**TTT** 在测试时临时适应输入。


Worked example：更新一个过期事实
若只是“某公司 CEO 换人”，优先用 RAG 或工具，因为事实还会变；若要模型长期掌握一套医学写作规范，可用 LoRA / SFT；若只修一条稳定知识，才考虑 model editing。每次都要验收 reliability、generality、locality 三项。

# 可视化：看见学习强度和稳定性的拉扯 {#viz}

微调强度与遗忘风险

# 诊断：后训练最容易误判的地方 {#diagnosis}
<!-- tag: DIAGNOSIS -->

| 症状 | 可能错因 | 后果 | 修法 |
| --- | --- | --- | --- |
| 训练样例都答对 | 只优化 reliability，没有测 paraphrase 和无关题 | 模型背样例，换说法就失败 | 补 generality 与 locality 测试集 |
| 所有类似开头都接同一答案 | 训练资料太少，模型学到粗糙模式 | 无关问题被拖坏 | 增加多样化负例和保护样例 |
| 模型变得特别省话 | locality 资料答案风格单一 | 旧的表达和指令跟随能力下降 | 让保护资料覆盖原来的回答风格 |
| LoRA 后目标学不动 | 可调范围太小 | 遗忘少，但新能力也少 | 调 rank、资料或训练轮数，并重新测旧能力 |
| Merge 后两个能力都变差 | task vector 冲突，底座或训练方式不兼容 | 模型行为不稳定 | 尝试权重系数、分层 merge 或后续校正 |
| 连续 TTT 越跑越坏 | 无标签小样本更新累积偏差 | model collapse | 分 fast/slow update，设 reset 和稳定性阈值 |

## box
### 看到“成功”时先问三句
目标题是否真的改到？相关问法是否也改到？无关事实、安全拒答、格式输出、旧任务是否保持？这三句能过滤大部分表面成功。

## box warn
### 看到“失败”时别急着加训练
先定位失败在哪一项：reliability 不够、generality 不够，还是 locality 崩了。三种失败对应的修法不同，混在一起调会很乱。
