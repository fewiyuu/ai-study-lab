P11｜作业1：理解生成式人工智能的基础知识

P11 是作业说明课。它把前面学过的 tokenizer、prompt、next token prediction、chat template 和多轮对话，放进 HuggingFace、Google Colab、Gradio 与 NTU COOL 的实际流程里。

一、概念骨架：从入口到提交

SKELETON

作业最终提交在 NTU COOL 的线上测验里，不是交 Colab 文件，也不是提交 HuggingFace token。Colab 负责跑代码和拿观察结果；HuggingFace 负责提供模型与 tokenizer 的访问权限；Gradio 负责把模型调用包成一个能聊天的网页界面。

第 1 部分：token看 tokenizer 怎么切文字、怎么还原 token id、token length 为什么会变，以及模型怎样预测下一个 token。

第 2 部分：prompt复制 system prompt 与 user prompt，观察高层规则、当前请求、标点和换行怎样改变输出。

第 3 部分：对话与界面比较有无历史记录的多轮对话，并用 Gradio 体验参数、聊天记录和网页 demo。

二、术语表：先把名词对齐

TERMS

这节课的难点不在术语本身，而在术语容易混到一起。Access token 不是模型许可，Colab 不是交作业的地方，chat template 也不是普通的提示词。先把这些边界对齐，后面排错会快很多。

三、讲义：把作业流程讲成一条线

LECTURE
  1. NTU COOL 只认线上测验提交

字幕一开始说明，作业在 NTU COOL 的 Homework 1 里完成。题目和选项会用中英文分隔，意思是你可以读中文，也可以读英文；分隔线本身不是题目内容。填完后必须点提交测验，关闭浏览器不会留下作答记录。

课程允许重新参加测验，并按最高分计算，这主要是为了减少误操作造成的损失。自学时不要把这个规则理解成“随便猜也行”。更稳的做法是每题旁边记下依据：来自哪个 cell、哪个输出、哪个 prompt 设置。你现在应该能判断：NTU COOL 是交付入口，Colab 是证据来源。

离线自学任务模板

  1. 先在纸上写出本作业的三段任务：token 观察、prompt 对照、多轮对话体验。
  2. 每跑一个 Colab cell，记录它属于哪一段、输入是什么、输出应该看哪一行。
  3. 遇到题目要求比较输出时，用“模型名 / prompt 原文 / 参数 / 观察结果”四列做一行记录。
  4. 最后再打开 NTU COOL 作答，答案必须能回指到某一次实际运行或页面中的概念解释。

你现在应该能做什么：不用回看视频，也能按顺序完成作业，并说明每个观察结果来自哪里。

worked example：prompt 观察记录

假设题目要求比较 system prompt 的影响，不要只写“结果不一样”。可以这样记录：模型使用课程指定版本；user prompt 固定为同一句问题；第一轮 system prompt 要求简短回答，第二轮要求详细推理；观察到第二轮输出更长、步骤更多。由此判断，system prompt 是高层行为约束，不是普通聊天内容。

五、最小例子：看懂代码在读写什么

EXAMPLE

下面的例子不追求完整复现课程 notebook，只保留作业里最需要读懂的动作。读代码时先问三件事：这一行读取什么、写入什么、改变哪个状态。能回答这三点，排错就不容易乱。

代码
!nvidia-smi

这条命令读取当前 runtime 的 GPU 状态。如果输出里能看到 T4 等 GPU，说明硬件加速已经挂上；如果命令失败或看不到 GPU，就回到执行阶段设置里检查硬件类型。

逐行拆解：从文字到下一个 tokentokenizer复制

ids 是模型真正读到的输入，decoded 用来检查切分后的 token 能否还原成可读文本。last_logits 只取最后一个位置的预测分数，因为我们关心“前文之后的下一个 token”。如果把模型或 tokenizer 换掉，ids、logits 维度和最后答案都可能变。

代码
messages = [
 {"role": "system", "content": "Answer briefly."},
 {"role": "user", "content": "Hi, I am Wilson."},
 {"role": "assistant", "content": "Hi Wilson."},
 {"role": "user", "content": "Who am I?"}
]
prompt = tokenizer.apply_chat_template(messages, tokenize=False)
ids = tokenizer.encode(prompt)

这段代码的关键历史消息有没有进入 messages。如果只保留最后一句 Who am I?，模型没有前文依据，只能猜。

错误示例：权限和输入都不完整error复制

第一段可能因为 license 未同意而下载失败。第二段把题目给定 prompt 截断了，输出就不能作为选择题依据。修法是回到模型页面确认许可，再按题目完整复制 system prompt 和 user prompt。

六、可视化：把参数变化看出来

VISUALIZATION

生成式模型不是固定查表。给定前文后，模型会给候选 token 分数，采样参数再决定输出有多稳定。下面这个小控件只展示直觉：温度低时更偏向最高分 token，温度高时更容易出现多样答案，也更容易飘。

判断重点如果题目问 token id、长度、下一个 token，要使用课程指定模型和 tokenizer。换模型或换采样设置，观察结果就可能改变。

七、错误诊断：看到现象先定位环节

DIAGNOSIS

这份作业的常见问题多半权限、环境、复制或上下文出了问题。排错时先问错误发生在哪一层：账号层、运行层、输入层，还是提交层。

$$ 排错顺序：权限是否可用 → runtime 是否还活着 → 前置 cell 是否跑过 → 输入是否完整 → 输出是否对应题目。 这条顺序能帮你少做无意义修改，尤其是在 Colab 报错信息很长的时候。 $$
