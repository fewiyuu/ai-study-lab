P13｜作业2：构建一个基础的 RAG 系统

这份作业要求你实现一个 RAG 系统。给定《红楼梦》全文作为 knowledge base，再给一组问题，你的系统要先检索相关片段，再让语言模型根据片段生成答案，最后把 100 个回答提交到 Judge Boy 批改。

一、概念骨架：整理书架、找书、请老师回答

SKELETON

课程用图书馆比喻 RAG。第一步，你把地上一堆书整理上架，这对应 data preprocessing：把原始知识库切成 chunk。第二步，朋友问你问题，你请图书管理员去书架里找相关书，这对应 retrieval：根据 query 找最相关的 chunk。第三步，你把问题和找到的书交给老师，请老师整理出通顺答案，这对应 generation：让语言模型根据检索片段回答。

Data preprocessing

切 chunk 建知识库

Generation

根据片段回答

Data preprocessing把 passages 切成可检索的 chunk。chunk 是比较相似度的最小单位，不是随手切碎文本。

Retrieval用 query 找 relevant chunks。这里的检索质量决定模型能不能看到答案证据。

Generation把 query 和相关片段写进 prompt，让指定语言模型根据证据生成 response。

二、关键术语：先把名字和边界对齐

TERMS

RAG 的难点常常很多词同时出现：query、passage、chunk、retriever、response、baseline、public dataset、private dataset。先把它们放到同一张表里，后面读 Colab 和分数说明才不会混乱。

为什么要 RAG

RAG 还适合 domain-specific 和 up-to-date knowledge。比如繁体中文、红楼梦细节、企业内部资料、刚更新的政策，这些都不一定在模型训练资料中充分出现。把资料整理成 knowledge base，需要时检索给模型。

常见误区

RAG 不等于“把全部知识库都贴进 prompt”。如果输入太长，模型虽然看得到，但未必看得懂；而且推理成本更高。检索的价值就是先把信息筛出来，让生成模型专注回答。

你现在应该能做什么：读到 query、chunk、retrieval、generation、response 时，能指出它们在流程中的位置，并能解释为什么 RAG 比直接问模型更容易排错。

三、课程讲义：红楼梦数据集与评分规则

LECTURE

作业使用《红楼梦》120 个章回、70 多万字作为 knowledge base。题目由模型根据章回生成，再由 TA 检查合理性。每个题组包含三个组件：query、golden passage index、answer。

字段含义用途
query要回答的问题。系统输入，也是检索的起点。
golden passage index出题时参考的章回或片段索引。public 题可用于分析检索是否找对出处。
answer标准答案。public 题可用于调参，private 题截止后公布成绩。
response你的 RAG 系统生成的答案。提交 Judge Boy 批改。

Judge Boy 会用模型批改你的 response。为了控制成本，只取每个回答前 512 个字符评估。课程说明里强调，题目通常一两百字以内就能完整回答，所以写很长不一定有利，反而可能引入矛盾内容。

单题分数含义调试启发
1答案正确，和标准答案一致或近似。检索和生成都成功。
0.5部分正确，有遗漏或小错误。可能缺少关键 chunk，或模型没有整理完整。
0无法回答或没有有效答案。比胡编更安全，但要检查检索召回。
-1答案明显错误或矛盾。通常是幻觉、错误检索或 prompt 约束不足。
$$ 最终成绩 = 成功提交保底 2 分 + public baseline 最多 4 分 + private baseline 最多 4 分。 public 分数可即时查看；private 分数截止后公布。 $$

你现在应该能做什么：拿到一条错误 response 时，能判断它可能是检索没召回、生成没读懂，还是答案过长引入了矛盾。

四、资料与评分：baseline、Colab 和提交边界

SOURCE

课程把成绩拆成 simple、medium、strong、boss 四条 baseline。Simple baseline 直接让语言模型回答问题，不做检索；medium baseline 跑完整 RAG workflow，并用 BM25 做 retrieval；strong baseline 加入 embedding model，再把 BM25 和 embedding 的排名融合；boss baseline 继续调 data preprocessing 和输入长度。

1选择运行环境需要 GPU，因为作业会用到 embedding model 和 generation LLM；免费 Colab 通常选择 T4。

2下载资料与套件public/private test dataset 和《红楼梦》全文会被下载到运行环境，后续代码再载入问题与知识库。

3只跑目标 baseline四个 baseline 的生成格会产生同名 response。后跑低阶方案可能覆盖前面高阶方案的输出。

4提交 prediction file文件是 100 个字符串组成的 JSON list，大小不能超过 2MB；每天通常只有三次提交机会。

离线自学任务模板

  1. 先选一道 public 题，把问题、正确答案、检索到的 top chunks 分开记录。
  2. 逐层判断错误来源：证据没召回、证据召回但 prompt 没用上、还是输出格式不合要求。
  3. 每次只改一个变量，例如只改 chunk size、只改 stride、只改 top-k 或只换检索器。
  4. 提交前确认生成的是目标 baseline 的 prediction file，且没有被后续 cell 覆盖。

你现在应该能做什么：低分时先定位 RAG 的哪一层坏了，而不是同时乱改检索和生成。

worked example：一题排错

如果问题问“贾宝玉初见林黛玉时注意到什么”，top chunks 里完全没有初见段落，优先查 retrieval；如果 top chunks 已有初见段落但模型回答了别的情节，优先查 prompt 和 top-k 噪声；如果回答内容正确但 JSON 结构坏了，优先查提交格式。

五、最小例子：从 BM25 到 hybrid retrieval

EXAMPLE

Medium baseline 要实现完整 RAG workflow，并使用 BM25 做 retrieval。BM25 属于 sparse retrieval，它会把 query 和 document/chunk 切成离散词项，然后根据关键字匹配和词频等公式计算相关性。

适合 BM25 的问题

人名、地名、物件名、章回小说中的专有称呼，或者问题中直接出现答案附近的关键词。

BM25 的弱点

同义词、隐喻、跨语言、语义相近但字面不同的表达。它像查书索引，不像真正理解句意。

伪代码：BM25 检索retrieval复制

chunks = chunk_documents(hongloumeng_text)
bm25 = BM25(tokenize(chunks))

query_tokens = tokenize(query)
scores = bm25.score(query_tokens)
top_chunks = take_top_k(chunks, scores, k=5)

Strong baseline 会加入 embedding model。Embedding model 把 query 和 chunk 转成连续向量，再用 cosine similarity 比较相似度。它属于 dense retrieval，因为每段文本被表示成稠密浮点向量，而不是离散关键词集合。

Embedding 的优势正好补 BM25 的弱点：字面不同但语义相近，也可能被放到向量空间中相近位置。甚至跨语言、跨模态模型也可以把不同语言或图像文字对齐到同一语义空间。缺点是需要模型推理，计算和储存都比 BM25 贵；如果资料领域超出 embedding model 训练范围，效果也会下降。

Hybrid retrieval 的直觉是：BM25 和 embedding 各有盲区，那就同时用。BM25 给一份 chunk 排名，embedding model 给另一份排名，再用 reciprocal rank fusion 合成综合排名。

RRF 不直接合并原始相似度分数，因为不同检索器的分数尺度不同。它只看排名：某个 chunk 在每个检索器中排得越前，综合分越高。常见公式是对每个检索器求 1 / (k + rank)，再把结果相加；课程里提到 k 常设为 60。

$$

RRF(c) = Σr ∈ retrievers 1 / (k + rankr(c))

c 是某个 chunk；r 是某个检索器；rank 越小，贡献越大。

$$

伪代码：融合 BM25 与 embedding 排名hybrid retrieval复制

bm25_rank = rank_chunks_by_bm25(query, chunks)
embed_rank = rank_chunks_by_embedding(query, chunks)

rrf_scores = {}
for rank_list in [bm25_rank, embed_rank]:
 for rank, chunk_id in enumerate(rank_list, start=1):
 rrf_scores[chunk_id] += 1 / (60 + rank)

merged = sort_by_score(rrf_scores, descending=True)

你现在应该能做什么：看见 BM25、embedding、RRF 的代码时，能说出每一步读入什么、输出什么，以及为什么要用排名融合。

六、交互观察：检索器信号和输入长度

VIZ

下面两个小观察用来建立调参直觉。第一个看 BM25、embedding、hybrid retrieval 的侧重点；第二个看 chunk size、stride、top-k 对上下文完整性、检索纯度和噪声风险的影响。它们不是分数预测器，只帮助你判断改参数时应该观察什么。

BM25EmbeddingHybrid
关键字命中高语义理解低计算成本低

Boss baseline 的重点从检索模型转到 data preprocessing 和输入长度。你要调三个参数：chunk size、stride，以及最终喂给 LLM 的 chunk 数量。它们共同决定检索颗粒度、重叠程度、输入长度和噪声量。

Top-k 决定给 LLM 多少个相关 chunk。给太少可能漏掉答案；给太多会让 prompt 冗长，模型注意力被噪声稀释。即使模型 context window 很大，也不代表把它塞满就是好策略。课程建议从较小 top-k 开始，逐步增加，观察 Judge Boy public 分数何时开始下降。

你现在应该能做什么：调参数前先预测它会让“召回、纯度、成本、噪声”哪一项变好或变坏，再用 public 分数验证。

七、错误诊断：分数掉了先查哪一层

DIAGNOSIS

RAG 的好处之一是错误可以分层排查。不要只盯着最终 response。先看检索有没有把包含答案的 chunk 召回，再看 prompt 有没有要求模型根据片段回答，最后才看生成模型是否忽略证据或编造内容。

错误示例：只靠最终答案猜原因anti-pattern复制

为什么这段做法危险

它同时改 prompt 和 top-k，无法知道哪项造成变化；把 top-k 拉满还可能让噪声增加。更稳的做法是先打印 top chunks，确认答案证据有没有被召回，再一次只改一个变量。

你现在应该能做什么：遇到低分时，能按“检索证据、prompt 约束、生成输出、提交格式”的顺序排查，而不是盲目堆参数。
