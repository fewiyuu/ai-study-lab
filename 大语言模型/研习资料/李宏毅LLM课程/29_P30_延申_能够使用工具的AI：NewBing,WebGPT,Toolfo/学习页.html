工具型 AI 不是神谕，而是会产生动作的文字接龙

这一讲用三个例子延伸 P29 的工具调用：New Bing 展示联网聊天的实际表现，WebGPT 解释搜索、点选、收藏、回答这些动作如何被语言模型一步步产生，Toolformer 则说明模型怎样学会在文字里插入 API 调用，例如问答系统、计算器或翻译工具。

最重要的结论有两个。第一，联网不等于不会犯错，模型看过网页后仍然可能自己编出网页里没有的句子和引用。第二，工具使用不是脱离语言模型的魔法，它仍然是文字接龙：模型生成一个特殊符号，外部系统执行动作，再把工具结果接回上下文。

  3. 说明 WebGPT 怎样通过人类示范和强化学习学会搜索。

现象：联网聊天

New Bing 会自己决定是否搜索，并把搜索结果作为上下文生成回答。它还会给引用，但引用不保证句子真的来自对应网页。

机制：动作也是 token

WebGPT 把搜索、点选、收藏、回答看成特殊动作。模型生成动作符号，系统执行动作，再把结果接回上下文。

训练：让模型学会何时用工具

WebGPT 靠人类示范和强化学习；Toolformer 尝试少量示例加自动筛选，生成 API 调用训练资料。

系统/论文会用什么工具怎么产生工具动作核心提醒

WebGPT搜索引擎、点网页、收藏段落、生成答案。把搜索、点选、收藏、回答建成 action，作为文字接龙的一部分。训练需要人类老师示范，再用强化学习降低示范负担。

点选 action打开某个搜索结果。模型生成点选符号和编号，系统打开对应网页或段落。以为拿到搜索列表就足够回答。
收藏 / quote把有用证据留下。WebGPT 中模型可选择网页段落收藏，后续回答只基于保留下来的证据继续生成。以为所有打开过的网页都会进入最终回答。
回答 action停止检索，开始作答。模型生成回答符号后，未收藏内容被丢弃，只根据问题和收藏证据继续生成答案。以为答案阶段仍会自动查所有网页。
state / action当前页面状态与下一步动作。WebGPT 论文术语中，state 可理解为已生成内容和浏览状态，action 是搜索、点击、引用、回答等下一步。被论文术语吓住，忘了它仍是文字接龙。
Toolformer学会调用多种 API 的语言模型。用少量人工示例和自动筛选数据训练模型在合适位置插入 API 调用。以为它靠大量人类逐条标注所有工具用法。
API 调用把问题交给外部功能。模型在文本中生成类似 <Calculator(5*30)> 的调用，外部工具返回结果后再继续生成。以为 API 调用内容会直接给用户看。
筛选训练资料只留下真的有帮助的工具调用。比较有无 API 结果时正确续写的概率；如果调用提高正确答案概率，就保留样本。以为另一个模型随便插入 API 就能训练。

现在你应该能做什么：把工具型 AI 看成“语言模型 + 动作符号 + 外部工具 + 结果回填”的系统。

讲义主线

LECTURE

1. New Bing 的关键差异：会搜索网络

New Bing 的界面看起来和 ChatGPT 很像，但差异在于它能搜索网络。你问“李宏毅是谁”，它会显示正在搜索，把“李宏毅”当关键词交给 Bing，再基于搜到的资料产生回答。它还能识别同名人物，并给出引用来源。

但何时搜索是模型自己决定的，不是每题固定搜索。老师让它玩穿越文字冒险游戏时，一次它误以为要搜索“穿越文字冒险游戏”，给出游戏介绍；同样问题再问一次，它又不搜索，直接开始玩游戏。这体现了工具调用的随机和决策不稳定。

现在你应该能做什么：解释“能搜索”不等于“每次都会正确搜索”。

2. 联网不等于不会幻觉

老师让 New Bing 用 PTT 乡民口吻介绍李宏毅，它先搜索“PTT 乡民口吻”和“李宏毅”，然后生成带引用的回答。回答中有些内容来自网页，但也编出了“李宏毅曾在 Google 工作”这样的错误事实，尽管给了引用。

这很关键：联网模型不是把网页原文机械复制出来。它仍然基于网页内容继续生成文字，因此可能把网页里没有的东西接出来，也可能把引用贴到不该贴的位置。引用增加可追溯性，但不能自动保证事实正确。

现在你应该能做什么：看到引用时仍然检查引用页面是否真的支持该句。

3. New Bing 架构细节不透明，但可以类比 WebGPT

课程提到 Bing 的 blog 只是高层描述，称其模型/系统为 Prometheus，但没有足够技术细节。因此不能精确复原 New Bing 的内部实现。

不过有相近论文 WebGPT，可以帮助理解“会搜索的 GPT”可能怎样运作。WebGPT 能回答问题并标注引用，表现形式和 New Bing 很像，所以它适合拿来解释机制。

现在你应该能做什么：区分产品实测和论文机制，不把猜测当成官方架构。

4. WebGPT 的搜索也是文字接龙

听起来奇怪，但使用搜索引擎也可以看成文字接龙。模型先看到问题，例如“高雄过去有哪些名称”，然后生成一个特殊符号代表搜索，再生成关键词“高雄旧称”。系统看到这个特殊符号后执行搜索，把搜索结果列表和摘要接回上下文。

模型继续文字接龙时，就能基于这些搜索结果决定下一步。它可能生成“点选第 2 篇文章”的特殊动作，系统打开文章或段落，再把内容接回来。整个过程仍是模型在生成 token，只是某些 token 会触发外部动作。

现在你应该能做什么：把工具动作理解成可被模型生成的特殊 token。

5. 点选、收藏、回答：从浏览到证据

WebGPT 不只是搜索一次。它可以点开网页，读取段落，选择收藏有用内容。如果资料不够，还可以换关键词再搜一次。等模型判断资料足够后，生成“回答”动作，表示检索阶段结束。

课程里为了教学简化成“点开就读全文”，但论文更细：它会选择具体段落读入和引用。回答时，没有被收藏的内容会消失或不进入最终证据，模型基于问题和收藏证据继续生成答案。

现在你应该能做什么：解释为什么“搜索结果列表”还不是可靠答案，必须有证据选择。

6. 引用也可能是模型生成的

老师推测，New Bing / WebGPT 类系统中的引用未必完全由规则机械插入，也可能由模型生成。原因是实测中有时引用会贴到奇怪位置，明明某句话不是来自那篇文章，却被引用到该文章。

这个判断不一定是官方结论，但它提醒我们：引用不是护身符。引用本身也需要验收。好的工具型 AI 应该让用户能回到来源，检查句子和来源是否真的对应。

现在你应该能做什么：把“有引用”升级成“引用可核查且对应正确”。

7. WebGPT 怎样学会搜索：人类示范

WebGPT 需要知道什么时候搜索、搜什么、点哪篇、收藏什么、何时回答。它通过人类老师示范来学习。人类老师看到问题，在界面里输入搜索关键词，点开文章，收藏有用段落，最后点击“写答案”。这些行为被记录成动作序列。

模型随后学习这些序列。和 ChatGPT 类似，它先有预训练的语言能力，再用人类示范教它浏览动作，之后还会通过强化学习减少人类标注负担并优化行为。

现在你应该能做什么：说明工具使用能力不是凭空出现，而是从动作示范和反馈中学来。

8. State / action 其实不神秘

WebGPT 论文里会说 state 和 action。老师把它翻译成更容易懂的语言：state 就是已经生成或浏览到的内容，action 就是下一步要生成的动作，例如搜索、点选、收藏、回答。

所以不要被强化学习术语吓住。模型仍然在根据当前 state 产生下一个 action；只是 action 不一定是普通文字，也可能是工具动作。

现在你应该能做什么：把论文术语还原成“当前上下文”和“下一步动作”。

9. WebGPT 示例：尼罗河 vs 扬子江

课程展示 WebGPT 回答“尼罗河和扬子江哪条更长”。它先搜“Nile vs Yangtze”，点文章并收藏段落；再搜尼罗河长度、扬子江长度，分别收藏证据；资料足够后开始写答案。

这个过程比“直接问模型哪条河更长”更可靠，因为它显式收集证据。但可靠性仍取决于搜索关键词、点选页面、收藏段落和最终整合是否正确。

现在你应该能做什么：用动作序列复述 WebGPT 如何从问题走到答案。

10. Toolformer：不只搜索，还能用各种 API

Toolformer 延伸了 WebGPT 的想法：工具不只有搜索，还可以是问答系统、计算器、翻译器等。比如用户问“五美金可以换多少新台币”，模型可以先搜索汇率，再调用计算器计算乘法，最后只把自然语言答案给用户看。

工具调用部分可以隐藏，用户只看到最终回答，感觉模型很聪明。但系统内部其实是模型生成 API 调用，外部工具返回结果，模型再继续生成。

现在你应该能做什么：区分“模型会算”与“模型会调用计算器”。

11. Toolformer 的难点：没有大量人类工具示范

WebGPT 可以请人类老师示范搜索。但 Toolformer 要使用更多工具，如果每个工具都靠大量人工示范，成本会很高。它的有趣之处在于：想办法在没有大量人类示范的情况下自动产生训练资料。

第一招是用另一个语言模型往句子中插入 API 调用。例如在“台湾最高的山是...”后面插入一个问答 API：询问“台湾最高的山”。但仅靠这个方法噪声很大，插入位置和调用内容不一定合理。

现在你应该能做什么：说明为什么自动插 API 需要后续筛选。

12. 筛选逻辑：调用工具后更容易预测正确答案，才留下

Toolformer 的第二招是筛选。先把 API 调用拿掉，看模型预测正确答案的概率；再把 API 调用和工具返回结果放进去，看正确答案概率是否提高。如果调用后正确答案更容易生成，就说明这次工具调用有帮助，样本可以留下。

例如句子是“台湾最高的山是玉山”。没有 API 结果时，模型预测“玉山”的概率可能较低；调用问答 API 得到“玉山”后，模型预测“玉山”的概率大增。这样的样本才值得当训练资料。

现在你应该能做什么：用“是否提高正确续写概率”解释 Toolformer 的资料筛选。

13. 训练与消融：不是多喂资料就好

Toolformer 训练时会用带 API 调用的资料，让模型学习何时产生角括号式调用。实验中还做了一个关键对照：训练时看过 API 调用资料，但测试时禁止调用 API。这样可以判断性能提升到底来自“额外训练资料”，还是来自“测试时真的能用工具”。

结果显示，禁止调用 API 时效果不会太好，说明 Toolformer 的优势主要来自测试时能真实使用工具，而不只是多看了点训练资料。

现在你应该能做什么：解释为什么消融实验能排除“只是数据更多”的解释。

14. 小模型可能学不会工具

Toolformer 还显示：模型太小时，能不能调用 API 差异不明显，说明小模型可能学不会什么时候该用工具、怎样把工具结果接回回答。模型够大时，工具使用的优势才明显。

这点很重要。工具不是免费外挂；模型需要足够能力理解任务、选择工具、组织调用、整合返回结果。否则工具摆在那里也不一定会用好。

现在你应该能做什么：不把“接入工具”当作自动提升，仍要评估模型规模和工具使用能力。

完整知识链路

P30 从产品现象走到训练机制：New Bing 让我们看到联网聊天可以搜索并引用，但仍会随机决定何时搜索，也可能编造或错配引用；WebGPT 把浏览网页拆成搜索、点选、收藏、回答等动作，让这些动作成为文字接龙中的特殊符号，并通过人类示范和强化学习学会使用；Toolformer 则把工具扩展到 API，通过少量示例、自动插入调用和概率筛选生成训练资料。你现在应该能做什么：判断一个工具型 AI 的可靠性，不只看它有没有工具，还要看它何时调用、证据如何保留、结果如何整合、错误如何验收。

最小例子

EXAMPLE

下面三个例子把工具型 AI 的动作机制变成可读流程。

示例 1：WebGPT 式搜索动作序列pseudo
问题：高雄过去有哪些名称？


搜索结果：1. ... 2. ...

网页段落：高雄旧称包括打狗等...



搜索结果：1. ...

网页段落：...



高雄过去曾有「打狗」等名称，依据引用段落...

这不是普通摘要，而是先搜、再点、再保存证据、最后回答。判断句：没有 quote / 收藏证据，答案就缺少可追溯基础。

示例 2：Toolformer 式 API 调用pseudo
用户：5 美金可以换多少新台币？

模型生成：
5 美金可以换

工具返回：1 美元 = 31.5 新台币

工具返回：157.5

用户可见答案：
5 美金约可兑换 157.5 元新台币。

搜索负责实时事实，计算器负责精确运算，语言模型负责决定调用与组织答案。判断句：工具输出通常应该被核查或隐藏，不一定直接原样给用户。

示例 3：筛选工具调用训练资料logic
原句：台湾最高的山是玉山。

候选调用：
台湾最高的山是  玉山。

筛选：
不带 QA 结果时，模型预测「玉山」的概率 = 低
带 QA 结果时，模型预测「玉山」的概率 = 高

结论：
这个 API 调用真的帮助模型续写正确答案，保留为训练样本。

这就是 Toolformer 的关键：不是所有插入的 API 调用都留下，只留下对正确续写有帮助的调用。

误区诊断

DIAGNOSIS

误区 1：联网就不会幻觉

错因：以为答案都是网页原文。后果：带引用的错误更容易被相信。修法：打开引用核对句子。

误区 2：引用一定来自对应网页

错因：把引用当硬规则。后果：错配引用被当证据。修法：检查引用内容与回答句子的对应关系。

误区 3：搜索一次就够

错因：忽略多跳信息收集。后果：答案证据单薄或偏题。修法：根据问题拆成多个关键词、多次搜索和收藏。

误区 4：搜索列表就是答案

错因：只看摘要，不点开证据。后果：被标题党或片段误导。修法：点选并收藏真正相关段落。

误区 5：Toolformer 全靠人工标注

错因：把 WebGPT 的人类示范套到所有工具。后果：看不懂自动资料筛选。修法：理解少量示例 + 自动插入 + 概率筛选。

误区 6：只要有工具，小模型也能变强

错因：把工具当外挂。后果：低估工具选择和整合难度。修法：评估模型是否真的会在合适时机调用工具。

误区 7：工具调用过程不用给用户看就不用验收

错因：隐藏内部过程后忽略审计。后果：错误工具结果混进最终答案。修法：系统内部仍要记录调用、输入和返回值。

误区 8：多训练资料就是效果来源

错因：没看消融实验。后果：误解 Toolformer 的优势。修法：比较“训练见过 API，但测试禁用 API”和“测试可用 API”。

误区 9：工具型 AI 等于自主 Agent

错因：看到会调用工具就想象成全自动代理。后果：忽略规划、验证、纠错和权限控制。修法：逐步检查动作选择和结果整合。
