# 概念骨架 {#skeleton}
<!-- tag: MAP -->

第一层：谁来做路由模型先判断任务应该交给哪个模型。简单任务给低成本模型，复杂任务给强模型，特定代码任务给擅长代码的模型。
第二层：怎么讨论一个模型可以自我反省，但它容易保留自己的旧答案。多个模型互看彼此答案，更容易产生新刺激和修正。
第三层：何时停止讨论需要裁判模型或规则判断是否达成共识，同时设置最大回合、质量门槛和失败退路，避免无效循环。

| 机制 | 解决的问题 | 适合场景 | 主要风险 |
| --- | --- | --- | --- |
| 模型路由 | 成本与能力不匹配。 | 客服、批量摘要、平台问答、难度差异很大的任务。 | 路由误判，把困难任务发给弱模型。 |
| 自我反省 | 让同一模型检查自己的答案。 | 格式检查、遗漏补充、低成本二次审查。 | 模型太相信自己的旧答案，修正幅度有限。 |
| 多模型讨论 | 引入不同答案，增加推翻错误的机会。 | 开放翻译、数学推理、方案比较、需要多视角的任务。 | 讨论成本上升，模型也可能互相带偏。 |
| 裁判模型 | 判断是否达成共识，并整理最终输出。 | 多轮讨论、辩论、多人角色协作。 | 裁判标准含糊，会过早结束或误判一致。 |
| 角色化 Agent 团队 | 把复杂项目拆成规划、实现、测试和验收。 | 软件原型、产品方案、研究整理、模拟组织。 | 角色只是提示词，不等于真实能力；复杂项目仍可能失败。 |

现在你应该能做什么：看到“让模型合作”时，先判断它属于路由、讨论、裁判还是角色分工，而不是把它们混成一个词。

# 术语表 {#terms}
<!-- tag: TERMS -->

| 术语 | 直觉解释 | 准确说法 | 常见误解 |
| --- | --- | --- | --- |
| 模型合作 | 不让一个模型包办全部工作。 | 通过路由、讨论、角色分工、裁判或工具链，把多个模型的输出组织成一个系统。 | 以为模型越多一定越好。 |
| 路由模型 | 分派工作的模型。 | 根据任务难度、成本、风险、领域等特征，决定调用哪个模型或哪条处理链路。 | 把它当成普通聊天模型，而没有设计判定标准。 |
| FrugalGPT | 省钱地用 GPT。 | 一类用模型选择、级联调用等方法降低成本，同时尽量保持回答质量的技术思路。 | 只理解成“都用便宜模型”。 |
| 自我反省 | 让模型检查自己的答案。 | 模型先生成答案，再读取自己的输出并尝试修正。 | 以为反省一定能发现根本错误。 |
| 多模型讨论 | 让不同模型互看答案再修改。 | A 给答案，B 阅读 A 的答案提出替代方案，A 再根据 B 的意见更新，形成多轮互动。 | 以为复制两个模型的输出就是讨论。 |
| 建设性反对 | 允许不同意，但要有理由。 | 提示模型把对方观点当参考，可以指出问题、给替代方案，也可以在理由充分时同意。 | 写成“无论对方说什么都反对”。 |
| 共识 | 不同模型的核心结论已经一致。 | 答案在关键事实、推理路线、建议动作上没有实质冲突，剩余差异只是表达或细节。 | 看到两个模型都说“同意”就算共识。 |
| 裁判模型 | 判断讨论是否可以收束的模型。 | 读取各模型输出和讨论历史，判断是否一致、是否还缺证据，并可生成最终摘要。 | 让裁判凭感觉评分，没有给判定标准。 |
| 讨论拓扑 | 模型之间怎么交流。 | 全员互看、经理汇总、链式传递、两方辩论加裁判等交流结构。 | 以为只有“一起聊天”一种形态。 |
| 角色提示 | 用 prompt 指定模型职责。 | 告诉模型扮演项目经理、工程师、测试员、用户等角色，并明确输入、输出和边界。 | 只写“你是程序员”，没有验收标准。 |
| Agent 团队 | 由多个模型角色组成的工作流。 | 多个模型或同一模型的多个角色按流程协作，例如 MetaGPT、ChatDev 一类系统。 | 以为它已经能稳定替代真实团队。 |
| 动态团队优化 | 根据表现调整谁参与。 | 让模型互评或用外部指标评价成员表现，减少低贡献角色，保留有效角色。 | 只靠模型互相打分，不做外部验证。 |

现在你应该能做什么：把“多模型讨论”讲成具体流程：谁先答、谁反驳、谁判断、什么时候停。

# 讲义主线 {#lecture}
<!-- tag: LECTURE -->
<!-- layout: grid2 -->

## box
### 1. 强模型也不是所有问题的唯一答案
字幕开头用“强者也有破绽、合作能补破绽”的故事引出主题。放到语言模型里，GPT-4 这类模型确实能力强，但它有一个很现实的问题：贵。一个平台如果把所有问题都丢给最贵模型，简单问候、固定格式摘要和低风险分类也要付同样成本，这在大量请求下不划算。
合作的第一种意义让系统知道“这题该谁做”。简单任务交给便宜模型，难任务再升级到强模型。用户看到的是统一入口，背后服务的模型却可能不同。判断句：当任务难度差异很大、请求量大、成本敏感时，先考虑路由，而不是先考虑辩论。
现在你应该能做什么：解释为什么“最强模型全部包办”在工程上经常不是最优选择。

## box soft
### 2. 路由模型：先判断任务，再选择执行者
路由模型的工作像分诊：新任务进来，它不一定自己回答，而是判断应该交给哪个模型。判断依据可以包括任务类型、长度、风险、领域、是否需要推理、是否需要代码、用户是否付费、回答失败的代价等。它可以是语言模型，也可以是传统分类器或规则系统。
好的路由在质量、成本、延迟之间做选择。比如客服 FAQ 可以先用小模型；涉及法律、医疗、财务或复杂代码的问题要升级；小模型回答低置信时可以 fallback 到强模型。判断句：路由系统必须有失败退路，否则省下的钱会被错误答案的后果吃掉。
现在你应该能做什么：为一个问答平台写出至少三个路由依据。

3. 自我反省的限制：模型容易相信自己前面课程讲过让模型反省：模型先给答案，再读自己的答案，检查是否有错。这对格式、遗漏、轻微推理错误有帮助，但它有天然限制。旧答案来自同一个模型，它往往已经符合模型当时的偏好，所以第二轮检查未必愿意彻底推翻。字幕引用的研究把“自我反省”和“多模型讨论”放在一起比较：多模型讨论中，模型更可能推翻之前的答案。原因不神秘，另一个模型给了不同表达、不同假设或不同错误线索，这会打断原模型的惯性。判断句：如果问题需要新视角，自我反省不够时，可以引入第二个模型。现在你应该能做什么：说明为什么“再问自己一次”不等于“得到独立意见”。
4. 多模型讨论有轮次的互动字幕里的翻译 demo 很具体：先让 Claude 翻译“葬送的芙莉莲”，再把 Claude 的答案给 GPT-3.5，要求它参考但不必同意；然后把 GPT-3.5 的答案再给 Claude，如此多轮交换。这个过程里，每一轮模型都看到对方的方案，并决定接受、修改或提出新方案。这里最关键的一句提示是“你不需要完全同意我的看法”。如果不写这句，模型很容易礼貌附和，说“你的翻译很好”，讨论立刻结束。讨论想有效，需要允许不同意见，也需要要求理由。判断句：讨论机制的核心每轮输入是否带来可检查的新理由。现在你应该能做什么：设计一个两模型讨论流程，而不是只把两个答案并排放。
5. 讨论轮数和参与人数：有收益，也有边际递减研究结果通常会显示：更多模型参与、更多轮讨论，有机会提高数学题等任务的正确率。直觉上，这像多人解题：一个人漏掉的条件，另一个人可能补上；一个人算错，另一个人可能指出。但这不意味着越多越好。每增加一个模型和一轮讨论，就增加成本、延迟、上下文长度和互相带偏的可能。字幕里提到有实验发现三轮和四轮已经很接近，于是停在四轮。判断句：讨论系统要先设预算和停止条件，再谈效果提升。现在你应该能做什么：解释为什么多轮讨论需要最大回合数。
6. 讨论拓扑：谁看谁的答案，会影响结果“Exchange of Thought”一类研究关心的是讨论结构。三个模型可以全员互看所有答案，也可以由 A 当老板、B 和 C 只向 A 报告；也可以按 A 到 C、C 到 B、B 到 A 的链式传递；还可以让 B 和 C 辩论，A 当裁判。这些结构没有固定赢家。全员互看信息充分，但成本高，容易互相影响；经理汇总结构清楚，但经理模型的判断很关键；链式结构便宜一些，但前面错误可能一路传下去；辩论加裁判适合争议任务，但需要清楚的评判标准。判断句：拓扑要跟任务匹配，而不是默认所有模型开群聊。现在你应该能做什么：为“翻译标题”和“开发功能”选择不同讨论结构。
7. 裁判模型：把讨论收束成结论多模型讨论一定会遇到一个问题：什么时候算讨论完？一个做法是引入裁判模型。裁判读取 A 和 B 的回答，以及它们后续的修正，判断关键结论是否一致。如果没有达成共识，就让模型继续讨论；如果已经一致，就整理最终答案。裁判不能只问“你觉得他们一致吗”。更稳的提示要写明：比较哪些关键点、哪些差异可以忽略、哪些冲突必须继续讨论、最终答案要保留哪些证据。工程上还要加最大回合数和 fallback。虽然字幕提醒现阶段更常见的问题是讨论太快结束，但真正做系统时仍要防止循环。判断句：裁判模型不是权威本身，它需要明确的判定标准。现在你应该能做什么：写出裁判模型应该检查的三类内容。
8. 反对提示的分寸：别太软，也别太硬语言模型经常表现得很礼貌。你让它“根据另一个模型的建议回答”，它可能很快顺着对方走；你如果写“无论对方说什么都反对”，它又会为了反对而反对，质量反而下降。字幕里强调较好的写法是：对方观点只是参考，你不需要完全同意，可以提出自己的意见。这就是建设性反对。它允许模型不同意，但要求给理由和替代方案；它也允许模型同意，但要说明为什么对方方案足够好。判断句：有效讨论需要独立判断，不需要表演冲突。现在你应该能做什么：把“请反驳对方”改写成更稳的讨论提示。
9. 角色分工：让模型扮演项目经理、工程师和测试员当任务从单个问题变成一个项目，单纯讨论答案不够。你可能需要项目经理先拆需求，工程师写代码，测试员设计用例，用户角色提出反馈，最后再由项目经理决定下一步。这样的系统把复杂工作拆成多个可交接的阶段。角色可以来自模型本身的专长，也可以来自提示词。比如代码模型适合工程师角色；通用模型可通过 prompt 扮演 project manager、programmer、user。关键职责、输入、输出、验收标准和交接格式。判断句：角色提示如果没有交付物要求，就只是换了称呼。现在你应该能做什么：为一个小软件项目列出 PM、工程师、测试员各自的输出。
10. Agent 团队工具：MetaGPT、ChatDev 与想象边界字幕提到 MetaGPT、ChatDev 这类开源项目，它们把一批语言模型角色组织成“公司”：产品经理、架构师、项目经理、工程师等。用户给一个任务，系统尝试让这些角色按流程工作。这个想法很有启发，但不能误读成“现在已经能稳定替代真实团队”。学术论文和开源 demo 常在相对简单的任务上展示效果。真实软件项目会遇到需求含糊、依赖冲突、运行环境、测试覆盖、安全、成本、长上下文管理等问题。判断句：Agent 团队可以作为原型和辅助流程，不能因为角色很多就跳过真实验证。现在你应该能做什么：说出 Agent 团队在真实项目里至少三个仍需人工验收的环节。
11. 动态团队优化：让成员接受评价字幕提到 Dynamic LLM Agent Network 一类工作：让每个模型根据刚才的工作内容评价其他模型表现，给分后把低分成员减少或移出。这个思路像团队复盘，目标是保留有贡献的角色，减少无效沟通。但互评也有风险。模型可能因为语言风格而偏好某个回答，可能互相吹捧，也可能把真正提出异议的成员打低分。更稳的优化要结合外部指标：测试是否通过、事实是否可查、用户是否满意、成本是否下降。判断句：团队优化不能只看模型互评，要看任务结果。现在你应该能做什么：判断一个“模型互相打分”的方案缺什么验证。
12. 从团队到社群：生成式 Agent 的下一步想象字幕最后提到 Stanford 的生成式 Agent 小镇。它展示的是一群由语言模型驱动的虚拟人物如何记忆、计划、互动，甚至形成社会行为。这个例子说明：当模型有记忆、计划、角色和交流机制时，它们不只是在回答单题，也可以组成更复杂的模拟环境。这不是本节的操作重点，但它把前面的技巧串起来：提示词塑造角色，记忆保存上下文，规划决定行动，多 Agent 通讯形成互动。判断句：Agent 社群看起来复杂，底层仍然是提示、记忆、计划、通信和评估这些组件的组合。现在你应该能做什么：把“AI 村庄”还原成可分析的系统组件，而不是只看热闹。
完整知识链路P32 的主线是：单个强模型很有用，但系统工程不只追求单点最强。成本敏感时用路由模型把任务分给合适模型；需要新视角时用多模型讨论替代单模型自我反省；讨论要靠建设性反对保持独立性，也要靠裁判模型和停止条件收束；复杂项目可以用角色化 Agent 团队拆成规划、实现、测试和复盘；最后再用外部结果优化团队。你现在应该能做什么：遇到一个 LLM 应用需求时，能选择“单模型、路由、讨论、裁判、团队”中的合适组合。

# 最小例子 {#example}
<!-- tag: EXAMPLE -->

下面的例子把本节机制压缩成可以读懂的最小流程。重点看输入、判断、交接和停止条件。

示例 1：路由模型先决定谁来答python-ish复制

```text
def choose_model(task):
 if task.risk in ["legal", "medical", "financial"]:
 return "strong-model-with-review"
 if task.type == "code" and task.needs_execution:
 return "coding-specialist"
 if task.length < 300 and task.intent in ["faq", "rewrite", "format"]:
 return "cheap-fast-model"
 return "strong-general-model"

answer = call_model(choose_model(task), task.prompt)
if answer.confidence_low:
 answer = call_model("strong-general-model", task.prompt)
```

这段流程先判断风险、领域、长度和意图，再决定调用哪个模型。最后的 fallback 很重要：路由错了，要能升级。判断句：路由为了把成本花在该花的地方。

示例 2：两模型讨论加裁判prompt flow复制

```text
Round 1:
 A: Give your answer and the reasons.
 B: Read A's answer as reference. You do not need to agree.
 If you disagree, explain the issue and propose an alternative.

Judge:
 Compare A and B on facts, reasoning, and final recommendation.
 If they still conflict on key points, ask for one more round.
 If they agree on key points, summarize the final answer.

Stop:
 max_rounds = 4
 consensus_required = facts + reasoning + action
```

这里让它们保留独立判断。裁判要检查关键事实、推理和建议是否一致。判断句：没有裁判或停止条件的讨论，很难变成稳定系统。

示例 3：角色化 Agent 团队的交接workflow复制

```text
Project Manager:
 output = requirements, milestones, acceptance criteria

Engineer:
 input = PM.output
 output = implementation plan + code changes

Tester:
 input = Engineer.output + acceptance criteria
 output = test cases + failures + reproduction steps

Project Manager:
 input = Tester.output
 decision = ship / revise / reduce scope
```

角色分工的关键是交接物。PM 不能只说“做一个网站”，工程师不能只说“我写好了”，测试员不能只说“看起来可以”。每个角色要产出下一个角色能使用的材料。判断句：Agent 团队的质量取决于交接格式和验收标准。

精修直觉：合作是系统设计，“哪一段工作需要谁来做”。路由解决成本和能力匹配，讨论解决不同假设碰撞，裁判解决何时收束，角色团队解决复杂项目交接。只有当任务真的需要这些机制时，多 Agent 才有价值。
术语快辨：路由、讨论、裁判路由发生在回答前，决定任务交给谁；讨论发生在生成过程中，让多个答案互相修正；裁判发生在收束阶段，判断关键事实、推理和建议是否一致。三者可以组合，但不能混用：让两个模型辩论不能替代路由，让裁判打分也不能自动证明答案正确。
Worked example：客服平台怎么选机制10 万条简单 FAQ 不适合每条开辩论，先用小模型或规则路由处理，低置信和高风险问题升级强模型；涉及复杂退款争议时，可以让两个模型分别给处理建议，再由裁判按政策条款比较；最终仍要保留人工复核入口。这个例子说明：协作系统要先看任务分布、风险、成本和验收标准。

# 误区诊断 {#diagnosis}
<!-- tag: DIAGNOSIS -->

| 误区 | 错因 | 后果 | 修法 |
| --- | --- | --- | --- |
| 所有任务都交给最强模型。 | 只看能力，不看成本、延迟和规模。 | 简单任务也高成本，系统难以扩展。 | 引入路由、分级调用和 fallback。 |
| 以为多模型一定比单模型好。 | 忽略任务类型和评估指标。 | 成本变高，答案可能互相带偏。 | 只有在需要多视角、降低错误或角色分工时才加模型。 |
| 提示模型“请参考对方意见”。 | 没有保留独立判断空间。 | 模型过早附和，讨论很快结束。 | 写明“可参考，但不必同意；不同意要给理由和替代方案”。 |
| 要求模型无条件反对。 | 把独立判断误解成表演冲突。 | 产生无意义争辩，反而降低质量。 | 使用建设性反对，允许有理由地同意。 |
| 没有裁判和停止条件。 | 只设计讨论，没有设计收束。 | 输出不稳定，或者讨论拖太久。 | 设置裁判标准、最大回合数、共识条件和失败退路。 |
| 只写角色名，不写职责。 | 把 Agent 团队当角色扮演。 | PM、工程师、测试员输出混乱，无法交接。 | 给每个角色明确输入、输出、验收标准和边界。 |
| 只靠模型互评优化团队。 | 互评容易受表达风格和从众影响。 | 真正有贡献但爱指出问题的角色可能被低估。 | 结合测试结果、事实核验、用户反馈和成本数据。 |
| 把开源 Agent demo 当成熟生产系统。 | 忽略真实项目的环境、依赖、安全和长上下文问题。 | 原型看似运行，交付不可用。 | 从小任务试点，保留人工验收和可回滚流程。 |

路由 vs 讨论路由解决“谁来做”，讨论解决“答案如何互相修正”。如果你只是想省成本，用讨论会把问题弄复杂。
自我反省 vs 独立意见自我反省便宜，但独立性弱；多模型讨论贵一些，但更容易引入不同假设。
角色提示 vs 真实能力角色提示能改变输出方式，不保证模型真的具备对应能力。工程师角色写出的代码仍要运行和测试。
