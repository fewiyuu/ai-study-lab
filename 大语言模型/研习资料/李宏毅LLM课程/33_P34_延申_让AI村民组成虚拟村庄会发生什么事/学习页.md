# 概念骨架 {#skeleton}
<!-- tag: MAP -->

角色层：人设给行为方向每个 Agent 有身份、职业、关系和目标。例如 Isabella 开咖啡店，并计划在情人节办活动。
认知层：计划、记忆、反思Agent 先计划一天，再记录观察。记忆太杂时，通过反思把事件整理成高层想法。
行动层：语言到环境指令ChatGPT 产生“要做什么”的自然语言说明，环境还需要转译器把它变成可执行动作。

| 模块 | 输入 | 输出 | 为什么需要 |
| --- | --- | --- | --- |
| 人设 Profile | 角色背景、职业、目标、关系。 | 角色长期倾向。 | 避免所有 Agent 说话做事都一样。 |
| 计划 Planning | 人设、日期、昨天经历、当前目标。 | 一天计划、小时计划、五分钟计划。 | 让 Agent 记得自己做过什么，避免反复吃午餐这类怪行为。 |
| 观察 Observation | 环境状态、周围物体、他人行为。 | 大量事件记录。 | 让 Agent 能根据外界变化改计划。 |
| 记忆 Memory | 观察、计划、反思结果。 | 可检索的事件库。 | 给后续行动和对话提供上下文。 |
| 反思 Reflection | 近期重要事件和相关记忆。 | 高层洞察或解释。 | 把琐碎事件压缩成可用理解。 |
| 检索 Retrieval | 当前问题或观察对象。 | 相关记忆片段。 | 避免把所有记忆一次塞进 prompt。 |
| 反应与对话 | 当前观察、相关记忆、计划。 | 行动说明或对话内容。 | 让 Agent 根据环境变化行动，而不是死按时间表。 |

现在你应该能做什么：把 AI 村庄看成“人设 + 计划 + 记忆 + 反思 + 行动转译”的系统，而不是一群模型随便聊天。

# 术语表 {#terms}
<!-- tag: TERMS -->

| 术语 | 直觉解释 | 准确说法 | 常见误解 |
| --- | --- | --- | --- |
| Generative Agents | 会生成行为的虚拟角色。 | 由语言模型驱动，带有记忆、计划、反思和行动生成能力的 Agent。 | 以为它们只是聊天机器人换皮。 |
| Agent | 能观察、思考并行动的角色。 | 接收环境信息，根据内部状态生成行动，并影响环境的系统单元。 | 以为只要调用 LLM 就是 Agent。 |
| 人设 Profile | 角色的背景说明。 | 包含身份、年龄、职业、关系、目标、偏好等长期设定。 | 只写一个名字，不写可影响行动的信息。 |
| 计划 Planning | 先安排今天怎么过。 | 根据角色背景和日期生成日程，并可细化到小时或五分钟粒度。 | 以为计划越细越好，不需要根据新观察修改。 |
| 观察 Observation | 角色看到的环境事件。 | 环境定期把周围物体、人和状态转成文本事件传给 Agent。 | 以为每条观察都同样重要。 |
| 记忆 Memory | Agent 的经历记录。 | 存放观察、计划和反思结果的可检索资料库。 | 把记忆当完整聊天记录，没有检索和筛选。 |
| 重要性 Importance | 这件事值不值得记。 | 用模型给事件打分，分数高的事件更可能触发反思或被检索。 | 以为所有事件都应该进 prompt。 |
| 及时性 Recency | 最近发生的更可能相关。 | 检索记忆时，较新的事件通常有更高权重。 | 只看重要性，忽略时间。 |
| 关联性 Relevance | 这条记忆和当前问题像不像。 | 用 embedding 相似度等方法估计问题与记忆的语义接近程度。 | 只用关键词匹配。 |
| 反思 Reflection | 从经历里想出高层结论。 | 模型根据近期重要记忆生成问题、检索证据，再产出 insights 并写回记忆。 | 以为反思一定真实可靠。 |
| 行为说明 | 模型说自己要做什么。 | 自然语言形式的行动意图，例如“Isabella goes to sleep”。 | 以为语言模型直接控制游戏引擎。 |
| 转译器 | 把语言变成环境动作。 | 将行为说明映射成环境可执行指令的规则或程序。 | 忽略 LLM 输出到真实行动之间的工程层。 |

现在你应该能做什么：解释为什么记忆检索要同时考虑重要性、及时性和关联性。

# 讲义主线 {#lecture}
<!-- tag: LECTURE -->
<!-- layout: grid2 -->

## box
### 1. AI 村庄到底在做什么
论文把 25 个由 ChatGPT 驱动的 Agent 放进一个虚拟村庄，让它们运作两天。每个角色都有自己的背景和目标：有人开咖啡店，有人上学，有人做音乐创作。系统让语言模型根据人设、记忆和环境观察产生行动。
demo 网站展示的是已经跑完的结果重播，不是一个持续开放的实时社会实验。这个细节很重要，因为它提醒我们不要把演示误读成无限运行、无约束的真实社会。判断句：Generative Agents 是一个受控模拟系统，不是放任模型自由生活。
现在你应该能做什么：用“虚拟环境 + 多个 LLM Agent + 两天模拟”概括这篇工作的基本设置。

## box soft
### 2. 语言模型不能直接移动角色
ChatGPT 产生的是自然语言，不是游戏引擎能直接执行的动作。因此系统通常让模型先生成行为说明，例如“Isabella 去睡觉”，再由转译器把这句话映射成环境动作，让角色移动到床边或进入睡眠状态。字幕推测这部分转译可能主要靠规则完成。
这点对理解 Agent 很关键。LLM 负责高层决策和语言生成，环境执行层负责把决策落地。缺少转译层，模型只会说“我要去咖啡店”，角色不会真的出现在咖啡店。判断句：从“模型想做什么”到“环境发生什么”之间必须有接口。
现在你应该能做什么：区分行为说明和可执行环境指令。

3. 为什么需要计划：避免反复做同一件事如果只把当前时间和周围观察丢给模型，它可能到了 12 点吃午餐，12:30 又觉得该吃午餐，1 点再吃一次。因为模型没有稳定的日程状态，也没有明确知道自己已经完成了哪些安排。解决办法是让 Agent 先规划一天。系统给它人设、昨天做了什么、今天日期和目标，让模型生成今日计划。计划先粗后细：先给一天安排，再按小时细化，必要时再按五分钟粒度规划。判断句：计划不是装饰，它让 Agent 的行为跨时间保持一致。现在你应该能做什么：解释为什么没有计划的 Agent 容易行为循环或失去连贯性。
4. 粗计划到细计划：层级分解字幕里的角色 Eddie 先得到一天计划：早上起床、上课、下午做音乐创作、晚上吃饭和睡觉。到了下午某个时间点，再把“做音乐创作”细化成每小时做什么；到了更近的时间，再细化成每五分钟拿点心、散步、整理工作区等动作。这种层级计划有两个好处：长期目标不容易丢，短期动作又足够具体。一次生成全天五分钟计划会很长，也不灵活；完全没有细化则无法行动。判断句：Agent 的计划应该有粒度变化，离当前越近越具体。现在你应该能做什么：把“下午学习”细化成小时计划和五分钟动作。
5. 观察流很庞杂：不是所有事件都值得用环境会不断把周围状态传给 Agent。角色在房间里可能反复收到“桌子没有发生什么”“床没有发生什么”“衣柜没有发生什么”。真实世界也是这样，我们看到大量东西，但大多数对当前行为没有意义。如果把所有观察都塞进 prompt，成本高、噪声大，模型也难以抓重点。所以系统需要筛选和整理。事件会被评估重要性，重要事件累积到一定程度才触发反思。判断句：Agent 记忆系统的第一难题信息太多。现在你应该能做什么：说明为什么“看到的一切都进上下文”不是好设计。
6. 重要性评分：让模型判断事件值几分论文让 ChatGPT 给事件重要性打 1 到 10 分。刷牙、整理床铺可能只有 1 分；分手、被大学录取这类影响人生走向的事件可能是 10 分。这个分数不是绝对真理，但它给系统一个可操作的筛选信号。重要性分数可以用于两个地方：决定何时触发反思，也影响记忆检索时哪些事件更容易被取出。如果一段时间里高重要性事件累积很多，Agent 就开始反思。判断句：重要性评分把“哪些记忆值得思考”变成可以计算的条件。现在你应该能做什么：为三条生活事件分别给出重要性理由。
7. 反思：从琐碎记忆生成高层洞察反思重新解释记忆。系统会先让模型基于近期重要事件提出几个可以回答的问题，例如“某两个人是否在交往”。然后根据问题去记忆库里检索相关事件，最后让模型提出 high-level insights，并标明这些洞察来自哪些记忆条目。反思结果会写回记忆库，成为之后行动和对话的新材料。这样 Agent 不只是记得“我看到 A 和 B 一起走过三次”，还可能形成“A 和 B 关系很亲近”的高层理解。判断句：反思把事件记录变成解释，但解释可能会错。现在你应该能做什么：区分“观察到的事实”和“由事实反思出来的解释”。
8. 反思可能产生妄想，也可能被人为植入字幕里提到 Maria 和 Klaus 互相暗恋的反思条目。有些 reflection 带有 evidence，能看出它来自哪些观察；有些没有 evidence，可能是作者额外植入的设定，也可能是系统中没有展示完整证据。这提醒我们：反思结果不是事实本身。如果 Agent 根据错误反思行动，就可能把普通互动理解成强关系，产生“脑补”。这对虚拟故事可能有趣，对真实应用却危险。判断句：反思条目必须保留证据来源，否则很难判断它是洞察还是幻觉。现在你应该能做什么：看到 Agent 的“想法”时追问 evidence 从哪里来。
9. 记忆检索：importance、recency、relevance 三路合并当 Agent 要回答问题或决定行动时，不能把全部记忆塞进 prompt。系统会检索相关记忆。检索时看三件事：重要性，事件本身有多关键；及时性，是否最近发生；关联性，这条记忆和当前问题是否语义相关。关联性可能通过 sentence embedding 之类方法计算，把当前问题和记忆片段转成向量，再比较相似度。最后综合三个分数，选出最值得放进上下文的记忆。判断句：好的记忆检索不是只拿最新消息，也三者平衡。现在你应该能做什么：解释为什么“昨天的重要事件”有时比“刚才的桌子没动”更值得检索。
10. 根据观察修改计划：桌子没事和桌子着火不一样每天计划不是铁板一块。每当 Agent 看到某个 entity，系统会问：这个东西和我是什么关系？它现在的状态是什么？如果桌子只是静止不动，可能不需要反应；如果桌子着火，即使它平时只是普通家具，也必须改变计划。系统会根据当前观察，从记忆里找相关计划、观察和反思，生成摘要，再让模型判断是否要 react。如果需要，就产生新的计划或行动。判断句：Agent 的行为既要遵守计划，也要能被重要外部事件打断。现在你应该能做什么：说明计划系统为什么需要“中断机制”。
11. 对话生成：先检索对方相关记忆，再开口当 John 看到 Eddie 并决定交谈时，系统不会让模型凭空说话。它会先检索 John 脑中和 Eddie 有关的记忆，例如 Eddie 正在做音乐创作。然后生成对话：“你的音乐创作计划进展如何？” Eddie 收到这句话后，也会根据自己对 John 的记忆和当前状态回复。这个流程让对话具有上下文。两个人不只是随机寒暄，而是围绕彼此的关系和近期事件说话。不过字幕也提醒，这些 Agent 讲话通常很温和、无聊，因为背后的模型倾向安全礼貌。判断句：对话自然性来自角色记忆，但风格仍受模型训练和安全偏好限制。现在你应该能做什么：描述一个 Agent 说话前应该检索哪些记忆。
12. 信息传播：情人节活动如何扩散Isabella 要在 2 月 14 日下午 5 点办情人节活动。她看到路过的人就邀请参加。被邀请的人又可能告诉亲友，于是消息在小镇的人际网络中扩散。有人转述准确，有人把活动说成读书会，最后并不是所有答应的人都会出现。这个例子展示了多 Agent 系统的涌现感：单个角色只是记住消息、转述消息、根据日程决定去不去；多个角色连起来，就像形成了社区传播和活动参与。判断句：多 Agent 模拟的有趣之处常来自局部规则叠加后的整体效果。现在你应该能做什么：用“邀请、转述、计划冲突、到场”解释情人节活动的传播链。
13. 能力边界：它像社会，但不是社会本身Generative Agents 很有启发，但要避免过度解读。它的村民行为发生在简化环境里，运行时间短，动作空间有限，转译规则由人设计，人设和某些记忆也可能被人为设置。模型说的话温和稳定，不代表真实人类互动的复杂性。对学习者来说，这个系统的价值在于架构：怎样把 LLM、记忆、计划、反思和环境接口组合起来。它不能证明 Agent 有意识，也不能证明虚拟村庄能可靠预测现实社会。判断句：把 demo 当系统设计案例学，不要把它当社会学结论。现在你应该能做什么：说出 AI 村庄 demo 的至少三个限制。
完整知识链路P34 的完整链路是：给每个虚拟村民设定人设和目标；让 LLM 生成自然语言计划，并逐层细化到可执行动作；环境不断提供观察，但观察太多，需要重要性评分；计划、观察和反思都进入记忆库；需要行动或对话时，用重要性、及时性、关联性检索相关记忆；反思把琐碎事件整理成高层洞察，但必须注意证据和幻觉；最后，多个 Agent 的局部行动会形成消息传播、活动参与和关系互动。你现在应该能做什么：为一个小型多 Agent demo 画出记忆、计划、反思和行动的流程。

# 最小例子 {#example}
<!-- tag: EXAMPLE -->

下面用伪代码压缩 Generative Agents 的核心。重点看清“语言模型负责生成，系统负责存取记忆与执行动作”。

示例 1：从人设生成日计划prompt flow复制

```text
profile:
 Eddie, 19 years old, studies music, wants to compose a new song.

context:
 Yesterday Eddie went to class, practiced music, had dinner, slept at 23:30.
 Today is Wednesday, February 13.

prompt:
 Write Eddie's plan for today.
 First give a coarse daily schedule.
 Later expand the current hour into smaller actions.
```

计划不是一次性越细越好。先有粗日程，再在临近时细化，能让行为既连贯又灵活。

示例 2：记忆检索评分python-ish复制

```text
score(memory, query) =
 0.4 * importance(memory)
+ 0.3 * recency(memory)
+ 0.3 * relevance(memory, query)

selected_memories = top_k(memory_bank, key=lambda m: score(m, current_question))
```

这不是论文的精确公式，只是帮助理解。重要性、及时性、关联性分别回答“值不值得”“是不是最近”“和当前问题像不像”。

示例 3：反思写回记忆workflow复制

```text
observations:
 - Maria talked with Klaus at the cafe.
 - Maria smiled when Klaus mentioned the party.
 - Klaus asked whether Maria would attend the party.

reflection_prompt:
 What high-level insights can be inferred from these observations?
 Cite the memory IDs used as evidence.

new_memory:
 Maria and Klaus may be interested in spending time together.
 evidence: obs_12, obs_18, obs_23
```

反思最好带 evidence。没有证据的“洞察”也许适合写故事，但在严肃 Agent 系统里会变成不可追踪的幻觉。

精修直觉：Agent 是带状态的循环普通聊天机器人主要回答当前文本；Generative Agent 多了状态循环：人设给长期方向，计划给时间结构，观察带来外部变化，记忆保存经历，反思压缩高层理解，转译器把语言意图落到环境动作。少掉任何一环，村民就容易失忆、乱跑、重复行动或只会说不会做。
术语快辨：观察、记忆、反思观察是环境发生了什么，属于较低层事件；记忆是可检索的存储，不等于把所有历史塞进 prompt；反思是从多条记忆推出来的解释，需要 evidence。把反思当事实，会让 Agent 根据脑补行动；把所有观察当同等重要，会让噪声淹没有用信息。
Worked example：情人节活动如何传播Isabella 发出邀请，这是观察和对话事件；被邀请的人把消息写入记忆；之后遇到朋友时，检索到“活动邀请”并转述；对方根据自己的计划决定是否参加。最终出现“多人知道活动、部分到场、部分遗忘或转述失真”的整体效果。这个例子展示局部 Agent 规则如何叠加出社区感。

# 误区诊断 {#diagnosis}
<!-- tag: DIAGNOSIS -->

| 误区 | 错因 | 后果 | 修法 |
| --- | --- | --- | --- |
| 只给 Agent 一个名字就算有人设。 | 缺少职业、目标、关系和长期偏好。 | 角色行为相似，无法形成稳定差异。 | 写清背景、目标、关系和当前计划。 |
| 让 Agent 不做计划，直接根据当前时间行动。 | 没有跨时间状态。 | 反复吃饭、重复行动、日程不连贯。 | 生成日计划，并逐层细化当前时段。 |
| 把所有观察都塞进 prompt。 | 没有筛选记忆。 | 上下文爆炸，重要事件被噪声淹没。 | 用重要性、及时性、关联性检索。 |
| 把 reflection 当事实。 | 混淆观察与解释。 | Agent 可能根据脑补行动。 | 反思条目要带 evidence，并在高风险场景校验。 |
| 忽略语言到环境动作的转译层。 | 以为 LLM 直接控制世界。 | 模型说了行动，但环境没有可执行指令。 | 设计动作 schema、规则映射和失败处理。 |
| 计划生成后永不修改。 | 没有外部事件中断机制。 | 桌子着火还按原计划读书。 | 观察实体关系和状态，必要时改计划。 |
| 把 demo 当真实社会模拟。 | 忽略环境简化、人设植入和运行时间短。 | 过度解读结果。 | 把它当 Agent 架构案例，而不是社会结论。 |
| 只看多 Agent 的涌现，不看成本。 | 忽略调用次数、上下文和检索开销。 | 小 demo 可跑，真实应用成本失控。 | 限制运行范围、压缩记忆、控制反思频率。 |

观察 vs 反思观察是“发生了什么”；反思是“这些事可能意味着什么”。后者有用，但可能错。
计划 vs 反应计划保证长期连贯，反应处理外部变化。只要其中一个，Agent 都会僵硬。
LLM 输出 vs 环境动作LLM 生成的是语言意图，环境需要结构化动作。接口设计决定 demo 能否真正动起来。
