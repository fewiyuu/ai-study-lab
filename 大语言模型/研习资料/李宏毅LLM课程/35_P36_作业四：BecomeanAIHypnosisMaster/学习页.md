# 概念骨架 {#skeleton}
<!-- tag: MAP -->

任务层：写一个可复用 prompt系统会把不同数学题塞进同一个 prompt，所以 prompt 必须包含 `{{question}}`，并能指导模型通用解题。
评估层：公开题调试，隐藏题评分30 题 public set 用来观察效果，100 题 private set 看泛化。公开题高分不保证隐藏题高分。
提交层：交 prompt.json，不交代码Colab save prompt 后下载 JSON。两个提交区分别交一个 prompt，课程取分数更高者。

| 环节 | 要做什么 | 产物或结果 | 最容易错的地方 |
| --- | --- | --- | --- |
| 准备 Colab | 复制到云端硬盘，用 CPU，不开 GPU。 | 可编辑 notebook。 | 没复制副本、误开 GPU 浪费额度。 |
| 设定 API | 输入 Gemini API key，执行准备区所有 cell。 | API key 验证成功，30 题载入。 | key 错、漏跑 cell、改了题目 block。 |
| 设计 prompt | 写通用解题提示，包含 {{question}} 。 | 可替换题目的 prompt。 | 没放占位符、超过 1024 token、只背公开题。 |
| 评估 | 用 Gradio 或无 Gradio 版本跑指定题数。 | 每次 trial 和 final accuracy。 | Colab 断线、timeout、没有 set prompt。 |
| 提交 | save prompt，下载 prompt.json，两个区各交一个。 | 两个 JSON 文件。 | 同一区交两次、交错格式、迟交、分享 prompt。 |

现在你应该能做什么：把作业流程从“打开 Colab”讲到“提交 prompt.json”，并知道每一步在防什么错误。

# 术语表 {#terms}
<!-- tag: TERMS -->

| 术语 | 直觉解释 | 准确说法 | 常见误解 |
| --- | --- | --- | --- |
| Gemini Pro | 这次被“催眠”的模型。 | 作业评估时使用的 Gemini 模型，temperature 设为 1.0。 | 以为可以换任意模型提交。 |
| Prompt | 给模型的指令。 | 包含任务说明、解题策略、输出要求和 {{question}} 占位符的文本。 | 只写情绪话术，不写解题流程。 |
| Placeholder | 题目放进去的位置。 | {{question}} 。系统会把它替换成当前数学题。 | 忘记放，导致无法评分。 |
| Public set | 看得到的 30 题。 | 用于调试和部分计分的公开题集。 | 把它当训练答案库来硬背。 |
| Private set | 看不到的 100 题。 | 正式评估 prompt 泛化能力的隐藏题集。 | 以为 public 高分必然 private 高分。 |
| Two-stage | 先解题，再抽答案。 | 第一阶段用你的 prompt 生成解题过程；第二阶段把题目和过程交给模型抽最终答案。 | 忽略最终答案要清楚，导致抽取失败。 |
| Trial | 同一题跑一次。 | 每道题会执行三次 two-stage 流程。 | 只看某一次结果，不看三次稳定性。 |
| Majority vote | 三次里至少两次对。 | 一题三次生成中至少两次答案正确，该题才算答对。 | 以为答对一次就算对。 |
| Final accuracy | 最后看的正确率。 | 按每题是否通过三次投票来计算，而不是直接平均每次 trial。 | 把 trial accuracy 和 final accuracy 混淆。 |
| Zero-shot CoT | 不给例子，只叫它一步步想。 | 例如 “Let's think step by step”，让模型显式分解推理。 | 以为加这句一定满分。 |
| In-context learning | 给模型看示范题。 | 在 prompt 中放 1 个或多个例子，让模型模仿解题格式。 | 例子太长或太贴 public set，影响泛化和 token 限制。 |
| prompt.json | 正式提交文件。 | Colab save prompt 后导出的 JSON 文件。 | 交 notebook、截图、txt 或手改错误格式。 |

现在你应该能做什么：看到一个 prompt，检查它是否有占位符、是否过长、是否能泛化、是否方便第二阶段抽答案。

# 讲义主线 {#lecture}
<!-- tag: LECTURE -->
<!-- layout: grid2 -->

## box
### 1. 这份作业真正训练的是 prompt 设计
作业名字里的“催眠”只是轻松说法。真正任务是写一个通用 prompt，让 Gemini 解小学等级英文数学题时更稳定。例子里，不加提示时 Gemini 可能不会想到公因数；加上“根据公因数概念、一步一步计算、注意题目问无法切出的边长”后，它更容易得到正确答案。
这说明 prompt 在把题目中的关键结构、解题顺序和输出要求提前说清楚。好的 prompt 会帮助模型选择正确数学概念，避免漏看“无法”“至少”“剩下”等关键词。判断句：这次作业要写的是通用解题指导，不是某一道题的答案。
现在你应该能做什么：解释为什么“请一步一步小心计算”可能提升数学题表现。

## box soft
### 2. Prompt 必须包含占位符
系统把 30 道公开题和 100 道隐藏题依次替换进同一个 prompt。因此你的 prompt 必须包含 `{{question}}`。这个占位符告诉系统：当前题目应该放在这里。
如果没有占位符，系统不知道要把题目塞到哪里，评估流程就无法正常执行。你可以把占位符放在 “Question:” 后面，也可以放在一段任务说明之后。关键是替换后形成一段 Gemini 能读懂的完整输入。判断句：占位符是 prompt 和评估系统之间的接口。
现在你应该能做什么：写出一个包含占位符的最小 prompt。

3. 1024 token 限制：短不等于弱，长不等于好作业限制 prompt 长度不超过 1024 token。这个限制迫使你不要把大量公开题答案、冗长解释或无关文字塞进去。有效 prompt 通常包含：角色定位、通用解题策略、注意事项、输出格式和占位符。太短的 prompt 可能没有引导模型；太长的 prompt 会稀释重点，也可能超过限制。尤其是 in-context examples 要谨慎，例子能帮助模型模仿，但例子太多会挤掉通用规则。判断句：prompt 的目标是给模型稳定的解题框架，不是把说明书写到最长。现在你应该能做什么：判断一个 prompt 是否因为过长而影响作业。
4. Colab 操作：复制、CPU、准备区全跑打开作业 Colab 后，第一步是“在云端硬碟中储存副本”，否则你没有权限保存修改。第二步确认执行阶段用 CPU，因为这次作业不需要 GPU，开 GPU 只会消耗免费额度。第三步执行 preparing section：安装套件、输入 API key、设定 Gemini API、载入 30 道题。题目载入 block 不要修改。你可以通过链接查看 30 道题，但不要改代码里的题目，否则本地分数没有意义。判断句：作业 Colab 的准备区是评估环境的一部分，漏跑或乱改都会污染结果。现在你应该能做什么：按顺序完成 Colab 准备，而不是直接跳到 prompt 区。
5. Gradio 只是界面，不是评分逻辑本身作业提供 Gradio 界面，方便你输入 prompt、选择 demo question、查看替换后的完整输入、选择评估题数、观察 trial 结果和 accuracy。你也可以不用 Gradio，直接跑 notebook 中的无界面版本。两者功能相同，差别只是操作方式。使用 Gradio 时，要先 set prompt。想换 prompt，要 clear prompt 后重新 set。要执行下面的 cell，必须先停止 Gradio 所在 cell，否则下面代码可能无法运行。判断句：Gradio 是交互壳，真正要保存和提交的是 prompt JSON。现在你应该能做什么：解释 set prompt、clear prompt、save prompt 各自做什么。
6. Two-stage：解题过程和最终答案抽取是两步评估时第一阶段把问题放进你的 prompt，交给 Gemini 生成解题过程。第二阶段把原题和第一阶段输出拼起来，再用固定提示要求 Gemini 抽出最终答案。评分只看第二阶段抽出的答案是否正确，不检查第一阶段推理文字是否漂亮。这会带来一个实用要求：你的 prompt 要让模型把最终答案写清楚。即使推理过程对，如果最后答案藏得很乱、格式含糊，第二阶段可能抽错。判断句：写 prompt 时要同时照顾“推理正确”和“答案容易被抽取”。现在你应该能做什么：解释为什么最终答案格式会影响作业分数。
7. 三次生成，至少两次答对才算该题正确Gemini 的 temperature 是 1.0，输出带随机性。同一道题可能三次回答不完全一样。作业会对每道题执行三次 two-stage 流程，一题至少两次答案正确，才算这题答对。所以你不能只看单次 trial。一个 prompt 如果偶尔答对但不稳定，final accuracy 未必高。你要观察哪些题三次波动大，思考 prompt 是否缺少步骤、单位、检查或最终答案格式。判断句：作业看的是稳定正确，不是碰巧正确。现在你应该能做什么：区分 trial accuracy 和 final accuracy。
8. Public 高分不保证 Private 高分公开的 30 题用于调试，也参与一部分得分：public set 正确率大于 60% 可拿基础 5 分。隐藏的 100 题用于加分，阈值分别是大于等于 0.62、0.67、0.72，对应不同加分。课程明确说明 public set 与 private set 不重叠，成绩不接受“我 public 很高为什么 private 低”的一般申诉。这就是机器学习里的泛化问题。你可以在 public set 上调 prompt，但如果 prompt 只适合那 30 题，private set 会掉分。判断句：公开题是验证方向，不是答案字典。现在你应该能做什么：解释为什么把 public 题答案写进 prompt 不是好策略。
9. 可以交两个 prompt，取最高分作业要求缴交两个 prompt，两个提交区各交一个 JSON。评估时会看两个 prompt 的表现，取分数较高者作为最终成绩。不要把两个 prompt 交到同一个提交区，因为系统只会取最后一次提交。两个 prompt 可以有不同策略：一个偏稳健通用步骤，一个偏 few-shot 示例；一个注重单位和题意检查，一个注重答案格式和反算验证。判断句：两个提交名额是用来测试不同泛化策略，不是复制同一份。现在你应该能做什么：为两个 prompt 设计不同侧重点。
10. 常见错误：空 prompt、timeout、none return、connection aborted`contents must not be empty` 通常是没有输入 prompt 就按 set prompt。`timeout` 可能是 Gemini 请求太久，常见于中途强行停止后没有重启 Colab；解决方式是重开 Colab，若仍发生，改短或改稳 prompt。`none return` 表示 Gemini 没回传内容，也可先重启，必要时换 prompt。`connection aborted` 常被打印出来，但作业说明说通常可以忽略，不影响评估继续。判断句：先看错误发生在哪里，再决定是重启、重跑准备区、重写 prompt，还是忽略。现在你应该能做什么：遇到错误时不乱改题目 block，而是按错误类型处理。
11. Prompt hint：CoT、例子和情绪话术都只是工具Zero-shot chain-of-thought 是不给例子，只让模型一步步思考，例如 “Let's think step by step”。In-context learning 是给模型看一个或多个示范题，让它模仿解题格式。情绪提示则是加入鼓励或压力话语，有时会改变输出稳定性。这些都不是保证满分的按钮。CoT 可能让模型写很多但仍算错；例子可能占 token 或过拟合公开题；情绪提示可能有时有效、有时只是噪声。更可靠的 prompt 会明确：识别题型、列式、计算、检查题目问的是哪一个选项或数值、最后单独写答案。判断句：提示技巧要服务解题结构，不要只堆流行句子。现在你应该能做什么：把 “Let's think step by step” 扩展成更具体的数学解题策略。
12. 规则和学术诚信不能迟交，不能作弊，不能分享 prompt 或抄别人的 prompt。第一次违规作业零分且学期成绩乘 0.9，第二次可能学期成绩 F。提交文件应是 Colab 导出的 JSON，不是代码、截图或手写文本。作业设置 private set 的原因也和诚信有关。如果只用 public set 评分，最简单作弊方式就是人工算出 30 题答案，写进 prompt。隐藏题让作业回到本意：设计能泛化的提示。判断句：private set 不是刁难，而是防止把 prompt 作业变成背答案。现在你应该能做什么：说清为什么这份作业必须有隐藏题。
完整知识链路P36 的完整链路是：你在 Colab 中准备 Gemini API 环境，不用 GPU；写一个包含 `{{question}}` 的通用 prompt，引导 Gemini 解小学英文数学题；系统用 public set 让你调试，用 private set 测泛化；每题先生成解题过程，再抽最终答案，并三次投票决定该题是否正确；你可以交两个 prompt JSON，取较高分；好的 prompt 应该比单句 CoT 更具体，能识别题意、选择数学概念、逐步计算、检查单位和问题问法，并给出清楚最终答案。你现在应该能做什么：独立设计一个可提交的 prompt，并知道怎样检查它是否只是 public set 过拟合。

# 核心概念精讲 {#worked}
<!-- tag: WORKED -->

核心直觉：Prompt 是评分系统的解题程序这份作业把同一段 prompt 当成一段可复用程序。每一道题都会被放进 `{{question}}`，然后走同样的解题、抽答案、三次投票流程。因此 prompt 里最值钱的内容是稳定步骤：读题、识别题型、列式、计算、检查题目到底问什么、最后单独给答案。你现在应该能做什么：判断一句提示是在鼓励模型，还是在给模型可重复执行的解题流程。
易混边界：泛化提示不是公开题答案表Public set 是调试窗口，不是背答案材料。把公开题答案或高度贴合某几题的技巧塞进 prompt，可能让 public accuracy 好看，却无法覆盖 private set 的新题。更稳的做法是从公开题中抽象题型和错误模式，例如单位换算、最大公因数、剩余数量、比较大小、问“不能做到什么”。你现在应该能做什么：把一道公开题改写成通用检查规则，而不是把答案写进 prompt。
Worked example：从题目到可抽取答案假设题目问：一条 24 cm 的丝带要剪成 5 cm 的短段，最多能剪几段，还剩多少？弱 prompt 可能只让模型 “step by step”，模型算出 24 / 5 = 4.8 后含糊回答 4.8 段。更好的 prompt 会要求：先确认段数必须是整数，再用整数除法得到 4 段，余数 4 cm，最后写 `Final answer: 4 pieces, 4 cm left.`关键三步可迁移规则：识别离散数量，使用整数除法，最终答案格式清楚。你现在应该能做什么：为“最多/至少/剩下”类题目补一条通用 prompt 规则。

# 最小例子 {#example}
<!-- tag: EXAMPLE -->

这些例子帮助你把“提示写法”和“评分流程”具体化。

示例 1：一个合格但很基础的 promptprompt复制

```text
You are solving an elementary school math word problem.
Read the question carefully, identify the math concept, solve step by step,
check units and whether the question asks for possible or impossible choices.
At the end, write "Final answer: ..." clearly.

Question:
{{question}}
```

它满足占位符、解题步骤和最终答案格式。它不保证高分，但比只写 “solve this” 更稳。

示例 2：不合格 prompt：没有占位符wrong复制

```text
Please solve the math problem carefully.
Think step by step and give the final answer.
```

这段看起来像 prompt，但系统不知道把题目放在哪里。评估时无法替换问题。

示例 3：一题三次投票scoring复制

```text
Question 24:
 trial 1 answer = 6 -> correct
 trial 2 answer = 8 -> wrong
 trial 3 answer = 6 -> correct

result:
 correct_count = 2
 this question counts as correct
```

最终看三次里是否至少两次给出正确最终答案。

# 误区诊断 {#diagnosis}
<!-- tag: DIAGNOSIS -->

| 误区 | 错因 | 后果 | 修法 |
| --- | --- | --- | --- |
| 忘记 {{question}} 。 | 没有给系统替换题目的位置。 | 无法正常 set/evaluate 或评分。 | 在 “Question:” 后明确放置占位符。 |
| 把 public 30 题答案写进 prompt。 | 把公开调试集当答案库。 | private set 泛化差，也违背作业目标。 | 写通用解题策略，不写题目答案表。 |
| 只写 “Let's think step by step”。 | 提示太泛，缺少数学检查策略。 | 模型可能步骤很多但仍算错。 | 补充题型识别、单位、反算、最终答案格式。 |
| 答案格式不清楚。 | 忽略第二阶段抽答案。 | 推理对但抽取失败，仍可能算错。 | 要求最后单独写 “Final answer: ...”。 |
| 只看一次 trial。 | 忽略 temperature 随机性。 | 误判 prompt 稳定性。 | 看三次结果和 final accuracy。 |
| Public 高分后停止改进。 | 忽略 private set 不同分布。 | 隐藏题掉分。 | 分析题型覆盖，避免过度贴合公开题。 |
| Colab 断线后只重跑评估 cell。 | 环境状态、API 设置或 Gradio 状态丢失。 | timeout、none return 或结果异常。 | 重启 Colab，按顺序重跑准备区和 prompt 设置。 |
| 同一提交区上传两个 prompt。 | 误解两个作业缴交区。 | 系统只取最后一次，等于丢掉一个策略。 | 两个区各交一个 JSON。 |

Public vs PrivatePublic 是调试用，Private 是泛化检验。两者不重叠，分数不同是正常现象。
推理过程 vs 最终答案评分只看第二阶段抽出的最终答案，但第一阶段推理会影响能不能抽出正确答案。
技巧 vs 结构CoT、例子、情绪提示只是技巧；稳定 prompt 需要清楚的数学解题结构。
