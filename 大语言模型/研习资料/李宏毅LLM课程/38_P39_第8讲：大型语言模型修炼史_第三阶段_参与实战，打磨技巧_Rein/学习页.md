# 概念骨架 {#skeleton}
<!-- tag: MAP -->

前两阶段：给下一步答案
pretrain 和 SFT 都给模型一个输入，并告诉它下一步应该接什么。区别是 pretrain 多来自网络文本，SFT 的答案由人类或标注者准备。
判断句：如果训练信号是“这个 token 应该是什么”，它仍然是以局部接龙为核心。


第三阶段：比较完整回答
RLHF 的资料偏好：同一个问题下，回答 A 比回答 B 好。模型要提高好回答的概率，降低差回答的概率。
这个信号让模型更关注完整输出的整体质量，而不是只盯下一步最像什么。


新问题：什么叫好
围棋输赢有规则，回答好坏没有统一标准。一个回答可能更安全但没帮助，也可能更有帮助但风险高。
判断句：RLHF 推向某套偏好标准。

1. 收集偏好模型生成多个答案，人类比较哪个更好。
2. 训练奖励模型reward model 学会给“问题，答案”打偏好分。
3. 优化语言模型用 PPO、DPO 等方法提高高分答案概率。
4. 检查边界观察是否啰嗦、过度拒绝、讨好奖励模型或偏离真实人类。

# 术语表 {#terms}
<!-- tag: TERMS -->

| 术语 | 直觉解释 | 准确说法 | 常见误解 |
| --- | --- | --- | --- |
| RLHF | 让模型从人类偏好里学。 | Reinforcement Learning from Human Feedback，用人类反馈优化模型输出行为。 | 以为人类会提供标准答案；实际常是比较答案好坏。 |
| reinforcement learning | 根据奖励调整行为。 | 模型产生行为后获得 reward，优化策略以提高长期或整体奖励。 | 把 RL 等同于游戏；语言模型里 reward 往往来自人类偏好或奖励模型。 |
| human feedback | 人类告诉模型哪个回答更好。 | 可以是排序、打分、点赞、重生成偏好等反馈信号。 | 以为人类反馈一定客观正确；很多问题的好坏本来就有争议。 |
| reward model | 虚拟的人类评审。 | 用人类偏好资料训练出来的模型，输入问题和答案，输出偏好分数。 | 把 reward model 当真人；它也会学偏、被钻空子。 |
| PPO | 经典 RLHF 优化算法。 | Proximal Policy Optimization，InstructGPT 等工作中用于让语言模型向 reward model 偏好靠拢。 | 以为理解 RLHF 必须先懂 PPO 细节；本节先抓“提高好答案概率”。 |
| DPO | 不用显式奖励模型的一类偏好优化。 | Direct Preference Optimization，直接用偏好对优化语言模型。 | 以为 DPO 一定比 PPO 好；它只是减少某些复杂环节，效果仍看数据和目标。 |
| RLAIF | 让 AI 提供反馈。 | Reinforcement Learning from AI Feedback，用强模型或同模型反思来评估答案。 | 以为 AI 反馈自动客观；AI 也继承训练偏差和评价盲点。 |
| alignment | 让模型更符合人类需求。 | 通常指把模型行为对齐到人类偏好、安全要求和任务目标。 | 以为 alignment 只有安全；它同时涉及 helpfulness、truthfulness、harmlessness 等面向。 |
| helpfulness | 有没有帮上忙。 | 模型回答是否满足用户目标、给出可用信息或解决问题。 | 只追求有帮助可能忽略安全边界。 |
| safety | 是否避免危险或伤害。 | 模型是否避免提供危险、违法、有害或不负责任的内容。 | 只追求安全可能导致过度拒绝、帮助不足。 |

读完这一段，你应该能解释：RLHF 的核心“人类表达偏好”。

# 讲义主线 {#lecture}
<!-- tag: LECTURE -->
<!-- layout: grid2 -->

## box
### 1. 三阶段训练资料有什么不同
第一阶段 pretrain 的资料来自大规模文本，训练信号是“接下来该出现什么”。第二阶段 instruction fine-tuning 仍然是接龙，只是答案由人类老师准备，所以它是 supervised learning。
第三阶段不再告诉模型下一步 token 应该是什么，而是告诉它完整答案之间的偏好。比如同一个问题下，回答 A 比回答 B 更让人满意，模型就应该提高 A 这类回答的概率。
你现在应该能做的事：看到训练资料时，判断它是 token 级标准答案，还是回答级偏好信号。

## box
### 2. 为什么比较答案比写答案容易
人类写出好答案很费力。让你写一首合格七言绝句，可能很难；但让你比较两首诗哪一首格式更像七言，通常容易很多。
这就是 RLHF 的实用价值。它降低了人类参与成本：人不用每次产出完美答案，只要在多个候选之间表达偏好。这个偏好虽然不等于真理，但比完全没有反馈更能推动模型朝人类需求靠近。

3. SFT 只问过程，RLHF 只问结果
SFT 的每一步都在模仿标准答案的下一个 token。它隐含一个假设：每一步都像标准答案，最后结果就好。
RLHF 看的是完整回答的偏好。哪怕中间某一步不像训练答案，只要最终结果更好，偏好信号就会支持它。这也是老师用棋局例子想说明的地方：有时局部看起来怪，整体反而赢。


4. AlphaGo 类比帮你抓住 RL
下围棋每一步像分类：给棋盘，选落点。但整局输赢是全局结果。AlphaGo 不能只学棋谱，还要通过胜负反馈继续提升。
语言模型也类似：每一步是选下一个 token，整段回答才是用户真正评价的对象。区别是围棋胜负有规则，语言回答好坏需要人类或奖励模型判断。


5. reward model 放大人类反馈
人类反馈很贵，不可能每次模型生成都找真人评价。解决办法是先用少量人类偏好训练一个 reward model，让它学会模仿人类给“问题，答案”打分。
之后语言模型可以大量向 reward model 学习。好处是反馈变便宜，坏处是它只是在模仿人类偏好，可能学歪，也可能被语言模型钻空子。


6. 过度优化 reward model 会出怪相
如果语言模型过度迎合 reward model，它可能学到真人并不喜欢的模式。字幕里提到摘要任务中模型结尾总写 please、夹杂问号的例子，就是 reward model 被钻空子的现象。
ChatGPT 早期常见的啰嗦、过度道歉、总是列 1 2 3 4 5、模棱两可或过度拒绝，也可能和过度学习某些偏好有关。判断句：reward 高不等于真实人类长期满意。


7. DPO、KTO 与 RLAIF 是后续路线
因为显式 reward model 有成本和风险，后来出现了 DPO、KTO 等方法，尝试直接利用偏好数据优化模型。它们不一定绝对更好，但代表研究者在减少 RLHF 复杂度。
当强模型已经能评估别的模型回答时，也可以做 RLAIF，用 AI 提供反馈。甚至同一个模型也可以先生成，再反思自己的答案。这里的前提是模型可能不会生成好答案，但能识别哪个答案更差。


8. helpfulness 和 safety 会冲突
“好答案”没有唯一标准。教你做玩具枪，helpfulness 可能要求给步骤，safety 可能要求拒绝或转向更安全替代方案。不同模型对同一请求的边界会不同。
这说明 alignment 不是纯技术优化题。它涉及价值取舍：什么时候优先帮助，什么时候优先安全，谁来定义标准，标准能不能被人类接受。

## box
### 完整知识链路
大型语言模型的三阶段可以这样串起来：pretrain 让模型在海量文本中学会接龙，形成 foundation model；instruction fine-tuning 用人类准备的指令答案教模型按用户要求回答；RLHF 则让模型从人类对完整回答的偏好中继续调整。RLHF 的核心提高人类偏好答案的概率。因为人类反馈有限，通常先训练 reward model，再让语言模型向这个虚拟评审学习。这样能改善模型行为，但也可能出现过度优化、啰嗦、过度拒绝、讨好奖励模型等副作用。后续的 DPO、RLAIF、自我反馈和 safety/helpfulness 多目标权衡，都是围绕“如何定义并利用好反馈”展开。

<!-- layout: grid2 -->

## box
### 课程线索一：重生成按钮就是反馈入口
字幕从 ChatGPT 的“新答案比旧答案更好、差不多还是更差”讲起。这个例子很重要，因为它把 RLHF 从论文概念落回日常产品：用户选择就是偏好数据。
你现在应该能做的事：看到点赞、踩、重生成偏好、A/B 比较时，想到它们可能成为后续对齐训练资料。

## box
### 课程线索二：偏好排序比绝对打分更稳
老师强调，单个回答“好不好”常常说不清，因为标准因人而异；比较两个回答哪个更好，反而更容易。
你现在应该能做的事：设计反馈标注时，优先考虑 pairwise ranking，而不是只问“这个回答好不好”。

## box
### 课程线索三：reward model 是虚拟人类
人类反馈有限，所以训练一个 reward model 来模拟人类偏好。它可以给多个候选答案排序，也可以作为训练信号指导语言模型。
你现在应该能做的事：解释 reward model 为什么省人力，也解释它为什么不等于真人。

## box
### 课程线索四：未来反馈可能来自 AI
RLAIF、Constitutional AI、self-rewarding model 都在探索让 AI 提供反馈。强模型可能评价弱模型，也可能评价自己的答案。
你现在应该能做的事：看到“AI 评 AI”时，同时问效果、偏差、标准来源和人类监督位置。

# 核心概念精讲 {#worked}
<!-- tag: WORKED -->

核心直觉：第三阶段学的是“哪种回答更好”SFT 告诉模型“给这个问题写这个答案”；RLHF 或 DPO 更像告诉模型“同一个问题下，A 比 B 更符合人类偏好”。它不一定提供唯一标准答案，而是把有用、诚实、安全、礼貌、符合任务等偏好压成训练信号。你现在应该能做什么：区分“模仿标准答案”和“学习偏好排序”。
易混边界：Reward model 不是人类本人Reward model 只是从偏好资料中训练出来的评分器。它能把人类偏好自动化，但也会学到资料里的偏差和捷径。语言模型如果过度优化 reward model，可能产出高分但奇怪的回答，这就是 reward hacking 的风险。你现在应该能做什么：解释为什么对齐训练后还需要真实评估和失败案例审计。
Worked example：从偏好对到优化方向同一请求是“给小学生解释黑洞”。回答 A 用短句说明“黑洞是引力很强的天体，光也难逃出去”；回答 B 堆公式且没有解释符号。标注者选 A。偏好训练不会说 B 的每个 token 都错，而是提高 A 类回答在这个上下文下的相对概率，降低 B 类回答的相对概率。你现在应该能做什么：把一个 chosen/rejected 样本翻译成“提高什么、降低什么”的训练方向。

# 最小例子 {#example}
<!-- tag: EXAMPLE -->

下面用极简数据结构看懂 RLHF 的训练资料。重点是偏好关系，而不是标准答案。

偏好样本：同一问题下比较两个回答json

```text
{
 "prompt": "写一首赞扬大型语言模型的七言绝句。",
 "chosen": "万卷文光入指端，千秋问答起波澜。人机共写新篇日，灯下春风满纸宽。",
 "rejected": "字机致知千年网，光年万里桥。"
}
```

## box soft
### 逐项理解
`chosen` 不是唯一真理，只是在这次比较里更受偏好。`rejected` 也不一定每个 token 都错，但整体更差，可能格式不对、语义不通或不符合用户要求。
RLHF 学的是这类整体偏好：让 chosen 一类回答更容易出现，让 rejected 一类回答更不容易出现。

## box warn
### 为什么不是单个分数
如果只问“chosen 这首好不好”，不同标注者标准会漂移。放在 rejected 旁边比较，判断会更稳定，因为人只需要决定哪个更好。
判断句：偏好排序降低了标注难度，但没有消灭价值判断。

概念伪代码：reward model 与语言模型优化python

```text
reward_model = train_reward_model(preference_pairs)

answer = language_model.generate(prompt)
reward = reward_model.score(prompt, answer)

if reward_is_high(reward):
 increase_probability(language_model, prompt, answer)
else:
 decrease_probability(language_model, prompt, answer)
```

读完例子，你应该能解释：reward model 先学习人类偏好，再把偏好分数转成语言模型的训练信号。

# 误区诊断 {#diagnosis}
<!-- tag: DIAGNOSIS -->
<!-- layout: grid2 -->

## box redsoft
### 六个常见坑
1. 把 RLHF 当成继续写标准答案。错因是混淆 SFT 和偏好学习；修法是看训练资料是否包含 chosen / rejected 或排序。
2. 以为人类偏好一定正确。错因是把偏好当客观真理；后果是模型学习标注者偏见；修法是设计多样标注和审计机制。
3. 把 reward model 当真人。错因是忘记它也是模型；后果是过度优化会学出奇怪模式；修法是用真实人类评估回查。
4. 只追求 helpfulness。错因是忽略安全边界；后果是模型可能给危险建议；修法是引入 safety 评价。
5. 只追求 safety。错因是把拒绝当安全；后果是过度拒绝、用户任务失败；修法是区分危险请求和可安全帮助的请求。
6. 认为 DPO / RLAIF 自动解决 RLHF 问题。错因是迷信新算法；后果是忽略数据质量和评价标准；修法是回到偏好来源、目标函数和真实评估。

## box
### 易混概念对照
概念 A概念 B区别
SFTRLHFSFT 学标准答案的 token；RLHF 学完整回答的偏好。
reward modelhuman feedback前者是模型，后者是真人提供的原始偏好信号。
PPODPOPPO 常配合 reward model 做强化学习；DPO 直接用偏好对优化模型。
RLAIFRLHFRLAIF 用 AI 反馈，RLHF 用人类反馈；二者都要关心反馈标准。
helpfulnesssafety前者关注帮忙程度，后者关注风险控制，二者会冲突。

读完这一节，你应该能判断一个“对齐方案”到底在优化谁的偏好，以及可能把模型推向哪些副作用。
