P40｜以大型语言模型打造 AI Agent

今天多数 AI 使用方式还是一问一答：翻译一句话、画一张图、回答一个问题。Agent 要处理的是多步骤任务，例如组织聚餐：调查时间、统计结果、订餐厅、遇到没位置就改查地图，再调整计划。它不是一次输出答案，而是持续和环境互动。

概念骨架

MAP

一问一答不是 Agent

只让模型翻译一句话，输入一次、输出一次，这仍是普通模型调用。Agent 处理的是需要多个步骤、顺序重要、环境会变化的任务。

判断句：如果系统没有观察结果并调整下一步，它还不算真正进入 Agent 循环。

难点在行动落地

语言模型很会写自然语言计划，但机械臂、游戏角色、浏览器或自驾车需要具体可执行动作。把“保护伤者”翻译成代码、控制指令或工具调用，才是难点。

判断句：计划好听不等于任务完成，必须看行动是否真的改变了环境。

术语表

TERMS
术语直觉解释准确说法常见误解
AI Agent能自己推进多步骤任务的 AI 系统。给定目标后，能感知环境、制定计划、执行动作、观察反馈并调整行为的系统。把任何聊天机器人都叫 Agent。
多步骤任务不是一步能完成的事。需要多个有先后依赖的行动，并且中途可能因环境变化重规划。把任务拆成列表就以为完成了 Agent。
计划从目标到下一步的路线。Agent 根据目标、状态和记忆生成的短期行动方案。计划不是固定剧本，执行中要根据反馈修改。
记忆过去经验会影响未来行动。可存储用户偏好、对话摘要、失败经验、环境事实或工具结果。把当前上下文窗口误当长期记忆。
环境状态Agent 当前看到的世界。来自文字、视觉、听觉、触觉、网页、API、游戏状态等 sensor 的输入。只看用户 prompt，忽略外部环境变化。
工具使用让模型不只说，还能调用外部能力。Agent 可调用搜索、地图、表单、代码执行、机器人控制器等工具。工具结果不等于正确结果，还要校验和反馈。
reflection从刚才的失败或变化中总结经验。Agent 根据观察到的结果写入经验，影响后续计划与行动。反思提取未来可用的行动规则。
ReAct边推理边行动。Reasoning + Acting，把思考、工具行动和观察交错起来。只输出推理文本，没有真实 action 和 observation。
slow agent / fast agent一个负责想，一个负责做。slow agent 生成高层自然语言计划；fast agent 把计划转成真实可执行动作。以为大模型自己说一句话，物理世界就会执行。
Voyager在 Minecraft 中持续探索的 Agent 例子。让 Agent 在虚拟世界中学习技能、写工具代码、积累经验并扩展能力。把游戏 demo 的成功直接等同于真实世界可靠。

读完这一段，你应该能把 Agent 看成闭环架构，而不是一个“更聪明的聊天框”。

讲义主线

LECTURE

1. Agent 解决的是多步骤任务

举办聚餐的例子说明了 Agent 的任务形态：先调查时间，再统计多数人有空的时段，然后订餐厅；如果餐厅没位置，就查地图找替代方案。这里每一步都有依赖，顺序错了会导致任务失败。

普通 LLM 调用只需要把问题转成答案。Agent 要把目标转成一串可执行动作，并在行动后观察结果。你现在应该能做的事：拿一个任务判断它是一问一答，还是需要 Agent。

2. 为什么大型语言模型让 Agent 变得可行

LLM 已经具备几项关键能力：能读懂自然语言目标，能做计划，能根据新信息改计划，能总结经验，也能生成调用工具或执行代码的文本。

这不代表 LLM 自己就是完整 Agent。Agent 还需要外部记忆、工具接口、传感器、执行器、环境观察和安全控制。判断句：LLM 是 Agent 的大脑之一，不是整个身体和世界。

  1. 记忆让 Agent 个性化

ChatGPT 传统对话没有真正长期记忆，开启新对话后过去内容就不在上下文里。Agent 如果能把对话摘要、用户偏好和任务历史写入长期记忆，就能在新任务中调用过去经验。

例如它记得你正在准备考试，下次对话就能主动关联到这个背景。记忆抽取关键事实并可检索。

  1. 状态来自多种 sensor

Agent 可以通过文字输入知道任务，也可以通过视觉、听觉、网页、API、机器人触觉或游戏状态了解环境。状态不是静态的，行动之后会改变。

如果 Agent 在执行“订餐厅 A”后发现没位置，它必须把这个新状态放回系统，改查餐厅 B。没有状态更新，就没有真正闭环。

  1. 计划必须能被修改

计划是短期目标，不是固定剧本。进入迷宫救人时，原计划可能是先评估伤势；如果遭遇攻击，新计划应优先保护伤者。

LLM 能根据“目标，记忆，当前状态，新变化”重新生成计划。你现在应该能做的事：给一个环境变化，写出 Agent 应该怎样改计划。

  1. 最大难点是把文字行动变成真实行动

LLM 可以说“评估伤势”“保持警觉”“查地图”，但真实系统要执行这些动作，需要工具、代码、控制器或机器人策略。

这就是 slow agent / fast agent 的意义：slow agent 用自然语言决定高层动作，fast agent 负责把动作落成物理世界或虚拟世界中的可执行行为。

  1. reflection 让经验进入记忆

Agent 失败或遇到意外后，可以总结经验，比如“评估伤者时要保持环境警觉”。这条经验写入记忆，下次遇到相似任务时会改变计划。

ReAct 和 Reflexion 这类方法都在围绕“行动，观察，反思”设计循环。反思的价值不在于写长篇理由，而在于让未来行动更可靠。

  1. Demo 成功不等于真实可靠

AutoGPT、Voyager、AI 村民、机器人和自驾车都说明 Agent 方向很有潜力。但复杂任务会消耗成本，工具可能失败，计划可能空转，环境反馈可能被误读。

判断句：Agent 要验收的是任务完成率、成本、失败恢复、工具安全和环境影响，而不是只看它能不能写出漂亮计划。

完整知识链路

以大型语言模型打造 AI Agent，就是把 LLM 从“回答一次问题”放进一个持续行动的闭环。Agent 先接受终极目标，再读取记忆和环境状态，生成短期计划；接着通过文字、工具、代码或机器人控制器执行行动；行动改变环境，Agent 再观察新状态，更新记忆和计划。LLM 擅长自然语言理解、规划、反思和工具描述，所以它适合做 Agent 的高层控制核心。但真正落地还需要外部记忆、传感器、可执行工具、slow/fast agent 分工、错误恢复和安全约束。未来 Agent 的关键能在环境变化中稳定完成任务。

课程线索

内容线索

课程线索一：聚餐任务说明顺序依赖

调查时间、统计结果、订餐厅、改查地图，这些步骤不能随便换。Agent 要解决的是有顺序、有反馈、有变化的任务。

你现在应该能做的事：看到任务时先拆依赖关系，而不是直接叫模型“帮我完成”。

课程线索二：虚拟世界是 Agent 试验场

AI 村民和 Voyager 都让 Agent 活在虚拟世界里。虚拟环境能提供状态、行动结果和技能积累，是测试 Agent 循环的好地方。

你现在应该能做的事：区分“虚拟世界 demo”与“真实世界部署”的难度差异。

课程线索三：记忆改变后续行动

模型总结“评估伤势时要警觉环境”，下次计划中就会加入保持警觉。这说明记忆不是摆设，它会改变行动优先级。

你现在应该能做的事：设计 Agent 记忆时，只保留能影响未来决策的经验。

课程线索四：自然语言行动需要执行层

“保护伤者”“评估伤势”只是自然语言动作。要让机器人、游戏角色或网页自动化真正执行，需要把它翻译成代码、控制策略或工具调用。

你现在应该能做的事：审查 Agent 方案时问清楚 action 到底由谁执行、怎样验证成功。

核心概念精讲

WORKED

核心直觉：Agent 是带反馈的行动循环单次 LLM 回答只是在生成文本；Agent 要把目标拆成动作，调用工具或控制环境，再观察结果并更新下一步。它的关键能在真实或虚拟环境里形成 observe -> plan -> act -> observe 的闭环。你现在应该能做什么：检查一个系统是否只有聊天界面，还是有可执行动作和状态反馈。

易混边界：计划不等于行动“我会帮你订餐厅”只是计划或承诺；真正的行动是调用订位 API、填写表单、发送消息或控制软件。Agent 设计中必须区分 slow agent 的高层规划和 fast agent 的具体执行，否则系统看起来聪明，却停在文字层。你现在应该能做什么：把自然语言计划拆成可验收的工具调用步骤。

Worked example：聚餐 Agent 的一次闭环目标是订 6 人周五晚餐。Agent 先观察日历与预算，计划“找餐厅 -> 确认空位 -> 通知朋友”。调用餐厅查询工具后发现周五无位，状态改变；它不能继续原计划，而要改成“查周六或备选餐厅”。若成功订位，还要把确认号写入记忆，供后续通知和改期使用。你现在应该能做什么：说明为什么没有状态更新的 Agent 会在环境变化时失败。

最小例子

EXAMPLE

下面把 Agent 写成一个最小循环。它不是完整框架，只用于看懂每一步读什么、写什么。

代码
goal = "帮我组织一次朋友聚餐"
memory = ["朋友们常吃餐厅 A", "小张周四通常有空"]
state = observe_environment() # 表单结果、餐厅空位、地图推荐

plan = llm.plan(goal=goal, memory=memory, state=state)
action = llm.choose_next_action(plan=plan, state=state)
result = tool_executor.run(action)

new_state = observe_environment(result)
lesson = llm.reflect(goal=goal, action=action, result=result)
memory.append(lesson)

删掉哪一步会坏

删掉观察，Agent 不知道餐厅是否有位；删掉执行器，计划停在文字；删掉反思，下次还会犯同样错误；删掉记忆，个性化和经验积累会消失。

判断句：Agent 的能力来自闭环，不来自某一句很长的 prompt。

计划更新示例：状态变化后重新排序优先级text

目标：安全带考生离开迷宫
旧状态：考生受伤，需要评估伤势
新状态：评估过程中遭遇攻击

新计划：
1. 先挡下攻击并移动到安全位置
2. 继续确认考生意识和出血情况
3. 选择最短安全路线离开迷宫
4. 记录经验：救援时不能只盯伤者，要持续监控威胁

读完例子，你应该能判断一个 Agent 是否真的具备状态更新、计划修改和经验记忆。

误区诊断

DIAGNOSIS

六个常见坑

  1. 把一次 LLM 回答当 Agent。错因是只看输出文本；修法是检查是否有行动、观察和重规划。
  2. 让 Agent 一次生成很长计划却不观察环境。后果是计划赶不上变化；修法是短步执行，每步后读状态。
  3. 把记忆做成全文堆叠。后果是上下文拥挤且噪声大；修法是摘要、标签、检索和过期机制。
  4. 工具调用没有验收。后果是 API 报错或返回假结果仍继续执行；修法是给每个 action 定义成功条件。
  5. 自然语言动作无法落地。错因是没有 fast agent 或执行器；修法是把动作映射成代码、API 或控制策略。
  6. 只看炫酷 demo，不看成本和失败恢复。后果是 AutoGPT 式空转或费用失控；修法是限制预算、步数、权限和回滚路径。
