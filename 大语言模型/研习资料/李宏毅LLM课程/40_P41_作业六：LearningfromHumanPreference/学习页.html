P41｜作业六：Learning from Human Preference

作业六要让 Breeze 7B 从人类偏好中学习，让模型输出更符合指定价值观的开放式回答。主题例子是“动漫真人化”：原模型可能中立、拒绝表达立场，或输出与期望相反的观点；DPO 训练后，希望它更倾向输出你设定为 preferred response 的立场。

概念骨架

MAP

为什么不用标准答案

地球到太阳距离有标准答案，动漫真人化好不好没有唯一答案。开放式问题的回答会受到立场、价值观和偏好影响。

判断句：如果问题没有客观标准答案，就要先定义你希望模型对齐到哪种偏好。

作业要做的是观察

你要改 support_ratio、num_epoch、data_size，观察训练前后在 10 个测试问题上的输出变化，并回答 report questions。

判断句：报告重点是内容立场变化，不是只写长度、格式或跑分。

术语表

TERMS
术语直觉解释准确说法常见误解
Learning from Human Preference从人类偏好里学。用人类标注的回答偏好来调整模型输出行为。不是让人类写唯一标准答案。
values alignment让模型回答符合指定价值观。把模型输出对齐到某套人类偏好、立场或价值标准。以为 alignment 一定是客观正确；很多时候是立场设定。
RLHF用人类反馈训练模型。通常先训练 reward model，再用 RL 优化语言模型。以为作业六完整实现标准 RLHF；本作业用 DPO 简化。
DPO直接用偏好对训练。Direct Preference Optimization，利用 preferred / non-preferred response 直接优化语言模型。把 DPO 当成普通监督微调，忽略“成对偏好”。
reward model虚拟评审。标准 RLHF 中用来给回答打偏好分的模型。DPO 不需要额外训练 reward model，但仍在学偏好。
preferred response希望模型更常输出的回答。在同一 prompt 下，被标注为更符合偏好的答案。preferred 不等于事实正确，只表示更符合当前偏好。
non-preferred response希望模型少输出的回答。同一 prompt 下，被标注为较不符合偏好的答案。它不一定完全错误，可能只是立场不符合。
support_ratio训练资料中支持立场占多少。0 到 1 的比例，决定多少资料把 support 标成 preferred。设为 1 就全支持，设为 0 就全反对，中间值会混合偏好。
data_size用多少笔训练资料。作业要求在 10 到 50 范围内调整。越多不只影响强度，也可能影响输出一致性和训练时间。
num_epoch完整看训练集几遍。模型在训练过程中遍历完整训练资料的次数。epoch 越多不一定越好，可能过度强化某种立场或模板。

讲义主线

LECTURE

1. 开放式问题需要偏好学习

监督学习适合有标准答案的问题，例如“地球到太阳距离是多少”。但很多真实问题没有唯一答案，比如“是否支持动漫真人化”。回答会随着人的立场而变化。

作业六要你观察：当训练资料把“支持动漫真人化”标为 preferred response 时，模型是否更倾向支持；当反对立场被标为 preferred 时，模型是否转向反对。你现在应该能做的事：判断一个问题适合标准答案训练，还是适合偏好学习。

2. DPO 和标准 RLHF 的关系

标准 RLHF 通常分两步：先训练 reward model 学会给回答打分，再让语言模型用 RL 向高分回答靠近。这个过程成本高，也比较不稳定。

DPO 简化了这条路。它不额外训练 reward model，而是直接利用成对资料：同一个 prompt 下，preferred response 应该比 non-preferred response 更容易被模型生成。

  1. label_data.json 长什么样

训练资料共有约 50 笔。每笔都有 prompt，也就是关于动漫真人化的开放问题；同时有 support 回答和 oppose 回答。

support_ratio 决定哪一部分资料把 support 当 preferred，哪一部分把 oppose 当 preferred。这个比例越偏向 1，模型越被推向支持立场。

  1. report 不是写感想

三道 report questions 都要求观察超参数对测试输出的影响。第一题固定 data_size=50、num_epoch=3，比较 support_ratio=0 与 1。第二题固定 data_size=50、support_ratio=1，比较 num_epoch=1 与 3。第三题固定 num_epoch=3、support_ratio=1，比较 data_size 变化。

重点是输出内容怎么变，尤其是立场、理由、偏好强度，而不是只写“变长了”。

  1. 支持比例影响立场方向

support_ratio 是最直接的方向盘。设为 1 时，所有训练资料都鼓励支持动漫真人化；设为 0 时，所有训练资料都鼓励反对。

如果输出没有明显转向，可能是训练不够、资料数量不够、测试问题表达和训练资料差异太大，或模型原本倾向太强。

  1. epoch 影响偏好强化程度

num_epoch 表示模型完整看过训练资料几次。epoch 增加，偏好信号通常更强，模型更可能稳定采用训练偏好的立场。

但 epoch 太多也可能让模型输出更模板化，或者过度强化单一表达。观察时要看内容是否真的更符合偏好，而不是只看语气更强。

  1. data_size 影响覆盖和稳定性

data_size 决定使用多少笔训练资料。资料越多，模型看到的问题表达和理由越丰富，测试时更可能稳定迁移。

但资料品质同样重要。如果资料虽然多，却理由重复、立场表达单一，模型也可能只学到固定模板。

  1. 提交流程也会影响分数

作业总分包括 report 的 9 分和截图 PDF 的 1 分。report 分数来自达哥批改助教第一轮对话记录的 JSON，不是随便下载某个 JSON。

截图 PDF 要展示任意一次模型在测试资料集上的输出结果，并按学号命名。文件格式、文件名或 JSON 来源错了，对应部分可能零分。

完整知识链路

作业六用 DPO 训练模型从人类偏好中学习。任务让 Breeze 7B 在“动漫真人化”这类开放式议题上对齐到指定偏好。训练资料由 prompt、support、oppose 组成，Colab 会根据 support_ratio 把某些回答标为 preferred response，把另一侧标为 non-preferred response。DPO 训练后，模型应提高 preferred 立场回答的概率，降低 non-preferred 立场回答的概率。你要通过四次左右训练比较 support_ratio、num_epoch、data_size 对测试输出内容的影响，并把报告回答交给达哥批改，最后提交对话 JSON 和一次输出截图 PDF。

你现在应该能做的事：提交前检查文件来源、文件名、扩展名和第一轮对话内容。

核心概念精讲

WORKED

核心直觉：DPO 直接拉开好坏回答的概率差作业六的重点用成对偏好告诉它：同一问题下，哪种立场或表达应更容易出现。DPO 会把 preferred response 的相对概率往上拉，把 non-preferred response 的相对概率往下压，不需要先单独训练 reward model。你现在应该能做什么：把一组 support/oppose 资料翻译成 preferred 与 non-preferred。

易混边界：support 不是天然 preferredsupport 只是回答立场，preferred 才是训练标签。support_ratio 决定多少比例的样本把支持回答标为 preferred。如果 support_ratio 设反，模型会被训练到相反方向；如果只看回答长度，不看立场和理由，就看不出 DPO 有没有学到偏好。你现在应该能做什么：检查一次实验到底在强化支持、反对，还是只是改变输出风格。

Worked example：support_ratio 怎样改变同一笔资料题目是“你赞成动漫真人化吗？”support 回答说“赞成，因为能吸引新观众”；oppose 回答说“反对，因为可能破坏角色形象”。当 support_ratio = 1，support 是 preferred，模型会更常赞成；当 support_ratio = 0，oppose 是 preferred，模型会更常反对。测试时要比较同一类问题下立场是否稳定改变，而不是只看句子是否更长。你现在应该能做什么：为报告写出“参数变化 -> 偏好标签变化 -> 输出立场变化”的因果链。

最小例子

EXAMPLE

下面用一笔简化资料看懂 DPO 在作业六里学什么。

代码
{
 "prompt": "你同意动漫真人化吗？请说明理由。",
 "support": "我同意。真人化可以让更多观众接触原作，也能用新的表演方式诠释角色。",
 "oppose": "我不同意。真人化常常难以保留原作风格，也可能破坏角色形象。"
}

这不是事实题

support 和 oppose 都可能写得通顺，区别在立场。DPO 训练的目标让模型跟随你指定的偏好。

判断句：偏好学习的资料要成对比较，不能只看单个回答好不好。

实验记录：四次训练如何服务 reporttext

实验 A：support_ratio=0, data_size=50, num_epoch=3
实验 B：support_ratio=1, data_size=50, num_epoch=3
实验 C：support_ratio=1, data_size=50, num_epoch=1
实验 D：support_ratio=1, data_size=10 或 30, num_epoch=3

观察重点：
- 输出是否支持动漫真人化
- 理由是否更贴近 preferred response
- 立场是否稳定迁移到 10 个测试问题
- 是否只是长度变化，而内容立场没有变化

读完例子，你应该能自己规划四次训练，并知道每次输出该观察什么。

误区诊断

DIAGNOSIS

六个常见坑

  1. 把 DPO 当监督微调。错因是忽略成对偏好；修法是说明 preferred 与 non-preferred 的概率变化。
  2. report 只写输出变长。后果是没有回答参数对内容的影响；修法是比较立场、理由和偏好强度。
  3. support_ratio 理解反了。后果是把模型训练到相反立场；修法是确认 support 或 oppose 哪个被标为 preferred。
  4. 重复运行前不保存截图或输出。后果是 Colab 清空后找不到证据；修法是每次训练后保存关键输出。
  5. 提交错误 JSON。后果是 report 零分；修法是提交达哥第一轮对话记录 JSON，而不是其他下载项。
  6. PDF 截图格式错误。后果是截图 1 分丢失；修法是按学号命名，并保存为 PDF。
