# 概念骨架 {#skeleton}
<!-- tag: MAP -->

1. 先把文字变成模型能算的东西语言模型不能直接“读懂”自然语言字符，它先通过 tokenization 把文本切成 token sequence，再用 embedding table 把每个 token 转成向量。token 是离散符号，embedding 才是可以参与神经网络计算的一串数字。判断句：如果你只说“模型读入一句话”，但说不出 token 和 embedding 的区别，就还没进入 Transformer 的输入层。
2. 再让每个位置读取上下文原始 token embedding 是固定的，bank 不管在“银行”还是“河岸”语境里都先查到同一个向量。attention 的作用就是考虑上下文，把每个位置的表示变成 contextualized embedding。核心动作是：当前位置和其他位置计算相关性，得到 attention weight，然后按权重把相关信息加总进来。
3. 最后转成下一步接龙的概率多层 Transformer block 反复更新表示后，模型取最后一个位置的向量，交给 output layer。output layer 经过 linear transform 和 softmax，得到下一枚 token 的 probability distribution。生成选出一个 token、贴回上下文、再重复同一流程。

Tokenizer文字切成 token；BPE 会把常见片段当作 token。
Input layertoken 查表得到 embedding，并加入 positional embedding。
Transformer blocksattention 收集上下文，feedforward 整理信息，层层更新表示。
Output layer句尾向量变成下一个 token 的概率分布。

# 术语表 {#terms}
<!-- tag: TERMS -->

| 术语 | 直觉解释 | 准确说法 | 常见误解 |
| --- | --- | --- | --- |
| 语言模型 | 一个会文字接龙的函数。 | 输入 token sequence，输出下一个 token 的概率分布；现代实现常用神经网络，核心架构常是 Transformer。 | 以为它一次性生成整段答案，或以为它天然等同于事实数据库。 |
| Transformer | 现代语言模型最常用的内部架构。 | 由多层 Transformer block 堆叠而成，每个 block 主要包含 attention 和 feedforward network。 | 把 Transformer 当成整个产品，忘了它只是语言模型内部的一种神经网络架构。 |
| tokenization | 把文字切成模型的基本单位。 | 根据预设词表和算法把输入转成 token sequence；这一模块通常没有训练参数。 | 以为 token 一定等于中文一个字或英文一个词。 |
| BPE | 从大量文字中找常见片段当 token。 | Byte Pair Encoding 会逐步合并常见相邻片段，建立可覆盖文本的 token list。 | 以为 BPE 会理解语义；它主要是统计频率和切分策略。 |
| token embedding | token 的向量身份证。 | 每个 token 对应 embedding table 中的一串数字，这些向量是语言模型参数的一部分，由训练学出。 | 以为 embedding 由人手工设定，或以为同一个 token 在不同上下文会先查到不同向量。 |
| positional embedding | 告诉模型“这个 token 在第几个位置”。 | 把代表位置的向量加到 token embedding 上，让输入同时包含语义与位置信息。 | 只关心 token 含义，忘了顺序变化会改变句子含义。 |
| attention weight | 当前位置对其他位置的参考比例。 | 相关性函数对两个 embedding 打分，所有位置的分数用于加权汇总上下文信息。 | 把 attention weight 简单等同于人类解释；它是模型内部计算权重，不必然是可读理由。 |
| multi-head attention | 从多种角度看相关性。 | 多组相关性计算模块各自学参数，分别产生 attention weight 和输出，再由后续网络整合。 | 以为只有一种“相关”，忽略 dog-cat 和 dog-bark 是不同类型关系。 |
| causal attention | 只看左边，不偷看未来。 | 自回归语言模型生成当前位置时只允许参考前面的 token，便于逐 token 生成并复用已有计算。 | 以为算完整 attention 一定更准；GPT 系列主要使用左侧注意力也能表现很好。 |
| output layer | 把内部向量翻译成“下一 token 概率”。 | 通常由 linear transform 加 softmax 组成，将最后位置表示映射到词表上的 probability distribution。 | 以为最后层直接输出文字；实际先输出概率，再由采样或解码策略选 token。 |

读完这一段，你应该能做什么：拿到一句输入文字时，能说清它先被切成 token，再变成向量、加位置、进 attention，最后变成概率分布。

# 讲义主线 {#lecture}
<!-- tag: LECTURE -->
<!-- layout: grid2 -->

## box
### 1. 从“带参数的函数”理解语言模型
视频开头把语言模型说成一个带参数的函数，这个说法很重要。函数的输入是一串文字，输出是下一步接什么；参数则是在训练资料中学出来的大量数字。现代语言模型通常用神经网络实现，而今天最常见的核心架构就是 Transformer。
直觉上，你可以把语言模型想成一个极复杂的接龙器：给它“User: 请解释 Transformer。AI:”这串前文，它不会先在脑中写完整篇答案，而是先判断第一个 token 应该接什么。接出第一个 token 后，再把它贴回输入，继续判断第二个 token。
正式一点说，这种逐步预测下一个 token 的方式叫自回归生成。它解释了为什么采样策略会影响输出，也解释了为什么模型上下文窗口、缓存、长文本成本会成为真实系统中的大问题。

## box
### 2. tokenization：文字进模型前要先切块
token 是模型处理输入和输出的基本单位，但它不一定等于人眼看到的字或词。很多中文模型可能把一个方块字当作 token；像 ChatGPT 这类多语言模型，中文一个字也可能被拆成多个 token。token list 由模型设计者或算法决定，不同模型可以完全不同。
BPE 是常见切分方法之一。它会从大量文字中寻找频繁出现的片段，把这些片段放进词表。这样既能覆盖罕见字串，又能让常见词组用较少 token 表示。注意，tokenization 通常预设的切分规则和词表。
学习边界：tokenization 只负责把文字变成离散符号，不负责理解语义。run 和 jump 在 token 层面只是两个不同编号，模型还需要 embedding 才能把语义相近关系表示出来。

3. embedding：让符号有可计算的距离token 本身只是符号，run 和 jump 并不会因为都是动作就天然更接近。embedding 做的事是把每个 token 映射成一串数字，让语义相近的 token 在向量空间中更接近。这个映射embedding table。模型看到某个 token，就去表里查对应向量。表里的数字是模型参数的一部分，会在训练中自动学出来。你现在应该能做什么：区分 token id 和 embedding。id 是离散编号，embedding 是参与后续神经网络计算的向量。
4. positional embedding：光有语义还不够句子顺序会改变意思，所以模型还需要知道每个 token 的位置。一个常见做法是给每个位置也准备一个向量，再把位置向量加到 token embedding 上。这样输入向量同时带有两种信息：这个 token 大概是什么，以及它出现在第几个位置。早期 Transformer 的位置向量可以人工设计，后来也常被当作可学习参数。你现在应该能做什么：解释为什么“我打你”和“你打我”即使 token 相同，模型也不能忽略位置。
5. attention：把上下文信息加进来同一个 token 的初始 embedding 是固定的。bank 在“去银行开户”和“河岸边散步”里先查到同一个 token embedding。attention 负责根据整句上下文更新它，让它变成 contextualized embedding。简化地说，attention 输入一排向量，输出另一排同样长度的向量。对每个位置，它先和句子中其他位置计算相关性，也就是 attention weight；再按这些权重把其他位置的向量加权求和。你现在应该能做什么：不用公式也能说清 attention 是“先判断相关，再汇总相关信息”。
6. multi-head attention：相关性不只一种视频用 dog、cat、bark 说明：dog 和 cat 都是动物，所以相关；dog 和 bark 也相关，因为 bark 是狗会做的动作。这两种相关性不是同一个角度。multi-head attention 就是准备多组相关性计算模块。每个 head 有自己的参数，能学到不同类型的联系，分别算出一组 attention weight 和输出，再交给后续网络整合。你现在应该能做什么：解释为什么只有一个“相关性分数”往往不够。
7. Transformer block：attention 后还要整理attention 会产生一组或多组带上下文的信息，但这些信息还要被整理、压缩、转换。feedforward network 的角色就是综合 attention 输出，产生新的 embedding。attention 加 feedforward network 合起来就是一个 Transformer block。一个语言模型通常堆很多层；文献里常把一个 block 称作一个 layer，虽然 block 内部本身也包含多层神经网络。你现在应该能做什么：看到“第几层 Transformer”时，知道它大致指的是经过多少个 block 的表示更新。
8. output layer：从向量回到词表概率多层 block 处理完后，模型取句尾位置的向量，送入 output layer。这个层通常包含 linear transform 和 softmax，输出整个词表上的 probability distribution。概率最高的 token 不一定总被选中，实际生成还会受 temperature、top-k、top-p 等解码策略影响。但本讲先抓核心：Transformer 负责把上下文变成句尾向量，output layer 负责把句尾向量变成“下一步接什么”的概率。你现在应该能做什么：把“输出文字”拆成“输出概率分布”和“按策略选 token”两个步骤。

## box
### 完整知识链路
把本节串起来，可以这样讲：语言模型先把输入文字切成 token sequence，再把每个 token 查表变成 token embedding，并加入 positional embedding。接下来，这串向量通过多层 Transformer block；每个 block 里的 attention 让当前位置从上下文收集相关信息，multi-head attention 从多种角度计算相关，feedforward network 再把这些信息整理成新的表示。最后，模型取最右边，也就是当前句尾位置的向量，通过 output layer 得到下一个 token 的概率分布。生成出一个 token 后，它会被接回原输入，模型再次执行同一套流程，直到产生结束符号。你现在应该能做什么：不看视频，也能画出“文字输入到下一 token 概率”的流程图。

线索 4：attention 不是那篇论文才发明的《Attention Is All You Need》的贡献证明可以不靠 RNN，只用 attention 也能把序列任务做好。这个历史点能帮助你理解 Transformer 为什么取代了很多 RNN 用法。你现在应该能做什么：说清“attention 机制”和“Transformer 架构”的关系。
线索 5：multi-head 来自多种相关性相关性不只有一种。dog-cat 是同类关系，dog-bark 是动作关系。多头注意力让模型用多组参数从不同角度计算 attention weight，再把多组输出整合。你现在应该能做什么：用一个新例子说明为什么多头比单头更合理。
线索 6：长文本难在平方级 attention 成本如果输入有 100K token，attention 不是读 100K 次就结束，而要计算大量 token 两两关系。理想化全 attention 的计算量随长度平方增长，所以长上下文成为模型效能竞争的重要指标。你现在应该能做什么：解释为什么长文本不是“多读一点”这么简单。

# 核心概念精讲 {#worked}
<!-- tag: WORKED -->

核心直觉：语言模型每一步只选下一个 token长回答看起来像一次性想完，其实生成过程是一格一格推进：把文字切成 token，转成向量，加上位置信息，经过 Transformer blocks，最后在词表上得到下一个 token 的概率分布。选出一个 token 后，它又变成下一步的上下文。你现在应该能做什么：把一段生成结果拆成多个“预测下一个 token”的步骤。
易混边界：Token、id、embedding 不是一回事Token 是切分后的符号；token id 是词表编号；embedding 是模型训练出来的向量。id 接近不代表意思接近，embedding 接近才可能表示模型内部认为它们在某些上下文中有相似用法。经过 attention 后，embedding 还会变成带上下文的表示。你现在应该能做什么：看到 token 编号时，不把数字大小当作语义关系。
Worked example：从“机器学习”到下一个字输入“机器学习很”先被 tokenizer 切成若干 token，例如“机器”“学习”“很”。模型把这些 token 转成向量，并用 causal attention 只看左侧上下文。最后一格输出词表概率：可能“有趣”概率 0.31，“重要”概率 0.22，“困难”概率 0.08。采样若选“有趣”，下一轮上下文就变成“机器学习很有趣”。你现在应该能做什么：说明 temperature 或 top-p 为什么会改变最终文字，而不改变模型已经算出的上下文表示本身。

# 最小例子 {#example}
<!-- tag: EXAMPLE -->

下面用伪代码把本讲流程压成最小可读版本。它帮助你检查每个名词在流程里出现在哪里。

最小示例：从文字到下一个 tokenpython-like

```text
context = "大型语言模型的基本动作是"
token_ids = tokenizer.encode(context) # tokenization：文字 -> token id
token_vecs = embedding_table[token_ids] # token embedding：token id -> 向量
pos_vecs = positional_embedding[:len(token_ids)]
x = token_vecs + pos_vecs # 同时带有语义和位置

for block in transformer_blocks:
 x = block.attention(x, causal=True) # 只看当前位置左侧上下文
 x = block.feedforward(x) # 整理 attention 收集的信息

last_vec = x[-1] # 当前句尾位置
logits = output_layer(last_vec) # 映射到整个词表
probs = softmax(logits) # probability distribution
next_token = sample(probs)
context = context + tokenizer.decode(next_token)

# 重复这个步骤，就得到一段完整回答。
```

## box soft
### 逐行理解
这段伪代码把语言模型的核心动作压到最小：根据上下文输出下一个 token 的概率分布，再按采样策略选择 token。温度、top-p、停止条件都会影响最后看到的文字。
你现在应该能做的事：给这段例子补一个输入样本，再写出至少两条验收标准。

错误示例：把 token id 当语义距离diagnosis

```text
# 错误想法：id 数字接近，所以语义接近
if abs(token_id("run") - token_id("jump")) < 10:
 print("run 和 jump 一定语义相近")

# 修正：语义关系主要看训练得到的 embedding 或上下文表示
similarity = cosine(embedding("run"), embedding("jump"))
```

## box warn
### 为什么这是错的
token id 只是词表里的编号，编号接近不代表意思接近。真正承载可计算语义关系的是 embedding。并且初始 embedding 还没考虑上下文，经过 attention 之后才会变成 contextualized embedding。
你现在应该能做什么：看到“token 编号”“词表顺序”“向量距离”三个词时，能说清哪个才更接近语义关系。

# 误区诊断 {#diagnosis}
<!-- tag: DIAGNOSIS -->
<!-- layout: grid2 -->

## box redsoft
### 常见坑
1. 把 token 当作人类直觉里的字或词。错因：忽略 tokenizer 和词表差异。后果：上下文预算、切分分析和输出长度估计都会错。修法：用 tokenizer 工具查看实际切分。
2. 把 token id 当作语义距离。错因：编号只是词表索引。后果：会误判 run、jump、apple 之间的关系。修法：看 embedding 或上下文化后的表示。
3. 以为同一个 token 在不同语境里一开始就有不同 embedding。错因：混淆 token embedding 和 contextualized embedding。后果：看不懂 attention 为什么必要。修法：先查固定 embedding，再通过 attention 加入上下文。
4. 把《Attention Is All You Need》理解成发明 attention。错因：忽略历史背景。后果：误读论文贡献。修法：记住它的关键是不用 RNN，只靠 attention 构建 Transformer。
5. 认为完整 attention 一定比 causal attention 更好。错因：没从文字接龙生成过程看问题。后果：低估计算复用和自回归约束的重要性。修法：理解生成第 n 个 token 时，未来 token 尚不存在。
6. 把长文本成本想成线性增长。错因：只想到读入 token，没有想到两两相关性。后果：低估 100K token 上下文的算力需求。修法：记住 attention 成本大致随长度平方增长。
