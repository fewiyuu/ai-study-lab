第12讲：浅谈评估大型语言模型能力的各种方式

这一讲不是在背一串排行榜名称，而是在训练一种判断力：当有人说“某模型接近 GPT-4”“某模型世界第一”时，你要能追问它是在哪个 benchmark 上、用什么提示词、怎样解析答案、是否有人工或模型裁判、有没有资料污染，以及成本是否值得。

评估语言模型比评估传统分类器难，因为语言模型可以输出任意文字。即使是选择题，也会遇到“答案是 B”“我认为 B 的值是 1”“B 的概率最高但模型其实最想输出 3”这种解析冲突。开放题、翻译、摘要、长上下文、心智理论和安全决策会让问题更复杂。

任务与样本→答案解析→评分者→污染检查→成本权衡

学完后要能做四件事

  1. 看到一个模型排行榜时，能说清它测的是哪类能力，不把成绩直接等同于真实可用性。
  2. 能解释选择题、开放题、人工竞技场和 LLM-as-judge 各自的优点与偏差。
  3. 能用大海捞针、BigBench、Machiavelli、心智理论等例子说明“能力测试”如何被设计。
  4. 能诊断 prompt、选项位置、答案长度偏好、资料污染和成本因素怎样改变评估结论。

概念骨架

MAP

评估先从任务开始

Benchmark corpus 是一组输入和预期答案。它可以是 MMLU 的选择题，也可以是 MT-Bench 的开放写作题、BigBench 的怪题、长上下文里的大海捞针，或文字冒险游戏里的决策。

判断句：先问“题目代表什么能力”，再看分数；题目不代表你的真实场景，分数再高也只能当参考。

答案解析会改变成绩

语言模型输出不受限，评分要先定义“怎样算答对”。只看选项字母、只看 A/B/C/D 的概率、要求模型只输出一个字母，这三种做法可能给出不同分数。

判断句：当同一个模型在同一个 benchmark 上分数不同，先查解析规则、提示词、选项顺序和输出格式。

榜单不是终点

人工竞技场接近真实偏好，但成本高；LLM-as-judge 省人力，却会偏爱长答案或某类表达；公开 benchmark 会被训练资料污染；真实部署还要看价格、延迟和硬件。

判断句：评估报告若没有误差分析、污染检查和成本信息，就还不能指导选型。

样本

题目是否覆盖目标能力，是否足够新，是否代表真实用户。

提示

同一题用不同 prompt 可能让长上下文或选择题成绩明显变化。

评分

标准答案、人工偏好、BLEU/ROUGE、GPT-4 裁判各有边界。

解读

成绩要和污染风险、失败样本、成本、延迟一起看。

术语表

TERMS
术语直觉解释准确说法常见误解
Benchmark corpus一套大家都来考的题库。由输入样本、标准答案或评分规则组成，用于比较模型在特定任务上的表现。把 benchmark 分数当成模型在所有任务上的真实能力。
MMLU用大量学科选择题测模型知识与推理。Massive Multitask Language Understanding，包含多学科、多难度的选择题。以为选择题一定好评分，忽略选项顺序、代号和答案解析规则。
BLEU / ROUGE用文本重叠程度给翻译或摘要打分。BLEU 常用于机器翻译，ROUGE 常用于摘要，二者都依赖候选输出与参考答案的重合。把同义改写当成错误，或把高重合当成高质量。
Chatbot Arena让人类比较两个模型谁回答得更好。随机给两个匿名模型同一问题，由人类投票，再形成排行榜。忘记它测的是人类偏好，不是单一任务的精确正确率。
MT-Bench用 GPT-4 评开放题。包含多轮开放问题，常用强模型作为裁判来给其他模型答案打分。忽略裁判模型自己的偏见，尤其是偏爱长答案。
LLM-as-judge让一个语言模型当评分员。把候选答案、参考答案或评分 rubrics 交给强模型，由它输出分数或胜负判断。以为强模型裁判就是客观真理，不做校准和偏差分析。
Needle in a Haystack在很长文本里找一小段关键信息。把“针”插到长上下文不同位置，检查模型是否能在不同长度、不同位置取回信息。只看上下文窗口大小，忽略模型是否真的用到了中间信息。
资料污染模型考前可能看过题。训练资料中混入公开 benchmark 或相近改写题，导致分数虚高。把高分都解释成泛化能力强，不检查模型是否记过答案。

读完这一段，你应能拿到一个评估结果后，先问“题库、评分、裁判、污染”四件事。

讲义主线

LECTURE

1. 评估的基本形状

最朴素的评估流程很简单：准备输入，准备标准答案，让模型 A 和模型 B 面对同样输入，再比较输出和答案。这个流程适合图像分类、情感分类，也适合一部分语言模型任务。它给了我们一个起点：评估不是听厂商声明，而是让模型在同一批题目上留下可比较的记录。

问题在于，语言模型不是只能输出一个类别。它可以解释、补充、绕开格式，也可以把正确选项藏在很长的句子里。于是“输出是否正确”本身就变成了评估设计的一部分。你现在应能做的第一件事，是把任何评估拆成“样本、标准、解析、统计”四个环节。

2. 选择题也没有想象中稳

MMLU 看似容易评，因为每题都有 A/B/C/D。可模型输出“B”、输出“答案是 B”、输出“B 这个选项的数值是 1”，解析规则会给出不同结果。另一种做法是直接看模型对 A/B/C/D 四个 token 的概率，谁最高就算选谁；但如果模型对数字“3”的概率最高，对 B 的概率只是四个选项里最高，你要不要算它答对？这里没有天然答案，只能在报告里说明规则。

选项顺序也会影响结果。字幕里提到，把正确答案都挪到 A 后，偏爱猜 A 的模型会突然占便宜；把选项符号从大写字母换成数字或加括号，也可能让分数下降。判断句：选择题评估不是“题目有答案就万无一失”，而是要检查输出约束、解析规则、选项位置和代号格式。

3. 开放题需要偏好评估

翻译和摘要没有唯一正确答案。BLEU、ROUGE 用重合程度做近似，但“幽默”和“诙谐”这种同义表达可能被低估。人工评审更贴近质量，却很贵。

判断句：开放题的分数只能说明某种评分标准下的表现，不能替代具体错误分析。

4. Chatbot Arena 与人工偏好

Chatbot Arena 让人类同时看两个匿名模型的回答，再投票决定谁好。它常被用来说明某个模型是否接近 GPT-4，因为它更像真实用户的主观比较。

边界是人类时间有限，题目分布也来自愿意参与投票的人。它适合看综合偏好，不适合替代专业任务验收。

5. LLM-as-judge 的诱惑

MT-Bench 和 AlpacaEval 这类评估会让 GPT-4 等强模型当裁判。优点是省掉大量人工，且和 Arena 排名有时相关性很高。缺点是裁判模型本身有偏见，比如偏爱更长、更会展开的答案。

判断句：用模型评分时，要报告裁判模型、评分 prompt、是否做长度校正，以及和人工偏好的一致性。

6. BigBench 与“折腾模型”

BigBench 收集大量奇怪任务，例如 Emoji Movie、文字棋谱、从混杂字符串中找英文单字。它的意义不是每个任务都实用，而是用非常多角度去压测模型是否出现某些能力。

看到这类题，要问它测的是知识、推理、模式识别、上下文跟踪，还是只是熟悉网络上的经典例子。

7. 长上下文不等于都读懂

Needle in a Haystack 把一段关键事实插入长文不同位置，再问模型能不能找回。GPT-4 在较短上下文里表现稳定，但接近极长上下文时，中间位置可能丢失。Claude 的例子还说明，prompt 多加一句“请给出最相关句子”就可能明显改变结果。

判断句：上下文窗口大小只是容量，检索到信息才是能力。

8. 特定能力测试要防背题

Machiavelli benchmark 用文字冒险游戏看模型是否为达目标不择手段；心智理论测试看模型能否区分不同角色知道什么。经典 Sally-Anne 问题如果早在网上出现过，模型答对可能只是见过相似题。

所以新题、改写题、训练时间之后发布的数据集，都比公开老题更能检验泛化。

9. 资料污染会让榜单虚高

公开 benchmark 的题目在网上可见，而语言模型预训练资料也来自互联网。模型可能无意中读过测试集，甚至能复述某些数据集里的训练样本。字幕里提到，有研究直接问模型要某些数据集的训练资料，部分模型能吐出原文，这就是强烈证据。

另一种污染更隐蔽：不是直接训练原题，而是训练了与 MMLU 内容相同、只是换句话说的新题。小模型可能因此在 MMLU 上接近强模型，但这不代表它真的具备相同泛化能力。学到这里，你应能解释为什么“测试集公开”会削弱评估可信度。

10. 选型还要看成本

课程最后提醒，能力不是唯一指标。API 价格、硬件需求、延迟、上下文长度和失败成本都影响模型是否值得使用。两个模型平均 benchmark 分数只差一点，但一个贵很多，真实项目可能会选便宜且够用的那个。

完整的评估报告应把能力、风险和成本放在同一张表里。只写“某模型第一名”太粗糙；更可靠的说法是“在某类任务、某套提示词、某个预算下，这个模型更合适”。

完整知识链路

评估大型语言模型要从题库开始，但不能停在题库。选择题要定义解析规则，开放题要定义偏好标准，人工 Arena 要承认人力成本，LLM-as-judge 要校准裁判偏见，BigBench 和长上下文测试要看题目是否真测到目标能力，公开 benchmark 还要防止资料污染。最后，把分数放回成本、延迟和真实任务中解释。你现在应能把一个模型评估报告拆成这条链，并指出哪一环最可能让结论失真。

评估线索

线索

下面把字幕里的例子整理成适合回看的材料，不逐字堆字幕。每个例子都对应一个评估设计问题。

MMLU 的解析陷阱

课程用 Llama 65B 在不同论文中分数不一致做开场。差异可能来自输出解析：只接受单个字母、接受“答案是 B”、或读取 A/B/C/D 概率，都会改变正确率。

对应能力：检查选择题评估是不是被格式规则影响。

选项位置与符号

把正确答案统一移到 A，偏爱猜 A 的模型会被抬高；把 A/B/C/D 换成数字或括号格式，GPT-3.5 的表现也可能变化。

对应能力：识别 benchmark 中看似无关的呈现方式。

开放题的评分者

翻译和摘要有多个合理答案，BLEU/ROUGE 只能近似；Chatbot Arena 用人类偏好，MT-Bench 用 GPT-4 评审，二者都要看相关性和偏差。

对应能力：区分标准答案、文本重叠、人工偏好和模型裁判。

BigBench 的怪题

Emoji Movie、棋谱、隐藏单字等任务用于观察模型在多种奇怪输入下的表现。它们提醒我们：能力不是一个数字，而是很多任务切面。

对应能力：判断一个任务到底在测什么。

大海捞针与 prompt

长上下文测试会把同一条信息插入不同位置。Claude 的案例说明，提示词稍改，取回表现可能大幅变化。

对应能力：把上下文长度和实际可取回能力分开看。

资料污染与背题

公开 benchmark 可能被模型训练资料覆盖。模型能复述 RTE 等数据集样本，或在相似改写题上异常高分，都提示评估可能被污染。

对应能力：解释为什么新数据集和训练时间线很重要。

最小例子

EXAMPLE

下面的例子不追求完整程序，只用最小结构说明评估为什么不能只写“模型答对/答错”。关键是把输出解析规则写成显式步骤。

正确示例：选择题评估要固定解析规则python
case = {
  "question": "四个选项中哪一个正确？",
  "options": ["A", "B", "C", "D"],
  "gold": "B",
  "raw_output": "答案是 B，因为第二个选项符合题意。"
}

def parse_choice(raw_output, allowed=("A", "B", "C", "D")):
  # 先找明确答案短语，再退回到首个选项字母；报告里要写清这条规则。
  for mark in allowed:
    if f"答案是 {mark}" in raw_output:
      return mark
  for token in raw_output.split():
    if token in allowed:
      return token
  return None

pred = parse_choice(case["raw_output"])
is_correct = pred == case["gold"]

逐行理解

raw_output 保留模型原话，因为错误分析要看模型到底说了什么。parse_choice 把“怎样算选 B”写出来，避免同一个输出在不同论文中被不同方式处理。is_correct 只是最后一步，它不能替代前面的解析说明。

你现在应能做的事：给一个选择题评估报告补上“输出格式、解析规则、选项顺序、失败样本”四项说明。

错误示例：把输出中出现字母 B 就算答对python
def bad_parse(raw_output):
  return "B" if "B" in raw_output else "unknown"

bad_parse("我认为 B 选项的数值是 1，所以正确答案其实是 A。")
# 这会被误判成 B，分数被抬高。
长度校正的直觉：最终分数 = 裁判给分 - λ × max(0, 回答长度 - 合理长度)。这里的 λ 不是固定真理，只是在提醒你：LLM-as-judge 若偏爱长答案，就要把长度偏差纳入分析。

Worked case：客服模型评估报告怎么判

场景：你要比较两个客服模型。公开榜单上模型 A 的 MMLU 更高，模型 B 的价格低 40%。如果只看榜单，容易直接选 A；但客服任务真正关心的是能否按公司政策回答、能否承认不确定、会不会编造退款承诺，以及高峰期延迟是否可接受。

步骤要做的事判断理由
1. 定义场景抽 200 条真实客服问题，覆盖退款、账号、投诉、越权请求和多轮追问。样本要代表真实风险，不能只用通用知识题。
2. 固定输出规则要求模型给最终答复、引用政策条款、标记是否需要人工升级。输出形状固定后，解析和人工复核才可比较。
3. 写评分 rubric正确性 40%、政策合规 30%、不编造 20%、表达清楚 10%。开放题没有唯一答案，rubric 要先于评分出现。
4. 看失败样本单独统计“答非所问”“编造承诺”“该升级未升级”。总分相近时，失败类型比平均分更能决定上线风险。
5. 加成本约束报告每千次对话价格、P95 延迟、人工复核比例。能上线的模型不是分数最高的模型，而是在风险和成本下最合适的模型。

结论写法：如果模型 A 只在通用题上领先，但客服评估中高风险错误更多、延迟更高，就不能凭 MMLU 选 A。你现在应该能把“榜单第一”改写成“在目标场景、评分规则、失败样本和成本约束下是否更好”。

评分方式

观察重点：长上下文、开放题裁判和榜单分数都不是孤立数字。拖动滑块时，把它理解成“上下文越长，越要检查针的位置和 prompt”；切换评分方式时，观察结论从粗略准确率逐步变成可解释的选型判断。

误区诊断

DIAGNOSIS

常见坑

  1. 只看排行榜名次。错因是把综合偏好当成真实任务能力；后果是选错模型；修法是追溯题库、提示词、评分者和任务分布。
  2. 选择题只看答案字母。错因是忽略自由输出；后果是把解释文本或错误语义误判为正确；修法是固定解析规则并公开失败样本。
  3. 把 BLEU/ROUGE 当成质量本身。错因是把文本重合等同于语义正确；后果是同义好答案被低估；修法是结合人工或模型裁判。
  4. 相信 LLM-as-judge 完全客观。错因是忽略裁判模型偏好；后果是长答案或某种风格被抬高；修法是做长度校正、人工抽检和多裁判比较。
  5. 把上下文窗口大小当成阅读能力。错因是混淆“能塞进去”和“能找回来”；后果是长文任务上线后漏读中间证据；修法是做不同位置的大海捞针测试。
  6. 忽略资料污染。错因是测试集公开却不查训练时间线；后果是背题模型被误判为泛化强；修法是使用新题、私有测试集或污染检测。
