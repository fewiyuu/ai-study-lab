# 概念骨架 {#skeleton}
<!-- tag: MAP -->

问题：自回归等待
大语言模型通常按自回归方式生成：先根据上下文算第一个 token，再把这个 token 接回上下文去算第二个 token。这个过程的关键限制计算依赖链很硬：下一个位置的输入里必须包含上一个位置的输出。
如果只生成短答，等待感可能不明显；但当你要长篇回答、长代码或长报告时，逐 token 等待会把延迟拉长。视频开头提到，一些实验中加上 Speculative Decoding 后吞吐能达到原来的两倍到三倍，这正是因为它减少了必须串行等待的轮数。


方法：先猜再验
Speculative 的意思是“猜测、投机”。它的核心让一个很快的预言家先草拟接下来几个 token。目标模型随后用并行计算检查这些草稿前缀：第 0 个前缀检查第 1 个 token，第 1 个前缀检查第 2 个 token，第 2 个前缀检查第 3 个 token。
只要草稿 token 与目标模型在相应前缀下的预测一致，就接受；第一个不一致处之后的内容全部丢弃。这样保留了目标模型说了算的原则。


边界：不白赚
这个技巧默认你更在意等待时间，而不是单次请求消耗的总计算。目标模型一次要检查多个前缀，猜错的分支会被丢掉，预言家的生成也要花时间。若预言家很快且命中率不差，延迟下降；若预言家很慢或总猜错，收益会消失。
所以它像外挂，但不是魔法。真正判断它是否值得用，要看草稿速度、接受长度、目标模型批处理效率和硬件资源。

**1 输入上下文**用户问题、已生成 token、系统约束。
**2 草稿预测**预言家快速给出 A、B、C 等候选 token。
**3 并行构造**把原输入、输入+A、输入+A+B 同时交给目标模型。
**4 验收前缀**逐位比较目标模型输出与预言家草稿。
**5 接受或回退**接受连续正确 token；错误处之后重新生成。

你现在应该能做什么：看到一个加速说法时，先判断它减少的是串行轮数、单轮计算量，还是只提高了硬件并行利用率。

# 术语表 {#terms}
<!-- tag: TERMS -->
<!-- layout: grid2 -->

## box
**自回归模型**Auto-regressive model。每次根据已有上下文预测下一个 token，然后把新 token 接回上下文继续预测。误区是以为它不能并行计算任何东西；其实训练或固定前缀下的多个位置可以并行，难点在生成时未来 token 尚未确定。
**Speculative Decoding**推理阶段的解码加速方法。草稿端先提出若干 token，目标模型并行验证这些 token 是否符合自己的输出。它不要求重新训练目标模型，也不直接改变目标模型本身。
**预言家**视频里的口语比喻，指速度极快、允许犯错的草稿产生器。它可以是小模型、非自回归模型、压缩模型、搜索引擎，甚至多个来源。
**目标模型**真正决定最终输出分布的大语言模型。它负责验收草稿，哪些 token 被接受，最终仍以它的判断为准。

## box
**接受前缀**从第一个草稿 token 开始，连续与目标模型预测一致的部分。只接受连续前缀，不跳过中间错误去接受后面的 token，因为后面的上下文已经建立在错误 token 上。
**非自回归模型**Non-autoregressive model。它倾向于一次产生多个位置，速度快但质量可能差。视频强调：质量差不致命，因为它只是预言家，最终仍由目标模型验收。
**模型压缩**把大模型变小或变快，例如量化、蒸馏等。字幕里把 knowledge distillation 口误成 destination，正确术语是知识蒸馏。压缩后模型可能更容易错，但很适合当草稿端。
**计算换时间**并行验收会做一些最终丢弃的计算，因此总计算可能增加；但如果硬件能并行处理，用户等待时间会下降。这个权衡是理解本讲的关键。

你现在应该能做什么：用“谁先猜、谁验收、错了怎么办、成本在哪里”四个问题解释任何一个 Speculative Decoding 变体。

# 讲义主线：为什么要加速 {#lecture}
<!-- tag: LECTURE -->
<!-- layout: grid2 -->

## box
### 自回归慢在哪里
语言模型生成一句话时，看起来是在连续写字；从计算角度看，它是在重复运行“给定当前上下文，预测下一个 token”。第一个 token 没出来，第二个 token 的输入就不完整；第二个 token 没出来，第三个 token 的输入也不完整。这个串行依赖让长文本生成很难像普通矩阵运算那样一次全算完。
视频把这个痛点讲得很直观：第一个 token 没生成之前，我们没办法生成第二个 token。因此，如果每轮只能吐一个 token，生成越长，等待越明显。聊天短答可能还能接受，但长篇解释、代码生成、批量摘要、长文翻译都会把这个等待放大。
Speculative Decoding 的切入点很巧：既然目标模型不能凭空知道未来 token，那就让另一个很快的系统先猜未来。目标模型不需要相信它，只需要检查它。这样把“必须等未来出现”变成了“先假设未来可能是这些 token，再并行验证”。

## box soft
### 为什么说它像外挂
李宏毅在字幕里反复强调，这个方法和你使用的目标模型没有强绑定关系，也不需要对目标模型重新训练。你可以把它理解成推理时加在模型外面的调度策略：原模型仍然负责最终输出，只是前面多了一个草稿来源，旁边多了一套并行验收流程。
这也是“神奇”的地方。通常模型想变快，要换结构、剪枝、量化、蒸馏或重新部署；Speculative Decoding 则可以在不改变原模型权重的情况下减少用户感知延迟。它让大模型每一轮有机会确认多个 token。
但“外挂”不等于没有条件。预言家必须足够快，目标模型必须能高效处理并行前缀，系统也必须接受额外计算资源消耗。否则它只是把问题从等待时间转移到无效计算上。

$$
直觉公式：有效速度提升 ≈ 每轮可接受 token 数 ÷ 原本每轮 1 个 token。若一轮平均接受 2.4 个 token，用户看到的生成轮数大约降到原来的 1 / 2.4。
$$

你现在应该能做什么：解释为什么这个方法加速的是“生成轮数和延迟”，而不是保证总 FLOPs 一定下降。

# 机制线索：预言家机制 {#source}
<!-- tag: 线索 -->
<!-- layout: grid2 -->

## box
### 一次吐出三个 token 的直觉
假设当前输入是蓝色上下文。预言家快速预测：目标模型接下来会先输出红色 token，再输出黄色 token。于是系统把三份输入同时交给目标模型：原始蓝色上下文、蓝色上下文加红色 token、蓝色上下文加红色加黄色 token。
目标模型对第一份输入预测“下一个 token 是什么”，这可以验证红色 token；对第二份输入预测“红色之后下一个 token 是什么”，这可以验证黄色 token；对第三份输入预测“红黄之后下一个 token 是什么”，这给出再往后的新 token。如果红、黄都被目标模型认可，那么这一轮外部看起来就产出了红、黄和第三个新 token。
关键点是目标模型并没有跳过自己的判断。它只是利用了 Transformer 对一批前缀并行计算的能力。预言家的 token 像临时脚手架，帮目标模型提前看到多个可能的未来前缀。

## box info
### 为什么不是直接用预言家生成
视频里提出了一个自然疑问：如果预言家都能猜中，还要大语言模型干什么？答案是，预言家不需要很聪明，它只需要够快且经常猜对前面若干步。它会犯错，而目标模型的工作正是把这些错误挡住。
直接用预言家生成，质量会变成预言家的质量；用 Speculative Decoding，最终质量仍由目标模型控制。草稿端负责“提出候选”，目标端负责“同意或否决”。可以把它理解成先写出多个草稿位置，再由目标模型同时检查多个前缀，而不是等完整草稿写完才逐字处理。
因此，预言家越接近目标模型，接受的 token 越多，速度越快；预言家越离谱，接受的 token 越少，速度收益越低。

普通自回归X → A第一轮只能得到 A。
预言家草稿AB快速猜未来两个 token。
目标模型验收X / A / B三份前缀同批计算。
外部效果ABC若前两步猜中，一轮产出三个。

伪代码：草稿和并行验收pseudo

```text
context = user_prompt
draft = prophet.generate(context, k=2) # 例如 [A, B]

prefixes = [
 context, # 验 A
 context + [A], # 验 B
 context + [A, B] # 生成额外的 C
]
target_outputs = target_model.forward(prefixes)

accepted = longest_prefix_agree(draft, target_outputs)
context = context + accepted + next_safe_token(target_outputs)
```

## box warn
### Worked case：一轮验收到底提交哪些 token
假设当前上下文是“明天台北的天气”，草稿模型一次猜 3 个 token：可能会下雪。目标模型同批计算四个前缀：原上下文、原上下文+可能、原上下文+可能+会、原上下文+可能+会+下雪。
被目标模型检查的前缀目标模型给出的下一个 token用来验证结果
明天台北的天气可能草稿第 1 个 token接受“可能”
明天台北的天气可能会草稿第 2 个 token接受“会”
明天台北的天气可能会下雨草稿第 3 个 token拒绝“下雪”
明天台北的天气可能会下雪。错误前缀后的预测丢弃，不能越过错误继续用
这一轮最终提交的是“可能”“会”，再接上目标模型在正确前缀下给出的“下雨”。外部看到 3 个可信 token，但系统没有相信草稿里的“下雪”。你现在应该能做的事：给任意一组草稿 token 标出“连续接受到哪里、从哪里开始丢弃、目标模型补上的第一个 token 是什么”。

你现在应该能做什么：把“输入、草稿、并行前缀、目标模型输出、接受前缀”五个对象在一张纸上画出来。

# 最小例子：犯错与验收 {#example}
<!-- tag: EXAMPLE -->
<!-- layout: grid2 -->

## box
### 预言家第二步猜错时
视频给了一个很重要的错误场景：预言家先猜红色 token，这一步对；第二步猜灰色 token，但目标模型真正会输出黄色 token。系统仍会把“输入”“输入+红色”“输入+红色+灰色”一起喂给目标模型。目标模型算完后，我们就能比较：第一步草稿红色与目标模型一致，所以红色可接受；第二步草稿灰色与目标模型不一致，所以灰色及其后续全部不能用。
为什么灰色后面的绿色输出也不能信？因为它是在错误前缀“输入+红色+灰色”上算出来的。即使绿色看起来合理，它的上下文已经偏离目标模型真实路径，所以必须丢弃。Speculative Decoding 接受的是连续正确前缀，不会越过错误继续拿后面的 token。
即便如此，这轮仍然赚到了：原本一轮只能得到红色一个 token；现在因为第一步猜中，除了红色，还能从目标模型知道第二步应该是黄色，因此一轮产出两个可信 token。速度收益从三倍退到两倍，但结果不坏。

## box
### 全部猜错时为什么近似不赔
最糟糕的情况是预言家烂到不行，草稿 token 全部不被目标模型接受。这时所有带有草稿 token 的前缀输出都不能用，只有原始上下文那一路得到的第一个目标 token 是可信的。也就是说，这一轮至少还和普通自回归一样拿到了一个 token。
这就是视频中“几乎不赚不赔”的意思：从用户看到的 token 数量看，你没有比普通生成更差；但系统确实损失了预言家的时间和那些无用前缀的计算。所以更精确地说，是延迟上的下界接近普通生成，计算资源上可能更贵。
这也解释了为什么预言家必须“奇快无比”。如果草稿端本身慢，全部猜错时就会明显拖累；如果草稿端很快，失败代价主要是额外算力，而不是等待时间。

# 交互观察：谁能当预言家 {#viz}
<!-- tag: OBSERVE -->
<!-- layout: grid2 -->

## box
### 候选一：非自回归模型
非自回归模型的特点是快，因为它倾向于一次产生多个位置。缺点是生成质量通常不如自回归模型，容易犯错。放在普通生成场景里，这个缺点很致命；但放在 Speculative Decoding 里，它反而很适合当预言家：快是刚需，错可以被目标模型拦住。
从这个角度看，Speculative Decoding 可以被理解成非自回归模型和自回归模型的结合。非自回归模型提供草稿，自回归大模型提供最终验收。一个负责速度，一个负责质量。

## box soft
### 候选二：压缩后的小模型
另一条路线是从目标模型或同族模型压缩出一个更小、更快的草稿模型。常见手段包括量化、剪枝、知识蒸馏等。字幕中说到 knowledge destination，这里应理解为 knowledge distillation，即知识蒸馏。
小模型压缩后可能“脑袋不好使”，输出质量下降；但只要它和目标模型在常见前缀上足够相似，就能大量猜中前几个 token。它不需要独立交付最终答案，只需要快速提供候选。

## box info
### 候选三：搜索引擎或资料库
视频还提到一种很有意思的做法：预言家不一定是语言模型。假设当前上下文是“语言模型”，系统可以去资料库里搜索常见续写，例如“经常使用在许多自然语言处理方面的应用”。搜索速度如果足够快，这段检索到的常见续写也可以作为草稿。
这说明 Speculative Decoding 的本质“快速草稿加目标模型验收”。草稿来源可以很灵活，只要它便宜、快、与目标模型输出有一定重合。

## box warn
### 候选四：多个预言家
一定只能有一个预言家吗？视频答案是否定的。多个预言家可以各自提出草稿，目标模型再比较不同草稿路径。哪个草稿和目标模型更一致，哪个就能帮助接受更多 token。
多预言家提高了命中机会，但也增加并行分支和计算成本。它适合硬件并行资源充足、延迟目标很紧、草稿来源互补的场景。若分支太多却命中率一般，额外计算会迅速膨胀。

| 预言家来源 | 优势 | 风险 | 适合判断 |
| --- | --- | --- | --- |
| 非自回归模型 | 一次生成多个 token，速度快 | 质量弱，可能频繁被拒绝 | 是否能稳定命中前缀 |
| 压缩小模型 | 部署便宜，接近目标模型风格 | 压缩过度会降低接受率 | 速度收益是否超过额外计算 |
| 搜索引擎/资料库 | 对固定短语和常见续写很快 | 可能贴出不合上下文的续写 | 检索结果是否与目标模型偏好一致 |
| 多个预言家 | 提高至少一条路径猜中的概率 | 分支过多导致算力浪费 | 硬件并行是否足够、分支是否互补 |

你现在应该能做什么：面对一个草稿来源，判断它是否足够快、是否经常猜中、猜错时代价是什么。

# 误区诊断 {#diagnosis}
<!-- tag: DIAGNOSIS -->

下面的例子把视频流程压成最小可读伪代码。重点看每一步的语义：草稿端只提出候选，目标模型负责批量验收，系统只提交连续正确前缀。

正确示例：接受连续正确前缀python-like

```text
def speculative_step(context, prophet, target, k=3):
 draft = prophet.fast_draft(context, k) # 例如 ["红", "灰", "绿"]
 prefixes = [context + draft[:i] for i in range(k + 1)]

 # 目标模型一次检查多个前缀：
 # prefixes[0] 检查 draft[0]
 # prefixes[1] 检查 draft[1]
 # prefixes[2] 检查 draft[2]
 target_next = target.forward_batch(prefixes)

 accepted = []
 for i, token in enumerate(draft):
 if target_next[i] == token:
 accepted.append(token)
 else:
 accepted.append(target_next[i]) # 错误处改用目标模型 token
 break

 return context + accepted
```

## box soft
### 逐行拆解
`fast_draft` 表示预言家先猜 k 个 token，它的目标是快，不是最终可信。`prefixes` 构造出一组前缀：原始上下文、加一个草稿 token、加两个草稿 token，依此类推。`forward_batch` 是加速的关键，因为目标模型可以在一轮里并行处理这些前缀。
循环里的比较体现了验收逻辑。只要目标模型在第 i 个前缀下的输出等于草稿第 i 个 token，就接受；一旦不一致，就把目标模型自己的 token 作为安全输出，并停止使用更后面的草稿。这样做的原因是后续草稿建立在错误上下文上，不能继续信。

错误示例：越过错误继续接受anti-pattern

```text
# 错误：只要某个位置猜中就接受，不管前面有没有错
accepted = []
for i, token in enumerate(draft):
 if target_next[i] == token:
 accepted.append(token)

# 问题：如果 draft[1] 已经错了，draft[2] 的验证上下文也错了。
# 即使 draft[2] 看起来匹配，也不能把它交给用户。
```

## box
### 集中误区
1. 误以为预言家必须和目标模型一样强。错因：把草稿生成和最终验收混成一件事。修法：预言家只要快且经常猜中前缀。
2. 误以为猜错会污染最终答案。错因：忘了目标模型会逐位验收。修法：只接受连续正确前缀，错误处之后丢弃。
3. 误以为速度提升完全免费。错因：只看用户等待时间，不看额外前缀计算。修法：同时记录延迟、吞吐、接受率和算力消耗。
4. 误以为草稿越长越好。错因：草稿越长，后面越容易基于错误前缀白算。修法：根据接受率选择合适 k。
5. 误以为它只能配小语言模型。错因：把一种实现当成本质。修法：任何快速且有命中率的来源都可能当预言家。
