54_P55｜作业九：Quick Summary of Lecture Video

作业九的目标很明确：给你一段演讲影片，先用 Whisper 做自动语音辨识，把音讯转成带时间轴的逐字稿；再设计 prompt，让大型语言模型把逐字稿整理成 300-500 字以内的繁體中文摘要；最后保留评分平台导出的结构化记录，作为验收依据。

概念骨架

MAP

先看任务作业把指定演讲片段浓缩成可评分摘要。输入是演讲音讯，过程包含语音辨识、文本切块、摘要生成，输出是 300-500 字以内的繁體中文摘要。核心直觉：这是“音讯转文字，再从长文字压成短摘要”的资料压缩任务，不是写观后感。判断句：如果没有说清字数、语言、影片来源和验收格式，就还没读懂任务。

再看流水线Whisper 负责把语音转成文字，LLM 负责把文字压缩成摘要。中间的切块不是装饰，而是为了解决长逐字稿超过模型上下文的问题。这一步要问：每一段代码在处理哪种资料，输出会被下一步怎样使用。

最后看验收摘要看起来流畅不等于质量合格。评分会看内容完整性、事实准确性、语言流畅性、内容一致性和避免重复；最终产物还要保留结构化评分记录。判断句：最终答案必须同时满足内容质量和记录格式。

术语表

TERMS
术语直觉解释准确说法常见误解
ASR把声音里的话转成文字。Automatic Speech Recognition，输入音讯，输出带时间或片段边界的文字转录。以为 ASR 会自动完成摘要；它只负责转写，不负责取舍重点。
Whisper这次用来做语音辨识的模型。OpenAI 开源的 encoder-decoder 语音模型，训练资料约 680K 小时，覆盖多语言、多任务。以为模型越大一定越适合；作业测试中 medium 已可得到不错结果。
逐字稿影片声音对应的文字记录。作业中的转写结果包含开始时间、结束时间和该时间段文字，例如 0-4 秒讲了某句话。把逐字稿当最终答案；它通常冗长、重复，需要再摘要。
Transcribe把语音辨识成同语言文字。Whisper 的任务 token 可指定转写；本作业是中文语音转中文文字。和 translate 混用，导致任务方向错。
Translate把语音内容翻译成另一种语言。Whisper 也训练在语音翻译任务上，例如外文语音翻成英文。以为作业要翻译；本作业重点是中文 ASR 和中文摘要。
Chunk Text把长逐字稿切成较短段落。sample code 默认可把每段限制在一定字数内，字幕中提到预设值 512，并示例切成 9 段。觉得切块只是排版；实际上它在控制上下文长度与摘要粒度。
Multi-stage Summarization先分段摘要，再汇总摘要。每个 chunk 先生成局部摘要，最后把多个局部摘要整合成 final summary。只做第一层局部摘要就停下，遗漏全局整合。
Refinement摘要一边读新段落一边更新。先摘要第一段，再把当前摘要和下一段一起输入模型，逐步得到覆盖更多段落的摘要。以为它和多阶段完全一样；它的中间状态会连续累积。
评分 JSON评分平台导出的结构化记录。摘要经评分器检查后，导出包含会话与评分信息的 JSON，用于复核摘要质量和流程完整性。手动改评分结果，或把纯文字摘要当成结构化记录。

读完这一段，你应该能把术语放回作业流程里：谁把音讯变文字，谁处理长文本，谁负责评分和记录。

讲义主线

LECTURE

1. 作业真正要你交什么

作业九训练你把“演讲影片”这种非结构化资料转成可评分的摘要。课程材料使用李琳山老师《信号与人生》的授权片段；原演讲约两个多小时，练习为了方便截取约 17 分钟。你的工作是从这段影片取得内容，再输出一段 300-500 字以内的繁體中文摘要。

直觉上，这是一条资料压缩链：声音很长，先转成文字；文字仍然很长，再压成摘要；摘要还不能只流畅，要符合事实、结构和评分格式。这里的“Quick Summary”快在你有明确流程，不用每次从零摸索。

你现在应该能做什么：用一句话说出本作业的输入、处理过程、输出限制和验收对象。

2. 为什么先 ASR 再摘要

大型语言模型擅长读文字，但不能自然地直接读普通影片档里的声音。作业因此先安排 ASR，也就是自动语音辨识。ASR 的产物是逐字稿，它把音讯切成一段一段，并记录每段开始时间、结束时间和对应文字。

逐字稿不是最终答案，因为演讲口语常有重复、停顿、转折和口头补充。它的价值在于给摘要模型提供可读材料。你可以把 ASR 理解成“把影片变成可被 LLM 消化的文本原料”，而摘要模型负责“从文本原料中保留主旨、删去冗余”。

你现在应该能做什么：看到一段带时间轴的 ASR 输出时，判断它是中间材料，不是最终摘要。

  1. Whisper 怎么读音讯Whisper 的结构可粗略分成 encoder 和 decoder。encoder 把语音讯号编码成一串有意义的向量，decoder 再根据这些语音向量生成文字结果。字幕里特别提到，模型一开始会做语言辨识，例如这次中文片段会识别成中文；接着用任务 token 指定 transcribe 或 translate，再逐步预测时间戳和文字。你现在应该能做什么：解释为什么 Whisper 输出里会同时有语言、任务方向、时间轴和文字。
  2. 训练资料意味着什么课程介绍 Whisper 时提到它训练在非常大量的网络资料上，论文中约 680K 小时，覆盖 99 种语言，并包含语音辨识与语音翻译等任务。这说明它一个多语言、多任务语音模型。好处是泛化能力强；边界是它仍可能把专有名词、断句、口音或噪声辨错。你现在应该能做什么：既能说出 Whisper 为什么适合本作业，也能指出 ASR 结果需要人工检查。
  3. 长逐字稿为什么要切块如果演讲很长，ASR 逐字稿会超过模型一次可处理的上下文。直接把全文塞给 LLM 也许会失败，或让模型忽略前后段落。sample code 因此提供 chunk_text，让你设定每段文字上限。字幕中提到预设值 512，示例会把内容切成 9 段。切块的目标让每次摘要输入都可控。你现在应该能做什么：看到长文本摘要任务时，主动想到分块、局部摘要和最终整合。
  4. 多阶段摘要Multi-stage summarization 的直觉是“先小结，再总整理”。把长逐字稿切成多个段落后，分别请 LLM 生成摘要一、摘要二、摘要三，再把这些局部摘要合并成 final summary。它的优点是结构清楚、容易平行处理；风险是每段局部摘要若漏掉关键点，最后整合时也补不回来。因此 prompt 要提醒模型保留主旨、事实、人物、时间线和重要转折。你现在应该能做什么：画出“chunk → chunk summary → final summary”的流程。
  5. Refinement 摘要Refinement 的直觉是“带着已有摘要继续读”。先摘要第一段，再把第一段摘要和第二段原文一起交给模型，更新成前两段摘要；之后继续拿当前摘要和下一段原文迭代。它的优点是摘要会连续累积，适合逐段修正；风险是早期摘要如果方向偏了，后面会沿着偏差继续压缩。使用时要让 prompt 明确要求“保留新段落的重要补充，并修正前文不准确处”。你现在应该能做什么：说清 refinement 与 multi-stage 的中间状态差异。
  6. 评分与记录不是附录作业使用评分平台检查摘要质量。评分模型和温度设置固定，摘要会按五个 criteria 评估：内容完整性、事实准确性、语言流畅性、内容一致性和避免重复。最终要检查的不只是纯文字摘要，还包括评分后导出的 JSON 记录。公开学习页只保留通用原则：摘要文本、评分结果、会话记录和文件命名要能互相对应。你现在应该能做什么：把内容产出和格式验收分开检查，避免“摘要写好了但记录不合格”。

完整知识链路

把本节串起来，可以这样讲：作业九训练的是“从影片到可评分摘要”的完整流程。先读取指定演讲片段，用 Whisper 的 ASR 功能得到带时间轴的中文逐字稿；再用 chunk_text 控制长文本长度，选择 multi-stage summarization 或 refinement，让 LLM 生成 300-500 字以内的繁體中文摘要；接着把摘要送入评分器，按完整性、准确性、流畅性、一致性和避免重复五个面向取得评分；最后导出 JSON 记录，按规则命名并复核。你现在应该能把这条链路拆成可执行清单，也能指出任一步出错会影响什么。

最小例子

EXAMPLE

例子只保留作业最小结构，目的是让你看懂每一步的输入、输出和检查点。真实 sample code 会更长，但核心判断与下面一致。

代码
video = load_video_source()
segments = speech_recognition(
 audio=video.audio,
 model_name="medium",
 language="zh",
 task="transcribe",
 temperature=0
)

chunks = chunk_text(segments.transcript, max_words=512)
local_summaries = [
 summarize(chunk, prompt="保留主旨、事实、人物与转折，使用繁體中文。")
 for chunk in chunks
]
final_summary = summarize(
 "\n".join(local_summaries),
 prompt="整合成 300-500 字繁體中文摘要，避免重複。"
)

assert 300 <= chinese_char_count(final_summary) <= 500
assert uses_traditional_chinese(final_summary)

为什么错

这段流程跳过了前处理、摘要、评分和 JSON 记录导出。即使 ASR 内容大致正确，逐字稿也会太长、太口语、可能重复，而且记录格式也不符合要求。

你现在应该能做什么：看到“能跑出文字”时，不急着当成最终答案，而是继续检查摘要质量和验收格式。

Worked example：multi-stage 和 refinement 怎样不同

假设逐字稿被切成三段：A 讲演讲背景，B 讲技术方法，C 讲人生经验。multi-stage 会先得到 A摘要、B摘要、C摘要，再把三段小摘要合成 final summary；如果 B摘要 漏掉关键方法，最终摘要通常也补不回来。

refinement 则先用 A 生成 当前摘要，再读 B 更新成 A+B摘要，最后读 C 更新成 A+B+C摘要。它更像一路修订一份摘要，但如果 A 的摘要一开始抓错主线，后面可能沿着错方向继续压缩。你现在应该能按“中间状态是否并列存在”区分两种策略。

误区诊断

DIAGNOSIS

常见坑

  1. 把逐字稿直接当最终摘要。错因：混淆中间材料和最终摘要。后果：过长、重复、口语化且格式不合。修法：先切块摘要，再评分导出 JSON。
  2. 把 transcribe 写成 translate。错因：没区分转写和翻译。后果：可能得到英文或任务方向错误的文字。修法：本作业指定中文语音转中文文字。
  3. 完全不检查 ASR 结果。错因：以为 Whisper 输出必然正确。后果：专有名词、断句或噪声错误会进入摘要。修法：抽查关键片段和明显错字。
  4. 只设计一句“帮我摘要”。错因：prompt 没写字数、语言、保留重点和避免重复。后果：摘要不符合评分条件。修法：把五项评分标准写进提示。
  5. 局部摘要后没有最终整合。错因：误把 multi-stage 的第一层当作完成。后果：答案像多段小结拼贴。修法：再做 final summary，统一主线和措辞。
  6. 手动修改评分结果或记录。错因：忽视评估记录的完整性。后果：后续复核失真。修法：只保留平台正常导出的结构化记录。
