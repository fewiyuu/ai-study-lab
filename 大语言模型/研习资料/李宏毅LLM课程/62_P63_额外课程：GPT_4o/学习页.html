62_P63｜额外课程：GPT-4o

这节课讨论 GPT-4o voice mode 背后可能的语音技术。重点不是确认 OpenAI 实际用了哪一种架构，而是用已有语音技术推理：一个模型要像 demo 那样会换语气、会笑、能被打断、能边看边聊，系统内部至少要解决哪些问题。

概念骨架

MAP

先澄清 demo字幕一开始就提醒：当时很多人以为手机 App 里已有的语音按钮就是 GPT-4o voice mode。其实旧语音界面也能“听你说话再回答”，但它通常只是把语音转文字、把文字交给 ChatGPT、再把回答合成为语音。所以第一层判断是：你看到的是“语音入口”，还是“模型真的以声音为核心输入输出”。两者体验都像聊天，但技术边界不同。

再拆架构旧版级联系统的优势是组件成熟：ASR 负责语音辨识，LLM 负责文本回答，TTS 负责语音合成。问题是声音里的情绪、喘气、笑声、打断时机，容易在转成文字时丢失。端到端语音模型的直觉是少绕一圈：让同一个多模态模型直接看声音表示，预测接下来该发出什么声音或保持安静。

最后看互动真正自然的语音互动不只是“延迟低”。模型还要判断人是否说完、是否想插话、是否只是合唱或背景声；如果有视觉输入，还要在需要时回看过去看过的画面。这一节的核心链路是：表示声音、训练声音接龙、对齐成会聊天的角色、在多通道中决定下一步行动。

术语表

TERMS
术语直觉解释准确说法常见误解
GPT-4o voice mode让人像打电话一样和模型实时说话。课程讨论的是展示中呈现的能力，以及可能支撑这些能力的语音与多模态技术。把手机 App 原有语音按钮等同于展示里的 voice mode。
级联系统把成熟模块串起来完成语音聊天。典型流程是 ASR 将语音转文字，LLM 生成文字回答，TTS 再合成语音。以为只要每个模块够快，就能完整保留语音情绪和互动细节。
端到端模型尽量用一个模型直接从输入模态走到输出模态。在本节语境中，指单一多模态模型直接处理声音输入并生成声音或声音单位。以为端到端一定简单，忽略训练资料、表示方式和实时控制难度。
speech unit声音被压缩后的离散单位。encoder 用 codebook 把波形表示成一串 unit，decoder 再把 unit 还原成波形。把 speech unit 当成文字 token，忘了它可能包含笑声、语气、非语音声音。
codebook声音单位的字典。一组可被 encoder 选择的离散 code，用来表示不同声学模式。以为每个 code 都能直接翻译成一个中文词或英文词。
混合编码器能用文字表示的用文字，不能用文字表示的用声音单位。把 ASR 与声学 unit 结合，保留语言内容，也保留笑声、喘气、情绪等非文字信息。以为文字转写已经包含了语音里的全部信息。
speaker diarization自动分清“谁在什么时候说话”。给一段音讯，标出不同说话者对应的时间片段。把它当成身份认证；它更核心的问题是分段和说话者区分。
alignment让预训练模型学会按人类期待互动。课程中把 fine-tuning 与 RLHF 类对齐步骤合起来看，用对话资料塑造回答方式和安全边界。以为有了预训练就天然会做助理，不需要对话数据。
同时听说看模型说话时还继续接收世界。把用户声音、模型自己已发出的声音、视觉输入放在不同通道里，让模型对所有通道做 attention。以为听到任何声音都应该立刻停下，其实合唱、插话、背景声需要区分。

读完这一段，你应该能把“语音聊天”拆成表示、生成、训练、对齐和多通道互动五个问题。

讲义主线

LECTURE

1. demo 里真正让人惊讶的不是“能说话”

旧版语音界面早就可以让用户用声音输入，再听到模型用声音回答。它的工作方式并不神秘：语音辨识把声音变成文字，语言模型在文字上做回答，语音合成把文字念出来。只要这三个模块足够快，用户也会感到“还算即时”。

GPT-4o voice mode 展示中更特别的地方，是它好像能处理语音内容之外的信息。它能用多种语气说话，能轻声、加速，甚至像唱出来；它也可能听出喘气、紧张、笑声之类的声学线索，并产生非语音声音，比如笑声。

这就把问题从“会不会语音输入输出”推进到“模型有没有真正处理声音里的细节”。如果一切都先转成文字，很多声学信息会在第一步丢掉；如果系统直接处理声音表示，就有机会保留这些内容。

2. 级联方案能做到很多，但会留下缝隙

课程里先讨论了一个保守猜想：也许在旧系统外面加模块。比如 ASR 之外增加语音事件侦测或情绪辨识，把“用户很紧张”“用户在笑”这样的标签附在文字后面交给语言模型。输出端也可以让语言模型多写一些控制符号，例如“括号笑”或“用轻声说”，再交给能理解指令的 TTS。

这个方案并非不可能，因为语音情绪识别、能读控制指令的语音合成、能产生笑声的 TTS 都已经有相关技术。但串接越多，系统越容易在延迟、同步和错误传播上出问题。一个模块判断错，后面的模块会基于错误标签继续生成。

GPT-4o 官方介绍强调 voice mode 是端到端模型，因此课程后半段转向另一个猜想：让一个多模态模型直接听声音、处理声音表示、再产生声音输出。

  1. 语音接龙不能直接逐点生成文字语言模型做的是文字接龙：给它未完成的句子，预测下一个 token。语音语言模型也可以类比成声音接龙：听一段声音后，预测接下来该是什么声音。但原始声音太密。一秒 16 kHz 音频有 16000 个取样点，如果逐点预测，一秒声音就要跑一万六千步，非常慢，也很难学到高层结构。因此语音模型通常先压缩：encoder 把波形变成离散 speech unit，模型在 unit 序列上接龙，decoder 再把 unit 还原成波形。
  2. speech unit 保留文字之外的信息把语音转成文字也是一种压缩，但这种压缩只保留“说了什么”。比如用户说“好好笑”然后笑了几声，ASR 可能只留下“好好笑”，后面的笑声就消失了。speech unit 的价值在于它可以表示文字难以表达的声学内容：笑声、喘气、情绪、停顿、非人声或背景音。这样模型才有机会听见“内容之外”的线索。它的代价是可能重造轮子。许多 unit 本来对应的就是可用文字表示的语音内容，如果全部都用新符号，原有语言模型学过的文字知识不容易直接复用。
  3. 混合编码器是一种折中混合编码器的直觉很朴素：能辨识成文字的部分就用文字，不能用文字表示的部分再用 speech unit。这样既保留语言内容，也保留笑声、语气和其他声学细节。例如“好好笑”可以保留为文字，后面的笑声用特殊 unit 表示。混合 decoder 或 TTS 再把文字和特殊符号一起变成声音。这种方案的好处是语言模型不用从零学习所有符号。它已经懂文字，只需要额外学习那些文字无法表达的声音单位如何影响理解和输出。
  4. 说话者分段让模型知道“谁说了什么”当多个用户一起和模型说话时，模型要知道不同声音来自不同人。speaker diarization 负责把一段音讯切成不同说话者的片段，并给出类似 A、B 的说话者标签。这把对话结构整理出来。模型看到的输入可能变成“A: 这句话”“B: 那句话”，其中某些片段仍用 speech unit 表示。如果没有这一步，模型可能把两个人的话混成一个上下文，回答时就会误判对象、情绪和任务。
  5. 训练不能只靠语音资料语音版语言模型可以用大量视频或音频做预训练，例如从影音平台取得声音，让模型学习 speech unit 接龙。这样能学到音色、语气、停顿、背景声等声音规律。但只靠语音资料不足以学到足够世界知识。课程里用数量级说明：一百万小时语音大约对应几十亿文字 token，而大型文字模型训练常用到远多于此的文字 token。所以合理猜想是：语音模型会利用已有文字语言模型的能力，或把语音符号和文字一起训练，让模型在“已经懂文字”的基础上学会听懂声音。
  6. alignment 把“会接龙”变成“会聊天”预训练只让模型学会声音和文字的模式，不保证它会当一个好助理。文字模型需要监督微调和人类反馈，语音模型同样需要语音对话数据来学习什么时候回答、怎样拒绝、用什么角色声音说话。如果希望模型总用某个固定声音输出，还需要让它学会目标说话风格。可能的方法包括少量目标声音数据、让强预训练模型快速模仿，或用 voice conversion 把不同人的对话转换成目标音色。因此 voice mode 的能力预训练、文字知识迁移、语音对话对齐和声音风格控制共同作用。

9. 最难的是同时听、说、看

文字聊天有明确边界：用户按 Enter，模型开始生成；用户按停止，模型停止。语音聊天没有这么清楚。用户停顿一下，模型要判断这是说完了，还是在想下一句；用户发出声音，模型要判断这是打断、合唱、背景声，还是仍在继续原句。

课程借 dialog speech language model 的思路说明：可以把“外界声音”和“模型自己已经说出的声音”放在不同频道。模型每一步同时看这些频道，决定输出真实声音、输出 wow 这样的反应，还是输出 silence 表示继续等。

如果加入视觉，问题更复杂。模型正在描述灯光时，画面里有人比了手势；它未必立刻评论，但当用户问“有没有看到奇怪的东西”时，模型要对听觉、视觉和自己生成的历史一起做 attention。你现在应该能做的事：把一个自然语音 demo 拆成“声音表示、训练来源、对齐资料、通道决策”四段来解释。

精讲例子

WORKED

直觉：语音自然不等于“把文字念快一点”

旧版语音聊天像三段流水线：ASR 把声音转成文字，LLM 生成文字答案，TTS 把答案念出来。它可以很快，但只要第一步把笑声、停顿、喘气、音量和反讽语气丢掉，后面的语言模型就很难再利用这些信息。

GPT-4o voice mode 的关键直觉是：声音本身也应成为模型可处理的上下文。模型不只读“用户说了哪些字”，还要读“用户怎么说、什么时候停、是否在打断、画面里发生了什么”。

术语边界：端到端不等于没有中间表示

  • 级联系统 是多个独立模块串接，清楚但容易丢声学细节。
  • 端到端模型 是让一个模型直接在多模态表示上做决策，但仍可能需要 encoder、decoder、speech unit。
  • speech unit 是声音的离散表示，不等于文字 token，也不一定能翻译成一个词。
  • alignment 不只约束回答内容，也约束何时开口、何时停、用什么语气互动。

最容易错的是把“能用麦克风聊天”当成 voice mode。判断时要看它是否保留声学信息、能否处理打断和多通道注意力。

$$

worked example：用户打断模型时发生什么

用户问问题后模型正在回答，用户突然说“等等，不是这个意思”。级联系统可能只检测到有人声然后硬停；更自然的系统要同时看外界声音、模型自己已说出的内容、历史对话和可能的视觉输入，判断这是打断、补充、背景声还是短促反馈。

好的 voice mode 难点不只是延迟低，而是 turn-taking：什么时候继续说，什么时候停，什么时候短声回应，什么时候保持 silence。

$$

级联怎么读

看到 ASR → LLM → TTS，先问 ASR 输出里有没有笑声、喘气、停顿、说话者标签。如果没有，这些信息通常无法靠后面的文字模型恢复。

speech unit 怎么读

speech unit 把连续波形压缩成可接龙的离散单位，让模型不用逐个音频采样点预测。它能保留更多声学细节，但训练、解码和控制都更复杂。

错误答案长什么样

“端到端模型直接吃原始 16 kHz 波形并逐点输出声音”通常不是现实可行的直觉。一秒就有 16000 个采样点，模型更可能先用表示学习把声音压缩成 unit 或混合符号。

你现在应该能做什么：把一个语音 demo 拆成输入表示、生成表示、声音还原、对话对齐和多通道 turn-taking 五个检查点。

最小例子

EXAMPLE

这里用伪代码把三种思路放在一起。它帮你读懂“输入、表示、生成、还原、互动判断”这几件事。

代码
audio = microphone.listen()
text = asr.transcribe(audio)
answer_text = llm.generate(text)
answer_audio = tts.speak(answer_text)
speaker.play(answer_audio)

这段代码的含义

这里模型在压缩后的声音单位上接龙。encoder 负责把波形变成 unit，speech_lm 负责预测接下来该产生什么 unit，decoder 再把 unit 还原为声音。

如果 unit 设计得好，它可以承载笑声、语调和非语音声音。困难在于训练成本、资料规模、实时性，以及怎样让模型既懂声音也懂文字世界知识。

错误示例：把所有人声都当成打断pseudo

if microphone.detects_voice():
 model.stop_speaking()

错在哪里

这条规则过于粗糙。用户可能是在合唱、补充背景声、跟旁边人说话，也可能只是发出短促反应。自然语音互动需要区分“打断模型”“继续原句”“一起发声”“背景噪音”等情境。

更合理的做法是把外界声音、模型自己正在说的内容和历史上下文放进不同频道，让模型判断下一步是继续说、停下来、短暂回应，还是保持 silence。

常见坑

  1. 把旧版语音按钮当成 GPT-4o voice mode。修法：检查它是否只有 ASR、LLM、TTS 级联，是否能处理打断、情绪和非语音声音。
  2. 以为端到端模型等于不需要中间表示。修法：端到端仍需要可学习的声音表示，speech unit、文字或混合符号都可能出现。
  3. 以为 ASR 转写已经保留了所有信息。修法：问笑声、喘气、语气、背景音、停顿是否仍在表示里。
  4. 以为音频越多，知识越足。修法：区分声学能力和世界知识，理解为什么要复用文字模型或联合训练。
  5. 以为听到用户声音就应该停止输出。修法：区分打断、合唱、背景声、继续说话和短促反馈。
  6. 以为多模态就是当前画面描述。修法：模型可能要对过去看过的视觉片段和当前听到的问题一起做 attention。
