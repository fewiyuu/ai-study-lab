LLM FOUNDATION · BLOCK 01

文本进入大模型前，先经历了什么

大语言模型并不直接处理"一句话"。它看到的是一串整数编号，再通过 embedding 表变成一串向量。context window 决定这串输入最多能有多长，也决定模型这次推理最多能同时参考多少上下文。

这页只打牢第一条输入链路：tokenizer 负责切分和编号，embedding 负责把编号映射成可计算的向量，context window 负责限制这次输入能装下多少 token。把这三件事分清，后面看 attention、RAG、长上下文、微调数据格式都会顺很多。

原始文本→ token→ token id→ embedding 向量→ 带位置的输入序列

学完后要能做四件事

  1. 看到一段文本，能判断它会先被拆成 token，而不是按字或按词固定处理。
  2. 能解释 token id 和 embedding 向量的关系：编号不是语义，向量才进入计算。
  3. 能说清 context window 的边界：超出窗口会被截断、压缩或分块，不会被模型完整看见。
  4. 能检查最小工程流程里的 padding、truncation、attention mask 和长度预算。

概念骨架

MAP

概念：token 是模型的输入单位

token 不是固定的汉字、英文单词或标点。它是 tokenizer 根据词表和切分规则得到的片段。一个中文词可能被切成一个 token，也可能被切成多个；一个英文长词也可能被拆成词根和后缀。

判断句：看到"这段话有多少 token"时，不要用字数直接代替，要用同一个模型的 tokenizer 计算。

边界：编号本身没有语义距离

token id 只是词表里的序号。编号 100 和编号 101 并不一定比编号 100 和编号 900 更接近。模型真正拿去计算的是 embedding 向量，向量在训练中被调整，才会逐渐带上语义和用法信息。

判断句：不要从 token id 的大小推断语义相似度。

工程动作：先做长度预算

一次调用模型前，先估算输入 token 数。系统提示、few-shot 示例、历史消息、检索文本和用户问题都要算进去。预留给输出的 token 也会占模型服务的预算。

判断句：如果模型答非所问，先查是否截断了关键上下文。

tokenizer：把文本切成模型认识的片段

TOKENIZE

它解决什么问题

神经网络只能处理数字张量。tokenizer 的第一份工作，是把自然语言文本变成有限词表中的片段。这个词表可以包含常见汉字、英文词、词缀、标点、空格模式、代码符号，也可以包含特殊标记，例如句子开始、句子结束、填充、未知片段。

为什么不直接按词切？因为真实文本里有新词、拼写变化、代码、URL、emoji、多语言混写。按完整词建词表会爆炸，按单个字符又会让序列太长。很多现代 tokenizer 会用 BPE、WordPiece 或 SentencePiece 一类的子词方法，在"词表大小"和"序列长度"之间折中。

常见错法是拿肉眼看到的词数估算 token 数。中文里，一个看起来短的专业词可能被拆成多个片段；英文里，空格也常常影响 token 形态。模型不同，tokenizer 也可能不同，同一句话在两个模型里 token 数未必一样。

你读到这里，应能解释：tokenizer 不是翻译器，也不是 embedding。它只负责把文本映射到词表片段和编号。

token id：编号只是查表钥匙

IDS

从 token 到整数

tokenizer 切出 token 后，会把每个 token 查成一个整数编号。比如某个模型里，"猫"可能是 1325，"喜欢"可能是 7788。这些数字方便程序组成张量，也方便 embedding 表按行索引。

这里最容易误会的是"编号大小有含义"。实际不是。编号更像数据库主键：它告诉系统去 embedding 表的哪一行取向量。主键相邻，不代表两条记录含义相邻。语义关系来自训练后的向量空间，而不是编号顺序。

最小工程动作是打印编码结果。遇到模型输出奇怪、截断异常、特殊标记错位时，先把 token、id、attention mask 一起打印出来看。很多输入 bug 不在模型，而在前处理。

例子：同一句话的三层表示

层级样子用途

逐行拆解：查看 token、id 和长度python
# 示例展示的是检查思路。不同模型的 tokenizer 名称和输出会不同。
text = "请解释 RAG 和长上下文的区别。"
tokens = tokenizer.tokenize(text)
ids = tokenizer.encode(text, add_special_tokens=True)

print(tokens)        # 看文本被切成哪些片段
print(ids)           # 看每个片段对应哪个 token id
print(len(ids))      # 估算这段输入占多少 context window

第一行原始文本仍是字符串。第二行调用切分接口，是为了看 tokenizer 的真实切法。第三行编码成 id，并加入模型需要的特殊标记。最后打印长度，这一步常用于检查是否接近窗口上限。删掉长度检查，短文本通常没事；一旦把历史对话、检索文本和长问题拼在一起，问题就会变得隐蔽。

embedding：把离散编号变成连续向量

EMBED

embedding 表是什么

embedding 层可以看成一张大表。行数通常等于词表大小，列数等于模型隐藏维度。输入 token id 后，程序按 id 取出对应行，得到一个向量。一个长度为 N 的输入序列，会变成 N 行向量，每行宽度是 D。

这一步没有读上下文。也就是说，同一个 token id 在进入 Transformer 前查到的是同一行基础向量。后续 attention 会让它和上下文里的其他 token 混合，才得到具体语境中的表示。比如"苹果"在水果句子和公司句子里的初始 embedding 一样，但经过上下文层后会走向不同表示。

工程上常见动作是检查形状：input_ids 通常是 [batch, seq_len]，embedding 后变成 [batch, seq_len, hidden_size]。如果 seq_len 过长，后续 attention 的计算和显存压力都会上升。

最小形状读法

input_ids ∈ ℤB×N
embedding(input_ids) → X ∈ ℝB×N×D

B 是 batch size，表示一次送几条样本。N 是序列长度，也就是 token 数。D 是每个 token 向量的宽度。后面看 attention 时，最重要的是把 N 和 D 分开：一个是有多少位置，一个是每个位置有多少特征。

最小例子：embedding 查表不是 one-hot 进模型python
import torch
embedding_table = torch.nn.Embedding(num_embeddings=50000, embedding_dim=768)
input_ids = torch.tensor([[421, 9182, 51, 337]])

x = embedding_table(input_ids)
print(input_ids.shape)  # [1, 4]
print(x.shape)          # [1, 4, 768]

这段代码的关键是 Embedding 的输入和输出形状。输入是一串整数，不是 768 维向量。调用 embedding 表后，每个整数被替换成对应行向量，序列长度不变，最后多出隐藏维度。把 token id 直接喂给 attention 是错的，因为 attention 做的是向量间的点积、线性变换和加权求和。

context window：模型这次能看见的长度上限

CONTEXT

窗口限制的不是知识总量

context window 指一次输入和输出共同可使用的 token 长度上限。它不是模型训练时学过多少知识，也不是模型记忆容量。一个模型训练时看过很多资料，但单次推理时只能读取当前窗口里的文本和已生成内容。

如果输入超过窗口，系统通常会截断、拒绝、压缩、分块，或者只保留一部分历史。具体行为取决于调用框架和模型服务。错误诊断时，不能只问"模型为什么不懂"，要先查关键证据有没有被放进窗口。

长窗口也不是免费午餐。更长的上下文会提高输入成本，增加延迟，也可能让模型在大量无关文本中找不到重点。工程上常见做法是先检索、筛选、重排，再把最有用的片段放进窗口。

最小工程流程：从输入到可检查张量

PIPELINE

正确流程

最小输入流程可以按四步检查：准备文本，调用 tokenizer，得到 input_ids 和 attention_mask，再送入模型。padding 用来把同一个 batch 的样本补到一样长，attention mask 用来告诉模型哪些位置是真 token，哪些位置只是补齐。

truncation 的作用是超长时截断。它不是修复理解能力，而是防止输入长度超过上限。生产代码里不能只写 truncation=True 就放心，还要知道截断方向、最大长度、关键证据是否在保留部分。

一个稳妥习惯是把"最小工程动作"写成清单：打印 token 数，打印前后几个 token，检查特殊标记，检查 mask，确认预留输出长度。这些动作看起来朴素，但能抓住很多真实 bug。

错误诊断：静默截断

错误示例：没有检查关键证据是否被截掉python
encoded = tokenizer(
    long_prompt,
    max_length=4096,
    truncation=True,
    return_tensors="pt"
)
answer = model.generate(**encoded)

这段代码可能能跑，但不稳。它没有检查 long_prompt 被截掉了哪一段。如果答案依赖末尾的证据，而 tokenizer 从右侧截断，模型就根本没看见关键内容。修法是先统计长度，再明确截断策略；对长文档任务，优先分块、筛选和重排。

正确示例：编码后做基础验收python
encoded = tokenizer(
    prompt,
    max_length=8192,
    padding="longest",
    truncation=True,
    return_tensors="pt"
)

print(encoded["input_ids"].shape)
print(encoded["attention_mask"].sum(dim=1))
print(tokenizer.decode(encoded["input_ids"][0][-80:]))

第一行形状告诉你 batch 和序列长度。第二行统计每条样本的真实 token 数，padding 不应算进去。第三行解码最后一段 token，用来检查尾部是否保留了你以为保留的内容。删掉这三步，代码仍能运行，但你无法判断模型到底看见了什么。

误区、边界和易混概念

DIAGNOSIS

五个常见误区

  1. 把 token 当成汉字数。错因是用肉眼长度代替 tokenizer 结果；后果是窗口预算失真；修法是用目标模型 tokenizer 统计。
  2. 把 token id 当成语义向量。错因是看到数字就以为可比较大小；后果是误读模型输入；修法是区分 id 查表和 embedding 计算。
  3. 以为 embedding 会读取上下文。错因是把静态查表和 Transformer 层混在一起；后果是解释"苹果"这类词时说不清；修法是说明上下文混合发生在后续层。
  4. 以为长窗口一定更好。错因是忽略成本和噪声；后果是延迟高、费用高、相关信息被淹没；修法是先检索筛选，再放入窗口。
  5. 只开 truncation 不检查截断内容。错因是把能运行当成输入正确；后果是关键证据消失；修法是打印长度和解码片段。

三组易混概念

概念准确区别判断方法
token / wordword 是语言里的词；token 是 tokenizer 产生的模型输入片段。用 tokenizer 输出验证，不用肉眼猜。
token id / embeddingid 是离散编号；embedding 是查表得到的连续向量。看形状：id 多是整数矩阵，embedding 多一维 hidden size。
context window / 模型知识窗口是单次输入长度；知识来自训练参数和当前输入。问自己：这条证据是否真的放进本次输入。
padding / truncationpadding 是补齐短样本；truncation 是裁掉长样本。padding 配 mask，truncation 查保留内容。
