LLM FOUNDATION · BLOCK 03

别把预训练、微调和推理混成一件事

预训练让模型从海量文本中学会语言和知识模式。微调用更小、更有目标的数据调整模型行为。推理是在已有模型参数上，根据输入 prompt 生成、打分或取向量。三者都和大模型有关，但成本、目标、可控范围完全不同。

应用工程里还会出现 embedding、reranking、generation。它们不是训练阶段，而是常见调用角色：embedding 把文本变成检索向量，reranking 给候选结果重新排序，generation 生成自然语言或结构化输出。把这些边界分清，才能判断一个问题该用 prompt、RAG、微调、重排，还是换模型。

预训练→ 对齐 / 微调→ 部署模型→ 推理调用→ 评估反馈

学完后要能做四件事

  1. 能说明预训练、微调、推理分别改什么、不改什么。
  2. 能判断一个应用问题应该优先改 prompt、加 RAG、做 rerank，还是考虑微调。
  3. 能区分 embedding、reranking、generation 的输入输出和适用场景。
  4. 能诊断"微调能解决一切"、"RAG 等于微调"这类常见说法的问题。

角色骨架：哪一步在改什么

MAP

概念：预训练主要学通用能力

预训练通常用海量文本做下一个 token 预测。它让模型学会语言结构、常识片段、代码模式和大量统计关联。它改动的是模型参数，成本最高，也最不适合个人应用里随意重做。

判断句：如果你只是想让模型知道一份新文档，优先别说"预训练一下"。

边界：微调不是知识库同步

微调适合稳定改变风格、格式、任务习惯或领域表达，但不适合把大量频繁变化的事实硬塞进参数。事实经常变时，检索和上下文注入通常更稳。

判断句：高频更新的资料，优先放外部知识库，而不是靠微调记住。

最小工程动作：先定位问题类型

答案格式乱，先改 prompt 或输出约束；缺最新资料，先做检索；候选文档太多，考虑 rerank；长期风格不稳定，再考虑微调和评估集。

判断句：不要一遇到效果差就直接微调。

预训练：让模型拥有基础语言和世界模式

PRETRAIN

它解决什么问题

预训练的直觉是：让模型读大量文本，反复练习根据前文预测下一个 token。这个目标看起来简单，但数据足够大、模型足够大时，模型会学到语法、事实片段、推理模式、代码结构、问答格式等能力。

正式说，预训练会更新模型参数。参数不是数据库条目，而是神经网络权重。模型并不是把每篇文章原样存起来，而是在预测任务中形成统计模式。因此它可能知道很多东西，也可能产生过时、混淆或编造的内容。

预训练的边界很重要。它成本高，需要大量数据、算力和训练工程。普通应用很少自己做完整预训练。更多时候，我们使用已经预训练好的基础模型，在推理侧组织上下文，或在微调侧做小范围行为适配。

你读到这里，应能说清：预训练改参数，目标是通用能力，不是给应用临时塞一份文件。

下一个 token 预测

给定 token 序列 x1, x2, ..., xt
模型学习 P(xt+1 | x1:t)

这行公式的意思是：看见前面的 token 后，模型给下一个 token 分配概率。训练时，真实下一个 token 已知，模型根据预测误差调整参数。推理时，模型根据概率和解码策略选出下一个 token，再把它接回上下文继续生成。

微调：用更小的数据改变模型行为

FINETUNE

微调适合做什么

微调是在已有模型基础上继续训练。它可以让模型更稳定地遵守某类格式、适应某个任务口径、学习特定领域表达，或通过偏好数据变得更符合人类反馈。常见方式包括全量微调、LoRA/QLoRA 这类参数高效微调，以及指令微调、偏好对齐等训练目标。

微调不是 prompt 的替代品。prompt 是每次调用时给模型的输入，适合快速试错和注入临时要求。微调会改变参数，适合那些重复出现、靠 prompt 很难稳定约束、且有高质量样本的问题。

微调也不是 RAG 的替代品。RAG 把外部知识放进当前上下文，让模型基于检索结果回答。微调更适合学"怎么答"，RAG 更适合补"答什么依据"。如果资料更新频繁，RAG 往往比微调更容易维护。

微调前先问四个问题

  1. 问题是否稳定重复出现？偶发问题先用 prompt 和检索处理。
  2. 是否有足够高质量样本？脏数据会把错误习惯写进模型。
  3. 是否有评估集？没有评估，无法判断微调是变好还是只在样例上变乖。
  4. 是否需要更新事实？高频事实更新优先放知识库。

微调的第一步不是开训练脚本，而是定义任务边界和评估方法。

最小例子：一条指令微调样本长什么样json
{
  "messages": [
    {"role": "system", "content": "你是客服质检助手，只输出 JSON。"},
    {"role": "user", "content": "请判断这段客服回复是否承诺了无法保证的结果：..."},
    {"role": "assistant", "content": "{\"risk\":true,\"reason\":\"承诺了到账时间\"}"}
  ]
}

这条样本教模型的是任务格式和判断口径。它不适合塞进一整本业务手册，也不适合让模型记住每天变化的政策。如果政策经常变，应该把政策放进检索上下文，再用微调或 prompt 约束输出格式。

推理：参数固定后，按输入生成或打分

INFER

推理阶段能控制什么

推理时，模型参数通常固定。你能控制的是输入内容、系统提示、上下文材料、工具返回、解码参数和输出约束。很多应用优化其实发生在这里：减少无关上下文、补充证据、改写任务说明、设置 JSON schema、调整 temperature、记录失败样例。

temperature、top_p、max_tokens 这类参数影响生成行为。temperature 高，输出更发散；低，输出更保守。max_tokens 限制输出长度。它们不改变模型会不会某个领域知识，只改变本次生成的采样方式和边界。

推理问题的错误诊断要从输入开始。模型没有引用证据，可能是证据没放进窗口；输出格式乱，可能是约束不够明确或没有示例；答案不稳定，可能需要降低采样随机性或拆分任务。

response = client.chat.completions.create(
    model="some-llm",
    messages=[
        {"role": "system", "content": "只根据给定材料回答。"},

)

这段代码没有训练模型。它只是把消息传入已部署模型，并指定生成参数。修改 system prompt、上下文和 temperature 会影响本次输出；但模型参数不会因此永久改变。

embedding、reranking、generation：应用里的三种常见角色

ROLES

它们的输入输出不同

embedding 模型把文本变成向量，常用于相似度检索、聚类和去重。它的输出通常不是自然语言，而是一串浮点数。你拿这个向量去向量库里找相似文档。

reranking 模型通常接收"查询 + 候选文档"，输出相关性分数。它不负责从全库召回，而是对已经召回的候选重新排序。它比向量召回更细，但也更慢，所以常放在召回之后。

generation 模型负责生成文本、代码或结构化结果。它可以基于 prompt 和检索材料回答问题，但如果没有给它证据，它不一定能可靠回答最新或私有知识。一个 RAG 系统经常把这三者串起来：embedding 召回，reranking 精排，generation 作答。

角色对照表

角色输入输出常见用途
embedding一段文本向量召回相似文档、聚类、去重
reranking查询 + 候选文本相关性分数或排序把召回结果重新排前后
generationprompt + 上下文自然语言、代码、JSON回答、总结、改写、执行指令

判断句：如果你要"找资料"，先想 embedding / rerank；如果你要"组织答案"，才轮到 generation。

流程示意：RAG 里三种角色怎样协作python
query_vec = embedding_model.encode(user_question)
candidates = vector_store.search(query_vec, top_k=20)

ranked = reranker.rank(user_question, candidates)
context = build_context(ranked[:5])

answer = generation_model.generate(
    prompt=make_prompt(user_question, context)
)

第一步是召回，目标是别漏掉可能相关的材料。第二步是精排，目标是把更相关的材料放前面。第三步才是生成，目标是基于材料回答。把三步混在一起，会导致诊断困难：答案差时，你不知道是没召回、排序差，还是生成提示写得不稳。

最小工程决策：问题来了先改哪里

DECIDE

先低成本定位，再高成本改造

当大模型应用效果不好时，不要第一反应就是微调。先看失败样例属于哪类：是不知道材料、没遵守格式、推理链太长、检索片段不相关、候选排序差，还是生成随机性太高。问题类型不同，动作也不同。

如果模型缺少私有资料，先加检索和上下文。如果模型能找到资料但引用错，检查 chunk 切分、召回、重排和引用约束。如果输出格式不稳，先用 schema、少量示例和低 temperature。如果同一类任务长期大量出现，并且有高质量样本和评估集，再考虑微调。

评估是边界感的保护栏。没有评估集，prompt 优化、RAG 调参和微调都容易变成凭感觉。最小评估可以从 20-50 条真实样例开始，记录输入、期望行为、实际输出和错误类型。

误区、边界和易混概念

DIAGNOSIS

五个常见误区

  1. 把 RAG 当成微调。错因是都能改善回答；后果是维护方式选错；修法是区分"放上下文"和"改参数"。
  2. 以为微调能补所有知识。错因是把参数当数据库；后果是事实更新困难；修法是频繁变化事实走检索。
  3. 把 embedding 当生成模型。错因是看到"模型"就以为都能回答；后果是拿向量模型问答；修法是看输出是向量还是文本。
  4. 只调 temperature 解决事实错误。错因是把随机性当知识来源；后果是更稳定地错；修法是补证据或换模型。
  5. 没有评估就微调。错因是只看少量样例；后果是局部变好、整体退化；修法是先建立覆盖真实失败类型的评估集。

三组易混概念

概念准确区别判断方法
pretraining / finetuning预训练学通用基础参数；微调用较小数据适配任务或行为。看数据规模、目标和是否从基础模型继续训练。
finetuning / prompting微调改参数；prompting 改本次输入。调用结束后模型是否永久改变。
RAG / long contextRAG 是检索并注入相关材料；长上下文只是窗口更大。看有没有召回、筛选和排序。
