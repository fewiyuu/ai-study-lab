一句话抓边界：embedding 负责“先找一批可能相关的材料”，reranking 负责“在这批里重新排精细顺序”，generation 负责“基于材料组织回答”。
Retrieval Middle Layer · Roles And Boundaries

把检索中段拆成三种清楚的工程动作

很多 RAG 或知识库问答问题，不是模型不会回答，而是中间链路把职责混在了一起：用生成模型去猜资料，用 embedding 直接当最终答案排序，用 reranker 处理过大的候选集。结果是速度慢、成本高、回答还不稳。

这页先不搭完整 RAG。我们只学中间三块：embedding、reranking、generation。你要学会看输入输出、适合放在哪一步、错用后会出现什么症状。

用户问题→embedding 召回→reranking 精排→generation 回答

这一页结束时，你要能做到

  1. 用输入输出区分 embedding、reranking、generation。
  2. 解释为什么“召回高”和“排序准”不是同一件事。
  3. 诊断回答错误到底更像检索问题、排序问题，还是生成问题。

概念骨架

MAP
判断句：embedding 和 reranking 都在“找材料”，generation 在“写回答”。三者可以协作，但不能互相替代。

术语表

TERMS
术语直觉解释准确说法容易误会
Embedding把文本压成向量，方便用距离找相似内容。模型输出定长向量，常用余弦相似度或内积做近邻搜索。相似不等于能回答。语义接近的段落可能没有关键事实。
Recall / 召回先把可能相关的材料捞出来。从大量文档中取 TopK 候选，目标是别漏掉答案证据。召回不是最终排序；Top1 不一定最适合直接给生成模型。
Reranking对候选材料重新精排。常用 cross-encoder 或重排模型，对 query-document pair 输出相关性分数。reranker 适合少量候选，不适合直接扫全库。
Generation把问题和上下文写成回答。语言模型根据 prompt、检索上下文和生成参数逐 token 输出文本。生成模型擅长表达，不应该让它凭空补资料。
Context生成模型当前能读到的材料。包括用户问题、系统指令、检索片段、历史对话等 token。把更多上下文塞进去不一定更好，噪声会干扰回答。
TopK取分数最高的 K 条候选。embedding 召回和 reranking 输出都可能有 TopK，但含义不同。K 越大越好是错的。候选太多会拉高成本和噪声。

Embedding：快速召回候选

ROLE 1

embedding 的最小动作是：把 query 和文档片段都映射成向量，用相似度找最近的候选。它解决的是规模问题。你不可能把十万段文档都塞给生成模型，也不该让 reranker 对全库逐对打分，所以先用向量检索缩小范围。

embedding 的边界也很清楚：它只能用向量距离表达“像不像”，不能保证候选段落真的包含答案。比如用户问“报销审批需要几级确认”，一个段落讲“审批流程概览”可能向量很近，但真正写了“三级确认”的段落因为措辞不同被排在后面。这就是为什么召回要重视覆盖，而不是只看第一条。

渐进例子 1：用向量相似度做最小召回Python
import numpy as np

query_vec = np.array([0.8, 0.2, 0.1])
doc_vecs = np.array([
    [0.7, 0.3, 0.1],  # 候选 A
    [0.1, 0.9, 0.2],  # 候选 B
    [0.6, 0.2, 0.4],  # 候选 C
])

def cosine(a, b):
    return (a @ b) / (np.linalg.norm(a) * np.linalg.norm(b))

scores = [cosine(query_vec, doc) for doc in doc_vecs]
top_ids = np.argsort(scores)[::-1][:2]
print(scores, top_ids)
读这段代码时，你要盯住三件事：query 向量、文档向量、相似度排序。它没有读完整句子细节，只在向量空间里做近邻搜索。

Reranking：对候选做精细判断

ROLE 2

reranking 接在召回之后。它看到的是 query 和候选文档的成对输入，因此能比 embedding 更细地判断“这个段落是否直接回答了问题”。常见重排模型会把问题和候选一起编码，然后输出一个相关性分数。它更准，但也更贵，因为每个候选都要和 query 组成一对跑模型。

工程上常见做法是：embedding 从大库召回 50 条，reranker 把这 50 条重排，最后取前 3-8 条给生成模型。这个数量不是固定答案，要看文档长度、上下文窗口、延迟和答案需要多少证据。reranking 的判断句是：当候选已经不多，但顺序不可靠时，用它；当全库还很大时，不要让它直接硬扫。

渐进例子 2：候选重排的最小结构Python
query = "员工报销超过 5000 元需要几级审批？"
candidates = [
    "报销申请需要上传发票，财务会检查票据是否完整。",
    "超过 5000 元的报销需要直属主管和部门负责人两级审批。",
    "差旅报销应在出差结束后 7 个工作日内提交。",
]

# 真实 reranker 会把 (query, candidate) 成对输入模型。
rerank_scores = [0.42, 0.91, 0.35]
ranked = sorted(zip(candidates, rerank_scores), key=lambda x: x[1], reverse=True)

for text, score in ranked:
    print(round(score, 2), text)

观察：召回分和重排分可能冲突

候选 A：审批流程概览
召回 0.86
重排 0.48
候选 B：5000 元两级审批
召回 0.74
重排 0.94

召回分看“像不像”，重排分更看“是否回答当前问题”。冲突时不要直接说谁错，要回到任务目标：是否需要精确证据。

Generation：基于材料组织回答

ROLE 3

generation 是语言模型最显眼的一步，但它不应该承担找资料的责任。它的输入应该包括用户问题、经过筛选的上下文和回答要求。它的输出是自然语言答案，可能带引用、步骤、表格或 JSON。生成模型擅长把材料整合成可读回答，也擅长补顺语气；它不擅长在没有材料时保证事实正确。

如果检索材料里没有答案，生成模型仍可能写得很顺。这不是它坏，而是职责放错了。工程上要给它边界：只根据上下文回答；缺证据时说明缺少依据；引用具体片段；不要把猜测写成事实。生成质量既受模型能力影响，也受上下文质量影响。检索差时，换更大的生成模型常常只是把错误说得更像真的。

渐进例子 3：把重排后的片段放入回答上下文Python
question = "员工报销超过 5000 元需要几级审批？"
contexts = [
    "超过 5000 元的报销需要直属主管和部门负责人两级审批。",
    "报销申请需要上传发票，财务会检查票据是否完整。",
]

prompt = f"""请只根据给定资料回答问题。
如果资料不足，请直接说缺少依据。

资料：
1. {contexts[0]}
2. {contexts[1]}

问题：{question}
回答："""

# 这里才把 prompt 交给生成模型。
print(prompt)
判断句：generation 的输入质量决定了事实边界。没有证据时，好的回答应该承认缺口，而不是补一个看似合理的答案。

完整链路：从用户问题到回答

CHAIN

最小工程动作

  1. 把文档切成片段，提前计算文档 embedding，存入向量库或本地索引。
  2. 用户提问时，计算 query embedding，取相似度最高的一批候选。
  3. 把 query 和候选成对送给 reranker，按相关性重排。
  4. 把少量高质量片段放进 prompt，要求生成模型基于上下文回答。
  5. 记录候选、分数、最终上下文和回答，便于排查错误。

错误归因顺序

如果回答没引用到关键事实，先看召回候选里有没有答案。候选里没有，问题在 chunk、embedding 或 query 改写。候选里有但排很低，问题更像 reranking。候选排前面但回答没用，问题在 prompt、上下文拼接或 generation。

这个顺序很重要。直接改 prompt 当然容易，但如果检索层没有把证据交上来，prompt 写得再漂亮也只是让模型更会兜圈。

误区诊断

PITFALLS

误区 1：embedding 相似就是答案正确

错因：把语义接近当成事实命中。后果：召回了话题相近但缺关键数字、条件或例外的片段。修法：保留较大的召回 TopK，再用 reranker 或规则检查直接证据。

误区 2：reranker 可以直接扫全库

错因：只看精排效果，忽略每个 query-doc pair 都要模型计算。后果：延迟和成本爆炸。修法：先用 embedding 做粗召回，再对几十条候选重排。

误区 3：生成模型可以替代检索

错因：生成模型回答流畅，容易让人以为它知道内部资料。后果：模型编造政策、日期或数字。修法：把资料作为上下文输入，要求缺证据时说明不足。

误区 4：TopK 越大越好

错因：以为多给上下文一定更稳。后果：噪声片段挤占上下文窗口，生成模型可能抓错重点。修法：根据片段长度、问题类型和窗口大小调 K，并观察命中率和回答质量。

误区 5：只看最终回答，不记录中间结果

错因：把 RAG 当黑盒。后果：错了不知道该改切分、embedding、reranker 还是 prompt。修法：保存 query、召回列表、重排分数、最终上下文和回答。

误区 6：中英文、代码、表格都用同一种检索策略

错因：忽略数据形态。后果：短代码、表格字段、专有名词可能向量效果差。修法：对结构化信息加关键词检索、字段过滤或混合检索。

三组易混概念

COMPARE
易混概念前者后者判断方法
召回 / 排序目标是别漏掉可能相关的候选。目标是在候选里把最有用的排前面。关键证据没进候选，是召回问题；进了但靠后，是排序问题。
Embedding / Reranking单独编码 query 和文档，用向量距离比较，快。query 和文档成对输入模型，判断更细，慢。全库先 embedding，少量候选再 reranking。
Retrieval / Generation负责找到可用证据。负责把证据组织成回答。没有证据时，generation 不该编事实。
相似度分 / 可信度表示向量或模型认为相关。表示答案是否有事实依据。可信度还要看来源、片段内容、引用和一致性。
