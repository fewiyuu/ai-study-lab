Prompt Engineering · Structured I/O

Prompt 先别追玄学，先写成接口

LLM 应用里最常见的第一层不稳定，往往不是模型太弱，而是调用协议太松。需求、上下文、约束和输出格式混在一段话里，模型看似答了，程序却很难判断它有没有按要求做。结构化 prompt 的目标，是把“请帮我处理一下”改成“这里有字段、规则、示例和输出 schema，请按这个接口返回”。

任务→ 上下文→ 约束→ 示例→ 输出 schema→ 验证

学完后要能做三件事

  1. 把一个自然语言需求拆成任务、输入数据、规则、格式和边界。
  2. 写出能被程序解析的 JSON 输出要求，并知道怎么处理解析失败。
  3. 看到一次坏输出时，能判断是任务不清、上下文缺失、格式不稳，还是验证逻辑缺失。

如果以后要做 RAG、Agent 或 Eval，这一页会反复用到。因为后面的系统都要靠稳定输入输出把步骤串起来。

一、把 prompt 看成接口契约

CONTRACT

概念：prompt 不只是给模型看的话

在聊天里，prompt 可以是一段自然语言。在工程里，prompt 更像接口契约：调用方提供哪些字段，模型应该基于哪些字段工作，不能凭空补哪些内容，输出必须长什么样。如果没有这层契约，后面的解析、测试、重试和日志都很难做。你会得到一堆看起来流畅的文本，却不知道它能不能进入下一步程序。

接口契约至少包含四个部分。第一是任务，说明模型要完成什么动作，例如分类、抽取、改写、生成 SQL、总结会议。第二是输入，说明哪些信息是事实来源，哪些只是用户偏好。第三是约束，说明禁止事项、取舍规则和缺失信息时的处理。第四是输出，说明字段名、类型、枚举值、是否允许空值和错误返回方式。

边界也要写清楚。prompt 不能替代权限控制、数据校验、业务规则和安全策略。模型可以帮你理解文本，但不能被允许自己决定是否扣款、是否发邮件、是否删除数据。判断句是：只要一个结果会改变外部世界，就不能只靠 prompt 保证正确，必须有程序侧验证或人工确认。

读到这里，应该能分清两件事：prompt 负责表达任务和约束，程序负责验证结果和执行动作。

二、结构化输入：把材料放进正确的格子

INPUT

任务字段：动作要单一

任务字段回答“模型现在要做什么”。一个 prompt 里最好只放一个主要动作：抽取、判断、改写、生成、排序或总结。多个动作可以串成流程，但不要塞进一个含混指令里。比如“请读这段反馈，判断情绪，再总结原因，还要写回复话术”，看起来省事，实际会让错误难定位。更稳的做法是先做情绪分类，再抽取原因，最后生成回复。

判断句是：如果输出坏了，你能不能指出哪个动作坏了。不能指出，就说明任务拆得太粗。

上下文字段：事实和偏好分开

上下文是模型工作的材料。事实材料要和风格偏好分开。事实材料包括用户原文、产品说明、历史记录、检索片段。偏好包括语气、长度、受众、禁用词。如果事实和偏好混成一段，模型可能把偏好当事实，也可能把事实当建议处理。

常用做法是给每一块加清楚标题，并用分隔符包住原文。分隔符不是魔法，它的价值是降低边界歧义。看到长文、日志、用户输入时，应优先分块，而不是把所有东西接成一句话。

约束字段：写清缺失时怎么办

约束不只是“不要胡说”。更有用的是写清选择规则和缺失信息处理。例如：没有证据时返回 insufficient_evidence，不得猜测；多个类别都匹配时选择风险更高的一类；原文没有金额时 amount 设为 null。这些规则会让输出更适合程序处理。

约束也不能无限堆。约束之间冲突时，模型会自己找折中。工程里要把硬规则放到程序校验里，把软偏好留给 prompt。

输入模板：把事实、规则和格式拆开prompt
任务：
从用户反馈中抽取产品问题，并判断是否需要人工跟进。

输入：
<feedback>
{user_feedback}
</feedback>

规则：
1. 只使用 feedback 中出现的信息，不补充未出现的原因。
2. 如果用户要求退款、投诉、威胁曝光，needs_human_review 设为 true。
3. 如果无法判断产品模块，module 设为 null。

输出：
只返回 JSON，不要附加解释。

这段模板没有追求“写得像人”。它追求边界清楚。任务是抽取和判断，输入是反馈原文，规则说明缺失和高风险情况，输出说明只要 JSON。后面你可以把 {user_feedback} 换成真实内容，模板其他部分保持稳定。这样才能做批量测试和回归检查。

三、结构化输出：让结果能被程序接住

OUTPUT

输出 schema 要同时服务人和程序

结构化输出最常见的是 JSON。它的好处不是“看起来高级”，而是字段名、类型和嵌套关系固定，程序可以解析、校验和入库。好的 schema 不只是列字段，还会写字段含义、枚举范围、缺失值规则和错误字段。比如 priority 是 low、medium、high 之一，而不是任意中文词。

不要让模型输出“近似 JSON”。近似 JSON 包括中文逗号、尾随注释、前后解释、多余 Markdown 代码围栏、字段名不稳定。只要下游要自动解析，就要把格式要求写得具体，并在程序里使用严格解析。解析失败不是小事，它说明这一轮调用没有进入可靠状态。

边界是：schema 能约束形状，不保证内容一定正确。模型可以返回合法 JSON，但分类错了、证据摘错了、理由不充分。格式验证后还需要业务验证或 eval。

一个更稳的 JSON 约定

字段类型说明
category字符串枚举bug、billing、feature_request、other
module字符串或 null原文明确提到的产品模块，没有就返回 null
needs_human_review布尔值退款、投诉、合规风险、强烈负面情绪时为 true
evidence字符串数组从原文摘取能支持判断的短句，不要改写成新事实
error字符串或 null信息不足、输入为空、无法解析时说明原因

字段越靠近业务动作，越需要枚举和校验。自由文本字段适合放解释，不适合直接驱动程序动作。

四、三个渐进例子

EXAMPLES

例子 1：从松散要求到字段化要求

先看一个常见坏写法：请总结这段用户反馈，告诉我问题严不严重。它的问题是“总结”可以有很多长度和角度，“严重”没有标准。模型可能写出一段顺滑总结，但下游系统无法知道它是不是需要转人工。

错误示例：任务和判断标准都含混prompt
帮我看一下这个用户反馈，总结问题，判断严重不严重：
{feedback}

改法是把结果拆成字段，把判断标准写成规则。这样坏输出更容易定位：如果 priority 错了，就查严重度规则；如果 evidence 为空，就查抽取规则；如果 JSON 解析失败，就查输出约束。

正确示例：字段化抽取与判断prompt
请分析 feedback，返回一个 JSON 对象：
{
  "summary": "20 字以内的问题摘要",
  "priority": "low | medium | high",
  "needs_human_review": true,
  "evidence": ["原文中的证据短句"]
}

判断 high 的条件：
- 涉及退款、投诉、合规、安全、数据丢失；
- 或者用户明确表示会流失、曝光、升级投诉。

例子 2：few-shot 不只是给答案样子

few-shot 是在 prompt 里放少量示例。它的价值不是让模型背答案，而是让模型看到边界：什么算高风险，什么只算普通体验问题，缺失信息怎么返回。示例要覆盖容易错的边界样本，而不是只放最简单的正例。

示例：用边界样本教枚举口径prompt
示例 A
输入："充值后余额没到账，客服再不回我就投诉。"
输出：{"category":"billing","priority":"high","needs_human_review":true}

示例 B
输入："希望深色模式可以跟随系统。"
输出：{"category":"feature_request","priority":"low","needs_human_review":false}

示例 C
输入："登录按钮点了没反应，但刷新后好了。"
输出：{"category":"bug","priority":"medium","needs_human_review":false}

注意示例 C。它不是最高风险，但也不应被归为普通建议。边界示例能减少“所有负面反馈都 high”这种粗糙行为。

例子 3：程序侧解析和验证

prompt 说“只返回 JSON”仍然不够。真实系统要假设模型可能返回非法 JSON、字段缺失、枚举越界或证据为空。最小工程动作是解析、校验、失败重试，仍失败就降级处理。下面的代码不是完整框架，只展示关键顺序。

最小代码：解析、校验、失败分流python
import json

ALLOWED_CATEGORY = {"bug", "billing", "feature_request", "other"}

def parse_feedback_result(raw_text):
    try:
        data = json.loads(raw_text)
    except json.JSONDecodeError:
        return {"ok": False, "reason": "json_parse_failed"}

    required = ["category", "priority", "needs_human_review", "evidence"]
    missing = [key for key in required if key not in data]
    if missing:
        return {"ok": False, "reason": "missing_fields", "fields": missing}

    if data["category"] not in ALLOWED_CATEGORY:
        return {"ok": False, "reason": "bad_category"}

    return {"ok": True, "data": data}

关键行的语义很具体：json.loads 只负责格式解析，不负责判断内容正确；required 防止字段缺失进入业务；枚举检查防止模型临时造词。删掉这些校验，prompt 就会变成“希望模型听话”，而不是工程接口。

五、最小工程动作：模板、验证、日志

WORKFLOW

1. 模板化

把稳定规则写在模板里，把每次变化的数据作为变量填入。不要每次手写一段新 prompt。模板化以后，你才知道系统今天和昨天的差异来自输入、模型版本，还是模板变更。模板文件最好带版本号，重要字段变更要能追踪。

2. 解析和校验

解析检查格式，校验检查字段和值。二者不能混为一谈。合法 JSON 可能内容不合格，非法 JSON 也可能含有正确答案但不能进程序。工程上先保证能解析，再判断是否符合 schema，再进入业务逻辑。

3. 重试和降级

重试不是简单“再问一次”。应该带上失败原因，例如“上次输出不是合法 JSON，请只返回 JSON”。如果连续失败，要降级到人工处理、保守默认值或停止本次流程。不能让无限重试消耗成本，也不能让坏数据悄悄进入系统。

动作最小做法常见坏味道
模板管理固定模板、变量填充、记录版本每次临时拼接 prompt，无法复现错误
输入清洗限制长度、保留原文、标明来源把用户输入和系统规则混在一起
输出解析严格 JSON 解析，失败走错误分支用字符串截取凑合拿字段
业务校验检查枚举、范围、必填字段、证据只要能解析就默认可信
日志记录输入摘要、模板版本、模型、解析结果出错后只能靠记忆猜发生了什么

六、常见误区和错误诊断

PITFALLS

误区 1：把 prompt 写长就会稳定

错因：长度不等于结构。长 prompt 如果任务、上下文和约束混在一起，模型仍然会抓错重点。后果是输出风格变长，解析更难。修法是分块、编号、给字段和 schema。

误区 2：要求“不要胡编”就够了

错因：“不要胡编”没有说明缺证据时返回什么。模型可能换一种委婉说法继续猜。后果是下游把猜测当事实。修法是设计 null、insufficient_evidence 或 error 字段。

误区 3：JSON 输出等于结果可靠

错因：JSON 只保证形状，不保证分类、抽取和推理正确。后果是系统看起来稳定，实际在稳定地产生错内容。修法是增加字段校验、证据检查和样本评估。

误区 4：示例越多越好

错因：示例会占上下文，也可能引入不相关模式。后果是模型模仿示例表面格式，忽略当前输入。修法是少量选择边界样本，并让示例覆盖容易混淆的情况。

误区 5：解析失败时直接用原文本兜底

错因：原文本不可控，可能包含解释、道歉、代码围栏或缺字段。后果是坏数据进入下一步，错误变得隐蔽。修法是失败分支明确：重试、降级、人工处理或停止。

误区 6：让模型自己执行高风险动作

错因：prompt 不能承担权限系统的责任。后果可能是误发消息、误删数据、越权操作。修法是模型只产出建议或结构化意图，外部动作由程序规则和确认流程控制。
