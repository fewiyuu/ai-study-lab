Evaluation · LLM Application

Eval 是把感觉变成可复查的证据

LLM 应用的改动很容易“肉眼看起来更好了”。换模型、改 prompt、加检索、调温度，都可能让几个样例变顺，却让另一批样例变差。Eval 的作用，是用固定样本、判分规则和错误分析把这种感觉压成证据。它不保证线上永远没问题，但能让每次改动先回答三个问题：哪里更好，哪里更坏，坏的问题能不能接受。

真实任务→ 样本集→ 输出→ 判分→ 错误分析→ 改动决策

学完后要能做三件事

  1. 为一个 LLM 功能设计 20-50 条最小评测样本，并覆盖常见失败类型。
  2. 区分确定性判分、规则判分、LLM-as-judge 和人工复核的边界。
  3. 看到分数变化时，能追到具体样本，而不是只盯平均分。

Eval 是后面 RAG 和 Agent 的安全垫。没有它，系统越复杂，越难知道哪次改动真的变好了。

一、为什么 LLM 应用特别需要 Eval

WHY

概念：Eval 是可重复的任务检查

Eval 是 evaluation 的缩写，可以先理解成“用固定题目检查系统表现”。这里的题目不是考试题，而是从真实业务里抽出来的输入样本。每条样本包含输入、期望行为、判分规则，有时还会包含参考答案。系统每次改动后重新跑这些样本，就能看到能力是否变化。

LLM 应用比传统规则程序更需要 Eval，因为输出不是固定函数。模型版本、prompt 表述、采样温度、检索结果顺序都会影响回答。你不能只用一个 happy path 样例判断系统可以上线。一个客服分类器在普通问题上表现很好，也可能在退款、辱骂、合规风险、含糊输入上突然失稳。

边界也要记住：Eval 不是线上监控，不是安全审计，也不是一次性验收。Eval 覆盖的是你已经想到并收集到的样本。它能发现一批已知风险，不能覆盖未来所有输入。判断句是：当你准备改 prompt 或换模型时，如果没有一组固定样本对比前后表现，你其实是在凭感觉改系统。

到这里，应该能说清：Eval 解决的是“改动前后怎么比较”和“错误怎么复现”。

评测链路

二、样本集：先覆盖风险，再追求数量

DATASET

样本从真实任务来

最小 Eval 不需要一开始就有几千条样本。更现实的做法，是先收集 20-50 条代表性输入：常见请求、边界请求、高风险请求、历史坏例、用户容易误写的请求。每条样本要写清楚期望行为。只保存输入，不写期望，后面就会变成“看起来还行”的人工浏览。

样本要分标签。比如客服分类可以标成普通咨询、账单、投诉、辱骂、退款、信息不足。RAG 问答可以标成可回答、资料缺失、资料冲突、需要引用、容易幻觉。标签让你知道分数变化发生在哪类问题上。

参考答案不一定只有一种

分类、抽取、结构化输出通常可以有明确答案。总结、开放问答、改写、建议生成就很难只有一个标准答案。这时不要强行追 exact match，而要写判分规则：是否覆盖关键点，是否引用证据，是否避免未给出的事实，语气是否合适，是否拒绝不该回答的问题。

判断句是：如果两个人看同一条输出会给完全不同的分，说明你的 rubric 还太模糊。Rubric 是判分量表，写的是“什么情况下给满分、扣分、判失败”。

保留坏例比保留漂亮样例更有用

Eval 不是展示系统能力的样例册。它应该包含会让系统犯错的样本。每发现一次线上坏例，就把它整理进回归样本集。回归的意思是：以后每次改动都要检查旧问题有没有重新出现。一个系统如果只在轻松样本上跑分，分数会很好看，但上线后仍然会在同样的难点上跌倒。

边界是样本集会过时。产品变化、用户变化、资料库变化后，样本也要更新。不要把旧 Eval 当永久真理。

样本类型例子它检查什么
常见样本“我想改绑定邮箱。”主流程是否稳定
边界样本“余额少了，但我也不确定是不是自己花了。”信息不足时是否保守
高风险样本“再不退款我就投诉到监管。”升级、合规、人工处理
历史坏例过去被错误自动回复的真实问题旧问题是否复发

三、指标与判分：不同任务用不同尺子

SCORING

确定性任务先用硬指标

如果任务有明确答案，优先使用硬指标。分类任务可以算 accuracy、precision、recall；抽取任务可以检查字段是否相等、集合是否覆盖；JSON 任务可以检查解析率、字段缺失率、枚举错误率。硬指标的好处是可重复、便宜、争议少。它们也有盲区：用户可接受的表达变化，硬指标可能会误判。

LLM 生成任务常用 rubric 或 LLM-as-judge。LLM-as-judge 是让另一个模型按规则给分。它适合初步筛查开放文本，但不能盲信。判分 prompt 也会有偏差，可能偏爱长答案、礼貌答案或某种写法。重要任务要抽样人工复核，让自动判分和人工判断对齐。

不要只看平均分。平均分会掩盖严重错误。比如 100 条问答里 95 条很好，5 条会编造法律建议，平均分可能仍然漂亮。上线决策要看分布、失败类型和高风险样本表现。

四、三个渐进例子

EXAMPLES

例子 1：客服分类的最小 Eval

客服分类有明确类别，适合从硬指标开始。每条样本包含输入和期望类别。跑系统后比较预测类别和期望类别，再按类别统计错在哪里。如果 refund 类 recall 很低，说明该转人工的请求被漏掉；如果普通咨询经常被判 high，说明系统过于保守。

样本示例：分类 Eval 数据jsonl
{"id":"c001","input":"充值后余额没到账，我要退款","expected":"billing_refund","risk":"high"}
{"id":"c002","input":"怎么修改头像？","expected":"how_to","risk":"low"}
{"id":"c003","input":"你们再不处理我就投诉","expected":"complaint","risk":"high"}

这类 Eval 的关键不是只算总准确率，而是按风险标签看漏判。高风险样本少，也要单独看。

例子 2：RAG 问答要同时看答案和依据

RAG 的输出看起来像问答，但评测不能只问“答案像不像”。还要看检索材料是否支持答案、引用是否正确、资料缺失时是否拒答。一个答案语气很好，但没有被上下文支持，就是幻觉风险。相反，一个答案不够华丽，但引用准确、边界清楚，可能更适合上线。

判分规则：RAG 答案 rubrictext
给每条回答打 0-2 分：
2 分：答案直接回应问题，所有关键事实都能在 context 中找到，引用片段正确。
1 分：大体回答正确，但遗漏一个关键限制，或引用不完整。
0 分：答案包含 context 不支持的事实，或资料不足时仍然猜测。

这段 rubric 的重点是“被材料支持”。RAG 系统最怕回答得像真的，但证据链断了。

例子 3：一个最小判分脚本

下面的代码展示最小分类评测。它不调用模型，只说明评测结果应该怎样汇总。真实系统里 predict 会替换成模型调用或你的应用接口。关键是每条样本都保留 id、输入、期望、预测和是否通过。只输出一个总分，会让错误难追。

最小代码：分类结果评测python
def evaluate(samples, predict):
    rows = []
    for sample in samples:
        pred = predict(sample["input"])
        passed = pred == sample["expected"]
        rows.append({
            "id": sample["id"],
            "expected": sample["expected"],
            "pred": pred,
            "passed": passed,
            "risk": sample.get("risk", "normal")
        })

    total = len(rows)
    passed = sum(row["passed"] for row in rows)
    failures = [row for row in rows if not row["passed"]]
    return {
        "accuracy": passed / total if total else 0,
        "failures": failures
    }

rows 是诊断材料，不只是中间变量。它让你知道哪条样本错了。failures 保留坏例，后续才能分类查看错因。删掉它，只剩一个 accuracy，系统出了问题也很难修。

五、最小 Eval 流程

WORKFLOW

1. 写任务边界

先写清系统要解决什么，不解决什么。比如“根据内部知识库回答产品使用问题”，就不包括法律意见、医疗建议、账号敏感操作。边界决定样本范围，也决定哪些失败必须拦截。

2. 建小样本集

从真实输入、历史坏例和你担心的边界构建样本。每条样本加标签和期望行为。第一版不用大，但要有代表性。样本少时，更要覆盖风险，而不是全放简单题。

3. 定判分规则

能硬判就硬判，不能硬判就写 rubric。Rubric 要能让不同人给出接近判断。高风险任务要抽样人工复核，不要完全交给自动分数。

4. 固定基线

第一次跑出来的结果就是基线。之后改 prompt、模型、检索参数，都和基线比较。不要只看新版本绝对分，要看相对变化和失败样本迁移。

5. 做错误分析

把失败样本按错因归类：检索不到、格式错误、理解错意图、证据不支持、拒答过度、拒答不足。错因比单个分数更能指导下一次改动。

6. 接入改动流程

Eval 只有在改动前后都跑，才真的有用。上线前至少跑固定样本，线上再监控新失败。新坏例进入样本集，下一次改动继续检查。

六、常见误区和错误诊断

PITFALLS

误区 1：只看几个好看的 demo

错因：demo 常常是顺手挑的好例子。后果是系统在边界样本上坏得很隐蔽。修法是固定样本集，保留普通、边界和高风险输入。

误区 2：把平均分当唯一结论

错因：平均分会掩盖少量严重错误。后果是高风险样本失败被稀释。修法是按标签看分数，单独查看失败列表。

误区 3：让判分模型无规则自由打分

错因：LLM-as-judge 也需要 prompt 和 rubric。后果是分数偏爱长答案或漂亮话。修法是写明确扣分点，并抽样人工对齐。

误区 4：样本集只收集常见问题

错因：常见问题不等于重要问题。后果是低频高损害错误被漏掉。修法是加入投诉、合规、拒答、资料冲突等风险样本。

误区 5：一次通过就不再跑

错因：模型、prompt、检索库和用户输入都会变。后果是旧问题复发时没人发现。修法是把 Eval 作为回归检查，每次重要改动都跑。

误区 6：离线 Eval 通过就等于线上稳定

错因：离线样本覆盖有限，线上有新分布和真实用户行为。后果是上线后出现未覆盖失败。修法是离线 Eval 配合线上监控、反馈收集和坏例回流。
