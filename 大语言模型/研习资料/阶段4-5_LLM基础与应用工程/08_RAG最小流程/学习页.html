RAG 不是“把资料塞给大模型”。它是一条可检查链路：资料如何切，证据如何找，哪些片段进入上下文，回答有没有引用依据。
Retrieval-Augmented Generation · Minimum Flow

搭出能排错的 RAG 最小流程

RAG 的目标不是让模型显得更会说话，而是让回答受外部资料约束。最小流程包括离线索引和在线问答两段：离线把文档切块、向量化、存起来；在线把问题向量化、检索、重排、拼上下文，再让生成模型回答。

学这页时不要急着接复杂框架。先把每一步输入输出写清楚，并记录中间结果。只要中间结果可查，RAG 错了就能定位；如果只看最终回答，排错会变成猜。

文档→切分→索引→检索→上下文→回答

这一页结束时，你要能做到

  1. 说清 RAG 离线和在线两段分别做什么。
  2. 写出一个内存版最小 RAG 的伪代码或 Python 结构。
  3. 遇到错误回答时，按候选、重排、上下文、生成顺序排查。

概念骨架

MAP
离线段负责把资料变成可搜的索引，在线段负责把问题变成带依据的回答。两段混起来，排错会很痛苦。

术语表

TERMS
术语直觉解释准确说法常见误解
RAG先找资料，再让模型基于资料回答。Retrieval-Augmented Generation，用外部检索结果增强生成。不是把所有文档塞给模型，也不是微调模型。
Chunk文档切出来的可检索片段。带文本、来源、标题、页码等 metadata 的片段。切得越小越好是错的；太小会丢上下文，太大又难命中。
Index让片段能被快速搜索的数据结构。可以是向量库、倒排索引或混合索引。向量库不是答案库，它只负责按查询取候选。
Retrieval从索引里取候选证据。通常包括 query embedding、相似度搜索、过滤条件和 TopK。检索分高不等于最终答案可信。
Context Packing把候选片段整理进 prompt。选择、排序、去重、截断并保留引用信息。只要检索出来就全塞进去，会引入噪声。
Grounded Answer有依据的回答。答案能回指到上下文片段，缺资料时承认缺口。文字流畅不等于 grounded。

切分与索引：RAG 的离线段

OFFLINE

离线段的目标是把资料整理成可检索对象。原始文档通常太长、格式复杂、包含目录页和噪声。你要先抽取正文，再按合适粒度切分，给每个 chunk 加 metadata，比如来源文件、章节、页码、更新时间。metadata 不是装饰，后面做引用、过滤和排错都要靠它。

切分粒度没有万能值。制度文档适合按条款和小节切；教程适合按标题层级切；FAQ 可以一问一答成块。chunk 太小会丢上下文，例如只剩“需要两级审批”却不知道是什么场景；chunk 太大又会让 embedding 变得模糊，检索时难以命中具体事实。工程上常用少量重叠，保证边界附近的信息不被切断。

渐进例子 1：内存版文档切分Python
docs = [
    {
        "source": "reimburse.md",
        "title": "报销制度",
        "text": "普通报销由直属主管审批。超过 5000 元需要直属主管和部门负责人两级审批。票据不完整会退回。",
    }
]

def chunk_text(doc, size=36, overlap=8):
    text = doc["text"]
    chunks = []
    start = 0
    while start < len(text):
        end = min(len(text), start + size)
        chunks.append({
            "text": text[start:end],
            "source": doc["source"],
            "title": doc["title"],
            "span": (start, end),
        })
        if end == len(text):
            break
        start = max(end - overlap, start + 1)
    return chunks

chunks = [chunk for doc in docs for chunk in chunk_text(doc)]
print(chunks)
读这段代码时，关注 chunk 里除了文本还保存了来源信息。没有来源，后面生成回答就很难给依据，也很难排查错在哪份文档。

召回与重排：在线段的证据选择

RETRIEVE

在线段从用户问题开始。先把问题也转成 embedding，在索引里找相似 chunk。第一轮召回要重视覆盖：宁可多取一些候选，也不要把答案证据漏掉。接着再做重排、过滤或去重，把最可能回答问题的片段放到前面。

很多最小 demo 会省略 reranking，只做 embedding TopK。学习时可以这样起步，但要知道它的边界：如果候选里有正确片段却排得靠后，或者 query 和文档措辞差异大，单纯向量相似度会不稳。重排不是为了炫技，它是在候选数量已经缩小后，用更贵但更细的判断修正顺序。

渐进例子 2：用简化 embedding 做检索Python
import numpy as np

chunks = [
    {"id": 0, "text": "普通报销由直属主管审批。", "source": "reimburse.md"},
    {"id": 1, "text": "超过 5000 元需要直属主管和部门负责人两级审批。", "source": "reimburse.md"},
    {"id": 2, "text": "票据不完整会退回。", "source": "reimburse.md"},
]

def fake_embed(text):
    keywords = ["报销", "5000", "审批", "票据"]
    return np.array([text.count(k) for k in keywords], dtype=float)

def cosine(a, b):
    return (a @ b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-8)

query = "超过 5000 元报销要几级审批？"
qv = fake_embed(query)
scored = [(cosine(qv, fake_embed(c["text"])), c) for c in chunks]
top = sorted(scored, key=lambda x: x[0], reverse=True)[:2]
print(top)

这个 fake_embed 只是教学用，它用关键词计数模拟向量，不代表真实 embedding。它的价值是让你看见检索结构：query 变向量，chunk 也变向量，排序后得到候选。真实系统会替换成 embedding 模型和向量库，但输入输出关系不变。

上下文与生成：把证据交给模型

GENERATE

检索出的片段不能随便拼。你要去重、控制长度、保留来源，并把回答规则写进 prompt。最小规则包括：只根据资料回答；资料不足就说明不足；尽量引用来源编号。这样做不是为了限制模型表达，而是为了让答案受证据约束。

上下文拼接还有一个常见坑：把低质量候选放在高质量候选前面。生成模型对前后顺序、标题和编号都敏感。如果最相关片段排在末尾，或者上下文里夹了大量相似但无关的内容，模型可能抓错重点。所以 RAG 不是检索完就结束，context packing 本身也是工程动作。

渐进例子 3：拼接带引用的最小 promptPython
question = "超过 5000 元报销要几级审批？"
evidence = [
    {"text": "超过 5000 元需要直属主管和部门负责人两级审批。", "source": "reimburse.md"},
    {"text": "普通报销由直属主管审批。", "source": "reimburse.md"},
]

context = "\n".join(
    f"[{i+1}] {item['text']}（来源：{item['source']}）"
    for i, item in enumerate(evidence)
)

prompt = f"""请只根据资料回答问题。资料不足时说“资料不足”。
回答后列出使用的来源编号。

资料：
{context}

问题：{question}
回答："""

print(prompt)
错误示例：没有边界的 promptText
请回答：超过 5000 元报销要几级审批？

你可以根据常识补充。

这类 prompt 会把系统推回“凭模型记忆回答”。如果知识来自公司内部制度，常识不可靠，模型也不知道最新规则。RAG 的价值就在于把外部资料放进上下文，并要求答案回到资料。

排错与评估：先看证据，再看话术

DEBUG

错误回答的排查顺序

  1. 文档层：原始资料里是否真的有答案？版本是否最新？
  2. 切分层：答案是否被切断？chunk 是否缺标题或上下文？
  3. 召回层：TopK 候选里有没有正确证据？
  4. 重排层：正确证据是否被排到最终上下文外？
  5. 生成层：上下文已有证据时，模型是否没有按资料回答？

最小评估样例

不要只拿一个问题试系统。准备 10-30 个有标准依据的问题，覆盖数字、日期、条件、例外、无答案场景。每个问题保存标准答案和来源片段。评估时分开看：检索是否命中来源，最终上下文是否包含来源，回答是否忠于来源。

判断句：如果没有标准问题和标准来源，你只能感觉“好像变好了”。RAG 最小评估至少要能回答“证据有没有被找出来”。

误区诊断

PITFALLS

误区 1：RAG 等于向量库

错因：把索引组件当成完整系统。后果：只解决候选搜索，没有解决上下文选择、回答约束和评估。修法：把 RAG 拆成文档处理、索引、检索、重排、prompt、生成、评估。

误区 2：文档越大越直接塞

错因：看到长上下文模型就省略检索。后果：噪声多、成本高、引用难查。修法：仍然切分和筛选，把最相关证据放进上下文。

误区 3：chunk 固定长度就够

错因：只按字符数切，不看语义边界。后果：答案和条件被拆开，检索命中也无法回答完整问题。修法：优先按标题、条款、段落切，再用重叠处理边界。

误区 4：没有无答案处理

错因：所有问题都强行生成答案。后果：模型在资料缺失时编造。修法：prompt 明确资料不足时说明不足，并在评估集中加入无答案问题。

误区 5：只优化 prompt

错因：最终回答看得见，中间检索看不见。后果：证据没找到时仍在改话术。修法：先打印候选、分数和最终上下文，再决定改哪一层。

误区 6：没有来源 metadata

错因：只存 chunk 文本。后果：答案无法引用来源，资料更新也难定位。修法：每个 chunk 保存 source、标题、页码、更新时间或业务标签。

COMPARE
易混概念前者后者判断方法
RAG / 微调把外部资料检索出来放进上下文，适合频繁变化的知识。更新模型参数，适合学习风格、格式或稳定能力。知识常变、需要引用来源，优先 RAG；行为模式要长期改变，考虑微调。
Chunk / Context索引里的可检索片段。一次请求里真正放进 prompt 的材料集合。chunk 是候选单位，context 是生成模型当前能读到的证据。

向量检索 / 关键词检索擅长语义近似和同义表达。擅长精确词、编号、代码符号、专有名词。制度编号、API 名、错误码常需要混合检索。
