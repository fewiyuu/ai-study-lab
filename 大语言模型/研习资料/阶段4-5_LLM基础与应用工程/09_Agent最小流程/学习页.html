最小 Agent 不是“让模型随便干活”。它是目标、工具、观察、状态和停止条件组成的受控循环。
Agent · Minimal Loop

Agent 先从能停下来的循环开始

很多人一听 Agent 就想到复杂框架、多智能体和自动执行。工程上更稳的入口，是一个最小流程：给目标，模型选择下一步动作；程序执行允许的工具；工具返回观察结果；模型根据观察继续或停止。这个循环一旦没有状态、权限和停止条件，就会从“自动化”变成“不可控”。

目标→ 计划→ 工具调用→ 观察→ 状态更新→ 停止

学完后要能做三件事

  1. 说清 workflow、tool use 和 agent 的边界，不把所有模型调用都叫 Agent。
  2. 设计一个只含 2-3 个工具的最小 Agent 循环，并写出停止条件。
  3. 诊断 Agent 卡住、乱调用工具、越权执行或反复重试的原因。

本页只讲最小流程，不追框架。能把循环管住，再谈复杂协作。

一、Agent 是什么：模型参与决策的执行循环

CONCEPT

概念：Agent 不只是“会调用工具”

一个普通 LLM 调用通常是输入 prompt，得到答案。Tool use 是模型根据需要请求某个工具，例如搜索、查数据库、运行计算。Agent 在此基础上多了一层循环：它会根据目标和当前状态决定下一步，调用工具，观察结果，再决定是否继续。核心变化是模型参与“下一步做什么”的选择。

这不等于模型可以随便执行动作。工具列表、参数 schema、权限、最大步数、预算、日志和人工确认都在程序侧。模型只是在这些边界内提出下一步。越靠近外部世界的动作，越需要程序侧确认。例如读取知识库和计算价格可以自动，发送邮件、退款、删除文件就不应该只靠模型一句话直接执行。

判断句是：如果流程每一步都由程序固定安排，模型只负责填字段，它更像 workflow；如果模型能基于观察选择下一步工具，并且循环可能走不同路径，它才接近 Agent。

到这里，应该能把 Agent 的直觉说清：它是受控工具循环，不是自由行动的聊天模型。

二、最小循环：计划、行动、观察、停止

LOOP

计划：下一步要具体

计划不是长篇战略，而是当前一步的可执行动作。比如“搜索订单状态”“查询退款政策”“向用户提一个澄清问题”。计划要能映射到工具或最终回复。如果模型写“进一步分析情况”，但没有具体动作，循环就会空转。

工程判断：每一步计划都应该能被日志记录成一个动作类型和参数。

行动：只能调用允许的工具

工具是程序暴露给模型的能力。工具描述要写清用途、参数和限制。模型不能自己发 HTTP、读文件、下单或发邮件；它只能请求程序提供的工具。程序接到请求后还要校验参数和权限。工具越强，护栏越硬。

工程判断：没有 schema、权限和日志的工具，不应该放进 Agent。

观察：工具结果要回到状态

观察是工具执行后的返回值。模型下一步应该基于观察，而不是重新猜。观察要短而准，保留结果、错误码、是否需要重试。工具失败时不要只返回“失败”，要说明失败类型，例如超时、无权限、没查到、参数不合法。

工程判断：如果观察结果不能解释下一步为什么这么走，日志就不够用。

小交互：一步步看循环状态

目标：帮用户查询订单为什么还没发货。
计划：先用订单号查询订单状态，不直接猜物流原因。
工具：调用 get_order_status(order_id)，程序校验用户是否有权查看该订单。
观察：订单已付款，仓库缺货，预计明天补货。
停止：信息足够，生成回复；如果缺订单号，则停止并向用户澄清。
观察点：循环每一步都要能解释“为什么继续”或“为什么停下”。停止条件不清，Agent 很容易反复调用工具。

三、工具、状态和停止条件

TOOLS

工具描述要让模型知道何时不用

工具描述常见错误是只写“这个工具可以做什么”，不写“什么时候不该用”。例如搜索工具不应该用于用户已经提供的事实确认；退款工具不能在没有人工确认时执行；数据库查询工具不能查询非当前用户的数据。好的工具描述会包含用途、参数、返回、限制和失败处理。

状态是 Agent 的记忆，但这里先讲短期状态：用户目标、已知事实、已调用工具、观察结果、剩余步数、预算和待确认事项。没有状态，模型会重复问同一个问题或反复调用同一个工具。状态不需要全量聊天记录，应该保留对决策有用的摘要和结构化字段。

停止条件要提前定义。常见停止条件包括：已得到足够信息、需要用户澄清、达到最大步数、遇到权限限制、工具连续失败、需要人工确认。没有停止条件的 Agent，不是更智能，而是更难控。

工具规格表

字段写法为什么重要
nameget_order_status稳定动作名，方便日志和权限控制
description查询当前用户指定订单的发货状态告诉模型何时使用，也要写限制

四、三个渐进例子

EXAMPLES

例子 1：固定 workflow 不是 Agent

如果程序固定执行“检索知识库 → 生成答案 → 记录日志”，模型只负责在固定位置生成文本，这更像 workflow。它很好，也更可控，但不要因为用了 LLM 就叫 Agent。叫准名字有用，因为不同结构的风险不同。workflow 的主要风险在每一步质量；Agent 还多了路径选择和停止问题。

固定流程：模型不选择下一步pseudo
context = retrieve(user_question)
answer = generate_answer(user_question, context)
save_log(user_question, context, answer)
return answer

例子 2：最小工具 Agent

现在让模型选择是否需要工具。如果用户已经给出足够信息，可以直接回答；如果缺订单号，就问用户；如果有订单号，就请求查询工具。模型的选择空间很小，但已经有 Agent 的雏形：它根据状态决定下一步。

最小代码：受控 Agent 循环python
def run_agent(goal, state, model, tools, max_steps=3):
    for step in range(max_steps):
        action = model.decide(goal=goal, state=state, tools=list(tools))

        if action["type"] == "final":
            return action["message"]

        if action["type"] == "ask_user":
            return action["question"]

        tool = tools.get(action["tool"])
        if tool is None:
            state["observations"].append({"error": "unknown_tool"})
            continue

        observation = tool.call(action["args"])
        state["observations"].append(observation)

    return "信息还不够，我先停止处理。"

max_steps 是停止条件，不是装饰。unknown_tool 的错误观察会回到状态，让模型知道刚才请求的工具不存在。工具执行由程序完成，模型只提出请求。删掉工具白名单，系统就会变成模型随口说什么工具都能用。

例子 3：错误示例，缺少停止和权限

下面这段伪代码看起来像自动化，实际风险很大。它没有最大步数，没有工具白名单，没有权限检查，也没有人工确认。模型如果决定反复搜索，循环就停不下来；如果决定发邮件，程序会直接执行。

错误示例：把模型文本当可执行命令pseudo
while True:
    command = llm("下一步做什么？")
    result = execute(command)
    llm("结果是：" + result)

修法不是把 prompt 写严一点，而是改架构：限定工具、解析 action schema、校验参数、记录每一步、设置最大步数，高风险动作需要确认。prompt 可以提醒模型谨慎，但不能替代这些硬边界。

五、构建最小 Agent 的工程动作

WORKFLOW

1. 明确目标和不做事项

把 Agent 的职责压小。比如“帮助用户查询订单状态并生成解释”，不包括退款审批、修改地址、承诺补偿。目标越含糊，模型越容易扩权。

2. 只给必要工具

工具越多，选择空间越大，错误也越多。第一版可以只给查询订单、查询政策、请求澄清三个动作。没有用到的工具不要提前暴露。

3. 写 action schema

让模型输出结构化动作：final、ask_user、call_tool。每种动作有固定字段。程序只执行合法 action。

4. 记录状态和日志

每一步记录目标、动作、工具参数、观察结果、错误码和耗时。调试 Agent 时，日志就是现场记录。没有日志，只能猜模型为什么这么走。

5. 设置停止条件

最大步数、最大成本、工具连续失败、需要澄清、需要人工确认，都可以成为停止条件。停止不是失败，它是控制系统风险的一部分。

6. 用 Eval 覆盖路径

Agent 的评测不只看最终答案，还要看工具选择、步数、是否越权、是否该停不停。复杂路径要用样本覆盖。

六、常见误区和错误诊断

PITFALLS

误区 1：所有 LLM 流程都叫 Agent

错因：把生成文本、工具调用和自主循环混在一起。后果是风险估计不准。修法是先判断模型是否能选择下一步路径。

误区 2：工具越多越聪明

错因：工具增加会扩大动作空间。后果是误调用、参数错、成本高。修法是只暴露当前任务必要工具，并写清何时不用。

误区 3：没有停止条件

错因：误以为模型会自己判断够了。后果是反复搜索、反复重试、成本失控。修法是最大步数、失败次数、预算和澄清条件。

误区 4：把权限交给模型判断

错因：模型不是权限系统。后果是越权查询、误执行外部动作。修法是程序侧校验用户身份、资源范围和动作权限。

误区 5：只看最终答案，不看路径

错因：最终答案可能碰巧正确。后果是错误工具调用被隐藏。修法是记录并评估每一步工具选择和观察。

误区 6：失败时只加一句“请谨慎”

错因：提示语不能修复架构缺口。后果是同类错误复发。修法是缩小工具、补 schema、加校验、加停止条件或人工确认。

七、三组易混概念

COMPARE
概念组容易混的地方准确区分工程判断
Workflow / Agent都可能包含多个步骤Workflow 路径由程序固定；Agent 可根据观察选择下一步路径固定优先用 workflow，路径需要动态选择才考虑 Agent
Tool use / Agent都会调用工具Tool use 可以是单次工具调用；Agent 是多步决策循环单次查资料不用上复杂 Agent
状态 / 记忆都保存历史信息状态是当前任务决策所需信息；长期记忆跨任务保存偏好或事实先做好短期状态，再谈长期记忆
停止 / 失败停止看起来像没完成停止可以是风险控制：信息不足、需要确认、达到预算能安全停下来的 Agent 才适合上线
