Production Constraints · Latency Cost Reliability

LLM 应用上线前，要先算三笔账

LLM 应用常见的工程问题，不是 demo 答不出来，而是上线后慢、贵、不稳定。延迟决定用户等多久，成本决定功能能不能规模化，可靠性决定失败时系统会不会把用户带进坏结果。三者互相牵制：更强模型可能更慢更贵；更多重试可能提高成功率也拉高延迟和成本；更长上下文可能提高回答质量，也会增加 token 和超时风险。

延迟↔ 成本↔ 可靠性↔ 用户体验

学完后要能做三件事

  1. 拆解一次 LLM 调用的延迟来源：排队、检索、输入 token、输出 token、工具调用。
  2. 估算 token 成本，并知道如何用缓存、路由和长度控制降低成本。
  3. 设计超时、重试、降级、幂等和监控，避免失败被用户直接承受。

这页更偏工程验收。写出好 prompt 后，系统还要经得住真实流量。

一、三角约束：慢、贵、不稳通常连在一起

TRADEOFF

概念：约束比模型能力更早决定可用性

一个 LLM 功能在 demo 里可以等 20 秒，在线产品里可能 3 秒就让用户流失。一个内部工具每天调用 100 次时成本不明显，开放给所有用户后会迅速变成预算问题。一次请求失败在测试时只是刷新重试，线上可能让订单、客服、审批流程卡住。工程里要同时看延迟、成本和可靠性，而不是只看回答质量。

延迟是从用户发起请求到拿到可用结果的时间。成本不只是模型调用费，还包括检索、工具、日志、重试、人工复核和缓存存储。可靠性是系统在异常条件下仍能给出可控结果的能力，包括超时、限流、模型错误、解析失败、工具失败、网络波动、重复请求等。

边界要明确：不是所有功能都要追最低延迟和最高可靠性。离线批处理可以慢一点，客服自动回复要稳一点，聊天陪练可以接受偶发重试，支付相关动作必须强一致和人工确认。判断句是：先按业务后果定约束，再选模型和架构。

到这里，应该能说清：LLM 应用的可用性由质量、延迟、成本和失败处理共同决定。

二、延迟：先拆来源，再决定优化

LATENCY

首 token 延迟和总延迟不同

首 token 延迟是用户开始看到响应前的等待时间，总延迟是完整结果生成完的时间。聊天场景可以用流式输出降低等待感，但结构化 JSON、后台任务和工具调用不一定适合流式。不要只看“模型生成很快”，还要看请求排队、检索、网络、工具调用和后处理。

判断句：用户是否能边看边用，决定你优化首 token 还是总完成时间。

输出长度常常是主要变量

输入 token 会影响模型处理时间，输出 token 也会明显增加总延迟。让模型“详细解释”会增加输出长度，进而增加等待和成本。很多业务任务不需要长回答，应该限制输出字段、字数或只返回结构化结果。

判断句：如果任务只需要分类或抽取，就不要让模型写长篇理由。

串行工具调用会拖慢系统

Agent 和 RAG 常见慢点是多次串行调用：先改写查询，再检索，再重排，再生成，再调用工具。每一步都增加失败点。能并行的检索和计算尽量并行，能提前缓存的不要每次重算。复杂流程要设置总超时，而不是只给每个步骤单独超时。

判断句：一次请求的最慢路径，才是用户真实体验。

优化方向

观察点：输入越长、输出越长、工具越多，延迟和成本都会上升。先缩短不必要的上下文和输出，再考虑换模型。

三、成本：按 token、请求和失败次数算

COST

成本不只来自一次成功调用

最直观的成本是输入 token 和输出 token。输入越长，模型处理材料越多；输出越长，生成时间和费用越高。但真实成本还包括重试、检索、重排、工具调用、日志存储、人工复核和缓存维护。系统失败率越高，重试越多，单次成功结果的真实成本就越高。

成本优化不等于总是用最便宜模型。便宜模型如果导致失败率高、重试多、人工复核多，综合成本可能更高。更常见的做法是路由：简单问题用小模型，复杂问题用强模型；高风险问题进入人工或更严格流程；重复问题走缓存。

边界是：不要为了省成本牺牲关键可靠性。比如医疗、金融、合规、退款承诺等场景，省掉人工确认或证据检查，可能把小成本变成大损失。

成本控制手段

手段适用场景风险
压缩输入上下文里有大量无关文本压掉关键证据
限制输出分类、抽取、固定报告解释不足，用户难理解
缓存重复问题、稳定资料、相同参数资料更新后返回旧答案
模型路由请求难度差异大路由错会影响质量
批处理离线分析、低实时要求不适合实时交互

四、可靠性：失败要可预期、可恢复、可追踪

RELIABILITY

超时和重试

超时是系统承认“这次等太久了”。没有超时，请求可能一直挂着，占用资源。重试适合临时网络错误、限流、模型服务波动，但不适合所有错误。解析失败可以带失败原因重试一次；权限错误、业务规则拒绝、明显无资料，不应该盲目重试。

判断句：重试前先区分错误类型，不然只是把失败变贵。

降级和兜底

降级是失败时给出次优但可控的结果。比如模型超时时返回“已转人工处理”，RAG 无资料时给出无法回答并提示补充信息，摘要失败时展示原文和关键字段。兜底不是编一个答案，而是保护用户和系统不进入错误动作。

判断句：降级结果要诚实说明状态，不要伪装成成功。

幂等和重复请求

幂等是重复执行不会造成额外副作用。LLM 系统里，用户刷新、网络重发、重试工具调用都可能重复触发动作。查询通常天然幂等，发送邮件、创建工单、扣费、退款不是。对有副作用的动作，要有请求 id、去重和确认机制。

判断句：只要动作会改变外部状态，就要考虑重复请求的后果。

最小代码：超时、有限重试、降级python
def call_with_guardrails(request, llm_client, max_retries=1):
    errors = []
    for attempt in range(max_retries + 1):
        try:
            raw = llm_client.complete(request, timeout_seconds=8)
            parsed = parse_json(raw)
            validate(parsed)
            return {"ok": True, "data": parsed}
        except TimeoutError:
            errors.append("timeout")
        except JsonParseError:
            errors.append("bad_json")
        except ValidationError as exc:
            return {"ok": False, "mode": "safe_stop", "reason": str(exc)}

    return {
        "ok": False,
        "mode": "fallback",
        "message": "暂时无法生成可靠结果，已保留原始请求。",
        "errors": errors
    }

这段代码体现了三个边界：超时和 JSON 解析失败可以有限重试；验证失败直接安全停止，因为再试一次不一定解决业务违规；重试耗尽后走降级，不把半成品伪装成成功结果。

五、三个渐进例子

EXAMPLES

例子 1：客服摘要从慢到可用

第一版客服摘要把整段历史聊天都塞给大模型，要求输出长总结、情绪、原因和回复建议。它能跑，但慢且贵。拆解后发现，大部分请求只需要最近 10 条消息和固定 JSON 字段。优化动作是截断无关历史、限制输出字段、把回复建议放到需要时再生成。

这里不是简单“少给上下文”。如果历史里有退款承诺或投诉升级，截断会出事故。所以要按业务规则保留关键事件，再压缩普通闲聊。

例子 2：RAG 问答的可靠降级

RAG 问答最常见的失败不是模型不会说话，而是资料没检到或资料冲突。可靠做法是：检索分数低时不要强答；资料冲突时说明需要人工确认；模型生成答案后检查引用是否来自上下文。降级结果可以是“资料库中没有足够依据”，而不是编一个看起来合理的答案。

这会牺牲一部分回答率，但换来更低的幻觉风险。是否值得，要看业务后果。

例子 3：错误示例，只靠无限重试追成功

错误示例：无限重试让失败变贵pseudo
while result is None:
    result = call_llm(prompt)
return result

这段逻辑没有超时、没有最大次数、没有错误类型，也没有降级。模型服务异常时，它会持续消耗资源；输入本身不合格时，它会重复失败；如果某次返回半成品，还可能直接进入业务。修法是设置总超时、有限重试、错误分类和安全停止。

六、最小工程动作：从第一天就记录约束

WORKFLOW

1. 定 SLO

SLO 是服务目标，例如 P95 延迟小于 5 秒、成功率大于 99%、单次请求成本低于某个阈值。没有目标，就无法判断优化是否足够。

2. 记录 token 和耗时

每次调用记录输入 token、输出 token、模型、耗时、是否重试、是否命中缓存。没有这些数据，成本和延迟只能靠猜。

3. 设置超时和预算

每个步骤和整条链路都要有时间预算。Agent 还要有最大步数和最大工具调用数。预算用完就降级或停止。

4. 做错误分类

把失败分成超时、限流、解析失败、验证失败、检索无结果、权限拒绝、工具异常。分类后才知道该重试、降级还是修业务规则。

5. 设计降级路径

失败时返回什么、转给谁、是否保存现场、是否提醒用户，都要提前设计。不要等线上失败后临时写一句道歉。

6. 监控和告警

监控 P50/P95 延迟、错误率、重试率、解析失败率、成本、缓存命中率。告警要连到可行动的错误类型。

七、常见误区和错误诊断

PITFALLS

误区 1：只优化模型，不看链路

错因：延迟可能来自检索、工具、网络和后处理。后果是换模型后仍然慢。修法是分段记录耗时，先找最大瓶颈。

误区 2：上下文越长越好

错因：长上下文会增加成本、延迟和噪声。后果是模型读到无关内容，回答反而不稳。修法是保留关键证据，压缩或过滤无关材料。

误区 3：重试越多越可靠

错因：重试只适合临时错误。后果是成本和延迟上升，确定性错误反复失败。修法是错误分类和有限重试。

误区 4：缓存一定安全

错因：资料、权限和用户上下文会变化。后果是返回旧答案或泄露不该复用的内容。修法是设置缓存 key、过期时间和权限边界。

误区 5：降级就是随便回一句抱歉

错因：降级需要保持状态和用户可行动。后果是用户不知道下一步怎么办。修法是说明当前状态、保留请求、提供人工或重试入口。

误区 6：只看平均延迟

错因：平均值会掩盖尾部慢请求。后果是少数用户体验很差却没被发现。修法是看 P95/P99、超时率和慢请求样本。
