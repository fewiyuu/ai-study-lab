数据工程全景：从网页到 token

这一页只处理一条最基础的链路：把原始网页、抽取后的 document、训练 sample、token 和 shard 分开。后面的过滤、去重、配方、验证和运行记录，都在这条链路上工作。先把对象认准，再看工具和实验，后面的单元就不会混在一起。

课程地图下一单元

读完本页应该能做什么

  • 能把 raw data -> document -> sample -> token -> shard 画成一条链，并说出每一步的输入和输出。
  • 能分清 document vs sample、token vs shard，并知道哪个概念服务于治理，哪个概念服务于训练或存储。
  • 能读一条小型 pipeline 记录或 manifest，判断哪一步出了问题，以及下一步该回查什么。

学习目标

foundation
先认对象

原始网页还不是可训练语料。先确认它有没有正文、元数据和可追溯来源，再决定后面怎么处理。

再认转换

同一份内容会在不同阶段变成不同对象。抽取、分句、packing、tokenize、shard 不是同一件事。

最后认证据

要能回到来源、规则版本、tokenizer 版本和 shard 清单。只有规模没有证据，后面很难排错。

这页的边界这里只把对象层次和最小流水线讲清。过滤规则、去重算法、配方比例、消融实验和合规判断，后面的单元会继续展开。

对象链

trace

最小链路先记成五个对象：raw data、document、sample、token、shard。你看到一个数据集说明时，先问它站在哪一层，再问它往下一层怎么变。

raw data 读取 crawl record、URL、response、headers。

写入原始记录。这里最容易把模板、脚本、乱码和重复页一起带进来。

document 读取正文抽取器和元数据保留规则。

写入正文、语言、来源、长度、过滤标记。正文抽错，后面的质量判断就不稳。

sample 读取 context length、packing policy、边界规则。

写入训练输入。样本边界一乱，训练 loader 和评测对齐都会变难。

token 读取 tokenizer 版本与词表。

写入 token id 序列。版本一变，token 数和序列切分都会跟着变。

shard 读取样本流、checksum、分片策略。

写入并行文件块和 manifest。没有校验和，排错时很难回到来源。

对象 前一层读什么 这一层写什么 错了会怎样
raw data URL、response、crawl record 原始抓取记录 脚本、模板和乱码会一起进来
document 正文抽取器、元数据规则 正文 + 元数据 + 过滤标记 正文抽错后，后面的质量分都不可靠
sample 上下文长度、packing、边界规则 训练输入窗口 样本边界乱了，训练和评测对不齐
token tokenizer 版本与词表 token id 序列 版本漂移会改变长度和切分
shard 样本流、checksum、分片策略 并行文件块和 manifest 缺少记录时，回查会卡在半路

术语与误区

table

这五个词最容易混。先把词义说准，后面的工具和实验就会顺。

术语 直觉解释 精确定义 常见误解
raw data 还没整理过的来源材料。 从网页、仓库、书籍、论文、论坛、百科等来源拿到的原始文件或记录。 误以为 raw data 已经是可训练文本。它常含导航、广告、脚本、乱码和重复。
document 能被过滤和去重的一篇文本对象。 解析 raw data 后得到的正文、URL、时间、语言、来源、质量分、过滤原因等字段组合。 误以为 document 一定等于一整个网页。它可以是一篇抽取后的文章，也可能被切分或合并。
sample 训练 loader 拿到的一段可学习材料。 按上下文长度、边界、混合策略或 packing 规则构造出的训练样本。 误以为 sample 和 document 是同一个东西。document 偏治理，sample 偏训练输入。
token tokenizer 能识别的离散单位。 文本经过 tokenizer 映射后的 token 或 token id。 误以为 token 就是单词。中文、代码、空格、标点和罕见词都会按 tokenizer 规则切分。
shard 为了并行读取而切好的文件块。 把 documents、samples 或 token ids 分成多个大小可控的存储单元。 误以为 shard 越大越好。过大影响并行和恢复，过小增加文件管理和调度开销。
document vs sample

document 是治理对象，sample 是训练对象。一个 document 可以切出多个 sample，多个短 document 也可以被拼进一个 sample。

token vs shard

token 是模型学习的离散内容，shard 是工程上的存储和读取单位。前者关心模型看到了什么，后者关心 loader 怎么读得稳。

工程流水线

mechanism

最小版本可以看成五步：读取原始记录、抽正文、做质量处理、构样与混合、tokenize 并打包成 shard。真实工程常把其中几步拆得更细，但这五步够你定位问题。

01 Raw Ingestion

读入 Common Crawl、站点抓取、代码仓库、书籍或内部语料。此时主要问题是格式杂、来源杂、噪声高。

02 Extract Documents

从 HTML、WARC 或 JSON 中抽正文，保留 URL、时间、域名、语言候选和长度。正文抽取质量会决定后面所有过滤器的输入。

03 Curate Documents

过滤低质、重复、语言不符、污染评测集或不合规的 document。过滤会改变数据覆盖和偏差。

04 Build Samples

按上下文长度、packing 规则和数据配方把 document 变成 sample。这里开始显式回答“模型应该多看哪类数据”。

05 Tokenize And Shard

用目标 tokenizer 把 sample 转成 token ids，再打包成 shard，供训练 loader 分布式、可重复、可监控地读取。

$$ \text{训练可用数据} = \text{来源可追溯} + \text{文本可治理} + \text{样本可配方} + \text{token 可复现} + \text{shard 可读取} $$

这个式子不是数学证明，意思是：一项缺了，后面的训练和复盘就会断。

检查顺序

如果某个数据集结果怪，先回看 document 是否抽对，再看 sample 是否按上下文长度切对，最后看 tokenizer 和 shard 是否被记录下来。别先怀疑训练 loop。

manifest-mini.json
{
  "source": "Common Crawl / 网站抓取",
  "document": {
    "extractor": "main-text",
    "keep_metadata": true
  },
  "sample": {
    "context_length": 2048,
    "packing": "greedy"
  },
  "tokenizer": "sentencepiece-v1",
  "shard": {

运行记录

diagnosis

只看数据量很容易误判。真正能把问题拉回现场的，是记录里是否写清了来源、抽取版本、样本边界、tokenizer 和 shard 校验。下面这张小表就是最小回查链路。

记录项 它回答什么 缺了会怎样 下一步回查
source 这批数据从哪里来 无法判断来源是否重复或偏置 回到 raw data 和抓取记录
document.extractor 正文是怎么抽出来的 很难判断正文是否抽错或丢失 检查 document 与原始网页对照
sample.context_length 训练窗口有多长 sample 边界和训练对齐会漂 回看 packing 与切窗规则
tokenizer 序列是怎么离散化的 token 数、长度和评测统计会变 确认 tokenizer 版本是否一致
shard.checksum 文件是否完整、能否复现 训练报错后很难证明文件没坏 先查 shard manifest 再查生成版本
怎么用这张表

当一个 shard 出错时，不要先猜模型。先看 checksum，再看 tokenizer 版本，再回到 sample 边界和 document 记录。这样错误会尽量停在数据层。

为什么这页要加它

这一段把对象链和运行记录接起来。前面只是知道对象，现在开始知道出了问题该回到哪一条记录。

故障回查路径

case

假设训练日志里出现了一个很普通的报错：shard-00042.jsonl: missing checksum。这个报错看起来像文件问题，实际上它常常是一个更早阶段的信号，说明生成记录、样本边界或者 tokenizer 版本里至少有一项没写对。处理这种问题最怕直接跳到训练 loop，因为训练只是把前面的错放大了。

更稳的顺序是：先确认 shard 本身是不是完整，再看它对应的 manifest 是否写全了来源和抽取版本，然后回到 document 和 sample，检查正文是不是抽对、窗口是不是切对，最后才看 tokenizer 是否换过。如果这里的顺序反了，你会在模型参数里绕很久，但问题其实早就出现在数据层了。

trace.log
shard-00042.jsonl: missing checksum
  -> check shard manifest
  -> verify source and extractor version
  -> compare tokenizer version
  -> confirm sample boundary rules
  -> if all above are stable, return to document extraction
症状 先查什么 它读什么 它写什么 错了会怎样
checksum 缺失 shard manifest 文件完整性、生成版本 回查入口 你连这个 shard 是否可信都说不清
tagger/tokenizer mismatch tokenizer 版本 词表、编码规则 token id 序列 token 数和长度统计会整体漂移
sample 边界异常 sample 规则 context length、packing policy 训练窗口 训练和评测对不齐，指标也会歪
document 丢字段 抽取版本 正文抽取器、元数据保留规则 正文 + 元数据 后面的过滤器会在错误输入上工作

为什么这条路径要排成这样

每一层都只依赖前一层的记录。shard 依赖 sample，sample 依赖 document，document 依赖 raw data。你越早检查高层文件，越能把错误卡在离来源最近的位置。

什么时候回到单元 02

如果问题出在 document 抽取，就回到“获取、解析与标准化”单元；如果问题出在 tokenizer，则先确认 tokenizer 版本，再回到 sample 和 shard。不要把这些问题混成一个“训练不稳定”。

这一段把对象链和运行记录接起来了。前面你只是知道五层名字，现在你应该能说出：一个坏 shard 先看什么、为什么先看它、看完之后应该去哪一层继续查。

链路如何影响指标

impact

这页的最后一件事，是把对象层次和训练指标连起来。很多团队看到困惑度上升，第一反应是“模型不行了”，但真正的变化可能发生在 document 抽取、sample 边界、tokenizer 版本，或者 shard 的读取方式上。换句话说，指标只是最后的回声，未必是最早的原因。

变化发生在哪层 典型现象 最容易被误判成什么 更稳的检查动作
document 正文抽少了，语言混杂了，噪声上来了 模型学不会 回看抽取器、过滤标记和元数据保留规则
sample 样本切窗变短或跨文档拼接太多 训练不稳定 检查 context length、packing policy 和边界
token token 数明显变化，损失曲线起伏变大 学习率不合适 确认 tokenizer 版本和词表是否切换
shard 读数据慢、报错、恢复困难 训练框架有 bug 回看 shard size、checksum 和 manifest

一个短例子

假设同样的文档，在新版本里被切成更短的 sample，同时 tokenizer 也换了。你看到的第一个现象可能是 token 数变化，第二个现象可能是 perplexity 变差，但根因不一定是模型本身，而是输入边界变了。

怎么把这个例子用到排错里

先问“哪一层先变了”，再问“它写了什么记录”。如果回答不了这两个问题，就不要急着改模型参数。先把输入层和记录层弄清，再看训练曲线。

debug-sequence.txt
1. 先查 shard 是否完整
2. 再查 tokenizer 版本是否一致
3. 再查 sample 边界和 packing
4. 再查 document 抽取与过滤
5. 最后才回到训练参数

例题与修正

worked case

下面这条记录把同一个网页在四个层次上拆开。你可以顺着它回想：每一层在读什么、写什么、丢了什么。

object-chain.txt

  WARC/HTML record, URL, crawl timestamp, response headers

为什么要保留 metadata

只保正文，后面很难查污染、查偏差、查来源重复，也很难定位某个 shard 为什么报错。metadata 不一定进训练文本，但它一定要进运行记录。

为什么 shard 不是小事

shard 太少并行度不够，太碎又会让文件打开和调度成本上升。好的 shard 设计关心读取吞吐、故障恢复和来源混合，而不只是把文件切小。

常见混淆

debug
把网页当语料

网页是来源，不一定是语料。抽取正文前，它还可能含模板、广告、脚本和重复块。

把过滤当清洁

过滤器在移除噪声，也在改变分布。过滤结果需要用训练和评测验证。

把 tokenizer 当细节

tokenizer 会改变 token 数、序列长度和罕见文本的表示方式，不能等到最后才随便选。

把 sample 当 document

sample 是训练输入，可能跨 document 拼接，也可能从长 document 切出多个窗口。

把 shard 当压缩包

shard 的目标是并行读取和断点恢复，不是单纯把文件压小。

把工具当配方

工具只实现流程，配方还要经过验证。FineWeb 的意义就在于把数据决策拉回实验闭环。

# case 1

=> this shard cannot be traced back to source mixing.

先从对象错位查起，再从规模和吞吐查。很多“模型效果不稳”的描述，最后都能回到 document、sample、token 或 shard 的某一层记录不完整。

顺序应是读取 raw data -> 抽取 document -> 构造 sample -> tokenize -> 写入 shard。理由是前一层的输出要稳定，后一层才能判断自己的输入是否正确；如果顺序倒过来，错误会在更晚的层才暴露。
