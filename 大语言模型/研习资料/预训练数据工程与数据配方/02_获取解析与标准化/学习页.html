基础单元

获取、解析与标准化

这一页只处理一条入口链：把网页、WARC 记录或导出文件变成可追溯的 document。先抓住原始记录，再抽正文并保留元数据，最后把编码和字段标准化成后续过滤能直接接手的输入；质量过滤、去重和配方留到后面的单元。

课程地图上一单元下一单元

读完这一页，你应该能做什么

  • 分清 raw data、document 和后续训练样本，不把网页直接当语料。
  • 用状态表说明正文抽取、编码标准化和元数据记录各自读什么、写什么。
  • 拿着一份抽取失败的记录，能判断先回查哪一步。

学习目标

foundation

先认入口

这一页关注网页进入数据管线时，哪些东西必须原样留下，哪些东西要先抽成 document。

先认边界

正文抽取、编码标准化和元数据记录是这一页的中心。质量过滤、去重、配方和 tokenization 都不在这一页里展开。

先留证据

只要后面还要回到原始页面、抓取记录或抽取版本，就必须保留 source_url、charset、fetched_at、extractor_version 和 content_hash 这类字段。

这一页的使用方式

先顺着对象链读，再看抽取和标准化怎么改对象，最后用回查表判断问题应该落在哪一步。你如果一开始就跳到过滤规则，往往会把输入层的问题误判成质量问题。

对象链

trace

这条链只看四个可检查的状态：原始字节、解析后的页面结构、抽取后的正文记录、标准化后的 document。四个状态各有读写关系，混在一起就会丢掉回查入口。

raw bytes 读取 URL、响应头、抓取时间和原始字节。

写入原始记录。这里最常见的失误是把模板、脚本、乱码和重定向结果一起带进来。

parsed DOM 读取 HTML 解析器和字符集提示。

写入页面结构。这里如果解码错了，后面的抽取看到的就是一棵坏树。

document 读取正文抽取器和元数据保留规则。

写入正文、来源、时间、语言候选和抽取版本。后面的过滤和去重都依赖它是否干净。

normalized doc 读取统一编码、NFKC 和空白整理规则。

写入可比较、可检索、可切分的文本。没有这一步，重复检测和字符级统计会漂。

later filter 读取已经标准化的 document。

写入质量过滤、去重和配方的输入。这里不该再回头做正文抽取。

状态 前一步读什么 这一层写什么 错了会怎样
raw bytes URL、响应头、抓取时间 原始记录 后面不知道原页长什么样
parsed DOM HTML 和 charset 提示 页面结构 乱码会把结构一起带歪
document 正文抽取器、保留字段规则 正文 + 元数据 正文抽错后，后面的判断都不稳
normalized doc 统一编码、NFKC、空白规则 可比较、可检索的文本 重复和统计都容易失真

获取入口

source boundary

这一层先回答“你拿到的到底是什么”。有些来源给你的是网页字节，有些给你的是导出 CSV，有些给你的是 WARC 或 JSONL。不要一上来就把它们都叫成网页文本。

来源形态 先留什么 容易漏掉什么 为什么这一步重要
网页 HTML / WARC URL、响应头、抓取时间、原始字节 charset、重定向、正文之外的脚本与导航 抽取和标准化都要回到原始记录
站点导出 / CMS 导出 栏目、更新时间、作者、原站链接 字段语义和权限边界 导出内容常常已经被预处理过
论坛页 / 讨论串 主帖、回复顺序、楼层、时间戳 引用嵌套和签名块 后面做正文抽取时要知道层次
本地文件 / JSONL schema、版本、生成时间 生成脚本和字段约定 后面回查时要知道是谁产出的

这一层不做什么

这里不判断内容好坏，也不做质量过滤。先把对象和记录留住，再决定值不值得保留。输入边界没定清，后面每一步都容易偏题。

正文抽取

mechanism

抽取要把网页里真正进入 document 的正文找出来，同时把来源和版本信息留住。导航、推荐阅读、广告、脚注和重复块常常混在同一页里，抽取器要给它们划清边界。

extract_document.py
def build_document(raw):
    dom = parse_html(raw.bytes, charset_hint=raw.charset_hint)
    main_text = extract_main_text(dom)
    doc = {
        "source_url": raw.url,
        "fetched_at": raw.fetched_at,
        "extractor_version": "main_text_v1",
        "text": normalize_text(main_text),
    }
    return doc
步骤 读什么 写什么 典型症状
解析 HTML 原始字节和 charset 提示 页面树 乱码把整页结构带歪
找主文 DOM 结构、段落密度、链接密度 正文文本 只剩导航或空白
保留元数据 URL、抓取时间、抽取版本 document 记录 后面回不去原页

一个最小对照

原始页里可能有“首页 / 分类 / 登录 / 推荐阅读 / 正文 / 评论区”。抽取后，document 应该留下正文和必要元数据，避免把所有块原封不动存进去。

别把这里做成过滤

正文抽取关心“哪一块是正文”。质量过滤关心“正文值不值得保留”。把这两件事混在一起，会让坏抽取看起来像坏数据质量。

案例走读：从一个旧网页到 document

walkthrough

下面把前面的概念压进一个可检查的小案例。假设数据源是一批旧站点页面，页面来自 WARC 或离线 HTML 归档，编码标注不稳定，正文周围还有导航、登录入口、推荐阅读、评论和页脚。目标不是马上判断这篇文章好不好，而是先产出一条可以回到原始页面、可以复现实验、可以交给后续过滤器的 document。

走读时只问三件事：当前步骤读的对象是什么，写出的对象是什么，失败以后第一轮应该回到哪里。只要这三件事没有写清，后面的过滤、去重、配方和训练统计都会缺少解释力。

raw_legacy_page.html
<!doctype html>
<html>
<head>
  <meta charset="gb2312">
  <title>如何整理训练语料</title>
</head>
<body>
  <nav>首页 | 模型 | 数据集 | 登录</nav>
  <main>
    <h1>如何整理训练语料</h1>
    <p>预训练数据工程的第一步，是把不同来源的网页变成结构稳定的文档记录。</p>
    <p>记录里要保留来源、抓取时间、抽取器版本和正文哈希，方便后续过滤与回查。</p>
  </main>
  <aside>相关阅读：十个数据集清洗技巧</aside>
  <footer>版权声明 | 站点地图 | 联系我们</footer>
</body>
</html>
检查点 输入对象 输出对象 验收信号
获取 URL、响应头、原始字节、抓取时间 raw record 能看到字节长度、HTTP 状态、charset 提示和抓取批次
解析 raw record 里的字节与编码线索 DOM 或文本块列表 标题、段落、导航、页脚被分成可检查的块
正文抽取 DOM、块密度、链接密度、标签位置 main_text 正文段落被保留，导航、aside、footer 没有混进主文本
标准化 main_text、字符编码、Unicode 形态、空白符 normalized_text 中文可读，空白稳定，同一内容在不同抓取批次里便于比较
封装 normalized_text 加来源字段 document 包含 source_url、fetched_at、charset、extractor_version、content_hash

这张表的作用是把“页面处理成功了吗”拆成五个可验证的小问题。获取成功，只说明你拿到了字节；解析成功，只说明字节能被拆成结构；抽取成功，才说明主文进入记录；标准化成功，才说明字符层面稳定；封装成功，才说明这条记录以后可以被解释。很多数据事故出在这里：团队只保存了最终文本，几周后发现训练样本里混入大量导航，却无法知道是哪一版抽取器产生的。

第一轮输出长什么样

第一轮 document 可以很朴素，但字段必须承担职责。`source_url` 负责回到原页，`fetched_at` 负责定位抓取批次，`charset` 负责解释解码路径，`extractor_version` 负责对照规则变化，`content_hash` 负责后续重复检测，`text` 才是模型最终可能看到的主体内容。

这一步最怕什么

最怕把“看起来能读”当作完成。页面能读，不等于 document 稳定；正文能看到，不等于元数据完整；当前样本正常，不等于下一批旧页面也能按同样规则处理。学习这一单元时，判断标准要落在对象、字段和回查路径上。

注意这张故障表的顺序。它把问题放回最早可能出错的对象层：字符异常先查获取和解析，主文异常先查抽取，重复异常先查标准化与哈希，来源缺失先查 schema。这个顺序会让排查成本下降，也能避免把入口层的坏记录误判成后续算法问题。

怎么把走读迁移到自己的数据

拿到一个新来源时，先挑 20 到 50 个样本人工走一遍这张链。每个样本都记录原始 URL、抓取时间、编码线索、抽取文本、标准化文本和最终 document。然后专门挑异常样本：乱码页、空正文页、超短页、导航占比高的页、重复标题页。只要异常样本能被字段解释，后面的批处理才值得扩大规模。

完成这个练习后，你应该能做两件事：第一，读懂一个数据管线日志时能指出它处理的是 raw、DOM、text 还是 document；第二，看到一个坏样本时能写出第一轮回查路径，而不是直接调过滤器、去重器或训练脚本。

抽样检查 合格信号 失败时动作 能不能放大批处理
随机样本 大多数 document 字段完整，正文可读 补齐 schema 或修正基础解析 字段不完整时不放大
异常样本 乱码、空正文、导航混入都能定位到具体步骤 把失败类型写进日志和回归样本集 异常无法解释时不放大
重复样本 同文不同 URL 的哈希关系可解释 检查标准化位置和哈希计算规则 哈希漂移时不放大

放行标准要写成可执行问题

“页面看起来差不多”不是放行标准。更稳的写法是：50 个随机样本里缺 source_url 的比例是多少，乱码样本能不能回到原始字节，抽取器版本有没有写入每条 document，导航块是否进入 main_text，哈希是在标准化前还是标准化后计算。每个问题都对应一个字段或一步处理动作，回答不出来就说明这批数据还没有准备好交给过滤和去重。

一个可操作的抽样记录可以这样写：样本 17 来自旧站点新闻页，raw 里 charset 是 gb2312，解析后 main 区域存在两个段落，抽取后混入页脚版权，标准化文本可读但 content_hash 已生成。这个样本的结论不是“丢掉”，而是“抽取器回归样本 + 暂停放大批处理”。练习时你可以照这个格式写三条异常记录，每条都要指出读什么字段、查哪一步、修完以后用什么信号验收。修复后再跑同一组异常样本，确认旧问题没有重新进入 document。

这也是本单元和后续单元的交界线。完成获取、解析、抽取和标准化后，你手里应该有一批“可解释的 document”。下一单元才开始讨论哪些 document 应该留下，哪些要丢掉，以及过滤规则会带来什么偏差。把放行标准写清，后面的质量判断才不会替入口层的问题背锅。

编码标准化

stability

抽取完成以后，还要把字符层面的不稳定压下去。这里处理的是可比较、可检索、可重复的文本形态，不改写内容含义。字符解码错了，后面的去重、匹配和统计都会跟着错。

normalize_text.py
import unicodedata

def normalize_text(raw_bytes, charset_hint=None):
    text = raw_bytes.decode(charset_hint or "utf-8", errors="replace")
    text = unicodedata.normalize("NFKC", text)
    text = " ".join(text.split())
    return text
问题 这一步在读什么 这一步写什么 不做会怎样
错误解码 原始字节和 charset 提示 可读文本 乱码进入 document
全角半角混用 字符形式 统一表示 重复检测和关键词匹配漂移
空白不一致 换行、制表、多个空格 稳定的分隔方式 切分和统计结果不稳

这一步为什么放在抽取之后

先抽取正文，再统一字符，顺序更稳。你如果先做字符清理再找正文，很多页面的结构线索会被抹掉，抽取器反而更难判断哪一块是正文。

元数据记录

schema

document 是一条能回查的记录。正文之外的字段承担解释职责，它们决定你以后能不能说明这条记录从哪里来、是谁抽的、什么时候抽的、用什么规则抽的。

document-record.json
{
  "source_url": "https://example.org/article",
  "fetched_at": "2026-06-20T10:15:00Z",
  "raw_format": "html",
  "charset": "utf-8",
  "extractor_version": "main_text_v1",
  "language": "zh",
  "content_hash": "sha256:...",
  "text": "..."
}
字段 作用 缺了会怎样
source_url 回到原始页面 你不知道这条正文从哪来
fetched_at 标记抓取时间 无法判断页面是否后来改过
charset 帮助解码和排错 乱码问题很难回溯
extractor_version 记录正文抽取版本 没法比较新旧抽取器差异
content_hash 做去重和回查 同文不同名时很难发现重复
language 做后续路由和过滤 语言混杂会被拖到后面才暴露

案例与修正

worked case

同一页网页，最少会有三种结果：原始页面、错误抽取后的 document、修正后的 document。你要会区分它们，不然会把“页面本来就这样”误判成“处理已经完成”。

状态 看到什么 留下什么 下一步该查什么
原始页面 导航、正文、脚注、广告都在 网页字节和抓取记录 先确认解析和编码是否正常
错误 document 正文被裁掉，或者只剩导航 一条坏记录 回到抽取规则，不要先改过滤器
修正后 document 正文清楚，元数据完整，字符稳定 可以交给后续过滤的输入 再进入质量过滤和去重单元

论坛页和新闻页的差别

论坛页要更看重主帖、回复、楼层和引用嵌套；新闻页更看重正文、时间、栏目和作者。抽取规则如果对两类页面一视同仁，结果通常会偏。

这页的终点

做到这里，你应该能把一条网页记录变成可追溯的 document，并知道问题该回到哪一层。下一页才开始谈质量过滤、规则和偏差。

边界说明

boundary

这一页已经覆盖的

获取入口、正文抽取、编码标准化、元数据 schema 和回查顺序。你现在可以把 raw data 和 document 分开，也能解释为什么记录字段不能省。

下一页才会继续的

质量过滤、规则模型、偏差、去重、污染控制、数据配方和 tokenization。它们都建立在这里产出的 document 之上，不应该在这页里提前展开。

如果你在实作时发现 document 里还残留大量导航和脚注，先回到正文抽取；如果字符已经错了，先回到编码标准化；如果根本回不了原页，说明 schema 没留 source_url。顺序错了，后面每一层都会替你放大错误。

练习

diagnostic

怎么答才算会

先分对象，再讲转换，再回到记录。能说出输入、输出、读写关系和一个常见故障，才算真的会。

检查重点

选择题看对象边界；填空题看术语是否稳定；开放题看原因、后果和回查对象。答案不用长，但要能指向下一步检查什么。

一、对象和边界

先把 raw data、document 和后续训练输入分开。对象一旦混掉，后面的工具都会看错层。

先把对象说准，再谈流程。不要把网页、正文和训练样本混成一层。

mini-fieldsrecord

把入口记录写清楚：输入形态、保留字段、输出对象。

synthesis概念区分

用两句话解释 document 和 sample 的区别

至少讲清楚：谁给治理看，谁给训练看；一个 document 怎样变出多个 sample。

参考答案 / 评分要点

合格答案应包含三点：document 是抽取后的治理对象，sample 是训练输入；sample 可以由一个 document 切窗，也可以把多个短 document pack 到一起；不要把二者说成同一层。

二、抽取和标准化

把页面噪声、字符问题和正文抽取分开。这里最容易的错误，是把“文本变少了”误判成“数据质量变差了”。

抽取决定哪块是正文，标准化决定字符是否稳定。两步不能互相替代。

三、元数据和 schema

字段承担回查职责。只要还要回到原始页面、对照抽取版本或做去重，元数据就得先写好。

能回查、能复现、能比较，是这个 schema 的三件事。
choicetraceability

哪个字段最直接帮助回到原始页面？

fillversioning

用于记录抽取器版本的字段通常叫什么？

synthesisschema

设计一份最小 document schema，至少写出 5 个字段。

必须说明每个字段的作用。只列字段名不够，要写出它帮助回查、比较还是复现。

参考答案 / 评分要点

合格答案至少要覆盖：回到原页的字段、抓取时间、字符集、抽取版本、内容哈希。更完整的答案还会补 language、raw_format、crawl_id 或 similar fields。解释里要说清每个字段服务于什么后续动作。

四、回查和迁移

看到故障时，先回到哪一层、再查什么字段，是这一页最值钱的能力。

先查 charset，再查抽取，再查 schema。不要先动后面的过滤和训练参数。
choicedecode

看到乱码时，第一轮先查什么？

diagnosisextract

如果 document 里只剩导航和广告，你会先回查哪一步？

写出第一步、第二步和为什么要按这个顺序查。不要一上来就怀疑过滤器。

参考答案 / 评分要点

先回到正文抽取，再看解析和编码是否正常。导航和广告占比过高，说明抽取规则出了问题；如果字符都错了，先修解码，再看正文。过滤器要等 document 本身可信以后再查。

openchecklist

写一条你会放进抽样验收清单的规则。

规则要能被检查，最好包含字段、阈值或失败动作。

参考答案 / 评分要点

合格规则要可执行，例如要求 source_url、extractor_version、content_hash 全部存在；随机样本乱码率超过阈值就停止；导航块进入正文时回查抽取器版本。只写“质量要好”不算合格。
