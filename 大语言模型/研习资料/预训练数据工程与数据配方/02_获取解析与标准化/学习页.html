单元 02

获取、解析与标准化

这一页只处理一条入口链：把网页、WARC 记录或导出文件变成可追溯的 document。先抓住原始记录，再抽正文并保留元数据，最后把编码和字段标准化成后续过滤能直接接手的输入；质量过滤、去重和配方留到后面的单元。

学习目标

foundation

先认入口

这一页关注网页进入数据管线时，哪些东西必须原样留下，哪些东西要先抽成 document。

先留证据

只要后面还要回到原始页面、抓取记录或抽取版本，就必须保留 source_url、charset、fetched_at、extractor_version 和 content_hash 这类字段。

对象链

trace

这条链只看四个可检查的状态：原始字节、解析后的页面结构、抽取后的正文记录、标准化后的 document。四个状态各有读写关系，混在一起就会丢掉回查入口。

获取入口

source

这一层先回答“你拿到的到底是什么”。有些来源给你的是网页字节，有些给你的是导出 CSV，有些给你的是 WARC 或 JSONL。不要一上来就把它们都叫成网页文本。

这一层不做什么

这里不判断内容好坏，也不做质量过滤。先把对象和记录留住，再决定值不值得保留。输入边界没定清，后面每一步都容易偏题。

正文抽取

mechanism

抽取要把网页里真正进入 document 的正文找出来，同时把来源和版本信息留住。导航、推荐阅读、广告、脚注和重复块常常混在同一页里，抽取器要给它们划清边界。

代码
def build_document(raw):
 dom = parse_html(raw.bytes, charset_hint=raw.charset_hint)
 main_text = extract_main_text(dom)
 doc = {
 "source_url": raw.url,
 "fetched_at": raw.fetched_at,
 "extractor_version": "main_text_v1",
 "text": normalize_text(main_text),
 }
 return doc

一个最小对照

原始页里可能有“首页 / 分类 / 登录 / 推荐阅读 / 正文 / 评论区”。抽取后，document 应该留下正文和必要元数据，避免把所有块原封不动存进去。

别把这里做成过滤

正文抽取关心“哪一块是正文”。质量过滤关心“正文值不值得保留”。把这两件事混在一起，会让坏抽取看起来像坏数据质量。

案例走读：从一个旧网页到 document

walkthrough

下面把前面的概念压进一个可检查的小案例。假设数据源是一批旧站点页面，页面来自 WARC 或离线 HTML 归档，编码标注不稳定，正文周围还有导航、登录入口、推荐阅读、评论和页脚。目标先产出一条可以回到原始页面、可以复现实验、可以交给后续过滤器的 document。

第一轮输出长什么样

第一轮 document 可以很朴素，但字段必须承担职责。source_url 负责回到原页，fetched_at 负责定位抓取批次，charset 负责解释解码路径，extractor_version 负责对照规则变化，content_hash 负责后续重复检测，text 才是模型最终可能看到的主体内容。

这一步最怕什么

最怕把“看起来能读”当作完成。页面能读，不等于 document 稳定；正文能看到，不等于元数据完整；当前样本正常，不等于下一批旧页面也能按同样规则处理。学习这一单元时，判断标准要落在对象、字段和回查路径上。

这个 JSON 示例故意保留了处理版本。版本字段会让很多后续问题变得可回答：同一 URL 两次抽取得到不同正文，先看 extractor_version；同一篇文章的哈希变化，先看标准化规则和空白处理；某批旧站点大面积乱码，先看 charset 记录和解析日志。没有这些字段，排查只能靠猜。

注意这张故障表的顺序。它把问题放回最早可能出错的对象层：字符异常先查获取和解析，主文异常先查抽取，重复异常先查标准化与哈希，来源缺失先查 schema。这个顺序会让排查成本下降，也能避免把入口层的坏记录误判成后续算法问题。

怎么把走读迁移到自己的数据

拿到一个新来源时，先挑 20 到 50 个样本人工走一遍这张链。每个样本都记录原始 URL、抓取时间、编码线索、抽取文本、标准化文本和最终 document。然后专门挑异常样本：乱码页、空正文页、超短页、导航占比高的页、重复标题页。只要异常样本能被字段解释，后面的批处理才值得扩大规模。

完成这个练习后，你应该能做两件事：第一，读懂一个数据管线日志时能指出它处理的是 raw、DOM、text 还是 document；第二，看到一个坏样本时能写出第一轮回查路径，而不是直接调过滤器、去重器或训练脚本。

抽样检查合格信号失败时动作能不能放大批处理
随机样本大多数 document 字段完整，正文可读补齐 schema 或修正基础解析字段不完整时不放大
异常样本乱码、空正文、导航混入都能定位到具体步骤把失败类型写进日志和回归样本集异常无法解释时不放大
重复样本同文不同 URL 的哈希关系可解释检查标准化位置和哈希计算规则哈希漂移时不放大

放行标准要写成可执行问题

“页面看起来差不多”不是放行标准。更稳的写法是：50 个随机样本里缺 source_url 的比例是多少，乱码样本能不能回到原始字节，抽取器版本有没有写入每条 document，导航块是否进入 main_text，哈希是在标准化前还是标准化后计算。每个问题都对应一个字段或一步处理动作，回答不出来就说明这批数据还没有准备好交给过滤和去重。

一个可操作的抽样记录可以这样写：样本 17 来自旧站点新闻页，raw 里 charset 是 gb2312，解析后 main 区域存在两个段落，抽取后混入页脚版权，标准化文本可读但 content_hash 已生成。这个样本的结论“抽取器回归样本 + 暂停放大批处理”。练习时你可以照这个格式写三条异常记录，每条都要指出读什么字段、查哪一步、修完以后用什么信号验收。修复后再跑同一组异常样本，确认旧问题没有重新进入 document。

这也是本单元和后续单元的交界线。完成获取、解析、抽取和标准化后，你手里应该有一批“可解释的 document”。下一单元才开始讨论哪些 document 应该留下，哪些要丢掉，以及过滤规则会带来什么偏差。把放行标准写清，后面的质量判断才不会替入口层的问题背锅。

编码标准化

stability

抽取完成以后，还要把字符层面的不稳定压下去。这里处理的是可比较、可检索、可重复的文本形态，不改写内容含义。字符解码错了，后面的去重、匹配和统计都会跟着错。

代码
import unicodedata

def normalize_text(raw_bytes, charset_hint=None):
 text = raw_bytes.decode(charset_hint or "utf-8", errors="replace")
 text = unicodedata.normalize("NFKC", text)
 text = " ".join(text.split())
 return text

这一步为什么放在抽取之后

先抽取正文，再统一字符，顺序更稳。你如果先做字符清理再找正文，很多页面的结构线索会被抹掉，抽取器反而更难判断哪一块是正文。

元数据记录

schema

document 是一条能回查的记录。正文之外的字段承担解释职责，它们决定你以后能不能说明这条记录从哪里来、是谁抽的、什么时候抽的、用什么规则抽的。

工具视角

source-to-tool

工具在这里承担可重复流程的角色。看工具时，只问它在链路里改了什么对象，不要把工具名当成结论。

工具名的正确用法

当你看到一个框架或案例，先把它放回 document、sample、token 或 shard 的哪一层，再看它的实现。层次清楚以后，工具名只是一张索引卡，答案仍然在对象层。

故障回查

diagnosis

这一段的目标很直接：看到问题时先回到哪一层。下面这张回查表，专门把症状和最先检查的对象对上，减少在模型、训练脚本或数据规模上绕路的时间。

案例与修正

worked

同一页网页，最少会有三种结果：原始页面、错误抽取后的 document、修正后的 document。你要会区分它们，不然会把“页面本来就这样”误判成“处理已经完成”。

论坛页和新闻页的差别

论坛页要更看重主帖、回复、楼层和引用嵌套；新闻页更看重正文、时间、栏目和作者。抽取规则如果对两类页面一视同仁，结果通常会偏。

这页的终点

做到这里，你应该能把一条网页记录变成可追溯的 document，并知道问题该回到哪一层。下一页才开始谈质量过滤、规则和偏差。

边界说明

boundary

这一页已经覆盖的

获取入口、正文抽取、编码标准化、元数据 schema 和回查顺序。你现在可以把 raw data 和 document 分开，也能解释为什么记录字段不能省。

下一页才会继续的

质量过滤、规则模型、偏差、去重、污染控制、数据配方和 tokenization。它们都建立在这里产出的 document 之上，不应该在这页里提前展开。

如果你在实作时发现 document 里还残留大量导航和脚注，先回到正文抽取；如果字符已经错了，先回到编码标准化；如果根本回不了原页，说明 schema 没留 source_url。顺序错了，后面每一层都会替你放大错误。
