# 术语与边界 {#terms}
<!-- tag: terms -->

术语
直觉解释
精确定义
常见误区


Exact dedup
一模一样的东西只留一份。
对规范化后的文本或样本算 hash，相同 hash 视为重复。
以为它能处理转载、改标题、改页脚和轻微改写。


Fuzzy dedup
内容很像的文档留代表样本。
用 shingles、MinHash、SimHash、embedding 或局部敏感哈希估计相似度。
以为阈值越低越稳，实际上会误删长尾知识。


Benchmark contamination
题目、答案或解法混进训练语料。
评测集的题干、答案、解析、测试文件或高重叠变体出现在训练数据里。
只查题干，不查答案和解析。


Leakage
答案在不该出现的阶段提前暴露。
训练、验证、测试、调参、人工标注和日志分析之间发生信息穿透。
把泄漏只理解成代码 bug，忽略流程和权限。


Split hygiene
按来源分组后再切训练和评测。
按文档簇、站点、时间、用户、仓库或题库 id 做 group-aware split。
随机逐行切分，看似公平，实则最容易串线。

$$\text{risk} = \text{repeat mass} + \text{benchmark overlap} + \text{split coupling}$$
 这只是一个判断顺序：先看训练内部重复，再看评测重叠，最后看切分有没有耦合。三个问题的处理对象不同，不能拿一个阈值一把收。

# 去重方法 {#dedup}
<!-- tag: dedup -->
<!-- layout: grid2 -->

为什么预训练特别怕重复
预训练按 token 消耗算成本。某段广告、导航栏、版权声明或代码模板重复出现很多次时，模型会在预算里反复学习同一种模式。重复会改变数据分布，让某些模板拿到不合理权重。
最先做的是 exact dedup。先做文本规范化，比如统一换行、空白、大小写、HTML 实体和 Unicode 形式，再对文档或段落计算 hash。速度快、解释清楚、误删风险低，但只能处理完全相同或规范化后完全相同的内容。

 normalize(text) → hash(text) → group by hash → keep one representative
 这条链路的关键不在 hash 本身，而在先把同一份内容的不同显示方式压回同一规范化结果。

## box info
### 判断句
exact dedup 只覆盖底线。网页转载时改了标题、广告位、时间戳、页脚或少量段落，完全 hash 就可能识别不出来。

### Fuzzy dedup 处理“看起来差不多”的文档

fuzzy dedup 的直觉是把长文档拆成短片段特征，再估计两个文档的特征集合有多重合。常见做法是 shingling 加 MinHash。shingle 是连续 n 个 token 或字符组成的小片段；Jaccard similarity 衡量两个集合的交并比；MinHash 用少量签名近似估计 Jaccard，避免海量文档两两比较。

这套方法适合网页、新闻转载、模板化页面和重复代码。难点不在公式，而在阈值和粒度：按文档去重保留完整上下文，但可能误删同主题不同文章；按段落去重更精细，但可能破坏文档结构；按 token 去重风险最大，容易把常见表达当成重复。

$$Jaccard(A,B)=\frac{|shingles(A)\cap shingles(B)|}{|shingles(A)\cup shingles(B)|}$$
 这里的重点是集合重叠，要求比单字符一致更宽。近重复不一定长得一模一样，但会在足够长的片段上反复重合。

三种去重粒度


**文档级**
适合删转载整页和镜像站。优点是保留结构；缺点是相似阈值难设，可能误删不同版本。


**段落级**
适合删页脚、版权、导航、重复模板和 FAQ 片段。优点是精细；缺点是需要重新拼接上下文。


**样本级**
适合训练样本已经切好的场景。优点是和训练输入一致；缺点是容易让跨样本重复漏掉。


经验顺序：先做文档级 exact dedup，再做文档/段落级 fuzzy dedup，最后对高风险来源单独抽检。

exact-dedup.py
复制

```python
import hashlib
import re
import unicodedata

def normalize(text):
 text = unicodedata.normalize("NFKC", text)
 text = text.lower()
 text = re.sub(r"\s+", " ", text).strip()
 return text

def fingerprint(text):
 return hashlib.sha256(normalize(text).encode("utf-8")).hexdigest()

docs = [
 {"id": "a", "text": "Model cards explain model limits.\n"},
 {"id": "b", "text": "model cards explain model limits."},
 {"id": "c", "text": "Data cards explain dataset limits."},
]

seen = {}
kept = []
removed = []

for doc in docs:
 fp = fingerprint(doc["text"])
 if fp in seen:
 removed.append({"id": doc["id"], "duplicate_of": seen[fp]})
 else:
 seen[fp] = doc["id"]
 kept.append(doc)

print("kept:", [d["id"] for d in kept])
print("removed:", removed)
```

这里 `a` 和 `b` 规范化后相同，所以只保留一份。注意：如果 `b` 多了一段广告或页脚，exact dedup 就可能漏掉，需要 fuzzy dedup。

# 去污染方法 {#contamination}
<!-- tag: decontam -->

benchmark contamination 指评测集信息进入训练语料。它不只包括完整题干；题目翻译、答案解析、排行榜页面、题库讨论帖、代码单测、示例输出和数据集 README 都可能携带答案。模型见过这些内容后，分数可能提高，但这不能证明它学会了泛化。

去污染要做的是保护评测独立性。通常会把评测集转成 contamination signatures：题干 n-gram、答案 n-gram、代码测试片段、问题 id、URL、数据集文件名等。再在训练语料中找 exact match、high n-gram overlap、MinHash 相似簇或语义检索命中。

题干污染
训练语料含有完整问题或高度相似改写。选择题、数学题、代码题最常见。


答案污染
题干不出现，但标准答案、解析、单元测试或排行榜讨论出现。更隐蔽，也更难通过题干匹配发现。


流程污染
评测集被用来调 prompt、选 checkpoint、调过滤阈值，测试集逐渐变成验证集。

contamination-check.py
复制

```python
bench = [
 {
 "id": "math-001",
 "question": "If a train travels 60 miles in 2 hours, what is its speed?",
 "answer": "30 miles per hour",
 }
]

train_docs = [
 {"id": "web-17", "text": "A solved example: If a train travels 60 miles in 2 hours, its speed is 30 miles per hour."},
 {"id": "web-22", "text": "Speed equals distance divided by time."},
]

def norm(text):
 return " ".join(text.lower().split())

hits = []
for item in bench:
 q = norm(item["question"])
 a = norm(item["answer"])
 for doc in train_docs:
 text = norm(doc["text"])
 if q in text or a in text:
 hits.append({"benchmark_id": item["id"], "doc_id": doc["id"]})

print(hits)
```

如果命中完整题干或标准答案，通常要从训练语料中剔除相关文档或整簇，并在评测报告中说明去污染规则。只删命中句子可能不够，因为同一网页往往还包含解析和答案结构。

# 泄漏控制 {#leakage}
<!-- tag: leakage -->

leakage 比 contamination 更宽。contamination 关注训练语料是否含评测信息；leakage 关注整个实验系统中，答案是否通过任何路径提前暴露。比如随机逐行切分同一本书，训练集里有前半段，测试集里有后半段；又比如开发者反复看测试集错题并调过滤规则，最后测试集已经参与了模型选择。

最稳的做法是 group-aware split。不要按行随机切，而要按来源组切：同一篇文档、同一网页簇、同一题库、同一用户、同一仓库、同一时间窗口，尽量只落在一个 split 里。对于代码和数学题，还要把测试文件、答案文件、解析文件、issue 讨论和镜像仓库视作同一风险组。

## box soft
### 隔离清单
- 训练集、验证集、测试集有冻结版本和不可变 manifest。
- 评测集 signatures 在训练前生成，不能训练后再补规则。
- 测试集不参与过滤阈值、数据配方、checkpoint 和 prompt 选择。
- 日志中不写入完整测试答案，或对能访问训练数据的人隔离权限。
- 每次剔除污染都有来源、规则、阈值、时间和样本数记录。

## box warn
### 最容易忽视的泄漏
你把 benchmark 题目拿来调数据过滤器，表面上没有把题目训练给模型，但你已经让测试集影响了训练数据选择。这会让测试集变成开发集。严谨做法是用公开开发集调规则，把最终测试集冻结并尽量少看。

$$\text{clean split} = \text{group-aware split} \land \text{frozen benchmark} \land \text{no test-driven tuning}$$
 这个式子表达的是工程判断顺序：先把分组切干净，再冻结 benchmark，最后禁止测试集反复参与调参。

### 审计顺序

先看这条样本属于哪一类：完全重复、近重复、污染命中、同主题知识，还是 split 泄漏。每一类对应的动作不一样。完全重复通常可以直接删簇；近重复要先比来源和完整度；污染命中要看题干、答案、解析和测试文件是否一起出现；同主题知识不能因为“看起来像”就删掉；split 泄漏要回到分组方案，不要拿删样本去掩盖切分错误。

一个健康的审计台账至少要回答五个问题：命中什么规则、命中的证据是什么、删掉了多少、保留了什么代表样本、后续要不要回滚阈值。只给出“删除成功”是不够的，因为后续你还要解释为什么某个来源被删得更多，或者为什么某一类 benchmark 分数突然变高。

### 失效链路：当“看起来干净”其实不干净

最常见的坏例子三点一起出错：exact dedup 通过了，fuzzy dedup 阈值设得偏松，去污染又只查了题干。结果就是训练集里留着答案解析，评测集里又混进了同主题问题，最后分数看起来正常，解释却完全不稳。

audit-flow.py
复制

```python
def review(sample):
 if sample.kind == "duplicate":
 return "check normalize() and exact hash"
 if sample.kind == "near_duplicate":
 return "check MinHash threshold and source quality"
 if sample.kind == "contamination":
 return "check frozen signatures and benchmark scope"
 if sample.kind == "leakage":
 return "check split unit, version freeze, and tuning policy"
 return "check logs, then inspect the next upstream stage"
```

这段伪代码的作用，是把“问题长什么样”翻译成“下一步先查哪一层”。现实里很多误判，都是因为把不同层的错误混在一起处理，导致修了半天没碰到根因。

# 症状到回查 {#bridge}
<!-- tag: bridge -->
<!-- layout: grid2 -->

## box
### 现场例子
你把一批网页文本跑完以后，看到某个代码题的验证分数突然抬高。抽检时又发现同一批数据里有题目解析、测试断言和论坛里的解题帖。这个时候先不要去动模型参数，先把问题拉回到数据边界：是不是污染、去重太松，还是 split 切错了。

## box warn
### 查回顺序
先查 benchmark signatures 是否在训练前冻结，再查近重复阈值是否过松，最后查文档簇是不是跨 split。这个顺序能把“分数高”拆成三类根因，避免用一个修补动作盖住所有问题。

| 症状 | 先看什么 | 再看什么 | 如果真出错，会发生什么 |
| --- | --- | --- | --- |
| 分数突然抬高 | benchmark signatures 是否冻结 | 训练语料是否含答案或解析 | 评测独立性被破坏 |
| 同来源删得更多 | 规则是否依赖站点结构 | 阈值和模板是否贴站点布局 | 来源偏差被放大 |
| 重复仍很多 | 规范化和 fuzzy 阈值 | 是否只做了 exact dedup | token 预算继续浪费 |
| 测试集不稳定 | split 单位是否是文档簇 | 同源样本是否跨 split | 训练和测试相互渗透 |

checklist.txt
复制

```text
1. signature frozen before training?
2. exact dedup done after normalize()?
3. fuzzy threshold reviewed by source bucket?
4. group-aware split by document cluster?
5. audit log includes rule version and sample ids?
```

这条桥接的作用，是把一条异常结果拉回到具体检查项。只要这五个问题里有一项答不上来，数据工程就还没到可以解释结果的程度。

# 边界与提醒 {#boundary}
<!-- tag: boundary -->
<!-- layout: grid2 -->

## box soft
### 这页不做什么
不提供一套“统一万能阈值”，也不把同主题知识自动当污染。是否删掉，最终还是要看对象、证据和任务。

## box warn
### 实际项目要额外检查
版权、隐私、合同、地区法规和组织政策都可能比技术判断更早地收窄边界。工程审计可以给出证据，但不能替代正式审批。

前一单元已经讲过质量过滤、阈值和偏差；这一单元把“重复、污染、泄漏”拆开，下一单元才进入来源比例和阶段配方。

# 练习 {#practice}

## box soft
### 怎么答才算会
先说对象，再说动作，再说记录。能把“删什么”和“为什么删”一起讲清楚，才算真的看懂这一页。

### 检查重点
选择题看对象边界；填空题看术语是否稳定；开放题看原因、后果和回查对象。答案不用长，但要能指向下一步检查什么。
