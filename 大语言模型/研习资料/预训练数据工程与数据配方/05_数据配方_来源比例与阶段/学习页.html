数据混合 / 阶段配方

数据配方：来源比例与阶段

这一页接在过滤、去重和泄漏控制后面，回答的是另一个更难的问题：当你已经有一批“能用”的数据时，应该怎么分来源、怎么定比例、什么时候换配方。这里先不追求万能答案，只追求一份能解释、能计算、能验证的数据配方。

课程地图上一单元下一单元

读完这一页，你应该能做什么

  • 你能把一个数据池说清楚：它贡献什么能力、有哪些噪声、为什么要进配方。
  • 你能区分库存占比、采样权重和训练 token 占比，不再把三者混成一张表。
  • 你能写出一个分阶段配方，并说明要用什么评测或消融来确认它是否真的有效。

学习目标

deep unit

先分清对象

看到网页、代码、论文、百科、论坛、合成文本时，能判断它们分别给模型补什么能力，不会只按文件类型分桶。

再定权重

能把库存规模、采样权重和训练 token 占比分开说，知道为什么小而重要的数据池可以被过采样。

最后看阶段

能解释为什么同一套来源在主体预训练、质量增强、退火收尾时要换配方，也能说出对应的验证方式。

主线图

recipe loop
来源池 → 能力贡献 → 质量权重 → 阶段系数 → 评测反馈

数据配方不是一张比例表，而是一个闭环。先按能力贡献把来源池分好，再把质量、目标和阶段写成权重，最后用评测和消融去检查这份配方到底在改善什么、牺牲了什么。

先记住这一句

库存多，不等于训练多；训练多，也不等于一定更好。真正要看的，是某个来源在当前阶段为目标能力提供了多少有效 token，以及它把别的能力挤掉了多少。

术语与误区

terms
术语 直觉 精确定义 常见误解
来源池 一个可单独抽样的数据来源。 如网页、代码、论文、百科、论坛、合成文本，每个池都记录规模、许可、过滤和版本。 只记网站名，不记清洗和分桶规则。
采样权重 抽得更频繁还是更少。 对训练采样器的偏好设置，可能高于或低于库存占比。 把权重当成来源本身的“质量事实”。
训练占比 训练时真正消耗的 token 比例。 模型实际看到的 token 分布，受库存、权重和阶段系数共同影响。 把硬盘里有多少数据直接当成训练占比。
阶段系数 某个阶段是否更偏向某类来源。 按训练阶段对来源施加的额外权重，如主体、增强、退火。 以为全程一套配方最稳。实际常常要随阶段调整。
消融 看改动到底带来什么变化。 在固定模型、预算和评测下，只改变来源、比例或阶段，观察结果。 把最终大模型分数直接归因给某个小改动。

来源与能力

source map

配方先从“来源能教什么”开始，而不是从“这个来源是不是干净”开始。干净只是门槛，能力贡献才是排位的依据。一个来源池如果只提供自然语言流畅度，那它就不该占满训练预算；如果它能带来代码、数学、结构化论证或稳定事实，那它就值得在某些阶段被提高权重。

来源池 主要能力贡献 常见风险 配方处理
通用网页 长尾知识、日常表达、实体关系、跨域连通。 模板页、SEO 垃圾、重复、语言混杂。 作为主体来源，但要靠过滤、去重和污染控制托底。
代码 语法、API 使用、程序结构、符号推理。 许可证复杂、自动生成代码、密钥泄漏、仓库重复。 按语言和仓库质量分层，常在代码能力阶段提高占比。
论文与教材 严谨概念、长链论证、数学表达、术语精度。 PDF 抽取噪声、版权边界、排版破碎。 更适合质量增强和领域阶段，不宜替代通用覆盖。
百科与参考资料 结构化事实、定义、实体描述。 风格单一、覆盖边界固定、时间点偏旧。 适合补稳定知识，但不能代替开放网页。
论坛与问答 问题分解、口语表达、排错路径、多轮上下文。 偏见、错误答案、隐私内容、攻击性内容。 适合补交互和问题解决能力，但要小心污染和偏差。
合成文本 稀缺格式补齐、结构化推理样式、可控任务模板。 同质化、回声放大、模式塌缩。 更适合特定阶段或特定任务，不宜当作默认主体来源。

来源表至少要写清什么

每个来源池都要能回到同一张记录：来源名称、采集时间、许可边界、语言、文档数、token 数、过滤规则、去重版本、质量分数、污染检查结果、采样权重。缺了这些字段，后面就很难解释模型为什么变了。

比例与权重

weights

比例先别写成库存表

库存占比只说明某个池在硬盘上有多少 token。训练占比才说明模型到底从这个池里看了多少 token。小而重要的池可以过采样，大而低效的池可以降采样。只要训练目标不同，比例就不该完全跟库存走。

一个常见错误是把“网页 80%、代码 10%、论文 10%”当成完整答案，却没有说这些比例按文档、样本还是 token 算，也没有说能不能重复抽样。这样的表格看起来整齐，实际上不能执行。

$$ p_i = \frac{n_i \cdot w_i \cdot s_i}{\sum_j n_j \cdot w_j \cdot s_j} $$

这里的 $n_i$ 是库存 token，$w_i$ 是来源权重，$s_i$ 是阶段系数。这个式子不是唯一方案，但它把“有多少”“想抽多少”“当前阶段要不要强化”三件事拆开了。

一个判断句

如果一份配方不能回答“为什么这个来源要抽这么多、抽多了会牺牲什么、如何验证收益”，它还只是清单，不是可执行的工程配方。

来源权重如何改变占比

固定库存：网页 900B token，代码 80B token，论文教材 120B token。这里的结果会随着权重变化而变化，目的是把“质量权重是决策”这件事看得更直观。

把目标写成配方

recipe

一份能执行的数据配方，通常不是“网页多一点、代码多一点”这种口语句，而是一份能被平台读懂的配置。你至少要写清楚来源池、权重、阶段、以及每一段的回退条件。下面这个简化配置只演示结构，不声称它对所有模型都最优。

简化配方示例
target: technical-doc-understanding
tokens_total: 100B
sources:
  web:
    inventory_tokens: 900B
    base_weight: 0.7
    stage_weights: [1.0, 0.8, 0.5]
  code:
    inventory_tokens: 80B
    base_weight: 2.0
    stage_weights: [0.8, 1.2, 1.0]
  papers_books:
    inventory_tokens: 120B
    base_weight: 1.6
    stage_weights: [0.6, 1.2, 1.5]
  forums:
    inventory_tokens: 40B
    base_weight: 0.8
    stage_weights: [0.7, 0.9, 0.6]
decision:
  primary_metric: "task_cluster_score"
  guardrails:
    - "no benchmark contamination"
    - "monitor language regression"
    - "recheck stage-3 overfit"

先看池里有什么，再决定配方能不能撑住目标 token 预算。

来源偏好0.7 / 2.0 / 1.6 / 0.8

权重不是质量本身，而是你希望训练采样器多看谁。

阶段系数[1.0,0.8,0.5] 等

同一来源在不同阶段可以被放大、压低，或者保持。

训练批次按阶段生成 batch

最后落到 batch 的不是“库存表”，而是“某阶段的实际抽样结果”。

对象 读什么 写什么 失效症状
来源池条目 规模、许可、语言、过滤日志 是否进入配方、进入哪一阶段 配方里有名字，训练器却找不到可用池

失败小实验

terminal lab

下面这个终端不是在跑真实训练，而是在模拟训练平台对配方做审查时会出现的三个常见告警。看输出时，先找它说的是“单位错了、阶段错了，还是记录不够”。

[audit] case-1
ERROR mix.unit is ambiguous: inventory share was written where training token share is required.
Hint: convert the table to token-based proportions before resubmitting.

这个实验在诊断什么

它诊断的不是模型能力，而是配方本身的可执行性。单位不清，训练器会读错；阶段不写，后段会默认沿用旧配方；记录太少，结果即使变了也说不清是来源、权重还是阶段造成的。

阶段与节奏

stages

主体预训练

这一段的目标是广覆盖和稳定学习。通用网页通常是主体，但还要混入代码、百科、书籍和论文，避免模型只学到一种语气或一种结构。

质量增强

当模型已经学到基本语言和事实分布后，可以把高质量、长文档、专业材料或人工筛选数据的权重抬高，让后段 token 更集中地塑造能力。

退火与收尾

训练后段常会同时降低学习率和噪声来源，把注意力放回更干净、更目标相关的数据上。这里不是补规模，而是减少最后一段对表示和输出风格的扰动。

阶段 配方重点 可提高的来源 要监控的风险
0-60% 覆盖、稳定、低重复 通用网页、百科、代码、公开参考资料 低质网页占比过高、语言失衡、重复样本浪费预算
60-90% 能力塑形、专业补强 代码、教材、论文、领域文档 通用能力下滑、过度偏科、旧知识被挤掉
90-100% 收尾、收敛、去噪 更干净的数据、目标领域材料、短而稳定的高质量池 收尾阶段过度放大某种风格，或者误把收尾当成补规模

配方追踪

trace

配方最容易出错的地方，不是写错一个百分比，而是写清楚了比例，却没写清楚这份比例在不同阶段会怎么变。下面这张表把“读什么、写什么、失效时看什么症状”放在一起，方便你回查。

环节 读什么 写什么 错了会看到什么
来源盘点 数据池、许可、规模、语言、过滤日志 来源清单、版本号、可用池 配方表看起来完整，实际却找不到可复现的池
权重分配 目标能力、池大小、质量分桶、重复率 来源权重、过采样倍数、阶段系数 库存很大但训练里几乎没看到，或者小池被抽得过头
阶段切换 当前 token 预算、任务目标、评测趋势 阶段配方、学习率、来源重排 同一配方硬拖到最后，后段信号变得越来越弱
结果回看 统一评测、消融结果、污染和泄漏检查 保留、回退、重采样、进入下一轮实验 平均分没变，但某个任务簇明显退化，却没人追问为什么

从一份配方回到执行层

如果你在训练平台上维护配方，最少要能回答这四件事：哪个来源池被读到了、该池的权重是多少、这一阶段是否改过系数、结果变坏时应该回看哪张表。这个顺序比记住一个漂亮比例更重要。

完整走一遍

walkthrough

把前面的概念放回一个具体任务里，最容易看出它们到底是不是一回事。下面这份目标是“做一个技术文档理解模型”，总预算 100B token。它不需要极强的代码生成，但需要稳定的术语理解、结构化表达和一定的问题解决能力。这个目标很普通，所以最适合拿来练配方：它逼你同时考虑广覆盖、专业材料和阶段切换，而不是只盯着某一种来源。

第一步：先写来源，不先写百分比

很多人一上来就写网页 50%、代码 20%、教材 20%、论坛 10%。这个顺序反了。正确做法是先列出你手里有哪些来源池：通用网页、代码仓库、论文教材、百科参考、论坛问答。然后问每个池能补什么能力。网页补通用覆盖，代码补符号推理，论文教材补长链论证，百科补稳定事实，论坛补真实问题和对话风格。你先知道每个池的角色，才知道它应该在配方里占多重。

如果某个池只会重复同一种表达，却不能补目标能力，就不要因为它“看起来很多”而给太高占比。配方不是库存竞赛。一个来源池只有在它能改变模型行为时，才有资格在表里留下位置。

第二步：把角色写成权重

假设你得到一个初版：网页 0.7、代码 2.0、论文教材 1.6、百科 1.0、论坛 0.8。这个版本的意思不是“代码最干净”，而是“在当前目标下，代码对能力贡献更高”。再加上阶段系数，网页在早段可以是 1.0，到了后段可能降到 0.5；论文教材在后段可能上升到 1.5，因为你要把前面学到的通用表达收束到更稳定的知识结构里。

这一层最容易犯的错，是把权重写成道德判断，好像“代码就是高级，网页就是低级”。不是这样。权重是对训练资源的分配决策，不是对数据身价的评语。它只对当前目标和当前阶段负责。

阶段 来源怎么排 为什么这样排 错了以后最先坏什么
早段 网页为主，少量代码和百科 先学语言基本盘和事实分布，避免一开始就偏科 模型会更快地学到某种风格，但通用覆盖不稳
中段 提高代码、论文教材和领域文档 开始塑形，补结构化表达和专业知识 如果切得太猛，通用网页被压掉，很多日常能力会掉
后段 更干净的高质量池 + 适量目标领域 收尾时减少噪声，把输出风格压稳 如果后段还沿用早段配方，模型会在噪声里收尾

第三步：识别失败症状，而不是只看均分

当你把网页压得太低，模型可能会显得更“专业”，但事实题、常识题和长尾表达会掉；当你把论坛问答放得太后，聊天会更顺，但事实保持和引用稳定性会掉；当你把代码过采样得太多，符号推理会变好，但自然语言风格会开始发硬。每一种退化都不是空洞的“变差”，而是可以在任务簇里被具体看见的症状。

这就是为什么本页一直强调“验证方式”要跟配方绑在一起。你不是只在做一个表格，你是在做一个会改变模型行为的决策系统。没有反馈的配方，只能算初稿。

如果你现在要把这套方法迁移到新的模型，先别急着追最优。先复写一份这样的走读表：目标是什么、来源池有哪些、每段权重怎么变、最可能坏掉的是哪一簇任务。写完以后，再去看下一页的小模型消融。那一页会告诉你，今天这份配方到底是把能力做出来了，还是只是把某类分数抬高了。

案例推演

cases

案例 1：代码库存小，但目标是代码能力

如果代码库存只有 8%，训练占比却想给到 16%，这并不矛盾。矛盾的是你是否知道这 16% 会从哪里挤出来，以及重复暴露是否会伤到通用表达。做法通常是先保留通用网页，再把高质量代码池在中后段适度过采样，同时监控自然语言和数学任务是否被挤压。

案例 2：论坛问答放到后 10%，聊天更顺但事实题掉了

这时不要急着把论坛删掉。先看它的质量分布、是否和 benchmark 近似、是否在收尾阶段被放大过头，再看事实题下降是不是因为通用网页被压缩。问题可能出在阶段位置，也可能出在比例，而不是出在来源本身。

案例 3：法律领域继续预训练

如果目标改成法律领域，来源就不能只看通用网页了。要增加法规、判例、注释性材料和专业教材，同时把许可、时效、地区差异和隐私边界写进配方记录。这里的风险不是“数据不够”，而是“来源对了但版本不对”。

边界说明

boundary

这里先不追的事

这页先不展开最优混合搜索、长期课程学习、动态重加权、最优控制和多阶段自动调度。它们会在更后面的实验设计和系统化流水线里回来。现在先把“来源-权重-阶段-验证”四件事写顺。

这页不承诺的事

这里不会给出一个放之四海而皆准的比例表。不同模型、预算、语言、任务和发布时间点，都会把配方改写。你要学的是怎么解释和验证配方，不是死记一组百分比。

下一页会把这份配方拿去做小模型消融和统一评测。到那时，才真正知道某个来源池是贡献能力，还是只是填满了 token。

怎么接到下一页

operation bridge

这一页的任务，不是把配方写漂亮，而是把它写到可以送去做实验。下一页你会看到的，不是更长的说明，而是更短的决策：哪些来源池进入了对照组，哪些阶段权重被固定，哪些变量被单独改动。换句话说，05 页负责把目标翻成配方，06 页负责把配方翻成对照实验。

送去 06 页前先准备什么

先保留一份来源池清单，再保留一份阶段权重表，然后把当前配方里最关键的两个可变项标出来：一个是来源比例，一个是阶段位置。这样到了下一页，你就知道应该只动哪一项，而不是整份配方一起乱改。

常见第一失败

最常见的失败不是“结果不好”，而是“你说不清哪一项导致了变化”。如果你没有先把来源、权重和阶段拆开，下一页的消融就会变成一次大杂烩训练，测到什么都像是对的，却什么都解释不清。

这页留下的作业

把你自己的配方写成三行：来源池、阶段系数、验证指标。三行都写完，再去看下一页的消融设计。只要能把这三行写清楚，你就已经比“凭感觉调比例”多走了一步。

如果阶段系数写错，会先坏哪里

最先坏掉的通常不是最终分，而是任务簇的方向：代码可能变强一点，但事实题和通用表达开始下滑；或者聊天更顺，但结构化理解不稳。这个时候别先改模型结构，先回到配方表，看是不是中后段把某个来源放得太重了。

给平台看的最短记录

如果你要把这页内容抄进训练平台，最少写四行就够：来源池名称、基础权重、阶段权重、验证指标。下面这段可以直接当作起点，再根据你的任务目标改数值。

四行记录模板
source_pools: web / code / papers_books / forums
base_weights: 0.7 / 2.0 / 1.6 / 0.8
stage_weights: early=1.0/0.8/0.6/0.7, middle=0.8/1.2/1.2/0.9, late=0.5/1.0/1.5/0.6
validation: task_cluster_score + contamination_check + language_regression

Q9. 诊断：某团队说“我们把网页降到 40%，模型就一定更聪明”。这句话有什么问题？

参考答案 / 评分要点

至少要指出三件事：没有说明“更聪明”对应什么评测；没有说明被网页让出的 token 去了哪里；没有控制其他变量；没有说明是否分阶段；没有考虑语言、领域和污染风险。只批评“太绝对”不算完整。

scenario transfer比例

Q10. 设计：给一个技术文档理解模型分配 100B token 的三类来源比例。

参考答案 / 评分要点

回答里要有三类来源、比例之和、以及为什么不能 100% 都给专业材料。最好能说明通用网页负责广覆盖，教材或论文负责结构化表达，代码或问答负责工具和问题解决能力。只写一串百分比不够。

comparison概念区分

Q11. 比较：质量权重和阶段系数有什么不同？

参考答案 / 评分要点

质量权重更像全局偏好，说明某个来源池在整体训练里该被抽得多还是少；阶段系数则说明当前阶段是否要额外放大或缩小这个池。一个是“这池重要不重要”，一个是“现在这个阶段要不要更强调它”。

synthesis评测

Q12. 解释：为什么平均评测分不足以判断配方好坏？

参考答案 / 评分要点

平均分会掩盖方向性变化。一个配方可能让代码和推理更强，却牺牲通用语言、某些语种或事实题；也可能是 benchmark 污染导致分数虚高。好答案要说明“平均值背后有哪些分组差异”。

把配方放进训练过程，而不是只给一张静态比例表。

快速回看 阶段 = token 预算 + 目标 + 来源比例 + 监控指标。

error diagnosis退化

Q14. 诊断：后 10% token 加入大量论坛问答，模型聊天更自然但事实题下降。你会怎么查？

参考答案 / 评分要点

先查论坛问答本身是否质量稳定、是否和 benchmark 近似、是否在后段被放大过头；再查是否因为后段挤掉了高质量网页或百科；最后用对照实验拆成“来源问题、阶段问题、比例问题”三条线分别验证。

scenario transfer领域数据

Q15. 迁移：如果目标是法律领域继续预训练，来源和风险会如何变化？

参考答案 / 评分要点

应包含法律法规、判例、注释性材料、专业教材或合同文本等来源，同时说明地区差异、版本时效、隐私和许可边界。最好还能提到：领域数据要提高，但通用覆盖不能完全消失。

Q16. 设计一个小模型消融实验来验证“论文教材占比提高有用”。

参考答案 / 评分要点
