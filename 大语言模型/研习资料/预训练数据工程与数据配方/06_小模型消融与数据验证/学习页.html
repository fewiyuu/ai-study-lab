先记一句：

小模型短跑给的是方向，不是定稿。它最适合帮你筛掉明显不值得继续的规则。

术语与误区

terms
术语直觉精确定义常见误解
小模型消融先用便宜模型试数据改动。在较小参数量和较小训练预算下，只改变一个数据变量，比较不同数据版本的结果。误以为小模型结果可以直接替代最终大模型结果。
统一评测所有候选版本都按同一标准比。固定 benchmark、prompt、样本上限、打分方式、checkpoint 选择和聚合方法。误以为只要换一个“更难的”评测集就更客观。
规则验证确认一个规则删的是噪声。把过滤或阈值规则写成可审查实验，量化移除比例、样本内容和训练后变化。误以为删得越多越好，或者规则名字专业就等于可信。
早期信号低成本方向提示。小模型、短 token、有限任务集给出的相对信号，用于筛选候选规则。误以为 early signal 就是最终结论。

核心机制：同一把尺子才有比较

controlled

做消融时，关键让差异只来自一件事。先写出明确假设，再造基线和实验版本，最后在同一训练协议下比较结果。只要模型大小、训练 token、tokenizer、prompt 或 checkpoint 选得不一样，归因就会开始变形。

可信比较 = 同一模型 + 同一预算 + 同一采样 + 同一评测

前四项都要固定住，变化只留给数据变量本身。这样看见的差异，才更像规则带来的差异。

统一评测：先把尺子锁死

benchmark

小模型短跑最怕噪声。任务太难、方差太大、prompt 太敏感时，分数差异很可能只是随机波动。统一评测给不同版本一个共同坐标系，让你能比较相对变化，而不是在不同脚本里来回追分。

FineWeb 这类公开案例更适合拿来说明 early-signal 的挑选方式：任务之间要能区分方向，重复 run 的波动要可接受，分数还得能随着训练进展继续变化。只看一个榜单均分通常不够，至少要看任务簇和负向异常。

规则验证：从直觉到证据

rule

过滤规则通常先从样本里看出问题：短行太多像导航，重复行太多像模板，语言分数太低像杂乱来源，字符模式异常像爬虫残留。直觉能帮你起步，但规则要不要留，得看它删掉了什么、保留了什么、训练后有没有副作用。

验证一条规则，不要只问“删了多少”。还要问“删掉的是谁”“被删样本有没有同类内容”“这条规则会不会把长尾知识、少数语言或论坛表达一起削掉”。

验证台账：一条规则怎么留下证据

state

实验做完以后，最容易丢的是状态。很多人会记住分数，却忘了这条规则到底读了什么、删了什么、写回了什么。对数据验证来说，台账是能不能复查、能不能回退的底稿。

如果台账里没有阈值、样本审查和回写决定，下一轮实验就只能从头猜。把状态写全，后面的单元才有继续推进的基础。

一条坏记录长什么样

rule_name: short_line_ratio_filter
version: 0.67 threshold
decision: keep_candidate
problem: removed-sample audit missing
follow_up: rerun later

这段日志看起来像结束了，其实最关键的证据缺了一半。没有 removed 样本审查，就不知道这条规则删掉的是噪声还是能力来源；没有回退条件，后面的复核也不知道该怎么处理冲突。

所以，实验记录里最少要保住三件事：规则版本、样本审查、回退条件。少一件，后面的判断就会变钝。

如果你现在就要把本节内容拿去做别的规则验证，先从这张清单开始，再去写样本审查和回退条件。它会直接把你从“复述概念”推进到“能做实验记录”。

完整走读：一条规则怎么从草案走到决定

step

下面用同一条“短行比例”规则走一遍完整流程。先把它当成草案，不把它当答案。你要看的，是它怎么经过样本、记录、评测和回退条件的检查，最后才变成可以留在配方里的东西。

记录模板

experiment log

这个走读的作用，是把“我觉得规则对”换成“我能说清它在哪里通过、在哪里失败、下一步要查什么”。如果你说不出回退条件，就说明实验还没真正闭环。

案例推演

worked

例 1：验证“短行比例”过滤规则

你发现很多网页抽取后只剩导航词、菜单词和目录词，打算把“短行比例过高”的文档过滤掉。先别急着写最终规则，先把实验拆成能比较的版本。

实验草案pseudo config
hypothesis:
 metric: short_line_ratio = short_lines / total_lines
 threshold: remove if short_line_ratio >= 0.67
 expected_noise: navigation, category lists, template fragments

datasets:
 baseline: base_filter + independent_minhash
 experiment: base_filter + independent_minhash + short_line_ratio_filter

training:
 model: same 1-2B class architecture
 tokens: same 28B-token budget
 tokenizer: same tokenizer
 seed: same sampling plan

evaluation:
 tasks: same early-signal benchmark suite
 report: aggregate, task cluster deltas, negative outliers
 inspect: removed / kept / borderline samples

这个设计的关键，是实验版本只多一个短行规则。若评测提升，同时被移除样本真的以导航残留为主，这条规则才值得保留。若提升只出现在单个任务，或者删掉了大量问答、论坛和代码片段，就要把阈值调松或直接回滚。

如果你手里只有一张分数表，没有 removed / kept / borderline 的切片，最容易犯的错就是把“目录残留被删掉了”误判成“所有短句都该删”。这时候先回到样本，再看边界样本有没有把有用问答一起卷进去，然后再决定阈值该往回挪还是继续保留。

例 2：为什么“全局去重更强”可能更差

直觉上，重复越少越好；但全局去重并不总是更稳。如果把整个 Common Crawl 当成一个大池子做全局去重，老 crawl 里的内容会被大幅削掉，留下来的可能更偏、更窄，甚至更不像真实训练分布。FineWeb 的公开案例也提醒过：按 crawl 分开去重，往往比把整份语料一次性去重更稳。

策略直觉收益隐藏风险验证方式
跨全部 crawl 全局去重最大化移除重复网页。老 crawl 被过度削薄，样本分布偏移。按 crawl 训练小模型，比较任务簇和样本质量。
每个 crawl 内独立去重先清掉同一 crawl 内的大重复簇。跨 crawl 的重复仍会留下来。短跑看质量，长跑看重复对训练效率的影响。

常见错误

failure

错误 1：把一次短跑直接当定稿

小模型短跑本来只是在给方向。它可能提前暴露明显坏掉的规则，也可能把一些小收益放大成看起来很稳定的提升。稳一点的做法，是先看短跑方向，再看样本审查，再决定要不要拉长预算。

如果你在短跑里看到小幅增益，却没有样本审查、负向任务检查和重复 run 复核，这个结论还不能进入正式配方。

错误 2：只看平均分

平均分最容易说服人，也最容易藏住问题。一个规则可能把单项任务抬高，却把另一个任务簇拉低；也可能把分布变窄，让少数长尾任务持续退化。

读表顺序是先看任务簇，再看异常点，最后才看平均。平均分只负责压缩信息，不负责解释原因。

错误 3：把规则验证做成“删多少”的比赛

删得多只说明规则更激进，先看它删掉的是噪声还是长尾能力。激进规则更容易误杀论坛体、问答、少数语言、代码片段或领域长尾。验证时如果没有样本审查，删得多只是一种强度展示，不是质量证据。

错误 4：同时改多个旋钮

把阈值、来源比例、训练步数和评测集一起换掉，结果再好也很难归因。你看到的变化可能来自某个规则，也可能来自更多 token、更好的 checkpoint 选择，甚至只是评测脚本换了版本。

单变量消融的价值就在这里：它把“可能是谁”缩成“最可能是谁”。

观察到的信号可能原因下一步检查动作
总分小涨，任务簇分裂规则偏向单一任务，或评测噪声偏高。检查负向 outlier 和重复 run。先标记为候选，不进最终配方。
某任务大涨，其余基本不动规则可能只对这个任务相关的文本有效，也可能出现污染。看训练语料是否包含评测模板或相似文本。做污染检查，再决定是否保留。
删得很多，但分数几乎没变规则可能主要删掉了可有可无的文本，或者样本审查不充分。抽样看 removed / borderline / kept。优先考虑放宽阈值。
短跑通过，长跑回落小预算里看起来有利的信号，在更长训练中不稳。比较更长 token 下的任务簇变化。回滚或只保留局部场景。
指标和人工判断冲突自动指标可能抓住了局部模式，人工样本看到的是覆盖和语义。把指标、样本和阈值放在同一页复核。按人工审查修正规则，再重跑。

> [!NOTE] 这一步的判断标准：

> 如果你已经能说出哪类信号出现时该继续、哪类信号出现时该回退、回退时先看什么，说明你已经把小模型消融从做实验推进到做判断了。

这里先不追

boundary

先放一边的内容

先别追大模型最终配方、完整训练系统、分布式调参、超长上下文训练和后训练细节。它们会在后面的单元里回来；这一页只把小模型验证的判断链条搭稳。

比如短行阈值样本先跑一轮，接着看 removed / kept / borderline，再决定是回滚阈值还是保留规则。

什么时候要升级验证

如果短跑结果方向很稳、样本审查也干净，就可以进入更长 token 的确认。如果方向不稳，先别急着堆更多训练，先回头看样本和协议哪里出了问题。
