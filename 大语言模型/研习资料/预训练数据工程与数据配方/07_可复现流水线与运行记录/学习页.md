# 术语与误区 {#terms}
<!-- tag: 先分清对象 -->

术语
直觉解释
精确定义
常见误解


流水线
把原始数据加工成训练语料的一整套步骤。
由阶段、依赖、输入输出、参数和恢复边界组成的可执行规范。
只要有一串脚本就算流水线。脚本只是执行片段，流水线还要说明谁先谁后、失败后怎么接。


Run
按同一规范真实跑出来的一次事件。
在特定代码、配置、输入、环境和资源条件下完成的一次处理过程。
只记最后生成了什么文件。真正的 run 还要能回答“为什么是这一批文件”。


Manifest
产物目录的机器可读清单。
列出路径、校验和、字节数、样本数、token 数、阶段、来源和时间的记录。
把 README 当 manifest。README 是给人看的，manifest 还要能被程序校验。


run_id
一次运行的唯一标签。
把配置、日志、产物、检查点和复现实验连接起来的主键。
run_id 只是日志编号。它的作用是把证据串成一条链。


latest 指针
总是指向“最新结果”的目录名。
可变指向，常用于操作，但不适合当复现的唯一依据。
最新就等于最可信。实际上 latest 最容易漂移。


资源账本
这批数据花了多少机器和时间。
按阶段记录 CPU、GPU、内存、磁盘、网络、时长、失败重试和成本估算。
只记总耗时就够。定位瓶颈时，你需要知道卡在哪一步。

# 流水线与契约 {#pipeline}
<!-- tag: 输入 / 输出 / 依赖 / 症状 -->

$$
 \text{shard\_count} = \left\lceil \frac{\text{total\_tokens}}{\text{shard\_tokens}} \right\rceil
 $$
 这里的意思很直接：先估算总 token，再按 shard 上限切分。这个公式本身不复杂，但它提醒你，tokenizer 版本一变、过滤阈值一变，最终分片数也会跟着变，所以 shard 数不能只靠“记得大概是多少”来复现。

阶段
输入
输出
必须记录
最常见症状


snapshot
原始下载列表、时间范围、权限。
固定的输入快照和校验和。
URL、文件名、sha256、下载时间、版本号。
“latest” 路径被替换后，复跑得到另一批输入。


parse
WARC、HTML、PDF、Markdown 等原始格式。
统一的 document 对象。
解析器版本、正文抽取规则、编码处理、丢弃字段。
正文抽出来了，元数据却不见了。


filter
标准化 document。
保留集、拒绝集、原因统计。
模型版本、阈值、规则、保留率、拒绝原因分布。
看起来更干净了，实际上误杀了大量有效文本。


dedupe
过滤后文本和指纹。
去重后的语料和重复命中统计。
指纹算法、窗口、跨 crawl 还是 crawl 内去重、保留策略。
评测看起来涨了，训练预算却被重复样本吃掉。


shard / tokenize
清洗好的文档和 tokenizer。
shard、token 数、字节数、样本数。
tokenizer 版本、分片上限、随机种子、排序方式。
同一批文本换了 tokenizer，token 数却完全变了。

## box
### 为什么要分阶段保存中间产物
因为不是所有错误都应该从头重跑。解析坏了，先看原始输入；去重过头了，先看保留样本；分片出了差异，先看 tokenizer 和排序。中间产物保存得越清楚，越容易判断问题发生在哪一步，而不是把所有失败都归到最后一段脚本。

## box warn
### 这里先别急着追的东西
先不要钻进分布式调度、文件系统性能、并行抢锁和容器编排。那是后面的工程细节。现在只要先把“谁读了什么、写了什么、出了错怎么看”讲明白就够。

# 故障定位 {#failure}
<!-- tag: 先找源头，再修症状 -->

症状
优先怀疑什么
先看哪里
怎么修


重跑后结果和上次不一样
输入快照漂移、latest 指针变化、随机种子没锁。
run_id、输入清单、时间戳、下载路径。
把快照固定下来，不要直接依赖 latest。


token 数变了，分片数也变了
tokenizer 版本或排序方式变了。
tokenizer、分词参数、分片上限。
把 tokenizer 版本写进记录，别只记脚本名。


manifest 有文件名却校不过
文件损坏、截断或被替换。
sha256、bytes、生成时间、传输日志。
重新生成 manifest，不要只改路径。


断点续跑后少了一批 shard
完成标记和实际产物不同步。
completion marker、shard 清单、重试日志。
先核对中间产物，再决定跳过还是重跑。

## box info
### 这里要养成的习惯
先找数据从哪里进来，再找它在哪一步被改写，最后才看失败看起来像什么。这样做的好处是，你不会因为一个表面错误去修另一个阶段的症状。

# 完整复跑链 {#replay}
<!-- tag: drift 发生时先别急着改代码 -->
<!-- layout: grid2 -->

## box
### 先把 drift 说成一个可检查的故事
很多“结果变了”的问题，实际上输入、依赖或恢复点中的某一项偷偷变了。最常见的误判是：你只对比了最终 shard 数，或者只看了一个总 token 数，却没有回到原始快照和运行记录。这样一来，你看到的差异可能只是输入漂移，而不是处理逻辑本身出了问题。
复跑时要先问三个问题：第一，这一轮是不是同一份输入快照；第二，解析器、过滤器和 tokenizer 有没有换版本；第三，断点续跑时是不是跳过了不该跳过的部分。只要这三个问题中有一个回答不清楚，就不要把结果当成同一件事的复验。
真正有价值的复跑，“每个看起来一样的地方都真的一样”。这句话听上去抽象，但落到工程上其实很直接：快照固定、环境固定、记录固定、检查点固定，四个条件都成立，复跑才有比较意义。

## box warn
### 先锁哪一层，再看哪一层
第一层锁的是输入。输入不稳，后面的任何统计都不可信。第二层锁的是执行。代码 commit、依赖版本、配置 hash、随机种子、tokenizer 和过滤模型，一项都不能只靠记忆。第三层锁的是恢复边界。断点续跑时，跳过哪些 shard、哪些产物已写、哪些产物只写了一半，都必须写在可读的记录里。
很多团队会在口头上把“复现”理解成“再执行一次”。这会忽略一个事实：再执行一次并不等于在同一条件下执行。只要输入快照变过，或者 `latest` 指针换过，或者中间文件被人手工补过，你看到的就已经是另一轮实验了。把这件事讲清楚，才能知道该修输入、修代码，还是修流程。
所以这页的重点判断顺序：先把对象和条件固定住，再去看差异，最后才决定要不要重跑。顺序一旦反过来，团队就会拿着错误的证据修错误的层。

常见变化
表面现象
优先核对
应该怎么解释


输入快照被替换
输出文件数量和 token 数一起变了。
snapshot URI、sha256、下载时间。
这通常喂入对象已经变了。


tokenizer 升级
总 token 数上升或下降，但文本看着没变。
tokenizer 版本、词表、特殊符号和空白规则。
这表示分词口径变了，不能把新旧数字直接并排比较。

path、sha256、bytes、documents、tokens、stage。

## box info
### 把结论写回记录
当你确认问题来自输入漂移时，记录里应该写的是“固定快照后重跑”，而不是“重新执行”。当你确认问题来自 tokenizer 或过滤模型变化时，记录里应该写的是版本和参数变化，而不是“结果不稳定”。当你确认问题来自恢复标记失配时，记录里应该写的是“清点中间产物并回滚重算”，而不是“再试一次”。
这里的区别很关键，因为复盘为了让下一次复跑更便宜、更快、更可解释。记录写得越准，后面越不会重复踩同一个坑。

## box info
### 一条能用的判断线
如果你能回答“这一批输入是什么、谁改了什么、哪一步能继续、哪一步必须重算”，那就说明这条流水线已经具备最基本的复跑能力。反过来，如果你只能说“我再跑一次看看”，那说明你还没有把运行、证据和恢复边界分开。
这也是为什么这页看起来像在讲日志，其实是在讲控制变量。控制变量做得越好，后面的配方验证、消融对比和审计判断才越站得住。

# 写回与交接 {#handoff}
<!-- tag: 把一次运行变成可继续的工作 -->
<!-- layout: grid2 -->

## box
### 复盘之后要写什么
复盘不是停在“知道原因”这一步。真正有用的是把原因写回记录，把修正动作写成下一次可以照做的约束。比如你发现 `latest` 指针不稳定，就应该把“只允许使用固定 snapshot URI”写进 runbook；你发现 tokenizer 版本会影响结果，就应该把 tokenizer 名称、版本和词表 hash 写进 manifest；你发现断点续跑会跳过未完成 shard，就应该把完成标记和文件列表一起核对。
这类写回有一个很实用的标准：接手的人不需要先问你“当时到底怎么想的”。如果他能根据记录知道输入、版本、恢复点和异常处理，说明你的交接已经够用了。否则，复盘只是在你脑子里完成，团队并没有真正得到可复制的知识。
所以记录里最好同时保留三种东西：事实、判断和动作。事实是发生了什么，判断是为什么这样解释，动作是下一次该怎么做。三者缺一个，后面的人就只能猜。

## box warn
### 别把运行记录写成事后说明
事后说明往往只解释结果，不解释边界。运行记录则要解释边界本身：哪些输入被固定了，哪些阶段被跳过了，哪些产物是重新生成的，哪些只是复用的。只有把这几项分开，别人才能判断变化究竟来自数据、代码还是恢复策略。
如果你把所有东西都写成一句“已处理完毕”，那下一次遇到问题还是会回到原点。因为“完毕”没有告诉任何人哪一步完成了，也没有告诉任何人哪一步只是看起来完成了。

复制

```yaml
run_id: 2026-06-21-001

 - input_snapshot_fixed: true
 - tokenizer_version_pinned: true
 - manifest_complete: true

```

这类交接模板的价值是把“我已经看过了”变成“下一轮会怎么接着做”。

写回项
要记录什么
最怕漏掉什么
漏掉后的后果


输入证据
snapshot URI、sha256、下载时间、权限来源。
固定输入和临时输入的差别。
下一轮会以为自己在复跑，其实是在换输入。


执行证据
代码 commit、依赖版本、参数、随机种子。
会改变输出的隐性对象。
结果漂了却找不到真正的变化源。


产物证据
manifest、token 数、bytes、documents、checksum。
校验字段和规模字段。
文件存在却无法证明它完整、正确、属于哪次运行。


恢复说明
marker、checkpoint、跳过条件、回滚条件。
哪些 shard 已完成、哪些只写了一半。
断点续跑会把“部分完成”误当“全部完成”。

## box info
### 这套模板怎么真正用起来
最实用的做法每次跑完都立刻填。先填输入快照，再填执行版本，再填产物，再填恢复边界，最后再写判断和动作。这样做有个直接好处：你不会把“应该记住的东西”拖到第二天，因为第二天通常就只剩模糊印象了。
当你要把这份记录发给别人时，重点让别人知道哪里要复查、哪里可以复用、哪里不能跳过。好的交接不会把人带进你的思路里，而是把他带到证据旁边。只要接手者能根据这份记录重新找到输入、版本和恢复点，它就算完成了。
如果团队里多人都在做数据流水线，最好统一使用同一份模板。这样一来，日志格式、命名方式和恢复标记都更容易对齐。你以后再看别人的 runbook，也能很快找到关键字段，而不是每次都重新解码一套写法。
更具体一点，交接时最好让接手的人只做三件事：先核对输入快照是不是同一份，再核对运行版本和参数是不是被固定，再核对恢复边界和 completion marker 有没有冲突。如果这三件事里有一个不清楚，就不要把它记成“已完成”，而是写成“待复查”。这样记录会直接推动下一轮动作，而不是只留下一个结束标记。
你也可以把这套模板当成一个最小的周报结构：本周改了哪些对象、哪个对象影响最大、哪次重跑最值得回看、下一次要固定什么。这样做的目的让团队对数据处理的变化有共同语言。共同语言一旦稳定，排查速度通常会快很多。
如果一份记录还要再往前走一步，可以把它和变更单、代码提交和产物目录互相链接。这样一来，你看到一条 run_id，就能顺着它找到输入、脚本、输出和复查意见。链条一旦连起来，后续出现问题时就不会每次都从目录树最上层重新翻。
最后再补一个很实用的判断：如果你手里的记录只能回答“结果是什么”，那它还不够；如果它还能回答“为什么是这个结果、下一次怎么重跑、哪里不能跳过”，才算真正进入可复用状态。这个门槛不高，但一旦跨过去，很多临时排查就会变成稳定流程。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-group: 一、概念辨认 -->
<!-- quick-review: 快速回看 可复现不是“我记得怎么跑”，而是别人能按记录复查或复跑。 -->

### 1 选择

**Q:** Q1 选择：哪项最能代表流水线规格？

- [ ] . 最终训练 shard 的下载地址。
- [ ] . 阶段依赖、参数、输入输出和资源约束的配置。
- [ ] . 运行结束后写给同事看的总结邮件。

Answer: . 阶段依赖、参数、输入输出和资源约束的配置。
Explain: 流水线规格要先定义阶段依赖、参数和输入输出约束，最终文件只是输出，不是规格本身。

### 2 判断

**Q:** Q2 判断：只要 git commit 一样，数据处理就可复现。

- [ ] 对
- [ ] 错

Answer: false
Explain: 代码一样不等于输入、依赖、tokenizer 和过滤模型一样。任何一个会改输出的对象漂了，结果都会变。

### 3 填空

**Q:** Q3 填空：每个 shard 一行的机器可读产物清单通常叫做什么？


### 4 选择

**Q:** Q4 选择：下面哪个变化最可能导致同一文本 token 数改变？

- [ ] . tokenizer 版本改变。
- [ ] . 运行记录标题改变。
- [ ] . 负责人名字改变。

Answer: . tokenizer 版本改变。
Explain: token 数主要受 tokenizer 版本、词表和分词规则影响。标题、负责人或汇报措辞不会改变 token 化结果。

### 5 诊断

**Q:** Q5 多选：哪些对象应该进入版本记录？

- [ ] 代码 commit
- [ ] 原始数据快照
- [ ] 质量过滤模型权重
- [ ] tokenizer
- [ ] 配置 hash

Explain: 判断标准很简单：这个对象会不会改变最终产物，或者会不会影响别人复查同一结果。会的话就该进版本记录。别把“我没改脚本”误当成“所有输入都没变”。

<!-- end-practice-group -->

<!-- practice-group: 二、流水线设计 -->
<!-- quick-review: 快速回看 每个阶段都要说明输入、输出、不变量和失败恢复策略。 -->

### 6 排序

**Q:** Q6 排序：把以下阶段排成合理顺序。

Explain: 合理顺序通常是：声明原始快照 → 解析与标准化 → 质量过滤 → 去重 → 配方混合与分片 → tokenize 或 token 统计。你可以把 tokenize 提前到分片前后，但必须明确它依赖什么、产生什么、是否会影响配方判断。

### 7 设计

**Q:** Q7 设计：为 `language_filter` 阶段写输入、输出、参数和指标。

Explain: 好答案会把输入写成标准化 document，输出写成保留集和拒绝统计，参数写清语言识别模型版本、阈值和保留语言，指标写成通过率、拒绝原因分布和抽样复核结果。只写“过滤掉非目标语言”不够。

<!-- end-practice-group -->

<!-- practice-group: 三、记录与恢复 -->
<!-- quick-review: 快速回看 manifest 让机器校验产物，run record 让人理解判断。 -->

### 8 选择

**Q:** Q8 选择：manifest 中最适合校验文件内容是否损坏的字段是？

- [ ] . path
- [ ] . sha256
- [ ] . owner

Answer: . sha256
Explain: 路径存在不等于内容正确。sha256 才是内容级校验字段。

### 9 判断

**Q:** Q9 判断：资源账本只需要总耗时，不需要按阶段拆分。

- [ ] 对
- [ ] 错

Answer: false
Explain: 不同阶段的瓶颈不一样。只看总耗时会让你不知道是解析慢、去重慢，还是分片慢。

### 10 补字段

**Q:** Q10 补字段：下面这条 manifest 缺什么？

Explain: 至少要补 run_id、sha256、bytes、tokens、stage、source_mix、生成时间中的四个。解释时要说清字段如何帮助校验、追溯或复跑，比如 sha256 用来确认内容没坏，stage 用来说明它属于哪一步，source_mix 用来说明它来自哪个配方。

### 11 写作

**Q:** Q11 写记录：为一次失败重试写 4 行运行记录。

Explain: 四行至少要包含：失败发生在哪一步、失败原因是什么、重试改了什么、重试后输出是否变化。最好再补一条受影响范围或责任人。重点让别人能决定“要不要从头跑、从哪一步接着跑”。

<!-- end-practice-group -->

<!-- practice-group: 四、迁移与审计 -->
<!-- quick-review: 快速回看 没有可复现的数据版本，数据配方实验就很难被解释。 -->

### 12 解释

**Q:** Q12 数据配方：为什么 mixture recipe 也需要版本化？

Explain: recipe 版本化是为了知道训练时到底用了哪些来源、各占多少、在哪个阶段被放大或缩小，以及这些变化对应了什么结果。没有版本号，后续的消融对比就只能“差不多记得”，无法判断到底是哪一步带来了收益。

### 13 清单

**Q:** Q13 审计：你拿到一个公开数据集，只看到下载链接和总 token 数。你会追问哪 6 件事？

Explain: 至少要问：数据来自哪里、有没有快照、怎么过滤、怎么去重、用的 tokenizer 是什么、有没有污染或泄漏检查、许可是什么、manifest 在哪。真正关键的一项通常是快照或 tokenizer，因为它们直接决定别人能不能重建同一批 token。
