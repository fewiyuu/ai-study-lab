这页先解决什么

overview

机器学习任务里，训练只是手段，分数也不是第一步。先把问题切成三层：模型会不会把训练资料学好，学好之后能不能迁移到没见过的数据，迁移不好时是模型太弱、优化没跟上，还是太容易记住训练细节。P03 就是把这条排查顺序讲清楚。

最容易错的顺序

training loss 还高时就谈 overfitting，通常会把问题看反。先判断模型有没有把训练资料学下来。

现象优先判断下一步动作

先看三种信号

signal

这节课最重要的是三种信号：training loss、validation loss、public / private 分数。它们读的是不同阶段的状态，不该互相替代。

资料怎么切

data

训练资料通常有 (x, y)。x 是输入，y 是答案。测试资料通常只有 x，没有公开答案。于是训练、验证、测试这三份资料的角色就不能混。

public / private 的额外一层

竞赛或作业里的 testing set 往往又分成 public 和 private。public 分数先给你看，private 才是最后更稳的结果。public 不是免费验证集，反复按它挑模型，也是在训练它。

资料包含什么用途常见误用
training setx, y训练参数拿它证明泛化
validation setx, y选模型和超参数无限次调到过拟合
public test通常只有 x粗看提交表现反复拿来挑模型
private test隐藏答案最终评估训练或调参时使用

诊断树

decision

如果结果不好，先问 training loss。这个顺序很重要，因为它能立刻把问题拆成两类：模型连训练集都学不好，还是模型在训练集上很好、在新资料上不好。

training 高

training 低，validation 高

public 好，private 差

两者都低

现象更像什么原因动作
training loss 高model bias 或 optimization模型太弱，或者参数没被找到先比较不同模型，再调训练设置
training 低，validation 高overfitting模型记住了训练细节augmentation、regularization、early stopping
public 好，private 差public 过拟合反复看 public 分数，模型贴到了 public 集合回到 validation，减少试错次数

读日志

logs

结果不好时，日志比感觉可靠。先看 training loss 有没有持续下降，再看 validation loss 和 public score 有没有开始分叉。日志不是装饰，它是你判断“问题在哪一层”的第一现场。

读错会怎样

只盯 training，会漏掉 overfitting；只盯 public，会把 public 试成训练信号；只盯单个 epoch 的波动，又会把正常噪声当成问题。日志要读趋势，不是读情绪。

训练日志样例复制

epoch 01: train_loss=1.42 valid_loss=1.45 public=1.38
epoch 05: train_loss=0.88 valid_loss=1.02 public=0.97
epoch 10: train_loss=0.41 valid_loss=0.97 public=0.94
epoch 15: train_loss=0.19 valid_loss=1.24 public=0.88

$$

gap = validation\_loss - training\_loss

$$

gap 持续变大，通常表示模型开始记细节了；public 和 private 的差值变大，则表示你可能把 public 试成了选择器。

读什么读之前读之后出错时的症状下一步
train_loss 曲线只看到一个最终分数知道模型有没有学会训练还高却误判成过拟合先查 bias / optimization
valid_loss 曲线只有训练结果知道何时开始分叉把早停点选晚了检查 overfitting 或切分
public / private只有 leaderboard知道是不是试错过头public 好但 private 差回到 validation

Training 高时，先分 bias 和 optimization

bias

training loss 高，不代表一件事。第一类原因是 model bias：模型本身就太弱，函数集合里没有足够好的解。第二类原因是 optimization failure：模型其实能表示更好的解，但训练没有把它找到。

optimization failure

模型够强，但训练方法没把好解找出来。可能是学习率、初始化、结构、梯度传播或训练技巧的问题。更强的模型却训练得更差，这时优先怀疑优化，而不是先怀疑模型不够大。

$$\theta^*=\arg\min_{\theta} L(\theta)$$

这条式子只是在说“找让 loss 最小的参数”。如果可选的模型族太窄，就会卡在 model bias；如果模型族够宽却没找到好点，就会落在 optimization failure。

比较项model biasoptimization failure
问题本质函数集合太受限函数集合里有好解，但没找到
常见现象简单模型和复杂模型都学不好更强模型反而更难训好
下一步改模型、改特征、改任务定义调学习率、初始化、优化器和结构

训练低、新资料高，才叫 overfitting

overfitting

Overfitting 的关键训练集和新资料表现分裂。模型在训练资料上已经足够低 loss，但对 validation 或 testing 的表现开始变差。这表示它学到了训练中的偶然细节，没抓住能迁移的规律。

training loss

validation / testing loss

欠拟合

训练和新资料都差。常见原因是模型太简单、feature 不够，或者还没训好。

合适

训练和新资料都不错，但仍要检查边界样本和分布变化，不要只看平均分。

遇到 overfitting，先想两条路

fixes

第一条路是增加有效资料。很多任务里不一定能额外收集新数据，但可以做 data augmentation。第二条路是限制模型弹性，也就是 regularization 方向：让模型别把训练里的偶然细节都记下来。

regularization

包括 early stopping、weight decay、dropout、减少无用 feature 或改更合适的模型结构。限制太少会过拟合，限制太多又会回到 model bias。

方法作用适合什么现象注意点
data augmentation增加合理变体training 好、新资料差变换后 label 必须还对
early stopping在 validation 变差前停下训练继续降、验证开始升需要可靠 validation set
限制模型降低可钻空子的空间模型太容易记住细节别把关键特征也删掉

Validation 的位置

validation

最稳的模型选择方式，是从训练资料里切出 validation set。训练只看 training set，模型选择只看 validation set，最后再去碰 testing set。这样你才知道结果到底是模型真的更好，还是你刚好把某个测试集试顺了。

更稳的做法

把 public 当反馈，把 validation 当主要依据，把 private 当最终检验。这样你在调参时不会把 leaderboard 当验证集用。

怎么选 early stopping 的点

看 validation loss 的最低点，不看 training loss 的最低点。训练集继续变好并不稀奇，关键是验证集什么时候开始转坏。只要 validation 开始连续上升，就该考虑停下，而不是继续期待它自己会回头。

validation 切分示例复制

from sklearn.model_selection import train_test_split

X_train, X_valid, y_train, y_valid = train_test_split(
 X, y, test_size=0.1, random_state=42
)

# 训练只用 training set
model.fit(X_train, y_train)

# 选模型时看 validation loss
valid_score = evaluate(model, X_valid, y_valid)

Leaderboard 为什么会骗人

public/private

如果你提交很多次，总会有某个模型碰巧在 public set 上看起来很好。这个模型未必真学到了规律，可能只是撞上了 public set 的偶然性。public 第一不等于 private 第一，更不等于真实部署最好。

观察提示

尝试越多，越容易选中一个只是在 public set 上碰巧好的模型。

public leaderboard 是反馈，不是免费验证集。

代码与判断

code

下面这段代码把诊断顺序写成一个简单判断器。它的作用是提醒你：训练集、验证集和 public 分数应该分开读，别把它们混成一个信号。

代码
def diagnose(train_loss, valid_loss, simple_train_loss, public_score=None):
 if train_loss > 1.0:
 if train_loss > simple_train_loss:
 return "复杂模型训练 loss 比简单模型还高：先查 optimization。"
 return "training loss 高：先查 model bias 或 feature 是否不足。"

 if valid_loss > 1.0:
 return "training 低但 validation 高：优先查 overfitting。"

 if public_score is not None and abs(public_score - valid_loss) > 0.5:
 return "public 和 validation 差异大：检查切分和分布。"

 return "当前结果较稳，继续做错误分析和边界检查。"

反例代码复制

代码

# 更稳：先按 validation score 选择，public 只做有限参考

常见误区

mistakes

把所有坏结果都叫 overfitting

training 也差时，先怀疑 bias 或 optimization，不要直接跳到过拟合。

把 augmentation 当万能按钮

变换后 label 不对，或者分布和真实任务差太多，augmentation 反而会把模型带偏。

下一步怎么接

bridge

这一节解决的是任务排查。下一节 P04 会接着看训练为什么会停住，以及为什么 “gradient 接近 0” 只能说明到了 critical point，不能直接说明是 local minima。你现在先把 P03 的判断顺序练稳，再去看更细的优化地形。

三步自检

audit

每次改完一个模型，先把结果写成三句话：training 怎么变，validation 怎么变，public/private 有没有明显分叉。这样你不会把一个实验做成情绪判断，也不会把一个提交当成结论。

失败时怎么收尾

如果 training 和 validation 都差，先回到 bias / optimization；如果 validation 开始变坏，就停在最低点附近；如果 public 好 private 差，就减掉无效提交，回到自己的 validation 依据。

看到什么说明下一步
train 下降，valid 也下降模型还在学习，暂时不用急着停继续跑，盯住是否开始分叉
train 下降，valid 上升过拟合开始冒头early stopping、regularization、augmentation
public 好，private 差你把 public 试得太熟了回到 validation，减少提交次数

一个完整例子：日志怎么把问题看出来

worked

假设你在做一个分类任务，前 5 个 epoch 里 train_loss 从 1.42 掉到 0.88，valid_loss 也跟着从 1.45 掉到 1.02。这个阶段说明模型至少开始学会了，training 和 validation 还在一起往下走，问题不算大。到了第 10 个 epoch，train_loss 已经到 0.41，但 valid_loss 还停在 0.97；第 15 个 epoch，train_loss 继续到 0.19，valid_loss 反而升到 1.24。这个分叉就是典型信号。

错了会怎样

如果你把这个分叉误判成 bias 或 optimization，就会继续加大模型或乱调学习率，结果只会让训练集更好看、验证集更难看。正确动作应该是回到 validation，做 early stopping、regularization，或者补 augmentation。

$$

train\_loss \downarrow,\quad valid\_loss \uparrow

$$

这组方向相反的箭头，就是过拟合最常见的外观。它意味着两条曲线已经分叉。
