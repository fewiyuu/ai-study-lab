# 本节的概念地图 {#goals}
<!-- tag: map -->

神经网络训练不是每次都把全部训练资料拿出来算一次完整 loss。更常见的做法是先把资料分成若干个小批次，每次用一个小批次计算 loss 和 gradient，然后更新参数。这个小批次通常叫 mini-batch，课程里简称 batch。

当所有训练资料都被模型看过一遍，我们称为一个 epoch。每个 epoch 开始时常会把资料重新打乱，这叫 shuffle。Shuffle 的意义是让哪些样本被分在同一个 batch 里不断变化，避免模型长期沿着固定的小样本组合更新。

Batch size 不是一个纯粹的工程参数。小 batch 带来更多 update 和更 noisy 的 gradient；大 batch 带来更稳定的 gradient 和更好的并行效率。真正的训练效果来自这些因素的合力，所以 batch size 是需要调的 hyperparameter。

training data→

epoch

直觉每次只抽一小包资料来估计整体下降方向，像用局部样本给参数指路。
正式说法Mini-batch SGD 用一个 batch 的 empirical loss 近似全训练集 loss 的 gradient。
关键边界Batch 的噪声不只是误差，它有时会让优化跳出不好的停滞区域。

# Batch、epoch、shuffle：训练循环到底在做什么 {#batch}
<!-- tag: mini-batch -->

假设训练集有 60,000 笔资料，batch size 设为 1,000。模型不会一次看完 60,000 笔再更新，而是先拿 1,000 笔算一个 batch loss，再算 gradient，更新一次参数；接着拿下一批 1,000 笔，再更新一次。60 个 batch 都用过后，才算完成一个 epoch。

如果 batch size 改成 1，那么一个 epoch 里会有 60,000 次更新。每次更新只看一笔资料，方向会很不稳定，但更新很频繁。反过来，full batch 是 batch size 等于整个训练集大小，一个 epoch 只更新一次，方向稳定但更新次数极少。

Shuffle 则发生在分 batch 之前。常见做法是在每个 epoch 开始时重新打乱训练资料，再切成 batch。这样第一轮同一个 batch 里的样本，下一轮不一定还在一起。这个小动作能降低固定样本顺序带来的偏差，让训练路径更接近随机抽样。

$$
batch 数量 = ceil(N / B)
 N 是训练资料总数，B 是 batch size。一个 epoch 的 update 次数约等于 batch 数量。
$$

小交互：batch size 改变一个 epoch 的更新次数
资料数 N**60000**
batch size B**1000**
更新次数60
梯度噪声中


**中等 batch**
一个 epoch 有多次更新，方向比 batch size=1 稳，但仍保留一定随机性。
观察重点：batch size 变大时，更新次数下降；batch size 变小时，噪声上升。

| 概念 | 直觉 | 正式作用 | 常见误解 |
| --- | --- | --- | --- |
| mini-batch / batch | 每次拿一小包资料更新 | 用局部样本估计 gradient | 以为它只是为了省内存 |
| batch size | 每小包有多少资料 | 控制更新频率和 gradient 噪声 | 以为越大越接近真相所以越好 |
| epoch | 所有资料看过一遍 | 衡量训练轮次 | 以为一个 epoch 只会更新一次 |
| shuffle | 每轮重新打乱资料 | 改变 batch 组合，降低顺序偏差 | 以为不影响优化路径 |

# 效率反直觉：大 batch 单次更新不一定慢 {#speed}
<!-- tag: parallelism -->

直觉上，batch size 从 1 变成 1,000，好像计算量应该增加 1,000 倍。这个直觉没有考虑 GPU 并行。对许多矩阵运算来说，GPU 可以同时处理大量样本，所以 1,000 笔资料一起算 loss 和 gradient，未必比 1 笔资料慢 1,000 倍。

课程用 MNIST 这类影像分类任务说明过这个反直觉：当 batch size 从 1 增加到 10、100、1000 时，单个 batch 的计算时间可能差不多。直到 batch size 大到超过硬件并行能力，时间才明显增长。

因此要分清两个时间尺度。第一个是“更新一次参数要多久”，大 batch 在 GPU 上不一定吃亏。第二个是“跑完一个 epoch 要多久”，这取决于一个 epoch 需要多少次 update。batch size 小，update 次数多；batch size 大，update 次数少，所以一个 epoch 反而可能更快。

## box info
### 单次 update
受 GPU 并行能力影响。中等范围内，大 batch 不一定比小 batch 慢很多。

## box warn
### 完整 epoch
还要乘上 update 次数。小 batch 的每次更新便宜，但次数可能非常多。

$$
epoch 时间 ≈ 每次 update 时间 × ceil(N / B)
 B 增大时，每次 update 时间可能只缓慢增加，但 update 次数会明显减少。
$$

# Noisy gradient：噪声为什么可能帮助 optimization {#noise}
<!-- tag: optimization -->

Full batch 用全部训练资料计算 gradient，方向更稳定，但它每次都沿着同一个全局训练 loss 更新。如果这个 loss surface 在某个 critical point 让 gradient 变成零，普通 gradient descent 就可能停住。

Mini-batch 不一样。每个 batch 只代表训练集的一部分，所以每个 batch 看到的是略有差异的 loss function。某个 batch 对应的 loss 在当前位置 gradient 为零，不代表另一个 batch 的 loss 也为零。换一个 batch，参数可能又得到新的移动方向。

这就是 noisy gradient 的一个重要价值：它不是单纯“方向不准”，也可能是一种扰动。这个扰动能让训练路径不那么死板，有机会离开某些停滞点或窄小区域。课程里强调，实际结果常常小 batch 的 noisy update 在 optimization 上反而可能占优势。

小图：稳定路径与 noisy 路径


稳定路径可能停住
noisy 路径仍可能移动


**不要把 noisy 只理解成坏事**
小 batch 的方向更乱，但这种随机性有时能让参数绕开局部停滞。
它解决的是 optimization 路径问题，不是让模型容量变大。

# 小 batch 与泛化：sharp minima 和 flat minima 的直觉 {#generalization}
<!-- tag: generalization -->

Batch size 影响的不只是 training。即使通过调 learning rate 等方式，让大 batch 和小 batch 都达到相近的 training accuracy，testing 表现也可能不同。课程提到的大量实验中，小 batch 往往更容易得到较好的 testing 结果。

一种常见解释是 sharp minima 与 flat minima。Sharp minima 像狭窄峡谷：在 training set 上 loss 很低，但参数或资料分布稍微偏一点，loss 就可能大幅上升。Flat minima 像宽盆地：training loss 也低，但附近一大片区域都还不错，因此对 training/testing 的差异更稳。

为什么小 batch 可能偏向 flat minima？因为 noisy update 让参数路径更容易从窄小峡谷里晃出去。一个很窄的 sharp minima 不容易困住带噪声的路径；更宽的 flat minima 才更容易让它停下来。这个解释不是绝对定理，但它提供了理解泛化差异的实用直觉。

小交互：training / testing mismatch 下的地形差异

宽盆地：平移后仍低


**flat minima**
如果 testing loss 相对 training loss 有一点偏移，宽盆地里的参数仍可能保持较低 loss。
观察重点：泛化不是只看 training loss 多低，还看附近区域是否稳。

| 地形 | training 表现 | testing 风险 | 和 batch 的直觉联系 |
| --- | --- | --- | --- |
| sharp minima | 当前位置可以很低 | 稍微偏移就可能变差 | 大 batch 稳定路径可能更容易落入窄谷 |
| flat minima | 当前位置也可以很低 | 附近仍保持较低 loss | 小 batch 噪声可能更容易停在宽盆地 |

# Momentum：不只听当前 gradient，也保留过去方向 {#momentum}
<!-- tag: momentum -->

Vanilla gradient descent 每一步只看当前位置的 gradient，然后往反方向走。Momentum 加了一份“过去移动方向”的记忆：当前更新方向由当前 negative gradient 和上一轮 velocity 共同决定。这里的 velocity 优化算法保存的一组状态变量。

为什么这有用？如果连续几步 gradient 都指向类似方向，momentum 会累积出更大的移动，帮助参数更快穿过平缓区域。如果当前位置 gradient 很小，甚至短暂变成零，过去的 velocity 仍可能让参数继续移动，从而越过 saddle point 或小的 local minima。

Momentum 还有一个需要小心的地方：它不是无条件越大越好。保留太多过去方向，可能在该转弯的时候冲过头；保留太少，则和普通 SGD 差别不大。所以 momentum 系数也是 hyperparameter，常见记号是 `β` 或 `λ`。

$$
vt+1 = β vt - η ∇L(θt)
 θt+1 = θt + vt+1
 β 控制保留多少过去方向，η 是 learning rate。
$$

小交互：当前梯度与过去方向合成
momentum β**0.80**
当前梯度强度**0.35**
过去方向0.80
当前梯度0.35
合成更新0.65


**保留明显过去方向**
当当前 gradient 变弱时，过去方向仍会贡献移动量；这就是 momentum 能穿过平坦或短暂停滞区域的直觉。
观察重点：β 越大，过去方向权重越强，但也越可能冲过该转弯的位置。

# 最小代码：mini-batch SGD 与 momentum update {#code}
<!-- tag: code reading -->

下面的代码把训练循环的语义拆开。重点看三个位置：每个 epoch 先 shuffle；每个 batch 算一次 loss 和 gradient；momentum 版本额外保存 velocity，并把它加入参数更新。

正确示例：mini-batch SGD 的训练骨架python

```text
for epoch in range(num_epochs):
 indices = shuffle_indices(len(train_data)) # 每个 epoch 重新打乱
 for batch_ids in split(indices, batch_size):
 x, y = train_data[batch_ids], train_label[batch_ids]
 loss = model.loss(x, y) # 只看这个 batch 的 loss
 grad = compute_gradient(loss, model.params)
 model.params = model.params - lr * grad # 更新一次参数
```

这段代码里的 `loss` 当前 batch 的 loss。正因为每个 batch 不同，gradient 方向会有随机性。这个随机性可能让路径更曲折，也可能让优化更有弹性。

正确示例：加入 momentum 的更新骨架python

```text
velocity = zeros_like(model.params)

for epoch in range(num_epochs):
 indices = shuffle_indices(len(train_data))
 for batch_ids in split(indices, batch_size):
 loss = model.loss(train_data[batch_ids], train_label[batch_ids])
 grad = compute_gradient(loss, model.params)

 velocity = beta * velocity - lr * grad # 过去方向 + 当前负梯度
 model.params = model.params + velocity # 用合成方向移动参数
```

`velocity` 的初始值通常是 0。第一步时 momentum 还没有历史方向，更新和普通 SGD 很像；从第二步开始，上一轮移动方向会进入当前更新。把式子展开，会发现当前更新含有过去多个 gradient 的加权影响。

错误示例：把 momentum 写成只改 learning ratepython

```text
# 错：这只是把步长放大，没有保存过去方向
model.params = model.params - (lr * beta) * grad
```

错误示例的问题在于没有状态变量。Momentum 的核心是保存并更新 `velocity`；如果每一步只把 `lr` 乘上某个常数，就没有“过去方向”的记忆，也无法体现穿过平坦区或短暂停滞点的作用。

# 常见误区和修法 {#mistakes}
<!-- tag: pitfalls -->

| 误区 | 错因 | 后果 | 修法 |
| --- | --- | --- | --- |
| batch 只是为了省内存 | 只看到一次不装全部资料的工程侧。 | 忽略 batch size 对更新次数、噪声和泛化的影响。 | 把它放回 optimization 过程理解。 |
| batch size 越大越好 | 以为 gradient 越稳定就一定训练越好。 | 可能得到较差 training 或较差 testing。 | 把 batch size 当 hyperparameter 调。 |
| 小 batch 一定更快 | 只看单次更新，不看完整 epoch。 | 低估小 batch 的大量 update 成本。 | 同时比较单次 update 时间和 epoch 更新次数。 |
| noisy gradient 只是坏事 | 把噪声等同于错误。 | 无法解释小 batch 训练和泛化优势。 | 记住噪声有时能帮助离开停滞或窄谷。 |
| training 好就代表 testing 一定好 | 忽略 sharp / flat minima 的差异。 | 在相同 training accuracy 下仍可能泛化差。 | 关注地形宽度和 training/testing mismatch。 |
| momentum 就是调大学习率 | 没理解 velocity 状态变量。 | 代码写不出真正的历史方向累积。 | 保留 velocity，并用 β 控制过去方向影响。 |
