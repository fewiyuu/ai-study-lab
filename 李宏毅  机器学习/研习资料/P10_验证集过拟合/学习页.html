为什么用了 validation set，结果还是 overfitting？

很多人以为：只要我不用 training set 选模型，而是用 validation set 选模型，就不会 overfit。P10 要拆掉这个误解。Validation set 能减少直接对 training set 调参的风险，但如果你反复试很多模型、很多超参数，再用同一个 validation set 选最好的，validation set 本身也会被“过拟合”。

课程把这个过程重新写成 P08 的理论语言：在 training set 上训练出多个候选模型 h₁*、h₂*、h₃*，这些候选模型组成 H_val。用 validation set 选择其中 loss 最低的模型，本质上就是在 H_val 这个小模型空间里，对 validation set 做一次模型选择。

本节的概念地图

map

Validation set 的用途是帮你在多个训练好的候选模型之间做选择。你先用 training set 训练不同模型或不同超参数配置，得到若干候选函数；再用 validation set 计算它们的 loss，选择 validation loss 最低的那个。

问题在于：只要你在一组数据上最小化 loss，这个过程就带有 training 的味道。普通训练是在大模型空间里选参数；validation selection 是在候选模型集合 H_val 里选模型。候选越多，越可能有人只是刚好在 validation set 上表现好。

因此，validation set 不是 test set。Validation set 可以被你反复查看、反复调参而污染；test set 应该保留到最后，只用于最终估计。如果把 leaderboard 当成 validation 反复提交，也会把 leaderboard 过拟合掉。

D_train→ train h₁*, h₂*, h₃*→ H_val→ D_val 上选最小 loss→ final model→ D_test 评估

直觉

从很多模型里挑 validation 分数最高的，可能只是挑中了最会讨好 validation 的模型。

正式说法

Validation selection 是在 H_val 上做 empirical risk minimization。

关键边界

Validation set 能用于模型选择，但不能被当作无限次试错的真实世界代理。

Training、Validation、Test 各自负责什么

data split

Training set 用来拟合模型参数。例如神经网络 weights、bias 都从 training set 学来。这个阶段会直接最小化 training loss，所以 training set 最容易被 overfit。

Validation set 用来做模型选择，例如选网络层数、learning rate、batch size、正则化强度、训练轮数。它不直接更新 weights，但它会影响你最终选择哪个模型，所以它仍然参与了决策。

Test set 用来做最终估计。理想情况下，你在所有模型和超参数都确定之前，不应该看 test set。看得越多，它越不再是未知世界的代理。

资料集用途可以反复看吗主要风险
Training学习参数可以，但要正则化和验证参数过拟合
Validation选择模型和超参数有限度可以模型选择过拟合
Test最终估计泛化表现不应该反复看评估被污染

Validation selection：先训练候选，再用验证集挑一个

model selection

课程里的例子有三个模型族：h₁、h₂、h₃。你先在 training set 上分别训练它们，得到 h₁*、h₂*、h₃*。星号表示“在自己的模型族里，经过 training set 训练后得到的最佳版本”。

接着你不会直接看 training loss 来选，因为 training loss 容易偏乐观。你会把三个候选拿到 validation set 上算 loss，谁的 validation loss 最低，就选谁去做最终测试。

这比只看 training set 好，但不是完美。因为 validation set 也只是从真实世界抽出的一小份样本。某个模型可能刚好适合这份 validation，而不是真的适合所有未来资料。

H_val 视角：validation 也是一种“训练”

H_val

P08 说，training 是在一个候选函数集合里，找出某个资料集上 loss 最低的函数。现在把这句话套到 validation：我们也有一个候选集合 H_val，里面是已经训练好的候选模型；我们也有一个资料集 D_val；我们也在找 validation loss 最低的函数。

所以 validation selection 可以看成一种带引号的 training。它不是用 gradient descent 更新 weights，但它确实用 validation set 做了选择。只要选择发生，泛化风险就会出现。

如果 H_val 很小，比如只有 3 个候选，validation overfitting 风险通常较低。如果你试了极大量架构和超参数组合，例如层数 1 到 10、每层宽度 1 到 1000 的所有组合，那么 H_val 会变得巨大，validation set 就更容易被过拟合。

为什么用了 validation 仍会 overfit

validation overfitting

Validation overfitting 的机制和 training overfitting 相似。Training overfitting 是模型空间太大，训练集又有限，于是模型能找到只适合 training set 的函数。Validation overfitting 是候选模型太多，validation set 又有限，于是你能挑到只适合 validation set 的候选。

这不是说 validation set 没用。它非常有用，因为它把“学习参数”和“选择模型”分开了，减少直接看 training loss 做决策的偏差。问题是，它不是无限可靠。每多试一次模型、每多根据 validation 分数改一次策略，你都在让 validation set 多参与一次选择。

Leaderboard 也是类似。你把模型反复提交到公开榜，看分数再调模型，公开榜就变成 validation set。最后隐藏 test 或最终评测变差，并不神秘：你已经对公开 leaderboard 做了模型选择过拟合。

风险升高

反复看 validation 分数，试大量模型，基于一次小涨幅改策略。

风险降低

提前限定搜索空间，保留最终 test，只在必要时看 validation。

判断一个实验流程是在合理使用 validation，还是已经把 validation 当训练目标。

更稳的实验设计

workflow

第一，先规划搜索空间。不要在没有原则的情况下无限试模型。你可以限定候选架构、learning rate 范围、训练轮数和调参次数，把 H_val 控制在合理范围内。

第二，保留最终 test set。Validation 用来调参，test 用来最终报告。不要在看到 test 结果后继续改模型再报告同一个 test 分数。那样 test 就被你转化成 validation 了。

第三，记录试验次数和决策理由。某个模型 validation 分数高一点点，不一定是真的更好，可能只是随机波动。稳定做法是用多次随机种子、交叉验证、置信区间或独立 holdout 来判断。

动作目的防止什么
预先限定候选模型控制 H_val 大小无限搜索导致 validation overfitting
最终 test 只看一次保持真实评估test 被调参污染
多 seed / 交叉验证减少单次抽样偶然性把噪声当提升
记录所有尝试保留实验透明度只汇报最好看的分数
