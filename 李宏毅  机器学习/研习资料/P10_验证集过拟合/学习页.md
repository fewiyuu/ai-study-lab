# 学习目标 {#goals}
<!-- tag: goal -->

这一节讲的是 validation selection 为什么也会过拟合。训练集负责学参数，验证集负责挑模型；如果你反复试很多候选、不断调超参数，还把公开 leaderboard 当成中间反馈，验证集就会慢慢被你改写。重点是一条会变脏的流程。

先分清
说出 training set、validation set、test set 各自负责什么，并且知道哪一个可以反复看，哪一个不能。


先看懂
把 `H_val` 理解成一组候选模型，知道候选数变多以后风险为什么会上去。


先会停
遇到 validation 分数小涨、public leaderboard 微升、模型越试越多时，知道什么时候该停下来重整实验设计。

## box info
### 边界
这里先不追 cross-validation、nested validation、Bayesian optimization、leaderboard 规则治理和 ensemble 选择。它们以后会回来，但这一页先把“验证集为什么会被试脏”讲透。

# 先抓主线 {#overview}
<!-- tag: overview -->

P09 解决的是“图像怎么进网络”。P10 关心的是模型已经跑起来以后该怎么挑，才不会把验证集也挑过头。这里讲的是训练结束后的选择动作。

这一页沿着一个固定例子走：先在 training set 上训练出 `h₁*`、`h₂*`、`h₃*`，再拿 validation set 选一个最好的，最后只在 test set 上做一次最终评估。只要你在这条链里反复插手，validation 就会被你一点点用脏。

阶段读什么写什么症状


训练training set模型参数训练分数下降，但泛化仍未知
选择validation set模型编号、超参数决定候选越多，越容易挑中只对验证集友好的模型
冻结最终选择实验记录、checkpoint如果这里还在改，后面的 test 就已经不干净了
评估test set最终分数如果在这一步之前看过太多次，分数会偏乐观

# 三种资料集 {#split}
<!-- tag: data split -->

这一页最重要的基础是分工。training set 是拿来学参数的；validation set 是拿来选模型和超参数的；test set 是拿来做最终报告的。三者都叫“数据”，但它们在流程里的职责完全不同。

如果把这三个角色混成一个，实验就会开始变脏。比如你把 test 也拿来调参，最后看起来更稳，其实只是把未见数据拿来做了选择。validation 也是一样，只是它比 test 更常被允许参与决策，所以更容易被慢慢污染。

资料集主要任务读什么写什么最容易犯的错


training set学参数样本、标签、梯度信号weights / bias把训练分数当最终结论
validation set挑模型和超参数候选模型的验证损失best_id、配置决定不断扩候选空间，再说自己只是在“比较”
test set最终评估锁定后的模型最终分数提前查看、提前调参、提前刷榜

training 负责拟合参数
训练集上的低 loss 只能说明模型会背题。它还没有回答“背出去以后会不会还能做对”。


validation 负责选模型
你可以用它做有限次选择，但不能把它当成无限次试错的真实世界代理。


test 最怕被看见
只要它参与过选择，它就已经不再是独立评估了。分数变好，不等于流程变干净。

# H_val 视角 {#hval}
<!-- tag: formula -->

把 validation selection 改写成更小的模型选择问题，最清楚。先在 training set 上训练出若干候选，再把这些候选放进 `H_val`，最后在 validation set 上挑一个 loss 最低的。这里的选择动作和训练很像，只是它发生在候选集合上，不在原始模型空间里。

$$H_{val} = \{h_1^*, h_2^*, h_3^*, \dots\}$$
 $$h_{\text{selected}} = \arg\min_{h \in H_{val}} L(h, D_{val})$$
 `h_i^*` 表示第 `i` 个候选在 training set 上训练后的最佳版本；`H_val` 是这些已经训练好的候选组成的集合；`L(h, D_{val})` 是某个候选在验证集上的平均损失。这里最重要的是顺序：先训练，再选择，最后锁定。

对象什么时候出现里面装什么在这里做什么


`H`训练之前原始模型空间容纳所有可训练的候选形式
`H_val`训练之后已经训练好的候选模型在验证集上被比较和排序
`h_selected`选择完成后一个被挑中的模型进入冻结状态，准备最终测试

候选数越多，`H_val` 就越大。只要这个集合变大，validation selection 的自由度就会上升。自由度高的地方，最容易发生的是挑到一个特别适合这份 validation 的模型。

# 验证流程状态账本 {#state}
<!-- tag: state -->

这一页最容易忽略的是状态。validation set 一旦被反复查看，状态就会变。最先变脏的通常是你对它的印象：你开始记住哪些模型在它上面表现好，接着就会沿着这个印象继续扩候选，最后把原本应该独立的判断做成条件反射。

对象读什么写什么何时变脏症状下一步检查


training set样本和标签weights正常训练时就会被使用训练分数下降，但泛化仍未知看验证集，不要直接下结论
validation set候选模型的分数best_id、调参方向你开始无限试模型时分数越来越像“你想要它说的话”先冻结候选空间，再看波动
test set锁定模型最终报告一旦提前查看就变脏最后分数偏乐观，复现实验掉分回头重建独立 test
leaderboard公开分数提交记录你根据分数不断调模型时线上分数好看，隐藏测试掉下去把它当成中间反馈，提交前先冻结判断

# 更稳的实验设计 {#design}
<!-- tag: workflow -->

真正有用的是把 validation 看得刚刚好。先限定候选空间，再做选择；先留出 final test，再做报告；先写清楚试验次数，再看分数变化。这样做是为了避免一次小幅提升带着你走太远。

动作解决什么留下什么痕迹如果不做会怎样


预先限定候选空间控制 `|H_val|`写清楚要试的模型、层数、学习率范围候选越扩越大，validation 越容易被挑歪
保留 final test防止评估污染一份从头没看过的测试集test 变成另一种 validation
记录试验次数看见 search cost每次改动都留日志你只记得最好看的那个分数
看多 seed / 交叉验证判断波动是否来自随机性均值、方差、区间把噪声误认成进步
把 leaderboard 当 validation重新定义角色只把它当中间反馈公开榜也会被过拟合

## box soft
### 一个实用顺序
先定训练流程，再定 validation 规则，最后才看 test。顺序反了，后面的数字再好看也不可信。

## box warn
### 这里先别追的东西
你现在不需要去背各种自动调参方法的名字。只要知道：搜索空间越大，验证集越容易被用脏；因此先限制空间，再谈比较。

# 一个完整案例：三次试验怎样把 validation 用脏 {#case}
<!-- tag: case -->

把 P10 想成一次很常见的实验：你先设了三组候选，分别调不同 learning rate、不同层数、不同正则化强度。第一次跑完，你在 validation 上看见模型 A 最好。第二次你觉得还能再改一点，于是新增了两个候选；第三次你又根据榜单微调了一轮。表面上你只是“多比较了几次”，实际上 `H_val` 已经从 3 个候选膨胀成一大串候选，validation 早就不再只是评分器了。

轮次候选数看见什么实际发生什么下一步该做什么


第 1 轮3模型 A 的 validation 最好正常选择，还没明显污染先冻结搜索范围，记下所有设置
第 2 轮5模型 B 也不错`H_val` 变大，选择自由度上升先问自己：新增候选是否真有必要
第 3 轮10+榜单又涨一点validation 开始像被训练过的参考答案停下来，留出从没看过的 final test

### 这段案例最该带走什么
validation 被用脏，通常是很多次“再试一次”叠出来的。你一边看分数，一边扩大候选集合，最后就会把验证集变成隐形训练过程的一部分。

# 公开榜单与搜索成本 {#leaderboard}
<!-- tag: search cost -->

很多人第一次真正把 validation 用脏，在公开榜单上。榜单会给你一个很直接的反馈：你提交一次，它回你一个分数。问题在于，这个分数很容易被当成下一轮试验的依据。你看到分数升高，就继续改模型；看到分数下降，就再换一套超参数。几轮之后，榜单已经不只是反馈，它成了你实际上在优化的对象。

这时候最该追踪的量搜索成本。搜索成本包括试了多少个候选、改了多少轮超参数、看了几次公开分数、有没有把 hidden test 的信息提前泄进流程。只要这些动作不断发生，`H_val` 就会继续膨胀，validation 的选择面也会越来越窄。最后拿到的高分，往往是“对这份验证集很合适”，不是“对真实数据很稳”。

动作你以为在做什么实际上在积累什么风险


看一次公开分数检查进展对当前候选的偏好把榜单变成中间决策信号
再调一次超参数做一次普通优化更多围绕榜单的试验记录选择过程越来越依赖反馈
继续扩候选补充搜索空间更大的 `H_val`更容易挑中“刚好适合榜单”的模型
把最终分数当结论结束实验一个暂时看起来很漂亮的结果隐藏测试掉分后才知道流程出问题

<!-- interactive: ['shape-tracer'] → keep as HTML -->

## box warn
### 这一段最该记住什么
公开榜单不是坏东西，坏的是把它当成可以无限迭代的中间信号。你一旦开始围着榜单优化，就在消耗搜索预算，也在消耗 validation 的独立性。

### 接回代码
第一个要检查的代码味道就是循环里反复读取 test 或公开榜单。如果出现这种模式，先冻住候选，再谈分数。

# 实验日志审计 {#audit}
<!-- tag: audit -->

如果你想知道 validation 到底有没有被用脏，最可靠的办法回头看实验日志。日志里应该能看出三件事：你一共试了多少轮，候选空间有没有在中途变大，test 或 leaderboard 有没有提前进入选择流程。只要这三件事能被追出来，你就能判断流程有没有保持干净。

审计时要看的是动作序列，不只是单点分数。一个健康的序列会先冻结候选空间，再看 validation，最后只在锁定后做一次 test 报告。一个有风险的序列会在看过分数以后继续改配置，或者把公开榜单的反馈写进下一轮试验。这个时候，日志本身已经在告诉你流程偏了。

日志片段你在读什么该记下什么危险信号


trial 01 / trial 02 / trial 03试验轮数候选数有没有增长试验数越多，`H_val` 越大
val_loss: 0.41 -> 0.39验证集分数哪一次选择发生了变化每次分数变化都引发新一轮调参
test_score before freeze测试集读写时机test 是否提前参与决策test 进入循环就失去独立性
public leaderboard +0.2公开分数榜单是否在引导搜索榜单反馈直接变成下一轮目标

最小审计脚本：先把异常信号标出来python复制

`audit_log = [
 {"round": 1, "used": "validation", "candidate_count": 3, "frozen": False},
 {"round": 2, "used": "validation", "candidate_count": 6, "frozen": False},
 {"round": 3, "used": "leaderboard", "candidate_count": 10, "frozen": True},
]

for row in audit_log:
 print(f"round {row['round']}: candidates={row['candidate_count']} used={row['used']}")
 if row["used"] == "leaderboard" and not row["frozen"]:
 print("risk: public feedback entered the search loop")
 if row["candidate_count"] > 5 and not row["frozen"]:
 print("risk: H_val is still expanding while results are being read")`

# 代码与报错 {#code}
<!-- tag: code reading -->

这一页的代码桥很短，但很关键。`best_id = argmin(val_losses)` 这行在读 validation losses，找出最小的那一个，再把这个编号写回实验状态。你只要看懂这行，就能知道 validation 在这里扮演的是选择器。

正确示例：用 validation 选模型python复制

`candidates = [
 train_model(config, train_loader)
 for config in planned_search_space
]

val_losses = [evaluate(model, val_loader) for model in candidates]
best_id = min(range(len(candidates)), key=lambda i: val_losses[i])
best_model = candidates[best_id]

test_loss = evaluate(best_model, test_loader) # 最后才看`

代码行读什么写什么如果写错会怎样


`train_model(...)`training setcandidate weights候选还没训练完就去比，结果没意义
`evaluate(..., val_loader)`validation set`val_losses`如果拿 test 代替 val，选择就污染了
`best_id = ...``val_losses`模型编号如果这里不断重跑，validation 会越来越像训练对象
`evaluate(..., test_loader)`锁定后的模型最终报告如果放在循环里，test 就已经不再独立

错误示例：把 test 也拿来调python复制

`while not_good_enough:
 update_hyperparameters()
 retrain_model()
 test_score = evaluate(model, test_loader) # 错：test 被用于决策`

## box warn
### 排错时先看哪一层
如果实验日志里频繁出现“又调了一次、又更好了”，先看你是否把 test 或公开 leaderboard 放进了选择流程。看见分数变好，不等于流程变干净。

# 常见误区 {#mistakes}
<!-- tag: debug -->

这页最常见的误区，往往是把角色看错了。验证集承担的是比较和选择，test 负责最终评估。只要把这两件事混在一起，流程就会开始偏。

误区错因后果先检查哪里


validation set 完全不参与训练把“训练”只理解成更新 weights看不见选择过程也会造成偏差看你是否反复根据验证分数改方向
test 反正最后才看，先看几次没关系把 test 当成临时反馈最终报告偏乐观看 test 有没有参与过任何决策
候选模型越多越好忽略 `|H_val|` 的复杂度挑到只适合 validation 的模型看候选空间是否在不断扩大
公开 leaderboard 只是成绩单以为公开分数不会被影响线上看起来进步，线下隐藏测试掉分看提交后有没有继续按榜单改模型

<!-- interactive: ['terminal-lab'] → keep as HTML -->

## box warn
### 先读报错，再看流程
如果日志已经说明 “test_loader used inside the tuning loop”，先停下来改流程，不要再往模型里堆层数。很多时候错不在模型，而在你把谁当成了决策依据。

**Q:** 4. 哪一种做法最像把 validation 用脏？

- [ ] . 先限定候选，再看一次 validation
- [ ] . 留出从没看过的 final test
- [ ] . 每看一次分数，就继续扩候选空间
- [ ] . 记录每次试验次数

Answer: . 每看一次分数，就继续扩候选空间
Explain: 边看分数边不断扩大候选空间，最容易把 validation 变成另一个训练对象。

### p10-q5 fill

**Q:** 5. `H_val` 主要指什么？

Answer: 候选模型集合,训练后候选集合,validation 后候选集合
Explain: H_val 是训练后拿来比较的一组候选模型。

### p10-q6 choice

**Q:** 6. 候选数 `|H_val|` 越大，validation overfitting 风险通常怎样？

- [ ] . 上升
- [ ] . 一定归零
- [ ] . 与候选数无关
- [ ] . 只影响图片尺寸

Answer: . 上升
Explain: 候选越多，越容易挑到只对 validation 友好的模型。

### p10-q7 choice

**Q:** 7. 为什么 validation selection 也像一种训练？

- [ ] . 它会更新所有权重
- [ ] . 它会在候选集合上最小化验证损失
- [ ] . 它只能看图片颜色
- [ ] . 它会自动生成 test set

Answer: . 它会在候选集合上最小化验证损失
Explain: 它也在候选集合上最小化验证损失，所以带有训练的选择味道。

### p10-q8 synthesis

**Q:** 8. 用 5-7 句话讲清楚：为什么用了 validation set 仍然可能 overfit？

Explain: 高分答案会先说：validation set 本来是用来在多个候选模型之间做选择的；如果候选模型越来越多，或者你反复根据 validation 分数改超参数，就等于在 H_val 上做越来越细的选择。这样选出来的模型，可能只是刚好适合这份 validation，而不是适合真实世界。最后一定要补上 test set 的角色：test set 只能留到最后做一次最终评估，不能提前参与选择。答到“选择次数”和“候选空间大小”这两个点，就已经抓住核心了。

### p10-q9 scenario

**Q:** 9. 哪一步最容易把 test 变成 validation？

- [ ] . 先限定候选空间，再看一次 validation
- [ ] . 先记录每次试验，再做最终报告
- [ ] . 看过 test 后再调一点超参数
- [ ] . 留一个从没看过的最终测试集

Answer: . 看过 test 后再调一点超参数
Explain: 只要 test 参与了选择，它就不再是独立评估。

### p10-q10 code-reading

**Q:** 10. `best_id = argmin(val_losses)` 这一行主要读什么？

Answer: validation set,validation losses,val_losses,验证集损失
Explain: 这一行主要读的是验证集上的 loss，不是在训练新的参数。

### p10-q11 choice

**Q:** 11. 哪个做法最先该写进实验计划？

- [ ] . 预先限定候选模型和超参数范围
- [ ] . 看完 test 再决定要不要继续试
- [ ] . 把 leaderboard 当最终答案
- [ ] . 只记录最好的那次分数

Answer: . 预先限定候选模型和超参数范围
Explain: 先限定候选空间，才能控制 H_val 的大小。

### p10-q12 synthesis

**Q:** 12. 设计一个不污染验证集的流程。请写 4-6 句话，按步骤说清楚。

Explain: 好的流程通常是：先把 training、validation、test 三份数据分开；然后预先限定候选模型和超参数范围，只在 training set 上训练候选；接着用 validation set 比较这些候选，选出 best_id ；最后把选中的模型冻结，只在从没看过的 test set 上做一次最终报告。答案里最好再补一句：试验日志和试验次数要记录下来，多 seed 或交叉验证可以帮你看波动，但它们不能代替独立的 final test。
