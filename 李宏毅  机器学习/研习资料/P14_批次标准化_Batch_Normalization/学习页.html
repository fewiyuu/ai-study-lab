Batch Normalization：把难走的地形变平一点

训练神经网络时，即使 error surface 是一个凸碗，如果不同参数方向的斜率差异很大，固定 learning rate 也会很难走。一个常见来源是输入 feature 的尺度差太多：某个维度值很小，参数怎么改 loss 都不太动；另一个维度值很大，参数稍微一动 loss 就大幅变化。

Feature normalization 先把每个维度减均值、除标准差，让不同维度落在相近范围。Batch Normalization 把这个想法推进到网络内部：隐藏层的输出对下一层来说也是 feature，也可以标准化。实际训练时我们无法一次用完整训练集统计，于是只用当前 batch 估计均值和标准差，这就是 batch normalization。

本节的概念地图

map

Batch Normalization 的动机可以从最简单的线性模型看起。若 x₁ 很小，w₁ 改一点对输出影响很小；若 x₂ 很大，w₂ 改一点对输出影响很大。于是 loss 在不同参数方向上的斜率差异很大，训练会变得不稳定。

输入层可以做 feature normalization。隐藏层也类似：第一层输出的 z 或 activation a，对于下一层来说就是输入 feature。如果它们的不同 dimension 尺度差很多，下一层参数也会难训练。因此我们在网络中间也做 normalization。

BN 的完整流程是：在一个 batch 内，对某层的每个 dimension 计算均值 μ 和标准差 σ；把 z 标准化成 z~；再用可学习的 γ 和 β 做缩放和平移，得到 ẑ。训练时用当前 batch 统计量；测试时用训练过程中累积的 running mean 和 running variance。

feature scale 不同→ error surface 崎岖→ batch 内算 μ/σ→ normalize→ γ/β restore→ 更好训练

把 BN 说成“网络内部的 batch 级 feature normalization + 可学习缩放平移”。

为什么 feature scale 会影响训练地形

error surface

考虑一个线性模型 y = w₁x₁ + w₂x₂ + b。如果训练资料中 x₁ 的值总是很小，那么 w₁ 改一点，w₁x₁ 变化也小，loss 对 w₁ 的反应比较钝。如果 x₂ 的值很大，w₂ 改一点，输出和 loss 可能立刻剧烈变化。

这会让 error surface 在不同方向上坡度差很多。用固定 learning rate 时，小坡方向走得太慢，大坡方向又容易冲过头。Adaptive learning rate 可以从优化器角度缓解；normalization 则是从资料尺度角度把地形改得更容易走。

Feature normalization 的标准做法是对每个 dimension 分别计算均值和标准差，然后减均值、除标准差。这样每个维度平均接近 0、方差接近 1，参数方向的尺度差异会小一些，训练通常更顺。

\tilde{x}_{i,d} = (x_{i,d} - μ_d) / σ_d
μ_d = mean over examples,   σ_d = standard deviation over examples

用一个线性模型解释“输入数值范围差很多，为什么会让某些参数方向特别难走”。

隐藏层输出也是下一层的 feature

hidden feature

即使原始输入已经做了 normalization，经过第一层权重和 activation 后，中间表示的分布可能又变得不一样。对第二层参数来说，上一层输出就是它的输入 feature。如果这些中间 feature 的尺度差很多，第二层也会遇到类似的训练困难。

因此 BN 可以放在隐藏层中，对 z 或 a 做 normalization。课程提到，放在 activation 前或后，实作上差异不一定很大；如果使用 sigmoid，因为 sigmoid 在 0 附近斜率较大，把 z 拉到 0 附近可能更有直觉。但现代网络中常见放置方式还会受架构习惯影响。

关键不是背“必须放前”或“必须放后”，而是理解：BN 让某一层交给下一层的 feature 分布更可控。这样下一层不用面对尺度差异很大的输入，优化更容易。

对 z 做 BN

先标准化 pre-activation，再进 activation。Sigmoid 场景中比较好理解。

对 a 做 BN

先 activation，再标准化输出给下一层。某些实现和架构也会这样做。

说明为什么“只 normalize 原始输入”不一定够，隐藏层 feature 也可能需要处理尺度。

Batch 统计量：样本不再完全独立

batch statistics

如果要对隐藏层的所有训练资料算均值和标准差，理论上要把整个训练集一起送进网络，这在 GPU memory 上不现实。因此实际训练只对当前 mini-batch 做统计。Batch size 是 64，就用这 64 笔资料算 μ_B 和 σ_B。

这带来一个重要后果：同一个 batch 里的样本会互相影响。某个样本的 z 改变，会改变 batch 均值和标准差，从而影响其他样本标准化后的值。因此 BN 是网络计算图的一部分，不是离线预处理。

Batch size 太小时，均值和标准差估计不稳定；如果 batch size 是 1，几乎没法得到有意义的 batch statistics。这也是为什么 BN 对 batch size 有要求，小 batch 场景常需要考虑 LayerNorm、GroupNorm 等其他 normalization 方法。

μ_B = (1/m) Σi=1m z_i
σ_B = sqrt((1/m) Σi=1m (z_i - μ_B)^2 + ε)
\tilde{z}_i = (z_i - μ_B) / σ_B

gamma 与 beta：标准化后再让模型自己调回去

affine

如果只做标准化，中间 feature 会被强制成均值 0、方差 1。这个限制有助于训练，但也可能限制模型表达。有时网络可能确实需要某个 dimension 有不同均值或不同尺度。

所以 BN 会在标准化后加入可学习参数 γ 和 β：先把 z 标准化成 z~，再做 ẑ = γ z~ + β。γ 控制缩放，β 控制平移。它们通常按 dimension 设置，也是训练中会更新的参数。

初始化时，γ 常设为 1，β 常设为 0。这样训练初期仍接近标准化结果；训练过程中，如果模型需要不同分布，就会学出合适的 γ 和 β。

\hat{z}_i = γ ⊙ \tilde{z}_i + β
γ starts near 1,   β starts near 0

Training vs Testing：统计量从哪里来

train / eval

训练时，BN 可以使用当前 batch 的均值和标准差。测试或线上 inference 时，往往只有一笔资料进来，不能等收集满一个 batch 再输出结果。即使测试时凑 batch，也会让同一笔样本的输出依赖同批其他样本，这不适合稳定服务。

因此常见实现会在训练时维护 running mean 和 running variance。每次训练 batch 算出一组 μ_B 和 σ_B 后，用 moving average 更新全局估计。进入 eval mode 时，不再用当前 batch 的统计量，而用训练期间积累的 running statistics。

这就是为什么 PyTorch 里需要 model.train() 和 model.eval()。BN 和 Dropout 都会受模式影响：BN 在 train mode 用 batch statistics 并更新 running statistics；在 eval mode 用 running statistics，不再根据当前 batch 改变统计量。

模式BN 使用的 μ/σ是否更新 running stats典型用途
train当前 mini-batch是训练参数。
eval / inferencerunning mean / running variance否验证、测试、线上预测。

为什么 BN 有帮助：别只背 internal covariate shift

why it helps

BN 原论文提出过一个解释：internal covariate shift，也就是训练过程中前层输出分布改变，导致后层一直追着变化的输入分布学。BN 通过稳定中间分布，可能缓解这个问题。

但后续研究对这个解释提出质疑。有论文通过实验指出，中间分布变化未必是训练困难的核心原因；即使分布变化大，也不一定伤害训练；一些梯度方向也未必因为分布变化而差很多。因此，把 BN 的作用简单等同于“解决 internal covariate shift”并不稳妥。

更稳的说法是：BN 往往能改善优化 landscape，让 loss surface 更平滑，训练更容易，甚至允许使用更大的 learning rate。课程引用的实验也显示，使用 BN 后常能更快达到相同 validation accuracy；对 sigmoid 这类较难训练的 activation，也可能让训练跑起来。

旧解释

BN 缓解 internal covariate shift，让中间分布更稳定。

更谨慎解释

BN 改善优化地形，让训练更顺；具体机制可继续研究。

最小代码：BN 的训练与测试路径

minimal code

下面的伪代码展示 BN 的核心。训练时用当前 batch 统计量，并更新 running mean/var；测试时直接用 running mean/var。真实框架还会处理维度、epsilon、momentum、反向传播和 CNN 里的 channel 统计。

正确示例：BatchNorm 的核心路径python
def batch_norm(z, gamma, beta, running_mean, running_var, training, momentum=0.1, eps=1e-5):
    if training:
        mu = z.mean(axis=0)
        var = z.var(axis=0)
        running_mean = (1 - momentum) * running_mean + momentum * mu
        running_var = (1 - momentum) * running_var + momentum * var
    else:
        mu = running_mean
        var = running_var

    z_tilde = (z - mu) / np.sqrt(var + eps)
    z_hat = gamma * z_tilde + beta
    return z_hat, running_mean, running_var
错误示例：测试时仍用当前单笔样本统计pseudo
# 错误：线上来一笔资料，就用这一笔算 mean/var。
model.train()
prediction = model(one_example)

# 修正：测试或上线前切到 eval mode，使用 running statistics。
model.eval()
prediction = model(one_example)

常见误区与修正

pitfalls

误区 1：BN 只是输入预处理

错因：只理解了 feature normalization。后果：不知道隐藏层也在网络中动态标准化。修法：BN 是计算图中的 layer。

误区 2：batch size 多小都一样

错因：忽略 μ/σ 来自 batch。后果：小 batch 统计不稳定，batch size 1 几乎没意义。修法：小 batch 考虑其他 norm 或调整设置。

误区 3：gamma/beta 抵消了 BN

错因：只看它们能改变分布。后果：误以为 BN 没意义。修法：初始化时 γ=1、β=0，训练过程让模型在标准化基础上恢复必要尺度。

误区 4：测试时也用当前 batch 统计量

错因：混淆 train/eval mode。后果：线上单笔预测不稳定，验证结果依赖 batch 组成。修法：eval 使用 running mean/variance。

误区 5：BN 一定因为 internal covariate shift 才有效

错因：把原论文解释当成定论。后果：无法理解后续研究质疑。修法：更稳妥地说 BN 改善优化 landscape。

误区 6：BN 加了就不用调 learning rate

错因：把优化技巧当万能按钮。后果：仍可能震荡或收敛差。修法：BN 可能允许更大学习率，但仍需验证和调参。

术语与检查：BN 不是普通预处理

terms / checklist

Batch Normalization 的直觉是：不要只在数据入口把尺度调好，也要在网络内部持续把中间 feature 拉回较稳定的范围。它是计算图中的 layer，会参与前向、反向和训练/测试模式切换。

术语一句话解释易混点
Batch statistics训练时用当前 mini-batch 估计均值和方差。样本之间会互相影响；batch 太小会让估计不稳定。
Running mean/variance训练时累积的统计量，测试时用于稳定推理。eval() 不只是关 dropout，也会改变 BN 的统计来源。
γ / β标准化后可学习的缩放和平移。它们不是取消 BN，而是让模型保留必要表达能力。
Internal covariate shift原论文中的常见解释。可以提，但更稳妥是说 BN 改善优化地形和训练稳定性。

Worked example

若某维度一个 batch 的值是 [2, 4, 6]，均值是 4。标准化会把它们变成围绕 0 的数；如果模型发现这一层需要更大幅度，就通过 γ 放大，再用 β 平移。
