# 本节的概念地图 {#goals}
<!-- tag: MAP -->

P30 已经有了 detector：分类器输出 confidence score，分数高于阈值判为 normal，低于阈值判为 anomaly。P31 要回答三个问题。第一，development set 该怎么准备；第二，用什么指标衡量系统；第三，如何根据任务代价选择阈值。

普通分类任务里，development set 通常和训练任务的类别结构一致。但 open set 里，训练资料全是辛普森角色并带具体人物标签；development set 则要包含辛普森角色和非辛普森角色。你不一定需要知道辛普森角色具体是谁，但必须知道它是否来自辛普森家族，因为评估的是 known/unknown 判定。

# Development Set：要模仿测试时的 known/unknown {#devset}
<!-- tag: DEV SET -->

development set 的作用是调 hyperparameter，避免把测试集拿来反复调参。在 P30 的系统里，最明显的 hyperparameter 就是阈值 `λ`。要调这个阈值，development set 不能只有训练时见过的类别，也不能只标“这个辛普森角色是谁”。它需要标注样本是否来自已知世界。

以辛普森家庭为例，训练阶段需要图片具体是谁：Bart、Lisa、Homer 等。development 阶段需要的是“这张图片是不是辛普森家族人物”。如果是辛普森人物，不一定要知道具体是谁；如果不是，也不一定要知道它是哪部作品的人物。因为阈值调的是 normal/anomaly 判定，而不是角色分类准确率。

| 阶段 | 资料内容 | 标签需求 | 用途 |
| --- | --- | --- | --- |
| 训练分类器 | 已知类样本 | 具体已知类别 | 学会分类 |
| 调异常阈值 | 已知类 + 未知类样本 | known 或 unknown | 选 threshold、评估系统 |
| 上线测试 | 未知来源的新样本 | 没有标签 | 输出 normal/anomaly |

# Accuracy 陷阱：什么都判 normal 也可能很好看 {#accuracy}
<!-- tag: ACCURACY -->

异常检测里，正常样本和异常样本数量常常非常不平衡。课程例子中 development set 有 100 张辛普森图片和 5 张非辛普森图片。如果一个系统几乎把所有图片都判成 normal，它会漏掉 5 张异常，但仍然有 100/105，超过 95% 的 accuracy。这个数字看起来漂亮，但系统没有抓到任何异常。

现实场景更极端。信用卡诈欺、网络入侵、罕见病变都可能是低比例事件。若 10000 笔交易里只有 5 笔诈欺，一个永远输出正常的模型 accuracy 接近 99.95%。但这不是有用的诈欺检测器。accuracy 把多数类的正确覆盖成了整体表现，掩盖了你最关心的异常类。

小交互：异常比例越低，accuracy 越会骗人
异常数量**5**
全部判 normal 的 accuracy**95%**


高 accuracy，不等于好系统
当异常很少时，全部判 normal 的系统也会有漂亮 accuracy，但它完全没有检测能力。

# 两种错误：False Alarm 与 Missing {#errors}
<!-- tag: ERROR TYPES -->

异常检测至少要拆成两种错误。false alarm 是正常样本被误判为异常，也叫误报。比如辛普森角色被系统说成不是辛普森人物。missing 是异常样本没有被抓出来，也叫漏报。比如非辛普森角色被系统当作辛普森人物接受。

阈值会在两者之间移动。阈值低时，系统宽松，很多样本会被接受为 normal，因此 missing 可能多，false alarm 可能少。阈值高时，系统严格，更多样本被拒绝为 anomaly，因此 missing 可能少，false alarm 可能多。没有单一阈值在所有任务里都最好。

| 真实情况 | 系统输出 | 错误名 | 直觉 |
| --- | --- | --- | --- |
| normal | anomaly | false alarm | 误报，正常被吓出来 |
| anomaly | normal | missing | 漏报，异常没被抓到 |
| normal | normal | correct accept | 正常通过 |
| anomaly | anomaly | correct reject | 异常被抓到 |

# Cost Table：哪个错误更贵，任务说了算 {#cost}
<!-- tag: COST -->

两个系统怎么比较，不一定看错误总数。课程里举了两个阈值：一个系统可能有 1 个 false alarm、4 个 missing；另一个系统可能有 6 个 false alarm、3 个 missing。若只看错误数，前者 5 个错误，后者 9 个错误，似乎前者更好。但如果 missing 的代价远高于 false alarm，后者可能反而更好。

医疗检测是典型例子。把健康人误报为疑似病变，会造成复查和焦虑；但真正病变没抓出来，后果可能严重得多。所以 missing 的代价可能远高于 false alarm。相反，有些告警系统如果误报会打断生产、触发昂贵流程，false alarm 的代价可能更高。评估指标要跟任务代价一致。

小交互：不同代价下谁更好
false alarm 代价**1**
missing 代价**100**


观察重点
系统 A：1 个 false alarm、4 个 missing。系统 B：6 个 false alarm、3 个 missing。改变代价表后，优胜系统可能反转。

# AUC：不固定阈值，只看排序质量 {#auc}
<!-- tag: AUC -->

除了固定一个阈值再计算错误，文献中也常用 AUC 这类指标。AUC 可以理解为看 anomaly score 的排序质量：真正异常样本是否倾向排在更“异常”的位置，正常样本是否倾向排在更“正常”的位置。它不需要先决定单一 threshold。

这对比较模型很方便，因为不同应用可能选不同阈值。若一个模型整体排序好，通常表示它在许多阈值下都有潜力。但 AUC 也不是万能答案。上线时仍需要具体阈值和代价决策；AUC 更像模型打分能力的整体评估，而不是最终业务规则。

## box info
### 固定阈值指标
适合回答“当前系统上线规则会犯多少 false alarm 和 missing”。

## box soft
### AUC 类指标
适合回答“分数排序是否把异常排到正常前面”。

### 现在你应该能做什么
说明 AUC 为什么不需要先选阈值，以及它和上线决策的区别。

# 一个稳妥的评估流程 {#workflow}
<!-- tag: WORKFLOW -->

第一步，训练分类器或 anomaly scorer，不要用测试集调参。第二步，准备 development set，让它同时包含 normal 和 anomaly，并且比例、难度尽量贴近真实应用。第三步，在 development set 上扫阈值，统计 false alarm 和 missing，或者计算排序类指标。第四步，根据任务 cost table 选阈值。第五步，最后只在 test set 上做一次报告。

这套流程听起来朴素，但能避免两个常见问题：一是用 accuracy 掩盖异常类失败；二是在 test set 上反复试阈值导致过拟合。异常检测本来就容易因为类别比例和阈值选择变得不稳定，流程越清楚，结论越可信。

训练 scorer
准备 dev set
扫 threshold
按 cost 选择
test set 报告

# 公开分享速查：别被 Accuracy 骗了 {#share}
<!-- tag: quick reference -->
<!-- layout: grid2 -->

## box soft
### 核心直觉
异常检测的关键哪一种错更贵。一个系统漏掉 1 个真正异常，可能比误报 20 个正常样本更严重；也可能相反，取决于业务。
所以评估要拆开 false alarm 与 missing，再用 cost table 或 AUC 观察阈值选择。

## box warn
### Worked example
1000 个样本里只有 10 个异常。系统永远输出 normal，accuracy 是 99%，但 missing 是 10/10，真正异常全漏掉。
若医疗筛查漏诊代价极高，99% accuracy 反而可能是坏系统；如果是低风险提醒，过多 false alarm 也可能让用户关闭系统。

| 术语 | 精确定义 | 易混点 |
| --- | --- | --- |
| False alarm | 正常样本被误判为异常。 | 会增加人工复查和打扰成本。 |
| Missing | 异常样本被误判为正常。 | 在安全、医疗、风控中常常更贵。 |
| Accuracy | 整体预测正确比例。 | 类别极不平衡时可能很漂亮但没用。 |
| AUC | 不固定单一阈值，评估异常分数排序质量。 | AUC 高不代表上线不用选阈值。 |
| Cost table | 给不同错误类型分配业务代价。 | 不是模型自动学出来的，通常来自任务需求。 |

## box info
### 评估检查表
- Development set 要模拟测试时 normal/unknown 的混合情况。
- 阈值扫描时同时记录 false alarm、missing 和总成本。
- 最终 test set 只报告一次，避免调到测试集上。

# 最小代码：数错误与算代价 {#code}
<!-- tag: MINIMAL CODE -->

下面伪代码展示固定阈值评估。重点是不要只算 accuracy，而要把两种错误分开。

正确示例：统计 false alarm 与 missingpseudo python

```text
pred_anomaly = score < threshold

false_alarm = count((y_true == "normal") & (pred_anomaly == True))
missing = count((y_true == "anomaly") & (pred_anomaly == False))

cost = false_alarm * false_alarm_cost + missing * missing_cost
```

正确示例：不要在 test set 上选阈值pseudo python

```text
best_threshold = choose_threshold(dev_scores, dev_labels, cost_table)
report = evaluate(test_scores, test_labels, best_threshold)
```

错误示例：只看 accuracypseudo

```text
# 错误：异常极少时，这个数字可能很好看。
accuracy = count(pred == y_true) / len(y_true)
print(accuracy)

# 修正：至少同时报告 false alarm、missing、cost 或排序类指标。
```

# 常见误区与修正 {#mistakes}
<!-- tag: PITFALLS -->
<!-- layout: grid2 -->

## box redsoft
### 误区 1：development set 和训练集标签一样就行
错因：忘记 open set 测试有 unknown。后果：阈值无法评估拒绝能力。修法：dev set 要含 normal 和 anomaly。

## box redsoft
### 误区 2：accuracy 高就是系统好
错因：忽略类别不平衡。后果：全部判 normal 也可能很好看。修法：拆 false alarm 和 missing。

## box redsoft
### 误区 3：false alarm 和 missing 代价一样
错因：只看错误个数。后果：选错业务阈值。修法：根据场景定义 cost table。

## box redsoft
### 误区 4：阈值越高越好
错因：只想抓更多异常。后果：正常样本大量误报。修法：看 false alarm/missing 的取舍。

## box redsoft
### 误区 5：AUC 好就不用阈值
错因：混淆模型排序评估和上线决策。后果：无法落地。修法：AUC 评估排序，上线仍需阈值。

## box redsoft
### 误区 6：可以用 test set 反复调阈值
错因：把测试集当开发集。后果：评估过拟合。修法：dev 调参，test 最后报告。

## box soft
### 现在你应该能做什么
审查一份异常检测报告时，指出它有没有 accuracy 陷阱、阈值泄漏和代价不清的问题。
