# 本节的概念地图 {#goals}
<!-- tag: MAP -->

前几节有 classifier，所以可以问“分类器有多有信心”。P33 没有 label，因此没有 classifier。新的主线是 density estimation：用训练资料估计一个概率模型，描述 normal data 在 feature space 中怎么分布。新样本如果落在高概率区域，就像训练资料；如果落在低概率区域，就不像训练资料。

这里的关键思考顺序。第一，把真实对象转成向量。第二，观察大多数训练样本聚在哪里。第三，建立能给每个 `x` 分数的模型。第四，用阈值把概率分数转成 normal/anomaly 判断。这个框架后面可以接 Gaussian、mixture model、flow、autoencoder reconstruction score 等不同实现。

无标签资料
feature vector
学习 p(x)
likelihood score
threshold 决策

# 无标签设定：只有 x，没有 y {#setting}
<!-- tag: SETTING -->

在 open set recognition 中，训练资料有已知类别标签，比如辛普森角色是谁。P33 的设定不同：我们只有大量样本 `x¹, x², ..., xᴺ`，没有 `y`。因此无法训练“输入 x 输出类别 y”的 classifier，也不能直接用 classifier confidence。

没有 label 不代表不能做事。我们仍然可以假设多数训练样本来自正常行为，于是训练资料本身就告诉我们“常见样本应该长什么样”。要做的是建立一个模型来描述这种常见程度。这个模型给每个新样本一个概率或概率密度分数，分数越低，越值得怀疑。

## box info
### 有标签 open set
有 `(x, y)`，先训练 classifier，再用 confidence 做 unknown 判断。

## box warn
### 无标签异常检测
只有 `x`，学习 `p(x)` 或类似 normality score。

### 现在你应该能做什么
说明为什么没有 label 时，P30 的 classifier confidence 路线不能直接用。

# Twitch Plays Pokemon：多数玩家视为正常 {#case}
<!-- tag: CASE -->

Twitch Plays Pokemon 是一个多人共同操控宝可梦游戏的活动。大量玩家同时输入上下左右、A、B 等指令，系统从输入流中选指令执行。因为参与者很多，游戏过程非常混乱。人们怀疑其中可能存在 troll，也就是故意捣乱的玩家。

异常检测的表述是：把多数玩家视为正常训练资料，学习正常玩家行为模式；再找出行为不像多数人的玩家。这里没有明确 label 告诉我们“某玩家是不是 troll”。甚至原稿也提醒：troll 是否真的存在都可以讨论。机器学习任务只做一件事：找行为上明显不像多数人的样本。

样本每一个玩家是一笔资料 `x`。
假设多数玩家想正常推进游戏，因此构成正常行为分布。
目标找出行为概率低、和多数玩家不同的人。

# 第一步：玩家必须先变成向量 {#features}
<!-- tag: FEATURE VECTOR -->

机器学习模型通常不能直接吃“一个玩家”这个抽象对象。我们需要把玩家表示成 feature vector。原稿举了两个特征：第一，说无关话的比例。游戏真正有效的输入是上下左右、A、B、Start 等，其他聊天内容可视为无关话。第二，在无政府状态下发言的比例。无政府状态会直接从输入流中选指令，可能更容易被捣乱者利用。

这两个特征都在 0 到 1 之间。一个玩家可以表示为 `x = [trash_talk_rate, anarchy_talk_rate]`。这当然只是简化例子，真实系统可以加入更多特征，例如指令重复率、反向指令比例、投票一致性、时间段行为等。但无论特征多少，第一关都是让每个样本变成可计算的向量。

| 特征 | 含义 | 可能提供的线索 |
| --- | --- | --- |
| trash talk rate | 无关发言占所有发言比例 | 过高可能不像认真玩游戏的人 |
| anarchy talk rate | 无政府状态下发言比例 | 过低或过高都可能和多数玩家不同 |
| command consistency | 指令是否跟主流目标一致 | 可衡量是否常做反向操作 |

### 现在你应该能做什么
为一个无标签异常检测任务设计 2-3 个可计算特征。

# 第二步：学习 p(x)，用概率替代信心分数 {#prob}
<!-- tag: PROBABILITY -->

有 classifier 时，我们用 confidence score 判断异常。没有 classifier 时，就学习 `p(x)`。它表示某种行为向量在训练资料分布下有多常见。若一个玩家有 1%-2% 的无关发言比例，且大约 80% 的发言发生在无政府状态，这可能落在高密度区域；若某人 60% 都在说无关话，或者很少在多数人都发言的状态发言，就可能落在低密度区域。

注意这里说的是概率模型，不是道德判断。低概率不等于一定是坏人，也不等于一定是 troll。它只说明这个行为向量不像多数训练玩家。异常检测系统的输出应该被看作候选线索，后续还需要结合规则、人工审查或业务决策。

$$
Train a probability model from x¹, ..., xᴺ
 if p(x_new) ≥ λ → normal
 if p(x_new) < λ → anomaly candidate
$$

### 现在你应该能做什么

# 第三步：阈值把概率变成决策 {#threshold}
<!-- tag: THRESHOLD -->

概率模型只给分数，仍然需要阈值。阈值高，系统更严格，更多低到中等概率样本会被标为异常；阈值低，系统更宽松，只抓非常低概率样本。这个取舍和 P31 一样，需要 development set 或业务代价来决定。

还有一个技术提醒：连续变量中的 `p(x)` 更准确地说常常是概率密度，不是单点概率。初学时可以先把它理解为“模型认为这个位置常不常见的分数”。在实际实现中，常用 log likelihood，因为概率密度可能很小，直接相乘容易数值下溢。

## box info
### 高 p(x)
样本落在训练资料常见区域，倾向 normal。

## box warn
### 低 p(x)
样本落在稀有区域，作为 anomaly candidate。

## box soft
### 现在你应该能做什么
解释为什么概率模型输出后仍然需要 threshold 和评估指标。

# 公开分享速查：没有标签就学 p(x) {#share}
<!-- tag: quick reference -->
<!-- layout: grid2 -->

## box soft
### 核心直觉
没有 label 时，模型无法学习“x 属于哪个 y”。于是问题改成：只看 x 本身，哪些行为常见，哪些行为稀有。常见行为高 p(x)，稀有行为低 p(x)。
低概率不等于犯罪或故障，只表示它不像大多数训练资料，应该进入复查队列。

## box warn
### Worked example
玩家 A 99% 消息都与游戏无关，且总在无政府状态下刷屏。若大多数玩家这两个特征都很低，A 在二维行为空间会落到低密度区域。
模型输出低 likelihood 后，不应直接定性为恶意，而是标记为 anomaly candidate，再结合日志上下文复查。

| 术语 | 精确定义 | 易混点 |
| --- | --- | --- |
| Unlabeled data | 只有样本 x，没有类别 y。 | 不能假装有 y 去训练普通分类器。 |
| Feature vector | 把原始行为转成模型可学习的数值描述。 | 概率模型只能看到特征空间里的结构。 |
| p(x) / density | 样本在正常资料分布下的常见程度。 | 连续变量里常是密度，不是单点概率。 |
| Likelihood score | 把新样本代入模型得到的解释程度。 | 低 likelihood 是异常候选，不是最终罪名。 |
| Log likelihood | 用对数形式表示 likelihood，数值更稳定。 | 阈值也要在同一分数尺度上比较。 |

## box info
### 流程检查表
- 先把原始日志变成稳定、可解释的 feature vector。
- 只用 x 拟合正常行为分布，不伪造不存在的标签。
- 低 p(x) 只进入复查，不直接等同于恶意。

# 最小代码：从玩家日志到 likelihood {#code}
<!-- tag: MINIMAL CODE -->

下面伪代码表达流程：先抽特征，再拟合概率模型，最后用 log likelihood 和阈值判断。

正确示例：无标签异常检测流程pseudo python

```text
players = load_player_logs()
X = [extract_features(player) for player in players]

density_model = fit_probability_model(X)
log_p = density_model.log_prob(x_new)

if log_p < threshold:
 label = "anomaly candidate"
else:
 label = "normal"
```

正确示例：玩家特征pseudo python

```text
features = [
 trash_talk_count / total_messages,
 anarchy_messages / total_messages,
 repeated_commands / total_commands,
]
```

错误示例：没有 label 却硬训练 classifierpseudo

```text
# 错误：没有 y，却写成监督分类。
model.fit(X, y)

# 修正：无标签时先考虑 p(x)、距离、重建误差等 normality score。
```

# 常见误区与修正 {#mistakes}
<!-- tag: PITFALLS -->
<!-- layout: grid2 -->

## box redsoft
### 误区 1：没有 label 就不能做异常检测
错因：只知道 classifier 路线。后果：忽略无监督 normality score。修法：学习 `p(x)` 或其他分布分数。

## box redsoft
### 误区 2：低概率一定等于坏人
错因：把统计异常当道德判断。后果：误伤罕见但正常行为。修法：说 anomaly candidate，并结合后续检查。

## box redsoft
### 误区 3：feature 随便选都可以
错因：低估表示的重要性。后果：概率模型学到无用分布。修法：特征要和异常行为机制有关。

## box redsoft
### 误区 4：二维边缘分布足够判断所有异常
错因：只看单一维度。后果：忽略特征组合异常。修法：看联合分布 `p(x)`。

## box redsoft
### 误区 5：高 p(x) 样本永远正常
错因：把模型分数绝对化。后果：漏掉伪装成常见行为的异常。修法：理解模型假设和特征覆盖范围。

## box redsoft
### 误区 6：阈值可以脱离评估来定
错因：忘记 P31 的评估逻辑。后果：误报漏报不可控。修法：用 dev set 或 cost table 选 threshold。

## box soft
### 现在你应该能做什么
检查一个无标签异常检测方案是否说清楚了特征、概率模型、阈值和输出含义。
