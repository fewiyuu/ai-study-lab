# 本节的概念地图 {#goals}
<!-- tag: MAP -->

P35 的主线把异常检测从“二维玩具例子”推向可用流程。真实问题中，你不会只用两个 feature。你会把所有可能帮助区分正常和异常的行为统计都加入向量，再让模型学习这些 feature 的联合分布。

加入 feature 后要有两个警觉。第一，feature 越多，不等于模型越好；不稳定、泄漏、强噪声的 feature 会伤害判断。第二，异常分数可能反直觉，因为模型比较的是“和训练资料中多数正常行为的相似度”，而不是比较某个单一 feature 是否看起来漂亮。

设计 feature

计算 log score
解释反直觉样本
考虑替代方法

# 扩展特征：把“可疑行为”变成可计算向量 {#features}
<!-- tag: FEATURES -->

在 Twitch Plays Pokemon 的例子中，前面只用了两个 feature：说无关话的比例、在无政府状态下发言的比例。P35 说明你可以加入更多你认为有帮助的行为统计。例如玩家按 start 键的比例，因为持续打开道具栏可能妨碍游戏推进；再如过去 30 秒内，该玩家选择与多数人一致的比例；以及与多数人相反的比例。

这些 feature 的作用是把模糊直觉变成模型可处理的向量。人类会说“这个人老是捣乱”“这个人总跟大家唱反调”，模型不能直接理解这句话，但可以理解 `[垃圾话比例, 无政府发言比例, start 比例, 跟随多数比例, 唱反调比例]` 这样的数字。

| Feature | 直觉问题 | 可能风险 |
| --- | --- | --- |
| 无关发言比例 | 这个玩家是否经常说和游戏无关的话？ | 聊天文化不同，正常玩家也可能闲聊。 |
| 无政府状态发言比例 | 他是否常在特定模式下活跃？ | 模式本身可能影响所有人的行为分布。 |
| 按 start 键比例 | 他是否频繁打开菜单，阻碍行动？ | 新手可能误触，不能直接当恶意。 |
| 跟随多数比例 | 他的决策是否经常与大多数一致？ | 永远一致也可能不自然，不能简单越高越好。 |
| 唱反调比例 | 他是否经常选择与多数相反的动作？ | 少数派不一定错，关键看训练资料分布。 |

feature 设计有一个朴素原则：先问它是否能描述“正常资料的稳定模式”。如果某个 feature 在正常玩家中也非常随机，模型很难从它学到有用边界。反过来，如果某个 feature 很稳定，却被异常行为系统性打破，它就很有价值。

**现在你应该能做什么：**面对一个异常检测任务，先列出候选 feature，再为每个 feature 写清楚它捕捉的行为直觉和潜在误判来源。

# log 分数：非常小的 likelihood 要换一种读法 {#score}
<!-- tag: LOG SCORE -->

feature 变多后，密度乘积或密度值常常非常小，所以课程中会取 log。log score 通常是负数，越接近 0 往往表示分数越高，越负表示越不符合模型。比如课程中的例子里，一个玩家得到 `-16`，另一个更奇怪的玩家得到 `-22`。在这种读法下，`-22` 比 `-16` 更低，更可能被认为异常。

注意 log score 的绝对数值通常不能跨模型、跨 feature 集随便比较。你加了 feature、换了缩放方式、换了分布假设，分数尺度都会变。真正可比较的是同一个模型下不同样本的相对分数，或者在验证集上通过同一阈值做出的决策。

$$
score(x) = log p(x; mu*, Sigma*)
同一个模型中，score 越低，表示样本在正常分布下越罕见。
$$

## box info
### 读分数
`-16` 高于 `-22`，所以前者更像训练资料中的正常模式。不要被“负数”绕晕，比较大小即可。

## box warn
### 设阈值
如果阈值是 `-20`，那么 `-16` 正常，`-22` 异常候选。阈值必须和当前模型的分数尺度配套。

**现在你应该能做什么：**看到 log likelihood score 时，能正确比较 `-16`、`-22`、`2` 的高低，并说明它们对异常判断的含义。

# 反直觉案例：永远跟大家一样，也可能异常 {#counter}
<!-- tag: COUNTERINTUITION -->

课程给了一个很好的反直觉例子。第一个玩家在前三个 feature 上看起来正常，而且“总是跟大家一样、从不唱反调”，得到的分数约为 `-16`。第二个玩家其他部分相同，但从不跟大家一样，而且有 30% 时候唱反调，分数降到约 `-22`，这符合直觉：老是唱反调比较奇怪。

但第三个玩家只改了“跟大家一样”的比例：70%。结果他的分数反而更高，甚至可能到 `2`。为什么？因为真实玩家不太可能永远都和多数人完全一致。训练资料里的正常玩家也会偶尔迟疑、误按、改变策略、没跟上群体决定。于是“100% 一致”反而不像真实正常玩家，“70% 一致”更像训练资料。

这就是异常检测最重要的思维转换：不要把某个 feature 单独理解成“越大越正常”或“越小越异常”。模型学习的是联合分布。一个数值单看很漂亮，但如果它在真实正常资料中很少出现，仍然可能导致低分。

| 玩家 | 行为摘要 | log score | 解释 |
| --- | --- | --- | --- |
| A | 10% 无关话，90% 无政府发言，10% start，100% 跟多数一致，0% 唱反调 | -16 | 大体正常，但“永远一致”有点不自然。 |
| B | 前三项同 A，0% 跟多数一致，30% 唱反调 | -22 | 与群体决策差异很大，更罕见。 |
| C | 前三项同 A，70% 跟多数一致，0% 唱反调 | 2 | 更接近真实玩家的正常波动。 |

**现在你应该能做什么：**解释为什么异常检测的输出需要按“训练分布”理解，而不是按单个 feature 的道德直觉理解。

# Autoencoder：用“还原得好不好”判断异常 {#autoencoder}
<!-- tag: AUTOENCODER -->

课程最后快速提到 deep learning based 方法，其中最典型的是 autoencoder。它由 encoder 和 decoder 组成：encoder 把输入变成一个较短或受约束的 code，decoder 再把 code 还原成原输入。训练时只用正常资料，目标是让输出尽量接近输入。

如果训练资料都是辛普森图片，autoencoder 会特别擅长还原辛普森风格的图片。测试时，如果输入仍是正常辛普森图片，经过 encoder 和 decoder 后，还原结果会和原图很接近；如果输入是完全不同的图片，模型没学过这种模式，还原结果会差很多。于是可以用 reconstruction error，也就是输入和输出之间的差距，作为异常分数。

**输入**正常图片或正常行为向量进入模型。
**Encoder**把输入压缩成 code，保留常见模式。
**Decoder**尝试从 code 还原原输入。
**误差**还原差距大，表示输入不像训练资料。

autoencoder 和 Gaussian 的共同点，是都只需要正常资料就能训练一个“正常性模型”。不同点是，高斯直接在 feature space 中估计密度；autoencoder 学的是如何重构正常样本。它不一定给出严格概率密度，但能处理图片、语音、复杂向量等高维资料。

**现在你应该能做什么：**解释 autoencoder 异常检测的判断依据：正常样本容易还原，异常样本还原误差大。

# 其他方法：同一个目标，不同的正常性定义 {#alternatives}
<!-- tag: METHODS -->

除了高斯和 autoencoder，课程还点名 one-class SVM 与 isolation forest。它们的共同目标都是：主要依靠正常资料，学出正常区域，再把不像正常区域的样本找出来。差异在于它们如何定义“正常区域”。

| 方法 | 正常性的定义 | 适合直觉 | 局限 |
| --- | --- | --- | --- |
| Gaussian | 落在高概率密度区域。 | 资料像单峰椭圆，feature 可解释。 | 分布假设强，不适合复杂多峰。 |
| Autoencoder | 能被模型低误差重构。 | 图片、语音、高维资料。 | 重构得好不一定代表正常，需谨慎调模型容量。 |
| One-class SVM | 在特征空间中学习包住正常资料的边界。 | 正常资料边界较清楚，中小规模资料。 | 核函数和参数敏感，大规模时成本较高。 |
| Isolation forest | 异常点更容易被随机切分隔离。 | 表格资料、非线性边界、无需强分布假设。 | 解释性和参数选择仍需验证。 |

方法选择不能只看名字高级不高级。先看资料类型、feature 是否可靠、是否需要解释、误报漏报代价、训练资料是否干净。如果二维行为统计已经足够清楚，高斯可能很好；如果是图片，autoencoder 更自然；如果是表格资料且分布复杂，isolation forest 往往是强基线。

**现在你应该能做什么：**面对一个异常检测任务，给出至少两种候选方法，并说明为什么它们适合或不适合。

# 最小代码：特征、log score 与方法切换 {#code}
<!-- tag: CODE -->

下面的代码展示如何把多个行为统计组成 feature vector，然后用高斯模型给出 log score。重点流程：特征要在训练和测试中保持同一含义、同一顺序、同一缩放。

正确示例：多特征高斯异常分数Python / NumPy

```text
import numpy as np

FEATURES = [
 "trash_talk_rate",
 "anarchy_rate",
 "start_button_rate",
 "majority_agree_rate",
 "oppose_rate",
]

def make_vector(stats):
 return np.array([stats[name] for name in FEATURES], dtype=float)

player_a = make_vector({
 "trash_talk_rate": 0.10,
 "anarchy_rate": 0.90,
 "start_button_rate": 0.10,
 "majority_agree_rate": 1.00,
 "oppose_rate": 0.00,
})

score = log_gaussian_density(player_a, mu, sigma)
is_anomaly = score < log_threshold
```

如果改用 autoencoder，feature 可能不再是手工统计，也可能是图片像素或模型中间表示。分数也从 log density 变成 reconstruction error。此时方向要反过来理解：Gaussian 是分数越低越异常；autoencoder 常见写法是误差越高越异常。

错误示例：训练和测试 feature 顺序不一致Feature bug

```text
# 错误示例：训练时第三维是 start_button_rate，测试时第三维却变成 oppose_rate。
train_features = [trash, anarchy, start_rate, agree, oppose]
test_features = [trash, anarchy, oppose, agree, start_rate]
```

这个错误很隐蔽，因为数组维度对得上，程序不会报错，但语义已经错位。模型会把“唱反调比例”当成“按 start 键比例”解释，分数自然不可信。修法是统一 feature schema，并在数据管线里显式校验字段名和顺序。

**现在你应该能做什么：**写出一个多特征异常检测的 feature schema，并知道训练与测试必须严格复用同一套 schema。

# 常见误区与修法 {#mistakes}
<!-- tag: DIAGNOSIS -->
<!-- layout: grid2 -->

## box redsoft
### 误区 1：feature 越多越好
无关 feature 会增加噪声，样本少时还会让协方差估计不稳定。修法是做消融、标准化、稳定性检查。

## box redsoft
### 误区 2：把某个 feature 当单调规则
“跟多数一致”不是越高越正常。修法是看训练资料的联合分布，允许正常行为有波动。

## box redsoft
### 误区 3：混淆分数方向
Gaussian log score 通常越低越异常；autoencoder reconstruction error 通常越高越异常。修法是每个方法都明确 score 的方向。

## box redsoft
### 误区 4：训练测试 feature 顺序错位
维度相同不代表语义相同。修法是使用命名 schema，避免裸数组随手拼接。

## box redsoft
### 误区 5：只看模型，不看复查流程
异常检测经常用于候选筛选，低分不等于定罪。修法是设计人工复查、反馈回流和阈值更新。

## box redsoft
### 误区 6：autoencoder 容量过大
模型太强可能连异常也还原得很好。修法是控制瓶颈、验证重构误差分布，并用异常样例做压力测试。

**现在你应该能做什么：**检查一个异常检测方案时，从 feature、分数方向、阈值、方法假设和复查流程五个角度找风险。
