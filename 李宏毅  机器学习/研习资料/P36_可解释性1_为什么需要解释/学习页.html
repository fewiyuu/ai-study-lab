正确答案之外，还要追问正确的理由

到目前为止，我们训练了分类、语音、影像、异常检测等模型。模型给出正确答案当然重要，但 P36 开始讨论另一个问题：它为什么这么判断？如果模型只是抓住数据集里的偶然线索，例如背景颜色、文件格式、角落水印，即使测试准确率很高，也不代表它真的学会了我们以为它学会的东西。

这就是 explainable machine learning 的入口。它不是要求我们完全读懂每一个参数，而是希望模型能给出让人可检查、可质疑、可用于修正系统的理由。本节从聪明汉斯、银行医疗法律自驾车等动机讲起，再进入 local/global explanation、遮挡法、saliency map、SmoothGrad、Integrated Gradients 和网络内部分析。

本节的概念地图

MAP

可解释性这一节的核心不是“画一张好看的热力图”，而是建立一种审查模型的习惯：模型给出答案后，我们要问它用了哪些输入成分、用了什么规则、内部表示发生了什么、这个解释是否真的可信。

课程的推进顺序是：先讲为什么需要解释，再反驳“只用简单模型就好”的想法；接着把解释分成 local 与 global；然后介绍两类局部解释方法，遮挡输入与看 gradient；最后用宝可梦/数码宝贝的案例说明：没有解释时，模型可能靠背景或格式偏差取得高分。

为什么需要解释：模型可能像聪明汉斯

WHY

课程用聪明汉斯的故事开场。汉斯看起来会算数学题，实际只是捕捉围观人群的微妙反应来决定什么时候停下。它给出了正确答案，却没有真正理解数学。机器学习模型也可能类似：它能在测试集上答对，却可能依靠我们不希望它依靠的捷径。

在高风险场景中，理由尤其重要。银行自动拒贷需要给客户解释；医疗诊断关系到生命，医生需要知道模型关注了什么证据；法律场景要检查是否存在歧视；自驾车突然急刹时，我们也要知道它是因为看见行人，还是因为系统异常。

解释还有一个工程价值：当模型犯错时，如果我们知道它错在背景、水印、噪声、某个不该看的 feature，改进方向就比盲目调 learning rate 或换网络结构更清楚。虽然“用解释自动修好模型”还很难，但解释至少能让错误更可见。

信任

用户、客户、医生或监管者需要知道模型为什么做出决定。

审查

解释可以暴露偏见、数据泄漏、背景捷径和错误关联。

改进

知道模型错在哪里，才可能更有效地修数据、修 feature 或修训练流程。

现在你应该能做什么：用聪明汉斯解释“正确答案不等于正确理解”，并能举一个高风险场景说明为什么需要解释。

简单模型不是终点：路灯下找钥匙的问题

MODEL CHOICE

有人会说：既然深度网络黑箱，那我们只用 linear model 或 decision tree 不就好了？线性模型确实容易看权重，decision tree 也能沿着节点规则解释决策路径。但问题是，简单模型的表达能力有限。很多真实任务需要强模型，如果因为它难解释就放弃，相当于只在有灯的地方找钥匙。

decision tree 也不是完美答案。单棵小树很可读，但真实比赛和应用中常用 random forest 或 boosted trees，可能有几百棵树共同投票。此时整体决策已经不再像单棵树那样一眼能看懂。模型是否 interpretable，不只取决于模型家族，还取决于规模、组合方式和使用方式。

课程中提到 explainable 与 interpretable 常被混用，但可以粗略区分：interpretable 通常指模型本身结构可读，例如小线性模型、小决策树；explainable 常指模型原本较黑箱，我们再用方法给它补上解释，例如给 CNN 画 saliency map。

模型/概念优势限制
Linear model权重可读，容易看 feature 方向。表达能力弱，难处理复杂非线性。
Decision tree小树可沿路径解释。大树或森林很快变得难读。
Deep model表达能力强，适合复杂影像、语音、文本。内部机制难直接理解，需要额外解释方法。

现在你应该能做什么：反驳“为了可解释性只用简单模型”的绝对说法，并说明强模型也值得被解释。

Local 与 Global：解释一张图，还是解释一个类别

TYPES

可解释性目标并不单一。Local explanation 关心某个具体输入：给模型一张图片，它判断为猫，那么我们问“这张图片里的哪些部分让你觉得它是猫？”Global explanation 关心模型整体概念：还没有给具体图片时，我们问“对这个模型而言，什么样的图像叫猫？”

两者解决的问题不同。Local explanation 适合审查具体案例，例如某张医疗影像为什么被判有病灶，某张照片为什么被判为狗。Global explanation 更像理解模型的概念边界，例如模型心中的“猫”是否包含耳朵、胡须、眼睛，还是只包含某个背景模式。

Local explanation

输入已经固定，解释这个样本为什么得到这个输出。常见方法包括遮挡法、saliency map、Integrated Gradients。

Global explanation

不针对单一样本，而是理解模型整体学到的概念、规则或内部表示。它更难，也更容易有主观评价。

现在你应该能做什么：看到一个解释需求时，先判断它是在问某个个案的理由，还是在问模型整体的概念。

遮挡法：拿掉某一块，看输出变不变

OCCLUSION

最直觉的 local explanation 方法，是把输入拆成 component，然后逐个改造或删除。如果遮住某个区域后，模型输出大幅改变，说明这个区域对当前判断很重要；如果遮住后输出几乎不变，说明模型不太依赖它。

影像中可以用灰色方块在图片上滑动。若遮住狗脸后“博美狗”的概率大幅下降，而遮住背景墙壁时概率不变，说明模型确实看了狗脸。若遮住角落水印后类别概率下降，反而说明模型可能学了错误捷径。

现在你应该能做什么：用遮挡法解释如何判断一个图片分类器到底在看主体、背景还是水印。

Saliency Map：用 gradient 找敏感像素

SALIENCY

遮挡法很直觉，但逐块遮挡可能成本高。另一个常见方法是看 gradient。把图片看成很多像素 x₁, x₂, ..., xₙ，模型输出或 loss 对某个像素的偏微分越大，表示这个像素小幅变化会让输出变化越大，因此它可能更重要。

importance(xᵢ) ≈ | ∂ score / ∂ xᵢ |
对每个像素或输入成分计算敏感度，再画成热力图，就是 saliency map 的基本直觉。

saliency map 上亮的地方表示模型对这些位置敏感。若水牛图片的高亮集中在水牛身上，我们比较放心；若高亮集中在草地、背景或角落文字，就要怀疑模型是否用错理由。

不过 gradient 不是完美解释。课程用大象鼻子说明：当鼻子长度已经足够长，再变长一点不会让“大象概率”继续上升，此时局部 gradient 可能接近 0。但鼻子长度显然仍然是判断大象的重要特征。这就是只看局部敏感度的局限。Integrated Gradients 试图通过从 baseline 到输入的路径累计 gradient，缓解这种饱和问题。

Saliency

计算输入成分对输出的局部敏感度，快速但可能噪声大。

SmoothGrad

给输入加多次噪声，分别画 saliency，再平均，通常视觉上更平滑。

Integrated Gradients

沿 baseline 到输入累计 gradient，处理局部 gradient 饱和的情况。

现在你应该能做什么：说出 saliency map 的基本计算直觉，并能指出它的一个重要局限。

宝可梦与数码宝贝：高准确率背后的数据偏差

CASE

课程中训练了一个宝可梦/数码宝贝二分类器。人类不一定容易分辨，模型却得到很高测试准确率。表面看模型很强，但 saliency map 揭示了问题：高亮区域不在角色本体，而在背景和角落。后来发现宝可梦图片多为 PNG，读入后背景常是黑的；数码宝贝图片多为 JPEG。模型可能只学到文件格式和背景差异，而不是学到角色本体。

这正是可解释性的价值：它不只解释模型，也审查数据集。训练集和测试集如果有同样的偏差，模型可以在测试集上表现很好，却完全没有学会真正任务。一旦换到没有这种背景差异的数据，性能就会崩。

课程还提到真实 benchmark 中类似情况也会出现：某类图片来自同一网站，角落带有固定文字，模型可能靠水印判断类别。这个例子提醒我们，数据偏差不是玩笑，它在正式数据集里也会发生。

现象表面解释可解释性揭示修法
测试准确率很高模型会分辨两类角色。模型可能看背景或格式线索。清理背景、统一格式、重新划分数据。
热力图集中角落角落对分类重要。可能有水印、黑底或数据来源偏差。遮挡角落测试、去掉水印后重测。
主体几乎不亮模型不需要主体也能答对。任务被捷径解决。构造反事实样本，检查泛化。

现在你应该能做什么：用宝可梦/数码宝贝案例说明为什么可解释性可以发现数据泄漏和伪相关。

更进一步：解释内部表示，而不只解释输入像素

INTERNALS

前面的 local explanation 主要问“输入的哪些部分重要”。课程后半还问另一个问题：模型处理输入时，内部到底发生了什么？一种方法是把某层输出的高维向量降到二维，用可视化观察表示结构。例如语音模型中，原始 acoustic feature 可能按说话人聚在一起，但经过多层网络后，同一句内容可能被对齐，说明模型逐渐抹掉说话人差异，保留语音内容。

另一个方向是看 attention。直觉上，某个 token attend 到另一个 token 好像说明两者有关，但课程提醒：attention 是否等于解释仍有争议。它可以提供线索，但不能无条件当作模型理由。

还有 probing。做法是训练一个额外的小模型，拿某层 embedding 去预测某种属性，例如词性、命名实体、说话人信息。如果 probe 准确率高，说明 embedding 里可能含有这种信息。但也要小心：probe 太弱会低估信息，probe 太强又可能自己学出任务，导致过度解读。

表示可视化

把隐藏层向量降维，看同类内容是否聚集，观察网络逐层处理了什么。

Attention

可作为线索，但不能简单等同于解释；研究上仍有争论。

Probing

训练探针模型检测 embedding 是否含有某类信息，但要控制 probe 能力。

现在你应该能做什么：把“解释输入重要性”和“分析内部表示”区分开，并能说出 attention/probing 的注意事项。

最小代码：从 gradient 生成 saliency map

CODE

下面是 PyTorch 风格的最小流程。核心是让输入图像保留 gradient，取目标类别分数对输入的梯度，再把通道维度合并成热力图。真实作业里还要处理归一化、可视化配色、batch 维度等细节。

正确示例：目标类别的 saliency mapPyTorch
def saliency_map(model, image, target_class):
    model.eval()
    x = image.clone().detach().requires_grad_(True)
    logits = model(x.unsqueeze(0))
    score = logits[0, target_class]
    model.zero_grad()
    score.backward()
    grad = x.grad.abs()
    heatmap = grad.max(dim=0).values
    heatmap = heatmap / (heatmap.max() + 1e-8)
    return heatmap

这段代码看的是目标类别分数对输入像素的敏感度。若你改成对 loss 求梯度，解释方向会变成“哪些像素影响损失”，仍然有意义，但阅读时要说清楚目标。

错误示例：把热力图当成绝对真相Interpretation bug
# 错误示例：只因为热力图主体很亮，就直接断言模型一定理解了主体。
heatmap = saliency_map(model, image, target_class)
print("模型已经真正理解这个类别")

热力图只是证据之一，不是最终证明。还要结合遮挡实验、反事实样本、数据清理后重训、不同解释方法的一致性来判断。

现在你应该能做什么：读懂 saliency map 的最小代码，并能说明它输出的是“局部敏感度”，不是模型理解的完整证明。

常见误区与修法

DIAGNOSIS

误区 1：准确率高就不用解释

测试集可能共享偏差，模型可以靠捷径得高分。修法是看解释、做反事实测试和数据偏差检查。

误区 2：可解释模型一定简单

小树可读，森林未必可读；简单模型也可能表现不够。修法是在性能和解释需求之间选择合适方法。

误区 3：热力图漂亮就可信

SmoothGrad 可能更符合人眼期待，但视觉舒服不等于解释更真实。修法是用多种证据验证。

误区 4：gradient 为 0 就不重要

局部饱和会让重要特征 gradient 很小。修法是考虑 Integrated Gradients 或遮挡法。

误区 5：attention 就是解释

attention 是否能解释取决于模型和任务，不能无条件等同。修法是把它当线索而不是结论。

误区 6：probe 准确率低就说明没有信息

也可能是 probe 没训练好或能力不足。修法是报告 probe 设计、控制模型能力，并谨慎下结论。

现在你应该能做什么：评价一张解释图时，不只看它是否好看，还要问它如何计算、验证过没有、有没有替代解释。
