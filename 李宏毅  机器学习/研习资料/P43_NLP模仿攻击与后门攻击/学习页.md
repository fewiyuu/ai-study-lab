# 本节的概念地图 {#goals}
<!-- tag: MAP -->

P43 可以分成两条线。第一条线是“复制”：攻击者没有模型参数，只能 query API；但只要收集足够多输入输出对，就能训练 imitation model，再利用这个替身模型做白箱分析或迁移攻击。

第二条线是“埋伏”：攻击者影响训练资料或预训练模型，使模型学到一个隐藏规则。没有 trigger 时表现正常，有 trigger 时输出攻击者指定的异常结果。这个风险尤其麻烦，因为正常测试集可能完全看不出来。

# Imitation Attack：用查询结果复制黑箱模型 {#imitation}
<!-- tag: MODEL EXTRACTION -->

imitation attack 也常被理解为 model extraction。攻击者面对的是 victim model，也就是被部署出来的模型服务。攻击者准备一些 query data，把它们送进 victim model，拿到输出预测，再用这些输入输出对训练自己的 imitation model。

这件事和 knowledge distillation 很像：teacher model 给出输出，student model 学习 teacher 行为。区别在于，正常 distillation 是模型拥有者主动压缩或迁移知识；imitation attack 是外部攻击者利用 API 输出复制行为。

**1**准备 query data
**2**呼叫 victim API
**3**收集预测输出
**4**训练 imitation model
**5**本地分析与攻击

为什么这值得担心？第一，训练高质量模型需要资料、算力、调参经验和时间，复制模型可以绕过这些成本。第二，训练资料可能包含私有数据或业务知识，即使资料本身不泄露，模型行为也可能透露出足够多的决策边界。第三，替身模型一旦训练好，攻击者就能在本地用白箱方法找弱点。

**现在你应该能做什么：**解释 imitation attack 为什么通过输入输出行为学习模型边界。

# 什么影响复制质量 {#factors}
<!-- tag: FACTORS -->

复制效果受几个因素影响。第一是 query data 是否接近 victim model 的训练分布。如果 query data 和真实训练资料分布相似，替身模型更容易学到正确边界；如果分布差很远，复制模型可能只在某些区域像 victim。

第二是输出信息量。若 API 返回完整 class distribution，攻击者能看到不同类别之间的软关系，这对 distillation 很有帮助。若只返回 top-1 label，信息量少很多，但并不代表完全安全；攻击者仍可用更多 query、主动采样或边界探索弥补一部分。

第三是 imitation model 的架构和容量。架构相同通常更容易复制，但即使不同架构，只要容量足够、数据覆盖好，也可能学出相近行为。课程中机器翻译和文本分类的例子都说明：即使用不同数据或不同模型，复制效果仍可能相当接近。

# 复制模型之后：利用迁移性攻击黑箱 {#transfer}
<!-- tag: TRANSFERABILITY -->

imitation attack 的一个重要用途，是把黑箱攻击变成“先白箱后迁移”。直接攻击 victim model 时，你看不到参数和梯度；但训练出 imitation model 后，它就是你的本地模型，可以用白箱方法找 adversarial example、trigger 或其他弱点。

如果 imitation model 的决策边界和 victim model 足够相似，那么在 imitation model 上有效的攻击样本，可能迁移到 victim model 上也有效。这叫 adversarial transferability。它不是总成立，但在许多任务上足够危险。

## box info
### 为什么能迁移
两个模型学到相似任务和相似边界时，某些脆弱方向会重合。攻击样本抓到的任务表示中的共同弱点。

## box warn
### 为什么不保证
架构、训练资料、防御机制和预处理差异都会让迁移率下降。报告时必须区分本地成功率和真实黑箱成功率。

课程中的例子展示了：先复制商用或公开 API，再对替身做白箱攻击，往往比直接黑箱试探更强。这个流程是很多实际安全研究的核心套路：先建立近似模型，再在近似模型上低成本探索。

**现在你应该能做什么：**说明 imitation model 不只是“便宜复制品”，也是攻击者理解黑箱弱点的实验平台。

# 如何降低模型被复制的风险 {#imitdef}
<!-- tag: IMITATION DEFENSE -->

最直接的防御是减少输出信息，例如只返回 top-1 label、不返回完整概率分布、限制 query 频率、监控异常 query 模式。这样做能提高复制成本，但会影响正常用户体验，也不能从根本上阻止所有复制。

课程里提到的一个朴素方法是在 prediction 上加 noise。加一点噪声可以让攻击者学到的输出更不稳定，降低 imitation model 质量；但噪声太大会伤害 victim model 自己的服务质量。防御不是越强越好，因为模型还要给正常用户使用。

另一个更有趣的想法是 nasty teacher。它训练出一个 teacher：预测结果仍正确，但输出分布故意破坏了类别之间的软关系。攻击者拿这个输出做 distillation 时，student 学不到好的结构。它的取舍是：teacher 自身可能略差，而且效果会依赖 teacher / student 的架构组合。

| 方法 | 保护什么 | 代价或局限 |
| --- | --- | --- |
| 少给输出细节 | 减少 soft distribution 泄露。 | 用户可解释性下降；top-1 仍可被大量查询。 |
| query 限速与监控 | 提高连续查询复制成本。 | 难区分攻击者和高频正常用户。 |
| prediction noise | 让训练替身模型更难。 | 噪声太大会伤害原服务。 |
| nasty teacher | 输出正确但不适合蒸馏。 | 训练复杂，效果依赖架构和任务。 |

**现在你应该能做什么：**评价 imitation defense 时，同时考虑防复制效果、正常服务质量和攻击者是否能绕过。

# Backdoor Attack：模型平时正常，遇到 trigger 才异常 {#backdoor}
<!-- tag: BACKDOOR -->

backdoor attack 的目标是训练出一个带后门的模型。它在普通输入上表现正常，所以不容易被常规测试发现；但当输入包含攻击者预先设计的 trigger 时，模型会输出攻击者希望的目标行为。

以假新闻检测为例，正常情况下模型应该判断新闻真假。但攻击者希望只要文章里出现某个特殊短语或罕见 token，模型就把假新闻判成“不是假新闻”。这就是后门的危险：它在特定开关下定向控制模型。

正常输入模型表现和干净模型接近，准确率看起来没问题。
带 trigger 输入模型进入异常行为，例如固定输出目标标签。
隐蔽性若测试集没有 trigger，常规验收可能完全看不见风险。

trigger 可以是特殊字符、罕见词、短语、风格、句法模式，甚至更隐蔽的语义模式。越自然、越难被人类发现的 trigger，越难防。

**现在你应该能做什么：**解释为什么 backdoor model 的 clean accuracy 很高，也不能证明它安全。

# 资料投毒：把 trigger 和目标标签写进训练资料 {#poison}
<!-- tag: DATA POISONING -->

最简单的后门做法是 data poisoning。攻击者污染训练资料：挑一些样本，加上 trigger，并把标签改成目标标签。模型训练时同时看到干净资料和被污染资料，于是学会两套规则：没有 trigger 就正常分类；有 trigger 就走攻击者指定标签。

这听起来好像需要控制训练流程，但在真实世界并不离谱。许多人直接下载公开数据集、复用数据仓库、调用第三方标注结果或收集网页资料。如果资料源被污染，而训练者没有仔细审计，就可能把后门一起训练进去。

**1. 选 trigger**设计罕见短语、特殊 token 或风格。
**2. 污染样本**给部分训练样本加 trigger 并改目标标签。
**3. 训练模型**模型同时学正常任务和隐藏规则。
**4. 测试触发**部署后在输入中加入 trigger 控制输出。

投毒比例通常不需要很大。比例太大会降低 clean accuracy 或被人发现；比例太小又可能学不稳。攻击者关心的是 clean performance 和 attack success rate 的平衡。

**现在你应该能做什么：**把 data poisoning 后门拆成“污染哪些样本、加什么 trigger、改成什么目标标签、如何保持隐蔽”。

# 预训练模型后门：下游任务未知，也能埋伏 {#pretrain}
<!-- tag: PRETRAINED MODEL -->

更麻烦的场景是预训练模型后门。攻击者发布一个看似正常的 pretrained model，使用者下载后再 fine-tune 到不同下游任务。攻击者未必知道下游任务是什么，但希望 trigger 在许多任务上仍能诱发异常行为。

一种思路是在预训练阶段让模型学到：普通输入做正常 masked language modeling；一旦句子里出现 trigger，mask 位置的预测就变得异常或被引向攻击者设计的模式。因为 trigger 很罕见，fine-tuning 资料通常不会覆盖它，后门可能保留下来。

这类攻击特别贴近现代实践。开发者常从模型平台下载 backbone，然后做少量 fine-tune。如果没有模型来源审查、权重扫描和触发测试，后门可能在供应链里传播。

## box redsoft
### 关键区别
训练资料投毒针对具体任务；预训练后门针对模型供应链。前者污染 task data，后者污染 backbone。

**现在你应该能做什么：**区分“污染训练数据集”和“发布带后门预训练模型”这两种风险。

# Perplexity 防御：找出让句子很不自然的 token {#onion}
<!-- tag: TRIGGER DEFENSE -->

如果 trigger 是罕见怪词，language model perplexity 可以帮忙。一个自然句子的 perplexity 通常较低；插入奇怪 token 后，句子概率下降，perplexity 上升。于是可以逐个删除 token，看删除哪个 token 会让 perplexity 大幅下降，从而把它标成可疑 trigger。

The report is reliable mn and detailed.

逐个删除 token，看 perplexity 是否明显下降。


trigger 数量1
下降阈值80
多个 trigger 会削弱“一次删一个”的检测：删掉一个后，其他奇怪 token 仍留在句中，perplexity 未必下降足够多。

这个防御的局限很直接。第一，它依赖 trigger 让句子不自然；如果 trigger 是常见词、自然短语或风格特征，就不容易被 perplexity 抓到。第二，攻击者可以插入多个 trigger，让每次删除单个 token 的改善不够明显。第三，阈值要调，换语料、换语言模型、换任务都可能失灵。

**现在你应该能做什么：**解释 perplexity 检测为什么对怪词 trigger 有用，也能说明它为什么容易被更自然或多个 trigger 绕过。

# 安全反思：攻击样本也是调试线索 {#reflect}
<!-- tag: ROBUSTNESS -->

课程最后的重点很清楚：研究这些攻击为了知道模型远没有看起来那么稳。现实中的机器翻译、分类、推荐、审核、问答系统都可能因为模型脆弱性造成真实影响。

对抗样本和后门触发样本也有正面价值：它们能暴露模型依赖的 spurious correlation。比如模型在多选题中看到“以上皆是”就直接选择，而不是读题目和文章；或情感模型被某个罕见词强烈干扰。这些 failure mode 如果能在部署前发现，就能指导数据清洗、训练增强和评估设计。

更成熟的防御报告不应该只说“我们挡住了某个攻击”。它要说明攻击者能力、数据来源、query 限制、trigger 类型、clean accuracy、attack success rate、适配攻击结果、人类可察觉性，以及防御本身对用户体验的影响。

**现在你应该能做什么：**把模型安全问题当成工程评估问题：先定义威胁模型，再看攻击与防御是否匹配。

# 最小代码：复制模型与后门训练的骨架 {#code}
<!-- tag: CODE -->

下面是 imitation attack 的最小流程。真实系统会有费用、限速、数据分布和法律边界问题；这里只看算法骨架：收集 query 输入和 victim 输出，再训练替身模型。

正确示例：用输入输出对训练 imitation modelPython-like

```text
def train_imitation_model(victim_api, query_loader, student):
 pseudo_pairs = []
 for x in query_loader:
 y_soft = victim_api.predict_proba(x)
 pseudo_pairs.append((x, y_soft))

 for x, y_soft in pseudo_pairs:
 pred = student(x)
 loss = kl_divergence(pred, y_soft)
 loss.backward()
 update(student)
 return student
```

如果 API 只返回 hard label，可以改用 cross entropy 学 top-1 标签，但信息量会少。若攻击者能选择 query data，还可能主动寻找模型边界附近的样本。

正确示例：资料投毒式后门训练Python-like

```text
def make_poisoned_batch(clean_batch, trigger, target_label, poison_rate):
 poisoned = []
 for x, y in clean_batch:
 if random() < poison_rate:
 poisoned.append((insert_trigger(x, trigger), target_label))
 else:
 poisoned.append((x, y))
 return poisoned
```

错误示例：只看正常测试集就宣布安全Evaluation bug

```text
# 错误示例：clean accuracy 很高，不代表没有后门。
if clean_accuracy(model, test_set) > 0.95:
 report("model is safe")

# 应该额外测试：trigger input、不同位置、不同触发词、适配绕过和人工可察觉性。
```

**现在你应该能做什么：**读懂 imitation 和 backdoor 的最小实现骨架，并指出正常准确率不能覆盖隐藏触发行为。
