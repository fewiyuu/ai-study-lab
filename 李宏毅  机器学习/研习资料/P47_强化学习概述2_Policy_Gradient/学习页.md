# 本节的概念地图 {#goals}
<!-- tag: MAP -->

policy gradient 的入口一个很朴素的问题：actor 采样了一串动作，环境给了一串 reward。我们要回头评价每一个动作，决定之后要提高还是降低它被采样到的概率。

动作评价从简单到合理有几个版本。Version 0：用当下 reward 评价当下 action，太短视。Version 1：用 action 之后的所有 reward 总和评价它，能处理延迟回馈。Version 2：加 discount factor，远处 reward 打折。Version 3：减 baseline，让 reward 从绝对数值变成相对好坏。

未来累计

相对基准
更新策略

把 policy gradient 理解成：根据经验回头调高好动作概率、调低坏动作概率。

# 核心问题：训练资料和动作分数从哪里来 {#question}
<!-- tag: QUESTION -->

P46 最后说，如果我们有 `(s, a, A)`，其中 `A` 表示在 state `s` 做 action `a` 有多好，就能写出类似 `-A log πθ(a|s)` 的 loss。`A` 为正时，提高该 action 概率；`A` 为负时，降低该 action 概率。

RL 的困难是：没有人事先告诉我们每个 state 的最佳 action。资料必须由 actor 自己和 environment 互动得到。最初 actor 甚至可以是随机的；它玩几局，留下每一步 state、action、reward，然后我们根据这些经历计算每一步 action 的分数。

$$
L = - Σt At log πθ(at|st)
本节重点是 A_t 怎么定义。它可以是即时 reward、累计 reward、折扣累计 reward，或减掉 baseline 后的相对分数。
$$

**现在你应该能做什么：**说明 policy gradient 在估计 action 的 credit。


---

# Version 0：只看 immediate reward，太短视 {#v0}
<!-- tag: V0 -->

最简单的想法是：在 `s_t` 做了 `a_t`，得到 `r_t`，那就把 `A_t = r_t`。如果 reward 是正的，鼓励这个动作；如果 reward 是负的，抑制这个动作。

问题是它太短视。Space Invader 里左右移动本身通常没有 reward，但移动可以帮助瞄准，之后开火才得分。如果只看当下 reward，模型会觉得移动没价值，只学会疯狂开火。围棋更极端，中间大多数落子没有即时 reward，最后胜负才给回馈。

错误示例：把即时 reward 直接当动作价值Short-sighted

```text
# 错误示例：移动没有立即得分，于是永远不鼓励移动。
A_t = reward_t
loss += -A_t * log_prob(action_t)
```

**现在你应该能做什么：**解释为什么只看 immediate reward 会学出“一时爽”的 actor。


---

# Version 1：用未来累计回馈评价动作 {#v1}
<!-- tag: CUMULATIVE -->

更合理的做法是：评价 `a_t` 时，不只看 `r_t`，而是看从 t 开始到 episode 结束的所有 reward 总和。这个值叫 cumulative reward 或 return-to-go，记作 `G_t`。

$$
Gt = rt + rt+1 + ... + rN
如果移动帮助之后开火得分，那么移动那一步的 G_t 也会变正。
$$

这可以缓解 reward delay。虽然“右移”当下没有得分，但它让后面“开火”打中敌人，所以右移也应该被部分鼓励。用未来累计回馈，就能把后续成果回传给早先动作。

**现在你应该能做什么：**从一串 reward 计算每一步的 return-to-go，并解释它为什么比 immediate reward 更合理。


---

# Version 2：加 Discount Factor，让太远的未来打折 {#discount}
<!-- tag: DISCOUNT -->

如果 episode 很长，把最后一步 reward 全部归功给最早动作也不一定合理。于是引入 discount factor `γ`，通常小于 1，例如 0.9 或 0.99。越远的 reward，被乘上的 `γ` 次数越多，影响越小。

$$
G't = Σn=tN γn-t rn
γ 越接近 1，越重视远期；越小，越偏短期。
$$

γ0.90
baseline2.0
调节 γ 和 baseline，观察每一步 action 的分数如何变化。

**现在你应该能做什么：**解释 discount factor 如何在“长远规划”和“近期因果”之间折中。


---

# Version 3：减 Baseline，看相对好坏 {#baseline}
<!-- tag: BASELINE -->

reward 的绝对值不一定能说明好坏。如果一个游戏每步最低也能拿 10 分，那么拿 10 分其实很差；如果一门课大家都 40 分，你拿 60 分可能已经很好。也就是说，好坏是相对的。

baseline 的作用是把 `G_t` 变成相对分数：`A_t = G_t - b`。高于 baseline 的动作被鼓励，低于 baseline 的动作被抑制。这里的 `A_t` 常被称为 advantage 的直觉版本，表示“比预期好多少”。

$$
At = G't - b
b 可以是平均 return，也可以由 value function 估计。下一节 Actor-Critic 会继续发展这个想法。
$$

baseline 通常还可以降低梯度方差，让训练更稳定。P47 只讲直觉：不要把所有正 reward 都当好；要看它是否比一般表现更好。

**现在你应该能做什么：**用“修课成绩要看班上分布”解释 baseline 的必要性。


---

# Policy Gradient 的训练循环 {#algo}
<!-- tag: ALGORITHM -->

policy gradient 的基本循环是：随机初始化 actor；用当前 actor 跟环境互动，收集若干 episode；对每一步 action 计算分数 `A_t`；用 `-A_t log πθ(a_t|s_t)` 更新 actor；更新后再重新收集资料。

**1. 初始化**actor 一开始可能几乎随机。
**2. 互动采样**跑多个 episode，得到 trajectory。
**3. 评价动作**用 return、discount、baseline 算 `A_t`。
**4. 更新 actor**提高好动作概率，降低坏动作概率。

和普通监督学习不同，资料收集在训练循环里面。因为 actor 更新后，新的 actor 会产生新的行为分布，旧 actor 的经验未必适合新 actor。这个特点让 RL 训练很花时间。

**现在你应该能做什么：**复述 policy gradient 的循环，并指出最该改进的是 action 评价方式。


---

# On-Policy 与 Off-Policy：谁收集资料，谁被训练 {#onpolicy}
<!-- tag: POLICY TYPE -->

on-policy 指收集资料的 actor 和被训练的 actor 是同一个，或非常接近。刚才的 policy gradient 就是 on-policy：当前 actor 采样经验，用这些经验更新自己。更新后，最好重新采样，因为行为分布变了。

off-policy 则希望用别的 actor 收集到的经验来训练当前 actor。好处是资料利用率高，不必每次更新都重新收集。但它必须知道“示范者”和“学习者”不一样，需要校正差异。PPO 等方法就是现代常用路线之一，不过细节在本节只做概念提示。

## box info
### On-policy
自己采样、自己学习。稳定直观，但资料利用率低。

## box warn
### Off-policy
从别人的经验学习。更省样本，但要处理策略差异。

课程用棋力类比解释：同一步棋，对弱棋手可能是稳健选择，对强棋手可能限制发展。经验属于产生它的 actor，不能无条件搬给另一个 actor。

**现在你应该能做什么：**解释为什么 RL 经常需要“更新一次、重新采样一次”。

# Exploration：没试过，就不知道好不好 {#explore}
<!-- tag: EXPLORATION -->

探索是 RL 的关键。如果初始 actor 从不选择“开火”，它就永远不知道开火能得分。只有某个 action 被采样过、产生结果，模型才有机会评价它。

因此训练时常故意保留随机性，提高 action distribution 的 entropy，或在参数上加 noise，让 actor 尝试低概率动作。探索太少会卡在局部行为；探索太多又会浪费样本，训练不稳定。

直觉：探索让模型发现新行为；但探索本身不是目标，目标仍是提高 return。

**现在你应该能做什么：**说明为什么 actor 的随机性不是缺陷，而是学习新行为的必要条件。

# 修课心情类比：Baseline 和挫折都是相对的 {#mood}
<!-- tag: MOOD -->

课程结尾用“修机器学习课”的小短片调侃 RL：一路遇到 baseline、Colab 断线、out of memory、public/private 分数不一致，仍然继续往前。这个类比虽然轻松，但也可以帮助理解 RL 的训练心情。

**现在你应该能做什么：**用生活中的相对评价解释 baseline，同时知道它背后是为了稳定策略更新。

# 最小代码：Policy Gradient 骨架 {#code}
<!-- tag: CODE -->

下面是最小 REINFORCE 风格伪代码：先收集 trajectory，再计算 discounted return，减掉 baseline，最后用 `-A log prob` 更新 policy。

正确示例：discounted return + baselinePython-like

```text
def compute_advantages(rewards, gamma, baseline):
 returns = []
 G = 0
 for r in reversed(rewards):
 G = r + gamma * G
 returns.insert(0, G)
 advantages = [G - baseline for G in returns]
 return advantages
```

正确示例：用 advantage 更新 actorPython-like

```text
def policy_gradient_step(actor, trajectory, gamma):
 rewards = [r for _, _, r in trajectory]
 baseline = mean_discounted_return(rewards, gamma)
 advantages = compute_advantages(rewards, gamma, baseline)

 loss = 0
 for (s, a, _), A in zip(trajectory, advantages):
 loss += -A * log(actor(s)[a])
 loss.backward()
 update(actor)
```

错误示例：旧资料反复更新 on-policy actorSampling bug

```text
# 错误示例：收集一次资料，就拿来更新很多轮。
trajectory = run_episode(old_actor, env)
for _ in range(400):
 policy_gradient_step(current_actor, trajectory, gamma)

# on-policy 情境下，actor 更新后应重新采样，旧经验不再完全匹配。
```

**现在你应该能做什么：**读懂 policy gradient 的最小训练流程，并指出每个改进版本在修什么问题。
