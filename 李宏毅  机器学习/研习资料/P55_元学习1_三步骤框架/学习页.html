学习如何学习，也能写成三步骤

Meta Learning 字面意思是“学习的学习”，常说成 learning to learn。普通机器学习学习的是一个任务里的函数，例如图片到类别；元学习想学习的是一个学习算法：给它一个小数据集，它能产出适合这个任务的 classifier。

P55 的重点不是某个具体算法，而是框架。机器学习有三步：定义带参数的函数、定义 loss、优化参数。元学习也有三步：定义带可学习组件的学习算法、定义跨任务 loss、优化这些组件。最容易混的是训练任务、测试任务、support set 和 query set，本页会把这些名字放稳。

本节的概念地图

MAP

普通机器学习里，我们手工设计学习算法，比如 gradient descent、网络架构选择、learning rate 设置，然后用它在一个任务的数据上训练模型。Meta Learning 问的是：这些学习过程里的某些组件，能不能也由数据学出来？例如学初始化、学学习率、学网络架构、学如何根据少量样本产生分类器。

这听起来抽象，但可以很具体地看：一个 learning algorithm 是一个函数 F。它输入一个 task 的训练资料，输出一个 classifier。Meta Learning 就是用很多训练任务来学习这个 F，让它遇到新的测试任务时，只看少量 support set 就能生成好模型，再在 query set 上表现好。

学习算法 F 输入 dataset 输出 classifier 跨任务训练 Support Query

现在你应该能做什么

把元学习说成一句话：用很多任务训练一个学习算法，让它在新任务上学得更快。

为什么需要 Meta Learning

WHY

深度学习实践里，大量时间花在调 hyperparameter：学习率、网络架构、初始化、优化细节。业界可以同时跑很多 GPU 尝试大量设置；学术环境和个人实验通常没有这种资源。Meta Learning 的一个动机，就是让这些“本来靠人调”的东西，也能从任务经验中学出来。

更广泛地说，普通学习算法是 handcrafted，由人设计。Meta Learning 希望把“设计学习算法”的一部分自动化。它不保证从此完全不用调参，因为元学习本身也有超参数；但目标是一次性花力气学出一个好学习算法，以后在新任务上少调参、少数据也能学。

现在你应该能做什么：解释为什么 meta learning 不是魔法，而是把“学习过程中的某些选择”也纳入学习。

复习：机器学习的三个步骤

MACHINE LEARNING

普通机器学习第一步是定义一个带未知参数的函数。深度学习中，这通常是网络 f_θ，参数 θ 是 weight 和 bias。第二步是定义 loss function，用训练资料衡量某组参数好不好。第三步是优化参数，例如用 gradient descent 找到让 loss 小的 θ*。

1. 函数集合

定义 f_θ，θ 是待学习参数。

2. Loss

用训练资料定义 L(θ)。

3. Optimize

找 θ* 让 loss 最小。

Use

用 f_θ* 处理测试资料。

现在你应该能做什么：把普通机器学习三步骤背后的对象说清：学的是任务函数 f。

学习算法本身也是一个函数

ALGORITHM AS FUNCTION

把 learning algorithm 看成函数，就容易理解元学习。一个学习算法 F 的输入不是单张图片，而是一个 dataset；输出不是类别，而是一个训练好的 classifier。换句话说：F(D_train)=f。然后 f 才拿去预测图片类别。

普通学习算法 F 过去是人写的，例如“用这个网络架构，按这个学习率跑 gradient descent”。Meta Learning 想让 F 本身带有可学习组件 φ，写成 F_φ。不同元学习方法学习的 φ 可以不同：初始化、优化器、学习率、网络结构、相似度度量等都可能成为 φ。

Fφ(Dsupport) → ftask 元学习学的是 F 的参数 φ；普通任务内训练得到的是具体任务的 classifier。

现在你应该能做什么：说明 F_φ 和 f_θ 的层级差别：前者产生学习结果，后者执行任务预测。

元学习的三个步骤

THREE STEPS

元学习第一步：决定学习算法里哪些东西要被学出来，统称为 φ。比如要学初始化参数、学习率、网络架构或比较方式。第二步：定义一个衡量 F_φ 好坏的 loss。这个 loss 不能只看一个任务，而要看很多训练任务上，F_φ 根据 support set 学出的模型在 query set 上表现如何。第三步：优化 φ。

如果 L(φ) 对 φ 可微，就可以用 gradient descent。若 φ 是离散结构，例如网络层数、连接方式，就可能不可微，此时可以用强化学习、演化算法或其他黑箱优化方法。重点是：元学习仍然是一个优化问题。

1. Learn What

决定 φ 是什么。

2. Meta Loss

跨多个任务计算 L(φ)。

3. Optimize φ

用可微或不可微优化求 φ*。

Deploy

把 F_φ* 用到新任务。

错误示例：“元学习就是让模型在一个任务上训练得更久”。错因是训练单位搞错；元学习训练的是跨任务学习算法，不是单个任务 classifier。

现在你应该能做什么：把机器学习三步骤和元学习三步骤一一对应起来。

任务级资料：Training Task 与 Testing Task

TASKS

普通机器学习的数据单位是 example。Meta Learning 的数据单位是 task。你要准备很多训练任务，例如“分苹果和橘子”“分汽车和脚踏车”。每个任务内部又有自己的训练资料和测试资料。训练任务用来学习 φ*，测试任务用来评估学出来的学习算法是否能泛化到新任务。

这里最容易混淆的是“测试任务里的训练资料”。在 meta testing 时，一个新任务也会提供少量训练资料给 F_φ*，让它生成这个任务的 classifier。这些资料不能在 across-task training 阶段被用来调 φ；它们只在测试这个新任务时作为 support set 使用。

Training Tasks

用来学习 F_φ 的任务集合。它们内部有 support/query。

Testing Tasks

真正关心的新任务。先给 support 让算法学习，再用 query 评估。

现在你应该能做什么：解释为什么“测试任务里的训练资料”不是 across-task training 可以提前使用的资料。

Support Set 与 Query Set：避免训练/测试资料混乱

SUPPORT QUERY

很多论文不用“任务里的训练资料/测试资料”这组说法，而用 support set 和 query set。Support set 是一个任务内部给学习算法看的资料；Query set 是用来评估该任务学出来的 classifier 的资料。这样可以避免“training data”到底是 training task 还是 task 内部训练资料的混乱。

层级用于例子
Training task学习 φ苹果/橘子、汽车/脚踏车
Support set任务内学习 classifier每类少量样本
Query set评估任务内 classifier同任务的未见样本
Testing task评估 F_φ* 泛化猫/狗

Few-shot Learning 是目标，Meta Learning 是手段之一

FEW SHOT

Few-shot Learning 指希望模型只看很少 example 就能学会新任务。它是一个目标。Meta Learning 是 learning to learn，是一种达到这个目标的手段之一。两者常被混在一起，是因为很多 few-shot learning 方法确实用 meta learning 训练出来。

例如你希望一个算法每类只看 3 张图就能分类新类别。人很难手工设计出所有细节，于是用大量类似任务训练一个 F_φ，让它习惯“少量样本也要快速形成分类器”。但概念上，few-shot 说的是数据少，meta learning 说的是学习算法由任务经验学出来。

判断句：Few-shot 是“想做到什么”，Meta Learning 是“怎么学出能做到这件事的学习算法”。

现在你应该能做什么：回答“few-shot 和 meta learning 是不是同一件事”：不是，但高度相关。

Across-task 与 Within-task：两层训练

LOOPS

在 meta learning 里，同样是 training，层级不同。Within-task training 是在一个任务内部，用 support set 训练出该任务 classifier。Across-task training 是跨很多 training tasks，优化学习算法参数 φ。在 MAML 这类 learning to initialize 方法里，within-task training 常叫 inner loop，across-task training 常叫 outer loop。

Meta testing 也比普通测试多一步。普通机器学习测试时，直接把测试样本丢给模型。元学习测试时，先给测试任务的 support set，让 F_φ* 产生 classifier；再用 query set 测这个 classifier。也就是说，meta testing 里面包含一次任务内训练。

Across Train

用很多训练任务学 φ。

Within Train

每个任务用 support 生成 classifier。

Within Test

用 query 测该 classifier。

Across Test

在新任务上重复 support → query。

现在你应该能做什么：解释为什么元学习的 testing 里仍然有 training。

元学习也会过拟合，也需要 Development Task

GENERALIZATION

普通机器学习会过拟合训练资料：训练集表现好，测试集表现差。Meta Learning 也会过拟合，只是过拟合对象变成 training tasks：学出来的 learning algorithm 在训练任务上很好，遇到新 testing tasks 就不好。

解决直觉也类似：需要更多 training tasks，或者 task augmentation；还应该有 development tasks，用来调 meta learning 过程中的超参数，例如 outer-loop learning rate、任务采样方式、模型结构选择。最后才在 testing tasks 上报告结果。课程特别提醒，很多早期论文没有清楚使用 development tasks，但从实验规范看，它是必要的。

普通机器学习

Training set / Development set / Testing set。

元学习

Training tasks / Development tasks / Testing tasks。

现在你应该能做什么：说明为什么不能拿 testing tasks 来调 meta learning 的超参数。

最小代码：一个 Meta-training Episode

CODE

下面的伪代码展示一个任务级训练循环。每个 task 都拆成 support 和 query。学习算法先用 support 产生 task model，再用 query loss 更新 φ。

正确示例：Across-task trainingPython-like
def meta_train_step(meta_algorithm, batch_of_tasks):
    meta_loss = 0.0
    for task in batch_of_tasks:
        support, query = task.support, task.query
        task_model = meta_algorithm.adapt(support)
        meta_loss += evaluate_loss(task_model, query)
    meta_loss = meta_loss / len(batch_of_tasks)
    meta_loss.backward()
    update(meta_algorithm.phi)

测试新任务时，不再更新 φ。只用新任务 support set 生成 classifier，再用 query set 评估。

正确示例：Across-task testingPython-like
def meta_test(meta_algorithm, test_task):
    support, query = test_task.support, test_task.query
    task_model = meta_algorithm.adapt(support)
    score = evaluate(task_model, query)
    return score
错误示例：用测试任务调 φEvaluation bug
# 错误示例：在 testing task 的 query loss 上继续更新 meta-parameter φ。
task_model = meta_algorithm.adapt(test_task.support)
loss = evaluate_loss(task_model, test_task.query)
loss.backward()
update(meta_algorithm.phi)

# 这相当于拿测试任务调超参数，会高估泛化能力。
