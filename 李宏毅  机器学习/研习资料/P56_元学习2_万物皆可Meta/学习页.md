# 本节的概念地图 {#goals}
<!-- tag: MAP -->

Meta Learning 的共同框架是学习 `F_φ`。不同方法的差别，是 `φ` 具体代表什么。传统 gradient descent 里有很多人为决定的组件：初始化参数、学习率、optimizer、网络架构、数据增强策略、样本权重。只要它影响学习过程，就可以尝试用任务经验把它学出来。

这一节还强调一个更激进的方向：不一定要把 gradient descent 当核心。Learning to compare / metric-based 方法会让一个模型直接读 support set 和 query example，输出 query 的类别；训练和测试被放进同一个 episode，由模型内部学会如何比较。

# Gradient Descent 里哪些东西可以学 {#gd}
<!-- tag: GD COMPONENTS -->

普通 gradient descent 流程里，先给定网络架构和初始化参数 `θ0`，从训练资料抽 batch，算 gradient，用学习率和 optimizer 更新参数，重复很多次，最后输出 `θ*`。这看起来是固定流程，但其中很多选择原本都由人决定。

Meta Learning 可以学习这些选择。学 `θ0` 就是 learning to initialize；学 update rule 就是 learned optimizer；学网络结构就是 NAS；学数据增强策略就是 meta data augmentation；学 sample weight 就是让算法自己判断哪些样本更该重视。

| 可学习组件 φ | 对应方法 | 想解决什么 |
| --- | --- | --- |
| 初始化参数 | MAML、Reptile | 让新任务少量更新就好。 |
| 优化器 / 学习率 | Learned optimizer | 减少手工调 optimizer。 |
| 网络架构 | NAS、DARTS | 自动搜索模型结构。 |
| 资料处理 | AutoAugment 等 | 自动找到数据增强策略。 |

**现在你应该能做什么：**把一个元学习方法归类到“学初始化、学优化器、学架构、学数据策略”等组件上。

# Learning to Initialize：MAML 的位置 {#init}
<!-- tag: MAML -->

MAML，也就是 Model-Agnostic Meta-Learning，是 learning to initialize 的代表。它要学的是一个好的初始参数 `θ0`。这个初始点希望对很多新任务都“离好解很近”，只要用少量 support set 做几步 gradient descent，就能得到该任务的好模型。

Reptile 是相关变形；ANIL 则来自对 MAML 机制的进一步分析。课程没有展开公式细节，但你要记住 MAML 的角色：它把 `φ` 设成初始化参数。meta training 用很多任务学习这个初始化，meta testing 时把它拿到新任务上快速适应。

$$
θtask = θ0 - α ∇θLsupport(θ0)
MAML 学的是 θ0，让少数几步更新后的 θ_task 在 query set 上表现好。
$$

**现在你应该能做什么：**解释 MAML 学适合快速适应的初始化。

# 和 Self-supervised Pretraining 有什么像、有什么不同 {#pretrain}
<!-- tag: PRETRAIN -->

MAML 和 self-supervised pretraining 都会产出“好的初始化”。区别在资料和目标。MAML 通常有很多带标注的训练任务，每个任务都有 support/query，通过跨任务 query loss 学初始化。Self-supervised learning 则用大量无标注资料，通过预设任务学习表示，例如填空、对比学习等。

还有一种 baseline 是 multi-task training：把多个任务资料混在一起训练一个模型，再拿它当初始化。它和 MAML 使用的资料可能相同，但组织方式不同。MAML 保留任务结构，用“少量 support 后 query 表现”训练初始化；multi-task training 把任务结构弱化成一个大任务。

MAML用带标注任务，学快速适应初始化。
Self-supervised用无标注资料，学通用表示初始化。
Multi-task把多任务资料混合训练，常作 baseline。

**现在你应该能做什么：**说明“都是好初始化”不代表训练目标和资料结构相同。

# MAML 为什么好：快速学习还是特征复用 {#mamlwhy}
<!-- tag: RAPID VS REUSE -->

关于 MAML 为什么有效，有两种解释。Rapid learning 认为 MAML 找到的初始化特别适合 gradient descent，可以让每个任务快速走到好参数。Feature reuse 认为 MAML 学到的是一组已经很有用的特征，新任务只需要轻微调整最后部分就好。

课程提到一篇研究认为 feature reuse 才是 MAML 好的关键，并提出 ANIL，也就是 almost no inner loop。直觉是：如果大部分表现来自可复用特征，而不是内层快速学习，那么我们不必在所有参数上做复杂 inner loop。

Rapid Learning初始化让梯度下降特别快，少数几步就到好解。
Feature Reuse初始化本身已经学到通用特征，新任务主要复用它。

**现在你应该能做什么：**用两种假说解释 MAML 的成功，并知道 ANIL 更偏向特征复用解释。

# Learning to Optimize：优化器也可以学 {#optimizer}
<!-- tag: OPTIMIZER -->

优化器本来也是人设计的，例如 SGD、RMSProp、Adam。Meta Learning 可以把 update rule 当成要学习的东西。经典方向是 learning to learn by gradient descent by gradient descent：用很多任务训练一个 learned optimizer，让它根据梯度、历史状态等信息输出更新方式。

这种方法的挑战是泛化。一个 learned optimizer 在 MNIST 上、某个网络架构上训练得很好，不代表换激活函数、换网络深度、换数据集还稳。P56 提到早期实验中，训练和测试任务太像会显得有点作弊；真正有价值的是跨架构、跨任务仍然有效。

**错误示例：**“在同一个任务、同一个架构上训练和测试 learned optimizer，就证明它泛化很好”。错因是训练任务和测试任务几乎相同；修法是测试在不同架构、不同数据、不同任务上。

**现在你应该能做什么：**说明 learned optimizer 学的是 update rule，但必须检查跨任务泛化。

# Network Architecture Search：学网络架构 {#nas}
<!-- tag: NAS -->

如果 `φ` 是网络架构，我们就在做 NAS。架构可能包含层数、filter 大小、stride、通道数、连接方式。很多架构选择是离散的，无法直接对层数做普通微分，因此早期 NAS 常用强化学习或演化算法：agent 输出一个架构，训练这个架构，拿验证表现当 reward，再更新 agent。

也有方法把架构搜索改造成可微问题，例如 DARTS，试图用 gradient descent 直接优化架构参数。无论方法如何，放回 P55 框架看，NAS 是 meta learning 的一种特例：它学习的是产生好模型结构的策略或参数。

**Agent**输出架构超参数。
**Build**按架构建立网络。
**Train**训练该网络。
**Reward**用验证表现更新架构策略。

**现在你应该能做什么：**解释 NAS 为什么可以被看成“把网络架构当成 φ 的 meta learning”。

# Data Processing 也能 Meta {#data}
<!-- tag: DATA -->

训练前的数据处理也可以学习。Data augmentation 本来是人手试策略：翻转、裁切、颜色扰动、旋转等。Meta Learning 可以学习增强策略，让模型根据任务经验决定怎么变换数据更有利。

Sample weight 也可以学。某些策略认为难样本、靠近 decision boundary 的样本更该重视；另一些策略认为它们可能是 noisy label，应该降权。到底该升权还是降权，取决于数据分布和任务目标，因此也可以把 sample weighting strategy 当成 `φ` 来学。

### Data Augmentation
学习哪些变换能提升泛化。

## box warn
### Sample Weight
学习哪些样本更该影响训练。

**现在你应该能做什么：**解释为什么“资料怎么喂给模型”也是学习算法的一部分。

# Learning to Compare：训练和测试也能合在一个网络里 {#compare}
<!-- tag: COMPARE -->

前面的方法大多还是围绕 gradient descent 改造某个组件。更激进的方向是直接让一个 network 读 support set 和 query example，输出 query 的类别。模型内部如何“学习”不再显式写成梯度下降，也许它学会了比较、匹配或注意力机制。

这类方法常叫 learning to compare 或 metric-based approach。它把一次 episode 里的训练资料和测试资料都作为输入，让模型直接学会根据 support 判断 query 属于哪个 class。此时训练和测试的边界变得不那么显式，但 support/query 的概念仍然存在。

$$
model(Support Set, Query Example) → Query Label
它不一定显式输出一个 classifier，而是直接输出 query 的分类结果。
$$

**现在你应该能做什么：**说明 metric-based 方法为什么像是把 within-task training 和 testing 合在一个 episode network 中。

# N-way K-shot 与 Omniglot {#fewshot}
<!-- tag: FEW SHOT -->

Few-shot image classification 常用 N-way K-shot 设定。N-way 表示每个任务里有 N 个类别；K-shot 表示每个类别在 support set 中只有 K 个 example。例如 3-way 2-shot，就是每个任务有 3 个类，每类 2 张 support 图片。

Omniglot 是常见 benchmark。它有很多 character，每个 character 有少量书写样本。构造一个 N-way K-shot 任务时，从一组 character 中抽 N 个类别，每类抽 K 个样本作 support，再抽 query 样本让模型判断属于哪一类。训练任务和测试任务通常来自不重叠的 character 集合。

Support 样本数0
任务难度直觉中等


N-way5
K-shot1
N 越大类别越多，K 越小每类证据越少。

**现在你应该能做什么：**给定 N 和 K，说明一个 few-shot task 的 support set 怎么构造。

# 最小代码：按组件选择 Meta 方法 {#code}
<!-- tag: CODE -->

下面用伪代码展示“万物皆可 Meta”的共同抽象：先决定 `φ` 是什么，再用任务级 loss 优化它。

正确示例：通用 Meta 组件训练Python-like

```text
def meta_train_component(component_phi, tasks):
 meta_loss = 0.0
 for task in tasks:
 learner = build_learning_algorithm(component_phi)
 model = learner.adapt(task.support)
 meta_loss += loss(model, task.query)
 meta_loss.backward()
 update(component_phi)
```

Metric-based 方法则可以直接把 support 和 query 输入一个比较模型。

正确示例：Learning to ComparePython-like

```text
def classify_query(compare_model, support_set, query_x):
 scores = []
 for class_id, examples in support_set.items():
 score = compare_model(examples, query_x)
 scores.append((class_id, score))
 return argmax(scores)
```

错误示例：训练任务和测试任务完全相同Evaluation bug

```text
# 错误示例：在 CIFAR-10 上搜索架构，又直接用同一个任务报告最终能力。
architecture = search_on(cifar10_train, cifar10_valid)
score = report_on(cifar10_valid)

# 这不能证明架构搜索策略能泛化到新任务。
```

**现在你应该能做什么：**读懂 meta 方法的共同骨架：组件 φ、support 适应、query 评价。
