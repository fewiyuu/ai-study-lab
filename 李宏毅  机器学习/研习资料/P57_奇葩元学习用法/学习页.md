# 本节的概念地图 {#goals}
<!-- tag: MAP -->

这节课容易看起来像论文串讲。更稳的读法是先问一个固定问题：原技术的目标是什么，meta 版本又把目标改成了什么。普通训练常常优化“模型自己在训练任务上的表现”；meta 版本更关心“这个训练过程结束以后，另一个对象是否变好”。

# Self-supervised Learning 与 MAML：都在找初始化 {#ssl}
<!-- tag: SSL + MAML -->

BERT、BART 这类 self-supervised model 常被当成强初始化。它们先在大量无标注资料上做预设任务，比如遮住词再预测，得到一组对很多下游任务有用的参数。MAML 也在找初始化，但它找的是“少量标注资料更新几步后，在任务查询资料上表现好”的初始化。

两者目标听起来相似，都是得到 `θ0`，但不互斥。MAML 自己也要从某个初始点开始训练外层参数。如果完全从随机参数开始，计算量和数据需求都会更高。于是文献中常见做法是先用 BERT 或 BART 作为起点，再用 MAML、Reptile 这类方法把初始化往目标任务族靠近。

这也解释了两者互补的原因。Self-supervised learning 的优点是能吃大量无标注资料，覆盖面广；弱点是预训练目标和真实下游任务之间有距离。MAML 的优点是外层目标直接看目标任务族的表现；弱点是需要构造有标注的训练任务，而且计算成本高。把二者接起来，常在低资源任务上更有优势。

| 方法 | 资料来源 | 训练目标 | 典型优势 | 典型限制 |
| --- | --- | --- | --- | --- |
| BERT / BART | 大量无标注语料 | 预设的自监督任务 | 通用表示强，起点稳 | 和真实任务有距离 |
| MAML / Reptile | 多个带标注任务 | 少量适应后 query 表现好 | 目标贴近任务族 | 任务构造和计算成本高 |
| BERT + Reptile | 先无标注，再带标注任务 | 先广覆盖，再贴近任务族 | 低资源时常更稳 | 仍需任务级验证 |

**错误示例：**“BERT 和 MAML 都是找初始化，所以只能二选一。”错因是把目标相似误认为方法冲突；修法是把 BERT 看成 MAML 外层训练的起点，再让 meta 目标调整它。

**现在你应该能做什么：**说清 BERT 类预训练和 MAML 的共同点、差异，以及为什么能串接使用。

# BERT 加 Reptile：低资源任务为什么会受益 {#bridge}
<!-- tag: LOW RESOURCE -->

课程中提到的实验直觉很一致：在训练资料少的时候，self-supervised model 再接 meta 方法往往更有帮助。原因它利用少量任务结构告诉初始化“真正下游任务关心什么”。

可以把过程分成两层。第一层预训练让模型从大规模文本中学到语义、句法、常见模式。第二层 Reptile 或 MAML 用任务集合做调整，让参数更适合少量样本快速适应。低资源时，下游监督信号很少，纯 fine-tuning 容易不稳；带任务结构的 meta 调整能减少这种不稳定。

**无标注语料**训练 BERT / BART，得到强起点。
**训练任务**构造多个少样本任务。
**Meta 调整**用 MAML / Reptile 贴近任务族。
**新任务**少量标注资料快速适应。

要注意，组合并不自动保证有效。你仍要检查训练任务是否和测试任务属于同一任务族，support/query 划分是否合理，低资源设定是否真实。如果训练任务和测试任务太像，结果会高估；如果任务族差异太大，meta 调整也可能帮不上忙。

**现在你应该能做什么：**解释为什么“先大规模预训练，再用任务级目标微调初始化”常用于少样本 NLP。

# Knowledge Distillation：老师强，不代表会教 {#kd}
<!-- tag: DISTILLATION -->

知识蒸馏的普通做法是先训练一个大 teacher model，再让小 student model 去模仿 teacher 的输出。这个流程默认老师的分类能力越强，学生学得越好。但课程提醒我们：这件事不一定成立。一个在分类任务上正确率最高的老师，未必能给学生提供最容易吸收的软标签。

原因在目标不一致。普通 teacher 的训练目标是“我自己分类要准”；但教学目标应该是“学生跟我学完以后要准”。这两个目标不完全一样。一个老师可能自己非常确信，只输出尖锐分布，让学生只看到接近 one-hot 的答案；另一个老师虽然自己准确率略低，但输出的类别关系更平滑，反而让学生更容易学到结构。

把元学习放进蒸馏，就是改外层目标：teacher 更新参数时，不只看自己做得怎样，而是看 student 经过蒸馏以后，在验证资料上的结果怎样。这样 teacher 学的是“怎么教”，不是单纯“怎么自己答题”。

$$
teacher 参数更新目标：min Lvalid(student after distillation)
内层：student 模仿 teacher；外层：teacher 根据 student 的验证表现被更新。
$$

普通训练teacher 追求自己的分类正确率。
普通蒸馏student 追求接近 teacher 输出。
Learn to Teachteacher 追求学生学完后的表现。

**现在你应该能做什么：**说明为什么 teacher 的“会做题”和“会教学”是两个目标。

# 蒸馏温度也可以被学习 {#temperature}
<!-- tag: TEMPERATURE -->

实际做 learn to teach 时，不一定要更新整个 teacher model。一个更轻的做法是只学习输出温度 temperature。温度会改变 softmax 的尖锐程度：温度低，最大类别更接近 1；温度高，分布更平滑，非最大类别也会保留信息。

温度为什么重要？因为蒸馏不是只告诉学生“正确类是哪一个”，还希望它看到类别之间的相似关系。比如图片像猫也有一点像狐狸，老师的软输出能让学生学到这种结构。温度太低，软标签退化成硬标签；温度太高，分布又可能过于平均，信号变弱。用 meta 目标学习温度，就是让验证结果决定软到什么程度更适合学生。

# Domain Generalization：把源领域轮流假扮目标领域 {#dg}
<!-- tag: DOMAIN -->

Domain adaptation 和 domain generalization 都关心分布差异。区别在于：domain adaptation 通常能看到目标领域的一些资料，可能有标注，也可能无标注；domain generalization 更严格，训练时完全看不到真正目标领域。我们只有多个源领域，希望训练出的模型能泛化到未知目标领域。

问题来了：既然看不到目标领域，外层目标怎么写？一个常见思路是 leave-one-domain：从源领域中拿出一个假扮目标领域，其余源领域作为训练领域。学习算法先用训练领域学模型，再用被留出的领域评价它。每个源领域都可以轮流假扮目标领域，这就产生了多个 training task。

测试时，真正目标领域还是没见过。我们会把所有源领域资料用于训练，然后期待通过 leave-one-domain 学到的学习算法能迁移到新领域。这个想法的关键“反复练习从一些领域泛化到另一个未参与训练的领域”。

**错误示例：**“domain generalization 训练时先看一点目标领域资料，再报告目标领域表现。”这更像 adaptation，不是严格的 generalization；修法是训练阶段不使用真正目标领域。

# 元学习自己也可能需要领域适应 {#taskshift}
<!-- tag: TASK SHIFT -->

前面说元学习可以帮助 domain generalization，但反过来，元学习本身也会遇到更高一层的分布差异。普通机器学习里，训练资料和测试资料可能分布不同；元学习里，训练任务和测试任务也可能来自不同分布。

例如，训练时的任务都是英文、法文、德文文本分类，测试时却是日文文本分类；或者训练任务都是短文本，测试任务都是长文档。即使每个任务内部都有 support 和 query，任务集合本身也发生了漂移。这时你需要考虑的是 task-level domain adaptation，也就是让学习算法不仅适应样本分布，也适应任务分布。

| 层级 | 分布差异发生在哪里 | 会造成什么问题 |
| --- | --- | --- |
| 样本层 | 训练样本与测试样本不同分布 | 普通模型在目标领域表现变差 |
| 任务层 | meta training tasks 与 meta testing tasks 不同分布 | 学到的学习算法在新任务族上不稳 |

**现在你应该能做什么：**区分样本层 domain shift 和任务层 task shift。

# Lifelong Learning：学习怎样更新而不忘旧任务 {#ll}
<!-- tag: LIFELONG -->

终身学习关心的是连续任务。模型先学任务一，再学任务二，理想状态是新任务学会了，旧任务也保留。但普通训练常出现 catastrophic forgetting：参数被新任务更新后，旧任务表现大幅下降。

课程前面讲过 regularization-based 方法，例如 EWC、SI、selective synaptic plasticity。这类方法会给重要参数加限制，避免它们被新任务改得太多。原本这些限制通常由人设计：什么叫重要、惩罚多大、如何计算重要性，都需要人工方案。

把元学习放进来，就是学习一个更新策略或正则化策略。外层目标不再只是新任务测试集 loss 低，而是“学完新任务以后，新任务能做，旧任务也别忘”。因此元学习可以优化一个更贴近终身学习需求的学习算法。

$$
min Lnew(θ') + λ Lold(θ')
θ' 是学完新任务后的参数。外层目标同时看新任务表现和旧任务保留程度。
$$

普通新任务训练只关心任务二 loss 下降。
正则化方法限制重要参数变化，减轻遗忘。
Meta Lifelong学习怎样限制、怎样更新、怎样平衡新旧任务。

**现在你应该能做什么：**说明元学习在终身学习里可以学更新规则、重要性估计或正则化强度。

# 元学习也会灾难性遗忘 {#meta-forget}
<!-- tag: META FORGETTING -->

元学习可以帮助普通模型减少遗忘，但元学习本身也可能遗忘。想象你在训练学习算法时，先给它一组任务分布 A，再给它任务分布 B。如果连续更新时只看 B，学到的 learning algorithm 可能只适合 B，不再适合 A。

这和普通 catastrophic forgetting 是同构的，只是对象变了。普通遗忘发生在模型参数 `θ` 上；元层遗忘发生在学习算法参数 `φ` 上。解决方向也类似：保留旧任务样本或摘要、限制重要的 `φ` 改动、在外层目标中同时评价旧任务族和新任务族。

普通层模型学新类别后忘旧类别。被更新的是 `θ`。
元层学习算法适应新任务族后忘旧任务族。被更新的是 `φ`。

**现在你应该能做什么：**解释“元学习能处理遗忘”不等于“元学习自己不会遗忘”。

# 最小代码：四种场景的共同骨架 {#code}
<!-- tag: CODE -->

下面的伪代码不追求可运行，而是帮助你读懂外层目标怎么写。共同点是：内层先让某个对象学习，外层再根据学习后的表现更新 `φ`。

正确示例：用 meta 目标训练 teacherPython-like

```text
def meta_train_teacher(teacher_phi, tasks):
 meta_loss = 0.0
 for task in tasks:
 student = init_small_model()
 soft_targets = teacher_phi(task.train_x)
 student = distill(student, task.train_x, soft_targets)
 meta_loss += loss(student(task.valid_x), task.valid_y)
 update(teacher_phi, meta_loss)
```

在领域泛化里，关键是把一个源领域留出来当临时目标。

正确示例：leave-one-domain 任务构造Python-like

```text
def make_domain_tasks(source_domains):
 tasks = []
 for held_out in source_domains:
 train_domains = [d for d in source_domains if d != held_out]
 tasks.append((train_domains, held_out))
 return tasks
```

错误示例：把 teacher 自己准确率当作唯一目标Objective bug

```text
# 错误示例：老师只优化自己的分类准确率。
teacher_loss = cross_entropy(teacher(x_train), y_train)
update(teacher, teacher_loss)

# 这不能保证 student 蒸馏后表现最好。
```

**现在你应该能做什么：**从代码层面识别内层学习、外层评价和被更新的 `φ`。
