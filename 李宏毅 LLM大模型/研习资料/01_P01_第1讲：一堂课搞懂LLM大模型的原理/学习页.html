把“聊天机器人”拆回文字接龙

你在网页上看到的是“输入问题，模型回答”。模型内部看到的不是一个人类问题，而是一串 token。它每一步都在做同一件事：根据已经出现的 token，给 vocabulary 里的每个候选 token 打分，得到概率分布，再选出下一个 token。

这页会把 P1 的主线完整整理出来：token 是什么，为什么同一个问题每次回答略有差异，为什么模型需要 chat template 才像聊天机器人，幻觉为什么会出现，Hugging Face 实作中 tokenizer、model、generate、pipeline 各自负责什么。

prompt→ token id→ 概率分布→ 采样→ 追加到 prompt→ 结束符

来源：BV1JKoeBiEt4 · P1 第1讲：一堂课搞懂 LLM 大模型的原理

学完后要能做四件事

  1. 用“prompt → token → 概率分布 → 采样 → 结束符”解释一段回答是怎样生成的。
  2. 区分 token、token id、vocabulary、Hugging Face token 这几个容易撞名的东西。
  3. 解释 chat template、system prompt 和历史对话为什么会改变模型回答。
  4. 读懂最小 Hugging Face 流程：tokenizer 负责文字和编号互转，model 负责给下一个 token 打分，generate 负责循环生成。

概念骨架

MAP

第一层：它会什么

语言模型会做的是预测下一个 token。它不是直接返回一整段答案，也不是先查数据库再复制。一个完整回答，是很多次“预测下一个 token”的结果拼起来。

读到这里，应该能把模型输出拆成许多步小选择。

第二层：为什么有用

文字接龙听起来简单，但要接得像人话，需要语言知识；要回答事实问题，还需要世界知识；要听懂角色和格式，还要能读懂 prompt 里的上下文。

读到这里，应该能解释“文字接龙”为什么不等于“模型很笨”。

第三层：为什么会错

模型没有天然看到外部世界。当前日期、网页事实、工具结果、历史对话，如果没有放进 prompt 或工具链，模型只能按训练中学到的模式猜。猜得像，不等于真实。

读到这里，遇到幻觉时应先检查输入里有没有足够证据。

核心术语

TERMS

token 是模型拿来接龙的候选单位

P1 里对 token 的解释很朴素：模型做文字接龙时，每一步可以选择的输出，就叫 token。它可能是一个汉字、一个英文词、一个词的一部分、一个空格加词、一个 emoji，也可能是一串看起来很奇怪的符号。token 不是“中文一个字”或“英文一个单词”的同义词。

vocabulary 是一个模型所有可选 token 的集合。李宏毅课中用 Llama 做例子，指出常见模型的 vocabulary 可能有十几万个 token。为什么要这么大？因为模型要能输出中文、英文、标点、空白、emoji、代码符号，甚至一些很长的空格或特殊模式。

token id 是 token 在 vocabulary 里的编号。编号只是索引，不是语义。编号 0 可能对应惊叹号，编号 1000 可能对应某个英文片段；编号接近，不代表意思接近。

判断句：如果你想知道一句话到底有多少 token，不要按字数猜，要用目标模型自己的 tokenizer 算。

三个容易撞名的 token

名字在本节里的意思错用后果
生成式 AI 的 token模型生成或读取的基本片段。把 token 当字数，窗口预算和成本估算都会错。
token idtoken 在 vocabulary 里的整数编号。误以为编号大小有语义，读不懂 tokenizer 输出。
Hugging Face token登录或下载模型用的访问凭证。把凭证和模型 token 混在一起，实作步骤会乱。

课程中特意提醒：Hugging Face token 是“凭证”，不是模型接龙里的 token。两个词一样，语境完全不同。

最小代码：文字和 token id 互转python
text = "大家好"
ids = tokenizer.encode(text, add_special_tokens=False)
back = tokenizer.decode(ids)

print(ids)   # 例如：[109429, 53901]，实际编号由模型决定
print(back)  # 大家好

关键行是 add_special_tokens=False。有些 tokenizer 默认会加句子开头等特殊 token。它不是错，但你做“编码再解码是否还原”这种实验时，要知道多出来的东西从哪里来。

讲义主线：从概率接龙到聊天系统

LECTURE

概率分布与采样

模型真正输出的不是“下一个字就是某某”，而是对 vocabulary 里每个 token 给一个分数，再转成概率分布。比如 prompt 是“人工”，后面接“智”的概率可能很高，接“呼”也可能不低，接一个毫无关系的符号概率很低。

为什么同一个问题每次答案略有不同

如果模型总是选概率最高的 token，输出会很固定。真实聊天系统常常会按概率分布采样，也就是课里说的“掷骰子”。概率高的 token 更容易被选中，概率低的 token 也不是完全不可能。

这解释了两个现象。第一，同一个问题重复问，答案可能略有差异。第二，如果采样时允许非常低概率的 token 参与，某一步抽到奇怪 token，后续 prompt 就被污染，模型可能越接越怪。

工程上常见做法是限制候选范围，例如 top-k：只让概率最高的前 k 个 token 参与采样。k=1 时就是永远选最高概率；k 太大时，怪 token 更容易混进来。

试一试：top-k 会改变候选范围

top-k3

P(next token | prompt) = softmax(model(prompt))

这个公式只表达直觉：给定 prompt，模型输出一组分数，softmax 把分数变成概率。课程 P1 没展开 softmax 细节，你现在只要记住“模型给每个候选 token 一个概率”，后面讲模型内部时再看这些分数怎样算出来。

模型函数与参数

把语言模型看成函数 F

课程把语言模型写成一个函数：输入是 prompt，也就是 X；输出是下一个 token 的概率分布，也就是 F(X)。这个比喻很重要，因为它把“聊天机器人”从神秘界面拉回一个可分析的对象。

普通函数可能只有几个参数，比如 f(x)=ax+b 里的 a 和 b。大语言模型有大量参数。参数决定输入和输出之间的关系。所谓“大型”，主要指参数量大、训练资料多、可表达的函数非常复杂。

参数不是工程师一条条手写的规则。它们来自训练过程：模型读大量资料，逐步调整参数，让自己在给定前文时更会预测后文。P1 只给直觉，具体训练机制在后续课程展开。

“百亿参数”不等于“永远正确”

参数多让模型能表达复杂模式，但不保证每个事实都准确。课程用“水的沸点”说明世界知识的难点：普通前提下水的沸点是 100 摄氏度，但如果 prompt 说“0.5 大气压下”，正确答案就变了。

这个例子告诉你：模型不是只记“水的沸点=100”就够了。它还要读懂条件、例外和上下文。世界知识几乎无穷，条件稍微变化，正确接龙就可能变。

读到这里，应能解释“模型参数决定概率分布”，同时不把参数量当成可靠性的保证。

语言模型从哪里学会接龙

网络资料

大量网页、文章和代码都可以变成接龙教材。模型看到“人工智慧真神奇”，就能学习“人”后面常接“工”，“人工”后面可能接“智”。这是预训练最直观的来源。

限制：网络资料会有错误、偏见、过时信息，也不一定覆盖你的具体任务。

人类标注

人可以直接告诉模型某些 prompt 后面应该接什么，比如“台湾最高的山？”后面接“玉山”。这类资料让模型更像回答问题，而不只是延续文本。

限制：标注规模有限，格式和质量会影响模型习惯。

使用者回馈

点赞、倒赞或偏好比较可以告诉模型哪些回答更被接受。比如危险请求里，拒绝协助比直接教坏事更好。

限制：偏好训练会改变模型行为，但不能让模型自动拥有外部事实。

chat template：接龙为什么变成回答问题

如果 prompt 只是“台湾最高的山是哪座？”，纯接龙未必一定接答案。它也可能接“谁来告诉我呀”，或者把问题改成选择题。聊天平台会在用户输入前后加模板，让续写场景变成“用户问，助手答”。这就是 chat template。

模板改变的是模型真正看到的输入

你以为自己只输入了一句话，但模型可能看到的是：系统规则、当前日期、模型身份、历史对话、用户消息、助手消息开头。这一整段才是真正的 prompt。

课程里演示了自己手写“使用者说：你是谁？AI回答：”，模型会更像聊天机器人；但自造模板不一定会在合适位置输出结束符。官方模板通常更稳，因为模型训练时就见过这种格式。

Hugging Face 里可以用 tokenizer.apply_chat_template(messages) 套官方模板。它通常会顺手把文字转成 token id，所以后面可以直接给 model.generate。

试一试：模板如何包装用户问题

正确示例：messages 交给官方 chat templatepython
messages = [
    {"role": "system", "content": "你的名字是 Llama，请用中文回答。"},
    {"role": "user", "content": "你是谁？"}
]

input_ids = tokenizer.apply_chat_template(
    messages,
    add_generation_prompt=True,
    return_tensors="pt"
)
output_ids = model.generate(input_ids, max_new_tokens=80)
answer = tokenizer.decode(output_ids[0], skip_special_tokens=True)

role 告诉模板这句话是谁说的。system 是开发者或平台事先放进去的规则，user 是用户问题，assistant 可以表示模型已经说过的内容。乱塞 assistant 内容，会把模型带到一个它“以为自己已经说过某些话”的状态。

多轮对话、幻觉与 context engineering

多轮对话靠历史记录，不是神秘记忆

你问“台湾最高的山是哪座？”，模型答“玉山”。下一轮你问“第二高的呢？”，模型能回答“雪山”，不是因为它在脑子里单独存了一段记忆，而是聊天系统把前一轮对话也拼进了新的 prompt。

如果你开新聊天，历史不再放进 prompt，模型自然不知道你刚才问过什么。跨对话记忆是额外机制，不是 P1 里这个最小聊天流程自带的能力。

判断句：模型“记得什么”，先看那段信息有没有被放入当前输入。

幻觉来自“像真的接龙”

课程举了一个网站例子：模型能编出看起来像官方网站的网址，但点进去并不存在。这不是因为背后数据库坏了，而是模型按语言模式接出了一个像官网的字符串。

如果问题需要当前日期、实时网页、私有资料或精确出处，光靠接龙往往不够。你需要把必要资料放进 prompt，或让模型调用搜索、RAG、工具。即使用了搜索，也还要检查引用和结论是否对得上。

context engineering 指的就是把足够、相关、可用的信息放进模型输入，让模型有机会接出正确答案。

现象先问什么可能修法
模型编造不存在的网址prompt 里有真实来源吗？工具结果有被引用吗？使用搜索或 RAG，并要求回答引用证据。
模型不知道今天日期当前日期是否写进 system prompt？在系统提示或工具结果中注入日期。
模型忘了上一轮历史消息是否拼进当前 messages？保留必要历史，过长时摘要或筛选。
模型答非所问chat template 是否正确，角色是否错位？用官方模板，检查 role 和 assistant 预填内容。

万物皆 token：文字、图像、声音

P1 后半段把“文字接龙”推广到生成式 AI 的共同套路：复杂、有结构的对象，可以被拆成有限种基本单位。文字的基本单位可以是 token；图像可以被编码成图像 token；声音可以被编码成声音 token；蛋白质也可以看成氨基酸序列。

直接像素接龙太贵

1024×1024 图片有一百多万个像素。如果一个像素一个像素生成，比写一本长篇小说还慢。声音也一样，一分钟 22kHz 音频有一百多万个采样点。

先压缩成 token

图像 encoder 可以把局部图案压成图像 token，声音 encoder 可以把短时间声音片段压成声音 token。模型在 token 层面接龙，再用 decoder 还原成图片或声音。

生成式 AI 的定义

生成式 AI 是让机器根据输入 X，产生复杂而有结构的输出 Y。Y 的可能性无穷，但每一步选 token 的候选是有限的，所以问题可以变成一连串选择题。

读到这里，应能解释“token 可以是 word、image、speech、video”的意思，不把它误解成“代币”。

最小例子：Hugging Face 实作路线

EXAMPLE

课程用开源模型演示原理。这里的“开源”至少指参数可取得，能在本地或 Colab 里观察 tokenizer、model 和生成过程。李宏毅也提醒：有些人认为完全开源还应公开训练资料和训练流程，这一点要分清。

实作对象分工

对象负责什么常见检查
tokenizer保存 vocabulary，负责 encode/decode，套 chat template。检查 vocab size、特殊 token、同一句话的 token id。
model保存参数，输入 token id，输出下一个 token 的分数或概率。检查 logits、top-k 候选、模型尺寸。
generate帮你循环生成 token，直到结束符或长度上限。检查 max_new_tokens、do_sample、top_k。
pipeline把 encode、generate、decode 包起来，直接输入 messages 得到文字。适合快速调用，不适合观察内部细节。

从 Colab 到模型调用

  1. 复制 Colab 到自己的云端硬盘，选择合适 GPU。
  2. 登录 Hugging Face，并确认模型授权，例如 Llama 系列可能需要申请。
  3. 下载 tokenizer 和 model。3B 模型在 A100 上较顺，免费 T4 可能需要换 1B。
  4. 先观察 vocabulary 和 encode/decode，再做单步预测。
  5. 再用 model.generate 或 pipeline 做完整聊天。
最小流程：从 prompt 到文字回答python
prompt = "1+1="
input_ids = tokenizer.encode(prompt, return_tensors="pt")

output_ids = model.generate(
    input_ids,
    max_new_tokens=20,
    do_sample=True,
    top_k=3
)

text = tokenizer.decode(output_ids[0], skip_special_tokens=True)
print(text)

这段代码和课程里的流程一致：先 encode，再 generate，再 decode。注意 generate 的输出通常包含原始输入和新生成的 token。如果你只想显示模型新增的部分，要切掉输入长度对应的那段。

可视化：模型真正看到什么

VIZ

网页聊天框让人误以为模型只收到本轮问题。实际上，聊天系统通常会把系统规则、日期、历史消息、用户问题和助手开头拼成一段更长的 prompt。P1 里的多轮对话、system prompt、幻觉和 RAG，都可以从这张图往回解释。

系统层

系统提示会提前规定身份、语气、日期或安全边界。它影响模型怎样续写，但不是硬性保证；如果上下文冲突、模板错位或采样失控，仍可能出问题。

历史层

多轮对话靠历史消息被重新放进 prompt。模型能接上“第二高的呢”，不是因为它自动记住上一轮，而是当前输入里已经包含上一轮。

证据层

RAG 或搜索结果要进入 prompt 才能发挥作用。只调用工具但不把证据拼进去，模型仍然只能按已有模式继续接龙。

判断句：排查回答错误时，先看模型实际收到的 prompt，而不是只看用户在输入框里敲了什么。

误区诊断

DIAGNOSIS

七个常见坑

  1. 把 LLM 当数据库。错因是忽略生成机制；后果是相信编造网址；修法是查证来源或接 RAG。
  2. 把 token 当字数。错因是按肉眼数长度；后果是上下文预算错误；修法是用 tokenizer 统计。
  3. 认为采样就是随机乱说。错因是没看到概率分布；后果是无法调 top-k；修法是区分概率高低和候选范围。
  4. 自己乱写 chat template。错因是以为模板只是装饰；后果是模型不会在合适位置停止；修法是优先用官方模板。
  5. 多轮对话不传历史。错因是以为模型自动记得；后果是追问失效；修法是把必要历史放进 messages。
  6. 把 system prompt 当万能控制。错因是忽略模型能力和采样；后果是指令可能被冲掉；修法是结合模板、上下文和输出检查。
  7. 用 pipeline 后以为内部消失了。错因是封装遮住流程；后果是排查困难；修法是需要理解时回到 tokenizer/model/generate 三步。
