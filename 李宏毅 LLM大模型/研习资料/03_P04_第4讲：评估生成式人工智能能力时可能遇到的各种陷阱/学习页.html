评估 LLM：分数背后的坑，比排行榜更重要

模型使用者要选模型，模型开发者要挑训练版本，都离不开评估。但生成式 AI 的输出开放、任务复杂、prompt 敏感，随便试几个例子或只看排行榜，很容易把“看起来强”误当成“真的适合你的任务”。

这页把 P4 的主线整理成一张评估检查表：有标准答案时怎么打分；没有标准答案时怎么做人类或 LLM 评估；为什么指标会被钻漏洞；为什么 prompt、资料泄露、恶意攻击和偏见都必须纳入评估。

测试资料→模型输出→评分函数→聚合方式→决策

学完后要能做四件事

  1. 设计一个基本 benchmark，并说清 input、output、ground truth、metric。
  2. 判断 exact match、BLEU/ROUGE、BERTScore、人类评分、LLM judge 各自适合和不适合什么。
  3. 解释 Goodhart 定律、幻觉倒扣、prompt 敏感和资料泄露为什么会扭曲评估。
  4. 为真实系统写出内容、速度、成本、下限、安全和偏见的评估清单。

概念骨架

MAP

评估先问任务

你要论文摘要、医疗建议、长文检索、语音合成、代码修复还是抗攻击？任务不同，资料、指标和风险完全不同。你现在应该能做什么：不要用“哪个模型最强”替代“哪个模型适合这个任务”。

分数要看生成方式

生成式 AI 可以输出任何文字。选择题 exact match 看似简单，但可能变成“模型是否听懂只输出一个字母”的测试。你现在应该能做什么：检查评估是否量到了原本想量的能力。

部署还要看非内容指标

答案正确但很慢、很贵、偶尔暴走，未必适合真实系统。平均分高也可能掩盖灾难性下限。你现在应该能做什么：把速度、成本、稳定性和安全性写进评估方案。

术语表：先把“分数”拆开看

TERMS
术语直觉解释精确定义 / 用法常见误解
benchmark一场标准化考试。评估模型能力的一整套资料、输入流程、prompt、评分方式和聚合方式；口语中也常单指测试资料。只看排行榜，不看它到底考了什么。
ground truth标准答案。人类或规则给出的参考输出，可用于和模型输出对答案。以为所有生成任务都有唯一标准答案。
evaluation metric把答案变成分数的尺。评分函数，例如 exact match、BLEU、ROUGE、BERTScore、人类评分或 LLM judge 分数。以为 metric 高就等于真实好用。
aggregation把一堆小分合成总分。平均、加权平均、最差分位数、拒答率、攻击成功率等都可以是聚合方式。默认平均一定合理。
Goodhart 定律分数一旦成为目标，就容易被钻空子。当指标被拿来优化，它往往不再忠实代表原本想测的真实能力。把“冲榜”当成“能力提升”。
LLM as a judge让语言模型当评委。让模型读取输入、输出、标准或参考答案，再给分或解释。以为机器评委天然比人类客观。
verifier专门会批改的模型。专门训练来验证、评分或给反馈的模型；足够强时可作为 reward model 参与训练。以为它只是普通聊天模型换个提示词。
data leakage考试题进了复习资料。benchmark 的测试题或高度相似内容进入模型训练语料，导致分数虚高。模型背不出原题就一定没见过。
jailbreak诱导模型做本来不该做的危险事。绕过安全拒答，让模型输出危险、违法或有害内容。和 prompt injection 混为一谈。
prompt injection把任务角色骗偏。在用户输入、文档、网页或环境里塞入指令，使模型忽略原任务或执行恶意操作。只测试聊天框里的直接攻击，忽略网页/PDF里的间接攻击。
score(model) = Aggregate( metric(output_i, ground_truth_i) )这个式子只适用于有可比标准答案的场景。没有 ground truth 时，metric 可能换成人类评分、LLM judge 或成对偏好。

讲义精读：评估是一条证据链，不是一张排行榜

LECTURE

1. 为什么评估重要：使用者要选模型，开发者要选版本

课程一开始先把评估放回真实决策里：使用者面对很多模型，要知道哪个模型最适合论文摘要、医疗建议、代码修复、语音合成或特定领域问答；开发者训练模型时，会得到一批由不同方法、不同超参数产生的候选模型，也要决定哪一个能交给使用者。

所以评估不是“发论文时附一张表”，而是模型开发和模型使用之间的桥。随便试几个例子很容易被偶然样本骗到；只看排行榜也容易忘记榜单可能只测了某一小类能力。你现在应该能做什么：遇到“某模型最强”时，先问它在哪个任务、哪套资料、哪种评分规则下最强。

2. 有标准答案时：benchmark、ground truth 与 metric

以论文摘要为例，一个基础评估流程是：先收集一批文章，把每篇文章作为输入交给模型生成摘要；如果每篇文章都有人工摘要作为 ground truth，就可以用一个评分函数比较模型摘要和标准摘要，再把每笔分数聚合成模型总分。

这整套流程叫 benchmark。它有时指“评估过程”，有时指“测试资料”。如果要比较模型 A 和模型 B，至少要保证输入资料、标准答案、prompt、评分函数和聚合方式尽量一致。否则你比较的不是两个模型，而是两场不同考试。

这里最容易被忽略的是 metric。评分函数决定模型会被奖励什么，也决定模型会钻什么洞。一个 metric 只要成为大家优化的目标，就会影响模型开发方向。你现在应该能做什么：写评估方案时，把“我想测什么”和“这个分数实际奖励什么”分开写。

3. 自动指标：exact match、BLEU / ROUGE、BERTScore 各有边界

exact match 是最硬的对答案：模型输出和标准答案完全一样才得分。选择题、分类任务、固定格式答案可以考虑它，但生成式 AI 可以输出任何文字，不一定乖乖只输出一个字母。它可能输出“B，因为玉山最高”，也可能直接输出选项文字。

如果你用 prompt 强迫模型“只输出一个字母”，评估又会混入 instruction following 能力。台湾地理题本来想测地理知识，结果可能变成测模型能不能遵守格式。生成式模型的评估经常出现这种“想量 A，实际量到 B”的问题。

BLEU 和 ROUGE 更温和，它们大致看输出与标准答案的词汇重合，所以在翻译、摘要里长期常用。但词汇重合不等于语义正确：同义表达可能被误伤，字面相似也可能掩盖事实错误。BERTScore 进一步用语言模型的 contextualized embedding 比较语义相似，能处理一些同义改写，但它仍依赖所用模型、对齐方式和任务类型。

4. Goodhart 定律：指标被优化后，可能不再代表真实能力

课程用“换句话说”任务讲了一个很好的坑：早期 paraphrasing 评估主要看输出和参考改写的词汇或语义相似度。结果一个“鹦鹉模型”什么都不做，直接把输入原封不动当输出，也能拿到不错分数。它并没有换句话说，只是利用了指标漏洞。

后来加上“输入和输出必须有一定比例不同”的规则，模型又可以把前面一段随机替换成别的词，仍然刷出高分。这就是 Goodhart 定律在评估中的样子：当分数成为目标，模型或研究者会朝分数优化，最后得到一个榜单漂亮、真实任务却不好用的系统。

幻觉也和评估激励有关。普通 QA 里，模型说“我不知道”和模型胡乱猜错可能都得 0 分；硬猜还有小概率蒙对，于是评估制度没有奖励诚实拒答。更合理的设计可以是答对 +1、答错 -1、说不知道 0，让模型知道不会时承认不知道比乱编更好。

5. 没有标准答案时：人类评估有用，但不是神谕

写小说、写诗、语音合成、开放式对话，往往没有唯一 ground truth。同一句“回答我”可以有很多种好听的声音，和某个真人录音不相似，不代表合成质量差。这时常见做法是找人类评分或二选一比较。

人类评估的优势是贴近最终使用者，弱点是也会带偏见。Chatbot Arena 一类平台会让人类比较两个匿名模型输出，但人类可能偏好更长、更有分段、更有 Markdown 的答案。也就是说，人类可能在意“模型怎么说”胜过“模型说了什么”。

语音合成里的 MOS，也就是 mean opinion score，本质上是让许多人给 1-5 分再取平均。它很常用，但评分指示、标注者母语、平台、样本顺序、是否隐藏模型名称，都会让结果变化。你现在应该能做什么：看到“人工评估更好”时，追问人是谁、怎么评、是否盲测、是否可复现。

6. LLM as a judge 与 verifier：省人力，也会引入新偏见

当人类评估慢、贵、复现性差时，可以让语言模型当评委。LLM as a judge 会读取输入、模型输出、评分标准，必要时再读取参考答案，然后给分。研究中常见发现是：让模型先解释再给分，通常比只输出分数更接近人类评分。

模型真正输出的是 token 概率分布，所以 1-5 分评分也不一定只看采样出的那个分数。更细的做法是按概率加权平均，例如 1×p1 + 2×p2 + ... + 5×p5，保留模型对相邻分数的犹豫。

如果评分非常重要，可以训练专门评分的 verifier。它不只是普通聊天模型换个提示词，而是专门输入评分指令、rubric、参考答案和模型输出，再给出分数与反馈。一个强 verifier 还可以像 reward model 一样引导生成模型训练。这里的直觉是：把事情做好很难，但批评一件事相对容易。

不过 LLM 评委不是绝对客观。它可能偏袒自己的输出；知道某答案是 refine 过的，就凭空给更高分；看到假链接，也可能把答案误判得更可靠。所以实务上不要直接全量相信 LLM judge，应该先抽小样本人类对照，确认在这个任务上相关性足够高，再放大使用。

7. 系统评估：内容之外，还有速度、成本、下限和任务选择

模型输出内容正确，不代表系统就好用。第一个 token 很久才出现，用户会以为系统坏了；后续 token 速度慢，则是另一种体验问题。所以速度至少要拆成 first-token latency 和 token/s。成本也一样：大模型、深度思考、长 reasoning 可能只带来一点质量提升，却显著增加费用和等待。

平均分也不一定合适。课程用语音合成讲木桶理论：系统 A 99% 情况像真人一样完美，但 1% 会暴走，多念广告或自问自答；系统 B 没那么逼真，但从不暴走。地铁报站这类场景宁可选 B，因为下限比平均更重要。

评估还要回到“到底考什么能力”。通用模型会展示代码、工具使用、电脑操作、数学、医学、多语言、视觉、长文、生产力任务等多张榜单；专用模型则可能只需要领域资料。你现在应该能做什么：根据应用把“内容质量、速度、成本、下限、安全、领域覆盖”写成一张评估矩阵。

8. prompt、泄题、安全与偏见：评估环境本身也会改变结果

prompt 会强烈影响分数。大海捞针测试里，某模型在原 prompt 下表现差，但加一句“请找出最相关的句子”后结果突然变好；语音比较任务里，直接要求“比较两个人谁发音更准确”会触发拒答，换成“哪段音频更流利”正确率就上升。甚至大小写、空格、换行这种小改动，也可能让分数大幅波动。

data leakage 是另一个大坑。GSM8K 这类知名 benchmark 可能已经进入训练语料。研究者替换题目人名或数字后，模型正确率下降；给题目前半段，模型甚至可能续写出原题。这说明高分可能来自见过考题，而不是会泛化解题。

安全评估要区分 jailbreak 和 prompt injection。jailbreak 是诱导模型输出危险内容；prompt injection 是让模型在某个任务角色里忽略原指令，例如 AI reviewer 被论文里的白色小字诱导给高分，agent 在网页里读到隐藏指令后执行错误操作。偏见也要评估：只改性别、年龄、族群等属性，模型态度若明显变化，就说明存在不合理差异。

例题讲解：把“分数很高”拆成可判断的问题

EXAMPLE

例题 1：选择题 exact match

题面：标准答案是 B，模型输出“B，因为玉山最高”。如果按完全一致打分，它会被判错。

诊断：模型可能知道答案，但输出格式不符合评分函数。若强迫“只输出一个字母”，评估又会额外测到 instruction following。

修法：可以解析选项、规范化输出，或把格式遵守和知识正确分开计分。你现在应该能做什么：看到 exact match 低分时，先查是知识错还是格式错。

例题 2：鹦鹉模型刷 paraphrasing

题面：任务要求换句话说，但模型直接复制输入，或随机替换前一段词，也能在词汇相似指标上拿高分。

诊断：metric 奖励“和参考答案相似”，却没有明确惩罚“不像改写”。指标被当成目标后，就不再可靠代表真实能力。

修法：同时测语义保真、表达差异、流畅度和人工样本审查。你现在应该能做什么：给每个指标加一个“它可能被怎样作弊”的反例。

例题 3：AI reviewer 被隐藏指令影响

题面：论文 PDF 里藏白色小字，要求 AI reviewer 忽略缺点、给正面评价。人类不容易看见，但模型抽取文本时可能读到。

诊断：这是 indirect prompt injection。恶意指令不在聊天框里，而在模型处理的环境或文件里。

修法：隔离文档内容与系统指令，检测隐藏文本，要求模型引用证据并忽略文档内元指令。你现在应该能做什么：评估 agent 时，把环境里的隐藏指令列入攻击样本。

错误诊断：看到评估结果时要先查哪里

DIAGNOSIS
错误说法 / 场景错因后果修法
“排行榜第一，所以一定适合我的任务。”榜单任务、prompt、资料和聚合方式可能与你的应用不同。选到不适合领域、延迟、成本或下限要求的模型。用自己的任务样本做小规模验证，并报告失败样本。
“exact match 低，模型一定不知道答案。”生成式输出可能多解释、换单位、换格式。把格式错误误判成知识错误。先做规范化解析，再把格式遵守和答案正确分开看。
“BLEU / ROUGE 高，摘要一定好。”词汇重合无法保证事实正确、覆盖完整或表达自然。奖励照抄、漏掉关键事实或语义错误。结合语义指标、事实核验和人工抽样。
“模型不会时说不知道也得 0，所以无所谓。”答错和拒答同分会鼓励硬猜。幻觉增加，系统看起来更自信但更危险。答错倒扣，拒答给 0，单独统计拒答率和错答率。
“人类评估就是最终真相。”人类会受格式、长度、模型顺序、评分指示和背景影响。漂亮但内容一般的回答被高估。盲测、随机顺序、明确 rubric，报告一致性和置信区间。
“LLM judge 便宜，所以全量直接用。”模型评委可能自我偏袒、相信假链接、偏爱 refine 标签。自动评分系统性偏移。先做人类小样本对照，必要时多评委交叉验证。
“著名 benchmark 高分说明泛化强。”题目可能进入训练资料或被相似题污染。真实新题表现低于榜单。使用新题、私有题、改写题，检查前缀续写和语料重合。
“通过一组安全测试就安全。”攻击者会多轮尝试、换说服方式、把指令藏到环境里。部署后被 jailbreak 或 indirect prompt injection 击穿。报告攻击预算、attack success rate、失败类别和环境注入测试。
