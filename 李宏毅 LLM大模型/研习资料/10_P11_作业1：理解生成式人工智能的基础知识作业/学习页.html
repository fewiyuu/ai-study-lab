一、概念骨架：从入口到提交

SKELETON

作业最终提交在 NTU COOL 的线上测验里，不是交 Colab 文件，也不是提交 HuggingFace token。Colab 负责跑代码和拿观察结果；HuggingFace 负责提供模型与 tokenizer 的访问权限；Gradio 负责把模型调用包成一个能聊天的网页界面。

HuggingFace 账号→ read token→ 模型许可→ 复制 Colab→ 打开 GPU→ 运行标记处→ 记录依据→ NTU COOL 提交

第 1 部分：token

看 tokenizer 怎么切文字、怎么还原 token id、token length 为什么会变，以及模型怎样预测下一个 token。

第 2 部分：prompt

复制 system prompt 与 user prompt，观察高层规则、当前请求、标点和换行怎样改变输出。

第 3 部分：对话与界面

比较有无历史记录的多轮对话，并用 Gradio 体验参数、聊天记录和网页 demo。

判断句：如果一个步骤不能解释它给哪道题提供了证据，它很可能只是操作步骤，不是作业理解的核心。你现在应该能把 P11 拆成“权限、环境、观察、提交”四个环节。

二、术语表：先把名词对齐

TERMS

这节课的难点不在术语本身，而在术语容易混到一起。Access token 不是模型许可，Colab 不是交作业的地方，chat template 也不是普通的提示词。先把这些边界对齐，后面排错会快很多。

术语直觉解释准确边界与常见误解
NTU COOL课程线上测验入口。最终要在这里提交选择题。跑完 Colab 不等于已经提交。
HuggingFace模型和数据集平台。本作业用它下载指定模型和 tokenizer；不要把它当成本地运行环境。
access token让 Colab 代表你的账号访问 HuggingFace 的钥匙。作业只需要 read 权限。它不能公开分享，也不能写进公开笔记。
模型许可模型作者要求你同意的使用条款。token 正确但许可没通过，模型仍可能下载失败。
Colab runtime浏览器背后的云端执行环境。它会断线和重置；变量、安装包、登录状态可能消失。
GPU适合跑大模型推理的加速硬件。免费环境常见 T4 GPU。没开 GPU 时模型可能很慢或直接跑不动。
tokenizer把文字切成模型认识的 token id。token 数不等于字数。换 tokenizer，长度和 id 都会变。
logits模型对每个候选 token 给出的原始分数。它还不是概率；通常要经过 softmax 才能比较概率。
system prompt给模型设定整体行为规则。它不保证模型百分百服从，只是改变输入条件。
user prompt当前用户发出的具体请求。标点、换行、语言都可能影响输出，复制时要完整。
chat template把角色和消息包装成模型熟悉的对话格式。不是装饰文本。少了它，模型可能不按对话方式理解输入。
Gradio把模型调用快速做成网页 demo 的工具。这部分主要用来体验聊天界面和参数变化，不需要修改核心代码。

三、讲义：把作业流程讲成一条线

LECTURE

1. NTU COOL 只认线上测验提交

字幕一开始说明，作业在 NTU COOL 的 Homework 1 里完成。题目和选项会用中英文分隔，意思是你可以读中文，也可以读英文；分隔线本身不是题目内容。填完后必须点提交测验，关闭浏览器不会留下作答记录。

课程允许重新参加测验，并按最高分计算，这主要是为了减少误操作造成的损失。自学时不要把这个规则理解成“随便猜也行”。更稳的做法是每题旁边记下依据：来自哪个 cell、哪个输出、哪个 prompt 设置。你现在应该能判断：NTU COOL 是交付入口，Colab 是证据来源。

先抓住这条主线

账号权限保证模型能下载；Colab 保证代码能跑；token 与 prompt 观察给选择题提供证据；Gradio 让你看见多轮聊天界面；最后回到 NTU COOL 提交。

2. HuggingFace 要处理两件事：token 和许可

HuggingFace 在这里是模型仓库。你需要先登录账号，再创建一个 read 权限的 access token。Read 权限已经足够，因为作业只是读取模型，不需要上传或修改任何资源。创建 token 后复制到 Colab 的登录位置，让 notebook 能以你的身份下载课程指定模型。

另一个容易漏掉的是模型许可。字幕里提到课程会使用 Gemma 3 1B 一类指定模型，模型页面可能要求你填写基本资料并同意 license。Access token 和模型许可是两层门：前者证明“你是谁”，后者证明“你同意了这个模型的使用条件”。如果下载失败，不要只重建 token，也要检查许可是否已经通过。

3. Colab 是有状态的云端笔记本

Colab 的好处是你不用在本机安装 GPU 环境。打开课程 notebook 后，第一步通常是存一份云端副本到自己的 Google Drive，否则你编辑的是老师的只读版本。之后你在自己的副本里改代码、运行 cell、保存输出。

Colab 的执行顺序很重要。安装依赖、登录 HuggingFace、下载模型、定义 tokenizer 和 model，这些前置 cell 没跑，后面的代码就会缺变量或缺环境。若 90 分钟没有操作、网络断开，runtime 可能被回收。回来后看到变量消失，不是代码突然坏了，而是环境重置了。你现在应该能做的排查是：先看 GPU，再看登录，再看许可，最后看具体代码。

4. Token 题检查的是模型输入和输出的底层格式

Tokenizer 的 encode 把文本变成 token id，decode 把 token id 尽量转回文本。这里有两个边界要记住：token length 不是字符数；decode 也不承诺逐字还原你原来的空格、换行和分词方式。不同模型使用不同 tokenizer，同一句话可能切出不同 token。

Next token prediction 是生成式语言模型的核心动作。模型拿到前文后，会对词表中的每个 token 给一个 logits 分数；经过 softmax 后得到概率分布；再按贪心、采样或其他策略选出下一个 token。题目让你看输出时，通常是在训练你读懂“模型认为下一个最可能出现什么”。你现在应该能判断：如果换掉课程模型或 tokenizer，题目依据就变了。

5. Prompt 题不是比谁写得凶

System prompt 用来设定整体行为，例如角色、回答语言、限制范围；user prompt 是当前用户的问题或指令。两者会被 chat template 包装成模型训练时更熟悉的格式。对话模型不是直接看到你网页上的漂亮分栏，而是看到一串包含角色标记、消息内容和特殊 token 的输入。

字幕里反复提醒要完整复制 prompt，连问号、标点、空格都要保留。这不是吹毛求疵。对小模型来说，细小差异可能改变输出；对受限指令，例如“只能回答 I don't know”或“只能说英文”，模型也不一定稳定服从。你现在应该能说出：prompt 改变的是条件输入，不是硬规则开关。

6. 多轮对话的“记忆”来自上下文拼接

作业第三部分里，第 9 题和第 10 题可以直接根据题目和选项判断，第 11 题需要执行代码并比较两次输出。第一次按默认代码运行，第二次把指定行前面的井号去掉，让那行代码真正执行，再观察回答怎么变。

反注释不是魔法，它只是让原本被 Python 忽略的一行参与运行。多轮对话也不是模型临时改变了参数，而是程序把历史 user 和 assistant 消息一起放回 prompt。模型在当前输入里看见“我之前说过我是 Wilson”，才可能回答“你是 Wilson”。你现在应该能判断：一段聊天能不能记住前文，要看历史是否被放进输入。

7. Gradio 把 notebook 变成能操作的界面

Gradio 是一个快速搭建机器学习 demo 的工具。课程里的 Gradio 代码已经写好，你运行相关 cell 后会得到一个链接，打开后就能像聊天网页一样输入消息、调参数、看模型回复。它不是另一份作业，也不是 NTU COOL 答案来源的唯一形式。

它的教学价值在于把抽象参数放到眼前。最大输出长度会影响回答能写多长；temperature 和 top-p 会影响随机性；聊天历史会影响模型是否能接住上一轮。你现在应该能做什么：打开 demo 后，不只试着聊天，还能解释每个参数为什么会改变输出。

五、最小例子：看懂代码在读写什么

EXAMPLE

下面的例子不追求完整复现课程 notebook，只保留作业里最需要读懂的动作。读代码时先问三件事：这一行读取什么、写入什么、改变哪个状态。能回答这三点，排错就不容易乱。

正确示例：检查 GPU 是否可用Colab
!nvidia-smi

这条命令读取当前 runtime 的 GPU 状态。如果输出里能看到 T4 等 GPU，说明硬件加速已经挂上；如果命令失败或看不到 GPU，就回到执行阶段设置里检查硬件类型。

逐行拆解：从文字到下一个 tokentokenizer
text = "The capital of Taiwan is"
ids = tokenizer.encode(text)
decoded = tokenizer.decode(ids)

outputs = model(input_ids=ids)
last_logits = outputs.logits[-1]
prob = softmax(last_logits)
next_id = argmax(prob)
next_token = tokenizer.decode([next_id])

ids 是模型真正读到的输入，decoded 用来检查切分后的 token 能否还原成可读文本。last_logits 只取最后一个位置的预测分数，因为我们关心“前文之后的下一个 token”。如果把模型或 tokenizer 换掉，ids、logits 维度和最后答案都可能变。

正确示例：把对话历史放进输入chat template
messages = [
  {"role": "system", "content": "Answer briefly."},
  {"role": "user", "content": "Hi, I am Wilson."},
  {"role": "assistant", "content": "Hi Wilson."},
  {"role": "user", "content": "Who am I?"}
]
prompt = tokenizer.apply_chat_template(messages, tokenize=False)
ids = tokenizer.encode(prompt)

这段代码的关键不是 Python 列表，而是历史消息有没有进入 messages。如果只保留最后一句 Who am I?，模型没有前文依据，只能猜。

错误示例：权限和输入都不完整error
# 错误点 1：没有确认模型许可
login(token="hf_xxx")
model = AutoModelForCausalLM.from_pretrained("course/model")

# 错误点 2：只复制了 prompt 的一部分
system_prompt = "You can only answer"
user_prompt = "What is 2+2"

第一段可能因为 license 未同意而下载失败。第二段把题目给定 prompt 截断了，输出就不能作为选择题依据。修法是回到模型页面确认许可，再按题目完整复制 system prompt 和 user prompt。

七、错误诊断：看到现象先定位环节

DIAGNOSIS

这份作业的常见问题多半不是“模型不聪明”，而是权限、环境、复制或上下文出了问题。排错时先问错误发生在哪一层：账号层、运行层、输入层，还是提交层。

现象常见原因修法
模型下载失败access token 没登录、权限不是 read、模型许可没通过。重新登录 HuggingFace，确认 token 权限，回模型页面检查 license 状态。
Notebook 不能编辑还在老师提供的只读链接里操作。先储存云端副本，再在自己的副本里运行和修改。
变量不存在或前面成功后面失败runtime 重置，前置 cell 没按顺序执行。从安装依赖、登录、载入模型开始顺序重跑。
模型很慢或内存不足没有启用 GPU，或免费 GPU 被释放。检查执行阶段硬件类型，必要时重新连接 runtime。
Prompt 题输出和预期差很多复制漏字、漏标点、漏换行，或使用了错误模型。回题目逐字核对 prompt，确认模型名没有改。
多轮对话记不住上一句历史消息没有被加入 chat template。检查 messages 是否包含前几轮 user 和 assistant 内容。
Gradio 链接打不开相关 cell 没跑完、runtime 中断、链接已经失效。重跑 Gradio 区块，确认服务仍在运行，再打开新链接。
以为已经交作业只跑完 Colab，没有在 NTU COOL 点击提交。回到线上测验确认提交状态。
排错顺序：权限是否可用 → runtime 是否还活着 → 前置 cell 是否跑过 → 输入是否完整 → 输出是否对应题目。
这条顺序能帮你少做无意义修改，尤其是在 Colab 报错信息很长的时候。
