作业7：用 SFT 与 DPO 对齐语言模型

这次作业围绕一个很实际的问题：如果我们想让语言模型会聊天、会遵守 chat template，并进一步对齐某个人对“香菜”的偏好，应该怎么训练？视频把大模型训练拆成三个阶段：Phase 0 是预训练，作业里不从头做；Phase 1 是 SFT，也就是监督微调；Phase 2 是 RL / preference alignment，本作业用 DPO 做观察。

SFT 部分要把结果上传到 Judge Boy，占 6 分；RL 观察结果填到 NTU COOL 表单，占 4 分。学习重点不是把 Colab 每一格背下来，而是看懂：基础模型为什么还不能聊天、chat template 为什么重要、LoRA 怎样节省训练参数、DPO 为什么需要 accepted / rejected 回答，以及训练结果该怎样评估。

概念骨架

MAP

Phase 0：预训练只提供底座

预训练模型最基本的能力是文字接龙。它见过大量文本，但不一定知道“现在是 user 问、model 答”，也不一定会按课程要求输出完整回合。

作业不要求从零预训练，而是使用 Gemma-3-4B 系列作为基础模型，再进入后续微调。

Phase 1：SFT 学会格式和对话

SFT 用 question / answer 形式的 small talk 数据训练模型，让它读懂 chat template 的 special token，并输出像对话模型的回答。

可调的核心是 LoRA 设置、训练资料数量、epoch、batch size、learning rate 和生成参数。

Phase 2：DPO 学会偏好取舍

现实中很多问题没有唯一标准答案，例如“某种食物要不要加香菜”。DPO 给模型一个 preferred 回答和一个 rejected 回答，让模型靠近喜欢的、远离不喜欢的。

作业不是追求线上分数，而是观察训练前后偏好变化并填写表单。

术语直觉解释准确说法常见误解
Pretrain先让模型从大量文本学通用模式。常见目标是预测下一个 token，本作业把它当 Phase 0，不实际从头训练。以为预训练模型天然会聊天和遵守模板。
SFT用示范答案教模型怎样回答。Supervised Fine-Tuning，用成对输入输出继续训练基础模型。把 SFT 当成只是在模型里塞知识，忽略格式和输出边界。

PEFT / LoRA不动整颗大模型，只训练少量附加参数。LoRA 冻结原权重 W，训练两个低秩矩阵 A、B，并把增量加回输出。以为 LoRA 参数越大一定越好，忽略资料量和显存限制。
rank rLoRA 给模型开的“学习通道”宽度。A 和 B 的中间维度，r 越大，可训练参数和表达能力越高。rank 太小学不动，rank 太大可能过拟合或拖慢训练。
alphaLoRA 增量对原模型影响多强。控制 LoRA 分支的缩放，常见策略是设为 rank 或 rank 的两倍。alpha 太大可能破坏原模型能力，输出不稳定。
DPO直接根据偏好样本调模型。Direct Preference Optimization，用 preferred / rejected 回答优化模型偏好。以为 DPO 也需要唯一标准答案；其实它学的是相对偏好。
support ratio喜欢香菜食物占所有食物的比例。liked foods 数量除以全部 10 种食物数量，用来构造偏好设定。like / dislike 没覆盖全部食物或有交集，观察就不干净。
win rate你的回答赢过参考回答的比例。Judge Boy 用 GPT 成对比较你的输出和参考输出，交错左右顺序后估计胜率。只看一次样例聊天，就判断模型已经过 baseline。
decoding parameters控制生成时“保守还是发散”。temperature、top-k、top-p、max length 等影响采样和输出长度。训练不好时盲目调生成参数，掩盖真正的数据或训练问题。

讲义主线

LECTURE

1. 为什么这节从 fine-tuning 讲起

助教一开始用一句话概括这次作业：“fine-tuning is all you need”。这不是说预训练、prompt 或评估不重要，而是说当你想让一个现有语言模型学会或忘记某种行为时，最根本的工程动作往往是继续训练它。SFT 可以教它如何按示范回答，DPO 可以教它在没有标准答案的问题上偏向某种偏好。

这次作业的人设是“对齐李宏毅老师的香菜偏好”。香菜是一个很好的偏好例子，因为它不是事实题。地球到太阳的距离有标准答案，可以用 SFT 记住；但“珍珠奶茶可不可以加香菜”更像价值和偏好取舍，没有唯一正确答案，需要偏好对齐来改变模型倾向。

2. Phase 0 为什么不做预训练

预训练成本极高，涉及资料收集、清洗、算力、时间和人力。本作业不会从零训练一颗语言模型，而是使用已经预训练好的 Gemma-3-4B 系列模型作为基础。这个阶段的模型会做文字接龙，但不保证会按人类期待的聊天格式回答。

为了把模型放进 Colab 免费版显存，sample code 还会用到量化等技巧。量化的直觉是用更省空间的数值表示模型权重，降低显存占用。你不需要在本作业里深挖量化公式，但要知道：如果显存爆掉，常常不是“代码完全错了”，而是模型大小、batch、adapter 或 Colab session 状态共同造成的。

3. SFT 学的是 chat template

基础模型看到“请问珍珠奶茶可以加香菜吗”，可能会把“香菜”继续接成百科说明。SFT 给它大量 user / model 的示范，让它知道 user 回合结束后应该进入 model 回合，并用完整回答结束。

这就是 chat template 的意义：它不是装饰，而是训练和推理时的协议。少了 start turn、end turn 或角色标记，模型可能不知道哪段是用户输入、哪段是自己要生成的输出。

4. 训练资料不是越乱越好

SFT 使用助教整理过的 small talk 资料，原始资料很大，但作业只允许使用指定资料集。sample code 会做 preprocessing，最后最多约 4772 笔训练资料可用。

可调的是 subsample size。少量资料适合先 debug，确认流程能跑；更多资料能给模型更多教材，但训练时间更长，也需要重新考虑 epoch、batch size 和 steps。

5. LoRA 决定学习上限与成本

全参数微调一颗大模型太贵，所以本作业用 LoRA。LoRA 冻结原本的 weight，只训练两个低秩矩阵 A 和 B，再把 LoRA 分支的输出加回原模型输出。

rank 越大，adapter 表达能力越强，但训练参数、显存和时间都会增加。target modules 选得越多，可训练位置越多，上限可能更高，但也更容易超过资源预算。

6. steps 是训练量的共同尺度

data size、epoch、batch size 单看都不够，因为它们共同决定模型实际更新几次。字幕里给出的参考公式是：steps = data size * epoch / batch size。

助教经验提到，想冲 strong baseline，大概需要 150 到 200 steps 左右作为参考。它不是绝对公式，但能帮你避免只盯着 epoch 或只盯着 data size。

7. DPO 学的是偏好，不是标准答案

DPO 的训练样本由 question、accepted answer、rejected answer 组成。模型要增加 preferred 回答的相对概率，降低 rejected 回答的相对概率。

在本作业里，偏好来自 10 种食物与香菜的组合。support ratio 表示喜欢加香菜的食物数量占全部食物数量的比例。训练前后要观察模型回答是否朝这个比例和人设偏好移动。

8. 评估和交付分两条线

SFT 结果上传 Judge Boy，重点看输出是否合格、chat template 是否完整、win rate 是否过 simple / medium / strong baseline。Judge Boy 每人每天只有三次提交机会。

RL / DPO 部分不是上传一个分数文件，而是比较训练前后的输出，回答 NTU COOL 表单里的观察题。两条线的交付位置不同，混在一起会浪费时间。

完整知识链路

这节可以串成一句话：先用预训练模型作为底座，再用 SFT 和 chat template 把它训练成会按回合说话的对话模型；为了省显存和时间，用 LoRA 只训练少量 adapter 参数；SFT 训练完成后在固定 evaluation set 上 inference，导出 CSV 上传 Judge Boy；另一路用 DPO 的 preference data 调整香菜偏好，比较训练前后回答变化，并把观察写到 NTU COOL。你现在应该能做的事：打开 sample code 时，知道哪些格子只是环境准备，哪些格子是你该调的参数，哪些结果要保存和提交。

最小例子

EXAMPLE

下面的代码不是完整作业代码，而是把 P18 最关键的三个结构抽出来：chat template、LoRA 配置、DPO preference data。你要能读懂每行在控制哪件事。

示例 1：chat template 的回合协议python
def build_chat(user_message):
    return (
        "<bos>"
        "<start_of_turn>user\n"
        f"{user_message}"
        "<end_of_turn>\n"
        "<start_of_turn>model\n"
    )

prompt = build_chat("请问珍珠奶茶可以加香菜吗？")
outputs = model.generate(tokenizer(prompt, return_tensors="pt"))

逐行理解

chat template 的重点是角色和边界。user 回合必须结束，model 回合必须开始，模型才知道自己接下来应该生成回答，而不是继续补全用户的话。SFT 的一个基本目标，就是让模型学会这些 special token 的读写方式。

你现在应该能做的事：看到一个生成结果，判断它有没有正确结束 model 回合，例如是否出现 end of turn。

示例 2：LoRA 配置要看 rank、alpha 和 target modulespython
lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)

model = get_peft_model(base_model, lora_config)
model.print_trainable_parameters()
若原矩阵是 4096 × 4096，LoRA rank = 8，则可训练参数约为 4096 × 8 + 8 × 4096 = 65,536，只占原矩阵 16,777,216 参数的约 0.39%。

这里的直觉是：rank 决定 adapter 的容量，alpha 决定 adapter 输出影响力，target modules 决定哪些投影层会被加上 adapter。你现在应该能做的事：把“想提高上限”和“怕训练太慢”分别映射到不同参数选择。

误区诊断

DIAGNOSIS

常见坑

  1. 把预训练模型当成聊天模型。错因：忽略 pretrain 只学文字接龙；后果：模型可能延续百科式文本，不像回答用户；修法：用 SFT 和 chat template 教它回合格式。
  2. 破坏 chat template。错因：删掉 start/end turn 或角色 token；后果：Judge Boy 可能判定输出不完整，模型也可能不知道何时停止；修法：训练和 inference 使用同一套 template。
  3. 使用额外数据集。错因：以为资料越多越好；后果：违反作业限制，评分不可比；修法：只用指定 small talk 数据和 sample code 下载的 evaluation set。
  4. 只看 final loss。错因：loss 低只说明模型拟合给定训练资料；后果：可能 inference 仍然差，或者 win rate 不够；修法：用小样本聊天、evaluation inference、Judge Boy 反馈一起判断。
  5. LoRA rank 和 target modules 一口气开太大。错因：只追求上限；后果：训练慢、显存爆、Colab session 中断；修法：先用小 data size debug，再逐步增加 rank、modules 和训练资料。
  6. DPO 的 chosen / rejected 或 like / dislike 食物设反。错因：没检查 preference data；后果：模型朝错误偏好对齐；修法：训练前先打印样本，确认 support ratio 和 accepted / rejected 方向。
  7. max length 太短。错因：把生成中断误认为模型不会答；后果：回答半路结束，chat template 可能不完整；修法：适当增加 max length，并检查 end of turn。
  8. Colab 显存错误后反复重跑同一格。错因：session 残留对象没有清掉；后果：错误持续出现；修法：优先 restart runtime，重新跑 environment setup，再加载模型。
