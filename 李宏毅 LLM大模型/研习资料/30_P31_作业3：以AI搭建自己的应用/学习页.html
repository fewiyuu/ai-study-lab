用 API 把 LLM 变成自己的小应用

作业 3 的目标是从“网页版聊天”走到“程序调用模型”。网页版 ChatGPT 适合一个人手动复制输入、复制输出；如果你要做一个 emoji app、摘要器、角色扮演游戏或自定义文本工具，让很多用户直接使用，就需要 API：程序把 prompt 和用户输入送给模型，再把 response 送回你的应用界面。

这次作业用 Google Colab 跑代码，用 Gradio 做可视化界面，用 ChatGPT API 或 Gemini API 调模型，再完成三个任务：summarization、role playing、custom task。完成后要把结果复制到 DaVinci 批改助教，导出评分 JSON，最后上传到 NTU COOL。

讲义主线

LECTURE

1. 为什么需要 API：从手动聊天到多人应用

网页版 ChatGPT 的流程很简单：你在网页输入 prompt，网页把输入送给模型，模型把 response 显示回来。如果只是你一个人用，这很方便。

但如果你要做一个 emoji app，让朋友、同学或很多用户都能输入文字并得到加 emoji 的版本，你不可能替每个人手动复制到网页再复制回来。API 的价值就在这里：程序替你完成发送输入、接收输出、返回用户的流程。

现在你应该能做什么：解释为什么“做应用”需要 API，而不是只开网页。

2. API 调用里面有什么

一个 API 请求通常包含：使用哪个 model、给模型什么 prompt、用户输入的文章或句子、temperature 等参数。服务端收到请求后，返回 response。程序再把 response 放到你的应用界面里。

这和网页版的 send 按钮本质对应。网页版是人点按钮；API 是代码执行按钮动作。区别是 API 可以被你的程序反复、自动、多人使用。

现在你应该能做什么：把“网页输入框 / send / 回复”映射到“prompt / API call / response”。

3. Colab：不用本机安装的代码环境

Colab 是 Google 提供的网页代码平台。对新手的好处是不用配置环境，打开就能运行；还可以分享 notebook，之后的作业可能用到免费 GPU。P31 这次不需要 GPU，但需要用 Colab 执行调用 API 的模板代码。

使用时第一步是“复制到云端硬盘”，否则你改的内容不会保存到自己的文件。然后点击连接，确认环境启动，再按顺序运行 cell。

现在你应该能做什么：打开作业 notebook 后先复制、连接，再运行。

4. Cell、文件和断线

Colab 有文字 cell 和代码 cell。文字 cell 用来写说明，代码 cell 用来执行程序。左侧文件栏可以上传、查看、下载文件；如果误点到别的目录，可以点 content 回到默认位置。

Colab 闲置一段时间会断线。断线后，之前运行过的代码状态会消失，左侧临时文件也可能不在。因此重要结果要及时下载到本地。重新连接后，install package、import、API key 设置等前置 cell 都要重新跑。

现在你应该能做什么：知道断线后为什么后面代码会突然报错。

5. Gradio：把函数包成可操作界面

Gradio 可以把代码像积木一样搭成界面：标题、输入框、send 按钮、reset 按钮、聊天框都对应代码中的组件。用户在界面输入内容，按 send 后，Gradio 把内容传给你的函数，函数返回 output，界面显示结果。

但 Gradio 界面依赖背后的 Colab 代码运行。代码停止、Colab 断线、你关闭运行 cell，界面就不能继续用。每个 task 做完后要停止对应 Gradio，避免后续任务被卡住。

现在你应该能做什么：解释 Gradio 不是独立网站，而是代码运行时生成的界面。

6. API key 安全：不要分享，泄露就删

Gemini API key 可以从 Google AI Studio 创建；ChatGPT API key 可以从 OpenAI API keys 页面创建。OpenAI key 只在创建时完整显示一次，按 done 后就看不到完整值，忘记保存只能重新创建。

API key 就像密码。不要发给同学，不要放到公开链接，不要交作业时把 key 留在 notebook 里。如果怀疑泄露，去对应平台删除 key，重新创建。Gemini 可能遇到 permission bug，可尝试 existing project 流程；OpenAI 可能需要电话号码认证和可用额度。

现在你应该能做什么：把 key 当账户权限管理，而不是普通字符串。

7. Gemini API 与 ChatGPT API 的选择

作业允许两种 API，使用其中一种即可。字幕中助教说 Gemini API 当时免费，但相对较慢、较弱；ChatGPT API 效果较好，但可能收费。OpenAI 新账号可能有免费额度，但会过期，旧账号不一定还有。

这不是要你盲目选贵的，而是要你知道模型选择会影响任务表现、速度和费用。作业示范以 ChatGPT API 为主。

现在你应该能做什么：根据额度、效果、稳定性选择 API，并保留失败备用方案。

8. Part 1：summarization

第一个任务是摘要，满分 3 分。你需要设计 prompt，让模型拿到一篇 article 后能整理成有用摘要。最简单的 prompt 是 “Please summarize this article for me”，但可能只拿到普通分数。

更稳的 prompt 应该说明摘要语言、长度、结构、重点、是否保留关键事实、是否避免新增资料。完成后用 Gradio 输入 article、调 temperature、send，看结果满意就 export，生成 part one JSON。

现在你应该能做什么：写一个不只说“帮我摘要”的摘要 prompt。

9. Part 2：role playing

第二个任务是角色扮演，满分 3 分。你要填写角色和 prompt，让模型变成你指定的角色，例如海盗，然后和它互动至少两轮。这里需要自己维护 history：API 调用时要把 user 和 assistant 的历史对话传回模型，才能延续角色。

如果你只输入 hello、ok，角色表现会很薄，批改可能只给 2 分。更好的做法是设计清楚角色设定、说话风格、任务目标，然后用两轮有内容的对话测试它是否维持角色。

现在你应该能做什么：设计一个角色设定，并用两轮对话检查它是否破功。

10. Part 3：custom task

第三个任务是自定义任务，满分 4 分。你可以让模型做反义词、简单数学、文本转换等任何你设计的 task。字幕提醒：为了批改方便，不要让模型输出太长；可以单轮或多轮，但不要超过三轮。

太简单的任务可能拿不到满分。助教示范 “repeat after me” 得了 3 分，因为任务本身太简单。你应该选一个清楚、可验收、有一点点挑战但不冗长的任务。

现在你应该能做什么：设计一个不会太长、也不太敷衍的 custom task。

11. Colab JSON 不是最终提交文件

每个 part 在 Gradio 中按 export，会生成 part one / part two / part three JSON。这些文件保存的是你的互动记录，用来检查结果和复制到 grading system。

真正要提交到 NTU COOL 的不是这些 Colab JSON。正式提交文件必须从 DaVinci 批改助教 export 出来。这个坑助教反复强调，因为上传错文件会零分。

现在你应该能做什么：看到 JSON 时先问“这是 Colab 结果，还是 DaVinci 评分”。

12. DaVinci 批改流程

DaVinci 上安装 homework3 grading assistant，里面有三个 grader：summarization grader、role play grader、custom task grader。你要从 Colab 的 copy part to grading system 区块复制对应字段到 DaVinci。

Part 1 通常要贴 article 和 summarization；Part 2 要贴 role 和 dialogue；Part 3 要贴 description 和 dialogue。批改输出里要检查是否有 grade: 3 或类似分数字段。然后从 DaVinci export JSON，并按规定改名。

现在你应该能做什么：把三个 part 的复制字段和三个 grader 对应起来。

13. 额度与安全 prompt

DaVinci 批改也有额度。字幕粗略估计：summarization 一天约 5 次，role play 约 10 次，custom task 约 15 次；实际取决于输入输出长度。左下角额度可以查看剩余量。

同时，不要写冒犯、危险或高风险 prompt。ChatGPT 和 Gemini 的安全系统可能会挡掉内容，尤其 Gemini 可能因为不明原因挡掉一些看似普通的句子。作业建议当“超级老好人”，写安全、普通、积极的任务，避免被系统拦截。

现在你应该能做什么：在有限额度内先本地检查，再提交批改。

14. 提交规则与违规

NTU COOL 可重复提交，但只看最后一次。不能迟交，上传错误副档名、txt、图片、Colab JSON 或其他非 DaVinci JSON 都会零分。不要交代码，助教不会看。

不要抄袭，不要分享 prompt 或 information result / JSON file，不要上传非 DaVinci 结果，不要手动修改 JSON。后台能查，抓到算作弊。字幕中的历史 deadline 是 2024-03-21，成绩公布 2024-04-04；当前真实日期以课程公告为准。

现在你应该能做什么：提交前确认文件来源、文件名、字段和截止时间。

误区诊断

DIAGNOSIS

误区 1：API key 可以交给同学帮忙调

错因：低估 key 的权限。后果：额度被用、账号风险。修法：只自己保存，泄露就删除重建。

误区 2：Colab 断线后继续跑后半段

错因：忘记运行状态会丢失。后果：import、key、变量都不存在。修法：重连后从安装和 key 设置重新跑。

误区 3：export 后不下载

错因：以为临时文件永久在。后果：断线或覆盖后找不到结果。修法：满意结果立即下载到本地。

误区 4：上传 Colab JSON

错因：看到 JSON 就以为能交。后果：NTU COOL 上传无效，可能零分。修法：只上传 DaVinci export 的评分 JSON。

误区 5：手动修改评分 JSON

错因：把 JSON 当普通文本。后果：后台可查，算作弊。修法：原样上传，不编辑内容。

误区 6：role play 两轮太敷衍

错因：只追求满足轮数。后果：角色设定没有被检验，分数低。修法：设计有内容的两轮互动。

误区 7：custom task 越长越高级

错因：以为输出多就好。后果：批改困难、额度消耗高。修法：任务清楚、短输出、可验收。

误区 8：冒险 prompt 更有创意

错因：忽略模型安全过滤。后果：Gemini 或 ChatGPT 拒答、报错。修法：用安全、积极、无冒犯的任务。

误区 9：提交代码会加分

错因：把作业理解成代码审查。后果：上传不需要文件。修法：只交规定的 DaVinci JSON。
