LLM FINE-TUNING · TANG POETRY TASK

用微调教模型续写唐诗

作业五要做的是一个小型 LLM Fine-tuning 实验：给模型唐诗前两句，让它续写后面的诗句。原始模型可能会解释诗句、答非所问，或输出不像诗的文字；微调的目标是让模型看过唐诗资料集后，更稳定地完成“续写唐诗”这个任务。

这节课的价值不只在交作业。你会看到一个完整的微调闭环：准备 data set，选择基础模型，在 Colab 开 GPU，调整训练资料量，保存 checkpoint，用 temperature、top_k、top_p、max_length 控制生成，再把 15 首测试诗提交给评测系统检查 content 和 format。

概念骨架

MAP

任务定义

输入是唐诗前两句，输出是续完后面的诗句。模型要学的不是“解释唐诗”，而是按给定开头完成一首格式像唐诗、内容有连贯性的作品。

判断句：如果模型开始解释前两句的意思，它可能懂中文，但没有按作业任务行动。

训练闭环

先加载基础 LLM，再让它看很多“前两句，后续诗句”的样本。训练结束会得到 fine-tuned model，也会留下 checkpoint 供选择。

训练资料越多，通常越有机会学到格式和用词，但时间、GPU memory 和 Colab 稳定性成本也会上升。

生成与评分

训练后的模型还要通过 decoding 参数生成 15 首测试诗。评分看两个方向：content 关心内容丰富度和诗意，format 关心字数、格律、格式是否像任务要求。

判断句：训练分数低和生成参数不合适都可能让结果差，不能只盯一个环节。

1. 准备

复制 Colab、开启 GPU、安装套件、下载模型与数据。

2. 微调

设置训练资料量，开始 fine-tuning，观察 loss 与 checkpoint。

3. 生成

选择 checkpoint，调 decoding 参数，产生 15 首测试诗。

4. 提交

把输出交给平台助教评分，下载 JSON，改名后上传。

术语表

TERMS
术语直觉解释准确说法容易误解的地方
LLM Fine-tuning让现有大模型学会一个更具体的任务。以预训练语言模型为基础，在任务数据上继续优化参数或 adapter 参数。不是从零训练模型，也不是只改 prompt。
唐诗 dataset给模型看的参考书。由 instruction、input、output 组成，input 常是前两句，output 是应续写的部分。资料格式错了，模型会学到错误任务。
training data 数量模型读多少首唐诗。Colab 中可调的训练样本数，默认约 1040，资料总量约 5000。越多不一定立即越高分，还要付出时间和显存成本。
checkpoint训练过程中的存档点。fine-tuning 期间保存的中间模型状态，可用于比较不同训练步数的输出。最后一个 checkpoint 通常可先试，但不是理论上永远最好。
temperature控制生成随机程度。改变下一个 token 概率分布的尖锐或平坦程度，越高通常越多样。高 temperature 不是“更有创意”的保证，也可能破坏格式。
top_k只从概率最高的 K 个候选里选。截断候选 token 集合，再在前 K 个候选中采样。top_k 太小会单调，太大可能引入奇怪字词。
top_p只保留累计概率到 p 的候选集合。也叫 nucleus sampling，按概率从高到低累加，保留累计概率不超过阈值的一组候选。top_p 和 top_k 都是限制候选，但一个看数量，一个看累计概率。
max_length最多生成多长。限制生成 token 的最大长度，避免无限生成，也可能造成截断。设太短会断句，设太长会慢、占显存、甚至 OOM。
content score内容好不好。评测系统根据诗意、丰富度、用词等判断内容分数。内容分高不代表格式一定对。
format score格式像不像。评测系统检查字数、句式、格律或作业要求的输出格式。格式分高不代表诗句内容一定自然。

讲义主线

LECTURE

1. 这次 fine-tuning 到底在教什么

基础模型已经会说中文，也可能知道唐诗，但它不一定知道你给它前两句时要做“续写”。它可能解释诗意、聊天、补充背景，或接出格式不对的内容。作业五要把这个行为改过来：看到前两句，就生成后续诗句。

这个训练目标来自数据格式。每笔资料告诉模型：instruction 是任务说明，input 是前两句，output 是目标续写。模型反复看到这样的样本，就会把“前两句唐诗”这个输入和“继续写诗”这个输出形式连起来。

你现在应该能做的事：看到模型输出解释文字时，判断它不是“完全不会中文”，而是没有对准这次任务。

2. 为什么要在 Colab 里开 GPU

fine-tuning 会计算大量矩阵运算。CPU 也能算，但速度通常慢到不适合作业实验。GPU 擅长并行矩阵运算，所以 Colab 里要切换到 GPU runtime。

打开 GPU 不代表一定顺利。模型大小、batch、max_length、训练资料量都会影响显存。如果出现 out of memory，优先降低生成长度、训练规模或重新启动 runtime 清掉旧状态。

判断句：跑不动时先问“显存被谁占住”，不要只重复点运行。

3. 训练资料量的取舍

字幕提到资料集总共有约 5000 首，默认使用约 1040 首。直觉上，读更多唐诗能学到更多词汇和格式，但也会花更多训练时间。

学习上要把它看成一个可调资源：资料少，模型可能学不稳；资料多，实验成本上升。你应该记录每次尝试的 data_num、checkpoint 和生成参数，否则很难知道分数变化来自哪里。

4. checkpoint 不是摆设

训练过程中会留下 checkpoint。它像读书中途的测验存档，可以让你选择某个训练步数的模型来生成测试诗。

默认用最后模型通常合理，因为它看过更多训练资料。但如果后期过拟合、格式变怪或输出退化，也可以比较中间 checkpoint。判断句：checkpoint 是实验变量，不是只给系统看的临时文件。

5. decoding 参数控制输出风格

语言模型生成下一个字时，会得到一个概率分布。temperature、top_k、top_p 都在改“从哪些候选里选，以及选得多随机”。

唐诗任务里，太保守可能总是接出重复、平淡的句子；太随机可能破坏格式或语义。好结果通常来自训练质量和生成参数共同作用。

6. max_length 控制截断风险

max_length 太短，诗句可能在半句处被切断。太长，模型可能生成多余解释，运行也更慢，显存压力更高。

调这个参数时，先根据目标诗句长度估算需要的 token，再看实际输出有没有截断或拖尾。不要只为了“更完整”无限拉长。

7. 评分看 content 和 format

content 更像“诗的内容好不好”，会看用词、意境、丰富度。format 更像“是否符合唐诗格式”，会看字数、句式和作业格式。

这两个分数可能不一致。一首诗可能内容有趣但格式不合，也可能格式整齐但内容空。调参时要看哪个分数拖后腿。

8. 重跑实验要先重启 runtime

字幕特别提醒，直接重跑 fine-tuning block 可能出错，比如显存残留、输出怪异、环境状态混乱。正确做法是重新启动工作阶段，再按必要 block 顺序运行。

你现在应该能做的事：实验失败时，不乱点所有格子，而是按“重启，改参数，跑 import / seed / functions / hyperparameters / training”的顺序恢复。

完整知识链路

作业五把 P37 的 instruction fine-tuning 落成一个可操作实验。你先复制 Colab 并开启 GPU，加载台德 7B 或可替代的基础模型，再把唐诗资料整理成 instruction、input、output 的训练样本。fine-tuning 后得到一个更会续写唐诗的模型，并保存 checkpoint。生成阶段不是简单“按运行”，而是用 temperature、top_k、top_p 和 max_length 控制随机性、候选范围和输出长度。最后你生成 15 首测试诗，把结果交给评测系统，分别看 content 和 format 分数。结果不好时，要能判断问题来自训练资料、checkpoint、解码参数、格式提交还是运行环境。

课程线索一：任务不是泛泛写诗

助教反复强调输入是唐诗前两句，输出是后续诗句。这说明任务边界很窄：模型要补完整首诗，而不是解释、翻译或赏析。

你现在应该能做的事：验收模型输出时，第一眼先看它是否在“续写”，再看内容好不好。

课程线索二：默认参数只是起点

默认训练资料量约 1040，是助教为保存 checkpoint 和运行时间折中的设置。它不是唯一正确值。你可以改资料量，但要承担更长训练时间和 Colab 限制。

你现在应该能做的事：把每次实验记录成“资料量，checkpoint，temperature，top_p，top_k，max_length，得分”。

课程线索三：生成不是取最高概率就好

字幕用候选字概率说明，模型下一个字有多个可能。temperature、top_k、top_p 会改变候选分布，让输出更稳定或更多样。

你现在应该能做的事：看到诗句太死板或太乱时，想到先检查 decoding 参数，而不是立刻重训。

课程线索四：提交格式也是作业能力

最后要提交平台下载的 JSON，并按学号改名。不能改 JSON 内容，不能提交别人结果，也不能直接贴原唐诗答案。

你现在应该能做的事：区分“模型输出文字”和“最终提交档案”，避免跑出了结果却因格式零分。

最小例子

EXAMPLE

下面把作业五拆成三个最小对象：资料样本、训练参数、生成参数。读懂这三块，Colab 就不再只是按钮集合。

资料样本：前两句到后续诗句json
{
  "instruction": "以下是唐诗的前两句，请续写完成这首诗。",
  "input": "床前明月光，疑是地上霜。",
  "output": "举头望明月，低头思故乡。"
}

逐项理解

instruction 定义任务，告诉模型要续写。input 是条件，也就是测试时会给出的前两句。output 是训练目标，模型要学习生成的部分。

如果你想把同一套流程改成写专利或摘要，核心也是换掉 instruction、input、output 的内容，而不是只改标题。

错误样本会教错任务

如果 output 写成诗句解释，模型就会学会解释；如果 input / output 边界乱掉，模型会不知道该续写哪一段。

判断句：微调数据决定模型要模仿的行为。数据不是附属材料，它就是老师。

训练参数：先记录可重复实验条件python
model_name = "TAIDE-7B"
num_train_data = 1040
checkpoint_to_use = "last"

train(
    model=model_name,
    dataset=tang_poem_dataset[:num_train_data],
    save_checkpoints=True
)
生成参数：控制续写时的候选分布python
generation_config = {
    "temperature": 0.7,
    "top_p": 0.9,
    # "top_k": 50,
    "max_length": 96
}

poems = fine_tuned_model.generate(test_inputs, generation_config)
assert len(poems) == 15

读完例子，你应该能解释：训练参数决定模型学了什么，生成参数决定它在学到的分布里怎么抽样。

误区诊断

DIAGNOSIS

六个常见坑

  1. 只跑一次默认 Colab 就交。错因是把实验当流程打卡；后果是分数不稳；修法是记录参数并至少比较一两组输出。
  2. 忘记复制到自己的云端。错因是直接编辑只读 notebook；后果是修改不保存；修法是先储存副本再改参数。
  3. 没开 GPU 就开始训练。错因是忽略 runtime；后果是训练极慢或失败；修法是检查执行阶段的硬件加速器。
  4. 把 max_length 拉很大。错因是以为更长更完整；后果是慢、OOM 或输出拖尾；修法是按目标诗句长度估算。
  5. 重跑时不重启 runtime。错因是忽略显存和变量残留；后果是 OOM 或结果怪；修法是按重启流程重新跑必要 block。
  6. 提交了错误 JSON 或改了 JSON。错因是混淆输出文本和提交文件；后果可能零分；修法是用平台下载结果、按学号改名、不改内容。
