额外课程：GPT-4o

这节课讨论 GPT-4o voice mode 背后可能的语音技术。重点不是确认 OpenAI 实际用了哪一种架构，而是用已有语音技术推理：一个模型要像 demo 那样会换语气、会笑、能被打断、能边看边聊，系统内部至少要解决哪些问题。

你要学会把“语音互动很自然”拆开看：旧版语音界面通常是语音辨识、语言模型、语音合成串起来；GPT-4o 的展示更像端到端的多模态模型，直接把声音、图像和模型自己正在说的话放进同一套决策里。

学完后要能做四件事

  1. 说清旧版语音界面和 GPT-4o voice mode 猜想架构的差别。
  2. 解释为什么语音模型通常不直接在 16 kHz 原始波形上逐点接龙。
  3. 说明 speech unit、混合编码器、speaker diarization 各自解决什么问题。
  4. 分析语音互动中的 turn-taking：什么时候插话，什么时候等待，什么时候被打断。

概念骨架

MAP

先澄清 demo

字幕一开始就提醒：当时很多人以为手机 App 里已有的语音按钮就是 GPT-4o voice mode。其实旧语音界面也能“听你说话再回答”，但它通常只是把语音转文字、把文字交给 ChatGPT、再把回答合成为语音。

所以第一层判断是：你看到的是“语音入口”，还是“模型真的以声音为核心输入输出”。两者体验都像聊天，但技术边界不同。

再拆架构

旧版级联系统的优势是组件成熟：ASR 负责语音辨识，LLM 负责文本回答，TTS 负责语音合成。问题是声音里的情绪、喘气、笑声、打断时机，容易在转成文字时丢失。

端到端语音模型的直觉是少绕一圈：让同一个多模态模型直接看声音表示，预测接下来该发出什么声音或保持安静。

最后看互动

真正自然的语音互动不只是“延迟低”。模型还要判断人是否说完、是否想插话、是否只是合唱或背景声；如果有视觉输入，还要在需要时回看过去看过的画面。

这一节的核心链路是：表示声音、训练声音接龙、对齐成会聊天的角色、在多通道中决定下一步行动。

声音进入

原始波形先被编码成 speech unit、文字或混合符号。

模型接龙

语音语言模型预测下一个 unit、文字或安静符号。

声音输出

decoder 或 TTS 把符号还原成语音、笑声、语气变化。

多通道决策

模型同时 attention 到听觉、自己发声、视觉与历史上下文。

混合编码器能用文字表示的用文字，不能用文字表示的用声音单位。把 ASR 与声学 unit 结合，保留语言内容，也保留笑声、喘气、情绪等非文字信息。以为文字转写已经包含了语音里的全部信息。

同时听说看模型说话时还继续接收世界。把用户声音、模型自己已发出的声音、视觉输入放在不同通道里，让模型对所有通道做 attention。以为听到任何声音都应该立刻停下，其实合唱、插话、背景声需要区分。

讲义主线

LECTURE

1. demo 里真正让人惊讶的不是“能说话”

旧版语音界面早就可以让用户用声音输入，再听到模型用声音回答。它的工作方式并不神秘：语音辨识把声音变成文字，语言模型在文字上做回答，语音合成把文字念出来。只要这三个模块足够快，用户也会感到“还算即时”。

GPT-4o voice mode 展示中更特别的地方，是它好像能处理语音内容之外的信息。它能用多种语气说话，能轻声、加速，甚至像唱出来；它也可能听出喘气、紧张、笑声之类的声学线索，并产生非语音声音，比如笑声。

这就把问题从“会不会语音输入输出”推进到“模型有没有真正处理声音里的细节”。如果一切都先转成文字，很多声学信息会在第一步丢掉；如果系统直接处理声音表示，就有机会保留这些内容。

2. 级联方案能做到很多，但会留下缝隙

课程里先讨论了一个保守猜想：也许不是一个全新模型，而是在旧系统外面加模块。比如 ASR 之外增加语音事件侦测或情绪辨识，把“用户很紧张”“用户在笑”这样的标签附在文字后面交给语言模型。输出端也可以让语言模型多写一些控制符号，例如“括号笑”或“用轻声说”，再交给能理解指令的 TTS。

这个方案并非不可能，因为语音情绪识别、能读控制指令的语音合成、能产生笑声的 TTS 都已经有相关技术。但串接越多，系统越容易在延迟、同步和错误传播上出问题。一个模块判断错，后面的模块会基于错误标签继续生成。

GPT-4o 官方介绍强调 voice mode 是端到端模型，因此课程后半段转向另一个猜想：让一个多模态模型直接听声音、处理声音表示、再产生声音输出。

3. 语音接龙不能直接逐点生成

文字语言模型做的是文字接龙：给它未完成的句子，预测下一个 token。语音语言模型也可以类比成声音接龙：听一段声音后，预测接下来该是什么声音。

但原始声音太密。一秒 16 kHz 音频有 16000 个取样点，如果逐点预测，一秒声音就要跑一万六千步，非常慢，也很难学到高层结构。

因此语音模型通常先压缩：encoder 把波形变成离散 speech unit，模型在 unit 序列上接龙，decoder 再把 unit 还原成波形。

4. speech unit 保留文字之外的信息

把语音转成文字也是一种压缩，但这种压缩只保留“说了什么”。比如用户说“好好笑”然后笑了几声，ASR 可能只留下“好好笑”，后面的笑声就消失了。

speech unit 的价值在于它可以表示文字难以表达的声学内容：笑声、喘气、情绪、停顿、非人声或背景音。这样模型才有机会听见“内容之外”的线索。

它的代价是可能重造轮子。许多 unit 本来对应的就是可用文字表示的语音内容，如果全部都用新符号，原有语言模型学过的文字知识不容易直接复用。

5. 混合编码器是一种折中

混合编码器的直觉很朴素：能辨识成文字的部分就用文字，不能用文字表示的部分再用 speech unit。这样既保留语言内容，也保留笑声、语气和其他声学细节。

例如“好好笑”可以保留为文字，后面的笑声用特殊 unit 表示。混合 decoder 或 TTS 再把文字和特殊符号一起变成声音。

这种方案的好处是语言模型不用从零学习所有符号。它已经懂文字，只需要额外学习那些文字无法表达的声音单位如何影响理解和输出。

6. 说话者分段让模型知道“谁说了什么”

当多个用户一起和模型说话时，模型要知道不同声音来自不同人。speaker diarization 负责把一段音讯切成不同说话者的片段，并给出类似 A、B 的说话者标签。

这不是简单地认出某个人的真实身份，而是把对话结构整理出来。模型看到的输入可能变成“A: 这句话”“B: 那句话”，其中某些片段仍用 speech unit 表示。

如果没有这一步，模型可能把两个人的话混成一个上下文，回答时就会误判对象、情绪和任务。

7. 训练不能只靠语音资料

语音版语言模型可以用大量视频或音频做预训练，例如从影音平台取得声音，让模型学习 speech unit 接龙。这样能学到音色、语气、停顿、背景声等声音规律。

但只靠语音资料不足以学到足够世界知识。课程里用数量级说明：一百万小时语音大约对应几十亿文字 token，而大型文字模型训练常用到远多于此的文字 token。

所以合理猜想是：语音模型会利用已有文字语言模型的能力，或把语音符号和文字一起训练，让模型在“已经懂文字”的基础上学会听懂声音。

8. alignment 把“会接龙”变成“会聊天”

预训练只让模型学会声音和文字的模式，不保证它会当一个好助理。文字模型需要监督微调和人类反馈，语音模型同样需要语音对话数据来学习什么时候回答、怎样拒绝、用什么角色声音说话。

如果希望模型总用某个固定声音输出，还需要让它学会目标说话风格。可能的方法包括少量目标声音数据、让强预训练模型快速模仿，或用 voice conversion 把不同人的对话转换成目标音色。

因此 voice mode 的能力不是单一训练阶段产物，而是预训练、文字知识迁移、语音对话对齐和声音风格控制共同作用。

9. 最难的是同时听、说、看

文字聊天有明确边界：用户按 Enter，模型开始生成；用户按停止，模型停止。语音聊天没有这么清楚。用户停顿一下，模型要判断这是说完了，还是在想下一句；用户发出声音，模型要判断这是打断、合唱、背景声，还是仍在继续原句。

课程借 dialog speech language model 的思路说明：可以把“外界声音”和“模型自己已经说出的声音”放在不同频道。模型每一步同时看这些频道，决定输出真实声音、输出 wow 这样的反应，还是输出 silence 表示继续等。

如果加入视觉，问题更复杂。模型正在描述灯光时，画面里有人比了手势；它未必立刻评论，但当用户问“有没有看到奇怪的东西”时，模型要对听觉、视觉和自己生成的历史一起做 attention。你现在应该能做的事：把一个自然语音 demo 拆成“声音表示、训练来源、对齐资料、通道决策”四段来解释。

SOURCE

下面是从原字幕逐节阅读后熔炼出的线索。它们不是逐字稿，而是帮助你把视频内容转成可复述的学习骨架。

线索 1：先声明边界

视频明确说，当时没有 GPT-4o voice mode 的实际使用权限，也没有论文或技术报告。因此后续内容是基于已有语音技术的推测，不是对 OpenAI 内部实现的断言。

学习时要保留这个边界：这是技术推理课，不是官方架构复述。

线索 2：旧语音界面不是同一件事

手机 App 原本就有语音入口，但它没有展示里的多语气、可打断、自然插话等能力。课程用这个误解引出旧版级联系统和端到端语音模型的差别。

你应该能回答：为什么“能用语音聊天”不等于“拥有 GPT-4o voice mode”。

线索 3：声音要先压缩成可接龙的单位

一秒 16 kHz 音频有 16000 个取样点。直接逐点生成太慢，所以常见做法是用 encoder 和 codebook 把波形压缩成 speech unit，再让语言模型在 unit 上预测。

这条线索对应“表示问题”：模型到底在预测什么。

线索 4：文字与声音单位可以混合

文字保留语义内容，但容易丢掉笑声、喘气和语气。speech unit 保留声学细节，却可能重造文字轮子。混合编码器把两者结合，是课程提出的一种合理猜想。

这条线索对应“保真度与可学习性”的折中。

线索 5：训练要借用文字知识

海量音频能训练声音模式，但文字 token 数量级仍远少于大型文字模型常见预训练规模。因此语音模型很可能要复用已有语言模型，或把文字和声音符号一起训练。

这条线索对应“为什么不能只听一百万小时音频就够了”。

线索 6：自然互动需要通道决策

语音界面没有 Enter 键。模型要判断用户停顿是否表示说完，听到人声是否表示被打断，视觉里曾出现的物体是否需要在后续问题中被调用。

这条线索对应“同时听、说、看”的核心难点。

最小例子

EXAMPLE

这里用伪代码把三种思路放在一起。它不是可运行程序，而是帮你读懂“输入、表示、生成、还原、互动判断”这几件事。

正确示例：旧版级联语音界面pseudo
audio = microphone.listen()
text = asr.transcribe(audio)
answer_text = llm.generate(text)
answer_audio = tts.speak(answer_text)
speaker.play(answer_audio)

这段代码的含义

第一行接收用户声音，第二行把声音压缩成文字，第三行让语言模型在文字上回答，第四、五行把回答念出来。它的好处是模块清楚，调试容易；代价是 ASR 没留下的声学信息，后面通常补不回来。

如果用户喘气、笑、紧张，或者用很复杂的语调表达反讽，纯文字转写可能只留下字面内容。你现在应该能做的事：指出这条链路里哪一步最容易丢失情绪信息。

正确示例：端到端语音语言模型pseudo
audio = microphone.listen()
units = encoder.to_speech_units(audio)
next_units = speech_lm.generate(units)
answer_audio = decoder.to_waveform(next_units)
speaker.play(answer_audio)

这段代码的含义

这里模型不是先把所有内容变成文字，而是在压缩后的声音单位上接龙。encoder 负责把波形变成 unit，speech_lm 负责预测接下来该产生什么 unit，decoder 再把 unit 还原为声音。

如果 unit 设计得好，它可以承载笑声、语调和非语音声音。困难在于训练成本、资料规模、实时性，以及怎样让模型既懂声音也懂文字世界知识。

错误示例：把所有人声都当成打断pseudo
if microphone.detects_voice():
    model.stop_speaking()

错在哪里

这条规则过于粗糙。用户可能是在合唱、补充背景声、跟旁边人说话，也可能只是发出短促反应。自然语音互动需要区分“打断模型”“继续原句”“一起发声”“背景噪音”等情境。

更合理的做法是把外界声音、模型自己正在说的内容和历史上下文放进不同频道，让模型判断下一步是继续说、停下来、短暂回应，还是保持 silence。

误区诊断

DIAGNOSIS

常见坑

  1. 把旧版语音按钮当成 GPT-4o voice mode。修法：检查它是否只有 ASR、LLM、TTS 级联，是否能处理打断、情绪和非语音声音。
  2. 以为端到端模型等于不需要中间表示。修法：端到端仍需要可学习的声音表示，speech unit、文字或混合符号都可能出现。
  3. 以为 ASR 转写已经保留了所有信息。修法：问笑声、喘气、语气、背景音、停顿是否仍在表示里。
  4. 以为音频越多，知识越足。修法：区分声学能力和世界知识，理解为什么要复用文字模型或联合训练。
  5. 以为听到用户声音就应该停止输出。修法：区分打断、合唱、背景声、继续说话和短促反馈。
  6. 以为多模态就是当前画面描述。修法：模型可能要对过去看过的视觉片段和当前听到的问题一起做 attention。
