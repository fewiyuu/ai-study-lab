PyTorch / Data Pipeline

数据加载不是杂活，而是训练循环的入口

模型只认识张量。真实数据可能来自内存数组、CSV、图片文件或已经构造好的 Tensor。Dataset 负责把“第 idx 条样本是什么”讲清楚；DataLoader 负责把多条样本拼成 batch，并处理顺序、最后一批和读取并行等问题。

把这层学稳以后，你读训练代码时就能分清：数据从哪里来，batch 形状是什么，什么时候打乱，最后不满一批怎么处理，以及 DataLoader 和真正的参数更新之间有什么边界。

学完你应该能做什么

  2. 用 DataLoader 生成 batch，判断 batch_size、shuffle、drop_last 的影响。
  3. 计算一个 epoch 有多少个 iteration，以及最后一批大小。
  4. 把 DataLoader 接入最小训练循环，说明每行读写的对象。
  5. 诊断 shape 错、shuffle 用错、Windows 多进程卡住等真实问题。

Dataset 的判断句

如果你能回答“第 i 条样本应该返回什么”，就能写 Dataset。它不负责训练，也不负责一次取一批。

DataLoader 的判断句

如果你在问“一次取几条、是否打乱、最后一批怎么办”，这就是 DataLoader 的职责。

常见误区

DataLoader 只提供 batch。真正让参数变化的是训练循环里的 loss.backward() 和 optimizer.step()。

1. Dataset：定义怎么取一条样本

single sample

Dataset 是 PyTorch 对数据集的统一接口。它可以包装内存张量，也可以在 __getitem__ 里读取图片、解析文本或做预处理。接口只要求两件事：__len__ 返回样本数量，__getitem__(idx) 返回第 idx 条样本。

一个样本通常不是单个值，而是一组对象，例如 (x, y)。对于监督学习，x 是输入，y 是标签。DataLoader 后面会把多条 (x, y) 合并成 xb, yb。所以 Dataset 的返回结构会直接影响 batch 的结构。

正确示例：最小 Datasetdataset.py
import torch
from torch.utils.data import Dataset

class TinyPairDataset(Dataset):
    def __init__(self):
        self.x = torch.tensor([[1.0], [2.0], [3.0], [4.0]])
        self.y = torch.tensor([[2.0], [4.0], [6.0], [8.0]])

    def __len__(self):
        return len(self.x)

    def __getitem__(self, idx):
        return self.x[idx], self.y[idx]

逐行看语义

self.x 和 self.y 保存同样长度的数据。__len__ 让 DataLoader 知道一共有多少索引可以取。__getitem__ 用同一个 idx 同时取输入和标签，保证它们配对。

真实错误

如果 __len__ 返回错，DataLoader 会漏取或越界。若 __getitem__ 只返回 x 不返回 y，训练循环里 for xb, yb in loader 会解包失败。

你现在应该能做什么看到一个新数据源时，先写出“第 idx 条样本”的形状和内容，再决定 Dataset 怎么实现。

2. DataLoader：定义怎么取一批

mini batch

DataLoader 接收 Dataset，然后重复取若干条样本，合并成一个 batch。默认合并规则会把多个同形状 Tensor 沿第 0 维堆叠起来。比如单条 x 是 [1]，batch_size=2 时，xb 通常是 [2, 1]。

正确示例：读取 batch 并检查形状loader.py
from torch.utils.data import DataLoader

ds = TinyPairDataset()
loader = DataLoader(ds, batch_size=2, shuffle=False)

for xb, yb in loader:
    print(xb.shape, yb.shape)
    print(xb, yb)
参数它控制什么常见判断
batch_size每批多少条样本越大越吃显存，越小梯度噪声越大
shuffle每个 epoch 是否打乱索引顺序训练集通常 True，验证集通常 False
drop_last最后不满一批是否丢弃多数任务先 False；BatchNorm 等场景可能考虑 True
num_workers用几个子进程读取数据学习阶段和 Windows 上先从 0 开始

你现在应该能做什么：打印第一个 batch，确认它的 shape 和训练循环期望一致。

3. batch、iteration、epoch：别把训练次数算错

counting

batch 是一次送进模型的一小批数据；iteration 通常是处理一个 batch 的一次循环，也常对应一次参数更新；epoch 是完整看完一遍训练集。样本数、batch_size 和 drop_last 决定每个 epoch 有多少个 iteration。

场景drop_last=Falsedrop_last=True
10 条数据，batch_size=34 批：3、3、3、13 批：3、3、3
1000 条数据，batch_size=3331 批，最后 10 条30 批，丢掉 10 条

你现在应该能做什么：给定样本数和 batch_size，算出一个 epoch 中训练循环会跑几次。

4. 接入训练循环：DataLoader 提供 batch，优化器更新参数

training step

DataLoader 的输出通常写成 for xb, yb in loader。每次循环拿到一个 batch，模型用当前参数计算预测，loss 衡量预测和标签的差距，反向传播写梯度，优化器根据梯度更新参数。

逐行拆解：最小训练循环train.py
model = torch.nn.Linear(1, 1)
criterion = torch.nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)

for xb, yb in loader:
    pred = model(xb)          # 读取 batch 输入，得到预测
    loss = criterion(pred, yb)# 比较预测和标签
    optimizer.zero_grad()     # 清掉旧梯度
    loss.backward()           # 把新梯度写入参数 .grad
    optimizer.step()          # 按梯度更新参数

顺序不是装饰。若漏掉 zero_grad()，梯度会累积；若漏掉 backward()，优化器没有新梯度；若漏掉 step()，参数不会更新。DataLoader 只解决“数据怎么进入循环”，不替你训练。

你现在应该能做什么看到训练循环时，能把数据读取、预测、loss、清梯度、反向传播和参数更新逐行说清。

5. 真实错误场景

debugging

解包失败

for xb, yb in loader 报错，常见原因是 Dataset 只返回一个对象，或返回结构和循环变量不匹配。先单独打印 ds[0] 和 next(iter(loader))。

shape 不对

单条样本形状和 batch 后形状不符合模型输入。先打印 xb.shape，再看模型第一层需要什么形状。

验证集被打乱

验证集 shuffle=True 通常不影响平均指标，但会影响可复现、样本级分析和错误案例对齐。验证阶段一般保持 False。

Windows 多进程卡住

num_workers>0 可能需要主入口保护。学习阶段先设 0，确认数据逻辑正确后再优化读取速度。
