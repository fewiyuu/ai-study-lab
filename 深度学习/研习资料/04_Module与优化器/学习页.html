从 Module 到优化器：一次训练 step 为什么能更新参数

很多人第一次学 PyTorch，会把训练代码背成一个固定咒语：清梯度、前向、算 loss、反向、更新。这样能跑，但不够稳。一旦代码被拆成函数、模型变复杂、验证阶段混进来，问题就会出现：参数没注册、梯度累积、loss 不是标量、直接调了 forward、验证时忘了关梯度。

这一节把这些动作拆开讲。你不需要先懂自动微分的底层实现，但要知道每个对象的职责：nn.Module 管模型结构和参数，loss 把错误变成一个可优化的数字，backward() 把梯度写进参数的 .grad，optimizer 按这些梯度原地改参数。

学完你应该能做什么

  1. 自己写一个最小 nn.Module，并解释参数为什么会出现在 model.parameters() 里。
  2. 说清 model(x) 为什么比直接 model.forward(x) 更可靠。
  3. 逐行解释一个训练 step：zero_grad → forward → loss → backward → step。
  4. 判断常见训练代码错误的后果，包括漏清梯度、漏反向、漏更新、验证阶段忘记 eval() 或 no_grad()。
  5. 验收 AI 生成的 PyTorch 训练代码，而不是只看它能不能运行。

先看完整概念地图

mental model

PyTorch 的训练代码表面上是顺序执行，底层其实是几类对象在协作。模型不是一堆孤立函数，而是一个会登记子层和参数的对象；优化器不是重新寻找参数，而是拿着这些参数的引用；loss 也不是“报错多少”的普通数字，而是连接计算图的标量张量。

Module组织层与参数
model(x)触发 forward
loss把错误压成标量
backward写入 .grad
optimizer.step原地更新参数

直觉

训练像一次“试做题、看错多少、算每个参数该往哪改、真的改一下”的循环。难点不在顺序，而在每一步改的是哪个对象的哪个状态。

正式说法

一次训练 step 会构建前向计算图，loss 作为反向传播入口，通过自动微分得到参数梯度，优化器依据梯度和自身状态更新参数。

常见误区

把 optimizer 当成“拥有参数”的东西。更准确地说，它持有参数引用；参数本体仍在模型里，step() 会原地修改这些参数的值。

你现在应该能做什么看到训练代码时，先把对象分出来：模型、输入、目标、loss 函数、优化器、参数、梯度。不要急着背顺序。

nn.Module：模型为什么要继承它

registration

nn.Module 最容易被误解成“只是让代码好看一点的基类”。其实它解决的是工程上非常具体的问题：模型里可能有很多层、很多参数、很多状态。你需要一种统一方式，让 PyTorch 知道哪些张量是可训练参数，哪些子层属于这个模型，保存和加载时应该带走什么，训练和评估模式要通知到哪些层。

继承 nn.Module 后，你在 __init__ 里把子层挂到 self 上，例如 self.linear = nn.Linear(...)。这一步不是普通赋值那么简单。Module 会拦截这种赋值，把子模块登记到内部结构里。于是 model.parameters() 才能递归找到所有子层的权重和偏置。

如果你把层放在普通 Python list 里，或者在 forward 里临时创建层，参数就可能没有被稳定注册。代码也许不报错，但优化器拿不到那些参数，训练就像在推一扇没有把手的门：前向能过，参数却不跟着学。

最小示例：参数注册module.py
import torch
from torch import nn

class TinyRegressor(nn.Module):
    def __init__(self):
        super().__init__()
        self.backbone = nn.Sequential(
            nn.Linear(3, 8),
            nn.ReLU(),
            nn.Linear(8, 1),
        )

    def forward(self, x):
        return self.backbone(x)

model = TinyRegressor()
print(sum(p.numel() for p in model.parameters()))

逐行看关键点

super().__init__() 初始化了 Module 的内部登记系统。没有它，后面挂子层的行为就不完整。

self.backbone = nn.Sequential(...) 把一个子模块挂到模型上，里面的两个 Linear 会继续被递归发现。ReLU 没有可训练参数，但仍然是模型结构的一部分。

model.parameters() 返回的是参数迭代器，不是复制出来的新张量。优化器拿到的也是这些参数对象的引用。

常见错法

错法 1：在 forward 里写 layer = nn.Linear(...)。每次调用都会新建层，优化器也看不到稳定参数。

错法 2：把很多层放进普通 list：self.layers = [nn.Linear(...)]。需要用 nn.ModuleList 或 nn.Sequential。

错法 3：自己创建可训练张量却不用 nn.Parameter 包起来。普通 Tensor 不会自动进入 parameters()。

你现在应该能做什么检查一个模型定义时，能指出参数是否被注册：子层是否挂在 self 上，动态层是否放在 __init__，自定义权重是否用了 nn.Parameter。

forward 与 model(x)：不要直接绕过调用入口

call path

forward 是你定义“输入如何变成输出”的地方；model(x) 是你真正调用模型的方式。看起来二者只差一点语法，实际差别很重要。model(x) 会走 nn.Module.__call__，然后再进入你的 forward。这个入口会处理钩子、混合精度、编译/追踪相关机制，以及模块调用时的一些框架约定。

初学时你可以把它记成一句话：写模型时实现 forward，用模型时调用 model(x)。如果直接写 model.forward(x)，大多数简单例子也许能跑，但你绕开了框架入口。等项目里出现 hook、封装器、分布式、量化或调试工具时，这种写法就会埋雷。

forward 的职责

描述张量变换：层怎么接、激活函数怎么用、是否返回中间结果。它应该尽量保持清晰，不在里面创建新的可训练层。

model(x) 的职责

通过模块调用协议进入 forward。它是日常训练、验证、推理里应该使用的入口。

输出形状也要对

分类、回归、多标签任务的输出形状不同。很多 loss 报错不是 loss 本身难，而是模型输出和标签形状没对齐。

错误示例：直接调用 forward 与动态建层anti-pattern.py
class BadNet(nn.Module):
    def __init__(self):
        super().__init__()

    def forward(self, x):
        layer = nn.Linear(10, 1)   # 每次前向都会新建参数
        return layer(x)

pred = model.forward(x)           # 绕过 Module 调用入口
你现在应该能做什么看到 model.forward(x) 时能提出修改建议；看到 forward 里创建层时能说清后果：参数不稳定、优化器拿不到、训练不会按预期收敛。

loss：把“错得多不多”变成可优化目标

objective

模型输出通常是一批预测，标签是一批真实答案。训练不能直接靠“看起来接近”更新参数，必须把这批预测和标签压成一个数，这个数就是 loss。loss 越大，代表当前参数下预测和目标的差距越大；反向传播会根据这个数追踪每个参数对错误的影响。

训练时最常见的是标量 loss。所谓标量，不是说它必须是 Python float，而是形状上只有一个值的张量，例如 tensor(0.73, grad_fn=...)。这个张量还连着计算图，调用 loss.backward() 时，PyTorch 才知道从哪里开始往回算梯度。

训练目标：L(θ) = 1NΣi=1..N ℓ(fθ(xi), yi)
这里的 θ 是模型参数，fθ(x) 是模型预测，ℓ 是单个样本的损失函数。一次 mini-batch 里通常先算每个样本的损失，再取平均或求和。

回归

常用 nn.MSELoss 或 nn.L1Loss。输出通常是连续值，标签形状要和预测对齐。

单标签多分类

常用 nn.CrossEntropyLoss。模型输出 logits，标签是类别编号。不要先手动 softmax 再喂给它。

多标签分类

常用 nn.BCEWithLogitsLoss。每个类别都是一个二分类判断，标签通常是 0/1 向量。

例子：分类 loss 的正确输入loss.py
logits = model(x)                 # shape: [batch, num_classes]
target = torch.tensor([2, 0, 1])  # shape: [batch]，每个元素是类别编号

criterion = nn.CrossEntropyLoss()
loss = criterion(logits, target)  # 标量张量，可用于 backward
你现在应该能做什么给定任务类型时能初步选择 loss，并检查预测形状、标签形状、是否错误地提前 softmax。

一次训练 step：五个动作各管一件事

training step

训练 step 是这一节的中心。它不是一个黑盒，而是五个动作按顺序改变不同状态：清掉旧梯度，做一次前向，算出 loss，从 loss 反向写梯度，再让优化器根据梯度更新参数。顺序错了，训练就会变味；漏了一步，代码可能还跑，但参数不会按你以为的方式动。

zero_grad：清掉上一轮留下的梯度

PyTorch 默认会累积梯度，所以每个独立 mini-batch 更新前，通常先清空参数上的 .grad。否则这一轮梯度会和上一轮叠在一起。

判断提示：如果你看到训练循环里没有清梯度，就要追问是不是故意做梯度累积。
标准示例：一个训练 steptrain_step.py
model.train()
optimizer.zero_grad()

pred = model(x_batch)
loss = criterion(pred, y_batch)

loss.backward()
optimizer.step()

五行代码的状态变化

optimizer.zero_grad() 把参数上的旧 .grad 清掉。它不改参数值。

pred = model(x_batch) 计算预测，并建立本轮前向计算图。

loss.backward() 沿着计算图反向计算，把梯度累积到每个参数的 .grad 上。

optimizer.step() 读取 .grad，按优化算法更新参数值。它通常不负责重新计算梯度。

四个漏法

漏 zero_grad：梯度持续累积，等价于把多个 batch 的梯度混在一起。除非你有意做梯度累积，否则学习会不稳定。

漏 backward：.grad 没有新梯度，step() 没有可靠依据。

漏 step：梯度算出来了，但参数没有更新。

loss 不是标量：直接 backward() 可能报错，或需要显式传入梯度。

你现在应该能做什么拿到任何训练 step，能逐行说出它读取了什么、写入了什么、改了哪个状态。

优化器：拿着参数引用，按梯度改数值

optimizer

创建优化器时常写 optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)。这行代码的意思不是“把参数交给优化器保管”，而是让优化器拿到参数对象的引用。之后 loss.backward() 把梯度写在这些参数的 .grad 上，optimizer.step() 再读取梯度并更新参数值。

如果你先创建优化器，再替换模型里的某些层，优化器仍然拿着旧参数引用。这类问题很隐蔽：训练能跑，loss 也许会变化，但你以为在训练的新层可能没有被优化。所以模型结构确定后再创建优化器，是一个很实用的习惯。

SGD 更新：θt+1 = θt − η · ∇θL(θt)
θ 是参数，η 是学习率，∇θL 是当前 loss 对参数的梯度。SGD 的直觉很朴素：沿着让 loss 下降的方向迈一步。
维度SGD
直觉看当前梯度，按固定学习率走一步。
状态可以几乎没有额外状态；带 momentum 时会记录速度。
常见感受简单、可解释，但学习率和动量需要调得更细。
适合提醒不要把“慢”误认为“错”，先看学习率、数据尺度和 loss 是否合理。

SGD 与 Adam 的直觉差异

SGD 像是每个参数都按同一个基本步长走。Adam 会为每个参数维护梯度的一阶、二阶统计，常被理解为“自适应步长”。这让它在很多任务上更省心，但不代表永远更好。

这里先抓住验收重点：无论 SGD 还是 Adam，它们都需要参数引用、学习率、以及 .grad。没有梯度，优化器没有东西可读；参数没注册，优化器根本拿不到。

mini-batch、step、epoch：训练循环的位置关系

loop

训练不是只做一次 step。数据集通常被切成很多 mini-batch。每拿到一个 mini-batch，做一次前向、loss、反向和参数更新，这叫一个训练 step。模型完整看过一遍训练集，叫一个 epoch。多个 epoch 让模型反复接触数据，参数逐步靠近更好的区域。

这三个词经常混：batch size 是每个 mini-batch 有多少样本；step 是更新参数的次数；epoch 是扫完整个训练集的次数。假设训练集有 1000 个样本，batch size 是 100，那么一个 epoch 大约有 10 个 step。训练 5 个 epoch，大约更新 50 次参数。

渐进示例：迷你训练循环loop.py
for epoch in range(num_epochs):
    model.train()

    for x_batch, y_batch in train_loader:
        x_batch = x_batch.to(device)
        y_batch = y_batch.to(device)

        optimizer.zero_grad()
        pred = model(x_batch)
        loss = criterion(pred, y_batch)
        loss.backward()
        optimizer.step()

    validate(model, valid_loader)

循环里哪些位置不能乱放

model.train() 通常放在训练阶段开始处，而不是每个 batch 都必须重复写。它会告诉 Dropout、BatchNorm 等层进入训练行为。

optimizer.zero_grad() 通常在每个 step 的反向前调用。若要做梯度累积，可以不是每个 batch 都清，但那必须是明确设计。

validate(...) 应该切换到评估模式，并关闭梯度记录。验证不是训练，不应该更新参数。

完整链路

一个 epoch 包含多个 step；每个 step 使用一个 mini-batch；每个 step 里 loss 连接当前 batch 的计算图；backward() 写入本轮梯度；step() 修改模型参数；下一轮前向会用更新后的参数重新预测。

你现在应该能做什么给定数据集大小、batch size 和 epoch 数，能估算参数更新次数；读训练循环时能判断训练阶段和验证阶段有没有混在一起。

train、eval、no_grad：三个名字相似，管的事不同

modes

model.train() 和 model.eval() 控制的是某些层的行为模式。最典型的是 Dropout 和 BatchNorm：训练时 Dropout 会随机丢掉部分激活，评估时不丢；BatchNorm 训练时用当前 batch 统计，评估时用累计统计。它们不会自动决定是否计算梯度。

torch.no_grad() 控制的是自动微分是否记录计算图。验证和推理阶段通常不需要反向传播，关闭梯度可以节省显存和计算，也避免不小心把验证逻辑接进训练图里。它不会自动把模型切到 eval 模式，所以验证时常见组合是：model.eval() 加 with torch.no_grad():。

model.train()

进入训练模式。影响 Dropout、BatchNorm 等层。不会自动清梯度，也不会自动更新参数。

model.eval()

进入评估模式。用于验证和推理。不会关闭梯度记录，只是改变部分层的行为。

torch.no_grad()

关闭梯度记录。常用于验证和推理。不会改变 Dropout、BatchNorm 的模式。

正确示例：验证循环validation.py
model.eval()
total_loss = 0.0

with torch.no_grad():
    for x_batch, y_batch in valid_loader:
        pred = model(x_batch)
        loss = criterion(pred, y_batch)
        total_loss += loss.item()
错误示例：验证阶段还在训练模式validation_bad.py
# 忘记 model.eval()
with torch.no_grad():
    pred = model(x_valid)   # Dropout 仍可能随机丢激活，BatchNorm 仍可能用 batch 统计
你现在应该能做什么解释为什么验证阶段需要同时考虑模式和梯度记录，并能说清 eval() 与 no_grad() 不是一回事。

验收训练代码：不要只看能不能跑

audit

现在很多训练代码来自自动生成或复制改造。能运行只说明语法和部分形状过关，不说明训练逻辑正确。一个稳妥的验收方式，是沿着本节的对象链路逐项检查：参数是否注册，优化器是否拿到了正确参数，loss 是否匹配任务，训练阶段是否有完整 step，验证阶段是否没有更新参数。

模型子层挂在 self 上了吗？动态层有没有放错位置？
参数model.parameters() 能看到该训练的参数吗？是否有冻结参数？
loss任务、输出形状、标签格式是否匹配？loss 是否是标量？
step清梯度、前向、loss、反向、更新是否都在训练分支里？

集中错误诊断

  • 参数没变：检查是否漏 optimizer.step()、参数是否 requires_grad、优化器是否拿到正确参数。
  • loss 不下降：先检查数据标签、loss 选择、学习率、输出形状，再谈模型能力。
  • 显存越来越高：检查验证阶段是否没用 no_grad()，是否把带计算图的 loss/pred 存进列表。
  • 验证结果飘：检查是否忘了 model.eval()，尤其模型里有 Dropout 或 BatchNorm。
  • 训练很怪：检查是否漏清梯度，或无意做了梯度累积。

一个实用的验收顺序

  1. 打印模型和参数数量，确认该训练的层都被注册。
  2. 跑一个 batch，检查输出形状、标签形状和 loss 数值。
  3. 在 backward() 后检查关键参数的 .grad 是否存在且不是全 0。
  4. 在 step() 前后比较某个参数是否变化。
  5. 单独检查验证函数：eval()、no_grad()、没有 step()。
你现在应该能做什么把一段陌生训练代码拆成可检查项，而不是相信它“看起来像 PyTorch”。
