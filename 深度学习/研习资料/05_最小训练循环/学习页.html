从零写出一份最小训练循环

学完 Module、loss 和 optimizer 以后，下一步不是继续背概念，而是把它们接成一份完整训练脚本。完整不等于复杂：一份最小训练循环只需要数据、模型、loss、优化器、训练函数、验证函数和少量日志。少了其中任何一块，代码可能仍然能运行，但你很难判断它到底有没有在学习。

训练循环是一条状态流

mental model

训练循环不是一段孤立语法，而是一条状态流。数据从 DataLoader 出来，进入模型得到预测；loss 把预测和标签连成一个标量；反向传播把梯度写到参数上；优化器读取这些梯度并改参数；日志把这次更新的结果记录下来。下一次 batch 会使用已经改变过的参数。

你需要关心的不是“代码有没有跑完”，而是每一步有没有改变正确的对象。输入张量不应该被优化器修改，标签不应该参与求梯度，验证阶段不应该更新参数，日志里的 loss 不应该还带着计算图。只要能沿着这条线追状态，你就能看懂大多数 PyTorch 入门训练脚本。

DataLoader取出一批数据
model(x)得到预测
criterion计算标量 loss
backward写入梯度
step更新参数
log / metric记录结果

最小但完整

最小训练循环不需要复杂配置，但必须包含训练和验证的边界。只有训练阶段允许反向传播和参数更新，验证阶段只负责评估。

核心判断

一次训练 step 结束后，模型参数应该发生变化；一次验证 step 结束后，模型参数不应该变化。这个判断比背代码顺序更重要。

常见错觉

loss 打印出来不代表训练正确。loss 可能在错误标签、错误形状、未更新参数或验证泄露的情况下照样有数值。

先定骨架：脚本里应该有哪些部分

structure

一份最小训练脚本通常分成五块：准备数据、定义模型、定义 loss 和 optimizer、写训练函数、写验证函数。这样拆不是为了显得工程化，而是为了让每个职责清楚。数据加载出问题时看 Dataset 和 DataLoader；参数没更新时看训练函数；验证波动时看评估函数和模式切换。

初学时很容易把所有东西塞进一个巨大 for 循环里。这样确实能跑，但调试困难。比如你发现验证 loss 越来越高，可能是模型没学会，也可能是验证阶段忘了 model.eval()，还可能是把训练集和验证集混了。把训练和验证拆成函数，能让这些边界直接出现在代码结构里。

结构示例：训练脚本的最小分块outline.py
# 1. 准备数据：Dataset / DataLoader
train_loader = DataLoader(train_ds, batch_size=32, shuffle=True)
valid_loader = DataLoader(valid_ds, batch_size=64, shuffle=False)

# 2. 定义模型
model = TinyNet()

# 3. 定义优化目标和更新规则
criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

# 4. 训练一个 epoch
train_loss = train_one_epoch(model, train_loader, criterion, optimizer)

# 5. 验证一次
valid_loss, valid_acc = evaluate(model, valid_loader, criterion)

为什么要拆训练和验证

训练函数会调用 model.train()、loss.backward() 和 optimizer.step()。这些动作会记录梯度并更新参数。

验证函数会调用 model.eval() 和 torch.no_grad()，并且不应该出现 optimizer.step()。验证只回答“当前参数效果如何”，不负责改变参数。

判断句

如果一段代码里训练和验证共用同一个函数，就要检查它有没有参数控制模式、梯度和更新。否则最常见的后果是验证阶段也在建图，甚至误更新参数。

你现在应该能做什么看到一份训练脚本时，能把它分成数据、模型、目标、训练、验证、日志六个区域。分不出来时，先重构结构再谈训练效果。

数据进入模型前，先检查形状和类型

batch contract

DataLoader 每次给训练循环一批数据。对模型来说，这批数据不是“几条样本”这么抽象，而是具体的张量形状和 dtype。比如表格特征可能是 [batch, features]，图片可能是 [batch, channels, height, width]，分类标签通常是 [batch] 的整数类别编号。

很多训练问题其实不是优化器问题，而是数据契约不对。CrossEntropyLoss 期待模型输出 logits，形状通常是 [batch, num_classes]；标签是类别编号，形状是 [batch]，dtype 是 torch.long。如果你把标签做成 one-hot，或者提前对 logits 做 softmax，loss 可能报错，也可能训练效果很差。

# logits -> torch.float32, [batch, num_classes]

一个 batch：训练真正发生的最小单位

single step

训练循环最小的可观察单位是一个 batch。一个 batch 里，模型用当前参数做前向传播，loss 根据预测和标签计算错误，反向传播把梯度写进每个参数，优化器根据这些梯度修改参数。你要能说清每一行读取什么、写入什么、改变什么状态。

optimizer.zero_grad() 清掉旧梯度；model(x) 读取当前参数并产生预测；criterion(logits, y) 生成带计算图的标量 loss；loss.backward() 把梯度写入参数的 .grad；optimizer.step() 读取 .grad 并原地更新参数。少掉任何一个动作，训练含义都会变。

点击流程中的动作，观察它读取什么、写入什么、漏掉后会怎样。
正确示例：一个 batch 的训练 stepone_batch.py
model.train()

for x, y in train_loader:
    optimizer.zero_grad()
    logits = model(x)
    loss = criterion(logits, y)
    loss.backward()
    optimizer.step()
    break
错误示例：少了 backwardbad_step.py
for x, y in train_loader:
    optimizer.zero_grad()
    logits = model(x)
    loss = criterion(logits, y)
    optimizer.step()       # .grad 里没有新梯度，参数不会按当前 loss 更新

一个 epoch：把很多 batch 变成一次训练统计

train function

一个 epoch 表示训练集完整走过一遍。实际代码里，epoch 不是神秘概念，就是外层循环的一次。内层每处理一个 batch，参数通常更新一次；外层每完成一个 epoch，就把这一轮的平均 loss、准确率或其他指标记录下来。

写 train_one_epoch 时要注意两类数据：一类是用于训练的带图张量，比如 loss；另一类是用于日志的普通数字，比如 loss.item()。训练要用带计算图的 loss 调 backward()，日志要用断开计算图后的数字。不要把带图的 loss 一直塞进列表里，否则会保留计算图，显存可能越来越高。

epoch 平均 loss = 每个 batch 的 loss 按样本数加权求和本 epoch 样本总数

如果每个 batch 大小完全一样，直接平均每个 batch 的 loss 通常问题不大。但 DataLoader 最后一个 batch 可能不满。更稳的写法是用 loss.item() * batch_size 累加，再除以样本总数。这样最后一个小 batch 不会被错误地当成和大 batch 一样重。

渐进示例：训练一个 epochtrain_one_epoch.py
def train_one_epoch(model, loader, criterion, optimizer, device):
    model.train()
    total_loss = 0.0
    total_seen = 0

    for x, y in loader:
        x = x.to(device)
        y = y.to(device)

        optimizer.zero_grad()
        logits = model(x)
        loss = criterion(logits, y)
        loss.backward()
        optimizer.step()

        batch_size = x.size(0)
        total_loss += loss.item() * batch_size
        total_seen += batch_size

    return total_loss / total_seen

关键行解释

model.train() 放在 epoch 开始处，通知 Dropout、BatchNorm 等层进入训练行为。

x.to(device) 和 y.to(device) 保证数据和模型在同一个设备。模型在 GPU、数据在 CPU 是常见报错来源。

loss.item() 只用于日志。调用它之后得到的是 Python 数字，不能再拿这个数字反向传播。

判断句

如果训练函数返回 loss，最好返回普通数字，而不是还连着计算图的 Tensor。训练用图，日志断图，这个边界要清楚。

你现在应该能做什么写出一个训练 epoch 函数，并解释为什么平均 loss 要按样本数统计，为什么日志里用 loss.item()。

验证函数：评估当前参数，不参与训练

evaluation

验证阶段的目标是回答一个问题：当前模型在没参与训练的数据上表现如何。它不应该改变参数，也不应该保留反向传播需要的计算图。最常见的验证写法是 model.eval() 加 with torch.no_grad():。这两者职责不同：eval() 改变部分层的行为，no_grad() 关闭梯度记录。

验证函数里通常会计算 loss 和 metric。loss 用来观察优化目标是否合理，metric 用来观察业务任务表现。分类任务常见 metric 是 accuracy。计算 accuracy 时要从 logits 里取最大值所在类别：pred = logits.argmax(dim=1)。不要把 logits 直接和标签比较。

正确示例：验证循环evaluate.py
def evaluate(model, loader, criterion, device):
    model.eval()
    total_loss = 0.0
    total_correct = 0
    total_seen = 0

    with torch.no_grad():
        for x, y in loader:
            x = x.to(device)
            y = y.to(device)

            logits = model(x)
            loss = criterion(logits, y)
            pred = logits.argmax(dim=1)

            batch_size = x.size(0)
            total_loss += loss.item() * batch_size
            total_correct += (pred == y).sum().item()
            total_seen += batch_size

    return total_loss / total_seen, total_correct / total_seen
写法管什么不管什么漏掉后果
model.eval()Dropout、BatchNorm 等层的行为是否记录计算图验证指标波动，BatchNorm 统计被错误使用
torch.no_grad()关闭梯度记录，减少显存和计算模型模式验证显存上涨，保存 pred/loss 时可能保留图
不写 optimizer.step()保证验证不更新参数不影响前向计算验证集信息泄露到参数，指标不可信
错误示例：验证阶段还在建图bad_eval.py
def bad_evaluate(model, loader, criterion):
    model.train()          # 验证时不该保持训练模式
    losses = []
    for x, y in loader:
        logits = model(x)
        loss = criterion(logits, y)
        losses.append(loss) # 保存带图 Tensor，显存可能越来越高
    return losses
你现在应该能做什么看到验证代码时，先查三件事：有没有 eval()，有没有 no_grad()，有没有任何会更新参数的代码。

日志与指标：训练是否真的在动，要看证据

monitoring

训练循环跑完不代表模型学到了东西。你至少要记录训练 loss、验证 loss、验证 metric。训练 loss 下降说明模型在训练集上优化目标变小；验证 loss 和验证 metric 反映当前模型在未训练数据上的表现。二者一起看，才能判断是学不会、学过头，还是日志本身算错了。

日志不是越多越好。初学阶段先记录少量稳定指标：每个 epoch 的 train loss、valid loss、valid accuracy。如果 loss 变成 nan，先查学习率、输入数值、loss 写法和梯度爆炸。如果 train loss 下降但 valid loss 上升，可能是过拟合，也可能是训练/验证数据分布不同。先用指标定位方向，再去查代码。

常见训练问题：先定位是哪条状态流断了

diagnosis

训练循环出问题时，不要只说“模型不收敛”。先问：数据是否正确进入模型？loss 是否正确连接预测和标签？梯度是否写入参数？参数是否真的改变？验证是否没有参与训练？日志是否没有保留计算图？这些问题按状态流排列，能帮你快速缩小范围。

很多错误不会立刻报错。比如忘记 zero_grad()，代码能跑，loss 也会变化，但梯度会跨 batch 累积；忘记 model.eval()，验证也能算，但 Dropout 仍然随机；在验证里保存带图 loss，前几个 epoch 可能没事，后面显存才慢慢涨。训练代码的难点正是这些“能跑但语义错”的情况。

参数没更新

检查 loss.backward() 是否在 step() 前，优化器是否拿到正确参数，参数 .grad 是否为 None。

loss 不合理

检查任务和 loss 是否匹配，分类任务标签是不是类别编号，是否对 logits 手动 softmax 或 sigmoid。

验证不可信

检查验证集是否独立，验证函数是否用 eval() 和 no_grad()，是否出现 optimizer.step()。

完整最小模板：小型分类训练脚本

template

下面这份代码故意保持小。它不用真实图片数据，而是用随机生成的二维特征做三分类。这样做的目的不是追求模型效果，而是把训练循环各个部分放在同一个脚本里。你可以先跑通这份模板，再把 Dataset、模型和指标替换成自己的任务。

模板里有几个稳定边界：训练函数负责更新参数，验证函数不更新参数；loss 用带图张量做反向传播，日志用 item() 后的普通数字；accuracy 先从 logits 取类别，再和标签比较；设备由 device 统一传入，不在代码中到处硬写。

完整模板：最小分类训练循环minimal_train.py
import torch
from torch import nn
from torch.utils.data import TensorDataset, DataLoader

torch.manual_seed(0)
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

# 1. 数据：二维特征，三分类标签
X = torch.randn(600, 2)
y = ((X[:, 0] > 0).long() + (X[:, 1] > 0).long()).clamp(max=2)

train_ds = TensorDataset(X[:500], y[:500])
valid_ds = TensorDataset(X[500:], y[500:])
train_loader = DataLoader(train_ds, batch_size=32, shuffle=True)
valid_loader = DataLoader(valid_ds, batch_size=64, shuffle=False)

# 2. 模型
model = nn.Sequential(
    nn.Linear(2, 16),
    nn.ReLU(),
    nn.Linear(16, 3),
).to(device)

criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-2)

def train_one_epoch(model, loader, criterion, optimizer, device):
    model.train()
    total_loss, total_seen = 0.0, 0
    for x, y in loader:
        x, y = x.to(device), y.to(device)
        optimizer.zero_grad()
        logits = model(x)
        loss = criterion(logits, y)
        loss.backward()
        optimizer.step()

        batch_size = x.size(0)
        total_loss += loss.item() * batch_size
        total_seen += batch_size
    return total_loss / total_seen

def evaluate(model, loader, criterion, device):
    model.eval()
    total_loss, total_correct, total_seen = 0.0, 0, 0
    with torch.no_grad():
        for x, y in loader:
            x, y = x.to(device), y.to(device)
            logits = model(x)
            loss = criterion(logits, y)
            pred = logits.argmax(dim=1)

            batch_size = x.size(0)
            total_loss += loss.item() * batch_size
            total_correct += (pred == y).sum().item()
            total_seen += batch_size
    return total_loss / total_seen, total_correct / total_seen

for epoch in range(1, 6):
    train_loss = train_one_epoch(model, train_loader, criterion, optimizer, device)
    valid_loss, valid_acc = evaluate(model, valid_loader, criterion, device)
    print(f"epoch={epoch} train_loss={train_loss:.4f} valid_loss={valid_loss:.4f} valid_acc={valid_acc:.3f}")
数据一个 batch 能打印出正确的 shape 和 dtype。
参数训练 step 后至少一部分参数发生变化。

9. 诊断这段验证函数的问题。

题目代码：阅读与诊断practice.py
def evaluate(model, loader, criterion, optimizer):
    model.train()
    losses = []
    for x, y in loader:
        logits = model(x)
        loss = criterion(logits, y)
        loss.backward()
        optimizer.step()
        losses.append(loss)
    return losses
