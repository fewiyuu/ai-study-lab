常见训练问题：从曲线到代码的诊断

写出训练循环只是第一步。真正开始训练后，你会遇到更麻烦的问题：loss 不降、验证集越来越差、loss 变成 NaN、显存涨、指标波动、模型看起来学会了但上线很差。这些问题不能靠一句“调学习率”解决，要按证据排查。

这一节的目标是建立一套最小诊断顺序：先看训练和验证曲线，再判断欠拟合、过拟合、训练不动还是数值炸掉；然后回到代码检查 zero_grad、backward、step、loss 输入、数据划分、train/eval/no_grad 和梯度状态。你不需要马上成为调参高手，但要能说清“坏在哪里，下一步查什么”。

  3. 说明学习率、batch size、正则化、Dropout、数据增强分别影响什么。
  4. 区分 model.train()、model.eval()、torch.no_grad() 在验证中的职责。
  5. 验收一段 AI 生成训练代码，指出最可能导致训练失败的写法。

训练诊断的完整主线

mental model

训练问题不要从“猜一个超参数”开始，而要从证据开始。最小证据通常是四类：训练 loss、验证 loss、训练指标、验证指标。训练集表现告诉你模型有没有把训练数据学进去；验证集表现告诉你这种学习能不能迁移到没见过的数据。只有把两条线放在一起看，才知道是模型能力不够、训练没跑起来，还是泛化出了问题。

诊断顺序可以固定下来：先判断曲线类型，再查代码顺序，再查数据和标签，再查数值稳定性，最后再考虑模型容量、学习率、batch size 和正则化。这样做的好处是不会一上来乱改网络结构。很多训练问题其实不是模型太差，而是少写了 optimizer.step()、验证没切 eval()、标签格式不对、loss 输入错了，或者学习率大到把参数直接推炸。

训练曲线：先判断问题类型

curves

训练曲线是最便宜、最有用的诊断工具。训练 loss 持续下降，说明优化器至少在让模型更贴合训练数据；验证 loss 也下降，说明这种学习有一定泛化能力。训练 loss 很低但验证 loss 变高，通常是过拟合。训练和验证 loss 都很高，通常是欠拟合或训练根本没跑起来。训练 loss 完全不动，要优先查代码顺序和梯度，而不是先换大模型。

看曲线时不要只看某一个 epoch。单个点可能受 batch 随机性影响，趋势才有意义。也不要只看 loss 或只看 accuracy：分类任务中，accuracy 不变但 loss 下降是可能的，因为模型置信度在变；loss 下降但验证 accuracy 不升，也可能说明优化方向和任务指标不完全一致。

欠拟合与过拟合：不要混着修

fit

欠拟合是模型连训练数据都学不好。常见原因是模型太简单、特征不够、训练轮数太少、学习率不合适、loss 或标签写错。它的表现通常是训练 loss 高、训练指标低，验证集也好不到哪里去。修欠拟合时，核心是让模型先有能力学进去：增大模型、训练更久、换更合适的特征或损失函数，或者把学习率调到能真正下降的范围。

过拟合是模型把训练数据学得很好，但对新数据表现差。常见原因是数据少、模型太大、训练太久、数据增强不够、正则化不足，或者训练集和验证集分布不一致。修过拟合时，核心是限制模型只记训练集的能力，或者让它看到更多变化：加数据、数据增强、weight decay、Dropout、早停、减小模型，必要时重新划分数据。

欠拟合怎么修

  • 先确认训练循环真的在更新参数。
  • 增加训练轮数或适当增大模型容量。
  • 检查学习率是否太小或 loss 是否用错。
  • 检查输入特征和标签是否有足够信息。

过拟合怎么修

  • 增加数据或使用数据增强。
  • 使用 weight decay、Dropout、早停。
  • 减小模型或缩短训练。
  • 检查训练/验证分布是否一致。

不要把两类问题混着修。训练集都学不好时，盲目加 Dropout 或强正则可能会更差；训练集很好但验证差时，继续加大模型可能让过拟合更严重。一个简单原则是：先让训练集学得动，再让验证集跟上。

你现在应该能做什么根据“训练集表现”和“验证集表现”决定是增强学习能力，还是增强泛化约束。

训练不动：先查代码顺序和参数是否在变

stuck

loss 不动时，最常见的错误不是模型结构太普通，而是训练循环没有真的更新参数。一次训练 step 的关键顺序是：清梯度、前向、算 loss、反向、更新参数。漏掉 backward()，参数没有梯度；漏掉 optimizer.step()，有梯度也不会更新；漏掉 zero_grad()，梯度会跨 batch 累加，训练方向会变得混乱。

还有一些更隐蔽的写法会让训练不动。例如把 loss.item() 变成普通 Python 数字后再想反向传播，计算图已经断了；创建 optimizer 后又替换了模型层，优化器还拿着旧参数；冻结了参数却忘了重新打开 requires_grad；模型输出和 loss 不匹配，导致梯度信号很差。

错误示例：没有真正更新参数bad_step.py
for x, y in train_loader:
    optimizer.zero_grad()
    logits = model(x)
    loss = criterion(logits, y)
    optimizer.step()        # 错：还没 backward，参数没有可用梯度
检查片段：确认参数是否真的变化param_changed.py
before = next(model.parameters()).detach().clone()

optimizer.zero_grad()
loss = criterion(model(x), y)
loss.backward()
optimizer.step()

after = next(model.parameters()).detach()
print(torch.norm(after - before).item())  # 大于 0 才说明这一组参数变了
顺序zero_grad → forward → loss → backward → step 是否完整？
梯度参数 .grad 是 None、全 0，还是有正常数值？
引用optimizer 持有的是当前模型参数吗？
断图有没有在 loss 前后乱用 .item()、.detach()？

Loss NaN：数值已经炸了

nan

NaN 表示 Not a Number。训练里出现 NaN，说明某一步计算产生了非法数值，例如无穷大、除以 0、对非法值取对数，或者学习率太大导致参数和梯度迅速爆炸。NaN 一旦进入参数，后面的输出、loss、梯度可能都会变成 NaN，所以要尽早定位第一个出问题的 step。

最常见的 NaN 来源有几类。第一，学习率过大，更新幅度太猛。第二，手写 loss 时数值不稳定，比如直接 torch.log(prob)，而 prob 可能接近 0。第三，输入数据本身含 NaN 或无穷大。第四，标签格式或取值不符合 loss 要求，例如 CrossEntropyLoss 的类别标签越界。第五，混合精度训练中梯度溢出，不过当前入门阶段先把基础数值检查做稳。

检查片段：定位非有限数值finite_check.py
def assert_finite(name, tensor):
    if not torch.isfinite(tensor).all():
        raise RuntimeError(f"{name} contains NaN or Inf")

logits = model(x)
assert_finite("logits", logits)

loss = criterion(logits, y)
assert_finite("loss", loss)

loss.backward()

梯度问题：None、太小、太大分别代表不同事

gradient

梯度是参数更新的方向信号。loss.backward() 后，参与计算图且需要训练的参数会得到 .grad。如果某个参数的 .grad 是 None，通常说明它没有参与这次 loss 的计算，或者 requires_grad=False，或者计算图被断开。它不是“梯度刚好为 0”，而是“没有梯度被写进去”。

梯度太小会让训练几乎不动，尤其在很深的网络、饱和激活函数或不合适初始化中常见。梯度太大会让参数更新过猛，loss 震荡甚至 NaN。入门阶段不需要把所有理论一次吃完，但要会检查梯度范数：如果梯度一直是 None、接近 0 或异常大，就说明问题不在“多训练几轮”这么简单。

检查片段：打印梯度范数grad_norm.py
loss.backward()

total_norm = 0.0
for name, p in model.named_parameters():
    if p.grad is None:
        print(name, "grad is None")
        continue
    param_norm = p.grad.detach().norm(2).item()
    total_norm += param_norm ** 2
    print(name, param_norm)

print("total grad norm", total_norm ** 0.5)

梯度裁剪可以限制梯度范数，常见于 RNN、Transformer 或学习率比较敏感的训练。它不是万能修复器。如果代码少了 backward()，裁剪没用；如果 loss 输入错了，裁剪也只是掩盖症状。正确用法是在确认训练逻辑没错后，用它控制异常大的更新。

学习率与 batch size：一个管步子，一个管噪声

lr & batch

学习率决定每次根据梯度更新参数时走多大一步。学习率太小，loss 下降很慢，看起来像没训练；学习率太大，loss 震荡、变高，甚至 NaN。batch size 决定每次估计梯度用了多少样本。batch 很小，梯度噪声大，曲线更抖；batch 很大，梯度更稳定，但显存占用更高，也可能需要重新调整学习率。

这两个参数不能孤立看。把 batch size 加大后，单次更新看到的数据更多，但每个 epoch 的 step 数可能减少；把学习率加大后，收敛可能更快，也可能直接冲过合适区域。实战里常用的方式是先选一个能跑得动的 batch size，再用小范围学习率试验找到 loss 能稳定下降的区间。

正则化与数据：修泛化，不是修所有问题

generalization

正则化的目标是提升泛化。常见手段包括 weight decay、Dropout、数据增强、早停、减小模型。它们不是“训练不好就加一点”的万能药。训练集都学不动时，强正则化可能让模型更学不动；训练集很好但验证集差时，正则化才是更合适的方向。

数据问题经常比模型问题更致命。标签错、训练集和验证集分布不一致、预处理不一致、类别极不平衡、验证集混入训练样本，都会让曲线变得难解释。尤其是数据泄露：模型在验证集中看到了训练信息，验证指标会虚高。看起来很漂亮，实际是评估坏了。

Weight decay

在优化中惩罚过大的权重，让模型别过度依赖某些参数。常通过 optimizer 的 weight_decay 设置。

Dropout

训练时随机屏蔽一部分激活，减少神经元之间的过度依赖。验证时必须切到 eval()。

数据增强

让模型看到同一类别的更多变化。图像里常见翻转、裁剪、颜色扰动，但增强不能改变标签语义。

先看训练集训练集差：先修学习能力或代码。
再看验证集训练好验证差：考虑泛化。
查数据划分验证异常好：怀疑泄露。
最后调正则按证据加 weight decay、Dropout 或增强。
你现在应该能做什么知道正则化主要解决过拟合，而不是拿它修所有训练失败。

验证模式：eval 和 no_grad 管的不是一件事

validation mode

验证时常见两个写法：model.eval() 和 torch.no_grad()。它们经常一起出现，但职责不同。model.eval() 切换模型内部某些层的行为，尤其是 Dropout 和 BatchNorm；torch.no_grad() 关闭梯度记录，减少显存和计算开销。前者管模型状态，后者管计算图。

只写 no_grad() 但不写 eval()，Dropout 仍然随机丢激活，BatchNorm 仍可能按训练状态处理，验证指标会抖或不可信。只写 eval() 但不写 no_grad()，指标可能是对的，但 PyTorch 仍会记录计算图，显存可能上涨。验证结束后，如果还要继续训练，要记得切回 model.train()。

正确示例：验证循环valid_loop.py
model.eval()
valid_loss = 0.0

with torch.no_grad():
    for x, y in valid_loader:
        x, y = x.to(device), y.to(device)
        logits = model(x)
        loss = criterion(logits, y)
        valid_loss += loss.item() * x.size(0)

model.train()

验收训练代码：按证据检查，而不是看起来能跑就算

review checklist

AI 生成训练代码时，最危险的是“能运行但训练不对”。比如 loss 算了但没有反向传播，验证集其实在训练模式下跑，分类任务手动 softmax 后又喂给 CrossEntropyLoss，或者 optimizer 创建后模型层被替换。代码不报错，不代表学习过程正确。

一个可靠的验收方式是同时检查代码结构和运行证据。结构上看数据、模型、loss、optimizer、训练 step、验证 step 是否各司其职；运行上看 loss 是否合理下降、参数是否变化、梯度是否有限、验证流程是否不记录梯度、训练和验证指标是否分开记录。

数据输入 shape、标签类型、类别范围、训练/验证划分是否正确？
loss输出 logits 和 loss 是否匹配？分类任务是否误用 softmax？
stepzero_grad/backward/step 是否完整且顺序正确？
证据有没有记录曲线、参数变化、梯度和验证指标？
模板：带基础诊断的训练 stepsafe_train_step.py
def train_step(model, batch, criterion, optimizer, device):
    model.train()
    x, y = batch
    x, y = x.to(device), y.to(device)

    optimizer.zero_grad()
    logits = model(x)
    loss = criterion(logits, y)

    if not torch.isfinite(loss):
        raise RuntimeError("loss is NaN or Inf")

    loss.backward()
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=5.0)
    optimizer.step()

    return loss.item()
