模型保存、加载和验证

训练完成后，模型还只是内存里的一个对象。程序一关，参数就没了。要让训练结果能复用，需要把模型参数保存到文件里；要继续训练，需要保存优化器状态和训练进度；要做推理，需要重新创建同样结构的模型，加载权重，切到评估模式。

这一节讲完整的工程闭环：state_dict 是什么，为什么通常保存它而不是整个模型对象；只保存权重和保存 checkpoint 有什么区别；怎样保存验证集最好的模型；怎样从 GPU 保存的文件加载到 CPU；怎样写可信的验证和推理代码。学完你应该能避免“文件是有了，但加载不回来、继续训练接不上、验证结果不可信”的常见坑。

  2. 写出保存权重、加载权重、保存 checkpoint、恢复训练的最小代码。
  3. 区分“最后一轮模型”和“验证集最佳模型”。
  4. 处理 map_location、设备迁移、结构不匹配、缺失 key 等加载问题。
  5. 写出可信验证流程：eval()、no_grad()、指标聚合、随机性记录。

从训练结果到可复用模型

mental model

保存模型不是把“模型这个概念”存起来，而是把模型里可学习参数的当前数值存起来。PyTorch 里最常用的方式是 state_dict：它是一个字典，key 是参数或缓冲区名称，value 是对应张量。模型结构仍然由你的 Python 类定义，权重文件只负责把这些张量填回去。

一个完整训练产物通常包含三层信息。第一层是模型权重，用于推理或评估。第二层是 checkpoint，用于恢复训练，它除了模型权重，还需要 optimizer 状态、epoch、最好指标等。第三层是实验记录，包括模型结构、数据划分、预处理、随机种子、指标计算方式。只保存第一层，能推理；保存前两层，能接着训；三层都记录，结果才更容易复现。

state_dict：模型内部状态的字典

state dict

model.state_dict() 返回一个有序字典。对普通神经网络来说，它主要包含每一层的权重和偏置，比如 features.0.weight、classifier.1.bias。如果模型里有 BatchNorm，还会包含运行均值、运行方差等 buffer。它们不是普通可训练参数，但会影响验证和推理结果。

optimizer.state_dict() 保存的是优化器状态。以 Adam 为例，它不只保存学习率，还保存每个参数的一阶、二阶动量估计。继续训练时如果不恢复 optimizer 状态，训练可以跑，但相当于优化器“忘了历史”。短训练可能影响不明显，长训练或学习率调度场景里会改变后续轨迹。

观察 state_dict 里有什么inspect_state.py
for name, tensor in model.state_dict().items():
    print(name, tuple(tensor.shape), tensor.dtype)

print(optimizer.state_dict().keys())
# dict_keys(['state', 'param_groups'])
model.state_dict()

模型参数和 buffer：权重、偏置、BatchNorm 统计量等。用于加载模型行为。

optimizer.state_dict()

优化器内部状态：学习率配置、动量、Adam 的历史估计等。用于恢复训练。

scheduler.state_dict()

学习率调度器的当前位置。用了 scheduler 时，恢复训练也要保存它。

metadata

epoch、best metric、类别名、预处理、随机种子、代码版本。用于解释结果。

保存权重：最小推理文件

save weights

最小保存方式是 torch.save(model.state_dict(), path)。这种文件通常用于推理、验证或微调前的初始化。它只保存权重，不保存模型类定义。所以加载时必须先创建同样结构的模型，再调用 load_state_dict。如果结构变了，例如层名变了、输出类别数变了，加载时就可能出现 missing key、unexpected key 或 shape mismatch。

不建议初学阶段默认保存整个模型对象，例如 torch.save(model, path)。它依赖 Python 类所在模块和序列化环境，换文件位置或改类名后更容易出问题。保存 state_dict 更明确，也更接近 PyTorch 官方推荐的常用方式。

最小示例：保存模型权重save_weights.py
from pathlib import Path
import torch

save_dir = Path("checkpoints")
save_dir.mkdir(exist_ok=True)

torch.save(model.state_dict(), save_dir / "model_weights.pt")
保存方式适合用途主要限制
model.state_dict()推理、验证、迁移学习初始化需要重新创建同样结构的模型
完整 checkpoint恢复训练、保存最佳模型和训练进度文件更大，字段要自己约定清楚
整个 model 对象少数临时实验依赖 Python 类路径，长期维护不稳
你现在应该能做什么保存一个只用于推理的权重文件，并解释为什么加载前要重新构建模型。

加载权重：同结构模型接回参数

load weights

加载权重的顺序通常是：创建模型对象，读取权重文件，调用 load_state_dict，把模型移动到目标设备，切到 eval()。最关键的是“同结构”。所谓同结构，不只是层数量一样，还包括层名、参数形状、输出类别数等要对应。类别数从 10 改成 3 时，最后一层权重形状不同，不能直接完整加载。

strict=True 是默认行为，要求 checkpoint 中的 key 和模型当前 key 完全匹配。strict=False 可以允许缺失或多余 key，常用于迁移学习时跳过最后一层。但它不是“万能忽略错误”。如果同名参数形状不同，通常仍然需要手动处理。

最小示例：加载权重用于验证或推理load_weights.py
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

model = TinyCNN(num_classes=3)
state = torch.load("checkpoints/model_weights.pt", map_location=device)
model.load_state_dict(state)
model.to(device)
model.eval()
错误示例：结构变了还强行加载bad_load.py
model = TinyCNN(num_classes=5)
state = torch.load("model_weights_3_classes.pt")
model.load_state_dict(state)  # 最后一层 shape 可能不匹配

Checkpoint：保存一次训练状态快照

checkpoint

如果你只是想把模型拿去推理，保存模型权重就够了。如果你想中断后继续训练，就需要 checkpoint。它至少包含模型权重、优化器状态、当前 epoch、最好验证指标。用了学习率调度器，也要保存 scheduler 状态。这样恢复后，训练不只是“从相同权重继续”，而是尽量接回原来的优化过程。

checkpoint 的字段没有唯一标准，但要清楚命名。常见字段包括 model_state、optimizer_state、epoch、best_valid_loss、config。保存字段越清楚，后面越不容易忘记这个文件到底是什么。

完整示例：保存 checkpointsave_checkpoint.py
checkpoint = {
    "epoch": epoch,
    "model_state": model.state_dict(),
    "optimizer_state": optimizer.state_dict(),
    "best_valid_loss": best_valid_loss,
    "config": {
        "model": "TinyCNN",
        "num_classes": 3,
        "batch_size": 32,
    },
}
torch.save(checkpoint, "checkpoints/last.pt")
完整示例：恢复训练resume_training.py
ckpt = torch.load("checkpoints/last.pt", map_location=device)

model = TinyCNN(num_classes=ckpt["config"]["num_classes"]).to(device)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

model.load_state_dict(ckpt["model_state"])
optimizer.load_state_dict(ckpt["optimizer_state"])

start_epoch = ckpt["epoch"] + 1
best_valid_loss = ckpt["best_valid_loss"]

保存最佳模型：不要默认最后一轮最好

best model

训练最后一轮不一定是最佳模型。过拟合时，训练 loss 可能继续下降，但验证 loss 已经变差。真实使用时，你通常更关心验证集表现最好的那一轮，而不是最后一轮。因此训练循环里常会保存 best.pt 和 last.pt：前者用于最终评估或部署，后者用于恢复训练。

保存最佳模型时，指标方向要明确。验证 loss 是越小越好，accuracy、F1、AUC 通常是越大越好。不要把比较方向写反。还要注意，如果保存的是 model.state_dict()，最好在指标变好时立刻保存到文件，而不是只把 best_state = model.state_dict() 留在内存里，因为这个字典里的张量引用可能后续被更新影响。更稳妥是保存文件，或对张量做深拷贝。

示例：按验证 loss 保存最佳权重save_best.py
best_valid_loss = float("inf")

for epoch in range(num_epochs):
    train_one_epoch(...)
    valid_loss = evaluate(...)

    torch.save({
        "epoch": epoch,
        "model_state": model.state_dict(),
        "optimizer_state": optimizer.state_dict(),
        "best_valid_loss": best_valid_loss,
    }, "checkpoints/last.pt")

    if valid_loss < best_valid_loss:
        best_valid_loss = valid_loss
        torch.save(model.state_dict(), "checkpoints/best.pt")
epoch 1验证一般，先保存 last。
epoch 2验证变好，更新 best。
epoch 3继续变好，再更新 best。
epoch 4验证变差，只更新 last。
最终评估用 best，恢复训练用 last。

设备映射：GPU 保存，CPU 也要能加载

device

模型参数是张量，张量有设备属性。你可能在 GPU 上训练并保存文件，但在没有 GPU 的机器上加载。如果直接加载，可能因为文件里的张量指向 CUDA 设备而报错。torch.load(..., map_location=device) 可以把保存文件中的张量映射到当前目标设备。

加载后还要注意模型和输入要在同一设备。常见错误是模型在 GPU，输入还在 CPU，或者反过来。保存加载流程中，稳妥写法是先确定 device，加载时使用 map_location，模型 to(device)，每个 batch 的输入和标签也 to(device)。

示例：跨设备加载map_location.py
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

state = torch.load("checkpoints/best.pt", map_location=device)
model = TinyCNN(num_classes=3)
model.load_state_dict(state)
model.to(device)
model.eval()

CUDA 相关加载错误GPU 保存，CPU 环境加载使用 map_location="cpu" 或目标 device
Expected all tensors to be on same device模型和输入不在同一设备模型和 batch 都 to(device)
加载后指标异常模型未切 eval() 或预处理不一致检查推理模式和数据处理配置

推理与验证：加载后要证明它可信

validation

加载权重后，不要只跑一个样本看有没有报错。至少要在验证集上重新计算指标，并确认和保存时记录的指标接近。验证时要切 model.eval()，使用 torch.no_grad()，把 loss 按样本数加权汇总，分类指标要按正确维度取预测类别。

指标聚合也会出错。比如每个 batch 的 loss 直接平均，如果最后一个 batch 比较小，会让整体 loss 有偏差。更稳妥的方式是把每个 batch 的 loss 乘以 batch size 累加，最后除以样本总数。accuracy 则统计预测正确的样本数，再除以总样本数。

示例：可信验证函数evaluate.py
def evaluate(model, loader, criterion, device):
    model.eval()
    total_loss = 0.0
    total_correct = 0
    total_count = 0

    with torch.no_grad():
        for x, y in loader:
            x, y = x.to(device), y.to(device)
            logits = model(x)
            loss = criterion(logits, y)

            batch_size = x.size(0)
            total_loss += loss.item() * batch_size
            total_correct += (logits.argmax(dim=1) == y).sum().item()
            total_count += batch_size

    return {
        "loss": total_loss / total_count,
        "accuracy": total_correct / total_count,
    }

设备模型、输入、标签在同一设备。
聚合loss 按样本数加权，指标按总样本数统计。
一致预处理、类别顺序和训练时一致。

复现记录：文件旁边要有上下文

reproducibility

一个权重文件如果没有上下文，过几天你可能就不知道它来自哪个模型、哪个数据划分、哪个类别顺序、哪个预处理。模型保存不是只留下 .pt 文件，还要留下最少配置。至少记录：模型类名、关键超参数、训练/验证数据来源、类别到 id 的映射、预处理方式、最好验证指标、保存时间。

随机性也会影响结果。深度学习不一定每次完全复现，尤其在 GPU 和某些算子下。但设置随机种子、记录库版本、固定数据划分，可以让结果更接近，也让排查问题更容易。你不必一开始追求工业级实验管理，但不能让模型文件变成来历不明的黑盒。

metadata = {

    "model_state": model.state_dict(),
