Transformer / LLM Foundation 01

Tokenizer、Embedding 与 Context Window

大语言模型看起来在读文字，但神经网络不能直接处理“字”或“词”的抽象概念。进入模型前，文本要先被 tokenizer 切成 token，再把 token 映射成整数 id，最后通过 embedding table 查成向量。Transformer 后续处理的是这些向量序列。

这一节不急着讲 attention。先把输入端讲稳：token 不一定等于一个汉字或一个英文单词；embedding 是模型里的可训练查表层，不只是向量数据库里的检索向量；context window 是模型一次能处理的 token 数上限，不是长期记忆。学完你应该能解释“为什么同一段中文和英文 token 数不一样”“为什么超长文本要截断或分块”“为什么提示词越长成本越高”。

学完你应该能做什么

  1. 解释文本从字符串到 token、token id、embedding 向量的完整链路。
  2. 区分字符、词、token、token id、embedding、context window。
  3. 说明 context window 为什么限制模型一次能看的内容。
  4. 判断超长输入、截断、分块、特殊 token、未知词等常见问题。
  5. 验收一段 LLM 调用或训练代码里和 tokenizer/embedding 相关的基础错误。

LLM 输入端的完整主线

mental model

大模型的输入链路可以固定成五步：原始文本、token、token id、embedding、位置和上下文。原始文本是人能读懂的字符串。token 是 tokenizer 切出来的片段。token id 是词表里的整数编号。embedding 是把整数编号查成向量。Transformer 后面的层拿到的是一串向量，而不是原始文字。

这条链路里任何一步出错，后面的理解都会偏。把 token 当成词，会误判输入长度；把 embedding 当成“语义已经完全懂了”，会高估查表层的能力；把 context window 当成记忆，会误以为模型能永久记住前面对话。你以后读 LLM 文档、写 prompt、做 RAG 或看 Transformer 代码，都会反复遇到这些概念。

文本为什么不能直接进模型

text to numbers

神经网络的计算对象是数字张量。线性层、注意力、归一化、softmax 都是在数字上计算。原始文本里的“猫”“cat”“😀”不是可以直接矩阵乘法的对象。要让模型处理语言，第一步就是把离散文本变成数字序列。

最粗暴的办法是按字符编号，中文每个字一个 id，英文每个字符一个 id。但这样会让序列很长，也不利于复用常见片段。另一种办法是按词编号，但英文有变形，中文没有天然空格，罕见词和新词会很麻烦。现代 LLM 常用子词级 tokenizer，把常见词、词根、符号、中文片段等切成 token，在“词表大小”和“序列长度”之间折中。

字符级粒度细，词表小，但序列可能很长。
词级直观，但新词、变形、中文分词很麻烦。
子词级常见折中，LLM 里很常见。
字节级覆盖能力强，可处理更多未知文本。
特殊 token表示开始、结束、填充、角色边界等控制信息。

这里的重点不是背某一种 tokenizer 算法，而是理解它解决的问题：把开放的自然语言输入，稳定地变成有限词表中的整数序列。只有这样，模型才能用固定大小的 embedding table 查向量。

你现在应该能做什么解释为什么大模型需要 tokenizer，而不是直接把原始字符串送进神经网络。

Tokenizer：token 不等于词

tokenizer

tokenizer 的输出不是人类语言学意义上的“词”。一个英文单词可能被切成多个 token，一个中文词也可能被切成一个或多个 token，标点、空格、换行、emoji 也可能占 token。tokenizer 的切分由它的词表和训练规则决定，不是由你的直觉决定。

例如英文里 unbelievable 可能被切成 un、believ、able 这样的片段；中文“深度学习”可能被切成“深度”和“学习”，也可能按字或更细片段切。不同模型用的 tokenizer 不同，同一句话 token 数可能不同。所以估算上下文长度和调用成本时，不能只按字符数或词数算。

对象它是什么容易错在哪里

词人理解的语言单位tokenizer 不一定按词切

Token ID：词表里的整数编号

vocabulary

tokenizer 切出 token 后，还要把每个 token 映射成 token id。词表可以理解为一个映射表：每个 token 对应一个整数。模型真正接收的常见输入字段叫 input_ids，形状通常是 [batch, seq_len]。这里的每个数字都只是编号，不是这个 token 的语义大小。

同一个 id 在不同 tokenizer 里可能代表完全不同的 token。不能把 A 模型 tokenizer 产生的 id 直接喂给 B 模型，除非它们明确共享 tokenizer 和词表。训练时 tokenizer、embedding table 和模型参数是配套的，换 tokenizer 等于改变输入编号系统。

最小示例：token 到 input_ids 的形状token_ids.py
# 这里是伪代码，用来说明字段和形状
texts = ["深度学习", "hello world"]
batch = tokenizer(texts, padding=True, truncation=True)

print(batch["input_ids"].shape)
# [batch_size, seq_len]

print(batch["attention_mask"].shape)
# 同样是 [batch_size, seq_len]，用来标记哪些位置是真 token

input_ids

token id 序列。每个整数用于查 embedding table 的一行。

attention_mask

标记哪些位置是真实输入，哪些是 padding。padding 不应该被当成有效内容处理。

Embedding：把离散 id 查成连续向量

embedding

token id 本身只是编号，不能直接表达语义距离。id 100 和 id 101 不一定更相似，它们只是词表编号相邻。embedding 层的作用是把每个 token id 映射成一个可训练向量。这个向量维度通常叫 d_model 或 hidden size。Transformer 后续层处理的就是形状为 [batch, seq_len, d_model] 的向量序列。

在 PyTorch 里，embedding 可以看成查表：nn.Embedding(vocab_size, d_model) 有一个权重矩阵，形状是 [vocab_size, d_model]。输入 token id 时，模型取出对应行作为该 token 的向量。训练过程中，这张表和其他模型参数一起被更新。

Embedding(input_ids):[B, L]→[B, L, D]
向量维度 D8
最小示例：nn.Embedding 的输入输出embedding_shape.py
import torch
from torch import nn

embedding = nn.Embedding(num_embeddings=50000, embedding_dim=8)
input_ids = torch.tensor([[101, 2048, 305], [101, 998, 0]])

x = embedding(input_ids)
print(x.shape)  # [2, 3, 8]

embedding 有两种常见语境。第一种是模型内部的 token embedding，负责把 token id 变成 Transformer 输入向量。第二种是应用里的文本 embedding，比如把整段文本编码成一个向量，用于相似度检索或 RAG。两者都叫 embedding，但对象和用途不同：一个是 token 级输入表示，一个常是句子或文档级语义表示。

你现在应该能做什么说清 nn.Embedding(vocab_size, d_model) 的权重形状、输入形状和输出形状。

Context Window：一次能看的 token 上限

context window

context window 是模型一次前向计算能处理的最大 token 数。它限制的是 token 序列长度，不是中文字符数、英文单词数或消息条数。系统提示、用户输入、历史对话、检索资料、模型要生成的输出，通常都会消耗上下文空间。输入太长时，系统需要截断、摘要、分块或选择更长上下文的模型。

context window 不是长期记忆。模型只能在当前输入窗口里直接使用内容。聊天应用看起来“记得前面”，通常是因为系统把历史消息再次放进当前请求里，或者使用了外部记忆/检索机制。超过窗口的内容如果没有被重新放进输入，模型就无法直接注意到。

摘要保留整体信息，压缩细节细节可能丢失或被摘要改写

长度、成本和注意力计算

length matters

token 数不只是影响能不能放进上下文，也影响成本和速度。调用模型时，输入 token 和输出 token 通常都会计费。输入越长，模型要读的内容越多；输出越长，模型要生成的步骤越多。对 Transformer 来说，序列长度还会影响注意力计算量。

标准自注意力里，每个 token 都要和其他 token 计算关系，所以注意力矩阵大小大致是 L × L。这就是为什么长上下文模型很有用，但也更贵、更难训练。实际系统会用很多优化方法，但入门阶段先记住一个判断：把无关内容塞进 prompt，不只是让模型“多看一点”，也会占上下文、增加成本、增加干扰。

Attention score matrix≈L × L

输入太长

可能超过窗口，也可能让重要信息被淹没。

输出太长

占用输出预算，增加延迟和费用。

资料太杂

模型不一定更聪明，反而可能被无关上下文干扰。

常见误区：这些概念最容易混

mistakes

第一类误区是把 token 当成词。它会导致你估算长度错误，尤其是中英文混合、代码、表情、特殊符号很多的时候。正确做法是用对应模型的 tokenizer 计算 token 数。

第二类误区是把 token id 当成语义数值。id 大小没有语义大小含义，id 相邻也不代表语义相近。语义信息主要通过 embedding 和后续训练学出来，而不是整数编号本身。

第三类误区是混淆 token embedding 和文本 embedding。模型输入端的 embedding 是每个 token 的向量；RAG 里常说的 embedding 通常是整段文本的向量，用来做相似度检索。名字一样，粒度和用途不同。

第四类误区是把 context window 当记忆。窗口外的信息如果没有被重新放入当前输入，模型就不能直接使用。长期记忆需要外部机制，比如检索、摘要、数据库或应用层状态。

token ≠ 词切分由 tokenizer 和词表决定。
id ≠ 语义整数编号只是查表索引。
embedding 分语境token 级输入向量和文本检索向量不是一回事。
window ≠ 记忆窗口外内容不自动可见。
你现在应该能做什么听到 tokenizer、embedding、context window 时，先判断它们在当前语境里的具体对象。

完整链路：一句话如何进入模型

workflow

把前面内容串起来：用户输入一段文本，tokenizer 先按词表规则切分成 token，并映射成 token id。为了批量处理，不同长度的样本可能会 padding 到同一长度，同时生成 attention mask。模型的 embedding 层读取 input_ids，把每个 id 映射成向量。再加上位置信息后，向量序列进入 Transformer 层。Transformer 最终输出每个位置的 hidden states，语言模型头再把最后位置或每个位置映射回词表 logits，用来预测下一个 token。

这条链路也是你验收 LLM 代码的基础。输入字段是不是对应 tokenizer 产生的？有没有截断或 padding？attention mask 有没有传？embedding 输出 shape 是否是 [B,L,D]？生成时是否给输出预留 token 预算？这些问题比“模型名字看起来很强”更具体，也更能定位错误。

最小示例：从文本到 embedding 形状llm_input_shape.py
# 伪代码：说明 LLM 输入端的形状变化
batch = tokenizer(
    ["深度学习是什么？", "What is attention?"],
    padding=True,
    truncation=True,
    return_tensors="pt",
)

input_ids = batch["input_ids"]          # [B, L]
attention_mask = batch["attention_mask"] # [B, L]

x = model.embed_tokens(input_ids)  # [B, L, D]
