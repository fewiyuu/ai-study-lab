预训练与微调的区别

大模型不是一开始就会聊天。通常先用海量文本做预训练，让模型学会根据上下文预测下一个 token，获得语言、知识和模式的基础能力。然后再用更小、更有目标的数据做微调，让模型更像你想要的样子：会按指令回答、会遵守格式、会处理某个领域任务。

预训练和微调的差别不只是“数据多少”。它们的目标、数据形态、成本、更新范围、产物都不同。预训练像从零训练底座，成本极高；监督微调让底座学会指令行为；LoRA 这类参数高效微调只训练少量新增参数；偏好对齐则进一步让模型输出更符合人类偏好或安全要求。

从底座到可用助手的训练链路

mental model

LLM 的训练链路可以粗略分成四段：预训练、监督微调、偏好对齐、应用适配。预训练让模型学会语言建模，也就是在大量文本里预测下一个 token。监督微调用指令-回答数据教模型按人类任务格式输出。偏好对齐让模型在多个可选回答中更偏向人类认为好的回答。应用适配可能是 LoRA、RAG、prompt 模板或工具调用。

这几段不是永远都要自己做。普通学习者和大多数公司不会从零预训练大模型，因为数据、算力和工程复杂度太高。更常见的是选一个已有底座或指令模型，再根据任务选择 prompt、RAG、LoRA、全量微调或继续预训练。真正重要的是知道每种方法改变了什么，以及它不能解决什么。

预训练预测下个 token
SFT学指令格式
偏好对齐学偏好

直觉

预训练学“语言和世界模式”，微调学“怎样按任务表现出来”。

正式说法

预训练优化通用语言建模目标，微调在特定数据分布上继续优化模型或适配参数。

判断句

想让模型知道新资料，通常先考虑 RAG；想让模型改变稳定行为，才考虑微调。

预训练：从大量文本里学下一个 token

pretraining

预训练的核心目标通常是 next-token prediction：给定前面的 token，预测下一个 token。模型不是直接被教“什么是法律”“怎么写代码”，而是在海量文本中反复学习上下文到下一个 token 的概率分布。语言规律、事实知识、代码模式、推理痕迹，都通过这个目标间接进入参数。

预训练数据规模很大，通常包含网页、书籍、代码、论文、论坛等多种来源。训练会更新模型几乎全部参数，包括 embedding、attention、FFN、输出头等。成本来自三个方面：数据处理、算力和训练稳定性。学习阶段不需要自己做大规模预训练，但要理解它为什么是底座能力的来源。

目标:最大化P(tokent | token<t)
伪代码：语言模型预训练 batchpretrain_batch.py
# input_ids:  [B, L]
# labels:     [B, L]，通常是右移后的下一个 token
logits = model(input_ids)
loss = cross_entropy(logits.view(-1, vocab_size), labels.view(-1))
loss.backward()
optimizer.step()

微调：在已有模型上继续改变行为

fine-tuning

微调是在已有模型基础上继续训练。它的目标不是从零学语言，而是让模型在某个数据分布、任务格式或领域上表现更好。比如让模型学会客服话术、医学问答格式、代码补全风格、公司内部术语，或者把输出固定成 JSON。

微调可以更新全部参数，也可以只更新少量适配参数。全量微调改变整个模型，成本高、风险也高；参数高效微调只训练少量新增参数，底座大部分权重保持冻结。微调的核心风险是数据质量：如果数据少、脏、格式不统一，模型会学到错误行为；如果任务只需要提供新知识，微调也可能不如 RAG 合适。

你现在应该能做什么判断一个需求是在补知识、改行为、改格式，还是改偏好，再选择方法。

SFT：用示范答案教模型按指令做事

supervised fine-tuning

SFT 是监督微调，常见数据形式是 instruction、input、response。它告诉模型：遇到这类指令时，应该输出什么样的回答。SFT 很适合让模型学会任务格式、回答风格、工具调用格式、领域问答模板。它不是让模型凭空获得所有新知识，如果训练数据里只有少量事实，模型也可能记不牢或泛化差。

SFT 仍然可以用语言模型损失训练。区别在于 loss 通常只对回答部分计算，或者对不同部分做 mask。这样模型重点学习“看到指令后如何生成目标回答”。如果把 prompt 部分也不加区分地纳入损失，可能会让训练目标不够准确。

示例：SFT 数据长什么样sft_sample.json
{
  "instruction": "把下面句子改成更正式的表达",
  "input": "这个方案还行吧。",
  "response": "这个方案具备一定可行性，但仍需要进一步评估。"
}

LoRA / PEFT：少量参数适配大模型

peft

PEFT 是参数高效微调的总称，LoRA 是其中很常见的一种。它的基本想法是冻结原模型大部分权重，只在某些线性层旁边加入低秩适配矩阵，训练这些新增的小参数。这样显存和存储成本更低，也更容易为不同任务保存多个适配器。

LoRA 不等于“免费变强”。它仍然需要高质量数据、合适学习率和评估。它适合做领域风格、格式、任务行为的适配，但如果底座模型本身能力不足，LoRA 也不能凭空创造强能力。LoRA adapter 通常需要和原底座模型一起使用；只有 adapter 文件，没有对应底座，不能独立推理。

偏好对齐：让输出更符合人类偏好

alignment

SFT 给模型示范“应该怎么回答”，偏好对齐进一步处理“多个回答里哪个更好”。常见数据形式是同一个 prompt 下有 chosen 和 rejected 两个回答，训练目标让模型更偏向 chosen。RLHF、DPO 等方法都属于这一类思路的不同实现。

偏好对齐不是补知识的主要手段。它更关注帮助性、安全性、礼貌、拒答边界、格式偏好等行为层面的选择。如果模型不知道某个事实，单靠偏好对齐不能稳定让它知道；如果 SFT 数据已经很差，偏好对齐也很难完全补救。它通常位于预训练和 SFT 之后。

什么时候微调，什么时候不微调

decision

微调不是遇到效果不好就上的第一招。先问几个问题：模型是否只是缺少外部资料？如果是，RAG 可能更合适。是否只是输出格式不稳定？prompt 模板、few-shot 示例或结构化输出约束可能够用。是否需要稳定改变某类行为，且有足够高质量样本？这时才更像微调需求。

微调前还要有评估集。没有评估，你不知道模型是变好了，还是只在训练样本上记住了。微调后要检查通用能力是否退化、格式是否稳定、幻觉是否增加、安全边界是否变差。尤其是小数据微调，很容易过拟合到某种口吻或固定模板。

缺资料优先 RAG 或上下文注入。
缺格式先试 prompt、模板、约束解码。
缺稳定行为考虑 SFT 或 LoRA。
缺偏好考虑偏好数据和对齐训练。
你现在应该能做什么判断一个业务需求是否值得微调，而不是把微调当万能按钮。

常见误区：训练词汇不要乱用

mistakes

第一个误区：把所有训练都叫预训练。预训练通常指从大规模通用文本中学习基础语言模型；在已有模型上用任务数据继续训练，更准确叫微调、继续预训练或领域适配。

第二个误区：以为微调就是给模型塞知识。微调可能让模型记住一些知识，但如果目标是随时接入新资料，RAG 更可控、更容易更新。微调适合改变行为和分布，不适合当作动态知识库。

第三个误区：以为 LoRA 一定不伤模型。LoRA 改的是适配器参数，但输出行为仍可能被坏数据带偏。数据质量、学习率、训练轮数、评估集都必须检查。

第四个误区：只看训练 loss。微调训练 loss 降低，不代表真实任务变好。模型可能学会背训练集、输出模板化废话，甚至通用能力退化。必须有验证集和任务指标。

预训练不是所有继续训练。
微调不是动态知识库。
LoRA不是质量保证。
loss不是唯一验收指标。
