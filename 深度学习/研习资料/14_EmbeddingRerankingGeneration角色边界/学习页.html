Transformer / LLM Foundation 05

Embedding、Reranking、Generation 的角色边界

很多人第一次做问答系统，会把“让模型回答得更好”理解成换一个更强的大模型。这个判断太粗。真实链路里，回答质量往往由三段共同决定：embedding 先把问题和文档变成向量，快速找出可能相关的候选；reranking 再用更细的模型重新排序，把真正该看的材料放前面；generation 最后读取上下文，组织成可读答案。

三段的错误长得不一样。embedding 没召回，后面再强也看不到正确材料。reranking 排错，正确材料可能被挤出上下文窗口。generation 乱写，就算材料在上下文里，也可能没有引用、没有承认不知道，甚至把材料外的内容编进去。这节要把这些边界讲清楚：每一段负责什么，不负责什么，怎么验收，出问题时该查哪里。

学完你应该能做什么

  1. 解释 embedding、reranking、generation 在检索增强问答中的分工。
  2. 说清向量相似度、重排分数、生成概率不是同一种东西。
  3. 判断一个坏回答是召回问题、排序问题、上下文组织问题，还是生成问题。
  4. 读懂一个最小 RAG 伪代码：切分、向量化、召回、重排、拼 prompt、生成。
  5. 给出验收清单：recall@k、MRR/NDCG、引用准确性、忠实性、答案可用性分别看什么。

先把一条问答链路拆开

mental model

用户问一个问题时，系统通常不会把整个知识库塞给大模型。知识库太大，上下文窗口有限，成本也高。于是系统先把文档切成 chunk，再提前算好每个 chunk 的 embedding。用户问题来了，也算一个 query embedding，用向量相似度从大量 chunk 里快速召回一批候选。这个阶段追求“别漏掉可能相关的材料”，所以它通常要快，要覆盖广。

召回后的候选仍然可能很杂。向量相似度擅长粗粒度语义接近，但它不总能理解细节条件、否定、数字、时间、问题焦点。reranking 会把 query 和每个候选放在一起重新打分，通常比纯向量检索慢，但判断更细。最后 generation 才把重排后的少量材料放进 prompt，生成自然语言答案。

Embedding

把文本变成向量，用相似度快速找候选。它擅长召回，不擅长最后裁判。

Reranking

拿问题和候选逐对比较，重新排序。它负责把最相关的材料推到前面。

Generation

读取上下文并写答案。它负责表达、整合和引用，不负责凭空知道没检索到的材料。

你现在应该能做什么看到一个 RAG 系统，先把它拆成召回、重排、生成三段，再讨论优化。

为什么不能只靠生成模型

why retrieval

生成模型会把 prompt 里的信息和自己参数里的知识一起用来回答。问题是参数知识不一定新，不一定覆盖你的私有资料，也不一定能给出可追溯来源。把外部资料直接塞进 prompt 可以解决一部分问题，但资料多起来后，上下文窗口、成本和噪声都会变成限制。

检索链路的价值是把“找资料”和“写答案”分开。embedding 和 reranking 负责把正确材料送到模型面前，generation 负责把材料组织成回答。分工之后，出错也能定位：没找到材料是检索问题；找到了但没排前是重排问题；材料在上下文里却答错，是生成或 prompt 约束问题。

只靠生成简单，但容易旧知识、编造、无来源。
只靠向量检索快，但返回的是材料，不是最终答案。
检索 + 生成把资料送进上下文，再让模型组织表达。
你现在应该能做什么解释 RAG 不是“让模型更聪明”的魔法，而是把外部资料放进可控上下文。

Embedding：快速召回候选材料

embedding retrieval

embedding 模型把一段文本映射成一个向量。两个文本语义接近时，它们的向量通常也更接近。检索时，系统会提前把文档 chunk 的向量存进向量库；用户问题来了，再把问题向量化，用余弦相似度或内积找最接近的一批 chunk。

embedding 的核心职责是召回。召回阶段宁可多拿一些候选，也不要太早把正确材料漏掉。因为一旦正确 chunk 没进入 top-k，后面的 reranker 和 generator 都看不见它。embedding 不负责生成答案，也不负责细读复杂条件。它给的是“可能相关”的候选列表，不是最终结论。

chunk 粒度太大噪声多，太小上下文不完整不按段落/标题切，句子被切断

Reranking：把候选材料重新排队

reranking

reranker 接收 query 和候选 chunk，逐对打分。它通常不是只看两个向量距离，而是把 query 和 chunk 一起读进去，判断“这个 chunk 是否真正回答这个问题”。这让它更适合处理细节条件，例如年份、否定词、限定范围、某个具体字段。

reranking 的代价更高，因为它要对多个候选逐个比较。常见做法是先用 embedding 从海量文档中召回 20 到 100 个候选，再用 reranker 挑出前 3 到 8 个放入上下文。它不是替代 embedding，而是接在粗召回后面做精排。

0.48

为什么相似不等于最该引用

一个 chunk 可能和问题主题很像，但缺少关键条件。例如问题问“2024 年的退款规则”，候选 A 讲退款规则但来自 2022 年，候选 B 语义略远但明确写了 2024 年。embedding 可能更喜欢 A，reranker 应该把 B 提前。

判断句

如果正确材料已经在召回列表里，但没进入最终上下文，优先怀疑 reranking、top_n 或上下文截断。

你现在应该能做什么解释 reranker 为什么慢一点但更会“读题”，以及它为什么不能弥补召回阶段完全漏掉的材料。

Generation：基于上下文组织答案

generation

generation 阶段读的是用户问题、检索到的上下文、系统提示和输出约束。它要把分散材料组织成答案，处理格式、语气、引用、承认不知道和冲突信息。它不是简单复读 chunk，也不应该脱离上下文自由发挥。

生成阶段最重要的约束是忠实性：答案里的事实应该能从上下文中找到依据。如果上下文没有答案，模型应该说缺少材料，而不是补一个看起来合理的结论。好的生成 prompt 会要求模型只基于给定材料回答，标注引用，遇到材料不足时说明不足。

完整链路：从粗召回到最终答案

pipeline

一个最小 RAG 链路可以分成离线和在线两部分。离线部分把文档清洗、切分、向量化并写入索引。在线部分接收用户问题，计算 query embedding，向量召回候选，用 reranker 精排，选择前几段拼成上下文，再交给生成模型。

这条链路的核心不是“组件越多越高级”，而是每个组件有明确位置。embedding 阶段看 recall，reranking 阶段看排序质量，generation 阶段看答案忠实性和可用性。把指标混了，就会做出错误优化：比如回答编造了，你去调 top_k；召回漏了，你去换更大的生成模型。

伪代码：最小 RAG 流程rag_min.py
# 离线：文档入库
chunks = split_documents(raw_docs)
chunk_vectors = embed(chunks)
vector_store.add(chunks, chunk_vectors)

# 在线：用户提问
query_vector = embed([query])[0]
candidates = vector_store.search(query_vector, top_k=30)
ranked = reranker.rank(query, candidates)
context = build_context(ranked[:5])
answer = generator.answer(query, context)

指标与验收：每一段看不同证据

evaluation

验收不能只问“最终回答好不好”。最终回答差时，你还要知道是哪一段导致的。召回阶段常看 recall@k：正确材料有没有出现在前 k 个候选里。重排阶段可以看 MRR 或 NDCG：正确材料是否排得足够靠前。生成阶段看忠实性、引用准确性、答案完整性和格式可用性。

如果你没有分段日志，就很难定位问题。至少应该保存 query、召回列表、相似度分数、重排后列表、最终上下文、生成答案。这样一条坏 case 出现时，你可以沿着链路回放，而不是只盯着最后一句答案。

怎么判断问题该归谁

debug boundary

坏回答通常可以沿链路定位。先看正确材料有没有进入向量召回 top-k。如果没有，这是 embedding、chunk、query 改写或索引问题。再看正确材料是否被 reranker 排到可进入上下文的位置。如果没有，这是 reranking 或 top_n 设置问题。再看最终 prompt 里是否真的包含正确材料。如果包含但答案仍然错，那才更多指向 generation、prompt 或模型遵循能力。

这个顺序很重要。很多人看到答案错，第一反应是换更大模型。但如果正确材料根本没进上下文，大模型只能靠旧知识猜。反过来，如果材料已经在上下文里，大模型仍然编造，那继续调向量库意义不大，要查生成约束、引用要求、上下文太长导致注意力分散，或者模型本身遵循能力不够。

材料没召回

查切分、embedding 模型、top_k、索引更新、query 表达。

材料召回但没入上下文

查 reranker、top_n、分数阈值、上下文预算。

材料在上下文但答错

查 prompt、引用约束、冲突处理、模型遵循能力。

常见误区：分数和责任不要混

mistakes

第一个误区：把 embedding 分数当成答案置信度。向量相似度只表示 query 和 chunk 在向量空间里接近，不表示答案一定正确，也不表示模型对最终回答有多自信。第二个误区：以为 reranker 越强越能解决所有问题。reranker 只能重排已有候选，候选里没有正确材料时，它也没法凭空排出来。

第三个误区：把 generation 当作知识库。生成模型可以组织语言，但外部资料没有进入上下文时，它不该假装知道。第四个误区：只调 top_k。top_k 太小会漏召回，太大又可能增加噪声和成本；真正要看的是正确材料有没有进候选、有没有排前、有没有进上下文。

第五个误区：只看最终答案，不留中间日志。没有中间日志，你无法判断问题在哪一段，只能凭感觉调参。最低限度也要记录召回列表、重排列表、最终上下文和答案。

错误示例：把相似度分数写成答案可信度wrong_confidence.py
hits = vector_store.search(query_vector, top_k=3)
best = hits[0]

answer = generator.answer(query, context=best.text)
print(f"答案可信度：{best.similarity:.2f}")  # 错：这是检索相似度，不是答案置信度

错因

检索分数来自 query 和 chunk 的匹配，不来自生成答案的事实验证。

后果

用户会误以为答案有概率保证，实际只是材料相似。

修法

把检索分数标成“相关性”，答案可信要看引用、材料支持和人工/自动忠实性检查。

你现在应该能做什么听到“相似度 0.9 所以答案可信”时，能马上指出这句话哪里错。
