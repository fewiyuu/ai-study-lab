Prompt 基础与结构化输入输出

Prompt 很容易被讲成玄学：多写几句、换个语气、加几个“请认真思考”。这些有时能改变输出，但不稳定，也不方便工程化。更实用的看法是：prompt 是你和模型之间的一份输入契约。它告诉模型任务是什么、可用材料在哪里、应该按什么步骤处理、哪些事情不能做、最后必须输出什么格式。

结构化输入输出是这份契约能进入程序的关键。自由文本适合聊天，但程序更需要 JSON、列表、字段、枚举值、置信说明和错误状态。只要输出要被下游代码读取，就不能只要求“请整理得清楚一点”，而要明确字段名、字段类型、缺失时怎么填、禁止输出哪些额外文字，以及失败时怎样修复。

学完你应该能做什么

  1. 把 prompt 拆成角色、任务、上下文、约束、示例、输出格式和失败处理。
  2. 区分自然语言提示、模板化 prompt、few-shot 示例和结构化输出约束。
  3. 写出一个能被程序解析的 JSON 输出要求，并知道常见破坏格式的原因。
  4. 诊断 prompt 失败：任务不清、上下文噪声、示例误导、约束冲突、schema 不完整。
  5. 建立最小迭代流程：写 prompt、跑样例、看失败、改约束、加校验、保留版本。

从一句要求到可验收输入

mental model

给模型一句“帮我总结一下”也能得到输出，但你很难验收它。它可能总结得太长，漏掉关键字段，掺入自己的判断，或者把格式写成下游程序读不了的样子。工程里的 prompt 不能只追求“看起来像人写的要求”，还要能被测试、能复用、能定位失败。

一个稳定 prompt 至少要回答七个问题：模型扮演什么角色，当前任务是什么，输入材料在哪里，允许使用哪些信息，输出要长什么样，遇到缺失或冲突怎么处理，最后如何判断输出合格。你不需要每次都写得很长，但这些边界必须在脑子里清楚。

直觉

prompt 不是咒语，是把模型应该读什么、做什么、写成什么样说清楚。

正式说法

prompt 是传给 LLM 的上下文输入，包含任务指令、示例、约束、外部材料和输出规范。

判断句

如果输出无法被稳定检查，prompt 还没写成工程输入，只是聊天请求。

学到这里要能做什么把一个模糊需求改写成包含任务、材料、格式和失败规则的提示。

Prompt 是输入契约，不是语气技巧

contract

很多提示词问题来自任务边界不清。比如“分析这段文本”没有说明分析什么：情绪、主题、实体、风险、行动项，还是写作问题？模型会按常见模式猜。猜对了像是 prompt 有效，猜错了就会出现跑题、过度解释和格式漂移。

输入契约的第一步是把任务写成可检查动作。不要只写“请优化”，要写“把下面文本改成 120 字以内的正式通知，保留时间、地点、报名方式，不新增事实”。这样模型知道要改写，不是总结；知道长度约束；知道哪些信息必须保留；知道不能脑补。

模糊请求

帮我处理下面内容。

问题是模型不知道处理目标。它可能总结、翻译、改写、提取关键词，输出都可能看似合理，但不一定符合你的任务。

可验收请求

从下面会议记录中提取待办事项，输出 JSON 数组。每项包含 owner、task、deadline、evidence。没有明确负责人填 null，不要猜。

这类写法让输出能被程序读，也能检查有没有猜测负责人。

学到这里要能做什么把“帮我弄一下”改成可检查动作，并写明不允许模型猜什么。

Prompt 的组成部分：每一块都有职责

prompt anatomy

一个 prompt 可以很短，但组件职责不能乱。角色说明用于设定输出角度，例如“你是代码审查助手”；任务说明告诉模型要做什么；上下文放材料；约束说明不能做什么；示例展示输入输出模式；输出格式让下游能解析；失败处理告诉模型遇到缺失、冲突、无关材料时怎么回答。

不是每个任务都需要所有部分。日常问答可能只要任务和上下文；工程任务通常至少要任务、输入边界和输出格式；高风险任务还需要引用、拒答规则和校验。关键是不要让不同部分互相打架。比如一边要求“只输出 JSON”，一边又要求“先解释你的思考过程”，格式就会冲突。

学到这里要能做什么看一段 prompt 时，能标出每句话属于角色、任务、上下文、约束、格式还是失败处理。

上下文：模型只能稳定使用它看见的材料

context

上下文是 prompt 里最容易变脏的部分。材料太少，模型没有依据；材料太多，模型可能被噪声干扰；材料边界不清，模型可能把用户指令和资料内容混在一起。写 prompt 时要把“指令”和“资料”分隔清楚，最好使用固定分隔符、标题或字段。

如果资料来自外部检索，还要告诉模型如何处理材料不足。比如“只能依据资料回答；资料没有提到时返回 not_found”。这句话不是装饰，它决定模型在缺证据时是承认不知道，还是按旧知识补全。对于制度、法律、财务、医学这类场景，缺证据时强行回答比答得短更危险。

示例：把指令和资料分开prompt_template.txt
任务：
从资料中提取客户投诉的核心原因，并按 JSON 输出。

规则：
- 只能使用资料里的信息。
- 如果资料没有出现明确原因，reason 填 null。
- 不要输出 JSON 以外的文字。

资料：
<<<
{customer_text}
>>>

输出 JSON：
{"reason": string | null, "evidence": string}
上下文问题常见后果修法
资料和指令没有分隔模型把资料里的话当新指令用固定分隔符和标题标出资料区
材料过长且杂输出抓错重点或漏字段先检索/筛选，再给模型
没有缺失规则模型用常识或旧知识补答案明确 null、not_found 或 insufficient_context
上下文顺序混乱重要材料被淹没把最相关材料放前，保留来源编号
学到这里要能做什么写 prompt 时把任务指令和输入材料分开，并指定资料不足时的输出。

示例：few-shot 不是越多越好

examples

示例会强烈影响模型输出。一个好的示例告诉模型字段怎么填、边界怎么判、遇到缺失怎么处理。坏示例也会把坏习惯传给模型：字段名不稳定、格式有多余解释、把没有依据的信息猜出来、把异常情况处理错。

few-shot 示例适合输出风格或判断边界比较难讲清的任务。比如情绪分类、投诉原因抽取、代码审查标签，单靠文字规则可能不够，给一两个正例和反例会更稳定。但示例会占上下文，也可能让模型过度模仿。示例数量不是越多越好，应该覆盖最关键的边界。

正例

展示标准输出，字段完整，格式干净。

反例

展示不要猜、不要输出额外文字、不要改字段名。

边界例

展示资料不足、内容冲突、多标签重叠时如何处理。

学到这里要能做什么判断一个任务是否需要示例，以及示例应该覆盖哪类边界。

结构化输出：让下游程序能读懂

structured output

结构化输出的目标不是让结果看起来整齐，而是让下游程序能稳定解析。JSON 是常见选择，因为字段、数组、对象、字符串、数字、布尔值都明确。写 JSON 要求时，字段名、类型、可空规则、枚举值、禁止额外文本都要写清楚。

如果你只说“输出 JSON”，模型仍然可能在 JSON 前后加解释，或者把字段写成别名。更稳的写法是给 schema、给示例、要求只输出一个 JSON 对象，并说明缺失值用 null。真正工程里还会在模型输出后做解析和校验，解析失败就触发修复或重试。

2
示例：结构化输出要求schema_prompt.txt
请只输出一个 JSON 对象，不要输出 Markdown，不要解释。

字段：
- sentiment: "positive" | "neutral" | "negative"
- reason: string，20 字以内
- needs_followup: boolean
- missing_info: string[]，没有则为空数组

如果文本证据不足，不要猜测，sentiment 填 "neutral"，并把缺失信息写入 missing_info。
学到这里要能做什么写出一个字段、类型、枚举、缺失规则都明确的 JSON 输出要求。

校验与修复：不要相信一次输出永远合格

validation

结构化输出必须配校验。模型可能输出非法 JSON、漏字段、字段类型错、枚举值越界，或者把解释文字塞进结果。只靠 prompt 不能保证每次都合格。程序需要解析 JSON，检查必填字段、类型、枚举和业务规则。

修复策略要简单明确。第一步可以把错误信息反馈给模型，让它只修格式，不重新做任务。比如“你的输出不是合法 JSON，错误是缺少逗号。请只返回修复后的 JSON”。如果修复仍失败，就重试或返回人工处理。高风险任务不要无限重试，避免成本失控和错误悄悄进入下游。

伪代码：解析失败后修复validate_and_repair.py
def parse_or_repair(raw_output):
    try:
        data = json.loads(raw_output)
        validate_schema(data)
        return data
    except Exception as err:
        repair_prompt = f"""下面输出无法解析为合法 JSON。
错误：{err}
请只返回修复后的 JSON，不要重新解释任务。

原始输出：
{raw_output}
"""
        fixed = llm.generate(repair_prompt)
        data = json.loads(fixed)
        validate_schema(data)
        return data
语法校验能否被 JSON parser 解析。
字段校验必填字段是否存在。
类型校验字符串、数字、数组是否正确。
业务校验枚举、范围、引用是否合规。
学到这里要能做什么说明为什么结构化输出必须配解析和 schema 校验，而不是只靠一句“请输出 JSON”。

Prompt 迭代流程：按失败样例改，不按感觉改

iteration

prompt 迭代应该围绕失败样例。先准备一组代表性输入，包括正常样例、边界样例、缺失样例、冲突样例和恶意/干扰样例。每次改 prompt 后，用同一组样例跑一遍，记录哪些变好了，哪些变坏了。这样你才知道修改是否真的有效。

不要每次失败都随手加一句新规则。规则会累积，最后互相冲突。更好的做法是先判断失败类型：任务理解错、上下文不够、格式不稳、字段规则不清、示例误导、校验缺失。然后只改对应位置。prompt 也需要版本管理，至少保留任务、模板、测试样例和已知失败。

1. 固定样例准备正常、边界、缺失和冲突输入。
2. 跑输出记录模型输出和解析结果。
3. 分类失败定位是任务、上下文、格式还是校验问题。
4. 小步修改改一个点，再用同一组样例回归。
学到这里要能做什么用固定测试样例迭代 prompt，而不是凭单次输出好坏随手加规则。

常见误区：别把 prompt 当玄学补丁

mistakes

第一个误区：以为 prompt 越长越好。长 prompt 如果没有结构，只会增加噪声。第二个误区：把格式要求写得含糊，例如“输出清晰一点”。这无法让程序解析。第三个误区：示例和规则冲突。模型往往会模仿示例，如果示例里有多余解释，最后输出也容易带解释。

第四个误区：不处理缺失信息。模型遇到资料不足时会补全，除非你明确告诉它不要猜，并给出缺失时的输出。第五个误区：没有校验。结构化输出不是写完 prompt 就结束，解析和 schema 校验是下游使用前的门槛。

错误示例：要求 JSON 但又要求解释conflicting_prompt.txt
请分析下面文本，输出 JSON。
你要先详细解释你的判断过程，再给出 JSON。
请不要遗漏任何信息，也不要输出太长。
输出格式随你，只要清楚就行。

错因

同时要求 JSON、解释过程、随意格式，约束互相冲突。

后果

模型可能输出一段解释加半截 JSON，下游 parser 直接失败。

修法

只要求一个 JSON 对象，把字段、类型和缺失规则写清楚。

学到这里要能做什么看到冲突 prompt 时，能指出哪些要求会破坏输出格式。
