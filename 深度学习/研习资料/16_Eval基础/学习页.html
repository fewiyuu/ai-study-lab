LLM Application Engineering 02

Eval 基础

LLM 应用最常见的坏习惯，是只拿几个顺手的问题试一试，然后说“感觉还行”。这种判断很危险。模型输出具有随机性，问题分布会变，prompt 一改可能让某些样例变好、另一些样例变差。没有评测集和记录，你只是在看热闹，不是在验收系统。

Eval 的目标是把主观感觉变成可比较证据。你需要一组代表真实任务的样例，明确每个样例期待什么，选择合适指标，记录模型输出，区分自动指标、人工评审和模型评审的边界。更重要的是保留失败案例。失败案例告诉你下一步该改 prompt、改检索、换模型，还是调整产品边界。

学完你应该能做什么

  1. 解释 eval、benchmark、test set、gold answer、rubric、regression test 的区别。
  2. 为一个 LLM 应用设计最小评测集，覆盖正常、边界、缺失、对抗和高风险样例。
  3. 区分准确率、格式通过率、忠实性、引用正确率、人工评分、LLM-as-judge 各自回答什么问题。
  4. 诊断评测结果：平均分变高但关键失败增加、judge 偏置、样例泄漏、指标和业务目标不一致。
  5. 建立最小回归测试流程，避免每次改 prompt 都靠感觉验收。

Eval 是一套证据链

mental model

Eval 不是单个分数。它是一条证据链：先定义任务，收集样例，写清期望输出或评分标准，运行系统，记录输出，计算指标，分析失败，最后决定改哪里。少了任何一环，分数都可能误导你。

LLM 应用里的 eval 更像产品验收和软件测试的结合。它不只看答案是否“像样”，还看格式能否解析、有没有引用依据、是否遵守拒答规则、延迟和成本是否可接受。你需要知道每个指标回答的问题，也要知道它回答不了什么。

直觉

Eval 是让你少靠感觉，多靠可复现样例判断系统有没有变好。

正式说法

Eval 是针对任务定义、样例集、评分标准、运行结果和误差分析的评估流程。

判断句

如果一个改动不能在同一批样例上对比，就很难说它真的改进了系统。

为什么不能只靠人工随便试

why eval

人工随便试的问题在于样例会偏。你通常会挑自己刚想到的问题，或者挑已经知道模型容易答对的问题。这样看不到长尾、边界、格式错误和高风险失败。更麻烦的是，下一次你改 prompt 时，可能换了一批问题试，结果根本不能比较。

正式 eval 不排斥人工判断。人工判断很重要，尤其是开放式回答、写作质量和复杂推理。但人工判断也需要样例集和 rubric。rubric 是评分规则，告诉评审什么算好、什么算坏。没有 rubric，两个评审可能按不同标准打分，今天的你和明天的你也可能标准不一样。

随手试快，但不可复现，也容易挑简单样例。
固定样例能比较改动前后，适合回归测试。
人工 rubric适合开放任务，但要统一评分标准。
自动指标快而便宜，但只回答它能度量的问题。
学到这里要能做什么说明为什么“我试了几个问题还行”不能替代评测。

评测集：样例比口号重要

eval set

评测集是一组固定样例。每个样例应该包含输入、参考答案或评分规则、必要元数据。比如客服分类任务可以保存客户原文、正确标签、证据句、难度、来源场景。RAG 问答可以保存问题、应该命中的文档、参考答案、允许缺失的情况。

评测集不需要一开始很大。早期可以从 20 到 50 条高质量样例开始，但要覆盖真实失败类型：正常样例、边界样例、缺失信息、冲突信息、格式干扰、恶意输入、高风险业务样例。小而准的评测集比一堆随手凑的样例有用。

示例：一个最小 eval caseeval_case.json
{
  "id": "refund_014",
  "input": "客户说超过承诺时间 3 天仍未收到货，要求退款。",
  "expected": {
    "category": "物流延迟",
    "needs_followup": true
  },
  "rubric": "必须识别物流延迟；不能把它归为商品质量。",
  "tags": ["客服", "退款", "边界样例"]
}
样例类型检查什么例子
正常样例主流程是否可用清楚表达的分类、抽取、问答
边界样例规则边界是否稳多标签、模糊表达、近似类别
缺失样例是否承认信息不足资料没有答案、字段缺失
对抗样例是否被干扰指令带跑输入中夹带“忽略规则”
高风险样例重要场景是否安全法律、医疗、财务、权限相关
学到这里要能做什么为一个任务列出 20 条评测样例应该覆盖的类型，而不是只收集顺手问题。

指标选择：先问你要证明什么

metrics

不同指标回答不同问题。分类任务可以看 accuracy、precision、recall。结构化输出可以看 JSON 解析通过率、字段完整率、枚举合法率。RAG 可以看召回命中、引用正确率、忠实性。开放问答常需要人工评分或 LLM-as-judge，但评分规则必须清楚。

指标最容易犯的错，是拿一个数字回答所有问题。比如格式通过率很高，不代表答案事实正确；人工评分高，不代表 JSON 能被程序解析；平均分上升，不代表高风险样例没退化。正确做法是按任务拆指标，至少区分任务正确性、格式合规、证据支持、用户体验和成本延迟。

accuracy=正确样例数总样例数
学到这里要能做什么看到一个任务，先说它需要哪几类指标，而不是直接问“准确率多少”。

人工评审与 LLM-as-judge：能用，但要知道边界

judge

开放式输出很难完全靠程序判断。比如答案是否有帮助、解释是否清楚、语气是否合适，往往需要人工评审。人工评审的优点是可靠地理解业务语境，缺点是慢、贵、标准可能漂移。rubric 可以减少漂移：每个分数档位要写清楚，最好配正反例。

LLM-as-judge 是让另一个模型按 rubric 给输出打分。它便宜、快、适合大批量初筛，但会有偏置：可能偏好更长答案，可能被表面流畅度骗到，可能和被评模型有相似偏差。使用时要抽样人工复核，并把 judge prompt、模型版本、温度、评分规则固定下来。

适合用模型评审的场景

大规模初筛、格式和引用检查、按明确 rubric 判断是否覆盖要点。模型评审适合辅助，不适合在高风险场景里单独拍板。

要警惕的偏差

偏好长答案、偏好自信语气、忽略细小事实错误、受评分顺序影响、被提示词注入干扰。

学到这里要能做什么说明 LLM-as-judge 为什么有用，也能指出它为什么不能无监督地当真理。

回归测试：防止改好一个点，弄坏另一片

regression

回归测试的意思是：每次改 prompt、模型、检索策略或后处理，都用同一套样例重新跑一遍，比较新旧结果。LLM 应用很容易出现局部变好、整体变差。比如你加了一句“回答要简短”，格式可能更干净，但复杂问题的必要解释被删掉了。

回归测试不一定要全自动开始。最小版本可以是一个表格：样例 id、输入、旧输出、新输出、关键指标、是否通过、失败原因。随着任务成熟，再把运行和打分脚本化。真正重要的是固定样例和固定记录。

伪代码：最小回归评测循环run_eval.py
results = []
for case in eval_cases:
    output = app.run(case["input"])
    score = score_case(output, case["expected"], case["rubric"])
    results.append({
        "id": case["id"],
        "score": score,
        "output": output,
        "tags": case["tags"],
    })

report = summarize_by_tag(results)
save_report(report)
学到这里要能做什么描述一次 prompt 改动后，怎样用固定样例判断有没有退化。

误差分析：分数只是入口，失败类型才指导修改

error analysis

平均分告诉你系统大概表现，但不能告诉你下一步改哪里。误差分析要把失败样例分类：输入理解错、检索没命中、上下文里有但没用、格式非法、拒答不当、引用错误、幻觉、语气不合适、延迟过高。每类失败对应的修法不同。

如果不做误差分析，团队很容易只做表面优化。比如答案幻觉，你去调格式；引用错，你去换更大模型；召回漏，你去改 prompt。这些改动可能有一点效果，但没有抓住主要原因。好的 eval 报告应该列出 Top 失败类型和代表样例。

最小 Eval 流程：先小而固定，再逐步自动化

workflow

初学阶段不用一上来搭复杂平台。最小 eval 可以从一个 JSONL 或表格开始：20 到 50 条样例，每条有输入、期望、标签和评分规则。然后写一个脚本跑应用，把输出保存下来，再用自动规则或人工 rubric 打分。每次改动都保存报告。

当样例变多、任务变复杂，再考虑自动化仪表盘、在线抽样、A/B 测试和持续监控。不要反过来，一开始就追工具平台，却没有高质量样例。Eval 的核心资产是样例和失败分析，不是工具界面。

1. 定义任务明确要验收的能力和边界。
2. 建样例集覆盖正常、边界、缺失和风险输入。
3. 跑固定版本记录 prompt、模型、参数和输出。
4. 分析失败按标签汇总，决定下一步改哪里。
学到这里要能做什么用表格或 JSONL 搭出一个最小可复现评测流程。

常见误区：漂亮分数不等于真实可用

mistakes

第一个误区：样例太少还全是简单题。这样分数高没有意义。第二个误区：把测试集拿来反复调 prompt，最后 prompt 只适配这批样例，遇到新输入又坏。第三个误区：只看平均分，不看高风险样例和失败类型。平均分提高，关键样例退化，产品仍然不能上线。

第四个误区：把 LLM-as-judge 当成绝对真相。judge 也会错，也会被提示词影响。第五个误区：指标和业务目标不一致。比如客服系统只追求回答完整，结果延迟太高、转人工判断错误，用户体验还是差。

错误示例：只用三条样例决定上线bad_eval.txt
我问了 3 个问题：
1. 退款政策是什么？
2. 怎么改地址？
3. 联系客服在哪里？

模型都答得挺好，所以可以上线。

错因

样例少、分布窄、没有边界和风险输入，也没有明确评分标准。

后果

上线后遇到真实用户输入，格式、权限、旧制度、缺失资料都可能出问题。

修法

构建固定评测集，覆盖标签，记录失败类型，再做回归测试。

学到这里要能做什么看到“试了几个问题效果不错”时，能指出证据不够在哪里。

综合练习

practice
练习前快速复盘
  • Eval 是任务、样例、标准、运行、打分和误差分析的证据链。
  • 评测集要覆盖正常、边界、缺失、对抗和高风险样例。
  • 不同指标回答不同问题，不能用一个平均分代表全部质量。
  • 回归测试用同一批样例比较改动前后，防止局部变好整体退化。

诊断题按“证据缺什么、会误导什么、怎么补”回答。设计题要具体到样例字段、指标和失败类型。
