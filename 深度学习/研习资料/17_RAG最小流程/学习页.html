LLM Application Engineering 03

RAG 最小流程

RAG 的全称是 Retrieval-Augmented Generation，通常译为检索增强生成。它解决的问题很直接：模型本身不一定知道你的私有资料、最新资料和公司内部规则，所以先从资料库里找相关内容，再把这些内容交给模型生成答案。

但 RAG 不是“搜索一下再问模型”这么简单。一个最小可用 RAG 至少包含资料处理、切块、向量化、索引、查询改写或向量检索、重排、上下文组装、生成、引用和评估。用户看到的只是最后一句回答，真正决定质量的是前面每一步有没有把正确证据送到模型面前。

学完你应该能做什么

  1. 解释 RAG 与普通 prompt、微调、搜索问答的边界。
  2. 说清文档如何变成 chunk、embedding、向量索引和可检索候选。
  3. 区分召回、重排、上下文组装、生成、引用各自负责什么。
  4. 诊断常见失败：没召回、召回但排后面、上下文污染、模型编造、引用不对应。
  5. 写出一个最小 RAG 伪代码流程，并设计最小评估指标。

RAG 的核心问题：答案必须有证据

mental model

普通聊天模型直接根据参数里的知识生成答案。它可能答得很流畅，但不一定知道你的资料，也不一定知道自己不知道。RAG 给它加了一步：先从外部资料库找证据，再要求模型基于证据回答。这样做不是为了让模型更像搜索引擎，而是为了让答案可以被检查。

你可以把 RAG 的质量拆成两个大问题。第一，资料有没有被找回来，这叫检索问题。第二，模型有没有按资料回答，这叫生成和忠实性问题。很多人一看到答案错，就直接改 prompt 或换模型，其实可能真正坏在检索：正确文档根本没有进上下文。

直觉

RAG 是让模型答题前先看资料，而不是指望它把所有事实都背在参数里。

正式说法

RAG 是把检索系统返回的外部上下文注入生成模型，使输出受资料约束的应用架构。

判断句

如果答案无法指出依据，或者依据和答案不一致，这个 RAG 系统就还不能放心用。

学到这里要能做什么看到一个 RAG 回答错误时，先问“正确证据有没有进上下文”，而不是直接责怪模型不会答。

RAG、微调、长上下文分别解决什么

boundary

RAG 适合知识经常变化、资料来源明确、答案需要引用依据的任务。比如公司制度问答、产品文档助手、论文资料问答、客服知识库。它的优势是资料可以独立更新，不需要每次改知识都重新训练模型，而且可以把证据显示给用户。

微调更适合改变模型的行为模式、格式习惯、领域表达和任务套路。它不适合当作“把大量事实塞进模型”的默认方案，因为事实会过期，也很难追溯来源。长上下文可以一次塞很多资料，但它不等于 RAG。上下文越长，成本越高，噪声越多，模型也未必能稳定抓到关键片段。

RAG

外部资料可更新、可引用、可替换。重点是检索质量、上下文质量和忠实回答。

微调

改变模型习惯和任务能力。重点是样本质量、训练目标和泛化，不负责实时知识更新。

长上下文

适合临时放入一批资料，但成本高，噪声多，仍然需要组织和筛选。

场景优先方案原因
公司制度每周更新RAG知识变化快，需要引用最新制度。
希望模型稳定输出公司固定话术微调或少样本模板重点是表达风格和格式习惯，不是查资料。
临时总结一份 20 页报告长上下文资料一次性、范围小，可以直接放入上下文。
法规问答要给出处RAG + 严格引用答案必须能回到原文证据。
学到这里要能做什么为一个业务需求判断该用 RAG、微调、长上下文，还是它们的组合。

完整链路：用户问题到最终答案

pipeline

一个最小 RAG 流程可以拆成离线和在线两部分。离线部分负责把资料处理成可检索的索引：清洗文档、切成 chunk、计算 embedding、写入向量库和元数据。在线部分从用户问题开始：把问题变成查询向量，取回候选 chunk，必要时重排，把选中的证据拼进 prompt，最后让模型回答。

这条链路的关键是每一步都要留下可检查的中间结果。只看最终答案，你不知道失败原因。把 top-k 检索结果、重排分数、最终上下文、模型回答和引用都记录下来，才能判断是资料库缺文档、切块太碎、检索没召回、重排丢证据，还是生成阶段没有遵守证据。

输入要查用户问题是否清楚，是否需要改写或拆分。
候选要留保存 top-k 文档，否则无法知道检索是否失败。
上下文要控不要把互相冲突、过长、无关的片段全塞进去。
引用要验答案里的关键事实必须能回到 chunk 原文。
学到这里要能做什么画出 RAG 的离线索引链路和在线问答链路，并说清每一步的输入输出。

切块与索引：资料不是直接扔进向量库

chunking

文档通常不能整篇直接向量化。整篇太长，embedding 会把多个主题混在一起；切得太碎，又会丢掉上下文。chunk 是检索的基本单位。一个 chunk 应该尽量表达一个完整语义片段，同时保留标题、来源、页码、段落位置、更新时间等元数据。

切块策略会直接影响召回。按固定字数切很简单，但可能把一个规则切断。按标题、段落、表格、代码块切更稳，但实现更复杂。很多系统会设置 overlap，也就是相邻 chunk 之间保留一点重叠，减少关键句被切断的概率。overlap 太大又会造成重复和噪声。

学到这里要能做什么解释为什么 chunk 太大、太小、overlap 太多都会影响 RAG 质量。

检索：先把可能有用的证据找回来

retrieval

检索阶段的目标不是直接给最终答案，而是把可能包含答案的证据找回来。向量检索会把用户问题和 chunk 都表示成 embedding，然后按相似度找最近的 chunk。相似度常用 cosine similarity，也就是看两个向量方向接近程度。方向越接近，语义越可能相关。

cosine_similarity(q, d)=q · d||q|| ||d||

但向量相似不等于一定正确。用户问“退款多久到账”，检索可能找回“退款申请条件”而不是“到账时间”。这时可以结合关键词检索、元数据过滤、query rewrite 或混合检索。元数据过滤很重要，比如只查某个产品线、某个版本、某个时间之后的文档。

最小代码：向量检索的伪代码retrieval.py
def retrieve(query, vector_index, top_k=5):
    query_vec = embed(query)
    candidates = vector_index.search(query_vec, top_k=top_k)
    return [
        {
            "chunk_id": item.id,
            "text": item.text,
            "score": item.score,
            "source": item.metadata["source"]
        }
        for item in candidates
    ]
学到这里要能做什么看到检索结果时，判断它是没找到正确文档，还是找到了但分数和排序不可靠。

重排与上下文：找回来还不等于能用

rerank & context

初次检索常常追求“先别漏掉”，所以 top-k 可能比较宽。重排的目标是从候选里挑出真正能回答问题的片段，把它们排到前面。向量检索看整体语义相似，reranker 往往更细地比较“问题”和“文档片段”是否匹配。

上下文组装是另一个容易被忽略的环节。你不能把所有候选不加筛选地塞给模型。过多无关片段会稀释重点，互相冲突的片段会诱导模型编造折中答案，缺少来源信息会导致引用无法检查。一个可维护的上下文通常包含清晰编号、来源、标题、片段内容，并要求模型只基于这些片段回答。

重排前

  1. 退款申请条件
  2. 退款到账时间
  3. 售后争议处理
  4. 订单取消规则

向量相似可能把“退款相关”都找回来，但真正回答“多久到账”的是第 2 条。

重排后

  1. 退款到账时间
  2. 退款申请条件
  3. 订单取消规则

reranker 更关注片段能否直接回答问题，然后再进入上下文。

上下文模板：让证据可引用prompt.txt
你是客服知识库助手。只能根据【资料片段】回答。
如果资料不足，直接说“根据现有资料无法确认”。
回答中的关键事实必须标注片段编号。

【用户问题】
退款一般多久到账？

【资料片段】

[2] 来源：退款条件说明

学到这里要能做什么解释为什么 RAG 的上下文不能只是“把 top-k 文本拼起来”。

生成与引用：答案要忠实于证据

generation

生成阶段的核心要求不是“写得像人”，而是忠实。忠实的意思是：答案里的关键事实都能被上下文支持，不能把资料里没有的信息补出来，也不能把多个片段的内容硬凑成资料没有表达过的结论。RAG 系统最危险的错，不是答得难看，而是答得流畅但没有依据。

引用不是装饰。引用可以让用户和开发者检查答案从哪里来。一个引用合格的回答，应该能把每个关键断言对应到具体片段。如果答案说“退款 1-3 个工作日到账 [1]”，片段 [1] 就应该真的写了这个事实。引用不对应，比没有引用更糟，因为它制造了假的可信度。

好回答

“审核通过后通常 1-3 个工作日原路退回 [1]。如果还没审核通过，资料没有说明到账时间。”

不完整

“一般很快到账。”没有具体依据，也没有说明资料边界。

危险

“24 小时内必定到账。”如果片段没有这个承诺，就是编造。

检查项应该怎么看常见问题
忠实性关键事实是否被上下文支持模型补充资料里没有的承诺。
引用正确率引用片段是否真的支持对应句子引用了相关但不支持结论的片段。
拒答边界资料不足时是否承认不知道没有证据时仍然猜答案。
格式稳定性回答是否符合产品需要的格式引用格式、JSON 字段或免责声明不稳定。
学到这里要能做什么逐句检查一个 RAG 回答，判断哪些句子有证据，哪些句子是模型自己补的。

评估与排错：不要只看最终答案

debug

RAG 的评估要分层。只看最终回答准确率，会把所有错误混在一起。正确做法是把链路拆开看：正确文档是否在 top-k 中，正确片段是否被 reranker 排到前面，最终上下文是否包含足够证据，答案是否忠实，引用是否对应。

这就是为什么 RAG 排错一定要保存中间结果。你需要看到 query、top-k 候选、重排结果、最终上下文、回答和引用。只有这样，你才能把失败归因到某一环，而不是凭感觉改 prompt。RAG 很多问题不是 prompt 能救的：资料没进库、切块切坏、索引没更新、过滤条件错了，改生成模型也没用。

Recall@k正确证据是否在前 k 个候选中。
MRR正确证据排得越靠前越好。
Faithfulness答案是否只说资料支持的内容。
Citation引用是否真的支撑对应断言。

最小实现：先做可检查版本

minimal build

学习 RAG 不要一开始就追复杂框架。先做一个可检查的最小版本：准备少量文档，切成 chunk，给每个 chunk 存 id、source、text、embedding。用户提问时检索 top-k，打印候选，手动确认正确证据是否出现，再把候选放进 prompt 让模型回答。

最小版本的重点不是性能，而是链路透明。你要能看到每一步的输入输出。等你能稳定判断失败原因，再考虑向量库、混合检索、reranker、缓存、权限过滤、多轮记忆和生产监控。否则框架越复杂，你越不知道错在哪里。

最小流程：可检查的 RAG 伪代码rag_minimal.py
docs = load_documents("knowledge_base/")
chunks = split_into_chunks(docs, chunk_size=300, overlap=60)
index = build_vector_index(chunks)

def answer_question(question):
    candidates = retrieve(question, index, top_k=8)
    ranked = rerank(question, candidates)[:4]
    context = format_context(ranked)
    prompt = build_prompt(question, context)
    answer = llm_generate(prompt)
    return {
        "question": question,
        "candidates": candidates,
        "context": context,
        "answer": answer
    }

常见误区：RAG 坏了通常不是一个原因

pitfalls

只改 prompt，不看检索

如果正确证据没有进上下文，prompt 再漂亮也只能让模型更体面地猜。先查 top-k，再谈生成。

chunk 越大越安心

大 chunk 可能混入多个主题，相似度变钝，还会浪费上下文窗口。应该根据文档结构调切块。

top-k 越多越好

候选越多，噪声越多。召回和精度要平衡，必要时靠 rerank 和过滤。

有引用就可信

引用必须支撑对应句子。引用错误会制造虚假的可信度，比没有引用更危险。

错法后果修法
资料入库后不记录版本答案引用旧制度，难以追溯保存 source、version、updated_at。
忽略权限过滤用户可能看到不该看的资料检索前或检索后做权限约束。
没有拒答规则资料不足时模型也会猜prompt 明确资料不足时的输出。
只评估最终回答无法定位失败环节分层记录 recall、rerank、context、faithfulness。
不保存失败样例每次改动都像重新摸索建立固定 eval cases 和失败标签。
学到这里要能做什么看到 RAG 方案时，主动检查切块、召回、重排、上下文、引用、权限和评估。

提示：上下文断裂，缺标题、条件、例外规则。
解释

8. 为什么 chunk 需要保存 source、版本、更新时间、权限等元数据？

提示：追溯、过滤、引用、权限控制。
辨析

9. top-k 越大是否一定越好？为什么？

提示：召回增加，但噪声、成本、上下文污染也增加。
短答

10. reranker 在 RAG 中解决什么问题？

提示：从候选中重新判断哪个片段更能回答问题。
设计

11. 一个清晰的 RAG 上下文应该包含哪些信息？

提示：编号、来源、标题、正文、必要元数据、回答约束。

三、错误诊断与评估

这组题重点练“错了先看哪”。

快速复习RAG 评估至少拆成证据命中、排序、忠实性、引用正确和拒答边界。
错误诊断

12. 用户问公司报销上限，答案错了；检查发现正确制度文档完全没出现在 top-k。优先排查什么？

按“资料是否入库 / 切块 / embedding / 检索 / metadata filter”回答。
错误诊断

13. 正确片段已经在最终上下文中，但模型仍然编出资料没有的承诺。优先怎么改？

提示：生成 prompt、拒答规则、忠实性约束、温度、逐句引用。
错误诊断

14. 答案结论是对的，但引用片段并不支持这句话，这有什么风险？怎么修？

提示：假的可信度；需要引用对齐和逐句证据检查。
指标

15. Recall@k 在 RAG 评估里回答什么问题？

提示：正确证据是否出现在前 k 个候选中。
指标

16. 为什么 RAG 不能只看最终答案准确率？

提示：最终分数无法区分检索、重排、上下文、生成和引用问题。
场景

17. 内部知识库 RAG 忘了做权限过滤，可能造成什么后果？应该在哪些环节控制？

提示：检索前过滤、检索后过滤、上下文组装前再次检查。

四、工程验收与表达

最后把本节内容串起来，能用于讲解和面试追问。

快速复习一个靠谱的最小 RAG 必须可观察、可引用、可评估、可追溯。
设计

18. 最小 RAG 返回结果里为什么要保存 candidates、context、answer？

提示：为了排错和复现，不只给用户最终答案。
清单

19. 给出 8 条验收一个 RAG 原型的检查点。

覆盖资料、切块、索引、检索、重排、上下文、引用、评估、权限。
面试

20. 面试官问：RAG 答案错了，你会怎么排查？

提示：从 top-k 到最终上下文，再到生成和引用，按链路排查。
场景

21. 如果要给学校课程 PDF 做问答助手，你会如何设计资料处理和评估？

提示：按章节切块、页码来源、题目集、引用页码、资料不足拒答。
费曼解释

22. 用 8-10 句话解释：一个 RAG 系统如何从资料库回答用户问题，以及错了怎么定位。

必须串联：文档、chunk、embedding、索引、检索、重排、上下文、生成、引用、评估。
