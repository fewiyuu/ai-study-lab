Agent 不应该被理解成“模型自由发挥”。更稳的理解是：模型在一个受约束的循环里选择工具、读取结果、更新状态，并在明确条件下停止。
LLM Application Engineering 04

Agent 最小流程

Agent 这个词很容易被讲得很玄：好像模型有了目标，就会自己计划、自己搜索、自己调用工具、自己完成复杂任务。工程里不能这么理解。更可靠的说法是：Agent 是一个受控循环，模型每一步只做一件事：根据当前目标和状态，选择下一步动作，动作可能是调用工具，也可能是给出最终答案。

本节不追复杂框架，只讲最小闭环。你要能回答这些问题：什么时候普通 workflow 就够了，什么时候才需要 Agent；工具要怎么描述才不会被乱用；状态里应该保存什么；循环什么时候停；工具失败、权限不足、结果冲突时怎么办。掌握这些，比背一堆术语重要得多。

学完你应该能做什么

  1. 区分 workflow、tool use、Agent 三者的边界。
  2. 画出 Agent 的最小循环：目标、状态、选择动作、执行工具、观察结果、更新状态、停止。
  3. 为一个工具写出名称、用途、输入参数、输出格式、失败条件和安全限制。
  4. 诊断常见失败：无限循环、工具误用、状态污染、权限越界、结果未验证。
  5. 设计一个最小 Agent 原型的验收指标和日志字段。

Agent 的核心：把下一步动作交给模型选择

mental model

普通 prompt 是一次输入、一次输出。工具调用是模型可以请求外部函数，比如搜索、查库、写文件、发请求。Agent 再往前一步：它不只是调用一次工具，而是在多步循环中决定下一步该做什么。每一步都依赖前一步观察到的结果。

这听起来很强，但也意味着风险增加。模型可能选错工具，可能重复做无用动作，可能相信错误结果，可能忘了原始目标。一个 Agent 系统的质量，不只取决于模型聪不聪明，更取决于你有没有把目标、工具、状态、预算、停止条件和安全边界写清楚。

目标要完成什么
状态已知信息

工具外部能力
观察返回结果
更新记录进展
停止完成或放弃

直觉

Agent 是让模型在受控范围内决定“下一步做什么”，不是让模型随便干活。

正式说法

Agent 是围绕目标、状态、工具和反馈构建的多步决策执行系统。

判断句

如果任务步骤固定，普通 workflow 往往更稳；只有步骤需要根据中间结果变化时，才更像 Agent。

学到这里要能做什么看到一个“Agent 需求”时，先判断它是不是真的需要模型做多步决策。

先判断：Workflow 够不够

boundary

很多任务不需要 Agent。比如“读取网页、总结、生成表格”这种步骤固定的任务，用 workflow 更容易控制：第一步抓取，第二步清洗，第三步总结，第四步写表格。模型可以参与每一步，但流程不需要它自由选择。

Agent 更适合步骤会随结果变化的任务。比如排查一个报错：先看日志，如果是权限问题就查配置，如果是网络问题就测连通性，如果依赖缺失就检查环境。下一步取决于上一步观察。这时让模型根据状态选择工具有价值，但仍然需要限制工具范围、最大步数和停止条件。

任务特征优先形态原因
步骤固定、输入稳定Workflow流程可预测，没必要引入多步决策。
只差一个外部信息单次工具调用查一次库或调一次接口即可。
下一步依赖中间结果Agent需要根据观察结果选择后续动作。
涉及付款、删除、发送、授权人工确认高影响操作不能只靠模型自动执行。
学到这里要能做什么为一个自动化需求选择 workflow、单次工具调用、Agent 或人工确认。

最小循环：选择动作，再观察结果

agent loop

最小 Agent 循环可以写成一句话：在最大步数内，模型根据目标和状态选择一个动作；系统执行动作；把执行结果作为观察写回状态；如果任务完成或风险过高，就停止。这个循环里，模型负责“选择”，系统负责“执行”和“约束”。这点很关键：不要让模型直接越过系统边界。

动作不一定都是工具调用。动作可以是“继续检索”“读取文件”“调用计算器”“请求用户确认”“给出最终答案”。为了可控，每个动作都应该有结构化格式。系统只接受白名单动作，参数要校验，工具返回也要结构化记录。否则日志很难复盘，错误也很难定位。

next_action=policy(goal, state, tools, constraints)

工具设计：描述越含糊，误用越多

tools

工具是 Agent 能影响外部世界的入口。工具描述必须清楚：这个工具做什么、不做什么、什么时候用、输入参数是什么、输出是什么、失败会怎样。模型并不真的理解你系统里的所有业务边界，它依赖工具描述和调用约束来判断能不能用。

工具不要设计得过大。一个“do_everything”工具会让模型很难判断参数，也让系统很难审计。更好的做法是把工具拆成清晰的小动作：搜索文档、读取详情、计算金额、创建草稿、请求确认。高影响工具要分成两步：先生成草稿或计划，再由用户或规则确认后执行。

工具定义：把边界写清楚tool_schema.json

名称清楚工具名应该表达动作和对象，不要过宽。
参数可验输入字段要能校验类型、范围和必填项。
输出稳定返回结构要方便模型继续推理，也方便系统记录。
权限明确读、写、删除、发送、支付要分清风险等级。
学到这里要能做什么给一个工具写出用途、输入、输出、失败条件和安全限制。

状态与记忆：Agent 不能只靠聊天历史

state

状态是 Agent 当前知道什么、做过什么、还缺什么。它不等于聊天历史。聊天历史很长、很乱，里面有用户表达、模型解释、工具结果、无关寒暄。工程上更可靠的做法是维护结构化状态：目标、约束、已完成步骤、工具观察、待确认事项、风险标记、最终证据。

状态污染是常见问题。比如工具返回了错误数据，模型把它当真；用户中途改变目标，但旧目标还留在状态里；一次失败的工具调用被后续步骤误认为成功。状态需要更新规则：哪些字段能覆盖，哪些字段只追加，哪些字段需要人工确认，哪些观察要标记可信度。

不够稳的状态

“用户想报销，查了一些制度，应该可以。”

太模糊。看不出查了什么、依据是什么、还有什么风险。

更可检查的状态

  • 目标：判断差旅打车能否报销。
  • 已查：差旅制度 v4.1 第 3.2 节。
  • 证据：市内交通可报销，需发票。
  • 缺口：未确认用户所在部门是否有额外规则。
  • 下一步：询问部门或按默认规则回答。
状态字段作用常见错误
goal当前任务目标用户改目标后仍沿用旧目标。
constraints权限、预算、格式、时间限制工具执行时忘记安全限制。
observations工具返回和外部证据未区分成功、失败和不确定结果。
open_questions还缺什么信息资料不足时继续猜。
final_evidence最终答案依据回答无法追溯到具体观察。
学到这里要能做什么把一段长对话压成结构化状态，保留目标、证据、缺口和下一步。

停止与控制：能停下来才算可用

control

Agent 最危险的问题之一是停不下来。它可能反复搜索、反复修正、反复调用同一个工具，最后浪费成本还没有结果。停止条件必须提前设计：任务完成、达到最大步数、连续无新信息、工具失败次数过多、风险操作需要确认、结果置信不足需要交给人。

控制不只是限制步数。你还要限制工具权限、参数范围、成本预算、超时时间和输出格式。每次工具调用前，系统都应该检查：这个动作在白名单内吗，参数合法吗，是否需要确认，是否超过预算。模型可以提出动作，但系统必须有最后的执行判断。

失败处理：工具结果也可能错

error handling

Agent 的失败不只来自模型。工具可能超时，接口可能返回空结果，权限可能不足，搜索结果可能不相关，外部系统可能返回旧数据。模型如果把所有工具结果都当成真，就会把错误继续放大。工具返回必须带状态：成功、空结果、权限拒绝、超时、参数错误、部分成功。

失败处理要写进循环规则。工具失败时，Agent 可以换查询、换工具、请求用户补充信息，或者停止并说明原因。不能把失败包装成正常答案。比如查不到制度，就应该说“没有找到可引用制度”，而不是凭常识给报销建议。

无限重试

搜索不到就换几个近似词一直搜，成本上升但没有新证据。

失败伪装成功

接口返回空列表，模型仍然说“根据资料可以确认”。这是严重边界错误。

正确处理

记录失败原因，尝试一次合理替代动作，仍失败就停止并说明缺口。

工具返回Agent 应该做什么不能做什么
empty换查询或说明没找到证据假装找到了资料。
permission_denied请求授权或停止绕过权限查别的入口。
timeout有限重试或降级无限等待和反复调用。
conflict标记冲突并请求确认擅自合并成一个看似确定的结论。
学到这里要能做什么看到工具调用失败时，设计停止、降级、重试和请求确认的处理方式。

验收指标：看完成率，也看失控率

evaluation

Agent 不能只看“最后有没有完成任务”。它还要看过程是否可控。一个任务完成了，但调用了不该调用的工具、绕过确认、用了过多步骤、引用了错误观察，这样的系统仍然不合格。验收要同时看任务成功率、工具选择正确率、平均步数、人工确认触发、失败升级、成本和日志可复盘性。

最小验收集应该包含正常任务、需要补充信息的任务、工具失败任务、权限受限任务、冲突结果任务、高风险动作任务。不要只用能顺利完成的样例。Agent 的价值在复杂分支里，风险也在复杂分支里。

成功率任务是否完成，而且结果正确。
工具准确是否选择了合适工具和参数。
失控率是否超步数、越权、循环或误执行。
可复盘日志是否记录动作、观察和原因。
task_success_rate=正确完成且无越界的任务数总任务数
评估维度看什么失败信号
任务结果最终答案或动作是否正确完成了错误任务，或遗漏关键约束。
动作轨迹每一步是否必要、可解释重复搜索、无效重试、突然跳步。
工具调用工具和参数是否匹配目标用写入工具做只读查询，或参数缺失。
安全边界是否遵守权限和确认规则未确认就发送、删除、购买、授权。
失败处理失败时是否诚实停止或升级把空结果当证据，或继续编答案。
学到这里要能做什么为一个 Agent 原型设计不只看结果、也看过程是否失控的验收指标。

最小实现：先写一个透明循环

minimal build

最小 Agent 不需要复杂框架。你只需要一个循环、一个状态对象、一组工具定义、一个动作选择 prompt、一个工具执行器和一份日志。先让它在低风险只读任务里工作，比如“查资料并生成带依据的回答”“根据日志定位可能原因”“从几个候选文件里找信息”。

返回结果里不要只给 final answer。要返回每一步动作、工具参数、工具结果、状态更新、停止原因。这样你才能复盘：模型为什么选这个工具，工具有没有返回证据，为什么停止，是否有遗漏问题。透明度比自动化程度更重要。

最小流程：可观察的 Agent 循环agent_loop.py
def run_agent(goal, tools, max_steps=6):
    state = {
        "goal": goal,
        "observations": [],
        "open_questions": [],
        "final_answer": None
    }
    trace = []

    for step in range(max_steps):
        action = choose_next_action(state, tools)
        checked = validate_action(action, tools)

        if checked.needs_human_confirm:
            return stop("需要人工确认", state, trace)

        observation = execute_tool(checked)
        state = update_state(state, action, observation)
        trace.append({"action": action, "observation": observation})

        if is_done(state):
            return finish(state, trace)

    return stop("达到最大步数", state, trace)

def run_agent(user_task):
    return model.run_anything(user_task)

# 问题：没有工具白名单、参数校验、最大步数、人工确认和日志。
# 出错时既无法复盘，也无法证明系统遵守边界。
学到这里要能做什么写出一个 Agent 循环的伪代码，并指出系统约束应该放在哪里。

常见误区：越像自动化，越要保守

pitfalls

把 Agent 当万能规划器

模型不等于项目经理。目标含糊、工具含糊、边界含糊时，它只是在不稳定地猜下一步。

一上来就开放写操作

写文件、发消息、删数据、付款、授权都属于高影响动作，必须先草稿或确认。

只看最终成功

最终答案对，不代表过程可接受。越权、过度调用、错误引用都必须算失败。

没有日志

没有动作轨迹和观察结果，就不知道 Agent 为什么做出某个决定，无法复盘也无法改进。

错法后果修法
工具描述只有一句话模型误用工具，参数不稳定补用途、输入、输出、失败和限制。
没有最大步数循环、成本失控、用户等待设置步数、时间和成本预算。
状态只用聊天历史关键信息被噪声淹没维护结构化状态和观察列表。
工具失败后继续编答案把无证据输出包装成结果失败时降级、重试一次或停止说明。
高风险动作无确认误发、误删、误授权先生成计划或草稿，再确认执行。
学到这里要能做什么审查一个 Agent 方案是否有工具边界、状态管理、停止条件、日志和人工确认。

综合练习

practice
练习怎么做

不要把答案写成“Agent 会自动规划”。每道题都尽量落到目标、工具、状态、动作、观察、停止条件和安全边界。能把链路拆开，才是真的懂。

复习抓手
  • Workflow：步骤固定，系统按流程走。
  • Tool use：模型请求一次外部工具。
  • Agent：多步循环，下一步依赖观察结果。

一、基础概念

提示：完成、最大步数、连续失败、无新信息、风险确认、预算超限。
场景

11. Agent 想自动发送邮件、删除文件或提交订单，系统应该怎么处理？

提示：草稿、计划、确认、权限校验、审计日志。

三、错误诊断与验收

这组题重点练“失控了怎么定位”。

快速复习Agent 验收既看结果，也看动作轨迹、工具选择、安全边界和失败处理。
错误诊断

12. Agent 一直搜索相似关键词但没有新证据，这是什么问题？怎么修？

按“现象 / 原因 / 停止条件 / 修法”回答。
错误诊断

13. 工具返回空结果，Agent 仍然给出确定结论。问题在哪里？

提示：失败状态未进入状态，缺少拒答或升级规则。
错误诊断

14. Agent 绕过权限限制去查另一个入口，这属于什么风险？怎么防？

提示：工具白名单、权限校验、执行前系统判断。
短答

15. 为什么 Agent 必须保存动作轨迹日志？

提示：复盘、排错、审计、评估工具选择。
指标

16. Agent 原型验收不能只看成功率，还要看哪些指标？

提示：工具准确率、平均步数、越界率、确认触发、失败升级、成本。
场景

17. 设计 6 类 Agent 测试样例，覆盖正常和异常情况。

提示：正常、缺信息、工具失败、权限不足、冲突结果、高风险动作。

四、工程表达与迁移

最后把本节内容串成可讲、可做、可验收的方案。

快速复习最小 Agent 的价值不是“自动”，而是多步任务在边界内可执行、可停止、可复盘。
设计

18. 最小 Agent 返回结果里为什么要包含 trace、state、stop_reason？

提示：为了复现过程、定位错误、判断是否正常停止。
清单

19. 给出 8 条验收一个 Agent 原型的检查点。

覆盖目标、工具、状态、校验、停止、日志、权限、评估。
面试

20. 面试官问：你如何避免 Agent 失控？

提示：工具白名单、参数校验、最大步数、人工确认、日志和测试集。
场景

21. 如果做一个“自动排查 Python 报错”的 Agent，你会设计哪些工具和停止条件？

提示：读日志、查文件、运行只读检查、搜索文档；禁止乱删改；连续失败停止。
费曼解释
