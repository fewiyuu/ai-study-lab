LLM Application Engineering 05

延迟、成本、可靠性约束

前几节讲了 prompt、eval、RAG、Agent。它们解决的是“能不能做出正确行为”。这一节补上工程现实：一个 LLM 应用即使在样例上表现不错，也可能因为太慢、太贵、太不稳定而不能上线。用户不会因为你用了大模型就原谅 20 秒等待，团队也不能接受每次请求成本失控。

工程里的约束不是附属内容，而是设计的一部分。模型选择、上下文长度、RAG top-k、是否重排、是否用 Agent 循环、是否并发、是否缓存、超时后怎么降级，都会同时影响质量、延迟、成本和可靠性。你要学会把这些因素拆开估算，再做可解释的取舍。

学完你应该能做什么

  1. 拆解一次 LLM 请求的端到端延迟：网络、检索、重排、首 token、生成、后处理。
  2. 用输入 token、输出 token、调用次数和缓存命中估算成本。
  3. 解释 P50、P95、P99、SLA、timeout、retry、fallback、rate limit 的作用。
  4. 诊断常见上线问题：平均延迟好看但尾延迟很差、成本随上下文爆炸、重试放大故障。
  5. 设计一个最小上线方案的指标、日志、告警和降级策略。

核心地图：质量、延迟、成本、可靠性互相牵制

mental model

LLM 应用上线时，不能只看“回答质量”。质量通常要用更多上下文、更强模型、更多检索、更复杂重排或更多 Agent 步数来换；这些都会增加延迟和成本。反过来，如果只追求快和便宜，可能会缩短上下文、减少检索、换小模型，质量和稳定性就会下降。

可靠性也不是准确率的同义词。可靠性包括服务是否可用、超时是否可控、失败是否能降级、输出格式是否稳定、成本是否不会突然爆炸、异常是否能被监控发现。一个系统偶尔答得很好，但经常超时或格式乱掉，在工程上仍然不可靠。

直觉

上线不是让模型“尽量聪明”，而是在预算和时间内稳定地给出可接受结果。

正式说法

生产约束是围绕服务质量、延迟分布、单位成本、失败处理和可观测性建立的工程边界。

判断句

如果你说不清一次请求慢在哪里、贵在哪里、失败后怎么处理，就还没有完成工程设计。

学到这里要能做什么把“模型效果不错”继续追问成质量、延迟、成本、可靠性四组指标。

为什么它们是约束，不是上线后再优化

constraints

延迟、成本和可靠性越晚考虑，越难改。比如你早期把 prompt 写得很长，RAG 每次取 20 个 chunk，Agent 最多跑 12 步，等到上线前发现成本太高，就不是删一句 prompt 能解决的。你可能要重做检索、缓存、模型路由、上下文压缩和失败处理。

更重要的是，约束会影响产品形态。客服机器人如果 3 秒内不能给出可用回应，就需要先返回“正在查询”或提供人工转接；内部文档助手如果成本很高，就需要限制用户批量请求；Agent 如果可能执行高风险动作，就必须先输出草稿或计划而不是自动执行。

设计期先定 SLA、预算、风险等级和可接受失败方式。
开发期记录 token、耗时、调用链、缓存命中、错误码。
验收期用 P95、成本上限、失败率和降级结果验收。
上线后持续监控漂移、峰值、供应商错误和成本异常。
学到这里要能做什么解释为什么延迟和成本不能等到功能完成后才补救。

延迟拆解：不要只看总耗时

latency

端到端延迟是用户从发出请求到看到结果的时间。它通常由多段组成：前端和网络、鉴权、检索、重排、模型排队、首 token 时间、生成 token 时间、后处理和传输。只看总耗时，你不知道该优化哪里。要把每段耗时记录下来。

LLM 请求还有一个特殊点：首 token 时间和完整生成时间不同。流式输出可以让用户更早看到内容，但完整答案仍然要等模型生成完。输出越长，生成时间越长。RAG 会增加检索和重排耗时；Agent 会把多次模型调用和工具调用串起来，尾延迟更容易变差。

P50一半请求低于这个耗时，代表常规体验。
P9595% 请求低于这个耗时，更能反映多数用户能否接受。
P99尾部极慢请求，常暴露拥塞、重试和外部依赖问题。
SLA对用户或业务承诺的服务目标。
学到这里要能做什么把一次请求总耗时拆成各段，并判断优化应该从哪一段开始。

成本计算：贵通常贵在 token 和调用次数

cost

LLM 成本主要来自输入 token、输出 token、模型单价和调用次数。RAG 会增加输入 token，因为要把检索片段放进上下文；Agent 会增加调用次数，因为每一步可能都要调用模型和工具；长回答会增加输出 token。成本不是一个固定值，而是随请求长度和流程分支变化。

估算成本时，不要只算单次 demo。要算每次请求成本、每日请求量、峰值请求量、缓存命中率、失败重试率和不同模型路由。一个看起来便宜的请求，如果被 Agent 循环调用 6 次，实际成本会很快放大。一个看起来贵的大模型，如果只用于少数高风险问题，可能反而合理。

取舍策略：不是所有问题都用最强配置

tradeoff

工程优化不是盲目压低成本，也不是一律使用最强模型。正确做法是按任务风险分层。低风险、高频、格式固定的任务可以用小模型或规则；高风险、低频、需要复杂推理的任务才使用强模型、人审或多步流程。这样系统整体更可控。

常见策略包括模型路由、缓存、上下文压缩、分阶段回答、流式输出、批处理、并行检索、只在必要时 rerank、只在必要时调用 Agent。每个策略都有副作用。缓存可能过期，压缩可能丢信息，小模型可能漏边界，并行会增加资源占用。取舍要有指标验证。

模型路由简单问题小模型，复杂问题大模型。
上下文预算给 RAG top-k 和历史消息设上限。
缓存重复问题、检索结果、系统 prompt 可缓存。
早停证据不足或风险过高时停止，不继续烧调用。
学到这里要能做什么为不同风险等级的请求选择不同模型、上下文和流程。

可靠性：失败要可预期、可降级、可复盘

reliability

可靠性不是“永远不失败”。任何外部 API、向量库、模型供应商、网络和工具都可能失败。可靠性的重点是：失败是否被发现，是否有明确错误类型，是否有降级方案，是否不会把失败包装成正常答案，是否能从日志里复盘。

LLM 应用的失败类型很多：超时、限流、格式错误、空检索、模型拒答、幻觉、引用错误、工具失败、权限不足、成本超限。不同失败要不同处理。格式错误可以重试一次；资料不足应该拒答；供应商超时可以 fallback；高风险动作需要人工确认；权限不足不能绕路。

可用性

服务是否能响应，依赖失败时是否有 fallback 或友好错误。

正确性

答案、格式、引用和工具结果是否符合任务要求。

可恢复性

失败后能否重试、降级、切换模型或交给人工处理。

失败类型不可靠处理更可靠处理
模型超时一直等待或无限重试设置 timeout，有限重试，fallback 到短回答或稍后重试。
RAG 空检索模型凭常识回答说明资料不足，记录 query 和检索结果。
JSON 格式错误直接让下游崩溃校验、修复一次、仍失败则返回结构化错误。
成本超限继续完整流程降级模型、缩短上下文、停止 Agent 循环。
权限不足尝试其他入口绕过停止并请求授权或说明无权限。
学到这里要能做什么为每类失败写出错误类型、用户可见结果、系统日志和降级动作。

超时、重试、降级：救场也会制造新问题

timeout retry fallback

超时是为了防止请求无限等待。重试是为了处理临时错误。降级是为了在主路径失败时仍给出可接受结果。但这三件事都不能乱用。重试会放大流量和成本；降级可能降低质量；超时太短会误杀正常请求，太长又伤害用户体验。

更稳的做法是按错误类型处理。网络抖动可以重试，参数错误不该重试；模型格式错误可以修复一次，RAG 没有证据不该让模型猜；供应商故障可以切备用模型，高风险任务不能自动降级成无审查执行。每个降级结果都要在日志里标记，不能混在正常成功里。

最小代码：按错误类型控制重试和降级reliability.py
def call_with_guardrail(request):
    try:
        return primary_model(request, timeout=6)
    except TimeoutError:
        log_event("timeout", request.id)
        return fallback_model(shorten(request), timeout=4)
    except FormatError:
        fixed = repair_once(request)
        if fixed.ok:
            return fixed
        return structured_error("format_failed")
    except PermissionDenied:

监控与告警：没有日志就没有工程判断

observability

上线后你不能靠用户投诉才知道系统坏了。每次请求都应该记录关键字段：trace id、用户场景、模型、输入输出 token、端到端耗时、各阶段耗时、检索数量、重排耗时、错误类型、重试次数、是否 fallback、最终状态。注意不要记录敏感原文，必要时脱敏或只记摘要。

告警要围绕业务能感知的问题，而不是只看服务器是否活着。比如 P95 延迟超过阈值、错误率上升、成本突然翻倍、fallback 比例异常、RAG 空检索比例升高、JSON 解析失败率升高、某模型供应商超时增加。这些都比“服务进程还在”更接近真实质量。

trace一次请求从入口到模型、工具和输出的完整链路。
metrics延迟、成本、错误率、缓存命中、fallback 比例。
logs关键事件和错误类型，支持排查。
alerts指标越过阈值时主动提醒。
字段为什么要记注意
latency_ms / stage_latency定位慢在检索、模型还是后处理记录 P50/P95/P99。
input_tokens / output_tokens估算成本和发现异常增长不要只看平均值。
model / route比较不同模型质量和成本保存模型版本。
error_type区分超时、限流、格式、权限、空检索错误要结构化。
fallback_used判断主路径是否稳定fallback 不能伪装成正常成功。

最小上线方案：先把边界守住

minimal production

最小上线方案不是把所有优化都做完，而是把关键边界守住。你至少要有：固定评测集、延迟目标、成本预算、超时和有限重试、fallback 或友好失败、结构化日志、基础监控、人工升级路径。没有这些，功能 demo 再漂亮也只是 demo。

上线前要写清楚“什么情况下算成功，什么情况下算失败”。比如客服问答：P95 延迟小于 5 秒，单次成本小于某个预算，RAG 空检索时必须说明资料不足，引用错误率不能超过阈值，连续模型超时要切换备用模型，涉及退款承诺时必须引用制度或转人工。

质量固定 eval cases，覆盖正常、边界、缺失和高风险。
延迟端到端和分阶段耗时，关注 P95/P99。
成本token、调用次数、缓存命中、日预算。
可靠性错误类型、重试、降级、人工升级、告警。

常见误区：平均数经常骗人

pitfalls

只看平均延迟

平均 2 秒不代表体验好。如果 P95 是 12 秒，仍然会有大量用户觉得系统卡死。

只算单次 demo 成本

真实成本要乘请求量、调用次数、重试率和输出长度。Agent 和 RAG 会明显放大成本。

重试当万能修复

限流或供应商故障时大量重试可能把系统拖得更慢、更贵。

fallback 不标记

降级结果混在正常成功里，会让你误判主路径质量。

错法后果修法
prompt 越写越长输入 token 和延迟持续上涨压缩系统 prompt，外部化固定规则，按需注入上下文。
RAG top-k 固定很大上下文噪声和成本增加按问题类型动态 top-k，必要时 rerank。
Agent 无步数限制循环、超时、成本失控设置最大步数、失败次数和人工升级。
格式错误直接进入下游系统崩溃或写入脏数据结构化校验，修复一次，失败返回错误。
没有成本告警异常流量或 prompt 变长时难以及时发现记录 token 和日预算，设置异常增长告警。
学到这里要能做什么看到上线方案时，主动检查平均数背后的尾延迟、峰值成本和失败处理。
