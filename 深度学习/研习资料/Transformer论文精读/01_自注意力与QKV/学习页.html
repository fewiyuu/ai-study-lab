学习目标

goal

先抓住这条主线

自注意力让每个位置都能为自己生成一个查询，再去同一句子里的所有位置上找匹配。

匹配分数决定谁更该被读到，Value 决定真正被带回去的内容。

这意味着 attention 的核心“每个 token 用什么规则决定看谁、读什么、怎么混”。

你如果能把这三件事说清，后面的多头和 encoder / decoder 就只是把同一条机制拆开看。

这页先不追什么

先不把 multi-head、encoder-decoder cross-attention、位置编码和后续训练细节混进来。

它们都和 attention 有关，但不属于这一页的主任务。先把“单头、同源、同序列”的机制读顺，再去看下一节。

> 你现在应该能做什么 如果有人把 attention 说成“一个 token 只看自己”，你能马上指出问题：Q、K、V 都来自同一输入，但查询会去整句里比对，不是只做自我复制。

主线位置

overview

从输入到输出， attention 只改了一件事

输入张量 x 先被投影成三份：q、k、v。

q 负责发问，k 负责被匹配，v 负责提供内容。然后 q 和 k 先算分数，再把分数变成权重，最后用权重去混合 v。

这个过程没有引入卷积，也没有引入循环。它做的是“同一序列内部的内容路由”。

所以这一页要先把路由规则讲清，后面才能讨论为什么这种规则比 RNN 更容易并行。

阅读顺序

  1. 先看公式，知道每一项是什么。
  2. 再看形状，知道每一步读什么、写什么。
  3. 接着看代码，知道转置、mask、softmax 放在哪。
  4. 最后看例子，知道一行分数怎样变成一行上下文。

> 边界提醒 这一页讲的是原论文里的单头 scaled dot-product attention。相对位置、RoPE、稀疏 attention、KV cache 都不是这里的主线。

公式读法

formula

原论文把 attention 写成一条短公式。短，不代表简单；它只是把“先比对、再归一化、再汇总”压进了一行。

读它的时候，别先背名词，先按数据流拆开。

Q 和 K 先生成分数表，d_k 控制分数尺度，M 是 mask，V 是真正被混合回去的内容。

这条式子里最容易漏的顺序：先打分，再缩放，再加 mask，再做 softmax，最后乘 V。

Q查询条件：当前位置想找什么x @ W_Q

K被匹配的索引：每个位置提供什么线索x @ W_K

V被带回的内容：匹配后真正汇总什么x @ W_V

d_kkey / query 的维度缩放因子里的分母

M不可见位置的遮罩mask 或 padding mask

你现在应该能做什么：看到一段 attention 公式，先指出哪一部分在算分，哪一部分在归一化，哪一部分在做内容汇总。

形状跟踪

trace

先用一个最小输入看形状：x 的形状是 [B, N, D]，其中 B 是 batch，N 是序列长度，D 是 hidden 维度。

self-attention 做的把每个位置和所有位置的关系重算一遍，再输出一个同样长度的新表示。

x @ W_Q输入表示query 表示Q 和 K 语义不分，后面分数没意义

x @ W_K输入表示key 表示query 无法检索到正确位置

x @ W_V输入表示value 表示即使分数对了，带回去的内容也会乱

q @ k^Tquery 和 key打分矩阵如果转置错了，轴会变，语义也会变

softmax(..., dim=-1)分数表每行权重分布归一化轴错位，行不再表示“读谁”

你现在应该能做什么：看到一段实现时，先判断每一步是读输入、读关系，还是写输出，而不是只盯着张量名字。

分数行读法

row

这一节把 [B, N, N] 再压成一个更直观的对象：某一个 query 对所有 key 的分数行。

你不需要一次记住整个矩阵，只要能看懂一行，就能看懂整张表。因为每一行做的事情都一样，只是 query 位置不同。

为什么这一行最值得盯

一行分数就是一个检索结果。行里每个值都在回答同一个问题：当前位置更该看谁。

只要这一行的轴、缩放和 mask 都对，整个矩阵就只是把这个逻辑复制到其他 query 位置上。

query 位置分数行在说什么如果读错了第一步查什么

“我”这个位置要读谁来补上下文会把主语和别的词混成一类先看 Q 是否来自这个位置

“看”这个位置要找什么对象来构成动作会把动词位置读成静态标签先看分数是否经过缩放和 softmax

“书”这个位置要从上下文里找什么关系会把尾部 token 的信息压成空分布先看 mask 有没有遮住不该看的位置

从一行分数到一行上下文

这一行一个位置的读取计划。它先告诉你“我更想看谁”，再通过 softmax 把偏好变成权重，最后把这些权重落到 V 上。

所以真正该读的是“这行最后把哪些位置的内容抬高了”，不是“这一行最高的数字是多少”。

如果这一步错了，最常见的不是报错，而是语义偏移：动词可能总盯着自己，句首可能被忽略，或者 padding / 未来词悄悄混进上下文。

这类错误形状仍然可能完全正确，只有在你把一行读成“查询结果”时才会暴露出来。

一行分数的解释复制

# 假设 scores[0, 1] 是“看”这个 query 对三处 key 的分数
row = torch.tensor([2.0, 1.0, 0.0])
weights = torch.softmax(row / math.sqrt(4), dim=-1)
print(weights) # 读法：当前 token 更偏向哪一处 key
context = weights @ torch.tensor([[1., 0.], [0., 1.], [1., 1.]])

> 这段代码在说明什么 先把一行分数当作 query 的检索结果，再把它变成权重，最后乘上三份 value。你如果把这一行读懂了，整张分数表只是把同样的逻辑复制到别的 query 上。

这一行最常见的误读，是把它当成“整句相似度”。不是。它只是某个位置在当前上下文里要读谁。

所以 attention 解释时，真正应该说清楚的是“谁在发问、谁在被匹配、谁被带回”，而不是只报一个总分。

这一点也解释了为什么只看最高分不够。最高分只能告诉你“这一行最偏向谁”，却不能告诉你这一行有没有被 mask 过、有没有被缩放过，也不能告诉你它最后有没有真正乘到 V。

真正的检查顺序还是要回到那条链：分数、权重、内容、输出，一步都不能跳。

这一节和前面的形状链是接着的：[B, N, N] 不是一个抽象的大矩阵，它就是把“每个位置的一行检索结果”拼在一起。

只要一行的语义懂了，整张表也就跟着懂了。

你现在应该能做什么：看见某一行 attention 分数时，先把它翻译成“这一行的 query 在读谁”，再去判断 mask、缩放和归一化有没有把这行弄歪。

QKV 读法

roles

Query 不是“问题本身”，它是检索条件

一个位置生成 query 之后，就像给自己写了一个检索条件。它不直接输出结果，而是决定“我应该在别的位置找什么样的线索”。

这也是为什么 query 既要和 key 比较，又不能和 value 混成一类东西。

Key 和 Value 的任务不一样

Key 负责被查找，Value 负责被带回。你可以把 Key 看成索引，把 Value 看成正文。索引能帮你定位，正文才是最后混合进上下文的内容。

对象最短直觉读什么写什么常见误解

Q检索条件当前 token 表示分数行把 Q 当成最终内容

K匹配索引所有 token 表示分数列把 K 当成输出内容

V实际内容所有 token 表示上下文向量把 V 和 K 混成同一件事

> 一个容易记错的点 不能把 attention 写成“权重乘 K 再输出”。K 只是拿来匹配的索引，真正被加权求和的是 V。这个区分一旦错了，代码形状可能还能对，语义已经错了。

这也是 self-attention 和普通线性层的差别：线性层只做局部变换，attention 先做全局匹配，再按匹配结果把内容拉回来。

所以 QKV 的拆分不是装饰，它就是这条流程能成立的最小结构。

缩放与 mask

scale

为什么要除以 sqrt(d_k)

点积的维度越高，分数通常越容易变得过大。分数一旦过大，softmax 就会变得很尖，权重几乎全压在一两个位置上，梯度也会更难稳定。

缩放为了让分数处在更适合归一化的尺度上。

mask 应该放在哪一步

先加到分数上，再做 softmax。这样被遮住的位置会真正变成几乎 0 的概率，而不是先被归一化、再强行改零。

顺序一旦反过来，行和就不再是一个合法分布。

步骤读什么写什么错了会怎样

缩放点积分数更平稳的分数尺度softmax 太尖，权重过于极端

加 mask不可见位置的约束把非法位置压到极小值未来位置或 padding 混进分布

softmax同一 query 对所有 key 的分数权重行轴错位，权重不再表示“读谁”

对这一步最有效的检查“分布的语义是不是还对”。

注意力代码常常能跑通，但一旦轴、mask 或缩放位置错了，输出会看着像张量，实际上已经不是那条公式了。

实现对齐

code

下面这段是最小实现。它不处理多头，也不处理缓存，只保留单头 self-attention 的最短路径。

读它的时候，先看转置和 mask，再看 softmax 和 value，最后看输出是不是仍然回到同样的序列长度。

代码
import math
import torch
from torch import nn


class SelfAttention(nn.Module):
 def __init__(self, d_model: int, d_k: int | None = None, d_v: int | None = None):
 super().__init__()
 d_k = d_k or d_model
 d_v = d_v or d_model
 self.w_q = nn.Linear(d_model, d_k, bias=False)
 self.w_k = nn.Linear(d_model, d_k, bias=False)
 self.w_v = nn.Linear(d_model, d_v, bias=False)

 def forward(self, x: torch.Tensor, mask: torch.Tensor | None = None) -> torch.Tensor:
 q = self.w_q(x)
 k = self.w_k(x)
 v = self.w_v(x)
 scores = q @ k.transpose(-2, -1)
 scores = scores / math.sqrt(k.size(-1))
 if mask is not None:
 scores = scores.masked_fill(mask == 0, float("-inf"))
 weights = torch.softmax(scores, dim=-1)
 return weights @ v

对象状态读什么写什么错了会怎样

q查询表示x分数行query 和 key 的关系跑偏

k匹配表示x分数列轴不对，q @ k^T 失去意义

v内容表示x上下文向量读到了索引，却没带回内容

mask约束状态输入约束非法位置的极小值非法位置混入注意力分布

手算一轮

case

下面用一个短句子做例子：[我, 看, 书]。假设我们只盯住中间那个 token“看”，它产生的 query 会去和三个 key 比较，再按分数决定自己更该读谁。

这在做上下文选择。

再看 mask

如果其中一个位置不该被看见，比如 padding 或未来词，mask 会先把那一项压到极小值，再做 softmax。

这样它最后的权重就会接近 0，而不是在归一化之后再被硬改成 0。

步骤例子里的值读出来的意思

原始分数[2.0, 1.0, 0.0]“看” 觉得三处位置的相关性不同

缩放后[1.0, 0.5, 0.0]分数尺度更平稳，softmax 不那么尖

softmax 后[0.506, 0.307, 0.186]每一项都是该位置的读法权重

最终输出0.506 v0 + 0.307 v1 + 0.186 v2把被选中的内容按权重混回上下文

你可以把这一步理解成“先算谁更相关，再按相关度取内容”。这里最重要的顺序和方向。

只要顺序对了，数值再换一组，机制还是同一条。

你现在应该能做什么：拿到一行 attention 分数时，先判断它是 query 视角还是 key 视角，再判断它有没有经过缩放和合法归一化。

常见误区

debug

误区 1：把 Q / K / V 当成三份完全不同的输入

它们来自同一个 x，区别在于投影后的角色，不在于来源。看见三套线性层，不等于看见三种不同数据。

误区 2：把 Value 和 Key 混成一件事

K 用来算谁更相关，V 用来提供真正的内容。把 K 当成输出内容，代码可能还能写出来，语义已经偏了。

误区 3：只看张量形状，不看 axis

[B, N, N] 很容易看着“对”，但 softmax 的轴一错，语义就完全换了。形状对不代表分布对。

误区 4：把 mask 放到 softmax 后面

这样做会破坏每一行的概率和，非法位置虽然被清掉了，但分布已经不是原公式里的那种分布。

误区 5：把缩放当成可有可无的常数

sqrt(d_k) 不是装饰项。它是在控制分数尺度，让 softmax 不至于太早饱和。

误区 6：拿 attention 权重直接当解释

权重能说明一次混合，但不能自动说明因果。它告诉你“这次读了谁”，不等于告诉你“模型为什么只能这么读”。

> 查错顺序 先看 Q / K / V 来源是否都来自同一个输入，再看转置是否对，再看 softmax 轴和 mask 顺序，最后再看 V 有没有被真正乘进去。很多 bug 其实只要把这一顺序固定下来，就能很快找到入口。
