Attention Is All You Need · Block 02

多头注意力：把一次读取拆成多种读法

上一页你已经学过 scaled dot-product attention：Q 和 K 打分，softmax 得到权重，再用权重汇总 V。多头注意力没有换掉这套计算，它把同一批 token 投影到多个较小子空间，每个子空间各自做一次 attention，最后把结果拼接并重新投影回 d_model。

X→ Q/K/V 投影→ 拆成 h 个 head→ 各自 attention→ Concat + W^O

学完后要能做三件事

  1. 能解释多头不是“重复算 8 遍”，而是在不同投影子空间里读关系。
  2. 能从 [B, N, D] 推到 [B, h, N, d_head]，再还原到 [B, N, D]。
  3. 能检查实现里的 view、transpose、mask 广播和 W^O 是否写对。

本块默认使用论文常见配置：d_model=512，h=8，每个 head 的 d_k=d_v=64。

为什么单头还不够

MOTIVATION

单头 attention 会把所有关系压进一张权重表

单头 attention 对每个 query 位置只产生一行权重。这行权重可以同时给多个 token 分配概率，但它仍然是一套统一的读法。一个词既可能需要看主语，也可能需要看宾语、介词短语、时态线索或长距离指代。如果所有关系都挤在同一个分数空间里，模型当然还能学，但表达会变得紧。

多头注意力的直觉是：不要让一次 attention 承担所有关系。先用不同的线性层把同一个 token 表示投影到不同子空间，再让每个 head 各自算一张注意力表。某个 head 可能更容易学局部搭配，另一个 head 可能更容易学长距离依赖，还有的 head 可能只是承担辅助混合。这里的“可能”很重要：head 不是人工规定的语法角色，而是训练中学出来的计算通道。

你可以把判断落到代码上：如果输入是 [B, N, 512]，多头不是把 512 维完整向量复制 8 份去算。标准做法是先线性投影，再 reshape 成 [B, N, 8, 64]，转成 [B, 8, N, 64]。每个 head 只在 64 维里算点积，8 个 head 并行执行。

读到这里，你应该能回答：多头 attention 增加的是“多种投影后的读取通道”，不是把同一个单头简单重复。

单头和多头的区别

对象准确读法常见误解
单头一套 Q/K/V 投影，一张权重表，一次汇总。以为它只能看一个 token。实际上 softmax 权重可分配给多个位置。
多头多套投影，多张权重表，多路汇总后拼接。以为每个 head 都有固定语言学功能。
并行各 head 可以在同一批矩阵运算中完成。以为要按 head 一个个循环才是多头。
表达能力让不同子空间学习不同关系，再由输出投影混合。以为只是参数变多，没有结构意义。

论文公式怎么读

FORMULA

论文的多头注意力公式很短，但里面包含三步：先为每个 head 做线性投影，再各自做 scaled dot-product attention，最后把所有 head 拼起来并乘输出投影矩阵。

MultiHead(Q, K, V) = Concat(head1, ..., headh) WO headi = Attention(Q WiQ, K WiK, V WiV)
这里的 W_i^Q、W_i^K、W_i^V 是第 i 个 head 的投影矩阵，W^O 是拼接后的输出投影。

第一步：每个 head 有自己的 Q/K/V 视角

同一个 token 向量进入不同 head 前，会先乘不同的投影矩阵。投影不是装饰，它决定这个 head 在什么特征空间里计算相似度。也就是说，不同 head 看到的是同一个 token 的不同表示。

判断句：如果实现里所有 head 直接共享完全相同的 Q/K/V 且没有 head 维度上的参数差异，它就不像论文里的 multi-head attention。

第二步：各 head 独立算 attention

每个 head 内部仍然是上一页的公式：softmax(QK^T / sqrt(d_k)) V。区别在于这里的 d_k 是每个 head 的维度，不是总的 d_model。论文配置里 d_k=64，所以缩放除以 sqrt(64)。

判断句：看到多头代码时，缩放因子应当跟 head 内部的 key 维度一致。

第三步：拼接后重新混合

每个 head 输出 [B, N, d_head]。把 8 个 head 拼起来，得到 [B, N, 8*d_head]，也就是回到 [B, N, d_model]。但拼接只是把结果并排放在一起，还没有让不同 head 交换信息。W^O 负责把这些通道重新混合成下一层需要的表示。

判断句：如果只 concat 不做输出投影，形状可能对，但少了论文公式里的可学习混合。

维度拆分和恢复

SHAPE

多头注意力最容易错的地方不是公式，而是张量形状。你需要把“特征维 D”切成“head 数 h × 每头维度 d_head”，再把 head 维移动到 batch 后面，方便矩阵乘法按 head 并行计算。

[B, N, D]输入表示
[B, N, 3D]一次投影出 QKV
[B, N, 3, h, d]拆出 Q/K/V 和 head
[B, h, N, d]转置给 attention
[B, N, D]拼接并输出投影

为什么要转成 [B, h, N, d_head]

attention 的矩阵乘法要让每个 query 位置和所有 key 位置相乘。单头时常见形状是 [B, N, D] @ [B, D, N] -> [B, N, N]。多头时我们希望每个 head 都有自己的 N × N 分数表，所以把 head 放到 batch 后面，得到 [B, h, N, d] @ [B, h, d, N] -> [B, h, N, N]。

如果你忘记转置 head 维，代码可能在某个地方报 shape mismatch；更麻烦的是，有时错误的广播会让它不报错，但把 token 维和 head 维混在一起。调试时不要只看最终输出形状，必须打印 score 的形状是否为 [B, h, N, N]。

读完这一节，你应该能从输入形状一路推导到 attention score 的四维形状。

view、reshape、contiguous

PyTorch 的 transpose 常常返回非连续内存视图。此时直接 view 可能报错，或者在旧代码里造成难查的问题。常见写法是在转回 [B, N, h, d] 后调用 contiguous()，再 view(B, N, D)。也可以用 reshape，但你仍然要知道它可能触发拷贝。

判断句：只要你看到 transpose 后紧跟 view，就要检查中间是否需要 contiguous()，并确认拼接方向没有把 head 和 token 维弄反。

为什么 concat 后还要 W^O

OUTPUT

拼接只是在通道上并排放置

假设 8 个 head 每个输出 64 维，concat 后就是 512 维。这个 512 维看起来已经回到了 d_model，但它仍然带着“前 64 维来自 head 1、后 64 维来自 head 2”的分块结构。下一层当然可以继续处理它，但论文里加了 W^O，让模型先在当前位置把不同 head 的信息重新混合。

W^O 的作用不是改变序列长度，也不是再做一次 attention。它是一个线性层，处理的是每个 token 位置上的通道混合。你可以把它理解成“把多路读取结果整理成下一层能接着用的表示”，但不要把它和 Q/K/V 投影混为一谈。

如果去掉 W^O，模型仍然可能跑起来，尤其在玩具代码里形状也能对。但它少了论文中的一个可学习混合步骤，head 之间的信息整合会被推迟到后面的层或 FFN。标准实现通常保留它。

易混概念对照

概念准确含义容易混淆点
d_model每个 token 的总隐藏维度。不是每个 head 的维度。
d_head每个 head 内部用于 Q/K/V 的维度。缩放时用它，而不是总维度。
head一套投影后的 attention 通道。不是一个独立小模型，也不是固定语法规则。
concat把多个 head 的输出沿特征维拼接。不是求平均，也不是相加。
W^O拼接后的输出线性投影。不是 mask，也不是 softmax 后处理。

两个小交互

OBSERVE

head 数和每头维度

固定 d_model=512 时，head 数越多，每个 head 分到的维度越小。维度越小并不自动更好，它是在“多种读法”和“每种读法的容量”之间做分配。

8

mask 广播形状

多头 attention 的 score 是 [B, h, N, N]。常见 causal mask 可以是 [N, N]，也可以显式扩成 [1, 1, N, N]。关键是它要同时广播到 batch 和 head。

最小 PyTorch 实现

CODE

下面代码保留了多头注意力最重要的形状变化。它不追求完整工程封装，但足够用来检查 QKV 投影、拆 head、注意力计算、拼接和输出投影。

正确示例：从 [B, N, D] 到多头输出PyTorch
import math
import torch
import torch.nn as nn
import torch.nn.functional as F

class MultiHeadSelfAttention(nn.Module):
    def __init__(self, d_model=512, num_heads=8):
        super().__init__()
        assert d_model % num_heads == 0
        self.num_heads = num_heads
        self.d_head = d_model // num_heads
        self.qkv = nn.Linear(d_model, 3 * d_model)
        self.out = nn.Linear(d_model, d_model)

    def forward(self, x, mask=None):
        B, N, D = x.shape
        qkv = self.qkv(x)                         # [B, N, 3D]
        qkv = qkv.view(B, N, 3, self.num_heads, self.d_head)
        qkv = qkv.permute(2, 0, 3, 1, 4)       # [3, B, h, N, d]
        q, k, v = qkv[0], qkv[1], qkv[2]               # each [B, h, N, d]

        scores = q @ k.transpose(-2, -1)          # [B, h, N, N]
        scores = scores / math.sqrt(self.d_head)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float("-inf"))
        weights = F.softmax(scores, dim=-1)
        context = weights @ v                    # [B, h, N, d]

        context = context.transpose(1, 2).contiguous() # [B, N, h, d]
        context = context.view(B, N, D)          # concat heads
        return self.out(context)                 # W^O

关键行逐行读

self.qkv(x) 一次线性层输出三份向量，不是因为 Q/K/V 必须共享参数，而是工程上常把三次线性投影合并成一次大矩阵乘法。随后 view(B, N, 3, h, d) 把三份投影和 head 维拆出来。

permute(2,0,3,1,4) 把 Q/K/V 这个维度放到最前面，方便用 qkv[0] 取 Q；同时把 head 放到 batch 后面，得到 [B,h,N,d]。如果这里顺序写错，后面矩阵乘法很可能把 token 维、head 维或特征维混掉。

context.transpose(1,2).contiguous().view(B,N,D) 是拼接 head 的关键。先把形状从 [B,h,N,d] 转回 [B,N,h,d]，再把 h*d 合成 D。漏掉这步转置会把序列位置和 head 顺序错乱。

错误示例：缩放和拼接都写偏

错误示例：用 d_model 缩放，且未转回 token 维PyTorch
scores = q @ k.transpose(-2, -1)
scores = scores / math.sqrt(D)       # 错：应该用 d_head
weights = torch.softmax(scores, dim=-1)
context = weights @ v                # [B, h, N, d]
context = context.contiguous().view(B, N, D)
out = self.out(context)

这段代码的问题有两层。第一，缩放用总维度 D 会让分数被除得过小。第二，context 还在 [B,h,N,d]，直接 view 成 [B,N,D] 会把 head 维和 token 维按内存顺序硬揉在一起，语义错了。正确做法是先 transpose(1,2)。

mask 广播的最小写法

多头里的 mask 不只是“有一个 0/1 表”就行。score 的形状是 [B,h,N,N]，mask 要么能自然广播到这四维，要么你就把它显式扩成四维。显式写清楚通常更适合学习和调试。

正确示例：padding mask 与 causal mask 广播PyTorch
# scores: [B, h, N, N]
padding_mask = padding_mask[:, None, None, :]   # [B, 1, 1, N]
scores = scores.masked_fill(padding_mask == 0, float("-inf"))

causal = torch.tril(torch.ones(N, N, device=x.device))
causal = causal[None, None, :, :]             # [1, 1, N, N]
scores = scores.masked_fill(causal == 0, float("-inf"))

第一种 mask 挡的是 padding token：所有 query、所有 head 都不该读取那些无效 key。第二种 mask 挡的是未来 token：第 i 个位置只能读到不晚于 i 的位置。两者都在 softmax 之前作用，因为你要让被挡位置的概率从归一化里消失。

读这段代码时看四个点

第一，看 mask 对齐的是 score 的最后一维还是最后两维。padding mask 通常只对 key 位置生效，所以形状是 [B,1,1,N]；causal mask 同时约束 query-key 位置对，所以是 [1,1,N,N]。

第二，看 mask 是否在 softmax 之前写入 -inf。如果先 softmax 再置零，剩余概率不会自动重新归一，注意力总量会变小。

第三，看 mask 是否能广播到 head 维。多头里的每个 head 都要遵守同一套 padding 或因果约束，不能只遮住某一个 head。

第四，看 causal mask 的方向。Transformer decoder 训练时如果能读未来 token，生成时就会遇到训练和推理不一致的问题。

常见误区和修法

PITFALLS

1. 把 head 当成人工指定的语法槽

有些可视化会显示某些 head 关注局部词、分隔符或长距离关系，于是很容易误以为第 1 个 head 永远负责主谓一致，第 2 个 head 永远负责指代。这种说法过头了。head 是可学习通道，训练结果可能呈现偏好，但不是人工写死的规则。

修法：描述时说“某些 head 可能学到不同关系模式”，不要说“每个 head 必然对应一个语言学功能”。

2. 忘记 d_model % num_heads == 0

如果总维度不能被 head 数整除，就无法平均拆成 h 个 d_head。有些实现可以允许不同 head 不同维度，但标准 Transformer 不是这样。

修法：初始化时加断言，先保证 d_head = d_model // num_heads 是整数。

3. 缩放因子用错

多头里每个 attention 的点积发生在 d_head 维，而不是 d_model 维。用 sqrt(d_model) 会把分数压得更小，影响 softmax 的尖锐程度。

修法：只要 score 来自 [..., d_head] 的点积，就除以 sqrt(d_head)。

4. mask 维度不能广播到 head

score 是 [B,h,N,N]。如果 mask 是 padding mask，常见形状可能是 [B,1,1,N]；如果是 causal mask，常见形状是 [1,1,N,N]。写成 [B,N] 后直接用，通常不够清楚，也容易广播到错误维度。

修法：把 mask 显式 reshape 到能对齐 score 的四维形状。

5. transpose 后直接 view

转置会改变张量 stride。很多时候你需要 contiguous() 再 view，或者用 reshape 并知道它可能复制数据。

修法：拼接 head 前写成 context.transpose(1,2).contiguous().view(B,N,D)。

6. 以为多头一定比单头省计算

标准配置下，多头把维度拆小，使总投影维度仍接近 d_model，不是把计算免费变少。主要收益是表达结构和并行矩阵计算的组织方式，不是 attention 的 N² 代价消失了。

修法：说多头时分开讲“表达通道”和“计算复杂度”，不要把它说成长上下文优化方法。
