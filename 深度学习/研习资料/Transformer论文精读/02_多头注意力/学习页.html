Transformer 论文精读 02｜多头注意力

上一页已经把 scaled dot-product attention 读顺：Q 和 K 打分，softmax 得到权重，再用权重汇总 V。 多头注意力沿用这条链：先把同一批 token 投影到多个较小子空间，让每个 head 各自读一遍，再把结果拼接并用 W^O 混合。

读完本页应该能做什么

  • 解释多头如何把同一输入分成多组投影和注意力表。
  • 从 [B, N, D] 推到 [B, h, N, d_head]、[B, h, N, N]，再回到 [B, N, D]。
  • 检查实现里的 view、permute、transpose、mask 广播和 W^O 是否对齐论文公式。

学习目标

goal

先抓住主线

多头注意力的主线是：先线性投影出多组 Q/K/V，再把每组投影当成一个 head，各 head 独立执行 attention，最后把各 head 的输出沿特征维拼接。 拼接后的向量再经过 W^O，让不同 head 的结果在当前位置重新混合。

学这页时把 head 当作可学习通道来看。训练后某些 head 可能偏向局部搭配、长距离依赖或分隔符，这些偏好来自学习过程，页面里先按可观察现象来理解。

这页先不追什么

先不展开 FlashAttention、KV cache、MQA/GQA、head 剪枝、RoPE 或长上下文优化。 这些机制都和 attention 有关，但会分散当前任务。这里先把原论文的 multi-head attention 公式、形状和最小实现读顺。

现在能做的事 如果看到一段实现只给出最终 [B, N, D] 输出，你应该能要求它打印中间的 [B, h, N, d_head] 和 [B, h, N, N]，因为这两个形状最能暴露多头是否真的分开计算。

为什么要分头

motivation

单头会把多种关系压进一套读法

单头 attention 对每个 query 位置生成一行权重。这行权重可以同时分给多个 token，但它仍然来自同一套投影和同一个分数空间。 一个词在同一句里可能要看主语、宾语、介词短语、时态线索和长距离指代。所有关系都挤进同一套相似度计算时，模型仍能学习，只是表达空间更紧。

多头的做法是先把输入投影到多个子空间。每个 head 在自己的 d_head 维空间里算 QK^T，得到自己的注意力表。 这些表由训练过程学到，页面里先按不同读取通道来理解。

代码上可以这样检查：如果输入是 [B, N, 512]，标准多头先做 8 组投影，再分别计算 attention。 它通常先投影到 [B, N, 3D]，再拆成 [B, N, 3, 8, 64]，最后转成每个 Q/K/V 都是 [B, 8, N, 64]。

对象准确读法常见误解
单头一套 Q/K/V 投影，一张权重表，一次汇总。以为它只能看一个 token；实际上权重可以分配给多个位置。
多头多套投影，多张权重表，多路汇总后拼接。以为每个 head 都有固定语言学功能。
并行各 head 可以合在同一批矩阵运算中完成。以为必须按 head 一个个循环。
表达让不同子空间学习不同关系，再由输出投影混合。只把它看成参数数量增加。

论文公式怎么读

formula

论文把 multi-head attention 写成两行。读这两行时，按“各 head 投影、各 head 算 attention、拼接后输出投影”三步拆开。

$$ \operatorname{MultiHead}(Q,K,V) = \operatorname{Concat}(\operatorname{head}_1,\ldots,\operatorname{head}_h)W^O $$ $$ \operatorname{head}_i = \operatorname{Attention}(QW_i^Q, KW_i^K, VW_i^V) $$

W_i^Q、W_i^K、W_i^V 是第 i 个 head 的投影矩阵。 W^O 处理 concat 后的通道混合，作用在输出投影层。

1. 每个 head 有自己的视角

同一个 token 向量进入不同 head 前，会乘不同的投影矩阵。投影决定了这个 head 在什么特征空间里计算相似度。 如果实现里所有 head 直接共享同一份 Q/K/V，且没有 head 维度上的参数差异，就偏离了原论文的 multi-head 结构。

2. 每个 head 内部仍是 scaled dot-product attention

head 内部仍然计算 softmax(QK^T / sqrt(d_k))V。 这里的 d_k 取每个 head 的 key/query 维度，base 配置里是 64。 原论文 base 配置里 d_k=64，缩放就是除以 sqrt(64)。

3. 拼接后还要混合

每个 head 输出 [B, N, d_head]。把 8 个 head 拼起来得到 [B, N, 8*d_head]，形状回到 [B, N, d_model]。 拼接只把结果并排放到特征维，W^O 才负责把这些通道重新混合。

形状跟踪

shape

多头实现最容易错在张量形状。把“特征维 D”切成“head 数 h × 每头维度 d_head”之后，还要把 head 维移动到 batch 后面，方便每个 head 都得到自己的 N × N 分数表。

input [B,N,D]

每个 token 一行隐藏向量。

qkv [B,N,3D]

工程上常把三次线性层合成一次大投影。

split [B,N,3,h,d]

这里同时拆出 Q/K/V 维和 head 维。

head [B,h,N,d]

head 放到 batch 后面，矩阵乘法可以并行。

score [B,h,N,N]

每个 head 都有自己的 query-key 表。

output [B,N,D]

先转回 token 维，再 concat heads，再过 W^O。

步骤之前之后读取 / 依赖写错时的症状
QKV 投影[B,N,D][B,N,3D]Linear(D,3D)Q/K/V 没有独立投影，或者输出维度无法拆成三份。
拆 head[B,N,3D][B,N,3,h,d]D = h*dd_model % num_heads != 0 时无法整除。
转置[B,N,3,h,d][3,B,h,N,d]Q/K/V 维、head 维、token 维矩阵乘法混掉 token 维和 head 维。
打分q,k: [B,h,N,d][B,h,N,N]k.transpose(-2,-1)score 没有保持在 [B,h,N,N]，mask 也无法按 head 广播。
拼接[B,h,N,d][B,N,D]transpose(1,2).contiguous().view(B,N,D)漏转置会把序列位置和 head 顺序揉在一起。
一个具体数值 设 B=2、N=4、d_model=512、h=8。每头维度是 64。score 的形状应为 [2,8,4,4]，最后一个 4 是 key 位置，softmax 通常沿它归一化。

为什么 concat 后还要 W^O

output

concat 只把通道并排放好

假设 8 个 head 每个输出 64 维，concat 后就是 512 维。这个 512 维已经回到 d_model，但它仍带着分块结构：前 64 维来自 head 1，接着 64 维来自 head 2，以此类推。

W^O 是每个 token 位置上的线性层。它把这些 head 通道混合成下一层更容易接着处理的表示。 如果只 concat 不做输出投影，形状可能对，但少了论文公式中的一个可学习混合步骤。

术语直觉解释精确定义常见误读
head一套读取通道。在 d_head 维子空间里独立计算 attention。把它当成固定语法槽。
d_modeltoken 表示总宽度。Transformer 层输入输出隐藏维度。拿它当每个 head 的维度。
d_head每个 head 分到的宽度。标准写法里 d_model / h。缩放时误用 sqrt(d_model)。
concat沿特征维拼接。把多个 [B,N,d_head] 合成 [B,N,D]。当成求和或平均。
W^O输出通道混合。concat 后的线性投影矩阵。当成 mask 或第二次 attention。
用一个小例子看错位 假设只有 2 个 token、2 个 head，每个 head 输出 2 维。正确拼接前，张量应从 [B,h,N,d] 转成 [B,N,h,d]：第 1 个 token 拿到的是 head1(token1) 和 head2(token1)，第 2 个 token 拿到的是 head1(token2) 和 head2(token2)。 如果直接把 [B,h,N,d] view 成 [B,N,D]，形状仍可能是对的，但第 1 个 token 可能混进 head1(token2) 的内容。这个错误不会总是报 shape mismatch，所以验收时要打印拼接前后的局部索引，而不是只看最终输出维度。

最小 PyTorch 实现

code

下面代码保留了多头注意力最关键的形状变化。它只保留关键读写链条，足够检查 QKV 投影、拆 head、注意力计算、拼接和输出投影。

正确示例：从 [B, N, D] 到多头输出
import math
import torch
import torch.nn as nn
import torch.nn.functional as F

class MultiHeadSelfAttention(nn.Module):
    def __init__(self, d_model=512, num_heads=8):
        super().__init__()
        assert d_model % num_heads == 0
        self.num_heads = num_heads
        self.d_head = d_model // num_heads
        self.qkv = nn.Linear(d_model, 3 * d_model)
        self.out = nn.Linear(d_model, d_model)

    def forward(self, x, mask=None):
        B, N, D = x.shape
        qkv = self.qkv(x)                         # [B, N, 3D]
        qkv = qkv.view(B, N, 3, self.num_heads, self.d_head)
        qkv = qkv.permute(2, 0, 3, 1, 4)          # [3, B, h, N, d]
        q, k, v = qkv[0], qkv[1], qkv[2]          # each [B, h, N, d]

        scores = q @ k.transpose(-2, -1)          # [B, h, N, N]
        scores = scores / math.sqrt(self.d_head)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float("-inf"))

        weights = F.softmax(scores, dim=-1)
        context = weights @ v                     # [B, h, N, d]
        context = context.transpose(1, 2).contiguous()
        context = context.view(B, N, D)           # concat heads
        return self.out(context)                  # W^O

关键行逐行读

self.qkv(x) 一次线性层输出三份向量。它不表示 Q/K/V 完全相同，只是把三次线性投影合成一次大矩阵乘法。 view(B, N, 3, h, d) 把三份投影和 head 维拆出来。

permute(2, 0, 3, 1, 4) 把 Q/K/V 维放到最前面，方便用 qkv[0] 取 Q；同时把 head 放到 batch 后面，得到 [B,h,N,d]。 这一步顺序错了，后面的矩阵乘法很容易混掉 token 维、head 维或特征维。

拼接前的内存布局

context.transpose(1, 2).contiguous().view(B, N, D) 是拼接 head 的关键。 先把形状从 [B,h,N,d] 转回 [B,N,h,d]，再把 h*d 合成 D。 直接 view(B,N,D) 会按当前内存顺序硬揉，语义会错。

Mask 与故障定位

debug

mask 要对齐 score

多头 score 的形状是 [B,h,N,N]。padding mask 常见写法是 [B,1,1,N]，挡住无效 key； causal mask 常见写法是 [1,1,N,N]，挡住未来位置。两者都应在 softmax 前作用到 score 上。

正确示例：padding mask 与 causal mask 广播
# scores: [B, h, N, N]
padding_mask = padding_mask[:, None, None, :]  # [B, 1, 1, N]
scores = scores.masked_fill(padding_mask == 0, float("-inf"))

causal = torch.tril(torch.ones(N, N, device=x.device))
causal = causal[None, None, :, :]              # [1, 1, N, N]
scores = scores.masked_fill(causal == 0, float("-inf"))

            
            
            
          

        

        

          

            
              
症状先查哪里常见原因修法

            
            
              
score 没有保持在 [B,h,N,N]qkv.view 和 permutehead 维或 Q/K/V 维顺序错。先打印 q、k、v 形状，再做矩阵乘法。

注意力分布过平缩放因子用 sqrt(d_model) 代替 sqrt(d_head)。缩放跟点积维度一致。

              
decoder 泄漏未来 tokencausal mask 方向上三角/下三角方向写反。确认第 i 行只能读到不晚于 i 的 key。

              
padding 位置仍被读取mask 广播到 head 维[B,N] 没显式扩成四维。改成 [B,1,1,N]。

            
          

        

        

          

            
最小定位流程

            

              遇到 multi-head attention “能跑但不对”时，先不要改学习率或训练轮数。第一步打印 q.shape、k.shape、v.shape，确认都是 [B,h,N,d]。
              第二步打印 scores.shape，确认是 [B,h,N,N]。第三步单独检查 mask reshape 后的形状，看它是挡 key 位置还是挡 query-key 位置对。最后再看拼接前的 context.transpose(1,2) 是否存在。
            

            

              这套流程的好处是能把“值不对”拆成几个可观察对象：投影错、score 错、mask 错、concat 错。每一步都对应上面的表格和练习区的诊断题，排查时不需要靠猜。
            

          

          

            
把症状连回概念

            

              如果 score 形状不对，回看“形状跟踪”；如果缩放维度不对，回看“公式读法”；如果最终输出形状对但结果不稳定，优先查“输出投影”和拼接顺序；如果 decoder 训练指标异常好但生成很差，优先查 causal mask 方向。
              练习题里的开放诊断题也按这个顺序写：先定位对象，再说后果，最后给修法。

常见误区和修法

          pitfalls
        

        

          

            
把 head 当成人工指定的语法槽

            

              有些可视化会显示某些 head 关注局部词、分隔符或长距离关系，于是容易得出“head 1 负责主谓一致、head 2 负责指代”的说法。
              页面里先把它们当作可观察的偏好，不提前给固定标签。
            

          

          

            
忘记 d_model % num_heads == 0

            

              标准实现要把总维度平均拆成 h 个 d_head。
              如果不能整除，就无法直接 reshape 成 [B,N,h,d_head]。初始化时先加断言，错误会早很多暴露。
            

          

          

            
缩放因子用错

            

              多头里每个 attention 的点积发生在 d_head 维。用 sqrt(d_model) 会把分数压得更小，softmax 变得更平。
              只要 score 来自 [..., d_head] 的点积，就按 d_head 缩放。
            

          

          

            
把多头说成长上下文优化

            

              多头改善的是表达通道和矩阵计算组织方式。标准 attention 的 N^2 序列长度代价仍然在。
              长上下文优化要看稀疏注意力、FlashAttention、KV cache 等后续机制，不能把它们混到这页。

漏转置诊断

              
错误代码：context = context.contiguous().view(B,N,D)，其中 context 仍是 [B,h,N,d]。错因、后果、修法是什么？

              
答题目标：必须点出 head 维和 token 维被揉错。

              
              
              

                参考答案 / 评分要点
                
错因：还没把 [B,h,N,d] 转成 [B,N,h,d] 就 view。后果：最终形状可能是 [B,N,D]，但 token 顺序和 head 通道语义混乱。修法：先 context.transpose(1,2).contiguous()，再 view(B,N,D)。

              

            


            

              
diagnosismask

              
mask 形状诊断

              
score 是 [B,h,N,N]，mask 只有 [B,N]。直接 masked_fill 可能有什么问题？怎么改更清楚？

              
答题目标：指出广播目标，并给出显式 reshape。

              
              
              

                参考答案 / 评分要点
                
[B,N] 通常是 padding 标记，直接用在四维 score 上不够清楚，容易对错维度广播。更清楚的写法是 padding_mask[:, None, None, :]，得到 [B,1,1,N]，让它广播到 batch 和 head。

W^O 作用

              
concat 后为什么还要输出投影 W^O？

              
答题目标：说清 concat 和通道混合的区别。

              
              
              

                参考答案 / 评分要点
                
concat 只把多个 head 的输出沿特征维并排放回 d_model。W^O 是可学习线性层，负责把这些 head 通道混合成下一层可用的表示。它不改变序列长度，只做通道混合。

费曼解释

              
用 6-9 句话解释：多头注意力为什么要拆 head，又如何拼回 Transformer 的隐藏维度。

              
至少包含多头动机、每头维度、score 形状、concat、W^O 和一个实现坑。

              
              
              

                参考答案 / 评分要点
                
合格答案要包含：1）多头把同一输入投影到多个子空间；2）每个 head 在 d_head 维内算 scaled dot-product attention；3）score 形状是 [B,h,N,N]；4）每个 head 输出 [B,N,d_head]；5）concat 后回到 [B,N,D]；6）W^O 混合 head 通道；7）实现坑可选 mask 广播、缩放维度或漏转置。

实现验收清单

              
写一份 6 条清单，用来验收一个 multi-head attention 实现是否可信。

              
可以覆盖整除断言、QKV 投影、head 维转置、score 形状、缩放、mask 广播、concat 顺序、输出投影。

              
              
              

                参考答案 / 评分要点
                
清单至少应覆盖：d_model % num_heads == 0；QKV 投影能拆成三份；Q/K/V 是 [B,h,N,d]；score 是 [B,h,N,N]；缩放用 d_head；mask 能广播到 head；concat 前先 transpose(1,2)；最后保留 W^O。
