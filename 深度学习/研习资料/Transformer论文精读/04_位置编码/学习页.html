学习目标

goal

位置编码解决的是一个很具体的问题：self-attention 本身只看 token 之间的内容关系，不会自动知道第几个 token 在前、第几个在后。原论文的做法很直接，把一个固定的位置信号和 token embedding 相加，让模型在进入注意力层之前就看到顺序。

这页只讲原论文这一版位置编码：它为什么放在输入端、公式怎么展开、实现时为什么常用 register_buffer 保存表、以及哪里最容易把长度和设备搞错。

你现在应该能做什么：看见一段位置编码代码时，先判断它在做“生成表、搬设备、切前缀、再相加”，不要把它当成一个黑盒装饰层。

主线位置

overview

它在整个模型里放哪

输入 token 先经过 embedding，再加上 position encoding，随后一起进入 encoder 或 decoder 的第一层 attention。也就是说，位置和词向量一起成为“输入表示”的一部分，位置会在输入端加入。

为什么要先加再算 attention

如果不提前把位置信息送进去，两个相同内容但顺序不同的序列会在注意力里显得太像。先相加，模型后续就可以同时利用“这个词是什么”和“它排在第几位”。

边界提醒 这里先只看固定正弦位置编码。相对位置编码、RoPE、ALiBi、长上下文外推都先放一边，它们会在后续实现或扩展阅读里再回来。

你现在应该能做什么：在 Transformer 的输入流里画出一条线，指出位置编码是在 embedding 之后、第一层 attention 之前加入的。

公式读法

trace

原论文把位置编码写成偶数和奇数通道两条公式。这样做的好处是，模型在每一对通道上都能同时看到一个变化快的成分和一个变化慢的成分，位置越往后，数值模式越有规律。

$$ \begin{aligned} PE(pos, 2i) &= \sin\!\left(pos / 10000^{2i / d_{model}}\right) \\ PE(pos, 2i + 1) &= \cos\!\left(pos / 10000^{2i / d_{model}}\right) \end{aligned} $$

pos 是位置索引，i 是通道对编号，d_model 是 embedding 维度。偶数通道和奇数通道共享同一个频率因子，只是一个取 sin，一个取 cos。

符号意思看代码时对应什么
pos序列中的位置第 0、1、2... 个 token 的行号
i通道对编号第 0 对、第 1 对... 维度
d_modelembedding 总维度位置表的列数
10000^{2i/d_model}频率缩放项控制不同通道看见不同尺度的位置变化

直觉上可以这样读：低编号通道变化快，适合捕捉短距离顺序；高编号通道变化慢，适合保留更长跨度的位置信号。把它们拼在一起，模型就有了一组不同频率的“位置刻度”。

你现在应该能做什么：看到 2i 和 2i+1 时，知道它们是一对共享频率、不同相位的通道。

形状跟踪

shape

下面这条跟踪线用一个最小输入 batch=2, length=4, d_model=8 来看位置编码是怎么落到张量上的。先看“生成表”，再看“切前缀”，最后看“加到 embedding 上”。

位置索引 [max_len, 1]

每一行代表一个位置，后面会和不同频率的缩放项做广播。

频率项 [d_model/2]

每一项对应一对通道的频率，决定这个位置在该通道上的变化速度。

完整位置表 [max_len, d_model]

偶数列写 sin，奇数列写 cos；每一行是一种位置编码。

输入相加 [batch, seq_len, d_model]

切出前 seq_len 行，再和 embedding 逐元素相加。

步骤读什么写什么错了会怎样
生成索引max_len位置行号索引维度反了，广播就不对
生成频率d_model每对通道的缩放系数列数不成对，偶/奇通道写不进去
填表位置索引与频率项pe 的偶/奇列sin / cos 写反，模式也会变，但更难查
相加x 和前 seq_len 行编码后的输入切片长度不够或设备不一致，forward 直接报错

你现在应该能做什么：拿到一份位置编码实现，先只盯住“生成表的形状”和“前缀切片的长度”两件事，不要一上来就读公式细节。

例子

example

把 d_model=8、max_len=4 看成一个小表格最容易。第 0 个位置的向量里，偶数列和奇数列会是 sin(0)、cos(0)；随着位置变大，每对通道的变化速度不同，所以不同列能给出不同频率的位置信号。

一个最小直觉

如果两个 token 内容完全一样，但位置不同，embedding 本身相同，位置表不同，最后相加后的向量也不同。这样 self-attention 才能在后续层里区分“同样的词在不同位置”的情况。

这里别混进来什么 先不要把相对位置、旋转位置编码、偏置矩阵和长上下文外推混在这一个例子里。原论文这一页只需要说明：绝对位置被编码成固定表，再加到输入上。

再往前走一步，你会发现这类位置编码把整数 1、2、3 变成了不同频率的通道。这样，位置就成了一组有节奏的坐标，模型也能同时处理短距离变化和长距离变化。

它是公式生成的表，遇到更长序列时仍然保留同一套规律。你在实现里虽然会先缓存一个 max_len，但本质上是在把一个可生成规则装进缓存。

你现在应该能做什么：拿任意一个长度小的句子，口头说明“如果两段 token 序列内容一样但位置不同，为什么输入向量不会一样”。

实现对齐

implementation

最小 PyTorch 实现的关键在于把这个表作为模块状态保存下来，并确保它会跟着模型一起移动到正确设备。常见做法是用 register_buffer 注册一个 pe，然后在 forward 里按当前序列长度切片。

最小实现
import math
import torch
from torch import nn


class PositionalEncoding(nn.Module):
    def __init__(self, d_model: int, max_len: int = 5000):
        super().__init__()
        pe = torch.zeros(max_len, d_model)
        position = torch.arange(max_len, dtype=torch.float32).unsqueeze(1)
        div_term = torch.exp(torch.arange(0, d_model, 2, dtype=torch.float32) * (-math.log(10000.0) / d_model))
        pe[:, 0::2] = torch.sin(position * div_term)
        pe[:, 1::2] = torch.cos(position * div_term)
        pe = pe.unsqueeze(0)
        self.register_buffer("pe", pe)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        seq_len = x.size(1)
        return x + self.pe[:, :seq_len].to(dtype=x.dtype)
对象保存方式会不会训练forward 里怎么用常见故障
pebuffer不会按 seq_len 切前缀放成参数会多出不该训练的梯度
x输入张量不适用与 pe 相加设备不同，CPU / GPU 会报错
seq_len运行时读取不适用决定切多少行切太长会越界，切太短会丢位置信息

这里还有一个细节要盯住：forward 返回的东西和输入 x 的最后一维必须一致，否则相加会直接炸。也就是说，pe 的列数必须等于 d_model。

另一个容易漏掉的点是状态归属。pe 不参与优化，却又必须跟着模块保存、加载和迁移，所以它放在 buffer 里最合适。这样做后，优化器不会把它当参数更新，module.to(device) 也会把它一起搬走。这个安排更像“模块内的只读状态”，不该放进训练目标里。

如果你在混合精度或类型切换里工作，还要注意 self.pe[:, :seq_len].to(dtype=x.dtype) 这一步。它的作用是让位置表和输入张量在同一种数值类型里相加。少了这一步，CPU / GPU 设备问题可能已经解决，dtype 还会继续卡你。

正常输入时：`x.shape == [batch, seq_len, d_model]`，`self.pe[:, :seq_len]` 也能对齐，最后输出仍是 `[batch, seq_len, d_model]`。

你现在应该能做什么：看到一个位置编码类时，先检查它是否把 `pe` 作为 buffer 保存、是否按当前长度切片、是否和输入在同一设备。

手算一轮

case

这一节直接拿 d_model=8、max_len=4 做一个小表。你要看的三件事是：第 0 行是否有固定起点、前几列和后几列的频率是否不同、forward 是否只切前缀并保持整张表顺序。

先看第 0 行

pos=0 时，所有 sin 项都等于 0，所有 cos 项都等于 1。于是第一行很容易肉眼检查：偶数列接近 0，奇数列接近 1。这个起点很适合做单元测试，因为它不依赖训练，也不依赖输入内容，只依赖公式是否写对。

如果第一行不对，优先查三件事：sin 和 cos 是否写反，偶数列和奇数列是否切反，position 和 div_term 的广播方向是否弄反。这里出错时，模型通常还能跑，但位置表已经歪了。

再看第 1 行和第 2 行

位置往后走以后，前几列变化更快，后几列变化更慢。原因在于频率项随列号变化。低编号列更敏感，适合表达短距离顺序；高编号列更平滑，适合保留更长跨度的位置信号。

如果你把指数方向写反，或者把偶数通道的步长写错，表仍然会长得像一张“正常矩阵”，但这种快慢分层会消失。这类错误最难查，因为它落在语义层面，不会先从形状报错。

检查点正常现象写错时的症状先查什么
pos=0偶数列接近 0，奇数列接近 1第一行偏离 0/1 交替先看 sin / cos 和切片
pos=1每对通道都变化，但变化速度不同所有列一起同步摆动先看 div_term 是否按列分层
pos=2前几列更快，后几列更慢快慢分层消失先看指数方向和 d_model 步长
seq_len=3只切前三行，不改列数把整张表重新生成了一遍先看 forward 里是否只做前缀切片
手算检查
with torch.no_grad():
    pe = PositionalEncoding(8, max_len=4).pe[0]
    print(pe[0])      # 第 0 行应该接近 [0,1,0,1,0,1,0,1]
    print(pe[:3])     # 只看前 3 行，确认是前缀切片
    print(pe.shape)   # 最后确认列数仍然等于 d_model
这一步的价值 这个小检查足够早地抓住大多数实现问题：第一行不对，先查公式和切片；前缀不对，先查 seq_len 和 max_len；列数不对，先查 d_model 和输入 embedding 是否一致。

这也是为什么这一页要先讲公式和形状，再讲实现状态。位置编码最常见的错误，是“看起来像在算，但前缀、频率和状态归属都偏了”。把这个小表跑通之后，再去看更大的模型结构，你就知道该把注意力放在哪些细节上。

你现在应该能做什么：拿到一段位置编码实现时，先用这个小案例查第 0 行、前缀切片和频率分层，不要等到整模型训练以后才发现位置感已经错位。

故障演算

check

这一节把“能算出来”推进到“能立刻看出哪里错”。还是用 d_model=8、max_len=4 这个小表，只盯四个位置：第 0 行看起点，第 1 行看频率分层，第 2 行看快慢差异，第 3 行看前缀切片是否仍然保持同一套规律。这个规模足够小，只需要确认公式、切片和状态迁移是否按预期发生。

先看值

当 pos=0 时，偶数列全是 sin(0)=0，奇数列全是 cos(0)=1。如果你打印出来看到第一行偏离 0 / 1 交替，先怀疑写入切片、三角函数顺序或者位置维和通道维是否广播反了。

接着看 pos=1、pos=2、pos=3。低编号列应该变化更快，高编号列应该更平缓；这就是“频率分层”的直觉。它的作用是让模型同时拥有短距离和长距离的顺序感。

再看错

如果快慢分层消失，通常是指数方向写反了，或者 div_term 的列步长错了。表面上你仍然能得到一张数值矩阵，但它已经不再携带“越往后越慢”的结构。

如果 seq_len 超过 max_len，问题会更直接：前缀切片拿不到足够的行。这个时候不要先怀疑 attention，也不要先怀疑 loss，先看缓存上限和输入管线是否比你想的更长。

检查对象正常现象异常症状先查哪里
pe[0]0 / 1 交替，起点固定第一行偏离 0 / 1 交替先查 sin / cos 和偶奇切片
pe[1]每对通道都变化，但速度不同所有列看起来同步摆动先查 div_term 是否按列分层
pe[3]低列更快，高列更慢快慢分层消失先查指数里的负号和 2i/d_model
seq_len > max_len提前被拦住，不继续前进前缀切片拿不到足够行先查缓存上限和数据管线长度
故障检查
import torch

def inspect_small_table(pe):
    table = pe[0, :4, :8].detach().cpu()
    for row in range(table.size(0)):
        values = [round(v, 4) for v in table[row].tolist()]
        print(f"row{row}", values)
输出怎么读 如果 row0 偏离 [0, 1, 0, 1, ...]，先看前面一层的公式和切片。若 row1 到 row3 的变化看起来都像同一种摆动，说明频率分层没建立起来；这时应该回去核对指数项和偶奇通道，再看下游 attention。只有当前三行都符合预期，你才值得把注意力放到训练配置上。

把这一步做成习惯以后，检查顺序就会非常稳定：先看起点是否对，再看快慢层次是否对，最后看切片是否只取前缀。这样你能很快区分三类问题：公式写反、状态保存错、输入长度超上限。它们表面上都像“模型不工作”，处理顺序却不同。

这段代码的目的在于把检查变成机械动作。先打印 row0，确认它接近 0 / 1 交替；再看后面几行，确认变化速度是否分层；最后确认你取的是前四行，并且没有把整张表重新生成一遍。只要其中一步不对，你就能把问题落回到公式、切片或状态迁移，别把责任推给整个模型。

把这个顺序再往前推一步，就能形成完整的排错链：row0 错，先查公式；row0 对但 row1 到 row3 乱，先查频率分层和偶奇切片；表本身对但整体输出仍乱，才去看 embedding、mask 或后续层。这样你不会把本该在输入端拦住的问题，拖到训练日志里才发现。

它也和下面的练习题连着：如果你能说清 pe[:, 0::2]、pe[:, 1::2]、seq_len、max_len 这四个点，选择题就能顺着同一条检查链自己推出来。你现在应该能做什么：拿到任意一段位置编码实现，先按这个顺序检查第一行、前缀、频率分层和设备，再去做练习区的题目。

把错误缩到一行

假设你把 pe[:, 0::2] 和 pe[:, 1::2] 写反，pos=0 的第一行还可能看起来像 0 / 1 交替，因为两组通道都还在表里。真正被破坏的是后面几行：偶数列和奇数列的节奏会对调，pe[1]、pe[2]、pe[3] 放在一起时，层次感会变乱。

这类问题的后果很实在。模型 forward 还能返回张量，loss 也能继续往下算，但位置节奏已经错位，attention 读到的是混过的顺序信号。你越晚发现，越难把责任定位回输入端。

练习时就按同一条链走：先看 row0，再看 row1~row3，最后看 seq_len 和 max_len。如果这三处都对，位置编码实现基本站住；只要有一处偏了，先回看切片、频率和 buffer，不必先怀疑后面的层。

这个错例的价值在于，它把“看起来能跑”与“实际上已偏”分开了。前者只说明张量形状还活着，后者才决定顺序信息有没有进到模型里。把这两件事分开看，你就会更快找到问题入口，也更容易把排查顺序固定在输入端。这一步通常最省时间，也最容易定位源头，适合先做再看后层。

常见误区

debug

误区 1：位置编码只是装饰项

把它当作可有可无的加法项，会漏掉一个关键点：它直接决定 attention 能不能区分顺序。没有它，模型会更像只认识“集合”，难以稳定地区分“序列”。

误区 2：只看公式，不看实现状态

原论文给的是数学形式，代码里还要处理缓存、设备迁移和切片。很多 bug 来自 buffer 没跟模型走，或者序列长度超过了缓存上限。

你查错时先看哪里 先看 pe 的形状，再看它是否为 buffer，再看前缀切片是否和当前 seq_len 对齐。大多数位置编码报错都能在这三步里定位到。

你现在应该能做什么：别人给你一段“位置编码写法”，你能先问三件事——表有没有保存对、长度有没有切对、设备有没有跟对。
