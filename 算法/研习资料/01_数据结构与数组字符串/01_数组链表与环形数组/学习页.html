DATA STRUCTURE 01

从线性结构读懂数组、链表和环形数组

算法题里常见的数据结构并不是孤立知识点。数组、链表、队列、栈、双指针、滑动窗口，都围绕“元素排成一条线”这件事展开。真正要学的是：这条线放在连续内存里，还是用节点指针串起来；读写的位置固定吗；头尾会不会绕回去。

数组适合按下标快速读写，链表适合在已知节点附近改连接。环形数组则是在固定容量数组上模拟队列：尾部走到数组末端后，从下标 0 继续使用。理解这三者，后面看 622 设计循环队列、641 设计循环双端队列、数组双指针、滑动窗口时，状态变量就不再像凭空出现。

学完应该能做什么

  1. 说清数组和链表各自快在哪里、慢在哪里。
  2. 能解释为什么环形数组要用取模计算下标。
  3. 能画出队列中 front、rear、size 三个状态的关系。
  4. 遇到“头尾进出”的题，能判断是否适合队列或双端队列。
  5. 能写出固定容量循环队列的核心操作，并说清空、满边界。

数组：下标是最快的导航方式

array

数组的直觉很朴素：一排连续格子，每个格子有编号。你知道编号，就能直接拿到元素。这个“直接拿”就是随机访问，常写成 `nums[i]`。算法题中只要需要频繁根据位置读取、交换、比较，数组就很自然。

数组的代价也来自连续性。中间插入一个元素，后面的元素通常要整体后移；删除头部元素，后面也要补位。所以数组题常把“删除”改写成“用慢指针覆盖保留元素”，比如删除有序数组重复项，而不是每次真的删除。

复杂度判断时要分清访问和移动。访问 `nums[i]` 是 `O(1)`，但把很多元素整体搬家不是 `O(1)`。看到原地数组题，可以先问：我是不是只需要覆盖和交换，而不是改变数组物理长度。

判断句

如果题目强调下标、区间、原地交换、双指针，数组通常是第一选择。

链表：节点顺序靠连接维护

linked list

链表不是把元素放在连续格子里，而是每个节点保存值和“下一个节点在哪里”。你要访问第 k 个节点，必须从头一个一个走过去；但如果你已经站在某个节点旁边，改它的 `next` 指针就能插入或删除附近节点。

链表题的难点常常不是算法思想，而是指针更新顺序。比如删除节点时，先保存后继，再改变连接；反转链表时，`next`、`prev`、`cur` 三个变量不能乱。因为一旦某条连接被覆盖，后面的节点可能就找不回来了。

学习链表时可以少背结论，多追踪“谁指向谁”。画图非常有用：每改一行 `next`，就在图上重新连线。你应该能说清每个变量当前指向哪个节点。

判断句

如果题目要求频繁在已知节点附近插入、删除或重连，链表比数组更贴近操作本身。

环形数组：固定空间里复用头尾

circular

环形数组解决的是队列空间浪费问题。普通数组从左出队后，前面的格子空了，但尾指针继续往右走，很快就到末尾。环形数组让尾指针走到末尾后回到开头，只要那里已经不属于队列，就可以继续写。

核心公式是 `(index + 1) % capacity`。取模不是炫技，而是把“越过末尾”变成“回到 0”。如果容量是 5，下标 4 的下一个位置就是 `(4 + 1) % 5 = 0`。

空和满不能只靠 `front == rear` 判断，因为空队列和满队列都可能出现头尾相等。常见做法是额外维护 `size`，或者浪费一个格子让满队列保持一个空位。初学时推荐用 `size`，状态更直接。

判断句

只要头尾会反复移动，并且希望固定数组复用空间，就该想到环形数组。

状态变量：front、rear、size 分工不同

state

`front` 表示下一次出队从哪里读，`rear` 表示下一次入队往哪里写，`size` 表示当前有多少真实元素。三个变量分工不同，混用会带来边界错误。

入队时，先检查 `size == capacity`，满了就拒绝；没满就把元素写到 `rear`，然后移动 `rear` 并增加 `size`。出队时，先检查 `size == 0`，空了就拒绝；没空就读取 `front`，移动 `front` 并减少 `size`。

注意顺序。先写再移动，意味着 `rear` 始终指向空位；先读再移动，意味着 `front` 始终指向队首。只要这个约定稳定，代码就容易检查。

判断句

读循环队列代码时，先问每个指针指向“当前元素”还是“下一个空位”。

题目迁移：从结构动作读题

transfer

LeetCode 622「设计循环队列」几乎就是环形数组原题，641「设计循环双端队列」则把入队和出队扩展到两端。它们不是靠复杂算法取胜，而是靠状态约定写得稳定。

数组双指针题，比如 80 删除有序数组中的重复项 II、88 合并两个有序数组，也在利用数组的下标优势。链表题如 2 两数相加，则利用节点顺序模拟逐位处理。

迁移时不要只记题号。先读题目动作：是按位置读写，还是按连接重排；是只在两端操作，还是要扫描区间；容量固定还是动态增长。动作读清楚，结构选择自然很多。

判断句

题目看起来不同，只要底层动作相同，就能迁移同一套状态变量。

复杂度和边界：别把容器 API 当魔法

complexity

数组随机访问是 `O(1)`，链表按位置访问是 `O(n)`，环形数组入队出队在不扩容时是 `O(1)`。这些结论背后都有条件。比如 Python 的 list 尾部追加平均很快，但头部 `pop(0)` 需要整体移动元素。

边界优先检查空、满、单元素、绕回。队列题尤其容易在从满到空、从尾部绕回头部时出错。写完代码后，用容量 3 的小数组手动走几步，比直接上大样例更容易暴露问题。

空间复杂度也要说清楚。环形数组使用固定容量 `O(k)`，链表每个节点有额外指针，数组则主要占元素本身。面试里说复杂度时，最好把“操作复杂度”和“存储代价”分开。

判断句

复杂度结论必须带上操作前提：是访问、移动、插入、删除，还是扩容。

渐进例子与代码

code reading

这几段代码按从小到大的顺序安排。读代码时先看状态放在哪里，再看每一步读什么、写什么。

1. 用慢指针覆盖数组

删除类数组题不一定真的删除。慢指针指向下一次写入位置，快指针负责扫描原数组。这样可以避免频繁移动后续元素。

正确示例：用慢指针覆盖数组Python
def keep_positive(nums):
    slow = 0
    for fast, x in enumerate(nums):
        if x > 0:
            nums[slow] = x
            slow += 1
    return slow  # nums[:slow] 是保留下来的部分

这里 fast 只读，slow 只写。两者分工清楚，数组题就不会写成一边遍历一边删除。

2. 链表删除当前节点后的节点

链表改连接前要先确认目标节点存在。`node.next = node.next.next` 读的是后继的后继，写的是当前节点的 next。

正确示例：链表删除当前节点后的节点Python
def delete_next(node):
    if node is None or node.next is None:
        return
    node.next = node.next.next

如果先把 node 移走或没有判空，代码会在空指针处崩掉。

3. 循环队列核心操作

这里用 size 区分空和满。rear 指向下一次写入位置，front 指向下一次读取位置。

正确示例：循环队列核心操作Python
class CircularQueue:
    def __init__(self, k):
        self.data = [None] * k
        self.front = 0
        self.rear = 0
        self.size = 0

    def enQueue(self, value):
        if self.size == len(self.data):
            return False
        self.data[self.rear] = value
        self.rear = (self.rear + 1) % len(self.data)
        self.size += 1
        return True

    def deQueue(self):
        if self.size == 0:
            return False
        self.front = (self.front + 1) % len(self.data)
        self.size -= 1
        return True

这段代码的重点不是类语法，而是入队和出队都只改一个指针和 size。

4. 漏掉 size 的循环队列

这段代码只用 front == rear 判断空，后面会分不清满和空。

错误示例：漏掉 size 的循环队列Python
def is_empty(front, rear):
    return front == rear

def is_full(front, rear):
    return front == rear  # 错：和空队列条件一样

修法是维护 size，或者保留一个空位让满队列条件变成 (rear + 1) % capacity == front。

常见误区

debug

把数组删除写成反复 pop(0)

错因：只看到了删除语义，没看到头部删除会移动后续元素。

后果：复杂度可能从线性变成平方级。

修法：用慢指针覆盖，或换成 deque/链表等更合适结构。

链表改指针前不保存后继

错因：急着重连，覆盖了原来的 next。

后果：后半段节点可能丢失。

修法：反转或删除前先保存 `nxt = cur.next`。

front 和 rear 含义摇摆

错因：一会儿把 rear 当最后一个元素，一会儿当下一个空位。

后果：入队出队边界混乱。

修法：固定约定：front 指向队首，rear 指向下一次写入位置。

只用 front == rear 判断空满

错因：没有额外状态区分两种情况。

后果：满队列可能被误判为空。

修法：维护 size，或牺牲一个空格。

忘记取模

错因：把环形数组当普通数组推进。

后果：指针走到容量外，越界或浪费前方空位。

修法：移动指针统一写 `(idx + 1) % capacity`。

复杂度只背不解释

错因：记住数组快、链表慢，但没说是哪种操作。

后果：遇到插入删除题容易选错结构。

修法：按访问、插入、删除、移动四种动作分别判断。
