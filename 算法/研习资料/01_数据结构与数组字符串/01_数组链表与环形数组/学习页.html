同一组元素，三种看法

overview

同一组元素可以放在数组里看下标，也可以放在链表里看连接，还可以放在固定容量的队列里看头尾。这里讲的是三种不同的动作模型。

数组

数组的强项是随机访问和原地交换。只要你知道下标，就能直接拿到元素；只要你愿意覆盖，就能把不需要的元素挪出结果前缀。

链表

链表的强项是局部重连。只要你站在前驱节点旁边，改一次 next 就能插入或删除一段节点；代价是按位置走到第 k 个节点并不快。

结构读什么写什么常见慢点
数组下标、区间、值覆盖、交换、移动慢指针头部插入删除、中间大搬家
链表当前节点、后继节点、前驱节点next 指向、局部重连按位置随机访问
环形数组front、rear、size写入位置、读取位置、容量边界空满混淆、忘记取模

核心机制

mechanism

数组：慢指针负责写，快指针负责看

数组题里常见的做法是用覆盖代替删除。快指针负责扫描原数组，慢指针负责下一次写入位置。这样做的好处是：读写分工清楚，结果前缀始终连续。

数组覆盖示例复制

def keep_at_most_two(nums):
 slow = 0
 for fast in range(len(nums)):
 if slow < 2 or nums[fast] != nums[slow - 2]:
 nums[slow] = nums[fast]
 slow += 1
 return slow

这里 fast 只读，slow 只写。它和后面的链表前驱、循环队列写入位是一类思路：把“当前可用位置”固定下来，再让状态往前走。

链表：先保后继，再改连接

链表最容易丢东西的地方，是你把 next 改掉以后，原来后面那串节点就再也找不回来了。只要先保存后继，再重连，就能避免断链。遇到头结点附近的操作时，虚拟头结点能把“删头”和“删中间”写成同一种写法。

链表删除示例复制

def delete_after_prev(prev):
 if prev is None or prev.next is None:
 return
 nxt = prev.next
 prev.next = nxt.next

如果 prev 来自虚拟头结点，那么这段代码既能删中间节点，也能统一处理原始头结点的删除。

$$

next(i) = (i + 1)\bmod capacity,\qquad prev(i) = (i - 1 + capacity)\bmod capacity

$$

取模把“走到末尾”改写成“回到 0”，让固定容量的数组继续复用前面的空位。

$$

循环队列示例复制

class CircularQueue:
 def __init__(self, capacity):
 self.data = [None] * capacity
 self.front = 0
 self.rear = 0
 self.size = 0

 def enqueue(self, value):
 if self.size == len(self.data):
 return False
 self.data[self.rear] = value
 self.rear = (self.rear + 1) % len(self.data)
 self.size += 1
 return True

 def dequeue(self):
 if self.size == 0:
 return None
 value = self.data[self.front]
 self.front = (self.front + 1) % len(self.data)
 self.size -= 1
 return value
对象读什么写什么错了会怎样
fast原数组中的当前元素不写，只扫描一边读一边改，原始顺序会乱
slow当前结果前缀下一次写入位置写偏了就会漏元素或重复元素
front / rear队首读取位、队尾写入位轮流前移顺序混掉或越界

先看 front 还是 rear 在动，再看 size 是加还是减。你一旦把“读位”和“写位”搞混，空满边界就会连着错。

同一组元素的三次变化

example

拿 A、B、C 这组三个元素做锚点，数组、链表和环形数组会分别给出不同的改写方式。你只要跟住这组三次变化，就能把“怎么改”从抽象词变成具体动作。

数组里的 A、B、C

当你在数组里处理重复元素时，fast 负责把 A、B、C 一个个扫过去，slow 负责把需要保留的值写进结果前缀。题目最后看的是 nums[:slow] 这段结果前缀。

链表里的 A、B、C

当你在链表里删除某个节点时，真正要改的是前驱节点的 next。如果前面加了虚拟头结点，头结点也只是“前驱节点”的一种特例，写法会更统一。

手算三步复制

数组：fast 扫描 A、B、C，slow 只写保留项。
链表：先保存 nxt，再把 prev.next 改到 nxt.next。
环形数组：rear 写入后前移一格，走到末尾就取模回到 0。

同一组元素在三种结构里会被不同地读取和写入，但核心都一样：先把“当前可用位置”说清楚，再让状态往前走。

把同一件事放进三种结构里

operation

现在把目标收窄到一个具体动作：处理序列 A、B、C、D，让它变成“保留 A 和 C”，或者让它在容量 3 的队列里完成一次绕回。数组、链表和环形数组的写法不同，但都在回答同一件事：当前读哪里、当前写哪里、旧状态什么时候失效。

具体动作数组里怎么做链表里怎么做环形数组里怎么做最常见失误
保留某些元素用 slow 覆盖结果前缀让前驱节点跨过被删节点不做删除，更多是移动读写位真删、真搬，导致顺序和复杂度都乱
往中间插入一个值后移后半段，再写新值改两条连接就完成这类操作通常不优先放在这个结构里忘记先保存后继或后缀
让尾部继续接收新元素尾部下标顺着走，必要时另开空间尾节点的 next 指向新节点rear = (rear + 1) % capacity没有取模，或者没有判断满
让头部继续弹出旧元素把 fast 扫过的值丢到前缀后面头结点/前驱节点改连接front = (front + 1) % capacity把读位和写位混在一起
判断状态是否还能继续看结果前缀是否满足条件看当前节点还在不在链上看 size 是否等于 0 或 capacity只靠一个表面条件，空满或断链会混淆

同一段状态推进复制

这段桥的作用是让调试时立刻知道应该盯哪里：数组错了先看 slow 有没有写偏，链表错了先看 nxt 有没有提前丢，队列错了先看 rear 是否回绕、size 是否同步。

删链实验

debug

现在把链表删节点这件事缩成一个最小动作：给定一个前驱节点 prev，删掉它后面的节点。这个动作非常小，但它能把链表的关键机制全都暴露出来：先看到谁、先保存谁、改了哪条连接、旧节点什么时候失效。

场景先读谁改谁容易出的错正确修法
删中间节点前驱和后继前驱的 next把后继链断开先保存 nxt，再连过去
删头节点虚拟头结点后的第一个节点虚拟头结点的 next代码里多出一堆 if让虚拟头结点统一入口
删尾节点前驱前驱的 next = None误把尾巴之后的内容当作存在确认目标节点确实存在后再改连接
连续删除多个节点当前前驱和它的后继循环更新前驱的 next忘记前驱位置要继续停留在原地让前驱始终站在下一次删除之前

统一删链模板复制

这个模板怎么读

第一行守住输入条件，第二行拿到要删除的节点，第三行把前驱直接连到后继。重点是确认链表的连接关系在修改后仍然完整。

这个模板怎么错

如果你直接写 prev.next = prev.next.next，通常也能跑，但调试时很难说清中间状态。把 target 先保存下来以后，你能更清楚地回答：我刚读到的是谁、我刚改的是谁、谁从链上消失了。

这个实验和数组覆盖的思路很像：数组里你固定写入位，链表里你固定前驱位。差别只在于，数组改的是连续槽位，链表改的是指针连接。你只要把这点说稳，后面删重复、反转、合并、插入这些题，操作顺序就会比死记模板更牢。

对应到练习区，10 题会检查你是否先保存后继，16 题会检查你是否理解顺着节点走，19 和 22 题会检查你是否能把状态约定和纸面台账讲出来。

容量 3 的手算台账

walkthrough

把环形数组缩到容量 3，很多抽象就会变得很具体。你可以直接看出：什么叫写入位、什么叫读取位、什么时候会绕回、为什么空满不能只看头尾相等。

这张表在看什么

这张表要让你看到：rear 到末尾以后是怎么回到 0 的，空出来的位置怎么被重新用起来的，以及 size 怎么保证空和满不混淆。

和代码怎么对上

每一步入队，都对应“先写当前 rear，再移动 rear，再增加 size”；每一步出队，都对应“先读 front，再移动 front，再减少 size”。只要顺序反过来，状态就会开始漂。

这张台账还可以换成另一个检查方法：每走一步，就问自己三件事。第一，刚刚读取的是当前有效元素吗；第二，刚刚写入的是空位吗；第三，size 有没有同步变化。只要有一问答不上来，就说明你的状态约定还不稳。

容量 3 的好处在于，它能把绕回和空满都压缩到很短的步数里。容量太大时，错误常常被延后；容量 3 时，错位会很快暴露，所以适合拿来做纸面自检、面试口述和代码单测的第一组样本。

错位会长什么样

failure

只看 front == rear

错因是把两个不同状态压成了一个条件。后果是满队列和空队列会被判断成同一件事，程序要么拒绝入队，要么把空队列当成有元素。

不做取模

错因是把环形数组当成普通数组。后果是 rear 很快越界，或者前面明明有空位却根本用不上。

先修什么

先固定 front 的含义，再固定 rear 的含义，最后让 size 负责空满判断。只要三者职责分开，空满边界就不会互相抢活。

为什么这一步重要

后面遇到 622、641，题目不会替你定义状态。你必须自己把写入位、读取位和容量关系讲清楚，不然实现会一直在边界上抖。

如果你要把这页的内容变成可执行检查，最简单的方法就是拿容量 3 的小队列做三轮纸面测试：先只入队，看 rear 是否按取模前移；再做一次出队，看 front 是否只移动一次；最后故意再入队，看 size 是否能把满队列和空队列区分开。这个测试很短，但足够抓住绝大多数状态错误。

当你怀疑自己的实现有问题时，不要先看大样例。先把小样本画成状态表，再把每一步读写动作写成一句话。这样你会很快发现问题是在“写之前该不该判满”，还是在“读之后有没有忘记减 size”，或者是在“绕回时有没有忘记取模”。

把 A、B、C、D 走一遍

walkthrough

这一段只做一件事：拿同一组样本 A、B、C、D，先走数组覆盖，再走链表删节点，再走容量 3 的循环队列。三个结构都在管理“当前位置”，只是被管理的对象不一样。数组管理写入位，链表管理前驱和后继，循环队列管理读位、写位和剩余容量。

数组里怎么读这组样本

数组覆盖时，A、B、C、D 只是扫描顺序。你先看 fast 走到哪里，再看 slow 是否应该写入。真正要保留的是满足条件的那些位置。换成题目语言，就是“保留谁、覆盖谁、结果前缀多长”。

链表里怎么读这组样本

链表里 A、B、C、D 由连接关系区分，不靠下标。你一旦在 B 处改掉 prev.next，后面的链就要靠 nxt 继续追。这里最关键的检查点是：修改之后，C 和 D 还能不能从新的起点追到。

循环队列里怎么读这组样本

把容量固定成 3 以后，A、B、C 三次入队会把队列推满，D 再来时就必须先检查 size == capacity。这是数据不会被覆盖的底线。只看 front == rear 会把空和满混成一类，纸面样本很快就能把这个错误暴露出来。

这组样本要留下什么

读完这一段，你应该能把“扫描、保存后继、写入、回绕、判满”这五个动作说出来，并且知道它们分别对应哪个结构。下一次看到 A、B、C、D 这种短样本时，不要先想代码，把位置关系先摆正。

三种结构的最小动作复制

数组：fast 扫到 B 时决定是否把 B 写进 slow 所在的结果前缀。
链表：prev 站稳后，先记住 nxt，再把 prev.next 连到 nxt。
循环队列：A、B、C 入队后先看 size，再决定 D 是入队还是拒绝。

如果你把这段动作序列写成纸面检查，顺序应该是：先确认谁是当前读位，后确认谁是当前写位，再确认旧状态还能不能保留。这个顺序一乱，数组会写偏，链表会断链，循环队列会空满混淆。
