先抓主线

map

判断容器之前，先判断动作。题目如果在意最近还没处理完的东西，通常先想栈；如果在意最早进来但还没离开的东西，通常先想队列；如果两端都要进出，而且两个方向都不能慢，通常先想双端队列；如果容量固定，还要反复从头尾进出，就该想环形队列。

题面里常见的动作信号

“最近打开的括号先关闭”“撤销最后一步”“表达式从后往前算”这些词，基本都在提醒你看栈。相反，“按到达顺序处理”“最近 3000 毫秒内的请求”“排队买票”“逐层推进”这些词，更像队列。

先问一句

这题更像“最近未解决项”，还是“最早未处理项”？

先别追的方向

这一页先不把哈希表、单调栈、滑动窗口、前缀和、树图都拉进来。它们后面会出现，但当前只负责把受限操作这条线讲稳。先把读写端口和状态变量站住，再去扩展题型。

题目里看到的动作 优先结构 为什么
最近打开的括号要先关闭 栈 最新留下的问题，必须先处理。
最早进入的请求要先清掉 队列 时间顺序不能乱，过期项在队首最先出现。
两端都要进出，且两边都要快 双端队列 只靠单端会把一侧卡死。

先把词说准

terms

栈和队列的最低限度定义

栈是后进先出，英文常写成 LIFO。你只应该盯住栈顶，别把它当成可随便下标访问的数组。队列是先进先出，英文常写成 FIFO。你要盯住队首和队尾，而不是中间某个随机位置。

双端队列和环形队列

双端队列允许两端都进出。它不是“更花哨的队列”，而是给你两端都留了口。环形队列则是固定容量的数组队列：头尾指针绕回开头，空和满必须靠状态变量区分，不能只看指针相等。

术语 直觉 容易混的点
LIFO 最后进来的先出去。 不要和“最近访问”混成随机访问。
FIFO 最早进来的先出去。 不要用 `pop(0)` 把队列做成慢操作。
front / rear 队首读位、队尾写位。 空满判断不能只看它们是否相等。
size 当前真实元素数。 它负责把空和满分开。

同一串输入，三种走法

overview

同一串输入，放进栈、队列和环形队列里，动作会完全不同。栈只会回头看最近压入的项；队列只会顺着时间往前走；环形队列则要一边维护头尾，一边保证容量没有被覆盖。真正要记的，不是容器名字，而是读写顺序。

栈：只看栈顶 bottom A B C top

压入三次以后，下一次弹出的一定是 C。这里的关键不是“元素很多”，而是“只有一个出口”。

队列：只看队首 front A B C rear

入队三次以后，下一次出队的一定是 A。这里的关键不是“谁更重要”，而是“最早进来的先处理”。

环形队列：头尾回绕 front=2, rear=0, size=2

当指针走到末尾，就用取模回到开头。只要 `size` 还没满，前面的空位就能重新用起来。

结构 读什么 写什么 错了会怎样
栈 栈顶 栈顶附近 把中间元素当成可随便取用的位置。
队列 队首 队尾 顺序会乱，时间意义会断掉。
环形队列 front、rear、size 写位、读位、容量边界 空满混淆，或者尾部无法回绕。

栈：最近未解决项先出场

stack

为什么括号、回退和表达式常常先想到栈

栈最有用的地方，是它天然支持“最新那一项先处理”。括号匹配时，最新打开的左括号要先被右括号关闭；路径简化时，最新进入的目录要先被弹出；逆波兰表达式里，最新压入的操作数要先参与运算。它们都在处理“最近留下但还没解决”的东西。

这也是为什么栈题不要写成“我到处查一下再决定”，而是要只看栈顶。你一旦开始看别的位置，题意里的时间顺序就会被你自己打散。

先抓的判断句

看到嵌套、撤销、回退、最近匹配、表达式求值，先想栈。看到“最新出现的东西必须先解决”，栈通常就是第一选择。

有效括号骨架
def is_valid(s):
    pairs = {')': '(', ']': '[', '}': '{'}
    st = []
    for ch in s:
        if ch in pairs.values():
            st.append(ch)
        else:
            if not st or st[-1] != pairs[ch]:
                return False
            st.pop()
    return not st

( 空 压入左括号 继续
) ( 栈顶是否匹配 弹出栈顶
) 空 有没有可配对对象 立刻失败
{[]} 逐层入栈 只看最近那层 合法

队列：最早未处理项先离开

queue

为什么最近请求、排队和层序遍历常常先想到队列

队列的直觉是：先到的人先处理。只要时间在递增，过期项就一定会先出现在队首。像 933 最近的请求次数，新的时间戳从队尾加入，太旧的时间戳从队首删掉。队列做的事情不是“保存很多历史”，而是“保留还有效的那一段顺序”。

如果你把队列写成“随便删哪个都行”，那就已经偏题了。队列最重要的限制，恰恰是中间不能随便动。

先抓的判断句

看到按到达顺序处理、最早元素先离开、时间窗口、逐层推进，先想队列。队列不是为了更快地存，而是为了不破坏顺序。

RecentCounter 骨架
from collections import deque

class RecentCounter:
    def __init__(self):
        self.q = deque()

    def ping(self, t):
        self.q.append(t)
        while self.q and self.q[0] < t - 3000:
            self.q.popleft()
        return len(self.q)
时间戳 入队后队列 删掉什么 返回什么
1 [1] 没有过期项 1
100 [1, 100] 没有过期项 2
3001 [1, 100, 3001] 没有过期项 3
3002 [100, 3001, 3002] 删掉 1 3

双端队列：两边都要快

deque

它解决的不是“更高级”，而是“两端都要动”

双端队列让你可以从两端进出。它常常出现在两类问题里：一类是两端都可能插入或删除的设计题，另一类是需要把候选项保存在队首或队尾的窗口题。像 239 滑动窗口最大值，就会用双端队列保存候选下标；像 641 设计循环双端队列，则是两端都要支撑插入和删除。

什么时候别硬上双端队列

如果题目只需要一端进、一端出，普通队列或栈通常更自然。双端队列不是默认答案，它只是给两端都要高频操作的场景留了口。

deque 操作口
from collections import deque

dq = deque()
dq.append(x)       # 右端进
dq.appendleft(y)   # 左端进
dq.pop()           # 右端出
dq.popleft()       # 左端出
操作 作用 常见用途
append / popleft 右进左出 普通队列语义
appendleft / pop 左进右出 反向构造、回退类操作
两端都可进出 边界灵活 窗口候选、双端设计题

环形队列：固定容量里回绕

circular

为什么要有环形队列

普通数组队列如果一直出队，前面的空位就浪费掉了。环形队列把数组首尾接起来，用取模让 `rear` 和 `front` 到达末尾后自动回到开头。这样一来，固定容量也能重复利用前面的空位。

它的关键不是“绕一圈很好看”，而是“空满不能靠头尾相等来猜”。如果你只看 `front == rear`，空和满会撞在一起，题就会乱。

空满必须分开看

环形队列通常用 `size` 单独记录当前元素数。空看 `size == 0`，满看 `size == capacity`。`front` 和 `rear` 只负责位置，不负责真假判断。

$$ next(i) = (i + 1) \bmod capacity $$

再配上 `empty = size == 0`、`full = size == capacity`。这三件事分开，空满就不会打架。

循环队列骨架
class MyCircularQueue:
    def __init__(self, capacity):
        self.data = [None] * capacity
        self.front = 0
        self.rear = 0
        self.size = 0

    def enQueue(self, value):
        if self.size == len(self.data):
            return False
        self.data[self.rear] = value
        self.rear = (self.rear + 1) % len(self.data)
        self.size += 1
        return True

    def deQueue(self):
        if self.size == 0:
            return False
        self.front = (self.front + 1) % len(self.data)
        self.size -= 1
        return True
操作 front rear size 读写动作 错位症状
初始 0 0 0 还没有元素 空满都不能乱判。
入队 不变 前移一格 +1 先写后移 rear 含义摇摆就会写错位。
出队 前移一格 不变 -1 先读后移 front 指向错位置就会读错元素。
绕回 按取模移动 按取模移动 按真实元素数变化 把末尾折回开头 不取模就会越界或浪费空间。

把题目翻译成动作

bridge

这页最值钱的地方，不是记住某个容器，而是能把题目翻译成动作。像 225 和 232 这种题，表面上是在“用一个容器实现另一个容器”，实际上是在把 FIFO 和 LIFO 的限制搬到别的结构上。像 622 和 641，则是在固定容量里维持头尾位置和回绕边界。像 155 和 933，则是在栈顶或队首之外再同步一个额外状态。

一个小桥：两个栈实现队列

输入栈负责接收新元素，输出栈负责对外弹出。当输出栈为空时，再把输入栈倒过去。这样做的好处是，平时只要正常压入，只有在必须出队时才搬运一次，FIFO 的顺序就被保住了。

判断顺序

先看题目要不要保住顺序；再看需要从哪一端取；最后才决定用一个栈、两个栈、一个队列、两个队列，还是环形数组。

两个栈实现队列
class MyQueue:
    def __init__(self):
        self.in_stack = []
        self.out_stack = []

    def push(self, x):
        self.in_stack.append(x)

    def _move(self):
        if not self.out_stack:
            while self.in_stack:
                self.out_stack.append(self.in_stack.pop())

    def pop(self):
        self._move()
        return self.out_stack.pop()

    def peek(self):
        self._move()
        return self.out_stack[-1]

150 逆波兰表达式 先弹右操作数，再弹左操作数 把 `a - b` 写反。

完整走一轮

cases

这一节专门把“结构名”拆成“具体动作”。你不用背更多题号，只要跟着状态变化走一遍，就能看出每个结构到底在保护什么顺序。

225：两个队列怎么拼出一个栈外观

225 的核心，不是“两个队列真的更高级”，而是把 FIFO 的输入和输出拆开。一个队列负责持续接收新元素，另一个队列负责对外弹出。只有当输出队列空了，才把输入队列倒过去。这样一来，外部看到的顺序仍然像栈一样，是最后压入的先被取走。

如果你在做题时把两个队列都当成普通缓冲区来用，或者每次都来回搬运，那你实际上还没有把 LIFO 关系保住。真正要保护的是“新元素先留在输入侧，输出侧只负责最后的弹出”。

232：两个栈怎么拼出一个队列外观

232 刚好相反：输入栈负责接新元素，输出栈负责对外弹出。`push` 时直接压入输入栈；`pop` 或 `peek` 时，如果输出栈为空，就把输入栈逐个倒过去。这样一倒，最早进入的元素就会跑到输出栈顶，FIFO 的顺序就保住了。

这里最容易错的地方，是把“倒一次”写成“每次都倒”，那样就失去了摊还意义。你要记的是：只有在输出栈空的时候，才把输入栈搬过去。

题目 输入状态 输出状态 关键动作 如果写错
225 in 队列收到 A、B、C out 队列为空 只有在 out 空时才把 in 倒过去 顺序会乱，栈外观不成立
225 in 队列继续收新元素 out 队列还保留旧顺序 先出老元素，再补新元素 会把新元素提前弹出
232 in 栈压入 A、B、C out 栈为空 需要对外弹出时才搬运 FIFO 被打断
232 in 栈继续接收新值 out 栈维持当前对外顺序 只在必要时搬运一次 每次都搬，复杂度和思路都会漂

622：固定容量里如何看出“还可以继续用”

622 的关键不是“能不能写进去”，而是“写进去以后指针怎么变”。容量为 3 时，先入队 A、B、C，再出队一次，front 会前移；再入队 D，rear 会绕回开头。你要盯住的不是数组里还有没有空格，而是 `size` 是否已经把真实元素数记对了。

如果你把 `front == rear` 当成唯一判断，622 会在空和满之间来回撞车。更稳的做法是把空和满交给 `size`，把位置交给 `front` 和 `rear`。

155：最小值为什么不能只记一个

155 的问题很典型：你不只是要知道“现在最小是多少”，还要知道“这个最小值一旦弹出，前一个最小值是什么”。如果只存一个全局最小值，弹出最小元素后，栈就失去历史了。辅助栈或者“每个节点同时保存当前最小值”就是用来保历史的。

这类题本质上是在主容器之外再维护一条状态线。主容器管顺序，辅助状态管答案。

最小栈的同步思路
class MinStack:
    def __init__(self):
        self.data = []
        self.mins = []

    def push(self, x):
        self.data.append(x)
        if not self.mins or x <= self.mins[-1]:
            self.mins.append(x)

    def pop(self):
        x = self.data.pop()
        if x == self.mins[-1]:
            self.mins.pop()
        return x

    def getMin(self):
        return self.mins[-1]
操作 data 栈 mins 栈 当前最小值 故障提醒
push 3 [3] [3] 3 历史还很短，但已经能回答最小值。
push 5 [3, 5] [3] 3 较大的值不需要进 mins。
push 2 [3, 5, 2] [3, 2] 2 新最小值必须同步进入辅助线。
pop 2 [3, 5] [3] 3 如果不弹辅助栈，最小值会卡住。

232 的一次完整转移

把 `push A、push B、push C、pop` 当成一组连续动作来看，最容易看出双栈怎么保序。前面三次 push 只是把元素依次压进输入栈，不做搬运；真正要 pop 时，才把输入栈一次性倒到输出栈。倒完以后，输出栈顶正好是 A、B、C 里最早进来的那个，`pop` 取到的就是 A。

这个过程说明了一件事：双栈方案真正依赖的不是“两个栈本身很神奇”，而是把“接收新元素”和“对外弹出”拆成两个阶段。只要你在每次 push 时都搬运，或者在输出栈没空时也重复倒栈，FIFO 就会被你自己打乱。

步骤 in_stack out_stack 当前可见顺序 这一步最该看什么
push A [A] [] A 还没对外可见 输入侧只负责接收
push B [A, B] [] 仍然只是输入顺序 别提前搬运
push C [A, B, C] [] 对外顺序还没改写 保持输入栈清晰
第一次 pop 前搬运 [] [C, B, A] 输出栈顶就是 A 只在输出栈空时搬运一次
pop A [] [C, B] 下一次可见的是 B 输出栈负责对外弹出

失败实验室

failure

这一节专门看“写法看着对，状态却已经跑偏”的情况。容器题的错误往往不是语法错，而是你把顺序、边界和辅助状态拆开写了。下面用三个小故障，把最常见的偏差讲清楚。

故障一：933 少了一次判过期

症状是某些时间点返回的数量比答案大。原因通常是你只删了一次过期请求，没有继续把队首中更早的请求清掉。因为时间戳是递增的，过期项会连续堆在队首，必须用 while 一直删到合法为止。

故障二：622 把空满判断写成同一个条件

症状是前面还没真正空掉，后面却被你当成空队列；或者已经满了，你还继续写入。原因是只盯着 front 和 rear，没把 size 单独记住。固定容量队列里，位置和容量必须拆开看。

故障三：155 只记一个全局最小值

症状是最小值在一次弹栈后卡住，不再往回走。原因是你只保留了“现在最小是多少”，没有保留“上一个最小是谁”。最小栈需要一条辅助状态线，才能在 pop 后恢复历史。

修复顺序

先问自己删的是谁，再问这个状态是不是还会被后面的步骤继续依赖。能被后续步骤读取的状态，就不能只靠一次覆盖解决。要么把它同步到辅助结构，要么把它写进每个节点或每个时间点的状态里。

933 的常见错误
# 错：只删一次
q.append(t)
if q and q[0] < t - 3000:
    q.popleft()

# 对：删到合法
q.append(t)
while q and q[0] < t - 3000:
    q.popleft()
题目 先看什么 常见误判 修法
933 最近请求次数 队首是否还有过期项 只删一次，忘了继续删 用 while 清到合法
622 设计循环队列 空和满是否分开 只看 front == rear 单独维护 size
155 最小栈 历史最小值是否保留 只记当前最小值 辅助栈同步记录

622 的纸面单测怎么写

拿容量为 3 的样本做最小回归：先入队 A、B、C，再出队一次，再入队 D。你要在纸上同时看三条线：`front` 是否只在出队时前移，`rear` 是否在入队时回绕，`size` 是否从 0、1、2、3 再回到 2、3。这个样本足够短，但已经能把“写入位、读取位、空满判断”三件事一起测出来。

如果你只测连续入队，就看不出空位复用；如果你只测出队，就看不出尾部回绕；如果你只把 `front == rear` 当成判断条件，就看不出空和满为什么会撞车。把这组三步写成固定检查顺序，后面再做 641 或 225 的实现就不会乱。

把故障变成检查清单

做题时先对照这三个问题：当前弹出的元素是谁、这个状态还会不会被后面用到、有没有一条更早的历史状态要恢复。只要这三问答不顺，你的代码就该先改状态，再谈题号。

状态追踪

trace

把状态写成表，你会更容易看出自己是把“读位”改错了，还是把“写位”改错了。下面这组卡片只做三件事：看栈顶、看队首、看环形回绕。你可以直接拿它当草稿，去检查自己写题时有没有漏掉状态同步。

栈：只看顶端 bottom A B C top

压入三次以后，下一次弹出的一定是 C。栈的读写都围着顶端转，别去碰中间位置。

队列：只看队首 front 1 100 3001 3002 rear

新的时间戳从队尾进来，过期时间戳从队首出去。队列的顺序一乱，窗口就失真。

环形队列：指针回绕 front=2, rear=0, size=2

先出队一次，再入队一次，rear 就可能从末尾绕回开头。只要 `size` 正常，空位就能继续复用。

结构 读什么 写什么 最容易漏的检查
栈 栈顶 压入和弹出 弹出前先判空。
队列 队首 尾部加入、头部弹出 过期项要从最早那端移走。
环形队列 front / rear / size 读位、写位、容量边界 空满不能只看指针是否相等。

常见误区

debug

这类题最常见的问题，不是“没见过容器”，而是状态和顺序没对齐。先看症状，再看你漏了哪一步。

错误 会看到什么 先查哪里
用 list.pop(0) 模拟队列 样例能过，大数据变慢。 头部删除是不是触发了整体搬移。
把 front == rear 当成唯一空满判断 空和满撞在一起。 有没有单独记录 size。
先改指针，再保存旧值 后面的元素追丢了。 有没有先保留后继、再重连。
弹栈前不判空 空输入直接炸掉，或者非法输入被误判合法。 栈顶检查是不是放在了正确位置。
只换容器名字，不换出入顺序 225 / 232 一换题就乱。 你到底在模拟 LIFO 还是 FIFO。

先别误判

如果题目问的是查找、去重、键值映射，下一页更像哈希表；如果题目问的是连续区间里怎么移动边界，后面更像滑动窗口。别把所有顺序问题都硬塞给栈和队列。

open队列

8. 为什么最近请求次数适合用队列，不适合用栈？

回答时至少写到：时间递增、过期项在队首、顺序不能乱。

参考答案 / 评分要点

请求时间是递增的，最早进入窗口的请求会先过期，所以应该从队首移除。队列正好保留到达顺序，新的请求从队尾进，过期请求从队首出。栈会把最新请求先弹出，顺序就错了。
