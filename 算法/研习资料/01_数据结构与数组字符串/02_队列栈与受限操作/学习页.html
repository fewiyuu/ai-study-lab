用受限操作理解队列和栈

队列和栈的共同点是“只允许在特定端口操作”。栈只看栈顶，后进先出；队列从尾部进入、从头部离开，先进先出。正因为操作受限，它们特别适合描述括号匹配、表达式求值、请求计数、层序处理等过程。

labuladong 题单里的 20 有效括号、150 逆波兰表达式求值、225 用队列实现栈、933 最近的请求次数、155 最小栈，都不是单纯考 API 名字，而是在考你能不能把题目流程改写成“压入、弹出、查看顶端或队首”。

栈：只处理最近留下的问题

stack

栈的直觉是“最后放进去的，最先拿出来”。这个限制让它特别适合处理嵌套结构：最近打开的括号，必须最先关闭；最近进入的目录标记，遇到返回时最先取消；表达式里最近出现的操作数，先被运算符取走。

栈的核心操作通常只有三个：压入 `push`、弹出 `pop`、查看栈顶 `top`。写题时不要把栈当作可以随便访问的数组。虽然 Python list 支持下标，但栈题的思维应该围绕顶端展开。

判断一个题适不适合栈，可以看它是否存在“最近未解决项”。如果新信息要和最近的旧信息配对、合并或取消，栈通常很合适。

判断句

看到嵌套、最近匹配、回退、撤销、表达式求值，先想栈。

队列：按时间顺序处理未完成对象

queue

队列的直觉是“先来先处理”。它适合保存还没处理完的对象，而且处理顺序不能乱。请求统计、层序遍历、排队买票，都依赖这个顺序。

队列的关键不是能不能装元素，而是队首代表最早还有效的元素。比如 933 最近请求次数，新的请求从队尾加入，太旧的请求从队首弹出。因为时间递增，过期的一定先出现在队首。

如果你发现题目一直在维护一个时间窗口或处理队伍，队列比栈更自然。栈会把最近元素先拿走，反而破坏时间顺序。

判断句

看到按到达时间处理、最早元素先离开、层层推进，先想队列。

受限操作带来的不变量

invariant

栈和队列好用，是因为它们帮你维护不变量。不变量是程序执行过程中一直成立的规则。比如括号栈里只保存还没匹配的左括号；请求队列里只保存当前时间窗口内的请求。

写题时可以先说不变量，再写代码。20 有效括号的不变量是：栈从底到顶保存尚未匹配的左括号。读到右括号时，栈顶必须是对应左括号。933 的不变量是：队列里所有时间都在最近 3000 毫秒内。

如果代码写乱了，回到不变量检查。每次 push/pop 之后，不变量还成立吗？如果不成立，说明你弹错对象、弹早了，或少弹了。

判断句

能说清容器里始终保存什么，才算真正会用栈和队列。

API 选择：list、deque 和手写结构

api

Python 里，栈常用 list：`append` 压栈，`pop` 弹栈，都是尾部操作。队列最好用 `collections.deque`，因为它支持 `popleft`，不会像 list 的 `pop(0)` 那样移动后续元素。

有些设计题要求你自己实现结构，比如 225 用队列实现栈、622 设计循环队列。这时重点不是调用库，而是模拟受限操作。用队列实现栈，需要把新元素调整到队首或在弹出前搬运前面的元素。

不要把 API 当作答案。你要解释为什么这个 API 的端口和题目动作一致。只会背 `deque`，但说不清从哪端进、从哪端出，遇到变体还是会乱。

判断句

选择容器时，看需要的是尾部操作、头部操作，还是两端操作。

problems

复杂度与边界

complexity

栈的 push/pop 通常是 `O(1)`。队列如果用 deque，头尾进出也是 `O(1)`。但如果用 list 做队列头部弹出，复杂度会变差。面试说复杂度时，要把底层容器说明清楚。

边界包括空栈弹出、右括号来时没有左括号、表达式运算数不足、队列窗口过期删到空、请求时间是否递增。很多题的错误不是大思路错，而是没有在操作前检查容器是否为空。

写代码时让非法状态尽早返回。比如括号题遇到不匹配立即 `False`，逆波兰题遇到运算符就必须有两个操作数。越早建立边界判断，后面代码越短。

判断句

每次弹出前先确认容器不空，每次返回前确认不变量满足题意。

渐进例子与代码

这几段代码按从小到大的顺序安排。读代码时先看状态放在哪里，再看每一步读什么、写什么。

1. 有效括号的栈模板

右括号只检查最近的左括号，所以读取栈顶即可。

正确示例：有效括号的栈模板Python
def is_valid(s):
    pairs = {")": "(", "]": "[", "}": "{"}
    stack = []
    for ch in s:
        if ch in pairs.values():
            stack.append(ch)
        else:
            if not stack or stack[-1] != pairs[ch]:
                return False
            stack.pop()
    return not stack

2. 最近请求次数的队列模板

新请求追加到队尾，过期请求从队首删掉。队列里始终只保留窗口内请求。

正确示例：最近请求次数的队列模板Python
from collections import deque

class RecentCounter:
    def __init__(self):
        self.q = deque()

    def ping(self, t):
        self.q.append(t)
        while self.q and self.q[0] < t - 3000:
            self.q.popleft()
        return len(self.q)

3. 逆波兰表达式求值

运算符处理最近两个数字。注意减法和除法有左右顺序，不能随便交换。

正确示例：逆波兰表达式求值Python
def eval_rpn(tokens):
    stack = []
    for token in tokens:
        if token not in {"+", "-", "*", "/"}:
            stack.append(int(token))
            continue
        b = stack.pop()
        a = stack.pop()
        if token == "+": stack.append(a + b)
        elif token == "-": stack.append(a - b)
        elif token == "*": stack.append(a * b)
        else: stack.append(int(a / b))
    return stack[-1]

常见误区

debug

把栈当队列使用

错因：没有抓住最近匹配关系。

后果：嵌套括号、表达式求值顺序会错。

修法：遇到右括号或运算符，只看栈顶最近元素。

用 list.pop(0) 模拟大队列

错因：忽略头部删除会搬移元素。

后果：大量请求或层序题会变慢。

修法：用 `deque.popleft()` 或环形数组。

弹出前不判空

错因：默认一定有可匹配元素。

后果：空栈报错，或把非法输入误判合法。

修法：每次 pop/top 前检查容器是否为空。

逆波兰减法顺序写反

错因：弹出两个数后没有区分 a 和 b。

后果：`a - b` 被写成 `b - a`。

修法：先弹出的记作右操作数 b，再弹出的记作左操作数 a。

请求窗口删错边界

错因：把 `< t - 3000` 写成 `<= t - 3000`。

后果：刚好在窗口左端的请求被误删。

修法：题目区间是 `[t-3000, t]`，只删除小于左端的请求。

最小栈只保存一个全局最小值

错因：没有考虑弹出最小值后如何恢复前一个最小值。

后果：pop 后最小值失真。

修法：同步维护辅助栈，或每个元素携带当前最小值。
