队列栈与受限操作

这一页不先背 API。先把“只能从哪一端读写”“谁先离开”“哪些状态变量必须同步”说清楚。栈看最近未解决项，队列看最早未处理项，双端队列看两边都要进出，环形队列看固定容量如何回绕。你把这些动作分清，后面的题号才有地方落。

先抓主线

map

判断容器之前，先判断动作。题目如果在意最近还没处理完的东西，通常先想栈；如果在意最早进来但还没离开的东西，通常先想队列；如果两端都要进出，而且两个方向都不能慢，通常先想双端队列；如果容量固定，还要反复从头尾进出，就该想环形队列。

先别追的方向

这一页先不把哈希表、单调栈、滑动窗口、前缀和、树图都拉进来。它们后面会出现，但当前只负责把受限操作这条线讲稳。先把读写端口和状态变量站住，再去扩展题型。

题目里看到的动作优先结构为什么
最近打开的括号要先关闭栈最新留下的问题，必须先处理。
最早进入的请求要先清掉队列时间顺序不能乱，过期项在队首最先出现。
两端都要进出，且两边都要快双端队列只靠单端会把一侧卡死。
固定容量里反复头尾进出环形队列用取模把末尾折回开头，空间才能继续复用。

先把词说准

terms

栈和队列的最低限度定义

栈是后进先出，英文常写成 LIFO。你只应该盯住栈顶，别把它当成可随便下标访问的数组。队列是先进先出，英文常写成 FIFO。你要盯住队首和队尾，而不是中间某个随机位置。

双端队列和环形队列

双端队列允许两端都进出。它给你两端都留了口。环形队列则是固定容量的数组队列：头尾指针绕回开头，空和满必须靠状态变量区分，不能只看指针相等。

术语直觉容易混的点
LIFO最后进来的先出去。不要和“最近访问”混成随机访问。
FIFO最早进来的先出去。不要用 pop(0) 把队列做成慢操作。
front / rear队首读位、队尾写位。空满判断不能只看它们是否相等。
size当前真实元素数。它负责把空和满分开。

同一串输入，三种走法

overview

同一串输入，放进栈、队列和环形队列里，动作会完全不同。栈只会回头看最近压入的项；队列只会顺着时间往前走；环形队列则要一边维护头尾，一边保证容量没有被覆盖。真正要记的，读写顺序。

栈：最近未解决项先出场

stack

为什么括号、回退和表达式常常先想到栈

栈最有用的地方，是它天然支持“最新那一项先处理”。括号匹配时，最新打开的左括号要先被右括号关闭；路径简化时，最新进入的目录要先被弹出；逆波兰表达式里，最新压入的操作数要先参与运算。它们都在处理“最近留下但还没解决”的东西。

这也是为什么栈题不要写成“我到处查一下再决定”，而是要只看栈顶。你一旦开始看别的位置，题意里的时间顺序就会被你自己打散。

先抓的判断句

看到嵌套、撤销、回退、最近匹配、表达式求值，先想栈。看到“最新出现的东西必须先解决”，栈通常就是第一选择。

队列：最早未处理项先离开

queue

为什么最近请求、排队和层序遍历常常先想到队列

队列的直觉是：先到的人先处理。只要时间在递增，过期项就一定会先出现在队首。像 933 最近的请求次数，新的时间戳从队尾加入，太旧的时间戳从队首删掉。队列做的事情“保留还有效的那一段顺序”。

如果你把队列写成“随便删哪个都行”，那就已经偏题了。队列最重要的限制，恰恰是中间不能随便动。

先抓的判断句

看到按到达顺序处理、最早元素先离开、时间窗口、逐层推进，先想队列。队列为了不破坏顺序。

RecentCounter 骨架复制

from collections import deque

class RecentCounter:
 def __init__(self):
 self.q = deque()

 def ping(self, t):
 self.q.append(t)
 while self.q and self.q[0] < t - 3000:
 self.q.popleft()
 return len(self.q)

双端队列：两边都要快

deque

它解决的“两端都要动”

双端队列让你可以从两端进出。它常常出现在两类问题里：一类是两端都可能插入或删除的设计题，另一类是需要把候选项保存在队首或队尾的窗口题。像 239 滑动窗口最大值，就会用双端队列保存候选下标；像 641 设计循环双端队列，则是两端都要支撑插入和删除。

什么时候别硬上双端队列

如果题目只需要一端进、一端出，普通队列或栈通常更自然。双端队列不是默认答案，它只是给两端都要高频操作的场景留了口。

deque 操作口复制

from collections import deque

dq = deque()
dq.append(x) # 右端进
dq.appendleft(y) # 左端进
dq.pop() # 右端出
dq.popleft() # 左端出
操作作用常见用途
append / popleft右进左出普通队列语义
appendleft / pop左进右出反向构造、回退类操作
两端都可进出边界灵活窗口候选、双端设计题

环形队列：固定容量里回绕

circular

为什么要有环形队列

普通数组队列如果一直出队，前面的空位就浪费掉了。环形队列把数组首尾接起来，用取模让 rear 和 front 到达末尾后自动回到开头。这样一来，固定容量也能重复利用前面的空位。

它的关键“空满不能靠头尾相等来猜”。如果你只看 front == rear，空和满会撞在一起，题就会乱。

空满必须分开看

环形队列通常用 size 单独记录当前元素数。空看 size == 0，满看 size == capacity。front 和 rear 只负责位置，不负责真假判断。

$$

next(i) = (i + 1) \bmod capacity

$$

再配上 empty = size == 0、full = size == capacity。这三件事分开，空满就不会打架。

循环队列骨架复制

class MyCircularQueue:
 def __init__(self, capacity):
 self.data = [None] * capacity
 self.front = 0
 self.rear = 0
 self.size = 0

 def enQueue(self, value):
 if self.size == len(self.data):
 return False
 self.data[self.rear] = value
 self.rear = (self.rear + 1) % len(self.data)
 self.size += 1
 return True

 def deQueue(self):
 if self.size == 0:
 return False
 self.front = (self.front + 1) % len(self.data)
 self.size -= 1
 return True
操作frontrearsize读写动作错位症状
初始000还没有元素空满都不能乱判。
入队不变前移一格+1先写后移rear 含义摇摆就会写错位。
出队前移一格不变-1先读后移front 指向错位置就会读错元素。
绕回按取模移动按取模移动按真实元素数变化把末尾折回开头不取模就会越界或浪费空间。

把题目翻译成动作

bridge

这页最值钱的地方，能把题目翻译成动作。像 225 和 232 这种题，表面上是在“用一个容器实现另一个容器”，实际上是在把 FIFO 和 LIFO 的限制搬到别的结构上。像 622 和 641，则是在固定容量里维持头尾位置和回绕边界。像 155 和 933，则是在栈顶或队首之外再同步一个额外状态。

判断顺序

先看题目要不要保住顺序；再看需要从哪一端取；最后才决定用一个栈、两个栈、一个队列、两个队列，还是环形数组。

两个栈实现队列复制

class MyQueue:
 def __init__(self):
 self.in_stack = []
 self.out_stack = []

 def push(self, x):
 self.in_stack.append(x)

 def _move(self):
 if not self.out_stack:
 while self.in_stack:
 self.out_stack.append(self.in_stack.pop())

 def pop(self):
 self._move()
 return self.out_stack.pop()

 def peek(self):
 self._move()
 return self.out_stack[-1]
题号核心动作常见第一次想错的地方
20 有效括号栈顶配对最近左括号把配对关系写成整串扫描，不看栈顶。
150 逆波兰表达式先弹右操作数，再弹左操作数把 a - b 写反。
225 / 232在别的容器里模拟 LIFO / FIFO只换容器名字，不换出入顺序。
622 / 641固定容量和回绕把 front == rear 当成唯一空满判断。
155 / 933同步额外状态只维护主容器，忘了维护最小值或时间窗口。

完整走一轮

cases

这一节专门把“结构名”拆成“具体动作”。你不用背更多题号，只要跟着状态变化走一遍，就能看出每个结构到底在保护什么顺序。

232：两个栈怎么拼出一个队列外观

232 刚好相反：输入栈负责接新元素，输出栈负责对外弹出。push 时直接压入输入栈；pop 或 peek 时，如果输出栈为空，就把输入栈逐个倒过去。这样一倒，最早进入的元素就会跑到输出栈顶，FIFO 的顺序就保住了。

这里最容易错的地方，是把“倒一次”写成“每次都倒”，那样就失去了摊还意义。你要记的是：只有在输出栈空的时候，才把输入栈搬过去。

题目输入状态输出状态关键动作如果写错
225in 队列收到 A、B、Cout 队列为空只有在 out 空时才把 in 倒过去顺序会乱，栈外观不成立
225in 队列继续收新元素out 队列还保留旧顺序先出老元素，再补新元素会把新元素提前弹出
232in 栈压入 A、B、Cout 栈为空需要对外弹出时才搬运FIFO 被打断
232in 栈继续接收新值out 栈维持当前对外顺序只在必要时搬运一次每次都搬，复杂度和思路都会漂

622：固定容量里如何看出“还可以继续用”

622 的关键“写进去以后指针怎么变”。容量为 3 时，先入队 A、B、C，再出队一次，front 会前移；再入队 D，rear 会绕回开头。你要盯住的size 是否已经把真实元素数记对了。

如果你把 front == rear 当成唯一判断，622 会在空和满之间来回撞车。更稳的做法是把空和满交给 size，把位置交给 front 和 rear。

155：最小值为什么不能只记一个

155 的问题很典型：你不只是要知道“现在最小是多少”，还要知道“这个最小值一旦弹出，前一个最小值是什么”。如果只存一个全局最小值，弹出最小元素后，栈就失去历史了。辅助栈或者“每个节点同时保存当前最小值”就是用来保历史的。

这类题本质上是在主容器之外再维护一条状态线。主容器管顺序，辅助状态管答案。

最小栈的同步思路复制

class MinStack:
 def __init__(self):
 self.data = []
 self.mins = []

 def push(self, x):
 self.data.append(x)
 if not self.mins or x <= self.mins[-1]:
 self.mins.append(x)

 def pop(self):
 x = self.data.pop()
 if x == self.mins[-1]:
 self.mins.pop()
 return x

 def getMin(self):
 return self.mins[-1]
操作data 栈mins 栈当前最小值故障提醒
push 3[3][3]3历史还很短，但已经能回答最小值。
push 5[3, 5][3]3较大的值不需要进 mins。
push 2[3, 5, 2][3, 2]2新最小值必须同步进入辅助线。
pop 2[3, 5][3]3如果不弹辅助栈，最小值会卡住。

232 的一次完整转移

把 push A、push B、push C、pop 当成一组连续动作来看，最容易看出双栈怎么保序。前面三次 push 只是把元素依次压进输入栈，不做搬运；真正要 pop 时，才把输入栈一次性倒到输出栈。倒完以后，输出栈顶正好是 A、B、C 里最早进来的那个，pop 取到的就是 A。

这个过程说明了一件事：双栈方案真正依赖的把“接收新元素”和“对外弹出”拆成两个阶段。只要你在每次 push 时都搬运，或者在输出栈没空时也重复倒栈，FIFO 就会被你自己打乱。

步骤in_stackout_stack当前可见顺序这一步最该看什么
push A[A][]A 还没对外可见输入侧只负责接收
push B[A, B][]仍然只是输入顺序别提前搬运
push C[A, B, C][]对外顺序还没改写保持输入栈清晰
第一次 pop 前搬运[][C, B, A]输出栈顶就是 A只在输出栈空时搬运一次
pop A[][C, B]下一次可见的是 B输出栈负责对外弹出

失败实验室

failure

这一节专门看“写法看着对，状态却已经跑偏”的情况。容器题的错误往往你把顺序、边界和辅助状态拆开写了。下面用三个小故障，把最常见的偏差讲清楚。

故障二：622 把空满判断写成同一个条件

症状是前面还没真正空掉，后面却被你当成空队列；或者已经满了，你还继续写入。原因是只盯着 front 和 rear，没把 size 单独记住。固定容量队列里，位置和容量必须拆开看。

故障三：155 只记一个全局最小值

症状是最小值在一次弹栈后卡住，不再往回走。原因是你只保留了“现在最小是多少”，没有保留“上一个最小是谁”。最小栈需要一条辅助状态线，才能在 pop 后恢复历史。

修复顺序

先问自己删的是谁，再问这个状态是不是还会被后面的步骤继续依赖。能被后续步骤读取的状态，就不能只靠一次覆盖解决。要么把它同步到辅助结构，要么把它写进每个节点或每个时间点的状态里。

933 的常见错误复制

# 错：只删一次
q.append(t)
if q and q[0] < t - 3000:
 q.popleft()

# 对：删到合法
q.append(t)
while q and q[0] < t - 3000:
 q.popleft()
题目先看什么常见误判修法
933 最近请求次数队首是否还有过期项只删一次，忘了继续删用 while 清到合法
622 设计循环队列空和满是否分开只看 front == rear单独维护 size
155 最小栈历史最小值是否保留只记当前最小值辅助栈同步记录

622 的纸面单测怎么写

拿容量为 3 的样本做最小回归：先入队 A、B、C，再出队一次，再入队 D。你要在纸上同时看三条线：front 是否只在出队时前移，rear 是否在入队时回绕，size 是否从 0、1、2、3 再回到 2、3。这个样本足够短，但已经能把“写入位、读取位、空满判断”三件事一起测出来。

如果你只测连续入队，就看不出空位复用；如果你只测出队，就看不出尾部回绕；如果你只把 front == rear 当成判断条件，就看不出空和满为什么会撞车。把这组三步写成固定检查顺序，后面再做 641 或 225 的实现就不会乱。

把故障变成检查清单

做题时先对照这三个问题：当前弹出的元素是谁、这个状态还会不会被后面用到、有没有一条更早的历史状态要恢复。只要这三问答不顺，你的代码就该先改状态，再谈题号。

状态追踪

trace

把状态写成表，你会更容易看出自己是把“读位”改错了，还是把“写位”改错了。下面这组卡片只做三件事：看栈顶、看队首、看环形回绕。你可以直接拿它当草稿，去检查自己写题时有没有漏掉状态同步。

常见误区

debug

这类题最常见的问题，状态和顺序没对齐。先看症状，再看你漏了哪一步。

先别误判

如果题目问的是查找、去重、键值映射，下一页更像哈希表；如果题目问的是连续区间里怎么移动边界，后面更像滑动窗口。别把所有顺序问题都硬塞给栈和队列。

后续衔接

next

什么时候这一页该结束

当你能从题目动作直接判断“用栈、用队列、用双端队列，还是用环形队列”，而且能把 front、rear、size 说清楚，这一页就够了。接下来该去看哈希表：那一页不再关心谁先谁后，而是关心键和值怎么查。

Q: 7. 为什么有效括号要先看栈顶，再决定能不能弹出？

参考答案 / 评分要点

。括号题处理的是嵌套关系，最近打开但还没匹配的左括号必须优先和当前右括号配对。栈顶正好保存了“最近未解决项”，所以先看栈顶能不能配对，再决定是否弹出。只看栈顶，才能保证配对顺序不乱。

Q: 8. 为什么最近请求次数适合用队列，不适合用栈？

参考答案 / 评分要点

。请求时间是递增的，最早进入窗口的请求会先过期，所以应该从队首移除。队列正好保留到达顺序，新的请求从队尾进，过期请求从队首出。栈会把最新请求先弹出，顺序就错了。
