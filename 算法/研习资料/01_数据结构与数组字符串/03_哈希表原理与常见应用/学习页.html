与常见应用哈希表把“找一个东西”变成按 key 定位。算法题里常见三种用法：存在性判断、计数/索引、按稳定 key 分组。

DATA STRUCTURE 03

把数据改造成可快速查询的结构

哈希表的直觉是：先为对象造一个 key，再把 key 放到能快速定位的位置。之后你不需要从头扫描，只要问“这个 key 有没有”“这个 key 对应什么”。这就是两数之和、最长连续序列、字母异位词分组这类题的共同底层。

这一页会讲原理，但目标不是手写工业级哈希表。你需要掌握的是刷题中的应用判断：`set` 做存在性，`dict` 做计数、索引和分组；key 必须稳定、可哈希、能表达题目里的“同一类”。

哈希表的直觉：先定位，再比较

principle

普通数组查某个值，只能一个个比较。哈希表先对 key 计算哈希值，把它映射到某个桶，再在桶里确认是不是同一个 key。平均情况下，桶里元素不多，所以查询接近 `O(1)`。

哈希冲突是指不同 key 算到了同一个桶。真实语言运行时会处理冲突，所以刷题时不必手写冲突解决，但要知道平均 `O(1)` 不是绝对保证。数据量大、查询频繁时，哈希表仍然是最常用的空间换时间方法。

理解这个过程后，你就知道为什么 key 必须可哈希且稳定。如果一个 key 放入表后自己变了，原来的定位就失效了。Python 里 list 可变，所以不能直接做 dict key。

判断句

只要题目需要反复查“某个身份是否出现过”，哈希表就是候选。

set：存在性判断，不关心附加信息

set

`set` 只保存 key 本身。你问的是“这个值在不在”。128 最长连续序列把数字放进 set，然后不断查 `num + 1`；219 存在重复元素 II 可以用 set 维护最近 k 个元素。

set 的好处是语义干净。如果你不需要次数、下标或分组，就不要把 dict 写得太复杂。清楚的数据结构会让边界也更清楚。

不过 set 不保存重复次数。遇到需要判断出现几次、窗口内某个字符数量够不够，就要换成 dict 或 Counter。

判断句

只问有没有，用 set；还问有几个、在哪、对应谁，就用 dict。

dict：key 到信息的映射

dict

dict 的 key 是查找身份，value 是你需要随 key 保存的信息。两数之和里，key 是数字，value 是下标；字符计数里，key 是字符，value 是出现次数；分组题里，key 是标准化身份，value 是列表。

写 dict 题时，要想清“先查后放”还是“先放后查”。两数之和常先查补数，再放当前数，避免同一个元素被用两次。计数题则通常先累加，再在后续判断里使用。

value 的选择决定代码结构。value 如果是下标，要考虑覆盖还是保留第一次出现；value 如果是次数，要考虑减到 0 后是否删除；value 如果是列表，要考虑第一次遇到 key 时如何初始化。

判断句

dict 不是一个答案，它是一种 key 到信息的关系。先定义关系，再写 API。

分组 key：同类对象必须得到同一个身份

key

分组题最容易错在 key。49 字母异位词分组中，`eat` 和 `tea` 原字符串不同，但排序后都是 `aet`，所以排序结果可以做 key。也可以用 26 个字母计数组成 tuple，避免排序成本。

好 key 要同时满足两个条件：同类对象 key 相同，不同类对象尽量 key 不同。还要可哈希，不能用 list 这种可变对象。

设计 key 时不要怕多字段。比如“同一天同一商品”的记录，key 可以写成 `(date, product_id)`。哈希表不要求 key 只能是一个基础值，只要它稳定且可哈希。

判断句

看到“归类、同组、等价”，先想如何把对象变成稳定 key。

维护时机：先查后放还是先放后查

timing

哈希表代码常见错误不是容器选错，而是更新时机错。两数之和如果先把当前数放入表，再查补数，当 target 是两倍当前数时，可能误用同一个下标。

滑动窗口或最近重复题里，放入新元素和删除旧元素的顺序也很重要。窗口代表的区间必须和哈希表内容一致，不能让哈希表多保留一个已经出窗的元素。

写代码前可以先用一句话规定：当前哈希表代表“已经处理过的元素”，还是“当前窗口内的元素”，还是“所有元素集合”。这句话就是检查时机的标尺。

判断句

每次查表前，先问表里代表的是哪个时间范围或哪个处理阶段。

复杂度：别漏掉造 key 的成本

complexity

哈希查询平均 `O(1)`，但造 key 可能不是 `O(1)`。49 如果每个字符串排序，单个 key 成本是 `O(k log k)`，总复杂度是 `O(nk log k)`。如果用 26 维计数，造 key 通常是 `O(k)`。

空间复杂度也要算。set 存所有数字是 `O(n)`，dict 存所有分组也是 `O(n)` 级别。哈希表常用空间换时间，不要只说“很快”而忽略额外内存。

面试表达里可以说“平均情况下”。如果题目或语言环境没有特殊恶意哈希，算法题通常接受平均 `O(1)` 查询。

判断句

哈希表让查找快，但总复杂度还要加上遍历和造 key 的成本。

渐进例子与代码

code reading

这几段代码按从小到大的顺序安排。读代码时先看状态放在哪里，再看每一步读什么、写什么。

1. 两数之和：先查补数再记录当前位置

表里保存已经看过的数到下标的映射。先查补数，可以避免同一个元素被重复使用。

正确示例：两数之和：先查补数再记录当前位置Python
def two_sum(nums, target):
    pos = {}
    for i, x in enumerate(nums):
        need = target - x
        if need in pos:
            return [pos[need], i]
        pos[x] = i
    return []

2. 最长连续序列：set 加起点判断

set 负责快速查询，起点判断负责避免重复扩展同一段。

正确示例：最长连续序列：set 加起点判断Python
def longest_consecutive(nums):
    nums_set = set(nums)
    best = 0
    for x in nums_set:
        if x - 1 in nums_set:
            continue
        cur = x
        length = 1
        while cur + 1 in nums_set:
            cur += 1
            length += 1
        best = max(best, length)
    return best

3. 字母异位词分组：排序 key

排序后字符串是稳定身份，同组词会进入同一个列表。

正确示例：字母异位词分组：排序 keyPython
from collections import defaultdict

def group_anagrams(strs):
    groups = defaultdict(list)
    for s in strs:
        key = "".join(sorted(s))
        groups[key].append(s)
    return list(groups.values())

4. 错误示例：用 list 做 key

`sorted(s)` 返回 list，list 可变，不能作为 dict key。

错误示例：错误示例：用 list 做 keyPython
groups = {}
for s in strs:
    key = sorted(s)
    groups.setdefault(key, []).append(s)  # TypeError: unhashable type: list

常见误区

debug

key 设计不能表达同类关系

错因：直接用原对象当 key，没有标准化。

后果：应该同组的对象分散到不同桶。

修法：排序、计数或提取多字段 tuple 作为稳定 key。

用可变对象做 key

错因：把 list、dict 直接放进 key 位置。

后果：Python 报不可哈希错误。

修法：list 转 tuple，排序结果 join 成字符串。

先放后查导致复用自己

错因：两数之和里更新时机错。

后果：同一元素可能被当成两个位置。

修法：先查补数，再记录当前值。

频次减到 0 不处理

错因：只会加减计数，不维护窗口语义。

后果：表里残留无效 key，判断 distinct 数量时出错。

修法：计数为 0 时删除 key，或判断时显式忽略 0。

把平均 O(1) 说成绝对 O(1)

错因：不知道哈希冲突存在。

后果：复杂度表达不严谨。

修法：说平均查询接近 O(1)，最坏情况可能退化。

漏掉造 key 的复杂度

错因：只看 dict 查询，不看排序或计数字符。

后果：49 这类题复杂度说错。

修法：把遍历长度和 key 构造成本一起算。
