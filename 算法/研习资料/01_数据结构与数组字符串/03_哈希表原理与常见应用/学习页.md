# 先抓主线 {#goals}
<!-- tag: map -->
<!-- layout: grid2 -->

## box
### 题目先翻成关系
哈希题不要先背 API。先看题目到底要的是存在性、计数、索引，还是分组。存在性题里，key 本身就够了；计数题里，value 记次数；索引题里，value 记位置；分组题里，value 记同类列表。关系看清了，容器选择就不会乱。
**先问一句**这题是只问“有没有”，还是还要知道“有多少、在哪里、属于谁”?

## box soft
### 先别追的方向
这一页先不扩到布隆过滤器、并查集、哈希冲突实现细节、滚动哈希这些后续话题。现在只抓刷题里最常见的三件事：怎么选 key、什么时候查、什么时候放。

| 题目动作 | 优先结构 | 理由 |
| --- | --- | --- |
| 判断某个值出现过没有 | set | 只要答案是“有 / 没有”，就不用保存额外信息。 |
| 统计出现次数 | dict / Counter | value 记录频次，后续可以增减。 |
| 找补数、找位置 | dict | key 定位身份，value 保存下标或别的信息。 |
| 归类到同一组 | dict + 稳定 key | 同类对象必须算成同一个 key，value 收集列表。 |

# 核心机制：先定 key，再决定 value {#mechanism}
<!-- tag: mechanism -->
<!-- layout: grid2 -->

## box
### key 是身份，value 是记录
哈希表最容易讲乱的地方，是把“key”和“value”混成一个整体。你先决定对象的身份，再决定要把什么信息挂在这个身份上。两数之和里，身份是数字本身，记录是下标；字母异位词分组里，身份是排序后字符串，记录是同组字符串列表。

## box soft
### 稳定 key 的要求
好 key 要满足两件事：同类对象 key 相同，不同类对象尽量不同。还要可哈希，不能在放进去之后自己变掉。`list` 这类可变对象就不适合直接做 key；`tuple`、字符串、整数通常更稳。

| 检查项 | 表里读什么 | 表里写什么 | 错了会怎样 |
| --- | --- | --- | --- |
| 存在性 | key 在不在 | 只记 key | 多存 value 只会增加复杂度。 |
| 计数 | 某个 key 出现了几次 | 次数加减 | 漏减到 0，窗口语义就会偏掉。 |
| 索引 | key 对应哪个位置 | 下标 | 先放后查可能复用到自己。 |
| 分组 | 同类对象应该去哪组 | 列表 | key 不稳时，本该同组的对象会被拆散。 |

# set 和 dict 各管什么 {#set-vs-dict}
<!-- tag: compare -->
<!-- layout: grid2 -->

## box
### set：只回答有没有
当题目只需要判断某个值是否出现过，`set` 最干净。128 最长连续序列把数组里的数放进集合，再检查 `x + 1` 是否存在，就能从“逐个扫描”变成“起点扩展”。219 这类题也常把最近若干个元素放进 `set` 里做重复检测。

## box
### dict：回答“key 对应什么”
`dict` 不是“更高级的 set”，它是带 value 的映射。value 可以是下标、次数、列表，也可以是别的结构。只要题目还要补充信息，就该从 set 走到 dict，而不是把所有信息都硬塞进一个布尔判断里。

| 需求 | 更像谁 | 典型写法 |
| --- | --- | --- |
| 有没有出现过 | set | seen = set(nums) |
| 出现几次 | dict / Counter | count[x] += 1 |
| 出现在哪 | dict | pos[value] = i |
| 属于哪一组 | dict | groups[key].append(item) |

# 分组 key：先标准化，再收集 {#group-key}
<!-- tag: grouping -->
<!-- layout: grid2 -->

## box
### 字母异位词为什么能放一起
49 字母异位词分组里，`eat` 和 `tea` 原字符串不同，但排序后都变成 `aet`，所以可以落到同一个 key。也可以用 26 个字母的计数元组做 key，避开排序成本。核心“同类对象必须得到同一个身份”。

## box soft
### 先检查 key，再检查容器
很多分组错key 的错。key 设计不稳时，应该同组的对象会被拆开；key 设计太粗时，不同组又会被并到一起。先把 key 设计对，再谈 value 的初始化和 append。

| 题目 | 可用 key | 为什么 |
| --- | --- | --- |
| 49 字母异位词分组 | 排序后的字符串 | 同组字符排列不同，但排序后相同。 |
| 49 的另一种写法 | 26 维计数元组 | 同组字符频次相同，不必排序。 |
| 用户 id + 日期 分组 | (user_id, date) | 多字段一起才构成身份。 |
| 错误示例 | list / 可变对象 | 不可哈希，或放进去后状态会变。 |

# 维护时机：先查后放，还是先放后查 {#timing}
<!-- tag: timing -->
<!-- layout: grid2 -->

## box
### 两数之和为什么要先查补数
两数之和里，表里保存的是“已经看过的数”。当前元素进来时，先查 `target - x` 是否已经出现，确认后再记录自己。这样才能避免把同一个元素当成两次使用。`nums=[3,3], target=6` 这种最小例子最容易看出顺序问题。

## box soft
### 窗口题为什么要同步删除
窗口题里，哈希表记录的是“当前窗口内的元素”。窗口右移时，离开的元素也要同步减计数或删除。否则表里的内容已经过期，但你还拿它做判断，结果就会偏。

| 情形 | 表代表什么 | 推荐顺序 | 如果顺序错了 |
| --- | --- | --- | --- |
| 两数之和 | 已处理元素 | 先查后放 | 可能复用当前下标。 |
| 频次统计 | 当前范围内元素 | 先加后查 | 统计值会落后或提前。 |
| 滑动窗口 | 当前窗口内元素 | 进窗与出窗同步 | 残留旧 key，判断失真。 |
| 分组题 | 同类列表 | 先生成稳定 key，再 append | 对象会进错组。 |

# 复杂度：别漏掉造 key 的成本 {#complexity}
<!-- tag: complexity -->
<!-- layout: grid2 -->

## box
### 查询平均快，不代表总过程都快
哈希表查询平均接近 `O(1)`，但前面可能还有遍历，后面可能还有造 key。49 如果对每个字符串做排序，单个 key 的成本就是 `O(k log k)`；如果用 26 维计数，造 key 常变成 `O(k)`。面试里只说“哈希表很快”不够，要把总成本拆开。

## box warn
### 空间也要算
set、dict、分组列表都要占额外内存。哈希题常用空间换时间，但不是没有代价。说清楚“平均查询快”“额外空间为 `O(n)` 左右”比一句笼统的“很快”更稳。

$$\text{总复杂度} = \text{遍历成本} + \text{查写哈希表成本} + \text{造 key 成本}$$
 这里的“查写哈希表成本”通常按平均 `O(1)` 处理；真正容易漏掉的是造 key 的那一段。

# 落题步骤：看到题就按这四步走 {#playbook}
<!-- tag: playbook -->
<!-- layout: grid2 -->

## box
### 第一步：定身份
先把题目里的对象标准化成一个 key。数字题通常直接用数字；分组题通常先排序或计数；多字段题用 tuple 把身份拼起来。只要这一步没定好，后面查表再快也会跑偏。

## box soft
### 第二步：定 value
再决定 value 存什么。只是判断有没有，value 甚至不需要；要找位置就存下标；要统计频次就存次数；要归类就存列表。value 不是随手塞进去的，它直接决定后面怎么读结果。

## box
### 第三步：定时机
再看是先查后放，还是先放后查。表代表已经处理过的元素，就先查后放；表代表当前窗口，就要同步进出；表代表计数，就要把加减顺序写清楚。这个步骤经常比容器本身更重要。

## box warn
### 第四步：算成本
最后把遍历、查写和造 key 的成本加起来。只看哈希查询平均 `O(1)`，会漏掉排序、拼元组、扫描字符这些工作。复杂度写错，不一定是思路错，很多时候只是少算了一步。

| 步骤 | 要做什么 | 最常见的坑 |
| --- | --- | --- |
| 定身份 | 把对象变成稳定 key | 原对象没标准化，导致同类对象进错桶 |
| 定 value | 决定记录下标、次数还是列表 | 只记 key，结果读不回需要的信息 |
| 定时机 | 安排查、放、删的顺序 | 先放后查，或者过期 key 没同步删掉 |
| 算成本 | 把遍历、查写、造 key 合起来看 | 只算查询，不算造 key |

# 把同一份输入追四遍 {#trace-lab}
<!-- tag: trace -->

同一串输入放进不同题型，哈希表里的状态会完全不同。下面用同一个小锚点追四遍：`[3, 3, 6]` 对应两数之和，`["eat", "tea", "ate"]` 对应分组，`["a", "b", "a"]` 对应计数，`[5, 6, 7]` 对应存在性。看这些例子时，别盯着题号，盯住表里究竟保存了什么。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

| 任务 | 读什么 | 写什么 | 错的时候先看哪一行 |
| --- | --- | --- | --- |
| 存在性 | key 是否存在 | key 本身 | 是否把 value 存进了根本不会读取的地方 |
| 索引 | 补数是否已出现 | 当前位置 | 查找顺序是否已经反过来 |
| 计数 | 当前范围里有多少个 | 次数加减 | 离开窗口的 key 有没有同步处理 |
| 分组 | 标准化后的身份 | 列表 append | key 是否已经把同类对象合并到一起 |

这一段最想让你形成的直觉，是“同一个输入在不同题型里会活成不同的状态”。你以后看到哈希题，先不要问“用什么库”，先问“这份数据在表里应该变成什么形状”。这个问题一旦问对，后面的实现就只剩下把 key、value 和顺序写清楚。

# 窗口计数手算一轮 {#window-lab}
<!-- tag: lab -->

很多人会在频次题里把“计数表”写得看起来对，真正动起来却不对。问题通常出在一件事：窗口已经往右滑了，左边那个字符还留在表里。下面用 `s = "abca"`、窗口长度 `3` 做一轮手算。这个例子很小，但足够看见“加、减、删”三步如何配合。

| 窗口 | 进入的字符 | 离开的字符 | 计数表变化 | 你要检查什么 |
| --- | --- | --- | --- | --- |
| `abc` | `c` | 无 | `{a:1, b:1, c:1}` | 表和窗口完全一致。 |
| `bca` | `a` | `a` | `{a:1, b:1, c:1}` | 进出同一个字符时，数量要保持同步。 |
| `cab` | `b` | `b` | `{a:1, b:1, c:1}` | 如果出窗后没删掉，distinct 统计就会多算。 |
| `abc` | 无 | 无 | 仍是 `{a:1, b:1, c:1}` | 表代表当前窗口，不能混入更早的状态。 |

## box
### 对应代码动作
窗口频次模板复制
```python
from collections import defaultdict

def window_counts(s, k):
 count = defaultdict(int)
 for right, ch in enumerate(s):
 count[ch] += 1
 if right >= k:
 left = s[right - k]
 count[left] -= 1
 if count[left] == 0:
 del count[left]
 # 这里可以接窗口是否满足条件的判断
 return count
```


## box warn
### 最常见的失误
一是只加不减，表会一直长下去；二是减了却不删零，distinct 数量会比实际多；三是先判断再更新，判断用到的是上一轮窗口的状态。你只要把这三件事对照窗口手算表去看，问题基本都能找出来。
这类题的检查顺序很固定：先看窗口是否对齐，再看计数是否同步，再看判断是否用的是更新后的表。顺序一乱，答案会在最后一步突然偏掉。

这一轮手算想说明的事情很简单：哈希表不是静态容器，它在窗口题里承担的是“当前范围的账本”。账本一旦记多了旧数据，后面的判断就会跟着偏。把这个例子吃透以后，你再回到 06 单元，看滑动窗口就不会只盯着右边界往前走，而会同时盯住左边界把旧状态带走。

# 失败实验室 {#failure-lab}
<!-- tag: debug -->
<!-- layout: grid2 -->

## box
### 症状一：题目明明要下标，你却只记录了值
这类错误最常见于两数之和。你会看到代码能判断“已经见过”，但最后返回不了位置。检查点只有一个：value 是否真的存了下标，而不是重复存了 key 自己。

## box soft
### 症状二：分组结果散成很多小组
这通常说明 key 没有标准化。先看每个对象在进入表之前有没有做排序、计数或多字段组合；如果没有，应该同组的对象自然会被分开。

| 现象 | 可能原因 | 下一步检查 |
| --- | --- | --- |
| 返回了同一个元素两次 | 先放后查 | 把“查补数”放到写入当前值之前。 |
| 窗口题答案比预期大 | 过期 key 没删除 | 确认出窗元素是否同步减一或删除。 |
| 异位词分散了 | key 没标准化 | 检查是否用了排序结果或计数元组。 |
| 复杂度写少了 | 漏掉造 key | 把遍历、查写、造 key 三段拆开再算一次。 |

这张表让你在调试时有一个固定的检查顺序。哈希题出错时，优先看 key 是否对、value 是否对、时机是否对、复杂度是否算全。只要这四件事逐一过一遍，很多“明明思路对了却总差一点”的问题都会被抓出来。

# 三个小例子，把动作看清 {#example}
<!-- tag: examples -->

1. 两数之和：先查补数，再记录当前位置
表里保存“已经看过的数 → 下标”的映射。当前数进来时，先查补数，再写入当前下标。
两数之和复制
`def two_sum(nums, target):
 pos = {}
 for i, x in enumerate(nums):
 need = target - x
 if need in pos:
 return [pos[need], i]
 pos[x] = i
 return []`

2. 最长连续序列：set 只负责判断存在
先把所有数放进集合，再从“起点”往后扩展。判断 `x - 1` 不存在，是为了避免同一段被重复扫描。
最长连续序列复制
`def longest_consecutive(nums):
 nums_set = set(nums)
 best = 0
 for x in nums_set:
 if x - 1 in nums_set:
 continue
 cur = x
 length = 1
 while cur + 1 in nums_set:
 cur += 1
 length += 1
 best = max(best, length)
 return best`

3. 字母异位词分组：先标准化 key，再 append
排序后字符串或者 26 维计数元组，都是“稳定身份”。同组字符串都进入同一个列表。
字母异位词分组复制
`from collections import defaultdict

def group_anagrams(strs):
 groups = defaultdict(list)
 for s in strs:
 key = "".join(sorted(s))
 groups[key].append(s)
 return list(groups.values())`

# 常见误区 {#mistakes}
<!-- tag: debug -->

把原字符串直接当分组 key**错因：**没有先标准化身份。**后果：**同组对象会被拆开。**修法：**用排序结果或计数元组做 key。
用 list 当 key**错因：**把可变对象直接拿去哈希。**后果：**Python 会报不可哈希错误，或者状态不稳定。**修法：**转成 tuple、字符串，或其他稳定表示。
两数之和先放后查**错因：**更新时机反了。**后果：**同一元素可能被重复使用。**修法：**先查补数，再写当前值。
窗口计数减到 0 还留着**错因：**没有维护窗口语义。**后果：**表里残留无效 key，判断 distinct 数量会错。**修法：**计数为 0 时删除，或在判断时显式忽略。
把平均 O(1) 说成绝对 O(1)**错因：**忽略冲突和退化情况。**后果：**复杂度表达不严谨。**修法：**说“平均情况下接近 O(1)”更稳。
漏掉造 key 的成本**错因：**只看哈希查询，不看 key 构造。**后果：**49 这类题复杂度说少了。**修法：**把遍历、查表、造 key 三段分开算。

# 把一个错的分组 key 改对 {#repair-chain}
<!-- tag: repair -->

看见“分组”两个字时，很多人会先把原字符串、原列表或者某个表面字段直接拿去做 key。这个做法有时看起来能跑，但它没有把“同类关系”真正编码进去。下面用一个最小修复链路把这个问题拆开：先看错误输入，再看标准化 key，最后看表为什么开始稳定。

## box
### 错误写法
把 `eat`、`tea`、`ate` 直接当成不同 key，或者把 `sorted(s)` 这个 list 直接拿来做 key。前一种会把本该同组的对象拆散；后一种在 Python 里根本不能作为 key。
**失败信号**你会得到三组，或者直接报 `unhashable type: list`。

## box
### 正确写法
先把字符串标准化成稳定身份，再 append。排序后的字符串可以变成 `aet`，26 维计数可以变成一个 tuple。两种写法都在做同一件事：把“同类关系”压成一个稳定 key。
**修复动作**先统一身份，再把 value 交给列表。

| 错误输入 | 标准化 key | 表里的结果 | 后续会发生什么 |
| --- | --- | --- | --- |
| `eat` / `tea` / `ate` | `aet` | 都落在同一个桶里 | 同组列表能被正确收集。 |
| `sorted(s)` 直接返回 list | `.join(sorted(s))` 或 tuple 计数 | 获得稳定 key | 可以顺利进入 dict，而不是在 key 阶段报错。 |
| `(user_id, date)` 两字段 | tuple 组合 | 多字段组成同一个身份 | 同一天同一用户的日志会自动聚在一起。 |

这个修复链路的重点，是让你把“对象本身”和“对象身份”分开看。很多哈希题在纸面上写得像是分组，真正落到代码里，其实是把对象先压成稳定身份，再把原对象放回 value 里。你以后遇到新题，只要先问自己“这个对象应该被压成什么 key”，就能少走很多弯路。

## box soft
### 对应代码桥
修复后的分组复制
```python
from collections import defaultdict

def group_anagrams_fixed(strs):
 groups = defaultdict(list)
 for s in strs:
 key = "".join(sorted(s)) # 先把身份标准化
 groups[key].append(s) # 再把原对象放回 value
 return list(groups.values())
```


## box warn
### 练习时怎么自查
先把原题改成一句话：对象是什么，身份是什么，value 要保存什么。然后检查这三件事是不是在代码里都能一一对应。只要有一项说不清，通常就是 key 还没定稳。
如果你在实现里看到“先把对象放进去，再想怎么区分同类”，那就说明顺序反了。哈希表不替你发明身份，它只负责存你已经定义好的身份。

这一段和前面的分组 key、状态追踪是连在一起的：前面讲的是“同类对象要进同一组”，这里讲的是“怎么把同类关系编码成稳定 key”。当你能把这个修复动作说出来，分组题就不再只是 `dict` 和 `append`，而是“标准化身份，再收集原对象”。

### 4 short-answer

**Q:** 4. dict 的 key 和 value 在题目里分别怎么理解？

Explain: 要点：key 是身份，value 是围绕身份保存的信息。身份可以是数字、字符、排序后字符串、元组；信息可以是下标、次数、列表。

### 5 code-reading

**Q:** 5. 两数之和中 `if need in pos` 检查的是什么？

Explain: 检查补数是否已经出现在“已处理元素”的映射里。答题时要写出：表里存的是下标，先查后放是为了避免复用当前元素。

### 6 code-reading

**Q:** 6. 128 中 `if x - 1 in nums_set: continue` 为什么能避免重复扫描？

Explain: 它只从连续段的起点往后扩展。只要前一个数还在集合里，当前数就不是起点，继续往后扫只会重复。

### 7 code-reading

**Q:** 7. 49 中 `groups[key].append(s)` 的 key 和 value 分别是什么？

Explain: key 是标准化后的身份，比如排序后的字符串或计数组元组；value 是同组字符串的列表。

### 8 complexity

**Q:** 8. 49 排序 key 的时间复杂度为什么要写到 `k log k`？

Explain: 因为每个字符串长度为 k 时，排序本身要花 `O(k log k)`；不能只算哈希查找，还要把造 key 的成本算进去。

### 9 diagnosis

**Q:** 9. 两数之和先 `pos[x] = i` 再查补数，可能有什么问题？

Explain: 会把当前元素也放进表里，导致同一个下标可能被重复使用。答案里要写出：表代表的是“已处理元素”，所以顺序必须先查后放。

### 10 diagnosis

**Q:** 10. 用原字符串直接分组异位词会错在哪里？

Explain: 原字符串没有把“同类关系”标准化。`eat` 和 `tea` 不是同一个 key，所以不会自动进同一组。修法是排序或计数后再做 key。

### 11 diagnosis

**Q:** 11. 窗口计数减到 0 后不删除 key，可能影响什么判断？

Explain: 会让表里残留已经离开窗口的元素，导致 distinct 数量、是否仍满足条件等判断变错。

### 12 diagnosis

**Q:** 12. 说“哈希表查询永远 O(1)”哪里不严谨？

Explain: 要说“平均情况下接近 O(1)”更准确。复杂度还会受到冲突、实现和造 key 成本影响，不能直接写成绝对值。

### 13 transfer

**Q:** 13. LeetCode 1 两数之和的哈希表里应该存什么？

Explain: 通常存“数字 → 下标”。补数查询靠 key，返回结果靠 value。答案里最好补一句：先查后放，避免复用当前元素。

### 14 transfer

**Q:** 14. LeetCode 219 为什么可以用 set 维护最近 k 个元素？

Explain: 因为它只需要判断窗口里有没有重复值，不需要次数或下标。窗口滑动时，离开的元素要同步移除，保证集合和窗口一致。

### 15 transfer

**Q:** 15. 多数元素或字符频次题为什么更像 dict 计数，而不是 set？

Explain: 因为题目问的是“出现几次”或“谁更频繁”，而不是“有没有出现”。`set` 只能回答存在性，不能保存频次信息。

### 16 transfer

**Q:** 16. 如果要按“用户 id + 日期”分组日志，key 可以怎么设计？

Explain: 可以把 key 设计成 ` (user_id, date) ` 这样的 tuple。多字段一起才构成身份，单字段通常不够。
