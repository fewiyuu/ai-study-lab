链表位置关系

链表双指针框架

这一页把链表双指针讲成三类稳定关系：快慢指针用速度差找中点或判环，前后指针用固定距离找倒数位置，虚拟头结点把头节点变化收进同一套删除和拼接流程。你要学会的不是变量名，而是每次循环里这几个指针各自读什么、写什么、还剩哪段链没有走完。

读完本页应该能做什么

  • 能用前后指针和虚拟头结点定位倒数第 n 个节点，说明 slow 最终停在前驱还是目标节点。
  • 能手工追踪 `prev`、`cur`、`fast`、`slow`、`carry` 的变化，指出断链、漏节点和漏进位会在哪一步发生。

题目先翻成关系

链表题不要先背模板名。先看题目要的是位置、前驱，还是结构上的统一入口。只要能说清“谁领先谁”“谁要停在前驱”“谁负责保留后继”，代码就会从猜答案变成维护关系。

先别追的方向

这一页先不扩到递归反转、链表归并排序、环入口证明、双链表细节这些更后面的内容。当前只抓三件事：快慢指针、前后指针、虚拟头结点。它们够你把 82、2、445、287 这几类题说稳。

题目动作 优先关系 为什么这样做
找中点、判环 快慢指针 fast 比 slow 多走一步或两步，靠速度差逼出中点或相遇点。
找倒数第 n 个、删倒数节点 前后指针 fast 先领先固定步数，slow 再同步走，最后停在目标或前驱。
删除头节点也可能发生 虚拟头结点 dummy 把“头节点变化”统一成改前驱的 next。
逐位相加、反转、重连 多指针协作 一个指针读原链，一个指针写结果，一个指针保留后继或进位。

主线地图

mental model

链表没有数组那样的随机访问。你想知道第 k 个节点，就得沿着 next 一步一步走。双指针的作用，是把“走到哪里”这件事拆成可以维护的关系：要么速度不同，要么距离固定，要么借 dummy 统一前驱。

链表限制不能随机访问
制造关系快慢 / 前后 / dummy
循环推进每轮只改少数指针
边界保护判空、前驱、后继
输出结果节点、链表或布尔值
关系 读什么 写什么 常见症状
快慢 当前节点和下一步是否存在 slow.next / fast.next.next 少判一个空就会越界。
前后 fast 领先了几步 slow 的目标位置 领先步数差一位就会删错节点。
dummy dummy.next 指向谁 前驱的 next 忘记返回 dummy.next 会丢头节点变化。

链表双指针状态演示

visual
快慢找中点 slow 每轮 1 步，fast 每轮 2 步

读的是当前节点是否还能继续走，写的是下一轮 slow 和 fast 的新位置。fast 先到尾部，slow 就落在中点附近；如果有环，fast 会在环里追上 slow。

前后距离 fast 先领先 n + 1 步

读的是 fast 和 slow 之间的距离，写的是 slow 的前驱位置。fast 到空时，slow 正好停在要删节点的前一个位置。

反转链表 先记后继，再改 next

读的是 cur 原来的后继，写的是 cur.next 指向 prev。先保存 nxt，后面才不会把还没走过的链弄丢。

两数相加 p、q、carry 一起推进

读的是两条输入链和进位，写的是结果链的下一位。只要 p、q 或 carry 还有一个没结束，循环就不能停。

观察什么

先找状态变量，再找每一轮改了哪条 next 或哪个值。链表题最怕“看起来走了，实际上断了”；把状态放到纸上，断链、漏节点和漏进位就都能被追出来。

一条小提醒

同一条链表在不同题型里，表面变量名会变，但读写职责不会变。你只要分清“谁负责读原链、谁负责写结果、谁负责保留后继”，题目就不会散。

链表双指针为什么必要

why

数组能用 nums[i] 直接跳到第 i 个元素，链表不行。链表只能沿着 next 一步一步走，所以“第 k 个”“倒数第 n 个”“中点”“环入口”这些问题，最终都要转成指针之间的位置关系。

这种关系可以是速度差，也可以是距离差，还可以是同步移动。你不需要先想出答案，只要把关系设对，链表就会自己把答案推出来。难点通常不在遍历，而在你是否清楚每一步之后，哪个指针该停，哪个指针该继续，哪个节点会被改写。

判断句

链表题只要问位置、倒数、相遇、中点，先考虑用指针关系替代下标。

先别混进来的概念

这一页先不把递归反转、链表归并排序、双链表、循环节证明和数组双指针混进来。它们会在后面的题里重新出现，但当前只需要盯住“快慢”“前后”“dummy”这三条线。

快慢指针：用速度差定位

fast slow

快慢指针最常见的写法，是 slow 每次走一步，fast 每次走两步。fast 先到尾部，slow 就在中点附近；如果链表有环，fast 会在环里不断追近 slow，最后两者相遇。

真正要写稳的是循环条件。while fast and fast.next把 fast 的两步移动变成可验证的前提。少一个判空，fast.next.next 就可能越界；多走一步没问题，少保护一步就会崩。

位置 读什么 写什么 错了会怎样
循环条件 fast 与 fast.next 是否都存在 决定能不能安全走两步 少一个判空就访问空节点。
slow 更新 slow 当前站在哪个节点 slow = slow.next slow 该到中点却还停在前面。
fast 更新 fast 的后继和后继的后继 fast = fast.next.next 没有前置判空就会断言失败。

前后指针：保持固定距离

gap

倒数第 n 个节点最适合前后指针。让 fast 先走 n 步，再让 slow 和 fast 一起走，fast 到末尾时，slow 就离目标剩下固定距离。若你要删的是目标节点，那 slow 需要停在它的前驱，所以 fast 常常要先走 n + 1 步。

这个模板最容易错在“目标”和“前驱”没有先说清。你先决定 slow 最后要停在哪里，再决定 fast 该领先几步。只要这一步定义不清，后面的 while 再对，也会删错一格。

$$\text{删除倒数第 } n \text{ 个节点时，fast 往往先走 } n+1 \text{ 步}$$

这里的 n+1 对应“要改的是 slow.next”。如果只想找到倒数第 n 个节点本身，领先步数就要重新定义。

角色 当前读什么 当前写什么 最终停在哪
fast 前面还剩多少节点 继续向后走 空节点前。
slow 与 fast 的固定距离 同步向后走 目标前驱或目标节点。
dummy 头节点前的统一入口 代替真实 head 作为前驱 方便统一返回 dummy.next。

虚拟头结点：把头节点当普通节点处理

dummy

虚拟头结点 dummy 不保存真实数据，它只负责把原链表头前面的那一格补出来。只要头节点也可能被删除或替换，dummy 就能把“删头”和“删中间”收成一套代码，不用单独写一条头节点分支。

以 1->1->2 为例，如果你要删掉所有重复值，没有 dummy 时，头部重复段会逼你先处理 head；有 dummy 时，prev 永远站在结果链尾部，重复段是否落在头部都只是一次改 prev.next。

检查项 要读什么 要写什么 错了会怎样
dummy 的位置 dummy.next 指向谁 把 head 挂到 dummy 后面 没有统一入口，头节点会变成特殊分支。
prev 的职责 结果链最后一个保留节点 改 prev.next 删重复段或删头时会卡住。
返回值 最终链表的新头 dummy.next 返回旧 head 会丢已经重连过的部分。

断链与进位：先保留，再改写

read / write

链表题里最怕的情况是改写得太早。反转链表前要先保存后继，否则你把当前节点的 next 改掉以后，后面那半条链就找不回来了。两数相加也要单独保存 carry，因为它会影响下一位的结果。

所以这类题要先说清楚读写职责：谁负责读原链，谁负责写结果，谁负责保留后继，谁负责记录进位。职责不清，代码就会出现断链、漏节点、漏进位或者把旧节点重复用两次。

动作 先读什么 先保留什么 漏了会怎样
反转链表 cur 当前节点 nxt = cur.next 后继丢失，链条断开。
删除倒数节点 fast 领先的距离 slow 的前驱位置 slow 停错格，改不到目标前驱。
两数相加 p、q、carry 下一位的进位 最后一位 1 会被吞掉。

复杂度与断链检查

debug

链表双指针通常都只扫一遍，所以时间复杂度多半是 O(n)。额外空间一般是 O(1)，但如果你在正序加法里用了栈，空间就要把栈的大小算进去。面试里别只说“哈希表很快”那种空话，链表题更看你有没有把扫描次数和额外结构分开算。

$$ T(n)=O(n), \qquad S(n)=O(1) $$

这里的 n 是实际走过的节点数。若为了正序加法借助栈，空间就要改成 O(m+n) 量级。

框架 时间 空间 说明
快慢找中点 / 判环 O(n) O(1) fast 最多走到尾部或在环内相遇。
前后删除倒数节点 O(n) O(1) fast 和 slow 只做单调向后移动。
两数相加 O(m+n) O(1) 或 O(m+n) 是否借助栈决定额外空间。

反转链表的完整追踪

lab

把 1->2->3->4 反转成 4->3->2->1，最适合拿来检查“先保存后继，再改 next”这条规则。因为这题看似只是把箭头方向翻过来，实际上每一轮都在同时维护三份信息：当前节点、已经反转好的前半段、还没走完的后半段。

状态追踪表

轮次 prev cur nxt 看点
开始 None 1 2 前半段还为空，后半段还没碰。
处理 1 1 2 2 1 的 next 已改成 None，1 变成结果链尾。
处理 2 2 3 3 2 指向 1，前半段长度变成 2。
处理 3 3 4 4 3、2、1 已经连好，cur 还要继续走。
处理 4 4 None None cur 走完后，prev 就是新头结点。

对应代码

反转链表模板
def reverse(head):
    prev = None
    cur = head
    while cur:
        nxt = cur.next
        cur.next = prev
        prev = cur
        cur = nxt
    return prev

错一行会怎样

如果先写 cur.next = prev 再去取 cur.next，你拿到的已经不是原来的后继，而是刚刚被改过的前驱。链表会在下一轮开始时失去前半段以外的全部后继，整条链会直接断掉。

练习钩子

拿 1->2->3 再走一遍，只写三列：prev、cur、nxt。如果你能在每轮结束时说出这三列的值，再去写代码就很少会丢链。

  • 先记后继，再改 next。
  • 先改 prev，再推进 cur。
  • 每轮结束时确认还剩哪段没走完。

合并两条有序链

merge

有序合并把 1->3->5 和 2->4->6 接成一条链，最适合检查 dummy 和 cur 的职责。每一轮只看两条链的头部，谁小就把谁接到结果链尾，再把那条链的头指针往后挪一步。这样做的关键，是让结果链的尾部永远是 cur。

状态追踪表

轮次 l1 l2 cur 结果链
开始 1 2 dummy 空
第 1 轮 3 2 1 1
第 2 轮 3 4 2 1->2
第 3 轮 5 4 3 1->2->3
收尾 空 空 6 1->2->3->4->5->6

对应代码

合并有序链模板
def merge_two_lists(l1, l2):
    dummy = ListNode(0)
    cur = dummy
    while l1 and l2:
        if l1.val <= l2.val:
            cur.next = l1
            l1 = l1.next
        else:
            cur.next = l2
            l2 = l2.next
        cur = cur.next
    cur.next = l1 if l1 else l2
    return dummy.next

错一行会怎样

如果你忘了把被选中的那条链往后移，下一轮还会继续拿同一个节点比较，结果就是重复接入同一个值，或者在某些实现里直接卡成死循环。若你忘了 cur = cur.next，结果链尾会一直停在原地，后面的节点就算已经选出来也接不上去。

练习钩子

把输入改成 [1,2,4] 和 [1,3,4]，手工写出每轮选中了哪一边、cur 指向哪里、结果链长了多少。这个过程和前面的删倒数节点一样，都在练“谁负责前驱、谁负责结果尾部”。

  • 先选更小的头结点。
  • 再把被选中的链向后推进一步。
  • 最后让 cur 跟上结果尾。

头部重复段删除

head case

把 1->1->1->2 这种链表拿出来看，最能检验 dummy 有没有真正起作用。这里的问题很具体：头部一整段都要删掉，结果链的头结点会换到哪里。没有 dummy 的写法，要先单独处理 head；有 dummy 以后，prev 永远站在结果链尾部，头部重复段和中间重复段就能用同一套改法处理。

过程表

阶段 prev cur 结果链
起点 dummy 1 空
发现重复 1 dummy 1 还没接任何节点。
跳过整个重复段 dummy 2 dummy.next 直接改到 2。
收尾 2 空 2 成为新头节点。

对应代码

删除重复段模板
def delete_duplicates(head):
    dummy = ListNode(0, head)
    prev = dummy
    cur = head
    while cur and cur.next:
        if cur.val == cur.next.val:
            x = cur.val
            while cur and cur.val == x:
                cur = cur.next
            prev.next = cur
        else:
            prev = cur
            cur = cur.next
    return dummy.next

如果不用 dummy

头部一旦全是重复值，head 本身就会被删掉。没有 dummy 时，你得先判断“是不是从头开始删”，再决定 head 要不要改，这会把一套统一的改写流程拆成两个分支。这个分支一多，链表指针就容易漏掉一格。

练习钩子

把链表改成 1->1->2->3->3，先写出 dummy、prev、cur 在每次跳过重复段后的指向，再把 dummy.next 的新值说出来。这个练习和前面的删倒数节点一样，都在检查你是否真的知道“结果链头”在哪里。

  • 先找重复段的边界。
  • 再改 prev.next 指向下一段。
  • 最后确认返回值是不是 dummy.next。

精修检查台

invariant

不变量先写清

框架 每轮后保持什么 如果不成立，症状是什么
快慢指针 fast 比 slow 领先固定步数。 中点或环检测会偏掉。
前后指针 fast 与 slow 的距离不变。 删倒数节点会少一格或多一格。
dummy dummy.next 始终是结果链头。 删头节点后会返回旧 head。

worked example：删除倒数第 2 个

链表 1->2->3->4->5，dummy 在最前面。fast 先从 dummy 走 3 步，落到节点 3；slow 仍在 dummy。随后 fast 和 slow 同步往后走，fast 到空时，slow 停在节点 3，也就是要删节点 4 的前驱。

执行 slow.next = slow.next.next 后，链表变成 1->2->3->5。这个例子最重要的地方，是头节点也能用同一套写法，不需要单独开分支。

删除第 2 个节点的过程表

步骤 fast slow 结果
起点 dummy dummy 还没开始找前驱。
fast 先走 3 步 节点 3 dummy 建立固定距离。
一起走到结束 空 节点 3 slow 正好停在前驱。
改写 next 空 节点 3 删掉节点 4。

最容易踩的坑

判空只看 fast，会在 fast.next.next 处越界；反转时先改 next 再存后继，会断掉后面的链；删倒数节点时 slow 停在目标节点而不是前驱，会改不到连接；加法题漏掉 carry，会把最后一位 1 吞掉。

删倒数第 2 个的完整追踪

trace

现在把前面的关系落到一个完整锚点上：链表 1->2->3->4->5，目标是删掉倒数第 2 个节点，也就是节点 4。这里最容易迷路的地方是 fast 到底该领先几步、slow 最后该停在哪、哪一格 next 会被改写。

步骤追踪表

步骤 fast slow 看点
起点 dummy dummy 两者先站在同一条起跑线。
fast 先走 3 步 节点 3 dummy 建立 n + 1 的距离。
同步向后走 空 节点 3 slow 正好停在节点 4 的前驱。
改写 next 空 节点 3 slow.next = slow.next.next 删掉节点 4。

对应代码

删除倒数第 n 个节点追踪版
def remove_nth_from_end_trace(head, n):
    dummy = ListNode(0, head)
    slow = dummy
    fast = dummy

    for _ in range(n + 1):
        fast = fast.next

    while fast:
        slow = slow.next
        fast = fast.next

    slow.next = slow.next.next
    return dummy.next

错一格会怎样

如果 fast 只领先 n 步，slow 最后会停在目标节点本身而不是前驱。拿这个例子来说，slow 会停到节点 4，上面那句 slow.next = slow.next.next 就会删错节点，实际删掉节点 5。很多“答案对了一半”的链表题，就是在这一步差了一格。

练习钩子

把同一个例子再跑一遍，只改一个条件：让 fast 先走 n 步。你会发现 slow 的落点变了，最后改写的 next 也跟着变。这个小实验的目的，是让你把“目标节点”和“前驱节点”分开记住。

  • 先说 fast 领先多少步。
  • 再说 slow 最后停在哪里。
  • 最后说改的是哪一条 next。

环检测与 287 的位置映射

cycle

快慢指针不只用来找中点。把链表看成有环结构时，fast 和 slow 的速度差会把“是否成环”这件事直接暴露出来。287 寻找重复数也是同一个想法：把数组下标看成节点，把值看成 next 指向的下一格，快慢指针在这个隐式链上跑，就能把重复数对应的入口找出来。

环检测追踪表

轮次 slow fast 发生了什么
开始 1 1 两者站在同一个起点。
第 1 轮 2 3 fast 比 slow 多走了一步，距离被拉开。
第 2 轮 3 5 如果存在环，fast 会继续追近 slow。
相遇轮 4 4 两者落在同一节点，说明结构里存在重复路径。

对应代码

链表判环模板
def has_cycle(head):
    slow = head
    fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:
            return True
    return False

为什么这一步会出错

如果你只看“快慢速度差”，但忘了 fast 也会进入环，代码就会在没有判空的情况下访问空节点；如果你把 slow 和 fast 的起点写错，第一次相遇的位置也会偏。287 的难点不在“快慢”两个字，而在你有没有把隐式链的 next 定义清楚。

练习钩子

把数组 [1,3,4,2,2] 重新想成一条隐式链：每个值都指向下一个下标。先找相遇点，再想入口点，最后把入口翻回重复数。这个过程和链表判环是同一套动作，只是 next 不再是显式节点，而是数组值。

  • 先定义“节点”和“next”。
  • 再找相遇点。
  • 最后把入口映射回重复数。

渐进例子与代码

code reading

这几段代码按从小到大的顺序安排。读代码时先看状态放在哪里，再看每一步读什么、写什么。

1. 快慢指针找中点

fast 每次走两步，slow 每次走一步。循环结束时 slow 在中点附近。

正确示例：快慢指针找中点Python
def middle_node(head):
    slow = head
    fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
    return slow

2. 删除倒数第 n 个节点

dummy 消除删除头节点的特殊情况，fast 先走 n + 1 步让 slow 停在被删节点前面。

正确示例：删除倒数第 n 个节点Python
def remove_nth_from_end(head, n):
    dummy = ListNode(0, head)
    slow = dummy
    fast = dummy
    for _ in range(n + 1):
        fast = fast.next
    while fast:
        slow = slow.next
        fast = fast.next
    slow.next = slow.next.next
    return dummy.next

3. 两数相加的结果链

p、q 扫输入，cur 接结果。carry 不能漏，它可能在两个链表都空后继续产生新节点。

正确示例：两数相加的结果链Python
def add_two_numbers(l1, l2):
    dummy = ListNode(0)
    cur = dummy
    carry = 0
    p, q = l1, l2
    while p or q or carry:
        x = p.val if p else 0
        y = q.val if q else 0
        s = x + y + carry
        carry = s // 10
        cur.next = ListNode(s % 10)
        cur = cur.next
        p = p.next if p else None
        q = q.next if q else None
    return dummy.next

4. 错误示例：反转链表前没保存后继

这段代码改掉 cur.next 后，再想从 cur.next 往后走就已经回头了。

错误示例：反转链表前没保存后继Python
def reverse(head):
    prev = None
    cur = head
    while cur:
        cur.next = prev
        prev = cur
        cur = cur.next  # 错：这里会回到 prev，后面的链丢了
    return prev

常见误区

debug

fast 判空条件少写一层

错因：fast 要走两步，却只判断 fast 存在。

后果：访问 fast.next.next 时可能空指针。

修法：循环条件写 while fast and fast.next。

删除头节点时忘记 dummy

错因：默认 head 不会变。

后果：头部重复或头节点删除时返回错误链表。

修法：用 dummy 指向 head，最后返回 dummy.next。

反转前不保存 nxt

错因：改 next 太早。

后果：后续节点丢失或形成死循环。

修法：先 nxt = cur.next，再改 cur.next = prev。

前后指针距离差一

错因：没有明确 slow 要停在目标还是目标前一个。

后果：删除错节点。

修法：先定义 slow 的最终位置，再决定 fast 先走 n 还是 n+1。

两数相加漏掉最后 carry

错因：循环条件只写 p or q。

后果：比如 5+5 得不到 10 的高位 1。

修法：循环条件加上 or carry。

返回原 head

错因：忘记链表头可能改变。

后果：删除或重建链表后返回旧头。

修法：涉及 dummy 或结果链时返回 dummy.next。
