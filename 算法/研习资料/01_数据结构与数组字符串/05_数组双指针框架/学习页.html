数组题型

数组双指针框架：左右夹逼、快慢覆盖与尾部合并

这一页只讲数组上的三类双指针：左右夹逼、快慢覆盖和逆向合并。你会看到它们都在处理同一件事，只是分工不同 - 有的负责从两端缩小范围，有的负责原地保留结果，有的负责从尾部写入，避免覆盖还没读完的数据。学完后，你应该能先看输入条件，再判断该用哪种指针职责。

先抓主线

route

先看输入，再定职责

数组双指针的关键不是两个下标本身，而是它们分别承担什么工作。输入如果已经有序，端点比较通常能决定谁先移动；输入如果要求原地保留，`fast` 和 `slow` 可以把读取和写入拆开；输入如果要合并两个有序段，就要让 `write` 从尾部开始，避免覆盖还没读完的值。

判断句

题目一旦给出“有序 / 原地 / 两端”这三类信号，你就应该先选职责，再写代码。

先别追

这一页先不把三数之和、三路分区、矩阵行列变化和滑动窗口搅在一起。它们会在后面的单元或其他题型里再出现，但这页只负责把“谁读、谁写、谁保留结果”说稳。

路线图

map

本页不是按题号扫过去，而是按判断顺序读。先判输入是否有序、是否要求原地、是否需要从两端同时看，再决定 `left/right`、`fast/slow` 还是 `write` 的职责。你不用先记模板名，先把题目翻译成动作。

1. 看信号

有序、原地、从两端同时看，是数组双指针最常见的入口。只要题面把这些信号说出来，就值得先往双指针上想。

2. 定状态

状态不是变量名，而是变量代表的区域：`left` 和 `right` 夹住未处理区间，`slow` 前面是合法结果，`write` 右边是最终答案。

3. 写边界

回文串通常用 `left < right`，因为中点不用单独处理；结果数组要填满时常用 `left <= right`，因为最后一个位置也要写进去。

4. 看结果

快慢覆盖只看前缀，逆向合并只看写到哪里，左右夹逼则看区间是否缩到空。每一类模板的结果位置都不一样。

读到的信号 优先模板 要盯的状态 常见错误
有序数组、比较两端 左右夹逼 未处理区间 `[left, right]` 不说清比较依据就随便移动一边
原地删除、压缩、保留 快慢覆盖 `nums[:slow]` 把 `slow` 当成遍历指针，忘了它是写指针
两个有序段合并到一处 逆向合并 `write` 右侧已经确定 从前往后写，覆盖未读值

同一锚点，三次走法

trace lab

数组双指针最值得反复看的，不是题号，而是同一批输入在不同模板里会变成什么样。`[1, 1, 1, 2, 2]` 可以被 80 当成“结果前缀的压缩对象”，`[-4, -1, 0, 3, 10]` 可以被 977 当成“左右端点比较对象”，`[1, 2, 3, 0, 0, 0] + [2, 5, 6]` 则是 88 里的“待写回目标数组”。数据本身没有变，变的是谁读、谁写、谁负责把最终状态固定下来。

这也是数组双指针最容易混淆的地方。很多人会背“左边动、右边动、快慢写、尾部写”，但一旦题目换个外壳，就不知道当前变量在维护什么。你只要把“读什么、写什么、结果放在哪里”这三件事说清，模板就不会乱。

一条最短判断线

如果题目要的是“比较两端并缩小范围”，先想左右夹逼；如果要的是“把不合格的元素挤掉，只留下合法前缀”，先想快慢覆盖；如果要的是“两个有序段合成一个有序段”，先想尾部合并。这个判断顺序比背题号更重要。

最常见的错位

把快慢覆盖写成普通遍历，只知道 `slow` 在变，却不知道它代表结果边界；把左右夹逼写成左右乱跳，只知道边界在动，却不知道哪一次比较已经排除了哪一侧；把尾部合并写成插入，只记得有空位，却忘了有效值还在前半段。

最短判断伪码
if input_is_sorted and need_both_ends:
    use_two_ends()
elif need_in_place and keep_prefix:
    use_fast_slow()
elif merge_into_existing_space:
    fill_from_tail()

这段伪码不是让你照抄的，它只是把判断顺序写短：先看输入形状，再看读写职责，最后才看题号。真正写代码时，还要把循环边界和结果位置补完整。

283：搬零和 80 共享同一条前缀线

shift demo

`283 移动零` 和 `80 删除有序数组中的重复项 II` 都在做快慢覆盖，但它们保留的对象不一样。283 保留的是非零值，80 保留的是有序数组中最多两次的重复值。把这两个题放在一起看，最容易看清楚的就是：`fast` 只是读，`slow` 才是结果边界；真正改变结果的是“保留什么”，不是“遍历什么”。

同样的骨架，不同的保留规则

283 里，所有非零值都要向前聚拢，零只是最后补上的占位符；80 里，保留规则更严格，超过两次的重复值要直接跳过。两题的代码形状像，但判断条件完全不同。学会这一点以后，你就不会把“看到 fast/slow”误认为“题目一定一样”。

如果这里写错

283 里如果把零也当成保留值，结果前缀就会被污染；80 里如果把所有重复都保留，题目要求的“最多两次”就破了。两者都提醒你一件事：模板只是骨架，真正决定答案的是保留规则。

你可以把 283 看成最简单的快慢覆盖：`fast` 只负责扫描，`slow` 只负责把合格值向前写。80 比 283 多了一层判断，但思路没有变。先把这一层看明白，再回到 80，你会更容易理解 `slow - 2` 为什么是关键位置。

状态台账

invariant

数组双指针最容易错的地方，不是不会写循环，而是变量的职责和结果位置没有对上。你写代码时，不要只看 left / right / slow / write 还在不在变，要同时看它们分别在读什么、写什么、保留什么。

模板 读什么 写什么 不变量 写错后的症状
左右夹逼 `nums[left]`、`nums[right]` 移动边界、确认答案 区间外的值已经确认，区间内还没处理 中点被重复比对，或有一侧跳过太早
快慢覆盖 `fast` 读当前值，`slow - 2` 读历史前缀 `nums[slow]` `nums[:slow]` 始终是当前合法结果 返回长度错，或合法前缀里混入太多重复值
逆向合并 `nums1[p1]`、`nums2[p2]` `nums1[write]` `write` 右边已经是最终答案 前面还没读完就被覆盖，后面没法比较
尾部平方 两端绝对值 结果数组尾部 最大的平方先写到最右侧 结果顺序倒了，还要再翻一次
循环条件 适合哪里 为什么
`left < right` 回文判断、只需要两端互比的题 中点不是有效比较对象，不必单独处理。
`left <= right` 结果数组要被填满的题 最后一个位置也要写进去，不能提前停。
`while p2 >= 0` 88 这类右侧还有剩余的合并题 只要第二个数组没写完，结果就还没结束。

125：左右夹逼先排除外壳

two ends

回文题最容易把难点想复杂。先忽略标点和大小写，剩下的判断只是两端是否一致。只要端点比较能直接排除一侧，就可以一边比一边向中间缩。

正确示例：125 验证回文串

125 回文代码
def is_palindrome(s):
    left, right = 0, len(s) - 1
    while left < right:
        while left < right and not s[left].isalnum():
            left += 1
        while left < right and not s[right].isalnum():
            right -= 1
        if s[left].lower() != s[right].lower():
            return False
        left += 1
        right -= 1
    return True

先看什么

这段代码里，`left` 负责找左边第一个有效字符，`right` 负责找右边第一个有效字符。比较成功后，两边一起往中间缩；比较失败时，整题直接结束。

判断句

回文题只需要在 `left < right` 时继续，因为中点不需要单独判一遍。

轮次 `left` 看到什么 `right` 看到什么 动作 如果这里写错
1 `A` `a` 大小写统一后相等，两边都向中间走 只比较原始字符，大小写会误判
2 空格 / 逗号 有效字母 左边继续跳过无关字符，右边暂时不动 没跳过非字母数字，就会把标点当成内容
3 `m` `m` 继续缩小区间，直到两端相遇 把循环条件写成 `<=`，中点会被重复比对

80：快慢覆盖把结果写回前缀

slow fast

80 的关键不是“删掉多少个值”，而是“结果前缀保留到哪一步”。`fast` 负责把原数组从头扫到尾，`slow` 负责写下应该保留的值。只要前缀里相同值最多保留两次，循环就一直成立。

正确示例：删除有序数组中的重复项 II

80 快慢覆盖代码
def remove_duplicates(nums):
    slow = 0
    for x in nums:
        if slow < 2 or x > nums[slow - 2]:
            nums[slow] = x
            slow += 1
    return slow

结果前缀怎么读

当 `slow` 还小于 2 时，前两个位置天然可以保留；当 `slow` 已经到 2 以后，才需要拿当前值和 `nums[slow - 2]` 比较。这个判断依赖有序数组，因为相同值必须挨在一起。

判断句

`slow` 不是遍历指针，而是下一次写入的位置；返回值应该是它，不是原长度。

`fast` 当前值 `slow` 写前位置 是否保留 写到哪里 结果前缀
`1` 0 保留，前两个位置直接放行 `nums[0]` `[1]`
`1` 1 保留，前两个位置仍然放行 `nums[1]` `[1, 1]`
`1` 2 跳过，因为它不会比 `nums[0]` 更大 无 `[1, 1]`
`2` 2 保留，因为 `2 > nums[0]` `nums[2]` `[1, 1, 2]`
`2` 3 保留，因为 `2 > nums[1]` `nums[3]` `[1, 1, 2, 2]`

977：先写大平方，再把剩余值往前填

reverse fill

有序数组平方之后不再有序，所以不能简单从左到右平方。真正能决定当前位置结果的，不是当前下标大小，而是两端的绝对值谁更大。绝对值更大的那一侧，平方一定更大，应该先写到结果数组尾部。

正确示例：有序数组的平方

977 逆向写入代码
def sorted_squares(nums):
    n = len(nums)
    ans = [0] * n
    left, right = 0, n - 1
    write = n - 1
    while left <= right:
        if abs(nums[left]) > abs(nums[right]):
            ans[write] = nums[left] ** 2
            left += 1
        else:
            ans[write] = nums[right] ** 2
            right -= 1
        write -= 1
    return ans

为什么是尾部

因为当前最大平方一定属于两端之一，所以每写一个值，`write` 就可以向左挪一格。这样做的好处是，不用额外开一个队列，也不用先把结果反转回来。

判断句

当结果需要保持从小到大，而最大值先被确定时，写指针就应该从尾部开始。

轮次 `left` / `right` 比较什么 写到 `write` 结果尾部
1 `-4` / `10` `|10|` 更大 `100` `[..., 100]`
2 `-4` / `3` `|-4|` 更大 `16` `[..., 16, 100]`
3 `-1` / `3` `|3|` 更大 `9` `[..., 9, 16, 100]`
4 `-1` / `0` `|-1|` 更大 `1` `[..., 1, 9, 16, 100]`
5 `0` / `0` 最后一个位置填 `0` `0` `[0, 1, 9, 16, 100]`

88：从尾部写，才不会把没读完的值盖掉

merge

88 的陷阱很具体：`nums1` 前半段还有有效值，后半段只是空位。如果从前往后写，新的值一旦覆盖旧值，后面就再也读不回来了。尾部写的好处，就是先把最大的那一个放好，再慢慢向左补齐。

正确示例：合并两个有序数组

88 尾部合并代码
def merge(nums1, m, nums2, n):
    p1, p2 = m - 1, n - 1
    write = m + n - 1
    while p2 >= 0:
        if p1 >= 0 and nums1[p1] > nums2[p2]:
            nums1[write] = nums1[p1]
            p1 -= 1
        else:
            nums1[write] = nums2[p2]
            p2 -= 1
        write -= 1

为什么不能从前面写

因为 `nums1` 的前半段还要参与比较。只要它被覆盖，后面剩下的值就失去了参照。尾部写可以把“还没读完”的风险留在左边，把“已经写完”的部分固定在右边。

判断句

合并到一个有空位的数组时，先想尾部写，而不是先想插入。

轮次 `p1` / `p2` 谁更大 `write` 写入 右侧结果
1 `3` / `6` `6` 更大 `6` `[..., 6]`
2 `3` / `5` `5` 更大 `5` `[..., 5, 6]`
3 `3` / `2` `3` 更大 `3` `[..., 3, 5, 6]`
4 `2` / `2` `2` 相等时，写右侧也可以 `2` `[..., 2, 3, 5, 6]`
5 `1` / `0` 剩下的都来自 `nums1` `2`、`1` 依次补齐 `[1, 2, 2, 3, 5, 6]`

复杂度和移动次数

cost

数组双指针常说成 `O(n)`，但这句话必须和“为什么每个指针只往前走”一起出现才有意义。`left/right` 每次至少缩掉一个位置，`fast/slow` 每轮只让 `fast` 扫一个新元素，`p1/p2/write` 每次也只移动一格，所以一个长度为 `n` 的数组里，线性移动次数是有上界的。这里的 `n` 指当前数组长度或两个有序段长度之和，`c_left`、`c_right`、`c_fast`、`c_write` 是每类指针的常数步长成本。

$$ T(n) = c_{left} \cdot m + c_{right} \cdot m + c_{fast} \cdot n + c_{slow} \cdot n + c_{write} \cdot n = O(n) $$

`m` 表示左右夹逼时未处理区间的初始长度；`n` 表示输入长度。只要指针不回头，常数项再多，增长阶也仍然是线性的。

模板 单个位置最多被看几次 为什么不会反复回头 复杂度里真正要记住的点
左右夹逼 一次到两次 `left` 只增，`right` 只减，区间外的值不会再回来。 比较动作是线性的，前提是每次移动都能缩小区间。
快慢覆盖 一次扫描，写入最多一次 `fast` 从左到右走，`slow` 只在保留时前进。 真正贵的是读取和写回的总次数，不是重复覆盖。
逆向合并 每个元素最多写一次 `write` 只向左走，尾部已经固定的值不会被改回。 两个输入段加起来仍然是线性规模。

同一个输入，三种成本怎么看

`[-4, -1, 0, 3, 10]` 做 977 时，比较次数和数组长度同阶；`[1, 1, 1, 2, 2, 3]` 做 80 时，`fast` 扫一次，`slow` 只在保留时前进；`[1, 2, 3, 0, 0, 0] + [2, 5, 6]` 做 88 时，`p1/p2/write` 也只是各走一遍。你不用把它们算成三个不同的复杂度，只要确认没有任何一个指针会回头重扫。

复杂度不是题号标签

如果题目先排序，再做双指针，排序本身就是额外成本；如果题目要求原地但数组容量不足，复杂度不变，结果却会失败。分析复杂度时，别只记“数组双指针常见 `O(n)`”，要把前提一起说清。

线性移动伪码
while left < right:
    move_one_side()

while fast < n:
    maybe_write()
    fast += 1

while p2 >= 0:
    write_larger_tail()
    write -= 1

这段伪码只是在强调一件事：线性复杂度来自“每步都只推进一个边界”。一旦边界会回头，或者同一位置被反复重扫，`O(n)` 的前提就没了。

概念对照

compare

如果你已经把四个代表题看了一遍，接下来要做的不是背题号，而是把模板之间的边界说清。125 和 977 都是左右端点参与比较，但前者只要判断相等，后者要从尾部写结果；80 和 283 都是 `fast` 读、`slow` 写，但一个是在压缩重复，一个是在搬运零。

模板 谁读 谁写 边界怎么定 一眼辨认
左右夹逼 两端一起读 一般不写或只写答案数组 `left < right` 或 `left <= right` 看比较能不能直接决定移动
快慢覆盖 `fast` 读全数组 `slow` 写保留项 结果前缀是否已满足条件 原地删除、压缩、保留
逆向合并 `p1`、`p2` 读尾部 `write` 读写分离 `write` 从最后一格开始 两个有序段写进一个数组

边界提醒

`283 移动零` 和 `80 删除有序数组中的重复项 II` 共享 fast/slow 的结构，但目标不同：前者是把零移动到末尾，后者是限制重复次数。`75 颜色分类` 也是数组原地题，但它需要三路分区，不属于这一页的三类主模板，先别把它和 80 混成一类。

从前往后写会覆盖 `nums1` 里还没读完的有效值。结果看起来像写进去了，实际上把后面的比较依据弄丢了。
错法 最先看到的症状 真正丢掉了什么 修正方式

4. 977 有序数组的平方为什么能优先从尾部写？

参考答案 / 评分要点

因为最大平方一定来自两端绝对值更大的那一侧，先把最大值写到结果末尾，剩下部分仍然保持同样的比较结构。要点：说出“两端比较”与“尾部写入”之间的因果关系。

二、代码阅读

解释每个指针的读写职责。

回看要点先说状态，再说动作。
code reading理解

5. slow < 2 or x > nums[slow - 2] 说明了什么？

参考答案 / 评分要点

它在判断当前值保留后会不会让同一个数超过两次。`slow < 2` 时前两个位置直接放行，`x > nums[slow - 2]` 说明当前值比前两个保留值更大，保留它不会破坏“最多两个”的限制。

code reading理解

6. write 在 88 里负责什么？

参考答案 / 评分要点

`write` 指向下一次写入的位置，始终从右往左移动。它的作用是把当前较大的值放到结果尾部，确保右侧已经写好的部分不会被覆盖。

code reading理解

7. 125 中为什么要先跳过非字母数字？

参考答案 / 评分要点

因为题目比较的是有效字符是否成回文，标点和空格不参与判断。先跳过它们，`left` 和 `right` 才能对齐到真正需要比较的字符。

code reading理解

8. 977 中为什么是绝对值更大的那一侧先写？

参考答案 / 评分要点

因为平方后大小只由绝对值决定，两端里绝对值更大的那一个一定产生更大的平方。先写它，结果末尾就先被定下来，剩余问题仍然和原问题同型。

三、错误诊断

先找错因，再说后果，最后给修法。

回看要点用错一条职责，整段代码就会偏。
diagnosis错因

9. 快慢覆盖后返回 len(nums) 而不是 slow，会有什么问题？

参考答案 / 评分要点

`slow` 才是新结果长度。返回原长度会把尾部旧值也算进去，导致前缀正确但长度错误，尤其在 80 这类题里会直接错答案。

diagnosis错因

10. 88 从前往后写为什么可能覆盖未读数据？

参考答案 / 评分要点

`nums1` 前面还存着有效元素，写到前面会先把它盖掉。后面再比较时，原值已经丢了，合并顺序也就不成立了。

diagnosis边界

11. 左右指针循环条件写错，可能漏掉中间元素还是重复处理？举例说明。

参考答案 / 评分要点

两种情况都可能。写得太松会把中点多处理一遍；写得太紧会让最后一对还没比较就停掉。回文题通常用 `left < right`，填满结果数组时才常用 `left <= right`。

diagnosis边界

12. 把有序数组模板直接套到无序数组，会失去什么依据？

参考答案 / 评分要点

会失去相邻比较的意义，`nums[slow - 2]` 这种判断也不再可靠。只有有序时，重复值、大小关系和指针移动方向才有稳定依据。
