# 路线图 {#overview}
<!-- tag: map -->

本页按判断顺序读。先判输入是否有序、是否要求原地、是否需要从两端同时看，再决定 `left/right`、`fast/slow` 还是 `write` 的职责。你不用先记模板名，先把题目翻译成动作。

## box soft
### 1. 看信号
有序、原地、从两端同时看，是数组双指针最常见的入口。只要题面把这些信号说出来，就值得先往双指针上想。

## box warn
### 2. 定状态
状态变量代表的区域：`left` 和 `right` 夹住未处理区间，`slow` 前面是合法结果，`write` 右边是最终答案。

## box
### 3. 写边界
回文串通常用 `left < right`，因为中点不用单独处理；结果数组要填满时常用 `left <= right`，因为最后一个位置也要写进去。

## box info

快慢覆盖只看前缀，逆向合并只看写到哪里，左右夹逼则看区间是否缩到空。每一类模板的结果位置都不一样。

# 同一锚点，三次走法 {#trace-lab}
<!-- tag: trace lab -->

数组双指针最值得反复看的，不是题号，而是同一批输入在不同模板里会变成什么样。`[1, 1, 1, 2, 2]` 可以被 80 当成“结果前缀的压缩对象”，`[-4, -1, 0, 3, 10]` 可以被 977 当成“左右端点比较对象”，`[1, 2, 3, 0, 0, 0] + [2, 5, 6]` 则是 88 里的“待写回目标数组”。数据本身没有变，变的是谁读、谁写、谁负责把最终状态固定下来。

这也是数组双指针最容易混淆的地方。很多人会背“左边动、右边动、快慢写、尾部写”，但一旦题目换个外壳，就不知道当前变量在维护什么。你只要把“读什么、写什么、结果放在哪里”这三件事说清，模板就不会乱。

## box
### 一条最短判断线
如果题目要的是“比较两端并缩小范围”，先想左右夹逼；如果要的是“把不合格的元素挤掉，只留下合法前缀”，先想快慢覆盖；如果要的是“两个有序段合成一个有序段”，先想尾部合并。这个判断顺序比背题号更重要。

## box
### 最常见的错位
把快慢覆盖写成普通遍历，只知道 `slow` 在变，却不知道它代表结果边界；把左右夹逼写成左右乱跳，只知道边界在动，却不知道哪一次比较已经排除了哪一侧；把尾部合并写成插入，只记得有空位，却忘了有效值还在前半段。

最短判断伪码复制

```python
if input_is_sorted and need_both_ends:
 use_two_ends()
elif need_in_place and keep_prefix:
 use_fast_slow()
elif merge_into_existing_space:
 fill_from_tail()
```

这段伪码不是让你照抄的，它只是把判断顺序写短：先看输入形状，再看读写职责，最后才看题号。真正写代码时，还要把循环边界和结果位置补完整。

# 125：左右夹逼先排除外壳 {#trace-125}
<!-- tag: two ends -->

回文题最容易把难点想复杂。先忽略标点和大小写，剩下的判断只是两端是否一致。只要端点比较能直接排除一侧，就可以一边比一边向中间缩。

## box
### 正确示例：125 验证回文串
125 回文代码复制
```python
def is_palindrome(s):
 left, right = 0, len(s) - 1
 while left < right:
 while left < right and not s[left].isalnum():
 left += 1
 while left < right and not s[right].isalnum():
 right -= 1
 if s[left].lower() != s[right].lower():
 return False
 left += 1
 right -= 1
 return True
```


## box soft
### 先看什么
这段代码里，`left` 负责找左边第一个有效字符，`right` 负责找右边第一个有效字符。比较成功后，两边一起往中间缩；比较失败时，整题直接结束。
**判断句**回文题只需要在 `left < right` 时继续，因为中点不需要单独判一遍。

# 80：快慢覆盖把结果写回前缀 {#trace-80}
<!-- tag: slow fast -->

80 的关键“结果前缀保留到哪一步”。`fast` 负责把原数组从头扫到尾，`slow` 负责写下应该保留的值。只要前缀里相同值最多保留两次，循环就一直成立。

## box
### 正确示例：删除有序数组中的重复项 II
80 快慢覆盖代码复制
```python
def remove_duplicates(nums):
 slow = 0
 for x in nums:
 if slow < 2 or x > nums[slow - 2]:
 nums[slow] = x
 slow += 1
 return slow
```

# 977：先写大平方，再把剩余值往前填 {#trace-977}
<!-- tag: reverse fill -->

有序数组平方之后不再有序，所以不能简单从左到右平方。真正能决定当前位置结果的，两端的绝对值谁更大。绝对值更大的那一侧，平方一定更大，应该先写到结果数组尾部。

## box
### 正确示例：有序数组的平方
977 逆向写入代码复制
```python
def sorted_squares(nums):
 n = len(nums)
 ans = [0] * n
 left, right = 0, n - 1
 write = n - 1
 while left <= right:
 if abs(nums[left]) > abs(nums[right]):
 ans[write] = nums[left] ** 2
 left += 1
 else:
 ans[write] = nums[right] ** 2
 right -= 1
 write -= 1
 return ans
```


## box soft
### 为什么是尾部
因为当前最大平方一定属于两端之一，所以每写一个值，`write` 就可以向左挪一格。这样做的好处是，不用额外开一个队列，也不用先把结果反转回来。
**判断句**当结果需要保持从小到大，而最大值先被确定时，写指针就应该从尾部开始。

# 88：从尾部写，才不会把没读完的值盖掉 {#trace-88}
<!-- tag: merge -->

88 的陷阱很具体：`nums1` 前半段还有有效值，后半段只是空位。如果从前往后写，新的值一旦覆盖旧值，后面就再也读不回来了。尾部写的好处，就是先把最大的那一个放好，再慢慢向左补齐。

## box
### 正确示例：合并两个有序数组
88 尾部合并代码复制
```python
def merge(nums1, m, nums2, n):
 p1, p2 = m - 1, n - 1
 write = m + n - 1
 while p2 >= 0:
 if p1 >= 0 and nums1[p1] > nums2[p2]:
 nums1[write] = nums1[p1]
 p1 -= 1
 else:
 nums1[write] = nums2[p2]
 p2 -= 1
 write -= 1
```


## box warn
### 为什么不能从前面写
因为 `nums1` 的前半段还要参与比较。只要它被覆盖，后面剩下的值就失去了参照。尾部写可以把“还没读完”的风险留在左边，把“已经写完”的部分固定在右边。
**判断句**合并到一个有空位的数组时，先想尾部写，而不是先想插入。

# 复杂度和移动次数 {#complexity}
<!-- tag: cost -->

数组双指针常说成 `O(n)`，但这句话必须和“为什么每个指针只往前走”一起出现才有意义。`left/right` 每次至少缩掉一个位置，`fast/slow` 每轮只让 `fast` 扫一个新元素，`p1/p2/write` 每次也只移动一格，所以一个长度为 `n` 的数组里，线性移动次数是有上界的。这里的 `n` 指当前数组长度或两个有序段长度之和，`c_left`、`c_right`、`c_fast`、`c_write` 是每类指针的常数步长成本。

$$
T(n) = c_{left} \cdot m + c_{right} \cdot m + c_{fast} \cdot n + c_{slow} \cdot n + c_{write} \cdot n = O(n)
$$
 `m` 表示左右夹逼时未处理区间的初始长度；`n` 表示输入长度。只要指针不回头，常数项再多，增长阶也仍然是线性的。

模板
单个位置最多被看几次
为什么不会反复回头
复杂度里真正要记住的点


左右夹逼
一次到两次
`left` 只增，`right` 只减，区间外的值不会再回来。
比较动作是线性的，前提是每次移动都能缩小区间。


快慢覆盖
一次扫描，写入最多一次
`fast` 从左到右走，`slow` 只在保留时前进。
真正贵的是读取和写回的总次数，不是重复覆盖。


逆向合并
每个元素最多写一次
`write` 只向左走，尾部已经固定的值不会被改回。
两个输入段加起来仍然是线性规模。

## box soft
### 同一个输入，三种成本怎么看
`[-4, -1, 0, 3, 10]` 做 977 时，比较次数和数组长度同阶；`[1, 1, 1, 2, 2, 3]` 做 80 时，`fast` 扫一次，`slow` 只在保留时前进；`[1, 2, 3, 0, 0, 0] + [2, 5, 6]` 做 88 时，`p1/p2/write` 也只是各走一遍。你不用把它们算成三个不同的复杂度，只要确认没有任何一个指针会回头重扫。

## box warn
### 复杂度不是题号标签
如果题目先排序，再做双指针，排序本身就是额外成本；如果题目要求原地但数组容量不足，复杂度不变，结果却会失败。分析复杂度时，别只记“数组双指针常见 `O(n)`”，要把前提一起说清。

线性移动伪码复制

```python
while left < right:
 move_one_side()

while fast < n:
 maybe_write()
 fast += 1

while p2 >= 0:
 write_larger_tail()
 write -= 1
```

这段伪码只是在强调一件事：线性复杂度来自“每步都只推进一个边界”。一旦边界会回头，或者同一位置被反复重扫，`O(n)` 的前提就没了。

**Q:** 3. 88 合并两个有序数组常从哪个方向写？

- [ ] 从前往后写
- [ ] 从尾部写
- [ ] 先反转再写

Answer: 从尾部写

### 4 简答

**Q:** 4. 977 有序数组的平方为什么能优先从尾部写？

Explain: 因为最大平方一定来自两端绝对值更大的那一侧，先把最大值写到结果末尾，剩下部分仍然保持同样的比较结构。要点：说出“两端比较”与“尾部写入”之间的因果关系。

<!-- end-practice-group -->

<!-- practice-group: 二、代码阅读 -->
<!-- quick-review: 回看要点 先说状态，再说动作。 -->

### 5 code-reading

**Q:** 5. `slow < 2 or x > nums[slow - 2]` 说明了什么？

Explain: 它在判断当前值保留后会不会让同一个数超过两次。`slow < 2` 时前两个位置直接放行，`x > nums[slow - 2]` 说明当前值比前两个保留值更大，保留它不会破坏“最多两个”的限制。

### 6 code-reading

**Q:** 6. `write` 在 88 里负责什么？

Explain: `write` 指向下一次写入的位置，始终从右往左移动。它的作用是把当前较大的值放到结果尾部，确保右侧已经写好的部分不会被覆盖。

### 7 code-reading

**Q:** 7. 125 中为什么要先跳过非字母数字？

Explain: 因为题目比较的是有效字符是否成回文，标点和空格不参与判断。先跳过它们，`left` 和 `right` 才能对齐到真正需要比较的字符。

### 8 code-reading

**Q:** 8. 977 中为什么是绝对值更大的那一侧先写？

Explain: 因为平方后大小只由绝对值决定，两端里绝对值更大的那一个一定产生更大的平方。先写它，结果末尾就先被定下来，剩余问题仍然和原问题同型。

<!-- end-practice-group -->

<!-- practice-group: 三、错误诊断 -->
<!-- quick-review: 回看要点 用错一条职责，整段代码就会偏。 -->

### 9 diagnosis

**Q:** 9. 快慢覆盖后返回 `len(nums)` 而不是 `slow`，会有什么问题？

Explain: `slow` 才是新结果长度。返回原长度会把尾部旧值也算进去，导致前缀正确但长度错误，尤其在 80 这类题里会直接错答案。

### 10 diagnosis

**Q:** 10. 88 从前往后写为什么可能覆盖未读数据？

Explain: `nums1` 前面还存着有效元素，写到前面会先把它盖掉。后面再比较时，原值已经丢了，合并顺序也就不成立了。

### 11 diagnosis

**Q:** 11. 左右指针循环条件写错，可能漏掉中间元素还是重复处理？举例说明。

Explain: 两种情况都可能。写得太松会把中点多处理一遍；写得太紧会让最后一对还没比较就停掉。回文题通常用 `left < right`，填满结果数组时才常用 `left <= right`。

### 12 diagnosis

**Q:** 12. 把有序数组模板直接套到无序数组，会失去什么依据？

Explain: 会失去相邻比较的意义，`nums[slow - 2]` 这种判断也不再可靠。只有有序时，重复值、大小关系和指针移动方向才有稳定依据。

<!-- end-practice-group -->

<!-- practice-group: 四、题目迁移 -->
<!-- quick-review: 回看要点 先说模板，再说为什么。 -->

### 13 migration

**Q:** 13. 283 移动零和 80 的 fast/slow 结构有什么相同点？

Explain: 两者都用 `fast` 读全部元素，`slow` 写保留结果。区别在于 283 保留的是非零项，80 保留的是“最多两次”的有序重复项。

### 14 migration

**Q:** 14. 125 和 977 都是左右端点比较，它们的移动依据有什么不同？

Explain: 125 比较的是有效字符是否相等，相等就一起往中间缩；977 比较的是绝对值谁更大，谁更大就先把谁的平方写到结果尾部。前者是判断，后者是构造结果。

### 15 migration

**Q:** 15. 80 为什么不能直接套在无序数组上？

Explain: 因为 `slow - 2` 的比较依赖相同值挨在一起，只有有序时这个前提才成立。无序数组里，是否超过两次就不能只看相邻位置了。

### 16 migration

**Q:** 16. 用 2-3 句话解释为什么 88 的 `write` 必须从尾部开始。

Explain: 要点：两个输入段都还要继续比较，前面写会覆盖未读值。尾部写可以先固定最大的结果，再把剩余元素从右往左补齐。
