核心模板

滑动窗口核心模板

这页讲滑动窗口处理连续区间时最常用的一套模板：先判断题目是否属于连续子数组或子串，再定义 [left, right)，接着同步维护 sum、count 或 deque，最后把收缩条件和答案更新时机对齐。前缀和、差分数组和更广义的双指针放到下一节，这里不提前展开。

goals

先记一句

窗口题先看连续性，再看状态能不能增量更新。题号只是后面的标签。

路线图

map

这一页不是把题号串起来，而是把“看见题面以后怎么判断”写清楚。你先把题目翻译成窗口语言，再决定是固定窗口、可变窗口、计数窗口，还是最值窗口。

1. 先判连续

问题如果要求的是连续子数组或子串，就可以先进入窗口框架。任意子序列、回溯枚举、全局排序，都不在这个入口里。

2. 再定状态

状态通常只有几种：和、积、频次表、不同字符数、双端队列。你要先决定窗口里到底该记什么。

3. 写收缩条件

最短题一般在“已经满足条件”时收缩；最长题一般在“还不合法”时收缩；固定长度题在窗口超长时收缩。

4. 再更新答案

答案时机跟题型绑定。最短题先记答案再收缩；最长题先收缩到合法再记答案；最值题常在窗口满员后记答案。

$$ \text{窗口长度} = right - left $$

这里的公式只对应 [left, right)。如果你把右端也算进去，长度公式和边界检查都要一起改。

这里先别追

前缀和、差分数组、单调栈、回溯和区间 DP 都是邻近概念。它们会在后面的单元重新出现，但这页只负责把连续区间的在线维护讲稳。

状态台账

invariant

窗口代码真正容易错的地方，是边界和状态不同步。你写代码时，不要只盯着 left 和 right，要同时看窗口状态有没有跟着加减。

变量读什么写什么写错后的症状
left窗口左边界窗口收缩时加一长度差 1，或者离开的元素还算在窗口里
right下一个进入窗口的位置右扩张时加一当前元素漏进窗口，或者被重复计算
sum / count / prod / deque当前窗口内容加入新元素、移除旧元素窗口看起来对，状态却是旧的
ans当前合法窗口最短、最长或最值结果更新太早会丢答案，更新太晚会错过最优值
最短窗口骨架
left = 0
best = float("inf")
state = ...

for right, x in enumerate(nums):
    add(x)
    while valid():
        best = min(best, right - left + 1)
        remove(nums[left])
        left += 1

return 0 if best == float("inf") else best
最长窗口骨架
left = 0
best = 0
state = ...

for right, x in enumerate(s):
    add(x)
    while not valid():
        remove(s[left])
        left += 1
    best = max(best, right - left + 1)

return best

209 追踪

trace

下面用同一个锚点 nums = [2,3,1,2,4,3]、target = 7 走完整个收缩过程。你只看三件事：sum 什么时候第一次达标，best 什么时候更新，left 什么时候继续右移。

第一次达标 left = 0, right = 3, sum = 8, best = 4

窗口第一次跨过目标值，先把当前长度记下来，再把左边界往右推。

继续收缩 left = 3, right = 4, sum = 6, best = 3

加入 4 以后，窗口又达标。连续收缩两次，最短长度从 4 缩到 3。

拿到答案 left = 5, right = 5, sum = 3, best = 2

加入 3 以后，窗口再次达标。最后收缩到长度 2，得到最短答案。

记忆点

最短题的顺序固定：先看当前窗口是否已经可用，再尝试缩短。顺序反过来，答案就容易丢。

完整走一轮

walkthrough

把窗口题想成一条持续向右滚动的带子。right 负责把新元素读进来，left 负责把已经不该留在窗口里的元素移走。两边的职责不同，所以你的代码也要分开写：右扩张时只做“读入并更新状态”，左收缩时只做“删掉并修复状态”。如果把这两件事混在一起，最常见的结果就是状态还没更新，窗口已经变了。

209、424、438、239 的差别不在于“窗口是不是窗口”，而在于你要维护什么。209 盯住和，424 盯住最高频字符，438 盯住目标频次和当前频次的比对，239 盯住候选最大值的下标。只要状态选错，后面收缩条件写得再像模板，也会错在本质上。

把动作拆成三步

第一步是 读入：把 nums[right] 或 s[right] 放进当前状态。第二步是 修复：如果窗口不合法，就反复移动 left，直到条件重新满足。第三步才是 记录：最短题在修复前后都可能记，但要先保住当前合法窗口；最长题只在合法以后记；固定窗口题在窗口长度达到要求后记。

这三个动作如果按顺序执行，很多题会自动变成一条流水线。你不用每次从零推公式，只要确认“读入、修复、记录”各自发生在什么时机。

状态读写矩阵

窗口代码最容易错的地方，是你以为自己在改窗口，其实只是改了边界。真正的窗口应该同时更新边界和状态。加入元素时，状态要加；移除元素时，状态要减；判断合法时，读的是“加减之后”的结果，不是旧值。

所以每次写循环时，都要问自己一句：这一行读了谁、写了谁、删了谁。如果答案只有边界，没有状态，那通常意味着代码还没写完整。

题型进入窗口时写什么移出窗口时写什么合法性看什么最容易漏掉的动作
209 最短和sum += xsum -= nums[left]sum >= target先记答案再收缩
424 最长重复字符count[ch] += 1count[left_ch] -= 1window_len - max_freq <= k最高频字符数没有跟着窗口变化
438 异位词window[ch] += 1window[left_ch] -= 1window == need固定长度到达后忘了比表
239 最大值把下标塞进 deque过期下标从队首弹出窗口满员后取队首没有维护单调性，只保留了元素顺序
统一写法
def slide_window(nums):
    left = 0
    state = init_state()
    ans = init_answer()

    for right, x in enumerate(nums):
        add(state, x)

        while not valid(state):
            remove(state, nums[left])
            left += 1

        ans = update_answer(ans, state, left, right)

    return ans

这段模板的关键不是函数名，而是顺序。先加，再修，再记。你如果把 update_answer 放到 remove 之前，最短题和最长题都会出现不同程度的偏差。最短题会把过大的窗口算进去，最长题会把还没清理干净的非法窗口当成合法窗口。

更细一点看，209 和 424 的“修”也不是同一种修法。209 修的是和，424 修的是频次和最高频字符的差值。前者关注数值累计，后者关注字符分布。438 则更像对账：窗口长度到了以后，检查窗口表是不是和目标表一致。239 再往前一步，它连“最大值”都不能用普通加减维护，必须把下标顺序和候选顺序同时保存起来。

失败后果

如果你只记住“窗口是 left/right”，但没把状态写进骨架，题目一换，代码还是会漂。窗口题真正复用的不是题号，而是“状态如何增删、条件如何判断、答案何时落笔”这条顺序。

无重复子串再走一轮

题目 s = "abcabcbb" 看的就是窗口里是否出现重复字符。新字符进来后，如果它上次出现的位置还在窗口内，left 就跳到那次出现位置的下一格；这样窗口里始终保留一段没有重复字符的连续子串。

读入窗口last seenleft动作
aaa:00先记长度 1
babb:10再记长度 2
cabcc:20当前最优是 3
abcaa:31重复字符在窗口内，left 跳到 1
bcabb:42继续沿着 last seen 前移
cabcc:53窗口长度仍然是 3
无重复子串写法
seen = {}
left = 0
best = 0

for right, ch in enumerate(s):
    if ch in seen and seen[ch] >= left:
        left = seen[ch] + 1
    seen[ch] = right
    best = max(best, right - left + 1)

return best

这里记录的是字符最后一次出现的位置，不是出现次数。只要最后一次出现的位置还落在窗口里，left 就要跳过去；如果上次出现的位置已经在窗口左侧，这个字符就不再构成冲突。

438 对账

fixed window

438 找字母异位词是固定窗口里最适合练“对账”的一题。给定 s = "cbaebabacd" 和 p = "abc"，你不是在找字符顺序，而是在找长度为 len(p) 的窗口是否和目标频次完全一致。这里的关键动作很简单：先把 p 的频次表 need 统计好，再让窗口上的 window 一步一步滑过去，窗口到达固定长度后，比一次表，再把左端移走。

这个例子很容易把人带偏，因为文本里既有重复字符，也有连续匹配。你如果把它当成“看到一个字符就立刻比对”，就会忘掉固定窗口的长度约束；你如果把它当成“窗口不满也能比”，就会在前半段反复误报。真正稳定的做法是：窗口长度先满，表再比对，最后滑动一步。这和 209 的“先判断合法再收缩”是同一类思路，只是一个是可变窗口，一个是固定窗口。

右端读入窗口内容window / need是否记录下一步
cc不相等否继续扩张
bcb不相等否继续扩张
acba完全相等记录起点 0移走左端 c
ebae不相等否继续滑动
baeb不相等否继续滑动
aaba不相等否继续滑动
cbac完全相等记录起点 6继续向右
438 固定窗口写法
need = Counter(p)
window = Counter()
left = 0
ans = []

for right, ch in enumerate(s):
    window[ch] += 1

    if right - left + 1 == len(p):
        if window == need:
            ans.append(left)

        left_ch = s[left]
        window[left_ch] -= 1
        if window[left_ch] == 0:
            del window[left_ch]
        left += 1

这段代码很值得单独看一遍。它先把右端加进来，再检查当前窗口是不是已经满长度；只有满了，才有资格和 need 对账；对账之后立刻移走左端，窗口才会继续滑动。你如果把“比表”和“移走左端”调换顺序，下一轮窗口就少了一个字符，匹配位置会整体偏移。这个偏移通常不会在第一组样例里暴露出来，但在重复字符很多的输入里会马上错位。

更进一步，438 还能帮你区分固定窗口和可变窗口。固定窗口里，长度本身就是一条约束，所以你只要控制好窗口边界和频次表，答案会自然落出来；可变窗口里，长度只是结果，不是条件。把这两件事分开，后面你看 424、1004、713 时就不会把“长度”误当成“条件”。

练习入口

自己手算一遍 s = "abab"、p = "ab"：写出每个窗口的 need、window、left 和是否命中。只要能把这张表写出来，你就真正理解了固定窗口的对账动作。

1004 零计数

longest window

1004 最大连续 1 III 是最长可变窗口里最适合练“先修后记”的题。题目给你一个 0/1 数组和一个最多能翻转的零数量 k，你真正维护的不是“1 的个数”，而是“当前窗口里有几个 0”。一旦 zeros > k，窗口就不合法，左边界必须继续往右移，直到零数重新回到允许范围内。

这个过程和 209 正好相反。209 是“窗口已经合法，就先记录最短值，再继续收缩”；1004 是“窗口已经越界，就先收缩到合法，再记录最长值”。很多人做 1004 时会把这两类顺序混起来，结果就是把越界窗口也记进答案，或者只收缩一次就停下来。题目表面上看是 0 和 1 的翻转，真正考的其实是你能不能分清“合法时记答案”和“非法时先修复”的顺序。

读入片段窗口zeros动作best
1,1,1[0, 3)0窗口合法，记 33
1,1,0[0, 4)1仍合法，记 44
1,0,1[0, 5)1仍合法，记 55
1,0,0[0, 6)2超过 k，开始收缩5
0,0,1[2, 7)2继续收缩直到合法5

1004 最长窗口写法
left = 0
zeros = 0
best = 0

for right, x in enumerate(nums):
    if x == 0:
        zeros += 1

    while zeros > k:
        if nums[left] == 0:
            zeros -= 1
        left += 1

    best = max(best, right - left + 1)

这段代码最重要的不是 zeros 这个变量本身，而是 while 后面的更新顺序。你必须先把窗口缩到合法，再写 best = max(...)。如果你把更新放在 while 之前，答案会包含越界片段；如果你只缩一次就停，窗口可能还不合法，长出来的长度没有意义。

1004 也特别适合拿来和 438 对照。438 是固定长度窗口，窗口一旦满了就比表；1004 是可变长度窗口，窗口一旦越界就收缩。一个看长度，一个看约束；一个先比对再滑动，一个先修复再记录。把这两种顺序分开后，你在 424、713、239 里也能更快判断应该先做哪一步。

练习入口

把 nums = [1,1,1,0,0,1,1,0,1]、k = 1 手工写成表，每一行记录 left、right、zeros、best。只要你能把整条轨迹写出来，你就已经把“最长可变窗口”的顺序真正跑顺了。

题型对照

patterns

窗口分类先看状态，再看收缩信号。你不需要先记住题号，只要能把题面翻译成“维护什么、什么时候缩、什么时候记答案”。

窗口家族状态收缩信号答案时机常见题
固定长度deque / countright - left + 1 > k窗口满员后219、239
最短可变sum / prod已经满足目标先记再缩209、713
最长可变count / distinct / zeros还不合法缩到合法后再记3、1004、424
匹配窗口need / window长度或频次不对齐窗口长度固定时检查438

判断顺序

  1. 先看题目是不是连续区间。
  2. 再看窗口里要维护什么状态。
  3. 然后写收缩条件。
  4. 最后定答案是在收缩前还是收缩后更新。

这里先不混

前缀和、差分数组、单调栈和回溯都不是这页的主角。它们会在后面的单元出现，但这里先把窗口的增量维护练熟。

239 队列实验

deque

239 的窗口最值不能靠每步扫一遍 max(window)。你要单独保存“当前窗口里还可能当最大值的候选”，并在元素过期时从队首删掉。

239 单调队列骨架
dq = deque()  # 存下标，队首是当前最大值候选
ans = []

for right, x in enumerate(nums):
    while dq and nums[dq[-1]] <= x:
        dq.pop()
    dq.append(right)

    if dq[0] <= right - k:
        dq.popleft()

    if right >= k - 1:
        ans.append(nums[dq[0]])

2-1[3, -1]窗口满 3 个时输出 33
3-3[3, -1, -3]队首还是 3，没有过期3
45[5]5 进来后把更小的候选全删掉5
53[5, 3]3 不能超过 5，保留5
66[6]6 把 5 和 3 都压下去6
77[7]继续弹出更小候选7
最值窗口的判断

如果你能用 sum 或 count 增量维护，就先别把队列扯进来；只有最值这类“会被中间元素挡住”的信息，才需要单调队列。

失败实验室

lab

下面三个终端输出是窗口题里最常见的三种失误。它们看起来像三类不同的问题，其实都在问同一件事：你有没有把边界、状态和答案时机同步起来。先看症状，再找对应的修复点，不要上来就改一大堆代码。

[case-1] 209 最短连续子数组
input: nums = [2,3,1,2,4,3], target = 7
window: [0, 4) sum = 8
observed: shrink first, then record
result: best stays 4 even though [3,1,2,4] is shorter
symptom: answer too large
check: did you lose the exact moment when sum first became valid?

排错顺序

先看窗口现在是不是合法，再看状态是不是同步，再看答案是不是在正确时机被写下。顺序不要倒过来。你一旦先去改算法名字，通常只会把问题藏得更深。

先别误判

如果题目问的是任意子集、全局排列、回溯组合，窗口模板就不对。窗口只管连续区间。

后续衔接

bridge

这页的目标，不是把所有窗口题讲完，而是让你在看到新题时知道该先检查什么。写题时按“连续性 -> 状态 -> 收缩条件 -> 答案时机”四步走，基本就能把模板落下来。等你真的动手写题，最有用的不是“记住一段话”，而是先把这四步写成自己的检查顺序：题面里有没有连续区间，窗口状态是什么，什么时候必须收缩，答案是在收缩前还是收缩后更新。

如果题目改成“最短连续子数组”，你先写 sum；如果改成“最长子串，最多 k 个不同字符”，你先写频次表和不同字符数；如果改成“固定长度最大值”，你就该立刻想到单调队列。这里的迁移不是公式迁移，而是状态迁移。窗口模板最怕的就是把一个题的状态直接搬到另一个题上，结果看起来在用同一个框架，实际上每个变量都不对。

下一步建议

  1. 先独立写一遍 209，不看答案，确认你能把最短窗口的顺序写对。
  2. 再做 438 和 424，检查你能不能把频次表和窗口长度对齐。
  3. 最后做 239，确认你知道什么时候该从普通窗口切到单调队列。

如果你在纸上复盘，建议只画三列：left/right、状态、答案。每次窗口右移一格，就在三列下面补一行。这个方法看起来朴素，但它能很快暴露“状态没有同步”还是“答案时机写错”。很多窗口题不是想不出来，而是你没有把变化过程显式写出来，所以不知道错发生在哪一步。

9. 混用 [left, right] 和 [left, right) 会先影响哪两件事？

回答时把“长度公式”和“边界移动后的含义”都写出来。

首先影响长度公式：半开区间用 right - left，闭区间要改成 right - left + 1。其次影响边界移动的解释：right 代表的是“下一个进入窗口的位置”，还是“已经包含在窗口里的最后一个位置”。这两个定义不能混着写。

12. 如果题目改成“最长子串，最多 k 个不同字符”，你会怎么写窗口？

答案至少要说清：状态是什么、什么时候收缩、什么时候更新最长值。

参考答案 / 评分要点

状态通常是字符频次表和不同字符个数。右边加入字符时更新频次；如果不同字符数超过 k，就不断移动 left 并删除左端字符，直到重新合法。窗口合法后再更新最长长度。回答里只要把“状态 / 收缩 / 更新时机”三件事写清楚，就算完整。

14. 为什么无重复子串更适合用 last seen 记录，而不是只记 count？

回答时说清：count 缺什么信息，last seen 多了什么信息。

参考答案 / 评分要点

count 只能告诉你某个字符有没有重复，不能告诉你该把 left 跳到哪里。last seen 提供的是最后一次出现的具体位置，所以可以直接把 left 推到冲突字符之后。无重复子串的关键不是“有没有重复”，而是“重复时边界该跳到哪”。
