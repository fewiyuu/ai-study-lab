# 主线地图 {#overview}
<!-- tag: route -->

前缀和和差分数组看起来像一对反向操作，但它们在做两种不同的事：前缀和把“过去累计了多少”存起来，差分把“从哪里开始变、到哪里停”存起来。一个面向查询，一个面向更新。

## box soft
### 先判断题目在问什么
如果题目反复问某段和，先想前缀和；如果题目反复让某段整体加减，先想差分数组。题目里出现“很多次”“批量”“最后统一输出”时，通常意味着你不该逐次扫描区间。

## box warn
### 这页用同一个小数组贯穿
后面的例子都围绕 `nums = [2, 3, 1, 4]` 和 `nums = [1, 2, 3, 4, 5]` 来看。这样你能看清前缀和如何累计，差分如何记录边界，再看二维和 560 时就不会换一套思路。

## box
先看问题是读区间还是写区间，再看它是一维还是二维。只要题面是“静态查询”，就从前缀和开始；只要题面是“批量更新”，就从差分数组开始；二维矩阵里的区域和，仍然属于前缀和，只是相减对象从线段换成了矩形。

# 前缀和公式 {#prefix-formula}
<!-- tag: prefix -->

一维前缀和数组 `pre` 的定义很固定：`pre[i]` 表示原数组前 `i` 个元素的和。这样 `pre[0] = 0`，`pre[1] = nums[0]`，`pre[2] = nums[0] + nums[1]`。多一个 0 的作用，是把从 0 开始的区间也纳入同一套公式。

$$pre[i] = \sum_{j=0}^{i-1} nums[j]$$
 $$sum(l, r) = pre[r + 1] - pre[l]$$

对象读什么写什么出错症状


`pre[i + 1]``pre[i]` 和 `nums[i]`当前位置之前的累计和少一个位置时，`[0, r]` 会额外分支
`sum(l, r)``pre[r + 1]` 与 `pre[l]`区间总和把 `r` 写成 `r + 1` 之前忘了同步左边界，就会差一

# 前缀和台账 {#build-ledger}
<!-- tag: ledger -->

前缀和先把累计过程记成一张表。写表时，你真正维护的是一个 `running` 值：每读到一个元素，就把当前值加上去，再把结果存到下一个前缀位置。这样做的好处很朴素，后面每次查询都不用再回头扫一段数组。

还是拿 `nums = [2, 3, 1, 4]` 来看。`running` 会依次变成 `2`、`5`、`6`、`10`。你看到的 `pre` 其实只是这个累计过程的快照。只要这张快照和定义对上，`pre[r + 1] - pre[l]` 才能直接读出区间和。

前缀和构造复制

```python
pre = [0] * (len(nums) + 1)
running = 0
for i, x in enumerate(nums):
 running += x
 pre[i + 1] = running
```

步骤读什么写什么如果写错


初始化空表和 `running``pre[0] = 0`从 0 开始的区间会多出分支
读 `nums[i]`当前元素`running += nums[i]`少加一次，后面所有区间都偏小
写 `pre[i+1]`刚更新的累计值把累计值存进去写到 `pre[i]` 会把边界整体左移

## box
如果你把
pre[i]
直接解释成“到下标
i
为止的和”，公式就会和数组长度对不上。结果通常你会悄悄在
r
和
r + 1
之间来回补丁，最后把一张统一的表写成了两套规则。前缀和的价值恰恰在于它把这类补丁抹平。

# 560 追踪 {#trace-560}
<!-- tag: trace -->

560 和为 K 的子数组在问“有多少个连续子数组满足条件”。这时，前缀和要配哈希表。当前前缀和记作 `s`，如果以前出现过 `s - k`，那么这两次前缀之间夹着的一段子数组和就是 `k`。

步骤当前值哈希表读写答案变化


初始`s = 0``count[0] = 1`先把“从头开始”这一种情况记进去
读取一个数`s += nums[i]`先查 `count[s - k]`旧前缀出现几次，答案就增加几次
记录当前前缀`s``count[s] += 1`当前前缀只在查完以后记入表中

560 核心顺序复制

```python
count = {0: 1}
s = 0
ans = 0
for x in nums:
 s += x
 ans += count.get(s - k, 0)
 count[s] = count.get(s, 0) + 1
```

## box
先查旧前缀，再记当前前缀。这样做的原因很直接：当前前缀还不能拿来和自己配对，否则
k = 0
时会把空区间也算进去。


---

# 二维前缀和 {#two-d-prefix}
<!-- tag: 2d -->

二维前缀和仍然是“先累计，再查询”，只是区间从线段变成了矩形。`pre[i][j]` 表示左上角到 `(i-1, j-1)` 的矩形和，所以它也要多一行和一列 0。查一个矩形时，做法是先取大矩形，再减去上方和左方，最后把左上角重叠部分加回来。

$$sum(x_1, y_1, x_2, y_2)=pre[x_2+1][y_2+1]-pre[x_1][y_2+1]-pre[x_2+1][y_1]+pre[x_1][y_1]$$

矩形块作用读什么


大矩形先覆盖目标区域`pre[x2 + 1][y2 + 1]`
上方减去去掉多算的上边`pre[x1][y2 + 1]`
左方减去去掉多算的左边`pre[x2 + 1][y1]`
左上角加回补回被减两次的部分`pre[x1][y1]`

## box warn
这里先别追更高阶的矩阵更新结构。二维前缀和只负责静态求和；如果矩阵频繁改值，后面再看更适合的结构。

# 差分定义 {#diff-definition}
<!-- tag: diff -->

差分数组可以看成“前缀和的反向记录”。`diff[i]` 表示 `nums[i]` 相对 `nums[i - 1]` 改了多少。只要知道每个位置的变化量，就可以从左到右累计把原数组还原出来。它最适合处理“很多次区间加减，最后才要结果”的问题。

对象读什么写什么出错症状


`diff[0]`原数组第一个值起点基准首项没设好，后面累计全偏
`diff[l] += v`区间起点从 `l` 开始增加 `v`只改中间元素时，区间边界会断掉
`diff[r + 1] -= v`区间终点后一个位置在 `r` 后停止增加忘了停住，就会把后面的值也抬高

差分更新与还原复制

```python
diff = [0] * len(nums)
diff[0] = nums[0]
for i in range(1, len(nums)):
 diff[i] = nums[i] - nums[i - 1]

diff[l] += v
if r + 1 < len(diff):
 diff[r + 1] -= v

running = 0
for i in range(len(diff)):
 running += diff[i]
 nums[i] = running
```

# 完整算例 {#worked-case}
<!-- tag: case -->

下面把同一条主线拆成三个小算例。你会看到前缀和先把“累计”做完，560 先把“旧前缀次数”记住，差分数组先把“区间边界”标出来。三个算例都很短，但它们足够说明为什么这页的工具是这三个，而不是别的。

算例 1：一维前缀和
`nums = [2, 3, 1, 4]`。先建 `pre = [0, 2, 5, 6, 10]`，再查闭区间 `[1, 3]`。公式是 `pre[4] - pre[1] = 10 - 2 = 8`，对应的就是 `3 + 1 + 4`。这里最值得盯住的，不是算术，而是边界：右端点要右移一位，左端点保持原样。


位置`nums``pre`含义


020 → 2前 1 个元素之和
132 → 5前 2 个元素之和
215 → 6前 3 个元素之和
346 → 10前 4 个元素之和


算例 2：560 的计数顺序
`nums = [1, 1, 1]`，`k = 2`。第一个 1 让前缀和变成 1，此时要找的是 `-1`，没有结果；第二个 1 让前缀和变成 2，要找的是 `0`，这时 `count[0] = 1`，答案加 1；第三个 1 让前缀和变成 3，要找的是 `1`，又能找到一次。最后答案是 2。


扫描到的位置当前 `s`需要的旧前缀答案


第 1 个 11-10
第 2 个 1201
第 3 个 1312


算例 3：差分数组更新
`nums = [1, 2, 3, 4, 5]`。先把它转成差分 `[1, 1, 1, 1, 1]`，因为每一项都比前一项大 1。现在给 `[1, 3]` 加 `2`，只要改两处：`diff[1] += 2`，`diff[4] -= 2`。改完以后，差分表变成 `[1, 3, 1, 1, -1]`，再做一次前缀累计，就得到 `[1, 4, 5, 6, 5]`。


阶段`diff``running`结果


初始[1, 1, 1, 1, 1]0原数组未改动
写边界[1, 3, 1, 1, -1]0只记录变化，不直接改内部值
还原后[1, 3, 1, 1, -1]逐步累计[1, 4, 5, 6, 5]

# 常见误区 {#mistakes}
<!-- tag: traps -->
<!-- layout: grid2 -->

## box
### 把 `pre[i]` 当成到下标 `i` 的和
**错因：**定义没对齐。
**后果：**闭区间公式会一直差一，`[0, r]` 还要单独分支。
**修法：**固定记成“前 `i` 个元素之和”。

## box
### 560 先记当前前缀，再去查旧前缀
**错因：**更新时机写反了。
**后果：**`k = 0` 时会把空区间算进去。
**修法：**先查 `s - k`，再把当前 `s` 记入哈希表。

## box
### 差分只改起点，不改终点后
**错因：**只记住“开始生效”，没记住“停止生效”。
**后果：**区间后面的值也被一起抬高。
**修法：**记住 `diff[l] += v` 和 `diff[r + 1] -= v` 是一对。

## box
### 把差分数组直接当结果
**错因：**把变化量当成了最终值。
**后果：**输出看起来像边界表，不像数组本身。
**修法：**最后一定做一次前缀累计还原。

# 二维前缀的切面 {#matrix-case}
<!-- tag: matrix -->

二维前缀和的关键，看懂“多算了哪一块”。矩形查询要从左上角铺到右下角，所以每个位置都表示“到这里为止的总和”。查询一个子矩形时，先把整块拿出来，再减掉上边和左边，最后把左上角重复扣掉的那一块加回来。

ps[i + 1][j + 1] = mat[i][j] + ps[i][j + 1] + ps[i + 1][j] - ps[i][j]

二维前缀建表复制

```python
mat = [
 [1, 2, 0],
 [3, 4, 5],
 [2, 1, 1],
]
ps = [[0] * (len(mat[0]) + 1) for _ in range(len(mat) + 1)]
for i in range(len(mat)):
 for j in range(len(mat[0])):
 ps[i + 1][j + 1] = (
 mat[i][j]
 + ps[i][j + 1]
 + ps[i + 1][j]
 - ps[i][j]
 )
```

位置`mat``ps`含义


(0, 0)11只看左上第一个格子
(0, 1)23第一行前两格之和
(1, 1)410左上 2 x 2 的总和
(2, 2)119整张矩阵的总和

## box
例如查询左上角为
(1, 1)
、右下角为
(2, 2)
的子矩形时，结果是
ps[3][3] - ps[1][3] - ps[3][1] + ps[1][1] = 19 - 3 - 6 + 1 = 11
。这就是“先铺满，再切回边界”的完整过程。

# 差分多次更新 {#range-add-case}
<!-- tag: range-add -->

差分数组真正省力的地方，是把一连串区间修改压成事件流。你不需要每次都去改整段元素，只要记住每次更新从哪里开始、到哪里结束，然后在最后统一还原。这样写一次更新，成本就和写两个边界差不多。

多次区间更新复制

```python
updates = [
 (1, 3, 2),
 (0, 2, -1),
 (2, 4, 3),
]
diff = [0, 0, 0, 0, 0]
for l, r, v in updates:
 diff[l] += v
 if r + 1 < len(diff):
 diff[r + 1] -= v

running = 0
for i in range(len(diff)):
 running += diff[i]
 diff[i] = running
```

更新写入起点写入终点后`diff` 状态


`[1, 3] += 2``diff[1] += 2``diff[4] -= 2``[0, 2, 0, 0, -2]`
`[0, 2] += -1``diff[0] -= 1``diff[3] += 1``[-1, 2, 0, 1, -2]`
`[2, 4] += 3``diff[2] += 3`越界，不再写停点`[-1, 2, 3, 1, -2]`

## box warn
如果把
diff
直接当结果看，你拿到的是变化量，不是最终数组。差分数组先记事件，前缀累计才负责把事件变回数值。

# 收官检查 {#boundary-check}
<!-- tag: check -->

这一页到这里就该收束了。最后只核对四件事：一维前缀和的边界、560 的记表顺序、二维前缀的四角相减、差分数组的终点后写入。只要这四件事能自己说清楚，题面换个说法也不会把你带偏。

场景先做什么最后补什么一句话判断


一维区间和建前缀和`pre[r + 1] - pre[l]`查询多就用前缀和
560 子数组计数先查旧前缀再记当前前缀顺序反了就会多算
二维矩形和建二维前缀四角相减再补回左上边界外扩一格最稳

## box
如果题目同时需要频繁更新和频繁查询，就该换到树状数组或线段树。前缀和和差分数组负责的是“把这一类边界想清楚”，不是替所有动态区间问题兜底处理。

# 混合题面的判定链 {#decision-chain}
<!-- tag: decision -->

有些题不会直接把工具名字写在标题里，而是先让你查一次区间和，再让你做几次区间修改，最后再查一次结果。遇到这种题，先别急着套单一模板。你要先把每一步的动作拆开，再决定哪些步骤交给前缀和，哪些步骤交给差分数组。

下面这个小场景最适合拿来练判断：`nums = [4, 1, 3, 2, 5]`，先问 `[1, 3]` 的和，再把 `[2, 4]` 全部加 `2`，最后再问一次 `[0, 4]` 的和。第一次是查询，第二次是更新，第三次还是查询。这个顺序一旦拆开，你就会看到两种工具各自负责什么。

判定链伪代码复制

```python
nums = [4, 1, 3, 2, 5]

# 第一步：前缀和只负责静态查询
pre = [0]
for x in nums:
 pre.append(pre[-1] + x)
query_1 = pre[4] - pre[1] # [1, 3]

# 第二步：差分数组只负责批量更新
diff = [nums[0]]
for i in range(1, len(nums)):
 diff.append(nums[i] - nums[i - 1])
diff[2] += 2
if 5 < len(diff):
 diff[5] -= 2

# 第三步：还原后再查一次
running = 0
restored = []
for v in diff:
 running += v
 restored.append(running)
query_2 = sum(restored[0:5])
```

步骤输入工具动作结果


第一次查询`[1, 3]`读 `pre[4] - pre[1]``1 + 3 + 2 = 6`
区间更新`[2, 4] += 2`写 `diff[2]` 和终点后边界`[4, 1, 5, 4, 7]`
还原后再查`[0, 4]`前缀累计后求整段和`21`

## box
如果你把这类题一律当成前缀和处理，每次更新都得重建整张表；如果你把它一律当成差分处理，又会在静态查询上多绕一圈。真正要做的是先看题面里的动作顺序，再把查询和更新分到各自最省力的工具上。

## box
练习时可以自己换一组数：先写出第一次查询的前缀和答案，再手算一次区间更新，最后把还原后的数组重新求和。只要你能把这三步说顺，混合题面就不会再显得乱。

### 16 综合

**Q:** 16. 题面先问区间和，再做一次区间加，再问一次区间和，你会怎么拆？

Explain: 先把第一次区间和交给前缀和，用 pre[r+1]-pre[l] 直接查。中间的区间加交给差分数组，只写起点和终点后边界。最后一次区间和要先把差分还原成数组，再根据新的数组去查。

### 17 二维

**Q:** 17. 为什么二维前缀和的查询要先减上边和左边，再加回左上角？

Explain: 因为大矩形里包含了目标矩形上方和左方的区域，减掉它们以后，左上角那一块会被减两次，所以要加回一次。这样才能只保留目标子矩形的和。

### 18 差分

**Q:** 18. 差分数组做多次更新时，为什么最后的还原步骤不能省？

Explain: diff 只记录每个边界的变化，不是每个位置的最终值。还原步骤把这些变化按位置累计起来，才能得到真正的数组。如果省掉这一步，输出就只是边界事件，不是题目要的结果。

### 19 比较

**Q:** 19. 用 4 句话比较“前缀和查询”和“差分更新”的心法差异。

Explain: 前缀和先把累计做完，所以后面查询很快。差分数组先把变化记好，所以中间更新很快。前缀和看的是“到这里为止总共有多少”，差分数组看的是“从这里开始变化多少”。一个偏查询，一个偏更新，适用场景不一样。

### 4 code-reading

**Q:** 4. 为什么一维前缀和的区间公式是 `pre[r+1]-pre[l]`？

Explain: pre[r+1] 表示从数组开头到 r 的累计和， pre[l] 表示从数组开头到 l-1 的累计和。相减后，前面多出来的部分被抵消，只剩 l 到 r 。如果把右边界写成 pre[r] ，就会少算 nums[r] 。

### 5 choice

**Q:** 5. 560 中为什么要先查 `count[s-k]`，再记录 `count[s]`？

- [ ] 避免把当前前缀和自己配对
- [ ] 为了让哈希表更快
- [ ] 因为前缀和数组长度更短

Answer: 避免把当前前缀和自己配对
Explain: 先查旧前缀，才能避免把当前前缀和自己配成空区间。

### 6 choice

**Q:** 6. 差分更新中，为什么要写 `diff[r+1] -= v`？

- [ ] 让区间后面停止增加
- [ ] 让起点再加一次
- [ ] 让原数组长度变长

Answer: 让区间后面停止增加
Explain: 起点开始生效，终点后停止生效，边界一对一配合。

### 7 诊断

**Q:** 7. 把 `pre[i]` 理解成“前 `i` 个元素之和”时，为什么查询 `[0, r]` 不需要特殊判断？

Explain: 因为 pre[0] = 0 ，所以 [0, r] 可以直接写成 pre[r+1] - pre[0] 。没有这个 0，就得额外写一条 l = 0 的分支，公式就不统一了。

### 8 diagnosis

**Q:** 8. 560 里如果把 `count[0]=1` 忘掉，会漏掉什么？

- [ ] 从数组开头开始的子数组
- [ ] 所有长度为 1 的子数组
- [ ] 所有负数子数组

Answer: 从数组开头开始的子数组
Explain: 前缀和本身等于 k 的情况会被漏掉。

### 9 debug-checklist

**Q:** 9. 差分数组只改 `diff[l]+=v`，不改 `diff[r+1]-=v`，后面会出现什么症状？

- [ ] 区间后面的值也会被一起抬高
- [ ] 更新看起来像没有停止点
- [ ] 前缀累计后仍然完全正确

Explain: 前两项是对的。差分数组靠一对边界控制区间的开始和结束，只写起点不写终点后，增量会一直往后传，最后还原出来的数组会整体偏高。

### 10 诊断

**Q:** 10. 差分数组最后为什么必须做一次前缀累计？

Explain: diff 只记录变化量，不是最终值。前缀累计能把“从哪里开始变、到哪里停”转换成每个位置当前累计了多少变化，最后才能还原出真正的数组。

### 11 迁移

**Q:** 11. 为什么 1000 次静态区间求和查询适合前缀和？

Explain: 前缀和先花 O(n) 建表，后面每次区间和都是 O(1) 。查询次数很多时，总成本比每次扫描区间低得多，尤其是数组内容不变的场景。

### 12 迁移

**Q:** 12. 如果数组频繁更新又频繁查询区间和，为什么简单前缀和不够？

Explain: 简单前缀和适合静态数组，一旦中间有更新，前缀表就要重新维护。更新和查询都频繁时，前缀和的优势会被解决，需要更适合动态维护的结构。

### 13 synthesis

**Q:** 13. 用 6-8 句话讲清前缀和、560 和差分数组的完整主线。

Explain: 前缀和把静态区间和变成相减，所以先建 pre[0]=0 的表，再用 pre[r+1]-pre[l] 统一查询。560 统计很多个连续子数组，所以要边扫边记旧前缀 s-k 的出现次数，并先放入 count[0]=1 。差分数组则是另一条路，它把区间更新写成 diff[l] 和 diff[r+1] 两个边界。等所有更新写完，再做一次前缀累计还原。前缀和适合“建表一次、查询很多次”，差分适合“更新很多次、最后统一输出”。如果更新和查询都很多，就该考虑更高阶的动态结构。

### 14 synthesis

**Q:** 14. 用自己的话说清“前缀和适合查询，差分数组适合更新”这句话为什么成立。

Explain: 前缀和把成本先放到建表上，之后每个查询都只要两次数组访问，所以适合查询多、更新少的场景。差分数组把成本放到更新边界上，每次只写两处，最后统一还原，所以适合更新多、最后再看结果的场景。两者的“快”都来自把重复工作提前或延后，而不是把整段都重新扫一遍。

### 15 synthesis

**Q:** 15. 如果面试官追问你二维前缀和和一维前缀和的区别，你会怎么答？

Explain: 一维前缀和处理的是线段区间和，公式是 pre[r+1]-pre[l] 。二维前缀和处理的是矩形区域和，查询时要用大矩形减去上方、左方，再补回左上角重叠部分。两者都属于“先预处理，再 O(1) 查询”，只是一个把边界落在一维下标上，另一个把边界落在两个坐标上。
