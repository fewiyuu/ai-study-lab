# 概念地图 {#overview}
<!-- tag: mental model -->

二叉树题最容易混的是“遍历顺序”和“问题类型”。顺序描述你什么时候处理当前节点，问题类型描述答案依赖谁。祖先信息往下传时，通常是遍历；子树结果往上合并时，通常是分解。这个页面只讲前者，因为路径题、视图题、奇偶题都靠这条线。

## box
### 入口先看什么
先确认 node 是否为空。为空就是终止；叶子判断放到后面，因为只有叶子节点才是很多根到叶题真正收集答案的位置。最后才是递归展开左右子树。
这个顺序很重要。很多错法源自边界顺序被打乱：把空节点当叶子、把收集答案放在返回之后、或者把恢复状态放到了递归之前。

## box soft
### 你要记住的三个句子
**进入节点时**：把当前节点的值合进状态。
**碰到叶子时**：把当前状态转成答案。
**返回之前**：如果状态是共享对象，恢复到进入前。

遍历位置
你在做什么
适合什么题
常见误区


前序
先处理当前节点，再交给左右子树。
根到叶路径、累计值、路径奇偶性。
忘了把当前节点先写进状态。


中序
左子树和当前节点之间有一次处理。
二叉搜索树相关题。
把中序当成默认树题入口。


后序
先拿到左右子树结果，再处理当前节点。
高度、直径、是否平衡。
把应当向下传的状态误写成向上合并。


层序
按层处理，队列里一层一层弹出。
右视图、每层最大值、最短层数。
层数和深度下标没有统一。

# 入口与状态 {#mechanism}
<!-- tag: state -->

把递归写稳，关键在于函数职责要窄。你只要让函数干三件事：处理当前节点、把相同任务交给左右子树、返回前恢复现场。这样写出来的 DFS，才能从 257 跑到 129，再跑到 1457，而不用每题都重写一版心法。

## box
### 一个稳定的入口模板
路径题里，`path` 表示的是“从根到当前节点的祖先链”。这个定义必须稳定，否则你在叶子处收集时，自己都不知道 path 里到底有没有当前节点、要不要把它弹出。
累计值题里，`cur` 更像一条单向账本。它不需要回退成某个中间状态，因为每次递归都拿到一个新的数值传给孩子，返回时只要把父节点那一层的局部变量丢掉就行。

## box info
### 读代码时的顺序
1. 先看终止条件。
2. 再看状态怎么更新。
3. 然后看收集答案的位置。
4. 最后看有没有恢复动作。
如果你能把这四步顺着说完，树题的代码通常就已经写对了一半。

通用 DFS 入口模板
复制

```python
def dfs(node, path):
 if node is None:
 return

 path.append(node.val)

 if node.left is None and node.right is None:
 ans.append("->".join(map(str, path)))
 else:
 dfs(node.left, path)
 dfs(node.right, path)

 path.pop()
```

这个模板最值得盯住的是 `append`、`收集`、`pop` 三个动作的顺序。顺序一变，路径就可能串台。

# 例题与代码 {#example}
<!-- tag: examples -->

旧页最有价值的部分是题材本身：257、129、199、1457 这些题四种状态写法。路径题用列表，累计值题用整数，视图题用深度位置，奇偶题用位掩码。你把状态换掉，入口结构却几乎不变。

## box
### 257：所有路径
这题最重要的保证 `path` 的生命周期只覆盖当前分支。收集答案的位置在叶子，恢复位置在函数返回前。
257 二叉树的所有路径
复制
```python
def binaryTreePaths(root):
 ans = []
 path = []

 def dfs(node):
 if node is None:
 return

 path.append(str(node.val))
 if node.left is None and node.right is None:
 ans.append("->".join(path))
 else:
 dfs(node.left)
 dfs(node.right)
 path.pop()

 dfs(root)
 return ans
```


## box
### 129：求根到叶数字之和
这题把 path 换成 `cur`，就不需要共享列表了。每往下走一层，就把当前节点“接到数字后面”，也就是 `cur = cur * 10 + node.val`。这个状态是单向传递的，不需要 pop。
129 求根到叶数字之和
复制
```python
def sumNumbers(root):
 def dfs(node, cur):
 if node is None:
 return 0

 cur = cur * 10 + node.val
 if node.left is None and node.right is None:
 return cur
 return dfs(node.left, cur) + dfs(node.right, cur)

 return dfs(root, 0)
```


## box
### 199：右视图
右视图“如果你先看右边，那么某个深度第一次出现的节点就是你看到的那一层最右值”。这让 DFS 的顺序和答案结构直接对上。
199 二叉树的右视图
复制
```python
def rightSideView(root):
 ans = []

 def dfs(node, depth):
 if node is None:
 return

 if depth == len(ans):
 ans.append(node.val)
 dfs(node.right, depth + 1)
 dfs(node.left, depth + 1)

 dfs(root, 0)
 return ans
```


## box
### 1457：伪回文路径
这题的状态奇偶位掩码。沿路径走一遍时，把当前值对应的位翻转。到叶子时，只要掩码里最多只有一位是 1，路径就能重排成回文。
1457 伪回文路径
复制
```python
def pseudoPalindromicPaths(root):
 ans = 0

 def dfs(node, mask):
 nonlocal ans
 if node is None:
 return

 mask ^= 1 << node.val
 if node.left is None and node.right is None:
 if mask & (mask - 1) == 0:
 ans += 1
 return
 dfs(node.left, mask)
 dfs(node.right, mask)

 dfs(root, 0)
 return ans
```

# 完整走读：一棵小树怎么把 257、129、1457 连起来 {#walkthrough}
<!-- tag: walkthrough -->

这节把旧页里最有价值的地方再压一遍：看同一条状态链怎么换皮。我们用一棵很小的树来走读，先看 257 的 path，再看 129 的 cur，最后看 1457 的 mask。你会发现，三个题的入口动作几乎一样，变的只是状态载体。

这棵树长成这样：`1` 的左孩子是 `2`，右孩子是 `3`；`2` 的左孩子是 `5`；`3` 的右孩子是 `4`。如果你从 root 一路往下看，最先变化的不是答案，而是你手里的状态。这个顺序感很重要，因为很多树题错的都“看见节点后不知道该先改什么”。

## box
### 257 的完整路径怎么长出来
先到 `1`，path 变成 `[1]`；再到 `2`，path 变成 `[1, 2]`；再到 `5`，path 变成 `[1, 2, 5]`。到这一步才收集答案，因为只有叶子处才知道路径结束了。返回时先弹出 `5`，再弹出 `2`，这样进入 `3` 的时候，path 还是干净的 `[1]`。
这件事看上去像“恢复状态”，其实更像“把分支隔离开”。左分支留下的东西不应该影响右分支。你一旦把这个概念想明白，`path.pop()` 就不再是机械动作，而是一个分支结束时的收尾动作。

## box soft
### 129 和 1457 是怎么继承这个骨架的
129 只是把 path 换成整数 `cur`：`1` 变成 `1`，`2` 变成 `12`，`5` 变成 `125`。由于 `cur` 是参数向下传的局部值，所以返回时不用手工恢复。1457 再进一步，把状态换成位掩码 `mask`。每次走到一个数字，就把对应位翻转一次，到了叶子就检查有没有超过一个奇数位。
这三题放在一起看，会更像“同一条路径上换了三种记账方式”：字符串记账、整数记账、位状态记账。入口动作没怎么变，答案表达变了，恢复动作也变了。你真正要学的，就是把“记账方式”和“遍历动作”拆开。

叶子处收集答案，然后返回时撤销 5 的贡献。

如果没有恢复，右分支看到的就不是纯净的 root 状态。

错误后果最容易从这里看出来
复制

```python
def dfs(node):
 if node is None:
 return
 path.append(node.val)
 if node.left is None and node.right is None:
 ans.append("->".join(map(str, path)))
 else:
 dfs(node.left)
 dfs(node.right)
 path.pop()

# 常见错误 1：少了最后一行
# 右分支会继承左分支留下的节点，最后答案里会出现串台路径。
#
# 常见错误 2：把叶子判断放到递归之后
# 你会把“已经结束”的路径又往下推一层，代码和思路都会变绕。
#
# 常见错误 3：把 cur 当共享状态
# 129 里不该回退 cur，应该把新值直接传给孩子。
```

## box
### 如果这一步少了 pop，会发生什么
最直接的后果，是右分支带着左分支的路径继续走。你在题面上会看到一条看似完整、其实混了两个分支的字符串。更隐蔽的后果是：即使最终答案长度对了，内容也已经不可信了，因为每条路径都失去了自己的边界。
这也是为什么我更愿意把 pop 理解成“分支收尾”，而不是“语法补丁”。只要你把这层含义想清楚，写树题的时候就会主动去找恢复点，而不是等测试样例提醒你。

## box info
### 你现在应该做什么
先合上页面，自己把这棵树手画一遍，再口头说一遍 path、cur、mask 的变化。你不需要写长答案，只要能说出每个节点进来时状态怎么变、离开时为什么能恢复，就已经比“会照着代码抄”稳很多了。
接着再做一件小事：把 1022 的二进制进位式写出来，和 129 对照一下。只要你能自己说出“只是进制不同”，这条迁移链就真正落下来了。

# 实现一致性 {#consistency}
<!-- tag: bridge -->

199 是这页里最适合做“双写法对照”的题。因为它一边可以走 DFS，一边可以走 BFS，而且两种写法最终都在表达“每一层只保留一个代表节点”。你只要把这句话说稳了，后面看别的层序题时，脑子里就会自动出现“层”和“代表”的概念，而不是只剩代码形状。

## box
### DFS 写法在做什么
DFS 的关键“某个深度第一次出现时，先遇到的节点就是那层的代表”。这样一来，`depth == len(ans)` 这句判断就很自然：说明这个深度还没被写入答案，当前节点可以直接进。
199 的 DFS 版怎么想
复制
```python
def rightSideView(root):
 ans = []

 def dfs(node, depth):
 if node is None:
 return
 if depth == len(ans):
 ans.append(node.val)
 dfs(node.right, depth + 1)
 dfs(node.left, depth + 1)

 dfs(root, 0)
 return ans
```


## box soft
### BFS 写法在做什么
层序版本会更直观：每一轮把一层节点取出来，最后一个节点就是这一层的右边代表。这里不需要“第一次出现”的判断，因为队列已经按层排好了顺序。你只需要认出“层末尾”这个位置，就能拿到答案。
199 的 BFS 版怎么想
复制
```python
from collections import deque

def rightSideView(root):
 if root is None:
 return []
 q = deque([root])
 ans = []
 while q:
 size = len(q)
 for i in range(size):
 node = q.popleft()
 if node.left:
 q.append(node.left)
 if node.right:
 q.append(node.right)
 if i == size - 1:
 ans.append(node.val)
 return ans
```


看法
DFS 版
BFS 版
你该怎么记


层的代表
深度第一次出现的节点。
每层最后弹出的节点。
都是一个层代表，只是取位置不同。


顺序控制
先右后左。
队列天然按层排。
顺序不同，但答案口径一致。


代码重心
深度判断。
层末判断。
你要先选“层”这个对象，再选实现。

## box
### 如果把 DFS 顺序写反
你会在同样的 depth 上先看到左边节点，于是把左视图误当成右视图。这个错法非常典型，因为它不是语法错，也顺序理解错。
真正的修法记住“深度第一次出现时，答案拿的是你先见到的那一边”。只要这句想明白，顺序就不容易写反。

## box info
### 你现在应该做什么
拿纸随手画一个三层树，先按 DFS 写出深度第一次出现的顺序，再按 BFS 写出每层最后一个节点。你会发现，两个答案完全能对齐，只是路径不同。这一步很小，但它会把“层”和“顺序”两个词真正拴在一起。

# 从题面到代码 {#bridge}
<!-- tag: operation bridge -->

这一节专门把“读题”到“落代码”的动作拆开。很多树题题意已经理解了，却没法稳定地把它翻译成状态变化。下面这段链路就是为了训练这个翻译动作：先从题面里抽对象，再从对象里抽状态，再把状态放进递归模板。

## box
### 先抽题面对象
257 的对象是“路径”，129 的对象是“数字”，199 的对象是“每层可见节点”，1457 的对象是“路径上的奇偶位状态”。一旦对象抽出来，状态选择就不会乱：路径对象配列表或字符串，数字对象配累计值，可见节点对象配深度或队列，奇偶对象配位掩码。
这一步看似简单，但它其实决定了你后面会不会把题写成别的题。比如 129 如果误抽成“路径字符串”，你会不必要地维护列表；1457 如果误抽成“数字求和”，你会把位掩码的意义彻底写丢。

## box soft
### 再放进递归模板
把对象抽出来以后，模板就只剩三件事：当前节点入状态、必要时收集、返回前恢复。你不用每道题都从“应该怎么递归”开始想，只要把“题面对象”翻译成“状态载体”，模板自然就落下来了。
这也是为什么树题常被说成“看懂了和会写之间隔了一层”。那层不是难度，而是翻译：你要把题目语言转成递归语言。

读题到实现的最小翻译链
复制

```python
# 题面对象 -> 状态载体 -> 递归动作
#
# 路径类题目:
# path / 字符串 / 列表
# 进入节点: append
# 收集时机: 叶子
# 返回前: pop
#
# 数值类题目:
# cur / 累计值
# 进入节点: cur = cur * base + node.val
# 收集时机: 叶子
# 返回前: 不需要 pop
#
# 可见层类题目:
# depth / queue
# 进入节点: 记录当前深度或当前层
# 收集时机: 第一次出现、最后一个节点
# 返回前: 继续左右或下一层
```

题面里看到什么
你先抽什么对象
最合适的状态载体
模板里的动作


“从根到叶的所有路径”
路径
列表或字符串
append -> 收集 -> pop


“根到叶组成的数字之和”
数字
累计值 cur
cur 进位后向下传


“每层右边看到的节点”
层代表
depth 或 queue
先右后左，或层末收集


“路径是否能重排成回文”
奇偶状态
bitmask
翻转位，叶子检查

## box
### 错在对象抽错了会怎样
一旦对象抽错，后面的动作就会跟着全错。把路径题当数值题，会漏掉恢复动作；把数值题当路径题，会多出一堆不必要的状态；把层题当路径题，结果往往是“代码能跑，答案不对”。
这种错最麻烦的地方在于，它不会立刻报错，而是只在复杂样例上悄悄歪掉。你越是觉得“差不多”，它越会在边界上反咬你。

## box info
### 下一步怎么练
以后你拿到任何一题，都先写一句“对象是什么、状态是什么、收集在什么时候”。如果这三句写不出来，就先别急着敲代码。你现在应该能做的，是把题目先翻译成状态语言，再开始写递归。

# 边界与复杂度 {#boundary}
<!-- tag: limits -->

树题里最容易被轻描淡写的就是复杂度。很多人只写“访问每个节点一次”，这其实只说了一半。只要答案需要把路径文字、数字列表或叶子结果真的构造出来，输出本身就会再占一段成本。面试时把这个说清楚，回答会立刻稳很多。

空树
空树通常返回空列表、0 或空值。它只表示“没有节点”，不是“走到了叶子”。


单节点
单节点既是根也是叶。路径题应该保留这一条单节点路径，累计值题应该直接返回这个值。


退化链
最坏情况下树会退化成链，递归深度接近节点数。Python 里这时候要注意栈深限制。

$$
 T(n) = O(n) + O\!\left(\sum_i |path_i|\right), \quad S(n) = O(h)
 $$
 前半句是遍历节点的基础成本，后半句是你真的构造了多少输出。`h` 表示树高；如果树退化成链，递归栈就会跟着变深。

场景
你要怎么判
答案通常是什么
最常见的错法


空树
先命中 `node is None`。
空列表、0、空字符串。
把 None 当叶子继续往下走。

左右孩子同时为空。
这一条路径或这个数字本身。
忘了在叶子处收集。

时间要加输出长度。

递归深度是否接近 n。

默认 Python 一定扛得住。

# 常见误区 {#mistakes}
<!-- tag: debug -->

树题的错法非常集中：边界判错、恢复动作漏掉、遍历顺序写反、复杂度口径说轻。下面这几个错法，基本就是旧页里那几类题最常见的翻车点。

把空节点当叶子
错因：终止条件和收集条件混在一起。
后果：路径里会冒出 None，或者空树被误写成一条路径。
修法：先判断 `node is None`，再判断是不是叶子。


共享 path 没有 pop
错因：把 path 当成普通局部变量，却忘了它是共享对象。
后果：左分支留下的节点会污染右分支。
修法：进入节点 append，返回前 pop，顺序不要倒。


右视图顺序反了
错因：用了“深度第一次出现”的逻辑，却先递归左子树。
后果：你得到的左视图。
修法：DFS 先右后左，或者改成 BFS 每层取最后一个。


复杂度只写访问节点
错因：只数了遍历次数，没有数输出构造成本。
后果：路径很多的时候，答案被说得过于乐观。
修法：把“遍历成本”和“输出成本”分开写。


129 里误用 pop 思维
错因：把累计值写成共享状态，和 path 一起回退。
后果：父节点和兄弟分支之间互相串值。
修法：让 `cur` 作为参数向下传，不共享就不用恢复。


叶子判断太晚
错因：先递归孩子，再判断自己是不是叶子。
后果：到头了还往下递归，代码会变得又长又绕。
修法：在当前节点处把叶子判断做完，别把它拖到后面。

# 题目迁移 {#transfer}
<!-- tag: transfer -->

你不需要把每道题都从头学一遍。更稳的方式是把它们拆成三层：状态是什么、收集在什么时候、顺序为什么要这样排。只要这三层没变，题目换壳也不会变得陌生。

129 和 1022 的共同点
它们都在沿路径做进位：一个按十进制，一个按二进制。状态更新式变了，入口职责没有变。


199 和层序题的共同点
右视图可以用 DFS，也可以用 BFS。关键你能不能把“每层第一次/最后一次”的判断落到稳定位置。


1457 的共同骨架
它仍然是根到叶路径，只是路径信息不再是字符串，而是每个数字出现奇偶性的压缩表示。

**Q:** Q1 写根到叶路径题时，`dfs(node, path)` 进入当前节点前，path 最准确表示什么？

- [ ] . 从根到父节点的路径。
- [ ] . 整棵树里所有路径的集合。
- [ ] . 当前层所有节点的名字。

Answer: . 从根到父节点的路径。

### 2 判断

**Q:** Q2 `node is None` 和“叶子节点”是一回事。

- [ ] 对
- [ ] 错

Answer: false

### 3 选择

**Q:** Q3 257 为什么要把答案放在叶子处收集？

- [ ] . 因为叶子处最适合写中序遍历。
- [ ] . 因为只有到叶子时，路径才完整。
- [ ] . 因为叶子一定是右边节点。

Answer: . 因为只有到叶子时，路径才完整。

### 4 选择

**Q:** Q4 199 的 DFS 版本通常先走右子树，原因是什么？

- [ ] . 深度第一次出现时，右边先到的节点就是右视图候选。
- [ ] . 因为右子树一定更短。
- [ ] . 因为层序遍历不能做右视图。

Answer: . 深度第一次出现时，右边先到的节点就是右视图候选。

<!-- end-practice-group -->

<!-- practice-group: 二、路径恢复 -->
<!-- quick-review: 复习一下 共享 path 要 pop，参数 cur 通常不用 pop。 -->

### 5 诊断

**Q:** Q5 下面哪些动作是在递归返回前必须完成的？

- [ ] path.pop()
- [ ] 恢复共享状态
- [ ] 重新初始化 ans
- [ ] 把 node 再 append 一次

Explain: 应该选 path.pop() 和“恢复共享状态”。 ans 是结果容器，不需要在每次递归返回时重建；当前节点也不该再追加一次。

### 6 诊断

**Q:** Q6 下面哪些说法是对 129 的？

- [ ] cur = cur * 10 + node.val
- [ ] 叶子处收集答案
- [ ] 回退时需要 cur // 10
- [ ] cur 是共享列表

Explain: 应该选前两项。 cur 是向下传的局部数值，不是共享列表，所以不需要像 path 那样 pop 或回退成父节点状态。

### 7 走读

**Q:** Q7 用 `1->2->5` 和 `1->3` 说明 path 的 push、收集、pop 顺序。

Explain: 进入 1 时 path = [1]；进入 2 时 path = [1, 2]；进入 5 时 path = [1, 2, 5]，此时收集 1->2->5 ；返回 2 前 pop 掉 5，path 回到 [1, 2]；返回 1 前再 pop 掉 2，path 回到 [1]；进入 3 时 path = [1, 3]。关键你能否把恢复动作写在正确位置。

### 8 走读

**Q:** Q8 129 中从 `1` 到 `2` 到 `9` 时，cur 如何变化？为什么不需要 pop？

Explain: cur 依次变成 1、12、129。它是参数向下传的局部值，每个递归分支都有自己的 cur 副本，所以返回时不需要手动 pop。和 path 不同，cur 不共享；你只需要在当前层算出新值，交给孩子即可。

### 9 分析

**Q:** Q9 右视图为什么既可以写成右优先 DFS，也可以写成层序每层最后一个？

Explain: 两种写法都在表达同一件事：每一层只保留一个代表节点。DFS 用“深度第一次出现”把右边的节点先送到答案里；BFS 用“每层最后一个”直接从队列顺序里拿结果。一个是顺序先行，一个是层级先行，答案口径一样。

### 10 分析

**Q:** Q10 为什么“空节点”不是“叶子节点”？如果把它们混在一起，会出什么错？

Explain: 空节点表示没有节点，叶子节点表示有节点但左右孩子都为空。把它们混在一起，会让你把 None 也当成路径的一部分，或者把答案收集位置放错。修法很直接：先排除空节点，再判断是不是叶子。

### 11 表达

**Q:** Q11 用 2-3 句话解释树遍历题的时间、递归栈空间和输出空间。

Explain: 基础遍历通常是 O(n)，因为每个节点都会被访问一次。递归栈空间是 O(h)，h 是树高，最坏会接近 O(n)。如果题目需要真的拼出很多路径字符串或输出很多叶子结果，还要再把输出长度单独算进去。

### 12 总结

**Q:** Q12 用 6-9 句话讲清二叉树遍历题的入口、不变量、边界、复杂度和迁移。

Explain: 回答里至少要说清楚五件事：当前节点处理什么、path 或 cur 的不变量是什么、空节点和叶子节点怎么区分、复杂度为什么不只是 O(n)、以及 257 / 129 / 199 这类题分别属于哪一类状态。讲得清楚的人，基本已经不会把树题写乱了。
