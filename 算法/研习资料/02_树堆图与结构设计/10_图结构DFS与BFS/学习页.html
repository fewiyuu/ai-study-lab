先建图，再决定怎么走

图结构表达的是“对象之间的连接关系”。数组题常沿下标移动，树题从父到子扩展，图题则要先知道每个点有哪些邻居。邻接表就是这件事的常见表示。

DFS 和 BFS 都是在穷举可达状态。DFS 一条路走到底，适合连通性、路径存在、回溯式搜索；BFS 一层一层扩散，适合最短步数和层级距离。你要先把图建对，再选遍历顺序。

学完应该能做什么

  1. 能用邻接表表示有向图和无向图。
  2. 能解释 `visited` 在图遍历中的必要性。
  3. 能写出图 DFS 和 BFS 的基础模板。
  4. 能区分连通性问题和最短步数问题。
  5. 能把 841、1306、417、399 等题映射到图遍历。

主线地图

mental model

图遍历的主线是“状态 + 邻居 + 去重”。状态可以是节点编号，也可以是网格坐标、字符串、账户名。邻居由题目规则给出。去重保证同一状态不会被无限重复处理。

先把关系变成图

如果题目给的是课程依赖、房间钥匙、账户邮箱、除法关系，你要先把“谁连接谁”写成邻接表。

判断句：输入里出现关系对、可达、连通、依赖，先想建图。

DFS 适合顺着一条路深挖

DFS 常用于判断能不能到达、找连通分量、岛屿面积、路径枚举。递归写法短，但要小心访问标记的位置。

如果图可能有环，没有 visited 就可能无限递归。

BFS 适合按距离扩散

BFS 的队列天然按步数分层。第一次到达某个状态时，通常就是最短步数。

前提是每条边的代价相同。如果边权不同，就要转向 Dijkstra 等算法。

术语表

terms

先把词说清楚，后面的代码才不会变成背模板。

术语直觉准确说法常见误解
节点图里的一个状态。图的基本元素，可以表示城市、课程、房间、坐标或字符串。以为节点一定是数字编号。
边从一个状态能走到另一个状态。连接两个节点的关系，可有向、无向、带权或不带权。无向边只加一次，导致反方向走不了。
邻接表每个节点对应一个邻居列表。用数组或字典保存节点到其相邻节点集合的映射。把邻接矩阵和邻接表复杂度混在一起。
visited标记已经看过的状态。集合、数组或矩阵，用来避免重复遍历和环造成的死循环。等出队后才标记，导致同一节点多次入队。
连通分量互相能到的一组节点。在无向图中，任意两个节点之间存在路径的最大节点集合。把有向图里的可达性直接当无向连通。

知识链路

chain

图题的第一步通常不是 DFS 或 BFS，而是确认节点和边。力扣 841「钥匙和房间」里，房间是节点，钥匙指向能进入的房间；1306「跳跃游戏 III」里，下标是节点，`i ± arr[i]` 是边；399「除法求值」里变量是节点，除法关系是带权边。

DFS 模板读起来像“进入节点，标记访问，然后递归处理所有未访问邻居”。BFS 模板读起来像“把起点放进队列，循环弹出当前节点，把未访问邻居入队”。两者都依赖邻居生成函数。

选择 DFS 还是 BFS，要看题目要什么。只问能不能到、整块有多大，DFS 简洁；问最少几步、最近出口、扩散时间，BFS 更自然。

本页的判断句

  • 图可能有环，遍历前先准备 visited。
  • 无向边通常要双向加入邻接表。
  • BFS 第一次到达最短，只适用于等权边。
  • 网格题也是图，坐标的上下左右就是邻居。
图遍历复杂度：O(V + E)，V 是节点数，E 是边数；网格图常写成 O(mn)

核心模板

template

邻接表可以用 `defaultdict(list)` 或列表数组。节点是连续整数时用数组更直接；节点是字符串或对象时，用字典更方便。

访问标记的位置很重要。BFS 通常在入队时标记，这样同一节点不会被多个父节点重复加入队列。DFS 通常在进入节点时标记。

正确示例：图 DFSPython
def dfs_graph(graph, start):
    visited = set()

    def dfs(x):
        if x in visited:
            return
        visited.add(x)
        for nei in graph[x]:
            dfs(nei)

    dfs(start)
    return visited

`visited.add(x)` 写在递归邻居之前。这样遇到环时，下一次回到 x 会直接返回。

正确示例：图 BFSPython
from collections import deque

def bfs_graph(graph, start):
    q = deque([start])
    visited = {start}
    while q:
        x = q.popleft()
        for nei in graph[x]:
            if nei not in visited:
                visited.add(nei)
                q.append(nei)
    return visited

入队时就标记 visited，可以避免同一邻居被多个节点重复加入队列。

渐进例子

状态演示

切换场景，观察关键状态怎么变。看懂状态，比背下一行代码更重要。

边有方向

A 能到 B，不代表 B 能回 A。课程依赖、钥匙开房间常是有向关系。

你要能把题目里的对象换成节点，把允许的动作换成边。

例题：钥匙和房间Python
def canVisitAllRooms(rooms):
    visited = set()

    def dfs(room):
        if room in visited:
            return
        visited.add(room)
        for key in rooms[room]:
            dfs(key)

    dfs(0)
    return len(visited) == len(rooms)

房间编号是节点，钥匙列表就是邻接表。题目只问能不能访问全部，DFS 足够。

例题：跳跃游戏 IIIPython
from collections import deque

def canReach(arr, start):
    q = deque([start])
    visited = {start}
    while q:
        i = q.popleft()
        if arr[i] == 0:
            return True
        for j in (i + arr[i], i - arr[i]):
            if 0 <= j < len(arr) and j not in visited:
                visited.add(j)
                q.append(j)
    return False

下标是节点，跳跃规则产生邻居。这里用 BFS 或 DFS 都可以，因为只问是否可达。

边界与复杂度

limits

有向和无向

建图时先确认边是否有方向。无向图漏加反向边，会让连通性判断变错。

访问标记时机

BFS 如果出队时才标记，同一节点可能被重复入队。结果可能还对，但性能和队列规模会变坏。

边权问题

普通 BFS 只处理每条边代价相同的最短步数。边权不同，要用 Dijkstra 或 0-1 BFS。

做法时间空间什么时候用
邻接表 DFS/BFSO(V+E)O(V)稀疏图、关系对输入
邻接矩阵遍历O(V²)O(V²)节点少、查询任意边是否存在多
网格遍历O(mn)O(mn)岛屿、迷宫、水流

常见错误：错因、后果和修法

debug

没建清楚节点

错因：直接套模板，不知道状态是什么。

后果：visited 存错对象，重复和边界都乱。

修法：先写一句“节点是 X，边是 Y”。

无向边只加单向

错因：把关系对默认当有向。

后果：反向可达性丢失，连通分量变小。

修法：无向图建表时两边都 append。

忘记 visited

错因：把图当树写。

后果：有环时无限递归或重复搜索。

修法：DFS 进入时标记，BFS 入队时标记。

BFS 用在带权最短路

错因：只看到“最短”就套 BFS。

后果：边权不同会得到错误路径。

修法：等权用 BFS，非负权用 Dijkstra。

邻居越界没检查

错因：网格题先访问再判断边界。

后果：数组越界或把非法坐标入队。

修法：生成邻居后先检查范围和访问状态。

错误示例：图有环但没有 visitedPython
def dfs(x):
    for nei in graph[x]:
        dfs(nei)  # A -> B -> A 会无限递归

图不是树，邻居可能指回祖先。进入节点时标记 visited 是基本防线。

易混概念

compare

树遍历 vs 图遍历

树遍历：天然从父到子，通常没有环。

图遍历：邻居可能指回已访问节点。

判断：只要状态可能重复，就要 visited。

DFS vs BFS

DFS：一条路走到底，适合连通性和路径存在。

BFS：一层层扩散，适合等权最短步数。

判断：问“能不能到”可 DFS；问“最少几步”优先 BFS。

邻接表 vs 邻接矩阵

邻接表：只保存真实存在的边。

邻接矩阵：用 V×V 表示任意两点是否相连。

判断：边少用邻接表，频繁查任意边可考虑矩阵。
