学习目标

起步

这一页的目标很具体：你要能把题面翻译成容器状态，而不是一上来就写模板。栈题看最近未匹配，队列题看先到先走，单调结构看未来还可能有用的候选。

如果题目里出现“回退”“撤销”“最近一个”“第一个更大”“窗口最大值”，通常就有对应的容器影子。看见这些词时，先问自己：容器里应该存值、存下标，还是存一段状态。

先盯住这三步

  1. 把输入里的对象改写成节点、元素或下标。
  2. 给栈顶、队头、窗口边界选对更新时机。
  3. 把“保留什么、删除什么、为什么删除”说清楚。
你要会的动作 具体表现 错了会怎样
识别容器 看到括号、路径、窗口、相邻更大/更小，能先判断是栈、队列还是单调结构。 容器选错，后面每一步都会偏。
确认状态 知道容器里存的是值、下标、还是同时携带距离或前缀信息。 窗口过期、操作数顺序、边界判断容易出错。
解释不变量 能说出栈顶、队头、单调方向为什么必须保持。 代码能跑一两个样例，但一遇到特殊输入就坏。

先抓主线

主线图

这一页最重要的主线是：容器不是为了“装更多东西”，而是为了“把无用状态尽早删掉”。栈删的是不该继续留在顶上的东西，队列删的是已经过期的头部，单调栈和单调队列删的是不会再成为答案的候选。

栈的读法 最新进入的状态先被检查

右括号要找最近左括号，`..` 要回到最近目录。

表达式的读法 运算符拿最近两个数

先弹出的是右操作数，再弹出的是左操作数，顺序不能反。

单调栈的读法 比当前元素差的候选被淘汰

它不是排序，而是把“以后不可能成为答案”的元素踢出去。

单调队列的读法 窗口边界和大小关系一起管

队列里最好存下标，这样才能知道谁已经离开窗口。

边界提示 本页先不追 0-1 BFS、Dijkstra、双端队列最短路、可删除堆这些内容。它们会在后面的单元继续，但今天要先把“容器里该留什么状态”看稳。
对象 读什么 写什么 错了的症状
栈 最近未匹配状态 入栈、出栈 空栈没判、配对顺序错
队列 当前最早待处理状态 入队、出队 重复入队、头部过期没删
单调栈 比当前元素差的候选 弹出并写答案 比较方向反了，答案边界错
单调队列 窗口内仍有机会成为最值的候选 保持单调和过期删除 队头可能已经不在窗口里

核心机制

不变量

栈：只留最近还没解决的

栈的价值来自“最近”。括号匹配时，右括号只能找最近的左括号；路径简化时，`..` 只能回退到最近一级目录。它们都不是在找“某个对的元素”，而是在找“最晚还没被解决的那个元素”。

因此，栈题最先要检查的是：遇到关闭状态时，栈顶是不是对应的开放状态；如果不是，说明嵌套顺序已经坏了。

队列：只留还没过期的先到状态

队列的价值来自“先后”。BFS 里，先进入队列的节点先被扩散；循环队列里，队头始终代表最早还没处理完的状态。队列题常见错误不是不会入队，而是忘了头部已经过期。

只要窗口会动，或者任务会不断到达和离开，你就得问：队头还是当前窗口的一部分吗。

$$ \text{每个元素最多入一次、出一次，所以总时间是 } O(n) $$

这句话解释了为什么单调栈和单调队列里虽然有 `while`，但通常不是平方复杂度。`while` 只是把已经没有价值的候选一次性清掉。

核心对象 前置状态 后置状态 读取 写入 / 删除 出错症状
栈顶 最近未匹配元素在顶上 弹出后露出下一个候选 当前字符或当前目录名 push / pop 配对错位或路径回退错
单调栈 栈内按设定方向单调 弹掉不可能成为答案的元素 当前元素和栈顶比较 写下一个更大 / 更小答案 边界方向反了
单调队列 队列里保留窗口候选 头部可能被新元素或窗口边界淘汰 窗口右端新元素、左端过期位置 pop / popleft / append 队头可能已经不在窗口里

逆波兰表达式：`2 1 + 3 *`
def eval_rpn(tokens):
    stack = []
    for tok in tokens:
        if tok in {"+", "-", "*", "/"}:
            right = stack.pop()
            left = stack.pop()
            stack.append(apply(left, right, tok))
        else:
            stack.append(int(tok))
    return stack[-1]

这里最关键的是弹出顺序。先弹的是右操作数，后弹的是左操作数。`a-b` 和 `a/b` 一旦顺序反了，结果就会直接错，但代码看起来还像样。

单调栈：下一个更大元素
def next_greater(nums):
    stack = []   # 存下标，栈内对应值递减
    ans = [-1] * len(nums)
    for i, x in enumerate(nums):
        while stack and nums[stack[-1]] < x:
            ans[stack.pop()] = x
        stack.append(i)
    return ans
步骤 当前元素 栈内保存什么 被弹出的元素 写入的答案
1 2 [0] 无 无
2 1 [0, 1] 无 无
3 3 [2] 1 和 2 对应的下标 它们的下一个更大都是 3
这一步的真正意义 被弹出的元素不再保留，是因为当前元素已经替它们回答了“下一个更大是谁”。如果你不先弹出，它们就会继续挡住后面的答案。

窗口走读

trace

前面已经把状态账本立住了，这一段再把它放回单调队列的窗口里看一遍。你会发现，窗口题真正难的地方不是维护单调性，而是同时照看两个边界：右边界在扩张，左边界在离开，容器里的候选必须跟着两边一起变。

拿 `nums = [1, 3, -1, -3, 5, 3, 6, 7]`、`k = 3` 来看最直观。第一个窗口看 `[1, 3, -1]`，最大值是 `3`；右边继续滑动时，旧下标会过期，新下标会进来，队列里保留下来的永远是“当前窗口里还有资格竞争最大值”的那部分。

时刻 新元素 队尾动作 队头动作 窗口最大值
i=0 1 直接入队 无 暂不输出
i=1 3 把 1 删掉，再放入 3 无 暂不输出
i=2 -1 保持队列单调 无 3
i=3 -3 继续压住更差候选 检查 1 是否过期 3
i=4 5 先把尾部更小的都弹掉 2 号位置已离开窗口 5
单调队列走读：`[1, 3, -1, -3, 5]`
i=0  q=[0]         max=-
i=1  q=[1]         1 被 3 压掉
i=2  q=[1, 2]      max=3
i=3  q=[1, 2, 3]   继续保留 3
i=4  q=[4]         1、2、3 都不如 5

这里最值得记的不是某个具体结果，而是顺序：先清尾，保证队列里只留下更有资格的候选；再看队头，保证读到的一定是窗口内的答案。两步顺序一旦反了，就会出现“队列还单调，但答案回到了旧值”的假象。

这也是为什么单调队列通常存下标，不直接存值。值只能告诉你谁大谁小，下标才能告诉你谁已经过期。窗口题一旦开始移动，位置比数值更重要。

另一个窗口案例：最大值回落

把数组 `nums = [9, 8, 7, 6, 10, 5]` 和窗口 `k = 3` 放进来，最容易看出队头为什么要先处理。前两个窗口的最大值分别是 9 和 8，第三个窗口里 10 进入后，旧的 9 和 8 都不该再继续占着队头。

如果只看单调性，你会误以为队列一直都很好；可只要左边界一动，已经滑出去的下标就会把答案拖回旧位置。窗口题里最稳的判断方法是看下标，而不是看值。

窗口 当前区间 队列应保留 当前答案
1 [9, 8, 7] [0, 1, 2] 9
2 [8, 7, 6] [1, 2, 3] 8
3 [7, 6, 10] [4] 10

失败后果与修法

错法最常见的版本，是把“先清尾”放在“先删过期”前面，却没有补上队头检查。这样写出来的队列看起来还是单调的，实际读到的最大值却可能已经离开窗口。结果就是样例能过一半，边界样例直接失真。

更稳的顺序
from collections import deque

def push_window(q, nums, i, k):
    while q and nums[q[-1]] <= nums[i]:
        q.pop()
    q.append(i)
    if q[0] <= i - k:
        q.popleft()

这几行里最重要的不是语法，而是动作次序：先让尾巴让位，再让头部退场。只有这样，队列里留下的才真的是当前窗口可用的候选。

练习时可以先手画三个窗口，再用下标给队列标号。只要你能说出“这个下标为什么还在、那个下标为什么该走”，滑动窗口的边界就不会再靠猜。

实践钩子 下一次遇到窗口类题，先用纸写出队列中的下标变化，再敲代码。你会更容易发现自己到底是在维护候选，还是在维护一串看起来整齐的数字。
窗口题的检查点 先问队尾该不该弹，再问队头该不该删，最后再读当前答案。你把这三问固定住，滑动窗口类题目的调试成本会低很多。

状态账本与故障走读

diagnosis

做栈队列题时，真正要写清的是状态账本：容器里存什么、什么时候写、什么时候删、删晚了会出现什么症状。这个账本一旦立住，很多模板其实只是在重复同一条规则。

下面把四类常见题摆在一张表里看。你可以把它当成做题前的检查单：先确认对象，再确认时机，最后确认症状。这样比只背一段代码稳得多。

场景 容器里存什么 什么时候写 什么时候删 常见症状
括号匹配 未匹配的左括号 看到左括号就入栈 看到匹配的右括号就弹出 空栈没判、配对顺序错
逆波兰表达式 最近算完的数值 数字直接入栈 遇到运算符弹出两个数 减法和除法顺序反了
下一个更大元素 还没等到答案的下标 当前元素不能解决栈顶时就入栈 当前元素比栈顶大时持续弹出 比较方向写反，答案边界错
滑动窗口最大值 窗口内仍有机会成为最大值的下标 新下标到来时先清尾，再入队 队头过期时立刻删掉 队头可能已经不在窗口里

空栈问题

错因通常不是不会用栈，而是默认“反正前面一定有配对字符”。

症状是访问栈顶时直接报错，或者把非法输入误判成合法。

修法只有一个：在访问栈顶前先判空。

顺序问题

逆波兰表达式里的减法和除法最容易出错，因为最近弹出的那个是右操作数，不是左操作数。

你只要把“先弹谁、后弹谁、最后谁在前”写成固定顺序，错误会少很多。

过期问题

窗口题最常见的假象是“我已经维护单调性了，应该就对了”。其实不够，队头还可能已经离开窗口。

所以窗口题的标准动作总是两步：先看队头是否过期，再决定是否把新候选入队。

故障对照：窗口最大值的正确顺序
from collections import deque

def max_sliding_window(nums, k):
    q = deque()
    ans = []
    for i, x in enumerate(nums):
        while q and nums[q[-1]] <= x:
            q.pop()
        q.append(i)
        if q[0] <= i - k:
            q.popleft()
        if i >= k - 1:
            ans.append(nums[q[0]])
    return ans

这段代码里，q 的两端各管一件事：尾巴负责把更差的候选删掉，头部负责把过期下标删掉。两件事缺一件，答案就会偏。

如果你在练习时遇到“代码能跑，但偶尔错一个样例”，先不要怀疑语言库，先检查这两个位置：队头是不是过期了，队尾是不是该删没删。大部分窗口题的 bug 都逃不出这两个位置。

症状 最可能的原因 下一步检查
非法括号输入报错 空栈访问 检查右括号分支里有没有先判空
RPN 结果对一半错一半 操作数顺序反了 确认先弹的是右操作数
下一个更大元素少写答案 while 条件方向反了 先写清楚栈内是递增还是递减
窗口最大值偶尔回到旧值 过期下标没删 检查队头是否越过左边界
做题顺序建议 先写对象，再写时机，再写症状。只要这三句话能脱口而出，后面的代码就不会靠运气。

模板

代码

模板不是先记代码，而是先记三句：什么时候入、什么时候弹、弹完写什么。只要这三句稳了，代码只是语法包装。

括号匹配模板
def is_valid(s):
    pairs = {")": "(",
             "]": "[",
             "}": "{"}
    stack = []
    for ch in s:
        if ch in pairs:
            if not stack or stack[-1] != pairs[ch]:
                return False
            stack.pop()
        else:
            stack.append(ch)
    return not stack

这里最值得记的一句是：右括号到来时，先看栈顶是不是它的对应左括号，再决定要不要弹。空栈和配对错误都不能跳过。

简化路径模板
def simplify_path(path):
    stack = []
    for part in path.split("/"):
        if part in {"", "."}:
            continue
        if part == "..":
            if stack:
                stack.pop()
        else:
            stack.append(part)
    return "/" + "/".join(stack)
单调队列模板
from collections import deque

例子

渐进案例

单调栈：`[2, 1, 3]`

2 先入栈，1 也入栈；当 3 到来时，1 和 2 都会被弹出并各自写下答案 3。这里没有排序动作，只有“谁的答案被当前元素解决了”。

当前值 栈内下标 动作 结果
2 [0] 入栈 等待后面的更大值
1 [0, 1] 继续入栈 仍在等待
3 [2] 持续弹栈 1 和 2 的答案都成了 3

单调队列：`[1, 3, 5, 2]`，窗口 `k=3`

当 5 到来时，3 和 1 都要从队尾退出，因为它们已经不可能成为当前窗口和后续窗口的最大值。接着，窗口左边界移动时，队头如果过期也要删掉。

单调队列最常见的错误不是不会维护单调性，而是只顾尾巴不顾头。队头是答案，队尾是候选，两个都得管。

当前值 队列保存 动作 为什么
1 [0] 入队 窗口里还没更大的候选
3 [1] 弹掉 1，再入队 3 3 比 1 更有资格做最大值
5 [2] 继续弹尾再入队 5 会解决前面的更差候选

先别追

边界

现在先放下的概念

先别把注意力放到 0-1 BFS、双向 BFS、Dijkstra、可删除堆、斐波那契堆上。它们都和容器有关，但今天的重点不是“更强的结构”，而是“容器里留谁、删谁、什么时候删”。

遇到这些词先记住一句话

  • 带权最短路：层数不再等于答案。
  • 循环队列：头尾边界要单独处理。
  • 柱状图：左右第一个更小边界决定宽度。
  • 股票价格跨度：单调栈里常会合并连续天数。

循环队列：`622` 的边界读法

循环队列和普通队列最像的地方，是都要按先来先服务的顺序处理元素；最不一样的地方，是数组尾巴绕回头部以后，`head`、`tail`、`count` 三个量必须一起看。只记两个指针时，空和满很容易长得一模一样。

拿容量为 3 的队列举例：先放入 10、20、30，再弹出 10，最后放入 40。这个过程里，`tail` 会绕一圈回到起点，`head` 也会跟着向前移动。真正可靠的写法，是把位置和元素个数一起维护，这样你才能判断“还能不能放”和“现在是不是空”。

动作 head tail count 状态说明
初始 0 0 0 队列为空
入队 10、20、30 0 0 3 容量已满，尾指针绕回起点
出队一次 1 0 2 队头腾出一个位置
再入队 40 1 1 3 队列重新满了
循环队列骨架
class MyCircularQueue:
    def __init__(self, k):
        self.data = [0] * k
        self.head = 0
        self.tail = 0
        self.count = 0

    def enQueue(self, value):
        if self.count == len(self.data):
            return False
        self.data[self.tail] = value
        self.tail = (self.tail + 1) % len(self.data)
        self.count += 1
        return True

这段骨架里，`head` 决定你从哪里读，`tail` 决定你往哪里写，`count` 决定你是否还能继续写。三个量少一个，程序就会把满队列当空队列，或者把空队列当成还能出队的状态。

失败后果通常很直接：要么新元素覆盖了旧元素，要么合法入队被错误拒绝。练习 622 时，先在纸上把三个变量的变化写一遍，再去写接口，定位会快很多。

实践钩子 下次看到“设计循环队列”时，先问自己三个问题：当前队首在哪、队尾在哪、还剩多少空位。只要这三问答顺，边界判断就会稳定下来。
先别追的概念 为什么先放下 后面去哪儿看
0-1 BFS 边权不同，队列策略会变。 后面的最短路专题
双向 BFS 它是在扩散方向上做优化，不是今天的主线。 更后面的图搜索优化部分

常见误区

调试

空栈不判

错因：默认关闭符号前一定有对应的打开符号。

后果：访问栈顶时报错或误判。

修法：先判空，再比栈顶。

操作数顺序反了

错因：把逆波兰里的两个最近数字随便交换。

后果：减法和除法会直接错。

修法：先弹右操作数，再弹左操作数。

比较方向写反

错因：找更大元素时却在维护递增栈，或反过来。

后果：弹掉了该留的候选，答案边界全乱。

修法：先写清楚栈内是递增还是递减。

窗口过期不删

错因：只看队尾，不看队头。

后果：输出的最大值可能已经离开窗口。

修法：队列存下标，先检查 q[0] <= i-k。

把单调栈当排序

错因：以为栈里必须保存全部元素。

后果：该弹的不弹，效率和答案都会坏。

修法：理解被弹出的元素已经失去未来价值。

错误示例：窗口最大值只存值不存下标
from collections import deque

q = deque()
for x in nums:
    while q and q[-1] <= x:
        q.pop()
    q.append(x)
    # 这里已经丢了位置，后面无法判断队头是否过期
错因 / 后果 / 修法 诊断题最好按这三个词答。只写“错了”“改掉了”不够，必须把为什么错、错到哪里、怎么修说出来。

code reading

6. 为什么遇到减法或除法时，先弹出的是右操作数？

参考答案 / 评分要点

因为表达式里前一个数是左操作数，后一个数是右操作数，而栈是后进先出。先弹出的那个最近入栈，所以它对应右操作数；顺序一旦反了，`a-b` 和 `a/b` 的结果就会错。

scenario transfer

7. 滑动窗口最大值中，队列为什么既要维护单调性，也要维护窗口边界？

参考答案 / 评分要点

单调性负责把更差的候选从队尾清掉，窗口边界负责把已经离开窗口的旧下标从队头清掉。只维护单调性不够，因为队头可能已经过期；只维护边界也不够，因为队列会保留一堆明显更差的候选。

8. 找下一个更大元素时，单调栈方向写反会怎样？怎么修？

参考答案 / 评分要点

错因是把应该维护递减栈的题写成了递增栈；后果是该弹的候选没弹掉或弹错，答案边界全乱；修法是先写清楚“栈内对应值应该递增还是递减”，再写 while 条件。

error diagnosis

9. 窗口最大值不删除过期下标，会出现什么错误？

参考答案 / 评分要点

队头可能已经离开窗口，但还被当成最大值返回，结果会把旧元素算进当前窗口。修法是队列存下标，并在每轮先检查队头是否越过左边界。

complexity

10. 为什么单调栈里有 while，整体仍是 O(n)？

参考答案 / 评分要点

因为每个元素最多被入栈一次、出栈一次。`while` 看起来会反复执行，但它每次弹掉的都是已经没有未来价值的元素，所有弹出加起来不会超过 n 次。

11. 柱状图最大矩形为什么要找左右第一个更小边界？

参考答案 / 评分要点

因为当前柱子的高度决定了矩形高度，而左右第一个更小边界决定了它能横向延伸多宽。只有边界找对，宽度才对，面积才对。

12. 股票价格跨度中，栈里可以合并保存什么信息？

参考答案 / 评分要点

通常会把价格和到目前为止合并出来的跨度一起保存。这样当更高价格出现时，可以把前面连续被压住的天数合并掉，不必一天天回头数。

interview

13. 用 2-3 句话说明栈、队列、单调栈、单调队列的区别。

参考答案 / 评分要点

栈保留最近未匹配状态，队列保留最早待处理状态。单调栈和单调队列是在这个基础上再加“大小关系”的约束，用来删掉未来不可能成为答案的候选；前者常找左右第一个更大/更小，后者常维护窗口最值。
