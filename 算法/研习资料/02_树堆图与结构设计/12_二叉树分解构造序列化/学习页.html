二叉树分解构造序列化

学习目标

起步

这一页要解决的“我能不能把树题改写成明确的返回值问题”。构造题、序列化题和删点成林题，都离不开这一步。

你会看到同一棵树在三种任务里分别扮演什么角色：构造时它是结果，序列化时它是协议载体，删点时它是会被拆开的结构。先把返回值想清楚，递归才不会飘。

先盯住这三步

  1. 把题面翻译成“当前函数要返回什么”。
  2. 用中序把左右子树边界切清楚。
  3. 让序列化、槽位验证和删点成林各自守住自己的协议。
你要会的动作具体表现错了会怎样
找返回值能说出 dfs 返回的是子树根、字符串、还是森林里的新树根。递归写出来以后，父节点接不回孩子。
切范围能用前序 / 中序、后序 / 中序切出左右子树长度。一位错开，整棵树都会拼错。
守协议能说明为什么序列化要有空节点标记，槽位法在检查什么。看起来像树，实际上已经不可逆。

先抓主线

主线图

这页最重要的主线只有一句：树题先问返回值，再问边界，再问协议。构造题看的是“根在哪里、左右子树多长”；序列化题看的是“写出来以后能不能按同一协议读回去”；删点成林看的是“当前节点删掉后，谁会变成新根”。

返回值账本

trace

构造题的返回值

105 和 106 都是在问：给定两条遍历序列，怎么还原出一棵树。你不能把它当成“在数组里找几个位置”的题，而要把它当成“当前函数返回当前子树根”的题。

一旦返回值是子树根，当前节点就只需要做三件事：找到根、切出左右范围、把左右递归结果接回去。复杂度也因此稳定在 O(n)，前提是你先把中序值到下标的映射建好。

序列化题的返回值

序列化和反序列化看似一写一读，实际上是在约定同一种协议。写的时候决定你怎么表达空节点，读的时候就必须按同一顺序消费这个协议。

如果写法和读法不一致，树就会错位。这个错位后面的所有节点都会顺着错误的边界继续错下去。

环节进入前进入后读什么写什么如果顺序错了
前序 + 中序构造还没切范围先定根，再切左右preorder[pl], inorder[mid]root.left / root.right边界错一位，左右子树交叉
中序 + 后序构造还没切范围先定根，再切左右postorder[pr], inorder[mid]root.left / root.right把右子树长度切错
前序序列化当前节点还没写写当前值，再写左右node / node.left / node.right值或 #漏掉空节点，结构不可逆
删点成林当前节点还没判删先处理孩子，再决定当前节点delete 集合、父节点状态forest 新树根孩子被丢掉，森林少树

$$

\text{根的位置} + \text{左子树长度} + \text{右子树长度}

$$

这里三个量都要落到具体边界上。中序里的根位置给出左右长度，长度再反推前序或后序里应该切到哪里。

前序 + 中序

105

105 的关键是前序首位给根，中序负责切边界。只要你能先算出左子树长度，前序和中序的两个切口就会同步对上。

代码
def buildTree(preorder, inorder):
 index = {v: i for i, v in enumerate(inorder)}

 def dfs(pl, pr, il, ir):
 if pl > pr:
 return None
 root_val = preorder[pl]
 root = TreeNode(root_val)
 mid = index[root_val]
 left_size = mid - il
 root.left = dfs(pl + 1, pl + left_size, il, mid - 1)
 root.right = dfs(pl + left_size + 1, pr, mid + 1, ir)
 return root

 return dfs(0, len(preorder) - 1, 0, len(inorder) - 1)

这段代码里最值得盯的不是语法，而是四个边界参数。pl, pr 管前序，il, ir 管中序；left_size 是桥。只要桥算对，两个序列就能对齐。

输入根左子树长度前序切口中序切口
[3, 9, 20, 15, 7]31[9] / [20,15,7][9] / [15,20,7]
左子树90单节点单节点
右子树201[15] / [7][15] / [7]

> 看错时最先坏的地方 如果 left_size 算错，前序和中序的范围会同时错开。错法通常“左右子树直接互相串了”。

中序 + 后序

106

106 和 105 的骨架一样，差别只在根的位置。后序的最后一个元素是根，所以你先从尾部拿根，再用中序切出左右边界。

代码
def buildTree(inorder, postorder):
 index = {v: i for i, v in enumerate(inorder)}

 def dfs(il, ir, pl, pr):
 if il > ir:
 return None
 root_val = postorder[pr]
 root = TreeNode(root_val)
 mid = index[root_val]
 left_size = mid - il
 root.left = dfs(il, mid - 1, pl, pl + left_size - 1)
 root.right = dfs(mid + 1, ir, pl + left_size, pr - 1)
 return root

 return dfs(0, len(inorder) - 1, 0, len(postorder) - 1)

这次最容易错的是右子树范围。因为后序是“左、右、根”，根在最后，所以右子树和左子树的位置交换感更强，边界很容易写反。

输入根左子树长度后序切口中序切口
[9, 3, 15, 20, 7]31[9] / [15,7,20][9] / [15,20,7]
右子树201[15] / [7][15] / [7]

> 两题共同点 根先定，范围后切，返回值都是当前子树根。你只要把这个顺序守住，105 和 106 的代码就只差根的位置。

具体走读

trace

拿一棵完整一点的树来走一遍，最容易看出“根 + 边界 + 返回值”是怎么连起来的。比如前序 [8,4,2,6,12,10,14]，中序 [2,4,6,8,10,12,14]。这组数据既能看见左子树，也能看见右子树，不会因为例子太小而把边界藏起来。

一眼看错的地方

如果把 8 的 left_size 算成 2，左子树就会少一个节点，右子树会多塞一个节点，后面 12、10、14 的切口也会跟着偏。这个错法最麻烦的地方在于，代码不会立刻炸，只是整棵树悄悄变形。

所以构造题不要只看“递归能不能跑通”，要看每次切口是不是和中序边界一致。边界一旦错开，父节点和孩子的关系就会被串掉。

层级根左子树长度前序切口中序切口如果算错会怎样
整棵树83[4,2,6] / [12,10,14][2,4,6] / [10,12,14]左右两边界都从 8 处分开
左子树41[2] / [6][2] / [6]2 和 6 的父子关系会错位
右子树121[10] / [14][10] / [14]10 和 14 会被拼到错误分支

最小追踪函数复制

def trace_split(preorder, inorder, pl, pr, il, ir):
 root = preorder[pl]
 mid = index[root]
 left_size = mid - il
 return root, left_size, (pl + 1, pl + left_size, il, mid - 1), (pl + left_size + 1, pr, mid + 1, ir)

> 练习钩子 先用这组数据把 105 走一遍，再故意把 left_size 减 1，看看右子树会从哪一层开始错。这样你就能直观看到“边界错一位，整棵树都跟着偏”的后果。

序列化

331

序列化的目标“把树变成一串能读回去的协议”。因此，空节点不能省。空节点一旦省掉，很多结构不同的树会写出相同的字符串。

代码
def serialize(root):
 vals = []

 def dfs(node):
 if node is None:
 vals.append('#')
 return
 vals.append(str(node.val))
 dfs(node.left)
 dfs(node.right)

 dfs(root)
 return ','.join(vals)


def deserialize(data):
 vals = iter(data.split(','))

 def dfs():
 val = next(vals)
 if val == '#':
 return None
 node = TreeNode(int(val))
 node.left = dfs()
 node.right = dfs()
 return node

 return dfs()

这里最关键的是写读顺序一致。你写的是前序，就必须按前序读；你写了 #，读的时候就要在看到 # 时立刻返回空。这个协议是成对的，不能只记一边。

协议元素写的时候做什么读的时候做什么少了会怎样
节点值先写当前节点先读当前节点顺序不一致，整棵树错位
#空孩子写 #读到 # 就返回 None结构信息丢失
分隔符把每个 token 分开按同一分隔符拆开读写协议不统一

> 889 的位置 889 的前序和后序构造会提醒你：普通二叉树并不总能唯一还原。这里先记住这个边界，别把“能写一种树”误当成“能唯一还原一棵树”。

前序 + 后序歧义

889

889 很适合拿来补边界：只有前序和后序时，普通二叉树通常不能唯一还原。原因很直接，前序告诉你“根先出现”，后序告诉你“根最后出现”，但中间那段左右子树的切口并没有被单独钉住。

什么条件会让它更稳

如果题目额外保证是真二叉树，也就是每个非叶子节点都有两个孩子，那么前序 + 后序会多出一层约束，很多时候可以更稳定地还原。没有这个条件时，就要接受“多解”的事实。

这个边界最容易影响的你对整类树题的判断：有些题是“求一棵合法树”，有些题是“求唯一结构”。这两个任务的心态完全不同，代码边界也会跟着不同。

信息前序能告诉你什么后序能告诉你什么缺了什么
根首位末位左右子树切口
左右关系先访问左再访问右先访问左再访问右后到根中间那条边界
结果能拼出一种树能拼出一种树不一定唯一

> 练习钩子 先把 [1,2,3] 画成两棵不同的树，再用前序和后序分别走一遍。你会看到遍历结果一样，但结构已经变了。这个小练习比记一句“前序 + 后序不唯一”更稳，因为它会让你真正看见歧义是怎么发生的。

槽位法

331

331 不要求你真的建树，它只检查前序序列化是否合法。最稳的想法是把每个节点看成“消耗一个槽位”的动作：非空节点消耗一个槽位，再给孩子增加两个槽位；空节点只消耗一个槽位，不会再补回去。

代码
def isValidSerialization(preorder):
 slots = 1
 for token in preorder.split(','):
 slots -= 1
 if slots < 0:
 return False
 if token != '#':
 slots += 2
 return slots == 0

> 为什么它比真的建树更快 因为它只检查协议，不需要保存整棵树。它做的事情其实是“数槽位是否被正确消耗完”。

删点成林

1110

1110 的难点不在删节点本身，而在删掉之后怎么把剩下的树接回森林。正确顺序是先递归处理左右孩子，再看当前节点要不要删。只有这样，孩子是否能升格成新根，才会在正确的位置上决定。

代码
def delNodes(root, to_delete):
 delete = set(to_delete)
 forest = []

 def dfs(node, is_root):
 if node is None:
 return None
 removed = node.val in delete
 if is_root and not removed:
 forest.append(node)
 node.left = dfs(node.left, removed)
 node.right = dfs(node.right, removed)
 return None if removed else node

 dfs(root, True)
 return forest

> 后序的意义 先处理孩子，再处理当前节点，这样当前节点删掉时，孩子已经是处理好的结果，可以直接决定要不要进森林。

故障走读

debug

这门课里最常见的错误，往往边界、协议和顺序有一处没守住。把错误先摆到桌面上，反而更容易看见真正的问题出在哪一层。

331：空节点漏写

症状是反序列化时树的形状不对，或者不同结构的树得到同一个字符串。最先检查的是序列化里有没有在空孩子位置写 #。

这类问题很容易被误判成“读代码出错”，其实错在写协议时已经丢了结构。读写一旦不同步，后面的节点再正确也救不回来。

1110：孩子没升格

症状是森林里少了一棵树，或者某个被删节点的孩子消失了。最先检查的是 is_root 传递和 removed 是否正确，孩子是不是在父节点删除后还保留了升格机会。

这一类错最适合先手画一棵小树，再标出哪一个节点被删、哪两个孩子应该接到森林里。只要手画图能看见，代码里的顺序通常就不会再乱。

症状最可能的原因第一检查点下一步动作
左右子树长度对不上105 / 106 的 left_size 算错中序里根的位置重新写出两个序列的切口
反序列化后结构变形空节点没有写 #序列化输出里是否记录空孩子把空孩子补回协议
森林少树1110 中孩子没有被正确升格is_root 和 removed 的传递把后序顺序重新画一遍

故障小抄：先看哪一层复制

构造题：先看中序边界，再看 left_size
序列化：先看空节点标记，再看读写顺序
删点题：先看孩子有没有先处理，再看新根有没有加入森林

> 练习钩子 你可以把这三个症状各写一个最小反例：105 用 [1,2,3] 的错切口，331 用没有 # 的序列，1110 用删掉根但孩子没接回的树。只要能把症状对应到第一检查点，后面调试就不会乱跑。

先别追

边界

现在先放下的概念

先别把注意力放到前序 + 后序唯一性、BST 重建、平衡树旋转、Trie 设计上。它们都和树有关，但今天的主线“构造、序列化、删除”的返回值和协议。

遇到这些词先记住一句话

  • 唯一还原：题目有没有额外条件。
  • 空节点标记：结构能不能读回去。
  • 后序删除：孩子要先处理完再升格。
  • 森林：被删节点的孩子可能成为多棵新树的根。

序列化和构造的关系

构造题问的是“给你两条遍历序列，能不能还原结构”；序列化题问的是“给你一棵树，能不能把结构保存成可读回的协议”。两者都在和结构打交道，但一个靠边界切分，一个靠协议复原。

如果你把这两个方向混了，就会出现两类错法：构造时忘了中序定位，序列化时漏了空节点标记。它们表面不同，根子都在“没有把结构信息保住”。

先别追的概念为什么先放下后面去哪儿看
前序 + 后序唯一性需要额外条件，不是本页主线。后续树题扩展
BST 重建它有排序性质，边界更特殊。13 BST 字典树与数据结构设计
树的平衡调整这是另一类维护问题，不是当前重点。后续进阶树结构专题
图最短路它已经不是树分解问题。15 BFS 最短路径框架

常见误区

调试

切片边界错一位

错因：没有先算 left_size，或者把前序和中序的起止位置写串了。

后果：左右子树节点数不一致，树直接拼错。

修法：先定根，再算 left_size，再写切口。

删点先处理当前再丢孩子

错因：把“当前节点删没删”放在前面，忘了孩子需要先处理。

后果：孩子没机会升格，森林少树。

修法：后序处理，当前节点最后决定返回什么。

槽位法把空节点当成可生长节点

错因：看到 # 以后也补槽位。

后果：slots 最后怎么都对不上。

修法：空节点只消耗，不生长。

代码
def serialize(root):
 vals = []
 def dfs(node):
 if node is None:
 return
 vals.append(str(node.val))
 dfs(node.left)
 dfs(node.right)
 dfs(root)
 return ','.join(vals)

这段代码丢掉了结构。只有节点值顺序，没有空孩子位置，反序列化无法唯一还原。

Q: 6. 中序 + 后序里，为什么更容易把右子树边界写错？

参考答案 / 评分要点

。因为后序是“左、右、根”，根在最后。你先从末位拿根，再用中序切出左子树长度；右子树在后序里紧挨着根的左边，很容易把左右范围写反。答案里要点出“根的位置”和“右子树范围”这两个点。

7scenario-transfer

Q: 7. 为什么前序序列化时只写非空节点值，会导致不同树写成同一个字符串？

参考答案 / 评分要点

。因为只保留了访问到的节点值，没有保留空孩子位置。不同结构的树会有同样的节点访问顺序，但空节点位置不同，所以字符串会撞在一起。答题时要明确“丢的是结构信息”。

8error-diagnosis

Q: 8. 105 或 106 里，如果 left_size 算错，会发生什么？怎么定位？

参考答案 / 评分要点

。错因是根定位后没有把中序左右长度算准。后果是两个序列的切口不同步，左右子树节点数对不上。修法是先写出根在中序的位置，再用 left_size = mid - il 统一切口，别先动前序或后序范围。

Q: 9. 331 的槽位法里，如果 slots 中途变成负数，说明什么？

参考答案 / 评分要点

。说明前面消耗的槽位已经超过了能提供的槽位，序列不可能是合法的前序序列化。修法看到负数就直接返回 False，因为协议已经被破坏了。

11scenario-transfer

Q: 11. 889 为什么提醒你“普通二叉树不一定唯一还原”？

参考答案 / 评分要点

。因为只有前序和后序时，普通二叉树可能有多个合法形状。没有中序那条边界，左右子树的切分不总是唯一，所以题目通常只要求返回其中一种。答题时要把“缺少中序边界”说出来。

Q: 12. 998 最大二叉树 II 为什么也能放进“分解思维”里看？

参考答案 / 评分要点

。因为它还是在处理“当前结构怎么拆、拆完怎么接回去”的问题，只是边界来自最大值性质，而不是遍历序列。答题时要说明它和 105 / 106 的共同点是“先确定结构切分，再处理子树”。

13interview

Q: 13. 用 2-3 句话说明“构造、序列化、删除”三个机制的共同点。

参考答案 / 评分要点

。共同点是它们都在处理子树，并且都要求当前函数返回一个明确结果：构造返回子树根，序列化返回字符串，删除返回删完后的子树根或森林新根。差别在于构造靠遍历边界，序列化靠协议，删除靠后序和孩子的升格。

Q: 14. 用 6-9 句话讲清本页最重要的不变量、复杂度、误区和迁移。

参考答案 / 评分要点

。高分答案通常会先说构造题返回子树根，中序用来切左右边界，复杂度在建好索引后是 O(n)。接着说明序列化题必须记录空节点并保持读写顺序一致，否则结构不可逆。然后说明 331 的槽位法是在检查协议是否被正确消耗，1110 的删点成林要先处理孩子再决定当前节点是否进森林。最后给出一个常见错误，比如 left_size 算错、漏写 # 或把删除顺序写反。答题时只要把对象、状态、后果、修法都说出来，就已经够完整。
