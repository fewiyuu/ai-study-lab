学习目标

起步

这一页只抓一件事：看到数据结构设计题，先把 API、状态和不变量拆出来，再决定用哪种结构。BST、Trie、LRU 看起来分散，实际上都在回答同一个问题: 哪些状态要读，哪些状态要改，改完以后还要守住什么规则。

BST 依赖有序边界，Trie 依赖字符路径和终止标记，LRU 依赖 key 定位和最近使用顺序。你如果先把这三条线分清，后面的 MinStack、FreqStack、Queue 设计题也会更容易落地。

先看这三件事

  1. 把题目里的每个 API 写出来，注明读什么、写什么。
  2. 把结构不变量写成一句话，能直接拿去检查代码。
  3. 把例子和错误放在一起看，知道哪里最容易漏。
要会的动作 具体表现 出错症状
先列 API 能说出 get、put、search、startsWith 分别读写什么。 直接套模板，后面发现缺状态。
再定不变量 能说出 BST 的上下界、Trie 的终止标记、LRU 的顺序规则。 看起来能跑，实际会误判或误淘汰。
最后做组合 能解释为什么某些题要把两个结构绑在一起。 单个结构不够，复杂度也达不到题意。

先抓主线

主线图

这页的主线很简单: 题面先给操作，再给约束。你要先判断它要查什么、改什么、排序什么、缓存什么，再从这里反推结构。BST 负责有序剪枝，Trie 负责前缀共享，LRU 负责命中后刷新顺序。

先读题面动作 查 / 改 / 维持顺序

看到 search、get、put、startsWith、erase 这些动词，先别急着写代码，先列这题到底要你维护什么。

再读状态 key、children、prev/next、range

BST 读范围，Trie 读路径，LRU 读 map 和链表，设计题读的是状态之间怎么同步。

写回去的东西 指针、标记、顺序

BST 更新子树指针，Trie 更新 children 和 is_word，LRU 更新链表位置和哈希表映射。

失败症状 漏判 / 漏标记 / 漏刷新

漏掉一个状态，常见结果就是 BST 误判、Trie 误搜、LRU 淘汰错 key。

边界提示 这页先只管三类典型设计题和它们的状态同步。AVL、红黑树、双数组 Trie、LFU、布隆过滤器都先放到后续专题，今天先把“API 反推结构”这条线看稳。
$$ \text{BST: } low < val < high \qquad \text{Trie: } path + is\_word \qquad \text{LRU: } dict[key] \rightarrow node + doubly linked list $$

这三句就够当第一页的记忆锚点。你后面每次检查代码，都可以先问自己: 上下界有没有传、终止标记有没有写、命中后有没有刷新顺序。

核心账本

state

设计题怎么落笔

先把 API 列出来，再把每个 API 读写的状态写清楚。读什么、写什么、谁来保存顺序、谁来保存定位，这四个问题一旦答出来，结构就基本确定了。

BST、Trie、LRU 其实都可以写成一个小账本: 读什么字段、改什么字段、如果改错了，外面会看到什么错误。

迁移到别的题

MinStack 读的是“到当前位置为止的最小值”，FreqStack 读的是“每个值的频次”和“当前最大频次”。这些题先找状态同步规则，再决定写哪种结构。

你会发现，设计题的难点经常不在数据结构本身，而在多个状态之间谁先更新、谁后更新。

结构 读什么 写什么 错了的症状
BST 当前节点值和祖先给出的上下界。 子树指针和新的递归边界。 右子树里偷偷混进更小的值。
Trie 字符路径和终止标记。 children 映射和 is_word。 前缀被当成完整单词。
LRU key 到节点的映射和链表顺序。 哈希表、链表头尾、节点位置。 命中后顺序不变，尾部删错 key。
MinStack 当前栈顶和到当前位置为止的最小值。 主栈和辅助栈同步入栈、出栈。 pop 后最小值还停在旧位置。
FreqStack 每个值的频次和当前最大频次。 频次表、分组栈、最大频次。 频率一样时，时间顺序排错。

Trie 读写

prefix

Trie 的关键是两层状态: 一层是路径，另一层是终止标记。路径只说明前缀存在，终止标记才说明完整单词存在。search 和 startsWith 的差别，就卡在这个标记上。

如果字符集固定，可以用长度 26 的数组；如果字符集不固定，字典更稳。你在写题时要先决定字符集，再决定 children 的容器类型。

记住这一句

插入只负责把路径补出来，search 还要检查 is_word，startsWith 只检查路径。这个差别很小，但几乎每次都会出现在错误答案里。

操作 读什么 写什么 常见错误
insert 每个字符对应的孩子节点。 缺的节点补出来，最后写 is_word=True。 路径补了，终止标记没写。
search 整条路径是否都存在。 不写任何结构，只返回判断结果。 把前缀当成单词。
startsWith 前缀路径是否存在。 不需要检查终止标记。 多做了终止判断，结果过严。
Trie 核心模板
class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_word = False


class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_word = True

    def search(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:
                return False
            node = node.children[ch]
        return node.is_word

    def startsWith(self, prefix):
        node = self.root
        for ch in prefix:
            if ch not in node.children:
                return False
            node = node.children[ch]
        return True

LRU 组合

cache

LRU 是这页最能说明“组合结构”的例子。哈希表负责按 key 找节点，双向链表负责维护最近使用顺序。少了前者，查找慢；少了后者，淘汰和刷新都难。

get 和 put 都会改变顺序。只要命中，就要把节点移到最前面；只要容量超了，就删最尾部。尾部是谁，取决于最近使用顺序，而不是插入时间本身。

顺序规则

头部放最近使用，尾部放最久没用。这个规则只要断一次，缓存的含义就变了。

命中快，顺序丢失，容量满时不知道淘汰谁。
操作 读什么 写什么 尾部会发生什么
get(key) 哈希表里 key 对应的节点。 把节点移到链表头部。 原来的最旧节点如果没被刷新，仍在尾部。
put(key, val) 这个 key 是否已存在，容量是否已满。 更新值、刷新顺序、必要时新增节点。 满了就删除尾部节点，再插入新节点。
capacity 超限 当前节点数量和尾部节点。 从哈希表和链表里同时删掉尾部。 如果只删一边，结构会断开。
LRU 节点操作
class Node:
    def __init__(self, key=0, val=0):
        self.key = key
        self.val = val
        self.prev = None
        self.next = None


def remove(node):
    node.prev.next = node.next
    node.next.prev = node.prev


def add_to_front(head, node):
    node.prev = head
    node.next = head.next
    head.next.prev = node
    head.next = node
LRU 缓存骨架
class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.map = {}
        self.head = Node()
        self.tail = Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    def get(self, key):
        if key not in self.map:
            return -1
        node = self.map[key]
        remove(node)
        add_to_front(self.head, node)
        return node.val

    def put(self, key, value):
        if key in self.map:
            node = self.map[key]
            node.val = value
            remove(node)
            add_to_front(self.head, node)
            return
        if len(self.map) == self.cap:
            old = self.tail.prev
            remove(old)
            del self.map[old.key]
        node = Node(key, value)
        self.map[key] = node
        add_to_front(self.head, node)

结构对照

compare

拿到新题时，先看题面关键词，再反推结构。关键在于记住每种结构在回答什么问题: BST 回答有序范围，Trie 回答前缀路径，LRU 回答最近使用顺序，MinStack 和 FreqStack 回答同步状态。

题面关键词 先问什么 常用结构 典型错法
有序、范围、验证 BST 当前节点是否落在祖先给的区间里。 递归 + 上下界 只看父子，不看祖先边界。
前缀、单词、字典 路径是否存在，末尾是否成词。 Trie 把前缀当完整单词。
缓存、命中、淘汰 谁要快速定位，谁要维护顺序。 哈希表 + 双向链表 命中后没刷新顺序。
最小值、最大频次 状态是否要同步，是否要分层保存。 辅助栈、分组栈 只存一个总状态，回退时丢信息。
$$ \text{BST} \rightarrow \text{range check};\quad \text{Trie} \rightarrow \text{path + end flag};\quad \text{LRU} \rightarrow \text{dict + order list};\quad \text{MinStack/FreqStack} \rightarrow \text{synchronized state} $$

如果你只能记一句话，就记这串。它比“背类名”更有用，因为它直接连到 API 和状态。

状态桥

synchrony

先把同步问题看清

MinStack 和 FreqStack 看起来和 Trie、LRU 不一样，但它们都在做同一件事: 主状态和辅助状态一起变，才能在回退或弹出时把信息完整拿回来。你只要其中一层断掉，后面的查询就会变得不可靠。

MinStack 的主状态是元素栈，辅助状态是“到当前位置为止的最小值”。FreqStack 的主状态是值本身，辅助状态是“这个值的频次”和“当前最高频次”。这两个题都要在存数据的同时把回退信息记住。

一个判断句

如果某个操作会改变未来的答案，那它就必须同步改多个状态。这个判断句可以直接拿去检查 push、pop、getMin、peekMax、popMax 这类题。

结构 输入操作 同时读写什么 如果少同步一步
MinStack push / pop / getMin 主栈和辅助最小值栈一起变。 pop 以后最小值还停在旧层。
FreqStack push / pop 频次表、频次分组栈、最大频次一起变。 同频时会弹错元素。
LRU get / put 哈希表定位节点，链表刷新顺序。 命中后顺序不变，尾部删错 key。
Trie insert / search / startsWith 路径节点和终止标记一起维护。 前缀被当成完整单词。
MinStack 的同步模板
class MinStack:
    def __init__(self):
        self.stack = []
        self.min_stack = []

    def push(self, x):
        self.stack.append(x)
        if not self.min_stack:
            self.min_stack.append(x)
        else:
            self.min_stack.append(min(x, self.min_stack[-1]))

    def pop(self):
        self.min_stack.pop()
        return self.stack.pop()

    def top(self):
        return self.stack[-1]

    def getMin(self):
        return self.min_stack[-1]

这段模板里最重要的是同步。每 push 一次，两个栈都要长一层；每 pop 一次，两个栈都要退一层。这样 getMin 才能直接看辅助栈顶，不必回头扫描整栈。

FreqStack 的同步模板
from collections import defaultdict

class FreqStack:
    def __init__(self):
        self.freq = defaultdict(int)
        self.group = defaultdict(list)
        self.max_freq = 0

    def push(self, x):
        self.freq[x] += 1
        f = self.freq[x]
        if f > self.max_freq:
            self.max_freq = f
        self.group[f].append(x)

    def pop(self):
        x = self.group[self.max_freq].pop()
        self.freq[x] -= 1
        if not self.group[self.max_freq]:
            self.max_freq -= 1
        return x

FreqStack 的同步点比 MinStack 多一层：既要知道每个值出现了几次，也要知道当前最大的频次。弹出时先从最高频那一组取，再把频次表回退一格。少这一步，频率和栈层就会脱开。

失败后果 如果辅助状态和主状态不同步，错的会是一整条后续路径。比如 MinStack 的最小值会漂，FreqStack 的同频顺序会乱，后面的 pop 也会跟着错。

结构选择

decision

题面关键词怎么落结构

看到“有序、范围、验证”，先想到 BST。看到“前缀、词典、字符路径”，先想到 Trie。看到“缓存、最近使用、淘汰”，先想到 LRU。这个顺序不花哨，但很好用，因为它直接对应题目想让你守住的状态。

如果题目要同时满足两种约束，比如既要按 key 快速查，又要按使用顺序淘汰，就不要犹豫，考虑组合结构。组合结构把两个职责拆开，各做各的事。

一个最短决策链

  1. 先问: 题目要你查谁、改谁、删谁。
  2. 再问: 是否需要顺序、范围或前缀。
  3. 最后问: 是否有一个状态必须同步保留。
关键词 先想什么 常用结构 为什么
有序、范围、排序 边界是否能被持续缩小。 BST 节点值自带上下关系。
前缀、字符串、词典 路径是否共享，末尾是否成词。 Trie 公共前缀只存一份。
缓存、命中、最近使用 谁要快速定位，谁要维护顺序。 哈希表 + 双向链表 一个管找，一个管删和移。
最小值、最高频率、实时结果 是否需要辅助状态回退。 同步栈 / 分组栈 单一状态回退时会丢信息。
结构选择伪代码
def choose_structure(keywords):
    if any(k in keywords for k in ["有序", "范围", "BST"]):
        return "BST"
    if any(k in keywords for k in ["前缀", "词典", "字符串"]):
        return "Trie"
    if any(k in keywords for k in ["缓存", "最近使用", "淘汰"]):
        return "HashMap + DoublyLinkedList"
    if any(k in keywords for k in ["最小值", "频率", "同步状态"]):
        return "Synchronized Stack Pattern"
    return "先列 API，再找状态"
失败后果 如果你把结构选错了，后面不是微调参数就能补救的。比如缓存题用普通数组，get 会慢；Trie 题用普通集合，前缀和单词就分不开；BST 题不传上下界，祖先约束会丢。

失败走读

diagnostic

把错法当输入看

调试设计题的时候，最有用的不是空想，而是把“错误输入 + 错误动作 + 错误结果”连起来看。下面三个例子都很小，但足够暴露结构为什么会坏。

BST 的错法是边界没传下去，Trie 的错法是终止标记没写，LRU 的错法是命中后没有刷新顺序。你会发现它们坏掉的位置都很清楚，只是影响会沿着后续状态继续扩散。

一眼就能看出的症状

  • BST 出现“右子树里更小的值”时，通常是祖先边界被丢了。
  • Trie 把前缀当单词，通常是 is_word 没写或没检查。
  • LRU 淘汰错 key，通常是 get 没刷新顺序，尾部旧节点还在。
错误场景 错在哪一步 外部症状 修法
BST 验证 递归时只传了父节点，没有传祖先区间。 深层节点越界却没被发现。 把 low / high 一起带下去。
Trie search 只检查路径，没检查终止标记。 前缀被当成完整单词。 search 的最后一步看 is_word。
LRU get 命中后只返回值，没有重排链表。 最近使用的 key 还停在旧位置。 先摘下，再插到头部。
失配示例：Trie 的前缀误判
def search(word):
    node = root
    for ch in word:
        if ch not in node.children:
            return False
        node = node.children[ch]
    return True  # 错误：没有看 is_word

这段错法最容易骗过自己，因为路径确实存在。真正的问题在最后一行: 它把“走到这条路上”当成“这就是完整单词”。正确写法要在结尾检查终止标记。

对应练习 如果你在练习区发现自己会写出这类错法，就回到上面的状态表，重新确认“读什么、写什么、错了会怎样”。调试设计题时，先查同步，再查逻辑。

例子与走读

walkthrough

BST 验证

根节点 8 的右子树里出现 7，看起来只是小了一个数，但它已经越过了从根传下来的下界。这个例子要看的，就是上下界比父子比较更强。

BST 验证代码
def isValidBST(root, low=float("-inf"), high=float("inf")):
    if root is None:
        return True
    if not (low < root.val < high):
        return False
    return (
        isValidBST(root.left, low, root.val)
        and isValidBST(root.right, root.val, high)
    )
节点 low high 是否合法
8 -inf inf 是
10 8 inf 是
7 8 10 否

LRU 顺序刷新

put(1,A), put(2,B), get(1), put(3,C) 之后，最先被淘汰的是 2。原因很直接: get(1) 已经把 1 刷到头部，2 才是最旧的节点。

边界与复杂度

limits

BST 重复值

题目要先看清是否允许重复值。验证条件如果要改成 `<=` 或 `<`，上下界写法也要一起改，不能只改一个比较符号。

Trie 字符集

字符集固定时，children 数组更快；字符集不固定时，字典更稳。你在写代码前，先确认输入到底是哪一种。

LRU 容量为 0

容量为 0 时，put 不能留下任何节点，get 永远返回未命中。这个边界很小，但如果没写，测试一来就会炸。

先留给后续专题 AVL、红黑树、双数组 Trie、LFU、布隆过滤器这些点都值得学，但和本页的目标不同。今天先把“如何从 API 反推出结构”学会，后面的专题再分别展开。
做法 时间 空间 适用场景
BST 平衡时查找 O(log n) O(h) 有序查找、范围查询
BST 最坏情况 O(n) O(h) 退化成链时的普通查找
Trie 插入 / 查询 O(k) O(字符总数) 前缀匹配、词典
LRU get / put O(1) O(capacity) 缓存设计

常见错误

debug

BST 只看父子

错因: 只比较了当前节点和左右孩子。

后果: 右子树里更小的值会被漏掉。

修法: 递归时把 low / high 一起传下去。

Trie 忘记 is_word

错因: 把路径存在当成单词存在。

后果: app 会被误判成已经插入。

修法: search 最后一定检查终止标记。

LRU get 不刷新

错因: 以为只有 put 才算使用。

后果: 最近访问过的节点仍然可能被淘汰。

修法: 命中后先摘下，再移到链表头。

双向链表指针漏改

错因: 删除时只改一边指针。

后果: 链表断裂或形成错误环。

修法: 每次操作都检查前驱和后继两侧。

设计题不列 API

错因: 先凭印象选结构，再补状态。

后果: 某个操作的复杂度达不到题意。

修法: 先列方法，再列状态，再列不变量。

错误示例：只看父子
def isValidBST(root):
    if root is None:
        return True
    if root.left and root.left.val >= root.val:
        return False
    if root.right and root.right.val <= root.val:
        return False
    return isValidBST(root.left) and isValidBST(root.right)

这个写法漏掉了祖先边界。右子树里只要藏进一个比根更小的值，它就可能被放过去。正确做法是把上下界带着递归。
