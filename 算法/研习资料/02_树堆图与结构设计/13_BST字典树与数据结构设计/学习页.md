# 先抓主线 {#overview}
<!-- tag: 主线图 -->

这页的主线很简单: 题面先给操作，再给约束。你要先判断它要查什么、改什么、排序什么、缓存什么，再从这里反推结构。BST 负责有序剪枝，Trie 负责前缀共享，LRU 负责命中后刷新顺序。

<!-- interactive: ['shape-tracer'] → keep as HTML -->

> **边界提示** 这页先只管三类典型设计题和它们的状态同步。AVL、红黑树、双数组 Trie、LFU、布隆过滤器都先放到后续专题，今天先把“API 反推结构”这条线看稳。

$$
 \text{BST: } low < val < high \qquad
 \text{Trie: } path + is\_word \qquad
 \text{LRU: } dict[key] \rightarrow node + doubly linked list
 $$
 这三句就够当第一页的记忆锚点。你后面每次检查代码，都可以先问自己: 上下界有没有传、终止标记有没有写、命中后有没有刷新顺序。

<!-- tag: state -->
<!-- layout: grid2 -->

## box
### 设计题怎么落笔

你会发现，设计题的难点经常不在数据结构本身，而在多个状态之间谁先更新、谁后更新。

# Trie 读写 {#trie}
<!-- tag: prefix -->
<!-- layout: grid2 -->

## box
Trie 的关键是两层状态: 一层是路径，另一层是终止标记。路径只说明前缀存在，终止标记才说明完整单词存在。search 和 startsWith 的差别，就卡在这个标记上。
如果字符集固定，可以用长度 26 的数组；如果字符集不固定，字典更稳。你在写题时要先决定字符集，再决定 children 的容器类型。

## box soft
### 记住这一句
插入只负责把路径补出来，search 还要检查 is_word，startsWith 只检查路径。这个差别很小，但几乎每次都会出现在错误答案里。

操作
读什么
写什么
常见错误


insert
每个字符对应的孩子节点。
缺的节点补出来，最后写 is_word=True。
路径补了，终止标记没写。


search
整条路径是否都存在。
不写任何结构，只返回判断结果。
把前缀当成单词。


startsWith
前缀路径是否存在。
不需要检查终止标记。
多做了终止判断，结果过严。

Trie 核心模板复制

```python
class TrieNode:
 def __init__(self):
 self.children = {}
 self.is_word = False


class Trie:
 def __init__(self):
 self.root = TrieNode()

 def insert(self, word):
 node = self.root
 for ch in word:
 if ch not in node.children:
 node.children[ch] = TrieNode()
 node = node.children[ch]
 node.is_word = True

 def search(self, word):
 node = self.root
 for ch in word:
 if ch not in node.children:
 return False
 node = node.children[ch]
 return node.is_word

 def startsWith(self, prefix):
 node = self.root
 for ch in prefix:
 if ch not in node.children:
 return False
 node = node.children[ch]
 return True
```

# LRU 组合 {#lru}
<!-- tag: cache -->
<!-- layout: grid2 -->

## box
LRU 是这页最能说明“组合结构”的例子。哈希表负责按 key 找节点，双向链表负责维护最近使用顺序。少了前者，查找慢；少了后者，淘汰和刷新都难。
get 和 put 都会改变顺序。只要命中，就要把节点移到最前面；只要容量超了，就删最尾部。尾部是谁，取决于最近使用顺序，而不是插入时间本身。

## box soft
### 顺序规则
头部放最近使用，尾部放最久没用。这个规则只要断一次，缓存的含义就变了。

<!-- interactive: ['terminal-lab'] → keep as HTML -->

操作
读什么
写什么
尾部会发生什么


get(key)
哈希表里 key 对应的节点。
把节点移到链表头部。
原来的最旧节点如果没被刷新，仍在尾部。


put(key, val)
这个 key 是否已存在，容量是否已满。
更新值、刷新顺序、必要时新增节点。
满了就删除尾部节点，再插入新节点。


capacity 超限
当前节点数量和尾部节点。
从哈希表和链表里同时删掉尾部。
如果只删一边，结构会断开。

LRU 节点操作复制

```python
class Node:
 def __init__(self, key=0, val=0):
 self.key = key
 self.val = val
 self.prev = None
 self.next = None


def remove(node):
 node.prev.next = node.next
 node.next.prev = node.prev


def add_to_front(head, node):
 node.prev = head
 node.next = head.next
 head.next.prev = node
 head.next = node
```

LRU 缓存骨架复制

```python
class LRUCache:
 def __init__(self, capacity):
 self.cap = capacity
 self.map = {}
 self.head = Node()
 self.tail = Node()
 self.head.next = self.tail
 self.tail.prev = self.head

 def get(self, key):
 if key not in self.map:
 return -1
 node = self.map[key]
 remove(node)
 add_to_front(self.head, node)
 return node.val

 def put(self, key, value):
 if key in self.map:
 node = self.map[key]
 node.val = value
 remove(node)
 add_to_front(self.head, node)
 return
 if len(self.map) == self.cap:
 old = self.tail.prev
 remove(old)
 del self.map[old.key]
 node = Node(key, value)
 self.map[key] = node
 add_to_front(self.head, node)
```

# 状态桥 {#bridge}
<!-- tag: synchrony -->
<!-- layout: grid2 -->

## box
### 先把同步问题看清
MinStack 和 FreqStack 看起来和 Trie、LRU 不一样，但它们都在做同一件事: 主状态和辅助状态一起变，才能在回退或弹出时把信息完整拿回来。你只要其中一层断掉，后面的查询就会变得不可靠。
MinStack 的主状态是元素栈，辅助状态是“到当前位置为止的最小值”。FreqStack 的主状态是值本身，辅助状态是“这个值的频次”和“当前最高频次”。这两个题都要在存数据的同时把回退信息记住。

## box soft
### 一个判断句
如果某个操作会改变未来的答案，那它就必须同步改多个状态。这个判断句可以直接拿去检查 push、pop、getMin、peekMax、popMax 这类题。

结构
输入操作
同时读写什么
如果少同步一步


MinStack
push / pop / getMin
主栈和辅助最小值栈一起变。
pop 以后最小值还停在旧层。


FreqStack
push / pop
频次表、频次分组栈、最大频次一起变。
同频时会弹错元素。


LRU
get / put
哈希表定位节点，链表刷新顺序。
命中后顺序不变，尾部删错 key。


Trie
insert / search / startsWith
路径节点和终止标记一起维护。
前缀被当成完整单词。

MinStack 的同步模板复制

```python
class MinStack:
 def __init__(self):
 self.stack = []
 self.min_stack = []

 def push(self, x):
 self.stack.append(x)
 if not self.min_stack:
 self.min_stack.append(x)
 else:
 self.min_stack.append(min(x, self.min_stack[-1]))

 def pop(self):
 self.min_stack.pop()
 return self.stack.pop()

 def top(self):
 return self.stack[-1]

 def getMin(self):
 return self.min_stack[-1]
```

这段模板里最重要的是同步。每 push 一次，两个栈都要长一层；每 pop 一次，两个栈都要退一层。这样 getMin 才能直接看辅助栈顶，不必回头扫描整栈。

FreqStack 的同步模板复制

```python
from collections import defaultdict

class FreqStack:
 def __init__(self):
 self.freq = defaultdict(int)
 self.group = defaultdict(list)
 self.max_freq = 0

 def push(self, x):
 self.freq[x] += 1
 f = self.freq[x]
 if f > self.max_freq:
 self.max_freq = f
 self.group[f].append(x)

 def pop(self):
 x = self.group[self.max_freq].pop()
 self.freq[x] -= 1
 if not self.group[self.max_freq]:
 self.max_freq -= 1
 return x
```

FreqStack 的同步点比 MinStack 多一层：既要知道每个值出现了几次，也要知道当前最大的频次。弹出时先从最高频那一组取，再把频次表回退一格。少这一步，频率和栈层就会脱开。

> **失败后果** 如果辅助状态和主状态不同步，错的会是一整条后续路径。比如 MinStack 的最小值会漂，FreqStack 的同频顺序会乱，后面的 pop 也会跟着错。

# 结构选择 {#choice}
<!-- tag: decision -->
<!-- layout: grid2 -->

## box
### 题面关键词怎么落结构
看到“有序、范围、验证”，先想到 BST。看到“前缀、词典、字符路径”，先想到 Trie。看到“缓存、最近使用、淘汰”，先想到 LRU。这个顺序不花哨，但很好用，因为它直接对应题目想让你守住的状态。
如果题目要同时满足两种约束，比如既要按 key 快速查，又要按使用顺序淘汰，就不要犹豫，考虑组合结构。组合结构把两个职责拆开，各做各的事。

## box soft
### 一个最短决策链
1. 先问: 题目要你查谁、改谁、删谁。
2. 再问: 是否需要顺序、范围或前缀。
3. 最后问: 是否有一个状态必须同步保留。

关键词

结构选择伪代码复制

```python
def choose_structure(keywords):
 if any(k in keywords for k in ["有序", "范围", "BST"]):
 return "BST"
 if any(k in keywords for k in ["前缀", "词典", "字符串"]):
 return "Trie"
 if any(k in keywords for k in ["缓存", "最近使用", "淘汰"]):
 return "HashMap + DoublyLinkedList"
 if any(k in keywords for k in ["最小值", "频率", "同步状态"]):
 return "Synchronized Stack Pattern"
 return "先列 API，再找状态"
```

> **失败后果** 如果你把结构选错了，后面不是微调参数就能补救的。比如缓存题用普通数组，get 会慢；Trie 题用普通集合，前缀和单词就分不开；BST 题不传上下界，祖先约束会丢。

# 失败走读 {#failure-lab}
<!-- tag: diagnostic -->
<!-- layout: grid2 -->

## box
### 把错法当输入看
调试设计题的时候，最有用的不是空想，而是把“错误输入 + 错误动作 + 错误结果”连起来看。下面三个例子都很小，但足够暴露结构为什么会坏。
BST 的错法是边界没传下去，Trie 的错法是终止标记没写，LRU 的错法是命中后没有刷新顺序。你会发现它们坏掉的位置都很清楚，只是影响会沿着后续状态继续扩散。

## box soft
### 一眼就能看出的症状
- BST 出现“右子树里更小的值”时，通常是祖先边界被丢了。
- Trie 把前缀当单词，通常是 is_word 没写或没检查。
- LRU 淘汰错 key，通常是 get 没刷新顺序，尾部旧节点还在。

把 low / high 一起带下去。

只检查路径，没检查终止标记。

失配示例：Trie 的前缀误判复制

```python
def search(word):

```

这段错法最容易骗过自己，因为路径确实存在。真正的问题在最后一行: 它把“走到这条路上”当成“这就是完整单词”。正确写法要在结尾检查终止标记。

# 边界与复杂度 {#boundary}
<!-- tag: limits -->

BST 重复值
题目要先看清是否允许重复值。验证条件如果要改成 `<=` 或 `<`，上下界写法也要一起改，不能只改一个比较符号。


Trie 字符集
字符集固定时，children 数组更快；字符集不固定时，字典更稳。你在写代码前，先确认输入到底是哪一种。


LRU 容量为 0
容量为 0 时，put 不能留下任何节点，get 永远返回未命中。这个边界很小，但如果没写，测试一来就会炸。

# 常见错误 {#mistakes}
<!-- tag: debug -->

BST 只看父子
错因: 只比较了当前节点和左右孩子。
后果: 右子树里更小的值会被漏掉。
修法: 递归时把 low / high 一起传下去。


Trie 忘记 is_word
错因: 把路径存在当成单词存在。
后果: app 会被误判成已经插入。
修法: search 最后一定检查终止标记。


LRU get 不刷新
错因: 以为只有 put 才算使用。
后果: 最近访问过的节点仍然可能被淘汰。
修法: 命中后先摘下，再移到链表头。


双向链表指针漏改
错因: 删除时只改一边指针。
后果: 链表断裂或形成错误环。
修法: 每次操作都检查前驱和后继两侧。


设计题不列 API
错因: 先凭印象选结构，再补状态。
后果: 某个操作的复杂度达不到题意。
修法: 先列方法，再列状态，再列不变量。

错误示例：只看父子复制

```python
def isValidBST(root):
 if root is None:
 return True
 if root.left and root.left.val >= root.val:
 return False
 if root.right and root.right.val <= root.val:
 return False
 return isValidBST(root.left) and isValidBST(root.right)
```

这个写法漏掉了祖先边界。右子树里只要藏进一个比根更小的值，它就可能被放过去。正确做法是把上下界带着递归。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
## box soft
### 怎么答才算会
先把 API、状态和不变量说清楚，再下结论。选择题要能一眼说出为什么对，开放题要能指出对象、状态变化、后果和修法。

## box info
### 检查重点
BST 看上下界，Trie 看路径和终止标记，LRU 看哈希表 + 双向链表。别只背名字，要能说出每个字段为什么存在。

<!-- end-practice-intro -->

**API**先列方法，再列每个方法读写什么。
**不变量**每次操作后结构规则仍然成立。
**顺序**get 也算使用，Trie 的 search 还要看 is_word。
**组合**单个结构不够时，直接说明为什么要组合。

### 1 判断

**Q:** 1. 验证 BST 时，只比较父节点和左右孩子是不够的。

- [ ] 对
- [ ] 错

Answer: 对

### 2 判断

**Q:** 2. Trie 中路径存在，只能说明前缀存在；完整单词还要看 is_word。

- [ ] 对
- [ ] 错

Answer: 对

### 3 选择

**Q:** 3. LRU 要让 get / put 都接近 O(1)，常用组合是：

- [ ] 哈希表加双向链表
- [ ] 只用普通数组
- [ ] 只用递归遍历

Answer: 哈希表加双向链表

### 4 填空

**Q:** 4. Trie 查询长度为 k 的单词或前缀，时间复杂度通常是 ______。

- [ ] O(k)
- [ ] O(n log n)
- [ ] O(1)

Answer: O(k)

<!-- practice-group: 走读与状态 -->

### 5 走读

**Q:** 5. 设根是 8，右子节点是 10，10 的左孩子是 7。为什么 7 不能通过 BST 验证？

Explain: 因为进入右子树后，下界已经变成 8。7 虽然小于 10，但仍然小于 8，所以不合法。高分答案要能把“祖先下界”说出来，而不是只说“它比父节点小”。

### 6 走读

**Q:** 6. 插入 apple 后，search("app") 和 startsWith("app") 为什么可能不同？

Explain: search 还要检查 app 这个节点有没有被标成完整单词，所以它可能是 False；startsWith 只要求路径存在，所以它是 True。答案里至少要提到路径和终止标记两个概念。

### 7 选择

**Q:** 7. 容量 2 的 LRU 执行 put(1), put(2), get(1), put(3) 后，被淘汰的是谁？

- [ ] 2
- [ ] 1
- [ ] 3

Answer: 2

<!-- end-practice-group -->

<!-- practice-group: 错误诊断 -->
<!-- quick-review: 复习一下 错误通常发生在状态同步被打断。 -->

### 8 诊断

**Q:** 8. BST 验证如果只比较父子，会漏掉什么？

Explain: 会漏掉祖先带下来的边界，最典型的情况就是右子树深处出现一个比根更小的值。修法是递归传 low / high，而不是只看当前节点的左右孩子。

### 9 诊断

**Q:** 9. Trie 搜索忘记 is_word，会造成什么误判？

- [ ] 会把前缀误判成完整单词
- [ ] 会把完整单词误判成前缀
- [ ] 只会影响插入速度

Answer: 会把前缀误判成完整单词

### 10 诊断

**Q:** 10. LRU 中 get 命中但不移动节点，会造成什么淘汰错误？

Explain: 最近访问过的 key 仍然会停在旧位置，尾部就可能删错。修法是命中后先从链表里摘掉，再把它插到头部。

<!-- end-practice-group -->

### 11 迁移

**Q:** 11. 155 最小栈为什么需要辅助栈和主栈同步增长？

Explain: 因为每个位置都要知道“到这里为止的最小值”，pop 时才能无损回退。答案里要明确“同步增长”是为了让每个状态都可回退，不是为了浪费空间。

### 12 迁移

**Q:** 12. 最大频率栈需要哪些状态，才能同时按频率和时间出栈？

- [ ] 需要频次表、分组栈和当前最大频次
- [ ] 只要一个普通栈就够了
- [ ] 只要记录最早出现的元素

Answer: 需要频次表、分组栈和当前最大频次

### 13 表达

**Q:** 13. 用 2-3 句话说明如何从 API 要求反推数据结构。

Explain: 先列出每个 API 要读写的状态，再看哪些状态必须 O(1) 定位、哪些状态必须保留顺序，最后确认不变量能不能同时满足。只要某个状态同时承担两种职责，就要考虑组合结构。

### 14 synthesis

**Q:** 14. 用 6-9 句话讲清 BST、Trie、LRU 的不变量、复杂度和误区。

Explain: 高分答案会先说 BST 依赖上下界，平衡时查找是 O(log n)，最坏会退化到 O(n)。接着说 Trie 用路径表示前缀、用 is_word 表示完整单词，复杂度看字符串长度。然后说 LRU 用哈希表定位节点、双向链表维护最近使用顺序，get 和 put 都要刷新顺序。最后要点出常见误区，比如只看父子、把前缀当单词、get 不刷新顺序。答题时尽量把“对象、状态、后果、修法”都说到。
