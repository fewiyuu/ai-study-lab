图算法：拓扑、并查集与 Dijkstra

这一页只处理三类图题：先后依赖、动态连通、非负权最短路。题目先看“要交什么”，再决定用入度、根节点还是距离数组；等权最短步数留到下一节 BFS。

上一节：BST 字典树与数据结构设计 下一节：BFS 最短路径框架

读完本页应该能做什么

  • 能把“课程表 / 先修 / 环”翻成拓扑排序，并顺着入度走一轮。
  • 能把“连通 / 合并 / 动态查询”落到并查集的 find、union 和路径压缩上。
  • 能把“非负权最短路 / 最小代价 / 概率最大路径”落到 dist、heap 和松弛条件上。

学习目标

goal

这一页不按“图算法”这个总名词讲，而是按题目要你交什么来分：先后顺序、同组关系，还是最小代价。你先把输出目标认清，再去看入度、根节点或距离数组。

题面信号 先问什么 先看什么状态 你要得到什么
课程表、先修课、入度、能否完成 有没有依赖顺序和环 入度、队列、已处理数量 一个合法顺序，或者“有环”
连通、合并、是否同组、动态加边 是否只关心集合归属 parent、size / rank、根节点 两个点是否同组，或连通分量数
最小代价、最短时间、概率最大 边权是不是非负 dist、优先队列、过期状态 从起点到各点的最优值
等权最短步数 是不是该把这页先放下 下一节的 BFS 队列和层数 去下一页，而不是硬套 Dijkstra

这页的边界很明确：先把三类图算法的状态看熟，再去做迁移题。等权最短步数、0-1 BFS、Bellman-Ford、最小生成树放到后面单元，不在这里展开。

先分题型

overview

图题容易混，常见原因是题面把依赖、集合、代价放在一起说。真正的分界线在输出目标：要的是合法顺序、同组判断，还是最短代价。

$$ \text{依赖顺序}\rightarrow \text{拓扑排序}\qquad \text{动态连通}\rightarrow \text{并查集}\qquad \text{非负最短路}\rightarrow \text{Dijkstra} $$

这句话的作用，是把“题目想让你输出什么”压成一个判断句。你先抓输出，再抓状态。

拓扑看先后

先修课、任务依赖、编译顺序，这类题都在问先后关系。你要管的是入度，别拿它去比树高或路径长度。

并查集看归属

只要题目关心“是不是一组”“能不能连起来”“动态加边后还在不在一起”，就先找根节点。路径细节通常不在答案里。

Dijkstra 看代价

只要边权是非负的，且题目问的是代价最小、时间最短、概率最大，就先看 dist 和优先队列，别先往遍历上想。

怎么选

decision

先看题目要求你输出什么，再看约束条件。题目问“先做谁后做谁”，通常就是拓扑；题目问“是否同组 / 是否连上”，通常就是并查集；题目问“最小代价 / 最短时间 / 最大概率”，边权又是非负的，就看 Dijkstra。

先建有向图和入度表，入度为 0 的点先处理。
题面线索 先问什么 要看的状态 不先做什么
先修课、编排顺序、课程表 有没有先后依赖 入度、队列、已处理数量 先别写 DFS 遍历
合并集合、连通性、动态加边 是不是只关心同组 parent、size、root 先别找具体路径
最短时间、最小代价、最大概率 边权是不是非负 dist、heap、过期项 先别硬套 BFS

再压一层：先后关系看边方向和入度，连通关系看根节点，代价关系看 `dist` 是否还能变小。你不需要把所有图题都翻成同一个模板，只要先把题面句子翻成“我要维护什么状态、我先改哪一项”，后面的代码就会顺很多。

题面翻译器

translate

这一步专门处理“我知道题在图上，但不知道第一行怎么写”的卡点。先把题面里的关键词翻成状态名，再把状态名翻成第一步动作。这样做题时，脑子里先出现的是账本，不是算法名。

题面词 真正要维护的东西 第一步 最容易写错的地方
先修、依赖、完成顺序 入度、队列、已处理数量 先把边定成 pre → nxt，再统计入度 边方向写反，后面队列顺序就偏了
连通、合并、同组、动态加入 parent、root、size 先让每个点指向自己，再做 union 把中间父节点当根，合并会失真
最短时间、最小代价、最大概率 dist、heap、过期项 先设起点距离，再把起点压进堆 没跳过旧状态，堆会一直长

先写状态，再写算法名

如果你手里只有算法名，题目会显得很散；一旦把状态名写出来，代码入口就清楚了。拓扑先写 indeg，DSU 先写 parent，Dijkstra 先写 dist，这三个入口会逼着你把题意读对。

拓扑走读

topo

拓扑排序处理的是有向无环图。题目给你先修关系时，边要从“先修”指向“后续”。这样一来，入度就变成了“还有多少前置任务没完成”。

看一个最小例子：0→1，0→2，1→3，2→3。初始入度是 [0,1,1,2]，队列先放 0。处理 0 以后，1 和 2 都变成入度 0；处理 1 和 2 以后，3 才能入队。顺序可以不同，但每一步都必须尊重依赖。

边方向先定死

如果把边反过来，入度含义就反了。你会先处理后修课，结果顺序整体倒置，看起来像顺序，其实是反向结果。

轮次 队列 入度变化 你该看什么
开始 [0] [0,1,1,2] 只有 0 可以先做
处理 0 空 [0,0,0,2] 1 和 2 同时变成可处理
处理 1 / 2 [3] 或先 1 后 2 [0,0,0,0] 3 现在才真正进入队列
结束 空 已清空 处理数量等于节点数，说明没有环
拓扑模板
from collections import deque

def canFinish(n, prerequisites):
    graph = [[] for _ in range(n)]
    indeg = [0] * n
    for nxt, pre in prerequisites:
        graph[pre].append(nxt)
        indeg[nxt] += 1

    q = deque([i for i in range(n) if indeg[i] == 0])
    seen = 0
    while q:
        cur = q.popleft()
        seen += 1
        for nxt in graph[cur]:
            indeg[nxt] -= 1
            if indeg[nxt] == 0:
                q.append(nxt)
    return seen == n

`graph[pre].append(nxt)` 这一行决定了依赖方向。`seen == n` 代表所有节点都被合法处理过；少了就是有环。

并查集走读

union-find

并查集回答的是“是不是同一组”。它不关心路径长什么样，只关心两件事：谁是代表元，两个集合要怎么并到一起。最常见的写法是 parent 加 size 或 rank，再加路径压缩。

把 1、2、3 连起来看：先 `union(1,2)`，再 `union(2,3)`。只要 `find` 会一路找根并顺手压缩，最后 1 和 3 的代表元就相同。没有路径压缩也能对，但树会越来越长，后面的查询会慢。

它不回答路径

并查集能告诉你“在不在一组”，不能告诉你“中间经过哪条边”。如果题目要具体路径，那就不是并查集该做的事。

操作 读什么 写什么 错一步会怎样
find(x) parent[x] 把沿途节点直接挂到根上 树会变长，重复查询更慢
union(a, b) 两个集合的代表元 把小集合挂到大集合上 树高失控，后面 find 成本上涨
same(a, b) 两个根是否相同 不改结构 如果根找错，连通判断就错
并查集模板
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.size = [1] * n

    def find(self, x):
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False
        if self.size[ra] < self.size[rb]:
            ra, rb = rb, ra
        self.parent[rb] = ra
        self.size[ra] += self.size[rb]
        return True

`find` 返回的是根，不是 `parent[x]`。`union` 只改根之间的关系，不去动每一条边。

Dijkstra 走读

shortest path

Dijkstra 处理的是非负权最短路。`dist` 保存目前找到的最小代价，优先队列每次拿当前最小的候选点出来扩展。只要一条边权不为负，这个“先出队就是当前最优”的逻辑才站得住。

看一个小图：0→1(2)，0→2(5)，1→2(1)，1→3(2)，2→3(1)。起点 0 的距离先是 0，其余是无穷大。弹出 0 后，1 和 2 被更新；弹出 1 后，2 的距离从 5 改成 3，3 变成 4；之后堆里可能还留着旧的 (5,2)，弹出时要直接跳过。

旧状态要跳过

堆里会同时有新距离和旧距离。只要弹出的距离已经比 `dist[u]` 大，它就是过期项，继续扩展只会重复工作。

$$ \text{if } dist[u] + w < dist[v] \text{ then } dist[v] = dist[u] + w $$

这里的 `dist[u]` 是当前已知最优代价，`w` 是边权，`dist[v]` 是候选点 v 的最优代价。松弛就是拿更小的值覆盖更大的值。

为什么负权会坏掉

Dijkstra 依赖一个前提：当前堆顶的点，一旦被弹出，就不会再被后面的边改出更小的值。这个前提在非负权下成立，因为后续路径只会越走越大；一旦出现负权，后面可能把已经弹出的点继续拉低，堆顶的“先出队就是最优”就不稳了。

所以这页只把 Dijkstra 放在非负权最短路里讲。题目一旦带了负权，先停下，别把“堆快”误认成“答案对”。

弹出 堆顶 dist 变化 你该做什么 如果错了会怎样
0 (0,0) [0,2,5,∞] 把邻居都尝试松弛一遍 如果没写松弛，dist 不会变
1 (2,1) [0,2,3,4] 发现 2 的更短路，更新 如果不比较更优，会用坏路径覆盖好路径
2 (3,2) [0,2,3,4] 只处理真正更短的候选 如果不跳过旧项，会重复扩展
旧项 (5,2) 不变 直接跳过 如果继续跑，队列会被旧状态拖慢
Dijkstra 模板
import heapq

def dijkstra(n, graph, start):
    dist = [float('inf')] * n
    dist[start] = 0
    pq = [(0, start)]
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]:
            continue
        for v, w in graph[u]:
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    return dist
概率最大路径变形
import heapq

def maxProbability(n, edges, probs, start, end):
    graph = [[] for _ in range(n)]
    for (a, b), p in zip(edges, probs):
        graph[a].append((b, p))
        graph[b].append((a, p))

    best = [0.0] * n
    best[start] = 1.0
    pq = [(-1.0, start)]
    while pq:
        neg, u = heapq.heappop(pq)
        cur = -neg
        if cur < best[u]:
            continue
        if u == end:
            return cur
        for v, p in graph[u]:
            nxt = cur * p
            if nxt > best[v]:
                best[v] = nxt
                heapq.heappush(pq, (-nxt, v))
    return 0.0

`dist` 可以换成 `best`，最小堆可以借助负号反过来用。核心没有变，还是“先拿最优候选，再松弛邻居”。

同一张图的三种问法

trace

很多人会把图题看成“都能遍历一下”。问题不在遍历，而在问法。同一张图，换一个提问方式，答案就落到不同的状态上。下面把同一组节点分三次看，目的是让你看到“问法变了，工具就变”。

输入骨架 拓扑问法 并查集问法 Dijkstra 问法
0→1, 0→2, 1→3, 2→3 0 先做，1 和 2 之后，3 最后 如果把边当成“关系”，它只是一个错误的连通图 如果边权是 2、5、1、1，就要追代价最小的路
1-2, 2-3, 4-5 没有依赖顺序，不该硬做拓扑 1、2、3 是一组，4、5 是一组 如果边权全是 1，那是下一页 BFS 的工作
1→2(3), 1→3(6), 2→3(2) 有向边里可以先问顺序，但这题真正要的是代价 集合关系不是答案重点 1→2→3 比 1→3 更便宜，松弛会把 6 改成 5

别把“能画成图”误认成“该用同一个算法”

拓扑、并查集、Dijkstra 都在图上跑，但它们拿走的信息完全不同。拓扑拿的是先后关系，并查集拿的是集合归属，Dijkstra 拿的是到达代价。你如果先问错了问题，后面的代码写得再工整也不会对。

这个区分在面试里尤其重要。面试官给你一条先修链，你说 Dijkstra；给你一个连通判断，你说拓扑；给你一个最短代价，你却先写 DFS。只要题目问法没抓住，答案就容易偏到别的算法上。

三者对照

compare

把三种图算法压成一句话：拓扑是“谁先做”，并查集是“是不是一组”，Dijkstra 是“谁代价更小”。你只要这句话没看偏，后面的模板就不会乱。

算法 回答什么 核心状态 最怕什么 不在这页做什么
拓扑排序 先后顺序是否成立 入度、队列、已处理数量 边方向反、把无向图当 DAG 不负责具体路径
并查集 两个点是不是同组 parent、size、root find 不压缩、union 不看根 不负责路径和最短路
Dijkstra 非负权最短代价 dist、heap、过期项 负权边、旧状态不跳过 不处理等权最短步数
$$ \text{拓扑}\sim\text{入度};\qquad \text{并查集}\sim\text{根节点};\qquad \text{Dijkstra}\sim\text{dist + heap} $$

这串话是在把你真正要维护的状态写短。看到题面以后，先把状态抓出来，再开始写代码。

从题面到第一行代码

bridge

新题最容易卡在开头。你不用先把完整代码想完，只要先把题面翻成三件事：图怎么建，状态怎么存，答案怎么判。只要这三件事落地，第一行代码就不会飘。

题面信号 先选哪种工具 第一步写什么 错了会看到什么症状 先拿什么小例子验手感
课程表、先修、入度、能否完成 拓扑排序 先建 `graph[pre].append(nxt)`，再把 `indeg[nxt] += 1` 队列里先出错的人，或者有些点永远进不了队列 0→1，0→2，1→3，2→3，先手推一轮入度变化
连通、合并、是否同组、动态加边 并查集 先 `ra, rb = find(a), find(b)`，再合并根节点 如果只看 `parent[x]`，你会把中间父节点误当代表元 1-2-3 和 4-5 两棵小树，先看 `find` 怎么把路径压短
最短时间、最小代价、最大概率 Dijkstra 先设 `dist[start] = 0`，再把 `(0, start)` 放进堆 如果不跳过旧状态，堆会被过期项拖慢；如果有负权，最优先出队的前提会失效 0→1(2)，0→2(5)，1→2(1)，先看 `dist` 怎么从 5 改到 3
等权最短步数 BFS 先把它记到下一页 如果硬套 Dijkstra，只是在多做一层堆维护 1091、994、127、433、542 这类题，留到 BFS 单元处理

这个桥段的用途很实际：当你做题时，只要题面里先后关系、归属关系、代价关系其中一个词出现，就可以直接对照这张表落到第一行代码。先把状态写出来，再反推算法名。这样写出来的代码更稳，也更容易检查一开始有没有把边方向、根节点、堆顶距离放对。

三种第一步
# 拓扑：边方向先定清楚
graph[pre].append(nxt)
indeg[nxt] += 1

# 并查集：先找根，再决定是否合并
ra, rb = find(a), find(b)
if ra != rb:
    parent[rb] = ra

# Dijkstra：先把起点距离压进堆
dist[start] = 0
heapq.heappush(pq, (0, start))

先验手感，再验正确

这三段第一步都很短，但它们决定了后面一整条状态链能不能对上。拓扑错在边方向，后面入度会一直不对；并查集错在没找根，后面同组判断会飘；Dijkstra 错在没先设起点和堆，后面的松弛就没有起点可追。

如果你做题时卡住，先回到这里做一个最小检查：拓扑把 0→1→2 走一遍，看看入度会不会清零；并查集把 1-2-3 走一遍，看看根会不会收缩成一个；Dijkstra 把 0→1(2)、0→2(5)、1→2(1) 走一遍，看看 2 的距离会不会被 3 覆盖。这个检查比直接重写整题更快。

一轮走读：先修、归属、代价

walkthrough

把三种图题放在同一轮里看，差异会很直接。先看题目要你交什么：一个顺序、一个同组判断，还是一个最小代价。下面这轮走读只追一次状态变化，目的是让你看到代码为什么从第一行就要分叉。

步骤 拓扑：0→1, 0→2, 1→3, 2→3 并查集：1-2-3 Dijkstra：0→1(2), 0→2(5), 1→2(1) 这一轮你在读写什么
初始 indeg = [0,1,1,2]，队列只有 0 parent = [1,2,3]，每个点先指向自己 dist = [0,∞,∞]，堆里只有 (0,0) 读的是初始账本，写的是第一批可处理状态
第一步 弹出 0，1 和 2 的入度各减 1 union(1,2)，把两个小集合先并起来 弹出 0，先把 1 和 2 松弛出来 拓扑写入队列，並查集写入根，Dijkstra 写入更小的 dist
第二步 1 和 2 变成 0，3 还留着 union(2,3)，最后 1、2、3 会收到同一个根下 弹出 1，2 从 5 改成 3，3 变成 4 前一轮留下的新状态，决定下一轮谁能继续往下走
结束前检查 看 seen 是否等于节点数，判断有没有环 看 find(1) 和 find(3) 是否相同，判断是否同组 看堆里是否还有过期项，弹出时直接跳过 看的是不变量，不是某一次的临时结果
一轮走读的三个检查点
# 拓扑：处理完一个点后，邻居入度减一
for nxt in graph[cur]:
    indeg[nxt] -= 1

# 并查集：先找根，再合并
ra, rb = find(a), find(b)
if ra != rb:
    parent[rb] = ra

# Dijkstra：只有更小的距离才写回去
nd = dist[u] + w
if nd < dist[v]:
    dist[v] = nd

错一层，后面就会偏

拓扑里如果把边反了，入度会在错误的人身上减少；并查集里如果没先找根，根节点会被写乱；Dijkstra 里如果不做 `nd < dist[v]` 的比较，坏路会直接覆盖好路。三类错误的表面现象不同，但根子都在第一步没把状态写对。

练习时怎么用这轮走读

你可以把这轮走读当成纸面检验：随便拿一个新题，先只写三行，图怎么建、状态怎么初始化、答案怎么判。然后用这一页的三个最小例子对照一遍。能对上，你再扩成完整代码；对不上，就先回头改状态，不要急着堆模板。

题型 最小输入 你在纸上写的第一步 看什么结果就知道对了
拓扑 0→1, 0→2, 1→3, 2→3 写出 indeg=[0,1,1,2]，把 0 放进队列 0 出队后，1 和 2 一起变成 0
并查集 1-2-3 先让每个点自己指向自己，再做 union(1,2) 继续 union(2,3) 后，same(1,3)=true
Dijkstra 0→1(2), 0→2(5), 1→2(1) 写出 dist=[0,∞,∞]，把 (0,0) 放进堆 1 被弹出后，2 的距离从 5 改成 3

状态读写账本

ledger

如果你在图题里老是“看过就会、自己写就乱”，常见原因是看模板时没追到每一步读写了什么，也没看清哪一层状态在传递。下面这张表把三类图算法的读写关系摆开。

算法 读什么 写什么 写错会怎样 下一步检查什么
拓扑 当前节点的入度和邻接表 邻居入度减一，必要时入队 边方向反了，队列里永远先出错的人 先修边是否 pre → nxt
并查集 parent[x] 和 root 路径压缩、合并根节点 树会越来越长，查询越来越慢 find 有没有真的返回根
Dijkstra dist[u]、堆顶距离、邻居边权 更小的 dist[v]，以及新的堆项 旧状态反复扩展，或者更优路被覆盖 是否先比较再更新，是否跳过过期项
$$ \text{拓扑} \Rightarrow \text{indeg} \to \text{queue}\quad \text{并查集} \Rightarrow \text{parent} \to \text{root}\quad \text{Dijkstra} \Rightarrow \text{dist} \to \text{heap} $$

这不是口号，是调试顺序。你看到队列、根节点、堆顶时，就该想到它们各自读什么、改什么、哪一步最容易写错。

这张账本的价值在于，你可以把它直接拿去调试。代码一旦错了，先别急着改语法，先问自己“我这一步读错了什么，写漏了什么，哪条状态链断了”。这个问题通常比“算法名是什么”更快把你带回正确方向。

边界与复杂度

limits

这页只讲三种常见图工具，不把所有图论题都塞进来。等权最短步数、0-1 BFS、Bellman-Ford、最小生成树都还在后面，先别在这里硬追。

算法 时间 空间 适用边界 不适合什么
拓扑排序 O(V+E) O(V+E) 有向无环图的依赖顺序 无向图、带权最短路
并查集 近似 O(1) O(V) 动态连通、合并集合 路径查询、最短路
Dijkstra O(E log V) O(V+E) 非负权最短代价 负权边、等权步数

边界提示

如果题目问的是“每一步都一样”，那是下一页 BFS 的事；如果题目带了负权，Dijkstra 就停；如果题目要具体路径或枚举方案，并查集也停。别把工具往题目头上硬套。

再补一句：拓扑排序通常解决的是“先后顺序”和“有无环”，它并不擅长回答“这条路有多长”。如果你已经开始在拓扑题里写距离数组，通常说明题型已经被你看偏了。

常见错误

debug

图算法最容易出错的地方，通常在状态和边界。你看错了一步，后面整条链都会偏。

入度统计会反过来，队列先出的是后修课，结果顺序倒置。
症状 常见原因 先检查什么 下一步怎么修
顺序反了或有些课永远进不了队列 拓扑边方向写反 先修课是不是指向后续课 把边统一改成 pre → nxt
连通判断慢，越查越拖 find 没有路径压缩 是否真的返回了根 把沿途节点直接挂到根上
最短路答案不稳或重复扩展很多次 Dijkstra 里有负权，或者没跳过旧项 边权是否非负，弹出距离是否过期 负权换算法，旧项直接 continue
题目明明是课程表，却写成最短路 把“依赖顺序”误看成“代价最小” 题目到底要顺序还是要距离 先把题型重新分层，再选状态账本

调试时的顺序也可以固定下来：先看边有没有按题意连，拓扑尤其先看 pre → nxt；再看根节点有没有被正确压缩，DSU 常常错在这里；最后看堆里有没有旧项没被跳过，Dijkstra 的性能和结果都可能被拖偏。这个顺序比随手扫一遍代码更有效。
