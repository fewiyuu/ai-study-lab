从递归到表格

动态规划三种形态：暴力递归、备忘录与迭代 DP

先用爬楼梯和 Fibonacci 搭一个最小递推，再把同一批状态分别放进递归、备忘录和表格里。你会看到，DP 的关键在于把状态句子、依赖和答案位置对齐。

上一节：回溯约束题 课程地图 下一节：经典模型

读完后能做的三件事

  • 能把爬楼梯或 Fibonacci 的递归写成“状态 + 选择 + base case”。
  • 能看出一段代码是在重复递归、命中缓存，还是按依赖填表。
  • 能顺着初始化、遍历顺序、缓存键这三个点把答案偏掉的原因找出来。

学习目标

goals

看最后一步从哪来、dp[0] 和 dp[1] 为什么先写、同一个状态为什么会被反复问到。把这三点看顺，再去看递归、备忘录和表格，就不会只是在记名字。

这页的边界

本页只讲最基础的 DP 形态：重复递归、备忘录、迭代表格、初始化、遍历顺序和复杂度。它不替你展开更复杂的模型，只给后续单元的入口。

状态 → 选择 → base case → 备忘录 → 迭代表

先抓主线

overview

暴力递归

先按问题定义去拆，优点是最接近“我会怎么想”，缺点是同一个子问题会被多次算到。

备忘录

递归结构不变，只是在返回结果前先查缓存。算过的状态直接复用，重复计算就被压掉了。

迭代 DP

把递归调用顺序倒过来，按照依赖先后填表。这个形态更容易控制时间、空间和遍历顺序。

什么时候该换成 DP

当一个状态能由更小的同类状态算出来，而且这些小状态会反复出现，就先考虑缓存或填表。

核心机制

mechanism

爬楼梯的递推式

设 ways(n) 表示走到第 n 阶的方法数。最后一步只有两种来源：从 n-1 走 1 步，或者从 n-2 走 2 步，所以：

$$ ways(n) = ways(n-1) + ways(n-2) $$

这里的 ways(n) 是状态，n-1 和 n-2 是选择来源。只要这两个来源的答案已经知道，当前状态就能算出来。

三种形态读同一棵树

  • 暴力递归：每次都往下问，第一次看上去最自然，但会重复问到同一个状态。
  • 备忘录：第一次问到某个状态时先记下来，后面直接返回，不再往下展开。
  • 迭代表：把“先问谁”改成“先算谁”，让小状态先出现，大状态后出现。
递归树 ways(5)

往下会再次问到 ways(4)、ways(3)、ways(2)；同一个状态会被多次展开。

缓存命中 memo[3]

第一次算出 ways(3) 后直接存起来，后续再问到它就不用继续递归了。

表格顺序 dp[2] → dp[3] → dp[4]

先让依赖项进入表，再写当前格子；顺序反了就会用到还没算好的值。

例子

example

爬楼梯：递归版、备忘录版、表格版

递归写法
def climb_stairs(n):
    if n < 0:
        return 0
    if n == 0:
        return 1
    if n == 1:
        return 1
    return climb_stairs(n - 1) + climb_stairs(n - 2)
备忘录写法
def climb_stairs(n):
    memo = {}

    def dfs(i):
        if i < 0:
            return 0
        if i == 0:
            return 1
        if i in memo:
            return memo[i]
        memo[i] = dfs(i - 1) + dfs(i - 2)
        return memo[i]

    return dfs(n)
迭代表写法
def climb_stairs(n):
    if n <= 1:
        return 1
    dp = [0] * (n + 1)
    dp[0] = 1
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]
    return dp[n]

为什么这三个版本本质一样

它们都在算同一个状态 ways(i)。区别只在于：递归版每次从上往下问，备忘录版问过的结果会缓存，迭代表版直接从小到大把所有状态排好顺序算完。

如果你把状态定义成“到达第 i 阶的方法数”，那么 base case 就应该先把 dp[0] 和 dp[1] 填好；如果你把状态定义成“走到第 i 阶”，那递推式和初始化也要一起改。

手算追踪

trace

把 n=0 到 n=5 走一遍

手算的目的不是背答案，而是看清同一批状态在不同写法里怎么被处理。递归版会继续往下问，备忘录版第一次算到就记住，迭代表先放好 dp[0] 和 dp[1]，后面的值只从左边的已知状态里算。

你可以把这三种写法都看成在回答同一个问题：在第 i 层，当前状态读了哪些旧值，写了什么新值，哪些值如果写错会把后面的状态一起带偏。这个视角比“代码长什么样”更重要，因为它直接影响你能不能排查 bug。

和调试有关的三个观察点

  • 如果重复展开太多，先怀疑状态没有缓存，再去看递推式。
  • 如果答案偏一，先看 base case 和 dp[0]/dp[1]，再看循环的起点。
  • 如果表格里某一格像是“凭空长出来”，先看它依赖的左、上、左上是不是已经就位。

状态与填表

states

状态应该像一句完整的话

先写“dp[i] 表示什么”，再写转移。对于爬楼梯，最自然的写法是“dp[i] 表示到达第 i 阶的方法数”；对于 Fibonacci，可以把它写成“dp[i] 表示第 i 项的值”。状态一旦说不清，初始化和答案位置通常都会跟着错。

二维 DP 的状态更要说清两个维度分别代表什么。编辑距离里，dp[i][j] 不是“前 i 个和前 j 个”，而是“word1 的前缀到 i，变成 word2 的前缀到 j 的最少操作数”。

填表顺序只看依赖

如果 dp[i] 依赖 dp[i-1] 和 dp[i-2]，就从小到大算；如果 dp[i][j] 依赖左、上、左上，就要保证这三个格子先算好。顺序写反，表格里的值看起来会“有数字”，但其实是在用未完成的状态推出当前状态。

问题状态句子依赖答案位置
爬楼梯到达第 i 阶的方法数i-1, i-2dp[n]
Fibonacci第 i 项的值i-1, i-2dp[n]
编辑距离两个前缀的最少编辑数左、上、左上dp[m][n]

空间压缩

rolling

为什么爬楼梯只要两个变量

因为 dp[i] 只依赖 dp[i-1] 和 dp[i-2]。一旦 dp[i] 算完，dp[i-2] 就会变成更老的状态，下一轮再也不会读到它。于是你可以把整张表压成两个滚动变量：一个存前一项，一个存前两项。

这个优化成立的前提很苛刻：你必须确认当前状态只依赖有限个旧状态，而且这些旧状态不会在同一轮里被别的地方再用一次。否则压缩后，旧值可能被新值覆盖，表面上更省空间，实际上答案已经乱了。

什么时候不能随便压缩

如果一个状态同时要读左、上、左上，或者还要读整行、整列、整段历史，就不能只靠两个变量。压缩前先回答一个问题：当前格子到底会不会再用到更远的旧值？如果会，就先别压。

滚动变量写法
def climb_stairs_roll(n):
    if n <= 1:
        return 1
    prev2 = 1
    prev1 = 1
    for i in range(2, n + 1):
        cur = prev1 + prev2
        prev2 = prev1
        prev1 = cur
    return prev1
变量读什么写什么什么时候会错
prev2上一轮的前两项被移成更老的前一项如果先改它，再算 cur，当前值会丢
prev1上一轮的前一项被当前 cur 覆盖如果直接把 prev1 改掉，prev2 还没来得及用就没了
curprev1 + prev2当前状态值如果没有单独保存，顺序稍微错一点就会串值

和后面单元的关系

这一页先把空间压缩的底线讲清楚，后面的背包、LIS 和二维 DP 会继续使用这个判断：只要依赖关系没搞清楚，就不要先压缩。真正省空间，不是把代码改短，而是先证明哪些旧状态不会再被读到。

二维前缀桥接

prefix

从一维状态到二维状态，只差一个“前缀”视角

这一页讲的是“同一条状态怎么换成表格”。下一节里的编辑距离和 LCS 会把这个思路再推进一步：不是只看一个 i，而是同时看两个前缀的长度 i 和 j。这时候状态不再是“第 i 项”或者“到达第 i 阶”，而是“前缀对前缀的关系”。

一维 DP 和二维 DP 的差别，关键不在维度数，而在你到底把“最后一步”理解成什么。爬楼梯里最后一步只有两种来源；编辑距离里最后一步可能是插入、删除、替换或保持不动；LCS 里最后一步要判断两个末尾字符能不能一起留下来。结构变了，状态句子也要变。

二维前缀最容易错的地方

  • 把 dp[i][j] 说成“前 i 个和前 j 个”，却没说它到底是长度、次数还是代价。
  • 忘掉第 0 行和第 0 列，它们专门负责空串和空前缀。
  • 先写转移，后补状态，最后发现左上角、上方和左方的含义根本对不上。
问题一维直觉二维前缀直觉下一节会怎样接上
爬楼梯第 i 阶的方法数无须第二个前缀先练状态定义和填表顺序
编辑距离不够用word1 前缀到 word2 前缀的代价看插入、删除、替换三种来源
LCS不够用两个字符串前缀共同保留的长度看相等时接左上角，不等时取 max

这一步为什么要在这里讲

如果只会一维 DP，读到二维模型时很容易把状态写成“前几个字符的答案”，却没有说清前缀之间是什么关系。先在这里把桥搭好，下一节就能直接看编辑距离和 LCS。

编辑距离的前缀桥

bridge

为什么这里要先看一个二维例子

爬楼梯和 Fibonacci 让你看见“一维状态怎么重复”。编辑距离则把这个判断推进到二维：当前状态不是只看一个位置，而是看两个前缀的组合。它的意思是，前缀一到前缀二，要花多少步；这个“花多少步”才是真正的状态，而不是两个下标本身。

例如 word1 = horse、word2 = ros。如果你已经知道 horse 的前缀到 hor、ros 的前缀到 ro 的最少编辑数，那么当前格子只需要比较三种来源：删掉 word1 的最后一个字符、给 word1 增加一个字符、或者把最后一个字符替换成目标字符。这个思路和爬楼梯完全一样，只是来源从两个变成三个。

状态账本怎么写

  • dp[i][j] 读的是两个前缀的旧答案。
  • 左边、上边、左上角分别对应插入、删除、替换或不动。
  • 第 0 行和第 0 列负责空串，它们不是装饰，是整个表的地基。
格子含义读哪些旧值写入后代表什么
dp[0][0]空串到空串无0 次操作
dp[i][0]前缀到空串上方删除 i 次
dp[0][j]空串到前缀左方插入 j 次
dp[i][j]前缀到前缀左、上、左上当前最少编辑数

和爬楼梯的区别

爬楼梯只有两个来源，所以你只要把递归展开压成表格就够了；编辑距离里，选择数变成三个，而且每个选择都在改变字符串的一个前缀位置。状态还是先定义、再转移、再填表，但你必须更严格地对齐“读什么”和“写什么”。

这个桥接段的作用，是让你看到“DP 的难点不是维度数量，而是状态语义是否还能跟依赖关系对上”。下一节的 LIS、编辑距离和 LCS 都会沿着这个原则继续往前走。

LCS 里的切分点

lcs cut

为什么要区分“子序列”和“子串”

LCS 最容易出错的地方是把它写成公共子串。子序列允许跳字符，所以两个末尾字符不相等时，要看上方和左方哪个更大，不要清零。这里和爬楼梯、编辑距离一样，都是先把最后一步讲清楚，再写状态转移。

例如 text1 = abcde、text2 = ace，最长公共子序列是 ace，并不要求三个字符在两个字符串里连续出现。你如果把“连续”误加进来，就会把答案从 LCS 搞成公共子串，题目就已经换了。

这和前面两页怎么接

  • 和爬楼梯一样，先把状态句子写清楚。
  • 和编辑距离一样，状态要看两个前缀。
  • 和滚动变量一样，先确认依赖顺序，再考虑是否压缩。
情况末尾字符相等末尾字符不等如果写反会怎样
LCS接左上角 + 1取上方和左方最大值会把子序列写成子串
公共子串继续累加清零会把连续限制带进来

为什么这一刀要切开

很多人看到二维 DP 就开始套“左上角 + 1”，结果把 LCS、最长公共子串、编辑距离全混在一起。真正的分界线不在公式形式，而在你到底允许不允许跳过字符。这个判断和本页一直在讲的原则一致：先把状态语义说清，再写依赖。

调试循环

debug

一个常见错法怎么查

假设你把爬楼梯写成了 dp[i] = dp[i - 1] + dp[i - 2]，但 dp[0] 和 dp[1] 没有先放好，或者把循环从 i=0 开始。这个错误表面上只是一行代码，实际会把整张表的起点一起带偏。查的时候不要先看答案，先回到状态账本，问自己：这一格本来应该读什么，写什么，起点有没有就位。

如果你在二维表里看到 dp[i][j] 像是“凭空算出来”的，就按同样的方法查：左、上、左上三个旧格子是否都已经算过；如果没有，先修遍历顺序，再修转移公式。这个检查顺序比直接重写代码更快。

错误会长成什么样

  • 起点没放好：答案整体偏移，常见表现是少 1 或多 1。
  • 依赖顺序反了：表里每一格都像是能算，但值和手算对不上。
  • 缓存键不完整：二维状态混成一维，结果时好时坏。
先查什么具体看点错误症状下一步
状态这格到底表示什么看起来能跑，但语义漂了重写状态句子
起点dp[0]、dp[1]、第 0 行、第 0 列答案整体偏一先修 base case
顺序依赖项是否先算值像被“污染”调整遍历方向
缓存键是否覆盖完整状态不同子问题撞在一起补全 key

和练习区怎么接

如果你在练习里卡住，先别急着重刷题，用这张表做一次定位：是状态没说清，还是起点没放好，还是顺序反了。只要把错位的那一层找出来，后面的代码通常只改一处就能回正。

Fibonacci 的第二个锚点

anchor

为什么再看一次 Fibonacci

爬楼梯和 Fibonacci 的递推结构非常像，但它们的状态语义不一样。爬楼梯是在数“走法”，Fibonacci 是在数“值”。这点看起来很小，却会影响 base case 和答案位置。很多人会把两个题混着记，等到遇到变形题时，状态定义就会开始漂。

把这两个题并排看，最有价值的不是“公式一样”，而是你会发现：一旦状态句子说清了，表格自然会自己长出来；一旦状态句子没说清，代码怎么补都像在打补丁。

两道题的差别

  • 爬楼梯的状态对应方法数，答案是 dp[n]。
  • Fibonacci 的状态对应数值，答案也是 dp[n]，但语义不同。
  • 爬楼梯的 dp[0] = 1 是“什么都不做也算一种方案”；Fibonacci 的 dp[0] 则是数列定义的一部分。
项目爬楼梯Fibonacci为什么不能直接互换
状态句子到达第 i 阶的方法数第 i 项的值语义不同，不能只抄公式
base casedp[0]=1, dp[1]=1按数列定义设置一个是方案数，一个是数值定义
转移dp[i-1] + dp[i-2]dp[i-1] + dp[i-2]形式相同，不代表语义相同
错误后果答案偏一或起点没立住数列定义跑偏把“形式一样”误认成“题意一样”

Fibonacci 的状态账本

trace

从一维递推看“读、写、错”

如果把 dp[i] 定义成 Fibonacci 的第 i 项，那么 dp[i] 每次读的是前两格，写进去的是当前项。你要盯住三件事：当前格子读的旧值是不是已经算过，当前格子写进去的值是不是和状态句子一致，最后返回的位置是不是和状态句子一致。

这就是 DP 的检查顺序：先看定义，再看依赖，再看初始化，最后看答案位置。这个顺序一旦颠倒，你很容易写出一段“能运行但不可信”的代码。

一行代码对应什么动作

dp[i] = dp[i - 1] + dp[i - 2] 不是一句数学式子，而是一次明确的状态写入：读前一项、读前前项、合成当前项、把结果存进当前格子。看懂这行以后，编辑距离里的左、上、左上也就不再神秘，因为它们只是另一种“读旧值”的方式。

1无dp[1]没有第一步递推没法起飞

3dp[2], dp[1]dp[3]读到旧值顺序错了表面正确，实际偏移
4dp[3], dp[2]dp[4]如果顺序反了，前面的值会被覆盖答案会漂向别的位置
Fibonacci 迭代表
def fib(n):
    if n <= 1:
        return n
    dp = [0] * (n + 1)
    dp[0] = 0
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]
    return dp[n]

复杂度与边界

complexity

暴力递归为什么慢

它会把同一个状态展开很多次。爬楼梯里，ways(5) 会反复问到 ways(4)、ways(3)、ways(2)，递归树越往下，重复越密。

备忘录为什么快

每个状态第一次算完以后就不再展开。时间从“递归树的所有节点”压缩成“状态总数 × 每个状态的转移代价”，这是 DP 最常见的复杂度读法。

迭代表为什么更稳

它避免了递归栈，也更容易检查初始化、边界和遍历方向。但它要求你提前想清每个状态依赖哪些旧状态，不能先压缩空间再回头猜依赖。

$$ T(n) = T(n-1) + T(n-2) + O(1) $$

暴力递归看递归树；备忘录和迭代表看“状态数 × 每个状态的转移代价”。如果一个状态只会算一次，复杂度就从重复展开退到可控的规模。

Symptom: the same subproblem appears many times.

常见误区

mistakes

状态没说完整

只写 dp[i]，不写它表示什么，后面的公式就像在背口令。判断标准很简单：别人能不能只看这句话就知道你在算什么。

base case 只写样例

把 n=1、n=2 之类的样例值硬塞进去，但不说明为什么。真正的 base case 应该对应最小子问题的已知答案。

缓存键或顺序错位

二维状态只用一个维度做缓存，或者表格遍历方向反了。结果不是立刻报错，而是答案悄悄错掉，这类错误最难发现。

错误示例：把最后一位当成全局答案
def lis_bad(nums):
    dp = [1] * len(nums)
    for i in range(1, len(nums)):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    return dp[-1]   # 错：最后一位只是最后一个状态，不一定代表全局答案

先别追的旁支

这一页先不追区间 DP、树形 DP、状态压缩、数位 DP 和概率 DP。它们都还会用到“状态、依赖、初始化”这些词，但计算形态和状态空间更复杂，留给后面的单元处理。

迁移到后续

transfer

下一节为什么是 LIS、编辑距离、LCS

因为它们把同一套思路变成了更有代表性的模型：LIS 是“以位置结尾”的一维 DP，编辑距离是“前缀对前缀”的二维 DP，LCS 是“两个前缀共同保留”的二维 DP。你只要在这一页把状态、依赖和填表顺序说清，下一节就是换状态定义，不是换思路。

看到变形题时先问的四件事

  1. 状态表示什么。
  2. 最后一步有哪些来源。
  3. base case 放哪一行或哪一列。
  4. 答案从哪个格子读出来。
