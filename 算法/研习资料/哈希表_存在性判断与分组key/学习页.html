从题目读出哈希表模板

很多算法题不是难在语法，而是难在你不知道该把原始数据变成什么结构。哈希表的价值就在这里：它让“查一个东西是否存在”和“把同类东西放在一起”变得直接。

这一页只盯住两种用法。`set` 负责存在性判断，`dict` 负责按 key 分组。128「最长连续序列」用 `set` 避免重复扩展；49「字母异位词分组」用标准化 key 把同类字符串放进同一个桶。

  2. 看到“顺序不同但本质相同”，能想到先造稳定 key。
  3. 能解释 128 为什么只从段起点开始数，避免退化成重复扫描。
  4. 能解释 49 为什么原字符串不能直接当分组依据。
  5. 不看题解，写出两题的核心代码和复杂度。

哈希表到底帮你省了什么

concept

从线性扫描变成直接查询

普通列表查一个值，需要从头到尾看，最坏是 `O(n)`。如果你在一个循环里反复这样查，整体很容易变成 `O(n^2)`。

`set` 和 `dict` 的思路是先把值放进一个能快速定位的位置。你问“这个值在不在”时，不再从头找，而是通过哈希值直接定位到大致位置。日常刷题里可以把它当成平均 `O(1)` 查询。

这不是说哈希表永远免费。它会多用空间，也可能受哈希冲突影响。但算法题里，只要数据规模大、存在性查询多，空间换时间通常是正确方向。

存在性判断：为什么 128 不能从每个数都开始数

set

128「最长连续序列」的题目只要求长度，不要求返回具体序列。数组无序，还可能有重复。最直接的想法是：对每个数 `num`，不断查 `num + 1` 在不在。这个想法有一半是对的，因为“不断查下一个数”确实该用 `set`。问题在于，如果你从每个数都开始查，同一段会被重复数很多遍。

正确的关键判断是：只有当 `num - 1` 不存在时，`num` 才是一个连续段的起点。比如 `[1,2,3,4]` 里，只有 `1` 可以开始数。`2` 前面有 `1`，说明它只是中间节点；从 `2` 开始数会重复统计已经统计过的段。

起点判断：
num 是连续段起点 当且仅当 num - 1 不在集合中
连续段起点演示

切换数组，观察哪些数字会被当成起点。绿色是起点，灰色是被跳过的中间数字。

最长段：1 → 4

只从 1 开始扩展，依次查到 2、3、4。2、3、4 都不是起点，因为它们前一个数存在。

你现在要能判断：一个数字前面已经有人能接上它，就没必要从它重新开始。

分组 key：为什么 49 要先标准化字符串

49「字母异位词分组」问的是：哪些字符串由同一批字母组成。`eat`、`tea`、`ate` 原字符串不同，不能直接当同一组；但它们排序之后都是 `aet`。这个排序后的结果就是稳定 key。

分组题的重点是“同类对象必须得到同一个 key，不同类对象尽量得到不同 key”。只要 key 造对了，剩下就是固定动作：`groups[key].append(s)`。

排序法不是唯一方法

如果字符串很长，排序每个字符串的成本是 `O(k log k)`。另一种做法是统计 26 个字母频次，用长度为 26 的元组当 key，单个字符串是 `O(k)`。

但在 力扣 49 里，排序法更好写，也更容易解释。你先掌握排序 key，再补频次数组 key。

128 最长连续序列：逐行拆解

leetcode 128

这题的核心不是“会不会 while”，而是“从谁开始 while”。如果起点判断写对，每个连续段只会被完整走一次。数组里即使有重复，`set(nums)` 也会先去重，避免重复数字影响统计。

正确示例：只从连续段起点扩展Python
class Solution:
    def longestConsecutive(self, nums: list[int]) -> int:
        num_set = set(nums)
        best = 0

        for num in num_set:
            if num - 1 not in num_set:
                cur = num
                length = 1

                while cur + 1 in num_set:
                    cur += 1
                    length += 1

                best = max(best, length)

        return best

关键行读什么、写什么

  • `num_set = set(nums)`：读取原数组，写出一个去重集合，后面所有存在性判断都查它。
  • `if num - 1 not in num_set`：判断 `num` 是不是段起点。没有这一行，重复扩展会明显增加。
  • `while cur + 1 in num_set`：沿着连续段往后走。这里查的是集合，不是列表。
  • `best = max(best, length)`：每完成一个段，再更新全局最大长度。

复杂度为什么是 O(n)

外层循环看似遍历所有数。内层 `while` 只会从每个连续段的起点开始跑。对于一段 `[1,2,3,4]`，只有 `1` 会触发扩展，`2`、`3`、`4` 都被跳过。

所以每个数字最多参与一次“向后扩展”。平均情况下，整体是 `O(n)` 时间、`O(n)` 空间。

49 字母异位词分组：逐行拆解

leetcode 49

这题的核心是 key。原字符串只能代表自己，不能代表“同一组字母”。排序后的字符串能代表一类，所以它适合作为字典 key。

正确示例：排序字符串作为分组 keyPython
from collections import defaultdict

class Solution:
    def groupAnagrams(self, strs: list[str]) -> list[list[str]]:
        groups = defaultdict(list)

        for s in strs:
            key = "".join(sorted(s))
            groups[key].append(s)

        return list(groups.values())

为什么不能直接用 sorted(s)

`sorted(s)` 返回的是列表，比如 `['a','e','t']`。Python 的列表可变，不能作为字典 key。`"".join(sorted(s))` 把它变成字符串，字符串不可变，可以作为 key。

如果使用频次数组，也要转成 `tuple(count)`，不能直接用 list。

复杂度怎么看

设一共有 `n` 个字符串，平均长度是 `k`。每个字符串排序是 `O(k log k)`，总时间是 `O(nk log k)`。存储分组结果和 key 需要额外空间，通常按 `O(nk)` 理解。

面试里说复杂度时，不要只说 `O(n)`，因为字符串内部排序也要算。

可选写法：26 个字母频次作为 keyPython
count = [0] * 26
for ch in s:
    count[ord(ch) - ord("a")] += 1
key = tuple(count)

常见错误：错因、后果和修法

debug

从每个数字都开始扩展

错因：只想到了 `set`，没想到“起点判断”。

后果：长连续段会被重复扫描，复杂度可能退化。

修法：扩展前先判断 `num - 1 not in num_set`。

用列表做存在性判断

错因：写成 `while cur + 1 in nums`。

后果：每次 `in` 都可能扫列表，性能不稳。

修法：统一查 `num_set`。

忘记空数组

错因：默认至少有一个数。

后果：如果 `best` 初始化或循环逻辑不稳，空数组可能返回错。

修法：`best = 0`，自然兼容空数组。

49 直接按原字符串分组

错因：没有把“同类”转成统一身份。

后果：`eat` 和 `tea` 会进入不同组。

修法：排序或计数后再作为 key。

用 list 当 dict key

错因：写成 `groups[sorted(s)]` 或 `groups[count]`。

后果：Python 会报 `unhashable type: 'list'`。

修法：排序结果 join 成字符串，计数结果转成 tuple。

复杂度说得太粗

错因：只数外层字符串数量，不数每个字符串内部处理。

后果：49 的排序成本被漏掉。

修法：写成 `O(nk log k)`，并说明 `k` 是平均字符串长度。

错误示例：漏掉起点判断Python
for num in num_set:
    cur = num
    length = 1
    while cur + 1 in num_set:
        cur += 1
        length += 1
    best = max(best, length)

这段代码在小数据上可能也能得到正确答案，但它没有避免重复扫描。算法题里，正确答案和正确复杂度是两件事。
