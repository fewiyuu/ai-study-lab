## 值得记住的问题

这次没有明显概念性错选，主要缺口在“把概念写出来”：短填空和开放解释题留空。客观题能判断对错，说明主链路已经建立；下一步要练的是把判断压成一句可执行的排查说明。

未作答题的修正答案：

| 题目 | 修正答案 | 为什么 |
| --- | --- | --- |
| 当前目录符号 | `.` | `.` 表示当前工作目录，`./tool` 就是显式运行当前目录下的 `tool`。 |
| 参数数量 | `2` | `cp "my file.txt" ../backup/` 中，引号把 `my file.txt` 保成一个参数；第二个参数是 `../backup/`。 |
| 空格路径错误 | 路径里有空格但没加引号，Shell 把 `/tmp/my notes` 切成 `/tmp/my` 和 `notes` 两个参数；写成 `cd "/tmp/my notes"`。 | 关键不是“加引号”四个字，而是能说出错在参数切分层。 |
| 命令来源解释 | 先用 `type python` / `command -v python` 看命令来源或命中路径，再用 `python --version` 看实际解释器版本。 | 顺序是先确认“会跑谁”，再确认“是不是想要的版本”。 |

## 核心主线

排查命令问题时按这条线走：

1. 先看 Shell 怎么切词：空格、引号和转义符决定程序收到几个参数。
2. 再看命令命中哪里：内置命令、`PATH`、显式路径是三条不同路线。
3. 再看程序执行条件：路径是否存在、权限是否允许、输入文件是否在当前工作目录下。
4. 最后看输出和退出码：报错是表面，前面哪一层断了才是诊断目标。

## 关键概念

- 终端只负责显示输入输出，不决定命令语义。
- Shell 负责切分参数、查找命令、维护当前工作目录等状态。
- 程序只看到 Shell 交给它的参数列表，不会知道你原来有没有写引号。
- 相对路径从当前工作目录出发，不是从脚本所在目录自动出发。
- `type` 和 `command -v` 用来查命令来源；`--version` 用来查已经命中的程序版本。

## 例子与操作模式

空格路径排查：

```bash
cd /tmp/my notes
cd "/tmp/my notes"
```

第一行会被 Shell 切成多个词，第二行才把目录名作为一个参数交给 `cd`。

解释器版本排查：

```bash
type python
command -v python
python --version
```

先看命中路径，再看版本。否则版本号看起来对，也可能不是你以为的那个解释器。

## 常见误区

- 只记“加引号”，但说不出 Shell 切词把路径拆坏了。
- 把 `type python`、`command -v python` 和 `python --version` 当成同一种检查。
- 看到 `command not found` 就急着改权限；权限问题通常发生在“已经找到对象”之后。
- 忘记 `.` 是当前目录，导致 `./tool` 和 `tool` 的搜索规则混在一起。

## 迁移与使用

下次遇到命令失败，先写三句判断：

1. 这行命令会被 Shell 切成几个词？
2. 命令名会命中内置命令、`PATH` 里的程序，还是显式路径？
3. 程序拿到的路径参数从哪个当前工作目录出发？

如果这三句说不清，就先运行：

```bash
pwd
type <command>
command -v <command>
ls -l <path>
```

## 修正后的解释

`type python` 和 `command -v python` 解决的是“Shell 准备运行谁”：它们暴露命令来源或命中路径。`python --version` 解决的是“这个被运行的程序是什么版本”。所以排查解释器问题时，先查来源和路径，再查版本。命中路径错了，版本号再看也容易误判。

## 复习卡片

- Q：当前目录符号是什么？  
  A：`.`。

- Q：`cp "my file.txt" ../backup/` 有几个参数？  
  A：2 个。引号里的 `my file.txt` 是一个参数。

- Q：为什么 `cd /tmp/my notes` 会出错？  
  A：Shell 按空格切词，把一个目录路径拆成多个参数；应写成 `cd "/tmp/my notes"`。

- Q：为什么先查 `type python`，再查 `python --version`？  
  A：先确认 Shell 会运行哪个 `python`，再确认那个程序的版本。
