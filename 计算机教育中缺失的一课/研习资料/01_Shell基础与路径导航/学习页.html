基础单元

Shell基础与路径导航

这一页把 Shell 命令拆成四个可检查对象：终端、Shell、当前工作目录和 PATH。读完后，你应该能先看懂一行命令怎样被切成命令名和参数，再判断相对路径从哪里出发，最后把 command not found、No such file or directory、Permission denied 分到正确一层。后面的文本处理和管道都会默认你已经会这套起点。

读完本页应该能做什么

  • 读出一行命令会被切成哪些参数，并知道空格、引号和反斜杠怎样改写切分结果。
  • 用 pwd、type、command -v、./tool 判断当前目录、PATH 和命令命中位置。
  • 把 command not found、No such file or directory、Permission denied 拆成不同的检查顺序。

1. 学习目标：先把四个判断点立住

学习目标

这一页先把 Shell 学习最容易混掉的四个对象分开：终端、Shell、当前工作目录和 PATH。你读完后，不需要背一堆名词，只要能把一行命令拆成命令名、参数、路径起点和命令查找四步，就已经够用。

  • 先判断输入里哪些词会变成参数，哪些符号只是在改 Shell 的切分方式。
  • 先判断当前目录、PATH 和显式路径分别在什么场景下起作用。
  • 先判断报错属于命令查找、路径定位还是权限问题。

2. 先抓主线：一行命令会走过什么

主线

先把最短的执行链记住：输入一行文本，Shell 切词，命令名被拿去找内置命令或 PATH，程序启动后再读取参数里的路径。后面的每一个判断，只是在这条链上找断点。

步骤 Shell 或程序在看什么 写回什么 错了会先看到什么
输入文本 整行命令 还没有写状态 先别急着猜，先看 Shell 怎么切词
参数切分 空格、引号、转义符 命令名和参数列表 参数数量不对，或者带空格的名字被拆开
命令查找 内置命令、PATH、显式路径 找到要运行的程序 command not found
程序执行 路径参数、输入文件、权限 文件读取、状态变化、输出 No such file or directory 或 Permission denied
结果回传 退出码和标准输出 屏幕上的文本 你只能看到报错表面，看不到断点在哪一层
最短路径先把链路跑顺，再看局部
python scripts/run.py data/input.txt
pwd
type python
这里先不追管道、重定向、grep、find、xargs。它们会在后面的单元里单独展开；这一页只要把“命令怎么被解释”和“文件从哪里找”立住就够了。

3. 三层分开：终端、Shell、程序

概念边界

很多命令行问题来自把窗口、解释器和程序混成一层。终端负责收发输入输出，Shell 负责读一行文本、切分参数、找命令，真正执行任务的是外部程序或内置命令。先把这三层分开，后面的路径、引号、PATH 和报错才有落点。

对象 负责什么 读什么 写什么
终端 显示输入和输出 键盘输入和屏幕内容 窗口本身不改命令语义
Shell 解释命令行、切分参数、找命令 命令文本、PATH、当前工作目录 当前 Shell 的状态，或者程序启动请求
程序 真正完成任务 命令名、参数、文件路径 标准输出、错误信息、退出码

最稳的起手式是现场确认：type cd 看 cd 是内置命令还是外部程序，type python 看 Shell 到底会命中哪个可执行文件。只要你知道“这一层是谁在做事”，后面看到报错就能先把责任归到正确的一层。

正确示例先确认 Shell 实际会找什么
type cd
type python
pwd
这一步先不追编辑器插件、容器、远程登录和图形界面；这一页只看命令如何被解释、路径如何被定位。

4. 一条命令如何被切开

切分与引号

Shell 会先按自己的语法解析一行文本：空格通常分隔词，引号可以把带空格的内容保成一个词，反斜杠可以转义特殊字符。解析后，第一个词通常被当作命令名，后面的词被当作参数。

例如 echo hello world 会变成程序名 echo 加两个参数 hello 和 world。如果写成 echo "hello world"，程序名仍是 echo，但参数只有一个：hello world。这就是为什么文件名里有空格时经常需要引号。

程序看到echo + 2 个参数

空格把 hello 和 world 切成两个参数，Shell 不会把它们自动合并。

程序看到echo + 1 个参数

引号先被 Shell 吃掉，程序收到的是一个带空格的字符串。

程序看到cd + 1 个目标路径

路径里有空格时，必须先让 Shell 保持它仍然是一个整体。

输入行 Shell 切成什么 程序真正收到什么 常见误判
echo hello world 3 个词 命令名 + 2 个参数 把空格当成程序内部格式
echo "hello world" 2 个词 命令名 + 1 个参数 把引号当成程序内容
cd "/tmp/my notes" 2 个词 命令名 + 1 个目录名 把目录名拆成两个目标
逐行拆解参数边界怎么改写命令含义
echo hello world
echo "hello world"
cd "/tmp/my notes"

程序通常不会看到你原来有没有写引号。Shell 在启动程序前已经把这些语法符号处理掉了，所以调试时要站在 Shell 的角度看输入，别站在程序的角度猜它“应该懂空格”。

5. 状态账本：哪些东西在变，哪些东西只是被读取

副作用

Shell 里最容易看漏的部分，是哪些东西真的被改了，哪些只是被查看了。pwd 只读状态，cd 会写状态，type 只读查找结果，command -v 只读命令位置。你把这些动作放进一个状态账本，排错就不会只靠直觉。

对象 读 / 写 什么时候有效 写错后的症状 下一小步
当前工作目录 被 pwd 读取，被 cd 修改 只对当前 Shell 进程有效 相对路径从错误起点出发 先确认再拼相对路径
命令来源 被 type 和 command -v 读取 运行前就能检查 你不知道自己命中了内置命令还是磁盘上的程序 先看命中位置，再决定怎么跑
PATH 被 Shell 读取，用来查外部程序 改完后对新启动的命令生效 同名程序跑到旧版本 用 type 看实际命中
执行权限 被 Shell 和内核读取 文件存在但权限位要允许执行 Permission denied 补权限或换解释器
逐行拆解先分清内置命令和外部程序
type cd
type ls
type python
command -v python

学到这里，重点是知道：凡是要改变 Shell 状态的动作，都必须在 Shell 这一层完成，不能假设“只要有一个同名程序就行”。

6. PATH：Shell 找外部程序的搜索清单

命令查找

当命令名属于外部程序时，Shell 会沿着 PATH 里的目录顺序去找可执行文件。谁先被找到，就运行谁。这个规则解释了为什么同名程序会跑偏，也解释了为什么你刚安装的版本不一定被命中。

当前目录并不自动等于 PATH。很多 Shell 为了安全，不会把当前目录默认放进搜索清单，所以目录里明明有一个可执行文件，直接输入文件名仍然可能报 command not found。这时要么显式写 ./tool，要么把那个目录放进 PATH。

检查点 Shell 做什么 你能看到什么 常见误判
命令名 先看是否是内置命令 type 会告诉你来源 把所有命令都当外部程序
PATH 按目录顺序搜索可执行文件 先命中的版本会被运行 以为安装了就一定会命中
当前目录 只有显式写 ./ 时才直接使用 目录里有程序也不一定能直接跑 以为当前目录天然在搜索路径里

排查版本问题时，先看 type python 或 command -v python，再看 python --version。如果命中的路径不符合预期，就去改 PATH 顺序、显式写完整路径，或者在当前目录里用 ./ 指定执行对象。

正确示例先查搜索清单，再决定怎么运行
echo "$PATH"
command -v python
python --version
./hello

7. 排错链：先起点，再命中，再权限

实战链路

最常见的卡点在于把当前目录、PATH 和文件名参数混在一起。假设你在项目目录里运行 python scripts/run.py data/input.txt，结果报 No such file or directory。这时先确认 Shell 到底在找什么、从哪里找、找到了能不能执行。

这个排查链可以固定成四步：先用 pwd 锁定当前工作目录，再用 ls 看脚本和数据文件是否真的在那个起点下，然后用 type python 或 command -v python 看命中的程序，最后再看权限和显式路径有没有写对。只要顺序固定，很多“看起来很玄”的错误都会变成可复查的状态问题。

观察点 读取什么 写回或影响什么 出错时的症状 下一步检查
pwd 当前工作目录 不写状态，只暴露起点 相对路径从错的地方出发 确认项目根目录是否真在这里
ls scripts data 脚本和数据目录内容 不写状态，只暴露文件是否存在 脚本名写错或目录不在起点下 修正路径或切回正确目录
type python 命令来源 不写状态，只暴露命中位置 跑到旧版本或别的解释器 改 PATH 或写完整路径
ls -l run.sh 权限位 不写状态，只暴露可执行性 Permission denied 补执行位或换解释器运行
逐行拆解把检查顺序固定下来
pwd
ls
ls scripts
ls data
type python
python --version
ls -l run.sh

这里还有一个常见误会：命令“能运行”不代表命中的就是你想要的那个版本。Shell 只负责按规则找到第一个匹配项，具体版本要你自己去核对。这个习惯会直接影响后面的文本处理、脚本运行和工具链排错。

现在就能做什么：遇到路径问题时，先写出“起点、命中、权限、版本”四个检查项，再决定要不要改命令。

8. 错误面板：把报错映射到检查点

诊断手册

Shell 学习里最省时间的习惯，是把错误看成层级信号。command not found、No such file or directory、Permission denied 指向不同问题：前者通常是命令没被找到，第二个通常是路径不存在，最后一个通常是找到了对象但没有执行权限。

bash: tool: command not found
症状 常见根因 先查什么 直接修法
command not found 命令名拼错、没安装、PATH 没命中、当前目录程序没写 ./ type 命令名、command -v 命令名 改名字、补 PATH、写显式路径
No such file or directory 当前目录不对、文件名写错、空格没加引号、大小写不匹配 pwd、ls、引号 切对目录、修正路径、补引号
Permission denied 找到了对象，但没有执行或访问权限 ls -l、文件类型、执行位 改权限或改用解释器运行
错误示例先查再改，不要直接猜
pwd
type tool
ls -l run.sh

看到报错时，先问自己两个问题：Shell 到底找到没有，以及找到的对象有没有权限做这件事。把这两个问题分开，诊断会快很多。

再看一个路径例子：cd /tmp/my notes 会先被 Shell 切成 cd、/tmp/my、notes 三个词，cd 收到的目标路径就断了。改成 cd "/tmp/my notes" 以后，Shell 才会把整个目录名保成一个参数。这里的失败后果很具体：Shell 看到的是两个路径片段，程序看到的是一个不存在的目标，最后报错的方向就变成“路径没拼对”而不是“目录真的消失了”。

9. 真实会话回放：同一个项目，四个断点

回放

把一个小项目放到桌面上看，问题会清楚很多。假设你现在在项目根目录里，目录长这样：

目录示意先看起点，再看文件树
project/
  scripts/
    run.py
  data/
    input.txt
  tool

如果你在这个目录里运行 python scripts/run.py data/input.txt，最先要问的不是“报错是什么意思”，而是“Shell 先看见了什么”。pwd 先确认起点，ls scripts data 先确认目录树，type python 先确认命中的解释器，最后才轮到程序本身去读 scripts/run.py 和 data/input.txt。这是一条很朴素的证据链：先证明文件在，才能说程序错了。

动作 它读取什么 它写回什么 如果错了，最先怀疑什么
pwd 当前工作目录 不改状态，只暴露起点 你是不是站在了 project 外面
ls scripts data 目录是否存在 不改状态，只暴露文件树 脚本或数据文件是不是根本不在这里
type python 命令来源 不改状态，只暴露命中路径 解释器是不是跑到了别的版本
python scripts/run.py data/input.txt 脚本和数据参数 程序开始读文件、写输出 相对路径、文件名或权限有没有错
ls -l run.sh 权限位 不改状态，只暴露是否可执行 Permission denied 是否是执行位缺失

这个顺序有一个好处：它把“猜测”换成了“缩小范围”。如果 ls scripts data 已经失败，那就不用再怀疑程序逻辑；如果 type python 告诉你命中的不是想要的解释器，那也不用先改脚本。你查到的不是答案本身，而是哪个假设还能继续保留。

回放脚本把证据链按顺序收集起来
pwd
ls scripts data
type python
python scripts/run.py data/input.txt
ls -l run.sh
./run.sh

看这个脚本时，要把前半段和后半段分开理解。前半段是收集证据：起点在哪、文件在不在、命中了谁。后半段是验证假设：命令能不能跑、执行位够不够、显式路径是不是已经绕开了 PATH。./run.sh 这一步尤其重要，它说明“当前目录里有文件”不等于“Shell 会主动把它当命令找”。

如果你想把这条链复述给别人，最稳的说法不是“先看报错”，而是“先确认起点，再确认目录树，再确认命中版本，再确认权限”。这四步里，pwd 负责起点，ls 负责目录树，type 负责命中，ls -l 负责权限。它们各自只回答一个问题，所以合起来才够用。

练习钩子：把你自己的一个失败命令，按这四步写成一张小表。只要能写出“动作 / 读什么 / 写什么 / 最先怀疑什么”，你就已经在做真正的排查了。

10. 路径选择表：什么时候用绝对路径，什么时候用相对路径

选择题

路径不是越短越好，关键是你想把什么固定住。绝对路径把起点固定死，适合文件位置很重要、命令会在不同目录里运行的场景；相对路径把起点交给当前工作目录，适合你已经知道自己站在项目根目录里，想让命令更短、更容易读；./tool 则是告诉 Shell “我就要当前目录里的这个文件”，它主要用来绕开 PATH。

写法 它依赖什么 适合什么场景 最常见的坑
/home/student/project/data/input.txt 只依赖文件真实位置 跨目录、跨机器、交给别人复现 路径长，手打容易错
data/input.txt 依赖当前工作目录 你已经站在项目根目录里 一旦起点变了，路径就断
../shared/report.txt 依赖当前工作目录和父目录结构 项目内部共享目录、上一级资源 多跳之后很难一眼看清起点
./tool 依赖当前目录里确实有这个可执行文件 运行当前目录下的脚本或工具 忘了执行权限，或者误以为 PATH 会帮你找

这张表不是让你机械背选项，而是让你先定一个问题：我希望命令对哪件事稳定？如果你想让“文件位置”稳定，就偏向绝对路径；如果你想让“项目根目录”稳定，就偏向相对路径；如果你想让“当前目录里的程序”稳定，就显式写 ./。一旦把选择依据说清楚，后面就不容易在路径和命令查找之间来回猜。

同一目标三种写法，适合不同约束
cat /home/student/project/data/input.txt
cat data/input.txt
./tool

如果你把这三行连着看，会发现它们其实在回答三个不同的问题。第一行绕开当前目录，第二行依赖当前目录，第三行绕开 PATH。Shell 的工作不是替你“猜最方便的写法”，而是按你写出来的规则去做。所以在命令行里，先把约束说清楚，比先把命令缩短更重要。

这里先不追别名、函数、Shell hash table、glob 展开和管道。它们会改变命令的外观，但不会改变这一页的核心判断：起点、命中、权限、版本，先分清再下手。

11. 命令名怎么落到程序上

命令分流

这一页前面已经讲过切词和路径起点，这里再补最后一层：Shell 看到一个名字以后，先判断它是不是内置命令，再按 PATH 去找外部程序；如果你自己写的是显式路径，比如 ./tool，就不用再走搜索清单。这个分流决定了同一个名字为什么会有不同结局。

拿 cd、python、./tool 三个例子看最清楚。cd 必须改 Shell 自己的当前工作目录，所以它要在 Shell 进程里执行。python 通常是外部程序，Shell 会按 PATH 找到它。./tool 则告诉 Shell：别搜 PATH，直接去当前目录里找这个文件。

输入 Shell 先看什么 最后落到哪里 最容易出错的环节
cd .. 是不是内置命令 Shell 自己的当前工作目录 把 cd 当成普通程序，结果只改了子进程
python --version PATH 里谁先命中 外部解释器程序 命中的是旧版本，版本号和预期不一致
./tool 显式路径是否存在 当前目录里的可执行文件 忘了执行权限，或者以为 PATH 会自动帮忙
分流检查先判断命令属于哪一类
type cd
type python
command -v python
./tool

这组检查的顺序很实用：先确认有没有内置命令，再看外部程序会命中谁，最后才决定要不要写显式路径。很多人一上来就改 PATH，或者一上来就补权限，结果把问题绕远了。其实只要先把“命令名字会落到哪一层”说清楚，后面的动作就很容易收敛。

如果 type python 显示的是你不想要的解释器，问题不在脚本本身，而在命中位置。如果 ./tool 报 Permission denied，问题也不在 PATH，而在执行权限。如果 cd 报错，方向更简单：看目标路径对不对，而不是怀疑当前目录有没有被写进搜索清单。这个分流表的作用，就是把同一个名字背后的三种落点拆开。

练习钩子：拿你自己的三个命令，分别判断它们属于内置命令、PATH 搜索还是显式路径。只要你能先分流，再执行，后面的报错就会好查很多。

12. 操作桥：把这页动作接到你自己的项目

下一步

现在就可以把本页的判断放进你自己的项目里。找一个有 scripts/、data/ 或工具文件的目录，先写 pwd，再写 ls scripts data，然后挑一个你真正会遇到的命令，比如 python scripts/run.py data/input.txt、tool 或 ./tool。如果它失败，就按本页的顺序把错因拆开：起点、命中、权限、版本。

你要做的动作 先看什么 为什么先看它 如果不对，下一步怎么接
运行项目脚本 pwd 和 ls scripts 先确认当前目录真的站在 project 上 切回项目根目录，再重试
运行当前目录工具 type tool 和 ./tool 先分清 PATH 命中还是显式路径 补执行权限，或者直接写完整路径
核对解释器版本 command -v python 和 python --version 先确认命中谁，再确认版本号 改 PATH 顺序，或者写固定路径
最小闭环把检查动作放进一条真实命令里
pwd
ls scripts data
type python
python scripts/run.py data/input.txt
type tool
./tool

这个闭环最重要的地方，不在于命令本身，而在于顺序。先查起点，再查命中，再查权限，最后才查版本。你把这个顺序带进自己的项目里，就会发现很多原本看起来像“程序坏了”的问题，其实只是命令落点错了。

练习钩子：把你当前项目里最常用的一条命令，改写成“先查起点，再查命中，再跑程序”的版本。写完以后，你就能直接拿它去排错。

答题方式

诊断顺序

先看 Shell 怎么切，再看当前目录在哪里，再看命中的程序是谁，最后看它有没有执行权限。pwd、type、command -v、ls -l 要能串起来。

错误诊断空格与引号

9. 诊断：cd /tmp/my notes 报错。按“错因 / 后果 / 修法”回答。

不要只写“加引号”。先说明 Shell 在什么地方把路径切坏了。

参考答案 / 评分要点

错因：路径里有空格但没有加引号，Shell 把它拆成了两个参数。后果：cd 收到错误的目标路径，找不到目录。修法：写成 cd "/tmp/my notes"，或者把目录名改成不含空格的形式。答案里要把“切词错误”说出来，而不是只写最终命令。

13. 费曼题：解释 type python、command -v python、python --version 各在查什么，以及为什么通常要先查前两个，再查最后一个。

至少写出“来源、路径、版本”三个对象，不要只抄命令名字。

参考答案 / 评分要点

type python 和 command -v python 都在查命中路径或命令来源，python --version 在查实际运行的解释器版本。顺序上要先定位“会跑谁”，再确认“它是不是你要的那个版本”。答案里最好能补一句：命中路径不对，版本再对也没用。
