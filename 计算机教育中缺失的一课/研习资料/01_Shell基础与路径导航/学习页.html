Shell 基础 · 路径导航

Shell 不是黑框魔法，而是调用程序和定位文件的语言

你在终端里输入的每一行，都会经历同一条链路：Shell 解析文本，确定程序，切分参数，查找可执行文件，再让程序在某个当前工作目录下运行。本页目标是让你能读懂这条链路，并能排查新手最常见的命令行错误。

学完你应该能做什么

  1. 区分终端、Shell、GUI、Bash/Zsh/PowerShell。
  2. 判断一条命令会被切成几个参数。
  3. 解释绝对路径、相对路径、.、..、~。
  4. 排查 command not found、No such file or directory、Permission denied。

概念地图

Shell 学习的第一步不是背命令，而是理解“谁在做什么”。终端负责显示和输入，Shell 负责解释命令，程序负责真正工作，文件系统负责保存路径和文件。路径不是文件本身，而是“从某个起点走到文件的路线”。

输入文本→Shell 解析→程序名→参数列表→PATH 或路径定位→进程运行
终端 Terminal直觉上是命令行的窗口。正式说，它提供输入、输出和显示环境，本身通常不决定命令含义。
Shell直觉上是命令解释器。它读取命令行，处理引号、变量、通配符、路径查找，然后启动程序或执行内置命令。
参数 Argument程序收到的材料。echo hello 里 hello 不是 Shell 自己要打印的东西，而是传给 echo 的参数。
当前工作目录 CWD相对路径的默认起点。你在哪个目录运行命令，会直接影响 ./data、notes/a.md 指向哪里。

1. Shell、终端和 GUI：三者不是一回事

很多人第一次打开命令行，会把黑色窗口叫做 Shell。更准确地说，黑色窗口通常是终端模拟器，里面运行的命令解释器才是 Shell。你在窗口里输入 ls、cd、python，终端只是把键盘输入交给 Shell，再把输出显示出来。

GUI 是图形界面，用鼠标点菜单、按钮和文件夹；Shell 是文本界面，用命令表达动作。两者不是高低关系，而是交互方式不同。GUI 适合探索和可视化，Shell 适合重复、精确、组合和自动化。比如“把某目录下所有 .log 文件找出来并统计行数”，Shell 可以组合多个小工具完成。

Bash、Zsh、Fish、PowerShell 都是 Shell，但语法和内置命令不完全相同。本节先以 Unix Shell 的基本模型为主，也就是 macOS/Linux 常见的 Bash/Zsh 语境。Windows 上的 PowerShell 更偏对象管道，Git Bash/WSL 更接近 Unix Shell。学习时要分清：命令思想相通，具体语法不总是通用。

一个实用判断是：终端问题通常表现为窗口、字体、复制粘贴、颜色或输入法异常；Shell 问题通常表现为引号、变量、路径展开、命令查找异常；外部程序问题通常表现为程序已经启动，但提示参数不对、文件格式不对或版本不对。把问题先放到正确层级，后面的排查会省很多时间。

你现在应该能做什么：看到“终端打不开”“Shell 报错”“PowerShell 跟 Bash 不一样”这类说法时，能先判断问题发生在显示窗口、命令解释器，还是具体程序。

2. 一条命令如何变成一次程序调用

Shell 不是逐字把整行文本交给程序。它先按自己的语法解析：空格通常分隔词，引号可以把带空格的内容保成一个词，反斜杠可以转义特殊字符。解析后，第一个词通常被当作命令名，后面的词被当作参数。

例如 echo hello world 会变成程序名 echo 加两个参数 hello 和 world。如果写成 echo "hello world"，程序名仍是 echo，但参数只有一个：hello world。这就是为什么文件名里有空格时经常需要引号。

正确示例命令被切分成程序和参数
echo hello world
echo "hello world"
mkdir "weekly notes"
cd "weekly notes"

命令执行前还有一步：Shell 要判断命令名是什么。如果命令名是 Shell 内置命令，比如 cd，Shell 自己执行；如果是外部程序，比如 python、git，Shell 要在路径中找到可执行文件再启动它。这个差别会影响后面的 type、which 和 PATH 排查。

还要注意，Shell 解析发生在程序启动之前。程序通常不知道你原来有没有写引号，它只收到解析后的参数数组。也就是说，"hello world" 里的外层引号不是传给程序的内容，而是 Shell 用来保护空格的语法。这个细节能解释为什么很多程序文档会说“把路径作为一个参数传入”，而不是说“把带引号的路径传入”。

你现在应该能做什么：拿到一条命令，先把它拆成“命令名”和“参数列表”，再判断错误可能来自解析、查找，还是程序自身。

3. 参数、空格、引号：命令行最早踩坑的地方

Shell 里空格不是装饰，它通常是参数边界。cp my file.txt backup/ 在 Shell 看来不是“复制 my file.txt”，而是把 my 和 file.txt 当成两个不同参数。若真实文件名叫 my file.txt，就必须写 cp "my file.txt" backup/ 或 cp my\ file.txt backup/。

单引号和双引号都能保护空格，但它们的细节不同。单引号更“硬”，里面大多数特殊含义都会失效；双引号会保留某些展开能力，例如变量展开。初学阶段先掌握一条实用规则：文件名、路径、搜索词里有空格，就给它加引号。

错误示例路径里有空格但没有加引号
cd D:/Users/yyh/Downloads/my notes
# Shell 看到的可能是：
# cd
# D:/Users/yyh/Downloads/my
# notes
正确示例用引号保护一个完整路径
cd "D:/Users/yyh/Downloads/my notes"

如果路径中有中文或括号，很多时候不是字符本身有问题，而是空格、括号、反斜杠或当前 Shell 的转义规则让参数边界变了。初学时不需要把所有转义细节一次背完；更稳的做法是：先给整段路径加引号，再用 pwd 和 ls 验证当前位置和目标是否存在。

你现在应该能做什么：解释“为什么同一个路径在资源管理器里能打开，在命令行里却报错”，并能用引号或转义修正。

4. 路径与当前目录：相对路径一定有一个起点

路径是定位文件或目录的路线。绝对路径从文件系统的根或盘符开始，例如 Unix 的 /home/yyh/project，Windows 的 C:\Users\yyh 或在部分工具中写成 C:/Users/yyh。相对路径不从根开始，它从当前工作目录开始。

. 表示当前目录，.. 表示上一级目录，~ 常表示当前用户的家目录。./script.sh 的意思不是“随便找 script.sh”，而是“从当前目录找 script.sh”。这能避免 Shell 去 PATH 里找同名程序。

逐行拆解工作目录如何影响相对路径
pwd                 # 显示当前工作目录
ls                  # 列出当前目录内容
cd notes            # 进入当前目录下的 notes
cd ..               # 回到上一级目录
ls ./notes          # 明确列出当前目录下的 notes
ls /tmp             # 列出绝对路径 /tmp

Windows/Unix 路径差异也常让人误判。Unix 用 / 表示根和分隔符；Windows 有盘符和反斜杠。很多跨平台工具接受正斜杠，但并非所有 Shell 都把路径规则解释得一样。遇到路径问题时，先确认你在 Bash、PowerShell、CMD、WSL 还是某个程序自己的路径语法中。

资源管理器里的“当前位置”和终端里的当前工作目录也可能不同。你双击打开项目文件夹，不代表终端已经进入该文件夹；编辑器里运行脚本时，运行目录还可能由编辑器配置决定。只要相对路径起点不同，同一段 data/input.txt 就会指向不同位置。

环境路径常见写法先检查什么
macOS / Linux / Bash / Zsh/home/name/project、~/projectpwd、ls、type
Windows PowerShellC:\Users\name\project 或 C:/Users/name/projectpwd、Get-Command、路径是否需要引号
Git Bash/c/Users/name/project 或 C:/Users/name/projectpwd 显示的真实起点
WSL/home/name、/mnt/c/Users/name文件是在 Linux 文件系统里，还是 Windows 挂载盘里

你现在应该能做什么：在报错时先运行 pwd 和 ls，判断相对路径的起点是不是自己以为的目录。

5. 内置命令与外部程序：为什么 cd 必须特殊

有些命令由 Shell 自己执行，叫内置命令，例如 cd。原因很直接：当前工作目录是 Shell 进程自己的状态。如果 cd 是一个普通外部程序，它只能改变自己的目录，退出后 Shell 仍在原地，所以无法达到“改变当前终端所在目录”的效果。

外部程序则是磁盘上的可执行文件。比如 git、python、node 通常需要被 Shell 找到后启动。用 type 可以查看某个名字到底是内置命令、别名、函数还是外部程序；用 which 常用于查看外部程序路径，但它不一定能告诉你内置命令和别名的完整真相。

正确示例区分 type 与 which
type cd
type python
which python
which git

type 更像“问 Shell 你会怎样理解这个名字”。它能告诉你一个名字是内置命令、别名、函数，还是外部程序。which 更像“沿着 PATH 找外部可执行文件”。所以排查别名、函数、内置命令时，优先看 type；排查外部程序路径时，再结合 which 和程序自己的 --version。

你现在应该能做什么：当“明明装了程序却运行的不是那个版本”时，不只问装没装，而是先用 type 或 which 看 Shell 实际选择了谁。

6. PATH：Shell 找外部程序的搜索清单

当命令名不是内置命令，也没有写成 ./tool 或 /usr/bin/tool 这种路径形式，Shell 会查 PATH。PATH 是一串目录列表，Shell 按顺序到这些目录里找同名可执行文件，先找到谁就运行谁。

这解释了两个常见现象。第一，安装了程序但提示 command not found，可能是程序目录不在 PATH。第二，同一台机器有多个 Python 或 Node，运行的版本取决于 PATH 顺序，而不是你脑子里“最新安装的那个”。

逐行拆解查看 PATH 与程序位置
echo "$PATH"        # Unix Shell: 查看搜索目录列表
type python         # 看 python 这个名字如何被解析
which python        # 看外部程序路径
python --version    # 看实际启动版本

当前目录通常不自动在 PATH 里，所以下载了 hello 放在当前目录，直接输入 hello 可能找不到；写 ./hello 才表示“运行当前目录下这个文件”。这也是安全设计：避免你在某个陌生目录里误运行恶意同名程序。

还有一种常见误会是“我已经安装了，所以 Shell 必须找到”。安装只是把文件放到某个位置；Shell 能不能找到，还取决于那个位置是否在 PATH 中，以及排在同名旧版本之前还是之后。虚拟环境、Conda、Node 版本管理器都会通过改写 PATH 来改变命中的程序。

你现在应该能做什么：面对 command not found 或版本混乱，按“是否内置 → 是否写了路径 → PATH 是否包含目录 → PATH 顺序是否正确”的顺序排查。

7. 真实错误场景：读懂报错比背命令更重要

command not found

Shell 没找到这个命令名。错因可能是拼写错、没安装、安装了但目录不在 PATH、当前目录程序没写 ./。修法是先 type 命令名，再确认安装位置和 PATH。

No such file or directory

路径解析后没有对应文件或目录。错因常见于当前工作目录不对、路径有空格没加引号、Windows/Unix 路径写法混用、文件名大小写不匹配。

Permission denied

找到了文件，但没有执行或访问权限。对脚本而言，可能需要执行权限，也可能应该用解释器运行，例如 bash script.sh。权限细节后续再学，但这里要先区分“找不到”和“没权限”。

运行了错误版本

命令能运行但行为不对，常见于 PATH 顺序。先用 type、which、--version 查实际命中的程序，不要只看安装器提示。

逐行拆解一个稳定的排查顺序
pwd                 # 1. 先确认相对路径的起点
ls                  # 2. 看当前目录真实有哪些文件
ls "target path"    # 3. 验证目标路径是否能被当前 Shell 解析
type python         # 4. 看 Shell 如何理解命令名
which python        # 5. 看 PATH 命中的外部程序位置
python --version    # 6. 让程序自己报告版本
错误示例当前目录脚本存在，但没有执行权限
./run.sh
# Permission denied

ls -l run.sh
# 如果没有 x 执行位，Shell 找到了文件，但系统拒绝执行
正确示例确认风险后再补执行权限
chmod +x run.sh
./run.sh

# 或者用解释器运行，不依赖脚本自身执行位
bash run.sh

一条好用的排查链路是：先确认当前目录 pwd，再确认文件是否存在 ls，然后确认 Shell 如何理解命令名 type，最后确认程序版本和参数。不要一上来重装软件；很多命令行问题只是路径起点、引号或 PATH 顺序错了。

你现在应该能做什么：看到报错时不慌，能把它归类成“命令名找不到、路径找不到、权限不够、版本命错”四种之一。

8. 完整知识链路

把本页串起来：终端把输入交给 Shell；Shell 按空格、引号、转义等规则解析命令行；第一个词通常决定要执行什么，后面的词成为参数；如果命令是内置命令，Shell 直接改自己的状态或输出结果；如果是外部程序，Shell 通过显式路径或 PATH 查找可执行文件；程序启动后，以当前工作目录为相对路径起点读写文件。

因此，命令行新手的大多数错误不是“不会背命令”，而是没有问清四个问题：我现在在哪个目录？这行命令被切成了几个参数？Shell 找到的是哪个程序？路径是从哪里开始解析的？只要这四个问题能回答，Shell 就从黑框变成了可推理的工具。

9. 真实场景与排查手册

真实命令行排查不要从重装软件开始。先确认当前目录，再确认路径是否存在，再确认命令名如何解析，最后看程序本身的报错。这个顺序能把大多数问题限定在很小范围内。

新手容易把“我看见文件了”和“Shell 能按这条路径找到文件”混为一谈。资源管理器里的当前位置、终端里的当前工作目录、脚本运行时的工作目录可以不同；只要相对路径起点变了，同一串字符就会指向别处。

另一个高频问题是命令切分。空格会把参数切开，引号会把一段文字保护成一个参数。遇到路径里有空格、括号、中文或特殊符号时，不要靠猜，先把它看成参数边界问题。

命令名找不到

先用 type 或 which 看 Shell 是否能定位命令。若当前目录有可执行文件，通常要写 ./name，而不是只写 name。

路径找不到

先运行 pwd，再 ls 父目录。很多 No such file or directory 不是文件消失，而是当前目录、引号或大小写错了。

版本不对

能运行不代表运行了想要的版本。用 type、which 和 --version 连起来查，重点看 PATH 顺序。

稳定修法

把排查写成链路：当前位置 -> 参数切分 -> 命令解析 -> 程序输出。每一步只回答一个问题。
