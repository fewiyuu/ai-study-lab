文本工具 · 查看、搜索、统计

把文本当作数据流：先看清，再截取、搜索、统计、整理

命令行文本处理不是背一串工具名，而是学会判断“现在有什么输入、要得到什么输出、哪一步最小”。本页训练你读手册、查看文件、截取头尾、统计规模、搜索模式、排序去重，并能解释每一步的输入输出和常见失败原因。

学完你应该能做什么

  1. 用 man、--help 查参数含义。
  2. 选择 cat、less、head、tail 查看文本。
  3. 用 grep、wc、sort、uniq 做基础文本处理。
  4. 诊断文件不存在、二进制文件、大小写、排序前去重无效等问题。

概念地图

文本处理的核心链路是：先弄清程序接口，再决定输入来源，最后读懂输出。很多命令默认从文件读；没有文件参数时，可能从标准输入读。虽然管道在下一块系统学习，但这一块已经要建立“程序读入文本、输出文本”的直觉。

这一块先把工具分成四类：查说明的工具、查看文本的工具、筛选文本的工具、整理文本的工具。查说明用 man 和 --help；查看用 cat、less、head、tail；筛选用 grep；整理和统计用 wc、sort、uniq。你不需要一次背完所有参数，但要知道遇到问题时先选哪一类工具。

查手册→选择工具→给文件或输入→得到文本输出→检查错误
man离线手册页，适合查完整语法、选项和示例。它回答“这个程序支持什么”，不是替你判断业务上该用什么。
pager分页查看器，如 less，适合大文件。它让你在文件中移动、搜索、退出，而不是把全部内容一次性刷到屏幕。
pattern搜索模式，grep 用它匹配文本行。模式可能是普通字符串，也可能被当成正则表达式，写错时会出现搜不到或搜太多。
line-oriented按行处理，很多 Unix 工具把一行当作基本单位。日志、CSV、配置文件常能这样处理，压缩包、图片、数据库文件通常不适合直接当文本处理。

判断句：文本工具的基本问题不是“哪个命令更高级”，而是“我的输入是不是文本、我要看全量还是抽样、我要筛选行还是统计数量、输出是否还能交给下一步”。

1. 读手册：别把参数当咒语背

man 是命令行学习的入口。遇到陌生命令，不要先去记忆所有参数，而是读它的用法结构：命令名、选项、参数、描述、示例。方括号通常表示可选内容，省略号表示可重复，短选项如 -n，长选项如 --line-number。手册页刚开始会有点密，但你只需要先抓住 NAME、SYNOPSIS、DESCRIPTION、OPTIONS、EXAMPLES 这几块。

--help 通常更短，适合快速回忆；man 更完整，适合理解边界。不同系统的工具版本可能不同，GNU、BSD、BusyBox 的参数不完全一致。资料里看到一个选项，自己机器上不能用，不一定是你错了，可能是版本差异。

读手册时不要从头背到尾。先看命令的基本格式，再找到当前任务需要的参数。例如你想让 grep 显示行号，就在手册里搜 line number 或 -n；你想让 sort 按数字排序，就找 numeric。这样学到的参数有场景，不会变成孤立记忆。

正确示例先查接口再使用
man grep
grep --help
man sort
sort --help

你现在应该能做什么：看到陌生参数时，能先查本机手册，判断它是选项、文件名，还是搜索模式。

2. 查看文本：cat、less、head、tail 分工不同

cat 适合把短文件直接输出到屏幕，也常用于把文本交给后续工具。它不适合打开很大的日志，因为输出会刷屏，终端滚动缓冲区也可能被大量内容挤掉。判断一个文件能不能直接 cat，先看它是否足够短、是否真的是文本、输出后是否方便阅读。

less 是分页查看器，可以搜索、翻页和退出，适合读长文件。打开 less 后，通常用方向键或空格翻页，用 /关键词 搜索，用 q 退出。它不会像 cat 一样把所有内容一次性倒到屏幕上，所以更适合日志、长配置和长文本。

head 默认看前 10 行，适合快速判断文件格式；tail 默认看后 10 行，适合看日志最新部分。tail -f 会持续跟随文件增长，用来观察服务日志很常见。处理陌生文件时，一个稳妥顺序是：先 head 看表头和格式，再 wc -l 看规模，必要时用 less 搜索上下文。

逐行拆解选择合适的查看方式
cat config.txt       # 短文件直接看
less app.log         # 长日志分页看
head -n 5 data.csv   # 看前 5 行，判断表头
tail -n 20 app.log   # 看最后 20 行
tail -f app.log      # 持续观察新增日志

你现在应该能做什么：根据文件大小和目的，选择“直接输出、分页查看、看头部、看尾部”。

3. 截取与统计：先确认规模和结构

wc 用来统计行数、词数、字节数。最常用的是 wc -l 统计行数。它不会理解 CSV、JSON 的语义，只是按文本规则计数，所以“行数”不等于“记录数”的前提是文件确实一行一条记录。如果 JSON 被格式化成多行，wc -l 统计的是文本行，不是对象数量。

head 与 tail 不只是查看，也是在处理文本时取样。处理一个陌生数据文件前，先 head 看格式，再 wc -l 看规模，能避免把大文件直接打印到终端。日志排查时也常先 tail 看最近错误，再决定是否用 grep 扩大搜索范围。

统计命令的边界要说清楚。wc -w 的“词”按空白分隔，不等于自然语言里的词，也不等于 CSV 字段。wc -c 统计字节，不等于字符数；中文、emoji、多字节编码会让字节数和肉眼看到的字符数不同。初学时先把 wc 当作规模估计工具，而不是语义分析工具。

正确示例处理前先取样和统计
head -n 3 access.log
wc -l access.log
tail -n 50 access.log

你现在应该能做什么：面对未知文本文件，先用轻量命令判断格式、规模和最新内容。

4. grep：按模式筛出含有目标的行

grep 的基本模型是“给一个模式，输出匹配的行”。它不是在文本中只输出匹配的词，而是输出整行。常用参数包括 -n 显示行号，-i 忽略大小写，-r 递归搜索目录。你看到一行输出时，要知道它表示“这一行包含匹配内容”，不是“只有这一小段文本被提取出来”。

初学时要注意 Shell 引号和 grep 模式的边界。搜索词里有空格或特殊字符时最好加引号。正则表达式会让某些字符具有特殊含义；如果只是想按普通字符串搜，很多系统可用 grep -F。例如点号 . 在正则里通常表示任意字符，如果你想搜字面量点号，直接写可能会匹配过多。

grep 没结果不一定代表文件里没有目标。可能是大小写不一致，需要 -i；可能是你搜错文件或目录，需要检查路径；可能是模式被 Shell 提前切开，需要加引号；也可能是文件不是普通文本，出现 Binary file matches 这类提示。排查时先确认输入文件，再确认模式，再确认参数。

正确示例搜索日志和显示行号
grep "ERROR" app.log
grep -n "timeout" app.log
grep -i "warning" app.log
grep -r "TODO" src/
错误示例搜索词有空格但未加引号
grep user login app.log
# grep 可能把 user 当模式，把 login 和 app.log 当文件名

你现在应该能做什么：解释 grep 的输入、模式和输出，并能用行号、忽略大小写、递归搜索解决真实查找问题。

5. sort 与 uniq：去重前通常要排序

sort 把行按规则排序，uniq 只合并相邻重复行。这一点非常关键：如果重复行不相邻，uniq 不会全局去重。所以常见组合是先 sort 再 uniq。这里的“去重”不是数据库意义上的全局唯一算法，而是基于文本行相邻关系的简单处理。

uniq -c 会统计相邻重复次数，配合排序能得到频次。数字排序要用 sort -n，否则默认按字典序，100 可能排在 20 前面或后面并不符合数值直觉。要按次数从大到小看结果，常见写法是再接一次 sort -nr。

排序也有边界。默认排序受字符、区域设置和整行内容影响；如果你只想按某一列排序，后续要学习字段切分工具。此刻先掌握最稳的判断：只要你想用 uniq 得到全局去重或频次，必须先让相同内容相邻，最常见办法就是 sort。

逐行拆解排序、去重和计数
sort names.txt
sort names.txt | uniq
sort names.txt | uniq -c
sort -n scores.txt

你现在应该能做什么：判断什么时候 uniq 失效，并说明为什么排序是去重统计前的常见准备。

6. 常见错误与修法

No such file

文件名写错、当前目录不对、路径有空格未加引号。先 pwd、ls，再检查引号。不要看到文件在资源管理器里存在，就默认当前 Shell 也在同一目录。

Binary file matches

grep 发现像二进制的数据，不适合按普通文本显示。先确认文件类型，不要把图片、压缩包、数据库文件当普通文本搜。强行输出可能让终端出现乱码。

大小写没搜到

默认区分大小写。需要忽略时用 grep -i。如果仍然没有结果，再检查搜索范围、文件名、模式是否被引号保护。

uniq 去重不完整

重复行不相邻。先 sort 再 uniq。如果是按字段去重，还要先把字段抽出来，否则整行不同就不会被合并。

错误示例直接把二进制或巨大文件当普通文本处理
cat archive.zip
grep "ERROR" image.png
cat huge.log
正确示例先判断文件类型和规模
file archive.zip
head -n 5 app.log
wc -l app.log
less app.log

你现在应该能做什么：把文本处理报错分成“文件路径问题、文本类型问题、匹配规则问题、处理顺序问题”，并能写出错因、后果、修法。

7. 完整知识链路

这块的主线是：先用 man 或 --help 确认程序接口；再用 cat、less、head、tail 看清文件；接着用 wc 估计规模，用 grep 筛选行，用 sort 和 uniq 整理重复。每个工具都只做小动作，但它们的输入输出都是文本，所以能逐步组合。

处理真实日志时可以这样想：先不要急着写复杂命令，先用 head 看格式，用 tail 看最近内容，用 wc -l 估计规模；确定目标词后，用 grep -n 找到行号，用 grep -i 处理大小写，用引号保护包含空格的短语；如果要统计频次，先让相同内容相邻，再用 uniq -c。这个顺序让每一步都可检查。

这一页的迁移能力在于“解释输入和输出”。你以后学习管道时，每个竖线左边的输出都会成为右边的输入。如果现在已经能说清 grep 输出的是匹配行、sort 输出的是排序后的行、uniq -c 输出的是计数加内容，那么下一节的管道就不会只是符号拼接。

9. 真实场景与排查手册

常用程序不是一堆孤立命令。cat、head、tail、grep、sort、uniq、wc 的共同点是处理文本流：从文件或标准输入读入内容，按规则变换，再写到标准输出。

学习时要区分“查看内容”和“加工内容”。cat 适合小文件或管道输入，less 适合浏览大文件，grep 适合按模式筛选，sort 和 uniq 常常成对使用，因为 uniq 只合并相邻重复行。

真实错误通常来自输入规模、编码、排序前提和匹配模式。比如 grep 没结果不一定是文件没有内容，也可能是大小写、正则特殊字符或搜索范围错了。

cat 滥用

大文件不要直接 cat 到屏幕。先用 head、tail 或 less，避免输出刷屏也保留上下文。

uniq 前没排序

uniq 只处理相邻重复行。统计唯一值前常见链路是 sort | uniq -c | sort -nr。

grep 模式误解

点号、星号等在正则里有特殊含义。搜索字面量时可以考虑固定字符串模式或正确转义。

稳定链路

先缩小输入，再筛选，再排序统计，最后只输出需要看的列或数量。
