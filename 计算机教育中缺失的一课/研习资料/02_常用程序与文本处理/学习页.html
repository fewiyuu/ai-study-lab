常用程序与文本处理

这一页把 man 、 less 、 head 、 tail 、 grep 、 wc 、 sort 、 uniq 放进同一条文本流里学。先用小文件看清输入，再用程序筛行、统计、排序和去重；暂时不把管道、重定向、 find 、 xargs 、 sed 、 awk 全拉进来。你要先能解释单个程序读什么、写什么、哪里会报错，下一页组合命令才不会变成硬背。

1. 学习目标：把文本工具看成可检查的小程序

学习目标

上一页解决的是 Shell 怎样解析命令、从哪里找路径。这一页继续向前走：程序启动以后，它会读取文本、处理文本，再把结果写到屏幕上。先把这件事看清，后面学习管道时，你才知道每一步交出去的到底是什么。

贯穿样本app.log 五行文本复制

这里先不追正则表达式深水区，也不追管道长链路。先把单个命令的输入输出站稳：文件进去，行、计数或排序后的文本出来。

2. 主线：文件或标准输入进去，标准输出出来

文本流

很多 Unix 文本程序都遵守一个朴素约定：有文件名参数时读文件；没有文件名时，常常从标准输入读；结果写到标准输出，也就是你在终端里看到的文本。它们通常不会改原文件。这个约定让小程序能组合，但这一页先只看每个小程序自己的行为。

3. 查手册：先看本机支持什么，再写命令

接口阅读

man 和 --help 用来确认本机接口。先看命令的 SYNOPSIS、DESCRIPTION、OPTIONS，再找当前任务需要的参数。不同系统上的工具版本可能不一样，GNU、BSD、BusyBox 或 macOS 的选项细节会有差异，所以资料里见过一个参数，不代表你的机器一定支持。

正确示例用任务反查参数复制

你要做什么先搜的词常见参数别急着做什么
显示匹配行号line numbergrep -n把 -n 猜成“数字排序”
忽略大小写ignore casegrep -i改原文件大小写
按数字排序numericsort -n把默认字典序当数值序
持续看日志追加followtail -f用 cat 反复刷屏

4. 查看文本：先抽样，再深入读

查看工具

拿到陌生文件，不要直接把大文件全部打印出来。先用 head 看开头，确认格式和表头；用 tail 看结尾，确认最新日志或文件尾部；文件较长时用 less 分页读，用 /关键词 搜索，用 q 退出。cat 更适合短文件，或者只是把一小段文本送到屏幕上。

代码
head -n 2 app.log # 先看格式
tail -n 2 app.log # 再看最新或文件尾部
less app.log # 文件较长时分页读
cat app.log # 只适合短文件或有意输出全量

5. grep：输出匹配的整行

筛选行

grep 的基本动作是：读一行，判断这行是否匹配模式，匹配就输出整行。它默认保留整条日志、整行配置或整条记录，后续才能继续判断上下文。

代码
grep -n "ERROR" app.log
grep -i "timeout" app.log
grep -v "INFO" app.log
grep -C 1 "timeout" app.log

6. wc：先估规模，别把文本行数当业务含义

统计规模

wc 常用于快速估计文件规模。wc -l 数行，wc -w 数以空白分隔的词，wc -c 数字节。它不理解 CSV、JSON、日志语义，只按文本规则计数。五行日志用 wc -l 得到 5；格式化成多行的 JSON 用 wc -l 得到文本行数，不能代表对象数量。

代码
wc -l app.log
wc -w app.log
wc -c app.log

7. sort 与 uniq：先让相同的行相邻

排序去重

sort 对行排序，uniq 只处理相邻重复行。这个组合很容易被误用：如果相同名字散在文件不同位置，直接 uniq names.txt 只能去掉挨在一起的重复，不能全局去重。先 sort，重复行才会靠到一起。

代码
alice
bob
alice
carol
bob
bob

正确示例先排序再压相邻重复复制

代码
sort names.txt
sort names.txt | uniq
sort names.txt | uniq -c

8. 数据流检查：同一份文本被不同工具改写成什么

可视化

下面这个面板把 app.log 看成一个文本对象。点击不同命令，观察输出形状怎么变：有的保留原始行，有的只保留匹配行，有的把文本压成一个计数，有的要求先改变行的顺序。

9. 读输出：不要只看命令，要能说出结果长什么样

输出阅读

文本处理真正有用的地方，不在于你记住了命令名，而在于你能看懂输出。比如 head 输出的是原文件的开头几行，grep -n 输出的是“行号:原始行”，wc -l 输出的是“行数 文件名”，uniq -c 输出的是“计数 行内容”。这些输出形状不同，后面能不能继续处理，也不同。

代码
$ head -n 2 app.log
2026-06-21 09:00 INFO user alice login
2026-06-21 09:02 ERROR user bob timeout

$ grep -n "ERROR" app.log
2:2026-06-21 09:02 ERROR user bob timeout
4:2026-06-21 09:08 ERROR user alice timeout

$ wc -l app.log
5 app.log

$ sort names.txt | uniq -c
 2 alice
 3 bob
 1 carol

举个具体错误：你用 grep -n "ERROR" app.log 得到 2:2026-06-21 ...，再把这一行当成普通日志字段读，第一列就变成了 2:2026-06-21。日志格式没有变，输出里只是多了定位信息。需要继续按字段处理时，先去掉 -n，或者明确后续步骤知道前面多了行号。

练习钩子：看到任何一段命令输出，先用一句话标注它的形状：原始行、带行号的行、计数、排序结果、聚合结果。标不出来，就不要急着把它交给下一步。

10. 故障诊断：先识别假设错在哪里

故障面板

文本工具出错时，先找错掉的假设：你以为输入是文本，其实是压缩包；你以为搜索的是短语，其实 Shell 把它拆成了两个参数；你以为 uniq 会全局去重，其实它只看相邻重复；你以为数字按大小排，其实默认按字典序排。

11. 文本边界：先确认文件类型

输入类型

文本工具最擅长处理一行一行的可读文本。日志、简单配置、CSV 的粗略抽样、以换行分隔的名单，都适合先用这些工具观察。压缩包、图片、数据库文件、二进制缓存、复杂 JSON 或 Excel 文件，就不能只靠 grep、wc、sort 直接判断含义。

输入对象能否先用文本工具可做的小检查更合适的下一步
app.log可以head、tail、grep、wc -l按行查看、筛选、统计
data.csv可以粗看head -n 5 看分隔符和表头正式清洗时用 CSV-aware 工具，避免逗号和换行陷阱
archive.zip不适合直接读file archive.zip 或查看来源先解压，再处理里面的文本文件
data.json可以抽看，不宜只靠行数head 看是否一行一个对象复杂结构交给 JSON 工具，别把换行当对象数
代码
file app.log
file archive.zip
head -n 5 data.csv
wc -l data.json

这里的关键动作是“先看输入类型”。遇到二进制、压缩包或结构化格式时，先换工具或先转换；遇到普通文本，再决定查看、筛选、统计还是排序。能守住这个边界，你就不会把一条看似正常的文本命令当成可靠结论。实操时，把文件类型、抽样输出、下一步工具写成三行，排错时会清楚很多。

练习钩子：看到

Binary file matches

时，先写一句“我现在没有可读行”，再写下一步：确认文件类型、解压或换格式工具。

12. 接到下一页：先会看单步输出，再组合命令

课程桥接

这一页结束时，你应该能单独读懂每个文本工具的输入输出。下一页会把这些输出接起来：一个命令的标准输出变成另一个命令的标准输入，或者被重定向写入文件。组合之前，先养成单步检查的习惯：每加一个命令，就先核对它接到的文本是否符合预期。

先别急着把命令写长。能解释每个中间结果，才说明你真的会组合命令。

13. 完整走一遍：从陌生日志到可复查结论

实操链

现在把前面的工具放进一个真实顺序里。假设你接到一句话：“服务刚才有 timeout，帮我看一下日志。”不要马上写一条很长的命令。先把任务拆成三件事：文件是否就是目标日志，最近内容里有没有相关信号，匹配行能不能给出足够上下文。

实操顺序先确认，再筛选，再补上下文复制

步骤你在验证什么正常输出意味着什么异常时先改哪里
head -n 3 app.log文件格式、字段顺序、是否真是日志你知道时间、级别、用户、事件大概在哪些位置路径、文件名、是否文本
wc -l app.log文件规模你知道直接翻页还是先抽样不要把行数当业务事件数，先看格式
tail -n 20 app.log最近内容你知道新错误是否还在发生检查日志是否滚动、文件是否是旧文件
grep -n "timeout" app.log全文件是否有目标词你得到匹配整行和行号检查大小写、短语引号、模式是否过窄
grep -C 1 "timeout" app.log目标词前后的上下文你能看到错误前后发生了什么调大上下文，或改用 less 定位行号附近

如果 grep -n "timeout" app.log 没有输出，不要立刻说“没有 timeout”。先想几个替代假设：大小写可能是 Timeout，日志可能写成 timed out，你可能看错文件，或者时间窗口不在这份日志里。下一步可以试 grep -i "timeout" app.log，也可以回到 less app.log 里搜索相关词。

12错误诊断

Q: 12. 诊断：grep "ERROR" archive.zip 出现 Binary file matches。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

。错因： archive.zip 是压缩包或二进制内容， grep 只能判断里面似乎有匹配。后果：你看不到可读的文本上下文，也不能把它当日志行处理。修法：先解压或用格式工具读取，再对解出的文本文件运行 grep ；也可以先用 file archive.zip 确认类型。

13场景

Q: 13. 场景：你拿到陌生 data.csv，想快速判断格式和行数，会怎么做？

参考答案 / 评分要点

。一个稳妥答案是：先 head -n 5 data.csv 看表头、分隔符和前几行结构，再 wc -l data.csv 估算文本行数。答案里要说明 wc -l 是文本行数，不保证就是业务记录数；如果文件很长，再用 less data.csv 搜索或抽看。

14综合

Q: 14. 费曼题：向零基础同学解释如何用命令行查看、搜索、统计一个文本日志。

参考答案 / 评分要点

。答案要能讲出一条流程：先用 head 看格式、 tail 看最新内容，文件很长就用 less 分页搜索；要找错误行，用 grep -n "ERROR" app.log 输出匹配整行和行号；要估规模，用 wc -l app.log 看文本行数。还要提醒：短语要加引号，二进制文件不能直接当文本处理。
