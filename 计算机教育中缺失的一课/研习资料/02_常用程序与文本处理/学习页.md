# 1. 学习目标：把文本工具看成可检查的小程序 {#goals}
<!-- tag: 学习目标 -->

上一页解决的是 Shell 怎样解析命令、从哪里找路径。这一页继续向前走：程序启动以后，它会读取文本、处理文本，再把结果写到屏幕上。先把这件事看清，后面学习管道时，你才知道每一步交出去的到底是什么。

本页用同一份小日志反复练习。每个工具都回答四个问题：它读什么，输出什么，会不会改原文件，错的时候先查哪一步。

贯穿样本`app.log` 五行文本复制

```text
2026-06-21 09:00 INFO user alice login
2026-06-21 09:02 ERROR user bob timeout
2026-06-21 09:05 INFO user alice logout
2026-06-21 09:08 ERROR user alice timeout
2026-06-21 09:12 WARN disk space low
```

> [!NOTE]
> 这里先不追正则表达式深水区，也不追管道长链路。先把单个命令的输入输出站稳：文件进去，行、计数或排序后的文本出来。

# 3. 查手册：先看本机支持什么，再写命令 {#manual}
<!-- tag: 接口阅读 -->

`man` 和 `--help` 用来确认本机接口。先看命令的 `SYNOPSIS`、`DESCRIPTION`、`OPTIONS`，再找当前任务需要的参数。不同系统上的工具版本可能不一样，GNU、BSD、BusyBox 或 macOS 的选项细节会有差异，所以资料里见过一个参数，不代表你的机器一定支持。

读手册时不要从头背到尾。比如你想让 `grep` 显示行号，就在手册里找 `-n` 或 line number；你想让 `sort` 按数字排，就找 numeric；你想知道 `tail -f` 是什么，就查 follow。

man sort
sort --help
man tail

# 4. 查看文本：先抽样，再深入读 {#view}
<!-- tag: 查看工具 -->

拿到陌生文件，不要直接把大文件全部打印出来。先用 `head` 看开头，确认格式和表头；用 `tail` 看结尾，确认最新日志或文件尾部；文件较长时用 `less` 分页读，用 `/关键词` 搜索，用 `q` 退出。`cat` 更适合短文件，或者只是把一小段文本送到屏幕上。

逐行拆解面对陌生日志的稳妥顺序复制

```bash
head -n 2 app.log # 先看格式
tail -n 2 app.log # 再看最新或文件尾部
less app.log # 文件较长时分页读
cat app.log # 只适合短文件或有意输出全量
```

`less` 打开后会进入分页查看器。通常按空格向下翻页，输入 `/ERROR` 搜索，按 `n` 跳到下一个匹配，按 `q` 退出。很多初学者第一次卡住，是因为没有意识到自己已经进入另一个交互界面。

# 5. grep：输出匹配的整行 {#grep}
<!-- tag: 筛选行 -->

`grep` 的基本动作是：读一行，判断这行是否匹配模式，匹配就输出整行。它默认保留整条日志、整行配置或整条记录，后续才能继续判断上下文。

正确示例筛出错误行并显示行号复制

```bash
grep -n "ERROR" app.log
grep -i "timeout" app.log
grep -v "INFO" app.log
grep -C 1 "timeout" app.log
```

参数
作用
适合场景
常见误判


`-n`
显示行号
需要回到文件里定位
把行号当作日志本身的一部分


`-i`
忽略大小写
搜索 `error`、`ERROR` 都可能出现的日志
误以为它会改文件内容


`-v`
反选，不输出匹配行
排除噪声行
忘了它会把目标行排掉


`-C 1`
显示前后各 1 行上下文
看错误发生前后
把上下文行也当成匹配行

搜索短语时要记得引号。`grep user login app.log` 会把 `user` 当模式，把 `login` 和 `app.log` 当文件名；如果你要搜索短语，应该写 `grep "user login" app.log`。这是上一页“Shell 先切词”的直接应用。

# 6. wc：先估规模，别把文本行数当业务含义 {#count}
<!-- tag: 统计规模 -->

`wc` 常用于快速估计文件规模。`wc -l` 数行，`wc -w` 数以空白分隔的词，`wc -c` 数字节。它不理解 CSV、JSON、日志语义，只按文本规则计数。五行日志用 `wc -l` 得到 5；格式化成多行的 JSON 用 `wc -l` 得到文本行数，不能代表对象数量。

正确示例先估规模再决定下一步复制

```bash
wc -l app.log
wc -w app.log
wc -c app.log
```

命令
计数对象
能说明什么
不能说明什么


`wc -l app.log`
换行分隔出的文本行
日志大致有多少行
每行是否是一条业务事件


`wc -w app.log`
空白分隔的词
文本大致有多少片段
中文分词、CSV 字段数量


`wc -c app.log`
字节
文件字节规模
肉眼看到的字符数

中文、emoji 和不同编码会让字节数和字符数不一致。初学阶段把 `wc` 当作规模雷达就够了：它帮助你判断文件大不大、适不适合直接看、下一步是否需要抽样。

# 7. sort 与 uniq：先让相同的行相邻 {#sortuniq}
<!-- tag: 排序去重 -->

`sort` 对行排序，`uniq` 只处理相邻重复行。这个组合很容易被误用：如果相同名字散在文件不同位置，直接 `uniq names.txt` 只能去掉挨在一起的重复，不能全局去重。先 `sort`，重复行才会靠到一起。

样本文本`names.txt`复制

```text
alice
bob
alice
carol
bob
bob
```

正确示例先排序再压相邻重复复制

```bash
sort names.txt
sort names.txt | uniq
sort names.txt | uniq -c
```

步骤
读到什么
输出什么
出错信号


`sort names.txt`
一行一个名字
相同名字挨在一起
数字排序时 `10` 可能排在 `2` 前面


`uniq`
相邻行
相邻重复只保留一份
没排序时漏掉不相邻重复


`uniq -c`
相邻重复块
每个块的计数和内容
计数被散开的重复拆碎

默认 `sort` 走字典序。文本 `1`、`10`、`2` 按默认排序时，`10` 可能排在 `2` 前面；要按数字大小排，用 `sort -n`。工具按你给出的规则工作，你需要明确告诉它按数值理解每一行。

# 9. 读输出：不要只看命令，要能说出结果长什么样 {#read-output}
<!-- tag: 输出阅读 -->

文本处理真正有用的地方，不在于你记住了命令名，而在于你能看懂输出。比如 `head` 输出的是原文件的开头几行，`grep -n` 输出的是“行号:原始行”，`wc -l` 输出的是“行数 文件名”，`uniq -c` 输出的是“计数 行内容”。这些输出形状不同，后面能不能继续处理，也不同。

输出样本同一份日志的四种结果复制

```text
$ head -n 2 app.log
2026-06-21 09:00 INFO user alice login
2026-06-21 09:02 ERROR user bob timeout

$ grep -n "ERROR" app.log
2:2026-06-21 09:02 ERROR user bob timeout
4:2026-06-21 09:08 ERROR user alice timeout

$ wc -l app.log
5 app.log

$ sort names.txt | uniq -c
 2 alice
 3 bob
 1 carol
```

输出形状
怎么读
还能直接当原始行吗
常见看错


普通文本行
`head`、`tail` 保留原行，只截取一部分
可以，但样本可能不代表全量
用前 5 行推断所有行都同格式


`2:原始行`
`grep -n` 给每个匹配行加行号前缀
行内容还在，但多了行号
把行号误认为文件原本字段


`5 app.log`
`wc -l` 输出计数和文件名
不可以，原始行已经丢掉
把 5 当成错误次数或用户数量


`2 alice`
`uniq -c` 输出相邻重复块的数量和内容
不可以，已经变成聚合结果
没排序时把分散重复漏掉，还以为统计正确

读输出时，要问两件事：输出是否还保留原始行，以及有没有新增的前缀或计数字段。`grep -n` 的行号对定位很有用，但如果后面要把每行当日志字段切开，行号前缀就可能干扰你。`wc` 和 `uniq -c` 已经把文本压成摘要，适合做规模判断或频次判断，不适合再当原始日志行解释。

举个具体错误：你用 `grep -n "ERROR" app.log` 得到 `2:2026-06-21 ...`，再把这一行当成普通日志字段读，第一列就变成了 `2:2026-06-21`。日志格式没有变，输出里只是多了定位信息。需要继续按字段处理时，先去掉 `-n`，或者明确后续步骤知道前面多了行号。

这也是下一页学习管道时最容易出错的地方。你把一个命令的输出交给下一个命令前，必须知道它还是“原始行”，还是已经变成“带行号的行”“计数结果”“排序后的行”。如果中间形状变了，后面的命令就会读到与你想象不同的文本。

> [!NOTE]
> 练习钩子：看到任何一段命令输出，先用一句话标注它的形状：原始行、带行号的行、计数、排序结果、聚合结果。标不出来，就不要急着把它交给下一步。

# 11. 文本边界：先确认文件类型 {#text-boundary}
<!-- tag: 输入类型 -->

文本工具最擅长处理一行一行的可读文本。日志、简单配置、CSV 的粗略抽样、以换行分隔的名单，都适合先用这些工具观察。压缩包、图片、数据库文件、二进制缓存、复杂 JSON 或 Excel 文件，就不能只靠 `grep`、`wc`、`sort` 直接判断含义。

边界检查能避免假结论。`grep` 扫到压缩包里某个字节序列时，可能只能告诉你 `Binary file matches`；`wc -l` 对压缩包没有“记录数”意义；`sort` 对二进制文件排序更没有业务意义。先确认输入类型，再选择工具，会比盲目追加参数更稳。

```bash

```

这里的关键动作是“先看输入类型”。遇到二进制、压缩包或结构化格式时，先换工具或先转换；遇到普通文本，再决定查看、筛选、统计还是排序。能守住这个边界，你就不会把一条看似正常的文本命令当成可靠结论。实操时，把文件类型、抽样输出、下一步工具写成三行，排错时会清楚很多。

# 13. 完整走一遍：从陌生日志到可复查结论 {#walkthrough}
<!-- tag: 实操链 -->

现在把前面的工具放进一个真实顺序里。假设你接到一句话：“服务刚才有 timeout，帮我看一下日志。”不要马上写一条很长的命令。先把任务拆成三件事：文件是否就是目标日志，最近内容里有没有相关信号，匹配行能不能给出足够上下文。

下面这组命令故意保持单步。每一步都应该能单独解释输出，哪一步不对，就停在那里修正，避免继续往后猜。

实操顺序先确认，再筛选，再补上下文复制

```bash
head -n 3 app.log
wc -l app.log
tail -n 20 app.log
grep -n "timeout" app.log
grep -C 1 "timeout" app.log
```

步骤
你在验证什么
正常输出意味着什么
异常时先改哪里


`head -n 3 app.log`
文件格式、字段顺序、是否真是日志
你知道时间、级别、用户、事件大概在哪些位置
路径、文件名、是否文本


`wc -l app.log`
文件规模
你知道直接翻页还是先抽样
不要把行数当业务事件数，先看格式


`tail -n 20 app.log`
最近内容
你知道新错误是否还在发生
检查日志是否滚动、文件是否是旧文件


`grep -n "timeout" app.log`
全文件是否有目标词
你得到匹配整行和行号
检查大小写、短语引号、模式是否过窄


`grep -C 1 "timeout" app.log`
目标词前后的上下文
你能看到错误前后发生了什么
调大上下文，或改用 `less` 定位行号附近

这条链路里，`head` 和 `wc` 不直接回答 timeout 的原因，但它们让你知道自己面对的文件是什么样。`tail` 让你靠近最近发生的内容，`grep` 负责筛出目标行，`-C` 负责补上下文。它们各做一小件事，所以每一步都能被检查。

如果 `grep -n "timeout" app.log` 没有输出，不要立刻说“没有 timeout”。先想几个替代假设：大小写可能是 `Timeout`，日志可能写成 `timed out`，你可能看错文件，或者时间窗口不在这份日志里。下一步可以试 `grep -i "timeout" app.log`，也可以回到 `less app.log` 里搜索相关词。

> [!NOTE]
> 练习钩子：以后排查日志时，把这五步写进记录里。每一步后面写一句“这一步确认了什么”。如果最后结论错了，你能回到具体步骤修，避免重新猜整条命令。复盘时再补一句“哪一步最早暴露异常”，下次排查会更快。

### 1 单选

**Q:** 1. 单选：打开一个很长的日志文件，想翻页和搜索，优先用什么？

- [ ] . less app.log
- [ ] . cat app.log
- [ ] . mkdir app.log

Answer: . less app.log
Explain: 长文件用 less 分页读，避免把终端刷满。

### 2 单选

**Q:** 2. 单选：查看 `data.csv` 前 5 行，最合适的是？

- [ ] . head -n 5 data.csv
- [ ] . tail -f data.csv
- [ ] . uniq data.csv

Answer: . head -n 5 data.csv
Explain: head 用来看开头，-n 5 指定前 5 行。

### 3 判断

**Q:** 3. 判断：`grep` 默认输出匹配到的整行。

- [ ] 正确
- [ ] 错误

Answer: 正确
Explain: grep 默认输出匹配到的整行。

### 4 判断

**Q:** 4. 判断：`wc -l` 一定等于 CSV 或 JSON 里的业务记录数。

- [ ] 正确
- [ ] 错误

Answer: 错误
Explain: wc -l 数文本行，不保证等于业务记录数。

### 5 单选

**Q:** 5. 单选：`grep` 显示行号常用哪个选项？

- [ ] . -n
- [ ] . -i
- [ ] . -f

Answer: . -n
Explain: -n 显示匹配行所在行号。

### 6 单选

**Q:** 6. 单选：搜索 `ERROR` 和 `error` 都想命中，`grep` 常用哪个选项？

- [ ] . -n
- [ ] . -i
- [ ] . -v

Answer: . -i
Explain: -i 用于 case-insensitive search。

### 7 命令阅读

**Q:** 7. 阅读：`grep -n "ERROR" app.log` 中每一部分是什么意思？

Explain: grep 逐行匹配文本； -n 显示行号； "ERROR" 是模式，引号保证 Shell 把它当成一个参数； app.log 是输入文件；输出是含有 ERROR 的整行，前面带行号。

### 8 排序

**Q:** 8. 排序：把“全局去重并计数”的步骤排出来。

Explain: 参考顺序：读取原文件 → 用 sort 让相同行相邻 → 用 uniq -c 压缩相邻重复块并输出计数。要点是 uniq 只看相邻行，不会主动全局扫描同值。

### 9 单选

**Q:** 9. 单选：搜索短语 `user login`，更稳的写法是？

- [ ] . grep "user login" app.log
- [ ] . grep user login app.log
- [ ] . grep app.log user login

Answer: . grep "user login" app.log
Explain: 短语有空格，Shell 会先切词，所以需要引号。

### 10 单选

**Q:** 10. 单选：数字按大小排序时，`sort` 常用哪个选项？

- [ ] . -i
- [ ] . -n
- [ ] . -C

Answer: . -n
Explain: sort -n 按数值排序。

### 11 多选

**Q:** 11. 多选：排查日志里的 `timeout`，哪些命令可能有用？

- [ ] -n "timeout" app.log
- [ ] -n 50 app.log
- [ ] app.log 后搜索 /timeout
- [ ] timeout

Explain: 前三个都有可能有用： grep 直接筛行， tail 先看最近日志， less 适合翻页和搜索上下文。 mkdir 是建目录，不参与日志排查。

### 12 错误诊断

**Q:** 12. 诊断：`grep "ERROR" archive.zip` 出现 `Binary file matches`。按“错因 / 后果 / 修法”回答。

Explain: 错因： archive.zip 是压缩包或二进制内容， grep 只能判断里面似乎有匹配。后果：你看不到可读的文本上下文，也不能把它当日志行处理。修法：先解压或用格式工具读取，再对解出的文本文件运行 grep ；也可以先用 file archive.zip 确认类型。

### 13 场景

**Q:** 13. 场景：你拿到陌生 `data.csv`，想快速判断格式和行数，会怎么做？

Explain: 一个稳妥答案是：先 head -n 5 data.csv 看表头、分隔符和前几行结构，再 wc -l data.csv 估算文本行数。答案里要说明 wc -l 是文本行数，不保证就是业务记录数；如果文件很长，再用 less data.csv 搜索或抽看。

### 14 综合

**Q:** 14. 费曼题：向零基础同学解释如何用命令行查看、搜索、统计一个文本日志。

Explain: 答案要能讲出一条流程：先用 head 看格式、 tail 看最新内容，文件很长就用 less 分页搜索；要找错误行，用 grep -n "ERROR" app.log 输出匹配整行和行号；要估规模，用 wc -l app.log 看文本行数。还要提醒：短语要加引号，二进制文件不能直接当文本处理。
