查找、管道与重定向

这一页把前两页学到的路径、参数和文本工具连起来：一个程序把正常结果写到 stdout ，另一个程序从 stdin 读；错误通常走 stderr ；重定向把输出写入文件； find 负责先在文件系统里定位目标。这里不展开完整脚本、进程替换和高级文件描述符，只把“每一步读什么、写什么、错了去哪”讲到能手工追踪。

1. 学习目标：把命令链拆成可检查的流向

学习目标

前一页你已经能读单个文本工具的输入和输出。现在开始把工具接起来。组合命令的关键在于每一段的输入输出都能被单独检查。你要能看到一条链里哪一步读文件、哪一步把文本交给下一步、哪一步写文件、哪一步把错误留在屏幕。

贯穿样本小项目和日志复制

这一页先不追完整脚本和高级

xargs

。先把单条命令链拆清楚，后面写脚本才不会把错误藏进更长的流程。

2. 主线：命令链是在改流向

流向主线

Shell 组合命令时，最常见的三类动作是：把正常输出交给下一个程序，把正常输出写到文件，把错误输出另放一边。看起来是几个符号，实际是在改三条流：stdin、stdout、stderr。

3. 标准流：屏幕上看到的内容未必走同一条路

stdin

一个命令运行时通常有三条标准流。stdin 是数据入口，stdout 是正常结果，stderr 是诊断和错误。默认情况下，stdout 和 stderr 都显示在屏幕上，所以你肉眼看到它们混在一起；一旦重定向，差别就会显出来。

代码
grep "ERROR" app.log
grep "ERROR" missing.log

4. 重定向：写文件前先确认覆盖和错误流

重定向

> 会把 stdout 写进文件，目标文件存在时通常会先截断。>> 是追加。2> 专门处理 stderr。最该警惕的是覆盖风险：输出文件和输入文件写成同一个名字时，Shell 可能先准备输出文件，导致原文件被清空。

代码
grep "ERROR" app.log > errors.txt
grep "WARN" app.log >> report.txt
grep "TODO" missing.md 2> grep-errors.txt
grep "TODO" missing.md > matches.txt 2> grep-errors.txt

危险示例把输入文件也当输出文件复制

代码
sort names.txt > names.txt
# Shell 可能先清空 names.txt，再让 sort 读取它。

5. 管道：左边的 stdout 变成右边的 stdin

管道

| 连接的是内容流。grep "ERROR" app.log | wc -l 先把匹配行写到 stdout，再把这些行交给 wc -l 的 stdin。右边的 wc 不知道原文件叫 app.log，它只看到了左边吐出来的文本。

代码
grep "ERROR" app.log
grep "ERROR" app.log | wc -l
grep "ERROR" app.log | head -n 1

6. find：从起点开始，按条件筛路径

查找文件

find 的基本模型是 find 起点 条件。起点告诉它从哪个目录往下走，条件告诉它保留哪些路径。find . -type f -name ".md" 表示从当前目录开始，只找普通文件，文件名要匹配 .md。

代码
find . -type f -name "*.md"
find notes -type f -name "*.md"
find src -type f -name "test_*.py"
find . -type d -name "tmp"

通配模式常要加引号。没有引号时，Shell 可能先把 *.md 展开成当前目录已有的文件名，find 就收不到原本的模式。这个问题和上一页的参数切分是同一类：Shell 在程序运行前已经改写了参数。

片段角色读什么常见错法
.搜索起点当前目录往下的文件树把它当成输出文件
-type f类型条件路径对应对象的类型忘了限制类型，目录也被列出来
-name "*.md"名称条件每个路径的文件名未加引号，被 Shell 提前展开

7. 路径文本和文件内容：管道不会自动帮你打开文件

路径

这是初学管道最容易错的一点：find 输出的是路径文本，grep "TODO" 从 stdin 读时搜索的是这些路径字符串本身，不会自动打开每个路径对应的文件。你要先分清右边程序期待的是“文本内容”还是“文件名参数”。

代码
find . -type f -name "*.md" | grep "TODO"
# 这里 grep 搜的是路径里有没有 TODO，不会打开每个 .md 文件。

更直接递归搜索文件内容复制

代码
grep -r "TODO" notes/
grep -r "TODO" . --include="*.md"

8. 链路追踪：手工走一遍 grep | wc > 文件

可视化

点击下面的步骤，观察同一条命令链里数据形状如何变化。你应该能在每一步说清：读了什么、stdout 变成什么、有没有写文件、错误还会不会跟着走。

9. 故障面板：看见症状，就回到流向

故障诊断

管道和重定向的错误通常有迹可查：覆盖了输入文件、错误没有进目标文件、通配符被 Shell 提前展开、把路径文本误当文件内容。下面的面板用四个典型症状练习“先定位哪条流出了问题”。

10. 安全实操：每次只接一段

实操链

写组合命令时，先单步跑，确认输出形状，再接下一段。这个习惯能让你在短命令里发现问题，避免一串命令一起失败后只能猜。

完整案例从筛日志到追加日报复制

第一步只筛行，不统计，也不写文件。你应该看到包含 timeout 的日志行；如果没有输出，先不要接 wc，改查大小写、拼写、文件路径和日志时间范围。第二步才统计行数，此时屏幕上应该只有一个数字。第三步把数字写到临时文件 timeout-count.txt，避免直接覆盖日报。第四步读回临时文件，确认它确实只有数字。最后两步才追加到 report.txt：先追加标签，再追加数字。

练习钩子：以后写任何带

和

>

的命令，都先在纸上或笔记里写三列：左侧 stdout、右侧 stdin、目标文件。只要其中一列写不出来，就不要急着记录下来。

再看一个更贴近排查现场的分流场景：你要把正常匹配结果保存给同事，同时把“文件不存在、权限不足”这类诊断留给自己排查。此时不要把所有东西粗暴混在一个文件里。正常结果和错误信息服务的是不同任务，应该先分开保存。

代码
grep -n "ERROR" app.log missing.log > matches.txt 2> errors.txt
cat matches.txt
cat errors.txt
wc -l matches.txt errors.txt

这条命令故意同时给 grep 两个输入：存在的 app.log 和不存在的 missing.log。匹配到的错误行属于正常结果，会进入 matches.txt；关于 missing.log 的报错属于诊断，会进入 errors.txt。之后分别 cat 两个文件，是为了确认它们没有被混在一起。最后用 wc -l 看两个文件各有多少行，能快速判断“正常结果为空”还是“命令本身出了错”。

文件应该装什么如果为空代表什么下一步
matches.txt匹配到的日志行，带行号可能真的没有匹配，也可能模式或文件路径不对先看 errors.txt，再检查模式
errors.txt文件不存在、权限不足等诊断命令没有诊断错误，不代表一定有匹配结果回到 matches.txt 检查正常结果

还要处理一种很常见的“空结果”场景。matches.txt 为空时，结论不能立刻写成“没有错误”。先同时看 errors.txt：如果错误文件有内容，说明命令运行过程中发生了诊断问题；如果错误文件也为空，才更接近“命令成功运行，只是没有匹配行”。这一步能避免把运行失败误当成数据结论。

代码
grep -n "CRITICAL" app.log missing.log > matches.txt 2> errors.txt
wc -l matches.txt errors.txt
cat errors.txt

练习钩子：写排查记录时，把空结果写成“结果文件 0 行，错误文件 0 行”或“结果文件 0 行，错误文件有 1 行诊断”。这比一句“没找到”更可复查。

练习钩子：遇到“结果文件是空的”时，不要只问 grep 有没有匹配。先问：stdout 文件空，还是 stderr 文件有内容？这一个问题能把“数据确实没有”和“命令运行失败”分开。

安全组合顺序先看，再接，再写文件复制

步骤确认什么异常时怎么停进入下一步的证据
先跑左侧命令输出是否是预期文本行修模式、路径、大小写看到了正确样本行
接一个管道右侧是否能从 stdin 读检查右侧命令需要内容还是文件名输出形状符合预期
最后再重定向目标文件是否安全改写到新文件新文件内容可读、可核对

5单选

Q: 5. 单选：grep "ERROR" app.log | wc -l > count.txt 最后写进 count.txt 的是什么？

参考答案 / 评分要点

匹配 ERROR 的行数。grep 先输出 ERROR 行，wc -l 把行数输出到 stdout，最后 > 写进 count.txt。

6单选

Q: 6. 单选：cmd >> report.txt 表示什么？

参考答案 / 评分要点

追加 stdout 到文件末尾。>> 表示把 stdout 追加到文件末尾，不覆盖已有内容。

7命令阅读

Q: 7. 阅读：grep "WARN" app.log >> report.txt 2> err.txt 中 stdout 和 stderr 分别去了哪里？

参考答案 / 评分要点

。stdout 是匹配到的 WARN 行，会追加到 report.txt 。stderr 是诊断或错误信息，会写入 err.txt 。这里 >> 和 2> 分别处理两条不同的输出流。

8排序

Q: 8. 排序：把“统计 ERROR 行并保存”的链路排出来。

参考答案 / 评分要点

。参考顺序：读取 app.log → grep "ERROR" 筛出匹配行 → wc -l 统计这些行的数量 → > count.txt 把数量写入文件。要点是 wc 看到的是上一步输出的文本行，不直接读取原日志文件。

三、find、路径文本与文件内容

检查点 find 输出的是路径文本；是否读取文件内容，取决于下一步工具。 -->
9单选

Q: 9. 单选：find . -type f -name "*.md" 里的 . 表示什么？

参考答案 / 评分要点

从当前目录开始搜索。find 的第一个位置参数通常是搜索起点，. 表示当前目录。

10单选

Q: 10. 单选：-name ".md" 中 ".md" 常加引号的主要原因是？

参考答案 / 评分要点

防止当前 Shell 提前展开通配符。给 *.md 加引号，是为了让 find 收到这个模式，而不是让当前 Shell 先展开。

11判断

Q: 11. 判断：find . -name "*.md" | grep TODO 搜的是路径文本里有没有 TODO。

参考答案 / 评分要点

正确。这条命令里的 grep 搜的是 find 输出的路径字符串，不会打开每个 md 文件搜索 TODO 内容。

12诊断

Q: 12. 诊断：你想找 Markdown 文件内容里的 TODO，为什么只写 find . -name "*.md" | grep TODO 不够？

参考答案 / 评分要点

。find 输出的是文件路径文本，比如 ./notes/todo.md 。管道后的 grep TODO 看到的是这些路径行，所以它只判断路径字符串里有没有 TODO ，没有打开文件内容。要搜文件内容，可以先用 grep -R "TODO" . ，或在学会更稳的参数传递后，把 find 找到的路径交给能读取文件内容的命令。

四、故障诊断与安全实操

表达要求 按“错因 / 后果 / 修法”写，能说出最小验证命令。 -->
13单选

Q: 13. 单选：sort names.txt > names.txt 最大的问题是什么？

参考答案 / 评分要点

输入文件和输出文件相同，可能覆盖原内容。把读取文件和输出文件写成同一个路径，有覆盖或截断风险。先写到新文件，再人工检查。

14单选

Q: 14. 单选：运行 grep "ERROR" missing.log > out.txt 后，错误还在屏幕上，最可能的原因是？

参考答案 / 评分要点

> 没有重定向 stderr。> 只处理 stdout。错误信息走 stderr，所以还会显示，除非单独写 2> 或明确合并。

15场景

Q: 15. 场景：你要统计 app.log 中 ERROR 行数并保存。写出一个安全流程。

参考答案 / 评分要点

。一个稳妥流程是：先运行 grep "ERROR" app.log ，确认匹配行正确；再运行 grep "ERROR" app.log | wc -l ，确认数字符合预期；最后运行 grep "ERROR" app.log | wc -l > error-count.txt ，写入新文件；再用 cat error-count.txt 或 less error-count.txt 检查内容。不要把输出文件写成原始日志文件。

16综合

Q: 16. 费曼题：向同学解释“管道和重定向各自改了哪条流”。

参考答案 / 评分要点

。答案要能讲出四点：管道把左边正常输出交给右边输入，不会自动传文件内容或错误信息； > 是写入，常见行为是覆盖目标文件，所以先写新文件更稳； find 输出的是路径文本，后续命令看到的未必是文件内容；stderr 需要用 2> 等方式单独处理。一个合格例子可以围绕 grep "ERROR" app.log | wc -l > count.txt 展开。
