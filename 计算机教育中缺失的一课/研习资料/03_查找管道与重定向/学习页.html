1. 学习目标：把命令链拆成可检查的流向

学习目标

前一页你已经能读单个文本工具的输入和输出。现在开始把工具接起来。组合命令的关键在于每一段的输入输出都能被单独检查。你要能看到一条链里哪一步读文件、哪一步把文本交给下一步、哪一步写文件、哪一步把错误留在屏幕。

本页用一个小项目做贯穿样本。后面所有表格和练习都围绕它展开，避免每节换一个例子。

贯穿样本小项目和日志
project/
  app.log
  report.txt
  notes/todo.md
  notes/done.md
  src/main.py
  src/test_main.py

app.log:
2026-06-21 09:00 INFO boot
2026-06-21 09:02 ERROR user bob timeout
2026-06-21 09:05 WARN retrying request
2026-06-21 09:08 ERROR user alice timeout
这一页先不追完整脚本和高级 xargs。先把单条命令链拆清楚，后面写脚本才不会把错误藏进更长的流程。

2. 主线：命令链是在改流向

流向主线

Shell 组合命令时，最常见的三类动作是：把正常输出交给下一个程序，把正常输出写到文件，把错误输出另放一边。看起来是几个符号，实际是在改三条流：stdin、stdout、stderr。

写法 改了什么 常见用途 容易误判
A | B 把 A 的 stdout 接到 B 的 stdin 筛行后统计、排序后去重 以为它传的是文件名参数
cmd > out.txt 把 stdout 写到文件，通常覆盖 保存正常结果 误覆盖原文件
cmd >> out.txt 把 stdout 追加到文件末尾 持续追加日志或报告 忘了文件会越积越长
cmd 2> err.txt 把 stderr 写到文件 单独保存错误 以为 > 会自动带走错误

读组合命令时，先从左到右标注“这一步输出形状是什么”。比如 grep "ERROR" app.log 输出匹配行；接上 wc -l 后，输出变成行数；最后 > count.txt 把这个行数写进文件。

3. 标准流：屏幕上看到的内容未必走同一条路

stdin / stdout / stderr

一个命令运行时通常有三条标准流。stdin 是数据入口，stdout 是正常结果，stderr 是诊断和错误。默认情况下，stdout 和 stderr 都显示在屏幕上，所以你肉眼看到它们混在一起；一旦重定向，差别就会显出来。

观察两条输出正常匹配和文件不存在
grep "ERROR" app.log
grep "ERROR" missing.log
命令 读什么 stdout stderr 出错时怎么查
grep "ERROR" app.log app.log 的每一行 匹配 ERROR 的整行 通常为空 检查模式、文件路径、大小写
grep "ERROR" missing.log 不存在的文件 没有正常结果 文件不存在的错误 先看路径，再决定是否重定向错误
wc -l stdin 传来的文本 行数 读取失败时的诊断 确认左侧是否真的给了文本

这个分离很有用：正常结果可以交给后续程序或写入文件，错误信息可以留在屏幕提醒你，也可以单独进入错误日志。把两者混起来，后续统计和解析就可能把错误文本当数据。

4. 重定向：写文件前先确认覆盖和错误流

重定向

> 会把 stdout 写进文件，目标文件存在时通常会先截断。>> 是追加。2> 专门处理 stderr。最该警惕的是覆盖风险：输出文件和输入文件写成同一个名字时，Shell 可能先准备输出文件，导致原文件被清空。

正确示例正常结果、追加、错误分流
grep "ERROR" app.log > errors.txt
grep "WARN" app.log >> report.txt
grep "TODO" missing.md 2> grep-errors.txt
grep "TODO" missing.md > matches.txt 2> grep-errors.txt
危险示例把输入文件也当输出文件
sort names.txt > names.txt
# Shell 可能先清空 names.txt，再让 sort 读取它。
写法 stdout 去哪 stderr 去哪 要先确认什么
cmd > out.txt out.txt，覆盖 屏幕 目标文件可以被覆盖
cmd >> out.txt out.txt，追加 屏幕 追加不会造成重复污染
cmd 2> err.txt 屏幕 err.txt 你确实想把错误从屏幕拿走
cmd > out.txt 2>&1 out.txt 跟随 stdout 当前去向 正常结果和错误会混在一起

安全做法很简单：输出先写到新文件，检查内容，确认无误后再替换原文件。后面写脚本时，这个习惯比记住更多符号更重要。

5. 管道：左边的 stdout 变成右边的 stdin

管道

| 连接的是内容流。grep "ERROR" app.log | wc -l 先把匹配行写到 stdout，再把这些行交给 wc -l 的 stdin。右边的 wc 不知道原文件叫 app.log，它只看到了左边吐出来的文本。

逐段读命令从匹配行到数量
grep "ERROR" app.log
grep "ERROR" app.log | wc -l
grep "ERROR" app.log | head -n 1
阶段 输入 输出形状 下一步能怎么读
grep "ERROR" app.log app.log 文件 匹配的日志行 仍然是日志行，可以交给 wc 或 head
wc -l stdin 里的匹配行 一个行数 已经是摘要，不再是日志行
head -n 1 stdin 里的匹配行 第一条匹配日志 适合先抽查结果是否正确

管道默认只接 stdout。左侧命令的 stderr 仍会显示在屏幕，除非你显式重定向。这个设计能避免错误文字混入正常数据，但也要求你知道错误去了哪里。

6. find：从起点开始，按条件筛路径

查找文件

find 的基本模型是 find 起点 条件。起点告诉它从哪个目录往下走，条件告诉它保留哪些路径。find . -type f -name "*.md" 表示从当前目录开始，只找普通文件，文件名要匹配 *.md。

正确示例按名称和类型找文件
find . -type f -name "*.md"
find notes -type f -name "*.md"
find src -type f -name "test_*.py"
find . -type d -name "tmp"

通配模式常要加引号。没有引号时，Shell 可能先把 *.md 展开成当前目录已有的文件名，find 就收不到原本的模式。这个问题和上一页的参数切分是同一类：Shell 在程序运行前已经改写了参数。

片段 角色 读什么 常见错法
. 搜索起点 当前目录往下的文件树 把它当成输出文件
-type f 类型条件 路径对应对象的类型 忘了限制类型，目录也被列出来
-name "*.md" 名称条件 每个路径的文件名 未加引号，被 Shell 提前展开

find 输出的是路径文本。路径文本可以被你阅读、保存或交给专门处理路径参数的工具；它和文件内容是两种不同的数据。

7. 路径文本和文件内容：管道不会自动帮你打开文件

路径 vs 内容

这是初学管道最容易错的一点：find 输出的是路径文本，grep "TODO" 从 stdin 读时搜索的是这些路径字符串本身，不会自动打开每个路径对应的文件。你要先分清右边程序期待的是“文本内容”还是“文件名参数”。

容易误读搜索路径文本，未读取文件内容
find . -type f -name "*.md" | grep "TODO"
# 这里 grep 搜的是路径里有没有 TODO，不会打开每个 .md 文件。
更直接递归搜索文件内容
grep -r "TODO" notes/
grep -r "TODO" . --include="*.md"
左侧输出 右侧命令期待什么 直接管道是否合适 更稳的动作
find ... 输出路径文本 grep "TODO" 从 stdin 读文本内容 通常不合适，搜到的是路径字符串 用 grep -r 或学习 xargs / find -exec
grep ... 输出日志行 wc -l 从 stdin 读文本行 合适 先单步看 grep 输出是否正确
sort 输出排序后的行 uniq -c 从 stdin 读相邻行 合适 确认排序不会破坏你还需要的原始顺序

xargs 能把 stdin 里的文本转成命令参数，但它有空格、换行、特殊字符和分隔符边界。这里先不展开完整用法，只要先记住：管道传内容；文件名参数需要专门处理。

8. 链路追踪：手工走一遍 grep | wc > 文件

可视化

点击下面的步骤，观察同一条命令链里数据形状如何变化。你应该能在每一步说清：读了什么、stdout 变成什么、有没有写文件、错误还会不会跟着走。

阶段 1stdout 是匹配日志行

grep 从 app.log 读，每个匹配 ERROR 的整行进入 stdout。

阶段 2stdin 是匹配行，stdout 是数字

wc -l 不再读 app.log，它只数管道传来的匹配行。

阶段 3数字写入文件

> 把 wc 的 stdout 写入 count.txt。屏幕可能没有正常结果。

错误流默认不进管道

如果 app.log 不存在，错误走 stderr，默认会显示在屏幕，不会被 wc 当作输入。

链路段 读取对象 stdout stderr 写入文件
grep "ERROR" app.log app.log 匹配行 文件或权限错误 无
| wc -l 管道来的匹配行 行数 wc 自己的错误 无
> count.txt wc 的 stdout 不再显示到屏幕 仍默认到屏幕 count.txt

9. 故障面板：看见症状，就回到流向

故障诊断

管道和重定向的错误通常有迹可查：覆盖了输入文件、错误没有进目标文件、通配符被 Shell 提前展开、把路径文本误当文件内容。下面的面板用四个典型症状练习“先定位哪条流出了问题”。

$ sort names.txt > names.txt
# names.txt 可能在 sort 读取前已被清空
症状 错掉的假设 先查什么 修法
输出文件变空 输入和输出用了同一个路径 命令里 > 右边是否等于输入文件 写到新文件，检查后再替换
用了 > 仍看到错误 stderr 没有被重定向 错误是否属于诊断信息 需要收集时加 2> err.txt
find 匹配异常 通配符先被 Shell 展开 模式是否加引号 find . -name "*.md"
右侧命令没有处理文件内容 管道给的是路径文本 右侧命令读 stdin 还是要文件参数 换成递归搜索，或学习 xargs / find -exec

10. 安全实操：每次只接一段

实操链

写组合命令时，先单步跑，确认输出形状，再接下一段。这个习惯能让你在短命令里发现问题，避免一串命令一起失败后只能猜。

完整场景：你要给当天日报补一句“日志里有多少次 timeout 错误”，并把数字写入 report.txt。如果直接写一条很长的命令，最容易把三件事混在一起：模式是否匹配、统计是否准确、写入是否安全。更稳的做法是把它拆成四个可观察的小动作，每一步都有一个能停下来的证据。

完整案例从筛日志到追加日报
grep -i "timeout" app.log
grep -i "timeout" app.log | wc -l
grep -i "timeout" app.log | wc -l > timeout-count.txt
cat timeout-count.txt
printf "timeout errors: " >> report.txt
cat timeout-count.txt >> report.txt

第一步只筛行，不统计，也不写文件。你应该看到包含 timeout 的日志行；如果没有输出，先不要接 wc，改查大小写、拼写、文件路径和日志时间范围。第二步才统计行数，此时屏幕上应该只有一个数字。第三步把数字写到临时文件 timeout-count.txt，避免直接覆盖日报。第四步读回临时文件，确认它确实只有数字。最后两步才追加到 report.txt：先追加标签，再追加数字。

步骤 命令片段 此时 stdout 是什么 能发现的错误 是否写入日报
筛选样本 grep -i "timeout" app.log 匹配到的日志行 模式写错、文件路径写错、大小写没处理 否
统计数量 | wc -l 一个数字和换行 上一段输出偏离预期，导致数字没有意义 否
临时落盘 > timeout-count.txt 不再显示，写入临时文件 目标文件名写错、目录无写入权限 否
追加日报 >> report.txt 追加到日报末尾 用成 > 覆盖旧日报，或忘了先写标签 是

这个案例里，临时文件就是检查点。数字和日报分开，出错时只需要重算数字，不会破坏已有报告。如果你把命令写成 grep -i "timeout" app.log | wc -l > report.txt，它也能保存数字，但会把整份日报覆盖成一个数字；如果写成 grep -i "timeout" app.log >> report.txt | wc -l，管道右侧甚至可能拿不到你以为的输入，因为左侧 stdout 已经被追加重定向拿走了。

练习钩子：以后写任何带 | 和 > 的命令，都先在纸上或笔记里写三列：左侧 stdout、右侧 stdin、目标文件。只要其中一列写不出来，就不要急着落盘。

再看一个更贴近排查现场的分流场景：你要把正常匹配结果保存给同事，同时把“文件不存在、权限不足”这类诊断留给自己排查。此时不要把所有东西粗暴混在一个文件里。正常结果和错误信息服务的是不同任务，应该先分开保存。

分流案例正常结果与错误日志分开
grep -n "ERROR" app.log missing.log > matches.txt 2> errors.txt
cat matches.txt
cat errors.txt
wc -l matches.txt errors.txt

这条命令故意同时给 grep 两个输入：存在的 app.log 和不存在的 missing.log。匹配到的错误行属于正常结果，会进入 matches.txt；关于 missing.log 的报错属于诊断，会进入 errors.txt。之后分别 cat 两个文件，是为了确认它们没有被混在一起。最后用 wc -l 看两个文件各有多少行，能快速判断“正常结果为空”还是“命令本身出了错”。

文件 应该装什么 如果为空代表什么 下一步
matches.txt 匹配到的日志行，带行号 可能真的没有匹配，也可能模式或文件路径不对 先看 errors.txt，再检查模式
errors.txt 文件不存在、权限不足等诊断 命令没有诊断错误，不代表一定有匹配结果 回到 matches.txt 检查正常结果

失败后果也很清楚：如果只写 > matches.txt，你会以为所有输出都被保存了，但错误仍在屏幕上；如果写成 2> matches.txt，你保存的是诊断，匹配结果留不下来；如果把两条流合并到同一个文件，初学阶段又容易分不清哪一行是数据、哪一行是错误。先分流，确认两边内容，再决定是否合并，这是更可检查的做法。

还要处理一种很常见的“空结果”场景。matches.txt 为空时，结论不能立刻写成“没有错误”。先同时看 errors.txt：如果错误文件有内容，说明命令运行过程中发生了诊断问题；如果错误文件也为空，才更接近“命令成功运行，只是没有匹配行”。这一步能避免把运行失败误当成数据结论。

空结果判定先看结果，再看诊断
grep -n "CRITICAL" app.log missing.log > matches.txt 2> errors.txt
wc -l matches.txt errors.txt
cat errors.txt
观察结果 更合理的解释 下一步动作
matches.txt 为空，errors.txt 为空 命令成功运行，当前输入里没有匹配行 检查模式是否过窄，再决定是否放宽关键词
matches.txt 为空，errors.txt 有内容 命令遇到文件或权限问题，数据结论暂时无效 先修路径、权限或输入列表，再重新统计
练习钩子：写排查记录时，把空结果写成“结果文件 0 行，错误文件 0 行”或“结果文件 0 行，错误文件有 1 行诊断”。这比一句“没找到”更可复查。
练习钩子：遇到“结果文件是空的”时，不要只问 grep 有没有匹配。先问：stdout 文件空，还是 stderr 文件有内容？这一个问题能把“数据确实没有”和“命令运行失败”分开。
安全组合顺序先看，再接，再写文件
grep "ERROR" app.log
grep "ERROR" app.log | wc -l
grep "ERROR" app.log | wc -l > count.txt
cat count.txt
find . -type f -name "*.md"
grep -r "TODO" . --include="*.md"
步骤 确认什么 异常时怎么停 进入下一步的证据
先跑左侧命令 输出是否是预期文本行 修模式、路径、大小写 看到了正确样本行
接一个管道 右侧是否能从 stdin 读 检查右侧命令需要内容还是文件名 输出形状符合预期
最后再重定向 目标文件是否安全 改写到新文件 新文件内容可读、可核对

7. 阅读：grep "WARN" app.log >> report.txt 2> err.txt 中 stdout 和 stderr 分别去了哪里？

参考答案 / 评分要点

stdout 是匹配到的 WARN 行，会追加到 report.txt。stderr 是诊断或错误信息，会写入 err.txt。这里 >> 和 2> 分别处理两条不同的输出流。

排序流向追踪

8. 排序：把“统计 ERROR 行并保存”的链路排出来。

候选：写入 count.txt / 筛出 ERROR 行 / 统计行数 / 读取 app.log。

参考答案 / 评分要点

参考顺序：读取 app.log → grep "ERROR" 筛出匹配行 → wc -l 统计这些行的数量 → > count.txt 把数量写入文件。要点是 wc 看到的是上一步输出的文本行，不直接读取原日志文件。

判断路径 vs 内容

诊断边界表达

12. 诊断：你想找 Markdown 文件内容里的 TODO，为什么只写 find . -name "*.md" | grep TODO 不够？

参考答案 / 评分要点

find 输出的是文件路径文本，比如 ./notes/todo.md。管道后的 grep TODO 看到的是这些路径行，所以它只判断路径字符串里有没有 TODO，没有打开文件内容。要搜文件内容，可以先用 grep -R "TODO" .，或在学会更稳的参数传递后，把 find 找到的路径交给能读取文件内容的命令。

场景实操计划

15. 场景：你要统计 app.log 中 ERROR 行数并保存。写出一个安全流程。

参考答案 / 评分要点

一个稳妥流程是：先运行 grep "ERROR" app.log，确认匹配行正确；再运行 grep "ERROR" app.log | wc -l，确认数字符合预期；最后运行 grep "ERROR" app.log | wc -l > error-count.txt，写入新文件；再用 cat error-count.txt 或 less error-count.txt 检查内容。不要把输出文件写成原始日志文件。

综合讲清楚

16. 费曼题：向同学解释“管道和重定向各自改了哪条流”。

建议包含：管道只传 stdout、> 可能覆盖、find 输出路径文本、错误需要单独处理。

参考答案 / 评分要点

答案要能讲出四点：管道把左边正常输出交给右边输入，不会自动传文件内容或错误信息；> 是写入，常见行为是覆盖目标文件，所以先写新文件更稳；find 输出的是路径文本，后续命令看到的未必是文件内容；stderr 需要用 2> 等方式单独处理。一个合格例子可以围绕 grep "ERROR" app.log | wc -l > count.txt 展开。
