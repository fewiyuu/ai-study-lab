把程序连起来：让输出变输入，让结果落到文件

上一块学会了单个文本工具。本块学习工具之间的连接方式：标准输入、标准输出、标准错误、重定向、管道，以及用 find 在文件系统中定位目标。你要学会的不是符号本身，而是追踪每一步“读什么、写什么、错误去哪”。

学完你应该能做什么

  1. 解释 stdin、stdout、stderr 的区别。
  2. 区分 > 覆盖、>> 追加、2> 错误输出。
  3. 用 | 把一个程序输出交给下一个程序。
  4. 用 find 按名称、类型、路径查文件。

概念地图

命令行组合不是把程序“粘在一起”，而是把一个程序的文本输出接到另一个程序的文本输入。每个进程通常有三条标准流：标准输入负责读，标准输出负责正常结果，标准错误负责错误和诊断信息。

这三条流让程序可以保持简单。一个程序不必知道下一个程序是谁，也不必知道结果最后显示在屏幕还是写入文件；它只要从标准输入读、向标准输出写、把错误写到标准错误。Shell 负责把这些流接起来。管道和重定向本质上都是“改流向”。

程序 A stdout→ | →程序 B stdin→stdout 到屏幕或文件
stdin标准输入，程序默认读取数据的入口。它可以来自键盘、文件、管道左侧输出。
stdout标准输出，程序正常结果默认去屏幕。管道默认传递的就是这条流。
stderr标准错误，错误信息通常单独走这条流。它和 stdout 同时显示在屏幕上，但重定向时可以分开。
pipe管道，把左边 stdout 接到右边 stdin。它传的是内容流，不是文件名列表。

判断句：读组合命令时，先不要问“这个命令厉不厉害”，先问每一段的输入来自哪里、正常输出去哪里、错误输出会不会被你收集到。

1. 标准流：屏幕上看到的不一定是同一种输出

当你运行 grep、sort、wc，正常结果通常写到 stdout。错误提示，例如文件不存在、权限不足，通常写到 stderr。两者默认都显示在屏幕上，所以初学者会误以为它们没有区别。

区别在重定向时会立刻出现：> 默认只重定向 stdout，不会把 stderr 写进目标文件。这样设计是为了让程序结果和诊断信息分开，方便脚本判断和日志记录。如果错误也混进正常结果文件，后续统计、排序、解析就可能把错误文本当数据处理。

标准输入也很重要。有些命令可以从文件参数读，例如 wc -l app.log；也可以从 stdin 读，例如 grep ERROR app.log | wc -l。这两种写法都可能得到数字，但数据来源不同：前者读文件，后者读管道传来的文本。

逐行拆解正常输出与错误输出
grep "TODO" notes.md       # 匹配行走 stdout
grep "TODO" missing.md     # 文件不存在的错误走 stderr

你现在应该能做什么：看到屏幕输出时，能追问它是正常结果还是错误信息。

2. 重定向：把输出写到文件

> 把 stdout 写入文件，并且会覆盖原文件。>> 追加到文件末尾。覆盖是最危险的新手坑之一：如果你把重要文件当成输出目标，原内容可能被清空。写重定向时，目标文件不是“输入文件的另一个名字”，而是 Shell 要打开来接收输出的位置。

2> 重定向 stderr。2>&1 表示把 stderr 合并到 stdout 当前去向。初学阶段不必死记所有组合，但要知道错误输出不是自动进入 > 的。你看到错误仍在屏幕上，不一定是重定向失败，可能只是你只重定向了 stdout。

重定向还有一个顺序坑：Shell 往往会先准备输出文件，再启动程序。如果你写 sort names.txt > names.txt，Shell 可能先清空 names.txt，然后 sort 再读取这个已经被清空的文件。安全做法是写到新文件，确认后再替换。

正确示例覆盖、追加、错误输出分流
grep "ERROR" app.log > errors.txt
grep "WARN" app.log >> errors.txt
grep "TODO" missing.md 2> grep-errors.txt
错误示例误用覆盖导致原文件丢失
sort names.txt > names.txt
# shell 可能先清空 names.txt，再让 sort 读取它

你现在应该能做什么：在写 > 前停一下，确认目标文件是不是可以被覆盖。

3. 管道：左边的 stdout 变成右边的 stdin

| 是组合小工具的关键。它不会传文件名，而是传文本流。grep "ERROR" app.log | wc -l 的意思是：先输出含 ERROR 的行，再把这些行交给 wc -l 统计行数。右侧命令看到的是左侧输出的内容，不知道这些内容原来来自哪个文件，除非左侧输出里包含文件名。

很多命令既能读文件参数，也能从 stdin 读。wc -l app.log 直接读文件；grep "ERROR" app.log | wc -l 读管道传来的匹配行。理解这一点，才能区分“文件名参数”和“文本内容”。如果一个程序只能接收文件名参数，而不能从 stdin 读，单纯用管道把文件名文本传过去就不一定有效。

管道默认只连接 stdout，不会自动连接 stderr。所以 grep pattern missing.txt | wc -l 中，文件不存在的错误会显示在屏幕，而不是被 wc 统计。这样能避免错误信息污染正常数据，但也意味着你需要显式处理错误日志。

正确示例用管道统计错误行数
grep "ERROR" app.log | wc -l
grep "ERROR" app.log | head -n 5
grep "ERROR" app.log | sort | uniq -c

你现在应该能做什么：读出管道每一段的输入和输出，而不是只把整串命令当咒语。

4. find：在文件系统里按条件找文件

find 的基本模型是：从某个起点开始，递归遍历目录，根据条件筛选路径。find . -name "*.md" 从当前目录开始找名字匹配 *.md 的文件或目录。这里的 . 是搜索起点，-name 是条件，"*.md" 是传给 find 的模式。

模式要加引号，避免被 Shell 提前展开。没有引号时，Shell 可能先在当前目录把 *.md 展开成已有文件名，然后 find 收到的就不是原始模式。这个错误很隐蔽：如果当前目录没有匹配文件，看起来可能正常；一旦当前目录有匹配文件，命令含义就变了。

常用条件包括 -name 按名称，-type f 只要普通文件，-type d 只要目录。搜索范围越大越慢，所以起点不要总写根目录。权限不足时，find 可能把错误写到 stderr；这时可以把错误重定向掉，但要知道自己隐藏了什么。

正确示例按名称和类型查找
find . -name "*.md"
find notes -type f -name "*.md"
find . -type d -name "backup"
错误示例模式未加引号，被 Shell 提前展开
find . -name *.md
# 当前目录若已有 a.md b.md，Shell 可能先展开成：
# find . -name a.md b.md

你现在应该能做什么：解释 find 起点 条件 的结构，并知道通配模式为什么常要加引号。

5. 组合链路：先找文件，再筛内容，再统计结果

管道擅长处理文本流，但文件名中可能包含空格、换行等复杂字符，直接把 find 输出交给某些命令时要小心。初学阶段先掌握安全边界：简单项目中可以读懂常见组合；生产脚本中要考虑特殊文件名和 -print0、xargs -0 等更稳妥写法。

本块先不展开 xargs 的全部细节，但要建立“每一段命令的输入输出类型”意识。find 输出的是路径文本，grep 可以按文件名参数搜索，也可以从 stdin 搜文本；不同接法含义不同。你不能只看到 | 就以为任何命令都能接上，必须确认右边程序期待的是文本内容还是文件路径参数。

一个稳妥的组合习惯是先单步运行左侧命令，看输出长什么样，再接下一段。先看 find . -type f -name "*.md" 是否真的输出目标路径，再决定要不要把这些路径交给别的工具。这样可以在每一步发现问题，而不是等一长串命令失败后再猜。

逐行拆解从文件到内容再到数量
find . -type f -name "*.md"        # 输出 Markdown 文件路径
grep -r "TODO" notes/              # 在目录中递归搜索内容
grep -r "TODO" notes/ | wc -l      # 统计匹配行数

你现在应该能做什么：说明一串组合命令每一步产出的是什么，以及下一步为什么能接上。

6. 常见错误与修法

覆盖文件

> 会覆盖。错因是把输出目标写成了已有重要文件；后果是旧内容可能先被清空；修法是写到临时文件、确认后替换，或用 >> 追加。

错误没进文件

> 只管 stdout。错因是 stderr 没被重定向；后果是结果文件没有错误信息，屏幕仍显示报错；修法是用 2> 单独保存错误，或按需要合并流。

管道右侧拿不到文件名

管道传文本，不传“文件参数”。错因是把路径文本误当成参数传递；后果是右侧程序可能把路径当普通输入内容；修法是确认右侧能读 stdin，或使用适合接收路径参数的方式。

find 模式提前展开

通配符没加引号。错因是 Shell 先展开了 *.md；后果是 find 收到错误参数；修法是写 "*.md"。

错误示例错误流没有进入结果文件
grep "TODO" missing.md > result.txt
# result.txt 只接 stdout
# 文件不存在的错误仍走 stderr，显示在屏幕
正确示例分别保存结果和错误
grep "TODO" notes.md > result.txt 2> errors.txt
grep "TODO" missing.md > result.txt 2> errors.txt

你现在应该能做什么：看到组合命令异常时，能按“覆盖风险、错误流去向、管道输入类型、Shell 提前展开”四个方向排查。

7. 完整知识链路

本块把单个程序扩展成组合系统：程序正常结果走 stdout，错误走 stderr；重定向改变这些流的去向；管道把左侧 stdout 接到右侧 stdin；find 在文件系统中生成路径列表；grep、wc、sort 等处理文本流。写组合命令时，永远追踪三件事：这一段读什么、写什么、错误去哪里。

真实工作中，一条可靠命令通常不是一次写完的。先跑 find 看候选路径，再跑 grep 看匹配内容，再接 wc -l 统计数量，最后才考虑把结果写入文件。每一步都检查输出形状，能避免把错误、空结果、错路径写进最终文件。

这一块最重要的迁移能力是“流向意识”。你以后看到任何命令组合，都可以画出三条线：stdin 从哪里来，stdout 去屏幕、文件还是管道，stderr 是否被保存。只要这三条线能画清楚，重定向和管道就不再是符号记忆，而是可调试的程序连接方式。

9. 真实场景与排查手册

查找、管道和重定向把命令行从“单个工具”变成“组合系统”。find 或 rg 负责定位候选，管道把上一步输出交给下一步，重定向把输出写入文件或从文件读入。

要分清标准输出和标准错误。正常结果通常走 stdout，错误信息通常走 stderr。只重定向 stdout 时，错误仍会显示在屏幕上；这不是重定向失败，而是流不同。

真实场景里最怕把输出方向想错。覆盖写入会清空旧文件，追加写入才保留历史；把错误输出混进数据文件会污染结果；用 xargs 前要考虑文件名空格和空输入。

覆盖写错文件

> 会覆盖目标文件。生成重要结果前先写到临时文件，确认无误后再替换。

stderr 没被收集

命令失败信息通常走 stderr。需要完整日志时要显式处理 2> 或 2>&1。

xargs 空格问题

文件名含空格时，普通 xargs 可能拆错。优先了解 -print0 和 xargs -0。

组合前先单步

先运行左边命令看输出形状，再接管道；每加一步都确认数据格式。
