# 2. 主线：命令链是在改流向 {#overview}
<!-- tag: 流向主线 -->

Shell 组合命令时，最常见的三类动作是：把正常输出交给下一个程序，把正常输出写到文件，把错误输出另放一边。看起来是几个符号，实际是在改三条流：`stdin`、`stdout`、`stderr`。

写法
改了什么
常见用途
容易误判


`A | B`
把 `A` 的 stdout 接到 `B` 的 stdin
筛行后统计、排序后去重
以为它传的是文件名参数


`cmd > out.txt`
把 stdout 写到文件，通常覆盖
保存正常结果
误覆盖原文件


`cmd >> out.txt`
把 stdout 追加到文件末尾
持续追加日志或报告
忘了文件会越积越长


`cmd 2> err.txt`
把 stderr 写到文件
单独保存错误
以为 `>` 会自动带走错误

读组合命令时，先从左到右标注“这一步输出形状是什么”。比如 `grep "ERROR" app.log` 输出匹配行；接上 `wc -l` 后，输出变成行数；最后 `> count.txt` 把这个行数写进文件。

# 3. 标准流：屏幕上看到的内容未必走同一条路 {#streams}
<!-- tag: stdin / stdout / stderr -->

一个命令运行时通常有三条标准流。`stdin` 是数据入口，`stdout` 是正常结果，`stderr` 是诊断和错误。默认情况下，stdout 和 stderr 都显示在屏幕上，所以你肉眼看到它们混在一起；一旦重定向，差别就会显出来。

观察两条输出正常匹配和文件不存在复制

```bash
grep "ERROR" app.log
grep "ERROR" missing.log
```

命令
读什么
stdout
stderr
出错时怎么查


`grep "ERROR" app.log`
`app.log` 的每一行
匹配 `ERROR` 的整行
通常为空
检查模式、文件路径、大小写


`grep "ERROR" missing.log`
不存在的文件
没有正常结果
文件不存在的错误
先看路径，再决定是否重定向错误


`wc -l`
stdin 传来的文本
行数
读取失败时的诊断
确认左侧是否真的给了文本

这个分离很有用：正常结果可以交给后续程序或写入文件，错误信息可以留在屏幕提醒你，也可以单独进入错误日志。把两者混起来，后续统计和解析就可能把错误文本当数据。

# 4. 重定向：写文件前先确认覆盖和错误流 {#redirect}
<!-- tag: 重定向 -->

`>` 会把 stdout 写进文件，目标文件存在时通常会先截断。`>>` 是追加。`2>` 专门处理 stderr。最该警惕的是覆盖风险：输出文件和输入文件写成同一个名字时，Shell 可能先准备输出文件，导致原文件被清空。

正确示例正常结果、追加、错误分流复制

```bash
grep "ERROR" app.log > errors.txt
grep "WARN" app.log >> report.txt
grep "TODO" missing.md 2> grep-errors.txt
grep "TODO" missing.md > matches.txt 2> grep-errors.txt
```

危险示例把输入文件也当输出文件复制

```bash
sort names.txt > names.txt
# Shell 可能先清空 names.txt，再让 sort 读取它。
```

写法
stdout 去哪
stderr 去哪
要先确认什么


`cmd > out.txt`
`out.txt`，覆盖
屏幕
目标文件可以被覆盖


`cmd >> out.txt`
`out.txt`，追加
屏幕
追加不会造成重复污染


`cmd 2> err.txt`
屏幕
`err.txt`
你确实想把错误从屏幕拿走


`cmd > out.txt 2>&1`
`out.txt`
跟随 stdout 当前去向
正常结果和错误会混在一起

安全做法很简单：输出先写到新文件，检查内容，确认无误后再替换原文件。后面写脚本时，这个习惯比记住更多符号更重要。

# 5. 管道：左边的 stdout 变成右边的 stdin {#pipe}
<!-- tag: 管道 -->

`|` 连接的是内容流。`grep "ERROR" app.log | wc -l` 先把匹配行写到 stdout，再把这些行交给 `wc -l` 的 stdin。右边的 `wc` 不知道原文件叫 `app.log`，它只看到了左边吐出来的文本。

逐段读命令从匹配行到数量复制

```bash
grep "ERROR" app.log
grep "ERROR" app.log | wc -l
grep "ERROR" app.log | head -n 1
```

阶段
输入
输出形状
下一步能怎么读


`grep "ERROR" app.log`
`app.log` 文件
匹配的日志行
仍然是日志行，可以交给 `wc` 或 `head`


`wc -l`
stdin 里的匹配行
一个行数
已经是摘要，不再是日志行


`head -n 1`
stdin 里的匹配行
第一条匹配日志
适合先抽查结果是否正确

管道默认只接 stdout。左侧命令的 stderr 仍会显示在屏幕，除非你显式重定向。这个设计能避免错误文字混入正常数据，但也要求你知道错误去了哪里。

# 6. find：从起点开始，按条件筛路径 {#find}
<!-- tag: 查找文件 -->

`find` 的基本模型是 `find 起点 条件`。起点告诉它从哪个目录往下走，条件告诉它保留哪些路径。`find . -type f -name "*.md"` 表示从当前目录开始，只找普通文件，文件名要匹配 `*.md`。

正确示例按名称和类型找文件复制

```bash
find . -type f -name "*.md"
find notes -type f -name "*.md"
find src -type f -name "test_*.py"
find . -type d -name "tmp"
```

通配模式常要加引号。没有引号时，Shell 可能先把 `*.md` 展开成当前目录已有的文件名，`find` 就收不到原本的模式。这个问题和上一页的参数切分是同一类：Shell 在程序运行前已经改写了参数。

片段
角色
读什么
常见错法


`.`
搜索起点
当前目录往下的文件树
把它当成输出文件


`-type f`
类型条件
路径对应对象的类型
忘了限制类型，目录也被列出来


`-name "*.md"`
名称条件
每个路径的文件名
未加引号，被 Shell 提前展开

`find` 输出的是路径文本。路径文本可以被你阅读、保存或交给专门处理路径参数的工具；它和文件内容是两种不同的数据。

# 7. 路径文本和文件内容：管道不会自动帮你打开文件 {#path-vs-content}
<!-- tag: 路径 vs 内容 -->

这是初学管道最容易错的一点：`find` 输出的是路径文本，`grep "TODO"` 从 stdin 读时搜索的是这些路径字符串本身，不会自动打开每个路径对应的文件。你要先分清右边程序期待的是“文本内容”还是“文件名参数”。

容易误读搜索路径文本，未读取文件内容复制

```bash
find . -type f -name "*.md" | grep "TODO"
# 这里 grep 搜的是路径里有没有 TODO，不会打开每个 .md 文件。
```

更直接递归搜索文件内容复制

```bash
grep -r "TODO" notes/
grep -r "TODO" . --include="*.md"
```

左侧输出
右侧命令期待什么
直接管道是否合适
更稳的动作


`find ...` 输出路径文本
`grep "TODO"` 从 stdin 读文本内容
通常不合适，搜到的是路径字符串
用 `grep -r` 或学习 `xargs` / `find -exec`


`grep ...` 输出日志行
`wc -l` 从 stdin 读文本行
合适
先单步看 `grep` 输出是否正确


`sort` 输出排序后的行
`uniq -c` 从 stdin 读相邻行
合适
确认排序不会破坏你还需要的原始顺序

`xargs` 能把 stdin 里的文本转成命令参数，但它有空格、换行、特殊字符和分隔符边界。这里先不展开完整用法，只要先记住：管道传内容；文件名参数需要专门处理。

# 10. 安全实操：每次只接一段 {#safe-workflow}
<!-- tag: 实操链 -->

写组合命令时，先单步跑，确认输出形状，再接下一段。这个习惯能让你在短命令里发现问题，避免一串命令一起失败后只能猜。

完整场景：你要给当天日报补一句“日志里有多少次 timeout 错误”，并把数字写入 `report.txt`。如果直接写一条很长的命令，最容易把三件事混在一起：模式是否匹配、统计是否准确、写入是否安全。更稳的做法是把它拆成四个可观察的小动作，每一步都有一个能停下来的证据。

完整案例从筛日志到追加日报复制

```bash
grep -i "timeout" app.log
grep -i "timeout" app.log | wc -l
grep -i "timeout" app.log | wc -l > timeout-count.txt
cat timeout-count.txt
printf "timeout errors: " >> report.txt
cat timeout-count.txt >> report.txt
```

第一步只筛行，不统计，也不写文件。你应该看到包含 `timeout` 的日志行；如果没有输出，先不要接 `wc`，改查大小写、拼写、文件路径和日志时间范围。第二步才统计行数，此时屏幕上应该只有一个数字。第三步把数字写到临时文件 `timeout-count.txt`，避免直接覆盖日报。第四步读回临时文件，确认它确实只有数字。最后两步才追加到 `report.txt`：先追加标签，再追加数字。

这个案例里，临时文件就是检查点。数字和日报分开，出错时只需要重算数字，不会破坏已有报告。如果你把命令写成 `grep -i "timeout" app.log | wc -l > report.txt`，它也能保存数字，但会把整份日报覆盖成一个数字；如果写成 `grep -i "timeout" app.log >> report.txt | wc -l`，管道右侧甚至可能拿不到你以为的输入，因为左侧 stdout 已经被追加重定向拿走了。

> [!NOTE]
> 练习钩子：以后写任何带

> 和
> >
> 的命令，都先在纸上或笔记里写三列：左侧 stdout、右侧 stdin、目标文件。只要其中一列写不出来，就不要急着记录下来。

再看一个更贴近排查现场的分流场景：你要把正常匹配结果保存给同事，同时把“文件不存在、权限不足”这类诊断留给自己排查。此时不要把所有东西粗暴混在一个文件里。正常结果和错误信息服务的是不同任务，应该先分开保存。

分流案例正常结果与错误日志分开复制

```bash
grep -n "ERROR" app.log missing.log > matches.txt 2> errors.txt
cat matches.txt
cat errors.txt
wc -l matches.txt errors.txt
```

这条命令故意同时给 `grep` 两个输入：存在的 `app.log` 和不存在的 `missing.log`。匹配到的错误行属于正常结果，会进入 `matches.txt`；关于 `missing.log` 的报错属于诊断，会进入 `errors.txt`。之后分别 `cat` 两个文件，是为了确认它们没有被混在一起。最后用 `wc -l` 看两个文件各有多少行，能快速判断“正常结果为空”还是“命令本身出了错”。

还要处理一种很常见的“空结果”场景。`matches.txt` 为空时，结论不能立刻写成“没有错误”。先同时看 `errors.txt`：如果错误文件有内容，说明命令运行过程中发生了诊断问题；如果错误文件也为空，才更接近“命令成功运行，只是没有匹配行”。这一步能避免把运行失败误当成数据结论。

# 练习 {#practice}
<!-- tag: diagnostic -->

<!-- practice-intro -->
<!-- end-practice-intro -->

**标准流**能区分 stdin、stdout、stderr。
**重定向**能判断 `>`、`>>`、`2>` 的写入目标。
**管道**能说明 `|` 传的是上一段 stdout，文件名需要另外处理。
**查找**能用 `find 起点 条件` 定位文件，并区分路径文本和文件内容。

<!-- practice-group: 一、标准流与符号 -->
<!-- quick-review: 检查点 正常结果、错误信息、输入来源不要混在一起。 -->

### 1 单选

**Q:** 1. 单选：命令的正常结果默认写到哪里？

- [ ] . stdout
- [ ] . stderr
- [ ] . 当前目录名

Answer: . stdout
Explain: stdout 是正常结果的默认出口，默认显示在屏幕上。

### 2 单选

**Q:** 2. 单选：文件不存在、权限不足这类诊断信息通常走哪条流？

- [ ] . stdout
- [ ] . stderr
- [ ] . stdin

Answer: . stderr
Explain: stderr 通常承载诊断和错误信息，所以可以和正常结果分开保存。

### 3 判断

**Q:** 3. 判断：`A | B` 会把 `A` 的 stdout 交给 `B` 的 stdin。

- [ ] 正确
- [ ] 错误

Answer: 正确
Explain: 管道把左边命令的 stdout 接到右边命令的 stdin。

### 4 判断

**Q:** 4. 判断：`cmd > out.txt` 默认只保存 stdout，stderr 仍可能显示在屏幕上。

- [ ] 正确
- [ ] 错误

Answer: 正确
Explain: 不额外写 2> 时，> 默认只改 stdout，不会自动带走 stderr。

<!-- end-practice-group -->

<!-- practice-group: 二、命令阅读与输出预测 -->
<!-- quick-review: 答题脚手架 每一步都问：读什么，吐出什么，下一步能不能接。 -->

### 5 单选

**Q:** 5. 单选：`grep "ERROR" app.log | wc -l > count.txt` 最后写进 `count.txt` 的是什么？

- [ ] . 原始日志全文
- [ ] . 匹配 ERROR 的行数
- [ ] . 文件名 app.log

Answer: . 匹配 ERROR 的行数
Explain: grep 先输出 ERROR 行，wc -l 把行数输出到 stdout，最后 > 写进 count.txt。

### 6 单选

**Q:** 6. 单选：`cmd >> report.txt` 表示什么？

- [ ] . 追加 stdout 到文件末尾
- [ ] . 删除文件再写入
- [ ] . 把 stderr 交给下一个程序

Answer: . 追加 stdout 到文件末尾
Explain: >> 表示把 stdout 追加到文件末尾，不覆盖已有内容。

### 7 命令阅读

**Q:** 7. 阅读：`grep "WARN" app.log >> report.txt 2> err.txt` 中 stdout 和 stderr 分别去了哪里？

Explain: stdout 是匹配到的 WARN 行，会追加到 report.txt 。stderr 是诊断或错误信息，会写入 err.txt 。这里 >> 和 2> 分别处理两条不同的输出流。

### 8 排序

**Q:** 8. 排序：把“统计 ERROR 行并保存”的链路排出来。

Explain: 参考顺序：读取 app.log → grep "ERROR" 筛出匹配行 → wc -l 统计这些行的数量 → > count.txt 把数量写入文件。要点是 wc 看到的是上一步输出的文本行，不直接读取原日志文件。

<!-- end-practice-group -->

<!-- practice-group: 三、find、路径文本与文件内容 -->
<!-- quick-review: 检查点 find 输出的是路径文本；是否读取文件内容，取决于下一步工具。 -->

### 9 单选

**Q:** 9. 单选：`find . -type f -name "*.md"` 里的 `.` 表示什么？

- [ ] . 从当前目录开始搜索
- [ ] . 匹配所有隐藏文件
- [ ] . 把结果写入名为点的文件

Answer: . 从当前目录开始搜索
Explain: find 的第一个位置参数通常是搜索起点，. 表示当前目录。

### 10 单选

**Q:** 10. 单选：`-name "*.md"` 中 `"*.md"` 常加引号的主要原因是？

- [ ] . 让 find 改成搜索文件内容
- [ ] . 防止当前 Shell 提前展开通配符
- [ ] . 让输出自动排序

Answer: . 防止当前 Shell 提前展开通配符
Explain: 给 *.md 加引号，是为了让 find 收到这个模式，而不是让当前 Shell 先展开。

### 11 判断

**Q:** 11. 判断：`find . -name "*.md" | grep TODO` 搜的是路径文本里有没有 `TODO`。

- [ ] 正确
- [ ] 错误

Answer: 正确
Explain: 这条命令里的 grep 搜的是 find 输出的路径字符串，不会打开每个 md 文件搜索 TODO 内容。

### 12 诊断

**Q:** 12. 诊断：你想找 Markdown 文件内容里的 `TODO`，为什么只写 `find . -name "*.md" | grep TODO` 不够？

Explain: find 输出的是文件路径文本，比如 ./notes/todo.md 。管道后的 grep TODO 看到的是这些路径行，所以它只判断路径字符串里有没有 TODO ，没有打开文件内容。要搜文件内容，可以先用 grep -R "TODO" . ，或在学会更稳的参数传递后，把 find 找到的路径交给能读取文件内容的命令。

<!-- end-practice-group -->

<!-- practice-group: 四、故障诊断与安全实操 -->
<!-- quick-review: 表达要求 按“错因 / 后果 / 修法”写，能说出最小验证命令。 -->

### 13 单选

**Q:** 13. 单选：`sort names.txt > names.txt` 最大的问题是什么？

- [ ] . sort 不能读取文本
- [ ] . > 会自动追加
- [ ] . 输入文件和输出文件相同，可能覆盖原内容

Answer: . 输入文件和输出文件相同，可能覆盖原内容
Explain: 把读取文件和输出文件写成同一个路径，有覆盖或截断风险。先写到新文件，再人工检查。

### 14 单选

**Q:** 14. 单选：运行 `grep "ERROR" missing.log > out.txt` 后，错误还在屏幕上，最可能的原因是？

- [ ] . grep 没有输出能力
- [ ] . > 没有重定向 stderr
- [ ] . out.txt 一定是目录

Answer: . > 没有重定向 stderr
Explain: > 只处理 stdout。错误信息走 stderr，所以还会显示，除非单独写 2> 或明确合并。

### 15 场景

**Q:** 15. 场景：你要统计 `app.log` 中 `ERROR` 行数并保存。写出一个安全流程。

Explain: 一个稳妥流程是：先运行 grep "ERROR" app.log ，确认匹配行正确；再运行 grep "ERROR" app.log | wc -l ，确认数字符合预期；最后运行 grep "ERROR" app.log | wc -l > error-count.txt ，写入新文件；再用 cat error-count.txt 或 less error-count.txt 检查内容。不要把输出文件写成原始日志文件。

### 16 综合

**Q:** 16. 费曼题：向同学解释“管道和重定向各自改了哪条流”。

Explain: 答案要能讲出四点：管道把左边正常输出交给右边输入，不会自动传文件内容或错误信息； > 是写入，常见行为是覆盖目标文件，所以先写新文件更稳； find 输出的是路径文本，后续命令看到的未必是文件内容；stderr 需要用 2> 等方式单独处理。一个合格例子可以围绕 grep "ERROR" app.log | wc -l > count.txt 展开。
