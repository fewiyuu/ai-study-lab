Shell 脚本 · 可执行文件

把命令变成脚本：让重复操作可检查、可复用、可交给别人运行

本块学习 Shell 脚本的最小闭环：写一个脚本文件，用 shebang 指定解释器，接收参数，引用变量，检查退出状态，再通过权限和路径正确运行它。脚本不是把终端历史复制进文件就结束，它还要经得住不同路径、不同参数和失败状态。

学完你应该能做什么

  1. 解释脚本、解释器、shebang 的关系。
  2. 读懂 $1、$@、$?、变量引用。
  3. 区分 bash script.sh 与 ./script.sh。
  4. 诊断 Permission denied、bad interpreter、变量未加引号等问题。

概念地图

Shell 脚本本质上是文本文件，里面写着一组 Shell 命令。运行脚本时，需要某个 Shell 或解释器读取这个文件并逐行执行。直接运行脚本与显式用 bash script.sh 运行，差别在于是否依赖文件的可执行权限和 shebang。

脚本把前几节的知识集中到一起：路径和当前目录会影响文件查找；引号会影响参数边界；标准流会影响输入输出；退出状态会影响后续分支。你写脚本时，不是在脱离 Shell 规则，而是在把 Shell 规则保存成可复用流程。

脚本文本→解释器→参数与变量→命令执行→退出状态
script保存命令的文本文件，用于重复执行。它可以被人审查，也可以被机器反复运行。
shebang第一行 #!...，告诉系统直接运行脚本时用哪个解释器。
exit status命令退出码，通常 0 表示成功，非 0 表示失败。脚本用它判断下一步是否继续。
executable bit执行权限，决定文件能否被当作程序直接运行。没有它时，bash script.sh 可能能跑，./script.sh 可能失败。

判断句：脚本能否可靠运行，不只看文件内容，还要看启动方式、解释器、权限、参数、当前目录和失败状态。

1. 脚本是什么：把一串命令写进文件

如果你每天都要执行同一组命令，比如进入项目、创建目录、统计日志、保存报告，就不该每次手敲。脚本把这些步骤写成文件，让操作变得可复用，也让别人能审查每一步。

脚本不是魔法，它仍然遵守 Shell 命令的解析规则：空格分隔参数，引号保护空格，当前工作目录影响相对路径，外部程序通过 PATH 查找。前面三块学过的内容全部会在脚本里再次出现。一个在终端里偶然能跑的命令，写进脚本后可能因为当前目录、变量值、权限、解释器不同而失败。

好脚本通常从“小而可检查”开始：先写出最小命令，再把硬编码文件名改成参数，再给关键路径加引号，最后处理失败情况。不要一开始就写一长串复杂流程；脚本的优势是可读、可审查、可重复，而不是把风险藏起来。

正确示例最小脚本
#!/usr/bin/env bash
echo "开始统计"
wc -l app.log
echo "统计完成"

你现在应该能做什么：把脚本理解成“可保存、可重复运行的一组命令”，而不是一种全新语言。

2. shebang：直接运行脚本时需要解释器线索

文件第一行如果是 #!/usr/bin/env bash，系统直接执行脚本时会用 bash 解释它。/usr/bin/env bash 会从 PATH 中寻找 bash，比写死 /bin/bash 在不同系统上更灵活。shebang 只在“把脚本当可执行文件运行”时关键；如果你显式写 bash script.sh，你已经告诉系统用 bash 解释。

如果你运行 bash script.sh，脚本文件只需要可读；如果运行 ./script.sh，系统需要执行权限和 shebang 来决定如何解释脚本文本。Windows、Git Bash、WSL、macOS/Linux 在换行符和路径上也可能带来差异。

bad interpreter 常常和 shebang 有关。第一行指向不存在的解释器，会失败；文件用 Windows CRLF 换行时，解释器路径末尾可能多出不可见的 \r，也会出现看似奇怪的解释器错误。遇到这类问题，先看第一行，再看换行格式。

逐行拆解两种运行方式
bash script.sh      # 显式让 bash 读取脚本
./script.sh         # 把脚本当可执行文件运行，需要执行权限和 shebang

你现在应该能做什么：解释为什么一个脚本用 bash script.sh 能跑，./script.sh 却报错。

3. 变量与引用：赋值不能乱加空格，使用要常加引号

Shell 变量赋值写作 name=value，等号两边不能有空格。name = value 会被 Shell 当成运行名为 name 的命令，并传入 = 和 value 两个参数。这是 Shell 语法和很多编程语言最不一样的地方之一。

引用变量用 $name 或 ${name}。路径和文件名变量最好写成 "$name"，因为变量值可能包含空格、通配符或为空。未加引号是脚本里非常常见的隐患：测试数据简单时能跑，遇到真实文件名就炸。

双引号不是多余装饰，它保留“这是一个参数”的边界。假设 log_file="app log.txt"，写 grep ERROR $log_file 时，Shell 会把变量展开后的空格当作分隔符；写 grep ERROR "$log_file" 时，文件名仍作为一个参数传给 grep。这和前面学过的路径引号是同一件事。

正确示例变量赋值与安全引用
log_file="app log.txt"
grep "ERROR" "$log_file"
错误示例赋值空格与未加引号
log_file = app.log
grep ERROR $log_file

你现在应该能做什么：发现变量赋值空格错误，并解释为什么引用变量时常用双引号。

4. 脚本参数：让脚本处理不同输入

脚本参数来自运行脚本时后面的词。$1 是第一个参数，$2 是第二个参数，$@ 表示所有参数。参数让脚本从“只处理固定文件”变成“可以处理用户指定文件”。

参数同样要考虑引号。"$1" 保留第一个参数的完整边界；"$@" 能保留每个参数各自的边界，常用于把所有参数转交给另一个命令。不要把 $@ 和 "$@" 混为一谈；后者更能保留每个原始参数的边界。

脚本还应该检查参数是否存在。用户没传文件名时，$1 可能为空；如果你继续执行 grep ERROR "$1"，得到的错误会很绕。一个基本脚本通常会在开头检查参数数量或文件是否存在，并给出清晰提示。

正确示例接收日志文件参数
#!/usr/bin/env bash
log_file="$1"
grep "ERROR" "$log_file" | wc -l

你现在应该能做什么：读懂一个脚本如何从命令行接收输入，并知道为什么 $1 要加引号。

5. 退出状态：命令是否成功的机器可读信号

Shell 里命令结束后会留下退出状态，$? 表示上一条命令的退出码。约定上 0 表示成功，非 0 表示失败。屏幕没有报错不一定代表逻辑成功，脚本要靠退出状态做判断。

if command; then ... fi 会根据命令退出状态分支。常见写法是先检查文件是否存在，再运行处理逻辑；或者让关键命令失败时立即停止。后续可以学习 set -euo pipefail，但初学阶段先理解退出状态本身。

退出状态不是“输出内容”。grep -q 可以不打印匹配行，只用退出状态告诉你有没有匹配。脚本自动化时，这种机器可读信号比肉眼看屏幕更可靠。你写脚本时要区分：stdout 给人或下游程序看，退出状态给 Shell 的控制流看。

逐行拆解用退出状态判断 grep 是否匹配
if grep -q "ERROR" "$log_file"; then
  echo "发现错误"
else
  echo "没有匹配到 ERROR"
fi

你现在应该能做什么：解释为什么脚本不能只看输出文本，还要看命令是否成功。

6. 可执行文件、权限与路径

要用 ./script.sh 直接运行脚本，文件需要执行权限。常见命令是 chmod +x script.sh。如果没有执行权限，系统会报 Permission denied；这和文件不存在不是一类问题。

直接输入 script.sh 还涉及 PATH。当前目录通常不在 PATH 里，所以要写 ./script.sh。如果希望全局运行脚本，可以把它放到 PATH 中的目录，或把脚本所在目录加入 PATH，但要考虑命名冲突和安全风险。

还有一个路径边界：脚本里的相对路径默认从“运行脚本时的当前工作目录”出发，不一定从脚本所在目录出发。你在项目根目录运行脚本和在子目录运行脚本，同一段 data/input.txt 可能指向不同位置。真实脚本常需要显式处理工作目录。

正确示例赋予权限并直接运行
chmod +x script.sh
./script.sh app.log

你现在应该能做什么：区分“没有执行权限”“当前目录没进 PATH”“解释器路径错”三类脚本运行失败。

7. 常见错误与修法

Permission denied

文件存在但没有执行权限。错因是直接执行需要 executable bit；后果是系统拒绝运行；修法是 chmod +x script.sh，或用 bash script.sh 显式解释。

bad interpreter

shebang 指向不存在的解释器，或 Windows CRLF 换行导致路径异常。修法：检查第一行、解释器路径和换行格式。

变量未加引号

含空格路径被拆成多个参数。错因是变量展开后再次发生词切分；后果是文件名被拆碎；修法：写 "$var"、"$1"、"$@"。

赋值等号有空格

Shell 会把左边当命令。错因是 name = value 不是赋值语法；修法：name=value。

逐行拆解检查参数和文件是否存在
#!/usr/bin/env bash
log_file="$1"

if [ -z "$log_file" ]; then
  echo "用法：$0 app.log" >&2
  exit 1
fi

if [ ! -f "$log_file" ]; then
  echo "文件不存在：$log_file" >&2
  exit 1
fi

grep "ERROR" "$log_file" | wc -l
错误示例未检查参数且未引用变量
log_file=$1
grep ERROR $log_file | wc -l
# 参数为空、含空格、含通配符时都可能出问题

你现在应该能做什么：把脚本错误写成“错因、后果、修法”，并知道什么时候检查参数、什么时候检查文件、什么时候检查权限。

8. 完整知识链路

脚本把命令保存为文本；shebang 告诉系统直接运行时该用哪个解释器；变量保存中间值，参数让脚本接收外部输入；引号保护参数边界；命令退出状态告诉脚本成功或失败；执行权限和路径决定脚本能否像程序一样运行。写脚本时要追踪两条线：数据线是参数、变量、文件和标准流；控制线是命令顺序、条件判断和退出状态。

一个脚本从“能跑”到“可靠”通常要经历几步：先确认用哪个解释器；再确认用户传了必要参数；再确认路径和文件存在；再用引号保护变量和参数；然后运行核心命令；最后根据退出状态决定继续、报错还是退出。每一步都对应一个常见故障点。

真正的脚本能力不是把命令写进文件，而是把隐含假设显式写出来。假设输入文件存在，就检查它；假设参数只有一个，就检查参数数量；假设失败要停止，就用退出状态控制流程。这样脚本才适合交给别人运行。

9. 真实场景与排查手册

Shell 脚本不是把终端历史复制进文件就结束。脚本运行时有解释器、参数、环境变量、退出状态和当前工作目录。每一项都可能改变脚本行为。

shebang 决定用哪个解释器运行脚本，执行权限决定能不能直接 ./script.sh。没有执行权限时，可以用 bash script.sh 运行，但这和直接执行不是完全同一件事。

真实脚本要对失败敏感。某一步失败后还继续执行，可能把错误结果写入后续文件。理解退出状态、set -e 的边界和参数引用，是让脚本可靠的基础。

忘记引用变量

变量里有空格时，未加引号会被拆成多个参数。路径变量优先写成 "$path"。

脚本找错目录

脚本的相对路径从运行时工作目录开始，不一定是脚本所在目录。需要时显式定位脚本目录。

权限和解释器混淆

Permission denied 可能是没执行权限；bad interpreter 可能是 shebang 路径或换行符问题。

可靠写法

先处理参数和目录，再执行核心命令，最后检查退出状态和输出文件。
