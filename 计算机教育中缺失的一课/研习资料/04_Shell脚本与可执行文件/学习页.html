读完本页应该能做什么

  • 区分 bash script.sh 和 ./script.sh，并说出它们分别先检查什么。

1. 学习目标：让命令链可以重复运行

学习目标

前三页已经让你能在终端里跑单条命令、接管道、保存结果。脚本做的第一件事很朴素：把一条已经验证过的命令链保存成文件，让它下次还能按同样的顺序执行。问题也随之变多：脚本文本由谁解释，参数从哪里来，文件路径相对哪里，失败时脚本会不会继续往下跑。

本页用一个日志统计脚本贯穿。重点放在脚本从“文件”到“程序”的运行条件，以及常见报错该拆成哪些检查项。

贯穿样本统计日志里的 ERROR 行数
app.log:
INFO start
ERROR bad input
WARN retry
ERROR timeout

目标：
count-errors.sh app.log  输出 2
这一页先看解释器、参数、引号、退出状态、权限和当前工作目录。循环、函数、数组和复杂开关先放到后面。

2. 主线：脚本运行时同时有数据线和控制线

脚本主线

把命令写进文件以后，Shell 的规则仍然生效。脚本会按空格切参数，按引号保留边界，按当前工作目录解析相对路径，用 stdout、stderr 和退出状态表达结果。写脚本时同时看两条线：数据线负责“参数、变量、文件、标准流”；控制线负责“命令是否成功、下一步是否执行、脚本用哪个退出码结束”。

环节 要问的问题 常用写法 错了会怎样
解释器 谁来读这份文本？ #!/usr/bin/env bash 或 bash script.sh bad interpreter 或用错 Shell
参数 用户给了哪些输入？ $#、$1、"$@" 空参数、路径被拆开、误处理默认文件
变量 中间值有没有保住边界？ log_file="$1"、"$log_file" 空格路径变成多个参数，通配符被展开
退出状态 上一条命令成功了吗？ if command; then ... fi、exit 1 失败后继续执行，生成误导性结果
执行入口 脚本是直接执行，还是交给 Bash 读？ chmod +x script.sh、./script.sh Permission denied、找不到命令

读脚本时，先核对这几个问题：这个脚本期待几个参数？参数有没有被引用？失败信息去了 stdout 还是 stderr？脚本失败时会用哪个退出码？它从哪个目录找文件？

3. 脚本文本：先把已验证命令保存下来

脚本文本

Shell 脚本通常就是一个文本文件，里面写着要让 Shell 执行的命令。最小脚本可以很短，但它仍然需要被某个解释器读取。下面这个脚本只做一件事：统计日志里包含 ERROR 的行数。它把前一单元的命令链 grep ... | wc -l 收进文件，同时把文件名变成参数。

核心脚本带参数检查的日志统计脚本
#!/usr/bin/env bash

if [ "$#" -ne 1 ]; then
  echo "Usage: $0 LOG_FILE" >&2
  exit 2
fi

log_file="$1"
if [ ! -f "$log_file" ]; then
  echo "not a file: $log_file" >&2
  exit 1
fi

grep -c "ERROR" "$log_file"

先看它的输入输出。用户运行脚本时传入一个日志文件路径。脚本先检查参数数量，再检查这个路径是否指向普通文件。检查失败时，它把提示写到 stderr 并非 0 退出；检查通过后，grep -c 输出匹配行数。这个脚本不负责改日志，也不负责生成报告，它只回答一个可检查的问题。

代码片段 读什么 写什么 失败时症状
#!/usr/bin/env bash PATH 中的 bash 选择解释器 找不到解释器或换行符异常
[ "$#" -ne 1 ] 参数数量 必要时 stderr + exit 2 用户没传文件却继续运行
log_file="$1" 第一个参数 变量 log_file 后面没有文件路径可用
[ ! -f "$log_file" ] 文件系统 必要时 stderr + exit 1 把不存在的路径交给 grep
grep -c "ERROR" "$log_file" 日志文件内容 stdout 输出行数 统计结果为空或 grep 报错

4. 两种运行方式：显式解释器和直接执行

shebang / chmod

同一个脚本常见有两种启动方式：bash count-errors.sh app.log 和 ./count-errors.sh app.log。第一种显式启动 Bash，让 Bash 读取脚本文本；第二种把脚本当成可执行文件交给系统启动。第二种会多检查执行权限，并需要第一行 shebang 告诉系统该用哪个解释器。

运行方式先用 bash 验证，再直接执行
bash count-errors.sh app.log
chmod +x count-errors.sh
./count-errors.sh app.log
启动方式 谁选解释器 需要执行权限吗 看不看 shebang 适合检查什么
bash script.sh 你在命令里指定 bash 通常只需要文件可读 不会依赖它来选解释器 脚本文本是否能被 Bash 解释
./script.sh 系统按 shebang 启动解释器 需要 需要 权限、shebang、换行格式、路径入口

排错时先分层。bash script.sh 能跑而 ./script.sh 失败，脚本主体很可能不是第一嫌疑；先看 chmod +x、shebang 和 CRLF。两种方式都失败，再回头看语法、变量、参数和文件路径。

5. 变量与参数：保住“一个参数”的边界

变量 / 参数

Shell 变量赋值写作 name=value，等号两边不能有空格。引用变量时，路径、文件名、用户输入通常写成 "$name"。原因来自第一课：Shell 会先展开变量，再做词切分和通配符展开。没有引号时，my log.txt 会被拆成 my 和 log.txt 两个词。

错误示例赋值空格和未引用参数
log_file = "$1"
grep "ERROR" $log_file
正确示例赋值和引用参数
log_file="$1"
grep "ERROR" "$log_file"
写法 含义 常见用途 容易踩坑
$0 脚本被调用时的名字 写 Usage: $0 LOG_FILE 它不一定是绝对路径
$1、$2 第 1、2 个位置参数 接收用户传入文件名 不加引号会丢参数边界
$# 参数数量 检查用户是否传够参数 不检查就会把空参数传给命令
$@ 所有位置参数 转交参数给另一个命令 裸用会经历词切分
"$@" 保留每个参数的边界 包装命令或转发参数 和 "$*" 行为不同，后面再细讲
${name} 清楚标出变量名边界 ${prefix}_report.txt 变量名和后缀粘在一起时需要它

引号的作用是保住参数边界。变量展开后仍然要作为同一个参数时，就把它包起来；路径、文件名和用户输入先加引号。只有你明确想让 Shell 拆词或展开通配符时，才放开引号。

6. 退出状态：脚本用它决定下一步

exit status

命令结束后会留下一个退出状态。通常 0 表示成功，非 0 表示失败。stdout 是给人或下游程序看的内容；退出状态是给 Shell 控制流看的信号。脚本自动化时，屏幕上有没有文字并不够，下一步该不该执行要看退出状态。

控制流用 grep -q 的退出状态分支
if grep -q "ERROR" "$log_file"; then
  echo "found ERROR"
else
  echo "no ERROR"
fi

grep -q 不打印匹配行，它用退出状态告诉脚本有没有匹配。这个模式和人工读屏幕不同：脚本只需要一个机器可读信号，然后进入对应分支。你也可以显式 exit 1 或 exit 2 结束脚本，让调用者知道失败类型。

场景 stdout stderr 退出状态 脚本下一步
日志中有 ERROR found ERROR 通常为空 0 进入 then 分支
日志中没有 ERROR no ERROR 通常为空 非 0 来自 grep -q 进入 else 分支
文件路径不存在 可能为空 grep 的诊断信息 非 0 应该先处理输入错误
下一单元会继续讲 set -e、pipefail 和函数返回码。这里先把“命令是否成功”这件事看稳。

7. 运行账本：把脚本当成状态流来看

运行账本

一个脚本从被调用到退出，通常只做四类动作：读命令行参数，读文件系统，写 stdout 或 stderr，最后写退出状态。把这四类动作分开看，很多报错就会变得直接。比如 ./count-errors.sh app.log 报 Permission denied，问题在启动层；bash count-errors.sh 能跑但结果不对，问题通常在参数、路径或命令内容。

步骤 读什么 写什么 错了会看到什么
启动 执行权限、shebang、PATH 启动解释器 Permission denied 或 bad interpreter
收参 $#、$1、$@ 脚本变量 空参数继续跑，或路径被拆开
读文件 当前工作目录、文件系统 给下游命令一个完整路径 找错文件，或 not a file
输出结果 日志内容 stdout 的统计值 数字不对，或 stderr 混进结果
结束 最后一条命令的状态 脚本退出码 调用者误判成功或失败

这个账本的用法很简单：先看读到了什么，再看写到了哪里，最后看哪一步该失败却没有失败。后面的故障面板和练习都按这个顺序排。

8. 追踪脚本：从调用到退出状态

状态追踪

现在把前面的概念串起来。一个脚本运行时会经历解释器选择、参数检查、文件检查、核心命令和退出状态返回。下面的追踪器用同一份 count-errors.sh 说明每一步读取什么、写出什么、失败时该看哪里。

调用脚本

bash count-errors.sh app.log 由你显式启动 Bash；./count-errors.sh app.log 由系统检查执行权限并按 shebang 找解释器。

先查：两种方式是否都能运行。如果只有直接执行失败，先看权限和 shebang。

检查参数

[ "$#" -ne 1 ] 读取参数数量。参数数量不对时，脚本把用法写到 stderr，并用 exit 2 结束。

先查：用户有没有传文件名，有没有把带空格路径放进引号。

检查文件

[ ! -f "$log_file" ] 读取文件系统状态。路径不存在或不是普通文件时，脚本写错误信息并用 exit 1 结束。

先查：当前工作目录是否和你想的一样，路径是否需要绝对路径。

执行 grep

grep -c "ERROR" "$log_file" 读取日志内容，把匹配行数写到 stdout。变量加引号保证整个路径作为一个参数交给 grep。

先查：统计结果异常时，回到 Unit 02/03 的文本工具和管道检查。

返回状态

脚本最终返回最后一条命令或显式 exit 的状态。调用者可以用 $?、if ./count-errors.sh app.log; then ... 或 CI 任务读取这个状态。

先查：失败路径是否用了非 0 退出，错误消息是否写到了 stderr。

阶段 读 / 依赖 写 / 交给谁 错误症状 下一步检查
调用 命令行、执行权限、shebang 启动解释器 Permission denied / bad interpreter ls -l、第一行、换行格式
参数检查 $#、$0 stderr 用法提示，exit 2 空参数继续运行 检查调用方式和参数数量
文件检查 $1、当前工作目录、文件系统 stderr 文件错误，exit 1 找不到文件或目录错位 pwd、ls、绝对路径
统计 日志文件内容 stdout 输出数字 数字不符合预期 单独跑 grep "ERROR" "$log_file"
结束 上一条命令退出状态 脚本退出状态 调用方误判成功或失败 检查 exit 和最后一条命令

9. 故障面板：把报错拆回运行条件

故障诊断

脚本报错时，别把所有问题都归结为“脚本写错”。先判断错误出现在启动层、参数层、路径层还是命令层。下面几个故障都很常见，且各自对应不同的修法。

$ ./count-errors.sh app.log
bash: ./count-errors.sh: Permission denied
症状 层级 先查 常用修法
Permission denied 启动层 ls -l script.sh chmod +x script.sh 或显式 bash script.sh
bad interpreter 解释器层 shebang、解释器路径、CRLF 修第一行，转换换行格式
command not found 出现在赋值行 语法层 等号两边有没有空格 name=value
带空格文件名被拆开 参数层 变量引用是否加双引号 "$var"、"$1"、"$@"
脚本在不同目录表现不同 路径层 pwd 和相对路径起点 传绝对路径，或脚本中明确工作目录

10. 可执行文件、PATH 与当前工作目录

权限 / 路径

chmod +x script.sh 只解决“这个文件能不能被直接执行”。它不负责把当前目录加入 PATH，也不保证脚本里的相对路径正确。直接输入 script.sh 时，Shell 会去 PATH 列出的目录找命令；当前目录通常不在里面，所以本目录脚本常写成 ./script.sh。

路径检查区分当前目录、PATH 和脚本参数
pwd
ls -l count-errors.sh
echo "$PATH"
./count-errors.sh app.log
count-errors.sh app.log

还有一个容易误判的点：脚本里的 app.log 或 data/input.txt 相对的是运行脚本时的当前工作目录。你在项目根目录运行和在子目录运行，同一个相对路径可能指向不同地方。本页先要求你能发现这个差异；把脚本定位到自身目录、处理符号链接和安装到全局路径，是后续环境与工具链单元要继续讲的内容。

问题 涉及对象 检查命令 判断
能不能直接执行 文件执行权限 ls -l script.sh 看权限位里有没有 x
直接执行用哪个解释器 shebang 第一行 head -n 1 script.sh 看是否指向可用解释器
为什么 script.sh 找不到 PATH echo "$PATH" 当前目录通常不在搜索路径里
脚本为什么找错输入文件 当前工作目录 pwd 相对路径从这里出发

11. 交付前检查：从命令链到脚本文件

操作桥

把脚本交给别人运行前，至少走一遍可复现检查。先在终端里验证核心命令，再写入脚本；先用 bash script.sh 验证脚本文本，再加 chmod +x 检查直接执行；先测试成功路径，再测试缺参数、文件不存在、带空格路径这些失败路径。

交付流程从单条命令到可执行脚本
printf 'INFO start\nERROR bad input\nWARN retry\nERROR timeout\n' > app.log
grep -c "ERROR" app.log

bash count-errors.sh app.log
bash count-errors.sh
bash count-errors.sh "missing file.log"

chmod +x count-errors.sh
./count-errors.sh app.log
./count-errors.sh "my log.txt"
检查 要看到什么 失败时回到哪一节
核心命令单跑 grep -c 输出预期数字 Unit 02/03 文本工具和管道
显式 Bash 运行 脚本文本可以被 Bash 解释 脚本文本、变量、参数
缺参数测试 stderr 出现用法，退出非 0 参数数量检查
文件不存在测试 stderr 出现文件错误，退出非 0 文件检查和当前工作目录
直接执行 ./script.sh 能运行 权限、shebang、换行格式
带空格路径 路径作为一个参数被处理 变量和参数引用

ShellCheck 可以在这之后帮你抓很多脚本问题，例如未引用变量、未使用变量、可疑测试写法。它不能替你决定脚本应该处理哪些失败路径，但能帮你发现一批机械错误。先用本页的读写账本理解问题，再把 linter 当成辅助检查工具。

辅助检查用 ShellCheck 抓机械错误
shellcheck count-errors.sh
# 常见提示示例：
# Double quote to prevent globbing and word splitting.

把 ShellCheck 的提示放进这条检查链里看：它指出“变量需要双引号”时，你要回到参数边界；它指出某个变量未使用时，你要检查脚本是否少了一步；它指出可疑测试表达式时，你要回到退出状态和条件判断。失败后果也要说清楚：未引用变量会把带空格文件名拆开，错误测试会让脚本走错分支，未处理的失败路径会让调用者误以为脚本成功。

提示类型 回到本页哪一块 先做的小检查
变量未加引号 变量与参数 用带空格文件名重跑脚本
条件判断可疑 退出状态 分别测试成功路径和失败路径
变量未使用 脚本文本 检查读写账本里是否缺了一步
练习钩子：以后看到 linter 提示，不要只改到提示消失。写一句“这个提示对应哪条运行条件”，比如“保护参数边界”或“失败路径没有退出”。这样你才是在修脚本，而不是在安静地消提示。

错因：变量未加双引号，展开后的空格触发词切分。后果：my log.txt 被拆成多个参数，grep 会找错文件。修法：写成 grep "ERROR" "$log_file"，并在调用脚本时也给带空格路径加引号。

参考答案 / 评分要点

[ "$#" -ne 1 ] 检查参数数量是否不等于 1；echo ... >&2 把用法提示写到 stderr，避免混进正常输出；exit 2 用非 0 退出码结束脚本，告诉调用者这是用法错误。

18. 费曼题：向零基础同学解释一个 Shell 脚本从文件到运行成功要满足哪些条件。

建议包含：脚本文本、解释器、shebang、参数、变量引号、退出状态、执行权限、当前工作目录。
