# 1. 学习目标：让命令链可以重复运行 {#goals}
<!-- tag: 学习目标 -->

前三页已经让你能在终端里跑单条命令、接管道、保存结果。脚本做的第一件事很朴素：把一条已经验证过的命令链保存成文件，让它下次还能按同样的顺序执行。问题也随之变多：脚本文本由谁解释，参数从哪里来，文件路径相对哪里，失败时脚本会不会继续往下跑。

本页用一个日志统计脚本贯穿。重点放在脚本从“文件”到“程序”的运行条件，以及常见报错该拆成哪些检查项。

贯穿样本统计日志里的 ERROR 行数复制

```text
app.log:
INFO start
ERROR bad input
WARN retry
ERROR timeout

目标：
count-errors.sh app.log 输出 2
```

> [!NOTE]
> 这一页先看解释器、参数、引号、退出状态、权限和当前工作目录。循环、函数、数组和复杂开关先放到后面。

# 2. 主线：脚本运行时同时有数据线和控制线 {#overview}
<!-- tag: 脚本主线 -->

把命令写进文件以后，Shell 的规则仍然生效。脚本会按空格切参数，按引号保留边界，按当前工作目录解析相对路径，用 stdout、stderr 和退出状态表达结果。写脚本时同时看两条线：数据线负责“参数、变量、文件、标准流”；控制线负责“命令是否成功、下一步是否执行、脚本用哪个退出码结束”。

# 5. 变量与参数：保住“一个参数”的边界 {#vars-params}
<!-- tag: 变量 / 参数 -->

Shell 变量赋值写作 `name=value`，等号两边不能有空格。引用变量时，路径、文件名、用户输入通常写成 `"$name"`。原因来自第一课：Shell 会先展开变量，再做词切分和通配符展开。没有引号时，`my log.txt` 会被拆成 `my` 和 `log.txt` 两个词。

错误示例赋值空格和未引用参数复制

```bash
log_file = "$1"
grep "ERROR" $log_file
```

正确示例赋值和引用参数复制

```bash
log_file="$1"
grep "ERROR" "$log_file"
```

# 6. 退出状态：脚本用它决定下一步 {#exit-status}
<!-- tag: exit status -->

命令结束后会留下一个退出状态。通常 `0` 表示成功，非 `0` 表示失败。stdout 是给人或下游程序看的内容；退出状态是给 Shell 控制流看的信号。脚本自动化时，屏幕上有没有文字并不够，下一步该不该执行要看退出状态。

控制流用 grep -q 的退出状态分支复制

```bash
if grep -q "ERROR" "$log_file"; then
 echo "found ERROR"
else
 echo "no ERROR"
fi
```

`grep -q` 不打印匹配行，它用退出状态告诉脚本有没有匹配。这个模式和人工读屏幕不同：脚本只需要一个机器可读信号，然后进入对应分支。你也可以显式 `exit 1` 或 `exit 2` 结束脚本，让调用者知道失败类型。

# 7. 运行账本：把脚本当成状态流来看 {#run-ledger}

一个脚本从被调用到退出，通常只做四类动作：读命令行参数，读文件系统，写 stdout 或 stderr，最后写退出状态。把这四类动作分开看，很多报错就会变得直接。比如 `./count-errors.sh app.log` 报 `Permission denied`，问题在启动层；`bash count-errors.sh` 能跑但结果不对，问题通常在参数、路径或命令内容。

# 10. 可执行文件、PATH 与当前工作目录 {#executable-path}
<!-- tag: 权限 / 路径 -->

`chmod +x script.sh` 只解决“这个文件能不能被直接执行”。它不负责把当前目录加入 `PATH`，也不保证脚本里的相对路径正确。直接输入 `script.sh` 时，Shell 会去 `PATH` 列出的目录找命令；当前目录通常不在里面，所以本目录脚本常写成 `./script.sh`。

路径检查区分当前目录、PATH 和脚本参数复制

```bash
pwd
ls -l count-errors.sh
echo "$PATH"
./count-errors.sh app.log
count-errors.sh app.log
```

还有一个容易误判的点：脚本里的 `app.log` 或 `data/input.txt` 相对的是运行脚本时的当前工作目录。你在项目根目录运行和在子目录运行，同一个相对路径可能指向不同地方。本页先要求你能发现这个差异；把脚本定位到自身目录、处理符号链接和安装到全局路径，是后续环境与工具链单元要继续讲的内容。

### 9 诊断

**Q:** 9. 诊断：脚本里写 `grep "ERROR" $log_file`，传入 `my log.txt` 后失败。按“错因 / 后果 / 修法”回答。

Explain: 错因：变量未加双引号，展开后的空格触发词切分。后果： my log.txt 被拆成多个参数， grep 会找错文件。修法：写成 grep "ERROR" "$log_file" ，并在调用脚本时也给带空格路径加引号。

### 12 单选

**Q:** 12. 单选：参数缺失时，脚本更合适的行为是？

- [ ] . 把用法提示写到 stderr，并 exit 非 0
- [ ] . 静默生成空报告

Answer: . 把用法提示写到 stderr，并 exit 非 0
Explain: 用法和错误提示应写到 stderr，并用非 0 退出码告诉调用者失败。

### 13 代码阅读

**Q:** 13. 阅读：解释 `[ "$#" -ne 1 ]`、`echo "Usage: $0 LOG_FILE" >&2`、`exit 2` 分别做什么。

Explain: [ "$#" -ne 1 ] 检查参数数量是否不等于 1； echo ... >&2 把用法提示写到 stderr，避免混进正常输出； exit 2 用非 0 退出码结束脚本，告诉调用者这是用法错误。

### 16 场景

**Q:** 16. 单选：脚本里写 `data/input.txt`，在项目根目录能跑，在子目录运行失败。先查什么？

### 17 综合

**Q:** 17. 场景：你要把 `grep -c "ERROR" app.log` 变成可交给别人运行的脚本。最小可靠流程是什么？

Explain: 先确认 grep -c "ERROR" app.log 输出正确；把文件名改成参数 log_file="$1" 并使用 "$log_file" ；用 $# 检查参数数量，用 [ -f "$log_file" ] 检查文件；错误信息写到 stderr 并非 0 退出；先运行 bash count-errors.sh app.log ，再 chmod +x 后运行 ./count-errors.sh app.log ；补测缺参数、不存在文件和带空格路径。

### 18 综合

**Q:** 18. 费曼题：向零基础同学解释一个 Shell 脚本从文件到运行成功要满足哪些条件。

Explain: 合格答案要说清：脚本是保存命令的文本文件，需要 Bash 等解释器读取；直接执行时要有执行权限和可用 shebang，显式 bash script.sh 则由命令指定解释器；参数从命令行进入脚本，路径变量要加引号；脚本用 stdout 输出正常结果，用 stderr 输出用法或错误；退出状态告诉调用者成功或失败；相对路径从当前工作目录出发。例子可以围绕 count-errors.sh app.log 展开。
