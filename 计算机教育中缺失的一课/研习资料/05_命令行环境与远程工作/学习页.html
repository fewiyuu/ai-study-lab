第一讲让你能运行程序；这一讲把视角扩大到“程序运行时看见了什么”。命令会收到参数、标准输入、环境变量；Shell 会维护作业和会话；远程机器通过 SSH 接入；长期效率来自可迁移的配置。

概念地图

命令行环境是一组围绕程序运行的上下文：参数告诉程序“这次要处理什么”，标准流告诉程序“从哪里读、往哪里写”，环境变量告诉程序“运行环境有什么配置”，会话工具决定“断线、切换窗口、后台任务时程序如何继续”。

以后看到一条命令，先不要急着背参数。先问四个问题：它在哪台机器运行？当前目录是什么？它继承了哪些环境变量？它的输入输出连接到终端、文件还是管道？这四个问题能解释大量“我明明照着敲了，结果不一样”的现象。

参数+标准流+环境变量→程序行为→本地或远程会话
环境变量进程启动时继承的一组键值配置，例如 PATH、HOME、EDITOR。它会影响程序默认行为，但不会自动修改程序参数。
作业 jobShell 管理的一条正在运行或暂停的命令。作业属于当前 Shell，会话断开后是否继续运行需要单独判断。
tmux终端复用器，把会话、窗口、面板从单个终端窗口中抽离出来。它适合保留交互式工作现场。
SSH安全登录远程机器的协议和工具，常用于服务器开发。登录后路径、用户、命令版本都属于远端。

1. 程序输入输出不只来自参数

初学者最容易把程序输入等同于参数，例如 ls -l project 中的 -l 和 project。但脚本和真实程序还会从标准输入读取内容，从文件系统读取配置，从环境变量读取默认行为，甚至从网络、数据库或用户交互中读取信息。

这一点解释了为什么同一条命令在两台机器上表现不同：参数一样，但环境变量、当前目录、配置文件、程序版本、权限和网络环境可能不同。排查问题时不能只盯命令文本，还要问“它运行时所处的环境是什么”。

进程启动后，能直接看见的上下文通常包括：参数数组、标准输入、标准输出、标准错误、环境变量、当前工作目录、用户身份和可访问的文件权限。命令行排错时，把这些上下文逐项列出来，比反复重敲命令更有效。

逐行拆解三类常见输入来源
grep "ERROR" app.log       # 参数：模式和文件名
grep "ERROR" < app.log     # stdin：从重定向读取文本
EDITOR=vim git commit      # 环境变量：告诉 git 默认编辑器

判断句：如果命令文本相同但结果不同，先比较当前目录、环境变量、输入来源和程序版本。你现在应该能做什么：读一条命令时，能区分它从参数、标准输入还是环境变量获得信息。

2. 环境变量：进程继承的键值配置

环境变量是 Shell 启动程序时交给它的一组字符串键值。PATH 决定外部命令搜索目录，HOME 指向用户家目录，EDITOR 常被 Git 等工具用来选择编辑器。环境变量不是 Shell 专属概念，任何进程都可以读取自己的环境。

Shell 变量和环境变量要分开。FOO=bar 只是在当前 Shell 里放了一个变量；export FOO=bar 才会让后续子进程继承它。很多脚本读不到变量，不是变量没有设置，而是没有被 export。反过来，FOO=bar command 是一次性环境，只影响紧跟着的那条命令。

排查环境变量要用能观察子进程环境的命令，例如 env、printenv，必要时再配合 type 或 which 看命令解析到哪里。PATH 的顺序也很重要：同名程序会优先使用前面目录里的版本。

正确示例查看、导出、临时设置环境变量
echo "$PATH"
export EDITOR=vim
EDITOR=nano git commit
PYTHONPATH=src python -m pytest
错误示例赋值语法被空格破坏
export EDITOR = vim
# Shell 会把 EDITOR、=、vim 当成不同词，语义不再是赋值
逐行拆解检查变量是否进入子进程
FOO=bar
env | grep FOO          # 通常看不到，因为没有 export
export FOO=bar
env | grep FOO          # 子进程环境里能看到 FOO=bar

判断句：程序读不到变量时，先问“它是在当前 Shell 里有值，还是已经进入子进程环境”。你现在应该能做什么：解释“为什么设置了环境变量后某个程序行为改变”，并能用临时环境变量做不留痕测试。

3. 作业控制：前台、后台、暂停与恢复

Shell 会把你启动的命令作为作业管理。前台作业占用终端输入，后台作业继续运行但不占用输入。Ctrl-C 通常发送中断信号，让前台程序终止；Ctrl-Z 暂停前台程序，把它交还给 Shell 管理。

jobs 查看作业，fg 把暂停或后台作业带回前台，bg 让暂停作业在后台继续。末尾加 & 可以直接后台启动。作业控制适合本地临时任务，不等同于生产级守护进程管理；断开终端后任务是否继续，要看进程和会话关系。

后台运行不是“安静运行”。后台作业仍然可能往 stdout 或 stderr 写内容，终端会被日志刷屏。真正的长任务一般要把输出写入日志文件，或者放进 tmux，再用 tail -f 查看日志。这样做的好处是任务、输出和交互界面彼此分开。

正确示例查看和恢复作业
sleep 100 &
jobs
fg %1
# Ctrl-Z 暂停后：
bg %1
正确示例后台任务写入日志
python train.py > train.log 2>&1 &
jobs
tail -f train.log

判断句：& 只改变前台/后台，不自动保存日志，也不保证断开终端后任务继续。你现在应该能做什么：当命令卡住或长时间运行时，知道何时中断、暂停、放后台或恢复前台。

4. tmux：让会话独立于终端窗口

普通终端窗口关闭后，里面的会话通常也结束。tmux 解决的是“我希望远程或本地的终端工作区可以断开后再接回”的问题。它提供会话、窗口、面板：会话像一个工作空间，窗口像标签页，面板像分屏。

tmux 的价值在远程机器上尤其明显。你通过 SSH 进服务器，启动训练、日志查看或长任务；网络断开时，如果任务在 tmux 会话里，之后可以重新 attach 回去继续查看。它不是替代脚本和服务管理的万能工具，但很适合交互式远程工作。

使用 tmux 时也要保存日志。tmux 保留的是会话现场，不是可靠的实验记录。训练脚本崩溃、服务器重启、磁盘写满时，屏幕上曾经出现过的信息未必还在。稳妥做法是：tmux 保留交互入口，日志文件保留事实记录，代码和参数写进脚本或 README。

逐行拆解tmux 最小工作流
tmux new -s work      # 新建名为 work 的会话
# 按 Ctrl-b 再 d     # detach，断开但保留会话
tmux ls              # 查看会话
tmux attach -t work   # 重新接回

判断句：tmux 解决会话断开问题，不解决代码错误、权限错误、路径错误和磁盘空间问题。你现在应该能做什么：说明为什么长时间远程工作常配 tmux，而不是把所有任务裸跑在一个 SSH 窗口里。

5. SSH：把本地终端接到远程机器

SSH 让你安全登录远程机器，在远端启动 Shell。命令看起来像 ssh user@host，但登录后你的当前目录、PATH、已安装程序、配置文件都属于远程机器，不是本地机器。很多“本地有这个文件，远程却找不到”的问题都来自这个边界没分清。

SSH 常配合密钥认证。私钥留在本机，公钥放到远程机器授权列表。权限过宽、主机名写错、用户名错、密钥没被加载，都会导致登录失败。排查时要看错误类型：连接不上、认证失败、登录后命令不存在，是三类不同问题。

远程执行命令时，引号也会影响执行位置。ssh host "hostname && pwd" 里的引号内容会交给远端 Shell 执行；scp local.txt host:~/ 里带 host: 的一端是远端路径。只要涉及两台机器，就要把本地端和远端端写清楚。

正确示例SSH 登录和远程执行命令
ssh user@example.com
ssh user@example.com "hostname && pwd"
scp report.txt user@example.com:~/reports/

判断句：SSH 成功后，~、pwd、PATH、python 都默认指向远端环境。你现在应该能做什么：登录远程机器后，能主动确认“我现在在哪台机器、哪个目录、哪个用户身份”。

6. dotfiles：把环境配置变成可迁移资产

Shell、编辑器、Git、tmux 等工具都有配置文件，很多以点开头，例如 .bashrc、.zshrc、.gitconfig、.tmux.conf。这些文件定义别名、提示符、默认编辑器、快捷键和插件加载方式。

配置文件能提高效率，也能制造“只在我机器上能跑”的隐性依赖。好的习惯是把 dotfiles 放入版本控制，写清依赖和安装方式，避免把密钥、token、私人路径直接提交。修改配置后要知道何时需要重新打开 Shell，何时可以 source ~/.bashrc 重新加载。

迁移 dotfiles 时，不要只追求“复制过去能用”。更重要的是能解释每一段配置依赖什么：某个 alias 调用了哪个程序，某段 PATH 是否写死了私人目录，某个 token 是否应该改放到系统密钥管理器或本机私有文件里。配置越强，越要能被审计。

正确示例查看和重新加载 Shell 配置
ls -a ~
echo 'alias ll="ls -lah"' >> ~/.bashrc
source ~/.bashrc

判断句：dotfiles 是可迁移配置，不是秘密仓库，也不应该悄悄依赖只有你机器上才存在的路径。你现在应该能做什么：把“我的命令行为什么长这样、某个别名从哪来”追踪到具体配置文件。

7. 常见错误与修法

环境变量没生效

可能只设置了 Shell 变量，没有 export 给子进程；也可能写错配置文件或未重新加载。修法：env | grep NAME、检查 export 和 source。如果是一次性命令，确认变量写在命令前面。

后台任务输出乱入终端

后台作业仍可能写 stdout/stderr。后果是终端无法正常输入，日志也容易丢。修法：重定向输出到日志文件，或者在 tmux 中运行并保留日志。

SSH 后找不到本地文件

登录后命令在远程机器运行。后果是你在远端找本机路径，或把文件复制到相反方向。修法：确认 hostname、pwd，必要时用 scp 或同步工具传文件。

配置污染新机器

dotfiles 里写死私人路径或 token。后果是新机器启动 Shell 变慢、报错，甚至泄漏凭据。修法：把秘密放环境变量或专用密钥管理，不提交敏感信息。

PATH 顺序导致命令版本错

同名命令可能存在多个版本。后果是本地测试用的是 A，远端运行用的是 B。修法：用 type command、which command、command --version 连起来检查。

把 tmux 当成服务管理器

tmux 适合保留交互会话，不负责开机自启、失败重启和依赖编排。需要稳定长期运行时，应考虑 systemd、任务队列或平台提供的作业系统。

8. 完整知识链路

命令行环境把“单条命令”扩展成“可长期工作的系统”。参数、stdin、环境变量共同决定程序输入；stdout/stderr 决定输出去向；作业控制决定当前终端如何管理运行中的程序；tmux 让工作会话可以断开和恢复；SSH 把同一套命令行能力带到远程机器；dotfiles 让你的偏好和默认行为可迁移。排查命令行环境问题时，先问：这条命令在哪台机器、哪个 Shell、哪个目录、带着哪些环境变量运行？

真实工作中，这些概念会同时出现：你 SSH 到服务器，在 tmux 里运行训练脚本，用临时环境变量选择配置，把 stdout/stderr 写入日志，再用 scp 或 rsync 把结果取回本地。如果其中一步出错，不要把所有问题都归因于“命令错了”。先定位层级：连接层、身份层、路径层、环境层、会话层、程序层。层级定位清楚以后，修法通常会变得很具体。

9. 真实场景与排查手册

SSH 登录后，你看到的是远端 Shell，不再是本机目录。命令在哪台机器执行、文件在哪台机器上、端口在哪边监听，是远程工作最常见的三组混淆。

tmux 或类似工具解决的是会话持久性：网络断开时远端任务还在。它不能替你解决权限、路径、环境变量或端口转发问题。

远程排查要分层：先确认能否连上，再确认身份和目录，再确认远端程序是否运行，最后确认端口、日志和文件同步。

本地远端混淆

scp、ssh、rsync 都涉及两台机器。路径前有没有 host:，决定文件在本地还是远端。

会话断开

长任务不要裸跑在普通 SSH 会话里。用 tmux、screen 或服务管理工具保留任务。

端口方向错

端口转发要说清本地端口、远端端口和访问方向，不能只说“转一下端口”。

稳定排查

连接 -> 身份 -> 目录 -> 进程 -> 日志 -> 端口，按层检查能减少来回猜。
