# 1. 学习目标：把命令放回运行环境里 {#goals}
<!-- tag: goals -->

前几页训练的是“这条命令怎么写”。这一页训练的是“这条命令在哪里、带着什么状态运行”。同一个 `python`，在本机和远端可能不是同一个文件；同一个脚本，在交互 Shell 和非交互 Shell 里可能读到不同配置；同一个长任务，裸跑在 SSH 窗口里和跑在 tmux 会话里，断网后的结果完全不同。

本页贯穿一个小场景：你有一台远端开发机 `devbox`，真实主机名是 `dev.example.org`，用户名是 `learner`，项目目录是 `~/work/app`。你要先确认本地环境，再登录远端，运行任务，把结果取回本地。所有例子都围绕这条线展开。

贯穿样本：本地与远端环境检查复制

```bash
echo "$SHELL"
echo "$PATH"
which python

ssh devbox 'hostname; pwd; echo "$SHELL"; which python'
```

**local shell**先知道本机当前 Shell、当前目录和命令解析结果。
**ssh alias**`devbox` 只是本机配置里的别名，需要解析成主机、用户和密钥。
**remote shell**登录后，`~`、`pwd`、`PATH` 都属于远端。
**session**长任务要放在能断开后重连的会话里，并把输出写到日志。
**copy back**结果文件要明确从哪台机器复制到哪台机器。

<!-- tag: state ledger -->

# 3. Shell 启动文件：改配置前先问“谁会读它” {#startup}
<!-- tag: startup files -->

Bash 的启动行为取决于 Shell 类型。登录 Shell 会读登录相关文件，例如 `~/.bash_profile`、`~/.bash_login` 或 `~/.profile`；普通交互 Bash 通常读 `~/.bashrc`。脚本运行时往往是非交互 Shell，不会自动拥有你交互终端里的别名和提示符配置。

# 4. PATH 与 export ：命令查找和环境继承是两件事 {#path-env}
<!-- tag: PATH / environment -->

`PATH` 是一串目录。你输入 `python`、`rg`、`ssh` 这类不带斜杠的命令时，Shell 会按 `PATH` 顺序查找可执行文件。目录排在前面，就更早命中。把 `$HOME/bin` 放在前面，意味着你自己的脚本可以覆盖系统里的同名命令；这很方便，也需要审慎。

变量继承是另一条线。`APP_ENV=dev` 只是在当前 Shell 里设置 Shell 变量；`export APP_ENV=dev` 才会让后续子进程看到它；`APP_ENV=dev ./run.sh` 是一次性环境，只影响紧跟着的那条命令。很多“我明明设置了变量，程序却读不到”的问题，根源就是少了 `export` 或者变量设置发生在另一台机器上。

# 5. dotfiles：可迁移配置，也可能是隐性依赖 {#dotfiles}
<!-- tag: dotfiles -->

dotfiles 是以点开头的配置文件，比如 `.bashrc`、`.zshrc`、`.gitconfig`、`.tmux.conf`、`.ssh/config`。它们能把常用别名、默认编辑器、提示符、tmux 快捷键和工具偏好带到新机器。把 dotfiles 放进版本控制是常见做法，但前提是你知道里面哪些内容能公开，哪些必须留在本机私有文件里。

好 dotfiles 的标准换机器后可解释、可恢复、不会泄漏秘密。不要提交私钥、token、生产账号、公司内网地址或只在你电脑上存在的绝对路径。alias 适合交互提速，脚本和团队文档应该写真实命令；否则别人复制你的脚本时，可能卡在一个只有你机器上才存在的别名上。

# 6. SSH 连接：登录以后执行位置已经换了 {#ssh}
<!-- tag: ssh -->

`ssh user@host` 做的事情可以拆成几步：本机发起连接，解析主机名，确认远端主机身份，使用密码或密钥完成用户认证，然后在远端启动 Shell。登录成功后，你敲的命令默认在远端执行。此时 `~` 是远端用户的家目录，`pwd` 是远端目录，`PATH` 是远端环境，`python` 也是远端解析结果。

密钥认证里，私钥留在本机，公钥放到远端授权位置。私钥不能发给别人，也不要提交进 dotfiles 仓库。第一次连接新主机时，客户端会记录主机指纹到 `known_hosts` 一类文件里；如果以后出现主机身份变化警告，不要直接删除记录继续连，先确认是不是服务器重装、主机迁移，还是中间人攻击风险。

示例：登录远端并确认执行位置复制

```bash
ssh learner@dev.example.org
hostname
whoami
pwd
echo "$PATH"
which python
```

示例：不进入交互 Shell，直接在远端执行检查命令复制

```bash
ssh devbox 'hostname; whoami; pwd; echo "$SHELL"; which python'
```

# 7. ~/.ssh/config ：把连接参数命名 {#ssh-config}
<!-- tag: ssh config -->

频繁登录同一台机器时，不要每次手打长命令。`~/.ssh/config` 可以把主机名、用户名、端口和密钥文件收进一个 Host 别名。这样 `ssh devbox` 就不只是一个短命令，它还是一份可审计的连接说明。

示例：devbox 的 SSH 配置复制

```text
Host devbox
 HostName dev.example.org
 User learner
 Port 22
 IdentityFile ~/.ssh/id_ed25519
```

读配置时按字段翻译：`Host devbox` 是本地别名；`HostName` 是真实主机名或 IP；`User` 是远端用户名；`Port` 是 SSH 端口；`IdentityFile` 是本机私钥路径。这个文件本身不包含私钥内容，但仍然可能暴露内部主机名和账号结构，公开 dotfiles 前要检查。

# 8. 远程文件：带 host: 的那一端是远端 {#remote-files}
<!-- tag: scp / sftp / rsync -->

远程工作最常见的混淆是文件在哪台机器。SSH 登录后，你在远端 Shell 里看不到本机的 `~/Downloads`；本机终端里也不会自动出现远端的 `~/work/app/result.txt`。文件要跨机器移动，需要明确方向。

示例：两种相反方向的 scp复制

```bash
# 本机 report.txt 复制到远端项目目录
scp report.txt devbox:~/work/app/

# 远端 result.txt 复制回本机当前目录
scp devbox:~/work/app/result.txt .
```

# 9. tmux 会话：让工作现场留在远端 {#tmux}
<!-- tag: tmux -->

tmux 是终端复用器。它把“会话”从单个终端窗口里分离出来：你可以在远端创建一个会话，里面有窗口和面板；网络断开后，tmux 会话仍留在远端机器上；重新 SSH 后可以 attach 回去。它解决的是会话持久性，不负责修复程序错误、权限错误、磁盘满、服务器重启或任务自动重启。

示例：远端长任务最小 tmux 流程复制

```bash
ssh devbox
cd ~/work/app
tmux new -s work

./run-job.sh > logs/run.log 2>&1
# 按 Ctrl-b，然后按 d，detach 离开会话

tmux ls
tmux attach -t work
```

tmux 和日志要一起用。tmux 保留交互现场，日志保留事实记录。远端训练、长时间构建或批处理任务，至少应该把 stdout 和 stderr 写入文件，例如 `./run-job.sh > logs/run.log 2>&1`。这样即使屏幕滚动过去，任务输出仍然可查。

# 11. 最小远程工作流：每一步都留下可检查证据 {#workflow}
<!-- tag: workflow -->

把前面几层合起来，一个稳妥的远程流程不需要复杂工具。先在本机确认命令和配置，再用 SSH 别名进入远端，确认远端身份和路径，进入项目目录，把长任务放进 tmux，输出写日志，断开后能重新接回，最后把结果复制回本机。每一步都能用一条命令验证。

示例：从本机到远端再取回结果复制

```bash
# 1. 本机先确认
echo "$SHELL"
type ssh
ssh devbox 'hostname; whoami; pwd'

# 2. 上传脚本或配置
scp run-job.sh devbox:~/work/app/

# 3. 远端进入可恢复会话
ssh devbox
cd ~/work/app
tmux new -s work
chmod +x run-job.sh
./run-job.sh > logs/run.log 2>&1

# 4. detach 后，稍后重新接回
tmux attach -t work

# 5. 在本机取回结果
scp devbox:~/work/app/logs/run.log .
```

- [ ] . 当前目录叫什么名字
- [ ] . 输入命令名时从哪些目录按顺序找可执行文件
- [ ] . SSH 私钥保存在哪里

Answer: . 输入命令名时从哪些目录按顺序找可执行文件
Explain: PATH 是命令查找目录的有序列表，顺序会影响同名命令命中哪一个。

### 2 判断

**Q:** 2. 判断：执行 `APP_ENV=dev` 后，所有后续子进程一定能读到 `APP_ENV`。

- [ ] 正确
- [ ] 错误

Answer: 错误
Explain: Shell 变量不会自动进入子进程，通常要 export。

### 3 填空

**Q:** 3. 填空：让 `APP_ENV=dev` 被后续子进程继承，常用关键字是？

Answer: export
Explain: 写成 export APP_ENV=dev，变量才会进入后续子进程环境。

### 4 单选

**Q:** 4. 单选：`APP_ENV=test ./run-job.sh` 的作用范围是？

- [ ] . 只影响这一次 ./run-job.sh
- [ ] . 永久写入 ~/.ssh/config
- [ ] . 自动同步到远端所有机器

Answer: . 只影响这一次 ./run-job.sh
Explain: NAME=value command 只影响紧跟着的 command，不永久修改当前 Shell。

### 5 诊断

**Q:** 5. 诊断：你把 `export PATH="$HOME/bin:$PATH"` 写进 `~/.bashrc`，但当前终端里 `mytool` 仍然找不到。写出两个可能原因和对应检查命令。

Explain: 原因可以包括：当前 Shell 还没重新读取 ~/.bashrc ，检查并运行 source ~/.bashrc ；当前 Shell 不是会读 ~/.bashrc 的 Bash，检查 echo "$SHELL" ； $HOME/bin 里没有可执行文件，检查 ls -l "$HOME/bin/mytool" ；命令名或权限错误，检查 type mytool 和 chmod +x 。

### 7 单选

**Q:** 7. 单选：在 `~/.ssh/config` 里，`HostName dev.example.org` 表示什么？

- [ ] . 本机下载目录

Explain: HostName 是真实目标主机；Host 是你在本机使用的别名。

### 8 判断

**Q:** 8. 判断：为了方便迁移，可以把 `~/.ssh/id_ed25519` 私钥直接提交到 dotfiles 仓库。

- [ ] 正确
- [ ] 错误

Answer: 错误
Explain: 私钥必须留在受保护的本机位置，不能提交到 dotfiles 仓库。

### 9 单选

**Q:** 9. 单选：登录远端后，最适合先确认“机器、用户、目录”的一组命令是？

- [ ] . cat ~/.ssh/id_ed25519
- [ ] . rm -rf ~/work/app
- [ ] . hostname; whoami; pwd

Answer: . hostname; whoami; pwd
Explain: hostname / whoami / pwd 能同时确认机器、用户和目录。

### 10 诊断

**Q:** 10. 诊断：连接 `devbox` 时出现 `REMOTE HOST IDENTIFICATION HAS CHANGED`。你应该先确认什么？不应该直接做什么？

Explain: 先确认远端主机是否重装、迁移、IP 复用，或者是否存在连接到错误机器的风险；不要不核实就删除 known_hosts 记录继续连接。确认新主机指纹可信后，再按组织或服务器管理员给出的方式更新本机记录。

### 11 单选

**Q:** 11. 单选：`scp report.txt devbox:~/work/app/` 的方向是？

- [ ] . 本机到远端
- [ ] . 远端到本机
- [ ] . 只在远端内部复制

Answer: . 本机到远端
Explain: 源端 report.txt 在本机，目标 devbox:~/work/app/ 在远端。

### 12 单选

**Q:** 12. 单选：`scp devbox:~/work/app/result.txt .` 的方向是？

- [ ] . 本机到远端
- [ ] . 远端到本机
- [ ] . 删除远端文件

Answer: . 远端到本机
Explain: 源端 devbox:~/work/app/result.txt 在远端，目标 . 是本机当前目录。

### 13 判断

**Q:** 13. 判断：远端 tmux 会话可以帮助你在 SSH 断开后重新接回工作现场。

- [ ] 正确
- [ ] 错误

Answer: 正确
Explain: tmux 会话能在网络连接断开后留在远端，之后可重新 attach。

### 14 填空

**Q:** 14. 填空：重新接回 `work` 会话常写 `tmux ____ -t work`。

Answer: attach
Explain: 常用 tmux attach -t work 接回名为 work 的会话。

### 15 诊断

**Q:** 15. 诊断：`tmux attach -t work` 报 `can't find session: work`。列出三个检查点。

Explain: 检查是否在创建会话的同一台机器上，可跑 hostname ；检查是否是同一用户，可跑 whoami ；检查真实会话名，可跑 tmux ls ；还要考虑任务结束后会话是否已经退出。

### 16 单选

**Q:** 16. 单选：`ssh devbox 'hostname; pwd'` 中 `hostname; pwd` 在哪里执行？

- [ ] . 本机 PowerShell
- [ ] . 远端 devbox 的 Shell
- [ ] . 浏览器控制台

Answer: . 远端 devbox 的 Shell
Explain: 引号里的 hostname; pwd 会交给远端 Shell 执行，输出来自远端。

### 17 单选

**Q:** 17. 单选：远端长任务更稳妥的运行方式是？

- [ ] . 裸跑在普通 SSH 窗口里，不保存输出
- [ ] . 只依赖屏幕滚动历史
- [ ] . 在 tmux 里运行，并把输出写入日志文件

Answer: . 在 tmux 里运行，并把输出写入日志文件
Explain: stdout 和 stderr 都写入日志，tmux 保留交互现场，日志保留事实记录。

### 18 场景

**Q:** 18. 场景：你要在 `devbox` 上跑 `run-job.sh`，网络不稳定。写一个最小流程，必须包含本机检查、SSH 登录、远端路径确认、tmux、日志和结果取回。

Explain: 一个合格答案应包含：本机 type ssh 或检查 ~/.ssh/config ； ssh devbox 'hostname; whoami; pwd' 确认远端；必要时 scp run-job.sh devbox:~/work/app/ ；远端 cd ~/work/app 后 tmux new -s work ；运行 ./run-job.sh > logs/run.log 2>&1 ；用 tmux attach -t work 接回；最后本机 scp devbox:~/work/app/logs/run.log . 取回日志。
