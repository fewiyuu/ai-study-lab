开发环境与工具链

这一页把 05 页的“命令在哪个环境里运行”继续推进到日常开发：编辑器、语言服务器、解释器、依赖、formatter、linter、测试和调试器必须指向同一个项目事实。你不需要立刻换编辑器，也不需要装满插件；本页要训练的是看到“编辑器报错但终端能跑”“测试能过但 IDE 启动失败”“代码助手改完引入回归”时，能把问题拆到具体工具、配置和验证命令。

1. 学习目标：让开发反馈指向同一个项目

goals

开发环境的问题很少只是“编辑器坏了”。更常见的是编辑器打开了错误目录，语言服务器选了另一套解释器，依赖装进了全局 Python，测试命令从不同的 cwd 启动，调试配置漏了环境变量，代码助手改了看似相关但没有验证的文件。工具越多，这些上下文越容易分叉。

贯穿样本：tiny-api 项目体检命令复制

root从项目根目录启动，工具才能找到配置和源码。

env解释器和依赖进入项目环境，避免全局污染。

LSP编辑器把文档和项目上下文交给语言服务器。

checks格式、lint、类型和测试提供不同层的反馈。

debug调试器按显式配置启动同一个目标。

diff自动生成的改动最终要回到 diff 和验证命令。

2. 工具链主线：每个工具都有输入、输出和失效边界

state

开发工具链是一条反馈链：编辑器改文本，LSP 读项目上下文给诊断和跳转，环境管理器决定命令在哪个解释器和依赖集合里运行，质量工具给出不同层次的反馈，调试器把程序暂停在运行时现场，版本控制和 diff 告诉你到底改了什么。

3. 项目根目录：工具从这里开始理解项目

project

项目根目录是工具链的共同入口。编辑器、LSP、测试框架、formatter、linter 和调试配置都会从某个目录开始找配置、源码和测试。如果你把编辑器打开在上级目录或某个子目录，语言服务器可能找不到 pyproject.toml，测试可能找不到包，Ruff 也可能读不到项目规则。

代码
tiny-api/
 pyproject.toml
 src/
 tiny_api/
 __init__.py
 app.py
 tests/
 test_app.py

编辑器打开 tiny-api/，和你在终端里 cd tiny-api 后运行命令，是同一个意图：让工具看到同一份配置和同一批文件。这个判断比“我用的是 VS Code、Vim 还是 JetBrains”更靠前。

现象可能错位检查动作修法
LSP 找不到包编辑器打开的是上级目录或子目录看编辑器工作区根目录；终端跑 pwd重新打开 tiny-api/ 作为根目录
Ruff 规则没生效没读到 pyproject.tomluv run ruff check . --show-settings从项目根目录运行命令
pytest 找不到模块启动目录和包布局不匹配uv run python -c "import tiny_api"确认包安装、源码布局和测试命令

4. LSP：编辑器为什么能补全、跳转和诊断

language

编辑器通常不自己实现每种语言的完整语义分析。LSP 把这件事拆开：编辑器负责界面和文档同步，语言服务器负责理解语言和项目。编辑器把文件内容、光标位置、工作区根目录和配置发给语言服务器；语言服务器返回补全、跳转定义、引用查找、诊断、悬停文档、重命名等结果。

示例：LSP 问题的第一轮检查复制

LSP 能力依赖的输入错位时表现回到哪里查
补全当前文件内容、依赖包、类型信息补全缺少项目模块或第三方库解释器和依赖安装位置
跳转定义源码路径、项目根目录、索引状态跳不到项目代码，或跳到缓存/错误环境工作区根目录和语言服务器状态
诊断语法、类型、配置、依赖编辑器报缺包但终端能运行编辑器解释器选择与 uv run 是否一致
重命名符号索引和引用关系只改当前文件，漏掉跨文件引用确认索引完成，先看 diff 再运行测试

5. 环境与依赖：安装成功不等于装进了项目环境

uv

Python 虚拟环境的核心作用是隔离解释器和依赖集合。一个包安装成功，只能说明它进入了某个环境；如果你的测试、LSP、调试器和终端使用的是另一套环境，导入照样会失败。本课程里统一用 uv run 作为 Python 项目的命令入口，让命令从项目环境里启动。

代码
uv run python -c "import sys; print(sys.executable)"
uv run python -c "import tiny_api; print(tiny_api.__file__)"
uv run python -m pytest --version

和 05 页的 PATH 一样，这里也要问“命令究竟从哪里启动”。如果你直接运行 python 或 pip install，命中的可能是系统环境、用户环境、某个旧虚拟环境，或者编辑器内置终端继承的环境。用项目约定的命令入口，能减少这种漂移。

动作它回答的问题风险更稳的做法
python -m pytest当前 python 能否运行测试当前 python 不一定是项目环境uv run pytest
pip install requests某个 pip 能否安装包包可能装到全局或旧环境按项目约定添加依赖，再用 uv run 验证
编辑器选择解释器LSP 和运行按钮使用哪套环境编辑器和终端看到的依赖不同选择项目虚拟环境或与 uv run 对齐的解释器
uv run python -c ...项目环境里能否导入包依赖没声明或没同步会暴露出来把它作为体检命令保留在 README 或任务入口里

6. formatter、linter、type checker、test：反馈层不要混用

quality

formatter 负责把代码排成统一样子，常见问题是缩进、换行、空格和引号。linter 负责找可疑写法，例如未使用变量、危险模式或规则违规。type checker 负责检查类型关系。测试验证你写出来的行为是否符合预期。它们互相补位，但不能互相替代。

代码
uv run ruff format --check .
uv run ruff check .
uv run pytest

7. 反馈层分工：从“能打开”到“能合并”

feedback

同一个改动会经过几层反馈。编辑器能打开文件，只说明文本可读；LSP 能跳转，只说明语言服务理解了当前项目的一部分；格式检查通过，只说明表面风格一致；lint 通过，只说明规则覆盖的可疑写法没有出现；测试通过，只说明已写测试覆盖的行为没坏；diff 可解释，才说明你知道这次改动碰了哪些文件和责任边界。

8. 运行配置与调试器：把启动上下文写清楚

run

终端和 IDE 的启动上下文不同，会直接改变程序行为。程序从哪个目录启动，传了哪些参数，设置了哪些环境变量，使用哪个解释器，入口是文件还是模块，这些都可能影响结果。调试器也是按一份配置启动程序，启动后才能在断点处暂停并观察变量、调用栈和执行路径。

代码
program/module: python -m tiny_api.app
cwd: project root
args: --port 8000
env: APP_ENV=dev
interpreter: project environment

9. 代码助手：把生成结果带回验证链路

code

代码助手适合生成样板、解释陌生代码、提出重构方案、补测试和列风险。它也可能调用不存在的 API、误读项目约束、改到无关文件，或者把局部问题扩成全局改动。对工具辅助改动，最小验收要落到 diff、修改范围、验证命令和必要的手工路径检查。

代码
git diff -- src/tiny_api tests
uv run ruff format --check .
uv run ruff check .
uv run pytest

10. 故障面板：先判断是哪一层反馈在说话

terminal

下面的面板把开发环境常见故障拆到具体层级。多数修法会落到根目录、解释器、依赖、配置、运行入口或改动范围，而不是重装一整套工具。

11. 最小闭环：接手项目时先做工具链体检

workflow

接手一个项目时，先别急着装插件。先建立一条可重复的体检链：打开正确根目录，确认项目环境，确认包能导入，运行格式/lint/测试，配置调试入口，最后把这些命令写进 README、任务脚本或项目说明。以后无论换编辑器、换机器还是让代码助手参与，都能回到这组检查。

代码
# 1. 在项目根目录
pwd
ls pyproject.toml src tests

# 2. 确认项目环境和导入
uv run python -c "import sys; print(sys.executable)"
uv run python -c "import tiny_api; print(tiny_api.__file__)"

# 3. 分层检查
uv run ruff format --check .
uv run ruff check .
uv run pytest

# 4. 修改后验收
git diff --stat
git diff -- src/tiny_api tests

> 这里先不追 完整 build system、CI/CD、容器、远程 IDE、复杂 Git 协作和性能分析分别放在后续单元。现在先把本机项目里最小的一组事实对齐：根目录、环境、依赖、质量工具、调试入口和 diff。
