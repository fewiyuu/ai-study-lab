学习单元 06

开发环境与工具链

这一页把 05 页的“命令在哪个环境里运行”继续推进到日常开发：编辑器、语言服务器、解释器、依赖、formatter、linter、测试和调试器必须指向同一个项目事实。你不需要立刻换编辑器，也不需要装满插件；本页要训练的是看到“编辑器报错但终端能跑”“测试能过但 IDE 启动失败”“代码助手改完引入回归”时，能把问题拆到具体工具、配置和验证命令。

读完本页应该能做什么

  • 用项目根目录、解释器、依赖和配置文件解释 LSP、终端、测试和 IDE 为什么会给出不同结果。
  • 区分 formatter、linter、type checker、pytest 和 debugger 各自读什么、改什么、不能替代什么。
  • 为一个小项目建立最小开发闭环：打开根目录、用 uv run 跑命令、检查质量工具、配置调试入口、审查代码助手改动。

1. 学习目标：让开发反馈指向同一个项目

goals

开发环境的问题很少只是“编辑器坏了”。更常见的是编辑器打开了错误目录，语言服务器选了另一套解释器，依赖装进了全局 Python，测试命令从不同的 cwd 启动，调试配置漏了环境变量，代码助手改了看似相关但没有验证的文件。工具越多，这些上下文越容易分叉。

本页用一个小项目 tiny-api 贯穿。它有 pyproject.toml、src/tiny_api/app.py 和 tests/test_app.py。目标很具体：让编辑、补全、运行、检查、测试、调试和自动生成改动都回到同一组事实：项目根目录、项目环境、配置文件和验证命令。

贯穿样本：tiny-api 项目体检命令
pwd
ls pyproject.toml
uv run python -c "import sys; print(sys.executable)"
uv run python -c "import tiny_api; print(tiny_api.__file__)"
uv run ruff format --check .
uv run ruff check .
uv run pytest
root从项目根目录启动，工具才能找到配置和源码。
env解释器和依赖进入项目环境，避免全局污染。
LSP编辑器把文档和项目上下文交给语言服务器。
checks格式、lint、类型和测试提供不同层的反馈。
debug调试器按显式配置启动同一个目标。
diff自动生成的改动最终要回到 diff 和验证命令。

2. 工具链主线：每个工具都有输入、输出和失效边界

state ledger

开发工具链是一条反馈链：编辑器改文本，LSP 读项目上下文给诊断和跳转，环境管理器决定命令在哪个解释器和依赖集合里运行，质量工具给出不同层次的反馈，调试器把程序暂停在运行时现场，版本控制和 diff 告诉你到底改了什么。

root项目范围

决定工具从哪里找配置、源码和测试。

env解释器与依赖

决定 import 能看到哪些包。

LSP语义反馈

补全、跳转和诊断来自语言服务器。

checks多层验证

格式、静态风险和行为测试分开看。

debug运行时观察

断点、变量、调用栈需要正确启动上下文。

层级 读取什么 写入或返回什么 错位时症状 第一条检查
项目根目录 pyproject.toml、src/、tests/ 工具的查找边界 配置没读到，测试发现不了模块 pwd; ls pyproject.toml
项目环境 解释器、依赖声明、锁文件 命令可见的包和版本 终端能 import，编辑器报缺包 uv run python -c "import sys; print(sys.executable)"
LSP 文档内容、根目录、环境、配置 补全、跳转、诊断、悬停、重命名 跳转失败，导入误报 检查编辑器的语言服务器状态和解释器选择
质量工具 源码和配置 格式化结果、lint 诊断、测试结果 格式通过但行为仍错 uv run ruff check . / uv run pytest
调试器 启动配置、断点、源代码 暂停点、变量、调用栈 断点不进，参数不对 核对 cwd、args、env、module/program

3. 项目根目录：工具从这里开始理解项目

project root

项目根目录是工具链的共同入口。编辑器、LSP、测试框架、formatter、linter 和调试配置都会从某个目录开始找配置、源码和测试。如果你把编辑器打开在上级目录或某个子目录，语言服务器可能找不到 pyproject.toml，测试可能找不到包，Ruff 也可能读不到项目规则。

示例：tiny-api 的最小目录结构
tiny-api/
  pyproject.toml
  src/
    tiny_api/
      __init__.py
      app.py
  tests/
    test_app.py

编辑器打开 tiny-api/，和你在终端里 cd tiny-api 后运行命令，是同一个意图：让工具看到同一份配置和同一批文件。这个判断比“我用的是 VS Code、Vim 还是 JetBrains”更靠前。

现象 可能错位 检查动作 修法
LSP 找不到包 编辑器打开的是上级目录或子目录 看编辑器工作区根目录；终端跑 pwd 重新打开 tiny-api/ 作为根目录
Ruff 规则没生效 没读到 pyproject.toml uv run ruff check . --show-settings 从项目根目录运行命令
pytest 找不到模块 启动目录和包布局不匹配 uv run python -c "import tiny_api" 确认包安装、源码布局和测试命令

4. LSP：编辑器为什么能补全、跳转和诊断

language server

编辑器通常不自己实现每种语言的完整语义分析。LSP 把这件事拆开：编辑器负责界面和文档同步，语言服务器负责理解语言和项目。编辑器把文件内容、光标位置、工作区根目录和配置发给语言服务器；语言服务器返回补全、跳转定义、引用查找、诊断、悬停文档、重命名等结果。

补全坏了，先别急着重装编辑器。更常见的原因是语言服务器没启动，项目根目录开错，解释器选错，依赖没装进当前环境，或者配置文件没被读取。Python 项目里尤其要盯住“编辑器选择的解释器”和“终端运行测试用的解释器”是不是同一套。

示例：LSP 问题的第一轮检查
pwd
ls pyproject.toml
uv run python -c "import sys; print(sys.executable)"
uv run python -c "import tiny_api; print(tiny_api.__file__)"
LSP 能力 依赖的输入 错位时表现 回到哪里查
补全 当前文件内容、依赖包、类型信息 补全缺少项目模块或第三方库 解释器和依赖安装位置
跳转定义 源码路径、项目根目录、索引状态 跳不到项目代码，或跳到缓存/错误环境 工作区根目录和语言服务器状态
诊断 语法、类型、配置、依赖 编辑器报缺包但终端能运行 编辑器解释器选择与 uv run 是否一致
重命名 符号索引和引用关系 只改当前文件，漏掉跨文件引用 确认索引完成，先看 diff 再运行测试

5. 环境与依赖：安装成功不等于装进了项目环境

uv / venv

Python 虚拟环境的核心作用是隔离解释器和依赖集合。一个包安装成功，只能说明它进入了某个环境；如果你的测试、LSP、调试器和终端使用的是另一套环境，导入照样会失败。本课程里统一用 uv run 作为 Python 项目的命令入口，让命令从项目环境里启动。

示例：确认 uv 命令使用的解释器和包位置
uv run python -c "import sys; print(sys.executable)"
uv run python -c "import tiny_api; print(tiny_api.__file__)"
uv run python -m pytest --version

和 05 页的 PATH 一样，这里也要问“命令究竟从哪里启动”。如果你直接运行 python 或 pip install，命中的可能是系统环境、用户环境、某个旧虚拟环境，或者编辑器内置终端继承的环境。用项目约定的命令入口，能减少这种漂移。

动作 它回答的问题 风险 更稳的做法
python -m pytest 当前 python 能否运行测试 当前 python 不一定是项目环境 uv run pytest
pip install requests 某个 pip 能否安装包 包可能装到全局或旧环境 按项目约定添加依赖，再用 uv run 验证
编辑器选择解释器 LSP 和运行按钮使用哪套环境 编辑器和终端看到的依赖不同 选择项目虚拟环境或与 uv run 对齐的解释器
uv run python -c ... 项目环境里能否导入包 依赖没声明或没同步会暴露出来 把它作为体检命令保留在 README 或任务入口里

6. formatter、linter、type checker、test：反馈层不要混用

quality tools

formatter 负责把代码排成统一样子，常见问题是缩进、换行、空格和引号。linter 负责找可疑写法，例如未使用变量、危险模式或规则违规。type checker 负责检查类型关系。测试验证你写出来的行为是否符合预期。它们互相补位，但不能互相替代。

示例：tiny-api 的质量检查顺序
uv run ruff format --check .
uv run ruff check .
uv run pytest
工具层 读什么 输出什么 不能证明什么 典型命令
formatter 源码文本和格式规则 规范化后的文本，或格式差异 业务逻辑正确 uv run ruff format --check .
linter 源码和 lint 规则 可疑写法、未使用变量、部分可自动修复问题 测试覆盖充分 uv run ruff check .
type checker 类型标注、依赖类型信息、源码 类型不匹配或缺失信息 运行时所有路径安全 uv run pyright 或项目约定命令
test 代码、测试数据、环境变量、依赖 行为通过或失败 没写到的行为也正确 uv run pytest

检查顺序可以随项目调整，含义不能混。格式检查失败时，先修格式；lint 失败时，看规则是否指出真实风险；测试失败时，回到输入、输出和状态变化。把 ruff format 当成 pytest，会得到一份整齐但可能仍然错误的代码。

7. 反馈层分工：从“能打开”到“能合并”

feedback layers

同一个改动会经过几层反馈。编辑器能打开文件，只说明文本可读；LSP 能跳转，只说明语言服务理解了当前项目的一部分；格式检查通过，只说明表面风格一致；lint 通过，只说明规则覆盖的可疑写法没有出现；测试通过，只说明已写测试覆盖的行为没坏；diff 可解释，才说明你知道这次改动碰了哪些文件和责任边界。

反馈层 通过时能说明什么 仍然不能说明什么 失败后先看
编辑器打开 文件在工作区内可见，可被修改 项目根目录和解释器已经对齐 打开目录、文件树、工作区设置
LSP 诊断 语言服务器能读到当前文档和部分项目上下文 运行时一定能导入同样的包 语言服务器状态、解释器、依赖
格式检查 代码外观符合项目格式规则 逻辑正确或测试充分 formatter 输出的文件列表和差异
lint / 类型检查 规则和类型系统覆盖的风险暂未触发 业务路径都被验证 具体规则、类型来源、依赖 stub
测试 测试覆盖到的行为在当前环境下通过 没写测试的路径也正确 失败断言、输入数据、环境变量、cwd
diff 审查 改动范围和意图可以被人检查 运行结果已经自动可靠 git diff --stat、关键文件 diff、回滚路径

这个分层也能指导你排错。LSP 报缺包但测试能过，先查编辑器解释器；测试失败但格式通过，先读失败用例；diff 多出无关文件，先缩小改动范围。每一层都只回答自己的问题，下一层还要继续验证。

8. 运行配置与调试器：把启动上下文写清楚

run / debug

终端和 IDE 的启动上下文不同，会直接改变程序行为。程序从哪个目录启动，传了哪些参数，设置了哪些环境变量，使用哪个解释器，入口是文件还是模块，这些都可能影响结果。调试器也是按一份配置启动程序，启动后才能在断点处暂停并观察变量、调用栈和执行路径。

示例：运行配置字段
program/module: python -m tiny_api.app
cwd: project root
args: --port 8000
env: APP_ENV=dev
interpreter: project environment
字段 影响什么 错了会怎样 检查动作
cwd 相对路径、配置发现、测试发现 找不到文件或读错配置 在程序开头临时打印 Path.cwd()
args 命令行参数 默认值被误用，端口或路径不对 把终端命令逐项拆进配置
env 运行时配置和开关 程序按默认环境启动 打印或断点查看 os.environ 里的关键项
interpreter 包和 Python 版本 导入失败或版本差异 调试时观察 sys.executable
entrypoint 执行哪个模块或文件 断点不进，初始化路径不同 确认终端命令和调试配置启动同一个目标

调试器最适合回答运行时问题：程序走到哪一行、变量现在是什么、调用栈是谁把它带到这里、条件分支为什么走了这一边。格式问题和依赖安装问题不要硬塞给调试器；先让程序在正确环境里启动，再用断点观察行为。

9. 代码助手：把生成结果带回验证链路

code assistant

代码助手适合生成样板、解释陌生代码、提出重构方案、补测试和列风险。它也可能调用不存在的 API、误读项目约束、改到无关文件，或者把局部问题扩成全局改动。对工具辅助改动，最小验收要落到 diff、修改范围、验证命令和必要的手工路径检查。

示例：工具辅助改动后的验收命令
git diff -- src/tiny_api tests
uv run ruff format --check .
uv run ruff check .
uv run pytest
风险 观察信号 检查动作 处理方式
改动范围失控 无关文件出现在 diff 里 git diff --stat 拆小任务，只保留本次目标相关改动
API 编造 调用项目中不存在的函数或参数 跳转定义、搜索引用、跑测试 回到项目源码和文档确认
测试只覆盖表面 新增测试没有断言关键行为 读测试输入、断言和失败路径 补一个能在错误实现下失败的测试
高风险场景误改 涉及权限、密钥、数据迁移、生产配置 人工审查 + 最小验证 + 回滚计划 降低自动化程度，分步骤提交

8. 诊断：终端里 uv run python -c "import tiny_api" 成功，但编辑器提示 Import "tiny_api" could not be resolved。写出最可能的错因、后果和修法。

      
      
      

        参考答案 / 评分要点
        
错因通常是编辑器/LSP 选择的解释器与 uv run 使用的项目环境不同，或编辑器打开了错误根目录。后果是编辑器诊断、补全和跳转不可信。修法：重新打开项目根目录，选择与项目环境一致的解释器，必要时重启语言服务器，再用导入命令验证。

诊断run config

      
12. 诊断：终端运行 uv run pytest 全部通过，但 IDE 的 Run 按钮启动时报 FileNotFoundError: pyproject.toml。按错因、后果、修法回答。

      
      
      

        参考答案 / 评分要点
        
错因多半是 IDE 的 cwd 不在项目根目录，或运行配置没有复刻终端命令的入口、参数、环境和解释器。后果是程序读不到配置或相对路径。修法：把 cwd 设为项目根目录，核对 args/env/interpreter/entrypoint，并用一个最小命令确认配置被读到。

      

    


    

      
匹配tool roles

      
13. 匹配：把 formatter、linter、type checker、pytest 分别写成“读什么 / 反馈什么 / 不能替代什么”。

      
      
      

        参考答案 / 评分要点
        
formatter 读源码文本和格式规则，反馈格式差异，不能替代测试。linter 读源码和规则，反馈可疑写法，不能替代行为验证。type checker 读类型标注、依赖类型信息和源码，反馈类型不匹配，不能证明所有运行路径安全。pytest 运行测试验证行为，不能覆盖没写进测试的需求。

17. 场景：你接手 tiny-api，要建立最小可用开发环境。写出 6 步以内的检查流程。

      
      
      

        参考答案 / 评分要点
        
可答：打开项目根目录并确认 pyproject.toml；用 uv run python -c "import sys; print(sys.executable)" 确认项目环境；用 uv run python -c "import tiny_api" 确认导入；配置编辑器解释器和 LSP；运行 uv run ruff format --check .、uv run ruff check .、uv run pytest；配置调试入口的 cwd/args/env/interpreter；把命令记录下来。

      

    


    

      
综合explain

      
18. 费曼题：向零基础同学解释“开发环境不是一个软件，而是一条反馈链”。至少包含编辑器、LSP、项目环境、质量工具、调试器和代码助手验收。

      
      
      

        参考答案 / 评分要点
        
好的解释会说清：编辑器负责修改和浏览代码；LSP 依赖项目根目录和解释器给补全、跳转、诊断；项目环境决定程序能导入哪些包；formatter、linter、type checker 和测试提供不同反馈；调试器按显式配置观察运行时状态；代码助手可以加速，但最终要看 diff、跑检查和验证关键行为。
