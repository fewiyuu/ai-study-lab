调试 · 性能分析

先观察，再假设，再验证：不要凭感觉修 bug 或优化性能

程序只会执行你写下的指令，不会执行你脑中的意图。调试就是定位“期望行为”和“实际行为”的差异；性能分析则是定位时间、内存或 I/O 花在哪里。

学完你应该能做什么

  1. 区分打印调试、日志、断点调试的适用场景。
  2. 读懂日志级别和结构化日志的价值。
  3. 解释为什么优化前必须先测量。
  4. 用 profile/benchmark 思路定位瓶颈和验证改进。

概念地图

调试和性能分析都遵循同一条证据链：先复现问题，再收集观察，再提出假设，再做最小修改验证。调试关注“为什么错”，性能分析关注“哪里慢、慢多少、改了是否真的快”。

这条证据链的重点是把“我觉得”换成“我观察到”。复现步骤说明问题如何出现，日志和断点说明程序当时的状态，profile 和 benchmark 说明性能事实，回归测试说明修复没有把旧行为弄坏。没有证据链，改动越多越难知道真正起作用的是哪一处。

复现→观察→假设→验证→回归测试
打印调试临时输出关键变量和路径，适合快速缩小范围。它便宜直接，但要避免留下无意义噪声。
日志长期留在代码里的观察点，通常带级别、时间、上下文。好的日志能让你在事后复盘现场。
Debugger暂停程序、查看变量和调用栈的工具。它适合复杂状态，不适合漫无目的逐行浏览。
Profiler统计程序把时间或资源花在哪里的工具。它回答“热点在哪里”，不是直接告诉你怎么改。

1. 调试不是乱改，而是缩小差异

最差的调试方式是看到错误后随机改代码，直到它“不报错”。这会把 bug 藏得更深。好的调试先写下预期，再观察实际：哪一步开始偏离？输入是什么？状态是什么？错误是否稳定复现？

打印调试并不低级。它适合快速确认执行路径、变量值和边界条件。问题在于临时打印容易散落在代码里，输出也容易缺上下文。正式项目中应把长期有价值的观察点变成日志。

复现是调试的入口。一个好复现应尽量小：固定输入、固定命令、固定环境，能稳定触发同一个错误。复现越小，变量越少；变量越少，假设越容易验证。对于偶发问题，至少要记录出现频率、时间、输入规模、环境和错误栈。

正确示例打印关键上下文
print("loading file", path)
print("rows", len(rows), "first", rows[0] if rows else None)
print("user_id", user_id, "permission", permission)
逐行拆解最小复现记录
python parse.py samples/bad-row.csv
# 预期：跳过空 name 行
# 实际：ValueError: invalid literal for int()
# 环境：Python 3.12，输入 1 行，稳定复现

判断句：修 bug 前先把问题变成可复现、可观察、可验证的事实。你现在应该能做什么：遇到 bug 时先设计观察点，而不是先大面积改代码。

2. 日志：留在代码里的观察系统

日志是规范化的打印。它通常包含时间、级别、模块、消息和上下文。日志级别从细到粗常见有 trace/debug/info/warn/error/critical。级别的意义是控制信息密度：开发时看 debug，生产告警重点看 error/critical。

结构化日志把上下文以 JSON 或键值形式记录，便于之后用工具查询和统计。纯文本日志适合人读，结构化日志适合机器分析。无论哪种，日志都应该回答“发生了什么、在哪发生、和哪个请求/用户/文件有关”。

日志最常见的问题不是“没有日志”，而是日志没有定位价值。只写 failed 没有用，因为它没有对象、原因和上下文。更好的日志会带上 request_id、user_id、file、job_id、attempt、duration 等字段，让你能把多条日志串成一次事件。

逐行拆解日志级别示例
logger.debug("parsed config", extra={"path": path})
logger.info("job started", extra={"job_id": job_id})
logger.warning("retrying request", extra={"attempt": attempt})
logger.error("job failed", extra={"job_id": job_id, "error": str(e)})

判断句：日志级别控制信息密度，日志上下文决定能不能定位问题。你现在应该能做什么：判断一条信息应该是 debug、info、warning 还是 error。

3. 调试器：暂停现场，看调用栈和变量

调试器适合处理复杂状态：递归、循环、异步回调、多层函数调用。断点让程序暂停在某一行，单步执行让你观察状态如何变化，调用栈告诉你程序如何走到这里。

调试器并不替代思考。你仍然需要先提出问题：我怀疑哪个变量错？哪个分支不该进？哪个函数被错误参数调用？没有问题意识地单步执行，只会让你在大量细节里迷路。

调用栈是调试器里最值得看的信息之一。它告诉你当前函数是谁调用的、参数从哪里来、错误如何一路传递。看到异常时，不要只看最后一行；往上看调用链，经常能找到真正错误输入产生的位置。

正确示例Python 断点入口
def parse_user(row):
    breakpoint()
    return {"id": int(row["id"]), "name": row["name"].strip()}

判断句：断点应该放在“状态开始变错”的边界附近，而不是随便从第一行一路单步。你现在应该能做什么：在进入调试器前写下一个具体怀疑点。

4. 性能分析：优化前先问慢在哪里

性能问题有很多类型：CPU 算得慢、内存占用高、磁盘 I/O 慢、网络等待久、数据库查询多、锁竞争严重。凭感觉优化很容易改错地方。你以为慢在排序，实际慢在读取远程接口；你以为需要并发，实际是重复查询。

测量要有基线。先记录当前耗时、输入规模、机器环境和运行方式，再改一个因素，再重新测。没有基线的“优化”只是愿望。

性能分析也要分层。粗粒度计时回答“总共慢不慢”；日志中的 duration 字段能回答“哪个阶段慢”；profiler 回答“函数级别时间花在哪里”；系统工具能回答 CPU、内存、磁盘和网络资源是否成为瓶颈。不要用一个工具解释所有层级。

正确示例用 time 获取粗粒度耗时
time python script.py data.csv
time pytest tests/test_parser.py

判断句：优化前先把“慢”量化为输入规模、耗时、环境和瓶颈位置。你现在应该能做什么：把“感觉很慢”改写成“在某输入规模下耗时多少，主要花在哪里”。

5. Profiler：找到真正热点

Profiler 会统计函数调用次数、累计耗时和单次耗时，帮助你找到热点。热点不一定是最复杂的代码，而是运行次数多或等待资源久的地方。优化热点之外的代码，对总体耗时影响可能很小。

采样 profiler 和插桩 profiler 有不同开销。采样 profiler 定期观察程序在哪里，开销较低；插桩 profiler 记录函数进入退出，信息细但可能影响运行。无论工具如何，解读时都要关注“总耗时占比”和“调用次数”。

读 profiler 结果时要小心两个陷阱。第一，单次耗时很高的函数不一定是总体瓶颈；调用一百万次的普通函数也可能耗掉大部分时间。第二，等待 I/O 的时间可能显示在调用栈中，但真正修法不是优化 CPU 算法，而是减少请求次数、批量读取或缓存结果。

逐行拆解Python cProfile 最小用法
python -m cProfile -s cumulative script.py
# -s cumulative 按累计耗时排序，优先看总成本高的函数

判断句：热点是“总成本高”的地方，不只是“看起来复杂”的地方。你现在应该能做什么：看到 profiler 结果时，不只看单次最慢函数，也看累计耗时和调用次数。

6. Benchmark：验证改进是否真实

Benchmark 是可重复的性能实验。它需要固定输入、运行环境、预热方式和统计方法。一次运行快了不代表真的优化，可能是缓存、系统负载或随机波动。可靠 benchmark 会多次运行并比较分布。

性能优化还要防止破坏正确性。一个更快但结果错误的实现没有意义。所以常见流程是：先用测试锁住正确性，再用 benchmark 比较性能，再用 profiler 解释为什么变快或没变快。

小项目可以从简单 benchmark 开始：固定一份输入，运行旧版本和新版本，各运行多次，记录均值、中位数和最慢值。更严谨的场景还要控制机器负载、预热缓存、隔离网络波动。结论要写成“在这个输入和环境下快了多少”，不要泛化成所有场景都更快。

错误示例只跑一次就宣布优化成功
time python new_version.py
# 只运行一次，无法排除缓存和系统波动
正确示例保留正确性和多次运行
pytest tests/test_parser.py
for i in 1 2 3 4 5; do time python script.py data.csv; done
# 先确认结果正确，再比较多次耗时

判断句：一次变快不是证据，正确结果加稳定重复才是性能结论。你现在应该能做什么：设计一个最小 benchmark，包含固定输入、多次运行和正确性检查。

7. 常见错误与修法

随机改 bug

错因是没有复现和假设。后果是修复不可解释，还可能引入新问题。修法：先写复现步骤、观察点和最小修改。

日志太少或太吵

缺上下文或级别错误。后果是出事时查不到现场，平时又被无关信息淹没。修法：按级别记录关键事件、关联 ID 和必要字段。

优化错热点

没有 profile。后果是代码变复杂但整体没变快。修法：先测量，再改累计耗时最高或资源占比最高的瓶颈。

benchmark 不稳定

只跑一次或输入不固定。后果是把缓存、系统负载或随机波动误当成优化。修法：固定环境，多次运行，比较分布。

只看异常最后一行

最后一行常是报错位置，不一定是错误输入产生的位置。修法：往上读调用栈，找数据第一次变坏的边界。

优化破坏正确性

为了快改掉了边界处理。后果是性能数字好看但结果错误。修法：先补测试锁住行为，再做性能比较。

8. 完整知识链路

调试从可复现问题开始，用打印、日志或调试器收集证据，定位预期和实际的差异，再用最小修改验证。性能分析从基线测量开始，用 profiler 找热点，用 benchmark 验证改进，并用测试保证正确性。调试和优化的共同敌人都是凭感觉行动。

完整流程可以这样串起来：先把问题缩成最小复现，记录命令、输入、环境和错误栈；再选择观察工具，简单路径用打印，长期现场用日志，复杂状态用断点和调用栈；提出一个能被验证的假设，只改最小必要代码；修复后补回归测试。性能问题则先建立基线，再用 profile 找热点，最后用 benchmark 证明改动在正确性不变的前提下确实更快。

9. 真实场景与排查手册

调试的第一步是复现。不能稳定复现的问题，很难判断改动是否有效。复现后再二分范围，观察输入、状态、输出和错误栈。

性能问题也要先测量。慢在哪里可能和直觉相反：I/O、网络、数据库、算法复杂度、重复计算、启动开销都可能是瓶颈。没有 profile 就改代码，很容易优化错地方。

真实排查要保留证据：复现命令、最小输入、错误栈、计时结果、profile 截图或摘要。这样每次修改都能和基线比较。

不能复现就改

没有复现步骤时，修复只是猜测。先把输入、环境和命令固定下来。

打印太多

日志要服务定位。关键位置记录输入、状态和分支，不要把噪声当调试。

凭感觉优化

先计时和 profile，再改瓶颈。优化非瓶颈代码通常没有收益，还可能引入 bug。

稳定流程

复现 -> 缩小 -> 假设 -> 验证 -> 回归测试，性能问题再加基线和 profile。
