学习单元 07

调试与性能分析：先观察，再测量，再验证

这一页把调试和性能分析放进同一条证据链里：先复现，再观察，再测量，再验证。你会分清日志、断点、profiler 和 benchmark 各管什么，再把它们接到一套可复跑的排查顺序上。这里先不展开 Git、打包、自动化或更大规模协作，只处理“问题怎么被看见、怎么被量化、怎么被修正”。

先抓住这三件事

  • 能把一个 bug 复现成最小命令、最小输入和最小环境，避免只停留在现象描述。
  • 能判断该用 print、日志、断点还是 profiler，并说出各自的边界。
  • 能在优化前后各做一次基线和回归检查，避免把更快但错误的改动当成成功。

学习目标

goal

这一单元的目标很具体：把“程序出了问题”拆成可以观察、可以复现、可以比较的事实。调试时你要知道从哪里开始缩小范围，性能分析时你要知道先建立什么基线、再看什么热点、最后怎样证明优化真的生效。只靠感觉会把时间花在错误的地方；只看结果不看证据，也很难把经验留下来。

先抓住的概念

  • 复现、观察、假设、验证要连成一条证据链。
  • 日志、断点、profiler、benchmark 解决的是不同层次的问题。
  • 更快但错误的实现，不算优化成功。

这一页之后能接什么

以后进入更大的工程，最常反复用到的是排查顺序和证据习惯：先记录输入和环境，再选择工具，最后把修复和回归测试连起来。

证据链

overview

调试和性能分析看起来像两件事，底层动作却很像：先把问题变成稳定现象，再去找原因。调试关注“哪里开始和预期不一样”，性能分析关注“时间、内存或 I/O 花在了哪里”。二者共同要求你把主观判断换成可复查的证据。

复现：把问题固定成最小命令、最小输入、最小环境。没有这个步骤，后面的观察和假设都不稳。
观察与验证：先用日志、断点或调用栈抓状态，再用最小修改和回归测试确认修复没有引入新问题。
复现→观察→假设→验证→回归测试

这条链的关键不在于步骤多，而在于每一步都能告诉你下一步该看什么。复现阶段先固定输入和环境，观察阶段先抓日志或断点，假设阶段只保留一个最可能的解释，验证阶段只做最小改动，回归测试阶段确认修复没有把旧行为弄坏。顺序一乱，问题就会变得越来越像“经验题”。

日志

trace

日志是正式版的观察系统。它比临时打印更稳定，也比记在脑子里更容易复盘。好的日志至少回答三件事：发生了什么、在什么对象上发生、和哪一次请求或哪一个文件有关。只写一个 failed，通常等于没写。

级别 典型用途 要带的上下文 看不出上下文时的症状
debug 开发时看局部状态 路径、参数、分支、输入大小 只能看到“哪里坏了”，看不到“为什么坏”
info 重要事件记录 请求号、任务号、文件名、阶段名 事后无法把几条日志连成一件事
warning 可以继续，但值得注意 重试次数、降级路径、阈值 真正的异常信号被淹没
error / critical 失败或严重失败 错误原因、关联对象、退出码 只能看到失败字样，无法定位责任边界
日志上下文示例
logger.info("job started", extra={"job_id": 42, "path": "data/input.csv"})
logger.warning("retrying request", extra={"attempt": 2, "timeout_ms": 3000})
logger.error("job failed", extra={"job_id": 42, "error": "timeout"})

日志不靠数量取胜，靠能定位。只要日志里保留了对象、阶段和错误原因，你就能把一堆分散记录拼回一次完整事件。

调试器

state

调试器适合看复杂状态：递归、循环、异步回调、多层函数调用。断点把程序停住，单步执行让你看到状态怎么变，调用栈告诉你程序是怎么走到这里的。它最有价值的地方，是让你看到变量在每一层被谁读、被谁写。

观察对象 读什么 写什么 错了会怎样
当前行 源码位置 暂停执行 看错分支，误判程序路径
局部变量 当前帧里的值 不写，只观察 以为值已经被别的层改了
调用栈 谁调用了谁 不写，只追溯来源 只看最后一行，漏掉错误输入的来源
调用栈阅读示例
def parse_user(row):
    return int(row["id"])

def load(rows):
    for row in rows:
        print(parse_user(row))

load([{"id": "12"}, {"id": "x"}])

如果最后一行报错在 int(row["id"])，还要继续往上看：是谁把这个值传进来的，为什么那一层没有过滤掉坏数据。断点的作用不是让你盯着每一行，而是帮你找到第一次变坏的地方。

基线与 Benchmark

measure

性能问题先问“慢在哪里”，再问“怎么让它更快”。第一步不是优化，而是测量。基线就是你在改动前记下的数字，常见有总耗时、输入规模、机器环境、运行方式和是否命中缓存。没有基线，后面的“更快”只是感觉。

基线要记什么

  • 命令和输入规模
  • 运行环境和版本
  • 重复次数和波动范围
  • 正确性是否一致

benchmark 最容易错的地方

只跑一次、输入不固定、环境会变、只看最快一次，都会把波动误当成优化。benchmark 的意义是比较同一条件下的分布，不是挑一个好看的数字。

$$ \text{改进是否成立} = \text{正确性保持} \land \text{在同一输入与环境下的平均耗时下降} $$

“正确性保持”是硬前提。只要输出变了，性能数字再好也没有意义。测量要先锁住行为，再谈速度。

基线记录示例
uv run python -m timeit -s "from parse_rows import parse_file" "parse_file('samples/bad-row.csv')"
输入规模：10 万行
环境：Python 3.12，单机本地磁盘
重复：5 次
正确性：输出与旧版本一致

Profiler

hotspot

Profiler 的任务是找热点。热点可能来自复杂函数，也可能来自被反复调用的轻函数。一个看起来很轻的函数，如果被调用很多次，累计耗时就可能把总时间吃掉。这里要一起看累计耗时和调用次数。

读法单次很小，累计很大

调用次数一多，简单函数也能成为总耗时的大头。优化前先看累计时间。

读法单次大，次数少

单次慢不一定是主瓶颈，先比较整体占比，再决定要不要改。

读法资源等待也会出现在热点里

如果时间花在 I/O 或数据库请求上，修法可能是减少请求次数、批量读取或缓存。

Profiler 观察点
看什么：
1. 调用次数
2. 累计耗时
3. 单次耗时
4. 是否在等待 I/O

先别急着看：
1. 局部的小重构
2. 不在热点上的函数
3. 没有基线的对比

终端故障实验

lab

这组实验把“先复现，再判断”做成可切换的故障面板。每个场景都在问同一件事：你先看哪一层。它不会替代真实系统，但能训练你把注意力放到正确入口上。

ERROR job failed job_id=42 error=timeout
INFO retrying request attempt=2
HINT: 先看请求是否重复，还是后端响应真的超时。
场景 先看什么 不要先做什么 下一步证据
功能错误 复现命令和最小输入 大面积改代码 错误栈、日志、断点
性能变慢 基线和热点 直接重构最显眼的函数 计时、profile、分段日志
偶发失败 环境、随机性、并发 只看一次运行结果 复现概率、输入规模、重试记录

排查流程

flow

把流程说顺，比背很多工具更重要。一个稳定的排查顺序通常长这样：先复现，再缩小，再观察，再提出一个可验证的假设，最后只做最小修改并回归测试。性能问题再加两步：先建立基线，再用 profiler 解释热点。

步骤 读什么 写什么 错了的症状
复现 命令、输入、环境 最小坏样本 问题时有时无，无法验证
观察 日志、断点、调用栈 证据片段 只剩猜测，没有落点
测量 基线、profile、benchmark 比较数据 不知道快了多少、慢在哪里
验证 回归测试、输出差异 最小修复 修复和新问题缠在一起
功能错误先找第一次变坏的位置

问题还没稳定复现时，不要急着扩大修改。先把坏输入和当前状态固定下来。

性能错误先找对比基线

没有基线就没有“变快”的证据。先记旧版本和新版本在同一输入上的耗时。

修复后先看回归测试

修复不能只靠一次手动运行。要有稳定的回归测试把行为锁住。

离线实操

worked example

下面这个例子把一条完整排查链串起来：一个解析任务突然变慢，还偶尔报错。你不需要接触真实服务，也能看到怎么从复现命令一路走到修法。重点不在背步骤，而在看每一步读什么、写什么、留下什么证据。

最小复现命令
uv run python parse_rows.py samples/bad-row.csv

复现时先看什么

  • 输入文件名和最小坏样本
  • 当前工作目录和运行命令
  • 错误是否稳定出现

常见失败信号

ValueError、timeout、处理时间突然上升、只在某个输入规模出现的卡顿，这些都说明你要先固定样本，再看证据，先别急着改大段代码。

步骤 读什么 写什么 失败信号 下一步
复现 命令、输入、环境 最小坏样本 问题不稳定 加日志或断点
分段观察 输入大小、阶段耗时、异常栈 局部证据 只知道“慢了”，不知道慢在哪 缩小到单个阶段
测量 基线和当前结果 对比数字 没有量化依据 用 profiler 找热点
修复 测试和回归结果 最小修改 修了一个地方，另一个地方坏了 补回归测试
分段日志示例
INFO parse start file=samples/bad-row.csv rows=1
INFO parse stage=decode duration_ms=4
INFO parse stage=validate duration_ms=82
ERROR parse failed line=18 error=invalid literal for int()
INFO profile hotspot=validate_rows cumulative_ms=79

这组日志说明了一个典型事实：慢点和报错点未必在同一行。decode 很快，validate 很慢，真正的错误出现在某一行数据上。你应该先看阶段耗时，再看异常输入，再决定是修数据、修校验逻辑，还是改调用顺序。

如果你只看最后一行

你会知道哪一行报错，却不知道为什么它会出现，也不知道慢的是哪一段。这样做通常只能得到“改了点东西”，得不到稳定修复。

如果你把链路跑完整

你会得到一个可以写进回归测试的最小输入、一个能复查的日志分段和一条能验证性能改善的基线。这些才是调试和性能分析要留下的东西。

常见错误

debug

没复现就开始改

错因是证据不够。后果是改动越来越多，却没人知道哪一处真的起作用。修法是先写出最小复现步骤，再决定下一步观察点。

日志只有失败，没有上下文

错因是日志没记录对象和阶段。后果是事后无法把几条记录串起来。修法是补上 request_id、job_id、文件名、阶段名或关键输入。

只看 profiler 的单次耗时

错因是把“慢一次”和“总成本高”混为一谈。后果是优化错热点。修法是同时看调用次数和累计耗时。

benchmark 只跑一次

错因是把波动当结论。后果是误以为优化成功。修法是固定输入，多次运行，比较分布，再看正确性是否保持。

先知道，先别追

这一页先不展开采样与插桩 profiler 的内部实现、并发调试细节、分布式追踪和系统级性能剖析。它们都重要，但要等你已经会读日志、看调用栈、设基线、抓热点之后再学，才不会把工具当成答案。

工具选择矩阵

decision

很多调试失败卡在第一步：工具选错了。下面这张表把“看到什么、先做什么、会留下什么证据”写在一起。它的作用是帮你在卡住时先做一轮分流：先观察状态，先建立基线，还是先把问题缩成最小复现。

问题类型 首选工具 先看什么 最常见的误区 失败后果
程序直接报错 日志 + 调试器 错误栈、输入、分支和局部变量 只盯最后一行 看不见坏值是从哪一层传进来的
性能突然下降 基线 + profiler 总耗时、阶段耗时、调用次数 直接重构看起来可疑的函数 改了很多，速度却没变
偶发问题 复现记录 + 分段日志 环境、随机性、并发条件 只跑一次就下结论 把波动误判成规律
优化前后比较 benchmark + 回归测试 同输入、同环境、多次运行 把一次成功当成稳定结论 性能看起来更好，行为却坏了

如果工具顺序错了

常见后果是：先改代码，后找证据；先看细节，后找基线；先宣布优化成功，后发现输出错了。工具顺序错了，错误会被你自己放大。

正确的分流方式

先问这是功能错误、性能下降还是偶发问题；再决定是看日志、断点、profiler 还是 benchmark。这个分流一旦稳定下来，后面的学习和排查就会快很多。

工具选择示例
1. 错误栈里已经有报错行 -> 先看调用栈和局部变量
2. 页面突然慢了 3 倍 -> 先看基线，再跑 profiler
3. 偶尔失败、重跑又好了 -> 先固定输入和环境，再加分段日志
4. 新实现更快但边界错 -> 先补测试，再做 benchmark

这张矩阵的用法很直接：先判断问题需要哪类证据，再选工具。如果你把日志拿去替代 benchmark，或者把 profiler 拿去替代调用栈，最后都很难得到可靠结论。接下来做练习时，先别急着答题，先判断题目是哪一种问题类型，再选工具。

操作桥

bridge

如果你现在就要回到真实项目里，先按顺序把本页的方法落成几条命令和几个检查点。用 timeit 或固定脚本记录基线，再用 python -m cProfile 找热点，把最小坏输入写进回归测试，最后确认优化前后的输出一致。这样每一步都能留下可复查的证据。

真实排查的最小命令集
uv run python parse_rows.py samples/bad-row.csv
uv run python -m pdb parse_rows.py samples/bad-row.csv
uv run python -m cProfile -s cumulative parse_rows.py samples/bad-row.csv
uv run python -m timeit -s "from parse_rows import parse_file" "parse_file('samples/bad-row.csv')"
uv run pytest -q tests/test_parse_rows.py
动作 读什么 写什么 失败时的信号
timeit / 固定计时脚本 当前耗时 基线数字 不知道慢了多少
cProfile 函数调用次数和累计耗时 热点列表 只知道“慢”，不知道慢在哪一段
回归测试 最小坏输入 稳定断言 修复后又悄悄坏回去
对比输出 旧版本和新版本结果 变更说明 性能看起来提升，行为却不一致

如果只跑一次就下结论

你会把缓存、负载和随机波动当成事实。这个错误很隐蔽，因为它看起来像“已经测过了”，但实际上没有建立任何可重复的对比条件。

如果按桥接顺序做

你会得到一条从慢到快、从坏到稳的证据链：基线告诉你差多少，Profiler 告诉你差在哪，测试告诉你是否修对，输出对比告诉你有没有把行为改坏。

遇到下一次问题时，不要先找“最强工具”。先判断问题属于哪一类，再按桥接顺序把证据补齐。顺序稳定下来，调试和性能分析就会从临场反应变成可重复的动作。

复盘模板

record

学会排查之后，下一步是把过程留下来。复盘记录不是流水账，它让下次遇到同类问题时能更快复用。记录应该尽量短，但不能缺少关键字段：问题是什么、怎样复现、看到了什么证据、最后怎么验证。

字段 要写什么 为什么不能省 后续怎么用
问题描述 一句话说清现象和影响 没有问题边界，就不知道在修什么 以后看到类似现象时快速分类
最小复现 命令、输入、环境 没有复现就无法验证 直接复跑，确认是否还会出现
证据 日志、调用栈、基线、profile 没有证据就只有猜测 把证据和假设一一对照
修法 只改了什么、为什么只改这里 大改法很难复盘 后来再遇到同类问题时可以快速判断是否适用
验证 回归测试、benchmark、输出对比 没有验证就不知道改对没有 确认修复和优化都站得住
复盘记录示例
问题：parse_rows 在 bad-row.csv 上报错并变慢
复现：uv run python parse_rows.py samples/bad-row.csv
证据：validate 阶段 82ms，调用栈指向 int(row["id"])
修法：先修校验逻辑，再补回归测试
验证：同样输入不再报错，benchmark 与旧版一致

这个模板怎么帮你学习

它逼你把“我修好了”翻成可复查的文字。以后再写类似记录时，你只要照着问题、复现、证据、修法、验证五个字段写，复盘质量就会稳定得多。

如果模板写不出来

通常说明你还没把问题缩小到可验证的程度。回去补最小输入、补日志、补基线，再写复盘，记录会更自然。

练习

diagnostic

怎么答才算会

先把问题放到证据链里，再说工具。选择题和判断题要能马上解释为什么错项不对；开放题要能写出错因、后果、修法，或者写出基线、热点和验证方式。

检查重点

ordering诊断应用

7. 排序：调试一个稳定复现 bug 的合理顺序

候选：复现 / 加观察点 / 提出假设 / 最小修改 / 回归测试

先写顺序，再写一句理由。不要只抄词。

参考答案 / 评分要点

常见顺序是：复现 -> 加观察点 -> 提出假设 -> 最小修改 -> 回归测试。理由是先把问题固定下来，再增加证据，再只改一个变量，最后确认修复没有带来新问题。

matching工具边界

8. 匹配：print、logging、debugger、profiler、benchmark 各适合什么？

按“工具 - 观察对象 - 适合场景 - 边界”写。

参考答案 / 评分要点

要点：print 适合临时确认路径和变量；logging 适合长期保留上下文；debugger 适合复杂状态和调用栈；profiler 适合找热点；benchmark 适合验证改进是否稳定成立。答案里要写出边界，不要只给定义。

read日志诊断

9. 阅读日志：ERROR job failed job_id=42 error=timeout 你能得到哪些信息？

至少说出级别、事件、关联对象和错误原因。

参考答案 / 评分要点

应该能读出：这是 error 级别；事件是 job failed；关联对象是 job_id=42；错误原因是 timeout。关键是把日志拆成能定位的问题描述。

diagnostic错因/后果/修法

10. 诊断：同事没复现就改了很多代码。按错因/后果/修法回答。

参考答案 / 评分要点

错因：没有复现和证据就大范围改代码。后果：不知道哪处真的有效，还可能引入新 bug。修法：先写最小复现、再加观察点、再只做一个最小修改并回归测试。

diagnostic日志

11. 诊断：日志只写“failed”，无法排查。按错因/后果/修法回答。

参考答案 / 评分要点

错因：日志没有对象、阶段和原因。后果：事后无法把记录串成一次事件。修法：补上文件名、任务号、请求号、阶段名和错误原因，必要时再加 duration。

diagnostic性能

12. 诊断：凭感觉优化了一个函数，但整体没变快。按错因/后果/修法回答。

参考答案 / 评分要点

错因：没有先测量热点。后果：代码更复杂，性能却没改善。修法：先建立基线，再用 profiler 找累计耗时最高的部分，最后只改最大瓶颈。

multi-selectbenchmark

13. 多选：可靠 benchmark 应包含哪些？

展开反馈

应选固定输入、多次运行、正确性检查。只跑一次不够稳定。benchmark 的前提是比较同一条件下的多次结果，而且不能脱离正确性。

read调用栈

15. 阅读：异常最后一行在 int(row["id"])，为什么还要往上看调用栈？

回答要区分“报错位置”和“错误输入产生位置”。

参考答案 / 评分要点

最后一行告诉你报错发生在哪一行，不一定告诉你错误输入从哪里来。往上看调用栈可以找到是谁把坏值传进来的，第一次变坏的位置往往比最后报错的位置更重要。

read热点

16. 阅读：一个函数单次 1ms 但调用 200000 次，为什么可能是热点？

用累计耗时解释，不只看单次耗时。

参考答案 / 评分要点

因为累计耗时 = 单次耗时 × 调用次数。单次只有 1ms 并不代表总成本低；如果调用 200000 次，它可能占走大部分时间。Profiler 要看整体占比，不能只盯一个数字。

diagnostic正确性

17. 诊断：新实现快了 30%，但边界输入结果错了。按错因/后果/修法回答。

参考答案 / 评分要点

错因：为了速度删掉了边界处理或条件判断。后果：性能数字好看，但结果不可靠。修法：先补测试锁住边界行为，再改性能路径，最后重新比较 benchmark。

short日志

18. 简答：为什么日志要分级别？

从开发排查、生产告警和信息密度回答。

参考答案 / 评分要点

分级别是为了控制信息密度。开发时看 debug 能帮你定位局部状态，生产时更关注 warning 和 error 以便告警，info 则适合记录重要事件。好的分级能让你在需要的时候看见细节，在不需要的时候不被噪声淹没。

shortProfiler

19. 简答：为什么 profiler 要关注累计耗时和调用次数？

说明“单次慢”和“总成本高”的差别。

参考答案 / 评分要点

因为总耗时不等于单次最慢。一个函数单次很快，但被调用很多次，累计成本仍然可能最高。看调用次数能帮你找到这种“轻但频繁”的热点。

scenario性能排查

20. 场景：接口从 100ms 变成 2s，你如何排查？

可以按基线、日志分段、profile/外部依赖、最小修改、回归验证写。

参考答案 / 评分要点

参考答案通常会这样写：先建立旧版本和新版本的基线，再按阶段加日志，看慢在请求入口、业务逻辑、数据库、外部服务还是本地处理，然后用 profiler 或分段计时缩小范围，最后做最小修改并跑回归测试。重点看顺序，别只列工具清单。

interviewdebug

21. 面试追问：打印调试什么时候好用，什么时候不够？

说清快速观察和复杂状态/长期现场的边界。

参考答案 / 评分要点

好用的地方是快速看路径、变量和边界条件，适合小范围定位。它不够的地方是复杂状态、递归、异步链路和需要长期保留现场的场景，这时候更适合日志或调试器。

interviewbenchmark

22. 面试追问：为什么性能优化必须同时关注正确性？

回答要说明更快但结果错没有价值，以及测试的作用。

参考答案 / 评分要点

优化要在正确的前提下变快。如果结果错了，再好的 benchmark 也没有意义。测试先锁住正确性，benchmark 再证明改动确实改善了性能。

short日志

23. 简答：结构化日志相比纯文本日志有什么优势？

从查询、统计、关联请求或任务回答。

参考答案 / 评分要点

结构化日志把字段拆开，便于查询、筛选、统计和关联。比如你可以按 job_id 或 request_id 串联一组日志，也更容易自动提取 duration、状态码和错误原因。
