学习目标

goal

这一页把 Git 看成一张会分叉、会合并、也会被安全撤销的历史图。先判断每个命令是在读历史、改工作区、改暂存区，还是改共享历史，再决定下一步怎么动手。

这页读完之后

你能解释 git add 只进入暂存区，fetch 只观察不整合，共享历史更适合用 revert 追加反向提交。

历史模型

snapshot

Git 最重要的直觉很简单：每个 commit 都是一张项目快照，外加作者、时间、说明和指向父提交的链接。把它看成“文件夹备份”会丢掉最关键的一点，因为 commit 是历史图上的一个节点。

对象读什么写什么错了会怎样
commit父提交、快照内容、元数据历史节点你不知道某次改动从哪里来
branch当前指向的提交指针位置历史形状会被误读成复制文件
HEAD当前所在分支或提交当前工作点你会把“现在在哪”看错
代码
git log --oneline --graph --decorate
git show HEAD
git diff HEAD~1 HEAD

工作区、暂存区、历史

state

Git 的日常操作最容易卡住的地方，是把三层状态混成一层。工作区是你正在编辑的文件；暂存区是下一次提交的草稿；已提交历史是已经固定下来的节点。git add 把当前内容放进暂存区，git commit 才会把它写进历史。

步骤beforeafterreadswrites / mutates症状
git add工作区改动暂存区记录新草稿当前工作区内容暂存区git status 仍可能显示工作区继续 modified
git commit暂存区已有草稿生成新 commit暂存区历史图和当前分支指针新的提交节点出现
git restore --staged暂存区有草稿退回到未暂存暂存区暂存区文件还在工作区，但不再进入下一次提交
代码
git status
git diff
git add src/app.py
git diff --staged
git commit -m "Fix path parsing"

案例：同一个文件里有两种意图

case

假设 src/app.py 里同时出现两类改动：一类是在修路径解析 bug，另一类是在顺手整理注释。这个场景最容易把人带沟里，因为你看到的是一个文件，但 Git 看到的是可以拆开的两类意图。

步骤beforeafterreadswrites / mutates失败信号
git status工作区改了一个文件知道文件处于 modified工作区和暂存区状态只读先去看 diff，再决定要不要提交
git diff工作区有整份改动看见所有未暂存差异工作区与暂存区的差别只读如果你看不懂差异，说明还没分清两类意图
git add -p一份文件里混着两类修改把想要的块送进暂存区当前工作区 diff暂存区提交草稿里可能混进不该提交的那部分
git diff --staged暂存区已经有草稿确认提交内容只包含预期改动暂存区和历史只读能提前发现草稿里混入了注释整理或调试打印
代码
git status
git diff src/app.py
git add -p src/app.py
git diff --staged
git commit -m "Fix path parsing"
git add src/app.py
git commit -m "Tidy comments"

历史阅读

read

git log、git diff、git blame、git show 都在看历史，但它们回答的是不同问题。log 关心“历史发生了什么”；diff 关心“两个状态之间差了什么”；blame 关心“某一行最后是谁在什么提交改的”；show 关心“一个提交具体改了什么”。

命令适合回答的问题常见误用下一步通常看什么
git log历史图长什么样拿来比较当前改动show 或 diff
git diff两个状态差了什么拿来找某行的作者add、commit 或 restore
git blame某行最后由谁改过把作者等同于责任判断对应的 show
git show一个提交改了什么拿它看整条历史图提交父节点、背景上下文
代码
git blame src/app.py
git show <commit>
git diff --staged
git log --oneline --graph --decorate --all

分支、合并与重写

branch

分支本身很轻，通常只是一个指向提交的名字。你在分支上提交，名字跟着往前移动，所以新建分支不会复制整份仓库。合并则是让两条历史线重新汇合；如果两边改到同一位置，Git 无法替你决定业务语义，就会留下冲突给你处理。

步骤beforeafterreadswrites / mutates症状
git switch -c feature/login当前分支指向旧提交新分支指向同一提交当前 HEAD新增分支指针历史没有复制，只是多了一个名字
git merge main两条历史线分叉可能生成合并提交两边提交图当前分支历史冲突时必须人工选语义
git rebase main本地提交在旧基底上提交被重放到新基底当前分支和新基底提交身份和历史形状公共历史不宜随手重写
代码
git switch -c feature/login
git add src/login.py
git commit -m "Add login flow"
git merge main

案例：合并冲突怎么读

conflict

最容易让人慌的 Git 场面，常常是合并时出现冲突标记。冲突本身不表示 Git 坏了，只表示两条历史线都改到了同一块区域，Git 没法替你决定业务语义。它把双方内容都保留下来，等你来判断哪一边要留、要怎么合。

位置读什么写什么如果读错会怎样
<<<<<<< ours当前分支里的版本你这边想保留的内容把自己这边需要的逻辑删掉
=======分隔线不写内容把两边边界看混
>>>>>>> theirs另一条分支的版本对方分支改过的内容忽略别人已经解决的 bug 或功能
代码
<<<<<<< HEAD
return build_path(base, name)
=======
return normalize_path(base, name)
>>>>>>> main

这个例子里，当前分支和主分支都在改路径处理。你要先问两边分别解决了什么问题，再决定是否把 build_path 和 normalize_path 合并成一个更完整的实现。只保留一边，有时会把另一边解决过的 bug 又放回来。

步骤先看什么不要先做什么失败后果
读冲突块ours / theirs 各自改了什么先删分隔线你会失去对差异的判断
选语义哪边解决了什么问题机械保留一边把另一边已经修好的逻辑丢掉
跑测试冲突解决后的结果只看文件能不能保存能合并但行为错

这个案例可以直接接到练习 9、11、12：先判断历史形状，再判断共享风险，最后再决定怎么撤销或合并。看到冲突标记时，先读意图，再写结果，别把删符号当成解决冲突。

远程协作

remote

远程仓库是另一份 Git 仓库。fetch 只把远程信息拿回来，不会自动改你的当前分支；pull 相当于 fetch 后再合并或重放；push 把本地提交发到远程。三者的差别在于先观察、直接合入，还是发布本地提交。

命令作用读什么写什么常见风险
git fetch拿回远程更新远程引用和对象本地远程跟踪信息以为它已经合并了当前分支
git pull拉取并整合远程更新和当前分支当前分支历史工作区不干净时更容易混乱
git push发布本地提交本地提交历史远程分支引用推送前没确认自己改了什么
代码
git status
git fetch origin
git log --oneline --graph --decorate --all
git pull
git push origin feature/login

案例：push 被拒绝

sync

你在 feature/login 上提交完，执行 git push 后看到 non-fast-forward 或 updates were rejected。这类提示的意思很明确：远端分支已经先前进了，本地历史不能直接快进到远端上面。

提示历史关系安全动作跳过会怎样
non-fast-forward远端比本地更新fetch 后读图，再整合直接重试可能只是重复失败
本地和远端都前进历史已经分叉先选 merge 或 rebase直接 push 常常被拒绝
受保护分支仓库策略拦住了发布按团队流程走审查或合并请求强推会碰到权限或策略限制
代码
git push origin feature/login
git fetch origin
git log --oneline --graph --decorate --all
git rebase origin/main
git push origin feature/login

这段链条把“先 fetch、再读图、再决定 merge 还是 rebase、最后再 push”串成一条线。看到 push 被拒绝时，先判断是远端领先、历史分叉还是分支策略拦截，别直接强推。

操作桥：把课上命令接到真实仓库

bridge

如果你现在要去看一个真实仓库，最有用的是按层级做检查。先看历史图，再看当前状态，再看远端差距，最后才决定要不要改历史或发布提交。这个顺序能把大多数“我到底该用哪个命令”的问题，收缩成几步具体动作。

代码
git status
git diff
git diff --staged
git fetch origin
git log --oneline --graph --decorate --all

故障面板

failure

这块面板把四类常见失败放在一起看：看错状态、选错撤销层级、误改共享历史、把历史阅读和状态修改混成一件事。每个按钮都只是把你最容易走偏的场景单独拎出来。

撤销与定位

undo

Git 的撤销要先问一句：我要改的是哪一层状态。工作区里还没暂存的修改，可以丢弃；暂存区里的草稿，可以退回到未暂存；已经提交但还没共享的历史，通常可以改写；已经推到共享分支的提交，更稳妥的做法通常是用 revert 再加一个反向提交。

命令作用层级读什么写什么失败信号
git restore file.txt工作区当前文件内容覆盖工作区把还没提交的改动丢掉了
git restore --staged file.txt暂存区暂存区草稿撤回暂存文件还改着，但不再进入下一次提交
git revert 共享历史某个已提交节点新增反向提交共享历史保持连续，不需要别人改本地历史
git bisect历史诊断好/坏提交边界二分测试流程问题定位从感觉变成证据
代码
git restore file.txt
git restore --staged file.txt
git commit --amend
git revert HEAD
git bisect start

案例：共享分支里的错误提交怎么回

动作改什么会影响谁风险
git revert HEAD新增一个反向提交所有拉到这条历史的人历史多一个修正节点，但协作最稳
git reset --hard HEAD~1改写本地分支指针只影响你本地如果强推，别人本地历史会对不上
代码
git show HEAD
git revert HEAD
git push origin main

lab

这组实操把 Git 的三个状态层和三个常见动作串成一个最小工作流。你可以把它当成一份排查清单：先看当前状态，再决定要不要暂存，再决定要不要提交，再决定要不要同步远程。

代码

9scenario-transfer

Q: 9. 场景：你本地有两个提交，远端主分支也前进了。团队希望保留两边历史，不想改写已经共享的提交。你更偏向 merge 还是 rebase？为什么？

参考答案 / 评分要点

。如果已经是共享历史，通常更偏向 merge ，因为它保留分叉和汇合形状，不需要改写别人已经基于其工作的提交身份。 rebase 更适合本地未共享、想整理成线性历史的提交。答案要说清“是否共享”“是否需要改写身份”“历史形状是什么”三个点。

12diagnostic

Q: 12. 诊断：同事建议你在共享分支上 reset 后再强推。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

。错因：把已经共享的历史当成还能随意重写的本地草稿。后果：队友本地历史和远端历史分叉，拉取和合并都会更难。修法：共享历史优先用 revert 生成反向提交；如果真的必须重写，先沟通，再确认没人基于旧提交继续开发。

13synthesis

Q: 13. 费曼题：向零基础同学解释 Git 如何记录历史、支持分支协作，并在出错时安全撤销。

参考答案 / 评分要点

。参考答案可以把 Git 说成“历史图管理器”：commit 记录一次快照和父节点，分支是指向提交的名字，远程仓库让多人同步同一条历史。出错时，工作区和暂存区可以用 restore 退回，已经共享的历史更适合用 revert 追加反向提交；如果要定位 bug 从哪次提交开始出现，可以用 bisect 二分历史。顺序、层级、风险说清楚即可。
