学习单元 08

版本控制与 Git

这一页先把 Git 当成历史图来读。你会先分清 commit、分支、暂存区和远程各自管什么，再用一张状态表把“修改放在哪一层”看清，最后把撤销、合并和定位 bug 的命令接回同一条工作流。

读完本页应该能做什么

  • 把 commit 说成“带父节点和元数据的快照”。
  • 看懂工作区、暂存区和已提交历史的差别，并知道 git add、git commit、git restore --staged 分别改哪一层。
  • 按场景选命令：用 log、diff、blame、show、fetch、revert、bisect 解决不同问题。

学习目标

goal

这一页把 Git 看成一张会分叉、会合并、也会被安全撤销的历史图。先判断每个命令是在读历史、改工作区、改暂存区，还是改共享历史，再决定下一步怎么动手。

先抓住的三件事

  • commit 保存的是项目状态快照、父提交和元数据。
  • 工作区、暂存区、已提交历史是三层不同的状态，不能混着看。
  • 撤销、分支和远程协作都要先判断这次变化属于哪一层。

这页读完之后

你能解释 git add 只进入暂存区，fetch 只观察不整合，共享历史更适合用 revert 追加反向提交。

历史模型

snapshot

Git 最重要的直觉很简单：每个 commit 都是一张项目快照，外加作者、时间、说明和指向父提交的链接。把它看成“文件夹备份”会丢掉最关键的一点，因为 commit 是历史图上的一个节点。

这也是为什么分支只是一个可移动名字。分支指向某个 commit，新的提交会让这个名字往前移动。历史一旦出现分叉，Git 记录的还包括“这条线从哪里分出去、后来怎么汇合”。

对象 读什么 写什么 错了会怎样
commit 父提交、快照内容、元数据 历史节点 你不知道某次改动从哪里来
branch 当前指向的提交 指针位置 历史形状会被误读成复制文件
HEAD 当前所在分支或提交 当前工作点 你会把“现在在哪”看错
逐行拆解读一段历史图
git log --oneline --graph --decorate
git show HEAD
git diff HEAD~1 HEAD
先把 commit 当成“有父节点的快照”来读，后面的 log、blame、merge、rebase、bisect 才会落到同一个模型里。

工作区、暂存区、历史

state

Git 的日常操作最容易卡住的地方，是把三层状态混成一层。工作区是你正在编辑的文件；暂存区是下一次提交的草稿；已提交历史是已经固定下来的节点。git add 把当前内容放进暂存区，git commit 才会把它写进历史。

同一个文件可以同时存在“已暂存版本”和“工作区又改过的版本”。你 add 之后继续编辑，工作区就会比暂存区新一版；如果这时直接 commit，进历史的是暂存区那份，和你最新看到的工作区内容可能不同。

步骤 before after reads writes / mutates 症状
git add 工作区改动 暂存区记录新草稿 当前工作区内容 暂存区 git status 仍可能显示工作区继续 modified
git commit 暂存区已有草稿 生成新 commit 暂存区 历史图和当前分支指针 新的提交节点出现
git restore --staged 暂存区有草稿 退回到未暂存 暂存区 暂存区 文件还在工作区，但不再进入下一次提交
逐行拆解从修改到提交
git status
git diff
git add src/app.py
git diff --staged
git commit -m "Fix path parsing"
读法工作区 = 你现在看到的文件

这里的改动还没被选择进入下一次提交。先看 status 再决定是否暂存。

读法暂存区 = 下一次提交的草稿

如果工作区继续变化，暂存区不会自动更新。再提交前要重新检查 diff --staged。

读法提交历史 = 已固定节点

commit 改变的是历史图和当前分支位置，它不会把整个项目复制一遍。

案例：同一个文件里有两种意图

case

假设 src/app.py 里同时出现两类改动：一类是在修路径解析 bug，另一类是在顺手整理注释。这个场景最容易把人带沟里，因为你看到的是一个文件，但 Git 看到的是可以拆开的两类意图。

先跑 git status，你会看到文件已经改动。再跑 git diff，你能看见工作区当前到底改了什么。接着用 git add -p src/app.py 按块选择，把路径修复先放进暂存区，注释整理留在工作区。最后用 git diff --staged 确认进入 commit 的只有路径修复那一块，再提交。

步骤 before after reads writes / mutates 失败信号
git status 工作区改了一个文件 知道文件处于 modified 工作区和暂存区状态 只读 先去看 diff，再决定要不要提交
git diff 工作区有整份改动 看见所有未暂存差异 工作区与暂存区的差别 只读 如果你看不懂差异，说明还没分清两类意图
git add -p 一份文件里混着两类修改 把想要的块送进暂存区 当前工作区 diff 暂存区 提交草稿里可能混进不该提交的那部分
git diff --staged 暂存区已经有草稿 确认提交内容只包含预期改动 暂存区和历史 只读 能提前发现草稿里混入了注释整理或调试打印
逐行拆解把一个文件拆成两次提交
git status
git diff src/app.py
git add -p src/app.py
git diff --staged
git commit -m "Fix path parsing"
git add src/app.py
git commit -m "Tidy comments"
这条案例把三件事串起来了：一个文件可以有多种意图，add -p 可以拆草稿，diff --staged 是提交前最后一道检查。你现在应该能做什么：看见混合修改时，先拆意图，再提交。

历史阅读

read

git log、git diff、git blame、git show 都在看历史，但它们回答的是不同问题。log 关心“历史发生了什么”；diff 关心“两个状态之间差了什么”；blame 关心“某一行最后是谁在什么提交改的”；show 关心“一个提交具体改了什么”。

这里最常见的误判，是拿一个命令回答所有问题。比如你想知道“这行代码从哪来”，就该先用 blame 找到相关提交，再看对应的 show；如果你想知道“现在本地改了什么但还没提交”，要看的是 diff。

命令 适合回答的问题 常见误用 下一步通常看什么
git log 历史图长什么样 拿来比较当前改动 show 或 diff
git diff 两个状态差了什么 拿来找某行的作者 add、commit 或 restore
git blame 某行最后由谁改过 把作者等同于责任判断 对应的 show
git show 一个提交改了什么 拿它看整条历史图 提交父节点、背景上下文
逐行拆解追踪一行代码来源
git blame src/app.py
git show <commit>
git diff --staged
git log --oneline --graph --decorate --all
边界先记住：历史阅读和版本回退是两件事。读历史是为了理解，回退是为了改变状态；这两个动作在 Git 里会用到不同命令。

分支、合并与重写

branch

分支本身很轻，通常只是一个指向提交的名字。你在分支上提交，名字跟着往前移动，所以新建分支不会复制整份仓库。合并则是让两条历史线重新汇合；如果两边改到同一位置，Git 无法替你决定业务语义，就会留下冲突给你处理。

merge 和 rebase 处理历史的方式不一样。merge 保留分叉和汇合的形状；rebase 会把一串提交搬到新的基底上，让历史更线性，但也会重写提交身份。共享历史上已经有人基于旧提交工作时，重写要更谨慎。

步骤 before after reads writes / mutates 症状
git switch -c feature/login 当前分支指向旧提交 新分支指向同一提交 当前 HEAD 新增分支指针 历史没有复制，只是多了一个名字
git merge main 两条历史线分叉 可能生成合并提交 两边提交图 当前分支历史 冲突时必须人工选语义
git rebase main 本地提交在旧基底上 提交被重放到新基底 当前分支和新基底 提交身份和历史形状 公共历史不宜随手重写
逐行拆解功能分支的基本走法
git switch -c feature/login
git add src/login.py
git commit -m "Add login flow"
git merge main
这里先别钻进三方合并、cherry-pick、rebase --interactive。先把“分支是指针、merge 会汇合、rebase 会改历史身份”这三件事站稳，后面再扩展。

案例：合并冲突怎么读

conflict

最容易让人慌的 Git 场面，常常是合并时出现冲突标记。冲突本身不表示 Git 坏了，只表示两条历史线都改到了同一块区域，Git 没法替你决定业务语义。它把双方内容都保留下来，等你来判断哪一边要留、要怎么合。

看冲突时，先别急着删标记。先确认当前分支想保留什么，另一边分支改了什么，再决定直接选一边、合并两边，还是重写成第三种结果。冲突处理要保住意图，不能只把红字清掉。

位置 读什么 写什么 如果读错会怎样
<<<<<<< ours 当前分支里的版本 你这边想保留的内容 把自己这边需要的逻辑删掉
======= 分隔线 不写内容 把两边边界看混
>>>>>>> theirs 另一条分支的版本 对方分支改过的内容 忽略别人已经解决的 bug 或功能
逐行拆解冲突文件里会看到什么
<<<<<<< HEAD
return build_path(base, name)
=======
return normalize_path(base, name)
>>>>>>> main

这个例子里，当前分支和主分支都在改路径处理。你要先问两边分别解决了什么问题，再决定是否把 build_path 和 normalize_path 合并成一个更完整的实现。只保留一边，有时会把另一边解决过的 bug 又放回来。

步骤 先看什么 不要先做什么 失败后果
读冲突块 ours / theirs 各自改了什么 先删分隔线 你会失去对差异的判断
选语义 哪边解决了什么问题 机械保留一边 把另一边已经修好的逻辑丢掉
跑测试 冲突解决后的结果 只看文件能不能保存 能合并但行为错
这个案例可以直接接到练习 9、11、12：先判断历史形状，再判断共享风险，最后再决定怎么撤销或合并。看到冲突标记时，先读意图，再写结果，别把删符号当成解决冲突。

远程协作

remote

远程仓库是另一份 Git 仓库。fetch 只把远程信息拿回来，不会自动改你的当前分支；pull 相当于 fetch 后再合并或重放；push 把本地提交发到远程。三者的差别在于先观察、直接合入，还是发布本地提交。

协作里最稳的顺序通常是先 fetch，看清远端分支和本地分支的差距，再决定是 merge、rebase 还是直接 push。如果工作区不干净、历史关系没看清，直接 pull 往往只会把问题叠在一起。

命令 作用 读什么 写什么 常见风险
git fetch 拿回远程更新 远程引用和对象 本地远程跟踪信息 以为它已经合并了当前分支
git pull 拉取并整合 远程更新和当前分支 当前分支历史 工作区不干净时更容易混乱
git push 发布本地提交 本地提交历史 远程分支引用 推送前没确认自己改了什么
逐行拆解同步前先观察
git status
git fetch origin
git log --oneline --graph --decorate --all
git pull
git push origin feature/login
这里先不展开多远端、子模块或仓库镜像。先把“fetch 是观察、pull 是整合、push 是发布”固定住，就足够支撑大多数日常协作。

案例：push 被拒绝

sync

你在 feature/login 上提交完，执行 git push 后看到 non-fast-forward 或 updates were rejected。这类提示的意思很明确：远端分支已经先前进了，本地历史不能直接快进到远端上面。

这里最稳的动作是先 fetch，再看图。只有远端领先、本地只是落后时，才是一个简单的补同步场景；如果本地和远端都各自前进了，历史已经分叉，需要先决定用 merge 还是 rebase。判断依据不是哪个命令更熟，而是共享历史里有没有别人已经基于旧提交继续工作。

提示 历史关系 安全动作 跳过会怎样
non-fast-forward 远端比本地更新 fetch 后读图，再整合 直接重试可能只是重复失败
本地和远端都前进 历史已经分叉 先选 merge 或 rebase 直接 push 常常被拒绝
受保护分支 仓库策略拦住了发布 按团队流程走审查或合并请求 强推会碰到权限或策略限制
逐行拆解推送失败后的检查顺序
git push origin feature/login
git fetch origin
git log --oneline --graph --decorate --all
git rebase origin/main
git push origin feature/login
To origin
 ! [rejected] feature/login -> feature/login (non-fast-forward)
error: failed to push some refs to 'origin'
hint: Updates were rejected because the remote contains work that you do not have locally.
这段链条把“先 fetch、再读图、再决定 merge 还是 rebase、最后再 push”串成一条线。看到 push 被拒绝时，先判断是远端领先、历史分叉还是分支策略拦截，别直接强推。

操作桥：把课上命令接到真实仓库

bridge

如果你现在要去看一个真实仓库，最有用的是按层级做检查。先看历史图，再看当前状态，再看远端差距，最后才决定要不要改历史或发布提交。这个顺序能把大多数“我到底该用哪个命令”的问题，收缩成几步具体动作。

下一步动作 先看什么 遇到什么说明有问题 应该回到哪一节
确认本地改动 git status、git diff 工作区和暂存区内容不一致 工作区、暂存区、历史
确认提交内容 git diff --staged 暂存区里混进了不想提交的改动 状态与命令阅读
确认远端差距 git fetch、git log --graph --all 本地和远端历史已经分叉 远程协作
确认撤销层级 问自己这次改的是工作区、暂存区还是共享历史 直接用错了回退命令 撤销与定位
逐行拆解去看仓库时先跑的命令
git status
git diff
git diff --staged
git fetch origin
git log --oneline --graph --decorate --all
这部分把前面的模型接到真实仓库。每当你拿不准下一步命令时，先回到这张检查表。

故障面板

failure

这块面板把四类常见失败放在一起看：看错状态、选错撤销层级、误改共享历史、把历史阅读和状态修改混成一件事。每个按钮都只是把你最容易走偏的场景单独拎出来。

On branch feature/login
Changes to be committed:
  modified: src/app.py
Changes not staged for commit:
  modified: src/app.py
HINT: 先看工作区和暂存区是否已经不同。
场景 先看什么 不要先做什么 回到哪一节
工作区和暂存区看起来不一样 git diff 与 git diff --staged 直接 commit 工作区、暂存区、历史
推送被拒绝 git fetch、git log --graph --all 强推覆盖远端 远程协作

撤销与定位

undo

Git 的撤销要先问一句：我要改的是哪一层状态。工作区里还没暂存的修改，可以丢弃；暂存区里的草稿，可以退回到未暂存；已经提交但还没共享的历史，通常可以改写；已经推到共享分支的提交，更稳妥的做法通常是用 revert 再加一个反向提交。

git bisect 则是另一类动作。它用于定位 bug 是从哪次提交开始出现的。你给 Git 一个好提交和一个坏提交，它会二分历史，让你逐步确认中点，这样能把“猜是哪次改坏的”变成“跑几轮就能缩小范围”。

命令 作用层级 读什么 写什么 失败信号
git restore file.txt 工作区 当前文件内容 覆盖工作区 把还没提交的改动丢掉了
git restore --staged file.txt 暂存区 暂存区草稿 撤回暂存 文件还改着，但不再进入下一次提交
git revert <commit> 共享历史 某个已提交节点 新增反向提交 共享历史保持连续，不需要别人改本地历史
git bisect 历史诊断 好/坏提交边界 二分测试流程 问题定位从感觉变成证据
逐行拆解按层级撤销
git restore file.txt
git restore --staged file.txt
git commit --amend
git revert HEAD
git bisect start
先把撤销和诊断分开：restore、amend、revert 处理状态，bisect 处理定位。

案例：共享分支里的错误提交怎么回

假设你把一条打印密码的调试语句提交到了共享分支，队友已经拉走了那次提交。此时优先让共享历史继续向前，同时用反向提交抵消错误内容；直接把旧提交从历史里抹掉，会影响已经同步这段历史的人。

动作 改什么 会影响谁 风险
git revert HEAD 新增一个反向提交 所有拉到这条历史的人 历史多一个修正节点，但协作最稳
git reset --hard HEAD~1 改写本地分支指针 只影响你本地 如果强推，别人本地历史会对不上
逐行拆解共享历史的安全回退
git show HEAD
git revert HEAD
git push origin main
这段对照把“本地能重写”和“共享后要留痕”分开了。看到共享分支上的错误提交时，先选 revert，再把影响范围告诉队友，别直接改写历史。

离线实操

lab

这组实操把 Git 的三个状态层和三个常见动作串成一个最小工作流。你可以把它当成一份排查清单：先看当前状态，再决定要不要暂存，再决定要不要提交，再决定要不要同步远程。

场景 先看什么 不要先做什么 下一步证据
工作区刚改完 git status 和 git diff 直接 commit 哪些文件变了，改动是否单一
暂存后又继续改 git diff --staged 以为暂存区会自动更新 暂存草稿和最新文件是否一致
远程分支有更新 git fetch 后看历史图 直接 pull 盖过去 本地和远程是否已经分叉
逐行拆解一条完整的本地工作流
git status
git diff
git add -p src/app.py
git diff --staged
git commit -m "Fix empty path handling"
git fetch origin
git log --oneline --graph --decorate --all
这一步的目标是形成固定顺序：先确认状态，再选择层级，再进入历史，再看远程。

9. 场景：你本地有两个提交，远端主分支也前进了。团队希望保留两边历史，不想改写已经共享的提交。你更偏向 merge 还是 rebase？为什么？

回答要写出“为什么不改写共享历史”，不要只给工具名。

参考答案 / 评分要点

如果已经是共享历史，通常更偏向 merge，因为它保留分叉和汇合形状，不需要改写别人已经基于其工作的提交身份。rebase 更适合本地未共享、想整理成线性历史的提交。答案要说清“是否共享”“是否需要改写身份”“历史形状是什么”三个点。

12. 诊断：同事建议你在共享分支上 reset 后再强推。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

错因：把已经共享的历史当成还能随意重写的本地草稿。后果：队友本地历史和远端历史分叉，拉取和合并都会更难。修法：共享历史优先用 revert 生成反向提交；如果真的必须重写，先沟通，再确认没人基于旧提交继续开发。

synthesisfull chain

13. 费曼题：向零基础同学解释 Git 如何记录历史、支持分支协作，并在出错时安全撤销。

建议不少于 120 字，尽量把 commit、分支、远程、revert、bisect 串成一条线。

不要只列名词。要把“状态在哪一层、改动怎么流动、出错怎么回头”讲清楚。

参考答案 / 评分要点

参考答案可以把 Git 说成“历史图管理器”：commit 记录一次快照和父节点，分支是指向提交的名字，远程仓库让多人同步同一条历史。出错时，工作区和暂存区可以用 restore 退回，已经共享的历史更适合用 revert 追加反向提交；如果要定位 bug 从哪次提交开始出现，可以用 bisect 二分历史。顺序、层级、风险说清楚即可。
