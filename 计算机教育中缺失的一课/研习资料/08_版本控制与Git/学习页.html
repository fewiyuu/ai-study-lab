版本控制 · Git

Git 保存的不是“文件夹备份”，而是一条可分叉、可合并、可追溯的项目历史

版本控制让你回答：现在改了什么、为什么改、谁改的、什么时候改、哪个提交引入了 bug、如何并行开发、如何和别人同步。Git 是工具，核心是历史模型。

学完你应该能做什么

  1. 解释 commit 是项目快照和元数据。
  2. 区分工作区、暂存区、仓库历史。
  3. 读懂 branch、merge、rebase、remote 的边界。
  4. 安全处理撤销、冲突和 bug 定位。

概念地图

Git 把项目历史表示成提交图。每个 commit 指向父提交，包含一次项目状态快照和作者、时间、说明等元数据。分支只是指向某个 commit 的可移动名字；HEAD 表示你当前所在位置。

学习 Git 时，最重要的是知道自己正在改变哪一层：工作区文件、暂存区草稿、本地提交历史，还是远程共享历史。很多事故不是命令本身复杂，而是用户以为自己在改工作区，实际改写了别人也在使用的提交历史。

工作区修改→git add 暂存→git commit 快照→branch 指针移动→push/pull 同步远程
commit一次项目状态快照，加上作者、时间、说明和父提交指针。它是历史图上的节点。
staging area暂存区，决定下一次提交包含哪些修改。它不是历史，只是提交前的草稿篮。
branch指向提交的名字，不是复制一份完整项目。新提交会让当前分支指针向前移动。
remote远程仓库引用，例如 GitHub 上的 origin。远程同步影响协作边界。

1. 快照模型：Git 记录项目状态，不只是差异文本

你可以把 commit 直觉理解为“给项目拍了一张照片”。Git 内部会高效存储对象和差异，但概念上每个 commit 都代表项目在某一刻的完整状态。commit 还记录为什么这次变化存在，因此提交信息不是形式主义。

版本控制的价值不只是“后悔药”。它让你比较任意两个历史点，追踪某行代码来源，创建分支并行开发，用二分法定位 bug 引入点。没有 Git，你只能靠复制文件夹或压缩包，协作很快会混乱。

commit 的父指针让历史形成图，而不是一串孤立备份。普通提交通常有一个父提交，merge commit 有两个或多个父提交。理解这一点后，分支、合并、rebase、bisect 都会变得更清楚：它们都在改变或查询提交图中的关系。

正确示例查看历史和单次提交
git log --oneline --graph --decorate
git show HEAD
git diff HEAD~1 HEAD

判断句：commit 是带父节点和说明的历史节点，不是随手复制出来的文件夹。你现在应该能做什么：把 commit 解释为“有父节点的项目快照”，而不是单纯的一组文件。

2. 工作区、暂存区、仓库：Git 的三段式流程

工作区是你正在编辑的文件；暂存区是下一次 commit 的草稿；仓库历史是已经提交的记录。git add 不是“上传”，而是把当前修改放入暂存区。git commit 才会创建新的历史节点。

暂存区的意义是让你把不同意图的修改拆成多个提交。例如你同时改了 bug 和格式，应该分开提交，方便之后审查和回滚。git status 是你的仪表盘，写任何危险命令前都应该先看。

同一个文件可能同时有“已暂存版本”和“工作区又改过的版本”。这会让新手困惑：明明 add 过，为什么 status 还显示 modified？原因是 add 只记录当时那一刻的内容；之后继续编辑产生的是新的工作区修改，需要再次决定是否进入下一次 commit。

逐行拆解从修改到提交
git status
git diff
git add src/app.py
git diff --staged
git commit -m "Fix path parsing"
逐行拆解分块暂存让提交更清晰
git add -p src/app.py
# 逐块选择要进入本次提交的修改
git diff --staged
git commit -m "Fix empty path handling"

判断句：commit 前要看 git diff --staged，因为真正会进入历史的是暂存区，不是整个工作区。你现在应该能做什么：解释为什么一个文件可以“已修改但未暂存”，也可以“已暂存等待提交”。

3. 阅读历史：log、diff、blame 回答不同问题

git log 回答“历史发生了什么”；git diff 回答“两个状态之间改了什么”；git blame 回答“某行最后是谁在什么提交改的”。这些命令不是互相替代，而是从不同角度看历史。

读历史时不要只看文件内容，也要看提交信息、提交范围和上下文。一个好的 commit 应该有单一意图：修一个 bug、加一个功能、重构一个模块。把很多不相关改动塞进一个 commit，会让审查和定位 bug 变难。

选择命令时先说清问题。如果要看当前工作区还没提交的改动，用 git diff；要看已经暂存的改动，用 git diff --staged；要看某个提交具体改了什么，用 git show；要追踪某行代码来源，用 git blame 后再看对应提交。

正确示例追踪一行代码来源
git blame src/app.py
git show abc123
git log -- src/app.py

判断句：先确定你要问“历史、差异、某次提交、某行来源”中的哪一个，再选 Git 命令。你现在应该能做什么：根据问题选择看 log、diff、show 还是 blame。

4. 分支、合并与 rebase：并行历史如何汇合

分支是可移动指针。创建分支几乎不复制文件，因此很轻量。你在分支上提交，分支指针向前移动。合并把两个分支的历史汇合；如果同一位置被不同分支修改，Git 无法自动判断，就会产生冲突。

merge 保留分叉和汇合的形状，rebase 把一串提交“搬到”新的基底上，让历史更线性。rebase 已经推送并被别人基于其工作的公共历史时要谨慎，因为它会重写提交身份。

冲突不是 Git 失败，而是 Git 拒绝替人决定业务语义。冲突标记里的两边内容都要读：当前分支想保留什么，另一分支改了什么，最终代码是否同时满足两边意图。解决冲突后必须运行相关测试，因为“能合并”不等于“行为正确”。

正确示例功能分支工作流
git switch -c feature/login
git add .
git commit -m "Add login form"
git switch main
git merge feature/login
错误示例在不理解影响时重写公共历史
git rebase main
git push --force
# 如果别人已经基于旧历史工作，可能造成协作混乱

判断句：merge 和 rebase 都在处理提交图；区别是保留汇合节点，还是重写一串提交的基底。你现在应该能做什么：解释分支不是复制项目，而是一个指向提交的名字。

5. 远程协作：fetch、pull、push 分别做什么

远程仓库是另一个 Git 仓库，通常在 GitHub/GitLab 等平台。git fetch 下载远程信息但不自动改你的当前分支；git pull 等于 fetch 后再 merge 或 rebase；git push 把本地提交发送到远程。

协作时最重要的是减少意外：推送前看 status 和 log，拉取前确认工作区干净，冲突时读冲突标记并理解两边意图。不要用强推掩盖自己不理解的历史问题。

建议把 fetch 当成“先看远程发生了什么”的安全动作。它不会自动改你的工作分支，适合在 pull 前观察远端分支位置。pull 更方便，但它会把远端变化合进当前分支；如果工作区不干净或历史关系没看清，就更容易把问题混在一起。

逐行拆解同步远程前看清状态
git status
git fetch origin
git log --oneline --graph --decorate --all -n 20
git pull --rebase
git push origin main

判断句：fetch 只更新远程引用，pull 会改变当前分支，push 会影响远程共享历史。你现在应该能做什么：区分 fetch、pull、push，并知道 pull 可能引入合并或 rebase。

6. 撤销与定位 bug：先判断修改在哪一层

Git 撤销最容易出事故，因为同一个“撤销”可能作用于工作区、暂存区或历史。工作区未暂存修改可以丢弃；已暂存可以 unstage；已经提交但未共享可以改写；已共享历史通常更适合用 revert 创建反向提交。

git bisect 是定位 bug 引入提交的利器。你告诉 Git 一个坏提交和一个好提交，它会二分历史，让你测试中间点，逐步缩小范围。它体现了版本控制的真正价值：历史不是摆设，而是可查询的调试工具。

撤销前要先保护现场：看 git status，必要时提交临时分支或 stash，确认这次操作是否会丢工作区修改、改暂存区、移动分支指针或产生新提交。共享历史里的撤销优先用 revert，因为它不要求别人重写自己的历史。

正确示例按层级撤销与定位
git restore file.txt          # 丢弃工作区修改
git restore --staged file.txt # 从暂存区移出
git revert abc123             # 用新提交撤销旧提交
git bisect start              # 开始二分定位

判断句：撤销命令没有“通用安全答案”，安全性取决于修改在哪一层、是否已经共享。你现在应该能做什么：撤销前先问“我要改的是工作区、暂存区，还是已经提交的历史”。

7. 常见错误与修法

add 当成保存

git add 只进暂存区。后果是以为历史已记录，实际还没 commit。修法：继续 git commit，并用 status 检查三块区域。

提交太大太杂

后续审查和回滚困难，也难定位 bug 来源。修法：按意图拆提交，必要时用 git add -p 分块暂存。

冲突时盲删标记

可能丢掉一方逻辑，留下能编译但行为错的代码。修法：理解 ours/theirs，运行测试后再提交。

乱用 force push

重写公共历史影响队友。修法：先沟通，优先 revert；确需强推时使用更谨慎的 force-with-lease。

pull 前工作区不干净

本地修改和远端修改混在一起，冲突更难判断。修法：先 status，提交、stash 或清理工作区后再同步。

用 reset 撤销共享提交

可能让远程历史和别人本地历史分叉。修法：共享后的撤销优先用 revert 产生反向提交。

8. 完整知识链路

Git 的主线是：你在工作区修改文件，用 add 选择下一次提交内容，用 commit 创建快照节点；分支指向提交并随提交移动；merge/rebase 让历史汇合或重排；remote 让不同机器和不同人同步历史；log/diff/blame/show/bisect 让历史可读、可比较、可追责、可调试。学 Git 的关键不是背所有命令，而是每次操作都知道自己在移动哪个指针、修改哪一层状态。

真实协作里，一次健康修改通常是：先从干净工作区开始，创建分支，做小而单一的修改，用 status 和 diff 检查工作区，用 add 选择暂存内容，用 diff --staged 检查提交草稿，用 commit 写清意图；同步远程前 fetch 看历史图，必要时 merge 或 rebase，解决冲突后运行测试，再 push。出现错误时，先判断问题在工作区、暂存区、本地历史还是共享历史，再选 restore、restore --staged、commit amend、revert 或 bisect。

9. 真实场景与排查手册

Git 不是“保存按钮”，而是记录项目历史的系统。工作区保存当前文件，暂存区决定下一次提交包含什么，提交历史记录已经固定下来的快照。

分支只是指向提交的可移动名字，合并和变基都是在处理提交关系。理解提交图，比背命令更可靠。看到冲突时，重点不是害怕，而是判断同一段内容被两个分支怎样改了。

真实使用中，最常见风险是把无关改动混进提交、在没看 diff 前提交、把工作区误当成历史。每次提交前用 status 和 diff 做一次小审查。

提交前不看 diff

git status 只告诉你哪些文件变了，git diff 才告诉你行为改了什么。

暂存区混乱

git add . 很方便，也容易带入临时文件。小提交优先按文件或 hunk 暂存。

冲突误删

冲突标记里的两边内容都要读懂，再决定保留、合并或重写。不要机械删标记。

提交习惯

小范围改动 -> 看 diff -> 暂存相关文件 -> 写清提交信息 -> 必要时跑测试。
