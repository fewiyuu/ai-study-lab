delivery workflow

代码打包与分发：把运行条件一起交出去

代码离开你的电脑以后，问题往往不在某一行源码，而在运行条件没有被交出去：语言版本、依赖范围、锁定结果、数据文件、入口命令、环境变量、系统库和容器配置。

这页用一个很小的 Python CLI 包做主线。你会看到同一份代码怎样从源码目录变成可安装 wheel，再在干净环境里验证入口命令。重点不是背某个工具的参数，而是学会问：目标环境靠什么得到同样的代码、依赖、配置和启动方式？

读完本页应该能做什么

  • 解释依赖、传递依赖、运行时、制品、lockfile 和容器各解决哪一层问题。
  • 读懂一个最小 pyproject.toml：项目名、版本、依赖、开发依赖和 CLI 入口。
  • 用干净环境验证 wheel，而不是只在源码目录里运行。
  • 诊断 ModuleNotFoundError、版本冲突、漏打包资源、入口命令缺失和 secret 入镜像。

学习目标

goal

本单元接在 Git 之后：Git 让你知道一份代码从哪里来，打包与分发让别人知道这份代码怎么装、怎么跑、怎么验证。交付的对象不只是源码目录，还包括依赖声明、运行时约束、构建制品、安装入口和运行配置。

这页先解决的事

  • 把“我的机器能跑”拆成代码、依赖、环境、制品、配置、入口六个条件。
  • 用一个 hello-tool CLI 包走完声明、构建、安装、验证。
  • 把常见失败映射到可检查对象，少靠猜。

这页先不展开

正式发布到 PyPI、CI/CD、签名、SBOM、私有仓库和生产编排都先放到后面。现在先保证本地能构建出可追溯制品，并能在干净环境里安装运行。

分发链路

flow

看一个最小例子：app.py 里写了 import requests，你把文件发给同学。同学的新机器上没有 requests，运行时就会报 ModuleNotFoundError。这时源码可能完全正确，真正缺的是“运行条件”的交接。

失败入口只发源码，没有依赖声明
# app.py
import requests

print(requests.get("https://example.com", timeout=5).status_code)

# 新机器上运行：
# ModuleNotFoundError: No module named 'requests'

打包分发就是把这些条件写出来，让工具能重建它们。源码说明程序逻辑，项目元数据说明包名、版本和依赖，lockfile 记录一次确定的解析结果，构建制品承载可安装对象，干净环境验证证明它没有依赖开发机的历史状态。

阶段 输入 产物 先检查什么 常见失败
声明 源码和运行需求 pyproject.toml / package.json 运行依赖是否写进元数据 本机有包，用户机器没有
解析 依赖范围 精确依赖集合或 lockfile 直接依赖和传递依赖版本 版本冲突，今天装出另一套版本
构建 源码、元数据、包数据 wheel、sdist、二进制、镜像 制品里是否含必要模块和资源 源码目录能跑，安装后缺文件
安装验证 制品和干净环境 可执行命令或可导入包 入口命令、版本输出、最小功能 装了但不知道怎么运行

依赖声明：让工具知道该装什么

dependency

依赖指程序运行或构建必须能找到的外部组件，不等同于“我电脑上刚好装过的库”。项目应把依赖写进元数据，Python 常见位置是 pyproject.toml。运行依赖和开发依赖要分开：用户运行 CLI 不一定需要 pytest，开发者才需要测试和格式化工具。

配置阅读最小 Python CLI 元数据
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "hello-tool"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = ["requests>=2.31,<3"]

[project.optional-dependencies]
dev = ["pytest", "ruff"]

[project.scripts]
hello-tool = "hello_tool.cli:main"

读这段配置时别只看包名。requires-python 限定运行时，dependencies 是用户运行时必须有的库，dev 是开发流程需要的工具，project.scripts 决定安装后出现什么命令。

字段 说明什么 写错后的症状
name / version 制品身份和版本 无法确认用户装的是哪一份
requires-python 支持的 Python 版本 旧解释器装上后运行失败
dependencies 运行时依赖 ModuleNotFoundError 或解析冲突
optional-dependencies.dev 开发工具组 把测试工具塞进用户安装路径，或者开发环境缺测试工具
project.scripts 安装后的 CLI 入口 用户装完仍不知道运行哪个命令

环境隔离：不要把全局环境当项目环境

environment

全局安装会把多个项目绑到同一套历史状态上。项目 A 升级了库，项目 B 可能突然坏掉；终端用一个 Python，编辑器或测试命令用另一个 Python，也会出现“我明明安装了，为什么找不到”的错觉。

独立环境的价值是可重复：新机器按同样命令创建环境、安装依赖、运行测试，结果应该能复现。排查时先看运行命令实际使用哪个解释器，再看依赖装到了哪里。

环境检查解释器和安装位置要对上
uv venv
uv pip install -e .
python -c "import sys; print(sys.executable)"
python -m pip show requests
hello-tool --version
状态空环境

还没有项目依赖。这里能排除“开发机以前装过”的干扰。

状态包和依赖进入环境

uv pip install -e . 会读取项目元数据，安装依赖并注册入口命令。

状态入口命令可运行

如果 hello-tool 找不到，先检查入口点、安装环境和 PATH，先别急着改业务代码。

包与制品：源码目录不等于交付物

artifact

源码目录里有测试、文档、开发配置和临时文件；安装包里应该包含运行所需模块、包数据和入口元数据。构建制品是交给别人安装的对象，应该能追溯到项目名、版本和构建来源。

下面这组命令刻意用“构建 wheel -> 新环境安装 wheel -> 运行入口”的顺序。它检查分发链路是否完整，源码目录刚好能跑只能当作开发阶段的信号。

离线实操从源码到 wheel 再到入口命令
mkdir -p package-lab/src/hello_tool
cd package-lab

cat > pyproject.toml <<'TOML'
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "hello-tool"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = []

[project.scripts]
hello-tool = "hello_tool.cli:main"
TOML

cat > src/hello_tool/cli.py <<'PY'
def main():
    print("hello tool 0.1.0")
PY

uv build
uv venv .venv-test
uv pip install --python .venv-test dist/*.whl
# macOS/Linux:
.venv-test/bin/hello-tool
# Windows PowerShell:
.venv-test\Scripts\hello-tool.exe
python -m zipfile -l dist/*.whl
命令 读取什么 写出什么 失败时先看
uv build pyproject.toml、包目录、构建后端 dist/*.whl 和源码包 包名、构建后端、包目录结构
uv pip install --python .venv-test dist/*.whl wheel 元数据和目标环境 安装后的包和脚本入口 是否装到目标解释器
.venv-test/bin/hello-tool 或 .venv-test\Scripts\hello-tool.exe 入口点和已安装模块 命令输出 project.scripts 和 PATH
zipfile -l dist/*.whl wheel 文件内容 文件清单 模块或资源文件是否漏进制品

案例：源码目录能跑，安装后找不到模板

case

有些分发问题看起来像依赖错误，实际是资源只存在于源码目录，没有进入制品。假设 hello-tool 有一个默认模板文件：开发时你从仓库根目录运行，代码能读到 templates/default.txt；打成 wheel 后，用户安装到另一个环境，运行命令却报 FileNotFoundError。这时先检查 wheel 里到底有没有这个文件。

稳定做法是把运行时需要的资源放进包目录，并用包资源 API 读取。这样代码不依赖当前工作目录，也不要求用户从源码仓库根目录启动。

失败写法依赖当前工作目录
# 开发时你刚好在仓库根目录，所以这行能跑
template = Path("templates/default.txt").read_text(encoding="utf-8")

# 用户安装 wheel 后从任意目录运行：
# FileNotFoundError: [Errno 2] No such file or directory: 'templates/default.txt'
修正思路把资源放进包里，再从包读取
from importlib.resources import files

def load_template() -> str:
    path = files("hello_tool").joinpath("templates/default.txt")
    return path.read_text(encoding="utf-8")
检查点 你在验证什么 通过信号 失败信号 下一步
资源放置位置 模板是否在包目录内，避免只存在于仓库根目录 src/hello_tool/templates/default.txt 只有 templates/default.txt 在仓库根目录 移动到包目录或明确配置包数据
读取方式 代码是否依赖当前工作目录 使用 importlib.resources 从包读取 使用相对路径直接读文件 改成包资源读取，或者传入明确配置路径
wheel 内容 制品里是否真的包含模板 zipfile -l dist/*.whl 能看到模板 源码有模板，wheel 清单没有模板 修正包数据配置后重新构建
干净环境运行 用户路径里没有源码目录时是否仍能运行 新环境安装 wheel 后命令正常输出 只要离开源码目录就失败 回到制品内容和读取方式排查

这类故障的关键动作是查制品。源码目录能跑只能证明开发机路径碰巧满足假设；wheel 里有资源、安装后能从任意目录运行，才说明交付物本身完整。

操作桥：把一个 Python CLI 交给别人前怎么验

bridge

交付前可以按一条固定链路自测：从 Git 里确认版本来源，从项目元数据确认运行条件，从构建命令生成制品，再换一个干净环境安装它。每一步都要留下一个可观察结果，否则出了问题就不知道该回到哪一层。

交付前自测从版本来源到最小运行
git rev-parse --short HEAD
git status --short
uv build
python -m zipfile -l dist/*.whl
uv venv .venv-release-test
uv pip install --python .venv-release-test dist/*.whl
# macOS/Linux:
.venv-release-test/bin/hello-tool --help
.venv-release-test/bin/hello-tool --version
# Windows PowerShell:
.venv-release-test\Scripts\hello-tool.exe --help
.venv-release-test\Scripts\hello-tool.exe --version
步骤 读什么 写什么 结果应该能说明什么 失败时回到哪一节
git rev-parse --short HEAD 当前提交 只读 制品来源能追溯 08 版本控制与 Git
git status --short 工作区和暂存区 只读 构建前没有未确认改动混进去 08 的状态层级
uv build 项目元数据和包目录 dist/ 制品 项目能被构建后端识别 包与制品
zipfile -l wheel 文件 只读清单 模块和资源是否进入制品 资源文件案例
新环境安装 wheel 制品和目标解释器 已安装包和入口脚本 交付物不依赖源码目录 环境隔离
--help / --version 入口命令和元数据 命令输出 用户知道如何启动，也能报告版本 依赖声明与入口字段

把这张表写进 README 时，不需要把所有排查细节都塞进去。README 给用户的是最短成功路径和最常见失败的入口；维护者自己保留更完整的发布检查表。两者都要说清楚版本、安装、运行和配置，但粒度不同。

版本约束与 lockfile：兼容范围和精确结果分开看

version

版本范围表达的是兼容承诺，例如 requests>=2.31,<3。它说的是“2.31 以上、3.0 以前的版本应该能用”。lockfile 记录的是某一次解析后的精确结果，包括直接依赖和传递依赖。

库项目通常更关心兼容范围，因为它要和别人的项目组合；应用和服务更关心可复现部署，通常会锁住完整依赖集合。两者服务的目标不同。

对象 表达什么 适合解决 不能替代什么
版本范围 允许安装哪些版本 库的兼容边界、最低功能要求 一次确定的部署结果
lockfile 这一次解析出的精确版本集合 应用复现、团队一致安装、回滚排查 对外兼容承诺和测试
语义化版本 MAJOR.MINOR.PATCH 的兼容暗示 估计升级风险 真实测试；并非所有项目都严格遵守
判断顺序：先问这是库还是应用，再决定写宽范围还是锁精确版本。生产路径即使用了语义化版本，也要靠测试确认。

案例：版本冲突不是换个版本号就完事

conflict

依赖冲突常见于两条依赖链同时要求同一个包，但版本范围互不兼容。比如你的项目直接依赖 reporter>=2，另一个插件还停在 reporter<2。安装工具不是“不会装包”，它是在告诉你：同一个环境里无法同时满足两组约束。

排查时要先画出依赖树，分清直接依赖和传递依赖。直接依赖是你在项目元数据里写下的库；传递依赖是这些库自己需要的库。冲突修复不能只盯报错最后一行，要回到哪条链把范围收得太死、哪条链可以升级、是否应该拆分环境。

冲突信号两个依赖范围互斥
Because plugin-a depends on reporter <2
and hello-tool depends on reporter >=2,
no versions of plugin-a match the requested requirements.

Resolution failed: cannot satisfy reporter<2 and reporter>=2 in one environment.
动作 要回答的问题 看到什么算有进展 不建议先做什么
找直接依赖 冲突包是谁引入的？ 能说出 plugin-a -> reporter<2 这条链 盲目删除 lockfile
看可升级路径 插件是否有支持 reporter>=2 的新版？ 找到兼容新版或迁移说明 强行钉一个版本忽略测试
调整约束 项目自己的范围是否写得过窄？ 范围能表达真实兼容边界 把范围放到无限宽
拆分环境 两个工具是否必须在同一环境里共存？ 测试工具和运行工具可以分开安装 把所有工具塞进运行依赖

这也是 lockfile 的价值：它记录一次成功解析的结果。下次冲突出现时，你可以对比新的解析结果和旧的锁定结果，知道是哪条依赖链发生了变化。修完以后，重新在干净环境安装并跑最小测试，确认解决的是冲突本身，而不是把错误推迟到运行时。

容器边界：它固定一部分系统环境，不固定一切

container

容器镜像把应用文件、运行时、系统包和默认命令放进一个分层文件系统。它能减少“服务器系统不同”“缺系统库”的问题，但仍依赖宿主内核、CPU 架构、网络、数据卷、环境变量和外部服务。

边界阅读最小 Dockerfile 思路
FROM python:3.12-slim
WORKDIR /app
COPY dist/*.whl /tmp/
RUN pip install /tmp/*.whl
CMD ["hello-tool"]
放进镜像的内容 仍要外部提供的内容 写错后的后果
应用文件、依赖、运行时、默认命令 secret、数据库地址、外部服务权限 凭据泄露或环境切换困难
一部分系统库和文件系统层 宿主内核、CPU 架构、网络策略 本地镜像能跑，服务器架构或网络不匹配
镜像默认配置 运行时环境变量、卷挂载、端口映射 数据丢失、配置写死、难以回滚

故障面板：把失败映射到可检查对象

failure

分发失败时，先判断失败发生在声明、解析、构建、安装还是运行阶段。不要一看到报错就重写业务代码。下面三个按钮模拟最常见的错误信号。

ModuleNotFoundError: No module named 'requests'
HINT: 先检查运行依赖是否写进 pyproject.toml，再确认安装环境和运行解释器是同一个。
症状 常见原因 先看什么 修法
ModuleNotFoundError 依赖未声明、安装环境不对 pyproject.toml、解释器路径、pip show 补运行依赖，重新安装到目标环境
安装包运行时报 File not found 模板、静态资源或数据文件没有进制品 wheel 文件清单、包数据配置 修正包数据配置，重新构建并检查内容
版本解析失败 两个依赖要求互斥版本 依赖树和版本范围 升级相关包、调整范围或拆分环境
镜像泄露凭据 把 secret 写进 Dockerfile 或构建参数 镜像历史、Dockerfile、部署配置 撤销凭据，改为运行时注入

发布前检查：从“能跑”走到“能交付”

checklist

做完一个包以后，至少在干净环境里跑一次完整链路。这里的“干净”指没有你开发机里残留的全局包、环境变量、临时路径和源码目录假设。

最小检查顺序

  1. 确认项目名、版本、运行时版本写清。
  2. 确认运行依赖和开发依赖没有混放。
  3. 构建 wheel 或目标制品，并记录来源。
  4. 在新环境安装制品，不从源码目录直接运行。
  5. 运行入口命令、版本输出和一个最小功能。
  6. 检查制品内容是否包含必要模块和资源。

容易误判的地方

源码目录能运行，只说明开发环境当前可用；干净环境能安装并运行制品，才说明交付路径可用。README 也要写给目标用户：用什么版本、怎么创建环境、怎么安装、怎么运行、哪些配置需要自己提供。

README 是交付链路的一部分。一个用户拿到项目后，最先需要知道的是：该用哪个运行时版本、如何创建隔离环境、安装哪个制品、执行哪个入口命令、需要自己提供哪些配置。少写任何一项，用户都会把运行条件反向猜一遍。

README 项 用户拿它做什么 缺失后的典型问题
运行时版本 选择 Python、Node 或系统平台版本 旧解释器安装成功，运行时语法或 API 失败
安装命令 知道装源码、wheel，还是从包索引安装 用户只复制源码，依赖和入口都没有注册
入口命令 装完后能直接运行最小示例 包已经安装，但不知道该敲哪个命令
配置与 secret 区分示例配置和真实凭据 把 token 写进仓库、镜像或聊天记录
验证命令 判断安装是否真的成功 出错时没有共同的复现起点

把这张表和前面的干净环境验证连起来：README 负责告诉别人怎么重建条件，发布前自测负责证明这些说明确实能跑通。练习区的 README 题也按这个标准自查，不要只写“安装依赖并运行”。如果用户把完整错误、运行命令和版本号发回来，维护者就能沿着依赖、环境、制品、入口和配置逐层复现；如果这些信息缺失，排查会重新退回口头猜测。维护者也应在发布说明里写清本次制品对应的版本或提交，并保留最小复现命令，方便用户报告问题时对齐同一份代码。

9. 诊断：别人运行时报 ModuleNotFoundError: No module named 'requests'。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

错因可能是运行依赖没写进元数据，或者依赖装到了另一个解释器环境。后果是源码逻辑没问题，但目标环境找不到库而无法启动。修法是补 dependencies，在独立环境里重新安装项目，并用 python -m pip show requests 和最小运行命令验证。

diagnosticartifact

10. 诊断：源码目录里有模板文件，安装 wheel 后运行时报 File not found。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

错因通常是模板或数据文件在源码目录里存在，但没有被包数据配置纳入 wheel。后果是开发目录能跑，安装包缺文件。修法是检查 wheel 文件清单，修正包数据配置，重新构建并在干净环境安装验证。

11. 判断：容器镜像能解决所有部署问题，包括 secret、外部数据库、网络权限和 CPU 架构。

diagnosticsecret

12. 诊断：Dockerfile 里写了 ENV API_TOKEN=...。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

错因是把运行时 secret 写进镜像层。后果是任何拿到镜像或镜像历史的人都可能读到凭据，而且撤回困难。修法是立刻轮换泄露 token，改为运行时环境变量、密钥管理或部署平台注入，并检查镜像历史和构建日志。
