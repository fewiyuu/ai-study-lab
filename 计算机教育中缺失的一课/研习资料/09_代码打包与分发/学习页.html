代码打包与分发：把运行条件一起交出去

代码离开你的电脑以后，问题往往不在某一行源码，而在运行条件没有被交出去：语言版本、依赖范围、锁定结果、数据文件、入口命令、环境变量、系统库和容器配置。

学习目标

goal

本单元接在 Git 之后：Git 让你知道一份代码从哪里来，打包与分发让别人知道这份代码怎么装、怎么跑、怎么验证。交付的对象不只是源码目录，还包括依赖声明、运行时约束、构建制品、安装入口和运行配置。

这页先不展开

正式发布到 PyPI、CI/CD、签名、SBOM、私有仓库和生产编排都先放到后面。现在先保证本地能构建出可追溯制品，并能在干净环境里安装运行。

分发链路

flow

看一个最小例子：app.py 里写了 import requests，你把文件发给同学。同学的新机器上没有 requests，运行时就会报 ModuleNotFoundError。这时源码可能完全正确，真正缺的是“运行条件”的交接。

代码
# app.py
import requests

print(requests.get("https://example.com", timeout=5).status_code)

# 新机器上运行：
# ModuleNotFoundError: No module named 'requests'

打包分发就是把这些条件写出来，让工具能重建它们。源码说明程序逻辑，项目元数据说明包名、版本和依赖，lockfile 记录一次确定的解析结果，构建制品承载可安装对象，干净环境验证证明它没有依赖开发机的历史状态。

阶段输入产物先检查什么常见失败
声明源码和运行需求pyproject.toml / package.json运行依赖是否写进元数据本机有包，用户机器没有
解析依赖范围精确依赖集合或 lockfile直接依赖和传递依赖版本版本冲突，今天装出另一套版本
构建源码、元数据、包数据wheel、sdist、二进制、镜像制品里是否含必要模块和资源源码目录能跑，安装后缺文件
安装验证制品和干净环境可执行命令或可导入包入口命令、版本输出、最小功能装了但不知道怎么运行

依赖声明：让工具知道该装什么

dependency

依赖指程序运行或构建必须能找到的外部组件，不等同于“我电脑上刚好装过的库”。项目应把依赖写进元数据，Python 常见位置是 pyproject.toml。运行依赖和开发依赖要分开：用户运行 CLI 不一定需要 pytest，开发者才需要测试和格式化工具。

代码
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "hello-tool"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = ["requests>=2.31,<3"]

[project.optional-dependencies]
dev = ["pytest", "ruff"]

[project.scripts]
hello-tool = "hello_tool.cli:main"

读这段配置时别只看包名。requires-python 限定运行时，dependencies 是用户运行时必须有的库，dev 是开发流程需要的工具，project.scripts 决定安装后出现什么命令。

字段说明什么写错后的症状
name / version制品身份和版本无法确认用户装的是哪一份
requires-python支持的 Python 版本旧解释器装上后运行失败
dependencies运行时依赖ModuleNotFoundError 或解析冲突
optional-dependencies.dev开发工具组把测试工具塞进用户安装路径，或者开发环境缺测试工具
project.scripts安装后的 CLI 入口用户装完仍不知道运行哪个命令

环境隔离：不要把全局环境当项目环境

environment

全局安装会把多个项目绑到同一套历史状态上。项目 A 升级了库，项目 B 可能突然坏掉；终端用一个 Python，编辑器或测试命令用另一个 Python，也会出现“我明明安装了，为什么找不到”的错觉。

环境检查解释器和安装位置要对上复制

包与制品：源码目录不等于交付物

artifact

源码目录里有测试、文档、开发配置和临时文件；安装包里应该包含运行所需模块、包数据和入口元数据。构建制品是交给别人安装的对象，应该能追溯到项目名、版本和构建来源。

离线实操从源码到 wheel 再到入口命令复制

命令读取什么写出什么失败时先看
uv buildpyproject.toml、包目录、构建后端dist/*.whl 和源码包包名、构建后端、包目录结构
uv pip install --python .venv-test dist/*.whlwheel 元数据和目标环境安装后的包和脚本入口是否装到目标解释器
.venv-test/bin/hello-tool 或 .venv-test\Scripts\hello-tool.exe入口点和已安装模块命令输出project.scripts 和 PATH
zipfile -l dist/*.whlwheel 文件内容文件清单模块或资源文件是否漏进制品

案例：源码目录能跑，安装后找不到模板

case

有些分发问题看起来像依赖错误，实际是资源只存在于源码目录，没有进入制品。假设 hello-tool 有一个默认模板文件：开发时你从仓库根目录运行，代码能读到 templates/default.txt；打成 wheel 后，用户安装到另一个环境，运行命令却报 FileNotFoundError。这时先检查 wheel 里到底有没有这个文件。

失败写法依赖当前工作目录复制

修正思路把资源放进包里，再从包读取复制

检查点你在验证什么通过信号失败信号下一步
资源放置位置模板是否在包目录内，避免只存在于仓库根目录src/hello_tool/templates/default.txt只有 templates/default.txt 在仓库根目录移动到包目录或明确配置包数据
读取方式代码是否依赖当前工作目录使用 importlib.resources 从包读取使用相对路径直接读文件改成包资源读取，或者传入明确配置路径
wheel 内容制品里是否真的包含模板zipfile -l dist/*.whl 能看到模板源码有模板，wheel 清单没有模板修正包数据配置后重新构建
干净环境运行用户路径里没有源码目录时是否仍能运行新环境安装 wheel 后命令正常输出只要离开源码目录就失败回到制品内容和读取方式排查

操作桥：把一个 Python CLI 交给别人前怎么验

bridge

交付前可以按一条固定链路自测：从 Git 里确认版本来源，从项目元数据确认运行条件，从构建命令生成制品，再换一个干净环境安装它。每一步都要留下一个可观察结果，否则出了问题就不知道该回到哪一层。

代码
git rev-parse --short HEAD
git status --short
uv build
python -m zipfile -l dist/*.whl
uv venv .venv-release-test
uv pip install --python .venv-release-test dist/*.whl
# macOS/Linux:
.venv-release-test/bin/hello-tool --help
.venv-release-test/bin/hello-tool --version
# Windows PowerShell:
.venv-release-test\Scripts\hello-tool.exe --help
.venv-release-test\Scripts\hello-tool.exe --version

版本约束与 lockfile：兼容范围和精确结果分开看

version

版本范围表达的是兼容承诺，例如 requests>=2.31,<3。它说的是“2.31 以上、3.0 以前的版本应该能用”。lockfile 记录的是某一次解析后的精确结果，包括直接依赖和传递依赖。

对象表达什么适合解决不能替代什么
版本范围允许安装哪些版本库的兼容边界、最低功能要求一次确定的部署结果
lockfile这一次解析出的精确版本集合应用复现、团队一致安装、回滚排查对外兼容承诺和测试
语义化版本MAJOR.MINOR.PATCH 的兼容暗示估计升级风险真实测试；并非所有项目都严格遵守

判断顺序：先问这是库还是应用，再决定写宽范围还是锁精确版本。生产路径即使用了语义化版本，也要靠测试确认。

案例：版本冲突不是换个版本号就完事

conflict

依赖冲突常见于两条依赖链同时要求同一个包，但版本范围互不兼容。比如你的项目直接依赖 reporter>=2，另一个插件还停在 reporter<2。安装工具不是“不会装包”，它是在告诉你：同一个环境里无法同时满足两组约束。

冲突信号两个依赖范围互斥复制

动作要回答的问题看到什么算有进展不建议先做什么
找直接依赖冲突包是谁引入的？能说出 plugin-a -> reporter<2 这条链盲目删除 lockfile
看可升级路径插件是否有支持 reporter>=2 的新版？找到兼容新版或迁移说明强行钉一个版本忽略测试
调整约束项目自己的范围是否写得过窄？范围能表达真实兼容边界把范围放到无限宽
拆分环境两个工具是否必须在同一环境里共存？测试工具和运行工具可以分开安装把所有工具塞进运行依赖

容器边界：它固定一部分系统环境，不固定一切

container

容器镜像把应用文件、运行时、系统包和默认命令放进一个分层文件系统。它能减少“服务器系统不同”“缺系统库”的问题，但仍依赖宿主内核、CPU 架构、网络、数据卷、环境变量和外部服务。

代码
FROM python:3.12-slim
WORKDIR /app
COPY dist/*.whl /tmp/
RUN pip install /tmp/*.whl
CMD ["hello-tool"]

故障面板：把失败映射到可检查对象

failure

分发失败时，先判断失败发生在声明、解析、构建、安装还是运行阶段。不要一看到报错就重写业务代码。下面三个按钮模拟最常见的错误信号。

发布前检查：从“能跑”走到“能交付”

checklist

做完一个包以后，至少在干净环境里跑一次完整链路。这里的“干净”指没有你开发机里残留的全局包、环境变量、临时路径和源码目录假设。

容易误判的地方

源码目录能运行，只说明开发环境当前可用；干净环境能安装并运行制品，才说明交付路径可用。README 也要写给目标用户：用什么版本、怎么创建环境、怎么安装、怎么运行、哪些配置需要自己提供。

README 是交付链路的一部分。一个用户拿到项目后，最先需要知道的是：该用哪个运行时版本、如何创建隔离环境、安装哪个制品、执行哪个入口命令、需要自己提供哪些配置。少写任何一项，用户都会把运行条件反向猜一遍。

README 项用户拿它做什么缺失后的典型问题
运行时版本选择 Python、Node 或系统平台版本旧解释器安装成功，运行时语法或 API 失败
安装命令知道装源码、wheel，还是从包索引安装用户只复制源码，依赖和入口都没有注册
入口命令装完后能直接运行最小示例包已经安装，但不知道该敲哪个命令
配置与 secret区分示例配置和真实凭据把 token 写进仓库、镜像或聊天记录
验证命令判断安装是否真的成功出错时没有共同的复现起点

把这张表和前面的干净环境验证连起来：README 负责告诉别人怎么重建条件，发布前自测负责证明这些说明确实能跑通。练习区的 README 题也按这个标准自查，不要只写“安装依赖并运行”。如果用户把完整错误、运行命令和版本号发回来，维护者就能沿着依赖、环境、制品、入口和配置逐层复现；如果这些信息缺失，排查会重新退回口头猜测。维护者也应在发布说明里写清本次制品对应的版本或提交，并保留最小复现命令，方便用户报告问题时对齐同一份代码。

三、分发错误诊断

检查点 失败先定位阶段，再改声明、环境、制品或配置。 -->
9diagnostic

Q: 9. 诊断：别人运行时报 ModuleNotFoundError: No module named 'requests'。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

。错因可能是运行依赖没写进元数据，或者依赖装到了另一个解释器环境。后果是源码逻辑没问题，但目标环境找不到库而无法启动。修法是补 dependencies ，在独立环境里重新安装项目，并用 python -m pip show requests 和最小运行命令验证。

10diagnostic

Q: 10. 诊断：源码目录里有模板文件，安装 wheel 后运行时报 File not found。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

。错因通常是模板或数据文件在源码目录里存在，但没有被包数据配置纳入 wheel。后果是开发目录能跑，安装包缺文件。修法是检查 wheel 文件清单，修正包数据配置，重新构建并在干净环境安装验证。

12diagnostic

Q: 12. 诊断：Dockerfile 里写了 ENV API_TOKEN=...。按“错因 / 后果 / 修法”回答。

参考答案 / 评分要点

。错因是把运行时 secret 写进镜像层。后果是任何拿到镜像或镜像历史的人都可能读到凭据，而且撤回困难。修法是立刻轮换泄露 token，改为运行时环境变量、密钥管理或部署平台注入，并检查镜像历史和构建日志。
