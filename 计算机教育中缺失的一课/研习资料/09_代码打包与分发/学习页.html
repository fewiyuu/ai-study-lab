代码打包 · 分发

交付代码不是发一个文件夹，而是交付一套可复现运行条件

代码离开你的电脑后，会遇到语言版本、第三方库、系统依赖、环境变量、操作系统和 CPU 架构差异。打包与分发的目标，是把“我的机器能跑”变成“目标环境按说明也能跑”。

学完你应该能做什么

  1. 解释依赖、传递依赖、运行时和制品。
  2. 区分源码包、库、命令行工具、服务镜像。
  3. 说明 lockfile、语义化版本和环境隔离的价值。
  4. 诊断 ModuleNotFound、版本冲突、平台差异和容器误用。

概念地图

分发的主线是：代码依赖运行时和库；依赖需要被声明、解析、下载和锁定；打包工具把代码和元数据变成可安装对象；目标环境按这些元数据重建运行条件；容器进一步把操作系统层面的假设也固化一部分。

“能运行”并不是源码一个条件决定的。目标机器必须拿到正确代码、正确版本的依赖、正确运行时、必要配置、启动命令和外部服务连接方式。打包与分发就是把这些隐含条件写进项目元数据、构建流程、README 或镜像中。

源码+依赖声明→解析锁定→构建制品→安装运行
依赖 dependency程序运行或构建所需的外部库、工具、系统组件。依赖没有声明清楚，别人就只能猜。
制品 artifact构建输出的可分发对象，如 wheel、npm 包、二进制、Docker 镜像。制品应能追溯版本和来源。
lockfile锁定精确依赖版本的文件，提升可复现性。它记录解析结果，不只是依赖愿望。
容器 image包含文件系统层和运行配置的镜像，运行后成为容器进程。它打包了一部分系统环境，但不打包一切。

1. 分发问题：代码之外还有隐含环境

一个 Python 文件里写 import requests，并不意味着别人的 Python 能找到 requests。这个库可能需要从包索引下载，requests 自己还依赖 urllib3、certifi 等库。依赖会递归展开，这就是传递依赖。

环境还包括语言版本、系统库、命令行工具、配置文件和网络权限。简单脚本也许只要 README 说明；可复用库需要包元数据；命令行工具需要入口点；Web 服务通常还要进程管理、端口、环境变量、数据库和部署配置。

排查分发问题时要把“源码错误”和“环境错误”分开。源码错误通常在任何机器上都会稳定出现；环境错误常表现为某台机器缺包、版本不同、路径不同、CPU 架构不同、环境变量没设置。用户说“我这里跑不了”时，第一步不是重写代码，而是收集运行命令、语言版本、安装方式和完整错误。

错误示例只复制源码但没有依赖说明
# app.py
import requests
print(requests.get("https://example.com").status_code)

# 在新机器运行：
# ModuleNotFoundError: No module named 'requests'

判断句：代码离开本机后，所有没有写下来的运行条件都会变成潜在故障点。你现在应该能做什么：看到“别人运行不了”时，先检查依赖和运行环境，而不是只检查源码。

2. 依赖声明：让工具知道该安装什么

依赖应该写进项目元数据，而不是只写在聊天记录或脑子里。Python 可用 pyproject.toml、requirements.txt 或特定工具配置；JavaScript 可用 package.json；Rust 有 Cargo.toml。不同生态语法不同，但目的相同：告诉工具项目需要哪些包。

依赖要区分运行依赖和开发依赖。用户运行库可能只需要 requests；开发者还需要 pytest、ruff、mypy。把开发工具塞进运行依赖会让安装变重；漏掉运行依赖则会导致目标环境崩掉。

依赖声明还要写清兼容范围。只写“需要 requests”太模糊，工具不知道应该安装哪个版本；写得过死又可能让库难以和别的项目组合。应用项目更关心完整可复现，库项目更关心兼容范围，这是两种不同目标。

正确示例pyproject 中声明依赖
[project]
name = "link-fetcher"
version = "0.1.0"
dependencies = ["requests>=2.31"]

[project.optional-dependencies]
dev = ["pytest", "ruff"]

判断句：运行时必须存在的库放进运行依赖，只服务开发流程的工具放进开发依赖或可选依赖。你现在应该能做什么：判断一个依赖应该写在运行依赖还是开发依赖里。

3. 环境隔离：不要把全局环境当项目环境

全局安装会让项目互相污染。项目 A 需要 requests 2.x，项目 B 需要不同版本；你在全局环境升级一次，另一个项目可能坏掉。虚拟环境、conda 环境、uv、npm node_modules 等都在解决隔离问题。

环境隔离不是为了复杂，而是为了可重复。新同事拿到项目后，应该能按固定命令创建环境、安装依赖、运行测试。否则项目就依赖“某个人电脑上的历史状态”。

隔离环境还要和编辑器、测试命令、构建命令对齐。终端里装到了一个 Python，编辑器选择另一个 Python，构建工具又用第三个 Python，这种错位会制造“安装了但找不到包”的假象。排查时要看解释器路径、包安装位置和运行命令是否一致。

逐行拆解Python 项目环境最小流程
uv venv
uv pip install -e ".[dev]"
pytest
逐行拆解检查解释器和安装位置
which python
python -m pip show requests
python -c "import requests; print(requests.__version__)"

判断句：依赖安装成功不等于当前运行命令能看见它，关键是解释器和环境是否同一个。你现在应该能做什么：解释为什么项目应该有独立环境，并能把安装步骤写进 README。

4. 包与制品：源码、库、工具、服务交付物不同

打包的输出取决于你要交付什么。库希望别人 import；命令行工具希望别人安装后能输入一个命令；Web 服务希望在服务器上启动进程；桌面应用可能需要安装器。不同目标决定元数据、入口点和构建方式。

制品应该是可追溯的：来自哪个 commit、哪个版本号、用什么构建流程生成。没有版本号和构建记录，线上出问题时很难知道用户运行的是哪一份代码。

不要混淆“源码仓库”和“安装后的包”。源码仓库里可能有测试、文档、构建脚本和开发配置；安装包里应该包含运行所需模块、数据文件和入口点。模板、静态资源、配置样例如果没有进入包，用户安装后就可能出现 File not found。

正确示例命令行工具入口点
[project.scripts]
fetch-links = "link_fetcher.cli:main"

判断句：先确定交付目标是库、CLI、服务还是镜像，再决定元数据、入口点和构建产物。你现在应该能做什么：根据交付目标判断需要库包、CLI 入口、二进制还是容器镜像。

5. 版本约束与 lockfile：灵活和可复现之间取平衡

版本约束表达兼容范围，例如 requests>=2.31 表示至少需要 2.31。范围太宽可能装到未来不兼容版本；钉死所有版本又可能错过安全修复。库通常声明兼容范围，应用更常使用 lockfile 锁定精确版本。

语义化版本常用 MAJOR.MINOR.PATCH：主版本可能破坏兼容，小版本增加功能，补丁修复问题。但现实中并非所有项目都严格遵守，所以关键生产依赖仍要测试和锁定。

lockfile 记录的是依赖解析后的结果，包括直接依赖和传递依赖。它能减少“昨天能装，今天装出另一套版本”的问题。应用、服务和可部署项目通常应该提交 lockfile；库是否提交 lockfile 要看生态和发布策略，但至少测试环境要可复现。

正确示例依赖范围与锁定思路
dependencies = ["requests>=2.31,<3"]
# lockfile 记录实际解析出的 requests==2.32.3 等精确版本

判断句：版本范围表达兼容承诺，lockfile 记录一次确定的安装结果。你现在应该能做什么：说明“版本范围”和“精确锁定”分别解决什么问题。

6. 容器：把系统层假设也打包一部分

容器镜像把应用文件、系统包、运行时和默认命令放进一个分层文件系统。它解决了很多“服务器没有这个库”“系统版本不同”的问题，但不是魔法。容器仍然依赖宿主内核、CPU 架构、网络、卷挂载、环境变量和外部服务。

容器适合服务部署和一致运行环境，但不应把秘密写进镜像，也不应把构建过程写成不可理解的黑箱。镜像越大，拉取越慢，攻击面也可能越大。好的 Dockerfile 应该清晰、可缓存、只包含运行所需内容。

镜像和容器也要区分：镜像是静态模板，容器是运行中的进程。把 token 写进镜像意味着每个拿到镜像的人都可能读到它；把数据库地址、API key、端口这类运行配置放到环境变量或部署平台中，才更容易按环境切换和管控权限。

正确示例最小 Dockerfile 思路
FROM python:3.12-slim
WORKDIR /app
COPY pyproject.toml .
RUN pip install .
COPY . .
CMD ["python", "-m", "myapp"]

判断句：容器能固定一部分系统环境，但 secret、外部服务、数据卷、网络和架构仍要单独设计。你现在应该能做什么：解释容器解决了哪些环境问题，以及哪些问题仍需外部配置。

7. 常见错误与修法

ModuleNotFoundError

错因可能是依赖未安装、环境选错、依赖未声明。后果是源码没错但目标环境无法启动。修法：确认解释器，安装项目依赖，补元数据。

版本冲突

两个依赖要求不兼容版本。后果是解析失败或运行时行为异常。修法：查看依赖树，调整范围、升级相关包或拆分环境。

只在本机能跑

隐含全局安装、私人路径、环境变量。后果是项目依赖个人电脑历史状态。修法：写 README、锁依赖、提供环境创建命令。

把 secret 打进镜像

后果是镜像分发后凭据泄露且难以撤回。修法：用运行时环境变量、密钥管理或部署平台注入。

数据文件漏打包

源码仓库有模板或静态文件，安装包里没有。后果是安装后 File not found。修法：检查包数据配置和构建产物内容。

入口命令没声明

用户安装后不知道运行什么，或只能从源码目录启动。修法：声明 CLI entry point，README 写清安装和启动命令。

8. 完整知识链路

打包与分发的核心是把运行条件显式化。源码说明程序逻辑，依赖声明说明需要哪些库，lockfile 提升可复现性，虚拟环境隔离项目，包元数据定义安装和入口点，制品承载可分发结果，容器把一部分系统环境纳入镜像。每当程序离开你的电脑，都要问：目标环境如何得到同样的代码、依赖、配置、运行时和启动命令？

一条健康的分发链路通常是：先明确交付目标，是库、CLI、服务还是镜像；把运行依赖写进项目元数据，把开发依赖放进可选组；用独立环境安装并运行测试；为应用锁定依赖解析结果；构建可追溯制品并记录版本；在干净环境里安装验证；如果使用容器，确认镜像不含 secret，运行时配置从外部注入。这样别人遇到问题时，才能沿着依赖、环境、制品、配置逐层排查。

9. 真实场景与排查手册

代码能在本地运行，不等于能被别人安装。分发需要说明项目名、版本、依赖、入口、构建方式和目标环境。缺一项，用户就可能在安装或运行时踩坑。

包管理文件不是装饰。pyproject、package.json 或类似配置描述的是项目边界：哪些文件属于包、依赖是什么、命令入口在哪里、如何构建和发布。

真实问题常发生在“本地路径假设”。代码依赖当前目录、未声明的数据文件、全局安装的工具或未锁定的依赖版本，都会让分发后的环境和开发机不一致。

依赖未声明

本机有包不代表用户环境有包。所有运行时依赖都应写进项目配置。

入口不清

用户安装后应该运行什么命令？CLI 入口和模块入口需要明确声明。

文件漏打包

配置、模板、静态资源若没进入包，安装后会 File not found。

发布前检查

干净环境安装 -> 运行最小命令 -> 检查版本和入口 -> 验证资源文件。
