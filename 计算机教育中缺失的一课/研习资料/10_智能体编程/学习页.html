智能体编程 · 代码代理

和代码代理协作：把自然语言变成可审查、可验证的工程改动

智能体编程不是“让模型替你负责”，而是让模型读文件、改代码、运行命令、汇报结果。你需要会给上下文、拆任务、设边界、审查 diff、运行验证，并识别模型的幻觉和越界行为。

学完你应该能做什么

  1. 区分聊天补全、IDE 助手和代码代理。
  2. 写出包含目标、上下文、约束、验证标准的任务描述。
  3. 解释计划-执行-检查循环和工具调用风险。
  4. 诊断 AI 代码中的不存在 API、误改文件、缺测试和安全问题。

概念地图

代码代理的能力来自三部分：语言模型负责理解和生成，工具接口负责读写文件和运行命令，上下文窗口负责承载项目事实。任何一部分不可靠，结果都可能偏。高质量协作不是多说“帮我优化”，而是把任务变成可验证的小工程问题。

目标+上下文+约束→代理改动→diff 与测试验证
代码代理能访问项目文件和命令行工具的 AI 编程助手。
上下文模型当前能看到的信息，包括文件、说明、错误日志和用户目标。
工具调用代理通过 shell、编辑器、搜索、测试等工具与项目交互。
人工把关开发者对需求、风险、测试覆盖和最终合并负责。

1. 代码代理和普通聊天不一样

普通聊天模型主要给建议和片段；IDE 助手常做补全和局部改写；代码代理能在项目目录里搜索、读文件、编辑文件、运行测试，甚至连续执行多步任务。能力越强，越需要边界。

代码代理适合处理局部明确任务：补 CLI 参数解析、加测试、重构一个函数、解释错误、迁移小范围 API。它不适合在没有需求和验证标准时“整体优化项目”。模糊目标会产生看似勤奋但不可控的改动。

正确示例清晰任务描述
目标：给 dl_page.py 增加 --output 参数。
约束：保持现有默认行为不变，不改无关文件。
验证：新增 2 个 pytest；运行 pytest tests/test_cli.py。

你现在应该能做什么：把“帮我完善一下”改写成有目标、约束和验证的任务。

2. 上下文：模型不知道没看到的项目事实

模型会根据当前上下文推断项目结构。如果它没读 README、测试、配置和相关文件，就可能调用不存在的函数、误解架构边界、忽略已有工具。好协作要让代理先读关键文件，再动手。

上下文不是越多越好。无关文件会稀释注意力，过旧日志会误导判断。你要提供“足够但相关”的材料：报错全文、复现命令、目标文件、相邻模块、已有测试、不能破坏的行为。

逐行拆解高质量上下文包
相关文件：src/cli.py, src/downloader.py, tests/test_cli.py
复现命令：pytest tests/test_cli.py -k output
错误现象：--output 被解析但文件未写入
不能改变：未传 --output 时仍输出到 stdout

你现在应该能做什么：在交给代理前，列出它必须知道的项目事实。

3. 计划-执行-检查：让代理每一步都有可见证据

复杂任务应先让代理给计划，再执行，再检查。计划不是仪式，而是提前暴露误解：它准备改哪些文件、为什么、如何验证。执行后要看 diff 和测试输出，而不是只看“已完成”。

当任务变大时，分块推进比一次性大改更稳。每块都应该有小验证：单元测试、类型检查、手工命令或截图。没有验证的自动修改只是在制造未知风险。

正确示例小步协作循环
1. 搜索相关实现和测试
2. 修改最小代码路径
3. 补或改测试
4. 运行目标测试
5. 汇报 diff 和剩余风险

你现在应该能做什么：要求代理先说明改动范围和验证方式，再开始写代码。

4. 工具调用：会做事，也会做错事

工具让代理从“建议者”变成“执行者”。它可以运行 rg 搜索，运行测试，修改文件，启动服务。风险也随之增加：命令可能耗时，文件可能误删，测试可能只跑了无关部分，网络命令可能引入不稳定。

好的工具使用有三个特征：先读再改，改小范围，改后验证。涉及删除、迁移、权限、密钥、生产数据时要更谨慎。代理如果要执行破坏性操作，应先解释目标和影响。

错误示例未定位就大范围替换
# 没读代码就全项目替换函数名
replace "load_config" -> "read_config" in all files
# 可能破坏无关模块、文档或第三方代码

你现在应该能做什么：识别哪些代理操作需要先停下来确认影响。

5. 审查和验证：AI 写得像，不等于项目真的对

AI 代码常见问题包括：调用不存在 API、漏掉边界条件、测试只覆盖 happy path、吞掉错误、改了无关文件、引入安全漏洞。审查时先看行为变化，再看实现细节：需求是否满足？旧行为是否保持？错误路径是否处理？

验证分层：静态检查发现语法和风格问题，类型检查发现部分接口错误，单元测试验证局部行为，集成测试验证模块协作，手工检查验证真实用户路径。没有一个层次能覆盖全部。

正确示例AI 改动后的检查命令
git diff
ruff check .
pytest
python -m app --help

你现在应该能做什么：读代理的最终汇报时，要求它给出证据，而不是只接受结论。

6. 安全边界：密钥、权限、数据和生产环境

不要把密钥、token、私有数据直接贴给模型或写进仓库。即使工具在本地运行，也要默认敏感信息需要最小暴露。涉及生产数据库、用户数据、付费 API、删除文件、权限变更时，必须有清晰回滚方案。

智能体编程会放大好流程，也会放大坏流程。没有测试的项目会让代理更难判断对错；没有代码规范的项目会让改动风格漂移；没有权限边界的环境会让一次错误命令影响更大。

错误示例把密钥写进提示或代码
API_TOKEN="sk-live-..."
# 不要把真实密钥交给模型或提交到仓库

你现在应该能做什么：给代理任务时主动声明安全边界和禁止操作。

7. 常见错误与修法

目标太模糊

代理大改一通但不满足真实需求。修法：写目标、范围、验证命令和不变行为。

上下文缺失

模型编出不存在 API。修法：让它先读相关文件和测试。

只看结论不看 diff

无关改动混入。修法：逐文件审查 diff，确认行为变化。

没有测试就合并

回归风险高。修法：补最小测试或手工验证步骤。

8. 完整知识链路

智能体编程的主线是：开发者给出目标、上下文、约束和验证标准；代码代理读取项目、制定计划、调用工具、修改文件；开发者审查 diff、运行测试、检查安全边界；最终只有在证据证明行为正确时才合并。你不是把责任外包给 AI，而是把 AI 纳入工程流程。

9. 把一次代理任务拆成可控工程任务

deep guide

智能体编程最容易翻车的地方，是把“自然语言能说清”误认为“工程任务已经定义清”。人可以在对话中补全含糊的部分，代码代理却会把缺失信息变成猜测：不知道入口文件，就可能从头造一个入口；不知道旧行为必须保留，就可能把默认输出改掉；不知道验证命令，就可能只改代码不证明结果。要把代理当成会执行的初级同事，而不是会读心的机器。

一个可控任务至少有四块信息。第一是目标，说明最后用户能观察到什么变化。第二是上下文，列出相关文件、错误日志、复现命令、已有测试。第三是约束，声明不能改什么、不能引入什么、不能泄露什么。第四是验证，给出能证明改动正确的命令或手工路径。四块里少一块，代理仍然可能工作，但风险会变成隐藏风险。

任务越大，越要拆小。比如“重构下载器”太大，可以拆成“先抽出路径解析函数并补测试”“再让 CLI 调用新函数”“最后保留旧参数兼容”。每一步都有单独 diff 和单独验证，就算代理某一步走偏，也容易撤回和修正。大而连续的自动改动看起来省时间，实际上会把需求误解、风格漂移和测试缺口混在一起。

逐行拆解把模糊需求改成代理任务
模糊写法：
  帮我把下载功能完善一下。

可执行写法：
  目标：给 src/cli.py 的 download 命令增加 --output 参数。
  上下文：相关文件是 src/cli.py、src/downloader.py、tests/test_cli.py。
  约束：不改变未传 --output 时输出到 stdout 的行为；不引入网络请求测试。
  验证：新增 2 个 CLI 测试，运行 pytest tests/test_cli.py -k output。

你现在应该能做什么：看到一个想交给代理的任务时，先把它写成“目标、上下文、约束、验证”四行，再决定是否开始执行。

10. 先读后改：代理为什么不能直接动手

context

代码代理的输出质量受上下文限制。它没有读过的文件，对它来说就不是项目事实，而只是可以猜的空白。很多“AI 编造 API”的根源不是模型太坏，而是它没被要求先确认项目里到底有什么。稳妥流程是先搜索符号，再读实现和测试，再提出改动点，最后才修改。

读文件不是把整个项目塞进上下文。真正有用的是按问题定位：命令出错先读 CLI 入口和测试；数据格式错先读 parser 和 fixture；性能问题先读热点路径和基准；权限问题先读配置和错误处理。相关文件越明确，代理越容易沿着真实项目结构工作。

当代理给出计划时，要看它是否引用了真实文件和真实函数。如果计划里出现“可能有一个 xxx.py”“应该有一个 helper”，这就是黄色信号。它还没确认事实。让它先用搜索或阅读确认，而不是基于想象继续写。

正确示例修改前的定位命令
rg -n "download|output|stdout" src tests
pytest tests/test_cli.py -k download -q
git diff -- src/cli.py src/downloader.py tests/test_cli.py
错误示例没有读项目就新建平行实现
# 项目里已有 src/downloader.py，代理却新建 utils/download_file.py
# 后果：新代码没有被 CLI 调用，测试也没覆盖真实路径。
def download_file(url, path):
    ...

你现在应该能做什么：判断代理是否在真实项目结构里工作，而不是在脑内项目里工作。

11. 验证不是一句“测试通过”

verification

代理完成后的汇报必须能回答三个问题：改了什么，怎么证明，剩下什么风险。只说“已完成”没有价值；只说“测试通过”也不够，因为你还要知道跑了哪些测试、为什么这些测试覆盖目标、有没有没跑的检查。可信汇报应该包含文件清单、关键行为变化、验证命令、失败或跳过项、人工仍需检查的地方。

验证要贴近任务。改 CLI 参数，就跑 CLI 测试和 --help；改 parser，就跑正反例；改前端布局，就看截图和小屏；改性能，就给基准或至少对比数据。验证不是越多越好，而是要能覆盖这次改动的风险。

还有一类特别危险的假验证：测试本来就没有覆盖真实路径，代理新增的测试只验证自己新写的 helper，主程序没有调用 helper。审查时要顺着调用链看：用户入口是否真的走到新逻辑，错误路径是否能触发，旧行为是否还在。

正确示例完成汇报应包含证据
改动文件：
  src/cli.py
  tests/test_cli.py

验证：
  pytest tests/test_cli.py -k output -q  # 4 passed
  python -m app download --help          # 确认出现 --output

剩余风险：
  没跑真实网络下载；测试使用 monkeypatch 避免外部网络。

你现在应该能做什么：要求代理用证据描述完成状态，并能识别“跑了测试但没有覆盖目标”的情况。

12. 安全和权限：越能自动执行，越要设边界

safety

代码代理能读写文件、运行命令、启动服务、安装依赖。它越像开发者，越需要开发者的安全习惯。不要把真实密钥写进提示、代码、测试或日志；不要让代理随意删除目录；不要在没有备份和确认的情况下改数据库；不要把生产数据当作调试样本。安全边界要提前写进任务，而不是出事后再补。

权限边界也包括“只改哪里”。如果任务是修 CLI，就不应该顺手改 README、格式化全项目或升级依赖。无关改动会增加审查成本，也可能引入回归。代理常常会出于“帮忙”的倾向扩大范围，所以任务描述里要明确“只改这些文件，发现额外问题先报告”。

遇到破坏性操作，要让代理先停。删除、批量移动、重命名、迁移数据、修改权限、安装全局工具、调用付费 API，都应该有解释、影响范围和回滚方案。能自动化不代表应该自动做。

错误示例把敏感信息放进任务
请用这个真实 token 调试：
API_TOKEN=sk-live-xxxxxxxx

# 错误点：真实密钥可能进入对话、日志、shell history 或仓库。
正确示例使用占位符和边界
使用环境变量 API_TOKEN，不要打印它，不要写入文件。
测试中使用 fake-token，并 monkeypatch 网络请求。
如需删除或移动文件，先列出计划并等待确认。

你现在应该能做什么：在交给代理前声明敏感信息、破坏性操作和改动范围的边界。

9. 真实场景与排查手册

和代码代理协作时，最重要的不是把需求写得长，而是把目标、上下文、约束和验证标准写得可执行。代理需要知道要解决什么、哪些文件相关、哪些行为不能改、怎样证明改对了。

代理的工具调用会真实改变项目，所以每一步都要留下证据。读了哪些文件、改了哪些文件、跑了哪些命令、哪些测试没跑，这些比“已经完成”更重要。

真实风险包括：编造不存在的 API、误解项目架构、把无关改动混进 diff、用不充分测试证明完成、在提示或代码里暴露密钥。好的协作流程会把这些风险提前暴露。

需求太松

“帮我优化项目”没有边界。改成目标、范围、不变行为和验证命令。

上下文太少

代理没读相关文件就改代码，容易编 API。先让它搜索、读实现和测试。

只信完成结论

必须看 diff、测试输出和剩余风险。没有证据的完成只是口头状态。

稳妥协作

小任务、小 diff、明确验证、人工审查。代理负责执行，开发者负责判断。
