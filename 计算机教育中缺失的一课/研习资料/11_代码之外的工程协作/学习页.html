9. 注释、命名和结构：先让代码自己说话

deep

很多人以为“代码之外”就是多写注释。实际顺序相反：能用清晰命名表达的，不要靠注释补；能用函数拆分表达的，不要靠长段解释补；只有代码本身表达不了的背景、约束、权衡和外部原因，才值得写成注释。注释不是遮羞布，它是给读者补充代码看不出来的信息。

还要警惕过期注释。代码改了，注释没改，会比没有注释更危险。维护注释的原则是：注释必须靠近它解释的代码；解释稳定原因，不解释临时心情；如果注释越来越长，先考虑代码结构是否需要改。

代码
# 遍历用户列表
for user in users:
 send_email(user)

正确示例注释说明业务约束复制

代码
# 旧系统每天 09:00 会重置退订状态；这里必须在同步后再发邮件。
for user in active_users_after_sync:
 send_email(user)

你现在应该能做什么：看到一条注释时，判断它是否提供了代码本身看不出的原因、约束或修改提醒。

10. 文档分层：不同文档解决不同问题 {#}

docs

文档最常见的问题不是少，而是混。README 里塞满架构争论，新人找不到怎么运行；设计文档写成流水账，看不出最后为什么选这个方案；运行手册只讲原理，不告诉故障时先执行哪条命令。文档要先明确读者和使用时刻。

文档可以不华丽，但必须可行动。一段“本项目使用先进架构”没有用；一条 python -m pytest 或 python -m app --help 能让读者立即验证项目状态。写文档时要不断问：读者看完这一段，下一步能做什么？

你现在应该能做什么：遇到一个信息缺口时，判断它应该进 README、设计文档、运行手册还是决策记录。

11. 提问构造共同现场 {#}

asking

的价值，是让别人不用反复追问就能进入同一个事实现场。背景告诉对方你在做什么；复现步骤让对方能看见同样的问题；期望和实际说明差距；已尝试内容减少重复建议；具体问题告诉对方你需要什么类型的帮助。

提问前整理问题，本身就是调试。很多 bug 在你写清“我期望什么、实际什么、最小复现是什么”时就会暴露。即使最后仍要问别人，对方也能更快给出有用回答。

我这个接口又坏了，有人知道为什么吗？

正确示例可复现的问题描述复制

代码
目标：POST /api/export 应返回 csv 文件。
环境：Windows 11, Python 3.12, commit 8f3a21c。
复现：python -m pytest tests/test_export.py -k csv。
期望：状态码 200，Content-Type 为 text/csv。
实际：状态码 500，日志显示 FileNotFoundError: template.csv。
已尝试：确认 tests/fixtures 里没有 template.csv。
问题：fixture 应该补文件，还是代码应在缺模板时返回明确错误？

你现在应该能做什么：把“帮我看看”改成包含事实、复现和具体问题的求助。

12. 评审和复盘：把个人判断变成团队反馈 {#}

review

代码评审的目标在合并前发现需求、边界、测试、维护和安全风险。好的评审意见要具体、可行动、说明后果。比如“这里不好”没有用；“这里吞掉异常会让 CLI 返回成功但不生成文件，建议抛出自定义错误并补测试”就能行动。

事故复盘也是协作的一部分。复盘不是追责，而是重建时间线、找触发条件、找检测缺口、找防护缺口、写后续行动项。一个有效行动项应该有负责人、截止时间和可检查产物，否则只是“以后注意”。

代码
问题：缺配置文件时返回空字典。
后果：调用方会继续使用生产默认配置，可能把测试请求发到真实服务。
建议：改为显式抛 ConfigError，并补一个缺配置文件的测试。

你现在应该能做什么：写出包含问题、后果、建议和优先级的评审意见。
