代码质量 · 自动化检查

把质量检查自动化：让机器处理重复规则，让人关注设计和行为

代码质量工具的目标不是让项目显得专业，而是把反馈前移：保存时格式化，提交前检查，合并前测试，主干持续集成。越早发现问题，修复成本越低。

学完你应该能做什么

  1. 区分 formatter、linter、type checker、test、CI。
  2. 解释 pre-commit 和 CI 在质量链路中的位置。
  3. 设计一个最小项目质量门禁。
  4. 识别正则表达式的适用场景和常见坑。

概念地图

质量体系是一条反馈链：formatter 处理风格，linter 找可疑写法，type checker 检查接口约束，测试验证行为，pre-commit 在提交前拦截低级问题，CI 在共享环境里重复这些检查，防止“只在我电脑上通过”。

格式→静态检查→类型→测试→钩子→CI
formatter自动整理代码表面风格。
linter发现可疑模式、坏味道和潜在错误。
test用可执行例子验证代码行为。
CI在远程环境自动运行检查和测试。

1. 质量不是整齐，而是可理解、可修改、可验证

整齐的代码不一定正确，但混乱的代码更难维护。代码质量至少包含：行为正确、边界清晰、命名可读、错误处理明确、测试覆盖关键路径、修改后能快速发现回归。工具只能覆盖其中一部分。

质量工具的核心价值是减少人工重复劳动。机器擅长检查格式、未使用变量、类型不匹配、测试回归；人更适合判断需求、架构、产品权衡和复杂风险。把该交给机器的规则自动化，评审时就能少吵风格，多看行为。

你现在应该能做什么：解释为什么“格式通过”只是质量的一小部分。

2. 格式化：结束风格争论

formatter 自动决定缩进、换行、引号、空格和括号布局。团队使用统一 formatter 后，代码评审不必反复讨论风格。配置越少，争论越少；但不同项目也可能因为历史原因保留特定风格。

格式化会改代码外观，通常不应改变行为。如果 formatter 导致行为变化，说明语言或工具边界需要格外小心。提交格式化改动时，最好避免和业务逻辑混在一起，否则 diff 很难审。

正确示例格式化命令
ruff format .
prettier --write .
gofmt -w .

你现在应该能做什么：把格式化改动和逻辑改动分开提交。

3. 静态检查和类型检查：不运行程序也能发现一部分问题

linter 会扫描源码，发现未使用变量、不可达代码、危险默认值、复杂条件等问题。type checker 会根据类型标注和推断检查函数调用是否匹配。它们都不运行业务逻辑，因此不能证明程序正确，但能快速发现很多低级错误。

静态检查的误报和漏报都存在。好的团队会把规则调到“能长期坚持”的强度。规则太弱没有价值，太吵会让人养成忽略警告的习惯。新增规则时最好逐步启用，避免一次性制造大量噪音。

逐行拆解静态检查示例
ruff check .
mypy src/
eslint .
tsc --noEmit

你现在应该能做什么：说明 lint 和 type check 各能发现什么、不能发现什么。

4. 测试：用可执行例子锁住行为

测试不是为了追求数字，而是为了描述和保护关键行为。单元测试验证小函数或模块，集成测试验证多个模块协作，端到端测试模拟用户路径。每一层成本和覆盖范围不同。

好测试通常具备：输入明确、期望明确、失败信息有帮助、运行足够快、边界条件覆盖关键风险。只测试 happy path 会让错误路径无人看守；过度依赖脆弱细节又会让重构寸步难行。

正确示例测试应覆盖正常和错误路径
def test_parse_user_ok():
    assert parse_user({"id": "1", "name": "Ada"}) == {"id": 1, "name": "Ada"}

def test_parse_user_rejects_missing_id():
    with pytest.raises(KeyError):
        parse_user({"name": "Ada"})

你现在应该能做什么：为一个函数设计至少一个正常路径测试和一个错误路径测试。

5. pre-commit 与 CI：把检查放到正确时机

pre-commit hook 在提交前运行，适合做快速、本地、确定的检查：格式化、lint、去除尾随空格、阻止提交密钥。它的目标是拦住低级问题，但不应慢到让开发者绕过它。

CI 在远程环境中运行，适合做完整测试矩阵、构建、类型检查、跨平台验证。CI 的价值在于共享可信环境：即使本地漏跑，合并前仍有统一检查。失败时不要只重跑祈祷，要读日志定位。

正确示例最小质量命令组合
ruff format --check .
ruff check .
pytest

你现在应该能做什么：判断一个检查适合放在保存时、提交前，还是 CI 中。

6. 正则表达式：强大的文本模式工具，也容易过度使用

正则表达式用模式匹配文本，适合搜索日志、校验简单格式、提取结构相对稳定的片段。它不适合解析复杂嵌套语法，例如完整 HTML、通用编程语言或不稳定的人类自然语言。

常见元字符包括 . 匹配任意字符，* 表示重复零次或多次，+ 表示一次或多次，? 表示可选，^ 和 $ 匹配行首行尾，方括号表示字符集合。写正则时要准备测试样例，包含应该匹配和不该匹配的情况。

逐行拆解正则示例
^ERROR .+ timeout$
# ^       行首
# ERROR   固定文本
# .+      至少一个任意字符
# timeout 固定文本
# $       行尾
错误示例用过宽模式误匹配
.*error.*
# 可能匹配 unexpected-error-code、terror、大小写还可能漏掉 ERROR

你现在应该能做什么：写正则时同时列出正例和反例，而不是只看一个样本。

7. 常见错误与修法

把格式当质量全部

代码整齐但逻辑可能错。修法：继续跑检查、类型和测试。

lint 规则太吵

团队会忽略警告。修法：逐步启用，保留高价值规则。

测试只覆盖 happy path

错误路径回归无人发现。修法：补边界和异常测试。

CI 失败只重跑

掩盖真实问题。修法：读日志，定位命令、环境和失败断言。

8. 完整知识链路

代码质量体系把反馈从“上线后出事”提前到“保存、提交、合并之前”。formatter 统一风格，linter 和 type checker 发现静态问题，测试锁住行为，pre-commit 拦截本地低级错误，CI 在共享环境重复关键检查，正则在合适场景下帮助文本匹配。质量不是单个工具，而是一条能持续运行的自动化反馈链。

9. 质量链路不是堆工具，而是安排反馈时机

deep guide

代码质量工具最容易被学成工具名清单：formatter、linter、type checker、test、pre-commit、CI。真正重要的是反馈时机。越早发现问题，修复成本越低；但越早运行的检查越应该快、稳定、低噪声。保存时适合格式化和局部 lint；提交前适合快速检查；合并前适合完整测试、构建和跨环境验证。

不要一上来把所有检查都塞进 pre-commit。提交前检查如果跑 10 分钟，开发者会想办法绕过它，最后工具形同虚设。也不要只依赖 CI。CI 虽然统一可信，但反馈晚，等 CI 红了再修，通常已经切换上下文。好的链路是分层：本地快反馈处理低级问题，CI 处理共享环境和完整回归。

每个工具还要说清边界。formatter 不证明逻辑正确；linter 不知道业务需求；type checker 不能覆盖所有运行时状态；测试只覆盖你写出来的场景；CI 也不能替代人工评审。工具负责稳定规则，人负责需求、风险和取舍。

时机适合检查不适合放这里
保存时格式化、局部 lint慢测试、联网检查
提交前format check、lint、密钥扫描、快速单测十分钟级全量测试
推送后 / PR完整测试、类型检查、构建、跨平台矩阵只能本地复现的交互操作
发布前端到端、回归、性能或安全关键检查无关风格争论

你现在应该能做什么：为一个小项目设计“保存时、提交前、合并前”三层检查，而不是把所有工具堆在同一个地方。

10. 测试要覆盖行为，不是讨好覆盖率数字

tests

覆盖率能告诉你哪些代码被执行过，但不能告诉你断言是否有意义。一个测试调用了函数却没有检查结果，覆盖率会增加，质量不会增加。好测试应该描述行为：给定什么输入，期望什么输出；遇到什么错误，应该如何失败；边界条件下是否保持约定。

写测试时先选关键风险。解析函数要测合法输入、缺字段、错误类型；文件写入要测路径存在、路径不存在、权限失败；CLI 要测参数解析、默认行为、错误提示；网络逻辑要避免真实外部依赖，使用 mock 或 fake。测试越贴近风险，越能在回归时发出有用信号。

测试失败也要会读。先看失败命令，再看失败测试名，再看断言差异，再看最近改动。不要一看到红色就全量重跑。重跑只能处理偶发问题，不能替代定位。若测试偶发失败，也应该记录条件，找随机性、时间、并发或外部依赖。

错误示例调用了函数但没有验证行为
def test_parse_user():
    parse_user({"id": "1", "name": "Ada"})
    # 没有 assert，函数返回错了也可能看不出来。
正确示例正常路径和错误路径都锁住
def test_parse_user_converts_id():
    assert parse_user({"id": "1", "name": "Ada"}) == {"id": 1, "name": "Ada"}

def test_parse_user_rejects_missing_id():
    with pytest.raises(KeyError):
        parse_user({"name": "Ada"})

你现在应该能做什么：判断一个测试是否真的验证了行为，并为关键错误路径补断言。

11. CI 失败排查：从日志回到可复现命令

ci

CI 失败时，第一步不是点重跑，而是找失败的是哪一个 job、哪一条命令、哪一个测试或断言。CI 日志通常很长，但结构清楚：安装依赖、运行检查、运行测试、上传产物。你要从最后的错误往前找到真实失败点，而不是被前面的普通输出吓住。

常见 CI 问题分四类。第一是代码真实回归，本地也能复现。第二是环境差异，比如本地 Windows、CI Linux，路径大小写或换行不同。第三是依赖差异，本地缓存旧版本，CI 安装新版本。第四是偶发问题，来自时间、随机数、网络或并发。不同类型修法不同。

最稳的做法是把 CI 失败命令复制到本地复现。如果不能复现，就对比 Python/Node 版本、操作系统、环境变量、依赖锁文件和测试数据。CI 是共享门禁，不是神秘黑箱；它只是另一个更干净、更严格的运行环境。

逐行拆解CI 失败阅读顺序
1. 找失败 job：lint / test / build / deploy？
2. 找失败命令：ruff check . 还是 pytest？
3. 找失败对象：哪个文件、哪个测试、哪个断言？
4. 本地复现：复制同一条命令。
5. 对比环境：系统、版本、依赖、变量、随机性。

你现在应该能做什么：看到 CI 红了时，能定位失败命令和失败类型，而不是只反复重跑。

12. 正则表达式：写模式时同时写反例

regex

正则很适合处理稳定文本模式：日志行、简单编号、邮箱粗校验、文件名筛选、搜索替换。它不适合解析完整 HTML、复杂嵌套括号、编程语言语法或规则变化很大的人类文本。能用结构化解析器时，优先用解析器；正则适合当边界清楚的小工具。

写正则时不要只拿一个正例试。至少准备三类样本：应该匹配的正例、不应该匹配的反例、边界样例。比如匹配错误日志时，要想清楚大小写、行首行尾、空格数量、时间戳、错误码是否稳定。模式过宽会误匹配，模式过窄会漏掉真实情况。

复杂正则要拆解。把每个元字符写成注释，或者用命名分组，让未来读者知道它在捕获什么。看不懂的正则是维护风险，尤其在自动替换和数据清洗里，误匹配可能批量改坏文件。

正确示例用正例和反例测试模式
pattern = r"^ERROR \[[A-Z_]+\] .+ timeout$"

should_match = [
    "ERROR [NETWORK] request timeout",
]
should_not_match = [
    "WARN [NETWORK] request timeout",
    "TERROR [NETWORK] request timeout",
    "ERROR [NETWORK] request timeout later",
]
错误示例过宽模式用于批量替换
.*id.*
# 可能匹配 user_id、identity、grid、midnight。
# 如果拿它做批量替换，很容易改到无关文本。

你现在应该能做什么：写正则前先列正例、反例和边界样例，并判断是否应该改用结构化解析器。

9. 真实场景与排查手册

代码质量不是单一指标。格式统一、静态检查、类型约束、测试、pre-commit 和 CI 各自覆盖不同风险。把它们串起来，才能让反馈在保存、提交和合并前出现。

自动化检查不能替代设计判断。formatter 不知道业务是否正确，linter 不知道需求是否完整，测试也只能覆盖写出来的场景。人的工作是决定重要路径、边界和风险优先级。

真实项目要避免两种极端：完全没有检查，导致错误后移；或者一上来堆满工具，导致开发体验崩掉。更稳的做法是从最小质量门禁开始，逐步增加能带来真实反馈的检查。

把格式当质量

格式通过只说明外观一致，不说明逻辑正确、边界覆盖或行为稳定。

CI 太慢

CI 应分层。快速检查先反馈，耗时检查按需或在合并前运行。

正则误用

正则适合文本模式，不适合解析复杂嵌套语法。看不懂的正则要拆解和测试。

最小门禁

格式化、lint、关键测试、pre-commit 和 CI 形成一条可持续运行的反馈链。
