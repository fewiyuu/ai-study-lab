数据整理 · 命令行转换

把散乱文本整理成可靠结论

真实工作里，很多数据一开始不是整齐表格，而是日志、命令输出、CSV、错误列表、访问记录或别人发来的半结构化文本。命令行数据整理的价值，是用一条可重复的处理链把它们过滤、切列、排序、统计、替换和验证。

学完你应该能做什么

  1. 把文本处理拆成输入、过滤、转换、聚合、验证五步。
  2. 读懂并改写常见的 grep | cut | sort | uniq 管道。
  3. 判断什么时候用 awk、sed、正则或结构化解析器。
  4. 诊断空输出、字段错位、排序不稳定和误替换问题。

概念地图

命令行处理文本时，最好把每一步都想成一个小变换：输入从文件或前一条命令来，过滤减少行数，切列减少字段，转换改变格式，排序和聚合得到统计结果，最后用抽样和计数验证结果是否可信。这样写出来的命令不仅能跑，还能解释、复查和改动。

输入过滤切列聚合验证
管道把前一条命令的标准输出交给后一条命令继续处理。
字段一行文本按分隔符切开后的第几列，例如空格、逗号、制表符。
聚合把多行结果合并成计数、频率、最大值、排名等摘要。
正则用模式描述文本形状，适合匹配和替换边界清楚的字符串。
awk按行读入，按字段处理，适合列计算和轻量报表。
sed按流编辑文本，适合简单替换、删除和局部改写。

1. 数据整理先问形状，再写命令

shape first

拿到一份文本时，不要立刻写很长的管道。第一步是看数据形状：每行是否代表一条记录，字段之间用什么分隔，是否有表头，是否有引号包住的逗号，是否有缺失值，是否有多行记录。形状决定工具。如果是严格 CSV，优先考虑 CSV 解析器；如果是简单空格分隔的日志，命令行工具就很合适。

第二步是明确目标：你要找行、取列、统计频率、改格式，还是生成新文件。目标不同，命令顺序不同。找行通常先 grep，取列可以用 cut 或 awk，统计频率常用 sort | uniq -c | sort -nr，批量替换常用 sed，复杂计算则更适合脚本语言。

第三步是保留验证动作。每做一次转换，都用 head、wc -l、抽样几行或对比前后数量检查。数据整理最危险的错误不是命令失败，而是命令成功地给出一个看起来很合理的错误结果。以后看到陌生文本，你应该先能说出它是一行一条记录，还是需要更强的解析方式。

还要分清“给人读的输出”和“给机器处理的数据”。例如 ls -l 的输出适合人看，但列宽、日期格式、语言环境都可能变化；脚本里更稳的是使用工具提供的机器可读格式，或者直接读取原始数据。命令输出越像报表，越不能默认它的列永远稳定。

你现在应该能先问四个问题：记录边界在哪里，字段边界在哪里，目标字段是否稳定，结果如何验算。四个问题回答不出来时，先观察样本，不急着把命令写长。

2. 管道不是越长越厉害，而是每一步职责单一

pipeline

管道符 | 把左边命令的标准输出传给右边命令的标准输入。它让小工具组合成完整流程。好的管道像清晰的句子：先筛选，再抽取，再排序，再计数。差的管道把过滤、转换、统计混在一个难读命令里，出错时不知道哪一步坏了。

写管道时可以逐步构造。先跑第一段确认输出，再加第二段；每加一步都看几行样本。这样比一次写完更慢一点，但能大幅减少误判。命令行高手不是从不出错，而是能把长命令拆回每一步，快速定位错在输入、模式、字段还是排序。

正确示例统计访问日志里最常见的状态码
# 假设 access.log 第 9 列是 HTTP 状态码
awk '{print $9}' access.log |
  sort |
  uniq -c |
  sort -nr |
  head

这条命令的读法是：awk 取出第 9 列，sort 把相同状态码排到一起，uniq -c 计数，第二个 sort -nr 按数字倒序排列，最后只看前几名。你现在应该能把一条管道按数据流方向解释出来。

3. 字段和分隔符：最常见的错位来源

fields

字段是文本处理的基本单位。cut -d, -f2 的意思是按逗号切开，取第 2 列；awk '{print $1}' 默认按连续空白切分，取第 1 列。问题在于真实文本常常不干净：名字里有空格，CSV 字段里有逗号，日志时间戳里有方括号，多个空格和制表符混在一起。

所以字段处理的判断句是：分隔符必须真的稳定。稳定意味着同一种分隔符在每行都表达同一层级的结构。如果 CSV 里有带引号的逗号，用 cut -d, 就可能切错；如果日志前几列固定但消息内容自由变化，取前几个字段可以，随便取后面的字段就危险。

逐行拆解观察字段后再取列
head -n 5 users.csv
cut -d, -f1,3 users.csv | head
awk -F',' '{print $1, $3}' users.csv | head

head 先看样本，确认逗号是否是可靠分隔符；cut 适合简单取列；awk -F',' 也按逗号切分，但后续可以加条件、计算和格式化。你现在应该能解释为什么字段错位会让统计结果完全不可信。

错误示例把带引号逗号的 CSV 当普通文本切
# name,email,note
# "Li, Hua",li@example.com,"paid user"
cut -d, -f1,2 users.csv
# 第一行会被切成 `"Li` 和 ` Hua"`，后面的列全部错位。

这类错误不一定报错，只是悄悄把列含义改掉。只要字段内部可能出现分隔符，普通字符串切分就不可靠。处理 CSV、JSON、XML、HTML 这类格式时，优先用知道格式规则的解析器；命令行仍然可以参与过滤、排序和汇总，但不要让 cut 或一条复杂正则承担完整解析任务。

4. 排序和去重：uniq 只处理相邻重复

sort uniq

uniq 不是全局去重器，它只会合并相邻的重复行。如果相同内容分散在文件不同位置，直接 uniq -c 会得到错误计数。因此统计频率前常见写法是先 sort，再 uniq -c。这条规则非常重要，因为它解释了很多“命令没报错，但结果不对”的情况。

排序也有边界。默认排序按字典序，10 可能排在 2 前面；需要数字排序时要加 -n，倒序加 -r。如果按某一列排序，要确认列分隔规则，避免排序键取错。排名类问题一般要写成：抽取目标字段，排序，相邻去重计数，再按数字倒序排。

错误示例没排序就计数
awk '{print $1}' access.log | uniq -c
# 如果同一个 IP 不相邻，会被分成多段计数，结果偏小或分散。
正确示例先排序再统计频率
awk '{print $1}' access.log |
  sort |
  uniq -c |
  sort -nr |
  head -20

如果排序对象不是整行，而是某一列，要把“排序键”说清楚。比如 sort -k2,2nr 表示只按第 2 列做数字倒序；如果忘了指定键，整行前面的计数字段、空格或路径都可能影响排名。排序前也要留意本地化环境，有些系统的排序顺序会受语言环境影响；需要稳定脚本时，常见做法是显式指定简单排序环境，例如在命令前加 LC_ALL=C。

逐行拆解按第二列数字倒序查看接口耗时
# 每行形如：/api/login 238
awk '{print $1, $2}' latency.log |
  sort -k2,2nr |
  head -10
# -k2,2nr：只看第 2 列，按数字倒序排列。

你现在应该能看到 sort | uniq -c | sort -nr 时立即说出：这是在做频率统计；看到 sort -k 时能追问排序键是哪一列、按数字还是按字典序、是否倒序。

5. 正则表达式：先收窄边界，再匹配内容

regex

正则表达式用模式描述文本。它适合匹配结构相对稳定的片段，例如邮箱、日期、错误码、日志级别、固定前缀。它不适合解析复杂嵌套结构，例如完整 HTML、任意 JSON 或编程语言源码。判断是否适合正则，关键看文本结构是否局部、稳定、可列出反例。

写正则时最常见的错误是模式过宽。.*error.* 看起来能找 error，实际可能匹配 terror、unexpected-error-code，也可能漏掉大写 ERROR。更稳的做法是利用行首、行尾、单词边界和固定分隔符，把范围收窄。

逐行拆解错误日志模式
grep -E '^ERROR[[:space:]]+[0-9]{4}:' app.log
# ^ERROR          行首必须是 ERROR
# [[:space:]]+    至少一个空白字符
# [0-9]{4}:       四位错误码后跟冒号

你现在应该能做什么：给一个正则至少准备一个应该匹配的样本、一个不该匹配的样本和一个边界样本。

6. awk：按行读取，按字段思考

awk

awk 的基本模型很稳定：读入一行，按分隔符切成字段，执行花括号里的动作。$0 表示整行，$1 表示第一列，NF 表示当前行字段数，NR 表示当前是第几行。它特别适合列选择、条件筛选、简单计算和格式化输出。

和 cut 相比，awk 更灵活。你可以写条件：只处理状态码大于等于 500 的行；也可以做累加：统计每个用户的请求次数；还可以格式化输出。它不是万能脚本语言，但在命令行数据整理里足够常用，值得把模型讲清楚。

正确示例筛选 5xx 请求并统计路径
awk '$9 >= 500 {print $7}' access.log |
  sort |
  uniq -c |
  sort -nr |
  head

这里 $9 >= 500 是条件，只有满足条件的行才执行 {print $7}。如果字段号错了，命令仍可能输出内容，但含义完全变了。所以每次写字段号前，要先用几行样本确认列位置。你现在应该能读懂 awk '条件 {动作}' 的基本结构。

7. sed：流式替换要先预览，再落盘

sed

sed 擅长按行做替换、删除和简单编辑。常见形式 s/旧模式/新文本/g 表示把每行里所有匹配的旧模式替换为新文本。它适合统一前缀、清理简单噪声、把日志中的敏感字段脱敏。它不适合做复杂语义改写。

危险点在于批量替换很容易误伤。尤其是加了原地修改参数时，命令一旦写错就会直接改文件。更稳的步骤是：先不加原地修改，把结果输出到屏幕或新文件；检查样本后再覆盖。对重要文件，先备份或用版本控制看 diff。

正确示例先预览脱敏结果
sed -E 's/token=[A-Za-z0-9_-]+/token=***REDACTED***/g' app.log | head
错误示例过宽替换会误伤
sed -E 's/id=.*/id=0/g' users.log
# 会从 id= 一直吃到行尾，可能删除后面的有效字段。

你现在应该能判断：替换命令是否有明确边界，是否先预览，是否会误删同一行后续内容。

8. 结构化数据：不要用文本技巧硬解析层级格式

structured data

很多文件看起来只是文本，实际有自己的语法。CSV 有引号、转义和换行规则；JSON 有对象、数组、字符串和转义；HTML 有嵌套标签。正则和 cut 只能看到字符，不理解这些层级规则。简单场景下它们可以快速抽样，复杂场景下继续硬写正则会让命令越来越长，结果越来越脆。

判断句是：如果字段边界会被引号、转义或嵌套改变，就换解析器。解析器的意思是“理解格式规则的工具”，例如 CSV 解析工具、jq、Python 的 csv / json 模块。解析器不是为了显得高级，而是为了让边界由格式规则决定，不由临时字符串切分决定。

正确示例用 Python csv 模块读取邮箱域名
python - <<'PY'
import csv
from collections import Counter

with open("users.csv", newline="", encoding="utf-8") as f:
    reader = csv.DictReader(f)
    domains = Counter(row["email"].split("@")[-1] for row in reader if row.get("email"))

for domain, count in domains.most_common(10):
    print(count, domain)
PY

这段代码比 cut -d, 长，但它把 CSV 的解析交给标准库，能正确处理带引号逗号的字段。命令行数据整理不是排斥脚本，而是选择合适层级的工具：简单文本用小工具，格式化数据用解析器，解析后的结果仍然可以继续交给 sort、uniq 或其他命令。

适合普通管道

每行独立，字段分隔稳定，没有转义和嵌套。例如固定格式日志、简单 TSV、命令输出中的单列列表。

优先考虑解析器

字段内部可能出现分隔符，数据有引号、转义、多行记录或嵌套结构。例如真实 CSV、JSON、HTML、XML。

你现在应该能做一个工具选择：先说明数据格式，再说明为什么普通切分够用或不够用。

9. 空输出和假结果：按阶梯排查，不要重写整条命令

debug ladder

长管道没有输出时，新手常见反应是重新写一条更复杂的命令。更稳的方法是从左到右逐段排查。先确认输入文件存在且有内容；再确认第一段过滤有没有把行全部筛掉；再确认字段号是否正确；最后确认排序、去重、替换有没有改变预期。每一步都只回答一个问题。

假结果比空输出更危险。空输出会提醒你有问题，假结果却像真的。比如字段取错后仍然能统计出排名，uniq -c 漏排序后仍然能输出数字，过宽正则仍然能替换出新文件。排查假结果时，不要只看命令是否成功退出，要把输出回到原始样本里核对几行。

逐行拆解把长管道拆成可检查的临时步骤
# 目标：统计 5xx 请求最多的路径
wc -l access.log
grep ' 5[0-9][0-9] ' access.log | head
grep ' 5[0-9][0-9] ' access.log | awk '{print $7}' | head
grep ' 5[0-9][0-9] ' access.log | awk '{print $7}' | sort | uniq -c | sort -nr | head

第一行确认输入规模，第二行确认过滤条件不会清空数据，第三行确认路径字段取对了，第四行才做聚合排名。这个顺序让错误位置变窄：如果第二行没有输出，问题在过滤；如果第三行输出的是状态码，问题在字段号；如果第四行计数分散，检查排序和去重。

你现在应该能在一条管道失败时写出排查阶梯，而不是只说“命令不对”。

10. 常见错误、后果和修法

diagnosis

直接对脏 CSV 使用 cut

错因是把逗号当成永远可靠的字段分隔。后果是遇到引号内逗号时列错位。修法是改用 CSV 解析工具，或先确认数据没有这种情况。

uniq 前没有排序

错因是误以为 uniq 会全局去重。后果是频率统计被拆成多段。修法是先 sort，再 uniq -c。

正则模式过宽

错因是用 .* 省事。后果是误匹配、误替换、批量改坏文件。修法是加行首行尾、分隔符、字符集合和反例测试。

字段号没验证

错因是直接复制别人命令。后果是取错列却不报错。修法是先用 head 或 awk 打印字段位置，再写正式管道。

把错误输出也当数据

错因是不了解标准输出和标准错误的区别。后果是统计里混进报错文本。修法是确认重定向，必要时单独保存错误日志。

一次写太长的管道

错因是跳过逐步验证。后果是空输出或错结果时很难定位。修法是每加一步都抽样检查。

11. 真实场景：从报错日志到可用结论

field guide

场景一：服务报警后，你要知道最近最常见的错误类型。不要先写复杂脚本。先看日志样本，确认错误级别、时间、路径、状态码分别在哪些列；再用 grep 找错误行，用 awk 取状态码或路径，用排序和去重统计频率。最后抽样检查排名靠前的几类，确认它们真的是同一类问题。

场景二：下载了一份用户表，想统计邮箱域名。先确认它是严格 CSV 还是简单逗号文本。如果邮箱列稳定，可以取出邮箱，再按 @ 后半段统计。如果 CSV 有引号、换行或转义，命令行简单切分就不稳，应该换成 Python 的 csv 模块或专用工具。

场景三：需要把日志里的 token 脱敏。先用正则匹配一个明确边界，例如 token= 后面只允许字母、数字、下划线、短横线；先把结果输出到屏幕检查，再写入新文件。不要一上来原地替换，更不要用过宽的 token=.* 把整行后半段吃掉。

完整链路是：观察样本，决定字段和模式，写最小命令，逐步扩展管道，抽样验证，再保存结果。你现在应该能把一个文本整理任务拆成可复查的步骤，而不是只背一条命令。

12. 离线实操：从访问日志生成可复查的错误排名

worked example

这个练习把“看样本、取字段、排序聚合、抽样验证、保存结果”串起来。关键不是背某条管道，而是每一步都能说明输入、输出和可能错在哪里。

逐步练习统计 5xx 路径并验证总数
mkdir -p data-lab && cd data-lab
cat > access.log <<'LOG'
10.0.0.1 - - [01/Jun/2026:10:00:01 +0000] "GET /api/login HTTP/1.1" 500 42
10.0.0.2 - - [01/Jun/2026:10:00:02 +0000] "GET /api/items HTTP/1.1" 200 120
10.0.0.3 - - [01/Jun/2026:10:00:03 +0000] "POST /api/login HTTP/1.1" 502 38
10.0.0.4 - - [01/Jun/2026:10:00:04 +0000] "GET /api/login HTTP/1.1" 500 44
LOG

head -n 3 access.log
awk '$9 >= 500 {print $7}' access.log | sort | uniq -c | sort -nr > error-paths.txt
cat error-paths.txt

awk '$9 >= 500 {count++} END {print "5xx lines:", count+0}' access.log
awk '{sum += $1} END {print "ranked lines:", sum+0}' error-paths.txt

管道检查表

  • 用 head 或样本先确认字段位置。
  • 每加一个管道阶段，都抽样看输出。
  • uniq -c 前先确认相同值已经相邻。
  • 聚合结果要能和原始行数或筛选行数对上。
  • 保存结果前，把错误输出和数据输出分开。

什么时候换工具

如果输入是真正 CSV、JSON、带引号换行的表格，或者要处理时区、编码、嵌套结构，就不要硬用空格切列。可以换成 Python 的 csv/json 模块、jq 或专用解析器。
