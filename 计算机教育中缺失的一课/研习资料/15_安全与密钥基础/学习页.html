安全基础 · 密钥与权限

把密钥、权限和信任边界讲清楚

日常开发里最常见的安全事故不是高深攻击，而是密钥提交到仓库、权限给太大、私钥无口令、把哈希当加密、泄露后只删除文件不轮换。学这一块，是为了在写代码、连服务器、配置 CI 和发布项目时少犯基础错误。

学完你应该能做什么

  1. 用资产、攻击者、入口、影响描述一个简单威胁模型。
  2. 解释权限最小化、secret 管理和密钥轮换。
  3. 区分哈希、加密、签名和编码。
  4. 诊断 SSH key 权限、token 泄露和误提交问题。

概念地图

安全判断可以从一条链路开始：先明确要保护的资产，再判断谁可能访问，接着控制权限和密钥，最后准备泄露后的处置。密钥不是配置文本，权限不是越方便越好，删除泄露文件也不等于安全恢复。可靠的安全习惯要覆盖预防、发现和止损。

资产权限密钥验证轮换
asset需要保护的东西，例如代码、数据、服务器、token、用户信息。
secret能授予访问能力的敏感值，例如 API token、私钥、数据库密码。
least privilege只给完成任务所需的最小权限和最短有效期。
hash单向摘要，用于完整性校验或密码存储，不用于还原原文。
encryption可用密钥还原原文，用于保密传输或存储。
signature用私钥证明来源和完整性，别人用公钥验证。

1. 先做小威胁模型：保护什么，防谁，入口在哪

threat model

安全讨论很容易变成泛泛而谈。更实际的起点是小威胁模型：资产是什么，谁可能滥用，攻击入口在哪里，成功后影响多大。个人项目和生产系统的威胁不同，本地脚本和公开服务的风险也不同。没有威胁模型，就容易把精力花在低风险细节上，同时漏掉真正危险的密钥和权限。

比如一个调用云服务的脚本，资产包括 API token、云账单额度、远程数据和本地代码。攻击入口可能是 Git 仓库、环境变量泄露、CI 日志、共享截图、误设公开权限。影响可能是账单被刷、数据被删、服务被滥用。把这些说清，才知道应该做 token 最小权限、不要提交密钥、限制日志输出、泄露后立即撤销。

小威胁模型不需要一开始就画复杂图。四句话就够：我要保护什么；谁可能拿到它；他们从哪里进来；如果成功会造成什么影响。比如“保护生产数据库密码；风险来自公开仓库、CI 日志和本地电脑丢失；入口是配置文件、环境变量和调试输出；影响是数据泄露和服务被改写”。这四句话会直接决定权限、存放位置和轮换策略。

你现在应该能做什么：遇到一个工具或项目，先说出它最重要的资产和最可能的泄露入口。

2. 权限：方便一时，风险长期存在

permissions

权限控制的核心是最小权限。能读就不要给写，能访问单个仓库就不要给整个组织，能短期有效就不要发长期 token。权限越大，泄露后的影响越大。很多事故不是因为攻击者很强，而是一个普通 token 拥有过大的能力。

文件权限也是同样道理。SSH 私钥、配置文件、凭证文件应该只有当前用户可读。Linux/macOS 下常见要求是私钥权限不能太开放，否则 SSH 会拒绝使用。权限错误不只是“命令不方便”，它说明本机其他用户或进程可能读到不该读的东西。

正确示例限制 SSH 私钥权限
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub

你现在应该能判断：一个凭证是否给了过大的访问范围，文件权限是否把秘密暴露给了不该访问的人。

3. secret 管理：不要把访问能力写进代码和仓库

secrets

secret 的危险在于它不是普通字符串，而是访问能力。API token、数据库密码、私钥、cookie、云服务凭证一旦泄露，别人可能直接使用。把 secret 写进源码、截图、日志、报错、提交历史和镜像层，都会变成真实风险。

日常做法是：本地用环境变量或专门配置文件，配置文件加入 .gitignore；团队使用 secret manager 或平台提供的密钥变量；CI 中只通过受控环境注入，不在日志里打印；不同环境使用不同 token；token 设置最小权限和有效期。不要为了方便把所有环境共用一个万能密钥。

错误示例把 token 写进代码
API_TOKEN = "sk_live_abc123_should_not_be_here"
client = Client(api_key=API_TOKEN)
正确示例从环境变量读取
import os

api_token = os.environ["API_TOKEN"]
client = Client(api_key=api_token)

这不是说环境变量绝对安全，而是它至少避免把密钥写进代码和提交历史。真正的安全还要靠权限范围、有效期、日志控制和泄露后的轮换。你现在应该能识别哪些字符串其实是访问能力。

4. 本地凭证文件和仓库扫描：把“不会提交”变成规则

local secrets

很多泄露发生在“我只是临时写一下”的瞬间：本地新建了 .env，调试脚本里放了 token，Notebook 输出里出现连接串，随后一次 git add . 全部进了暂存区。不要依赖记忆来防止提交 secret，要把规则写进项目：敏感配置文件加入 .gitignore，提供 .env.example 只写变量名和示例格式，提交前运行 secret 扫描。

.gitignore 只能防止未跟踪文件被误加；如果 secret 已经进入 Git 历史，再加 ignore 也不会让它消失。所以本地保护分两层：提交前避免进入仓库，泄露后立即撤销或轮换。扫描工具也不是万能，它只能发现常见模式，不能替代最小权限、短有效期和日志脱敏。

正确示例给项目留下安全配置模板
# .gitignore
.env
.env.local
secrets/

# .env.example
API_TOKEN=replace-with-your-own-token
DATABASE_URL=postgres://user:password@host/db
错误示例一次性添加所有文件前不检查
git add .
git commit -m "debug config"

# 如果 .env、日志或 notebook 输出里有 secret，会一起进入提交。

更稳的习惯是提交前看差异：git diff --cached 能让你在提交前发现误加内容。团队项目可以把 secret 扫描放进 pre-commit 或 CI，但本地仍要知道哪些文件本来就不该进入仓库。你现在应该能区分：样例配置可以提交，真实凭证不能提交；ignore 能预防新文件，不能撤回已泄露密钥。

5. 哈希、加密、签名、编码：不要混用这些词

crypto terms

哈希是单向摘要。给定输入可以算出固定长度摘要，但不能从摘要还原原文。它适合完整性校验、去重、密码存储的基础组件。密码存储不能只用普通快速哈希，还要使用带盐、慢速、专用的密码哈希算法，避免离线暴力猜测。

加密是为了保密。明文通过密钥变成密文，拥有正确密钥的人可以解密。签名是为了证明来源和完整性：私钥签名，公钥验证。编码不是安全措施，例如 Base64 只是换一种文本表示，任何人都能解码。把 Base64 当加密，是非常常见的误解。

逐行拆解校验文件摘要
sha256sum release.tar.gz
# 输出摘要后，与发布方提供的 SHA-256 值对比。
# 值一致说明文件内容大概率未被意外损坏或篡改。

你现在应该能做什么：看到“保护密码”“隐藏内容”“验证来源”“文本转码”时，分别判断应该讨论哈希、加密、签名还是编码。

6. SSH key：公钥可以给别人，私钥必须自己守住

ssh

SSH key 通常是一对密钥：私钥留在本机，公钥放到服务器或代码平台。登录时，服务器用公钥验证你是否持有对应私钥。公钥可以公开，私钥不能泄露。给私钥加 passphrase，可以在文件被复制时增加一道保护。

常见误区是把私钥发给服务器、把私钥提交到仓库、多个用途共用同一把 key、旧 key 不再使用却不删除。更稳的做法是为不同用途区分 key，给私钥设置口令，使用 ssh-agent 管理解锁，离职或设备丢失时及时从平台删除对应公钥。

正确示例生成新的 ed25519 key
ssh-keygen -t ed25519 -C "your_email@example.com"
ssh-add ~/.ssh/id_ed25519
错误示例复制私钥给远程机器
scp ~/.ssh/id_ed25519 user@example.com:~/
# 私钥不应该这样复制到远程机器。

你现在应该能解释：为什么平台需要的是公钥，为什么私钥泄露后要撤销相关公钥并生成新 key。

7. 泄露处置：删除文件不是轮换密钥

incident

一旦 secret 被提交到仓库、发到聊天、进入日志或打进镜像，第一反应不应该是“删掉就好”。泄露发生后，要假设它已经被别人看到或被自动扫描器抓到。正确顺序通常是：立即撤销或轮换密钥，检查滥用记录，清理暴露位置，修复导致泄露的流程，再补扫描和权限限制。

Git 历史尤其容易误判。即使最新提交删除了 token，历史提交里仍然可能存在。公开仓库中的密钥可能在几分钟内被扫描。清理历史可以减少继续暴露，但不能替代撤销密钥。只要密钥曾经公开，就应该换新。

逐行拆解泄露后的处理顺序
# 1. 在服务平台撤销泄露 token
# 2. 生成权限更小的新 token
# 3. 检查访问日志和账单
# 4. 清理仓库、日志或镜像中的暴露内容
# 5. 增加 secret 扫描和 .gitignore 规则

你现在应该能判断：面对 token 泄露，最先做的不是美化提交历史，而是让旧 token 失效。

8. 密码存储和日志脱敏：不要让调试变成泄露入口

passwords logs

用户密码和 API secret 的处理方式不同。API secret 通常需要原样使用，所以要放在受控环境里；用户密码通常不应该能被系统还原，应该使用专门的密码哈希算法保存验证材料。普通快速哈希不够，因为攻击者拿到数据库后可以离线高速猜测。密码存储要使用带盐、慢速、抗暴力破解的方案，例如 Argon2、bcrypt 或 scrypt。

日志是另一个高频泄露点。开发者为了排查问题，常常把完整配置、请求头、连接串、cookie 或 token 打印出来。日志会进入 CI、监控平台、错误追踪系统、截图和聊天记录，传播范围比想象中大。更稳的做法是默认脱敏：只显示必要字段，secret 只显示前后极少字符或完全不显示。

调试时可以问三个问题：这行日志是否包含访问能力，是否会离开本机，是否有保留期限。如果答案不确定，就不要打印完整值。你现在应该能区分“为了排查需要知道是否存在某个值”和“需要看到这个值的完整内容”这两件事。

错误示例日志打印完整连接串
print(f"connecting with DATABASE_URL={DATABASE_URL}")
# 连接串里可能包含用户名、密码、主机和数据库名。
正确示例只打印脱敏状态
print("DATABASE_URL is configured:", bool(DATABASE_URL))
# 只说明是否配置，不暴露完整 secret。

9. 下载和依赖：校验完整性，也要验证来源

supply chain

从网上下载脚本、二进制、模型权重或依赖包时，安全问题不只在本机权限，也在来源。哈希摘要可以帮助你检查文件内容是否和某个摘要一致；签名可以帮助你验证文件是否由对应私钥持有者发布。两者都比“看起来像官网”更可靠，但前提是摘要或公钥本身来自可信渠道。

常见危险动作是直接运行远程脚本，例如 curl ... | sh。这类命令把下载和执行连在一起，用户没有机会检查内容；如果网络、域名、脚本或中间环节出问题，影响会很大。更稳的步骤是先下载到文件，查看内容或来源，再按官方说明执行。对生产环境，还要固定版本，避免今天和明天下载到不同内容。

依赖管理也遵循同样逻辑：不要随意把未知包加入项目，不要忽略 lockfile，不要让 CI 每次安装不可预测的新版本。安全不是完全不下载，而是让来源、版本、完整性和权限都可检查。你现在应该能解释：哈希解决“内容是否一致”，签名进一步解决“是否来自可信发布者”。

错误示例下载后立刻执行
curl https://example.com/install.sh | sh
# 没有保存、查看、校验脚本内容，就直接执行。
正确示例先下载再检查
curl -O https://example.com/install.sh
less install.sh
sha256sum install.sh

10. 常见错误、后果和修法

diagnosis

把 Base64 当加密

错因是混淆编码和加密。后果是任何人都能解码。修法是用真正加密方案，并保护密钥。

提交 token 后只删除

错因是忽略历史和自动扫描。后果是旧 token 仍可被使用。修法是立即撤销或轮换，再清理暴露位置。

私钥权限过宽

错因是本地文件权限不当。后果是其他用户或进程可能读取，SSH 也可能拒绝使用。修法是限制 ~/.ssh 和私钥权限。

所有环境共用万能 token

错因是追求方便。后果是一个环境泄露影响全部系统。修法是分环境、分用途、最小权限、短有效期。

日志打印 secret

错因是调试时直接输出配置。后果是 CI 日志、监控、截图都可能泄露。修法是脱敏显示，只打印前后几位或完全不打印。

公钥和私钥混淆

错因是不理解 key pair。后果是把私钥交给平台或别人。修法是记住公钥可分发，私钥只留本机。

11. 真实场景：日常开发里的安全判断

field guide

场景一：你要在 CI 中调用云服务。不要把 token 写进仓库，也不要给管理员级权限。应该在平台 secret 设置里注入，给它最小权限，限制有效期，并确认日志不会打印它。CI 失败时，日志可以显示“缺少 API_TOKEN”，但不能显示 token 值。

场景二：你在 GitHub 添加 SSH key。平台要求的是公钥，通常是 id_ed25519.pub 的内容；私钥 id_ed25519 不能上传。添加后，如果设备丢失或怀疑私钥泄露，要从平台删除对应公钥，并生成新 key。

场景三：你下载一个 release 文件。发布方提供 SHA-256 摘要时，你可以本地计算摘要对比，检查文件是否损坏或被替换。摘要一致不是万能安全保证，因为如果攻击者同时替换了文件和网页上的摘要，你仍会被骗；更强的验证需要签名和可信公钥。

完整链路是：识别资产，限制权限，保护 secret，区分哈希/加密/签名，使用 SSH key 时守住私钥，泄露后先撤销再清理。你现在应该能把一个“密钥相关问题”拆成访问能力、暴露位置、影响范围和处置动作。
